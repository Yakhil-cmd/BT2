### Title
Unverified Participant Contributions in CKD Protocol Allow Malicious Participant to Corrupt Derived Confidential Key - (`File: src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates `(norm_big_y, norm_big_c)` shares from all participants by simple addition, with no commitment scheme, hash-binding, or zero-knowledge proof verifying that each participant computed their share honestly. A malicious participant can send arbitrary elliptic-curve points, corrupting the final `CKDOutput` and causing all honest parties to accept an unusable or attacker-biased derived confidential key.

### Finding Description

The `do_ckd_coordinator` function collects each participant's `(norm_big_y, norm_big_c)` pair and unconditionally adds them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation each participant is supposed to perform is:

```rust
let big_s = hash_point * private_share.to_scalar();  // S = x_i · H(pk, app_id)
let big_c = big_s + app_pk * y.0;                    // C = S + y·A
let big_y = ElementG1::generator() * y.0;            // Y = y·G
``` [2](#0-1) 

Each participant sends their Lagrange-weighted `(λ_i · Y_i, λ_i · C_i)` privately to the coordinator: [3](#0-2) 

There is **no commitment round**, **no hash-binding**, and **no zero-knowledge proof** that the sent points were derived from the participant's actual private share and a valid random `y`. The coordinator has no way to distinguish an honest contribution from an arbitrary group element.

Compare this to the DKG protocol, which uses a two-round hash-commitment scheme (`commitment_hash` broadcast before the actual commitment) and a Schnorr proof-of-knowledge to bind each participant's polynomial commitment to their secret: [4](#0-3) [5](#0-4) 

The OT-based presign similarly verifies aggregated shares against public commitments before accepting them: [6](#0-5) 

The CKD protocol has no equivalent guard.

### Impact Explanation

A malicious participant sends `(big_y_malicious, big_c_malicious)` = `(honest_big_y + Δ_Y, honest_big_c + Δ_C)` for attacker-chosen group elements `Δ_Y`, `Δ_C`. The coordinator computes:

```
Y_final  = Σ λ_i · Y_i  +  λ_malicious · Δ_Y
C_final  = H(pk,app_id)·msk + app_pk·y  +  λ_malicious · Δ_C
```

After the TEE unmasks with `app_sk`: the derived confidential key becomes `H(pk,app_id)·msk + λ_malicious·Δ_C - app_sk·λ_malicious·Δ_Y`, which is an attacker-biased value that no honest party can predict or use. Setting `Δ_C = identity` and `Δ_Y = identity` (i.e., sending the identity point) silently drops the malicious participant's contribution, causing the output to be computed over an incomplete set of shares — producing a key that is cryptographically inconsistent with the master secret key. In either case, all honest parties accept and store a corrupted `CKDOutput` with no indication of failure.

This maps to: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The participant list is caller-supplied and can include an adversary. No special privilege is required — the malicious party only needs to be included in `participants` and send a message to the coordinator. The attack requires no cryptographic break, no leaked key, and no external dependency.

### Recommendation

Before the coordinator aggregates contributions, introduce a commitment-then-reveal round analogous to the DKG protocol:

1. **Round 1 (commit):** Each participant broadcasts `H(session_id, participant_id, norm_big_y, norm_big_c)`.
2. **Round 2 (reveal):** Each participant sends `(norm_big_y, norm_big_c)` to the coordinator.
3. **Verify:** The coordinator checks each revealed pair hashes to the committed value before summing.

Additionally, require each participant to provide a zero-knowledge proof of knowledge of `y_i` and that `big_c_i` was formed as `H(pk,app_id)·x_i + app_pk·y_i` (a Chaum-Pedersen or similar DLEQ proof), binding the contribution to the participant's actual private share.

### Proof of Concept

1. Honest participants `{P1, P2, P3}` run `ckd(...)` with coordinator `P1`.
2. Malicious `P2` intercepts the protocol at `do_ckd_participant` and instead of calling `compute_signature_share`, sends `(identity, identity)` to the coordinator.
3. `do_ckd_coordinator` sums contributions: `norm_big_y = P1_Y + identity + P3_Y`, `norm_big_c = P1_C + identity + P3_C`.
4. The resulting `CKDOutput` corresponds to a key derived from only `P1` and `P3`'s shares — not the full threshold secret — producing a key that is inconsistent with `msk` and cannot be used to decrypt any data encrypted to the legitimate derived key.
5. No error is raised; all honest parties accept the corrupted output. [7](#0-6)
