### Title
Reward Address Can Add Funds to Pool But Cannot Initiate Exit — Inconsistent Authorization Causes Permanent Fund Freeze - (File: `src/pool/pool.cairo`)

---

### Summary

`exit_delegation_pool_intent` in `Pool` only permits the pool member itself to call it (via `get_caller_address()`), while `add_to_delegation_pool` and `claim_rewards` explicitly authorize **both** the pool member and its `reward_address`. Because `add_to_delegation_pool` transfers tokens **from the caller** (i.e., from the reward_address's own balance when it is the caller), the reward_address can permanently lock its own tokens inside a pool position it can never unwind.

---

### Finding Description

The Pool contract establishes a two-party authorization model for pool-member operations. Two functions explicitly permit either the pool member or its registered reward address to act:

**`add_to_delegation_pool`** — pool_member OR reward_address authorized: [1](#0-0) 

Crucially, when the reward_address is the caller, tokens are pulled **from the reward_address's own balance**: [2](#0-1) 

**`claim_rewards`** — pool_member OR reward_address authorized: [3](#0-2) 

However, **`exit_delegation_pool_intent`** silently restricts the caller to the pool member only by using `get_caller_address()` as the identity with no reward_address fallback: [4](#0-3) 

The interface documentation confirms this restriction is intentional but does not justify the asymmetry: [5](#0-4) 

When exit finally executes, funds are returned exclusively to `pool_member`, not to whoever contributed them: [6](#0-5) 

The same asymmetry exists in the Staking contract: `increase_stake` allows staker OR reward_address to contribute tokens from their own balance, but `unstake_intent` only allows the staker: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

The reward_address can call `add_to_delegation_pool` and spend tokens **from its own wallet** to top up a pool member's position. Those tokens are credited to the pool member's balance and, on exit, are returned to the pool member address — not to the reward_address that funded them. Because `exit_delegation_pool_intent` is gated exclusively to the pool member, the reward_address has no path to initiate recovery of its contributed tokens. If the pool member address is inaccessible (lost key, compromised EOA, or a smart contract that does not implement exit calls), the reward_address's contributed funds are **permanently frozen** inside the pool position.

This matches the allowed impact: **Permanent freezing of funds (High)**.

---

### Likelihood Explanation

The scenario is realistic under two common patterns:

1. **Smart-contract pool members**: A protocol or DAO deploys a contract as the pool member and sets a separate EOA or multisig as the reward_address for operational management. If the pool-member contract does not implement a call to `exit_delegation_pool_intent`, the reward_address management key cannot trigger exit.

2. **Reward-address top-ups**: A user sets their reward_address to a hot wallet or automation contract and uses it to call `add_to_delegation_pool` to compound returns. If the cold-wallet pool_member key is later lost, the hot wallet's contributed tokens are unrecoverable.

Both patterns are natural uses of the two-party authorization model the protocol itself establishes.

---

### Recommendation

Extend `exit_delegation_pool_intent` to accept an explicit `pool_member` parameter (mirroring `add_to_delegation_pool`) and add a reward_address authorization check, consistent with the rest of the authorization model:

```cairo
fn exit_delegation_pool_intent(
    ref self: ContractState,
    pool_member: ContractAddress,
    amount: Amount,
) {
    let caller = get_caller_address();
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::CALLER_CANNOT_EXIT_POOL,
    );
    // ... rest of logic unchanged
}
```

Apply the same fix to `unstake_intent` in `Staking` to allow the reward_address to initiate unstake on behalf of the staker.

---

### Proof of Concept

1. Alice deploys a smart contract `SC` as her pool member and registers her EOA `R` as the reward_address.
2. `R` calls `add_to_delegation_pool(pool_member: SC, amount: 1000)` — succeeds; 1000 tokens are transferred from `R`'s balance and credited to `SC`'s pool position. [9](#0-8) 
3. `SC` does not implement a call to `exit_delegation_pool_intent`.
4. `R` attempts to call `exit_delegation_pool_intent(amount: 1000)` — the call resolves `pool_member = get_caller_address() = R`, which has no pool position, and reverts with `POOL_MEMBER_DOES_NOT_EXIST`. [4](#0-3) 
5. There is no other entry point through which `R` can initiate exit for `SC`'s position.
6. The 1000 tokens contributed by `R` are permanently frozen inside `SC`'s pool position. [6](#0-5)
