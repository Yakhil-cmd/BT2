### Title
Off-by-One in `is_commission_commitment_active` Allows Commission Increase One Epoch Before Commitment Expires - (File: src/staking/staking.cairo)

### Summary
`is_commission_commitment_active` uses a strict `<` comparison instead of `<=`, causing a commission commitment to be treated as expired at `expiration_epoch` itself rather than after it. A malicious staker can exploit this to increase their commission during the epoch that delegators believed was still protected by the commitment, stealing one epoch of delegator yield.

### Finding Description
The function `is_commission_commitment_active` is defined as:

```rust
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() < commission_commitment.expiration_epoch
}
```

The spec states the commitment "expires **in** `expiration_epoch`", meaning the commitment should remain active **through** `expiration_epoch` (inclusive) and only lapse afterward. The strict `<` operator causes the commitment to be treated as expired the moment `current_epoch == expiration_epoch`, one epoch too early.

This is the direct analog of the reported Solidity bug: a deadline comparison is reversed/off-by-one, allowing an action to be taken before the deadline has truly passed.

When the commitment is expired, `update_commission` falls into the else-branch:

```rust
} else {
    assert!(
        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
    );
}
```

This prevents a direct commission increase. However, `set_commission_commitment` also checks expiry:

```rust
assert!(
    !self.is_commission_commitment_active(:commission_commitment),
    "{}",
    Error::COMMISSION_COMMITMENT_EXISTS,
);
```

Because the old commitment is already considered expired at `expiration_epoch`, the staker can immediately set a **new** commitment with a higher `max_commission` and then call `set_commission` to raise the commission — all within the same epoch that delegators believed was still protected.

### Impact Explanation
Delegators who joined a pool under a commission commitment with `expiration_epoch = E` and `max_commission = X` expect their commission to be capped at X through epoch E. Due to the off-by-one, at epoch E the staker can:

1. Call `set_commission_commitment(max_commission: Y, expiration_epoch: E+1)` — succeeds because the old commitment is already considered expired.
2. Call `set_commission(commission: Y)` — succeeds because the new commitment is active and `Y <= Y`.

Delegators' rewards for epoch E are reduced by the higher commission rate. This constitutes **theft of unclaimed yield** (High severity).

### Likelihood Explanation
Any staker who has set a commission commitment and later wishes to raise their commission can exploit this at the exact `expiration_epoch`. The entry point (`set_commission_commitment` → `set_commission`) is publicly callable by the staker with no privileged role required beyond being a registered staker. The window is one epoch, but the staker controls when they act within that epoch.

### Recommendation
Change the strict less-than to less-than-or-equal so the commitment remains active during `expiration_epoch`:

```rust
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() <= commission_commitment.expiration_epoch
}
```

This aligns with the spec's language ("expires **in** `expiration_epoch`") and ensures delegators are protected through the full epoch they were promised.

### Proof of Concept

1. Current epoch = 5. Staker calls `set_commission_commitment(max_commission: 500, expiration_epoch: 6)`. Delegators join expecting commission ≤ 500 through epoch 6.
2. Epoch advances to 6. `is_commission_commitment_active` evaluates `6 < 6` → `false` (expired).
3. Staker calls `set_commission_commitment(max_commission: 900, expiration_epoch: 7)`. Passes because `!false` = `true` at line 762–766.
4. Staker calls `set_commission(commission: 900)`. Passes because new commitment is active (`6 < 7`) and `900 <= 900`.
5. Delegators' rewards for epoch 6 are computed with 9% commission instead of the promised ≤ 5%.
