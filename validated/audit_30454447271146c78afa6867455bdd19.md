### Title
Gateway Admits Unsigned Invoke Transactions for Accounts with Pending Deploy-Account via `skip_stateful_validations` Bypass - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (and therefore all signature verification) for any invoke transaction with `nonce = 1` targeting an address whose `sender_address` appears in `account_tx_in_pool_or_recent_block`. An attacker who observes a victim's `deploy_account` transaction in the mempool can submit an invoke transaction with an arbitrary/garbage signature on behalf of the victim's undeployed address and have it admitted to the mempool without any authorization check.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations`, when `validate = false`, execution returns immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
```

`perform_pre_validation_stage` only checks nonce (with `strict_nonce_check = false`), fee bounds, and balance — it does **not** verify the transaction signature.

The `account_tx_in_pool_or_recent_block` check is purely address-based:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

It returns `true` for any address that has ever had a transaction in the mempool or a recent committed block — it does not verify that the *submitter* of the new invoke transaction is the legitimate owner of that address.

**Attack scenario:**

1. Victim pre-funds address `V` and submits a `deploy_account` transaction (nonce 0). This transaction enters the mempool, so `account_tx_in_pool_or_recent_block(V) == true`.
2. Attacker crafts an invoke transaction: `sender_address = V`, `nonce = 1`, arbitrary calldata, garbage/empty signature, sufficient resource bounds.
3. Gateway stateless validation passes (no signature check there).
4. `validate_nonce` passes: `0 <= 1 <= 0 + max_allowed_nonce_gap`.
5. `validate_by_mempool` passes: mempool only checks for duplicate hash/nonce, not signatures.
6. `skip_stateful_validations` returns `true`: nonce=1 ✓, account_nonce=0 ✓, `account_tx_in_pool_or_recent_block(V)` = true ✓.
7. `__validate__` is never called. The transaction is admitted to the mempool with a garbage signature.

### Impact Explanation

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- An attacker can inject a fake nonce-1 invoke transaction for any account that has a pending `deploy_account`. The mempool then holds a `DuplicateNonce` entry for that slot, causing the victim's legitimate nonce-1 invoke to be rejected with `MempoolError::DuplicateNonce` when submitted.
- The attacker can spam the mempool with invalid transactions targeting all accounts in the deploy-account UX flow, consuming mempool capacity and degrading sequencer throughput.
- The invalid transactions will eventually revert during batcher execution, wasting block resources.

### Likelihood Explanation

The `deploy_account + invoke` UX flow is an explicitly supported and documented feature. Any account going through this flow is vulnerable during the window between submitting `deploy_account` and having it committed. The mempool is observable (transactions are propagated via P2P), so an attacker can identify targets. The attack requires only crafting a syntactically valid invoke transaction with a garbage signature — no special privileges or resources are needed.

### Recommendation

The `skip_stateful_validations` check should verify that the submitter is the legitimate owner of the sender address, not merely that *some* transaction for that address exists in the mempool. One approach: require that the `deploy_account` transaction hash for the account is provided and matches a known pending deploy_account in the mempool (as the `native_blockifier` path does via `deploy_account_tx_hash`). Alternatively, perform a lightweight signature pre-check (e.g., ECDSA recovery) before skipping the full `__validate__` call, or restrict the skip to cases where the deploy_account transaction was submitted in the same batch/request.

### Proof of Concept

1. Victim submits `deploy_account` for address `V` (nonce 0). Mempool now has `account_tx_in_pool_or_recent_block(V) == true`.
2. Attacker submits:
   ```
   RpcInvokeTransactionV3 {
       sender_address: V,
       nonce: 1,
       calldata: [/* arbitrary */],
       signature: [],  // empty / garbage
       resource_bounds: /* valid non-zero bounds */,
       ...
   }
   ```
3. Gateway stateless validator passes (no signature check).
4. `validate_nonce`: `0 <= 1 <= max_allowed_nonce_gap` → passes.
5. `validate_by_mempool` → `Mempool::validate_tx` → no duplicate hash, nonce gap allowed → passes.
6. `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(V)=true` → returns `true`.
7. `run_validate_entry_point` with `skip_validate=true` → `execution_flags.validate=false` → `perform_validations` returns after `perform_pre_validation_stage` (balance/nonce only) without calling `__validate__`.
8. `mempool_client.add_tx(...)` is called. The unsigned transaction is now in the mempool.
9. Victim submits their legitimate nonce-1 invoke → mempool rejects with `DuplicateNonce`.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
