### Title
WalletConnect `take_offer` Confirmation Displays Understated Total Cost Due to Incorrect Multi-NFT Royalty Calculation - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

The `formatAmountWithRoyalties` function in the WalletConnect command display pipeline incorrectly calculates the "Total Amount with Royalties" shown to users before they approve a `chia_wallet.take_offer` command. When an offer involves multiple NFTs, the function divides the purchase amount by the number of NFTs (`splitAmount = amount / BigInt(royaltyPercentages.length)`) before computing each NFT's royalty, systematically understating the true total by a factor of N. A malicious WalletConnect-connected dApp can exploit this to make a victim approve an offer that costs significantly more XCH/CAT than the confirmation dialog displays.

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` is the sole function responsible for computing the `amountWithRoyalties` field shown in the WalletConnect approval dialog:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts, lines 354-375
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length); // ← BUG: divides before royalty calc
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The correct formula for total royalties when buying N NFTs for a total price P is:

```
royaltyTotal = Σ (P × royalty_i / 10000)   for each NFT i
```

The buggy formula computes:

```
splitAmount  = P / N                         (integer division, truncates)
royaltyTotal = Σ (splitAmount × royalty_i / 10000)
             = (P / N) × Σ (royalty_i / 10000)
```

This is exactly `1/N` of the correct royalty total. With N=2 NFTs the displayed total is half the actual; with N=4 it is one-quarter.

The wallet daemon's own royalty path, used in the interactive offer builder, passes the **full** fungible asset amount to `calculate_royalties` for each NFT independently, confirming the correct per-NFT-on-full-price semantics:

```typescript
// packages/gui/src/components/offers2/OfferBuilderProvider.tsx, lines 101-113
const offeredFungibleAssets = [
  ...(offeredXCH ?? []).map(({ amount }) =>
    fungibleAssetFromAssetIdAndAmount('xch', chiaToMojo(amount))),
  ...
];
const requestedRoyaltiesRequest: CalculateRoyaltiesRequest = {
  royaltyAssets: requestedRoyaltyAssets,
  fungibleAssets: offeredFungibleAssets,   // full amounts, not split
};
``` [2](#0-1) 

The `amountWithRoyalties` value is rendered verbatim in the WalletConnect confirmation dialog as the authoritative "Total Amount with Royalties" the user will pay:

```tsx
// packages/gui/src/electron/dialogs/Confirm/Confirm.tsx, lines 109-114
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties}{' '}
    {networkPrefix ? networkPrefix.toUpperCase() : 'XCH'}
  </div>
)}
``` [3](#0-2) 

After the user approves, the actual `take_offer` RPC is dispatched with the original offer blob; the wallet daemon enforces the correct (higher) royalties on-chain.

### Impact Explanation

A victim who approves a WalletConnect `take_offer` for a multi-NFT offer will pay more XCH/CAT than the confirmation dialog stated. The gap scales with both the number of NFTs and their royalty percentages.

**Concrete example — 2 NFTs, each 50 % royalty, purchase price 1 XCH:**

| | Displayed | Actual |
|---|---|---|
| Base price | 1 XCH | 1 XCH |
| NFT-1 royalty (50 %) | 0.25 XCH | 0.5 XCH |
| NFT-2 royalty (50 %) | 0.25 XCH | 0.5 XCH |
| **Total** | **1.5 XCH** | **2 XCH** |

The user approves believing they spend 1.5 XCH; they actually spend 2 XCH — a 33 % overcharge relative to what was shown. With 4 NFTs the displayed total is 25 % of the actual royalty burden.

This satisfies the **High** impact criterion: "Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."

### Likelihood Explanation

**Low.** The attacker must:
1. Establish a WalletConnect session with the victim (requires victim to pair with the dApp).
2. Have minted NFTs with high royalty percentages on-chain.
3. Construct a valid multi-NFT offer and send `chia_wallet.take_offer` via WalletConnect.

The victim must approve the dialog. No leaked keys or host compromise are required; the entire path is reachable by an unprivileged dApp operator.

### Recommendation

Replace the `splitAmount` division with per-NFT royalty calculation on the full purchase amount:

```typescript
// Correct: each NFT's royalty is calculated on the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3) 

This aligns the WalletConnect confirmation display with the wallet daemon's own `calculate_royalties` semantics used in `OfferBuilderProvider`.

### Proof of Concept

The existing test suite already encodes the buggy behavior and can serve as a regression baseline. The discrepancy is observable by comparing the test's expected `amountWithRoyalties` against the correct value:

```
// packages/gui/src/electron/commands/parseCommandDisplay.test.ts, lines 251-334
// Offer: 2 NFTs (royalty_percentage 500 and 10) for 100_000_000 mojos (0.0001 XCH)

// Current (buggy) output:
amountWithRoyalties: '0.00010255'
// splitAmount = 100_000_000 / 2 = 50_000_000
// royalty = (50_000_000 * 500 / 10_000) + (50_000_000 * 10 / 10_000)
//         = 2_500_000 + 50_000 = 2_550_000
// total   = 102_550_000 mojos = 0.00010255 XCH  ← shown to user

// Correct output (per-NFT on full amount):
// royalty = (100_000_000 * 500 / 10_000) + (100_000_000 * 10 / 10_000)
//         = 5_000_000 + 100_000 = 5_100_000
// total   = 105_100_000 mojos = 0.0001051 XCH   ← what user actually pays
``` [5](#0-4) 

A malicious dApp mints two NFTs each with `royalty_percentage = 5000` (50 %), constructs an offer requesting 1 XCH for both NFTs, and sends `chia_wallet.take_offer` via WalletConnect. The victim's confirmation dialog shows "Total Amount with Royalties: 1.5 XCH"; the wallet daemon enforces 2 XCH on-chain. The victim loses 0.5 XCH beyond what they consented to.
