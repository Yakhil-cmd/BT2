### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Factory Fee Cap System — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no factory-level cap validation, while every other admin fee setter (`setPoolAdminFees`, `setPoolProtocolFee`, `setFeeCaps`) is strictly bounded by `maxAdminSpreadFeeE6` / `HARD_MAX_SPREAD_FEE_E6`. A pool admin can set per-bin additional fees to any value up to `uint16` max (65 535 in E6 = ~6.55%) on any bin, bypassing the factory's fee cap invariant and causing swaps through that bin to execute at prices above what the oracle and cap system permit.

---

### Finding Description

The factory enforces a layered fee cap system for spread fees:

- Hard ceiling: `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%)
- Owner-configurable cap: `maxAdminSpreadFeeE6 ≤ HARD_MAX_SPREAD_FEE_E6`
- Admin setter enforces the cap: `if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh()` [1](#0-0) [2](#0-1) 

However, `setPoolBinAdditionalFees` performs **no cap check** — it forwards the caller-supplied values directly to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitude: [4](#0-3) 

These per-bin fees are then **added on top of** the oracle-derived `baseFeeX64` in every swap path through that bin:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7) 

The `BinState` struct stores `addFeeBuyE6` and `addFeeSellE6` as `uint16`, giving a maximum of 65 535 in E6 units (~6.55%): [9](#0-8) 

---

### Impact Explanation

The effective ask price for a swap through a bin with `addFeeBuyE6 = 65535` becomes:

```
ask = marginalPrice × (1 + baseFee + 0.065535) × (1e8 / (1e8 - notionalFee))
```

This exceeds the oracle-implied price by up to **6.55% additional spread** on top of the oracle's own spread and the capped global spread fee — entirely outside the factory's cap enforcement. The effective bid/ask quote reaching the swap is therefore unbounded relative to the cap system's intent, constituting bad-price execution for traders swapping through that bin.

The per-bin fee is split between LPs and the protocol/admin via `spreadFeeE6`, so the admin indirectly benefits from the inflated fee through their share of the spread fee split. [10](#0-9) 

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained to operate "inside caps." The factory's cap system is the primary protection for traders. Any pool admin — including one that is compromised, malicious, or simply misconfigured — can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` immediately, with no timelock, no protocol approval, and no revert. The call is permissioned only by `onlyPoolAdmin`, which is a single address set at pool creation.

---

### Recommendation

Add a factory-level cap check in `setPoolBinAdditionalFees` analogous to the check in `setPoolAdminFees`. Either reuse `maxAdminSpreadFeeE6` as the ceiling for per-bin additional fees, or introduce a dedicated `maxBinAdditionalFeeE6` cap:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Apply the same cap at pool creation time when bin states are initialized from `nonNegativeBinDataArray` / `negativeBinDataArray`.

---

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 200_000` (20% hard cap).
2. Pool admin calls `factory.setPoolAdminFees(pool, 200_000, 0)` — this succeeds and is the maximum allowed global spread fee.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — this succeeds with **no revert**, setting an additional ~6.55% fee on bin 0 in both directions.
4. A trader swaps through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, which is the oracle spread plus an additional 6.55% — entirely outside the factory's cap system.
5. The trader pays ~6.55% more than the oracle-implied price + capped spread fee, with no on-chain protection. [3](#0-2) [4](#0-3)
