### Title
Malicious Pool Server Can Supply Unbounded `relativeLockHeight` to Permanently Lock a User's Plot NFT — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
When a user joins or changes a farming pool, the GUI fetches pool parameters (`targetPuzzleHash`, `relativeLockHeight`) directly from the pool operator's `/pool_info` endpoint and passes them to `pw_join_pool` / `create_new_pool_wallet` with no bounds validation. A malicious pool operator can serve an arbitrarily large `relativeLockHeight` (e.g., `4294967295`), locking the user's Plot NFT on-chain for an effectively permanent duration, preventing them from ever switching pools or self-pooling.

### Finding Description

The join-pool flow in `PlotNFTSelectPool.tsx` calls `getPoolInfo(normalizedPoolUrl)` at submit time and extracts two critical parameters from the pool server's response:

```
const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
if (!targetPuzzleHash) { throw ... }
if (relativeLockHeight === undefined) { throw ... }
``` [1](#0-0) 

The only guards are existence checks — there is no validation that `relativeLockHeight` falls within a safe range, and no format/ownership check on `targetPuzzleHash`. These values are then forwarded verbatim to the wallet RPC:

- `createNewPoolWallet({ initialTargetState, fee })` in `PlotNFTAdd.tsx` [2](#0-1) 

- `pwJoinPool({ walletId, poolUrl, relativeLockHeight, targetPuzzlehash, fee })` in `PlotNFTChangePool.tsx` [3](#0-2) 

The electron main process handler for `FETCH_POOL_INFO` performs no validation on the returned JSON fields either — it simply parses and returns the raw response:

```
ipcMainHandle(AppAPI.FETCH_POOL_INFO, async (poolUrl: string) => {
  const poolInfoUrl = `${poolUrl}/pool_info`;
  return fetchJSON(poolInfoUrl);
});
``` [4](#0-3) 

The `PoolInfo` display component does render `relativeLockHeight` and `targetPuzzleHash` to the user under a "Verify Pool Details" step, but with no contextual warning about dangerous values: [5](#0-4) 

The label "Relative Lock Height" gives no indication that a value of, say, `4294967295` means the user can never leave the pool. The Chia pool protocol specification caps this at 4608 blocks (~2 weeks), but the GUI enforces no such bound.

### Impact Explanation

`relative_lock_height` is an on-chain parameter embedded in the Plot NFT singleton at creation/join time. Once committed, the user must wait that many blocks before they can switch pools or self-pool. A value of `4294967295` (~83 years at 18.75s/block) is functionally permanent. This constitutes a **custody restriction** on pooled farming rewards: the user's XCH farming output is locked to a pool they cannot exit, matching the Critical/High impact class of "payout change or balance/accounting change affecting pooled farming rewards" and "bypass of custody/clawback restrictions with direct security impact."

### Likelihood Explanation

The attack requires only that a user be directed to a pool URL controlled by the attacker — via a phishing link, a community post, or a QR code. The pool server is a simple HTTP endpoint returning JSON. No cryptographic capability or host compromise is needed. The user sees a "Verify Pool Details" panel but has no basis to evaluate whether `relativeLockHeight: 4294967295` is dangerous.

### Recommendation

1. **Enforce a maximum `relativeLockHeight`** in `prepareSubmitData` before passing to the RPC. The Chia pool protocol maximum is 4608; reject or warn on any value exceeding it:
   ```ts
   const MAX_RELATIVE_LOCK_HEIGHT = 4608;
   if (relativeLockHeight > MAX_RELATIVE_LOCK_HEIGHT) {
     throw new Error(t`Pool relativeLockHeight ${relativeLockHeight} exceeds the maximum allowed value of ${MAX_RELATIVE_LOCK_HEIGHT}.`);
   }
   ```
2. **Display a human-readable lock duration** in the "Verify Pool Details" panel (e.g., "~2 weeks") so users can evaluate the commitment they are making.
3. **Validate `targetPuzzleHash` format** (64-character hex string) before use.

### Proof of Concept

1. Attacker stands up an HTTP server at `https://evil-pool.example.com` with a `/pool_info` endpoint returning:
   ```json
   {
     "name": "Legit Pool",
     "description": "Best pool ever",
     "pool_url": "https://evil-pool.example.com",
     "fee": "0.01",
     "logo_url": "",
     "minimum_difficulty": 1,
     "protocol_version": "1.0",
     "relative_lock_height": 4294967295,
     "target_puzzle_hash": "0xaaaa...attacker_controlled_hash"
   }
   ```
2. Victim enters `https://evil-pool.example.com` in the "Pool URL" field of the "Join a Pool" or "Change Pool" dialog.
3. `usePoolInfo` fetches and displays the pool info. The "Relative Lock Height" field shows `4294967295` with no warning.
4. Victim clicks "Create" / "Change". `prepareSubmitData` passes `relativeLockHeight: 4294967295` and the attacker's `targetPuzzleHash` to `createNewPoolWallet` / `pwJoinPool`.
5. The Plot NFT is committed on-chain with a lock height of `4294967295`. The victim can never switch pools or self-pool. All farming rewards flow through the attacker-controlled pool indefinitely.
