Audit Report

## Title
In-flight `SignWithThresholdContext` entries for migrated canisters are silently orphaned on subnet A' during an online subnet split, with no reject response delivered — (`rs/replicated_state/src/metadata_state.rs`)

## Summary

`SystemMetadata::online_split()` calls `reject_in_progress_management_calls_after_split()` to clean up subnet call contexts for canisters migrated to subnet B, but this function handles only `install_code`, `stop_canister`, and `raw_rand` contexts. `sign_with_threshold_contexts` (covering `sign_with_ecdsa`, `sign_with_schnorr`, and `vetkd_derive_key`) are entirely omitted. When a canister with an in-flight signing request is migrated to subnet B, the `SignWithThresholdContext` is orphaned on subnet A' with no reject response ever delivered to the canister. Chain-fusion minter canisters (ckBTC, ckETH) that issue `sign_with_ecdsa` calls for user withdrawals will have those withdrawals permanently stuck with no protocol-level recovery path.

## Finding Description

`SubnetCallContextManager` holds `sign_with_threshold_contexts: BTreeMap<CallbackId, SignWithThresholdContext>` alongside the other context maps. [1](#0-0) 

`SignWithThresholdContext` carries a `request: Request` field containing the originating canister's sender ID and reply callback — exactly the information needed to construct a reject response. [2](#0-1) 

During `online_split()` on subnet A', `reject_in_progress_management_calls_after_split()` is called to clean up contexts for migrated canisters. [3](#0-2) 

The function iterates over `install_code`, `stop_canister`, and `raw_rand` contexts, removing non-local ones and enqueuing reject responses. There is no equivalent `remove_non_local_sign_with_threshold_calls` call — `sign_with_threshold_contexts` is completely absent from this cleanup. [4](#0-3) 

On subnet B, the entire `subnet_call_context_manager` is reset to `Default::default()`. [5](#0-4) 

However, the `SignWithThresholdContext` was stored on subnet A' (where the canister originally lived), not on subnet B. So the context is not dropped on subnet B — it is orphaned on subnet A' with no reject response sent to the canister.

The change-guard function in `subnet_call_context_manager.rs` explicitly lists `sign_with_threshold_contexts` as a field that must be considered during subnet splitting, confirming the omission is unintentional. [6](#0-5) 

After the split, subnet A' will eventually produce a signing response for the orphaned context via `retrieve_context()`, but the originating canister no longer exists on subnet A'. The response is dropped. The canister on subnet B never receives a signing response (success or reject). [7](#0-6) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

Chain-fusion minter canisters (ckBTC, ckETH) record user withdrawals as "in-progress" when issuing `sign_with_ecdsa` calls. Any such in-flight call at split time results in a permanently stuck withdrawal with no protocol-level recovery. The user's funds are locked indefinitely. The bug is deterministic: every affected signing context produces a permanently stuck withdrawal. Depending on withdrawal traffic at split time, multiple users' funds can be affected simultaneously.

## Likelihood Explanation

Subnet splits are rare NNS governance operations, but the IC is explicitly designed to support them and they are a planned operational capability. The bug is triggered by a legitimate governance operation (not a malicious actor), and the harm is caused by the code bug, not by malicious governance intent. Chain-fusion minters process continuous withdrawal traffic, so any in-flight signing request at split time — a window spanning multiple consensus rounds — triggers the stuck state. No attacker action is required; the bug fires automatically for every affected context.

## Recommendation

In `reject_in_progress_management_calls_after_split()` in `rs/replicated_state/src/metadata_state.rs`, add handling for `sign_with_threshold_contexts` analogous to the existing `raw_rand` handling: add a `remove_non_local_sign_with_threshold_calls` method to `SubnetCallContextManager` that iterates over all contexts whose `request.sender()` is not a local canister, removes them, and returns them. Then in `reject_in_progress_management_calls_after_split()`, iterate over the returned contexts and call `reject_management_call_after_split` with `CanisterCall::Request(Arc::new(context.request))` and the migrated canister ID, so the canister receives a definitive `SysTransient` reject and can attempt recovery. [8](#0-7) 

## Proof of Concept

State-machine test outline:

1. Create a canister on subnet A with an active `sign_with_ecdsa` call — push a `SignWithThresholdContext` into `SubnetCallContextManager` with `request.sender()` set to the canister's ID.
2. Assign the canister to subnet B in the routing table.
3. Call `ReplicatedState::online_split(subnet_b_id, subnet_a_id)` (the subnet A' path).
4. Assert that `state_a.metadata.subnet_call_context_manager.sign_with_threshold_contexts` is empty (context was cleaned up with a reject).
5. Assert that `subnet_queues` contains an output reject response addressed to the canister.

Currently steps 4 and 5 both fail: the context remains on subnet A' and no reject is enqueued, confirming the orphan. The existing test infrastructure in `rs/replicated_state/src/metadata_state/tests.rs` already tests the analogous `raw_rand` path and can be extended directly. [9](#0-8)
