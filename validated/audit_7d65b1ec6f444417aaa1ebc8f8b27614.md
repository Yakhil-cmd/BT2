### Title
Deployment Authorization Bypass via Unrestricted Contract Update in `SetContract` - (File: `fvm/environment/contract_updater.go`)

### Summary
When contract deployment is restricted to a whitelist of authorized accounts, the `SetContract` function in the FVM skips the authorization check entirely for updates to already-existing contracts. Any account owner holding an existing contract — regardless of whether they are on the authorized deployment list — can replace that contract's code with arbitrary Cadence code, bypassing the deployment restriction.

### Finding Description

In `fvm/environment/contract_updater.go`, the `SetContract` function enforces the deployment authorization check only when the contract does not yet exist:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
if err != nil {
    return err
}

if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf(
        "deploying contract failed: %w",
        errors.NewOperationAuthorizationErrorf(
            "SetContract",
            "deploying contracts requires authorization from specific accounts"))
}

updater.draftUpdates[location] = ContractUpdate{
    Location: location,
    Code:     code,
}
```

The comment explicitly states "Contract updates are always allowed." When `exists == true`, `isAuthorizedForDeployment` is never called, and the draft update is unconditionally queued.

The deployment authorization check (`isAuthorizedForDeployment`) reads the authorized-accounts list from the service account's storage at `ContractDeploymentAuthorizedAddressesPath`. When `RestrictedDeploymentEnabled()` returns `true`, only accounts on that list may deploy new contracts. However, this restriction is entirely absent for the update path.

The analog to the external report's `isContract` bypass is direct: in the EVM case, the type-check returns "not a contract" during the constructor phase, allowing a contract to pass a whitelist gate. Here, the existence-check returns `true` for any already-deployed contract, causing the whitelist gate to be skipped entirely.

The attack entry point is a standard Cadence transaction signed by the account owner with `auth(UpdateContract) &Account` — a capability any account owner possesses over their own account. No privileged keys, no admin access, and no staked node role is required.

This is confirmed by the existing test at `fvm/fvm_blockcontext_test.go` lines 563–617, which explicitly asserts that a contract update succeeds without service-account authorization:

```go
t.Run("account update with update code succeeds if not signed by service account", func(t *testing.T) {
    ...
    txBodyBuilder = testutil.UpdateUnauthorizedCounterContractTransaction(accounts[0])...
    ...
    require.NoError(t, output.Err)
})
```

### Impact Explanation

An account that is **not** on the authorized deployment list can deploy arbitrary Cadence contract code on-chain by:

1. Obtaining an account that already has any contract deployed to it (e.g., via `templates.CreateAccount` with a benign contract, signed by the service account during bootstrapping).
2. Submitting a transaction with `auth(UpdateContract) &Account` to replace the contract body with arbitrary code.

The malicious contract is then live on-chain and callable by any other transaction or contract. Depending on the contract's content, this can lead to unauthorized state mutation, theft of assets from callers, or subversion of protocol-level invariants enforced by Cadence contracts.

### Likelihood Explanation

The precondition — owning an account with an existing contract — is easily satisfied. Account creation with an initial contract is a standard Flow operation (used in bootstrapping, staking setup, and SDK templates). Any account created this way can immediately exploit the update path. The attacker needs only their own account keys and the ability to submit a transaction, both of which are available to any unprivileged network participant.

### Recommendation

Apply the same `isAuthorizedForDeployment` check to contract updates as to new deployments when `RestrictedDeploymentEnabled()` is `true`. The guard in `SetContract` should be:

```go
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying/updating contract failed: %w", ...)
}
```

If unrestricted updates are intentional for account-owned contracts, the deployment restriction should be documented as a "new deployment only" control, and downstream security assumptions that rely on it as a full code-deployment gate must be revisited.

### Proof of Concept

1. Enable deployment restriction on the network (`isContractDeploymentRestricted = true` in service account storage).
2. Confirm that account `A` (not on the authorized list) cannot deploy a new contract — the FVM returns `"deploying contracts requires authorization from specific accounts"`.
3. Have the service account create a new account `A` with a benign placeholder contract (e.g., `access(all) contract Placeholder {}`). The service account's signature satisfies the deployment authorization for the initial deploy.
4. From account `A`, submit a transaction:
   ```cadence
   transaction {
     prepare(signer: auth(UpdateContract) &Account) {
       signer.contracts.update(name: "Placeholder", code: "<malicious_code>".decodeHex())
     }
   }
   ```
   signed only by account `A` (no service account signature).
5. The FVM executes `SetContract` → `ContractExists` returns `true` → the `isAuthorizedForDeployment` check is skipped → the malicious code is committed to chain.
