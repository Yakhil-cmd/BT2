### Title
Signature Verification Bypassed for Invoke Transactions via `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point call (which performs account-level signature verification) for any invoke transaction whose nonce is `1` and whose sender address appears in the mempool or a recent block. Because `account_tx_in_pool_or_recent_block` checks for **any** transaction from the address — not specifically a `deploy_account` — an unprivileged attacker can submit an invoke transaction with an arbitrary/invalid signature for a victim account that has a pending `deploy_account` in the mempool, and the gateway will admit it without ever verifying the signature.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke` transaction.
2. The transaction nonce equals `Nonce(Felt::ONE)`.
3. The on-chain account nonce equals `Nonce(Felt::ZERO)`.
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

**The overly broad membership check:** [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address has **any** transaction in the pool or state — it does not verify that the matching transaction is specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch is circular: future-nonce transactions can themselves have been admitted via this same skip path.

**Effect when skip is triggered:** [3](#0-2) 

`execution_flags.validate` is set to `false`, so `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling `validate_tx`: [4](#0-3) 

`validate_tx` is the only place the account's `__validate__` entry point is invoked: [5](#0-4) 

No other code in the gateway path verifies the cryptographic signature of the transaction.

**Fee-escalation amplification:** [6](#0-5) 

When `enable_fee_escalation` is `true`, an attacker who submits an invalid invoke with a higher tip and `max_l2_gas_price` than the victim's legitimate nonce-1 transaction will **replace** the victim's transaction in the mempool. The victim's transaction is permanently removed; the attacker's invalid transaction occupies the slot until the batcher rejects it at execution time.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions.**

An invoke transaction carrying an arbitrary or forged signature is admitted to the mempool without any account-level signature check. This breaks the invariant that every transaction in the mempool has been authorized by its sender. Concretely:

- The invalid transaction occupies the nonce-1 slot for the victim account.
- With fee escalation enabled, the attacker can permanently displace the victim's legitimate nonce-1 transaction.
- The batcher will eventually reject the invalid transaction (because `__validate__` is called during actual execution), but the victim's original transaction is already gone and must be resubmitted.
- An attacker can repeat this for every account that submits a `deploy_account` + `invoke` pair (a common UX pattern), causing systematic disruption of account onboarding.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The trigger condition (account nonce = 0, tx nonce = 1, address in mempool) is reliably met for every user who follows the recommended UX of submitting `deploy_account` + `invoke` together.
- The mempool is observable (transactions are propagated over P2P), so an attacker can monitor for new `deploy_account` transactions and immediately race with a crafted invoke.
- No privileged access is required; any network participant can submit transactions to the gateway.
- The only constraint is that the attacker must act before the `deploy_account` is committed to a block (after which the account nonce advances to 1 and the skip condition no longer applies).

---

### Recommendation

1. **Narrow the membership check**: Replace `account_tx_in_pool_or_recent_block` with a check that specifically verifies a `deploy_account` transaction for the sender address is present in the mempool. Add a dedicated `deploy_account_in_pool(address)` query to the mempool API.

2. **Alternatively, verify the signature even when skipping the entry point**: The `__validate__` skip is a UX optimization; the cryptographic signature can still be verified off-chain (e.g., by recovering the public key from the class hash and checking the ECDSA/Stark signature) before admitting the transaction.

3. **Add a test**: Add a test case that submits an invoke with an invalid signature when a `deploy_account` is in the mempool and asserts that the gateway rejects it.

### Proof of Concept

```
1. Alice submits deploy_account(class_hash=C, constructor_calldata=..., nonce=0, valid_sig)
   → Gateway fully executes __validate_deploy__, admits to mempool.
   → mempool.account_tx_in_pool_or_recent_block(Alice) == true

2. Eve observes Alice's deploy_account in the mempool (P2P propagation).

3. Eve submits invoke(sender=Alice, nonce=1, calldata=<anything>, sig=[0x0, 0x0],
                      tip=Alice_tip * 2, max_l2_gas_price=Alice_price * 2)

4. Gateway stateless validation: passes (no signature check).

5. Gateway stateful validation:
   a. get_nonce_from_state(Alice) → Nonce(0)   [account not yet deployed]
   b. validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap → OK
   c. validate_by_mempool: nonce=1 not yet in pool → OK
   d. skip_stateful_validations:
        tx.nonce() == Nonce(1) ✓
        account_nonce == Nonce(0) ✓
        account_tx_in_pool_or_recent_block(Alice) == true ✓  ← Alice's deploy_account
      → returns true (skip __validate__)
   e. run_validate_entry_point(skip_validate=true):
        execution_flags.validate = false
        StatefulValidator::perform_validations returns Ok(()) immediately
        __validate__ is NEVER called; Eve's forged signature is never checked.

6. Eve's invalid invoke is admitted to the mempool.
   If fee escalation is enabled and Eve's tip > Alice's tip:
     → Alice's legitimate nonce-1 invoke is replaced and permanently removed.

7. Batcher later executes the block:
   - deploy_account executes → Alice's account is deployed.
   - Eve's invoke executes → __validate__ is called → signature [0x0, 0x0] fails → tx rejected.
   - Alice's nonce-1 tx is gone; Alice must resubmit.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
