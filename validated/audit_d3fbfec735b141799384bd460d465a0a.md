### Title
WalletConnect Offer Confirmation Understates "Total Amount with Royalties" via Incorrect Split-Royalty Accounting — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount equally among all NFTs before computing each NFT's royalty contribution. When a WalletConnect-connected dApp sends a `take_offer` or `create_offer_for_ids` command involving multiple NFTs with royalties, the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog is systematically understated. The user approves believing they will spend less than the offer actually requires, causing them to unknowingly authorize a higher spend than displayed.

### Finding Description

`parseCommandDisplay` is the function that produces the wallet-delta summary shown in the WalletConnect `Confirm.tsx` dialog before the user clicks "Approve". For offers that include NFTs with royalties, it calls `formatAmountWithRoyalties` to compute the displayed total:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length); // ← divides amount by NFT count
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The Chia protocol computes each NFT's royalty on the **full** fungible amount. The code instead divides the amount by the number of NFTs (`splitAmount = amount / n`) and applies each royalty to that reduced share. For `n` NFTs the displayed royalty total is `1/n` of the correct value.

**Concrete arithmetic (from the existing test):**
- Offer: 2 NFTs (royalties 500 = 5%, 10 = 0.1%) for 100,000,000 mojos (0.0001 XCH)
- **Code displays:** `splitAmount = 50,000,000`; royalties = 2,500,000 + 50,000 = 2,550,000 → total **0.00010255 XCH**
- **Protocol actually charges:** royalties = 5,000,000 + 100,000 = 5,100,000 → total **0.0001051 XCH**
- **Understatement:** 2,550,000 mojos (≈ 50% of actual royalties hidden from user) [2](#0-1) 

The incorrect `amountWithRoyalties` value is then rendered in `Confirm.tsx` as the authoritative "Total Amount with Royalties" label the user reads before clicking Approve:

```tsx
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties} ...
  </div>
)}
``` [3](#0-2) 

The same path applies to `create_offer_for_ids` commands via `withRoyaltyTotals` → `formatAmountWithRoyalties`. [4](#0-3) 

### Impact Explanation

A user who has paired a WalletConnect dApp sees the confirmation dialog showing "Total Amount with Royalties: X XCH". Because `X` is understated (by up to ~50% of actual royalties for a 2-NFT offer, more for larger bundles), the user approves a transaction that actually spends more XCH than they consented to. The excess goes to NFT royalty recipients — a direct, irreversible balance reduction beyond what the user authorized based on the displayed figure. This matches **High: WalletConnect state that causes a user to approve the wrong amount**.

### Likelihood Explanation

Any WalletConnect-paired dApp can craft a `take_offer` payload containing a multi-NFT offer with royalties. No special privilege beyond an established WalletConnect session is required. WalletConnect pairing is a routine user action (scanning a QR code or clicking a deep link on a legitimate-looking site). The understatement scales with the number of NFTs and their royalty percentages, making it easy to engineer a meaningful discrepancy.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be calculated against the full fungible `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectation for the multi-NFT case from `'0.00010255'` to `'0.0001051'`.

### Proof of Concept

1. Pair a WalletConnect dApp with the Chia GUI wallet.
2. From the dApp, send:
   ```json
   {
     "command": "chia_wallet.take_offer",
     "params": { "offer": "<offer_with_2_NFTs_5pct_and_0.1pct_royalties_for_0.0001_XCH>" }
   }
   ```
3. The confirmation dialog displays **"Total Amount with Royalties: 0.00010255 XCH"**.
4. User clicks Approve.
5. The wallet backend executes the offer; the actual spend is **0.0001051 XCH** — 2,550,000 mojos more than the user was shown.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
