### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State (Level 1) Instead of Restoring Pre-Pause Activity — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` hardcodes the target pause level to `1` (admin-paused) regardless of whether the pool was active (`0`) or admin-paused (`1`) before the protocol pause. When the protocol pauses an active pool and later unpauses it, the pool is left permanently stuck in the admin-paused state, blocking all swaps until the pool admin separately calls `unpausePool`.

---

### Finding Description

The factory defines three pause levels:

| Level | Meaning |
|---|---|
| `0` | Active |
| `1` | Paused by pool admin |
| `2` | Paused by protocol |

`protocolPausePool` accepts both level `0` and level `1` as valid pre-conditions, meaning it can override an active pool: [1](#0-0) 

`protocolUnpausePool` then unconditionally transitions to level `1`, never to `0`: [2](#0-1) 

The pool's `_checkNotPaused` guard blocks swaps for **any** non-zero pause level: [3](#0-2) 

The `swap` function carries this guard: [4](#0-3) 

The pool admin's `unpausePool` only accepts level `1` as input and only transitions to `0`: [5](#0-4) 

So after `protocolUnpausePool` sets the pool to level `1`, the pool admin must separately call `unpausePool` to reach level `0`. If the pool was never admin-paused, this is an unexpected extra step that the protocol owner's "unpause" action silently creates.

---

### Impact Explanation

After the protocol unpauses a pool that was previously active (level `0`), the pool remains at level `1` (admin-paused). All `swap` calls revert with `PoolPaused`. The pool is completely non-functional for trading until the pool admin explicitly calls `unpausePool`. If the pool admin is a multisig or DAO with governance delay, the pool can remain broken for an extended period. This constitutes broken core pool functionality (unusable swap flow) with no direct path for the protocol owner to fully restore it.

---

### Likelihood Explanation

The scenario is straightforward: the protocol owner uses `protocolPausePool` during an emergency on an active pool (the common case), then calls `protocolUnpausePool` believing the pool is restored. The pool remains paused. This is a normal operational sequence with no special preconditions required.

---

### Recommendation

`protocolUnpausePool` should restore the pool to level `0` (fully active), not level `1`. The protocol pause is a temporary override; removing it should return the pool to its pre-pause state. Change:

```solidity
// Before
IMetricOmmPoolFactoryActions(pool).setPause(1);

// After
IMetricOmmPoolFactoryActions(pool).setPause(0);
```

If the intent is to require pool admin sign-off before resuming, that design decision must be documented and the pool admin must be notified out-of-band, which is an operational risk. The cleaner fix is to store the pre-pause level and restore it on unpause.

---

### Proof of Concept

```
Initial state: pauseLevel == 0 (active)

1. Protocol calls protocolPausePool(pool)
   → guard: cur (0) != 0 && cur (0) != 1 → false, passes
   → setPause(2) → pauseLevel == 2

2. Protocol calls protocolUnpausePool(pool)
   → guard: cur (2) != 2 → false, passes
   → setPause(1) → pauseLevel == 1   ← BUG: should be 0

3. User calls swap(...)
   → _checkNotPaused(): pauseLevel (1) != 0 → revert PoolPaused()

4. Pool admin must now call unpausePool(pool)
   → guard: cur (1) != 1 → false, passes
   → setPause(0) → pauseLevel == 0   ← only now is the pool active again
```

The protocol owner's `protocolUnpausePool` call does not restore the pool to a usable state when the pool was active before the protocol pause. Swaps remain blocked until the pool admin takes a separate, unannounced action.
