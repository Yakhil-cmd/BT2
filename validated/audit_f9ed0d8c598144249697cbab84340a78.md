### Title
Unchecked `approve()` Return Value in `transfer_to_staking_contract` Breaks BTC-Token Delegation — (File: `src/pool/pool.cairo`)

### Summary

The `Pool.transfer_to_staking_contract` internal function calls `token_dispatcher.approve(...)` using the raw `IERC20Dispatcher` without checking the boolean return value. For non-standard BTC tokens that return `false` instead of reverting on a failed approval, the pool contract silently proceeds to call `add_stake_from_pool`, which then reverts inside the staking contract with a misleading "insufficient allowance" error. This permanently breaks delegation for any such BTC token pool, preventing all delegators from entering or adding to the pool.

### Finding Description

In `src/pool/pool.cairo`, the internal function `transfer_to_staking_contract` (lines 668–682) is called by both `enter_delegation_pool` and `add_to_delegation_pool` whenever a delegator deposits tokens:

```cairo
fn transfer_to_staking_contract(
    self: @ContractState,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
    staker_address: ContractAddress,
) {
    let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
    token_dispatcher
        .approve(spender: staking_pool_dispatcher.contract_address, amount: amount.into()); // ❌ return value ignored

    staking_pool_dispatcher.add_stake_from_pool(:staker_address, :amount);
}
``` [1](#0-0) 

The call uses the raw `IERC20Dispatcher` (not `CheckedIERC20DispatcherTrait`), so a `false` return from `approve()` is silently discarded. The contract already imports `CheckedIERC20DispatcherTrait` at line 47 and uses it correctly for all other token operations (`checked_transfer_from` in `transfer_from_delegator`, `checked_transfer` in `exit_delegation_pool_action` and `claim_rewards`), making this omission inconsistent. [2](#0-1) [3](#0-2) 

In `add_stake_from_pool` (staking.cairo lines 1003–1046), the staking contract calls `checked_transfer_from` after updating state:

```cairo
token_dispatcher
    .checked_transfer_from(
        sender: pool_contract, recipient: get_contract_address(), amount: amount.into(),
    );
``` [4](#0-3) 

Because `checked_transfer_from` reverts on failure, the entire transaction reverts — but with a misleading "insufficient allowance" error originating from the staking contract rather than from the approve step in the pool. For any BTC token whose `approve()` returns `false` instead of reverting, every call to `enter_delegation_pool` and `add_to_delegation_pool` will always fail, making the pool permanently unusable for that token.

### Impact Explanation

Delegation to any BTC-token pool whose underlying token returns `false` on a failed `approve()` is permanently broken. No delegator can ever enter or add to such a pool. The pool contract is deployed and initialized, the staker has opened it for delegation, but no funds can flow through it. This constitutes griefing with damage to users (delegators cannot participate) and to the protocol (the pool is a dead contract). The error surface is also misleading: the revert originates inside the staking contract's `checked_transfer_from`, not at the approve step, making diagnosis difficult.

This maps to the **Medium** allowed impact: *Griefing with no profit motive but damage to users or protocol*.

### Likelihood Explanation

The protocol explicitly supports non-STRK BTC tokens via `add_token` / `enable_token` (staking.cairo lines 1344–1389). The token admin can add any ERC20-compatible token with 5–18 decimals. Wrapped BTC tokens on Starknet may follow non-standard ERC20 patterns. Any such token that returns `false` on approve (rather than reverting) will trigger this path. The entry point (`enter_delegation_pool`, `add_to_delegation_pool`) is fully unprivileged and reachable by any delegator. [5](#0-4) 

### Recommendation

Replace the unchecked `approve` call with the `checked_approve` method from `CheckedIERC20DispatcherTrait`, which is already imported and used elsewhere in the same file:

```cairo
fn transfer_to_staking_contract(
    self: @ContractState,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
    staker_address: ContractAddress,
) {
    let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
    // Use checked_approve instead of approve
    token_dispatcher
        .checked_approve(
            spender: staking_pool_dispatcher.contract_address, amount: amount.into()
        );
    staking_pool_dispatcher.add_stake_from_pool(:staker_address, :amount);
}
```

This ensures that a `false` return from `approve()` causes an immediate, meaningful revert in the pool contract rather than propagating silently to the staking contract.

### Proof of Concept

1. Token admin adds and enables a BTC token whose `approve()` returns `false` on failure (non-standard behavior).
2. A staker calls `set_open_for_delegation(token_address)` — a BTC-token pool is deployed.
3. A delegator calls `enter_delegation_pool(reward_address, amount)`:
   - `transfer_from_delegator` succeeds: delegator's tokens move to the pool contract.
   - `transfer_to_staking_contract` calls `token_dispatcher.approve(staking_contract, amount)` → returns `false`, **not detected**.
   - `add_stake_from_pool` is called; inside, `checked_transfer_from(pool_contract → staking_contract)` fails because allowance is still 0.
   - The entire transaction reverts (including the `transfer_from_delegator` step), so no funds are permanently locked.
4. Every subsequent attempt by any delegator to enter or add to this pool will fail identically.
5. The pool is permanently unusable for this token, with no recovery path short of a contract upgrade. [6](#0-5) [7](#0-6)
