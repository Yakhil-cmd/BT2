Audit Report

## Title
XNet Peer Selection Permanently Biased via RTT EMA Manipulation in ProximityMap - (File: rs/xnet/payload_builder/src/proximity.rs)

## Summary
`XNetClientImpl::query()` records the RTT observation immediately after receiving HTTP response headers, before reading the response body. A malicious registered node can send headers instantly (~1 µs) while stalling or dropping the body, driving its EMA to the minimum clamp (1,000 ns) and its selection weight to the maximum (1,000,000,000). This creates up to a 100,000× selection bias over honest nodes, allowing the malicious node to monopolize XNet pulls and stall cross-subnet message delivery indefinitely.

## Finding Description
In `rs/xnet/payload_builder/src/lib.rs`, `XNetClientImpl::query()` calls `self.http_client.get(endpoint.url.clone()).await` which resolves when HTTP response headers are received. `observe_roundtrip_time` is called immediately after at lines 1831–1834, before the body is read at lines 1838–1843: [1](#0-0) 

In `rs/xnet/payload_builder/src/proximity.rs`, the observed duration is clamped to `[1_000, 1_000_000_000]` ns: [2](#0-1) 

The EMA is updated as `(*ema * 9 + duration_nanos) / 10`, and the first observation sets the EMA directly via `or_insert_with`: [3](#0-2) 

The weight formula `1_000 * NANOS_PER_SEC / ema_nanos` maps a 1,000 ns EMA to weight 1,000,000,000 and a 100 ms EMA to weight 10,000 — a 100,000× ratio: [4](#0-3) 

`pick_node()` is called on every XNet pull attempt: [5](#0-4) 

There are no existing guards: no penalty RTT on body failure or timeout, no minimum weight floor, and no minimum selection probability. The existing test `pick_node_extreme_roundtrip_times` explicitly validates the 1,000,000× weight disparity between a 1 µs and 1 s RTT node: [6](#0-5) 

## Impact Explanation
Once the EMA is poisoned, the malicious node is selected for nearly all XNet pulls from the victim subnet. It can then serve `204 No Content` on every pull, leaving the victim subnet's XNet slice pool for the remote subnet permanently empty. No XNet messages from the remote subnet are included in the victim subnet's blocks, stalling cross-subnet message delivery indefinitely. This is a concrete, persistent, targeted subnet availability impact — matching the High ($2,000–$10,000) category: *"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation
The attacker requires control of one registered node on the target remote subnet. Node operators are permissioned via NNS governance, but the IC has hundreds of node operators across many subnets. A single compromised or malicious node operator is a realistic threat model explicitly within the HackenProof scope ("protocol peer below the consensus fault threshold," "xnet origin"). No threshold corruption, key compromise, DNS hijack, or user interaction is required. The attack is fully automated and self-reinforcing: once the EMA is at the minimum clamp, a single 1 µs header-only response keeps it there indefinitely.

## Recommendation
1. **Move `observe_roundtrip_time` after successful body read** — only record RTT when a complete, valid response body is received (after line 1843).
2. **Record a penalty RTT on body failure or timeout** — treat `BodyReadError` and `Timeout` as a high-RTT observation (e.g., 1 s / `NANOS_PER_SEC`) to penalize nodes that respond with headers but fail to deliver content.
3. **Cap the maximum weight ratio** — enforce a floor on node weights (e.g., no node may have weight less than `max_weight / 1000`) to bound the maximum selection bias regardless of RTT spread.
4. **Apply a minimum selection probability** — guarantee every registered node has at least a small baseline probability of being selected, preventing complete monopolization.

## Proof of Concept
**Setup**: Attacker controls node `N` (operator `OP_A`) on subnet B. Subnet A's XNet payload builder pulls streams from subnet B.

**Step 1 — Poison the EMA**: Node `N` handles XNet HTTP requests by sending a `200 OK` response with headers in ~1 µs, then immediately closing the connection without a body. `XNetClientImpl::query()` records RTT ≈ 1 µs (clamped to 1,000 ns) at lines 1831–1834, then fails with `BodyReadError` at line 1843. The EMA for `OP_A` is set to 1,000 ns → weight = 1,000,000,000.

**Step 2 — Monopolize selection**: With one malicious node (weight 1,000,000,000) and two honest nodes at 100 ms RTT (weight 10,000 each), `pick_node()` selects the malicious node with probability ≈ 1,000,000,000 / 1,020,000 ≈ 98%.

**Step 3 — Stall XNet**: Node `N` serves `204 No Content` on every pull. Subnet A's slice pool for subnet B remains empty. Cross-subnet message delivery stalls indefinitely.

**Verification**: Run the existing test `pick_node_extreme_roundtrip_times` in `rs/xnet/payload_builder/src/proximity/tests.rs` — it already asserts the 1,000,000× weight disparity. A new integration test can extend this by simulating a `BodyReadError` path and confirming the EMA is updated before the error is returned.
