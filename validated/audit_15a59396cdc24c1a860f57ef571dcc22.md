### Title
Unverified Participant Contributions in CKD Coordinator Allow Malicious Participant to Corrupt Confidential Key Derivation Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` aggregates participant-supplied `(big_y, big_c)` shares by simple addition with no proof of correctness. A single malicious participant can substitute arbitrary curve points, causing the coordinator to assemble a structurally valid but cryptographically wrong `CKDOutput`. Every honest requester who calls `unmask` on that output receives an incorrect confidential key. This is the direct analog of the Lybra finding: both protocols consume an unprotected, caller-supplied value in a security-critical aggregation step instead of a committed or verified one.

### Finding Description

`do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs` collects one `CKDOutput` per participant and accumulates the two G1 elements unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

The honest computation that each participant is supposed to perform is defined in `compute_signature_share`:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

so that the aggregate satisfies `C − app_sk · Y = msk · H(pk ‖ app_id)`.

No zero-knowledge proof, commitment, or consistency check is attached to the message sent by `do_ckd_participant`. The coordinator has no way to distinguish a correctly formed share from an arbitrary pair of G1 points. A malicious participant `j` can replace its honest `(norm_big_y_j, norm_big_c_j)` with any `(Y_j', C_j')` of its choice. The resulting aggregate becomes:

```
Y' = Y_honest − norm_big_y_j + Y_j'
C' = C_honest − norm_big_c_j + C_j'
```

and `C' − app_sk · Y'` equals `msk · H(pk ‖ app_id) + (C_j' − norm_big_c_j) − app_sk · (Y_j' − norm_big_y_j)`, which is an arbitrary G1 point under the attacker's control.

### Impact Explanation

The coordinator returns a `CKDOutput` that is structurally well-formed (two non-identity G1 points) but encodes a wrong secret. Every honest party that calls `unmask(app_sk)` on this output obtains a G1 point that is not `msk · H(pk ‖ app_id)`. The derived confidential key is therefore unusable or, worse, silently wrong. This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

Any single participant in a CKD session is a reachable, unprivileged attacker. The `ckd` public API accepts any `Participant` value; no additional privilege is required. The malicious participant simply deviates from `compute_signature_share` and sends hand-crafted bytes. The coordinator has no detection mechanism, so the attack succeeds with probability 1 whenever one participant is malicious.

### Recommendation

Each participant must attach a non-interactive zero-knowledge proof of correct formation alongside its `(norm_big_y, norm_big_c)` share. Concretely, the proof must demonstrate knowledge of scalars `(x_i, y_i)` satisfying:

```
norm_big_y_i / λ_i = y_i · G₁
norm_big_c_i / λ_i = x_i · H(pk ‖ app_id) + y_i · app_pk
x_i · G₂ = vk_share_i   (links x_i to the committed verification key share)
```

This is a standard three-statement sigma protocol (Schnorr-style). The coordinator must verify all proofs before accumulating any share, rejecting the session if any proof fails. This is the direct analog of replacing `balanceOf(pair)` with `getReserves()`: it replaces a freely manipulable input with one that is cryptographically committed.

### Proof of Concept

1. Run a 3-of-3 CKD session. Participants are `P1`, `P2`, `P3`; `P3` is the coordinator.
2. `P1` (malicious) computes its honest `(norm_big_y_1, norm_big_c_1)` but instead sends `(norm_big_y_1 + δ·G₁, norm_big_c_1 + Δ)` for arbitrary attacker-chosen `δ` and `Δ ≠ δ·app_pk`.
3. `P3` (coordinator) receives the tampered share, adds it to the running sum without verification, and returns `CKDOutput { big_y: Y', big_c: C' }`.
4. The requester calls `ckd_output.unmask(app_sk)` and obtains `C' − app_sk·Y' = msk·H(pk‖app_id) + (Δ − δ·app_pk)`, which is not `msk·H(pk‖app_id)`.
5. The derived confidential key is wrong; any downstream use (e.g., decryption, re-encryption) silently fails or produces attacker-influenced output.

The root cause is at: [1](#0-0) 

where contributions are accumulated, and at: [2](#0-1) 

where `do_ckd_participant` sends its share with no attached proof, mirroring the unprotected `balanceOf` read in the Lybra report.
