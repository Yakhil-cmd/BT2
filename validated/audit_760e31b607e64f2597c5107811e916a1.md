### Title
Missing Minimum-Threshold Validation in FROST `assert_sign_inputs` and `presign` Allows Threshold-1 Signing to Corrupt Protocol Output — (`src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` validate that the threshold does not exceed the participant count, but omit the symmetric lower-bound check (`threshold < 2`). This check is consistently enforced in every other entry-point in the library — `assert_key_invariants` (DKG/reshare) and `validate_triple_inputs` (triple generation) — but is silently absent from the FROST signing path. A caller who passes `threshold = 1` to the FROST signing or presigning entry-points will not receive an error; the protocol will proceed with a single participant, producing a cryptographically invalid, unusable signature share aggregation.

---

**Missing in FROST presign (`src/frost/mod.rs`, `presign`):**

// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check

When `threshold = 1` is accepted, the protocol proceeds with a single participant. The Lagrange coefficient for a lone participant in a one-element list evaluates to 1, so the participant's raw signing share is used directly as the interpolated secret — which is not the actual secret key (the constant term of the shared polynomial). The aggregated signature is therefore cryptographically invalid and will fail external verification.

---

### Impact Explanation

A caller who passes `threshold = 1` to `assert_sign_inputs` or `presign` receives no error. The FROST signing protocol runs to completion, the coordinator aggregates the single share, and the output is an unusable signature. Honest parties who rely on the library's initialization checks to guarantee a valid output will silently receive a corrupted, unverifiable signature. This matches the allowed High impact: **corruption of sign outputs so honest parties accept unusable cryptographic outputs**.

---

### Likelihood Explanation

The `threshold` parameter is caller-supplied and is a plain `usize`-backed newtype with no compile-time lower bound. Any integrator who misconfigures the signing call (e.g., passes the number of *malicious* parties instead of the reconstruction lower bound, or simply passes `1` for a "single-signer fast path") will silently produce broken signatures. The asymmetry with keygen — where the same mistake is caught immediately — makes this a realistic misconfiguration path.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
``` [5](#0-4) 

---

### Proof of Concept

1. Run keygen with 3 participants and `threshold = 2` (accepted correctly).
2. Call `assert_sign_inputs` (or `presign`) with the same 3 participants but `threshold = 1`.
3. Observe: no `InitializationError` is returned; the protocol initialises successfully.
4. Run the signing protocol with only 1 participant present (satisfies the unchecked `threshold = 1` guard).
5. The coordinator aggregates the single share; the resulting signature fails standard Ed25519/RedJubjub verification against the public key produced in step 1.

Compare: calling `keygen` with `threshold = 1` on the same participant set immediately returns `InitializationError::ThresholdTooSmall { threshold: 1, min: 2 }`. [6](#0-5) [7](#0-6)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
