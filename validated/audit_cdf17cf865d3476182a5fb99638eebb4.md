### Title
SPoCK Proof Not Verified by Consensus Node During Sealing — (`engine/consensus/approvals/approval_collector.go`)

### Summary

The Flow consensus node's sealing path accepts `ResultApproval` messages from Verification Nodes and aggregates their attestation signatures to produce a block seal. While the attestation signature and the outer verifier signature are cryptographically checked, the **SPoCK (Specialized Proof of Confidential Knowledge) proof** embedded in every `ResultApprovalBody.Spock` is **never verified** by the consensus node before the seal is generated. This is an explicit, acknowledged `TODO` in the production sealing code. An unprivileged, staked Verification Node can submit a `ResultApproval` carrying a valid attestation signature but a completely fabricated or zeroed SPoCK proof, and the consensus node will count it toward the sealing threshold and produce a valid seal.

### Finding Description

The `ResultApprovalBody` struct carries a `Spock` field that is supposed to be a Specialized Proof of Confidential Knowledge — a BLS-based proof that the Verification Node actually re-executed the chunk and witnessed the same register touches as the Execution Node. [1](#0-0) 

The `VerifyingAssignmentCollector.validateApproval` function verifies:
1. The attestation signature (`verifyAttestationSignature`)
2. The outer verifier signature (`verifySignature`) [2](#0-1) 

Neither of these checks touches the `Spock` field. The SPoCK proof is accepted as long as it is non-empty (enforced only at construction time). [3](#0-2) 

When `ApprovalCollector.SealResult()` is called to generate the candidate seal, there is an explicit `// TODO: Check SPoCK proofs` comment with no implementation: [4](#0-3) 

The seal is generated and placed in the `IncorporatedResultSeals` mempool without any SPoCK verification. The `SealValidator.validateSeal` called by consensus nodes when validating a candidate block also only verifies the aggregated attestation signatures, not SPoCK proofs: [5](#0-4) 

The design documentation explicitly acknowledges that SPoCK verification is the dominant computational cost and is supposed to be performed, but it is not: [6](#0-5) 

### Impact Explanation

SPoCK is the cryptographic mechanism that proves a Verification Node actually re-executed the chunk. Without SPoCK verification, a staked Verification Node can:

1. Submit a `ResultApproval` with a valid attestation signature (over `{BlockID, ExecutionResultID, ChunkIndex}`) but an **arbitrary or fabricated SPoCK proof**.
2. The consensus node counts this approval toward the sealing threshold.
3. If enough such approvals are collected (or a single malicious VN is assigned to enough chunks), a seal is produced for an execution result that was never actually re-executed by the approving VN.

This breaks the core security invariant of Flow's verification system: that sealing a block requires cryptographic proof that assigned Verification Nodes independently re-executed the chunks. A malicious staked Verification Node can attest to execution results it never verified, undermining the fraud-detection guarantee that the entire verification layer is designed to provide. In the worst case, a colluding Execution Node and sufficient Verification Nodes could seal a fraudulent execution result (e.g., one that mints tokens or redirects transfers) without any honest re-execution.

### Likelihood Explanation

The attacker must be a staked Verification Node — a role reachable without privileged access beyond the standard staking requirement. The attack path is direct: craft a `ResultApproval` with a valid attestation signature (trivially computed from public data: `{BlockID, ExecutionResultID, ChunkIndex}`) and any non-empty byte sequence as the `Spock` field. The `validateApproval` function will accept it. No brute force, no key compromise, and no quorum control is required for the individual approval submission. The number of malicious VNs needed to reach the sealing threshold depends on the `RequireApprovalsForSealConstruction` parameter, but the missing check is a necessary step regardless of threshold.

### Recommendation

Implement SPoCK verification in `ApprovalCollector.SealResult()` or in `VerifyingAssignmentCollector.validateApproval()`. The SPoCK proof in `approval.Body.Spock` must be verified against the corresponding SPoCK secret from the Execution Receipt using `crypto.SPOCKVerifyAgainstData` with the verifier's staking public key. The Execution Receipt's `Spocks[chunkIndex]` field provides the execution node's SPoCK proof, and the two must be cross-verified. This is already demonstrated in test helpers: [7](#0-6) 

### Proof of Concept

1. Obtain a staked Verification Node identity (standard protocol participation).
2. Observe an `ExecutionResult` for a block and compute the attestation for an assigned chunk: `Attestation{BlockID, ExecutionResultID, ChunkIndex}`.
3. Sign the attestation ID with the VN's staking key to produce a valid `AttestationSignature`.
4. Set `Spock` to any non-empty byte slice (e.g., `[]byte{0x01}`).
5. Sign the full `ResultApprovalBody.ID()` to produce a valid `VerifierSignature`.
6. Submit the `ResultApproval` to consensus nodes via the `PushApprovals` channel.
7. `VerifyingAssignmentCollector.validateApproval` passes all checks (identity authorized, attestation sig valid, verifier sig valid).
8. The approval is counted toward the sealing threshold. The fabricated SPoCK proof is never checked.
9. Once the threshold is met, `ApprovalCollector.SealResult()` generates a seal — the `// TODO: Check SPoCK proofs` block is skipped entirely. [8](#0-7)

func (ac *VerifyingAssignmentCollector) validateApproval(approval *flow.ResultApproval) error {
	// check that approval is for the expected result to reject incompatible inputs
	if approval.Body.ExecutionResultID != ac.ResultID() {
		return fmt.Errorf("AssignmentCollector processes only approvals for result (%x) but got one for (%x)", ac.ResultID(), approval.Body.ExecutionResultID)
	}

	// approval has to refer same block as execution result
	if approval.Body.BlockID != ac.BlockID() {
		return engine.NewInvalidInputErrorf("result approval for invalid block, expected (%x) vs (%x)",
			ac.BlockID(), approval.Body.BlockID)
	}

	chunkIndex := approval.Body.ChunkIndex
	if chunkIndex >= uint64(ac.result.Chunks.Len()) {
		return engine.NewInvalidInputErrorf("chunk index out of range: %v", chunkIndex)
	}

	identity, found := ac.authorizedApprovers[approval.Body.ApproverID]
	if !found {
		return engine.NewInvalidInputErrorf("approval not from authorized verifier")
	}

	err := ac.verifyAttestationSignature(&approval.Body, identity)
	if err != nil {
		return fmt.Errorf("validating attestation signature failed: %w", err)
	}

	err = ac.verifySignature(approval, identity)
	if err != nil {
		return fmt.Errorf("validating approval signature failed: %w", err)
	}

	return nil
}

func (s *sealValidator) verifySealSignature(aggregatedSignatures *flow.AggregatedSignature,
	chunk *flow.Chunk, executionResultID flow.Identifier) error {
	// TODO: replace implementation once proper aggregation is used for Verifiers' attestation signatures.

	atst, err := flow.NewAttestation(flow.UntrustedAttestation{
		BlockID:           chunk.BlockID,
		ExecutionResultID: executionResultID,
		ChunkIndex:        chunk.Index,
	})
	if err != nil {
		return fmt.Errorf("could not build attestation: %w", err)
	}

	atstID := atst.ID()

	for i, signature := range aggregatedSignatures.VerifierSignatures {
		signerId := aggregatedSignatures.SignerIDs[i]

		nodeIdentity, err := identityForNode(s.state, chunk.BlockID, signerId)
		if err != nil {
			return err
		}

		valid, err := nodeIdentity.StakingPubKey.Verify(signature, atstID[:], s.signatureHasher)
		if err != nil {
			return fmt.Errorf("failed to verify signature: %w", err)
		}

		if !valid {
			return engine.NewInvalidInputErrorf("Invalid signature for (%x)", nodeIdentity.NodeID)
		}
	}

	return nil
}
