### Title
Hardcoded `EXIT_TO_NEAR_CALLBACK_GAS` (10 TGas) Is Insufficient to Execute `refund_on_error` EVM Call, Causing Permanent Loss of Burned ERC-20 Tokens - (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile schedules a NEAR callback (`exit_to_near_precompile_callback`) with a hardcoded gas budget of `EXIT_TO_NEAR_CALLBACK_GAS = 10_000_000_000_000` (10 TGas). When the upstream `ft_transfer` promise fails, this callback must execute `engine::refund_on_error`, which re-runs the full Aurora EVM to re-mint burned ERC-20 tokens. The 10 TGas budget is structurally insufficient to cover the Aurora engine overhead plus EVM execution, causing the callback to run out of NEAR gas, the refund to silently fail, and the user's ERC-20 tokens to be permanently destroyed.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` precompile constructs a two-step NEAR promise: a base `ft_transfer` (or `near_withdraw`) call, followed by a callback to `exit_to_near_precompile_callback` on the Aurora engine contract itself. [1](#0-0) 

The callback gas is unconditionally set to `EXIT_TO_NEAR_CALLBACK_GAS`: [2](#0-1) 

Inside `exit_to_near_precompile_callback`, when the base promise fails and the `error_refund` feature is compiled in, the code calls `engine::refund_on_error`: [3](#0-2) 

`refund_on_error` instantiates a full `Engine` and calls `engine.call()` with `u64::MAX` EVM gas to either re-mint burned ERC-20 tokens or transfer ETH back from the precompile address: [4](#0-3) 

Running the Aurora EVM inside a NEAR callback requires:
- Loading engine state from NEAR storage
- Executing the EVM (including ERC-20 `mint` logic)
- Writing modified state back to NEAR storage

The Aurora engine's own overhead per EVM call is well above 10 TGas for any non-trivial ERC-20 contract. The constant itself carries a `TODO` comment acknowledging the value was only "determined experimentally based on tests" — tests that exercise the happy path, not the refund path under load. [5](#0-4) 

By contrast, the `return_promise` helper used for other connector operations dynamically computes remaining gas: [6](#0-5) 

No such dynamic calculation is applied to `EXIT_TO_NEAR_CALLBACK_GAS`.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The ERC-20 burn happens inside the EVM *before* the `ft_transfer` promise is dispatched. If `ft_transfer` fails (e.g., the NEAR recipient account is not registered with the NEP-141 contract) and the refund callback subsequently runs out of NEAR gas, the callback panics, the re-mint never executes, and the user's ERC-20 tokens are permanently destroyed with no recovery path. The NEAR runtime does not retry failed callbacks.

---

### Likelihood Explanation

**Moderate.**

Three conditions must hold simultaneously:
1. The `error_refund` Cargo feature is compiled into the production binary (it is present in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml` and exercised in integration tests).
2. A user calls the `ExitToNear` precompile for an ERC-20 token whose NEP-141 recipient is not registered (a common user mistake).
3. The ERC-20 contract's `mint` function consumes enough EVM gas that the total NEAR gas for `refund_on_error` exceeds 10 TGas.

Condition 3 is realistic for any ERC-20 with non-trivial mint logic (e.g., fee-on-transfer tokens, tokens with hooks, or tokens with complex storage layouts). The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` demonstrates the refund path is a live, exercised code path. [7](#0-6) 

---

### Recommendation

Replace the hardcoded `EXIT_TO_NEAR_CALLBACK_GAS` with a dynamic calculation that forwards all remaining prepaid gas minus a small reservation for bookkeeping, mirroring the pattern already used in `calculate_attached_gas`:

```rust
// In ExitToNear::run, instead of:
attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,

// Use remaining gas forwarding (analogous to calculate_attached_gas):
attached_gas: env.prepaid_gas()
    .saturating_sub(env.used_gas())
    .saturating_sub(GAS_FOR_PROMISE_CREATION),
```

Alternatively, raise `EXIT_TO_NEAR_CALLBACK_GAS` to a value that covers the worst-case Aurora EVM overhead (at minimum 50–100 TGas), consistent with `WITHDRAWAL_GAS = 100 TGas` used elsewhere. [8](#0-7) 

---

### Proof of Concept

1. Deploy a NEP-141 token on NEAR with a complex `mint` function (e.g., one that writes to many storage slots, consuming >50,000 EVM gas when called via the ERC-20 mirror).
2. Bridge the NEP-141 to Aurora via `deploy_erc20_token`, creating an ERC-20 mirror.
3. From an Aurora EVM account that holds the ERC-20, call the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x01` (ERC-20 exit), specifying a NEAR recipient account that is **not registered** with the NEP-141 contract.
4. The ERC-20 tokens are burned in the EVM. The `ft_transfer` promise fails because the recipient is unregistered.
5. `exit_to_near_precompile_callback` is invoked with 10 TGas. It attempts `refund_on_error` → `engine.call()` → EVM mint. The NEAR gas budget is exhausted before the EVM mint completes.
6. The callback panics. The ERC-20 tokens remain burned. The user's balance is zero on both NEAR and Aurora with no recovery mechanism. [9](#0-8) [10](#0-9)

fn calculate_attached_gas<E: Env>(env: &E) -> NearGas {
    let required_gas = env.used_gas().saturating_add(GAS_FOR_PROMISE_CREATION);

    if required_gas >= env.prepaid_gas() {
        NearGas::new(0)
    } else {
        env.prepaid_gas() - required_gas
    }
}

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
