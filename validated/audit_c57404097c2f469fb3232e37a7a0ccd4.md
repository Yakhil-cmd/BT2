### Title
WalletConnect `chia_pushTransactions` Opaque Spend Bundle Bypasses Meaningful User Confirmation - (`packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

A malicious dApp connected via WalletConnect can send a crafted spend bundle through `chia_pushTransactions` (with `sign: true`) that transfers the user's XCH, CAT, or NFT to an attacker-controlled address. The GUI confirmation dialog cannot humanize the `transactions` parameter — it is accepted as opaque JSON and `parseCommandDisplay` returns `undefined` for this command — so the user sees only a raw JSON blob and cannot evaluate what they are approving. This is a direct analog to the external report's class: unconstrained input parameters that cause asset-affecting state changes without adequate validation or user understanding.

### Finding Description

The `chia_wallet.push_transactions` command is registered in `Commands.ts` with its `transactions` parameter typed as `'json'` — a completely opaque, structurally unvalidated blob: [1](#0-0) 

The `parseDappParams` function validates declared parameter types (bigint, string, number, bool) but for `type: 'json'` it performs no structural validation — the value is passed through as-is: [2](#0-1) 

The `parseCommandDisplay` function, which is responsible for humanizing command parameters into a readable wallet-delta display (showing "you are spending X XCH, receiving Y CAT"), only handles `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. For `chia_wallet.push_transactions` it returns `undefined`: [3](#0-2) 

This `undefined` display is passed directly to the `Confirm` dialog in `main.tsx`: [4](#0-3) 

The result is that the user's confirmation dialog shows only the raw `transactions` JSON blob — an unreadable spend bundle — with no indication of which coins are being spent, to which address, or in what amount.

The `sign` parameter (optional bool) means the dApp can submit an **unsigned** spend bundle and request the wallet daemon to sign it on the user's behalf: [5](#0-4) 

### Impact Explanation

A malicious dApp that has been granted `chia_pushTransactions` can craft a spend bundle that transfers the user's XCH, CAT, or NFT to an attacker-controlled address. With `sign: true`, the wallet daemon signs the bundle using the user's private key. The user is shown a "Confirm Push Transactions" dialog containing raw JSON they cannot evaluate. If the user clicks "Push" — a reasonable action for a dApp they believe is legitimate — their assets are transferred to the attacker. This constitutes unauthorized transfer of XCH/CAT/NFT caused by the user approving the wrong transaction due to spoofed/opaque WalletConnect state.

### Likelihood Explanation

The attacker must first convince the user to connect their wallet to a malicious dApp via WalletConnect and grant `chia_pushTransactions` permission. The pairing dialog labels it as a "Spending command": [6](#0-5) 

Once granted, the dApp can invoke `chia_pushTransactions` with any spend bundle at any time. The user sees a confirmation dialog each time, but the content is unreadable raw JSON. A user who has granted this permission to what they believe is a legitimate dApp (e.g., a DEX or game) will likely approve the dialog, especially if the dApp presents a plausible UI alongside the WalletConnect prompt.

### Recommendation

1. `parseCommandDisplay` must be extended to handle `chia_wallet.push_transactions`. The GUI should parse the spend bundle structure and display a human-readable summary of which coins/amounts/addresses are involved before the user confirms.
2. Until a full spend-bundle parser is implemented, the confirmation dialog for `chia_pushTransactions` should display a prominent warning that the transaction content cannot be verified by the GUI and the user should only approve if they fully trust the dApp.
3. Consider whether `sign: true` should be allowed at all via WalletConnect, or whether it should require a separate, more explicit permission.

### Proof of Concept

**Attacker preconditions:** Malicious dApp connected via WalletConnect with `chia_pushTransactions` in its granted command list.

**Attack steps:**

1. Malicious dApp calls `chia_pushTransactions` via WalletConnect with:
   ```json
   {
     "transactions": [<crafted_spend_bundle_spending_user_xch_to_attacker>],
     "sign": true,
     "push": true
   }
   ```

2. `parseDappParams` accepts `transactions` as opaque JSON with no structural validation. [7](#0-6) 

3. `dispatchPairRequest` checks pair, command grant, network, and fingerprint — all pass since the dApp was legitimately paired. [8](#0-7) 

4. `parseCommandDisplay` is called with `commandId = 'chia_wallet.push_transactions'` and returns `undefined` — no wallet delta is computed. [3](#0-2) 

5. The `Confirm` dialog opens showing the raw `transactions` JSON blob. The user sees "Confirm Push Transactions" with unreadable data and clicks "Push".

6. `sendCommand` forwards the spend bundle to the wallet daemon with `sign: true`. The daemon signs and submits it. The user's XCH is transferred to the attacker.

        }

          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);

          if (trimmed === '') {
            throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
          }
        }

          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
}

export async function dispatchPairRequest<T>(
  topic: string,
  command: string,
  params: Record<string, unknown>,
  process: (context: DispatchPairRequestContext) => Promise<T>,
  confirm: () => Promise<boolean>,
): Promise<T> {
  const [loggedInFingerprint, isMainnetValue] = await Promise.all([getLoggedInFingerprint(), isMainnet()]);

  const pair = findPair(topic);
  if (!pair) {
    throw new WcError(`Pair not found`, WcErrorCode.USER_REJECTED);
  }

  // verify if the command is allowed for this pair
  if (!pair.commands.includes(command)) {
    throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const { fingerprint } = params;

  // verify if the network is the same as the pair's network
  if (isMainnetValue !== pair.mainnet) {
    throw new WcError(`Network mismatch`, WcErrorCode.UNSUPPORTED_CHAINS);
  }

  // verify if the requested fingerprint is allowed for this pair
  const requestedFingerprint = fingerprint ?? loggedInFingerprint;
  if (typeof requestedFingerprint !== 'number' || !requestedFingerprint || requestedFingerprint !== pair.fingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const context = {
    pair,
    fingerprint: requestedFingerprint,
  };

  // Dapps may not switch the active key for an existing pair.
  if (fingerprint !== undefined && fingerprint !== loggedInFingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  // if command is bypassed return true
  if (pair.bypass.includes(command)) {
    return process(context);
  }

  const isAllowed = await confirm();
  if (isAllowed === true) {
    return process(context);
  }

  throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
