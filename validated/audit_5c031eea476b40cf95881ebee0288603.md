### Title
Missing Proof-of-Correctness Validation on CKD Share Contributions Allows Malicious Participant to Corrupt Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator accumulates per-participant share contributions `(big_Y_i, big_C_i)` without any cryptographic proof that each participant used their actual private share or correctly formed their ElGamal component. A single malicious participant can send arbitrary group elements, shifting the aggregated output by an attacker-controlled offset. Honest parties accept the corrupted result, producing an unusable or attacker-influenced derived confidential key. This is a direct data-validation analog to the external report: just as Salty.IO assumed USDS = $1 without on-chain verification, this protocol assumes each received CKD share is honestly computed without any on-protocol verification.

---

### Finding Description

**Root cause — `do_ckd_coordinator`, lines 50–55:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant privately sends `(λ_i · Y_i, λ_i · C_i)` to the coordinator, where the intended values are:

- `Y_i = y_i · G` (a fresh random commitment)
- `C_i = x_i · H(pk, app_id) + y_i · app_pk` (ElGamal ciphertext component using the participant's actual private share `x_i`)

The coordinator sums all received values and returns the aggregate as the CKD output. **No zero-knowledge proof, commitment binding, or consistency check is performed on any received pair.** A malicious participant `j` can substitute arbitrary group elements `(big_Y_j', big_C_j')` for their correct contribution. The coordinator has no mechanism to detect the substitution.

**Contrast with DKG (`src/dkg.rs`):** The DKG protocol enforces correctness at every analogous step — `verify_proof_of_knowledge` (lines 452–460), `verify_commitment_hash` (lines 463–469), and `validate_received_share` (lines 520–522) — before accepting any participant contribution. The CKD protocol has no equivalent guards.

**Attacker-controlled entry path:**

1. Caller invokes `ckd(participants, coordinator, me, key_pair, app_id, app_pk, rng)` — a public API function (`src/confidential_key_derivation/protocol.rs`, line 66).
2. The malicious participant's instance of `do_ckd_participant` (line 17) calls `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))` (line 30) with attacker-chosen values instead of the output of `compute_signature_share`.
3. The coordinator's `do_ckd_coordinator` (line 35) receives and unconditionally accumulates the malicious values (lines 50–55).
4. The corrupted `CKDOutput` is returned to all honest callers as the protocol result.

---

### Impact Explanation

The CKD output `(big_Y, big_C)` is an ElGamal encryption of `msk · H(pk, app_id)` under `app_pk`, where `msk` is the distributed master secret. A malicious participant `j` who sends `(big_Y_j', big_C_j')` shifts the aggregate by `(big_Y_j' − λ_j · y_j · G, big_C_j' − λ_j · C_j)`. The resulting output is an encryption of a value that differs from `msk · H(pk, app_id)` by an attacker-controlled offset. Honest parties accept this output without any indication of corruption. The TEE that subsequently calls `unmask(app_sk)` on the corrupted output derives a wrong confidential key, rendering the CKD protocol output cryptographically unusable.

**Impact category:** High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

Any participant enrolled in a CKD session can trigger this with zero additional privilege. The attack requires only the ability to participate in the protocol (the normal caller role) and the ability to substitute arbitrary serializable group elements in the private message sent to the coordinator. No key material, side-channel access, or external oracle is needed. The protocol provides no detection mechanism, so the attack succeeds silently every time it is attempted.

---

### Recommendation

Add a non-interactive zero-knowledge proof of correct share formation to each participant's contribution. Concretely, each participant should prove in zero knowledge that:

1. `big_Y_i = λ_i · y_i · G` for the same `y_i` used in `big_C_i`.
2. `big_C_i = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` where `x_i` is consistent with the participant's public verification share from `key_pair.public_key`.

A standard Chaum–Pedersen or Schnorr-based DLEQ proof over BLS12-381 G1 is sufficient. The coordinator must verify all proofs before accumulating any contribution, mirroring the `verify_proof_of_knowledge` pattern already present in `src/dkg.rs` (lines 452–460).

### Proof of Concept

```
Setup: N=3 participants, threshold=2. Participants P1, P2 are honest; P3 is malicious.

1. All three participants call ckd(...) with the same (participants, coordinator=P1, app_id, app_pk).

2. P1 and P2 run compute_signature_share() correctly and send their outputs to P1 (coordinator).

3. P3, instead of calling compute_signature_share(), sends:
       big_Y_3' = ElementG1::generator()   // arbitrary non-zero point
       big_C_3' = ElementG1::generator()   // arbitrary non-zero point

4. P1's do_ckd_coordinator() receives all three contributions and sums them:
       norm_big_y = λ1·Y1 + λ2·Y2 + G        // G instead of λ3·Y3
       norm_big_c = λ1·C1 + λ2·C2 + G        // G instead of λ3·C3

5. The returned CKDOutput encodes an encryption of a value ≠ msk·H(pk, app_id).

6. Any TEE calling ckd_output.unmask(app_sk) derives a wrong key with no error signal.
   The correct confidential key is permanently unrecoverable from this session's output.
```
