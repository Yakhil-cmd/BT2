### Title
`skip_stateful_validations` admits invoke transactions with arbitrary signatures for accounts with a pending `deploy_account` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function bypasses the `__validate__` entry-point (signature verification) for any `Invoke` transaction with `nonce=1` when the sender's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from that address in the pool — not specifically a `deploy_account` — an unprivileged attacker who observes a victim's pending `deploy_account` can inject an `Invoke(nonce=1)` carrying an arbitrary/invalid signature. The gateway admits it without calling `__validate__`, placing it in the mempool and blocking the victim's legitimate first post-deployment invoke.

### Finding Description

**Broken invariant:** Every admitted `Invoke` transaction must have its signature verified by the account's `__validate__` entry point before entering the mempool.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`. [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator` returns immediately without calling `__validate__`: [3](#0-2) [4](#0-3) 

**Why the guard is insufficient:**

`account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the address, not specifically a `deploy_account`: [5](#0-4) [6](#0-5) 

**Nonce gap allows nonce=1 for account_nonce=0:**

The gateway's `validate_nonce` permits `tx_nonce` up to `account_nonce + max_allowed_nonce_gap`. The production default is `max_allowed_nonce_gap = 200`, so `nonce=1` is always within range when `account_nonce=0`: [7](#0-6) [8](#0-7) [9](#0-8) 

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

Attack steps (all unprivileged, no special access required):

1. Victim broadcasts `deploy_account` for address `A`. It enters the mempool; `tx_pool.contains_account(A)` becomes `true`.
2. Attacker submits `Invoke(sender=A, nonce=1, calldata=<anything>, signature=<garbage>)`.
3. Gateway: `account_nonce=0`, `tx_nonce=1` passes the nonce-gap check (1 ≤ 200).
4. `validate_by_mempool` passes (nonce ≥ account_nonce).
5. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(A)` is `true`.
6. `run_validate_entry_point` is called with `skip_validate=true` → `__validate__` is **not called** → invalid signature is never checked.
7. Attacker's `Invoke(nonce=1)` is admitted to the mempool.
8. Victim's legitimate `Invoke(nonce=1)` is now rejected as a duplicate nonce (or requires fee escalation to displace the attacker's transaction).
9. Batcher executes `deploy_account` (nonce=0) → account `A` is deployed.
10. Batcher executes attacker's `Invoke(nonce=1)` → `__validate__` fails → transaction reverted; account nonce advances to 2.
11. Victim's legitimate `Invoke(nonce=1)` is now stale; victim must re-submit with `nonce=2`.

Consequences:
- **Invalid transaction admitted to mempool** (the core invariant violation).
- **Victim's first post-deployment invoke is blocked** and their nonce is burned by the attacker.
- If the account has balance after deployment, the reverted transaction may still charge a fee.
- Attacker can repeat this for every new `deploy_account` observed in the mempool, at negligible cost.

### Likelihood Explanation

**Likelihood: High.** The mempool is observable (P2P propagation, RPC). Any `deploy_account` transaction is visible before it is executed. The attacker needs only to submit a single `Invoke(nonce=1)` with a random signature before the victim does. No privileged access, no special contract, no stake required. The `max_allowed_nonce_gap=200` production default ensures the nonce check never blocks the attack.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the pool for the sender address. Introduce a dedicated mempool query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, and use that in `skip_stateful_validations` instead of the generic presence check.

Alternatively, restrict `skip_stateful_validations` to only skip validation when the **caller** of the gateway is the same entity that submitted the `deploy_account` (e.g., by requiring the two transactions to arrive in the same batch/request), which is the intended UX scenario.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
