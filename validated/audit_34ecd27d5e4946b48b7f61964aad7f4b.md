### Title
Gateway Skips Signature Validation for Nonce-1 Invoke Transactions When Victim Account Has Any Pending Mempool Entry — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (the on-chain signature check) for any invoke transaction with `nonce=1` when the sender's on-chain nonce is `0` and the sender address appears in `account_tx_in_pool_or_recent_block`. Because that mempool check returns `true` for **any** account that has **any** transaction in the pool — not specifically a deploy-account transaction — an unprivileged attacker can inject a signature-invalid invoke transaction for any victim account that has a pending deploy-account transaction, causing the victim's account to be charged fees for a reverted transaction and potentially displacing the victim's own legitimate nonce-1 invoke transaction via fee escalation.

``` [3](#0-2) 
This returns `true` for **any** account that has **any** transaction in the pool — including a deploy-account transaction submitted by a different party. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is incorrect: the attacker's own nonce-1 invoke transaction (with a bad signature) is the first invoke transaction for the account and passes the mempool's `validate_tx` check (which only checks for duplicate hashes and nonce range, not signatures). The `validate_by_mempool` call that precedes `skip_stateful_validations` also performs no signature check: [5](#0-4) 

The `StatelessTransactionValidator` only checks signature **length**, not validity: [6](#0-5) 

**Attack flow:**

1. Alice submits a `deploy_account` tx for address `A` (nonce 0). The gateway accepts it and it enters the mempool.
2. Attacker observes `A` in the mempool and submits an invoke tx: `sender=A`, `nonce=1`, **arbitrary/invalid signature**, high tip.
3. Gateway stateless check: signature length is within bounds → passes.
4. Gateway stateful check: `validate_state_preconditions` → nonce 1 is within `[0, 200]` → passes. `validate_by_mempool` → no duplicate, nonce valid → passes. `skip_stateful_validations` → `A` is in pool → returns `true`. `run_validate_entry_point` with `validate=false` → `__validate__` is never called → tx accepted.
5. Attacker's tx enters the mempool. If Alice also submitted a nonce-1 invoke, the attacker can set a higher tip to displace it via fee escalation.
6. Batcher executes deploy-account tx → `A` is deployed with some STRK balance.
7. Batcher executes attacker's invoke tx using `AccountTransaction::new_for_sequencing` (which sets `validate: true`). `__validate__` is called, fails on the bad signature, the tx reverts, and fees are charged from `A`'s balance. [7](#0-6) 

### Impact Explanation

- **Admission of invalid transaction:** A signature-invalid invoke transaction is accepted by the gateway and enters the mempool. This directly satisfies "High. Mempool/gateway/RPC admission accepts invalid transactions."
- **Economic harm to victim:** The reverted transaction's fee is charged from the victim's account. The attacker pays nothing to submit the transaction.
- **Displacement of legitimate transaction:** Via fee escalation, the attacker can replace the victim's own nonce-1 invoke transaction, delaying or blocking the victim's first post-deployment action.
- **Unprivileged trigger:** Any party can submit transactions to the public gateway endpoint.

### Likelihood Explanation

The attack is straightforward: monitor the public mempool (or P2P gossip) for deploy-account transactions, then immediately submit a nonce-1 invoke with a bad signature and a high tip. No special knowledge or privilege is required. The window is open from the moment the deploy-account tx enters the mempool until the block containing it is committed.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction exists for the sender address in the mempool. Alternatively, add a dedicated mempool query such as `has_pending_deploy_account(address)` that only returns `true` when a deploy-account transaction (not just any transaction) is pending for that address. This preserves the intended UX feature while closing the signature-bypass window.

### Proof of Concept

```
1. Alice: POST /add_transaction  →  deploy_account { sender=A, nonce=0, valid_sig }
   Gateway accepts; mempool now contains A's deploy_account.

2. Attacker: POST /add_transaction  →  invoke { sender=A, nonce=1, sig=[0xdead,0xbeef], tip=HIGH }
   - StatelessTransactionValidator: sig length 2 ≤ max → OK
   - validate_state_preconditions: nonce 1 ∈ [0, 200] → OK
   - validate_by_mempool: no duplicate, nonce valid → OK
   - skip_stateful_validations: account_tx_in_pool_or_recent_block(A) == true → returns true
   - run_validate_entry_point(skip_validate=true): validate=false → __validate__ skipped → OK
   Gateway accepts; mempool now contains both txs.
   If Alice also submitted nonce=1 invoke with lower tip, attacker's tx replaces it.

3. Batcher builds block:
   - Executes deploy_account(A, nonce=0) → A deployed, balance funded.
   - Executes attacker's invoke(A, nonce=1, validate=true):
       __validate__ called → bad signature → PanicInValidate / InvalidValidateReturnData
       tx reverts; fee charged from A's balance.

Result: Alice's account drained of one tx-worth of fees; her own nonce-1 invoke displaced.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
