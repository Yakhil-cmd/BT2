### Title
Unprivileged Caller Can Advance `last_reward_block` Without Distributing Rewards, Permanently Skipping Block Rewards for All Stakers - (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in the Staking contract is publicly callable by any non-zero address. When called with `disable_rewards: true`, it advances the global `last_reward_block` to the current block without distributing any rewards. Because `last_reward_block` is a **global** gate — not per-staker — any subsequent legitimate call to `update_rewards` in the same block is rejected with `REWARDS_ALREADY_UPDATED`. An attacker can front-run every block's first legitimate reward call, permanently skipping block rewards for all stakers in that block.

---

### Finding Description

`update_rewards` is defined in `IStakingRewardsManager` and enforces only `general_prerequisites()` (pause check + non-zero caller). No role-based access control exists. [1](#0-0) 

The function immediately writes `last_reward_block` to the current block number — before any reward logic runs: [2](#0-1) 

If `disable_rewards` is `true`, the function returns right after that write, distributing nothing: [3](#0-2) 

Because `last_reward_block` is a single global storage slot, the gate check at the top of the function now blocks every other caller for the rest of that block: [4](#0-3) 

The attacker only needs to supply any currently-active staker address with non-zero STRK balance — information that is fully public on-chain. [5](#0-4) 

---

### Impact Explanation

In consensus-rewards mode (after `consensus_rewards_first_epoch`), block rewards are the sole reward mechanism for stakers. By front-running the first `update_rewards` call each block with `disable_rewards: true`, an attacker causes `last_reward_block` to advance without `_update_rewards` ever being reached. The reward supplier's `unclaimed_rewards` is never incremented for that block, and the block's inflation allocation is permanently lost. Repeated across every block, this constitutes **permanent freezing of unclaimed yield** for all stakers and delegators. [6](#0-5) 

---

### Likelihood Explanation

The attack requires no special privilege, no capital, and no coordination. The attacker only needs to:
1. Know any valid staker address with non-zero balance (trivially readable from on-chain events).
2. Submit a transaction with `disable_rewards: true` before any legitimate `update_rewards` call in each block.

This is a straightforward front-running griefing attack executable by any external account. The consensus-rewards path is active once `consensus_rewards_first_epoch` is set, which is a planned protocol milestone. [7](#0-6) 

---

### Recommendation

Restrict `update_rewards` to callers with a designated role (e.g., `only_app_governor` or a dedicated rewards-updater role), or remove the `disable_rewards` parameter from the public interface entirely and handle the "skip rewards" case through an internal function that is not externally callable. Additionally, consider making `last_reward_block` a per-staker variable rather than a global gate, so one call cannot block all others.

---

### Proof of Concept

```
// Attacker (any EOA) front-runs every block:
// 1. Pick any active staker address with non-zero balance (e.g., from NewStaker events).
let victim_staker: ContractAddress = ...; // any valid staker

// 2. Call update_rewards with disable_rewards = true.
//    - last_reward_block is set to current_block_number.
//    - No rewards are distributed (early return at `if disable_rewards`).
staking.update_rewards(staker_address: victim_staker, disable_rewards: true);

// 3. Any legitimate staker trying to claim block rewards in the same block now hits:
//    assert!(current_block_number > self.last_reward_block.read(), REWARDS_ALREADY_UPDATED)
//    => PANICS. Block rewards are permanently lost for this block.
staking.update_rewards(staker_address: legitimate_staker, disable_rewards: false); // REVERTS
```

Repeating this every block permanently freezes all consensus-mode block rewards for every staker and delegator in the protocol.
