### Title
Precision Loss in `native_per_pubdata` Floor Division Allows Bypassing Pubdata Native Resource Charges - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

The `native_per_pubdata` ratio is computed with floor division (`wrapping_div`) while `native_per_gas` uses ceiling division (`div_ceil`). When `pubdata_price < native_price`, `native_per_pubdata` truncates to zero, completely eliminating the native resource charge for all pubdata written during a transaction. Any user can then publish unlimited state changes without paying the corresponding native (proving) resource cost, shifting the full proving burden to the operator.

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 142, `native_per_pubdata` is computed with floor division:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

Compare with `native_per_gas` at line 135, which explicitly uses ceiling division to ensure the user always pays at least the actual cost:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same floor-division pattern is replicated in `api/src/helpers.rs` line 427:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `pubdata_price < native_price`, `native_per_pubdata` truncates to `0`. This zero value propagates through three critical downstream functions:

**1. `create_resources_for_tx` — no native reserved for intrinsic pubdata:**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0, so native_limit is not reduced
``` [4](#0-3) 

**2. `get_resources_to_charge_for_pubdata` — zero native charged for any pubdata:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // → pubdata_bytes * 0 = 0
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

**3. `check_enough_resources_for_pubdata` — pubdata check always passes:**
```rust
let enough = resources.has_enough(&resources_for_pubdata); // resources_for_pubdata = 0 → always true
``` [6](#0-5) 

This check gates whether a transaction is reverted for insufficient pubdata resources: [7](#0-6) 

Additionally, in `compute_gas_refund`, the `delta_gas` adjustment that compensates for native overconsumption also uses floor division:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [8](#0-7) 

When `native_per_pubdata = 0`, pubdata native costs are excluded from `native_used`, so `delta_gas` also underestimates the gas equivalent of native consumption, compounding the undercharge.

### Impact Explanation

**Extreme case (`pubdata_price < native_price` → `native_per_pubdata = 0`):**
- Users can write to an arbitrary number of storage slots (generating large pubdata) without paying any native resources for it.
- The full native limit remains available for computation (not reduced by pubdata overhead).
- The operator/prover bears the entire proving cost for the pubdata.
- This is a direct resource accounting loss for the operator: the user pays EVM gas but zero native resources for pubdata proving.

**General case (`pubdata_price % native_price != 0`):**
- Users are systematically undercharged by up to `(native_price − 1)` native units per byte of pubdata.
- For `native_price = 1000`, `pubdata_price = 1999`: `native_per_pubdata = 1` instead of `2`. For 10,000 bytes of pubdata, the user is undercharged by ~10,000 native units (≈33% of the actual cost).

The ZKsync OS documentation explicitly states that native resource consumption drives proving cost and that running out of native resources reverts the transaction. Bypassing this charge undermines the economic security of the proving system. [9](#0-8) 

### Likelihood Explanation

The extreme case (`native_per_pubdata = 0`) requires `pubdata_price < native_price`. While the operator sets these values, this condition can arise from misconfiguration or from an operator intentionally subsidizing pubdata. There is no guard in the validation path that rejects a transaction when `native_per_pubdata == 0` while `pubdata_price > 0`.

The systematic undercharge (floor vs. ceil) is always present whenever `pubdata_price % native_price != 0`, which is the common case for arbitrary operator-set prices. Any user submitting a transaction that generates pubdata benefits from this undercharge.

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — undercharges user):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. Additionally, add an explicit check rejecting transactions where `native_per_pubdata == 0` but `pubdata_price > 0`, analogous to the existing `native_price != 0` guard.

### Proof of Concept

1. Operator sets block context: `pubdata_price = 999`, `native_price = 1000`.
2. `native_per_pubdata = 999.wrapping_div(1000) = 0`.
3. User submits an EIP-1559 transaction calling a contract that writes to 500 storage slots (generating ~16,000 bytes of pubdata).
4. In `create_resources_for_tx`: `intrinsic_pubdata_overhead = 0 × intrinsic_pubdata = 0` — full native limit available.
5. After execution, `get_resources_to_charge_for_pubdata` returns `(16000, Resources{native: 0})`.
6. `check_enough_resources_for_pubdata` returns `has_enough = true` regardless of remaining native.
7. `compute_gas_refund` charges 0 native for pubdata; `delta_gas` does not account for pubdata proving cost.
8. Transaction succeeds. User paid EVM gas only. Operator/prover bears the full native proving cost for 16,000 bytes of pubdata.
