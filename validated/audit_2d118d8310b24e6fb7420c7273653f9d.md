### Title
Rewards Permanently Frozen in Pool Contract When Total Pool Balance Falls Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When the total delegated balance in a Pool contract is below `min_delegation_for_rewards`, the pool's `compute_rewards_per_unit` returns zero, causing the cumulative rewards trace to record no increase. However, the Staking contract has already transferred the STRK reward tokens to the Pool contract before this check occurs. Those tokens are permanently unclaimable by any delegator.

---

### Finding Description

The Pool contract stores a `cumulative_rewards_trace` that tracks the per-unit reward index. Each time the Staking contract calls `update_rewards_from_staking_contract`, the pool appends a new entry:

```cairo
// src/pool/pool.cairo  lines 569–587
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    self.assert_caller_is_staking_contract();
    let (_, last) = self.cumulative_rewards_trace.last().unwrap();
    self
        .cumulative_rewards_trace
        .insert(
            key: self.get_current_epoch(),
            value: last
                + self
                    .compute_rewards_per_unit(
                        staking_rewards: rewards, total_stake: pool_balance,
                    ),
        );
}
```

`compute_rewards_per_unit` contains an explicit early-return of zero when `total_stake < min_delegation_for_rewards`:

```cairo
// src/pool/pool.cairo  lines 967–978
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    mul_wide_and_div(
        lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
    )
        .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
}
```

The thresholds are set at construction time via `get_token_rewards_config`:

- **STRK**: `min_for_rewards = 10^18` (1 STRK)
- **BTC (8 dec)**: `min_for_rewards = 10^3`

```cairo
// src/pool/utils.cairo  lines 124–138
pub(crate) fn get_token_rewards_config(token_address: ContractAddress) -> TokenRewardsConfig {
    if token_address == STRK_TOKEN_ADDRESS {
        STRK_CONFIG   // min_for_rewards: 10^18
    } else {
        TokenRewardsConfig {
            decimals,
            min_for_rewards: 10_u128.pow(decimals.into() - 5),
            ...
        }
    }
}
```

On the Staking side, `update_pool_rewards` **first** transfers STRK tokens to the Pool contract, **then** calls `update_rewards_from_staking_contract`:

```cairo
// src/staking/staking.cairo  lines 1865–1891
fn update_pool_rewards(...) {
    for (pool_contract, token_address, pool_balance, pool_rewards) in pools_rewards_data {
        self.send_rewards_to_delegation_pool(   // ← tokens transferred HERE
            :staker_address,
            pool_address: pool_contract,
            amount: pool_rewards,
            ...
        );
        pool_dispatcher
            .update_rewards_from_staking_contract(  // ← trace updated HERE (may record 0)
                rewards: pool_rewards,
                pool_balance: pool_balance.to_native_amount(:decimals),
            );
    }
}
```

The Staking contract decides whether to send rewards based solely on whether `pool_rewards > 0` (computed from the 18-decimal normalized balance), **without checking** whether the native-decimal pool balance is below `min_delegation_for_rewards`:

```cairo
// src/staking/staking.cairo  lines 1994–1999
if pool_rewards.is_non_zero() {
    pool_rewards_array
        .append(
            (pool_contract, token_address, pool_balance_curr_epoch, pool_rewards),
        );
}
```

The pool contract's own comment acknowledges the mismatch:

```cairo
// src/pool/pool.cairo  lines 960–966
/// **Note**: Delegation rewards lost when pool balance is less than
/// `min_delegation_for_rewards`. The staking contract continues to forward
/// `pool_rewards` to the pool contract even in this case.
```

**Concrete example (STRK pool):**

| Parameter | Value |
|---|---|
| Total pool balance | `5 × 10^17` fri (0.5 STRK) |
| `min_delegation_for_rewards` | `10^18` fri (1 STRK) |
| Total staker STRK stake | `10^21` fri (1000 STRK) |
| Block rewards | `10^18` fri (1 STRK) |
| `pool_rewards_including_commission` | `10^18 × 5×10^17 / 10^21 = 5×10^14` (non-zero) |
| Tokens sent to pool | `5×10^14` STRK |
| `compute_rewards_per_unit` result | **0** (because `5×10^17 < 10^18`) |
| Cumulative trace delta | **0** |
| Delegator claimable rewards | **0** |
| Tokens stuck in pool contract | **`5×10^14` STRK permanently** |

There is no admin-withdrawal function in the Pool contract. `claim_rewards` reads only from the cumulative trace, and `exit_delegation_pool_action` returns only the delegated principal. The frozen STRK has no recovery path in the current implementation.

---

### Impact Explanation

STRK reward tokens are transferred into the Pool contract but are never recorded in the cumulative rewards trace. Delegators calling `claim_rewards` receive zero for those epochs. The tokens cannot be recovered by any delegator, staker, or admin through any existing function. This constitutes **permanent freezing of unclaimed yield** (High).

---

### Likelihood Explanation

- There is no minimum delegation amount enforced in `enter_delegation_pool` beyond `amount.is_non_zero()`.
- A single delegator with less than 1 STRK (or the BTC equivalent) is sufficient to trigger the condition.
- The condition is also reachable naturally when a pool is newly opened and has only a small initial delegation.
- No attacker profit motive is required; the freeze occurs automatically whenever the total pool balance dips below the threshold while the staker is actively earning rewards.

---

### Recommendation

Before transferring rewards to the pool contract, the Staking contract should check whether the pool's native-decimal balance meets `min_delegation_for_rewards`. If it does not, the rewards should either be withheld (and credited to the staker or burned) or the pool contract should be updated to track and redistribute them in a future epoch when the balance crosses the threshold. The current split — where the Staking contract uses 18-decimal normalized balances to decide whether to send, while the Pool contract uses native-decimal balances to decide whether to record — creates the mismatch.

---

### Proof of Concept

1. Deploy the system with a staker holding 1000 STRK own stake and a STRK pool.
2. A delegator calls `enter_delegation_pool` with `amount = 5 × 10^17` (0.5 STRK).
3. Advance K epochs so the balance becomes effective.
4. The sequencer calls `update_rewards` (V3) or the attestation contract calls `update_rewards_from_attestation_contract` (V2).
5. In `calculate_staker_pools_rewards`, `pool_rewards > 0` because `5×10^17 / 10^21 × total_rewards > 0`.
6. `send_rewards_to_delegation_pool` transfers STRK to the pool contract.
7. `update_rewards_from_staking_contract` is called; `compute_rewards_per_unit` returns 0 because `5×10^17 < 10^18`.
8. The cumulative trace records `last + 0`.
9. The delegator calls `claim_rewards` and receives 0.
10. The STRK balance of the pool contract is non-zero but permanently unclaimable.
