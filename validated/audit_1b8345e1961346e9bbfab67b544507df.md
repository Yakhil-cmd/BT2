### Title
Unrestricted `update_rewards` with `disable_rewards: true` Allows Permanent Griefing of Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `Staking` contract is callable by any unprivileged address. It uses a **global** `last_reward_block` to enforce a one-reward-update-per-block invariant. However, the function accepts a `disable_rewards` flag that, when `true`, causes it to advance `last_reward_block` to the current block **without distributing any rewards**. An attacker can call `update_rewards(any_valid_staker, true)` every block, consuming the global slot and permanently preventing all stakers from receiving consensus rewards.

---

### Finding Description

`update_rewards` is a public, permissionless function. Its body:

1. Checks `current_block_number > last_reward_block` (the per-block guard).
2. Validates the supplied `staker_address` is active with non-zero balance.
3. **Writes `last_reward_block = current_block_number`** unconditionally.
4. Then checks `if disable_rewards || self.is_pre_consensus() { return; }` — returning early with no reward distribution. [1](#0-0) [2](#0-1) 

The critical ordering flaw: `last_reward_block` is updated at line 1485 **before** the `disable_rewards` branch at line 1487. Any caller can therefore:

- Pass `disable_rewards: true` with any valid, active staker address.
- Advance `last_reward_block` to the current block.
- Return without distributing rewards.

Any legitimate call to `update_rewards` in the same block then fails with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

There is no access-control check beyond `general_prerequisites()`, which only asserts the contract is unpaused and the caller is non-zero: [4](#0-3) 

The `last_reward_block` is a single global storage slot shared across all stakers: [5](#0-4) 

---

### Impact Explanation

In the consensus rewards phase (after `consensus_rewards_first_epoch` is set), `update_rewards` is the sole mechanism for distributing per-block STRK and BTC rewards to stakers. By calling `update_rewards(valid_staker, true)` every block, an attacker causes `last_reward_block` to advance each block with zero rewards distributed. All stakers are denied their consensus yield indefinitely. This constitutes **permanent freezing of unclaimed yield** (sustained attack) or **temporary freezing** (intermittent attack), both of which are High-impact in the allowed scope.

---

### Likelihood Explanation

Medium. There is no financial profit motive for the attacker. However, the attack is trivially executable: the attacker only needs to identify any one active staker with non-zero balance (publicly observable on-chain) and submit one transaction per block. On Starknet, transaction fees are low, making sustained griefing economically feasible for a motivated adversary.

---

### Recommendation

Move the `last_reward_block` write to **after** the `disable_rewards` guard, so that consuming the per-block slot only occurs when rewards are actually distributed:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Update last block rewards only when rewards are actually distributed.
self.last_reward_block.write(current_block_number);
// ... rest of reward distribution logic
```

Alternatively, add access control so that only the staker's registered operational address (or a designated consensus contract) may call `update_rewards`.

---

### Proof of Concept

1. Consensus rewards phase is active (`consensus_rewards_first_epoch` has been set and passed).
2. Attacker identifies any active staker `S` with non-zero STRK balance (readable from public state).
3. Each block, attacker submits: `staking.update_rewards(S, disable_rewards: true)`.
4. Inside the call:
   - `current_block_number > last_reward_block` → passes (new block).
   - Staker `S` is valid and active → passes.
   - `last_reward_block` is written to `current_block_number`.
   - `disable_rewards == true` → early return, zero rewards distributed.
5. Any legitimate call to `update_rewards` for any staker in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeated every block: no staker ever accumulates consensus rewards. `unclaimed_rewards_own` for all stakers remains frozen at its pre-attack value. [6](#0-5)
