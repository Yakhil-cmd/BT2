### Title
`dlogeq::verify` propagates `Err(IdentityElement)` instead of `Ok(false)` for crafted proofs, enabling DoS of triple generation — (`src/crypto/proofs/dlogeq.rs`, `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

A malicious participant in triple generation can craft a `Proof{e, s}` with `s = x·e` (where `x = dlog(public0)`, a value the attacker knows because it is their own secret) such that the reconstructed commitment `big_k0` in `verify` evaluates to the group identity. This causes `encode_two_points` to return `Err(ProtocolError::IdentityElement)`, which propagates via `?` through `verify` and then again via `?` in `generation.rs`, aborting the honest verifier's protocol execution entirely rather than returning `Ok(false)` (proof rejected). The result is permanent denial of triple generation for any honest party that receives the crafted message.

---

### Finding Description

**Step 1 — `verify` computes the commitment and calls `encode_two_points`** [1](#0-0) 

```rust
let (phi0, phi1) = statement.phi(&proof.s.0);
let big_k0 = phi0 - *statement.public0 * proof.e.0;
let big_k1 = phi1 - *statement.public1 * proof.e.0;
let enc = encode_two_points::<C>(&big_k0, &big_k1)?;
```

If `big_k0` is the group identity, `encode_two_points` fails: [2](#0-1) 

```rust
let mut ser1 = C::Group::serialize(point_1)
    .map_err(|_| ProtocolError::IdentityElement)?
```

The `?` on line 157 propagates `Err(ProtocolError::IdentityElement)` out of `verify`.

**Step 2 — The caller in `generation.rs` uses `?` on the result of `verify`** [3](#0-2) 

```rust
if !dlogeq::verify(
    &mut transcript.fork(b"dlogeq0", &from.bytes()),
    statement,
    their_phi_proof,
)? {
    return Err(ProtocolError::AssertionFailed(...));
}
```

The `?` on line 487 propagates `Err(ProtocolError::IdentityElement)` directly, aborting the entire async protocol task — not just rejecting the proof.

**Step 3 — The attacker can trivially trigger this**

The statement constructed for the incoming proof is: [4](#0-3) 

```rust
let statement = dlogeq::Statement::<C> {
    public0: &big_e_j_zero.index(from)?.value(),
    generator1: &big_f.eval_at_zero()?.value(),
    public1: &big_c_j,
};
```

`public0` is indexed by `from` — the sender's own public commitment. The attacker knows the discrete log `x` of their own `public0 = x·G` because they created it. They choose any scalar `e`, compute `s = x·e`, and send `Proof{e, s}`.

During verification:
```
big_k0 = s·G − public0·e
       = (x·e)·G − (x·G)·e
       = x·e·G − x·e·G
       = identity
```

`encode_two_points` is called with the identity point and returns `Err(ProtocolError::IdentityElement)`.

---

### Impact Explanation

The `Err` propagates out of the honest party's protocol coroutine, permanently aborting triple generation. The honest party cannot complete the protocol and cannot produce triples needed for ECDSA presigning/signing. A single malicious participant can target any honest party by sending one crafted message per round, causing permanent denial of triple generation for that party.

This matches: **High: Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The attacker is a legitimate protocol participant (required by the threat model). They know their own secret scalar `x` trivially. The crafted proof requires only one scalar multiplication (`s = x·e`). No cryptographic assumption needs to be broken. The attack is deterministic and reproducible.

---

### Recommendation

In `verify`, catch the identity-element error from `encode_two_points` and convert it to `Ok(false)` rather than propagating it as `Err`. A malformed proof that produces an identity commitment is simply an invalid proof and should be rejected gracefully:

```rust
let enc = match encode_two_points::<C>(&big_k0, &big_k1) {
    Ok(enc) => enc,
    Err(ProtocolError::IdentityElement) => return Ok(false),
    Err(e) => return Err(e),
};
```

This preserves the invariant that `verify` returns `Ok(false)` for any structurally invalid or maliciously crafted proof, and only returns `Err` for genuine internal errors.

---

### Proof of Concept

```rust
// Attacker knows x = dlog(public0) because public0 is their own commitment.
// Choose arbitrary e, compute s = x*e.
// The verifier computes big_k0 = s*G - public0*e = x*e*G - x*G*e = identity.
// encode_two_points returns Err(IdentityElement).
// The ? in verify propagates it.
// The ? in generation.rs:487 propagates it, aborting the honest party.

let x: Scalar = attacker_secret;
let public0 = ProjectivePoint::GENERATOR * x; // attacker's own commitment

let e = Scalar::from(42u64); // arbitrary
let s = x * e;               // crafted s

let crafted_proof = Proof { e: SerializableScalar(e), s: SerializableScalar(s) };

// Verifier side:
let result = dlogeq::verify(&mut transcript, statement, &crafted_proof);
// result == Err(ProtocolError::IdentityElement)  ← not Ok(false)
assert!(matches!(result, Err(ProtocolError::IdentityElement)));
// Protocol aborts instead of rejecting the proof.
```
