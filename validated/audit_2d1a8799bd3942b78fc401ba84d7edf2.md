### Title
Unbounded `pool_member_epoch_balance` Trace Growth Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over a per-delegator balance trace (`pool_member_epoch_balance`) in an unbounded loop. Because `add_to_delegation_pool` enforces only `amount.is_non_zero()` with no minimum-amount floor, a delegator can append one new trace entry per epoch at negligible cost (1 wei per call). After enough epochs of balance changes without claiming, the loop in `calculate_rewards` will exceed the Starknet gas limit, permanently reverting every future `claim_rewards` call for that pool member and freezing their accumulated yield.

---

### Finding Description

**Root cause — no minimum amount floor in `add_to_delegation_pool`:**

`add_to_delegation_pool` accepts any non-zero amount:

```cairo
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
``` [1](#0-0) 

Every call to `add_to_delegation_pool` (or `exit_delegation_pool_intent`) invokes `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The trace's `insert` logic appends a **new checkpoint** whenever the epoch key differs from the last stored key:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Because the key is always `current_epoch + K`, one call per epoch appends one new entry. There is no deduplication across epochs.

**The unbounded loop — `calculate_rewards`:**

`claim_rewards` calls `calculate_rewards`, which iterates over every trace entry since the last claim:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
``` [3](#0-2) 

Each iteration performs multiple storage reads (`pool_member_trace.at(...)` and `find_sigma`). Once the number of accumulated entries is large enough, the gas cost of the loop exceeds the Starknet block gas limit and the transaction reverts.

**`entry_to_claim_from` does not help once the trace is large enough:**

`entry_to_claim_from` is only advanced when `claim_rewards` **succeeds**:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [4](#0-3) 

If the delegator accumulates N epochs of balance changes without claiming, the loop must process all N entries in a single transaction. Once N is large enough to exceed the gas limit, every subsequent `claim_rewards` call reverts, and there is no mechanism to process the trace in smaller batches.

**`min_delegation_for_rewards` does not prevent trace growth:**

`min_delegation_for_rewards` is only checked inside `compute_rewards_per_unit` to guard against overflow in the reward-per-unit calculation. It is never checked in `add_to_delegation_pool` and does not prevent a 1-wei addition from creating a new trace entry. [5](#0-4) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace for a delegator is large enough that `calculate_rewards` exceeds the gas limit, every call to `claim_rewards` reverts. The delegator's accumulated STRK rewards — already transferred to the pool contract by the staking contract — are permanently inaccessible. `exit_delegation_pool_action` only returns the staked principal; it does not transfer rewards. There is no batch-claim or trace-pruning mechanism. This constitutes **permanent freezing of unclaimed yield** (High impact).

---

### Likelihood Explanation

A delegator calling `add_to_delegation_pool` with 1 wei once per epoch pays only gas. The `min_for_rewards` threshold (1 STRK for STRK pools) applies to the total pool balance, not to individual additions. The attack requires sustained activity over many epochs without claiming rewards, but the per-epoch cost is negligible. It can also occur accidentally for a delegator who frequently adjusts their stake and rarely claims. Likelihood is **Medium**.

---

### Recommendation

Enforce a meaningful minimum amount in `add_to_delegation_pool` (and optionally `enter_delegation_pool`) — analogous to `min_delegation_for_rewards` — so that each balance-change call requires a non-trivial token amount. This bounds the rate at which the trace can grow relative to the rewards earned, making it economically irrational to spam trace entries. Additionally, consider adding a batch-claim mechanism that processes only a bounded window of trace entries per call, advancing `entry_to_claim_from` even on partial completion.

---

### Proof of Concept

```
1. Delegator calls enter_delegation_pool(amount = STRK_CONFIG.min_for_rewards)
   → trace length = 1

2. For epoch i in [1 .. N]:
     advance_epoch()
     delegator calls add_to_delegation_pool(pool_member, amount = 1)
     → trace length = i + 1   (new entry at epoch i + K)

3. After N epochs without claiming:
     delegator calls claim_rewards(pool_member)
     → calculate_rewards loops over N entries
     → each iteration: storage read (pool_member_trace.at) + find_sigma (storage reads)
     → at sufficiently large N, transaction reverts with out-of-gas
     → delegator's accumulated STRK rewards are permanently frozen in the pool contract
```
