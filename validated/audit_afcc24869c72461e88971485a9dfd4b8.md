### Title
Permanent Token Freeze on Failed `ExitToNear` When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is not enabled (which is the case for the default feature set), any ERC-20 or ETH tokens burned during an `ExitToNear` precompile call are permanently destroyed if the subsequent NEP-141 `ft_transfer` promise fails. No callback is attached to detect the failure, and no refund mechanism is invoked, resulting in a permanent, irrecoverable fund freeze for the user.

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) burns EVM-side tokens (ERC-20 or ETH) and then schedules a NEAR-side `ft_transfer` (or `ft_transfer_call`) promise to deliver the corresponding NEP-141 tokens to the recipient. A callback (`exit_to_near_precompile_callback`) is optionally attached to handle failure and re-mint the burned tokens.

The refund field in the callback args is gated entirely behind the `error_refund` compile-time feature: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

When `error_refund` is not enabled and `transfer_near` is also `None` (the common ERC-20 exit case), `callback_args` equals its default value, so **no callback is attached at all**: [2](#0-1) 

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

Even when a callback is attached (e.g., for the wNEAR unwrap path where `transfer_near` is `Some`), the failure branch in `exit_to_near_precompile_callback` silently returns `None` without refunding: [3](#0-2) 

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    ...
} else if let Some(args) = args.refund {   // ← None when error_refund disabled
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← tokens are gone, no refund
};
```

The `error_refund` feature is **not** in the `default` feature set: [4](#0-3) 

```toml
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
```

The test suite explicitly acknowledges the two behaviors: [5](#0-4) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls the `ExitToNear` precompile to bridge ERC-20 tokens (or ETH) back to NEAR:

1. The EVM-side tokens are **burned** immediately and irreversibly.
2. The NEP-141 `ft_transfer` promise is dispatched.
3. If the promise fails (e.g., recipient account not registered with the NEP-141 contract, insufficient storage deposit, paused contract, or any other NEAR-side error), **no refund is issued**.
4. The user's tokens are permanently destroyed — they no longer exist on either side of the bridge.

### Likelihood Explanation

**High.** The failure condition is easily triggered by any user who:
- Specifies a recipient NEAR account that is not registered with the NEP-141 token contract (a common mistake, since NEP-141 requires explicit storage registration).
- Attempts to exit to a NEAR account that has been deleted or does not exist.
- Exits during a period when the NEP-141 contract is paused.

The `ExitToNear` precompile is a core, publicly accessible bridge function reachable by any EVM user via a standard EVM call. No special privileges are required.

### Recommendation

Enable the `error_refund` feature in the production WASM build, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` so that a callback is always attached and failed exits always trigger `refund_on_error`. The refund logic in `engine::refund_on_error` already correctly handles both ERC-20 re-minting and ETH restoration: [6](#0-5) 

The `error_refund` feature should be promoted to a default feature, or the conditional compilation guards should be removed so the refund path is always compiled in.

### Proof of Concept

1. Deploy an ERC-20 token on Aurora backed by a NEP-141 token.
2. As an EVM user, call the `ExitToNear` precompile to bridge ERC-20 tokens to a NEAR account that is **not registered** with the NEP-141 contract.
3. The ERC-20 tokens are burned in the EVM immediately.
4. The `ft_transfer` NEAR promise fails because the recipient is unregistered.
5. Without `error_refund` enabled, no callback is attached (or the callback's `refund` field is `None`), so `refund_on_error` is never called.
6. The user's tokens are permanently gone — the ERC-20 balance is zero and no NEP-141 tokens were delivered.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` demonstrates exactly this scenario and explicitly documents that without `error_refund`, `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` — i.e., the exited tokens are lost. [7](#0-6)
