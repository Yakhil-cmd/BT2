Audit Report

## Title
Off-by-One in SNS Governance `register_vote` Deadline Check Allows Vote at Exact Deadline Second - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS `register_vote` function guards the voting window with `now_seconds > deadline` (strictly greater-than), while the canonical `accepts_vote` helper in `rs/sns/governance/src/proposal.rs` uses `now_seconds < deadline` (strictly less-than). At exactly `now_seconds == deadline`, `register_vote` passes the guard and permanently records the ballot via `cast_vote_and_cascade_follow`, while `accepts_vote` simultaneously returns `false` for the same timestamp. The NNS governance implementation correctly delegates to `accepts_vote` and does not have this inconsistency.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 3914–3922, the deadline guard is:

```rust
let deadline = proposal.get_deadline_timestamp_seconds();
if now_seconds > deadline {
    return Err(...);
}
```

This allows execution to continue when `now_seconds == deadline`. Control then falls through to `cast_vote_and_cascade_follow` (lines 3931–3942) and `process_proposal` (line 3944), permanently writing the ballot and potentially deciding the proposal.

The canonical definition in `rs/sns/governance/src/proposal.rs` lines 2100–2103 is:

```rust
pub fn accepts_vote(&self, now_seconds: u64) -> bool {
    now_seconds < self.get_deadline_timestamp_seconds()
}
```

This returns `false` at `now_seconds == deadline`. The SNS `register_vote` never calls `accepts_vote`; it uses its own inline check with the wrong operator. By contrast, NNS governance at `rs/nns/governance/src/governance.rs` lines 5629–5637 delegates to `proposal.accepts_vote(now_seconds, voting_period_seconds)` and rejects when it returns `false`, closing the window correctly.

No other guard in `register_vote` or `cast_vote_and_cascade_follow` re-checks the deadline, so the late ballot is unconditionally committed.

## Impact Explanation
This matches the **High ($2,000–$10,000)** impact class: *Significant SNS security impact with concrete user or protocol harm.* An unprivileged neuron holder can cast a governance vote after the canonical deadline has expired, with full knowledge of the live tally (tallies are readable via public query calls). In a close vote, this allows a large-stake neuron to observe the outcome and flip it at the last moment — after all other participants believe the window has closed. The late vote is counted in the tally and can decide the proposal, while the attacker may forgo voting rewards (since `accepts_vote` returns `false`), creating an asymmetric incentive for a well-capitalized actor to manipulate high-stakes SNS proposals at zero reward cost.

## Likelihood Explanation
The Internet Computer advances `now_seconds` in discrete consensus rounds (~1–2 seconds each). An attacker can monitor on-chain time via query calls and submit a `manage_neuron::RegisterVote` ingress message in the round where `now_seconds` equals `current_deadline_timestamp_seconds`. No privileged access is required — any principal controlling a neuron with an uncast ballot on the proposal can exploit this. The attack is deterministic, requires no victim interaction, and is repeatable across every SNS deployment.

## Recommendation
Replace the inline `>` comparison in `register_vote` with the canonical `accepts_vote` helper, matching the NNS pattern:

```rust
// Before (buggy):
let deadline = proposal.get_deadline_timestamp_seconds();
if now_seconds > deadline {
    return Err(...);
}

// After (correct):
if !proposal.accepts_vote(now_seconds) {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Proposal deadline has passed.",
    ));
}
```

This eliminates the duplicate deadline logic and ensures a single, consistent definition of "voting period open" is enforced throughout the SNS governance canister.

## Proof of Concept
1. Deploy an SNS with a proposal whose `current_deadline_timestamp_seconds = T`.
2. At time `T − 1`, observe the live tally via a query call showing a close result.
3. At time `T` (the exact deadline second), submit `manage_neuron { RegisterVote { proposal_id, vote: <decisive_vote> } }`.
4. Observe that `register_vote` evaluates `T > T` → `false` → no error is returned.
5. Confirm `cast_vote_and_cascade_follow` records the ballot and `process_proposal` decides the proposal with the late vote included.
6. Confirm `proposal.accepts_vote(T)` returns `false` for the same timestamp, proving the inconsistency.

A deterministic integration test using PocketIC can set the mock time to exactly `T` and assert that `register_vote` succeeds while `accepts_vote` returns `false`.
