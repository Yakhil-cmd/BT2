### Title
Pausing the Contract Mid-Flight Permanently Freezes NEAR Tokens During wNEAR Unwrap Operations — (File: `engine/src/contract_methods/connector.rs`)

### Summary

`exit_to_near_precompile_callback` enforces `require_running` before it executes the NEAR transfer step of a wNEAR unwrap. Because the wNEAR ERC-20 burn and the NEP-141 `near_withdraw` are committed in the base promise before the callback fires, pausing the contract between those two NEAR receipts causes the unwrapped NEAR to be permanently stranded in Aurora's account with no recovery path.

### Finding Description

The wNEAR unwrap flow is a two-step cross-contract sequence:

**Step 1 (base promise — committed atomically with the EVM call):**
- User's EVM transaction calls the wNEAR ERC-20 `withdrawToNear` function.
- The ERC-20 contract burns the user's wNEAR tokens from EVM state.
- The `EXIT_TO_NEAR` precompile creates a NEAR promise to call `near_withdraw` on the wNEAR NEP-141 contract, which burns the NEP-141 wNEAR and credits the unwrapped NEAR to Aurora's account.

**Step 2 (callback — separate NEAR receipt):**
- `exit_to_near_precompile_callback` is invoked.
- If `near_withdraw` succeeded, it transfers the unwrapped NEAR from Aurora's account to the user's target NEAR account.

The callback is defined as:

```rust
// engine/src/contract_methods/connector.rs

    io: I, env: &E, handler: &mut H,

    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // ← blocks if contract is paused
        env.assert_private_call()?;
        ...

            if let Some(args) = args.transfer_near {
                // Transfer unwrapped NEAR to user — never reached if paused
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }
            None
        } else if let Some(args) = args.refund {
            // Refund ERC-20 tokens — never reached if paused
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
            ...
        }
```

If the owner calls `pause_contract` between the `near_withdraw` receipt and the `exit_to_near_precompile_callback` receipt, `require_running` returns `ERR_PAUSED`, the callback panics, and the NEAR transfer to the user is never scheduled. Because NEAR promise results are not rolled back on callback failure, the NEAR that was already credited to Aurora's account in Step 1 is permanently stranded there.

The `pause_contract` function itself performs no accounting for in-flight two-step operations:

```rust
// engine/src/contract_methods/admin.rs
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
```

There is no mechanism to enumerate or drain pending callbacks before the pause takes effect, and no recovery function exists to reclaim stranded NEAR after the fact.

This is structurally identical to the reported `VoterV3` bug: an admin action (killing a gauge / pausing the contract) does not account for the in-flight first step of a two-step operation, leaving the second step's funds permanently inaccessible.

The CHANGES.md entry for v3.6.2 confirms the team already recognised this class of problem for `ft_resolve_transfer`:

> "The `ft_resolve_transfer` callback doesn't require running the contract to finish the `ft_transfer_call` correctly" ([#906])

The same fix was never applied to `exit_to_near_precompile_callback`.

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the race condition is triggered:
- The user's wNEAR ERC-20 tokens are irreversibly burned from EVM state.
- The NEP-141 wNEAR is irreversibly burned by `near_withdraw`.
- The resulting NEAR is credited to Aurora's account but can never be forwarded to the user.
- No on-chain recovery path exists; the NEAR is permanently frozen inside the Aurora contract.

### Likelihood Explanation

The owner may legitimately pause the contract for emergency maintenance or upgrades. Any wNEAR unwrap operation whose `near_withdraw` receipt lands in a block before the pause, but whose `exit_to_near_precompile_callback` receipt lands in a block after the pause, triggers the freeze. Because NEAR processes receipts across multiple blocks, this window is real and non-negligible whenever the contract is paused under load. The owner has no tooling to detect or drain pending callbacks before pausing.

### Recommendation

Remove `require_running` from `exit_to_near_precompile_callback`, mirroring the fix already applied to `ft_resolve_transfer` in [#906]. The callback must always be allowed to complete regardless of the contract's pause state, because the base promise's side-effects (wNEAR burn, NEAR credit to Aurora) are already irreversible by the time the callback fires.

Alternatively, add a pre-pause drain step that verifies no `exit_to_near_precompile_callback` receipts are pending before setting `is_paused = true`.

### Proof of Concept

1. User calls the wNEAR ERC-20 `withdrawToNear(recipient, amount)` from an EVM contract. This burns `amount` wNEAR ERC-20 tokens and schedules a NEAR receipt to call `near_withdraw` on the wNEAR NEP-141 contract, with `exit_to_near_precompile_callback` as the callback.

2. The `near_withdraw` receipt executes successfully in block N: NEP-141 wNEAR burned, `amount` NEAR credited to Aurora's account.

3. The owner calls `pause_contract` in block N (or N+1 before the callback receipt is processed).

4. The `exit_to_near_precompile_callback` receipt executes in block N+1 (or later). `require_running` returns `ERR_PAUSED`; the callback panics.

5. The NEAR transfer to the user is never scheduled. The `amount` NEAR is permanently frozen in Aurora's account. The user has lost their wNEAR with no recourse.
