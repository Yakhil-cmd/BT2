### Title
Ethereum Transaction Flow `precharge_fee` Silently Drops `SIMULATION` Flag, Causing Pubdata Underestimation in Gas Estimation - (`basic_bootloader/src/bootloader/transaction_flow/ethereum/mod.rs`)

---

### Summary

The Ethereum-type transaction flow's `precharge_fee` function hardcodes `false` for the `fee_payment_in_simulation` parameter when calling `update_account_nominal_token_balance`, instead of forwarding `Config::SIMULATION`. The ZK transaction flow's `precharge_fee` correctly passes `Config::SIMULATION`. This inconsistency means that when simulation mode is active for Ethereum-type transactions (EIP-1559, EIP-2930, legacy), the `not_compress_balance` flag is never set on the sender's account, causing pubdata to be underestimated during gas estimation. Transactions that pass simulation may then run out of gas for pubdata during actual execution.

---

### Finding Description

The system has a `fee_payment_in_simulation` flag that propagates through balance updates to set `not_compress_balance` on account metadata. This flag exists specifically to ensure simulation never underestimates pubdata: when `gas_price = 0` in simulation, fee precharge produces no balance diff, so without the flag the pubdata estimator would not count the sender's balance change. In actual execution with `gas_price > 0`, the fee precharge does produce a balance diff that must be published as pubdata.

**ZK flow — correct:** [1](#0-0) 

The ZK flow passes `Config::SIMULATION` as the `fee_payment_in_simulation` argument.

**Ethereum flow — broken:** [2](#0-1) 

The Ethereum flow hardcodes `false`, regardless of whether `Config::SIMULATION` is `true`.

The `fee_payment_in_simulation` flag flows down through the storage layer: [3](#0-2) 

In the flat storage model, it sets `not_compress_balance` on the account: [4](#0-3) 

That flag is then consumed by `calculate_pubdata_used_by_tx` to force worst-case pubdata accounting: [5](#0-4) 

Because the Ethereum flow passes `false`, `not_compress_balance` is never set for the sender's account during fee precharge in simulation, so `calculate_pubdata_used_by_tx` silently omits the sender's balance-change pubdata cost from the estimate.

The simulation entry point uses `BasicBootloaderCallSimulationConfig` (where `SIMULATION = true`): [6](#0-5) [7](#0-6) 

---

### Impact Explanation

When a user or wallet calls `simulate_tx` with an Ethereum-type transaction (EIP-1559, EIP-2930, legacy):

1. Simulation runs with `gas_price = 0` (typical for `eth_estimateGas`).
2. Fee precharge produces no balance diff → no pubdata counted for sender balance.
3. `not_compress_balance` is never set → `calculate_pubdata_used_by_tx` returns a value that omits the sender's fee-related balance change.
4. The returned gas estimate is too low by the pubdata cost of the sender's balance diff.
5. The user submits the actual transaction with the underestimated gas limit.
6. During actual execution with `gas_price > 0`, fee precharge produces a real balance diff that must be published as pubdata.
7. The transaction runs out of gas for pubdata and fails, wasting the user's gas fees.

This is a **resource accounting bug** with direct user-fund impact: gas fees are consumed but the transaction fails. The `not_compress_balance` mechanism was introduced precisely to prevent this class of simulation/execution divergence, as evidenced by the regression test: [8](#0-7) 

That test only covers the ZK flow. The Ethereum flow has no equivalent protection.

---

### Likelihood Explanation

- EIP-1559 and legacy transactions are the dominant transaction type for any EVM-compatible chain.
- `eth_estimateGas` / `eth_call` are called by every wallet and dApp before submitting a transaction.
- The bug is triggered whenever `gas_price > 0` in actual execution after a simulation with `gas_price = 0`, which is the standard pattern.
- No special attacker capability is required; any unprivileged user submitting a standard Ethereum transaction through the normal RPC path is affected.

---

### Recommendation

In `basic_bootloader/src/bootloader/transaction_flow/ethereum/mod.rs`, change the hardcoded `false` in `precharge_fee` to `Config::SIMULATION`:

```rust
system.io.update_account_nominal_token_balance(
    ExecutionEnvironmentType::NoEE,
    resources,
    from,
    &value,
    true,
    Config::SIMULATION,  // was: false
)
```

Apply the same fix to any refund or operator-payment balance updates in the Ethereum flow that also hardcode `false` where `Config::SIMULATION` should be used, mirroring the pattern already established in the ZK flow: [9](#0-8)

### Proof of Concept

**Step 1 — Observe the inconsistency:**

ZK flow passes `Config::SIMULATION`: [10](#0-9) 

Ethereum flow hardcodes `false`: [2](#0-1) 

**Step 2 — Trace the missing flag:**

`fee_payment_in_simulation = false` → `update_nominal_token_value_inner` → `m.not_compress_balance |= false` (no-op): [11](#0-10) 

**Step 3 — Pubdata underestimation:**

`calculate_pubdata_used_by_tx` skips the sender's balance entry because `not_compress_balance` is `false` and the balance is unchanged (gas_price=0 in simulation): [12](#0-11) 

**Step 4 — Divergence at execution time:**

With `gas_price > 0` in actual execution, the sender's balance decreases by the fee. This diff must be published as pubdata, but the gas limit was set based on the underestimated simulation result, causing the transaction to run out of gas for pubdata.

/// Regression for an issue where the lack of fee payment during
/// simulation resulted in underestimated pubdata length, and thus
/// underestimated gas usage.
#[test]
fn test_simulation_gas_used_regression() {
    let mut tester = TestingFramework::new();
    let wallet = tester.random_signer();
    let target_address = common_target_address();
    tester = tester.with_balance(wallet.address(), U256::MAX);

    // First tx, 0 gas price.
    let tx = {
        let tx = TxLegacy {
            chain_id: 37u64.into(),
            nonce: 0,
            gas_price: 0,
            gas_limit: 2_000_000,
            to: TxKind::Call(target_address),
            value: U256::ZERO,
            input: Default::default(),
        };
        ZKsyncTxEnvelope::from_eth_tx(tx, wallet.clone())
    };
    let block_context = BlockContext {
        eip1559_basefee: U256::from(91161500u64),
        native_price: U256::from(911615u64),
        pubdata_price: U256::from(10303657632u64 * 4),
        ..Default::default()
    };
    tester = tester.with_block_context(block_context.clone());
    let result_simulation = tester.simulate_block(vec![tx.clone()]);
    let first_tx = result_simulation.tx_results[0]
        .clone()
        .expect("Must succeed");

    // Second tx, realistic gas price
    let tx = {
        let tx = TxLegacy {
            chain_id: 37u64.into(),
            nonce: 0,
            gas_price: 91161500,
            gas_limit: 2_000_000,
            to: TxKind::Call(target_address),
            value: U256::ZERO,
            input: Default::default(),
        };
        ZKsyncTxEnvelope::from_eth_tx(tx, wallet.clone())
    };

    tester = tester.with_block_context(block_context);
    let result_simulation = tester.simulate_block(vec![tx.clone()]);
    let second_tx = result_simulation.tx_results[0]
        .clone()
        .expect("Must succeed");
    assert_eq!(first_tx.gas_used, second_tx.gas_used);
    assert_eq!(first_tx.native_used, second_tx.native_used);
    assert_eq!(first_tx.pubdata_used, second_tx.pubdata_used);
}
