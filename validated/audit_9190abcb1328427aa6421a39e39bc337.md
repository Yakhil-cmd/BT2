### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures When Deploy-Account Is in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function unconditionally bypasses the `__validate__` entry-point (signature verification) for any invoke transaction with nonce=1 whenever the sender account has *any* transaction visible in the mempool. An unprivileged attacker who observes a legitimate deploy-account transaction in the public mempool can immediately inject an invoke transaction with an arbitrary/invalid signature for that account. The gateway accepts it without signature verification, admitting an unauthorized transaction to the mempool.

---

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, `skip_stateful_validations` returns `true` when all four conditions hold: [1](#0-0) 

1. Transaction type is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `!tx.execution_flags.validate`, execution returns `Ok(())` immediately after `perform_pre_validation_stage` — `__validate__` is never called: [3](#0-2) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` passes only nonce, tip, and gas-price fields — **no signature field exists in `ValidationArgs`**: [4](#0-3) 

The guard condition `account_tx_in_pool_or_recent_block` is too broad: it returns `true` if the account has **any** transaction in the pool or recent-block state, not specifically a deploy-account: [5](#0-4) 

**Attack path:**

1. Legitimate owner pre-funds address A with STRK and submits a deploy-account transaction.
2. Attacker observes the deploy-account in the public mempool for address A.
3. Attacker submits an invoke transaction with nonce=1 for address A with an invalid/arbitrary signature.
4. Gateway stateful validation:
   - `account_nonce = 0` → nonce check passes (1 ≥ 0)
   - `validate_by_mempool` → no signature field, passes
   - `skip_stateful_validations` → nonce=1, account_nonce=0, deploy-account in pool → returns `true`
   - `run_validate_entry_point` → `validate = false` → `__validate__` **not called**
5. Invalid invoke transaction accepted into the mempool.
6. Batcher executes deploy-account first, then the invalid invoke: `__validate__` runs and fails (invalid signature), transaction reverts.
7. Address A pays fees for the reverted, unauthorized transaction.
8. Legitimate user's intended nonce=1 invoke is blocked (nonce slot occupied by attacker's tx).

The full gateway flow showing where the nonce is returned and the transaction forwarded to the mempool without signature verification: [6](#0-5) 

---

### Impact Explanation

- **Admission invariant broken**: the gateway accepts an invoke transaction whose signature has never been verified.
- **Nonce-slot griefing**: the attacker occupies the nonce=1 slot for any pre-funded, not-yet-deployed account whose deploy-account is visible in the mempool. The legitimate user must either wait for the revert or pay a fee-escalation premium to replace the attacker's transaction.
- **Unauthorized fee drain**: when the batcher executes the invalid invoke, it reverts and address A pays fees for a transaction it never authorized.
- Matches impact: **High — Mempool/gateway admission accepts invalid transactions before sequencing**.

---

### Likelihood Explanation

Medium. The attacker only needs to monitor the public mempool for deploy-account transactions and submit an invoke with nonce=1 before the deploy-account is sequenced. No privileges, keys, or special knowledge are required.

---

### Recommendation

`skip_stateful_validations` must verify that the account specifically has a **deploy-account** transaction in the mempool, not merely any transaction. The current `account_tx_in_pool_or_recent_block` check is insufficient. Options:

1. Add a dedicated mempool query `has_deploy_account_in_pool(address)` and use it instead.
2. Alternatively, remove the `skip_stateful_validations` UX shortcut entirely and require users to wait for the deploy-account to be confirmed before submitting invoke transactions.

---

### Proof of Concept

```
1. Pre-fund address A with STRK (e.g., transfer from another account).
2. Submit deploy_account tx for address A (legitimate owner).
3. Attacker submits invoke tx:
     sender_address = A
     nonce          = 1
     signature      = [0x0, 0x0]   // invalid
     resource_bounds = <sufficient to pass gateway fee check>
4. Gateway accepts the invoke tx (skip_stateful_validations returns true).
5. Batcher executes deploy_account, then the invalid invoke.
6. Invalid invoke reverts; address A is charged fees.
7. Legitimate nonce=1 invoke from the real owner is rejected as DuplicateNonce.
``` [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
