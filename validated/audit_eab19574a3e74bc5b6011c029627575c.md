### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` on pool-level admin spread fees but applies **no equivalent cap** to per-bin additional fees set via `setPoolBinAdditionalFees`. A pool admin can set `addFeeBuyE6`/`addFeeSellE6` to any value up to `uint16` max (65,535 = 6.5535% in E6 units) on any bin, charging traders more than the protocol-enforced cap permits.

---

### Finding Description

`setPoolAdminFees` correctly enforces the cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` performs **no cap check** at all — it passes the caller-supplied values directly to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

In the swap path, the bin additional fee is added directly on top of the base spread fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [3](#0-2) 

The pool-level `spreadFeeE6` (which is `protocolSpreadFeeE6 + adminSpreadFeeE6`) is bounded by the factory caps. The bin additional fee is **not** bounded by any factory cap — only by the `uint16` type (max 65,535 = 6.5535%). This means:

1. **When the owner lowers `maxAdminSpreadFeeE6` below 65,535** (e.g., to 0 to restrict pool admins), the pool admin can still charge up to 6.5535% per bin via `setPoolBinAdditionalFees`, completely bypassing the cap.

2. **Even at the default cap** (`HARD_MAX_SPREAD_FEE_E6 = 200,000`), a pool admin who has already set `adminSpreadFeeE6 = maxAdminSpreadFeeE6` can additionally set bin fees of up to 65,535, making the total effective admin-controlled fee 26.5535% for that bin — exceeding the hard cap. [4](#0-3) 

---

### Impact Explanation

Traders swapping in a bin where the pool admin has set high additional fees pay more than the protocol-enforced cap allows. This is a **direct loss of user funds** — excess fees are extracted by the pool admin beyond what the cap system is designed to permit. The cap system exists precisely to protect users from semi-trusted pool admins overcharging; this bypass nullifies that protection for per-bin fees.

---

### Likelihood Explanation

Any pool admin can trigger this at any time by calling `setPoolBinAdditionalFees` with `addFeeBuyE6` or `addFeeSellE6` set to values exceeding `maxAdminSpreadFeeE6`. No special conditions are required beyond being the pool admin. The impact is highest when the factory owner has lowered `maxAdminSpreadFeeE6` to restrict pool admins, since the bypass is then most pronounced.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce a combined cap so that `adminSpreadFeeE6 + addFeeBuyE6 ≤ maxAdminSpreadFeeE6` for any bin.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(200_000, 1_000, 1_000_000, 1_000_000)` — lowering `maxAdminSpreadFeeE6` to 1,000 (0.1%) to restrict pool admins.
2. Pool admin calls `setPoolAdminFees(pool, 1_001, 0)` → **reverts** with `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` → **succeeds**. No cap check.
4. Traders swapping in bin 0 now pay 6.5535% additional fee — 65× the intended 0.1% cap — with no revert or guard. [2](#0-1)
