### Title
Unchecked Precompile Return Value in `EvmErc20.withdrawToNear` Enables Silent Token Destruction Without Bridge Promise — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens before invoking the `ExitToNear` precompile, and never checks the assembly `call` return value. Passing an empty `recipient` causes the precompile to fail with `ERR_INVALID_RECEIVER_ACCOUNT_ID` after the burn has already committed. Because the Solidity function ignores `res`, the outer transaction succeeds, leaving the tokens permanently destroyed with no corresponding NEP-141 credit on NEAR.

---

### Finding Description

**Step 1 — Burn before precompile, return value ignored**

In `EvmErc20.withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // (A) burn first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // (B) res is NEVER checked
    }
}
``` [1](#0-0) 

There is no `pausable` modifier on `withdrawToNear`, and `AdminControlled` provides no input validation on `recipient`. [2](#0-1) 

**Step 2 — Empty recipient passes `validate_input_size` but fails `parse_recipient`**

With `recipient = bytes('')`, `input_size = 33` (flag=1 + amount=32). The precompile's `parse_input` enforces `MIN_INPUT_SIZE = 3` (without `error_refund` feature), so 33 bytes passes: [3](#0-2) [4](#0-3) 

After stripping the flag byte, `try_from` reaches the ERC-20 branch:

```rust
0x1 => {
    let amount = parse_amount(&input[..32])?;       // succeeds
    let Recipient { .. } = parse_recipient(&input[32..])?;  // empty slice → fails
``` [5](#0-4) 

`parse_recipient(&[])` converts the empty slice to `""` via `str::from_utf8` (succeeds), then attempts `"".parse::<AccountId>()`, which fails because an empty string is not a valid NEAR account ID: [6](#0-5) 

The precompile returns `Err(ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))`.

**Step 3 — EVM semantics: failed subcall does not revert caller**

Standard EVM behavior (which Aurora's executor follows) treats a failed precompile call as a failed subcall: the `call` opcode returns `0` to the caller, but the caller's own state changes — including the already-committed `_burn` — are **not** rolled back. Since `res` is never inspected, `withdrawToNear` returns normally. The NEAR-side bridge promise log is never emitted. [7](#0-6) 

---

### Impact Explanation

The ERC-20 token supply is reduced by `amount` on the Aurora/EVM side, but no `ft_transfer` or `ft_transfer_call` promise is scheduled on the NEAR side. The NEP-141 balance held by the Aurora engine contract is unchanged. The burned tokens are irrecoverable: they no longer exist in the EVM state and were never credited on NEAR. This constitutes **permanent freezing (destruction) of funds**.

---

### Likelihood Explanation

- No special privilege is required; any token holder can call `withdrawToNear`.
- The trigger is a single call with `recipient = bytes('')` and any nonzero `amount`.
- No admin action, governance vote, or external dependency is needed.
- The function has no `pausable` guard and no Solidity-level input validation on `recipient`.
- The path is fully deterministic and locally reproducible.

---

### Recommendation

1. **Check the precompile return value** and revert if it is `0`:
   ```solidity
   assembly {
       let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                       0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Validate `recipient` before burning**: require `recipient.length > 0` (and optionally enforce a maximum length) before calling `_burn`.
3. **Invert the order**: call the precompile first (in a view/static simulation or with a pre-check), and only burn if the precompile call would succeed. Alternatively, use a two-phase commit pattern.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IEvmErc20 {
    function withdrawToNear(bytes memory recipient, uint256 amount) external;
    function balanceOf(address account) external view returns (uint256);
}

contract PoC {
    function exploit(IEvmErc20 token, uint256 amount) external {
        uint256 before = token.balanceOf(address(this));
        // Call with empty recipient — burn succeeds, precompile fails silently
        token.withdrawToNear(bytes(""), amount);
        uint256 after_ = token.balanceOf(address(this));
        // assert: before - after_ == amount (tokens burned)
        // assert: no NEP-141 balance increase on NEAR side
        require(before - after_ == amount, "tokens not burned");
        // Tokens are permanently lost
    }
}
```

Deploy `EvmErc20` on a local Aurora sandbox, mint tokens to `PoC`, call `exploit`. Confirm ERC-20 balance decreases by `amount` and no NEAR-side `ft_transfer` promise was scheduled (observable via the absence of the `ExitToNear` event log). [1](#0-0) [6](#0-5) [5](#0-4)
