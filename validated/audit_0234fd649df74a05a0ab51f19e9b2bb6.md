### Title
Push-Based Token Transfers in `collectFees` Block Fee Collection and Fee-Rate Updates When `adminFeeDestination` Is Blacklisted — (`File: metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.collectFees` pushes tokens directly to `adminFeeDestination` and `FACTORY` via `safeTransfer`. Both `MetricOmmPoolFactory.setPoolAdminFees` and `MetricOmmPoolFactory.setPoolProtocolFee` call `collectFees` as a mandatory prerequisite before writing new fee rates. If either push transfer reverts (e.g., USDC/USDT blacklists `adminFeeDestination`), all three entry-points — `collectPoolFees`, `setPoolAdminFees`, and `setPoolProtocolFee` — revert permanently until the pool admin manually rotates the destination. The protocol owner has no independent path to unblock `setPoolProtocolFee`.

---

### Finding Description

`MetricOmmPool.collectFees` (lines 416–427) performs four conditional `safeTransfer` calls:

```solidity
if (totalFee0ToAdmin > 0) transferToken0(adminFeeDestination_, totalFee0ToAdmin);
if (totalFee1ToAdmin > 0) transferToken1(adminFeeDestination_, totalFee1ToAdmin);
if (totalFee0ToProtocol > 0) transferToken0(FACTORY, totalFee0ToProtocol);
if (totalFee1ToProtocol > 0) transferToken1(FACTORY, totalFee1ToProtocol);
```

`transferToken0/1` call `IERC20.safeTransfer`, which reverts on failure. [1](#0-0) [2](#0-1) 

`MetricOmmPoolFactory.setPoolProtocolFee` (owner-only) calls `collectFees` before writing the new rate:

```solidity
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← push target
);
// ... then updates poolFeeConfig and calls setPoolFees
``` [3](#0-2) 

`MetricOmmPoolFactory.setPoolAdminFees` (pool-admin-only) has the identical pattern: [4](#0-3) 

The permissionless `collectPoolFees` also routes through `collectFees`: [5](#0-4) 

If `adminFeeDestination` is added to the USDC or USDT blacklist after pool deployment, every call to `collectFees` that computes a non-zero admin share reverts. Because `setPoolProtocolFee` and `setPoolAdminFees` both embed `collectFees` atomically, they also revert. The only escape valve is `setPoolAdminFeeDestination`, which does **not** call `collectFees`: [6](#0-5) 

However, `setPoolAdminFeeDestination` is callable only by the pool admin. The factory owner (protocol) has no independent path to change `adminFeeDestination` and therefore cannot unblock `setPoolProtocolFee` without pool-admin cooperation.

---

### Impact Explanation

1. **Protocol fee loss / stuck funds**: `collectPoolFees` is permissionless and is the normal operational path for extracting accrued fees. While the pool is blocked, `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` continue to accumulate but cannot be flushed. Spread-fee surplus also grows. All accrued protocol and admin fees are trapped in the pool contract for the duration of the block.

2. **Broken admin-boundary**: `setPoolProtocolFee` (factory owner) is gated behind a push transfer to an address the owner cannot change. The protocol owner's ability to adjust per-pool protocol fees — a core governance action — is held hostage by the pool admin's fee destination state.

3. **`setPoolAdminFees` blocked**: The pool admin cannot change fee rates without first calling `setPoolAdminFeeDestination` to rotate to a non-blacklisted address, which may not be obvious and requires an additional transaction.

---

### Likelihood Explanation

- USDC and USDT are the most common pool tokens in oracle-based AMMs. Circle and Tether both maintain on-chain blacklists and have historically blacklisted addresses linked to sanctions or exploits.
- `adminFeeDestination` is set at pool creation and can be any address (multisig, DAO treasury, custodian). Any of these can be blacklisted post-deployment.
- No validation at creation time checks whether `adminFeeDestination` can receive the pool's specific tokens.
- The trigger requires no privileged attacker — it is an external action by a token issuer (Circle/Tether) on a legitimately-set address.

---

### Recommendation

Adopt a pull-pattern for fee distribution, mirroring the fix described in the external report:

1. In `collectFees`, instead of pushing tokens to `adminFeeDestination` and `FACTORY`, credit internal balance mappings (e.g., `claimableAdminFee0[pool]`, `claimableProtocolFee0`).
2. Add separate `withdrawAdminFees(pool)` and `withdrawProtocolFees()` functions that pull from those mappings. A failed withdrawal for one recipient does not affect the other.
3. This decouples fee-rate management (`setPoolAdminFees`, `setPoolProtocolFee`) from token delivery, so a blacklisted destination never blocks governance operations.

---

### Proof of Concept

```
Setup:
  - Pool with token0 = USDC, token1 = WETH
  - adminFeeDestination = 0xABCD (e.g., a DAO treasury)
  - adminSpreadFeeE6 > 0, so collectFees will compute totalFee0ToAdmin > 0

Step 1: Swaps occur; spread fees accumulate as surplus in the pool.

Step 2: Circle blacklists 0xABCD (e.g., OFAC sanction).

Step 3: Anyone calls factory.collectPoolFees(pool)
  → pool.collectFees(..., adminFeeDestination=0xABCD)
  → transferToken0(0xABCD, totalFee0ToAdmin)
  → USDC.safeTransfer(0xABCD, amount)  ← REVERTS (blacklisted)
  → collectPoolFees reverts. Fees stuck.

Step 4: Factory owner calls factory.setPoolProtocolFee(pool, newFee, 0)
  → same collectFees call → same revert.
  → Protocol owner cannot change protocol fee rate. Permanently blocked
    until pool admin calls setPoolAdminFeeDestination.

Step 5: Pool admin calls factory.setPoolAdminFees(pool, newFee, 0)
  → same collectFees call → same revert.
  → Admin cannot change their own fee rate either.

Resolution path: pool admin must first call setPoolAdminFeeDestination(pool, newDest),
then retry. Protocol owner has no independent path.
```
