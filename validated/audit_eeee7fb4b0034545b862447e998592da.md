### Title
Staged Upgrade Code Persists After Owner Change, Enabling Malicious Contract Deployment — (File: `engine/src/lib.rs`)

---

### Summary

The Aurora Engine's two-phase upgrade mechanism (`stage_upgrade` / `deploy_upgrade`) stores upgrade bytecode in contract storage when the owner stages it, but **never clears that code when ownership is transferred**. Because `deploy_upgrade` has **no caller authorization check**, anyone—including a former owner or an attacker who compromised the former owner's key—can deploy the stale staged bytecode after the delay expires, even after ownership was changed specifically to revoke that authority.

---

### Finding Description

**Phase 1 — `stage_upgrade`** requires `require_owner_only` and writes the upgrade bytecode to `CODE_KEY` and the deployment block height to `CODE_STAGE_KEY`: [1](#0-0) 

**Phase 2 — `deploy_upgrade`** has **no owner or caller check**. It only verifies `require_running` and that the current block height exceeds the staged index, then unconditionally deploys whatever is stored in `CODE_KEY`: [2](#0-1) 

This was an intentional design decision documented in the changelog: [3](#0-2) 

**`set_owner`** updates the owner in state but does **not** clear `CODE_KEY` or `CODE_STAGE_KEY`: [4](#0-3) 

There is no `cancel_upgrade` or equivalent function anywhere in the codebase.

**Analog to the external report — attack path:**

1. The current owner's key is compromised. The attacker calls `stage_upgrade(malicious_bytecode)`. The bytecode is written to `CODE_KEY`; the deployment block height (`current_block + upgrade_delay_blocks`) is written to `CODE_STAGE_KEY`.
2. The legitimate owner detects the compromise and calls `set_owner(new_owner)` to revoke the attacker's authority.
3. `set_owner` updates `state.owner_id` but leaves `CODE_KEY` and `CODE_STAGE_KEY` untouched. The malicious staged code persists.
4. After `upgrade_delay_blocks` blocks pass, the attacker (or anyone) calls `deploy_upgrade()`. The function passes all checks and calls `Runtime::self_deploy(&CODE_KEY)`, deploying the malicious bytecode to the Aurora Engine contract account. [5](#0-4) 

The new owner's only recourse is to race-call `stage_upgrade` with safe code to overwrite `CODE_KEY` before the delay expires. If `upgrade_delay_blocks` is 0 or very small, this window is insufficient or nonexistent.

---

### Impact Explanation

Deploying arbitrary malicious bytecode to the Aurora Engine contract gives the attacker full control over all contract logic. This enables:

- **Direct theft** of all ETH and ERC-20 tokens held by the ETH connector bridge.
- **Permanent freezing** of all user funds.

Impact: **Critical**.

---

### Likelihood Explanation

The attack requires a compromised owner key (or a malicious outgoing owner who knows they are being replaced). For a high-value bridge contract like Aurora Engine, owner key compromise is a realistic threat. The `upgrade_delay_blocks` parameter provides a detection window, but:

- If `upgrade_delay_blocks` is 0, the attacker can call `stage_upgrade` and `deploy_upgrade` in the same block, before the new owner can react.
- If the delay is short, the race window for the new owner to overwrite `CODE_KEY` may be too narrow.
- The intentional removal of the owner check from `deploy_upgrade` eliminates the last line of defense after an ownership transfer. [6](#0-5) 

---

### Recommendation

1. **Add a `cancel_upgrade` function** (owner-only) that clears `CODE_KEY` and `CODE_STAGE_KEY` from storage.
2. **Clear staged upgrade code inside `set_owner`**: when ownership transfers, remove any pending staged code so the new owner starts with a clean slate.
3. **Add an owner check to `deploy_upgrade`**, or require the current owner to explicitly re-authorize deployment after an ownership change.

---

### Proof of Concept

```
Block N:   attacker (compromised key) calls stage_upgrade(malicious_bytecode)
           → CODE_KEY = malicious_bytecode
           → CODE_STAGE_KEY = N + upgrade_delay_blocks

Block N+1: legitimate owner calls set_owner(new_owner)
           → state.owner_id = new_owner
           → CODE_KEY unchanged, CODE_STAGE_KEY unchanged

Block N+D: attacker calls deploy_upgrade()
           → require_running: passes
           → block_height (N+D) > CODE_STAGE_KEY (N+D): passes
           → Runtime::self_deploy(&CODE_KEY) deploys malicious_bytecode
           → Aurora Engine is now under attacker control
           → all bridged ETH and ERC-20 tokens are stolen / frozen
```

The root cause is structurally identical to the external report: an authority (`owner`) is stored at action-initiation time (`stage_upgrade`), the authority is later changed (`set_owner`), but the previously authorized pending action (`CODE_KEY`) is never revoked, and the finalization step (`deploy_upgrade`) does not re-verify the current authority. [7](#0-6)

pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;

        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;
        state::set_state(&mut io, &state)?;

        Ok(())
    })
}

#[named]
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}

    #[unsafe(no_mangle)]
    pub extern "C" fn deploy_upgrade() {
        // This function is intentionally not implemented in `contract_methods`
        // because it only makes sense in the context of the NEAR runtime.
        let mut io = Runtime;
        let state = state::get_state(&io).sdk_unwrap();
        require_running(&state)
            .map_err(ContractError::msg)
            .sdk_unwrap();
        let index = internal_get_upgrade_index();
        if io.block_height() <= index {
            sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
        }
        Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
    }

    /// Deploy code from given key in place of the current contract.
    /// Not implemented in terms of higher level traits (e.g. IO) for efficiency reasons.
    pub fn self_deploy(code_key: &[u8]) {
        unsafe {
            // Load current account id into register 0.
            exports::current_account_id(0);
            // Use register 0 as the destination for the promise.
            let promise_id = exports::promise_batch_create(u64::MAX as _, 0);
            // Remove code from storage and store it in register 1.
            // if the code_key exists call `promise_batch_action_deploy_contract`, otherwise panic with error.
            match exports::storage_remove(code_key.len() as _, code_key.as_ptr() as _, 1) {
                1 => {
                    exports::promise_batch_action_deploy_contract(promise_id, u64::MAX, 1);
                    Self::promise_batch_action_function_call(
                        promise_id,
                        b"state_migration",
                        &[],
                        0,
                        Self::GAS_FOR_STATE_MIGRATION.as_u64(),
                    );
                }
                _ => panic!("ERR_NO_CODE_STAGED"),
            }
        }
    }
