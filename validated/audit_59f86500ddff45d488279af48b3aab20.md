### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` Due to Missing `evmAddress` Binding in `signedData` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of signatures is valid over `signedData` and that the COA resource at the given capability path matches the supplied `evmAddress`, but it does not enforce that `signedData` commits to `evmAddress`. A signature produced to prove ownership of one COA (COA_A) belonging to a Cadence account can be replayed verbatim to prove ownership of any other COA (COA_B) belonging to the same Cadence account, because the only binding between the signature and the target COA is the `evmAddress` field — which is supplied by the caller, not covered by the signature.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verify that the supplied signatures are valid over `signedData` using the Cadence account's keys.
2. Verify that the COA resource at `path` in account `address` has an EVM address equal to `evmAddress`. [1](#0-0) 

The function itself documents the gap:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [2](#0-1) 

Because `signedData` is never checked to contain `evmAddress`, the two checks are entirely decoupled. An attacker who possesses a valid `COAOwnershipProof` for COA_A (i.e., `{Address: alice, CapabilityPath: "coa_a", KeyIndices: [0], Signatures: [sig_D]}` over data `D`) can construct a modified proof `{Address: alice, CapabilityPath: "coa_b", KeyIndices: [0], Signatures: [sig_D]}` and call `verifyCOAOwnershipProof(COA_B_evm_address, D, modified_proof)`. The function will:

- Accept `sig_D` as valid over `D` for `alice`'s key 0 (unchanged).
- Accept that the COA at `alice`'s `/public/coa_b` has EVM address `COA_B_evm_address` (true by construction).
- Return `isValid: true`.

The `COA.isValidSignature` ERC-1271 implementation in `fvm/evm/handler/coa/coa.sol` passes `address(this)` as the `evmAddress` argument: [3](#0-2) 

This means any EVM contract that calls `COA_B.isValidSignature(H, proof_for_coa_b)` will receive `ValidERC1271Signature` for a hash `H` that was only ever signed in the context of COA_A.

The precompile entry point is `verifyCOAOwnershipProof(address,bytes32,bytes)` on the Cadence Arch contract, callable by any EVM transaction or contract: [4](#0-3) 

The Go-side decoder `DecodeABIEncodedProof` passes the caller-supplied `address` field directly as `EVMAddress` into `COAOwnershipProofInContext` without any binding check: [5](#0-4) 

---

### Impact Explanation

An attacker who obtains a valid COA ownership proof for any one COA belonging to a Cadence account can replay it to impersonate any other COA belonging to the same account in ERC-1271 signature verification contexts. Concretely:

- EVM smart contracts that gate privileged actions behind `isValidSignature` (e.g., Safe multisig, ERC-4337 account abstraction, NFT marketplaces, DeFi protocols) will accept the replayed proof as a valid signature from COA_B even though the account owner only ever signed for COA_A.
- This constitutes unauthorized authorization: an attacker can forge a valid ERC-1271 response for a COA they do not control, as long as they have any prior valid proof for a sibling COA of the same Cadence account.

---

### Likelihood Explanation

The preconditions are:
1. The victim Cadence account holds at least two COAs (not enforced by the protocol; the comment acknowledges this is currently uncommon but not impossible).
2. The attacker has obtained a valid `COAOwnershipProof` for one of those COAs (e.g., intercepted from a prior authentication flow, a public on-chain proof, or a proof the victim shared with a different dApp).

Both conditions are realistic as multi-COA accounts become more common and as COA ownership proofs are used in off-chain authentication flows. The attack requires no privileged access, no admin keys, and no staked node control — only a valid proof blob that was already produced by the victim.

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. Before verifying signatures, check that the first 20 bytes (or a well-defined encoding) of `signedData` equal `evmAddress`. Alternatively, reconstruct the expected signed payload as `hash(evmAddress || application_data)` and verify signatures over that. The fix should be applied in `fvm/evm/stdlib/contract.cdc` at the `validateCOAOwnershipProof` function, and the `COAOwnershipProof` encoding/signing convention should be updated accordingly. [6](#0-5)

### Proof of Concept

**Setup:** Cadence account `alice` owns two COAs:
- COA_A stored at `/storage/coa_a`, published at `/public/coa_a`, with EVM address `0xAAAA...`
- COA_B stored at `/storage/coa_b`, published at `/public/coa_b`, with EVM address `0xBBBB...`

**Step 1 — Obtain a valid proof for COA_A:**
Alice signs hash `H` and produces:
```
proof_a = {
  Address: alice,
  CapabilityPath: "coa_a",
  KeyIndices: [0],
  Signatures: [sig_H]   // sig_H = sign(alice_key_0, H)
}
```
This proof is valid: `verifyCOAOwnershipProof(0xAAAA..., H, encode(proof_a))` returns `true`.

**Step 2 — Construct a replay proof for COA_B:**
The attacker modifies only `CapabilityPath`:
```
proof_b = {
  Address: alice,
  CapabilityPath: "coa_b",   // changed
  KeyIndices: [0],
  Signatures: [sig_H]        // same signature, same data
}
```

**Step 3 — Call the precompile with COA_B's address:**
```
verifyCOAOwnershipProof(0xBBBB..., H, encode(proof_b))
```

**Result:** `validateCOAOwnershipProof` verifies:
- `sig_H` is valid over `H` for `alice`'s key 0 → ✓ (unchanged)
- COA at `alice`'s `/public/coa_b` has EVM address `0xBBBB...` → ✓ (true by construction)
- Returns `isValid: true`

The attacker has successfully impersonated COA_B using a signature that was only produced for COA_A. Any EVM contract calling `COA_B.isValidSignature(H, encode(proof_b))` will receive `0x1626ba7e` (valid). [7](#0-6) [8](#0-7)

	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
