### Title
Pending-state `is_deployed` check ignores pending `deployed_contracts`, causing `replace_class` on a pending-only contract to be misclassified as `deployed_contracts` instead of `replaced_classes` in simulation/trace state diffs — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`convert_thin_state_diff` calls `is_deployed(prev_block_number, address)` to decide whether a class-hash change in a simulation's `induced_state_diff` is a new deployment or a class replacement. For pending-block simulations, `prev_block_number` is the latest accepted block number. `is_deployed` constructs a `BlockId::HashOrNumber` (not `Tag::Pending`), so `maybe_get_class_hash_at` skips the pending overlay entirely and only queries accepted storage. A contract first deployed in the pending state (not yet accepted) is invisible to this check, so any `replace_class` call on it during simulation is misclassified as `deployed_contracts` instead of `replaced_classes` in the returned trace state diff.

### Finding Description
`simulate_transactions` and `trace_transaction`/`trace_block_transactions` all call `convert_thin_state_diff` to post-process the blockifier's `induced_state_diff`. [4](#0-3) 

The blockifier emits a flat `StarknetApiThinStateDiff` where every class-hash write — whether from a `deploy` syscall or a `replace_class` syscall — appears in `deployed_contracts`. The RPC layer is the only place that splits these into `deployed_contracts` (first-ever deployment) vs `replaced_classes` (class swap on an existing contract). It does so by calling `is_deployed` against the state *before* the simulated transaction.

For a pending simulation, "before" means: accepted storage up to `block_number` **plus** the pending state diff. But `is_deployed` constructs a non-pending `BlockId`: [5](#0-4) 

This causes `maybe_get_class_hash_at` to take the `else { None }` branch and skip the pending overlay: [6](#0-5) 

Contrast this with the correct pending-aware path used by `get_class_hash_at` (the public RPC endpoint), which does pass `Tag::Pending` and therefore does include the pending overlay.

### Impact Explanation
Any caller of `starknet_simulateTransactions`, `starknet_traceTransaction`, or `starknet_traceBlockTransactions` against the pending block receives a state diff where a `replace_class` on a pending-only contract is reported as `deployed_contracts` instead of `replaced_classes`. This is an authoritative-looking wrong value returned by the RPC simulation/trace path, matching the **High** impact tier: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation
The scenario requires:
1. A contract deployed in the current pending block (routine — every pending block can contain deploy transactions).
2. A simulation request against `"block_id": "pending"` that invokes `replace_class` on that contract.

Both conditions are reachable by any unprivileged user with no special privileges. The pending state is publicly readable and simulation is a public RPC endpoint.

### Recommendation
In `is_deployed`, when the caller intends to check the pending-inclusive state, pass `Tag::Pending` rather than a bare block number. Alternatively, refactor `convert_thin_state_diff` to pass the original `block_id` through to `is_deployed` so that the pending overlay is included when appropriate:

```rust
async fn is_deployed(
    &self,
    block_id: BlockId,   // was: block_number: BlockNumber
    contract_address: ContractAddress,
) -> RpcResult<bool> {
    Ok(self.maybe_get_class_hash_at(block_id, contract_address).await?.is_some())
}
```

And in `convert_thin_state_diff`, call:
```rust
// For pending: pass Tag::Pending so the pending overlay is consulted.
// For accepted: pass the previous block number as before.
let prev_block_id = match block_id {
    BlockId::Tag(Tag::Pending) => BlockId::Tag(Tag::Pending),
    _ => BlockId::HashOrNumber(BlockHashOrNumber::Number(prev_block_number)),
};
if self.is_deployed(prev_block_id, address).await? { ... }
```

### Proof of Concept
Scenario (Rust integration test sketch):

1. Write block 0 to accepted storage (no contract at address `0xABCD`).
2. Populate `pending_data.state_update.state_diff.deployed_contracts` with `{0xABCD → class_hash_v1}` (simulating a pending deploy transaction).
3. Call `simulate_transactions` with `block_id = "pending"` and a transaction that invokes `replace_class(class_hash_v2)` from within contract `0xABCD`.
4. The blockifier executes against pending state (sees `0xABCD` as deployed with `class_hash_v1`) and produces `induced_state_diff.deployed_contracts = {0xABCD → class_hash_v2}`.
5. `convert_thin_state_diff` calls `is_deployed(block_number=0, 0xABCD)` → accepted storage has no entry → returns `false`.
6. **Observed**: `0xABCD` remains in `deployed_contracts` in the returned trace.
7. **Expected**: `0xABCD` should be in `replaced_classes` because it was already deployed (in pending) before the simulated transaction ran.
