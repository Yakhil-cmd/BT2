Audit Report

## Title
Incomplete OFAC Blocklist Check in ckBTC Minter `retrieve_btc` — Caller's Bitcoin Deposit Address Not Validated Against Blocklist - (File: rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs)

## Summary
The ckBTC minter's `retrieve_btc` and `retrieve_btc_with_approval` functions check only the **destination** Bitcoin address against the OFAC blocklist, not the caller's derived Bitcoin deposit address. The blocklist carries an explicit bidirectional comment ("BTC is not accepted from nor sent to addresses on this list"), but the withdrawal path enforces only the outbound ("sent to") half. A user whose derived deposit address appears on the OFAC SDN list can freely convert ckBTC back to BTC by routing to any clean destination address, bypassing the compliance control entirely.

## Finding Description
In `rs/bitcoin/checker/lib/blocklist.rs` (L9–11), the `BTC_ADDRESS_BLOCKLIST` is annotated with a bidirectional intent: "BTC is not accepted from nor sent to addresses on this list." The deposit path (`update_balance`) partially enforces the inbound direction by calling `check_transaction` on UTXO input addresses. However, in the withdrawal path:

- `retrieve_btc` (L186–202): calls `check_address(btc_checker_principal, args.address.clone(), runtime)` — only the destination `args.address` is screened.
- `retrieve_btc_with_approval` (L281–307): calls `check_address(btc_checker_principal, parsed_address.display(btc_network), runtime)` — again only the destination.

The `check_address` helper (L534–547) is a single-address lookup against the blocklist. The `CanisterRuntime` trait exposes `derive_user_address` (lib.rs L1793–1795), which deterministically computes the caller's Bitcoin deposit address from their IC principal, but this function is never invoked in either withdrawal function. As a result, if a caller's derived deposit address is present on the OFAC SDN list, neither `retrieve_btc` nor `retrieve_btc_with_approval` will detect it, and the withdrawal proceeds normally.

## Impact Explanation
This is a **High** severity finding matching: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."* The ckBTC system explicitly implements OFAC compliance as a documented security property. The gap allows a sanctioned entity — one whose derived deposit address has been added to the OFAC SDN list — to exit the ckBTC system freely by withdrawing to any non-sanctioned Bitcoin address. This constitutes a concrete regulatory compliance bypass in an in-scope financial integration (ckBTC/Chain Fusion), with direct protocol-level harm: the minter facilitates a BTC transfer on behalf of a sanctioned party in violation of its own stated compliance invariant.

## Likelihood Explanation
The attack path requires no special privileges: any unprivileged user can call `retrieve_btc` or `retrieve_btc_with_approval`. The precondition — that a user's derived deposit address appears on the OFAC SDN list — is realistic because: (1) ckBTC deposit addresses are deterministic and publicly derivable from IC principals; (2) OFAC updates its SDN list periodically and independently of NNS upgrade cycles; (3) a user may have legitimately minted ckBTC before their address was sanctioned, or via UTXOs whose input addresses were not blocked at deposit time. The window between an SDN list update and the next NNS-approved blocklist upgrade creates a period during which the gap is exploitable, but even after a blocklist upgrade the withdrawal path remains unprotected for the caller's own address.

## Recommendation
In both `retrieve_btc` and `retrieve_btc_with_approval`, after obtaining the `caller`, derive the caller's Bitcoin deposit address using `runtime.derive_user_address(state, &Account { owner: caller, subaccount: None })` and call `check_address` on it before proceeding. If the result is `BtcAddressCheckStatus::Tainted`, return an appropriate error (e.g., `RetrieveBtcError::GenericError` with `error_code: ErrorCode::TaintedAddress`) and abort the withdrawal. This mirrors the existing destination check and closes the bidirectional compliance gap documented in the blocklist comment.

## Proof of Concept
1. Derive the caller's deposit address `A` for a known IC principal using `account_to_p2wpkh_address_from_state`.
2. Add address `A` to the `BTC_ADDRESS_BLOCKLIST` (simulating an SDN list update via NNS upgrade).
3. Ensure the caller holds ckBTC (minted prior to the blocklist update).
4. Call `retrieve_btc` with a destination address `B` not present in the blocklist.
5. Observe: `check_address(B)` returns `Clean`; the minter burns the caller's ckBTC and schedules a BTC transaction to `B`. Address `A` is never checked.
6. Confirm: an integration test analogous to `should_block_deposit_from_blocked_address` (ckETH, `rs/ethereum/cketh/minter/tests/cketh.rs` L221–239) for the withdrawal path would fail — no such test exists for ckBTC withdrawal, confirming the gap is untested and unmitigated.
