### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Payment Bypass - (File: engine-precompiles/src/xcc.rs)

### Summary

The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller before executing a NEAR cross-contract call. The EVM sub-call's exit reason is checked, but the ABI-encoded boolean return value of `transferFrom` is never decoded or validated. If the wNEAR ERC-20 contract returns `false` instead of reverting on a failed transfer, the precompile proceeds as if payment was received, allowing the caller to execute cross-contract calls without paying the required wNEAR.

### Finding Description

In `engine-precompiles/src/xcc.rs`, the `run_with_handle` method of `CrossContractCall` collects wNEAR from the caller when `required_near != ZERO_YOCTO`. It does so by constructing a raw `transferFrom` calldata and invoking it via `handle.call()`: [1](#0-0) 

The destructured `return_value` is only used as the raw revert payload in the `Revert` branch. In the `Succeed` branch (line 203), `return_value` — which contains the ABI-encoded `bool` returned by `transferFrom` — is silently discarded. The code never decodes it and never asserts it equals `true`.

The ERC-20 `transferFrom` specification (EIP-20) states the function MUST return a boolean: `true` on success, `false` on failure. Non-reverting ERC-20 implementations signal failure via this return value rather than by reverting. The `TRANSFER_FROM_SELECTOR` constant confirms the precompile is calling the standard `transferFrom(address,address,uint256) → bool` function: [2](#0-1) 

The `transfer_from_args` helper encodes the call correctly, but the response is never validated: [3](#0-2) 

### Impact Explanation

When `required_near != ZERO_YOCTO`, the precompile is supposed to debit the caller's wNEAR balance before scheduling a NEAR cross-contract call. If the wNEAR ERC-20 contract returns `false` (without reverting), the precompile emits the `AMOUNT_TOPIC` log and schedules the promise as if payment succeeded. The engine's implicit address never receives the wNEAR, yet the NEAR cross-contract call is still dispatched and paid for by the engine. This constitutes **theft of unclaimed yield / temporary fund freeze** at minimum, and **direct theft of NEAR funds** at maximum, because the engine subsidizes the cross-contract call cost without compensation.

**Impact: High — Theft of NEAR funds (the engine pays for XCC gas without receiving wNEAR).**

### Likelihood Explanation

The wNEAR ERC-20 address is set via `factory_set_wnear_address` and stored in engine state. The standard Aurora-deployed wNEAR mirror contract is an OpenZeppelin-style ERC-20 that reverts on failure, making this harder to trigger with the canonical deployment. However:

1. The code is structurally incorrect regardless of the current wNEAR implementation — it is one contract upgrade or address change away from being exploitable.
2. Any ERC-20 token set as the wNEAR address that follows the non-reverting pattern (e.g., USDT-style) would make this immediately exploitable by any unprivileged EVM user.
3. The attacker-controlled entry path is direct: call the XCC precompile from any EVM address with insufficient wNEAR allowance or balance, using a non-reverting wNEAR token.

**Likelihood: Medium** (depends on the wNEAR contract's revert behavior, but the code is unconditionally wrong).

### Recommendation

After the `Succeed` branch, decode `return_value` as an ABI-encoded `bool` and revert if it is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // ERC-20 transferFrom returns a bool; false means the transfer failed
    // without reverting (non-reverting ERC-20 pattern).
    let success = return_value
        .get(31)  // last byte of the 32-byte ABI bool
        .copied()
        .unwrap_or(0);
    if success == 0 {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

Alternatively, use `SafeERC20`-style logic or require that the wNEAR contract always reverts on failure (document and enforce this invariant).

### Proof of Concept

1. Deploy a non-reverting wNEAR ERC-20 contract whose `transferFrom` always returns `false` without reverting.
2. Set this contract as the wNEAR address via `factory_set_wnear_address`.
3. Call the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) from any EVM address with a valid `CrossContractCallArgs` payload requiring NEAR payment.
4. Observe: `handle.call` returns `ExitReason::Succeed` with `return_value = [0u8; 32]` (ABI-encoded `false`).
5. The `match` arm at line 203 takes the `Succeed` branch and does nothing.
6. The XCC promise log is emitted and the cross-contract call is scheduled — without any wNEAR being transferred from the caller.
7. The engine pays for the NEAR cross-contract call without receiving the required wNEAR compensation. [4](#0-3)
