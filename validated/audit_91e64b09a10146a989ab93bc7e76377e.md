### Title
`execute_scheduled` in XCC Router is `#[payable]` but permanently locks any attached NEAR — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC router contract is marked `#[payable]`, signaling to callers that it accepts attached NEAR tokens. However, the function body never reads or uses `env::attached_deposit()`. Any NEAR attached to a call is silently absorbed by the router contract and permanently locked, because the router has no general NEAR withdrawal mechanism.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, `execute_scheduled` is declared `#[payable]`:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [1](#0-0) 

The `#[payable]` attribute in NEAR SDK means the function will not panic when NEAR is attached. Without it, any attached deposit causes an automatic panic. Its presence therefore signals to callers that NEAR is accepted and used. However, the function never calls `env::attached_deposit()` and never forwards the deposit anywhere. The attached NEAR is received by the router account and stays there.

The stored `PromiseArgs` already encode their own `attached_balance` field, which is drawn from the router's existing NEAR balance (funded separately via `fund_xcc_sub_account`). Attaching NEAR to `execute_scheduled` does not fund the promise — it is simply lost.

The router contract has no general NEAR withdrawal function. The only recovery path is `send_refund`, which:
- Can only be called by the parent (Aurora engine)
- Sends a fixed hardcoded `REFUND_AMOUNT` of 2 NEAR
- Only executes when there are no failed promise results [2](#0-1) 

There is no path to recover arbitrary NEAR locked by user deposits to `execute_scheduled`.

The comment on the function explicitly states it is intentionally open to anyone:

> "It is intentional that this function can be called by anyone (not just the parent)." [3](#0-2) 

---

### Impact Explanation

Any NEAR attached to a call to `execute_scheduled` is permanently frozen in the router contract. This matches the **Critical — Permanent freezing of funds** impact category. The XCC flow involves NEAR payments (for storage staking and promise funding), so users interacting with the XCC system may reasonably believe attaching NEAR to `execute_scheduled` is required or helpful to fund promise execution.

---

### Likelihood Explanation

The function is callable by any account on NEAR (not restricted to the parent engine). The XCC system is designed around NEAR token flows, and the `#[payable]` attribute is a direct signal in NEAR SDK that the function accepts deposits. A user who schedules a promise requiring NEAR and then calls `execute_scheduled` may attach NEAR thinking it funds the execution. The likelihood is **medium**: the mistake is plausible for any user interacting with the XCC router directly, and the function is publicly callable.

---

### Recommendation

Remove the `#[payable]` attribute from `execute_scheduled` if attached NEAR is not intended. Without `#[payable]`, the NEAR SDK will automatically panic if any NEAR is attached, preventing accidental fund loss. If NEAR is intentionally accepted (e.g., to top up the router's balance), add explicit handling via `env::attached_deposit()` and document the behavior.

---

### Proof of Concept

1. User schedules a promise via the XCC precompile that requires NEAR (e.g., `attached_balance: Yocto::new(1_000_000)`).
2. The promise is stored in `scheduled_promises` via `schedule`.
3. User (or anyone) calls `execute_scheduled(nonce)` and attaches, say, 1 NEAR thinking it funds the promise.
4. The function executes successfully — the promise runs using the router's existing balance, not the attached NEAR.
5. The 1 NEAR attached to the call is now held by the router contract.
6. No function exists to recover it: `send_refund` only sends a fixed 2 NEAR to the parent under specific conditions; there is no `withdraw` or equivalent.
7. The user's NEAR is permanently frozen. [1](#0-0) [2](#0-1)

        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
