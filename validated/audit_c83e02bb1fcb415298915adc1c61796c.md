### Title
Unguarded Division by Zero in `calculate_staker_total_staking_power` for STRK Total Stake — (`File: src/staking/utils.cairo`)

### Summary
`calculate_staker_total_staking_power` divides by `strk_total_stake` without a zero-guard, while the analogous BTC branch explicitly checks for zero. If `strk_total_stake` is zero at call time, the `mul_wide_and_div(...).unwrap()` panics, reverting any transaction that triggers staking-power computation.

### Finding Description
In `src/staking/utils.cairo`, the function `calculate_staker_total_staking_power` computes two weighted components of a staker's power:

```cairo
// lines 130-145
let strk_staking_power = mul_wide_and_div(
    lhs: staker_strk_total_amount.to_amount_18_decimals(),
    rhs: STRK_WEIGHT_FACTOR,
    div: strk_total_stake.to_amount_18_decimals(),   // ← no zero-guard
)
    .unwrap();                                        // ← panics if div == 0

let btc_staking_power = if btc_total_stake.is_zero() {  // ← zero-guard present
    Zero::zero()
} else {
    mul_wide_and_div(
        lhs: staker_btc_total_amount.to_amount_18_decimals(),
        rhs: BTC_WEIGHT_FACTOR,
        div: btc_total_stake.to_amount_18_decimals(),
    )
        .unwrap()
};
``` [1](#0-0) 

The BTC branch is explicitly guarded with `if btc_total_stake.is_zero()` returning `Zero::zero()`, but the STRK branch has no equivalent guard. `mul_wide_and_div` returns `None` on division by zero; the immediate `.unwrap()` then panics. [2](#0-1) 

The asymmetry mirrors exactly the original M07 pattern: one divisor is protected, the other is not.

### Impact Explanation
Any protocol path that calls `calculate_staker_total_staking_power` while `strk_total_stake == 0` will revert. This function is internal (`pub(crate)`) and is invoked during staking-power queries and reward distribution flows. A revert in those flows would temporarily freeze reward accrual and attestation for all participants until the condition is resolved — matching the **Temporary freezing of funds / unclaimed yield** impact category.

### Likelihood Explanation
`strk_total_stake` is the aggregate STRK staked across all stakers. It reaches zero only when no STRK is staked in the protocol (e.g., immediately after deployment before any staker joins, or after all stakers have fully exited). With the BTC multi-token extension, a state where BTC stakers exist but STRK total stake is zero is theoretically reachable. Likelihood is **low** in a live network but non-zero during protocol bootstrapping or a mass-exit event.

### Recommendation
Add the same zero-guard for `strk_total_stake` that already exists for `btc_total_stake`:

```cairo
let strk_staking_power = if strk_total_stake.is_zero() {
    Zero::zero()
} else {
    mul_wide_and_div(
        lhs: staker_strk_total_amount.to_amount_18_decimals(),
        rhs: STRK_WEIGHT_FACTOR,
        div: strk_total_stake.to_amount_18_decimals(),
    )
        .unwrap()
};
```

This is consistent with the existing BTC guard at line 136 and eliminates the panic path.

### Proof of Concept
1. Deploy the protocol with no stakers (or reach a state where all STRK stakers have fully exited).
2. Call any public function that internally invokes `calculate_staker_total_staking_power` (e.g., a staking-power query or reward-distribution trigger).
3. The call reaches `mul_wide_and_div(..., div: 0)`, which returns `None`.
4. `.unwrap()` panics, reverting the transaction.
5. All reward and attestation flows are blocked until at least one STRK staker re-enters. [3](#0-2)
