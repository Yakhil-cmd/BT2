### Title
Unbounded Native Resource Passed to Callees Enables Unexpected Transaction Failure via Storage-State Manipulation — (File: `basic_bootloader/src/bootloader/runner.rs`)

---

### Summary

ZKsync OS passes the entire native resource budget (proving cost) to every callee frame with no cap, directly analogous to the external report's "no gas limit on external calls." Because native consumption is invisible to the EVM and depends on storage state that can change between `eth_estimateGas` and actual execution, an attacker can front-run a victim's transaction to inflate its native cost, causing an out-of-native revert that exhausts all gas.

---

### Finding Description

ZKsync OS implements a **dual-resource model**: EVM gas (ergs) and a separate *native* resource that tracks proving cost (RISC-V cycles). The two resources are handled asymmetrically at every external call boundary.

**EVM gas — capped by the 63/64 rule:**

In `evm_interpreter/src/ee_trait_impl.rs`, `calculate_resources_passed_in_external_call` explicitly applies the 63/64 rule to ergs before handing them to the callee:

```rust
let max_passable_ergs =
    gas_utils::apply_63_64_rule(resources_available_in_caller_frame.ergs());
let ergs_to_pass = core::cmp::min(call_request.ergs_to_pass, max_passable_ergs);
``` [1](#0-0) 

**Native resource — no cap, transferred in full:**

Immediately after the ergs calculation, `runner.rs` calls `give_native_to`, which moves *all* remaining native from the caller frame into the callee frame:

```rust
// Give native resource to the callee.
resources_in_caller_frame.give_native_to(&mut callee_resources);
``` [2](#0-1) 

The implementation of `give_native_to` confirms the full transfer with no reservation:

```rust
fn give_native_to(&mut self, other: &mut Self) {
    let n = core::mem::replace(&mut self.native, Native::empty());
    other.native = n;
}
``` [3](#0-2) 

This design is explicitly documented:

> "The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend." [4](#0-3) 

**Why this creates a resource-accounting bug:**

The native limit for a transaction is derived at validation time as `nativeLimit = gasLimit × nativePerGas`. The `deltaGas` mechanism then converts any excess native consumption back into EVM gas and charges it from the remaining ergs:

> "`deltaGas := (nativeUsed / nativePerGas) − gasUsed`. If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs." [5](#0-4) 

Because the callee receives *all* native resources and can consume them freely, any increase in the callee's proving cost between estimation and execution directly inflates `deltaGas`. If the inflated `deltaGas` pushes `gasUsed` past `gasLimit`, the transaction reverts and all gas is consumed — the user cannot protect against this because there is no per-call native cap they can set.

The EVM gas path has the 63/64 rule as a safety valve; the native path has none.

---

### Impact Explanation

- A victim's transaction is estimated to succeed with gas limit G.
- An attacker front-runs the transaction to change storage state (e.g., writing to a slot the victim's callee reads, making it cold at execution time).
- The callee's native consumption rises; `deltaGas` grows accordingly.
- `gasUsed + deltaGas > G` → the transaction reverts with all gas consumed.
- The victim loses the full prepaid fee (`gasLimit × gasPrice`) with no execution benefit.

This is a **resource accounting bug** causing **user fund loss** (full gas prepayment forfeited) triggered by an unprivileged on-chain action (a front-running write to shared storage).

---

### Likelihood Explanation

- Any DeFi protocol with shared mutable storage (AMM reserves, lending positions, ERC-20 balances) is a candidate target.
- Front-running is a well-understood, routinely executed technique on EVM-compatible chains.
- The attacker pays only the cost of a storage write; the victim loses an entire transaction's gas.
- No privileged access is required; any EOA can execute the attack.

---

### Recommendation

1. **Introduce a per-call native cap** analogous to the 63/64 rule for ergs. Allow callers to specify a maximum native budget for each external call, retaining the remainder in the caller frame.
2. **Expose native gas remaining** via a system opcode or precompile so contracts and users can reason about native consumption the same way they reason about EVM gas.
3. **Bound `deltaGas` at estimation time** by computing a worst-case native consumption estimate and including it in the gas estimate returned by `eth_estimateGas`, so users are not surprised at execution time.

---

### Proof of Concept

1. **Deploy** a contract `VictimCallee` that reads a storage slot `S` (cold read = high native cost) and performs a loop whose iteration count depends on `S`.
2. **Victim** calls `eth_estimateGas` for a transaction to `VictimCallee`. At estimation time, `S` is warm (recently written), so native consumption is low; the estimate fits within the victim's chosen gas limit G.
3. **Attacker** observes the pending transaction in the mempool and submits a higher-priority transaction that writes a new value to `S`, evicting it from the warm set.
4. **Victim's transaction** executes. `VictimCallee` now performs a cold SLOAD of `S`, incurring higher native cost. `give_native_to` has already transferred the full native budget to `VictimCallee`; it consumes it all.
5. `deltaGas = (nativeUsed / nativePerGas) − gasUsed` exceeds the remaining ergs budget.
6. The transaction reverts; `gasUsed` is reported as G (full gas limit consumed).
7. Victim loses `G × gasPrice` with no state change.
