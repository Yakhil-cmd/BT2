### Title
Duplicate Authorizer Addresses Not Rejected in Transaction Validation — (File: `access/validator/validator.go`, `fvm/environment/transaction_info.go`)

---

### Summary

The Flow transaction validation pipeline checks for duplicate *signatures* (by address+key-index pair) but never enforces uniqueness of the *addresses* in `tx.Authorizers`. An unprivileged transaction sender can submit a transaction whose `Authorizers` slice contains the same address more than once. All existing validation gates pass, and the FVM propagates the duplicate list verbatim to the Cadence runtime, allowing a single account to satisfy a multi-authorizer `prepare` block that was designed to require two or more *distinct* signers.

---

### Finding Description

**Step 1 – Access-node pre-validation (`access/validator/validator.go`)**

`checkAccounts` iterates over `tx.Authorizers` and verifies that each entry has at least one observed signature, but it uses a `map[flow.Address]bool` keyed only by address. If `Authorizers = [addrA, addrA]`, both iterations hit the same map entry and both pass silently. [1](#0-0) 

No step in `initValidationSteps` checks for duplicate authorizer addresses: [2](#0-1) 

`checkAddresses` only validates that each address is a valid chain address, not that the list is free of duplicates: [3](#0-2) 

**Step 2 – FVM execution-time verification (`fvm/transactionVerifier.go`)**

`newSignatureEntries` correctly deduplicates *signatures* by `(address, keyIndex)` pair, but the authorizer list itself is never deduplicated. The subsequent weight loop iterates over the raw `tx.Authorizers` slice; for a duplicate entry the same `payloadWeights[addr]` value is read twice, so both iterations pass the threshold check: [4](#0-3) 

**Step 3 – Runtime authorizer list construction (`fvm/environment/transaction_info.go`)**

`NewTransactionInfo` appends every element of `tx.Authorizers` into `runtimeAddresses` without a uniqueness guard. If the same address appears twice, the Cadence runtime receives it twice: [5](#0-4) 

The Cadence runtime uses `runtimeAuthorizers` to bind `AuthAccount` objects to the positional parameters of the transaction's `prepare` block. With `Authorizers = [addrA, addrA]` and a script `prepare(a: &Account, b: &Account)`, both `a` and `b` resolve to the same account, satisfying a two-signer requirement with only one real signer.

---

### Impact Explanation

Any Cadence contract or transaction script that uses a multi-authorizer `prepare` block to enforce a 2-of-N (or N-of-N) signing policy can be bypassed by a single account that lists itself N times in `Authorizers`. Because `AuthAccount` grants full storage and capability access to the named account, an attacker who controls one account can impersonate a second required co-signer, potentially:

- Withdrawing assets from a shared vault that requires two distinct owners to authorize.
- Executing privileged contract operations gated behind a multi-party approval pattern.

This is a **transaction authorization bypass** leading to unauthorized mutation of on-chain assets.

---

### Likelihood Explanation

The attack requires only a standard, unprivileged transaction submission — no staked node, no admin key, no social engineering. Any user who can send a transaction to an Access Node can exploit this. The only prerequisite is the existence of a deployed contract whose security model relies on the distinctness of authorizer addresses, which is a common and reasonable assumption for multi-sig patterns in Cadence.

---

### Recommendation

1. **In `access/validator/validator.go` → `checkAccounts`**: After building `observedSigs`, add a uniqueness check over `tx.Authorizers` using a `map[flow.Address]struct{}` and return a `DuplicatedAuthorizerError` if any address appears more than once.

2. **In `fvm/environment/transaction_info.go` → `NewTransactionInfo`**: As a defense-in-depth measure, deduplicate `runtimeAddresses` before passing them to the Cadence runtime, or assert that no duplicates exist and panic/error if they do.

3. **In `fvm/transactionVerifier.go` → `verifyTransaction`**: Add an explicit uniqueness assertion over `tx.Authorizers` before the weight-check loop.

---

### Proof of Concept

```
Attacker controls account addrA (weight 1000, threshold 1000).

Craft transaction:
  Authorizers = [addrA, addrA]
  Script:
    transaction {
      prepare(owner: &Account, cosigner: &Account) {
        // contract assumes owner != cosigner
        // e.g. withdraw from a shared vault requiring two distinct signers
        SharedVault.withdraw(from: owner, approvedBy: cosigner)
      }
    }

Submit to Access Node:
  - checkAccounts: observedPayloadSig[addrA] = true → both authorizer entries pass.
  - verifyTransaction: payloadWeights[addrA] = 1000 ≥ 1000 → both entries pass.
  - NewTransactionInfo: runtimeAddresses = [addrA, addrA].
  - Cadence runtime binds owner = AuthAccount(addrA), cosigner = AuthAccount(addrA).
  - SharedVault.withdraw executes with a single real signer satisfying a 2-of-2 gate.
```

This is directly analogous to the Basin `tokens[0] = tokens[1]` scenario: a list that is assumed to contain distinct entities does not enforce that invariant at construction time, and downstream logic that iterates the list produces incorrect (doubled/mis-attributed) results.

```go
func (v *TransactionValidator) checkAccounts(tx *flow.TransactionBody) error {
	// check for duplicate account key
	type uniqueKey struct {
		address flow.Address
		index   uint32
	}
	observedSigs := make(map[uniqueKey]bool)
	for _, sig := range append(tx.PayloadSignatures, tx.EnvelopeSignatures...) {
		if observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] {
			return DuplicatedSignatureError{Address: sig.Address, KeyIndex: sig.KeyIndex}
		}
		observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] = true
	}
	// check for minimum account signatures
	observedEnvelopeSig := make(map[flow.Address]bool)
	observedPayloadSig := make(map[flow.Address]bool)
	for _, sig := range tx.EnvelopeSignatures {
		observedEnvelopeSig[sig.Address] = true
	}
	for _, sig := range tx.PayloadSignatures {
		observedPayloadSig[sig.Address] = true
	}

	if !observedEnvelopeSig[tx.Payer] {
		return MissingSignatureError{Address: tx.Payer, Message: "payer envelope signature is missing"}
	}

	if !observedEnvelopeSig[tx.ProposalKey.Address] && !observedPayloadSig[tx.ProposalKey.Address] {
		return MissingSignatureError{Address: tx.ProposalKey.Address, Message: "proposer signature on either payload or envelope is missing"}
	}

	for _, authorizer := range tx.Authorizers {
		if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
			// at this point, payer and proposer are guaranteed to have signatures
			continue
		}
		if !observedEnvelopeSig[authorizer] && !observedPayloadSig[authorizer] {
			return MissingSignatureError{Address: authorizer, Message: "authorizer signature on either payload or envelope is missing"}
		}
	}
```

```go
func NewTransactionInfo(
	params TransactionInfoParams,
	tracer tracing.TracerSpan,
	serviceAccount flow.Address,
) TransactionInfo {

	isServiceAccountAuthorizer := false
	runtimeAddresses := make(
		[]common.Address,
		0,
		len(params.TxBody.Authorizers))

	for _, auth := range params.TxBody.Authorizers {
		runtimeAddresses = append(
			runtimeAddresses,
			common.Address(auth))
		if auth == serviceAccount {
			isServiceAccountAuthorizer = true
		}
	}

	return &transactionInfo{
		params:                     params,
		tracer:                     tracer,
		runtimeAuthorizers:         runtimeAddresses,
		isServiceAccountAuthorizer: isServiceAccountAuthorizer,
	}
}
```
