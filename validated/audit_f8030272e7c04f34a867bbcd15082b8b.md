### Title
`tick()` All-or-Nothing Batch Minting Blocked by Partial Period Cap Consumption — (`L1/starkware/solidity/stake/RewardSupplier.sol` / `PeriodMintLimit.sol`)

### Summary

`tick()` always mints for the full batch of up to `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5` messages at once. The period cap is set to exactly `5 × TOKENS_PER_MINT_REQUEST = 6_500_000e18`. If any prior `tick()` call in the same period consumed even one message (1_300_000e18), the remaining cap (5_200_000e18) is insufficient for a full 5-message batch, causing every subsequent `tick()` call with 5 pending messages to revert with `EXCEED_PERIOD_MINTING` for up to one week.

### Finding Description

**Constants (all production code):**

| Constant | Value | File |
|---|---|---|
| `TOKENS_PER_MINT_REQUEST` | 1_300_000e18 | `RewardSupplier.sol:10` |
| `MAX_MESSAGES_TO_PROCESS_PER_TICK` | 5 | `RewardSupplier.sol:11` |
| `PERIOD_MINT_CAP` | 6_500_000e18 | `PeriodMintLimit.sol:8` |
| `MINTING_PERIOD_DURATION` | 1 weeks | `PeriodMintLimit.sol:9` |

Note: `PERIOD_MINT_CAP == MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST` exactly.

**Call chain:**

`tick()` (no access control) → `requiredMinting()` → returns `(TOKENS_PER_MINT_REQUEST × numMsgsToConsume, numMsgsToConsume)` → consumes messages → `mintManager().mintRequest(token(), amountToMint)` → `checkAndUpdatePeriodicalQuota(token, amountToMint)`. [1](#0-0) 

`checkAndUpdatePeriodicalQuota` enforces:

```
require(mintedThisPeriodBefore + amount <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
``` [2](#0-1) 

**Failure scenario (exact arithmetic):**

1. Period starts: `mintedThisPeriodBefore = 0`.
2. `tick()` called with 1 pending message → `amountToMint = 1_300_000e18` → `0 + 1_300_000e18 = 1_300_000e18 ≤ 6_500_000e18` → **succeeds**. Now `mintedThisPeriodBefore = 1_300_000e18`.
3. 5 messages accumulate in the L2→L1 queue.
4. `tick()` called → `numMsgsToConsume = min(5, 5) = 5` → `amountToMint = 6_500_000e18`.
5. `checkAndUpdatePeriodicalQuota`: `1_300_000e18 + 6_500_000e18 = 7_800_000e18 > 6_500_000e18` → **reverts**. [3](#0-2) 

Because the entire transaction is atomic, the `consumeMessageFromL2` calls at lines 117–119 also revert, so the 5 messages remain pending but unprocessable for the rest of the period (up to 7 days). [4](#0-3) 

There is no partial-batch path: `requiredMinting()` always returns the full `min(pending, 5)` count with no awareness of the remaining period quota. [5](#0-4) 

### Impact Explanation

Pending L2→L1 mint-request messages cannot be processed for up to one full `MINTING_PERIOD_DURATION` (1 week). Staking rewards that depend on those minted tokens being bridged to L2 are delayed. This is a temporary freeze of unclaimed yield delivery. The messages are not lost, but the delay is bounded only by the period reset. [6](#0-5) 

### Likelihood Explanation

The trigger condition is normal protocol operation: `tick()` is permissionless and is expected to be called whenever messages are pending. Any call that processes fewer than 5 messages (because fewer than 5 are pending at that moment) partially consumes the cap. This is a routine occurrence — L2 mint requests accumulate gradually. No privileged access, no key compromise, and no external dependency is required. [7](#0-6) 

### Recommendation

`requiredMinting()` should cap `numMsgsToConsume` not only by `MAX_MESSAGES_TO_PROCESS_PER_TICK` but also by the remaining period quota:

```solidity
uint256 remainingCap = PERIOD_MINT_CAP - mintedThisPeriod(token);
uint256 maxByQuota = remainingCap / TOKENS_PER_MINT_REQUEST;
uint256 numMsgsToConsume = Math.min(
    Math.min(messagingContract().l2ToL1Messages(msgHash), MAX_MESSAGES_TO_PROCESS_PER_TICK),
    maxByQuota
);
```

This makes `tick()` process as many messages as the remaining cap allows rather than reverting when the full batch exceeds it. [5](#0-4) [8](#0-7)

```

Trigger:
  3. pendingMessages = 5
  4. call tick()
       requiredMinting() → amountToMint = 5 * 1_300_000e18 = 6_500_000e18

  5. pendingMessages = 4
  6. call tick()
       amountToMint = 4 * 1_300_000e18 = 5_200_000e18
       1_300_000e18 + 5_200_000e18 = 6_500_000e18 <= 6_500_000e18
       → succeeds                               ✓

Result: with 5 pending messages the queue is frozen until period reset (≤ 7 days).
```

        uint256[] memory messagePayload = new uint256[](1);
        messagePayload[0] = TOKENS_PER_MINT_REQUEST;

        bytes32 msgHash = l2ToL1MsgHash(mintRequestSource(), address(this), messagePayload);
        // Limit the number of msgs to consume to limit.
        uint256 numMsgsToConsume = Math.min(
            messagingContract().l2ToL1Messages(msgHash),
            MAX_MESSAGES_TO_PROCESS_PER_TICK
        );

        return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
    }

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);
```

    function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
        bytes32 periodSlot = periodAccountingSlot(token);
        uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
        uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
        require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
        periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
    }
