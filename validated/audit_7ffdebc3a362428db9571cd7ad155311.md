### Title
Threshold Value Accepted as `usize` but Silently Narrowed to `u16` at Signing Time, Causing Permanent Denial of FROST Signing - (File: `src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

The FROST signing protocols (`eddsa` and `redjubjub`) accept a `ReconstructionLowerBound` threshold backed by a `usize` at initialization and DKG time, with no upper-bound check against `u16::MAX` (65535). However, inside `construct_key_package`, the threshold is narrowed to `u16` via `u16::try_from(threshold.value())`. If the threshold exceeds 65535, this conversion fails and returns a `ProtocolError`, causing every signing attempt to fail permanently after a successful DKG.

---

`construct_key_package` is called from every signing code path — coordinator v1, coordinator v2, participant v1, and participant v2: [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9) 

If `threshold > 65535`, every call to `construct_key_package` returns `Err(ProtocolError::Other("threshold cannot be converted to u16"))`, and signing is permanently broken for all participants.

---

### Impact Explanation

A caller who sets up DKG with a threshold value between 65536 and `usize::MAX` will complete DKG successfully (no u16 check exists there), receive valid `KeygenOutput` shares, and then find that every subsequent signing attempt — whether via `sign_v1`, `sign_v2`, or the redjubjub `sign` — fails with a `ProtocolError` at the `construct_key_package` step. There is no recovery path: the key shares are valid but signing is permanently denied. This matches the **High: Permanent denial of signing for honest parties under valid protocol inputs** impact category.

---

### Likelihood Explanation

The library exposes `ReconstructionLowerBound(usize)` as a public API type with no documented ceiling of 65535. A library integrator building a large MPC network (e.g., threshold = 70,000 in a permissioned validator set) would pass a threshold that clears all initialization guards, complete DKG without error, and only discover the failure at signing time. The gap between the accepted type (`usize`, up to ~1.8×10^19 on 64-bit systems) and the enforced ceiling (`u16::MAX` = 65535) is not surfaced anywhere in the public API or error messages at setup time.

---

### Recommendation

Add an explicit upper-bound check against `u16::MAX` in `assert_sign_inputs` (and ideally also in `assert_key_invariants`) before the protocol is initialized:

```rust
// In assert_sign_inputs (src/frost/mod.rs)
if threshold.value() > u16::MAX as usize {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: u16::MAX as usize,
    });
}
```

Alternatively, change the `KeyPackage::new` call to accept a `usize` or `u32` if the upstream `frost_core` / `frost_ed25519` / `reddsa` libraries permit it, eliminating the narrowing entirely.

---
