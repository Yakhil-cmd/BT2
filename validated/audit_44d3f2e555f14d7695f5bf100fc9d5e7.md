### Title
Cloneable `PresignOutput` and Reference-Based `rerandomize_presign` API Enable Presignature Nonce Reuse Leading to Secret Key Recovery - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based and Robust ECDSA implementations expose a `PresignOutput` type that derives `Clone` and whose `rerandomize_presign` function accepts the presignature by shared reference (`&PresignOutput`). This means the same presignature — containing secret nonce material — can be rerandomized and consumed in multiple independent signing sessions without any library-level enforcement preventing reuse. Two signatures produced from the same underlying presignature with different `(msg_hash, tweak)` pairs have multiplicatively related nonces, enabling full secret key recovery via standard ECDSA nonce-reuse algebra. The library documents this danger but provides no code-level guard.

---

### Finding Description

**OT-based ECDSA** (`src/ecdsa/ot_based_ecdsa/mod.rs`):

`PresignOutput` is declared with `#[derive(Clone)]`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
```

`RerandomizedPresignOutput::rerandomize_presign` takes the presignature by shared reference:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,          // ← shared reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { … }
``` [1](#0-0) [2](#0-1) 

**Robust ECDSA** (`src/ecdsa/robust_ecdsa/mod.rs`):

Identical pattern — `PresignOutput` derives `Clone` and `rerandomize_presign` takes `&PresignOutput`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub c: Scalar,
    pub e: Scalar,
    pub alpha: Scalar,
    pub beta: Scalar,
}
``` [3](#0-2) [4](#0-3) 

By contrast, `RerandomizedPresignOutput` deliberately does **not** derive `Clone` and the `sign` function consumes it by value — demonstrating that the library authors understood the one-time-use requirement for the downstream type but did not apply the same protection to the upstream `PresignOutput`. [5](#0-4) [6](#0-5) 

The security documentation explicitly acknowledges the danger:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [7](#0-6) 

Yet the API makes reuse trivially possible: a caller holds a `PresignOutput`, calls `rerandomize_presign(&presign_output, args_for_msg1)` and then `rerandomize_presign(&presign_output, args_for_msg2)`, obtaining two `RerandomizedPresignOutput` values backed by the same nonce `k`. No compile-time or runtime barrier prevents this.

The OT-based ECDSA README also states "Each output is consumed **exactly once** (one-time use)" but this invariant is not enforced by the type system. [8](#0-7) 

---

### Impact Explanation

**Impact: Critical — Extraction/reconstruction of private signing shares.**

In ECDSA, if two signatures `(R₁, s₁)` and `(R₂, s₂)` are produced from the same presignature with different messages `h₁ ≠ h₂`:

- After rerandomization with scalars `δ₁` and `δ₂` (derived from the respective messages), the effective nonces are `k·δ₁⁻¹` and `k·δ₂⁻¹`.
- The ratio `(k·δ₁⁻¹) / (k·δ₂⁻¹) = δ₂/δ₁` is publicly computable from the two signatures and the public rerandomization inputs.
- This multiplicative nonce relationship is sufficient to recover the secret key `x` using standard ECDSA nonce-reuse algebra, as explicitly confirmed by the library's own security documentation.

For the Robust ECDSA scheme, the documentation further notes that due to signature-share linearization, the attack requires as few as `2t+2` presigning participants and two signing sessions. [9](#0-8) 

---

### Likelihood Explanation

**Likelihood: High.**

The attack requires no privileged access. Any caller of the public API — an integrator, a malicious coordinator, or a participant who controls the signing orchestration — can:

1. Obtain a `PresignOutput` from a completed presigning round (this is the normal output of the presign protocol).
2. Call `rerandomize_presign` twice on the same `PresignOutput` reference with two different `RerandomizationArguments` (different `msg_hash` or `tweak`).
3. Drive two signing sessions to completion using the two `RerandomizedPresignOutput` values.
4. Collect the two resulting signatures and apply the nonce-reuse recovery.

No cryptographic break, no leaked key, and no out-of-scope assumption is required. The `Clone` derive makes the attack even simpler (clone the `PresignOutput`, hand one copy to each signing session). The library's own README and security documentation confirm the danger, meaning the attack surface is well-understood and the missing guard is a concrete omission.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput`** in both `src/ecdsa/ot_based_ecdsa/mod.rs` and `src/ecdsa/robust_ecdsa/mod.rs`. This mirrors the intentional design of `RerandomizedPresignOutput`, which already omits `Clone`.

2. **Change `rerandomize_presign` to consume `PresignOutput` by value** (`presignature: PresignOutput` instead of `presignature: &PresignOutput`). This makes reuse a compile-time error: once the presignature is rerandomized, it is moved and cannot be used again.

3. **Apply the same pattern to the FROST `PresignOutput<C>`** in `src/frost/mod.rs`, which also derives `Clone` and contains `SigningNonces` — secret nonce material whose reuse in Schnorr/FROST similarly enables signing-share recovery. [10](#0-9)

### Proof of Concept

```rust
// Attacker-controlled orchestration layer
use threshold_signatures::ecdsa::ot_based_ecdsa::{
    PresignOutput, RerandomizedPresignOutput, RerandomizationArguments,
};

// presign_output is the result of a completed presigning round
let presign_output: PresignOutput = run_presign_protocol(…);

// Rerandomize the SAME presignature for two different messages — no error, no warning
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,          // ← shared reference, not consumed
    &args_for_message_hash_1,
).unwrap();

let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,          // ← same presignature reused
    &args_for_message_hash_2,
).unwrap();

// Drive two signing sessions to completion
let sig1 = run_sign(participants, rerand1, msg_hash_1);
let sig2 = run_sign(participants, rerand2, msg_hash_2);

// sig1 and sig2 have multiplicatively related nonces (k·δ₁⁻¹ and k·δ₂⁻¹).
// δ₁ and δ₂ are publicly derivable from RerandomizationArguments.
// Standard ECDSA nonce-reuse algebra recovers the aggregate secret key x.
let recovered_secret_key = ecdsa_nonce_reuse_recovery(sig1, sig2, delta1, delta2, …);
```

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**

   * Do **not** allow $N_2 < N_1$.

   The coordinator must not be able to present different message hashes, tweaks, or
