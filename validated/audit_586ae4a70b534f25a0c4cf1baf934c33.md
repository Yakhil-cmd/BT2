### Title
ERC-20 Mirror Token Burned Before Cross-Chain Transfer Confirmation, No Refund Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile flow burns a user's ERC-20 mirror tokens (or deducts ETH) synchronously within the EVM execution, then schedules an asynchronous NEAR promise (`ft_transfer` / `ft_transfer_call`) to deliver the corresponding NEP-141 tokens. When the `error_refund` Cargo feature is **not** compiled in — which is the case for the default build, since `error_refund` is absent from the `default` feature set — no callback is registered to handle a failed promise. If the cross-chain transfer fails, the burned tokens are permanently lost with no refund path.

---

### Finding Description

**Step 1 — Tokens burned before promise executes.**

In `EvmErc20.withdrawToNear()`, the ERC-20 balance is reduced first:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← tokens gone from EVM state
    // ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...);
        // res is never checked
    }
}
``` [1](#0-0) 

**Step 2 — Refund args are `None` without the feature flag.**

Inside the `ExitToNear` precompile `run()`, the `refund` field of the callback args is unconditionally set to `None` when `error_refund` is not compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in default build
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

**Step 3 — No callback scheduled for the standard `ft_transfer` path.**

When `refund` is `None` and `transfer_near` is also `None` (the common `ft_transfer` path), `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise is scheduled as a bare `PromiseArgs::Create` with **no callback at all**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

**Step 4 — Callback silently does nothing on failure.**

Even in the `ft_transfer_call` path where a callback *is* registered, the callback's failure branch falls through to `None` when `args.refund` is `None`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← no refund, no error, tokens gone
};
``` [4](#0-3) 

**Step 5 — `error_refund` is not a default feature.**

```toml
[features]
default = ["std"]
# ...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [5](#0-4) 

The test suite explicitly documents the fund-loss outcome when the feature is absent:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

---

### Impact Explanation

When the `ft_transfer` promise fails (e.g., the NEAR recipient account is not registered with the NEP-141 token), the user's ERC-20 mirror tokens have already been burned on the Aurora side. Without `error_refund`, no refund is issued. The tokens are permanently destroyed with no recovery path. This constitutes **critical permanent freezing/loss of user funds**.

---

### Likelihood Explanation

NEAR NEP-141 tokens require explicit storage registration before an account can receive them. Any user who calls `withdrawToNear()` with a recipient that has not registered with the specific NEP-141 contract will trigger this failure. This is a realistic, unprivileged user action requiring no special access. The `error_refund` feature is not compiled into the default build, so the production binary is affected unless the feature is explicitly enabled at compile time.

---

### Recommendation

1. Add `error_refund` to the `default` feature set in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml` so the refund path is always compiled in for production builds.
2. Alternatively, restructure the `ExitToNear` precompile so that a failure-handling callback is always registered regardless of compile-time features, mirroring the fix applied in the referenced CoreDAO commit (move the accounting decrement to occur only after confirming the cross-chain transfer succeeded).
3. In `EvmErc20.withdrawToNear()`, check the return value of the `call` to the precompile and revert if it fails, so the `_burn` is also reverted.

---

### Proof of Concept

1. Deploy a NEP-141 token on NEAR. Bridge it to Aurora to get an ERC-20 mirror (`EvmErc20`).
2. As an EVM user holding ERC-20 mirror tokens, call `withdrawToNear(recipient, amount)` where `recipient` is a NEAR account **not registered** with the NEP-141 contract.
3. The ERC-20 `_burn` executes immediately — the user's EVM balance is reduced.
4. The `ExitToNear` precompile schedules `ft_transfer` on the NEP-141 contract with no callback (default build, `error_refund` absent).
5. The `ft_transfer` promise fails because the recipient is unregistered.
6. No callback fires. No refund is issued.
7. The user's ERC-20 tokens are permanently destroyed; the NEP-141 tokens remain locked in the Aurora contract account.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) reproduces this exact scenario and confirms the balance loss when `error_refund` is not enabled. [7](#0-6)
