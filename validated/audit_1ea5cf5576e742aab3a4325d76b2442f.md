### Title
Missing Access Control on `update_rewards` Allows Any Caller to Globally Block Staker Rewards Per Block — (`File: src/staking/staking.cairo`)

### Summary

`update_rewards` is documented as callable only by the Starknet sequencer, but the implementation enforces no such restriction. Any unprivileged address can call it with `disable_rewards: true`, which unconditionally advances the global `last_reward_block` counter and returns early. Because `last_reward_block` is a single contract-wide slot, this one call exhausts the per-block reward slot for every staker, preventing the legitimate sequencer from distributing rewards to any staker for that block.

### Finding Description

The spec for `update_rewards` states:

> **access control**: Only starkware sequencer.

The implementation, however, only calls `general_prerequisites()`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
``` [1](#0-0) 

`general_prerequisites` contains no sequencer check:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [2](#0-1) 

Immediately after the staker-validity assertions, `last_reward_block` is written to the current block number **before** the `disable_rewards` branch is evaluated:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [3](#0-2) 

`last_reward_block` is a single global storage slot shared across all stakers:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [4](#0-3) 

The guard at the top of `update_rewards` rejects any second call in the same block:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [5](#0-4) 

**Attack flow:**

1. Attacker observes the mempool or simply submits a transaction at the start of each block.
2. Attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)`.
3. The call passes all checks (contract not paused, caller non-zero, staker active, block not yet processed).
4. `last_reward_block` is set to the current block; the function returns without distributing any rewards.
5. The legitimate sequencer's call to `update_rewards` for any staker in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers and their pool members receive zero rewards for that block.
7. Repeated every block → permanent freezing of unclaimed yield.

The attacker needs only a valid active staker address, which is publicly enumerable from the `stakers` vector. [6](#0-5) 

### Impact Explanation

Every block in which the attacker front-runs the sequencer, **all** stakers and delegation-pool members earn zero block rewards. Sustained across many blocks this constitutes permanent freezing of unclaimed yield for all protocol participants. This matches the allowed High impact: *"Permanent freezing of unclaimed yield or unclaimed royalties"* and *"Temporary freezing of funds"*.

### Likelihood Explanation

The entry point is a public, permissionless function. The only cost to the attacker is gas per block. No privileged key, bridge compromise, or external dependency is required. The staker address argument can be any currently active staker, all of which are publicly readable on-chain. The attack is straightforward to automate.

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, consistent with the spec. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.roles.only_sequencer();   // enforce the documented access control
    self.general_prerequisites();
    ...
```

Alternatively, if a dedicated sequencer role is not yet implemented, restrict the caller to a stored `sequencer_address` similar to how `attestation_contract` is stored and checked.

### Proof of Concept

1. Deploy the protocol and advance past the consensus-rewards activation epoch.
2. From any EOA (non-zero address), call:
   ```
   staking.update_rewards(staker_address=<any_active_staker>, disable_rewards=true)
   ```
3. Observe that `last_reward_block` is now equal to the current block number and no `StakerRewardsUpdated` event was emitted.
4. In the same block, the sequencer attempts:
   ```
   staking.update_rewards(staker_address=<any_staker>, disable_rewards=false)
   ```
5. The call reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeat step 2 at the start of every block; no staker ever accumulates rewards. [7](#0-6) [8](#0-7)
