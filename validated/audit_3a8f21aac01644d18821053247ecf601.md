### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions with Nonce=1 When Any Account Transaction Exists in Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (the account's signature check) for any invoke transaction with `nonce=1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` for **any** transaction in the pool, not only a `deploy_account`. An attacker can submit a forged invoke with an arbitrary/invalid signature for any account that has on-chain nonce `0` and any transaction in the mempool, and the gateway will admit it without verifying the signature.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the simultaneous `deploy_account + invoke` flow: because the account contract does not exist yet, `__validate__` cannot run. The function returns `true` (skip) when three conditions hold:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

The comment justifies the broad check: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [2](#0-1) 

However, the second branch of that reasoning is the flaw. If the account has a **regular invoke (nonce=0)** in the pool, the account **is already deployed** and `__validate__` **can** run. The code skips it anyway.

`account_tx_in_pool_or_recent_block` checks for any transaction, not specifically `deploy_account`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

`StatefulValidator::perform_validations` then returns early without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce/fee/tip fields via `ValidationArgs`; it does not verify the signature:

```rust
pub struct ValidationArgs {
    pub address: ContractAddress,
    pub account_nonce: Nonce,
    pub tx_hash: TransactionHash,
    pub tx_nonce: Nonce,
    pub tip: Tip,
    pub max_l2_gas_price: GasPrice,
}
``` [6](#0-5) 

No other guard in the gateway path verifies the signature when `skip_validate=true`.

### Impact Explanation

An attacker can inject an invoke transaction with a completely invalid/forged signature into the mempool without any signature verification. This matches **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Secondary impact: if the attacker's forged invoke (nonce=1) is admitted **before** the victim's legitimate invoke (nonce=1), the mempool rejects the legitimate transaction as `DuplicateNonce`:

```rust
return Err(MempoolError::DuplicateNonce { address, nonce });
``` [7](#0-6) 

This is a targeted denial-of-service against the victim's post-deploy invoke.

### Likelihood Explanation

The preconditions are easily observable on-chain and in the public mempool:
- Account on-chain nonce = 0 (trivially visible)
- Any transaction for that account in the mempool (observable via RPC)
- Attacker submits a forged invoke with nonce=1 before the victim does

No privileged access is required. The attacker only needs to monitor the mempool and submit a crafted transaction.

### Recommendation

**Short term:** In `skip_stateful_validations`, replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the sender address. Add a dedicated mempool query such as `has_deploy_account_in_pool(address)` that inspects the transaction type.

**Long term:** Consider whether the skip-validate UX optimization is worth the admission invariant it breaks. If kept, the condition must be narrowed to only the case where the account contract genuinely does not exist yet (class hash = 0 at the sender address), which can be verified directly from state rather than inferred from mempool contents.

### Proof of Concept

1. Alice deploys account `A` (class hash ≠ 0, on-chain nonce = 0).
2. Alice submits a legitimate invoke `Tx_A` with nonce=0. It passes `__validate__` and enters the mempool. Now `account_tx_in_pool_or_recent_block(A)` returns `true`.
3. Mallory crafts `Tx_M`: an invoke for account `A`, nonce=1, signature = `[0x0, 0x0]` (invalid).
4. Gateway stateless validation passes (no signature content check).
5. Gateway stateful validation: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool=true` → `skip_stateful_validations` returns `true` → `execution_flags.validate=false` → `__validate__` is never called → `Tx_M` is admitted to the mempool.
6. Alice submits her legitimate `Tx_B` with nonce=1 and a valid signature. The mempool rejects it: `MempoolError::DuplicateNonce { address: A, nonce: 1 }`.
7. Alice's legitimate nonce=1 transaction is permanently blocked until `Tx_M` is evicted (e.g., by TTL or fee escalation). [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
