### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Unvalidated Group Elements — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator blindly accumulates `CKDOutput` group elements received from each participant with no cryptographic validation. A single malicious participant can transmit arbitrary BLS12-381 G1 points in place of their honest share, silently corrupting the aggregated `(Y, C)` output that all honest parties accept as the final CKD result.

---

### Finding Description

The CKD protocol is a one-round additive aggregation. Each participant computes a Lagrange-weighted share `(λ_i · Y_i, λ_i · C_i)` and sends it privately to the coordinator. The coordinator sums all received values:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-56
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

There is no check that the received `big_y` or `big_c` is consistent with the sender's committed public key share or with any zero-knowledge proof of correct computation. The participant's honest contribution is computed in `compute_signature_share`:

```rust
// src/confidential_key_derivation/protocol.rs  lines 165-181
let big_y = ElementG1::generator() * y.0;
let big_s = hash_point * private_share.to_scalar();
let big_c = big_s + app_pk * y.0;
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
Ok((norm_big_y, norm_big_c))
```

A malicious participant ignores this computation entirely and instead calls `chan.send_private(waitpoint, coordinator, &(arbitrary_y, arbitrary_c))` with any two G1 points of their choosing. The coordinator has no mechanism to detect or reject this substitution.

**Analogy to the 1inch finding:** In the 1inch vulnerability, the maker's `_getMakingAmount` hook returns an out-of-bounds value that triggers a recomputation of `actualTakingAmount` via `_getTakingAmount`, and the threshold check `actualMakingAmount * takingAmount < thresholdAmount * actualTakingAmount` is bypassed because the maker controls both sides. Here, the participant's send path is the equivalent "hook": it is entirely under the participant's control, the coordinator performs no bounds or consistency check on the received elements, and the final aggregated output is accepted without verification — the threshold-signatures analogue of the bypassed `MakingAmountTooLow` guard.

---

### Impact Explanation

The CKD output `(Y, C)` is used by the application to unmask the confidential derived key:

```
derived_key = C − app_sk · Y  =  x · H(pk ‖ app_id)
```

If a malicious participant substitutes arbitrary `(Y′, C′)`, the coordinator outputs `(Y_honest + Y′, C_honest + C′)`. The application unmasks this to:

```
(C_honest + C′) − app_sk · (Y_honest + Y′)
= x · H(pk ‖ app_id) + (C′ − app_sk · Y′)
```

The derived key is shifted by an attacker-controlled offset `C′ − app_sk · Y′`. Honest parties accept this corrupted value as the legitimate CKD output. This constitutes **corruption of CKD outputs so honest parties accept an incorrect derived key** — matching the High impact tier: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. The attacker needs only to be a registered participant (no special privilege beyond membership). The attack requires no cryptographic break, no leaked key, and no external dependency — only the ability to send a message on the protocol channel, which every participant already does. The protocol provides no detection mechanism, so the attack is silent and repeatable across every CKD invocation that includes the malicious participant.

---

### Recommendation

Add a zero-knowledge proof of correct share computation alongside each `(norm_big_y, norm_big_c)` message. Concretely, each participant should prove in zero knowledge that:

1. `norm_big_y = λ_i · y_i · G` for some scalar `y_i` they know.
2. `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)` where `x_i` is consistent with their committed public key share `X_i = x_i · G` established during DKG.

A standard Chaum-Pedersen or Schnorr-based proof suffices for (1); a proof of discrete-log equality between `norm_big_c − λ_i · y_i · A` and `λ_i · X_i` (in the `H(pk ‖ app_id)` base) covers (2). The coordinator must verify these proofs before accumulating any participant's contribution.

---

### Proof of Concept

**Setup:** 3-of-3 CKD session. Participants P1 (honest), P2 (honest), P3 (malicious). Coordinator = P1.

1. P1 and P2 compute honest `(norm_big_y_i, norm_big_c_i)` per `compute_signature_share` and send them to P1.
2. P3, instead of calling `compute_signature_share`, directly calls:
   ```rust
   chan.send_private(waitpoint, coordinator, &(ElementG1::identity(), ElementG1::generator()))
   ```
   sending `(identity, G)` — two arbitrary G1 points unrelated to P3's key share.
3. The coordinator at lines 50–56 accumulates without validation:
   ```
   norm_big_y = norm_big_y_1 + norm_big_y_2 + identity   // = honest Y
   norm_big_c = norm_big_c_1 + norm_big_c_2 + G           // = honest C + G
   ```
4. The coordinator returns `CKDOutput::new(honest_Y, honest_C + G)`.
5. The application unmasks: `(honest_C + G) − app_sk · honest_Y = x·H(pk‖app_id) + G`, a key shifted by `G` — an attacker-controlled, non-zero offset.
6. No error is raised anywhere in the protocol. All honest parties accept the corrupted output. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
