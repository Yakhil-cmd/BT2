Let me look at the mempool's `validate_tx` implementation and the `StatefulValidatorTrait` to confirm the full picture.

### Title
`skip_stateful_validations` Bypasses `__validate__` for Already-Deployed Accounts with Nonce 0 — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` uses `account_nonce == Nonce(Felt::ZERO)` to infer "account not yet deployed," but this is incorrect. A deployed account that has not yet executed any transaction also has nonce 0. When a legitimate nonce-0 invoke for such an account is already in the mempool, an attacker can submit a nonce-1 invoke with an arbitrary/fake signature, which passes all gateway checks and is admitted to the mempool without the `__validate__` entry point ever being called.

---

### Finding Description

`skip_stateful_validations` (lines 429–461) is the gateway's UX shortcut for the deploy-account + invoke(nonce=1) simultaneous-submission pattern. Its logic is:

```
if tx.nonce() == 1 && account_nonce == 0 {
    if account_tx_in_pool_or_recent_block(sender) {
        return true  // skip __validate__
    }
}
```

The code comment at line 436 states: *"the account nonce is zero, meaning the account was not deployed yet."* This assumption is wrong. An account can be deployed and have nonce 0 if it has not yet executed any transaction.

`account_tx_in_pool_or_recent_block` (line 697 of `mempool.rs`) returns `true` when:

```rust
self.state.contains_account(account_address)
    || self.tx_pool.contains_account(account_address)
```

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool — not only a `DeployAccount`. So when a legitimate nonce-0 invoke for a deployed-but-nonce-0 account is already in the pool, the condition fires and `skip_stateful_validations` returns `true`.

`run_validate_entry_point` then sets `ExecutionFlags { validate: !skip_validate = false, ... }` and calls `StatefulValidatorTrait::validate` → `perform_validations`. Inside `perform_validations` (line 79):

```rust
if !tx.execution_flags.validate {
    return Ok(());   // returns before __validate__ is called
}
```

The `__validate__` entry point — which is the account's signature-verification function — is never invoked. The mempool's `validate_tx` only checks nonce ordering and duplicate hashes; it does not verify signatures. The attacker's nonce-1 invoke is therefore admitted to the mempool with no signature check.

This is the direct analog of the original M-21 bug:

| Original (Malt) | Sequencer |
|---|---|
| `auctionActive` — is the auction currently running? | `account_nonce == 0` — is the account "not deployed"? |
| `auctionExists` — does the auction object exist at all? | `account_tx_in_pool_or_recent_block` — does any tx exist for this account? |
| Inactive-but-existing auction satisfies the outer check but causes `triggerAuction` to return early, yet the caller still gets the reward | Deployed-but-nonce-0 account satisfies the outer nonce check but causes `skip_stateful_validations` to return `true`, yet `__validate__` is never called |

---

### Impact Explanation

An attacker can submit an invoke transaction with nonce 1 carrying an arbitrary signature for any account that:
1. Is already deployed (contract exists at the address), and
2. Has `account_nonce == 0` in the last committed block (no transactions executed yet), and
3. Has any pending nonce-0 invoke in the mempool.

The gateway admits the attacker's nonce-1 invoke without calling `__validate__`, bypassing the account's signature-verification logic entirely at the admission layer. This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The preconditions are realistic:
- Newly deployed accounts with nonce 0 are common (every account starts here).
- A nonce-0 invoke in the mempool is the normal first transaction after deployment.
- The attacker only needs to observe the mempool for such a condition and race to submit a nonce-1 invoke with a fake signature before the nonce-0 invoke is committed.

No privileged access is required. The attack is fully unprivileged and requires only standard RPC access to the gateway.

---

### Recommendation

`skip_stateful_validations` must distinguish between "account not deployed" and "account deployed with nonce 0." The correct guard is to check whether the account contract actually exists at the address (i.e., `get_class_hash_at(address) != ClassHash::default()`). If the class hash is non-zero, the account is deployed and the skip must not be triggered regardless of the nonce. Alternatively, the mempool should expose a dedicated `deploy_account_in_pool(address)` query that returns `true` only when a `DeployAccount` transaction (not any transaction) is pending for the address, and `skip_stateful_validations` should call that instead of `account_tx_in_pool_or_recent_block`.

---

### Proof of Concept

1. Deploy account `A` (class hash set, nonce = 0 in committed state).
2. Legitimate user submits `Invoke(sender=A, nonce=0, valid_signature)` → gateway calls `__validate__`, passes, tx enters mempool pool.
3. Attacker submits `Invoke(sender=A, nonce=1, fake_signature)`.
4. Gateway `validate_nonce`: `0 <= 1 <= max_allowed_nonce_gap` → **passes**.
5. Gateway `validate_by_mempool` (`validate_tx`): checks nonce ordering and duplicate hash only → **passes**.
6. `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (account deployed but no committed txs)
   - `account_tx_in_pool_or_recent_block(A)` → `tx_pool.contains_account(A)` is `true` (nonce-0 invoke is in pool) → returns **`true`**.
7. `run_validate_entry_point` sets `ExecutionFlags { validate: false }`.
8. `StatefulValidator::perform_validations`: `if !tx.execution_flags.validate { return Ok(()); }` — **`__validate__` is never called**.
9. Attacker's nonce-1 invoke with fake signature is **admitted to the mempool** without any signature verification.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
