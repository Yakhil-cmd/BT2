### Title
Multi-NFT Royalty Amount Understated in Offer Confirmation Display - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
In `formatAmountWithRoyalties`, when an offer involves multiple NFTs with royalties, the code incorrectly divides the fungible amount by the number of NFTs before applying each royalty percentage. This causes the `amountWithRoyalties` shown in the signing/confirmation dialog to be understated, leading a user to approve an offer believing they will spend less than they actually will.

### Finding Description
The function `formatAmountWithRoyalties` in `parseCommandDisplay.ts` computes the total cost including royalties as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);  // line 363
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

The correct formula is: for each NFT, apply its royalty percentage to the **full** fungible amount, then sum. Instead, the code first divides the amount equally among all NFTs (`splitAmount`), then applies each royalty only to that fraction. This is mathematically equivalent to averaging the royalty percentages and applying the average once — which is wrong when percentages differ.

**Correct calculation** (2 NFTs: 5% and 0.1%, amount = 100,000,000 mojos):
- Royalty₁ = 100,000,000 × 500/10,000 = 5,000,000
- Royalty₂ = 100,000,000 × 10/10,000 = 100,000
- Total = 105,100,000 mojos → **0.0001051 XCH**

**Actual (buggy) calculation:**
- splitAmount = 100,000,000 / 2 = 50,000,000
- Royalty₁ = 50,000,000 × 500/10,000 = 2,500,000
- Royalty₂ = 50,000,000 × 10/10,000 = 50,000
- Total = 102,550,000 mojos → **0.00010255 XCH** (understated by ~2.5%)

The test fixture at line 320 confirms the codebase encodes this wrong value as the expected output: [2](#0-1) 

This `amountWithRoyalties` value is surfaced to the user in the WalletConnect/Electron confirmation dialog via `walletDeltaToDisplay`: [3](#0-2) 

### Impact Explanation
A user accepting a multi-NFT offer (via `take_offer` or `create_offer_for_ids`) sees a confirmation dialog showing a total spend amount that is **lower than what will actually be deducted** from their wallet. The on-chain royalty enforcement is correct; only the pre-approval display is wrong. The user approves based on a false (understated) total, constituting display of the wrong amount at the signing approval step. This maps to the High impact category: "corruption or spoofing of offer state that causes a user to approve the wrong amount."

### Likelihood Explanation
Triggered whenever a user accepts or creates an offer involving two or more NFTs that each carry royalties. Multi-NFT offers are a supported and documented feature of the Chia offer system. No special attacker capability is required — a legitimately crafted offer file with two royalty-bearing NFTs is sufficient to trigger the miscalculation.

### Recommendation
Remove the `splitAmount` division. Each royalty percentage must be applied to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

Update the corresponding test expectation in `parseCommandDisplay.test.ts` from `'0.00010255'` to `'0.0001051'`. [2](#0-1) 

### Proof of Concept
1. Craft an offer file containing two NFTs: NFT-A with `royalty_percentage = 500` (5%) and NFT-B with `royalty_percentage = 10` (0.1%), requesting 100,000,000 mojos (0.0001 XCH).
2. Present the offer to a Chia GUI user via WalletConnect or the offer import flow.
3. The confirmation dialog displays `amountWithRoyalties: 0.00010255 XCH`.
4. The user approves, believing total cost is 0.00010255 XCH.
5. Actual on-chain deduction is 0.0001051 XCH — approximately 2.5% more than displayed.

The discrepancy scales with the difference between royalty percentages and the number of NFTs involved; a high-royalty NFT paired with a low-royalty NFT maximizes the understatement.
