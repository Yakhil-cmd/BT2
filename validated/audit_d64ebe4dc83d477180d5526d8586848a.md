### Title
Missing Threshold Validation in CKD Protocol Allows Corrupted Key Derivation Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` omits the threshold-vs-participant-count validation that every other protocol entry point in the library enforces. An unprivileged library caller can invoke `ckd()` with fewer participants than the reconstruction threshold, the protocol completes without error, and the coordinator outputs a cryptographically incorrect `CKDOutput` whose embedded BLS signature is unverifiable — permanently denying the requesting application its derived key.

---

### Finding Description

Every other protocol entry point in the library accepts a `threshold` parameter and rejects calls where `participants.len() < threshold`:

- `assert_key_invariants` (keygen) — rejects if `threshold > participants.len()` [1](#0-0) 
- `assert_reshare_keys_invariants` (reshare) — rejects if intersection is smaller than old threshold [2](#0-1) 
- `assert_sign_inputs` (FROST sign) — rejects if `threshold.value() > participants.len()` [3](#0-2) 
- `validate_triple_inputs` (OT triple generation) — same guard [4](#0-3) 

`ckd()` is the sole exception. Its entire validation block is:

```rust
if participants.len() < 2 {
    return Err(InitializationError::NotEnoughParticipants { … });
}
// kick out duplicates
// ensure me is present
// ensure coordinator is present
// — no threshold check —
``` [5](#0-4) 

`ckd()` does not accept a threshold argument at all, and `KeygenOutput` stores no threshold field:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
}
``` [6](#0-5) 

Inside `compute_signature_share`, each participant computes their Lagrange-weighted contribution using only the participants present in the current call:

```rust
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [7](#0-6) 

The coordinator then blindly sums all received contributions:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [8](#0-7) 

When the participant count is below the reconstruction threshold, the Lagrange interpolation identity `∑ λᵢ · xᵢ = msk` does not hold. The resulting `C` does not equal `msk · H(pk, app_id) + a · Y`, so the app's decryption `sig = C − a · Y` does not equal `msk · H(pk, app_id)`, and BLS signature verification fails.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator returns a structurally well-formed `CKDOutput` with no error. The requesting application has no way to distinguish this from a correct output until it attempts to verify the embedded BLS signature, at which point the derived key is permanently unusable. Because the CKD output is deterministic per `(app_id, pk)`, re-running the protocol with the same (under-threshold) participant set produces the same wrong output; the application cannot recover without a correctly-sized participant set.

---

### Likelihood Explanation

**Medium.** The threshold is not stored in `KeygenOutput` and is not a parameter to `ckd()`, so any caller that does not independently track and enforce the threshold — including one that simply passes the currently-online participants without checking against the keygen threshold — silently triggers the corruption. The absence of a compile-time or runtime guard makes accidental misuse straightforward.

---

### Recommendation

1. **Short term:** Add a `threshold: impl Into<ReconstructionLowerBound>` parameter to `ckd()` and reject calls where `participants.len() < threshold.value()`, mirroring the pattern in `assert_sign_inputs`.
2. **Long term:** Store the reconstruction threshold inside `KeygenOutput` so that all downstream protocol functions can self-validate without requiring callers to track it separately.

---

### Proof of Concept

```rust
// DKG was run with threshold = 5, participants = [P1..P5]
// Caller mistakenly passes only 2 participants to ckd()
let result = ckd(
    &[p1, p2],          // only 2 of 5 threshold participants
    coordinator,
    me,
    key_pair,           // KeygenOutput from a threshold-5 DKG
    app_id,
    app_pk,
    rng,
)
.unwrap();              // succeeds — no threshold check fires

// Protocol runs to completion; coordinator emits CKDOutput
// App computes: sig = C − a·Y  ≠  msk·H(pk, app_id)
// BLS verification: FAIL — derived key is permanently unusable
```

The call succeeds at the `ckd()` entry point [9](#0-8)  because the only guard is `participants.len() < 2`. The corrupted output propagates to the application with no indication of failure.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
