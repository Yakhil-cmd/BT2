### Title
Cross-Address COA Ownership Proof Replay via Missing `evmAddress` Binding in `signedData` — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function that verifies signatures over arbitrary `signedData` but does not enforce that `signedData` commits to the target `evmAddress`. A signature produced by a Cadence account to prove ownership of one COA (Cadence-Owned Account) can be replayed by any unprivileged caller to pass the same proof for a *different* COA owned by the same Cadence account. The code itself acknowledges this gap in a developer comment, but the mitigation it cites (the EVM precompile always passes `address(this)`) does not protect direct Cadence-level callers of the function, which is `access(all)`.

---

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. It verifies that the provided `signatures` over `signedData` are cryptographically valid for the given Cadence account keys.
2. It verifies that the COA resource stored at `path` has the given `evmAddress`. [1](#0-0) 

Crucially, there is no step that binds the two checks together — the function never verifies that `signedData` encodes `evmAddress`. The developer comment at lines 1003–1009 explicitly acknowledges this: [2](#0-1) 

The function is declared `access(all)`: [3](#0-2) 

This means any Cadence script or transaction — submitted by any unprivileged sender — can call it directly with attacker-chosen arguments, including a substituted `path` and `evmAddress`.

The comment's cited mitigation — "the EVM-side precompile always passes the calling COA's address as the `evmAddress` argument" — applies only to the `verifyCOAOwnershipProof` precompile path invoked from EVM (e.g., from `coa.sol`'s `isValidSignature`): [4](#0-3) 

That path is safe because `address(this)` is always the calling COA. However, the Cadence-level `validateCOAOwnershipProof` is separately and directly callable by any Cadence script or transaction, with no such constraint. The precompile merely delegates to `validateCOAOwnershipProof` via `coaOwnershipProofValidator`: [5](#0-4) 

Any off-chain authentication flow or on-chain Cadence protocol that calls `EVM.validateCOAOwnershipProof` directly — and does not itself encode `evmAddress` into `signedData` — is vulnerable to cross-address replay.

---

### Impact Explanation

An attacker who observes a valid COA ownership proof (signature + `signedData`) produced by a victim for COA_A can replay it against COA_B if the victim's Cadence account also owns COA_B. The attacker supplies the same `address`, `signedData`, `keyIndices`, and `signatures`, but substitutes `path` and `evmAddress` to point to COA_B. The function returns `isValid: true` for COA_B even though the victim never signed anything binding to COA_B.

Any protocol that gates a privileged action (e.g., bridge withdrawal, asset transfer, role assignment) on a successful `EVM.validateCOAOwnershipProof` result — without independently verifying that `signedData` encodes the target `evmAddress` — can be bypassed. The impact is unauthorized proof of COA ownership, leading to unauthorized access to on-chain assets or privileged protocol actions. [6](#0-5) 

---

### Likelihood Explanation

- `validateCOAOwnershipProof` is `access(all)` and reachable by any unprivileged Cadence script or transaction submitted to any Access node.
- The attacker only needs to observe a valid proof (e.g., from a prior on-chain transaction or an off-chain authentication flow that publishes the proof).
- The precondition — victim has multiple COAs — is not enforced against by the protocol; the Cadence type system and storage model allow any number of COAs per account.
- The code comment itself documents the attack vector, indicating the design team is aware it is a real possibility. [7](#0-6) 

---

### Recommendation

Enforce the binding inside `validateCOAOwnershipProof` itself rather than relying on callers. Before verifying signatures, the function should assert that `signedData` encodes `evmAddress` (e.g., by hashing `evmAddress` into the data that is verified, or by requiring `signedData` to be a canonical encoding of `evmAddress`). This removes the trust assumption placed on callers and makes the function safe regardless of how it is invoked. [1](#0-0)

### Proof of Concept

**Setup:**
- Alice's Cadence account (`0xAlice`) owns two COAs:
  - COA_A at `/public/coa_a` with EVM address `0xAAAA...`
  - COA_B at `/public/coa_b` with EVM address `0xBBBB...`

**Step 1 — Victim produces a legitimate proof for COA_A:**
Alice signs `signedData = <some bytes>` with her account key to prove ownership of COA_A. The proof `(address=0xAlice, path=/public/coa_a, signedData, keyIndices, signatures, evmAddress=0xAAAA...)` is submitted on-chain or shared off-chain.

**Step 2 — Attacker observes the proof:**
The attacker extracts `(signedData, keyIndices, signatures)` from the on-chain transaction or off-chain channel.

**Step 3 — Attacker replays against COA_B:**
The attacker submits a Cadence script:
```cadence
import EVM from 0xEVM

access(all)
fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: 0xAlice,           // same Cadence account
        path: /public/coa_b,        // substituted: COA_B's path
        signedData: <same bytes>,   // replayed from COA_A proof
        keyIndices: <same>,         // replayed
        signatures: <same>,         // replayed
        evmAddress: <0xBBBB...>     // substituted: COA_B's EVM address
    )
}
```

**Step 4 — Result:**
The function verifies the signatures over `signedData` (valid, because Alice's keys signed it), then checks that the COA at `/public/coa_b` has address `0xBBBB...` (true). It returns `ValidationResult(isValid: true)` — falsely asserting that Alice authorized a proof for COA_B. [8](#0-7) [9](#0-8)
