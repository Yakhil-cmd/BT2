### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Non-Deploy-Account Trigger — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip the blockifier `__validate__` entry-point call only when a `deploy_account` transaction for the same sender is pending in the mempool (UX feature: deploy + invoke submitted together). The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account in the pool, not exclusively a `deploy_account`. An attacker who controls a deployed account with on-chain nonce 0 can first submit a valid invoke at nonce 0, then submit a second invoke at nonce 1 with an **arbitrary/invalid signature**. The second transaction passes every pre-check and then has its `__validate__` call suppressed, entering the mempool without signature verification.

### Finding Description

`skip_stateful_validations` fires when three conditions hold simultaneously:

1. The incoming transaction is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` (nonce = 1).
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain nonce = 0).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

The code comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is incorrect. `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` whenever the account has **any** transaction in the pool — including a regular invoke at nonce 0. There is no type-check for `DeployAccount`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

With `validate: false`, `validate_tx` returns `Ok(None)` immediately without executing `__validate__`:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [4](#0-3) 

Neither `validate_state_preconditions` nor `validate_by_mempool` checks the transaction signature; they only verify nonce range and mempool-level nonce ordering: [5](#0-4) 

The default `max_allowed_nonce_gap` is 200, so nonce 1 is always within the accepted range when the on-chain nonce is 0: [6](#0-5) 

### Impact Explanation

An attacker can inject an invoke transaction carrying an **invalid or forged signature** into the mempool without the sequencer ever calling the account's `__validate__` entry point. The transaction will be sequenced, fail at blockifier execution time (nonce increment already happened in `perform_pre_validation_stage`), and consume sequencer resources. More critically, the admission invariant — *every accepted invoke must have passed account-controlled signature verification* — is broken at the gateway/mempool boundary.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The preconditions are trivially achievable by any unprivileged user:
- Own or control any deployed Starknet account whose on-chain nonce is currently 0 (e.g., a freshly deployed account that has not yet had its first transaction committed).
- Submit one valid invoke at nonce 0 (passes full `__validate__`).
- Immediately submit a second invoke at nonce 1 with a garbage signature.

No privileged access, no special contract, no race condition beyond normal mempool timing is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a **`DeployAccount`** transaction for the sender is pending. Alternatively, add a dedicated `deploy_account_in_pool(address)` method to the mempool that inspects transaction types, and use that in `skip_stateful_validations`. [7](#0-6) 

### Proof of Concept

```
1. Deploy account A (class with __validate__ that checks ECDSA signature).
   On-chain state: nonce(A) = 0.

2. POST /gateway/add_transaction:
     Invoke V3, sender=A, nonce=0, signature=<valid ECDSA sig over tx_hash_0>
   → passes validate_nonce (0 == 0)
   → passes validate_by_mempool (nonce 0 >= account_nonce 0)
   → skip_stateful_validations: nonce != 1, returns false
   → run_validate_entry_point: __validate__ called, passes
   → tx_0 admitted to mempool; mempool.tx_pool.contains_account(A) = true

3. POST /gateway/add_transaction:
     Invoke V3, sender=A, nonce=1, signature=<all zeros / garbage>
   → passes validate_nonce (1 in [0, 200])
   → passes validate_by_mempool (staged nonce for A is 1, tx_nonce 1 >= 1)
   → skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(A) == true  ✓  (tx_0 is in pool)
       → returns true
   → run_validate_entry_point: validate=false → __validate__ NOT called
   → tx_1 with invalid signature admitted to mempool  ✓
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
