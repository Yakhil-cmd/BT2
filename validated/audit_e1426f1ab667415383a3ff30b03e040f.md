### Title
One-Sided `delta_gas` Adjustment in `compute_gas_refund` Silently Overcharges Users When Native Resource Consumption Is Lower Than EVM Gas Consumption — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` correction is applied **only when positive** (native resource cost exceeds EVM gas cost). When `delta_gas < 0` — meaning the transaction consumed *less* native resource than the EVM gas payment implies — the excess gas is **never refunded** to the user. The `TODO` comment in the code explicitly flags this as an unresolved asymmetry. The result is a systematic overcharge of users whose transactions are EVM-gas-heavy but native-resource-light, with the surplus flowing to the operator.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` computes the final `gas_used` figure that determines how much the user pays and how much is refunded:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native cost > EVM gas cost → charge extra gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The design intent, documented in `docs/double_resource_accounting.md`, is:

> If `deltaGas > 0`, we add it to `gasUsed` … This ensures that gas estimation will include additional gas to cover for native resources using just base fee. [2](#0-1) 

The documentation describes only the positive branch. The negative branch — where `native_used / native_per_gas < gas_used` — is left unhandled. The `TODO` comment is the codebase's own acknowledgement that this path is incomplete.

This is structurally identical to the external report's root cause: a cap/limit check that adjusts a resource amount in one direction but silently skips the symmetric adjustment in the other direction, causing the remainder to be permanently lost to the user.

---

### Impact Explanation

When `delta_gas < 0`:

- `gas_used` is **not reduced**, so the user is charged for more EVM gas than their native resource consumption warrants.
- `total_gas_refund = gas_limit - gas_used` is correspondingly smaller.
- The operator (coinbase) receives `gas_used * gas_price`, which includes the unrefunded surplus.

The overcharge per transaction is `|delta_gas| * gas_price`. For a transaction with `gas_limit = 1,000,000`, `gas_used = 500,000`, and `native_used / native_per_gas = 50,000`, the user is overcharged by `450,000 * gas_price` — a material loss at any non-trivial gas price. The funds are not burned; they are transferred to the operator, making this a direct, measurable financial loss for the transaction sender.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever a transaction consumes proportionally more EVM gas than native resources. This is the common case for:

- Transactions with large calldata or complex EVM logic that does **not** generate significant pubdata (storage writes).
- Transactions where the `native_per_gas` ratio is low (i.e., `gas_price` is low relative to `native_price`).
- Any transaction where the EVM execution path is gas-intensive but the state diff is small.

This is a routine scenario in normal protocol operation, not an edge case. Every such transaction silently overcharges the sender.

---

### Recommendation

Apply the `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (clamped to prevent underflow below `minimal_gas_used`):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else {
    // delta_gas < 0: native cost < EVM gas cost → refund excess gas
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This mirrors the fix in the external report: process the remainder up to the cap before finalising the accounting state.

---

### Proof of Concept

1. User submits an EIP-1559 transaction with `gas_limit = 500_000`, `max_fee_per_gas = 1000`, `max_priority_fee_per_gas = 1000`.
2. The transaction executes a computation-heavy EVM path (e.g., many arithmetic opcodes) that writes to **zero** storage slots, generating no pubdata.
3. After execution: `gas_used (EVM) = 400_000`; `native_used = 2_000_000`; `native_per_gas = 10` (derived from `gas_price / native_price`).
4. `delta_gas = (2_000_000 / 10) - 400_000 = 200_000 - 400_000 = -200_000`.
5. The `if delta_gas > 0` branch is **not taken**. `gas_used` remains `400_000`.
6. User refund: `(500_000 - 400_000) * 1000 = 100_000_000 wei`.
7. Correct refund should be: `(500_000 - 200_000) * 1000 = 300_000_000 wei`.
8. User is overcharged by `200_000_000 wei` (0.0002 ETH at these prices), which flows to the operator. [3](#0-2)
