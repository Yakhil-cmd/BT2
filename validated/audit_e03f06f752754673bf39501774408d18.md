### Title
Missing Verification of Participant Shares in CKD Coordinator Allows Output Corruption - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function aggregates participant-supplied ElGamal share pairs `(big_y, big_c)` without performing any verification of their correctness. A single malicious participant can send arbitrary group elements, silently corrupting the aggregated CKD output that the coordinator returns to the client. The DKG protocol (`do_keyshare`) performs the analogous step correctly by verifying proofs of knowledge and commitment hashes before accepting any participant contribution.

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and blindly adds them together: [1](#0-0) 

There is no check that:
- `norm_big_y` is of the form `λ_i · y_i · G` for any scalar `y_i` (i.e., a valid Lagrange-weighted nonce commitment), or
- `norm_big_c` is of the form `λ_i · (x_i · H(pk, app_id) + y_i · A)` (i.e., correctly uses the participant's secret share `x_i` and the same nonce `y_i`).

The correct share computation is: [2](#0-1) 

A malicious participant simply sends any two group elements instead of the values produced by `compute_signature_share`. The coordinator has no mechanism to detect the substitution.

By contrast, the DKG protocol (`do_keyshare` in `src/dkg.rs`) requires every participant to supply a proof of knowledge of their secret coefficient and a commitment hash, both of which are verified before any share is accepted: [3](#0-2) 

The CKD coordinator performs no equivalent step — the missing verification call is the direct analog of the missing `setTokenParams()` call in the external report.

### Impact Explanation

The final CKD output is:

```
big_Y = Σ λ_i · Y_i
big_C = Σ λ_i · C_i
```

The client unmasks with `big_C − a · big_Y` expecting `msk · H(pk, app_id)`. If one participant substitutes arbitrary `(Y_malicious, C_malicious)`, the sum is shifted by an attacker-chosen offset, producing an incorrect derived key. Honest parties receive and accept a corrupted CKD output — they have no way to detect the manipulation because the coordinator returns only the aggregated result.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the CKD session can trigger this. The participant role is explicitly in scope per the prompt. No special privilege is required beyond being a listed participant. The attack requires only sending two arbitrary `ElementG1` values instead of the correctly computed ones — a trivial deviation from the protocol.

### Recommendation

Before aggregating, the coordinator should require each participant to supply a Chaum-Pedersen (or equivalent) zero-knowledge proof demonstrating that:
1. `big_y` is a valid scalar multiple of the generator (`big_y = y · G`), and
2. `big_c` is correctly formed as `x_i · H(pk, app_id) + y · A` using the same `y` and the participant's committed public share.

This mirrors the proof-of-knowledge verification already present in `do_keyshare` for the DKG protocol.

### Proof of Concept

1. Honest participants 1…n−1 run `compute_signature_share` correctly.
2. Malicious participant `n` sends `(ElementG1::identity(), ElementG1::identity())` (or any chosen group elements) instead of its real share.
3. The coordinator executes: [4](#0-3) 

4. The aggregated `norm_big_y` and `norm_big_c` are shifted by `λ_n · (0 − Y_n_correct)` and `λ_n · (0 − C_n_correct)` respectively.
5. The coordinator returns a `CKDOutput` whose `unmask(app_sk)` does not equal `msk · H(pk, app_id)`.
6. The client decrypts an incorrect derived key with no error signal — the corruption is silent and permanent for this CKD invocation.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
