### Title
Missing Deadline in COA Ownership Proof Signatures Enables Indefinite Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `validateCOAOwnershipProof` function and the `verifyCOAOwnershipProof` EVM precompile accept COA ownership proof signatures that contain no deadline, expiry, or nonce. Once a Flow account owner signs any `signedData` hash to produce a `COAOwnershipProof`, that proof remains cryptographically valid forever. Any party who obtains the proof can replay it at any future time without the signer's current consent, enabling unauthorized ERC-1271 authorizations on behalf of the COA owner.

---

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies a set of Flow account key signatures over an arbitrary `signedData` blob, then confirms the signing account controls the named COA resource. The function signature is:

```cadence
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
``` [1](#0-0) 

The underlying `COAOwnershipProof` struct that carries the signatures contains only `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`:

```go
type COAOwnershipProof struct {
    KeyIndices     KeyIndices
    Address        FlowAddress
    CapabilityPath PublicPath
    Signatures     Signatures
}
``` [2](#0-1) 

There is **no deadline, expiry timestamp, or nonce** in either the proof struct or the signed message. The verification logic in `validateCOAOwnershipProof` performs only three checks: key-index/signature count match, cryptographic signature validity, and COA address match. No temporal bound is ever enforced. [3](#0-2) 

This proof is consumed by the `verifyCOAOwnershipProof` Cadence Arch precompile, which is called from the COA Solidity contract's ERC-1271 `isValidSignature` implementation:

```solidity
function isValidSignature(
    bytes32 _hash,
    bytes memory _sig
) external view virtual returns (bytes4){
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig
        )
    );
    ...
}
``` [4](#0-3) 

The precompile dispatcher routes the call through `coaOwnershipProofValidator`, which invokes `validateCOAOwnershipProof` with the caller-supplied `signedData` and proof: [5](#0-4) 

The code itself acknowledges a related replay concern (cross-address replay) but does not address the missing temporal bound:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [6](#0-5) 

---

### Impact Explanation

A COA owner who signs a `bytes32` hash for a specific ERC-1271 purpose (e.g., a Seaport order, a permit-style token approval, a meta-transaction, or a DAO vote) cannot revoke that authorization without revoking their entire Flow account key. Any party who obtains the `COAOwnershipProof` can present it to any ERC-1271-consuming EVM contract at any future time. Because the COA's `isValidSignature` delegates entirely to `verifyCOAOwnershipProof` with no expiry check, the contract will return `ValidERC1271Signature` indefinitely.

Concrete consequences include:
- Unauthorized execution of expired limit orders or permit approvals backed by a COA
- Inability to selectively revoke a single authorization without rotating the entire Flow account key
- Any EVM protocol that relies on ERC-1271 for access control (multisig, DAO, bridge) inherits this unbounded validity window for COA signers

---

### Likelihood Explanation

The ERC-1271 interface is live and deployed as part of the COA contract. Any unprivileged EVM transaction sender can call `isValidSignature` on a COA address with a previously captured proof. Obtaining a valid proof requires only that the COA owner has ever signed a hash for any ERC-1271 purpose and that the proof was observable (e.g., submitted in a prior transaction). No privileged access, key compromise, or node control is required to replay it.

---

### Recommendation

1. Add a `Deadline` (block height or timestamp) field to `COAOwnershipProof` and include it in the signed message.
2. In `validateCOAOwnershipProof`, enforce that the current block height/timestamp is before the deadline before accepting the proof.
3. Optionally, add a per-account nonce to `COAOwnershipProof` so that a COA owner can invalidate all outstanding proofs by incrementing the nonce, analogous to the `mapOperatorRegisterAgentsNonces` pattern in the referenced Solidity report.

---

### Proof of Concept

1. Alice owns a Flow account controlling a COA. She signs hash `H` (e.g., a Seaport order hash) and produces a `COAOwnershipProof P`.
2. The order is filled once; Alice considers the authorization consumed.
3. Six months later, Bob (an unprivileged EVM sender) submits a new EVM transaction calling `isValidSignature(H, P)` on Alice's COA.
4. `isValidSignature` calls `verifyCOAOwnershipProof(coaAddress, H, P)` on the Cadence Arch precompile.
5. `validateCOAOwnershipProof` verifies Alice's signatures over `H`, confirms the COA address matches, and returns `isValid: true`.
6. The ERC-1271-consuming contract accepts the result and executes the authorized action again — without Alice's current consent and with no way for Alice to have prevented it short of revoking her Flow account key.
