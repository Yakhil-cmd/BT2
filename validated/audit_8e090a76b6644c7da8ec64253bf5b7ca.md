### Title
Missing Caller Validation in `interop_root_reporter_event_hook` Allows Unauthorized Interop Root Injection - (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` and `system_context_event_hook` event hooks perform critical system state mutations (`add_interop_root`, `update_settlement_layer_chain_id`) without validating the address of the contract that triggered the event emission. The `SystemEventHook` function signature passes only `caller_ee` (the execution environment type), not the caller address. Both hooks explicitly discard even that: `_caller_ee: u8`. Any contract deployed at the registered hook address (`0x10008` or `0x800b`) that exposes a public function emitting the matching event signature with user-controlled arguments can trigger these hooks from an unprivileged transaction.

---

### Finding Description

**Call hooks** (L1 messenger `0x7001`, set-bytecode `0x7002`, mint `0x7100`) all perform an explicit caller address check at the top of their handler and return an empty-account response for unauthorized callers:

```rust
// l1_messenger_hook
if caller != L1_MESSENGER_ADDRESS { return Ok((empty_return, return_memory)); }
// mint_base_token_hook
if caller != L2_BASE_TOKEN_ADDRESS { return Ok((empty_return, return_memory)); }
``` [1](#0-0) [2](#0-1) 

**Event hooks** have no equivalent guard. The `SystemEventHook` function type signature does not carry the caller address at all — only `caller_ee: u8` (the execution environment type):

```rust
pub struct SystemEventHook<S: SystemTypes>(
    for<'a> fn(
        &ArrayVec<..., MAX_EVENT_TOPICS>,
        &[u8],
        u8,          // caller_ee — NOT the caller address
        &mut System<S>,
        &mut S::Resources,
    ) -> Result<(), SystemError>,
);
``` [3](#0-2) 

Both event hooks explicitly ignore even this field:

```rust
// interop_root_reporter_event_hook
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &...,
    data: &[u8],
    _caller_ee: u8,   // ← ignored
    system: &mut System<S>,
    ...
``` [4](#0-3) 

```rust
fn new_sl_chain_id_event_hook<S: EthereumLikeTypes>(
    topics: &...,
    data: &[u8],
    _caller_ee: u8,   // ← ignored
    system: &mut System<S>,
    ...
``` [5](#0-4) 

The hook dispatch in `System::emit_event` fires the event hook keyed only on the **emitting contract's address** (`address.try_into_low()`), with no information about who called that contract:

```rust
if let Some(address_low) = address.try_into_low() {
    let _ = hooks.try_intercept_event(address_low, topics, data, ee_type as u8, self, resources)?;
}
``` [6](#0-5) 

The hooks are registered for:
- `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW` = `0x10008` → `interop_root_reporter_event_hook`
- `SYSTEM_CONTEXT_ADDRESS_LOW` = `0x800b` → `system_context_event_hook` [7](#0-6) 

The `interop_root_reporter_event_hook` calls `system.io.add_interop_root(...)` unconditionally once the event signature and ABI layout match, with no validation of `root` being non-zero or `chain_id` being non-zero at the ZKsync OS layer:

```rust
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot { root, block_or_batch_number, chain_id },
)?;
``` [8](#0-7) 

`add_interop_root` in the IO subsystem performs no caller or content validation — it pushes the root directly: [9](#0-8) 

The service-transaction whitelist restricts **service transactions** (type `0x7d`) to specific `(address, selector)` pairs, but this does not prevent a regular EVM transaction from calling any public function on the system contracts at `0x10008` or `0x800b`: [10](#0-9) 

---

### Impact Explanation

If the `L2InteropRootStorage` contract at `0x10008` exposes any public function that emits `InteropRootAdded(uint256,uint256,bytes32[])` with caller-controlled arguments, an unprivileged user can:

1. Inject an arbitrary `(chain_id, block_or_batch_number, root)` tuple into the system's `InteropRootStorage`.
2. Corrupt the `interop_roots_rolling_hash` accumulated in `ZKBatchDataKeeper.apply_block`, which is committed to the settlement layer. [11](#0-10) 

This could allow fabrication of cross-chain asset claims or denial of legitimate interop operations by poisoning the rolling hash.

Similarly, if `SystemContext` at `0x800b` exposes a public function emitting `SettlementLayerChainIdUpdated(uint256)`, an attacker can overwrite the settlement layer chain ID used in batch commitments.

---

### Likelihood Explanation

The exploitability is **conditional** on the system contracts at `0x10008` and `0x800b` having public functions that emit the matching event signatures with user-controlled data. The ZKsync OS Rust code itself is the necessary vulnerable step: it unconditionally trusts any event emitted from the registered address, with no caller validation. The system contracts are the only barrier. If either contract has a permissionless emit path (e.g., a public relay function, a misconfigured access modifier, or a re-entrancy path), the attack is directly reachable from a standard EVM transaction.

---

### Recommendation

1. **Add caller address to `SystemEventHook` signature.** Extend the function type to include the caller address (the address of the contract that called the emitting contract), analogous to how call hooks receive `caller` from `ExternalCallRequest`.

2. **Validate caller in both event hooks.** In `interop_root_reporter_event_hook`, assert the emitting contract was called by the bootloader or a whitelisted service-transaction sender. In `system_context_event_hook`, assert the caller is the bootloader.

3. **Add content validation at the ZKsync OS layer.** `interop_root_reporter_event_hook` should reject `root == Bytes32::zero()` and `chain_id == U256::ZERO` directly, rather than relying solely on the Solidity contract.

---

### Proof of Concept

```
1. Deploy (or use existing) L2InteropRootStorage at 0x10008 with a public function:
   function injectRoot(uint256 chainId, uint256 batchNum, bytes32 root) external {
       emit InteropRootAdded(chainId, batchNum, abi.encode(32, 1, root));
   }

2. Send a regular EVM transaction:
   from: attacker_EOA
   to:   0x0000000000000000000000000000000000010008
   data: injectRoot(1, 999, 0xdeadbeef...)

3. The EVM interpreter executes LOG3 with:
   - address = 0x10008
   - topics[0] = INTEROP_ROOT_ADDED_EVENT_SIG
   - topics[1] = chainId = 1
   - topics[2] = batchNum = 999
   - data = abi.encode(32, 1, 0xdeadbeef...)

4. System::emit_event fires interop_root_reporter_event_hook (no caller check).

5. system.io.add_interop_root(...) stores the fake root.

6. ZKBatchDataKeeper.apply_block incorporates the fake root into interop_roots_rolling_hash,
   corrupting the batch commitment submitted to the settlement layer.
``` [12](#0-11) [13](#0-12) [3](#0-2)

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())
}

    pub fn emit_event(
        &mut self,
        hooks: &mut HooksStorage<S, S::Allocator>,

        resources: &mut S::Resources,

        data: &[u8],

        // If successful, intercept event hook, if any

            let _ = hooks.try_intercept_event(
                address_low,
                topics,
                data,
                ee_type as u8,
                self,
                resources,
            )?;
        }
        Ok(())
    }
