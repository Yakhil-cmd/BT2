### Title
WalletConnect Offer Confirmation Understates Total Royalty Cost for Multi-NFT Offers - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible trade amount equally among all NFTs before computing each NFT's royalty, rather than applying each NFT's royalty percentage to the full amount. This causes the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog to be materially understated when a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` request involves multiple NFTs with royalties. A user approves thinking they will spend less than the actual on-chain deduction.

### Finding Description

The external report's vulnerability class is **accounting bug**: a proportional fee calculation that uses current ownership ratios instead of the correct per-participant basis, causing one party to receive (or pay) the wrong share. The Chia GUI analog is in the WalletConnect confirmation display: the royalty total shown to the user is computed with the wrong proportional basis when multiple NFTs are present.

The buggy function is `formatAmountWithRoyalties`:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides amount by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

Each NFT's royalty is computed on `amount / N` (where N = number of NFTs) instead of on the full `amount`. On-chain, each NFT's royalty is independently applied to the full fungible amount, so the correct total royalty is:

```
Σ (amount × royaltyPercentage_i / 10_000)
```

But the code computes:

```
Σ ((amount / N) × royaltyPercentage_i / 10_000)  =  (1/N) × correct_total
```

The displayed total is therefore `1/N` of the correct royalty total, understated by a factor equal to the number of NFTs in the offer.

**Concrete proof from the existing test case** (`parseCommandDisplay.test.ts`, lines 251–335):

- Trade: 100,000,000 mojos for 2 NFTs (royalty_percentage = 500 and 10)
- Code displays: `amountWithRoyalties: '0.00010255'` XCH (102,550,000 mojos)
- Correct total: royalty₁ = 5,000,000 + royalty₂ = 100,000 → total = 105,100,000 mojos = **0.0001051 XCH**
- Understatement: **~50%** of the actual royalties are hidden from the user

The `amountWithRoyalties` field is rendered in the WalletConnect confirmation dialog (`Confirm.tsx`) under the label "Total Amount with Royalties" for both XCH and CAT spend lines.

### Impact Explanation

A WalletConnect-connected dApp sends `chia_wallet.take_offer` with an offer containing multiple NFTs that each carry royalties. The user's confirmation dialog shows a "Total Amount with Royalties" that is significantly lower than what the on-chain transaction will actually deduct. The user approves based on the understated figure; the wallet backend then executes the transaction at the correct (higher) royalty amount. This is a direct, concrete case of WalletConnect state causing a user to approve the wrong spend amount.

### Likelihood Explanation

Any WalletConnect-paired dApp can craft a `chia_wallet.take_offer` payload with a multi-NFT offer. The attacker controls the NFTs and their royalty percentages (within the on-chain royalty cap). No special privileges, leaked keys, or cryptographic breaks are required. The discrepancy grows linearly with the number of NFTs and their royalty rates, making it straightforward to engineer a large gap between the displayed and actual amounts.

### Recommendation

Replace the split-amount approach with a per-NFT full-amount calculation:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Each NFT's royalty is applied to the full fungible amount, not a split share.
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

The existing test expectation for the two-NFT case (`'0.00010255'`) would need to be updated to the correct value (`'0.0001051'`).

### Proof of Concept

1. Attacker pairs a dApp with the user's wallet via WalletConnect.
2. Attacker creates an offer: user pays 1 XCH (1,000,000,000,000 mojos) for 3 NFTs each with `royalty_percentage = 1000` (10%).
3. Attacker's dApp sends `chia_wallet.take_offer` via WalletConnect.
4. GUI calls `parseCommandDisplay` → `formatAmountWithRoyalties`:
   - `splitAmount = 1,000,000,000,000 / 3 = 333,333,333,333` (truncated)
   - royalty per NFT: `333,333,333,333 × 1000 / 10,000 = 33,333,333,333`
   - total royalty displayed: `99,999,999,999` mojos ≈ 0.1 XCH
   - **Displayed total: ~1.1 XCH**
5. Correct on-chain royalties: `3 × (1,000,000,000,000 × 1000 / 10,000) = 300,000,000,000` mojos = 0.3 XCH
   - **Actual total: 1.3 XCH**
6. User sees "Total Amount with Royalties: 1.1 XCH" and approves; wallet deducts 1.3 XCH.
