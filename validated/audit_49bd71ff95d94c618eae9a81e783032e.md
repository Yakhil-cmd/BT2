### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` and `presign` Allows Threshold = 1 to Bypass Minimum Quorum Requirement — (File: `src/frost/mod.rs`)

---

### Summary

The FROST signing and presigning entry points (`assert_sign_inputs` and `presign` in `src/frost/mod.rs`) validate that the threshold does not exceed the participant count, but omit the lower-bound check (`threshold >= 2`) that is enforced during key generation in `assert_key_invariants` (`src/dkg.rs`). A caller or malicious coordinator can pass `threshold = 1` (or `0`) to the signing phase, bypassing the minimum quorum requirement established at key generation time and corrupting the signing output.

---

### Finding Description

During key generation, `assert_key_invariants` in `src/dkg.rs` enforces both an upper and a lower bound on the threshold:

```rust
// Step 1.1
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// Step 1.1
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

During FROST signing, `assert_sign_inputs` in `src/frost/mod.rs` only validates the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

The lower-bound check (`threshold < 2`) is entirely absent. The same omission exists in the `presign` function in the same file:

```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

The `ReconstructionLowerBound` type is a plain `usize` wrapper with no enforced invariants:

```rust
pub struct ReconstructionLowerBound(usize);
``` [4](#0-3) 

This creates an exact analog to the ProfitNotifier bug: the parameter is validated in one context (keygen) but not in another (signing), producing an inconsistency that any caller can exploit.

---

### Impact Explanation

In FROST, each participant computes a partial signature using a Lagrange coefficient `λ_i` that depends on the signing threshold. The key shares were generated under a polynomial of degree `threshold_keygen - 1` (enforced ≥ 1, i.e., threshold ≥ 2). If a caller passes `threshold_sign = 1` to `assert_sign_inputs`, the Lagrange coefficients are computed as if only one participant is needed (coefficient = 1 for the sole participant), which is inconsistent with the degree-≥-1 polynomial used during key generation. The aggregated partial signatures will not reconstruct a valid signature. Honest participants complete the protocol and accept the output, but the resulting signature is cryptographically unusable.

**Impact class:** High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

The `assert_sign_inputs` function is the public-facing validation gate for FROST signing. Any library caller — including a malicious coordinator — can supply an arbitrary `threshold` value. The `ReconstructionLowerBound` type imposes no compile-time or runtime lower-bound constraint. A misconfigured integration or a malicious coordinator can trivially pass `threshold = 1` without triggering any error, causing the signing session to proceed and produce a corrupt output.

---

### Recommendation

Add the same lower-bound check that exists in `assert_key_invariants` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing guard in `assert_key_invariants`: [5](#0-4) 

Long-term, consider encoding the lower-bound invariant directly into the `ReconstructionLowerBound` type (e.g., via a validated constructor that rejects values below 2), so that the constraint is enforced at the type level and cannot be bypassed by any call site.

---

### Proof of Concept

1. A key is generated with `threshold = 2` and `N = 3` participants via `keygen`. The `assert_key_invariants` check enforces `threshold >= 2`.
2. A malicious coordinator initiates a FROST signing session by calling `assert_sign_inputs` with the same 3 participants but `threshold = 1`.
3. `assert_sign_inputs` checks only `1 > 3` (false) — no error is returned.
4. The signing protocol proceeds. Each participant computes their Lagrange coefficient as if only 1 participant is needed (λ = 1), but their share `f(i)` was generated from a degree-1 polynomial (threshold = 2). The partial signatures are computed with the wrong coefficients.
5. The coordinator aggregates the partial signatures. The result is a signature `(R, z)` that does not verify against the public key, because the Lagrange interpolation is inconsistent with the key generation threshold.
6. All honest participants have completed the protocol and accepted the (invalid) output — the signing session is permanently corrupted.
