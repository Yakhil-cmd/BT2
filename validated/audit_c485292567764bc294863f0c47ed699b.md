### Title
Gateway `validate_resource_bounds` checks against stale `l2_gas_price` instead of `next_l2_gas_price`, causing wrong admission decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The gateway's stateful resource-bounds check compares a transaction's `max_price_per_unit` against the **current block's** `l2_gas_price`, but the block being built will execute at `next_l2_gas_price` (the EIP-1559-adjusted price stored in the same block header). The code even carries a self-documenting TODO acknowledging the wrong field is used. This causes the gateway to accept transactions that will fail in the batcher (price rising) or reject transactions that would succeed (price falling).

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` by reading `block_header.l2_gas_price.price_in_fri`:

```rust
// crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs:46-49
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
```

`BlockHeaderWithoutHash` carries **two distinct L2 gas price fields**:

```rust
// crates/starknet_api/src/block.rs:237,239
pub l2_gas_price: GasPricePerToken,   // price used IN this block
pub next_l2_gas_price: GasPrice,      // EIP-1559 price for the NEXT block
```

`validate_resource_bounds` then reads the value that was populated from `l2_gas_price` and uses it as the threshold for the **next** block:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:229-240
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
```

The threshold is then:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:366-372
let threshold = (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer();
if tx_l2_gas_price.0 < threshold { return Err(...) }
```

`next_l2_gas_price` is computed by `calculate_next_base_gas_price` (EIP-1559 formula) and can differ from `l2_gas_price` by up to `price / gas_price_max_change_denominator` per block. The orchestrator's `update_l2_gas_price` / `calculate_next_l2_gas_price` path is what actually sets the gas price used during batcher execution — it uses `next_l2_gas_price` from the synced block header, not `l2_gas_price`.

The same wrong value is also forwarded into `run_validate_entry_point`'s `BlockContext`, so the blockifier validation at the gateway also runs at the wrong price, masking the mismatch until the transaction reaches the batcher.

### Impact Explanation

**When the L2 gas price is rising** (`next_l2_gas_price > l2_gas_price`): the gateway threshold is too low. A transaction with `max_price_per_unit` in the range `[threshold_old, next_l2_gas_price)` passes both the gateway pre-check and the gateway blockifier validation (both use the stale price), is admitted to the mempool, and then fails `check_fee_bounds` during actual batcher execution with `MaxGasPriceTooLow`. The gateway has admitted an economically invalid transaction.

**When the L2 gas price is falling** (`next_l2_gas_price < l2_gas_price`): the gateway threshold is too high. A transaction with `max_price_per_unit` in the range `[threshold_new, threshold_old)` is rejected by the gateway even though it would succeed in the batcher. Valid transactions are incorrectly denied admission.

Both directions match the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The EIP-1559 L2 gas price adjusts every block. Any period of sustained congestion (price rising) or low utilization (price falling) produces a persistent gap between `l2_gas_price` and `next_l2_gas_price`. The gap is bounded per block but accumulates. Any user submitting a transaction with `max_price_per_unit` near the boundary will be affected. The TODO comment confirms the developers are aware the wrong field is used, meaning the condition is present in every production deployment.

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, replace the `l2_gas_price` field population with `next_l2_gas_price`:

```rust
// crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.next_l2_gas_price.try_into()?,
},
```

This ensures both the pre-validation threshold and the blockifier `BlockContext` used during gateway validation reflect the price that will actually govern execution in the next block, eliminating the admission/execution mismatch.

### Proof of Concept

1. Observe that `BlockHeaderWithoutHash` stores both `l2_gas_price` (current block) and `next_l2_gas_price` (next block). [1](#0-0) 

2. `GatewayFixedBlockSyncStateClient` reads only `l2_gas_price`, ignoring `next_l2_gas_price`. [2](#0-1) 

3. `validate_resource_bounds` uses this stale value as the admission threshold, with a TODO acknowledging the wrong field. [3](#0-2) 

4. The threshold comparison rejects or accepts based on the stale price. [4](#0-3) 

5. The orchestrator's `update_l2_gas_price` uses the EIP-1559 formula, so `next_l2_gas_price ≠ l2_gas_price` whenever gas usage deviates from the target. [5](#0-4) 

6. The `StorageBlockHeader` confirms `next_l2_gas_price` is a distinct, separately stored field. [6](#0-5)
