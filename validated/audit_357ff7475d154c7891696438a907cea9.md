### Title
Centralized ERC-20 Token Custody via `erc20_fallback_address` Silently Redirects All Non-Whitelisted Bridge Deposits to a Single Address - (File: `engine/src/engine.rs`)

### Summary
In Silo mode, every ERC-20 token bridged into Aurora via `ft_on_transfer` whose intended recipient is not on the address whitelist is silently redirected to a single, centrally-controlled EVM address (`erc20_fallback_address`). This mirrors the `salesAddress` pattern in the reported vulnerability: a single address accumulates all user funds, is undocumented as a risk to bridging users, and creates a concentrated point of failure.

### Finding Description
When a NEP-141 token is bridged into Aurora, the NEAR runtime calls `ft_on_transfer`, which dispatches to `receive_erc20_tokens` in `engine/src/engine.rs`. Inside that function, lines 818–822 perform the following check:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

`is_allow_receive_erc20_tokens` delegates entirely to the `Address` whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [2](#0-1) 

If the whitelist is enabled and the recipient is absent from it, the token mint is silently re-targeted to `erc20_fallback_address`. The original sender receives no error and no refund — the NEP-141 tokens are consumed by the bridge and the ERC-20 equivalent is minted entirely to the fallback address. The `erc20_fallback_address` is a single EVM address stored in contract storage and set exclusively by the owner: [3](#0-2) 

The `SiloParamsArgs` struct documents this address only as "used for withdrawing ERC-20 base tokens in case a recipient is not in the silo white list," with no user-facing warning that all such tokens accumulate in one place: [4](#0-3) 

### Impact Explanation
Every ERC-20 token bridged to any non-whitelisted address in Silo mode is minted to the single `erc20_fallback_address`. The bridging user's funds are immediately inaccessible to them — they cannot withdraw, transfer, or recover the tokens without action from the fallback address controller. This constitutes **High: Temporary freezing of funds** for every affected user. Because all such tokens accumulate at one address, the total value at risk grows with every bridge deposit to a non-whitelisted recipient. If the fallback address key is ever compromised, the impact escalates to **Critical: Direct theft** of all accumulated tokens.

### Likelihood Explanation
In any deployed Silo instance with the `Address` whitelist enabled, this condition is triggered by every ordinary user who bridges NEP-141 tokens to an address that has not been explicitly whitelisted. No special privilege is required — the attacker-controlled entry path is the standard `ft_on_transfer` NEAR callback, callable by any NEP-141 token contract that has been registered with Aurora. The concentration of funds grows passively over time as users bridge tokens.

### Recommendation
1. When `erc20_fallback_address` is set and a recipient is not whitelisted, return the full token amount to the sender (i.e., return `args.amount` instead of `0`) rather than silently minting to the fallback address. This eliminates the centralized custody risk entirely.
2. If the redirect behavior is intentional, document it prominently in user-facing materials and emit an on-chain event identifying the original intended recipient so funds can be traced and returned.
3. Consider requiring the fallback address to be a multisig or a NEAR account (not a bare EVM address) to reduce key-compromise risk.

### Proof of Concept
1. Deploy Aurora Engine in Silo mode with `set_silo_params` specifying `erc20_fallback_address = 0xAAAA...` and enable the `Address` whitelist.
2. Register a NEP-141 token and deploy its ERC-20 mirror via `deploy_erc20_token`.
3. As an unprivileged NEAR account (`alice.near`), call `ft_transfer_call` on the NEP-141 contract, transferring 1000 tokens to Aurora with `msg = hex(alice_evm_address)` where `alice_evm_address` is **not** on the whitelist.
4. Aurora's `ft_on_transfer` → `receive_erc20_tokens` executes. Because `alice_evm_address` is not whitelisted, `recipient` is overwritten with `erc20_fallback_address`.
5. The ERC-20 `mint(erc20_fallback_address, 1000)` call succeeds. Alice's NEP-141 tokens are consumed; her EVM address balance is 0; `erc20_fallback_address` holds 1000 ERC-20 tokens.
6. Alice has no mechanism to recover her tokens without cooperation from the fallback address controller.

This is confirmed by the workspace test `test_transfer_nep141_to_non_whitelisted_address`, which demonstrates exactly this flow: [5](#0-4)
