### Title
Dust Delegation Grows `pool_member_epoch_balance` Trace Unboundedly, Permanently Freezing Unclaimed Yield — (File: src/pool/pool.cairo)

---

### Summary

`add_to_delegation_pool` in `Pool.cairo` enforces only `amount > 0`, permitting dust amounts (1 token unit). A pool member (or their reward address) can call this function with dust amounts across successive epochs, appending a new checkpoint to their `pool_member_epoch_balance` trace each epoch. The `calculate_rewards` loop inside `claim_rewards` iterates over every trace entry since the last claim — a loop the codebase itself acknowledges as "unbounded" — and will eventually exhaust gas, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

**Root cause — no minimum amount in `add_to_delegation_pool`:**

`add_to_delegation_pool` only rejects a zero amount:

```cairo
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
```

Any value ≥ 1 is accepted, including dust. [1](#0-0) 

**Trace growth mechanism:**

Every call to `add_to_delegation_pool` invokes `increase_member_balance` → `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. [2](#0-1) 

The underlying `insert` implementation appends a **new** checkpoint whenever the epoch key differs from the last stored key. Because `get_epoch_plus_k()` returns `current_epoch + K`, one new checkpoint is appended per epoch in which the function is called. [3](#0-2) 

`exit_delegation_pool_intent` also calls `set_member_balance`, so alternating between adding and removing dust amounts across epochs grows the trace even faster. [4](#0-3) 

**Unbounded loop in `calculate_rewards`:**

`claim_rewards` calls `calculate_rewards`, which iterates from `entry_to_claim_from` to `pool_member_trace_length`. The code itself contains the comment:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. **This loop is unbounded but unlikely to exceed gas limits.** [5](#0-4) 

`entry_to_claim_from` is only advanced when `claim_rewards` succeeds. If the pool member (or their reward address) never calls `claim_rewards` while continuously adding dust across epochs, the loop length grows proportionally to the number of elapsed epochs. Once the loop is large enough to exhaust the Starknet transaction gas limit, `claim_rewards` reverts on every attempt.

`pool_member_info_v1` (a read function) also calls `calculate_rewards` and is equally affected. [6](#0-5) 

---

### Impact Explanation

When `claim_rewards` permanently reverts due to gas exhaustion, the pool member's accumulated STRK rewards are locked in the pool contract with no recovery path — matching the **"Permanent freezing of unclaimed yield"** (High) impact category. The `pool_member_info_v1` view also becomes unusable, breaking off-chain tooling that reads reward state.

---

### Likelihood Explanation

The entry path requires only an unprivileged pool member (or their reward address). The cost per epoch is minimal: one `add_to_delegation_pool` call with `amount = 1`. The attacker must refrain from calling `claim_rewards` while growing the trace. On Starknet, epochs are measured in blocks/time, so reaching a gas-exhausting trace length takes sustained effort, but the per-epoch cost is negligible. The scenario is also reachable accidentally by a legitimate delegator who adds small top-ups frequently over a long period without claiming.

---

### Recommendation

Enforce a minimum delegation amount in both `enter_delegation_pool` and `add_to_delegation_pool`, analogous to the existing `min_stake` check in the staking contract:

```cairo
assert!(amount >= self.min_delegation_for_rewards.read(), "{}", Error::AMOUNT_LESS_THAN_MIN_DELEGATION);
```

The pool already stores `min_delegation_for_rewards` (set to `10^18` for STRK, i.e. 1 STRK) in its constructor — this value should be reused as the entry/top-up floor. [7](#0-6) [8](#0-7) 

Additionally, consider bounding the maximum number of unclaimed trace entries (e.g., require `claim_rewards` before `add_to_delegation_pool` if the trace exceeds a threshold), or compress consecutive same-epoch entries.

---

### Proof of Concept

1. Staker deploys a pool with `pool_enabled: true`.
2. Attacker calls `enter_delegation_pool(reward_address, amount: 1)` — accepted because `1 > 0`.
3. For each epoch `i` from 1 to N:
   a. Advance one epoch.
   b. Call `add_to_delegation_pool(pool_member: attacker, amount: 1)`.
   c. Each call appends a new checkpoint at epoch `current + K` to `pool_member_epoch_balance`.
4. After N epochs, `pool_member_epoch_balance` has N+1 entries; `entry_to_claim_from` is still 0 (no `claim_rewards` called).
5. Call `claim_rewards(pool_member: attacker)`. The `calculate_rewards` loop iterates over all N+1 entries. For sufficiently large N, the transaction runs out of gas and reverts.
6. All subsequent `claim_rewards` calls also revert — the pool member's yield is permanently frozen.

The only cost is N × (gas for `add_to_delegation_pool` + 1 token unit), which is negligible over time. [9](#0-8) [10](#0-9) [11](#0-10)

```text
            // Calculate rewards and update entry_to_claim_from.

            // Emit event.

                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );
```

```text
    /// This is done by either inserting a new checkpoint, or updating the last one.
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
```
