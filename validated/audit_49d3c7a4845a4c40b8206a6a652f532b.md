### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze a Pool Member's Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in the `Pool` contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace since their last reward claim. A pool member (or their designated reward address) can inflate this trace by making one balance change per epoch across many epochs without ever claiming rewards. Once the trace grows large enough, the `claim_rewards` transaction will always exceed the Starknet block gas limit, permanently freezing the pool member's accrued but unclaimed yield with no recovery path.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function (lines 837–888) iterates over the `pool_member_epoch_balance` trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
```

The trace (`pool_member_epoch_balance`) grows by exactly one entry per epoch in which the pool member changes their balance. The trace insertion logic in `src/pool/pool_member_balance_trace/trace.cairo` (lines 163–173) confirms that multiple changes within the same epoch update the same entry, but a new epoch always appends a new checkpoint.

The `entry_to_claim_from` cursor stored in `pool_member_info` is only advanced on a **successful** `claim_rewards` call. If the pool member never claims, the loop must traverse every balance-change entry since their first delegation. Each iteration performs multiple storage reads (`pool_member_trace.at(...)` and `find_sigma` which reads `cumulative_rewards_trace`), making the per-iteration gas cost non-trivial.

The `claim_rewards` function (lines 335–377) is the sole caller of `calculate_rewards` and is the only mechanism to advance `entry_to_claim_from`. There is no partial-claim or pagination mechanism.

The `add_to_delegation_pool` function (lines 221–254) permits the **reward address** — which is a separate, caller-controlled address — to add funds on behalf of the pool member:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
```

This means the reward address can unilaterally inflate the pool member's balance trace by calling `add_to_delegation_pool` once per epoch, each time in a new epoch, without the pool member's ongoing consent.

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace grows beyond the gas budget of a single Starknet transaction, every call to `claim_rewards` for that pool member will revert with an out-of-gas error. Because `entry_to_claim_from` is only updated inside `claim_rewards`, and because there is no alternative path to advance the cursor or withdraw rewards in chunks, the pool member's entire accumulated unclaimed yield becomes permanently inaccessible. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The attack requires one balance-change transaction per epoch for a sustained period. The cost per epoch is a single `add_to_delegation_pool` call (or `exit_delegation_pool_intent` / `enter_delegation_pool`). The reward address can execute this without the pool member's per-epoch approval once the reward address is set. On a protocol with short epochs (hours to days), thousands of epochs accumulate over months to years, making this a slow-burn griefing vector. The likelihood is **medium-low** for deliberate exploitation but **medium** for accidental self-infliction by yield-aggregator integrations that auto-compound by calling `add_to_delegation_pool` every epoch.

---

### Recommendation

1. **Paginate reward claims**: Allow `claim_rewards` to accept a `max_entries` parameter so the loop is bounded per call, and the cursor advances incrementally across multiple transactions.
2. **Cap balance-change frequency**: Enforce a minimum epoch gap between balance changes for the same pool member, preventing trace inflation faster than claims can keep up.
3. **Charge gas proportional to trace length**: Add a gas surcharge on `add_to_delegation_pool` and `exit_delegation_pool_intent` that scales with the current trace length, making trace inflation prohibitively expensive.

---

### Proof of Concept

```
1. Pool member P delegates to a pool and sets reward_address = R (an EOA or contract).
2. R calls add_to_delegation_pool(pool_member=P, amount=1) once per epoch for N epochs,
   never triggering claim_rewards.
   - Each call in a new epoch appends one entry to pool_member_epoch_balance[P].
3. After N epochs, P (or R) calls claim_rewards(pool_member=P).
4. calculate_rewards loops N times, each iteration reading from pool_member_epoch_balance
   and cumulative_rewards_trace (multiple storage reads per iteration).
5. For sufficiently large N, the transaction exceeds the Starknet block gas limit and reverts.
6. Because entry_to_claim_from is never updated, every subsequent claim_rewards call
   also reverts. P's unclaimed yield is permanently frozen.
```

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();

            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self

                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;

            // Emit event.
