### Title
ERC-20 Tokens Burned in Excess of NEP-141 Tokens Transferred When `ft_transfer_call` Partially Accepts — (`engine/src/contract_methods/connector.rs`)

---

### Summary

When a user exits ERC-20 tokens to NEAR via the Omni flow, the ERC-20 tokens are burned for the full requested amount before the cross-chain call is made. If the NEP-141 receiver's `ft_on_transfer` returns a non-zero refund amount (partial acceptance), the `exit_to_near_precompile_callback` ignores the actual amount transferred and does not re-mint the ERC-20 tokens for the unaccepted portion. The user permanently loses the difference.

---

### Finding Description

**Step 1 — ERC-20 burn happens before the cross-chain call.**

In `EvmErc20.sol` and `EvmErc20V2.sol`, `withdrawToNear` burns the full `amount` from the caller's balance unconditionally, then calls the `ExitToNear` precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← burned first, no rollback possible
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...)
    }
}
``` [1](#0-0) 

**Step 2 — The precompile schedules `ft_transfer_call` for the Omni path.**

In `engine-precompiles/src/native.rs`, the Omni branch selects `ft_transfer_call` as the method:

```rust
Some(Message::Omni(msg)) => (
    nep141_account_id,
    ft_transfer_call_args(&exit_params.receiver_account_id, exit_params.amount, msg)?,
    "ft_transfer_call",
    None,
    ...
),
``` [2](#0-1) 

A callback `exit_to_near_precompile_callback` is chained only when `callback_args != default()`. For the Omni path without `error_refund`, `callback_args` equals the default (both fields `None`), so **no callback is scheduled at all**. With `error_refund` enabled, a callback is scheduled but, as shown below, it still does not handle partial acceptance. [3](#0-2) 

**Step 3 — The callback ignores the actual transferred amount.**

`exit_to_near_precompile_callback` checks only whether the promise succeeded or failed. It uses `_` to discard the promise return value, which for `ft_transfer_call` is the amount **actually** transferred (i.e., `requested − refunded`):

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // ↑ return value ignored; partial acceptance treated identically to full acceptance
    if let Some(args) = args.transfer_near { ... }
    None
} else if let Some(args) = args.refund {
    // only fires on complete failure
    engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None
};
``` [4](#0-3) 

**Step 4 — NEP-141 standard allows partial acceptance.**

Under NEP-141, a receiver's `ft_on_transfer` may return `Y > 0` tokens as unused. `ft_resolve_transfer` then refunds those `Y` tokens back to the sender (Aurora). The promise result seen by the callback is `X − Y` (the amount actually transferred). Since the callback ignores this value, the `Y` ERC-20 tokens that were burned are never re-minted.

---

### Impact Explanation

| Asset | Before | After (partial acceptance of Y) |
|---|---|---|
| User ERC-20 balance | X | X − X = 0 (burned) |
| Receiver NEP-141 balance | — | +X − Y |
| Aurora NEP-141 balance | B | B + Y (refunded, inaccessible to user) |

The user burns X ERC-20 tokens but receives only X−Y NEP-141 tokens at the destination. The Y NEP-141 tokens are silently absorbed into Aurora's NEP-141 balance with no mechanism for the user to reclaim them. This is a **permanent, irrecoverable loss** of user funds — matching the Critical impact tier (direct theft/permanent freeze of funds in motion).

---

### Likelihood Explanation

The Omni flow is the intended path for cross-chain routing (e.g., bridging to another chain via an intermediary contract). Any NEP-141 receiver that implements capacity limits, rate limits, or conditional acceptance logic can legitimately return a non-zero refund from `ft_on_transfer`. This is explicitly supported by the NEP-141 standard and is not an adversarial condition. A user routing tokens through such a receiver triggers the loss without any attacker involvement. Likelihood is **medium** given the Omni flow's purpose.

---

### Recommendation

In `exit_to_near_precompile_callback`, parse the `PromiseResult::Successful(bytes)` return value for the `ft_transfer_call` case. Compute `refunded = requested − actually_transferred`. If `refunded > 0`, call `refund_on_error` (or an equivalent re-mint path) to restore the `refunded` ERC-20 tokens to the original sender. The `RefundCallArgs` struct already carries the necessary `erc20_address`, `recipient_address`, and `amount` fields for this purpose. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. Deploy a NEP-141 receiver contract whose `ft_on_transfer` accepts only half the tokens (returns `amount / 2`).
2. Bridge NEP-141 tokens into Aurora; receive ERC-20 tokens (amount = X).
3. Call `withdrawToNear(recipient_bytes, X)` on the ERC-20 contract with an Omni message targeting the receiver.
4. ERC-20 contract burns X tokens from the caller.
5. `ExitToNear` precompile schedules `ft_transfer_call(receiver, X, msg)` on the NEP-141 contract.
6. Receiver's `ft_on_transfer` returns `X/2`; NEP-141 refunds `X/2` tokens to Aurora.
7. `exit_to_near_precompile_callback` fires with `PromiseResult::Successful(X/2)`.
8. Callback discards the return value, takes the success branch, does nothing.
9. Caller's ERC-20 balance: 0. Receiver's NEP-141 balance: +X/2. Aurora's NEP-141 balance: +X/2 (locked). Net loss to caller: X/2 ERC-20 tokens, permanently.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
