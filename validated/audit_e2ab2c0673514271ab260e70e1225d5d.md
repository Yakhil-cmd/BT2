### Title
Missing Emitter-Caller Authorization in Event Hooks Enables Unauthorized Interop Root Injection and Settlement Layer Chain ID Manipulation - (File: system_hooks/src/event_hooks/interop_root_reporter.rs, system_hooks/src/event_hooks/system_context.rs)

---

### Summary

ZKsync OS implements two classes of system hooks: **call hooks** and **event hooks**. All call hooks explicitly verify the `caller` address before executing privileged state changes. The two event hooks — `interop_root_reporter_event_hook` and `system_context_event_hook` — perform **no such caller authorization check**. They fire on any event emitted from the registered system contract address matching the expected event signature, regardless of who called the system contract to produce that event. This is the direct analog of the KYC report's "access control enforced only at the frontend layer": the ZKsync OS backend (the event hook) trusts the Solidity system contract (the frontend) to enforce caller restrictions, but does not enforce them itself.

---

### Finding Description

**Call hooks enforce caller identity — event hooks do not.**

Every call hook in the codebase explicitly checks the `caller` field of the `ExternalCallRequest` before performing any privileged operation:

- `l1_messenger_hook`: rejects any caller that is not `L1_MESSENGER_ADDRESS` (`0x8008`).
- `mint_base_token_hook`: rejects any caller that is not `L2_BASE_TOKEN_ADDRESS` (`0x800a`).
- `set_bytecode_on_address_hook`: rejects any caller that is not `CONTRACT_DEPLOYER_ADDRESS` or `COMPLEX_UPGRADER_ADDRESS`.
- `contract_deployer_temp_hook`: rejects any caller that is not `COMPLEX_UPGRADER_ADDRESS`. [1](#0-0) [2](#0-1) [3](#0-2) 

The two event hooks receive only `topics`, `data`, `caller_ee` (execution environment type, not an address), and `system`. **Neither receives nor checks the address of the account that called the system contract to produce the event.**

`interop_root_reporter_event_hook` — registered for `L2_INTEROP_ROOT_STORAGE_ADDRESS` (`0x10008`):

```rust

**How the hook dispatch works.** `System::emit_event` fires the event hook whenever any contract at the registered address emits an event. The hook receives only the event payload — not who called the emitting contract: [6](#0-5) 

The `SystemEventHook` type signature confirms: no caller address is passed to the hook function: [7](#0-6) 

**The intended access control is only at the service-transaction whitelist level.** The bootloader's service transaction parser restricts `addInteropRootsInBatch` (on `0x10008`) and `setSettlementLayerChainId` (on `0x800b`) to service transactions originating from `BOOTLOADER_FORMAL_ADDRESS`: [8](#0-7) 

However, this whitelist only restricts **service transaction type** (`0x7d`). A regular L2 transaction (EIP-1559, legacy, etc.) can call any address, including `0x10008` and `0x800b`, without restriction. If the Solidity system contracts at those addresses do not independently enforce caller restrictions, any unprivileged sender can call them, cause the event to be emitted, and trigger the hook.

---

### Impact Explanation

**Interop root injection (`0x10008`):** `add_interop_root` appends an attacker-controlled `InteropRoot { root, chain_id, block_or_batch_number }` to the system's interop root storage. Interop roots are cross-chain commitments used to verify messages from other chains. Injecting a fabricated root allows an attacker to forge valid-looking cross-chain message proofs, enabling unauthorized asset withdrawals or message replay across chains that rely on this chain as a settlement layer. [9](#0-8) 

**Settlement layer chain ID manipulation (`0x800b`):** `update_settlement_layer_chain_id` sets the chain ID of the settlement layer for the current block. This value is read during block finalization and used in cross-chain commitment logic. An attacker who sets it to an arbitrary value can corrupt the block's settlement-layer commitment, causing proof verification failures or enabling cross-chain fraud. [10](#0-9) 

The `NewSettlementLayerChainIdStorage` enforces at most one update per block, so the attacker only needs to win a race to the first call in a block: [11](#0-10) 

---

### Likelihood Explanation

- An unprivileged EOA can send a standard L2 transaction to `0x10008` or `0x800b` with no special privileges.
- The service-transaction whitelist is enforced only for transaction type `0x7d`; regular transactions bypass it entirely.
- The event hooks are registered unconditionally in the ZK post-init path: [12](#0-11) 

- The only remaining barrier is whether the Solidity contracts at `0x10008` and `0x800b` independently check `msg.sender`. The ZKsync OS runtime provides no such guarantee and does not enforce it.

---

### Recommendation

1. **Add emitter-caller authorization to event hooks.** Pass the address of the account that triggered the event (the transaction originator or the immediate caller of the system contract) into the event hook dispatch, and reject events not originating from the expected privileged caller — mirroring the pattern used by all call hooks.

2. **Alternatively, enforce caller checks inside the system contracts.** The Solidity contracts at `0x10008` and `0x800b` must independently verify that `msg.sender` is `BOOTLOADER_FORMAL_ADDRESS` (or another trusted address) before emitting the privileged events, so that the ZKsync OS event hook cannot be triggered by an arbitrary caller even if the hook itself does not check.

3. **Audit the asymmetry.** The `SystemEventHook` type should be extended to include the emitter-caller address in its function signature, making it structurally impossible to write an event hook that cannot check caller identity.

---

### Proof of Concept

**Attack: Inject a fake interop root**

1. Attacker sends a regular L2 transaction to `L2_INTEROP_ROOT_STORAGE_ADDRESS` (`0x10008`) calling `addInteropRootsInBatch(...)` with a crafted `InteropRoot` payload (arbitrary `chainId`, `blockOrBatchNumber`, `root`).
2. If the Solidity contract at `0x10008` does not check `msg.sender`, it executes and emits `InteropRootAdded(chainId, blockOrBatchNumber, [root])`.
3. `System::emit_event` intercepts the event from address `0x10008`, matches `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW`, and calls `interop_root_reporter_event_hook`.
4. The hook validates only the event signature and data format — both of which the attacker controls — and calls `system.io.add_interop_root(...)` with the attacker's values.
5. The fabricated interop root is now committed to the block's interop root storage, indistinguishable from a legitimate root submitted by the sequencer.

**Attack: Corrupt settlement layer chain ID**

1. Attacker sends a regular L2 transaction to `SYSTEM_CONTEXT_ADDRESS` (`0x800b`) calling `setSettlementLayerChainId(uint256)` with an arbitrary chain ID.
2. If the Solidity contract does not check `msg.sender`, it emits `SettlementLayerChainIdUpdated(newChainId)`.
3. `system_context_event_hook` fires, validates only the event signature and topic count, and calls `system.io.update_settlement_layer_chain_id(newChainId)`.
4. The block's settlement layer chain ID is now set to the attacker's value, corrupting cross-chain commitment data for the entire block.

pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())
```

    /// Emit an event, potentially capturing some using an event hook.
    pub fn emit_event(
        &mut self,
        hooks: &mut HooksStorage<S, S::Allocator>,
        ee_type: ExecutionEnvironmentType,
        resources: &mut S::Resources,
        address: &<S::IOTypes as SystemIOTypesConfig>::Address,
        topics: &ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
        data: &[u8],
    ) -> Result<(), SystemError> {
        // First, emit the event using io subsystem
        self.io
            .emit_event(ee_type, resources, address, topics, data)?;

        // If successful, intercept event hook, if any
        if let Some(address_low) = address.try_into_low() {
            let _ = hooks.try_intercept_event(
                address_low,
                topics,
                data,
                ee_type as u8,
                self,
                resources,
            )?;
        }
        Ok(())
    }
