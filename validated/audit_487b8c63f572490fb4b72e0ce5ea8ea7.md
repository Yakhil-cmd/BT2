### Title
`uint32` overflow in `_afterTimelock` lets pool admin bypass LP-protection timelocks — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
`_afterTimelock` truncates `block.timestamp + timelock` to `uint32`. When the pool admin sets a `timelock` value large enough to overflow `uint32`, the resulting `executeAfter` wraps to a timestamp already in the past, making every `_requireElapsed` check pass immediately. The admin can then execute drawdown, decay, or watermark changes with zero effective delay, bypassing the LP-protection timelock entirely.

### Finding Description
The helper function: [1](#0-0) 

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

adds `block.timestamp` (`uint256`) and `timelock` (`uint32`) in 256-bit space, then truncates to `uint32`. The `timelock` field accepts any `uint32` value — neither `initialize` nor `proposeOracleStopLossTimelock` validate it: [2](#0-1) 

With `block.timestamp ≈ 1,753,000,000` (July 2025) and `type(uint32).max = 4,294,967,295`, any `timelock > 2,541,967,295` (~80 years) causes the sum to exceed `type(uint32).max`. The truncated `executeAfter` wraps to a value **smaller than the current timestamp**.

The enforcement check: [3](#0-2) 

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

promotes `executeAfter` to `uint256` for comparison. With a wrapped value (e.g., `1,752,999,999`), `block.timestamp < executeAfter` evaluates to `false` — the revert never fires.

All four proposal paths share this flaw: [4](#0-3) [5](#0-4) [6](#0-5) 

All `pendingXxxExecuteAfter` fields are also `uint32`: [7](#0-6) 

**Attack path (starting from timelock = 0):**

1. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`. `_afterTimelock` returns `uint32(block.timestamp + 0) = block.timestamp` (no overflow yet). `executeAfter = block.timestamp`.
2. Admin immediately calls `executeOracleStopLossTimelock`. `_requireElapsed(block.timestamp)` → `block.timestamp < block.timestamp` → `false` → passes. Timelock is now `type(uint32).max`.
3. Admin calls `proposeOracleStopLossDrawdown(pool, 0)`. `_afterTimelock` returns `uint32(1_753_000_000 + 4_294_967_295)` = `uint32(6_047_967_295)` = `1_752_999_999` (wrapped, in the past).
4. Admin immediately calls `executeOracleStopLossDrawdown`. `_requireElapsed(1_752_999_999)` → `1_753_000_000 < 1_752_999_999` → `false` → passes. Drawdown is set to 0 with zero delay.

The same applies to decay and watermark proposals.

### Impact Explanation
The timelock is the sole on-chain mechanism giving LPs advance notice before the pool admin alters drawdown, decay rate, or high watermarks. Bypassing it lets the admin:

- **Instantly disable the stop-loss** (drawdown → 0), stripping LP value protection without warning.
- **Instantly set arbitrarily high watermarks**, causing the stop-loss to trigger on any subsequent swap and freezing pool trading — a broken core pool functionality impact.

This is an admin-boundary break: the pool admin exceeds the timelock bound that caps their power over LP-protection parameters.

### Likelihood Explanation
The pool admin must set a `timelock` value greater than ~2.54 billion seconds. This is within the valid `uint32` range and requires no external privileges or special conditions. If the current timelock is 0, the entire attack executes in two transactions. If the current timelock is non-zero, the admin must wait it out once before escalating to the overflow value — still a reachable path for a malicious admin.

### Recommendation
Use `uint256` for all `executeAfter` storage fields and the `_afterTimelock` return type, or add an explicit upper bound on the `timelock` parameter in both `initialize` and `proposeOracleStopLossTimelock` (e.g., `require(newTimelock <= 365 days)`).

```diff
- function _afterTimelock(address pool_) private view returns (uint32) {
-     return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
+ function _afterTimelock(address pool_) private view returns (uint256) {
+     return block.timestamp + oracleStopLossConfig[pool_].timelock;
  }
```

And in `proposeOracleStopLossTimelock`:
```diff
+ uint256 private constant MAX_TIMELOCK = 365 days;
  function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
+     if (newTimelock > MAX_TIMELOCK) revert TimelockTooLarge(newTimelock);
      ...
  }
```

### Proof of Concept

```solidity
// block.timestamp = 1_753_000_000 (July 2025)
// timelock = type(uint32).max = 4_294_967_295

uint32 executeAfter = uint32(1_753_000_000 + 4_294_967_295);
// sum = 6_047_967_295
// uint32 wraps: 6_047_967_295 - 4_294_967_296 = 1_752_999_999

// _requireElapsed:
// block.timestamp < executeAfter
// 1_753_000_000   < 1_752_999_999  →  false  →  no revert  →  timelock bypassed
```
