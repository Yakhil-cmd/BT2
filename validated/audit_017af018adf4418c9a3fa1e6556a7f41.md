### Title
Blocklist Destination-Address Check Applied Only to EVM Transactions, Bypassed via Cosmos SDK Messages — (`app/block_address.go`, `app/proposal.go`)

### Summary
Cronos enforces a destination-address blocklist check for `MsgEthereumTx` transactions at both the mempool (CheckTx) and proposal (PrepareProposal/ProcessProposal) layers. The identical check is absent for Cosmos SDK messages such as `bank.MsgSend` and `ibc.MsgTransfer`. An unprivileged user can therefore transfer tokens to a blocklisted address by using a Cosmos SDK message instead of an EVM transaction, bypassing the destination-address restriction entirely.

### Finding Description

**Enforcement in `BlockAddressesDecorator.AnteHandle`** (`app/block_address.go`):

The decorator runs its entire body only when `ctx.IsCheckTx()` is true. Inside that block it checks:
1. **Signers** — for every transaction type.
2. **EVM destination** — only when the message is `*evmtypes.MsgEthereumTx`.
3. **EIP-7702 authorization list** — only when the message is `*evmtypes.MsgEthereumTx`. [1](#0-0) 

For any non-EVM Cosmos SDK message (e.g., `bank.MsgSend`, `ibc.MsgTransfer`, `cronos.MsgConvertVouchers`), the decorator only checks whether the **signer** is blocked. The **recipient/receiver** field of those messages is never inspected.

**Enforcement in `ProposalHandler.ValidateTransaction`** (`app/proposal.go`):

The same asymmetry exists at the proposal layer. Signers are checked for all transaction types, but the destination address check is gated behind `msg.(*evmtypes.MsgEthereumTx)`: [2](#0-1) 

Cosmos SDK messages pass through `ValidateTransaction` with only a signer check, regardless of their recipient field.

**Contrast with the EVM path:**

The destination check for EVM transactions was deliberately added (CHANGELOG v1.6.0, PR #1922 "Feat: check destination address in the blocklist") and is covered by the integration test `test_block_list_contract`, which confirms that sending an EVM transaction *to* a blocked address is rejected: [3](#0-2) 

No equivalent test or enforcement exists for Cosmos SDK message recipients.

### Impact Explanation

The blocklist is a compliance/sanctions mechanism. The explicit addition of the EVM destination check (and its test coverage) demonstrates that the design intent is to prevent tokens from flowing *to* blocked addresses, not only to prevent blocked addresses from *sending*. The Cosmos SDK message path silently omits this check, creating an inconsistency directly analogous to the external report: one operation path (EVM) enforces the restriction; the parallel path (Cosmos SDK) does not.

A non-blocked user can send any amount of CRO, IBC vouchers, or CRC20/CRC21 tokens to a blocked address via `bank.MsgSend` or `ibc.MsgTransfer`. If the blocked address is later removed from the blocklist, those accumulated funds become fully spendable. This constitutes a **bypass of the block-list authorization check**, which falls under the High impact category.

### Likelihood Explanation

Exploitation requires no special privileges. Any user who knows an address is on the blocklist can immediately route a Cosmos SDK transfer to it. The two code paths (EVM vs. Cosmos SDK) are both standard, publicly documented transaction types on Cronos.

### Recommendation

Apply the same destination-address check to Cosmos SDK messages inside both `BlockAddressesDecorator.AnteHandle` and `ProposalHandler.ValidateTransaction`. For each non-EVM message, extract the recipient field (e.g., `bank.MsgSend.ToAddress`, `ibc.MsgTransfer.Receiver`) and verify it against the blocklist map, mirroring the existing EVM destination check at lines 51–54 of `app/block_address.go`.

### Proof of Concept

1. Admin adds address `A` to the blocklist via `MsgStoreBlockList`.
2. User `B` (not blocked) attempts an EVM transfer of 1 CRO to `A` → **rejected** by `BlockAddressesDecorator` (destination check, line 52 of `app/block_address.go`).
3. User `B` instead submits a Cosmos SDK `bank.MsgSend{FromAddress: B, ToAddress: A, Amount: coins}` → **accepted**, because `AnteHandle` only checks the signer (`B`), not the recipient (`A`).
4. The transaction is included in a block; `A` receives the tokens.
5. If `A` is later removed from the blocklist, it can freely spend those tokens.

The same bypass works via `ibc.MsgTransfer` with `Receiver = A`, or via `cronos.MsgConvertVouchers` where the signer is `B` but the converted coins are sent to `A`'s address indirectly through a contract call.

func (bad BlockAddressesDecorator) AnteHandle(ctx sdk.Context, tx sdk.Tx, simulate bool, next sdk.AnteHandler) (newCtx sdk.Context, err error) {
	if ctx.IsCheckTx() {
		if sigTx, ok := tx.(signing.SigVerifiableTx); ok {
			signers, err := sigTx.GetSigners()
			if err != nil {
				return ctx, err
			}
			for _, signer := range signers {
				if _, ok := bad.blockedMap[sdk.AccAddress(signer).String()]; ok {
					return ctx, fmt.Errorf("signer is blocked: %s", sdk.AccAddress(signer).String())
				}
			}
		}

		for _, msg := range tx.GetMsgs() {
			msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
			if ok {
				ethTx := msgEthTx.AsTransaction()
				// check the destination address
				if ethTx.To() != nil {
					if _, ok := bad.blockedMap[sdk.AccAddress(ethTx.To().Bytes()).String()]; ok {
						return ctx, fmt.Errorf("destination address is blocked: %s", sdk.AccAddress(ethTx.To().Bytes()).String())
					}
				}
				// check EIP-7702 authorisation list
				if ethTx.SetCodeAuthorizations() != nil {
					for _, auth := range ethTx.SetCodeAuthorizations() {
						addr, err := auth.Authority()
						if err == nil {
							if _, ok := bad.blockedMap[sdk.AccAddress(addr.Bytes()).String()]; ok {
								return ctx, fmt.Errorf("signer is blocked: %s", addr.String())
							}
						}
						// check the target address
						if _, ok := bad.blockedMap[sdk.AccAddress(auth.Address.Bytes()).String()]; ok {
							return ctx, fmt.Errorf("authorisation address is blocked: %s", sdk.AccAddress(auth.Address.Bytes()).String())
						}
					}
				}
			}
