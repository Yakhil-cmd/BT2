### Title
Consensus Key Re-utilization in `CreateValidator` Allows Reuse of Decommissioned Keys - (File: `sei-cosmos/x/staking/keeper/msg_server.go`)

---

### Summary

`CreateValidator` only checks whether a consensus key is currently registered in the live validator store. Because `RemoveValidator` deletes the `ValidatorsByConsAddrKey` entry for a fully removed validator, a previously used consensus key passes the uniqueness check and can be registered again by any party who holds the corresponding private key.

---

### Finding Description

`CreateValidator` performs two uniqueness checks before registering a new validator:

1. **Operator address check** — `GetValidator(ctx, valAddr)` at line 43 [1](#0-0) 
2. **Consensus key check** — `GetValidatorByConsAddr(ctx, sdk.GetConsAddress(pk))` at line 52 [2](#0-1) 

`GetValidatorByConsAddr` reads exclusively from the `ValidatorsByConsAddrKey` KV store prefix: [3](#0-2) 

When a validator is fully removed (unbonded, zero tokens), `RemoveValidator` **deletes** that same store entry at line 177: [4](#0-3) 

There is no secondary historical-key store, no tombstone, and no other mechanism that would cause the check at line 52 of `msg_server.go` to reject a previously used key after its validator has been removed. The `HistoricalInfoKey` prefix in `keys.go` stores block-level historical info, not a registry of used consensus keys. [5](#0-4) 

---

### Impact Explanation

An attacker who possesses a consensus private key that was previously used by a now-removed validator can call `CreateValidator` (directly via Cosmos tx or via the EVM staking precompile) with that key. The check at line 52 returns `found = false` because the key's store entry was erased by `RemoveValidator`. A new validator is then registered and can sign blocks with the recycled key.

Concrete harm:
- If the original validator was decommissioned **because its key was compromised**, the attacker holding the leaked private key can immediately re-enter the active validator set under a new operator identity.
- If the original validator node is still running (e.g., misconfiguration during decommission), two validators sharing the same consensus key will produce conflicting votes, triggering double-sign slashing on the new validator and potentially causing consensus instability.

Severity: **Medium** — staking integrity violation; does not directly cause fund loss at scale but can corrupt the validator set and undermine the security intent of key decommissioning.

---

### Likelihood Explanation

The prerequisite is possession of a previously used consensus private key. This is realistic in the following scenarios:
- A validator operator leaked or lost control of their consensus key and decommissioned to mitigate it; the attacker who obtained the key can now re-register.
- A validator operator intentionally decommissions and later wants to reuse the same key under a different operator address (self-inflicted, but still violates the protocol's intended invariant).

The precompile entry points (`precompiles/staking/staking.go` and legacy versions) all ultimately call `p.stakingKeeper.CreateValidator`, so the same path is reachable from EVM contracts. [6](#0-5) 

---

### Recommendation

Maintain a persistent set of "used consensus addresses" that is written to in `RemoveValidator` (or `AfterValidatorRemoved`) and checked in `CreateValidator` before registering a new key. The check at line 52 of `msg_server.go` should be extended to also query this historical set:

```go
// In RemoveValidator, after deleting the live entry:
k.SetHistoricalConsAddr(ctx, valConsAddr)  // write to a tombstone store

// In CreateValidator, alongside the existing check:
if k.IsHistoricalConsAddr(ctx, sdk.GetConsAddress(pk)) {
    return nil, types.ErrValidatorPubKeyUsedHistorically
}
```

---

### Proof of Concept

1. Alice creates validator `V1` with consensus key `K` (private key `k`).
2. All delegators undelegate from `V1`; after the unbonding period, `V1`'s tokens reach zero.
3. `RemoveValidator` is called for `V1`, deleting `ValidatorsByConsAddrKey[K]`. [7](#0-6) 
4. Attacker (holding `k`) submits `MsgCreateValidator` with `Pubkey = K` and a new operator address `V2`.
5. `GetValidatorByConsAddr(K)` returns `found = false` (entry was deleted). [3](#0-2) 
6. `CreateValidator` proceeds; `V2` is registered with key `K` and begins signing blocks. [2](#0-1)

func (k Keeper) GetValidatorByConsAddr(ctx sdk.Context, consAddr sdk.ConsAddress) (validator types.Validator, found bool) {
	store := ctx.KVStore(k.storeKey)

	opAddr := store.Get(types.GetValidatorByConsAddrKey(consAddr))
	if opAddr == nil {
		return validator, false
	}

	return k.GetValidator(ctx, opAddr)
}
