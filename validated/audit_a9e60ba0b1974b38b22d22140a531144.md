### Title
Signature Verification Bypass via `skip_stateful_validations` Allows Unauthenticated Invoke Admission — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX feature for the deploy-account + invoke flow unconditionally skips the account's `__validate__` entry point for any invoke transaction with `nonce == 1` targeting an address that has any transaction in the mempool. Because the check uses only a mempool-presence query — not a cryptographic binding to the deploy-account — any unprivileged attacker can submit an invoke with an arbitrary/invalid signature for a victim's undeployed address and have it admitted to the mempool without signature verification.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements the following logic: [1](#0-0) 

The condition to skip `__validate__` is:
1. Transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in committed state)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

When all four conditions hold, `skip_validate = true` is returned, and `run_validate_entry_point` is called with `validate: false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate == false` for an Invoke, execution returns immediately after `perform_pre_validation_stage` — the account's `__validate__` entry point is never called: [3](#0-2) 

The mempool-presence check used to justify skipping validation is: [4](#0-3) 

This returns `true` if the account has **any** transaction in the pool — including a deploy-account submitted by the victim. There is no cryptographic binding between the deploy-account and the subsequent invoke.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an `Invoke` with `nonce = 1` for the victim's address carrying an **arbitrary or invalid signature**. Because `skip_stateful_validations` returns `true`, the gateway calls `run_validate_entry_point` with `validate: false`, so the account's `__validate__` entry point is never executed. The transaction is forwarded to the mempool and accepted.

The attacker's invalid invoke occupies `nonce = 1` for the victim's address. When the victim subsequently submits their own legitimate invoke with `nonce = 1`, the mempool rejects it as a duplicate nonce (or forces the victim to pay a higher fee to replace it via fee escalation). The attacker's invalid invoke will fail at execution time (blockifier calls `__validate__` with `strict_nonce_check = true` during block production), but because `__validate__` failure results in transaction rejection with no fee charged, the attacker bears zero cost and can repeat the attack indefinitely.

**Broken invariant**: Every transaction admitted to the mempool must have passed the account's `__validate__` entry point. This invariant is violated for invoke transactions with `nonce == 1` targeting any address with a pending deploy-account.

### Likelihood Explanation

The attack requires only:
1. Observing a victim's `deploy_account` transaction in the public mempool (trivially observable via P2P gossip or the gateway API)
2. Submitting an invoke with `nonce = 1` for the victim's address before the victim does

No privileged access, special keys, or funds are required. The attacker pays no fees on failure. The attack window is the time between the victim's `deploy_account` entering the mempool and the victim's first invoke being submitted — a window that is explicitly widened by the UX feature itself.

### Recommendation

Replace the mempool-presence check with a cryptographic binding. The deploy-account transaction hash should be included in the invoke's transaction hash or verified as a prerequisite. Concretely:

- Require the invoke to carry the deploy-account transaction hash as a field, and verify it matches a pending deploy-account for the same address before skipping `__validate__`.
- Alternatively, do not skip `__validate__` entirely; instead run it against a speculative state that includes the pending deploy-account execution, so the account contract exists and can verify the signature.

The current check in `skip_stateful_validations`: [5](#0-4) 

must be strengthened to verify that the account in the mempool is specifically a `DeployAccount` transaction for the same address, and that the invoke's signature is cryptographically bound to that specific deploy.

### Proof of Concept

1. Victim generates a new account key pair; computes deploy address `A`; submits `deploy_account` for `A` → accepted into mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker observes `A` in the mempool. Attacker constructs `Invoke { sender_address: A, nonce: 1, signature: [0xdead, 0xbeef] }` (arbitrary invalid signature).
3. Gateway processes attacker's invoke:
   - `validate_nonce`: `0 <= 1 <= max_allowed_nonce_gap` → passes. [6](#0-5) 
   - `validate_by_mempool`: no duplicate nonce=1 exists → passes.
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_in_pool==true` → returns `true`. [7](#0-6) 
   - `run_validate_entry_point` called with `validate: false` → `__validate__` never executed. [8](#0-7) 
4. Attacker's invalid-signature invoke is admitted to the mempool, occupying `nonce = 1` for address `A`.
5. Victim submits their legitimate invoke with `nonce = 1` → mempool rejects it as `DuplicateNonce`.
6. Attacker's invoke fails at blockifier execution (`__validate__` called with `strict_nonce_check: true`, signature invalid → rejected, no fee charged). Attacker repeats from step 2 at zero cost.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
