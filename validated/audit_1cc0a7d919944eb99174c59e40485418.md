### Title
Wrong `nftSellerNetAmount` Formula Causes NFT Offer Viewer to Display Inflated "Net Proceeds" and Suppress Negative-Amount Warning, Inducing NFT Sellers to Accept Underpaying Offers - (File: `packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` hardcodes `nftSellerNetAmount = amount` instead of `amount - royaltyAmount`. The correct formula was commented out. This wrong value propagates into two security-relevant display paths in `NFTOfferViewer`: the "Net Proceeds" figure shown to an NFT seller before they click "Accept Offer", and the `overrideNFTSellerAmount` that replaces the raw on-chain amount in the "You will receive" row of the offer summary. Both show the full pre-royalty amount. Additionally, the negative-amount guard in `NFTOfferEditor` is permanently suppressed.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The correct formula was intentionally commented out, leaving `nftSellerNetAmount` equal to the gross `amount` regardless of royalty percentage or maker fee.

This value is consumed in two places:

**Path 1 — NFTOfferViewer "Net Proceeds" and `overrideNFTSellerAmount`:**

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // wrong: equals full amount
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

`overrideNFTSellerAmount` is passed into `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it **replaces** the raw on-chain amount:

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [3](#0-2) 

The "You will receive" row therefore shows the full gross amount instead of `amount - royaltyAmount`.

The "Net Proceeds" label (explicitly described in its tooltip as "the asking price, minus any associated creator fees") also renders the wrong value:

```typescript
<FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} /> {displayName}
``` [4](#0-3) 

**Path 2 — NFTOfferEditor negative-amount guard permanently suppressed:**

```typescript
const showNegativeAmountWarning = (nftSellerNetAmount ?? 0) < 0;
``` [5](#0-4) 

Because `nftSellerNetAmount` is always the positive `amount`, this warning never fires even when `royaltyAmount + makerFee > amount`.

### Impact Explanation

For a `TokenForNFT` offer (buyer creates offer, NFT seller accepts), the NFT seller views the offer before clicking "Accept Offer". The GUI shows:

- **"You will receive: 10 XCH"** (wrong — should be 5 XCH after a 50% royalty)
- **"Net Proceeds: 10 XCH"** (wrong — same bug)

The NFT seller accepts believing they will receive 10 XCH. On-chain, 5 XCH is routed to the creator's royalty address and the seller receives only 5 XCH. The discrepancy is proportional to the royalty percentage and scales with offer size.

This fits the allowed High impact: corruption of offer state that causes a user to approve the wrong amount.

### Likelihood Explanation

Every `TokenForNFT` offer involving an NFT with a non-zero `royaltyPercentage` triggers the bug. No special attacker capability is required — any buyer can create such an offer. The NFT seller has no indication from the GUI that the displayed "Net Proceeds" is wrong; the royalty percentage is shown separately but the GUI explicitly presents `nftSellerNetAmount` as the post-royalty figure.

### Recommendation

Restore the correct formula in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12))
    : amount; // NFTForToken: seller receives full asking price; buyer pays royalties on top
``` [6](#0-5) 

Also verify that `overrideNFTSellerAmount` in `NFTOfferViewer` is only applied to the "offered" (maker) side of the summary row, not both sides, to avoid unintended overrides.

### Proof of Concept

1. Mint an NFT with `royaltyPercentage = 5000` (50%).
2. As a buyer, open the Chia GUI and create a `TokenForNFT` offer: offer 10 XCH for that NFT.
3. Share the offer file with the NFT seller.
4. NFT seller opens the offer file in the GUI (`NFTOfferViewer`).
5. Observe: "You will receive" row shows **10 XCH**; "Net Proceeds" shows **10 XCH**.
6. NFT seller clicks "Accept Offer".
7. On-chain result: NFT seller receives **5 XCH**; creator receives **5 XCH** as royalty.
8. The GUI never showed the correct net proceeds of 5 XCH, and the negative-amount guard at `NFTOfferEditor.tsx:271` would also never have fired even if the royalty exceeded 100% of the offered amount.
