### Title
Unchecked Precompile Return Value After `_burn` in `withdrawToNear` Allows Permanent Fund Loss — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`withdrawToNear` in `EvmErc20V2` burns the caller's ERC-20 tokens before calling the `ExitToNear` precompile, and never checks the assembly `call` return value. If the precompile rejects the input (e.g., empty `recipient`), the burn is committed but no NEAR-side transfer is scheduled, permanently destroying the tokens.

---

### Finding Description

In `EvmErc20V2.sol`, `withdrawToNear` follows this sequence:

```solidity
// etc/eth-contracts/contracts/EvmErc20V2.sol lines 53-64
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // (1) burn commits unconditionally

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // (2) res is captured but NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

The `_burn` on line 55 executes and commits state before the precompile is called. The assembly block captures `res` (the EVM `call` success flag) but performs no conditional check on it — there is no `if iszero(res) { revert(0,0) }` or equivalent.

On the precompile side, `ExitToNearParams::try_from` parses the input:

```rust
// engine-precompiles/src/native.rs lines 758-763
0x1 => {
    let amount = parse_amount(&input[..32])?;
    let Recipient { receiver_account_id, message } =
        parse_recipient(&input[32..])?;   // empty slice → parse fails → ExitError
``` [2](#0-1) 

When `recipient` is `bytes('')` (zero-length), `parse_recipient` receives an empty slice. A NEAR `AccountId` requires at least 2 characters, so parsing fails and the precompile returns an `ExitError`. The EVM translates this into a `call` return value of `0`. Because `EvmErc20V2` never inspects `res`, execution continues normally and the function returns without reverting.

Additionally, `parse_input` calls `validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)`:

```rust
// engine-precompiles/src/native.rs lines 788-791
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
``` [3](#0-2) 

If `MIN_INPUT_SIZE` exceeds the 53-byte payload produced by an empty recipient (`1 + 20 + 32 + 0`), the call fails even earlier at size validation — still returning `0` to the Solidity caller, still unchecked.

The same unchecked-return pattern exists in the V1 contract (`EvmErc20.sol` line 61), but the question is scoped to `EvmErc20V2.sol`. [4](#0-3) 

---

### Impact Explanation

The ERC-20 balance is permanently reduced with no corresponding NEP-141 transfer on NEAR. The tokens are neither on the EVM side (burned) nor on the NEAR side (no promise scheduled). They are unrecoverable. This constitutes **permanent freezing/destruction of funds** — a Critical impact.

---

### Likelihood Explanation

Any token holder can trigger this by passing `bytes('')` as the `recipient` argument. No special privilege, admin access, or external dependency is required. The call is a standard EVM transaction available to any EOA or contract holding a nonzero balance. Likelihood is **High**.

---

### Recommendation

Check the precompile return value and revert if it indicates failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively, validate `recipient.length > 0` (and ideally enforce a minimum valid NEAR account ID length) before calling `_burn`. The safest pattern is: validate inputs → call precompile → only burn on success (or use a checks-effects-interactions ordering where the precompile is called first).

Apply the same fix to `withdrawToEthereum` and to `EvmErc20.sol` (V1).

---

### Proof of Concept

1. Deploy `EvmErc20V2` on a local Aurora sandbox.
2. Mint `1000` tokens to `attacker`.
3. Call `withdrawToNear(bytes(''), 1000)` from `attacker`.
4. Observe: `balanceOf(attacker) == 0` (burn committed).
5. Observe: no `ft_transfer` NEAR promise was scheduled (precompile rejected the call silently).
6. Tokens are permanently lost.

Fuzz variant: iterate `recipient` byte lengths `[0, 1]`; for length 0, assert the call reverts OR a valid promise is emitted — the assertion fails for length 0, confirming the bug.
