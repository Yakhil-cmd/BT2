### Title
Gateway Admits Invoke Transactions with Invalid Signatures via `skip_stateful_validations` Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call for invoke transactions when `tx.nonce == 1` and `account_nonce == 0` and `account_tx_in_pool_or_recent_block(sender) == true`. The mempool check does not verify that the matching pool entry is specifically a `deploy_account` transaction. An attacker who observes a legitimate `deploy_account` for address X in the mempool can submit an invoke from X with nonce=1 and an **invalid signature**; the gateway skips signature validation, admits the transaction, and the attacker can use fee escalation to replace the victim's valid invoke.

### Finding Description

`skip_stateful_validations` is designed for the UX pattern where a user submits `deploy_account + invoke` simultaneously. Because the account does not yet exist on-chain, `__validate__` cannot be called for the invoke. The gateway instead checks whether the account has any transaction in the mempool or a recent block as a proxy for "a deploy_account is pending":

### Impact Explanation

**Attack scenario:**

1. Alice submits `deploy_account(X)` + `invoke(X, nonce=1, valid_sig, fee=F)`. Both are admitted to the mempool.
2. Bob observes Alice's `deploy_account` in the mempool (via P2P gossip or gateway monitoring).
3. Bob submits `invoke(X, nonce=1, invalid_sig, fee=F+1)`.
4. Gateway evaluates: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` (Alice's deploy_account is in the pool) → `skip_validate=true`.
5. `__validate__` is **not called**; Bob's invalid-signature invoke is admitted to the mempool.
6. Bob's invoke has a higher fee, so it replaces Alice's invoke via fee escalation.
7. Batcher executes: `deploy_account(X)` succeeds; Bob's `invoke(X, nonce=1)` is executed with `validate=true` (default batcher flags), `__validate__` fails, transaction reverts — but fees are charged from X (Alice's account).
8. Alice's valid invoke was evicted from the mempool and must be resubmitted.

**Concrete broken invariant:** The gateway's admission invariant — "every admitted invoke transaction has passed `__validate__` or is provably safe to skip" — is violated. An attacker with no relationship to address X can bypass signature validation by piggybacking on any legitimate `deploy_account` for X that is in the mempool.

This matches the allowed impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The condition is easy to trigger: the attacker only needs to observe a `deploy_account` transaction in the mempool (publicly visible via P2P) and submit a competing invoke with a higher fee. No privileged access is required. The `deploy_account + invoke` UX pattern is explicitly supported and documented, making it a predictable target. [6](#0-5) 

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** for the sender address is present in the mempool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query that inspects transaction type, not just address presence. This ensures the skip is only granted when the account's own deploy is pending, not when any arbitrary transaction for that address exists.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to
