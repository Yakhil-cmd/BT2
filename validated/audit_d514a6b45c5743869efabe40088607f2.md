### Title
FLOW Tokens Permanently Frozen When `EVM.CadenceOwnedAccount` Resource Is Destroyed With Non-Zero EVM Balance - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVM.CadenceOwnedAccount` (COA) resource in Flow's EVM contract has no `destroy` hook to automatically withdraw the EVM-side FLOW balance back to Cadence before the resource is destroyed. Because a COA's EVM address is not controlled by any cryptographic key, destroying the Cadence resource while the EVM address holds a non-zero balance permanently and irrecoverably freezes those FLOW tokens.

---

### Finding Description

`EVM.CadenceOwnedAccount` is the sole Cadence-side controller of a deterministically derived EVM address. The contract explicitly states that "users are never able to access a key for their account, meaning that they cannot control their COA's address on other EVM blockchains." [1](#0-0) 

The resource exposes a `withdraw` function that correctly moves FLOW tokens from the EVM balance back to a Cadence `FlowToken.Vault`: [2](#0-1) 

However, the `CadenceOwnedAccount` resource definition contains **no `destroy` hook** that would automatically call `withdraw` (or otherwise drain the EVM balance) before the Cadence resource is destroyed: [3](#0-2) 

When `destroy coa` is executed on a COA that holds a non-zero EVM balance:
1. The Cadence resource object is destroyed.
2. The EVM address retains its balance in the EVM state.
3. There is no private key for the EVM address.
4. There is no mechanism to recreate a COA bound to the same EVM address.
5. The FLOW tokens are permanently inaccessible.

This is structurally identical to the Alchemix `merge` bug: the `withdraw` path performs the necessary cleanup (reward claim / balance withdrawal) but the destruction path does not.

Multiple test cases in the repository demonstrate this pattern — depositing FLOW into a COA and then destroying it without withdrawing — and they succeed without error: [4](#0-3) [5](#0-4) [6](#0-5) 

The token-accounting test even labels this outcome as expected: `"all tokens should be accounted for (EVM deposit is a known sink)"` — confirming the protocol treats the deposited tokens as gone from the Cadence side, with no recovery path once the COA is destroyed. [7](#0-6) 

---

### Impact Explanation

Any FLOW tokens deposited into a `CadenceOwnedAccount` that is subsequently destroyed without a prior `withdraw` call are permanently frozen at the orphaned EVM address. The address has no key, no recreatable COA, and no other access path. This matches the target impact: **permanent freezing of user-owned on-chain assets**.

---

### Likelihood Explanation

The pattern is reachable by any unprivileged transaction sender who owns a COA. No admin action is required. The risk is elevated because:
- The `deposit` function is `access(all)` (no entitlement required), making it easy to fund a COA.
- The `destroy` keyword is a standard Cadence resource disposal pattern; users unfamiliar with the COA architecture may use it without first calling `withdraw`.
- The codebase itself contains multiple test examples of `deposit` followed immediately by `destroy` without an intervening `withdraw`.

An additional aggravating scenario: the `withdraw` function has a `pre` condition `!EVM.isPaused()`. If EVM is paused at the time a user wishes to close their COA, they cannot withdraw first, and destroying the COA while paused permanently loses the balance. [8](#0-7) 

---

### Recommendation

Add a `destroy` hook (Cadence `ResourceDestroyedEvent` or a pre-condition) to `CadenceOwnedAccount` that either:

1. **Auto-withdraws**: Automatically withdraws the full EVM balance to a designated Cadence vault before the resource is destroyed, or
2. **Guards destruction**: Adds a pre-condition that panics if `self.balance().attoflow > 0`, forcing the caller to drain the balance explicitly before destroying.

Option 2 is simpler and consistent with how Cadence `FungibleToken.Vault` resources are expected to be handled (destroyed only when empty).

---

### Proof of Concept

```cadence
import EVM from 0x...
import FlowToken from 0x...
import FungibleToken from 0x...

transaction() {
    prepare(account: auth(Storage) &Account) {
        let vaultRef = account.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
                from: /storage/flowTokenVault)
            ?? panic("no vault")

        // Fund a COA with 10 FLOW
        let coa <- EVM.createCadenceOwnedAccount()
        let amount <- vaultRef.withdraw(amount: 10.0) as! @FlowToken.Vault
        coa.deposit(from: <-amount)

        // coa.balance() == 10.0 FLOW at this point.
        // Destroy without withdrawing — no error, no refund.
        destroy coa

        // The EVM address still holds 10.0 FLOW.
        // No key exists for it. No COA can be recreated at that address.
        // The 10.0 FLOW are permanently inaccessible.
    }
}
```
