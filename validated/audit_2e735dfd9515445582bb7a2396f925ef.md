### Title
Pool `change_reward_address()` Missing STRK Reward Token Guard Allows Permanent Yield Freeze in Non-STRK Pools - (File: src/pool/pool.cairo)

### Summary
In `pool.cairo`, `change_reward_address()` only validates that the new reward address is not equal to the pool's own staking token. For non-STRK pools (e.g., BTC pools), this check does not cover the STRK reward token address. A pool member can set their reward address to the STRK token contract, causing all future reward transfers to be permanently locked inside the STRK ERC-20 contract with no recovery path.

### Finding Description
The pool contract's `change_reward_address` performs a single-token guard:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
``` [1](#0-0) 

`self.token_dispatcher` holds the pool's **staking** token (e.g., BTC for a BTC pool). The check therefore only rejects `reward_address == BTC_TOKEN_ADDRESS`. It does **not** reject `reward_address == STRK_TOKEN_ADDRESS`.

By contrast, the staking contract's `change_reward_address` uses `does_token_exist`, which iterates over **all** registered tokens (both STRK and BTC):

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

The pool's `claim_rewards` always pays out in STRK regardless of the pool's staking token:

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [3](#0-2) 

When `reward_address` has been set to `STRK_TOKEN_ADDRESS`, this transfer sends STRK into the STRK ERC-20 contract itself. Standard ERC-20 contracts have no mechanism to recover tokens sent to their own address, so the yield is permanently frozen.

The same incomplete guard is present in `enter_delegation_pool`, which also emits `REWARD_ADDRESS_IS_TOKEN` only for the pool's own staking token, as confirmed by the existing test suite:

```cairo
// Catch REWARD_ADDRESS_IS_TOKEN.
let token_address = token.contract_address();   // pool's staking token only
let result = pool_safe_dispatcher.enter_delegation_pool(reward_address: token_address, :amount);
assert_panic_with_error(:result, expected_error: GenericError::REWARD_ADDRESS_IS_TOKEN.describe());
``` [4](#0-3) 

No test covers the case where `reward_address == STRK_TOKEN_ADDRESS` for a BTC pool, confirming the gap.

### Impact Explanation
A pool member in a BTC delegation pool who sets `reward_address = STRK_TOKEN_ADDRESS` (accidentally or through a UI bug) will have all accrued STRK rewards transferred into the STRK token contract on every `claim_rewards` call. Those tokens are permanently unrecoverable. This matches the allowed impact: **Permanent freezing of unclaimed yield** (High). [3](#0-2) 

### Likelihood Explanation
BTC pool support is an explicit, deployed feature of the protocol. [5](#0-4) 

Any pool member of a BTC pool can trigger this with a single call to `change_reward_address(STRK_TOKEN_ADDRESS)`. No privileged access is required. The call succeeds silently — no error is raised, no warning is emitted — making accidental triggering plausible (e.g., a user mistakenly pasting the STRK token address instead of their wallet address).

### Recommendation
Replace the single-token check in `pool.cairo`'s `change_reward_address` (and `enter_delegation_pool`) with a check against all registered tokens, mirroring the staking contract's approach. The simplest fix is to also assert that `reward_address != STRK_TOKEN_ADDRESS`:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    let pool_token = self.token_dispatcher.contract_address.read();
    assert!(
        pool_token != reward_address && STRK_TOKEN_ADDRESS != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

A more future-proof approach would be to call back into the staking contract's `does_token_exist` view, so that any newly added token is automatically covered.

### Proof of Concept

1. Deploy the system with a BTC pool enabled (as the protocol supports).
2. A pool member calls `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)` on the BTC pool — the call succeeds because `STRK_TOKEN_ADDRESS != BTC_TOKEN_ADDRESS`.
3. Alternatively, an existing BTC pool member calls `change_reward_address(STRK_TOKEN_ADDRESS)` — again succeeds.
4. Rewards accrue over epochs.
5. Anyone calls `claim_rewards(pool_member)`.
6. `pool.cairo` line 365–366 executes: `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
7. STRK tokens are transferred into the STRK ERC-20 contract itself and are permanently frozen — no recovery function exists. [6](#0-5) [7](#0-6)

        fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
            assert!(
                self.token_dispatcher.contract_address.read() != reward_address,
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let old_address = pool_member_info.reward_address;

            // Update reward_address and commit to storage.
            pool_member_info.reward_address = reward_address;
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardAddressChanged {
                        pool_member, new_address: reward_address, old_address,
                    },
                );
        }
