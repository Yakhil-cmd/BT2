### Title
Permanent ERC-20 Token Loss via Missing Refund State Transition in `ExitToNear` Precompile Callback ŌĆö (File: `engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` compile-time feature is disabled, ERC-20 tokens burned by the `ExitToNear` precompile are permanently destroyed if the subsequent NEP-141 `ft_transfer` NEAR promise fails. The callback `exit_to_near_precompile_callback` has no state transition to a "refunded" state when `args.refund` is `None`, leaving the exit flow permanently stuck in a failed terminal state with no recovery path for the user.

---

### Finding Description

The `ExitToNear` precompile burns ERC-20 tokens inside the EVM and emits a log that schedules a NEAR-side `ft_transfer` (or `near_withdraw`) promise. The promise args are constructed in `engine-precompiles/src/native.rs`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ŌåÉ always None when feature is off
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is disabled, `callback_args.refund` is unconditionally `None`. For a standard ERC-20 exit (no wNEAR unwrap), `transfer_near` is also `None`, so `callback_args == default()` and **no callback is attached at all** ŌĆö the promise is a bare `PromiseArgs::Create`:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ŌåÉ no callback, no refund path
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

For wNEAR exits, a callback is attached, but the callback body in `exit_to_near_precompile_callback` has no refund branch when `args.refund` is `None`:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path: optionally transfer NEAR
    None
} else if let Some(args) = args.refund {
    // refund path: only reachable when error_refund feature is ON
    ...
} else {
    None   // ŌåÉ silent no-op; tokens already burned, no recovery
};
``` [3](#0-2) 

The `ExitToNearPrecompileCallbackArgs` struct confirms `refund` is optional and defaults to `None`: [4](#0-3) 

The ERC-20 burn is committed as part of the EVM execution (the EVM state is finalised before any NEAR promise runs). If the NEAR-side `ft_transfer` then fails ŌĆö e.g., because the recipient account is not registered with the NEP-141 contract ŌĆö the tokens are gone with no state transition to recover them.

The test suite explicitly acknowledges this behaviour:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

---

### Impact Explanation

**Critical ŌĆö Permanent freezing / destruction of user funds.**

When `error_refund` is disabled in the production binary:

- ERC-20 tokens are burned inside the EVM (irreversible EVM state change).
- The NEP-141 `ft_transfer` promise fails (e.g., unregistered recipient, zero balance on the connector side).
- No callback refund path exists; the exit flow terminates silently.
- The user has lost their bridged tokens with no on-chain recovery mechanism.

---

### Likelihood Explanation

**Medium.** The failure condition is a routine operational mistake: sending to a NEAR account that has not called `storage_deposit` on the NEP-141 contract. This is a common error for new users bridging tokens. The attacker-controlled entry path is the standard `ExitToNear` precompile, callable by any EVM address via a normal EVM transaction submitted through `submit`. No special privilege is required. [6](#0-5) 

---

### Recommendation

- **Short term:** Enable the `error_refund` feature in the production WASM build so that `refund_call_args` is populated and the callback can mint back the burned ERC-20 tokens on failure.
- **Long term:** Decouple the refund logic from a compile-time feature flag. The refund state transition should be unconditionally present in the callback so that every exit flow has a defined recovery path regardless of build configuration. Document the expected state transitions (burn ŌåÆ transfer ŌåÆ [success | refund]) explicitly.

---

### Proof of Concept

1. Compile Aurora Engine **without** the `error_refund` feature (default build).
2. Bridge a NEP-141 token to Aurora as an ERC-20.
3. Call the `ExitToNear` precompile from an EVM contract, specifying a NEAR recipient that has **not** called `storage_deposit` on the NEP-141 contract.
4. The ERC-20 tokens are burned in the EVM (balance decremented, `Transfer` event emitted to zero address).
5. The NEAR-side `ft_transfer` promise fails with `"The account ... is not registered"`.
6. Because `callback_args == default()`, no `exit_to_near_precompile_callback` is scheduled; there is no refund promise.
7. Observe: ERC-20 balance is zero, NEP-141 balance of recipient is unchanged, and the user has permanently lost their tokens. [7](#0-6) [8](#0-7)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
