### Title
Missing Zero Address Validation in `change_reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both `change_reward_address` in the staking contract and `change_reward_address` in the pool contract accept `ContractAddress` zero (`0x0`) as a valid new reward address. Any staker or pool member can call these functions with a zero address, permanently redirecting all future reward transfers to the burn address and freezing their unclaimed yield.

### Finding Description
`staking.cairo::change_reward_address` (line 517) validates only that the new address is not a registered token address, but performs no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address guard
    staker_info.reward_address = reward_address;
    ...
}
``` [1](#0-0) 

`pool.cairo::change_reward_address` (line 505) has the same gap — it only rejects the token contract address:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address guard
    pool_member_info.reward_address = reward_address;
    ...
}
``` [2](#0-1) 

When rewards are later claimed, `send_rewards_to_staker` unconditionally transfers to whatever `reward_address` is stored:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [3](#0-2) 

And the pool's `claim_rewards` does the same:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

The same missing check exists at initial registration: `stake` and `enter_delegation_pool` also accept `reward_address = 0x0` without rejection. [5](#0-4) [6](#0-5) 

### Impact Explanation
If `reward_address` is set to `0x0`, every subsequent `claim_rewards` call transfers STRK tokens to the zero address. On Starknet, ERC-20 transfers to address `0x0` succeed without reverting, so the tokens are permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope. [7](#0-6) 

### Likelihood Explanation
The entry path requires only that a registered staker or pool member call `change_reward_address(0x0)`. No privileged role is needed. The call can be made accidentally (e.g., a front-end bug passing an uninitialized address) or by a user who later loses access to their account and attempts to "burn" their rewards. The function is publicly callable by any staker or delegator. [8](#0-7) [9](#0-8) 

### Recommendation
Add an explicit zero-address guard in both `change_reward_address` implementations, and in `stake` / `enter_delegation_pool` at initial registration:

```cairo
assert!(!reward_address.is_zero(), "REWARD_ADDRESS_IS_ZERO");
```

This mirrors the existing `REWARD_ADDRESS_IS_TOKEN` guard pattern already present in both contracts. [10](#0-9) [11](#0-10) 

### Proof of Concept
1. Deploy the system normally.
2. As a registered staker, call `staking.change_reward_address(reward_address: 0x0)`. The call succeeds — no assertion fires.
3. Advance epochs, attest, and accumulate rewards.
4. Call `staking.claim_rewards(staker_address)`. The contract calls `checked_transfer(recipient: 0x0, amount: rewards)`. Tokens are transferred to the zero address and are permanently lost.
5. The staker's `unclaimed_rewards_own` is zeroed out, but the tokens are unrecoverable.

The identical path applies to pool members via `pool.change_reward_address(0x0)` followed by `pool.claim_rewards(pool_member)`. [12](#0-11) [13](#0-12)
