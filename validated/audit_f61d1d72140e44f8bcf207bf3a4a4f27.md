Audit Report

## Title
Silent Clamping of Over-Range `icrc2_approve` Amount to `Tokens::MAX` Grants Unintended Maximum Allowance — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary
In `icrc2_approve_not_async`, a user-supplied `amount` field that exceeds `u64::MAX` is silently replaced with `Tokens::MAX` (`u64::MAX` e8s) rather than returning an error. The resulting maximum allowance is committed to the ledger, enabling the designated spender to drain the approver's full balance via `icrc2_transfer_from`. The same function correctly rejects an out-of-range `expected_allowance` with `ApproveError::AllowanceChanged`, making the inconsistency in the `amount` path an unambiguous defect.

## Finding Description
At `rs/ledger_suite/icrc1/ledger/src/main.rs` line 840:

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

`Tokens::try_from(Nat)` (`tokens.rs` lines 291–296) returns `Err` for any `Nat` whose underlying `BigUint` does not fit in `u64`. The `unwrap_or_else` branch discards that error and substitutes `Tokens::MAX` (`u64::MAX` e8s, `tokens.rs` line 142). The resulting `amount` is then embedded directly into an `Approve` operation and committed via `apply_transaction` (lines 863–876). No subsequent check validates that the stored allowance matches the caller's intent.

By contrast, lines 841–854 handle `expected_allowance` conversion failure by returning `ApproveError::AllowanceChanged` — the correct pattern — confirming the `amount` path is not an intentional design choice.

## Impact Explanation
Any caller (no privilege required) can record an allowance of `u64::MAX` e8s (~184 billion tokens) for an arbitrary spender by submitting a single crafted `icrc2_approve` call. The spender can then call `icrc2_transfer_from` to drain the approver's entire balance. This constitutes unauthorized ledger fund drainage affecting any ICRC-2 ledger deployed with this code, matching the **High** impact class: *unauthorized access to ledger/canister-controlled funds where exploitation requires meaningful per-target work or other constraints* (the constraint being that the approver must submit an out-of-range `Nat`, which occurs naturally in canisters performing arithmetic or in wallets encoding "unlimited" approvals).

## Likelihood Explanation
The attack requires only a standard ingress `icrc2_approve` call with a `Nat` value of `2^64` or larger, encodable in Candid by any client. No privileged access, key material, or governance action is needed. Canisters that compute approval amounts arithmetically or that attempt to express "unlimited" allowances are natural sources of such values. The exploit is repeatable and affects every ICRC-2 ledger instance running this code.

## Recommendation
Replace the silent fallback with an explicit error return, mirroring the `expected_allowance` pattern already present in the same function:

```rust
let amount = Tokens::try_from(arg.amount).map_err(|_| ApproveError::GenericError {
    error_code: Nat::from(0u64),
    message: "amount exceeds maximum token value".to_string(),
})?;
```

## Proof of Concept
```rust
// Attacker (or buggy canister acting as approver) submits:
ApproveArgs {
    amount: Nat::from(u64::MAX) + Nat::from(1u64), // 2^64, does not fit in u64
    spender: Account { owner: attacker_principal, subaccount: None },
    from_subaccount: None,
    expected_allowance: None,
    expires_at: None,
    fee: None,
    memo: None,
    created_at_time: None,
}
// Tokens::try_from returns Err → unwrap_or_else substitutes Tokens::MAX (u64::MAX e8s)
// Ledger records allowance = u64::MAX for attacker
// Attacker calls icrc2_transfer_from to drain approver's balance
```

A deterministic integration test using PocketIC can confirm this by asserting that `icrc2_approve` with `amount = 2^64` returns `ApproveError::GenericError` (after the fix) rather than `Ok(block_index)`, and that no allowance is recorded.
