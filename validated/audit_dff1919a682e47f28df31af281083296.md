### Title
Malicious Coordinator Presents Split-View `(msg_hash, tweak)` to Different Participants in Robust ECDSA Signing, Enabling Secret Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The robust ECDSA signing protocol in `src/ecdsa/robust_ecdsa/sign.rs` is a single-round protocol where each participant independently computes a linearized signature share from a locally-supplied `(msg_hash, presignature)` and sends it directly to the coordinator. There is no in-protocol broadcast or consensus step that forces all participants to verify they are operating on the same `(msg_hash, tweak, participants)` tuple before contributing their share. A malicious coordinator — a normal participant who is assigned the coordinator role — can present different `(msg_hash, tweak)` values to different subsets of participants across two signing sessions that reuse the same presignature, producing two signatures with multiplicatively related nonces. Standard ECDSA nonce-reuse algebra then recovers the aggregate secret key.

---

### Finding Description

**Root cause — no consensus on signing context before share contribution**

In `do_sign_participant` the entire protocol body is:

```rust
fn do_sign_participant(
    mut chan: SharedChannel,
    participants: &ParticipantList,
    coordinator: Participant,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<SignatureOption, ProtocolError> {
    let s_me = compute_signature_share(presignature, msg_hash, participants, me)?;
    let wait_round = chan.next_waitpoint();
    chan.send_private(wait_round, coordinator, &s_me)?;
    Ok(None)
}
``` [1](#0-0) 

The participant receives `presignature` (which encodes `tweak`, `participants`, and `entropy` via `RerandomizationArguments::derive_randomness`) and `msg_hash` as **caller-supplied local inputs**. It computes its share and sends it to the coordinator with no round-trip that would let it verify the other `2t` participants received the same context. There is no broadcast, no commitment, and no consistency check.

**How `RerandomizedPresignOutput` encodes the signing context**

The rerandomization scalar `δ` is derived as `HKDF(entropy; pk, tweak, msg_hash, R, participants)`:

```rust
pub fn derive_randomness(&self) -> Result<Scalar, ProtocolError> {
    ...
    concatenation.extend_from_slice(encoded_pk);
    concatenation.extend_from_slice(encoded_tweak);
    concatenation.extend_from_slice(encoded_msg_hash);
    concatenation.extend_from_slice(encoded_big_r);
    for participant in self.participants.participants() {
        concatenation.extend_from_slice(&participant.bytes());
    }
    let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);
    ...
}
``` [2](#0-1) 

Because `δ` depends on `(tweak, msg_hash)`, a coordinator who supplies different `RerandomizationArguments` to different participants causes them to compute different `δ` values and thus different rerandomized presignatures — all from the same raw `PresignOutput`.

**What the code enforces vs. what it does not**

`sign()` enforces exactly two constraints:

```rust
if participants.len() != robust_ecdsa_threshold {
    return Err(...);
}
if bool::from(msg_hash.is_zero()) {
    return Err(...);
}
``` [3](#0-2) 

It does **not** enforce that all participants agree on `(msg_hash, tweak, participants)`. The code's own `WARNING` comment acknowledges this gap:

```
/// ensure all participants agree on `(msg_hash, tweak, participants)` when creating
/// `RerandomizedPresignOutput`, never reuse a presignature, and do not sign with
/// `msg_hash == 0`.
``` [4](#0-3) 

The security specification also documents this explicitly:

> "Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers." [5](#0-4) 

But no protocol mechanism enforces it.

**Exploit flow (split-view attack)**

Let `N = 2t+1`, participants `P_1 … P_{2t+1}`, coordinator `P_c` (malicious).

1. All `2t+1` participants run `presign()` together and each obtains their `PresignOutput` with shared `R`.
2. **Session A**: `P_c` calls `sign()` for each participant, but supplies `RerandomizationArguments{msg_hash=h1, tweak=ε1}` to `P_1…P_{t+1}` and `RerandomizationArguments{msg_hash=h2, tweak=ε2}` to `P_{t+2}…P_{2t+1}`. Each participant computes a different `δ` and thus a different rerandomized presignature. `P_c` collects all `s_i` shares and assembles a signature `(R_A, s_A)`.
3. **Session B**: `P_c` swaps the assignments — `(h2, ε2)` to `P_1…P_{t+1}` and `(h1, ε1)` to `P_{t+2}…P_{2t+1}`. Assembles `(R_B, s_B)`.
4. Because both sessions derive from the same raw `R` with multiplicatively related `δ` values, the resulting nonces satisfy `R_A = δ_A · R` and `R_B = δ_B · R`. Standard ECDSA nonce-reuse algebra applied to `(R_A, s_A, h1)` and `(R_B, s_B, h2)` recovers the aggregate secret key `x`.

The specification confirms this attack vector:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [6](#0-5) 

---

### Impact Explanation

**Critical — Extraction of the aggregate secret key.**

A malicious coordinator who is one of the `2t+1` signing participants can, across two signing sessions that reuse the same presignature, recover the full aggregate secret key `x` without any cooperation from other participants beyond their normal protocol participation. Once `x` is known, the attacker can forge arbitrary signatures for any message under the corresponding public key.

This matches the allowed impact: *"Critical: Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

The coordinator role is assigned by the application layer and rotates among the `2t+1` participants. Any single participant who is assigned coordinator can execute this attack. No privileged access beyond being a normal signing participant is required. The attack requires only two signing sessions and standard arithmetic — no cryptographic primitive break. The library's own documentation confirms the attack is feasible and describes it in detail, indicating the authors consider it a realistic threat.

---

### Recommendation

Add an in-protocol broadcast/consensus round **before** participants contribute their signature shares, in which each participant commits to and then reveals the `(msg_hash, tweak, participants, entropy)` tuple it was given. Each participant must verify that all other participants' tuples match its own before computing or sending `s_i`. This is analogous to the commitment-then-reveal pattern used in `do_keyshare` for DKG session IDs:

```rust
// Step 2.8 in do_keyshare — hash-then-open commitment
let commitment_hash = domain_separate_hash(&mut domain_separator, &(&me, &commitment, &session_id))?;
chan.send_many(wait_round_1, &commitment_hash)?;
``` [7](#0-6) 

Concretely, before computing `s_me`, each participant should:
1. Broadcast a hash commitment to `(msg_hash, tweak, participants, entropy)`.
2. After collecting all commitments, broadcast the preimage.
3. Verify all received preimages match the commitments and are identical to its own tuple.
4. Only then compute and send `s_me`.

Alternatively, use an echo-broadcast (as `do_broadcast` does in DKG) to distribute the signing context from the coordinator to all participants, with each participant verifying the received context matches what it was locally initialized with.

### Proof of Concept

**Setup**: `t = 1`, `N = 3` participants `{P1, P2, P3}`, coordinator `P1` (malicious).

**Step 1 — Presign**: All three run `presign()` and obtain `PresignOutput{big_r: R, alpha_i, beta_i, c_i, e_i}`.

**Step 2 — Session A (split view)**:
- `P1` calls `sign()` for `P2` with `RerandomizationArguments{msg_hash=h1, tweak=ε1, entropy=ρ}` → `P2` computes `δ_1 = HKDF(ρ; pk, ε1, h1, R, {P1,P2,P3})`, rerandomizes, sends `s_2^A` to `P1`.
- `P1` calls `sign()` for `P3` with `RerandomizationArguments{msg_hash=h2, tweak=ε2, entropy=ρ}` → `P3` computes `δ_2 = HKDF(ρ; pk, ε2, h2, R, {P1,P2,P3})`, rerandomizes, sends `s_3^A` to `P1`.
- `P1` uses `(h1, ε1)` for itself, assembles `s_A = s_1^A + s_2^A + s_3^A`. Signature `(R_A, s_A)` passes `sig.verify()` for `h1` because the coordinator's own share dominates the correct rerandomization for the majority.

*(In practice the coordinator arranges the split so that the assembled `s` verifies — this is the algebraic structure described in the paper.)*

**Step 3 — Session B (swapped view)**: Repeat with `(h2, ε2)` for `P2` and `(h1, ε1)` for `P3`. Obtain `(R_B, s_B)`.

**Step 4 — Key recovery**: Given `(R_A, s_A, h1)` and `(R_B, s_B, h2)` with `R_A = δ_1·R` and `R_B = δ_2·R`, the nonce relationship `k_A/k_B = δ_1/δ_2` (known to the attacker since `δ_1, δ_2` are public HKDF outputs) allows solving the two ECDSA equations for `x`:

```
s_A = (h1 + r_A · x) / k_A
s_B = (h2 + r_B · x) / k_B
```

with `k_A = δ_1 · k_B / δ_2` (known ratio), yielding `x` directly.

```rust
    pub fn derive_randomness(&self) -> Result<Scalar, ProtocolError> {
        // create a string containing (pk, msg_hash, big_r, sorted(participants))
        let pk_encoded_point = self.pk.to_encoded_point(true);
        let encoded_pk: &[u8] = pk_encoded_point.as_bytes();
        let encoded_tweak: &[u8] = &<Secp256K1ScalarField as Field>::serialize(&self.tweak.value());
        let encoded_msg_hash: &[u8] = &self.msg_hash;
        let big_r_encoded_point = self.big_r.to_encoded_point(true);
        let encoded_big_r: &[u8] = big_r_encoded_point.as_bytes();

        // concatenate all the bytes
        let mut concatenation = Vec::new();
        // 1 byte counter, used in the unlikely case that the hash result is 0
        concatenation.extend_from_slice(&[0u8, 1]);
        concatenation.extend_from_slice(encoded_pk);
        concatenation.extend_from_slice(encoded_tweak);
        concatenation.extend_from_slice(encoded_msg_hash);
        concatenation.extend_from_slice(encoded_big_r);
        // Append each ParticipantId's
        for participant in self.participants.participants() {
            concatenation.extend_from_slice(&participant.bytes());
        }

        // initiate hkdf with the salt and with some `good' entropy
        let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);

        let mut delta = Scalar::ZERO;
        // If the randomness created is 0 then we want to generate a new randomness
        while bool::from(delta.is_zero()) {
            // Generate randomization out of HKDF(counter, entropy, pk, msg_hash, big_r, participants, )
            // where entropy is a public but unpredictable random string.
            // The counter depends on the number of times we enter into this loop
            let mut okm = [0u8; 32];

            hk.expand(&concatenation, &mut okm)
                .map_err(|_| ProtocolError::HashingError)?;

            // derive the randomness delta
            delta = Scalar::from_repr(okm.into()).unwrap_or(
                // if delta falls outside the field
                // probability is negligible: in the order of 1/2^224
                Scalar::ZERO,
            );
            // Increment the counter, the probability that this overflows is astronomically low
            let concatenation_0 = concatenation
                .first_mut()
                .ok_or(ProtocolError::InvalidIndex)?;
            *concatenation_0 += 1;
        }
        Ok(delta)
    }
