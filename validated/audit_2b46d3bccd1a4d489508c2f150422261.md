### Title
BTC Pool `change_reward_address` and `enter_delegation_pool` Omit STRK Reward Token Guard, Enabling Permanent Burning of Delegator Yield — (File: src/pool/pool.cairo)

---

### Summary
The pool contract validates that a delegator's reward address is not the pool's **staked** token address. For BTC pools, this check only guards against the BTC token address, not the STRK token address — which is the **actual reward token** for every pool type. A BTC pool member can therefore set their reward address to `STRK_TOKEN_ADDRESS`. When `claim_rewards` is called, the STRK reward transfer goes to the STRK token contract itself, permanently burning the delegator's unclaimed yield.

---

### Finding Description

The pool contract stores the pool's staked token in `token_dispatcher` and uses it as the sole guard in two entry points.

**`enter_delegation_pool` (line 195):** [1](#0-0) 

```rust
let token_address = token_dispatcher.contract_address;
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

**`change_reward_address` (line 505–510):** [2](#0-1) 

```rust
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

For a **STRK pool**, `token_dispatcher` holds the STRK token address, so the check incidentally also blocks `STRK_TOKEN_ADDRESS` as a reward address — the correct behavior.

For a **BTC pool**, `token_dispatcher` holds the BTC token address. The check only blocks `BTC_TOKEN_ADDRESS`; it does **not** block `STRK_TOKEN_ADDRESS`.

`claim_rewards` always pays out in STRK, hardcoded unconditionally: [3](#0-2) 

```rust
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

The staking contract confirms this design explicitly: [4](#0-3) 

```rust
// Rewards are always in STRK.
self.send_rewards_to_delegation_pool(
    :staker_address,
    pool_address: pool_contract,
    amount: pool_rewards,
    token_dispatcher: strk_token_dispatcher,
);
```

Because the reward denomination (STRK) is decoupled from the staked denomination (BTC), the guard that was designed to prevent reward-address-equals-token-address only works for STRK pools. For BTC pools the guard is blind to the actual reward token.

---

### Impact Explanation

If a BTC pool member's `reward_address` is `STRK_TOKEN_ADDRESS`, every call to `claim_rewards` transfers STRK into the STRK token contract itself. Those tokens are permanently inaccessible — there is no recovery path. This constitutes **permanent freezing of unclaimed yield** (High impact).

---

### Likelihood Explanation

Low-to-medium. The pool member must supply or later change to `STRK_TOKEN_ADDRESS` as their reward address. Realistic vectors:

1. **Accidental misconfiguration** — a BTC pool member who is unfamiliar with the dual-token design might paste the STRK token address (the most prominent address in the system) as their reward address, believing rewards will arrive in STRK at that address.
2. **Griefing via social engineering** — an attacker convinces a pool member to call `change_reward_address(STRK_TOKEN_ADDRESS)`, framing it as a "direct STRK reward" address.

The check exists precisely to prevent this class of mistake; its absence for BTC pools is an unintended gap.

---

### Recommendation

Extend both guards to also reject the STRK reward token address, regardless of pool type:

```rust
// In enter_delegation_pool and change_reward_address:
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
assert!(STRK_TOKEN_ADDRESS != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

Alternatively, factor the check into a helper that always validates against both the pool's staked token and the protocol-wide reward token (`STRK_TOKEN_ADDRESS`).

---

### Proof of Concept

1. A staker calls `set_open_for_delegation(token_address: BTC_TOKEN_ADDRESS)` — a BTC pool is deployed with `token_dispatcher = BTC_TOKEN`.
2. A delegator calls `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)`.  
   — The guard `BTC_TOKEN_ADDRESS != STRK_TOKEN_ADDRESS` evaluates to `true`; the call succeeds. [1](#0-0) 
3. Alternatively, after entering normally, the delegator calls `change_reward_address(STRK_TOKEN_ADDRESS)`.  
   — Same guard, same result: passes. [2](#0-1) 
4. The staker attests; `update_pool_rewards` sends STRK to the pool contract and calls `update_rewards_from_staking_contract`. [5](#0-4) 
5. The delegator (or anyone permitted) calls `claim_rewards`.  
   `claim_rewards` transfers STRK to `reward_address = STRK_TOKEN_ADDRESS`. [3](#0-2) 
6. The STRK tokens are now locked inside the STRK token contract with no retrieval mechanism. The delegator's yield is permanently burned.
