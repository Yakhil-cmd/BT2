### Title
Missing Slippage Protection in `enter_delegation_pool` / `add_to_delegation_pool` Allows Staker to Silently Increase Commission Before Delegation Executes - (File: `src/pool/pool.cairo`)

---

### Summary

`enter_delegation_pool` and `add_to_delegation_pool` in `src/pool/pool.cairo` accept no `max_commission` parameter. A staker who holds an active `CommissionCommitment` can legally increase their commission up to `max_commission` at any moment. A delegator who signs a transaction after observing a low commission may have that transaction execute against a materially higher commission, with no on-chain protection.

---

### Finding Description

**Entry points (no slippage guard):**

`enter_delegation_pool` and `add_to_delegation_pool` in `src/pool/pool.cairo` take only `reward_address` and `amount` (or `pool_member` and `amount`). Neither accepts a `max_commission` bound. [1](#0-0) 

**Commission increase path:**

`set_commission` in `src/staking/staking.cairo` calls `update_commission`. When an active `CommissionCommitment` exists, the only constraint is `commission <= max_commission` and `commission != old_commission`. This explicitly permits **increases** up to `max_commission` with no delay or notice period. [2](#0-1) 

The `CommissionCommitment` is set by the staker via `set_commission_commitment`, which only requires `current_commission <= max_commission`. A staker can therefore set `max_commission` far above the current rate at any time. [3](#0-2) 

The staking contract's own interface comment acknowledges this: *"Current commission increase safeguards still allow for sudden commission changes."* [4](#0-3) 

**No commission check at delegation time:**

Neither `enter_delegation_pool` nor `add_to_delegation_pool` reads or validates the staker's current commission before accepting funds. [5](#0-4) 

The same gap exists in `switch_delegation_pool`, where a pool member moves stake to a new staker/pool without specifying an acceptable commission for the destination. [6](#0-5) 

---

### Impact Explanation

A delegator who delegates or switches pools expecting commission `C_low` may end up bound to commission `C_high` (up to `max_commission`). All future rewards earned while delegated are split at the higher rate, permanently reducing the delegator's yield for the duration of their delegation. This constitutes **griefing with damage to users** (Medium), and depending on the magnitude of the commission jump, can approach theft of unclaimed yield (High).

---

### Likelihood Explanation

**Medium.** The preconditions are:

1. The staker must have an active `CommissionCommitment` with `max_commission` materially above the current commission — a deliberate, observable on-chain action.
2. The staker must call `set_commission` to a higher value before the delegator's transaction is sequenced.

On Starknet there is no public mempool, so classic frontrunning is harder than on Ethereum. However, a malicious staker can advertise a low commission to attract delegators, then raise commission in the same or a preceding block. Because Starknet transaction ordering is controlled by the sequencer, a staker who can influence or predict sequencing (or simply races the delegator) can reliably exploit this. The `CommissionCommitment` mechanism was designed to protect delegators but paradoxically enables sudden increases up to `max_commission`.

---

### Recommendation

Add a `max_commission: Commission` parameter to `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool`. At the start of each function, read the staker's current commission from the staking contract and assert it does not exceed the caller-supplied bound:

```cairo
let current_commission = self.get_commission_from_staking_contract();
assert!(current_commission <= max_commission, Error::COMMISSION_EXCEEDS_MAX);
```

This mirrors the slippage-protection pattern recommended in the reference report: the user commits to an acceptable bound at signing time, and the transaction reverts if the on-chain state has moved outside that bound.

---

### Proof of Concept

1. Staker deploys a STRK pool, sets commission to **1%**, and calls `set_commission_commitment(max_commission: 5000, expiration_epoch: E+100)` (50% max).
2. Alice observes 1% commission, approves tokens, and signs `enter_delegation_pool(reward_address, 1_000_000_STRK)`.
3. Before Alice's transaction is sequenced, the staker calls `set_commission(5000)` (50%). This passes `update_commission` because `5000 <= max_commission` and `5000 != 100`. [7](#0-6) 
4. Alice's `enter_delegation_pool` executes. No commission check occurs; her funds are accepted. [8](#0-7) 
5. All future rewards for Alice are split at 50% commission instead of 1%. Alice loses ~49% of her expected yield with no recourse.

        fn add_to_delegation_pool(
            ref self: ContractState, pool_member: ContractAddress, amount: Amount,
        ) -> Amount {
            // Asserts.
            self.assert_staker_is_active();
            let pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            assert!(
                caller_address == pool_member || caller_address == pool_member_info.reward_address,
                "{}",
                Error::CALLER_CANNOT_ADD_TO_POOL,
            );
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);

            // Transfer funds from the delegator to the staking contract.
            let token_dispatcher = self.token_dispatcher.read();
            let staker_address = self.staker_address.read();
            transfer_from_delegator(pool_member: caller_address, :amount, :token_dispatcher);
            self.transfer_to_staking_contract(:amount, :token_dispatcher, :staker_address);

            // Update the pool member's balance checkpoint.
            let old_delegated_stake = self.increase_member_balance(:pool_member, :amount);
            let new_delegated_stake = old_delegated_stake + amount;

            // Emit events.
            self
                .emit(
                    Events::PoolMemberBalanceChanged {
                        pool_member, old_delegated_stake, new_delegated_stake,
                    },
                );

            new_delegated_stake
        }

                GenericError::MISSING_UNDELEGATE_INTENT,
            );
            assert!(amount <= pool_member_info.unpool_amount, "{}", GenericError::AMOUNT_TOO_HIGH);
            let reward_address = pool_member_info.reward_address;

            // Update pool_member_info and write to storage.
            pool_member_info.unpool_amount -= amount;
            if pool_member_info.unpool_amount.is_zero() {
                // unpool_amount is zero, clear unpool_time.
                pool_member_info.unpool_time = Option::None;

            self.write_pool_member_info(:pool_member, :pool_member_info);

                    :to_pool,

            let staker_address = get_caller_address();

                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
