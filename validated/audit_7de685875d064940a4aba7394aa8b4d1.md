### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Corrupted Signing Output - (File: src/frost/mod.rs)

### Summary
`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate only the upper bound of the `threshold` parameter (`threshold > participants.len()`), but omit the lower-bound check (`threshold < 2`) that is explicitly enforced in `assert_key_invariants` (`src/dkg.rs`). A malicious coordinator or misconfigured caller can pass `threshold = 1` (or `0`) to FROST signing, causing the Lagrange interpolation to use a degree-0 polynomial that does not match the degree-(threshold−1) polynomial used during key generation, producing an invalid and unusable signature.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` enforces only the upper bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
// ← no lower-bound check; threshold = 1 or 0 passes silently
``` [2](#0-1) 

The same omission exists in `presign`:

```rust
// src/frost/mod.rs lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
``` [3](#0-2) 

`ReconstructionLowerBound` itself is a plain `usize` wrapper with no invariant enforcement, so any value including 0 or 1 is accepted: [4](#0-3) 

### Impact Explanation

FROST signing uses the caller-supplied `threshold` to determine how many shares participate in Lagrange interpolation. If `threshold = 1` is supplied but the key was generated with `threshold = 2` (the minimum enforced by `assert_key_invariants`), the interpolation uses a degree-0 polynomial instead of the degree-1 polynomial that matches the secret sharing. The aggregated signature will fail to verify against the public key. All honest participants' signing contributions are wasted and the signing round produces an unusable cryptographic output.

**Matched impact:** *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation

`assert_sign_inputs` is a public API function. A malicious coordinator controlling the signing session can deliberately supply `threshold = 1` to corrupt every signing round without any cryptographic capability. A misconfigured integrator can trigger the same outcome accidentally. No privileged key material or external oracle is required.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces, in both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Alternatively, enforce the invariant inside `ReconstructionLowerBound::new` so it is impossible to construct a value below 2 at the type level, eliminating the need for repeated call-site checks.

### Proof of Concept

1. Run DKG with `threshold = 2`, `participants = [A, B, C]` — succeeds because `assert_key_invariants` enforces `threshold >= 2`.
2. Call `assert_sign_inputs(participants, /*threshold=*/1, me, coordinator)` — **returns `Ok`** because the lower-bound check is absent.
3. Proceed to FROST signing with `threshold = 1`; the coordinator aggregates shares using Lagrange coefficients for a single-point interpolation.
4. The resulting signature does not verify against the public key produced in step 1, because the interpolation polynomial degree (0) does not match the secret-sharing polynomial degree (1).
5. All honest participants have completed their signing round and cannot distinguish this failure from a network fault; the signing output is permanently corrupted for this session. [5](#0-4) [6](#0-5)
