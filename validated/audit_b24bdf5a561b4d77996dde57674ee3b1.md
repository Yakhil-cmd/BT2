### Title
Malicious Pool Server Can Substitute `targetPuzzleHash` Between Display and Submission, Redirecting All Farming Rewards to Attacker — (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

The pool-joining flow fetches pool info **twice** from the remote pool server: once for display (via `usePoolInfo` in `PlotNFTSelectBase`) and once at submission time (via `prepareSubmitData` → `getPoolInfo`). A malicious pool operator can serve a legitimate `targetPuzzleHash` for the display request and an attacker-controlled `targetPuzzleHash` for the submission request. The GUI has no mechanism to detect the substitution. The attacker-supplied hash is passed directly to `pw_join_pool` / `create_new_wallet`, permanently redirecting the victim's XCH farming rewards to the attacker's address.

---

### Finding Description

**Display path (check):**
`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)` which fetches `/pool_info` and renders the pool's name, fee, `relativeLockHeight`, and `targetPuzzleHash` for the user to review. [1](#0-0) 

**Submission path (use):**
When the user clicks "Create" / "Change", `handleSubmit` calls `prepareSubmitData`, which issues a **fresh, independent** HTTP request to the same pool URL:

```
const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
``` [2](#0-1) 

`getPoolInfo` delegates to `window.appAPI.fetchPoolInfo(poolUrl)` with no integrity check against the previously displayed data: [3](#0-2) 

The IPC handler in the Electron main process appends `/pool_info` and fetches without comparing to any prior response: [4](#0-3) 

The `targetPuzzleHash` from this second fetch is placed directly into `initialTargetState` and forwarded to `createNewPoolWallet` or `pwJoinPool`: [5](#0-4) [6](#0-5) [7](#0-6) 

In Chia's pooling protocol, `targetPuzzleHash` is the on-chain destination for all block rewards earned by the Plot NFT singleton. Whoever controls that puzzle hash receives the XCH.

---

### Impact Explanation

**Critical — Unauthorized payout change affecting XCH farming rewards.**

A malicious pool operator substitutes their own puzzle hash at submission time. The resulting on-chain transaction permanently locks the Plot NFT's reward destination to the attacker's address. Every block reward earned by the victim's plots is paid to the attacker for the entire lifetime of the Plot NFT (or until the user pays the double-fee to leave and rejoin a legitimate pool — if they ever discover the fraud).

---

### Likelihood Explanation

- The attacker only needs to operate a pool server (no privileged access required).
- The attack is invisible to the user: the displayed pool details (name, fee, `targetPuzzleHash`) are legitimate; only the submission-time response is malicious.
- The user has no UI indication that a second fetch occurred or that its result differs from what was shown.
- The attack works for both new Plot NFT creation (`PlotNFTAdd`) and pool changes (`PlotNFTChangePool`).

---

### Recommendation

Do not re-fetch pool info at submission time. Instead, pass the already-fetched `PoolInfo` object (obtained by `usePoolInfo` during the display phase) into `prepareSubmitData` directly, eliminating the second network round-trip and the TOCTOU window. If a second fetch is retained for any reason, compare `targetPuzzleHash` and `relativeLockHeight` byte-for-byte against the displayed values and abort with an error if they differ.

---

### Proof of Concept

1. Attacker operates a pool server at `https://evil-pool.example`.
2. For all requests to `/pool_info` that include a `User-Agent` or timing pattern matching the display fetch, the server returns:
   ```json
   { "target_puzzle_hash": "0xLEGITIMATE_POOL_HASH...", "relative_lock_height": 32, ... }
   ```
3. For the submission-time fetch (which arrives a few seconds later, after the user clicks "Create"), the server returns:
   ```json
   { "target_puzzle_hash": "0xATTACKER_CONTROLLED_HASH...", "relative_lock_height": 32, ... }
   ```
4. The user sees the legitimate hash in the UI. `prepareSubmitData` uses the attacker hash.
5. `createNewPoolWallet` is called with `targetPuzzleHash = "0xATTACKER_CONTROLLED_HASH"`.
6. All subsequent farming rewards from the victim's plots are paid to the attacker's address.
