### Title
Gateway Stateful Admission Uses Previous-Block L2 Gas Price as Threshold, Causing Systematic False Rejections and False Admissions When EIP-1559 Price Drifts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `validate_resource_bounds` function checks a transaction's `max_price_per_unit` against a threshold derived from the **previous block's** L2 gas price. However, the block actually being built uses the **next block's** L2 gas price, computed by the EIP-1559 fee market (`calculate_next_l2_gas_price_for_fin`). With `gas_price_max_change_denominator = 48`, the price can shift up to ~9.5% per block. This creates two systematic admission errors: valid transactions are rejected when the price drops, and transactions that cannot pay the actual block price are admitted when the price rises.

### Finding Description

In `validate_resource_bounds`, the gateway fetches the **previous** block's L2 gas price and uses it as the admission reference:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is then computed as:

```rust
let threshold = (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer();
if tx_l2_gas_price.0 < threshold { return Err(...); }
``` [2](#0-1) 

The default `min_gas_price_percentage` is `100`, meaning the threshold equals the previous block's price exactly: [3](#0-2) 

Meanwhile, the actual block being built uses the **next** L2 gas price, computed by the EIP-1559 formula in `calculate_next_base_gas_price`:

```rust
let denominator = gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
let price_change = (price_u256 * gas_delta) / denominator;
``` [4](#0-3) 

With `gas_price_max_change_denominator = 48` across all versioned constants: [5](#0-4) 

And `gas_target = 1,040,000,000`, `max_block_size = 5,800,000,000` (v0.14.4), the maximum per-block price change is:
- **Downward** (empty block): `price / 48 ≈ 2.08%`
- **Upward** (full block): `price × 4,760,000,000 / (1,040,000,000 × 48) ≈ 9.54%`

The `run_validate_entry_point` function also uses the previous block's gas prices (only the block number is incremented), so the blockifier pre-validation inside the gateway also passes using the stale price:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// gas_prices remain from the previous block
``` [6](#0-5) 

The batcher, however, passes the actual next-block gas prices to the blockifier during block building: [7](#0-6) 

The TODO comment in the code explicitly acknowledges the wrong price is being used: [8](#0-7) 

### Impact Explanation

Two broken admission invariants result:

**False rejection (High impact — "rejects valid transactions before sequencing"):** When the previous block is under-utilized, the next block's price drops by up to ~2.08%. Any transaction with `max_price_per_unit` in the range `[next_price, prev_price)` is valid for the next block but is rejected by the gateway. With `min_gas_price_percentage = 100` (default), the rejection band is the full price-drop delta — up to ~2.08% of the current price every block.

**False admission (High impact — "accepts invalid transactions before sequencing"):** When the previous block is congested, the next block's price rises by up to ~9.54%. Transactions with `max_price_per_unit` in the range `[prev_price, next_price)` pass the gateway check but fail during block building with `MaxGasPriceTooLow`. These transactions consume mempool slots and block-building resources without ever being includable.

### Likelihood Explanation

The EIP-1559 fee market adjusts the L2 gas price every block based on utilization. Any deviation from the target utilization (which is the normal operating condition) causes the next block's price to differ from the previous block's price. Both failure modes therefore occur continuously in production, not just under adversarial conditions. The magnitude is bounded per block (~2% down, ~9.5% up) but is compounded across consecutive blocks of consistent utilization.

### Recommendation

Replace `previous_block_l2_gas_price` with the computed next-block L2 gas price. The fee market calculation is already available via `calculate_next_l2_gas_price_for_fin` in `crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`. The gateway should call this function (or read `next_l2_gas_price` from the block header once it is stored there, as the TODO suggests) to derive the correct threshold. This is the exact fix the TODO comment points to.

### Proof of Concept

**False rejection (price drops ~2%):**

1. Previous block L2 gas price: `P = 1,000,000,000 FRI`; block was empty (`gas_used = 0`).
2. EIP-1559 computes next price: `P' = P - P/48 = 979,166,666 FRI`.
3. Gateway threshold (`min_gas_price_percentage = 100`): `1,000,000,000 FRI`.
4. User submits transaction with `max_price_per_unit = 990,000,000 FRI`.
5. Gateway calls `validate_tx_l2_gas_price_within_threshold` → `990,000,000 < 1,000,000,000` → **rejected**.
6. The transaction would have been valid for the next block: `990,000,000 > 979,166,666`.

**False admission (price spikes ~9.5%):**

1. Previous block L2 gas price: `P = 1,000,000,000 FRI`; block was full (`gas_used = 5,800,000,000`).
2. EIP-1559 computes next price: `P' ≈ 1,095,400,000 FRI`.
3. Gateway threshold: `1,000,000,000 FRI`.
4. User submits transaction with `max_price_per_unit = 1,000,000,000 FRI`.
5. Gateway admits the transaction: `1,000,000,000 >= 1,000,000,000` → **admitted to mempool**.
6. During block building, blockifier checks against `P' = 1,095,400,000 FRI` → `1,000,000,000 < 1,095,400,000` → transaction fails with `MaxGasPriceTooLow` and is dropped.
