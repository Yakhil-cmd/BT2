### Title
Sub-Minimum EVM Balance Permanently Locked Due to Minimum Withdrawal Check - (`fvm/evm/types/balance.go`, `fvm/evm/emulator/emulator.go`, `fvm/evm/impl/impl.go`)

### Summary
The `AttoFlowBalanceIsValidForFlowVault` check enforced on every EVM-to-Cadence withdrawal rejects any amount below `1e10` attoFlow. Because EVM gas fees can reduce a COA or EOA balance to an arbitrary sub-threshold value, any FLOW tokens remaining below that threshold are permanently locked in the EVM state with no recovery path.

### Finding Description
Flow EVM stores balances in atto-flow (1e-18 FLOW), while Cadence `FlowToken.Vault` uses `UFix64` (1e-8 FLOW precision). The conversion multiplier is `UFixToAttoConversionMultiplier = 10^10`. To prevent rounding loss, every withdrawal call enforces:

```go
// fvm/evm/types/balance.go:105-107
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
```

This guard is applied unconditionally to the **requested withdrawal amount** in two places:

1. `fvm/evm/emulator/emulator.go` line 441 — the low-level `withdrawFrom` direct-call path:
```go
if !types.AttoFlowBalanceIsValidForFlowVault(call.Value) {
    return types.NewInvalidResult(..., types.ErrWithdrawBalanceRounding), nil
}
```

2. `fvm/evm/impl/impl.go` line 779 — the Cadence host-function path (`InternalEVM.withdraw`):
```go
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
```

Both paths reject any withdrawal of an amount `< 1e10` attoFlow. Consequently, if an EVM account's **entire balance** is `< 1e10` attoFlow, no withdrawal of any size is possible, and the funds are irrecoverable.

EVM gas fees are deducted in attoFlow at arbitrary granularity. A COA that executes EVM calls (e.g., `EVM.CadenceOwnedAccount.call(...)`) or runs EVM transactions via `EVM.run(...)` will have gas costs subtracted from its balance at the EVM level. If the post-fee balance falls below `1e10` attoFlow, the owner can never bridge those tokens back to Cadence. Similarly, any EVM-level transfer of `< 1e10` attoFlow into a COA (e.g., from a Solidity contract paying out a small reward) leaves a permanently unrecoverable residue.

### Impact Explanation
FLOW tokens held in the EVM state become permanently locked whenever an account balance drops below `1e10` attoFlow (`0.00000001 FLOW`). There is no sweep, rescue, or alternative withdrawal path in the protocol. The tokens exist in the EVM state trie but can never be returned to a Cadence vault. This is a direct cross-VM asset loss: FLOW bridged from Cadence into EVM cannot be fully bridged back.

### Likelihood Explanation
The scenario is reachable by any unprivileged user who:
- Creates a COA and deposits a small amount of FLOW (e.g., exactly `2e10` attoFlow).
- Executes an EVM call or transaction whose gas cost leaves a remainder `< 1e10` attoFlow.

Gas prices in Flow EVM are set by the transaction sender and can be arbitrarily high. A simple EVM transfer costs 21,000 gas; at a gas price of `5e5` attoFlow/gas the fee is `1.05e10` attoFlow, which can easily leave a sub-threshold residue. Additionally, any Solidity contract that sends a small FLOW amount (e.g., a dust payment) to a COA address creates the same condition. No special privilege is required.

### Recommendation
Apply the minimum-amount check only when the requested withdrawal amount is **less than the account's full balance**. When the caller is withdrawing the entire remaining balance and that balance is `< 1e10` attoFlow, either:
- Allow the withdrawal to round down to zero and return an empty vault (treating the dust as a known precision loss), or
- Provide a dedicated "sweep dust" path that burns or redistributes sub-threshold balances rather than locking them.

The analogous fix in the referenced report was to skip the minimum-amount check when the owner is withdrawing their entire remaining balance.

### Proof of Concept
1. A user creates a COA and deposits exactly `2e10` attoFlow (`0.00000002 FLOW`).
2. The user calls `cadenceOwnedAccount.call(...)` targeting any EVM contract with a gas limit of `21000` and a gas price of `6e5` attoFlow/gas. Gas cost = `21000 × 6e5 = 1.26e10` attoFlow.
3. Remaining COA balance = `2e10 − 1.26e10 = 0.74e10 = 7.4e9` attoFlow.
4. The user attempts `cadenceOwnedAccount.withdraw(balance: EVM.Balance(attoflow: 7400000000))`.
5. `newInternalEVMTypeWithdrawFunction` in `fvm/evm/impl/impl.go` line 779 evaluates `AttoFlowBalanceIsValidForFlowVault(7.4e9)` → `false` → panics with `ErrWithdrawBalanceRounding`.
6. The `7.4e9` attoFlow remain in the EVM state trie permanently; no Cadence vault can ever receive them.

        /// Withdraws the balance from the cadence owned account's balance.
        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
            emit FLOWTokensWithdrawn(
                address: self.address().toString(),
                amount: balance.inFLOW(),
                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
