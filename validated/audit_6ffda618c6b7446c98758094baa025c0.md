### Title
Gateway Skips `__validate__` for Invoke Nonce-1 Transactions Based on Overly Broad Mempool Presence Check, Admitting Unsigned Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the account's `__validate__` entry point for any invoke transaction with `nonce == 1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The check is intended to confirm a pending `deploy_account` transaction, but it returns `true` for **any** transaction type from that address. An attacker can therefore submit a forged (arbitrarily-signed) invoke transaction with `nonce = 1` for any account that has **any** pending transaction in the mempool while its on-chain nonce is still `0`, and the gateway will admit it without ever calling `__validate__`.

The comment in `skip_stateful_validations` states the invariant it relies on:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This invariant is **broken**: an account with on-chain nonce `0` can have a regular invoke transaction (nonce `0`) in the pool that passed its own `__validate__` check. That presence is unrelated to whether a `deploy_account` is pending. The check does not filter by transaction type.

### Impact Explanation

An attacker who observes that account address `A` has on-chain nonce `0` and any pending transaction in the mempool can submit an invoke transaction with `nonce = 1`, an arbitrary (forged) signature, and arbitrary calldata. The gateway's stateful validator will:

1. Confirm `account_nonce == 0` from state.
2. Confirm `account_tx_in_pool_or_recent_block(A) == true`.
3. Set `skip_validate = true`, bypassing `__validate__`.
4. Admit the transaction to the mempool.

The mempool accepts an invalid (unsigned) transaction. This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

The batcher re-executes with `ExecutionFlags::default()` (`validate: true`), so the forged transaction will be rejected at execution time and produce no committed state change. However, the mempool is polluted with attacker-controlled transactions, wasting batcher resources and potentially displacing legitimate transactions via capacity pressure.

### Likelihood Explanation

The preconditions are observable on-chain and in the public mempool:
- Account nonce `0` is visible via any RPC call.
- Pending transactions are visible in the mempool.

The most common trigger is the intended UX flow itself: any user who submits a `deploy_account` transaction creates a window during which an attacker can inject a forged nonce-1 invoke for that address. No privileged access is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **deploy_account** transaction is pending for the address. The mempool should expose a `deploy_account_in_pool(address)` query, or the gateway should inspect the transaction type of the pending transaction before deciding to skip validation.

Alternatively, restrict the skip to the case where the account contract does not yet exist on-chain (i.e., `get_class_hash_at(sender_address)` returns the zero class hash), which is the only case where `__validate__` genuinely cannot be called.

### Proof of Concept

1. Alice submits a `deploy_account` transaction for address `A` (or any invoke with nonce `0` from an existing account with on-chain nonce `0`). The transaction enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.

2. Attacker submits an `InvokeV3` transaction:
   - `sender_address = A`
   - `nonce = 1`
   - `signature = [0xdead, 0xbeef]` (arbitrary)
   - `calldata` = drain Alice's ERC-20 balance

3. Gateway calls `extract_state_nonce_and_run_validations`:
   - `account_nonce = 0` (from state)
   - `skip_stateful_validations` returns `true` because `nonce == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(A) == true`
   - `run_validate_entry_point` sets `validate = false`, returns `Ok(())`

4. Gateway calls `mempool_client.add_tx(...)`. The forged transaction is now in the mempool.

5. Batcher pulls the forged transaction and attempts execution with `validate = true`. `__validate__` is called, fails on the bad signature, and the transaction is reverted — but it consumed batcher resources and occupied a mempool slot.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
