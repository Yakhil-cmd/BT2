### Title
Stale Delegated Permissions Persist After `CronosAdmin` Change, Enabling Unauthorized Token Mapping Corruption — (`x/cronos/keeper/permissions.go`, `x/cronos/keeper/msg_server.go`)

### Summary
The `x/cronos` permission system allows `CronosAdmin` to delegate `CanChangeTokenMapping` and `CanTurnBridge` permissions to arbitrary addresses via `MsgUpdatePermissions`. When governance changes the `CronosAdmin` parameter, all previously-delegated per-address permissions remain active in the KV store. Because there is no reverse index of permission holders, the new admin cannot enumerate and revoke stale grants. A previously-authorized address retains the ability to call `UpdateTokenMapping` and corrupt denom/contract bindings without any current-admin approval.

### Finding Description
`SetPermissions` writes a bitmask keyed by `AdminToPermissionsKey(address)` into the KV store:

```go
func (k Keeper) SetPermissions(ctx sdk.Context, address sdk.AccAddress, permissions uint64) {
    store := ctx.KVStore(k.storeKey)
    permissionsBytes := sdk.Uint64ToBigEndian(permissions)
    store.Set(types.AdminToPermissionsKey(address), permissionsBytes)
}
``` [1](#0-0) 

`HasPermission` checks the current `CronosAdmin` for implicit full access, then falls through to the stored per-address bitmask:

```go
admin := k.GetParams(ctx).CronosAdmin
for _, account := range accounts {
    if admin == account.String() {
        return true
    }
    permission := k.GetPermissions(ctx, account)
    if permission&permissionsToCheck == permissionsToCheck {
        return true
    }
}
``` [2](#0-1) 

When `CronosAdmin` is updated via governance, the old admin loses its implicit authority, but **no code clears the per-address grants the old admin issued**. There is no iterator or reverse index over `AdminToPermissionsKey` entries, so the new admin has no mechanism to discover and revoke them.

`UpdateTokenMapping` enforces only the `HasPermission` check: [3](#0-2) 

### Impact Explanation
An address X that was legitimately granted `CanChangeTokenMapping` by the old admin retains that permission after the admin rotates. X can call `UpdateTokenMapping` to remap any IBC denom to an attacker-controlled contract address. Subsequent `MsgConvertVouchers` calls will mint CRC20 tokens to the malicious contract instead of the legitimate one, constituting unauthorized token minting and corruption of denom/contract bindings — a **High** impact per the allowed scope ("Corruption of token mappings, denom/contract binding… with direct security impact").

### Likelihood Explanation
Admin rotation via governance is a normal, expected protocol operation. Any address that was ever granted `CanChangeTokenMapping` by a prior admin becomes a permanently privileged actor unless the new admin manually revokes it — which is impossible without a full KV store scan, since no enumeration API exists for permission holders.

### Recommendation
On every `CronosAdmin` parameter change, iterate all `AdminToPermissionsKey` entries and delete them (or emit an event requiring explicit re-grant). Alternatively, scope stored permissions to a specific admin epoch so that a parameter change automatically invalidates all prior grants.

### Proof of Concept
1. Governance sets `CronosAdmin = AdminA`.
2. `AdminA` calls `MsgUpdatePermissions` granting `CanChangeTokenMapping` to address `X`.
3. Governance proposal passes, changing `CronosAdmin = AdminB`.
4. `AdminB` is unaware of `X`'s grant (no enumeration possible).
5. `X` calls `MsgUpdateTokenMapping`, remapping `ibc/ATOM` denom to a malicious CRC20 contract.
6. All subsequent `MsgConvertVouchers` for `ibc/ATOM` mint tokens to the attacker's contract — unauthorized asset redirection with no current-admin approval.
