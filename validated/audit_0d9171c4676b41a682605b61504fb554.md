### Title
Unprotected `new()` Initializer Allows Anyone to Frontrun Engine Initialization and Seize Ownership — (`engine/src/contract_methods/admin.rs`)

### Summary
The Aurora Engine's `new()` function, which sets the contract's `owner_id` and all critical configuration, has no caller access control. Any NEAR account can call it before the legitimate deployer, installing themselves as owner. The owner controls contract upgrades, key management, and pause/resume — giving an attacker full control over all bridged assets.

### Finding Description
`admin::new()` is the NEAR contract initializer for the Aurora Engine. Its only guard is a check that state does not yet exist:

```rust
// engine/src/contract_methods/admin.rs, lines 56-59
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
```

There is no check on `env.predecessor_account_id()`. The function then deserializes attacker-supplied `NewCallArgs` and writes the resulting `EngineState` — including `owner_id` — directly to storage:

```rust
// lines 62-86
let args = NewCallArgs::deserialize(&input)...;
let state: EngineState = args.into();
// ...
state::set_state(&mut io, &state)?;
```

`NewCallArgs` (all versions V1–V4) include a caller-controlled `owner_id` field:

```rust
// engine-types/src/parameters/engine.rs, lines 107-108
pub owner_id: AccountId,
```

The WASM entrypoint in `engine/src/lib.rs` (lines 76–83) exposes this as a public `new()` function callable by any NEAR account.

### Impact Explanation
**Critical — Direct theft of all user funds.**

The `owner_id` stored by `new()` is the sole authorization gate for the most dangerous admin operations, including `upgrade()`:

```rust
// engine/src/contract_methods/admin.rs, lines 174-176
let state = state::get_state(&io)?;
require_running(&state)?;
require_owner_only(&state, &env.predecessor_account_id())?;
```

`upgrade()` issues a NEAR batch promise that deploys arbitrary WASM code to the Aurora Engine account itself. An attacker who controls `owner_id` can deploy a malicious contract that transfers all ETH and ERC-20 balances held by the engine to the attacker. The Aurora Engine on mainnet holds all assets bridged from Ethereum, making this a total-loss event for all users.

### Likelihood Explanation
**High.** In NEAR, contract deployment and initialization are two separate transactions unless the deployer explicitly uses a batch transaction. The window between `DeployContract` and the `new()` call is observable on-chain. A monitoring bot can detect the deployment transaction and immediately submit a `new()` call with a malicious `owner_id` in the same or next block. No special privileges, leaked keys, or social engineering are required — only the ability to submit a NEAR transaction.

### Recommendation
Restrict `new()` so it can only be called by the contract account itself (i.e., `env.current_account_id() == env.predecessor_account_id()`). In NEAR, a contract can only call itself via a batch transaction, which atomically combines deployment and initialization, eliminating the frontrunning window. Add the following check at the top of `new()`:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if env.current_account_id() != env.predecessor_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ...
}
```

This mirrors the pattern used by `set_owner`, `upgrade`, and other privileged functions that already call `require_owner_only`.

### Proof of Concept

1. Aurora team submits transaction T1: `DeployContract` (uploads WASM to `aurora` account).
2. Attacker observes T1 in the mempool / block and submits transaction T2 before T1's follow-up `new()` call:
   ```json
   {
     "receiver_id": "aurora",
     "actions": [{
       "type": "FunctionCall",
       "method_name": "new",
       "args": "<borsh(NewCallArgsV4 { chain_id: 1313161554, owner_id: 'attacker.near', ... })>",
       "gas": 30000000000000,
       "deposit": 0
     }]
   }
   ```
3. T2 succeeds: `state::get_state` returns `Err(NotFound)`, so the guard passes. `EngineState { owner_id: "attacker.near", ... }` is written to storage.
4. Aurora team's legitimate `new()` call fails with `ERR_ALREADY_INITIALIZED`.
5. Attacker calls `stage_upgrade()` then `upgrade()` (both gated only by `require_owner_only`) to deploy a malicious WASM contract.
6. The malicious contract drains all bridged ETH and ERC-20 token balances from the Aurora Engine account.
