### Title
Gateway L2 Gas Price Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Admitting Transactions That Fail Blockifier Execution — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current block's** `l2_gas_price` from the block header to compute the admission threshold, but the batcher executes the next block using `next_l2_gas_price` (the EIP-1559-derived price stored in the same header). When the L2 gas price is rising, transactions whose `max_price_per_unit` falls between the two values pass gateway admission and enter the mempool, then fail `check_fee_bounds` inside the batcher's blockifier execution. When the price is falling, the reverse holds: valid transactions are incorrectly rejected at the gateway.

---

### Finding Description

`validate_resource_bounds` fetches the block info and extracts the **current block's** STRK L2 gas price:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

The `GatewayFixedBlockSyncStateClient` builds `BlockInfo` from `block_header.l2_gas_price.price_in_fri` — the **current** block's price — and never reads `block_header.next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`BlockHeaderWithoutHash` carries **both** fields: `l2_gas_price` (the price used in the current block) and `next_l2_gas_price` (the EIP-1559-derived price for the **next** block): [3](#0-2) 

The consensus orchestrator writes `self.l2_gas_price` — the EIP-1559 result — into `next_l2_gas_price` when it commits a block:

```rust
next_l2_gas_price: self.l2_gas_price,
``` [4](#0-3) 

`self.l2_gas_price` is updated by `calculate_next_l2_gas_price_for_fin`, which applies the EIP-1559 formula to the current block's gas usage: [5](#0-4) 

When the batcher executes the next block it uses the gas prices supplied by the orchestrator (i.e., `next_l2_gas_price`). The blockifier's `check_fee_bounds` then enforces:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [6](#0-5) 

**The mismatch:**

| Stage | L2 gas price used |
|---|---|
| Gateway `validate_resource_bounds` | `block_header.l2_gas_price` (current block) |
| Gateway blockifier validation (`run_validate_entry_point`) | same `block_info.gas_prices` (current block) |
| Batcher blockifier execution | `next_l2_gas_price` (EIP-1559 result) |

When the network is busy and `next_l2_gas_price > l2_gas_price`, a transaction with:

```
min_gas_price_percentage% × l2_gas_price  ≤  tx.max_price_per_unit  <  next_l2_gas_price
```

passes both gateway checks but fails `check_fee_bounds` in the batcher with `InsufficientResourceBounds { MaxGasPriceTooLow }`.

The TODO comment in the source is a self-contained acknowledgement of the bug:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
``` [7](#0-6) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

- **Admission of invalid transactions**: Under rising gas prices (sustained high block utilization), transactions whose `max_price_per_unit` is between the stale gateway threshold and the actual next-block price are admitted to the mempool and forwarded to the batcher, where they fail `check_fee_bounds`. This wastes mempool slots, batcher CPU, and produces failed transaction receipts for users who believed their transaction was accepted.
- **Rejection of valid transactions**: Under falling gas prices, transactions that would be perfectly valid for the next block are rejected at the gateway because the threshold is computed from the higher stale price.

Both directions are reachable under normal EIP-1559 dynamics without any privileged access.

---

### Likelihood Explanation

**Medium.** The EIP-1559 formula (`calculate_next_base_gas_price`) adjusts the L2 gas price every block based on utilization relative to `gas_target`. During any sustained period of above-target or below-target block usage the divergence between `l2_gas_price` and `next_l2_gas_price` grows monotonically. No attacker action is required beyond submitting ordinary transactions; a busy network naturally creates the condition. The `min_gas_price_percentage` config parameter provides a partial buffer only for the admission direction, not for the rejection direction.

---

### Recommendation

In `validate_resource_bounds`, read `next_l2_gas_price` from the block header instead of `l2_gas_price`. The `GatewayFixedBlockStateReader` trait and its `GatewayFixedBlockSyncStateClient` implementation should expose `next_l2_gas_price` from `BlockHeaderWithoutHash` and pass it to `validate_tx_l2_gas_price_within_threshold`. This is exactly what the existing TODO comment requests. [8](#0-7) 

Additionally, `run_validate_entry_point` should build its `BlockContext` using `next_l2_gas_price` as the L2 gas price so that the gateway's blockifier validation matches the batcher's execution context. [9](#0-8) 

---

### Proof of Concept

1. Observe the latest committed block header: `l2_gas_price = P`, `next_l2_gas_price = P'` where `P' > P` (achievable by filling blocks above `gas_target`).
2. Compute the gateway threshold: `T = min_gas_price_percentage% × P`.
3. Submit an `InvokeV3` transaction with `l2_gas.max_price_per_unit = T` (satisfies `T ≥ T`, passes `validate_resource_bounds`) and `T < P'`.
4. The gateway's blockifier validation also passes because it builds `BlockContext` from the same stale `block_info` with `l2_gas_price = P`, so `check_fee_bounds` sees `T ≥ P` (if `T ≥ P`; otherwise set `T = P`).
5. The transaction enters the mempool.
6. When the batcher executes the next block with `l2_gas_price = P'`, `check_fee_bounds` fires `MaxGasPriceTooLow { max_gas_price: T, actual_gas_price: P' }` and the transaction fails — despite having been admitted by the gateway. [10](#0-9) [11](#0-10)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
