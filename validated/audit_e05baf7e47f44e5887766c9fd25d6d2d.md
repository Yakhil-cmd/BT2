### Title
Owner Can Front-Run `ft_on_transfer` by Atomically Replacing `erc20_fallback_address` to Steal Bridged ERC-20 Tokens - (File: `engine/src/engine.rs`)

### Summary

In Aurora Engine's Silo mode, when a NEP-141 token is bridged to a non-whitelisted EVM address via `ft_on_transfer`, the `receive_erc20_tokens` function reads `erc20_fallback_address` from storage **at execution time** and mints the tokens there instead. Because `set_erc20_fallback_address` (and `set_silo_params`) can be called by the owner at any time with no time-lock or delay, the owner can atomically replace the fallback address with their own controlled address immediately before a user's bridge transaction is processed. The user's tokens are then minted to the owner's address, constituting direct theft of bridged funds. This is the exact structural analog of the referral-code front-run in the external report: a privileged actor mutates a routing variable that is read at execution time, redirecting value away from the intended recipient.

### Finding Description

**Root cause — mutable routing state read at execution time with no delay protection.**

In `engine/src/engine.rs`, `receive_erc20_tokens` resolves the final mint recipient as follows:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

The value returned by `silo::get_erc20_fallback_address` is read directly from NEAR storage at the moment the receipt executes. There is no snapshot, no commit-reveal, and no time-lock.

The owner can update this value at any time via two separate entrypoints:

```rust
pub extern "C" fn set_erc20_fallback_address() {
    // ...
    require_owner_and_running(&state, &io.predecessor_account_id())...
    silo::set_erc20_fallback_address(&mut io, args.address);
}
``` [2](#0-1) 

```rust
pub extern "C" fn set_silo_params() {
    // ...
    require_owner_and_running(&state, &io.predecessor_account_id())...
    silo::set_silo_params(&mut io, args);

The underlying storage write has no guard beyond the owner check:

```rust
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }
}
``` [4](#0-3) 

**Exploit flow:**

1. Silo mode is active. The legitimate `erc20_fallback_address` is set to a treasury or burn address `T`.
2. User `U` calls `ft_transfer_call` on a NEP-141 token contract, specifying a non-whitelisted EVM address as the recipient. This schedules a `ft_on_transfer` callback on the Aurora engine.
3. The owner observes the pending NEAR receipt (NEAR receipts are publicly visible before they are applied).
4. The owner submits `set_erc20_fallback_address` (or `set_silo_params`) pointing to their own EVM address `A`. Because NEAR receipt ordering within a shard is deterministic and the owner controls when they submit, they can ensure this receipt executes before `U`'s `ft_on_transfer` receipt.
5. `U`'s `ft_on_transfer` receipt executes. `receive_erc20_tokens` reads the now-updated `erc20_fallback_address = A`, finds the recipient is not whitelisted, and mints the full token amount to `A`.
6. Optionally, the owner restores `erc20_fallback_address = T` to avoid detection.

The same attack works passively without any race: the owner can simply set `erc20_fallback_address` to their own address for a period of time, and every bridge transfer to any non-whitelisted address during that window is silently stolen.

### Impact Explanation

**Critical — Direct theft of user funds.**

Any user who bridges NEP-141 tokens to a non-whitelisted EVM address while the owner has substituted their own address as `erc20_fallback_address` loses the entire bridged amount. The tokens are minted to the owner's EVM address and are immediately withdrawable. There is no revert, no refund path, and no on-chain record distinguishing this from a legitimate fallback redirect. The loss is permanent and proportional to the bridged amount.

### Likelihood Explanation

**High.** The owner key is a single point of control with no time-lock, no multi-sig requirement enforced at the contract level, and no event that would alert users before the substitution takes effect. The attack requires no special tooling: a single NEAR transaction calling `set_erc20_fallback_address` is sufficient. In Silo deployments (enterprise or private chains), the owner is often a small set of operators, making this a realistic insider-threat vector. The passive variant (set-and-wait) requires no timing precision at all.

### Recommendation

1. **Immutable or time-locked fallback address.** Require a minimum delay (e.g., 24–48 h) between a `set_erc20_fallback_address` call and the new value taking effect, enforced on-chain.
2. **Two-step commit.** Separate the proposal and activation of a new fallback address into two transactions with a mandatory waiting period.
3. **Emit an event on change.** At minimum, log the old and new fallback address so off-chain monitors can alert users to suspend bridge transfers.
4. **Snapshot at deposit time.** Record the intended fallback address in the `ft_transfer_call` message itself and validate it matches the on-chain value at execution time, reverting if it has changed.

### Proof of Concept

```
// State before attack
erc20_fallback_address = T  (legitimate treasury)

// Step 1: User submits ft_transfer_call targeting non-whitelisted address R
// NEAR receipt R1: ft_on_transfer(sender=U, amount=1000, msg="<R_hex>") → Aurora

// Step 2: Owner front-runs with:
// NEAR receipt R0 (submitted before R1 is applied):
set_erc20_fallback_address(address = A)  // A = owner's EVM address

// Step 3: R0 executes → erc20_fallback_address = A

// Step 4: R1 executes → receive_erc20_tokens:
//   recipient = R  (from msg)
//   fallback_address = A  (read from storage NOW)
//   is_allow_receive_erc20_tokens(R) == false  (R not whitelisted)
//   → recipient = A
//   → mint(A, 1000)   ← user's 1000 tokens go to owner

// Step 5 (optional): Owner restores
set_erc20_fallback_address(address = T)
```

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
