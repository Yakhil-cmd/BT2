### Title
Gateway `skip_stateful_validations` Ignores `max_nonce_for_validation_skip` Config, Hardcoding Nonce == 1 and Incorrectly Rejecting Valid Invoke Transactions - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The Rust gateway's `skip_stateful_validations` function replicates the deploy-account UX skip logic from `PyValidator::should_run_stateful_validations` (the reference implementation in `native_blockifier`) but fails to use the `max_nonce_for_validation_skip` configuration field. It hardcodes `tx.nonce() == Nonce(Felt::ONE)` instead of checking the configured range `[1, max_nonce_for_validation_skip]`. The config field is defined, serialized, and documented in `StatefulTransactionValidatorConfig`, but is never read by the Rust gateway's validation path. When `max_nonce_for_validation_skip` is set to any value greater than 1, the gateway incorrectly requires blockifier validation for invoke transactions with nonces 2 through `max_nonce_for_validation_skip` on undeployed accounts, causing those valid transactions to be rejected at the gateway admission stage.

### Finding Description

**Reference implementation** (`crates/native_blockifier/src/py_validator.rs`, `should_run_stateful_validations`):

let nonce_small_enough_to_qualify_for_validation_skip =

    && is_post_deploy_nonce

### Impact Explanation

When `max_nonce_for_validation_skip` is configured to any value N > 1, the gateway will:

1. Accept a deploy_account + invoke(nonce=1) pair (skip works correctly for nonce == 1).
2. **Reject** invoke(nonce=2) through invoke(nonce=N) even though the reference implementation (`PyValidator`) would skip validation for all of them, because the account is not yet deployed.

The blockifier validation runs `StatefulValidator::validate` → `perform_pre_validation_stage` → `validate_tx` on an account whose contract class does not yet exist in state, causing a `ValidateFailure` error. The gateway returns this error to the caller, and the transaction is dropped before reaching the mempool.

This matches the allowed impact: **High — Mempool/gateway admission rejects valid transactions before sequencing.**

### Likelihood Explanation

The default production config sets `max_nonce_for_validation_skip = 0x1`, which happens to match the hardcoded value, masking the bug entirely under default settings. The bug activates only when an operator raises the config value above 1. The field is documented and exposed in the config schema, making it a plausible operational change. The `native_blockifier` path already supports values > 1 correctly, creating an expectation that the Rust gateway does too.

### Recommendation

Convert `skip_stateful_validations` from a free function to a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as an explicit parameter), and replace the hardcoded `Nonce(Felt::ONE)` equality check with the same three-condition logic used in `PyValidator::should_run_stateful_validations`:

```rust
// In run_pre_validation_checks, pass the config value:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    self.config.max_nonce_for_validation_skip,  // ← add this
    mempool_client.clone(),
).await?;

// In skip_stateful_validations:
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,       // ← new param
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        let tx_nonce = tx.nonce();
        if account_nonce == Nonce(Felt::ZERO)
            && Nonce(Felt::ONE) <= tx_nonce
            && tx_nonce <= max_nonce_for_validation_skip
        {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(...);
        }
    }
    Ok(false)
}
```

Add a test case for `nonce == 2` with `max_nonce_for_validation_skip = 2` to cover the previously untested range.

### Proof of Concept

1. Deploy a sequencer node with `gateway_config.static_config.stateful_tx_validator_config.max_nonce_for_validation_skip = 0x2`.
2. Generate a fresh account keypair (account not yet on-chain, nonce = 0).
3. Submit a `deploy_account` transaction (nonce = 0) to the gateway — accepted.
4. Submit an `invoke` transaction with nonce = 1 — accepted (skip fires, deploy_account found in mempool).
5. Submit an `invoke` transaction with nonce = 2 — **rejected** with `ValidateFailure` because `skip_stateful_validations` returns `false` (hardcoded `== Nonce(Felt::ONE)` fails), blockifier validation runs against the not-yet-deployed account, and the `__validate__` entry point cannot be found.
6. With the reference `PyValidator` logic, step 5 would skip validation and accept the transaction.
