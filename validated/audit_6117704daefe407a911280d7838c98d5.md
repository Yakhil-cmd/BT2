### Title
Unguarded `update_rewards` with `disable_rewards=true` Permanently Freezes Consensus-Phase Yield for All Stakers — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract has no caller access control. Any unprivileged address can invoke it with `disable_rewards = true`, which advances the global `last_reward_block` sentinel **before** the early-return guard is evaluated. This consumes the single per-block reward slot without distributing any yield, and blocks every subsequent call in that block. Repeated every block, this permanently freezes unclaimed consensus-phase rewards for all stakers.

---

### Finding Description

`update_rewards` is exposed as a public function under `StakingRewardsManagerImpl`. Its only gate is `general_prerequisites()`, which checks the pause flag and a non-zero caller — no role or identity check is performed. [1](#0-0) 

Inside the function, the global `last_reward_block` is written at line 1485, **before** the `disable_rewards` branch at line 1487: [2](#0-1) 

The per-block uniqueness guard at lines 1453–1458 then prevents any further call in the same block: [3](#0-2) 

Because `last_reward_block` is a **single global field** (not per-staker), the first successful call in any block exhausts the reward opportunity for every staker in that block: [4](#0-3) 

The attacker only needs to supply a valid, active staker address (trivially obtained from on-chain events). The staker's own balance is verified but the **caller** is never verified: [5](#0-4) 

---

### Impact Explanation

In the consensus-rewards phase (`is_pre_consensus()` returns `false`), `update_rewards` is the sole mechanism by which per-block STRK and BTC rewards are credited to stakers via `_update_rewards`. If the slot is consumed with `disable_rewards = true` every block, no staker ever accumulates new `unclaimed_rewards_own` and no pool ever receives forwarded rewards. This constitutes **permanent freezing of unclaimed yield** for all protocol participants. [6](#0-5) 

---

### Likelihood Explanation

The attack requires no stake, no role, and no privileged key — only a valid staker address (publicly observable) and the gas cost of one transaction per block. On Starknet, per-transaction fees are low. The attacker has no financial upside but can inflict sustained, protocol-wide yield denial at minimal cost, making this a realistic griefing scenario.

---

### Recommendation

1. **Add access control**: Restrict `update_rewards` to the authorized consensus sequencer address (analogous to how `update_rewards_from_attestation_contract` restricts to `attestation_contract`).
2. **Alternatively, reorder writes**: Move `self.last_reward_block.write(current_block_number)` to **after** the `if disable_rewards || self.is_pre_consensus() { return; }` guard, so a `disable_rewards = true` call does not consume the block slot.

---

### Proof of Concept

```
// Attacker (any EOA) repeats this every block:
staking_contract.update_rewards(
    staker_address = <any valid active staker>,  // obtained from NewStaker events
    disable_rewards = true
)
```

Step-by-step:

1. Block N begins. `last_reward_block` holds block N-1.
2. Attacker calls `update_rewards(valid_staker, disable_rewards=true)`.
3. `general_prerequisites()` passes (contract not paused, caller non-zero).
4. `current_block_number (N) > last_reward_block (N-1)` — assertion passes.
5. Staker existence and balance checks pass.
6. **`last_reward_block.write(N)` executes** — slot consumed.
7. `if disable_rewards { return; }` — function exits with zero rewards distributed.
8. Any legitimate call to `update_rewards` in block N now hits `REWARDS_ALREADY_UPDATED` and reverts.
9. Attacker repeats in block N+1, N+2, … indefinitely.

All stakers accumulate zero consensus-phase yield for the duration of the attack.
