### Title
Silo Whitelist Disabled by Default Allows Unauthorized EVM Transaction Submission and Contract Deployment Before Activation — (`engine/src/contract_methods/silo/whitelist.rs`)

---

### Summary

The Aurora Engine Silo access-control whitelist is **disabled by default** and is not activated during contract initialization. Before the owner explicitly calls `set_whitelist_status`, any unprivileged NEAR account can submit EVM transactions and deploy EVM contracts into a Silo instance, bypassing the intended access restrictions. This is the direct analog of the PrePO `_mintHook` uninitialized guard: the guard exists but is inert until a separate post-deployment call activates it, creating a frontrunnable window.

---

### Finding Description

`Whitelist::is_enabled()` in `engine/src/contract_methods/silo/whitelist.rs` reads a storage key and returns `false` when the key is absent — i.e., the default state after `new()` is called:

```rust
pub fn is_enabled(&self) -> bool {
    // White list is disabled by default. So return `false` if the key doesn't exist.
    let key = self.key(STATUS);
    self.io
        .read_storage(&key)
        .is_some_and(|value| value.to_vec() == [1])
}
```

Every access-check helper in `engine/src/contract_methods/silo/mod.rs` uses the short-circuit pattern `!list.is_enabled() || list.is_exist(account_id)`:

```rust
fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)   // true for everyone when disabled
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::EvmAdmin);
    !list.is_enabled() || list.is_exist(address)
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}

fn is_account_allowed<I: IO + Copy>(io: &I, account: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Account);
    !list.is_enabled() || list.is_exist(account)
}
```

`assert_access` in `engine/src/engine.rs` is the sole gate that enforces Silo restrictions on every EVM transaction:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
```

Because `is_allow_submit` and `is_allow_deploy` both return `true` when the whitelist is disabled, `assert_access` never rejects any caller during the window between `new()` and the first successful `set_whitelist_status` call.

The `new()` function in `engine/src/contract_methods/admin.rs` does not accept or set any whitelist configuration. The whitelist must be activated by a separate, post-deployment owner call. This is the same structural gap as the PrePO `_mintHook`: the guard is not wired up at construction time.

---

### Impact Explanation

**High — Temporary freezing of funds / Critical — Direct theft of user funds.**

In a Silo deployment the operator's intent is to restrict which NEAR accounts and EVM addresses may submit transactions or deploy contracts. During the window between `new()` and `set_whitelist_status(active: true)`:

1. Any NEAR account can call `submit` or `submit_with_args` and have `assert_access` pass unconditionally.
2. Any NEAR account can deploy arbitrary EVM bytecode via `deploy_code`.
3. If the Silo has already received bridged ERC-20 or ETH balances (via `ft_on_transfer`), an attacker who controls an EVM address with a balance — or who can interact with an already-deployed EVM contract holding funds — can drain those funds before the whitelist gate is raised.
4. Even if no funds are present at the moment of attack, the attacker can deploy a persistent malicious contract that will be callable after the whitelist is enabled (because the whitelist controls *submission*, not *existence* of already-deployed code), enabling future exploitation.

---

### Likelihood Explanation

**Medium-High.** NEAR transactions are publicly observable in real time. A Silo deployment follows a predictable multi-step sequence: `new()` → `set_eth_connector_contract_account()` → (bridge deposits) → `set_whitelist_status()`. An attacker monitoring the chain can detect the deployment, observe that the whitelist is still disabled, and submit transactions in any block before `set_whitelist_status` is confirmed. NEAR block times are ~1 second, so the window is narrow but reliably exploitable by an automated bot. The attack requires no privileged access and no leaked secrets.

---

### Recommendation

1. **Enable whitelists atomically at initialization.** Extend `NewCallArgs` to accept optional whitelist configuration (kinds + initial entries + enabled flags) and apply them inside `new()` before the state is committed. This eliminates the gap entirely.
2. **Alternatively**, add a dedicated `initialize_silo` method that atomically sets Silo params, enables all four whitelists, and seeds the initial allowed entries — and document that it must be called in the same block as `new()`.
3. **Minimum mitigation**: change the default of `is_enabled()` to `true` (deny-by-default) and require an explicit `set_whitelist_status(active: false)` to open access, inverting the current open-by-default posture.

---

### Proof of Concept

```
Block N:   Silo operator calls new() + set_eth_connector_contract_account()
           → whitelist storage key absent → is_enabled() == false for all four lists

Block N+1: Bridge relayer calls ft_on_transfer (predecessor == eth_connector)
           → engine.receive_base_tokens() mints ETH to operator's EVM address

Block N+2: Operator broadcasts set_whitelist_status(Admin, active=true) ...

Block N+2: Attacker sees the pending set_whitelist_status tx in the mempool.
           Attacker submits a higher-priority NEAR tx calling `submit` with a
           crafted EVM transaction that:
             - deploys a malicious EVM contract, OR
             - calls an existing EVM contract holding funds

           assert_access() evaluates:
             is_allow_submit(io, attacker_near_id, attacker_evm_addr)
             → is_address_allowed() → !Whitelist(Address).is_enabled() → true
             → is_account_allowed() → !Whitelist(Account).is_enabled() → true
             → returns true → no rejection

           Attacker's EVM transaction executes successfully before the whitelist
           is activated, achieving unauthorized EVM access.

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
