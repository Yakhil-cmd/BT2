### Title
Unsigned Invoke Transaction Admitted to Mempool via Overly Broad `skip_stateful_validations` Check - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for any invoke transaction with nonce=1 sent to an address that has **any** transaction in the mempool or a recent block. The check `account_tx_in_pool_or_recent_block` does not verify that the existing mempool transaction is specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` transaction in the mempool can inject an unsigned invoke transaction with nonce=1 for the victim's address, bypassing signature verification entirely at the gateway admission layer.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the "deploy_account + invoke" bundle pattern: when a user sends both transactions simultaneously, the invoke (nonce=1) should be admitted even though the account is not yet deployed on-chain. The guard condition is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` immediately without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [3](#0-2) 

The underlying `account_tx_in_pool_or_recent_block` implementation checks for the presence of **any** transaction for the address, regardless of type:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

The mempool's `validate_tx` (called before `skip_stateful_validations`) only checks nonce ordering and fee escalation — it does not verify signatures or transaction type:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

The nonce validation in the gateway allows nonce=1 when `account_nonce=0` as long as `max_allowed_nonce_gap >= 1` (which must be true in production for the UX feature to function):

```rust

    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
    return Err(create_error(...));

### Impact Explanation

An attacker can inject an invoke transaction with an arbitrary (invalid) signature for any address that has a pending `deploy_account` in the mempool. The transaction is admitted to the mempool without any signature check. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Secondary consequences:
- The attacker's unsigned nonce=1 tx occupies the `(address, nonce=1)` slot. The victim's legitimate nonce=1 invoke tx (sent as part of the deploy+invoke bundle) must pay a higher fee to replace it via fee escalation, or is rejected outright.
- The attacker can spam the mempool with invalid transactions for every address with a pending `deploy_account`, degrading mempool quality and wasting batcher execution resources (the batcher will call `__validate__` at execution time, which will fail and revert).

### Likelihood Explanation

- All pending `deploy_account` transactions are publicly visible in the mempool.
- The attacker needs only to know the victim's address (derivable from the public `deploy_account` transaction) and submit a well-formed invoke transaction with nonce=1 and any signature bytes.
- No privileged access, special keys, or on-chain state is required.
- The attack is trivially repeatable for every new `deploy_account` submission.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies the account has a **`deploy_account`** transaction in the mempool. For example, expose a `deploy_account_tx_in_pool(address)` query from the mempool, or store the transaction type alongside the address in the mempool state so `skip_stateful_validations` can confirm the pending transaction is actually a `deploy_account`.

Alternatively, mirror the stricter guard used in `PyValidator::should_run_stateful_validations`, which requires an explicit `deploy_account_tx_hash` parameter supplied by the caller rather than inferring it from pool membership:

```rust
// crates/native_blockifier/src/py_validator.rs:109-110
let deploy_account_not_processed =
    deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
``` [7](#0-6) 

### Proof of Concept

1. Victim broadcasts a `deploy_account` transaction for address `0xVICTIM` (nonce=0). It enters the mempool; `account_tx_in_pool_or_recent_block(0xVICTIM)` now returns `true`.

2. Attacker submits an `RpcInvokeTransaction::V3` with:
   - `sender_address = 0xVICTIM`
   - `nonce = 1`
   - `signature = [0x0, 0x0]` (invalid)
   - `resource_bounds` satisfying the stateless gas-price check

3. Gateway stateless validation passes (no signature check performed). [8](#0-7) 

4. Gateway stateful validation:
   - `validate_nonce`: `0 <= 1 <= max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: nonce=1 is not a duplicate, not too old → passes.
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(0xVICTIM)==true` → returns `true`.
   - `run_validate_entry_point`: `execution_flags.validate = false` → `__validate__` is **never called**. [9](#0-8) 

5. The attacker's unsigned invoke transaction is added to the mempool via `mempool_client.add_tx(...)`. The victim's legitimate nonce=1 invoke (if submitted afterward) must pay a higher fee to displace it or is rejected as a duplicate nonce. [4](#0-3)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
