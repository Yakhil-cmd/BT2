### Title
ETH Sent to `tick()` Is Permanently Frozen When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.sol`'s `tick()` function is `payable` and accepts ETH to cover L1→L2 message fees. However, when `amountToMint == 0` (no pending mint requests), the entire `msg.value` is silently consumed by the contract and never refunded. The contract has no `receive()`, `fallback()`, or `withdraw()` function, making any ETH sent in this scenario permanently irrecoverable.

---

### Finding Description

The `tick()` function in `RewardSupplier.sol` is declared `external payable` and uses `msg.value` to pay for two L1→L2 messages (a bridge deposit and a `sendMessageToL2` call to the minting curve). The ETH is only consumed inside the `if (amountToMint > 0)` branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        // ...
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // If amountToMint == 0, msg.value is silently discarded
}
```

When `amountToMint == 0`, the function returns without using or refunding `msg.value`. The ETH accumulates in `RewardSupplier.sol` with no mechanism to recover it. The contract inherits from `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl` — none of which provide ETH withdrawal functionality, and `RewardSupplier.sol` itself defines no `receive()`, `fallback()`, or `withdraw()` function. ---

### Impact Explanation

Any ETH sent to `tick()` when `requiredMinting()` returns `amountToMint == 0` is permanently frozen in `RewardSupplier.sol`. The caller (an external operator) loses their ETH with no recourse. Over time, as operators call `tick()` to check for pending requests, ETH accumulates and is irrecoverable. This matches the **Medium: Griefing with no profit motive but damage to users or protocol** impact category — the protocol's operational liveness depends on operators continuing to call `tick()`, and repeated ETH losses disincentivize this.

---

### Likelihood Explanation

`tick()` is `external` with no access control — any address can call it. An operator monitoring the system may call `tick()` with ETH preemptively (e.g., to ensure the message fee is covered), not knowing whether minting is currently needed. The `requiredMinting()` view function exists to check this, but there is no on-chain guard preventing a `tick()` call with ETH when `amountToMint == 0`. This is a realistic operational scenario. [2](#0-1) 

---

### Recommendation

Add a refund of unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ... existing logic ...
    }
    // Refund any unused ETH to the caller
    if (address(this).balance > 0) {
        (bool ok, ) = msg.sender.call{value: address(this).balance}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, revert if `msg.value > 0` and `amountToMint == 0`, so callers are not silently drained.

---

### Proof of Concept

1. The L2 `RewardSupplier` has sufficient `unclaimed_rewards` and has not yet triggered a new `MintRequest`. No pending L2→L1 messages exist.
2. An operator calls `tick{value: 0.01 ether}()` to proactively cover fees.
3. `requiredMinting()` returns `(0, 0)` because `l2ToL1Messages(msgHash) == 0`.
4. The `if (amountToMint > 0)` branch is skipped entirely.
5. The 0.01 ETH is now held by `RewardSupplier.sol`.
6. No `withdraw`, `receive`, or `fallback` function exists to recover it.
7. The ETH is permanently frozen. [3](#0-2) [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
