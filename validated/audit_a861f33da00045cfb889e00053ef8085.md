### Title
Pool Admin Front-Runs `setFeeCaps` to Lock Admin Fees Above the Newly Lowered Cap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolAdminFees` validates the new admin fee against `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` at call time. Because `setFeeCaps` has no timelock and both functions are immediately executable, a pool admin who observes a pending `setFeeCaps` transaction that lowers the admin cap can front-run it with `setPoolAdminFees`, locking in fees above the cap the protocol owner intended to enforce.

### Finding Description
`setFeeCaps` is an `onlyOwner` function that immediately writes new values to `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8`: [1](#0-0) 

`setPoolAdminFees` enforces the cap with a point-in-time check: [2](#0-1) 

If the pool admin's `setPoolAdminFees` transaction is mined before the owner's `setFeeCaps` transaction, the check passes against the old (higher) cap. After `setFeeCaps` executes, `poolFeeConfig[pool].adminSpreadFeeE6` (and the pool's live `spreadFeeE6`) can exceed the new `maxAdminSpreadFeeE6`. No subsequent automatic enforcement corrects this; the excess persists until the owner separately calls `setPoolProtocolFee`, which contains the only clamping logic: [3](#0-2) 

`setFeeCaps` itself only auto-clamps the global default protocol fee, not per-pool admin fee configs: [4](#0-3) 

### Impact Explanation
The pool's live `spreadFeeE6` (stored as an immutable-like value on the pool after `setPoolFees`) is the sum of protocol + admin components: [5](#0-4) 

Every swap executed against the pool pays this inflated spread fee. Traders receive worse execution than the protocol's cap was meant to guarantee. The admin captures the excess fee revenue. The broken invariant is `poolFeeConfig[pool].adminSpreadFeeE6 <= maxAdminSpreadFeeE6` post-`setFeeCaps`.

### Likelihood Explanation
- The pool admin is semi-trusted and has a direct financial incentive to maximize fee revenue.
- On Ethereum mainnet (a target chain), the mempool is public; a pool admin can trivially observe a pending `setFeeCaps` transaction and submit `setPoolAdminFees` with a higher gas price.
- No special permissions beyond the pool admin role are required.
- The attack window is a single block race, identical in structure to the ProofOfEmail front-run.

### Recommendation
1. **Enforce caps retroactively in `setFeeCaps`**: iterate over all pools (or accept a pool list) and clamp `poolFeeConfig[pool].adminSpreadFeeE6` / `adminNotionalFeeE8` to the new caps, then call `setPoolFees` on each affected pool.
2. **Alternatively, add a timelock to `setFeeCaps`**: a two-step propose/execute pattern (mirroring `proposePoolPriceProvider`) gives pool admins no actionable mempool signal before the cap change is final.
3. **Minimum fix**: after writing new caps in `setFeeCaps`, emit an event that monitoring bots can use to immediately trigger `setPoolProtocolFee` on all pools whose admin fees now exceed the new cap.

### Proof of Concept
```
State before:
  maxAdminSpreadFeeE6 = 200_000   (20%)
  poolFeeConfig[pool].adminSpreadFeeE6 = 50_000  (5%)

Step 1 — Owner broadcasts (not yet mined):
  factory.setFeeCaps(200_000, 50_000, ...)   // intends to lower admin cap to 5%

Step 2 — Pool admin observes pending tx, front-runs with higher gas:
  factory.setPoolAdminFees(pool, 150_000, 0)
  // check: 150_000 <= maxAdminSpreadFeeE6 (200_000) → passes
  // pool.spreadFeeE6 = protocolSpreadFeeE6 + 150_000

Step 3 — Owner's setFeeCaps mines:
  maxAdminSpreadFeeE6 = 50_000

Post-state:
  poolFeeConfig[pool].adminSpreadFeeE6 = 150_000  // exceeds new cap of 50_000
  pool.spreadFeeE6 = protocolSpreadFeeE6 + 150_000  // live fee used by every swap

Every subsequent swap pays 15% admin spread instead of the intended ≤5%.
The excess persists until the owner calls setPoolProtocolFee, which is the only
path that clamps admin fees to the current maxAdminSpreadFeeE6.
```
