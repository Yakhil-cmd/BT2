### Title
Storage Capacity Check Uses Maximum Possible Fees Instead of Actual Fees, Causing Valid Transactions to Fail - (`fvm/transactionInvoker.go`)

### Summary

In `normalExecution()`, the `maxTxFees` value computed from the transaction's **gas limit** (not actual execution effort) is passed to `CheckStorageLimits`. The storage capacity check then treats the payer's balance as `actual_balance - maxTxFees` when computing available storage. However, the subsequent `deductTransactionFees()` call uses the **actual** `computationUsed`, which is always ≤ the gas limit. This mismatch means the storage check applies a stricter constraint than what will actually be deducted, causing valid transactions to fail the storage check when they would have passed if the actual fees were used.

### Finding Description

In `fvm/transactionInvoker.go`, `normalExecution()` follows this sequence:

**Step 1 — Compute `maxTxFees` from gas limit:**

`CheckPayerBalanceAndReturnMaxFees` is called, which internally calls `CheckPayerBalanceAndGetMaxTxFees` with `maxExecutionEffort = txnState.TotalComputationLimit()` (the transaction's gas limit, not actual usage). The returned `maxFees` represents the maximum possible fee if the full gas limit is consumed. [1](#0-0) 

**Step 2 — Storage check uses `maxTxFees` as the fee deduction:**

`CheckStorageLimits` is called with `maxTxFees`, which is forwarded to `AccountsStorageCapacity`. The on-chain `FlowStorageFees` contract computes the payer's effective balance as `actual_balance - maxTxFees` to determine storage capacity. [2](#0-1) [3](#0-2) 

**Step 3 — Actual fee deduction uses `computationUsed` (actual effort):**

`deductTransactionFees()` uses `computationUsed` (actual execution effort, capped at the limit), which is always ≤ `TotalComputationLimit()`. [4](#0-3) 

**The discrepancy:** The storage check subtracts `maxTxFees` (gas-limit-based) from the payer's balance, but the actual deduction subtracts `computationUsed` (actual effort). Whenever `computationUsed < TotalComputationLimit()`, the storage check is stricter than necessary.

The code comment acknowledges the intent but does not acknowledge the over-conservatism: [5](#0-4) 

### Impact Explanation

A transaction that would succeed — where the payer has enough balance to cover actual fees and maintain storage capacity — can be rejected by the storage limit check because the check uses `maxTxFees` (gas-limit-based) instead of the actual fees that will be deducted. This is unintended protocol behavior: valid transactions are rejected. No funds are lost, but the transaction fails with `ErrCodeStorageCapacityExceeded` even though the payer's post-deduction balance would have been sufficient.

### Likelihood Explanation

Any unprivileged transaction sender can trigger this by submitting a transaction where:
1. The gas limit is set significantly higher than actual execution effort (common practice — users set conservative gas limits)
2. The payer's balance is in the range: sufficient to cover `actual_fees + storage_minimum`, but insufficient to cover `max_fees + storage_minimum`

This is a realistic scenario for any payer whose balance is near the storage capacity boundary and who submits transactions with conservative gas limits.

### Recommendation

Use the actual `computationUsed` (or a tighter upper bound) instead of `TotalComputationLimit()` when computing `maxTxFees` for the storage capacity check. Alternatively, perform the storage check after fee deduction, or pass the actual computation used at the time of the storage check rather than the pre-execution maximum.

### Proof of Concept

Concrete scenario (illustrative values):
- Gas limit: 1,000 units; fee weight: 1 token/unit → `maxTxFees = 1,000 tokens`
- Actual computation used: 100 units → `actualFees = 100 tokens`
- Payer balance: 600 tokens; storage minimum reservation: 500 tokens

Storage check: effective balance = `600 - 1,000 = -400` → capacity below minimum → **storage check fails** → transaction rejected.

Actual deduction: effective balance = `600 - 100 = 500` → capacity exactly at minimum → **would have passed**.

The transaction is rejected by `CheckStorageLimits` with `ErrCodeStorageCapacityExceeded` even though the payer's post-deduction balance would have been sufficient. [6](#0-5) [7](#0-6)
