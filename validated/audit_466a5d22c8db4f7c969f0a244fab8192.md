### Title
Unsigned Invoke Transaction Admitted via `skip_stateful_validations` Signature Bypass for Undeployed Accounts — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function completely bypasses the `__validate__` entry-point call (i.e., the account's signature check) for any Invoke transaction with `nonce == 1` sent to an address whose on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. The guard is too broad: it returns `true` whenever **any** transaction for the target address exists in the mempool, not specifically a `deploy_account`. An unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can immediately submit a malicious Invoke with `nonce = 1` for the victim's address, have the gateway admit it without any signature check, and front-run or replace the victim's legitimate first post-deployment transaction.

---

### Finding Description

`skip_stateful_validations` is the UX feature that allows a user to broadcast `deploy_account` + `invoke(nonce=1)` simultaneously, before the account exists on-chain. The gateway skips `__validate__` for the invoke because the account contract does not yet exist to run it against.

The skip condition is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The third guard delegates to:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction in the pool for that address — not specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch is circular: future-nonce transactions that passed validation were validated against an already-deployed account, which is not the case here.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the function returns `Ok(())` immediately, before the `__validate__` call:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 79-81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

The signature is therefore never checked at the gateway admission stage.

---

### Impact Explanation

An attacker who observes Alice's `deploy_account(nonce=0)` entering the mempool can immediately submit `Invoke(sender_address=Alice, nonce=1, calldata=<malicious>, tip=T+1)` to the gateway. All three skip conditions are satisfied:

| Condition | Value |
|---|---|
| `tx.nonce() == 1` | ✓ (attacker sets it) |
| `account_nonce == 0` | ✓ (Alice not yet deployed) |
| `account_tx_in_pool_or_recent_block(Alice)` | ✓ (Alice's `deploy_account` is in the pool) |

The gateway admits the attacker's transaction without any signature verification. If the attacker's tip exceeds Alice's, the mempool accepts it as a fee-escalation replacement, evicting Alice's legitimate `invoke(nonce=1)`. When the batcher executes the block:

1. Alice's `deploy_account` succeeds — her account is deployed and funded.
2. The attacker's `invoke(nonce=1)` executes with `validate: true` (batcher always validates), calls `__validate__`, fails signature check, reverts.
3. Alice's account is charged fees for the failed transaction; her intended first action is permanently blocked from that nonce slot.

The broken invariant is: **the gateway must not admit a transaction for an account without verifying the account's authorization (signature), regardless of deployment status.** This is the direct Sequencer analog of the BribeVault bug where `depositBribeERC20` accepted an arbitrary `briber` address as the `from` in `transferFrom` — here, the gateway accepts an arbitrary `sender_address` without the corresponding signature.

---

### Likelihood Explanation

The attack requires no special privilege. Any unprivileged actor can:
1. Watch the public mempool for `deploy_account` transactions (standard P2P gossip).
2. Craft an `Invoke(nonce=1)` for the victim's address with a higher tip.
3. Submit it to the gateway before the victim submits their own `invoke(nonce=1)`.

The race window is the time between the victim's `deploy_account` entering the mempool and the victim submitting their paired `invoke`. For users who submit both atomically (the intended UX), the window is small but non-zero due to network propagation. For users who submit them separately, the window is arbitrarily large.

---

### Recommendation

1. **Narrow the skip guard**: `account_tx_in_pool_or_recent_block` should be replaced with a check that specifically verifies a `deploy_account` transaction exists for the address in the mempool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query.

2. **Alternatively, defer validation to execution**: Do not skip `__validate__` at the gateway for invoke transactions targeting undeployed accounts. Instead, accept the UX cost of requiring the user to submit the invoke after the deploy_account is confirmed, or run a lightweight off-chain signature check against the expected public key embedded in the deploy constructor calldata.

3. **Use `max_nonce_for_validation_skip` consistently**: The config field `max_nonce_for_validation_skip` exists in `StatefulTransactionValidatorConfig` but is not used by `skip_stateful_validations` in the gateway (it is only used in `PyValidator`). The gateway hardcodes `nonce == 1`. This field should be wired into the gateway's skip logic to allow operators to tighten the window. [5](#0-4)

### Proof of Concept

```
Setup:
  Alice's address = A (derived from her deploy_account constructor args)
  Alice's on-chain nonce = 0 (account not yet deployed)

Step 1: Alice submits deploy_account(nonce=0, tip=10) + invoke(nonce=1, calldata=TRANSFER_FUNDS, tip=10)
        → Both enter the mempool. account_tx_in_pool_or_recent_block(A) = true.

Step 2: Attacker submits invoke(sender_address=A, nonce=1, calldata=DRAIN_FUNDS, tip=11)
        to the gateway.

Step 3: Gateway evaluates skip_stateful_validations:
          tx.nonce() == 1          → true
          account_nonce == 0       → true
          account_tx_in_pool(A)    → true (Alice's deploy_account is in pool)
        → skip_validate = true
        → run_validate_entry_point called with validate=false
        → __validate__ NOT called, no signature check performed
        → Transaction admitted to mempool.

Step 4: Mempool accepts attacker's invoke as fee-escalation replacement
        (tip 11 > 10), evicting Alice's invoke(nonce=1).

Step 5: Batcher builds block:
          execute deploy_account(A, nonce=0)  → Alice's account deployed, funded
          execute attacker's invoke(A, nonce=1, validate=true)
            → __validate__ called → signature mismatch → REVERT
            → Alice's account charged fees; DRAIN_FUNDS not executed
            → Alice's TRANSFER_FUNDS is gone from mempool

Result: Alice's account is deployed but her first intended action is blocked.
        Alice loses fees. Attacker spent only a higher tip.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
