### Title
Pool Admin Can Reset Price Provider Timelock Indefinitely by Re-Proposing, Bypassing the Oracle Rotation Delay — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` contains no guard against overwriting an existing pending proposal. Every call unconditionally overwrites `pendingPriceProvider[pool]` and resets `pendingPriceProviderExecuteAfter[pool]` to `block.timestamp + timelock`. A pool admin can call it repeatedly to keep the timelock from ever expiring, or — more critically — can silently substitute a different provider address after the community has already reviewed the original proposal, with no on-chain signal that the first proposal was cancelled.

---

### Finding Description

In `MetricOmmPoolFactory.proposePoolPriceProvider` (lines 474–491):

```solidity
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;          // unconditional overwrite
pendingPriceProviderExecuteAfter[pool] = executeAfter;  // timelock reset
emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
```

There is no check such as `if (pendingPriceProvider[pool] != address(0)) revert PendingProposalExists()`. The function simply overwrites whatever was pending. [1](#0-0) 

Contrast this with `cancelPoolAdminTransfer`, which has an explicit cancel path and emits a `PoolAdminTransferCancelled` event. No equivalent cancel function or guard exists for price provider proposals. [2](#0-1) [3](#0-2) 

The interface exposes no `cancelPoolPriceProviderProposal` function, confirming the omission is structural, not just missing from the implementation. [4](#0-3) 

---

### Impact Explanation

Two distinct harms follow from this:

**1. Indefinite timelock reset (oracle rotation DoS)**  
The admin proposes provider A, waits `timelock - 1 second`, then proposes provider A again. `executeAfter` is pushed forward by a full `timelock` period. Repeating this prevents `executePoolPriceProviderUpdate` from ever succeeding, permanently blocking oracle rotation for the pool.

**2. Silent provider substitution after community review (bad-price execution)**  
The admin proposes provider A (legitimate). The community reviews it and decides to remain in the pool. Just before the timelock expires, the admin calls `proposePoolPriceProvider(pool, providerB)` where `providerB` is a malicious oracle returning manipulated bid/ask prices. The only on-chain signal is a new `PoolPriceProviderChangeProposed` event for `providerB`; there is no cancellation event for `providerA`. After the new timelock elapses, `executePoolPriceProviderUpdate` installs `providerB`. Subsequent swaps execute against the manipulated oracle, causing direct loss of user principal. [5](#0-4) 

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted — trusted only within caps and timelocks. The timelock is the primary on-chain protection against a malicious pool admin rotating to a bad oracle. The bypass requires only repeated calls to a public function by the pool admin; no external conditions, flash loans, or token quirks are needed. The substitution attack is particularly realistic because the first proposal appears legitimate and the second proposal's event is indistinguishable from a normal re-proposal.

---

### Recommendation

Add a guard at the top of `proposePoolPriceProvider` that reverts if a pending proposal already exists:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PendingPriceProviderProposalExists();
```

Alternatively, if re-proposal is intentional (e.g., to correct a mistake), emit a distinct `PoolPriceProviderProposalCancelled` event for the overwritten proposal and preserve the original `executeAfter` (do not reset the clock). The admin transfer pattern — which has an explicit `cancelPoolAdminTransfer` — should be mirrored here.

```solidity

vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerA));
// pendingPriceProvider[pool] = providerA

// 2. Half the timelock elapses

// 3. Admin proposes provider B (silently overwrites A, resets clock)
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerB));
// pendingPriceProvider[pool] = providerB  (providerA silently gone, no cancel event)
// pendingPriceProviderExecuteAfter[pool] = T + timelock/2 + timelock  (reset!)

    factory.pendingPriceProviderExecuteAfter(pool),
    block.timestamp + timelock  // NOT T + timelock
);
assertEq(factory.pendingPriceProvider(pool), address(providerB));

// 5. executePoolPriceProviderUpdate now reverts — timelock not elapsed
vm.expectRevert(...PriceProviderTimelockNotElapsed...);
factory.executePoolPriceProviderUpdate(pool);
```
