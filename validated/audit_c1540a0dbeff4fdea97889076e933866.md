### Title
`uint64` Underflow in `ComputationRemaining` Bypasses EVM Batch Gas Limit — (`fvm/meter/computation_meter.go`)

---

### Summary

`ComputationMeter.ComputationRemaining()` performs an unchecked `uint64` subtraction. The guard `if remainingComputationUsage <= 0` is dead code for a `uint64` value and never fires on underflow. When `computationUsed > computationLimit`, the subtraction wraps to a near-`MaxUint64` value, which is then returned as the available EVM gas budget. `batchRun` in `fvm/evm/handler/handler.go` consumes this value directly to gate how many EVM transactions may be included in a batch, so the underflow silently removes the computation cap.

---

### Finding Description

In `fvm/meter/computation_meter.go`, `ComputationRemaining` is:

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }

    remainingComputationUsage := m.params.computationLimit - m.computationUsed  // ← uint64 subtraction
    if remainingComputationUsage <= 0 {                                          // ← DEAD CODE for uint64
        return 0
    }

    return remainingComputationUsage / w
}
```

`computationLimit` and `computationUsed` are both `uint64`. In Go, `uint64` arithmetic wraps silently; there is no panic. The condition `remainingComputationUsage <= 0` is a comparison of a `uint64` to `0`; it is only true when the value is exactly `0`. When `computationUsed > computationLimit`, the subtraction wraps to `(MaxUint64 - (computationUsed - computationLimit) + 1)`, a huge positive number, and the guard never fires.

`computationUsed` can exceed `computationLimit` because `MeterComputation` increments the counter *before* checking the limit:

```go
m.computationUsed += w * intensity          // mutates state first
if m.computationUsed > m.params.computationLimit {
    return errors.NewComputationLimitExceededError(...)
}
```

Additionally, `Merge` accumulates a child meter's `computationUsed` into the parent with no limit check at all:

```go
func (m *ComputationMeter) Merge(child ComputationMeter) {
    m.computationUsed = m.computationUsed + child.computationUsed
    ...
}
```

After a `Merge` that pushes the parent over its limit, the parent's `computationUsed > computationLimit` without any error having been returned. If `ComputationRemaining` is called in that window, it underflows.

`batchRun` in `fvm/evm/handler/handler.go` calls `ComputationRemaining` at the very start to gate the entire batch:

```go
remainingGasLimit := h.backend.ComputationRemaining(environment.ComputationKindEVMGasUsage)
...
for i, rlpEncodedTx := range rlpEncodedTxs {
    txGasLimit := tx.Gas()
    if remainingGasLimit < txGasLimit {
        return nil, types.ErrInsufficientComputation
    }
    remainingGasLimit -= txGasLimit
}
```

If `ComputationRemaining` returns a near-`MaxUint64` value due to underflow, the `remainingGasLimit < txGasLimit` check never triggers, and the entire batch is admitted regardless of the declared gas limits.

---

### Impact Explanation

An attacker who can cause `computationUsed > computationLimit` in the parent meter before `EVM.batchRun` is invoked can submit a batch of EVM transactions whose aggregate gas exceeds the FVM computation limit. The batch runs with metering disabled (`RunWithMeteringDisabled`), so the EVM transactions execute without the intended cap. This allows an unprivileged transaction sender to consume unbounded EVM execution resources within a single Flow transaction, bypassing the computation limit that protects Execution nodes from resource exhaustion.

---

### Likelihood Explanation

The trigger condition (`computationUsed > computationLimit` before `ComputationRemaining` is called) is reachable via the `Merge` path: a Cadence transaction that performs enough nested-transaction work to push the parent meter over its limit, then calls `EVM.batchRun`. The `Merge` function has no limit check, so the overflow is silent. The attacker controls the transaction script and can craft the sequence deliberately. The weight for `ComputationKindEVMGasUsage` is set in production (confirmed by the existing "Batch run evm gas overflow" integration test), so the underflow branch is live code, not a dead path.

---

### Recommendation

Replace the dead `<= 0` guard with an explicit pre-subtraction comparison:

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }
    if m.computationUsed >= m.params.computationLimit {
        return 0
    }
    return (m.params.computationLimit - m.computationUsed) / w
}
```

Additionally, add a limit check inside `Merge` so that merging a child that would push the parent over its limit is detected immediately rather than silently deferred.
