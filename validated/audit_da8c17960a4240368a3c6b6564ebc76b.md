### Title
WalletConnect `chia_setNFTDID` Approval Dialog Displays Raw DID String Without Profile Name Resolution — (`packages/gui/src/electron/commands/Commands.ts`)

### Summary

The WalletConnect approval dialog for `chia_setNFTDID` displays the `did` parameter as a raw string (hex or bech32m DID identifier) with no resolution to a human-readable profile name from the user's DID wallet list. A paired dApp can supply an attacker-controlled DID, and the user cannot distinguish it from their own DID in the approval prompt.

### Finding Description

**Entrypoint:** A paired dApp sends `chia_setNFTDID({ wallet_id, nft_coin_ids, did: attacker_did, fee })` via WalletConnect.

**Step 1 — Command registration.** `chia_setNFTDID` is registered in `DappCommands` by inheriting the params of `chia_wallet.nft_set_nft_did`. The `did` field is declared as `type: 'string'` with no `humanize` annotation and no special handler: [1](#0-0) 

**Step 2 — Param parsing.** `parseDappParams` converts the `did` value with `String(value)` and passes it through unchanged: [2](#0-1) 

**Step 3 — Humanization.** `humanizeDappCommand` calls `humanizeParams` → `humanizeParamValue`. For `type: 'string'` with no `humanize` field, the function returns `String(value)` — the raw DID string as supplied by the dApp: [3](#0-2) 

**Step 4 — No enrichment in `parseCommandDisplay`.** `parseCommandDisplay` only handles `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. For `chia_wallet.nft_set_nft_did` it returns `undefined` — no DID-to-name resolution occurs: [4](#0-3) 

**Step 5 — Dialog rendering.** The `Confirm` dialog renders the `rows` array verbatim. The user sees label `"DID"` with value equal to whatever raw string the dApp sent: [5](#0-4) 

**Step 6 — Execution.** If the user approves, `sendCommand('nft_set_nft_did', ...)` is called with the attacker-supplied `did` value, moving the NFT to the attacker's DID: [6](#0-5) 

**Contrast with `DIDProfileDropdown`.** The normal GUI flow resolves a DID to a wallet name using `useGetDIDsQuery()` and displays `wallet.name`. This lookup never happens in the WalletConnect approval path: [7](#0-6) 

### Impact Explanation

The user is shown a raw DID identifier (e.g., `did:chia:1abc...xyz` or a hex string) and asked "Are you sure you want to move this NFT to the specified profile?" They cannot determine from the dialog whether the DID belongs to one of their own profiles or to an attacker. If they approve, the NFT is irreversibly transferred to the attacker-controlled DID. This is a direct NFT asset loss.

This meets the **High** criterion: *"Corruption, spoofing, or unsafe trust of... WalletConnect state that causes a user to approve... the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation

- The dApp only needs to be paired (user has already connected it and granted `chia_setNFTDID` permission).
- No additional privileges are required.
- The attack is fully local-testable: pair a dApp, call `chia_setNFTDID` with an external DID, observe the raw string in the dialog.
- Users are accustomed to approving WalletConnect prompts from trusted dApps and are unlikely to manually verify a raw DID identifier.

### Recommendation

In `humanizeParamValue` or via a dedicated enrichment step in `humanizeDappCommand`/`parseCommandDisplay` for `chia_wallet.nft_set_nft_did`, resolve the `did` parameter against the user's DID wallet list (via `get_dids` RPC) and display the matched wallet name (or flag it as "External / Unknown DID" if not found). This mirrors what `DIDProfileDropdown` does in the normal GUI flow.

### Proof of Concept

1. Pair a dApp with the user's wallet, requesting `chia_setNFTDID` permission.
2. From the dApp, call:
   ```js
   await walletConnect.request({ method: 'chia_setNFTDID', params: {
     wallet_id: 1,
     nft_coin_ids: ['<victim_nft_coin_id>'],
     did: 'did:chia:1attacker000000000000000000000000000000000000000000000000',
     fee: 0
   }});
   ```
3. Observe the approval dialog: the "DID" row shows the raw attacker DID string with no profile name.
4. User clicks "Move" → NFT is transferred to the attacker's DID via `nft_set_nft_did`.

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
