### Title
Integer Truncation in Multi-NFT Royalty Display Understates WalletConnect Approval Amount - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

The `formatAmountWithRoyalties` function in the WalletConnect command display pipeline uses BigInt integer division to split the purchase amount equally across NFTs before computing per-NFT royalties. This produces a displayed `amountWithRoyalties` that is systematically and significantly lower than the amount the wallet daemon will actually deduct. A WalletConnect-connected dApp can exploit this by presenting a multi-NFT offer, causing the user to approve a transaction while seeing a materially understated total cost.

### Finding Description

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` computes the royalty-inclusive total shown in the WalletConnect confirmation dialog as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length); // ← integer truncation
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n, // ← integer truncation
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The function divides the total purchase amount equally among all NFTs (`amount / N`) before computing each NFT's royalty. In the Chia protocol, each NFT's royalty is calculated on the **full** trade amount, not a proportional share. This means the displayed royalty is understated by a factor of N (the number of NFTs in the offer).

**Concrete example** (confirmed by the existing test case):

- Offer: 2 NFTs, purchase price = 100,000,000 mojos (0.0001 XCH)
- NFT1 royalty: 500 bps (5%), NFT2 royalty: 10 bps (0.1%)

GUI calculation (wrong):
- `splitAmount = 100_000_000 / 2 = 50_000_000`
- NFT1 royalty: `50_000_000 * 500 / 10_000 = 2_500_000`
- NFT2 royalty: `50_000_000 * 10 / 10_000 = 50_000`
- **Displayed total: 102,550,000 mojos (0.00010255 XCH)**

Actual daemon calculation (correct):
- NFT1 royalty: `100_000_000 * 500 / 10_000 = 5_000_000`
- NFT2 royalty: `100_000_000 * 10 / 10_000 = 100_000`
- **Actual total: 105,100,000 mojos (0.0001051 XCH)**

The test suite itself encodes the wrong value as the expected output, confirming the bug is present and untested against the correct protocol behavior: [2](#0-1) 

The `amountWithRoyalties` field produced by this function is surfaced directly in the WalletConnect confirmation dialog via `walletDeltaToDisplay` → `withRoyaltyTotals`: [3](#0-2) 

The actual on-chain royalty deduction is computed by the Chia wallet daemon (via `nft_calculate_royalties`), which correctly applies each NFT's royalty to the full purchase price. The GUI display and the daemon's deduction are therefore inconsistent.

### Impact Explanation

A user approving a WalletConnect `take_offer` or `create_offer_for_ids` command involving multiple NFTs with royalties sees a materially understated total cost in the confirmation dialog. The actual transaction deducts significantly more than shown. The discrepancy grows linearly with the number of NFTs: for N NFTs, the displayed royalty is 1/N of the actual royalty. A user who would have declined the transaction at the true cost may approve it based on the false display.

This matches the High impact category: the WalletConnect confirmation state causes the user to approve the wrong amount.

### Likelihood Explanation

Any WalletConnect-connected dApp can issue a `take_offer` command referencing an offer with two or more NFTs that carry royalties. No special privilege is required. Multi-NFT offers are a supported and documented feature of the Chia offer system. The bug is triggered automatically by the display pipeline whenever `royaltyPercentages.length > 1`.

### Recommendation

Replace the `splitAmount` approximation with the correct per-NFT royalty calculation. Each NFT's royalty must be computed on the full `amount`, not on `amount / N`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Remove the `splitAmount` variable entirely. Also add a test asserting that the displayed `amountWithRoyalties` matches the sum of the full-amount per-NFT royalties, not the split-amount approximation.

### Proof of Concept

The existing test at `parseCommandDisplay.test.ts` line 251–334 demonstrates the bug: it asserts `amountWithRoyalties: '0.00010255'` for a 2-NFT offer with royalties of 500 bps and 10 bps on 100,000,000 mojos. The correct value is `'0.0001051'` (105,100,000 mojos). The test passes because it encodes the buggy output as the expected value. [4](#0-3) 

A dApp exploiting this would:
1. Construct or reference an offer containing ≥2 NFTs each with non-zero royalties.
2. Send `take_offer` via WalletConnect.
3. The GUI confirmation dialog shows `amountWithRoyalties` = `(full_price + full_price/N * sum(royalties))` instead of the correct `(full_price + full_price * sum(royalties))`.
4. The user approves, and the daemon deducts the correct (higher) amount.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
