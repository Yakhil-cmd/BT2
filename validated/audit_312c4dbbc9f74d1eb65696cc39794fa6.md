Looking at the external bug class — a factory-deployed entity retaining control that should belong to the intended owner — I need to find a nearcore analog where an authorization boundary is missing: specifically, where entity A can act on a resource that should only be controllable by entity B.

I traced `DeployGlobalContractAction` with `GlobalContractDeployMode::AccountId` through the full execution path.

**Key finding**: `action_deploy_global_contract` in `runtime/runtime/src/global_contracts.rs` never checks that `predecessor_id == receiver_id`. When `deploy_mode == AccountId`, the global contract identifier is derived from `account_id` (the receipt's `receiver_id`), not from the `predecessor_id`. This means any account can send a transaction targeting account B and overwrite the global contract stored under B's account-ID namespace.

Compare with `DeployContract`, `AddKey`, `Stake`, etc., which all require `predecessor_id == receiver_id` (enforced at receipt processing). `DeployGlobalContract` is absent from that list in both the spec and the runtime.

The validation path confirms no ownership check exists:
- `validate_deploy_global_contract_action` — only checks code size [1](#0-0) 
- `check_account_existence` — only checks the account exists [2](#0-1) 
- `action_deploy_global_contract` — no `predecessor_id` check, proceeds directly to `initiate_distribution` [3](#0-2) 
- `initiate_distribution` — identifier is `GlobalContractIdentifier::AccountId(account_id.clone())` where `account_id` is the receiver, not the caller [4](#0-3) 

The documentation explicitly states the `AccountId` mode is intended to let "the owner update the contract for all its users" [5](#0-4)  — but no ownership is enforced.

### Title
Unauthorized Global Contract Overwrite: Missing `predecessor_id == receiver_id` Guard in `DeployGlobalContract` with `AccountId` Mode — (`File: runtime/runtime/src/global_contracts.rs`)

### Summary
Any unprivileged account can overwrite another account's `AccountId`-mode global contract by sending a transaction targeting that account with `Action::DeployGlobalContract(deploy_mode: AccountId, code: <malicious>)`. The runtime charges storage to the victim's balance and installs the attacker's code under the victim's account-ID namespace, silently redirecting all downstream callers to execute attacker-controlled Wasm.

### Finding Description
`GlobalContractDeployMode::AccountId` stores the contract under `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(receiver_id) }`. The intent is that only the named account can publish or update code under its own identifier. However, `action_deploy_global_contract` receives `account_id = receipt.receiver_id()` and never verifies that `receipt.predecessor_id() == receipt.receiver_id()`.

Actions such as `DeployContract`, `AddKey`, `Stake`, `DeleteKey`, and `DeleteAccount` all carry an explicit `predecessor_id == receiver_id` requirement enforced at receipt-processing time. `DeployGlobalContract` carries no such guard.

Attack path:
1. Attacker (account `evil.near`) constructs a `SignedTransaction` with `signer_id = evil.near`, `receiver_id = victim.near`, `actions = [DeployGlobalContract { code: <malicious_wasm>, deploy_mode: AccountId }]`.
2. Transaction is valid: `evil.near` holds a `FullAccess` key on its own account; no rule forbids targeting a third-party receiver with this action.
3. Receipt arrives at `victim.near`'s shard. `action_deploy_global_contract` is called with `account_id = victim.near`.
4. Storage cost is deducted from `victim.near`'s balance (attacker pays only gas).
5. `initiate_distribution` creates a `GlobalContractDistributionReceipt` with `id = GlobalContractIdentifier::AccountId("victim.near")` and the attacker's code, propagating it to all shards.
6. `apply_distribution_current_shard` writes the malicious Wasm to `TrieKey::GlobalContractCode { identifier: AccountId("victim.near") }` on every shard, overwriting the legitimate contract.

The nonce mechanism (`increment_nonce` / `check_and_update_nonce`) only prevents *stale* re-delivery of the same distribution wave; it does not prevent a fresh, higher-nonce overwrite by an unauthorized party.

### Impact Explanation
Every account that previously called `UseGlobalContract { contract_identifier: AccountId("victim.near") }` now executes the attacker's Wasm on any subsequent function call. The attacker can drain funds, corrupt storage, or exfiltrate data from all such accounts. The victim's own balance is also reduced by the storage cost. The corrupted trie key is `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(victim_account_id) }` on every shard.

### Likelihood Explanation
The attack requires only a funded account and a standard signed transaction — no validator, relayer, or operator privilege. The `AccountId` deploy mode is the primary mechanism for mutable global contracts (the `CodeHash` mode is immutable by design), so any production deployment using it is exposed. The attacker pays only gas; the victim pays storage.

### Recommendation
Add a `predecessor_id == receiver_id` check inside `action_deploy_global_contract` when `deploy_mode == AccountId`, mirroring the guard already applied to `DeployContract`, `AddKey`, and similar privileged actions:

```rust
if matches!(deploy_contract.deploy_mode, GlobalContractDeployMode::AccountId)
    && receipt.predecessor_id() != account_id
{
    result.result = Err(ActionErrorKind::ActorNoPermission {
        account_id: account_id.clone(),
        actor_id: receipt.predecessor_id().clone(),
    }.into());
    return Ok(());
}
```

Alternatively, enforce this at the receipt-validation layer alongside the existing `predecessor_id == receiver_id` checks for other privileged actions.

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}

fn initiate_distribution(
    state_update: &mut TrieUpdate,
    account_id: AccountId,
    contract_code: Arc<[u8]>,
    deploy_mode: &GlobalContractDeployMode,
    current_shard_id: ShardId,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))
        }
        GlobalContractDeployMode::AccountId => {
            GlobalContractIdentifier::AccountId(account_id.clone())
        }
    };
    // Increment the nonce and write it to state immediately to prevent multiple
    // distributions with the same nonce from being initiated. This requires
    // allowing the same nonce in the freshness check when applying the
    // distribution receipt.
    let nonce = increment_nonce(state_update, &id)?;
    let distribution_receipt =
        GlobalContractDistributionReceipt::new(id, current_shard_id, vec![], contract_code, nonce);
    let distribution_receipts =
        Receipt::new_global_contract_distribution(account_id, distribution_receipt);
    // No need to set receipt_id here, it will be generated as part of apply_action_receipt
    result.new_receipts.push(distribution_receipts);
    Ok(())
}
