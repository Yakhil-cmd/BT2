### Title
Missing Lower-Bound Validation on `threshold` in FROST Presign/Sign Creates Parameter Bounds Mismatch with DKG - (File: `src/frost/mod.rs`)

---

### Summary

The DKG phase enforces `threshold >= 2` as a hard lower bound, but the FROST presign and sign phases omit this lower-bound check entirely. A caller can supply `threshold = 1` to `presign` or `assert_sign_inputs` and pass all validation, even though no valid keygen output can ever carry `threshold = 1`. This is a direct analog to H02: one layer enforces an inclusive bound, the other enforces only an exclusive upper bound, leaving an edge-case value that is accepted by the signing layer but was structurally rejected by the key-generation layer.

---

### Finding Description

**DKG lower-bound enforcement** — `assert_key_invariants` in `src/dkg.rs`:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

This means no `KeygenOutput` can ever be produced with `threshold = 1`.

**FROST `presign` — no lower-bound check** (`src/frost/mod.rs`):

```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: threshold >= 2 is absent
``` [2](#0-1) 

**FROST `assert_sign_inputs` — no lower-bound check** (`src/frost/mod.rs`):

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: threshold >= 2 is absent
``` [3](#0-2) 

By contrast, `validate_triple_inputs` for OT-based ECDSA triple generation **does** replicate the lower-bound check:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
``` [4](#0-3) 

This makes the omission in the FROST path a targeted gap, not a universal oversight.

**The mismatch table:**

| Phase | Lower-bound enforced? | Accepts `threshold = 1`? |
|---|---|---|
| `assert_key_invariants` (DKG) | Yes (`>= 2`) | No |
| `validate_triple_inputs` (OT triple gen) | Yes (`>= 2`) | No |
| `presign` (FROST) | **No** | **Yes** |
| `assert_sign_inputs` (FROST sign) | **No** | **Yes** |

---

### Impact Explanation

When `threshold = 1` is supplied to the FROST signing path:

1. All validation in `presign` and `assert_sign_inputs` passes.
2. The FROST library proceeds to aggregate signature shares using Lagrange coefficients computed for a 1-of-n scheme (coefficient = 1 for the single selected participant).
3. The actual key shares were generated under a 2-of-n (or higher) polynomial; the correct Lagrange coefficient for reconstruction is not 1.
4. The aggregated signature `z = nonce_i + 1 · key_share_i · challenge` does not satisfy `z·G = R + challenge · public_key`, so the output signature is cryptographically invalid and unusable.

Honest participants complete the protocol and accept the output, but the resulting signature is permanently broken. This maps to:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.** [5](#0-4) 

---

### Likelihood Explanation

`presign` and `assert_sign_inputs` are public library entry points. Any library caller — including a malicious coordinator or a buggy integration — can supply `threshold = 1`. No privilege, leaked key, or cryptographic break is required. The caller only needs to deviate from the documented convention of passing the same threshold used at keygen time. Because the library provides no enforcement, this deviation is silently accepted.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use, in both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This synchronises the valid threshold range across all protocol phases, eliminating the edge-case gap. [6](#0-5) [7](#0-6)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
