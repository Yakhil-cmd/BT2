### Title
`DepositAllowlistExtension` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension.beforeAddLiquidity` is supposed to gate who may deposit into a pool. It receives both `sender` (the actual `msg.sender` of `addLiquidity`) and `owner` (the LP position recipient). It ignores `sender` entirely and only checks whether `owner` is allowlisted. Any address can therefore call `pool.addLiquidity(owner = allowlistedAddr)` and pass the guard, depositing tokens and manipulating bin state without being on the allowlist.

### Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`: [3](#0-2) 

`_beforeAddLiquidity` forwards both arguments to the extension: [4](#0-3) 

Inside the extension, the first parameter (`sender`) is unnamed/discarded. The guard only checks `allowedDepositor[msg.sender][owner]`, where `msg.sender` is the pool address and `owner` is the caller-supplied second argument: [2](#0-1) 

A non-allowlisted `callerAddr` can call `pool.addLiquidity(owner = allowlistedAddr, ...)`. The extension evaluates `allowedDepositor[pool][allowlistedAddr]` → `true`, and the deposit proceeds. `callerAddr` pays the tokens via the modify-liquidity callback; the LP position is minted to `allowlistedAddr`.

### Impact Explanation
The `DepositAllowlistExtension` is an admin-configured access control. Its entire purpose is to restrict which addresses may deposit. That invariant is completely broken: any address can deposit by naming an allowlisted address as `owner`. This allows:
- Unauthorized injection of liquidity into specific bins, directly affecting the pool's bin cursor and swap price path.
- Circumvention of regulatory or risk-management deposit restrictions the pool admin intended to enforce.
- Potential front-running or bin-state manipulation attacks against other LPs or swappers.

This is an admin-boundary break: an unprivileged path bypasses an admin-configured guard, satisfying the contest's "Admin-boundary break" impact criterion.

### Likelihood Explanation
The pool's `addLiquidity` interface is fully public. The `owner` parameter is caller-controlled with no validation tying it to `msg.sender`. Any attacker who knows an allowlisted address (which is readable from `allowedDepositor` public mapping) can exploit this immediately with no special privileges. [5](#0-4)

### Proof of Concept
1. Deploy pool with `DepositAllowlistExtension` wired into `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, ownerAddr, true)` — `ownerAddr` is allowlisted; `callerAddr` is not.
3. `callerAddr` calls `pool.addLiquidity(owner = ownerAddr, salt, deltas, callbackData, "")`.
4. Extension evaluates `allowedDepositor[pool][ownerAddr]` → `true`; no revert.
5. `callerAddr`'s callback pays the tokens; LP shares are minted to `ownerAddr`.
6. Assert: deposit succeeded despite `callerAddr` never being allowlisted.
