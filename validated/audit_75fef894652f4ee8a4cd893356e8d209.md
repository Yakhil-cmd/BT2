### Title
Missing Zero-Address Validation in `change_reward_address` Permanently Freezes Unclaimed Yield - (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both the staking contract and the delegation pool contract allow a staker or pool member to set their `reward_address` to the zero address. Because the STRK token (OpenZeppelin ERC20 on Starknet) reverts on transfers to the zero address, any subsequent call to `claim_rewards` will permanently revert, freezing the user's accumulated unclaimed yield.

---

### Finding Description

`change_reward_address` in `staking.cairo` validates only that the new address is not a registered token address: [1](#0-0) 

`change_reward_address` in `pool.cairo` performs the same single check: [2](#0-1) 

Neither function checks `reward_address.is_non_zero()`. The same omission exists at initial entry: `stake()` in `staking.cairo` only guards against the token-address case: [3](#0-2) 

Once a zero `reward_address` is stored, every reward-claiming path that calls `checked_transfer(recipient: reward_address, ...)` will revert. For stakers this is `send_rewards_to_staker`: [4](#0-3) 

For pool members this is `claim_rewards` in `pool.cairo`: [5](#0-4) 

The error catalog already defines `ZERO_ADDRESS` as a recognised error string, confirming the protocol intended this guard to exist: [6](#0-5) 

---

### Impact Explanation

A staker or pool member who sets (or initially registers) a zero `reward_address` will have their `claim_rewards` call revert on every invocation because the STRK ERC20 token (OpenZeppelin Cairo) rejects transfers to `address(0)`. The accumulated `unclaimed_rewards_own` / `_unclaimed_rewards_from_v0` balance is locked in the contract with no recovery path, constituting **permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

Any staker or pool member can trigger this unilaterally by calling `change_reward_address(0)` — no privileged role is required. The call succeeds silently (no revert, event emitted), so the damage is not immediately apparent. An accidental mis-paste of a zero address during a routine reward-address rotation is a realistic scenario. The `ZERO_ADDRESS` error already existing in the error catalogue indicates the developers anticipated this check but did not implement it in these two functions.

---

### Recommendation

Add a non-zero assertion at the top of both `change_reward_address` implementations and at the `reward_address` parameter of `stake` / `enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `assert_caller_is_not_zero` pattern already used elsewhere in the codebase: [7](#0-6) 

---

### Proof of Concept

1. **Staker stakes normally** — `reward_address` is initially valid.
2. **Staker calls** `staking_contract.change_reward_address(reward_address: 0)`.  
   - The only guard (`REWARD_ADDRESS_IS_TOKEN`) passes because `0` is not a registered token.  
   - `staker_info.reward_address` is written as `0`. [8](#0-7) 
3. **Rewards accrue** over subsequent epochs into `staker_info.unclaimed_rewards_own`.
4. **Anyone calls** `staking_contract.claim_rewards(staker_address)`.  
   - `send_rewards_to_staker` executes `token_dispatcher.checked_transfer(recipient: 0, amount: ...)`.  
   - The STRK ERC20 reverts: `'ERC20: transfer to 0'`. [9](#0-8) 
5. **Result**: All unclaimed rewards for this staker are permanently frozen. The identical path applies to pool members via `pool.cairo`'s `change_reward_address` → `claim_rewards`. [10](#0-9)
