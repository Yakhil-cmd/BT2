### Title
Period Boundary Straddling Allows 2× `PERIOD_MINT_CAP` Bypass in a Single Block Pair — (`L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

`PeriodMintLimit.periodAccountingSlot` keys its accounting on `block.timestamp / MINTING_PERIOD_DURATION` (an epoch-aligned integer). Because `tick()` carries no access control, any caller can invoke it once in the last block of period *k-1* and once in the first block of period *k*, consuming up to 5 messages each time. Both calls pass the `PERIOD_MINT_CAP` check independently, allowing 13,000,000e18 tokens to be minted in two consecutive Ethereum blocks (~12 seconds) — exactly 2× the intended weekly cap.

---

### Finding Description

`periodAccountingSlot` computes the period bucket as:

```solidity
uint256 period_index = block.timestamp / MINTING_PERIOD_DURATION;
``` [1](#0-0) 

This means the counter resets to zero at every exact multiple of 604,800 seconds. The cap check is:

```solidity
require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
``` [2](#0-1) 

`tick()` is an unrestricted public entry point:

```solidity
function tick() external payable {
``` [3](#0-2) 

Each call processes at most `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5` messages, each worth `TOKENS_PER_MINT_REQUEST = 1_300_000e18`, for a maximum of exactly `PERIOD_MINT_CAP = 6_500_000e18` per call: [4](#0-3) [5](#0-4) 

The constants are deliberately sized so one full tick exactly saturates one period's cap. Straddling the boundary defeats this design entirely.

---

### Impact Explanation

- **2× inflation in ~12 seconds**: 13,000,000e18 STRK minted instead of the intended 6,500,000e18 weekly maximum.
- The extra tokens are bridged to L2 and enter the reward pool, inflating the supply and diluting the real value of every staker's unclaimed yield.
- This fits **High: Theft of unclaimed yield** — the economic value of existing unclaimed rewards is reduced by the unexpected supply expansion.

---

### Likelihood Explanation

- `tick()` requires no role, no stake, no special state — any EOA can call it.
- Period boundaries are fully predictable (every 604,800 seconds from the Unix epoch).
- The only precondition is ≥10 pending L2→L1 mint-request messages. After a week of normal staking activity, this is routine.
- No validator collusion or timestamp manipulation is needed; the attacker simply submits two transactions in two consecutive blocks that naturally straddle the boundary.
- The `mintingAllowance` in `MintManager` is a separate per-caller budget set by the token admin; it is expected to be set generously so it does not interfere with normal weekly minting, making it unlikely to block the second tick. [6](#0-5) 

---

### Recommendation

Replace the fixed-epoch period index with a **rolling window** that tracks the earliest timestamp in the current window:

```solidity
// Store: windowStart[token] → uint256
function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
    uint256 ws = windowStart[token];
    if (block.timestamp >= ws + MINTING_PERIOD_DURATION) {
        // Start a new window anchored to now
        ws = block.timestamp;
        windowStart[token] = ws;
        periodMintAccounting()[token] = 0;
    }
    uint256 after_ = periodMintAccounting()[token] + amount;
    require(after_ <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
    periodMintAccounting()[token] = after_;
}
```

This ensures the cap applies to any contiguous 604,800-second window, not just calendar-aligned epochs.

### Proof of Concept

```
Precondition: 10 L2→L1 mint-request messages are pending in the StarkNet messaging contract.

Block N   (block.timestamp = k*604800 - 1):
  attacker calls tick()
  → period_index = (k*604800 - 1) / 604800 = k - 1
  → periodSlot = keccak256(604800, token, k-1)
  → minted[k-1] = 0 + 6_500_000e18 ≤ PERIOD_MINT_CAP  ✓
  → 5 messages consumed, 6_500_000e18 STRK minted & bridged

Block N+1 (block.timestamp = k*604800):
  attacker calls tick()
  → period_index = k*604800 / 604800 = k
  → periodSlot = keccak256(604800, token, k)   ← DIFFERENT SLOT
  → minted[k] = 0 + 6_500_000e18 ≤ PERIOD_MINT_CAP  ✓
  → 5 messages consumed, 6_500_000e18 STRK minted & bridged

Total minted in ~12 seconds: 13_000_000e18 (2× PERIOD_MINT_CAP)
Invariant violated: no sliding 604800-second window should exceed PERIOD_MINT_CAP.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
