### Title
Multi-NFT Royalty Integer Division Causes WalletConnect Confirmation to Display Understated Total Cost - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function incorrectly divides the total fungible amount equally among NFTs before computing each royalty, using integer (`BigInt`) division that truncates remainders. When a WalletConnect dApp sends a `chia_wallet.take_offer` command containing multiple NFTs with royalties, the confirmation dialog shows a total cost that is materially lower than what the wallet daemon will actually charge, causing the user to approve a transaction based on a false amount.

### Finding Description

In `packages/gui/src/electron/commands/parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the displayed total-with-royalties as follows:

```javascript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← integer division, truncates
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

Two compounding errors exist:

1. **Wrong base for each royalty.** Each NFT's royalty should be applied to the full fungible `amount` (the actual sale price). Instead, `amount` is split equally across all NFTs first. For N NFTs the displayed royalty is `1/N` of the correct value.

2. **Integer truncation.** `amount / BigInt(royaltyPercentages.length)` silently drops the remainder, introducing an additional undercount.

The result is fed into `withRoyaltyTotals` → `parseCommandDisplay` and surfaced as `amountWithRoyalties` in the WalletConnect confirmation dialog for `chia_wallet.take_offer`. [2](#0-1) 

The existing test fixture confirms the buggy value is the one actually rendered:

```
// 2 NFTs: royalty_percentage 500 (5 %) and 10 (0.1 %)
// amount = 100 000 000 mojos
amountWithRoyalties: '0.00010255'   // ← what the dialog shows
``` [3](#0-2) 

Correct calculation (each royalty on the full amount):
- NFT1 royalty: `100 000 000 × 500 / 10 000 = 5 000 000`
- NFT2 royalty: `100 000 000 × 10 / 10 000 = 100 000`
- Correct total: **105 100 000 mojos (0.0001051 XCH)**

Displayed total: **102 550 000 mojos (0.00010255 XCH)**

Discrepancy: **2 550 000 mojos understated** — and it scales linearly with the number of NFTs and their royalty percentages.

### Impact Explanation

The WalletConnect confirmation dialog is the sole security gate between a dApp command and the user's wallet. A user who sees `amountWithRoyalties: 0.00010255 XCH` and approves will have the wallet daemon deduct the correct (higher) amount. The user approved a transaction based on a materially false cost figure. With high-royalty NFTs (e.g., 20 % each) and several NFTs in one offer, the understated amount can be a large fraction of the true cost. This satisfies the **High** impact criterion: *WalletConnect state causes a user to approve the wrong amount*.

### Likelihood Explanation

Any WalletConnect-connected dApp can craft a `chia_wallet.take_offer` payload containing two or more NFTs with non-zero royalties. No special privilege is required beyond having an active WalletConnect session, which is the normal operating mode for dApp integrations. The bug is deterministic and reproducible for every multi-NFT offer.

### Recommendation

Remove the equal-split and apply each royalty to the full `amount`:

```javascript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

This matches the wallet daemon's per-NFT royalty semantics and eliminates the integer-truncation undercount.

### Proof of Concept

The existing test at `packages/gui/src/electron/commands/parseCommandDisplay.test.ts` (lines 251–335) already demonstrates the discrepancy. With `royaltyPercentages = [500, 10]` and `amount = 100 000 000`:

| | Mojos | XCH |
|---|---|---|
| Dialog shows (`amountWithRoyalties`) | 102 550 000 | 0.00010255 |
| Correct total | 105 100 000 | 0.00010510 |
| **Understatement** | **2 550 000** | **0.00000255** | [4](#0-3) 

A dApp attacker scales the attack by increasing the number of NFTs or their royalty percentages. With 5 NFTs each at 20 % royalty, the dialog would show 20 % of the true total royalty cost, causing the user to approve a transaction that costs 5× more in royalties than displayed.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
