### Title
Blocklist Bypass via EVM Contract Intermediary — (`app/proposal.go`, `app/block_address.go`)

### Summary

The Cronos sanctions blocklist is enforced exclusively at the transaction-envelope level. Both `ValidateTransaction` and `BlockAddressesDecorator` inspect only `ethTx.To()` — the top-level EVM destination field. No hook, precompile guard, or post-tx processor checks the Cronos blocklist against internal EVM call targets. An unprivileged attacker can route an ERC20/CRC20/CRC21 transfer to a blocked address through a deployed proxy contract, bypassing the blocklist entirely.

### Finding Description

**Enforcement points and their scope:**

`ProposalHandler.ValidateTransaction` in `app/proposal.go` checks:
- Transaction signers
- `ethTx.To()` — the outer EVM tx destination
- EIP-7702 authorization list addresses [1](#0-0) 

`BlockAddressesDecorator.AnteHandle` in `app/block_address.go` performs the same three checks, but only during `CheckTx`: [2](#0-1) 

Neither function inspects internal EVM call targets — the addresses reached by `CALL`, `DELEGATECALL`, or `STATICCALL` opcodes executed inside the EVM after the transaction is admitted.

**The only other `checkBlockedAddr` in the codebase** lives in the bank precompile: [3](#0-2) 

This calls `bc.bankKeeper.BlockedAddr(to)`, which checks the Cosmos SDK bank module's hardcoded module-account blocklist — **not** the Cronos sanctions blocklist stored via `MsgStoreBlockList`. It provides zero protection against the sanctions list for internal calls.

**The EVM hook** (`LogProcessEvmHook.PostTxProcessing`) processes logs for bridge/IBC events but contains no blocklist check: [4](#0-3) 

**Attack path:**

1. Address `E` is added to the Cronos sanctions blocklist via `MsgStoreBlockList`.
2. Attacker deploys proxy contract `C` (any unprivileged user can do this; `C` is not blocked).
3. Attacker sends EVM tx: `to = C`, calldata encodes `erc20.transfer(E, amount)`.
4. `ValidateTransaction` evaluates `ethTx.To() == C` → not in blocklist → **tx accepted**.
5. `ProcessProposalHandler` includes the tx in the block.
6. EVM executes `C`; `C` issues an internal `CALL` to the ERC20 contract with `transfer(E, amount)`.
7. The ERC20 contract updates its storage: `E`'s balance increases. No blocklist check occurs anywhere in this path.
8. `E` has received tokens despite being on the sanctions list.

### Impact Explanation

A blocked address receives ERC20, CRC20, or CRC21 token balances through a contract intermediary. The blocklist's stated purpose — preventing sanctioned addresses from receiving asset transfers — is defeated for any token transfer routed through a smart contract. This is a direct bypass of the block-list authorization check.

### Likelihood Explanation

The attack requires only: (a) knowledge of which address is blocked (the blocklist is encrypted but the effect is observable on-chain), and (b) the ability to deploy a contract and send one transaction. Both are available to any unprivileged user. No special privileges, leaked keys, or validator compromise are needed.

### Recommendation

Enforce the blocklist at EVM execution time, not only at the transaction-envelope level. Options include:

- Add a `PostTxProcessing` hook that inspects all ERC20 `Transfer` event logs emitted during execution and reverts (or strips) any log whose `to` field is in the blocklist.
- Integrate a blocklist check into the EVM state-transition layer so that any `CALL` whose value or token transfer targets a blocked address causes an EVM revert.
- At minimum, document that the blocklist only covers direct `ethTx.To()` destinations and does not prevent indirect token flows.

	to, err := sdk.AccAddressFromBech32(addr.String())

		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {

	return nil

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
