### Title
Malicious Participant Can Equivocate on Commitment Hash in DKG to Cause Permanent Denial of Key Generation — (File: `src/dkg.rs`)

---

### Summary

In `do_keyshare`, the polynomial commitment hash (Round 2) is sent via `chan.send_many` — a simple point-to-point send with no reliable-broadcast guarantee — while the actual commitment (Round 3) is sent via `do_broadcast` (Authenticated Double-Echo Broadcast). A malicious participant can exploit this asymmetry to send a different commitment hash to each honest party. When the actual commitment is later reliably broadcast, some honest parties' hash-verification passes while others' fails, splitting the honest set: some abort with `InvalidCommitmentHash`, while the rest hang indefinitely waiting for the aborted parties' success broadcast. This permanently denies key generation, reshare, and refresh to all honest parties.

---

### Finding Description

`do_keyshare` in `src/dkg.rs` implements a commit-then-reveal scheme across five rounds:

**Round 2 — commitment hash sent via `send_many` (point-to-point):**

```
// Step 2.9
let wait_round_1 = chan.next_waitpoint();
chan.send_many(wait_round_1, &commitment_hash)?;   // ← NOT reliable broadcast
``` [1](#0-0) 

Each party stores the hashes it receives directly from each sender:

```rust
for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?
{
    all_hash_commitments.put(from, their_commitment_hash);
}
``` [2](#0-1) 

**Round 3 — actual commitment sent via `do_broadcast` (reliable broadcast):**

```rust
let commitments_and_proofs_map = do_broadcast(
    &mut chan,
    &participants,
    me,
    (commitment, proof_of_knowledge),
)
.await?;
``` [3](#0-2) 

**Round 4 — each party verifies the revealed commitment against the hash it received in Round 2:**

```rust
verify_commitment_hash(
    &session_id,
    p,
    &mut commit_domain_separator.clone(),
    commitment_i,
    &all_hash_commitments,
)?;
``` [4](#0-3) 

`verify_commitment_hash` recomputes `H(participant, commitment, session_id)` and compares it to the stored hash:

```rust
fn verify_commitment_hash<C: Ciphersuite>(
    ...
    all_hash_commitments: &ParticipantMap<'_, HashOutput>,
) -> Result<(), ProtocolError> {
    let actual_commitment_hash = all_hash_commitments.index(participant)?;
    let commitment_hash =
        domain_separate_hash(domain_separator, &(&participant, &commitment, &session_id))?;
    if *actual_commitment_hash != commitment_hash {
        return Err(ProtocolError::InvalidCommitmentHash);
    }
    Ok(())
}
``` [5](#0-4) 

**The attack:** Because the commitment hash is sent via `send_many` (not reliable broadcast), a malicious participant M can send `H(C₁)` to Alice and `H(C₂)` to Bob. In Round 3, M reliably broadcasts `C₁` (so all parties see the same `C₁`). Alice's check passes (`H(C₁) == H(C₁)`), but Bob's check fails (`H(C₂) ≠ H(C₁)`), causing Bob to return `Err(ProtocolError::InvalidCommitmentHash)` and exit.

Alice then reaches `broadcast_success`, which calls `do_broadcast` and waits for every participant — including Bob — to broadcast:

```rust
broadcast_success(&mut chan, &participants, me, session_id).await?;
``` [6](#0-5) 

Since Bob has already aborted and will never send its echo-broadcast message, Alice (and any other party that accepted M's commitment) hangs indefinitely. The entire DKG session is permanently stalled.

This vulnerability is inherited by all three public operations that call `do_keyshare`: `keygen`, `reshare`, and `refresh`. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A single malicious participant can permanently deny key generation, reshare, or refresh to all honest parties by sending different commitment hashes to different subsets of the honest set. The parties that received the "wrong" hash abort; the parties that received the "right" hash hang waiting for the aborted parties' success broadcast. No honest party can complete the protocol. This maps directly to:

> **High: Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The attacker is a protocol participant who controls their own node. They need only modify their implementation to call `send_private` with different hash values for different recipients instead of `send_many`. No cryptographic primitive needs to be broken. The attack is repeatable: every time honest parties restart DKG, the malicious participant can repeat it. Likelihood is **Medium-High** given that the attacker is already a trusted participant in the key-generation ceremony.

---

### Recommendation

Replace the `send_many` call for the commitment hash with `do_broadcast`, so all honest parties are guaranteed to hold the same hash before the actual commitment is revealed:

```rust
// Round 2: use reliable broadcast for the commitment hash
let all_hash_commitments_map =
    do_broadcast(&mut chan, &participants, me, commitment_hash).await?;
```

This mirrors the treatment of the actual commitment in Round 3 and closes the equivocation window. Alternatively, the commitment hash round can be eliminated entirely by sending the commitment itself via `do_broadcast` in a single round (removing the need for a separate hash-binding step), which is the approach taken by several PedPop variants.

---

### Proof of Concept

1. Three participants run DKG: honest Alice, honest Bob, malicious M (threshold 2-of-3).
2. **Round 2:** M sends `H(C₁)` to Alice and `H(C₂)` to Bob via `send_many` (different values per recipient).
3. **Round 3:** M reliably broadcasts `C₁` via `do_broadcast`; all parties receive the same `C₁`.
4. **Round 4:**
   - Alice: `H(C₁) == H(C₁)` → verification passes; Alice continues.
   - Bob: `H(C₂) ≠ H(C₁)` → `verify_commitment_hash` returns `Err(InvalidCommitmentHash)`; Bob exits.
5. **Round 5:** Alice calls `broadcast_success` → `do_broadcast` waits for Bob's echo; Bob has exited and never sends it.
6. Alice hangs indefinitely. DKG fails for all honest parties.
7. M can repeat this attack on every subsequent DKG attempt, permanently preventing key generation.

pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)
}
