### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no validation against `maxAdminSpreadFeeE6`, while the parallel function `setPoolAdminFees` does enforce that cap. A pool admin can set per-bin additional fees up to `uint16` max (65,535 ≈ 6.55% in E6 units) on any bin, bypassing the factory's admin fee boundary entirely.

### Finding Description

`setPoolAdminFees` enforces the cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no such check — it passes values straight through:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool also has no cap check — it stores the raw values: [3](#0-2) 

In the swap price calculation, `addFeeBuyE6`/`addFeeSellE6` are added directly on top of the oracle-derived `baseFeeX64`:

```solidity
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The ask price paid by a buyer is then:

```solidity
uint256 askBeforeNotional = Math.mulDiv(marginalPriceX64, ONE_X64 + buyFeeX64, ONE_X64, Math.Rounding.Ceil);
``` [5](#0-4) 

So the bin additional fee is a direct, uncapped surcharge on every swap through that bin, controlled entirely by the pool admin.

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` and impose an additional ~6.55% fee on every swap through that bin, with no factory-level guard. This is additive on top of the oracle-derived spread and the global `spreadFeeE6`. The `maxAdminSpreadFeeE6` cap (up to 20%) is rendered meaningless for per-bin fee extraction: the admin can charge traders up to `uint16` max in additional fees per bin, extracting value from swappers beyond the intended admin fee boundary. This is a direct loss of user principal on every swap routed through the affected bin.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly scoped as "semi-trusted only inside caps." The asymmetry between `setPoolAdminFees` (capped) and `setPoolBinAdditionalFees` (uncapped) is a clear omission. Any pool admin — including one who accepted the role after a two-step transfer — can exploit this immediately with a single transaction, with no timelock.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (uint24(addFeeBuyE6)  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (uint24(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, define a dedicated `maxAdminBinAdditionalFeeE6` cap and enforce it here.

### Proof of Concept

1. Deploy a pool with `adminSpreadFeeE6 = 0` and `maxAdminSpreadFeeE6 = 200_000`.
2. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Execute a swap through bin 0.
4. Assert the effective buy fee applied to the swap equals `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, which corresponds to ~6.55% additional fee — exceeding any reasonable interpretation of the `maxAdminSpreadFeeE6` boundary.
5. Confirm no revert occurred in step 2 despite the fee exceeding what `setPoolAdminFees` would allow if the same value were passed as `newAdminSpreadFeeE6` (which would revert only if `> 200_000`, but the point is there is zero validation at all in the bin path).
