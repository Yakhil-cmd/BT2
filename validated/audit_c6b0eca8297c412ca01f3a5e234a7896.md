### Title
Presignature Reuse by Malicious Coordinator Enables Secret Key Extraction via ECDSA Nonce-Reuse - (File: src/ecdsa/robust_ecdsa/mod.rs, src/ecdsa/ot_based_ecdsa/mod.rs)

---

### Summary

Both the Robust ECDSA and OT-based ECDSA schemes expose `PresignOutput` as a `Clone`-able struct with no single-use enforcement. A malicious coordinator can orchestrate two separate signing sessions against the same presignature material with different `(msg_hash, tweak, entropy)` inputs. The resulting pair of signatures have multiplicatively related nonces, enabling full secret key extraction via standard ECDSA nonce-reuse techniques. The library documents "never reuse a presignature" but provides no mechanism to enforce it.

---

`RerandomizedPresignOutput` also derives `Clone` in both schemes: [3](#0-2) [4](#0-3) 

The `sign()` function in the robust scheme takes `presignature: RerandomizedPresignOutput` by value (moved), which appears to enforce single-use at the type level: [5](#0-4) 

However, because `PresignOutput` is `Clone`, a caller can call `rerandomize_presign()` on the same `PresignOutput` twice with different `RerandomizationArguments`, producing two distinct `RerandomizedPresignOutput` values from the same underlying nonce material. The `rerandomize_presign()` function has no state and performs no usage tracking: [6](#0-5) 

The signing protocol itself has no cross-participant agreement round on `(msg_hash, tweak)`. Each participant independently computes their share and sends it only to the coordinator:

```rust
fn do_sign_participant(...) -> Result<SignatureOption, ProtocolError> {
    let s_me = compute_signature_share(presignature, msg_hash, participants, me)?;
    let wait_round = chan.next_waitpoint();
    chan.send_private(wait_round, coordinator, &s_me)?;
    Ok(None)
}
``` [7](#0-6) 

There is no broadcast or echo-broadcast of `(msg_hash, tweak, participants)` before participants commit their shares. Participants have no way to detect that the coordinator is running a second session against the same presignature.

The security documentation explicitly acknowledges this gap and the resulting attack:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [8](#0-7) 

The code enforces `N = 2t+1` and `msg_hash != 0` as partial mitigations: [9](#0-8) 

But the third and most critical constraint — "Never reuse a presignature" — is entirely unenforced: [10](#0-9) 

The README also claims "Each presignature is consumed exactly once (one-time use)" but this is contradicted by the `Clone` derivation: [11](#0-10) 

---

### Impact Explanation

A malicious coordinator orchestrates two signing sessions against the same `PresignOutput`. Both sessions use the full `2t+1` participant set (satisfying the enforced constraint). Session A uses `(h1, ε1, ρ1)` and session B uses `(h2, ε2, ρ2)`. Each session produces a valid signature:

- `(R_A, s_A)` where `R_A = δ_A · R`, nonce `k_A = k · δ_A^{-1}`
- `(R_B, s_B)` where `R_B = δ_B · R`, nonce `k_B = k · δ_B^{-1}`

The ratio `k_A / k_B = δ_B / δ_A` is fully known to the coordinator (since `δ_A` and `δ_B` are computed from public inputs). This is a textbook ECDSA related-nonce attack, yielding the aggregate secret key `x`. This matches the **Critical** impact: extraction of aggregate secret material.

---

### Likelihood Explanation

The coordinator role is reachable by any participant without privileged assumptions — the coordinator is simply one of the `2t+1` signers. A single malicious participant who is chosen as coordinator can execute this attack. The only prerequisite is participation in a presigning session, which is a normal protocol operation. The attack requires two signing requests to honest participants, both of which are indistinguishable from legitimate requests.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`** in both schemes. This forces callers to consume the presignature exactly once at the type level and prevents accidental or malicious cloning.

2. **Add a cross-participant agreement round** before participants commit their signature shares. Participants should broadcast a commitment to `(msg_hash, tweak, participants)` and verify that all other participants agree before sending `s_i` to the coordinator. This prevents a malicious coordinator from presenting different signing contexts to different participants.

3. **Add a presignature usage flag** (e.g., a `Cell<bool>` or `AtomicBool`) inside `PresignOutput` that is set on first use and causes `rerandomize_presign()` to return an error on subsequent calls.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
