### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Path Allows Single-Participant Signature — (`File: src/frost/mod.rs`)

### Summary
`assert_sign_inputs` in the FROST signing path validates that `threshold` does not exceed the participant count, but omits the lower-bound check (`threshold >= 2`) that is enforced in every other protocol entry point. A malicious coordinator can pass `threshold = 1` to the signing function, reducing the effective signing quorum to a single participant and enabling unauthorized production of a valid threshold signature without the required multi-party agreement.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both an upper and a lower bound on the threshold: [1](#0-0) 

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

The analogous validation function used in the FROST EdDSA and RedJubjub signing path, `assert_sign_inputs` in `src/frost/mod.rs`, only enforces the upper bound: [2](#0-1) 

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check for threshold < 2
```

The `ReconstructionLowerBound` type itself imposes no minimum: [3](#0-2) 

So `threshold = 1` (or even `threshold = 0`) passes `assert_sign_inputs` without error. The FROST protocol then uses this threshold to determine how many signature shares are required for aggregation. With `threshold = 1`, a single participant's share is sufficient to produce a valid aggregate signature, regardless of the threshold that was used during key generation.

The same gap exists in `presign` for FROST: [4](#0-3) 

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
```

### Impact Explanation

**Critical — Unauthorized creation of a valid threshold signature.**

A malicious coordinator who controls the signing invocation can supply `threshold = 1`. The FROST aggregation step then requires only one signature share to produce a valid signature over the group's public key. This completely bypasses the intended t-of-n security guarantee: a single compromised or colluding participant can unilaterally produce a signature that is cryptographically valid under the shared public key, without the knowledge or participation of the remaining honest parties.

### Likelihood Explanation

The coordinator role is explicitly modeled as a potentially adversarial participant in this library (the coordinator is required to be in the participant list but is otherwise untrusted for correctness). The signing API is a public library function; any caller who controls the `threshold` argument — including a malicious coordinator node — can trigger this path. No special privileges beyond being a valid participant are required.

### Recommendation

Add the same lower-bound check that `assert_key_invariants` already enforces to both `assert_sign_inputs` and the FROST `presign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing pattern in `src/dkg.rs` and `src/ecdsa/ot_based_ecdsa/triples/generation.rs`. [5](#0-4) 

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (valid, enforced by `assert_key_invariants`).
2. Call `frost::eddsa::sign` (or `redjubjub::sign`) with the same 3 participants but `threshold = 1`.
3. `assert_sign_inputs` accepts `threshold = 1` — no error is returned.
4. The FROST aggregation requires only 1 signature share.
5. The coordinator, acting alone, submits their single share; the aggregator produces a valid signature over the shared public key.
6. Standard Ed25519 verification against the group public key succeeds, confirming an unauthorized threshold signature was created by a single party.
