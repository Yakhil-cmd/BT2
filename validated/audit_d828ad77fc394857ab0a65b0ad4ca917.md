### Title
Unprivileged Caller Can Permanently Suppress Block Rewards via `update_rewards(disable_rewards=true)` - (File: src/staking/staking.cairo)

### Summary
`update_rewards` in the Staking contract has no access control and accepts a caller-controlled `disable_rewards` flag. Any unprivileged address can call it with `disable_rewards=true` to consume the one-call-per-block slot (tracked by the global `last_reward_block`) without distributing rewards, permanently denying block rewards for all stakers for that block.

### Finding Description
`update_rewards` is a public function gated only by `general_prerequisites`, which checks that the contract is not paused and the caller is non-zero — no role restriction exists. [1](#0-0) 

`general_prerequisites` itself: [2](#0-1) 

The function enforces a global, single-slot-per-block invariant via `last_reward_block`: [3](#0-2) 

After validating the staker, it unconditionally writes the current block to `last_reward_block` and then checks whether to actually distribute rewards: [4](#0-3) 

`last_reward_block` is a single global field, not per-staker: [5](#0-4) 

Because the write to `last_reward_block` happens **before** the `disable_rewards` early-return, any caller who invokes `update_rewards(any_valid_staker, disable_rewards=true)` will:

1. Advance `last_reward_block` to the current block.
2. Skip all reward distribution.
3. Cause every subsequent call to `update_rewards` in the same block to revert with `REWARDS_ALREADY_UPDATED`.

Block rewards for that block are permanently unrecoverable.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.** Block rewards that are never distributed via `_update_rewards` are never accounted for in `reward_supplier_dispatcher.update_unclaimed_rewards_from_staking_contract` and never credited to `staker_info.unclaimed_rewards_own` or forwarded to delegation pools. [6](#0-5) 

Because the slot is consumed globally, a single attacker call suppresses rewards for **all** stakers for that block, not just one.

### Likelihood Explanation
**High.** The attack requires no funds, no special role, and no prior state. Any non-zero address can execute it. It can be repeated every block. On Starknet, where transaction ordering within a block is deterministic and observable, a griever can reliably front-run the legitimate `update_rewards` call each block.

### Recommendation
Add role-based access control to `update_rewards` restricting it to the designated block-proposer role or the attestation contract. Alternatively, decouple the `last_reward_block` write from the reward-distribution path so that a `disable_rewards=true` call does not consume the block slot.

### Proof of Concept
1. Attacker observes that `current_block_number > last_reward_block` (slot is open).
2. Attacker calls `update_rewards(any_valid_staker_address, disable_rewards=true)`.
3. `last_reward_block` is set to `current_block_number`; no rewards are distributed.
4. The legitimate consensus caller invokes `update_rewards(staker, disable_rewards=false)` — it reverts with `REWARDS_ALREADY_UPDATED` because `current_block_number == last_reward_block`.
5. All stakers lose their block rewards for that block permanently.
6. Repeat every block.
