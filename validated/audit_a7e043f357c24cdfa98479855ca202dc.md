### Title
Revoking key_0 Does Not Revoke Duplicate Keys at Index > 0 â€” Revoked Key Material Remains Usable via Duplicate Key Index - (`fvm/environment/accounts.go`, `fvm/environment/account-key-metadata/metadata.go`)

### Summary

The Flow FVM stores revocation status for key_0 (index 0) inside the `apk_0` register as part of the full `AccountPublicKey` struct, while revocation status for keys at index > 0 is stored independently in `weightAndRevokedStatusBytes` within the account status register. When a duplicate key (key_1) is added that shares key material with key_0, its revoked status entry in `weightAndRevokedStatusBytes` is completely independent of key_0's revoked flag in `apk_0`. Revoking key_0 only updates `apk_0`; it does not cascade to key_1's independent revoked-status entry. The transaction verifier reads key_1's revoked status from `weightAndRevokedStatusBytes` (still `false`) and the public key bytes from `storedKeyIndex=0` (the `apk_0` register), authorizing the transaction with key material that was intended to be revoked.

### Finding Description

**Step 1 â€” Add key_0 (index 0):**

`appendPublicKey` special-cases `keyIndex == 0` and calls `setAccountPublicKey0`, storing the full `AccountPublicKey` struct (including `Revoked=false`) in the `apk_0` register. No entry is written to `keyMetadataBytes`. [1](#0-0) 

**Step 2 â€” Add key_1 (duplicate of key_0, index 1):**

`appendPublicKey` calls `appendKeyMetadataToAccountStatusRegister`, which calls `status.AppendAccountPublicKeyMetadata`. Inside `appendAccountPublicKeyMetadata`, `FindDuplicateKey` detects that key_1 is a duplicate of key_0 (`duplicateStoredKeyIndex=0`). `AppendDuplicateKeyMetadata` is then called, which:
- Appends key_1's own `revoked=false` and `weight` to `weightAndRevokedStatusBytes` (independent of key_0's status)
- Records the mapping `keyIndex=1 â†’ storedKeyIndex=0` in `mappingBytes`
- Returns `saveKey=false` so no new key bytes register is created [2](#0-1) [3](#0-2) 

**Step 3 â€” Revoke key_0 (index 0):**

`RevokeAccountPublicKey(addr, 0)` special-cases `keyIndex == 0` and calls `revokeAccountPublicKey0`, which reads the `apk_0` register, sets `Revoked=true`, and writes it back. The `keyMetadataBytes` (which contains key_1's independent `weightAndRevokedStatusBytes` entry) is **never touched**. [4](#0-3) 

**Step 4 â€” Transaction verifier checks key_1:**

`getAccountKeysAndAggregateWeights` calls `GetRuntimeAccountPublicKey(addr, 1)`. Since `keyIndex != 0`, it reads the account status and calls `status.AccountPublicKeyMetadata(1)` â†’ `GetKeyMetadata(keyMetadataBytes, 1, deduplicated=true)`:
- `revoked` is read from `weightAndRevokedStatusBytes` at position `keyIndex-1 = 0` â†’ key_1's own entry â†’ **`false`** (not revoked)
- `storedKeyIndex` is read from the mapping â†’ **`0`**

Then `getStoredPublicKey(a, address, 0)` retrieves the raw key bytes from `apk_0` (the same bytes as key_0). The returned `RuntimeAccountPublicKey` has `Revoked=false` and the key material of the "revoked" key_0. [5](#0-4) 

The verifier checks `accountKey.Revoked` â€” it is `false` â€” and proceeds to verify the cryptographic signature against key_0's public key bytes, which succeeds because the attacker still holds the private key. [6](#0-5) 

The sequence number checker (`transactionSequenceNum.go`) also calls `GetAccountPublicKeyRevokedStatus(addr, 1)` â†’ `status.AccountPublicKeyRevokedStatus(1)` â†’ reads from `weightAndRevokedStatusBytes` â†’ returns `false`. It does not catch the bypass either. [7](#0-6) 

### Impact Explanation

An attacker who controls account X can:
1. Add key_0 (full weight 1000)
2. Add key_1 as a duplicate of key_0 (full weight 1000)
3. Revoke key_0 (e.g., as part of a key rotation or to appear compliant)
4. Continue signing and submitting transactions using key_1's index, which the FVM authorizes because key_1's revoked status is independently stored as `false`

This violates the core security invariant that revoking a key must prevent all use of that key material for transaction authorization. An attacker can permanently retain signing capability over an account even after the account owner (or a Cadence contract) believes the key has been revoked.

### Likelihood Explanation

The attack requires only that the attacker control their own account â€” a standard, permissionless operation on Flow. Adding duplicate keys is not blocked by any guard. The exploit is deterministic and requires no brute force, front-running, or privileged access. It is locally reproducible on the Flow emulator/localnet without any modification.

### Recommendation

When revoking key_0 (index 0), the FVM must also iterate over all keys at index > 0 that have `storedKeyIndex=0` in their deduplication mapping and set their `weightAndRevokedStatusBytes` revoked flag to `true`. Alternatively, the revocation check for any key at index > 0 whose `storedKeyIndex` maps to a key that is itself revoked should propagate the revoked status. A simpler fix is: in `GetRuntimeAccountPublicKey` and `GetAccountPublicKeyRevokedStatus`, when `storedKeyIndex == 0`, additionally check the `apk_0` register's `Revoked` flag and OR it with the key's own revoked status from `weightAndRevokedStatusBytes`.

### Proof of Concept

```
1. On localnet emulator, create account A with key_0 (weight=1000, ECDSA_P256/SHA3_256).
2. Submit Cadence tx: account.keys.add(publicKey: key_0_pub, hashAlgorithm: SHA3_256, weight: 1000.0)
   â†’ key_1 is added at index 1 with the same public key bytes as key_0.
3. Submit Cadence tx: account.keys.revoke(keyIndex: 0)
   â†’ key_0 is revoked; apk_0 register now has Revoked=true.
4. Construct a new transaction with:
   - ProposalKey: {Address: A, KeyIndex: 1, SequenceNumber: <current>}
   - Authorizer: A
   - Payer: A
   Sign with the private key corresponding to key_0/key_1.
5. Submit and execute the transaction.
6. Assert: transaction succeeds (expected: rejected with "account key has been revoked").
```

The transaction at step 5 will be accepted by the FVM because `GetRuntimeAccountPublicKey(A, 1).Revoked == false`, demonstrating that the revocation of key_0 does not prevent key_1 from being used for authorization.

// AppendDuplicateKeyMetadata appends duplicate key metadata.
func (m *KeyMetadataAppender) AppendDuplicateKeyMetadata(
	keyIndex uint32,
	duplicateStoredKeyIndex uint32,
	revoked bool,
	weight uint16,
) (err error) {

	// Append revoked status and weight
	m.weightAndRevokedStatusBytes, err = appendWeightAndRevokedStatus(m.weightAndRevokedStatusBytes, revoked, weight)
	if err != nil {
		return err
	}

	if !m.deduplicated {
		// Set deduplication flag
		m.deduplicated = true

		// Save mapping start key index.
		m.startIndexForMapping = keyIndex
	}

	// Append duplicate stored key index to mapping
	m.mappingBytes, err = appendStoredKeyIndexToMappings(m.mappingBytes, duplicateStoredKeyIndex)
	if err != nil {
		return err
	}

	return nil
}
