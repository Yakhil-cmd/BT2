### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing and Presigning Allows Corrupted Signature Outputs — (`File: src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` accept a `threshold` parameter but only validate its upper bound (must not exceed participant count). They do not enforce the lower bound (`threshold >= 2`) that is explicitly required and enforced during key generation in `assert_key_invariants` (`src/dkg.rs`). A caller supplying `threshold = 0` or `threshold = 1` bypasses this guard entirely, causing the FROST signing protocol to proceed with an invalid threshold and produce corrupted, unusable signature outputs that honest parties accept as protocol completions.

---

### Finding Description

During DKG, `assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

However, `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

The lower-bound check (`threshold < 2`) is entirely absent. The same omission exists in the `presign` function:

```rust
// src/frost/mod.rs lines 71-77
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: args.threshold.into(),
        max: participants.len(),
    });
}
``` [3](#0-2) 

Neither function rejects `threshold = 0` or `threshold = 1`. Because `ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement of its own, any value is accepted: [4](#0-3) 

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

In FROST, the signing coordinator aggregates participant signature shares via Lagrange interpolation using the supplied `threshold` to determine the interpolation basis. If `threshold = 1` is supplied but keys were generated with `threshold = 2`, the Lagrange coefficients are computed for a 1-of-N scheme instead of the correct 2-of-N scheme. The resulting aggregate signature is cryptographically invalid (it will fail standard Ed25519/RedJubjub verification), yet the protocol completes without any internal error. All honest participants observe a successful protocol run and accept the output, unaware that the signature is unusable. This constitutes corruption of sign outputs within the allowed impact scope.

---

### Likelihood Explanation

**Medium.** The entry path is any direct caller of `assert_sign_inputs` or `presign` in `src/frost/mod.rs` — including a malicious coordinator or a misconfigured honest participant. The `threshold` parameter is caller-supplied with no type-level enforcement. A malicious coordinator deliberately passing `threshold = 1` can reliably trigger this on every signing round, permanently denying usable signatures to honest parties.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces, in both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check at `src/dkg.rs` lines 580–582 and closes the inconsistency between keygen and signing validation. [5](#0-4) 

---

### Proof of Concept

1. Run DKG with `participants = [P0, P1, P2]` and `threshold = 2` (accepted, keys generated correctly).
2. Call `assert_sign_inputs(&[P0, P1, P2], /*threshold=*/ 1usize, P0, P0)` — no error is returned because only the upper-bound check fires and `1 <= 3`.
3. Proceed to FROST signing with `threshold = 1`. The coordinator computes Lagrange coefficients for a 1-of-3 scheme (all coefficients = 1 for a single-element basis), which differs from the 2-of-3 coefficients used during keygen.
4. The aggregated signature fails Ed25519 verification, yet the protocol returned `Ok(...)` to all honest participants.
5. Signing is permanently broken for this session with no internal diagnostic, matching the "permanent denial of signing for honest parties" impact. [6](#0-5)
