### Title
Missing Caller Restriction on `update_rewards` Allows Anyone to Permanently Grief Staker Rewards - (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, which unconditionally advances `last_reward_block` to the current block without distributing any rewards. Because the per-block reward slot is consumed, the staker and all their pool members permanently lose the rewards for that block. Repeated every block, this completely freezes all yield accrual.

---

### Finding Description

The spec for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer.

(`docs/spec.md` lines 1644–1645) [1](#0-0) 

The implementation in `StakingRewardsManagerImpl` performs only `general_prerequisites()` (a pause check) and a block-number guard, with **no check on `get_caller_address()`**:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validity checks ...

    // ← last_reward_block is written BEFORE the disable_rewards branch
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits without distributing
    }
    // ... actual reward distribution ...
}
``` [2](#0-1) 

The critical sequence is:

1. `last_reward_block` is written to `current_block_number` at line 1485 **unconditionally**.
2. If `disable_rewards == true`, the function returns at line 1488 **without distributing any rewards**.
3. Any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED` (line 1454–1458).

The interface definition confirms there is no access-control annotation in the trait itself:

```cairo
pub trait IStakingRewardsManager<TContractState> {
    fn update_rewards(
        ref self: TContractState, staker_address: ContractAddress, disable_rewards: bool,
    );
}
``` [3](#0-2) 

The test suite calls `update_rewards` without any `cheat_caller_address_once`, confirming no access control is enforced at runtime:

```cairo
staking_rewards_dispatcher.update_rewards(:staker_address, disable_rewards: true);
``` [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Each Starknet block produces a fixed block reward for each active staker (calculated from `yearly_mint` and `avg_block_duration`). Once `last_reward_block` is set to block `N` with `disable_rewards: true`, the reward for block `N` is **permanently lost** — it is never added to `unclaimed_rewards_own` for the staker, and never transferred to the delegation pools. There is no recovery path; the slot is consumed.

An attacker who front-runs the sequencer's legitimate `update_rewards` call every block causes 100% of consensus-phase block rewards to be silently discarded for the targeted staker and all their pool members. [4](#0-3) 

---

### Likelihood Explanation

**High.** The function is publicly callable with no cost beyond gas. On Starknet L2, gas costs are low. An attacker needs only to submit a transaction per block targeting any staker. No privileged access, no tokens, no special state is required. The attack is trivially automatable and can be sustained indefinitely. [5](#0-4) 

---

### Recommendation

Add a caller check at the top of `update_rewards` to enforce that only the authorized sequencer address (or a designated rewards-manager role) can invoke it, matching the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

Alternatively, gate the function behind the existing `SECURITY_AGENT` or a new `REWARDS_MANAGER` role via the `RolesComponent` already present in the contract. [6](#0-5) 

---

### Proof of Concept

1. Staker `S` stakes STRK and K epochs pass — staker is active and eligible for consensus rewards.
2. `consensus_rewards_first_epoch` is set and the current epoch is past it.
3. Attacker `A` (any address) monitors the mempool. At each new block `N`:
   - `A` calls `update_rewards(staker_address: S, disable_rewards: true)`.
   - `last_reward_block` is set to `N`.
   - The function returns without distributing rewards.
4. When the legitimate sequencer later tries to call `update_rewards(S, false)` for block `N`, it reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` and all pool members receive zero rewards for block `N`.
6. Repeated every block: `S` and all delegators earn zero yield indefinitely.

The test `test_update_rewards_without_distribute` (lines 3985–4050) already demonstrates that calling with `disable_rewards: true` leaves `unclaimed_rewards_own` unchanged while consuming the block slot, confirming the mechanism. [4](#0-3) [7](#0-6)
