### Title
Disabled Per-Participant Signature Share Verification Enables Persistent Denial of Signing — (File: `src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

### Summary
The FROST signing coordinator deliberately passes an empty `BTreeMap` as the `verifying_shares` argument to `PublicKeyPackage::new()`, disabling per-participant signature share verification during aggregation. This is the direct analog of the commented-out governance replacement: a critical verification step that should execute has been neutralized. As a result, a single malicious participant can submit an invalid signature share, cause aggregation to fail, and cannot be identified or excluded — enabling persistent denial of signing for all honest parties.

### Finding Description
In `do_sign_coordinator_v1` and `do_sign_coordinator_v2` in `src/frost/eddsa/sign.rs`, the coordinator constructs the `PublicKeyPackage` with an empty verifying-shares map:

```rust
// We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
// Feature "cheater-detection" only points to a malicious participant, if there's such.
// It doesn't bring any additional guarantees.
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [1](#0-0) 

The identical pattern appears in `do_sign_coordinator_v2`: [2](#0-1) 

And in the RedJubjub coordinator: [3](#0-2) 

In the FROST protocol, `PublicKeyPackage` carries two pieces of data: the group verifying key and a map of per-participant verifying shares (`verifying_shares`). When the "cheater-detection" feature is active, `aggregate()` uses `verifying_shares` to verify each individual signature share before combining them. By supplying `BTreeMap::new()`, the per-participant verification step is entirely bypassed — only the final aggregated signature is checked against the group key.

The developers' comment claims this "doesn't bring any additional guarantees," but this is incorrect. Per-participant verification provides the critical guarantee that a misbehaving participant can be **identified and excluded**. Without it:

1. A malicious participant submits a crafted (invalid) signature share.
2. `aggregate()` sums all shares, the final signature fails the group-key check, and returns an error.
3. The coordinator receives only a generic failure — it cannot determine which participant submitted the bad share.
4. The malicious participant repeats this in every subsequent signing attempt.
5. Honest parties are permanently locked out of producing a valid signature.

### Impact Explanation
This matches **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.** FROST's documented trust model explicitly tolerates up to `t-1` malicious participants. A single malicious participant who is always included in the signing set (e.g., because the signing set equals the threshold) can cause every signing attempt to fail indefinitely. Without cheater detection, the coordinator has no mechanism to identify and exclude the culprit, making the DoS permanent rather than transient.

### Likelihood Explanation
Any participant who has completed DKG and holds a valid key share can act maliciously during signing by sending a crafted `round2::SignatureShare`. No special privilege or external capability is required. The attacker only needs to be included in the signing participant set, which is a normal protocol condition.

### Recommendation
Populate `verifying_shares` with the actual per-participant verifying shares derived from the DKG output, and enable the `cheater-detection` feature flag in `frost-ed25519` / `reddsa`. The `KeygenOutput` already carries the `public_key` (group verifying key); the per-participant verifying shares can be derived from the DKG commitment polynomials or stored alongside `KeygenOutput`. With a populated map, `aggregate()` will identify the exact participant whose share is invalid, allowing the coordinator to exclude them and retry with the remaining honest participants.

### Proof of Concept
1. Run DKG with participants `{P1, P2, P3}`, threshold `t = 2`.
2. Run presign (`sign_v2` path) to obtain `PresignOutput` for all three.
3. `P3` (malicious) computes a random scalar instead of the correct `round2::sign(...)` output and sends it as their `SignatureShare` to the coordinator.
4. The coordinator calls `aggregate()` with `PublicKeyPackage::new(BTreeMap::new(), vk_package)`.
5. `aggregate()` sums the shares; the result fails the final group-key verification; the coordinator receives `ProtocolError::AssertionFailed("...")`.
6. Because `verifying_shares` is empty, no per-participant check was performed; the coordinator cannot determine that `P3` is the culprit.
7. `P3` repeats step 3 in every subsequent signing round — signing is permanently denied for `P1` and `P2`.
