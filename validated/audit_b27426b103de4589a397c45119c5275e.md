### Title
Unauthenticated `ft_on_transfer` Allows Minting Unbacked ERC-20 Tokens Without Depositing NEP-141 Tokens - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entrypoint on the Aurora Engine contract can be called directly by any NEAR account. Because `deploy_erc20_token` also has no access control, an attacker can register their own NEAR account as a NEP-141 token in Aurora and then call `ft_on_transfer` directly — causing the engine to mint an arbitrary amount of ERC-20 tokens to any EVM address without any real NEP-141 tokens ever being deposited. This is a direct analog of the CryptoPunks frontrunning bug: a function intended to be called only by a legitimate token contract can be called by anyone, and the contract mints tokens to the caller-controlled address.

---

### Finding Description

The NEP-141 standard requires that `ft_on_transfer` is called by the token contract itself as a callback after `ft_transfer_call` has already debited the sender's balance. Aurora Engine's `ft_on_transfer` implementation does not enforce this invariant.

**Step 1 — No access control on `deploy_erc20_token`.**

`deploy_erc20_token` (Legacy variant) only checks `require_running` and performs no caller verification: [1](#0-0) 

Any NEAR account can call this to register itself as a NEP-141 token in Aurora's storage.

**Step 2 — `ft_on_transfer` has no caller whitelist.**

The function reads `predecessor_account_id` from the NEAR runtime and branches only on whether it equals the eth-connector account. For every other caller it unconditionally proceeds to mint ERC-20 tokens: [2](#0-1) 

There is no check that `predecessor_account_id` is a registered, legitimate NEP-141 token that has actually transferred tokens.

**Step 3 — `receive_erc20_tokens` mints from the privileged admin address.**

`receive_erc20_tokens` resolves the ERC-20 contract for the predecessor account and calls the ERC-20 `mint` function using `erc20_admin_address = current_address(current_account_id)` — the Aurora engine's own EVM address, which is the admin of every ERC-20 it deploys: [3](#0-2) 

The mint succeeds because it originates from the admin. The `amount` used is taken directly from the attacker-supplied `args.amount` with no backing check: [4](#0-3) 

---

### Impact Explanation

**Critical — Insolvency / Direct theft of funds.**

The attacker mints ERC-20 tokens that are not backed by any real NEP-141 tokens held by Aurora. These unbacked tokens can be:

- Supplied as collateral in DeFi protocols on Aurora to borrow real assets.
- Swapped for legitimate tokens in AMMs, draining liquidity pools.
- If the attacker also controls the NEP-141 contract, they can trigger the exit/withdraw flow to redeem other users' deposited NEP-141 tokens, directly stealing them.

The ERC-20 supply becomes permanently inflated relative to the actual NEP-141 reserves, making the bridge insolvent.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A NEAR account (free to create).
2. One call to `deploy_erc20_token` (permissionless).
3. One direct call to `ft_on_transfer` with an arbitrary `amount`.

No special privileges, no governance capture, no leaked keys. The entire attack is executable by any unprivileged NEAR account in a single block.

---

### Recommendation

Restrict `ft_on_transfer` so it only accepts calls from accounts that are already registered as NEP-141 tokens in Aurora's storage (i.e., accounts for which `get_erc20_from_nep141` would succeed). Reject calls from any other predecessor before any minting logic runs.

Additionally, add access control to `deploy_erc20_token` (at minimum for the Legacy variant) so that arbitrary accounts cannot self-register as NEP-141 tokens.

---

### Proof of Concept

```
// Attacker NEAR account: "attacker.near"

// 1. Register attacker's account as a NEP-141 token in Aurora (no access control)
aurora.deploy_erc20_token(
    DeployErc20TokenArgs::Legacy("attacker.near".parse().unwrap())
);
// Aurora now has an ERC-20 mapped to "attacker.near"

// 2. Call ft_on_transfer directly — no NEP-141 tokens are transferred
//    predecessor_account_id = "attacker.near" (set by NEAR runtime)
aurora.ft_on_transfer(
    sender_id: "victim.near",
    amount: 1_000_000_000_000_000_000_000_000,  // arbitrary, attacker-chosen
    msg: hex::encode(attacker_evm_address),
);

// 3. receive_erc20_tokens executes:
//    - get_erc20_from_nep141("attacker.near") → succeeds (registered in step 1)
//    - erc20_admin_address = current_address(aurora_account_id)  ← privileged admin
//    - ERC-20.mint(attacker_evm_address, 1_000_000_000_000_000_000_000_000) ← succeeds

// Result: attacker holds 1e24 ERC-20 tokens backed by zero NEP-141 deposits.
// Aurora bridge is insolvent.
```

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
