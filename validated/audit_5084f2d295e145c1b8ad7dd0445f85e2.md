### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Overly Broad `account_tx_in_pool_or_recent_block` Condition — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call (which performs signature verification) for an invoke transaction with nonce=1 when the account's on-chain nonce is 0, to support the deploy-account + invoke UX pattern. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from that account in the pool or recent block, not specifically a `deploy_account` transaction. An attacker can exploit this to inject an invoke transaction with nonce=1 and an **invalid or forged signature** into the mempool without triggering `__validate__`, as long as any other transaction from the target account is already pending.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions are simultaneously true:

```
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
account_tx_in_pool_or_recent_block(tx.sender_address()) == true
``` [1](#0-0) 

When it returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage` — the `__validate__` call is never made: [3](#0-2) 

The comment in `skip_stateful_validations` justifies the check as follows:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [4](#0-3) 

This reasoning is incorrect. `account_tx_in_pool_or_recent_block` checks:

```rust
self.state.contains_account(account_address) || self.tx_pool.contains_account(account_address)
``` [5](#0-4) 

`contains_account` returns `true` for **any** transaction type (invoke, declare, deploy_account) from that address. It does not distinguish whether the pooled transaction is a `deploy_account`.

**Attack path:**

1. Account `A` is deployed via `deploy_syscall` (not `deploy_account`), so its on-chain nonce is 0 and it has a class hash set.
2. A legitimate user submits an invoke tx for `A` with nonce=0 and a valid signature → passes `__validate__` → admitted to mempool.
3. `account_tx_in_pool_or_recent_block(A)` now returns `true` (via `tx_pool.contains_account`).
4. Attacker submits an invoke tx for `A` with nonce=1 and an **invalid/arbitrary signature**.
5. Gateway stateless validation passes (no signature check there).
6. `validate_nonce` passes: `0 <= 1 <= max_allowed_nonce_gap`. [6](#0-5) 

7. `validate_by_mempool` passes: no duplicate hash, nonce not too old.
8. `skip_stateful_validations` returns `true` (all three conditions met).
9. `run_validate_entry_point` is called with `skip_validate=true` → `__validate__` is **never called**.
10. Attacker's invalid-signature tx is admitted to the mempool.

An equivalent path exists for the intended deploy-account + invoke UX scenario: any account that has a **pending deploy_account tx** in the mempool (account_nonce=0 in state) is vulnerable — an attacker can race ahead and inject a nonce=1 tx with an invalid signature before the legitimate user does, occupying the nonce=1 slot.

The `validate_by_mempool` call does not check signatures: [7](#0-6) 

The mempool's `validate_tx` only checks for duplicate hashes and stale nonces: [8](#0-7) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unprivileged attacker can inject an invoke transaction with a completely invalid signature into the mempool for any account whose on-chain nonce is 0 and which has any pending transaction in the pool. The admitted transaction will fail during batcher execution (because `__validate__` is called with `validate=true` during actual execution), but it is accepted by the gateway and occupies the nonce=1 slot in the mempool. This enables:

- **Griefing/DoS**: The attacker's invalid nonce=1 tx occupies the slot; the legitimate user's nonce=1 tx is rejected with `DuplicateNonce` (or must pay a fee-escalation premium to replace it).
- **Admission of unauthorized transactions**: The gateway's invariant that every admitted transaction has passed `__validate__` is broken.

---

### Likelihood Explanation

**Moderate.** The preconditions are:

1. Target account has on-chain nonce=0 (deployed via `deploy_syscall`, or has a pending `deploy_account` tx).
2. Any transaction from that account is already in the mempool (observable on-chain/mempool).
3. Attacker submits a nonce=1 tx before the legitimate user.

The deploy-account + invoke UX pattern is explicitly supported and tested, meaning condition 1+2 are a normal operational state. The mempool is observable, so condition 3 is achievable by any network participant.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a specific check that verifies a **deploy_account** transaction exists for the account in the mempool. Add a dedicated mempool API (e.g., `deploy_account_tx_in_pool(address)`) that only returns `true` when the pending transaction for that address is of type `DeployAccount`. This preserves the UX intent while closing the signature-bypass window.

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await

// Use something like:
mempool_client.deploy_account_tx_in_pool(tx.sender_address()).await
```

### Proof of Concept

```
Precondition:
  - Account 0xABCD deployed via deploy_syscall; on-chain nonce = 0; has sufficient fee balance.
  - Legitimate user submits invoke_tx(sender=0xABCD, nonce=0, valid_sig) → admitted to mempool.

Attack:
  1. Attacker observes mempool: account_tx_in_pool_or_recent_block(0xABCD) == true.
  2. Attacker submits invoke_tx(sender=0xABCD, nonce=1, sig=[0xDEAD, 0xBEEF]).
  3. Gateway stateless validation: passes (no sig check).
  4. convert_rpc_tx_to_internal: passes (no sig check).
  5. extract_state_nonce_and_run_validations:
       get_nonce_from_state(0xABCD) → 0
       validate_nonce: 0 <= 1 <= max_allowed_nonce_gap → OK
       validate_by_mempool: no dup hash, nonce not stale → OK
       skip_stateful_validations:
         tx.nonce=1 == Nonce(ONE) ✓
         account_nonce=0 == Nonce(ZERO) ✓
         account_tx_in_pool_or_recent_block(0xABCD) = true ✓
         → returns true
       run_validate_entry_point(skip_validate=true):
         execution_flags.validate = false
         StatefulValidator::perform_validations → returns Ok(()) without __validate__
  6. Attacker's tx with invalid signature is added to mempool.

Result:
  - Legitimate user's nonce=1 tx is rejected with DuplicateNonce.
  - Attacker's invalid tx is executed by batcher, fails __validate__, is reverted.
  - Nonce=1 slot is consumed; user must fee-escalate or wait for eviction.
``` [9](#0-8) [10](#0-9) [5](#0-4)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
