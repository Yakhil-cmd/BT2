### Title
Unprivileged Caller Can Register Arbitrary NEAR Account as NEP-141 Token and Mint Unbacked ERC-20 Tokens — (`engine/src/contract_methods/connector.rs`)

---

### Summary

`deploy_erc20_token` has no access control. Any NEAR account can call it with an arbitrary `AccountId` as the NEP-141 argument, causing Aurora to deploy an ERC-20 and permanently register the mapping `attacker_account → erc20_address`. The attacker then calls `ft_on_transfer` directly from that account to mint an unlimited supply of the ERC-20 without locking any real NEP-141 tokens. These unbacked tokens can be sold on Aurora DEXes, draining real liquidity.

---

### Finding Description

**Step 1 — No access control on `deploy_erc20_token`**

`engine/src/contract_methods/connector.rs` `deploy_erc20_token` only checks `require_running`; there is no `require_owner_only` or any whitelist check: [1](#0-0) 

Any NEAR account can call this function with any `AccountId` as the NEP-141 argument.

**Step 2 — Registration is permanent and unconditional**

`engine::deploy_erc20_token` (called from the above) deploys an ERC-20 bytecode and calls `register_token`, which only rejects if the NEP-141 is *already* registered: [2](#0-1) 

Once registered, the mapping `attacker_account → erc20_address` is stored permanently in Aurora's storage.

**Step 3 — `ft_on_transfer` trusts `predecessor_account_id` as the NEP-141 identity**

`ft_on_transfer` branches on whether the caller is the ETH connector or any other registered NEP-141: [3](#0-2) 

If `predecessor_account_id` matches a registered NEP-141 (which the attacker's account now is), `receive_erc20_tokens` is called and ERC-20 tokens are minted to the address encoded in `msg`. There is no check that the caller actually locked real NEP-141 tokens.

**Step 4 — Confirmed by existing test**

The existing test `test_ft_on_transfer` confirms that after `deploy_erc20_token(nep141)`, calling `ft_on_transfer` with `predecessor = nep141` successfully mints ERC-20 tokens: [4](#0-3) 

The contrast test `test_ft_on_transfer_fail` confirms that an *unregistered* predecessor causes all tokens to be returned — meaning registration is the only gate: [5](#0-4) 

**Contrast with `mirror_erc20_token`**

The developers already recognized the need for access control in the related `mirror_erc20_token` function (which is owner-only and even has a `TODO` comment about adding an admin access list): [6](#0-5) 

`deploy_erc20_token` has no equivalent protection.

---

### Impact Explanation

An attacker mints an unbounded supply of an ERC-20 token on Aurora that is backed by nothing. If that token is listed on any Aurora DEX (e.g., Trisolaris), the attacker can sell the minted tokens for real bridged assets (USDC, ETH, wNEAR), directly draining liquidity providers. Users who hold or receive the unbacked ERC-20 cannot redeem it for real NEP-141 tokens because the "NEP-141" is the attacker's own account. This is **direct theft of user funds** — Critical impact.

---

### Likelihood Explanation

The entry path requires only:
1. A funded NEAR account (trivially obtained)
2. A call to `deploy_erc20_token` on Aurora with the attacker's own account ID
3. A call to `ft_on_transfer` on Aurora from that account

No privileged access, no leaked keys, no governance capture. Any NEAR account holder can execute this immediately.

---

### Recommendation

Add `require_owner_only` (or a dedicated whitelist of approved NEP-141 deployers) to `deploy_erc20_token`, consistent with the access control already applied to `mirror_erc20_token` and `factory_set_wnear_address`: [7](#0-6) 

Alternatively, validate that the supplied `AccountId` corresponds to an actual on-chain NEP-141 contract (e.g., by requiring a successful `ft_metadata` call before completing registration, and restricting who may initiate the flow).

---

### Proof of Concept

```
1. Attacker creates NEAR account `evil.near`.

2. Attacker calls `deploy_erc20_token` on Aurora with args = Legacy("evil.near").
   → Aurora deploys ERC-20 at address 0xABCD and stores evil.near → 0xABCD.

3. Attacker calls `ft_on_transfer` on Aurora directly from `evil.near`:
     sender_id: "victim.near"
     amount:    "1000000000000000000000000"
     msg:       hex(attacker_evm_address)
   → predecessor_account_id == "evil.near" == registered NEP-141
   → receive_erc20_tokens mints 1e24 of 0xABCD to attacker_evm_address.

4. Attacker swaps 0xABCD tokens on an Aurora DEX for real bridged USDC/ETH.
   Liquidity providers lose real assets; the ERC-20 is unbacked.
```

    pub fn register_token(
        &mut self,
        erc20_token: Address,
        nep141_token: AccountId,
    ) -> Result<(), RegisterTokenError> {
        match get_erc20_from_nep141(&self.io, &nep141_token) {
            Err(GetErc20FromNep141Error::Nep141NotFound) => (),
            Err(GetErc20FromNep141Error::InvalidNep141AccountId) => {
                return Err(RegisterTokenError::InvalidNep141AccountId);
            }
            Err(GetErc20FromNep141Error::InvalidAddress) => {
                return Err(RegisterTokenError::InvalidAddress);
            }
            Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered),
        }

        let erc20_token = ERC20Address(erc20_token);
        let nep141_token = NEP141Account(nep141_token);
        nep141_erc20_map(self.io).insert(&nep141_token, &erc20_token);
        Ok(())
    }

fn test_ft_on_transfer() {
    let mut runner = AuroraRunner::new();
    // Standalone runner presently does not support ft_on_transfer
    runner.standalone_runner = None;
    let nep141 = "tt.testnet";
    let alice = "alice";
    let token = runner.deploy_erc20_token(nep141);
    let amount = Balance::new(10);
    let recipient = runner.create_account().address;

    let balance = runner.balance_of(token, recipient, DEFAULT_AURORA_ACCOUNT_ID);
    assert_eq!(balance, U256::from(0));

    let res = runner.ft_on_transfer(nep141, alice, alice, amount, &recipient.encode());
    // Transaction should succeed so return amount is 0
    assert_eq!(res, "\"0\"");

    let balance = runner.balance_of(token, recipient, DEFAULT_AURORA_ACCOUNT_ID);
    assert_eq!(balance, U256::from(amount.as_u128()));
}

pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(

    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
