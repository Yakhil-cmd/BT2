### Title
Permanent ETH Freeze on Failed `ft_transfer` in `ExitToNear` Precompile When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile deducts ETH from a user's EVM balance before scheduling a `ft_transfer` promise on NEAR. When the `error_refund` compile-time feature is disabled, no callback is registered to handle `ft_transfer` failures. If the NEAR-side transfer fails (e.g., recipient account not registered with the eth-connector), the ETH is permanently frozen at the precompile address with no recovery path. This is the direct Aurora analog of Solidity's `.transfer` anti-pattern: a one-shot transfer with no fallback on failure.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` precompile constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a plain base-ETH exit (not wNEAR unwrap), `exit_base_token_to_near` returns `None` for `transfer_near_args`. [2](#0-1) 

This means `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()` (both fields `None`). The subsequent branch then schedules **only** the bare `ft_transfer` promise with no callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

The `exit_to_near_precompile_callback` NEAR entrypoint — which is the only place `engine::refund_on_error` is invoked — is therefore never called when the promise fails:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op; ETH stays frozen at precompile address
};
``` [4](#0-3) 

The integration test explicitly confirms the fund-loss outcome when `error_refund` is absent:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

The same pattern is confirmed for ERC-20 exits: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH is atomically deducted from the caller's EVM balance during EVM execution (it moves to `exit_to_near::ADDRESS`). The NEAR-side `ft_transfer` is an asynchronous promise. If that promise fails and no callback exists, the ETH is irrecoverably locked at the precompile address. There is no admin escape hatch, no retry mechanism, and no way for the user to reclaim the funds.

---

### Likelihood Explanation

**Medium.** The `ft_transfer` promise fails whenever the NEAR recipient account is not registered with the eth-connector NEP-141 contract (a common user mistake — NEAR accounts must explicitly register storage). Any user who supplies an unregistered or non-existent NEAR account ID as the exit destination will trigger this path. The failure mode is silent from the EVM perspective (the EVM transaction succeeds; only the async NEAR promise fails), making it easy to miss.

---

### Recommendation

1. **Enable `error_refund` unconditionally in production** or promote it from a feature flag to always-on behavior. The refund callback path in `exit_to_near_precompile_callback` already handles the failure case correctly when the feature is present.
2. If the feature flag must remain, add a compile-time or runtime assertion that prevents deployment without `error_refund` enabled.
3. Consider pre-validating the NEAR recipient account ID before deducting ETH from the EVM balance, so the EVM call reverts early rather than proceeding with an unrecoverable async transfer.

---

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature (or use a build where it is absent).
2. Fund an EVM address with ETH on Aurora.
3. Call the `ExitToNear` precompile (address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x00` and a NEAR account ID that is **not registered** with the eth-connector (e.g., a freshly created account that has never called `storage_deposit`).
4. The EVM transaction succeeds; ETH is deducted from the caller's balance.
5. The NEAR `ft_transfer` promise fails (insufficient storage deposit / unregistered account).
6. No `exit_to_near_precompile_callback` is invoked; no refund occurs.
7. Observe that the caller's EVM balance is permanently reduced by the exit amount, and the NEAR recipient received nothing.

This matches the confirmed test behavior at `engine-tests/src/tests/erc20_connector.rs:773-775` and `engine-precompiles/src/native.rs:449-483`. [7](#0-6) [8](#0-7) [9](#0-8)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
