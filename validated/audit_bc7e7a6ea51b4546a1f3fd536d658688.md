### Title
`PriceVelocityGuardExtension` Uses `block.number` for Velocity Measurement, Making the Guard Significantly Weaker on Fast-Block Chains — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` measures elapsed time between oracle price observations using `block.number` rather than `block.timestamp`. Because the protocol targets Ethereum (~12 s/block), Base (~2 s/block), and HyperEVM (sub-second blocks), the same `maxChangePerBlockE18` setting produces a guard that is up to 6× (Base) or more (HyperEVM) weaker per unit of wall-clock time than the pool admin intended, allowing oracle price movements that should have been blocked to reach the swap execution path.

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` computes the allowed price change budget as:

```
allowedSq = maxChange² × (1 + blockDiff)
```

where `blockDiff = block.number - prevBlock`. [1](#0-0) 

Both `lastUpdateBlock` (written in `beforeSwap` and `setLastMidPrice`) and `prevBlock` (read in `beforeSwap`) are raw block numbers: [2](#0-1) 

The `PriceVelocityState` struct stores `lastUpdateBlock` as a `uint64` block number, not a timestamp: [3](#0-2) 

The protocol explicitly targets Ethereum, Base, and HyperEVM: [4](#0-3) 

Block times differ dramatically across these chains:

| Chain | Block time | Blocks per 12 s | Effective velocity budget vs Ethereum |
|---|---|---|---|
| Ethereum | ~12 s | 1 | 1× |
| Base | ~2 s | 6 | √6 ≈ 2.45× |
| HyperEVM | ~1 s | 12 | √12 ≈ 3.46× |

A pool admin who calibrates `maxChangePerBlockE18` to allow, say, 1% per Ethereum block (12 s) is actually permitting ~2.45% per 12 s on Base and ~3.46% per 12 s on HyperEVM — without any configuration change.

---

### Impact Explanation

The velocity guard is the `beforeSwap` extension gate that prevents swaps from executing when the oracle mid-price has moved faster than the pool admin's configured threshold. When the guard is too permissive, swaps proceed at oracle prices that have moved at rates the admin intended to block. LPs are exposed to adverse selection from arbitrageurs exploiting rapid oracle movements (e.g., a flash-crash or a cross-venue price spike) that the guard was specifically deployed to stop. This constitutes bad-price execution reaching the swap path and potential LP principal loss.

---

### Likelihood Explanation

The protocol is designed for multi-chain deployment. Any pool on Base or HyperEVM that uses `PriceVelocityGuardExtension` with a `maxChangePerBlockE18` calibrated for Ethereum block times is immediately affected — no special attacker action is required. Any swap during a rapid oracle price movement that would have been blocked on Ethereum will pass through on Base/HyperEVM. The trigger is a normal `swap` call.

---

### Recommendation

Replace `block.number` with `block.timestamp` throughout `PriceVelocityGuardExtension`. Store `lastUpdateTimestamp` (seconds) instead of `lastUpdateBlock`, and express `maxChangePerSecondE18` instead of `maxChangePerBlockE18`. The velocity check becomes:

```solidity
uint256 timeDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This makes the guard chain-agnostic and consistent with how all other time-sensitive logic in the codebase (oracle staleness, timelocks, confidence cooldowns) is implemented — using `block.timestamp`. [5](#0-4) 

---

### Proof of Concept

**Setup:** Deploy a pool on Base with `PriceVelocityGuardExtension`. Pool admin calls `setMaxChangePerBlock` with `maxChangePerBlockE18 = 1e16` (1% per block, calibrated for Ethereum's 12-second blocks — intended to cap at ~1% per 12 seconds of real time).

**On Ethereum (12 s/block):**
- Two swaps 12 seconds apart → `blockDiff = 1`
- `allowedSq = (1e16)² × 2 = 2e32`
- `changeE18` must be ≤ `sqrt(2) × 1e16 ≈ 1.41%`

**On Base (2 s/block):**
- Two swaps 12 seconds apart → `blockDiff = 6`
- `allowedSq = (1e16)² × 7 = 7e32`
- `changeE18` must be ≤ `sqrt(7) × 1e16 ≈ 2.65%`

A 2.5% oracle price move in 12 seconds is **blocked** on Ethereum but **passes** on Base. An arbitrageur observing a 2.5% cross-venue price discrepancy can swap against the pool on Base, extracting value from LPs, while the same trade would have been halted on Ethereum. [6](#0-5)
