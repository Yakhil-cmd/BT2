### Title
Pool Admin Bypasses Drawdown Timelock via Two-Step Timelock-Reduction Attack, Disabling Stop-Loss Protection Without LP Reaction Window — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
After reducing the pool timelock to 0 through a single timelocked proposal, the pool admin can propose **and execute** a `drawdownE6 = 0` change in the same transaction, with zero LP reaction time. This completely disables the `OracleValueStopLossExtension` stop-loss guard for all subsequent swaps.

### Finding Description
The `_afterTimelock` helper snapshots the **live** `oracleStopLossConfig[pool_].timelock` at proposal time:

```solidity
// OracleValueStopLossExtension.sol:297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

And `_requireElapsed` uses a strict `<` comparison:

```solidity
// OracleValueStopLossExtension.sol:301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

The full attack sequence:

| Step | Call | Effect |
|------|------|--------|
| 1 | `proposeOracleStopLossTimelock(pool, 0)` | `pendingTimelockExecuteAfter = T + currentTimelock` |
| 2 | Wait `currentTimelock` seconds | — |
| 3 | `executeOracleStopLossTimelock(pool)` | `oracleStopLossConfig[pool].timelock = 0` |
| 4 (same tx) | `proposeOracleStopLossDrawdown(pool, 0)` | `pendingDrawdownExecuteAfter = block.timestamp + 0 = block.timestamp` |
| 5 (same tx) | `executeOracleStopLossDrawdown(pool)` | `block.timestamp < block.timestamp` → **false** → passes; `drawdownE6 = 0` |

Steps 3–5 execute atomically in one transaction. LPs have **zero blocks** to observe the drawdown proposal and exit.

Once `drawdownE6 = 0`, `_afterSwapOracleStopLoss` returns immediately on every swap:

```solidity
// OracleValueStopLossExtension.sol:216-217
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;
```

All watermark checks are skipped. The extension is effectively dead.

### Impact Explanation
The NatSpec contract-level comment explicitly states:

> "Drawdown and decay changes are timelocked so LPs can react; monitor at least as often as the timelock or trust the pool admin."

LPs who chose this pool based on the stop-loss guarantee receive zero reaction time for the drawdown change. With stop-loss disabled, swaps that cause unlimited per-share value loss in any bin are no longer blocked. Any actor (including the admin) can then execute swaps that drain LP value without the extension reverting. This is a direct admin-boundary break: the pool admin is supposed to be semi-trusted **only inside timelocks**, and this sequence bypasses the drawdown timelock entirely.

### Likelihood Explanation
The attack requires only the pool admin role and one timelock wait (the timelock reduction). No oracle admin, factory owner, or external collusion is needed. The sequence is fully on-chain and atomic in the final step.

### Recommendation
`proposeOracleStopLossDrawdown` must record the timelock value **at proposal time** and enforce it at execution, independent of the live timelock. One approach: store `pendingDrawdownTimelockSnapshot` alongside `pendingDrawdownExecuteAfter` and require that the elapsed duration satisfies the **snapshot** value, not the current one. Alternatively, enforce a minimum timelock floor that cannot be reduced below a protocol constant (e.g., 24 hours).
