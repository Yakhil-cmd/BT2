### Title
Unauthorized Burn of Victim's `evm/<callerContract>` Bank Tokens via Bank Precompile — (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `BankContract.Run` handler for `BurnMethodName` accepts an arbitrary `recipient` address as the burn target and derives the denom from `contract.Caller()`, but **never checks that the address being burned from equals the caller**. Any EVM contract can burn `evm/<itself>` tokens held by any victim address without the victim's consent.

### Finding Description

In `bank.go`, the `MintMethodName`/`BurnMethodName` branch unpacks `args[0]` as the target address and `args[1]` as the amount: [1](#0-0) 

The denom is derived from the **caller** (the invoking contract), not from the target address: [2](#0-1) 

For the burn path, `SendCoinsFromAccountToModule` is called with `addr` = the caller-supplied `recipient` argument — not the caller itself: [3](#0-2) 

There is no guard of the form `if addr != sdk.AccAddress(contract.Caller().Bytes()) { return error }` anywhere in this branch. The only check performed is `checkBlockedAddr`, which only rejects module accounts: [4](#0-3) 

The same missing authorization check also affects `TransferMethodName` (line 192), where `from` is caller-supplied with no verification against `contract.Caller()`. [5](#0-4) 

### Impact Explanation

- Deploy contract A at address `0xA`.
- Victim acquires a nonzero balance of denom `evm/0xA` (e.g., via a prior `mint` call or transfer).
- Contract A calls `bank.burn(victimAddr, amount)`.
- The precompile computes `denom = "evm/0xA"`, `addr = victimAddr`, and executes `SendCoinsFromAccountToModule(ctx, victimAddr, evm_module, {evm/0xA: amount})` followed by `BurnCoins`.
- Victim's `evm/0xA` balance is destroyed with no signature or approval from the victim.

This is **unauthorized destruction of user assets** — a Critical impact under the allowed scope (unauthorized burn of precompile-controlled assets).

### Likelihood Explanation

The attack requires only:
1. Deploying a contract (permissionless on Cronos).
2. A victim holding a nonzero balance of `evm/<attackerContract>` — which the attacker can arrange by minting to the victim first, or by targeting any existing holder.

No privileged access, leaked keys, or external assumptions are needed.

### Recommendation

Add an authorization check in the burn branch ensuring the address being burned from is the caller:

```go
// In the burn branch, before SendCoinsFromAccountToModule:
if !addr.Equals(sdk.AccAddress(contract.Caller().Bytes())) {
    return nil, errors.New("burn: caller is not the token holder")
}
```

Apply the same fix to `TransferMethodName` for the `sender`/`from` address.

### Proof of Concept

1. Deploy contract A (`0xA`) on Cronos EVM.
2. Call `bank.mint(victimAddr, 1000)` from A → victim now holds `1000 evm/0xA`.
3. Call `bank.burn(victimAddr, 1000)` from A.
4. Query `bank.balanceOf(0xA, victimAddr)` → returns `0`.
5. Victim's balance was destroyed without any transaction signed by the victim.

#Vulnerability found.
