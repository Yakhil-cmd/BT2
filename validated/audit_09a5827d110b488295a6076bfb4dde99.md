### Title
Stale `AdminToPermissions` entries persist after `CronosAdmin` rotation, allowing previously-authorized addresses to bypass the new admin's authority - (File: `x/cronos/keeper/msg_server.go`, `x/cronos/keeper/permissions.go`, `x/cronos/keeper/params.go`)

---

### Summary

When governance rotates `CronosAdmin` via `MsgUpdateParams`, the `AdminToPermissions` KVStore entries granted by the old admin are never cleared. Any address previously granted `CanChangeTokenMapping` by the old admin retains that permission indefinitely, allowing it to call `MsgUpdateTokenMapping` without the new admin's knowledge or authorization — a direct bypass of Cronos admin/governance authority over token mapping state.

---

### Finding Description

The Cronos module maintains two separate, independent permission stores:

1. **`CronosAdmin`** — stored inside `Params` in the KVStore; the single address that has all permissions and is the only one allowed to call `MsgUpdatePermissions`.
2. **`AdminToPermissions` mapping** — a separate KVStore prefix (`prefixAdminToPermissions`) that maps arbitrary addresses to a `uint64` permission bitmask (`CanChangeTokenMapping = 1`, `CanTurnBridge = 2`).

`HasPermission` in `x/cronos/keeper/permissions.go` grants access if **either** condition is true:

```go
// HasPermission check if an account has a specific permission.
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
    admin := k.GetParams(ctx).CronosAdmin
    for _, account := range accounts {
        if admin == account.String() {   // ← current admin check
            return true
        }
        permission := k.GetPermissions(ctx, account)  // ← KVStore check
        if permission&permissionsToCheck == permissionsToCheck {
            return true
        }
    }
    return false
}
```

When governance rotates the admin via `MsgUpdateParams`, `UpdateParams` calls `SetParams`, which only writes the new `Params` blob:

```go
func (k Keeper) SetParams(ctx sdk.Context, params types.Params) error {
    if err := params.Validate(); err != nil {
        return err
    }
    store := ctx.KVStore(k.storeKey)
    bz := k.cdc.MustMarshal(&params)
    store.Set(types.ParamsKey, bz)   // ← only params written; AdminToPermissions untouched
    return nil
}
```

No code in `SetParams`, `UpdateParams`, or anywhere in the admin-rotation path iterates over or clears the `AdminToPermissions` prefix. All entries written by the old admin survive the rotation intact.

The `UpdatePermissions` handler enforces that only the **current** admin can grant/revoke permissions:

```go
func (k msgServer) UpdatePermissions(...) {
    admin := k.Keeper.GetParams(ctx).CronosAdmin
    if admin != msg.From {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, ...)
    }
    k.SetPermissions(ctx, acc, msg.Permissions)
}
```

After rotation, the new admin B can grant new permissions, but has no automatic mechanism to discover or revoke the stale entries left by old admin A. The new admin would need to know every address A ever permissioned and manually revoke each one.

---

### Impact Explanation

**High — Bypass of Cronos admin/governance authority and corruption of token mapping state.**

An address X that was legitimately permissioned by old admin A retains `CanChangeTokenMapping` after A is replaced by B. X can then call `MsgUpdateTokenMapping` without B's authorization:

- **Register a new denom → malicious contract mapping**: For any IBC denom not yet mapped, X can register a malicious ERC20 contract. When IBC vouchers of that denom subsequently arrive via `OnRecvVouchers` → `ConvertVouchersToEvmCoins`, the module will interact with X's malicious contract to mint EVM tokens, potentially redirecting or locking user funds.
- **Delete an existing denom → contract mapping**: For non-source denoms, X can call `MsgUpdateTokenMapping` with an empty contract field, triggering `DeleteExternalContractForDenom`. This breaks the IBC auto-conversion flow for that denom, causing arriving IBC vouchers to remain as native tokens rather than being converted to EVM tokens, permanently disrupting the bridge for affected users.

Both actions are gated solely by `HasPermission(CanChangeTokenMapping)`, which X passes via the stale KVStore entry.

---

### Likelihood Explanation

Admin rotation via governance (`MsgUpdateParams`) is an explicitly documented, expected operational event — the spec states `CronosAdmin` "Can be updated at runtime." Any address that was ever granted `CanChangeTokenMapping` by a previous admin becomes a permanently privileged actor that the new admin cannot easily discover or revoke without out-of-band knowledge of the old admin's actions.

---

### Recommendation

In `SetParams` (or in `UpdateParams` before calling `SetParams`), detect when `CronosAdmin` is changing and clear all `AdminToPermissions` entries:

```go
func (k Keeper) SetParams(ctx sdk.Context, params types.Params) error {
    if err := params.Validate(); err != nil {
        return err
    }
    // If admin is rotating, purge all delegated permissions granted by the old admin.
    oldParams := k.GetParams(ctx)
    if oldParams.CronosAdmin != params.CronosAdmin {
        k.clearAllPermissions(ctx)
    }
    store := ctx.KVStore(k.storeKey)
    bz := k.cdc.MustMarshal(&params)
    store.Set(types.ParamsKey, bz)
    return nil
}
```

Where `clearAllPermissions` iterates over `KeyPrefixAdminToPermissions` and deletes all entries. Alternatively, scope the `AdminToPermissions` key to include the granting admin's address (analogous to the Open Dollar fix of adding `_owner` as a key field), so that permissions are automatically invalidated when the admin changes.

---

### Proof of Concept

**Step 1**: Old admin A grants address X `CanChangeTokenMapping`:
```
MsgUpdatePermissions { from: A, address: X, permissions: 1 }
```
`SetPermissions` writes `AdminToPermissionsKey(X) → 1` to KVStore.

**Step 2**: Governance passes `MsgUpdateParams` rotating `CronosAdmin` from A to B:
```
MsgUpdateParams { authority: gov_module, params: { cronos_admin: B, ... } }
```
`SetParams` writes new `Params` blob. `AdminToPermissionsKey(X) → 1` is **not touched**.

**Step 3**: X (no longer authorized by new admin B) calls:
```
MsgUpdateTokenMapping { sender: X, denom: "ibc/HASH", contract: "0xMalicious..." }
```

**Step 4**: `UpdateTokenMapping` calls `HasPermission(ctx, [X], CanChangeTokenMapping)`:
- `admin = B`, `B != X` → first check fails
- `GetPermissions(X)` returns `1` (stale entry) → `1 & 1 == 1` → **returns true**

**Step 5**: `RegisterOrUpdateTokenMapping` registers `ibc/HASH → 0xMalicious`. The new admin B never authorized this.

**Step 6**: When IBC vouchers of `ibc/HASH` arrive, `OnRecvVouchers` → `ConvertVouchersToEvmCoins` uses `0xMalicious` as the ERC20 contract, enabling X to steal or lock user funds during conversion.

func (k msgServer) UpdatePermissions(goCtx context.Context, msg *types.MsgUpdatePermissions) (*types.MsgUpdatePermissionsResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)
	admin := k.Keeper.GetParams(ctx).CronosAdmin
	// if admin is empty, no sender could be equal to it
	if admin != msg.From {
		return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
	}
	acc, err := sdk.AccAddressFromBech32(msg.Address)
	if err != nil {
		return nil, err
	}
	k.SetPermissions(ctx, acc, msg.Permissions)

	return &types.MsgUpdatePermissionsResponse{}, nil
}
