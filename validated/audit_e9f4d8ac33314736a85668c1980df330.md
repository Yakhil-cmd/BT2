### Title
`DepositAllowlistExtension` Checks `owner` Instead of `sender`, Allowing Any Address to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller/payer) and only validates `owner` (the position recipient). Because `pool.addLiquidity` accepts any `owner` address from any caller, an unlisted address can deposit tokens into a curated pool by nominating an allowlisted address as the position owner.

### Finding Description

The full call chain is:

**Step 1 — `MetricOmmPool.addLiquidity`** passes `msg.sender` as `sender` and the caller-supplied `owner` as `owner`:

```solidity
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
``` [1](#0-0) 

**Step 2 — `ExtensionCalling._beforeAddLiquidity`** forwards both to the extension:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeAddLiquidity, (sender, owner, salt, deltas, extensionData))
``` [2](#0-1) 

**Step 3 — `DepositAllowlistExtension.beforeAddLiquidity`** drops the `sender` argument entirely (unnamed `address,`) and only checks `owner`:

```solidity
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    ...
}
``` [3](#0-2) 

Here `msg.sender` is the pool, so the check resolves to `allowedDepositor[pool][owner]`. The actual caller/payer (`sender`) is never consulted.

`pool.addLiquidity` has no restriction on who may call it or what `owner` they supply: [4](#0-3) 

The `MetricOmmPoolLiquidityAdder` explicitly documents and tests that `owner` may differ from `msg.sender` (the payer): [5](#0-4) 

### Impact Explanation

The `DepositAllowlistExtension` is documented as "Gates `addLiquidity` by depositor address, per pool." [6](#0-5) 

The invariant is that only allowlisted addresses may pay tokens into the pool. Because the check is on `owner` (position recipient) rather than `sender` (actual payer), any unlisted address B can:

1. Call `pool.addLiquidity(owner=A, ...)` where A is allowlisted.
2. The extension sees `allowedDepositor[pool][A] = true` → passes.
3. B pays the tokens via the modify-liquidity callback; the position is credited to A.
4. B has deposited tokens into the pool despite being unlisted.

This completely defeats the purpose of the allowlist. Pool admins deploying this extension to enforce KYC, regulatory, or curation controls receive no protection against unlisted depositors.

### Likelihood Explanation

The attack requires only a direct call to `pool.addLiquidity` with a known allowlisted address as `owner`. No special privileges, flash loans, or oracle manipulation are needed. The `MetricOmmPoolLiquidityAdder` even provides a ready-made path (`addLiquidityExactShares(pool, owner, ...)`) that explicitly supports `owner != msg.sender`. [7](#0-6) 

### Recommendation

Change `beforeAddLiquidity` to validate `sender` (the actual caller/payer) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

Update `setAllowedToDeposit`, `isAllowedToDeposit`, and the `AllowedToDepositSet` event parameter name from `depositor` to `sender` for consistency. [8](#0-7) 

### Proof of Concept

```solidity
// Pool configured with DepositAllowlistExtension; only address A is allowlisted.
// Address B is NOT allowlisted.

// B calls pool directly:
vm.prank(B);
pool.addLiquidity(
    A,          // owner = allowlisted address A
    salt,
    deltas,
    callbackData,
    extensionData
);
// Extension checks allowedDepositor[pool][A] == true → passes.
// B pays tokens via callback; position credited to A.
// B has deposited into the curated pool despite being unlisted.
assertGt(pool.positionBinShares(A, salt, binIdx), 0);
```
