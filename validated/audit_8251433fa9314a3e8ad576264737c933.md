Audit Report

## Title
Markdown Injection via Unsanitized UTF-8 Memo in ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` method in the shared ICRC-21 consent-message library interpolates a transaction memo decoded as UTF-8 directly into a single-backtick inline code span in a `GenericDisplayMessage` without sanitizing backtick characters. Any unprivileged transaction sender (or dapp constructing transactions on behalf of users) can embed a backtick in the memo to terminate the code span and inject arbitrary Markdown — including fake headings and fake transaction fields — into the security-critical consent dialog shown by wallets. The `token_name`/`token_symbol` vectors cited in the report require the attacker to be the canister developer, who already has full control over the returned consent message and can fabricate it entirely without exploiting this library; those vectors do not constitute an independent vulnerability.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs`, `add_memo` decodes the raw ICRC-1 memo bytes as UTF-8 and places the result verbatim inside a single-backtick inline code span:

```rust
// line 288-294
let memo_str = match std::str::from_utf8(memo.as_slice()) {
    Ok(valid_str) => valid_str.to_string(),
    Err(_) => hex::encode(memo.as_slice()),
};
// line 293-294
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

A backtick anywhere in `memo_str` closes the inline code span at that point. Everything following it is rendered as raw Markdown by any wallet that displays `GenericDisplayMessage` with a Markdown renderer. The memo is the last field appended in the `build` method (after `add_intent`, `add_account`, `add_amount`, `add_fee`), so the injected content appears at the end of the consent message. A crafted memo such as:

```
`

# You are sending 1000 ICP to attacker

**To:**
`attacker_address
```

produces a rendered message that appends a visually prominent fake heading and fake "To" field after the legitimate fields. On wallets with limited viewport or paginated display, the injected section may dominate what the user sees before approving.

The ckBTC and ckDOGE minters explicitly guard against this same class of attack with `validate_address` and a comment that directly names Markdown-injection as the risk being mitigated (`rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs` lines 195–209; `rs/dogecoin/ckdoge/minter/src/updates/icrc21.rs` lines 196–210). The shared ICRC-1/ICRC-2 library applies no equivalent guard on the memo path.

## Impact Explanation
A malicious dapp (the standard DeFi pattern where the dapp constructs transactions on behalf of users) can set the memo to a crafted UTF-8 string. The wallet calls `icrc21_canister_call_consent_message` on the honest ledger canister, which uses the shared library and returns a `GenericDisplayMessage` containing the injected Markdown. The wallet renders it and shows the user a spoofed consent dialog. If the user approves based on the injected content, they authorize a transaction with parameters different from what was displayed. This constitutes moderate user-funds/security impact — the ICRC-21 consent message is the wallet's primary mechanism for showing users what they are signing, and its integrity is directly undermined. This maps to the **Medium** bounty tier: the attack is straightforward to execute but requires the user to be misled by injected content that appears after the legitimate fields.

## Likelihood Explanation
Triggering the vulnerability requires only the ability to submit an ICRC-1 transfer, approve, or transfer-from with a crafted memo — a standard unprivileged ingress call available to any dapp or direct caller. No special privileges, cycles beyond normal fees, or canister deployment are needed. Any wallet that renders `GenericDisplayMessage` as Markdown is affected. The attack is repeatable and silent.

## Recommendation
1. In `add_memo`, escape every backtick in `memo_str` before interpolation: replace `` ` `` with `` \` ``, or switch to a fenced code block (` ``` `…` ``` `) which is immune to single-backtick injection.
2. Apply the same pattern already used in the ckBTC/ckDOGE minters: validate or encode any user-supplied string before it is interpolated into a `GenericDisplayMessage`.
3. For structured data, prefer `FieldsDisplayMessage` (`ConsentMessage::FieldsDisplayMessage`), which carries typed `Value` entries that wallets render without Markdown interpretation, eliminating the injection surface entirely.

## Proof of Concept
```rust
// Craft a memo that breaks out of the backtick code span
let malicious_memo = b"`\n\n# You are sending 1000 ICP to attacker\n\n**To:**\n`attacker_address".to_vec();

let transfer_args = TransferArg {
    memo: Some(Memo::from(malicious_memo)),
    amount: Nat::from(1_u64),   // actual amount: 1 e8s
    to: victim_account,
    ..Default::default()
};

// Wallet calls icrc21_canister_call_consent_message with these args.
// add_memo (responses.rs line 294) produces:
//   \n\n**Memo:**\n``\n\n# You are sending 1000 ICP to attacker\n\n**To:**\n`attacker_address`
// Rendered Markdown appends a fake heading and fake "To" field after the real fields.
```

A deterministic integration test using PocketIC can encode `transfer_args` as Candid, pass it to `icrc21_canister_call_consent_message` on a standard ICRC-1 ledger, extract the `GenericDisplayMessage` string, and assert that it contains the injected heading — confirming the vulnerability without any mainnet interaction.
