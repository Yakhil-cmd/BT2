### Title
Pending Price-Provider Proposal Has No Expiration, Allowing Admin to Execute a Stale Oracle Swap at Any Opportune Moment After the Timelock Elapses — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` records a pending oracle change with a minimum execution timestamp (`block.timestamp + timelock`) but imposes **no maximum deadline**. Once the timelock elapses the proposal is valid forever. A pool admin can propose a replacement oracle, let the timelock pass, wait until the pool holds maximum TVL (or until new LPs who are unaware of the stale proposal have deposited), and then execute the change — defeating the entire protective purpose of the timelock.

---

### Finding Description

`proposePoolPriceProvider` writes:

```solidity
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool]             = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
``` [1](#0-0) 

`executePoolPriceProviderUpdate` enforces only the lower bound:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
``` [2](#0-1) 

There is no upper-bound check and no `cancelPoolPriceProviderProposal` function anywhere in the factory interface or implementation. [3](#0-2) 

The only validation applied to the proposed provider — both at proposal time and at execution time — is a token-pair identity check:

```solidity
if (IPriceProvider(priceProvider).token0() != token0 || IPriceProvider(priceProvider).token1() != token1)
    revert PriceProviderTokenMismatch();
``` [4](#0-3) 

A provider that returns the correct token pair but adversarial bid/ask prices passes this check.

---

### Impact Explanation

The timelock exists to give LPs a bounded warning window: see the proposal, decide whether to exit, and do so before the new oracle goes live. Without an expiration deadline:

1. The admin proposes a replacement oracle (possibly one that returns manipulated prices but has the correct `token0()`/`token1()`).
2. The timelock elapses; some LPs exit.
3. New LPs deposit over subsequent days or weeks, unaware of the stale pending proposal (it is not surfaced in any on-chain invariant they can rely on).
4. The admin calls `executePoolPriceProviderUpdate` at the moment of maximum TVL or minimum LP vigilance.
5. The pool's active oracle is replaced; subsequent swaps execute against adversarial bid/ask quotes, allowing the admin (or a colluding trader) to drain LP principal through bad-price execution.

This satisfies the **bad-price execution** and **admin-boundary break** allowed impacts: the timelock is the only constraint on the admin's ability to rotate the oracle, and the missing expiration nullifies it.

---

### Likelihood Explanation

The trigger requires a malicious or compromised pool admin. Pool admins are explicitly classified as "semi-trusted only inside caps and timelocks." The timelock is the sole mechanism bounding admin oracle power; its protective intent is fully defeated by the missing expiration. Any pool with a mutable price provider (`priceProviderTimelock != type(uint256).max`) is affected. [5](#0-4) 

---

### Recommendation

Add an expiration deadline to the proposal. When `executePoolPriceProviderUpdate` is called, revert if the current timestamp exceeds `pendingPriceProviderExecuteAfter[pool] + PROPOSAL_EXPIRY`. A reasonable value is `2 × timelock` or a fixed constant (e.g., 7 days). Additionally, expose a `cancelPoolPriceProviderProposal` function so the admin can explicitly withdraw a stale proposal, and emit an event on cancellation so off-chain monitors can track proposal lifecycle.

```solidity
// In executePoolPriceProviderUpdate:
uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
if (block.timestamp > execAfter + PROPOSAL_EXPIRY) revert PriceProviderProposalExpired(...);
```

### Proof of Concept

```
T=0      Admin calls proposePoolPriceProvider(pool, maliciousOracle)
         → pendingPriceProvider[pool]             = maliciousOracle
         → pendingPriceProviderExecuteAfter[pool] = T + timelock

T=timelock  Timelock elapses. Proposal is now executable — and remains so forever.
            Some LPs exit after seeing the event.

T=timelock+30days  New LPs deposit; they see no active proposal in any enforced state.
                   Pool TVL is at its highest.

T=timelock+30days+1  Admin calls executePoolPriceProviderUpdate(pool).
                     _validatePriceProvider passes (token0/token1 match).
                     Pool oracle is switched to maliciousOracle.
                     Next swap: maliciousOracle returns an adversarial bid/ask.
                     Trader (colluding with admin) receives more token1 than the
                     legitimate oracle would permit, draining LP token1 balances.
``` [6](#0-5)

    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.

    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
