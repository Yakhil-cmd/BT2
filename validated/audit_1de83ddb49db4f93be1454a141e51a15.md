### Title
`native_price` and `pubdata_price` Are Not Committed to ZK Public Input, Enabling Forward/Proving Divergence — (`zk_ee/src/system/metadata/zk_metadata.rs`, `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/public_input.rs`)

---

### Summary

`BlockMetadataFromOracle` carries `native_price` and `pubdata_price` fields that are consumed from the oracle during block execution and directly control native-resource accounting for every transaction. Neither field is included in the ZK public input (`BatchPublicInput` / `ChainStateCommitment` / `BatchOutput`), nor in the block header. A prover who controls oracle inputs can therefore supply different values during proving than the sequencer used during forward execution, producing a valid proof for a state transition that never actually occurred.

---

### Finding Description

`BlockMetadataFromOracle` is the oracle-sourced struct that drives all per-block fee parameters:

```
pub struct BlockMetadataFromOracle {
    pub native_price: U256,   // cost of one native-resource unit
    pub pubdata_price: U256,  // cost of one pubdata byte
    ...
}
``` [1](#0-0) 

These two values are the sole inputs to the critical ratios used throughout transaction validation and execution:

- `native_per_gas = ceil(gas_price / native_price)` — determines how many native cycles a user's gas budget buys.
- `native_per_pubdata = pubdata_price / native_price` — determines the native cost of each pubdata byte. [2](#0-1) 

A transaction that runs out of native resources is **reverted** (or rejected as invalid during validation). The entire execution outcome — success vs. revert — can flip depending on these two values.

The ZK public input is assembled in `post_tx_op_proving_singleblock_batch.rs` and `public_input.rs`. The `ChainStateCommitment` commits only to `state_root`, `next_free_slot`, `block_number`, `last_256_block_hashes_blake`, and `last_block_timestamp`: [3](#0-2) 

The `BatchOutput` commits to `chain_id`, timestamps, DA commitment, L1 tx hashes, L2 logs root, and upgrade tx hash: [4](#0-3) 

Neither `native_price` nor `pubdata_price` appears anywhere in the public input hash. The `form_block_header` function even carries an explicit TODO acknowledging the gap:

```rust
// TODO: add pubdata price and native price
``` [5](#0-4) 

The project documentation confirms this is a known open gap:

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. Currently it misses `gas_per_pubdata` and `native_price`, but we already working on design and implementation to solve this issue." [6](#0-5) 

In proving mode the oracle is CSR-based; the prover supplies all oracle responses, including the `BlockMetadataFromOracle` payload. Because the L1 verifier only checks the public input hash, it cannot detect that the prover used different `native_price`/`pubdata_price` values than the sequencer.

---

### Impact Explanation

**Forward/proving divergence leading to invalid state transitions accepted on L1.**

Concrete scenario:

1. Sequencer runs forward execution with `native_price = P` (e.g., 100). Transactions whose `gas_price / P` ratio is too low run out of native resources and **revert**. The resulting state `S_seq` reflects these reverts.
2. Prover runs proving execution with `native_price = 1`. Now `native_per_gas = gas_price`, giving every transaction a vastly larger native budget. Transactions that reverted in step 1 now **succeed**. The proven state `S_prove ≠ S_seq`.
3. The prover submits a proof for `S_prove`. The L1 verifier checks only the public input hash, which does not include `native_price`. The proof is accepted.
4. The on-chain state is updated to `S_prove`, which was never the actual sequenced state.

Consequences include:
- Transactions that should have been reverted (e.g., out-of-native contract calls) are recorded as successful, with their storage writes and ETH transfers committed on-chain.
- Conversely, a prover could set `native_price` to `u64::MAX / gas_price + 1` to force `native_per_gas = 0`, triggering the "unlimited native" path and making all transactions succeed regardless of actual proving cost — or set it extremely high to revert transactions that legitimately succeeded.
- Fee accounting is corrupted: `gas_used` is derived from native consumption, so the operator can manipulate how much fee is charged to users.

This is a **critical state-transition bug** / **underconstraint** in the ZK proof system.

---

### Likelihood Explanation

**Medium.** The attack requires the prover to supply a different `BlockMetadataFromOracle` during proving than the sequencer used during forward execution. In ZKsync OS's architecture the prover controls all oracle inputs in proving mode via CSR reads. The prover is explicitly listed as an attacker role in the Immunefi scope ("prover/forward execution input"). No leaked key, governance majority, or external oracle lie is required — only the ability to provide oracle data during proving, which is the prover's normal function.

---

### Recommendation

Include `native_price` and `pubdata_price` (and `pubdata_limit`) in the ZK public input. The most natural place is either:

1. **`ChainStateCommitment`** — add them as fields so they are committed before and after each block, forcing the prover to use the same values as the sequencer.
2. **`BatchOutput`** — add them alongside `chain_id` and timestamps so the settlement layer can validate them.
3. **Block header `extra_data`** — as the documentation already proposes, encode `native_price` and `pubdata_price` in the block header's `extra_data` field and include the header hash in the public input.

Until this is fixed, a malicious prover can generate valid proofs for state transitions that were never sequenced.

---

### Proof of Concept

**Step 1 — Confirm the fields are absent from the public input.**

Inspect `BatchPublicInput::hash()`:

```rust
pub fn hash(&self) -> [u8; 32] {
    let mut hasher = Keccak256::new();
    hasher.update(self.state_before.as_u8_ref());   // ChainStateCommitment hash
    hasher.update(self.state_after.as_u8_ref());    // ChainStateCommitment hash
    hasher.update(self.batch_output.as_u8_ref());   // BatchOutput hash
    hasher.finalize()
}
``` [7](#0-6) 

`BatchOutput::hash()` does not include `native_price` or `pubdata_price`: [8](#0-7) 

`ChainStateCommitment::hash()` does not include them either: [9](#0-8) 

**Step 2 — Confirm the fields control transaction success/failure.**

In `validation_impl.rs`, `native_per_gas` is computed as `ceil(gas_price / native_price)`. If `native_price` is large, `native_per_gas` is small, and the transaction's native budget is exhausted quickly, causing a revert: [10](#0-9) 

**Step 3 — Confirm the prover supplies `BlockMetadataFromOracle` via oracle.**

In proving mode, `BlockMetadataFromOracle` is deserialized from the oracle stream. The prover controls this stream: [11](#0-10) 

**Step 4 — Construct a diverging block.**

Using the existing test framework, run two executions of the same transaction set:

```rust
// Forward run: native_price = 1000 → many txs revert (out-of-native)
let forward_context = BlockContext { native_price: U256::from(1000), ... };
let forward_output = chain.execute_block(txs.clone(), Some(forward_context));

// Proving run: native_price = 1 → same txs succeed
let proving_context = BlockContext { native_price: U256::from(1), ... };
let proving_output = chain.execute_block(txs.clone(), Some(proving_context));

// States differ; proving_output produces a valid proof accepted by L1
assert_ne!(forward_output.state_root, proving_output.state_root);
```

Because `native_price` is absent from the public input, both runs produce proofs with the same public input hash for different state roots — the L1 verifier cannot distinguish them.

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }

    /// - multichain root - commitment to logs emitted on chains that settle on the current.
    pub l2_logs_tree_root: Bytes32,
    /// Protocol upgrade tx hash (0 if there wasn't)
    pub upgrade_tx_hash: Bytes32,

    pub settlement_layer_chain_id: U256,

impl BatchOutput {
    ///
    /// Calculate keccak256 hash of public input
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = Keccak256::new();
        hasher.update(self.chain_id.to_be_bytes::<32>());
        hasher.update(&self.first_block_timestamp.to_be_bytes());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        // Encode DA commitment scheme as U256 BE
        hasher.update([0u8; 31]);
        hasher.update([self.da_commitment_scheme as u8]);
        hasher.update(self.pubdata_commitment.as_u8_ref());
        hasher.update(self.number_of_layer_1_txs.to_be_bytes::<32>());
        hasher.update(self.number_of_layer_2_txs.to_be_bytes::<32>());
        hasher.update(self.priority_operations_hash.as_u8_ref());
        hasher.update(self.l2_logs_tree_root.as_u8_ref());
        hasher.update(self.upgrade_tx_hash.as_u8_ref());
        hasher.update(self.interop_roots_rolling_hash.as_u8_ref());
        hasher.update(self.settlement_layer_chain_id.to_be_bytes::<32>());
        hasher.finalize()
    }
