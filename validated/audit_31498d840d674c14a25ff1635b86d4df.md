### Title
Asymmetric Pause State Between `EXIT_TO_NEAR` Precompile and ETH Connector `engine_withdraw` Causes Permanent Fund Freeze - (File: `engine/src/pausables.rs`, `engine-precompiles/src/native.rs`)

---

### Summary

The Aurora Engine exposes two **independently controlled** pause mechanisms on two separate contracts: `pause_precompiles` on the Aurora Engine contract (EVM side) and `pa_pause_feature` on the ETH Connector contract (NEAR side). When the ETH Connector's `engine_withdraw` feature is paused while the `EXIT_TO_NEAR` precompile remains unpaused, any user who calls `EXIT_TO_NEAR` will have their EVM-side tokens burned with no corresponding NEAR-side transfer and no refund path, resulting in permanent fund freeze.

---

### Finding Description

The Aurora Engine defines two separate, independently operated pause surfaces:

**Surface 1 — EVM-side precompile pause** (`engine/src/pausables.rs`):

`PrecompileFlags` tracks two pausable precompiles via a bitmask stored in Aurora Engine contract storage: [1](#0-0) 

The pause check is enforced inside the EVM executor's `PrecompileSet::execute`: [2](#0-1) 

**Surface 2 — NEAR-side ETH Connector feature pause** (`pa_pause_feature`):

The ETH Connector contract (a separate NEAR account) exposes its own `pa_pause_feature` method that independently pauses named features such as `"deposit"` and `"engine_withdraw"`. This is demonstrated by the test: [3](#0-2) 

**The asymmetric gap:**

When a user calls the `EXIT_TO_NEAR` precompile from the EVM, the precompile's `run()` method burns the user's EVM-side tokens and schedules a cross-contract promise to call `engine_withdraw` on the ETH Connector: [4](#0-3) 

The callback `exit_to_near_precompile_callback` only issues a refund if the `error_refund` compile-time feature is enabled **and** `args.refund` is `Some`: [5](#0-4) 

The `refund` field is set to `None` when the `error_refund` feature is absent: [6](#0-5) 

If `engine_withdraw` is paused on the ETH Connector while `EXIT_TO_NEAR` is not paused on the Aurora Engine, the sequence is:

1. User submits an EVM transaction calling `EXIT_TO_NEAR` — succeeds, tokens burned from EVM balance.
2. Aurora Engine schedules a promise to `engine_withdraw` on the ETH Connector.
3. ETH Connector rejects the call with `"Pausable: Method is paused"`.
4. Callback fires; `args.refund` is `None` (no `error_refund` feature) → no refund is issued.
5. Tokens are permanently destroyed.

The two pause states are controlled by different admin calls (`pause_precompiles` on Aurora Engine vs. `pa_pause_feature` on the ETH Connector), with no enforcement that they must be toggled atomically. [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any EVM user who calls `EXIT_TO_NEAR` (for base ETH or ERC-20 tokens) during the window where `engine_withdraw` is paused on the ETH Connector but `EXIT_TO_NEAR` is not paused on the Aurora Engine will have their tokens permanently burned with no recovery path. ETH and all bridged ERC-20 tokens are in scope.

---

### Likelihood Explanation

**Medium.** The scenario requires an admin to pause `engine_withdraw` on the ETH Connector (a plausible emergency or migration action) without simultaneously pausing the `EXIT_TO_NEAR` precompile on the Aurora Engine. Because the two pause mechanisms are on different contracts, invoked by different method names, and have no shared atomicity guarantee, an operator acting under time pressure could easily pause only one side. The window between the two admin calls is sufficient for users to lose funds.

---

### Recommendation

1. Enforce atomic pause coordination: when `engine_withdraw` is paused on the ETH Connector, the `EXIT_TO_NEAR` precompile must also be paused on the Aurora Engine in the same transaction batch (or via a single coordinating admin contract).
2. Unconditionally enable the `error_refund` feature in production builds so that a failed `engine_withdraw` promise always triggers a refund of the burned EVM tokens, regardless of pause state.
3. Add a runtime guard inside `EXIT_TO_NEAR::run()` that reads the ETH Connector's pause state before burning tokens, rejecting the call early if `engine_withdraw` is paused.

---

### Proof of Concept

1. Admin calls `pa_pause_feature({"key": "engine_withdraw"})` on the ETH Connector contract.
2. Admin does **not** call `pause_precompiles` on the Aurora Engine (or calls it later).
3. User submits an EVM transaction to the `EXIT_TO_NEAR` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with their ETH amount and a NEAR recipient.
4. Aurora Engine executes the precompile — no pause flag set for `EXIT_TO_NEAR`, so execution proceeds. User's EVM ETH balance is decremented.
5. Aurora Engine schedules a promise: `engine_withdraw` on the ETH Connector.
6. ETH Connector rejects with `"Pausable: Method is paused"`.
7. `exit_to_near_precompile_callback` fires; `args.refund` is `None`; no refund is issued.
8. User's ETH is gone from the EVM and never arrives on NEAR — permanent freeze. [8](#0-7) [9](#0-8)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
