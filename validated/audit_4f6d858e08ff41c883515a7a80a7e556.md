Audit Report

## Title
Missing Minimum Fee Rate Floor in `DogecoinFeeEstimator::estimate_nth_fee` Can Cause ckDOGE Withdrawal Transactions to Become Permanently Stuck - (File: `rs/dogecoin/ckdoge/minter/src/fees/mod.rs`)

## Summary
`DogecoinFeeEstimator::estimate_nth_fee()` returns the raw fee percentile from the Dogecoin canister with no minimum floor, unlike `BitcoinFeeEstimator::estimate_nth_fee()` which applies a hard-coded `minimum_fee_per_vbyte()` floor added after a documented production incident on 2025-06-21. If the Dogecoin canister returns anomalously low fee percentiles, the ckDOGE minter will build and broadcast withdrawal transactions with a near-zero fee rate, causing them to never be mined. The shared resubmission path can then enter a deterministic panic, freezing all pending ckDOGE withdrawals and leaving users with burned ckDOGE but undelivered DOGE.

## Finding Description

**Root cause:** `DogecoinFeeEstimator::estimate_nth_fee()` returns the raw oracle value with no floor:

```rust
// rs/dogecoin/ckdoge/minter/src/fees/mod.rs, lines 53-66
fn estimate_nth_fee(&self, fee_percentiles: &[FeeRate], nth: usize) -> Option<FeeRate> {
    match &self.network {
        Network::Mainnet => {
            if fee_percentiles.len() < 100 || nth >= 100 {
                return None;
            }
            Some(fee_percentiles[nth])  // No floor applied
        }
        Network::Regtest => Some(DEFAULT_REGTEST_FEE),
    }
}
```

By contrast, `BitcoinFeeEstimator::estimate_nth_fee()` applies `f.max(self.minimum_fee_per_vbyte())` (1,500 millis/byte on Mainnet) as a stop-gap fix after the 2025-06-21 incident:

```rust
// rs/bitcoin/ckbtc/minter/src/fees/mod.rs, lines 112
median_fee.map(|f| f.max(self.minimum_fee_per_vbyte()))
```

**Exploit path:**
1. The Dogecoin canister returns anomalously low fee percentiles (same anomaly as the 2025-06-21 Bitcoin incident).
2. `DogeCanisterRuntime::get_current_fee_percentiles()` fetches these values and stores them in `last_median_fee_per_vbyte`.
3. Any non-anonymous user calls the public `#[update]` endpoint `retrieve_doge_with_approval` → `ic_ckbtc_minter::updates::retrieve_btc::retrieve_btc_with_approval()`. The minter burns ckDOGE from the user's ledger account (irreversible) and builds a Dogecoin transaction using the near-zero fee rate from `DogecoinFeeEstimator::estimate_nth_fee()`.
4. The transaction is broadcast but never mined (fee too low for relay/mining).
5. On the next timer tick, `ic_ckbtc_minter::timer()` (called directly from the ckDOGE minter's `canister_global_timer`) invokes `resubmit_transactions()`. The resubmission logic at `rs/bitcoin/ckbtc/minter/src/lib.rs` line 870 computes `fee_rate.max(prev_fee_rate + Fee::MIN_RELAY_FEE_RATE_INCREASE)`. With an anomalously low stored `effective_fee_per_vbyte`, this arithmetic can produce a deterministic panic (exactly as documented in PR #5713 for ckBTC), trapping the timer and blocking all subsequent ckDOGE withdrawals.

**Why existing checks are insufficient:** The only guard in `estimate_nth_fee` is a length check (`fee_percentiles.len() < 100`), which does not protect against valid-length but anomalously low-valued percentile arrays. The `MIN_RELAY_FEE_RATE_INCREASE` constant (`100_000` millikoinus/byte) does not prevent the initial submission at a near-zero fee rate.

## Impact Explanation

This matches the **High** bounty impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

- **User fund loss / frozen funds:** A user's ckDOGE is burned on the ledger (irreversible) but DOGE is never delivered. Funds remain frozen until a canister upgrade is deployed by operators.
- **Withdrawal functionality unavailability:** If the resubmission path panics deterministically, the minter's timer-driven `finalize_requests` loop becomes stuck, blocking all pending ckDOGE withdrawals — not just the one that triggered the low-fee transaction. This is a platform-level DoS on the ckDOGE withdrawal functionality.

## Likelihood Explanation

The identical failure mode already occurred in production with ckBTC on 2025-06-21, as documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`. The ckDOGE minter reuses the same shared `ic_ckbtc_minter` infrastructure (same `FeeEstimator` trait, same `resubmit_transactions`, same timer loop). The root cause — anomalously low fee percentiles returned by the canister oracle — is not Bitcoin-specific and can equally affect the Dogecoin canister. Any non-anonymous user calling `retrieve_doge_with_approval` during such a period triggers the stuck-transaction scenario. No special privileges are required.

## Recommendation

Add a `minimum_fee_per_vbyte()` method to `DogecoinFeeEstimator` (analogous to `BitcoinFeeEstimator::minimum_fee_per_vbyte()`) and apply it in `estimate_nth_fee()`:

```rust
impl DogecoinFeeEstimator {
    const fn minimum_fee_per_vbyte(&self) -> FeeRate {
        let rate = match &self.network {
            Network::Mainnet => 100_000, // 100 koinu/byte — calibrate to Dogecoin fee market
            Network::Regtest => 0,
        };
        FeeRate::from_millis_per_byte(rate)
    }
}

// In estimate_nth_fee, replace:
Some(fee_percentiles[nth])
// with:
Some(fee_percentiles[nth].max(self.minimum_fee_per_vbyte()))
```

The exact floor value should be calibrated to Dogecoin's typical fee market, but any meaningful non-zero floor prevents the near-zero fee scenario.

## Proof of Concept

**Unit/integration test plan (safe, no mainnet interaction):**

1. Construct a `DogecoinFeeEstimator` with `Network::Mainnet`.
2. Call `estimate_nth_fee(&vec![FeeRate::from_millis_per_byte(1); 100], 50)`.
3. Assert the returned value equals `FeeRate::from_millis_per_byte(1)` — confirming no floor is applied (current behavior).
4. Contrast with `BitcoinFeeEstimator` under the same inputs, which returns `FeeRate::from_millis_per_byte(1_500)` due to the floor.
5. Using PocketIC or the existing integration test harness, simulate a `retrieve_doge_with_approval` call while `last_median_fee_per_vbyte` is set to `FeeRate::from_millis_per_byte(1)`, and verify the minter builds a transaction with a 1 millikoinu/byte fee rate — confirming the stuck-transaction scenario is reachable from an unprivileged caller.
