### Title
`vm_resources_to_execution_resources` Returns `UnknownBuiltin` Error for Valid Transactions Using `range_check96`/`add_mod`/`mul_mod` Builtins - (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

The `match` arm for `range_check96`, `add_mod`, and `mul_mod` builtins is explicitly commented out with a `TODO` in `vm_resources_to_execution_resources`. Any valid transaction whose execution trace includes these builtins causes the function to fall through to the `_` catch-all and return `Err(ExecutionError::UnknownBuiltin)`, making `starknet_simulateTransactions` return an error for a transaction that the blockifier executed successfully.

### Finding Description

In `objects.rs`, the conversion from `VmExecutionResources` to the RPC `ExecutionResources` type iterates over `builtin_instance_counter` and maps each `BuiltinName` to its RPC counterpart: [1](#0-0) 

The three builtins introduced in newer Cairo/Sierra versions — `range_check96`, `add_mod`, `mul_mod` — are explicitly left unhandled. Any non-zero count for these builtins (the zero-count guard at line 369–371 skips zero entries) falls into the `_` arm and returns `Err(ExecutionError::UnknownBuiltin { builtin_name })`.

This function is called from `FunctionInvocation::try_from` during trace construction: [2](#0-1) 

The `?` propagates the error up through `get_trace_constructor`'s closures: [3](#0-2) 

So the full path is: `starknet_simulateTransactions` → `simulate_transactions` → `get_trace_constructor(tx)(execution_info)` → `execution_info.try_into()` → `FunctionInvocation::try_from` → `vm_resources_to_execution_resources` → `Err(UnknownBuiltin)`.

### Impact Explanation

The blockifier executes the transaction successfully (it supports these builtins natively), but the RPC trace-construction layer rejects the result. The caller of `starknet_simulateTransactions` receives an error response instead of a valid `TransactionSimulationOutput`. This is a concrete wrong value returned by the RPC simulation path: an error where a valid trace should appear.

Impact: **High** — RPC simulation returns an authoritative-looking error for a valid transaction.

### Likelihood Explanation

`range_check96` is used by Cairo programs compiled with Sierra ≥ 1.x for 96-bit range checks; `add_mod`/`mul_mod` are used for modular arithmetic. Any user can deploy a contract compiled with a modern Cairo compiler that emits these builtins and trigger the failure with a standard unprivileged `starknet_simulateTransactions` call. No special privileges are required.

### Recommendation

Add explicit match arms for the three missing builtins. The simplest fix is to map them to their closest RPC equivalents or introduce new `Builtin` enum variants, and remove the `TODO` comment:

```rust
BuiltinName::range_check96 => builtin_instance_counter.insert(Builtin::RangeCheck96, count),
BuiltinName::add_mod => builtin_instance_counter.insert(Builtin::AddMod, count),
BuiltinName::mul_mod => builtin_instance_counter.insert(Builtin::MulMod, count),
```

If the RPC `Builtin` enum does not yet have these variants, they should be added. Alternatively, if these builtins are intentionally excluded from the RPC resource representation, they should be silently skipped with `continue` rather than returning an error.

### Proof of Concept

1. Compile a Cairo contract that uses the `range_check96` builtin (any Sierra ≥ 1.x contract performing 96-bit range checks).
2. Declare and deploy it on a node running this code.
3. Call `starknet_simulateTransactions` with an invoke transaction targeting that contract.
4. Observe: the RPC returns `ExecutionError::UnknownBuiltin { builtin_name: range_check96 }` instead of a valid simulation trace, even though the transaction would execute and be accepted by the sequencer.
