### Title
WalletConnect `create_offer_for_ids` Silently Misclassifies NFT as CAT in Signing Confirmation Due to Missing Asset-Type Check — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary
In the WalletConnect command handler for `chia_wallet.create_offer_for_ids`, the receiving side of an offer omits the asset-type guard that the spending side enforces. Any bytes32 asset ID on the receiving side that lacks a `driver_dict` entry is unconditionally assumed to be a CAT. A malicious dApp can exploit this to display an NFT (singleton) as a CAT in the user-facing signing confirmation dialog, causing the user to approve an offer they believe requests a fungible token but actually requests an NFT.

---

### Finding Description

`parseCommandDisplay.ts` contains `createOfferAssetKinds`, which resolves the display type (`'chia'`, `'cat'`, `'nft'`, `'wallet'`) for each asset in a `create_offer_for_ids` WalletConnect command before showing the confirmation dialog.

**Spending side** (what the user gives up): [1](#0-0) 

```typescript
assetKinds.spending[assetId] = assetKindForDriverDictAssetId(assetId, driverDict);
```

If the driver dict has no entry for the asset, `assetKindForDriverDictAssetId` returns `undefined`. Downstream, `parseWalletDeltaItem` hits the guard at line 331–333 and throws `'Asset type is not valid'`, blocking the command.

**Receiving side** (what the user requests): [2](#0-1) 

```typescript
// Backend legacy behavior treats positive requested bytes32 assets with
// no explicit driver as CATs.
assetKinds.receiving[assetId] = assetKindForDriverDictAssetId(assetId, driverDict) ?? 'cat';
```

The `?? 'cat'` fallback silently promotes any unrecognized bytes32 asset to `'cat'`. There is no check equivalent to the spending-side guard. `parseWalletDeltaItem` then renders it as: [3](#0-2) 

```typescript
const { name } = await catAssetIdToName(key);
return { kind: 'cat', amount: mojoToCATLocaleString(value), assetId: key, symbol: name };
```

An NFT launcher ID passed as a receiving asset with no driver dict entry is therefore rendered as `kind: 'cat'` in the confirmation dialog — the NFT branch at lines 302–329 is never reached.

The asymmetry is exact: the spending path is protected; the receiving path is not. This mirrors the original report's pattern where one code path (filling with upfront reward) skipped the points-program check that every other path performed.

---

### Impact Explanation

A user who approves the spoofed confirmation creates a real, on-chain offer that requests the NFT (singleton) rather than the displayed CAT. Any counterparty can fill it, transferring the NFT to the user and collecting the offered XCH. The user loses XCH and receives an asset they did not intend to request. Because the confirmation dialog is the sole user-visible gate before the offer is signed and broadcast, bypassing its accuracy is a direct security impact.

This matches the **High** impact class: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong asset … or status."*

---

### Likelihood Explanation

Any dApp that has obtained a WalletConnect session (the normal, unprivileged entry point) can issue `chia_wallet.create_offer_for_ids` with an NFT launcher ID on the receiving side and omit `driver_dict`. No leaked keys, host compromise, or cryptographic break is required. The WalletConnect approval flow is the intended integration path, so the attacker surface is reachable by any connected dApp.

---

### Recommendation

Remove the `?? 'cat'` fallback on the receiving side and apply the same strict guard used for the spending side. If a bytes32 receiving asset has no driver dict entry, the command should be rejected with an error (matching the spending-side behavior at line 222). Alternatively, if legacy CAT-without-driver behavior must be preserved, add an explicit allowlist check (e.g., verify the asset ID resolves to a known CAT via `catAssetIdToName` before accepting the `'cat'` classification) and surface a clear warning to the user when the type cannot be confirmed.

---

### Proof of Concept

A malicious dApp sends the following WalletConnect request (no `driver_dict`):

```json
{
  "method": "chia_wallet.create_offer_for_ids",
  "params": {
    "offer": {
      "1": "-1000000000000",
      "<nft_launcher_id_hex>": "1"
    }
  }
}
```

1. `createOfferToWalletDelta` parses the dict: wallet `1` (XCH) is spending 1 XCH; `<nft_launcher_id_hex>` is receiving with amount `1`.
2. `createOfferAssetKinds` hits the receiving branch: `assetKindForDriverDictAssetId` returns `undefined` (no driver entry) → fallback assigns `'cat'`.
3. `parseWalletDeltaItem` renders the NFT launcher ID as `kind: 'cat'`, calling `mojoToCATLocaleString(1n)` → displays `"0.001 Unknown CAT"`.
4. The confirmation dialog shows: **"Offering 1 XCH — Receiving 0.001 Unknown CAT"**.
5. User approves. The offer is created on-chain requesting the NFT singleton.
6. Counterparty fills the offer; user loses 1 XCH and receives the NFT. [4](#0-3) [5](#0-4)

function createOfferAssetKinds(
  walletDelta: WalletDelta,
  walletInfos: Record<string, WalletInfo>,
  driverDict: Record<string, unknown>,
): AssetDisplayKinds {
  const assetKinds: AssetDisplayKinds = {
    spending: {},
    receiving: {},
  };

  for (const assetId of Object.keys(walletDelta.spending)) {
    if (isNumericKey(assetId)) {
      assetKinds.spending[assetId] = assetKindForWalletId(assetId, walletInfos);
    } else {
      assetKinds.spending[assetId] = assetKindForDriverDictAssetId(assetId, driverDict);
    }
  }

  for (const assetId of Object.keys(walletDelta.receiving)) {
    if (isNumericKey(assetId)) {
      assetKinds.receiving[assetId] = assetKindForWalletId(assetId, walletInfos);
    } else {
      // Backend legacy behavior treats positive requested bytes32 assets with
      // no explicit driver as CATs.
      assetKinds.receiving[assetId] = assetKindForDriverDictAssetId(assetId, driverDict) ?? 'cat';
    }
  }

  return assetKinds;
}
