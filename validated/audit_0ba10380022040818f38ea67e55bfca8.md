### Title
FLOW Tokens Permanently Frozen When `CadenceOwnedAccount` Resource Is Destroyed Without Withdrawing EVM Balance - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` function. When a COA is destroyed via `destroy coa` while holding a non-zero EVM balance, the FLOW tokens stored at the associated EVM address become permanently inaccessible. The Cadence resource that holds the `Owner`/`Withdraw` entitlement required to call `withdraw()` is gone, and the EVM address has no private key, so no path exists to recover the funds.

---

### Finding Description

The `CadenceOwnedAccount` resource is defined at line 518 of `fvm/evm/stdlib/contract.cdc`:

```cadence
access(all) resource CadenceOwnedAccount: Addressable {
    access(self) var addressBytes: [UInt8; 20]
    ...
}
``` [1](#0-0) 

The resource exposes a `withdraw` function gated behind the `Owner | Withdraw` entitlement:

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault { ... }
``` [2](#0-1) 

The resource has **no `destroy` function**. The grep search for `destroy|ResourceDestroyer|beforeDestroy` in `contract.cdc` returns only 2 incidental matches — neither is a resource destructor. This means Cadence will silently destroy the resource without any pre-destruction hook.

When a user calls `destroy coa`:
1. The Cadence resource object is destroyed.
2. The EVM balance at `addressBytes` remains in the EVM state (managed by `InternalEVM`).
3. No Cadence capability or entitlement to `withdraw()` from that EVM address exists anymore.
4. COA EVM addresses have no associated private key (they are controlled exclusively through the Cadence resource), so no EVM-native transaction can recover the funds either.

The FLOW tokens are permanently frozen at the orphaned EVM address.

---

### Impact Explanation

Any FLOW tokens deposited into a COA's EVM balance via `deposit()` and not withdrawn before `destroy` are permanently lost. The EVM state retains the balance at the address, but the only mechanism to move those tokens back to Cadence — the `withdraw()` function on the `CadenceOwnedAccount` resource — is gone. This is a **permanent freezing of FLOW tokens** with no recovery path. [3](#0-2) 

---

### Likelihood Explanation

Any unprivileged user who:
1. Creates a COA (`EVM.createCadenceOwnedAccount()`),
2. Deposits FLOW tokens into it,
3. Then calls `destroy coa` without first calling `withdraw()`

will permanently lose their funds. This is a realistic user-error scenario with no on-chain protection. The contract provides no guard (no pre-condition, no non-zero balance check, no automatic drain) in the destruction path. [4](#0-3) 

---

### Recommendation

Add a `destroy` function to `CadenceOwnedAccount` that either:

1. **Panics if the EVM balance is non-zero**, forcing the caller to withdraw first:
   ```cadence
   access(contract)
   fun destroy() {
       pre {
           self.balance().isZero(): "CadenceOwnedAccount: must withdraw EVM balance before destroying"
       }
   }
   ```

2. **Or automatically withdraws the full balance** to a caller-supplied Cadence vault before destruction (requires a design change to pass a receiver).

Option 1 is the minimal safe fix and directly mirrors the correct pattern: require the user to claim assets before destruction, rather than silently discarding them. [5](#0-4) 

---

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Mint some FLOW tokens
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 10.0)
        let vault <- minter.mintTokens(amount: 10.0)
        destroy minter

        // Create a COA and deposit 10 FLOW into its EVM balance
        let coa <- EVM.createCadenceOwnedAccount()
        let evmAddr = coa.address()
        coa.deposit(from: <-vault)

        // Balance is now 10.0 FLOW in EVM state
        // Destroy the COA WITHOUT withdrawing
        destroy coa

        // The 10.0 FLOW at evmAddr is now permanently frozen:
        // - No Cadence resource holds Owner/Withdraw entitlement to that address
        // - The EVM address has no private key
        // - Funds are irrecoverable
    }
}
``` [2](#0-1)

    /// Creates a new cadence owned account
    access(all)
    fun createCadenceOwnedAccount(): @CadenceOwnedAccount {
        pre {
            !self.isPaused(): "EVM operations are temporarily paused"
        }
        let acc <-create CadenceOwnedAccount()
        let addr = InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)
        acc.initAddress(addressBytes: addr)

        emit CadenceOwnedAccountCreated(address: acc.address().toString(), uuid: acc.uuid)
        return <-acc
    }
