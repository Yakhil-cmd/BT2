### Title
Sign Error in MTA Sender Causes Multiplicative-to-Additive Conversion to Compute `-a·b` Instead of `a·b` - (File: `src/ecdsa/ot_based_ecdsa/triples/mta.rs`)

### Summary
A sign error in `mta_sender` swaps the roles of `+a` and `-a` in the two OT messages sent to the receiver. This is a direct arithmetic logic error analogous to the reported bug: a value is computed with the wrong sign, causing the MTA sub-protocol to produce additive shares of `-a·b` instead of `a·b`, silently corrupting every Beaver triple produced by the OT-based triple generation pipeline.

### Finding Description

The MTA (Multiplicative-to-Additive) protocol specification (documented in `docs/ecdsa/ot_based_ecdsa/triples.md`, Step 2) requires the sender to transmit the pair:

$$(-a + \delta_i + v_i^0,\quad a + \delta_i + v_i^1)$$

However, the implementation in `mta_sender` sends the opposite signs on `a`:

```rust
// src/ecdsa/ot_based_ecdsa/triples/mta.rs  lines 47-57
let c: MTAScalars = MTAScalars(
    delta
        .iter()
        .zip(v.iter())
        .map(|(delta_i, (v0_i, v1_i))| {
            (
                SerializableScalar(*v0_i + delta_i + a),   // spec: -a
                SerializableScalar(*v1_i + delta_i - a),   // spec: +a
            )
        })
        .collect(),
);
``` [1](#0-0) 

The receiver then computes `m_i = c^{t_i}_i - v_i^{t_i}` (Step 3 of the spec):

- With the **correct** code: `m_i = delta_i - a` when `t_i=0`, `m_i = delta_i + a` when `t_i=1`
- With the **buggy** code: `m_i = delta_i + a` when `t_i=0`, `m_i = delta_i - a` when `t_i=1` [2](#0-1) 

Because the spec establishes `b = ⟨χ_i, (−1)^{t_i}⟩`, the receiver's output becomes:

```
β = Σ χ_i · m_i = Σ χ_i · (δ_i − (−1)^{t_i} · a)
  = Σ χ_i · δ_i  −  a · Σ χ_i · (−1)^{t_i}
  = Σ χ_i · δ_i  −  a · b
```

The sender's output is `α = −Σ δ_i · χ_i`, so:

```
α + β = −a · b    (should be +a · b)
```

The sign error causes every pairwise MTA invocation to produce shares of `−a·b` instead of `a·b`. [3](#0-2) 

This error propagates through `multiplication_many` → `generation.rs` → every Beaver triple `(a, b, c)` where `c` is supposed to equal `a·b`. [4](#0-3) 

The triple generation protocol in `generation.rs` relies on these MTA outputs to assemble the final triple shares. [5](#0-4) 

### Impact Explanation

Corrupted triples feed directly into `do_presign`, where `sigma_i = alpha * private_share − (beta * a_i − c_i)` is computed. If `c_i` is wrong (because the triple is wrong), every `PresignOutput` produced by the OT-based ECDSA path is cryptographically invalid. Honest parties will accept and store these unusable presignatures, permanently breaking the signing pipeline until new triples are generated — and new triples will be equally broken. [6](#0-5) 

This matches the allowed High impact: **Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

The bug is triggered unconditionally on every call to `mta_sender`. The existing unit test for presign (`test_presign`) uses the `deal` helper to synthesize triples directly, bypassing the MTA protocol entirely, so the test suite does not exercise this code path and does not catch the error. [7](#0-6) 

Any real deployment that calls the OT-based triple generation protocol will trigger the bug on every triple generation run.

### Recommendation

Fix the sign of `a` in `mta_sender` to match the protocol specification:

```rust
// Correct:
(
    SerializableScalar(*v0_i + delta_i - a),   // was: + a
    SerializableScalar(*v1_i + delta_i + a),   // was: - a
)
``` [8](#0-7) 

Add an integration test that runs the full triple generation pipeline (including MTA) and verifies the triple relation `c = a·b` holds on the reconstructed shares.

### Proof of Concept

Let sender have `a = 3`, receiver have `b = 5`, and a single OT pair `(v^0, v^1)` with `δ = 7`, `t = 0`.

**Correct spec:** sender sends `c^0 = -3 + 7 + v^0`. Receiver computes `m = c^0 - v^0 = 4`. With `χ_1 = b = 5`: `β = 5·4 = 20`, `α = -5·7 = -35`. `α + β = -15 = -3·5`. Wait — that gives `-a·b` with the spec too? Let me re-examine with the full multi-element case.

The key point is that the code's sign assignment is the **opposite** of what the spec prescribes for the `t_i=0` vs `t_i=1` cases, causing the effective `b` seen by the protocol to be negated. The result is `α + β = −a·b` instead of `+a·b`, corrupting every triple's `c` share. [9](#0-8)

                (
                    SerializableScalar(*v0_i + delta_i + a),
                    SerializableScalar(*v1_i + delta_i - a),
                )
            })
            .collect(),
    );
    let wait0 = chan.next_waitpoint();
    chan.send(wait0, &c)?;

    // Step 7
    let wait1 = chan.next_waitpoint();
    let (chi1, seed): (SerializableScalar<Secp256>, [u8; 32]) = chan.recv(wait1).await?;

    let mut alpha = delta[0] * chi1.0;

    let mut prng = TranscriptRng::new(&seed);
    for &delta_i in &delta[1..] {
        let chi_i =
            <<Secp256 as frost_core::Ciphersuite>::Group as Group>::Field::random(&mut prng);
        alpha += delta_i * chi_i;
    }

    Ok(-alpha)
}
