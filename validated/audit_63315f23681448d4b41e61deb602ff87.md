### Title
Lack of Slippage Control for `enter_delegation_pool` and `switch_delegation_pool` - (File: src/pool/pool.cairo)

### Summary
The `enter_delegation_pool` and `switch_delegation_pool` functions in the Pool contract accept no `max_commission` guard. A staker who holds an active commission commitment can atomically raise their commission before a delegator's transaction lands, causing the delegator to earn materially fewer rewards than they simulated off-chain.

### Finding Description

When a delegator calls `enter_delegation_pool` or `switch_delegation_pool`, the only parameters accepted are the destination pool address and the token amount. Neither function accepts a maximum-acceptable-commission value. [1](#0-0) [2](#0-1) 

The commission that will actually apply to the delegator's future rewards is read live from the staking contract at reward-distribution time. It is not locked at entry.

A staker can change their commission via `set_commission`. Without an active commitment the commission may only decrease, which is safe for delegators. However, when a staker has set an active `commission_commitment`, the internal `update_commission` helper permits the commission to be raised to any value up to `max_commission`: [3](#0-2) 

`set_commission` takes effect immediately with no time-lock or delay. A staker can therefore call `set_commission(max_commission)` in the same block (or even the same transaction bundle on Starknet) as the delegator's `enter_delegation_pool` call, front-running the delegator.

### Impact Explanation

The delegator deposits principal expecting reward rate `R` (derived from commission `C_old`). After the front-run, the effective commission is `C_new > C_old`. Every future reward epoch the delegator earns `rewards * (1 - C_new)` instead of `rewards * (1 - C_old)`. The difference flows to the staker. The delegator's principal is not at risk, but their yield stream is permanently reduced until they exit — matching **High: Theft of unclaimed yield**.

### Likelihood Explanation

- Any staker who has called `set_commission_commitment` (a public, permissionless action) can execute this attack.
- Starknet's sequencer ordering is observable in the mempool; a staker running their own node or monitoring the mempool can reliably front-run a delegator's delegation transaction.
- The attack requires no privileged access beyond being the staker of the target pool.

### Recommendation

Add an optional `max_commission: Option<Commission>` parameter to both `enter_delegation_pool` and `switch_delegation_pool`. Before completing the delegation, assert that the pool's current commission does not exceed the caller-supplied bound:

```rust
fn enter_delegation_pool(
    ref self: ContractState,
    reward_address: ContractAddress,
    amount: Amount,
    max_commission: Option<Commission>,  // <-- new
) {
    if let Option::Some(limit) = max_commission {
        let current = self.get_commission_from_staking_contract();
        assert!(current <= limit, "{}", Error::COMMISSION_EXCEEDS_SLIPPAGE_LIMIT);
    }
    // ... rest of existing logic
}
```

Apply the same guard to `switch_delegation_pool`.

### Proof of Concept

1. Staker calls `set_commission_commitment(max_commission: 2000, expiration_epoch: current + 10)` — commits to allow commission up to 20 %.
2. Staker calls `set_commission(500)` — sets commission to 5 %.
3. Delegator observes commission = 5 % off-chain and submits `enter_delegation_pool(reward_address, amount)`.
4. Staker observes the pending transaction and submits `set_commission(2000)` with higher gas / priority, landing first.
5. Delegator's `enter_delegation_pool` executes; `get_commission_from_staking_contract` now returns 2000 (20 %).
6. For every subsequent reward epoch the delegator receives 80 % of pool rewards instead of the 95 % they expected — a 15.8 % reduction in yield that accrues to the staker. [4](#0-3)
