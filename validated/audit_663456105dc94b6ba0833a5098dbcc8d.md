### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Unverified Signatures by Exploiting Any Pending Deploy-Account in Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point check for an invoke transaction when the sender address has a pending transaction in the mempool and the on-chain nonce is zero. The check does not verify that the incoming invoke transaction is authorized by the same key that submitted the deploy-account. Any observer can craft an invoke transaction with a victim's sender address, nonce=1, and a garbage signature, and the gateway will admit it to the mempool without ever verifying the signature.

---

### Finding Description

The gateway implements a UX feature that allows a user to submit a `deploy_account` + `invoke` pair simultaneously. Because the account does not exist on-chain yet when the invoke arrives, the gateway cannot call `__validate__` (the account contract is not deployed). The function `skip_stateful_validations` decides whether to skip this check: [1](#0-0) 

The skip condition is:
1. The transaction is an `Invoke`.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [2](#0-1) 

Condition 4 only checks whether **any** transaction from the sender address exists in the mempool or a recent block. It does not verify that the incoming invoke transaction carries a valid signature for that address, nor that it originates from the same user who submitted the deploy-account.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate: false`, the function returns immediately after `perform_pre_validation_stage` without ever calling `__validate__`: [4](#0-3) 

`perform_pre_validation_stage` only checks nonce, fee bounds, and proof facts — it does not verify the transaction signature: [5](#0-4) 

The result is that the gateway calls `mempool_client.add_tx` with a transaction whose signature has never been verified. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool (mempool contents are observable) can submit an `Invoke` transaction with:
- `sender_address` = victim's address
- `nonce` = 1
- `calldata` = arbitrary
- `signature` = garbage

The gateway will:
1. Pass `validate_nonce` (nonce 1 ≥ account_nonce 0).
2. Pass `validate_by_mempool` (no duplicate nonce yet).
3. Return `skip_validate = true` from `skip_stateful_validations` (victim's address is in the mempool).
4. Skip `run_validate_entry_point` entirely.
5. Admit the transaction to the mempool.

**Concrete harms:**
- **Mempool admission of invalid transactions**: A transaction with an unverified signature enters the mempool, violating the invariant that every mempool transaction has passed account-level validation.
- **Nonce squatting / victim DoS**: The attacker's transaction occupies nonce=1 for the victim's address. If the attacker sets a higher fee than the victim's legitimate nonce=1 invoke, the victim's transaction is rejected by the mempool's fee-escalation check (`DuplicateNonce` or insufficient escalation), blocking the victim's post-deploy invoke from being sequenced.

At execution time the blockifier calls `__validate__` with `validate: true` (via `new_for_sequencing`), so the attacker's transaction will revert — but the admission-time invariant is already broken and the nonce-squatting harm has already occurred. [7](#0-6) 

---

### Likelihood Explanation

- Mempool contents are observable by any network participant.
- The attack requires no special privilege: submit one `Invoke` transaction with the victim's address and nonce=1.
- The condition `account_nonce == 0 && tx_nonce == 1` is exactly the window during which the UX feature is active, and it is a predictable, short-lived state for every new account deployment.
- The `account_tx_in_pool_or_recent_block` check is trivially satisfied by the victim's own pending deploy-account.

---

### Recommendation

Before returning `skip_validate = true`, verify that the incoming invoke transaction carries a structurally valid signature for the account class that will be deployed. Concretely:

1. Retrieve the deploy-account transaction for the sender address from the mempool.
2. Extract the `class_hash` from that deploy-account.
3. Instantiate the class and call a lightweight off-chain signature-format check (e.g., verify the signature length and format match the expected account class), or at minimum verify the signature is non-empty and well-formed.

Alternatively, restrict the skip to cases where the gateway can cryptographically link the invoke to the deploy-account (e.g., require the invoke to include a commitment to the deploy-account transaction hash in its calldata or signature).

---

### Proof of Concept

```
Precondition:
  Victim submits deploy_account for address V (nonce=0).
  V is now in the mempool; account_nonce for V on-chain = 0.

Attack:
  Attacker submits Invoke {
      sender_address: V,
      nonce: 1,
      calldata: [<malicious>],
      signature: [0x0, 0x0],   // garbage
      resource_bounds: <valid>,
  }

Gateway path:
  1. validate_nonce(V, account_nonce=0, tx_nonce=1) → OK (0 ≤ 1 ≤ max_gap)
  2. validate_by_mempool(V, nonce=1) → OK (no existing nonce=1 for V)
  3. skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(V) == true  ✓  (victim's deploy_account)
       → returns true (skip __validate__)
  4. run_validate_entry_point skipped entirely.
  5. mempool.add_tx(attacker_invoke) → admitted.

Result:
  Attacker's transaction with garbage signature is in the mempool at nonce=1 for V.
  Victim's legitimate nonce=1 invoke is rejected (DuplicateNonce or fee-escalation failure).
  Victim's deploy_account + invoke UX flow is broken.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
