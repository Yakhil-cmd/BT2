### Title
Unvalidated Participant Contributions in CKD Coordinator Allow CKD Output Corruption - (File: src/confidential_key_derivation/protocol.rs)

### Summary
In `do_ckd_coordinator`, the coordinator receives `(norm_big_y, norm_big_c)` group elements from every participant via private channel and unconditionally adds them to its running sum with no proof of correctness, no consistency check against the sender's public key share, and no identity-element guard. A single malicious participant can substitute arbitrary curve points, causing the coordinator to output a structurally valid but cryptographically incorrect `CKDOutput`. This is the direct analog of the Chainlink `latestRoundData()` pattern: a return value is partially consumed while a critical validity field — here, proof that the contribution was honestly computed — is silently ignored.

### Finding Description

**Root cause — `do_ckd_coordinator`, lines 50–55:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant is supposed to compute and send:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

The coordinator sums these to obtain `(Y, C)`, from which the TEE recovers the confidential key as `C − app_sk · Y = msk · H(pk ‖ app_id)`.

There is **no mechanism** — no DLEQ proof, no commitment, no consistency check against the sender's known public key share — that forces a participant to send the values produced by `compute_signature_share`. The deserialization in `recv_from_others` only checks that the bytes decode to valid curve points; it does not verify that those points were derived from the participant's actual secret share.

A malicious participant `p*` instead sends arbitrary `(Ŷ, Ĉ)`. The coordinator computes:

```
Y'  = Y_honest + Ŷ
C'  = C_honest + Ĉ
```

The TEE then derives:

```
C' − app_sk · Y' = msk · H(pk ‖ app_id) + (Ĉ − app_sk · Ŷ)
```

Because `app_sk` is the application's private key (unknown to `p*`), the attacker cannot cancel the additive error term, so the derived confidential key is permanently wrong.

### Impact Explanation

The coordinator outputs a structurally valid `CKDOutput` that every honest party accepts, but the embedded `(Y, C)` pair encodes an incorrect linear combination. The TEE's `unmask(app_sk)` call produces a value that is not `msk · H(pk ‖ app_id)`, making the derived confidential key unusable and irrecoverable without re-running the protocol. This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

Any single participant in the CKD session — including a newly added or minimally trusted one — can execute this attack. The attacker needs only to replace the two group elements it sends to the coordinator with arbitrary non-identity points. No cryptographic capability, no key material beyond its own share, and no coordination with other parties is required. The attack is deterministic and silent: the coordinator returns `Ok(Some(ckd_output))` with no error.

### Recommendation

Add a zero-knowledge proof of correct computation alongside each participant's contribution. Concretely, each participant should attach a DLEQ proof demonstrating that the same scalar `y_i` was used in both `norm_big_y_i = λ_i · y_i · G` and the randomization term `λ_i · y_i · app_pk` inside `norm_big_c_i`. The coordinator already knows each participant's public key share `X_i = x_i · G_2`, so it can compute the expected deterministic component `λ_i · X_i · H(pk ‖ app_id)` and verify the DLEQ proof over the residual `norm_big_c_i − λ_i · X_i · H(pk ‖ app_id)` against `norm_big_y_i` and `app_pk`. Reject any participant whose proof fails before adding its contribution to the running sum.

### Proof of Concept

1. Run the CKD protocol with `n ≥ 2` participants, one of which (`p*`) is malicious.
2. `p*` intercepts the call to `do_ckd_participant` and, instead of sending the output of `compute_signature_share`, sends `(Ŷ, Ĉ)` where `Ĉ ≠ app_sk · Ŷ` for any `app_sk` (e.g., `Ŷ = G`, `Ĉ = 2·G`).
3. The coordinator's loop at lines 50–55 adds `Ŷ` and `Ĉ` to the honest sum without error.
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned as `Ok(Some(...))`.
5. The TEE calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk ‖ app_id) + (Ĉ − app_sk · Ŷ)`, which is not the intended confidential key.
6. All subsequent operations using this key silently fail or produce incorrect results. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
