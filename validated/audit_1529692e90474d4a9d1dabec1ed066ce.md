### Title
Silent Destruction of Recreated EVM Contract on CREATE2 Address Collision — (`fvm/evm/emulator/state/stateDB.go`)

### Summary

`stateDB.Commit` contains a mutually exclusive branch structure that permanently loses a recreated EVM account when `HasSelfDestructed` and `IsCreated` are both true for the same address. This state is set deterministically by `DeltaView.CreateAccount` whenever CREATE2 targets an address that already holds a non-zero balance in `BaseView`. After `Commit`, the address does not exist in persistent storage, silently destroying the deployed contract and burning any ETH balance.

### Finding Description

**Root cause — `DeltaView.CreateAccount` sets two conflicting flags simultaneously**

When `CreateAccount` is called on an address that already exists (e.g., a CREATE2 target that received ETH before deployment), the function:

1. Calls `d.SelfDestruct(addr)` → writes `d.toBeDestructed[addr]` (so `HasSelfDestructed` returns `true`)
2. Unconditionally writes `d.created[addr] = struct{}{}` (so `IsCreated` returns `true`) [1](#0-0) 

Both flags coexist on the same address after this call returns.

**Root cause — `stateDB.Commit` short-circuits on `deleted=true` before reaching the `IsCreated` branch**

```
if db.HasSelfDestructed(addr) {   // true  → DeleteAccount called
    ...
    deleted = true
}
if deleted {
    continue                       // ← jumps here, skipping IsCreated entirely
}
...
if db.IsCreated(addr) {           // never reached
    db.baseView.CreateAccount(...)
}
``` [2](#0-1) 

`baseView.CreateAccount` is never called. The `recreated` flag tracked in `DeltaView` is never consulted in `Commit`. [3](#0-2) 

### Impact Explanation

After `stateDB.Commit` returns:

- `BaseView` has no record of the address (it was deleted and never re-created).
- Any code set via `SetCode` during the transaction exists only in the `DeltaView` layer, which is discarded.
- Any ETH balance carried over into the new account is burned (the old account's balance was deleted; the new account was never written).
- A fresh `BaseView` constructed from the same ledger will report the address as non-existent, confirming the contract was silently destroyed.

This constitutes unauthorized destruction of an EVM resource: a successfully executing CREATE2 transaction produces no persistent contract, with no error returned to the caller.

### Likelihood Explanation

The precondition is reachable through a standard Ethereum pattern. The EVM's collision guard (in go-ethereum's `evm.go`) only blocks CREATE2 when `nonce != 0` or `codeHash != emptyCodeHash`. An address with balance but no code and nonce=0 passes the guard. Sending ETH to a CREATE2 address before deployment is a well-known factory pattern (e.g., counterfactual deployment). Any such deployment on Flow EVM triggers this path.

The scenario does not require admin access, staked node control, or key compromise. It is reachable via a normal EVM transaction submitted through the standard Access API path.

### Recommendation

In `stateDB.Commit`, after deleting a self-destructed account, check whether the address is also flagged as `IsCreated` (i.e., it was recreated in the same transaction). If so, call `baseView.CreateAccount` with the current balance/nonce/code instead of `continue`-ing. The `recreated` map already tracked in `DeltaView` can serve as the discriminator.

Concretely, the fix replaces the unconditional `continue` after `deleted = true` with:

```go
if deleted {
    if db.IsCreated(addr) {
        // account was recreated after self-destruct; persist the new account
        bal := db.GetBalance(addr)
        nonce := db.GetNonce(addr)
        code := db.GetCode(addr)
        codeHash := db.GetCodeHash(addr)
        if err = db.baseView.CreateAccount(addr, bal, nonce, code, codeHash); err != nil {
            return nil, wrapError(err)
        }
        // also call updateCommitter.CreateAccount(...)
    }
    continue
}
```

### Proof of Concept

1. Construct a `BaseView` backed by an in-memory ledger; call `baseView.CreateAccount(addrA, balance=100, nonce=0, code=nil, ...)` and `baseView.Commit()`.
2. Construct a `StateDB` over the same ledger. Call `stateDB.CreateAccount(addrA)` (simulating the EVM's CREATE2 path). At this point `DeltaView.toBeDestructed[addrA]` and `DeltaView.created[addrA]` are both set.
3. Call `stateDB.SetCode(addrA, someCode)` and `stateDB.AddBalance(addrA, 50)`.
4. Call `stateDB.Commit(true)`.
5. Construct a fresh `BaseView` from the same ledger. Call `baseView.Exist(addrA)` → returns `false`. Call `baseView.GetBalance(addrA)` → returns `0`. The recreated contract and its balance are gone. [4](#0-3) [5](#0-4)

func (d *DeltaView) CreateAccount(addr gethCommon.Address) error {
	// if is already created return
	if d.IsCreated(addr) {
		return nil
	}
	exist, err := d.Exist(addr)
	if err != nil {
		return err
	}
	if exist {
		// check if already destructed
		destructed, balance := d.HasSelfDestructed(addr)
		if !destructed {
			balance, err = d.GetBalance(addr)
			if err != nil {
				return err
			}
			err = d.SelfDestruct(addr)
			if err != nil {
				return err
			}
		}

		d.nonces[addr] = 0
		d.codes[addr] = nil
		d.codeHashes[addr] = gethTypes.EmptyCodeHash
		// carrying over the balance. (legacy behavior of the Geth stateDB)
		d.balances[addr] = balance

		// flag addr as recreated, this flag helps with postponing deletion of slabs
		// otherwise we have to iterate over all slabs of this account and set the to nil
		d.recreated[addr] = struct{}{}

		// remove slabs from cache related to this account
		for k := range d.slots {
			if k.Address == addr {
				delete(d.slots, k)
			}
		}
	}
	d.dirtyAddresses[addr] = struct{}{}
	d.created[addr] = struct{}{}
	return nil
}
