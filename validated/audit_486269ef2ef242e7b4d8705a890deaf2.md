### Title
Participant Set Used in Presignature Rerandomization Is Not Enforced to Match the Signing Participant Set — (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The robust ECDSA signing flow is a two-step sequential process: (1) rerandomize the presignature using `RerandomizationArguments` (which includes a `participants` list that is hashed into `delta`), then (2) run `sign()` with a separate `participants` list used for Lagrange coefficient computation. No code-level check enforces that the `participants` used in step 1 matches the `participants` used in step 2. A malicious coordinator can supply an inconsistent participant set across these two steps, breaking the rerandomization binding and causing honest parties to produce inconsistent or unusable signing outputs.

---

### Finding Description

The robust ECDSA signing protocol has two sequential steps that must use a consistent participant set:

**Step 1 — Rerandomization** (`src/ecdsa/robust_ecdsa/mod.rs`, `RerandomizedPresignOutput::rerandomize_presign`):

`RerandomizationArguments` carries a `participants: ParticipantList` field. [1](#0-0) 

`derive_randomness()` feeds this participant list directly into the HKDF to produce `delta`: [2](#0-1) 

`delta` is then used to rerandomize `big_r`, `alpha`, and `beta` inside `rerandomize_presign()`: [3](#0-2) 

The resulting `RerandomizedPresignOutput` stores only the rerandomized scalars (`big_r`, `alpha`, `beta`, `e`) — it does **not** store which `participants` were used to derive `delta`. [4](#0-3) 

**Step 2 — Signing** (`src/ecdsa/robust_ecdsa/sign.rs`, `sign()`):

`sign()` accepts a fresh `participants: &[Participant]` argument and a pre-built `presignature: RerandomizedPresignOutput`. [5](#0-4) 

The Lagrange coefficients used to linearize each signer's share are computed from this `participants` argument — not from the `participants` that were used to derive `delta` in step 1. There is **no check** anywhere in `sign()` that the two participant sets are equal. [6](#0-5) 

**Attack path (malicious coordinator):**

1. Coordinator runs presigning with participant set `P1 = {A, B, C}`.
2. Coordinator calls `rerandomize_presign()` with `RerandomizationArguments { participants: P_rerand = {A, B, D}, ... }` — a set that differs from the actual intended signing set. `delta` is now bound to `P_rerand`.
3. Coordinator distributes the resulting `RerandomizedPresignOutput` (with `big_r`, `alpha`, `beta` rerandomized under `delta(P_rerand)`) to signers.
4. Coordinator calls `sign()` with `participants = P_sign = {A, B, C}` — the real signing set.
5. Each signer computes Lagrange coefficients over `P_sign`, but their presignature shares were rerandomized under `delta(P_rerand)`. The two are inconsistent.
6. The combined signature is invalid/unusable; honest parties have accepted and processed an inconsistent signing transcript.

Alternatively, the coordinator can provide **different** `RerandomizationArguments.participants` to different honest signers, causing each to compute a different `delta`, producing mutually inconsistent rerandomized shares — the signing round produces garbage.

The security considerations document acknowledges the requirement: *"Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers."* [7](#0-6)  However, this is only a documentation note — the code never enforces it.

---

### Impact Explanation

**High.** A malicious coordinator can corrupt the signing output so that honest parties accept inconsistent signing transcripts or produce unusable cryptographic outputs. The rerandomization binding — which is the primary defense against Wagner attacks and split-view attacks — is broken because the participant set hashed into `delta` does not match the participant set used for Lagrange interpolation in signing. This maps directly to the allowed High impact: *"Corruption of… sign… outputs so honest parties accept inconsistent… participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

**Medium.** The coordinator is an explicitly in-scope attacker role. The attack requires only that the coordinator supply a `RerandomizationArguments` struct with a `participants` field that differs from the `participants` argument later passed to `sign()`. No cryptographic capability is needed; it is a pure API-level inconsistency that the library does not prevent.

---

### Recommendation

1. Store the `participants` used during rerandomization inside `RerandomizedPresignOutput`.
2. At the start of `sign()`, assert that the stored `participants` equals the `participants` argument passed to `sign()`, returning an `InitializationError` on mismatch.
3. Alternatively, move rerandomization inside `sign()` so that the same `participants` list is used for both `delta` derivation and Lagrange coefficient computation, eliminating the gap entirely.

### Proof of Concept

```rust
// Step 1: rerandomize with P_rerand = {A, B, D}
let args_rerand = RerandomizationArguments::new(
    pk, tweak, msg_hash, big_r,
    ParticipantList::new(&[A, B, D]).unwrap(),  // <-- P_rerand
    entropy,
);
let rerand_presig = RerandomizedPresignOutput::rerandomize_presign(&presig, &args_rerand).unwrap();
// delta is now HKDF(..., [A, B, D])

// Step 2: sign with P_sign = {A, B, C}  (different from P_rerand)
let protocol = sign(
    &[A, B, C],   // <-- P_sign, Lagrange coefficients computed over this
    coordinator,
    max_malicious,
    me,
    public_key,
    rerand_presig,  // rerandomized under delta([A, B, D])
    msg_hash,
).unwrap();
// Signing proceeds without error but produces an invalid/inconsistent signature
// because Lagrange(P_sign) ≠ Lagrange(P_rerand) while shares were rerandomized under delta(P_rerand)
```

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
