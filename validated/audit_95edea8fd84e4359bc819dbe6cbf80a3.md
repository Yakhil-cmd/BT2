### Title
Malicious CKD Participant Can Corrupt Protocol Output by Sending Arbitrary Contributions Without Verification - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator blindly accumulates `(big_y, big_c)` contributions from every participant with no zero-knowledge proof or consistency check. A single malicious participant can substitute arbitrary group elements for their correct share, causing the coordinator to output a corrupted CKD result that all honest parties accept as final, permanently denying the TEE application its correct confidential key.

### Finding Description
**Vulnerability class (analog mapping):** The external report describes a "poisonous token" attack where an attacker passes a swap-path ending with a token they control, so fees are paid in a worthless asset. The analog here is a malicious CKD participant who passes a "poisonous contribution" — arbitrary group elements — instead of the correct cryptographic computation, so the coordinator's aggregated output is silently corrupted.

**Root cause — `do_ckd_coordinator`, lines 44–57:**

In `src/confidential_key_derivation/protocol.rs`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and unconditionally adds it to the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is no proof of correct computation attached to each contribution. The correct per-participant computation is:

```
big_c_i = x_i · H(pk ‖ app_id) + y_i · app_pk
norm_big_c_i = λ_i · big_c_i
``` [2](#0-1) 

Nothing in the protocol forces a participant to use their actual secret share `x_i` or the agreed-upon `app_pk`. A malicious participant can deviate from `compute_signature_share` entirely and send any `(ElementG1, ElementG1)` pair they choose. The coordinator has no mechanism to detect the deviation. [3](#0-2) 

**Contrast with DKG:** Every other multi-party protocol in this library (DKG, reshare, refresh) enforces correctness of participant contributions via `verify_proof_of_knowledge`, `validate_received_share`, and `verify_commitment_hash`. [4](#0-3) 

The CKD protocol has no equivalent safeguard.

**Attacker-controlled entry path:**
1. Attacker is a legitimate participant in the CKD session (unprivileged, no special role required).
2. Instead of calling `compute_signature_share` honestly, the attacker's node sends `(G, 0)` (or any chosen pair) to the coordinator via the private channel.
3. The coordinator's loop adds the poisonous values to `norm_big_y` and `norm_big_c` without rejection.
4. The coordinator outputs the corrupted `CKDOutput`. [5](#0-4) 

### Impact Explanation
The correct CKD invariant is:

```
unmask(big_c, big_y, a) = big_c − a·big_y = msk · H(pk ‖ app_id)
``` [6](#0-5) 

If the malicious participant injects `(big_y_evil = G, big_c_evil = 0)`:

```
big_y_total  = Σ λ_i y_i G  +  G
big_c_total  = msk · H(pk‖app_id) + (Σ λ_i y_i) · app_pk
unmask       = msk · H(pk‖app_id) − a·G   ≠   msk · H(pk‖app_id)
```

The TEE application receives a permanently wrong confidential key. Because the coordinator outputs a single `CKDOutput` with no per-participant audit trail, honest parties cannot distinguish a corrupted result from a correct one. This matches the allowed High impact: **corruption of CKD output so honest parties accept an unusable cryptographic output**.

### Likelihood Explanation
Any participant in the CKD session — including a participant who legitimately holds a valid key share — can mount this attack with zero cryptographic capability beyond participation. No threshold of colluders is required; a single malicious node suffices. The attack is trivially reproducible by any library caller who controls one participant's network messages.

### Recommendation
Require each participant to attach a zero-knowledge proof of correct computation alongside their `(norm_big_y, norm_big_c)` contribution. Concretely, a Schnorr-style proof of knowledge of `(x_i, y_i)` satisfying:

```
norm_big_c_i = λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)
norm_big_y_i = λ_i · y_i · G
```

should be generated by each participant and verified by the coordinator before accumulating the contribution. This is consistent with the proof-of-knowledge pattern already used in `proof_of_knowledge` / `internal_verify_proof_of_knowledge` in `src/dkg.rs`. [7](#0-6)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}
