### Title
Bank Precompile `transfer` Executes Unauthorized Native Token Transfers Without Caller-Sender Verification - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile (`0x0000...0064`) accepts an arbitrary `sender` address from ABI-encoded input arguments and executes a native coin transfer from that address without verifying that `contract.Caller()` matches the supplied `sender`. Any EVM contract can therefore drain any account's balance of that contract's native denom (`evm/<caller_address>`) without the account holder's consent.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    args, err := method.Inputs.Unpack(contract.Input[4:])
    ...
    sender    := args[0].(common.Address)   // ← fully attacker-controlled
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    ...
    from  := sdk.AccAddress(sender.Bytes())
    to    := sdk.AccAddress(recipient.Bytes())
    ...
    denom := EVMDenom(contract.Caller())    // "evm/<calling_contract>"
    amt   := sdk.NewCoin(denom, ...)
    ...
    bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no assertion that `contract.Caller() == sender`. The `denom` is correctly scoped to the calling contract's address, but the `from` account is taken verbatim from the caller-supplied ABI arguments. The Cosmos SDK `SendCoins` call then moves native coins out of the victim's account with no approval or ownership check.

The same structural flaw exists in the `burn` branch: `recipient` (the address whose tokens are burned) is also taken from input args with no check that `contract.Caller() == recipient`. [2](#0-1) 

### Impact Explanation
**Critical — Unauthorized transfer of native tokens.**

Any EVM contract at address `A` can call `bank.transfer(victim, attacker, amount)` and move `evm/A`-denominated native coins out of `victim`'s Cosmos account without any approval. Because the bank precompile is the canonical mechanism for CRC20/CRC21 contracts to manage their native-side balances, users who hold `evm/A` tokens (e.g., as collateral in a DeFi protocol built on contract `A`) can have their entire balance stolen in a single transaction. The attacker needs no special privilege beyond deploying a contract.

### Likelihood Explanation
Any contract author can exploit this immediately. The bank precompile is a publicly callable address (`0x0000000000000000000000000000000000000064`). No leaked keys, governance access, or cryptographic break is required. The only precondition is that victim accounts hold a non-zero balance of the calling contract's native denom, which is the normal state for any protocol that uses the bank precompile to issue tokens.

### Recommendation
Add a caller-equality guard before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    // Require the EVM caller to be the sender, or implement an allowance model.
    if contract.Caller() != sender {
        return nil, errors.New("bank transfer: caller is not the sender")
    }
    ...
```

Apply the same guard to the `burn` branch (verify `contract.Caller() == recipient` before burning from an arbitrary address).

### Proof of Concept
1. Attacker deploys `MaliciousContract` at EVM address `0xATK`.
2. Victim legitimately acquires `evm/0xATK` tokens (e.g., by interacting with `0xATK` or receiving them as rewards).
3. Attacker calls `MaliciousContract.drain(victim_address)`, which internally calls:
   ```solidity
   IBankModule(0x0000000000000000000000000000000000000064)
       .transfer(victim_address, attacker_address, victim_balance);
   ```
4. The bank precompile unpacks `sender = victim_address`, computes `denom = "evm/0xATK"`, and calls `bankKeeper.SendCoins(victim, attacker, evm/0xATK coins)` — no authorization check is performed.
5. Victim's entire `evm/0xATK` balance is transferred to the attacker in one transaction. [3](#0-2)
