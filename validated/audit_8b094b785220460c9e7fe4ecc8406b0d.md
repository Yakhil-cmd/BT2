### Title
Malicious CKD Participant Can Corrupt Protocol Output Without Detection — (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator blindly aggregates elliptic-curve point contributions from all participants with no proof of correctness. A single malicious participant can substitute arbitrary group elements for their legitimate share, causing the coordinator to output an attacker-controlled `CKDOutput` that honest parties accept as valid.

### Finding Description
The vulnerability class from the external report is **missing validation on an operation that costs the attacker nothing**, allowing manipulation of shared state before honest parties can detect it. The direct analog here is in `do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs`.

Each participant is supposed to compute and send:

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

where `x_i` is their secret signing share and `y_i` is a fresh random scalar.

The coordinator aggregates these contributions:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no zero-knowledge proof, no commitment, and no consistency check** on the received `(norm_big_y, norm_big_c)` pair. The coordinator cannot distinguish a correctly computed share from an arbitrary group element. A malicious participant simply sends `(P, Q)` for any points `P, Q ∈ G₁` of their choosing.

Compare with the DKG in `src/dkg.rs`, which enforces a proof-of-knowledge (`proof_of_knowledge` / `verify_proof_of_knowledge`) and a hash commitment (`verify_commitment_hash`) before accepting any participant's polynomial contribution. The CKD protocol has no equivalent safeguard. [1](#0-0) [2](#0-1) 

### Impact Explanation
The final `CKDOutput` is the sum of all participant contributions. One malicious participant shifts both `norm_big_y` and `norm_big_c` by arbitrary offsets, producing an output that is cryptographically indistinguishable from a legitimately computed one to any observer who does not know the individual shares. The TEE application that calls `ckd_output.unmask(app_sk)` will derive a confidential key that is entirely wrong and unrelated to the master secret key. All honest parties accept this corrupted output as the protocol result.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable or inconsistent cryptographic outputs.** [3](#0-2) 

### Likelihood Explanation
Any participant in the CKD session is an attacker-controlled entry point. The attack requires no cryptographic capability beyond the ability to send a network message — the malicious participant simply omits the honest computation and sends two arbitrary `ElementG1` values. There is no detection mechanism, no abort, and no retry. The coordinator will silently produce and return the corrupted output. [4](#0-3) 

### Recommendation
Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that:

1. `norm_big_y` is a scalar multiple of `G` consistent with the participant's committed randomness `y_i`.
2. `norm_big_c` is formed using the participant's public verification share (derivable from the DKG output) and the same `y_i`.

A Chaum-Pedersen / Sigma-protocol proof of discrete-log equality between `norm_big_y` and the `y_i`-component of `norm_big_c` suffices. The coordinator must verify all proofs before summing. This mirrors the proof-of-knowledge pattern already used in `do_keyshare` for DKG contributions. [5](#0-4) 

### Proof of Concept

1. Honest participants `P_1 … P_{n-1}` run `compute_signature_share` correctly and send their `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant `P_m` skips `compute_signature_share` entirely and sends `(ElementG1::generator(), ElementG1::generator())` — two arbitrary fixed points — to the coordinator via `chan.send_private`.
3. The coordinator executes the loop at lines 50-55, summing all contributions including `P_m`'s bogus pair.
4. The resulting `CKDOutput` is `(Σ norm_big_y_i + G, Σ norm_big_c_i + G)` — shifted by `(G, G)` from the correct value.
5. The coordinator returns this as `Some(ckd_output)` with no error.
6. The TEE application calls `ckd_output.unmask(app_sk)` and derives a key that does not equal `msk · H(pk ‖ app_id)`, permanently breaking confidential key derivation for this session. [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,

    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,

    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
