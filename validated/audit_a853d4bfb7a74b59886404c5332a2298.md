### Title
`setPoolAdminFeeDestination` Omits Pre-Collection, Redirecting All Accrued Admin Fees to New Destination - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-config mutator (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` at the old rates and old destination before writing new state. The destination setter skips this step, so all spread and notional fees that accrued since the last collection are silently redirected to the new address on the next `collectPoolFees` call.

---

### Finding Description

`setPoolAdminFees` (lines 408–435) and `setPoolProtocolFee` (lines 318–360) both follow the same safe pattern: call `collectFees` with the current `poolAdminFeeDestination[pool]` first, then update state. This ensures fees earned under the old configuration are settled before any change takes effect.

`setPoolAdminFeeDestination` (lines 438–447) breaks this pattern:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

`collectPoolFees` (lines 379–389) reads `poolAdminFeeDestination[pool]` at call time:

```solidity
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← uses whatever is stored NOW
    );
}
```

After `setPoolAdminFeeDestination` executes, the next `collectPoolFees` call (permissionless — any caller) sends the entire accrued admin share (both spread surplus and `notionalFeeToken{0,1}Scaled`) to the new destination, not the old one.

---

### Impact Explanation

All admin fees that accrued between the last collection and the destination change are permanently redirected. The old destination (which may be a DAO treasury, revenue-sharing contract, or a different party from the admin) receives nothing for that period. The new destination receives fees it did not earn. The magnitude equals the full admin share of spread surplus plus notional accumulator at the moment of the next collection — potentially the entire lifetime of uncollected fees if collection has been infrequent.

---

### Likelihood Explanation

The pool admin is semi-trusted and can call `setPoolAdminFeeDestination` at any time with no timelock. A single transaction is sufficient to redirect all pending fees. The trigger is reachable by the pool admin alone, requires no special market conditions, and `collectPoolFees` is permissionless so any keeper or bot can finalize the redirect immediately after.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` at the current rates and current destination before overwriting `poolAdminFeeDestination[pool]`.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the OLD destination first
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed; admin sets `adminFeeDestination = DAOTreasury`.
2. Pool trades actively; spread surplus and `notionalFeeToken{0,1}Scaled` accumulate — say $50 k in admin-share fees.
3. Admin calls `setPoolAdminFeeDestination(pool, adminWallet)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is now `adminWallet`.
4. Any address calls `collectPoolFees(pool)`. `collectFees` is invoked with `adminFeeDestination_ = adminWallet`.
5. All $50 k in accrued admin fees are transferred to `adminWallet`. `DAOTreasury` receives zero.

The root cause is the missing `collectFees` call in `setPoolAdminFeeDestination`, directly analogous to the external report's missing `updateAccumulatedETHPerLP()` call — in both cases a state-update that must precede a configuration change is omitted, causing the wrong recipient to receive value that was earned under the old configuration.
