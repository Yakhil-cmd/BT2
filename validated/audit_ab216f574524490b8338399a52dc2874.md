### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke Transactions via Non-Deploy-Account Pool Presence — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call at the gateway only when a `deploy_account` transaction is pending for a not-yet-deployed account. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the pool or recent committed block, not exclusively for `deploy_account` transactions. An attacker who controls an account deployed via the `deploy` syscall (nonce=0, code present) can first submit a valid nonce-0 invoke (which passes full `__validate__`), then submit a nonce-1 invoke with an **invalid signature**. The gateway skips `__validate__` for the nonce-1 invoke and admits it to the mempool without signature verification.

---

### Finding Description

In `extract_state_nonce_and_run_validations`, after nonce and resource-bound pre-checks pass, the gateway calls `skip_stateful_validations`: [1](#0-0) 

The skip condition fires when `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` and `account_tx_in_pool_or_recent_block` returns `true`. The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This claim is incorrect: a nonce-0 invoke transaction (not a deploy_account) also satisfies the check, because `account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

which returns `true` for any transaction type in the pool. When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate = false` for an Invoke transaction, the `__validate__` entry point is never called: [4](#0-3) 

The transaction is then forwarded to the mempool without any account-side signature verification.

---

### Impact Explanation

An attacker controlling an account deployed via the `deploy` syscall (code present, nonce=0, no deploy_account transaction) can:

1. Submit a valid nonce-0 invoke from that account. This passes full `__validate__` and is admitted to the mempool. `account_tx_in_pool_or_recent_block` now returns `true` for this address.
2. Submit a nonce-1 invoke with an **invalid signature** (or arbitrary calldata targeting any contract). The gateway sees `tx_nonce=1, account_nonce=0`, calls `account_tx_in_pool_or_recent_block` → `true`, sets `validate=false`, and admits the transaction to the mempool without calling `__validate__`.

The invalid transaction is admitted to the mempool and will only be rejected during block building when the batcher calls `__validate__` with `validate=true`. This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The attacker can repeat this pattern (deploying fresh accounts via `deploy` syscall) to continuously inject invalid transactions, polluting the mempool and wasting batcher execution resources.

---

### Likelihood Explanation

Medium. The attacker must deploy a contract via the `deploy` syscall and pay fees for the nonce-0 invoke. However, this is a standard Starknet operation requiring no privileged access. The skip window is narrow (only nonce=1 transactions), but the setup cost is a one-time per-account expense, and the attack can be repeated with new accounts.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction. Alternatively, remove the skip entirely and always run `__validate__` for invoke transactions — the UX cost is a single failed validation for the post-deploy invoke, which is recoverable by the client.

Additionally, the `max_nonce_for_validation_skip` field in `StatefulTransactionValidatorConfig` is defined but unused in the gateway's `skip_stateful_validations`: [5](#0-4) 

The gateway hardcodes `Nonce(Felt::ONE)` instead of reading this config value, creating a silent divergence from the intended configurable behavior.

---

### Proof of Concept

```
1. Deploy contract C at address A via `deploy` syscall from another account.
   - A has code (valid __validate__), nonce=0, no deploy_account tx.

2. Submit InvokeV3 from A, nonce=0, valid signature.
   - Gateway: account_nonce=0, tx_nonce=0 → nonce check passes (0 <= 0 <= 200).
   - skip_stateful_validations: tx_nonce != 1 → returns false → __validate__ IS called → passes.
   - Admitted to mempool. account_tx_in_pool_or_recent_block(A) now returns true.

3. Submit InvokeV3 from A, nonce=1, INVALID signature (e.g., all-zero signature).
   - Gateway: account_nonce=0, tx_nonce=1 → nonce check passes.
   - skip_stateful_validations: tx_nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true
     → returns true (skip).
   - run_validate_entry_point: validate=false → __validate__ NOT called.
   - Transaction admitted to mempool with invalid signature.

4. Batcher fetches the nonce-1 invoke, calls __validate__ with validate=true → fails.
   Transaction rejected during block building.
``` [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
