### Title
Unbounded `ArrayVec::push` in `apply_to_array_vec` causes panic when cumulative L2-to-L1 logs across multiblock batch exceed 16 384 — (`zk_ee/src/common_structs/logs_storage.rs`)

### Summary
`LogsStorage::apply_to_array_vec` unconditionally calls `ArrayVec::push` into a fixed-capacity `ArrayVec<Bytes32, 16384>`. In a multiblock batch, this is called once per block. Because the per-block limit is also 16 384, two or more blocks with a combined log count exceeding 16 384 will overflow the accumulator and panic the RISC-V execution, making the batch unprovable.

### Finding Description
`ZKBatchDataKeeper.logs_storage` is declared as `ArrayVec<Bytes32, 16384>`. [2](#0-1) 

After each block executes, `post_op` calls:
```rust
io.logs_storage.apply_to_array_vec(&mut batch_data.logs_storage);
``` [4](#0-3) 

`apply_to_array_vec` iterates every log and calls `array_vec.push(log.hash())`: [3](#0-2) 

`arrayvec::ArrayVec::push` panics when `len == capacity`. There is no capacity check before or inside this function. The per-block limit (`MAX_NUMBER_OF_LOGS = 16_384`) only prevents a single block from exceeding 16 384 logs; it does not bound the batch total. [6](#0-5) 

### Impact Explanation
A panic in the RISC-V proving execution aborts the run. The batch cannot produce a valid public input. The operator must either discard the batch or re-sequence it with fewer logs per batch, causing a liveness disruption for the multiblock-batch proving path.

### Likelihood Explanation
The multiblock-batch feature is a production binary (`multiblock_batch.bin`). Any user can emit L2-to-L1 logs via the L1 Messenger hook. Filling two blocks with ~8 200 logs each (well within the per-block limit) is sufficient to trigger the panic. No privileged access is required.

### Recommendation
Replace `array_vec.push(log.hash())` with `array_vec.try_push(log.hash())` and propagate an error if capacity is exceeded, or enforce a batch-level cumulative log limit (e.g., `batch_data.logs_storage.remaining_capacity() >= block_log_count`) before calling `apply_to_array_vec`. The batch-level cap should be enforced during sequencing so that a batch is sealed before the 16 384 total is reached.

### Proof of Concept
1. Deploy a contract that calls `L1Messenger.sendToL1(bytes)` in a loop.
2. Submit transactions filling Block 1 with 8 193 L2-to-L1 logs (passes `check_for_block_limits`).
3. Submit transactions filling Block 2 with 8 192 L2-to-L1 logs (also passes per-block check).
4. Run the multiblock-batch prover over these two blocks.
5. Block 1's `apply_to_array_vec` fills `logs_storage` to 8 193 entries.
6. Block 2's `apply_to_array_vec` attempts to push entry 16 385 into a capacity-16 384 `ArrayVec` → **panic** → proving aborts.
