### Title
Missing Zero Address Check in `change_reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both `Staking::change_reward_address` and `Pool::change_reward_address` accept a zero address as the new `reward_address` without any validation. A staker or pool member can set their reward address to `ContractAddress::zero()`, causing all subsequent reward transfers to be sent to the zero address, permanently destroying unclaimed yield.

### Finding Description
`Staking::change_reward_address` (staking.cairo line 517) performs only one validation: that the new address is not a registered token address. There is no check that `reward_address.is_non_zero()`. [1](#0-0) 

Similarly, `Pool::change_reward_address` (pool.cairo line 505) only checks that `reward_address != token_dispatcher.contract_address`, with no zero address guard. [2](#0-1) 

When rewards are subsequently claimed, `send_rewards_to_staker` transfers STRK directly to `staker_info.reward_address`: [3](#0-2) 

And `Pool::claim_rewards` transfers to `pool_member_info.reward_address`: [4](#0-3) 

If `reward_address` is zero, the `checked_transfer` call sends tokens to address(0), permanently burning them. The same issue exists in the initial `stake()` call, which also accepts a zero `reward_address` without validation. [5](#0-4) 

### Impact Explanation
Any accumulated unclaimed STRK rewards are permanently sent to address(0) and irrecoverable. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**. The `unstake_action` flow also calls `send_rewards_to_staker` before removing the staker record, meaning rewards are burned at exit time with no recovery path. [6](#0-5) 

### Likelihood Explanation
The entry path is fully unprivileged — any registered staker or pool member can call `change_reward_address(0)` directly. No special role, leaked key, or external dependency is required. User error (e.g., passing a zero address by mistake) or a malicious actor deliberately burning their own rewards (griefing the accounting) are both realistic triggers.

### Recommendation
Add a zero address guard in both `change_reward_address` implementations and in `stake()`:

```cairo
// In Staking::change_reward_address and Pool::change_reward_address
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);

// In Staking::stake
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The error variant `GenericError::ZERO_ADDRESS` already exists in the codebase (used in `add_token`). [7](#0-6) 

### Proof of Concept

1. Staker stakes normally with a valid reward address.
2. Staker accumulates unclaimed STRK rewards over several epochs.
3. Staker calls `staking.change_reward_address(reward_address: ContractAddress::zero())` — succeeds with no revert.
4. Staker (or anyone) calls `staking.claim_rewards(staker_address)`.
5. `send_rewards_to_staker` executes `checked_transfer(recipient: 0x0, amount: rewards)` — tokens are sent to the zero address and permanently lost.
6. The same sequence applies to a pool member calling `pool.change_reward_address(0)` followed by `pool.claim_rewards(pool_member)`. [8](#0-7) [9](#0-8)

            // Asserts.

            let caller_address = get_caller_address();

                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.

            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
