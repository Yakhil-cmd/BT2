Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Loops Can Permanently Block All Periodic Tasks - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` function executes two unbounded loops — ballot aggregation over all proposals × all voters, and maturity distribution over all rewarded neurons — in a single synchronous message with no instruction-limit check. At the configured ceilings (200,000 neurons, 700 proposals), the computation can exceed the IC's 40-billion-instruction per-message limit, causing a trap. Because `latest_reward_event` is written only at the very end of the function, a trap rolls back all state, `should_distribute_rewards` returns `true` again on the next tick, and every subsequent `run_periodic_tasks` invocation re-traps at the same point — permanently blocking reward distribution, upgrade finalization, and maturity disbursements.

## Finding Description

**Loop 1 — ballot aggregation** at `rs/sns/governance/src/governance.rs` lines 5894–5931: iterates over every entry in `considered_proposals` and, for each proposal, over every `(voter, ballot)` pair in `proposal.ballots`. At the ceiling values (`MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700`, `MAX_NUMBER_OF_NEURONS_CEILING = 200_000`), this is up to 140,000,000 HashMap insert/update operations in a single synchronous call with no early-exit or instruction-limit check. [1](#0-0) [2](#0-1) 

**Loop 2 — maturity distribution** at lines 5954–5997: iterates over every entry in `neuron_id_to_reward_shares` (up to 200,000 neurons) and mutates each neuron's maturity fields. No instruction-limit check exists here either. [3](#0-2) 

**`latest_reward_event` is written only at the very end** of the function (lines 6084–6092). If the message traps before reaching this write, the IC rolls back all state changes. [4](#0-3) 

**`should_distribute_rewards`** checks `seconds_since_last_reward_event > round_duration_seconds` using `self.latest_reward_event().end_timestamp_seconds`. Because the event was never written, this condition remains `true` on every subsequent tick, causing the same failing computation to be retried indefinitely. [5](#0-4) 

**`run_periodic_tasks`** calls `distribute_rewards` synchronously (not in a separate message) at line 5513. A trap in `distribute_rewards` aborts the entire `run_periodic_tasks` execution, also blocking `maybe_finalize_disburse_maturity`, `maybe_move_staked_maturity`, and upgrade checks that follow it. [6](#0-5) 

**Contrast with NNS governance**, which introduced `RewardsDistributionInProgress` with a `continue_processing` method that checks `is_over_instructions_limit()` after each neuron and resumes across multiple messages. [7](#0-6) 

## Impact Explanation

When the instruction limit is exceeded, the IC traps the entire `run_periodic_tasks` message. Because `latest_reward_event` is never updated, every subsequent periodic timer invocation re-enters the same failing computation. The result is a permanent liveness failure of the SNS governance canister: voting rewards are never distributed, upgrade proposals cannot be finalized, and maturity disbursements stall. This matches the allowed impact: **High — Significant SNS security impact with concrete user or protocol harm**, specifically application-level DoS of the SNS governance canister with no recovery path short of an emergency canister upgrade.

## Likelihood Explanation

No attacker action is required. Normal protocol operation — users staking neurons, submitting proposals, and voting — drives the system toward this state. The `MAX_NUMBER_OF_NEURONS_CEILING = 200_000` and `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING = 700` are hard ceilings enforced by parameter validation. A popular SNS configured near these ceilings (e.g., 100,000+ neurons, 50+ proposals settling in one round) can trigger the instruction-limit trap. The NNS governance CHANGELOG explicitly documents that this exact problem was identified and fixed for NNS; SNS governance has no equivalent protection.

## Recommendation

Apply the same batched-distribution pattern used by NNS governance:

1. After computing `neuron_id_to_reward_shares`, persist it to stable storage (analogous to `RewardsDistributionInProgress`).
2. Write `latest_reward_event` immediately so the round is marked as settled and the computation is not retried.
3. Distribute maturity to neurons in a separate periodic timer task that checks `is_message_over_threshold` after each neuron and resumes in subsequent messages until distribution is complete.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700`.
2. Have 200,000 users stake neurons and vote on 700 proposals.
3. Wait for the reward round to end. `run_periodic_tasks` calls `distribute_rewards`.
4. `distribute_rewards` enters the ballot-aggregation loop (up to 140,000,000 HashMap operations) and the maturity-distribution loop (200,000 neuron mutations) in a single synchronous message.
5. The message exceeds the IC's 40-billion-instruction limit and traps.
6. `latest_reward_event` is not updated; `should_distribute_rewards` returns `true` on the next tick.
7. Every subsequent `run_periodic_tasks` invocation traps at the same point — permanently blocking all SNS periodic tasks.

A deterministic integration test can reproduce this using PocketIC with a mocked environment that counts instructions, or by constructing a `Governance` state with 200,000 neurons each having voted on 700 proposals and calling `distribute_rewards` directly, asserting that it panics or exceeds a simulated instruction budget.

            match neuron_store.with_neuron_mut(&id, |neuron| {
