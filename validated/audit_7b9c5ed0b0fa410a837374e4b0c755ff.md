### Title
Zero Native Resource Charge for All EIP-2537 BLS12-381 Precompiles Allows Prover Resource Exhaustion - (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

All seven EIP-2537 BLS12-381 precompiles (`G1ADD`, `G2ADD`, `G1MSM`, `G2MSM`, `PAIRING_CHECK`, `MAP_FP_TO_G1`, `MAP_FP2_TO_G2`) charge `cost_native = 0` for the native resource dimension of ZKsync OS's double resource accounting model. The native resource is the system's model for prover (RISC-V zkVM) complexity. Because BLS12-381 operations are among the most expensive cryptographic operations to prove, charging zero native resource allows an unprivileged transaction sender to force the prover to consume actual RISC-V cycles far in excess of what the native resource budget predicts, causing prover DoS.

---

### Finding Description

ZKsync OS implements a **double resource accounting model**: every operation charges both EVM gas (ergs) and a **native resource** that models proving complexity in RISC-V cycles. The native resource limit per transaction is derived from `gasLimit * nativePerGas` and capped at `MAX_NATIVE_COMPUTATIONAL`. [1](#0-0) 

The native resource is the mechanism that prevents a transaction from forcing the prover to do more work than the sender paid for. Every other computationally expensive precompile charges a non-zero native cost: [2](#0-1) 

However, all seven EIP-2537 BLS12-381 precompiles hard-code `cost_native = 0`:

**G1 Addition and G2 Addition** (`addition.rs`): [3](#0-2) [4](#0-3) 

**G1MSM and G2MSM** (`msm.rs`): [5](#0-4) [6](#0-5) 

**Pairing Check** (`pairing.rs`): [7](#0-6) 

**Field-to-G1 and Field-to-G2 Mappings** (`mappings.rs`): [8](#0-7) 

The EIP-2537 feature is enabled in the `eth_stf` configuration used for Ethereum-equivalent execution: [9](#0-8) 

The precompiles are registered and reachable from any EVM call: [10](#0-9) 

**Attacker-controlled entry path:**

1. Attacker submits a transaction calling the BLS12-381 G2MSM precompile (address `0x0e`) or pairing check (`0x0f`) with the maximum number of pairs that fit within the block gas limit.
2. The EVM gas cost is charged correctly (e.g., G2MSM with 128 pairs costs ~1.5M EVM gas per call).
3. The native resource charged is `0` regardless of how many pairs are processed.
4. The prover must execute the full Peppinger MSM algorithm over BLS12-381 G2 points, consuming actual RISC-V cycles that are orders of magnitude larger than what the native resource budget accounts for.
5. The attacker can repeat this across multiple transactions in a block, each consuming 0 native resource while forcing maximum prover work.

The `msm()` function itself is a full Peppinger multi-scalar multiplication with `O(n * 256 / log(n))` group operations over BLS12-381 G2, each of which is far more expensive to prove than BN254 operations (which charge `BN254_PAIRING_BASE_NATIVE_COST ≈ 13,000,000` native units per pair): [11](#0-10) 

---

### Impact Explanation

The native resource is the only mechanism that bounds the prover's actual RISC-V cycle consumption per transaction. With `cost_native = 0` for all BLS12-381 operations, the native resource budget is never depleted by these calls. An attacker can fill a transaction's entire EVM gas budget with BLS12-381 G2MSM or pairing calls, forcing the prover to execute an unbounded (relative to native resource) amount of cryptographic work.

Concrete impact:
- **Prover DoS**: The prover's actual RISC-V cycle consumption can exceed the cycle budget implied by the native resource charged, causing proof generation to fail or take disproportionately long.
- **Sequencer throughput degradation**: Blocks containing such transactions take far longer to prove than the native resource accounting predicts, stalling finalization.
- **Repeated on-demand**: Any unprivileged user can submit such transactions at will, paying only EVM gas (which is economically bounded) while forcing unbounded prover work.

This matches the Immunefi impact category: **Increasing network processing node resource consumption by at least 30% without brute force actions** (prover node), and potentially **Taking down the application** (prover crash/timeout).

---

### Likelihood Explanation

- **No privileged access required**: Any account with sufficient ETH to pay EVM gas can submit the attack transaction.
- **On-demand**: The attacker controls exactly when and how many BLS12-381 operations are triggered.
- **Deterministic**: The attack is reproducible and does not depend on network state.
- **EIP-2537 is Prague-hardfork standard**: The feature is enabled in the `eth_stf` configuration targeting Ethereum equivalence.

---

### Recommendation

Add non-zero native resource costs for all BLS12-381 precompiles, calibrated against actual RISC-V cycle measurements. As a reference, BN254 pairing charges `native_with_delegations!(13_000_000, 500_000, 0)` per pair. BLS12-381 G2MSM and pairing are significantly more expensive and should carry proportionally higher native costs. The pattern used by other precompiles (e.g., `BN254_ECMUL_NATIVE_COST`, `ECRECOVER_NATIVE_COST`) should be followed: [12](#0-11) 

A `TODO(EVM-1178)` comment already exists for `POINT_EVALUATION_NATIVE_COST`, indicating the team is aware of the need to benchmark and set native costs. The same process should be applied to all EIP-2537 precompiles before production deployment.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract BLS12DoS {
    // BLS12-381 G2MSM precompile address (0x0e)
    address constant G2MSM = address(0x0e);

    // Call G2MSM with 128 pairs (maximum discount tier)
    // Each call costs ~1.5M EVM gas but 0 native resource
    function attack() external {
        // 128 pairs * (G2_SERIALIZATION_LEN=256 + SCALAR_SERIALIZATION_LEN=32) = 36,864 bytes
        // Use identity point (infinity) with scalar=1 for valid but cheap-to-construct input
        bytes memory input = new bytes(128 * 288); // 128 pairs
        // ... populate with valid G2 points and scalars ...
        
        // Repeat until gas exhausted - each call charges 0 native resource
        for (uint i = 0; i < 10; i++) {
            (bool ok,) = G2MSM.call{gas: 2_000_000}(input);
            require(ok);
        }
    }
}
```

Submit a transaction calling `attack()` with `gasLimit = 30_000_000`. The transaction:
1. Passes EVM gas validation (gas is sufficient).
2. Passes native resource validation (0 native consumed by BLS12-381 calls).
3. Forces the prover to execute 1,280 BLS12-381 G2 multi-scalar multiplications (128 pairs × 10 calls), each requiring hundreds of field multiplications over the BLS12-381 base field in the RISC-V zkVM.
4. The prover's actual cycle consumption is orders of magnitude beyond what the native resource budget (0) predicted.

fn msm<G: CurveGroup, A: core::alloc::Allocator + Clone>(
    bases: &[G::Affine],
    mut bigints: Vec<<Fr as PrimeField>::BigInt, A>,
    allocator: A,
) -> G {
    assert_eq!(bases.len(), bigints.len());
    let size = bases.len();

    const NUM_BITS: usize = 256;

    // let's special-case for "short" invocations
    if size < 4 {
        // it'll amortize largely over final projective -> affine, and doubling loop
        let mut acc = G::ZERO;
        for bit in (0..NUM_BITS).rev() {
            let word_idx = bit / 64;
            let bit_idx = bit % 64;
            unsafe {
                core::hint::assert_unchecked(bases.len() == bigints.len());
            }

            for i in 0..size {
                if bigints[i].0[word_idx] & 1 << bit_idx > 0 {
                    acc += &bases[i];
                }
            }
            if bit > 0 {
                acc.double_in_place();
            }
        }

        return acc;
    }
