### Title
Owner Can Replace ETH Connector Account Without Migrating User Balances, Freezing Withdrawals - (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `set_eth_connector_contract_account` function allows the engine owner to atomically overwrite the stored ETH connector account pointer with no balance migration or settlement step. After the swap, every user `withdraw` call is routed to the new connector, which holds none of the users' NEP-141 collateral. Users who deposited before the swap cannot withdraw their ETH from Aurora to Ethereum through the engine, freezing those funds.

### Finding Description
`set_eth_connector_contract_account` (lines 418–438) performs two storage writes — `set_connector_account_id` and `set_connector_withdraw_serialization_type` — and returns immediately. There is no checkpoint, no balance migration, and no guard that verifies the new connector holds equivalent collateral. [1](#0-0) 

The `withdraw` function (lines 43–58) constructs an `EngineWithdrawCallArgs` and calls `return_promise`, which internally calls `get_connector_account_id` to resolve the target account at call time. [2](#0-1) 

`get_connector_account_id` reads the single storage slot written by `set_connector_account_id`, so it always returns the most recently set account. [3](#0-2) 

The engine maintains its own EVM balance ledger (`KeyPrefix::Balance`) independently of the connector's NEP-141 ledger. `get_balance` / `set_balance` read and write this engine-side storage. [4](#0-3) 

When a deposit is processed, the engine credits the EVM balance AND the old connector mints NEP-141 tokens as collateral. These are two separate representations of the same asset. After the connector pointer is swapped, the engine's EVM balance entries still exist, but the cross-contract call `engine_withdraw` is now directed at the new connector, which holds no NEP-141 tokens for those users. The call fails, the EVM balance is not decremented, and the user is stuck in a loop where every withdrawal attempt fails.

### Impact Explanation
Users who deposited ETH before the connector swap cannot withdraw their ETH from Aurora to Ethereum through the engine. Their EVM balance is intact in the engine's storage but unreachable via the standard withdrawal path. The old connector still holds the NEP-141 collateral, but `engine_withdraw` on the old connector is restricted to the engine account, so users cannot reclaim funds directly. This constitutes at minimum a **High — temporary freezing of funds**, and depending on whether the old connector exposes a permissionless withdrawal path, it may be **Critical — permanent freezing of funds**.

### Likelihood Explanation
The owner legitimately needs to upgrade or replace the ETH connector (e.g., to deploy a patched version). The function provides no warning, no migration step, and no atomic swap mechanism. A well-intentioned upgrade performed without a coordinated migration will silently freeze all pre-swap user balances. The likelihood of this occurring during a connector upgrade is realistic.

### Recommendation
Before overwriting the connector account pointer, the engine should either:
1. Require that the total NEP-141 supply on the old connector is zero (i.e., all users have already withdrawn), or
2. Perform an atomic migration that transfers all NEP-141 collateral from the old connector to the new one before updating the pointer, or
3. Emit a time-locked change with a mandatory notice period so users can withdraw before the pointer is updated.

This mirrors the MiniChefV2 recommendation: update (settle) the pool state before changing parameters.

### Proof of Concept
1. User calls `ft_on_transfer` / `deposit` → engine credits EVM balance; old connector mints NEP-141 tokens.
2. Owner calls `set_eth_connector_contract_account` with `new_connector` (e.g., an upgraded contract that holds zero NEP-141 tokens).
   - `set_connector_account_id` overwrites the storage slot immediately.
3. User calls `withdraw` on the engine.
   - `return_promise` resolves the connector account to `new_connector`.
   - NEAR schedules a call to `new_connector.engine_withdraw(sender_id, recipient_address, amount)`.
4. `new_connector` has no NEP-141 balance for the user → the call fails or reverts.
5. The engine's EVM balance for the user is unchanged (not decremented), but no Ethereum withdrawal is initiated.
6. Every subsequent `withdraw` attempt repeats steps 3–5. The user's ETH is permanently locked in Aurora with no standard exit path. [5](#0-4) [6](#0-5)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

pub fn set_balance<I: IO>(io: &mut I, address: &Address, balance: &Wei) {
    io.write_storage(
        &address_to_key(KeyPrefix::Balance, address),
        &balance.to_bytes(),
    );
}

pub fn remove_balance<I: IO + Copy>(io: &mut I, address: &Address) {
    io.remove_storage(&address_to_key(KeyPrefix::Balance, address));
}

pub fn get_balance<I: IO>(io: &I, address: &Address) -> Wei {
    let raw = io
        .read_u256(&address_to_key(KeyPrefix::Balance, address))
        .unwrap_or_else(|_| U256::zero());
    Wei::new(raw)
}
