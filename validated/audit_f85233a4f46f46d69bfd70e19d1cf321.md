### Title
RCAT Amount Uses Wrong Mojo Scaling Factor in NFT Offer Editor, Causing 1e9× Over-Offer — (File: `packages/gui/src/components/offers/NFTOfferEditor.tsx`)

---

### Summary

`buildOfferRequest` in `NFTOfferEditor.tsx` omits `WalletType.RCAT` from its CAT-type guard, falling back to `chiaToMojo` (÷ 1 × 10¹²) instead of `catToMojo` (÷ 1 × 10³) when a user creates an NFT offer denominated in RCAT. Because RCAT wallets are explicitly listed in `NFTOfferTokenSelector`, a user who selects RCAT and enters "1" will have an offer built for 1 000 000 000 RCAT instead of 1 RCAT — a 10⁹× accounting error that directly drains their RCAT balance when the offer is accepted.

---

### Finding Description

`buildOfferRequest` converts the user-entered token amount to mojos with this guard:

```typescript
// NFTOfferEditor.tsx line 493
const baseMojoAmount: BigNumber = [WalletType.CAT, WalletType.CRCAT].includes(tokenWalletInfo.walletType)
  ? catToMojo(tokenAmount)
  : chiaToMojo(tokenAmount);
``` [1](#0-0) 

`WalletType.RCAT = 132` is absent from the array `[WalletType.CAT (6), WalletType.CRCAT (57)]`, so the condition evaluates to `false` and `chiaToMojo` is used. `chiaToMojo` multiplies by 10¹², while RCAT (like all CATs) uses 10³ mojos per token. [2](#0-1) 

Meanwhile, `NFTOfferTokenSelector` explicitly includes RCAT wallets in its option list, making the path reachable by any user who has an RCAT wallet:

```typescript
// NFTOfferTokenSelector.tsx line 59
const catOptions = wallets
  .filter((wallet: Wallet) => [WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(wallet.type))
``` [3](#0-2) 

The correct path — `catToMojo` — is used everywhere else in the codebase for RCAT:

- `OfferEditor.tsx` `getUpdatedOffer`: `[WalletType.CAT, WalletType.RCAT, WalletType.CRCAT]` → `catToMojo` [4](#0-3) 
- `NFTOfferConditionalsPanel` balance display: `case WalletType.RCAT:` → `mojoToCAT` [5](#0-4) 

`buildOfferRequest` is the only site that forgets RCAT.

---

### Impact Explanation

When a user selects an RCAT wallet in the NFT offer editor and enters an amount (e.g., `1`):

| Step | Value |
|---|---|
| User input | 1 RCAT |
| Expected mojos (`catToMojo`) | 1 000 mojos |
| Actual mojos (`chiaToMojo`) | 1 000 000 000 000 mojos |
| Ratio | **10⁹ × over-offer** |

The offer is submitted to the node with this inflated mojo amount. If the counterparty accepts, the user's RCAT balance is debited by 10⁹ times what they intended to offer. This is a direct, irreversible asset loss affecting RCAT (a CAT-class token).

---

### Likelihood Explanation

Any user who holds an RCAT wallet and uses the NFT offer editor to buy or sell an NFT for RCAT will trigger this bug. RCAT wallets appear in the token selector dropdown without any warning. No attacker action is required — the user's own offer creation flow silently applies the wrong scaling factor.

---

### Recommendation

Add `WalletType.RCAT` to the type guard in `buildOfferRequest`, matching every other site in the codebase:

```typescript
// NFTOfferEditor.tsx
const baseMojoAmount: BigNumber = [WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(tokenWalletInfo.walletType)
  ? catToMojo(tokenAmount)
  : chiaToMojo(tokenAmount);
```

---

### Proof of Concept

1. User has an RCAT wallet with ≥ 1 000 000 000 RCAT (or any amount ≥ the inflated mojo value).
2. User opens the NFT offer editor ("Buy an NFT" tab).
3. User selects their RCAT wallet from the `NFTOfferTokenSelector` dropdown (RCAT is listed because the selector filters for `[CAT, RCAT, CRCAT]`).
4. User enters `1` in the amount field, intending to offer 1 RCAT.
5. `buildOfferRequest` is called; `[WalletType.CAT, WalletType.CRCAT].includes(WalletType.RCAT)` → `false` → `chiaToMojo("1")` → `1_000_000_000_000` mojos.
6. The offer is created requesting 1 000 000 000 RCAT from the user's wallet.
7. A counterparty accepts the offer; the user loses 10⁹ RCAT instead of 1 RCAT.
