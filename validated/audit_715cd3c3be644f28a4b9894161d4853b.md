### Title
Unbounded `MessageBuffer` Growth via Attacker-Injected Phantom Waitpoints — (`File: src/protocol/internal.rs`)

### Summary
The `MessageBuffer` inside `Comms` is a `HashMap<MessageHeader, SubMessageQueue>` with no size cap and no eviction mechanism. Any participant who can call `Protocol::message()` can inject messages tagged with arbitrary waitpoint values that the honest protocol code never polls. Each unique `(channel, waitpoint)` pair creates a permanent new HashMap entry backed by an unbounded MPSC channel. The codebase's own test (`attacker_can_fill_message_buffer_with_unused_waitpoints`) explicitly confirms this behavior is reachable and unmitigated.

### Finding Description

`MessageBuffer` is defined as an unbounded `HashMap`: [1](#0-0) 

Each `SubMessageQueue` is backed by a `futures::channel::mpsc::unbounded()` channel — itself unbounded: [2](#0-1) 

The `push` method unconditionally creates a new entry for any `MessageHeader` it has not seen before: [3](#0-2) 

`Protocol::message()` routes directly into this path: [4](#0-3) 

`push_message` only validates that the raw bytes are long enough to parse a `MessageHeader`; it performs no validation of whether the encoded `waitpoint` is one the running protocol will ever consume: [5](#0-4) 

Because `pop` only drains the entry for the specific `(channel, waitpoint)` the honest async code awaits, every entry created for a phantom waitpoint is **never consumed and never removed**. The HashMap grows monotonically for the lifetime of the `Comms` object.

The codebase's own test proves this end-to-end: 512 messages with distinct high-numbered waitpoints produce exactly 512 permanent HashMap entries: [6](#0-5) 

### Impact Explanation

A malicious participant in any threshold protocol (DKG, presign, sign, FROST, CKD) can send a stream of messages with distinct, never-polled waitpoint values. Each message allocates a new `HashMap` entry plus an unbounded MPSC channel pair. With no cap, the process heap grows without bound until the OS kills it or the node operator intervenes. This constitutes **unbounded memory exhaustion** causing **permanent denial of signing, key generation, reshare, refresh, or CKD** for all honest parties sharing that `Comms` instance — matching the Medium impact tier (griefing / resource-exhaustion by a malicious participant).

### Likelihood Explanation

Every participant in a threshold protocol is a valid caller of `Protocol::message()`. A single malicious co-signer, or any network peer that can deliver raw bytes to the protocol executor, can trigger this with no special privileges. The attack requires only the ability to send protocol messages — a capability every participant already has by design. The attack is trivially scriptable: send messages with sequentially incrementing waitpoint fields.

### Recommendation

Apply at least one of the following mitigations:

1. **Bound the HashMap**: Reject (or drop) any incoming message whose `(channel, waitpoint)` key would cause the `MessageBuffer` to exceed a configurable maximum entry count (e.g., `n_participants × max_expected_waitpoints`).
2. **Allowlist valid waitpoints**: Track which waitpoints the running protocol has registered interest in (via `pop` calls) and silently discard messages for unregistered waitpoints in `push`.
3. **Evict stale entries**: After a protocol round completes, remove all entries for waitpoints belonging to that round.

### Proof of Concept

The existing test in the repository is a direct PoC:

```rust
// src/protocol/internal.rs, lines 532-554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        comms.push_message(attacker, message);
    }

    // 512 permanent, never-consumed HashMap entries confirmed:
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
```

To escalate to OOM: replace `512` with an unbounded loop and run against a live DKG or signing session. The `Comms` object shared across all protocol rounds will accumulate entries until the process is killed, permanently aborting the threshold operation for all honest participants.
