### Title
Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price Instead of Next-Block Price for Resource-Bounds Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` and `run_validate_entry_point` both read the **previous committed block's** L2 gas price from `GatewayFixedBlockStateReader::get_block_info()` and use it to gate admission. The actual block being built by the batcher uses the **next** L2 gas price (computed via EIP-1559 mechanics from the previous block's gas consumption). When the two prices diverge — which is the normal case under any non-zero load — the gateway's admission decision is wrong: it admits transactions that will fail `check_fee_bounds` during execution, and rejects transactions that would be valid for the next block.

### Finding Description

**Root cause — `validate_resource_bounds` (lines 228–240):**

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
```

The developer-left TODO explicitly acknowledges the wrong value is used. The `next_l2_gas_price` field is already stored in `BlockHeaderWithoutHash` (used by the consensus orchestrator at `sequencer_consensus_context.rs:1727`) and in `BlockPostV0_13_1` (line 89), but `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` never reads it — it only copies `block_header.l2_gas_price` (the current block's price) into the returned `BlockInfo`.

**Second instance — `run_validate_entry_point` (lines 323–324):**

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
```

The block number is incremented to simulate the next block, but the gas prices inside `block_info` are **not** updated to the next block's prices. The `BlockContext` built from this stale `block_info` is then passed to `StatefulValidator::validate` → `AccountTransaction::perform_pre_validation_stage` → `check_fee_bounds`, which compares `resource_bounds.max_price_per_unit` against `block_info.gas_prices` — still the previous block's prices.

**The batcher uses the correct price.** When the batcher calls `propose_block`, it receives a `ProposeBlockInput` whose `block_info` is built from `ProposalInit.l2_gas_price_fri` — the freshly computed next-block price from `SequencerConsensusContext::l2_gas_price` (updated by `update_l2_gas_price` after each `decision_reached`). The batcher then re-runs `perform_pre_validation_stage` with this correct price.

**Divergence mechanics.** The next L2 gas price is computed by `calculate_next_l2_gas_price_for_fin` / `calculate_next_base_gas_price` using the previous block's gas consumption. Under any non-trivial load the two prices differ:

- **Rising price (block above target):** `next_price > previous_price`. Transactions with `previous_price ≤ max_price_per_unit < next_price` pass both gateway checks but fail `check_fee_bounds` in the batcher. The gateway admits invalid transactions.
- **Falling price (block below target):** `next_price < previous_price`. Transactions with `threshold(next_price) ≤ max_price_per_unit < threshold(previous_price)` are rejected by the gateway but would be valid for the next block. The gateway rejects valid transactions.

### Impact Explanation

**Admission of invalid transactions (rising price):** Transactions that satisfy the gateway's stale-price threshold enter the mempool and are handed to the batcher. The batcher's `check_fee_bounds` rejects them, wasting sequencer execution budget and polluting the mempool with transactions that can never be included. Under sustained high load — exactly when the price rises fastest — this effect is largest.

**Rejection of valid transactions (falling price):** Users whose `max_price_per_unit` is above the true next-block threshold but below the stale previous-block threshold receive a `GAS_PRICE_TOO_LOW` error and cannot submit transactions even though the next block would accept them. This is a liveness failure for those users.

Both outcomes match the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The L2 gas price changes every block via EIP-1559 mechanics. Any block that is not exactly at the gas target produces a different next price. This is the common case during normal network operation. No special attacker action is required; the discrepancy is structural and occurs on every transaction submitted while the price is moving.

### Recommendation

1. **Expose `next_l2_gas_price` through `GatewayFixedBlockStateReader`.** Add a `get_next_l2_gas_price` method (or include it in the returned `BlockInfo`) by reading `block_header_without_hash.next_l2_gas_price` in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`.

2. **Use it in `validate_resource_bounds`** (resolving the existing TODO):
   ```rust
   let next_block_l2_gas_price = self
       .gateway_fixed_block_state_reader
       .get_next_l2_gas_price()
       .await?;
   self.validate_tx_l2_gas_price_within_threshold(
       executable_tx.resource_bounds(),
       next_block_l2_gas_price,
   )?;
   ```

3. **Use it in `run_validate_entry_point`** when constructing the block context, replacing `block_header.l2_gas_price` with `block_header.next_l2_gas_price` in the `GasPrices` struct so that `check_fee_bounds` operates on the price the batcher will actually use.

### Proof of Concept

1. Observe the current committed block N has L2 gas price `P_N` and was above the gas target, so `P_{N+1} = P_N * (1 + δ)` for some `δ > 0`.
2. Submit an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P_N` (exactly the previous block's price, above the `min_gas_price_percentage` threshold).
3. `validate_resource_bounds` computes `threshold = (min_gas_price_percentage / 100) * P_N ≤ P_N` → **passes**.
4. `run_validate_entry_point` builds a `BlockContext` with `gas_prices.strk_gas_prices.l2_gas_price = P_N`; `check_fee_bounds` checks `P_N ≥ P_N` → **passes**. Transaction enters the mempool.
5. The batcher builds block N+1 with `l2_gas_price_fri = P_{N+1} > P_N`. When `perform_pre_validation_stage` runs with this context, `check_fee_bounds` checks `P_N ≥ P_{N+1}` → **fails**. The transaction is dropped.

/// Compute the next L2 gas price (for the fin or for updating state). Respects override when set.
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}

                            tx_context.get_gas_prices(),
