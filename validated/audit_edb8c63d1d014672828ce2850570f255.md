### Title
`calculateNFTRoyalties` Sets `nftSellerNetAmount` to Full Asking Price Instead of Net-of-Royalty Amount, Causing Incorrect "You Will Receive" Display in NFT Offer Flows - (File: `packages/gui/src/components/offers/utils.ts`)

### Summary

In `calculateNFTRoyalties`, the `nftSellerNetAmount` field is hardcoded to `amount` (the full asking price) instead of `amount - royaltyAmount`. The correct formula is commented out directly in the source. This causes the NFT offer editor and viewer to display an inflated "You will receive" / "Net Proceeds" figure to the NFT seller, misrepresenting the actual tokens they will receive after royalties are deducted.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, the `calculateNFTRoyalties` function computes royalty-related display values for NFT offers:

```typescript
export function calculateNFTRoyalties(...): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;          // ← BUG: should be amount - royaltyAmount
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  ...
}
``` [1](#0-0) 

The correct formula (`amount - royaltyAmountString - makerFee`) is explicitly commented out and replaced with just `amount`. As a result, `nftSellerNetAmount` always equals the full asking price regardless of the royalty percentage.

This value propagates into two user-facing flows:

**1. NFT Offer Editor (`NFTOfferEditor.tsx`):** When a seller creates an NFT-for-token offer, the UI renders "You will receive: [nftSellerNetAmount]" using the inflated value. [2](#0-1) 

**2. NFT Offer Viewer (`NFTOfferViewer.tsx`):** When viewing an offer, `overrideNFTSellerAmount` is computed from `nftSaleInfo?.nftSellerNetAmount` and passed to `NFTOfferSummary` to override the displayed seller-side amount. Additionally, the "Net Proceeds" section renders `nftSaleInfo?.nftSellerNetAmount` directly. [3](#0-2) [4](#0-3) 

### Impact Explanation

A seller creating an NFT-for-token offer with a 10% royalty on a 100 XCH asking price will see:

- **Displayed:** "You will receive: 100 XCH"
- **Actual on-chain receipt:** 90 XCH (10 XCH goes to the royalty recipient)

The seller approves and signs the offer based on a materially incorrect net proceeds figure. This is a direct accounting spoofing issue in the offer confirmation flow: the user is shown the wrong amount when deciding to create or accept an NFT offer, matching the "causes a user to approve … the wrong … amount" criterion.

The `overrideNFTSellerAmount` path additionally causes the offer summary shown to the accepting party to display an incorrect seller-side amount, potentially misleading both sides of the trade.

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this path. NFT royalties are a standard feature of the Chia NFT1 standard and are common in practice. The bug is always active whenever `calculateNFTRoyalties` is called with a non-zero royalty percentage, which is the normal case for royalty-bearing NFTs. [5](#0-4) 

### Recommendation

Restore the correct net-proceeds formula in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [6](#0-5) 

The `makerFee` inclusion should be conditioned on `exchangeType === NFTOfferExchangeType.NFTForToken` (i.e., only deduct the maker fee from the seller's net when the seller is paying it), consistent with how `totalAmount` is computed on line 318–319.

### Proof of Concept

1. Open the Chia GUI and navigate to the NFT Offer Editor.
2. Select "Sell an NFT" tab and choose an NFT with a 10% royalty (e.g., `royaltyPercentage = 1000` in raw units → 10%).
3. Enter a token amount of 100 XCH.
4. Observe: the UI displays **"You will receive: 100 XCH"** and **"Creator Fee: 10 XCH"** simultaneously — the net proceeds label does not subtract the royalty.
5. The correct display should be **"You will receive: 90 XCH"**.
6. The seller creates the offer believing they will net 100 XCH; on-chain they receive 90 XCH. [7](#0-6)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
