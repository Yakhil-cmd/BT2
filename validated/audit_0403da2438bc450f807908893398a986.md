### Title
Excess NEAR Attached to `fund_xcc_sub_account` Is Not Returned to Caller — (`engine/src/xcc.rs`)

---

### Summary

`fund_xcc_sub_account` accepts any NEAR deposit `>= STORAGE_AMOUNT` (2 NEAR) without returning the surplus. The entire `attached_deposit` is forwarded to the router sub-account, which the user cannot directly drain back. Any NEAR above the required storage-staking amount is silently locked in the router.

---

### Finding Description

`fund_xcc_sub_account` reads the caller's attached NEAR deposit and checks only that it is not *less than* `STORAGE_AMOUNT`:

```rust
// engine/src/xcc.rs  ~line 90, 116-122
let fund_amount = Yocto::new(env.attached_deposit());
...
if create_needed {
    if fund_amount < STORAGE_AMOUNT {          // non-strict: >= passes
        return Err(FundXccError::InsufficientBalance);
    }
    promise_actions.push(PromiseAction::Transfer {
        amount: fund_amount,                   // entire deposit forwarded
    });
``` [1](#0-0) [2](#0-1) 

The same pattern applies in the upgrade branch and the "no deploy needed" branch — in every code path the full `fund_amount` is transferred to the router sub-account:

```rust
// upgrade branch
promise_actions.push(PromiseAction::FunctionCall {
    attached_yocto: fund_amount,   // all of it
    ...
});

// no-deploy branch
promise_actions.push(PromiseAction::Transfer {
    amount: fund_amount,           // all of it
});
``` [3](#0-2) 

`STORAGE_AMOUNT` is a fixed constant of 2 NEAR:

```rust
pub const STORAGE_AMOUNT: Yocto = Yocto::new(2_000_000_000_000_000_000_000_000);
``` [4](#0-3) 

No refund promise is created for the excess. The `send_refund` callback that exists in the flow sends back exactly `REFUND_AMOUNT` (= `STORAGE_AMOUNT` = 2 NEAR) to the *engine* account — not to the original caller — and only when the engine itself fronted the storage cost:

```rust
// engine/src/xcc.rs
let refund_needed = match deploy_needed {
    AddressVersionStatus::DeployNeeded { create_needed } => create_needed,
    AddressVersionStatus::UpToDate => false,
};
if refund_needed {
    let refund_call = PromiseCreateArgs {
        target_account_id: promise.target_account_id.clone(),
        method: "send_refund".into(),
        ...
    };
``` [5](#0-4) 

The router sub-account (`{evm_address}.aurora`) is owned by the Aurora Engine contract, not by the user. The xcc-router exposes no user-callable withdrawal function for raw NEAR; the only egress is attaching NEAR to future XCC calls via the precompile. A user who never makes another XCC call, or who simply over-funded by mistake, has no path to recover the surplus.

---

### Impact Explanation

Any NEAR above `STORAGE_AMOUNT` attached to `fund_xcc_sub_account` is transferred into the router sub-account and cannot be directly recovered by the caller. The router is controlled by the Aurora Engine; the user can only spend the locked NEAR by routing it through future cross-contract calls. If the user does not intend to make further XCC calls, or if the over-payment is large, the funds are effectively frozen in the router sub-account for an indefinite period. This constitutes **temporary (potentially indefinite) freezing of user funds**, matching the High-severity impact tier.

---

### Likelihood Explanation

`fund_xcc_sub_account` is a public NEAR-level entry point callable by any NEAR account. A user who misreads the required amount, uses a wallet that rounds up, or follows documentation that does not warn about exact values will silently over-pay. The non-strict `<` check provides no feedback that the excess was not returned. Likelihood is **medium**: the function is not called on every transaction, but there is no guard preventing over-payment and no documentation warning.

---

### Recommendation

Replace the non-strict lower-bound check with a strict equality check, or add an explicit refund promise for the surplus:

```rust
// Option A – strict equality
if fund_amount != STORAGE_AMOUNT {
    return Err(FundXccError::IncorrectBalance);
}

// Option B – refund excess
let excess = fund_amount.checked_sub(STORAGE_AMOUNT).unwrap_or(ZERO_YOCTO);
if !excess.is_zero() {
    // schedule a Transfer back to env.predecessor_account_id()
}
```

---

### Proof of Concept

1. Alice calls `fund_xcc_sub_account` on the Aurora Engine NEAR contract, attaching **10 NEAR** (instead of the required 2 NEAR).
2. The check `fund_amount < STORAGE_AMOUNT` evaluates to `false` (10 ≥ 2), so execution continues.
3. `PromiseAction::Transfer { amount: fund_amount }` forwards all **10 NEAR** to `{alice_evm_address}.aurora`.
4. No refund promise is scheduled for the 8 NEAR surplus.
5. Alice's NEAR account is debited 10 NEAR; 8 NEAR sits in the router sub-account, inaccessible except through future XCC calls that attach NEAR to outgoing promises.
6. If Alice makes no further XCC calls, the 8 NEAR remains frozen in the router indefinitely. [6](#0-5)
