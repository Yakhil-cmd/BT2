### Title
`broadcast_success` Missing `err` Parameter Silently Drops Failure Signals, Permanently Blocking DKG/Reshare for Honest Parties - (`File: src/dkg.rs`)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` is documented to accept an `err` parameter so it can broadcast either success or failure to all participants at the end of DKG/reshare. The implementation is missing this parameter entirely and unconditionally broadcasts success. When a malicious participant causes an honest party to abort early (e.g., via an invalid proof of knowledge or invalid secret share), that honest party exits `do_keyshare` without ever reaching `broadcast_success`, leaving all other honest participants waiting indefinitely for a broadcast that will never arrive. This permanently blocks key generation, reshare, and refresh for all honest parties.

---

### Finding Description

**Root cause — missing `err` parameter in `broadcast_success`:** [1](#0-0) 

The doc-comment explicitly states the function "takes err as input" and should "broadcast failure" when that error is present. The actual signature has no such parameter; the body unconditionally passes `(true, session_id)` to `do_broadcast`. The failure-detection branch described in the comment was never implemented.

**How an honest participant aborts early:**

Inside `do_keyshare`, every verification step propagates errors with `?`: [2](#0-1) [3](#0-2) [4](#0-3) 

Any of these returning `Err` causes `do_keyshare` to return immediately. The participant never reaches the private-share sending step (lines 499–506) or the final broadcast step: [5](#0-4) 

**What other participants observe:**

All other honest participants are blocked at two subsequent `await` points:
1. `recv_from_others` at `wait_round_3` — waiting for the private share that the aborted participant will never send.
2. `do_broadcast` inside `broadcast_success` — waiting for the broadcast vote that the aborted participant will never emit.

Because `broadcast_success` has no way to carry a failure flag, the aborted participant cannot signal its peers to abort gracefully. The peers spin indefinitely.

**The check that was meant to catch failures is dead code:** [6](#0-5) 

This check would abort the protocol if any participant broadcast `false`. Because the `err` parameter was never added, no participant ever broadcasts `false`; the check is unreachable in practice.

---

### Impact Explanation

**High — Permanent denial of key generation, reshare, and refresh for honest parties.**

A single malicious participant can permanently stall every honest party's `keygen`, `reshare`, or `refresh` call. The honest parties have no mechanism to learn that a peer aborted, so they wait indefinitely. Because the library's `keygen`/`reshare` public API is a blocking async function, the entire signing group is rendered inoperable until an external timeout is applied — and even then, the DKG round cannot be cleanly restarted without external coordination, since some parties completed earlier rounds and some did not.

---

### Likelihood Explanation

**High.** Any participant in the protocol can trigger this by sending a syntactically valid but semantically invalid message — for example, a proof of knowledge that fails verification, or a secret share that does not lie on the committed polynomial. Both are trivially constructable. No special privilege, leaked key, or cryptographic break is required. The malicious participant need only be one of the `N` parties invited to the DKG/reshare session.

---

### Recommendation

Add the missing `err` parameter to `broadcast_success` and call it unconditionally — even on the error path — so that every participant always participates in the final broadcast round:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
    err: Option<&ProtocolError>,   // ← add this
) -> Result<(), ProtocolError> {
    let success = err.is_none();
    let vote_list = do_broadcast(chan, participants, me, (success, session_id)).await?;
    // ... existing checks ...
}
```

Then restructure `do_keyshare` so that any local error is captured, `broadcast_success` is called with that error (allowing peers to abort cleanly), and only then is the local error returned:

```rust
// Capture local verification result instead of propagating immediately
let local_result = verify_and_compute(...);
broadcast_success(&mut chan, &participants, me, session_id, local_result.err().as_ref()).await?;
local_result?;
```

This mirrors the intended design described in the doc-comment and ensures that a participant detecting malicious behavior can signal all peers to abort rather than silently disappearing.

### Proof of Concept

**Setup:** 3 honest participants (P1, P2, P3) and 1 malicious participant (M) run `keygen` with threshold 2.

1. **Round 1–3 proceed normally.** All four participants broadcast session IDs and commitment hashes.

2. **M sends an invalid proof of knowledge** in Round 3 (the `do_broadcast` of `(commitment, proof_of_knowledge)`). The proof is syntactically valid (correct byte length) but fails the Schnorr check.

3. **P1 reaches Round 4** (`verify_proof_of_knowledge` at line 452). The check fails: [2](#0-1) 
   `do_keyshare` returns `Err(ProtocolError::InvalidProofOfKnowledge(M))`.

4. **P1 never sends its private shares** to P2, P3 (lines 499–506 are never reached), and never participates in `broadcast_success` (line 531 is never reached).

5. **P2 and P3** are blocked at `recv_from_others(&chan, wait_round_3, ...)` waiting for P1's private share. They wait indefinitely.

6. **Result:** The entire keygen session is permanently stalled. P2 and P3 cannot complete or cleanly abort. The signing group is inoperable.
