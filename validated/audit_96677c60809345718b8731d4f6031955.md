### Title
Unchecked Precompile Return Value in `withdrawToNear` Causes Permanent Token Burn Without NEAR Transfer — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.withdrawToNear` burns the caller's ERC-20 tokens before invoking the `ExitToNear` precompile via inline assembly. The assembly `call` return value (`res`) is never checked. When the precompile rejects the call — specifically when `input_size > MAX_INPUT_SIZE` — the burn is already committed and no NEAR-side transfer is ever scheduled. The tokens are permanently destroyed with no refund path.

---

### Finding Description

**Step 1 — `_burn` before precompile call**

In `EvmErc20V2.withdrawToNear`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // ← burn commits here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never read or checked
    }
}
``` [1](#0-0) 

**Step 2 — `MAX_INPUT_SIZE` enforcement in the precompile**

`MAX_INPUT_SIZE` is `1_024` bytes: [2](#0-1) 

Both variants of `parse_input` call `validate_input_size` as the very first operation: [3](#0-2) 

`validate_input_size` returns `Err(ExitError::Other("ERR_INVALID_INPUT"))` when `input.len() > max`: [4](#0-3) 

**Step 3 — Trigger condition**

`EvmErc20V2` is deployed when the `error_refund` feature is enabled (the `sender` address is embedded in the input): [5](#0-4) 

The input layout is `[flag(1)] + [sender(20)] + [amount(32)] + [recipient(variable)]`, so `input_size = 53 + recipient.length`. The precompile rejects when `recipient.length > 971` (i.e., `53 + recipient.length > 1024`).

**Step 4 — No refund path for EVM-level precompile failure**

The `error_refund` callback (`exit_to_near_precompile_callback`) only fires as a NEAR-side callback after a promise completes: [6](#0-5) 

When `validate_input_size` fails inside `parse_input`, the precompile returns `Err(ExitError)` immediately — no promise is ever created, so no callback is ever scheduled. The `error_refund` mechanism is entirely bypassed. The `refund_on_error` / `setup_refund_on_error_input` path is unreachable: [7](#0-6) 

**Step 5 — EVM semantics confirm the burn persists**

In standard EVM semantics (and Aurora's SputnikVM), a failed sub-`call` returns `0` to the caller but does **not** revert the caller's own state changes. `_burn` is a direct internal call — its state change is part of the calling contract's frame and is not rolled back when the precompile sub-call fails. Since `res` is never checked and no `require`/`revert` follows, the function returns normally with the burn committed.

---

### Impact Explanation

Any token holder who calls `withdrawToNear` with `recipient.length >= 972` will have their tokens permanently burned with no corresponding NEP-141 credit on NEAR. The tokens are irrecoverable: the ERC-20 supply is reduced, no NEAR promise is scheduled, and no refund callback fires. This constitutes **permanent freezing (destruction) of funds**.

---

### Likelihood Explanation

The `recipient` parameter is `bytes memory` with no length guard in Solidity. The threshold (972 bytes) is reachable via:
- A legitimate Omni-bridge message of the form `receiver_id:{"recipient":"...","fee":"...","native_token_fee":"..."}` where the JSON payload is long
- Any frontend or contract that passes an oversized recipient without pre-validation
- A deliberate griefing call by the token holder themselves

No special privilege is required. Any address holding a balance can trigger this.

---

### Recommendation

Add a return-value check immediately after the assembly block, reverting if the precompile call failed:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Alternatively (and more robustly), validate `recipient.length` in Solidity before burning:

```solidity
require(1 + 20 + 32 + recipient.length <= 1024, "ERR_RECIPIENT_TOO_LONG");
_burn(sender, amount);
```

The same pattern applies to `withdrawToEthereum` and should be audited for the same class of issue.

contract BurnWithoutTransfer {
    function exploit(address erc20, uint256 amount) external {
        // Construct a recipient of length 972 (= 1024 - 53 + 1)
        // This makes input_size = 1 + 20 + 32 + 972 = 1025 > MAX_INPUT_SIZE
        bytes memory oversizedRecipient = new bytes(972);
        // Fill with valid ASCII so UTF-8 decode doesn't fail first
        for (uint i = 0; i < 972; i++) {
            oversizedRecipient[i] = 0x61; // 'a'
        }

        uint256 balanceBefore = IEvmErc20V2(erc20).balanceOf(address(this));

        // _burn commits; precompile rejects with ERR_INVALID_INPUT; res ignored
        IEvmErc20V2(erc20).withdrawToNear(oversizedRecipient, amount);

        uint256 balanceAfter = IEvmErc20V2(erc20).balanceOf(address(this));

        // Assert: tokens burned, no NEAR transfer scheduled
        assert(balanceBefore - balanceAfter == amount);
        // Tokens are gone permanently — no refund, no NEAR credit
    }
}
```

**Expected outcome:** `balanceBefore - balanceAfter == amount` holds, confirming the burn committed. No NEP-141 `ft_transfer` promise is ever created on the NEAR side. The tokens are permanently destroyed.
