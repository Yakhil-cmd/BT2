### Title
Unrestricted `deploy_erc20_token` Allows Any Caller to Register Arbitrary NEP-141 Mappings and Mint Unbacked ERC-20 Tokens - (File: engine/src/contract_methods/connector.rs)

### Summary
The `deploy_erc20_token` NEAR entry point has no caller access control. Any NEAR account can register an arbitrary NEP-141 account ID as an ERC-20 on Aurora. Because `ft_on_transfer` uses `predecessor_account_id` as the token identifier with no whitelist check, the same attacker-controlled account can then call `ft_on_transfer` directly — without transferring any real NEP-141 tokens — and cause Aurora to mint an unlimited quantity of the registered ERC-20 to any EVM address.

### Finding Description

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` performs only a liveness check (`require_running`) before deploying an `EvmErc20` contract and writing the NEP-141 → ERC-20 mapping into storage: [1](#0-0) 

There is no `require_owner_only` or any other caller restriction. Compare this to `mirror_erc20_token`, which correctly gates on `require_owner_only`: [2](#0-1) 

The comment inside `deploy_erc20_token` itself acknowledges the intent but does not enforce it: [3](#0-2) 

Once the attacker's NEAR account (e.g., `attacker.near`) is registered as a NEP-141, they call `ft_on_transfer` directly from that account. `ft_on_transfer` branches on `predecessor_account_id`: [4](#0-3) 

Since `attacker.near` is not the eth-connector, the else branch fires and calls `receive_erc20_tokens`, which looks up the ERC-20 for `attacker.near` and calls `mint(recipient, amount)` on it as the engine admin: [5](#0-4) 

The `EvmErc20` contract's `mint` function is guarded by `onlyAdmin`, and the admin is set to `current_address(current_account_id)` (the engine's own EVM address) at deploy time: [6](#0-5) 

Because `receive_erc20_tokens` calls `mint` *as* the engine admin address, the `onlyAdmin` guard is satisfied. No real NEP-141 tokens need to have been transferred; the attacker supplies an arbitrary `amount` in the `FtOnTransferArgs`.

### Impact Explanation

An attacker can mint an unbounded quantity of any ERC-20 token they registered via `deploy_erc20_token`. If that token is subsequently listed on an Aurora DEX or used as collateral in a lending protocol, the attacker can dump the minted supply to drain real bridged assets (ETH, USDC, etc.) from those protocols. This constitutes insolvency for any protocol that accepts the token, and direct theft of funds from liquidity providers.

**Impact: Critical — insolvency / direct theft of user funds.**

### Likelihood Explanation

The entry point `deploy_erc20_token` is a public NEAR function callable by any account with no deposit or special key requirement. The follow-on `ft_on_transfer` call is equally unrestricted. The attack requires only two NEAR transactions and a controlled NEAR account. No privileged access, leaked keys, or governance capture is needed.

**Likelihood: High.**

### Recommendation

Add `require_owner_only` (or an explicit admin allowlist) at the top of `deploy_erc20_token`, mirroring the pattern already used in `mirror_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?; // ADD THIS
        ...
    })
}
```

Additionally, consider whether `ft_on_transfer` should validate that the predecessor is a known, whitelisted NEP-141 contract before minting.

### Proof of Concept

1. Attacker controls NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on the Aurora engine with input `attacker.near` (borsh-encoded `AccountId`). No access control fires; an `EvmErc20` is deployed at address `A` and the mapping `attacker.near → A` is written to storage.
3. Attacker calls `ft_on_transfer` on the Aurora engine from `attacker.near` with JSON args:
   ```json
   {"sender_id": "attacker.near", "amount": "1000000000000000000000000", "msg": "<attacker_evm_address_hex>"}
   ```
4. `ft_on_transfer` sees `predecessor_account_id == "attacker.near"` ≠ eth-connector → calls `receive_erc20_tokens("attacker.near", ...)`.
5. `receive_erc20_tokens` resolves `attacker.near → A`, then calls `A.mint(attacker_evm_address, 1e24)` as the engine admin.
6. `EvmErc20.mint` passes `onlyAdmin` because the caller is the engine's EVM address.
7. Attacker now holds `1e24` unbacked ERC-20 tokens at address `A` on Aurora, which can be sold into any liquidity pool that accepts the token. [1](#0-0) [8](#0-7) [9](#0-8)

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
```

```rust
    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
```
