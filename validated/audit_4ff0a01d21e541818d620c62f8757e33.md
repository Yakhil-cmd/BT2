Audit Report

## Title
Unsolicited Pong Triggers Unconditional Disconnect, Enabling Persistent Reconnect Loop Against ckBTC Adapter — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
In `process_pong_message`, any `Pong` received while `PingState::Idle` unconditionally calls `conn.disconnect()` rather than being ignored. Because `disconnect()` (not `discard()`) is used, the peer's address is returned to `known_addresses` and the adapter immediately reconnects to the same peer. An attacker controlling Bitcoin peers that enter the adapter's address book can exploit this to hold connection slots in a permanent disconnect-reconnect cycle, stalling ckBTC block ingestion.

## Finding Description
In `connectionmanager.rs`, `process_pong_message` evaluates pong validity:

```rust
// L517–531
let valid_pong = match conn.ping_state() {
    PingState::ExpectingPong { nonce: ping_nonce, .. } => *ping_nonce == nonce,
    PingState::Idle { last_pong_at: _ } => false,
};
if valid_pong {
    conn.idle();
} else {
    conn.disconnect();  // triggered by unsolicited pong
};
```

`conn.disconnect()` (connection.rs L189–192) sets `ConnectionState::NodeDisconnected` and aborts the stream task. In `reap_disconnected` (L286–288), `NodeDisconnected` triggers `address_book.remove_from_active()`, which (addressbook.rs L282–287) removes the address from `active_addresses` and re-inserts it into `known_addresses`. This is the critical difference from `discard()`, which (when seeds are available) removes the address entirely from both sets. After reaping, `manage_connections` (L211–213) immediately calls `make_connection` to refill up to `max_connections`, and `pop()` (addressbook.rs L240–250) randomly selects from `known_addresses` — which now includes the malicious peer's address again. The cycle repeats indefinitely with no backoff or ban.

## Impact Explanation
The adapter maintains 2–5 connections (`connection_limits`, L703–710). If 2 or more malicious peers occupy slots in a tight disconnect-reconnect cycle, the adapter can be kept below `min_connections` for extended periods. Block fetching for ckBTC requires live connections; sustained deprivation stalls Bitcoin block delivery to the ckBTC canister, halting deposit and withdrawal processing. This constitutes a significant Chain Fusion / ck-token availability impact with concrete user harm — matching **High ($2,000–$10,000)**: "Significant Chain Fusion, ck-token … security impact with concrete user or protocol harm."

## Likelihood Explanation
The attacker must run Bitcoin nodes that enter the adapter's address book. The adapter discovers peers via DNS seeds and `addr` messages propagated by other peers — both are open, permissionless channels. A malicious operator can run standard Bitcoin nodes, complete the version handshake legitimately, and then send `Pong` messages at will. No privileged access, no key material, and no consensus participation is required. The Bitcoin P2P protocol places no restriction on sending `Pong` at any time. The attack is repeatable indefinitely and requires only that the malicious addresses remain in `known_addresses`, which the `disconnect()` path guarantees.

## Recommendation
Replace `conn.disconnect()` in the `PingState::Idle` branch with a no-op. An unsolicited `Pong` is not a protocol violation; only a `Pong` with a mismatched nonce when one is expected indicates misbehavior:

```rust
PingState::Idle { .. } => {
    // Unsolicited pong — not a protocol violation; ignore.
    trace!(self.logger, "Received unsolicited pong from {}", address);
    return Ok(());
}
```

Optionally, call `conn.disconnect()` (or `conn.discard()`) only when `PingState::ExpectingPong` and the nonce does not match, as that is the only case indicating genuine misbehavior.

## Proof of Concept
1. Run a Bitcoin node that completes the version handshake with the adapter (reaches `ConnectionState::HandshakeComplete`, `PingState::Idle`).
2. From the peer side, send `NetworkMessage::Pong(any_nonce)`.
3. Observe: `process_pong_message` → `PingState::Idle` → `valid_pong = false` → `conn.disconnect()`.
4. On next `tick()`: `reap_disconnected` moves address back to `known_addresses`; `manage_connections` reconnects to the same peer.
5. Repeat step 2 indefinitely — the connection slot cycles without ever becoming usable.
6. With 2 such peers, the adapter stays at or below `min_connections = 2` with zero available connections for block fetching.

A deterministic integration test can be written using the existing `test` module in `connectionmanager.rs`: inject a mock connection in `HandshakeComplete`/`Idle` state, call `process_pong_message` with an arbitrary nonce, assert `conn.state() == NodeDisconnected`, then call `reap_disconnected` and assert the address is back in `address_book.known_addresses`.
