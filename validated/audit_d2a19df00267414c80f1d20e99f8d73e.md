### Title
Gateway Stateful Validator Builds `BlockContext` with Stale Previous-Block L2 Gas Price, Making `check_fee_bounds` Inconsistent with Batcher Execution — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`run_validate_entry_point` constructs a `BlockContext` by fetching the **previous block's** `BlockInfo` and only incrementing the block number, leaving all gas prices at their previous-block values. The blockifier's `check_fee_bounds` then validates the transaction's `max_price_per_unit` against those stale prices. The batcher, however, builds its `BlockContext` from the **actual next-block** gas prices supplied by the consensus orchestrator. When the EIP-1559 L2 gas price moves between blocks, the gateway's admission decision diverges from the batcher's execution decision: valid transactions are rejected at the gateway, or transactions that pass the gateway fail at the batcher.

### Finding Description

In `run_validate_entry_point` (lines 323–330 of `stateful_transaction_validator.rs`):

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // only number is updated
let block_context = BlockContext::new(
    block_info,          // gas_prices still == P_prev
    self.chain_info.clone(),
    versioned_constants,
    BouncerConfig::max(),
);
```

The `block_info.gas_prices` field is **never updated**; it retains the previous block's STRK L2 gas price `P_prev`. This `BlockContext` is passed to `StatefulValidator::perform_validations`, which calls `AccountTransaction::perform_pre_validation_stage` → `check_fee_bounds`. Inside `check_fee_bounds` (lines 403–423 of `account_transaction.rs`), the check is:

```rust
let GasPriceVector { l2_gas_price, .. } = block_info.gas_prices.gas_price_vector(fee_type);
// ...
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    // MaxGasPriceTooLow error
}
```

`actual_gas_price` here is `P_prev`, not `P_next`.

The batcher's `BlockBuilderFactory::preprocess_and_create_transaction_executor` (lines 771–776 of `block_builder.rs`) builds its `BlockContext` from `block_metadata.block_info`, which is the **actual next-block info** supplied by the consensus orchestrator via `propose_block_input.block_info` (batcher line 429). That `block_info` carries `P_next`, computed by `calculate_next_base_gas_price` / `calculate_next_l2_gas_price_for_fin` using the previous block's gas usage and the EIP-1559 formula.

The same stale price is also used in the earlier soft threshold check in `validate_resource_bounds` (lines 230–236), where the comment explicitly acknowledges the problem:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
```

The two checks together mean the gateway enforces `max_price >= P_prev` (hard check in blockifier) and `max_price >= min_gas_price_percentage * P_prev / 100` (soft threshold), while the batcher enforces `max_price >= P_next`.

### Impact Explanation

**High — Mempool/gateway/RPC admission rejects valid transactions or accepts transactions that will fail before sequencing.**

*Case A — gas price decreases (`P_next < P_prev`):*
A user submits a V3 (`AllResources`) transaction with `max_l2_gas_price = P_next + ε` (sufficient for the next block). The gateway's `check_fee_bounds` compares against `P_prev > P_next + ε` and returns `MaxGasPriceTooLow`, rejecting a transaction that would have executed successfully in the batcher. The user's transaction is silently dropped from the network even though it is economically valid.

*Case B — gas price increases (`P_next > P_prev`):*
A user submits with `max_l2_gas_price = P_prev + ε`. The gateway passes it (price ≥ `P_prev`). The batcher's `check_fee_bounds` compares against `P_next > P_prev + ε` and reverts the transaction, wasting a mempool slot and the user's nonce increment at the gateway.

Both cases are triggered by any unprivileged user submitting a V3 transaction during a block where the EIP-1559 mechanism adjusts the L2 gas price. The `calculate_next_base_gas_price` function can move the price by up to `price * gas_delta / (gas_target * denominator)` per block; with a full block (`gas_used = max_block_size`) and `gas_price_max_change_denominator = 8` (typical), the price can shift by ~`(max_block_size - gas_target) / (gas_target * 8)` ≈ 5% per block, making the divergence practically reachable.

### Likelihood Explanation

Every block where L2 gas usage deviates from the target causes `P_next ≠ P_prev`. Under normal network load variation this occurs continuously. The window of affected transactions is those whose `max_l2_gas_price` falls between `P_next` and `P_prev` (or vice versa). The `min_gas_price_percentage` soft threshold widens this window further. No special privileges or adversarial conditions are required; any user submitting a V3 transaction during a price-change block can be affected.

### Recommendation

Compute the expected next-block L2 gas price before building the `BlockContext` in `run_validate_entry_point`, using the same `calculate_next_l2_gas_price_for_fin` / `calculate_next_base_gas_price` logic the orchestrator uses, and substitute it into `block_info.gas_prices` before constructing the `BlockContext`. The same corrected price should be used in `validate_resource_bounds`. The existing TODO comment at line 229 already identifies this gap.

### Proof of Concept

1. Previous block N has `P_N = 30_000_000_000 FRI` (30 gwei) and was nearly empty (`gas_used ≈ 0`).
2. EIP-1559 computes `P_{N+1} = P_N - P_N * gas_target / (gas_target * denominator)` ≈ `28_125_000_000 FRI` (~6% drop, matching the `low_congestion` snapshot test in `fee_market/test.rs`).
3. User submits a V3 invoke with `max_l2_gas_price = 29_000_000_000 FRI` (above `P_{N+1}`, below `P_N`).
4. Gateway's `run_validate_entry_point` builds `BlockContext` with `l2_gas_price = P_N = 30_000_000_000`.
5. `check_fee_bounds` evaluates `29_000_000_000 < 30_000_000_000` → `MaxGasPriceTooLow` → transaction rejected.
6. Batcher would have built `BlockContext` with `l2_gas_price = P_{N+1} = 28_125_000_000`; `29_000_000_000 >= 28_125_000_000` → would have passed.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
```
