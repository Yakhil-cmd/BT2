### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap validation, while the analogous global-fee setter `setPoolAdminFees` is strictly bounded by `maxAdminSpreadFeeE6`. A pool admin can therefore push the effective per-bin spread above the protocol's hard ceiling without any timelock or factory-owner approval.

### Finding Description
The factory enforces a hard ceiling on admin spread fees through `maxAdminSpreadFeeE6` (itself bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20 %). Every call to `setPoolAdminFees` checks this cap before writing:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

The per-bin setter has no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];
  s.addFeeBuyE6 = addFeeBuyE6;
  s.addFeeSellE6 = addFeeSellE6;
  emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

During every swap the per-bin additional fee is added directly on top of the global base fee before computing the bid/ask:

```solidity
// MetricOmmPool.sol L541-542
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The same additive pattern appears in all four swap paths: [5](#0-4) [6](#0-5) [7](#0-6) 

`uint16` allows values up to 65 535; in E6 units that is ≈ 6.55 %. The global spread fee is already capped at up to 40 % (20 % protocol + 20 % admin). Adding an uncapped 6.55 % per-bin surcharge pushes the effective spread for that bin to ≈ 46.55 %, far above the hard ceiling the factory is designed to enforce.

### Impact Explanation
Every trader whose swap routes through the manipulated bin pays a spread that exceeds the protocol's hard cap. The excess spread is extracted from the trader's input amount (worse execution price), constituting a direct loss of trader principal on each affected swap. Because the per-bin fee is applied inside the swap math before the output amount is computed, the trader receives fewer tokens than the oracle price and the capped fee schedule would permit — a bad-price execution impact.

### Likelihood Explanation
The pool admin role is semi-trusted and explicitly constrained by caps and timelocks for other actions (global fees, oracle rotation). `setPoolBinAdditionalFees` requires only `onlyPoolAdmin` and no timelock, so a pool admin can execute this in a single transaction at any time. The trigger is a valid, reachable, semi-trusted action — not a malicious setup assumption.

### Recommendation
Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the guard in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(
  address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
  // addFeeBuyE6 and addFeeSellE6 are E6; maxAdminSpreadFeeE6 is also E6 (uint24).
  // Casting is safe: uint16 max (65535) < uint24 max.
  if (addFeeBuyE6  > uint16(maxAdminSpreadFeeE6)) revert AdminFeeTooHigh();
  if (addFeeSellE6 > uint16(maxAdminSpreadFeeE6)) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` itself so the invariant is upheld regardless of which factory version calls it.

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 0` (factory owner disables admin spread fees for a pool).
2. Pool admin calls `factory.setPoolAdminFees(pool, 1, 0)` → reverts with `AdminFeeTooHigh` — cap is enforced.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → **succeeds** — no cap check.
4. A trader swaps through bin 0. The effective spread is now `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ 6.55 % above the oracle mid-price, despite the factory owner having set the admin spread cap to zero.
5. The trader receives fewer output tokens than the oracle price and the zero-admin-fee schedule would entitle them to, with the excess retained in the pool as inflated LP fees — a direct loss of trader principal on every swap through that bin.
