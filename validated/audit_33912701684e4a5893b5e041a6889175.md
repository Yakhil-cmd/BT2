### Title
Single Malicious Participant Can Permanently Abort DKG/Reshare/Refresh via `broadcast_success` False Vote - (File: src/dkg.rs)

### Summary
The `broadcast_success` function in `src/dkg.rs` requires **all N participants** to broadcast `(true, session_id)` in the final round of the DKG/reshare/refresh protocol. A single malicious participant can broadcast `(false, session_id)` to cause every honest participant to abort with `ProtocolError::AssertionFailed`, permanently denying key generation, resharing, and refresh — even though honest parties have already computed valid key shares.

### Finding Description
`do_keyshare` (the shared core of `do_keygen`, `do_reshare`, and `do_refresh`) concludes with a call to `broadcast_success`:

```rust
// Step 5.4 and Step 5.5
broadcast_success(&mut chan, &participants, me, session_id).await?;
``` [1](#0-0) 

`broadcast_success` uses `do_broadcast` (the echo-broadcast primitive) to collect a vote from every participant, then enforces a strict all-or-nothing check:

```rust
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant seems to have failed its checks. Aborting Protocol!"
    ));
}
``` [2](#0-1) 

There is a second abort path for a mismatched session ID:

```rust
if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
    return Err(ProtocolError::AssertionFailed(
        "A participant broadcast the wrong session id. Aborting Protocol!"
    ));
}
``` [3](#0-2) 

The echo-broadcast layer (`reliable_broadcast_receive_all`) faithfully delivers whatever value a participant sends — including `(false, session_id)` — to all other participants. [4](#0-3) 

By the time `broadcast_success` is reached, every honest participant has already:
- Verified all polynomial commitments and proofs of knowledge (Round 4)
- Verified all secret shares against commitments (Round 5, `validate_received_share`)
- Accumulated a valid signing share [5](#0-4) 

Despite this, a single malicious participant who broadcasts `(false, session_id)` causes every honest participant to discard their completed, valid key share and return an error.

### Impact Explanation
This is **permanent denial of key generation, reshare, and refresh** for all honest parties. The attack is repeatable: every time honest parties attempt DKG or reshare, the malicious participant can abort the final round. Because the malicious participant participates honestly in all earlier rounds (avoiding early detection), honest parties cannot distinguish this from a legitimate failure and must restart. The README explicitly states the protocol is designed to tolerate up to ⌊(N−1)/3⌋ malicious parties; a single-participant abort violates that documented guarantee. [6](#0-5) 

### Likelihood Explanation
Any participant enrolled in a DKG/reshare/refresh session can trigger this. No cryptographic capability, leaked key material, or external resource is required — only the ability to send a single message with `boolean = false` in the final broadcast round. The attacker participates honestly through all expensive earlier rounds to avoid triggering earlier abort paths, then strikes at the last step.

### Recommendation
The `broadcast_success` round should not abort all honest parties based on a single participant's self-reported failure. Two concrete mitigations:

1. **Remove the liveness dependency**: After Round 5, each honest participant has independently verified every input. The `broadcast_success` round adds no cryptographic guarantee; honest parties can safely output their key share without waiting for others to confirm success.

2. **Threshold-based abort**: If the success broadcast is retained for diagnostic purposes, only abort if more than ⌊(N−1)/3⌋ participants broadcast `false` — consistent with the BFT tolerance already claimed for the echo-broadcast layer.

### Proof of Concept

1. Initiate a DKG with N ≥ 2 participants; one participant M is malicious.
2. M participates honestly through all rounds: broadcasts its session ID (Round 1), sends its commitment hash (Round 2.9), broadcasts its commitment and proof of knowledge (Round 3), and sends valid secret shares to all participants (Round 4.6).
3. All honest participants successfully verify M's commitment, proof of knowledge, and secret share. They compute valid signing shares.
4. In the `broadcast_success` round, instead of broadcasting `(true, session_id)`, M broadcasts `(false, session_id)`.
5. The echo-broadcast layer delivers M's `(false, session_id)` reliably to all honest participants.
6. Every honest participant executes the check at `src/dkg.rs:329` — `vote_list.iter().all(|&(boolean, _)| boolean)` evaluates to `false` — and returns `ProtocolError::AssertionFailed("A participant seems to have failed its checks. Aborting Protocol!")`.
7. All honest participants discard their valid key shares. The DKG fails. M can repeat this attack on every subsequent attempt. [6](#0-5) [7](#0-6)
