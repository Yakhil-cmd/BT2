### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since the last reward claim. A delegator who makes one balance change per epoch (via `add_to_delegation_pool`) without claiming rewards will grow this trace without bound. Once the trace is large enough, the `claim_rewards` transaction will exceed the Starknet block gas limit, permanently freezing the delegator's unclaimed yield with no recovery path.

---

### Finding Description

**Root cause — the unbounded loop:**

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the pool member's balance trace:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop starts at `pool_member_info.entry_to_claim_from` and runs to `pool_member_trace_length`. The `entry_to_claim_from` cursor is only advanced when `claim_rewards` successfully completes and writes the updated value back to storage. [2](#0-1) 

**How the trace grows:**

Every call to `add_to_delegation_pool` (or `exit_delegation_pool_intent`, `switch_delegation_pool`, `enter_delegation_pool_from_staking_contract`) invokes `set_member_balance` / `increase_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: ...)`. The `insert` implementation appends a **new** checkpoint entry whenever the key (`current_epoch + K`) differs from the last stored key — i.e., whenever the operation occurs in a different epoch than the previous one. [3](#0-2) [4](#0-3) 

There is no cap on the number of entries in `pool_member_epoch_balance`. The minimum amount for `add_to_delegation_pool` is `amount.is_non_zero()` — i.e., 1 token unit. [5](#0-4) 

**Why the cursor does not help:**

`entry_to_claim_from` is only advanced past already-processed entries when `claim_rewards` **succeeds**. If the delegator never calls `claim_rewards` (or the call reverts due to gas exhaustion), the cursor stays at its old position and the next call must re-iterate over the entire accumulated range. Once the trace is large enough that a single `claim_rewards` call exceeds the gas limit, the cursor can never advance, and the rewards are permanently frozen.

**Access control on `claim_rewards`:**

Only the pool member address or the registered reward address may call `claim_rewards`. [6](#0-5) 

This means no third party can rescue the frozen rewards on behalf of the victim.

---

### Impact Explanation

**Impact: Permanent freezing of unclaimed yield (High)**

Once the `pool_member_epoch_balance` trace for a delegator grows beyond the threshold where a single `claim_rewards` call exceeds the Starknet transaction gas limit, the delegator's accumulated rewards become permanently unclaimable. There is no administrative escape hatch, no batch-claim mechanism, and no way to truncate the trace. The delegator's principal can still be withdrawn via `exit_delegation_pool_action`, but all accrued yield is lost forever.

---

### Likelihood Explanation

**Likelihood: Low-to-Moderate**

The developers themselves acknowledge the risk in a code comment: *"This loop is unbounded but unlikely to exceed gas limits."* The number of epochs required depends on Starknet's per-transaction gas ceiling and the per-iteration cost of the loop body (two storage reads, arithmetic, and a `find_sigma` call per iteration). Given that Starknet epochs are approximately one week long, a delegator who makes one balance change per epoch and never claims rewards would need on the order of hundreds to thousands of epochs to trigger the condition — a multi-year horizon under normal usage. However:

- A delegator who legitimately adds to their position frequently (e.g., dollar-cost averaging) and forgets to claim rewards for an extended period can reach this state organically.
- A malicious actor who controls both the pool member address and the reward address can deliberately grow the trace to grief themselves and permanently destroy their own yield, or to grief a reward address they do not control (if the reward address is a separate account that cannot prevent the pool member from making balance changes).

---

### Recommendation

1. **Introduce a per-claim iteration cap**: Allow `claim_rewards` to process at most `MAX_ENTRIES_PER_CLAIM` trace entries per call, storing the updated `entry_to_claim_from` cursor even when the full range is not processed. This converts a single potentially-unbounded call into a paginated series of bounded calls.

2. **Enforce a maximum trace length**: Reject `add_to_delegation_pool` calls that would push the trace beyond a safe bound (e.g., 1000 entries), or require the delegator to claim rewards before adding more.

3. **Periodic compaction**: After a successful `claim_rewards`, compact the trace by removing all entries before the new `entry_to_claim_from` cursor.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
