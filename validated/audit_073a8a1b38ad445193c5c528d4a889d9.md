### Title
Unsigned Invoke Transaction Admitted to Mempool via `skip_stateful_validations` Front-Run on Pending Deploy-Account - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for any invoke transaction with `nonce == 1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. The check is purely address-based: it returns `true` whenever **any** transaction from that address is in the mempool pool or committed state — it does not verify that the pending transaction is specifically a `deploy_account` from the same submitter. An unprivileged attacker who observes a victim's `deploy_account` transaction entering the mempool can immediately submit their own invoke transaction with `nonce == 1` for the same address, bypass `__validate__` entirely, and have it admitted to the mempool with an arbitrary or missing signature.

### Finding Description

**Trigger condition.** `skip_stateful_validations` fires when all three hold simultaneously:

```
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
mempool_client.account_tx_in_pool_or_recent_block(sender_address) == true
``` [1](#0-0) 

The third condition is satisfied as soon as **any** transaction from the target address is present in the pool:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for any transaction type — including the victim's `deploy_account` at nonce 0. [3](#0-2) 

**Attack path (no fee escalation required).**

1. Victim Alice submits `deploy_account` (nonce 0) for address `A`. It passes all gateway checks and is inserted into the mempool pool.
2. Attacker Bob observes Alice's `deploy_account` in the mempool. Bob constructs an invoke transaction for address `A` with `nonce = 1`, an arbitrary calldata, and an invalid (or empty) signature.
3. Bob submits his invoke tx. The gateway runs `run_pre_validation_checks`:
   - `validate_nonce`: passes — `account_nonce == 0`, `tx_nonce == 1`, within the allowed gap.
   - `validate_by_mempool`: passes — no existing nonce-1 tx for address `A` yet.
   - `skip_stateful_validations`: returns `true` — Alice's `deploy_account` is in the pool, so `account_tx_in_pool_or_recent_block` returns `true`.
4. `run_validate_entry_point` is called with `skip_validate = true`, so `execution_flags.validate = false` and the `__validate__` entry point is **never executed**. [4](#0-3) 

5. Bob's unsigned invoke tx is admitted to the mempool.
6. Alice subsequently submits her legitimate invoke tx (nonce 1). `validate_by_mempool` now returns `MempoolError::DuplicateNonce` because Bob's tx already occupies that slot. Alice's valid tx is **rejected**. [5](#0-4) 

**With fee escalation enabled**, Bob can additionally displace Alice's invoke tx even after she has already submitted it, by submitting a higher-fee replacement that also skips `__validate__`: [6](#0-5) 

**Root cause.** The `skip_stateful_validations` guard is designed to verify that a `deploy_account` for the sender exists, but the underlying `account_tx_in_pool_or_recent_block` check is not type-filtered — it returns `true` for any transaction type from that address. The comment in the code acknowledges this approximation:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [7](#0-6) 

The second branch of that reasoning ("transactions with future nonces that passed validations") is only valid for already-deployed accounts, which would have `account_nonce > 0` and would not reach the skip condition. The first branch ("has a deploy_account transaction") is not verified — any transaction type satisfies the check.

### Impact Explanation

**Broken invariant:** Every invoke transaction admitted to the mempool must have passed `__validate__` (signature verification), unless it is the legitimate first-invoke submitted alongside a `deploy_account` by the same user. This invariant is broken: an attacker can admit an invoke transaction with an arbitrary or missing signature by front-running the victim's `deploy_account` submission.

**Concrete effects:**
- An unsigned/malformed invoke transaction is accepted by the gateway and inserted into the mempool — a direct violation of the admission invariant.
- The victim's valid, signed invoke transaction is rejected with `DuplicateNonce` (or requires fee escalation to reclaim the slot).
- When the batcher executes the block, the attacker's tx fails at `__validate__` (the blockifier does call it during execution), but the victim's tx has already been displaced and is not executed in that block.
- The attack is repeatable at negligible cost: the attacker's tx fails during `__validate__` before any fee transfer, so no fee is charged.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The attack requires:
1. Observing a `deploy_account` transaction entering the mempool (public information via the P2P network or gateway).
2. Submitting an invoke tx with `nonce = 1` for the same address before the victim does — a race condition that is straightforward to win with a script.

No privileged access, special keys, or economic stake is required. The `deploy_account + invoke` UX flow is an explicitly supported and documented feature, making it a predictable target.

### Recommendation

Filter `account_tx_in_pool_or_recent_block` to only return `true` when the pending transaction for the address is specifically a `deploy_account` transaction. Alternatively, record the `deploy_account` transaction hash at submission time and require the invoke tx to reference it (analogous to the `deploy_account_tx_hash` parameter in `PyValidator::should_run_stateful_validations`): [8](#0-7) 

The `PyValidator` version already uses a `deploy_account_tx_hash` parameter to tie the skip to a specific deploy transaction. The Apollo gateway's `skip_stateful_validations` should adopt the same binding, or at minimum check that the transaction in the pool for the address is of type `DeployAccount`.

### Proof of Concept

```
1. Alice submits:
     deploy_account { class_hash, salt, constructor_calldata, nonce=0, signature=valid }
     → admitted to mempool; tx_pool.contains_account(A) = true

2. Bob (attacker) submits:
     invoke { sender_address=A, nonce=1, calldata=arbitrary, signature=[] }

3. Gateway processes Bob's invoke:
     validate_nonce(account_nonce=0, tx_nonce=1)          → OK (within gap)
     validate_by_mempool(nonce=1)                          → OK (no existing nonce-1 tx)
     skip_stateful_validations:
       tx.nonce() == 1                                     → true
       account_nonce == 0                                  → true
       account_tx_in_pool_or_recent_block(A)               → true (Alice's deploy_account)
       returns true  →  __validate__ SKIPPED
     run_validate_entry_point(skip_validate=true)          → no signature check, OK

4. Bob's unsigned invoke is in the mempool at (A, nonce=1).

5. Alice submits her legitimate invoke { sender_address=A, nonce=1, signature=valid }:
     validate_by_mempool → MempoolError::DuplicateNonce    → REJECTED

6. Block executes:
     deploy_account(A) succeeds; A is now deployed.
     Bob's invoke: __validate__ called → fails (invalid signature) → tx reverted, no fee.
     Alice's invoke: never executed.
``` [9](#0-8) [10](#0-9) [2](#0-1)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
