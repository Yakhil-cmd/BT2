### Title
Malicious Participant Can Corrupt CKD Output via Unvalidated Group Element Contributions — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary
In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates `(big_y, big_c)` contributions from all participants without any proof of correctness. A single malicious participant can send arbitrary BLS12-381 G1 elements, causing the coordinator to produce a structurally valid but cryptographically incorrect `CKDOutput`. Every honest party that relies on this output will derive a wrong confidential key.

### Finding Description

The coordinator's aggregation loop in `do_ckd_coordinator` receives each participant's `CKDOutput` and unconditionally adds the two group elements into the running totals: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant is supposed to compute: [2](#0-1) 

```
big_y  = y_i · G
big_c  = x_i · H(pk ‖ app_id) + y_i · app_pk
```

and then Lagrange-weight both before sending. There is **no zero-knowledge proof, commitment, or consistency check** that the received `(big_y, big_c)` pair was computed from the participant's actual signing share `x_i` and a freshly sampled `y_i`. The only gate is deserialization, which verifies the point is on the BLS12-381 G1 curve — it does not verify the discrete-log relationship between `big_y` and `big_c`.

Compare this with the DKG protocol, which enforces a proof-of-knowledge for every polynomial commitment before accepting it: [3](#0-2) 

No equivalent proof exists in the CKD path.

### Impact Explanation

The requester unmasks the coordinator's output as:

```
confidential_key = C − a · Y  =  msk · H(pk ‖ app_id)
```

If participant `j` sends `big_y_j' = δ_Y` and `big_c_j' = δ_C` (arbitrary G1 elements), the coordinator produces:

```
Y' = Y_honest + δ_Y
C' = C_honest + δ_C
```

The requester then computes `C' − a · Y' = msk · H + δ_C − a · δ_Y`, which is a wrong group element. The requester silently accepts this as their confidential key. There is no verification step in `CKDOutput::unmask` that would detect the corruption: [4](#0-3) 

This matches the **High** impact category: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation

Any single participant in the CKD session is a sufficient attacker. The participant is already a protocol-level principal (they hold a valid signing share and are listed in `participants`). No external capability, leaked key, or cryptographic break is required. The attack requires only that the participant deviate from the protocol by sending crafted G1 elements — a trivial modification for a library caller who controls their own `Protocol` implementation or message routing.

**Likelihood: Medium** — the attacker must be a protocol participant, but no threshold of colluders is needed; one suffices.

### Recommendation

Add a Schnorr-style proof of correct encryption alongside each `(big_y, big_c)` contribution. Concretely, participant `i` should prove in zero knowledge that:

1. `big_y = y_i · G` for some `y_i` they know, and  
2. `big_c = x_i · H(pk ‖ app_id) + y_i · app_pk` for the same `y_i` and their committed signing share `x_i`.

This is a standard ElGamal correctness proof (Chaum–Pedersen / sigma protocol). The coordinator must verify all proofs before summing. Alternatively, use a verifiable secret sharing approach where participants commit to their contribution before revealing it, analogous to the commitment-then-reveal pattern already used in DKG: [5](#0-4) 

### Proof of Concept

1. Honest participants `P1, P2, P3` run `ckd(...)` with a valid `KeygenOutput` and `app_id`.
2. Malicious participant `P2` intercepts the `chan.send_private` call and instead sends `CKDOutput::new(G1::identity(), G1::identity())` to the coordinator.
3. The coordinator's loop at line 50–55 adds the identity elements to the running totals without error.
4. The coordinator returns `CKDOutput { big_y: Y_honest, big_c: C_honest }` — but `C_honest` is now missing `P2`'s legitimate contribution `λ_2 · x_2 · H`.
5. The requester calls `ckd_output.unmask(app_sk)` and obtains `C_honest − a · Y_honest ≠ msk · H(pk ‖ app_id)`.
6. The requester silently accepts the wrong confidential key with no error or warning.

The root cause is the absence of any proof-of-correctness check between lines 50 and 56 of `src/confidential_key_derivation/protocol.rs`. [6](#0-5)
