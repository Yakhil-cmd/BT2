### Title
CRCAT Token Silently Dropped from WalletConnect Offer Preview, Causing Signing Context Confusion - (File: packages/gui/src/util/createOfferForIdsToOfferBuilderData.ts)

### Summary
`createOfferForIdsToOfferBuilderData`, which converts `chia_createOfferForIds` WalletConnect parameters into the offer preview shown to the user, handles `WalletType.CAT` and `WalletType.RCAT` but omits `WalletType.CRCAT`. A CRCAT wallet entry is silently dropped via the `default: break` path. Every other asset-selection surface in the codebase explicitly includes `WalletType.CRCAT` alongside `CAT` and `RCAT`. The omission means a WalletConnect dApp can craft a `chia_createOfferForIds` request that offers CRCAT tokens while the confirmation preview shown to the user displays nothing (or an incomplete offer), causing the user to approve an offer whose actual on-chain effect differs from what was displayed.

### Finding Description
`createOfferForIdsToOfferBuilderData` iterates over the `walletIdsAndAmounts` dict sent by a WalletConnect dApp and maps each entry to the offer builder preview:

```typescript
switch (asset.walletType) {
  case WalletType.STANDARD_WALLET:
    section.xch.push({ amount: mojoToChia(numericValue.abs()).toFixed() });
    break;
  case WalletType.CAT:
  case WalletType.RCAT:
    section.tokens.push({ amount: mojoToCAT(numericValue.abs()).toFixed(), assetId: asset.assetId });
    break;
  default:
    break;   // ← CRCAT falls here; silently discarded, no error, no warning
}
``` [1](#0-0) 

`WalletType.CRCAT` is absent from the `switch`. The `default: break` silently discards the entry. No exception is raised and no UI warning is emitted.

Every other asset-selection surface in the codebase that handles CAT/RCAT explicitly adds CRCAT:

- `OfferBuilderTokenSelector.tsx` filters wallets with `[WalletType.CAT, WalletType.RCAT, WalletType.CRCAT]` [2](#0-1) 

- `OfferBuilderTokensSection.tsx` counts available wallets with the same triple [3](#0-2) 

- `createDefaultValues.ts` pre-populates the offered tokens section for CRCAT [4](#0-3) 

- `OfferAssetSelector.tsx` includes CRCAT in the asset list [5](#0-4) 

The test file for this function explicitly labels its scope as "WalletConnect / RPC parity", confirming it is the preview path for WalletConnect `chia_createOfferForIds` requests: [6](#0-5) 

The actual WalletConnect command dispatch sends the **original** `commandParams` (including the CRCAT wallet ID) to the wallet daemon unchanged: [7](#0-6) 

This creates a split: the preview shown to the user is derived from the buggy conversion (CRCAT dropped), while the actual RPC call to the wallet daemon uses the original parameters (CRCAT present).

### Impact Explanation
A paired WalletConnect dApp sends `chia_createOfferForIds` with a CRCAT wallet ID and a negative amount (indicating an offer). The confirmation dialog renders the preview via `createOfferForIdsToOfferBuilderData`, which silently drops the CRCAT entry. The user sees an empty or incomplete offer (e.g., "offering nothing" or only the non-CRCAT side). The user approves. The wallet daemon receives the original parameters and creates an offer that locks and exposes the user's CRCAT tokens. The user has approved an offer whose actual content (CRCAT being offered) was never displayed. This matches the High impact category: WalletConnect state causes a user to approve the wrong asset.

### Likelihood Explanation
Any WalletConnect dApp that has been granted the `chia_createOfferForIds` command permission can trigger this. The pairing and command-permission grant are the only prerequisites; no key compromise or phishing is required beyond the initial (legitimate) pairing. The fingerprint and command-allowlist checks in `useWalletConnectCommand` are passed normally because the command itself is permitted. [8](#0-7) 

### Recommendation
Add `WalletType.CRCAT` to the `switch` statement in `createOfferForIdsToOfferBuilderData`, consistent with every other asset-handling surface:

```typescript
case WalletType.CAT:
case WalletType.RCAT:
case WalletType.CRCAT:   // add this
  section.tokens.push({ amount: mojoToCAT(numericValue.abs()).toFixed(), assetId: asset.assetId });
  break;
```

### Proof of Concept
1. Pair a WalletConnect dApp with the user's wallet and grant it `chia_createOfferForIds` permission.
2. The user has a CRCAT wallet with ID `3`.
3. The dApp sends:
   ```json
   { "method": "chia_createOfferForIds", "params": { "offer": { "3": -1000 }, "fee": 0 } }
   ```
4. `createOfferForIdsToOfferBuilderData` is called with `{ "3": -1000 }`. Wallet 3 resolves to `WalletType.CRCAT`. The `switch` hits `default: break`. The preview `OfferBuilderData` has empty `offered.tokens`, `offered.xch`, and `offered.nfts`.
5. The confirmation dialog shows the user an offer with nothing being offered.
6. The user approves.
7. `useWalletConnectCommand` dispatches the original `{ "3": -1000 }` to the wallet daemon via `window.permissionsAPI.dispatchAsPair`.
8. The wallet daemon creates an offer locking 1000 mojos of the user's CRCAT — an asset the user never saw in the approval dialog.
