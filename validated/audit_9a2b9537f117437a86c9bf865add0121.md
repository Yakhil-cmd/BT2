Audit Report

## Title
Missing Callback-ID/Response-Type Validation Allows Byzantine Proposer to Deliver `EmptyBlob` to a Pending `GetSuccessors` Callback — (`rs/bitcoin/consensus/src/payload_builder.rs`, `rs/replicated_state/src/bitcoin.rs`)

## Summary

`validate_self_validating_payload_impl` performs no check that a `BitcoinAdapterResponse`'s response variant matches the request type registered for the given `callback_id`, and no check that the `callback_id` exists in any context map. Separately, the `SendTransactionResponse` branch of `push_response` skips any lookup against `bitcoin_send_transaction_internal_contexts`, unconditionally enqueuing `Payload::Data(EmptyBlob)` for whatever `callback_id` is present. A single Byzantine block proposer (below the consensus fault threshold) can pair a `SendTransactionResponse` with a pending `GetSuccessors` callback ID; every honest replica will accept the block, and execution will deliver `EmptyBlob` to the bitcoin canister's `GetSuccessors` callback, corrupting chain-sync state and stalling ckBTC UTXO tracking.

## Finding Description

**Root cause 1 — `validate_self_validating_payload_impl` is a near-no-op.** [1](#0-0) 

The function short-circuits on an empty payload and otherwise returns only the byte count. It never reads the replicated state, never checks whether the `callback_id` in each response exists in `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts`, and never verifies that the response variant matches the request variant registered for that ID.

`validate_payload` calls this function and then only checks the raw byte length against `MAX_BITCOIN_PAYLOAD_IN_BYTES`: [2](#0-1) 

No state-based type/ID check is performed at any point during validation.

**Root cause 2 — `SendTransactionResponse` branch in `push_response` omits context lookup.** [3](#0-2) 

The branch extracts `callback_id` from the response but never consults `bitcoin_send_transaction_internal_contexts`. It unconditionally constructs `Payload::Data(EmptyBlob.encode())` and pushes `ConsensusResponse::new(callback_id, payload)` to the consensus queue.

Contrast with the `GetSuccessorsResponse` branch, which guards with a mandatory context lookup: [4](#0-3) 

No equivalent guard exists for `SendTransactionResponse`.

**Exploit flow.**

1. A Byzantine block proposer observes a pending `GetSuccessors` callback ID `G` in the certified state.
2. It crafts a `SelfValidatingPayload` containing `BitcoinAdapterResponse { response: SendTransactionResponse({}), callback_id: G }`.
3. All honest replicas call `validate_payload` → `validate_self_validating_payload_impl`, which accepts the payload (only size and decodability are checked).
4. The block is notarized and finalized.
5. During execution, `push_response` is called, enters the `SendTransactionResponse` arm, and enqueues `ConsensusResponse::new(G, Payload::Data(EmptyBlob.encode()))`.
6. The bitcoin canister's `GetSuccessors` callback for ID `G` receives `EmptyBlob` instead of block data. The `GetSuccessors` context for `G` is never removed from `bitcoin_get_successors_contexts` (the `SendTransactionResponse` branch does not touch that map), leaving a stale context.

## Impact Explanation

The bitcoin canister's chain-sync state machine receives an `EmptyBlob` where it expects a `BitcoinGetSuccessorsResponse`. This will cause the canister to trap, misinterpret the response, or stall. Because the `GetSuccessors` context for `G` remains in `bitcoin_get_successors_contexts` after the malicious response is processed, the canister may be permanently stuck waiting for a response that has already been "consumed." This directly stalls the ckBTC minter's UTXO tracking, which depends on continuous chain-sync progress — a concrete, significant Chain Fusion / ck-token security impact with direct user and protocol harm. This matches the **High** bounty impact: *Significant Chain Fusion, ck-token, ledger, or infrastructure security impact with concrete user or protocol harm.*

## Likelihood Explanation

Exploitation requires control of exactly one subnet node acting as block proposer — a single Byzantine node strictly below the consensus fault threshold (`f < n/3`). This is explicitly within the IC threat model. No threshold corruption, key compromise, or privileged access beyond node operation is needed. The pending `callback_id` is observable from the certified state. The attack is deterministic and locally reproducible.

## Recommendation

1. **In `validate_payload` / `validate_self_validating_payload_impl`**: Read the certified state at `validation_context.certified_height` and, for each response in the payload, verify that the `callback_id` exists in the appropriate context map and that the response variant matches the request variant (e.g., `SendTransactionResponse` only for IDs present in `bitcoin_send_transaction_internal_contexts`; `GetSuccessorsResponse` only for IDs present in `bitcoin_get_successors_contexts`). Return `InvalidArtifact` if either check fails.

2. **In `push_response` — `SendTransactionResponse` branch**: Add a lookup against `bitcoin_send_transaction_internal_contexts` (mirroring the guard in the `GetSuccessorsResponse` branch) and return `StateError::BitcoinNonMatchingResponse` if the `callback_id` is absent or belongs to the wrong context type.

## Proof of Concept

```rust
// State-machine test sketch (rs/replicated_state/src/bitcoin.rs test module)
let mut state = make_replicated_state();

// Register a GetSuccessors context with callback_id = 0
state.metadata.subnet_call_context_manager
    .bitcoin_get_successors_contexts
    .insert(CallbackId::from(0), make_get_successors_context());

// Byzantine proposer crafts a SendTransactionResponse for callback_id 0
let malicious_response = BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
        SendTransactionResponse {}
    ),
    callback_id: 0,
};

// push_response returns Ok(()) — no type-match check
let result = push_response(&mut state, malicious_response);
assert_eq!(result, Ok(()));

// GetSuccessors callback received EmptyBlob instead of block data
assert_eq!(
    state.consensus_queue[0].payload,
    Payload::Data(EmptyBlob.encode())
);

// Context for G=0 still present in bitcoin_get_successors_contexts — stale/leaked
assert!(state.metadata.subnet_call_context_manager
    .bitcoin_get_successors_contexts
    .contains_key(&CallbackId::from(0)));
```

        &self,
        payload: &SelfValidatingPayload,
        validation_context: &ValidationContext,

        let since = Instant::now();

        if *payload == SelfValidatingPayload::default() {
            return Ok(0.into());
        }

        self.metrics
            .observe_validate_duration(VALIDATION_STATUS_VALID, since);
        let size = NumBytes::new(payload.count_bytes() as u64);

        Ok(size)
    }
