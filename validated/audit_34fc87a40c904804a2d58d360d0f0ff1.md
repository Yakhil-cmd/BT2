### Title
Unprotected `new()` Initialization Allows Any Caller to Seize Engine Ownership - (File: engine/src/contract_methods/admin.rs)

### Summary
The `new()` function in the Aurora Engine contract performs no caller authentication. Any NEAR account can call it before the legitimate deployer does. Because the function permanently locks state on first call, a front-runner can set themselves as `owner_id`, permanently blocking the legitimate deployer from initializing the engine and gaining owner-level control over the contract.

### Finding Description
`contract_methods::admin::new()` is the one-time initialization entry point for the Aurora Engine. It checks only whether state already exists, then unconditionally writes the caller-supplied `NewCallArgs` (including `owner_id`) to storage:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
```

There is no check on `env.predecessor_account_id()` or `env.current_account_id()`. The public WASM entrypoint exposes this directly:

```rust
pub extern "C" fn new() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::new(io, &env)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
```

In the workspace deployment flow, contract deployment and initialization are two separate transactions:

```rust
let contract = account.deploy(&self.code...).await?;
// ...
engine.new(self.chain_id, self.owner_id, self.upgrade_delay_blocks).transact().await
```

Between these two transactions, any NEAR account can call `new()` with attacker-controlled arguments, including setting `owner_id` to themselves. All subsequent legitimate calls to `new()` will return `ERR_ALREADY_INITIALIZED`. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Once the attacker is recorded as `owner_id`, they hold all owner-gated privileges. Critically, `attach_full_access_key` is gated only by `require_owner_only`:

```rust
pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(...) {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;
    // adds a full access key to the engine account
```

An attacker who controls `owner_id` can immediately call `attach_full_access_key` to add their own full-access key to the Aurora Engine NEAR account. With a full-access key, they can drain all NEAR and bridged token balances held by the engine account. This constitutes **direct theft of user funds** — Critical impact. [4](#0-3) 

### Likelihood Explanation
NEAR has ~1-second block times and no traditional mempool, but the attack window is the gap between the `deploy` transaction and the `new()` transaction. Any observer watching the chain can detect the deploy and submit a `new()` call in the next block. The workspace deployment code does not batch these into a single atomic transaction, making the window real and exploitable by any monitoring actor. Likelihood: **Medium**. [3](#0-2) 

### Recommendation
Add a caller check inside `new()` to restrict initialization to the contract's own account (i.e., `env.predecessor_account_id() == env.current_account_id()`), or batch the deploy and `new()` call into a single NEAR batch transaction so no window exists between deployment and initialization.

### Proof of Concept
1. Deployer submits a NEAR transaction to deploy the Aurora Engine WASM to account `aurora`.
2. Attacker observes the deploy transaction on-chain.
3. Attacker submits a call to `aurora::new` with `NewCallArgs { owner_id: "attacker.near", ... }` before the deployer's `new()` lands.
4. Attacker's `new()` succeeds; `EngineState { owner_id: "attacker.near" }` is written to storage.
5. Deployer's `new()` call returns `ERR_ALREADY_INITIALIZED` and fails.
6. Attacker calls `aurora::attach_full_access_key` with their own public key (passes `require_owner_only` since `owner_id == "attacker.near"`).
7. Attacker now holds a full-access key to the `aurora` account and can transfer all funds. [1](#0-0) [4](#0-3)

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
```

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
```
