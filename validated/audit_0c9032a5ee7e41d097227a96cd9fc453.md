### Title
Malicious Participant Can Trigger Index-Out-of-Bounds Panic via Undersized `c_j_i_v` Vec at `wait6` — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

In `do_generation_many`, the `wait6` receive loop indexes into a peer-supplied `Vec<SerializableScalar<C>>` using `c_j_i_v[i]` for `i in 0..N` with no prior length check. A malicious participant can send a Vec of length `N-1` (or any length `< N`), causing an index-out-of-bounds Rust panic in the honest receiver's process.

---

### Finding Description

At `wait6`, the honest party receives private polynomial-share scalars from every other participant: [1](#0-0) 

```rust
for (_, c_j_i_v) in
    recv_from_others::<Vec<SerializableScalar<C>>>(&chan, wait6, &participants, me).await?
{
    for i in 0..N {
        let c_j_i = c_j_i_v[i].0;   // ← panics if c_j_i_v.len() < N
        c_i_v[i] += c_j_i;
    }
}
```

`recv_from_others` only enforces that exactly one message arrives from each peer; it performs no structural validation on the deserialized payload: [2](#0-1) 

There is no guard between deserialization and the `c_j_i_v[i]` index expression. A malicious participant constructs and sends a `Vec<SerializableScalar<C>>` with `len < N` (e.g., `N-1` elements). When the honest receiver reaches `c_j_i_v[N-1]`, Rust's bounds check fires and the thread/task panics.

The same structural defect exists at `wait3` (`a_j_i_v[i]`, `b_j_i_v[i]`), `wait4` (`big_c_j_v[i]`, `their_phi_proofs[i]`), and `wait5` (`their_hat_big_c_i_points[i]`, `their_phi_proofs[i]`): [3](#0-2) [4](#0-3) [5](#0-4) 

The question focuses on `wait6`; all four sites share the same root cause.

---

### Impact Explanation

A Rust index-out-of-bounds is an unrecoverable panic, not a `ProtocolError`. The `make_protocol` adapter polls the async future cooperatively; a panic propagates out of `poke()` to the caller and crashes the process (or the enclosing thread) unless the integrator wraps every `poke()` call in `std::panic::catch_unwind`, which is not done in this library. Triple generation is a mandatory prerequisite for every OT-based ECDSA presignature and signature. A single malicious participant can crash every honest party's process at any of the four vulnerable waitpoints, permanently blocking signing for the session. The attacker can repeat this across every new session, making the denial effectively permanent.

---

### Likelihood Explanation

Any participant in a triple-generation session controls the bytes they send. Constructing a short Vec is trivial — it requires only serializing a `Vec` of length `N-1` with `rmp_serde`. No cryptographic material, no leaked keys, and no external assumptions are needed. The attacker need only be an admitted participant.

---

### Recommendation

Before the inner `for i in 0..N` loop, validate the length of every received Vec:

```rust
if c_j_i_v.len() != N {
    return Err(ProtocolError::AssertionFailed(
        format!("c_j_i_v from {from:?} has wrong length: expected {N}, got {}", c_j_i_v.len())
    ));
}
```

Apply the same guard at `wait3`, `wait4`, and `wait5` for `a_j_i_v`, `b_j_i_v`, `big_c_j_v`, `their_phi_proofs`, `their_hat_big_c_i_points`, and `their_phi_proofs`. Consider a shared helper that deserializes and length-validates any `Vec` payload against `N` to prevent recurrence.

---

### Proof of Concept

```rust
// Attacker's participant sends at wait6:
let short_vec: Vec<SerializableScalar<Secp256K1Sha256>> = vec![]; // len = 0, N > 0
let payload = rmp_serde::encode::to_vec(&short_vec).unwrap();
// Deliver payload to honest receiver at wait6.
// Honest receiver reaches c_j_i_v[0], panics with:
//   "index out of bounds: the len is 0 but the index is 0"
// Process crashes; triple generation never completes.
```

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
