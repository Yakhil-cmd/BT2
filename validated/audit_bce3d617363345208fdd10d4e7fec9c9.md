### Title
Silent ERC-20 Token Misdirection via Silo Whitelist Redirect in `receive_erc20_tokens` — (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, when the `Address` whitelist is enabled and a fallback address is configured, NEP-141 tokens sent to Aurora via `ft_transfer_call` with a non-whitelisted EVM recipient are silently minted to the fallback address instead of being returned to the sender. The `ft_on_transfer` callback returns `0` (success), permanently transferring the sender's tokens to the fallback address without their knowledge or consent.

---

### Finding Description

The `receive_erc20_tokens` function silently redirects the ERC-20 mint recipient from the intended address to the fallback address under the following condition:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which returns `false` for any address not present in the enabled `Address` whitelist:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [2](#0-1) 

After the silent redirect, the function mints ERC-20 tokens to the fallback address and returns `Ok(Some(result))`. The `ft_on_transfer` caller then interprets this as success and returns `0` tokens to refund:

```rust
let amount_to_return = if let Err(_err) = &result {
    args.amount.as_u128()  // error: return tokens to sender
} else {
    0  // success: keep all tokens
};
``` [3](#0-2) 

This means the NEP-141 `ft_transfer_call` is considered fully settled. The sender's NEP-141 tokens are permanently consumed, and the ERC-20 mirror tokens are minted to the fallback address — not the intended recipient.

The fallback address is set exclusively by the Silo owner via `set_erc20_fallback_address` or `set_silo_params`: [4](#0-3) 

The `SiloParamsArgs` type documents this behavior:

```rust
/// EVM address, which is used for withdrawing ERC-20 base tokens in case
/// a recipient of the tokens is not in the silo white list.
pub erc20_fallback_address: Address,
``` [5](#0-4) 

The analog to the Liquity report is direct: Liquity's `_requireValidRecipient` causes unexpected **reverts** for 3rd-party callers. Aurora's Silo whitelist causes unexpected **silent token misdirection** — which is strictly worse, because the sender receives no error signal and cannot retry or recover.

---

### Impact Explanation

Any unprivileged NEP-141 token holder who calls `ft_transfer_call` on a registered NEP-141 token with Aurora as the receiver and a non-whitelisted EVM address as the `msg` recipient will have their tokens permanently redirected to the Silo fallback address. The NEP-141 transfer is marked complete (0 refunded), so the sender has no recourse through the standard token interface. The tokens are effectively frozen in the fallback address, inaccessible to the original sender.

**Impact: High — Temporary (potentially permanent) freezing of user funds.**

If the fallback address is a contract that does not expose a withdrawal path for arbitrary users, the freeze is permanent.

---

### Likelihood Explanation

This is reachable by any unprivileged token holder on any Aurora Silo deployment where:
1. The `Address` whitelist is enabled (`WhitelistKind::Address` active), and
2. A fallback address is configured via `set_silo_params` or `set_erc20_fallback_address`.

Both conditions are the normal operating state of a Silo deployment. The entry path is the standard NEP-141 `ft_transfer_call` interface — no special privileges required.

**Likelihood: Medium** (Silo-mode deployments only, but the trigger is any ordinary token transfer to a non-whitelisted address).

---

### Recommendation

When the intended recipient is not whitelisted, `receive_erc20_tokens` should return an error (causing `ft_on_transfer` to refund the full amount to the sender) rather than silently redirecting to the fallback address. The fallback address mechanism is appropriate only for cases where the engine itself initiates a deposit on behalf of an unknown address (e.g., bridge ingress), not for user-initiated `ft_transfer_call` flows where the sender has explicitly named a recipient.

At minimum, the function should return the full token amount to the sender when the recipient is not whitelisted, consistent with the error-handling contract already present in `ft_on_transfer`. [6](#0-5) 

---

### Proof of Concept

1. Silo mode is active: `Address` whitelist enabled, fallback address `F` configured.
2. Alice holds 1000 units of NEP-141 token `T`, which has a registered ERC-20 mirror on Aurora.
3. Alice calls `ft_transfer_call` on `T` with `receiver_id = aurora`, `amount = 1000`, `msg = alice_evm_address_hex`.
4. Aurora's `ft_on_transfer` is triggered; `receive_erc20_tokens` is called.
5. `is_allow_receive_erc20_tokens` returns `false` for Alice's EVM address (not whitelisted).
6. `get_erc20_fallback_address` returns `Some(F)`.
7. `recipient` is silently overwritten to `F`.
8. ERC-20 `mint(F, 1000)` is executed successfully.
9. `ft_on_transfer` returns `"0"` — zero tokens refunded.
10. Alice's 1000 NEP-141 tokens are permanently consumed. ERC-20 tokens are at `F`, not Alice's address. Alice has no recourse. [7](#0-6)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
