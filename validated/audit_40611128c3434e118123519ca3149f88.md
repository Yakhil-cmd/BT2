### Title
Malformed Payload from Any Participant Permanently Aborts DKG/Reshare/Refresh via `recv_from_others` — (`src/protocol/helpers.rs`, `src/protocol/internal.rs`)

---

### Summary

A single malicious participant can permanently abort an honest party's DKG, reshare, or refresh protocol by sending one message whose 40-byte header is valid (correct `ChannelTag` + correct `Waitpoint`) but whose payload is intentionally malformed (e.g., truncated or msgpack-invalid). The `recv_from_others` helper propagates the resulting `DeserializationError` immediately via `?`, with no retry or skip logic, permanently terminating the protocol future.

---

### Finding Description

**Attack path — step by step:**

**Step 1 — Entry point:** The library consumer calls `Protocol::message(from, data)` to deliver a network message to the protocol instance. [1](#0-0) 

This delegates directly to `Comms::push_message`.

**Step 2 — `push_message` accepts the message with no payload validation:** [2](#0-1) 

The only checks are: (a) length ≥ 40 bytes, and (b) the first 40 bytes parse as a valid `MessageHeader`. There is no check that `from` is in the participant list, and no check that the payload bytes are well-formed. A message with a correct 40-byte header and garbage payload bytes passes both checks and is stored in the `MessageBuffer`.

**Step 3 — `MessageBuffer::push` stores it in a FIFO queue keyed by `MessageHeader`:** [3](#0-2) 

**Step 4 — `Comms::recv` pops the message and attempts deserialization:** [4](#0-3) 

If `rmp_serde::decode::from_slice` fails on the malformed payload, `Comms::recv` returns `Err(ProtocolError::DeserializationError(...))`. `SharedChannel::recv` is a direct pass-through with no retry: [5](#0-4) 

**Step 5 — `recv_from_others` propagates the error immediately via `?`:** [6](#0-5) 

The `?` on line 20 means any `Err` from `chan.recv()` immediately exits the loop and propagates up. There is no retry, no skip-and-continue, no participant-list check before deserialization.

**Step 6 — `do_keyshare` uses `recv_from_others` in two non-broadcast rounds:**

Round 2 (commitment hashes): [7](#0-6) 

Round 4 (private share evaluations): [8](#0-7) 

Both calls propagate `ProtocolError` via `?`, permanently aborting the `do_keyshare` future.

**The attacker can compute the correct header deterministically:** `ChannelTag::root_shared()` is a fixed SHA-256 hash of public constants, and waitpoints are sequential starting from 0 — both are fully predictable by any protocol participant. [9](#0-8) 

---

### Contrast with Echo Broadcast (which is resilient)

The echo broadcast protocol, used in rounds 1, 3, and 5.5, explicitly handles deserialization errors by continuing the loop: [10](#0-9) 

`recv_from_others` has no equivalent guard. This asymmetry means the non-broadcast rounds (2 and 4) are vulnerable while the broadcast rounds are not.

---

### Impact Explanation

A single malicious participant (one of the n protocol participants) can permanently abort DKG, reshare, or refresh for all honest parties sharing that `Comms` instance. The protocol future terminates with `ProtocolError::DeserializationError` and cannot be resumed. This matches the **High** impact: *Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions.*

The DKG documentation states the protocol tolerates up to `floor((N-1)/3)` malicious parties, but this attack succeeds with a single malicious participant regardless of N or the threshold.

---

### Likelihood Explanation

- The attacker is a legitimate protocol participant (no external assumptions needed).
- The correct header bytes are fully deterministic and computable from public constants.
- The attack requires sending exactly one message with a valid 40-byte header and any garbage payload.
- No cryptographic assumptions need to be broken.
- The attack is locally testable.

---

### Recommendation

`recv_from_others` should mirror the echo broadcast pattern: catch deserialization errors and continue the loop rather than propagating them. Specifically, change line 20 of `src/protocol/helpers.rs` from:

```rust
let (from, msg) = chan.recv(waitpoint).await?;
```

to a match that skips malformed messages:

```rust
let Ok((from, msg)) = chan.recv(waitpoint).await else {
    continue;
};
```

This is exactly the pattern already used in `reliable_broadcast_receive_all` at `src/protocol/echo_broadcast.rs` line 179–182.

---

### Proof of Concept

```rust
// Attacker computes the correct header for round 2 (waitpoint 0 on shared channel)
let channel_tag = ChannelTag::root_shared(); // deterministic, public
let header = MessageHeader::new(channel_tag).with_waitpoint(0);
let mut malicious_msg = header.to_bytes().to_vec(); // valid 40-byte header
malicious_msg.extend_from_slice(&[0xFF, 0xFE, 0xFD]); // garbage msgpack payload

// Attacker delivers this before the honest participant's message
protocol.message(attacker_participant, malicious_msg);

// Honest party's poke() now returns ProtocolError::DeserializationError
// The protocol future is permanently terminated
assert!(matches!(protocol.poke(), Err(ProtocolError::DeserializationError(_))));
```
