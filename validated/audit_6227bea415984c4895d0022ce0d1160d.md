### Title
Gateway `skip_stateful_validations` Bypasses Signature Verification for Nonce-1 Invoke Transactions on Existing Accounts with Nonce=0 — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator skips the `__validate__` entry point — which is the sole on-chain signature-verification step at admission time — for any invoke transaction with `tx_nonce == 1` whenever the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` for **any** transaction type in the pool (including a plain invoke), not exclusively for a `deploy_account` transaction. This allows an attacker to submit an invoke with an arbitrary (invalid) signature for any existing account whose on-chain nonce is `0` and that already has a pending transaction in the mempool, bypassing gateway signature validation entirely.

---

### Finding Description

In `skip_stateful_validations` the skip decision is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type (invoke or deploy_account) in the pool. The code comment claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is incorrect for the case of a **pre-existing account** (on-chain nonce = 0, account already deployed). Such an account can have a valid invoke with `nonce=0` in the pool (which passed `__validate__` because the account exists). Once that nonce-0 invoke is in the pool, `account_tx_in_pool_or_recent_block` returns `true`, and the skip condition fires for any subsequent nonce-1 invoke — regardless of whether its signature is valid.

The `run_validate_entry_point` then sets `validate: !skip_validate`, so `__validate__` is never called:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 308-312
let only_query = false;
let charge_fee = enforce_fee(executable_tx, only_query);
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The upstream `validate_by_mempool` call only checks for duplicate hashes and nonce ordering — it does **not** verify signatures:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 162-174
fn validate_incoming_tx(...) -> MempoolResult<()> {
    ...
    if tx_nonce < account_nonce {
        return Err(MempoolError::NonceTooOld { ... });
    }
    Ok(())
}
``` [4](#0-3) 

The existing test suite confirms the skip fires for any `contains_tx=true` case without distinguishing transaction type:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator_test.rs  lines 152-156

``` [5](#0-4) 

By contrast, the legacy Python-path validator (`PyValidator::should_run_stateful_validations`) requires an explicit `deploy_account_tx_hash` to be supplied by the caller before it will skip:

```rust
// crates/native_blockifier/src/py_validator.rs  lines 109-110
let deploy_account_not_processed =
    deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
``` [6](#0-5) 

The Rust gateway path has no equivalent type-specific guard.

---

### Impact Explanation

An attacker can submit an invoke transaction with a completely invalid (or zero) signature for any account that (a) has on-chain nonce = 0 and (b) already has any pending transaction in the mempool. The gateway will admit the transaction without ever calling `__validate__`, violating the invariant that every admitted transaction carries a verified signature. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

Accounts with on-chain nonce = 0 that have a pending transaction are common (e.g., genesis-funded accounts, operator accounts, or any account that has just been deployed and is sending its first invoke). The attacker needs no privileged access — only knowledge of such an account address and the ability to submit RPC transactions.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific check that confirms a `deploy_account` transaction (and only a `deploy_account`) is present in the mempool for the sender address. Alternatively, expose a dedicated mempool query `deploy_account_in_pool(address) -> bool` and use that in `skip_stateful_validations`, mirroring the explicit `deploy_account_tx_hash` guard used in `PyValidator::should_run_stateful_validations`.

---

### Proof of Concept

1. Account `A` is pre-deployed on-chain with nonce = 0 (e.g., a genesis account).
2. The legitimate owner of `A` submits a valid invoke with `nonce=0`. It passes `__validate__` (account exists) and is admitted to the mempool. `tx_pool.contains_account(A)` now returns `true`.
3. An attacker constructs an invoke for `A` with `nonce=1` and a zeroed/garbage signature.
4. Gateway stateful validation:
   - `validate_nonce`: `0 ≤ 1 ≤ 200` → passes.
   - `validate_by_mempool`: no duplicate hash, `tx_nonce(1) ≥ account_nonce(0)` → passes.
   - `skip_stateful_validations`: `tx_nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → returns `true`.
   - `run_validate_entry_point`: `validate = false` → `__validate__` **never called**.
5. The attacker's transaction with invalid signature is admitted to the mempool.
6. Repeating step 3 with different garbage signatures and different `tx_hash` values allows unbounded injection of signature-invalid transactions into the mempool for account `A`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
