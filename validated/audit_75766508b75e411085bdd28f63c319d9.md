### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function skips the account's `__validate__` entry point (which performs signature verification) for any invoke transaction with `tx_nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The check does not verify that the transaction in the pool is specifically a `deploy_account`. An account deployed via the `deploy` syscall (nonce=0 on-chain) can have an ordinary invoke with nonce=0 in the mempool, satisfying the condition and causing a subsequent nonce=1 invoke with an **invalid signature** to bypass `__validate__` and be admitted to the mempool.

### Finding Description

In `skip_stateful_validations`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The function returns `true` (skip `__validate__`) when:
1. The transaction is an invoke with `tx_nonce == 1`
2. The on-chain `account_nonce == 0`
3. The account has **any** transaction in the pool or a recent block

The code comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second case is the flaw: a nonce=0 invoke that passed `__validate__` is treated as proof that all future transactions from the account are valid. This is incorrect — the signature for nonce=1 is independent of the signature for nonce=0.

The `run_validate_entry_point` then constructs the `AccountTransaction` with `validate: !skip_validate`, so when `skip_validate=true`, the `__validate__` entry point is never called:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };
``` [2](#0-1) 

And in `StatefulValidator::perform_validations`, when `validate=false`, the `__validate__` call is skipped entirely:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

### Impact Explanation

An attacker who controls an account deployed via the `deploy` syscall (nonce=0 on-chain) can:

1. Submit a valid invoke with nonce=0 (passes `__validate__` with a correct signature) → enters the mempool.
2. Submit an invoke with nonce=1 carrying an **invalid/arbitrary signature**.
3. `skip_stateful_validations` returns `true` because `tx_nonce==1`, `account_nonce==0`, and the account has a transaction in the pool.
4. `__validate__` is skipped at the gateway level.
5. The nonce=1 invoke with invalid signature is admitted to the mempool.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The transaction will fail during blockifier execution (where `__validate__` is always run via `run_or_revert`), but the gateway invariant — that every transaction in the mempool has passed account signature verification — is broken. This enables mempool pollution and potential displacement of legitimate nonce=1 transactions.

### Likelihood Explanation

The trigger requires an account deployed via the `deploy` syscall (not `deploy_account`) with nonce=0 on-chain, which is a valid and reachable Starknet state. The attacker must control the account (to submit the valid nonce=0 invoke), but then can freely submit nonce=1 invokes with arbitrary signatures. The `max_allowed_nonce_gap` default of 200 means the mempool accepts nonce=1 when account_nonce=0 without issue. [4](#0-3) 

### Recommendation

In `skip_stateful_validations`, replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the account in the mempool. Alternatively, add a dedicated `deploy_account_in_pool` query to the mempool client so the skip logic only fires when the account is genuinely undeployed and a deploy_account is pending.

### Proof of Concept

1. Deploy contract `A` via `deploy` syscall (not `deploy_account`). On-chain: `nonce(A) = 0`, account exists.
2. Submit `invoke(A, nonce=0, valid_signature)` → passes `validate_nonce` (0 ≤ 0 ≤ 200), passes `validate_by_mempool`, `skip_stateful_validations` returns `false` (nonce≠1), `__validate__` runs and succeeds → enters mempool.
3. Submit `invoke(A, nonce=1, INVALID_SIGNATURE)` → passes `validate_nonce` (0 ≤ 1 ≤ 200), passes `validate_by_mempool`, `skip_stateful_validations` checks: `tx_nonce==1` ✓, `account_nonce==0` ✓, `account_tx_in_pool_or_recent_block(A)` → `true` (nonce=0 invoke is in pool) → returns `true`.
4. `run_validate_entry_point` is called with `validate: false` → `__validate__` is **skipped**.
5. The nonce=1 invoke with invalid signature is admitted to the mempool without signature verification. [5](#0-4) [6](#0-5)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
