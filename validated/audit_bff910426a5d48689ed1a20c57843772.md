### Title
Unbounded `pool_member_epoch_balance` Trace Growth Enables Permanent Freezing of Unclaimed Yield via Repeated Balance-Change Calls — (`src/pool/pool.cairo`)

---

### Summary

A pool member can grow their personal `pool_member_epoch_balance` trace without bound by calling `add_to_delegation_pool` (amount = 1) or `exit_delegation_pool_intent` (amount = 0) once per epoch. The `calculate_rewards` function, invoked inside `claim_rewards`, iterates over this trace in a loop that the code itself labels **"unbounded"**. After enough epochs of trace inflation, the loop exceeds Starknet's per-transaction gas limit, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

**Step 1 — Trace growth mechanism.**

Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` ultimately calls `set_member_balance`, which calls `trace.insert(key: epoch + K, value: ...)`: [1](#0-0) 

The `insert` implementation appends a **new** checkpoint whenever the epoch key differs from the last stored key, and merely overwrites the last entry when the key is the same: [2](#0-1) 

Because the key is `current_epoch + K`, one new entry is appended per distinct epoch in which a balance-changing call is made. There is no cap on the number of entries.

**Step 2 — Unbounded loop in `calculate_rewards`.**

`claim_rewards` calls `calculate_rewards`, which iterates over every entry in the trace from `entry_to_claim_from` up to the current epoch: [3](#0-2) 

The developers themselves acknowledge the risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

**Step 3 — Attack path.**

A pool member who deliberately (or inadvertently) makes one balance-changing call per epoch without ever claiming rewards accumulates one new trace entry per epoch. After `N` such epochs, a single `claim_rewards` call must iterate `N` times. Once `N` crosses the gas-limit threshold, the transaction always reverts.

The cheapest reliable vector is `add_to_delegation_pool(pool_member, amount: 1)` — one token per epoch — which calls `increase_member_balance` → `set_member_balance`: [4](#0-3) 

An alternative zero-token-cost vector is `exit_delegation_pool_intent(amount: 0)`, which also unconditionally calls `set_member_balance`: [5](#0-4) 

**Step 4 — `entry_to_claim_from` does not protect against this.**

`entry_to_claim_from` advances only when `claim_rewards` succeeds. If the pool member withholds claims while continuing to append entries, the backlog grows monotonically. Once the backlog is large enough to exceed gas, `claim_rewards` can never succeed, so `entry_to_claim_from` can never advance — the freeze is permanent. [6](#0-5) 

---

### Impact Explanation

The pool member's accumulated unclaimed yield becomes permanently unclaimable once the trace is large enough to exhaust gas in a single `claim_rewards` call. The principal is still recoverable via `exit_delegation_pool_action` (which does not call `calculate_rewards`), but all accrued rewards are frozen forever.

**Impact: Permanent freezing of unclaimed yield** — matches the allowed High impact category.

---

### Likelihood Explanation

- The attacker controls the exact pace of trace growth (one entry per epoch).
- The minimum token cost is 1 unit per epoch (via `add_to_delegation_pool`), or zero tokens per epoch (via `exit_delegation_pool_intent(0)`).
- Starknet epochs are short (measured in blocks), so a determined attacker can accumulate hundreds of entries over weeks.
- The code comment explicitly acknowledges the loop is unbounded, confirming the developers are aware of the risk but have not bounded it.
- A legitimate long-term delegator who makes frequent small top-ups could trigger this unintentionally.

**Likelihood: Medium.**

---

### Recommendation

1. **Bound the trace per claim window**: After each successful `claim_rewards`, prune or compact trace entries with epoch < `entry_to_claim_from` so the live trace length stays O(K).
2. **Enforce a maximum trace length**: Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` calls that would push the trace beyond a safe bound (e.g., 256 entries).
3. **Alternatively, checkpoint on claim**: Store a single "last claimed epoch + balance" checkpoint and recompute rewards incrementally, eliminating the need to iterate over historical balance changes.

---

### Proof of Concept

```
// Setup: pool member enters with minimum stake.
pool.enter_delegation_pool(reward_address, min_amount);

// Each epoch for N epochs (no claim_rewards called):
for epoch in 0..N {
    advance_epoch();
    // Appends one new entry to pool_member_epoch_balance trace.
    token.approve(pool, 1);
    pool.add_to_delegation_pool(pool_member, amount: 1);
    // OR (zero token cost):
    // pool.exit_delegation_pool_intent(amount: 0);
}

// After N epochs, pool_member_epoch_balance trace has N entries.
// The following call iterates N times in calculate_rewards and OOGs:
pool.claim_rewards(pool_member);  // REVERTS: out of gas
// Unclaimed yield is permanently frozen.
```

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
