### Title
Missing `max_l2_gas_amount` Upper-Bound Check for Declare Transactions in Stateless Gateway Validator - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary
The stateless gateway validator enforces a `max_l2_gas_amount` cap on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. Any unprivileged user can submit a Declare V3 transaction with `l2_gas.max_amount` set to `u64::MAX` (or any value exceeding the configured cap of 1,210,000,000), and the gateway will admit it to the mempool without rejection.

### Finding Description

In `validate_resource_bounds`, the check for `max_l2_gas_amount` is guarded by a type-branch that short-circuits for `RpcTransaction::Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

The production configuration sets `max_l2_gas_amount = 1_210_000_000`: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms this bypass is reachable: a Declare transaction with `max_amount: GasAmount(200)` passes validation when `max_l2_gas_amount: 100`: [3](#0-2) 

By contrast, the same oversized amount on an Invoke or DeployAccount transaction is correctly rejected: [4](#0-3) 

The downstream execution path uses `l2_gas.max_amount` directly as the initial Sierra gas counter for `AllResources`-bounded transactions: [5](#0-4) 

A Declare transaction with `l2_gas.max_amount = u64::MAX` therefore enters the mempool with an unbounded initial gas claim, bypassing the gateway's own admission policy.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The gateway's `max_l2_gas_amount` policy exists to prevent transactions from claiming excessive L2 gas. For Declare transactions this invariant is broken: any user can submit a Declare V3 with `l2_gas.max_amount` set to any value (e.g., `u64::MAX`) and it will pass stateless validation and be forwarded to the mempool. The stateful validator only checks the L2 gas *price* against the previous block price, not the *amount*: [6](#0-5) 

A user with sufficient balance to cover `max_amount * max_price_per_unit` can therefore admit a Declare transaction that violates the gateway's own resource-bound policy, bypassing the cap that applies to every other transaction type.

### Likelihood Explanation

Any unprivileged user who can submit a Declare V3 transaction can trigger this. No special privileges, no race condition, no dependency on external state beyond having a valid account with sufficient balance to pass the stateful balance check. The bypass is unconditional for all Declare transactions.

### Recommendation

Apply the same `max_l2_gas_amount` upper-bound check to Declare transactions. Remove the type-branch exemption and apply the check uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The TODO comment at line 78 acknowledges this gap; it should be resolved rather than deferred. [1](#0-0) 

### Proof of Concept

1. Construct a valid Declare V3 transaction with:
   - `l2_gas.max_amount = u64::MAX` (or any value > 1,210,000,000)
   - `l2_gas.max_price_per_unit >= min_gas_price` (≥ 8,000,000,000 in production)
   - A valid Sierra contract class, valid signature, nonce, etc.
2. Submit to the gateway's `add_transaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare transactions.
4. The transaction proceeds to stateful validation and, if the account balance covers `max_amount * max_price_per_unit`, is admitted to the mempool — with a declared L2 gas amount far exceeding the configured policy cap.

The existing test already demonstrates step 3 in isolation: [3](#0-2)
