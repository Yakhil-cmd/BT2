### Title
`setPoolAdminFeeDestination()` Redirects Already-Accrued Admin Fees to New Destination Without Prior Collection — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination()` updates `poolAdminFeeDestination[pool]` in the factory without first flushing accrued fees to the old destination. All spread-surplus and notional fees that built up under the old address are silently redirected to the new address on the next `collectFees` call, causing a direct loss of owed admin fees for the old fee destination.

---

### Finding Description

The protocol stores the admin fee recipient per-pool in `poolAdminFeeDestination[pool]` inside `MetricOmmPoolFactory`. Every fee collection call passes this value as `adminFeeDestination_` to `MetricOmmPool.collectFees()`, which immediately transfers the admin share of both spread-surplus tokens and tracked notional fees to that address. [1](#0-0) 

The function performs no prior fee flush:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees() first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

The protocol's own sibling function `setPoolAdminFees()` demonstrates the correct pattern: it explicitly calls `collectFees` at the **old** destination before updating any fee parameter. [2](#0-1) 

`setPoolAdminFeeDestination()` omits this step entirely. After the destination is changed, the next call to `collectPoolFees()` (permissionless) or any fee-changing function will flush all previously accrued fees — both the spread surplus and `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` — to the **new** address. [3](#0-2) 

The `collectPoolFees` entry point in the factory reads `poolAdminFeeDestination[pool]` at call time, so there is no snapshot of the old destination: [4](#0-3) 

---

### Impact Explanation

All admin fees accrued since the last collection are permanently redirected to the new destination. The old fee destination — which may be a separate revenue-sharing contract, a DAO treasury, or a third-party beneficiary — receives nothing for the period it was the designated recipient. The loss is bounded by the total admin-share fees accrued in the pool since the last `collectFees` call, which can be arbitrarily large in a high-volume pool with a long collection interval.

---

### Likelihood Explanation

The pool admin is semi-trusted. The contest scope explicitly flags "fee collection destinations" as an admin-boundary bypass surface. A pool admin who controls both the admin role and a new destination address can atomically (1) call `setPoolAdminFeeDestination` to point to their own address, then (2) call the permissionless `collectPoolFees` to drain all accrued fees. No timelock, no cap, and no on-chain guard prevents this sequence. The old fee destination has no recourse.

---

### Recommendation

Mirror the pattern already used in `setPoolAdminFees()`: call `collectFees` at the **current** `poolAdminFeeDestination[pool]` before overwriting the mapping.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before rotating
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed; `poolAdminFeeDestination[pool] = addressA` (e.g., a revenue-sharing contract).
2. Swaps occur; spread surplus and notional fees accumulate in the pool.
3. Pool admin calls `setPoolAdminFeeDestination(pool, addressB)` — no fees are collected; `poolAdminFeeDestination[pool]` is now `addressB`.
4. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `addressB` and passes it to `collectFees`.
5. `MetricOmmPool.collectFees` transfers the entire admin share of all accrued spread surplus and notional fees to `addressB`.
6. `addressA` receives zero despite being the designated recipient for the entire accrual period.
