### Title
Total Stake Inflated When Pool Member Switches Delegation From an Unstaking Staker - (File: src/staking/staking.cairo)

### Summary
`switch_staking_delegation_pool` unconditionally adds the switched amount back to `total_stake`, but when the source staker has already called `unstake_intent`, the intent amount was **never subtracted** from `total_stake` in the first place. This permanently inflates `total_stake`, diluting rewards for every staker in the protocol.

### Finding Description

The protocol tracks `total_stake` as the denominator for reward and staking-power calculations. The invariant is:

- `remove_from_delegation_pool_intent` **subtracts** the intent amount from `total_stake` (so funds "in flight" are excluded).
- `switch_staking_delegation_pool` **adds** the switched amount back to `total_stake` (restoring it for the destination staker).

This invariant breaks when the source staker has already called `unstake_intent`.

**`unstake_intent`** removes the staker's entire current delegated balance from `total_stake`: [1](#0-0) 

**`remove_from_delegation_pool_intent`** calls `update_delegated_stake`, which explicitly **skips** the `total_stake` update when the staker is unstaking, because the amount was already removed: [2](#0-1) 

**`switch_staking_delegation_pool`** then **unconditionally** adds the switched amount to `total_stake`, with no check on the source staker's unstake status: [3](#0-2) 

The function only checks that the **destination** staker is not unstaking: [4](#0-3) 

### Impact Explanation

After the switch, `total_stake` is permanently inflated by `switched_amount`. Since rewards are proportional to `staker_stake / total_stake`, every staker and delegator in the protocol receives fewer rewards than they are entitled to. The inflation persists until the destination staker eventually calls `unstake_intent`, which may never happen. This constitutes **theft of unclaimed yield** from all protocol participants.

### Likelihood Explanation

The scenario requires only that a staker calls `unstake_intent` before a pool member in that staker's pool calls `exit_delegation_pool_intent` followed by `switch_delegation_pool`. This is a natural and common flow: a staker announces exit, and their delegators migrate to a new staker. Any pool member can trigger this unintentionally. No privileged access is required.

### Recommendation

In `switch_staking_delegation_pool`, look up the source staker associated with `from_pool` and check whether that staker has `unstake_time.is_some()`. If so, skip the `add_to_total_stake` call, mirroring the guard already present in `update_delegated_stake`.

### Proof of Concept

1. Staker A stakes with a delegation pool (pool P). Pool member delegates amount `D` to pool P. `total_stake = S + D`.
2. Staker A calls `unstake_intent()`. `total_stake` decreases by `D` (own stake also removed). `total_stake = 0` (for simplicity, own stake = 0).
3. Pool member calls `exit_delegation_pool_intent(D)` on pool P. Pool P calls `remove_from_delegation_pool_intent` on the staking contract. Because `staker_info.unstake_time.is_some()`, `update_delegated_stake` skips the `total_stake` update. `total_stake` remains `0`. Intent records `D`.
4. Pool member calls `switch_delegation_pool(to_staker=B, to_pool=P2, amount=D)` on pool P. Pool P calls `switch_staking_delegation_pool`. `add_to_total_stake(D)` is called unconditionally. `total_stake = D`.
5. **Result**: `total_stake = D` even though the only active staker (B) has `0` delegated balance from this switch yet (the balance trace is updated at `epoch + K`). The `D` that was removed during `unstake_intent` has been re-added without a corresponding real stake, permanently inflating `total_stake` and diluting all future reward distributions.
