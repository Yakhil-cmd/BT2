Audit Report

## Title
Unbounded `ingress_messages` deserialization and uncapped task spawning in `TryFrom<pb::StrippedBlockProposal>` / `assemble_message` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary
`TryFrom<pb::StrippedBlockProposal>` collects `value.ingress_messages` into a `Vec<SignedIngressId>` with no length cap, causing O(N) heap allocation before any bouncer check. If the bouncer subsequently accepts the artifact (e.g., the Byzantine peer is a legitimate block proposer advertising a real block hash), `assemble_message` spawns one unbounded tokio task per entry, enabling OOM or severe memory pressure on the receiving replica from a single Byzantine peer below the fault threshold.

## Finding Description
`TryFrom<pb::StrippedBlockProposal>` at `stripped.rs:100–108` iterates `value.ingress_messages` and collects into a `Vec<SignedIngressId>` with no length check. The consensus-layer constant `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` (`rs/limits/src/lib.rs:78`) is enforced only during block validation, not during P2P deserialization. The QUIC transport allows messages up to 128 MB (`rs/p2p/quic_transport/src/lib.rs:74`), accommodating ~1.56 M `StrippedIngressMessage` entries (~82 bytes each in proto encoding).

The deserialization (and thus the O(N) Vec allocation) occurs before the bouncer check: the `artifact` parameter arriving at `FetchStrippedConsensusArtifact::assemble_message` is already a deserialized `MaybeStrippedConsensusMessage`; the bouncer is evaluated only inside `FetchArtifact::download_artifact` at line 212 (`assembler.rs:183–190`). If the bouncer returns `Wants` (e.g., the Byzantine peer supplied a valid `unstripped_consensus_message_id` for a real block they proposed), `BlockProposalAssembler::new` maps the N entries 1:1 into a second `Vec<(SignedIngressId, Option<SignedIngress>)>` (`assembler.rs:572–588`), and the loop at `assembler.rs:214–226` spawns one `get_or_fetch` tokio task per entry with no upper bound. The developer comment at `assembler.rs:440–441` explicitly assumes "at most 1000 elements in the vector" but this invariant is never enforced at the deserialization layer. `HASHES_IN_BLOCKS_ENABLED = true` (`rs/consensus/features/src/lib.rs:6`) confirms this code path is active in production.

The final `try_assemble` ID-mismatch check (`assembler.rs:641–645`) would eventually reject the crafted proposal, but the task explosion and memory pressure occur before that point.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* A single Byzantine replica below the fault threshold can repeatedly crash or severely degrade individual honest replicas via OOM, reducing subnet fault tolerance and potentially blocking consensus progress if enough replicas are targeted.

## Likelihood Explanation
`HASHES_IN_BLOCKS_ENABLED = true` in production. A single Byzantine node that is a legitimate block proposer can craft a `StrippedBlockProposal` with a valid `unstripped_consensus_message_id` (for a real block it proposed) but with ~1.56 M fake `ingress_messages` entries, staying within the 128 MB QUIC cap. No subnet-majority corruption is required; one Byzantine node suffices. The attack is repeatable every consensus round in which the Byzantine node is the block proposer.

## Recommendation
Add a length cap in `TryFrom<pb::StrippedBlockProposal>` before collecting `ingress_messages`, matching the existing consensus-layer limit:

```rust
const MAX_INGRESS_MESSAGES_PER_STRIPPED_BLOCK: usize = 1000;

if value.ingress_messages.len() > MAX_INGRESS_MESSAGES_PER_STRIPPED_BLOCK {
    return Err(ProxyDecodeError::Other(format!(
        "Too many ingress_messages: {} > {}",
        value.ingress_messages.len(),
        MAX_INGRESS_MESSAGES_PER_STRIPPED_BLOCK,
    )));
}
```

Apply the same cap to `stripped_idkg_dealings`. This enforces the invariant the developer already assumed at `assembler.rs:440–441` at the correct deserialization layer.

## Proof of Concept
Construct a `pb::StrippedBlockProposal` with a valid `pruned_block_proposal` and `unstripped_consensus_message_id` (from a real block proposal) and `ingress_messages` containing 10^6 minimal `StrippedIngressMessage` entries (~82 bytes each, total ~82 MB, within the 128 MB cap). Call `StrippedBlockProposal::try_from(proto)` and observe RSS growth proportional to N. Pass the result to `BlockProposalAssembler::new` and call `missing_stripped_messages()` — confirm it returns 10^6 entries. Pass to `FetchStrippedConsensusArtifact::assemble_message` with a bouncer returning `Wants` — confirm 10^6 tokio tasks are spawned and RSS grows by several GB before `try_assemble` returns an error.
