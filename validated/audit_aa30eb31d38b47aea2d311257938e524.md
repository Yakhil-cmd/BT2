### Title
`DelegateAction` Expiry Boundary Off-by-One Allows Execution at `max_block_height` — (`File: runtime/runtime/src/actions.rs`)

### Summary

`apply_delegate_action` uses a strict `>` comparison to check whether a `DelegateAction` has expired, but the protocol specification and field documentation both require a `>=` comparison. At the exact boundary block (`block_height == max_block_height`), the code accepts and executes a delegate action that the user intended to be invalid, breaking the authorization invariant that `max_block_height` is meant to enforce.

### Finding Description

`DelegateAction.max_block_height` is documented as:

> "The maximal height of the block in the blockchain below which the given DelegateAction is valid."

The runtime spec (`docs/RuntimeSpec/Actions.md`) states the expiry condition as:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired`

The production execution path in `apply_delegate_action` checks:

```rust
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
``` [1](#0-0) 

The operator is `>` (strictly greater than), not `>=`. When `apply_state.block_height == max_block_height`, the condition is `false`, so the action proceeds to signature verification, nonce advancement, and inner-action receipt generation — all of which mutate on-chain state — at a block height where the user's signed intent says the action must be rejected.

The only existing unit test for this boundary (`test_delegate_action_max_height`) sets `block_height = max_block_height + 1`, leaving the `block_height == max_block_height` case untested and unguarded:

```rust
// Setup current block as higher than max_block_height. Must fail.
let apply_state =
    create_apply_state(signed_delegate_action.delegate_action.max_block_height + 1);
``` [2](#0-1) 

The `DelegateAction` struct definition and the `VersionedDelegateActionRef::max_block_height()` accessor confirm the field is user-controlled and signed: [3](#0-2) [4](#0-3) 

### Impact Explanation

A relayer (unprivileged, not a validator) who holds a user-signed `DelegateAction` with `max_block_height = N` can submit it at block `N` and have it accepted and executed on-chain. The user's access-key nonce is advanced and the inner actions (transfers, function calls, etc.) are dispatched as new receipts — all at a block height the user's signature explicitly excluded.

The corrupted values are:
- **Access key nonce**: advanced at block `N` when it must not be
- **Inner-action receipts**: created and queued at block `N` when they must not be
- **Account balances / contract state**: mutated by those receipts

The invariant broken is: *a `DelegateAction` signed with `max_block_height = N` must not produce any on-chain effect at or after block `N`*.

### Likelihood Explanation

Any relayer who holds a signed `DelegateAction` can exploit this by timing submission to block `N`. No special privilege is required beyond possessing the signed payload. The window is exactly one block, but in NEAR's ~1-second block time this is a deterministic, repeatable condition, not a race.

### Recommendation

Change the comparison from `>` to `>=`:

```rust
// Before (incorrect):
if apply_state.block_height > delegate_action.max_block_height() {

// After (correct, matches spec and field documentation):
if apply_state.block_height >= delegate_action.max_block_height() {
``` [5](#0-4) 

Add a unit test that asserts `DelegateActionExpired` when `block_height == max_block_height`, mirroring the existing `test_delegate_action_max_height` test but at the exact boundary.

### Proof of Concept

1. User creates and signs a `DelegateAction` with `max_block_height = N`, intending it to be invalid at block `N`.
2. Relayer waits until the chain reaches block `N`.
3. Relayer submits the signed `DelegateAction` inside a transaction at block `N`.
4. `apply_delegate_action` evaluates `N > N` → `false`; the expiry branch is skipped.
5. Signature verification passes (the signature is valid).
6. `validate_delegate_action_key` advances the access-key nonce.
7. A new receipt carrying the inner actions is pushed to `result.new_receipts` and executed.
8. On-chain state (balances, contract storage, nonces) is mutated at block `N`, one block past the user's intended expiry. [6](#0-5) [7](#0-6)
