### Title
`delta_gas` Floor Division in `compute_gas_refund` Favors User Over Protocol — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the conversion from native resource consumption to EVM gas units uses integer floor division (`native_used / native_per_gas`). This systematically rounds the extra gas charged to the user **down**, meaning the user pays slightly less than the true cost of their native resource consumption. The rounding direction should favor the protocol (ceiling), not the user (floor). This is a direct analog of the reported rounding-direction vulnerability class.

### Finding Description

`compute_gas_refund` implements the ZKsync OS double-resource accounting reconciliation step. After execution, it computes `delta_gas` — the additional EVM gas to charge when native resource consumption exceeds what the ergs-based gas accounting already covers:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs L69-L79
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

The expression `native_used / native_per_gas` is Rust integer division, which truncates toward zero (floor for positive values). This means the gas equivalent of native consumption is rounded **down**, so `delta_gas` is at most `floor(native_used / native_per_gas) - gas_used` rather than `ceil(native_used / native_per_gas) - gas_used`.

The consequence: up to `native_per_gas - 1` native units of consumption are never charged to the user. The user's `gas_used` is understated by up to 1 gas unit, and the operator receives a correspondingly smaller fee.

The same floor-rounding issue appears in the minimum gas floor:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs L426
let min_gas_used = context.minimal_ergs_to_charge.0 / ERGS_PER_GAS;
```

`minimal_ergs_to_charge.0 / ERGS_PER_GAS` also truncates down, meaning the enforced minimum gas is slightly below the true intrinsic cost in ergs.

Note that `native_per_gas` itself is computed with **ceiling** division (correctly favoring the protocol):

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs L135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

But the downstream `delta_gas` division undoes part of this protection.

### Impact Explanation

Every transaction whose native resource consumption exceeds the ergs-based gas consumption (i.e., whenever `delta_gas > 0`) underpays by up to 1 gas unit. The operator (coinbase) receives `gas_used * gas_price` tokens, where `gas_used` is understated by at most 1. The user retains the corresponding token amount as an unearned refund. While the per-transaction loss is bounded (< 1 gas unit × gas_price), it is systematic and always favors the user — the protocol never benefits from this rounding.

### Likelihood Explanation

The `delta_gas > 0` branch is triggered on any transaction where native resource consumption is the binding constraint (i.e., native-heavy transactions such as those with significant pubdata, complex proving work, or high `native_per_gas` ratios). This is a normal operating condition, not an edge case. Any unprivileged user submitting a standard EVM transaction can reach this code path.

### Recommendation

Replace the floor division in `delta_gas` with ceiling division to ensure the protocol is never shortchanged:

```rust
// In compute_gas_refund, replace:
(native_used / native_per_gas) as i64 - (gas_used as i64)

// With:
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

Similarly, replace the floor division for `min_gas_used`:

```rust
// Replace:
let min_gas_used = context.minimal_ergs_to_charge.0 / ERGS_PER_GAS;

// With:
let min_gas_used = context.minimal_ergs_to_charge.0.div_ceil(ERGS_PER_GAS);
```

### Proof of Concept

**Setup:**
- `native_price = 10`, `gas_price = 11`
- `native_per_gas = ceil(11/10) = 2` (computed correctly with `div_ceil`)
- `gas_limit = 100`, `native_limit = 200`
- After execution: `gas_used = 50` (from ergs), `native_used = 101`

**Current behavior (floor):**
```
delta_gas = floor(101 / 2) - 50 = 50 - 50 = 0
final gas_used = 50
```
The user consumed 101 native units, which at rate 2 native/gas corresponds to 50.5 gas. They are charged for only 50 gas. The 0.5 gas unit (worth `0.5 * gas_price` tokens) is silently absorbed by the user.

**Correct behavior (ceiling):**
```
delta_gas = ceil(101 / 2) - 50 = 51 - 50 = 1
final gas_used = 51
```
The user is charged for 51 gas, fully covering their native consumption.

The attacker-controlled entry path is a standard unprivileged EVM transaction submitted to the ZKsync OS bootloader. The vulnerable step is `compute_gas_refund` in [1](#0-0) , called from both the ZK transaction flow [2](#0-1)  and the L1 transaction flow [3](#0-2) .
