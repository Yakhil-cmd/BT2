### Title
Missing Verification of Participant Contributions in CKD Coordinator Allows Malicious Participant to Corrupt Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` blindly accumulates `(norm_big_y, norm_big_c)` shares from every participant without any cryptographic proof that each participant used their correct secret key share. A single malicious participant can send arbitrary group elements, silently corrupting the final `CKDOutput` delivered to the application.

---

### Finding Description

In `do_ckd_coordinator` (lines 35–58 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives each participant's contribution and unconditionally adds it to the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute, inside `compute_signature_share`:

- `big_y  = y_i · G`  (random blinding point)
- `big_c  = x_i · H(pk ‖ app_id) + y_i · app_pk`  (secret share contribution masked by blinding)
- `norm_big_y = λ_i · big_y`,  `norm_big_c = λ_i · big_c` [2](#0-1) 

The coordinator then forms the final output `(Y, C)` by summing all contributions, and the application unmasks it as `C − a·Y = msk · H(pk ‖ app_id)`. [3](#0-2) 

**The missing check:** there is no zero-knowledge proof (e.g., a Chaum-Pedersen / DLEQ proof) that the participant's `big_c` was formed using the same `y_i` as `big_y` and the correct secret share `x_i`. The coordinator has no way to distinguish a well-formed contribution from an arbitrary pair of group elements. This is structurally identical to the reported DeFi bug: a value is received from an external party and consumed without verifying it meets the required minimum/correctness criterion.

---

### Impact Explanation

A malicious participant sends `(norm_big_y', norm_big_c')` of their choosing. The coordinator computes:

```
Y_final = Y_honest + norm_big_y'
C_final = C_honest + norm_big_c'
```

The application unmasks to `C_final − a·Y_final`, which equals `msk·H(pk‖app_id) + (norm_big_c' − a·norm_big_y')` — an attacker-controlled offset. The derived confidential key is silently wrong. Every honest party accepts this corrupted `CKDOutput` as valid because no verification step exists.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

- Any single participant in the protocol can mount this attack; no threshold of colluders is required.
- The attack requires only sending two arbitrary `G1` group elements instead of the honestly computed ones — trivially achievable by any participant who controls their own protocol instance.
- There is no detection mechanism: the coordinator returns `Ok(Some(ckd_output))` regardless of whether contributions were well-formed. [4](#0-3) 

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a **DLEQ (Chaum-Pedersen) proof** demonstrating that:

1. `big_y = y_i · G` and `big_c − x_i · H(pk‖app_id) = y_i · app_pk` share the same discrete logarithm `y_i` (proving the blinding is consistent).
2. `x_i · G2 = vk_share_i` (proving the secret share used matches the public verification share from DKG).

The coordinator must verify all such proofs before accumulating contributions, and abort if any proof is invalid, identifying the malicious participant.

---

### Proof of Concept

1. Run the CKD protocol with `n` participants, one of which is malicious.
2. The malicious participant, instead of calling `compute_signature_share`, sends `(G1::identity(), G1::identity())` (or any arbitrary pair) to the coordinator.
3. The coordinator's loop at lines 50–55 adds these values without complaint.
4. The final `CKDOutput` is `(Y − λ_i·y_i·G, C − λ_i·(x_i·H + y_i·app_pk))`.
5. The application calls `ckd_output.unmask(app_sk)` and receives a value that is not `msk·H(pk‖app_id)`.
6. No error is raised anywhere in the protocol; the corruption is silent. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,

    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
