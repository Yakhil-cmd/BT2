### Title
Gateway Stateless Validator Unconditionally Applies `min_gas_price` to `l2_gas.max_price_per_unit` Even When `l2_gas.max_amount = 0`, Rejecting Valid L1-Only Transactions - (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies the `min_gas_price` floor to `l2_gas.max_price_per_unit` unconditionally, without first checking whether `l2_gas.max_amount` is zero. In production, where `min_gas_price = 8_000_000_000`, any V3 transaction that legitimately sets `l2_gas.max_amount = 0` and `l2_gas.max_price_per_unit = 0` is rejected with `MaxGasPriceTooLow`, even though L2 gas is not being used and its price is economically irrelevant. The test suite silently masks this by using a test-only config with `min_gas_price = 0`.

---

### Finding Description

In `validate_resource_bounds`, after the zero-bounds guard, the following check fires unconditionally:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs, line 71
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
``` [1](#0-0) 

There is no guard on `l2_gas.max_amount`. When `l2_gas.max_amount = 0`, the L2 gas price per unit is irrelevant: `max_possible_fee` for L2 gas is `0 * price = 0` regardless of price. No L2 gas fee will ever be charged. Yet the check still fires.

The production default config sets `min_gas_price = 8_000_000_000`:

```rust
// crates/apollo_gateway_config/src/config.rs, line 192
min_gas_price: 8_000_000_000,
``` [2](#0-1) 

This is also confirmed in the node schema:

```json
"gateway_config.static_config.stateless_tx_validator_config.min_gas_price": {
    "value": 8000000000
}
``` [3](#0-2) 

The test suite masks this by using `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` with `min_gas_price: 0` for the `valid_l1_gas` positive test case:

```rust
static DEFAULT_VALIDATOR_CONFIG_FOR_TESTING: LazyLock<StatelessTransactionValidatorConfig> =
    LazyLock::new(|| StatelessTransactionValidatorConfig {
        validate_resource_bounds: false,
        min_gas_price: 0,   // <-- masks the production bug
        ...
    });
``` [4](#0-3) 

The `valid_l1_gas` test case explicitly enables `validate_resource_bounds: true` but inherits `min_gas_price: 0` from the testing config, so it passes. With the production `min_gas_price = 8_000_000_000`, the same transaction would be rejected:

```rust
#[case::valid_l1_gas(
    StatelessTransactionValidatorConfig {
        validate_resource_bounds: true,
        ..*DEFAULT_VALIDATOR_CONFIG_FOR_TESTING  // min_gas_price = 0 here
    },
    RpcTransactionArgs {
        resource_bounds: AllResourceBounds {
            l1_gas: NON_EMPTY_RESOURCE_BOUNDS,
            ..Default::default()  // l2_gas = { max_amount: 0, max_price_per_unit: 0 }
        },
        ..Default::default()
    }
)]
``` [5](#0-4) 

The only negative test for this check (`max_l2_gas_price_below_min`) uses `NON_EMPTY_RESOURCE_BOUNDS` for `l2_gas` (i.e., `l2_gas.max_amount > 0`), so it never exercises the `max_amount = 0` path with a non-zero `min_gas_price`: [6](#0-5) 

The analog to the external report is direct: just as the Solidity `swap()` forwards `msg.value` on the non-native-token path (a field that is irrelevant to that path), the gateway applies the `min_gas_price` check to `l2_gas.max_price_per_unit` on the zero-L2-gas-amount path (a field that is irrelevant when `max_amount = 0`). In both cases, a field that should be ignored for a particular configuration is instead acted upon, causing incorrect behavior — there, tokens get stuck; here, valid transactions are dropped.

---

### Impact Explanation

**High. Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Any V3 transaction (Invoke, Declare, DeployAccount) that sets `l2_gas.max_amount = 0` and `l2_gas.max_price_per_unit = 0` — a legitimate configuration for transactions that only consume L1 gas or L1 data gas — is rejected at the stateless gateway with `MaxGasPriceTooLow`. The transaction never reaches the mempool or blockifier. The user receives a hard error and cannot proceed without artificially inflating `l2_gas.max_price_per_unit` to a value ≥ `min_gas_price`, even though no L2 gas will be consumed.

---

### Likelihood Explanation

The `AllResourceBounds` struct defaults `l2_gas` to `{ max_amount: 0, max_price_per_unit: 0 }`. Any client that constructs a V3 transaction specifying only L1 gas bounds and leaving L2 gas at its natural zero default will trigger this rejection. This is a common pattern for transactions that do not use L2 gas (e.g., pre-0.13.3 style transactions submitted as V3, or transactions explicitly opting out of L2 gas). The trigger is fully unprivileged and requires no special access.

---

### Recommendation

Guard the `min_gas_price` check with a condition on `l2_gas.max_amount`. The price per unit is only meaningful when gas will actually be consumed:

```diff
- if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
+ if resource_bounds.l2_gas.max_amount.0 > 0
+     && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
      return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
          gas_price: resource_bounds.l2_gas.max_price_per_unit,
          min_gas_price: self.config.min_gas_price,
      });
  }
```

Additionally, add a test case that runs the `valid_l1_gas` scenario against the production default config (`DEFAULT_VALIDATOR_CONFIG`, not `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING`) to prevent regression.

---

### Proof of Concept

Submit a V3 Invoke transaction to the gateway with the following resource bounds:

```json
{
  "resource_bounds": {
    "L1_GAS":      { "max_amount": "0x3E8", "max_price_per_unit": "0xE8D4A51000" },
    "L2_GAS":      { "max_amount": "0x0",   "max_price_per_unit": "0x0" },
    "L1_DATA_GAS": { "max_amount": "0x0",   "max_price_per_unit": "0x0" }
  }
}
```

**Expected:** Transaction accepted (L1 gas bounds are non-zero and above minimum; L2 gas is unused).

**Actual (production config, `min_gas_price = 8_000_000_000`):** Gateway rejects with:
```
MaxGasPriceTooLow { gas_price: 0, min_gas_price: 8000000000 }
```

The check at line 71 fires because `0 < 8_000_000_000`, even though `l2_gas.max_amount = 0` makes the L2 gas price irrelevant. [7](#0-6)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
