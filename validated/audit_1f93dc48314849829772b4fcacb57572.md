### Title
Missing `validate_no_value_attached_to_precompile` Guard Allows Permanent ETH Lock at Secp256r1 Precompile Address - (`engine-precompiles/src/secp256r1.rs`)

### Summary

`Secp256r1::run` does not call `validate_no_value_attached_to_precompile`, unlike every other Aurora-specific precompile. Because the EVM executor transfers value to the precompile address before invoking the precompile, and because `Secp256r1::run` always returns `Ok(...)` (never reverts), any ETH attached to a CALL targeting `0x0000...0100` is committed to that address with no recovery path.

### Finding Description

`Secp256r1::run` ignores the `context` parameter entirely (it is named `_context`): [1](#0-0) 

By contrast, every other Aurora-specific precompile that could receive value calls `utils::validate_no_value_attached_to_precompile(context.apparent_value)` as its first action:

- `PrepaidGas::run` — [2](#0-1) 
- `RandomSeed::run` — [3](#0-2) 
- `PredecessorAccount::run` — [4](#0-3) 
- `CurrentAccount::run` — [5](#0-4) 
- `PromiseResult::run` — [6](#0-5) 
- `CrossContractCall::run_with_handle` — [7](#0-6) 

That guard returns `Err(ExitError::Other("ATTACHED_VALUE_ERROR"))` when value > 0, which causes the EVM executor to revert the entire call frame — including the value transfer: [8](#0-7) 

The existence of this guard in every other precompile is direct evidence that the EVM executor (aurora_evm / sputnikvm `StackExecutor`) **does** transfer value to the precompile address before invoking the precompile logic. Returning `Err` is the only mechanism to revert that transfer. `Secp256r1::run` never returns `Err` due to value — it only errors on `OutOfGas`. For all other inputs (including invalid signatures), it returns `Ok(PrecompileOutput::without_logs(cost, empty_vec))`: [9](#0-8) 

Because the call succeeds, the EVM commits the state diff, including the balance increase at `0x0000...0100`. The precompile address has no EVM bytecode and no withdrawal function, so the ETH is unrecoverable.

### Impact Explanation

Any ETH sent as `value` in a CALL to `Secp256r1::ADDRESS` (`0x0000000000000000000000000000000000000100`) is permanently locked. The address has no code, no `receive`/`fallback`, and cannot be deployed to (precompile addresses take precedence). The funds are irrecoverable — **permanent freezing of funds**.

### Likelihood Explanation

The attack requires no privilege. Any EVM user with an ETH balance on Aurora can trigger it with a single transaction: `CALL 0x0000...0100 value=N data=<any 160 bytes>`. The call succeeds (returns `Ok`), the value is committed, and the ETH is gone. This is trivially reachable through the standard `submit` entrypoint.

### Recommendation

Add the value guard as the first statement in `Secp256r1::run`, matching the pattern used by all other Aurora precompiles:

```rust
fn run(
    &self,
    input: &[u8],
    target_gas: Option<EthGas>,
    context: &Context,
    _is_static: bool,
) -> EvmPrecompileResult {
    utils::validate_no_value_attached_to_precompile(context.apparent_value)?;
    // ... rest of implementation
}
```

    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        _context: &Context,
        _is_static: bool,
    ) -> EvmPrecompileResult {
        let cost = Self::required_gas(input)?;
        if let Some(target_gas) = target_gas
            && cost > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        // Return empty output on failure according to EIP-7951
        let output = Self::execute(input).unwrap_or_default();
        Ok(PrecompileOutput::without_logs(cost, output))
    }
}
