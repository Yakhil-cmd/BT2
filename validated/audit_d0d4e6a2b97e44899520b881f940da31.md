### Title
`protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, leaving pool permanently paused after protocol unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.protocolUnpausePool` is supposed to restore a protocol-paused pool (`pauseLevel=2`) to active state (`pauseLevel=0`). Instead it calls `setPause(1)`, transitioning the pool to admin-paused state. Swaps remain blocked because `_checkNotPaused` reverts on any non-zero `pauseLevel`. The factory owner has no further path to reach `pauseLevel=0`; only the pool admin can do so via `unpausePool`.

### Finding Description [2](#0-1) 

`protocolUnpausePool` reads the current `pauseLevel`, enforces `cur == 2`, then calls `IMetricOmmPoolFactoryActions(pool).setPause(1)`. The argument should be `0` (active), not `1` (admin-paused). [3](#0-2) 

`setPause` accepts any value 0–2 and writes it directly to `pauseLevel`. [4](#0-3) 

`_checkNotPaused` reverts on `pauseLevel != 0`, so `pauseLevel=1` still blocks all swaps. [5](#0-4) 

`unpausePool` (the only path to `pauseLevel=0`) is `onlyPoolAdmin`. The factory owner cannot call it.

### Impact Explanation
After the factory owner executes the full protocol-pause/unpause cycle (`protocolPausePool` → `protocolUnpausePool`), the pool is left at `pauseLevel=1`. All swaps revert with `PoolPaused`. The factory owner has no further lever to restore the pool to active state. The pool admin must call `unpausePool` to complete the unpause. If the pool admin key is unavailable, rotated, or the admin contract is non-functional, the pool is permanently bricked: swaps, and any extension hooks gated on swap, are permanently blocked. This constitutes broken core pool functionality and complete loss of swap utility for all LPs and traders.

### Likelihood Explanation
The factory owner is expected to use `protocolPausePool`/`protocolUnpausePool` in emergency scenarios (e.g., oracle manipulation, exploit response). Every invocation of `protocolUnpausePool` silently fails to restore the pool to active state. The bug is deterministic and triggered on every normal use of the function.

### Recommendation
Change line 402 in `MetricOmmPoolFactory.sol`:

```solidity
// Before (wrong):
IMetricOmmPoolFactoryActions(pool).setPause(1);

// After (correct):
IMetricOmmPoolFactoryActions(pool).setPause(0);
```

Also update the `InvalidPauseTransition` revert argument on line 401 from `1` to `0` to match the intended target level.

### Proof of Concept
1. Deploy pool; assert `pauseLevel == 0`.
2. Factory owner calls `protocolPausePool(pool)` → `setPause(2)` → `pauseLevel == 2`.
3. Factory owner calls `protocolUnpausePool(pool)` → `setPause(1)` → `pauseLevel == 1`.
4. Assert `pauseLevel == 1` (not 0).
5. Call `pool.swap(...)` → reverts with `PoolPaused`.
6. Factory owner has no function to call `setPause(0)` directly; only pool admin's `unpausePool` can do so.
