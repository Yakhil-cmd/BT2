### Title
`skip_stateful_validations` Admits Invoke Transactions With Invalid Signatures, Enabling Unauthorized Nonce Increment and Fee Drain - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function skips the account's `__validate__` entry point (signature verification) for any Invoke transaction with `nonce=1` when the sender address has *any* transaction in the mempool or recent committed block. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can submit a crafted Invoke with `nonce=1` and an **invalid signature** for the victim's address. The gateway admits it without signature verification. When the batcher later executes the block, the Invoke fails at `__validate__`, but the nonce is already incremented and a fee is charged — neither authorized by the victim.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. Transaction type is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in state).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

Condition 4 is checked via: [2](#0-1) 

This returns `true` if **any** transaction for the address is in the pool or committed state — not specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incomplete: an attacker-controlled Invoke with `nonce=1` that was previously admitted via this same skip mechanism would also satisfy the check.

**How the skip propagates to disable `__validate__`:** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate` is set to `false`. Inside `StatefulValidator::perform_validations`: [4](#0-3) 

The `__validate__` call is entirely skipped. No signature verification occurs at gateway admission time.

**Attack scenario:**

1. Victim submits a valid `deploy_account` for address `A` (nonce=0). It enters the mempool.
2. Attacker observes the mempool (P2P-propagated) and submits an Invoke for address `A` with `nonce=1` and a **forged/invalid signature**.
3. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (account not yet in state)
   - `account_tx_in_pool_or_recent_block(A)` → `true` (victim's `deploy_account` is in pool) ✓
4. `execution_flags.validate = false` → `__validate__` is skipped. The Invoke is admitted to the mempool.
5. Batcher executes the block: `deploy_account` runs first (nonce=0), deploying account `A`.
6. Batcher executes the attacker's Invoke (nonce=1). `perform_pre_validation_stage` increments the nonce **before** fee checks: [5](#0-4) 

7. `__validate__` runs (batcher uses default `validate=true` flags) and fails — invalid signature.
8. Transaction is reverted. Victim's nonce is now `2`. If the account has balance, a fee is charged.

**The `validate_by_mempool` call does not check signatures:** [6](#0-5) 

The mempool's `validate_tx` only checks for duplicate transactions and nonce ordering — not signatures.

---

### Impact Explanation

**High — Mempool/gateway admits invalid transactions before sequencing.**

- An attacker can submit an Invoke with an invalid (or entirely fabricated) signature for any address that has a pending `deploy_account` in the mempool.
- The gateway admits it without any signature check.
- At execution time, the victim's nonce is incremented from 1 to 2 without the victim's authorization.
- If the account has a non-zero balance (e.g., funded via L1→L2 message or constructor), a fee is charged for a transaction the victim never signed.
- The victim's legitimate Invoke (nonce=1) is displaced; they must resubmit with nonce=2.
- The attacker can also use fee escalation to replace the victim's own valid nonce=1 Invoke with the attacker's invalid one (by bidding a higher tip/gas price).

---

### Likelihood Explanation

**Medium.** The mempool is observable via P2P gossip. Any node can see pending `deploy_account` transactions. The attack window is the time between the `deploy_account` entering the mempool and being included in a block. This window can be several seconds to minutes. No privileged access is required; the attacker only needs to submit a standard Invoke RPC call with a crafted (invalid) signature.

---

### Recommendation

1. **Narrow the `account_tx_in_pool_or_recent_block` check**: Instead of checking for *any* transaction, verify that the mempool specifically contains a `deploy_account` transaction for the address. Add a dedicated `deploy_account_in_pool(address)` query to the mempool.

2. **Alternatively, perform a lightweight signature pre-check at the gateway**: Even when skipping the full `__validate__` execution, verify that the signature field is structurally consistent with the expected account class (e.g., correct length for a standard ECDSA account) before admission.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
