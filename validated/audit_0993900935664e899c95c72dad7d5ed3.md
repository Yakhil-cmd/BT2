### Title
Missing Validation of Participant Share Contributions in CKD Coordinator Allows Malicious Participant to Corrupt CKD Output - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` receives `(big_y, big_c)` ElGamal share contributions from every participant and blindly aggregates them without any validity check. A single malicious participant can substitute arbitrary group elements for their correct share, causing the coordinator to silently output a corrupted, unusable confidential key. No error is raised and no culprit is identified.

### Finding Description

**Root cause — lines 50–55 of `src/confidential_key_derivation/protocol.rs`:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is supposed to send `(λ_i · y_i · G,  λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk))`. The coordinator sums all contributions and returns the aggregate as the CKD output. There is no check that:

1. The received group elements are valid (on the curve, not the identity, in the correct subgroup).
2. The participant's `(big_y, big_c)` pair is consistent with their committed key share — i.e., there is no zero-knowledge proof of correct computation.

Contrast this with the DKG protocol (`src/dkg.rs`), which validates every received share with `validate_received_share` (lines 259–286) and every commitment with `verify_proof_of_knowledge` (lines 172–218) and `verify_commitment_hash` (lines 222–236). The CKD protocol performs none of these checks.

**Exploit path:**

1. A legitimate participant `P_m` is enrolled in a CKD session.
2. Instead of computing the correct `(norm_big_y, norm_big_c)`, `P_m` sends arbitrary group elements `(big_y', big_c')` to the coordinator.
3. The coordinator adds `big_y'` and `big_c'` to its running sum without complaint.
4. The final output is `(Y + big_y', C + big_c')`.
5. When the TEE application unmasks the output with `app_sk`, it obtains `(C + big_c') − app_sk · (Y + big_y') = msk · H(pk ‖ app_id) + (big_c' − app_sk · big_y')`, which is not the correct confidential key unless `big_c' = app_sk · big_y'` — a condition the attacker cannot satisfy without knowing `app_sk`.
6. The coordinator returns `Ok(Some(ckd_output))` with no indication of failure; the honest coordinator and the TEE application have no way to detect the corruption.

### Impact Explanation

A single malicious participant (no privileged access required beyond being a valid protocol member) can cause the coordinator to output an incorrect, unusable CKD result. The TEE application silently receives a wrong confidential key. This maps directly to the allowed High impact: *"Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs."* Because the corruption is silent and unattributable, the honest coordinator cannot retry with a different participant set or identify the culprit.

### Likelihood Explanation

Any participant enrolled in a CKD session can trigger this with a single malformed message. No cryptographic break, no leaked key, and no privileged network position is required. The attacker's only precondition is being a valid member of the participant list, which is the baseline attacker profile described in `RESEARCHER.md`.

### Recommendation

Add a zero-knowledge proof of correct share computation to each participant's message, analogous to the Schnorr proof-of-knowledge used in DKG. At minimum, validate that received group elements are on the curve, are torsion-free, and are not the identity — mirroring the checks already present in `verify_signature` (`src/confidential_key_derivation/ciphersuite.rs`, lines 224–229). A complete fix requires a non-interactive proof binding `big_c` to the participant's committed public key share and the public inputs `(pk, app_id, app_pk)`, so that an incorrect contribution is detectable and attributable.

### Proof of Concept

```
Setup:
  participants = [P1, P2, P3], coordinator = P1, threshold = 3-of-3

Normal execution:
  Each Pi sends (λ_i · y_i · G,  λ_i · (x_i · H(pk‖app_id) + y_i · app_pk))
  Coordinator sums → correct CKD output → unmask yields msk · H(pk‖app_id)

Attack (P3 is malicious):
  P3 sends (G, G)  ← arbitrary, not derived from x_3 or y_3
  Coordinator computes:
    Y'  = λ_1·y_1·G + λ_2·y_2·G + G
    C'  = λ_1·(x_1·H+y_1·A) + λ_2·(x_2·H+y_2·A) + G
  unmask: C' − app_sk·Y' = msk·H + G − app_sk·G  ≠  msk·H
  Result: coordinator returns Ok(Some(corrupted_output)) with no error.
  TEE application uses wrong key; no participant is identified as malicious.

    from: Participant,
