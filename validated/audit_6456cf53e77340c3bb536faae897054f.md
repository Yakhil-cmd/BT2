### Title
Unvalidated `offerURL` from Blockchain Notification Allows Arbitrary URI Fetch — (`packages/gui/src/hooks/useBlockchainNotifications.tsx`)

### Summary
An on-chain attacker can send a blockchain notification whose hex-encoded JSON payload contains a malicious URI in the `d.u` field. This value is extracted and stored verbatim as `offerURL` with no scheme or format validation, then passed directly to `useOffer()`, which is designed to fetch offer data from that URL.

### Finding Description

In `prepareNotifications`, the `u` field from the decoded notification JSON is assigned directly to `offerURL` with no validation: [1](#0-0) 

The `url` variable (which is `data.u` from attacker-controlled blockchain data) is placed into the `Notification` object at lines 108 and 118 without any check on URI scheme, format, or allowlist.

In `NotificationOffer.tsx`, this `offerURL` is extracted and passed directly to `useOffer()`: [2](#0-1) 

`useOffer` delegates to `OffersProvider` context via `getOffer(offerURLOrData)`: [3](#0-2) 

The designed purpose of `offerURL` is to be fetched — the notification flow exists specifically so the GUI downloads and displays an offer from a remote URL. There is no URL validation anywhere in this chain.

**Confirmed gaps:**
- No scheme validation (`http://`, `https://` only) in `prepareNotifications`
- No sanitization before passing to `useOffer()`
- The attacker entry point (blockchain `send_notification` RPC to victim's puzzle hash) is a documented, unprivileged on-chain operation

**Unconfirmed (requires reading `OffersProvider.tsx` and Electron main process config):**
- Whether `getOffer()` uses `fetch()` or another mechanism to retrieve the URL
- Whether Electron's `webSecurity` / `contextIsolation` settings restrict `file://` URI access in the renderer

### Impact Explanation
If `OffersProvider.getOffer()` uses `fetch()` (the expected behavior for offer URL loading), a `file:///etc/passwd` or `file:///path/to/wallet/keys` URI could cause the Electron renderer to read local files. The offer data would then be parsed and potentially displayed or logged. This falls under **High: Unsafe handling of external URLs... that produces direct asset loss, approval hijack, persistent lockout, or other concrete wallet security impact** — specifically SSRF or local file read in the Electron context.

Auto-acceptance of a crafted offer is **not** automatic; the user must click Accept, so the Critical "unauthorized spend" path requires social engineering on top.

### Likelihood Explanation
Any node operator can send a blockchain notification to any puzzle hash for a small XCH fee. The victim's GUI will process it automatically upon wallet sync. The missing validation is a single-line omission with no compensating control visible in the traced path.

### Recommendation
In `prepareNotifications`, validate `url` before storing it:
```ts
const { u: url, ph: puzzleHash } = data;
if (!url || !/^https?:\/\//i.test(url)) {
  throw new Error('Invalid or missing offer URL scheme');
}
```
Additionally, `OffersProvider.getOffer()` should enforce `https://`-only fetches at the fetch call site.

### Proof of Concept
1. Encode `{"t":1,"d":{"u":"file:///etc/passwd","ph":"<victim_puzzle_hash>"}}` as hex.
2. Call `send_notification` RPC targeting the victim's puzzle hash with that hex payload.
3. Victim's GUI receives the notification, `prepareNotifications` sets `offerURL = "file:///etc/passwd"`.
4. `NotificationOffer` renders and calls `useOffer("file:///etc/passwd")`.
5. `OffersProvider.getOffer()` fetches the URI — if using `fetch()` in Electron renderer without `webSecurity` restrictions, the local file contents are returned and parsed as offer data.

**Unit test plan:** Call `prepareNotifications` with a mocked blockchain notification where `d.u = "file:///etc/passwd"`. Assert the resulting `Notification` is `null` (rejected) or that `offerURL` is sanitized/absent.
