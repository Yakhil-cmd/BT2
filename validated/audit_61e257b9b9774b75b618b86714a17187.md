### Title
Permanent FLOW Token Loss When `CadenceOwnedAccount` Resource Is Destroyed With Non-Zero EVM Balance — (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` function to guard against or handle a non-zero EVM balance before the resource is destroyed. If a COA resource is destroyed while FLOW tokens remain at its EVM address, those tokens become permanently inaccessible. There is no sweep or recovery mechanism anywhere in the EVM contract.

---

### Finding Description

The `CadenceOwnedAccount` resource is the only Cadence-side controller of a COA's EVM address. The EVM balance is stored in the EVM state, not inside the Cadence resource itself. The resource's EVM address is deterministically derived from its Cadence UUID at creation time:

```cadence
let addr = InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)
``` [1](#0-0) 

Once the resource is destroyed, the UUID is gone. No new COA can be created with the same UUID (Cadence UUIDs are auto-incremented and unique), so the EVM address is permanently orphaned.

The `CadenceOwnedAccount` resource definition spans lines 518–803 of `contract.cdc` and contains `deposit`, `withdraw`, `call`, `deploy`, `withdrawNFT`, `withdrawTokens`, etc., but **no `destroy` function**: [2](#0-1) 

Because there is no `destroy` function, Cadence permits the resource to be destroyed without any balance check or pre-withdrawal step. The EVM balance at the orphaned address is then permanently inaccessible because:

1. The COA EVM address has **no private key** — it is not an EOA.
2. There is **no Cadence resource** left to call `withdraw()` on.
3. There is **no admin or protocol-level sweep function** in the EVM contract that can withdraw from an arbitrary EVM address.

The `withdraw` function itself requires the caller to hold the `CadenceOwnedAccount` resource (entitlement `Owner | Withdraw`): [3](#0-2) 

Additionally, FLOW tokens can arrive at a COA's EVM address via EVM-native transfers (e.g., another EVM account sending FLOW directly to the COA's address) without the COA owner initiating or even being aware of the deposit. This mirrors the PayrollManager scenario where tokens arrive via airdrops or accidental returns. The `EVMAddress.deposit()` function is `access(all)`, meaning any Cadence transaction can deposit FLOW to any EVM address: [4](#0-3) 

The Go-side deposit implementation explicitly keeps the vault alive in the EVM storage account to back the EVM balance, confirming the tokens are real and tracked: [5](#0-4) 

---

### Impact Explanation

**Cross-VM asset loss.** Any FLOW tokens held at a COA's EVM address at the time the COA resource is destroyed are permanently burned from the perspective of the Cadence/EVM bridge. The amount is unbounded — a user could have deposited hundreds or thousands of FLOW into their COA before destroying it. There is no recovery path: no admin sweep, no protocol-level rescue, no re-derivation of the address.

---

### Likelihood Explanation

**Moderate.** Several realistic paths lead to this loss:

- A user destroys their COA resource (e.g., during account cleanup or contract migration) without first checking or withdrawing the EVM balance.
- A Cadence contract that manages COAs on behalf of users destroys a COA as part of its logic without a balance check.
- A COA receives FLOW via an EVM-native transfer (from another EVM account) that the owner did not initiate and is unaware of, then the owner destroys the COA believing it is empty.

The `access(all)` `deposit` on `EVMAddress` means any third party can silently fund a COA's EVM address at any time, making the "owner believes it is empty" scenario realistic.

---

### Recommendation

Add a `destroy` function to the `CadenceOwnedAccount` resource that enforces one of the following:

1. **Panic if balance is non-zero** — prevents accidental destruction of a funded COA:
   ```cadence
   destroy() {
       pre {
           self.balance().isZero():
               "EVM.CadenceOwnedAccount: cannot destroy a COA with a non-zero EVM balance"
       }
   }
   ```

2. **Withdraw and return balance before destruction** — requires the caller to supply a receiver vault, analogous to a `sweep()` function.

Either approach closes the gap between the Cadence resource lifecycle and the EVM balance lifecycle.

---

### Proof of Concept

Any unprivileged Cadence transaction can trigger this loss:

```cadence
import EVM from <EVM_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>
import FungibleToken from <FUNGIBLE_TOKEN_ADDRESS>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Fund a COA with 10.0 FLOW
        let admin = account.storage
            .borrow<&FlowToken.Administrator>(from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 10.0)
        let vault <- minter.mintTokens(amount: 10.0)
        destroy minter

        let coa <- EVM.createCadenceOwnedAccount()
        // EVM address is now live; UUID is emitted in CadenceOwnedAccountCreated event
        coa.deposit(from: <-vault)

        // A third party could also send FLOW to coa.address() via EVM-native transfer
        // at any time, without the owner's knowledge.

        // Owner destroys the COA believing cleanup is safe.
        // No destroy() function exists to warn or prevent this.
        destroy coa

        // 10.0 FLOW (plus any third-party deposits) are now permanently inaccessible:
        // - No private key for the EVM address
        // - No Cadence resource to call withdraw()
        // - No admin sweep function in the EVM contract
    }
}
```

The root cause is confirmed at `fvm/evm/stdlib/contract.cdc` lines 518–803: the `CadenceOwnedAccount` resource has no `destroy` function, and the EVM balance is entirely decoupled from the Cadence resource lifecycle. [6](#0-5)
