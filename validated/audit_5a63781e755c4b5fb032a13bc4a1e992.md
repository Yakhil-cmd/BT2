### Title
Opaque `type: 'json'` Spend Bundle Accepted Without Structural Validation in WalletConnect Confirmation Dialog — (File: `packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/commands/parseDappParams.ts`)

### Summary
The WalletConnect command pipeline accepts `chia_pushTx` (`spend_bundle`) and `chia_pushTransactions` (`transactions`) as opaque `type: 'json'` blobs. `parseDappParams` enforces a top-level key allowlist but performs zero structural validation on the nested JSON content. The confirmation dialog renders these params as raw JSON without parsing amounts or destination addresses. A malicious dapp that has been granted either permission can craft a spend bundle that transfers user funds to an attacker-controlled address; the user sees only an unreadable JSON blob and has no way to identify the actual financial impact before approving.

### Finding Description

**Step 1 — Command schema declares `spend_bundle` and `transactions` as unvalidated `type: 'json'`**

`chia_full_node.push_tx` declares its sole param as `type: 'json'`: [1](#0-0) 

`chia_wallet.push_transactions` declares `transactions` as `type: 'json'`: [2](#0-1) 

**Step 2 — `parseDappParams` enforces only the top-level key allowlist; `type: 'json'` values pass through untouched**

The validation loop in `parseDappParams` only coerces `string`, `number`, `bool`, and `bigint` typed params. When `type === 'json'`, no branch executes — the value is forwarded as-is: [3](#0-2) 

A dapp can therefore supply any arbitrarily-structured spend bundle — including coin spends targeting attacker-controlled puzzle hashes — and `parseDappParams` will not reject it.

**Step 3 — The confirmation dialog renders raw JSON with no amount/address parsing**

`parseCommandDisplay` provides a human-readable wallet-delta view only for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. For `chia_pushTx` and `chia_pushTransactions` it returns `undefined`: [4](#0-3) 

`humanizeDappCommand` then falls back to `humanizeParams`, which renders the raw JSON value as a string in the dialog: [5](#0-4) 

The `Confirm` dialog renders this as `whitespace-pre-wrap` text — a raw JSON blob the user cannot meaningfully interpret: [6](#0-5) 

**Step 4 — Parsed params are forwarded directly to the daemon**

After user confirmation, `parsedParams` (including the unvalidated `spend_bundle` / `transactions` blob) is sent directly to the daemon via `sendCommand`: [7](#0-6) 

### Impact Explanation

A malicious dapp granted `chia_pushTx` or `chia_pushTransactions` permission can submit a spend bundle that spends user-owned coins to an attacker address. The user is shown a confirmation dialog containing only a raw JSON blob — no coin amounts, no destination addresses, no wallet delta. The user cannot distinguish a legitimate transaction from a fund-draining one. Upon approval, the spend bundle is submitted to the full node or wallet daemon and executed on-chain, resulting in irreversible loss of XCH or CAT funds.

`chia_pushTx` is particularly severe because it routes to `chia_full_node` (bypassing wallet-level coin management), meaning the dapp can reference any coin the user owns directly by coin ID.

### Likelihood Explanation

The attacker must be a dapp that the user has connected via WalletConnect and explicitly granted `chia_pushTx` or `chia_pushTransactions` permission. This is a realistic scenario: a dapp can request these permissions during pairing (they appear in the permission list), and a user who trusts the dapp's stated purpose (e.g., "push a pre-signed transaction") would grant them. Once granted, the dapp can send any spend bundle at any time, including after a bypass ("always allow") has been set, requiring zero further user interaction.

### Recommendation

1. **Structural validation**: For `type: 'json'` params on transaction-pushing commands, validate that the value conforms to the expected schema (e.g., `spend_bundle` must be an object with `coin_spends` array and `aggregated_signature` string; each `coin_spend` must have `coin`, `puzzle_reveal`, `solution`). Reject malformed inputs before the confirmation dialog is shown.

2. **Human-readable confirmation display**: Implement a `parseCommandDisplay` handler for `chia_wallet.push_transactions` and `chia_full_node.push_tx` that decodes the spend bundle and displays the affected coin amounts and destination puzzle hashes (or bech32m addresses where resolvable) in the wallet-delta section — analogous to the existing `take_offer` display handler.

3. **Restrict `chia_pushTx` scope**: Consider whether `chia_full_node.push_tx` should be available to WalletConnect dapps at all, given that it bypasses all wallet-level coin tracking and validation.

### Proof of Concept

1. Attacker publishes a dapp and requests `chia_pushTx` (or `chia_pushTransactions`) permission during WalletConnect pairing. The user grants it, believing the dapp has a legitimate use (e.g., "submit a pre-built transaction").

2. The dapp constructs a spend bundle that spends a user-owned coin (coin ID obtained via `chia_getCoinRecordsByNames`, also a grantable permission) to an attacker-controlled puzzle hash.

3. The dapp sends a `session_request` with `method: "chia_pushTx"` and `params: { spend_bundle: { coin_spends: [{ coin: { parent_coin_info: "...", puzzle_hash: "...", amount: 1000000000000 }, puzzle_reveal: "...", solution: "..." }], aggregated_signature: "..." } }`.

4. `parseDappParams` validates only that `spend_bundle` is a present top-level key; the nested structure is forwarded untouched.

5. The confirmation dialog shows: **Spend Bundle**: `{"coin_spends":[{"coin":{"parent_coin_info":"0x...","puzzle_hash":"0x...","amount":1000000000000},...}],...}` — no human-readable amount or destination.

6. The user clicks "Push", the spend bundle is submitted to the full node, and 1 XCH is transferred to the attacker's address irreversibly.

    const { name, type, isOptional } = param;

    const value = nextParams[name];

    // verify if all isOptional !== true params are provided
    if (isOptional !== true && value === undefined) {
      throw new Error(`param is required: ${name}`);
    }

    // parse value to the correct types
    if (value !== undefined) {
      if (type === 'string') {
        nextParams[name] = String(value);
      } else if (type === 'number') {
        nextParams[name] = Number(value);
        if (Number.isNaN(nextParams[name])) {
          throw new Error(`Invalid number value for argument ${name}. Value: ${value}`);
        }
      } else if (type === 'bool') {
        if (typeof value !== 'boolean') {
          throw new Error(`Invalid boolean value for argument ${name}. Value: ${value}`);
        }
        nextParams[name] = value;
      } else if (type === 'bigint') {
        if (typeof value !== 'string' && typeof value !== 'number' && typeof value !== 'bigint') {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        if (typeof value === 'number' && !Number.isSafeInteger(value)) {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        if (typeof value === 'string') {
          const trimmed = value.trim();
          if (trimmed === '') {
            throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
          }
        }

        const bigintValue = BigInt(value);
        if (bigintValue.toString() !== value.toString()) {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        nextParams[name] = bigintValue;
      }
    }

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
}

import { getDappCommandSchema } from './getDappCommandSchema';
import { humanizeParams } from './humanizeParams';

export async function humanizeDappCommand(dappCommand: string, data: Record<string, unknown>, networkPrefix?: string) {
  const dappCommandSchema = getDappCommandSchema(dappCommand);

  const rows = await humanizeParams(dappCommandSchema.params, data, networkPrefix);

  return {
    destructive: dappCommandSchema.destructive === true,
    title: dappCommandSchema.title(),
    message: dappCommandSchema.message(),
    confirmLabel: dappCommandSchema.confirmLabel(),
    rows,
  };
}
