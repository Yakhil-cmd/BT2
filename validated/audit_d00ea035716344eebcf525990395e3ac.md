### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Cannot Be Disabled and `CanTurnBridge` Permission Check Is Absent - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` gRPC message handler in `x/cronos/keeper/msg_server.go` is implemented as a bare `return nil, nil` stub. It performs no authorization check and makes no state change. Any unprivileged user can call `MsgTurnBridge` and receive a success response, while the bridge state is never modified. The intended gravity-bridge circuit breaker is permanently non-functional.

### Finding Description
The `TurnBridge` handler at lines 84–87 of `x/cronos/keeper/msg_server.go` reads:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

This stub:
1. **Performs no authorization check.** Compare with `UpdateTokenMapping`, which guards execution with `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` before doing any work. [2](#0-1) 

2. **Makes no state change.** No bridge-enabled flag is written to the KV store; the gravity bridge is unaffected regardless of the `Enable` field in the message.

3. **Returns success to every caller.** `nil, nil` is a successful gRPC response, so the transaction is accepted on-chain and the caller's account sequence is incremented.

The `CanTurnBridge` permission constant is defined in the permissions keeper and ADR-009 explicitly states that `MsgTurnBridge` must require this permission: [3](#0-2) 

The integration test `test_gravity_turn_bridge` confirms the intended behavior: a "community" (non-admin) caller is expected to receive a non-zero error code, and a "validator" (admin) caller is expected to succeed and actually disable the bridge. The current no-op implementation violates both expectations. [4](#0-3) 

The `MsgTurnBridge` message is a fully registered gRPC endpoint: [5](#0-4) 

### Impact Explanation
Two distinct High-severity impacts arise:

**1. Bypass of `CanTurnBridge` permission (High — auth bypass):** Any unprivileged address can submit `MsgTurnBridge` and receive a success response. The `CanTurnBridge` permission system defined in `x/cronos/keeper/permissions.go` is completely bypassed because the handler never calls `HasPermission`. This directly satisfies the allowed impact: *"Bypass of Cronos admin, governance authority, permission … authorization checks."*

**2. Permanent inability to disable the gravity bridge (High — long-lived DoS of bridge protection):** The gravity bridge circuit breaker is permanently non-functional. Even the admin or a permissioned address cannot disable the bridge in an emergency. If the gravity bridge is exploited (e.g., a batch-submission attack or oracle compromise), the intended `MsgTurnBridge(false)` call silently succeeds but leaves the bridge running, allowing continued unauthorized asset flows. This satisfies: *"Permanent or long-lived inability for honest users or validators to process valid … bridge/conversion flows … under normal network assumptions."*

### Likelihood Explanation
The entry path is fully reachable by any unprivileged user with a funded account. `MsgTurnBridge` is a standard Cosmos SDK transaction, broadcast via CLI (`CmdTurnBridge`) or gRPC. No special privileges, leaked keys, or cryptographic breaks are required. The no-op is present in the production keeper code and is directly contradicted by the integration test expectations, confirming it is an unimplemented stub shipped as production code. [6](#0-5) 

### Recommendation
Implement `TurnBridge` with:
1. A permission check: `if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) { return nil, sdkerrors.ErrUnauthorized }`.
2. A state write that persists the `Enable` flag (e.g., a dedicated KV key under the cronos store).
3. Enforcement of that flag in the gravity bridge hooks and/or the `EndBlocker` so that outbound batches are not created when the bridge is disabled.

### Proof of Concept
1. Unprivileged user `U` submits `MsgTurnBridge{Sender: U, Enable: false}`.
2. The handler executes `return nil, nil` — the transaction is accepted on-chain with code 0.
3. No bridge-enabled flag is written; the gravity bridge continues processing outbound batches.
4. Admin `A` submits `MsgTurnBridge{Sender: A, Enable: false}` during an emergency — identical result: code 0, bridge still running.
5. The gravity bridge cannot be stopped through the intended mechanism, and any ongoing exploit continues unimpeded. [1](#0-0) [3](#0-2)

// CmdTurnBridge returns a CLI command handler for enable or disable the bridge
func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{
		Use:   "turn-bridge [true/false]",
		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}
