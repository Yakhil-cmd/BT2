### Title
`verify` Accepts Forged Signatures When `digest = n` Due to Missing Zero-Scalar Guard — (`crypto/src/secp256r1/verify.rs`)

---

### Summary

`Scalar::reduce_be_bytes` silently maps `digest = n` (the P-256 curve order) to `Scalar::ZERO`. The `verify` function has no guard against `z = 0`, so the ECDSA equation degenerates to `u2 * pk` with no generator term. An unprivileged attacker can construct a valid-looking `(r, s)` for **any** target public key without knowing the corresponding private key, causing `p256verify` to return `1` for a signature that was never produced by the key holder.

---

### Finding Description

**Step 1 — `reduce_be_bytes` maps `n → 0`** [1](#0-0) 

`subtract_modulus` computes `overflowing_sub(self, MODULUS)`. When `self == MODULUS` (i.e., `digest = n`), the subtraction yields `(0, borrow=false)`, so `self` becomes `[0,0,0,0]` — `Scalar::ZERO`. No borrow means the add-back branch is skipped.

**Step 2 — `verify` does not guard against `z = 0`** [2](#0-1) 

After `let z = Scalar::reduce_be_bytes(digest)`, there is no `if z.is_zero() { return Ok(false); }` check. Execution continues with `u1 = 0 * s_inv = 0` and `u2 = r * s_inv`.

**Step 3 — `ecmult` skips the generator term when `ng = 0`** [3](#0-2) 

`ng = u1 = 0` → `wnaf_ng = Wnaf::default()` (zero bits). The loop only accumulates `u2 * pk`. The result is `u2 * pk` with no `G` contribution.

**Step 4 — Forgery construction for any target public key `P`**

Choose `t = 1`:
- `Q = t * P = P` (public scalar multiplication, no private key needed)
- `r = x(P) mod n`
- `s = r * t⁻¹ mod n = r`

Submit `(digest=n, r, s, x(P), y(P))`.

Verification trace:
- `z = 0`
- `u2 = r * s⁻¹ = r * r⁻¹ = 1`
- `ecmult(P, 1, 0) = 1 * P = P`
- `recovered = x(P) mod n = r` → **match → returns `true`**

The attacker never needed the private key of `P`.

---

### Impact Explanation

Any smart contract using the `p256verify` precompile (address `0x0100`, EIP-7951) for authentication — passkey/WebAuthn wallets, P-256-based account abstraction, multi-sig schemes — can be bypassed. The attacker submits `digest = n` with a trivially computed `(r, s)` for the victim's known public key. The precompile returns `1`, and the contract grants access. [4](#0-3) 

---

### Likelihood Explanation

- Requires no privileged access, no leaked key, no oracle manipulation.
- The attack is deterministic and requires only public information (the target's public key).
- The construction is trivial: `r = x(P) mod n`, `s = r`, `digest = n`.
- The fuzzer differential (`precompiles_p256.rs`) compares against revm; if revm's `p256_verify` also reduces `n → 0` and allows `z=0`, the differential test would not catch this. [5](#0-4) 

---

### Recommendation

Add an explicit zero-check on the digest scalar immediately after reduction in `verify`:

```rust
// crypto/src/secp256r1/verify.rs, after line 23
let z = Scalar::reduce_be_bytes(digest);
if z.is_zero() {
    return Ok(false);
}
```

This matches the intent of the signature-component guards already present in `Signature::from_scalars`: [6](#0-5) 

---

### Proof of Concept

```
P-256 curve order n (big-endian, 32 bytes):
  digest = ffffffff00000000ffffffffffffffffbce6faada7179e84f3b9cac2fc632551

Pick any valid P-256 public key (x, y), e.g. the generator G:
  x = 6b17d1f2e12c4247f8bce6e563a440f277037d812deb33a0f4a13945d898c29
  y = 4fe342e2fe1a7f9b8ee7eb4a7c0f9e162bce33576b315ececbb6406837bf51f5

Compute r = x mod n = x (since x < n for the generator):
  r = 6b17d1f2e12c4247f8bce6e563a440f277037d812deb33a0f4a13945d898c29

Set s = r:
  s = 6b17d1f2e12c4247f8bce6e563a440f277037d812deb33a0f4a13945d898c29

Call p256verify(digest=n, r, s, x, y) → returns 1 (VALID)
```

The 160-byte input to the precompile is `digest || r || s || x || y`. No private key knowledge required.
