### Title
ETH Permanently Locked in `tick()` When No Mint Requests Are Pending — (`File: L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
The `tick()` function in `RewardSupplier.sol` is declared `payable` and uses `msg.value` to fund two L1→L2 messages. However, the ETH-spending logic is gated inside an `if (amountToMint > 0)` block. When there are no pending L2 mint requests, the function returns silently without spending or refunding `msg.value`, permanently locking any ETH sent by the caller.

### Finding Description
`tick()` is the public entry point for processing L2 reward mint requests on L1. It is `payable` because it must forward ETH as fees to two downstream calls:

1. `bridge().depositWithMessage{value: msgFee}(...)` — StarkGate L1→L2 bridge fee
2. `messagingContract().sendMessageToL2{value: msgFee}(...)` — Starknet Core messaging fee

Both calls are wrapped inside `if (amountToMint > 0)`. When `requiredMinting()` returns zero (no pending L2 messages), the entire ETH-spending block is skipped and the function returns. Any `msg.value` sent by the caller is absorbed by the contract with no refund path. [1](#0-0) 

There is no `receive()`, `fallback()`, or admin ETH-withdrawal function anywhere in `RewardSupplier.sol`, and none is introduced by the inherited contracts (`RewardSupplierStorage`, `Identity`, `ProxySupportImpl`) visible in the repository. [2](#0-1) 

The root cause is structurally analogous to the reported `mekHoneyJarWithETH` bug: a function that is `payable` and uses `msg.value`, but whose ETH-handling path is not always executed, causing ETH to be irrecoverably consumed by the contract.

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users.**

Any caller who sends ETH to `tick()` when `amountToMint == 0` permanently loses that ETH. Because `tick()` is a public, permissionless function intended to be called by keepers or bots on a schedule, callers have no reliable on-chain guarantee that pending requests exist at the moment of their transaction. A front-runner or natural race condition (another keeper processes the last pending message in the same block) can cause a well-intentioned caller to lose their ETH silently.

### Likelihood Explanation
**Medium.** `tick()` is `external` with no access control, designed to be called by any keeper. Keepers routinely send ETH to cover L1→L2 message fees. A race condition where two keepers submit `tick()` in the same block, or where the last pending message is consumed between the keeper's off-chain check and on-chain execution, is a realistic and repeatable scenario. [3](#0-2) 

### Recommendation
Refund unused ETH at the end of `tick()` when `amountToMint == 0`:

```diff
 function tick() external payable {
     (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

     if (amountToMint > 0) {
         // ... existing logic ...
+    } else if (msg.value > 0) {
+        (bool ok, ) = msg.sender.call{value: msg.value}("");
+        require(ok, "ETH_REFUND_FAILED");
     }
 }
```

Alternatively, remove `payable` from `tick()` and require callers to pre-fund the contract via a separate deposit, or have the contract hold a persistent ETH balance for fees.

### Proof of Concept
1. Keeper A calls `requiredMinting()` off-chain and sees 1 pending message.
2. Keeper A submits `tick{value: 0.01 ether}()`.
3. Keeper B's `tick()` (with no ETH) is mined first in the same block, consuming the last pending message.
4. Keeper A's transaction executes: `requiredMinting()` now returns `(0, 0)`, the `if` block is skipped, and `0.01 ether` is permanently locked in the `RewardSupplier` contract.
5. No function exists to recover the locked ETH. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
