### Title
USDT Fee-on-Transfer Breaks Swap and Liquidity Callback Balance Checks, Causing Permanent DoS for USDT Pools — (File: `metric-core/contracts/MetricOmmPool.sol`)

### Summary
`MetricOmmPool.swap()` and `addLiquidity()` both use a balance-before/after pattern to verify that the caller's callback paid the required input amount. If USDT (explicitly in scope) is the input token and its built-in fee is activated (> 0), the callback sends the full computed delta but the pool receives `delta - fee`. The balance check then reverts unconditionally, permanently breaking all swaps and liquidity additions for any USDT pool.

### Finding Description

**Swap path** — `MetricOmmPool.swap()`:

After computing `amount0Delta` (the input the pool requires), the pool:
1. Sends output tokens to `recipient`
2. Calls `metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData)` — telling the caller to pay `amount0Delta`
3. Checks that its balance rose by exactly that amount [1](#0-0) 

```solidity
uint256 balance0Before = balance0();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
    revert IncorrectDelta();
}
```

The router's `_justPayCallback` calls `pay(token, payer, pool, amount0Delta)`, which resolves to `safeTransferFrom(payer, pool, amount0Delta)`. [2](#0-1) 

If `token0` is USDT with fee `f > 0`, the pool receives `amount0Delta - f`. The check becomes:

```
balance0Before + amount0Delta  >  balance0Before + amount0Delta - f
```

This is always `true` when `f > 0`, so every swap reverts with `IncorrectDelta`.

**Liquidity path** — `LiquidityLib.addLiquidity()`: [3](#0-2) 

```solidity
IMetricOmmModifyLiquidityCallback(msg.sender)
  .metricOmmModifyLiquidityCallback(amount0Added, amount1Added, callbackData);
if (amount0Added > 0 && balance0Before + amount0Added > IERC20(ctx.token0).balanceOf(address(this))) {
  revert IMetricOmmPoolActions.InsufficientTokenBalance();
}
```

Identical invariant: if USDT is `token0`, the callback sends `amount0Added` but the pool receives `amount0Added - f`, triggering `InsufficientTokenBalance` on every `addLiquidity` call.

**Fee collection silent loss** — `MetricOmmPool.collectFees()`: [4](#0-3) 

When fees are swept to `adminFeeDestination` and `FACTORY`, if USDT is a pool token the recipients silently receive `totalFee - f` instead of `totalFee`. No revert occurs here, but protocol and admin fee revenue is permanently lost on every collection.

### Impact Explanation

- **Swap DoS**: Every `swap()` call on a USDT-input pool reverts with `IncorrectDelta`. The pool is completely unusable for trading.
- **Liquidity DoS**: Every `addLiquidity()` call reverts with `InsufficientTokenBalance`. LPs cannot enter the pool.
- **Silent fee loss**: `collectFees()` silently under-delivers protocol and admin fees proportional to the USDT transfer fee rate on every collection cycle.

This satisfies "Broken core pool functionality causing loss of funds or unusable withdraw/swap/liquidity flows."

### Likelihood Explanation

USDT's fee mechanism exists in its deployed contract and is currently set to 0. Tether has the unilateral ability to activate it. The contest scope explicitly carves out USDC/USDT from the "non-standard ERC20 behavior" exclusion, making this scenario in-scope. Any USDT pool created via the permissionless `createPool` path is affected the moment the fee is non-zero.

### Recommendation

Replace the fixed-delta balance check with an actual-received check in both `swap()` and `LiquidityLib`:

```solidity
// swap path
uint256 balance0Before = balance0();
IMetricOmmSwapCallback(msg.sender).metricOmmSwapCallback(amount0Delta, amount1Delta, callbackData);
if (amount0Delta > 0 && balance0() < balance0Before + uint256(amount0Delta)) {
    revert IncorrectDelta();
}
```

For `collectFees`, record the balance before and after each transfer and account for the shortfall, or accept that fee-on-transfer tokens will deliver less and document the limitation.

### Proof of Concept

1. Deploy a USDT/WETH pool via `MetricOmmPoolFactory.createPool()` with USDT as `token0`.
2. Activate USDT's fee (e.g., 1 basis point).
3. Call `MetricOmmSimpleRouter.exactInputSingle()` for a USDT→WETH swap.
4. Router's `metricOmmSwapCallback` calls `safeTransferFrom(user, pool, amount0Delta)`.
5. USDT deducts its fee; pool receives `amount0Delta - fee`.
6. Pool evaluates `balance0Before + amount0Delta > balance0()` → `true` → reverts `IncorrectDelta`.
7. Repeat for `addLiquidity` → reverts `InsufficientTokenBalance`.
8. Both core operations are permanently broken for this pool. [5](#0-4) [6](#0-5) [7](#0-6)
