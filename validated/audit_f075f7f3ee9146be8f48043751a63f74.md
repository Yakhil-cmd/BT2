### Title
Bank Precompile `transfer` and `burn` Allow Any EVM Contract to Drain or Destroy Any Holder's `evm/` Native Tokens Without Authorization - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The bank precompile's `transfer` and `burn` methods accept an arbitrary address argument as the token source but perform no check that this address equals `contract.Caller()`. Any deployed EVM contract can therefore move or destroy `evm/<its_own_address>` native bank tokens held by any victim address without the victim's consent.

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes four methods. The denom for all state-mutating operations is always `EVMDenom(contract.Caller())` — i.e., `evm/<calling_contract_address>`. This correctly ties the denomination to the calling contract. However, the authorization model breaks for `transfer` and `burn`:

**`transfer` (lines 167–200):**

```go
sender := args[0].(common.Address)   // arbitrary, from calldata
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // evm/<calling_contract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`from` is taken directly from ABI calldata. There is no assertion that `sender == contract.Caller()`. Any contract can pass an arbitrary victim address as `sender` and drain that victim's `evm/<contract_address>` balance to any `recipient`. [1](#0-0) 

**`burn` (lines 113–156):**

```go
recipient := args[0].(common.Address)  // arbitrary, from calldata — the address to burn FROM
...
addr := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

Again, `addr` is taken from calldata with no check against `contract.Caller()`. Any contract can burn tokens from any holder of its denomination. [2](#0-1) 

Contrast this with the `utils.go` `exec` helper used by other precompiles (ICA, relayer), which explicitly enforces `caller == signer`:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: expected %s, got %s", ...)
}
``` [3](#0-2) 

The bank precompile has no equivalent guard.

### Impact Explanation

`evm/<contract_address>` tokens are real Cosmos SDK bank module assets. They can be held in any account, transferred via IBC, and used in DeFi flows. A malicious contract can:

1. **Steal**: call `transfer(victim, attacker, balance)` to move all of a victim's `evm/<malicious_contract>` tokens to the attacker with zero victim interaction.
2. **Destroy**: call `burn(victim, balance)` to permanently destroy a victim's holdings.

This is an unauthorized transfer and burn of precompile-controlled assets — Critical impact under the allowed scope. [4](#0-3) 

### Likelihood Explanation

The attacker needs only to deploy a contract (unprivileged) and have victims hold `evm/<attacker_contract>` tokens. This is trivially achievable: the attacker mints tokens to victims via the `mint` method (which is also unrestricted — any contract can mint its own denomination to any address), then immediately drains them. No admin key, governance vote, or leaked secret is required. [5](#0-4) 

### Recommendation

In the `transfer` case, verify that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

In the `burn` case, verify that the address to burn from equals `contract.Caller()`:

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn: can only burn from the calling contract's own account")
}
```

This mirrors the authorization model already enforced in `utils.go` for other precompiles and matches the expected invariant that a contract may only move tokens it itself holds, not tokens held by arbitrary third parties. [6](#0-5) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
    function burn(address addr, uint256 amount) external payable returns (bool);
}

contract BankDrainer {
    IBankPrecompile constant BANK = IBankPrecompile(address(100)); // 0x64

    // Step 1: attacker mints evm/<this> tokens to victim as "airdrop"
    function airdrop(address victim, uint256 amount) external {
        BANK.mint(victim, amount);
    }

    // Step 2: attacker drains victim's evm/<this> balance to themselves
    function drain(address victim, uint256 amount) external {
        // sender = victim, recipient = msg.sender (attacker EOA)
        // No authorization check in precompile — succeeds unconditionally
        BANK.transfer(victim, msg.sender, amount);
    }

    // Alternative: destroy victim's tokens
    function destroy(address victim, uint256 amount) external {
        BANK.burn(victim, amount);
    }
}
```

1. Attacker deploys `BankDrainer` at address `0xDEAD`.
2. Attacker calls `airdrop(victim, 1000)` → victim now holds `1000 evm/0xDEAD` in the Cosmos bank module.
3. Attacker calls `drain(victim, 1000)` → bank precompile executes `SendCoins(victim → attacker, 1000 evm/0xDEAD)` with no consent from victim.
4. Victim's balance is zero; attacker holds 1000 `evm/0xDEAD`.

The same path works for `burn`: step 3 becomes `destroy(victim, 1000)`, permanently destroying the victim's tokens. [7](#0-6)
