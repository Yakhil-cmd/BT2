### Title
User-Controlled `CREATE2` Salt in `createPool` Enables Address Collision Attack to Drain LP Funds - (File: `metric-core/contracts/MetricOmmPoolDeployer.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts a fully attacker-controlled `bytes32 salt` and passes it unmodified to `MetricOmmPoolDeployer.deploy`, which uses it directly in a `CREATE2` deployment of `MetricOmmPool`. A meet-in-the-middle brute-force attack can find a collision between a pre-computed pool address and an attacker-controlled contract address, allowing the attacker to pre-set unlimited token approvals on the future pool address before the pool is deployed, then drain all LP principal after liquidity is added.

---

### Finding Description

`PoolParameters.salt` is a raw `bytes32` field with no validation in `_validatePoolParameters`:

```solidity
// metric-core/contracts/types/FactoryOperation.sol
struct PoolParameters {
  ...
  bytes32 salt;
}
```

`createPool` forwards it verbatim to the deployer:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol L183
salt: params.salt,
```

The deployer uses it directly in `CREATE2`:

```solidity
// metric-core/contracts/MetricOmmPoolDeployer.sol L62
pool = address(
  new MetricOmmPool{salt: params.salt}(...)
);
```

The resulting pool address is:

```
keccak256(0xff ++ deployer_address ++ params.salt ++ keccak256(MetricOmmPool_initcode + args))
```

The attacker controls `params.salt` (full 256 bits) and also controls the constructor arguments (`token0`, `token1`, `priceProvider`, etc.), giving them significant influence over the init-code hash as well. This is strictly more exploitable than the Arcadia analog, where the salt was at least mixed with `tx.origin` via `keccak256(abi.encodePacked(salt, tx.origin))`.

**Attack steps:**

**Tx 1 (pre-collision setup):**
- Brute-force `params.salt` values to enumerate candidate pool addresses (one side of the meet-in-the-middle).
- Separately brute-force a CREATE2 salt for an attacker-controlled deployer to find a collision with any address in the candidate set.
- Deploy the attack contract at `0xCOLLIDED`, call `token0.approve(attacker, type(uint256).max)` and `token1.approve(attacker, type(uint256).max)`, then `selfdestruct` — all in one transaction (valid post-Dencun per EIP-6780 for same-tx creation).

**Tx 2 (drain):**
- Call `createPool` with the colliding `params.salt`. The pool deploys at `0xCOLLIDED`.
- LPs add liquidity; `token0` and `token1` are transferred into `0xCOLLIDED`.
- Attacker calls `token0.transferFrom(0xCOLLIDED, attacker, balance)` and `token1.transferFrom(...)` using the pre-set approvals.
- All LP principal is drained.

---

### Impact Explanation

Complete loss of all LP principal (token0 and token1) held in any targeted pool. The pool's token balances are fully accessible to the attacker via the pre-set ERC-20 approvals. Every pool deployed by the factory is independently vulnerable; the attacker can target pools with the highest TVL.

---

### Likelihood Explanation

`createPool` is permissionless — any caller can supply any `bytes32 salt`. The salt is not mixed with `msg.sender`, `tx.origin`, block data, or any other entropy. The meet-in-the-middle attack requires approximately 2^80 hash operations per side. As documented in EIP-3607 and the referenced Kyber Swap audit, the Bitcoin network alone achieves ~6×10^20 hashes/second, reaching 2^80 in ~33 minutes. A fraction of that hashrate (available commercially for millions of dollars) suffices. The cost is easily offset by the TVL of a single high-value pool.

---

### Recommendation

Remove the user-supplied `salt` from `PoolParameters` entirely and derive the salt internally from data the attacker cannot freely choose:

```solidity
bytes32 salt = keccak256(abi.encodePacked(
    params.token0,
    params.token1,
    params.priceProvider,
    nextPoolIdx,
    block.chainid
));
```

Alternatively, replace `CREATE2` with vanilla `CREATE` (opcode-level `new MetricOmmPool(...)`), whose address is determined by the deployer's address and its internal nonce — neither of which the caller controls.

---

### Proof of Concept

The vulnerability is structural and follows the same two-property proof as the Arcadia PoC:

1. **A contract can be deployed on top of an address that previously held a self-destructed contract** (EVM property, unchanged post-Dencun for same-tx selfdestruct).
2. **An ERC-20 approval set by a contract before it self-destructs persists at that address** — the token contract's storage records the allowance keyed by `(owner=0xCOLLIDED, spender=attacker)`, which survives the selfdestruct and remains valid when the pool is later deployed at the same address.

The exact salt that produces the collision is found off-chain. Once found, the two-transaction sequence above drains the pool atomically. No privileged access is required; `createPool` is fully permissionless. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
