### Title
CadenceOwnedAccount Resource Destruction Without Balance Check Causes Permanent Freezing of EVM FLOW Tokens - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

The `EVM.CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` hook that checks for a non-zero EVM balance before allowing destruction. When a user destroys their COA Cadence resource, any FLOW tokens held in the associated EVM address become permanently inaccessible. This is a direct analog to the reported Voter.sol pattern: a position-closing action (resource destruction) does not automatically recover held assets, and after destruction the only recovery path (the `withdraw` entitlement function) is permanently severed.

---

### Finding Description

A `CadenceOwnedAccount` (COA) is a Cadence resource that exclusively controls a deterministic EVM address. The EVM address is derived from the resource's `uuid` at creation time and has **no private key**. The contract documentation explicitly states this:

> "COAs are not controlled by a key. Instead, every COA account has a unique resource accessible on the Cadence side, and anyone who owns that resource submits transactions on behalf of this address." [1](#0-0) 

The only way to move FLOW tokens out of a COA's EVM address back to Cadence is through the `withdraw` function, which is gated behind the `Owner | Withdraw` entitlement — meaning the caller must hold the live resource: [2](#0-1) 

The `CadenceOwnedAccount` resource has **no `destroy` hook** (no `ResourceDestroyed` event, no pre-condition asserting `self.balance().isZero()`). Cadence allows the resource to be destroyed unconditionally: [3](#0-2) 

When `destroy cadenceOwnedAccount` is executed:
1. The Cadence resource is destroyed and its `uuid` is gone.
2. The EVM address (derived from that `uuid`) persists in EVM state with its full balance intact.
3. No new COA can ever be created with the same EVM address (address derivation is `uuid`-based and `uuid` is globally unique and non-reusable).
4. The `withdraw` entitlement function can never be called again because the resource no longer exists.

The EVM balance is permanently frozen.

---

### Impact Explanation

Any FLOW tokens held in the COA's EVM address at the time of destruction — including EVM-side yield, bridged tokens, or rewards paid by EVM DeFi contracts to the COA address — are permanently inaccessible. This matches the target impact scope: **permanent freezing of unclaimed yield**.

The EVM state retains the balance (it is not burned), but there is no mechanism to recover it. The `Account.Withdraw` interface requires the live resource: [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged user who:
1. Creates a COA via `EVM.createCadenceOwnedAccount()`,
2. Receives FLOW tokens in the EVM address (e.g., from an EVM DeFi protocol paying yield, a bridge deposit, or a direct EVM transfer),
3. Destroys the COA resource without first calling `withdraw` to recover the balance,

will permanently lose those tokens. The pattern is realistic: users routinely destroy resources to clean up storage, and the EVM balance is not visible as a Cadence resource field — it lives in a separate EVM state trie. A user inspecting the Cadence resource sees no balance field to remind them to withdraw first.

The existing test code demonstrates that `destroy cadenceOwnedAccount` is a normal, accepted operation even after a deposit: [5](#0-4) 

---

### Recommendation

Add a `destroy` hook (or a Cadence `ResourceDestroyed` event with a pre-condition) to `CadenceOwnedAccount` that asserts the EVM balance is zero before allowing destruction:

```cadence
// In CadenceOwnedAccount resource:
event ResourceDestroyed(address: String = self.address().toString(), balance: UInt = InternalEVM.balance(address: self.addressBytes))

// Or enforce via pre-condition:
// pre { self.balance().isZero() : "EVM.CadenceOwnedAccount: cannot destroy with non-zero balance; withdraw first" }
```

This mirrors the pattern used in standard Cadence `FungibleToken.Vault` implementations, which enforce zero-balance before destruction.

---

### Proof of Concept

The following Cadence transaction demonstrates the issue. After execution, the FLOW tokens deposited into the COA are permanently inaccessible:

```cadence
import EVM from <EVM_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Mint some FLOW tokens
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin
        )!
        let minter <- admin.createNewMinter(allowedAmount: 5.0)
        let vault <- minter.mintTokens(amount: 5.0)
        destroy minter

        // Create a COA and deposit 5 FLOW into its EVM address
        let coa <- EVM.createCadenceOwnedAccount()
        let evmAddr = coa.address()
        coa.deposit(from: <-vault)

        // At this point coa.balance() == 5.0 FLOW in EVM state.
        // Destroy the COA WITHOUT withdrawing first.
        destroy coa

        // The EVM address `evmAddr` still exists in EVM state with 5.0 FLOW,
        // but there is no resource, no key, and no mechanism to ever withdraw it.
        // The tokens are permanently frozen.
    }
}
```

The EVM address balance can be confirmed to be non-zero after the transaction via `EVM.EVMAddress(bytes: evmAddr.bytes).balance()`, but no transaction can ever move those tokens because `CadenceOwnedAccount.withdraw` requires the live resource.
