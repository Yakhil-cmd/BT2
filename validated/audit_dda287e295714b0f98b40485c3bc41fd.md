Looking at the exact code path in `ViewClientActor::handle(GetExecutionOutcome)` and `Chain::get_next_block_hash_with_new_chunk`, I need to trace whether `target_shard_index` computed from the old epoch's layout is used to index into the new epoch's `outcome_roots` vector.

**Tracing the code precisely:**

In `chain/client/src/view_client_actor.rs` lines 1148–1182:

1. `epoch_id` is fetched from the **outcome block** (old epoch).
2. `shard_layout` is the **old epoch's** layout.
3. `target_shard_id` = `account_id_to_shard_id(..., &epoch_id)` — old epoch's shard ID.
4. `target_shard_index` = `shard_layout.get_shard_index(target_shard_id)` — **old epoch's index**.
5. `get_next_block_hash_with_new_chunk` is called and may return `(h, child_shard_id)` where `h` is in the **new epoch** after resharding.
6. `outcome_roots` is built from block `h` (new epoch's chunk layout).
7. `merklize(&outcome_roots).1[target_shard_index]` — **old index into new epoch's vector**.

In `chain/chain/src/chain.rs` lines 3917–3931, `get_next_block_hash_with_new_chunk` correctly updates `shard_ids` to child shards and `shard_layout` to the new layout when crossing an epoch boundary with resharding. It returns `(block_hash, child_shard_id)` where `child_shard_id` is a shard ID in the **new** layout.

The returned `target_shard_id` (line 1163, shadowing the outer variable) is the new child shard ID. But `target_shard_index` is never recomputed — it remains the index from the old epoch's layout.

In V2/V3 shard layouts, `shard_id ≠ shard_index` (explicitly noted at `mod.rs:378`). After a split, the child shards have different indices in the new layout than the parent had in the old layout.

The bounds check at line 1174 uses the old `target_shard_index` against `outcome_roots.len()` (new epoch's count). Since the new epoch has more shards, the old index is likely still in-bounds, so no error is raised — a silently wrong proof is returned.

### Title
Stale `target_shard_index` from old epoch's shard layout used to index new epoch's `outcome_roots` after resharding, producing an unverifiable `outcome_root_proof` — (`chain/client/src/view_client_actor.rs`)

### Summary

`ViewClientActor::handle(GetExecutionOutcome)` computes `target_shard_index` from the outcome block's (old) epoch shard layout, then calls `get_next_block_hash_with_new_chunk` which may return a block `h` in a new epoch with a different shard layout. The `outcome_roots` vector is built from block `h` (new layout), but `merklize(&outcome_roots).1[target_shard_index]` still uses the old epoch's index, producing a `MerklePath` for the wrong leaf. The returned `outcome_root_proof` fails `verify_path` against the block's `outcome_root`.

### Finding Description

The handler computes `target_shard_index` once, before calling `get_next_block_hash_with_new_chunk`:

```rust
// Lines 1148-1158: epoch_id and shard_layout from the OUTCOME block (old epoch)
let epoch_id = *self.chain.get_block(&outcome_proof.block_hash)?.header().epoch_id();
let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id).into_chain_error()?;
let target_shard_id =
    account_id_to_shard_id(self.epoch_manager.as_ref(), &account_id, &epoch_id)
        .into_chain_error()?;
let target_shard_index = shard_layout   // <-- OLD epoch layout
    .get_shard_index(target_shard_id)
    .map_err(Into::into)
    .into_chain_error()?;
``` [1](#0-0) 

`get_next_block_hash_with_new_chunk` walks forward across the epoch boundary, updates its internal `shard_layout` and `shard_ids` to the new epoch's children, and returns `(h, child_shard_id)`:

```rust
// Lines 3917-3931: shard_layout updated to new epoch's layout internally

    shard_ids = shard_ids.into_iter()
        .flat_map(|id| next_shard_layout.get_children_shards_ids(id)...)
        .collect();
    shard_layout = next_shard_layout;  // new layout, NOT returned to caller

``` [2](#0-1) 

Back in the handler, `target_shard_id` is shadowed by the child shard ID, but `target_shard_index` is never recomputed:

```rust

        .collect::<Vec<_>>();              // ordered by NEW epoch's shard layout

    outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone()
    //                                            ^^^^^^^^^^^^^^^^^^^ OLD epoch's index!
``` [3](#0-2) 

In V2/V3 shard layouts, `shard_id` and `shard_index` are explicitly decoupled: [4](#0-3) 

The `id_to_index_map` lookup in V2/V3 means a parent shard at index `I` in the old layout may have children at completely different indices in the new layout: [5](#0-4) 

### Impact Explanation

The `outcome_root_proof` (`MerklePath`) returned in `GetExecutionOutcomeResponse` (and surfaced via `RpcLightClientExecutionProofResponse`) proves the wrong leaf of the Merkle tree built from `outcome_roots`. Any light client or bridge that calls `verify_path(block.outcome_root, &outcome_root_proof, &chunk_outcome_root)` will get `false`, making the proof unverifiable. Cross-chain bridges and light clients relying on NEAR's proof API cannot confirm transaction finality for outcomes that cross a resharding epoch boundary.

### Likelihood Explanation

This triggers whenever:
1. A transaction/receipt executes in epoch N (before resharding).
2. The next block with a new chunk for the account's shard is in epoch N+1 (after resharding).
3. The shard layout uses V2 or V3 (non-trivial shard ID ↔ index mapping).

Condition 2 is common: if the outcome block is near the end of epoch N, the very next block is in epoch N+1. Resharding is a planned, recurring event on mainnet. Any user querying `EXPERIMENTAL_light_client_proof` for a transaction executed just before a resharding boundary will receive a broken proof.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the **new epoch's** shard layout:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    // Recompute shard_index using the new block's epoch layout
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

### Proof of Concept

1. Configure a test-loop with a V2→V3 resharding at epoch boundary (e.g., 4 shards → 5 shards, where shard 1 splits into shards 4 and 5 at new indices 1 and 2).
2. Submit a transaction from an account in shard 1 (old epoch) in the last block of epoch N.
3. Advance 1 block into epoch N+1 (resharding boundary).
4. Call `GetExecutionOutcome` for the transaction.
5. Assert that `verify_path(block_h.outcome_root, &outcome_root_proof, &chunk_outcome_root)` returns `false`.
6. Confirm the fix: recompute `target_shard_index` from the new epoch's layout and assert `verify_path` returns `true`.

The existing test at `test-loop-tests/src/tests/light_client.rs` (`check_outcome_proofs`) does not cross a resharding epoch boundary and would not catch this bug. [6](#0-5)

    pub fn get_shard_index(&self, shard_id: ShardId) -> Result<ShardIndex, ShardLayoutError> {
        match self {
            // In V0 & V1 the shard id and shard index are the same.
            Self::V0(_) | Self::V1(_) => Ok(shard_id.into()),
            // In V2 & V3 the shard id and shard index are **not** the same.
            Self::V2(v2) => v2.get_shard_index(shard_id),
            Self::V3(v3) => v3.get_shard_index(shard_id),
        }
    }

fn check_outcome_proofs(
    env: &mut TestLoopEnv,
    account_id: &AccountId,
    seed_hash: CryptoHash,

    let outcome = env.rpc_runner().execute_tx(tx, Duration::seconds(5)).unwrap();
    // Advance so the outcome's block is final and included in later blocks' merkle roots.
    env.rpc_runner().run_for_number_of_blocks(4);

    let mut ids = vec![TransactionOrReceiptId::Transaction {
        transaction_hash: outcome.transaction_outcome.id,
        sender_id: account_id.clone(),
    }];
    for receipt_outcome in &outcome.receipts_outcome {
        ids.push(TransactionOrReceiptId::Receipt {
            receipt_id: receipt_outcome.id,
            receiver_id: account_id.clone(),
        });

    let mut rpc = env.rpc_node_mut();
    let view_client = rpc.view_client_actor();

    // The light client head and the block merkle root all proofs are checked against.
    let light_client_block = view_client
        .handle(GetNextLightClientBlock { last_block_hash: seed_hash })
        .unwrap()
        .unwrap();
    let light_client_head = light_client_block_hash(&light_client_block);
    let block_merkle_root = light_client_block.inner_lite.block_merkle_root;
