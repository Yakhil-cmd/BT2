### Title
ETH Sent to `tick()` Is Permanently Lost When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
The `tick()` function in `RewardSupplier.sol` is `external payable` and accepts ETH as message fees for L1→L2 bridge calls. However, when `amountToMint == 0` (no pending mint requests), the entire minting block is skipped and `msg.value` is neither used nor refunded. Any ETH sent by the caller is permanently frozen in the contract with no recovery path.

### Finding Description
`tick()` is declared `external payable` and splits `msg.value` between two downstream calls only inside the `if (amountToMint > 0)` branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // uses half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // uses other half
    }
    // ← if amountToMint == 0, msg.value is silently discarded
}
``` [1](#0-0) 

When `requiredMinting()` returns `(0, 0)` — which happens whenever there are no pending L2→L1 mint-request messages — the function exits immediately after the check, leaving `msg.value` trapped in the contract. [2](#0-1) 

The contract inherits from `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl` and exposes no `withdraw()`, no `receive()` fallback that routes ETH out, and no admin sweep function. There is no mechanism to recover stranded ETH. [3](#0-2) 

### Impact Explanation
Any ETH sent to `tick()` during a no-op call (no pending mint requests) is permanently frozen inside the `RewardSupplier` contract. Because `tick()` is `external` with no access control, any address — including automated keepers or off-chain bots — can trigger this loss. The ETH is irrecoverable: there is no admin withdrawal path in the contract or its storage base. This constitutes **permanent freezing of the caller's funds**.

Allowed impact matched: **High — Permanent freezing of funds / Temporary freezing of funds**.

### Likelihood Explanation
- `tick()` has no access control; any public caller can invoke it.
- Mint requests are only queued when the L2 `RewardSupplier` explicitly sends an L2→L1 message. Between reward epochs, `requiredMinting()` will routinely return `(0, 0)`.
- Callers (e.g., keepers, bots, or users trying to trigger reward distribution) are expected to attach ETH as messaging fees. Calling `tick()` speculatively with ETH attached — before confirming pending messages — is a natural usage pattern.
- The loss is silent: no revert, no event, no indication that the ETH was not used.

### Recommendation
Add a refund for unused `msg.value` in the no-op path, or revert when `msg.value > 0` and `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_NEEDED_REFUND_ETH");
        return;
    }
    // ... existing minting logic
}
```

Alternatively, refund any excess ETH after the two downstream calls complete, in case the bridge/messaging contract does not consume the full fee.

### Proof of Concept
1. No L2→L1 mint-request messages are pending (normal inter-epoch state).
2. A keeper calls `tick{value: 0.01 ether}()` to trigger reward distribution.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. `tick()` returns. The 0.01 ETH is now held by `RewardSupplier` with no withdrawal path.
6. The keeper's ETH is permanently lost. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
