### Title
Unknown-CAT Warning Suppressed via Attacker-Controlled `infos[assetId].type = 'singleton'` in Offer File — (`packages/gui/src/hooks/useAcceptOfferHook.tsx`)

---

### Summary

An attacker who controls an offer file can set `infos[catAssetId].type` to `'singleton'` in the driver dict. The GUI's `offerAssetTypeForAssetId` function trusts this field unconditionally and returns `OfferAsset.NFT` for the CAT. The `offeredUnknownCATs` filter in `acceptOffer` explicitly excludes NFT-typed assets, so the unknown-CAT warning is never shown. The user confirms a clean dialog and `takeOffer` executes against the original offer blob.

---

### Finding Description

**Root cause — `offerAssetTypeForAssetId` in `utils.ts`:**

The function reads `info.type` directly from the attacker-supplied `offerSummary.infos` with no cross-validation against the actual spend bundle:

```typescript
switch (info.type.toLowerCase()) {
  case 'cat':      assetType = OfferAsset.TOKEN; break;
  case 'singleton': assetType = OfferAsset.NFT;  break;
}
``` [1](#0-0) 

**Warning filter in `useAcceptOfferHook.tsx`:**

```typescript
const offeredUnknownCATs: string[] = Object.entries(offerSummary.offered)
  .filter(
    ([assetId]) =>
      offerAssetTypeForAssetId(assetId, offerSummary) !== OfferAsset.NFT &&
      lookupByAssetId(assetId) === undefined,
  )
  .map(([assetId]) => assetId);
``` [2](#0-1) 

When `infos[catAssetId].type = 'singleton'`, `offerAssetTypeForAssetId` returns `OfferAsset.NFT`, the `!== OfferAsset.NFT` predicate is `false`, and the CAT is excluded from `offeredUnknownCATs`. The dialog receives an empty array and shows no warning. [3](#0-2) 

**Pre-accept banner also suppressed — `OfferBuilderViewer.tsx`:**

`offerToOfferBuilderData` routes `info?.type === 'singleton'` entries into `requestedNfts` instead of `requestedTokens`. [4](#0-3) 

`OfferBuilderViewer` computes `requestedUnknownCATsLocal` only from `.requested.tokens`, so the CAT is absent there too, and `missingRequestedCATs` stays `false`. [5](#0-4) 

Both warning surfaces are suppressed by the same type-spoofing.

**`takeOffer` executes against the original blob regardless:**

```typescript
const response = await takeOffer({ offer: offerData, fee: feeInMojos }).unwrap();
``` [6](#0-5) 

`offerData` is the raw offer string passed in by the caller — it is never re-derived from the sanitized summary. The display layer and the execution layer are decoupled, so display-layer spoofing directly affects execution.

---

### Impact Explanation

The unknown-CAT warning is the only GUI gate that alerts a taker they are about to receive a CAT for which they have no wallet and whose legitimacy they have not verified. Suppressing it means a user can be induced to exchange XCH (or another known asset) for a worthless or malicious CAT with no on-screen indication that anything is unusual. This maps directly to **High: Bypass of offer confirmation … with direct security impact** and **High: Unsafe handling of external … import payloads … that causes a user to approve … the wrong asset**.

---

### Likelihood Explanation

The attacker only needs to craft and distribute an offer file — a standard, unprivileged operation. No local access, no key material, and no dependency compromise is required. The victim only needs to import and accept the offer through the normal GUI workflow.

---

### Recommendation

1. **Do not trust `infos[id].type` for security decisions.** Derive the asset type independently from the wallet list (CAT wallets, NFT wallets) keyed on `assetId`, not from the offer's self-reported driver dict.
2. **Treat any `offered` entry that is not XCH and not a known wallet as an unknown CAT**, regardless of what `infos[id].type` claims.
3. Alternatively, add a strict allowlist: if `infos[id].type` is `'singleton'` but the assetId does not match any known NFT launcher ID in the user's wallet, treat it as an unknown CAT and show the warning.

---

### Proof of Concept

```typescript
// Mocked offerSummary with a CAT assetId whose type is spoofed to 'singleton'
const catAssetId = 'aabbcc...'; // 64-char hex, no wallet exists for it
const offerSummary = {
  offered:   { [catAssetId]: 1000 },
  requested: { xch: 1_000_000_000_000 },
  fees: 0,
  infos: {
    [catAssetId]: {
      type: 'singleton',          // <-- spoofed; real spend is a CAT
      launcherId: `0x${catAssetId}`,
      launcherPh: '0x' + '00'.repeat(32),
    },
  },
};

// offerAssetTypeForAssetId returns OfferAsset.NFT for catAssetId
assert(offerAssetTypeForAssetId(catAssetId, offerSummary) === OfferAsset.NFT);

// offeredUnknownCATs filter excludes it
const offeredUnknownCATs = Object.entries(offerSummary.offered)
  .filter(([id]) =>
    offerAssetTypeForAssetId(id, offerSummary) !== OfferAsset.NFT &&
    lookupByAssetId(id) === undefined)
  .map(([id]) => id);

assert(offeredUnknownCATs.length === 0); // warning suppressed

// Dialog shows no warning; user clicks "Yes, Accept Offer"
// takeOffer({ offer: realCATOfferBlob, fee }) executes
```
