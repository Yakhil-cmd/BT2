### Title
Gateway Admits Invoke Transactions with Unverified Signatures via `skip_stateful_validations` Deploy-Account UX Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validation path contains a deliberate UX shortcut that skips the `__validate__` entry-point call for invoke transactions with nonce=1 when the account is not yet deployed. The guard used to allow this skip — `account_tx_in_pool_or_recent_block` — only checks whether the sender address appears anywhere in the mempool or recent-block state; it does not verify that the pending transaction is a `deploy_account`, nor does it verify the invoke's signature in any other way. An unprivileged attacker can exploit this to inject invoke transactions carrying arbitrary (invalid) signatures into the mempool for any account that has a pending `deploy_account`.

### Finding Description

In `skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`), the gateway skips the blockifier `__validate__` call when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in committed state).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` constructs the `AccountTransaction` with `validate: false`, so the `__validate__` entry point — the account's signature-verification function — is never called. [2](#0-1) 

The guard `account_tx_in_pool_or_recent_block` resolves to:

```rust
self.state.contains_account(account_address) || self.tx_pool.contains_account(account_address)
``` [3](#0-2) 

`MempoolState::contains_account` checks only whether the address key exists in the `staged` or `committed` maps — it does not verify that the pending transaction is a `deploy_account`, nor does it inspect the nonce value stored there. [4](#0-3) 

**Attack path:**

1. Victim Alice submits a valid `deploy_account` for address `A` (nonce=0). It is admitted to the mempool (full execution validation runs for `DeployAccount`).
2. Attacker Bob immediately submits an `Invoke` for address `A` with `nonce=1` and an **arbitrary/invalid signature**.
3. Gateway reads `account_nonce=0` from state (account not yet deployed). Nonce check passes (`0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`).
4. `validate_by_mempool` passes (mempool nonce check only, no signature check).
5. `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` (Alice's deploy_account is in the pool) → returns `true`.
6. `run_validate_entry_point` is called with `validate=false` → `__validate__` is **never called**.
7. Bob's invalid-signature invoke is **admitted to the mempool**. [5](#0-4) 

### Impact Explanation

The broken invariant is: *every invoke transaction admitted to the mempool must have its account signature verified*. This invariant is explicitly bypassed for nonce=1 invokes targeting accounts with a pending deploy.

Concrete consequences:

- **Mempool pollution / DoS**: Bob can flood the mempool with invalid-signature invokes for every account that has a pending `deploy_account`, consuming mempool capacity without paying any fee at admission time.
- **Nonce-slot squatting**: Bob's invalid invoke occupies Alice's nonce=1 slot. Alice must either wait for Bob's transaction to be evicted (after execution failure) or pay a higher tip to replace it via fee escalation, degrading UX and potentially blocking Alice's legitimate nonce=1 transaction.
- **Fee drain on execution**: When the batcher eventually executes Alice's `deploy_account` and then Bob's nonce=1 invoke, the blockifier calls `__validate__` with Bob's invalid signature. The transaction fails validation and is reverted, but Alice's account is charged a fee for the failed validation attempt (standard Starknet revert-fee behavior), draining funds from Alice's account without her consent.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

- The attack requires no special privileges — any unprivileged user can submit transactions to the gateway.
- The triggering condition (a pending `deploy_account` in the mempool) is a normal, common user action.
- The attacker only needs to know the target account address, which is publicly visible in the mempool.
- The `max_nonce_for_validation_skip` config field (defaulting to `Nonce(Felt::ONE)`) is not referenced in `skip_stateful_validations` at all — the check is hardcoded to `Nonce(Felt::ONE)`, so the window cannot be narrowed by configuration. [6](#0-5) 

### Recommendation

1. **Verify the pending transaction type**: Before skipping `__validate__`, confirm that the transaction found in the mempool for the sender address is specifically a `deploy_account` transaction, not just any transaction. The current `account_tx_in_pool_or_recent_block` check is too broad.

2. **Verify the invoke signature against the declared class**: Even when skipping the on-chain `__validate__` call (because the account contract doesn't exist yet), the gateway could perform an off-chain signature check using the class hash declared in the pending `deploy_account` transaction.

3. **Alternatively, restrict the skip to only when a `deploy_account` with matching address is present in the pool**: Query the mempool for a `deploy_account` transaction specifically for the sender address, rather than using the generic `account_tx_in_pool_or_recent_block` predicate.

### Proof of Concept

```
1. Alice submits: deploy_account(class_hash=C, salt=S, ...) → address A, nonce=0
   Gateway: full execution validation runs → admitted to mempool

2. Bob submits: invoke(sender=A, nonce=1, signature=[0xdead, 0xbeef, ...])
   Gateway stateful validation:
     - get_nonce_from_state(A) → 0  (A not deployed yet)
     - validate_nonce: 0 ≤ 1 ≤ 200 → OK
     - validate_by_mempool: nonce check passes → OK
     - skip_stateful_validations:
         tx.nonce() == 1 ✓
         account_nonce == 0 ✓
         account_tx_in_pool_or_recent_block(A) == true ✓  (Alice's deploy_account is in pool)
         → returns true (skip)
     - run_validate_entry_point(skip_validate=true):
         execution_flags.validate = false
         __validate__ is NOT called
         → returns Ok(())
   Bob's invalid-signature invoke is admitted to the mempool.

3. Batcher executes Alice's deploy_account → A is deployed with nonce=1
4. Batcher executes Bob's invoke (nonce=1):
   - __validate__ is called with Bob's invalid signature
   - Validation fails → transaction reverted
   - Alice's account is charged a fee
   - Alice's nonce=1 slot was consumed by Bob's invalid transaction

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
