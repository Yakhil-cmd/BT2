### Title
Gateway `skip_stateful_validations` admits invoke transactions with arbitrary signatures when any pending transaction exists for the sender - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function bypasses the `__validate__` entry point (the account's signature-verification step) for any invoke transaction with `nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The pool check is not restricted to deploy-account transactions — it returns `true` for **any** transaction from that sender. An unprivileged attacker who observes a victim's pending `deploy_account` via P2P can inject an invoke with an arbitrary signature for the victim's address, which the gateway admits to the mempool without ever calling `__validate__`.

### Finding Description

**Step 1 – The skip condition.**

`skip_stateful_validations` fires when three conditions hold simultaneously:

```
tx.nonce() == Nonce(Felt::ONE)   &&
account_nonce == Nonce(Felt::ZERO) &&
mempool_client.account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

**Step 2 – The pool check is not deploy-account-specific.**

`account_tx_in_pool_or_recent_block` returns `true` if *any* transaction from the address is in the pool or a recent committed block — including the victim's own `deploy_account` that an attacker observed via P2P:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

**Step 3 – When the skip fires, `__validate__` is never called.**

`run_validate_entry_point` sets `validate: !skip_validate`, so when `skip_validate == true` the flag is `false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`StatefulValidator::perform_validations` then short-circuits before the `__validate__` call:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

`AccountTransaction::validate_tx` also guards on the same flag:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [5](#0-4) 

**Step 4 – The nonce-gap check does not block the attack.**

`validate_nonce` in the gateway accepts `account_nonce <= incoming_tx_nonce <= account_nonce + max_allowed_nonce_gap`. For the deploy-account + invoke UX to function at all, `max_allowed_nonce_gap >= 1` must be configured, so an attacker's invoke with `nonce = 1` passes this check when `account_nonce = 0`: [6](#0-5) 

**Step 5 – The mempool's own `validate_tx` does not block the attack.**

`validate_by_mempool` calls `mempool.validate_tx`, which only checks for duplicate hashes, nonce-too-old, and fee-escalation rules. It does not verify the cryptographic signature: [7](#0-6) 

### Impact Explanation

An attacker can inject an invoke transaction carrying an arbitrary/garbage signature into the mempool for any account that has a pending `deploy_account`. This violates the admission invariant: **every non-skipped invoke admitted to the mempool must have passed `__validate__`**. Concrete consequences:

1. **Nonce-slot squatting.** The attacker's transaction occupies the victim's `nonce = 1` slot. The victim's legitimate invoke with `nonce = 1` is rejected with `DuplicateNonce` (or must pay a higher fee to replace it via fee escalation).
2. **Mempool pollution / batcher waste.** The batcher includes the invalid transaction in a block proposal; it fails at `__validate__` during execution and is rejected, wasting block space and batcher CPU.
3. **Transaction delay.** The victim's legitimate invoke is delayed by at least one block.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Likelihood Explanation

- The attacker needs only to observe a `deploy_account` transaction in the P2P mempool — a normal, public event.
- No privileged access, no special account, no prior relationship with the victim is required.
- The attack window is the time between the victim's `deploy_account` entering the mempool and being included in a block (typically seconds to minutes).
- The attack is cheap: the attacker pays nothing (the invalid transaction is rejected, not reverted, so no fee is charged to anyone).

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction exists for the sender. For example, expose a `deploy_account_tx_in_pool(address)` query from the mempool that inspects the transaction type, and use that in `skip_stateful_validations` instead of the generic pool-membership check.

Alternatively, restrict the skip to cases where the mempool already holds a transaction of type `DeployAccount` for the sender address, preventing any other transaction type from triggering the bypass.

### Proof of Concept

```
1. Alice submits deploy_account(sender=X, nonce=0, valid_sig).
   → Passes all gateway checks; admitted to mempool.

2. Attacker observes X in the mempool via P2P.

3. Attacker submits invoke(sender=X, nonce=1, calldata=anything, signature=[0xdead, 0xbeef]).

4. Gateway stateless check: passes (signature length ≤ max).

5. Gateway stateful – validate_nonce:
   account_nonce=0, tx_nonce=1, max_allowed_nonce_gap≥1 → passes.

6. Gateway stateful – validate_by_mempool:
   nonce=1 ≥ account_nonce=0, no duplicate → passes.

7. Gateway stateful – skip_stateful_validations:
   nonce==1 ✓, account_nonce==0 ✓,
   account_tx_in_pool_or_recent_block(X)==true (Alice's deploy_account) ✓
   → returns true (skip __validate__).

8. run_validate_entry_point: validate=false → __validate__ NOT called.
   Transaction admitted to mempool.

9. Alice submits invoke(sender=X, nonce=1, calldata=legitimate, valid_sig).
   → Mempool rejects: MempoolError::DuplicateNonce { address: X, nonce: 1 }.

10. Alice's transaction is blocked until the attacker's transaction is
    processed by the batcher (fails at __validate__, rejected, removed from mempool).
``` [8](#0-7) [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
