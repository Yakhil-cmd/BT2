### Title
GossipSub `BehaviourPenaltyThreshold` Default Value Mismatches Its Own Comment - (`File: config/default-config.yml`)

### Summary

The default configuration for the GossipSub peer scoring system sets `penalty-threshold` (i.e., `BehaviourPenaltyThreshold`) to `1000`, while the inline comment directly above the value explicitly states it should be `10`. This is a direct analog to the reported Solidity vulnerability: a numeric constant does not match the value described in the accompanying comment/documentation, causing the security parameter to behave contrary to its stated design intent.

### Finding Description

In `config/default-config.yml`, the `behaviour.penalty-threshold` field under `scoring-parameters.peer-scoring.internal` is set to `1000`:

```yaml
# We set it to 10, meaning that we at most tolerate 10 of such RPCs containing iHave broken promises.
penalty-threshold: 1000
```

The comment states the threshold is `10`, but the actual value is `1000` — a factor of 100 larger. This value is loaded at startup via `config.DefaultConfig()` and flows directly into `pubsub.PeerScoreParams.BehaviourPenaltyThreshold` through `NewScoreOption` in `network/p2p/scoring/score_option.go`.

The `BehaviourPenaltyThreshold` controls when GossipSub begins penalizing a peer for iHave broken promises and GRAFT flood attacks. A peer's misbehavior counter must exceed this threshold before any penalty weight is applied. With the threshold set to `1000` instead of the intended `10`, a malicious unstaked Access node or any unauthenticated peer can send up to `1000` iHave broken-promise RPCs before incurring any GossipSub-level penalty — 100× more than the design allows.

The README and multiple inline comments throughout the codebase consistently describe the intended threshold as `10`:
- `network/p2p/scoring/README.md` lines 193–199: "`BehaviourPenaltyThreshold` is set to `defaultBehaviourPenaltyThreshold`, i.e., `10`"
- `config/default-config.yml` lines 450–451: "We set it to 10, meaning that we at most tolerate 10 of such RPCs"
- `config/default-config.yml` line 497: "the generous threshold that we set (i.e., `defaultBehaviourPenaltyThreshold = 10`)"

The penalty weight math in the comments (lines 467–481) is also calibrated for a threshold of `10`, not `1000`.

### Impact Explanation

Any peer reachable over the network — including an unprivileged, unstaked Access node or Observer node — can send iHave broken-promise RPCs to victim nodes. With the threshold at `1000`, the attacker can sustain `1000` such RPCs (instead of `10`) before the GossipSub scoring system begins penalizing them. This effectively disables the iHave broken-promise spam protection for the first `1000` misbehaviors per peer, allowing a sustained iHave spam / lazy-pull DoS attack against any Flow node running with the default configuration. The attacker can exhaust victim nodes' iWant processing capacity without being graylisted.

### Likelihood Explanation

The default configuration is loaded by every node that does not explicitly override this parameter. The attack requires only network connectivity to a Flow node — no staking, no private keys, no privileged access. Any peer that can establish a GossipSub connection can exploit this. The misconfiguration is present in the shipped default and would affect all production nodes using `config.DefaultConfig()` without override.

### Recommendation

Change `penalty-threshold` in `config/default-config.yml` from `1000` to `10` to match the stated design intent in the comments, the README, and the penalty weight math:

```yaml
penalty-threshold: 10
```

Also verify that `penalty-decay` (currently `0.5`) matches the intended `0.99` described in the README and comments (lines 195, 496), as there is a secondary discrepancy there as well.
