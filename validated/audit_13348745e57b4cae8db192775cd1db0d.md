The vulnerability claim is **valid**. The code confirms it directly.

### Title
Block-list enforcement skipped during DeliverTx/FinalizeBlock — (`app/block_address.go`)

### Summary

`BlockAddressesDecorator.AnteHandle` wraps **all** signer, destination, EIP-7702 authority, and `MsgStoreBlockList` authorization checks inside a single `if ctx.IsCheckTx()` guard. During `DeliverTx`/`FinalizeBlock`, `ctx.IsCheckTx()` returns `false`, so the entire enforcement body is skipped and the ante handler unconditionally calls `next`.

### Finding Description [1](#0-0) 

Lines 33–82 are entirely inside `if ctx.IsCheckTx() { ... }`. The closing brace at line 82 is followed immediately by `return next(ctx, tx, simulate)` at line 83, which is unconditional. This means:

- **CheckTx** (mempool admission): blocked signers, blocked EVM destinations, blocked EIP-7702 authorities/targets, and unauthorized `MsgStoreBlockList` senders are all rejected.
- **DeliverTx / FinalizeBlock** (actual state execution): none of those checks run. The decorator is a no-op.

A validator can inject a transaction from a blocked address directly into a block proposal (bypassing the mempool entirely). Because the ante handler is re-executed during `FinalizeBlock` with `IsCheckTx() == false`, the block-list check is never reached and the transaction executes successfully.

### Impact Explanation

The block-list is the primary mechanism for preventing sanctioned or otherwise restricted addresses from interacting with the chain. Bypassing it during actual execution allows a blocked address to:

- Execute EVM transactions (including contract calls and ETH transfers)
- Perform IBC transfers
- Execute bridge/conversion operations (CRC20/CRC21)
- Submit `MsgStoreBlockList` from an unauthorized address

This directly matches the **High** impact category: *Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks*.

### Likelihood Explanation

Any validator — without needing compromised keys — can include an arbitrary signed transaction in a block proposal without routing it through the mempool. This is standard Cosmos SDK behavior (validators build blocks from their own transaction pool or directly). No special privilege beyond being a validator is required to exploit this.

### Recommendation

Remove the `if ctx.IsCheckTx()` guard so that all checks run unconditionally during both `CheckTx` and `DeliverTx`. If certain checks (e.g., the `MsgStoreBlockList` admin check) are intentionally mempool-only, they should be moved to message-server validation (`ValidateBasic` or the keeper's `MsgServer`), not left unguarded during execution.

### Proof of Concept

1. Add address `A` to the block-list via `MsgStoreBlockList`.
2. Have address `A` sign a valid `MsgEthereumTx` or IBC `MsgTransfer`.
3. Have a validator include that transaction directly in a block proposal (bypassing mempool/`CheckTx`).
4. Observe that during `FinalizeBlock`, `BlockAddressesDecorator.AnteHandle` is called with `ctx.IsCheckTx() == false`, the entire `if` body is skipped, `next` is called, and the transaction executes successfully — state changes are committed despite `A` being blocked. [2](#0-1) [3](#0-2)

func (bad BlockAddressesDecorator) AnteHandle(ctx sdk.Context, tx sdk.Tx, simulate bool, next sdk.AnteHandler) (newCtx sdk.Context, err error) {
	if ctx.IsCheckTx() {
		if sigTx, ok := tx.(signing.SigVerifiableTx); ok {
			signers, err := sigTx.GetSigners()
			if err != nil {
				return ctx, err
			}
			for _, signer := range signers {
				if _, ok := bad.blockedMap[sdk.AccAddress(signer).String()]; ok {
					return ctx, fmt.Errorf("signer is blocked: %s", sdk.AccAddress(signer).String())
				}
			}
		}

		for _, msg := range tx.GetMsgs() {
			msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
			if ok {
				ethTx := msgEthTx.AsTransaction()
				// check the destination address
				if ethTx.To() != nil {
					if _, ok := bad.blockedMap[sdk.AccAddress(ethTx.To().Bytes()).String()]; ok {
						return ctx, fmt.Errorf("destination address is blocked: %s", sdk.AccAddress(ethTx.To().Bytes()).String())
					}
				}
				// check EIP-7702 authorisation list
				if ethTx.SetCodeAuthorizations() != nil {
					for _, auth := range ethTx.SetCodeAuthorizations() {
						addr, err := auth.Authority()
						if err == nil {
							if _, ok := bad.blockedMap[sdk.AccAddress(addr.Bytes()).String()]; ok {
								return ctx, fmt.Errorf("signer is blocked: %s", addr.String())
							}
						}
						// check the target address
						if _, ok := bad.blockedMap[sdk.AccAddress(auth.Address.Bytes()).String()]; ok {
							return ctx, fmt.Errorf("authorisation address is blocked: %s", sdk.AccAddress(auth.Address.Bytes()).String())
						}
					}
				}
			}
		}

		admin := bad.getParams(ctx).CronosAdmin
		for _, msg := range tx.GetMsgs() {
			if blocklistMsg, ok := msg.(*types.MsgStoreBlockList); ok {
				if admin != blocklistMsg.From {
					return ctx, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
				}
			}
		}
	}
	return next(ctx, tx, simulate)
