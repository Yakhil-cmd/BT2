### Title
Missing Validation of Participant Share Contributions in CKD Coordinator Allows Malicious Participant to Corrupt CKD Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator receives `(norm_big_y, norm_big_c)` from each participant and blindly aggregates them with no cryptographic validation. There is no proof of knowledge that `norm_big_y` was correctly computed as `λ_i · y_i · G`, and no proof that `norm_big_c` was correctly computed as `λ_i · (x_i · H(pk, app_id) + y_i · app_pk)`. A single malicious participant can inject arbitrary group elements, causing all honest parties to accept a corrupted CKD output.

---

### Finding Description

In `do_ckd_coordinator` (lines 35–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator collects each participant's share and sums them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The `compute_signature_share` function (lines 148–182) defines the correct computation each participant must perform:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` [2](#0-1) 

However, the coordinator has no mechanism to verify that any participant actually followed this computation. There is no ZK proof of knowledge of `y_i` (to bind `norm_big_y` to a known discrete log), and no proof that `norm_big_c` is consistent with the participant's committed key share `x_i` and the claimed `y_i`. The coordinator simply trusts the received values.

This is the direct analog of the Chainlink `latestAnswer()` pattern: data is consumed from an external source (here, a protocol participant) without running the extra validations that would detect incorrect or malicious values. Just as `latestRoundData` exposes `answeredInRound` and `updatedAt` to allow staleness checks, a ZK proof attached to each share would allow the coordinator to verify correctness before aggregation.

**Attacker-controlled entry path:**

1. Attacker is a valid participant in the CKD protocol (no privilege required beyond being in the participant list).
2. In `do_ckd_participant` (lines 17–33), the attacker sends an arbitrary `(norm_big_y, norm_big_c)` pair to the coordinator instead of the correctly computed values. [3](#0-2) 

3. The coordinator in `do_ckd_coordinator` adds the malicious values without any check.
4. The final `CKDOutput` is corrupted; all honest parties accept it.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept incorrect cryptographic outputs.**

The final `CKDOutput` is the aggregated `(Y, C)` pair from which the confidential key is derived via `CKDOutput::unmask(app_sk)` as `C − app_sk · Y`. A malicious participant who injects arbitrary `(norm_big_y, norm_big_c)` shifts both `Y` and `C` by attacker-chosen group elements, producing a derived confidential key that is neither the intended `msk · H(pk, app_id)` nor detectable as wrong by any honest party. The TEE application receiving this key will silently operate on incorrect secret material.

---

### Likelihood Explanation

**Medium.** The attacker must be a legitimate participant in the CKD protocol session — a realistic threat model for threshold protocols where one of the `N` participants is compromised or malicious. No external access, leaked keys, or cryptographic breaks are required. The attack requires only sending a crafted message in a single protocol round.

---

### Recommendation

Attach a non-interactive ZK proof to each participant's share submission, proving:

1. Knowledge of `y_i` such that `norm_big_y = λ_i · y_i · G` (a standard Schnorr discrete-log proof).
2. That `norm_big_c` is consistent with the participant's public key share and the claimed `y_i` (a proof of discrete-log equality, analogous to the `dlogeq` proof already present in `src/crypto/proofs/dlogeq.rs`). [4](#0-3) 

The coordinator must verify both proofs before adding any participant's contribution to the aggregate. This mirrors the fix described in the external report: replacing a bare data fetch with a richer call that exposes fields enabling validation.

---

### Proof of Concept

**Setup:** 3-participant CKD session; participants `P1`, `P2` (honest), `P3` (malicious). Coordinator is `P1`.

**Attack:**

1. `P3` runs `do_ckd_participant` but instead of calling `compute_signature_share`, it constructs `norm_big_y = ElementG1::identity()` and `norm_big_c = arbitrary_target_point` and sends them to the coordinator.
2. The coordinator's loop at lines 50–55 adds `identity` to `norm_big_y` (no change) and adds `arbitrary_target_point` to `norm_big_c` (shifts the aggregate by an attacker-chosen offset).
3. The resulting `CKDOutput` satisfies `C_final = C_honest + arbitrary_target_point`, so `unmask(app_sk)` returns `msk · H(pk, app_id) + arbitrary_target_point` — a value entirely controlled by the attacker's injection, not the intended confidential key.
4. No error is raised; the coordinator returns `Some(ckd_output)` with the corrupted value. [5](#0-4)

/// We use a transcript in order to verify the Fiat-Shamir transformation.
pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
