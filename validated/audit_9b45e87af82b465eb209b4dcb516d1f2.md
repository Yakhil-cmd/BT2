### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Suppress Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `src/staking/staking.cairo` is specified to be callable only by the Starkware sequencer, but the implementation contains no such access control check. Any unprivileged address can call it with `disable_rewards: true`, consuming the single per-block reward slot and permanently denying stakers and pool members their consensus block rewards for that block.

### Finding Description
The protocol specification at `docs/spec.md:1645` explicitly states:

> **access control:** Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` only calls `self.general_prerequisites()`, which exclusively checks that the contract is not paused and that the caller is not the zero address:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);  // consumes the slot

    if disable_rewards || self.is_pre_consensus() {
        return;   // exits without distributing rewards
    }
    ...
```

There is no check that `get_caller_address() == sequencer_address` or any equivalent privileged-caller guard. The function writes `last_reward_block` to the current block number **before** the `disable_rewards` branch, meaning once any caller invokes it in a given block, no further call can succeed for that block (`REWARDS_ALREADY_UPDATED` is thrown). An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` first in a block permanently forfeits all staker and pool-member rewards for that block.

The analog to the reference bug is direct: the reference bug had a payment-validation check inside a loop that was silently skipped when no funds were sent, allowing state to be updated without payment. Here, the caller-identity check is entirely absent from the implementation despite being required by the spec, allowing state (`last_reward_block`) to be updated without the caller being the authorized sequencer, and causing the reward distribution branch to be skipped.

### Impact Explanation
- Every block in which an attacker front-runs the sequencer's `update_rewards` call with `disable_rewards: true` results in **permanent, irrecoverable loss of that block's STRK rewards** for the targeted staker and all pool members delegating to them.
- The `last_reward_block` guard ensures the slot cannot be reclaimed: once written for block N, no call can distribute rewards for block N ever again.
- An attacker can sustain this indefinitely at low cost (one transaction per block), causing **permanent freezing of unclaimed yield** for any or all stakers.
- This matches the allowed impact: **High: Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation
- The function is publicly callable with no role restriction in the deployed contract.
- The attacker only needs to submit a transaction before the sequencer's own `update_rewards` call each block — a straightforward front-running strategy on Starknet.
- No special privileges, leaked keys, or external dependencies are required.
- The cost is a single transaction per block; the attacker has no financial incentive requirement (pure griefing).

### Recommendation
Add a sequencer-only access control guard at the top of `update_rewards`, consistent with the specification. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // enforce spec: "Only starkware sequencer"
    ...
```

Where `assert_caller_is_sequencer` checks `get_caller_address()` against a stored sequencer address (or uses the existing roles component). Alternatively, remove the `disable_rewards` parameter from the public interface and handle the disable-rewards path through a separate privileged entrypoint.

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Staker Alice has staked and has an active balance.
3. Attacker Bob (any address) monitors the mempool and, at each new block, submits:
   ```
   update_rewards(staker_address: alice, disable_rewards: true)
   ```
   before the sequencer's legitimate call.
4. The function executes: `last_reward_block` is written to the current block number, then returns early due to `disable_rewards == true`.
5. When the sequencer subsequently attempts `update_rewards(alice, disable_rewards: false)`, it reverts with `REWARDS_ALREADY_UPDATED`.
6. Alice and her pool members receive zero rewards for that block. This loss is permanent.
7. Bob repeats this every block, permanently suppressing all of Alice's (and her delegators') consensus rewards.
