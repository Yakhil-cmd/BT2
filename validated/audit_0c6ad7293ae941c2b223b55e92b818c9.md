### Title
Malicious CKD Participant Can Inject Arbitrary Contributions to Corrupt the Coordinator's Output — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates `(norm_big_y, norm_big_c)` contributions from every participant by simple addition, with **no cryptographic verification** that each contribution is well-formed or consistent with the participant's committed secret share. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a silently corrupted `CKDOutput` that all honest parties accept as valid.

---

### Finding Description

The CKD coordinator aggregates participant shares in `do_ckd_coordinator`:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-56
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant computes its contribution in `compute_signature_share`:

```rust
// src/confidential_key_derivation/protocol.rs  lines 165-180
let big_y = ElementG1::generator() * y.0;
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();
let big_c = big_s + app_pk * y.0;
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
```

The protocol invariant is:

```
Y  = Σ λᵢ · yᵢ · G
C  = Σ λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)
   = msk · H(pk ‖ app_id) + Y · app_sk
```

so that `unmask(app_sk)` recovers `msk · H(pk ‖ app_id)`.

There is **no proof of knowledge**, **no commitment-then-reveal**, and **no consistency check** between `norm_big_y` and `norm_big_c`. A malicious participant simply sends any two group elements it chooses. The coordinator cannot distinguish a valid contribution from a forged one.

Compare this with the DKG protocol (`src/dkg.rs`), which protects against exactly this class of attack through:
- A commitment hash broadcast before the polynomial is revealed (lines 410–425)
- A Schnorr proof of knowledge of the secret coefficient (lines 118–141, 452–460)
- Per-share verification against the VSS commitment (lines 520–522)

The CKD protocol has none of these defences.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

A malicious participant sends `(norm_big_y′, norm_big_c′)` instead of the correct values. The coordinator computes:

```
Y′ = Y_honest + Δy
C′ = C_honest + Δc
```

`unmask(app_sk)` then yields:

```
C′ − app_sk · Y′ = msk · H(pk ‖ app_id) + (Δc − app_sk · Δy)
```

The result is a silently wrong confidential key. Because the coordinator has no way to detect the manipulation, it returns the corrupted `CKDOutput` as if it were correct. Every downstream consumer of the derived key (e.g., a BLS-based application) will silently receive an invalid key, permanently breaking the CKD output for that `(pk, app_id)` pair.

---

### Likelihood Explanation

**High.** Any single participant in the CKD protocol can trigger this. The attacker needs only to be a legitimate member of the participant set (no special privilege required beyond that). The attack requires sending two arbitrary group elements in place of the honest contribution — a trivial modification to the participant's local computation. No external assumptions, leaked keys, or cryptographic breaks are needed.

---

### Recommendation

Add a zero-knowledge proof of correct contribution, analogous to the Schnorr PoK used in the DKG. Specifically, each participant should prove in zero knowledge that:

1. `norm_big_y = λᵢ · yᵢ · G` for some scalar `yᵢ` (i.e., it is a valid scalar multiple of the generator).
2. `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)` using the **same** `yᵢ` as in (1) and the participant's committed secret share `xᵢ`.

A standard Chaum–Pedersen discrete-log equality proof covers both requirements. Alternatively, adopt a commit-then-reveal pattern (as in the DKG) so that the coordinator can verify each contribution against a prior commitment before aggregating.

---

### Proof of Concept

**Setup:** 3 participants `P1` (coordinator), `P2` (honest), `P3` (malicious). Threshold = 2.

1. `P3` runs `compute_signature_share` locally to obtain its correct `(norm_big_y_3, norm_big_c_3)`.
2. Instead of sending the correct values, `P3` sends `(G₁, G₁)` — the generator point for both components.
3. The coordinator `P1` aggregates:
   - `Y′ = norm_big_y_1 + norm_big_y_2 + G₁`
   - `C′ = norm_big_c_1 + norm_big_c_2 + G₁`
4. `P1` returns `CKDOutput::new(Y′, C′)` with no error.
5. `unmask(app_sk)` computes `C′ − app_sk · Y′ ≠ msk · H(pk ‖ app_id)`.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
