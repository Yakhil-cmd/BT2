### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Unverified Signatures into the Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX bypass in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (i.e., signature verification) for any Invoke transaction with `nonce == 1` when the sender's on-chain nonce is `0` and the sender address has *any* transaction in the mempool or a recent block. Because the check is on the sender address rather than on the submitter's identity, an unprivileged third party can submit an Invoke with a forged or empty signature for a victim's not-yet-deployed account and have it admitted to the mempool without any cryptographic verification.

---

### Finding Description

`run_pre_validation_checks` calls `skip_stateful_validations` after the nonce/resource-bounds pre-checks: [1](#0-0) 

`skip_stateful_validations` returns `true` (skip) when all four conditions hold: [2](#0-1) 

When `skip_validate == true`, `run_validate_entry_point` sets `validate: false`, so the account's `__validate__` entry point — the only place the signature is cryptographically checked — is never invoked: [3](#0-2) 

The guard condition is:

```
tx.nonce() == 1  &&  account_nonce == 0  &&  account_tx_in_pool_or_recent_block(sender) == true
```

`account_tx_in_pool_or_recent_block` checks whether *any* transaction for the sender address is present in the mempool or a recent block — it does **not** restrict to deploy-account transactions. A deploy-account transaction submitted by the victim satisfies this condition. Once it does, the gateway will admit a subsequent Invoke with `nonce == 1` for that same address regardless of who submitted it and regardless of whether the signature is valid.

The stateless validator only checks signature *length*, not cryptographic validity: [4](#0-3) 

`validate_by_mempool` (called before the skip check) validates nonce gaps and fee bounds via the mempool client, but does not verify the signature: [5](#0-4) 

**Concrete attack path:**

1. Victim broadcasts a `DeployAccount` for address `X`. It enters the mempool; `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Attacker submits an `Invoke` with `sender_address = X`, `nonce = 1`, and an arbitrary or empty signature.
3. Gateway evaluates: Invoke ✓, `nonce == 1` ✓, on-chain `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(X) == true` ✓ → `skip_validate = true`.
4. `run_validate_entry_point` is called with `validate: false`; `__validate__` is never executed.
5. The attacker's Invoke is forwarded to the mempool and stored alongside the victim's `DeployAccount`.

---

### Impact Explanation

**Mempool/gateway admission accepts an invalid transaction before sequencing (High).**

The attacker's Invoke carries an invalid signature but is unconditionally admitted. Consequences:

- **Mempool slot displacement / griefing**: If the mempool enforces one transaction per `(address, nonce)` and the attacker's Invoke arrives with a higher fee, it replaces the victim's legitimate `nonce == 1` Invoke. The victim's `DeployAccount` executes, the account is deployed, but the victim's intended action is never sequenced.
- **Mempool pollution / resource exhaustion**: An attacker can flood the mempool with signature-less Invokes for every address that has a pending `DeployAccount`, consuming mempool capacity without paying any fee (the transactions will be rejected at execution time by the batcher, which uses `validate: true` via `new_for_sequencing`). [6](#0-5) 

The batcher's `validate: true` flag means the invalid Invoke will ultimately be rejected during block building, but the damage (slot displacement, mempool pollution) occurs before that point.

---

### Likelihood Explanation

The attack requires no privileged access. Any observer of the public mempool can:

1. Watch for `DeployAccount` transactions (they are broadcast over P2P).
2. Immediately submit a competing `Invoke` with `nonce == 1` for the same address.

The window is the time between the `DeployAccount` entering the mempool and the batcher processing it — typically several seconds to minutes. The attacker needs no knowledge of the victim's private key.

---

### Recommendation

1. **Restrict the skip condition to deploy-account transactions only.** `account_tx_in_pool_or_recent_block` should be replaced with a dedicated `deploy_account_tx_in_pool_or_recent_block` check that returns `true` only when a `DeployAccount` for the address is present, not any arbitrary transaction.

2. **Alternatively, perform a lightweight signature-format check** (e.g., verify the ECDSA signature against the transaction hash using the class hash declared in the pending `DeployAccount`) before skipping the full `__validate__` call.

3. **Mempool-side guard**: The mempool's `validate_tx` path could reject an Invoke with `nonce == 1` for an undeployed account unless a `DeployAccount` for that exact address is already queued.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
