### Title
Signature Bypass via `skip_stateful_validations` Enables Nonce Front-Running DoS Against New Accounts - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry point for invoke transactions with nonce=1 when the account has **any** transaction in the mempool. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an invoke with nonce=1 for the victim's address carrying an arbitrary (invalid) signature. Because `__validate__` is skipped, the gateway admits this transaction without verifying the signature. The attacker's transaction occupies the victim's nonce=1 slot; the victim's legitimate invoke is then rejected by the mempool's `validate_tx` with `DuplicateNonce`. The attacker pays no fees (the invalid transaction fails `__validate__` at execution time and is dropped without fee charge), enabling a persistent, free denial-of-service against any new account deployment.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

This means the account's `__validate__` Cairo entry point is **never called** — the transaction's signature is not verified at the gateway level.

**The gateway flow that enables the attack**

In `run_pre_validation_checks`, `validate_by_mempool` is called **before** `skip_stateful_validations`: [4](#0-3) 

`validate_by_mempool` calls `mempool_client.validate_tx`, which internally calls both `validate_incoming_tx` and `validate_fee_escalation`: [5](#0-4) 

`validate_fee_escalation` returns `DuplicateNonce` when a transaction with the same `(address, nonce)` already exists in the pool and the incoming transaction does not meet the fee-escalation threshold: [6](#0-5) 

**Step-by-step attack**

| Step | Actor | Action | Result |
|------|-------|--------|--------|
| 1 | Victim | Submits `deploy_account` (nonce=0) for address A | Admitted to mempool |
| 2 | Attacker | Observes deploy_account in mempool (P2P) | Learns victim address A |
| 3 | Attacker | Submits `invoke` (nonce=1, sender=A, signature=`[0xdead]`) | — |
| 4 | Gateway | `validate_by_mempool` → nonce=1 not yet in pool → **ok** | — |
| 5 | Gateway | `skip_stateful_validations` → deploy_account in pool → **returns true** | — |
| 6 | Gateway | `run_validate_entry_point(skip_validate=true)` → `__validate__` **not called** | Signature never checked |
| 7 | Gateway | `mempool_client.add_tx` | Attacker's invalid invoke **admitted** |
| 8 | Victim | Submits `invoke` (nonce=1, sender=A, valid signature) | — |
| 9 | Gateway | `validate_by_mempool` → `validate_fee_escalation` → **`DuplicateNonce`** | Victim's invoke **rejected** |

The attacker's invalid invoke eventually fails `__validate__` at block-execution time. In Starknet v3, a failed `__validate__` results in transaction rejection with **no fee charged**. The attacker can repeat this cycle indefinitely at zero cost.

The comment in the code acknowledges the weaker check compared to the legacy `PyValidator`, which required an explicit `deploy_account_tx_hash` parameter: [7](#0-6) 

The new gateway replaces that explicit hash check with the broader `account_tx_in_pool_or_recent_block` query, which any transaction for the address can satisfy.

---

### Impact Explanation

**Broken invariant**: Every transaction admitted to the mempool must have its signature verified by the account's `__validate__` entry point before admission.

**Corrupted admission decision**: The gateway accepts an invoke transaction with an arbitrary/invalid signature (bypassing `__validate__`) and subsequently rejects the victim's legitimate invoke with `InvalidTransactionNonce` / `DuplicateNonce`.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The attacker does not need any privileged access — only the ability to submit transactions to the public HTTP endpoint or P2P network.

---

### Likelihood Explanation

- The victim's `deploy_account` is broadcast over P2P and visible to any network participant.
- The attacker only needs to submit one transaction per victim invoke attempt, with no fee cost on failure.
- The attack window is the latency between the victim's `deploy_account` appearing in the mempool and the victim's `invoke` being admitted — typically seconds, easily exploitable by an automated bot.
- The attack is repeatable: each time the victim resubmits after the attacker's invalid invoke is dropped from the block, the attacker can front-run again.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that verifies a **specific `deploy_account` transaction hash** is pending for the account, mirroring the legacy `PyValidator.should_run_stateful_validations` design: [8](#0-7) 

Concretely, the gateway should require the caller to supply the `deploy_account` transaction hash alongside the invoke, and only skip `__validate__` when that specific hash is present in the mempool. This prevents an attacker from exploiting the skip path without possessing the victim's private key (since submitting a valid `deploy_account` requires a valid signature from the account's own `__validate__`).

Alternatively, never skip `__validate__` at the gateway level; instead, handle the "account not yet deployed" case by catching the resulting execution error and returning a user-friendly message, analogous to the try-catch pattern recommended in the external Permit2 report.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
