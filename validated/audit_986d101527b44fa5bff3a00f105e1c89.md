### Title
Flow EVM Accepts Pre-EIP-155 Unprotected Transactions, Enabling Cross-Chain Replay Attacks - (`fvm/evm/emulator/signer.go`, `fvm/evm/emulator/config.go`, `fvm/evm/emulator/emulator.go`)

---

### Summary

Flow EVM's transaction execution pipeline uses go-ethereum's `types.MakeSigner`, which — even when `EIP155Block` is set to block 0 — still accepts pre-EIP-155 "unprotected" legacy transactions (those with `v=27` or `v=28`). This allows any unprivileged caller to replay old Ethereum mainnet transactions (signed before EIP-155 was activated in October 2016) against Flow EVM accounts, potentially stealing funds from users who share the same address on both networks.

---

### Finding Description

`MakeChainConfig` in `fvm/evm/emulator/config.go` sets `EIP155Block: bigZero`, signaling that EIP-155 is active from genesis:

```go
chainConfig := &gethParams.ChainConfig{
    ChainID:        chainID,
    HomesteadBlock: bigZero,
    EIP155Block:    bigZero,   // EIP-155 "active from block 0"
    ...
}
``` [1](#0-0) 

`GetSigner` in `fvm/evm/emulator/signer.go` then constructs the transaction signer via `types.MakeSigner`:

```go
func GetSigner(cfg *Config) types.Signer {
    return types.MakeSigner(
        cfg.ChainConfig,
        cfg.BlockContext.BlockNumber,
        cfg.BlockContext.Time,
    )
}
``` [2](#0-1) 

This signer is used directly in `RunTransaction` and `BatchRunTransactions` to recover the sender from the transaction signature:

```go
msg, err := gethCore.TransactionToMessage(
    tx,
    GetSigner(bl.config),
    proc.config.BlockContext.BaseFee)
``` [3](#0-2) 

**The critical behavior of go-ethereum's `MakeSigner`:** Even when `EIP155Block` is set to 0, the resulting signer (a `londonSigner`/`cancunSigner` wrapping `EIP155Signer`) still accepts pre-EIP-155 unprotected transactions. When a transaction has `v=27` or `v=28` (i.e., `tx.Protected() == false`), go-ethereum's `EIP155Signer.Sender()` falls back to `HomesteadSigner`, which recovers the sender without any chain ID binding. This means a transaction signed on Ethereum mainnet before EIP-155 is cryptographically valid on Flow EVM.

The public entry point is `EVM.run` in `fvm/evm/stdlib/contract.cdc`, callable by any unprivileged Cadence transaction sender:

```cadence
access(all)
fun run(tx: [UInt8], coinbase: EVMAddress): Result {
    pre { !self.isPaused(): "EVM operations are temporarily paused" }
    return InternalEVM.run(tx: tx, coinbase: coinbase.bytes) as! Result
}
``` [4](#0-3) 

No Flow account authorization is required to submit an arbitrary RLP-encoded EVM transaction.

---

### Impact Explanation

An attacker who possesses (or can observe on-chain) a pre-EIP-155 Ethereum transaction can replay it on Flow EVM:

1. **Fund theft:** If victim Alice sent a pre-EIP-155 ETH transfer with nonce `N` on Ethereum mainnet, and Alice's Flow EVM account (same address, derived from the same private key) currently has nonce `N`, the attacker can submit that exact old transaction bytes to `EVM.run` on Flow. The signature verifies (via `HomesteadSigner` fallback), the nonce matches, and Alice's Flow EVM funds are transferred to the original recipient — without Alice's consent.

2. **Fresh account exploitation (nonce=0):** Any early Ethereum user whose first-ever transaction (nonce=0) was pre-EIP-155 is immediately vulnerable the moment they fund a Flow EVM account at the same address, since the nonce starts at 0.

The impact is direct, unauthorized asset transfer from victim EVM accounts on Flow.

---

### Likelihood Explanation

- **Attacker-controlled entry path:** `EVM.run` is a public, permissionless Cadence function. Any Flow account can submit arbitrary RLP-encoded EVM transactions. No staked node, operator privilege, or leaked key is required.
- **Pre-EIP-155 transactions are publicly available:** All pre-October 2016 Ethereum transactions are permanently recorded on-chain and trivially retrievable.
- **Nonce constraint:** The victim's Flow EVM account nonce must match the nonce of the replayed transaction. For nonce=0 (fresh accounts), this is automatically satisfied. For higher nonces, the attacker must wait or engineer the nonce to match.
- **Address overlap:** Users who import their Ethereum private key into a Flow EVM wallet (a common onboarding path) will have the same address on both networks.

Likelihood is **medium**: the nonce constraint limits immediate exploitability, but the nonce=0 case is immediately exploitable for any early Ethereum user who funds a new Flow EVM account.

---

### Recommendation

Explicitly reject unprotected (pre-EIP-155) legacy transactions before passing them to the emulator. In `ContractHandler.decodeTransaction` (or in `BlockView.RunTransaction`), add a check:

```go
tx := gethTypes.Transaction{}
if err := tx.UnmarshalBinary(encodedTx); err != nil {
    return nil, err
}
// Reject unprotected (pre-EIP-155) transactions
if !tx.Protected() {
    return nil, types.ErrUnprotectedTransaction
}
``` [5](#0-4) 

This mirrors the recommendation from the zkSync finding: enforce chain-ID-bound signature verification and reject any transaction whose `v` value does not encode the Flow EVM chain ID.

---

### Proof of Concept

1. Retrieve any pre-EIP-155 Ethereum transaction from before block 2,675,000 (e.g., a simple ETH transfer with `v=27` or `v=28`). These are publicly available on Etherscan.

2. Confirm the sender address has a Flow EVM balance and that the Flow EVM account nonce matches the replayed transaction's nonce (nonce=0 for a freshly funded account).

3. Submit the raw RLP bytes via a Cadence transaction:

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>

transaction(encodedTx: [UInt8], coinbase: [UInt8; 20]) {
    execute {
        EVM.run(tx: encodedTx, coinbase: EVM.EVMAddress(bytes: coinbase))
    }
}
```

4. The `GetSigner` in `fvm/evm/emulator/signer.go` returns a signer that falls back to `HomesteadSigner` for `v=27/28`, successfully recovering the original sender. The transaction executes, transferring the victim's Flow EVM funds to the attacker-controlled recipient encoded in the old Ethereum transaction. [2](#0-1) [3](#0-2) [1](#0-0)

func (h *ContractHandler) decodeTransaction(encodedTx []byte) (*gethTypes.Transaction, error) {
	usage := common.ComputationUsage{
		Kind:      environment.ComputationKindRLPDecoding,
		Intensity: uint64(len(encodedTx)),
	}
	err := h.backend.MeterComputation(usage)
	if err != nil {
		return nil, err
	}

	tx := gethTypes.Transaction{}
	if err := tx.UnmarshalBinary(encodedTx); err != nil {
		return nil, err
	}

	return &tx, nil
}
