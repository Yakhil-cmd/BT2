Audit Report

## Title
SNS Governance `distribute_rewards` Unbounded Loops Can Permanently Block All Periodic Tasks - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `distribute_rewards` function executes two unbounded loops — ballot aggregation over all proposals × all voters, and maturity distribution over all rewarded neurons — in a single synchronous message with no instruction-limit check. At the configured ceilings (200,000 neurons, 700 proposals), the computation exceeds the IC's 40-billion-instruction per-message limit, causing a trap. Because `latest_reward_event` is written only at the very end of the function, a trap rolls back all state, `should_distribute_rewards` returns `true` again on the next timer tick, and the same failing computation is retried indefinitely — permanently blocking reward distribution and all other SNS periodic tasks.

## Finding Description

**Root cause — Loop 1 (ballot aggregation)**, `rs/sns/governance/src/governance.rs` lines 5894–5931: iterates over every `considered_proposals` entry and, for each proposal, over every `(voter, ballot)` pair in `proposal.ballots`. Worst case: `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING × MAX_NUMBER_OF_NEURONS_CEILING = 700 × 200,000 = 140,000,000` HashMap insert/update operations with no instruction-limit check or early exit. [1](#0-0) 

**Root cause — Loop 2 (maturity distribution)**, lines 5954–5997: iterates over every entry in `neuron_id_to_reward_shares` (up to 200,000 neurons), performing Decimal arithmetic and neuron mutation per entry, again with no instruction-limit check. [2](#0-1) 

**`latest_reward_event` is written only at the very end**, line 6084, after both loops complete. A trap before this point rolls back all state changes. [3](#0-2) 

**`should_distribute_rewards` re-triggers on every tick** because it checks `latest_reward_event.end_timestamp_seconds`, which was never updated after the trap. [4](#0-3) 

**`distribute_rewards` is called synchronously** from `run_periodic_tasks` at line 5513, meaning a trap aborts the entire periodic task message, blocking `maybe_finalize_disburse_maturity`, `maybe_move_staked_maturity`, upgrade checks, and `maybe_gc` as well. [5](#0-4) 

**No instruction-limit guard exists in SNS governance** — a grep for `is_message_over_threshold`, `DISTRIBUTION_MESSAGE_LIMIT`, and `is_over_instructions_limit` returns zero matches in `rs/sns/governance/`.

**NNS governance recognized and fixed this exact problem** via `RewardsDistributionInProgress` and `continue_processing`, which calls `is_over_instructions_limit()` after each neuron and resumes in subsequent messages. [6](#0-5) 

The ceiling constants that bound the worst-case computation are confirmed in `rs/sns/governance/src/types.rs`: [7](#0-6) 

## Impact Explanation

When the instruction limit is exceeded, the IC traps the entire `run_periodic_tasks` message. Because `latest_reward_event` is never updated, every subsequent invocation re-enters the same failing computation. The result is a **permanent liveness failure** of the SNS governance canister: voting rewards are never distributed, upgrade proposals cannot be finalized, maturity disbursements stall, and staked-maturity movement halts. Recovery requires an emergency canister upgrade by the SNS's controlling NNS proposal or SNS root. This matches the allowed impact **"Application/platform-level DoS … or SNS … security impact with concrete user or protocol harm"** — **High severity**.

## Likelihood Explanation

No attacker action is required. Normal protocol operation — users staking neurons and proposals being submitted and settling — drives the system toward the failure state. The SNS framework explicitly permits up to 200,000 neurons and 700 open-ballot proposals. A mature, popular SNS (e.g., post-decentralization-swap with many direct participants) can reach tens of thousands of neurons. At the ceiling values the computation is ~140 million HashMap operations plus 200,000 Decimal-arithmetic neuron mutations in a single message, which plausibly exceeds 40 billion instructions. The failure is deterministic and self-reinforcing once triggered.

## Recommendation

Apply the same batched-distribution pattern used by NNS governance:

1. After computing `neuron_id_to_reward_shares`, persist it to stable storage (analogous to `RewardsDistributionInProgress`) and **immediately** write `latest_reward_event` so the round is marked settled and the computation is not retried.
2. Distribute maturity to neurons in a separate periodic timer task that calls `is_over_instructions_limit()` (or the SNS equivalent) after each neuron and resumes in subsequent messages until the distribution is complete.
3. Alternatively, cap `considered_proposals` and neuron iteration to a safe per-message budget and carry forward any remainder to the next tick.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons = 200_000` and `max_number_of_proposals_with_ballots = 700`.
2. Have 200,000 principals stake neurons and vote on 700 proposals.
3. Wait for the reward round to end; `run_periodic_tasks` calls `distribute_rewards`.
4. `distribute_rewards` enters the ballot-aggregation loop (up to 140,000,000 HashMap operations) followed by the maturity-distribution loop (200,000 Decimal-arithmetic mutations) in a single message.
5. The message exceeds the IC's 40-billion-instruction limit and traps.
6. `latest_reward_event` is not updated; `should_distribute_rewards` returns `true` on the next tick.
7. Every subsequent `run_periodic_tasks` invocation traps at the same point — permanently blocking all SNS periodic tasks.

A deterministic integration test using PocketIC can reproduce this by populating the governance proto with the maximum number of neurons and proposals, invoking `run_periodic_tasks`, and asserting that `latest_reward_event` is never updated across repeated calls.

    fn continue_processing(
        &mut self,
        neuron_store: &mut NeuronStore,
        is_over_instructions_limit: fn() -> bool,
    ) {
        while let Some((id, reward_e8s)) = self.rewards.pop_first() {
            match neuron_store.with_neuron_mut(&id, |neuron| {
                let auto_stake = neuron.auto_stake_maturity.unwrap_or(false);
                if auto_stake {
                    neuron.staked_maturity_e8s_equivalent = Some(
                        neuron
                            .staked_maturity_e8s_equivalent
                            .unwrap_or_default()
                            .saturating_add(reward_e8s),
                    );
                } else {
                    neuron.maturity_e8s_equivalent =
                        neuron.maturity_e8s_equivalent.saturating_add(reward_e8s);
                }
            }) {
                Ok(_) => {}
                Err(e) => {
                    println!(
                        "{}Error rewarding neuron {:?} during reward_distribution.\
                    This should not be possible as neuron existence is checked when \
                    rewards are calculated: {}",
                        LOG_PREFIX, id, e
                    );
                }
            };
            if is_over_instructions_limit() {
                break;
            }
        }
    }
