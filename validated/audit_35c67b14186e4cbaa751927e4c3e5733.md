### Title
WalletConnect Offer Approval Displays Understated Royalty Total for Multi-NFT Offers Due to Incorrect Integer Division — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each NFT's royalty contribution. This causes the `amountWithRoyalties` field shown in the WalletConnect command-approval dialog to understate the true cost to the user. A user who accepts a multi-NFT offer via WalletConnect sees a lower total than they will actually spend, and approves based on that false figure.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect command display (via `parseCommandDisplay`, invoked from `main.tsx`) to compute the "total including royalties" shown to the user before they approve a `take_offer` or `create_offer_for_ids` command.

The current implementation:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // integer division
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

In the Chia NFT royalty model each NFT's royalty is independently calculated on the **full** fungible amount paid. The code instead first divides `amount` by the NFT count (integer division, truncating), then multiplies each royalty percentage by that reduced `splitAmount`. The result is that the displayed total is smaller than the amount the wallet will actually deduct.

The correct formula is:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

### Impact Explanation

The `amountWithRoyalties` string is the only figure shown to the user in the WalletConnect confirmation dialog for the fungible side of a multi-NFT offer. The user reads it, decides the cost is acceptable, and clicks Approve. The Chia wallet backend then executes the transaction using the correct (higher) royalty amounts enforced on-chain. The user's wallet is debited more than the approval screen stated. This is a direct, concrete financial loss caused by a corrupted approval display in the WalletConnect flow.

Impact category: **High** — "Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to … display the wrong … amount … or status."

### Likelihood Explanation

Any WalletConnect-connected dApp can send a `take_offer` command referencing an offer that contains two or more NFTs with non-zero royalties. No special privileges are required. The dApp author controls the offer contents and can maximise the discrepancy by choosing NFTs with high royalty percentages. The user has no other source of truth in the approval dialog.

### Recommendation

Remove the `splitAmount` division. Apply each NFT's royalty percentage to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [2](#0-1) 

Update the corresponding test expectations in `parseCommandDisplay.test.ts` to reflect the corrected values. [3](#0-2) 

### Proof of Concept

**Scenario:** WalletConnect dApp sends `take_offer` for an offer where the user pays **1 XCH** (1 000 000 000 000 mojos) to receive two NFTs — NFT-A with `royalty_percentage = 500` (5 %) and NFT-B with `royalty_percentage = 200` (2 %).

**Current (buggy) display:**
```
splitAmount = 1_000_000_000_000 / 2 = 500_000_000_000
royalty_A   = (500_000_000_000 × 500)  / 10_000 = 25_000_000_000
royalty_B   = (500_000_000_000 × 200)  / 10_000 = 10_000_000_000
total shown = 1_035_000_000_000 mojos  →  1.035 XCH
```

**Correct total (what the wallet actually deducts):**
```
royalty_A   = (1_000_000_000_000 × 500)  / 10_000 = 50_000_000_000
royalty_B   = (1_000_000_000_000 × 200)  / 10_000 = 20_000_000_000
total actual = 1_070_000_000_000 mojos  →  1.070 XCH
```

The user approves believing they spend **1.035 XCH**; their wallet is debited **1.070 XCH** — a silent overcharge of **0.035 XCH** per approval. The discrepancy scales linearly with the number of NFTs and their royalty percentages. [4](#0-3)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
