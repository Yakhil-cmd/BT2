### Title
XCC Router `add_batch_actions` Panics Without `all-promise-actions` Feature, Causing NEAR Fund Freeze — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `add_batch_actions` function is compiled as `unimplemented!()` when the `all-promise-actions` Cargo feature is absent. Any user who submits a valid `PromiseArgs::Recursive` payload containing a `SimpleNearPromise::Batch` variant through the XCC precompile will cause the router's `execute` receipt to panic. Because the NEAR unwrapped from wNEAR is transferred to the router in a prior receipt, that NEAR is left permanently stranded in the router sub-account with no automatic refund path.

---

### Finding Description

The XCC router exposes two compile-time branches for `add_batch_actions`:

    unimplemented!("NEAR batch transactions are not supported. …")
}

```rust

#[cfg(feature = "all-promise-actions")]

``` [2](#0-1) 

`add_batch_actions` is called unconditionally from `recursive_promise_create` whenever the incoming promise contains a `SimpleNearPromise::Batch` variant:

```rust
SimpleNearPromise::Batch(batch) => {
    let target = batch.target_account_id.as_ref().parse().unwrap();
    let id = env::promise_batch_create(&target);
    Self::add_batch_actions(id, &batch.actions);   // ← panics without feature
    id
}
``` [3](#0-2) 

`recursive_promise_create` is reached from `promise_create`, which is called by the public `execute` entry-point:

```rust
pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
    self.assert_preconditions();
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [4](#0-3) 

The XCC precompile in `engine-precompiles/src/xcc.rs` accepts arbitrary `CrossContractCallArgs` from the caller, serialises the inner `PromiseArgs` verbatim, and forwards it as the argument to the router's `execute` method:

```rust
let args = CrossContractCallArgs::try_from_slice(input)
    .map_err(|_| ExitError::Other(Cow::from(consts::ERR_INVALID_INPUT)))?;
let (promise, attached_near) = match args {
    CrossContractCallArgs::Eager(call) => {
        …
        let promise = PromiseCreateArgs {
            target_account_id,
            method: consts::ROUTER_EXEC_NAME.into(),
            args: borsh::to_vec(&call)…,
        };
``` [5](#0-4) 

No validation is performed on the shape of the `PromiseArgs` before it is forwarded. The precompile also transfers wNEAR from the caller to the engine's implicit address and then schedules `withdraw_wnear_to_router` to unwrap that wNEAR into real NEAR and send it to the router sub-account — all in receipts that execute **before** `execute` is called:

```rust
let withdraw_call = PromiseCreateArgs {
    target_account_id: current_account_id.clone(),
    method: "withdraw_wnear_to_router".into(),
    …
};
``` [6](#0-5) 

Because NEAR protocol does not revert state changes from earlier receipts when a later receipt panics, the NEAR that was already transferred to the router sub-account cannot be automatically recovered when `execute` panics.

---

### Impact Explanation

**High — Temporary/Permanent Fund Freeze.**

When a user attaches wNEAR to an XCC call that uses `PromiseArgs::Recursive` with a `SimpleNearPromise::Batch` payload:

1. The user's wNEAR is burned and real NEAR is sent to the router sub-account (prior receipt — irreversible).
2. The router's `execute` receipt panics via `unimplemented!()`.
3. The NEAR is stranded in the router sub-account. The only recovery path (`send_refund`) is only scheduled in the new-account creation flow, not for general failed executions.

---

### Likelihood Explanation

**Medium.** The `PromiseArgs::Recursive` variant with `SimpleNearPromise::Batch` is a natural choice for any user who wants to perform multi-action NEAR operations (e.g., create account + deploy + initialise) via XCC. The precompile accepts and forwards the payload without any type-level guard. The `all-promise-actions` feature is absent from the default build, so the panic branch is active in any standard deployment.

---

### Recommendation

1. **Enable `all-promise-actions` in the production XCC router build** so that `add_batch_actions` is fully implemented.
2. **Or** add a pre-flight validation step in the XCC precompile (`engine-precompiles/src/xcc.rs`) that rejects `PromiseArgs::Recursive` payloads containing `SimpleNearPromise::Batch` when the router is known to be built without `all-promise-actions`, returning a clear error before any wNEAR is transferred.
3. **Or** add a callback after `execute` that refunds the router's NEAR balance back to the engine on failure, analogous to the existing `send_refund` mechanism.

---

### Proof of Concept

1. Attacker (or innocent user) constructs a Borsh-encoded `CrossContractCallArgs::Eager(PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch { target_account_id: …, actions: [PromiseAction::CreateAccount] })))` with non-zero wNEAR attached.
2. Calls the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372`.
3. Precompile deducts wNEAR from the caller and schedules `withdraw_wnear_to_router` → NEAR lands in the router sub-account.
4. Precompile schedules `execute` on the router as a callback.
5. Router's `execute` dispatches to `recursive_promise_create` → `add_batch_actions` → `unimplemented!()` → receipt panics.
6. NEAR is permanently stranded in the router sub-account; the caller's wNEAR is gone.
