### Title
EVM Transaction Revert Does Not Roll Back FVM PRG State Advancement via `revertibleRandom()` Precompile — (`fvm/evm/handler/precompiles.go`)

---

### Summary

When an EVM transaction calls `revertibleRandom()` via the Cadence Arch precompile and then reverts, the EVM `StateDB` snapshot/revert mechanism rolls back EVM-layer state (balances, storage, logs), but the underlying FVM `randomGenerator` PRG object is advanced and **never rolled back**. This is a persistent in-memory side-effect that crosses the EVM/Cadence boundary and is invisible to EVM state observers.

---

### Finding Description

**Root cause — `revertibleRandomGenerator` directly advances the FVM PRG:** [1](#0-0) 

`backend.ReadRandom(rand)` calls through `WrappedEnvironment.ReadRandom()`: [2](#0-1) 

…into the FVM `randomGenerator.ReadRandom()`: [3](#0-2) 

Each call to `gen.prg.Read(buf)` irreversibly advances the ChaCha20 CSPRG stream position. The `randomGenerator` is a plain Go struct held in the FVM environment for the lifetime of the Cadence transaction. It has no snapshot/revert capability.

**EVM revert only rolls back EVM StateDB, not the FVM PRG:**

The EVM `StateDB.RevertToSnapshot()` truncates the `db.views` slice: [4](#0-3) 

This rolls back EVM account state, storage, logs, and access lists. The FVM `randomGenerator` is not part of the EVM StateDB and is not touched by this revert.

**The precompile is callable via `staticcall`:**

The test contract confirms `revertibleRandom()` is called via `staticcall` (which is supposed to be read-only): [5](#0-4) 

**The PRG is shared between EVM and Cadence within the same Cadence transaction:**

The PRG is seeded once per Cadence transaction using the block's random source and the transaction ID as a diversifier: [6](#0-5) 

Both Cadence `revertibleRandom<UInt64>()` and EVM `revertibleRandom()` consume from this same PRG stream within the same Cadence transaction.

---

### Impact Explanation

An attacker who controls an EVM transaction submitted via `EVM.run()` within a Cadence transaction can:

1. Call `revertibleRandom()` N times inside the EVM transaction (via `staticcall` to the Cadence Arch precompile), advancing the PRG by N×8 bytes.
2. Have the EVM transaction revert — EVM state is fully rolled back, leaving no EVM-visible trace.
3. The PRG position is now at step N+1.
4. Any subsequent call to `revertibleRandom<UInt64>()` in Cadence (or another EVM call) within the same Cadence transaction reads from position N+1 instead of position 1.

Since the PRG seed is deterministic (block random source + transaction ID), the attacker can precompute all PRG outputs and choose N such that position N+1 yields a desired value. This allows the attacker to steer randomness consumed by other Cadence contracts called later in the same transaction, without leaving any trace in EVM state.

---

### Likelihood Explanation

The attack requires the attacker to:
- Submit a Cadence transaction that calls `EVM.run(attackerTx)` followed by a randomness-consuming Cadence contract call.
- Know the PRG seed (computable from the block's random source and the transaction ID, both available at execution time).

This is a concrete, locally testable path on the unmodified Flow emulator. The attacker controls their own Cadence transaction and can freely compose `EVM.run()` with subsequent Cadence calls. No privileged access is required.

---

### Recommendation

The FVM PRG state advancement from EVM precompile calls should be rolled back when the EVM transaction reverts. Options include:

1. **Snapshot the PRG position before each EVM transaction** and restore it on revert. This requires the `randomGenerator` to expose a `Snapshot()`/`RevertToSnapshot()` API analogous to the EVM StateDB.
2. **Isolate the EVM PRG from the Cadence PRG**: give EVM transactions their own independent PRG instance (seeded from the Cadence PRG but not sharing state), so EVM reverts cannot affect the Cadence PRG position.
3. **Prohibit `revertibleRandom()` from being called in reverting EVM contexts**: detect revert and discard the PRG advancement (complex to implement correctly).

Option 2 is the cleanest: derive a separate PRG for EVM execution from the Cadence PRG at the start of each EVM transaction, and discard it on revert.

function callRevertibleRandomNTimesAndRevert(uint n) public {

        cadenceArch.staticcall(abi.encodeWithSignature("revertibleRandom()"));
    }
    revert(); // EVM state rolled back, PRG advancement persists
}

func revertibleRandomGenerator(backend backends.Backend) func() (uint64, error) {
	return func() (uint64, error) {

func (gen *randomGenerator) createPRG() (random.Rand, error) {
	// Use the protocol state source of randomness [SoR] for the current block's
	// execution
	source, err := gen.entropySource.RandomSource()
	// `RandomSource` does not error in normal operations.
	// Any error should be treated as an exception.
	if err != nil {
		return nil, fmt.Errorf("reading random source from state failed: %w", err)
	}

	// Use the state/protocol PRG derivation from the source of randomness:
	//  - for the transaction execution case, the PRG used must be a CSPRG
	//  - use the state/protocol/prg customizer defined for the execution environment
	//  - use the salt as an extra diversifier of the CSPRG. Although this
	//    does not add any extra entropy to the output, it allows creating an independent
	//    PRG for each transaction or script.
	csprg, err := prg.New(source, prg.ExecutionEnvironment, gen.salt)
	if err != nil {
		return nil, fmt.Errorf("failed to create a CSPRG from source: %w", err)
	}

	return csprg, nil
```

func (gen *randomGenerator) ReadRandom(buf []byte) error {
	defer gen.tracer.StartExtensiveTracingChildSpan(
		trace.FVMEnvRandom).End()

	// PRG creation is only done once.
	if !gen.isPRGCreated {
		newPRG, err := gen.createPRG()
		if err != nil {
			return err
		}
		gen.prg = newPRG
		gen.isPRGCreated = true
	}

	gen.prg.Read(buf)
	return nil
```

func (db *StateDB) RevertToSnapshot(index int) {
	if index > len(db.views) {
		db.cachedError = fmt.Errorf("invalid revert")
		return
	}
	db.views = db.views[:index]
}
