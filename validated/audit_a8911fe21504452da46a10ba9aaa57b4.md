### Title
WalletConnect `take_offer` Approval Silently Falls Back to Offer-Controlled Royalty When `nftGetInfo` Fails, Causing Understated Spending Total - (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

In the WalletConnect approval flow for `chia_wallet.take_offer`, the GUI computes and displays a "Total Amount with Royalties" to the user before they approve. The royalty percentage used in this calculation is first seeded from the offer blob's embedded `infos` field (fully controlled by the dApp/offer maker), then overridden by the on-chain value fetched via `nftGetInfo`. If `nftGetInfo` fails or returns `success: false` — which is the common case when the user is buying an NFT they do not yet own — the override never happens and the offer-controlled royalty (potentially 0%) is silently used. A malicious dApp can craft an offer blob with a suppressed or omitted royalty in the driver dict, causing the approval dialog to display a lower spending total than the transaction actually deducts.

---

### Finding Description

**Two-path royalty resolution in `parseWalletDeltaItem`**

In `parseCommandDisplay.ts`, `parseWalletDeltaItem` handles NFT assets for the approval display:

```
// Path 1 (seed): royalty from offer-provided infos field
const result: DisplayWalletDeltaItem = {
  kind: 'nft',
  nftId,
  royaltyPercentage,   // ← comes from offerSummaryRoyaltyPercentages(summary)
};

try {
  // Path 2 (override): on-chain royalty from nftGetInfo
  const nftInfo = await nftGetInfo(key);
  if (nftInfo && nftInfo.success && nftInfo.nft_info && nftInfo.nft_info.data_uris) {
    if ('royalty_percentage' in nftInfo.nft_info) {
      result.royaltyPercentage = parseRoyaltyPercentage(nftInfo.nft_info.royalty_percentage);
    }
  }
} catch {
  // NFT type has already been resolved from offer data; metadata is best effort.
}
``` [1](#0-0) 

**Seed value is offer-controlled**

`offerSummaryRoyaltyPercentages` extracts the royalty from `summary.infos[assetId]`, which is derived from the driver dict embedded in the offer blob. The offer blob is provided by the dApp. The wallet daemon's `get_offer_summary` RPC returns this driver dict as-is without independently verifying it against the on-chain NFT puzzle: [2](#0-1) 

**Override silently skipped on failure**

If `nftGetInfo` returns `{ success: false }` (NFT not indexed in the local wallet — the normal case when a user is *buying* an NFT they don't own yet), the `if (nftInfo && nftInfo.success ...)` guard prevents the override. If it throws, the `catch` block swallows the error with a comment calling it "best effort." In both cases `result.royaltyPercentage` stays at the offer-provided value. [3](#0-2) 

**Downstream: incorrect `amountWithRoyalties` shown to user**

`royaltyPercentagesForSide` collects royalty percentages from the NFT items on the receiving side. If the royalty is 0 or `undefined` (because the dApp omitted it and `nftGetInfo` failed), the array is empty. `formatAmountWithRoyalties` then returns `undefined`, so the spending line shows only the base amount with no royalty added: [4](#0-3) 

The approval dialog renders this as the authoritative total the user is agreeing to spend: [5](#0-4) 

---

### Impact Explanation

A malicious dApp sends a `chia_wallet.take_offer` WalletConnect command with an offer blob whose driver dict omits or zeroes the NFT's `royalty_percentage`. The WalletConnect approval dialog shows the user a spending total of, say, 1 XCH. The user approves. The actual transaction — enforced by the NFT's on-chain royalty puzzle — deducts 1.1 XCH (10% royalty). The user loses the royalty amount without having seen it in the approval screen.

This matches the **High** allowed impact: *"Corruption, spoofing, or unsafe trust of... WalletConnect state that causes a user to approve... the wrong... amount."*

---

### Likelihood Explanation

- Any dApp connected via WalletConnect can send `chia_wallet.take_offer` with an arbitrary offer blob.
- Omitting the royalty from the driver dict is trivial — `royaltyPercentageForDriverInfo` returns `undefined` if `also.also.transfer_program` is absent.
- `nftGetInfo` returning `success: false` for an NFT the user does not yet own is the expected behavior in a purchase flow, making the fallback path the *default* path for NFT buys.
- The existing test suite confirms `nftGetInfo` returning `success: false` is a handled case, not an edge case. [6](#0-5) 

---

### Recommendation

1. **Do not treat `nftGetInfo` failure as silent.** If the royalty cannot be verified on-chain, display a prominent warning (e.g., "Royalty information could not be verified") rather than silently using the offer-provided value.
2. **Do not use the offer's `infos` field as the authoritative royalty source.** It is dApp-controlled. Treat it only as a hint; require on-chain confirmation before computing the displayed total.
3. **Block or warn on approval** when `nftGetInfo` fails for an NFT being received, so the user is not shown a potentially understated total.

---

### Proof of Concept

1. Malicious dApp crafts an offer blob where the NFT's driver dict has `type: 'singleton'` but omits `also.also.transfer_program` (no royalty field).
2. dApp sends `chia_wallet.take_offer` via WalletConnect with this offer.
3. GUI calls `getOfferSummary` → wallet daemon returns `infos[nftId] = { type: 'singleton' }` (no royalty path).
4. `offerSummaryRoyaltyPercentages` calls `royaltyPercentageForDriverInfo({ type: 'singleton' })` → returns `undefined` (no `also` field).
5. `parseWalletDeltaItem` seeds `result.royaltyPercentage = undefined`.
6. `nftGetInfo(nftId)` returns `{ success: false }` — user doesn't own the NFT yet.
7. Override is skipped; `result.royaltyPercentage` stays `undefined`.
8. `royaltyPercentagesForSide([{ kind: 'nft', royaltyPercentage: undefined }])` → `[]`.
9. `formatAmountWithRoyalties` returns `undefined` → no `amountWithRoyalties` on the XCH spending line.
10. Approval dialog shows: **"Spending: 1 XCH"** — no royalty line.
11. User approves. Actual transaction deducts **1.1 XCH** (10% royalty enforced by NFT puzzle).
12. User loses 0.1 XCH beyond what the approval screen showed.

function offerSummaryRoyaltyPercentages(offerSummary: OfferSummaryForDisplay): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  const { infos } = offerSummary;
  if (!isPlainObject(infos)) {
    return royaltyPercentages;
  }

  for (const assetId of Object.keys(offerSummary.requested)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.spending[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  for (const assetId of Object.keys(offerSummary.offered)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.receiving[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  return royaltyPercentages;
}

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {
  return lines
    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
