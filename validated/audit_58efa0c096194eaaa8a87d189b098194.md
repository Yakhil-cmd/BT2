### Title
WalletConnect Multi-NFT Royalty Total Understated in Signing Confirmation Dialog — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible payment amount by the number of NFTs before computing each NFT's royalty contribution. When a WalletConnect dApp sends a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` request involving multiple NFTs with royalties, the "Total Amount with Royalties" line shown in the signing confirmation dialog is systematically understated. The user approves believing they will spend less than the blockchain will actually deduct.

### Finding Description

`formatAmountWithRoyalties` is called once per fungible asset line in the WalletConnect confirmation dialog. It receives the full payment `amount` and the array of `royaltyPercentages` collected from every NFT on the opposite trade side. [1](#0-0) 

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

The intent appears to be "split the payment evenly across NFTs and compute each royalty on that split." But NFT royalties are not split — each NFT's royalty is computed on the **full** payment amount. The correct formula is:

```ts
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The result of the buggy formula is placed in `amountWithRoyalties` on the `DisplayWalletDeltaItem` and rendered in the `Confirm` dialog as the authoritative "Total Amount with Royalties" label that the user reads before clicking Approve. [2](#0-1) 

The existing test suite encodes and asserts the **wrong** value, confirming the bug is live: [3](#0-2) 

Scenario from that test — 2 NFTs, royalties 5 % and 0.1 %, payment 0.0001 XCH:

| | Displayed (buggy) | Correct |
|---|---|---|
| splitAmount | 50 000 000 mojos | — |
| royaltyAmount | 2 550 000 mojos | 5 100 000 mojos |
| **Total shown** | **0.00010255 XCH** | **0.0001051 XCH** |

For high-royalty NFTs the gap is far larger. Two NFTs each with 50 % royalty and a 1 000 XCH payment:

| | Displayed | Correct |
|---|---|---|
| Total shown | 1 500 XCH | 2 000 XCH |
| **Understatement** | **500 XCH (33 %)** | |

### Impact Explanation

The `amountWithRoyalties` field is the only figure in the WalletConnect confirmation dialog that tells the user what will actually leave their wallet after royalties. Because it is understated, the user approves a transaction believing they will spend X XCH/CAT when the blockchain will deduct significantly more. This is a direct, quantifiable financial loss caused by corrupted WalletConnect state in the signing approval flow.

Fits: *"Corruption … of … WalletConnect state that causes a user to approve … the wrong … amount"* — **High**.

### Likelihood Explanation

- WalletConnect is a supported, documented feature.
- Any already-paired dApp can send `chia_wallet.take_offer` with a multi-NFT offer at any time.
- The bug is triggered by the ordinary case of buying two or more NFTs that carry royalties — a common NFT marketplace pattern.
- No special privileges, leaked keys, or social engineering are required beyond the existing WalletConnect pairing.

### Recommendation

Replace `splitAmount` with the full `amount` in `formatAmountWithRoyalties`:

```ts
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Each NFT's royalty is computed on the full payment amount, not a split.
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

Update the corresponding test assertion from `'0.00010255'` to `'0.0001051'`.

### Proof of Concept

1. Pair a WalletConnect dApp with the Chia GUI wallet.
2. Craft an offer string containing two NFTs, each with `royalty_percentage: 500` (5 %), requesting 1 000 XCH.
3. Send `chia_wallet.take_offer` via WalletConnect.
4. The confirmation dialog shows **"Total Amount with Royalties: 1 050 XCH"** (buggy: `splitAmount = 500`, each royalty = 25 XCH, total royalty = 50 XCH).
5. User clicks Approve.
6. The blockchain deducts **1 100 XCH** (correct: each royalty = 50 XCH, total royalty = 100 XCH).
7. User has lost 50 XCH more than the dialog indicated.

The discrepancy scales linearly with both the number of NFTs and their royalty percentages, making it exploitable for arbitrarily large losses in high-royalty, multi-NFT offers. [4](#0-3) [5](#0-4) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],

  if (royaltyPercentages.length === 0 || line.kind === 'nft') {

  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
