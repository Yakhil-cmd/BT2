### Title
Unprivileged caller can permanently suppress per-block consensus rewards via `update_rewards` with `disable_rewards: true` — (`src/staking/staking.cairo`)

### Summary
`update_rewards` is documented as "Only starkware sequencer" but the code enforces no such restriction. Any non-zero address can call it first in any block with `disable_rewards: true`, consuming the global `last_reward_block` slot and permanently discarding that block's rewards for all stakers.

### Finding Description
The spec states access control is "Only starkware sequencer." [1](#0-0) 

The actual implementation of `general_prerequisites` only checks the contract is unpaused and the caller is non-zero — no identity or role check is performed. [2](#0-1) 

Inside `update_rewards`, `last_reward_block` is written to storage **before** the `disable_rewards` branch is evaluated: [3](#0-2) 

`last_reward_block` is a single global slot. Once any call writes the current block number to it, the guard at line 1454–1458 causes every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

### Impact Explanation
An attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` as the first transaction in any block after consensus rewards are active. The call:
1. Passes all guards (staker exists, is active, has non-zero STRK balance).
2. Writes `current_block_number` to `last_reward_block`.
3. Returns early — zero rewards distributed.
4. All subsequent `update_rewards` calls in that block revert.

That block's rewards are permanently lost. Repeated across many blocks this constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

### Likelihood Explanation
The function is public with no caller restriction. The attacker only needs to know any valid active staker address (publicly observable on-chain) and front-run or simply be first in a block. No capital, no privilege, no special setup required.

### Recommendation
Enforce that only the authorized Starkware sequencer address (or a designated rewards-manager role) can call `update_rewards`. Add a caller check inside `update_rewards` analogous to `assert_caller_is_attestation_contract()` used in `update_rewards_from_attestation_contract`. [5](#0-4) 

### Proof of Concept
1. Deploy with two active stakers, consensus rewards active.
2. Attacker (any EOA) calls `update_rewards(staker_A, disable_rewards: true)` at block N.
3. `last_reward_block` is now N; staker_A receives zero rewards.
4. Legitimate sequencer call `update_rewards(staker_A, disable_rewards: false)` at block N → reverts `REWARDS_ALREADY_UPDATED`.
5. Block N rewards are permanently gone. Repeat every block to starve all stakers indefinitely.

The flow test at lines 2882–2885 already demonstrates that `disable_rewards: true` with consensus active yields zero rewards and sets `last_reward_block`, confirming the mechanism works exactly as described. [6](#0-5)
