### Title
`request_funds` Uses Live `balance_of` Instead of Internal Accounting, Enabling L1 Mint-Request Suppression — (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `RewardSupplier` contract's internal `request_funds` function determines whether to send a mint request to L1 by reading the contract's live STRK token balance via `balance_of(get_contract_address())`. Because any address can transfer STRK directly to the `RewardSupplier` without going through the bridge, an attacker can artificially inflate the on-chain balance, causing the contract to believe it has sufficient credit and suppressing L1 mint requests. When the donated tokens are eventually consumed paying out rewards, the contract's actual balance falls below `unclaimed_rewards`, causing `claim_rewards` calls from the staking contract to revert and temporarily freezing reward distribution.

---

### Finding Description

In `src/reward_supplier/reward_supplier.cairo`, the internal `request_funds` function is called every time the staking contract reports new rewards via `update_unclaimed_rewards_from_staking_contract`. Its job is to decide whether to send an L1 mint request:

```cairo
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // ← live balance, not internal tracker
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount;
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);
    if credit < debit + threshold {          // ← no request if credit is inflated
        ...send L1 mint messages...
    }
    self.l1_pending_requested_amount.write(l1_pending_requested_amount);
}
``` [1](#0-0) 

`credit` is computed as `balance_of(this) + l1_pending_requested_amount`. Because `balance_of` reflects any tokens held by the contract regardless of how they arrived, a direct ERC-20 transfer to the `RewardSupplier` address inflates `balance` and therefore `credit`, without updating any internal accounting variable.

The `on_receive` callback (the legitimate bridge path) correctly decrements `l1_pending_requested_amount` when bridged tokens arrive:

```cairo
fn on_receive(...) -> bool {
    ...
    if amount_u128 > l1_pending_requested_amount {
        self.l1_pending_requested_amount.write(Zero::zero());
    } else {
        l1_pending_requested_amount -= amount_u128;
        self.l1_pending_requested_amount.write(l1_pending_requested_amount);
    }
    true
}
``` [2](#0-1) 

A direct transfer bypasses `on_receive` entirely, so `l1_pending_requested_amount` is never adjusted, yet `balance` is higher. The contract therefore over-estimates its credit and under-requests from L1.

The downstream consumer is `claim_rewards`, which performs a hard token transfer to the staking contract:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
}
``` [3](#0-2) 

If the contract's actual token balance is less than `amount` at the time of this call, `checked_transfer` reverts, causing the staking contract's entire `update_rewards` / attestation reward flow to revert.

---

### Impact Explanation

When the attacker's donated tokens are consumed paying out rewards, the contract's balance drops below `unclaimed_rewards`. The next call to `claim_rewards` from the staking contract will revert because `checked_transfer` cannot transfer more tokens than the contract holds. This blocks the staking contract from distributing rewards to stakers and pool members until new funds arrive from L1 via the bridge — a delay that can span hours to days depending on L1 finality and bridge processing time.

This constitutes **temporary freezing of unclaimed yield** (High) or at minimum **griefing with damage to the protocol** (Medium). Stakers and delegators cannot claim rewards during the freeze window.

---

### Likelihood Explanation

The attack requires the attacker to spend real STRK tokens (a net loss to the attacker). There is no profit motive. However, the cost scales with the threshold value (`compute_threshold(base_mint_amount)`), which is a fixed protocol parameter. A well-funded adversary (e.g., a competing validator or a protocol antagonist) could execute this at moderate cost. The entry path is fully permissionless — any address can call `IERC20.transfer(reward_supplier_address, amount)`.

---

### Recommendation

Replace the live `balance_of` read in `request_funds` with an internal storage variable (e.g., `tracked_balance`) that is incremented only when tokens arrive via `on_receive` and decremented when tokens are transferred out via `claim_rewards`. Use `tracked_balance` in place of `balance_of(get_contract_address())` when computing `credit`. This mirrors the recommendation in the reference report: maintain an internal variable rather than relying on the live token balance.

---

### Proof of Concept

1. Observe the current `unclaimed_rewards` (`U`) and `l1_pending_requested_amount` (`P`) in the `RewardSupplier`.
2. Compute the current `credit = balance_of(reward_supplier) + P`.
3. Transfer `D` STRK tokens directly to the `RewardSupplier` address such that `credit + D >= U + threshold` for many future reward epochs.
4. For each subsequent epoch, when the staking contract calls `update_unclaimed_rewards_from_staking_contract`, `request_funds` reads the inflated balance and skips the L1 mint request.
5. After enough epochs, the donated tokens are consumed by `claim_rewards` calls. The contract's balance falls below `unclaimed_rewards`.
6. The next `claim_rewards` call from the staking contract reverts at `checked_transfer`, blocking reward distribution until a new L1 mint request is processed and bridged funds arrive. [4](#0-3) [5](#0-4)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
