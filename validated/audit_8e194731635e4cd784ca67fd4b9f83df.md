### Title
Incorrect Echo Threshold Constant for Small Participant Sets Breaks Broadcast Agreement, Corrupting DKG Outputs - (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

`echo_ready_thresholds` returns a hardcoded `(0, 0)` for `n ≤ 3` instead of computing the formula values from the specification. For `n = 3` this produces an echo threshold of `0` instead of the correct `1`, allowing a malicious DKG participant to cause honest parties to deliver different values from the same broadcast session, violating the Agreement property and corrupting DKG key material.

---

### Finding Description

The specification in `docs/network-layer.md` defines the echo threshold as:

> Once a counter for ECHO(m) is greater than `(N + MaxFaulty) / 2`, send READY(m)

where `MaxFaulty = floor((N-1)/3)`.

The implementation in `echo_ready_thresholds` short-circuits this formula for small `n`:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // ← hardcoded, bypasses the formula
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

For `n = 3`:
- `MaxFaulty = floor((3-1)/3) = 0`
- Spec echo threshold = `floor((3 + 0) / 2) = 1` → party sends READY when `count > 1` (needs ≥ 2 echoes)
- Implementation echo threshold = `0` → party sends READY when `count > 0` (needs ≥ 1 echo)

The ready threshold (`0`) is the same in both cases, so the delivery condition (`count > 2 * ready_t = 0`) fires on the very first READY message received.

**Attack scenario for n = 3 DKG:**

A malicious participant P3 (the sender in its broadcast session) sends two different polynomial commitment packages `v1` to P1 and `v2` to P2.

With the implementation's echo threshold of `0`:
1. P1 echoes `v1`; its own simulated echo gives `count(v1) = 1 > 0` → P1 immediately sends `READY(v1)`
2. P2 echoes `v2`; its own simulated echo gives `count(v2) = 1 > 0` → P2 immediately sends `READY(v2)`
3. P1 receives `READY(v2)` from P2: `count(v2) = 1 > 0 = ready_t` → P1 amplifies `READY(v2)`, but P1 already delivered `v1` (its own READY fired first)
4. P2 receives `READY(v1)` from P1: similarly delivers `v2`

**Result:** P1 delivers `v1`, P2 delivers `v2` — different values for the same broadcast session. Agreement is broken.

With the correct echo threshold of `1`:
- Neither P1 nor P2 ever accumulates `count > 1` for any value (each sees only 1 echo per value), so no READY is sent and the protocol stalls — the safe, expected behavior when a malicious sender violates the `f = 0` assumption.

The DKG (`do_keyshare`) calls `do_broadcast` in three rounds (Round 1.3 for session IDs, Round 3.2 for polynomial commitments `(C_i, R_i, s_i)`, Round 5.4 for success messages). A malicious P3 can exploit the broken broadcast in Round 3.2 to deliver different `C_3` to P1 and P2. If P3 also equivocates in Round 1.3 (sending different `sid_3` and thus different `h_3 = H_2(3, C_3, sid)` values), both honest parties pass all DKG verification checks but compute different master public keys:

- P1: `pk = C_1(0) + C_2(0) + C_3_v1(0)`
- P2: `pk = C_1(0) + C_2(0) + C_3_v2(0)`

---

### Impact Explanation

**High — Corruption of DKG outputs so honest parties accept inconsistent public keys.**

Honest parties complete the DKG protocol without error but hold shares of different master public keys. Any subsequent signing attempt will produce signatures that are invalid under the other party's view of the public key, permanently breaking the signing capability for the key set generated in that session. The corrupted key material cannot be detected without an out-of-band comparison of public keys.

---

### Likelihood Explanation

Any deployment using a 3-participant DKG (a common minimal configuration for threshold schemes) is affected. The malicious participant needs only to send different serialized polynomial commitment vectors to different honest parties — a straightforward network-layer manipulation requiring no cryptographic capability. The `n = 3` case is explicitly supported and validated by the library (DKG requires `1 < threshold < N`, so `n = 3, threshold = 2` is a valid and documented configuration).

---

### Recommendation

Remove the `n <= 3` special case and apply the formula uniformly:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n < 2 {
        return (0, 0);
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

For `n = 3` this yields `(1, 0)`, matching the specification. For `n = 2` it yields `(1, 0)` as well. The comment "no malicious parties are assumed when n ≤ 3" is misleading — the formula already encodes `f = 0` for these sizes; the special case is not needed and actively weakens the protocol.
