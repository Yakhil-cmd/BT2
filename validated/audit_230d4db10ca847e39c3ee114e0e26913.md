**Analysis of the invariant:**

The `surplus0Scaled` is computed as `balance - binTotals - notionalFee0AmountScaled`. Spread fees are then computed as a fraction of this reduced surplus. Notional fees are paid separately from `notionalFeeToken0Scaled`. The total payout is:

```
(balance - binTotals - notional) × spreadRatio + notional
```

Since `spreadRatio = adminSpreadFeeE6 / spreadSumE6 + protocolSpreadFeeE6 / spreadSumE6 = 1`, the total payout equals exactly `balance - binTotals` — the raw surplus. LP principal (`binTotals`) is never touched.

**The test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive`:** [3](#0-2) 

This test computes `surplus0Scaled` **without** subtracting the notional accumulator, then adds notional fees on top — a naive computation that would overallocate. The actual `collectFees` code avoids this by subtracting `notionalFee0AmountScaled` from surplus before computing spread fees. The test documents the pitfall, not an actual bug in the production code.
