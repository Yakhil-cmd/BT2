### Title
Unvalidated Participant Contributions in CKD Coordinator Allow Silent Output Corruption - (`src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function accepts `(norm_big_y, norm_big_c)` group elements from every participant and sums them directly, with no cryptographic proof of correct computation and no identity-element check. A single malicious participant can substitute arbitrary group elements — including the identity — causing the coordinator to silently output a wrong `CKDOutput` that all honest parties accept as legitimate.

### Finding Description
The vulnerability class from the external report is **missing validation of externally-provided values before using them in a critical operation**. The Oracle just forwarded the latest price without checking for zero or staleness; the CKD coordinator does the exact same thing with participant-supplied group elements.

In `do_ckd_coordinator` the coordinator computes its own share correctly via `compute_signature_share`, then receives every other participant's `(norm_big_y, norm_big_c)` pair and adds them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

The `compute_signature_share` function shows what a correct contribution looks like — `norm_big_y = λ_i · y · G` and `norm_big_c = λ_i · (x_i · H(pk, app_id) + y · app_pk)` — but nothing forces a participant to actually send those values. [2](#0-1) 

There is no:
- Zero-knowledge proof that `big_y` and `big_c` were derived from the participant's actual key share and a fresh random scalar.
- Check that the received elements are not the group identity (the additive zero for `ElementG1`).
- Consistency check between the received elements and the participant's public key share.

Compare this with the OT-based ECDSA presign, which explicitly rejects a zero `e_j` share and then verifies `E = e·G` before proceeding: [3](#0-2) 

And the robust ECDSA presign, which verifies `W = g^w` and rejects the identity `big_r`: [4](#0-3) 

No equivalent guard exists in the CKD coordinator path. [5](#0-4) 

### Impact Explanation
A malicious participant sends `(G1::identity(), G1::identity())` (or any attacker-chosen pair) instead of their legitimate Lagrange-weighted contribution. The coordinator adds these silently, producing a `CKDOutput` whose `big_y` and `big_c` are shifted by the attacker's chosen offset. Every honest party that receives this output treats it as the correct result of the threshold CKD protocol. The derived confidential key is therefore wrong — either unusable (if the downstream TEE application detects a mismatch) or silently incorrect (if it does not). This is a direct corruption of a CKD output accepted by honest parties, matching the **High** impact tier: *Corruption of CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs*.

### Likelihood Explanation
Any participant enrolled in a CKD session is the attacker. No special privilege, no key leakage, and no cryptographic break is required. The attacker simply deviates from the protocol by sending arbitrary bytes that deserialize to valid `ElementG1` values. Because the coordinator performs no proof verification, the attack succeeds on every invocation. The entry path is `ckd()` → `run_ckd_protocol()` → `do_ckd_coordinator()`, all reachable by an unprivileged library caller who is a legitimate participant. [6](#0-5) 

### Recommendation
Add a non-interactive zero-knowledge proof of correct computation alongside each `(norm_big_y, norm_big_c)` contribution. Concretely, each participant should prove in zero knowledge that:
- `norm_big_y = λ_i · y · G` for some scalar `y` they know, and
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y · app_pk)` for the same `y` and their committed key share `x_i`.

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
