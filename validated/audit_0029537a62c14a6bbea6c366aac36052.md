### Title
Failed NEAR `ft_transfer` Promise in `ExitToNear` Precompile Causes Permanent Fund Loss When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not compiled in (it is **not** a default feature), the `ExitToNear` precompile schedules a bare NEAR `ft_transfer` promise with **no failure callback**. If that promise fails — for example, because the recipient NEAR account is not registered in the NEP-141 contract — the ERC-20 tokens that were already burned (or the ETH already deducted from the user's EVM balance) are permanently lost with no refund path.

---

### Finding Description

The `ExitToNear` precompile's `run` function builds a `ExitToNearPrecompileCallbackArgs` struct. Under the `error_refund` feature gate, the `refund` field is populated with the data needed to re-mint tokens or return ETH on failure. Without the feature, `refund` is hardcoded to `None`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `transfer_near` is also `None` (the common case for non-wNEAR ERC-20 and ETH exits), `callback_args` equals the default value, so the promise is emitted as a bare `PromiseArgs::Create` — **no callback is attached**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `error_refund` feature is **not** listed under `default` in either crate's manifest: [3](#0-2) [4](#0-3) 

The callback that would perform the refund — `exit_to_near_precompile_callback` — only executes `refund_on_error` when `args.refund` is `Some(...)`. With `refund: None` baked into the serialized callback args, even if a callback were registered, no refund would occur: [5](#0-4) 

The `refund_on_error` function itself is correct — it re-mints ERC-20 tokens or transfers ETH back from the precompile address — but it is never reached: [6](#0-5) 

The tests explicitly document this behavior: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**ERC-20 exit path:** The ERC-20 contract burns the user's tokens before calling the precompile. If the subsequent `ft_transfer` NEAR promise fails, the burned tokens are gone — the NEP-141 balance remains in Aurora's account and the ERC-20 supply is permanently reduced. The user suffers a **permanent, irrecoverable loss** of their bridged tokens.

**ETH exit path:** The user's EVM ETH balance is debited and credited to the `exit_to_near` precompile address before the promise is dispatched. If the promise fails, the ETH is permanently locked in the precompile address with no mechanism to recover it.

Impact classification: **Critical — permanent freezing / loss of user funds.**

---

### Likelihood Explanation

The trigger condition is straightforward and requires no special privilege: a user supplies a NEAR recipient account that is not registered (has no storage deposit) in the target NEP-141 contract. This is a common mistake — NEAR's NEP-141 standard requires explicit `storage_deposit` registration before receiving tokens. Any user who exits to an unregistered account, a mistyped account ID, or a freshly created account that has not yet registered will silently lose their funds. The `error_refund` feature being non-default means any production deployment that does not explicitly opt into it is permanently exposed.

---

### Recommendation

1. Make `error_refund` a **default feature** in both `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, or unconditionally populate `refund` in `ExitToNearPrecompileCallbackArgs` without a feature gate.
2. Ensure the `exit_to_near_precompile_callback` is always registered as a callback on the `ft_transfer` promise so that any failure is detected and the refund path in `refund_on_error` is always reachable.
3. Remove the `#[cfg(not(feature = "error_refund"))] refund: None` branch entirely, as there is no safe scenario where a failed exit should result in no refund.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default build).
2. Bridge a NEP-141 token to Aurora, receiving ERC-20 tokens.
3. Call the ERC-20's `withdraw` function targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract (e.g., `"unregistered.near"`).
4. The ERC-20 tokens are burned in the EVM. The NEAR `ft_transfer` promise fails because the recipient is not registered.
5. No callback fires. No re-mint occurs.
6. Observe: ERC-20 balance is zero, NEP-141 balance of `"unregistered.near"` is zero, NEP-141 balance of Aurora contract is unchanged — the tokens are permanently lost.

This is directly confirmed by the existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs`, which asserts that without `error_refund`, the ERC-20 balance is `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` (i.e., the exit amount is gone with no refund). [9](#0-8)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
