### Title
Malicious Participant Can Permanently Abort DKG/Reshare/Refresh via Malformed Message at `recv_from_others` Waitpoint — (`src/protocol/helpers.rs`)

### Summary

`recv_from_others` propagates deserialization errors via `?`, while `reliable_broadcast_receive_all` silently ignores them with `continue`. A malicious participant in a valid DKG/reshare/refresh session can craft a single message with a syntactically valid routing header but an invalid msgpack payload, targeting the `wait_round_1` waitpoint. When an honest party's `recv_from_others` pops this message, `chan.recv` returns `Err(DeserializationError)`, which the `?` operator immediately propagates, aborting `do_keyshare` before all shares are collected.

**Root cause — `recv_from_others` propagates recv errors:**

```rust

    let (from, msg) = chan.recv(waitpoint).await?;   // ← ? aborts on DeserializationError
    if seen.put(from) {
        messages.push((from, msg));

}

**Call site in `do_keyshare`:**

The `?` here propagates the error all the way out of `do_keyshare`, which is called by `do_keygen`, `do_reshare`, and (transitively) refresh.

**Message routing has no payload validation:**

`push_message` only validates that the message is long enough and that the header bytes parse. It does not validate the sender is in the participant list, nor does it validate the payload:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);
}
``` [5](#0-4) 

**The routing header is fully predictable:**

The shared channel tag is deterministic — it is computed from a fixed domain separator and the string `"root shared"`, with no session-specific or secret input: [6](#0-5) 

The waitpoint for `wait_round_1` is sequential. `do_keyshare` calls `do_broadcast` first (consuming waitpoint 0), then `wait_round_1 = chan.next_waitpoint()` yields waitpoint 1: [7](#0-6) [8](#0-7) 

Any participant in the session can compute the exact 40-byte header for `wait_round_1` without any secret knowledge.

### Impact Explanation

A single malicious participant in an n-of-n DKG, reshare, or refresh session can permanently abort the protocol for all honest parties by delivering one message with a valid 40-byte routing header and a garbage msgpack payload. The honest party's `do_keyshare` future returns `Err(DeserializationError)` before collecting all commitment hashes, making it impossible to complete key generation or resharing. This matches the **High** impact: *Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions*.

---

### Likelihood Explanation

High. The attacker is a legitimate participant (already in the participant list). The attack requires only:
1. Computing the deterministic shared channel tag (public, no secrets needed).
2. Knowing that `wait_round_1` = waitpoint 1 (deterministic, sequential, inspectable from source).
3. Sending one message: `[40-byte header] + [0xFF 0xFF ...]` (any bytes that fail msgpack decode).

No cryptographic assumptions need to be broken. The attack is trivially reproducible in a unit test.

---

### Recommendation

Change `recv_from_others` to skip (not abort) on deserialization failure, consistent with how `reliable_broadcast_receive_all` handles it:

```rust
while !seen.full() {
    let Ok((from, msg)) = chan.recv(waitpoint).await else {
        continue;  // skip malformed messages instead of aborting
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

Alternatively, record which participant sent the malformed message and abort only after attributing the fault, so honest parties can identify the malicious participant rather than silently failing.
