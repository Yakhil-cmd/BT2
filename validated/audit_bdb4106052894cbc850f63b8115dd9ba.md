### Title
Gateway Validates L2 Gas Price Against Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Valid Transactions to Be Rejected and Invalid Transactions to Be Admitted — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validation path reads the **current block's** `l2_gas_price` from the block header to gate admission, but the batcher executes the next block using `next_l2_gas_price` (the EIP-1559-adjusted price stored separately in the same header). Because these two values diverge every block, the gateway systematically rejects transactions that would succeed in the batcher and admits transactions that will fail in the batcher.

---

### Finding Description

**Step 1 — Gateway reads the wrong gas price for admission.**

In `validate_resource_bounds`, the gateway fetches block info and uses `strk_gas_prices.l2_gas_price` as the threshold reference:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

The TODO comment is the developers' own acknowledgment that the wrong field is being read. `get_block_info()` is implemented in `GatewayFixedBlockSyncStateClient` and maps `block_header.l2_gas_price.price_in_fri` into the returned `BlockInfo`: [2](#0-1) 

The field `block_header.next_l2_gas_price` — the price that will actually govern the next block — is never read here.

**Step 2 — The same stale price is used inside `run_validate_entry_point`.**

When the gateway runs the blockifier's `__validate__` entry point, it builds a block context from the same `get_block_info()` call, only bumping the block number:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
``` [3](#0-2) 

The gas prices in `block_info` are **not** updated to `next_l2_gas_price`. Consequently, `check_fee_bounds` inside `perform_pre_validation_stage` also enforces `max_price_per_unit >= l2_gas_price` (block N's price) rather than `next_l2_gas_price` (block N+1's price): [4](#0-3) 

**Step 3 — The batcher executes with `next_l2_gas_price`, not `l2_gas_price`.**

The consensus orchestrator maintains `self.l2_gas_price`, which is the EIP-1559-adjusted price for the **next** block. After each decision it calls `update_l2_gas_price`, which runs `calculate_next_l2_gas_price_for_fin` over the just-finalized block's gas consumption: [5](#0-4) 

This updated price is stored in the block header as `next_l2_gas_price`: [6](#0-5) 

The batcher then uses `propose_block_input.block_info.gas_prices.strk_gas_prices.l2_gas_price` — which is `next_l2_gas_price` from the previous block — as the actual execution price: [7](#0-6) 

**The invariant broken:** `gateway_admission_price ≠ batcher_execution_price`. The gateway uses `P_N` (block N's `l2_gas_price`); the batcher uses `P_{N+1}` (block N's `next_l2_gas_price`). The EIP-1559 formula (`calculate_next_base_gas_price`) adjusts the price by up to `1/gas_price_max_change_denominator` per block, so the two values routinely differ: [8](#0-7) 

---

### Impact Explanation

**Case A — Price decreasing (`P_{N+1} < P_N`, block N below gas target):**

A user submits a transaction with `max_price_per_unit = V` where `P_{N+1} ≤ V < P_N`.

- Gateway `validate_resource_bounds`: `V < P_N` → **rejected** with `GAS_PRICE_TOO_LOW`.
- Batcher `check_fee_bounds`: `V ≥ P_{N+1}` → would **succeed**.

Result: a valid transaction is incorrectly rejected at the gateway admission boundary.

**Case B — Price increasing (`P_{N+1} > P_N`, block N above gas target):**

A user submits a transaction with `max_price_per_unit = V` where `P_N ≤ V < P_{N+1}`.

- Gateway `validate_resource_bounds`: `V ≥ P_N` → **admitted**.
- Batcher `check_fee_bounds`: `V < P_{N+1}` → **fails** during execution.

Result: an economically invalid transaction is admitted to the mempool and wastes sequencer resources.

Both cases match the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The EIP-1559 fee market adjusts the price every block based on gas consumption. Any block that is not exactly at the gas target produces a `next_l2_gas_price ≠ l2_gas_price`. Under normal network load this is the common case, not the exception. The magnitude of the discrepancy grows with sustained over- or under-utilization. The developers themselves flagged the root cause with a TODO comment, confirming the issue is known but unresolved.

---

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, replace `block_header.l2_gas_price` with `block_header.next_l2_gas_price` when populating the `gas_prices` field used for L2 gas validation, or expose `next_l2_gas_price` as a separate accessor and use it in both `validate_resource_bounds` and the block context constructed in `run_validate_entry_point`. This resolves the TODO at line 229 and aligns gateway admission with actual batcher execution semantics.

---

### Proof of Concept

1. Block N is finalized with `l2_gas_price = 10_000` fri and gas consumption below the gas target.
2. `calculate_next_base_gas_price` produces `next_l2_gas_price = 9_970` fri (≈0.3% decrease) and stores it in block N's header.
3. A user submits an invoke transaction with `l2_gas.max_price_per_unit = 9_980` fri.
4. Gateway calls `validate_resource_bounds`: reads `l2_gas_price = 10_000` from `get_block_info()`; with `min_gas_price_percentage = 100`, threshold = 10_000; `9_980 < 10_000` → **rejected** with `GAS_PRICE_TOO_LOW`.
5. The batcher would build block N+1 with `l2_gas_price = 9_970`; `check_fee_bounds` would see `9_980 ≥ 9_970` → **accepted**.
6. The user's valid transaction is permanently blocked from entering the mempool.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
