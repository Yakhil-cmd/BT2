### Title
`EVM.mustRun()` Silently Accepts Failed EVM Transactions, Enabling Cross-VM Asset Loss - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.mustRun()` in the production EVM contract asserts only that the EVM transaction status is not `invalid` or `unknown`, explicitly allowing `Status.failed` to pass through without reverting the enclosing Cadence transaction. This is the direct Flow analog of calling `transfer()`/`transferFrom()` without checking the return value: a Cadence transaction author who uses `mustRun` to perform EVM token transfers will have their Cadence-side state changes committed even when the EVM execution reverted, producing cross-VM asset accounting inconsistency.

### Finding Description
`EVM.mustRun()` is defined in `fvm/evm/stdlib/contract.cdc` as `access(all)` — callable by any unprivileged Cadence transaction author:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
```

The assert condition is `status == failed || status == successful`. This means a transaction that reverted inside the EVM (e.g., an ERC20 `transfer()` that returned `false` and reverted, or ran out of gas) satisfies the assertion and the Cadence transaction continues executing normally. The `Status` enum has four cases:

```cadence
access(all) enum Status: UInt8 {
    access(all) case unknown      // 0
    access(all) case invalid      // 1
    access(all) case failed       // 2  ← EVM reverted; still passes mustRun
    access(all) case successful   // 3
}
```

The function name `mustRun` implies a guarantee of execution success, but it only guarantees the transaction was not rejected at the validation stage. A Cadence transaction that:
1. Locks/burns Cadence-side tokens (state change committed)
2. Calls `EVM.mustRun()` to mint/transfer the equivalent ERC20 on the EVM side
3. Receives `Status.failed` back (EVM reverted)

…will commit the Cadence-side burn while the EVM-side mint never happened, permanently destroying assets.

### Impact Explanation
Cross-VM asset loss. Cadence-side state changes (token burns, escrow releases, balance decrements) are committed while the corresponding EVM-side token transfer silently failed. The result is a permanent discrepancy between Cadence and EVM asset accounting with no recovery path, because the EVM nonce is consumed and the failed transaction is included in the EVM block.

### Likelihood Explanation
Any unprivileged Cadence transaction author can call `EVM.mustRun()` directly. The function is `access(all)` with no entitlement guard. Bridge transactions that pair Cadence-side asset operations with EVM-side token transfers via `mustRun` are the primary risk surface. EVM execution can fail due to: out-of-gas conditions (attacker supplies a low gas limit), ERC20 tokens that revert on transfer failure, or race conditions that drain the EVM-side balance between the Cadence pre-check and the EVM call. All of these are attacker-controllable inputs.

### Recommendation
Change the assert in `mustRun` to require `Status.successful`:

```cadence
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction did not execute successfully"
    )
    return runResult
}
```

Any caller that intentionally wants to allow a failed-but-executed EVM transaction should use `EVM.run()` directly and inspect the result explicitly, mirroring the SafeERC20 pattern of making the success check mandatory at the call site.

### Proof of Concept

The following Cadence transaction demonstrates the silent failure path reachable by any unprivileged sender:

```cadence
import EVM from <EVMContractAddress>
import FungibleToken from <FungibleTokenAddress>

// Unprivileged transaction: burn Cadence tokens, then attempt EVM transfer via mustRun.
// If the EVM transfer reverts (e.g. out of gas), Cadence tokens are destroyed
// but no EVM tokens are minted — permanent cross-VM asset loss.
transaction(evmTx: [UInt8], coinbaseBytes: [UInt8; 20]) {
    prepare(signer: auth(BurnToken) &Account) {
        // Step 1: Cadence-side burn (state change committed on success)
        let vault <- signer.storage.load<@{FungibleToken.Vault}>(from: /storage/myVault)!
        destroy vault  // tokens destroyed on Cadence side

        // Step 2: EVM-side mint via mustRun — silently succeeds even if EVM reverts
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)
        let result = EVM.mustRun(tx: evmTx, coinbase: coinbase)
        // result.status may be Status.failed here — no panic, transaction commits
        // EVM tokens were never minted; Cadence tokens are gone
    }
}
```
