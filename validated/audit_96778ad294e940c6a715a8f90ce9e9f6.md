### Title
`skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures by Trusting User-Supplied `sender_address` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for an invoke transaction when the transaction's own `sender_address` field (user-supplied) is found in the mempool. An attacker who observes any address `A` with a pending deploy-account transaction can submit a crafted invoke with `sender_address = A`, nonce = 1, and an arbitrary/invalid signature. The gateway admits the transaction without ever verifying the signature.

### Finding Description

The gateway stateful path calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` — the nonce embedded in the attacker-controlled transaction.
3. `account_nonce == Nonce(Felt::ZERO)` — the on-chain nonce for the address named in the transaction.
4. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [2](#0-1) 

All four conditions are satisfied using only attacker-controlled data: `sender_address`, `nonce`, and the mempool lookup key are all taken from the transaction itself. No independent proof of ownership is required.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is then entirely bypassed: [4](#0-3) 

The only other admission check, `validate_by_mempool`, only verifies duplicate hashes and nonce range — it does not verify signatures: [5](#0-4) 

The nonce range check in `validate_nonce` also passes for nonce=1 with account_nonce=0 (within `max_allowed_nonce_gap`): [6](#0-5) 

### Impact Explanation

An attacker can submit an invoke transaction with an arbitrary/invalid signature for any address `A` that satisfies:
- On-chain nonce = 0 (account not yet deployed).
- `account_tx_in_pool_or_recent_block(A)` = true (A has a pending deploy-account or any other transaction in the mempool, or was recently committed).

The gateway admits the transaction to the mempool without signature verification. This:

1. **Admits invalid transactions**: Transactions with forged or empty signatures enter the mempool, violating the invariant that every admitted invoke must have a verified signature or be from a proven owner.
2. **Enables griefing of legitimate deploy-account + invoke flows**: The attacker can front-run a legitimate user's nonce-1 invoke by submitting a fake invoke with the same `(sender_address, nonce=1)` pair. If the attacker's transaction has a higher fee, it displaces the legitimate transaction via fee escalation. The attacker's transaction will fail at execution (when `__validate__` runs with the deployed account's real public key), but the legitimate user's transaction has been evicted and must be resubmitted.
3. **Mempool pollution**: The attacker can flood the mempool with invalid transactions targeting any address with a pending deploy-account.

### Likelihood Explanation

The attack requires only:
- Monitoring the public mempool for deploy-account transactions (trivially observable).
- Submitting a crafted invoke with `sender_address` set to the victim's address, nonce=1, and any signature bytes that pass the stateless size check.

No privileged access, special keys, or on-chain funds are required. The attack is fully unprivileged and reachable from the public gateway endpoint.

### Recommendation

The `skip_stateful_validations` check should not rely solely on the user-supplied `sender_address` to determine whether to skip signature verification. The mempool lookup key must be independently derived or the skip condition must be tightened. Specifically:

- Require that the deploy-account transaction in the mempool was submitted by the same party (e.g., verify that the deploy-account transaction's computed contract address matches the invoke's `sender_address` and that the deploy-account transaction's hash is provided and verified).
- Alternatively, restrict the skip to cases where the gateway itself received the deploy-account transaction in the same request batch (i.e., the deploy-account and invoke arrive together), rather than relying on a mempool state query keyed by the attacker-controlled `sender_address`.

### Proof of Concept

1. Observe that address `A` has a deploy-account transaction in the mempool (on-chain nonce = 0).
2. Construct an `RpcInvokeTransactionV3` with:
   - `sender_address = A`
   - `nonce = 1`
   - `signature = [0x1, 0x2]` (arbitrary, passes stateless size check)
   - Valid `resource_bounds` (passes stateless resource check)
3. Submit to the gateway's `add_transaction` endpoint.
4. The gateway calls `extract_state_nonce_and_run_validations`:
   - `account_nonce` = 0 (fetched from state for address `A`).
   - `validate_nonce`: passes (0 ≤ 1 ≤ max_gap).
   - `validate_by_mempool`: passes (no duplicate hash, valid nonce range).
   - `skip_stateful_validations`: `tx.nonce() == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(A) == true` → returns `true`.
   - `run_validate_entry_point` is called with `skip_validate = true` → `__validate__` is never executed.
5. The transaction is admitted to the mempool with an invalid signature, satisfying the "High" impact criterion: **mempool/gateway admission accepts invalid transactions before sequencing**. [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
