### Title
Malicious Equivocation in Echo Phase Triggers Premature Early-Exit Abort, Causing Inconsistent DKG Termination — (`src/protocol/echo_broadcast.rs`)

---

### Summary

The custom early-exit check inside `reliable_broadcast_receive_all` uses `n - received_echo_cnt` as the count of remaining possible Echo votes. A malicious party can exploit this by sending its Echo message with a conflicting value to a subset of honest parties (causing their early-exit check to fire and abort) while sending a consistent Echo to the remaining honest parties (allowing them to reach the echo threshold and proceed). This splits honest parties into those that abort with `AssertionFailed` and those that proceed — violating the echo broadcast safety invariant. Because `do_broadcast` is called directly inside `do_keyshare` (DKG and reshare), this translates to a permanent denial of key generation for the aborted parties.

---

### Finding Description

**Threshold computation for n=4:** [1](#0-0) 

For n=4: `broadcast_threshold = (4-1)/3 = 1`, `echo_threshold = usize::midpoint(4,1) = 2`. The protocol requires `count > echo_t` (i.e., ≥ 3 Echo votes) to proceed to Ready, and `count > 2*ready_t` (i.e., ≥ 3 Ready votes) to deliver.

**The vulnerable early-exit check:** [2](#0-1) 

The check computes `non_received_echo_cnt = n - received_echo_cnt` and aborts if no value can possibly reach `echo_t`. The flaw: `n` includes the malicious party. A malicious party can send its Echo to some honest parties (with a conflicting value, exhausting their `non_received_echo_cnt` budget) while withholding or sending a consistent Echo to others.

**Concrete attack (n=4: H1, H2, H3 honest; M malicious, M is the session sender):**

M sends:
- `Send(true)` to H1, H2 → they echo `Echo(true)` to all
- `Send(false)` to H3 → H3 echoes `Echo(false)` to all
- `Echo(false)` to H1 and H3 (conflicting)
- `Echo(true)` to H2 (consistent)

**H1 trace:**
1. `Echo(true)` [self]: count(true)=1, received=1, non_received=3 → 1+3=4 > 2 → ok
2. `Echo(true)` [H2]: count(true)=2, received=2, non_received=2 → 2+2=4 > 2 → ok
3. `Echo(false)` [H3]: count(false)=1, received=3, non_received=1 → true: 2+1=3 > 2 → ok
4. `Echo(false)` [M]: count(false)=2, received=4, non_received=0 → true: 2+0=2 **not** > 2; false: 2+0=2 **not** > 2 → **`AssertionFailed` abort**

**H2 trace:**
1. `Echo(true)` [self]: count(true)=1 → ok
2. `Echo(true)` [H1]: count(true)=2 → ok
3. `Echo(true)` [M]: count(true)=3 → **3 > 2 → sends `Ready(true)`, sets `finish_echo=true`**
4. `Echo(false)` [H3]: `finish_echo=true` → skipped

H2 is now in the Ready phase. It can only collect Ready votes from itself and M (2 votes), which is not > 2·ready_t=2, so H2 **hangs indefinitely**.

**H3 trace:** Same as H1 — receives 2 `Echo(true)` + 2 `Echo(false)` with non_received=0 → **`AssertionFailed` abort**.

**Result for n=4:** H1 and H3 abort; H2 hangs. The invariant "all honest parties either all deliver the same value or all abort" is violated.

**For n=5 (1 malicious party, same attack pattern):** `echo_t=3`, `ready_t=1`. M sends `Echo(false)` to H1 and H4 (causing them to abort), and `Echo(true)` to H2 and H3 (allowing them to reach 4 Echo(true) votes, send Ready(true), and collect 3 Ready(true) votes from self+H3/H2+M). H2 and H3 **deliver** `true` while H1 and H4 **abort** — a full safety violation.

**DKG impact:** `do_broadcast` is called three times inside `do_keyshare`: [3](#0-2) [4](#0-3) [5](#0-4) 

A broadcast abort at any of these call sites causes `do_keyshare` to return `Err`, meaning the affected honest party produces no `KeygenOutput`. Other honest parties that completed the broadcast proceed and produce a `KeygenOutput`. The resulting participant set is permanently inconsistent.

---

### Impact Explanation

- **n=4:** H1 and H3 abort with `AssertionFailed`; H2 hangs indefinitely. No honest party can complete DKG. **High: Permanent denial of key generation.**
- **n≥5:** A strict subset of honest parties abort while others deliver and complete DKG with a `KeygenOutput`. The threshold assumption is broken because the set of parties holding valid key shares is smaller than expected. **High: Corruption of DKG outputs / inconsistent participant sets.**

---

### Likelihood Explanation

The attack requires only that the malicious party can send different messages to different participants — a standard capability for any Byzantine participant. No cryptographic assumption needs to be broken. The malicious party controls only its own message delivery. The attack is deterministic and reproducible.

---

### Recommendation

Remove or correct the early-exit check. The check incorrectly assumes all `n` parties will eventually contribute an Echo. The correct bound for "remaining possible honest echos" is `(n - f) - received_honest_echo_cnt`, where `f` is the maximum number of malicious parties (`broadcast_threshold`). Alternatively, remove the optimization entirely and rely on the standard Bracha protocol's timeout/liveness mechanism, which does not introduce this asymmetric abort risk. [2](#0-1)

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
