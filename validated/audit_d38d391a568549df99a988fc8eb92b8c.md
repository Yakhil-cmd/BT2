### Title
Unregistered Computation Kind Silently Bypasses Execution Fee Metering - (File: `fvm/meter/computation_meter.go`)

### Summary
`MeterComputation()` in the FVM's weighted computation meter performs a registry lookup of `computationWeights[kind]`. When a computation kind is absent from the registry, the function silently returns `nil` — contributing **zero** to `computationUsed`. Because `computationUsed` is the execution effort passed directly to `DeductTransactionFees`, any operation whose kind is not registered in the on-chain weights map is executed for free. An unprivileged transaction sender can craft a transaction that heavily exercises such unregistered kinds (e.g., `ComputationKindEVMEncodeABI`, `ComputationKindEVMDecodeABI`, or any newly introduced kind not yet added to the on-chain weights) and pay only the flat inclusion fee regardless of actual resource consumption.

### Finding Description

`MeterComputation` in `fvm/meter/computation_meter.go` performs a map lookup against the active `computationWeights` registry:

```go
w, ok := m.params.computationWeights[kind]
if !ok {
    return nil          // ← zero computation charged, no error
}
m.computationUsed += w * intensity
```

The companion functions exhibit the same pattern:

- `ComputationAvailable`: returns `true` (unlimited capacity) when the kind is absent.
- `ComputationRemaining`: returns `math.MaxUint64` when the kind is absent or has weight 0.

The active weights are loaded from on-chain state and merged with `DefaultComputationWeights`, which contains only three kinds (`Statement`, `Loop`, `FunctionInvocation`). All other computation kinds — including every FVM-specific kind defined in `fvm/environment/meter.go` (e.g., `ComputationKindEVMGasUsage` = 2037, `ComputationKindEVMEncodeABI` = 2042, `ComputationKindEVMDecodeABI` = 2043, `ComputationKindHash` = 2001, `ComputationKindVerifySignature` = 2002, etc.) — are absent from `DefaultComputationWeights` and are only present if the on-chain weights registry has been explicitly updated to include them.

`DefaultExecutionMeteringParameters()` in `state/protocol/kvstore.go` initialises the on-chain weights with an **empty map**, meaning any network that has not explicitly set weights for a given kind treats that kind as free. When a new computation kind is introduced in a software upgrade but the corresponding governance transaction to register its weight has not yet been executed, every operation of that kind is metered at zero cost.

The total `computationUsed` is then passed as `executionEffort` to `DeductTransactionFees`:

```go
_, err = executor.env.DeductTransactionFees(
    executor.proc.Transaction.Payer,
    executor.proc.Transaction.InclusionEffort(),
    computationUsed)   // ← zero if all kinds were unregistered
```

A transaction that performs only unregistered-kind operations pays only the inclusion fee, regardless of actual work performed.

### Impact Explanation

An unprivileged transaction sender can submit a Cadence transaction that exclusively invokes operations whose computation kinds are absent from the on-chain weights registry. The execution effort reported to `DeductTransactionFees` is zero, so the payer is charged only the flat inclusion fee. For computationally expensive operations — particularly EVM calls metered via `ComputationKindEVMGasUsage`, or ABI encode/decode operations metered via `ComputationKindEVMEncodeABI`/`ComputationKindEVMDecodeABI` — this represents a severe undercharge relative to actual resource consumption, reducing protocol fee revenue and potentially enabling resource exhaustion at minimal cost.

### Likelihood Explanation

The condition is triggered whenever any computation kind used by the runtime is absent from the active on-chain weights. This is a structural property of the design: `DefaultComputationWeights` covers only three kinds, and `DefaultExecutionMeteringParameters` initialises with an empty map. Every new computation kind introduced in a software upgrade creates a window — between the software deployment and the governance transaction that registers the new weight — during which that kind is free. The EVM-specific kinds (`ComputationKindEVMGasUsage`, `ComputationKindEVMEncodeABI`, `ComputationKindEVMDecodeABI`) are particularly high-value targets because EVM operations can be arbitrarily expensive in gas terms.

### Recommendation

1. Change `MeterComputation` to treat an absent weight as a non-zero sentinel (e.g., the maximum weight or a configurable default) rather than silently returning `nil`. This mirrors the principle of fail-safe defaults.
2. Alternatively, maintain a compile-time exhaustive mapping that requires every defined `ComputationKind` to have an explicit weight entry, so that adding a new kind without a weight is a build error.
3. Ensure `DefaultExecutionMeteringParameters` is populated with at minimum the `DefaultComputationWeights` entries rather than an empty map, so the fallback is never fully unmetered.
4. Document the governance dependency explicitly: any software upgrade that introduces a new `ComputationKind` must be accompanied by a weight-registration transaction executed atomically with or before the upgrade activation.

func (m *ComputationMeter) MeterComputation(usage common.ComputationUsage) error {
	kind := usage.Kind
	intensity := usage.Intensity

	m.computationIntensities[kind] += intensity
	w, ok := m.params.computationWeights[kind]
	if !ok {
		return nil
	}
	m.computationUsed += w * intensity
	if m.computationUsed > m.params.computationLimit {
		return errors.NewComputationLimitExceededError(
			uint64(m.params.TotalComputationLimit()))
	}
	return nil
}

func (executor *transactionExecutor) deductTransactionFees() (err error) {
	if !executor.env.TransactionFeesEnabled() {
		return nil
	}

	computationLimit := executor.txnState.TotalComputationLimit()

	computationUsed, err := executor.env.ComputationUsed()
	if err != nil {
		return errors.NewTransactionFeeDeductionFailedError(
			executor.proc.Transaction.Payer,
			computationLimit,
			err)
	}

	if computationUsed > computationLimit {
		computationUsed = computationLimit
	}

	_, err = executor.env.DeductTransactionFees(
		executor.proc.Transaction.Payer,
		executor.proc.Transaction.InclusionEffort(),
		computationUsed)

	if err != nil {
		return errors.NewTransactionFeeDeductionFailedError(
			executor.proc.Transaction.Payer,
			computationUsed,
			err)
	}
	return nil

// checkGasLimit checks if enough computation is left in the environment

func (h *ContractHandler) checkGasLimit(limit types.GasLimit) error {
	// check gas limit against what has been left on the transaction side
	usage := common.ComputationUsage{
		Kind:      environment.ComputationKindEVMGasUsage,
		Intensity: uint64(limit),
	}
	if !h.backend.ComputationAvailable(usage) {
		return types.ErrInsufficientComputation
	}
	return nil
}

// decodeTransaction decodes RLP encoded transaction payload and meters the resources used.
func (h *ContractHandler) decodeTransaction(encodedTx []byte) (*gethTypes.Transaction, error) {
	usage := common.ComputationUsage{
		Kind:      environment.ComputationKindRLPDecoding,
		Intensity: uint64(len(encodedTx)),
	}
	err := h.backend.MeterComputation(usage)
	if err != nil {
		return nil, err
	}

	tx := gethTypes.Transaction{}
	if err := tx.UnmarshalBinary(encodedTx); err != nil {
		return nil, err
	}

	return &tx, nil
}

func (h *ContractHandler) meterGasUsage(res *types.Result) error {
	usage := common.ComputationUsage{
		Kind:      environment.ComputationKindEVMGasUsage,
		Intensity: res.GasConsumed,
	}
	return h.backend.MeterComputation(usage)
}
