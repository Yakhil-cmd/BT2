### Title
Missing proof-of-correctness on participant CKD contributions allows silent output corruption — (`src/confidential_key_derivation/protocol.rs`)

### Summary
A malicious participant can send an arbitrary `CKDOutput` (e.g., with `norm_big_c` negated) to the coordinator. The coordinator aggregates all contributions without any validation, producing a `CKDOutput` that decodes to the wrong confidential key. Honest parties accept and return this corrupted output.

### Finding Description
In `do_ckd_coordinator`, the coordinator collects `(norm_big_y, norm_big_c)` from every other participant via `recv_from_others::<CKDOutput>` and sums them directly:

```rust
// protocol.rs lines 50–55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

No zero-knowledge proof, commitment, or consistency check is attached to or verified for the received values. A malicious participant `i` can replace their honest contribution `(λ_i·Y_i, λ_i·C_i)` with `(λ_i·Y_i, -(λ_i·C_i))`. The coordinator's aggregated `big_c` becomes:

```
big_c_agg = (Σ_{j≠i} λ_j·C_j) - λ_i·C_i
          = correct_big_c - 2·λ_i·C_i
```

This is not equal to `msk·H(pk, app_id) + a·Y`, so `unmask(app_sk)` returns a wrong value. The app receives a `CKDOutput` that fails BLS signature verification, and the correct confidential key is unrecoverable from this run.

### Impact Explanation
**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator outputs a structurally valid `CKDOutput` (no error is raised), but `unmask(app_sk)` returns a garbage group element that is not `msk·H(pk, app_id)`. The app cannot derive its confidential key. Any single malicious participant among the `n` participants can trigger this on every CKD invocation they participate in.

### Likelihood Explanation
Any non-coordinator participant can execute this attack. The attack requires only that the participant deviate from the protocol by negating (or otherwise modifying) their `norm_big_c` before sending it. No cryptographic assumption needs to be broken. The coordinator has no mechanism to detect or attribute the deviation.

### Recommendation
Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that `norm_big_c = λ_i·(x_i·H(pk, app_id) + y_i·app_pk)` and `norm_big_y = λ_i·y_i·G`, consistent with the participant's committed public share. The coordinator must verify all proofs before aggregating. A Chaum-Pedersen-style DLEQ proof over G1 would suffice to bind `norm_big_y` and `norm_big_c` to the participant's known public key share.

```rust
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
```

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
```
