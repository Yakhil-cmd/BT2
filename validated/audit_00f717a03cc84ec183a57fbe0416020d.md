### Title
Global `last_reward_block` Manipulated via Unprivileged `update_rewards(disable_rewards: true)` Call — (File: `src/staking/staking.cairo`)

### Summary
Any unprivileged caller can invoke `update_rewards` with an arbitrary `staker_address` and `disable_rewards: true`. Because `last_reward_block` is a **global** (not per-staker) storage variable, this single call consumes the entire block's reward slot without distributing any rewards, permanently denying the targeted staker their consensus-block rewards for that block. Repeated across every block, this permanently freezes a staker's unclaimed yield.

### Finding Description

`update_rewards` in `src/staking/staking.cairo` is a public function gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. No check restricts who may call it or what value `disable_rewards` may take. [1](#0-0) 

The function immediately writes the current block number into the **global** `last_reward_block` field: [2](#0-1) 

Then, if `disable_rewards` is `true`, it returns **before** distributing any rewards: [3](#0-2) 

Because `last_reward_block` is a single global slot shared across all stakers, the guard at the top of the function prevents any second call in the same block: [4](#0-3) 

The storage declaration confirms `last_reward_block` is not keyed per staker: [5](#0-4) 

Block rewards are computed per-block with no catch-up mechanism; a skipped block's rewards are permanently lost: [6](#0-5) 

### Impact Explanation

An attacker who calls `update_rewards(staker_address: victim, disable_rewards: true)` every block:

1. Sets `last_reward_block` to the current block number.
2. Skips reward distribution entirely.
3. Blocks the victim (and every other staker) from calling `update_rewards` again in that block.
4. Causes the victim to permanently lose consensus block rewards for every targeted block.

This constitutes **permanent freezing of unclaimed yield** — a HIGH-severity impact under the allowed scope.

### Likelihood Explanation

`update_rewards` is a fully public, permissionless entry point. Any EOA or contract can call it with `disable_rewards: true` targeting any staker. No special role, leaked key, or privileged access is required. The only cost is gas per block, making sustained griefing economically feasible, especially against high-value stakers.

### Recommendation

1. **Restrict `disable_rewards: true`** to a privileged role (e.g., `security_agent`) or remove the parameter from the public interface entirely.
2. **Scope `last_reward_block` per staker** (i.e., `Map<ContractAddress, BlockNumber>`) so that one call for staker A cannot block staker B from receiving rewards in the same block.
3. Alternatively, require `get_caller_address() == staker_address || get_caller_address() == staker_info.reward_address` before allowing `disable_rewards: true`.

### Proof of Concept

```
// Attacker runs this every block to permanently freeze victim's yield:
loop {
    staking_contract.update_rewards(
        staker_address: victim_address,
        disable_rewards: true,   // no access control prevents this
    );
    // last_reward_block is now set to current block
    // victim calls update_rewards → reverts with REWARDS_ALREADY_UPDATED
    // victim's block rewards are permanently lost
    wait_for_next_block();
}
```
