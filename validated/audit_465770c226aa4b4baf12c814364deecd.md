### Title
Unimplemented `TurnBridge` Handler Bypasses Authorization and Permanently Disables Bridge Circuit Breaker - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is a stub that unconditionally returns `nil, nil` — it performs no authorization check and makes no state change. Any unprivileged user can call `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` permission system entirely. Simultaneously, no authorized party (admin or permissioned address) can ever actually disable the gravity bridge, permanently breaking the emergency circuit-breaker.

### Finding Description
`MsgTurnBridge` is a registered Cosmos SDK message with a defined permission bit (`CanTurnBridge`) and a dedicated permission system (ADR-009). The proto definition marks `sender` as the signer field, but the keeper implementation is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly gates execution behind `HasPermission`:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The `CanTurnBridge` permission constant is defined and the permission system is fully wired: [3](#0-2) 

ADR-009 explicitly states that `MsgTurnBridge` is one of the two messages that must require permission: [4](#0-3) 

The integration test `test_gravity_turn_bridge` confirms the intended behavior — community (unpermissioned) should fail, validator (permissioned) should succeed — but the stub implementation makes both succeed with no effect: [5](#0-4) 

### Impact Explanation
**High — Bypass of Cronos admin/permission authorization checks:**
Any unprivileged user can submit `MsgTurnBridge{sender: attacker, enable: false}` and receive a `code: 0` success response. The `CanTurnBridge` permission check is never evaluated, so the entire permission system for this message is bypassed.

**High — Permanent inability for authorized users to execute bridge circuit-breaker:**
Because the handler performs no state change, the gravity bridge can never be turned off by anyone — including the `CronosAdmin` or addresses holding `CanTurnBridge` permission. In an emergency (e.g., a bridge exploit), the intended safety mechanism is permanently non-functional.

### Likelihood Explanation
The entry path is a standard signed Cosmos SDK transaction requiring only a valid account and gas. No privileged keys, leaked secrets, or special conditions are needed. The `CmdTurnBridge` CLI command is publicly documented and callable by any user. [6](#0-5) 

### Recommendation
Implement `TurnBridge` with the same pattern as `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if the check fails.
2. Implement the actual bridge enable/disable state change (e.g., set a bridge-enabled flag in the cronos module params or interact with the gravity bridge keeper).

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    // TODO: persist bridge enabled/disabled state and enforce in gravity hooks
    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept
1. Create any funded Cronos account `attacker` (no admin key, no `CanTurnBridge` permission).
2. Submit: `cronosd tx cronos turn-bridge false --from attacker`
3. Observe: transaction succeeds with `code: 0` — no `ErrUnauthorized` is returned.
4. Observe: the gravity bridge continues to process batches normally — no state was changed.
5. Submit the same tx from the `CronosAdmin` address: also succeeds with `code: 0`, but the bridge is still not disabled.
6. Confirm: `cronosd query cronos params` shows no bridge-disabled flag; gravity bridge batches continue to be created. [1](#0-0) [7](#0-6)

// HasPermission check if an account has a specific permission. by default cronos admin has all permissions
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```

// CmdTurnBridge returns a CLI command handler for enable or disable the bridge
func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{
		Use:   "turn-bridge [true/false]",
		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}
