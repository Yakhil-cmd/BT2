### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Over-Minting and Bridge Insolvency - (`engine/src/engine.rs`)

### Summary

When a NEP-141 token with a fee-on-transfer mechanism is bridged into Aurora via `ft_on_transfer`, the engine mints ERC-20 mirror tokens equal to the **claimed** transfer amount (`args.amount`) rather than the **actual** amount received. This creates a permanent accounting mismatch: the ERC-20 mirror supply exceeds Aurora's real NEP-141 balance, making the bridge insolvent for that token and permanently freezing the last users' funds.

---

### Finding Description

The bridge entry flow is:

1. A user calls `ft_transfer_call` on a NEP-141 token contract, specifying Aurora as the receiver and an EVM address in the message.
2. The NEP-141 contract deducts a fee, transfers `amount - fee` tokens to Aurora, then calls `ft_on_transfer` on Aurora with the **original** `amount` (not the net received amount). This is standard NEP-141 behavior.
3. Aurora's `ft_on_transfer` entrypoint reads `args.amount` from the callback arguments and passes it directly to `receive_erc20_tokens`.
4. `receive_erc20_tokens` mints exactly `args.amount` ERC-20 tokens to the recipient — the full pre-fee amount — with no check against Aurora's actual NEP-141 balance change.

The root cause is in `receive_erc20_tokens`:

The `setup_receive_erc20_tokens_input` function encodes a `mint(recipient, amount)` call to the `EvmErc20` contract using this unchecked amount: [2](#0-1) 

The `EvmErc20.sol` `mint` function unconditionally mints the requested amount: [3](#0-2) 

On exit, `ExitToNear` burns the ERC-20 tokens and calls `ft_transfer` on the NEP-141 contract for the same burned amount: [4](#0-3) 

Since Aurora's actual NEP-141 balance is `amount - fee` but the ERC-20 supply is `amount`, the bridge is insolvent by the fee amount for every such deposit.

---

### Impact Explanation

**Insolvency / Permanent fund freeze (Critical/High).**

Aurora's ERC-20 mirror token supply for the affected NEP-141 exceeds the actual NEP-141 tokens Aurora holds. Users who exit first receive their full amount. The last users to exit — holding the "phantom" over-minted tokens — will have their `ft_transfer` calls on the NEP-141 contract fail because Aurora's balance is insufficient. Those ERC-20 tokens become permanently unwithdrawable: the user cannot exit to NEAR (insufficient NEP-141 balance) and cannot exit to Ethereum (same backing issue). The funds are permanently frozen in the EVM mirror.

The insolvency compounds with every fee-on-transfer deposit: each deposit widens the gap between ERC-20 supply and NEP-141 backing.

---

### Likelihood Explanation

**Medium.** Any NEP-141 token can be bridged to Aurora — there is no allowlist restricting which tokens can be registered via `deploy_erc20_token` or received via `ft_on_transfer`. An unprivileged user can trigger this by simply calling `ft_transfer_call` on any fee-on-transfer NEP-141 token with Aurora as the receiver. No admin access, no governance capture, and no special privileges are required. The attacker-controlled entry path is the standard public bridge deposit flow. [5](#0-4) 

---

### Recommendation

In `receive_erc20_tokens`, do not trust `args.amount` as the mint quantity. Instead, query Aurora's NEP-141 balance before and after the transfer (or use the return value of the NEP-141 `ft_transfer_call` resolution) to determine the actual net received amount, and mint only that amount. Alternatively, document and enforce that only NEP-141 tokens without fee-on-transfer mechanisms may be bridged, and add an on-chain check or registry to enforce this invariant.

---

### Proof of Concept

1. Deploy a NEP-141 token that charges a 10% fee on every transfer (deducted from the transferred amount before crediting the receiver).
2. Call `ft_transfer_call` on the NEP-141 with `amount = 1000`, `receiver_id = aurora`, `msg = <evm_address_hex>`.
3. The NEP-141 contract credits Aurora with `900` tokens (after 10% fee) and calls `ft_on_transfer` on Aurora with `args.amount = 1000`.
4. Aurora's `ft_on_transfer` → `receive_erc20_tokens` reads `amount = 1000` and mints `1000` ERC-20 mirror tokens to the EVM address.
5. Aurora's actual NEP-141 balance: `900`. ERC-20 mirror supply: `1000`. Gap: `100` tokens.
6. Repeat step 2–4 ten times. Aurora holds `9000` NEP-141 tokens but has minted `10000` ERC-20 tokens.
7. The first 9000 ERC-20 tokens can be successfully exited via `withdrawToNear`. The remaining 1000 ERC-20 tokens cannot be exited — Aurora's `ft_transfer` call on the NEP-141 will fail with insufficient balance. Those 1000 ERC-20 tokens are permanently frozen. [6](#0-5) [7](#0-6)

```rust
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self

                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());
```

```rust
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
```
