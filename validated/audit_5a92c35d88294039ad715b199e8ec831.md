### Title
Weak `IsValidIBCDenom` validation allows governance to set a non-hex `IbcCroDenom` that permanently breaks all CRO-to-IBC bridge transfers — (`x/cronos/types/types.go`, `x/cronos/types/params.go`)

### Summary
`IsValidIBCDenom` only validates string length and the `"ibc/"` prefix, but does not verify that the 64-character suffix is valid hexadecimal. A governance proposal can set `IbcCroDenom` to a syntactically conforming but non-hex value (e.g., `"ibc/" + "G"*64`) that passes `Validate()` and `SetParams`, but causes `GetSourceChannelID` to fail at runtime when `transferTypes.ParseHexHash` is called, permanently breaking all CRO-to-IBC bridge transfers.

### Finding Description
`IsValidIBCDenom` in `x/cronos/types/types.go` checks only length and prefix: [1](#0-0) 

`validateIsIbcDenom` in `x/cronos/types/params.go` delegates entirely to this function: [2](#0-1) 

`SetParams` calls `params.Validate()` which calls `validateIsIbcDenom`, so a value like `"ibc/" + "G"*64` is accepted and written to the KV store: [3](#0-2) 

At runtime, `IbcTransferCoins` constructs an `ibcCoin` using the stored (malicious) `IbcCroDenom` and passes it to `ibcSendTransfer`: [4](#0-3) 

`ibcSendTransfer` calls `GetSourceChannelID` on the malicious denom: [5](#0-4) 

`GetSourceChannelID` calls `transferTypes.ParseHexHash` on the non-hex suffix, which fails: [6](#0-5) 

The error propagates back through `ibcSendTransfer` → `IbcTransferCoins` → `TransferTokens` / `SendCroToIbcHandler`, causing every CRO-to-IBC transfer to revert deterministically.

### Impact Explanation
After the malicious governance proposal passes, every call to `MsgTransferTokens` with the EVM denom and every `SendCroToIbc` EVM hook invocation fails deterministically. The CRO bridge path is completely and permanently disabled until a corrective governance proposal is passed. This is a **High** impact: long-lived inability for honest users to process valid bridge/conversion flows.

### Likelihood Explanation
The attack requires a governance proposal to pass. This is a meaningful barrier (requires majority voting power or social engineering), but the proposal path is an explicitly in-scope entry point. The bug is a straightforward validation gap in production code, not a theoretical edge case.

### Recommendation
Strengthen `IsValidIBCDenom` to also validate that the 64-character suffix is valid hexadecimal, aligning it with what `GetSourceChannelID` requires at runtime:

```go
func IsValidIBCDenom(denom string) bool {
    if len(denom) != ibcDenomLen || !strings.HasPrefix(denom, ibcDenomPrefix) {
        return false
    }
    suffix := denom[len(ibcDenomPrefix):]
    _, err := hex.DecodeString(suffix)
    return err == nil
}
```

Alternatively, call `transferTypes.ParseHexHash` directly inside `validateIsIbcDenom` to ensure the stored value is always parseable by `GetSourceChannelID`.

### Proof of Concept
1. Submit `MsgUpdateParams` via governance with `IbcCroDenom = "ibc/" + strings.Repeat("G", 64)`.
2. Proposal passes; `SetParams` succeeds because `IsValidIBCDenom` only checks length and prefix.
3. User calls `MsgTransferTokens` with EVM denom coins.
4. `IbcTransferCoins` constructs `ibcCoin` with the malicious denom and calls `ibcSendTransfer`.
5. `GetSourceChannelID` calls `ParseHexHash("GGG...")` → error.
6. Transaction reverts; all subsequent CRO bridge transfers fail identically.
7. Bridge remains broken until a new governance proposal sets a valid hex `IbcCroDenom`.
