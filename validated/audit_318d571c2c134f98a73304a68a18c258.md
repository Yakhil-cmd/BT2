### Title
Permanent Fund Loss in `ExitToNear` Precompile: Hardcoded Fixed Gas With No Refund Fallback When `error_refund` Feature Is Disabled — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile attaches hardcoded, fixed NEAR gas amounts to cross-contract `ft_transfer` / `ft_transfer_call` promise calls. When the `error_refund` Cargo feature is disabled (the default), any failure of those NEAR-side promises leaves user funds permanently destroyed: ERC-20 tokens are burned (or ETH is deducted from the EVM balance) before the NEAR transfer is attempted, and there is no callback to refund the user on failure. This is the direct Aurora-Engine analog of the Solidity `transfer()` / fixed-2300-gas vulnerability: a hard dependency on a fixed gas budget, combined with no recovery path when that budget is exceeded or the call fails for any reason.

---

### Finding Description

**Root cause 1 — hardcoded gas constants**

In `engine-precompiles/src/native.rs`, the `costs` module defines fixed NEAR gas amounts for every outbound promise:

```rust
// engine-precompiles/src/native.rs  lines 42-62
mod costs {
    // TODO(#483): Determine the correct amount of gas
    pub(super) const EXIT_TO_NEAR_GAS: EthGas = EthGas::new(0);
    // TODO(#483): Determine the correct amount of gas
    pub(super) const EXIT_TO_ETHEREUM_GAS: EthGas = EthGas::new(0);

    pub(super) const FT_TRANSFER_GAS: NearGas = NearGas::new(10_000_000_000_000);
    pub(super) const FT_TRANSFER_CALL_GAS: NearGas = NearGas::new(70_000_000_000_000);
    pub(super) const EXIT_TO_NEAR_CALLBACK_GAS: NearGas = NearGas::new(10_000_000_000_000);
    // TODO(#332): Determine the correct amount of gas
    pub(super) const WITHDRAWAL_GAS: NearGas = NearGas::new(100_000_000_000_000);
}
```

These values are baked in at compile time. The `ft_transfer` promise is constructed with `attached_gas: FT_TRANSFER_GAS` (10 TGas):

```rust
// engine-precompiles/src/native.rs  lines 456-468
let attached_gas = if method == "ft_transfer_call" {
    costs::FT_TRANSFER_CALL_GAS
} else {
    costs::FT_TRANSFER_GAS
};

let transfer_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method,
    args: args.into_bytes(),
    attached_balance: Yocto::new(1),
    attached_gas,
};
```

If NEAR Protocol raises the gas cost of any operation inside `ft_transfer` (exactly as Ethereum's Istanbul hardfork raised `SLOAD` costs), the fixed 10 TGas budget will be exhausted and the promise will fail.

**Root cause 2 — refund mechanism gated behind an opt-in feature flag**

The callback that would refund the user on failure is compiled out unless the `error_refund` Cargo feature is explicitly enabled:

```rust
// engine-precompiles/src/native.rs  lines 449-455
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← no refund path in the default build
    transfer_near: transfer_near_args,
};
```

When `error_refund` is disabled and there is no `transfer_near` (i.e., for all plain ERC-20 and ETH exits), `callback_args` equals the default value, so the promise is scheduled **without any callback**:

```rust
// engine-precompiles/src/native.rs  lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback; failure is silent
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,
            ...
        },
    })
};
```

The `error_refund` feature is opt-in (not a default feature) in both `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`, so the production WASM binary ships without it unless explicitly requested.

**Combined effect**

1. A user calls the `ExitToNear` precompile (address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) from within the EVM.
2. The precompile burns the user's ERC-20 tokens (or deducts ETH from their EVM balance) immediately, before the NEAR promise executes.
3. A NEAR `ft_transfer` promise is scheduled with a fixed 10 TGas budget and no error-handling callback.
4. If the promise fails for any reason — insufficient gas due to a NEAR protocol gas-cost increase, recipient not registered with the NEP-141 contract, or any other NEAR-side error — the funds are gone: burned on the EVM side, never delivered on the NEAR side.

---

### Impact Explanation

**Permanent freezing / destruction of user funds.** The ERC-20 tokens (or ETH) are irreversibly removed from the user's EVM balance at the point the precompile runs. If the subsequent NEAR promise fails and the `error_refund` feature is absent, there is no on-chain path to recover those assets. This matches the "permanent freezing of funds" impact category.

---

### Likelihood Explanation

**Medium-High.** The `error_refund` feature is disabled in the default production build. Any of the following conditions — all reachable by an ordinary EVM user — triggers the loss:

- Specifying a NEAR recipient account that is not registered (has no storage deposit) with the target NEP-141 contract. The `ft_transfer` call will panic on the NEAR side and the promise will fail.
- A future NEAR Protocol gas-cost increase (analogous to EIP-1884 / Istanbul) that pushes the actual gas consumption of `ft_transfer` above the hardcoded 10 TGas ceiling.
- Any transient or permanent failure of the target NEP-141 contract.

The test suite explicitly documents the first scenario and acknowledges the fund loss:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 656-660
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
// ↑ FT_EXIT_AMOUNT is permanently lost
```

---

### Recommendation

1. **Enable `error_refund` unconditionally** in the production engine build, or promote it to a default feature, so that the `exit_to_near_precompile_callback` always carries a refund path.
2. **Replace hardcoded gas constants** with dynamic calculation (e.g., forward a fraction of the remaining prepaid gas, as `return_promise` already does in `engine/src/contract_methods/connector.rs` via `calculate_attached_gas`), so the budget is not frozen at compile time.
3. **Add a no-callback guard**: if `error_refund` is intentionally disabled, at minimum reject the precompile call before burning any tokens rather than silently losing them on NEAR-side failure.

---

### Proof of Concept

**Entry path (unprivileged EVM user):**

1. User holds ERC-20 tokens on Aurora that map to a NEP-141.
2. User calls the ERC-20's `withdrawToNear` function (or directly calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) specifying an unregistered NEAR account as recipient.
3. The precompile burns the ERC-20 tokens immediately.
4. A NEAR `ft_transfer` promise with 10 TGas is scheduled to the NEP-141 contract.
5. The NEP-141 `ft_transfer` panics because the recipient has no storage deposit → promise fails.
6. Because `error_refund` is disabled, `callback_args.refund == None` and no callback was attached → no refund is issued.
7. User's tokens are permanently destroyed.
