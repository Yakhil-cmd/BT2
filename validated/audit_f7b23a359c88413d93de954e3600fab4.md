### Title
Hardcoded `Fee(1)` placeholder in consensus L1Handler conversion produces wrong `paid_fee_on_l1` in every validator-executed L1Handler transaction — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary
`convert_consensus_l1_handler_to_internal_l1_handler` unconditionally supplies `Fee(1)` as `paid_fee_on_l1` for every `L1Handler` transaction that arrives through the consensus (P2P) path. The actual fee paid on L1 is discarded. The wrong value propagates into the blockifier fee-check, the `CentralL1HandlerTransaction` Cende record, and any downstream OS/proof pipeline that reads `paid_fee_on_l1`.

### Finding Description

`convert_consensus_tx_to_internal_consensus_tx` dispatches `ConsensusTransaction::L1Handler` to `convert_consensus_l1_handler_to_internal_l1_handler`: [1](#0-0) 

That helper calls `L1HandlerTransaction::create` with a hardcoded stub: [2](#0-1) 

`L1HandlerTransaction::create` stores the value verbatim in the `paid_fee_on_l1` field: [3](#0-2) 

The resulting `L1HandlerTransaction` (with `paid_fee_on_l1 = Fee(1)`) is then:

1. **Executed by the blockifier**, where the only fee guard is `paid_fee == Fee(0)` — `Fee(1)` trivially passes regardless of the actual L1 fee: [4](#0-3) 

2. **Serialized into `CentralL1HandlerTransaction`** for the Cende pre-confirmed block record, embedding the wrong fee: [5](#0-4) 

By contrast, the `native_blockifier` Python path correctly reads the real `paid_fee_on_l1` from the transaction object: [6](#0-5) 

### Impact Explanation

Every validator node that processes a consensus proposal containing an `L1Handler` transaction will execute it with `paid_fee_on_l1 = Fee(1)` instead of the actual L1 fee. This produces:

- **Wrong Cende record**: `CentralL1HandlerTransaction.paid_fee_on_l1` is `1` instead of the real value. Any OS/proof pipeline that reads this field to reconstruct or verify the L1 message will see incorrect data, potentially producing a wrong or unverifiable proof.
- **Fee-check bypass**: The blockifier's only guard (`paid_fee != Fee(0)`) is trivially satisfied by the stub. A future tightening of that check (already anticipated by the TODO) would be silently bypassed on the validator path.
- **Proposer/validator divergence**: The proposer builds the transaction from the real L1 event (correct fee); every validator reconstructs it with `Fee(1)`. Any logic that compares or hashes `paid_fee_on_l1` across nodes will diverge.

### Likelihood Explanation

This is triggered unconditionally for every `L1Handler` transaction that passes through the consensus path (i.e., every L1→L2 message included in a consensus proposal). No special attacker action is required; normal L1→L2 messaging is sufficient.

### Recommendation

Include `paid_fee_on_l1` in the `ConsensusTransaction::L1Handler` variant so it is transmitted through consensus alongside the rest of the transaction data, and remove the `Fee(1)` stub. Until then, the validator path produces structurally wrong `paid_fee_on_l1` for every L1Handler transaction.

### Proof of Concept

1. An L1 user calls `sendMessageToL2` on the Starknet core contract, paying a real fee (e.g., `Fee(1_000_000_000)`).
2. The proposer detects the L1 event, creates an `L1HandlerTransaction` with `paid_fee_on_l1 = Fee(1_000_000_000)`, and includes it in a consensus proposal.
3. A validator receives the proposal and calls `convert_consensus_tx_to_internal_consensus_tx` → `convert_consensus_l1_handler_to_internal_l1_handler`.
4. The resulting executable transaction has `paid_fee_on_l1 = Fee(1)`.
5. The blockifier fee check passes (`Fee(1) != Fee(0)`).
6. The Cende record for this transaction contains `paid_fee_on_l1 = 0x1` instead of `0x3B9ACA00`.
7. Any OS/proof component that reads `paid_fee_on_l1` from the Cende record will operate on the wrong value.
