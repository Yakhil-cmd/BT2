### Title
`_afterTimelock` uint32 Truncation Silently Bypasses All Timelocked Parameter Updates in OracleValueStopLossExtension — (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

---

### Summary

`_afterTimelock` computes `uint32(block.timestamp + timelock)` without a checked cast. When the configured `timelock` is large enough that the sum exceeds `type(uint32).max`, the result silently wraps to a past timestamp. Every subsequent `propose*` call stores that past value as `executeAfter`, so the matching `execute*` call passes `_requireElapsed` immediately — the intended delay is zero regardless of the configured timelock value.

---

### Finding Description

The helper at the bottom of `OracleValueStopLossExtension`:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `oracleStopLossConfig[pool_].timelock` is `uint32`. The addition is performed in `uint256` (no overflow at that width), but the result is then narrowed to `uint32` via an explicit cast. Solidity 0.8+ does **not** protect against explicit narrowing casts — only implicit ones — so the truncation is silent.

With `block.timestamp ≈ 1,750,000,000` (year 2025), the wrap occurs whenever:

```
timelock > type(uint32).max − block.timestamp
         = 4,294,967,295 − 1,750,000,000
         = 2,544,967,295 seconds  (≈ 80 years)
```

A pool admin who sets `timelock = type(uint32).max` (≈ 136 years) — perhaps intending to mirror the factory's `priceProviderTimelock == type(uint256).max` immutability pattern — gets:

```
block.timestamp + timelock = 1,750,000,000 + 4,294,967,295 = 6,044,967,295
uint32(6,044,967,295)      = 6,044,967,295 mod 2^32        = 1,749,999,999
```

That is a timestamp **one second in the past**.

`_requireElapsed` then evaluates:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

`block.timestamp` (uint256 = 1,750,000,000) is compared against `executeAfter` widened to uint256 (= 1,749,999,999). The condition is false, so the revert is never triggered. Every timelocked operation — drawdown, decay, watermarks, and the timelock itself — can be executed in the same block as the proposal.

There is no validation of the `timelock` value in `initialize()`:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock: no bounds check
```

Nor in `proposeOracleStopLossTimelock`:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // ← wraps if current timelock is large
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
```

---

### Impact Explanation

`OracleValueStopLossExtension` is the primary LP protection mechanism. Its timelocks give LPs a window to exit before the pool admin can alter stop-loss parameters. With the bypass active, the admin can in a single block:

1. Call `proposeOracleStopLossDrawdown(pool, 0)` then `executeOracleStopLossDrawdown(pool)` — disabling the stop-loss entirely.
2. Call `proposeOracleStopLossDecay(pool, type(uint32).max)` then `executeOracleStopLossDecay(pool)` — collapsing all watermarks to zero on the next swap.
3. Call `proposeOracleStopLossHighWatermarks` then `executeOracleStopLossHighWatermarks` — resetting per-bin watermarks to arbitrary values.
4. Call `proposeOracleStopLossTimelock(pool, 0)` then `executeOracleStopLossTimelock(pool)` — permanently removing the delay for all future changes.

This breaks the core LP protection invariant of the extension: that parameter changes are delayed long enough for LPs to react.

---

### Likelihood Explanation

The trigger requires `timelock > ~80 years`. This is within the `uint32` range (max ≈ 136 years) and has no validation gate. A pool admin may set `type(uint32).max` intending to make parameters immutable — a natural analogy to the factory's `priceProviderTimelock == type(uint256).max` immutability mode. The `initialize()` call accepts any `uint32` value without bounds checking, so the misconfiguration is reachable through a normal (non-malicious) pool creation flow.

---

### Recommendation

Replace the unchecked narrowing cast with a checked conversion. The simplest fix:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    // SafeCast reverts on overflow, preventing silent wrap-around
    return SafeCast.toUint32(result);
}
```

Alternatively, cap the maximum accepted `timelock` in `initialize()` and `proposeOracleStopLossTimelock()` to a value that cannot overflow (e.g., `type(uint32).max / 2`).
