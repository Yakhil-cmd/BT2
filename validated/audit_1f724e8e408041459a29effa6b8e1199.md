Audit Report

## Title
`spawn_neuron` Pre-Check Uses Stale 5% Worst-Case Modulation While Mission 70 Allows −10% — (File: rs/nns/governance/src/governance.rs)

## Summary
`spawn_neuron` guards against spawning a neuron whose eventual ICP stake would fall below `neuron_minimum_stake_e8s` using a hardcoded 5% worst-case modulation factor. The Mission 70 algorithm, now live in the same codebase, defines a lower bound of −10% (−1,000 permyriad). The pre-check therefore under-estimates the worst case by a factor of two, allowing neurons to be spawned whose `cached_neuron_stake_e8s` will be set below `neuron_minimum_stake_e8s` once the actual modulation is applied seven days later.

## Finding Description

**Pre-check (stale constant):** [1](#0-0) 

The hardcoded `0.05` was correct when the CMC-based modulation range was ±500 permyriad (±5%). The Mission 70 algorithm defines a new range: [2](#0-1) 

`compute_maturity_modulation_permyriad` clamps its output to `[MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70, MATURITY_MODULATION_MAX_PERMYRIAD_MISSION_70]` = `[−1000, 200]`: [3](#0-2) 

`maybe_spawn_neurons` reads `current_value_permyriad` (which can be −1,000) and passes it directly to `apply_maturity_modulation`: [4](#0-3) [5](#0-4) 

`apply_maturity_modulation` performs no range validation — it accepts any `i32` and computes `amount * (10_000 + basis_points) / 10_000`. With −1,000 permyriad this yields `amount * 9_000 / 10_000`: [6](#0-5) 

The `MIN_MATURITY_MODULATION_PERMYRIAD = -500` constant in `maturity_modulation/mod.rs` is a documentation constant only — it is not enforced as a guard inside `apply_maturity_modulation`: [7](#0-6) 

**Exploit window:** Any `maturity_to_spawn` in the range `(neuron_minimum_stake_e8s / 0.90, neuron_minimum_stake_e8s / 0.95]` passes the pre-check (×0.95 ≥ minimum) but produces a below-minimum stake when modulation is −10% (×0.90 < minimum). With `neuron_minimum_stake_e8s = 100_000_000 e8s`, the vulnerable range is approximately `(111_111_112, 105_263_158]` — a ~6M e8s window.

After seven days `maybe_spawn_neurons` sets `neuron.cached_neuron_stake_e8s = neuron_stake` where `neuron_stake < neuron_minimum_stake_e8s`: [8](#0-7) 

## Impact Explanation

A neuron is permanently created in NNS governance with `cached_neuron_stake_e8s` below `neuron_minimum_stake_e8s`, violating the core governance invariant that all neurons must hold at least the minimum stake. This corrupts voting-power accounting for the affected neuron and undermines the integrity of NNS governance state. This matches the allowed impact: **Significant NNS governance security impact with concrete protocol harm** (High, $2,000–$10,000).

## Likelihood Explanation

The Mission 70 modulation is live and its range is [−10%, +2%]. The bug manifests whenever the daily modulation is between −5% and −10%, which is a realistic scenario during sustained ICP price declines. The entry path is a standard unprivileged `manage_neuron` → `Spawn` call requiring only control of a neuron with sufficient maturity. No special permissions, no social engineering, and no external dependencies are required.

## Recommendation

Replace the hardcoded `0.05` with the Mission 70 lower bound using integer arithmetic consistent with `apply_maturity_modulation`:

```rust
// rs/nns/governance/src/governance.rs, replacing line 2666
// Use Mission 70 lower bound: -1_000 permyriad → multiply by 9_000/10_000
let least_possible_stake = maturity_to_spawn
    .saturating_mul(
        (BASIS_POINTS_PER_UNITY as i64 + MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70) as u64
    )
    / BASIS_POINTS_PER_UNITY as u64;
```

This mirrors the exact arithmetic in `apply_maturity_modulation` and avoids floating-point imprecision. The constant `MATURITY_MODULATION_MIN_PERMYRIAD_MISSION_70` should be imported from `timer_tasks::update_icp_xdr_rate_related_data` so the pre-check stays in sync with the algorithm's bounds automatically.

## Proof of Concept

1. NNS Governance is running with Mission 70 maturity modulation at −1,000 permyriad (−10%).
2. Neuron A has `maturity_e8s_equivalent = 105_263_158` (just above `100_000_000 / 0.95 = 105_263_158`).
3. Controller calls `manage_neuron` → `Spawn { percentage_to_spawn: 100 }`.
4. `spawn_neuron` computes `least_possible_stake = (105_263_158 as f64 * 0.95) as u64 = 100_000_000` — equals minimum, passes the check.
5. Child neuron is created in spawning state with `maturity_e8s_equivalent = 105_263_158`.
6. After 7 days, `maybe_spawn_neurons` calls `apply_maturity_modulation(105_263_158, -1_000)` → `105_263_158 * 9_000 / 10_000 = 94_736_842`.
7. `neuron.cached_neuron_stake_e8s` is set to `94_736_842`, which is below `neuron_minimum_stake_e8s = 100_000_000`.

A deterministic integration test can reproduce this by: (a) setting `heap_data.maturity_modulation.current_value_permyriad = Some(-1_000)`, (b) spawning a neuron with the maturity value above, and (c) advancing time by 7 days and calling `maybe_spawn_neurons`, then asserting `cached_neuron_stake_e8s < neuron_minimum_stake_e8s`.

```rust
pub fn apply_maturity_modulation(
    amount_maturity_e8s: u64,
    maturity_modulation_basis_points: i32,
) -> Result<u64, String> {
    let amount_e8s = u128::from(amount_maturity_e8s);

    let adjusted_maturity_modulation_basis_points = saturating_add_or_subtract_u128_i32(
        BASIS_POINTS_PER_UNITY,
        maturity_modulation_basis_points,
    );

    let modulated_amount_e8s: u128 = amount_e8s
        .checked_mul(adjusted_maturity_modulation_basis_points)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?
        .checked_div(BASIS_POINTS_PER_UNITY)
        .ok_or_else(|| "Underflow or overflow when calculating maturity modulation".to_string())?;

    u64::try_from(modulated_amount_e8s).map_err(|err| err.to_string())
```
