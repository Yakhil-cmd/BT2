### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Delegator's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded `while` loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace. Because there is no mechanism to process entries in batches, a pool member who accumulates a sufficiently large trace (by making many `add_to_delegation_pool` or `exit_intent` calls without claiming) can reach a state where every future `claim_rewards` transaction runs out of gas, permanently freezing their unclaimed STRK rewards.

---

### Finding Description

`calculate_rewards` is called by `claim_rewards` in `src/pool/pool.cairo`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop bound is `pool_member_trace_length`, which equals the total number of balance-change checkpoints written for this pool member since their last successful `claim_rewards` call. Each iteration performs at least two storage reads: `pool_member_trace.at(entry_to_claim_from)` and `self.find_sigma(...)`. There is no way to process entries in partial batches — `claim_rewards` must process all pending entries in a single transaction. [2](#0-1) 

The `entry_to_claim_from` cursor is persisted in `pool_member_info` and advanced after each successful claim: [3](#0-2) 

This means the loop only processes entries accumulated *since the last claim*. However, if a pool member makes N balance changes between two claims, the next `claim_rewards` call must process all N entries atomically. If N is large enough to exhaust the Starknet transaction gas limit, the call reverts, `entry_to_claim_from` is never advanced, and every subsequent `claim_rewards` attempt also reverts — permanently locking the member's accrued rewards.

A new balance-trace entry is written on every call to `add_to_delegation_pool` (or `exit_intent` followed by re-delegation) that occurs in a different epoch from the previous entry: [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the trace length between two claims exceeds the gas ceiling for a single Starknet transaction, `claim_rewards` will always revert. The pool member cannot split the work across multiple calls because the function processes the entire pending range atomically. The principal can still be recovered via `exit_intent` / `exit_action`, but all accrued STRK rewards are irrecoverably frozen in the pool contract.

---

### Likelihood Explanation

**Medium.**

A pool member who actively manages their delegation (e.g., repeatedly topping up with small amounts across many epochs, or cycling through `exit_intent` → re-delegation) will accumulate one trace entry per epoch of activity. Over a long protocol lifetime — or if a delegator deliberately avoids claiming for an extended period while making frequent balance changes — the trace can grow to hundreds or thousands of entries. Each entry requires multiple storage reads in the loop, making gas exhaustion realistic at scale. The developers themselves acknowledge the risk in the code comment: *"This loop is unbounded but unlikely to exceed gas limits."* [5](#0-4) 

---

### Recommendation

1. **Paginated claiming**: Add a `max_entries` parameter to `claim_rewards` so the loop processes at most `max_entries` checkpoints per call, advancing `entry_to_claim_from` and returning partial rewards. Callers can invoke the function multiple times to drain the full trace.
2. **Bounded trace growth**: Enforce a maximum number of unclaimed balance-change entries per pool member (e.g., require a claim before a new balance change is accepted once the trace exceeds a threshold).
3. **Estimate worst-case gas**: Measure the per-iteration gas cost and set a hard cap on `pool_member_trace_length` that is provably safe within Starknet's transaction gas limit.

---

### Proof of Concept

1. Pool member `A` joins a pool and delegates the minimum amount.
2. Over `N` epochs, `A` calls `add_to_delegation_pool` once per epoch with a small amount, never calling `claim_rewards`. Each call writes a new entry to `pool_member_epoch_balance`, so after `N` epochs the trace has `N` entries.
3. `A` calls `claim_rewards`. The `calculate_rewards` loop must iterate over all `N` entries, each requiring `pool_member_trace.at(i)` (storage read) and `find_sigma(...)` (additional storage reads into `cumulative_rewards_trace`).
4. For sufficiently large `N`, the transaction exceeds the Starknet gas limit and reverts.
5. Every subsequent `claim_rewards` call also reverts because `entry_to_claim_from` was never advanced. `A`'s accumulated STRK rewards are permanently frozen. [6](#0-5)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
