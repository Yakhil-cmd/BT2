### Title
Unauthorized Invoke Transaction Admitted via `skip_stateful_validations` Signature Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender. Because that mempool check returns `true` for **any** transaction from the sender — not specifically a `deploy_account` — an unprivileged attacker can submit an invoke transaction with an invalid or empty signature targeting any account that has a pending `deploy_account` in the mempool, and the gateway will admit it without ever verifying the signature.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all four conditions hold:

1. The transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

**The broken invariant — condition 4 is not type-specific:** [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or committed state — it does not verify that the transaction is a `deploy_account`. The comment in `skip_stateful_validations` claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is not enforced by the check.

**Effect — `__validate__` is never called:**

When `skip_validate=true`, `run_validate_entry_point` sets `validate: !skip_validate = false`: [3](#0-2) 

This is passed to `StatefulValidator::perform_validations`, which for Invoke transactions returns `Ok(())` immediately without calling `__validate__`: [4](#0-3) 

The account's signature verification logic is never executed at the gateway.

**Execution-time consequence:**

During batcher execution, `AccountTransaction` is created with `ExecutionFlags::default()` which has `validate: true`: [5](#0-4) 

So `__validate__` is called during block execution. For a real account with ECDSA validation, the transaction reverts. However, the victim's account is still charged fees for the reverted transaction — an unauthorized economic loss. For accounts with weak or absent `__validate__` logic (e.g., `account_with_dummy_validate`), the attacker's calldata executes successfully.

---

### Impact Explanation

**Matching impact**: *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

An attacker can inject an unsigned (or arbitrarily signed) invoke transaction for any victim account that has a pending `deploy_account` in the mempool. The gateway admits it without signature verification. The transaction reaches the batcher, is included in a block, and either:

- Reverts during execution (for accounts with real signature checks), charging the victim fees for a transaction they never authorized, or
- Succeeds (for accounts with dummy `__validate__`), executing arbitrary calldata on behalf of the victim.

---

### Likelihood Explanation

- Any account that submits a `deploy_account` transaction is vulnerable during the window between submission and block inclusion.
- The attacker only needs to observe the public mempool for pending `deploy_account` transactions.
- No special privileges are required; any unprivileged actor can submit an `RpcTransaction` to the gateway.
- The nonce constraint (`nonce=1`) is trivially satisfiable.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender in the mempool. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(sender_address) -> bool` that only returns `true` when the pending transaction is of type `DeployAccount`. This preserves the UX intent (allowing the paired invoke to be submitted simultaneously) while preventing unauthorized actors from exploiting the skip path.

---

### Proof of Concept

**Setup**: Victim `0xVICTIM` submits a `deploy_account` transaction (nonce=0). It enters the mempool.

**Attack**:

1. Attacker observes `deploy_account` for `0xVICTIM` in the mempool.
2. Attacker submits an `RpcInvokeTransactionV3` with:
   - `sender_address = 0xVICTIM`
   - `nonce = 1`
   - `calldata = [transfer_all_to_attacker]`
   - `signature = []` (empty)
3. Gateway stateless validation passes (no signature content check).
4. `extract_state_nonce_and_run_validations` is called:
   - `account_nonce = 0` (victim not deployed yet) ✓
   - `validate_nonce`: nonce=1 is within `[0, max_allowed_nonce_gap]` ✓
   - `skip_stateful_validations`: tx.nonce()==1, account_nonce==0, `account_tx_in_pool_or_recent_block(0xVICTIM)` = **true** (victim's deploy_account is in pool) → returns `true`
5. `run_validate_entry_point` is called with `skip_validate=true` → `validate=false` → `__validate__` is **never called**.
6. Transaction is admitted to the mempool.
7. Batcher includes both transactions in a block. The `deploy_account` executes first (nonce=0), deploying the victim's account. The attacker's invoke (nonce=1) executes next with `validate=true`; `__validate__` is called, the empty signature fails, the transaction reverts, and the victim is charged fees.

**Key code path**: [6](#0-5) 

The `account_tx_in_pool_or_recent_block` call at line 444–447 returns `true` solely because the victim's `deploy_account` is in the pool, with no check that the attacker's invoke was submitted by the same party or that the pool entry is a `deploy_account`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
