### Title
Gateway Admits Unsigned Invoke Transactions for Any Account with a Pending Deploy-Account in the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (the account's signature-verification step) for any invoke transaction with `nonce == 1` submitted to an address whose `account_tx_in_pool_or_recent_block` check returns `true`. Because the mempool is public and the stateless validator only checks signature *length* (not validity), an unprivileged attacker can submit an invoke with an empty or forged signature for any victim account that has a pending deploy-account in the mempool, and the gateway will admit it without ever calling `__validate__`.

---

### Finding Description

**Stateless validator — signature check is length-only.**

`StatelessTransactionValidator::validate_tx_signature_size` rejects signatures that exceed `max_signature_length` but accepts any signature of zero or more elements, including an empty one. [1](#0-0) 

**Stateful validator — `skip_stateful_validations` skips `__validate__` based on a mempool presence check.**

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate == true`, `run_validate_entry_point` is called with `validate: false`, so the blockifier's `StatefulValidator::validate` (which calls `__validate__`) is never invoked. [3](#0-2) 

**`account_tx_in_pool_or_recent_block` checks the victim's address, not the submitter's identity.**

The check is purely address-based: it returns `true` if the target address has *any* transaction in the pool or recent committed state. [4](#0-3) 

**Mempool nonce validation does not block the attacker's transaction.**

`MempoolState::validate_incoming_tx` only rejects a transaction if `tx_nonce < account_nonce` (`NonceTooOld`). A fresh invoke with `nonce == 1` for an account whose on-chain nonce is `0` passes this check. [5](#0-4) 

`validate_fee_escalation` (called from `validate_tx`) would raise `DuplicateNonce` only if a transaction with the *same nonce* already exists in the pool for that address. If the victim has not yet submitted their own `nonce == 1` invoke, no such entry exists, and the attacker's transaction is accepted. [6](#0-5) 

**Full gateway flow for the attacker's transaction.**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks` (which calls `validate_state_preconditions` → `validate_nonce` → `validate_by_mempool` → `skip_stateful_validations`) and then `run_validate_entry_point`. With `skip_validate == true`, the `__validate__` entry point is never reached. [7](#0-6) 

---

### Impact Explanation

An attacker can submit an invoke transaction for any victim account that has a pending deploy-account in the mempool, using an empty or invalid signature. The gateway admits the transaction to the mempool without verifying the account's `__validate__` entry point. Concrete consequences:

- **Nonce-slot squatting**: The attacker's transaction occupies `nonce == 1` for the victim's address. If the victim later submits their own legitimate `nonce == 1` invoke, the mempool rejects it with `DuplicateNonce`, blocking the victim's first post-deploy transaction.
- **Forced reverted transaction on-chain**: When the block is built, the deploy-account runs first (deploying the account), then the attacker's invoke runs with `__validate__` called on the now-deployed account. `__validate__` rejects the invalid signature, producing a reverted transaction in the victim's account history.
- **Broken admission invariant**: The gateway's core security property — that every admitted transaction has been validated by the account's `__validate__` — is violated for any account in the deploy-account + invoke UX window.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The mempool is public. Any observer can enumerate pending deploy-account transactions and identify victim addresses. The attack requires no privileges, no special knowledge beyond the victim's address, and no on-chain funds. The window of vulnerability is the time between the victim's deploy-account entering the mempool and the victim submitting their own `nonce == 1` invoke. This window is non-trivial in practice (users often submit the two transactions separately or with a delay).

---

### Recommendation

**Short term**: Before returning `true` from `skip_stateful_validations`, verify that the pending deploy-account transaction in the mempool was submitted by the *same sender* as the invoke transaction (i.e., the deploy-account's `contract_address` field matches the invoke's `sender_address` and the deploy-account's `tx_hash` is known to the gateway). This prevents an attacker from exploiting a victim's pending deploy-account.

**Medium term**: Instead of skipping `__validate__` entirely, simulate the account's post-deployment state (using the class hash from the pending deploy-account) and run `__validate__` against that simulated state. This preserves the UX benefit while enforcing signature verification.

**Long term**: Require that the deploy-account and its first invoke be submitted as an atomic bundle (a single RPC call), so the gateway can verify both together before admitting either to the mempool.

---

### Proof of Concept

```
1. Alice submits deploy_account(class_hash=C, salt=S) → alice_address is computed,
   deploy_account enters the mempool with nonce=0.

2. Attacker observes alice_address in the mempool via any public mempool API.

3. Attacker crafts:
     invoke_tx {
       sender_address: alice_address,
       nonce: 1,
       calldata: [<arbitrary>],
       signature: [],          // empty — passes stateless length check
       resource_bounds: <valid>,
     }

4. Attacker submits invoke_tx to the gateway.

5. Gateway stateless check:
   - validate_tx_signature_size: len([]) == 0 <= max_signature_length → OK

6. Gateway stateful check:
   - account_nonce = get_nonce_from_state(alice_address) = 0  (not deployed yet)
   - validate_nonce: 0 <= 1 <= max_allowed_nonce_gap → OK
   - validate_by_mempool: MempoolState::validate_incoming_tx: 1 >= 0 → OK
   - skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(alice_address) == true  ✓  (Alice's deploy_account)
       → returns true (SKIP __validate__)
   - run_validate_entry_point(skip_validate=true):
       ExecutionFlags { validate: false, ... }
       → __validate__ is NOT called

7. Attacker's invoke is admitted to the mempool for alice_address at nonce=1.

8. Alice submits her own invoke(nonce=1) → mempool returns DuplicateNonce → rejected.

9. Block is built:
   - Alice's deploy_account executes → alice_address is deployed.
   - Attacker's invoke executes → __validate__ runs on alice_address → signature invalid
     → transaction REVERTS, but is included in the block.
```

```rust
    fn validate_tx_signature_size(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        let signature = tx.signature();

        let signature_length = signature.0.len();
        if signature_length > self.config.max_signature_length {
            return Err(StatelessTransactionValidatorError::SignatureTooLong {
                signature_length,
                max_signature_length: self.config.max_signature_length,
            });
        }

        Ok(())
    }
```

```rust
    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
        &self,
        tx_reference: TransactionReference,

    ) -> MempoolResult<()> {
        let TransactionReference { address, nonce: tx_nonce, .. } = tx_reference;
        let account_nonce = self.resolve_nonce(address, incoming_account_nonce);
```
