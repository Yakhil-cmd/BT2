### Title
Owner Can Atomically Redirect ETH Connector to Malicious Account Without Timelock or Multi-Sig — (File: engine/src/contract_methods/connector.rs)

---

### Summary

`set_eth_connector_contract_account` allows the single owner account to instantly redirect all ETH bridge operations — `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw` — to an arbitrary NEAR account with no timelock, no multi-sig, and no delay. A malicious or compromised owner can point the connector at an attacker-controlled account, stealing all in-flight bridged ETH and enabling unbacked ETH minting (insolvency).

---

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` (lines 418–438) writes an arbitrary NEAR `AccountId` into the engine's persistent storage as the ETH connector:

```rust
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;  // sole guard
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);                     // overwrites connector
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

The stored account ID is consumed by `get_connector_account_id` inside `return_promise` (lines 598–617), which is the **sole routing function** for every bridge-facing call:

```rust
fn return_promise<I: IO + PromiseHandler, E: Env>(...) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,   // attacker-controlled after redirect
        method: method.to_string(),
        ...
    };
    ...
}
```

Every user-facing bridge method delegates through `return_promise`:

| Method | Routed call |
|---|---|
| `withdraw` (line 58) | `engine_withdraw` |
| `ft_transfer` (line 264) | `engine_ft_transfer` |
| `ft_transfer_call` (line 285) | `engine_ft_transfer_call` |
| `storage_deposit` (line 302) | `engine_storage_deposit` |
| `storage_unregister` (line 323) | `engine_storage_unregister` |
| `storage_withdraw` (line 338) | `engine_storage_withdraw` |

A second critical path exists in `ft_on_transfer` (line 81):

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)   // mints ETH — no supply cap
} else {
    engine.receive_erc20_tokens(...)
};
```

Once the connector account is set to `attacker.near`, that account can call `ft_on_transfer` on the engine and be treated as the legitimate ETH connector, minting arbitrary base ETH with no real backing.

There is **no timelock**, **no multi-sig**, and **no delay** between the owner calling `set_eth_connector_contract_account` and the redirect taking effect. The change is atomic and immediate.

---

### Impact Explanation

**Direct theft of user funds (Critical):** Every user who calls `withdraw` after the redirect has their `engine_withdraw` promise routed to the attacker's contract. The attacker's contract can absorb the call without processing the Ethereum-side unlock, permanently stealing the bridged ETH.

**Insolvency (Critical):** After the redirect, `attacker.near` can call `ft_on_transfer` on the engine with an arbitrary `amount`. Because `predecessor_account_id == get_connector_account_id(&io)?` evaluates to `true`, `receive_base_tokens` mints unbacked ETH into the EVM state, creating insolvency across all ETH-denominated balances on Aurora.

---

### Likelihood Explanation

The owner is a single NEAR account (`state.owner_id`) with no on-chain multi-sig or governance wrapper enforced by the engine contract itself. A single private-key compromise, insider action, or social-engineering event targeting the owner account is sufficient. The function requires one transaction and takes effect in the same block. Users have zero reaction time because there is no timelock or staged-change mechanism for this parameter.

---

### Recommendation

1. **Timelock**: Require a mandatory delay (e.g., 48–72 hours) between proposing and applying a new connector account, giving users time to exit.
2. **Multi-sig**: Gate `set_eth_connector_contract_account` behind a multi-sig or DAO governance contract rather than a single EOA owner.
3. **Allowlist**: Restrict the new connector account to a pre-approved set of known-good accounts, preventing arbitrary redirection.
4. **Emit an observable event**: At minimum, emit a log/event so monitoring infrastructure can detect and alert on connector changes immediately.

---

### Proof of Concept

1. Owner calls `set_eth_connector_contract_account` with `account = "attacker.near"` — one transaction, no delay.
2. `attacker.near` is now stored as the connector account via `set_connector_account_id`.
3. **Theft path**: Any user calls `withdraw(recipient_eth_address, amount)`. `return_promise` routes `engine_withdraw` to `attacker.near`. `attacker.near` ignores the call; user's NEP-141 ETH balance is debited but no Ethereum-side unlock occurs. Funds are lost.
4. **Insolvency path**: `attacker.near` calls `ft_on_transfer` on the engine contract with `amount = U128::MAX`. `predecessor_account_id == get_connector_account_id(&io)?` is `true`. `receive_base_tokens` mints `U128::MAX` wei of ETH into the attacker's EVM address. The ETH supply on Aurora is now unbacked, rendering all user balances insolvent.
