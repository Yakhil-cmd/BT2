Audit Report

## Title
Single Principal Can Exhaust the Global ckBTC Withdrawal Queue, Blocking All Other Users - (File: `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary
The ckBTC minter enforces a single global cap of `MAX_CONCURRENT_PENDING_REQUESTS = 5000` on pending withdrawal requests with no per-principal sub-limit. A single unprivileged user can sequentially submit minimum-amount withdrawal requests until the global queue is full, causing every subsequent `retrieve_btc` or `retrieve_btc_with_approval` call from any other user to fail with `TemporarilyUnavailable("too many pending retrieve_btc requests")`. The queue drains only as Bitcoin transactions confirm on-chain, creating a DoS window of hours to days.

## Finding Description
`MAX_CONCURRENT_PENDING_REQUESTS = 5000` is defined at the module level and checked globally before accepting any new request: [1](#0-0) [2](#0-1) [3](#0-2) 

The only per-account protection is `retrieve_btc_guard`, which uses a `BTreeSet<Account>` (`retrieve_btc_accounts`) to prevent two **concurrent** in-flight calls for the same `Account`: [4](#0-3) 

The guard is RAII-based and is dropped when the async function returns — after the burn is committed and the request is pushed to `pending_retrieve_btc_requests`. This means sequential calls from the same account are not blocked: [5](#0-4) 

For `retrieve_btc_with_approval`, the `Account` key includes `from_subaccount`, so an attacker using distinct subaccounts is treated as distinct accounts, bypassing even the concurrent guard entirely: [6](#0-5) 

Once accepted, requests are pushed unconditionally into `pending_retrieve_btc_requests` with no per-account cap enforced: [7](#0-6) 

`retrieve_btc_account_to_block_indices` tracks per-account block indices but is never consulted to enforce a per-account limit: [8](#0-7) 

`count_incomplete_retrieve_btc_requests` counts pending + in-flight + submitted-but-unconfirmed requests, meaning the queue stays full until Bitcoin confirmations arrive: [9](#0-8) 

## Impact Explanation
Once the global queue reaches 5000 entries, every call to `retrieve_btc` or `retrieve_btc_with_approval` from any user returns `TemporarilyUnavailable`. This is a complete, sustained DoS of the ckBTC withdrawal system on the mainnet canister `mqygn-kiaaa-aaaar-qaadq-cai`, preventing all legitimate ckBTC holders from converting their tokens to BTC. This matches the **High** impact category: *"Application/platform-level DoS... not based on raw volumetric DDoS"* and *"Significant Chain Fusion, ck-token... security impact with concrete user or protocol harm."*

## Likelihood Explanation
The attacker needs `5000 × fee_based_retrieve_btc_min_amount` ckBTC upfront, but the ckBTC is burned and the attacker receives BTC back minus miner fees. The net cost is only Bitcoin transaction fees for the batched outputs. With `MAX_CONCURRENT = 100` concurrent guard slots available, the attacker can fill the queue in ~50 batches of 100 parallel calls using distinct subaccounts. No privileged access is required — any unprivileged ingress caller can trigger this. The attack is repeatable as soon as the queue drains.

## Recommendation
- **Short term:** Add a per-account cap on entries in `pending_retrieve_btc_requests`. Reject a new request if the caller's account already has `N` (e.g., 10–50) incomplete requests. This can be enforced by querying `retrieve_btc_account_to_block_indices` (already populated in `accept_retrieve_btc_request`) and counting entries whose status is not finalized.
- **Long term:** Review `MAX_CONCURRENT_PENDING_REQUESTS` relative to the number of active users and Bitcoin block time. Consider rate-limiting per principal across a rolling time window.

## Proof of Concept
1. Eve acquires `5000 × fee_based_retrieve_btc_min_amount` ckBTC.
2. Eve calls `retrieve_btc_with_approval` with 100 distinct `from_subaccount` values concurrently (saturating the guard), waits for completion, then repeats ~50 times. Each call passes the guard (distinct `Account` keys), passes the queue-depth check, burns ckBTC, and enqueues a `RetrieveBtcRequest`.
3. After 5000 successful calls, `count_incomplete_retrieve_btc_requests()` equals `MAX_CONCURRENT_PENDING_REQUESTS`.
4. Alice calls `retrieve_btc_with_approval` for a legitimate withdrawal. The check at line 274 fires and returns `TemporarilyUnavailable("too many pending retrieve_btc requests")`.
5. Alice and all other users are blocked until Bitcoin confirms Eve's batched transactions — a window of hours to days.

A deterministic integration test using PocketIC can reproduce this by: initializing the minter state, submitting 5000 `accept_retrieve_btc_request` calls with distinct accounts, then asserting that a subsequent `retrieve_btc_with_approval` call returns `TemporarilyUnavailable`.

```rust
impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
```
