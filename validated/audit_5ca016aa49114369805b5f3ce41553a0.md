### Title
`sign()` Accepts Caller-Supplied `max_malicious` Not Validated Against Presignature Threshold — (`src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The robust ECDSA `sign()` function accepts a caller-supplied `max_malicious` parameter to compute the required signing-set size (`2*max_malicious+1`), but never validates this value against the threshold that was actually used during the `presign()` phase. Because `PresignOutput` does not encode the presigning threshold, a malicious coordinator can invoke `sign()` with a smaller `max_malicious` than was used in `presign()`, producing a signing set `N2 < N1`. The library's own documentation explicitly identifies this as the precondition for a split-view attack capable of extracting the secret key.

---

### Finding Description

**Root cause — `PresignOutput` carries no threshold field.**

`do_presign` in `src/ecdsa/robust_ecdsa/presign.rs` computes five scalar outputs (`big_r`, `alpha`, `beta`, `c`, `e`) but does not record the `max_malicious` / threshold used to produce them: [1](#0-0) 

**Root cause — `sign()` trusts the caller's `max_malicious` unconditionally.**

`sign()` in `src/ecdsa/robust_ecdsa/sign.rs` derives the required participant count solely from the caller-supplied `max_malicious`: [2](#0-1) 

The only enforcement is `participants.len() == 2*max_malicious_sign+1`. There is no check that `max_malicious_sign` equals the `max_malicious` used during `presign()`. Because `RerandomizedPresignOutput` (derived from `PresignOutput`) contains no threshold field, the signing phase has no ground truth to compare against.

**Analogy to the SuperPool bug.** `SuperPool._supplyToPools()` checks `poolCapFor[poolId]` (a locally-tracked value) instead of the pool's actual global capacity, so a deposit that should be capped by the real pool state is instead capped by a stale local value. Here, `sign()` checks `max_malicious` (a caller-supplied local value) instead of the threshold embedded in the presignature (the true global state), so a signing session that should be constrained by the presigning threshold is instead constrained by whatever the coordinator claims.

**Documented security constraint violated.**

`docs/ecdsa/robust_ecdsa/signing.md` states:

> Use exactly N1 = N2 = 2t+1 participants for both presigning and signing. Do **not** allow N2 < N1. Allowing larger presigning sets or smaller signing sets enables split-view and presignature-reuse attacks when a coordinator can run parallel or partially overlapping signing sessions. [3](#0-2) 

The same document states the attack can extract the secret key with as few as two signing sessions: [4](#0-3) 

---

### Impact Explanation

A malicious coordinator who controls which participants are invited to a signing session can:

1. Run `presign()` with `max_malicious=t` → `N1 = 2t+1` participants.
2. Run `sign()` with `max_malicious=t'` where `t' < t` → `N2 = 2t'+1 < N1` participants, passing all local checks.
3. Run two or more such sessions with overlapping but distinct subsets, presenting different `(msg_hash, tweak, participants)` to different signers.

This is the exact split-view scenario the library documents as enabling secret-key extraction. The impact is **Critical: extraction of the aggregate secret signing material**.

---

### Likelihood Explanation

The coordinator role is an explicit, reachable participant in every signing session. No privileged key material or external compromise is required — the coordinator simply passes a smaller `max_malicious` value to `sign()`. The `PresignOutput` struct provides no mechanism for honest participants to detect the mismatch, so the attack is silent from each individual signer's perspective.

---

### Recommendation

1. **Embed the threshold in `PresignOutput`** (and therefore in `RerandomizedPresignOutput`):

```rust
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub alpha: Scalar,
    pub beta: Scalar,
    pub c: Scalar,
    pub e: Scalar,
    pub max_malicious: MaxMalicious,   // ← add this
}
```

2. **Validate in `sign()`** that the caller-supplied `max_malicious` matches the value stored in the presignature before proceeding:

```rust
if max_malicious != presignature.max_malicious {
    return Err(InitializationError::BadParameters(
        "max_malicious must match the value used during presigning".to_string(),
    ));
}
```

This mirrors the SuperPool fix: instead of relying on a locally-tracked value, check the actual global state (the threshold recorded at presign time).

---

### Proof of Concept

```
1. Coordinator runs presign() with participants = [P1,P2,P3,P4,P5], max_malicious=2 (N1=5).
   Each Pi receives a PresignOutput with alpha_i, beta_i, c_i, e_i computed over all 5 parties.

2. Coordinator calls sign() twice:
   Session A: participants=[P1,P2,P3], max_malicious=1 (N2=3), msg_hash=h, tweak=ε
   Session B: participants=[P3,P4,P5], max_malicious=1 (N2=3), msg_hash=h, tweak=ε

   Both sessions pass the check `participants.len() == 2*1+1 == 3`. ✓
   No participant can detect that N2 < N1 because PresignOutput carries no threshold.

3. The coordinator now holds two sets of linearized signature shares computed with
   different Lagrange coefficients (different participant subsets) over the same
   presignature nonce — the precondition for the split-view key-extraction attack
   documented in docs/ecdsa/robust_ecdsa/signing.md.
``` [5](#0-4) [6](#0-5)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    if args.max_malicious.value() > participants.len() {
        return Err(InitializationError::BadParameters(
            "max_malicious must be less than or equals to participant count".to_string(),
        ));
    }

    let robust_ecdsa_threshold = args
        .max_malicious
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*max_malicious+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // To prevent split-view attacks documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during presigning must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args, rng);
    Ok(make_protocol(ctx, fut))
}

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
