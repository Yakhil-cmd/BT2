### Title
`MsgTurnBridge` Handler Is a No-Op Stub — Bridge Circuit Breaker Permanently Broken and Permission Check Absent - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` message handler in Cronos is implemented as a stub that unconditionally returns `nil, nil` without performing any permission check or state update. This means (1) any unprivileged user can call `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` permission system, and (2) the gravity bridge's enable/disable state can never be changed via this message, permanently disabling the emergency circuit breaker.

### Finding Description

`UpdateTokenMapping` correctly gates execution behind `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` before touching state. The analogous `TurnBridge` handler, which is supposed to set the gravity bridge's `bridge_active` parameter, does neither:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare with the correctly-implemented sibling handler:

```go
func (k msgServer) UpdateTokenMapping(...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, ...)
    }
    ...
}
``` [2](#0-1) 

The `CanTurnBridge` permission bit is defined and the permission infrastructure is fully wired: [3](#0-2) 

The proto definition confirms `MsgTurnBridge` carries a `sender` and an `enable` bool that are supposed to drive a real state change: [4](#0-3) 

The integration test `test_gravity_turn_bridge` demonstrates the intended contract: a permissioned validator calls `turn_bridge("false")` and subsequent bridge sends are expected to fail. With the no-op handler, the bridge state is never written, so the test's post-condition (bridge disabled) can never be satisfied: [5](#0-4) 

The gravity module's `bridge_active` parameter (queried as `cli.query_gravity_params()["bridge_active"]`) is what `TurnBridge` should be toggling, but no write ever reaches it. [6](#0-5) 

### Impact Explanation

**High — Bypass of Cronos admin bridge authorization check + permanent inability to disable the bridge.**

1. **Permission bypass**: Because there is no `HasPermission` guard, any unprivileged address can submit `MsgTurnBridge` and receive a `code: 0` success response. The `CanTurnBridge` permission bit is rendered meaningless.
2. **Bridge circuit breaker broken**: The gravity bridge's `bridge_active` flag can never be set to `false` through the intended administrative path. In an emergency (e.g., a bridge exploit draining funds), the admin and all permissioned operators are permanently unable to halt bridge/conversion flows via `MsgTurnBridge`, matching the allowed High impact: *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows."*

### Likelihood Explanation

The entry path is a live, registered gRPC message handler reachable by any network participant. No special privileges, leaked keys, or cryptographic breaks are required. The bug is a missing implementation body — it will trigger on every invocation.

### Recommendation

Implement `TurnBridge` with the same pattern as `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)

    sender, err := sdk.AccAddressFromBech32(msg.Sender)
    if err != nil {
        return nil, err
    }
    if !k.HasPermission(ctx, []sdk.AccAddress{sender}, CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }

    // Update the gravity module's bridge_active parameter
    gravityParams := k.gravityKeeper.GetParams(ctx)
    gravityParams.BridgeActive = msg.Enable
    k.gravityKeeper.SetParams(ctx, gravityParams)

    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept

1. Any unprivileged address submits `MsgTurnBridge{Sender: unprivilegedAddr, Enable: false}`.
2. The handler returns `nil, nil` — transaction succeeds with `code: 0`.
3. Query `gravity params` — `bridge_active` is still `true`.
4. Bridge send operations continue to succeed, confirming the circuit breaker has no effect.

The existing integration test at `integration_tests/test_gravity.py:661-662` already encodes the expected permission rejection for an unprivileged caller (`assert rsp["code"] != 0`), but with the current no-op implementation that assertion **fails** — the unprivileged call succeeds. [7](#0-6)
