### Title
Permissionless `collectPoolFees` Can Be Front-Run Against `setPoolAdminFeeDestination` to Divert Admin Fees to a Stale Destination — (`MetricOmmPoolFactory.sol`)

---

### Summary

`collectPoolFees` is explicitly permissionless — any address may call it. It reads the live `poolAdminFeeDestination[pool]` and forwards all accrued admin-share fees to that address. `setPoolAdminFeeDestination`, unlike `setPoolAdminFees` and `setPoolProtocolFee`, does **not** collect accrued fees before overwriting the destination. An attacker who monitors the mempool can front-run a pool admin's destination-change transaction with `collectPoolFees`, forcing all accrued admin fees to be sent to the old (potentially broken) address before the redirect takes effect.

---

### Finding Description

**Permissionless collection path:**

`collectPoolFees` carries no access-control modifier:

```solidity
// MetricOmmPoolFactory.sol L379
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← reads current destination
    );
}
```

The pool's `collectFees` then transfers the admin share directly to that address:

```solidity
// MetricOmmPool.sol L416-420
if (totalFee0ToAdmin > 0) transferToken0(adminFeeDestination_, totalFee0ToAdmin);
if (totalFee1ToAdmin > 0) transferToken1(adminFeeDestination_, totalFee1ToAdmin);
```

**Destination change without prior collection:**

`setPoolAdminFeeDestination` simply overwrites the mapping — no fee flush first:

```solidity
// MetricOmmPoolFactory.sol L438-447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees() call
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

**Contrast with sibling functions that do flush first:**

Both `setPoolAdminFees` (L418–425) and `setPoolProtocolFee` (L328–335) call `pool.collectFees(...)` at the old rates/destination before applying any change. `setPoolAdminFeeDestination` is the only fee-management mutator that skips this step, creating the inconsistency that enables the attack.

**Attack sequence:**

1. Pool admin fee destination is `oldDest` (e.g., a contract that cannot handle ERC-20 tokens).
2. Fees accrue over time from swaps.
3. Pool admin submits `setPoolAdminFeeDestination(pool, newDest)` to redirect fees.
4. Attacker sees the pending transaction in the mempool and submits `collectPoolFees(pool)` with higher gas.
5. `collectPoolFees` executes first: all accrued admin fees are transferred to `oldDest`.
6. `setPoolAdminFeeDestination` executes: destination is updated to `newDest`.
7. Fees sent to `oldDest` in step 5 are permanently locked.

---

### Impact Explanation

All admin-share fees accrued since the last collection are permanently sent to the old destination. If that address is a contract without token-withdrawal logic, the tokens are irrecoverable. The amount at risk equals the full admin-fee surplus held by the pool at the time of the attack. Because `collectPoolFees` is permissionless and costs only gas, the attacker has no economic barrier.

---

### Likelihood Explanation

The scenario requires: (a) the current admin fee destination to be an address that cannot usefully receive tokens, and (b) the pool admin to submit a destination-change transaction. Both conditions are realistic — integrators frequently deploy pools with treasury or multisig addresses that may later need to be rotated. On any chain with a public mempool (Ethereum mainnet, Base), front-running a single-transaction state change is straightforward. The attacker needs no special role or capital.

---

### Recommendation

`setPoolAdminFeeDestination` should flush accrued fees to the **current** destination before overwriting it, exactly as `setPoolAdminFees` and `setPoolProtocolFee` do:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush fees to the current destination before redirecting
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

This makes the destination change atomic with respect to accrued fees: whatever was owed to the old address is settled before the pointer moves, eliminating the front-running window.

---

### Proof of Concept

```
Pool deployed: adminFeeDestination = brokenContract (no ERC-20 withdrawal)
Swaps occur → admin fees accrue in pool surplus

Admin submits:
  setPoolAdminFeeDestination(pool, workingAddress)   [pending in mempool]

Attacker submits (higher gas):
  collectPoolFees(pool)

Execution order:
  1. collectPoolFees(pool)
       → pool.collectFees(..., poolAdminFeeDestination[pool])
       → poolAdminFeeDestination[pool] == brokenContract  ← stale read
       → transferToken0(brokenContract, adminFee0)        ← LOCKED
       → transferToken1(brokenContract, adminFee1)        ← LOCKED

  2. setPoolAdminFeeDestination(pool, workingAddress)
       → poolAdminFeeDestination[pool] = workingAddress   ← too late

Result: all accrued admin fees permanently locked in brokenContract.
```
