### Title
Asymmetric `delta_gas` Adjustment Silently Overcharges Users When Native Resource Consumption Is Below EVM Gas Consumption - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the dual-resource accounting adjustment (`delta_gas`) is applied asymmetrically: when native resource consumption implies *more* gas was used than EVM tracking shows, `gas_used` is increased and the user is charged extra. However, when native resource consumption implies *less* gas was used than EVM tracking shows (`delta_gas < 0`), `gas_used` is **not** reduced. The user is overcharged, and the excess fee silently flows to the operator rather than being refunded to the sender. A developer-authored `// TODO: return delta_gas to gas_used?` comment at the exact location confirms this gap is known but unresolved.

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` implements ZKsync OS's double-resource accounting reconciliation. After execution, it computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

The code only handles the `delta_gas > 0` branch (native cost exceeds EVM gas cost → inflate `gas_used`). The `delta_gas < 0` branch (EVM gas cost exceeds native cost → should *reduce* `gas_used`) is silently skipped, with a `// TODO: return delta_gas to gas_used?` comment marking the omission. [2](#0-1) 

The inflated `gas_used` then propagates directly into the refund and operator-payment calculations:

- **ZK flow** (`zk/mod.rs`): `token_to_refund = gas_price * (tx_gas_limit - gas_used)` — user receives a smaller refund than warranted; `token_to_pay_operator = gas_used * gas_price_for_operator` — operator receives more than warranted. [3](#0-2) [4](#0-3) 

- **Ethereum flow** (`ethereum/mod.rs`): same pattern — refund uses `tx_gas_limit - gas_used`, so the user is under-refunded by `|delta_gas| * gas_price`. [5](#0-4) 

The design documentation only describes the `delta_gas > 0` case and is silent on the negative case, confirming it is unintentionally unhandled: [6](#0-5) 

### Impact Explanation

Any user submitting a transaction where EVM gas consumption exceeds native resource consumption (computation-heavy, storage-light workloads) is silently overcharged. The overcharge amount is `|delta_gas| * gas_price` tokens, which are transferred to the operator instead of being refunded to the sender. The user has no way to recover these funds. The excess is not burned — it is a direct, unearned transfer to the operator, creating a systematic fee extraction from users.

**Impact: 4** — Direct, permanent loss of user funds on every qualifying transaction; no recovery path.

### Likelihood Explanation

The `delta_gas < 0` condition is triggered whenever a transaction's EVM gas consumption exceeds its native resource consumption. This is a common pattern for:
- Computation-heavy contracts with minimal storage writes
- Transactions where `native_per_gas` is low (low gas price relative to native price)
- Any transaction where the EVM gas model charges more than the native proving cost

This is not an edge case; it is a routine occurrence for a broad class of transactions.

**Likelihood: 3** — Regularly triggered by ordinary unprivileged transactions; no special setup required.

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor) so that the user is refunded the full amount corresponding to their actual resource consumption:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption is less than EVM gas consumption.
    // Reduce gas_used so the user is refunded the difference.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
// Remove the TODO comment once addressed.
```

This mirrors the symmetric intent described in the double-resource accounting documentation and resolves the TODO at line 80.

### Proof of Concept

1. Deploy a computation-heavy EVM contract that performs many arithmetic operations but writes zero or minimal storage (e.g., a loop computing hashes without `SSTORE`).
2. Submit a ZKsync L2 transaction calling this contract with a `gas_limit` of 500,000 and a `gas_price` that yields a non-trivial `native_per_gas` ratio.
3. After execution, observe that `gas_used` reported equals the full EVM gas consumed (e.g., 400,000), while `native_used / native_per_gas` is lower (e.g., 350,000), giving `delta_gas = -50,000`.
4. The user's refund is `(500,000 - 400,000) * gas_price = 100,000 * gas_price` instead of the correct `(500,000 - 350,000) * gas_price = 150,000 * gas_price`.
5. The operator receives `400,000 * gas_price_for_operator` instead of `350,000 * gas_price_for_operator`.
6. The user permanently loses `50,000 * gas_price` tokens with no recourse. [7](#0-6)
