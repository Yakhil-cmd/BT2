### Title
Silo Address Whitelist Bypassed for ERC-20 Token Recipients When No Fallback Address Is Configured — (`engine/src/engine.rs`)

### Summary
In Silo mode, `receive_erc20_tokens` only enforces the `Address` whitelist when an `erc20_fallback_address` is configured. When no fallback is set, the whitelist check is silently skipped, allowing ERC-20 tokens to be minted to non-whitelisted EVM addresses via the `ft_on_transfer` bridge entry point.

### Finding Description
The `receive_erc20_tokens` function in `engine/src/engine.rs` performs the recipient whitelist check inside a combined Rust `if let … &&` guard:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

Because Rust evaluates `if let Some(fallback_address) = …` first, the entire compound condition short-circuits to `false` whenever `get_erc20_fallback_address` returns `None`. The call to `is_allow_receive_erc20_tokens` is never reached, and the original (potentially non-whitelisted) `recipient` is used unchanged. Tokens are then minted to that address unconditionally. [2](#0-1) 

By contrast, the `submit`/`call` entry points always enforce the `Address` whitelist through `assert_access` before any EVM execution:

```rust
fn assert_access<I: IO + Copy, E: Env>(io: &I, env: &E, transaction: &NormalizedEthTransaction)
    -> Result<(), EngineError>
{
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    ...
}
``` [3](#0-2) 

`is_allow_receive_erc20_tokens` and `is_allow_submit` both delegate to the same `is_address_allowed` helper, which checks the `WhitelistKind::Address` list: [4](#0-3) [5](#0-4) 

The `ft_on_transfer` entry point routes to `receive_erc20_tokens` for any NEP-141 token that is not the base ETH connector, with no prior silo check: [6](#0-5) 

### Impact Explanation
In a Silo deployment where the `Address` whitelist is enabled but no `erc20_fallback_address` has been set, any caller can invoke `ft_transfer_call` on a registered NEP-141 contract, specifying a non-whitelisted EVM address as the recipient. The bridge mints ERC-20 tokens to that address. Because the non-whitelisted address is blocked from submitting EVM transactions by `is_allow_submit`, it cannot transfer, approve, or exit those tokens via the `ExitToNear` precompile. The NEP-141 tokens are consumed on the NEAR side and the ERC-20 tokens are frozen in an address that cannot act on them — a **temporary-to-permanent freezing of funds** (High impact).

### Likelihood Explanation
Medium. The precondition is a Silo deployment with the `Address` whitelist enabled and no `erc20_fallback_address` configured. The wiki and code show that the fallback address is an optional parameter: [7](#0-6) 

Any unprivileged NEAR account can trigger the path by calling `ft_transfer_call` on a registered NEP-141 token with a non-whitelisted EVM address in the `msg` field. No special permissions are required.

### Recommendation
Decouple the whitelist check from the fallback redirect. The `Address` whitelist should be enforced regardless of whether a fallback is configured. If the recipient is not whitelisted and no fallback exists, return an error so the NEP-141 tokens are refunded to the sender:

```rust
// Proposed fix:
if !silo::is_allow_receive_erc20_tokens(&self.io, &recipient) {
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io) {
        recipient = fallback_address;
    } else {
        return Err(/* ERR_ADDRESS_NOT_WHITELISTED */);
    }
}
```

### Proof of Concept
1. Deploy Aurora Engine in Silo mode; enable the `Address` whitelist; do **not** set an `erc20_fallback_address`.
2. Register a NEP-141 token and deploy its ERC-20 mirror.
3. From any NEAR account, call `ft_transfer_call` on the NEP-141 contract with `receiver_id = aurora` and `msg = <non_whitelisted_evm_address_hex>`.
4. `ft_on_transfer` is invoked on Aurora → `receive_erc20_tokens` is called.
5. The `if let Some(fallback_address) = get_erc20_fallback_address(…)` guard evaluates to `false` (no fallback set); `is_allow_receive_erc20_tokens` is never called.
6. ERC-20 tokens are minted to the non-whitelisted address.
7. Attempt any EVM transaction from that address (e.g., `transfer`, `ExitToNear`): `assert_access` → `is_allow_submit` → `NotAllowed` — the tokens are frozen.
