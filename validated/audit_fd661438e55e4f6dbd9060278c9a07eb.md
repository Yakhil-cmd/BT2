### Title
Pool Admin Can Permanently Lock Protocol Fees by Setting a Reverting `adminFeeDestination` - (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` accepts any non-zero address with no check that the destination can receive ERC-20 tokens. Because `collectFees` transfers admin fees **before** protocol fees in a single atomic call, a pool admin who sets a reverting destination permanently blocks all protocol fee collection with no factory-owner recovery path.

---

### Finding Description

`setPoolAdminFeeDestination` validates only that the new address is non-zero: [1](#0-0) 

`collectFees` on the pool transfers admin fees first, then protocol fees, then clears the notional accumulators — all in one atomic block: [2](#0-1) 

If `transferToken0(adminFeeDestination_, ...)` or `transferToken1(adminFeeDestination_, ...)` reverts (lines 417/420), the entire `collectFees` call reverts. Protocol transfers (lines 423/426) and the accumulator reset (lines 429-430) never execute.

Every factory path that collects fees passes `poolAdminFeeDestination[pool]` directly:

- `collectPoolFees` (permissionless): [3](#0-2) 
- `setPoolProtocolFee` (owner-only): [4](#0-3) 
- `setPoolAdminFees` (pool-admin): [5](#0-4) 

All three revert if the destination reverts. The factory owner has **no function** to override `poolAdminFeeDestination` — that mapping is exclusively writable by the pool admin via `setPoolAdminFeeDestination`. The factory owner's own `setPoolProtocolFee` is also bricked because it calls `collectFees` with the same bad destination before updating fee config.

---

### Impact Explanation

Protocol fees (spread surplus and notional accumulators) accrue inside the pool and can only leave via `collectFees`. With a reverting destination set, `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` are never cleared and the surplus is never transferred to the factory. These amounts are permanently locked in the pool — the protocol cannot recover them without the pool admin's cooperation. This is a direct, permanent loss of protocol fee revenue.

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The attack requires only one call to `setPoolAdminFeeDestination` with a contract that reverts on token receipt (e.g., a contract with no `receive`/fallback and no ERC-20 transfer hook). No special timing, no flash loan, no oracle manipulation is needed. A malicious or compromised pool admin can execute this at any time after pool deployment.

---

### Recommendation

Two complementary fixes:

1. **Collect fees before updating the destination** in `setPoolAdminFeeDestination` (mirror the pattern used in `setPoolAdminFees`), so any reverting destination is caught before it is persisted.

2. **Add a factory-owner override** for `poolAdminFeeDestination` (e.g., `setPoolAdminFeeDestinationOverride`) so the protocol can recover if a pool admin sets a broken destination.

Optionally, split `collectFees` into two independent transfers so a failing admin transfer does not block the protocol transfer.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import "forge-std/Test.sol";
// ... standard pool setup imports

contract RevertOnReceive {
    // reverts on any ERC-20 safeTransfer (transferFrom callback or direct transfer)
    fallback() external { revert("no tokens"); }
}

contract AdminFeeDestinationLockTest is Test {
    // 1. Deploy pool with normal adminFeeDestination
    // 2. Do swaps to accrue fees
    // 3. Pool admin calls setPoolAdminFeeDestination(pool, address(new RevertOnReceive()))
    // 4. Assert collectPoolFees reverts
    // 5. Assert notionalFeeToken0Scaled / notionalFeeToken1Scaled unchanged (fees locked)
    function test_revertingDestinationLocksProtocolFees() public {
        // ... setup pool, add liquidity, execute swaps ...
        RevertOnReceive bad = new RevertOnReceive();
        vm.prank(poolAdmin);
        factory.setPoolAdminFeeDestination(pool, address(bad));

        vm.expectRevert();
        factory.collectPoolFees(pool);

        // protocol fees remain locked in pool
        (uint128 n0,) = PoolStateLibrary._slot2(pool);
        assertGt(n0, 0, "notional fee still locked");
    }
}
```
