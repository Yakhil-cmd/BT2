Audit Report

## Title
Unsolicited `Pong` Triggers Unconditional Disconnection and Reconnect Loop — (`rs/bitcoin/adapter/src/connectionmanager.rs`)

## Summary
`process_pong_message` calls `conn.disconnect()` whenever a `Pong` is received while `PingState::Idle`, treating an unsolicited pong identically to a nonce-mismatched pong. Because `disconnect()` (not `discard()`) is used, the peer's address is returned to `known_addresses` and the adapter immediately reconnects to the same peer. An attacker controlling Bitcoin nodes in the adapter's address book can send unsolicited `Pong` messages in a tight loop to permanently occupy connection slots and stall ckBTC block ingestion.

## Finding Description
In `connectionmanager.rs`, `process_pong_message` evaluates pong validity:

```rust
let valid_pong = match conn.ping_state() {
    PingState::ExpectingPong { ping_sent_at: _, nonce: ping_nonce } => *ping_nonce == nonce,
    PingState::Idle { last_pong_at: _ } => false,
};
if valid_pong { conn.idle(); } else { conn.disconnect(); }
``` [1](#0-0) 

Both a nonce mismatch and an unsolicited pong (Idle state) unconditionally reach `conn.disconnect()`, which sets `ConnectionState::NodeDisconnected` and aborts the stream task: [2](#0-1) 

In `reap_disconnected`, a `NodeDisconnected` connection triggers `address_book.remove_from_active()`: [3](#0-2) 

`remove_from_active` moves the address from `active_addresses` back into `known_addresses`, making it immediately eligible for reconnection: [4](#0-3) 

This is the critical difference from `discard()`, which removes the address entirely when seeds are available: [5](#0-4) 

After reaping, `manage_connections` refills connections up to `max_connections`: [6](#0-5) 

The adapter thus reconnects to the same malicious peer, which can again send an unsolicited `Pong`, creating an indefinite disconnect-reconnect cycle for that slot.

## Impact Explanation
The adapter maintains between 2 (`min_connections`) and 5 (`max_connections`) connections when DNS seeds are available: [7](#0-6) 

If 2 or more peers in the address book are malicious and continuously send unsolicited `Pong` messages, the adapter can be kept below `min_connections` for extended periods. This stalls block delivery to the ckBTC canister, constituting a **High** severity impact: significant ck-token / Chain Fusion availability disruption with concrete user harm (ckBTC minting/redemption halted while the adapter cannot fetch blocks).

## Likelihood Explanation
The attacker must run Bitcoin nodes that appear in the adapter's address book. The adapter discovers peers via DNS seeds and `addr` messages from connected peers — a realistic path for an attacker to inject their nodes. No privileged access is required: completing the Bitcoin version handshake is sufficient. The Bitcoin P2P protocol places no restriction on sending `Pong` at any time, so the triggering message is fully protocol-conformant. The attack is repeatable indefinitely at negligible cost once the malicious nodes are in the address book.

## Recommendation
Replace `conn.disconnect()` in the `PingState::Idle` branch with a no-op. An unsolicited `Pong` is not a protocol violation; only a `Pong` with a mismatched nonce when one was expected warrants disconnection:

```rust
PingState::Idle { last_pong_at: _ } => {
    // Unsolicited pong — not a protocol violation; ignore.
    trace!(self.logger, "Received unsolicited pong from {}", address);
    return Ok(());
}
```

Only disconnect when `PingState::ExpectingPong` and the received nonce does not match the sent nonce.

## Proof of Concept
1. Run a Bitcoin node that completes the version handshake with the adapter (`ConnectionState::HandshakeComplete`, `PingState::Idle`).
2. From the peer side, send `NetworkMessage::Pong(42)` (any nonce).
3. `process_pong_message` is called → `PingState::Idle` branch → `valid_pong = false` → `conn.disconnect()`.
4. `reap_disconnected` fires → `NodeDisconnected` → `address_book.remove_from_active()` → address returned to `known_addresses`.
5. `manage_connections` reconnects to the same peer.
6. Repeat from step 2 indefinitely.

A deterministic integration test can be written using the existing test harness in `connectionmanager.rs` (see `mod test` at line 713): inject a mock peer that completes the handshake and immediately sends a `Pong`, then assert that the connection count drops and the address reappears in `known_addresses`, confirming the loop.
