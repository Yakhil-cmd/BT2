### Title
Missing Access Control in `update_rewards()` Allows Any Caller to Permanently Suppress Staker Consensus Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards()` function in the Staking contract has no caller restriction, allowing any unprivileged address to invoke it with `disable_rewards=true`. This permanently consumes the per-block reward slot (`last_reward_block`) without distributing any rewards, so the legitimate consensus mechanism can never recover rewards for that block. The parallel function `update_rewards_from_attestation_contract()` correctly gates its caller via `assert_caller_is_attestation_contract()`, making the missing guard in `update_rewards()` the direct root cause.

---

### Finding Description

`update_rewards()` is gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — no role or identity check is performed. [1](#0-0) 

The function unconditionally writes `current_block_number` into `last_reward_block` **before** branching on `disable_rewards`: [2](#0-1) 

If `disable_rewards=true`, the function returns immediately with no reward distribution, but the block slot is already consumed: [3](#0-2) 

Any subsequent call for the same block — including the legitimate consensus mechanism — is rejected: [4](#0-3) 

By contrast, the pre-consensus reward path correctly restricts its caller: [5](#0-4) 

`last_reward_block` is a single contract-wide value, not per-staker. One attacker call with any valid active staker address blocks the reward slot for **all** stakers for that block. [6](#0-5) 

---

### Impact Explanation

An attacker who front-runs the consensus mechanism's `update_rewards` call for block N with `disable_rewards=true` causes:

1. `last_reward_block` is set to N — the slot is permanently consumed.
2. No STRK or BTC block rewards are distributed to the staker or its delegation pools for block N.
3. The legitimate consensus call for block N reverts with `REWARDS_ALREADY_UPDATED`.

Because `last_reward_block` is shared, a single attacker transaction per block is sufficient to suppress rewards for every staker in the system for that block. Sustained over time, this constitutes **permanent freezing of unclaimed yield**, which is within the allowed High impact scope.

---

### Likelihood Explanation

- The function is fully public with no role requirement.
- The attacker only needs to know any active staker address (publicly readable from the `stakers` vector) and the current block number.
- The cost is one Starknet transaction per block. At current gas prices this is economically feasible for a motivated griever targeting a high-value staker.
- No privileged access, leaked key, or external dependency is required.

---

### Recommendation

Add a role check (e.g., `only_operator` or a dedicated `REWARDS_MANAGER_ROLE`) to `update_rewards()`, mirroring the pattern used in `update_rewards_from_attestation_contract()`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.roles.only_operator(); // <-- add this
    self.general_prerequisites();
    ...
}
```

Alternatively, move the `last_reward_block.write` to after the `disable_rewards` branch so that a no-op call does not consume the block slot.

---
