### Title
Unbounded `royalty_percentage` in WalletConnect Offer Confirmation Causes Spoofed Spending Amount Display - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The `parseRoyaltyPercentage` function in the Electron main process accepts any positive integer as a valid NFT royalty percentage with no upper-bound check. This value is used to compute and display `amountWithRoyalties` in the WalletConnect confirmation dialog shown to the user before they approve a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command. A malicious dApp or crafted offer file can embed an out-of-range `royalty_percentage` in the offer's driver dict, causing the displayed total spending amount to be arbitrarily inflated or deflated, deceiving the user into approving or rejecting an offer based on a false amount.

---

### Finding Description

In `packages/gui/src/electron/commands/parseCommandDisplay.ts`, the function `parseRoyaltyPercentage` validates only that the value is a positive integer or a digit-only string, with no upper bound:

```typescript
function parseRoyaltyPercentage(value: unknown): number | undefined {
  if (typeof value === 'number' && Number.isInteger(value) && value > 0) {
    return value;
  }
  if (typeof value === 'string' && /^\d+$/.test(value)) {
    return Number(value);
  }
  return undefined;
}
``` [1](#0-0) 

In Chia's NFT protocol, `royalty_percentage` is stored in basis points (0–10,000, representing 0%–100%). The function should reject values above 10,000, but it does not. The returned value is passed directly into `formatAmountWithRoyalties`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [2](#0-1) 

This `totalAmount` is formatted and returned as `amountWithRoyalties` on the `DisplayWalletDeltaItem`, which is rendered in the WalletConnect confirmation dialog before the user approves the transaction. [3](#0-2) 

The `royalty_percentage` value originates from two attacker-controlled sources:

1. **`chia_wallet.take_offer`**: The offer blob's embedded driver dict `infos` field, parsed by the daemon and returned in the offer summary. A malicious offer file can embed any `royalty_percentage`.
2. **`chia_wallet.create_offer_for_ids`**: The `driver_dict` parameter sent directly by the dApp over WalletConnect, with no sanitization before being passed to `createOfferRoyaltyPercentages`. [4](#0-3) 

---

### Impact Explanation

**Understated royalty attack (asset loss):** A malicious offer creator sets `royalty_percentage: "0"` in the driver dict while embedding a high actual royalty payment in the offer blob. The GUI displays `amountWithRoyalties` ≈ base price (no royalty shown). The user approves believing they pay the base price, but the actual offer enforces a high royalty, causing them to spend significantly more XCH/CAT than the confirmation dialog indicated.

**Overstated royalty attack (denial of service on legitimate offers):** A malicious dApp sets `royalty_percentage: "999999"` in the `driver_dict` of a `create_offer_for_ids` WalletConnect request. The GUI displays an astronomically large `amountWithRoyalties`, causing the user to reject a legitimate offer they would otherwise have accepted.

Both paths satisfy the High-severity criterion: "Corruption, spoofing, or unsafe trust of RPC, event, offer, NFT metadata, DataLayer, notification, or WalletConnect state that causes a user to approve, import, sign, send, revoke, burn, join, or **display the wrong asset, identity, amount, destination, or status**."

---

### Likelihood Explanation

- **Unprivileged entry**: Any dApp connected via WalletConnect can supply an arbitrary `driver_dict`, or any party can craft and share a malicious `.offer` file.
- **No existing guard**: `parseRoyaltyPercentage` explicitly accepts all positive integers. There is no clamping, rejection, or warning for values above 10,000.
- **User trust in the dialog**: The WalletConnect confirmation dialog is the primary security boundary the user relies on to verify what they are approving. Spoofing `amountWithRoyalties` directly undermines this boundary.
- **Contrast with NFTOfferEditor**: The legacy `NFTOfferEditor` does check `royaltyPercentage > 100` before submission, but `parseCommandDisplay.ts` — which handles the WalletConnect path — has no equivalent guard. [5](#0-4) 

---

### Recommendation

Add an upper-bound check in `parseRoyaltyPercentage` to reject values exceeding 10,000 (100% in basis points):

```typescript
function parseRoyaltyPercentage(value: unknown): number | undefined {
  let parsed: number | undefined;
  if (typeof value === 'number' && Number.isInteger(value) && value > 0) {
    parsed = value;
  } else if (typeof value === 'string' && /^\d+$/.test(value)) {
    parsed = Number(value);
  }
  if (parsed !== undefined && parsed <= 10_000) {
    return parsed;
  }
  return undefined;
}
```

Additionally, display a clear warning in the confirmation dialog when `amountWithRoyalties` differs significantly from the base amount, and consider cross-validating the driver dict's `royalty_percentage` against the on-chain NFT info fetched via `nftGetInfo` before rendering the confirmation.

---

### Proof of Concept

**Via WalletConnect (`create_offer_for_ids`):**

A malicious dApp sends the following WalletConnect request:
```json
{
  "method": "chia_createOfferForIds",
  "params": {
    "offer": { "1": "-1000000000000", "<nft_launcher_id>": "1" },
    "driver_dict": {
      "<nft_launcher_id>": {
        "type": "singleton",
        "also": {
          "type": "metadata",
          "also": {
            "type": "ownership",
            "transfer_program": {
              "type": "royalty transfer program",
              "royalty_percentage": "999999"
            }
          }
        }
      }
    }
  }
}
```

`parseRoyaltyPercentage("999999")` returns `999999`. `formatAmountWithRoyalties` computes `royaltyAmount = (1_000_000_000_000n * 999999n) / 10_000n ≈ 99_999_900_000_000n` mojos. The dialog displays `amountWithRoyalties ≈ "100,000.9 XCH"` instead of the actual `"1 XCH"`, causing the user to reject a legitimate offer. [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
