### Title
Gateway L2 Gas Price Admission Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful admission check validates a transaction's `l2_gas.max_price_per_unit` against the **previous block's execution price** (`block_header.l2_gas_price`). The next block being built will enforce the **EIP-1559-adjusted price** (`block_header.next_l2_gas_price`). Because these two values diverge every time block utilization is above or below the gas target, the gateway systematically admits transactions that the blockifier will reject, and rejects transactions the blockifier would accept.

---

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` reads the reference price for admission from `GatewayFixedBlockStateReader::get_block_info()`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates that field from `block_header.l2_gas_price.price_in_fri`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`block_header.l2_gas_price` is the price **used inside** the previous block. The price that will be enforced in the **next** block is `block_header.next_l2_gas_price`, which is the EIP-1559-adjusted value computed at the end of the previous block and stored separately in the header:

```rust
let block_header_without_hash = BlockHeaderWithoutHash {
    l2_gas_price,                          // price used in THIS block
    next_l2_gas_price: self.l2_gas_price,  // price enforced in the NEXT block
    ...
};
``` [3](#0-2) 

When the blockifier executes the next block it enforces `block_info.gas_prices.strk_gas_prices.l2_gas_price`, which equals `previous_block.next_l2_gas_price`:

```rust
let GasPriceVector { l1_gas_price, l1_data_gas_price, l2_gas_price } =
    block_info.gas_prices.gas_price_vector(fee_type);
// ...
if resource_bounds.max_price_per_unit < actual_gas_price.get() { ... }
``` [4](#0-3) 

The two checks therefore use different reference prices:

| Layer | Reference price |
|---|---|
| Gateway admission | `previous_block.l2_gas_price` |
| Blockifier `check_fee_bounds` | `previous_block.next_l2_gas_price` |

The EIP-1559 formula adjusts the price by up to `price / gas_price_max_change_denominator` per block, so the gap is non-trivial in every block where utilization differs from the gas target.

The TODO comment in the gateway code explicitly acknowledges the wrong value is being read: `// TODO(Arni): getnext_l2_gas_price from the block header.` [5](#0-4) 

---

### Impact Explanation

**When price is rising** (`next_l2_gas_price > l2_gas_price`, i.e., block utilization above gas target):

A transaction with `l2_gas.max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway threshold check but fails `check_fee_bounds` in the blockifier with `ResourceBoundsError::MaxGasPriceTooLow`. The transaction is admitted to the mempool, propagated to peers, and then rejected during block execution — wasting sequencer resources and producing a user-visible failure after the transaction was accepted.

**When price is falling** (`next_l2_gas_price < l2_gas_price`, i.e., block utilization below gas target):

A transaction with `max_price_per_unit` in `[next_l2_gas_price, threshold * l2_gas_price)` is rejected by the gateway with `GAS_PRICE_TOO_LOW` even though the blockifier would accept it. Valid transactions are denied admission.

Both directions match the "High" impact category: **Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The EIP-1559 adjustment fires on every block where gas usage differs from the gas target. In normal operation this is nearly every block. The discrepancy between `l2_gas_price` and `next_l2_gas_price` is therefore present continuously, not only under adversarial conditions. Any user submitting a V3 (`AllResources`) transaction with a `max_price_per_unit` near the current threshold will be affected.

---

### Recommendation

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` should populate `strk_gas_prices.l2_gas_price` from `block_header.next_l2_gas_price` instead of `block_header.l2_gas_price.price_in_fri`. Because `next_l2_gas_price` is a single `GasPrice` (not a `GasPricePerToken`), the `GatewayFixedBlockStateReader` trait or the `BlockInfo` struct may need a dedicated field, or the validator can call a new `get_next_l2_gas_price()` method on the reader. The TODO comment at line 229 already tracks this gap.

---

### Proof of Concept

1. Observe the previous block has `l2_gas_price = 1_000` FRI and `next_l2_gas_price = 1_100` FRI (block was 10% over gas target, EIP-1559 raised the price).
2. Submit a V3 `invoke` transaction with `l2_gas.max_price_per_unit = 1_050` FRI and `min_gas_price_percentage = 100`.
3. Gateway check: `1_050 >= 100% * 1_000` → **passes**, transaction admitted to mempool.
4. Blockifier `check_fee_bounds` for the next block: `1_050 >= 1_100` → **fails** with `MaxGasPriceTooLow`.
5. The transaction was accepted by the gateway but is unexecutable in the block it was admitted for.

Conversely, with `l2_gas_price = 1_000` and `next_l2_gas_price = 900` (block under-utilized):

1. Submit a transaction with `max_price_per_unit = 950`.
2. Gateway check: `950 >= 100% * 1_000` → **fails**, transaction rejected.
3. Blockifier would check: `950 >= 900` → would **pass**.
4. A valid transaction is incorrectly denied admission.
