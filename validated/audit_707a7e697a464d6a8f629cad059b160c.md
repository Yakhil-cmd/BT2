### Title
Missing Caller Access Control on `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as restricted to "Only starkware sequencer" but the implementation enforces no such check. Any non-zero address can call it first in any block with `disable_rewards = true`, consuming the global `last_reward_block` gate and permanently discarding that block's consensus rewards for all stakers.

---

### Finding Description

The spec at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

The actual implementation in `src/staking/staking.cairo` at `general_prerequisites` is:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [1](#0-0) 

There is no sequencer role check, no `assert_caller_is_sequencer`, and no role-gating of any kind. The full `update_rewards` entry point is:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only: unpaused + non-zero caller
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // global gate consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits with no rewards distributed
    }
    ...
}
``` [2](#0-1) 

`last_reward_block` is a **single global storage variable**, not per-staker. Once it is written to the current block number, every subsequent call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`, regardless of which staker is targeted or who the caller is. [3](#0-2) 

---

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards=true)` as the first transaction in every block:

1. Passes `general_prerequisites` (contract is live, caller is non-zero).
2. Passes the `last_reward_block` gate (new block, gate is open).
3. Writes `last_reward_block = current_block`.
4. Returns immediately — zero rewards distributed to anyone.
5. Every subsequent legitimate sequencer call in that block reverts with `REWARDS_ALREADY_UPDATED`.

The block's consensus rewards are **permanently lost** — they are never minted or credited to any staker. Repeating this every block across an epoch starves all validators of their yield indefinitely. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation

The call requires no stake, no privileged role, and no special setup — only a non-zero address and knowledge of any active staker's address (which is public on-chain). The attacker pays only gas. The attack is sustainable indefinitely and is trivially automatable.

---

### Recommendation

Add a sequencer-only guard to `update_rewards`. On Starknet, the sequencer address is available via `starknet::get_execution_info().unbox().tx_info.unbox().account_contract_address` or a stored sequencer address role. The check should be placed immediately after `general_prerequisites`:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this
    ...
}
```

Alternatively, store an authorized `rewards_updater` address (set at deployment to the sequencer) and assert `get_caller_address() == self.rewards_updater.read()`.

---

### Proof of Concept

1. Deploy the staking system with consensus rewards active.
2. Stake as `staker_A` and wait K epochs for balance to become effective.
3. From an arbitrary EOA (attacker), call:
   ```
   staking.update_rewards(staker_address=staker_A, disable_rewards=true)
   ```
   at the start of every block.
4. Observe that `staker_A.unclaimed_rewards_own` never increases.
5. Attempt the legitimate sequencer call in the same block — it reverts with `REWARDS_ALREADY_UPDATED`.
6. Advance many blocks/epochs; compare accumulated rewards to expected model — they remain zero.

The existing test `update_rewards_disable_rewards_consensus_rewards_flow_test` already demonstrates that `disable_rewards=true` with consensus active yields zero rewards and that a second call in the same block reverts — confirming the gate is consumed on the first call regardless of caller identity. [4](#0-3)

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`

            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(

                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
