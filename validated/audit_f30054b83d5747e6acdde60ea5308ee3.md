### Title
EVM Pause Guard Blocks Critical Asset-Withdrawal Operations, Permanently Locking User Funds During Pause - (File: fvm/evm/stdlib/contract.cdc)

### Summary
When the Flow EVM is paused via governance, the `!EVM.isPaused()` pre-condition guard is applied uniformly to all state-mutating operations — including the critical asset-recovery functions `CadenceOwnedAccount.withdraw()`, `CadenceOwnedAccount.withdrawTokens()`, and `CadenceOwnedAccount.withdrawNFT()`. Users with assets held in a Cadence-Owned Account (COA) in EVM cannot retrieve those assets back to Cadence for the entire duration of the pause, with no escape hatch.

### Finding Description
The `EVM` contract in `fvm/evm/stdlib/contract.cdc` implements a pause mechanism controlled by the Governance Committee. When paused, `EVM.isPaused()` returns `true` and every state-mutating function enforces a `pre { !EVM.isPaused() }` condition.

The three asset-withdrawal functions on `CadenceOwnedAccount` all carry this guard:

**`CadenceOwnedAccount.withdraw()`** — moves FLOW tokens from EVM back to Cadence: [1](#0-0) 

**`CadenceOwnedAccount.withdrawNFT()`** — bridges an NFT from EVM back to Cadence: [2](#0-1) 

**`CadenceOwnedAccount.withdrawTokens()`** — bridges fungible tokens from EVM back to Cadence: [3](#0-2) 

The pause state is read from contract storage: [4](#0-3) 

The design intent is documented as putting EVM into "read-only mode": [5](#0-4) 

However, `withdraw()`, `withdrawNFT()`, and `withdrawTokens()` are not EVM-state-mutating operations in the sense that they move assets *out of* EVM and back to Cadence. Blocking them during a pause does not protect EVM state integrity — it only prevents users from recovering their own assets.

Note also that `CadenceOwnedAccount.deposit()` (line 563) has no direct pause guard of its own but delegates to `EVMAddress.deposit()`, which does: [6](#0-5) 

So during a pause, assets can neither enter nor leave EVM via COA — but the critical harm is the inability to exit.

### Impact Explanation
Any user who holds FLOW tokens, bridged fungible tokens, or bridged NFTs inside a COA in Flow EVM is unable to retrieve those assets for the entire duration of the pause. There is no alternative withdrawal path. If the pause is extended (e.g., during a security incident requiring prolonged investigation), users are locked out of their own assets indefinitely. In DeFi contexts built on top of Flow EVM, this can trigger cascading effects: a user who needs their FLOW back to Cadence to meet an obligation (e.g., a Cadence-side collateral call, a staking requirement, or a time-sensitive redemption) cannot do so, resulting in direct, quantifiable asset loss or forced liquidation on the Cadence side.

### Likelihood Explanation
The Governance Committee can pause EVM for legitimate operational reasons (maintenance, security response). Pauses are a documented and expected operational event. Any pause of non-trivial duration (hours to days, as is common during security incidents) directly triggers this impact for all COA holders with non-zero balances. The affected users are unprivileged — any COA owner is exposed. No attacker action is required beyond the governance pause itself occurring.

### Recommendation
Exempt the three asset-recovery functions from the `isPaused()` guard, analogous to how `dryRun` and `dryCall` are already exempted because they do not mutate EVM state in a harmful way: [7](#0-6) 

Specifically, `withdraw()`, `withdrawNFT()`, and `withdrawTokens()` move assets *out of* EVM and back to Cadence. They do not introduce new EVM state or execute arbitrary EVM bytecode. Allowing them during a pause does not undermine the security rationale for the pause (preventing arbitrary EVM execution), while ensuring users retain access to their own assets. Alternatively, introduce a separate `emergencyWithdraw` path that bypasses the pause guard for these specific operations.

### Proof of Concept
1. Alice deposits 100 FLOW into her COA via `coa.deposit(from: <-vault)`.
2. The Governance Committee pauses EVM for a security investigation: `/storage/evmOperationsPaused` is set to `true`.
3. Alice submits a Cadence transaction:
```cadence
import EVM from 0x...
transaction {
    prepare(account: auth(Storage) &Account) {
        let bal = EVM.Balance(attoflow: 0)
        bal.setFLOW(flow: 100.0)
        let coa = account.storage.borrow<auth(EVM.Withdraw) &EVM.CadenceOwnedAccount>(
            from: /storage/coa
        )!
        let vault <- coa.withdraw(balance: bal)
        // use vault to meet Cadence-side obligation
        destroy <- vault
    }
}
```
4. The transaction panics: `"EVM operations are temporarily paused"`.
5. Alice cannot recover her 100 FLOW for the duration of the pause. If the pause lasts days (as is common during security incidents), and Alice has a time-sensitive Cadence-side obligation, she suffers direct asset loss.

The test suite confirms this exact behavior: [8](#0-7)
