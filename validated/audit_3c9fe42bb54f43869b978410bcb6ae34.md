### Title
RCAT wallet omitted from `catToMojo` branch in `buildOfferRequest` causes 1,000,000× mojo inflation in NFT offers — (`packages/gui/src/components/offers/NFTOfferEditor.tsx`)

### Summary

`NFTOfferTokenSelector` exposes RCAT wallets as selectable offer tokens, but `buildOfferRequest` omits `WalletType.RCAT` from the `catToMojo` guard. When a user selects an RCAT wallet, the amount conversion silently falls through to `chiaToMojo`, submitting an offer with 1,000,000× more mojos than the user entered and saw displayed.

### Finding Description

`NFTOfferTokenSelector.tsx` line 59 includes `WalletType.RCAT` alongside `WalletType.CAT` and `WalletType.CRCAT` in the wallet filter, making RCAT wallets appear as valid offer token choices. [1](#0-0) 

`buildOfferRequest` in `NFTOfferEditor.tsx` line 493 checks only `[WalletType.CAT, WalletType.CRCAT]` for the `catToMojo` branch. `WalletType.RCAT` is absent, so it falls through to `chiaToMojo`. [2](#0-1) 

The balance display code (lines 144–148) correctly handles RCAT with `mojoToCAT` (scale 1e3), so the spendable balance shown to the user is in CAT units. [3](#0-2) 

The pre-submission balance guard (lines 587–591) compares `spendableBalance` (in CAT units, set via `mojoToCAT`) against `tokenAmount` (the user-entered CAT-unit value). This guard passes correctly for small inputs, so it does **not** catch the inflated mojo amount. [4](#0-3) 

The result: for a user-entered amount of `"1"` RCAT:
- Displayed: 1 RCAT
- `catToMojo("1")` = 1,000 mojos (correct)
- `chiaToMojo("1")` = 1,000,000,000 mojos (what is actually submitted)
- Ratio: **1,000,000× inflation**

### Impact Explanation

A user with an RCAT wallet who creates an NFT offer (buy or sell side) will have `createOfferForIds` called with a mojo amount 1,000,000× larger than the amount they entered and confirmed on screen. If the user holds sufficient RCAT balance, the offer is created and broadcast locking far more tokens than intended. This directly causes an accounting/balance discrepancy between what the user approved and what is committed on-chain, fitting the High impact category: *corruption of offer state that causes a user to submit the wrong amount*. [5](#0-4) 

### Likelihood Explanation

Any user who holds an RCAT wallet and uses the NFT offer editor can trigger this. No external attacker is required — the user's own normal interaction with the dropdown is sufficient. RCAT is a first-class wallet type exposed in the selector with no warning or guard. [6](#0-5) 

### Recommendation

Add `WalletType.RCAT` to the `catToMojo` guard in `buildOfferRequest`:

```typescript
// NFTOfferEditor.tsx line 493
const baseMojoAmount: BigNumber = [WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(tokenWalletInfo.walletType)
    ? catToMojo(tokenAmount)
    : chiaToMojo(tokenAmount);
```

This mirrors the existing correct handling in the balance display switch-case and in the token selector filter. [2](#0-1)
