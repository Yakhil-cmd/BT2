### Title
Single Malicious Participant Can Permanently Stall DKG, Presign, and Signing via Message Withholding — (`src/protocol/helpers.rs`)

---

### Summary

The `recv_from_others` helper and equivalent collection loops throughout the library require **all N participants** to respond before any protocol round can advance. There is no timeout, no threshold-based early exit, and no mechanism to proceed with fewer than N responses. A single malicious participant who withholds their message in any round causes every honest party to block indefinitely, permanently denying DKG, presign, or signing completion. This directly contradicts the documented BFT tolerance of MaxFaulty = (N−1)/3 malicious parties.

---

### Finding Description

**Root cause — `src/protocol/helpers.rs`**

`recv_from_others` loops on `while !seen.full()`, which only becomes true when every participant (except `me`) has delivered a message: [1](#0-0) 

`seen.full()` is satisfied only when all N slots are filled. If one participant never sends, `chan.recv(waitpoint).await?` blocks forever with no timeout path.

**Call sites that inherit this blocking behaviour:**

1. **DKG Round 1** — commitment-hash collection: [2](#0-1) 

2. **DKG Round 5** — secret-share collection: [3](#0-2) 

3. **Robust ECDSA Presign Round 2** — polynomial-evaluation collection: [4](#0-3) 

4. **Robust ECDSA Presign Round 3** — `W`-share collection (`while !wshares_map.full()`): [5](#0-4) 

5. **Robust ECDSA Signing** — coordinator collects all signature shares: [6](#0-5) 

6. **Echo-broadcast** — `reliable_broadcast_receive_all` exits only when every broadcast instance has completed (`state.iter().all(|x| x.finish_ready)`), so a participant who never sends their initial `Send` message stalls the broadcast layer used by DKG Rounds 1, 3, and 5.5: [7](#0-6) 

---

### Impact Explanation

**High — Permanent denial of signing, key generation, reshare, and refresh for honest parties.**

The library documentation explicitly states the BFT tolerance:

> `MaxFaulty = (N − 1) / 3` [8](#0-7) 

For N = 4 this gives MaxFaulty = 1. A single malicious participant is therefore within the documented tolerance. Yet the implementation requires all N parties to respond in every round. One withheld message in any round permanently stalls the entire protocol for all honest parties. The gap between the documented safety claim and the liveness requirement is the vulnerability.

---

### Likelihood Explanation

**High.** The attack requires no cryptographic capability, no leaked material, and no coordination with other parties. Any single participant — including a newly joined participant in a reshare — can execute it by simply not transmitting their round message. The attacker can choose the most damaging round (e.g., Round 5 of DKG after all commitments are public, or the signing round after a presignature has been consumed) to maximise disruption.

---

### Recommendation

1. **Threshold-based collection**: Accept responses from at least N − MaxFaulty parties before advancing, using Lagrange interpolation or the existing polynomial reconstruction to recover from missing shares where the protocol permits.
2. **Timeout with participant exclusion**: Introduce a configurable per-round deadline; mark non-responding participants as faulty and restart the round (or the protocol) with the remaining honest set.
3. **Identify and attribute blame**: Log which participant failed to respond so the application layer can take corrective action (e.g., slashing, exclusion from future sessions).

---

### Proof of Concept

```
Setup: N = 4 participants (P1, P2, P3, P4), MaxFaulty = 1, threshold = 2.
       P4 is malicious.

Round 1 (DKG): P4 sends its commitment hash honestly.
Round 2 (DKG): P4 broadcasts its commitment and proof of knowledge honestly.
Round 3 (DKG): P4 broadcasts its commitment and proof honestly.
Round 4 (DKG): P4 sends private share evaluations to P1, P2, P3 honestly.
Round 5 (DKG): P4 withholds its private share evaluation — sends nothing.

Result: P1, P2, P3 each call recv_from_others at waitpoint wait_round_3.
        seen.full() never becomes true because P4's slot is never filled.
        chan.recv(waitpoint).await blocks indefinitely.
        DKG never completes; no party obtains a key share.
        The same attack applies identically to presign (withhold polynomial
        evaluation in Round 2) and signing (coordinator never receives P4's
        signature share, so do_sign_coordinator blocks forever).
``` [9](#0-8) [3](#0-2)

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)

## Appendix: On the Discussion of Threshold Types

The BFT threshold states that the maximum number of faulty nodes a distributed system ($\mathsf{MaxFaulty}$) can tolerate while still reaching consensus has the following bound $\mathsf{MaxFaulty} = \frac{N - 1}{3}$ where $N$ is the total number of nodes.

The cryptographic threshold refers to the maximum number of malicious parties plus one ($\mathsf{threshold}$) that a scheme can tolerate without compromising security, assuming the existence of an underlying reliable broadcast channel. $\mathsf{threshold}$ is scheme dependent and can have a different value than $\mathsf{MaxFaulty}$. For instance, in the OT based ECDSA, $\mathsf{threshold}$ can be up to $N$, but in Robust ECDSA scheme $\mathsf{threshold}$ must not exceed $\frac{N - 1}{2}+1$.
