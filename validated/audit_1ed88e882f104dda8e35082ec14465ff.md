### Title
Unbounded `move_to_block` Loop in `load_hashchain` Causes Permanent Freezing of All EVM Transactions When Block Gap Is Large - (File: `engine/src/hashchain.rs`, `engine-hashchain/src/hashchain.rs`)

---

### Summary

`Hashchain::move_to_block` contains an unbounded `while` loop that iterates one step at a time from the stored `current_block_height` to the actual NEAR block height, performing a keccak hash computation per iteration. This loop is invoked unconditionally on every call to `with_hashchain` and `with_logs_hashchain` ‚Äî which wrap every EVM transaction entry point (`submit`, `call`, `deploy_code`) and every admin function (`pause_contract`, `resume_contract`, `set_owner`, etc.). If the Aurora Engine contract is dormant for a sufficiently long period while the hashchain is active, the gap between the stored block height and the current NEAR block height grows unboundedly. Any subsequent transaction will exhaust NEAR's 300 TGas limit inside `move_to_block`, permanently bricking all contract methods that go through the hashchain wrapper ‚Äî including the admin `pause_contract` needed to invoke the only recovery path (`start_hashchain`).

---

### Finding Description

**Root cause ‚Äî `move_to_block` is an O(N) loop with no bound:**

In `engine-hashchain/src/hashchain.rs`, `move_to_block` iterates one block at a time from `current_block_height` to `next_block_height`:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(...);
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration calls `compute_block_hashchain`, which invokes `txs_merkle_tree.compute_hash()` and then `keccak(~100 bytes)`. There is no cap on the number of iterations.

**Trigger ‚Äî `load_hashchain` calls `move_to_block` on every transaction:**

In `engine/src/hashchain.rs`, `load_hashchain` is called at the top of both `with_hashchain` and `with_logs_hashchain`:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
```

The gap `block_height - hashchain.current_block_height` grows with every NEAR block that passes without a transaction being processed by the engine.

**All EVM entry points are affected:**

`submit`, `submit_with_args`, `call`, and `deploy_code` all call `with_logs_hashchain`. Every admin method (`pause_contract`, `resume_contract`, `set_owner`, `set_upgrade_delay_blocks`, `register_relayer`, etc.) calls `with_hashchain`. All of these will fail with out-of-gas once the gap is large enough.

**The deadlock ‚Äî no recovery path when the contract is running:**

The only mechanism to reset the hashchain is `start_hashchain`, which does not use `with_hashchain`. However, `start_hashchain` enforces `require_paused`. The only way to pause the contract is `pause_contract`, which uses `with_hashchain` and therefore also fails. This creates an irrecoverable deadlock:

- `submit` ‚Üí `with_logs_hashchain` ‚Üí `load_hashchain` ‚Üí `move_to_block` ‚Üí **out-of-gas**
- `pause_contract` ‚Üí `with_hashchain` ‚Üí `load_hashchain` ‚Üí `move_to_block` ‚Üí **out-of-gas**
- `start_hashchain` ‚Üí `require_paused` ‚Üí **ERR** (contract is not paused)

---

### Impact Explanation

**Critical ‚Äî Permanent freezing of funds.**

Once the block gap exceeds the threshold that exhausts 300 TGas in `move_to_block`, every method that wraps `with_hashchain` or `with_logs_hashchain` becomes permanently unreachable. This includes all EVM transaction submission paths. Users holding ETH or ERC-20 tokens on Aurora cannot execute any EVM transactions to withdraw or transfer their assets. The admin cannot pause the contract to invoke `start_hashchain` because `pause_contract` itself goes through the same failing wrapper. The contract is permanently bricked with no on-chain recovery path.

---

### Likelihood Explanation

NEAR produces approximately one block per second. The hashchain is an opt-in feature, but once enabled it is active for all subsequent transactions. A dormancy period of tens of thousands of blocks (hours to days, depending on the per-iteration gas cost of `keccak` in the NEAR WASM runtime) is sufficient to trigger the freeze. Realistic scenarios include:

- A period of low or zero Aurora usage while the hashchain is active.
- A planned maintenance pause that runs longer than expected: the contract is paused (preventing hashchain updates), and when the admin tries to resume via `resume_contract` (which uses `with_hashchain`), the call fails ‚Äî and `pause_contract` is also blocked, preventing `start_hashchain`.
- A network-level incident that halts Aurora relayers for an extended period.

---

### Recommendation

1. **Cap the loop in `move_to_block`:** Introduce a maximum number of iterations per call (e.g., 1 or a small constant). If the gap exceeds the cap, either process only the capped number of blocks and persist the intermediate state, or skip empty blocks entirely by jumping directly to `next_block_height` when no transactions were recorded in the skipped range.

2. **Decouple `pause_contract` from the hashchain wrapper:** `pause_contract` is the only recovery entry point for the deadlock. It should not go through `with_hashchain`, or at minimum should be allowed to proceed even if `move_to_block` would fail.

3. **Relax the `require_paused` guard on `start_hashchain`:** Allow the key manager to call `start_hashchain` even when the contract is running, so the hashchain can be reset without first needing to pause.

---

### Proof of Concept

**Attacker-controlled entry path:**

Any NEAR account can call `submit` on the Aurora Engine contract after a sufficiently long dormancy period. No special privileges are required.

**Step-by-step:**

1. The Aurora Engine is deployed and `start_hashchain` is called at NEAR block height N, storing `current_block_height = N` in the hashchain state.
2. The contract receives no transactions for a large number of blocks (gap = G blocks).
3. Any user calls `submit` with a valid EVM transaction at block height N + G.
4. Execution enters `with_logs_hashchain` ‚Üí `load_hashchain` ‚Üí `hashchain.move_to_block(N + G)`.
5. The loop runs G iterations, each computing `keccak(~100 bytes)`. For G large enough to exhaust 300 TGas, the NEAR transaction fails with out-of-gas.
6. The hashchain state is not updated (transaction rolled back). The next call faces the same gap.
7. The admin calls `pause_contract` ‚Üí same `with_hashchain` path ‚Üí same out-of-gas failure.
8. `start_hashchain` is blocked by `require_paused`. The contract is permanently frozen.

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

    mut io: I,
    env: &E,
    function_name: &str,
    f: F,

where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);

    if let Some(mut hashchain) = maybe_hashchain {

            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}

pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }

    hashchain.add_block_tx(
        block_height,
        function_name!(),
        &input,
        &[],
        &Bloom::default(),
    )?;
    crate::hashchain::save_hashchain(&mut io, &hashchain)?;

    state.is_paused = false;
    state::set_state(&mut io, &state)?;

    Ok(())
}
