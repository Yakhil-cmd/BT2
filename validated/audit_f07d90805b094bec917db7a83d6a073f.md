### Title
Missing Access Control on `update_rewards` Allows Any Caller to Block Per-Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for the current block and causing the sequencer's legitimate call to revert with `REWARDS_ALREADY_UPDATED`. Repeated across blocks, this permanently freezes unclaimed yield for all stakers.

### Finding Description
The protocol spec for `update_rewards` states:

> **Access control**: Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` only calls `general_prerequisites()` (a pause check) and then asserts `current_block_number > self.last_reward_block.read()`. There is no `assert_caller_is_sequencer()` or equivalent guard.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // pause check only
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // global slot consumed

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // no rewards distributed
    }
    ...
}
```

`last_reward_block` is a single global storage variable (not per-staker). Writing it once per block prevents any further `update_rewards` call in that block for **any** staker. An attacker who calls `update_rewards(valid_staker, true)` early in a block:

1. Passes all checks (contract not paused, block is new, staker is valid and active).
2. Writes `last_reward_block = current_block`.
3. Returns immediately because `disable_rewards = true`, distributing **zero** rewards.
4. The sequencer's subsequent call for any staker in the same block reverts with `REWARDS_ALREADY_UPDATED`.

### Impact Explanation
Every block in which the attacker fires this transaction, **no staker receives their consensus-phase block reward**. Because `update_rewards` is the sole mechanism for distributing per-block rewards in the post-consensus phase, sustained griefing permanently freezes unclaimed yield for the entire staker set. This matches the allowed High impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation
- The entry point is a public, permissionless external function — no tokens, no stake, no privileged role required.
- The attacker only needs a valid (non-exiting, non-zero-balance) staker address, which is publicly observable on-chain.
- The cost is one transaction per block; the attacker pays gas but causes disproportionate damage to all stakers.
- The only mitigation would be sequencer-level transaction ordering (placing its own `update_rewards` call first), but this is an off-chain operational assumption not enforced by the contract, and it is not guaranteed.

### Recommendation
Add an explicit sequencer-only access control check at the top of `update_rewards`, consistent with the spec and analogous to the `assert_caller_is_attestation_contract()` guard already present in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this guard
    ...
}
```

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Attacker observes any valid staker address `S` with non-zero balance.
3. At the start of block `N`, attacker submits:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. The call succeeds: `last_reward_block` is written to `N`, no rewards are distributed.
5. The sequencer attempts `update_rewards(staker_address: any, disable_rewards: false)` in block `N`.
6. The call reverts: `current_block_number (N) > last_reward_block (N)` is false → `REWARDS_ALREADY_UPDATED`.
7. All stakers miss their block reward for block `N`.
8. Repeated every block → permanent freezing of all consensus-phase unclaimed yield.
