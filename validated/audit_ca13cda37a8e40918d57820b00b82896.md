Audit Report

## Title
SNS Governance Neuron Slot Exhaustion DoS Blocks New Participant Staking - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `claim_neuron` path inserts neurons with `DissolveDelaySeconds(0)` and enforces only a global `max_number_of_neurons` ceiling with no per-principal limit and no minimum dissolve delay at creation time. An unprivileged attacker can fill all neuron slots with immediately-dissolved zero-voting-power neurons, permanently blocking new governance participation until the attacker voluntarily disburses or existing governance passes a parameter change. The staked tokens are fully recoverable, making the net cost only transaction fees.

## Finding Description
`claim_neuron` constructs a new `Neuron` with `dissolve_state: Some(DissolveState::DissolveDelaySeconds(0))` before any balance check, then calls `self.add_neuron(neuron.clone())?` which internally calls `check_neuron_population_can_grow()`. [1](#0-0) 

`check_neuron_population_can_grow()` enforces only a single global ceiling — `(self.proto.neurons.len() as u64) + 1 > max_number_of_neurons` — with no per-principal count, no minimum dissolve delay, and no rate limit. [2](#0-1) 

`add_neuron` itself adds no further guards beyond heap size and the global population check. [3](#0-2) 

A neuron with `DissolveDelaySeconds(0)` is immediately in the `Dissolved` state: it has zero voting power, contributes nothing to governance, yet permanently occupies a slot in `self.proto.neurons` and is counted by `check_neuron_population_can_grow()`. The attacker is never forced to disburse; disbursement is entirely voluntary. The default ceiling is 200,000 neurons, but individual SNS instances may configure a much lower `max_number_of_neurons`. [4](#0-3) 

The `neuron_minimum_dissolve_delay_to_vote_seconds` parameter exists in `NervousSystemParameters` but is only enforced for voting and proposal submission, not at neuron creation time. [5](#0-4) 

## Impact Explanation
**High** — Application/platform-level DoS with concrete user harm: new users cannot stake SNS tokens and create neurons, permanently blocking new governance participation for the affected SNS instance until the attacker voluntarily disburses or existing neuron holders pass a `ManageNervousSystemParameters` proposal to raise `max_number_of_neurons`. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Application/platform-level DoS… not based on raw volumetric DDoS."* The attack is targeted at the SNS governance layer, not the IC protocol itself, and the remediation path (governance proposal) is non-trivial and requires existing holders to act.

## Likelihood Explanation
**Low-to-Medium** — The attacker must acquire `max_number_of_neurons × neuron_minimum_stake_e8s` SNS tokens. For SNS instances with a low configured `max_number_of_neurons` (e.g., 100–1,000) and a low token price, the capital requirement can be negligible. The tokens are fully recoverable via `Disburse` at any time (neurons are already in `Dissolved` state), so the attacker bears no permanent financial loss beyond transaction fees. The attack is repeatable: after disbursing, the attacker can immediately re-execute.

## Recommendation
1. Enforce a minimum dissolve delay at neuron creation time — require at least `neuron_minimum_dissolve_delay_to_vote_seconds` in `claim_neuron` before inserting the neuron into state.
2. Add a per-principal neuron count limit in `check_neuron_population_can_grow()` or `add_neuron()` to prevent a single actor from filling all slots.
3. Add an SNS governance action allowing the community to forcibly disburse dissolved neurons that have been inactive beyond a configurable threshold.

## Proof of Concept
1. Identify an SNS with a low `max_number_of_neurons` (e.g., 100) and cheap SNS tokens.
2. Transfer `max_number_of_neurons × neuron_minimum_stake_e8s` tokens to `max_number_of_neurons` distinct subaccounts of the SNS governance canister.
3. Call `manage_neuron` → `ClaimOrRefresh` for each subaccount. Each call reaches `claim_neuron`, which inserts a neuron with `DissolveDelaySeconds(0)` — immediately dissolved, zero voting power — and increments the global neuron count.
4. After `max_number_of_neurons` neurons are created, `check_neuron_population_can_grow()` returns `PreconditionFailed` for any subsequent `ClaimOrRefresh` call by any user.
5. New users attempting to stake receive `"Cannot add neuron. Max number of neurons reached."` indefinitely.
6. The attacker recovers all staked tokens at any time via `Disburse` (neurons are already `Dissolved`), bearing only transaction fees.
7. A deterministic integration test can reproduce this by setting `max_number_of_neurons: Some(3)` in `NervousSystemParameters`, claiming 3 neurons with `DissolveDelaySeconds(0)` from one principal, then asserting that a fourth `ClaimOrRefresh` from a different principal returns `ErrorType::PreconditionFailed`.

    fn add_neuron(&mut self, neuron: Neuron) -> Result<(), GovernanceError> {
        let neuron_id = neuron
            .id
            .as_ref()
            .expect("Neuron must have a NeuronId")
            .clone();

        // New neurons are not allowed when the heap is too large.
        self.check_heap_can_grow()?;

        // New neurons are not allowed when the maximum configured is reached
        self.check_neuron_population_can_grow()?;

        if self.proto.neurons.contains_key(&neuron_id.to_string()) {
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                format!("Cannot add neuron. There is already a neuron with id: {neuron_id}"),

            &self.proto.id_to_nervous_system_functions,
            &neuron,
        );

        add_neuron_to_follower_index(&mut self.topic_follower_index, &neuron);

        self.proto.neurons.insert(neuron_id.to_string(), neuron);

        Ok(())
    }

    /// Checks whether new neurons can be added or whether the maximum number of neurons,
    /// as defined in the nervous system parameters, has already been reached.
    fn check_neuron_population_can_grow(&self) -> Result<(), GovernanceError> {
        let max_number_of_neurons = self
            .nervous_system_parameters_or_panic()
            .max_number_of_neurons
            .expect("NervousSystemParameters must have max_number_of_neurons");

        if (self.proto.neurons.len() as u64) + 1 > max_number_of_neurons {
            return Err(GovernanceError::new_with_message(
                ErrorType::PreconditionFailed,
                "Cannot add neuron. Max number of neurons reached.",
            ));
        }

        Ok(())
    }
