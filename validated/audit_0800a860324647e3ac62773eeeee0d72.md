### Title
Sensitive Secret Material Exposed via `Debug` Trait on `KeygenOutput`, `TripleShare`, and `PresignOutput` Types — (`src/lib.rs`, `src/ecdsa/ot_based_ecdsa/triples/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Multiple production types holding long-term private signing shares, Beaver triple secret scalars, and presignature nonce shares unconditionally derive `#[derive(Debug)]`. Any caller, logging framework, error handler, or tracing middleware that formats these types with `{:?}` will emit the raw secret material in plaintext. This is the direct Rust-library analog of the pprof endpoint being bound to `0.0.0.0` instead of `localhost`: a debug interface that was never hardened against exposure silently leaks the most sensitive values in the system.

---

### Finding Description

The following production structs derive `Debug` while containing secret cryptographic material:

**1. `KeygenOutput<C>` — long-term private signing share** [1](#0-0) 

`private_share: SigningShare<C>` is the participant's long-term secret key share produced by DKG/reshare/refresh. `Debug` is derived without suppressing this field, so `format!("{:?}", keygen_output)` prints the raw scalar.

**2. `TripleShare` — Beaver triple secret scalars** [2](#0-1) 

The module's own documentation states explicitly:

> "It's important that the value of the underlying scalars in the triple is kept secret, otherwise the private key used to create a signature with that triple could be recovered." [3](#0-2) 

Yet `a`, `b`, `c` are all printed by the derived `Debug` impl.

**3. `PresignOutput` (OT-based) — nonce shares `k` and `sigma`** [4](#0-3) 

**4. `RerandomizedPresignOutput` (OT-based) — rerandomized nonce shares** [5](#0-4) 

**5. `PresignOutput` (Robust ECDSA) — nonce shares `c`, `e`, `alpha`, `beta`** [6](#0-5) 

**6. `RerandomizedPresignOutput` (Robust ECDSA)** [7](#0-6) 

**7. `PresignArguments` (OT-based) — transitively contains `KeygenOutput`** [8](#0-7) 

**8. `ScalarWrapper` (CKD) — BLS secret scalar** [9](#0-8) 

---

### Impact Explanation

- **`KeygenOutput.private_share`**: Disclosure of the long-term private signing share. With this value an attacker can reconstruct the full threshold secret once they collect shares from `t` participants, enabling unauthorized signature creation for any message. This is **Critical** under the allowed scope: *"Extraction, reconstruction, or disclosure of private signing shares."*

- **`TripleShare.{a,b,c}`**: The module explicitly documents that leaking these scalars allows recovery of the private signing key used with that triple. This is **Critical**: *"disclosure of … presign secrets, nonce material."*

- **`PresignOutput.{k, sigma}` / `{c, e, alpha, beta}`**: Nonce share leakage in ECDSA directly enables private key extraction via the standard `k`-reuse / nonce-recovery attack. **Critical**.

---

### Likelihood Explanation

This is a library. Callers routinely log protocol outputs for observability, debugging, and error reporting. Rust's `tracing` and `log` crates, `anyhow`/`thiserror` error chains, and test harnesses all use `{:?}` formatting. The `Debug` bound is also required by many generic bounds in the ecosystem (e.g., `assert_eq!`, `unwrap_or_else` error messages). Because the derive is unconditional and the types are `pub`, any downstream consumer that logs or asserts on these values will silently emit secret material. No attacker action is required beyond being in a position to read logs or error output.

---

### Recommendation

1. **Remove `Debug` from all types containing secret scalars.** For `KeygenOutput`, `TripleShare`, `PresignOutput`, and `RerandomizedPresignOutput`, either delete `Debug` from the derive list entirely or implement a manual `Debug` that redacts secret fields:

```rust
impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        f.debug_struct("KeygenOutput")
            .field("private_share", &"[REDACTED]")
            .field("public_key", &self.public_key)
            .finish()
    }
}
```

2. Apply the same pattern to `TripleShare`, both `PresignOutput` types, both `RerandomizedPresignOutput` types, `PresignArguments`, and `ScalarWrapper`.

3. Consider adopting the `secrecy` crate (`Secret<T>`) to wrap scalar fields so that `Debug` redaction is enforced at the type level rather than per-struct.

---

### Proof of Concept

```rust
use threshold_signatures::{keygen, KeygenOutput, participants::Participant, thresholds::ReconstructionLowerBound};

// After running DKG and obtaining keygen_output: KeygenOutput<C>
// Any of the following will print the raw private signing share:
println!("{:?}", keygen_output);
// or via tracing:
tracing::debug!(?keygen_output, "DKG complete");
// or via assert_eq! on test failure:
assert_eq!(keygen_output, other_output); // prints both on failure

// For TripleShare, the module itself documents the consequence:
// "the private key used to create a signature with that triple could be recovered"
println!("{:?}", triple_share); // prints a, b, c in plaintext
```

The `Debug` derive is unconditional and the types are part of the public API surface, so no special privileges are needed — any library caller can trigger this exposure through ordinary Rust idioms.
