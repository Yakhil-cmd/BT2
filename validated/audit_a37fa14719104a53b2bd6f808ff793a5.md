### Title
`skip_stateful_validations` bypasses `__validate__` for nonce-1 invoke txs via any pooled tx — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call only when a pending `deploy_account` transaction exists for a fresh account (UX convenience). However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the pool for that address, not exclusively a `deploy_account` tx. An attacker who controls a deployed contract with nonce 0 can pre-seed the pool with a valid future-nonce invoke tx, then submit a nonce-1 invoke tx with an invalid signature that is admitted to the mempool without running `__validate__`.

### Finding Description

`skip_stateful_validations` fires when:

```
tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)
```

and `account_tx_in_pool_or_recent_block` returns `true`. [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool for that address. [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate = false` in `ExecutionFlags`. Inside `StatefulValidator::perform_validations`, the `__validate__` entry-point call is then skipped entirely:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
``` [4](#0-3) 

The execution flags are set here: [5](#0-4) 

`perform_pre_validation_stage` (nonce increment, fee bounds, balance) still runs, but the account's signature-checking `__validate__` Cairo function is never invoked. [6](#0-5) 

The production config sets `max_allowed_nonce_gap = 200`, so a nonce-2 invoke tx from an account with on-chain nonce 0 passes `validate_nonce` without issue: [7](#0-6) [8](#0-7) 

### Impact Explanation

An attacker can inject an invoke transaction with an **invalid signature** (or any payload that would fail `__validate__`) into the mempool without the gateway running the account's validation entry point. This is a direct violation of the admission invariant: every invoke tx in the mempool must have passed `__validate__`. The admitted invalid tx occupies mempool capacity, can displace legitimate transactions when the pool is full, and forces the batcher to waste execution budget discovering the failure at sequencing time.

Matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The precondition is low-cost: the attacker needs only a contract deployed on-chain with nonce 0 (achievable via the `deploy` syscall from any other account) and the ability to submit two RPC transactions. `max_allowed_nonce_gap = 200` makes the nonce-2 seed tx trivially admissible. No privileged access is required.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check in `skip_stateful_validations` with a check that specifically verifies a **`deploy_account` transaction** exists for the address in the pool or a recent block. The current check conflates "account has any pending tx" with "account has a pending deploy_account tx", which is the root cause.

Concretely, add a `deploy_account_tx_in_pool_or_recent_block(address)` query to the mempool that only returns `true` when the pooled or recently committed transaction for that address is of type `DeployAccount`.

### Proof of Concept

**Setup:** Contract `B` is deployed on-chain via `deploy` syscall from another account. `B` has a `__validate__` function that verifies ECDSA signatures. On-chain nonce of `B` = 0.

**Step 1 — Seed the pool:**
Submit `invoke_tx_seed` from `B` with `nonce = 2`, valid signature. Gateway checks:
- `validate_nonce`: `0 ≤ 2 ≤ 200` ✓
- `skip_stateful_validations`: `nonce != 1` → returns `false`
- `run_validate_entry_point`: `__validate__` runs, valid signature passes ✓
- `account_tx_in_pool_or_recent_block(B)` now returns `true`

**Step 2 — Inject invalid tx:**
Submit `invoke_tx_attack` from `B` with `nonce = 1`, **invalid/empty signature**. Gateway checks:
- `validate_nonce`: `0 ≤ 1 ≤ 200` ✓
- `validate_by_mempool`: no duplicate nonce ✓
- `skip_stateful_validations`: `nonce == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(B) == true` → returns `true`
- `run_validate_entry_point` called with `skip_validate = true` → `__validate__` **not called**
- `invoke_tx_attack` is **admitted to the mempool** with an invalid signature

**Result:** The mempool now contains a nonce-1 tx from `B` that would fail `__validate__` at execution time. The batcher will reject it, but it occupied a mempool slot and forced a wasted execution attempt. Repeated at scale this constitutes a targeted mempool-pollution attack against any account whose address is known.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
