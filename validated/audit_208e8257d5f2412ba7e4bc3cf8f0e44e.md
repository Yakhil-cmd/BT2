### Title
Uninitialized Aurora Engine Contract — Missing Caller Check in `new()` Allows Attacker to Seize Ownership - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine's `new()` initialization function does not validate the identity of the caller. Any NEAR account can call it before the legitimate deployer does. Because NEAR contract deployment and initialization are two separate transactions, an attacker who monitors the network can front-run the deployer's `new()` call, set themselves as `owner_id`, and gain full administrative control over the engine.

---

### Finding Description

The public entry point `new()` in `engine/src/lib.rs` dispatches to `contract_methods::admin::new()`:

```rust
// engine/src/lib.rs:76-80
#[unsafe(no_mangle)]
pub extern "C" fn new() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::new(io, &env)
```

The implementation in `engine/src/contract_methods/admin.rs` is:

```rust
// engine/src/contract_methods/admin.rs:56-88
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
```

The only guard is `ERR_ALREADY_INITIALIZED` — there is **no check on `env.predecessor_account_id()`**. The `env` object is used only for `env.block_height()` and `env.current_account_id()` (hashchain bookkeeping), never to authenticate the caller. [1](#0-0) 

In NEAR Protocol, deploying a contract and calling its initialization function are **two separate transactions**. The `deploy_and_init` helper in the workspace illustrates this two-step pattern:

```rust
// engine-workspace/src/lib.rs:107-125
let contract = account.deploy(&self.code...).await?;
...
engine.new(self.chain_id, self.owner_id, ...).transact().await...
``` [2](#0-1) 

Between the `deploy` transaction and the `new` transaction there is an observable window on the NEAR network. Any account can submit a `new` call during this window. The first successful call wins and permanently sets `owner_id`, `chain_id`, and `key_manager` in `EngineState`. [3](#0-2) 

---

### Impact Explanation

`owner_id` is the privileged account that controls the most dangerous administrative operations:

- `upgrade()` — deploys arbitrary new contract code to the engine account, replacing all logic.
- `stage_upgrade()` / `deploy_upgrade()` — stages and deploys replacement code.
- `set_eth_connector_contract_account()` — redirects the ETH connector to an attacker-controlled contract.
- `factory_update()` — replaces the XCC router code stored in the engine. [4](#0-3) 

An attacker who becomes `owner_id` can call `upgrade()` with a WASM payload that transfers all bridged ETH and ERC-20 balances to the attacker, or that redirects all future deposits. This constitutes **direct theft of all user funds at rest and in motion** — a Critical impact.

---

### Likelihood Explanation

NEAR transactions are publicly broadcast before finalization. An attacker running a NEAR node or watching the RPC can observe the `DeployContract` action for the `aurora` account and immediately submit a `new()` call with attacker-controlled arguments. Because NEAR processes receipts asynchronously across shards, the attacker's `new()` receipt can be scheduled in the same or the very next block before the deployer's `new()` receipt. No special privilege, leaked key, or governance capture is required — only the ability to submit a NEAR transaction.

---

### Recommendation

Add a caller check at the top of `new()` so that only the account that owns the contract (i.e., `env.current_account_id()`) can initialize it:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    // Only the contract account itself may initialize the engine.
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    ...
}
```

Alternatively, combine the `DeployContract` and `FunctionCall { name: "new", ... }` actions into a single NEAR batch transaction at deployment time, so both actions execute atomically and no external `new()` call can be inserted between them.

---

### Proof of Concept

1. Attacker monitors the NEAR RPC for a `DeployContract` action targeting the `aurora` account.
2. Immediately after observing the deploy transaction, attacker submits:
   ```
   near call aurora new '{"chain_id": [...], "owner_id": "attacker.near", "upgrade_delay_blocks": 0}' --accountId attacker.near
   ```
3. If the attacker's receipt is processed before the deployer's `new()` receipt, `state::set_state` stores `owner_id = "attacker.near"`.
4. The deployer's subsequent `new()` call returns `ERR_ALREADY_INITIALIZED` and fails silently from the deployer's perspective.
5. Attacker calls `upgrade()` with a malicious WASM payload that drains all bridged ETH and ERC-20 token balances. [5](#0-4) [6](#0-5)

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
