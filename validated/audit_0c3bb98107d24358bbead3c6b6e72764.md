Audit Report

## Title
Unchecked Index Panic on Empty `EnvelopePair` Vec Crashes Rosetta ICP Process — (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`)

## Summary
The `construction_hash` handler unconditionally accesses `envelope_pairs[0]` after finding a `RequestType::Send` entry via `find()`, without verifying the inner `Vec<EnvelopePair>` is non-empty. An unauthenticated attacker can POST a single crafted hex-CBOR `signed_transaction` containing `(RequestType::Send, vec![])` to `/construction/hash`, triggering a Rust index-out-of-bounds panic that terminates the Rosetta ICP process.

## Finding Description
`SignedTransaction::from_str` in `rs/rosetta-api/icp/src/models.rs` (lines 37–48) performs only CBOR deserialization with no structural validation — it accepts any well-formed CBOR, including a `requests` vec containing `(RequestType::Send, vec![])`. The type alias `pub type Request = (RequestType, Vec<EnvelopePair>)` (line 58) places no non-empty constraint on the inner vec.

In `construction_hash` (`rs/rosetta-api/icp/src/request_handler/construction_hash.rs`, lines 18–28), the handler calls `.find(|(rt, _)| rt.is_transfer())` on the deserialized requests. `RequestType::Send` satisfies `is_transfer()` (confirmed at `rs/rosetta-api/icp/src/request_types.rs`, lines 137–139). Once found, the handler immediately executes:

```rust
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pairs[0].update,  // line 27 — panics if vec is empty
)
```

This is a direct `Index` trait call on a `Vec`. When the vec is empty, Rust panics with an index-out-of-bounds. There is no `.get(0)` or `.first()` guard anywhere in this code path.

The safe pattern already exists in `rs/rosetta-api/icp/src/request.rs` (lines 237–239):
```rust
let payload: &models::EnvelopePair = calls
    .first()
    .ok_or_else(|| ApiError::invalid_request("No request payload provided."))?;
```
But `construction_hash` does not go through that path — it directly destructures the tuple from `find()` and indexes the vec without any bounds check.

## Impact Explanation
The Rosetta ICP node is a standalone HTTP server process serving as a financial integration component (ICP Rosetta API, explicitly in scope). A Rust panic with the default panic handler (or under `panic = "abort"`) terminates the process. A single malformed request crashes the entire Rosetta node, making it unavailable to all clients until manually restarted. This is a non-volumetric, single-request denial-of-service against the Rosetta ICP process. This matches the **High** bounty impact: "Application/platform-level DoS, crash... or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The `/construction/hash` endpoint requires no authentication. The attacker needs only network access to the Rosetta node and the ability to POST JSON. Crafting the payload is trivial: serialize `SignedTransaction { requests: vec![(RequestType::Send, vec![])] }` with `serde_cbor`, hex-encode it, and submit it as the `signed_transaction` field. No keys, credentials, or privileged access are required. The attack is repeatable and deterministic.

## Recommendation
Replace the unchecked index at line 27 of `construction_hash.rs` with a bounds-checked access that returns an `ApiError` on empty input, mirroring the existing safe pattern in `request.rs`:

```rust
let envelope_pair = envelope_pairs.first().ok_or_else(|| {
    ApiError::invalid_request("No envelope pairs provided for transfer request.")
})?;
TransactionIdentifier::try_from_envelope(
    request_type.clone(),
    &envelope_pair.update,
)
```

## Proof of Concept
```rust
#[test]
fn test_construction_hash_empty_envelope_pairs_panics() {
    use std::str::FromStr;
    use crate::models::{SignedTransaction, EnvelopePair};
    use crate::request_types::RequestType;

    let malicious = SignedTransaction {
        requests: vec![(RequestType::Send, vec![])],
    };
    let hex = malicious.to_string(); // hex-encoded CBOR

    let signed = SignedTransaction::from_str(&hex).unwrap();
    // find() succeeds because RequestType::Send satisfies is_transfer()
    // envelope_pairs[0] panics — process aborts under panic=abort
    let result = std::panic::catch_unwind(|| {
        let (_, envelope_pairs) = signed.requests.iter()
            .rev()
            .find(|(rt, _)| rt.is_transfer())
            .unwrap();
        let _ = &envelope_pairs[0]; // panics here
    });
    assert!(result.is_err(), "Should have panicked — this is the bug");
}
```

Under `panic = "abort"` (typical for production binaries), `catch_unwind` does not help — the process terminates immediately upon the first such request.
