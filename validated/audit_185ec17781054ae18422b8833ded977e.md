### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required — (`File: L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is `external payable` and accepts ETH to cover Starknet L1→L2 messaging fees. However, when `amountToMint == 0` (no pending mint requests), the function exits the `if` block without consuming or refunding `msg.value`. Any ETH sent by the caller in this case is permanently locked in the contract with no recovery path.

---

### Finding Description

`tick()` is callable by any address with no access control. Its intended flow is:

1. Check `requiredMinting()` — if there are pending L2 mint requests, consume them, mint tokens, and bridge them to L2 using two payable calls that split `msg.value`.
2. If there are no pending requests (`amountToMint == 0`), the function returns immediately.

The critical code path:

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... msg.value is consumed here via bridge and messaging calls
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // If amountToMint == 0: msg.value is silently trapped
}
```

When `amountToMint == 0`, `msg.value` is never forwarded to any external call and never refunded. The contract has no `withdraw` or ETH-recovery function. The ETH is permanently locked. [1](#0-0) 

---

### Impact Explanation

Any ETH sent to `tick()` when there are no pending mint requests is permanently locked in the `RewardSupplier` contract. There is no admin withdrawal function, no `receive`/`fallback` refund path, and no sweep mechanism visible in the contract or its storage base.

This maps to the allowed impact: **Griefing with no profit motive but damage to users or protocol (Medium)** — and also **Temporary/permanent freezing of funds** depending on the amount locked.

---

### Likelihood Explanation

`tick()` is a public, permissionless function intended to be called by off-chain keepers or bots on a regular cadence. A caller cannot atomically verify that `amountToMint > 0` before sending ETH in the same transaction (without a simulation). A keeper that sends ETH speculatively — or one that races with another keeper — will lose their ETH when the condition is already false. This is a realistic, recurring scenario in production keeper infrastructure.

---

### Recommendation

Add an ETH refund when `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic
    } else if (msg.value > 0) {
        // Refund unused ETH
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, remove `payable` from `tick()` and require callers to pre-fund the contract separately, or add a `require(amountToMint > 0)` guard before accepting ETH.

---

### Proof of Concept

1. Off-chain keeper calls `tick{value: 0.01 ether}()`.
2. `requiredMinting()` returns `(0, 0)` because no L2 mint requests are pending.
3. The `if (amountToMint > 0)` block is skipped entirely.
4. `tick()` returns. The 0.01 ETH is now held by `RewardSupplier` with no recovery path.
5. The keeper's ETH is permanently lost. [2](#0-1) [3](#0-2)

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)

            // Send a totalSupply update to L2MintCurve.

            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),

                messagePayload
            );
        }
    }
