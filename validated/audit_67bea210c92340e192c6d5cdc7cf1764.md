### Title
WalletConnect Commands Execute Against "Logged-Out" Wallet Due to No-Op `handleLogOut` — (`packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary
`handleLogOut` is an intentional no-op stub. Clicking the logout button navigates the UI to `/` but leaves the daemon fully logged in under the previous fingerprint. Because `WalletConnectProvider` is mounted at the app-provider level (not inside a wallet route), its `session_request` listener remains active. The fingerprint guard in `useWalletConnectCommand` compares `currentFingerprint` (still the old fingerprint, because the daemon was never told to log out) against `pair.fingerprint` (also the old fingerprint) — the check passes, and the command is dispatched to the daemon.

### Finding Description

**Step 1 — No-op logout.**
`handleLogOut` does nothing: [1](#0-0) 

`LogoutButton` calls `logOut()` then `navigate('/')`: [2](#0-1) 

No daemon RPC is issued. The daemon remains logged in with the previous fingerprint.

**Step 2 — WalletConnectProvider stays mounted.**
`WalletConnectProvider` is registered in `AppProviders.tsx` at the application root, not inside any wallet-specific route. It is never unmounted by navigation to `/`. The `session_request` listener registered in its `useEffect` remains active: [3](#0-2) 

**Step 3 — Fingerprint guard passes.**
`useWalletConnectCommand` fetches `currentFingerprint` via `useGetLoggedInFingerprintQuery`. Because the daemon was never logged out, this query still returns the previous fingerprint. The guard: [4](#0-3) 

evaluates `currentFingerprint !== pair.fingerprint` as `X !== X` → `false`. No error is thrown.

**Step 4 — Command dispatched.**
`dispatchAsPair` is called with the still-valid pair topic and the command reaches the daemon: [5](#0-4) 

### Impact Explanation
A WalletConnect dapp that had a previously approved session can send `chia_sendTransaction`, `chia_signCoinSpends`, or any other approved command after the user clicks "Log Out." The daemon executes the spend against the wallet the user believes is no longer active. This is unauthorized transaction execution against a wallet the user has explicitly attempted to deactivate — meeting the Critical threshold (unauthorized spend/transfer of XCH/CAT/NFT) or at minimum High (bypass of WalletConnect approval/session termination with direct asset impact).

### Likelihood Explanation
Any dapp with an existing approved WalletConnect session can exploit this. No special privileges are needed beyond the already-granted session. The window is open for as long as the GUI process remains running after logout (which is the normal usage pattern — users log out to switch wallets, not to quit the app).

### Recommendation
`handleLogOut` must issue a daemon logout RPC (e.g., `log_out` or equivalent) before navigating. Additionally, `WalletConnectProvider` should either: (a) disconnect all active sessions on logout, or (b) check that the current UI fingerprint (not just the daemon fingerprint) matches `pair.fingerprint` before processing any `session_request`. The daemon fingerprint alone is insufficient as a guard when the UI logout path deliberately avoids calling the daemon.

### Proof of Concept
1. Establish a WalletConnect session with a dapp for fingerprint X; grant `chia_sendTransaction`.
2. Click the "Log Out" button in the GUI — UI navigates to `/`, daemon remains logged in.
3. From the dapp, send a `session_request` for `chia_sendTransaction` with a valid recipient and amount.
4. Observe: `useWalletConnectCommand.handleProcess` runs, `currentFingerprint === pair.fingerprint === X`, guard passes, `dispatchAsPair` is called, and the daemon executes the transaction against the "logged-out" wallet without any user confirmation prompt.
