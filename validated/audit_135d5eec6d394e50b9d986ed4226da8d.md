### Title
Signature Validation Bypass via Overly Broad `account_tx_in_pool_or_recent_block` Check in `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry-point for an invoke transaction with `nonce=1` when the account's deploy-account transaction is still pending. The guard it uses — `account_tx_in_pool_or_recent_block` — checks whether **any** transaction from the sender address exists in the mempool or a recent block, not whether a `deploy_account` transaction specifically exists. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a crafted invoke transaction with `nonce=1` from the victim's address carrying an arbitrary (or empty) signature. The gateway skips `__validate__`, admits the transaction to the mempool, and the victim's legitimate first invoke transaction is blocked (DuplicateNonce) until the attacker's transaction is rejected during block execution. The attacker pays nothing because validation failures do not charge fees.

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have passed account signature validation (either via the `__validate__` entry point or via a legitimately scoped skip).

**Root cause — `skip_stateful_validations`:**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())   // ← checks ANY tx type
                .await
                ...;
        }
    }
    Ok(false)
}
```

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect. `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type from the address:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)   // staged or committed — any tx type
        || self.tx_pool.contains_account(account_address)  // any tx in pool
}
```

**Execution path that admits the attacker's transaction:**

1. `extract_state_nonce_and_run_validations` reads `account_nonce = 0` from state (account not yet deployed).
2. `run_pre_validation_checks` → `validate_state_preconditions` passes (nonce 1 ≥ 0).
3. `validate_by_mempool` passes (no duplicate hash, nonce 1 ≥ resolved nonce 0).
4. `skip_stateful_validations` fires: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block` returns `true` (victim's `deploy_account` is in the pool) → returns `true`.
5. `run_validate_entry_point` is called with `skip_validate = true` → `execution_flags.validate = false`.
6. Inside `StatefulValidator::perform_validations`, `perform_pre_validation_stage` runs (nonce and fee-balance checks pass because the victim's address is pre-funded), then the function returns `Ok(())` **without calling `__validate__`**.
7. The attacker's transaction is forwarded to the mempool via `mempool_client.add_tx`.

**Why `perform_pre_validation_stage` does not block the attacker:**

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 355-372
pub fn perform_pre_validation_stage<S: State + StateReader>(
    &self, state: &mut S, tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
    if self.execution_flags.charge_fee {
        self.check_fee_bounds(tx_context)?;
        verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
    }
    self.validate_proof_facts(&tx_context.block_context, state)?;
    Ok(())
}
```

`handle_nonce` uses `strict_nonce_check = false`, so nonce 1 ≥ 0 passes. `verify_can_pay_committed_bounds` checks the victim's pre-funded STRK/ETH balance, which is non-zero (Starknet accounts must be pre-funded before deployment). The attacker sets resource bounds to any value that passes the stateless validator's `min_gas_price` check.

**What happens during block execution:** The batcher retrieves the attacker's transaction from the mempool and executes it with `execution_flags.validate = true`. `__validate__` runs, fails (wrong/missing signature), and the transaction is rejected. The mempool is notified via `commit_block` with the rejected hash. The victim can then resubmit their nonce=1 transaction — but the attacker can immediately repeat the attack at zero cost.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway admits a transaction that carries an arbitrary signature and will unconditionally fail `__validate__` during block execution. The admitted transaction occupies the `(address, nonce=1)` slot in the mempool. The victim's legitimate first invoke transaction after deployment is rejected with `DuplicateNonce` (or must pay a higher fee to trigger fee escalation). Because failed-validation transactions are not charged fees, the attacker can repeat this at zero cost for every block cycle until the victim's `deploy_account` is committed and the account nonce advances past 0, permanently preventing the victim from landing their first invoke in the same block as their deployment.

---

### Likelihood Explanation

**Likelihood: Medium.**

The attacker must:
1. Monitor the mempool for `deploy_account` transactions (public information).
2. Submit a crafted invoke transaction with `nonce=1` before the victim's legitimate invoke transaction is admitted.

This is a narrow race window but is fully automatable with a mempool-watching bot. No privileged access is required. The victim's address and the required nonce are both observable on-chain/in-mempool.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the address. Either:

1. Add a dedicated `deploy_account_in_pool(address)` query to the mempool that inspects transaction types, or
2. Store a separate set of addresses for which a `deploy_account` transaction is currently pending, and query that set in `skip_stateful_validations`.

This ensures the validation skip is only granted when the account is genuinely in the process of being deployed, not whenever any transaction from the address happens to be present.

---

1. Victim submits RpcDeployAccountTransaction for address V (nonce=0).

   → account_tx_in_pool_or_recent_block(V) == true.

     nonce          = 1

       where Y >= min_gas_price (passes stateless validator)

3. Gateway stateful path:
   - get_nonce_from_state(V) → 0
   - validate_state_preconditions: nonce 1 in [0, 0+max_gap] ✓
   - validate_by_mempool: nonce 1 >= resolved_nonce 0 ✓, no duplicate ✓

       tx.nonce()==1 ✓, account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(V) == true  ← deploy_account(V) is in pool

   → validate_by_mempool: DuplicateNonce(V, 1) → REJECTED.

5. Attacker's tx reaches block execution:
   → __validate__ runs with invalid signature → FAILS → tx rejected.
   → mempool notified via commit_block(rejected=[attacker_tx_hash]).

6. Attacker repeats step 2 immediately. Zero cost per iteration.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
