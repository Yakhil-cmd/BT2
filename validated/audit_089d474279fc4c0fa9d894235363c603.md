### Title
Excess NEAR Attached to EVM Entry Points Is Permanently Locked in the Aurora Engine Contract - (File: engine/src/contract_methods/evm_transactions.rs)

---

### Summary

The primary EVM transaction entry points (`submit`, `submit_with_args`, `call`, `deploy_code`) in Aurora Engine never inspect or refund `env.attached_deposit()`. Any yoctoNEAR a caller attaches to these calls is silently absorbed into the Aurora Engine contract with no recovery path, directly mirroring the "excess value permanently locked" pattern from the reference report.

---

### Finding Description

In NEAR Protocol, any function call may carry an `attached_deposit` of yoctoNEAR. If the callee does not forward or refund that deposit and the call succeeds, the NEAR is permanently credited to the callee contract's account balance with no on-chain mechanism to retrieve it.

The four public EVM entry points in Aurora Engine:

- `submit` — submits a raw RLP-encoded Ethereum transaction
- `submit_with_args` — same, with additional relayer arguments
- `call` — allows a NEAR account to call an EVM contract
- `deploy_code` — deploys EVM bytecode

…all share the same pattern: they read `io.read_input()`, execute EVM logic, and return a result. None of them call `env.assert_one_yocto()`, check `env.attached_deposit() == 0`, or issue a refund promise for any attached NEAR. [1](#0-0) 

Compare this with functions that correctly enforce an exact deposit: [2](#0-1) [3](#0-2) 

Those functions call `env.assert_one_yocto()`, which enforces `attached_deposit == 1` and reverts otherwise: [4](#0-3) 

The EVM entry points have no equivalent guard and no refund path.

---

### Impact Explanation

**Impact: High — Temporary or Permanent Freezing of Funds.**

Any NEAR attached to a successful `submit`, `submit_with_args`, `call`, or `deploy_code` call is permanently absorbed into the Aurora Engine contract. There is no administrative withdrawal function for arbitrary NEAR balances, and NEAR Protocol does not automatically refund deposits on successful calls. The funds are irrecoverable by the sender.

---

### Likelihood Explanation

**Likelihood: Low-to-Medium.**

Realistic triggering scenarios:

1. **`call` entry point confusion**: A NEAR-native user wants to invoke an EVM contract that requires ETH value. They see `CallArgs` accepts a `value` field but mistakenly also attach NEAR to the NEAR-level call, believing it is required to fund the EVM value transfer. The EVM call succeeds using the sender's EVM balance; the attached NEAR is locked.

2. **Relayer or wallet UI bug**: A wallet or relayer that wraps `submit` could inadvertently attach NEAR (e.g., from a misconfigured gas-payment flow), locking the NEAR permanently.

3. **Direct user error**: A user unfamiliar with the NEAR/Aurora distinction attaches NEAR to `submit` thinking it pays for Aurora gas.

The `call` path is the most realistic because it is explicitly designed for NEAR accounts interacting with EVM contracts, making the "attach NEAR to pay for EVM value" mistake intuitive.

---

### Recommendation

Add a zero-deposit assertion at the top of each EVM entry point:

```rust
// In submit, submit_with_args, call, deploy_code:
if env.attached_deposit() != 0 {
    return Err(errors::ERR_INVALID_AMOUNT.into());
}
```

Alternatively, issue a refund promise for any non-zero deposit before returning. The pattern used by `withdraw` and `ft_transfer` (enforcing exactly 1 yocto) demonstrates the correct approach for functions that do require a deposit. [5](#0-4) 

---

### Proof of Concept

**Entry path for `call`:**

1. User holds ETH on Aurora and wants to call EVM contract `0xABCD...` with `value = 1 ETH`.
2. User constructs `CallArgs { contract: 0xABCD..., value: [1 ETH in Wei], input: [...] }` and calls the Aurora Engine `call` method on NEAR, attaching `1_000_000_000_000_000_000_000_000` yoctoNEAR (1 NEAR) believing it funds the EVM value.
3. `call` dispatches to `engine.call_with_args(args, handler)`, which executes the EVM call using the sender's EVM ETH balance for the value field.
4. The NEAR-level call succeeds. The 1 NEAR attached deposit is now credited to the Aurora Engine contract account.
5. No refund promise is issued. The 1 NEAR is permanently inaccessible to the user. [6](#0-5) [7](#0-6)

#[named]
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}

pub fn withdraw<I: IO + Copy + PromiseHandler, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    require_running(&state::get_state(&io)?)?;
    env.assert_one_yocto()?;

    let args: WithdrawCallArgs = io.read_input_borsh()?;
    let args = borsh::to_vec(&EngineWithdrawCallArgs {
        sender_id: env.predecessor_account_id(),
        recipient_address: args.recipient_address,
        amount: args.amount,
    })
    .unwrap();

    return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)
}
