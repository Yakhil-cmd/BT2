Audit Report

## Title
Unbounded Per-Stream Buffer Allocation Enables Memory Exhaustion by a Single Malicious Subnet Peer — (File: rs/p2p/quic_transport/src/lib.rs)

## Summary
The QUIC transport layer sets `MAX_MESSAGE_SIZE_BYTES` to 128 MiB and permits up to 1,000 concurrent bidirectional streams per connection. The `start_stream_acceptor` loop spawns an unbounded `JoinSet` task per accepted stream with no application-level cap, and each task calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`, allocating a `Vec<u8>` that grows up to 128 MiB per stream. A single authenticated malicious subnet peer can open 1,000 streams and slowly trickle data on each, forcing the victim replica to allocate up to 128 GiB of heap memory, triggering OOM and crashing the node.

## Finding Description
**Root cause — oversized buffer ceiling with no per-peer task cap:**

`MAX_MESSAGE_SIZE_BYTES` is set to 128 MiB: [1](#0-0) 

`MAX_CONCURRENT_BIDI_STREAMS` is set to 1,000: [2](#0-1) 

`start_stream_acceptor` spawns a task into an unbounded `JoinSet` for every accepted stream with no application-level concurrency limit. The code itself acknowledges this in a comment: [3](#0-2) 

Each spawned task calls `read_request`, which calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`, allocating a `Vec<u8>` that grows as data arrives: [4](#0-3) 

The same pattern exists on the outgoing RPC response path: [5](#0-4) 

And for response body serialization: [6](#0-5) 

**Why existing flow-control guards are insufficient:**

The QUIC-level `STREAM_RECEIVE_WINDOW` (4 MiB) and connection-level `RECEIVE_WINDOW` (200 MiB) are *initial* flow-control windows: [7](#0-6) 

Quinn's `read_to_end` reads data from the QUIC receive buffer into the application-level `Vec<u8>`, which frees the QUIC receive buffer and causes Quinn to automatically extend the flow-control window. The data is then held in the `Vec` until the stream closes. This means the connection-level 200 MiB window does not cap total heap allocation — it only limits in-flight bytes at any instant. As the attacker slowly sends data and the victim reads it into growing `Vec`s, the window advances and the attacker can send more, eventually filling each `Vec` to 128 MiB across all 1,000 streams.

**Exploit flow:**
1. Attacker (a legitimate subnet node with a valid registry TLS certificate) connects to the victim replica.
2. Attacker opens 1,000 concurrent bidirectional QUIC streams (the enforced maximum).
3. On each stream, the attacker slowly trickles data, respecting flow control to avoid idle timeout (keep-alive interval is 1 s, idle timeout is 5 s).
4. Each stream causes `start_stream_acceptor` to spawn a task calling `read_request` → `recv_stream.read_to_end(134_217_728)`.
5. Each task's `Vec<u8>` grows as data arrives; the QUIC window advances as data is read, allowing more data to be sent.
6. With 1,000 streams each accumulating up to 128 MiB: total heap pressure reaches up to **128 GiB**, triggering OOM and crashing the replica process.

## Impact Explanation
A crashed replica drops out of consensus participation, degrading subnet fault tolerance. If the attacker controls `f` nodes in an `f`-fault-tolerant subnet, it can simultaneously attack all honest nodes, halting the subnet entirely. This matches the allowed High impact: **Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000). The attack is protocol-level (exploiting the buffer limit and unbounded task spawning), not a raw volumetric DDoS.

## Likelihood Explanation
- **Attacker role**: A single subnet node below the consensus fault threshold. No majority corruption is required.
- **Authentication barrier**: Mutual TLS against the registry-verified subnet topology is enforced, so only a legitimate (or compromised) subnet peer can connect. A single malicious node operator suffices.
- **No rate limiting**: There is no per-peer stream rate limit or per-peer memory budget at the application layer. The code comment at `request_handler.rs:52–56` explicitly acknowledges the unbounded buffering as a known design limitation.
- **Repeatability**: The attack can be repeated after a node restart. The attacker only needs to maintain the QUIC connection alive (trivially done with keep-alive traffic).

## Recommendation
1. **Short term**: Enforce an application-level cap on the number of concurrently in-flight tasks per peer in `start_stream_acceptor`. Reject or back-pressure new streams when the `JoinSet` size exceeds a threshold (e.g., 10–50). Additionally, reduce `MAX_MESSAGE_SIZE_BYTES` to the smallest value that accommodates the largest legitimate message type (consensus summary blocks are cited as >5 MiB; a limit of 10–20 MiB would be far safer than 128 MiB).
2. **Long term**: Introduce per-peer memory accounting at the transport layer. Track total bytes buffered across all in-flight streams for a given peer and close the connection if the aggregate exceeds a configurable budget. Align `MAX_MESSAGE_SIZE_BYTES` with the actual maximum payload size enforced by the consensus and state-sync protocols.

## Proof of Concept
A malicious subnet node (with a valid registry TLS certificate) executes:

```
for i in 1..=1000:
    open bidirectional QUIC stream to victim node
    slowly write data on stream i (respecting flow control, never closing the stream)
    # each write causes the victim's read_to_end Vec to grow; the QUIC window advances
    # allowing further writes up to MAX_MESSAGE_SIZE_BYTES (128 MiB) per stream
```

**Deterministic integration test plan**: Using the existing `turmoil`-based test harness (as seen in `connection_handle.rs` tests), construct a test that:
1. Spins up a victim `QuicTransport` endpoint.
2. Connects a malicious peer with a valid self-signed cert (using `SkipServerVerification` pattern already present in tests).
3. Opens 1,000 bidirectional streams and writes data slowly on each.
4. Asserts that the victim's memory usage (tracked via `tokio::runtime::Handle::metrics()` or a custom allocator) grows proportionally, or that the victim's `inflight_requests` JoinSet reaches 1,000 tasks.

The relevant code path is: `start_stream_acceptor` (unbounded `JoinSet`) → `handle_bi_stream` → `read_request` → `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`. [8](#0-7) [1](#0-0)

```rust
pub async fn start_stream_acceptor(
    log: ReplicaLogger,
    peer_id: NodeId,
    conn_handle: ConnectionHandle,
    metrics: QuicTransportMetrics,
    router: Router,
) {
    let mut inflight_requests: JoinSet<Result<(), P2PError>> = tokio::task::JoinSet::new();
    let mut quic_metrics_scrape = tokio::time::interval(QUIC_METRIC_SCRAPE_INTERVAL);
    // The extreme result of a slow handler is that the stream limit will be reach, hence
    // having buffered up to the stream limit number of messages/requests.
    // A better approach will be to use a router implemented as a tower service and accept
    // streams iff the router is ready. Then the actual number of buffered messages is determined
    // by the handlers instead by the underlying implementation.
    loop {
        tokio::select! {
            Some(completed_request) = inflight_requests.join_next() => {
                match completed_request {
                    Ok(res) => {
                        let _ = res.inspect_err(|err| info!(every_n_seconds => 60, log, "{:?}", err));
                    }
                    Err(err) => {
                        // Cancelling tasks is ok. Panicking tasks are not.
                        if err.is_panic() {
                            std::panic::resume_unwind(err.into_panic());
                        }
                    }
                }
            },
            bi = conn_handle.conn().accept_bi() => {
                match bi {
                    Ok((bi_tx, bi_rx)) => {
                        let send_stream = ResetStreamOnDrop::new(
                            bi_tx,
                            metrics.connection_handle_incoming_streams_total.clone()
                        );
                        inflight_requests.spawn(
                            metrics.request_task_monitor.instrument(
                                handle_bi_stream(
                                    peer_id,
                                    conn_handle.conn_id(),
                                    metrics.clone(),
                                    router.clone(),
                                    send_stream,
                                    bi_rx
                                )
                            )
                        );
                    }
                    Err(err) => {
                        info!(
                            log,
                            "Exiting request handler event loop due to conn error {:?}",
                            err.to_string()
                        );
                        observe_conn_error(&err, "accept_bi", &metrics.request_handle_errors_total);
                        break;
                    }
                }
            },
            _ = conn_handle.conn().accept_uni() => {},
            _ = conn_handle.conn().read_datagram() => {},
            _ = quic_metrics_scrape.tick() => {
                metrics.collect_quic_connection_stats(conn_handle.conn(), &peer_id);
            }
        }
    }
}
```
