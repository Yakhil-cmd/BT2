### Title
COA Ownership Proof Signature Replay Across Multiple COAs Owned by the Same Cadence Account - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced for one Cadence-Owned Account (COA_A) can be replayed against a different COA (COA_B) published at a different capability path on the same Cadence account, causing `isValidSignature` (ERC-1271) to return `ValidERC1271Signature` for COA_B without the account owner's consent.

---

### Finding Description

`EVM.validateCOAOwnershipProof` performs two checks:

1. The provided signatures are cryptographically valid against the Cadence account's keys.
2. The COA resource published at the caller-supplied `path` has an EVM address matching the caller-supplied `evmAddress`.

What it does **not** check is whether `signedData` encodes `evmAddress`. The code itself documents this gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

Because `createCadenceOwnedAccount` imposes no limit on how many COAs a single Cadence account may create, an account can legitimately hold COA_A at `/public/coaA` and COA_B at `/public/coaB`. [2](#0-1) 

The ERC-1271 entry point in the deployed COA contract calls the precompile with `address(this)` as the `evmAddress` argument — i.e., the address of whichever COA `isValidSignature` is called on: [3](#0-2) 

The precompile decodes the proof (which includes the attacker-controlled `CapabilityPath`) from the raw `_sig` bytes and forwards it to `validateCOAOwnershipProof`: [4](#0-3) [5](#0-4) 

Because the `path` field inside the proof is attacker-controlled and is never bound to the `evmAddress` that the precompile pins to `address(this)`, an attacker can substitute the path of COA_B into a proof originally created for COA_A. The signature bytes remain unchanged and still verify correctly (same Cadence account keys), and the path-to-address check passes because COA_B is legitimately published at the substituted path. [6](#0-5) 

---

### Impact Explanation

Any EVM contract that relies on ERC-1271 (`isValidSignature`) to gate asset transfers, multi-sig approvals, or permit-style authorizations will accept a forged proof for COA_B. An attacker who observes a valid proof for COA_A (e.g., from a submitted transaction or an off-chain authentication flow) can replay it against COA_B to obtain an unauthorized `ValidERC1271Signature` response. This constitutes unauthorized access to on-chain assets or protocol actions gated behind COA_B's ERC-1271 identity.

---

### Likelihood Explanation

The precondition is that the victim Cadence account holds at least two published COAs. While wallets today typically create one COA per account, the protocol imposes no such restriction, and multi-COA setups are a natural pattern for accounts that interact with multiple EVM protocols. The attacker needs only to observe one valid proof (e.g., from a public transaction) and substitute the capability path — no private key material is required.

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress`. Concretely, before verifying the signature, check that the first 20 bytes (or a well-known prefix) of `signedData` equal `evmAddress`. Alternatively, reconstruct the expected signed payload as `hash(evmAddress || applicationData)` and verify against that, so that a proof created for COA_A is cryptographically bound to COA_A's address and cannot be replayed against COA_B. [7](#0-6) 

---

### Proof of Concept

```
Setup:
  - Cadence account Alice holds:
      COA_A  →  EVM address 0xAAAA…  published at /public/coaA
      COA_B  →  EVM address 0xBBBB…  published at /public/coaB

Step 1 (legitimate):
  Alice signs arbitrary data D (e.g., an off-chain login challenge for COA_A).
  Proof_A = { address: Alice, path: /public/coaA, keyIndices: [0], signatures: [sig] }
  Alice submits Proof_A to some EVM dApp that calls COA_A.isValidSignature(hash(D), encode(Proof_A))
  → verifyCOAOwnershipProof(0xAAAA…, hash(D), encode(Proof_A)) → true ✓

Step 2 (attack):
  Attacker observes Proof_A on-chain.
  Attacker constructs Proof_B = { address: Alice, path: /public/coaB, keyIndices: [0], signatures: [sig] }
  (only the path field is changed; the signature bytes are identical)

  Attacker calls COA_B.isValidSignature(hash(D), encode(Proof_B))
  → isValidSignature passes address(this)=0xBBBB… to the precompile
  → validateCOAOwnershipProof(Alice, /public/coaB, D, [0], [sig], 0xBBBB…)
      • keyList.verify(sig, D, "FLOW-V0.0-user") → true  (Alice's key is valid)
      • acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coaB).address() == 0xBBBB… → true
  → returns ValidationResult(isValid: true)
  → isValidSignature returns ValidERC1271Signature (0x1626ba7e)

Result: COA_B accepts a signature that Alice never produced for COA_B.
        Any EVM protocol gating actions on COA_B's ERC-1271 approval is bypassed.
```
