### Title
Missing Cross-Participant `app_id` Consistency Verification Corrupts CKD Output — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol passes `app_id` as a local parameter to each participant independently, but never commits to or broadcasts it as part of the protocol transcript. A malicious participant can silently substitute a different `app_id` when computing their share. The coordinator blindly sums all received shares, producing a CKD output that is cryptographically incorrect and unusable for any honest party.

---

### Finding Description

The `ckd()` entry point in `src/confidential_key_derivation/protocol.rs` accepts `app_id` as a caller-supplied argument and passes it into `run_ckd_protocol()`, which dispatches to either `do_ckd_participant()` or `do_ckd_coordinator()`. [1](#0-0) 

Inside `do_ckd_participant()`, the participant computes its share via `compute_signature_share()` and immediately sends the raw curve points `(norm_big_y, norm_big_c)` to the coordinator. No message in the protocol ever carries or commits to the `app_id` value itself. [2](#0-1) 

Inside `do_ckd_coordinator()`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and adds them together unconditionally, with no check that the shares were computed under the same `app_id`. [3](#0-2) 

The `app_id` enters the computation only inside `compute_signature_share()`, where it is hashed together with the master public key:

```
hash_point = H(pk || app_id)
S_i        = x_i · hash_point
C_i        = S_i + y_i · app_pk
``` [4](#0-3) 

The hash function used is `hash_app_id_with_pk`, which concatenates the compressed master public key with the raw `app_id` bytes and hashes to a G1 curve point. [5](#0-4) 

Because `app_id` never appears in any protocol message, a malicious participant can call `ckd()` with `app_id = "evil_app"` while all honest parties use `app_id = "real_app"`. The coordinator has no mechanism to detect the discrepancy.

---

### Impact Explanation

Let honest parties use `app_id` and the malicious party (index `m`) use `app_id'`. The coordinator accumulates:

```
C_total = Σ_{i≠m} λ_i · (x_i · H(pk||app_id) + y_i · A)
        + λ_m · (x_m · H(pk||app_id') + y_m · A)
```

After the client unmasks with its secret scalar `a`:

```
C_total − a·Y_total
  = msk · H(pk||app_id)
  + λ_m · x_m · (H(pk||app_id') − H(pk||app_id))
```

This is not equal to `msk · H(pk||app_id)`. The derived confidential key is wrong and cannot be verified against the expected BLS pairing check in `verify_signature`. [6](#0-5) 

Impact: **High — Corruption of CKD outputs so honest parties accept an unusable cryptographic output.**

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to call `ckd()` with a different `app_id` byte string. No cryptographic capability, key material, or external assumption is required. The coordinator cannot distinguish a corrupted share from an honest one because the protocol carries no `app_id` binding in its messages.

---

### Recommendation

Before participants compute their shares, add a broadcast round in which every participant commits to (or directly broadcasts) the `app_id` they intend to use. The coordinator should verify that all received `app_id` values are identical before proceeding to aggregate shares. A lightweight approach is to include `app_id` in the first broadcast message alongside the session nonce, analogous to how `session_id` is established in `do_keyshare`. [7](#0-6) 

---

### Proof of Concept

```
Participants: [P1 (honest), P2 (honest), P3 (malicious)]
Agreed app_id: b"real_app"

P1 calls: ckd(..., app_id = b"real_app", ...)
P2 calls: ckd(..., app_id = b"real_app", ...)
P3 calls: ckd(..., app_id = b"evil_app", ...)   ← substituted silently

P3 computes: S3 = x3 · H(pk || b"evil_app")
             C3 = S3 + y3 · app_pk

Coordinator sums: C_total = C1 + C2 + C3  (no app_id check)

Client unmasks: C_total − a·Y_total
              ≠ msk · H(pk || b"real_app")

Result: CKD output is cryptographically invalid; the derived
        confidential key does not satisfy the BLS pairing equation.
```

        norm_big_c += participant_output.big_c();

    key_pair: KeygenOutput,

        });
    }

    // kick out duplicates

        return Err(InitializationError::DuplicateParticipants);

        });

            participant: coordinator,

        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub fn verify_signature(
    verifying_key: &VerifyingKey,
    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle
    let base1 = hash_app_id_with_pk(verifying_key, msg).into();
    let base2 =
        <<BLS12381SHA256 as frost_core::Ciphersuite>::Group as frost_core::Group>::generator()
            .into();

    if blstrs::pairing(&base1, &element2).eq(&blstrs::pairing(&element1, &base2)) {
        Ok(())
    } else {
        Err(frost_core::Error::InvalidSignature)
    }
}
