### Title
BTC Pool Member Can Set Reward Address to STRK Token Contract, Permanently Freezing Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

`Pool.cairo`'s `enter_delegation_pool` and `change_reward_address` only validate that `reward_address` is not equal to the pool's own token. For BTC pools, this means `reward_address = STRK_TOKEN_ADDRESS` passes validation. Since pool rewards are always paid in STRK, calling `claim_rewards` then transfers STRK to the STRK token contract itself, permanently freezing the yield.

### Finding Description

In `pool.cairo`, both `enter_delegation_pool` and `change_reward_address` guard against the reward address being the pool's own token: [1](#0-0) [2](#0-1) 

For a BTC pool, `token_dispatcher.contract_address` is the BTC token address. The assertion `BTC_TOKEN_ADDRESS != reward_address` does **not** prevent `reward_address = STRK_TOKEN_ADDRESS`.

By contrast, the staking contract's `change_reward_address` calls `does_token_exist`, which iterates over **all** managed tokens (STRK + every BTC token): [3](#0-2) 

Pool rewards are always disbursed in STRK regardless of the pool's token: [4](#0-3) 

So when a BTC pool member has `reward_address = STRK_TOKEN_ADDRESS` and `claim_rewards` is invoked, STRK is transferred to the STRK token contract itself.

### Impact Explanation

STRK rewards sent to the STRK token contract are permanently inaccessible — the token contract has no withdrawal mechanism for accidentally received funds. This constitutes **permanent freezing of unclaimed yield**, which is a High-severity impact under the allowed scope.

### Likelihood Explanation

Any BTC pool member can trigger this by calling `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount)` or `change_reward_address(STRK_TOKEN_ADDRESS)` on a BTC pool. No privileged role is required. The entry path is fully reachable by an unprivileged delegator. Likelihood is low-to-medium because it requires a deliberate (or mistaken) choice of reward address, but the missing guard makes it possible where the staking contract explicitly prevents it.

### Recommendation

Mirror the staking contract's validation in `pool.cairo`. In both `enter_delegation_pool` and `change_reward_address`, query the staking contract (or pass a token list at construction) and assert that `reward_address` is not any managed token:

```cairo
// In pool.cairo change_reward_address / enter_delegation_pool
let staking_dispatcher = IStakingDispatcher {
    contract_address: self.staking_pool_dispatcher.contract_address.read(),
};
// Reuse the same guard the staking contract applies
assert!(
    !staking_dispatcher.get_active_tokens().contains(reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Alternatively, at minimum add a hard-coded check against `STRK_TOKEN_ADDRESS` since that is always the reward token:

```cairo
assert!(reward_address != STRK_TOKEN_ADDRESS, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

### Proof of Concept

1. A BTC token is added and enabled by the token admin.
2. A staker opens a BTC delegation pool via `set_open_for_delegation(btc_token_address)`.
3. An attacker (pool member) calls `pool.enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: 1_BTC_unit)`.
   - The check `btc_token_address != STRK_TOKEN_ADDRESS` passes — no revert.
4. Rewards accrue over epochs via `update_rewards` / `update_rewards_from_attestation_contract`.
5. The attacker (or anyone authorized) calls `pool.claim_rewards(pool_member: attacker_address)`.
6. `claim_rewards` executes:
   ```cairo
   let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
   reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards.into());
   ```
   STRK is transferred to the STRK token contract — permanently frozen. [5](#0-4) [6](#0-5)
