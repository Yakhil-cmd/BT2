### Title
No Mechanism for Protocol Owner to Cancel a Pending Price Provider Proposal — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` queues a pending oracle change in `pendingPriceProvider[pool]` / `pendingPriceProviderExecuteAfter[pool]`. No cancel function exists for this queue — neither for the pool admin nor for the protocol owner (factory `Ownable2Step` owner). By contrast, the analogous admin-transfer queue has an explicit `cancelPoolAdminTransfer`. Once a malicious or erroneous price provider is proposed, the protocol owner cannot remove it from the queue even while the pool is at pause level 2, and `executePoolPriceProviderUpdate` carries no pause guard, so the pool admin can finalize the bad oracle while the pool is halted.

---

### Finding Description

`proposePoolPriceProvider` writes unconditionally to two storage slots:

```solidity
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
``` [1](#0-0) 

`executePoolPriceProviderUpdate` — the only path that clears those slots — is gated `onlyPoolAdmin(pool)` and contains no check on the pool's pause level:

```solidity
function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    ...
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
``` [2](#0-1) 

`setPriceProvider` on the pool itself carries only `onlyFactory` — no `whenNotPaused` guard:

```solidity
function setPriceProvider(address newPriceProvider) external onlyFactory {
    priceProvider = newPriceProvider;
``` [3](#0-2) 

The protocol owner's entire incident toolkit is `protocolPausePool` (level 0/1 → 2) and `protocolUnpausePool` (2 → 1): [4](#0-3) 

Neither of those touches `pendingPriceProvider`. The `IMetricOmmPoolFactoryPoolAdmin` interface exposes no cancel entry-point for the price provider queue: [5](#0-4) 

Compare the admin-transfer queue, which does have `cancelPoolAdminTransfer` callable by the current admin: [6](#0-5) 

The price provider queue has no equivalent.

---

### Impact Explanation

After the timelock elapses the pool admin calls `executePoolPriceProviderUpdate` — even while the pool sits at pause level 2 — and the malicious oracle is written to `priceProvider` on the pool. When the pool is eventually unpaused, every subsequent `swap` call reads `_getBidAndAskPriceX64()` from the attacker-controlled provider. The attacker can return arbitrarily inverted or unbounded bid/ask prices, allowing them to drain LP reserves or extract value from traders. This maps directly to the **bad-price execution** and **pool insolvency** impact categories.

---

### Likelihood Explanation

The trigger is a compromised or malicious pool admin key — a realistic incident scenario that the timelock mechanism is explicitly designed to mitigate. The protocol owner's pause capability is the intended second line of defence, but because `executePoolPriceProviderUpdate` ignores pause level and no cancel path exists, that defence is ineffective once a proposal is live. Likelihood is **Medium**: requires pool admin compromise, but the missing cancel is the exact gap the timelock-plus-pause design was meant to close.

---

### Recommendation

Add a `cancelPoolPriceProviderUpdate(address pool)` function callable by the **protocol owner** (`onlyOwner`) — mirroring `cancelPoolAdminTransfer` — that simply deletes the two pending slots:

```solidity
function cancelPoolPriceProviderUpdate(address pool) external onlyOwner {
    if (pendingPriceProvider[pool] == address(0)) revert NoPriceProviderChangeProposed();
    address cancelled = pendingPriceProvider[pool];
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderChangeCancelled(pool, cancelled);
}
```

Additionally, add a pause-level guard inside `executePoolPriceProviderUpdate` (or inside `setPriceProvider`) so that a protocol-paused pool cannot have its oracle swapped without the owner's explicit consent.

---

### Proof of Concept

1. Pool is deployed with a mutable price provider and a finite `priceProviderTimelock`.
2. Pool admin key is compromised. Attacker calls `proposePoolPriceProvider(pool, maliciousOracle)`.
3. Protocol owner detects the proposal and calls `protocolPausePool(pool)` → pause level 2.
4. Timelock elapses. Attacker calls `executePoolPriceProviderUpdate(pool)`. No pause check → `setPriceProvider(maliciousOracle)` executes on the pool.
5. Protocol owner is forced to either keep the pool permanently halted (DoS for all LPs) or unpause, at which point `swap` reads prices from `maliciousOracle`.
6. Attacker's oracle returns an inverted spread; attacker swaps and extracts LP reserves.

The protocol owner has no on-chain action between steps 3 and 4 that can prevent step 4 from succeeding.
