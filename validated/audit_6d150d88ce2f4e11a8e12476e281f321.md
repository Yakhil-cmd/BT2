Audit Report

## Title
Silent Clamping of Over-Max `icrc2_approve` Amount to `Tokens::MAX` — (`rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary
In `icrc2_approve_not_async`, when a caller submits an `amount` field containing a `Nat` value exceeding `u64::MAX`, the `TryFrom<Nat>` conversion error is silently discarded via `unwrap_or_else(|_| Tokens::max_value())`, and the resulting `u64::MAX` allowance is committed to the ledger without any error being returned to the caller. The invariant "stored allowance equals requested amount, or the call fails" is broken, enabling a spender to receive an effectively unlimited allowance regardless of the approver's intent.

## Finding Description
`TryFrom<Nat> for Tokens` at [1](#0-0)  correctly returns `Err` when the `Nat` value does not fit in a `u64`. However, `icrc2_approve_not_async` at [2](#0-1)  discards that error:

```rust
let amount = Tokens::try_from(arg.amount).unwrap_or_else(|_| Tokens::max_value());
```

The silently-clamped `amount` (`u64::MAX`) is then embedded directly into `Operation::Approve` and committed via `apply_transaction` at [3](#0-2) . Neither `apply_transaction` nor `approvals.approve` performs any upper-bound check on the allowance amount — the `u64::MAX` value passes through unchallenged. [4](#0-3) 

The correct pattern already exists in the same function for `expected_allowance` at [5](#0-4) , which returns `Err(ApproveError::AllowanceChanged { ... })` on conversion failure instead of substituting a fallback. The `amount` branch inconsistently uses `unwrap_or_else` instead.

The public `#[update]` endpoint `icrc2_approve` is reachable by any principal with no privilege requirement. [6](#0-5) 

## Impact Explanation
A user interacting with a DApp that computes allowance amounts (e.g., "approve maximum possible") may submit a `Nat` value exceeding `u64::MAX`. Instead of receiving an error and retrying with a valid value, the call silently succeeds with a `u64::MAX` allowance. The designated spender — which may be a malicious or compromised DApp contract — can then call `icrc2_transfer_from` to drain the approver's entire balance. This constitutes unauthorized fund loss from an in-scope ICRC ledger, matching the **High** impact class: "Unauthorized access to ledger or canister-controlled funds where exploitation requires meaningful per-target work or other constraints." The constraint here is that the victim must submit an oversized `amount` (by mistake or through a deceptive DApp interface), which is realistic given that `Nat` is an arbitrary-precision type and clients may not know the `u64` ceiling.

## Likelihood Explanation
The attack requires no privileged access. Any principal can call `icrc2_approve`. The Candid encoding of a `Nat` larger than `u64::MAX` is trivially constructable by any client library. The most realistic path is a DApp that computes `u128::MAX` or `2^64` as an "unlimited approval" sentinel, submits it, receives a success response, and the user is unaware that the stored allowance is `u64::MAX` rather than an error. The existing test suite at [7](#0-6)  already tests `expected_allowance = Nat::from(u128::MAX)` and expects `AllowanceChanged`, confirming the developers are aware of the over-max pattern — but no analogous test exists for the `amount` field.

## Recommendation
Replace the silent fallback with an explicit error return, consistent with the `expected_allowance` branch:

```rust
let amount = Tokens::try_from(arg.amount)
    .map_err(|_| ApproveError::GenericError {
        error_code: Nat::from(0u64),
        message: "amount exceeds maximum token value".to_string(),
    })?;
```

## Proof of Concept
1. Encode `ApproveArgs { amount: Nat::from(u128::MAX), spender: attacker_account, from_subaccount: None, fee: None, memo: None, created_at_time: None, expected_allowance: None, expires_at: None }`.
2. Submit via `icrc2_approve` ingress call — the call returns `Ok(block_index)` instead of an error.
3. Query `icrc2_allowance` for `(owner, attacker_account)` — the returned allowance is `u64::MAX`, not `u128::MAX` and not an error.
4. Call `icrc2_transfer_from` from the attacker's principal to transfer the owner's full balance to any destination.
5. A deterministic integration test can be added alongside the existing `test_approve_expected_allowance` in [8](#0-7)  by sending `amount = Nat::from(u128::MAX)` and asserting the result is `Err(ApproveError::GenericError { ... })` rather than `Ok(...)`.

pub fn test_approve_expected_allowance<T>(ledger_wasm: Vec<u8>, encode_init_args: fn(InitArgs) -> T)
where
    T: CandidType,
{
    let from = PrincipalId::new_user_test_id(1);
    let spender = PrincipalId::new_user_test_id(2);

    let (env, canister_id) = setup(
        ledger_wasm,
        encode_init_args,
        vec![(Account::from(from.0), 100_000)],
    );

    let mut approve_args = default_approve_args(spender.0, 150_000);

    send_approval(&env, canister_id, from.0, &approve_args).expect("approval failed");

    // Wrong expected_allowance.
    approve_args.expires_at = None;
    approve_args.amount = Nat::from(400_000_u32);
    approve_args.expected_allowance = Some(Nat::from(100_000_u32));
    assert_eq!(
        send_approval(&env, canister_id, from.0, &approve_args),
        Err(ApproveError::AllowanceChanged {
            current_allowance: Nat::from(150_000_u32)
        })
    );
    let allowance = Account::get_allowance(&env, canister_id, from.0, spender.0);
    assert_eq!(allowance.allowance.0.to_u64().unwrap(), 150_000);
    assert_eq!(allowance.expires_at, None);
    assert_eq!(balance_of(&env, canister_id, from.0), 90_000);
    assert_eq!(balance_of(&env, canister_id, spender.0), 0);

    // Wrong expected_allowance - above u64::MAX
    approve_args.expires_at = None;
    approve_args.amount = Nat::from(400_000_u32);
    approve_args.expected_allowance = Some(Nat::from(u128::MAX));
    assert_eq!(
        send_approval(&env, canister_id, from.0, &approve_args),
        Err(ApproveError::AllowanceChanged {
            current_allowance: Nat::from(150_000_u32)
        })
    );
    let allowance = Account::get_allowance(&env, canister_id, from.0, spender.0);
    assert_eq!(allowance.allowance.0.to_u64().unwrap(), 150_000);
    assert_eq!(allowance.expires_at, None);
    assert_eq!(balance_of(&env, canister_id, from.0), 90_000);
    assert_eq!(balance_of(&env, canister_id, spender.0), 0);

    // Correct expected_allowance.
    approve_args.amount = Nat::from(400_000_u32);
    approve_args.expected_allowance = Some(Nat::from(150_000_u32));
