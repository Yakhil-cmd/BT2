### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since their last claim. A pool member who changes their balance in many distinct epochs without claiming rewards will accumulate enough trace entries to make every future `claim_rewards` call run out of gas, permanently freezing their unclaimed yield.

### Finding Description

`calculate_rewards` iterates linearly over the pool member's balance trace: [1](#0-0) 

The comment on line 858 explicitly acknowledges the loop is unbounded. Each iteration performs multiple storage reads: one `pool_member_trace.at(entry_to_claim_from)` and up to three reads inside `find_sigma` (via `find_sigma_standard_case`). [2](#0-1) 

The trace grows by one entry per epoch in which the pool member modifies their balance. `set_member_balance` inserts with key `current_epoch + K`; the underlying `insert` only appends a new checkpoint when the key differs from the last one, so at most one new entry is added per epoch. [3](#0-2) 

The following public functions all call `set_member_balance` or `increase_member_balance`, each potentially adding a new trace entry per epoch:
- `add_to_delegation_pool` (line 242)
- `exit_delegation_pool_intent` (line 278)
- `enter_delegation_pool` (line 201) [4](#0-3) 

The `entry_to_claim_from` cursor stored in `InternalPoolMemberInfoV1` is only advanced on a *successful* `claim_rewards` call: [5](#0-4) 

If a `claim_rewards` transaction reverts due to OOG, `entry_to_claim_from` is never updated. Every subsequent call restarts the loop from the same position, making the freeze permanent.

### Impact Explanation

A pool member who has changed their balance in N distinct epochs without claiming will have N entries in their trace. Once N is large enough that the loop exhausts the Starknet transaction gas limit, every `claim_rewards` call reverts. Because `entry_to_claim_from` is only written on success, the member can never advance past the bloated trace. Their accumulated STRK rewards are permanently frozen in the pool contract — they cannot be claimed by the pool member or transferred elsewhere.

This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation

Any pool member who:
1. Calls `add_to_delegation_pool` or `exit_delegation_pool_intent` in many different epochs (e.g., a delegator who regularly adjusts their stake), **and**
2. Does not call `claim_rewards` frequently enough to keep `entry_to_claim_from` advancing

will accumulate a large trace. Over a long-running protocol (hundreds or thousands of epochs), this is a realistic scenario for active delegators. No privileged access is required — only normal pool member operations.

### Recommendation

Replace the linear scan with a paginated claim mechanism: allow `claim_rewards` to accept a `max_entries` parameter and process only that many trace entries per call, advancing `entry_to_claim_from` incrementally. Alternatively, enforce a maximum number of unclaimed balance-change entries before a claim is required (i.e., block further balance changes if the trace exceeds a threshold without claiming).

### Proof of Concept

1. Pool member Alice joins a pool at epoch 0.
2. Alice calls `add_to_delegation_pool` once per epoch for N epochs (e.g., N = 2000), adding a minimal amount each time. Each call in a new epoch appends one entry to `pool_member_epoch_balance`.
3. Alice never calls `claim_rewards` during this period.
4. After N epochs, Alice calls `claim_rewards`. The `calculate_rewards` loop must iterate all N entries, each requiring ~4 storage reads. At sufficient N, the transaction runs OOG and reverts.
5. Because `entry_to_claim_from` was not updated (revert), every subsequent `claim_rewards` call also runs OOG. Alice's accumulated rewards are permanently frozen. [6](#0-5) [7](#0-6)

pub(crate) fn find_sigma_standard_case(
    cumulative_rewards_trace_vec: StorageBase<Trace>,
    cumulative_rewards_trace_idx: VecIndex,
    target_epoch: Epoch,
) -> Amount {
    // Three entries in the cumulative rewards trace are relevant (idx, idx - 1, idx - 2).
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx);
    if epoch < target_epoch {
        return sigma;
    }
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx - 1);
    if epoch < target_epoch {
        return sigma;
    }
    // Note: When handling a checkpoint from the old version, it never reaches here.
    let (epoch, sigma) = cumulative_rewards_trace_vec.at(cumulative_rewards_trace_idx - 2);
    assert!(epoch < target_epoch, "{}", InternalError::INVALID_EPOCH_IN_TRACE);
    sigma
}
