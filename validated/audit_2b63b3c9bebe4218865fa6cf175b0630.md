### Title
`PresignOutput` Not Consumed on Use Enables Presignature Reuse Leading to Private Key Extraction — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based and Robust ECDSA schemes expose `rerandomize_presign()` as a function that takes the raw `PresignOutput` by **shared reference** (`&PresignOutput`) rather than by value. Because the presignature is never consumed, the library provides no mechanism to enforce the documented one-time-use invariant. A malicious coordinator can drive two independent signing sessions against the same presignature, obtaining two ECDSA signatures whose nonces are algebraically related, and then recover the aggregate private key via a standard nonce-reuse attack.

---

### Finding Description

The documentation is explicit that presignatures must never be reused:

> "it's crucial that a presignature is never reused"
> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."

Despite this, `rerandomize_presign()` in both schemes accepts the presignature by immutable reference:

**OT-based ECDSA** (`src/ecdsa/ot_based_ecdsa/mod.rs`):
```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) 

**Robust ECDSA** (`src/ecdsa/robust_ecdsa/mod.rs`):
```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [2](#0-1) 

Both `PresignOutput` types also derive `Clone`, making it trivial to duplicate them before any consumption point: [3](#0-2) [4](#0-3) 

The downstream `sign()` functions do take `RerandomizedPresignOutput` by value (consuming it), which is the correct Rust pattern for one-time use: [5](#0-4) [6](#0-5) 

However, because `rerandomize_presign()` does not consume the underlying `PresignOutput`, the same raw presignature can be rerandomized an arbitrary number of times with different `RerandomizationArguments` (different tweaks `ε` and/or message hashes `h`), producing multiple distinct `RerandomizedPresignOutput` values — each of which can then be passed to `sign()` and consumed in a separate signing session.

The orchestration documentation explicitly identifies this as the critical invariant that must be enforced:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [7](#0-6) 

The library provides no state-tracking mechanism to detect or prevent this reuse. The `PresignOutput` struct carries no "used" flag, and there is no registry of consumed presignatures.

---

### Impact Explanation

When the same presignature `(R, k_i, σ_i)` is rerandomized twice with different parameters `(δ₁, ε₁, h₁)` and `(δ₂, ε₂, h₂)`, the two resulting ECDSA signatures share a multiplicatively related nonce:

- Session 1: nonce `R₁ = δ₁·R`, signature `(R₁, s₁)`
- Session 2: nonce `R₂ = δ₂·R`, signature `(R₂, s₂)`

Because both `R₁` and `R₂` are scalar multiples of the same base point `R = k·G`, the two signatures satisfy a linear relation in the secret key `x`. A coordinator who observes both `(R₁, s₁)` and `(R₂, s₂)` can solve for `x` using standard ECDSA nonce-reuse algebra. The security considerations section of the Robust ECDSA documentation explicitly confirms this:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [8](#0-7) 

**Impact: Critical — full extraction of the aggregate private signing key.**

---

### Likelihood Explanation

The malicious coordinator is an explicitly documented threat actor in this library. The coordinator role is required for both OT-based and Robust ECDSA signing. A coordinator who wishes to extract the private key needs only to:

1. Complete a legitimate presigning session with honest participants.
2. Initiate two signing sessions referencing the same presignature (e.g., with different message hashes or tweaks).
3. Collect the signature shares from both sessions.

Honest participants have no in-library mechanism to detect that the presignature was already used — the library provides no consumed-presignature registry, no "used" bit on `PresignOutput`, and no enforcement at the `rerandomize_presign()` call site. The attack requires only that the coordinator be malicious and that participants follow the protocol normally.

---

### Recommendation

Change `rerandomize_presign()` to consume the `PresignOutput` by value in both schemes:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // ← consume by value
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

This leverages Rust's ownership system to enforce the one-time-use invariant at compile time: once a `PresignOutput` has been passed to `rerandomize_presign()`, it is moved and cannot be used again. Additionally, remove the `Clone` derive from `PresignOutput` in both schemes to prevent callers from trivially circumventing the move semantics. [3](#0-2) [4](#0-3)

### Proof of Concept

```rust
// Both schemes — OT-based ECDSA shown
use threshold_signatures::ecdsa::ot_based_ecdsa::{PresignOutput, RerandomizedPresignOutput};

// After presigning, each participant holds presign_output: PresignOutput

// Session 1: coordinator requests signature on message M1 with tweak ε1
let rerandomized_1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,   // ← reference: presign_output is NOT consumed
    &args_for_m1,
).unwrap();
// sign() consumes rerandomized_1 → produces (R1, s1)

// Session 2: coordinator requests signature on message M2 with tweak ε2
// presign_output is still valid — library has no record it was used
let rerandomized_2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,   // ← same presign_output reused
    &args_for_m2,
).unwrap();
// sign() consumes rerandomized_2 → produces (R2, s2)

// Coordinator now holds (R1,s1) and (R2,s2) with related nonces → recovers private key x
``` [9](#0-8) [10](#0-9)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
