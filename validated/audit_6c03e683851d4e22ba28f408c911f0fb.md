### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State (Level 1) Instead of Active State (Level 0), Leaving Pool Swaps Blocked After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` in `MetricOmmPoolFactory` always sets the pool's pause level to **1** (admin-paused) rather than **0** (active). When the protocol pauses an active pool (0 → 2) and then calls `protocolUnpausePool` (2 → 1), the pool is still paused. Swaps remain blocked and the pool admin must take an additional, unsolicited action (`unpausePool`) to restore the pool to operational state — even though the admin never initiated a pause.

---

### Finding Description

The pause-level state machine has three states:

| Level | Meaning |
|---|---|
| 0 | Active — swaps allowed |
| 1 | Admin-paused — swaps blocked |
| 2 | Protocol-paused — swaps blocked |

`protocolUnpausePool` is defined as:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← always lands at level 1
}
``` [1](#0-0) 

The pool's `_checkNotPaused` guard blocks swaps whenever `pauseLevel != 0`:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) 

Because level 1 ≠ 0, the `whenNotPaused` modifier on `swap` still fires after `protocolUnpausePool` completes: [3](#0-2) 

The only path back to level 0 is the pool admin calling `unpausePool`, which requires `cur == 1`:

```solidity
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [4](#0-3) 

The admin cannot skip from level 2 directly to level 0 — `unpausePool` requires `cur == 1`, and `protocolUnpausePool` is the only way out of level 2. So the full sequence for an originally-active pool is:

```
active (0) → protocolPausePool → protocol-paused (2)
           → protocolUnpausePool → admin-paused (1)   ← still paused!
           → unpausePool (admin must act) → active (0)
```

The protocol's "unpause" action does not restore the pool to the operational state it was in before the protocol pause. Level 1 is a valid pause state, but it is unreachable as an "unpaused" outcome — swaps remain blocked.

---

### Impact Explanation

After `protocolUnpausePool` completes, the pool is still paused at level 1. The `swap` function reverts for every caller. LPs cannot earn trading fees, and traders cannot execute swaps. The pool's primary function is broken until the admin explicitly calls `unpausePool`. If the pool admin is a multisig or DAO requiring governance, this introduces an additional governance round-trip delay. If the admin key is slow to respond, the pool remains non-functional for an extended and unpredictable period after the protocol has already declared the emergency over.

---

### Likelihood Explanation

This occurs every time the protocol pauses and then unpauses a pool that was originally in the active state (level 0). This is the most common scenario for an emergency protocol pause — the pool is running normally, the protocol pauses it for an emergency, and then the protocol unpauses it. The broken transition is deterministic and reproducible on every such cycle.

---

### Recommendation

Change `protocolUnpausePool` to transition directly to level 0 (active) instead of level 1 (admin-paused):

```diff
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the intended design is to hand off control to the admin after a protocol unpause, the function should be renamed to reflect that it does not actually unpause the pool, and the documentation should explicitly state that the admin must call `unpausePool` afterward.

---

### Proof of Concept

1. Pool is deployed and active — `pauseLevel == 0`.
2. Protocol calls `protocolPausePool` → `pauseLevel` transitions to 2. Swaps blocked.
3. Emergency resolved. Protocol calls `protocolUnpausePool` → `pauseLevel` transitions to **1** (admin-paused). Swaps still blocked.
4. Any call to `swap` reverts with `PoolPaused` because `pauseLevel != 0`.
5. The pool admin (who never paused the pool) must now call `unpausePool` to reach level 0.
6. Until the admin acts, the pool is stuck in a paused state despite the protocol having "unpaused" it.
