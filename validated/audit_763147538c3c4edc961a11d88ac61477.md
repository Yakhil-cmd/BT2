### Title
Missing Validation of `reserved[1]` Refund Recipient Address in L1→L2 Transaction Parsing - (`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `validate_structure()` method of `AbiEncodedTransaction` does not validate that the `reserved[1]` field — the refund recipient address for L1→L2 and upgrade transactions — is a valid 20-byte Ethereum address. An L1 transaction submitter can supply a `U256` value with non-zero upper 12 bytes. The bootloader then silently converts it via `u256_to_b160_checked`, sending the gas refund to a truncated/unintended address, causing permanent loss of the refund for the original depositor.

---

### Finding Description

In `validate_structure()`, the `reserved[1]` field is explicitly left unvalidated with a `// TODO: validate address?` comment: [1](#0-0) 

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
```

The field is stored as a raw `U256` (not a `B160`): [2](#0-1) 

Later, in `process_l1_transaction`, the refund is minted to the address derived from this unvalidated `U256`: [3](#0-2) 

```rust
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    mint_base_token::<S, Config>(
        ...
        &refund_recipient,
        ...
    )?;
}
```

`u256_to_b160_checked` truncates the `U256` to its lower 20 bytes. If the upper 12 bytes of `reserved[1]` are non-zero (i.e., the value is not a valid Ethereum address), the truncation silently produces a different address than the submitter intended.

---

### Impact Explanation

The refund (unused gas × gas price) is minted to the truncated address rather than the intended recipient. Since L1→L2 transactions cannot be invalidated or replayed, the funds are permanently lost to the original depositor. The `validate_structure()` function is the only place where this could be caught before execution, and it explicitly skips the check.

**Impact**: High — permanent, irreversible loss of user funds (gas refund) for any L1→L2 transaction with a malformed `reserved[1]`.

---

### Likelihood Explanation

An L1 user submitting a deposit transaction with a `reserved[1]` value whose upper 12 bytes are non-zero triggers this path. This can happen:
- By accident (e.g., passing a full `uint256` storage slot value instead of an address)
- By a malicious actor deliberately crafting a transaction to grief a specific depositor

L1 bridge contracts may perform their own validation, but the ZKsync OS bootloader itself provides no defense-in-depth check, and the `// TODO: validate address?` comment confirms this is a known, unresolved gap.

**Likelihood**: Low-Medium — requires a malformed L1 transaction, but the path is fully reachable with no privileged access required.

---

### Recommendation

In `validate_structure()`, add an explicit check that `reserved[1]` fits within 20 bytes (upper 12 bytes must be zero) for `L1_L2_TX_TYPE` and `UPGRADE_TX_TYPE`:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    // Validate that reserved[1] is a valid 20-byte Ethereum address
    let reserved_1 = self.reserved[1].read();
    // Upper 96 bits (12 bytes) must be zero
    if reserved_1 >> 160 != U256::ZERO {
        return Err(());
    }
}
```

This mirrors the existing `parse_address` logic already used for `from`, `to`, and `paymaster` fields. [4](#0-3) 

---

### Proof of Concept

1. Construct an L1→L2 transaction where `reserved[1]` = `0x000000000000000000000001<valid_20_byte_address>` (upper 12 bytes non-zero, e.g., `0x01` in byte 11).
2. Submit it via the L1 priority queue.
3. The bootloader's `validate_structure()` accepts it (the TODO branch does nothing).
4. After execution, `u256_to_b160_checked(reserved[1])` truncates to the lower 20 bytes — a completely different address.
5. The gas refund is minted to that truncated address.
6. The original depositor's refund is permanently lost. [5](#0-4) [6](#0-5)
