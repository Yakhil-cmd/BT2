### Title
Rebasing NEP-141 Token Yield Permanently Locked in Aurora Bridge Connector - (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

When a rebasing NEP-141 token (one whose `ft_balance_of(aurora)` grows autonomously over time, analogous to Aave aTokens on Ethereum) is bridged into Aurora via the `ft_on_transfer` / `deploy_erc20_token` flow, the ERC-20 mirror mints a fixed amount equal to the deposited quantity. As the NEP-141 balance held by the Aurora contract grows due to rebasing, the total ERC-20 supply remains static. When users exit via `withdrawToNear`, only the burned ERC-20 amount is transferred back. The accrued rebasing yield accumulates in Aurora's NEP-141 balance with no mechanism to recover it, permanently locking those funds.

---

### Finding Description

**Deposit path** (`ft_on_transfer` → `receive_erc20_tokens`):

When a user calls `ft_transfer_call` on a NEP-141 contract targeting Aurora, the NEAR runtime invokes `ft_on_transfer` on the Aurora engine. Inside `receive_erc20_tokens`, the `amount` field from `FtOnTransferArgs` is used verbatim to mint exactly that many ERC-20 tokens:

```rust
// engine/src/engine.rs
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)
```

This mints `amount` ERC-20 tokens. The Aurora contract now holds `amount` NEP-141 tokens.

**Withdrawal path** (`withdrawToNear` → `ExitToNear` precompile → `ft_transfer`):

In `EvmErc20.sol`, `withdrawToNear` burns the caller's ERC-20 tokens and passes the burned `amount` directly to the `ExitToNear` precompile:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    // calls ExitToNear precompile at 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f
```

The `ExitToNear` precompile in `exit_erc20_token_to_near` then constructs an `ft_transfer` call using `exit_params.amount` — the burned ERC-20 amount — as the NEP-141 transfer amount:

```rust
// engine-precompiles/src/native.rs
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()
)
```

**The rebasing gap**:

If the NEP-141 token is rebasing, Aurora's actual NEP-141 balance grows beyond the total ERC-20 supply. For example:
- 1000 tokens deposited → 1000 ERC-20 minted; Aurora holds 1000 NEP-141
- After rebasing: Aurora holds 1100 NEP-141; total ERC-20 supply still 1000
- All 1000 ERC-20 burned → 1000 NEP-141 transferred out
- 100 NEP-141 tokens remain in Aurora's account permanently

There is no admin function, sweep function, or any other mechanism in the Aurora engine contract to recover excess NEP-141 token balances that exceed the total ERC-20 supply.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield / Permanent freezing of funds.**

All rebasing yield accrued on NEP-141 tokens held by the Aurora bridge contract is permanently locked. No user (depositor, ERC-20 holder, or admin) can recover the excess NEP-141 balance. The yield is effectively destroyed. If a popular yield-bearing NEP-141 token (e.g., a NEAR lending protocol's receipt token) is bridged, the locked amount grows continuously and unboundedly.

---

### Likelihood Explanation

**Likelihood: Medium.**

The Aurora bridge is designed to accept any NEP-141 token via `deploy_erc20_token` and `ft_on_transfer`. The NEAR ecosystem has yield-bearing NEP-141 tokens (e.g., staking derivatives, lending receipt tokens). Any user can permissionlessly deploy an ERC-20 mirror for such a token and begin bridging. No special privileges are required to trigger this condition — it is an inherent property of the bridge design when applied to rebasing tokens.

---

### Recommendation

1. Track the total NEP-141 amount deposited per token (analogous to total ERC-20 supply for that token).
2. Add a function allowing the contract owner or vault creator to sweep excess NEP-141 balance (i.e., `actual_nep141_balance - total_erc20_supply`) on a pro-rata basis.
3. Alternatively, document and enforce that only non-rebasing NEP-141 tokens may be bridged, and add a check or registry to prevent rebasing tokens from being registered via `deploy_erc20_token`.

---

### Proof of Concept

**Step 1**: Deploy a rebasing NEP-141 token on NEAR (e.g., a yield-bearing staking receipt token whose `ft_balance_of` for any holder increases each epoch).

**Step 2**: Call `deploy_erc20_token` on Aurora to create an ERC-20 mirror for it.

**Step 3**: Alice calls `ft_transfer_call(aurora, 1000, alice_evm_address)` on the NEP-141 contract. Aurora's `ft_on_transfer` mints 1000 ERC-20 tokens to Alice's EVM address. [1](#0-0) 

**Step 4**: Time passes. The NEP-141 token rebases. Aurora's NEP-141 balance is now 1100 (100 tokens of yield accrued). Total ERC-20 supply is still 1000.

**Step 5**: Alice calls `withdrawToNear(alice_near_account, 1000)` on the ERC-20 contract. This burns 1000 ERC-20 tokens and calls the `ExitToNear` precompile with `amount = 1000`. [2](#0-1) 

**Step 6**: The precompile calls `ft_transfer(alice_near_account, 1000)` on the NEP-141 contract. Alice receives 1000 tokens. [3](#0-2) 

**Step 7**: Aurora's NEP-141 balance is now 100. Total ERC-20 supply is 0. The 100 rebasing yield tokens are permanently locked in Aurora's account. No function exists to recover them. [4](#0-3)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);

                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());
