### Title
COA Ownership Proof Signature Replay via Missing `signedData`-to-`evmAddress` Binding — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over an arbitrary caller-supplied `signedData` blob using the fixed domain tag `"FLOW-V0.0-user"`, but never enforces that `signedData` encodes the `evmAddress` being proven. Any prior signature produced by a Flow account key under the `"FLOW-V0.0-user"` tag — for any purpose — can be replayed by an unprivileged attacker to make that account's COA (Cadence-Owned Account) appear to have validly signed the same data in an EVM context, satisfying ERC-1271 `isValidSignature` checks.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts `signedData` as a free byte array and verifies the provided signatures against it using the hardcoded domain separation tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The function's own NatSpec comment acknowledges the root cause:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The COA's ERC-1271 implementation in `fvm/evm/handler/coa/coa.sol` calls this path directly:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
    );
    ...
}
```

Here `_hash` becomes `signedData` and `_sig` is the RLP-encoded `COAOwnershipProof`. There is no nonce, no timestamp, and no enforcement that `signedData` encodes `evmAddress` or any EVM-context binding. The `"FLOW-V0.0-user"` domain tag is the standard tag used across all user-facing Cadence signature operations, so any signature Alice has ever produced under that tag — for any Cadence dApp, off-chain authentication flow, or other purpose — is a valid candidate for replay.

---

### Impact Explanation

An attacker who observes a valid Flow account signature over some 32-byte value `D` (produced under `"FLOW-V0.0-user"`) can:

1. Construct a `COAOwnershipProof` embedding Alice's Flow address, the observed key index, and the observed signature.
2. Call `isValidSignature(D, proof)` on Alice's COA contract from any EVM transaction.
3. The COA calls `verifyCOAOwnershipProof(address(this), D, proof)` → `validateCOAOwnershipProof` → signature check passes → returns `ValidERC1271Signature` (0x1626ba7e).

Any EVM protocol that relies on ERC-1271 for authorization (ERC-20 `permit`-style approvals, NFT marketplace listings, multisig confirmations, lending protocol collateral approvals) will treat Alice's COA as having authorized the action corresponding to hash `D`. This constitutes unauthorized movement of assets or unauthorized state changes in EVM protocols, directly analogous to the LGO permit replay.

**Impact:** Unauthorized ERC-1271 signature validation → unauthorized asset transfers or protocol actions on behalf of a victim's COA.

---

### Likelihood Explanation

- The attacker entry path is fully unprivileged: any EVM transaction can call `isValidSignature` on any COA.
- The `"FLOW-V0.0-user"` tag is the standard user-facing tag; signatures under it are produced routinely by wallets and dApps.
- The attacker only needs to observe one prior signature from the victim's Flow account key over a 32-byte value that is also meaningful in an EVM context (e.g., an EIP-712 struct hash, an ERC-20 permit hash).
- No special node access, key compromise, or social engineering is required.

**Likelihood:** Medium — requires observing a prior signature, but the `"FLOW-V0.0-user"` tag is ubiquitous and signatures are observable on-chain or via public APIs.

---

### Recommendation

`validateCOAOwnershipProof` must enforce that `signedData` commits to `evmAddress`. The simplest fix is to require that `signedData` is a canonical encoding of `(evmAddress, applicationData)` and verify this inside the function before the `keyList.verify` call, rather than leaving it as a caller convention. Alternatively, the function should hash `evmAddress` into the signed message itself before verification, similar to how EIP-712 binds a domain separator to every signed struct.

6. Returns `0x1626ba7e` (valid ERC-1271 signature) — attacker has forged Alice's COA authorization for hash `D` in any EVM protocol.

// COAOwnershipProofInContext contains all the data
// needed to verify a COAOwnership proof.
// The proof is verified by checking the signatures over the
// input signed data (SignedData), then loading the resource
// capability from the provided path in the proof, and
// at last checking if the EVMAddress of the resource matches
// the provided one.
type COAOwnershipProofInContext struct {
	COAOwnershipProof
	SignedData SignedData
	EVMAddress Address
}

func NewCOAOwnershipProofInContext(sd []byte, addr Address, encodedProof []byte) (*COAOwnershipProofInContext, error) {
	proof, err := COAOwnershipProofFromEncoded(encodedProof)
	if err != nil {
		return nil, err
	}
	return &COAOwnershipProofInContext{
		COAOwnershipProof: *proof,
		SignedData:        sd,
		EVMAddress:        addr,
	}, nil
}
