### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with **no factory-side cap check**, while the parallel function `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`. A pool admin can therefore set per-bin additional fees up to `uint16` max (65 535 ≈ 6.55 % in E6 units) on any bin, bypassing the protocol-enforced fee cap and extracting excess fees from traders.

---

### Finding Description

`setPoolAdminFees` enforces the cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has **no equivalent check**:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool-side `setBinAdditionalFees` also performs no cap check — it writes the raw values directly:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];
  s.addFeeBuyE6 = addFeeBuyE6;
  s.addFeeSellE6 = addFeeSellE6;
  emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

These per-bin fees are then **added directly to the base spread fee** in every swap path:

```solidity
// MetricOmmPool.sol:910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

```solidity
// MetricOmmPool.sol:1177
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [5](#0-4) 

The factory hard-caps the **global** spread fee at `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) and the admin component at `maxAdminSpreadFeeE6` (a configurable value ≤ 20 %). [6](#0-5)  The per-bin additional fee is a `uint16` whose maximum (65 535) represents ≈ 6.55 % in E6 units — entirely unconstrained relative to whatever `maxAdminSpreadFeeE6` is set to. If `maxAdminSpreadFeeE6` is, say, 5 000 (0.5 %), the pool admin can still push per-bin fees to 65 535 (6.55 %), a 13× overshoot of the intended cap.

---

### Impact Explanation

Every swap routed through the affected bin pays `baseFee + addFeeBuyE6/addFeeSellE6` instead of the capped `baseFee + maxAdminSpreadFeeE6`. The excess fee accrues to LPs (not the admin directly), but the pool admin controls LP positions and can drain the excess via `collectPoolFees`. Traders receive materially less output than the oracle price and the disclosed fee cap permit, constituting direct loss of trader principal beyond the protocol-enforced bound. This satisfies the "Admin-boundary break: pool admin exceeds caps" impact gate.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly supposed to be bounded by `maxAdminSpreadFeeE6`. The missing cap is a single-transaction call requiring only `poolAdmin[pool] == msg.sender`. No timelock, no multi-sig, no additional privilege is needed beyond the pool admin role itself.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  uint24 cap = maxAdminSpreadFeeE6;
  if (addFeeBuyE6 > cap) revert AdminFeeTooHigh();
  if (addFeeSellE6 > cap) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` constant or state variable if per-bin fees are intended to have a different ceiling than the global spread cap.

---

### Proof of Concept

1. Deploy a pool with `maxAdminSpreadFeeE6 = 5_000` (0.5 %).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert — `BinState.addFeeBuyE6` and `addFeeSellE6` are now 65 535 on bin 0.
4. A trader swaps token1 → token0 through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ base + 6.55 %, far exceeding the 0.5 % admin cap.
5. Assert that the trader's output is less than what `maxAdminSpreadFeeE6 = 5_000` would permit — the shortfall is the excess fee extracted beyond the protocol-enforced bound.
