### Title
`set_eth_connector_contract_account` Does Not Validate New Connector's Back-Reference to Engine, Permanently Freezing User Funds - (`engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` replaces the engine's stored connector account ID with any arbitrary NEAR account without verifying that the new connector contract has the current Aurora Engine's account ID set as its `aurora_engine_account_id`. If the new connector was initialized with a different engine account ID, every subsequent `engine_withdraw` call the engine makes to that connector will be rejected, permanently freezing all bridged ETH held in the new connector.

---

### Finding Description

The ETH connector contract is initialized with an `aurora_engine_account_id` field that it uses to gate access to privileged methods such as `engine_withdraw`. Only the account whose ID matches `aurora_engine_account_id` may call `engine_withdraw` on the connector; any other caller receives "Method can be called only by aurora engine".

The engine stores which connector account to use via `set_connector_account_id`, and all outbound connector calls (withdraw, ft_transfer, ft_transfer_call, storage operations) are routed through `return_promise`, which reads that stored account ID and dispatches a cross-contract call to it.

`set_eth_connector_contract_account` updates this stored account ID unconditionally:

```rust
// engine/src/contract_methods/connector.rs lines 418-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);          // ← no back-reference check
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

There is no assertion that `args.account`'s `aurora_engine_account_id` equals `env.current_account_id()`. A new connector deployed with a different engine account ID (e.g., a staging engine, a typo, or a future engine address) will silently be accepted. After the swap, every call the engine makes to the connector via `return_promise` targets the new connector, which will reject `engine_withdraw` because the caller (the engine) does not match the connector's stored `aurora_engine_account_id`. ---

### Impact Explanation

All bridged ETH held in the new connector becomes permanently inaccessible through the engine. Users who call `withdraw` on the engine receive a cross-contract call that the connector rejects. Because the engine has no fallback path and the connector's `aurora_engine_account_id` is set at construction (not by the engine), the engine cannot recover access without the connector owner separately updating `aurora_engine_account_id` on the connector side. If that is not done, the freeze is permanent.

Impact: **Critical — Permanent freezing of all user bridged-ETH funds held in the new connector.**

---

### Likelihood Explanation

The function is restricted to the engine owner or a private (self) call, which limits the attack surface to operational mistakes rather than arbitrary callers. However, the scenario is realistic: a new connector contract is deployed for an upgrade or migration, its `aurora_engine_account_id` is set to a staging or incorrect engine account, and the owner calls `set_eth_connector_contract_account` pointing to it. The engine accepts the new account without complaint, and the mismatch is only discovered when the first `withdraw` fails. The original FlywheelCore report was rated medium under identical conditions (auth-gated setter, honest operational mistake). The room for such a mistake is non-trivial given that connector deployment and engine configuration are separate operations performed independently.

---

### Recommendation

Add a cross-contract read-and-verify step before accepting the new connector account, or at minimum add an explicit require that the caller supplies the expected `aurora_engine_account_id` of the new connector and that it matches `env.current_account_id()`. A synchronous on-chain check is not possible in NEAR without a callback, so the safest mitigation is a two-step pattern:

1. In `set_eth_connector_contract_account`, schedule a cross-contract call to `new_connector.get_aurora_engine_account_id()`.
2. In the callback, assert the returned value equals `env.current_account_id()` before writing the new connector account ID to storage.

Alternatively, require the caller to pass the expected `aurora_engine_account_id` as an argument and reject the update if it does not equal the engine's own account ID (trusting the caller to have verified it off-chain), which is a weaker but simpler guard against accidental misconfiguration.

---

### Proof of Concept

1. Deploy a new ETH connector contract initialized with `aurora_engine_account_id = "some-other-engine.near"` (any account that is not the live Aurora Engine).
2. As the Aurora Engine owner, call `set_eth_connector_contract_account` with the new connector's account ID.
3. The engine stores the new connector account without any validation.
4. A user calls `withdraw` on the engine with 1 yoctoNEAR attached.
5. The engine dispatches a cross-contract call to `new_connector.engine_withdraw(...)`.
6. The new connector rejects the call: "Method can be called only by aurora engine" (because `predecessor_account_id` is the Aurora Engine, but `aurora_engine_account_id` stored in the new connector is `"some-other-engine.near"`).
7. The user's withdrawal fails. All funds deposited into the new connector are permanently inaccessible through the engine.
