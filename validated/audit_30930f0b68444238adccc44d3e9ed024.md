Audit Report

## Title
Unchecked Signed-to-Unsigned Cast in `ledgeramount_from_amount` Produces Phantom Token Amount - (File: rs/rosetta-api/icp/src/models/amount.rs)

## Summary
`from_amount` parses a user-supplied string into `i128` and validates only that the absolute value fits in `u64`, returning negative values unchanged. `ledgeramount_from_amount` then casts the result directly to `u64` via `as`, which in Rust performs two's-complement truncation, turning e.g. `-1000_i128` into `18_446_744_073_709_550_616_u64`. This produces a `Tokens` object with a phantom astronomically large e8s value that propagates through the Rosetta API's construction and parsing logic.

## Finding Description
`from_amount` at [1](#0-0)  parses the value string into `i128`, checks only that `val.abs()` fits in `u64`, and returns `Ok(val)` — negative values pass this check without error.

`ledgeramount_from_amount` at [2](#0-1)  calls `from_amount` and immediately casts the result with `inner as u64`. In Rust, `as` on a signed-to-unsigned conversion is defined as two's-complement truncation: `(-1000_i128) as u64 == 18_446_744_073_709_550_616`. No guard exists between the two functions.

A secondary issue exists in `State::transaction` at [3](#0-2)  where the debit branch computes `(-amount) as u64`. While `from_amount`'s range validation prevents `i128::MIN` from reaching this path via the normal call chain, the method is public and the negation is unguarded.

The existing check in `from_amount` — `u64::try_from(val.abs())` — is insufficient because it only validates magnitude, not sign. A value of `-1000` has `abs() == 1000`, which fits in `u64`, so the check passes and the negative value is returned to the caller.

## Impact Explanation
The Rosetta API is an explicitly in-scope financial integration component. An unprivileged caller can cause the API to construct `Tokens` objects with phantom values exceeding `u64::MAX / 2` e8s (≈184 billion ICP equivalent). This corrupted amount propagates through construction endpoints (`/construction/parse`, `/construction/payloads`), affecting fee calculations, balance comparisons, and operation serialization within the Rosetta layer. Exchanges and integrators that rely on the Rosetta API as a trusted intermediary for amount parsing will receive silently incorrect data. This constitutes a significant Rosetta API security impact with concrete harm to data integrity and exchange integrations, qualifying as **High** under the "Significant Rosetta... security impact with concrete user or protocol harm" category.

## Likelihood Explanation
The Rosetta API is a publicly reachable HTTP service requiring no authentication. The attacker supplies a JSON body with a negative `value` string (e.g., `"-1000"`) in the `Amount` field. The `from_amount` validator is explicitly designed to accept negative values (to represent debits), so the input passes validation unconditionally and reaches the unsafe cast. The attack is trivially repeatable with a single HTTP request.

## Recommendation
In `ledgeramount_from_amount`, add an explicit non-negativity check before the cast:

```rust
pub fn ledgeramount_from_amount(amount: &Amount, token_name: &str) -> Result<Tokens, String> {
    let inner = from_amount(amount, token_name)?;
    if inner < 0 {
        return Err(format!("Amount must be non-negative, got {inner}"));
    }
    Ok(Tokens::from_e8s(inner as u64))
}
```

In `State::transaction`, use `checked_neg()` before the cast to guard against overflow:

```rust
let abs_amount = amount.checked_neg()
    .ok_or_else(|| ApiError::InvalidTransaction(false, "Amount overflow".into()))?;
tokens: Tokens::from_e8s(abs_amount as u64),
```

## Proof of Concept
1. Start the ICP Rosetta API pointed at any ICP node.
2. POST to `/construction/parse` with a `value` of `"-1000"` and currency `ICP` with 8 decimals.
3. Execution reaches `ledgeramount_from_amount` → `from_amount` returns `-1000_i128` (passes the `abs()` check) → `(-1000_i128) as u64 == 18_446_744_073_709_550_616` → `Tokens::from_e8s(18_446_744_073_709_550_616)` is constructed.
4. A unit test can confirm this directly: assert that `ledgeramount_from_amount(&Amount { value: "-1000".into(), currency: icp_currency(), metadata: None }, "ICP")` returns an `Err`, which it currently does not.
