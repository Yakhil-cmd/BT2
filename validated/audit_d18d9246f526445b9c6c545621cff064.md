### Title
Permanent Loss of ERC-20 / ETH Tokens When NEAR-Side `ft_transfer` Fails Due to Disabled `error_refund` Feature in Production Build - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) from the user's EVM balance before dispatching a NEAR-side `ft_transfer` / `ft_transfer_call` promise. If that NEAR promise fails, the refund path that re-mints the burned tokens is gated behind the `error_refund` compile-time feature flag. The production WASM build does **not** enable this flag, so a failed NEAR-side transfer results in permanent, unrecoverable loss of the user's tokens.

---

### Finding Description

**Step 1 – Tokens are burned before the NEAR promise is dispatched.**

When a user calls the `ExitToNear` precompile (flag `0x01`, ERC-20 path), the ERC-20 contract's `burn` function is invoked inside the EVM. The precompile then constructs a NEAR promise to call `ft_transfer` or `ft_transfer_call` on the corresponding NEP-141 contract.

**Step 2 – The refund field is unconditionally `None` in production.**

In `engine-precompiles/src/native.rs`, the callback arguments are assembled as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

**Step 3 – No callback is attached for the standard ERC-20 exit path.**

For the standard `ft_transfer` path (`transfer_near_args = None`, `refund = None`), `callback_args` equals the `Default` value, so the promise is created **without any callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

If `ft_transfer` fails (e.g., recipient not registered, NEP-141 paused), there is no callback to detect the failure and no re-mint of the burned tokens.

**Step 4 – Even when a callback exists (wNEAR unwrap path), the refund branch is dead.**

For the wNEAR unwrap path, `transfer_near` is `Some(...)`, so a callback is attached. However, inside `exit_to_near_precompile_callback`, the refund branch is:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← taken when refund == None; no action
};
``` [3](#0-2) 

Because `args.refund` is `None` in production, the `else` branch is always taken and no refund occurs.

**Step 5 – Production build does not include `error_refund`.**

The `Makefile.toml` defines the production build feature set as:

```
CARGO_FEATURES_BUILD = "contract"
``` [4](#0-3) 

The `error_refund` feature is defined in `engine/Cargo.toml` as an opt-in:

```toml
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [5](#0-4) 

It is never included in `CARGO_FEATURES_BUILD`, confirming the refund path is compiled out of the deployed contract.

**Step 6 – The test suite explicitly acknowledges the fund loss.**

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

This confirms that without `error_refund`, `FT_EXIT_AMOUNT` tokens are permanently lost when the NEAR-side transfer fails.

---

### Impact Explanation

**Critical – Permanent freezing / loss of user funds.**

When the NEAR-side `ft_transfer` or `near_withdraw` promise fails after the EVM-side burn has already been committed, the user's ERC-20 tokens (or ETH, via the base-token path) are permanently destroyed with no recovery path. The NEP-141 balance on NEAR is unchanged (the transfer never completed), but the EVM balance is gone. The tokens are effectively burned into the void. [7](#0-6) 

---

### Likelihood Explanation

**Medium-High.** The NEAR-side `ft_transfer` can fail for several realistic, user-triggerable reasons:

- The recipient NEAR account is not registered with the NEP-141 token (storage deposit missing).
- The NEP-141 contract is paused or has a custom transfer guard.
- The `ft_transfer_call` receiver's `ft_on_transfer` panics or returns an error.
- Insufficient attached gas causes the promise to fail.

None of these require admin access or special privileges. Any user who calls `ExitToNear` targeting an unregistered or incompatible recipient triggers the loss.

---

### Recommendation

Enable the `error_refund` feature in the production WASM build by adding it to `CARGO_FEATURES_BUILD`:

```
CARGO_FEATURES_BUILD = "contract,error_refund"
```

This ensures `refund_call_args` is populated and the `exit_to_near_precompile_callback` can re-mint burned tokens when the NEAR-side promise fails. [4](#0-3) [8](#0-7) 

---

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora (backed by a NEP-141).
2. User calls the `ExitToNear` precompile (flag `0x01`) targeting a NEAR account that is **not registered** with the NEP-141 contract.
3. The EVM executes the ERC-20 `burn`, reducing the user's balance by 100.
4. A NEAR promise is dispatched to call `ft_transfer` on the NEP-141 contract.
5. The NEP-141 contract rejects the transfer because the recipient has no storage deposit.
6. Because `error_refund` is not compiled in, no callback is attached (or the callback's refund branch is `None`).
7. The user's ERC-20 balance on Aurora is 0. The NEP-141 balance on NEAR is unchanged (still held by Aurora's account). The 100 tokens are permanently lost. [9](#0-8) [10](#0-9)

#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
