Looking at the COA contract's `supportsInterface` implementation and comparing it against what the contract actually implements.

### Title
COA Contract `supportsInterface` Omits ERC-1271 Interface ID Despite Implementing `isValidSignature` - (File: `fvm/evm/handler/coa/coa.sol`)

---

### Summary

The `COA` contract deployed for every Cadence-Owned Account (COA) on Flow EVM implements `isValidSignature` (ERC-1271) but does not declare support for the ERC-1271 interface ID in its `supportsInterface` (ERC-165) function. Any EVM contract that performs the standard ERC-165 capability check before invoking `isValidSignature` will receive `false` and refuse to treat the COA as a valid signer, silently breaking signature-based authorization for all COA accounts.

---

### Finding Description

The `COA` contract in `fvm/evm/handler/coa/coa.sol` declares itself as implementing `IERC165` and provides a `supportsInterface` function:

```solidity
function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId;
}
``` [1](#0-0) 

The contract also fully implements `isValidSignature` per ERC-1271:

```solidity
// ERC1271 requirement 
function isValidSignature(
    bytes32 _hash,
    bytes memory _sig
) external view virtual returns (bytes4) { ... }
``` [2](#0-1) 

The ERC-1271 interface ID is `bytes4(keccak256("isValidSignature(bytes32,bytes)"))` = `0x1626ba7e`. This exact value is already defined in the contract as `ValidERC1271Signature`: [3](#0-2) 

However, `supportsInterface` never checks `id == 0x1626ba7e`. The ERC-1271 interface ID is therefore absent from the ERC-165 registry of the COA contract.

The `contract.cdc` documentation explicitly states that COAs support ERC-1271:

> "From the EVM perspective, COAs are smart contract wallets that accept native token transfers and support several ERCs including ERC-165, ERC-721, ERC-777, ERC-1155, **ERC-1271**." [4](#0-3) 

The implementation contradicts this specification.

---

### Impact Explanation

Any EVM-side contract that follows the standard defensive pattern of calling `supportsInterface(0x1626ba7e)` before invoking `isValidSignature` will receive `false` from a COA address and will reject the COA as a signer. This is the standard pattern used by protocols such as Safe, OpenSea Seaport, and many ERC-4337 paymasters. COA owners are therefore unable to use their accounts as ERC-1271 signers in any such protocol, despite the COA contract fully implementing the required function. The impact is loss of interoperability for all COA accounts in the Flow EVM environment.

---

### Likelihood Explanation

The likelihood is high. ERC-1271 signature validation is a widely used primitive in DeFi and NFT protocols. The standard defensive check (`supportsInterface` before `isValidSignature`) is common practice. Any unprivileged EVM contract author can deploy a contract that performs this check, and every COA on the network will silently fail it. No special privileges or conditions are required to trigger the issue.

---

### Recommendation

Add the ERC-1271 interface ID to the `supportsInterface` function in `coa.sol`:

```solidity
// bytes4(keccak256("isValidSignature(bytes32,bytes)"))
bytes4 constant internal ERC1271InterfaceId = 0x1626ba7e;

function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId ||
        id == ERC1271InterfaceId;  // add this
}
```

The compiled ABI in `fvm/evm/handler/coa/coa_abi.json` and any downstream bytecode embedding must also be regenerated after the fix. [5](#0-4) 

---

### Proof of Concept

1. A COA is created via `EVM.createCadenceOwnedAccount()` in Cadence, producing an EVM smart contract at a deterministic address.
2. An unprivileged EVM contract author deploys the following checker:

```solidity
interface IERC165 { function supportsInterface(bytes4) external view returns (bool); }
contract Checker {
    function check(address coa) external view returns (bool) {
        // ERC-1271 interface ID: bytes4(keccak256("isValidSignature(bytes32,bytes)"))
        return IERC165(coa).supportsInterface(0x1626ba7e);
    }
}
```

3. `check(coaAddress)` returns `false` for every COA address, even though calling `isValidSignature` directly on the same address works correctly.
4. Any protocol that gates `isValidSignature` calls behind this ERC-165 check will refuse to accept COA-signed messages, silently breaking signature-based authorization for all COA owners. [6](#0-5)
