### Title
EIP-7702 `chain_id = 0` Authorization Cross-Chain Replay Enables Unauthorized Code Delegation and Fund Theft - (File: `engine-transactions/src/eip_7702.rs`)

---

### Summary

The Aurora Engine's EIP-7702 implementation accepts `AuthorizationTuple` entries with `chain_id = 0` as valid on **any** chain. Because Aurora operates multiple live deployments (mainnet chain ID `1313161554`, testnet `1313161555`, local `1313161556`), a signed off-chain authorization with `chain_id = 0` that was created on one deployment can be replayed verbatim on any other deployment by an unprivileged attacker who merely observed the original transaction. There is no per-chain cancellation mechanism: the authority cannot invalidate the authorization on a chain they did not know it was applied to, directly analogous to the Putty M-14 finding where signed off-chain orders could not be cancelled without exact parameters and without a centralized service.

---

### Finding Description

In `engine-transactions/src/eip_7702.rs`, the `authorization_list()` method of `SignedTransaction7702` validates each `AuthorizationTuple`. The chain-ID check at line 181 is:

```rust
(auth.chain_id.is_zero() || auth.chain_id == current_tx_chain_id)
``` [1](#0-0) 

When `auth.chain_id` is zero the authorization passes chain-ID validation unconditionally, regardless of which Aurora deployment is processing the transaction. The resulting `Authorization` struct is then passed through `NormalizedEthTransaction::authorization_list` into the EVM executor: [2](#0-1) 

The engine then calls `engine.call(…, transaction.authorization_list, …)` which hands the validated authorization list to the SputnikVM executor, which installs the delegation designator (`EF0100 || address`) into the authority's code slot: [3](#0-2) 

The outer transaction's `chain_id` is validated against the engine state: [4](#0-3) 

But this check applies only to the **transaction sender's** chain ID, not to the `chain_id` field inside each `AuthorizationTuple`. A `chain_id = 0` authorization signed on Aurora testnet carries a valid signature over `keccak256(0x05 || rlp([0, address, nonce]))`. That exact byte string is equally valid on Aurora mainnet, because the hash does not commit to any specific chain. [5](#0-4) 

The `AuthorizationTuple` struct stores `chain_id`, `address`, and `nonce` as the only signed fields: [6](#0-5) 

There is no on-chain registry of "used" or "cancelled" authorizations. The only cancellation mechanism is for the authority to increment their nonce on the target chain — but they cannot do this if they do not know the authorization was replayed there.

---

### Impact Explanation

EIP-7702 delegation installs arbitrary bytecode into the authority's account code slot. Any subsequent call to that account executes the delegated code **in the authority's storage and balance context**. If the delegated contract address on the target chain is controlled by the attacker (e.g., deployed via CREATE2 to the same address as a legitimate contract on testnet), the attacker can:

1. Drain all ETH from the authority's account.
2. Approve all ERC-20 token balances to the attacker.
3. Execute arbitrary state mutations on behalf of the authority.

This constitutes **direct theft of user funds** — a Critical impact.

---

### Likelihood Explanation

- Aurora has three live deployments with distinct chain IDs. Any user who signs a `chain_id = 0` authorization on testnet (nonce 0, the common starting state for a new account) is immediately vulnerable on mainnet if their mainnet account also has nonce 0.
- The `chain_id = 0` path is explicitly documented and tested as "chain-agnostic authorization": [7](#0-6) 

- Wallets and dApps that want cross-chain compatibility may deliberately use `chain_id = 0`.
- The attacker only needs to monitor public testnet transactions for `chain_id = 0` authorizations, then replay them on mainnet. No privileged access is required.

---

### Recommendation

1. **Reject `chain_id = 0` authorizations in Aurora Engine.** Since Aurora is not Ethereum mainnet, there is no legitimate reason to accept chain-agnostic authorizations. Add a check:

```rust
if auth.chain_id.is_zero() {
    is_valid = false;
}
```

2. **Alternatively**, if cross-chain authorizations are intentionally supported, document the replay risk prominently and require dApps to use the specific Aurora chain ID.

3. **Add a per-authority nonce-range invalidation mechanism** (analogous to Seaport's counter or the `minimumValidNonce` pattern recommended in Putty M-14) so that an authority can invalidate all outstanding authorizations on a given chain with a single transaction, without needing to know which specific authorizations were replayed.

---

### Proof of Concept

1. Alice creates a fresh account on Aurora testnet (nonce = 0) and on Aurora mainnet (nonce = 0).
2. Bob deploys a malicious contract `M` on Aurora mainnet at address `X` using CREATE2. `M` contains: `CALLER SELFBALANCE PUSH1 0 PUSH1 0 PUSH1 0 PUSH1 0 CALLER GAS CALL` (transfers all ETH to caller).
3. Alice signs an EIP-7702 authorization on Aurora testnet: `keccak256(0x05 || rlp([0, X, 0]))` — `chain_id = 0`, `address = X`, `nonce = 0`.
4. Bob observes Alice's authorization tuple `(chain_id=0, address=X, nonce=0, parity, r, s)` from the public testnet transaction.
5. Bob constructs a new EIP-7702 transaction on Aurora **mainnet** (`chain_id = 1313161554`) with Alice's authorization tuple in `authorization_list`, signed by Bob as the transaction sender.
6. Bob submits the transaction via the `submit` entrypoint. The engine validates: `auth.chain_id.is_zero()` → `true` → authorization is valid. Alice's mainnet account nonce is 0, matching `auth.nonce`. The EVM installs `EF0100 || X` into Alice's code slot. Alice's mainnet nonce becomes 1.
7. Bob calls Alice's mainnet address. The delegated code `M` executes in Alice's context and transfers her entire ETH balance to Bob.
8. Alice has no way to prevent or detect this until after the theft, because she never submitted any transaction to Aurora mainnet and has no reason to monitor it. [1](#0-0) [8](#0-7)
