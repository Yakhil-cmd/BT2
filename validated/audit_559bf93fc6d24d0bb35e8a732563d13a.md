### Title
Unchecked Precompile Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent ERC-20 Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` burns ERC-20 tokens **before** calling the exit precompile and never checks the precompile call's return value. If the precompile rejects the call for any reason (amount exceeds `u128::MAX`, invalid NEAR account ID, oversized input), the tokens are permanently destroyed on Aurora while the corresponding NEP-141 tokens on NEAR are never transferred. The result is an irreversible accounting discrepancy and permanent fund loss for the caller.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow the same unsafe pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked — silent failure
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum`: [2](#0-1) 

In Solidity inline assembly, a failed `call` returns `0` in `res` but **does not revert the caller**. Execution continues normally after the assembly block. Because `_burn` has already executed, the tokens are gone.

The `ExitToNear` precompile (`native.rs`) enforces several hard rejections that can silently fail the call:

1. **Amount exceeds `u128::MAX`** — `parse_amount` returns `Err` for any `U256 > u128::MAX`: [3](#0-2) 

2. **Invalid NEAR account ID** — `parse_recipient` returns `Err` for malformed UTF-8 or an account ID that fails NEAR's validation: [4](#0-3) 

3. **Input exceeds `MAX_INPUT_SIZE` (1 024 bytes)** — `validate_input_size` rejects oversized payloads: [5](#0-4) 

In every one of these cases the precompile returns an `ExitError`, the EVM sub-call returns `0`, and `EvmErc20.sol` silently returns to the caller with the tokens already burned.

The `amount` parameter is declared as `uint256` (ERC-20 standard), while the precompile only accepts values up to `u128::MAX`. Any user whose balance exceeds `u128::MAX` — or who simply passes a value above that threshold — will trigger this path. The ERC-20 `mint` function accepts a full `uint256`: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ERC-20 tokens are burned on Aurora (supply decreases). The corresponding NEP-141 tokens on NEAR are never released (supply on NEAR is unchanged). The user has no recovery path: the ERC-20 tokens no longer exist, and the NEP-141 tokens remain locked in the connector with no mechanism to reclaim them. This is an irreversible accounting discrepancy in the ERC-20 mirror system — the direct Aurora-Engine analog of the CLMM bin accounting manipulation that causes one side of the pool to lose tokens without the other side compensating.

---

### Likelihood Explanation

**Medium.**

- Any user who calls `withdrawToNear` with `amount > u128::MAX` (a valid `uint256`) triggers the bug deterministically.
- Any user who supplies a malformed NEAR account ID (empty string, invalid characters, too-long string) triggers it.
- The `recipient` parameter is a raw `bytes` argument with no on-chain pre-validation before the burn; a simple typo in the recipient string is sufficient.
- No special privileges are required; any EVM address that holds tokens can call these functions.

---

### Recommendation

1. **Check the assembly return value and revert on failure:**

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Validate inputs before burning** (check-effects-interactions): validate `amount <= type(uint128).max` and that `recipient` is a syntactically valid NEAR account ID before calling `_burn`.

3. Apply the same fix to `withdrawToEthereum`.

---

### Proof of Concept

1. Admin mints `2**128` (> `u128::MAX`) tokens to attacker's address via `mint(attacker, 2**128)`.
2. Attacker calls `withdrawToNear("alice.near", 2**128)`.
3. `_burn(attacker, 2**128)` executes — attacker's ERC-20 balance drops to zero.
4. The precompile call is made; `parse_amount` rejects the value because `2**128 > u128::MAX`, returning `ExitError`.
5. The EVM sub-call returns `0` into `res`; the assembly block does not revert.
6. `withdrawToNear` returns normally.
7. Attacker holds zero ERC-20 tokens; zero NEP-141 tokens are transferred on NEAR.
8. Funds are permanently destroyed — no recovery path exists.

fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));

    Ok(amount)
}

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {

        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
