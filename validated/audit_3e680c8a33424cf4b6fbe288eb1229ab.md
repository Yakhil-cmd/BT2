### Title
Unbounded `MessageBuffer` Growth via Arbitrary Waitpoint Injection - (File: `src/protocol/internal.rs`)

### Summary
A malicious participant can send protocol messages with arbitrary, never-polled `Waitpoint` values, causing the `MessageBuffer` (a `HashMap<MessageHeader, SubMessageQueue>`) inside `Comms` to grow without bound. Each unique `(channel_tag, waitpoint)` pair creates a new `HashMap` entry backed by an `UnboundedSender`/`UnboundedReceiver` pair that is never drained. This is a confirmed, self-documented resource-exhaustion path: the codebase contains a test named `attacker_can_fill_message_buffer_with_unused_waitpoints` that explicitly proves the attack succeeds.

### Finding Description

The `MessageBuffer` struct holds a `HashMap<MessageHeader, SubMessageQueue>` protected by a `Mutex`: [1](#0-0) 

Each `SubMessageQueue` is backed by `futures::channel::mpsc::unbounded()` — an unbounded in-memory channel: [2](#0-1) 

The `push` method unconditionally inserts a new `HashMap` entry for any previously-unseen `MessageHeader`: [3](#0-2) 

The public entry point `Comms::push_message` performs only a length check and header parse before calling `push` — there is no cap on the number of distinct headers, no cap on the number of messages per header, and no eviction: [4](#0-3) 

`push_message` is called directly from the `Protocol` trait implementation's `message()` method, which is the standard external message-delivery interface: [5](#0-4) 

A `MessageHeader` is a 40-byte value: a 32-byte `ChannelTag` plus an 8-byte `Waitpoint` (a `u64`). An attacker can craft messages with the correct channel tag prefix but with `Waitpoint` values the honest protocol code never polls (e.g., `1_000_000`, `1_000_001`, …). Each such message inserts a new `HashMap` entry. The honest party never pops these entries, so they accumulate indefinitely.

The codebase itself documents this as a known, unmitigated attack surface in a test: [6](#0-5) 

The test asserts that after 512 injected messages with distinct unused waitpoints, the `HashMap` contains exactly 512 entries — confirming unbounded growth with no rejection or eviction.

### Impact Explanation

Memory is consumed at a rate proportional to the number of crafted messages the attacker sends. Each `HashMap` entry allocates:
- A `SubMessageQueue` struct (two heap-allocated channel endpoints)
- The message payload itself (stored in the unbounded channel)

An attacker who can deliver messages to an honest party's `Protocol::message()` can exhaust the process's heap, causing an OOM kill or severe memory pressure. This permanently terminates or stalls the honest party's participation in DKG, reshare, refresh, presign, or signing sessions — matching the **High: Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties** impact, or at minimum **Medium: Griefing or resource-exhaustion causing unbounded memory growth beyond documented behavior**.

### Likelihood Explanation

The `Protocol::message()` method is the standard, documented interface for delivering network messages to a protocol participant. Any participant in the session (or any network-level adversary who can inject messages) can trigger this path. No special privilege is required — the attacker only needs to be able to send bytes to the honest party's message-delivery interface. The `Waitpoint` field is a plain `u64` in the message header, trivially enumerable. The attack requires no cryptographic knowledge.

### Recommendation

1. **Cap the number of distinct `MessageHeader` entries** in `MessageBuffer`. Reject (or drop) messages whose header is not in a pre-approved set of expected waitpoints for the current protocol phase.
2. **Cap the per-header queue depth** — bound the number of messages stored per `(channel, waitpoint)` pair to a small constant (e.g., `N`, the participant count).
3. **Validate the waitpoint range** at ingress: honest protocol code only ever polls a small, statically-known set of waitpoints per session. Messages referencing out-of-range waitpoints should be silently dropped in `push_message` before any allocation occurs.

### Proof of Concept

The repository's own test demonstrates the attack directly:

```rust
// src/protocol/internal.rs, line 532
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());

        // Attacker injects messages for waitpoints the honest code never polls.
        comms.push_message(attacker, message);
    }

    // HashMap now has 512 entries — all leaked, none ever consumed.
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
```

Scale `attack_count` to `u64::MAX` iterations (or simply send at high rate) and the process OOMs. The attacker controls the `Waitpoint` field (bytes 32–39 of any message) and the channel tag is derivable from the public `ChannelTag::root_shared()` constant, making the crafted header trivially constructable by any network participant.
