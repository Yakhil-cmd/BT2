### Title
TOCTOU Split-Fetch: `prepareSubmitData` Re-fetches Pool Info Without Validation, Allowing Malicious Pool to Substitute `targetPuzzleHash` Between Display and Submission — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

`usePoolInfo` fetches and displays pool data (including `targetPuzzleHash`) for user review. At submit time, `prepareSubmitData` makes a **second, independent** call to `getPoolInfo` with no URL validation and no reuse of the already-fetched data. A malicious pool operator can serve a legitimate `targetPuzzleHash` on the first request (display) and a different one on the second request (submission), causing the user to join a pool whose payout address they never actually approved.

### Finding Description

**Display path — `usePoolInfo`:**

`usePoolInfo` validates the URL (enforcing `https` on mainnet), fetches pool info, and returns it for display in the UI. [1](#0-0) 

**Submission path — `prepareSubmitData`:**

`prepareSubmitData` calls `normalizeUrl` and then immediately calls `getPoolInfo` again — with **no** `isValidURL` check, no mainnet/testnet check, and no reuse of the data already fetched and shown to the user. [2](#0-1) 

The `targetPuzzleHash` and `relativeLockHeight` used in the actual join transaction come exclusively from this second fetch: [3](#0-2) 

`getPoolInfo` itself is a thin wrapper with no validation: [4](#0-3) 

**Additional gap — testnet http URLs:**

On testnet (`isMainnet=false`), `usePoolInfo` omits the `protocols: ['https']` restriction, so `http://` URLs pass validation. [5](#0-4) 

The electron-side `isValidURL` (used by `fetchPoolInfo` in the main process) only allows `https` and `ipfs`, which would block plain `http://` at the IPC boundary. However, the TOCTOU gap is protocol-agnostic and applies equally to `https://` pools on mainnet. [6](#0-5) 

### Impact Explanation

The `targetPuzzleHash` is the pool's declared payout address for farming rewards. By serving a legitimate-looking puzzle hash on the first request (display) and substituting their own address on the second request (submission), a malicious pool operator causes the user to sign and submit a join transaction that routes all future farming rewards to the attacker's address. The user sees one destination but the on-chain transaction encodes a different one. This is a direct, permanent misdirection of pooled farming rewards.

### Likelihood Explanation

Any pool operator can implement this trivially: serve the real `targetPuzzleHash` for the first `/pool_info` request and a different one for the second. No special access is required — the attacker only needs to operate a pool that a user chooses to join. The window between display and submission is the normal user interaction time (reading the UI, clicking submit).

### Recommendation

- In `prepareSubmitData`, do not re-fetch pool info. Instead, pass the already-validated `poolInfo` object from `usePoolInfo` through the form state or as a parameter, and use that data directly for the transaction.
- If a second fetch is deemed necessary, compare the `targetPuzzleHash` and `relativeLockHeight` from both responses and abort with an error if they differ.
- Apply the same URL validation logic in `prepareSubmitData` as in `usePoolInfo` (including the mainnet https enforcement).

### Proof of Concept

1. Stand up a pool server at `https://evil-pool.example.com` that serves `/pool_info`.
2. For the **first** request, return `{ "target_puzzle_hash": "<legitimate_hash>", "relative_lock_height": 32, ... }`.
3. For the **second** request (and all subsequent), return `{ "target_puzzle_hash": "<attacker_hash>", "relative_lock_height": 32, ... }`.
4. On a testnet node, open the GUI, navigate to "Join a Pool", enter `https://evil-pool.example.com`.
5. Observe the UI displays `<legitimate_hash>` (from `usePoolInfo`'s fetch).
6. Click submit. `prepareSubmitData` fires a second fetch, receives `<attacker_hash>`, and submits the join transaction with that hash.
7. Confirm on-chain that the PlotNFT's `pool_puzzle_hash` is `<attacker_hash>`, not `<legitimate_hash>`.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
