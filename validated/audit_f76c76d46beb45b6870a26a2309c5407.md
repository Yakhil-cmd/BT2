### Title
Echo-Broadcast Threshold Clamped to Zero for `n ≤ 3` Defeats Consistency Guarantee, Enabling DKG Output Corruption - (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

`echo_ready_thresholds` returns `(0, 0)` for any participant count `n ≤ 3`. This clamps both the echo and ready thresholds to zero, making every single-vote comparison trivially true and completely defeating the consistency guarantee of the echo-broadcast protocol. A malicious participant in a valid 3-party DKG can equivocate — sending different polynomial commitments to different honest parties — causing them to compute different aggregate public keys and corrupting the DKG output.

---

### Finding Description

In `src/protocol/echo_broadcast.rs`, the threshold computation function is: [1](#0-0) 

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // ← both thresholds clamped to zero
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

These thresholds are consumed directly in `reliable_broadcast_receive_all`: [2](#0-1) 

- **Echo phase delivery condition:** `count > echo_t` → with `echo_t = 0`, any single echo vote (including the node's own simulated echo) immediately triggers the ready phase. [3](#0-2) 

- **Final delivery condition:** `count > 2 * ready_t` → with `ready_t = 0`, `2 * 0 = 0`, so a single ready vote (the node's own simulated ready) immediately delivers. [4](#0-3) 

The critical consequence: each honest party processes its own simulated echo and simulated ready vote **before** receiving any real messages from other parties. With thresholds of 0, each party delivers immediately based solely on what the sender told it, with no cross-checking from other participants. A malicious sender can therefore present different values to different honest parties, and each honest party will deliver the value it was shown — with no opportunity for the discrepancy to be detected.

The echo-broadcast is used in DKG to broadcast polynomial commitments: [5](#0-4) 

The library enforces only `n ≥ 2` and `threshold ≥ 2` in `assert_key_invariants`, making `n = 3, threshold = 2` a fully valid and accepted configuration: [6](#0-5) 

**Structural analog to the reported bug:** Just as `properDepositRatio` clamped `depositRatio` and `hypeRatio` to `[0.1, 10]` — making the comparison trivially pass even when the true ratio was `10^200` — `echo_ready_thresholds` clamps both thresholds to `0` when `n ≤ 3`, making the consistency check trivially pass even when a malicious party is actively equivocating.

---

### Impact Explanation

With `n = 3` (parties A, B, C) and C malicious:

1. C sends `Send(commitment_1)` to A and `Send(commitment_2)` to B.
2. A echoes `commitment_1`, simulates its own echo → `count(commitment_1) = 1 > 0` → sends `Ready(commitment_1)`, sets `finish_echo = true`.
3. A simulates its own ready → `count(commitment_1) = 1 > 0` → delivers `commitment_1`, sets `finish_ready = true`.
4. A subsequently receives `Echo(commitment_2)` from B → skipped (`finish_echo = true`).
5. A subsequently receives `Ready(commitment_2)` from B → skipped (`finish_ready = true`).
6. B follows the symmetric path and delivers `commitment_2`.

A and B now hold inconsistent polynomial commitments for C. They compute:
- A: `public_key_A = A_0 + B_0 + C_0_for_A`
- B: `public_key_B = A_0 + B_0 + C_0_for_B`

These are different public keys. The DKG output is corrupted. Subsequent signing operations will produce signature shares that are inconsistent with each other, making valid threshold signatures impossible for honest parties.

**Impact category:** High — Corruption of DKG outputs so honest parties accept inconsistent public keys.

---

### Likelihood Explanation

`n = 3, threshold = 2` is a natural and common configuration for threshold protocols (e.g., 2-of-3 multisig). The library explicitly accepts it — `assert_key_invariants` enforces only `n ≥ 2` and `threshold ≥ 2` with no additional constraint requiring `n ≥ 4`. The attacker needs no privileged access: any one of the three participants who acts maliciously during DKG can execute this attack by simply sending different `Send` messages to different peers, which is a standard network-layer capability.

---

### Recommendation

1. **Enforce `n ≥ 4` at the DKG initialization layer** (in `assert_key_invariants`) to guarantee that the echo-broadcast protocol always operates with at least one Byzantine fault tolerance margin (`n ≥ 3f + 1` with `f ≥ 1`).
2. **Alternatively**, remove the `n ≤ 3` special case and compute thresholds uniformly. For `n = 3`, `broadcast_threshold = (3-1)/3 = 0` anyway under integer division, so the special case provides no benefit and only obscures the security gap.
3. **Document** the minimum participant count required for Byzantine fault tolerance if the special case is intentionally retained.

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0

    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
