### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield via Trace Inflation - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `Pool` contains an explicitly unbounded loop over a pool member's `pool_member_epoch_balance` trace. Because the reward address (a distinct, unprivileged actor) is permitted to call `add_to_delegation_pool` on behalf of a pool member, it can inflate the trace across many epochs. Once the trace is large enough, `claim_rewards` will exceed the Starknet transaction gas limit and revert on every attempt, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

`calculate_rewards` iterates over every unprocessed entry in the `pool_member_epoch_balance` trace from `entry_to_claim_from` up to the current trace length:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows by one entry per epoch in which `set_member_balance` is called with a new epoch key. The `insert` function only merges into the last entry when the key matches; otherwise it appends:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

`set_member_balance` is called with key `current_epoch + K` on every invocation of `add_to_delegation_pool`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(
        balance: amount,
        cumulative_rewards_trace_idx: self.cumulative_rewards_trace_length() + 1,
    );
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [3](#0-2) 

Critically, `add_to_delegation_pool` permits the **reward address** — a separate entity from the pool member — to call it:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [4](#0-3) 

The `entry_to_claim_from` cursor that bounds the loop is **only advanced on a successful `claim_rewards` call**:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
self.write_pool_member_info(:pool_member, :pool_member_info);
``` [5](#0-4) 

If `claim_rewards` reverts due to gas exhaustion, `entry_to_claim_from` is never written, so every subsequent call must re-process the same oversized trace and will also revert.

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace for a victim pool member is inflated beyond the gas budget of a single Starknet transaction, `claim_rewards` reverts on every call. Because `entry_to_claim_from` is only persisted on success, there is no way to process the trace in smaller batches — the pool member's entire accumulated unclaimed yield is permanently frozen. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation

The reward address is a first-class protocol actor that can be any address, including one controlled by a griefing party. The attack requires:

1. The reward address to hold a minimal token balance (1 unit per epoch).
2. Calling `add_to_delegation_pool` once per epoch while the staker is active.
3. Waiting for the pool member to accumulate enough unclaimed entries that `claim_rewards` exceeds the gas limit.

The cost is proportional to the number of epochs × minimum token amount, making this a low-cost griefing attack. The staker being active is the only prerequisite, which is the normal operating state of the protocol.

---

### Recommendation

1. **Paginate `calculate_rewards`**: Add a `max_entries` parameter so the loop processes at most N entries per call, advancing `entry_to_claim_from` and writing it to storage even on a partial run. This allows the pool member to drain the trace over multiple transactions.
2. **Restrict `add_to_delegation_pool` to the pool member only**: Remove the reward address's ability to call this function, eliminating the external inflation vector.
3. **Alternatively**, cap the number of unprocessed trace entries (e.g., require `claim_rewards` before a new balance change can be recorded if the backlog exceeds a threshold).

---

### Proof of Concept

**Setup**: Pool member `P` has reward address `R` (controlled by attacker). Staker is active.

**Attack loop** (repeat for N epochs, where N is large enough to exceed gas):
```
for each new epoch e:
    R calls add_to_delegation_pool(pool_member=P, amount=1)
    → set_member_balance called with key = e + K
    → new checkpoint appended to pool_member_epoch_balance[P]
```

After N iterations, `pool_member_epoch_balance[P]` has N entries and `entry_to_claim_from = 0` (P never claimed).

**Trigger**:
```
P calls claim_rewards(pool_member=P)
→ calculate_rewards loops over N entries
→ transaction runs out of gas and reverts
→ entry_to_claim_from remains 0
→ all future claim_rewards calls also revert
```

P's accumulated yield is permanently inaccessible. [6](#0-5) [7](#0-6) [8](#0-7)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
