### Title
Gateway L2 Gas Price Validation Uses Stale Previous-Block Price, Admitting Transactions the Batcher Rejects — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validator checks a transaction's `l2_gas.max_price_per_unit` against the **previous block's** L2 gas price, while the batcher executes those same transactions against the **next block's** L2 gas price (computed via EIP-1559). When blocks are full and the gas price is rising, transactions that clear the gateway's threshold are admitted to the mempool but then fail at batcher execution with `InsufficientResourceBounds / MaxGasPriceTooLow`. This is the direct Sequencer analog of the CIF lagging bug: two related accounting values (`previous_block_l2_gas_price` used for admission vs. `next_block_l2_gas_price` used for execution) are read from different points in time, making the admission check systematically less strict than the execution check.

---

### Finding Description

**Step 1 — Gateway admission check (stale price)**

`validate_resource_bounds` in `StatefulTransactionValidator` reads the gas price from the last committed block and checks the transaction against it:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is `min_gas_price_percentage% × previous_block_l2_gas_price` (default 100 %): [2](#0-1) 

The TODO comment is the developers' own acknowledgement that the correct value to use is the **next** block's price, not the previous block's price.

**Step 2 — Gateway blockifier validation also uses the stale price**

`run_validate_entry_point` builds a `BlockContext` from the same previous-block `BlockInfo` (only the block number is bumped, gas prices are unchanged):

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [3](#0-2) 

So `check_fee_bounds` inside `perform_pre_validation_stage` also runs against `previous_block_l2_gas_price` during gateway validation.

**Step 3 — Batcher execution uses the actual next-block price**

The consensus orchestrator computes the next block's L2 gas price via EIP-1559 (`calculate_next_base_gas_price`) and embeds it in the `ProposalInit`. The batcher builds its `BlockContext` from that proposal, so `check_fee_bounds` at execution time uses `next_block_l2_gas_price`: [4](#0-3) 

**Step 4 — The blockifier's `check_fee_bounds` enforces the actual price**

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(
        ResourceBoundsError::MaxGasPriceTooLow { ... }
    );
}
``` [5](#0-4) 

If `next_block_l2_gas_price > previous_block_l2_gas_price`, a transaction whose `max_price_per_unit` equals `previous_block_l2_gas_price` passes the gateway but fails here.

**Step 5 — The `GatewayFixedBlockSyncStateClient` returns the current block's price, not `next_l2_gas_price`**

The block header carries a `next_l2_gas_price` field (visible in the protobuf converter), but `get_block_info_from_sync_client` populates `gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price` (the current block's price), not `block_header.next_l2_gas_price`: [6](#0-5) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts transactions that are rejected before sequencing.**

Any `AllResources` V3 transaction submitted with `l2_gas.max_price_per_unit` equal to exactly the previous block's L2 gas price passes all gateway checks and enters the mempool. When the batcher picks it up and calls `perform_pre_validation_stage` with the next block's (higher) gas price, `check_fee_bounds` returns `InsufficientResourceBounds { MaxGasPriceTooLow }` and the transaction is rejected. The user's nonce is not advanced (the transactional state is rolled back), no fee is charged, and the transaction must be resubmitted. Under sustained high load (EIP-1559 price climbing every block), this affects every transaction priced at exactly the previous block's floor.

---

### Likelihood Explanation

The condition is triggered whenever the next block's L2 gas price exceeds the previous block's price, which happens every time a block is above the gas target. EIP-1559 allows the price to increase by up to `price × (gas_used − gas_target) / (gas_target × gas_price_max_change_denominator)` per block. During periods of sustained high demand this is a routine, predictable occurrence. The default `min_gas_price_percentage = 100` means the gateway accepts transactions priced at exactly the previous block's price, which is the most common "minimum viable" bid a wallet would construct.

---

### Recommendation

Replace the stale `l2_gas_price` field with `next_l2_gas_price` in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`. The `next_l2_gas_price` field is already present in `BlockHeaderWithoutHash` and is already serialised in the protobuf wire format. The gateway's `validate_resource_bounds` and the `BlockContext` built in `run_validate_entry_point` should both use this value so that the admission check is at least as strict as the execution check. The existing TODO comment (`// TODO(Arni): getnext_l2_gas_price from the block header.`) already identifies the correct fix location.

---

### Proof of Concept

```
Block N committed:
  l2_gas_price (fri) = 100
  next_l2_gas_price  = 112   (EIP-1559, block was full)

User submits AllResources invoke tx:
  l2_gas.max_price_per_unit = 100

Gateway stateful validation:
  previous_block_l2_gas_price = 100   (from get_block_info())
  threshold = 100% × 100 = 100
  100 >= 100  → PASS
  check_fee_bounds (BlockContext built with price=100) → PASS
  → Transaction admitted to mempool

Batcher builds block N+1:
  BlockContext.gas_prices.strk_gas_prices.l2_gas_price = 112

  perform_pre_validation_stage → check_fee_bounds

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
