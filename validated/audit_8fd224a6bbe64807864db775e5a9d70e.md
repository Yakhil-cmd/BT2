### Title
Missing Maximum Commission Guard in Delegation Entry and Switch Functions Allows Staker Front-Running - (File: src/pool/pool.cairo)

### Summary

`enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool` in `src/pool/pool.cairo` accept no `max_commission` argument. Because `set_commission_commitment` + `set_commission` allow a staker to raise commission up to an arbitrary `max_commission` ceiling in a single transaction, a staker can front-run a delegator's entry or pool-switch and silently impose a much higher commission rate than the delegator observed before signing.

### Finding Description

The `set_commission_commitment` function lets a staker publish a commitment that unlocks commission **increases** up to `max_commission` within a validity window: [1](#0-0) 

Once the commitment is active, `set_commission` permits raising the rate to any value ≤ `max_commission` in one call: [2](#0-1) 

The protocol itself acknowledges this risk in a developer note immediately above `set_commission_commitment`: [3](#0-2) 

The three delegation entry points that a delegator calls carry no `max_commission` guard:

```
fn enter_delegation_pool(ref self: ContractState, reward_address: ContractAddress, amount: Amount)
fn add_to_delegation_pool(ref self: ContractState, pool_member: ContractAddress, amount: Amount)
fn switch_delegation_pool(ref self: ContractState, to_staker: ContractAddress, to_pool: ContractAddress, amount: Amount)
``` [4](#0-3) [5](#0-4) [6](#0-5) 

None of these functions read or validate the current commission before executing. The commission is only applied later, at reward distribution time (attestation), via `split_rewards_with_commission`: [7](#0-6) 

### Impact Explanation

A delegator who switches to or enters a pool expecting a 10% commission rate can be silently placed into a pool charging 50% (or up to 100%) commission. Every future reward epoch the staker captures the excess commission share that the delegator never agreed to. This constitutes ongoing theft of the delegator's yield: the staker's commission rewards grow at the delegator's direct expense, epoch after epoch, until the delegator notices and exits.

This maps to **High: Theft of unclaimed yield** — the staker continuously siphons yield that would otherwise accrue to the delegator.

### Likelihood Explanation

- The commitment is a public on-chain event (`CommissionCommitmentSet`), but Starknet's mempool ordering means a staker can observe a pending `enter_delegation_pool` or `switch_delegation_pool` transaction and submit `set_commission` with higher gas to execute first.
- The window for the attack is the entire `expiration_epoch` duration (up to one year), giving the staker ample time to execute the front-run.
- The delegator has no on-chain mechanism to revert if the commission has changed since they signed.
- `switch_delegation_pool` is particularly exposed: the delegator is already in an exit window and cannot re-enter the original pool without a new intent cycle, so they are effectively trapped in the high-commission pool.

### Recommendation

Add a `max_commission: Commission` parameter to `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool`. At the start of each function, assert that the current commission (fetched from the staking contract) does not exceed the caller-supplied ceiling, reverting with a descriptive error if it does. This mirrors the standard slippage-protection pattern and gives delegators a trustless guarantee about the terms they are accepting.

### Proof of Concept

1. Staker deploys a STRK pool with commission = 1000 (10%).
2. Staker calls `set_commission_commitment(max_commission: 5000, expiration_epoch: current_epoch + 1)`.
3. Delegator observes commission = 10% and submits `switch_delegation_pool(to_staker, to_pool, amount)`.
4. Staker front-runs by calling `set_commission(commission: 5000)` — succeeds because the commitment is active. [8](#0-7) 
5. Delegator's `switch_delegation_pool` executes with no commission check; the delegator is now a member of the pool. [9](#0-8) 
6. At the next attestation, `split_rewards_with_commission` applies 50% commission, and the staker receives 5× the commission the delegator expected. [10](#0-9) 
7. The delegator must call `exit_delegation_pool_intent` and wait the full exit window before they can leave, during which time they continue to lose yield.

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );
        }

        fn switch_delegation_pool(
            ref self: ContractState,
            to_staker: ContractAddress,
            to_pool: ContractAddress,
            amount: Amount,
        ) -> Amount {
            // Asserts.
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            assert!(
                pool_member_info.unpool_time.is_some(),
                "{}",
                GenericError::MISSING_UNDELEGATE_INTENT,
            );
            assert!(amount <= pool_member_info.unpool_amount, "{}", GenericError::AMOUNT_TOO_HIGH);
            let reward_address = pool_member_info.reward_address;

            // Update pool_member_info and write to storage.
            pool_member_info.unpool_amount -= amount;
            if pool_member_info.unpool_amount.is_zero() {
                // unpool_amount is zero, clear unpool_time.
                pool_member_info.unpool_time = Option::None;
            }
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Serialize the switch pool data and invoke the staking contract to switch pool.
            let switch_pool_data = SwitchPoolData { pool_member, reward_address };
            let mut serialized_data = array![];
            switch_pool_data.serialize(ref output: serialized_data);
            self
                .staking_pool_dispatcher
                .read()
                .switch_staking_delegation_pool(
                    :to_staker,
                    :to_pool,
                    switched_amount: amount,
                    data: serialized_data.span(),
                    identifier: pool_member.into(),
                );
