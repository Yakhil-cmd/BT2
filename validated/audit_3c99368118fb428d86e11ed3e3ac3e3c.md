Audit Report

## Title
Unchecked Arithmetic in SNS Governance `distribute_rewards` Causes Silent Maturity Wrap-Around - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance `distribute_rewards` function uses bare `+` and `+=` operators on `u64` fields when accumulating neuron maturity rewards. In Rust release builds compiled to wasm32 (the IC canister target), unsigned integer overflow wraps silently. The NNS governance equivalent uses `saturating_add` for the same operations. This inconsistency means a neuron's `maturity_e8s_equivalent` or `staked_maturity_e8s_equivalent` can silently wrap to near-zero, causing permanent, undetected loss of maturity entitlements with no on-chain error or event emitted.

## Finding Description
In `rs/sns/governance/src/governance.rs`, the reward distribution loop contains three unchecked `u64` additions: [1](#0-0) 

Specifically:
1. `neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s` (line 5991) — bare `+` on `u64`
2. `neuron.maturity_e8s_equivalent += neuron_reward_e8s` (line 5994) — bare `+=` on `u64`
3. `distributed_e8s_equivalent += neuron_reward_e8s` (line 5996) — bare `+=` on `u64`

The NNS governance equivalent in `rs/nns/governance/src/reward/distribution.rs` uses `saturating_add` for both maturity fields: [2](#0-1) 

The workspace `Cargo.toml` does not set `overflow-checks = true` for release profiles, and no such setting was found in the Bazel build configuration for production canister builds. In Rust release/wasm32 builds, unsigned integer overflow wraps silently per the Rust reference. The `distributed_e8s_equivalent` value is subsequently stored directly into the `RewardEvent` record: [3](#0-2) 

The `debug_assert!` at line 6004–6007 that checks `distributed_e8s_equivalent <= rewards_purse_e8s` is a no-op in release builds, providing no protection. [4](#0-3) 

## Impact Explanation
This maps to **High ($2,000–$10,000)**: "Significant SNS security impact with concrete user or protocol harm." A neuron holder whose `maturity_e8s_equivalent` or `staked_maturity_e8s_equivalent` wraps to near-zero permanently loses their accumulated maturity — a direct financial loss with no recovery path, since maturity represents a future SNS-token minting entitlement. The `distributed_e8s_equivalent` wrap also corrupts the stored `RewardEvent`, potentially affecting downstream reward accounting. No privileged access is required; any neuron holder who votes is affected once the overflow threshold is reached.

## Likelihood Explanation
The overflow threshold is `u64::MAX ≈ 1.84 × 10^19 e8s ≈ 184 billion tokens`. This requires an SNS with an extremely large token supply (e.g., 200 billion tokens) and a neuron accumulating maturity over many rounds. For `distributed_e8s_equivalent`, the threshold must be crossed in a single reward round across all neurons combined. While this is a high bar, it is not architecturally prevented — the SNS framework imposes no hard cap on token supply at this scale, and the bug is reachable by an unprivileged neuron holder simply by voting. The NNS governance team already identified this pattern as requiring `saturating_add`, confirming awareness of the risk class.

## Recommendation
Replace all three bare arithmetic operations in `distribute_rewards` with `saturating_add`, matching the NNS governance implementation:

```rust
// staked maturity
neuron.staked_maturity_e8s_equivalent = Some(
    neuron.staked_maturity_e8s_equivalent.unwrap_or(0)
        .saturating_add(neuron_reward_e8s),
);
// unstaked maturity
neuron.maturity_e8s_equivalent =
    neuron.maturity_e8s_equivalent.saturating_add(neuron_reward_e8s);
// tracking variable
distributed_e8s_equivalent =
    distributed_e8s_equivalent.saturating_add(neuron_reward_e8s);
```

Add a property-based or invariant test asserting that `distribute_rewards` never decreases any neuron's maturity fields and that `distributed_e8s_equivalent` never exceeds `rewards_purse_e8s`.

## Proof of Concept
1. Deploy an SNS with `total_token_supply` set to a value whose e8s representation exceeds `u64::MAX` (e.g., 200 billion tokens = 2×10^19 e8s).
2. Create a single neuron holding the entire voting power with `auto_stake_maturity = true`.
3. Run reward distribution rounds until `staked_maturity_e8s_equivalent` approaches `u64::MAX`.
4. On the next `distribute_rewards` call, `neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s` wraps to a value near zero.
5. Observe that the neuron's `staked_maturity_e8s_equivalent` is now near zero despite having accumulated rewards for many rounds — permanent, silent loss with no on-chain error or event emitted.

A deterministic unit test can reproduce this without a live deployment by directly calling `distribute_rewards` with a mock neuron whose `staked_maturity_e8s_equivalent` is pre-set to `u64::MAX - 1` and a `neuron_reward_e8s` of 2.
