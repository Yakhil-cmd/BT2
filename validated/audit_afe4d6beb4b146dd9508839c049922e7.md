### Title
FROST Signing Coordinator Permanently Blocked by Any Single Unresponsive Participant - (`src/frost/eddsa/sign.rs`, `src/protocol/helpers.rs`)

---

### Summary

The FROST signing coordinator unconditionally waits for **all** selected participants to respond at every round using `recv_from_others`. There is no timeout, no fallback, and no mechanism to proceed with a different subset. A single malicious or unresponsive participant included in a signing session can permanently halt signing for all honest parties.

---

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` loops until it has received exactly one message from every participant in the provided list:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

`seen.full()` returns `true` only when **every** participant has been marked. There is no timeout, no partial-quorum exit, and no error path for a missing sender. [1](#0-0) 

The FROST signing coordinator (`do_sign_coordinator_v1`) calls `recv_from_others` twice per signing session:

- **Round 1** (line 126): waits for a commitment from every participant before building the `SigningPackage`.
- **Round 2** (line 150): waits for a signature share from every participant before aggregating. [2](#0-1) [3](#0-2) 

The same pattern appears in `do_sign_coordinator_v2` (line 206): [4](#0-3) 

FROST is explicitly designed so that any `t`-of-`N` subset can produce a valid signature. The library's own stress tests confirm that signing sessions are run with `actual_signers > min_signers` (i.e., more participants than the threshold):

```rust
for actual_signers in min_signers..=max_signers {
    let data = run_sign_v1(&key_packages, actual_signers.into(), coordinator, min_signers, ...)
``` [5](#0-4) 

When `actual_signers > threshold`, the coordinator still waits for all `actual_signers` to respond. A single participant who stops responding after being selected causes the coordinator to block indefinitely at `chan.recv(waitpoint).await` inside `recv_from_others`, with no way to recover or restart with a different subset. [6](#0-5) 

The same structural issue exists in DKG (`do_keyshare`) at the commitment-hash exchange round and the private-share distribution round, both of which call `recv_from_others` and require all N participants to respond before proceeding. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A single malicious participant included in a signing session can refuse to send their Round 1 commitment or Round 2 signature share. The coordinator's `recv_from_others` loop never exits, permanently blocking signature production for all honest parties in that session. Because the library exposes no timeout, no partial-quorum path, and no session-restart API, the denial is indefinite from the library's perspective. This matches: **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

Any participant who is legitimately included in a signing session (a valid protocol input) can trigger this by simply going silent after the session starts. No cryptographic break, no key compromise, and no external dependency is required. The attacker only needs to be selected as one of the `actual_signers` for a session — a normal operational event. In deployments where `actual_signers > threshold`, the attacker does not even need to be one of the strictly necessary `t` parties.

---

### Recommendation

1. **Enforce `participants.len() == threshold` at signing time**, so the coordinator only waits for the minimum required set and any dropout is immediately fatal rather than silently blocking.
2. **Alternatively**, implement a timeout on `recv_from_others` (or on the outer protocol driver) so that a non-responding participant causes a recoverable error rather than an indefinite block, allowing the coordinator to restart with a different `t`-subset.
3. **Document** the liveness assumption explicitly: every participant passed to `sign_v1`/`sign_v2` must remain online for the duration of the session, and callers are responsible for selecting only participants known to be available.

---

### Proof of Concept

1. Set up a signing session with `N=5`, `threshold=3`. Pass all 5 participants to `sign_v1`.
2. Participant 4 sends its Round 1 commitment but then goes silent (does not send a Round 2 signature share).
3. The coordinator's second `recv_from_others` call at line 150 of `do_sign_coordinator_v1` loops forever waiting for participant 4's share.
4. The remaining 4 honest participants (including the 3 needed for a valid threshold signature) are permanently blocked from completing the signing session.
5. The `while !seen.full()` condition in `recv_from_others` never becomes true because participant 4's entry is never marked, and there is no timeout or error path to exit the loop. [9](#0-8) [3](#0-2)

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
