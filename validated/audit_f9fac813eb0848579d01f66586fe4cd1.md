### Title
Missing Zero-Address Validation for `erc20_fallback_address` Causes Permanent ERC-20 Token Loss in Silo Mode - (`engine/src/contract_methods/silo/mod.rs`)

### Summary

In Silo mode, the `erc20_fallback_address` parameter — which redirects incoming ERC-20 bridge tokens when the recipient is not whitelisted — can be set to `Address::zero()` without any validation. If this occurs (by operator mistake or intentional misconfiguration), every NEP-141 token bridged to a non-whitelisted EVM address is minted to the zero address and permanently frozen.

### Finding Description

`set_silo_params` and `set_erc20_fallback_address` are the two owner-callable entry points that write the fallback address to storage. Neither performs any check that the supplied address is non-zero.

In `engine/src/contract_methods/silo/mod.rs`, `set_erc20_fallback_address` writes whatever `Option<Address>` it receives directly to storage:

```rust
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes()); // no zero-address guard
    } else {
        io.remove_storage(&key);
    }
}
``` [1](#0-0) 

`set_silo_params` delegates to the same function without adding any guard: [2](#0-1) 

The on-chain entry point `set_erc20_fallback_address()` in `engine/src/lib.rs` also performs no validation before forwarding the caller-supplied address: [3](#0-2) 

When a NEP-141 token is bridged into Aurora via `ft_on_transfer`, `receive_erc20_tokens` checks whether the intended EVM recipient is whitelisted. If not, it silently replaces the recipient with the stored fallback address: [4](#0-3) 

If the stored fallback is `Address::zero()`, the ERC-20 mint call succeeds — tokens are credited to the zero address — and `ft_on_transfer` returns `"0"` (refund nothing), so the NEP-141 tokens are permanently retained by the Aurora contract while the ERC-20 balance is irrecoverably locked at the zero address.

The test suite itself demonstrates that `Address::zero()` is an accepted value for this field, with no runtime rejection: [5](#0-4) 

The `SiloParamsArgs` type definition confirms there is no type-level constraint preventing a zero address: [6](#0-5) 

### Impact Explanation

**Permanent freezing of funds (Critical).** Any user who bridges NEP-141 tokens to a non-whitelisted EVM address while the fallback is `Address::zero()` loses those tokens permanently. The NEP-141 balance is consumed by the Aurora contract, and the corresponding ERC-20 tokens are minted to an address for which no private key exists. There is no recovery path.

### Likelihood Explanation

The likelihood is low-to-medium. The Silo operator must call `set_silo_params` or `set_erc20_fallback_address` with a zero address. This can happen by mistake (e.g., passing a zero address as a placeholder during initial configuration, as the test suite itself does) or intentionally. The absence of any on-chain guard means neither path is blocked. The pattern is directly analogous to the RocketJoe finding: a privileged actor sets a critical economic parameter to zero, either in error or to exploit users.

### Recommendation

Add a zero-address guard in `set_erc20_fallback_address` (and correspondingly in `set_silo_params`) before writing to storage:

```rust
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        // Reject the zero address; it has no owner and would permanently
        // freeze any tokens redirected to it.
        assert!(!address.is_zero(), "ERR_ZERO_FALLBACK_ADDRESS");
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }
}
```

The same guard should be applied inside `set_silo_params` when `params.erc20_fallback_address` is extracted.

### Proof of Concept

1. Silo operator calls `set_silo_params` with `erc20_fallback_address = Address::zero()` and enables the `Address` whitelist.
2. A user calls `ft_on_transfer` on a NEP-141 token contract, specifying a non-whitelisted EVM address as the recipient.
3. `receive_erc20_tokens` detects the recipient is not whitelisted and replaces it with `Address::zero()`.
4. The ERC-20 `mint` call succeeds, crediting `Address::zero()` with the token amount.
5. `ft_on_transfer` returns `"0"` (no refund), so the NEP-141 tokens are permanently held by Aurora.
6. The user has lost their NEP-141 tokens; the ERC-20 tokens are frozen at the zero address with no recovery mechanism. [7](#0-6)
