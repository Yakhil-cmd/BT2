### Title
Admin Fee Destination Not Reset on Pool Admin Transfer, Causing Ongoing Diversion of Admin Fees to Previous Admin — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

When a pool admin transfer completes via `acceptPoolAdmin`, the `poolAdminFeeDestination[pool]` mapping is not cleared or reset. Because `collectPoolFees` is a permissionless function, the previous admin (or anyone) can immediately call it after the transfer, routing all accrued admin fees to the old admin's address. The new admin loses every wei of admin fees until they explicitly call `setPoolAdminFeeDestination`.

---

### Finding Description

`acceptPoolAdmin` performs a two-step admin handover: it updates `poolAdmin[pool]` to the pending address and deletes `pendingPoolAdmin[pool]`, but it does **not** touch `poolAdminFeeDestination[pool]`.

```solidity
// MetricOmmPoolFactory.sol:518-526
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
    // ← poolAdminFeeDestination[pool] is never touched
}
```

`collectPoolFees` is permissionless — no `onlyOwner` or `onlyPoolAdmin` guard:

```solidity
// MetricOmmPoolFactory.sol:379-389
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← still points to old admin's address
    );
}
```

The fee destination is set at pool creation to `params.adminFeeDestination` (typically the deploying admin's treasury):

```solidity
// MetricOmmPoolFactory.sol:220
poolAdminFeeDestination[pool] = params.adminFeeDestination;
```

After `acceptPoolAdmin` completes, `poolAdmin[pool]` is the new admin, but `poolAdminFeeDestination[pool]` still resolves to the old admin's address. Any call to `collectPoolFees` — by anyone, including the old admin — sends the admin share of accrued fees to the old admin.

---

### Impact Explanation

Admin fees are a direct revenue stream belonging to the pool admin role. After a legitimate admin transfer, every admin-fee collection event sends tokens to the wrong address (the previous admin) until the new admin explicitly calls `setPoolAdminFeeDestination`. Because `collectPoolFees` is permissionless, the old admin can call it immediately after `acceptPoolAdmin` to drain all accrued admin fees before the new admin can react. The new admin suffers a direct, quantifiable loss of token0 and token1 equal to the entire admin-fee balance accrued in the pool at the time of transfer, plus any fees that accrue before the destination is updated.

**Severity: Medium** — direct loss of protocol fees (admin share) owed to the new admin; no loss of user principal or LP assets, but the admin fee stream is fully diverted.

---

### Likelihood Explanation

Every pool admin transfer triggers this condition automatically. The old admin need not be malicious — even a benign old admin who calls `collectPoolFees` out of habit, or any third-party keeper that calls it, will route fees to the wrong address. A malicious old admin can front-run the new admin's `setPoolAdminFeeDestination` call to maximise extraction. The window is open from the moment `acceptPoolAdmin` is mined until the new admin's `setPoolAdminFeeDestination` is confirmed.

---

### Recommendation

Reset `poolAdminFeeDestination[pool]` inside `acceptPoolAdmin`. The simplest safe default is to set it to the incoming admin's address, or to `address(0)` with a guard in `collectPoolFees` that skips the admin leg when the destination is zero:

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    // Reset fee destination to the incoming admin so fees are not misdirected
    poolAdminFeeDestination[pool] = pending;
    emit PoolAdminFeeDestinationUpdated(pool, pending);
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

Alternatively, collect all outstanding fees to the old destination atomically inside `acceptPoolAdmin` before updating `poolAdmin`, then set the destination to the new admin.

---

### Proof of Concept

```
1. Factory deploys pool with adminFeeDestination = AdminA_treasury.
   poolAdminFeeDestination[pool] = AdminA_treasury  ✓

2. Pool accumulates admin fees over time (spread + notional components).

3. AdminA calls proposePoolAdminTransfer(pool, AdminB).
   pendingPoolAdmin[pool] = AdminB

4. AdminB calls acceptPoolAdmin(pool).
   poolAdmin[pool]          = AdminB       ← updated
   pendingPoolAdmin[pool]   = address(0)   ← cleared
   poolAdminFeeDestination[pool] = AdminA_treasury  ← NOT updated

5. AdminA (or any keeper) calls collectPoolFees(pool).
   → collectFees(..., poolAdminFeeDestination[pool])
   → admin share of token0/token1 transferred to AdminA_treasury
   AdminB receives nothing.

6. AdminB calls setPoolAdminFeeDestination(pool, AdminB_treasury).
   Only fees accrued AFTER this block go to AdminB.
   All fees accrued before step 6 (including the entire pre-transfer balance)
   are permanently lost to AdminB.
