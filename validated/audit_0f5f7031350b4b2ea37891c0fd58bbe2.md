### Title
Missing Caller Authorization in `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo::update_rewards`)

### Summary

The spec mandates "Only starkware sequencer" access control for `update_rewards`, but the implementation enforces no such check. Any unprivileged caller can invoke `update_rewards(staker_address, disable_rewards=true)` to consume the global `last_reward_block` gate for a block, permanently discarding that block's consensus rewards.

### Finding Description

`general_prerequisites()` is the only guard called at the entry of `update_rewards`: [1](#0-0) 

It only checks that the contract is unpaused and the caller is non-zero — no role or identity check: [2](#0-1) 

The spec explicitly states the required access control: [3](#0-2) 

But the implementation at the `StakingRewardsManagerImpl` entry point has no `only_sequencer` or equivalent assertion: [4](#0-3) 

`last_reward_block` is a **single global** storage slot (not per-staker): [5](#0-4) 

The write to `last_reward_block` happens **before** the `disable_rewards` early-return branch: [6](#0-5) 

Once `last_reward_block` equals the current block number, every subsequent call in that block reverts with `REWARDS_ALREADY_UPDATED`, regardless of caller or `disable_rewards` value.

### Impact Explanation

An attacker who calls `update_rewards(any_valid_staker, disable_rewards=true)` before the sequencer in any block:

1. Passes all validation (staker exists, is active, has non-zero STRK balance).
2. Writes `last_reward_block = current_block`.
3. Returns immediately — zero rewards distributed.
4. The sequencer's legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. That block's rewards are **permanently lost** — there is no catch-up or retroactive distribution mechanism.

Repeated every block, this permanently freezes all consensus reward accrual for all stakers and delegators. Impact: **High — Permanent freezing of unclaimed yield**.

### Likelihood Explanation

- Requires no privilege, no stake, no special role — only a valid `staker_address` of an active staker with effective balance (i.e., K epochs after staking).
- On Starknet, transaction ordering within a block is sequencer-controlled, but the attacker only needs to submit a transaction that lands in the same block before the sequencer's `update_rewards` call. Since `update_rewards` is a regular external call (not a sequencer-internal operation), this is achievable.
- Cost is one transaction per block with minimal gas.

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, consistent with the spec. For example, assert that `get_caller_address() == sequencer_address` where `sequencer_address` is a stored, governance-controlled value, or use an existing role such as `OPERATOR`.

### Proof of Concept

```
// Block N:
// Attacker (any address) calls:
staking.update_rewards(staker_address=<any_active_staker>, disable_rewards=true)
// → passes all checks, writes last_reward_block = N, returns early, no rewards distributed

// Sequencer then calls (same block N):
staking.update_rewards(staker_address=<intended_staker>, disable_rewards=false)
// → REVERTS: REWARDS_ALREADY_UPDATED (current_block_number == last_reward_block)

// Block N rewards are permanently lost.
// Repeat every block to suppress all consensus rewards indefinitely.
```

The BTC-wrapper delegation angle in the question is not necessary to trigger the bug — the vulnerability is in the missing caller check and the global `last_reward_block` gate, which affects all stakers uniformly.
