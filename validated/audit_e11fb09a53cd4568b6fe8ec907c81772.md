### Title
Gateway Stateful Admission Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` for Resource-Bound Threshold Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful resource-bound check compares a transaction's `max_price_per_unit` for L2 gas against the **current block's** `l2_gas_price`. The correct reference is `next_l2_gas_price` — the EIP-1559-adjusted price that will actually govern the next block in which the transaction executes. Under sustained high or low load the two values diverge, causing the gateway to admit transactions that will fail at batcher execution (or to reject transactions that would succeed).

---

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` reads the reference gas price from `GatewayFixedBlockStateReader::get_block_info()`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`get_block_info_from_sync_client` constructs a `BlockInfo` from `block_header.l2_gas_price.price_in_fri` and silently discards `block_header.next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`BlockHeaderWithoutHash` carries both fields as distinct values:

```rust
pub l2_gas_price: GasPricePerToken,
pub l2_gas_consumed: GasAmount,
pub next_l2_gas_price: GasPrice,
``` [3](#0-2) 

`next_l2_gas_price` is the EIP-1559 output of `calculate_next_base_gas_price` and is what the batcher will embed as the block's gas price when the transaction actually executes. [4](#0-3) 

The threshold check then compares the transaction's `max_price_per_unit` against a percentage of the **wrong** reference:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold { ... }
``` [5](#0-4) 

The `run_validate_entry_point` path also builds its `BlockContext` from the same stale `BlockInfo`, so the blockifier validation inside the gateway is equally misaligned with the batcher's actual execution price. [6](#0-5) 

---

### Impact Explanation

**High — gateway admission accepts invalid transactions / rejects valid transactions.**

Under EIP-1559, when blocks are consistently full, `next_l2_gas_price` rises above `l2_gas_price`. The gateway's threshold is anchored to the lower stale value, so transactions whose `max_price_per_unit` clears the gateway check but falls below `next_l2_gas_price` are admitted to the mempool and forwarded to the batcher, where they fail the blockifier's `check_fee_bounds` pre-validation. The reverse holds when blocks are consistently empty: `next_l2_gas_price` falls below `l2_gas_price`, and the gateway rejects transactions that the batcher would accept.

---

### Likelihood Explanation

The EIP-1559 L2 gas price mechanism is always active. Any sustained period of above- or below-target block utilisation causes `next_l2_gas_price` to diverge from `l2_gas_price`. The divergence is bounded per block by `gas_price_max_change_denominator` but compounds across consecutive blocks. The condition is therefore reachable by any user submitting a transaction during normal network operation; no privileged access is required.

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a separate accessor) to expose `next_l2_gas_price` from `BlockHeaderWithoutHash`.
2. In `validate_resource_bounds`, replace `previous_block_l2_gas_price` with the `next_l2_gas_price` value from the block header, resolving the TODO comment.
3. In `run_validate_entry_point`, set `block_info.gas_prices.strk_gas_prices.l2_gas_price` to `next_l2_gas_price` so the blockifier validation inside the gateway is consistent with the batcher's execution context.

---

### Proof of Concept

1. Network is under sustained high load; blocks are full for several consecutive heights.
2. EIP-1559 causes `next_l2_gas_price` to be, say, 20 % above `l2_gas_price` of the latest committed block.
3. `min_gas_price_percentage` is configured at 50 (default-like value).
4. Attacker (or ordinary user) submits an invoke V3 transaction with `l2_gas.max_price_per_unit = 60 % × l2_gas_price`.
5. Gateway threshold: `60 % × l2_gas_price ≥ 50 % × l2_gas_price` → **PASSES** admission.
6. Batcher builds the next block with `l2_gas_price = next_l2_gas_price = 120 % × l2_gas_price`.
7. Blockifier `check_fee_bounds`: `60 % × l2_gas_price < 120 % × l2_gas_price` → **FAILS** with `InsufficientResourceBounds`.
8. The transaction was admitted to the mempool and consumed gateway/mempool resources but cannot be included in any block.
