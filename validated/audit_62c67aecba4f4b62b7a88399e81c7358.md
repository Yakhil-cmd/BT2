### Title
`native_price` and `pubdata_price` Oracle Values Not Committed in ZK Public Inputs, Enabling Prover-Controlled Resource Accounting Manipulation - (`basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/mod.rs`)

---

### Summary

The ZKsync OS oracle provides `native_price` and `pubdata_price` as part of `BlockMetadataFromOracle` via `BLOCK_METADATA_QUERY_ID`. These values directly control how much users pay for gas and pubdata. However, neither value is included in the block header, `ChainStateCommitment`, `BatchOutput`, nor `BatchPublicInput`. A malicious prover/sequencer can supply arbitrary values for these fields and the ZK proof will still verify, because the proof never commits to them. This allows the prover to silently manipulate the economic parameters of every transaction in a block.

---

### Finding Description

**Step 1 — Oracle provides `native_price` and `pubdata_price` at block start.**

`ZkMetadata::metadata_op()` queries the oracle for `BlockMetadataFromOracle`, which includes `native_price` and `pubdata_price`: [1](#0-0) 

These fields are deserialized from the oracle response with no constraint on their values: [2](#0-1) 

**Step 2 — `native_price` and `pubdata_price` directly control fee accounting for every transaction.**

In `validate_and_compute_fee_for_transaction()`, both values are read and used to compute `native_per_gas` and `native_per_pubdata`, which determine how many native resources a user must pre-pay: [3](#0-2) 

**Step 3 — Neither value is included in the block header.**

`form_block_header()` explicitly omits both fields with a TODO comment: [4](#0-3) 

The documentation confirms this is a known gap: [5](#0-4) 

**Step 4 — Neither value appears in any ZK public input structure.**

`ChainStateCommitment` commits only to `state_root`, `next_free_slot`, `block_number`, `last_256_block_hashes_blake`, and `last_block_timestamp`: [6](#0-5) 

`BatchOutput` commits to `chain_id`, timestamps, DA commitment, L1 tx counts, logs root, and upgrade hash — but not `native_price` or `pubdata_price`: [7](#0-6) 

`BatchPublicInput` is a hash of `state_before`, `state_after`, and `batch_output` — none of which contain the pricing parameters: [8](#0-7) 

**Step 5 — The block header hash feeds into `ChainStateCommitment` but does not carry the missing fields.**

The block header is hashed and included in `last_256_block_hashes_blake` of `ChainStateCommitment`, but since `native_price` and `pubdata_price` are absent from the header, they remain uncommitted even through this indirect path: [9](#0-8) 

---

### Impact Explanation

A malicious prover/sequencer can supply any `native_price` and `pubdata_price` values to the oracle. Because these values are never committed in the ZK proof's public inputs, the proof verifies regardless of what values were used during execution. This enables:

1. **Setting `pubdata_price` to 0**: All transactions in the block pay zero for pubdata. The operator/protocol bears L1 data availability costs without compensation — a direct loss of funds for the protocol.

2. **Setting `pubdata_price` to an arbitrarily high value**: `native_per_pubdata` becomes inflated, causing users to be overcharged for pubdata. The excess fees flow to the operator (`coinbase`) — theft from users.

3. **Setting `native_price` to an extreme value**: Manipulates `native_per_gas = gas_price / native_price`, either causing all transactions to fail with `NativeResourcesAreTooExpensive` (DoS) or allowing transactions to consume far more native resources than they paid for (resource accounting bypass).

The state transition proof remains valid in all cases because the settlement layer has no way to verify what `native_price` or `pubdata_price` were used.

---

### Likelihood Explanation

The attacker is the prover/sequencer, which is listed as a valid attacker in the scope ("prover/forward execution input"). The oracle response for `BLOCK_METADATA_QUERY_ID` is fully controlled by whoever constructs the oracle input. No special key or governance access is required beyond operating the prover. The attack is silent — the resulting proof is indistinguishable from a legitimate one on the settlement layer.

---

### Recommendation

Include `native_price` and `pubdata_price` in the block header's `extra_data` field (as already noted in the documentation as the intended fix), and ensure the block header hash — which feeds into `ChainStateCommitment` via `last_256_block_hashes_blake` — therefore commits to these values. Alternatively, add them directly to `BatchOutput` so the settlement layer can validate them. The `form_block_header()` function in `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/mod.rs` is the correct place to implement this fix.

---

### Proof of Concept

1. Construct a `BlockMetadataFromOracle` oracle response with `pubdata_price = U256::ZERO` and `native_price = U256::from(1)`.
2. Run the proving system with this oracle. All transactions in the block will have `native_per_pubdata = 0`, meaning no native resources are charged for pubdata.
3. The resulting `BatchPublicInput` hash is identical in structure to a legitimate block — `native_price` and `pubdata_price` do not appear in `ChainStateCommitment`, `BatchOutput`, or `BatchPublicInput`.
4. The ZK proof verifies on the settlement layer. The operator has processed a block where all pubdata costs were borne by the protocol, with no on-chain evidence of manipulation.

The existing test infrastructure confirms the pricing parameters are fully operator-controlled per block: [10](#0-9)

pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }

pub struct BatchOutput {
    /// Chain id used during execution of the blocks.
    pub chain_id: U256,
    /// First block timestamp.
    pub first_block_timestamp: u64,
    /// Last block timestamp.
    pub last_block_timestamp: u64,
    /// DA commitment scheme.
    pub da_commitment_scheme: DACommitmentScheme,
    /// Pubdata commitment.
    pub pubdata_commitment: Bytes32,
    /// Number of l1 -> l2 processed txs in the batch.
    pub number_of_layer_1_txs: U256,
    /// Number of processed L2 txs in the batch.
    pub number_of_layer_2_txs: U256,
    /// Rolling keccak256 hash of l1 -> l2 txs processed in the batch.
    pub priority_operations_hash: Bytes32,
    /// L2 logs tree root.
    /// Note that it's full root, it's keccak256 of:
    /// - merkle root of l2 -> l1 logs in the batch .
    /// - multichain root - commitment to logs emitted on chains that settle on the current.
    pub l2_logs_tree_root: Bytes32,
    /// Protocol upgrade tx hash (0 if there wasn't)
    pub upgrade_tx_hash: Bytes32,
    /// Linear keccak256 hash of interop roots
    pub interop_roots_rolling_hash: Bytes32,
    /// Settlement layer chain id.
    pub settlement_layer_chain_id: U256,
}

impl BatchOutput {
    ///
    /// Calculate keccak256 hash of public input
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = Keccak256::new();
        hasher.update(self.chain_id.to_be_bytes::<32>());
        hasher.update(&self.first_block_timestamp.to_be_bytes());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        // Encode DA commitment scheme as U256 BE
        hasher.update([0u8; 31]);
        hasher.update([self.da_commitment_scheme as u8]);
        hasher.update(self.pubdata_commitment.as_u8_ref());
        hasher.update(self.number_of_layer_1_txs.to_be_bytes::<32>());
        hasher.update(self.number_of_layer_2_txs.to_be_bytes::<32>());
        hasher.update(self.priority_operations_hash.as_u8_ref());
        hasher.update(self.l2_logs_tree_root.as_u8_ref());
        hasher.update(self.upgrade_tx_hash.as_u8_ref());
        hasher.update(self.interop_roots_rolling_hash.as_u8_ref());
        hasher.update(self.settlement_layer_chain_id.to_be_bytes::<32>());
        hasher.finalize()
    }

        hasher.update(self.state_before.as_u8_ref());
        hasher.update(self.state_after.as_u8_ref());
        hasher.update(self.batch_output.as_u8_ref());
        hasher.finalize()
