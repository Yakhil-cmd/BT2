### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Block Reward Distribution - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the staking contract is documented to be callable "Only starkware sequencer" but has no such enforcement in code. Any unprivileged address can call it with `disable_rewards: true`, which advances the global `last_reward_block` without distributing any rewards, permanently blocking the sequencer from distributing rewards for that block.

### Finding Description
The spec at `docs/spec.md` line 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` only calls `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — it does **not** verify the caller is the authorized sequencer address.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();  // only: unpaused + non-zero caller
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards — happens BEFORE the rewards check.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;  // exits without distributing any rewards
    }
    ...
}
```

The critical sequence is:
1. `last_reward_block` is written to `current_block_number` unconditionally (line 1485).
2. If `disable_rewards == true`, the function returns immediately without distributing rewards (line 1487–1489).

An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` before the sequencer in a given block will:
- Advance `last_reward_block` to the current block.
- Distribute zero rewards.
- Cause the sequencer's subsequent call to revert with `REWARDS_ALREADY_UPDATED`.

Because `last_reward_block` is a **global** (not per-staker) variable, a single attacker call blocks reward distribution for **all stakers** in that block. Rewards for missed blocks are permanently lost — there is no retroactive recovery mechanism.

### Impact Explanation
This matches **"High: Permanent freezing of unclaimed yield"**. If an attacker calls `update_rewards(any_valid_staker, disable_rewards: true)` consistently across blocks, all stakers permanently lose their block rewards. The `unclaimed_rewards_own` field in `InternalStakerInfoLatest` and the pool `cumulative_rewards_trace` are never updated for those blocks, and there is no mechanism to retroactively credit missed blocks.

### Likelihood Explanation
On Starknet, the sequencer controls transaction ordering, which limits classical front-running. However:
- The sequencer is not required to call `update_rewards` in every block (the design allows missed blocks, as shown in `test_update_rewards_miss_blocks_flow_test`).
- Any block in which the sequencer does not call `update_rewards` first is exploitable.
- The attacker only needs one valid staker address (public on-chain information) and no special privileges.
- The attack is cheap (one transaction per block) and requires no capital.

### Recommendation
Add an explicit sequencer address check inside `update_rewards`, analogous to how `update_rewards_from_attestation_contract` enforces `assert_caller_is_attestation_contract()`. Store the authorized sequencer address in contract storage and assert it at the top of `update_rewards`.

### Proof of Concept
1. Attacker reads any active staker address from on-chain state.
2. At the start of a block (before the sequencer's `update_rewards` call), attacker submits:
   ```
   staking.update_rewards(staker_address: known_staker, disable_rewards: true)
   ```
3. `last_reward_block` is set to the current block number; no rewards are distributed.
4. The sequencer's call to `update_rewards` reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers lose their block rewards for that block permanently.
6. Repeating this every block causes total, permanent freezing of all staker yield.
