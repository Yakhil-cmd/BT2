### Title
Type Confusion in `assetWalletId` Bypasses Offer Validation Guard, Silently Dropping Rows from Created Offer — (`packages/gui/src/components/offers/OfferEditorConditionsPanel.tsx`, `OfferEditor.tsx`)

### Summary
When a user adds a second row to either the maker or taker side of the offer editor, `useFieldArray.append` is called with `assetWalletId: ''` (empty string). The validation guard in `onSubmit` uses strict equality (`row.assetWalletId === 0`), which evaluates to `false` for `''`, so `missingAssetSelection` is never set. `getUpdatedOffer` then silently drops the row because `'' > 0` is `false`. The offer is submitted to `createOfferForIds` with fewer assets than the user sees on screen.

### Finding Description
`OfferRowData` declares `assetWalletId: number` with `0` as the sentinel for "no selection made." `defaultMakerRow` correctly initializes with numeric `0`. However, `OfferEditorConditionsPanel` appends new rows with `assetWalletId: ''` (string). This creates a type mismatch at runtime:

- `'' === 0` → `false` (strict equality, guard not triggered)
- `'' > 0` → `false` (JS coerces `''` to `0`, row silently dropped)

The result: a row is visible in the UI but absent from the offer sent to the wallet daemon.

### Impact Explanation
If the dropped row is on the **taker side** (what the user wants to receive), the offer is created requesting fewer assets than intended. If a counterparty accepts this offer, the user surrenders their maker assets and receives less than they expected — a direct financial loss. No external attacker is required beyond the user's normal offer creation workflow; the bug is triggered by the standard "add row" button.

### Likelihood Explanation
Any user who clicks "add row" on either side, leaves the new row's asset selector blank, and clicks "Create Offer" triggers this. The UI provides no error, no warning, and no confirmation dialog before calling `createOfferForIds`. The offer file is created and can be immediately shared.

### Recommendation
1. Change both `takerAppend` and `makerAppend` calls in `OfferEditorConditionsPanel.tsx` to use numeric `0` instead of `''`:
   ```ts
   makerAppend({ amount: '', assetWalletId: 0, walletType: WalletType.STANDARD_WALLET })
   ```
2. Change the validation guard in `OfferEditor.onSubmit` to use loose equality or an explicit type-safe check:
   ```ts
   if (!row.assetWalletId || row.assetWalletId === 0) { missingAssetSelection = true; }
   ```
3. Add a TypeScript strict-mode check or a runtime assertion in `getUpdatedOffer` to reject non-numeric `assetWalletId` values rather than silently dropping the row.

### Proof of Concept
1. Open the offer editor with at least two wallets available.
2. Click the "+" button on the maker side to add a second row.
3. Do **not** select an asset for the new row.
4. Enter an amount for the first maker row and both taker rows.
5. Click "Create Offer."
6. Observe: no validation error is shown; the offer is created.
7. Inspect the created offer file: it contains only the first maker row's asset, not the second (ghost) row.
8. Unit test: call `onSubmit` with `makerRows: [{ assetWalletId: 1, amount: '1', ... }, { assetWalletId: '', amount: '', ... }]` and assert `missingAssetSelection === true` — the assertion fails, confirming the bypass.

function getUpdatedOffer(offerParam: { [key: string]: BigNumber }, row: OfferEditorRowData, debit: boolean) {
  const offer = JSON.parse(JSON.stringify(offerParam));
  const { amount, assetWalletId, walletType: walletTypeLocal } = row;
  if (assetWalletId > 0) {
    let mojoAmount = new BigNumber(0);
    if (walletTypeLocal === WalletType.STANDARD_WALLET) {
      mojoAmount = chiaToMojo(amount);
    } else if ([WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(walletTypeLocal)) {
      mojoAmount = catToMojo(amount);
    }

    offer[assetWalletId] = debit ? mojoAmount.negated() : mojoAmount;
  } else {
    console.error('missing asset wallet id');
  }
  return offer;
