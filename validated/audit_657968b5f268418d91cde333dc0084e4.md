### Title
Burn-Before-Precompile With Unchecked Return Value Enables Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.withdrawToNear` burns the caller's ERC-20 tokens **before** invoking the `ExitToNear` precompile, and the inline-assembly call **never checks the return value `res`**. When the precompile rejects the call (e.g., because `input_size > MAX_INPUT_SIZE`), the burn is committed but no NEAR promise is emitted, permanently destroying the tokens.

---

### Finding Description

**Step 1 — Burn happens unconditionally first.**

In `EvmErc20V2.withdrawToNear` (lines 54-63):

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // ← burn is irreversible

    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;  // = 53 + recipient.length

    assembly {
        let res := call(gas(), 0xe9217bc7..., 0, add(input, 32), input_size, 0, 32)
        // res is NEVER read or checked
    }
}
``` [1](#0-0) 

**Step 2 — Precompile enforces `MAX_INPUT_SIZE = 1024`.**

In `native.rs`:

```rust
const MAX_INPUT_SIZE: usize = 1_024;

fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {
    if input.len() < min || input.len() > max {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_INPUT")));
    }
    Ok(())
}
``` [2](#0-1) [3](#0-2) 

When `recipient.length > 971`, `input_size = 53 + recipient.length > 1024`. The precompile's `try_from` parsing calls `validate_input_size` and returns `Err(ExitError::Other("ERR_INVALID_INPUT"))`. The EVM translates this into a failed sub-call (return value `0`).

**Step 3 — `res = 0` is silently ignored.**

The assembly block captures `res` but never branches on it. The outer Solidity function returns normally. No revert, no promise log, no exit event. [4](#0-3) 

**Step 4 — No promise → no NEP-141 transfer.**

A successful precompile call emits a `promise_log` (a NEAR cross-contract call to `ft_transfer`) as an EVM log. With the precompile failing, that log is never produced, so the NEAR runtime never releases the corresponding NEP-141 tokens to the recipient. [5](#0-4) 

**The `error_refund` feature does not help here.** The refund callback is only constructed and emitted when the precompile call itself *succeeds* (returns `Ok`). A precompile-level rejection before any promise is built means there is nothing to refund. [6](#0-5) 

---

### Impact Explanation

- ERC-20 `totalSupply` decreases (tokens burned).
- The NEP-141 tokens that back those ERC-20 tokens remain locked in the Aurora connector contract forever.
- The bridge becomes insolvent: fewer ERC-20 tokens exist than NEP-141 tokens backing them, but the "missing" NEP-141 tokens are permanently inaccessible.
- **Impact: Critical — Insolvency.**

---

### Likelihood Explanation

Any unprivileged EVM account that holds a balance of a registered ERC-20 mirror token can trigger this by calling `withdrawToNear` with a `recipient` bytes argument of length ≥ 972. No special role, no admin key, no governance action is required. The call is a standard EVM transaction. The same structural bug also fires for any other precompile rejection (invalid UTF-8 recipient, invalid NEAR account-ID format, etc.), widening the attack surface further.

---

### Recommendation

1. **Check `res` and revert on failure:**
   ```solidity
   assembly {
       let res := call(...)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Alternatively, validate input size before burning:**
   Add a Solidity-level guard `require(53 + recipient.length <= 1024, "recipient too long")` before `_burn`.
3. **Preferred fix — burn after precompile succeeds:** restructure so the burn is the last operation, or use a two-phase commit (lock → precompile → burn on callback success).

Note: `EvmErc20.withdrawToNear` (the V1 contract) has the same pattern but with `input_size = 1 + 32 + recipient.length`, so the threshold is `recipient.length > 991`. It carries the same bug. [7](#0-6)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Preconditions:
//   - `token` is a deployed EvmErc20V2 registered with the Aurora engine
//   - `attacker` holds `amount` tokens
//   - Aurora engine is running locally (e.g., aurora-standalone or workspaces)

function poc(EvmErc20V2 token, uint256 amount) external {
    // Craft a recipient longer than 971 bytes (e.g., 972 'a' bytes)
    bytes memory oversizedRecipient = new bytes(972);
    for (uint i = 0; i < 972; i++) oversizedRecipient[i] = 0x61; // 'a'

    uint256 supplyBefore = token.totalSupply();

    // This call burns `amount` tokens, then the precompile rejects the
    // oversized input and returns 0, which is silently ignored.
    token.withdrawToNear(oversizedRecipient, amount);

    uint256 supplyAfter = token.totalSupply();

    // Assert: supply decreased (tokens burned) ...
    assert(supplyAfter == supplyBefore - amount);
    // ... but no ExitToNear promise log was emitted and no NEP-141
    // transfer occurred on the NEAR side.
    // The tokens are permanently destroyed.
}
```

**Expected result:** `totalSupply` drops by `amount`; no `ExitToNear` event is emitted; the NEAR-side NEP-141 balance of the recipient is unchanged; the connector's locked balance is unchanged. The discrepancy is permanent.
