### Title
Unsigned `coinbase` Parameter in `EVM.run()` Enables Gas Fee Redirection by Unprivileged Cadence Sender — (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.run(tx: [UInt8], coinbase: EVMAddress)` and `EVM.batchRun(txs: [[UInt8]], coinbase: EVMAddress)` are `access(all)` Cadence functions that accept a `coinbase` (gas fee collector) address as a caller-supplied parameter. The `coinbase` is **not part of the signed EVM transaction** and is **not authenticated** in any way. Any unprivileged Cadence transaction sender who possesses a valid signed EVM transaction can submit it with an arbitrary `coinbase`, redirecting all EVM gas fees to an attacker-controlled address. This is the direct Flow analog of the unsigned `callbackData` vulnerability: a parameter that materially affects the financial outcome of a cross-party operation is left unsigned and freely replaceable by the submitter.

---

### Finding Description

**Root cause — unsigned, unauthenticated `coinbase` parameter:**

`EVM.run()` is declared `access(all)` with no restriction on who may call it or what `coinbase` value they may supply:

```cadence
// fvm/evm/stdlib/contract.cdc
access(all)
fun run(tx: [UInt8], coinbase: EVMAddress): Result {
    pre { !self.isPaused(): "EVM operations are temporarily paused" }
    return InternalEVM.run(tx: tx, coinbase: coinbase.bytes) as! Result
}
``` [1](#0-0) 

The same pattern applies to `batchRun`:

```cadence
access(all)
fun batchRun(txs: [[UInt8]], coinbase: EVMAddress): [Result] { ... }
``` [2](#0-1) 

The Go implementation passes the caller-supplied `coinbase` bytes directly to the handler without any validation:

```go
// fvm/evm/impl/impl.go
gasFeeCollector, err := interpreter.ByteArrayValueToByteSlice(context, gasFeeCollectorValue)
// ...
result := handler.Run(transaction, types.NewAddressFromBytes(gasFeeCollector))
``` [3](#0-2) 

The handler then transfers the entire gas fee balance accumulated at the internal `CoinbaseAddress` to the caller-supplied `gasFeeCollector`:

```go
// fvm/evm/handler/handler.go
func (h *ContractHandler) runWithGasFeeRefund(gasFeeCollector types.Address, f func()) {
    cb := h.AccountByAddress(types.CoinbaseAddress, true)
    initCoinbaseBalance := cb.Balance()
    f()
    afterBalance := cb.Balance()
    diff := new(big.Int).Sub(afterBalance, initCoinbaseBalance)
    if diff.Sign() > 0 {
        cb.Transfer(gasFeeCollector, diff)
    }
``` [4](#0-3) 

**What the EVM transaction signer actually signs:**

A standard RLP-encoded EVM transaction covers: `{nonce, gasPrice, gasLimit, to, value, data, chainId, r, s, v}`. The `coinbase` is a **block-level parameter** in standard EVM and is therefore **absent from the EVM transaction signature**. There is no mechanism for the EVM transaction signer to commit to a specific `coinbase` address.

**Attack path — malicious relayer (no front-running required):**

1. Alice signs an EVM transaction with `gasPrice > 0` and hands the RLP-encoded bytes to Bob (a relayer) with an off-chain agreement that gas fees go to `agreedAddress`.
2. Bob (malicious relayer) submits the Cadence transaction:
   ```cadence
   EVM.run(tx: aliceTx, coinbase: EVM.EVMAddress(bytes: bobAddress))
   ```
   instead of the agreed `agreedAddress`.
3. Alice's EVM transaction executes normally; all gas fees (`gasUsed × gasPrice` in attoFLOW) are transferred to `bobAddress` instead of `agreedAddress`.
4. Alice's EVM account is debited the correct amount; the theft is invisible to her at the EVM level.

**Attack path — front-running (requires mempool/off-chain access):**

If Alice submits her own EVM transaction via a Cadence transaction, an attacker who obtains the signed EVM bytes (e.g., from an off-chain relayer API or a shared channel) can submit `EVM.run(tx: aliceTx, coinbase: attackerAddress)` first. Alice's EVM nonce is consumed; her Cadence transaction fails; the gas fees go to the attacker.

---

### Impact Explanation

**Classification:** Cross-VM asset loss.

EVM gas fees are FLOW tokens held inside the EVM environment. The `runWithGasFeeRefund` mechanism unconditionally transfers the entire fee balance to the caller-supplied `gasFeeCollector`. An attacker who controls the Cadence transaction submission can redirect these fees to any EVM address. The EVM transaction signer has no on-chain mechanism to enforce that the `coinbase` is set to a specific address, because the `coinbase` is outside the scope of the EVM transaction signature.

The financial impact scales with `gasUsed × gasPrice`. For high-value EVM transactions (e.g., DeFi interactions, contract deployments) with non-zero gas prices, the redirected fees can be significant.

---

### Likelihood Explanation

**Medium.** The malicious relayer scenario requires no special capability beyond being the party who submits the Cadence transaction wrapping the EVM transaction — a role that is common in EVM relayer architectures. The front-running scenario additionally requires obtaining the signed EVM bytes before the legitimate submitter, which is feasible when EVM transactions are shared off-chain (e.g., via relayer APIs, gasless transaction services, or meta-transaction infrastructure). The `EVM.run()` function is `access(all)` with no access control on the `coinbase` parameter, making exploitation straightforward for any party who holds the signed EVM transaction bytes.

---

### Recommendation

1. **Enforce coinbase via EVM transaction data:** Allow EVM transaction senders to embed the intended `coinbase` address in the EVM transaction's `data` field or as a typed extension, and validate it on-chain before executing.
2. **Restrict `EVM.run()` to authenticated callers:** Require the Cadence transaction's authorizer to match a registered relayer or the EVM transaction sender's COA, so that the `coinbase` is implicitly authenticated by the Cadence transaction signature.
3. **Emit the `coinbase` in the `TransactionExecuted` event:** At minimum, make the `coinbase` address visible in the event log so that EVM transaction senders can detect fee redirection after the fact.

---

### Proof of Concept

Any unprivileged Cadence transaction sender can execute the following to redirect gas fees from a signed EVM transaction `aliceTxBytes` to `attackerEVMAddress`:

```cadence
import EVM from <EVMContractAddress>

transaction(aliceTxBytes: [UInt8], attackerBytes: [UInt8; 20]) {
    execute {
        let attackerCoinbase = EVM.EVMAddress(bytes: attackerBytes)
        // coinbase is not part of aliceTxBytes signature — freely replaceable
        let result = EVM.run(tx: aliceTxBytes, coinbase: attackerCoinbase)
        // All gas fees from Alice's EVM transaction now go to attackerCoinbase
    }
}
```

The `coinbase` parameter flows through `InternalEVM.run` → `ContractHandler.Run` → `runWithGasFeeRefund`, which unconditionally transfers the accumulated fee balance at `types.CoinbaseAddress` to the attacker-supplied address. [5](#0-4) [6](#0-5)

func (h *ContractHandler) runWithGasFeeRefund(gasFeeCollector types.Address, f func()) {
	// capture coinbase init balance
	cb := h.AccountByAddress(types.CoinbaseAddress, true)
	initCoinbaseBalance := cb.Balance()

	afterBalance := cb.Balance()
