### Title
Silent Promise Failure Causes Permanent Fund Loss in `ExitToNear` Precompile Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary

When the Aurora Engine is compiled without the `error_refund` feature flag, the `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) from a user's EVM balance and schedules a NEAR `ft_transfer` promise with **no failure callback**. If the NEAR promise fails for any reason (e.g., recipient account not registered with the NEP-141 contract), the burned tokens are permanently lost with no refund path. This is a direct structural analog to the BentoBox/Aave `_exit()` try-catch silent swallow: the failure of the external call is silently ignored and the accounting deficit is borne by the user.

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` precompile constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← hardcoded None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `refund` is `None` and `transfer_near` is also `None` (the common ERC-20 exit case), `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the promise is created **without any callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund on failure
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `ft_transfer` NEAR promise can fail for multiple reasons reachable by any unprivileged user:
- The recipient NEAR account is not registered with the NEP-141 contract.
- The NEP-141 contract is paused.
- Insufficient attached gas causes the cross-contract call to fail.

When `error_refund` is enabled, the callback `exit_to_near_precompile_callback` fires on failure and calls `engine::refund_on_error` to re-mint burned ERC-20 tokens or return ETH: [3](#0-2) 

Without `error_refund`, this entire recovery path is absent. The integration tests explicitly document the resulting fund loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

The same loss applies to ETH exits:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of user funds.**

When a user calls `withdrawToNear` on an ERC-20 mirror token (or sends ETH to the `ExitToNear` precompile address), the EVM-side balance is irrevocably reduced before the NEAR promise is dispatched. If the promise fails and `error_refund` is not compiled in, the tokens are permanently destroyed with no on-chain recovery path. The NEP-141 balance on the NEAR side is also unchanged (the `ft_transfer` failed), so the tokens exist nowhere — they are simply gone.

### Likelihood Explanation

**Medium.** The `error_refund` feature is an opt-in compile-time flag, not a default. Any production binary compiled without it is vulnerable. The failure condition (`ft_transfer` failing) is reachable by any user who specifies an unregistered recipient account, which requires no special privilege. The `ExitToNear` precompile is a core, publicly accessible bridge interface callable by any EVM transaction.

### Recommendation

Make `error_refund` a default feature in `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, or remove the conditional compilation entirely and always schedule the failure callback. The callback path in `exit_to_near_precompile_callback` already handles both the success and failure cases correctly. [6](#0-5) 

### Proof of Concept

1. Deploy Aurora Engine compiled **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora, receiving ERC-20 mirror tokens.
3. Call `withdrawToNear("unregistered.near", amount)` on the ERC-20 contract. This burns the ERC-20 tokens and schedules `ft_transfer` to `unregistered.near`.
4. The `ft_transfer` NEAR promise fails because `unregistered.near` is not registered with the NEP-141 contract.
5. No callback fires. No refund is issued.
6. Observe: ERC-20 balance reduced by `amount`, NEP-141 balance on NEAR side unchanged, tokens permanently lost.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–666) reproduces this exact scenario and confirms the balance discrepancy under `#[cfg(not(feature = "error_refund"))]`. [7](#0-6)

```rust
#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
