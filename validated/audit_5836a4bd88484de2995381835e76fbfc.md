### Title
Missing Threshold Validation in CKD Protocol Allows Corrupted Key Derivation Output - (`src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function validates only that `participants.len() >= 2`, but never validates that `participants.len() >= threshold` (the DKG reconstruction threshold). Because the threshold is neither stored in `KeygenOutput` nor accepted as a parameter by `ckd()`, the protocol silently proceeds with an under-sized participant set, producing a cryptographically incorrect derived key that is accepted as valid output.

### Finding Description
The external M-01 report describes a pattern where two sub-requirements (overhead and intrinsic cost) are each checked separately against the total, so a transaction can pass validation with `max(overhead, intrinsic)` instead of the required `overhead + intrinsic`. The analog here is that the CKD entry-point checks participant count against the wrong lower bound (`2` instead of `threshold`), allowing the protocol to proceed when the participant set is too small to correctly reconstruct the secret.

`ckd()` performs these initialization checks:

```rust
if participants.len() < 2 {
    return Err(InitializationError::NotEnoughParticipants { ... });
}
``` [1](#0-0) 

It then checks that `me` and `coordinator` are present, but **never** checks `participants.len() >= threshold`. [2](#0-1) 

Inside `compute_signature_share`, each participant computes a Lagrange coefficient over the provided participant set and uses it to normalize their secret contribution:

```rust
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [3](#0-2) 

Lagrange interpolation over a (t, n) secret sharing scheme requires at least `t` evaluation points to reconstruct the degree-(t-1) polynomial. When `participants.len() < t`, the interpolation silently evaluates a lower-degree polynomial, producing a wrong aggregate secret. The coordinator then sums all contributions and returns this incorrect value as the CKD output with no error:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [4](#0-3) 

The root cause is structural: `KeygenOutput` stores only `private_share` and `public_key` — the threshold used during DKG is not preserved — so `ckd()` has no way to enforce the correct lower bound even if it tried. [5](#0-4) 

Compare with other protocols in the library that do enforce the threshold. For example, `assert_key_invariants` in `src/dkg.rs` explicitly checks `threshold > participants.len()`: [6](#0-5) 

And `assert_sign_inputs` in `src/frost/mod.rs` checks `threshold.value() > participants.len()`: [7](#0-6) 

No equivalent guard exists in `ckd()`.

### Impact Explanation
When `ckd()` is called with `k < t` participants, the Lagrange interpolation reconstructs a wrong polynomial, and the coordinator returns a deterministically wrong `CKDOutput`. The calling application calls `ckd_output.unmask(app_sk)` and receives an incorrect BLS point as its derived secret — a value that is neither the intended `msk · H(pk ∥ app_id)` nor random noise, but a predictable function of the under-sized participant set and their shares. Honest parties accept this corrupted output without any protocol-level error, mapping directly to the allowed impact: **Corruption of CKD outputs so honest parties accept unusable or inconsistent cryptographic outputs**.

### Likelihood Explanation
The `ckd()` function is a public library API. Any unprivileged caller who assembles the participant list (e.g., an application layer that loses contact with some nodes and retries with a smaller set, or a malicious coordinator who selects the participant list) can trigger this path. Because no error is returned, the caller has no signal that the output is wrong. The threshold is not documented as a required constraint on the `participants` argument in the function signature or its doc-comment.

### Recommendation
1. **Store the threshold in `KeygenOutput`**: Add a `threshold: ReconstructionLowerBound` field so that `ckd()` can enforce `participants.len() >= threshold` at initialization time, mirroring the pattern used in `assert_key_invariants` and `assert_sign_inputs`.
2. **Alternatively, add a `threshold` parameter to `ckd()`** and validate it before proceeding, returning `InitializationError::NotEnoughParticipantsForThreshold` when the check fails.

### Proof of Concept
```
// DKG: threshold = 3, participants = [P1, P2, P3, P4, P5]
let keys = keygen(&[P1,P2,P3,P4,P5], threshold=3, ...);

// CKD called with only 2 participants — passes the `>= 2` check
let result = ckd(
    &[P1, P2],          // len=2 < threshold=3, no error returned
    coordinator=P1,
    me=P1,
    keys[P1].clone(),
    app_id,
    app_pk,
    rng,
);
// result is Ok(...) — protocol completes
// ckd_output.unmask(app_sk) returns a WRONG key, not msk·H(pk||app_id)
// The application silently uses the corrupted derived key
```

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
```
