### Title
Unbounded Per-Block Keccak Loop in `move_to_block` Causes Permanent Gas-Exhaustion DOS on All Hashchain-Protected Engine Operations — (File: `engine-hashchain/src/hashchain.rs`)

---

### Summary

`Hashchain::move_to_block` iterates **one NEAR block at a time**, computing a full keccak hash per iteration, to catch up from the stored block height to the current NEAR block height. NEAR produces roughly one block per second and caps each transaction at ~300 TGas. A single keccak256 call costs on the order of 1–2 TGas, so the loop can process only ~150–300 blocks before exhausting gas. If the Aurora Engine contract is idle for more than a few minutes, the next call to any hashchain-protected method (`submit`, `deposit`, `call`, etc.) will run out of gas and revert. Because the revert leaves the stored `current_block_height` unchanged, every subsequent call faces an even larger gap and also reverts — producing a **permanent** DOS on all hashchain-protected operations.

---

### Finding Description

**Root cause — `engine-hashchain/src/hashchain.rs`, `move_to_block` (lines 67–88):**

```rust
pub fn move_to_block(
    &mut self,
    next_block_height: u64,
) -> Result<(), BlockchainHashchainError> {
    if next_block_height <= self.current_block_height {
        return Err(BlockchainHashchainError::BlockHeightIncorrect);
    }

    while self.current_block_height < next_block_height {   // ← unbounded
        self.previous_block_hashchain = self.block_hashchain_computer
            .compute_block_hashchain(                        // ← keccak per iteration
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );
        self.block_hashchain_computer.clear_txs();
        self.current_block_height += 1;
    }
    Ok(())
}

These two wrappers guard every major engine entry point (EVM transaction submission, connector deposits, admin calls, etc.).

**Why the DOS becomes permanent:**
NEAR transactions are atomic. When `move_to_block` exhausts gas, the entire transaction reverts, including any update to the stored `Hashchain` state. The stored `current_block_height` therefore remains at its old value. The next call must iterate over an even larger gap (old height → new, higher current height) and also reverts. The engine is stuck until the hashchain is administratively reset or the feature is disabled — neither of which is available to ordinary users.

---

### Impact Explanation

All operations wrapped by `with_hashchain` / `with_logs_hashchain` — including EVM transaction submission (`submit`), ETH/token deposits, and cross-contract calls — become permanently unreachable once the block-height gap exceeds the per-transaction gas budget. User funds deposited into the engine cannot be moved or withdrawn via the normal EVM path, constituting a **permanent freeze of funds** for all users of the engine.

---

### Likelihood Explanation

NEAR produces approximately one block per second. A gap of ~200–300 blocks (3–5 minutes of engine inactivity) is sufficient to exhaust gas. Aurora Engine is a general-purpose EVM layer; periods of low activity lasting several minutes are routine (e.g., overnight, weekends, or during NEAR network congestion). No attacker action is required — the condition arises naturally from normal operation. Once triggered, no unprivileged user can recover the engine.

---

### Recommendation

Replace the unbounded per-block loop with a **O(1) catch-up**: when `next_block_height > current_block_height + 1`, compute the hashchain for the gap using a single aggregated hash (e.g., hash the range boundaries and the empty-block hash) rather than iterating every intermediate block. Alternatively, cap the number of iterations per call and allow the catch-up to be spread across multiple transactions, or skip intermediate empty blocks entirely and only hash blocks that actually contained transactions.

---

### Proof of Concept

1. Initialize the Aurora Engine with the hashchain feature enabled (hashchain state written to storage).
2. Allow the NEAR chain to advance by ≥ 300 blocks (~5 minutes) without any call to the engine.
3. Submit any EVM transaction via `submit` (or any other `with_hashchain`-wrapped method).
4. Observe: `load_hashchain` calls `move_to_block(current_near_block)`, which loops ≥ 300 times computing keccak, exhausting the 300 TGas NEAR transaction limit and reverting.
5. Repeat step 3 — the stored `current_block_height` is still the old value; the gap is now even larger; the call reverts again.
6. All subsequent calls revert indefinitely: **permanent DOS / fund freeze**. [6](#0-5) [3](#0-2)
