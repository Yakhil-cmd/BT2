### Title
Scheduled Callback Transactions Executed in Undefined Order Due to Missing EventIndex Sort in `ExecuteCallbacksTransactions` — (File: `fvm/blueprints/scheduled_callback.go`)

---

### Summary

The current (v1) implementation of `blueprints.ExecuteCallbacksTransactions` constructs scheduled callback transaction bodies from `PendingExecution` events without sorting them by `EventIndex`. The function even carries a self-acknowledged `// todo` comment noting this gap. The frozen v0 implementation (`builderV0.ExecuteCallbacksTransactions`) explicitly sorts by `EventIndex` before constructing transactions. Because the execution node's `blockComputer` uses the v1 path, the execution order of scheduled transactions in the system chunk is not guaranteed to match the order in which the `FlowTransactionScheduler` contract intended them to run, creating an ordering vulnerability analogous to the external report's "actions executed out of order due to lack of guard."

---

### Finding Description

`blueprints.ExecuteCallbacksTransactions` (v1, the live path used by the execution node) iterates over `processEvents` in whatever order they arrive, with no sort applied:

```go
// fvm/blueprints/scheduled_callback.go
for _, event := range processEvents {
    // todo make sure to check event index to ensure order is indeed correct
    // event.EventIndex
    ...
    txs = append(txs, tx)
}
```

The `// todo` comment is an explicit developer acknowledgment that the EventIndex ordering has not been enforced.

The frozen v0 implementation, used for historical block re-execution, **does** sort:

```go
// model/access/systemcollection/system_collection_v0.go
slices.SortFunc(processEvents, func(a, b flow.Event) int {
    return int(a.EventIndex - b.EventIndex)
})
```

The execution node's `executeProcessCallback` calls `blueprints.ExecuteCallbacksTransactions` with `txn.Output().Events` from the `FlowTransactionScheduler.process()` transaction. The resulting `callbackTxs` slice is then fed directly into `queueSystemTransactions`, which enqueues them in the order they appear in the slice, and `executeQueue` dispatches them to worker goroutines in that same order. No re-ordering or guard is applied at any point in the chain.

The `PendingExecution` event carries a `priority` field, indicating the scheduler contract is designed to express execution priority. If the scheduler contract's internal data structure (e.g., a Cadence dictionary or array) does not emit events in strict `EventIndex` order, or if any code path passes events in a different order (e.g., the Access Node's `LocalTransactionProvider.ScheduledTransactionsByBlockID` or `ENTransactionProvider.ScheduledTransactionsByBlockID`, both of which call `ExecuteCallbacksTransactions` with events retrieved from storage), the callback transactions will be constructed and executed in the wrong order.

---

### Impact Explanation

The `FlowTransactionScheduler` is designed to allow Cadence contract authors and users to schedule transactions with explicit priorities. If two scheduled transactions are ordered such that transaction A must execute before transaction B (e.g., A sets up a capability or state that B depends on, or A records a "last executed" timestamp that B checks), executing them in the wrong order bypasses the intended sequencing. This is the direct Flow analog of the DCA strategy example in the external report: a multi-step scheduled workflow (step 1: perform action; step 2: update state/timestamp) can be inverted, causing the state update to be skipped or the action to execute without the prerequisite state being set. The result is unauthorized or unintended on-chain state mutation within the system chunk, affecting any user who has scheduled transactions with ordering dependencies.

---

### Likelihood Explanation

Any Cadence contract author or user who can schedule transactions via the `FlowTransactionScheduler` contract (demonstrated as accessible in the integration test `TestScheduledTransactions`) and who schedules multiple transactions in the same block is exposed. The `// todo` comment confirms the ordering invariant is not enforced. The divergence between v0 (sorts) and v1 (does not sort) is a concrete regression. The Access Node paths (`LocalTransactionProvider` and `ENTransactionProvider`) also call the unsorted v1 path when reconstructing scheduled transactions for API responses, meaning the ordering gap is present across multiple code paths.

---

### Recommendation

Apply the same `slices.SortFunc` by `EventIndex` that v0 uses, inside `blueprints.ExecuteCallbacksTransactions`, before iterating over `processEvents`. Remove the `// todo` comment once the sort is in place. Additionally, add an explicit assertion or test that verifies callback transactions are constructed in EventIndex order regardless of the order in which events are supplied.
