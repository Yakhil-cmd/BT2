### Title
`ExitToEthereum` Precompile Schedules `withdraw` Promise With No Failure Callback, Causing Permanent Fund Loss on Promise Revert - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns user tokens inside the EVM and then schedules an async `withdraw` promise on the ETH connector contract. Unlike the `ExitToNear` precompile, which conditionally attaches an `exit_to_near_precompile_callback` to handle promise failure and refund tokens, `ExitToEthereum` attaches **no callback at all**. If the `withdraw` promise fails for any reason after the EVM burn has committed, there is no recovery path: the tokens are permanently destroyed on the Aurora side without being released on the Ethereum side.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run` function constructs the outbound promise as a bare `PromiseArgs::Create`:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

No callback is attached. Compare this to `ExitToNear`, which conditionally wraps the transfer promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [2](#0-1) 

The `exit_to_near_precompile_callback` handler explicitly checks the promise result and, when the `error_refund` feature is enabled, calls `engine::refund_on_error` to restore the user's EVM balance:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [3](#0-2) 

No equivalent mechanism exists for `ExitToEthereum`. The execution sequence is:

1. EVM receipt executes: user's ETH or ERC-20 balance is debited inside the EVM.
2. `withdraw` promise is scheduled on the ETH connector (or NEP-141 contract).
3. The `withdraw` receipt executes asynchronously. If it fails (ETH connector paused, contract error, gas exhaustion, etc.), NEAR does **not** roll back the EVM state from step 1.
4. No callback receipt exists to detect the failure and restore the user's balance.

The EVM accounting (token burn) is committed and irreversible; the bridge accounting (NEP-141 burn + Ethereum proof) never completes. The funds are gone. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing/loss of funds.**

Any user who calls the `ExitToEthereum` precompile while the ETH connector's `withdraw` method is unable to succeed (paused, contract error, or gas issue) will have their tokens permanently destroyed on the Aurora side with no Ethereum-side release and no on-chain recovery path. The `Accounting` struct that tracks gained/lost ETH supply is never corrected. [5](#0-4) 

---

### Likelihood Explanation

The ETH connector contract is a separate NEAR account. Its `withdraw` method can fail if:

- The connector is **paused** (a legitimate admin action during an incident). Any in-flight `ExitToEthereum` EVM receipts whose `withdraw` receipt executes during the pause window will silently lose funds.
- The connector contract is **upgraded** between the EVM receipt and the `withdraw` receipt, changing the method interface.
- **Gas exhaustion** if `costs::WITHDRAWAL_GAS` is underestimated for a given connector implementation.

The window is narrow but realistic during operational events (emergency pauses, upgrades). The user has no way to know the connector will be paused before their EVM transaction is submitted. [6](#0-5) 

---

### Recommendation

Mirror the `ExitToNear` pattern for `ExitToEthereum`: attach a callback promise (e.g., `exit_to_ethereum_precompile_callback`) that inspects the `withdraw` promise result. On failure, the callback should re-credit the user's EVM balance (ETH or ERC-20) using `engine::refund_on_error` or an equivalent mechanism. The callback must be gated on the same `error_refund` feature flag (or a new one) so it can be enabled for production builds.

---

### Proof of Concept

1. User holds 1 ETH on Aurora and calls the `ExitToEthereum` precompile (flag `0x00`, valid Ethereum recipient).
2. EVM receipt commits: 1 ETH is debited from the user's Aurora balance.
3. `withdraw` promise is scheduled on the ETH connector account.
4. Between step 2 and step 3 executing, the ETH connector is paused by an admin (legitimate emergency action).
5. The `withdraw` receipt fails with a "paused" error.
6. No callback receipt exists; NEAR discards the failed receipt.
7. User's Aurora ETH balance is 0; no Ethereum-side withdrawal proof was generated; 1 ETH is permanently lost. [7](#0-6)

pub struct Accounting {
    gained: U256,
    lost: U256,
}

impl Accounting {
    pub fn change(&mut self, amount: Change) {
        match amount.new_value.cmp(&amount.old_value) {
            Ordering::Greater => {
                let net_gained = amount.new_value - amount.old_value;
                self.gained = self.gained.saturating_add(net_gained);
            }
            Ordering::Less => {
                let net_lost = amount.old_value - amount.new_value;
                self.lost = self.lost.saturating_add(net_lost);
            }
            Ordering::Equal => (),
        }
    }

    pub fn remove(&mut self, amount: U256) {
        self.lost = self.lost.saturating_add(amount);
    }

    #[must_use]
    pub fn net(&self) -> Net {
        match self.gained.cmp(&self.lost) {
            Ordering::Equal => Net::Zero,
            Ordering::Greater => Net::Gained(self.gained - self.lost),
            Ordering::Less => Net::Lost(self.lost - self.gained),
        }
    }
