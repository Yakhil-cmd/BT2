Audit Report

## Title
Unbounded Per-Account Vec Growth in `retrieve_btc_account_to_block_indices` Causes Unbounded Query Cost in `retrieve_btc_status_v2_by_account` - (File: `rs/bitcoin/ckbtc/minter/src/state/audit.rs`, `rs/bitcoin/ckbtc/minter/src/state.rs`)

## Summary
The ckBTC minter maintains a `BTreeMap<Account, Vec<u64>>` called `retrieve_btc_account_to_block_indices` that accumulates every accepted withdrawal request's block index per account and is never pruned. The public query endpoint `retrieve_btc_status_v2_by_account` iterates over the entire unbounded Vec for a target account, calling `retrieve_btc_status_v2` for each entry. For high-volume accounts, this causes the query to consume an ever-growing number of instructions, eventually exceeding the IC per-query instruction limit and permanently breaking status lookup for that account.

## Finding Description
**Root cause — unbounded Vec, never pruned:**

In `accept_retrieve_btc_request`, every successful withdrawal call appends the new `block_index` to the per-account Vec:
```rust
state
    .retrieve_btc_account_to_block_indices
    .entry(account)
    .and_modify(|entry| entry.push(request.block_index))
    .or_insert(vec![request.block_index]);
```
The field is declared as an unbounded `Vec<u64>` per account at `rs/bitcoin/ckbtc/minter/src/state.rs` L462-463. A grep search across the entire codebase for any removal from this map returns zero matches — no code path ever removes entries from this Vec. The event log replay in `rs/bitcoin/ckbtc/minter/src/state/eventlog.rs` L395-403 also only appends to the map, never removes.

`push_finalized_request` caps `finalized_requests` at `MAX_FINALIZED_REQUESTS = 100` but leaves `retrieve_btc_account_to_block_indices` completely untouched (`rs/bitcoin/ckbtc/minter/src/state.rs` L1330-1337).

**Unbounded query cost:**

`retrieve_btc_status_v2_by_account` iterates over every block index in the Vec without any limit (`rs/bitcoin/ckbtc/minter/src/state.rs` L807-819). For each entry, `retrieve_btc_status_v2` first performs O(log n) BTreeMap lookups (`pending_reimbursements`, `pending_withdrawal_reimbursements`, `reimbursed_transactions`, `reimbursed_withdrawals`). Entries not found in those maps fall through to `retrieve_btc_status`, which performs three O(n) linear scans over `pending_retrieve_btc_requests` (up to 5,000 entries), `submitted_transactions`, and `finalized_requests` (`rs/bitcoin/ckbtc/minter/src/state.rs` L872-917). For entries that are "Unknown" (not in any BTreeMap), the full linear scan is triggered.

The query endpoint is publicly callable by any principal with an arbitrary `target` account (`rs/bitcoin/ckbtc/minter/src/main.rs` L186-189).

**The global cap does not prevent per-account Vec bloat:**

`MAX_CONCURRENT_PENDING_REQUESTS = 5000` limits only the number of currently incomplete requests across all accounts (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` L22, L174-179). It does not bound the historical accumulation in `retrieve_btc_account_to_block_indices`, which persists indefinitely across all finalized requests.

## Impact Explanation
This is a High-severity application-level DoS. As a single account accumulates withdrawal history (k entries), the instruction cost of `retrieve_btc_status_v2_by_account` grows without bound. Once k is large enough, the query exceeds the IC per-query instruction limit (~5 billion instructions), permanently breaking the status lookup endpoint for that account. Additionally, the `retrieve_btc_account_to_block_indices` map itself grows without bound in canister heap memory, contributing to long-term memory exhaustion of the production ckBTC minter canister. This matches the allowed impact: "Application/platform-level DoS... not based on raw volumetric DDoS" (High, $2,000–$10,000).

## Likelihood Explanation
Medium. Each `retrieve_btc` call requires burning real ckBTC, so the attack is economically constrained. However, a legitimate high-volume user (e.g., an exchange or automated service) naturally accumulates thousands of historical entries over months of operation, triggering the degradation without any malicious intent. An attacker with sufficient ckBTC can deliberately bloat a target account's Vec and then repeatedly call the query against it. The ckBTC minter is a production mainnet canister handling real Bitcoin value, making even medium-likelihood degradation significant.

## Recommendation
1. **Prune `retrieve_btc_account_to_block_indices` on finalization.** When a request is finalized (confirmed or reimbursed), remove its `block_index` from the per-account Vec. Only pending/in-flight indices need to be tracked for `retrieve_btc_status_v2_by_account`.
2. **Cap the Vec per account.** Enforce a maximum number of tracked block indices per account (e.g., the last N requests), evicting the oldest entries when the cap is reached.
3. **Add pagination to `retrieve_btc_status_v2_by_account`.** Accept `start` and `length` parameters so callers cannot force unbounded iteration in a single query call.

## Proof of Concept
1. An account calls `retrieve_btc` or `retrieve_btc_with_approval` successfully N times over time. Each call appends one entry to `retrieve_btc_account_to_block_indices[account]` (`rs/bitcoin/ckbtc/minter/src/state/audit.rs` L27-33).
2. The global `MAX_CONCURRENT_PENDING_REQUESTS` check passes because requests are processed and finalized between calls, keeping the incomplete count low.
3. After N successful withdrawals, `retrieve_btc_account_to_block_indices[account]` contains N entries. None are ever removed (confirmed by zero grep matches for removal).
4. Any caller invokes `retrieve_btc_status_v2_by_account(Some(account))`. The query iterates all N entries (`rs/bitcoin/ckbtc/minter/src/state.rs` L813-819), calling `retrieve_btc_status_v2` → `retrieve_btc_status` for each, performing O(N × pending_queue_size) work for entries that fall through to the linear scan path.
5. A deterministic integration test or PocketIC test can reproduce this by: (a) initializing the minter state, (b) inserting N synthetic entries directly into `retrieve_btc_account_to_block_indices` for a test account, (c) calling `retrieve_btc_status_v2_by_account` and measuring instruction consumption as N grows, confirming linear growth without bound.
