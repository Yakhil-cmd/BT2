### Title
Malicious Sender Can Permanently Abort Honest Parties via Echo Equivocation in `reliable_broadcast_receive_all` — (`src/protocol/echo_broadcast.rs`)

---

### Summary

The early-abort optimization in `reliable_broadcast_receive_all` (lines 241–263) fires based solely on each party's local view of received echo counts. A malicious sender can equivocate — sending different `Send` values to different honest parties, then sending different `Echo` values to different honest parties — to engineer a split echo distribution that triggers the early-abort for some honest parties while allowing others to proceed to Ready. This permanently desynchronizes the DKG/reshare/refresh state across honest parties.

---

### Finding Description

`echo_ready_thresholds` computes, for n=4:

```
broadcast_threshold = (4 - 1) / 3 = 1
echo_t = usize::midpoint(4, 1) = 2
```

A party must collect **strictly more than 2** echo votes for a single value to proceed to Ready. [1](#0-0) 

The early-abort check at lines 241–263 fires when `finish_amplification` is false (i.e., always during the Echo phase) and no candidate value satisfies `cnt + non_received_echo_cnt > echo_t`: [2](#0-1) 

`finish_amplification` is only set in the Ready phase (line 287), so it is always `false` when the Echo early-abort check runs: [3](#0-2) 

**Concrete attack for n=4 (H1, H2, H3, M — M is the sender):**

**Step 1 — Send equivocation:**
- M sends `Send(A)` to H1, H2
- M sends `Send(B)` to H3

Honest parties echo what they received:
- H1 → `Echo(A)` to all
- H2 → `Echo(A)` to all
- H3 → `Echo(B)` to all

**Step 2 — Echo equivocation:**
- M sends `Echo(B)` to H1 and H2
- M sends `Echo(A)` to H3

**H1's local state after all 4 echoes arrive:**
| Value | Count |
|-------|-------|
| A | 2 (from H1-self, H2) |
| B | 2 (from H3, M) |

`received_echo_cnt = 4`, `non_received_echo_cnt = 0`
Check: `max(2+0, 2+0) = 2`, not `> echo_t(2)` → **`return Err(AssertionFailed(...))`** [4](#0-3) 

**H3's local state after all 4 echoes arrive:**
| Value | Count |
|-------|-------|
| A | 3 (from H1, H2, M) |
| B | 1 (from H3-self) |

Check: `3+0 = 3 > echo_t(2)` → **proceeds to `Ready(A)`** [5](#0-4) 

H1 and H2 permanently abort with `AssertionFailed`; H3 proceeds. The DKG/reshare/refresh session is permanently split.

---

### Impact Explanation

`do_broadcast` is called directly from DKG, reshare, refresh, and CKD protocol flows. An `Err` return from `reliable_broadcast_receive_all` propagates up and terminates those protocols for the aborting parties. The parties that aborted cannot recover without restarting the entire protocol, and the parties that proceeded have already advanced their state — making a clean restart impossible without external coordination. This constitutes **permanent denial of DKG/reshare/refresh** for honest parties under valid protocol inputs and within the documented trust assumption of f < n/3 malicious parties. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only one malicious participant who controls their own network stack — a standard assumption in threshold protocol threat models. The equivocation is straightforward: send different byte strings to different peers in both the Send and Echo phases. No cryptographic assumption needs to be broken. The attack is deterministic and reproducible for any n ≥ 4 with f=1.

---

### Recommendation

Remove the early-abort optimization entirely, or make it safe:

1. **Safe option:** Only fire the early-abort after `finish_amplification` is set (i.e., after the party has already committed to a Ready message). At that point, the party's own echo is already locked in and the check is sound.
2. **Correct option:** Follow the standard Bracha reliable broadcast protocol, which does not include this optimization. Simply wait for `> echo_t` echoes for a single value; if the sender equivocates, no value will ever reach the threshold and the protocol will time out — which is the correct, safe behavior.

The guard `else if !state_sid.finish_amplification` at line 241 was intended to protect against the check firing after amplification, but it does not protect against the check firing during the Echo phase when the sender has equivocated. [7](#0-6)

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
