### Title
Inflated `balance_of` in `request_funds` Suppresses L1 Mint Requests, Temporarily Freezing Unclaimed Yield — (`File: src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`RewardSupplier::request_funds` reads the contract's live STRK token balance via `balance_of(get_contract_address())` to decide whether to send a mint request to L1. Because this is the raw ERC-20 balance, any tokens sent directly to the contract address (outside the normal `on_receive` / StarkGate path) inflate the apparent `credit`, suppressing L1 mint requests. When those donated tokens are later consumed by `claim_rewards`, the contract is left with an insufficient balance and no pending L1 request, causing `claim_rewards` to revert and temporarily freezing staker/delegator yield.

---

### Finding Description

`request_funds` is the sole mechanism that triggers L1 minting. Its logic is:

```
credit = balance_of(reward_supplier_address) + l1_pending_requested_amount
debit  = unclaimed_rewards
if credit < debit + threshold  →  send mint request(s) to L1
``` [1](#0-0) 

The `balance` term is obtained by calling `token_dispatcher.balance_of(account: get_contract_address())` — the live ERC-20 balance of the contract. [2](#0-1) 

An unprivileged actor can call `IERC20::transfer(reward_supplier_address, amount)` directly, depositing tokens into the contract without going through StarkGate. This:

1. Increases `balance` by the donated amount `D`.
2. Does **not** call `on_receive`, so `l1_pending_requested_amount` is unchanged.
3. Makes `credit` artificially large, so the `credit < debit + threshold` guard is not triggered and no L1 mint request is sent.

The donated tokens are real and will be consumed by subsequent `claim_rewards` calls (which transfer tokens to the staking contract). Once consumed, `balance` drops back to its true level. However, because no L1 request was sent during the donation window, `l1_pending_requested_amount` may be zero or insufficient. The next call to `update_unclaimed_rewards_from_staking_contract` will eventually trigger a new request, but there is a latency gap (L2 → L1 message + L1 mint + L1 → L2 bridge) during which `claim_rewards` will revert because the actual token balance is below `unclaimed_rewards`. [3](#0-2) 

The `on_receive` callback (the only path that decreases `l1_pending_requested_amount`) is restricted to the StarkGate address, so a direct ERC-20 transfer never updates the internal accounting. [4](#0-3) 

---

### Impact Explanation

When the donated tokens are consumed and no L1 mint was requested in time, `claim_rewards` attempts a `checked_transfer` that will revert because the contract's actual STRK balance is less than the requested `amount`. This blocks every staker and delegator from claiming their accrued STRK rewards until the next epoch's `update_unclaimed_rewards_from_staking_contract` fires, a new L1 request is processed, and the bridge delivers fresh tokens — a multi-epoch delay.

**Impact: Temporary freezing of unclaimed yield** (matches allowed scope).

---

### Likelihood Explanation

The attack requires only a standard ERC-20 `transfer` to the `RewardSupplier` contract address — no privileged role, no bridge compromise, no leaked key. Any address holding STRK can execute it. The attacker loses the donated tokens (they are consumed as legitimate rewards), making this a griefing attack with a direct cost, but the cost can be calibrated to the `base_mint_amount` / `threshold` parameters to maximise disruption per token spent.

---

### Recommendation

Replace the live `balance_of` call in `request_funds` with an internally tracked balance variable (e.g., `internal_balance`). Increment it in `on_receive` and decrement it in `claim_rewards`. Use only this internal variable when computing `credit`:

```rust
let balance = self.internal_balance.read(); // instead of balance_of(...)
let credit = balance + l1_pending_requested_amount;
```

This mirrors the fix described in the referenced report: maintain an internal record of the balance, distinct from the raw ERC-20 `balanceOf`, so that unsolicited token transfers cannot affect the mint-request logic.

---

### Proof of Concept

1. Deploy the system normally; let `unclaimed_rewards = U`, `l1_pending_requested_amount = P`, `balance = B` where `B + P < U + threshold` (i.e., a mint request would normally be triggered next epoch).
2. Before the next epoch's `update_unclaimed_rewards_from_staking_contract` call, an attacker calls `strk_token.transfer(reward_supplier_address, D)` where `D` is large enough that `(B + D) + P >= U + threshold`.
3. `update_unclaimed_rewards_from_staking_contract` fires; `request_funds` reads `balance_of = B + D`, computes `credit = B + D + P >= debit + threshold`, and sends **no** L1 mint request.
4. Stakers/delegators call `claim_rewards`; the staking contract calls `reward_supplier.claim_rewards(amount)`. The donated tokens `D` are transferred out, reducing the balance to `B`.
5. `B < U` (the true shortfall), so the next `claim_rewards` call triggers a `checked_transfer` that reverts — yield is frozen.
6. Recovery only occurs after the next epoch fires `request_funds` again (now seeing the true low balance), sends an L1 message, and the bridge delivers new tokens. [1](#0-0) [5](#0-4) [3](#0-2)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
