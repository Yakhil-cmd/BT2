### Title
`update_rewards` Lacks Access Control, Allowing Any Caller to Permanently Freeze Staker Yield via `disable_rewards: true` - (`src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function is specified to be callable only by the Starkware sequencer, but the implementation contains no access control check. Any unprivileged address can call it with `disable_rewards: true` to advance the global `last_reward_block` counter without distributing any rewards, permanently destroying one block's worth of yield for all stakers and blocking the legitimate sequencer call in the same block.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` unconditionally writes `last_reward_block` to the current block number **before** evaluating the `disable_rewards` flag:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // ← always executed

if disable_rewards || self.is_pre_consensus() {
    return;                                            // ← exits with no rewards
}
```

The protocol specification (`docs/spec.md`, line 1644–1645) states:

> **Access control**: Only starkware sequencer.

However, the only guard in the function is `self.general_prerequisites()`, which only checks whether the contract is paused. There is no `assert_caller_is_sequencer()` or equivalent check.

Because `last_reward_block` is a **single global storage slot** (not per-staker), a single call to `update_rewards(any_valid_staker, disable_rewards: true)` in block N:

1. Sets `last_reward_block = N` with zero rewards distributed.
2. Causes every subsequent `update_rewards` call in block N to revert with `REWARDS_ALREADY_UPDATED` (the check `current_block_number > self.last_reward_block.read()` fails).
3. Permanently destroys the rewards for block N, because the reward calculation is proportional to `current_block - last_reward_block`. When the sequencer calls in block N+1, it only receives rewards for 1 block (N+1 − N = 1), not for the skipped block N.

This is confirmed by the test `test_update_rewards_miss_blocks`, which shows that rewards accumulate across missed blocks — meaning a block whose slot is consumed by a no-op call loses its rewards forever.

The analog to the original report is exact:

| Original (symmio-core) | This codebase |
|---|---|
| `chargeFundingRate` accepts empty `quoteIds[]` | `update_rewards` accepts `disable_rewards: true` |
| Increments `partyANonces` with no real work | Advances `last_reward_block` with no rewards distributed |
| Blocks partyA's signature-dependent operations | Blocks the sequencer's legitimate reward update in the same block |
| Caller: any partyB | Caller: any address |

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

For every block in which an attacker front-runs the sequencer's `update_rewards` call with `disable_rewards: true`, all stakers permanently lose that block's rewards. The rewards are not deferred; they are destroyed because the reward calculation window is consumed by the no-op call. An attacker who does this continuously causes total, permanent loss of all consensus-phase staking yield.

---

### Likelihood Explanation

**High.** The function is publicly callable with no access control. The only cost to the attacker is gas. On Starknet, where the sequencer is a known entity submitting transactions, an attacker can reliably front-run the sequencer's `update_rewards` transaction in every block by submitting their own transaction first. No special privileges, leaked keys, or external dependencies are required.

---

### Recommendation

Add a caller check matching the specification. For example, introduce a stored `sequencer_address` and assert:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   assert!(
+       get_caller_address() == self.sequencer_address.read(),
+       "{}",
+       Error::CALLER_IS_NOT_SEQUENCER,
+   );
    ...
}
```

Alternatively, move the `last_reward_block.write(...)` call to after the `disable_rewards` guard so that a no-op call does not consume the block's reward slot.

---

### Proof of Concept

1. Staker S stakes and K epochs pass so their balance is effective.
2. Consensus rewards are active (`is_pre_consensus()` returns false).
3. In block N, attacker A (any address) calls:
   ```
   update_rewards(staker_address: S, disable_rewards: true)
   ```
4. `last_reward_block` is written to N. No rewards are distributed.
5. The legitimate sequencer calls `update_rewards(staker_address: S, disable_rewards: false)` in block N → reverts with `REWARDS_ALREADY_UPDATED`.
6. In block N+1, the sequencer calls `update_rewards(S, false)` → rewards for 1 block only (N+1 − N = 1). Block N's rewards are permanently lost.
7. Repeating step 3 every block causes total, continuous loss of staking yield. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
