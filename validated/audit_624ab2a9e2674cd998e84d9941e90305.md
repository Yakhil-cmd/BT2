### Title
Anyone Can Call `update_rewards` to Block All Staker Reward Distributions Per Block - (File: `src/staking/staking.cairo`)

### Summary

`IStakingRewardsManager::update_rewards` is callable by any address despite the specification requiring "Only starkware sequencer" access. Because it writes to the global `last_reward_block` storage variable before distributing rewards, any caller can front-run the legitimate sequencer call with `disable_rewards: true`, consuming the per-block reward slot without distributing any yield, and causing the sequencer's legitimate call to revert with `REWARDS_ALREADY_UPDATED`.

### Finding Description

The `update_rewards` function in `StakingRewardsManagerImpl` performs no caller authentication: [1](#0-0) 

It unconditionally writes `last_reward_block` to the current block number before checking whether rewards should be distributed: [2](#0-1) 

`last_reward_block` is a single global storage variable (not per-staker). The guard at line 1454–1458 enforces that only one call per block succeeds. Once an attacker calls `update_rewards(any_valid_staker, disable_rewards: true)`, the slot is consumed: `last_reward_block` equals the current block, no rewards are credited, and every subsequent call in that block reverts.

The specification explicitly restricts this function to the Starknet sequencer: [3](#0-2) 

The interface definition carries no such restriction in code: [4](#0-3) 

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who front-runs the sequencer's `update_rewards` call every block with `disable_rewards: true` permanently prevents all stakers from accumulating block rewards. Because `last_reward_block` is global, a single attacker transaction per block is sufficient to deny rewards to every staker and every delegation pool in the protocol. The yield is not deferred — it is simply never credited.

### Likelihood Explanation

**High.** The attack requires no special privilege, no capital at risk, and only one cheap transaction per block. On Starknet, transaction ordering within a block is controlled by the sequencer, but the attacker only needs to submit the call before the sequencer's own `update_rewards` inclusion. The cost is minimal gas; the attacker has no profit motive but can inflict continuous, protocol-wide yield denial.

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the configured sequencer address (or a dedicated rewards-manager role), mirroring the pattern already used for `update_rewards_from_attestation_contract`: [5](#0-4) 

Introduce an analogous `assert_caller_is_sequencer()` guard as the first statement inside `update_rewards`, before the `last_reward_block` write.

### Proof of Concept

1. Consensus rewards are active; staker `S` is registered and has non-zero balance.
2. Attacker `A` (any EOA) calls `update_rewards(S, disable_rewards: true)` in block `N`.
3. `last_reward_block` is written to `N`; the function returns early — no rewards credited.
4. The sequencer's legitimate `update_rewards(S, disable_rewards: false)` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` (and all delegation pool members) receive zero rewards for block `N`.
6. Repeating step 2 every block permanently freezes all yield accumulation across the protocol.

The existing test suite already demonstrates the `REWARDS_ALREADY_UPDATED` revert when a second call is made in the same block: [6](#0-5)
