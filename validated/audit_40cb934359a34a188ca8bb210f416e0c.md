### Title
Missing Caller Restriction in `execute_scheduled` Enables Unauthorized Premature Execution of Scheduled Cross-Contract Promises - (File: etc/xcc-router/src/lib.rs)

---

### Summary

The `execute_scheduled` function in the XCC router contract has no caller restriction, allowing any NEAR account to trigger execution of a user's scheduled cross-contract promises at an arbitrary time. This is the direct analog of the reported vulnerability: just as any connected dapp could invoke multi-chain functionality without user confirmation, any NEAR account can invoke a user's scheduled promise without the user's consent, bypassing the intended timing control.

---

### Finding Description

The XCC router contract (`etc/xcc-router/src/lib.rs`) stores scheduled promises in `scheduled_promises: LookupMap<u64, PromiseArgs>`. The `schedule` function correctly requires `assert_preconditions()`, which verifies the caller is the parent Aurora Engine: [1](#0-0) 

However, `execute_scheduled` has no such restriction: [2](#0-1) 

The comment at lines 146–148 acknowledges this design choice and claims "there is no security risk." This reasoning is incomplete. While the promise *content* is fixed (authorized by the engine), the *timing* of execution is entirely uncontrolled. Any NEAR account can call `execute_scheduled(nonce)` at any moment.

The NEAR tokens required for the scheduled promise are transferred to the router at scheduling time via `withdraw_wnear_to_router`: [3](#0-2) 

This means the router holds the user's NEAR from the moment the promise is scheduled. An attacker who calls `execute_scheduled` prematurely causes the router to spend those funds at an unfavorable time.

Critically, the `remove` at line 151 is committed to storage as soon as `execute_scheduled` returns — regardless of whether the downstream async promise succeeds or fails. If the downstream call fails, the promise is permanently gone from storage and the user must re-schedule. [4](#0-3) 

Compare with `execute` and `schedule`, which both call `assert_preconditions()`: [5](#0-4) [6](#0-5) 

The `assert_preconditions` check verifies `predecessor_account_id == self.parent` (the Aurora Engine): [7](#0-6) 

`execute_scheduled` is the only state-mutating method that skips this check entirely.

---

### Impact Explanation

**Theft of unclaimed yield (High):** A user schedules a DeFi operation (e.g., a token swap, liquidity deposit, or reward claim) via the XCC precompile using `CrossContractCallArgs::Delayed`. The user's wNEAR is converted and held in the router. An attacker monitors the router's storage, identifies the nonce, and calls `execute_scheduled` at an unfavorable moment — for example, after front-running the swap to inflate the price of the output token. The user receives fewer tokens than they would have at their intended execution time. The attacker profits from the price impact they induced.

**Temporary freezing of funds (High):** If the attacker triggers `execute_scheduled` when the target contract is not ready (e.g., a lending protocol in a paused state, or a DEX with insufficient liquidity), the downstream promise fails. The NEAR attached to the call is refunded to the router, but the scheduled promise entry is permanently deleted from `scheduled_promises`. The user must re-schedule the operation by calling the XCC precompile again, paying additional gas and wNEAR, and potentially missing a time-sensitive window (e.g., avoiding liquidation, claiming expiring rewards).

---

### Likelihood Explanation

**Medium.** The attacker needs:
1. The router contract address — publicly derivable as `{hex_evm_address}.aurora` from any EVM address.
2. The nonce of the scheduled promise — publicly readable from NEAR on-chain storage.
3. A NEAR account — no special privileges required.

All three are trivially obtainable by monitoring the NEAR blockchain. The function is `#[payable]` but imposes no minimum deposit, so the attacker can call it with 0 attached NEAR. No admin compromise, oracle error, or social engineering is required.

---

### Recommendation

Add a caller check to `execute_scheduled` consistent with `execute` and `schedule`. If open execution by third parties is desired for automation purposes, introduce a time-lock: store a `scheduled_at` block height alongside each promise and only allow execution after a user-specified delay. This preserves automation while preventing premature or adversarial execution.

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    self.assert_preconditions(); // Add this line
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

### Proof of Concept

1. User calls the XCC precompile (`engine-precompiles/src/xcc.rs`) with `CrossContractCallArgs::Delayed(call)` encoding a DEX swap.
2. The engine calls `schedule` on the user's router (`{hex_address}.aurora`), storing the `PromiseArgs` at nonce `N`. The user's wNEAR is converted to NEAR and held in the router via `withdraw_wnear_to_router`.
3. Attacker reads the router's `scheduled_promises` storage on-chain and identifies nonce `N` and the swap details.
4. Attacker front-runs: buys the output token on the DEX, inflating its price.
5. Attacker calls `execute_scheduled(N)` on the router. No caller check is performed.
6. The router removes the promise from storage (line 151) and executes the swap at the inflated price (line 153–154).
7. User receives fewer output tokens than expected. Attacker sells the output token at profit.
8. If the swap fails instead, the promise is permanently deleted from storage; the user must re-schedule and re-fund. [8](#0-7) [9](#0-8) [10](#0-9)

    /// Checks the following preconditions:
    ///   1. Contract is initialized
    ///   2. predecessor_account_id == self.parent
    ///   3. There are no failed promise results
    /// These preconditions must be checked on methods where are important for
    /// the security of the contract (e.g. `execute`).
    fn require_preconditions(&self) -> Result<(), Error> {
        let parent = self.get_parent()?;
        require_caller(&parent)?;
        require_no_failed_promises()?;
        Ok(())
    }

    /// Panics if any of the preconditions checked in `require_preconditions` are not met.
    fn assert_preconditions(&self) {
        self.require_preconditions().unwrap_or_else(env_panic);
    }
