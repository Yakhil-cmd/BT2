### Title
Unbounded `pool_member_epoch_balance` Trace Growth Enables Permanent Freezing of Delegator Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

`add_to_delegation_pool` in `src/pool/pool.cairo` enforces only `amount > 0` with no minimum-amount floor. Each call in a distinct epoch appends a new entry to the per-member `pool_member_epoch_balance` trace. The `calculate_rewards` helper — called by both `claim_rewards` and `pool_member_info_v1` — contains an **explicitly unbounded loop** over that trace. A delegator who repeatedly tops up with dust amounts across many epochs, while deferring `claim_rewards`, can grow the trace until the loop exceeds the Starknet transaction gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

`add_to_delegation_pool` validates only that `amount` is non-zero:

```cairo
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
``` [1](#0-0) 

Every successful call routes through `increase_member_balance` → `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. Because the key is `current_epoch + K`, one new trace entry is created per distinct epoch in which the member modifies their balance:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

`claim_rewards` calls `calculate_rewards`, which iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` up to the current epoch. The developers themselves annotate this loop as unbounded:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
``` [3](#0-2) 

`entry_to_claim_from` is only advanced on a **successful** `claim_rewards` transaction:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [4](#0-3) 

If the transaction reverts due to gas exhaustion, the pointer is never updated, so every subsequent `claim_rewards` attempt must re-iterate the same oversized trace and will also revert — permanently.

`exit_delegation_pool_intent` also calls `set_member_balance`, so alternating add/exit-intent operations across epochs can grow the trace even faster:

```cairo
self.set_member_balance(:pool_member, amount: new_delegated_stake);
``` [5](#0-4) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace is large enough that the `calculate_rewards` loop exceeds the Starknet transaction gas ceiling, every call to `claim_rewards` for that pool member will revert. Because `entry_to_claim_from` is only persisted on success, the pointer never advances, and the delegator's accumulated STRK rewards are **permanently frozen** — matching the allowed High impact: *"Permanent freezing of unclaimed yield."*

---

### Likelihood Explanation

- Any pool member (unprivileged delegator) can trigger this unilaterally; no privileged role is required.
- The minimum cost per epoch is 1 token unit (`amount = 1`), making the attack cheap in token terms.
- The attacker must sustain one transaction per epoch over many epochs without claiming rewards. Starknet epochs are short (on the order of hours), so hundreds to thousands of entries can accumulate within weeks to months.
- The developers' own comment acknowledges the loop is unbounded, confirming awareness of the risk.
- A legitimate delegator who makes frequent small top-ups and rarely claims can trigger this accidentally.

---

### Recommendation

1. **Enforce a minimum delegation amount** in `add_to_delegation_pool` (and `enter_delegation_pool`) analogous to the staking contract's `min_stake` check. Reject deposits below a protocol-defined `min_delegation` threshold.
2. **Cap trace growth** by refusing balance updates that would create a new trace entry when the trace already has more than a safe maximum number of unclaimed entries (i.e., require `claim_rewards` before adding again if the trace is too long).
3. **Bound the loop** in `calculate_rewards` with a maximum iteration count and allow partial reward claims, advancing `entry_to_claim_from` incrementally across multiple transactions.

---

### Proof of Concept

```
1. Delegator D enters pool with amount = 1 (minimum non-zero).
2. Each epoch, D calls add_to_delegation_pool(pool_member: D, amount: 1).
   → set_member_balance inserts a new entry at epoch + K each time.
   → pool_member_epoch_balance trace grows by 1 entry per epoch.
3. D never calls claim_rewards.
4. After N epochs (N large enough to exceed gas limit in the loop):
   D calls claim_rewards.
   → calculate_rewards iterates from entry_to_claim_from=0 to N.
   → Transaction runs out of gas and reverts.
   → entry_to_claim_from remains 0.
5. Every subsequent claim_rewards call also reverts.
   → D's accumulated STRK rewards are permanently frozen.
```
