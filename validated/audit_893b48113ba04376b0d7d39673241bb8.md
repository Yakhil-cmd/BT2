### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions When Any Non-Deploy-Account Transaction Exists in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway stateful validator is intended to skip the `__validate__` entry-point call only when a `deploy_account` transaction is pending for a new account (UX feature: deploy + invoke in one shot). However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type in the mempool or recent committed block, not exclusively for `deploy_account` transactions. An attacker who already has any valid transaction (e.g., an invoke with nonce 0) sitting in the mempool for a fresh account can exploit this to submit a subsequent invoke with nonce 1 carrying an **invalid signature** that bypasses the gateway's `__validate__` entry-point check and is admitted to the mempool.

### Finding Description

```rust

    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect. `account_tx_in_pool_or_recent_block` checks:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type (Invoke, Declare, DeployAccount) stored in the pool for that address. [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

This causes `StatefulValidator::perform_validations` to return `Ok(())` without ever calling the account's `__validate__` entry point:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [5](#0-4) 

The `__validate__` entry point is the account's primary mechanism for signature verification. The stateless validator only checks signature **length**, not cryptographic validity:

```rust

    let signature_length = signature.0.len();
    if signature_length > self.config.max_signature_length { ... }
``` [6](#0-5) 

The mempool's `validate_tx` also does not check the signature — it only checks nonce ordering and fee escalation:

```rust

    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;

### Impact Explanation

An invoke transaction with nonce 1 carrying an **invalid or forged signature** is admitted to the mempool without any cryptographic authorization check. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The transaction will ultimately fail during batcher execution (the blockifier re-runs `__validate__` with `validate: true` at execution time), but it has already been admitted to the mempool, consuming mempool capacity and sequencer resources. An attacker can use this to flood the mempool with signature-invalid transactions at low cost (only one valid nonce-0 transaction is needed as a "key").

### Likelihood Explanation

The preconditions are easily met by any unprivileged user:
1. Create a fresh account (nonce 0 in state — trivially satisfied for any new address).
2. Submit one valid invoke with nonce 0 (passes all validation, enters mempool).
3. Submit arbitrarily many invokes with nonce 1 and invalid signatures — all bypass `__validate__` at the gateway.

No privileged access, flash loans, or transaction-ordering control is required. The window is open for as long as the nonce-0 transaction remains unconfirmed in the mempool.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`DeployAccount` transaction** is pending for the address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address)` that only returns `true` when the pending transaction is of type `DeployAccount`. This preserves the intended UX feature while closing the bypass for arbitrary transaction types.

### Proof of Concept

```
1. Attacker controls address A (nonce = 0 in committed state, no deployed contract).

2. Attacker submits InvokeV3 from A, nonce=0, valid signature.
   → Passes stateless validation (signature length OK).
   → Passes stateful nonce check (0 == 0).
   → Passes __validate__ entry point (valid sig).
   → Admitted to mempool. account_tx_in_pool_or_recent_block(A) = true.

3. Attacker submits InvokeV3 from A, nonce=1, INVALID/FORGED signature.
   → Passes stateless validation (signature length OK).
   → Passes stateful nonce check (0 <= 1 <= 0+max_gap).
   → skip_stateful_validations: nonce==1, account_nonce==0,
     account_tx_in_pool_or_recent_block(A)==true → returns true.
   → run_validate_entry_point called with validate=false.
   → __validate__ entry point is NEVER called.
   → Transaction admitted to mempool with invalid signature.

4. Repeat step 3 indefinitely (same nonce-0 tx in pool acts as key).
   → Mempool flooded with signature-invalid transactions at minimal cost.
``` [9](#0-8) [10](#0-9)

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(

        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),

        let cur_span = Span::current();

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
