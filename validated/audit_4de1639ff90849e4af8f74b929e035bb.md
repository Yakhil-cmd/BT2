### Title
Incorrect Royalty Amount Displayed in WalletConnect Offer Approval Due to Division Math Error - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each NFT's royalty. When a WalletConnect dApp submits a `take_offer` or `create_offer_for_ids` command involving multiple NFTs with royalties, the `amountWithRoyalties` shown to the user in the signing-approval dialog is understated by a factor of N (the number of NFTs). A malicious dApp can exploit this to make a user approve an offer believing they will spend far less XCH/CAT than will actually be deducted.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect command parsing to compute the total spend including creator royalties, which is then surfaced to the user as `amountWithRoyalties` in the approval dialog:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts L363-L368
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

The correct formula for total royalties when buying N NFTs is:

```
totalRoyalty = Σ (amount × royaltyPercentage_i / 10_000)
```

The code instead computes:

```
splitAmount = amount / N
totalRoyalty = Σ (splitAmount × royaltyPercentage_i / 10_000)
             = (amount / N) × Σ royaltyPercentage_i / 10_000
```

This is exactly `1/N` of the correct value. For a single NFT the result is correct, but for N > 1 the displayed total is understated by a factor of N.

**Concrete example** (from the existing test at line 251):
- Offer: pay 100,000,000 mojos (0.0001 XCH) to receive 2 NFTs with royalties 500 (5%) and 10 (0.1%)
- Code displays: `0.00010255` XCH (royalty = 2,550,000 mojos)
- Correct total: `0.0001051` XCH (royalty = 5,100,000 mojos)
- Understatement: ~50% of actual royalties hidden

With 10 NFTs the displayed royalty would be 1/10 of the actual royalty. The actual on-chain spend is determined by the wallet daemon and is always correct; only the approval-dialog display is wrong. [1](#0-0) 

The `withRoyaltyTotals` function feeds this result into the `DisplayWalletDeltaItem` objects returned by `parseCommandDisplay`, which are rendered in the WalletConnect approval UI: [2](#0-1) 

### Impact Explanation

A WalletConnect dApp sends `chia_wallet.take_offer` with an offer that includes multiple NFTs carrying high royalty percentages. The approval dialog shows `amountWithRoyalties` as a small number (1/N of the true total). The user, trusting the displayed figure, approves. The wallet daemon executes the offer at the correct (much higher) royalty cost, draining more XCH or CAT than the user consented to based on the displayed information. This is a direct, concrete asset-loss scenario triggered by a WalletConnect state display error.

### Likelihood Explanation

Any WalletConnect-connected dApp can craft and submit a `take_offer` payload. No special privileges, leaked keys, or social engineering beyond the normal WalletConnect connection are required. The bug is triggered automatically whenever the offer contains two or more NFTs with non-zero royalties — a common real-world scenario (e.g., bundle NFT sales).

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be calculated against the full `amount`, not a fraction of it:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The existing test at line 251 must also be updated: the expected `amountWithRoyalties` for 0.0001 XCH with royalties 500 + 10 should be `'0.0001051'`, not `'0.00010255'`. [3](#0-2) 

### Proof of Concept

1. Attacker operates a WalletConnect-enabled dApp.
2. User connects their Chia wallet via WalletConnect.
3. DApp calls `chia_wallet.take_offer` with an offer string encoding: user pays 10 XCH, receives 10 NFTs each with `royalty_percentage = 1000` (10%).
4. `formatAmountWithRoyalties` computes: `splitAmount = 10_XCH_in_mojos / 10 = 1_XCH_in_mojos`; each NFT's royalty = `1_XCH * 1000 / 10_000 = 0.1 XCH`; total royalty shown = 1 XCH; displayed total = **11 XCH**.
5. Correct total: each NFT's royalty = `10_XCH * 1000 / 10_000 = 1 XCH`; total royalty = 10 XCH; actual total = **20 XCH**.
6. User sees "11 XCH total" and approves; wallet spends 20 XCH. The 9 XCH difference is paid as royalties to NFT creators without the user's informed consent. [4](#0-3)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
