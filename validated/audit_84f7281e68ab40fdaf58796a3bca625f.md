### Title
Presignature Reuse via Non-Consuming Rerandomization API Enables Secret Key Extraction - (File: `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

### Summary
`PresignOutput` is `Clone` and `rerandomize_presign` accepts `&PresignOutput` (a shared reference), so the same presignature can be rerandomized multiple times under different `RerandomizationArguments`. A malicious coordinator can drive two signing sessions from the same underlying nonce, producing two sets of signature shares that are algebraically related. Standard ECDSA nonce-reuse analysis then recovers the aggregate secret key.

### Finding Description
Both the robust ECDSA and OT-based ECDSA pipelines require that each `PresignOutput` is consumed **exactly once**. The documentation states this explicitly:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."

However, the API does not enforce this invariant at the type level. In `src/ecdsa/robust_ecdsa/mod.rs`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct PresignOutput { ... }          // Clone: can be copied freely

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,     // &-ref, not consumed
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(...);
        }
        let delta = args.derive_randomness()?;
        ...
    }
}
```

The only guard is `presignature.big_r == args.big_r`. Nothing prevents calling `rerandomize_presign` a second time with the same `presignature` and a different `args` (different `msg_hash`, `tweak`, or `entropy`). Each call produces a distinct `RerandomizedPresignOutput` with a different `delta`-scaled nonce, but all outputs share the same underlying secret nonce material from the original `PresignOutput`.

The identical pattern exists in `src/ecdsa/ot_based_ecdsa/mod.rs` at `RerandomizedPresignOutput::rerandomize_presign`.

This is the direct analog of the TokenBridge double-spend: the presignature is the "resource" that must be used only once; `RerandomizationArguments` provides two different "tracking contexts" (analogous to `_snapshots` vs `_events`); and the absence of a consuming API is the missing "never resume normal operations" guard.

### Impact Explanation
If a malicious coordinator drives two signing sessions from the same `PresignOutput` with different `(msg_hash, tweak, entropy)` tuples, honest participants will each call `rerandomize_presign` twice on their stored share and contribute signature shares to both sessions. The coordinator collects both sets of shares. Because both `RerandomizedPresignOutput` values are deterministic linear transforms of the same secret nonce, the coordinator can apply standard ECDSA nonce-reuse algebra to recover the aggregate secret key. This is explicitly acknowledged in `docs/ecdsa/robust_ecdsa/signing.md`:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."

Impact: **Critical** — full extraction of the aggregate private signing key.

### Likelihood Explanation
The coordinator is a normal participant role, reachable without any privileged key material. Any participant can be designated coordinator. The attack requires only that the coordinator initiates two signing requests referencing the same presignature `big_r`, which honest participants cannot distinguish from two legitimate sequential sessions (the library provides no per-participant presignature-usage ledger). The attack is fully deterministic once two sessions complete.

### Recommendation
Enforce single-use at the type level by consuming the presignature:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // take ownership, not &-ref
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

Additionally, remove the `Clone` derive from `PresignOutput` in both schemes so that callers cannot silently duplicate the one-time-use secret material before passing it in.

### Proof of Concept

1. All `N = 2t+1` participants complete the presigning protocol and each stores their `PresignOutput` share `(big_r, ...)`.
2. The malicious coordinator constructs two distinct `RerandomizationArguments`:
   - `args1 = (pk, T1, M1, big_r, participants, E1)`
   - `args2 = (pk, T2, M2, big_r, participants, E2)`
3. The coordinator broadcasts `args1`; each honest participant calls `rerandomize_presign(&presig, &args1)` — the borrow leaves `presig` intact — and contributes share `s_i^{(1)}` to the coordinator.
4. The coordinator broadcasts `args2`; each honest participant calls `rerandomize_presign(&presig, &args2)` on the **same** stored `presig` and contributes share `s_i^{(2)}`.
5. The coordinator now holds `(R_1, s^{(1)})` and `(R_2, s^{(2)})` where `R_1 = delta_1 * R`, `R_2 = delta_2 * R`. Because `s^{(1)}` and `s^{(2)}` are linear functions of the same secret nonce `k`, the coordinator solves the resulting linear system to recover `x` (the aggregate secret key).

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
