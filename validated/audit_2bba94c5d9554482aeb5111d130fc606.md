### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` applies no upper-bound check on `addFeeBuyE6` / `addFeeSellE6`, while the parallel admin-fee setter `setPoolAdminFees` is explicitly capped by `maxAdminSpreadFeeE6`. A pool admin can therefore set per-bin additional fees to the `uint16` maximum (65 535 E6 ≈ 6.55%) on every bin, bypassing the factory-owner-controlled cap and making the pool economically unusable for traders.

---

### Finding Description

`setPoolAdminFees` enforces the factory cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no equivalent check — it passes the caller-supplied values straight through:

```solidity

### Impact Explanation

The factory owner sets `maxAdminSpreadFeeE6` to bound how much fee the pool admin can impose on traders. That invariant is fully bypassed: a pool admin can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = addFeeSellE6 = 65535` on every bin in the pool. The effective per-bin spread fee becomes `baseFeeX64 + 6.5535%`, regardless of what `maxAdminSpreadFeeE6` is set to. With a typical `maxAdminSpreadFeeE6` of, say, 1–5%, the uncapped per-bin fee alone exceeds the intended ceiling. Traders receive materially less output than the oracle-permitted price allows, and the pool becomes economically dead for swap volume. LPs can still withdraw principal via `removeLiquidity` (which does not route through swap math), so there is no direct loss of LP principal, but the core swap functionality is broken for the pool's lifetime until the admin resets the fees.

---

### Likelihood Explanation

The pool admin role is semi-trusted and is explicitly in scope as an admin-boundary attack surface. The call requires only `poolAdmin[pool] == msg.sender` — no timelock, no second confirmation. A single transaction per bin is sufficient to corrupt all bins. The factory owner has no on-chain mechanism to override or clamp per-bin fees after the fact without the admin's cooperation.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` mirroring the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6  > maxAdminBinFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that `baseFee + addFeeBuyE6 ≤ maxAdminSpreadFeeE6` at the time of the call, or introduce a dedicated `maxAdminBinSpreadFeeE6` cap settable only by the factory owner.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 10_000` (1%).
2. Pool admin calls `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` for every bin in the pool.
3. No revert occurs — `MetricOmmPoolFactory` passes the values through unchecked.
4. A trader calls `swap`; the effective fee applied is `baseFeeX64 + 65535/1e6 ≈ baseFee + 6.55%`, far exceeding the 1% cap.
5. Assert: `effectiveFee > maxAdminSpreadFeeE6` — the factory-owner-controlled invariant is violated.
6. Repeat for all bins; the pool is now economically unusable for any swap direction.
