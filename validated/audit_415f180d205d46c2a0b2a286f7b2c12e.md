### Title
Unbounded Concurrent WalletConnect Signing-Request Dialogs Allow Approval Hijack via Dialog Flood - (File: packages/gui/src/components/walletConnect/WalletConnectProvider.tsx)

### Summary
A connected dapp can send an unlimited number of concurrent `session_request` events for spending commands (e.g., `chia_sendTransaction`, `chia_spendCAT`, `chia_nft_transfer_nft`). Because the `session_request` handler is not serialized and `openReactDialog` opens non-modal Electron `BrowserWindow` instances, each request spawns an independent confirmation dialog simultaneously. A user overwhelmed by dozens of stacked dialogs is likely to accidentally confirm one, causing an unauthorized asset transfer to the attacker's address.

### Finding Description

The `session_request` event listener in `WalletConnectProvider.tsx` fires `handleSessionRequest` for every incoming WalletConnect request without any queuing, rate-limiting, or in-flight count check:

```ts
client.on('session_request', handleSessionRequest);
```

`handleSessionRequest` is async and is never awaited between invocations — each event is processed independently and concurrently. [1](#0-0) 

Each request that requires user confirmation reaches `dispatchPairRequest`, which calls the `confirm` callback: [2](#0-1) 

The `confirm` callback in `main.tsx` calls `openReactDialog`, which creates a **new non-modal `BrowserWindow`** for every invocation: [3](#0-2) 

`openReactDialog` defaults `modal = false` and the WalletConnect call site passes no `modal: true` override, so all dialogs are independent, non-blocking windows: [4](#0-3) 

There is no cap on how many confirmation windows can be open simultaneously, no deduplication of identical pending requests, and no serialization that would force the user to resolve one dialog before the next appears.

Spending commands that require per-request confirmation (no `allowConfirmationBypass`) include: [5](#0-4) 

### Impact Explanation

A malicious dapp that has been granted `chia_sendTransaction` (or any other spending command) permission during pairing can send N concurrent `session_request` events, each targeting the attacker's address. N confirmation windows open simultaneously on the user's desktop. The user, attempting to dismiss them, accidentally clicks "Confirm" on one. That single confirmation executes a real on-chain spend — transferring XCH, CAT tokens, or NFTs to the attacker. This is a **Critical** impact: unauthorized spend/transfer of XCH, CAT, or NFT assets.

### Likelihood Explanation

The prerequisite is that the user has already paired with the malicious dapp and granted it a spending command. This is a realistic scenario: users connect dapps for legitimate purposes (DEX, NFT marketplace) and the dapp later turns malicious, or the dapp's backend is compromised. Once paired, the attack requires only sending many WebSocket messages — no cryptographic capability or host compromise is needed.

### Recommendation

1. **Serialize confirmation dialogs**: Maintain a per-pair (or global) async queue so that only one confirmation dialog is shown at a time. Subsequent requests wait until the current dialog is resolved.
2. **Rate-limit session requests**: Reject or throttle requests from a single pair that arrive faster than a human can reasonably review (e.g., more than 1 pending confirmation per pair at a time).
3. **Deduplicate identical pending requests**: If an identical command+params is already awaiting confirmation, reject the duplicate rather than opening a second dialog.
4. **Make confirmation dialogs modal**: Pass `modal: true` to `openReactDialog` for spending commands so the dialog blocks interaction with the parent window and cannot be buried under other windows.

### Proof of Concept

1. User connects a dapp via WalletConnect and grants it `chia_sendTransaction` permission (without bypass).
2. The dapp sends 50 concurrent `session_request` events over the WalletConnect relay, each with `method: "chia_sendTransaction"`, `params: { address: "<attacker_address>", amount: <user_balance>, fee: 0 }`.
3. `WalletConnectProvider.tsx` fires `handleSessionRequest` 50 times concurrently; each reaches `openReactDialog` in `main.tsx` and opens a new non-modal `BrowserWindow`.
4. 50 "Confirm Send Transaction" windows appear on the user's desktop simultaneously.
5. The user clicks what they believe is a "Cancel" button on one dialog but accidentally hits "Send" on an overlapping dialog.
6. `dispatchPairRequest` calls `process(context)` → `sendCommand('send_transaction', 'chia_wallet', ...)` → funds are transferred on-chain to the attacker.
