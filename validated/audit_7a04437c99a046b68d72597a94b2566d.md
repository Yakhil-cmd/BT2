### Title
Keygen Threshold Not Stored in `KeygenOutput`, Enabling Malicious Coordinator to Silently Reduce Effective Signing Threshold — (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `KeygenOutput` struct omits the threshold `t` used during distributed key generation. Because `presign()` and `sign()` accept caller-supplied thresholds with no cross-validation against the keygen threshold, a malicious coordinator can supply lower-threshold Beaver triples and a lower presign threshold. Honest parties have no way to detect the inconsistency, so the effective signing threshold is silently reduced from the intended `t` to an attacker-chosen `t' < t`, allowing fewer parties than intended to produce a valid ECDSA signature.

---

### Finding Description

`KeygenOutput` is the sole output of the keygen protocol and the only artifact honest parties retain between protocol phases:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
}
``` [1](#0-0) 

The threshold `t` used to generate the secret-sharing polynomial is **not recorded** in this struct. Consequently, when an honest party later calls `presign()`, it must accept the threshold from `PresignArguments` on faith:

```rust
pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,          // threshold is caller-supplied
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError>
``` [2](#0-1) 

The only threshold validation performed is:

1. `args.threshold <= participants.len()` — the presign threshold does not exceed the participant count.
2. `args.threshold == args.triple0.1.threshold == args.triple1.1.threshold` — the presign threshold matches the triple thresholds.

There is **no check** that `args.threshold >= keygen_threshold`. The comment in the code even explicitly acknowledges a related omission:

> "We omit the check that the new participant set was present for the triple generation, because presumably they need to have been present in order to have shares." [3](#0-2) 

The orchestration specification requires:

```
N_1 >= t_0 >= t
N_2 >= t_1 >= t
```

meaning every downstream threshold must be at least as large as the keygen threshold `t`. [4](#0-3) 

This invariant is **never enforced in code** because `t` is not stored anywhere after keygen completes.

The same gap exists in the online `sign()` function, which also accepts a caller-supplied threshold with no reference to the keygen threshold:

```rust
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,   // caller-supplied, unchecked
    ...
``` [5](#0-4) 

`PresignOutput` similarly stores no threshold:

```rust
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
``` [6](#0-5) 

`TriplePub` does record a threshold, but it is the triple threshold — not the keygen threshold — and honest parties have no stored keygen threshold to compare it against:

```rust
pub struct TriplePub {
    ...
    pub threshold: ReconstructionLowerBound,
}
``` [7](#0-6)

### Impact Explanation

The presign protocol linearizes each party's key share using Lagrange coefficients computed from the presign participant set. Provided `N_1 >= t`, the sum of linearized shares correctly reconstructs the master secret contribution. The resulting presignature `(R, k_i, sigma_i)` is then a `t_1`-threshold Shamir sharing of `k` and `k·x`.

If `t_1 < t`, only `t_1` parties are needed to reconstruct `k` and `sigma = k·x` during signing, producing a valid ECDSA signature — even though the keygen was configured to require `t` parties. The signature passes standard ECDSA verification against the master public key.

This constitutes **unauthorized creation of a valid threshold signature**: a coalition of `t' < t` parties (coordinated by the malicious coordinator) can sign arbitrary messages, violating the threshold security guarantee established at keygen time.

---

### Likelihood Explanation

The attack is reachable by a **malicious coordinator**, which is an explicitly documented attacker role for this library. The coordinator:

1. Controls which triples are distributed to honest parties (via `deal()` or by running `generate_triple()` with a chosen threshold).
2. Instructs each honest party's local `presign()` call to use the lower threshold — honest parties have no stored keygen threshold to contradict this.
3. The presign validation passes because `args.threshold == triple.threshold` (both are the attacker-chosen `t'`).

No cryptographic break, leaked key, or external dependency failure is required. The attack is purely a consequence of missing cross-phase state.

---

### Recommendation

Store the threshold in `KeygenOutput`:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
    pub threshold: ReconstructionLowerBound,   // add this
}
```

Then enforce the orchestration invariant in `presign()`:

```rust
if args.threshold.value() < args.keygen_out.threshold.value() {
    return Err(InitializationError::BadParameters(
        "Presign threshold must be >= keygen threshold".to_string(),
    ));
}
```

Apply the same check in `sign()` by propagating the threshold through `PresignOutput` or `RerandomizedPresignOutput`.

---

### Proof of Concept

```
1. Run keygen with N=5, t=4.
   → Each party holds a share of a degree-3 polynomial; KeygenOutput stores no threshold.

2. Malicious coordinator calls deal(&mut rng, &participants, 2.into())
   → Generates triples with threshold=2.

3. Coordinator distributes triple shares to all 5 honest parties and instructs
   each to call presign() with threshold=2.

4. Each honest party calls:
     presign(&participants, me, PresignArguments {
         triple0: (share0, pub0),   // pub0.threshold == 2
         triple1: (share1, pub1),   // pub1.threshold == 2
         keygen_out,                // no stored threshold to compare
         threshold: 2.into(),       // matches triple threshold → passes validation
     })
   Presign succeeds. The resulting PresignOutput is a 2-threshold sharing of (k, k·x).

5. Coordinator assembles only 2 parties for signing:
     sign(&two_parties, coordinator, 2.into(), me, public_key, presig, msg_hash)
   → participants.len()=2 >= threshold=2 → passes.
   → Lagrange interpolation over 2 parties correctly reconstructs k and sigma.
   → Valid ECDSA signature produced.

6. Signature verifies against the master public key established at keygen,
   despite only 2 of the intended 4 threshold parties participating.
```

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }
