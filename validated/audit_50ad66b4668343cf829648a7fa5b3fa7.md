### Title
`FeeEstimation` Omits `l2_gas_consumed` While Exposing `l2_gas_price`, Returning an Authoritative-Looking Incomplete Fee Estimation Response - (File: crates/apollo_rpc_execution/src/objects.rs)

### Summary
`tx_execution_output_to_fee_estimation` computes the full `GasVector` (including `l2_gas`) from the transaction receipt but never writes `gas_vector.l2_gas` into the returned `FeeEstimation` struct. The struct exposes `l2_gas_price` but has no `l2_gas_consumed` field. Callers of `starknet_estimateFee` and `starknet_simulateTransactions` therefore receive a response from which the L2 gas component of the fee cannot be reconstructed, and from which the correct `l2_gas.max_amount` resource bound for a V3 transaction cannot be derived.

### Finding Description
In `tx_execution_output_to_fee_estimation`:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;

Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),      // ✓ written
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // ✓ written
    l1_data_gas_price,
    l2_gas_price,                                  // price present …
    // gas_vector.l2_gas is computed but NEVER written anywhere
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

`gas_vector.l2_gas` is read from the receipt but silently dropped. The `FeeEstimation` struct has no `l2_gas_consumed` field. A TODO comment in the struct definition acknowledges the omission:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
/// The L2 gas price for execution.
pub l2_gas_price: GasPrice,
```

The `overall_fee` field's own documentation compounds the problem:

```rust
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
pub overall_fee: Fee,
```

This formula is wrong for V3 transactions: `overall_fee` is taken directly from `receipt.fee`, which the blockifier computes as `l1_gas_cost + l1_data_gas_cost + l2_gas_cost`. The L2 gas component is silently baked into `overall_fee` but is invisible to the caller because neither the field nor the documentation exposes it.

This is the direct sequencer analog of the external bug: a computed value (`gas_vector.l2_gas`) is never written back into the output structure, so the output is internally inconsistent — `l2_gas_price` is present but the corresponding consumed quantity is absent.

### Impact Explanation
Every caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction receives a `FeeEstimation` that:

1. **Cannot be used to set correct resource bounds.** To submit a V3 transaction a wallet or SDK must supply `l2_gas.max_amount`. The only authoritative source for this value is `starknet_estimateFee`, but the response omits `l2_gas_consumed`. Callers must guess or over-provision, or their transactions will revert with `MaxGasAmountExceeded`.

2. **Returns an internally inconsistent authoritative-looking value.** The response includes `l2_gas_price` (implying L2 gas is relevant) but no `l2_gas_consumed`. A caller who reconstructs the fee as `gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price` will obtain a value strictly less than `overall_fee` for any transaction that consumed L2 gas, with no way to account for the discrepancy.

3. **Documents `overall_fee` with a wrong formula.** The struct comment states the fee equals `gas_consumed * gas_price + data_gas_consumed * data_gas_price`, which is false for V3 transactions.

This matches the allowed impact: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation
All V3 (`AllResources`) transactions consume L2 gas. V3 is the current standard transaction type. Any wallet, SDK, or dApp that calls `starknet_estimateFee` to size its `l2_gas.max_amount` resource bound is affected on every such call. No special conditions are required; the omission is unconditional.

### Recommendation
Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it in `tx_execution_output_to_fee_estimation`:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
    l1_data_gas_price,
    l2_gas_consumed: gas_vector.l2_gas.0.into(),   // add this
    l2_gas_price,
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

Update the `overall_fee` doc-comment to reflect the correct three-component formula.

### Proof of Concept
1. Submit any V3 invoke transaction to `starknet_estimateFee`.
2. Observe the response: `l2_gas_price` is non-zero, but no `l2_gas_consumed` field is present.
3. Compute `gas_consumed × l1_gas_price + data_gas_consumed × l1_data_gas_price`. The result is less than `overall_fee` by exactly `gas_vector.l2_gas × l2_gas_price`.
4. A wallet that uses the response to set `l2_gas.max_amount = 0` (the only value derivable from the response) will have its transaction rejected by the blockifier with `MaxGasAmountExceeded` at execution time.

```rust
pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}
```
