### Title
Unbounded `stakers` Vec Growth Enables Permanent Griefing of `get_stakers()` Consensus Function - (File: `src/staking/staking.cairo`)

### Summary
Every call to `stake()` appends the caller's address to `stakers: Vec<ContractAddress>`, but stakers are **never removed** from this Vec even after completing `unstake_action`. The `get_stakers()` consensus function iterates over the **full range** of this Vec on every invocation. An attacker can inflate the Vec by cycling through many fresh addresses — each staking the minimum amount and then unstaking — at a net cost of only transaction fees (the staked principal is recovered). Once the Vec is large enough, `get_stakers()` exceeds the gas limit and becomes permanently uncallable, preventing the consensus layer from determining the validator set.

### Finding Description

**Root cause — unbounded Vec growth:**

Every `stake()` call unconditionally pushes the caller's address onto `stakers`:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [1](#0-0) 

The storage declaration carries an explicit developer acknowledgement that entries are never removed:

```cairo
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [2](#0-1) 

`remove_staker()`, called during `unstake_action`, clears `staker_info`, the operational-address mapping, commission fields, and pool contracts — but never touches `stakers`: [3](#0-2) 

**Root cause — full-range iteration in `get_stakers()`:**

`get_stakers()` iterates over every element ever pushed, including all long-since-unstaked addresses:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [4](#0-3) 

Each inactive entry still costs a storage read (`staker_address_ptr.read()`) plus the `is_staker_active` check, so the gas cost scales linearly with the total number of addresses ever registered, not just the currently active ones.

**Attack path:**

1. Attacker controls N fresh addresses (Starknet accounts are cheap to create).
2. Each address calls `stake()` with `min_stake` → address is pushed to `stakers`.
3. Each address calls `unstake_intent()`, waits `exit_wait_window`, then calls `unstake_action()` → principal is returned.
4. `assert_staker_address_not_reused` prevents re-staking from the same address, so each iteration consumes one address permanently.
5. Net cost per iteration: two transaction fees (stake + unstake cycle); `min_stake` is fully recovered.
6. After enough iterations, `get_stakers()` exceeds the Starknet node's gas cap for `starknet_call` and reverts on every invocation.

**Why address reuse does not protect against this:**

`assert_staker_address_not_reused` checks that `staker_own_balance_trace` is empty for the caller, preventing a single address from being reused. However, it does nothing to limit the total number of *distinct* addresses that can be registered. The attacker simply uses a new address each time. [5](#0-4) 

### Impact Explanation

`get_stakers()` is the consensus-layer entry point that returns the full validator set (address, staking power, public key, peer ID) for a given epoch. The consensus layer calls this function off-chain to build the committee for each epoch. If the function exceeds the gas limit, the consensus layer cannot determine the validator set, which disrupts the protocol's ability to operate. This constitutes **unbounded gas consumption** leading to permanent griefing of a critical protocol function.

Impact classification: **Medium — Griefing / Unbounded gas consumption** (matches allowed scope).

### Likelihood Explanation

The attack requires only transaction fees; the staked principal is fully recovered. Starknet account creation is permissionless and inexpensive. The attacker does not need any privileged role, leaked key, or external dependency. The only friction is the `exit_wait_window` delay between each stake/unstake cycle, which slows but does not prevent the attack. Given the low cost and the explicit developer note acknowledging that stakers are never removed, this is a realistic griefing vector.

### Recommendation

1. **Remove stakers from the Vec on `unstake_action`**: Replace the `Vec` with a data structure that supports O(1) deletion (e.g., a swap-and-pop pattern using a `Map<index, address>` plus a length counter), and clear the entry in `remove_staker()`.

2. **Alternatively, maintain a separate active-staker count**: Track only active stakers in the Vec and use a tombstone/active flag to skip deleted entries without iterating over them, or rebuild the Vec to exclude inactive entries during iteration.

3. **Short-term mitigation**: Add a cap on the total number of registered staker addresses, or require a non-refundable registration fee to raise the cost of the attack.

### Proof of Concept

```
// Pseudocode — repeat N times with fresh address_i
for i in 0..N:
    address_i.approve(staking_contract, min_stake)
    staking_contract.stake(reward_address, operational_address_i, min_stake)
    // address_i is now in stakers Vec at index i
    staking_contract.unstake_intent()          // called from address_i
    advance_time(exit_wait_window)
    staking_contract.unstake_action(address_i) // min_stake returned
    // address_i remains in stakers Vec forever

// After N iterations:
staking_contract.get_stakers(epoch_id)
// → iterates over N entries, each requiring storage reads
// → eventually reverts with out-of-gas once N is large enough
```

The `stakers` Vec length after the attack equals N (all entries retained), while the number of *active* stakers may be zero. Every future call to `get_stakers()` pays the full O(N) iteration cost. [2](#0-1) [1](#0-0) [4](#0-3)
