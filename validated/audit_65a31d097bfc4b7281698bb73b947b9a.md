### Title
Cross-COA Signature Replay via Missing `evmAddress` Binding in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's keys signed `signedData`, and that the COA resource at the given `path` matches the supplied `evmAddress`. However, it never checks that `signedData` itself encodes `evmAddress`. Because the `COAOwnershipProof` struct (which carries `CapabilityPath`) is RLP-encoded but **not** covered by the signature, an attacker who obtains a valid proof for COA_A can re-encode it with COA_B's `CapabilityPath` and replay the same signatures to make COA_B's `isValidSignature` return `ValidERC1271Signature` — without the account owner ever authorizing that COA.

---

### Finding Description

**Root cause — `validateCOAOwnershipProof` does not bind `evmAddress` into the signed message.**

`fvm/evm/stdlib/contract.cdc` lines 1082–1086 perform the only cryptographic check:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // ← only this is signed
    domainSeparationTag: "FLOW-V0.0-user"
)
```

`signedData` is whatever the caller supplies; `evmAddress` and `path` are checked structurally (lines 1095–1105) but are **not** part of the signed payload. The code itself acknowledges this at lines 1003–1009:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

**The proof structure is mutable by an attacker.**

`COAOwnershipProof` (in `fvm/evm/types/proof.go` lines 139–144) is:

```go
type COAOwnershipProof struct {
    KeyIndices     KeyIndices
    Address        FlowAddress
    CapabilityPath PublicPath   // ← not signed
    Signatures     Signatures
}
```

It is RLP-encoded (`proof.go` line 147) and passed as the opaque `bytes` argument to `verifyCOAOwnershipProof`. Nothing prevents an attacker from decoding it, swapping `CapabilityPath`, and re-encoding it.

**End-to-end exploit path.**

The COA EVM contract (`fvm/evm/handler/coa/coa.sol` lines 114–125) exposes ERC-1271:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4)
{
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this),   // ← always the calling COA's EVM address
            _hash,
            _sig             // ← the attacker-controlled proof bytes
        )
    );
```

The precompile (`fvm/evm/precompiles/arch.go` lines 143–155) decodes the proof and calls `validateCOAOwnershipProof` via `fvm/evm/handler/precompiles.go` lines 115–152.

**Step-by-step attack (Alice owns COA_A at `/public/coaA` and COA_B at `/public/coaB`):**

1. Alice legitimately signs `_hash` for COA_A, producing a `COAOwnershipProof` with `CapabilityPath = "coaA"`.
2. Attacker RLP-decodes the proof, changes `CapabilityPath` to `"coaB"`, and re-encodes it.
3. Attacker calls `COA_B.isValidSignature(_hash, modified_proof)`.
4. `isValidSignature` calls `verifyCOAOwnershipProof(address(COA_B), _hash, modified_proof)`.
5. The precompile calls `validateCOAOwnershipProof` with `evmAddress = address(COA_B)`.
6. Inside `validateCOAOwnershipProof`:
   - Signatures over `_hash` are valid ✓ (Alice signed `_hash`)
   - COA at `/public/coaB` matches `address(COA_B)` ✓
7. Returns `ValidationResult(isValid: true)` → `isValidSignature` returns `0x1626ba7e`.

Alice never authorized `_hash` for COA_B; the attacker manufactured a valid ERC-1271 response for it.

---

### Impact Explanation

Any EVM contract or off-chain protocol that relies on `COA_B.isValidSignature` to confirm that the COA's Cadence owner authorized a specific hash will accept the replayed proof. Concrete consequences include:

- **Unauthorized ERC-1271 approvals**: DeFi protocols, NFT marketplaces, or multisig contracts that call `isValidSignature` on COA_B will treat the replayed signature as a valid authorization from COA_B's owner.
- **Cross-COA asset theft**: If COA_B holds EVM assets and a protocol uses ERC-1271 to gate withdrawals or transfers, the attacker can drain COA_B by replaying a signature Alice produced for COA_A.

The Cadence protocol places no restriction on the number of COAs a single account may own; `EVM.createCadenceOwnedAccount()` can be called multiple times and stored at arbitrary paths.

---

### Likelihood Explanation

- **Multiple COAs per account are protocol-supported**: `EVM.createCadenceOwnedAccount()` has no uniqueness guard; test code and production helpers already demonstrate accounts with two COAs.
- **Proof bytes are attacker-controlled**: The `_sig` parameter of `isValidSignature` is fully attacker-supplied; RLP re-encoding is trivial.
- **Signatures are often public**: ERC-1271 proofs are submitted on-chain or via APIs; once Alice's proof for COA_A is observed, the attacker can replay it against COA_B immediately.
- **`validateCOAOwnershipProof` is `access(all)`**: It can also be called directly from any Cadence script or transaction, bypassing the EVM layer entirely.

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. The simplest fix is to prepend (or hash-commit) the 20-byte `evmAddress` into the data that is cryptographically verified:

```cadence
// Reconstruct the expected signed payload
let expectedSignedData: [UInt8] = evmAddress.concat(signedData)

let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: expectedSignedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Callers must then sign `evmAddress || payload` rather than bare `payload`. This mirrors the EIP-4337 fix of including the `EntryPoint` address in the user-operation hash, ensuring a signature produced for one COA cannot be replayed against another.
