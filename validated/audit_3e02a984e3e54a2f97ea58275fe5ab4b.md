### Title
Unpermissioned `update_rewards` with attacker-controlled `disable_rewards` permanently freezes consensus yield for all stakers — (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is a public function with no access control. It accepts a caller-controlled `disable_rewards: bool` parameter. When called with `disable_rewards: true`, it advances the **global** `last_reward_block` checkpoint to the current block without distributing any rewards. Because `last_reward_block` is a single protocol-wide variable, an attacker can call `update_rewards(any_valid_staker, disable_rewards: true)` every block to permanently prevent all stakers from ever receiving consensus block rewards.

---

### Finding Description

`update_rewards` is exposed in the `IStakingRewardsManager` public interface with no role check or caller restriction. [1](#0-0) 

Inside the implementation, the function first asserts `current_block_number > last_reward_block`: [2](#0-1) 

It then **unconditionally** writes `last_reward_block = current_block_number` before checking `disable_rewards`: [3](#0-2) 

`last_reward_block` is a single global storage slot, not per-staker: [4](#0-3) 

The ordering is the root cause: the global checkpoint is consumed **before** the `disable_rewards` guard is evaluated. Any call — even one that distributes nothing — exhausts the reward slot for that block for every staker in the protocol.

The analog to the external report's vulnerability class is direct: in the reference report, `unstakeToken` performs a state-changing action (burning voting power, transferring an NFT) without verifying the caller owns the target position. Here, `update_rewards` performs a state-changing action (advancing `last_reward_block`) without verifying the caller is authorized to decide whether rewards should be distributed. In both cases, an unprivileged caller uses a public entry point to affect the financial outcome of other users.

---

### Impact Explanation

An attacker calling `update_rewards(valid_staker, disable_rewards: true)` once per block:

1. Advances `last_reward_block` to the current block with no reward distribution.
2. Every subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
3. Repeated every block, this **permanently freezes all consensus block rewards** for every staker and pool member in the protocol.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- Requires no special role, no stake, and no prior setup — only a valid staker address, which is publicly observable from `NewStaker` events.
- Starknet transaction fees are low; automating one call per block is economically trivial.
- The attack is fully permissionless and requires no coordination or front-running beyond submitting a transaction before the legitimate reward call.

---

### Recommendation

Restrict `update_rewards` to a trusted caller (e.g., the attestation contract or a dedicated rewards-manager role), mirroring the access control already applied to `update_rewards_from_attestation_contract`: [5](#0-4) 

At minimum, add a role check at the top of `update_rewards`, or remove the `disable_rewards` parameter from the public interface and derive the disable condition from on-chain state (e.g., whether the staker attested in the current epoch).

---

### Proof of Concept

```cairo
// Attacker (any address, no stake required) runs this bot every block:
loop {
    staking.update_rewards(
        staker_address: any_valid_staker,  // observable from NewStaker events
        disable_rewards: true,
    );
    // Effect:
    //   last_reward_block := current_block_number  (global, shared by all stakers)
    //   reward distribution branch: skipped
    // All legitimate update_rewards calls in this block now revert:
    //   assert!(current_block_number > last_reward_block)  => REWARDS_ALREADY_UPDATED
}
// Result: zero consensus block rewards distributed to any staker, indefinitely.
```
