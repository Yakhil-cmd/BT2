### Title
Unpermissioned `deploy_erc20_token` Allows Any Caller to Register Arbitrary NEP-141 Account IDs, Enabling Unbacked ERC-20 Token Minting - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `deploy_erc20_token` contract method lacks any caller access control. Any unprivileged NEAR account can call it with an arbitrary NEP-141 account ID (including one they control), causing the engine to deploy a real ERC-20 contract and permanently register the `nep141 → erc20` mapping in storage. Because `ft_on_transfer` unconditionally trusts `predecessor_account_id` as the token identifier and mints ERC-20 tokens whenever that predecessor is found in the registry, the attacker can subsequently call `ft_on_transfer` directly from their registered NEAR account to mint an unlimited quantity of the newly-deployed ERC-20 token with no backing assets.

---

### Finding Description

**Root cause — missing access control in `deploy_erc20_token`:**

`engine/src/contract_methods/connector.rs` lines 112–159 implement `deploy_erc20_token`. The only guard present is `require_running` (line 118), which only checks whether the contract is paused. There is no `require_owner_only` or any other caller restriction:

The `Legacy` variant takes the caller-supplied `nep141: AccountId` and immediately calls `engine::deploy_erc20_token`, which deploys an ERC-20 bytecode and writes the bidirectional mapping to NEAR storage via `nep141_erc20_map(self.io).insert(...)`: [2](#0-1) [3](#0-2) 

No on-chain verification is performed that the supplied `nep141` account actually exists or is a legitimate fungible token. The NEAR entry point in `engine/src/lib.rs` also applies no guard: [4](#0-3) 

**Exploitation path — minting via `ft_on_transfer`:**

`ft_on_transfer` (connector.rs lines 62–109) uses `predecessor_account_id` directly as the token identifier. If the predecessor is not the ETH-connector account, it calls `receive_erc20_tokens` with that predecessor as the NEP-141 token:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(&predecessor_account_id, &args, ...)
};
``` [5](#0-4) 

`receive_erc20_tokens` looks up the ERC-20 address for the predecessor in the registry and, if found, calls the ERC-20 `mint` function: [6](#0-5) 

Because the attacker registered their own NEAR account as the NEP-141 token in step 1, the lookup succeeds and an arbitrary amount of ERC-20 tokens is minted to any EVM address the attacker specifies.

**Contrast with `mirror_erc20_token`**, which correctly applies `require_owner_only` before proceeding: [7](#0-6) 

The comment on line 149 even acknowledges the intent: *"this transaction could be executed by the owner of the contract only"* — but that restriction was never enforced for `deploy_erc20_token`. [8](#0-7) 

---

### Impact Explanation

**Critical — Insolvency / Direct theft of funds.**

An attacker mints an unbounded quantity of an ERC-20 token on Aurora that has no backing NEP-141 collateral. Those tokens are fully valid ERC-20 tokens on Aurora and can be:

- Sold into any Aurora DEX liquidity pool, draining real assets from liquidity providers.
- Used as collateral in Aurora lending protocols, borrowing real assets against worthless tokens.
- Transferred to other EVM addresses, spreading the inflation.

The bridge's invariant — that every ERC-20 token on Aurora is backed 1:1 by a locked NEP-141 token — is permanently broken for every fake token the attacker registers.

---

### Likelihood Explanation

**High.** The attack requires only two sequential NEAR transactions from any account:

1. Call `deploy_erc20_token` with `DeployErc20TokenArgs::Legacy(<attacker_account>)`.
2. Call `ft_on_transfer` from `<attacker_account>` with an arbitrary `amount` and the attacker's EVM address as `msg`.

No special privileges, no leaked keys, no governance capture, and no third-party dependency is needed. The `deploy_erc20_token` entry point is publicly callable on mainnet (`aurora` contract, chain ID 1313161554). The only prerequisite is that the attacker controls a valid NEAR account ID, which costs a trivial amount of NEAR to create.

---

### Recommendation

Add `require_owner_only` (or a dedicated role check) immediately after `require_running` inside `deploy_erc20_token`, mirroring the pattern already used in `mirror_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?; // ADD THIS
        ...
    })
}
```

Alternatively, introduce a dedicated `TokenDeployer` role (as the TODO comment in `mirror_erc20_token` already suggests) so that a set of trusted bridge relayers — but not arbitrary accounts — can register new tokens. [9](#0-8)

  '{"args": <borsh(DeployErc20TokenArgs::Legacy("attacker.near"))>}' \

# Aurora now stores: nep141_erc20_map["attacker.near"] = <ERC20_ADDR>

  --accountId attacker.near

# → get_erc20_from_nep141("attacker.near") succeeds (registered in step 1)
# → ERC-20 mint(attacker_evm_addr, 1000000000000000000000) executes
# → Attacker holds 1e21 unbacked ERC-20 tokens on Aurora

    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

            engine.receive_base_tokens(&args)

            )

        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
