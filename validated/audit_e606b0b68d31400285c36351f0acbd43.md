### Title
Token Mapping Deletion Permanently Locks IBC Vouchers Escrowed at CRC21 Contract Address - (File: x/cronos/keeper/keeper.go, x/cronos/keeper/evm.go)

### Summary
When a Cronos admin removes a non-source (IBC/gravity) token mapping via `MsgUpdateTokenMapping` with an empty `Contract` field, IBC vouchers that users had previously converted to CRC21 tokens become permanently locked at the CRC21 contract's Cosmos-side escrow address. No recovery path exists for affected users.

### Finding Description
The conversion flow for non-source IBC tokens in `ConvertCoinFromNativeToCRC21` works as follows: the user's IBC vouchers are sent to `sdk.AccAddress(contract.Bytes())` (the CRC21 contract's Cosmos address acting as an escrow), and the user receives minted CRC21 tokens in their EVM wallet. [1](#0-0) 

The reverse path, `ConvertCoinFromCRC21ToNative`, begins with a mandatory lookup: [2](#0-1) 

If the mapping is absent, the function immediately returns an error and the escrowed IBC vouchers cannot be released.

The deletion path is reachable via `MsgUpdateTokenMapping` with an empty `Contract` field, which calls `DeleteExternalContractForDenom`: [3](#0-2) 

`DeleteExternalContractForDenom` removes both the `DenomToExternalContractKey` and the reverse `ContractToDenomKey` entries. For IBC tokens there is no auto-deployed fallback contract, so after deletion `GetDenomByContract` returns `found=false` for that contract address permanently. [4](#0-3) 

The same `GetDenomByContract` guard also blocks the EVM-hook path (`SendToIbcHandler.handle`) and the `IbcTransferCoins` path checks `GetContractByDenom` which is equally broken: [5](#0-4) [6](#0-5) 

There is no balance check before deletion and no mechanism for users to recover escrowed vouchers once the mapping is gone.

### Impact Explanation
IBC vouchers (real cross-chain assets) are permanently locked at `sdk.AccAddress(contract.Bytes())`. Every conversion and bridge-out path for those vouchers requires a live denom↔contract mapping. After deletion, all three exit paths (`ConvertCoinFromCRC21ToNative`, `SendToIbcHandler`, `IbcTransferCoins`) are blocked. This constitutes a permanent, irreversible loss of IBC vouchers for every user who had converted before the mapping was removed — matching the **High** impact class: *Permanent or long-lived inability for honest users to process valid bridge/conversion flows*, and **High**: *Corruption of token mappings/denom-contract binding with direct security impact*.

### Likelihood Explanation
Admin may legitimately remove a token mapping for valid operational reasons (e.g., the external CRC21 contract is found to be faulty, the project behind the token shuts down, or a migration to a new contract is planned). The protocol provides no safeguard — no balance check, no user-notification window, no grace period — before the mapping is deleted. Any user who converted IBC vouchers to CRC21 tokens before the deletion loses their funds silently.

### Recommendation
Before executing `DeleteExternalContractForDenom`, verify that the CRC21 contract's Cosmos-side escrow balance for the associated denom is zero:

```go
func (k Keeper) DeleteExternalContractForDenom(ctx sdk.Context, denom string) bool {
    contract, found := k.getExternalContractByDenom(ctx, denom)
    if !found {
        return false
    }
    // Guard: reject deletion if escrowed balance is non-zero
    escrowAddr := sdk.AccAddress(contract.Bytes())
    bal := k.bankKeeper.GetBalance(ctx, escrowAddr, denom)
    if bal.IsPositive() {
        return false // or return an error
    }
    // ... existing deletion logic
}
```

Alternatively, provide an admin-callable migration function that sweeps the escrowed balance back to a recovery address before the mapping is deleted, ensuring users can reclaim their IBC vouchers.

### Proof of Concept
1. Admin registers mapping: `ibc/ATOM` → `CRC21_contract` via `MsgUpdateTokenMapping`.
2. Alice calls `MsgConvertVouchers` with 100 `ibc/ATOM`. `ConvertCoinFromNativeToCRC21` sends 100 `ibc/ATOM` to `sdk.AccAddress(CRC21_contract.Bytes())` and mints 100 CRC21 tokens to Alice's EVM address.
3. Admin calls `MsgUpdateTokenMapping{Denom: "ibc/ATOM", Contract: ""}`. `RegisterOrUpdateTokenMapping` calls `DeleteExternalContractForDenom("ibc/ATOM")`, removing both KV entries.
4. Alice calls `send_to_ibc` on the CRC21 contract to bridge back. The EVM hook `SendToIbcHandler.handle` calls `GetDenomByContract(CRC21_contract)` → `found=false` → returns `"contract X is not connected to native token"`.
5. Alice tries `MsgTransferTokens` directly. `IbcTransferCoins` calls `GetContractByDenom("ibc/ATOM")` → `found=false` → returns `"coin ibc/ATOM is not supported"`.
6. Alice's 100 `ibc/ATOM` remain permanently locked at `sdk.AccAddress(CRC21_contract.Bytes())` with no recovery path.

func (k Keeper) DeleteExternalContractForDenom(ctx sdk.Context, denom string) bool {
	store := ctx.KVStore(k.storeKey)
	contract, found := k.getExternalContractByDenom(ctx, denom)
	if !found {
		return false
	}
	store.Delete(types.DenomToExternalContractKey(denom))
	deleteReverseIfOwned(store, contract, denom)
	if auto, found := k.getAutoContractByDenom(ctx, denom); found {
		bz := store.Get(types.ContractToDenomKey(auto.Bytes()))
		if len(bz) == 0 {
			store.Set(types.ContractToDenomKey(auto.Bytes()), []byte(denom))
		} else if existingDenom := string(bz); existingDenom != denom {
			if k.contractOwnedByDenom(ctx, existingDenom, auto) {
				// auto address is already owned by another denom; drop local auto mapping
				store.Delete(types.DenomToAutoContractKey(denom))
			} else {
				// stale reverse entry
				store.Set(types.ContractToDenomKey(auto.Bytes()), []byte(denom))
			}
		}
	}
	return true
}
