### Title
COA Sub-1e10 AttoFlow Remainder Permanently Locked Due to Minimum Withdrawal Threshold - (File: fvm/evm/stdlib/contract.cdc)

### Summary
`CadenceOwnedAccount.withdraw()` enforces a hard minimum of 1e10 attoFlow per withdrawal, mirroring the UFix64 precision floor of Cadence Flow Vaults. Because the EVM environment tracks balances at full 1e-18 FLOW (attoFlow) precision, a COA can legitimately accumulate a balance whose sub-1e10 attoFlow remainder can never be withdrawn. The remainder is permanently locked inside the COA with no recovery path.

### Finding Description
The EVM contract `CadenceOwnedAccount` in `fvm/evm/stdlib/contract.cdc` exposes a `withdraw(balance: Balance)` function. The underlying host function `newInternalEVMTypeWithdrawFunction` in `fvm/evm/impl/impl.go` enforces the check:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
```

`AttoFlowBalanceIsValidForFlowVault` in `fvm/evm/types/balance.go` is defined as:

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0  // bal >= 1e10
}
```

Any withdrawal request below 1e10 attoFlow panics with `ErrWithdrawBalanceRounding`. The only special case that bypasses this is an exact zero balance (`balance.isZero()`), which returns an empty vault.

A COA can accumulate a non-multiple-of-1e10 attoFlow balance through:

1. **EVM-to-EVM transfers at attoFlow precision**: Any EVM account or Solidity contract can `transfer()` an arbitrary attoFlow amount (e.g., 5 attoFlow) directly to the COA's EVM address. This is a standard EVM operation and requires no special privilege.
2. **Truncation during withdrawal**: The withdraw implementation at `impl.go` line 784 truncates the requested amount to the nearest 1e10 attoFlow:
   ```go
   value := new(big.Int).Div(amountValue.BigInt, types.UFixToAttoConversionMultiplier)
   amount := types.NewBalanceFromUFix64(cadence.UFix64(value.Uint64()))
   ```
   If a user requests to withdraw `X` attoFlow where `X mod 1e10 != 0`, only `floor(X / 1e10) * 1e10` attoFlow is actually deducted. The remainder `X mod 1e10` (up to 9,999,999,999 attoFlow) stays in the COA and can never be withdrawn.

Once the COA balance drops below 1e10 attoFlow, the only valid call is `withdraw(balance: Balance(attoflow: 0))`, which returns an empty vault without touching the actual balance. There is no "withdraw all" or "sweep dust" path.

### Impact Explanation
Any attoFlow amount between 1 and 9,999,999,999 (inclusive) that ends up as the sole remaining balance in a COA is permanently inaccessible to the owner. The funds are not burned or redistributed — they remain credited to the COA's EVM address but cannot be moved to a Cadence Flow Vault. The maximum locked amount per COA is 9,999,999,999 attoFlow ≈ 0.00000001 FLOW. While small per instance, the lock is permanent and the user has no recourse.

### Likelihood Explanation
The scenario is reachable by any unprivileged transaction sender:
- A user creates a COA and deposits FLOW.
- The user calls an EVM contract (e.g., a DeFi protocol, a yield contract, or any contract that computes interest/fees at attoFlow precision) that sends back a non-multiple-of-1e10 attoFlow amount to the COA.
- Alternatively, the user simply withdraws an amount that is not a multiple of 1e10 attoFlow (e.g., `1230000789912345678` attoFlow), triggering the truncation and leaving `9912345678` attoFlow permanently locked — as demonstrated in the existing test `"test coa withdraw with remainder truncation"`.

No admin access, leaked keys, or staked node compromise is required.

### Recommendation
Add a "withdraw all" path analogous to the mitigation suggested in M-03: if the requested balance equals the full COA balance and the full balance is below 1e10 attoFlow, allow the withdrawal to proceed by rounding up to 1e10 attoFlow (crediting the difference from the EVM native token vault) or by burning the dust amount and returning an empty vault. Alternatively, expose a dedicated `sweepDust()` function that clears any sub-1e10 attoFlow remainder by burning it, so users are not permanently locked out of their own funds.

### Proof of Concept

```
1. Alice creates a COA and deposits 1.00000001 FLOW
   (= 1_000_000_010_000_000_000 attoFlow, a multiple of 1e10).

2. An EVM contract sends 5 attoFlow to Alice's COA EVM address
   (standard EVM transfer, no privilege required).
   COA balance is now 1_000_000_010_000_000_005 attoFlow.

3. Alice calls coa.withdraw(balance: EVM.Balance(attoflow: 1_000_000_010_000_000_005)).
   - AttoFlowBalanceIsValidForFlowVault check passes (>= 1e10).
   - Truncation: floor(1_000_000_010_000_000_005 / 1e10) * 1e10
                = 1_000_000_010_000_000_000 attoFlow withdrawn.
   - Remaining COA balance: 5 attoFlow.

4. Alice calls coa.withdraw(balance: EVM.Balance(attoflow: 5)).
   - AttoFlowBalanceIsValidForFlowVault(5) returns false (5 < 1e10).
   - Transaction panics: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".

5. Alice's 5 attoFlow is permanently locked. No recovery path exists.
```
