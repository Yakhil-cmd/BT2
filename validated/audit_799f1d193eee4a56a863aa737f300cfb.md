### Title
Unpermissioned `exit_delegation_pool_action` Allows Any Caller to Force Pool Member Exit - (File: src/pool/pool.cairo)

### Summary
The `exit_delegation_pool_action` function in the Pool contract has no caller restriction. Any address can invoke it for any pool member once the exit window has elapsed. A pool member who signaled exit intent but later changed their mind can be forcibly ejected before they can cancel, causing them to lose their staking position and all future rewards they would have earned.

### Finding Description
In `src/pool/pool.cairo` at lines 295–333, `exit_delegation_pool_action` accepts a `pool_member` address parameter but performs **no check** that `get_caller_address() == pool_member`. The function only verifies that an exit intent exists and that the exit window has elapsed:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    let unpool_time = pool_member_info
        .unpool_time
        .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
    assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);
    // ... transfers unpool_amount to pool_member, no caller check
``` [1](#0-0) 

The spec explicitly documents this as intentional: "Any address can execute." [2](#0-1) 

A pool member can cancel a pending exit intent at any time by calling `exit_delegation_pool_intent(amount: 0)`, which sets `unpool_time = None` and `unpool_amount = 0`, restoring their full delegated balance:

```cairo
if amount.is_zero() {
    pool_member_info.unpool_time = Option::None;
}
pool_member_info.unpool_amount = amount;
let new_delegated_stake = total_amount - amount;
``` [3](#0-2) 

However, once the exit window elapses, a malicious actor can race the pool member's cancellation transaction by calling `exit_delegation_pool_action(pool_member)` first. On Starknet the sequencer controls transaction ordering, so an adversary who is also a pool member (and thus benefits from the victim's exit) can submit the forced-exit transaction to be ordered before the victim's cancellation.

The forced exit:
1. Calls `remove_from_delegation_pool_action` in the staking contract, pulling `unpool_amount` tokens into the pool contract.
2. Transfers `unpool_amount` to `pool_member` (the victim receives their principal back).
3. Zeroes out `unpool_amount` and `unpool_time` in the victim's `pool_member_info`. [4](#0-3) 

After the forced exit, the victim's delegated balance is zero. They earn no further staking rewards until they re-enter the pool via `enter_delegation_pool` or `add_to_delegation_pool`, paying gas again. The attacker, if also a pool member, now holds a larger proportional share of the pool and earns a greater fraction of all future rewards distributed via `update_rewards_from_staking_contract`. [5](#0-4) 

The analogous `unstake_action` in the staking contract is also callable by any address: [6](#0-5) 

However, stakers have no mechanism to cancel an unstake intent, so forcing `unstake_action` does not change the staker's outcome — they were committed to unstaking regardless. The pool-member case is the exploitable one because cancellation is possible.

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users (or with indirect profit motive for an attacker who is also a pool member).**

- The victim pool member receives their `unpool_amount` principal back, so there is no direct theft of at-rest funds.
- The victim loses all future staking rewards they would have earned had they cancelled and remained in the pool.
- The victim must pay gas to re-enter the pool.
- An attacker who is also a pool member gains a larger share of future pool rewards at the victim's expense, constituting indirect extraction of yield.

### Likelihood Explanation
- The function is publicly callable with no access control; no special role or privilege is required.
- The only precondition is that the target pool member has a pending exit intent whose window has elapsed — a normal, observable on-chain state.
- An attacker who is also a pool member has a direct financial incentive (increased reward share) to execute this attack.
- On Starknet, the sequencer controls transaction ordering, making it feasible for a motivated attacker to ensure their `exit_delegation_pool_action` call is ordered before the victim's `exit_delegation_pool_intent(0)` cancellation.

### Recommendation
Add a caller restriction to `exit_delegation_pool_action` so that only the pool member themselves (or optionally their reward address) can trigger the exit:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let caller = get_caller_address();
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::UNAUTHORIZED_EXIT_ACTION,
    );
    // ... rest of function
```

This mirrors the access control already applied to `claim_rewards` in the same contract: [7](#0-6) 

If permissionless execution is desired for UX reasons (e.g., allowing a protocol relayer to finalize exits on behalf of users), the protocol should document this trade-off explicitly and consider whether the griefing/incentive-manipulation risk is acceptable, as was discussed in the referenced Perennial audit.

### Proof of Concept

1. Alice (pool member) calls `exit_delegation_pool_intent(amount: full_balance)` — she wants to exit but later changes her mind.
2. The exit window elapses (e.g., 21 days pass).
3. Alice prepares a transaction calling `exit_delegation_pool_intent(amount: 0)` to cancel her exit and remain in the pool.
4. Bob (also a pool member in the same pool) observes Alice's pending cancellation and submits `exit_delegation_pool_action(pool_member: alice)` to the sequencer with higher priority or via sequencer collusion.
5. Bob's transaction executes first: Alice's `unpool_amount` is transferred to her address, her pool balance becomes zero, and her `unpool_time` is cleared.
6. Alice's cancellation transaction now reverts with `MISSING_UNDELEGATE_INTENT` because `unpool_time` is already `None`.
7. Alice has been forcibly ejected. Bob's share of all future pool rewards increases proportionally.

        fn exit_delegation_pool_action(
            ref self: ContractState, pool_member: ContractAddress,
        ) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let unpool_time = pool_member_info
                .unpool_time
                .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
            assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberExitAction {
                        pool_member, unpool_amount: pool_member_info.unpool_amount,
                    },
                );

            // Perform removal action in the staking contract, receiving funds if needed.
            // Note that if the intent was done after the staker was removed (unstake_action),
            // the funds will already be in the pool contract, and the following call will do
            // nothing.
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_action(identifier: pool_member.into());

            let unpool_amount = pool_member_info.unpool_amount;
            pool_member_info.unpool_amount = Zero::zero();
            pool_member_info.unpool_time = Option::None;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer delegated amount to the pool member.
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());

            unpool_amount
        }
