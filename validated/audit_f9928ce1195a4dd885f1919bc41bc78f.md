### Title
`execute_scheduled` is unnecessarily `#[payable]` — attached NEAR is permanently trapped in XCC Router - (File: etc/xcc-router/src/lib.rs)

---

### Summary
The `execute_scheduled` function in the XCC Router contract is marked `#[payable]`, allowing callers to attach NEAR tokens. However, the function never reads `env::attached_deposit()` and has no mechanism to refund or use the attached NEAR. Any NEAR sent with this call is permanently frozen in the router contract.

---

### Finding Description
In `etc/xcc-router/src/lib.rs`, the `execute_scheduled` function is explicitly documented as callable by **anyone** (not just the parent Aurora engine):

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The `#[payable]` attribute causes the NEAR runtime to accept any attached deposit without reverting. The function body, however, never calls `env::attached_deposit()` and performs no operation with the attached NEAR. The only other method that could recover NEAR from the router is `send_refund`, which transfers a fixed `REFUND_AMOUNT` (2 NEAR) to the parent and is restricted to the parent caller — it has no mechanism to recover arbitrary user-attached deposits.

The second `#[payable]` method, `deploy_upgrade`, is gated by `self.assert_preconditions()` which enforces `predecessor_account_id == self.parent` (the Aurora engine), so it is not reachable by an unprivileged user.

---

### Impact Explanation
Any NEAR attached to a call to `execute_scheduled` is permanently frozen inside the XCC router sub-account. There is no function in the router contract that allows recovering arbitrary deposited NEAR — `send_refund` only transfers a hardcoded `REFUND_AMOUNT` to the parent and is parent-restricted. The trapped NEAR cannot be recovered by the user or any permissionless actor.

**Impact class: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation
`execute_scheduled` is intentionally open to any caller. A user or contract interacting with the XCC router may attach NEAR (e.g., due to a wallet default, a scripting error, or a mistaken assumption that NEAR is needed to pay for the promise execution). Because the NEAR runtime silently accepts the deposit when `#[payable]` is present, no error is raised and the funds are silently lost. The likelihood is **medium**: the function is publicly callable and the `#[payable]` attribute provides no warning to callers.

---

### Recommendation
Remove the `#[payable]` attribute from `execute_scheduled`. Since the function does not use the attached deposit in any way, there is no reason to accept one. Removing `#[payable]` causes the NEAR runtime to automatically reject any call that includes an attached deposit, protecting users from accidental fund loss.

```rust
// Before (vulnerable):
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) { ... }

// After (fixed):
pub fn execute_scheduled(&mut self, nonce: U64) { ... }
```

---

### Proof of Concept

1. An XCC router sub-account (e.g., `<hex_address>.aurora`) has a scheduled promise stored via a prior `schedule` call.
2. Any unprivileged user calls `execute_scheduled` on that router with an attached deposit of, say, 1 NEAR.
3. Because the function is `#[payable]`, the NEAR runtime accepts the deposit without reverting.
4. The function body executes the scheduled promise and returns — `env::attached_deposit()` is never read.
5. The 1 NEAR is now held by the router sub-account with no permissionless recovery path: `send_refund` is parent-restricted and transfers only a fixed `REFUND_AMOUNT`, not the user's deposit.
6. The user's 1 NEAR is permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2)
