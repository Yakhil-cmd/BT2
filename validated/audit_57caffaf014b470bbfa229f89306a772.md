### Title
Missing Verification of Participant Share Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The Confidential Key Derivation (CKD) coordinator blindly accumulates `(big_y, big_c)` share contributions from all participants with no cryptographic verification that each contribution was honestly computed. A single malicious participant can send arbitrary group elements, silently corrupting the final `CKDOutput` and causing every honest party to accept an unusable or attacker-biased derived key.

### Finding Description

In `do_ckd_coordinator`, the coordinator sums every participant's `(norm_big_y, norm_big_c)` pair without any proof of correct computation:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute:
- `big_y = y_i * G` (ephemeral public key)
- `big_c = x_i * H(pk || app_id) + y_i * app_pk` (masked share)
- then normalize both by the Lagrange coefficient `λ_i` [2](#0-1) 

There is no zero-knowledge proof, commitment, or any other mechanism that forces a participant to use their actual private share `x_i` and a well-formed ephemeral scalar `y_i`. The coordinator has no way to detect a deviation.

The participant simply sends the pair and returns `None`:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
Ok(None)
``` [3](#0-2) 

This is structurally identical to the external report's finding: a mechanism that *should* enforce correctness (on-chain royalty enforcement / participant share verification) is entirely absent, so any participant can bypass it by supplying alternative, unverified values.

### Impact Explanation

The correct unmasked output is `msk * H(pk || app_id)`. If participant `j` sends arbitrary `(big_y', big_c')` instead of their honest contribution, the coordinator computes:

```
final_big_c = honest_sum + big_c'
final_big_y = honest_sum_y + big_y'
unmasked = final_big_c - app_sk * final_big_y
         = msk * H(pk || app_id) + (big_c' - app_sk * big_y')
```

Unless `big_c' = app_sk * big_y'` (which requires knowing `app_sk`), the unmasked result is wrong. The coordinator outputs a `CKDOutput` that is cryptographically invalid, and every honest party that relies on it receives a corrupted derived key. There is no mechanism to identify the culprit or recover.

This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

Any single participant in the CKD session is a sufficient attacker. The CKD protocol is N-of-N (all participants must contribute), so even one malicious node can corrupt every run. No special privilege beyond being a listed participant is required. The attack requires only sending two arbitrary group elements instead of the honestly computed ones — a trivial protocol deviation.

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` contribution with a non-interactive zero-knowledge proof of correct formation, e.g., a Schnorr-style proof that:
1. `big_y = y_i * G` for a known `y_i`, and
2. `big_c - x_i * H(pk || app_id) = y_i * app_pk` (i.e., the same `y_i` was used in both).

This is a standard "proof of correct ElGamal encryption" (Chaum-Pedersen proof). The coordinator must verify all proofs before aggregating contributions, and abort if any proof fails, identifying the malicious participant.

### Proof of Concept

1. Honest participants `P_1 … P_{n-1}` compute correct `(norm_big_y_i, norm_big_c_i)`.
2. Malicious participant `P_n` sends `(G, G)` (the generator point for both fields) instead of their honest contribution.
3. The coordinator sums all values including `(G, G)` with no error.
4. The resulting `CKDOutput` satisfies `big_c - app_sk * big_y ≠ msk * H(pk || app_id)`.
5. The client calls `ckd_output.unmask(app_sk)` and receives a garbage group element, making the derived confidential key permanently unusable for that session. [4](#0-3) [5](#0-4)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
