Audit Report

## Title
Unbounded `inbound_connecting` JoinSet Enables Pre-Authentication Resource Exhaustion via QUIC Connection Flood — (`rs/p2p/quic_transport/src/connection_manager.rs`)

## Summary
The QUIC transport connection manager spawns an unbounded Tokio task into `inbound_connecting` for every accepted `Incoming` QUIC connection before any TLS or subnet-membership authentication occurs. An off-subnet attacker who can send UDP packets to the replica's transport port and respond to Quinn's Retry packets (completing address validation) can accumulate an unbounded number of tasks, each holding live QUIC connection state for up to 10 seconds, exhausting heap memory or Tokio scheduler capacity and starving legitimate subnet peer connections from being processed.

## Finding Description
`inbound_connecting` is declared and initialized as a plain, unbounded `JoinSet` with no capacity constraint:
- Declaration at line 121: `inbound_connecting: JoinSet<Result<ConnectionWithPeerId, ConnectionEstablishError>>`
- Initialization at line 256: `inbound_connecting: JoinSet::new()`

The event loop's `select!` branch at lines 354–362 accepts every `Incoming` from `self.endpoint.accept()` unconditionally and immediately calls `self.handle_inbound_conn_attemp(incoming)` with no size check on `inbound_connecting` beforehand.

`handle_inbound_conn_attemp` (lines 538–588) unconditionally calls `self.inbound_connecting.spawn(timeout_conn_fut)` at line 587. All authentication — TLS handshake completion (`incoming.await` at line 543), certificate extraction, `node_id_from_certificate_der`, and the `peer_id > node_id` dialer check — occurs inside the spawned future, after the task is already in the JoinSet.

The only existing use of `inbound_connecting.len()` is at line 375 for metrics collection, not as a guard. The `EndpointConfig::default()` at line 207 sets no `max_incoming` limit. The TLS server config update in `handle_topology_change` (lines 403–411) restricts accepted certificates to subnet nodes, but this restriction is enforced during the TLS handshake inside the spawned task, not before spawning.

In Quinn's architecture, `endpoint.accept()` yields an `Incoming` after QUIC address validation (retry token exchange) but before the TLS handshake. An attacker with a real IP address can complete address validation by responding to Quinn's Retry packet, causing `Incoming` objects to be yielded and tasks to be spawned without possessing any valid subnet TLS certificate. Each spawned task holds pending QUIC connection state and lives for up to `CONNECT_TIMEOUT = 10s` (line 82) before timing out.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

The attack exploits a specific code flaw — the absence of any bound on `inbound_connecting` combined with deferred authentication — rather than raw packet flooding. Each connection attempt that completes QUIC address validation (requiring a real source IP and a valid Retry token response) causes a task to be spawned that holds heap-allocated QUIC connection state for up to 10 seconds. At a sustained rate, the JoinSet grows without bound, exhausting heap memory or saturating the Tokio task scheduler. This prevents the `select!` loop from processing legitimate inbound connections from subnet peers, directly stalling consensus peer connectivity and subnet availability on the affected replica.

## Likelihood Explanation
The replica's QUIC transport port is publicly reachable. The attacker requires no cryptographic material, no privileged access, and no key compromise — only the ability to send UDP packets and respond to Quinn's Retry packets (standard QUIC address validation). The attack is sustainable: new tasks are spawned faster than they time out at 10 seconds, so a modest connection rate maintains a large and growing JoinSet indefinitely. No subnet-majority corruption or governance control is needed.

## Recommendation
1. **Cap `inbound_connecting` before spawning**: In `handle_inbound_conn_attemp`, check `self.inbound_connecting.len()` against a hard limit (e.g., `2 × subnet_size` or a fixed constant such as 1000) before calling `self.inbound_connecting.spawn(...)`. If the limit is exceeded, call `incoming.refuse()` or drop the `Incoming` to reject the connection at zero task cost.
2. **Use `Incoming::refuse()`**: Quinn's `Incoming` type exposes a `refuse()` method that sends a QUIC `CONNECTION_CLOSE` frame without completing the handshake, cleanly rejecting excess connections before any task is spawned.
3. **Per-IP rate limiting**: Track connection attempts per source IP in the event loop before spawning and drop `Incoming` objects from IPs exceeding a per-second threshold.

## Proof of Concept
```
1. Attacker opens N UDP sockets targeting the replica's transport port.
2. For each socket, send a valid QUIC Initial packet (any or no TLS cert).
3. Respond to Quinn's Retry packet with the correct token to pass address validation.
4. endpoint.accept() yields Incoming → handle_inbound_conn_attemp spawns task into inbound_connecting with no size check.
5. Repeat at rate R > (tasks completing per second ≈ 1/CONNECT_TIMEOUT = 0.1/s per task).
6. inbound_connecting.len() grows without bound.
7. At sufficient task count: heap pressure causes OOM or Tokio scheduler starvation.
8. Legitimate subnet peer Incoming objects are accepted but their tasks are never scheduled
   → CONNECT_TIMEOUT fires → peers retry → consensus stalls on the affected replica.

Deterministic local test: write a Quinn client loop that opens connections at rate R,
responds to Retry tokens, and never completes the TLS handshake. Assert that
inbound_connecting.len() grows monotonically and that a legitimate peer connection
added after the flood takes longer than CONNECT_TIMEOUT to be processed.
```
