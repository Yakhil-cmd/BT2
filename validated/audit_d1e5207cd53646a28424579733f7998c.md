### Title
Multiblock Batch Proving Panics When Cumulative L2→L1 Logs Exceed `ArrayVec` Capacity — (`File: basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

---

### Summary

In the multiblock batch proving path, `ZKBatchDataKeeper.logs_storage` is a fixed-capacity `ArrayVec<Bytes32, 16384>`. Logs from each block are appended to this accumulator via `apply_to_array_vec`, which calls `ArrayVec::push` without a capacity check. Because the per-block log limit equals the accumulator's total capacity, any multiblock batch whose first block fills the log limit will cause an unconditional panic when the second block's logs are appended. An unprivileged user can trigger this by emitting L2→L1 logs up to the block limit.

---

### Finding Description

`ZKBatchDataKeeper` holds a batch-level log accumulator:

```rust
pub logs_storage: ArrayVec<Bytes32, 16384>,
``` [1](#0-0) 

Its capacity is exactly `16384`, matching `MAX_NUMBER_OF_LOGS`:

```rust
pub const MAX_NUMBER_OF_LOGS: u64 = 16_384;
``` [2](#0-1) 

After each block is processed in the multiblock batch proving path, the block's logs are appended to the batch accumulator:

```rust
io.logs_storage
    .apply_to_array_vec(&mut batch_data.logs_storage);
``` [3](#0-2) 

`apply_to_array_vec` iterates every log and calls `array_vec.push(log.hash())` with no capacity guard:

```rust
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())
    });
}
``` [4](#0-3) 

`ArrayVec::push` panics unconditionally when the array is at capacity. The per-block limit check in `check_for_block_limits` only enforces that a single block does not exceed `MAX_NUMBER_OF_LOGS`:

```rust
} else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
    Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
``` [5](#0-4) 

This check is per-block and does not account for cumulative batch-level log count. Because the batch accumulator has the same capacity as the per-block maximum, a batch containing two or more blocks where the first block uses the full `16384`-log allowance will overflow the `ArrayVec` when the second block's logs are appended, causing a panic in the prover.

---

### Impact Explanation

The panic occurs inside the proving system's post-tx operation for multiblock batches. This makes the batch **unprovable**: the RISC-V prover aborts, the batch cannot be finalized, and the chain halts at that batch boundary. This is a **valid-execution unprovability** issue — the sequencer accepted and executed the transactions correctly, but the prover cannot produce a proof for the resulting batch. The chain is effectively stalled until the operator manually intervenes.

---

### Likelihood Explanation

The L1 Messenger system hook is callable by any unprivileged user via a standard EVM transaction to address `0x8008` with `sendToL1(bytes)` calldata. A single block can hold up to `16384` logs. An attacker needs only to fill one block to its log limit (which is a known, public constant), then ensure a second block with at least one log is included in the same multiblock batch. This requires no privileged access, no oracle manipulation, and no governance majority. The cost is bounded by the gas required to emit `16384` L2→L1 log events.

---

### Recommendation

Add a capacity check inside `apply_to_array_vec` before pushing, or change `batch_data.logs_storage` to a dynamically-sized structure. At minimum, replace the unchecked `push` with a checked variant and propagate an error if the batch accumulator is full:

```rust
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.try_push(log.hash())
            .expect("batch logs_storage overflow: increase capacity or enforce batch-level log limit");
    });
}
```

A more robust fix is to enforce a batch-level log limit (e.g., `MAX_NUMBER_OF_LOGS` × max blocks per batch) and size `logs_storage` accordingly, or switch to a heap-allocated `Vec` for the batch accumulator.

---

### Proof of Concept

1. Deploy a contract that calls the L1 Messenger hook (`address(0x8008)`) in a loop, emitting `sendToL1(bytes)` messages.
2. Submit transactions that fill block B₁ with exactly `16384` L2→L1 logs (staying within `check_for_block_limits`).
3. Submit one more transaction in block B₂ (same multiblock batch) that emits at least one L2→L1 log.
4. When the prover processes the multiblock batch and calls `post_op` for B₂, `apply_to_array_vec` attempts to push into a full `ArrayVec<Bytes32, 16384>`, triggering an unconditional panic.
5. The batch cannot be proven; the chain halts. [1](#0-0) [6](#0-5) [3](#0-2)
