### Title
Fixed 2300-Gas Stipend in `DefaultGasLimitForTokenTransfer` Causes Permanent Deposit/Transfer Failure for Smart Contracts with Complex Receive/Fallback Logic - (File: `fvm/evm/types/call.go`)

### Summary

`fvm/evm/types/call.go` defines `DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300` (i.e., 21,000 + 2,300 = 23,300 gas). This constant is used as the hard-coded gas limit for all three bridge-level token movement primitives: `NewDepositCall`, `NewWithdrawCall`, and `NewTransferCall`. Any EVM smart contract whose `receive()` or `fallback()` function consumes more than 2,300 gas will permanently fail to receive FLOW tokens via the bridge deposit path, causing the enclosing Cadence transaction to revert.

### Finding Description

In `fvm/evm/types/call.go` lines 31–37, the protocol defines:

```go
// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer
```

`NewDepositCall` (line 194–210), `NewWithdrawCall` (line 213–229), and `NewTransferCall` (line 231–247) all use this constant as their `GasLimit`. These are the only gas limits available for the bridge-level direct calls — callers cannot override them.

When `Account.Deposit` is invoked in `handler.go` (line 965–975), it constructs a `NewDepositCall` with `DepositCallGasLimit` and calls `executeAndHandleCall`. Inside `emulator.go`, `mintTo` (line 381–425) runs the call via `proc.run`. If the recipient EVM contract's `receive()` or `fallback()` function consumes more than 2,300 gas, `gethCore.ApplyMessage` returns `ErrExecutionReverted`, `res.Failed()` is true, the state is reset (line 418), and the failed result is returned. Back in `handler.go`, `panicOnErrorOrInvalidOrFailedState` panics, reverting the entire Cadence transaction.

The same applies to `Account.Transfer` (handler.go line 999–1010) via `NewTransferCall`.

The 2,300 gas stipend is the same fixed value used by Solidity's deprecated `.send()` and `.transfer()` — it is sufficient only for a simple ETH/FLOW receipt with no state changes. Any contract that emits an event, writes to storage, or calls another contract in its `receive()` function will exceed this limit.

### Impact Explanation

Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function consumes more than 2,300 gas is permanently unable to receive FLOW tokens via the bridge `Deposit` path or the COA `Transfer` path. This is a cross-VM asset routing failure: the contract is designed to accept tokens, but the protocol-level gas cap silently prevents it. The Cadence transaction reverts, and there is no caller-side mechanism to increase the gas limit for these direct calls. DeFi contracts that perform minimal bookkeeping (e.g., updating an internal balance mapping, emitting a `Received` event) in their `receive()` function are affected.

### Likelihood Explanation

Any unprivileged user can deploy an EVM smart contract on Flow EVM. A contract with a `receive()` function that writes one storage slot or emits one event easily exceeds 2,300 gas (a single `SSTORE` costs 20,000 gas cold; a `LOG1` with 32 bytes costs ~750 gas). This is a standard and common pattern in EVM contracts. The condition is trivially reachable by any contract author without any special privileges.

### Recommendation

Replace the fixed `2_300` gas stipend with a configurable or sufficiently large gas limit for deposit and transfer direct calls, similar to how `NewContractCall` and `NewDeployCall` accept a caller-supplied `gasLimit`. At minimum, `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` should be raised to a value that accommodates typical `receive()` implementations (e.g., 100,000 gas), or the callers of `Account.Deposit` and `Account.Transfer` should be allowed to specify the gas limit.

### Proof of Concept

1. Deploy the following Solidity contract on Flow EVM:
```solidity
contract LoggingReceiver {
    event Received(address sender, uint256 amount);
    receive() external payable {
        emit Received(msg.sender, msg.value); // ~750+ gas for LOG1
        // Any SSTORE would cost 20,000+ gas cold
    }
}
```
2. From a Cadence transaction, call `EVM.deposit(from: <-vault)` targeting the deployed contract's address (via `EVMAddress.deposit` in `contract.cdc` line 563–565, which calls `InternalEVM.deposit`, which calls `Account.Deposit` in `handler.go` line 959–976).
3. The `NewDepositCall` is constructed with `DepositCallGasLimit = 23_300`. After the 21,000 intrinsic cost, only 2,300 gas remains for the `receive()` body. The `LOG1` emission alone may exhaust this budget.
4. `mintTo` in `emulator.go` line 416 detects `res.Failed()`, resets state, and returns the failed result.
5. `panicOnErrorOrInvalidOrFailedState` in `handler.go` line 975 panics, reverting the Cadence transaction.
6. The contract can never receive FLOW tokens via the bridge deposit path regardless of how many times the operation is retried.

// and update the account balance with the new amount
func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}

// Withdraw deducts the balance from the account and
// withdraw and return flow token from the Flex main vault.
func (a *Account) Withdraw(b types.Balance) *types.FLOWTokenVault {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMWithdraw).End()

	res, err := a.executeAndHandleAuthorizedCall(
		types.NewWithdrawCall(
			a.fch.addressAllocator.NativeTokenBridgeAddress(),
			a.address,
			b,
			a.Nonce(),
		),
		b,
		true,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)

	return types.NewFlowTokenVault(b)
}

func (proc *procedure) mintTo(
	call *types.DirectCall,
) (*types.Result, error) {
	// check and convert value
	value, isValid := checkAndConvertValue(call.Value)
	if !isValid {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrInvalidBalance,
		), nil
	}

	// create bridge account if not exist
	bridge := call.From.ToCommon()
	if !proc.state.Exist(bridge) {
		proc.state.CreateAccount(bridge)
	}

	// add balance to the bridge account before transfer
	proc.state.AddBalance(bridge, value, gethTracing.BalanceIncreaseWithdrawal)
	// check state errors
	if err := proc.state.Error(); err != nil {
		return nil, err
	}

	// withdraw the amount and move it to the bridge account
	res, err := proc.run(call.Message(), call.Hash(), types.DirectCallTxType)
	if err != nil {
		return res, err
	}

	// if any error (invalid or vm) on the internal call, revert and don't commit any change
	// this prevents having cases that we add balance to the bridge but the transfer
	// fails due to gas, etc.
	if res.Invalid() || res.Failed() {
		// reset the state to revert the add balances
		proc.state.Reset()
		return res, nil
	}

	// commit and finalize the state and return any stateDB error
	res.StateChangeCommitment, err = proc.commit(true)
	return res, err
}
