### Title
Scheduled XCC Promise Removed Before Execution With No Failure Callback, Permanently Locking User NEAR - (`etc/xcc-router/src/lib.rs`)

### Summary

The XCC router's `execute_scheduled` function removes a scheduled promise from storage **before** executing it and attaches no failure callback. If the underlying NEAR promise fails, the NEAR that was already unwrapped from the user's wNEAR ERC-20 balance is refunded to the router account with no mechanism for the user to recover it.

### Finding Description

When a user calls the XCC precompile with `CrossContractCallArgs::Delayed`, the engine:

1. Burns the user's wNEAR ERC-20 tokens immediately (in the same EVM transaction).
2. Schedules a `withdraw_wnear_to_router` call that unwraps the wNEAR into actual NEAR and deposits it into the router sub-account.
3. Calls `schedule()` on the router, which stores the user's promise in `scheduled_promises`.

Later, `execute_scheduled` is called (by anyone — this is intentional per the comment):

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {  // ← removed first
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);  // ← then executed
    env::promise_return(promise_id);                 // ← no failure callback
}
``` [1](#0-0) 

The promise is **removed from `scheduled_promises` before execution**. If the promise fails (e.g., target contract panics, method does not exist, insufficient gas at the target), NEAR Protocol refunds the attached deposit back to the predecessor of the receipt — which is the router account itself. The promise is already gone from storage, so it cannot be retried.

The router contract exposes no function to withdraw arbitrary NEAR amounts. The only egress is `send_refund`, which:
- Requires the caller to be the parent (Aurora engine, not the user)
- Requires **no failed promises** (`require_no_failed_promises()`)
- Sends only a fixed `REFUND_AMOUNT` of 2 NEAR back to the engine [2](#0-1) 

This means the NEAR attached to the failed promise is permanently trapped in the router sub-account.

The analog to M-27 is direct: in M-27, the settlement owner is `address(0)` so retry/redeem is blocked; here, the promise record is deleted before execution so retry is impossible, and there is no refund path for the user's NEAR.

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any NEAR attached to a scheduled XCC promise (sourced from the user's wNEAR ERC-20 balance, which is burned at scheduling time) is permanently locked in the router sub-account if the promise execution fails. The user loses both their wNEAR ERC-20 tokens (burned) and the NEAR (stuck in the router). There is no recovery path. [3](#0-2) 

### Likelihood Explanation

**Medium.** Any EVM user who uses `CrossContractCallArgs::Delayed` with a non-zero `attached_balance` is exposed. Failure conditions are realistic: the target NEAR contract may not exist, may panic, may not have the called method, or may run out of gas. Additionally, `execute_scheduled` is callable by anyone, so a third party can trigger execution at an inopportune time (e.g., before the target contract is ready). [4](#0-3) 

### Recommendation

`execute_scheduled` should attach a failure callback that re-stores the promise (or refunds the NEAR to the user's router account) if the promise fails. Concretely:

1. Use `env::promise_then` to attach a callback to the created promise.
2. In the callback, check `env::promise_result(0)`. If it failed, either re-insert the promise into `scheduled_promises` (to allow retry) or transfer the refunded NEAR back to the originating EVM address via the parent engine.

Alternatively, do not remove the promise from `scheduled_promises` until a success callback confirms execution succeeded.

### Proof of Concept

1. User holds wNEAR ERC-20 tokens on Aurora.
2. User calls the XCC precompile with `CrossContractCallArgs::Delayed(PromiseArgs::Create(promise))` where `promise.attached_balance = 1_000_000_000_000_000_000_000_000` (1 NEAR) and `promise.target_account_id` is a non-existent NEAR account.
3. The engine burns 1 NEAR worth of wNEAR ERC-20 from the user, unwraps it to NEAR in the router, and stores the promise at nonce 0.
4. Anyone calls `router.execute_scheduled({"nonce": "0"})`.
5. The promise is removed from `scheduled_promises`. The call to the non-existent account fails. NEAR Protocol refunds 1 NEAR to the router account.
6. The user's wNEAR ERC-20 balance is 0 (burned in step 3). The promise is gone from storage (removed in step 4). The router has no function to return the 1 NEAR to the user. Funds are permanently frozen. [1](#0-0) [5](#0-4)

fn require_no_failed_promises() -> Result<(), Error> {
    let num_promises = env::promise_results_count();
    for index in 0..num_promises {
        // We can use deprecated `promise_result` rather than `promise_result_checked` safely here,
        // because the promise result could be received from the Aurora Engine itself,
        // and we can be sure that the len of the promise result is within bounds.
        #[allow(deprecated)]
        if env::promise_result(index) == PromiseResult::Failed {
            return Err(Error::CallbackOfFailedPromise);
        }
    }
    Ok(())
}
