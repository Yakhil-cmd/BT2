### Title
Pool Member's Reward Address Cannot Signal Exit Intent Despite Having Delegation Management Rights - (File: `src/pool/pool.cairo`)

---

### Summary

`exit_delegation_pool_intent` in `pool.cairo` restricts callers to only the pool member address, while the sibling functions `add_to_delegation_pool` and `claim_rewards` explicitly permit the pool member's `reward_address` as an authorized caller. This inconsistency means the `reward_address` — which the protocol treats as a co-manager of the pool position — cannot initiate withdrawal of delegated funds, leading to a potential permanent freeze of those funds.

---

### Finding Description

The pool contract defines three state-mutating functions that act on a pool member's position. Two of them explicitly allow either the pool member or their registered `reward_address` to call them:

**`add_to_delegation_pool`** — allows `pool_member` OR `reward_address`: [1](#0-0) 

**`claim_rewards`** — allows `pool_member` OR `reward_address`: [2](#0-1) 

However, **`exit_delegation_pool_intent`** derives the pool member identity exclusively from `get_caller_address()` with no fallback to `reward_address`: [3](#0-2) 

The interface NatSpec confirms this restriction is intentional but inconsistent: [4](#0-3) 

The `reward_address` is not merely a passive recipient — the protocol explicitly grants it the ability to increase the pool member's delegated stake (a capital-committing action). Granting the right to add funds while denying the right to exit is an asymmetric authorization model that leaves the `reward_address` unable to protect or recover the pool member's capital.

---

### Impact Explanation

If the pool member's private key is lost or permanently inaccessible, the `reward_address` — the only other address with recognized authority over the position — cannot call `exit_delegation_pool_intent`. Since `exit_delegation_pool_action` requires a prior intent (enforced by `MISSING_UNDELEGATE_INTENT`), and no other recovery path exists in the contract, the delegated funds are permanently frozen in the staking contract with no on-chain mechanism to retrieve them.

**Impact: High — Permanent freezing of funds (delegated stake).** [5](#0-4) 

---

### Likelihood Explanation

A pool member using a hot wallet as their `pool_member` key and a cold wallet or multisig as their `reward_address` is a standard operational security pattern. Key loss or compromise of the hot wallet is a realistic event. The `reward_address` being unable to exit is not a hypothetical edge case — it is a direct consequence of the current access control model for any such setup.

---

### Recommendation

Modify `exit_delegation_pool_intent` to accept calls from either the pool member or their registered `reward_address`, consistent with `add_to_delegation_pool` and `claim_rewards`:

```cairo
fn exit_delegation_pool_intent(ref self: ContractState, amount: Amount) {
    // Resolve pool_member: caller is either pool_member or their reward_address.
    let caller_address = get_caller_address();
    // Try caller as pool_member first; if not found, check if caller is a reward_address.
    let pool_member = ... // resolve pool_member from caller or explicit parameter
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller_address == pool_member || caller_address == pool_member_info.reward_address,
        "{}",
        Error::CALLER_CANNOT_EXIT_POOL,
    );
    ...
}
```

Note: Because `exit_delegation_pool_intent` currently uses `get_caller_address()` as the pool member identity (not a parameter), the signature may need to be updated to accept an explicit `pool_member: ContractAddress` parameter (as `add_to_delegation_pool` does), so the `reward_address` can specify which member it is acting on behalf of.

---

### Proof of Concept

1. Alice (`pool_member`) delegates 1000 STRK to a pool, registering Bob (`reward_address`) as her reward address.
2. Bob successfully calls `add_to_delegation_pool(pool_member: Alice, amount: 500)` — this succeeds because `reward_address` is authorized. [6](#0-5) 
3. Bob successfully calls `claim_rewards(pool_member: Alice)` — this succeeds because `reward_address` is authorized. [7](#0-6) 
4. Alice's private key is lost.
5. Bob attempts to call `exit_delegation_pool_intent(amount: 1500)` to recover the funds. The call **reverts** because `get_caller_address()` returns Bob's address, which does not match any stored `pool_member` key. [8](#0-7) 
6. No other address can call `exit_delegation_pool_intent` on Alice's behalf. The 1500 STRK are permanently frozen.
