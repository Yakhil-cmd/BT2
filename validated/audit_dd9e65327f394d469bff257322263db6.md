Audit Report

## Title
Ed25519 Non-Canonical Identity Element Encoding Bypasses Signature Forgery Protection — (`rs/crypto/internal/crypto_lib/basic_sig/ed25519/src/api.rs`)

## Summary

`verify_public_key()` checks only `is_torsion_free()` and omits `is_canonical()`. The three non-canonical encodings of the identity element are torsion-free but non-canonical, so they pass this guard. Because `verify_signature()` under ZIP215 multiplies the challenge scalar by the identity point (yielding zero), any `(R = S·G, S)` pair satisfies the verification equation for these keys. The ingress validation path never calls `verify_public_key()` at all; it calls `verify_basic_sig_by_public_key()` directly, which also performs no canonicality check. An unprivileged attacker can therefore forge a valid ingress signature for the three principals derived from these encodings.

## Finding Description

**`verify_public_key()` missing `is_canonical()` check:**

`verify_public_key()` calls only `pk.is_torsion_free()`: [1](#0-0) 

The `ic_ed25519` library explicitly tests and documents that the three non-canonical identity encodings are torsion-free but non-canonical: [2](#0-1) 

`is_canonical()` is defined as a round-trip check: [3](#0-2) 

**Signature verification with the identity element:**

`verify_signature()` computes `recomputed_r = k * minus_a + S * G`. When `A` is the identity, `minus_a = 0`, so `recomputed_r = S * G` regardless of `k`. The check `(recomputed_r - R) * cofactor == identity` passes for any `R = S * G`: [4](#0-3) 

This is confirmed by the ZIP215 test vectors, which include the canonical identity key paired with many distinct signatures, all of which pass: [5](#0-4) 

**`verify()` performs no canonicality check:**

`verify()` calls only `deserialize_raw()` (which succeeds for non-canonical identity encodings) and then `verify_signature()`: [6](#0-5) 

**Ingress validation path:**

`validate_user_id_and_signature()` calls `validate_user_id()` (which only checks `sender == PrincipalId::new_self_authenticating(sender_pubkey)`) and then `validate_signature()`: [7](#0-6) 

`validate_signature()` calls `public_key_from_bytes()` → `user_public_key_from_bytes()`, which calls `ic_ed25519::PublicKey::deserialize_rfc8410_der()` — no canonicality check: [8](#0-7) 

Then `validate_signature_plain()` → `verify_basic_sig_by_public_key()` calls `ic_ed25519::PublicKey::deserialize_raw()` and `pk.verify_signature()` — again no canonicality check: [9](#0-8) 

**Contrast with the threshold-sig group module**, which explicitly rejects all three non-canonical identity encodings before any further processing: [10](#0-9) 

## Impact Explanation

An attacker can authenticate as any of the three principals derived from the non-canonical identity encodings without possessing any private key. This constitutes unauthorized access to wallets, identities, and canister-controlled funds for those principals. The attack is constrained to exactly three specific principals (one per non-canonical encoding), which limits the scope relative to a universal forgery. This matches the **High ($2,000–$10,000)** impact class: "Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."

## Likelihood Explanation

The attack requires no privileged access, no key material, no network-level attack, and no social engineering. The three target encodings are publicly documented in the codebase itself. Computing `R = S·G` for any `S` is a single scalar multiplication. The exploit is fully local, deterministic, and repeatable. The only constraint is that the targets are three specific principals.

## Recommendation

Add an `is_canonical()` check inside `verify_public_key()`:

```rust
pub fn verify_public_key(pk: &types::PublicKeyBytes) -> bool {
    if let Ok(pk) = ic_ed25519::PublicKey::deserialize_raw(&pk.0) {
        pk.is_torsion_free() && pk.is_canonical()
    } else {
        false
    }
}
```

Additionally, add a canonicality check in `verify_basic_sig_by_public_key()` in `rs/crypto/standalone-sig-verifier/src/lib.rs` for the `AlgorithmId::Ed25519` branch, since the ingress path bypasses `verify_public_key()` entirely and calls this function directly.

## Proof of Concept

```rust
use hex_literal::hex;

// Non-canonical identity encoding #2
let non_canonical_identity: [u8; 32] =
    hex!("eeffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff7f");

// DER-wrap it
let der_pubkey = ic_ed25519::PublicKey::convert_raw32_to_der(non_canonical_identity);

// Derive the principal the attacker will impersonate
let principal = PrincipalId::new_self_authenticating(&der_pubkey);
let sender = UserId::from(principal);

// Forge a signature: pick S=1, R = 1·G (base point)
let s_scalar = Scalar::ONE;
let r_point = ED25519_BASEPOINT_POINT * s_scalar;
let mut sig = [0u8; 64];
sig[..32].copy_from_slice(&r_point.compress().0);
sig[32..].copy_from_slice(s_scalar.as_bytes());

// Submit ingress message with sender=principal, sender_pubkey=der_pubkey,
// signature=sig — all validation passes, ingress is accepted.
```

A deterministic integration test can be written using PocketIC: construct an `HttpRequest<SignedIngressContent>` with the above parameters and assert that `HttpRequestVerifierImpl::validate_request()` returns `Ok(...)`.
