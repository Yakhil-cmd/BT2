### Title
No L2 sequencer uptime check in `PriceProviderL2` and `ProtectedPriceProviderL2` allows stale oracle prices to reach pool swaps after sequencer restart — (`smart-contracts-poc/contracts/PriceProviderL2.sol`, `smart-contracts-poc/contracts/ProtectedPriceProviderL2.sol`)

---

### Summary

`PriceProviderL2` and `ProtectedPriceProviderL2` are explicitly designed for L2 deployment (they carry a `FUTURE_TOLERANCE` immutable to handle L2 clock skew) but contain no check against a Chainlink L2 sequencer uptime feed. The sole protection against stale oracle data is a `_isStale` comparison of the oracle's `refTime` against `block.timestamp`. When the L2 sequencer goes offline, oracle keepers cannot push new price data on-chain. When the sequencer restarts, there is a window — bounded by `MAX_TIME_DELTA` (up to 7 days) — during which the on-chain oracle price is stale relative to the current market but still passes the staleness check. Pool swaps during this window execute at prices that no longer reflect market reality, causing direct LP losses.

---

### Finding Description

Both L2 price providers share the same staleness guard:

```solidity
// PriceProviderL2.sol _isStale (lines 135–150)
function _isStale(
    uint256 refTime,
    uint256 nowTs,
    uint256 maxDelta,
    uint256 futureTol
) internal pure returns (bool) {
    if (refTime == 0) return true;
    if (refTime > nowTs) {
        return (refTime - nowTs) > futureTol;
    }
    return (nowTs - refTime) > maxDelta;
}
```

This check only measures the age of the last oracle push (`refTime`). It has no knowledge of whether the L2 sequencer was recently restarted.

**Attack / failure path:**

1. The L2 sequencer goes offline. `block.timestamp` stops advancing on L2; oracle keepers cannot push new price data.
2. The market price of the pool's asset moves significantly (e.g., +15%) during the outage.
3. The sequencer restarts. `block.timestamp` jumps forward by the outage duration `T`.
4. The on-chain oracle price still reflects the pre-outage value; its `refTime` is `T` seconds old.
5. If `T < MAX_TIME_DELTA`, `_isStale` returns `false` — the stale price passes.
6. `getBidAndAskPrice()` returns bid/ask computed from the pre-outage price.
7. Swaps execute at the stale price. LPs who hold the appreciating asset sell it at the old (lower) price, suffering a direct loss proportional to the price move.

The `FUTURE_TOLERANCE` immutable (lines 37–38 of `PriceProviderL2.sol`) shows the developers are aware of L2 timing subtleties (sequencer clock skew), but it only tolerates oracle timestamps slightly *ahead* of `block.timestamp` — it does not address the post-restart stale-price window.

`MAX_TIME_DELTA` is set at construction and is bounded only by `(0, 7 days]`:

```solidity
// PriceProviderL2.sol constructor, line 92
if (_maxTimeDelta == 0 || _maxTimeDelta > 7 days) revert MaxTimeDeltaOutOfBounds();
```

`PriceProviderFactoryL2.createPriceProvider` is permissionless and passes `_maxTimeDelta` directly to the constructor without any tighter cap, so any caller can deploy a provider with a multi-hour or multi-day staleness window.

The same structural gap exists in `ProtectedPriceProviderL2._computeBidAsk` (line 207) and in the L1 variants (`PriceProvider`, `ProtectedPriceProvider`) — but the L1 variants are not subject to sequencer downtime, so only the L2 contracts are in scope here.

---

### Impact Explanation

**Bad-price execution**: the bid/ask quote delivered to the pool during the post-restart window is stale. Traders can swap against the pool at the pre-outage price while the true market price has moved, extracting value from LPs. The loss per swap is proportional to the price move during the outage and the swap size. For a volatile asset with a multi-hour outage, this can be a large fraction of LP principal.

---

### Likelihood Explanation

L2 sequencer outages are rare but documented (Arbitrum experienced a ~7-hour outage in 2022; Optimism has had shorter outages). The protocol explicitly targets L2 deployment (dedicated `PriceProviderL2`, `ProtectedPriceProviderL2`, `PriceProviderFactoryL2`, and deployment scripts under `script/l2/`). The trigger requires no privileged action — any user can submit a swap immediately after the sequencer restarts. The window lasts until oracle keepers push a fresh price, which may take minutes to hours depending on keeper liveness.

---

### Recommendation

Add a Chainlink L2 sequencer uptime feed check inside `_getBidAndAskPrice()` (or a shared internal helper) in both `PriceProviderL2` and `ProtectedPriceProviderL2`. The standard pattern:

1. Read the sequencer uptime feed (`AggregatorV2V3Interface`).
2. If `answer == 1` (sequencer is down), revert or return the stale sentinel `(0, type(uint128).max)`.
3. If the sequencer recently restarted (i.e., `block.timestamp - startedAt < GRACE_PERIOD`), also revert — oracle keepers may not yet have pushed a fresh price. A grace period of 3600 seconds (1 hour) is the Chainlink-recommended default.

The sequencer uptime feed address should be stored as an immutable set at construction time (or passed through the factory), so it cannot be changed after deployment.

---

### Proof of Concept

```
Setup:
  - Deploy PriceProviderL2 with MAX_TIME_DELTA = 2 hours, FUTURE_TOLERANCE = 10s
  - Oracle price at T=0: mid = 1000 (refTime = T=0)

Scenario:
  T=0:    Sequencer goes offline. block.timestamp freezes on L2.
  T=0:    Market price moves from 1000 → 1150 (+15%).
  T=1h:   Sequencer restarts. block.timestamp = T+1h.
  T=1h:   Oracle on-chain still has mid=1000, refTime=T=0.
           _isStale check: (T+1h) - T = 3600s < MAX_TIME_DELTA (7200s) → NOT stale.
           getBidAndAskPrice() returns bid/ask around 1000.
  T=1h:   Attacker swaps, buying the pool's asset at ~1000 instead of ~1150.
           LP loss: ~13% on the swapped amount.
  T=1h+ε: Oracle keeper pushes fresh price (1150). Window closes.
