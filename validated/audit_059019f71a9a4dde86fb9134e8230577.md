### Title
Cryptographic Key Seeds Exposed via Shell History Through Command-Line Arguments - (File: `cmd/bootstrap/cmd/key.go`)

### Summary
The Flow bootstrap CLI tool (`cmd/bootstrap`) accepts cryptographic key seeds — which deterministically derive node private keys — as plaintext command-line arguments via `--networking-seed`, `--staking-seed`, and `--machine-seed` flags. Any operator who supplies these seeds on the command line exposes them in shell history and process listings, allowing an attacker with local access to regenerate the exact private keys used by the node.

### Finding Description
The `bootstrap key` command registers three seed flags as optional command-line arguments: [1](#0-0) 

These seeds are the sole input to the deterministic key derivation functions: [2](#0-1) 

The `bootstrap machine-account-key` command similarly exposes a `--seed` flag: [3](#0-2) 

The `bootstrap observer-network-key` command exposes a `--seed` flag for the observer networking key: [4](#0-3) 

The official README explicitly documents that seeds "can be specified as command line parameters": [5](#0-4) 

When a node operator runs:
```bash
go run ./cmd/bootstrap key \
  --address "node.example.com:3569" \
  --role consensus \
  --networking-seed aabbcc... \
  --staking-seed ddeeff...
```
the seed values appear verbatim in shell history (e.g., `~/.bash_history`) and in `ps aux` output during execution. Because `crypto.GeneratePrivateKey(algo, seed)` is deterministic, any party who recovers the seed can regenerate the identical private key.

### Impact Explanation
The three seed flags map to three distinct private keys:

- **Staking key** (BLS12-381): Used to sign consensus messages. Compromise enables equivocation (double-signing), which disrupts HotStuff consensus and is a slashable offense.
- **Networking key** (ECDSA P-256): Defines the node's libp2p identity. Compromise enables network-level impersonation of the node.
- **Machine account key** (ECDSA P-256): Controls the on-chain machine account used for DKG participation and QC voting. Compromise allows unauthorized signing of DKG/QC transactions, directly affecting epoch transitions and random beacon key generation.

The `NodePrivateKeys` struct is explicitly marked as containing "all sensitive information for a node": [6](#0-5) 

### Likelihood Explanation
The likelihood is **medium**. The README actively documents seed flags as a supported input method. Node operators running bootstrapping on shared or multi-user machines, or in CI/CD pipelines where command history is logged, are at realistic risk. An attacker who gains read access to the operator's home directory (e.g., via a compromised shell account, a misconfigured CI artifact store, or a co-located process) can extract the seed from `~/.bash_history` and immediately derive the node's private keys without any further brute-force.

### Recommendation
**Short term:** Remove the `--networking-seed`, `--staking-seed`, `--machine-seed`, and `--seed` flags from all bootstrap subcommands. When a seed is required, read it from a file path argument or from `stdin` (with terminal echo disabled), never from a positional or named CLI argument that appears in process arguments.

**Long term:** Audit all bootstrap subcommands (including `--clustering-random-seed` in `clustering.go` and `--random-seed` in `rootblock.go`) for similar exposure. Ensure that any value that directly or indirectly determines a private key is never passed as a shell argument. [7](#0-6) [8](#0-7) 

### Proof of Concept
```bash
# Operator runs bootstrap with explicit seed
go run ./cmd/bootstrap key \
  --address "consensus-node.example.com:3569" \
  --role consensus \
  --networking-seed aabbccddeeff00112233445566778899aabbccddeeff001122334455667788 \
  --staking-seed 99887766554433221100ffeeddccbbaa99887766554433221100ffeeddccbbaa

# Attacker reads shell history on the same machine
cat ~/.bash_history | grep networking-seed
# → reveals the full hex seed

# Attacker regenerates the identical staking private key
seed, _ := hex.DecodeString("99887766554433221100ffeeddccbbaa99887766554433221100ffeeddccbbaa")
stakingKey, _ := crypto.GeneratePrivateKey(crypto.BLSBLS12381, seed)
// stakingKey is now identical to the node's actual staking private key
```
