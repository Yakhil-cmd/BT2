### Title
Unbounded EVM-event scan in `eip6110_events_parser` during block finalization is uncharged against native resources, enabling prover-cycle exhaustion DOS — (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over **every EVM event emitted in the entire block** during block finalization to find EIP-6110 deposit events. This loop runs outside any native-resource accounting. Because EVM events (LOG0–LOG4) are bounded only by the block gas limit and not by a native-resource cap, an unprivileged attacker can fill a block with cheap LOG0 events, forcing the prover to execute an arbitrarily large finalization loop that was never charged against the block's native-resource budget. In the RISC-V proving context this can exhaust the prover's cycle budget, making the block unprovable and permanently stalling L1 finalization.

---

### Finding Description

`eip6110_events_parser` is called unconditionally during block finalization in both the sequencing and proving post-tx operations:

```
post_tx_op_proving.rs  → eip6110_events_parser(&*system, &mut intermediate_hasher)
post_tx_op_sequencing.rs → eip6110_events_parser(&system, &mut requests_hasher)
```

Its implementation is a plain `for` loop over the full block-level events iterator:

```rust
for event in system.io.events_iterator() {   // iterates ALL EVM events in the block
    if event.address != &DEPOSIT_CONTRACT_ADDRESS {
        continue;
    }
    ...
}
``` [1](#0-0) 

The block-limit enforcement function `check_for_block_limits` caps gas, blob-gas, computational native resources, pubdata, and **L2→L1 logs** (`MAX_NUMBER_OF_LOGS = 16 384`), but it has **no cap on the number of EVM events** (LOG0–LOG4 opcodes): [2](#0-1) 

EVM events are therefore bounded only by the block gas limit. With a ~15 M gas block limit and LOG0 costing 375 gas, an attacker can emit ≈ 40 000 events per block. Every one of those events causes one iteration of the `eip6110_events_parser` loop during finalization.

Critically, this finalization loop runs **after** all per-transaction native-resource accounting is complete. The `Resources::from_native(u64::MAX)` pattern used in the EIP-7002/7251 helpers shows that finalization-phase storage reads are intentionally given infinite resources; the same logic applies to the event scan — it is never charged. The `MAX_NATIVE_COMPUTATIONAL` guard only covers work accumulated during the transaction loop: [3](#0-2) 

In the RISC-V proving binary (`proof_running_system`), every instruction executed consumes prover cycles. The finalization loop's O(total_events) work is invisible to the resource model, creating an unbounded, uncharged computation path in the critical proving code path. [4](#0-3) 

---

### Impact Explanation

**Prover-cycle exhaustion / unprovable block (DOS on L1 finalization).**

The ZKsync Airbender prover operates on a RISC-V binary with a finite cycle budget. If the `eip6110_events_parser` loop consumes enough cycles beyond what the transaction-phase native-resource accounting reserved, the block exceeds the prover's cycle limit and cannot be proven. A block that cannot be proven cannot be settled on L1, permanently stalling the chain's state-root advancement and all dependent withdrawals and cross-chain operations.

Even below the hard cycle limit, the asymmetry between what the attacker pays (EVM gas, bounded by the block gas limit) and what the prover must compute (O(events) finalization work, uncharged) degrades proving throughput and increases proving cost per block.

---

### Likelihood Explanation

Any unprivileged L2 user can deploy a contract that executes `LOG0` in a tight loop and submit transactions that fill the block gas limit with such calls. No special role, governance access, or oracle manipulation is required. The attack is cheap: LOG0 with zero data costs 375 gas, so a 15 M gas block can hold ≈ 40 000 events. The attacker pays only standard transaction fees; the prover bears the unbounded finalization cost.

---

### Recommendation

1. **Add a block-level cap on EVM events.** Introduce a `MAX_EVM_EVENTS_PER_BLOCK` constant and enforce it in `check_for_block_limits`, analogous to the existing `MAX_NUMBER_OF_LOGS` check for L2→L1 logs.

2. **Charge native resources for the finalization scan.** Account for the `eip6110_events_parser` loop cost in the block's native-resource budget, either by pre-charging based on the event count before finalization or by running the scan inside a resource-metered context.

3. **Filter events at emission time.** Maintain a separate, bounded list of events emitted by `DEPOSIT_CONTRACT_ADDRESS` during transaction execution, so the finalization scan is O(deposit_events) rather than O(all_events).

---

### Proof of Concept

1. Deploy a contract on the ZKsync OS Ethereum-compatible chain:
   ```solidity
   contract Spammer {
       fallback() external {
           for (uint i = 0; i < 500; i++) {
               assembly { log0(0, 0) }  // 375 gas each, no data, no topics
           }
       }
   }
   ```
2. Submit enough transactions calling this contract to consume the full block gas limit (~15 M gas → ~40 000 LOG0 events across the block).
3. All transactions pass `check_for_block_limits` (gas limit, native limit, pubdata limit, and L2→L1 log limit are all respected; EVM events are uncapped).
4. During `post_op` (block finalization), `eip6110_events_parser` iterates all ~40 000 events. None of this work was charged against the block's native-resource budget.
5. In the RISC-V proving context, the prover must execute the full loop, consuming cycles that were not reserved. Repeated across blocks, this exhausts the prover's cycle budget or makes individual blocks unprovable, stalling L1 finalization. [5](#0-4) [6](#0-5) [7](#0-6)

pub fn eip6110_events_parser<S: EthereumLikeTypes>(
    system: &System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {
                event_encountered = true;
                requests_hasher.update(&[DEPOSIT_REQUEST_EIP_7685_TYPE]);
            }
            let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
                panic!("invalid deposit event structure");
            };
        }
    }

    Ok(event_encountered)
}
