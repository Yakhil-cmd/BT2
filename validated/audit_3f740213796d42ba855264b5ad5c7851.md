### Title
Missing Validation of `app_pk` Identity Element Causes Direct Disclosure of Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts `app_pk: PublicKey` (a `blstrs::G1Projective` element) without validating that it is not the group identity element. When a caller supplies `app_pk = G1::identity()`, the ElGamal masking term collapses to zero, causing the coordinator's output `C` to directly equal `msk · H(pk, app_id)` — the confidential derived key — in plaintext. This key is then broadcast on-chain, exposing it to any observer.

### Finding Description

The CKD protocol is an ElGamal-based scheme. Each node `i` computes:

```
S_i = x_i · H(pk, app_id)
C_i = S_i + y_i · A          // A = app_pk
```

The coordinator aggregates:

```
C = Σ λ_i · C_i = msk · H(pk, app_id) + (Σ λ_i · y_i) · A
Y = Σ λ_i · y_i · G
```

The app recovers the confidential key as `C − a · Y = msk · H(pk, app_id)`, which requires knowing `a` (the secret corresponding to `A = a · G`).

When `app_pk = G1::identity()`:

```
C_i = S_i + y_i · identity = S_i
C   = Σ λ_i · S_i = msk · H(pk, app_id)
```

The masking term vanishes entirely. The coordinator's output `C` is the confidential key itself, with no encryption. This output is sent on-chain per the protocol design.

The root cause is in `ckd()`: [1](#0-0) 

No check is performed on `app_pk` before it is forwarded into `compute_signature_share()`: [2](#0-1) 

The multiplication `app_pk * y.0` at line 174 silently produces the identity when `app_pk` is the identity, stripping the ElGamal mask.

By contrast, the `verify_signature` function in the ciphersuite module does explicitly reject identity elements: [3](#0-2) 

This guard exists for verification but is absent at the protocol entry point.

The `PublicKey` type alias is: [4](#0-3) 

`blstrs::G1Projective::identity()` is a valid Rust value that compiles and passes all existing checks in `ckd()`.

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

A malicious caller submitting `app_pk = G1Projective::identity()` causes the coordinator to broadcast `C = msk · H(pk, app_id)` on-chain without any ElGamal masking. This is the exact confidential derived key the protocol is designed to keep secret. Any on-chain observer can read it directly from the coordinator's response, without needing `app_sk`. The master secret key `msk` itself is not exposed, but the per-`app_id` derived secret — the primary security guarantee of CKD — is fully disclosed.

### Likelihood Explanation

**High.** The `app_pk` value `A` is supplied by the TEE application or developer contract as part of the CKD request. A malicious app or a compromised developer contract can trivially set `A = G1::identity()`. The identity element is a canonical, easily constructable value (`blstrs::G1Projective::identity()`). No cryptographic capability is required. The MPC network has no mechanism to distinguish a legitimate zero-key request from a malicious one without an explicit identity check at the library boundary.

### Recommendation

Add an explicit identity-element check for `app_pk` at the start of `ckd()` in `src/confidential_key_derivation/protocol.rs`, before the protocol is initialized:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used in `verify_signature` and in the `BLS12381G1Group::serialize` implementation, which both reject identity elements explicitly.

```rust
use blstrs::{G1Projective, Scalar};
use elliptic_curve::Group;
use threshold_signatures::confidential_key_derivation::{
    protocol::ckd, AppId, ElementG1,
};
use threshold_signatures::participants::Participant;

// Attacker supplies the identity element as app_pk
let app_pk: ElementG1 = G1Projective::identity(); // <-- malicious input

let participants = vec![Participant::from(1u32), Participant::from(2u32)];
let coordinator = participants[0];

// ckd() accepts this without error
let protocol = ckd(
    &participants,
    coordinator,
    participants[0],
    honest_keygen_output,
    AppId::try_from(b"victim_app").unwrap(),
    app_pk,  // identity — no validation
    rng,
).unwrap(); // succeeds

// After running the protocol, the coordinator's CKDOutput.big_c
// equals msk * H(pk, app_id) directly — the confidential key in plaintext.
// Any observer of the on-chain response can read it without app_sk.
```

```rust
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);

    // ensure my presence in the participant list

            role: "self",

            role: "coordinator",
```
