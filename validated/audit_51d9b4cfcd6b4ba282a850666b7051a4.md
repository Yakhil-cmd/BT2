### Title
Attacker Can DOS Reward Distribution by Donating Tokens to `RewardSupplier` - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `request_funds` internal function in `RewardSupplier` uses the contract's **total** STRK token balance to calculate available credit. An attacker can donate STRK tokens directly to the `RewardSupplier` contract, artificially inflating the apparent credit and suppressing L1 mint requests. Once the donated tokens are consumed by legitimate reward claims, the contract's balance falls below `unclaimed_rewards` with no pending L1 replenishment, causing `claim_rewards` to revert and temporarily freezing reward distribution.

---

### Finding Description

`request_funds` is called every time `update_unclaimed_rewards_from_staking_contract` is invoked. It reads the contract's live token balance to decide whether to send an L1 mint request:

```cairo
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // ← total balance, not tracked credit
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount;   // ← inflated by donation
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);
    if credit < debit + threshold {
        // send L1 mint request
    }
    ...
}
``` [1](#0-0) 

Because `balance` is the raw ERC-20 balance of the contract address, any tokens transferred directly to the contract (without going through `on_receive`) inflate `credit` without updating any internal accounting. The `on_receive` callback only reduces `l1_pending_requested_amount` for tokens arriving via StarkGate; a direct ERC-20 transfer bypasses it entirely. [2](#0-1) 

When `claim_rewards` is later called by the staking contract, it performs a `checked_transfer`. If the balance has been depleted below `unclaimed_rewards` (because the donated tokens were consumed by prior claims and no L1 replenishment was triggered), the transfer reverts:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    ...
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
    // ↑ reverts if actual balance < amount
}
``` [3](#0-2) 

This propagates up through `claim_from_reward_supplier` in the staking contract utilities, which asserts the balance increase equals the claimed amount: [4](#0-3) 

---

### Impact Explanation

Temporary freezing of unclaimed yield. When the donated tokens are consumed by legitimate claims and no L1 mint request was sent in advance (because the donation suppressed the threshold check), the `RewardSupplier` balance falls below `unclaimed_rewards`. All subsequent calls to `claim_rewards` revert, blocking reward distribution for all stakers until an L1 mint response eventually arrives — which can only happen after the next `request_funds` call finally triggers a new L1 request (once the balance drops low enough to pass the threshold check again). The L1→L2 message latency introduces a meaningful freeze window.

---

### Likelihood Explanation

High. The attack requires only a direct ERC-20 `transfer` of STRK tokens to the `RewardSupplier` contract address. No privileged role, no special access, and no protocol interaction is needed — any STRK token holder can execute this. The `on_receive` callback is gated to the StarkGate bridge, so a direct transfer is never intercepted or accounted for.

---

### Recommendation

Replace the live `balance_of` call with an internally tracked balance variable. Increment it in `on_receive` (when tokens arrive from L1) and decrement it in `claim_rewards` (when tokens are sent to the staking contract). Use this tracked value instead of `balance_of(get_contract_address())` in `request_funds` to compute `credit`. This mirrors the fix recommended in the referenced report: measure the expected balance, not the total balance.

---

### Proof of Concept

1. Protocol is operating normally: `unclaimed_rewards = U`, `balance = B`, `l1_pending_requested_amount = P`, with `B + P ≈ U + threshold` (healthy state).
2. Attacker calls `STRK.transfer(reward_supplier_address, X)` for a large `X`.
3. `balance` is now `B + X`. Next call to `request_funds(U)` computes `credit = B + X + P`, which satisfies `credit >= U + threshold` — no L1 request is sent.
4. Stakers claim rewards normally; `claim_rewards` transfers tokens out. After `U` tokens are claimed, `balance = B + X - U`, `unclaimed_rewards = 0`.
5. New rewards accumulate: `unclaimed_rewards = R`. `request_funds(R)` is called; `balance = B + X - U`, still large enough to suppress the L1 request if `X` was chosen appropriately.
6. This continues until `balance < unclaimed_rewards`. At that point, `claim_rewards` calls `checked_transfer` which reverts due to insufficient balance.
7. No L1 mint request was sent during steps 3–6, so no replenishment is in flight. Reward distribution is frozen until the next `request_funds` call finally triggers an L1 request and the bridge message is processed.

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
