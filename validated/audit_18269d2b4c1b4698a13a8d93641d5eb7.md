### Title
Unchecked `sender` Argument in Bank Precompile `transfer` Enables Unauthorized Drain of `evm/<caller>` Native Tokens — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the bank precompile accepts a `sender` address as an ABI argument and uses it directly as the debit account in `SendCoins`, with **no check that `sender == contract.Caller()`**. Any contract can therefore call `bank.transfer(victim, attacker, amount)` to drain the victim's `evm/<caller>` native-token balance without the victim's consent.

---

### Finding Description

In `BankContract.Run`, the `TransferMethodName` branch: [1](#0-0) 

```go
sender    := args[0].(common.Address)   // fully attacker-controlled ABI arg
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from  := sdk.AccAddress(sender.Bytes()) // no validation against contract.Caller()
to    := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // denom scoped to caller contract
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))  // debits `from`
```

The denom is correctly scoped to `evm/<contract.Caller()>`, but the **debit address** (`from`) is taken verbatim from the ABI input. There is no guard of the form `if sender != contract.Caller() { return error }` anywhere in the function.

Compare with `mint`/`burn`, which also use `contract.Caller()` for the denom but operate on a module account rather than an arbitrary user address — the asymmetry makes `transfer` uniquely dangerous. [2](#0-1) 

---

### Impact Explanation

A malicious (or later-compromised) contract `M` can:

1. Attract users to interact with it (e.g., as a DeFi vault), causing them to hold `evm/<M>` native tokens (via `bank.mint` or prior `bank.transfer` calls).
2. Call `bank.transfer(victim, attacker, victim_balance)` at any time — no user signature or approval required.
3. `SendCoins` executes unconditionally, moving the victim's `evm/<M>` tokens to the attacker.
4. If `M` also provides a redemption path (burn `evm/<M>` for underlying CRO/IBC assets), the attacker converts the stolen tokens to real value.

This is a direct, unprivileged, on-chain drain of user funds — Critical per the scope rules ("Unauthorized … transfer … of precompile-controlled assets").

The "stale state" framing in the question is a red herring: the vulnerability is unconditional and requires no special prior state. Any block where a victim holds a nonzero `evm/<M>` balance is sufficient.

---

### Likelihood Explanation

- Requires deploying one malicious contract — no privileged access, no leaked keys.
- The attack is a single EVM call; it is atomic and reverts cleanly on failure.
- The `TestBank` integration test already demonstrates the exact call pattern (`bank.transfer(msg.sender, recipient, amount)`), confirming the path is live and reachable. [3](#0-2) 

---

### Recommendation

Enforce that the `sender` argument equals `contract.Caller()` before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
```

Alternatively, restrict `transfer` so that only the contract itself (i.e., `contract.Caller()`) can be the debit account, matching the trust model already implied by denom scoping.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract DrainVault {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: lure victim — mint evm/<DrainVault> tokens to them
    function deposit(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain — no victim signature required
    function drain(address victim, address attacker, uint256 amount) external {
        // sender arg = victim, but caller = DrainVault
        // precompile uses DrainVault as denom authority, victim as debit account
        bank.transfer(victim, attacker, amount);
    }
}
```

**Expected result:** `drain()` succeeds; victim's `evm/<DrainVault>` balance decreases by `amount`; attacker's balance increases by `amount`. No revert, no authorization error. [4](#0-3)
