### Title
WalletConnect Offer Confirmation Displays Understated "Total Amount with Royalties" When Multiple NFTs Are Involved - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`formatAmountWithRoyalties` in the WalletConnect confirmation pipeline incorrectly divides the fungible spending amount equally across all NFTs before applying each royalty percentage, instead of applying each royalty percentage to the full amount. The result is that the "Total Amount with Royalties" shown to the user in the `Confirm.tsx` dialog is systematically understated whenever a `take_offer` or `create_offer_for_ids` WalletConnect command involves two or more NFTs with royalties. A user who relies on this figure to decide whether to approve the transaction will approve while believing they are paying less than the blockchain will actually deduct.

### Finding Description

`formatAmountWithRoyalties` is called by `withRoyaltyTotals` for every fungible spending line (XCH or CAT) in a WalletConnect offer confirmation. It receives the full spending `amount` and the array of royalty percentages extracted from all NFTs on the receiving side.

The current implementation:

```js
// packages/gui/src/electron/commands/parseCommandDisplay.ts  lines 363-367
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This computes:

```
royaltyAmount = (amount / n) × Σ(royaltyPercentage_i) / 10 000
```

The correct formula is:

```
royaltyAmount = amount × Σ(royaltyPercentage_i) / 10 000
```

The two are equal only when `n = 1`. For `n ≥ 2` the displayed royalty is `n` times too small.

The existing test at line 251–335 of `parseCommandDisplay.test.ts` asserts the wrong value (`0.00010255`) and therefore encodes the bug rather than catching it:

- amount = 100 000 000 mojos (0.0001 XCH)
- NFT1 royalty = 500 bp (5 %), NFT2 royalty = 10 bp (0.1 %)
- **Code produces**: `splitAmount = 50 000 000`; royalty = 2 500 000 + 50 000 = 2 550 000 mojos → total **0.00010255 XCH**
- **Correct**: royalty = 5 000 000 + 100 000 = 5 100 000 mojos → total **0.0001051 XCH**

The computed `amountWithRoyalties` string is placed directly into the `DisplayWalletDeltaItem` and rendered in `Confirm.tsx` as "Total Amount with Royalties" for XCH, CAT, and wallet-type lines. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The `amountWithRoyalties` field is the primary figure the user reads to understand their total cost before clicking "Approve" in the WalletConnect confirmation dialog. When it is understated, the user approves a transaction believing they will spend less XCH/CAT than the blockchain will actually deduct (royalties are enforced on-chain, not by the GUI). This constitutes displaying the wrong amount in a signing-approval context, matching the High impact criterion: *"WalletConnect state that causes a user to approve … the wrong … amount."*

With realistic royalty percentages the gap is material. Example: 1 XCH offered, two NFTs with 10 % and 5 % royalties — the dialog shows **1.075 XCH** but the user actually pays **1.15 XCH** (a 0.075 XCH shortfall). [4](#0-3) [5](#0-4) 

### Likelihood Explanation

Any WalletConnect-connected dapp can trigger this by issuing a `take_offer` command whose offer contains two or more NFTs with royalties, or a `create_offer_for_ids` command with a `driver_dict` containing multiple NFT entries with royalty transfer programs. Both command paths are fully supported and documented. The attacker needs only an existing WalletConnect session (already approved by the user) and a crafted offer string or parameter object — no additional privileges, leaked keys, or cryptographic breaks are required. [6](#0-5) 

### Recommendation

Remove the `splitAmount` division. Apply each royalty percentage to the full `amount`:

```js
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the test at line 310–335 of `parseCommandDisplay.test.ts` to assert the corrected value (`0.0001051` XCH for the two-NFT case). [7](#0-6) [8](#0-7) 

### Proof of Concept

1. Establish a WalletConnect session with the Chia GUI wallet.
2. Send a `chia_wallet.take_offer` command whose offer resolves (via `getOfferSummary`) to:
   - `requested`: `{ xch: "1000000000000" }` (1 XCH)
   - `offered`: `{ <nft1_launcher_id>: "1", <nft2_launcher_id>: "1" }`
   - `infos`: NFT1 with `royalty_percentage: 1000` (10 %), NFT2 with `royalty_percentage: 500` (5 %)
3. Observe the WalletConnect confirmation dialog. It displays **"Total Amount with Royalties: 1.075 XCH"**.
4. The correct value is **1.15 XCH** (10 % × 1 XCH + 5 % × 1 XCH = 0.15 XCH royalty).
5. The user approves, believing they spend 1.075 XCH; the blockchain deducts 1.15 XCH. [1](#0-0) [4](#0-3)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }
