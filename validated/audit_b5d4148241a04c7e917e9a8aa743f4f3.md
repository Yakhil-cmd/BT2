### Title
Missing `max_l2_gas_amount` Upper-Bound Check for Declare Transactions Admits Over-Resourced Transactions - (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces an upper bound on `l2_gas.max_amount` for Invoke and DeployAccount transactions but explicitly skips that check for Declare transactions. No downstream guard re-applies the limit. A Declare transaction with an arbitrarily large `l2_gas.max_amount` passes every gateway validation stage and is admitted to the mempool, violating the protocol invariant that all transactions must satisfy `l2_gas.max_amount ≤ config.max_l2_gas_amount`.

### Finding Description

In `validate_resource_bounds`, the code reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The empty `if` branch for `Declare` is a deliberate no-op: the upper-bound error is never returned for Declare transactions. The test `valid_l2_gas_amount_on_declare` explicitly asserts this behaviour — a Declare with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100`. [2](#0-1) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* against the previous block's price; it never checks the gas *amount*. [3](#0-2) 

The blockifier's `check_fee_bounds` enforces only a *lower* bound (actual minimal gas ≤ declared max), not an upper bound. [4](#0-3) 

`verify_can_pay_committed_bounds` checks whether the account balance covers `max_possible_fee()`. This is the only partial guard: if `l2_gas.max_price_per_unit` is large enough to make `max_possible_fee()` exceed the account balance, the transaction is rejected. However, an attacker who controls both `l2_gas.max_amount` and `l2_gas.max_price_per_unit` can choose a price low enough that the inflated max fee is still within their balance, while `max_amount` is many times the configured limit. [5](#0-4) 

Once admitted, `TransactionContext::initial_sierra_gas()` returns `l2_gas.max_amount` directly as the initial Sierra gas budget for the transaction. [6](#0-5) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` set to a multiple of `config.max_l2_gas_amount` (e.g. `N × 1,210,000,000`) passes stateless and stateful gateway validation and is admitted to the mempool. During execution it receives an initial Sierra gas budget of `N × max_l2_gas_amount` instead of the intended maximum. This allows a single Declare transaction to consume more per-transaction Sierra gas than the protocol permits, bypassing the per-transaction resource cap. The production configured limit is `1,210,000,000`. [7](#0-6) 

**Impact category**: High — Mempool/gateway admission accepts invalid transactions before sequencing.

### Likelihood Explanation

Any unprivileged user can craft a Declare transaction with an oversized `l2_gas.max_amount`. The only constraint is that the account must hold enough balance to cover `max_possible_fee()` at the chosen `max_price_per_unit`. By setting `max_price_per_unit` to the minimum accepted value (just above the stateful validator's price threshold), the required balance scales linearly with `max_amount`, making the attack feasible for any account with moderate funds.

### Recommendation

Remove the Declare-specific exemption and apply the same upper-bound check uniformly:

```rust
// In validate_resource_bounds:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas budget (e.g. for Sierra compilation), introduce a separate `max_l2_gas_amount_declare` config field with an appropriate limit rather than leaving the field unbounded.

### Proof of Concept

1. Construct a valid `RpcDeclareTransaction::V3` with:
   - `l2_gas.max_amount = GasAmount(config.max_l2_gas_amount * 10)` (10× the limit)
   - `l2_gas.max_price_per_unit` = minimum price that passes the stateful price-threshold check
   - `l1_gas` bounds set to cover actual L1 costs
2. Submit via the gateway's `add_transaction` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never triggered for Declare.
4. The stateful validator's `validate_resource_bounds` only checks the price, not the amount — passes.
5. `verify_can_pay_committed_bounds` passes because the account balance covers `max_possible_fee()` at the chosen low price.
6. The transaction is admitted to the mempool with `initial_sierra_gas() = GasAmount(max_l2_gas_amount * 10)`, giving it 10× the intended per-transaction gas budget.

The existing test `valid_l2_gas_amount_on_declare` already documents and asserts this bypass as expected behaviour: [2](#0-1)

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}
