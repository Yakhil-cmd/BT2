Audit Report

## Title
`liquid_cycles_balance` Does Not Account for Pending `ingress_induction_cycles_debit` During DTS Execution - (`rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs`)

## Summary
During Deterministic Time Slicing (DTS) multi-round execution, `SandboxSafeSystemState` is initialized with `system_state.balance()` as `initial_cycles_balance` rather than `system_state.debited_balance()`. This means `liquid_cycles_balance()` and `withdraw_cycles_for_transfer()` do not subtract the pending `ingress_induction_cycles_debit` when computing spendable cycles. A canister can therefore transfer up to `ingress_induction_cycles_debit` more cycles than it actually has available, causing its balance to fall below the freeze threshold once the debit is applied after execution completes.

## Finding Description

**Root cause:** `SandboxSafeSystemState::new()` passes `system_state.balance()` (the raw balance) at line 836 of `sandbox_safe_system_state.rs`, not `system_state.debited_balance()`. [1](#0-0) 

`SystemState` separately tracks `ingress_induction_cycles_debit`, which is not yet deducted from `cycles_balance`. The correct effective balance is `debited_balance()`: [2](#0-1) 

When a new ingress message arrives while a canister is paused between DTS slices, `charge_ingress_induction_cost` correctly guards using `debited_balance()` and adds to `ingress_induction_cycles_debit`: [3](#0-2) 

When the next DTS slice resumes, a fresh `SandboxSafeSystemState` is built with `initial_cycles_balance = balance()` (the pre-debit value). `liquid_cycles_balance()` then computes `cycles_balance() - freeze_threshold` without subtracting the pending debit: [4](#0-3) 

`withdraw_cycles_for_transfer()` uses the same inflated `self.cycles_balance()` as the starting point, permitting transfers up to `balance() - freeze_threshold` rather than the correct `debited_balance() - freeze_threshold`: [5](#0-4) 

After execution completes, `apply_ingress_induction_cycles_debit` deducts the full debit from the real balance: [6](#0-5) 

At that point the canister's balance is `balance() - transferred - debit`. Since `transferred` was computed against the inflated `balance()`, the final balance can fall below the freeze threshold. The maximum over-transfer equals `ingress_induction_cycles_debit`, which is itself bounded by `balance - freeze_threshold` (enforced by the guard in `charge_ingress_induction_cost`). In the worst case (multiple ingress messages inducted across multiple DTS pauses), the total debit approaches `balance - threshold`, and the canister can end up at or near zero balance.

## Impact Explanation

This is a cycles conservation / resource accounting bug. Cycles are effectively double-spent: transferred to another canister via `ic0_call_cycles_add128` and simultaneously consumed by the pending debit applied post-execution. The canister's balance drops below its freeze threshold, violating the protocol invariant that permitted operations must leave the balance at or above the freeze threshold. The canister becomes frozen as a result. This matches the **High** impact class: significant cycles/funds accounting impact with concrete user harm (canister frozen, cycles over-transferred), constrained by the requirement that the canister's Wasm code must be performing a liquid-balance transfer during the affected DTS execution.

## Likelihood Explanation

The scenario requires: (1) a canister running a DTS update call (any canister with a large instruction limit), (2) a concurrent ingress message inducted to the same canister while it is paused between slices — triggerable by any unprivileged external user, and (3) the canister's Wasm code querying `ic0_canister_liquid_cycle_balance128` and transferring the reported maximum via `ic0_call_cycles_add128`. Steps 1 and 2 are reachable by any unprivileged ingress sender. Step 3 is a common canister pattern (e.g., "drain my liquid balance to a child canister"). The combination is realistic for any canister that attempts to maximize cycle transfers during a long-running execution.

## Recommendation

Initialize `initial_cycles_balance` in `SandboxSafeSystemState::new()` with `system_state.debited_balance()` instead of `system_state.balance()`:

```rust
// rs/embedders/src/wasmtime_embedder/system_api/sandbox_safe_system_state.rs, line 836
system_state.debited_balance(),  // was: system_state.balance()
```

This ensures `liquid_cycles_balance()`, `withdraw_cycles_for_transfer()`, and `ic0_canister_liquid_cycle_balance128` all reflect the true spendable balance, consistent with how `charge_ingress_induction_cost` already guards new debits using `debited_balance()`.

## Proof of Concept

```
1. Canister C: balance = 10_000_000, freeze_threshold = 5_000_000, debit = 0.

2. C starts a DTS update call (long-running). After the first slice,
   C is paused with balance still = 10_000_000.

3. External user sends a new ingress message to C.
   charge_ingress_induction_cost() checks debited_balance() = 10_000_000 >= cost + threshold,
   adds cost = 2_000_000 to ingress_induction_cycles_debit.
   State: balance = 10_000_000, debit = 2_000_000, debited_balance = 8_000_000.

4. C's DTS execution resumes. SandboxSafeSystemState built with
   initial_cycles_balance = balance() = 10_000_000 (NOT debited_balance = 8_000_000).

5. C calls ic0_canister_liquid_cycle_balance128:
   liquid_cycles_balance() = 10_000_000 - 5_000_000 = 5_000_000 (inflated by 2_000_000).

6. C calls ic0_call_cycles_add128(5_000_000) to transfer to canister D.
   withdraw_cycles_for_transfer checks: 10_000_000 - 5_000_000 >= 5_000_000 → OK.
   Balance change recorded: -5_000_000.

7. Execution completes. apply_balance_changes: balance = 10_000_000 - 5_000_000 = 5_000_000.
   apply_ingress_induction_cycles_debit: balance = 5_000_000 - 2_000_000 = 3_000_000.

8. Final balance = 3_000_000 < freeze_threshold = 5_000_000.
   Canister C is frozen. Canister D received 5_000_000 cycles that C did not truly have available.
```

A deterministic integration test or PocketIC test can reproduce this by: deploying a canister with a Wasm that calls `ic0_canister_liquid_cycle_balance128` then `ic0_call_cycles_add128` with the full liquid amount; triggering a DTS execution; injecting a concurrent ingress message to accumulate debit; resuming the DTS slice; and asserting the final balance against the freeze threshold.
