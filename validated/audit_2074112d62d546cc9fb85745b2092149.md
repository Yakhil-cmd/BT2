Audit Report

## Title
Markdown Injection via Unsanitized Memo in ICRC-21 `GenericDisplayMessage` Consent Messages - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` function interpolates a UTF-8 memo string verbatim into a single-backtick inline code span in a Markdown `GenericDisplayMessage`. A backtick character in the memo closes the code span prematurely, and a newline breaks out of it entirely (CommonMark inline code spans do not span lines), allowing a dapp to inject arbitrary Markdown fields into the consent message shown to the user before signing. The ckBTC minter explicitly documents and mitigates this same class of attack via address validation; the shared ledger `add_memo` path has no equivalent protection.

## Finding Description
In `packages/icrc-ledger-types/src/icrc21/responses.rs` at line 294, `memo_str` is placed verbatim inside a single-backtick span:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

If `memo_str` is valid UTF-8 (checked at line 288–289), it is used as-is with no escaping. A memo byte sequence that decodes to `` ` \n\n**Amount:** `0.00000001 TOKEN` `` closes the opening backtick immediately, emits a newline (terminating the inline code span per CommonMark), and then injects a structurally identical bold field. Because `add_memo` is called after `add_amount` and `add_fee`, the injected field appears at the end of the message, where some wallet renderers may display it as the authoritative value or where users may anchor on the last-seen field.

The `icrc21_canister_call_consent_message` endpoint on both the ICRC-1 ledger (`rs/ledger_suite/icrc1/ledger/src/main.rs`, line 1189) and the ICP ledger (`rs/ledger_suite/icp/ledger/src/main.rs`, line 1478) is an `#[update]` method callable by any principal with no access control. The memo is decoded directly from the caller-supplied `ConsentMessageRequest.arg` blob with no sanitization before reaching `add_memo`.

A secondary surface exists in `add_intent` (line 53): `token_name` is interpolated unescaped as a Markdown `#` heading. The `validate_token_name` function in `rs/nervous_system/common/src/ledger_validation.rs` (lines 51–84) enforces only length (4–255 chars) and a banned-name list; it does not reject newlines or Markdown control characters, so a governance-approved SNS token name containing `\n# ` could inject a second heading. This surface requires governance majority and is a secondary concern.

The ckBTC minter (`rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs`, lines 195–209) explicitly validates the destination address against a strict parser before interpolation and documents the Markdown-injection risk in comments, confirming DFINITY's awareness of this attack class and the absence of equivalent protection in the shared ledger path.

## Impact Explanation
The ICRC-21 consent message flow is the sole mechanism by which a wallet presents a human-readable description of a transaction to a user before signing. Injecting fake bold fields (e.g., a second `**Amount:**` or `**To:**`) into this message allows a malicious dapp to cause a user to approve a transaction whose on-chain effect differs from what was displayed. This constitutes concrete user-funds harm across all ICRC-1 and ICRC-2 ledger deployments (ICP ledger, all SNS ledgers, and any third-party ICRC-1 ledger using the shared `build_icrc21_consent_info` library). This matches the allowed High impact: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
Exploitation requires only that a dapp include a backtick and newline in a memo — no privileged access, no key material, no threshold corruption. Any dapp can supply arbitrary memo bytes to `icrc21_canister_call_consent_message`. The ICRC-21 flow is specifically the highest-value moment to deceive a user (immediately before signing), and the attack is repeatable against any user of any ICRC-1/ICRC-2 ledger.

## Recommendation
Escape or strip Markdown special characters from `memo_str` before interpolation in `add_memo`. At minimum, backticks and newlines must be removed or escaped. A robust approach is to use a fenced code block (triple-backtick) instead of an inline code span, which is significantly harder to escape. Apply the same treatment to `token_name` in `add_intent` for the `GenericDisplayMessage` path. The ckBTC minter's pattern of validating input against a strict allowlist before interpolation should be adopted as the standard for all user-supplied strings entering `GenericDisplayMessage`.

## Proof of Concept
Call `icrc21_canister_call_consent_message` on any ICRC-1 ledger with:

```
method: "icrc1_transfer"
arg: Candid-encode TransferArg {
    to: <victim_account>,
    amount: 100_000_000,   // actual: 1 ICP
    memo: Some(Memo(b"` \n\n**Amount:** `0.00000001 TOKEN`".to_vec())),
    ...
}
```

The returned `GenericDisplayMessage` will contain a second `**Amount:**` field after the legitimate one, injected from the memo. A wallet rendering CommonMark will display both fields; users anchoring on the last-seen field will perceive the transfer amount as `0.00000001 TOKEN` while the actual on-chain debit is `1 ICP`. A deterministic integration test using PocketIC can assert that the returned `GenericDisplayMessage` string contains the injected bold field and that a CommonMark renderer produces two distinct `Amount` entries.
