### Title
Incorrect Argument Types in FVM System Contract Invocation Specs Cause Fee Deduction and Payer Balance Check Failures - (File: `fvm/environment/system_contracts.go`)

### Summary
The `ContractFunctionSpec` structs for `verifyPayersBalanceForTransactionExecution` and `deductTransactionFee` declare `sema.UInt64Type` for their effort/fee parameters, but the actual `cadence.Value` arguments passed at call sites are `cadence.UFix64`. `UInt64` and `UFix64` are distinct, incompatible types in Cadence. The Cadence runtime uses the declared `ArgumentTypes` to convert and validate arguments when invoking contract functions; passing a `UFix64` value where `UInt64` is declared causes the invocation to fail, breaking payer balance checks and fee deductions for every transaction.

---

### Finding Description

In `fvm/environment/system_contracts.go`, two `ContractFunctionSpec` structs are defined with incorrect `ArgumentTypes`:

**`verifyPayersBalanceForTransactionExecutionSpec`** (lines 89–107):
```go
ArgumentTypes: []sema.Type{
    sema.NewReferenceType(..., sema.AccountType),
    sema.UInt64Type,   // ← declared as UInt64
    sema.UInt64Type,   // ← declared as UInt64
},
``` [1](#0-0) 

But `CheckPayerBalanceAndGetMaxTxFees` passes `cadence.UFix64` values for those parameters (lines 120–127):
```go
[]cadence.Value{
    cadence.BytesToAddress(payer.Bytes()),
    cadence.UFix64(inclusionEffort),     // ← UFix64 value
    cadence.UFix64(maxExecutionEffort),  // ← UFix64 value
},
``` [2](#0-1) 

**`deductTransactionFeeSpec`** (lines 130–148) has the identical mismatch:
```go
ArgumentTypes: []sema.Type{
    sema.NewReferenceType(..., sema.AccountType),
    sema.UInt64Type,   // ← declared as UInt64
    sema.UInt64Type,   // ← declared as UInt64
},
``` [3](#0-2) 

While `DeductTransactionFees` passes `cadence.UFix64` values (lines 157–164):
```go
[]cadence.Value{
    cadence.BytesToAddress(payer.Bytes()),
    cadence.UFix64(inclusionEffort),   // ← UFix64 value
    cadence.UFix64(executionEffort),   // ← UFix64 value
},
``` [4](#0-3) 

The `ArgumentTypes` field is passed directly to `runtime.InvokeContractFunction` as the authoritative type annotation used by the Cadence runtime to convert and validate arguments: [5](#0-4) 

`cadence.UFix64` is a fixed-point decimal (stored as `uint64` scaled by 10^8). `sema.UInt64Type` is a plain unsigned integer. These are incompatible types in Cadence's type system. The Cadence runtime's argument conversion rejects a `cadence.UFix64` value when `sema.UInt64Type` is the declared expected type, returning a type-mismatch error.

The actual on-chain Cadence contract functions (`verifyPayersBalanceForTransactionExecution` and `deductTransactionFee` in `FlowFees.cdc`) accept `UFix64` parameters for effort values, consistent with how fees are denominated throughout the Flow protocol. The specs are therefore wrong: they should declare `sema.UFix64Type`.

---

### Impact Explanation

`CheckPayerBalanceAndGetMaxTxFees` is called by `TransactionPayerBalanceChecker.CheckPayerBalanceAndReturnMaxFees` before every transaction execution: [6](#0-5) 

A failure here returns `FailureCodePayerBalanceCheckFailure`, causing the transaction to be aborted as a system failure. `DeductTransactionFees` is called by `transactionExecutor.deductTransactionFees` at the end of every transaction: [7](#0-6) 

A failure here returns `ErrCodeTransactionFeeDeductionFailedError`. Both paths are triggered for every transaction submitted to the network. The result is that all transaction processing fails at the FVM level — payer balance checks and fee deductions both abort — breaking the core execution pipeline for every user-submitted transaction.

---

### Likelihood Explanation

Every transaction submitted to the network triggers both `CheckPayerBalanceAndGetMaxTxFees` (when fees are enabled) and `DeductTransactionFees`. There is no special condition or privilege required to reach these code paths — any unprivileged user submitting any transaction exercises the vulnerable code. Likelihood is **high**.

---

### Recommendation

Replace `sema.UInt64Type` with `sema.UFix64Type` in both `verifyPayersBalanceForTransactionExecutionSpec` and `deductTransactionFeeSpec` to match the actual `cadence.UFix64` values passed at the call sites and the `UFix64` parameter types declared in the on-chain `FlowFees.cdc` contract:

```go
// verifyPayersBalanceForTransactionExecutionSpec
ArgumentTypes: []sema.Type{
    sema.NewReferenceType(..., sema.AccountType),
    sema.UFix64Type,  // was sema.UInt64Type
    sema.UFix64Type,  // was sema.UInt64Type
},

// deductTransactionFeeSpec
ArgumentTypes: []sema.Type{
    sema.NewReferenceType(..., sema.AccountType),
    sema.UFix64Type,  // was sema.UInt64Type
    sema.UFix64Type,  // was sema.UInt64Type
},
```

---

### Proof of Concept

1. Submit any transaction to a Flow network with transaction fees enabled.
2. The FVM calls `CheckPayerBalanceAndGetMaxTxFees` → `sys.Invoke(verifyPayersBalanceForTransactionExecutionSpec, ...)`.
3. `runtime.InvokeContractFunction` receives `cadence.UFix64` values but `sema.UInt64Type` as the declared argument types.
4. The Cadence runtime's argument conversion rejects the `UFix64` value as incompatible with `UInt64`, returning a type-mismatch error.
5. `CheckPayerBalanceAndReturnMaxFees` wraps this as `FailureCodePayerBalanceCheckFailure` and aborts the transaction.
6. Even if step 2–5 were bypassed, `DeductTransactionFees` → `sys.Invoke(deductTransactionFeeSpec, ...)` hits the same mismatch, aborting with `ErrCodeTransactionFeeDeductionFailedError`.

No special privileges, keys, or node compromise are required — any transaction submission exercises this path.
