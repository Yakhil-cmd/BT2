Audit Report

## Title
`Approve` Operations Bypass Legacy Fee-Collector Credit in ICRC-1 Rosetta `update_account_balances`, Causing Inaccurate Balance Reporting - (File: `rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

## Summary
The `update_account_balances` function in the ICRC-1 Rosetta API processes `Approve` operations using only the ICRC-107 fee collector (`current_fee_collector_107`), while `Transfer`, `Burn`, and `Mint` operations all use the `get_107_fee_collector_or_legacy` helper that also handles legacy fee collectors embedded in block metadata. When an `icrc2_approve` transaction is processed on a ledger with a legacy fee collector configured, the fee is debited from the `from` account in `account_balances` but never credited to the legacy fee collector, causing the derived balance table to diverge from the true ledger state. The Rosetta `/account/balance` endpoint reads exclusively from this derived table, so callers receive incorrect balances.

## Finding Description
In `update_account_balances`, the `Burn` arm (lines 345–357), `Mint` arm (lines 374–386), and `Transfer` arm (lines 477–489) all resolve the fee collector via `get_107_fee_collector_or_legacy(&rosetta_block, connection, current_fee_collector_107)`, which checks both the ICRC-107 metadata-stored collector and the legacy collector embedded in the block's `fee_collector` / `fee_collector_block_index` fields.

The `Approve` arm (lines 418–445) instead uses:
```rust
if let Some(Some(collector)) = current_fee_collector_107 {
    credit(collector, fee, ...)?;
}
```
This only credits the ICRC-107 collector. If `current_fee_collector_107` is `None` (no ICRC-107 collector active) but the block carries a legacy `fee_collector` field, the credit is silently skipped. The fee is still debited from `from` (line 429–435), so the `account_balances` table loses tokens: the `from` account is debited but no account is credited. `get_account_balance_at_block_idx` (lines 863–891) queries only `account_balances`, so this divergence is directly returned to API callers.

## Impact Explanation
This matches the allowed High impact: "Significant Rosetta … security impact with concrete user or protocol harm." Any exchange, wallet, or financial service using the Rosetta `/account/balance` endpoint against a ledger with a legacy fee collector will receive an understated balance for the fee collector account after every `icrc2_approve` call. The total of all balances in `account_balances` will be less than the true token supply tracked in `blocks`, violating ledger conservation within the Rosetta index. This is a concrete, persistent data integrity failure in a production financial integration component.

## Likelihood Explanation
`icrc2_approve` is a standard, publicly callable ICRC-2 endpoint requiring no special privileges. Legacy fee collectors are a real production configuration. Every `Approve` transaction processed by the Rosetta node on such a ledger silently widens the divergence. No attacker action is required beyond submitting a normal approval transaction; the bug is triggered automatically by the Rosetta sync loop.

## Recommendation
In the `Approve` arm of `update_account_balances`, replace the ICRC-107-only check with the same helper used by `Transfer`, `Burn`, and `Mint`:

```rust
// Replace (lines 437–445):
if let Some(Some(collector)) = current_fee_collector_107 {
    credit(collector, fee, rosetta_block.index, connection, &mut account_balances_cache)?;
}

// With:
if let Some(collector) = get_107_fee_collector_or_legacy(
    &rosetta_block, connection, current_fee_collector_107)? {
    credit(collector, fee, rosetta_block.index, connection, &mut account_balances_cache)?;
}
```

After the fix, extend `repair_fee_collector_balances` to also repair historical `Approve` blocks that were processed under the old code path.

## Proof of Concept
1. Configure an ICRC-1 ledger with a legacy fee collector (set `fee_collector` in block metadata, no ICRC-107 `FeeCollector` block).
2. Submit an `icrc2_approve` transaction from any account with a non-zero fee.
3. Call `update_account_balances` to process the block.
4. Call `get_account_balance_at_block_idx` for the legacy fee collector account — it shows no credit for the approval fee.
5. Call `get_account_balance_at_block_idx` for the `from` account — it shows the fee was debited.
6. Sum all balances in `account_balances` at that block index; the total is less than the true supply recorded in `blocks`.

This can be implemented as a deterministic unit test mirroring the existing `repair_fee_collector_balances` test (tests.rs lines 469–487), substituting an `Approve` block for the `Transfer` block at index 2 and asserting the fee collector balance remains uncredited after `update_account_balances` runs without the fix.
