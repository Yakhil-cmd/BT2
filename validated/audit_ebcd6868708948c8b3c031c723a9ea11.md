### Title
Unauthenticated Invoke Transaction Bypasses `__validate__` via `skip_stateful_validations` Frontrunning — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point check for any invoke transaction with `nonce == 1` when `account_nonce == 0`, provided `account_tx_in_pool_or_recent_block` returns `true`. Because this pool-membership check is keyed only on the **sender address** — not on any proof of ownership — any unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a crafted invoke transaction for the victim's address, bypass signature validation at the gateway, and occupy the victim's `nonce = 1` slot in the mempool, permanently blocking the victim's legitimate first invoke transaction.

---

### Finding Description

**The UX feature and its invariant**

The gateway implements a "deploy-account + invoke" UX shortcut: a user can broadcast both transactions simultaneously. Because the account does not yet exist on-chain, the gateway cannot call `__validate__` on the account contract for the invoke transaction. Instead, `skip_stateful_validations` checks whether a deploy-account transaction for that address is already in the mempool, and if so, skips `__validate__`: [1](#0-0) 

The function returns `true` (skip validation) when:
- The transaction is an `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)`
- `account_nonce == Nonce(Felt::ZERO)`
- `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, so the blockifier's `StatefulValidator` returns `Ok(())` without ever calling `__validate__`: [3](#0-2) 

**The broken invariant**

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in the mempool pool or in the mempool's committed/staged state: [4](#0-3) 

This check is keyed **only on the contract address**. It does not verify that the incoming invoke transaction was signed by the owner of that address. Any actor who can observe that address `A` has a `deploy_account` transaction in the mempool can submit an arbitrary invoke transaction for address `A` with `nonce = 1`, and the gateway will accept it without calling `__validate__`.

**The attack path**

1. Victim broadcasts `deploy_account` for address `A` (nonce 0). The gateway validates it fully (including `__validate_deploy__`) and adds it to the mempool.
2. Attacker observes this and immediately submits an `Invoke` transaction for address `A` with `nonce = 1` and arbitrary calldata (e.g., a drain call).
3. At the gateway:
   - `validate_state_preconditions` passes: `account_nonce = 0`, `tx.nonce() = 1`, within `max_allowed_nonce_gap`.
   - `validate_by_mempool` passes: no duplicate hash, nonce not too old.
   - `skip_stateful_validations` calls `account_tx_in_pool_or_recent_block(A)` → `true` (victim's deploy_account is in the pool) → returns `true`.
   - `run_validate_entry_point` sets `validate = false` → `__validate__` is **not called**.
   - Attacker's invoke transaction is accepted and forwarded to the mempool. [5](#0-4) 

4. Victim now submits their legitimate invoke transaction with `nonce = 1`. The mempool rejects it with `MempoolError::DuplicateNonce` because the attacker's transaction already occupies that slot. [6](#0-5) 

5. The victim's deploy-account + invoke UX flow is permanently broken until the attacker's transaction is evicted (e.g., via TTL expiry or fee escalation).

---

### Impact Explanation

**Admission of invalid transactions:** The gateway accepts an invoke transaction that carries no valid signature for the sender account, violating the invariant that every admitted transaction must pass `__validate__`.

**Rejection of valid transactions:** The victim's correctly-signed invoke transaction is rejected by the mempool with `DuplicateNonce` because the attacker's unsigned transaction already occupies `(address, nonce=1)`.

This matches the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The attack requires only:
1. Monitoring the public mempool (or gateway) for `deploy_account` transactions — trivially observable.
2. Submitting a single well-formed invoke transaction with `nonce = 1` for the victim's address.

No privileged access, no special knowledge of the victim's private key, and no on-chain state is required. The window of opportunity is the entire time the victim's `deploy_account` transaction sits in the mempool before being committed.

---

### Recommendation

The `skip_stateful_validations` function must not grant the validation-skip solely based on address membership in the mempool. Two concrete options:

1. **Restrict the skip to the transaction submitter's own session:** Pass the deploy-account transaction hash alongside the invoke transaction (as an explicit linkage field), and verify that the deploy-account transaction in the mempool was submitted in the same gateway request or carries a matching commitment.

2. **Do not skip `__validate__` at the gateway:** Instead, handle the "account not yet deployed" case inside the blockifier's pre-validation stage by detecting that the account class hash is zero and deferring signature verification to execution time. This removes the gateway-level bypass entirely.

At minimum, `skip_stateful_validations` should verify that no other invoke transaction at `nonce = 1` is already present in the mempool for the same address before granting the skip: [1](#0-0)

### Proof of Concept

```
// Step 1: Victim submits deploy_account for address A (nonce 0).
// Gateway: __validate_deploy__ passes. Tx added to mempool.

// Step 2: Attacker submits invoke for address A (nonce 1, arbitrary calldata).
// Gateway stateful validation:
//   validate_state_preconditions: account_nonce=0, tx_nonce=1 → OK (within gap)
//   validate_by_mempool: no duplicate hash, nonce >= account_nonce → OK
//   skip_stateful_validations:
//     tx is Invoke, tx.nonce()==1, account_nonce==0 → enters branch
//     account_tx_in_pool_or_recent_block(A) → true (victim's deploy_account is in pool)
//     returns true  ← SKIP __validate__
//   run_validate_entry_point: validate=false → __validate__ NOT called → Ok(())
// Attacker's invoke tx added to mempool at (A, nonce=1).

// Step 3: Victim submits their legitimate invoke for address A (nonce 1).
// Mempool: DuplicateNonce { address: A, nonce: 1 } → REJECTED.
// Victim's deploy_account + invoke UX flow is broken.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
