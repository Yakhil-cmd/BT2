### Title
Unbounded Loop in `calculate_rewards` Over Attacker-Grown Balance Trace Permanently Freezes Unclaimed Yield — (`File: src/pool/pool.cairo`)

---

### Summary

`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace. Because there is no cap on how many checkpoints a pool member can accumulate, and because the loop is re-entered from `entry_to_claim_from = 0` if the member never claims, a pool member who changes their balance in many distinct epochs without claiming can grow the trace until a single `claim_rewards` call exceeds the Starknet gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal function `calculate_rewards` (lines 837–888) iterates over the entire `pool_member_epoch_balance` trace from `entry_to_claim_from` to `pool_member_trace_length`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance` (called from `enter_delegation_pool`, `add_to_delegation_pool`, and `exit_delegation_pool_intent`). The underlying `insert` function in `PoolMemberBalanceTrace` appends a **new** checkpoint whenever the epoch key differs from the last stored key:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

The key used is `self.get_epoch_plus_k()` — one unique key per epoch. Therefore, a pool member who modifies their balance in each distinct epoch adds one new checkpoint per epoch, with no upper bound enforced anywhere. [3](#0-2) 

`calculate_rewards` is called exclusively from `claim_rewards`:

```cairo
let (mut rewards, updated_entry_to_claim_from) = self
    .calculate_rewards(
        :pool_member,
        from_checkpoint: pool_member_info.reward_checkpoint,
        :until_checkpoint,
        entry_to_claim_from: pool_member_info.entry_to_claim_from,
    );
``` [4](#0-3) 

The `entry_to_claim_from` cursor is only advanced and persisted **after** a successful `claim_rewards` call. If the member never claims, the cursor stays at 0 and every future `claim_rewards` must traverse the entire accumulated trace from the beginning.

Each loop iteration calls `find_sigma`, which performs multiple storage reads from `cumulative_rewards_trace`. With enough accumulated checkpoints, the total gas consumed by the loop exceeds the Starknet per-transaction gas limit, causing the transaction to revert. Because the cursor is never advanced on a revert, **every subsequent `claim_rewards` call will also revert**, permanently freezing the member's unclaimed yield.

---

### Impact Explanation

**Permanent freezing of unclaimed yield** for the affected pool member. Once the trace is large enough that `claim_rewards` always reverts, the accumulated rewards are locked in the pool contract forever with no recovery path, since there is no partial-claim or cursor-reset mechanism.

This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

The protocol is designed for long-term participation. A pool member who:
1. Enters the pool once,
2. Calls `add_to_delegation_pool` or `exit_delegation_pool_intent` in each epoch (both are normal, incentivized actions), and
3. Defers claiming rewards (also normal — there is no protocol incentive to claim every epoch),

will grow their trace at a rate of one entry per epoch. Starknet epochs are short (on the order of hours to days). Over months of normal participation, a member can accumulate hundreds to thousands of entries. Each iteration performs multiple storage reads via `find_sigma`, making the gas cost per iteration non-trivial. The threshold for hitting the gas limit is reachable through ordinary protocol usage without any adversarial intent.

---

### Recommendation

1. **Enforce a maximum trace length per pool member** — reject `set_member_balance` / `increase_member_balance` calls that would push the trace beyond a safe cap (e.g., 500 entries), or require the member to claim rewards first.
2. **Alternatively, implement partial/paginated claiming** — allow `claim_rewards` to accept a `max_entries` parameter and advance `entry_to_claim_from` incrementally across multiple transactions.
3. **Remove the dismissive comment** ("unlikely to exceed gas limits") and replace it with an enforced invariant.

---

### Proof of Concept

```
1. Deploy the staking system and pool contract.
2. Pool member A calls enter_delegation_pool(reward_address, amount).
3. For each of N epochs (e.g., N = 500):
   a. Advance to the next epoch.
   b. Call add_to_delegation_pool(pool_member_A, 1) — adds one checkpoint to
      pool_member_epoch_balance[A] because the epoch key is new.
   c. Do NOT call claim_rewards (entry_to_claim_from stays at 0).
4. After N epochs, call claim_rewards(pool_member_A).
   → calculate_rewards loops N times, each iteration calling find_sigma
     (multiple storage reads from cumulative_rewards_trace).
   → Transaction reverts with OUT_OF_GAS.
5. Repeat step 4 — every attempt reverts.
   → Pool member A's accumulated rewards are permanently frozen.
```

The codebase itself acknowledges the risk at line 858: *"This loop is unbounded but unlikely to exceed gas limits"* — confirming the developers are aware of the unbounded nature but have not enforced any limit. [5](#0-4)
