### Title
Unbounded Loop Over Attacker-Grown Balance Trace Permanently Freezes Pool Member Unclaimed Yield — (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the delegation pool contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's `pool_member_epoch_balance` trace. Because a pool member can grow this trace by one entry per epoch (via repeated `add_to_delegation_pool` or `exit_delegation_pool_intent` calls across different epochs) without ever claiming rewards, the trace can grow large enough to cause the `claim_rewards` transaction to exceed Starknet's maximum gas limit, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over every entry in the pool member's balance trace between `entry_to_claim_from` and the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    // ... compute rewards ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace (`pool_member_epoch_balance`) is a `Vec`-backed structure that appends a new checkpoint whenever a balance change is recorded at a **new epoch key**. The key used is always `current_epoch + K`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(...);
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `insert` implementation only updates the last checkpoint in-place if the key matches; otherwise it **appends** a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

This means every epoch in which a pool member calls `add_to_delegation_pool` or `exit_delegation_pool_intent` appends exactly one new entry to the trace. Both functions are accessible to any pool member: [4](#0-3) [5](#0-4) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` **succeeds**:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [6](#0-5) 

If a pool member accumulates many trace entries across epochs without ever claiming, the loop in `calculate_rewards` must traverse all of them in a single transaction. Once the trace is large enough to exceed Starknet's per-transaction gas ceiling, `claim_rewards` will always revert, and because `entry_to_claim_from` is never updated on failure, the rewards are permanently unclaimable.

The same unbounded `calculate_rewards` call is also made inside the `pool_member_info_v1` view function: [7](#0-6) 

---

### Impact Explanation

A pool member who makes balance changes across a large number of epochs without claiming rewards will eventually be unable to call `claim_rewards` successfully. Because there is no partial-claim mechanism and `entry_to_claim_from` is only advanced on success, the unclaimed yield is permanently frozen in the contract with no recovery path. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The trace grows by at most one entry per epoch. Epochs are time-bounded (governed by `epoch_duration` and `epoch_length` in `EpochInfo`). A pool member would need to make balance changes across hundreds or thousands of epochs without ever claiming rewards to approach the gas limit. This is a low-likelihood scenario for an ordinary user, but it is reachable by:

1. A pool member who is a smart contract that programmatically makes small deposits each epoch as part of a strategy, without triggering reward claims.
2. A pool member who deliberately inflates their own trace (e.g., to grief themselves or to test the limit).

The developers themselves acknowledge the risk in the code comment: *"This loop is unbounded but unlikely to exceed gas limits."* The word "unlikely" confirms the risk is known and accepted without a hard bound.

---

### Recommendation

1. **Cap trace growth**: Enforce a maximum number of unclaimed balance-change entries per pool member (e.g., reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if `pool_member_trace_length - entry_to_claim_from` exceeds a safe threshold).
2. **Partial reward claiming**: Allow `claim_rewards` to accept a `max_entries` parameter so the loop can be bounded per call, advancing `entry_to_claim_from` incrementally.
3. **Force claim on balance change**: Require a successful reward claim before any balance-modifying operation, keeping `entry_to_claim_from` always close to the trace head.

---

### Proof of Concept

```
Epoch 1:  pool_member calls add_to_delegation_pool(1)  → trace length = 1
Epoch 2:  pool_member calls add_to_delegation_pool(1)  → trace length = 2
Epoch 3:  pool_member calls add_to_delegation_pool(1)  → trace length = 3
...
Epoch N:  pool_member calls add_to_delegation_pool(1)  → trace length = N
          (entry_to_claim_from = 0 throughout, no claim_rewards called)

Epoch N+1: pool_member calls claim_rewards
           → calculate_rewards loops N times
           → if N exceeds gas limit threshold, transaction reverts
           → entry_to_claim_from remains 0
           → all subsequent claim_rewards calls also revert
           → pool member's accumulated rewards are permanently frozen
```

```text
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
