### Title
Permissionless `exit_delegation_pool_action` Allows Any Caller to Force-Exit a Pool Member, Blocking Intent Cancellation — (File: src/pool/pool.cairo)

### Summary
`Pool.exit_delegation_pool_action` accepts an arbitrary `pool_member` address and performs no caller authentication. Because the pool protocol also allows a pool member to cancel a pending exit intent by calling `exit_delegation_pool_intent(0)` (which re-adds funds to the staking balance while the staker is active), the permissionless action creates a race condition: any attacker can call `exit_delegation_pool_action(victim)` the moment the exit window elapses, permanently preventing the victim from canceling their exit and forcing them out of the pool.

### Finding Description

`exit_delegation_pool_action` in `src/pool/pool.cairo` accepts a caller-supplied `pool_member` address and contains no access-control guard:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    let unpool_time = pool_member_info
        .unpool_time
        .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
    assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);
    // ... clears unpool state and transfers funds to pool_member
    token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());
``` [1](#0-0) 

The spec confirms this is intentional: `exit_delegation_pool_action` access control is documented as **"Any address can execute."** [2](#0-1) 

However, the same protocol also allows a pool member to **cancel** a pending exit intent by calling `exit_delegation_pool_intent(0)`. The pre-condition for `exit_delegation_pool_intent` is:

> "Pool member (caller) is not in an exit window **or** staker is active." [3](#0-2) 

This means that as long as the staker is active, a pool member can call `exit_delegation_pool_intent(0)` to cancel their exit intent even after the exit window has elapsed. The internal call to `undelegate_from_staking_contract_intent` with `amount = 0` re-adds the funds to the staking balance and clears the intent in the staking contract:

```cairo
fn undelegate_from_staking_contract_intent(
    self: @ContractState, pool_member: ContractAddress, amount: Amount,
) -> Timestamp {
    if !self.is_staker_active() {
        assert!(
            self.internal_pool_member_info(:pool_member).unpool_time.is_none(),
            "{}",
            Error::UNDELEGATE_IN_PROGRESS,
        );
        return Time::now();
    }
    // staker is active: forwards amount=0 to staking contract, clearing the intent
    staking_pool_dispatcher
        .remove_from_delegation_pool_intent(
            :staker_address, identifier: pool_member.into(), :amount,
        )
}
``` [4](#0-3) 

The race condition is:

1. Pool member calls `exit_delegation_pool_intent(amount)` — funds leave the active staking balance.
2. Exit window elapses.
3. Pool member changes their mind and prepares to call `exit_delegation_pool_intent(0)` to cancel.
4. **Attacker calls `exit_delegation_pool_action(pool_member)` first** — the exit is executed, funds are returned to `pool_member`, and the pool member record is cleared.
5. The pool member's cancellation attempt now fails (`MISSING_UNDELEGATE_INTENT`).

The attacker does not need to front-run in real time. They can simply call `exit_delegation_pool_action` at any point after the window elapses, before the victim submits their cancellation transaction.

### Impact Explanation

The pool member's principal is returned to them (no direct theft), but they are forcibly removed from the pool. To resume earning rewards they must re-enter the pool and wait K epochs for their new stake to become effective. During that waiting period they earn no rewards. This constitutes **griefing with no profit motive but measurable damage to the pool member** — loss of unclaimed/future yield and a forced disruption of their staking position.

This matches the allowed impact: **Medium — Griefing with no profit motive but damage to users or protocol.**

### Likelihood Explanation

The attack requires no special privileges, no tokens, and no capital. An attacker only needs to:
- Monitor on-chain events for `PoolMemberExitIntent` emissions.
- Wait for `Time::now() >= unpool_time`.
- Submit a single call to `exit_delegation_pool_action(victim)`.

This is a low-cost, permissionless, and repeatable attack against any pool member who has signaled exit intent and whose staker is still active.

### Recommendation

Add an access-control check to `exit_delegation_pool_action` so that only the pool member themselves or their registered `reward_address` can trigger the exit action:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    let caller = get_caller_address();
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::UNAUTHORIZED_EXIT_ACTION,
    );
    // ... rest of function
```

This preserves the ability for the pool member to cancel their intent while preventing any third party from forcing the exit.

### Proof of Concept

```
1. Alice (pool_member) calls pool.exit_delegation_pool_intent(1000).
   → Alice's 1000 tokens leave the active staking balance.
   → unpool_time = now + exit_wait_window.

2. exit_wait_window elapses.

3. Alice decides to stay. She prepares to call pool.exit_delegation_pool_intent(0).

4. Attacker Bob calls pool.exit_delegation_pool_action(alice_address).
   → Passes: pool_member exists, unpool_time is set, Time::now() >= unpool_time.
   → Alice's 1000 tokens are transferred back to alice_address.
   → Alice's pool member record is cleared (unpool_time = None, unpool_amount = 0).

5. Alice's call to exit_delegation_pool_intent(0) now panics with MISSING_UNDELEGATE_INTENT
   (or POOL_MEMBER_DOES_NOT_EXIST if the record was fully cleared).

6. Alice must call enter_delegation_pool() again and wait K epochs before her
   stake becomes effective and earns rewards.
``` [1](#0-0) [5](#0-4) [6](#0-5)

        fn exit_delegation_pool_action(
            ref self: ContractState, pool_member: ContractAddress,
        ) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let unpool_time = pool_member_info
                .unpool_time
                .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
            assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberExitAction {
                        pool_member, unpool_amount: pool_member_info.unpool_amount,
                    },
                );

            // Perform removal action in the staking contract, receiving funds if needed.
            // Note that if the intent was done after the staker was removed (unstake_action),
            // the funds will already be in the pool contract, and the following call will do
            // nothing.
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_action(identifier: pool_member.into());

            let unpool_amount = pool_member_info.unpool_amount;
            pool_member_info.unpool_amount = Zero::zero();
            pool_member_info.unpool_time = Option::None;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer delegated amount to the pool member.
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());

            unpool_amount
        }
