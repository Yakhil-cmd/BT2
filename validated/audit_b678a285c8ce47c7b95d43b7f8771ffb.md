### Title
`skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures When a `deploy_account` Is Pending in the Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function intentionally bypasses the `__validate__` entry-point call (the account's signature-verification step) for an invoke transaction whose nonce is `1` and whose sender has a pending `deploy_account` in the mempool. The guard that decides whether to skip is `account_tx_in_pool_or_recent_block`, which only checks whether *any* transaction from that address is known to the mempool — it does not verify that the pending transaction is actually a `deploy_account`. An attacker can therefore submit an invoke transaction carrying a completely fabricated signature for any pre-funded, not-yet-deployed account that has a `deploy_account` in flight, and the gateway will admit it to the mempool without ever running the account's `__validate__` function.

---

### Finding Description

`extract_state_nonce_and_run_validations` in `StatefulTransactionValidator` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range + resource-bound price check)
       ├─ validate_by_mempool            (duplicate / nonce-gap check in mempool)
       └─ skip_stateful_validations      ← decides whether __validate__ is called
```

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate` is `false`, execution returns immediately after `perform_pre_validation_stage` — the `__validate__` call is never reached:

```rust
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

`perform_pre_validation_stage` only checks nonce ordering, fee-bound arithmetic, and proof-facts structure. It does **not** invoke the account contract and therefore does **not** verify the transaction signature:

```rust
pub fn perform_pre_validation_stage<S: State + StateReader>(
    &self, state: &mut S, tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
    if self.execution_flags.charge_fee {
        self.check_fee_bounds(tx_context)?;
        verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
    }
    self.validate_proof_facts(&tx_context.block_context, state)?;
    Ok(())
}
``` [4](#0-3) 

The guard `account_tx_in_pool_or_recent_block` is defined as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

This returns `true` for any account that has *any* transaction in the pool or was seen in a recent committed block — not specifically a `deploy_account`. The comment in `skip_stateful_validations` acknowledges this looseness: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* That reasoning is circular: future-nonce transactions that "passed validations" may themselves have been admitted via this same skip path.

---

### Impact Explanation

An attacker who observes a `deploy_account` transaction for address `A` in the mempool can immediately submit an `InvokeV3` transaction with:
- `sender_address = A`
- `nonce = 1`
- `signature = [0x41, 0x41, ...]` (arbitrary garbage)
- resource bounds ≥ `min_gas_price` (to pass stateless validation)

The gateway will:
1. Pass stateless checks (valid address, non-zero bounds, correct DA modes).
2. Pass `validate_nonce` (account_nonce=0 ≤ tx_nonce=1 ≤ 0+gap).
3. Pass `validate_by_mempool` (no duplicate hash).
4. Call `skip_stateful_validations` → `account_tx_in_pool_or_recent_block(A)` = `true` → returns `true`.
5. Call `run_validate_entry_point` with `validate=false` → `__validate__` is never executed.
6. Admit the garbage-signature invoke to the mempool.

The admitted transaction occupies nonce slot 1 for address `A`. The legitimate owner's nonce-1 invoke will be rejected by the mempool as a duplicate nonce (or require fee escalation). When the batcher eventually executes the garbage-signature transaction, `__validate__` will run and revert, but the nonce is already consumed and the account is charged for the validation gas. The real owner's nonce-1 transaction is permanently blocked unless they perform fee escalation.

This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The preconditions are common in production:
- New account deployments always involve a pre-funded address followed by a `deploy_account` transaction.
- The window between the `deploy_account` entering the mempool and being included in a block is observable on-chain and via the gateway's pending-transaction API.
- No privileged access is required; any unprivileged actor can submit an `InvokeV3` transaction.

---

### Recommendation

1. **Short term**: In `skip_stateful_validations`, replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction (not just any transaction) is pending for the address. Introduce a dedicated mempool query such as `deploy_account_tx_in_pool(address)` that only returns `true` when the pending transaction is of type `DeployAccount`.

2. **Medium term**: Record the hash of the `deploy_account` transaction at the time the paired invoke is admitted (as the `native_blockifier` path does via `deploy_account_tx_hash`), and verify it still exists in the mempool at execution time.

3. **Long term**: Add an integration test that submits a garbage-signature invoke paired with a valid `deploy_account` and asserts the gateway rejects it.

---

### Proof of Concept

```
1. Alice submits RpcDeployAccountTransactionV3 for address A (nonce=0).
   → Mempool admits it; account_tx_in_pool_or_recent_block(A) now returns true.

2. Eve submits RpcInvokeTransactionV3:
     sender_address = A
     nonce          = 1
     signature      = [0xdead, 0xbeef]   // garbage
     resource_bounds.l2_gas.max_price_per_unit = min_gas_price   // passes stateless check
     resource_bounds.l2_gas.max_amount = 1

3. Gateway stateless validator: passes (non-zero bounds, valid address, correct DA modes).

4. convert_rpc_tx_to_internal: computes tx_hash, no compiled_class_hash check needed for invoke.

5. extract_state_nonce_and_run_validations:
   a. get_nonce_from_state(A) → Nonce(0)
   b. validate_nonce: 0 ≤ 1 ≤ 0+gap → OK
   c. validate_resource_bounds: price ≥ threshold → OK
   d. validate_by_mempool: no duplicate hash → OK
   e. skip_stateful_validations:
        tx.nonce() == 1  ✓
        account_nonce == 0  ✓
        account_tx_in_pool_or_recent_block(A) == true  ✓
      → returns true
   f. run_validate_entry_point(skip_validate=true):
        execution_flags.validate = false
        perform_pre_validation_stage: nonce incremented, fee-bound check passes
                                      (A is pre-funded with STRK)
        __validate__ NOT called  ← garbage signature never verified
      → returns Ok(())

6. Gateway calls mempool.add_tx with Eve's garbage-signature invoke at nonce 1.
   → Mempool admits it.

7. Alice's legitimate nonce-1 invoke is now rejected as DuplicateNonce.
```
