### Title
`validateCOAOwnershipProof` Signatures Do Not Commit to `evmAddress`, Enabling Cross-COA Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account key signatures are valid over caller-supplied `signedData`, then separately checks that the COA at the given `path` matches the supplied `evmAddress`. It never enforces that `signedData` encodes `evmAddress`. A signature produced to prove ownership of COA_A can therefore be replayed verbatim to prove ownership of COA_B — a different COA owned by the same Cadence account — by simply substituting a different `path` and `evmAddress` in the call. The code itself acknowledges this gap in a comment.

### Finding Description

`validateCOAOwnershipProof` is declared `access(all)` and is callable from any unprivileged Cadence script or transaction. [1](#0-0) 

The function performs two independent checks:

1. **Signature check** (line 1082–1086): `keyList.verify(signatureSet: signatureSet, signedData: signedData, ...)` — verifies the signatures are valid over `signedData`.
2. **Address check** (lines 1095–1105): borrows the COA capability at `path` and compares its `.address().bytes` against the caller-supplied `evmAddress`. [2](#0-1) 

Nowhere does the function verify that `signedData` encodes `evmAddress`. The two checks are entirely independent. The code comment at line 1003 explicitly acknowledges: *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [3](#0-2) 

**Exploit path:**

1. Alice owns a Cadence account with two COAs: COA_A published at `/public/coaA` and COA_B published at `/public/coaB`.
2. A protocol asks Alice to prove ownership of COA_A by calling `validateCOAOwnershipProof` with `signedData = D`, `path = /public/coaA`, `evmAddress = COA_A_addr`. Alice signs `D` with her account keys and the proof passes.
3. An attacker observes Alice's signatures over `D` (e.g., from an on-chain event or off-chain API response).
4. The attacker submits a Cadence script calling `EVM.validateCOAOwnershipProof(address: Alice, path: /public/coaB, signedData: D, keyIndices: [...], signatures: [...], evmAddress: COA_B_addr)`.
5. Step 1 passes: the signatures are valid over `D` (same data, same keys). Step 2 passes: the COA at `/public/coaB` has address `COA_B_addr`. The function returns `isValid: true`.

The attacker has produced a valid COA ownership proof for COA_B using a signature that was never intended for COA_B.

The EVM precompile path (`verifyCOAOwnershipProof` called from the COA's `isValidSignature`) is **not** affected because the precompile always passes `address(this)` as the EVM address: [4](#0-3) 

However, the Cadence-level function is `access(all)` and is directly reachable by any unprivileged script or transaction without going through the precompile.

### Impact Explanation

Any on-chain or off-chain protocol that calls `EVM.validateCOAOwnershipProof` directly from Cadence (rather than via the EVM precompile) and uses the result to gate asset transfers, authorization decisions, or bridge operations is vulnerable to cross-COA impersonation. An attacker who observes a valid ownership proof for COA_A can replay it to impersonate the owner of COA_B, potentially gaining unauthorized access to assets or privileges associated with COA_B in that protocol.

The `COAOwnershipProof` type is explicitly designed as a bridge authentication primitive: [5](#0-4) 

Unauthorized proof acceptance for the wrong COA address is a cross-VM asset-loss / bridge mis-accounting class of impact.

### Likelihood Explanation

Likelihood is **low-to-medium**:
- Requires the same Cadence account to hold multiple COAs at different public paths (uncommon today, but not prohibited by the protocol).
- Requires a protocol to call `validateCOAOwnershipProof` directly from Cadence rather than via the EVM precompile.
- The attacker must observe a prior valid signature over some `signedData` for the victim account.
- The code comment explicitly warns downstream callers, but the warning is advisory only — the function itself does not enforce the constraint.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` commits to `evmAddress`. The simplest fix is to require that `signedData` is the hash of `evmAddress || application_data`, or to internally prepend `evmAddress` to the data before calling `keyList.verify`, so that a signature produced for one EVM address cannot be accepted as proof of ownership for a different EVM address.

### Proof of Concept

```cadence
// Attacker script — no privileged access required
import EVM from 0x1

access(all)
fun main(): EVM.ValidationResult {
    // signedData, keyIndices, signatures were observed from Alice's prior
    // proof for COA_A. We substitute COA_B's path and address.
    return EVM.validateCOAOwnershipProof(
        address: aliceCadenceAddress,
        path: /public/coaB,          // COA_B's path, not COA_A's
        signedData: observedSignedData,
        keyIndices: observedKeyIndices,
        signatures: observedSignatures,
        evmAddress: coaBEvmAddress   // COA_B's EVM address
    )
    // Returns isValid: true — proof accepted for the wrong COA
}
```

The function verifies the signatures over `observedSignedData` (valid, same keys), then checks that the COA at `/public/coaB` has address `coaBEvmAddress` (valid, it does). No check ever ties the signature to `coaBEvmAddress`, so the replay succeeds. [6](#0-5) [7](#0-6)
