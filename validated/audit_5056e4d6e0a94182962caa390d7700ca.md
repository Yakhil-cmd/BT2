### Title
Rewards Permanently Frozen in Delegation Pool When Pool Balance Is Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a delegation pool's total balance is below `min_delegation_for_rewards` (1 STRK = 10^18 wei for STRK pools), the staking contract still calculates and physically transfers STRK reward tokens to the pool contract, but `compute_rewards_per_unit` silently returns zero, leaving the cumulative rewards trace unchanged. No delegator can ever claim those transferred tokens. They are permanently frozen in the pool contract.

---

### Finding Description

**Vulnerability class**: Token-freeze / accounting bug — analogous to M-6 (rewards forwarded to a pool with zero/sub-threshold balance are permanently lost).

**Root cause — `compute_rewards_per_unit` in `src/pool/pool.cairo`:**

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
```

The code's own comment at the call site acknowledges the problem:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [1](#0-0) 

**How rewards arrive at the pool despite the zero return:**

In `calculate_staker_pools_rewards` (staking.cairo), for each pool the staking contract computes `pool_rewards_including_commission` proportional to `pool_balance_curr_epoch / total_stake`. If `pool_balance_curr_epoch` is non-zero (even 1 wei), this product can be non-zero, so the pool is appended to `pool_rewards_array`:

```cairo
if pool_rewards.is_non_zero() {
    pool_rewards_array
        .append(
            (pool_contract, token_address, pool_balance_curr_epoch, pool_rewards),
        );
}
``` [2](#0-1) 

Then `update_pool_rewards` physically transfers STRK to the pool contract and calls `update_rewards_from_staking_contract`:

```cairo
self.send_rewards_to_delegation_pool(
    :staker_address,
    pool_address: pool_contract,
    amount: pool_rewards,
    token_dispatcher: strk_token_dispatcher,
);
...
pool_dispatcher
    .update_rewards_from_staking_contract(
        rewards: pool_rewards,
        pool_balance: pool_balance.to_native_amount(:decimals),
    );
``` [3](#0-2) 

Inside `update_rewards_from_staking_contract`, the cumulative rewards trace is updated by adding `compute_rewards_per_unit(...)`. When `pool_balance < min_delegation_for_rewards`, that function returns 0, so the trace is incremented by 0:

```cairo
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    self.assert_caller_is_staking_contract();
    let (_, last) = self.cumulative_rewards_trace.last().unwrap();
    self
        .cumulative_rewards_trace
        .insert(
            key: self.get_current_epoch(),
            value: last
                + self
                    .compute_rewards_per_unit(
                        staking_rewards: rewards, total_stake: pool_balance,
                    ),
        );
}
``` [4](#0-3) 

Because the trace is not advanced, `calculate_rewards` in the pool contract will compute zero rewards for every delegator for those epochs, even though STRK tokens are sitting in the pool contract. There is no sweep or recovery function in the pool contract; the tokens are permanently frozen.

---

### Impact Explanation

STRK tokens are physically transferred to the pool contract (`send_rewards_to_delegation_pool` executes an ERC-20 transfer) but the cumulative rewards index is not updated. No delegator can ever claim them via `claim_rewards`, and no function exists to recover them. This is **permanent freezing of unclaimed yield**, matching the allowed High-severity impact.

---

### Likelihood Explanation

The only guard on `enter_delegation_pool` is `amount.is_non_zero()` — there is no minimum delegation floor:

```cairo
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
``` [5](#0-4) 

Any unprivileged delegator can delegate 1 wei of STRK. If the total pool balance stays below `10^18` wei (1 STRK), every subsequent reward distribution permanently freezes a portion of STRK in the pool contract. The condition is reachable without any privileged access, and the `min_delegation_for_rewards` threshold for STRK is `10^18`:

```cairo
pub(crate) const STRK_CONFIG: TokenRewardsConfig = TokenRewardsConfig {
    decimals: 18, min_for_rewards: 10_u128.pow(18), base_value: 10_u128.pow(28),
};
``` [6](#0-5) 

---

### Recommendation

In `calculate_staker_pools_rewards`, skip sending rewards to a pool whose native balance is below `min_delegation_for_rewards`. Concretely, before appending to `pool_rewards_array`, check whether `pool_balance.to_native_amount(:decimals) >= min_delegation_for_rewards`. This mirrors the fix recommended in M-6: do not forward rewards when the effective supply is too small to account for them.

Alternatively, enforce a minimum delegation amount in `enter_delegation_pool` equal to `min_delegation_for_rewards`.

---

### Proof of Concept

1. Staker calls `stake(...)` with pool enabled (`set_open_for_delegation`).
2. Delegator calls `enter_delegation_pool(reward_address, amount: 1)` — delegates 1 wei of STRK.
3. After K epochs the pool balance becomes effective: `pool_balance_curr_epoch = 1` (normalized, converts back to 1 native wei).
4. `update_rewards` (or `update_rewards_from_attestation_contract`) is called.
5. `calculate_staker_pools_rewards` computes `pool_rewards_including_commission = total_rewards * 1 / total_stake > 0` (assuming total_stake is large but finite).
6. `pool_rewards > 0`, so the pool is appended to `pool_rewards_array`.
7. `update_pool_rewards` calls `send_rewards_to_delegation_pool` → STRK ERC-20 transfer to pool contract succeeds.
8. `update_rewards_from_staking_contract(rewards: pool_rewards, pool_balance: 1)` is called.
9. `compute_rewards_per_unit(staking_rewards: pool_rewards, total_stake: 1)` → `1 < 10^18` → returns `0`.
10. `cumulative_rewards_trace` is updated with `last + 0 = last` — no change.
11. Delegator calls `claim_rewards` → `calculate_rewards` computes `0` for all epochs → transfer of 0 STRK.
12. The STRK tokens transferred in step 7 are permanently frozen in the pool contract with no recovery path.
