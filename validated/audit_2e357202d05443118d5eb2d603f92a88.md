### Title
Malicious Coordinator Can Present Different Participant Sets to Different Signers, Enabling Secret Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`)

### Summary
The `sign()` function in `src/ecdsa/robust_ecdsa/sign.rs` accepts a caller-supplied `participants` list that is used to compute each signer's Lagrange coefficient, but provides no mechanism to ensure all participants agree on the same list. A malicious coordinator can present different participant sets to different signers within the same signing session, causing each signer to linearize their share with a different Lagrange coefficient. The resulting inconsistent shares, collected across two signing sessions with different message hashes, allow the coordinator to extract the aggregate secret key via a split-view / nonce-reuse attack.

### Finding Description

The robust ECDSA signing protocol computes each participant's signature share as:

```
s_i = λ_i(participants) · (α_i · h + β_i · Rx + e_i)
```

where `λ_i` is the Lagrange coefficient for participant `i` evaluated over the signing set `participants`. Correctness requires that every participant uses the **same** `participants` list so that the shares sum to the correct value `h/k + x·Rx/k + e/k`.

The `sign()` function enforces:
- `participants.len() == 2t+1`
- `me ∈ participants`
- `coordinator ∈ participants`
- `msg_hash ≠ 0` [1](#0-0) 

However, it does **not** enforce that all participants agree on the same `participants` list. Each participant trusts the list passed to `sign()` by the caller (the coordinator or orchestrator). There is no broadcast/consensus step where participants verify each other's view of the signing set. [2](#0-1) 

The Lagrange coefficient is computed locally in `compute_signature_share()` using whatever `participants` list was passed to `sign()`: [3](#0-2) 

A malicious coordinator can:
1. Run presigning with N₁ = 2t+1 participants `{P1, …, P_{2t+1}}`.
2. In a signing session, call `sign()` for `P1` with `list_A = {P1,P2,P3,P4,P5}` and for `P2,…,P5` with `list_B = {P1,P2,P3,P4,P6}` (where `P6` is a different participant ID, possibly fabricated).
3. `P1` computes `s_1 = λ_1(list_A) · s_1_raw`; `P2` computes `s_2 = λ_2(list_B) · s_2_raw`, etc.
4. The coordinator collects shares computed with **inconsistent** Lagrange coefficients.
5. Repeating across two sessions with different `h` values yields two linear equations in the secret key shares, enabling full key recovery via standard ECDSA nonce-reuse algebra.

The library's own security documentation explicitly acknowledges this class of attack and states the mitigation requirement that the code does not implement: [4](#0-3) 

The `do_sign_coordinator()` function does verify the final signature, but the coordinator is the attacker and can abort after collecting shares, or simply use the sums from two sessions without needing a valid output from either: [5](#0-4) 

### Impact Explanation
**Critical — secret key extraction.** The documentation states: *"a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."* Inconsistent Lagrange coefficients produce shares that, when combined across two sessions with different message hashes, allow the coordinator to solve for the aggregate private key `x` using standard ECDSA nonce-reuse techniques. This constitutes unauthorized extraction of the aggregate secret material.

### Likelihood Explanation
The coordinator is a protocol participant and is explicitly **not** fully trusted — the security documentation states *"The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers."* Any participant designated as coordinator can trivially execute this attack by constructing two different `ParticipantList` objects and passing them to different signers' `sign()` calls. No external assumptions, leaked keys, or cryptographic breaks are required.

### Recommendation
Add a broadcast/consensus step at the start of the signing protocol (analogous to `broadcast_success()` in DKG) where every participant broadcasts a commitment to their view of `(participants, msg_hash, tweak)` and verifies that all views are identical before computing their signature share. This prevents the coordinator from presenting inconsistent inputs to different signers. [6](#0-5) 

Alternatively, bind the participant list into the `RerandomizationArguments` HKDF derivation and verify inside `sign()` that the `participants` argument matches the list embedded in the `RerandomizedPresignOutput`, making any split-view attempt produce a cryptographically detectable inconsistency.

### Proof of Concept

```
Setup: t = 2, N = 2t+1 = 5 participants {P1, P2, P3, P4, P5}

Step 1: Run presign() with all 5 participants → each gets (big_r, α_i, β_i, c_i, e_i)

Step 2 (Session A, h₁):
  Coordinator calls sign() for P1 with participants = {P1,P2,P3,P4,P5}
  Coordinator calls sign() for P2..P5 with participants = {P1,P2,P3,P4,P6}
  → P1 uses λ_1({P1..P5}), P2..P5 use λ_i({P1..P5,P6})
  → Coordinator collects sum_A = Σ λ_i(S_i) · (α_i·h₁ + β_i·Rx + e_i)

Step 3 (Session B, h₂ ≠ h₁, same presignature):
  Coordinator repeats with a different split of participant lists
  → Coordinator collects sum_B = Σ λ_i(S_i') · (α_i·h₂ + β_i·Rx + e_i)

Step 4: sum_A and sum_B are two linear equations in the secret key shares.
  Standard ECDSA nonce-reuse algebra recovers x (the aggregate private key).
```

The `sign()` function has no check preventing the coordinator from supplying different `participants` slices to different callers, and `do_sign_participant()` sends its share privately to the coordinator with no cross-participant verification. [2](#0-1) [3](#0-2)

async fn do_sign_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<SignatureOption, ProtocolError> {
    let mut s = compute_signature_share(&presignature, msg_hash, &participants, me)?.0;
    let wait_round = chan.next_waitpoint();

    for (_, s_i) in
        recv_from_others::<SerializableScalar<C>>(&chan, wait_round, &participants, me).await?
    {
        // Sum the linearized shares
        s += s_i.0;
    }

    // raise error if s is zero
    if s.is_zero().into() {
        return Err(ProtocolError::AssertionFailed(
            "signature part s cannot be zero".to_string(),
        ));
    }
    // Normalize s
    s.conditional_assign(&(-s), s.is_high());

    let sig = Signature {
        big_r: presignature.big_r,
        s,
    };

    if !sig.verify(&public_key, &msg_hash) {
        return Err(ProtocolError::AssertionFailed(
            "signature failed to verify".to_string(),
        ));
    }

    Ok(Some(sig))

/// A common computation done by both the coordinator and the other participants
fn compute_signature_share(
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
    participants: &ParticipantList,
    me: Participant,
) -> Result<SerializableScalar<C>, ProtocolError> {
    // (beta_i + tweak * k_i) * delta^{-1}
    let big_r = presignature.big_r;
    let big_r_x_coordinate = x_coordinate(&big_r);
    // beta * Rx + e
    let beta = presignature.beta * big_r_x_coordinate + presignature.e;

    let s_me = msg_hash * presignature.alpha + beta;
    // lambda_i * s_i
    let linearized_s_me = s_me * participants.lagrange::<C>(me)?;
    Ok(SerializableScalar::<C>(linearized_s_me))
}

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
