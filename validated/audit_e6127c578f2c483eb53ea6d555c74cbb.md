Looking at the external report's vulnerability class — a publicly-visible state-changing transaction that can be front-run/back-run to extract value — I need to find an analog in Starknet Staking where a rate/parameter update can be exploited.

After thorough investigation, I found a valid analog: a missing access control on `update_rewards` that allows any caller to consume the global `last_reward_block` slot with `disable_rewards: true`, permanently denying rewards for that block to all stakers.

---

### Title
Missing Access Control on `update_rewards` Allows Anyone to Grief Stakers by Blocking Reward Distribution - (File: `src/staking/staking.cairo`)

### Summary

The `update_rewards` function in the `Staking` contract has no caller restriction in the implementation, despite the specification explicitly requiring "Only starkware sequencer." Any unprivileged caller can invoke `update_rewards(staker_address, disable_rewards: true)` to consume the global `last_reward_block` slot for the current block without distributing any rewards, permanently denying all stakers their block rewards for that block.

### Finding Description

The spec at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` only calls `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — no sequencer identity check is performed.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();  // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);  // global slot consumed

    if disable_rewards || self.is_pre_consensus() {
        return;  // exits WITHOUT distributing rewards
    }
``` [1](#0-0) 

The `last_reward_block` is a **global** storage variable — not per-staker. Once it is written to the current block number, no further `update_rewards` call can succeed in that block for **any** staker, because all subsequent calls revert with `REWARDS_ALREADY_UPDATED`. [2](#0-1) 

The `general_prerequisites` function confirms there is no sequencer check:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [3](#0-2) 

The spec's access control requirement is unambiguous: [4](#0-3) 

### Impact Explanation

An attacker calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block before the legitimate sequencer call. This:

1. Sets `last_reward_block` to the current block number.
2. Returns immediately without distributing any rewards.
3. Causes all subsequent `update_rewards` calls in that block to revert with `REWARDS_ALREADY_UPDATED`.

All stakers lose their block rewards for every block the attacker front-runs. Since `update_rewards` is the sole mechanism for distributing consensus (V3) block rewards to stakers and their pools, sustained griefing permanently freezes unclaimed yield accumulation across the entire protocol.

This matches the allowed impact: **Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation

- The function is publicly callable with no access restriction.
- The attacker only needs to submit a transaction with a valid (existing, active) `staker_address` and `disable_rewards: true`.
- The gas cost is low (no token transfers, no reward computation).
- On Starknet, the sequencer controls intra-block ordering, which mitigates this in the normal case. However, if the sequencer does not call `update_rewards` in every block (which is the expected operational pattern — it is called once per block per staker), an attacker can fill the slot in any block the sequencer skips.
- The attack is economically rational for a competitor or malicious actor wishing to damage the protocol's stakers at minimal cost.

### Recommendation

Add an explicit sequencer-only check to `update_rewards`, consistent with the specification. For example, store the authorized sequencer address at construction and assert it in `update_rewards`:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

Alternatively, if the function is intentionally public, the `disable_rewards` parameter should be removed or restricted so that only the sequencer can pass `disable_rewards: true`.

### Proof of Concept

1. Deploy the system and advance to consensus rewards epoch.
2. Attacker (any address) calls `update_rewards(staker_address, disable_rewards: true)` in block N.
3. `last_reward_block` is set to N; no rewards are distributed.
4. Sequencer attempts `update_rewards(staker_address, disable_rewards: false)` in block N → reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker receives zero rewards for block N.
6. Repeated every block: staker's `unclaimed_rewards_own` never increases despite being an active, attesting staker.

The `REWARDS_ALREADY_UPDATED` revert path is confirmed by the existing test suite: [5](#0-4)
