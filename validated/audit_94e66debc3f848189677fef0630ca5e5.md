### Title
Unauthorized ERC-20 Token Minting via Direct `ft_on_transfer` Call from Malicious NEAR Contract — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entry point in Aurora Engine performs no verification that the call originated from a legitimate NEP-141 `ft_transfer_call` flow. Any unprivileged NEAR account (including a malicious contract) can call `ft_on_transfer` directly on Aurora, supplying attacker-controlled `sender_id` and `amount` arguments, and cause the engine to mint ERC-20 tokens without any real NEP-141 token transfer having occurred. Because `deploy_erc20_token` is also unrestricted, the attacker can first register their own fake NEP-141 contract and then exploit this path end-to-end.

---

### Finding Description

`ft_on_transfer` is the NEP-141 receiver callback. In the legitimate flow it is invoked by a real token contract after `ft_transfer_call`, so the predecessor is the token contract and the arguments (`sender_id`, `amount`, `msg`) are set by that contract. Aurora's implementation reads those arguments from the raw NEAR input and branches on the predecessor: [1](#0-0) 

```rust
let args: FtOnTransferArgs = read_json_args(&io)?;
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
```

There is **no guard** that enforces the call came through a legitimate `ft_transfer_call`. A malicious NEAR contract can call `ft_on_transfer` directly on Aurora, making itself the predecessor and supplying arbitrary `amount` and `sender_id` values. Aurora will then execute `receive_erc20_tokens` using the attacker's contract as the NEP-141 token identity and the attacker-supplied amount, minting ERC-20 tokens with no real backing.

The prerequisite step — registering the attacker's fake NEP-141 contract as an ERC-20 token — is also open to any caller: [2](#0-1) 

`deploy_erc20_token` has only `require_running`, no `require_owner_only` or allowlist check, so any NEAR account can register any NEP-141 account ID and obtain an ERC-20 mirror address.

---

### Impact Explanation

An attacker mints ERC-20 tokens on Aurora that are backed by zero real NEP-141 tokens. These unbacked tokens can be:

- Deposited into Aurora-native DEX liquidity pools to drain real assets held by other users.
- Used as collateral in Aurora lending protocols to borrow real tokens.
- Transferred to other EVM addresses and sold before the discrepancy is detected.

This constitutes **direct theft of user funds** held in Aurora DeFi protocols that accept the minted token, and breaks the 1:1 bridge accounting invariant between NEP-141 and ERC-20 balances.

**Impact rating: High** — theft of funds / unexpected minting of tokens.

---

### Likelihood Explanation

The attack requires only:
1. Deploying a NEAR smart contract (permissionless on NEAR mainnet).
2. Calling `deploy_erc20_token` on Aurora (unrestricted).
3. Calling `ft_on_transfer` on Aurora from the malicious contract (unrestricted).

No admin keys, governance capture, or social engineering are needed. Any NEAR account holder can execute this sequence. **Likelihood: Low-to-Moderate** — the steps are non-trivial but fully permissionless.

---

### Recommendation

Add a predecessor allowlist check inside `ft_on_transfer`. The predecessor must be a NEAR account that has a registered ERC-20 ↔ NEP-141 mapping **and** must have initiated a real `ft_transfer_call`. At minimum, reject calls whose predecessor is not already registered in the ERC-20/NEP-141 map before attempting to mint. Additionally, restrict `deploy_erc20_token` to the contract owner or a governance-controlled allowlist to prevent arbitrary NEP-141 registration.

---

### Proof of Concept

1. **Attacker deploys** `evil.near` — a minimal NEAR contract that can call arbitrary methods on other contracts.

2. **Attacker registers** `evil.near` as a NEP-141 token on Aurora:
   ```
   aurora.call("deploy_erc20_token", borsh_encode(AccountId("evil.near")))
   ```
   No access control blocks this call. [3](#0-2) 

3. **Attacker calls** `ft_on_transfer` on Aurora directly from `evil.near`:
   ```json
   {
     "sender_id": "attacker.near",
     "amount": "1000000000000000000000",
     "msg": "<evm_address_of_attacker>"
   }
   ```
   The predecessor is `evil.near`, which is not the connector account, so the engine enters the `receive_erc20_tokens` branch. [4](#0-3) 

4. Aurora mints `1000000000000000000000` units of the ERC-20 mirror of `evil.near` to the attacker's EVM address — **with no real NEP-141 tokens ever transferred**.

5. Attacker swaps the minted tokens on an Aurora DEX for real ETH or other assets, draining liquidity providers.
