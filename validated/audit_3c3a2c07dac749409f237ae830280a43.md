### Title
Gateway stateful and stateless validators skip L1/L1-data gas price threshold checks, admitting transactions that fail blockifier pre-validation — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateless and stateful validators enforce a minimum gas price only on the **L2 gas** dimension. Neither validator checks `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` against the current block's L1/L1-data gas prices. The mempool's `ValidationArgs` likewise carries only `max_l2_gas_price`. The blockifier's `check_fee_bounds` is the sole place that validates all three gas prices, but it runs only after the transaction has already been admitted to the mempool and dequeued by the batcher. An attacker can therefore flood the mempool with transactions that carry a compliant L2 gas price but a trivially low L1/L1-data gas price (e.g., 1), paying near-zero effective fees while consuming mempool slots and batcher pre-validation cycles.

---

### Finding Description

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A transaction with `l1_gas.max_price_per_unit = 1` and `l1_data_gas.max_price_per_unit = 1` is definitionally invalid: the blockifier will reject it at `check_fee_bounds` because those prices are below the block's L1/L1-data gas prices. Yet the transaction clears every admission gate (stateless validator, stateful validator, mempool `validate_tx`) and occupies a mempool slot. An attacker can sustain a continuous stream of such transactions, displacing legitimate transactions from the bounded mempool and forcing the batcher to repeatedly dequeue-and-discard them. The cost to the attacker is only the L2 gas fee (which is checked), and that can be minimised by setting `l2_gas.max_amount` to the smallest non-zero value that satisfies the zero-bounds check.

---

### Likelihood Explanation

**Medium-High.** The attack requires no privileged access. The attacker only needs to know the current L2 gas price (publicly observable from the previous block header) to craft a transaction that passes the L2 threshold while setting L1/L1-data prices to 1. No special contract deployment or signature forgery is needed. The `validate_resource_bounds` flag defaults to `true` in production, so the L2 check is active, but the L1/L1-data gap is always present regardless of configuration. [8](#0-7) 

---

### Recommendation

1. **Stateless validator**: extend `validate_resource_bounds` to compare `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against configured minimums (analogous to the existing `min_gas_price` for L2).

2. **Stateful validator**: resolve the TODO in `validate_tx_l2_gas_price_within_threshold` by also comparing `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against the previous block's L1/L1-data gas prices (with the same configurable percentage tolerance already used for L2).

3. **Mempool `ValidationArgs`**: add `max_l1_gas_price` and `max_l1_data_gas_price` fields so the mempool can enforce the same threshold before admitting a transaction.

---

### Proof of Concept

```
Submit RpcInvokeTransactionV3 with:
  resource_bounds.l2_gas.max_price_per_unit  = <current block L2 gas price>   // passes stateless + stateful
  resource_bounds.l2_gas.max_amount          = 1                               // minimises L2 fee
  resource_bounds.l1_gas.max_price_per_unit  = 1                               // below block L1 price, NOT checked
  resource_bounds.l1_gas.max_amount          = 1_000_000
  resource_bounds.l1_data_gas.max_price_per_unit = 1                           // below block L1-data price, NOT checked
  resource_bounds.l1_data_gas.max_amount     = 1_000_000
```

Expected result:
- `StatelessTransactionValidator::validate` → **Ok** (L2 price ≥ 8 gwei, total fee > 0)
- `StatefulTransactionValidator::validate_tx_l2_gas_price_within_threshold` → **Ok** (L2 price meets threshold)
- `Mempool::validate_tx` → **Ok** (only L2 price checked)
- Transaction enters mempool.
- Batcher dequeues it; `AccountTransaction::perform_pre_validation_stage` → `check_fee_bounds` → **`ResourceBoundsError::MaxGasPriceTooLow { resource: L1Gas }`** — transaction discarded after consuming a mempool slot and a batcher cycle.

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
