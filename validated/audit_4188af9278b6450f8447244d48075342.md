### Title
Rewards Permanently Locked in Pool Contract When Total Delegation Falls Below `min_delegation_for_rewards` - (File: src/pool/pool.cairo)

### Summary
When the total delegated balance in a pool contract drops below `min_delegation_for_rewards` (1 STRK for STRK pools), the pool's `compute_rewards_per_unit` silently returns zero, discarding all reward accounting. However, the staking contract still physically transfers STRK reward tokens to the pool contract. These tokens accumulate in the pool with no mechanism for any party to claim or recover them, resulting in permanent freezing of unclaimed yield.

### Finding Description
The pool contract stores a `min_delegation_for_rewards` threshold (set to `10^18` fractions = 1 STRK for STRK pools) in its constructor:

```cairo
// src/pool/pool.cairo lines 164-166
let config = get_token_rewards_config(:token_address);
self.min_delegation_for_rewards.write(config.min_for_rewards);
```

The `STRK_CONFIG` constant sets this to exactly 1 STRK:

```cairo
// src/pool/pool.cairo lines 62-64
pub(crate) const STRK_CONFIG: TokenRewardsConfig = TokenRewardsConfig {
    decimals: 18, min_for_rewards: 10_u128.pow(18), base_value: 10_u128.pow(28),
};
```

When the staking contract distributes rewards to the pool, it calls `update_rewards_from_staking_contract`, which internally calls `compute_rewards_per_unit`:

```cairo
// src/pool/pool.cairo lines 960-978
/// **Note**: Delegation rewards lost when pool balance is less than
/// `min_delegation_for_rewards`. The staking contract continues to forward
/// `pool_rewards` to the pool contract even in this case.
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    mul_wide_and_div(
        lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
    )
        .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
}
```

The inline comment explicitly acknowledges the critical asymmetry: **the staking contract physically transfers STRK tokens to the pool contract even when `pool_balance < min_delegation_for_rewards`**, but the pool's accounting records zero rewards per unit. The cumulative rewards trace does not increase, so no delegator can ever claim those tokens. There is no sweep, recovery, or admin function to retrieve them.

The entry into the pool has no minimum amount guard beyond non-zero:

```cairo
// src/pool/pool.cairo line 191
assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
```

This means a delegator can legitimately enter with any amount from 1 friston upward, and if the total pool balance is below 1 STRK, every epoch's reward distribution is silently discarded at the accounting layer while the tokens accumulate in the pool contract.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

STRK reward tokens forwarded by the staking contract to the pool are irrecoverably locked. No delegator can claim them (the cumulative rewards trace shows zero growth), no staker can retrieve them, and no admin function exists to sweep them. The tokens sit in the pool contract balance permanently. This matches the allowed impact: *Permanent freezing of unclaimed yield or unclaimed royalties*.

### Likelihood Explanation
**Medium-High.** The condition is reachable by any unprivileged actor:

1. A delegator can enter a pool with any amount from 1 friston to `10^18 - 1` friston (less than 1 STRK), immediately triggering the condition for every subsequent epoch.
2. A pool that previously had sufficient balance can fall below the threshold as delegators call `exit_delegation_pool_intent` and `exit_delegation_pool_action`, reducing the total balance below 1 STRK. All future rewards are then permanently lost.
3. No special privilege is required. The entry path is `enter_delegation_pool` (public) or natural balance reduction via `exit_delegation_pool_intent` (callable by any pool member).

### Recommendation
Two complementary fixes:

1. **Enforce a minimum delegation amount on entry**: In `enter_delegation_pool` and `add_to_delegation_pool`, assert that the resulting total pool balance meets `min_delegation_for_rewards`. This prevents the pool from entering the loss-inducing state.

2. **Do not transfer reward tokens when accounting is zero**: In the staking contract's reward distribution logic, skip the physical STRK transfer to the pool when `pool_balance < min_delegation_for_rewards`, so no tokens are sent that cannot be accounted for. Alternatively, accumulate them in the staking contract and forward them once the pool balance recovers above the threshold.

### Proof of Concept

1. Staker registers and opens a STRK delegation pool.
2. Delegator calls `enter_delegation_pool` with `amount = 1` (1 friston, far below `min_delegation_for_rewards = 10^18`).
3. Staker attests for K epochs; the staking contract computes pool rewards and calls `update_rewards_from_staking_contract(rewards: R, pool_balance: 1)` on the pool.
4. Inside `compute_rewards_per_unit`: `1 < 10^18` → returns `0`. The cumulative rewards trace is updated with `last + 0 = last` (no change).
5. The staking contract has already transferred `R` STRK tokens to the pool contract's balance.
6. Delegator calls `claim_rewards` → receives `0` STRK.
7. `R` STRK tokens are now permanently locked in the pool contract with no recovery path.
