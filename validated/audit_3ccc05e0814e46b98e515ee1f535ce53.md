### Title
Unbounded Growth of `stakers` Vector Enables Griefing of `get_stakers()` via Stake-Unstake Cycling - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage vector in the Staking contract is append-only: every new staker address is pushed to it on `stake()` and **never removed**, even after a full `unstake_intent` + `unstake_action` cycle. Because `get_stakers()` iterates the entire vector on every call, an attacker can inflate the vector to an unbounded size by cycling a single `min_stake` deposit through many fresh addresses, causing `get_stakers()` to consume unbounded gas and eventually become uncallable.

### Finding Description

`stake()` appends every new staker address to `self.stakers` unconditionally: [1](#0-0) 

The storage field comment explicitly acknowledges the invariant: [2](#0-1) 

`get_stakers()` iterates the **full range** of this vector on every invocation: [3](#0-2) 

The only guard in `stake()` is a minimum-amount check: [4](#0-3) 

There is no cap on the total number of staker registrations, and there is no mechanism to compact or prune the vector.

Because `assert_staker_address_not_reused` prevents re-staking from the same address, each cycle requires a fresh address — but the attacker's capital is fully recovered after each `unstake_action`. With only `min_stake` tokens the attacker can, over successive exit windows, register an unbounded number of addresses, each leaving a permanent entry in `stakers`.

### Impact Explanation

`get_stakers()` is the function the consensus layer calls to obtain the validator set for a given epoch: [5](#0-4) 

As the vector grows, every call to `get_stakers()` performs proportionally more storage reads and computation. Once the vector is large enough, the function will exceed Starknet's per-transaction resource limits, making the validator-set query permanently uncallable. This breaks the consensus layer's ability to determine which stakers should attest, disrupting protocol operation.

**Impact class**: Medium — Griefing with no profit motive but damage to the protocol; unbounded gas consumption.

### Likelihood Explanation

The attack requires only `min_stake` tokens (recoverable after each exit window) and an arbitrary number of fresh addresses (trivially generated). The exit wait window adds a time cost per cycle but does not prevent the attack; it merely slows it. The attack is fully permissionless and requires no privileged access.

### Recommendation

1. **Compact the vector on unstake**: When `unstake_action` is called, swap-and-pop the staker's entry from `stakers` so the vector length reflects only currently-registered stakers.
2. **Alternatively, cap registrations**: Enforce a protocol-level maximum on the total number of staker registrations.
3. **Lazy iteration**: Replace the full-range scan in `get_stakers()` with a paginated or indexed approach so a single call cannot be forced to iterate an unbounded set.

### Proof of Concept

1. Attacker holds `min_stake` STRK tokens.
2. Attacker calls `stake(reward, op_A, min_stake)` from address `A` → `stakers` vector length = N+1.
3. Attacker calls `unstake_intent()` from `A`, waits `exit_wait_window`, calls `unstake_action(A)` → funds returned; `stakers` vector still length N+1 (address `A` remains).
4. Attacker repeats steps 2–3 with fresh addresses `B`, `C`, `D`, … each cycle adding one permanent entry.
5. After K cycles the attacker has spent only gas fees; `stakers` has grown by K entries.
6. Any call to `get_stakers(epoch_id)` now iterates K extra entries. At sufficient K the call exceeds resource limits and reverts, preventing the consensus layer from reading the validator set.
