### Title
`skip_stateful_validations` Admits Unsigned Invoke (nonce=1) from Undeployed Account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the `__validate__` entry-point (signature verification) for any invoke transaction with `nonce=1` sent from an address whose `account_nonce` is `0`, provided that address has **any** transaction in the mempool. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can immediately submit an invoke with `nonce=1` from that address carrying an empty or forged signature. The gateway accepts it without running `__validate__`, inserting an invalid transaction into the mempool and blocking the victim's legitimate first invoke.

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must either (a) have passed the account's `__validate__` entry point, or (b) be a `deploy_account` whose address does not yet exist in state. The `skip_stateful_validations` path violates (a) for invoke transactions without verifying that the caller is the same party who submitted the `deploy_account`.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is an `Invoke` with `tx.nonce() == 1`
2. `account_nonce == 0` (account not yet deployed in committed state)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

Condition 3 is satisfied by **any** transaction from that address in the pool — not specifically a `deploy_account`. The check does not verify the identity of the submitter.

**How the skip propagates to zero signature verification:** [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, … }` and calls `StatefulValidator::validate`. Inside the blockifier: [3](#0-2) 

`validate_tx` returns `Ok(None)` immediately — the `__validate__` Cairo entry point is never executed, so the signature bytes are never inspected.

**Attack path (analog to H-02 threshold race):**

| H-02 (multi-sig) | Sequencer analog |
|---|---|
| Controller A signs "decrease threshold to 50" | Victim submits `deploy_account` for address X (nonce=0) |
| Attacker B intercepts A's signature and pairs it with "remove A" instead | Attacker observes X in mempool; submits `invoke(nonce=1, calldata=arbitrary, signature=[])` from X |
| Threshold weakening (step 1) enables unauthorized removal (step 2) | `account_tx_in_pool_or_recent_block` returning `true` (step 1) enables unsigned invoke admission (step 2) |

The `account_tx_in_pool_or_recent_block` check is the "threshold" — once it is satisfied by the victim's `deploy_account`, the security invariant (signature check) is weakened for any nonce=1 invoke from that address.

**Concrete gateway flow for the attacker's transaction:**

1. **Nonce check** (`validate_nonce`): `account_nonce=0`, `tx_nonce=1`, `max_allowed_nonce_gap=200` → `0 ≤ 1 ≤ 200` ✓ [4](#0-3) 

2. **Mempool validation** (`validate_by_mempool` → `mempool.validate_tx`): checks only duplicate hash and nonce ordering — no signature check ✓ [5](#0-4) 

3. **`skip_stateful_validations`**: nonce=1, account_nonce=0, victim's `deploy_account` is in pool → returns `true` ✓ [6](#0-5) 

4. **`run_validate_entry_point`** with `validate=false` → `__validate__` never called, signature never checked ✓

The attacker's unsigned invoke is stored in the mempool at `(address=X, nonce=1)`.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway admits a transaction with an invalid (empty) signature. Consequences:

- **Nonce-slot squatting / DoS on first post-deploy invoke.** The attacker's invalid invoke occupies `(X, nonce=1)`. The victim's legitimate invoke is rejected as `MempoolError::DuplicateNonce` or must pay a fee-escalation premium to replace it. [7](#0-6) 

- **Persistent griefing.** Because `__validate__` failure at execution time does not increment the nonce, the attacker can repeatedly front-run the victim's nonce=1 invoke, forcing the victim to continuously escalate fees.

- **Incorrect admission decision.** The gateway's contract is to admit only transactions that pass account validation. Admitting an unsigned invoke violates this contract and produces an authoritative-looking "accepted" response to the attacker.

---

### Likelihood Explanation

**Medium.** The attacker requires only:
1. Mempool visibility (public RPC endpoint).
2. The ability to compute the deterministic deploy-account address from the observed `deploy_account` transaction parameters.
3. Submission of a single unsigned invoke before the victim's invoke arrives.

No privileged access, no special contract, no prior on-chain state is required. The window is the time between the victim's `deploy_account` entering the mempool and their `invoke(nonce=1)` arriving — a realistic race for any user following the documented deploy+invoke UX pattern. [8](#0-7) 

---

### Recommendation

1. **Verify the mempool transaction is a `deploy_account`.** `account_tx_in_pool_or_recent_block` returns `true` for any transaction type. The skip should only fire when the account's pending transaction is specifically a `deploy_account`.

2. **Tie the skip to the deploy-account transaction hash.** The gateway already receives the `deploy_account` hash in the `py_validator` path (`deploy_account_tx_hash` parameter). The same binding should be enforced in the Rust gateway path so the skip is scoped to the exact deploy being paired. [9](#0-8) 

3. **Alternatively, perform a lightweight signature-format check** (non-empty, correct length) even when `__validate__` is skipped, to reject obviously forged transactions at the gateway boundary.

---

### Proof of Concept

```

# Step 1 — victim submits deploy_account (signed)
POST /gateway/add_transaction

→ accepted, deploy_account enters mempool

# Step 2 — attacker observes mempool, extracts address X, submits unsigned invoke
POST /gateway/add_transaction
{ type: INVOKE, sender_address: X, nonce: 1, calldata: [0xdead], signature: [] }

# Gateway evaluation:
#   validate_nonce:  0 ≤ 1 ≤ 200  ✓
#   validate_by_mempool: no dup nonce  ✓
#   skip_stateful_validations:
#     nonce==1 ∧ account_nonce==0 ∧ account_tx_in_pool_or_recent_block(X)==true
#     → skip_validate = true
#   run_validate_entry_point(validate=false) → __validate__ NOT called
→ attacker's unsigned invoke ACCEPTED into mempool at (X, nonce=1)

# Step 3 — victim submits their legitimate invoke
POST /gateway/add_transaction
{ type: INVOKE, sender_address: X, nonce: 1, calldata: [intended], signature: [valid] }
→ REJECTED: MempoolError::DuplicateNonce  (or must pay fee-escalation premium)

# Step 4 — batcher executes block
deploy_account(nonce=0) → X deployed
invoke(nonce=1, attacker's, sig=[]) → __validate__ called → FAILS (invalid sig) → reverts

# Victim's intended action never executes; attacker can repeat indefinitely.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
```
