### Title
XCC Router `execute_scheduled` Accepts NEAR Deposits With No Withdrawal Path, Causing Permanent Fund Freeze - (`File: etc/xcc-router/src/lib.rs`)

### Summary
The XCC Router contract's `execute_scheduled` function is marked `#[payable]`, allowing any caller to attach NEAR tokens. However, the router has no mechanism to withdraw arbitrary accumulated NEAR. Any NEAR attached to `execute_scheduled` is permanently frozen in the router sub-account.

### Finding Description
The `Router` contract in `etc/xcc-router/src/lib.rs` exposes `execute_scheduled` as a `#[payable]` function that is intentionally callable by anyone:

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [1](#0-0) 

The function accepts attached NEAR (via `#[payable]`) but does nothing with it — the deposit is neither forwarded to the created promise nor returned to the caller. It silently accumulates in the router's account balance.

The only NEAR-recovery mechanism in the contract is `send_refund`, which:
1. Can only be called by the parent (Aurora engine), not by the user whose funds are stuck.
2. Transfers a hardcoded constant `REFUND_AMOUNT` (2 NEAR), not the actual accumulated balance. [2](#0-1) 

There is no function in the `Router` contract that allows withdrawing an arbitrary NEAR balance. The full public interface is: `initialize`, `get_version`, `execute`, `schedule`, `execute_scheduled`, `deploy_upgrade`, and `send_refund` — none of which provide a general NEAR withdrawal path. [3](#0-2) 

### Impact Explanation
**Impact: High — Permanent freezing of funds.**

Any NEAR attached by a caller to `execute_scheduled` is permanently frozen in the router sub-account. Because each EVM address gets its own router sub-account (e.g., `<hex_address>.aurora`), and because `execute_scheduled` is open to any caller, any user who attaches NEAR to this call loses those funds with no recovery path. The funds cannot be retrieved by the user, the router owner, or the Aurora engine (since `send_refund` only sends a fixed 2 NEAR, not the arbitrary stuck amount).

### Likelihood Explanation
**Likelihood: Medium.**

`execute_scheduled` is intentionally public and `#[payable]`. A user executing a scheduled XCC promise may reasonably attach NEAR to fund the promise's execution (e.g., to cover attached balances in the scheduled promise). Since the function signature gives no indication that attached NEAR is ignored, this is a realistic mistake. Additionally, NEAR Protocol callbacks from failed cross-contract calls can refund NEAR to the calling contract — if such a refund lands on the router account, it is similarly unrecoverable.

### Recommendation
Either:
1. Remove `#[payable]` from `execute_scheduled` (and `deploy_upgrade` if applicable) to reject any attached NEAR at the protocol level, or
2. Add a privileged withdrawal function (callable only by the parent Aurora engine) that transfers the full NEAR balance minus storage staking requirements back to the parent or to the associated EVM address owner.

### Proof of Concept
1. A user has a scheduled XCC promise stored in their router sub-account (created via `Router::schedule`).
2. The user (or any third party) calls `execute_scheduled` on the router, attaching e.g. 1 NEAR as a deposit, expecting it to fund the promise execution.
3. The function removes the promise from storage and creates it via `Self::promise_create`, but the 1 NEAR attached deposit is never forwarded — it stays in the router's account balance.
4. The user has no way to recover the 1 NEAR: `send_refund` is parent-only and sends only the hardcoded `REFUND_AMOUNT` (2 NEAR); no other withdrawal function exists.
5. The 1 NEAR is permanently frozen in the router sub-account. [4](#0-3) [5](#0-4)

#[near]
impl Router {
    #[init(ignore_state)]
    #[must_use]
    pub fn initialize(wnear_account: AccountId, must_register: bool) -> Self {
        // The first time this function is called there is no state and the parent is set to be
        // the predecessor account id. In subsequent calls, only the original parent is allowed to
        // call this function. The idea is that the Create, Deploy and Initialize actions are done in a single
        // NEAR batch when a new router is deployed by the engine, so the caller will be the Aurora
        // engine instance that the user's address belongs to. If we update this contract and deploy
        // a new version of it, again the Deploy and Initialize actions will be done in a single batch
        // by the engine.
        let caller = env::predecessor_account_id();
        let mut parent = LazyOption::new(StorageKey::Parent, None);
        match parent.get() {
            None => {
                parent.set(&caller);
            }
            Some(parent) => {
                // Allow self-calls to `initialize` also.
                // This happens during the upgrade flow.
                if (caller != parent) && (caller != env::current_account_id()) {
                    env::panic_str(ERR_ILLEGAL_CALLER);
                }
            }
        }

        if must_register {
            env::promise_create(
                wnear_account.clone(),
                "storage_deposit",
                b"{}",
                WNEAR_REGISTER_AMOUNT,
                WNEAR_REGISTER_GAS,
            );
        }

        let mut version = LazyOption::new(StorageKey::Version, None);
        if version.get().unwrap_or_default() != CURRENT_VERSION {
            // Future migrations would go here

            version.set(&CURRENT_VERSION);
        }

        let nonce = LazyOption::new(StorageKey::Nonce, None);
        let scheduled_promises = LookupMap::new(StorageKey::Map);
        Self {
            parent,
            version,
            nonce,
            scheduled_promises,
            wnear_account,
        }
    }

    pub fn get_version(&self) -> u32 {
        self.version.get().unwrap_or_default()
    }

    /// This function can only be called by the parent account (i.e. Aurora engine) to ensure that
    /// no one can create calls on behalf of the user this router contract is deployed for.
    /// The engine only calls this function when the special precompile in the EVM for NEAR cross
    /// contract calls is used by the address associated with the sub-account this router contract
    /// is deployed at.
    pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Allows the parent contract to trigger an update to the logic of this contract
    /// (by deploying a new contract to this account);
    #[payable]
    pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
        self.assert_preconditions();

        let promise_id = env::promise_batch_create(&env::current_account_id());
        env::promise_batch_action_deploy_contract(promise_id, &args.code);
        env::promise_batch_action_function_call(
            promise_id,
            INITIALIZE,
            &args.initialize_args,
            NearToken::default(),
            INITIALIZE_GAS,
        );
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
}
