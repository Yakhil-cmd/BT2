### Title
Permissionless `unwrapWETH9` with arbitrary `recipient` drains any WETH stranded on the router — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.unwrapWETH9` is `public payable` with no access control and accepts a caller-supplied `recipient`. Any WETH balance held by the router — including WETH deposited by a legitimate user's swap with `recipient: address(router)` — can be drained to an arbitrary attacker address in a subsequent transaction.

### Finding Description

`unwrapWETH9` reads the router's full WETH balance and transfers it as ETH to the caller-supplied `recipient` with no check on `msg.sender`:

```solidity
function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override {
    uint256 balanceWETH = IERC20(WETH).balanceOf(address(this));
    if (balanceWETH < amountMinimum) revert InsufficientWETH(amountMinimum, balanceWETH);
    if (balanceWETH > 0) {
        IWETH9(WETH).withdraw(balanceWETH);
        _transferETH(recipient, balanceWETH);
    }
}
``` [1](#0-0) 

The documented ETH-output pattern requires the user to set `recipient: address(router)` in the swap and then call `unwrapWETH9` **in the same multicall**: [2](#0-1) 

If a user issues the swap in one transaction and the unwrap in a second (a realistic mistake, especially for integrators or scripts that split calls), the WETH sits on the router between blocks. Any attacker observing the mempool or the chain state can immediately call `unwrapWETH9(0, attacker)` and receive all of it.

The same issue applies to `sweepToken`, which is also `public payable` with an arbitrary `recipient` and no access control: [3](#0-2) 

### Impact Explanation

Direct loss of user principal. Any WETH output from a swap that was routed through the router (`recipient: address(router)`) and not unwrapped atomically in the same multicall is permanently drainable by an unprivileged attacker. The attacker receives the full ETH value; the original swapper receives nothing.

### Likelihood Explanation

- The pattern is explicitly documented and tested: users are expected to compose `exactInput`/`exactInputSingle` with `recipient: address(router)` followed by `unwrapWETH9` in a multicall.
- Integrators, bots, or users who split these into two transactions (e.g., due to gas estimation, scripting errors, or partial multicall construction) will strand WETH.
- The attack requires no privileges, no special setup, and is trivially front-runnable from the mempool.

### Recommendation

Restrict `unwrapWETH9` and `sweepToken` so that the `recipient` can only be `msg.sender`, or add a `msg.sender == recipient` check. This matches the behavior of `refundETH`, which correctly sends only to `msg.sender`: [4](#0-3) 

Alternatively, enforce that these functions are only callable within a `multicall` context (e.g., via a reentrancy-style lock that is set during `multicall` execution).

### Proof of Concept

```solidity
// 1. Victim swaps token1 → WETH, sends output to router (intending to unwrap later)
router.exactInputSingle(ExactInputSingleParams({
    ...,
    tokenOut: address(weth),
    recipient: address(router),   // WETH lands on router
    ...
}));

// 2. In a separate transaction, attacker drains it
router.unwrapWETH9(0, attacker);  // no access control; attacker receives all ETH
assert(attacker.balance > 0);
assert(weth.balanceOf(address(router)) == 0);
```

The test `test_multicall_tokenForWeth_thenUnwrapEth` already demonstrates the exact stranding state (swap with `recipient: address(router)`) — the only difference is the attacker calls `unwrapWETH9` in a separate transaction with their own address as recipient: [5](#0-4)
