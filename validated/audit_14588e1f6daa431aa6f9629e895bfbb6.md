### Title
Malicious Participant Can Corrupt CKD Output Without Verification of Contributed Shares — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary
The CKD coordinator aggregates `(norm_big_y, norm_big_c)` contributions from all participants without any cryptographic verification that each contribution is correctly formed relative to the contributor's actual private share. A single malicious participant can inject arbitrary group elements, causing every honest party to accept a corrupted `CKDOutput` that encodes the wrong confidential key.

---

### Finding Description

In `do_ckd_coordinator` (lines 35–58 of `src/confidential_key_derivation/protocol.rs`), the coordinator collects each participant's `CKDOutput` and blindly accumulates it:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant is supposed to send:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is the participant's private signing share and `y_i` is a fresh random scalar (see `compute_signature_share`, lines 148–182). The coordinator then sums these to obtain an ElGamal ciphertext of `msk · H(pk ‖ app_id)`.

There is **no proof of correct formation** attached to the contribution. Nothing prevents a malicious participant from sending:

```
norm_big_y_malicious  = arbitrary G1 element
norm_big_c_malicious  = arbitrary G1 element
```

The coordinator has no mechanism to detect the deviation and will incorporate the poisoned values directly into the final `CKDOutput`.

Contrast this with the DKG protocol (`src/dkg.rs`), which enforces:
- A Schnorr proof-of-knowledge for every polynomial commitment (lines 452–460)
- A commitment-hash binding round (lines 462–469)
- Per-share Feldman verification (lines 520–522)

The CKD protocol provides none of these safeguards.

---

### Impact Explanation

The aggregated output is:

```
big_y = Σ norm_big_y_i   (sum over all participants, including the malicious one)
big_c = Σ norm_big_c_i
```

When the application unmasks the result with `app_sk`:

```
big_c − app_sk · big_y  ≠  msk · H(pk ‖ app_id)
```

The derived confidential key is silently wrong. Every honest party — including the coordinator and the downstream TEE application — accepts this corrupted output as valid, because there is no final consistency check. The malicious participant can additionally choose their injected values so that the resulting key is one they control, enabling them to impersonate the TEE application's derived secret.

**Matched allowed impact:** *High — Corruption of CKD outputs so honest parties accept unusable or attacker-controlled cryptographic outputs.*

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. No threshold of colluders is required; one malicious party suffices. The attack requires only the ability to participate in the protocol and send a crafted message to the coordinator — a capability every participant already possesses by design. There is no out-of-band detection mechanism.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` contribution with a non-interactive zero-knowledge proof of correct formation — specifically, a proof that:

1. `norm_big_y = λ_i · y_i · G` for some scalar `y_i`, and
2. `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` for the same `y_i` and the participant's committed private share `x_i`.

A standard Chaum–Pedersen / Schnorr sigma protocol over the BLS12-381 G1 group suffices. The coordinator must verify all proofs before accumulating any contribution, mirroring the verification discipline already present in `do_keyshare` (`src/dkg.rs`, lines 452–476).

---

### Proof of Concept

```
Setup: 3 participants P1 (honest), P2 (honest), P3 (malicious); coordinator = P1.

1. P1 and P2 compute correct (norm_big_y_i, norm_big_c_i) and send to P1.
2. P3 sends (0_G1, 0_G1) — the identity element — to P1.

3. Coordinator accumulates:
     big_y = λ1·y1·G + λ2·y2·G + 0  =  (λ1·y1 + λ2·y2)·G
     big_c = λ1·C1 + λ2·C2 + 0

4. Unmask:
     big_c − app_sk·big_y
       = λ1·x1·H + λ2·x2·H          (missing λ3·x3·H term)
       ≠ msk·H

5. The coordinator returns this corrupted CKDOutput as Some(ckd_output)
   with no error, and the TEE application derives the wrong key silently.

Variant: P3 sends (norm_big_y_fake, norm_big_c_fake) computed with a
chosen x3' ≠ x3, making the output encode msk + (x3'−x3)·λ3, a value
P3 knows and can use to impersonate the derived secret.

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in

        norm_big_c += participant_output.big_c();

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
