### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Delegator's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in `pool.cairo` contains an explicitly acknowledged unbounded `while` loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. A delegator who makes many balance changes across distinct epochs without claiming can grow this trace without bound, eventually causing `claim_rewards` to revert with an out-of-gas error, permanently freezing their unclaimed yield.

### Finding Description
In `src/pool/pool.cairo`, the `calculate_rewards` function iterates over the pool member's balance trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each call to `set_member_balance` inserts a new checkpoint into `pool_member_epoch_balance` keyed by `get_epoch_plus_k()`:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

The `insert` function in the trace only deduplicates if the key (epoch) is identical to the last entry; otherwise it appends a new checkpoint:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

This means every call to `add_to_delegation_pool`, `exit_delegation_pool_intent`, or `switch_delegation_pool` in a **different epoch** appends a new entry to the trace. The trace grows monotonically and is never pruned. [4](#0-3) 

### Impact Explanation
If a delegator accumulates a sufficiently large number of balance-change entries between two consecutive `claim_rewards` calls, the `while` loop in `calculate_rewards` will consume more gas than the Starknet transaction limit allows, causing the transaction to revert. Because the trace is append-only and `entry_to_claim_from` is only advanced on a successful claim, the delegator is permanently unable to claim their accrued yield. This matches **High: Permanent freezing of unclaimed yield**. [5](#0-4) 

### Likelihood Explanation
A delegator who actively manages their delegation position — increasing or partially exiting across many epochs without claiming — will naturally grow their trace. The code itself acknowledges the risk with the comment "This loop is unbounded but unlikely to exceed gas limits," confirming the developers are aware of the pattern but have not bounded it. No privileged access is required; any delegator can trigger this through normal protocol interactions. [6](#0-5) 

### Recommendation
Cap the number of trace entries processed per `claim_rewards` call (partial claiming), or enforce a maximum number of balance-change entries per pool member between claims. Alternatively, compress consecutive same-epoch entries at write time and document a hard upper bound on how many distinct-epoch changes are safe between claims.

### Proof of Concept
1. Delegator calls `add_to_delegation_pool` in epoch `E`.
2. Delegator calls `exit_delegation_pool_intent` (partial) in epoch `E+1`, then `add_to_delegation_pool` again in epoch `E+2`, and so on — one balance change per epoch — for `N` epochs without ever calling `claim_rewards`.
3. Each call in a new epoch appends one entry to `pool_member_epoch_balance` (key = current epoch + K).
4. After `N` epochs, `pool_member_trace_length` = N.
5. Delegator calls `claim_rewards`. The `calculate_rewards` loop iterates from `entry_to_claim_from` (0 or last saved index) up to N, executing a `find_sigma` storage read per iteration.
6. For sufficiently large N (on the order of thousands of entries, each requiring multiple storage reads), the transaction exceeds the Starknet gas limit and reverts.
7. Every subsequent `claim_rewards` call also reverts — the delegator's yield is permanently frozen. [7](#0-6) [8](#0-7)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
