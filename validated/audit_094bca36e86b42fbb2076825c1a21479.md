### Title
Incorrect NFT Net Proceeds Calculation Causes Wrong Amount Displayed in Offer Acceptance Flow - (File: packages/gui/src/components/offers/utils.ts)

### Summary
`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` sets `nftSellerNetAmount` to the gross offer amount instead of the amount minus royalties. This incorrect derived value is propagated into the offer acceptance UI, causing the "You will receive" / "Net Proceeds" figure shown to a taker to be inflated by the full royalty amount. An unprivileged attacker who creates an NFT offer with royalties can cause a victim to see a higher net payout than they will actually receive, inducing acceptance of an offer they might otherwise reject.

### Finding Description

`calculateNFTRoyalties` computes `royaltyAmount` correctly, but then assigns `nftSellerNetAmount = amount` (the gross price) instead of `amount - royaltyAmount`. The correct formula is commented out immediately below the assignment:

```typescript
// packages/gui/src/components/offers/utils.ts
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;          // BUG: gross, not net
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

`NFTOfferDetails` (the offer acceptance screen) calls `calculateNFTRoyalties` and then converts the incorrect `nftSellerNetAmount` into `overrideNFTSellerAmount` in mojos:

```typescript
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // gross, not net
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

This value is passed as `overrideNFTSellerAmount` to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it **replaces** the original amount in the "You will receive" display:

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [3](#0-2) 

The same incorrect `nftSellerNetAmount` is also rendered as the "Net Proceeds" label in the details panel: [4](#0-3) 

### Impact Explanation

When a victim opens an imported `TokenForNFT` offer (i.e., they are the NFT seller/taker), the "You will receive" line and the "Net Proceeds" line both display the **gross** token amount. The royalty deduction is shown separately as "Creator Fee", but the net figure that summarises what the seller actually pockets is wrong — it equals the gross price, not `gross - royalty`. A seller with a 10 % royalty NFT priced at 10 XCH would see "Net Proceeds: 10 XCH" when the correct figure is 9 XCH. This constitutes displaying the wrong amount in the offer-acceptance confirmation screen, which can cause the user to approve an offer they would otherwise reject.

The actual `takeOffer` RPC call uses the raw `offerData` blob and is unaffected, so the on-chain transaction executes at the correct amounts — but the user consented based on a materially incorrect display. [5](#0-4) 

### Likelihood Explanation

Any unprivileged actor can create a `TokenForNFT` NFT offer for an NFT that carries a royalty percentage and share the offer file with a victim. The victim opens the offer in the GUI, sees the inflated "You will receive" figure, and accepts. No special access, leaked keys, or social engineering beyond sharing an offer file is required. The bug fires on every `TokenForNFT` offer where the NFT has a non-zero `royaltyPercentage`.

### Recommendation

Restore the commented-out net-amount formula in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [6](#0-5) 

### Proof of Concept

1. Create an NFT with `royaltyPercentage = 1000` (10 %).
2. Create a `TokenForNFT` offer: offer 10 XCH, request the NFT.
3. Share the `.offer` file with a second wallet.
4. In the second wallet, open the offer via **Offers → Import Offer**.
5. Observe the "You will receive" / "Net Proceeds" line shows **10 XCH** instead of the correct **9 XCH**.
6. The `calculateNFTRoyalties` call returns `nftSellerNetAmount = 10` (gross) because line 314 assigns `amount` directly; the correct value `10 - 1 = 9` is in the commented-out block immediately below. [7](#0-6)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
