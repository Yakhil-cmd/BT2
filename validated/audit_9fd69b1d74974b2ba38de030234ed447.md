### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Can Permanently Freeze Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the `DelegationPool` contract contains an unbounded loop that iterates over every entry in a pool member's balance trace since their last claim. A delegator who repeatedly calls `add_to_delegation_pool` or `exit_delegation_pool_intent` across many epochs without claiming rewards will accumulate an ever-growing trace. Eventually, a `claim_rewards` call must process all accumulated entries in a single transaction, which can exceed Starknet's gas/step limits and permanently freeze the delegator's unclaimed yield.

The developers themselves acknowledge this in a code comment at the loop site.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function is called by both `claim_rewards` (line 349) and `pool_member_info_v1` (line 532). It iterates over the pool member's `pool_member_epoch_balance` trace from `entry_to_claim_from` to `pool_member_trace_length`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows every time `set_member_balance` is called, which happens on every invocation of `add_to_delegation_pool`, `exit_delegation_pool_intent`, `enter_delegation_pool`, or `enter_delegation_pool_from_staking_contract`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(
        balance: amount,
        cumulative_rewards_trace_idx: self.cumulative_rewards_trace_length() + 1,
    );
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `entry_to_claim_from` cursor is persisted in `pool_member_info` and advanced after each successful `claim_rewards` call:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [3](#0-2) 

This means the loop only processes entries since the **last successful claim**. If a delegator makes balance changes across many epochs without claiming, the number of unprocessed trace entries grows linearly with the number of balance-change epochs. When `claim_rewards` is finally called, it must process all of them in one transaction.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A delegator who accumulates a sufficiently large number of balance-change epochs without claiming rewards will find their `claim_rewards` transaction permanently reverting due to step/gas exhaustion. Because `entry_to_claim_from` is only advanced on a *successful* `claim_rewards` call, and because the trace is append-only and never pruned, there is no recovery path: every future `claim_rewards` attempt will also fail. The delegator's accrued yield is permanently frozen in the pool contract.

The same loop is also executed in the read-only `pool_member_info_v1` view function, meaning even querying the member's state can fail. [4](#0-3) 

---

### Likelihood Explanation

**Medium-High.** Any delegator who:
1. Regularly calls `add_to_delegation_pool` (e.g., to compound rewards or top up stake) across many epochs, **and**
2. Does not call `claim_rewards` frequently enough to keep `entry_to_claim_from` close to the trace head

will naturally accumulate this condition over time. No adversarial intent is required — this is a normal usage pattern. The protocol has no minimum claim frequency requirement and no mechanism to prune the trace. Over months or years of operation, early delegators with active balance management are at risk.

---

### Recommendation

1. **Enforce periodic claiming**: Require or incentivize `claim_rewards` to be called before each `add_to_delegation_pool` or `exit_delegation_pool_intent`, resetting `entry_to_claim_from` and keeping the unprocessed window small.
2. **Batch/paginate reward calculation**: Allow `claim_rewards` to accept a `max_entries` parameter, processing only a bounded number of trace entries per call and persisting the updated `entry_to_claim_from` even for partial claims.
3. **Prune processed trace entries**: After advancing `entry_to_claim_from`, mark or remove processed entries so the effective trace length does not grow without bound.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` at epoch 0.
2. Delegator calls `add_to_delegation_pool` once per epoch for N epochs (e.g., N = 10,000), never calling `claim_rewards`. Each call inserts a new entry into `pool_member_epoch_balance` via `set_member_balance`.
3. After N epochs, `pool_member_trace_length` = N + 1.
4. Delegator calls `claim_rewards`. The loop at line 859 must iterate over all N entries, calling `find_sigma` (which reads from `cumulative_rewards_trace`) on each iteration.
5. For sufficiently large N, the transaction exceeds Starknet's step limit and reverts.
6. All future `claim_rewards` calls also revert. The delegator's accumulated yield is permanently frozen. [5](#0-4) [6](#0-5)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
