### Title
Missing `collectFees` call before destination update causes accrued admin fees to be redirected to new address — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-mutating path (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` with the **current** destination before making any change. The missing flush means all fees that accrued under the old destination are silently redirected to the new one on the next `collectPoolFees` call.

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the correct pattern: call `collectFees(…, poolAdminFeeDestination[pool])` with the **old** destination, then update state. [1](#0-0) 

`setPoolAdminFeeDestination` skips this step entirely: [2](#0-1) 

After the write to `poolAdminFeeDestination[pool]`, the next call to `collectPoolFees` (permissionless) or any fee-changing function passes the **new** destination to `collectFees`: [3](#0-2) 

Inside `collectFees`, the admin share of both the spread surplus and the notional accumulator is transferred to whichever `adminFeeDestination_` is passed in: [4](#0-3) 

Because the flush never happened, the entire backlog of admin fees — spread surplus accumulated since the last collection and the full `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` accumulator — is paid to the new destination. [5](#0-4) 

### Impact Explanation

The old admin fee destination (which may be a DAO treasury, a multisig, or any address distinct from the pool admin key) permanently loses all fees that accrued before the destination change. Those tokens are transferred to the new destination instead. This is a direct, irreversible loss of ERC-20 principal owed to the old recipient. The magnitude equals the full spread surplus plus the notional accumulator at the moment of the next collection — potentially large for high-volume pools with infrequent collection schedules.

### Likelihood Explanation

`setPoolAdminFeeDestination` is a routine operational action (treasury rotation, multisig migration, handover to a new entity). The pool admin is semi-trusted and is expected to call this function in normal operations. No malicious intent is required: even an honest admin performing a routine treasury rotation triggers the loss. The permissionless `collectPoolFees` can be called by anyone immediately after the destination change, making the loss instant and unrecoverable.

### Recommendation

Add a `collectFees` call with the **old** destination before overwriting `poolAdminFeeDestination[pool]`, mirroring the pattern already used in `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the OLD destination before rotating
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool accumulates spread surplus and notional fees over time (e.g., 1 000 token0 owed to `oldDest`).
2. Pool admin calls `setPoolAdminFeeDestination(pool, newDest)`. No flush occurs; `poolAdminFeeDestination[pool]` is now `newDest`.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `newDest` and passes it to `collectFees`.
4. `collectFees` transfers the full 1 000 token0 admin share to `newDest`.
5. `oldDest` receives 0. Loss is permanent. [6](#0-5) [2](#0-1) [3](#0-2) [7](#0-6)
