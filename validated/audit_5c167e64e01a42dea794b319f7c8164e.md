### Title
Unbounded Recursion in `recursive_promise_create` Allows Permanent Freezing of NEAR Funds in XCC Router - (`File: etc/xcc-router/src/lib.rs`)

---

### Summary

The `recursive_promise_create` function in the XCC router contract processes a user-controlled `NearPromise` tree with no depth limit or element-count cap. A user can craft a `NearPromise::Then` chain of arbitrary nesting depth or a `NearPromise::And` vector of arbitrary length, submit it as a `CrossContractCallArgs::Delayed` XCC call, and cause every subsequent `execute_scheduled` invocation to fail. Because the promise is removed from storage before the failure and NEAR reverts state on panic, the promise remains in storage but is permanently unexecutable, freezing the NEAR tokens deposited to the router account.

---

### Finding Description

`Router::recursive_promise_create` in `etc/xcc-router/src/lib.rs` is a recursive function that walks a `NearPromise` tree:

```rust
fn recursive_promise_create(promise: &NearPromise) -> PromiseIndex {
    match promise {
        NearPromise::Simple(x) => { ... }
        NearPromise::Then { base, callback } => {
            let base_index = Self::recursive_promise_create(base); // unbounded recursion
            ...
        }
        NearPromise::And(promises) => {
            let indices: Vec<PromiseIndex> = promises
                .iter()
                .map(Self::recursive_promise_create) // unbounded iteration
                .collect();
            env::promise_and(&indices)
        }
    }
}
``` [1](#0-0) 

There is no maximum depth check for `NearPromise::Then` chains and no maximum element count check for `NearPromise::And` vectors. The `NearPromise` type itself is a recursive tree with no enforced size bound: [2](#0-1) 

The XCC precompile accepts `CrossContractCallArgs::Delayed(PromiseArgs::Recursive(NearPromise))` from any EVM caller and forwards it to the router's `schedule` method without validating the depth or element count of the `NearPromise` tree: [3](#0-2) 

The router's `execute_scheduled` is intentionally open to any caller: [4](#0-3) 

When `execute_scheduled` is called with a pathological promise, `Self::promise_create(promise)` panics (stack overflow for deep `Then` chains; NEAR runtime rejection for oversized `And` vectors). NEAR reverts state on panic, so the `scheduled_promises.remove` is undone and the promise remains in storage — permanently unexecutable. [5](#0-4) 

The router has no admin withdrawal function. The only recovery path is `send_refund`, which sends a fixed `REFUND_AMOUNT` (2 NEAR) and requires no failed promise results — a condition that can never be satisfied if the promise always panics: [6](#0-5) 

---

### Impact Explanation

NEAR tokens deposited to the router sub-account (the initial 2 NEAR storage stake plus any user-deposited NEAR for XCC calls) are permanently frozen. There is no admin escape hatch or recovery mechanism in the router contract that can drain the account when the scheduled promise is unexecutable. This satisfies the **permanent freezing of funds** impact criterion.

---

### Likelihood Explanation

Any EVM user who can call the XCC precompile can trigger this. The `CrossContractCallArgs::Delayed` path requires no special privilege. The attacker only needs to:
1. Fund their XCC router sub-account (a normal prerequisite for XCC usage).
2. Submit a single EVM transaction calling the XCC precompile with a crafted deeply-nested `NearPromise::Then` chain or an oversized `NearPromise::And` vector.

The EVM gas charged by the precompile is proportional to `input.len()` and `attached_gas`, not to recursion depth, so the cost to the attacker is bounded. The attack is deterministic and repeatable. Likelihood is **medium** — it requires a user to have funded an XCC router, but no admin compromise or external dependency.

---

### Recommendation

1. **Enforce a maximum depth** on `NearPromise::Then` chains and a **maximum element count** on `NearPromise::And` vectors before storing or executing a promise. Validate these limits in the XCC precompile before forwarding to the router, and again in the router's `schedule` and `execute_scheduled` methods.
2. **Add a recovery mechanism** to the router (callable only by the parent) that allows draining the account balance back to the parent in case a scheduled promise is permanently unexecutable.
3. Consider replacing the recursive `recursive_promise_create` with an iterative implementation using an explicit stack, which eliminates the Wasm call-stack overflow risk entirely.

---

### Proof of Concept

**Attack vector (Delayed XCC path):**

1. Attacker funds their XCC router sub-account via `fund_xcc_sub_account`, depositing, e.g., 10 NEAR.
2. Attacker submits an EVM transaction calling the XCC precompile at `cross_contract_call::ADDRESS` with:
   ```
   CrossContractCallArgs::Delayed(
       PromiseArgs::Recursive(
           NearPromise::And(vec![NearPromise::Simple(...); 10_000])
       )
   )
   ``` [7](#0-6) 
3. The engine calls `router.schedule(promise)`. Borsh deserialization of `NearPromise::And` is linear (not recursive), so it succeeds. The promise is stored at nonce `N`.
4. Anyone calls `router.execute_scheduled({nonce: N})`.
5. `self.scheduled_promises.remove(&N)` succeeds (promise removed from storage).
6. `Self::promise_create(promise)` → `recursive_promise_create` → `NearPromise::And` branch → iterates 10,000 elements, each calling `env::promise_create` → NEAR runtime rejects the transaction (exceeds promise-per-receipt limit) or exhausts 300 Tgas → panic.
7. NEAR reverts state: promise is back in storage at nonce `N`.
8. Step 4–7 repeats indefinitely. The 10 NEAR in the router account is permanently frozen. [8](#0-7) [9](#0-8)

fn require_no_failed_promises() -> Result<(), Error> {
    let num_promises = env::promise_results_count();
    for index in 0..num_promises {
        // We can use deprecated `promise_result` rather than `promise_result_checked` safely here,
        // because the promise result could be received from the Aurora Engine itself,
        // and we can be sure that the len of the promise result is within bounds.
        #[allow(deprecated)]
        if env::promise_result(index) == PromiseResult::Failed {
            return Err(Error::CallbackOfFailedPromise);
        }
    }
    Ok(())
}
