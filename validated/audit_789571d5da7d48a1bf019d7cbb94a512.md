### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions via Non-Deploy-Account Mempool Entry - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call only when a `deploy_account` transaction for the sender is already in the mempool (UX feature: send deploy + invoke simultaneously). However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type from that sender, not exclusively `deploy_account`. An attacker who observes that an already-deployed account has a nonce-0 invoke transaction in the mempool can submit a nonce-1 invoke transaction with an **invalid signature** for that account; the gateway skips `__validate__` and admits the invalid transaction to the mempool.

### Finding Description

In `skip_stateful_validations` the skip condition fires when:
- the incoming transaction is an `Invoke`,
- `tx.nonce() == 1`, and
- `account_nonce == 0` (from on-chain state). [1](#0-0) 

When those conditions hold, the function delegates to:

```rust
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())
```

which resolves to:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type from the sender — including a regular nonce-0 `Invoke` — not only a `DeployAccount`. The inline comment claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

The second branch of that reasoning ("future nonces that passed validations") does not apply to a nonce-0 invoke for an account whose on-chain nonce is also 0: nonce-0 is the *current* nonce, not a future one, and its presence in the pool does not imply a pending `deploy_account`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

and the blockifier's `perform_validations` returns early without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

No other check in the gateway pipeline verifies the signature. `validate_by_mempool` only checks for duplicate tx hash and nonce range: [5](#0-4) 

The production default `max_allowed_nonce_gap = 200` means nonce-1 is always within the accepted range when `account_nonce = 0`. [6](#0-5) 

### Impact Explanation

An attacker can inject an invoke transaction carrying an **arbitrary (invalid) signature** into the mempool for any account that currently has `account_nonce = 0` on-chain and any transaction in the mempool pool or recent-block state. Concretely:

1. The attacker observes account `A` (already deployed, on-chain nonce = 0) with a nonce-0 invoke in the mempool.
2. The attacker submits a nonce-1 invoke for `A` with a forged/invalid signature.
3. The gateway admits it: nonce check passes (0 ≤ 1 ≤ 200), `validate_by_mempool` passes (no duplicate hash, nonce in range), `skip_stateful_validations` returns `true`, `__validate__` is never called.
4. The invalid-signature transaction occupies the `(A, nonce=1)` slot in the mempool. If the attacker sets a higher fee, it displaces `A`'s legitimate nonce-1 transaction via fee escalation.

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**: an unauthorized party can inject a transaction that has never been validated by the account's `__validate__` entry point.

### Likelihood Explanation

- Requires `account_nonce = 0` on-chain and any transaction from that account in the mempool — a common state for newly active accounts.
- The attacker needs no privileged access; they only need to observe the public mempool.
- `max_allowed_nonce_gap = 200` (production default) ensures the nonce-1 tx always passes the nonce range check.
- The attack is a single RPC call (`add_transaction`) with a crafted payload.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with one that verifies a **`DeployAccount` transaction specifically** is present for the sender address. For example, expose a `deploy_account_tx_in_pool(address)` query on the mempool and use it in `skip_stateful_validations`:

```diff
- return mempool_client
-     .account_tx_in_pool_or_recent_block(tx.sender_address())
-     .await
-     ...
+ return mempool_client
+     .deploy_account_tx_in_pool(tx.sender_address())
+     .await
+     ...
```

This preserves the intended UX (deploy + invoke in one batch) while closing the path that allows any non-deploy-account mempool entry to trigger validation skip.

### Proof of Concept

```
1. Deploy account A on-chain (on-chain nonce becomes 0).
2. Submit a valid invoke tx for A with nonce=0 (valid signature).
   → Gateway runs __validate__, tx admitted to mempool.
3. Submit an invoke tx for A with nonce=1 and an INVALID/random signature.
   → validate_nonce: 0 ≤ 1 ≤ 200 → passes.
   → validate_by_mempool: no duplicate hash, nonce in range → passes.
   → skip_stateful_validations:
       tx.nonce()==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(A) == true (nonce-0 invoke is in pool) ✓
       → returns true (skip __validate__)
   → run_validate_entry_point: execution_flags.validate=false → returns Ok(()) immediately.
4. Invalid-signature nonce-1 tx is now in the mempool.
   If fee > A's legitimate nonce-1 tx, it displaces it via fee escalation.
```

async fn validate_by_mempool(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<()> {
    let validation_args = ValidationArgs::new(tx, account_nonce);
    mempool_client
        .validate_tx(validation_args)
        .await
        .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
}
