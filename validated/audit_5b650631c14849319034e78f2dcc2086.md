### Title
Missing Negative `deltaGas` Refund Causes User Overcharge in Gas Accounting - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

`compute_gas_refund` in ZKsync OS implements a dual-resource reconciliation step (`deltaGas`) that adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption. However, the symmetric case — when native resource consumption is *lower* than EVM gas consumption (`deltaGas < 0`) — is silently ignored. The user is not refunded for the difference, resulting in a direct financial overcharge. The code itself contains a `// TODO: return delta_gas to gas_used?` comment acknowledging the unresolved case.

---

### Finding Description

ZKsync OS uses a double resource accounting model: EVM gas (ergs) and a native resource that models RISC-V proving cost. At the end of transaction execution, `compute_gas_refund` reconciles the two via `deltaGas`:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
```

where `nativePerGas = gasPrice / nativePrice`.

The code in `refund_calculation.rs` handles only the `deltaGas > 0` branch:

```rust
if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` — meaning native resource consumption is *less* than EVM gas consumption — `gas_used` is not reduced. The user's refund is then computed in `refund_and_commit_fee` as:

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
```

Because `gas_used` was not reduced by `|delta_gas|`, the refund is smaller than it should be. The user is overcharged by `|delta_gas| * gas_price` tokens.

The official design document (`docs/double_resource_accounting.md`) only describes the `deltaGas > 0` case and is silent on the negative case, confirming this is an unresolved gap rather than an intentional design choice.

---

### Impact Explanation

**Vulnerability class**: Resource accounting bug / missing refund.

A user who submits a transaction where native resource consumption (in gas-equivalent units) is lower than EVM gas consumed will not receive the corresponding refund. The overcharge is:

```
overcharge = |delta_gas| * gas_price
           = (gas_used - native_used / native_per_gas) * gas_price
```

**Concrete example**:
- `gas_limit = 100,000`, `gas_price = 10,000`, `native_price = 100`
- `native_per_gas = 10,000 / 100 = 100`
- `native_limit = 100,000 * 100 = 10,000,000`
- Transaction uses `gas_used = 50,000` EVM gas and `native_used = 1,000,000` native units
- `delta_gas = (1,000,000 / 100) - 50,000 = 10,000 - 50,000 = -40,000`
- User is overcharged: `40,000 * 10,000 = 400,000,000` tokens

The overcharged amount flows to the operator (or is burned under `burn_base_fee`). The user's balance is permanently reduced by more than the actual cost of execution.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever `native_used / native_per_gas < gas_used`. This condition is reachable by any unprivileged transaction sender in at least two realistic ways:

1. **High gas price**: `native_per_gas = gas_price / native_price`. A user who sets a high `maxFeePerGas` or `maxPriorityFeePerGas` increases `native_per_gas`, which decreases `native_used / native_per_gas`, making `delta_gas` more negative. This is a normal user action (e.g., to get a transaction included quickly during congestion).

2. **EVM-gas-heavy, native-light operations**: Operations with high EVM gas cost but low RISC-V proving cost (e.g., certain precompile calls, memory-expansion-heavy code, or transactions that accumulate large EIP-3529 EVM refunds reducing `gas_used` while native cost remains fixed) can produce `delta_gas < 0`.

The `#[cfg(not(feature = "unlimited_native"))]` guard confirms this code path is active in production builds (where `unlimited_native` is not set).

---

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, floored at `minimal_gas_used` to preserve the intrinsic gas minimum:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // TODO: return delta_gas to gas_used?
    // Native cost is lower than EVM gas cost; refund the difference.
    gas_used = gas_used
        .saturating_sub((-delta_gas) as u64)
        .max(minimal_gas_used);
}
```

This mirrors the symmetric treatment already applied for `delta_gas > 0` and ensures users are not overcharged when native resource consumption is lower than EVM gas consumption.

---

### Proof of Concept

1. Deploy ZKsync OS with `unlimited_native` feature **disabled** (production configuration).
2. Submit an EIP-1559 transaction with a high `maxFeePerGas` (e.g., 100× the `native_price`), so `native_per_gas` is large.
3. Execute a transaction body that consumes a moderate amount of EVM gas (e.g., 50,000) but a relatively small amount of native resource.
4. Observe that `delta_gas = (native_used / native_per_gas) - gas_used < 0`.
5. Observe that `gas_used` returned by `compute_gas_refund` is **not** reduced.
6. Observe that `refund_and_commit_fee` credits the user with `(gas_limit - gas_used) * gas_price` — a smaller refund than `(gas_limit - (gas_used + delta_gas)) * gas_price`.
7. Confirm the user's final balance is lower than expected by `|delta_gas| * gas_price` tokens.
