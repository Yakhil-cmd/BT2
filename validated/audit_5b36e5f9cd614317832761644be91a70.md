### Title
Cross-Context COA Ownership Proof Replay via Missing `evmAddress` Domain Separation in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` is the Cadence-side implementation of COA ownership proof validation. It is reachable in two distinct contexts: via the EVM precompile `verifyCOAOwnershipProof` (where `evmAddress` is always enforced to be the calling COA's own address) and directly as a public Cadence function (where `evmAddress` is caller-supplied and unconstrained). In both contexts, the cryptographic signature is verified only against `signedData`, which is never required to encode `evmAddress`. This inconsistency means a signature produced to prove ownership of COA-A can be replayed in the direct-Cadence context to assert ownership of COA-B, a different COA owned by the same Cadence account.

---

### Finding Description

**Root cause — two verification contexts with different `evmAddress` enforcement:**

**Context 1 — EVM precompile path (enforced):**
The COA Solidity contract's ERC-1271 `isValidSignature` always passes `address(this)` as the `evmAddress` argument to the precompile:

```solidity
cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig   // evmAddress = calling COA's own address
    )
);
``` [1](#0-0) 

The precompile then calls `validateCOAOwnershipProof` via `coaOwnershipProofValidator`, forwarding the proof including the enforced `evmAddress`: [2](#0-1) 

**Context 2 — Direct Cadence call (unenforced):**
`EVM.validateCOAOwnershipProof` is `access(all)` and callable by any Cadence script or transaction. The caller supplies `evmAddress` freely. The function verifies the cryptographic signature against `signedData` only — it never checks that `signedData` encodes `evmAddress`: [3](#0-2) 

The signature verification step: [4](#0-3) 

After the signature passes, the function separately checks that the COA at `path` has the given `evmAddress` — but this check is structurally decoupled from the signature: [5](#0-4) 

The code itself acknowledges the inconsistency: [6](#0-5) 

**Exploit flow:**

1. Cadence account `A` owns two COAs: COA-A (EVM address `0xAAAA`) at `/public/coaA` and COA-B (EVM address `0xBBBB`) at `/public/coaB`.
2. Account `A` signs `signedData` (e.g., for a protocol that uses COA-A). The signature `S` is published on-chain or observable.
3. An attacker calls `EVM.validateCOAOwnershipProof(address: A, path: /public/coaB, signedData: signedData, keyIndices: [0], signatures: [S], evmAddress: 0xBBBB)` from a Cadence script.
4. The function verifies `S` against `signedData` using account `A`'s key — this passes, because `S` is a valid signature from `A`.
5. The function verifies that the COA at `/public/coaB` has EVM address `0xBBBB` — this also passes.
6. The function returns `isValid: true`, falsely asserting that the caller proved ownership of COA-B.

---

### Impact Explanation

Any Cadence protocol that calls `EVM.validateCOAOwnershipProof` directly to gate access to on-chain assets or privileges (e.g., token claims, bridge withdrawals, governance votes tied to a specific COA address) can be deceived. An attacker who observes any valid signature from the target Cadence account — regardless of the original purpose of that signature — can replay it to assert ownership of a different COA belonging to the same account. This constitutes unauthorized cross-asset impersonation: the attacker gains protocol-level access to resources associated with COA-B without ever controlling COA-B's resource or the Cadence account's keys.

---

### Likelihood Explanation

The attacker entry path is reachable by any unprivileged Cadence transaction sender. The preconditions are:
- The target Cadence account has more than one COA (the code comment acknowledges this is uncommon today but not prevented).
- A valid signature from the account's keys is observable on-chain (e.g., from a prior ERC-1271 interaction, a bridge operation, or any other protocol use).
- A Cadence protocol uses `EVM.validateCOAOwnershipProof` directly rather than relying solely on the EVM precompile path.

The first condition is the primary limiting factor. As the ecosystem matures and multi-COA accounts become more common (e.g., for DeFi vaults, multi-purpose wallets), likelihood increases. The second and third conditions are realistic in any active DeFi deployment.

---

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof` itself, rather than relying on callers to do so. Specifically, before verifying the signature, reconstruct the expected signed message as `hash(evmAddress || signedData)` (or a structured equivalent) and verify the signature against that reconstructed message. This makes the signature domain-specific to the target COA address, eliminating the cross-address replay. The EVM precompile path already passes the correct `evmAddress`; the Cadence-side function should enforce the same binding.

---

### Proof of Concept

```cadence
// Attacker Cadence transaction
import EVM from 0x<EVMContractAddress>

transaction {
    execute {
        // signedData and sig were observed from a prior on-chain interaction
        // where account 0xVICTIM proved ownership of COA-A (evmAddress 0xAAAA...)
        let signedData: [UInt8] = [/* observed signedData */]
        let sig: [UInt8]        = [/* observed signature S */]

        // Replay against COA-B (evmAddress 0xBBBB...) owned by the same account
        let result = EVM.validateCOAOwnershipProof(
            address:    0xVICTIM,
            path:       /public/coaB,
            signedData: signedData,
            keyIndices: [0],
            signatures: [sig],
            evmAddress: [0xBB, 0xBB, /* ... 20 bytes of COA-B address */]
        )
        // result.isValid == true — attacker has falsely proven ownership of COA-B
        assert(result.isValid, message: "replay succeeded")
    }
}
```

The signature `S` is cryptographically valid for account `0xVICTIM`'s key against `signedData`. Because `validateCOAOwnershipProof` never checks that `signedData` encodes the target `evmAddress`, the replay succeeds unconditionally as long as the COA capability exists at the given path.
