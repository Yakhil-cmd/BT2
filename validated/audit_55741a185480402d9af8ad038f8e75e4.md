### Title
No setter for `l1_reward_supplier` in `MintingCurve`, whose address may change after L1 upgrade — (File: `src/minting_curve/minting_curve.cairo`)

---

### Summary

The `MintingCurve` contract stores the L1 reward supplier address (`l1_reward_supplier`) as a `felt252` set only at construction time. This address is the sole authentication gate for the `update_total_supply` L1 handler. There is no setter to update it post-deployment. If the L1 reward supplier contract is redeployed or upgraded to a new address, all future `update_total_supply` L1→L2 messages will be permanently rejected, freezing `total_supply` at a stale value and causing `yearly_mint()` — and therefore all staker reward calculations — to return incorrect results.

---

### Finding Description

`MintingCurve` stores `l1_reward_supplier` in its `Storage` struct:

```cairo
// src/minting_curve/minting_curve.cairo:68
l1_reward_supplier: felt252,
```

It is written once in the constructor:

```cairo
// src/minting_curve/minting_curve.cairo:102
self.l1_reward_supplier.write(l1_reward_supplier);
```

The only place it is read is inside the `update_total_supply` L1 handler, which uses it as an allowlist check:

```cairo
// src/minting_curve/minting_curve.cairo:119-124
#[l1_handler]
fn update_total_supply(ref self: ContractState, from_address: felt252, total_supply: Amount) {
    assert!(
        from_address == self.l1_reward_supplier.read(),
        "{}",
        Error::UNAUTHORIZED_MESSAGE_SENDER,
    );
```

If the L1 reward supplier is redeployed at a new address (e.g., due to a non-proxy upgrade or contract migration), every subsequent `update_total_supply` call will revert at this assertion. The `total_supply` field is then permanently frozen at its last-written value.

`total_supply` is a direct input to `yearly_mint()`:

```cairo
// src/minting_curve/minting_curve.cairo:139-147
fn yearly_mint(self: @ContractState) -> Amount {
    let total_supply = self.total_supply.read();
    ...
    let yearly_mint = self.compute_yearly_mint(
        total_stake: total_stake.to_strk_native_amount(), :total_supply,
    );
```

And `compute_yearly_mint` uses it directly in the reward formula:

```cairo
// src/minting_curve/minting_curve.cairo:177-178
let stake_times_supply: u256 = total_stake.wide_mul(total_supply);
self.c_num.read().into() * stake_times_supply.sqrt() / C_DENOM.into()
```

`yearly_mint()` is called by `RewardSupplier` every epoch to compute both attestation-based and consensus-based rewards for all stakers and pool members:

```cairo
// src/reward_supplier/reward_supplier.cairo:156, 176
let yearly_mint = minting_curve_dispatcher.yearly_mint();
```

There is no `set_l1_reward_supplier` function anywhere in the `MintingCurve` contract or its interface (`src/minting_curve/interface.cairo`). The `IRewardSupplierConfig` interface only exposes `set_block_duration_config`.

The same pattern exists in `RewardSupplier` itself: `l1_reward_supplier` is stored with no setter and is used to route outbound L2→L1 mint-request messages:

```cairo
// src/reward_supplier/reward_supplier.cairo:333-336
fn send_mint_request_to_l1_reward_supplier(self: @ContractState) {
    let payload: Span<felt252> = array![self.base_mint_amount.read().into()].span();
    let to_address = self.l1_reward_supplier.read();
    send_message_to_l1_syscall(:to_address, :payload).unwrap_syscall();
```

If this address is stale, mint requests are silently sent to a dead address, and no new STRK is ever bridged to L2.

---

### Impact Explanation

**Primary (MintingCurve):** A frozen `total_supply` causes `yearly_mint()` to return a permanently understated value. Because STRK total supply grows monotonically as rewards are minted, a stale value means the computed reward rate is lower than it should be. Every staker and pool member receives less than their entitled yield for the entire duration of the staleness. This constitutes **theft of unclaimed yield** for all protocol participants.

**Secondary (RewardSupplier):** Mint requests routed to a stale L1 address are never fulfilled. The reward supplier's STRK balance eventually falls below `unclaimed_rewards`, causing `claim_rewards` to revert for stakers. This constitutes **temporary freezing of funds**.

Both impacts fall within the allowed scope.

---

### Likelihood Explanation

The L1 infrastructure includes an upgrade framework (proxy pattern referenced in the wiki under "L1 Proxy and Upgrade Framework"). If the L1 `RewardSupplier` is ever migrated to a new address — whether due to a bug fix, feature upgrade, or emergency redeployment — the L2 `MintingCurve` and `RewardSupplier` contracts have no mechanism to track the change. This is a latent, time-delayed vulnerability: it is harmless until the first L1 address change, at which point it becomes permanent and protocol-wide.

---

### Recommendation

Add a privileged setter (e.g., restricted to `app_governor` or `token_admin`) in both `MintingCurve` and `RewardSupplier` to update `l1_reward_supplier` post-deployment:

```cairo
fn set_l1_reward_supplier(ref self: ContractState, new_l1_reward_supplier: felt252) {
    self.roles.only_app_governor();
    self.l1_reward_supplier.write(new_l1_reward_supplier);
    // emit event
}
```

Expose this through the respective config interfaces (`IMintingCurveConfig`, `IRewardSupplierConfig`).

---

### Proof of Concept

1. Protocol deploys `MintingCurve` with `l1_reward_supplier = 0xABCD` (current L1 address). `total_supply` is updated correctly.
2. L1 `RewardSupplier` is upgraded/redeployed at `0xDEAD`.
3. L1 `tick()` sends an `update_total_supply` message from `0xDEAD` to `MintingCurve`.
4. `update_total_supply` asserts `from_address == self.l1_reward_supplier.read()` → `0xDEAD != 0xABCD` → **revert**.
5. `total_supply` is now frozen at its last value (e.g., 10B STRK), even as actual supply grows to 11B, 12B, etc.
6. `yearly_mint()` computes `C * sqrt(total_stake * 10B)` instead of `C * sqrt(total_stake * 12B)` — a ~9.5% underestimate at 12B supply.
7. Every epoch, `RewardSupplier.update_current_epoch_block_rewards()` uses this understated `yearly_mint`, distributing proportionally less to all stakers and pool members.
8. There is no admin function to fix `l1_reward_supplier` in `MintingCurve`, so the error compounds indefinitely.
