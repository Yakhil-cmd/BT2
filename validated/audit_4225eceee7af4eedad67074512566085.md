### Title
ETH Permanently Locked in `RewardSupplier.tick()` When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is a public `payable` function that accepts ETH to fund two cross-chain operations. When there are no pending mint requests (`amountToMint == 0`), the function silently succeeds without using or refunding the ETH, permanently locking any ETH sent by the caller inside the contract.

---

### Finding Description

`tick()` is the critical L1 function that mints STRK rewards and delivers them to L2. It performs two payable cross-chain operations:

1. `bridge().depositWithMessage{value: msgFee}(...)` — sends minted tokens to L2 via StarkGate
2. `messagingContract().sendMessageToL2{value: msgFee}(...)` — sends a total supply update to the L2 `MintingCurve`

Both operations are gated behind `if (amountToMint > 0)`. When `amountToMint == 0`, the entire block is skipped, but the function is still `payable` and accepts ETH without any refund path:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... uses msg.value for two cross-chain calls
    }
    // If amountToMint == 0: msg.value is silently consumed and locked
}
``` [1](#0-0) 

There is no `receive()`, `fallback()`, or ETH withdrawal function anywhere in the contract or its inherited contracts (`RewardSupplierStorage`, `Identity`, `ProxySupportImpl`), so any ETH sent when `amountToMint == 0` is irrecoverable. [2](#0-1) 

The `requiredMinting()` view function is public and can be called off-chain to check whether minting is needed, but there is no atomicity guarantee between that check and the `tick()` call. [3](#0-2) 

---

### Impact Explanation

Any ETH sent to `tick()` when `amountToMint == 0` is permanently locked in the `RewardSupplier` contract with no recovery path. This is direct, irreversible loss of ETH for the caller. The impact maps to **Medium: Griefing with no profit motive but damage to users or protocol** — the protocol itself has no incentive to cause this, but the design flaw causes real ETH loss for callers who trigger the function in a no-op state.

---

### Likelihood Explanation

`tick()` is callable by any unprivileged public account with no access control. A realistic path to loss:

1. The L2 `RewardSupplier` emits a `MintRequest` event (L2→L1 message).
2. Caller A observes the pending message via `requiredMinting()` and submits `tick()` with ETH.
3. Caller B front-runs Caller A, consuming the pending mint request.
4. Caller A's transaction executes with `amountToMint == 0`; the function returns successfully, and Caller A's ETH is locked.

This race condition is realistic on Ethereum mainnet where mempool visibility is public. Additionally, a naive caller who sends ETH without first checking `requiredMinting()` will also lose funds. Likelihood: **Medium**.

---

### Recommendation

Add an explicit refund of unused ETH when `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic
    } else if (msg.value > 0) {
        // Refund ETH if there is nothing to mint
        (bool success, ) = msg.sender.call{value: msg.value}("");
        require(success, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, revert immediately if `msg.value > 0` and `amountToMint == 0`.

---

### Proof of Concept

1. L2 `RewardSupplier` sends one mint request to L1 (L2→L1 message pending).
2. Caller B calls `tick{value: 0}()` — this succeeds with zero ETH (no fee needed for the messaging calls if `msg.value == 0` is accepted by the bridge/messaging contract, or it reverts harmlessly).
3. Alternatively, Caller B calls `tick{value: X}()` first, consuming the pending mint request.
4. Caller A then calls `tick{value: Y}()` where `Y > 0`.
5. `requiredMinting()` returns `(0, 0)` because no pending messages remain.
6. The `if (amountToMint > 0)` block is skipped.
7. `tick()` returns successfully. Caller A's `Y` wei is now permanently locked in `RewardSupplier`.
8. No withdrawal function exists; the ETH is unrecoverable. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
