### Title
Owner Can Permanently Freeze All User ETH and Token Withdrawals via `set_eth_connector_contract_account` - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` allows the engine owner to redirect all withdrawal-related cross-contract calls to an arbitrary NEAR account ID without checking existing user balances or validating that the new account implements the required connector interface. Pointing the connector to a non-functional or non-existent account permanently breaks every user-facing withdrawal path in Aurora.

---

### Finding Description

The function `set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` (lines 417–438) is callable by the engine owner (or via a private self-call). It unconditionally overwrites the stored connector account ID with no validation:

```rust
// engine/src/contract_methods/connector.rs:417-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();
        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }
        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;
        set_connector_account_id(io, &args.account);          // ← no balance check
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
```

The stored connector account ID is the single routing target for every connector-related operation. The internal helper `return_promise` (lines 598–617) reads this value and creates a cross-contract call to it for **every** user-facing connector method:

```rust
// engine/src/contract_methods/connector.rs:598-617
fn return_promise<I: IO + PromiseHandler, E: Env>(...) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,  // ← single point of failure
        method: method.to_string(),
        ...
    };
    ...
}
```

This `return_promise` helper is used by: `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, `storage_withdraw`, `ft_metadata`, `ft_balance_of`, and `ft_total_eth_supply_on_near`.

Additionally, both EVM-level exit precompiles read the same storage key at execution time:

- `ExitToEthereum::get_eth_connector_contract_account` (`engine-precompiles/src/native.rs`, lines 837–841)
- `ExitToNear::get_eth_connector_contract_account` (`engine-precompiles/src/native.rs`, lines 289–292)

If the owner sets the connector account to a non-existent or non-conforming NEAR account, every one of these paths fails at the cross-contract call stage. Because the EVM balance state (stored under `KeyPrefix::Balance`) is separate from the connector's NEP-141 ledger, the EVM balances remain recorded but are completely unreachable: users cannot withdraw ETH, cannot transfer tokens out, and cannot exit via the precompiles.

There is no check for existing user balances before the overwrite, no validation that the new account implements `engine_withdraw` / `engine_ft_transfer` / etc., and no migration mechanism for funds already held in the old connector.

---

### Impact Explanation

**Permanent freezing of funds — Critical.**

All ETH and ERC-20 token balances held by every Aurora user become permanently inaccessible the moment the connector account is changed to a non-functional address. The EVM state records the balances, but no withdrawal path exists: `withdraw`, `ft_transfer`, `ft_transfer_call`, `ExitToNear`, and `ExitToEthereum` all route through the stored connector account ID. There is no fallback or recovery mechanism in the contract.

---

### Likelihood Explanation

The engine owner is a single privileged NEAR account. The `set_eth_connector_contract_account` entrypoint is a legitimate administrative function (used during connector migrations and upgrades, as seen in test setup). A mistaken account ID, a typo, or a malicious owner action is sufficient to trigger the freeze. No user interaction is required after the owner call; the freeze is immediate and affects all users simultaneously.

---

### Recommendation

Before overwriting the connector account ID, `set_eth_connector_contract_account` should:

1. **Verify the new account exists and implements the required interface** (e.g., via a view call to `engine_withdraw` or a capability check) before committing the change.
2. **Enforce a time-lock or two-step migration** so that users have a window to withdraw funds from the old connector before the switch takes effect.
3. **Optionally**, record the old connector account ID and allow users to withdraw from it directly during a transition period.

---

### Proof of Concept

1. User deposits ETH into Aurora; their balance is recorded in the EVM state and in the old connector's NEP-141 ledger.
2. Owner calls `set_eth_connector_contract_account` with `account = "nonexistent.near"` (a valid `AccountId` string that resolves to no deployed contract).
3. User calls `withdraw(recipient_address, amount)` on the engine.
4. `withdraw` calls `return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)`.
5. `return_promise` reads `get_connector_account_id` → returns `"nonexistent.near"`.
6. The NEAR runtime creates a cross-contract call to `nonexistent.near::engine_withdraw` → the call fails (account does not exist).
7. The user's EVM balance is unchanged (no debit occurred), but the withdrawal promise failed — the user receives no ETH on NEAR or Ethereum.
8. Every subsequent withdrawal attempt by every user follows the same path and fails identically.
9. The `ExitToEthereum` and `ExitToNear` precompiles also call `get_eth_connector_contract_account` at EVM execution time and route to the same broken account, so EVM-level exits are equally broken.
10. All user funds are permanently frozen with no on-chain recovery path.
