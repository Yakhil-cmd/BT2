### Title
`PresignOutput` Does Not Cache `max_malicious` or Participant Set, Enabling Split-View Attacks via Threshold Mismatch — (`src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

The `PresignOutput` struct in the robust ECDSA scheme does not record the `max_malicious` threshold or the presigning participant set used to produce it. The `sign()` function accepts any caller-supplied `max_malicious` value independently, with no cross-validation against the presigning phase. A malicious coordinator can exploit this by presenting the same presignature to two different participant subsets with different message hashes, enabling a split-view attack that extracts the aggregate secret key.

---

### Finding Description

`PresignOutput` in `src/ecdsa/robust_ecdsa/mod.rs` stores only the cryptographic outputs `(big_r, c, e, alpha, beta)`: [1](#0-0) 

Neither `max_malicious` nor the presigning participant set is persisted in this struct.

The `presign()` entry point in `src/ecdsa/robust_ecdsa/presign.rs` enforces `participants.len() == 2*max_malicious + 1` at initialization time: [2](#0-1) 

However, because `PresignOutput` does not record these values, the downstream `sign()` function in `src/ecdsa/robust_ecdsa/sign.rs` accepts any independently supplied `max_malicious`: [3](#0-2) 

The only check is that `participants.len() == 2*max_malicious + 1` for the signing call itself — there is no verification that this matches the presigning call. The `compute_signature_share` function then linearizes shares using the *signing* participant set's Lagrange coefficients: [4](#0-3) 

The rerandomization step (`RerandomizationArguments`) binds the presignature to `(Y, ε, h, R, ρ)` via HKDF but does **not** include the participant set in the binding, as confirmed by the protocol specification: [5](#0-4) 

The security documentation explicitly states the required constraint but leaves enforcement entirely to the caller: [6](#0-5) 

This is the direct analog of the `passThresholdBps` bug: a threshold parameter that governs session validity is not cached in the session output, so a different value can be silently substituted at evaluation time.

---

### Impact Explanation

Each participant's `alpha_i` share is a share of `k^{-1}` (the inverse nonce), and `beta_i` is a share of `k^{-1} · x` (where `x` is the aggregate secret key): [7](#0-6) 

If a malicious coordinator runs two signing sessions against the same presignature with different message hashes `h1` and `h2`, each honest participant `i` computes:

- Session A: `s_i(h1) = alpha_i · h1 + beta_i · Rx + e_i`
- Session B: `s_i(h2) = alpha_i · h2 + beta_i · Rx + e_i`

Subtracting: `alpha_i = (s_i(h1) − s_i(h2)) / (h1 − h2)`

Since `alpha_i` is a share of `k^{-1}`, the coordinator reconstructs `k^{-1}`, then `k`, and finally recovers `x` from any valid signature via `x = (s·k − h) / Rx`. This is the split-view attack documented in the codebase itself: [8](#0-7) 

**Impact: Critical** — full extraction of the aggregate secret signing key.

---

### Likelihood Explanation

The malicious coordinator is an explicitly modeled adversary in this library (the `sign()` API designates a coordinator role). The coordinator controls which participant set and `max_malicious` value is passed to each signing invocation. Because `PresignOutput` carries no record of the presigning parameters, honest participants have no protocol-level means to detect that the coordinator is reusing a presignature across two sessions with different subsets or different thresholds. The attack requires only two signing sessions and is algebraically straightforward once the coordinator holds both sets of signature shares.

---

### Recommendation

Cache `max_malicious` and a commitment to the presigning participant set inside `PresignOutput` (analogous to how `totalVotingPower` is recommended to be cached per-proposal in the external report). The `sign()` function should verify that the caller-supplied `max_malicious` and participant set match the values recorded in the presignature before proceeding. Concretely:

```rust
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub c: Scalar,
    pub e: Scalar,
    pub alpha: Scalar,
    pub beta: Scalar,
    // Add these:
    pub max_malicious: MaxMalicious,
    pub participants_commitment: [u8; 32], // e.g. hash of sorted participant IDs
}
```

Then in `sign()`, assert:
```rust
assert!(presignature.max_malicious == max_malicious);
assert!(presignature.participants_commitment == hash(participants));
```

### Proof of Concept

1. Run `presign()` with `participants = [P1, P2, P3, P4, P5]`, `max_malicious = 2` (N1 = 5). Each participant receives a `PresignOutput`.

2. Malicious coordinator runs `sign()` **Session A** with `participants = [P1, P2, P3]`, `max_malicious = 1` (N2 = 3), `msg_hash = h1`, passing the same `PresignOutput` (after rerandomization). The library accepts this call because `3 == 2·1 + 1`.

3. Malicious coordinator runs `sign()` **Session B** with `participants = [P3, P4, P5]`, `max_malicious = 1` (N3 = 3), `msg_hash = h2 ≠ h1`, passing the same `PresignOutput`. Again accepted.

4. For participant P3 (present in both sessions), the coordinator holds `s_P3(h1)` and `s_P3(h2)`. It computes `alpha_P3 = (s_P3(h1) − s_P3(h2)) / (h1 − h2)`, recovering P3's share of `k^{-1}`.

5. Repeating across the overlap, the coordinator reconstructs `k^{-1}`, then `k`, then the secret key `x`. No valid signature is required; the share values alone suffice.

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

fn compute_signature_share(
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
    participants: &ParticipantList,
    me: Participant,
) -> Result<SerializableScalar<C>, ProtocolError> {
    // (beta_i + tweak * k_i) * delta^{-1}
    let big_r = presignature.big_r;
    let big_r_x_coordinate = x_coordinate(&big_r);
    // beta * Rx + e
    let beta = presignature.beta * big_r_x_coordinate + presignature.e;

    let s_me = msg_hash * presignature.alpha + beta;
    // lambda_i * s_i
    let linearized_s_me = s_me * participants.lagrange::<C>(me)?;
    Ok(SerializableScalar::<C>(linearized_s_me))
}

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.
