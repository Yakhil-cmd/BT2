### Title
`RewardSupplier.unclaimed_rewards` Exceeds Actual Token Balance, Causing `claim_rewards` and `unstake_action` to Fail - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `RewardSupplier` contract maintains an `unclaimed_rewards` counter that is incremented immediately when staking rewards are calculated, but the actual STRK token balance of the contract may be lower than `unclaimed_rewards` because L1 funds are requested and received asynchronously via the StarkGate bridge. Any call to `claim_rewards` on the `RewardSupplier` that exceeds the current token balance will revert, temporarily freezing staker and delegator reward withdrawals.

---

### Finding Description

When a staker attests or `update_rewards` is called, the staking contract calls `reward_supplier_dispatcher.update_unclaimed_rewards_from_staking_contract(rewards)`, which increments `unclaimed_rewards` and then calls `request_funds`: [1](#0-0) 

Inside `request_funds`, the contract computes:

```
credit = balance + l1_pending_requested_amount
debit  = unclaimed_rewards
```

If `credit < debit + threshold`, it sends L1 messages to request more STRK. After this call, the invariant maintained is:

```
balance + l1_pending_requested_amount >= unclaimed_rewards + threshold
``` [2](#0-1) 

Critically, this invariant does **not** guarantee `balance >= unclaimed_rewards`. The `l1_pending_requested_amount` represents funds requested from L1 but not yet received. During the L1→L2 bridge latency window, `balance < unclaimed_rewards` is a normal operating state.

When a staker calls `claim_rewards` or `unstake_action`, the staking contract calls `send_rewards_to_staker`, which calls `claim_from_reward_supplier`, which calls `RewardSupplier.claim_rewards(amount)`: [3](#0-2) 

The guard at line 214 only checks `amount <= unclaimed_rewards`. It does **not** check whether the contract's actual token balance is sufficient. The subsequent `checked_transfer` at line 219 will revert if `balance < amount`.

The same failure path exists in `_update_rewards` when pool rewards are claimed from the `RewardSupplier` before being forwarded to delegation pools: [4](#0-3) 

And in `send_rewards_to_staker` called from both `claim_rewards` and `unstake_action`: [5](#0-4) 

---

### Impact Explanation

Any staker or delegator calling `claim_rewards` or `unstake_action` during the window when `RewardSupplier.balance < unclaimed_rewards` will receive a revert. The staker's principal is not at risk, but their accrued, unclaimed STRK rewards are temporarily inaccessible. The `unstake_action` function is also blocked because it calls `send_rewards_to_staker` before returning the principal, meaning a staker cannot complete their exit from the protocol until the RewardSupplier's balance is replenished by the L1 bridge.

This matches the allowed impact: **High — Temporary freezing of unclaimed yield / Temporary freezing of funds**.

---

### Likelihood Explanation

This is a reachable, normal operating condition. The L1→L2 StarkGate bridge has non-trivial latency. Every time rewards are distributed (every block in consensus mode, or every attestation in pre-consensus mode), `unclaimed_rewards` grows. If multiple stakers accumulate rewards faster than L1 funds arrive, the balance will be insufficient. Any unprivileged staker or delegator can trigger the failure simply by calling `claim_rewards` at the wrong time. No special access or attack is required.

---

### Recommendation

Before executing the `checked_transfer` in `RewardSupplier.claim_rewards`, add a balance check and revert with a descriptive error if the actual token balance is insufficient:

```cairo
let balance: Amount = token_dispatcher
    .balance_of(account: get_contract_address())
    .try_into()
    .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
assert!(amount <= balance, "{}", Error::INSUFFICIENT_BALANCE_FOR_CLAIM);
```

Alternatively, the `request_funds` invariant should be tightened to ensure `balance >= unclaimed_rewards` at all times (not just `balance + l1_pending_requested_amount >= unclaimed_rewards + threshold`), or the staking contract should gracefully handle a failed claim from the reward supplier rather than reverting the entire user-facing transaction.

---

### Proof of Concept

1. Staker A attests → `update_rewards` is called → `update_unclaimed_rewards_from_staking_contract(1000 STRK)` → `unclaimed_rewards` becomes `1000 STRK`, `request_funds` sends an L1 message but the bridge has not yet delivered the tokens → `RewardSupplier.balance = 0`.
2. Staker A immediately calls `Staking.claim_rewards(staker_A)`.
3. Staking contract calls `send_rewards_to_staker` → `claim_from_reward_supplier(amount=1000)`.
4. `RewardSupplier.claim_rewards(1000)`: check `1000 <= unclaimed_rewards (1000)` passes.
5. `token_dispatcher.checked_transfer(recipient: staking_contract, amount: 1000)` reverts because `balance = 0`.
6. The entire `claim_rewards` transaction reverts. Staker A cannot claim rewards until L1 funds arrive.

The same revert occurs in `unstake_action`, blocking the staker from completing their exit. [6](#0-5) [7](#0-6)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

        fn unstake_action(ref self: ContractState, staker_address: ContractAddress) -> Amount {
            // Prerequisites and asserts.
            self.general_prerequisites();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let unstake_time = staker_info
                .unstake_time
                .expect_with_err(Error::MISSING_UNSTAKE_INTENT);
            assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Send rewards to staker's reward address.
            // It must be part of this function's flow because staker_info is about to be erased.
            let token_dispatcher = strk_token_dispatcher();
            self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
            // Update staker info to storage (it will be erased later).
            // This is done here to avoid re-entrancy.
            self.write_staker_info(:staker_address, :staker_info);

            let staker_amount = self.get_own_balance(:staker_address).to_strk_native_amount();
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            self.remove_staker(:staker_address, :staker_info, :staker_pool_info);

            // Return stake to staker.
            token_dispatcher
                .checked_transfer(recipient: staker_address, amount: staker_amount.into());

            self

                );
            // Clear staker pools.
            staker_pool_info.pools.clear();
            staker_amount
        }
