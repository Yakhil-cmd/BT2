### Title
Missing Proposer Key Weight Check Allows Unauthorized Sequence Number Increment - (`fvm/transactionVerifier.go`)

### Summary

`verifyTransaction` in `fvm/transactionVerifier.go` enforces the `AccountKeyWeightThreshold` for the payer and every authorizer, but never enforces it for the proposer when the proposer is distinct from both the payer and all authorizers. An unprivileged sender who holds any key — even a sub-threshold key — on a victim account can use that key as the proposer, causing the victim account's on-chain sequence number to be incremented without the account owner's full authorization.

### Finding Description

`verifyTransaction` performs three weight-related checks:

1. **Authorizers** — each address in `tx.Authorizers` (excluding the payer) must accumulate `payloadWeights[addr] >= keyWeightThreshold`.
2. **Payer** — `envelopeWeights[tx.Payer] >= keyWeightThreshold`.
3. **Proposer** — only that a signature matching `(tx.ProposalKey.Address, tx.ProposalKey.KeyIndex)` exists (`foundProposalSignature`); the accumulated weight for that address is **never compared against the threshold**. [1](#0-0) 

The `getAccountKeysAndAggregateWeights` helper aggregates the proposer's key weight into `payloadWeights` or `envelopeWeights` (depending on whether the proposer signed the payload or the envelope), but those maps are only consulted for authorizers and the payer respectively. If the proposer is neither an authorizer nor the payer, its accumulated weight is silently ignored. [2](#0-1) 

The same gap exists in the earlier, lighter-weight `checkAccounts` function in the access/collection validator: it verifies that at least one signature from the proposer's address is present, but imposes no weight requirement. [3](#0-2) 

### Impact Explanation

The Flow sequence number is stored as on-chain account state. Incrementing it without the account owner's authorization is an unauthorized mutation of that account's state. A victim whose sequence number has been advanced will find their own pending transactions rejected with a sequence-number mismatch, effectively locking them out until they re-issue transactions with the new (attacker-controlled) sequence number. In a multi-key account where the attacker holds only a partial-weight key, this constitutes unauthorized account mutation that the key-weight threshold was specifically designed to prevent.

### Likelihood Explanation

Multi-key accounts are a first-class Flow feature (e.g., hardware-wallet + hot-wallet setups, corporate accounts). Any party that has ever been granted even a single low-weight key on an account — a revoked employee, a compromised service key, a key shared for a limited purpose — can exploit this gap. No special node access, admin privilege, or social engineering is required; a standard signed transaction is the only entry point.

### Recommendation

After `getAccountKeysAndAggregateWeights` returns, add an explicit weight check for the proposer address, mirroring the existing payer and authorizer checks:

```go
// proposer must have sufficient weight
proposerWeights := payloadWeights
if _, signedEnvelope := envelopeWeights[tx.ProposalKey.Address]; signedEnvelope {
    proposerWeights = envelopeWeights
}
if !v.hasSufficientKeyWeight(proposerWeights, tx.ProposalKey.Address, keyWeightThreshold) {
    return errors.NewAccountAuthorizationErrorf(
        tx.ProposalKey.Address,
        "proposer account does not have sufficient signatures (%d < %d)",
        proposerWeights[tx.ProposalKey.Address],
        keyWeightThreshold)
}
```

The same guard should be added to `checkAccounts` in `access/validator/validator.go`.

### Proof of Concept

1. Account **A** has two keys: key-0 with weight 1000 (owner's key) and key-1 with weight 499 (attacker's key, below the 1000 threshold).
2. Attacker constructs a transaction:
   - `ProposalKey = {Address: A, KeyIndex: 1, SequenceNumber: <current>}`
   - `Payer = <attacker's own account>` (full-weight key)
   - `Authorizers = []`
3. Attacker signs the payload with key-1 on account A and signs the envelope with their own full-weight key.
4. `verifyTransaction` runs:
   - Authorizer loop: no authorizers → passes.
   - Payer check: attacker's own account has full weight → passes.
   - Proposer check: `foundProposalSignature = true` (key-1 signature found) → passes. **Weight of key-1 (499) is never compared against 1000.**
5. Transaction executes; account A's sequence number is incremented to `<current> + 1`.
6. Account A's owner's pending transaction (also using sequence number `<current>`) is now rejected. [4](#0-3) [5](#0-4)

```go
// verifyTransaction verifies the transaction from the given procedure,
// and check the Authorizers have enough weights.
func (v *TransactionVerifier) verifyTransaction(
	tracer tracing.TracerSpan,
	proc *TransactionProcedure,
	txnState storage.TransactionPreparer,
	keyWeightThreshold int,
) error {
	span := tracer.StartChildSpan(trace.FVMVerifyTransaction)
	if span.Tracer != nil {
		span.SetAttributes(
			attribute.String("transaction.ID", proc.ID.String()),
		)
	}
	defer span.End()

	tx := proc.Transaction
	if tx.Payer == flow.EmptyAddress {
		return errors.NewInvalidAddressErrorf(tx.Payer, "payer address is invalid")
	}

	// return the signature entries (both payload and envelope) and empty weight maps
	// that will be used to aggregate weights.
	// the account keys are deduplicated during this call.
	signatures, payloadWeights, envelopeWeights, err := newSignatureEntries(tx)
	if err != nil {
		return err
	}

	accounts := environment.NewAccounts(txnState)

	if keyWeightThreshold < 0 {
		return nil
	}

	// at this point, account keys are guaranteed to be unique across all signatures
	err = v.getAccountKeysAndAggregateWeights(txnState, accounts, signatures, tx.ProposalKey)
	if err != nil {
		return errors.NewInvalidProposalSignatureError(tx.ProposalKey, err)
	}

	// all authorizers must have sufficient weights
	for _, addr := range tx.Authorizers {
		// Skip this authorizer if it is also the payer. In the case where an account is
		// both a PAYER as well as an AUTHORIZER or PROPOSER, that account is required
		// to sign only the envelope.
		if addr == tx.Payer {
			continue
		}
		// hasSufficientKeyWeight
		if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
			return errors.NewAccountAuthorizationErrorf(
				addr,
				"authorizer account does not have sufficient signatures (%d < %d)",
				payloadWeights[addr],
				keyWeightThreshold)
		}
	}

	// payer must have sufficient weights
	if !v.hasSufficientKeyWeight(envelopeWeights, tx.Payer, keyWeightThreshold) {
		// TODO change this to payer error (needed for fees)
		return errors.NewAccountAuthorizationErrorf(
			tx.Payer,
			"payer account does not have sufficient signatures (%d < %d)",
			envelopeWeights[tx.Payer],
			keyWeightThreshold)
	}

	// Verify all cryptographic signatures against account public keys (concurrently)
	// and fail if at least one signature is invalid.
	// (at this point, signatures have been deduplicated and weights have been checked,
	// we wouldn't verify the signatures if any of those checks failed)
	err = v.verifySignatures(signatures)
	if err != nil {
		return errors.NewInvalidProposalSignatureError(tx.ProposalKey, err)
	}

	return nil
}
```
