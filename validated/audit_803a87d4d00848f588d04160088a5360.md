### Title
Unbounded Loop in `calculate_rewards` Over `pool_member_epoch_balance` Trace Causes Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

`DelegationPool.claim_rewards()` calls the internal `calculate_rewards()` function, which iterates over every entry in a pool member's `pool_member_epoch_balance` trace since the last claim. Because a pool member (unprivileged delegator) can grow this trace without bound by making balance changes across many epochs, the loop will eventually exceed the Starknet gas limit, permanently freezing the pool member's unclaimed yield. The code itself acknowledges this risk with a comment: *"This loop is unbounded but unlikely to exceed gas limits."*

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to `pool_member_trace_length`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each iteration also calls `find_sigma`, which performs additional storage reads, compounding the gas cost per iteration. [2](#0-1) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`. The `insert` implementation appends a **new** checkpoint whenever the key (epoch) differs from the last entry's key:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

So every call to `add_to_delegation_pool` (or any balance-modifying operation) in a **different epoch** appends one new entry to the trace. A pool member who adds to their delegation once per epoch, without claiming rewards, grows the trace by one entry per epoch indefinitely.

`claim_rewards` is the only function that advances `entry_to_claim_from`, so if a pool member delays claiming while accumulating many balance changes, the next `claim_rewards` call must iterate all accumulated entries in a single transaction: [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace grows large enough that iterating it in `calculate_rewards` exceeds the Starknet per-transaction gas limit, every call to `claim_rewards` for that pool member will revert. Because `entry_to_claim_from` is only advanced inside a successful `claim_rewards` call, there is no way to partially process the trace. The pool member's accumulated STRK rewards become permanently inaccessible.

---

### Likelihood Explanation

**Medium-to-High.** Any pool member who:
1. Makes balance changes (via `add_to_delegation_pool`) across many epochs, and
2. Does not claim rewards frequently enough to keep `entry_to_claim_from` close to the trace head,

will eventually hit this condition. This is a realistic pattern for active delegators who top up their stake regularly. The protocol imposes no upper bound on the trace length and no minimum claim frequency. The code comment at line 858 explicitly acknowledges the unbounded nature of the loop. [5](#0-4) 

---

### Recommendation

1. **Paginated claiming**: Add an `entry_count_limit` parameter to `claim_rewards` (or an internal variant) so the loop processes at most `N` entries per call, writing back the updated `entry_to_claim_from` even on a partial run. This mirrors the offset+length mitigation recommended in the original M-03 report.

2. **Trace compaction**: After a successful `claim_rewards`, compact already-processed entries from the trace so the length does not grow monotonically.

3. **Enforce a maximum balance-change frequency**: Reject `add_to_delegation_pool` calls that would create a new trace entry if the member already has an unprocessed entry from the same epoch window, limiting trace growth rate.

---

### Proof of Concept

**Setup:**
- Staker stakes with pool enabled.
- Delegator delegates a minimum amount.

**Attack / Trigger:**
```
for epoch in 1..N:
    advance_epoch()
    add_to_delegation_pool(delegator, pool, amount: 1)   // new trace entry each epoch
    // do NOT call claim_rewards

// After N epochs, attempt to claim:
claim_rewards(pool_member: delegator)   // iterates N entries × find_sigma cost → OOG revert
```

Each `add_to_delegation_pool` call in a new epoch invokes `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`, appending a new `PoolMemberBalanceCheckpoint`. [6](#0-5) 

After `N` such epochs, `pool_member_trace_length = N`. The `claim_rewards` call enters `calculate_rewards` with `entry_to_claim_from = 0` and must execute the loop body `N` times, each time calling `find_sigma` (storage reads). For sufficiently large `N`, the transaction exceeds the Starknet gas limit and reverts. Because `entry_to_claim_from` is never written on a revert, all subsequent `claim_rewards` calls also revert, permanently freezing the pool member's unclaimed STRK yield. [7](#0-6)

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(

                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Emit event.
            self

                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
