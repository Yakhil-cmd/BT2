Audit Report

## Title
Unsanitized Memo Bytes Enable Markdown Injection into ICRC-21 `GenericDisplayMessage`, Allowing Fake Field Spoofing — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` function in `responses.rs` interpolates user-controlled memo bytes directly into a single-backtick CommonMark code span without sanitizing backtick or newline characters. Because a single backtick in the memo closes the span early, an attacker-controlled dApp can inject arbitrary markdown — including fake bold field labels such as `**To:**` — into the ledger-generated `GenericDisplayMessage` returned by `icrc21_canister_call_consent_message`. This breaks the core security guarantee of ICRC-21, which is to provide wallets with a tamper-proof, ledger-generated description of what the user is actually signing, and can cause users to approve transfers to an attacker-controlled address while believing they are sending to an expected recipient.

## Finding Description
**Root cause — `add_memo`, `GenericDisplayMessage` branch** (`packages/icrc-ledger-types/src/icrc21/responses.rs`, L284–295):

```rust
pub fn add_memo(&mut self, memo: GenericMemo) {
    match memo {
        GenericMemo::Icrc1Memo(memo) => {
            let memo_str = match std::str::from_utf8(memo.as_slice()) {
                Ok(valid_str) => valid_str.to_string(),   // no sanitization
                Err(_) => hex::encode(memo.as_slice()),
            };
            match self {
                ConsentMessage::GenericDisplayMessage(message) => {
                    message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
                    // single-backtick span, breakable by any backtick in memo_str
```

In CommonMark, a code span opened with one backtick is terminated by the next single backtick in the string. If `memo_str` contains a backtick, the span closes early and everything after it is rendered as normal markdown.

**Exploit path:**

1. Any principal calls the `#[update]` endpoint `icrc21_canister_call_consent_message` (`rs/ledger_suite/icp/ledger/src/main.rs`, L1478–1481) — no access control.
2. The endpoint decodes `TransferArg` and passes the raw memo bytes as `GenericMemo::Icrc1Memo` (`packages/icrc-ledger-types/src/icrc21/lib.rs`, L392–395).
3. `ConsentMessageBuilder::build()` calls `message.add_memo(memo)` unconditionally if a memo is present (`lib.rs`, L290–292).
4. `add_memo` attempts UTF-8 decoding; if valid, embeds the string verbatim into the markdown template with no escaping.

**Concrete payload (22 bytes, within the 32-byte default limit):**

```
b"x`\n\n**To:**\n`attacker "
```

The format string produces:

```
\n\n**Memo:**\n`x`\n\n**To:**\n`attacker `
```

CommonMark parsing: `` `x` `` closes as a legitimate code span; `\n\n` creates a paragraph break; `**To:**` renders as a bold label visually identical to the legitimate field added by `add_account`; `` `attacker ` `` renders as a code span closed by the trailing backtick from the format string.

The legitimate `**To:**` field is added before the memo (`lib.rs`, L206), so the rendered consent message contains the real `**To:**` (attacker's address) followed by the injected fake `**To:**` (victim's expected address). A user scanning quickly for the "To:" label may fixate on whichever matches their expectation and approve the transaction.

**Existing guards:** None. There is no sanitization, escaping, or stripping of backtick or newline characters anywhere in the `add_memo` → `GenericDisplayMessage` path. The `FieldsDisplayMessage` branch is unaffected because it uses a structured `Value::Text` field rendered outside of markdown.

## Impact Explanation
ICRC-21 is a financial integration component explicitly in scope. Its security guarantee is that the ledger — not the dApp — generates a trustworthy description of what the user is signing. A malicious dApp that constructs a `TransferArg` with `to = attacker_address` and a crafted memo can cause the wallet's consent screen to display a fake `**To:**` field showing a victim-expected address, while the actual on-chain transfer goes to the attacker. The user approves what they believe is a legitimate transfer; funds are stolen. The same technique can inject a fake `**Amount:**` or `**Fees:**` field. This maps to **High** impact: significant ICRC ledger security impact with concrete user fund harm, constrained by the requirement for a malicious dApp and a wallet that renders `GenericDisplayMessage` as CommonMark markdown.

## Likelihood Explanation
- The attacker is any unprivileged principal who can call `icrc21_canister_call_consent_message` — no special role, key, or threshold corruption required.
- The attack requires only crafting a `TransferArg` with a memo containing a backtick, a trivial operation.
- The payload fits in the default 32-byte memo limit.
- Rendering `GenericDisplayMessage` as CommonMark is the explicit intent of the ICRC-21 standard, so any conforming wallet is a valid target.
- The attack is repeatable against any user of any dApp that uses the ICRC-1 transfer endpoint.

## Recommendation
Sanitize `memo_str` before interpolation in the `GenericDisplayMessage` branch. Minimal fix:

```rust
let safe_memo = memo_str
    .replace('`', "\\`")
    .replace('\n', " ")
    .replace('\r', "");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, hex-encode all memo bytes unconditionally for the `GenericDisplayMessage` branch (valid UTF-8 display is a convenience, not a requirement), or use a double-backtick span and strip any occurrence of ` `` ` from the memo.

## Proof of Concept

```rust
#[test]
fn memo_markdown_injection() {
    use icrc_ledger_types::icrc21::lib::GenericMemo;
    use icrc_ledger_types::icrc21::responses::ConsentMessage;
    use serde_bytes::ByteBuf;

    let mut msg = ConsentMessage::GenericDisplayMessage(String::new());

    // Simulate add_account("To", "real_attacker_address")
    msg.add_account("To", "real_attacker_address".to_string());

    // Craft memo: closes the backtick span, injects a fake **To:** field
    let malicious_memo = b"x`\n\n**To:**\n`victim_expected_address ";
    msg.add_memo(GenericMemo::Icrc1Memo(ByteBuf::from(malicious_memo.to_vec())));

    if let ConsentMessage::GenericDisplayMessage(s) = &msg {
        let to_count = s.matches("**To:**").count();
        assert_eq!(to_count, 2, "injected a fake To: field: {s}");
        assert!(s.contains("victim_expected_address"));
    }
}
```

This test exercises the exact code path at `responses.rs` L284–295 and will pass against the current unpatched code, confirming the injection.
