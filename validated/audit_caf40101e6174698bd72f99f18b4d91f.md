### Title
Gateway `skip_stateful_validations` admits invoke transactions with unverified signatures when a deploy-account is pending — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally bypasses the `__validate__` entry-point call (the only place where the account's signature is verified) for any Invoke transaction whose nonce is 1 and whose sender address has *any* transaction in the mempool or a recent block. Because a `DeployAccount` for address A is publicly visible in the mempool, an attacker can submit an Invoke for A with an arbitrary garbage signature, have it admitted without signature verification, and displace or corrupt the legitimate post-deploy invoke.

---

### Finding Description

**Trigger condition** — `skip_stateful_validations` returns `true` when all three hold:

1. The transaction is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 4 is satisfied by the legitimate `DeployAccount` that Alice already submitted, because `AccountTransaction::sender_address()` for a `DeployAccount` returns the newly computed `contract_address`, which is the same address the attacker targets. [2](#0-1) 

**Effect of `skip_validate = true`** — `run_validate_entry_point` sets `validate: !skip_validate = false` in `ExecutionFlags`, which causes `StatefulValidator::perform_validations` to return `Ok(())` immediately before the `__validate__` call: [3](#0-2) [4](#0-3) 

The `__validate__` entry point is the *only* place where the account's ECDSA/Stark signature is checked. Neither the stateless validator nor the mempool's `validate_tx` inspect the signature field. [5](#0-4) 

**Attack path**

1. Alice submits `DeployAccount(class_hash=C, salt=S, calldata=D)` → gateway validates it fully (runs `__validate_deploy__`), admits it to the mempool.
2. Attacker computes `A = calculate_contract_address(S, C, D, deployer=0)` — deterministic and public.
3. Attacker submits `Invoke(sender=A, nonce=1, calldata=arbitrary, signature=garbage)`.
4. Gateway stateful path: `account_nonce=0`, `tx.nonce()=1`, `account_tx_in_pool_or_recent_block(A)=true` → `skip_validate=true` → `__validate__` is **never called** → transaction admitted.
5. If the attacker's tip ≥ Alice's legitimate nonce-1 invoke, the mempool replaces Alice's invoke with the attacker's (fee-escalation path).
6. Batcher executes `DeployAccount(A)` → success; then executes `Invoke(A, nonce=1, garbage_sig)` → `__validate__` panics → transaction reverts; fee is charged from A's balance. [6](#0-5) 

The comment in the code claims the mempool presence check is "sufficient", but it is not: it proves only that *some* transaction from A is pending, not that the incoming invoke carries a valid signature.

---

### Impact Explanation

**Matches: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

- An Invoke with a completely fabricated signature is admitted to the mempool and forwarded to the batcher.
- The attacker can displace Alice's legitimate nonce-1 invoke via fee escalation, causing her first post-deploy action to be replaced by a failing transaction.
- The failing transaction is included in the block; the fee is deducted from Alice's account balance (the account was funded before the deploy-account to cover fees).
- The attack is repeatable for every new account deployment visible in the mempool.

---

### Likelihood Explanation

- The mempool is observable via P2P gossip or the RPC `starknet_getTransactionByHash` / pending-block endpoints; `DeployAccount` transactions are not secret.
- The target contract address is deterministically computable from public fields in the `DeployAccount` transaction.
- The attacker only needs to submit one transaction with a tip slightly above the victim's invoke tip; no privileged access is required.
- The window is bounded to nonce-1 invokes, but that is precisely the most sensitive transaction (first action of a newly deployed account).

---

### Recommendation

Replace the unconditional skip with a pre-validation that uses the class hash carried in the pending `DeployAccount` to run a *stateless* signature check against the transaction hash, without requiring the account to be deployed. Alternatively, restrict the skip to cases where the gateway can confirm the pending mempool entry is specifically a `DeployAccount` (not any transaction type), and add a signature-format check (e.g., length and curve-point validity) as a minimum gate even when full `__validate__` execution is deferred.

---

### Proof of Concept

```
# 1. Alice submits:
DeployAccount { class_hash: C, salt: S, constructor_calldata: D, nonce: 0, signature: valid_sig }
# → Gateway runs __validate_deploy__, admits to mempool.

# 2. Attacker computes:
A = calculate_contract_address(salt=S, class_hash=C, calldata=D, deployer=0)

# 3. Attacker submits (before or with higher tip than Alice's invoke):
Invoke { sender_address: A, nonce: 1, calldata: [steal_funds...], signature: [0x1, 0x2] }

# 4. Gateway stateful path:
#    account_nonce(A) = 0  ✓
#    tx.nonce()       = 1  ✓
#    account_tx_in_pool_or_recent_block(A) = true  ✓  (Alice's DeployAccount is there)
#    → skip_validate = true → __validate__ NOT called → ADMITTED

# 5. Batcher block N:
#    Execute DeployAccount(A)          → OK, A deployed
#    Execute Invoke(A, nonce=1, 0x1,0x2) → __validate__ fails → REVERTED, fee charged from A
#    Alice's legitimate Invoke(A, nonce=1) was displaced or never reached the batcher.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
