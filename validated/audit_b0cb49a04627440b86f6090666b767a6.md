### Title
Wrong Variable Stored in `PresignOutput`: Raw Nonce Share `k_i` Used Instead of Lagrange-Weighted `k_prime_i` - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

In `do_presign` within `src/ecdsa/ot_based_ecdsa/presign.rs`, the Lagrange-weighted nonce share `k_prime_i` is correctly computed and used during the presigning protocol, but the raw (unweighted) `k_i` is stored in the `PresignOutput` struct. The signing phase consumes `k` from `PresignOutput` directly; because `k_i ≠ k_prime_i`, the resulting signature shares will not reconstruct a valid ECDSA signature, permanently corrupting the signing output for every presignature produced.

---

### Finding Description

In `do_presign` (lines 64–185 of `src/ecdsa/ot_based_ecdsa/presign.rs`), the protocol performs the following sequence:

**Step 1 — Lagrange linearization (line 93–95):**
```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;   // Lagrange-weighted nonce share
```

**Step 2 — `k_prime_i` is used in the protocol broadcast (line 136):**
```rust
let alpha_i: Scalar = k_prime_i + a_prime_i;
```
`alpha_i` is broadcast to all parties and summed into `alpha`, which is then used to compute `sigma_i`:
```rust
let sigma_i = alpha * private_share - (beta * a_i - c_i);
```
`sigma_i` is therefore mathematically consistent with the Lagrange-weighted `k_prime_i`.

**Step 3 — Wrong variable stored in output (lines 181–185):**
```rust
Ok(PresignOutput {
    big_r,
    k: k_i,       // ← BUG: should be k_prime_i
    sigma: sigma_i,
})
```

The raw `k_i` is stored instead of `k_prime_i`. In the Cait-Sith signing equation, each party computes:
```
s_i = k_i * (hash(m) + r * sigma_i)
```
and the final signature is `s = Σ s_i`. For this to equal the correct `k * (hash(m) + r * sigma)`, the stored `k` must be the Lagrange-weighted share `k_prime_i`. Storing the raw `k_i` means `Σ k_i ≠ k`, so the reconstructed `s` is cryptographically invalid.

Because `sigma_i` was computed using `alpha = Σ(k_prime_j + a_prime_j)` (Lagrange-weighted), the two outputs `k` and `sigma` in `PresignOutput` are now inconsistent with each other: `sigma` is calibrated for Lagrange-weighted `k_prime_i`, but `k` holds the raw `k_i`. No downstream fix in the signing phase can reconcile this mismatch without access to the original Lagrange coefficient, which is not stored in `PresignOutput`.

---

### Impact Explanation

Every presignature produced by `do_presign` carries a structurally incorrect `k` field. When the signing phase consumes this presignature, it will produce signature shares that do not sum to a valid ECDSA signature. Honest parties will accept the output of the signing protocol (no verification failure occurs at the share-combination step), but the resulting `(r, s)` pair will fail standard ECDSA verification against the public key. This matches the allowed High impact:

> **High: Corruption of presign outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

All signing operations that use any presignature generated by this code path are permanently broken. There is no recovery path without regenerating presignatures with a corrected implementation.

---

### Likelihood Explanation

This bug is triggered on every single invocation of `do_presign` with more than one participant (i.e., whenever `lambda_me ≠ 1`, which is the normal case for any threshold > 1). No attacker action is required; the corruption is deterministic and unconditional. Any library caller invoking the OT-based ECDSA presigning flow will produce unusable presignatures.

---

### Recommendation

Replace `k_i` with `k_prime_i` in the `PresignOutput` construction:

```diff
- Ok(PresignOutput {
-     big_r,
-     k: k_i,
-     sigma: sigma_i,
- })
+ Ok(PresignOutput {
+     big_r,
+     k: k_prime_i,
+     sigma: sigma_i,
+ })
```

This ensures the stored nonce share is Lagrange-weighted and consistent with `sigma_i`, matching the Cait-Sith protocol specification.

---

### Proof of Concept

Consider a 2-of-3 threshold setup. Participant `me` has Lagrange coefficient `lambda_me = 3` (a typical non-unity value). Then:
- `k_i = 5` (raw nonce share from triple)
- `k_prime_i = 3 * 5 = 15` (correct Lagrange-weighted share)

The protocol broadcasts `alpha_i = k_prime_i + a_prime_i = 15 + ...`, so `sigma_i` is computed relative to `k_prime_i = 15`.

But `PresignOutput.k = k_i = 5` is stored.

In signing: `s_i = 5 * (hash(m) + r * sigma_i)`. The other parties similarly store their raw `k_j`. The sum `Σ s_i = Σ(k_j) * (hash(m) + r * sigma)` where `Σ k_j = Σ(raw shares)` ≠ `Σ(k_prime_j) = k`. The resulting `s` fails ECDSA verification. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
