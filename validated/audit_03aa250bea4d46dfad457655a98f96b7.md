### Title
Unprivileged Caller Can Front-Run `update_rewards` With `disable_rewards: true`, Permanently Freezing Block Rewards - (File: src/staking/staking.cairo)

### Summary
`update_rewards` in `src/staking/staking.cairo` is a public function with no access control. It accepts a `disable_rewards` boolean parameter and unconditionally writes `last_reward_block` to the current block number before checking that flag. Any unprivileged caller can invoke `update_rewards(any_valid_staker, disable_rewards: true)` to consume the per-block reward slot without distributing any rewards, permanently blocking the legitimate staker from receiving consensus block rewards for that block.

### Finding Description
`update_rewards` is gated only by `general_prerequisites`, which checks that the contract is not paused and the caller is not the zero address. [1](#0-0) 

The function writes `last_reward_block` to the current block number unconditionally, before the `disable_rewards` branch: [2](#0-1) 

`last_reward_block` is a single global storage slot, not per-staker: [3](#0-2) 

The guard at the top of `update_rewards` enforces that only one call per block can succeed across the entire contract: [4](#0-3) 

Because `last_reward_block` is written before the `disable_rewards` check, a malicious caller who supplies `disable_rewards: true` consumes the slot and returns without distributing rewards: [5](#0-4) 

Any subsequent legitimate call to `update_rewards` in the same block fails with `REWARDS_ALREADY_UPDATED`. The rewards for that block are permanently lost — there is no mechanism to retroactively distribute them.

The attacker only needs to supply any currently active staker address with non-zero STRK balance to pass the validity checks: [6](#0-5) 

### Impact Explanation
This is a **permanent freezing of unclaimed yield**. Consensus block rewards are computed per-block and distributed once per block via `update_rewards`. If the slot is consumed with `disable_rewards: true`, the rewards for that block are never credited to any staker and cannot be recovered. An attacker who calls this every block prevents the entire consensus reward system from ever distributing rewards to any staker.

### Likelihood Explanation
The attack requires no privileged role, no capital, and no special setup — only a valid active staker address (publicly observable on-chain) and enough gas to call `update_rewards` before the legitimate caller. The attacker has no profit motive but can cause continuous, compounding damage to all stakers and the protocol.

### Recommendation
Add an access control check to `update_rewards` so that only the staker themselves (or an authorized party such as the attestation contract) can call it for a given `staker_address`. Alternatively, move the `self.last_reward_block.write(current_block_number)` write to after the `disable_rewards` guard so that a call with `disable_rewards: true` does not consume the per-block slot.

### Proof of Concept
1. Staker `S` is active with non-zero STRK balance. Block `N` arrives.
2. Attacker calls `update_rewards(S, disable_rewards: true)` at block `N`.
3. The function passes all validity checks, writes `last_reward_block = N`, then returns early without distributing rewards.
4. The legitimate call `update_rewards(S, disable_rewards: false)` at block `N` reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` permanently loses block `N` rewards.
6. Repeating this every block freezes all consensus rewards indefinitely. [7](#0-6)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
