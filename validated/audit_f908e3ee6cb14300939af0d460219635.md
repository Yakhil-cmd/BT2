### Title
Partial Token Return in `ft_transfer_call` Exit Path Not Refunded to User — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

When a user exits ERC-20 tokens to NEAR using the Omni message format (triggering `ft_transfer_call` on the NEP-141 contract), the `exit_to_near_precompile_callback` only checks whether the promise succeeded or failed — it never inspects the actual amount of tokens used by the receiver. If the receiver's `ft_on_transfer` returns a non-zero unused amount, those NEP-141 tokens are credited back to Aurora's NEP-141 balance, but the corresponding ERC-20 tokens are never re-minted for the user. The user permanently loses the returned portion of their tokens.

### Finding Description

The `ExitToNear` precompile supports an Omni message format that routes the exit through `ft_transfer_call` instead of `ft_transfer`. In `exit_erc20_token_to_near`, when `exit_params.message` is `Some(Message::Omni(msg))`, the method chosen is `"ft_transfer_call"`: [1](#0-0) 

The NEP-141 `ft_transfer_call` standard works as follows:
1. Transfers `amount` tokens from Aurora to the receiver.
2. Calls `receiver.ft_on_transfer(aurora, amount, msg)`.
3. In `ft_resolve_transfer`, transfers back any unused tokens to Aurora.
4. Returns the amount **used** (not returned) as the promise result.

The callback `exit_to_near_precompile_callback` is attached to this promise: [2](#0-1) 

The callback only branches on `PromiseResult::Successful(_)` vs failure. It **ignores the bytes inside the successful result** — which encode how many tokens were actually used. If the receiver returns `Y` tokens (where `0 < Y ≤ X`), `ft_transfer_call` still resolves as `Successful` (with value `X - Y`), and the callback takes the success branch and does nothing.

The `refund_call_args` function, which populates the `RefundCallArgs` for the callback, always encodes the **full original amount** and is only triggered on complete promise failure: [3](#0-2) 

The `refund_on_error` function, which re-mints ERC-20 tokens, is therefore never called for partial returns: [4](#0-3) 

The result: the user burned `X` ERC-20 tokens, the receiver received `X - Y` NEP-141 tokens, and `Y` NEP-141 tokens are silently credited back to Aurora's NEP-141 balance with no corresponding ERC-20 tokens ever re-minted. Those `Y` tokens are permanently inaccessible.

### Impact Explanation

**Critical — Permanent freezing of user funds.**

The `Y` NEP-141 tokens returned to Aurora's balance cannot be claimed by anyone:
- There are no ERC-20 tokens to burn in exchange for them (the user's ERC-20 tokens were already burned for the full `X`).
- No other mechanism exists to withdraw NEP-141 tokens from Aurora's balance without burning the corresponding ERC-20 tokens.

The user loses `Y` tokens permanently. Aurora's NEP-141 balance becomes inflated relative to the ERC-20 supply, creating a bridge insolvency condition proportional to the accumulated returned amounts.

### Likelihood Explanation

**Medium.** The Omni message format is a documented, user-accessible feature of the `ExitToNear` precompile. Any receiver contract that implements partial usage (e.g., a DEX that can only fill part of an order, or any contract that conditionally rejects some tokens) will trigger this path. The user does not need to be malicious — this occurs in normal operation whenever a legitimate receiver returns unused tokens. The `ft_transfer_call` standard explicitly supports partial returns, making this a realistic scenario.

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `Successful`, decode the returned bytes to determine the amount of tokens actually used. If the used amount is less than the original exit amount, call `refund_on_error` (or an equivalent re-mint) for the difference (`original_amount - used_amount`). The `RefundCallArgs` struct already carries all necessary information (`erc20_address`, `recipient_address`, `amount`) to perform this partial re-mint.

### Proof of Concept

1. User holds `X = 1000` units of ERC-20 token `T` on Aurora (backed by NEP-141 `T.near`).
2. User calls `T.burn(1000, exit_to_near_precompile, omni_msg)` where `omni_msg` routes to a receiver contract `R.near` via `ft_transfer_call`.
3. ERC-20 burns 1000 tokens from user; `ExitToNear` precompile fires `ft_transfer_call(R.near, 1000, msg)` on `T.near`.
4. `R.near.ft_on_transfer` processes 600 tokens and returns `400` as unused.
5. `ft_resolve_transfer` transfers 400 tokens back to Aurora's `T.near` balance.
6. `ft_transfer_call` resolves `Successful` with value `600` (used amount).
7. `exit_to_near_precompile_callback` sees `Successful`, enters the success branch at line 214, does nothing.
8. Final state: User has 0 ERC-20 tokens. `R.near` has 600 NEP-141 tokens. Aurora has 400 extra NEP-141 tokens with no corresponding ERC-20 supply. User permanently lost 400 tokens. [5](#0-4) [6](#0-5)

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
