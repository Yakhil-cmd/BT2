Based on my investigation of the production code, this is a **real vulnerability**.

## Key Findings

### 1. No Caller Check in `update_rewards`

The spec at `docs/spec.md` line 1645 states access control is **"Only starkware sequencer"**, but the implementation contains no such check: [1](#0-0) 

`general_prerequisites()` only checks the pause flag — confirmed by the pause tests which show all functions revert with `"Contract is paused"` but no caller-identity check is present in `update_rewards`. Unit tests call `update_rewards` directly without any caller setup and expect only `REWARDS_ALREADY_UPDATED` on re-entry, not an access control error. [2](#0-1) 

### 2. `last_reward_block` is a Global Gate (Not Per-Staker)

The guard that prevents double-updates is a **single global** `last_reward_block`, not a per-staker value: [3](#0-2) 

Once any caller writes `last_reward_block = current_block_number`, **no staker** can receive rewards in that block.

### 3. `disable_rewards: true` Skips Reward Distribution

After writing `last_reward_block`, the function returns early with no rewards distributed: [4](#0-3) 

---

### Title
Unprivileged caller can suppress all consensus block rewards by calling `update_rewards` with `disable_rewards: true` — (`src/staking/staking.cairo`)

### Summary
`update_rewards` is callable by any address despite the spec requiring "Only starkware sequencer." The global `last_reward_block` gate means a single call per block with `disable_rewards: true` permanently discards that block's rewards for all stakers.

### Finding Description
`StakingRewardsManagerImpl::update_rewards` performs no `get_caller_address()` check. Any EOA or contract can call it each block with an arbitrary valid `staker_address` and `disable_rewards: true`. The function writes `last_reward_block = current_block_number` before the early-return branch, so the sequencer's legitimate call later in the same block reverts with `REWARDS_ALREADY_UPDATED`. The block's rewards are permanently lost.

### Impact Explanation
An attacker submitting one transaction per block can zero out all consensus rewards for all stakers indefinitely. This is permanent freezing of unclaimed yield — **High** severity under the allowed impact scope.

### Likelihood Explanation
Starknet L2 transaction fees are low. Submitting one griefing transaction per block is economically viable. No privileged access, no leaked keys, no external dependencies required.

### Recommendation
Add a caller check inside `update_rewards`:
```rust
let caller = get_caller_address();
assert!(caller == self.sequencer_address.read(), "{}", Error::ONLY_SEQUENCER);
```
The sequencer address should be stored at construction time or read from a trusted role registry, matching the spec's stated access control.

### Proof of Concept
1. Deploy with two active stakers, consensus rewards active.
2. Each block, attacker calls `update_rewards(staker_A, disable_rewards: true)` before the sequencer.
3. Sequencer's call reverts with `REWARDS_ALREADY_UPDATED`.
4. After N blocks, both stakers' `unclaimed_rewards_own` remain zero; expected cumulative rewards are N × block_reward. [5](#0-4) [6](#0-5)

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,

            let curr_epoch = self.get_current_epoch();
            assert!(

                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self

            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
