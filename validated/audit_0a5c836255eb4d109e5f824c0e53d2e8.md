### Title
`DelegateAction` (meta-transaction) signatures lack chain-ID binding, enabling cross-chain replay — (File: `core/primitives/src/action/delegate.rs`)

---

### Summary

`DelegateAction` and `DelegateActionV2` (NEP-366 / NEP-611 meta-transactions) are signed over a payload that contains no chain identifier, genesis hash, or network-specific context. A signature produced for NEAR mainnet is cryptographically valid on NEAR testnet or any NEAR hard-fork that shares the same account state, allowing an attacker to replay the signed authorization on a different network and execute the inner actions on behalf of the victim.

---

### Finding Description

`DelegateAction.get_nep461_hash()` constructs the signed digest as:

```
SHA-256( borsh( MessageDiscriminant(NEP-366) || DelegateAction ) )
``` [1](#0-0) 

`DelegateAction` contains only `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`. [2](#0-1) 

The `MessageDiscriminant` is a fixed NEP number (366 or 611). It distinguishes message *types* but carries no chain-specific information. [3](#0-2) 

By contrast, every regular `SignedTransaction` includes a `block_hash` field — a recent block hash that is chain-specific and validated by `check_transaction_validity_period` — which implicitly binds the transaction to one chain. [4](#0-3) 

`DelegateAction` has no equivalent binding. The only temporal guard is `max_block_height`, but block heights are independent counters on every NEAR network and do not identify a chain.

The same absence applies to `DelegateActionV2` / `VersionedDelegateActionPayload.get_nep461_hash()`: [5](#0-4) 

---

### Impact Explanation

An attacker who obtains a `SignedDelegateAction` intended for mainnet can submit it on testnet (or any NEAR fork) if:

1. The victim's account exists on the target network (common for developers; identical on hard-forks).
2. The same access key exists on the target network.
3. The nonce has not yet been consumed on the target network.
4. `max_block_height` has not been exceeded on the target network.

On success, the inner `actions` execute with `sender_id` as the predecessor — the victim's identity. This can authorize arbitrary transfers, `AddKey`/`DeleteKey`, `FunctionCall`, or `DeleteAccount` on the target chain without the victim's knowledge or consent.

**Impact: High** — full impersonation of the victim for any action the `DelegateAction` encodes, on any NEAR network sharing the same account/key state.

---

### Likelihood Explanation

**Realistic scenarios:**

- **Mainnet ↔ Testnet replay**: NEAR account names are identical across networks. Developers routinely hold the same named account with the same key pair on both. A relayer that receives a `SignedDelegateAction` for mainnet can trivially submit it on testnet.
- **Hard-fork replay**: At the fork block, every account, access key, and nonce is identical on both chains. Any `DelegateAction` signed before the fork is valid on both post-fork chains until the nonce is consumed on each independently.

The attacker needs only to intercept or observe the `SignedDelegateAction` (it is transmitted off-chain to a relayer and then on-chain in a transaction body, so it is publicly visible in the mempool and on-chain).

---

### Recommendation

Include a chain-specific identifier in the signed payload of `DelegateAction` and `DelegateActionV2`. The canonical choice is the genesis block hash (`GenesisId`), which is already used for peer-handshake chain discrimination in the network layer. Concretely:

- Add a `chain_id: CryptoHash` (genesis hash) field to `DelegateAction` and `DelegateActionV2`, or
- Incorporate the genesis hash into `SignableMessage` so that all NEP-461-tagged messages are automatically chain-bound.

This mirrors the fix applied in the external report (including `block.chainid` and the contract address in the signed hash).

---

### Proof of Concept

1. Alice holds account `alice.near` on both mainnet and testnet with the same ED25519 key pair and nonce `N` on both.
2. Alice signs a `DelegateAction` for mainnet: transfer 10 NEAR to `bob.near`, nonce `N+1`, `max_block_height = 200_000_000`.
3. The relayer (or an observer who copies the on-chain transaction) extracts the `SignedDelegateAction`.
4. The attacker wraps it in a new outer transaction on testnet and submits it.
5. The runtime on testnet calls `SignedDelegateAction::verify()`: [6](#0-5) 

6. `get_nep461_hash()` recomputes the same digest (no chain context) — verification passes.
7. The nonce check passes (`N+1 > N`).
8. The transfer executes on testnet, draining Alice's testnet balance without her consent.

The corrupted value is Alice's testnet account balance (debited) and nonce (advanced), caused entirely by a valid signature that was never intended for testnet.

    /// different messages before hashing
    ///
    /// For more details, see: [NEP-461](https://github.com/near/NEPs/pull/461)
    pub fn get_nep461_hash(&self) -> CryptoHash {
        let signable = SignableMessage::new(&self, SignableMessageType::DelegateActionV2);
        let bytes = borsh::to_vec(&signable).expect("failed to serialize");
        hash(&bytes)
    }
