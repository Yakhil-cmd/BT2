The code evidence is conclusive. Let me confirm one more detail about the `EvmErc20.sol` burn-before-call ordering and the absence of any `ExitToEthereum`-side callback in the engine.

### Title
No-Callback `PromiseArgs::Create` in `ExitToEthereum` ERC-20 Path Causes Permanent Token Loss on NEP-141 Withdraw Failure — (`engine-precompiles/src/native.rs`)

---

### Summary

`ExitToEthereum::run` unconditionally emits `PromiseArgs::Create` for the NEP-141 `withdraw` promise with no error-handling callback. `EvmErc20.withdrawToEthereum` burns the caller's ERC-20 tokens **before** invoking the precompile and does not check the assembly call's return value. If the downstream NEP-141 `withdraw` promise fails (e.g., eth-connector paused, insufficient engine balance), the burned ERC-20 tokens are permanently destroyed with no refund path.

---

### Finding Description

**Step 1 — Burn before precompile call, no return-value check.**

`EvmErc20.withdrawToEthereum` burns tokens at line 66, then calls the precompile at line 74 inside an `assembly` block whose `res` value is never inspected:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6..., 0, ...);  // res never checked
    }
}
``` [1](#0-0) 

**Step 2 — Precompile always emits `PromiseArgs::Create`, never a callback.**

Inside `ExitToEthereum::run`, regardless of whether `flag=0x0` (ETH) or `flag=0x1` (ERC-20), the promise is hardcoded as `PromiseArgs::Create`:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

There is no conditional branch, no `PromiseArgs::Callback`, and no `exit_to_ethereum_precompile_callback` method anywhere in the engine.

**Step 3 — Contrast with `ExitToNear`, which has a refund path.**

`ExitToNear::run` conditionally wraps the transfer promise in a `PromiseArgs::Callback` that calls `exit_to_near_precompile_callback`, which in turn calls `refund_on_error` to re-mint burned ERC-20 tokens on failure: [3](#0-2) [4](#0-3) 

`ExitToEthereum` has no equivalent. The `refund_on_error` function that re-mints ERC-20 tokens is only reachable via the `ExitToNear` callback path: [5](#0-4) 

**Step 4 — The NEP-141 `withdraw` can realistically fail.**

The eth-connector exposes a `pa_pause_feature` admin call with key `"engine_withdraw"` that pauses the `withdraw` method. This is a documented, production-accessible feature: [6](#0-5) 

When paused, any `withdraw` promise scheduled by `ExitToEthereum` will fail on the NEAR side. The EVM-side burn is already committed and irreversible.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToEthereum` on a bridged ERC-20 while the eth-connector's `engine_withdraw` feature is paused (or while the engine's NEP-141 balance is insufficient) will have their ERC-20 tokens permanently burned with no ETH minted on Ethereum and no refund issued. The NEP-141 tokens remain locked in the engine contract with no recovery path accessible to the user.

---

### Likelihood Explanation

The eth-connector pause is a live, admin-accessible production feature. A window between a user's EVM transaction and the NEAR promise execution is sufficient. Additionally, any transient NEP-141 balance shortfall or contract-level rejection of the `withdraw` call triggers the same outcome. The attacker does not need to be privileged — any user calling `withdrawToEthereum` during a pause window is affected.

---

### Recommendation

Mirror the `ExitToNear` refund pattern in `ExitToEthereum`:

1. Add an `exit_to_ethereum_precompile_callback` method to the engine that checks the promise result and, on failure, re-mints the burned ERC-20 tokens via `refund_on_error`.
2. Change `ExitToEthereum::run` to emit `PromiseArgs::Callback(PromiseWithCallbackArgs { base: withdraw_promise, callback: <exit_to_ethereum_precompile_callback> })` instead of the bare `PromiseArgs::Create`.
3. Pass the ERC-20 address, refund address, and amount into the callback args (analogous to `RefundCallArgs`).

---

### Proof of Concept

```
1. Deploy a NEP-141 token; bridge it to Aurora (creates EvmErc20 mirror).
2. Mint ERC-20 tokens to address A on Aurora.
3. Admin calls pa_pause_feature({"key": "engine_withdraw"}) on the eth-connector.
4. Address A calls EvmErc20.withdrawToEthereum(recipientEthAddr, amount):
   a. _burn(A, amount) executes — ERC-20 balance of A decreases by `amount`.
   b. ExitToEthereum precompile is called with flag=0x1.
   c. Precompile emits PromiseArgs::Create targeting NEP-141.withdraw.
   d. EVM transaction succeeds; burn is committed.
5. NEAR executes the scheduled withdraw promise → fails ("Pausable: Method is paused").
6. No callback fires; no refund is issued.
7. Assert: ERC-20 balance of A = original - amount (tokens gone).
8. Assert: No ETH minted at recipientEthAddr on Ethereum side.
9. Assert: NEP-141 balance of engine contract unchanged (tokens locked).
```

The invariant broken: **ERC-20 burn ↔ NEP-141 unlock** — the burn is committed but the unlock never occurs, and no compensating re-mint is possible.
