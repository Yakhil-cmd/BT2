### Title
Missing Caller Authorization in `update_rewards` Allows Griefing of Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is specified to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged address can call it with `disable_rewards: true`, consuming the single per-block reward slot and preventing legitimate reward distribution for that block.

### Finding Description
The protocol specification explicitly states the access control for `update_rewards`:

> **access control**: Only starkware sequencer. [1](#0-0) 

However, the implementation only calls `general_prerequisites()`, which checks that the contract is not paused and the caller is not the zero address — it does **not** verify the caller is the sequencer: [2](#0-1) 

The function unconditionally writes `current_block_number` to `last_reward_block` **before** the `disable_rewards` guard: [3](#0-2) 

Because the contract enforces at most one call per block via the `REWARDS_ALREADY_UPDATED` assertion, any caller who invokes `update_rewards(staker_address, disable_rewards: true)` first in a block will:
1. Advance `last_reward_block` to the current block.
2. Return early without distributing any rewards.
3. Cause every subsequent call in the same block — including the sequencer's — to revert with `REWARDS_ALREADY_UPDATED`.

The `general_prerequisites` helper confirms no sequencer check exists: [4](#0-3) 

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users or protocol.**

Stakers and pool members lose block rewards for every block in which the attacker successfully occupies the reward slot before the sequencer. Because `last_reward_block` is a single contract-wide value, one call with `disable_rewards: true` for any valid staker blocks reward distribution for all stakers in that block. [5](#0-4) 

### Likelihood Explanation
On Starknet the sequencer controls transaction ordering, so reliable front-running is not guaranteed. However, the missing guard still creates a reachable griefing path:

- In any block where the sequencer has not yet submitted its `update_rewards` transaction, an attacker's transaction can occupy the slot.
- The attacker requires no special role, no funds beyond gas, and no privileged access — only a valid `staker_address` (publicly readable from the `stakers` vector). [6](#0-5) 

### Recommendation
Add an explicit caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the authorized sequencer address (stored in contract configuration), mirroring the pattern used for other privileged entry points such as `update_rewards_from_attestation_contract`: [7](#0-6) 

### Proof of Concept
1. Attacker reads any active `staker_address` from the public `stakers` vector.
2. In block N (before the sequencer's transaction is included), attacker calls:
   `update_rewards(staker_address, disable_rewards: true)`
3. `last_reward_block` is written to N; the function returns early — no rewards distributed.
4. Sequencer's `update_rewards` call in block N reverts: `REWARDS_ALREADY_UPDATED`.
5. All stakers and pool members receive zero block rewards for block N.
6. Attacker repeats each block at negligible cost to suppress rewards indefinitely. [8](#0-7)

```text
        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
```
