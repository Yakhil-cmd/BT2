### Title
Cross-COA Signature Replay in `EVM.validateCOAOwnershipProof` — (`File: fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies that a Cadence account's key signatures are valid over caller-supplied `signedData`, then checks that the COA resource at the supplied `path` matches the supplied `evmAddress`. Because the function never enforces that `signedData` encodes `evmAddress`, a signature produced for one COA can be replayed against a different COA owned by the same Cadence account. The code itself documents this gap. Any unprivileged Cadence script or transaction can call the function directly with attacker-chosen arguments, making the replay reachable without any privileged access.

### Finding Description
`EVM.validateCOAOwnershipProof` is an `access(all)` function in the EVM system contract. Its logic is:

1. Verify that the supplied `signatures` (over `signedData`) satisfy the key-weight threshold for the Cadence account at `address`.
2. Borrow the `CadenceOwnedAccount` resource from `path` and confirm its EVM address equals `evmAddress`.

The function never checks that `signedData` encodes `evmAddress`. The code comment at lines 1001–1009 explicitly acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The signature verification step uses `Crypto.KeyList.verify` over the raw `signedData` with domain tag `"FLOW-V0.0-user"`: [2](#0-1) 

The EVM-address binding check is performed only after signature verification succeeds, and it only checks that the COA resource at `path` has the right address — it does not verify that the signer intended to authorize *that specific* COA: [3](#0-2) 

### Impact Explanation
If a Cadence account holds two COAs published at different public paths (e.g., `/public/coa1` and `/public/coa2`), a valid signature over some `signedData` that was produced for COA_1 can be replayed by any caller to make `validateCOAOwnershipProof` return `isValid: true` for COA_2. Any on-chain Cadence contract that gates asset transfers or capability grants on the result of `EVM.validateCOAOwnershipProof` called directly (not via the EVM precompile) is vulnerable to this cross-COA authorization bypass. The ERC-1271 path through `coa.sol`'s `isValidSignature` is not affected because the precompile always passes `address(this)` as `evmAddress`. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
The function is `access(all)` and callable from any unprivileged Cadence script or transaction with fully attacker-controlled arguments. The attacker needs only to observe a valid signature over some `signedData` for the target Cadence account (e.g., from a prior on-chain interaction), and the target account must have more than one COA published at different paths. The code comment acknowledges that wallets historically create at most one COA per account, which limits current exposure, but the protocol itself imposes no such restriction and multi-COA accounts are a supported configuration.

### Recommendation
Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` before accepting the proof as valid. Concretely, the function should verify that the 20-byte `evmAddress` is present within `signedData` (or that `signedData` is a deterministic commitment to `evmAddress`), rejecting proofs where this binding is absent. This mirrors the fix pattern for the analogous Solidity vulnerability: bind the signed message to the specific context (COA address) rather than relying on callers to do so.

### Proof of Concept
1. Cadence account `A` owns two COAs: `COA_1` at `/public/coa1` (EVM address `0xAAA…`) and `COA_2` at `/public/coa2` (EVM address `0xBBB…`).
2. Account `A` signs `signedData = <some_bytes>` for a legitimate interaction with `COA_1`. The signature is observed on-chain.
3. An attacker submits a Cadence script:
```cadence
import EVM from 0x…
access(all) fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: <account_A>,
        path: /public/coa2,          // COA_2, not COA_1
        signedData: <same_bytes>,    // original signedData, not encoding 0xBBB…
        keyIndices: <original_key_indices>,
        signatures: <original_signatures>,
        evmAddress: <0xBBB…>         // COA_2's EVM address
    )
}
```
4. The function verifies the signatures (valid, because they were produced by account `A`'s keys over `signedData`), then checks that the COA at `/public/coa2` has EVM address `0xBBB…` (true), and returns `isValid: true` — even though account `A` never signed anything authorizing `COA_2`.
