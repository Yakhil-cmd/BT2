### Title
Missing Proof of Knowledge for Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD protocol's coordinator blindly sums participant contributions `(norm_big_y, norm_big_c)` without any proof of knowledge or commitment verification. A single malicious participant can send arbitrary group elements, silently corrupting the final `CKDOutput` and causing the coordinator to derive a wrong confidential key — with no error raised.

### Finding Description
In `do_ckd_coordinator`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and adds them together unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute in `compute_signature_share`:
- `big_y = y * G` (randomness commitment)
- `big_s = hash_point * private_share` (BLS signature share)
- `big_c = big_s + app_pk * y` (masked share)
- `norm_big_y = lambda_i * big_y`, `norm_big_c = lambda_i * big_c` [2](#0-1) 

No proof of knowledge is generated or attached to the contribution. The coordinator has no way to verify that:
1. `norm_big_y` is a valid Lagrange-weighted commitment to a random nonce `y`
2. `norm_big_c` was correctly computed as `lambda_i * (big_s + app_pk * y)` using the participant's actual private share

This is the direct analog of the `sellProfits()` vulnerability: just as `amountOutMinimum: 0` allows an attacker to manipulate the swap output without bound, the absence of a proof of knowledge here allows a malicious participant to inject an arbitrary additive offset into the CKD output without detection.

Critically, unlike the robust ECDSA signing protocol which verifies the final signature before returning it: [3](#0-2) 

the CKD coordinator performs **no final correctness check** on the derived output before returning it: [4](#0-3)

### Impact Explanation
A malicious participant sends arbitrary `(norm_big_y_evil, norm_big_c_evil)` to the coordinator. The coordinator computes:
- `sum_big_c = honest_sum_big_c + norm_big_c_evil`
- `sum_big_y = honest_sum_big_y + norm_big_y_evil`

When the TEE unmasks via `sum_big_c - app_sk * sum_big_y`, it derives `honest_key + (norm_big_c_evil - app_sk * norm_big_y_evil)` — a wrong key that does not equal `msk * H(pk, app_id)`. The coordinator silently accepts this corrupted output. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation
Any single malicious participant (one of the threshold signers) can execute this attack. The attacker only needs to send a modified `(norm_big_y, norm_big_c)` pair to the coordinator instead of the correctly computed values. No special cryptographic knowledge is required. The attack is completely undetectable by the coordinator because there is no verification step.

### Recommendation
Add a zero-knowledge proof of knowledge to each participant's contribution. Each participant should prove (using a Sigma protocol or DLEQ proof) that their `(norm_big_y, norm_big_c)` was correctly computed from their private share and a random nonce. The coordinator should verify these proofs before summing the contributions. The existing `dlog::prove_with_nonce` / `dlog::verify` and `dlogeq::prove_with_nonce` / `dlogeq::verify` primitives in `src/crypto/proofs/` are directly applicable for this purpose. [5](#0-4) 

### Proof of Concept
1. Honest participants `P_1, ..., P_{n-1}` and malicious participant `P_m` all call `ckd()`.
2. `P_m` computes the correct values internally but sends `(identity_G1, arbitrary_point)` to the coordinator instead.
3. The coordinator sums all contributions: `sum_big_c = honest_sum_big_c + arbitrary_point`.
4. The coordinator returns `CKDOutput(sum_big_y, sum_big_c)` with no error.
5. The TEE unmasks: `sum_big_c - app_sk * sum_big_y = honest_key + arbitrary_point` — a wrong key.
6. The coordinator silently accepts the wrong key; no protocol error is raised.

pub fn prove_with_nonce<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    witness: Witness<C>,
    k: Scalar<C>,
) -> Result<Proof<C>, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (big_k_0, big_k_1) = statement.phi(&k);

    // This will never raise error as k is not zero and generator1 is not the identity
    let enc = encode_two_points::<C>(&big_k_0, &big_k_1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    let s = k + e * witness.x.0;
    Ok(Proof {
        e: SerializableScalar::<C>(e),
        s: SerializableScalar::<C>(s),
    })
}
