### Title
Unprivileged Caller Can Permanently Suppress Per-Block Rewards via `update_rewards` with `disable_rewards: true` — (File: `src/staking/staking.cairo`)

### Summary
`update_rewards` has no caller access control despite the spec stating "Only starkware sequencer." The global `last_reward_block` gate is consumed on every call regardless of the `disable_rewards` flag, allowing any address to permanently discard an entire block's rewards for all stakers by calling with `disable_rewards: true` first.

### Finding Description

The implementation at `StakingRewardsManagerImpl::update_rewards` performs no caller identity check: [1](#0-0) 

The only guard is the global `last_reward_block` check: [2](#0-1) 

After passing that check, `last_reward_block` is unconditionally written to the current block number **before** the `disable_rewards` branch: [3](#0-2) 

When `disable_rewards: true`, the function returns immediately after writing `last_reward_block`, distributing zero rewards. Because `last_reward_block` is now equal to `current_block_number`, every subsequent call in the same block — including the legitimate sequencer's call — reverts with `REWARDS_ALREADY_UPDATED`.

The spec explicitly states the access control should be "Only starkware sequencer": [4](#0-3) 

But the interface exposes it to any address: [5](#0-4) 

### Impact Explanation

Each block's reward opportunity is a one-shot slot gated by `last_reward_block`. Once consumed with `disable_rewards: true`, that block's rewards are **permanently lost** — there is no mechanism to retroactively distribute rewards for a past block. An attacker who front-runs the sequencer on every block can suppress 100% of staking yield for all stakers indefinitely. This matches **High: Permanent freezing of unclaimed yield**.

### Likelihood Explanation

The call is cheap (no token transfer, no stake required). Any address can execute it. The attacker only needs to submit a transaction before the sequencer's `update_rewards` call each block. On Starknet, where the sequencer ordering is observable, this is straightforwardly achievable.

### Recommendation

Add a caller check at the top of `update_rewards` restricting it to the authorized sequencer address (or a role-gated address), consistent with the spec's stated access control of "Only starkware sequencer."

### Proof of Concept

1. Deploy two active stakers with STRK pools, advance K epochs so both have effective balances.
2. Start consensus rewards.
3. On each new block, have an unprivileged attacker call `update_rewards(staker_A, disable_rewards: true)` before the sequencer.
4. Observe: `last_reward_block` is set to the current block; the sequencer's subsequent call reverts with `REWARDS_ALREADY_UPDATED`.
5. After N blocks, both stakers' `unclaimed_rewards_own` remain at zero while the model predicts N × block_reward accumulation.

The test at `src/staking/tests/test.cairo` line 3956–3960 already demonstrates that calling with `disable_rewards: true` followed by `disable_rewards: false` in the same block causes the second call to revert — confirming the slot is consumed: [6](#0-5)
