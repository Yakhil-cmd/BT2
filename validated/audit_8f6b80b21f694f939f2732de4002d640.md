### Title
ERC-20 Tokens Permanently Burned When `ft_transfer` Promise Fails in `ExitToNear` Flow Without `error_refund` Feature — (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

When a user calls `withdrawToNear` on a bridged ERC-20 token, the tokens are burned immediately and a NEAR `ft_transfer` promise is scheduled asynchronously. If that promise fails (e.g., the recipient account is not registered with the NEP-141 token), the NEAR tokens remain in the Aurora engine contract and are never returned to the user. Without the `error_refund` compile-time feature enabled, no refund mechanism exists, causing permanent loss of user funds. This is a direct structural analog to the Omnipool refund-stuck-in-contract bug: an intermediary (the Aurora engine account) receives the value that should be returned to the user, but has no logic to forward it.

---

### Finding Description

**Step 1 — Token burn in `EvmErc20.sol`**

`withdrawToNear` immediately and unconditionally burns the caller's tokens, then calls the `ExitToNear` precompile via inline assembly. The return value of the assembly `call` is stored in `res` but is never checked; the function does not revert on failure. [1](#0-0) 

**Step 2 — Asynchronous NEAR promise, no synchronous refund path**

The `ExitToNear` precompile schedules a NEAR `ft_transfer` promise. This promise executes asynchronously after the EVM transaction completes. If the promise fails (unregistered recipient, insufficient storage deposit, etc.), the NEAR tokens remain in the Aurora engine account. The ERC-20 tokens are already burned and cannot be recovered. [2](#0-1) 

**Step 3 — `error_refund` feature is the only recovery path, and it is opt-in**

The only mechanism that re-mints ERC-20 tokens to the user on promise failure is gated behind the `error_refund` compile-time feature flag. When this feature is absent, the `MIN_INPUT_SIZE` is 3 bytes (no refund address field), and no callback is registered to handle the failure case. [3](#0-2) 

The feature also changes the wire protocol (adds a 20-byte `refund_address` field), meaning it cannot be silently enabled without a coordinated upgrade. Enabling it after deployment requires callers to supply the extra field, breaking backward compatibility with existing integrations.

**Step 4 — Test explicitly confirms fund loss without the feature**

The integration test `test_exit_to_near_refund` documents the two outcomes: [4](#0-3) 

Without `error_refund`, the user's ERC-20 balance is reduced by `FT_EXIT_AMOUNT` with no corresponding NEAR credit — the funds are gone.

---

### Impact Explanation

**Critical — Permanent freezing / direct loss of user funds.**

A user who calls `withdrawToNear` with a recipient that is not registered with the NEP-141 token (or whose storage deposit has been reclaimed) will have their ERC-20 tokens burned with no recovery. The NEAR-side tokens remain in the Aurora engine account indefinitely. There is no admin function to sweep or redistribute them to the rightful owner. The loss is permanent and proportional to the amount withdrawn.

---

### Likelihood Explanation

**Medium.**

- Any user can trigger this by supplying an unregistered NEAR account ID as the recipient — a common mistake for new users bridging to NEAR for the first time.
- A recipient account can become unregistered between the time a user constructs the transaction and the time the NEAR promise executes (storage reclamation).
- The `error_refund` feature changes the input encoding, so existing tooling (wallets, dApps) that does not include the refund address field cannot benefit from the fix even if the feature is compiled in.
- No special privilege is required; any EVM address holding bridged ERC-20 tokens is a potential victim.

---

### Recommendation

1. **Enable `error_refund` in the production build** and coordinate a protocol upgrade so all callers supply the `refund_address` field.
2. Alternatively, derive the `refund_address` automatically from the EVM `msg.sender` inside the precompile, eliminating the need for callers to supply it and removing the backward-compatibility break.
3. In `EvmErc20.sol`, check the return value of the assembly `call` to the exit precompile and revert if it fails, preventing the burn from succeeding when the precompile cannot schedule the promise. [5](#0-4) 

---

### Proof of Concept

1. Deploy Aurora with `error_refund` feature **disabled** (the default when the flag is absent).
2. Bridge a NEP-141 token to Aurora, minting ERC-20 tokens to address `A`.
3. From address `A`, call `erc20.withdrawToNear("unregistered.near", amount)`.
4. The ERC-20 contract burns `amount` tokens from `A` and calls the `ExitToNear` precompile.
5. The precompile schedules a NEAR `ft_transfer` to `unregistered.near`.
6. The NEAR promise fails because `unregistered.near` is not registered with the NEP-141 contract.
7. No `refund_on_error` callback fires (feature disabled).
8. Observe: `A`'s ERC-20 balance is reduced by `amount`; `unregistered.near` receives nothing; the NEAR tokens remain in the Aurora engine account.
9. The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–666) reproduces exactly this outcome under the `#[cfg(not(feature = "error_refund"))]` branch. [6](#0-5)

    async fn test_exit_to_near_refund() {

            ft_owner,
