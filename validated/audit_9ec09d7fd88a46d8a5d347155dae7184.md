### Title
Missing Verification of Participant CKD Contributions Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator aggregates cryptographic contributions from all participants without verifying their correctness. A single malicious participant can send arbitrary group elements, corrupting the final derived key output accepted by the coordinator.

### Finding Description
In `do_ckd_coordinator`, the coordinator computes its own share via `compute_signature_share` (which is locally verified by construction) and then receives `(norm_big_y, norm_big_c)` from every other participant via `recv_from_others`. These received values are unconditionally added to the running sum with no proof of correctness, no range check, and no consistency check against the participant's public key share:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to send:
- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)`

where `x_i` is the participant's private signing share and `y_i` is a fresh random scalar. The coordinator has no mechanism to verify either of these invariants. By contrast, the coordinator's own contribution is computed locally in `compute_signature_share` and is therefore implicitly correct. [2](#0-1) 

The participant side (`do_ckd_participant`) simply sends the two group elements with no attached proof: [3](#0-2) 

This is structurally identical to the reference bug: the protocol only "approves" (verifies) the primary actor's own contribution while blindly consuming every other participant's contribution that happens to be present.

### Impact Explanation
The CKD output `(Y, C)` is the sum of all participants' `(norm_big_y, norm_big_c)` values. A malicious participant who substitutes arbitrary group elements `(Y', C')` shifts the aggregate:

```
Y_final  = Y_honest + (Y'_malicious - Y_correct_malicious)
C_final  = C_honest + (C'_malicious - C_correct_malicious)
```

The app unmasks the result as `C_final − y · Y_final`, which no longer equals `msk · H(pk ‖ app_id)`. The coordinator returns a structurally valid `CKDOutput` that is cryptographically wrong. Every downstream consumer of the derived key silently accepts an incorrect value. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs**. [4](#0-3) 

### Likelihood Explanation
Any single participant in the CKD session can trigger this. No privileged access, no leaked key material, and no external dependency is required. The attacker only needs to be a valid member of the `participants` list passed to `ckd()`, which is the normal operating condition for any participant. The malicious message is a standard protocol message on the existing channel; the coordinator cannot distinguish it from an honest message. [5](#0-4) 

### Recommendation
Each participant must attach a zero-knowledge proof demonstrating that `norm_big_y` and `norm_big_c` were computed correctly with respect to their committed public key share. Concretely:

1. **Proof for `norm_big_y`**: A Schnorr/dlog proof that `norm_big_y = λ_i · y_i · G` for a known `y_i` (i.e., the participant knows the discrete log of `norm_big_y / λ_i` with respect to `G`).
2. **Proof for `norm_big_c`**: A proof that `norm_big_c` is consistent with the participant's public key share `X_i = x_i · G` and the hash point `H(pk ‖ app_id)` — e.g., a `dlogeq`-style proof showing `norm_big_c − λ_i · y_i · A = λ_i · x_i · H(pk ‖ app_id)`.

The coordinator must verify both proofs before adding the contribution to the aggregate. This pattern is already used in the triple-generation protocol (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`) where `dlog` and `dlogeq` proofs are verified for every received polynomial commitment before they are summed. [6](#0-5) 

### Proof of Concept

1. Run a CKD session with participants `[P1, P2, P3]` and coordinator `P1`.
2. `P2` (malicious) intercepts its own `compute_signature_share` result and instead sends `(G, G)` — the generator point for both `norm_big_y` and `norm_big_c`.
3. The coordinator at `do_ckd_coordinator` receives `P2`'s message and adds `G` to both running sums unconditionally.
4. The final `CKDOutput` returned by the coordinator is `(Y_honest + G, C_honest + G)`.
5. The app unmasks: `(C_honest + G) − y · (Y_honest + G) = msk · H(pk ‖ app_id) + G(1 − y)`, which is not the correct derived key.
6. The coordinator returns `Ok(Some(ckd_output))` with no error, and the caller silently accepts the corrupted key. [7](#0-6)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,

    app_pk: PublicKey,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates

        return Err(InitializationError::DuplicateParticipants);

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
