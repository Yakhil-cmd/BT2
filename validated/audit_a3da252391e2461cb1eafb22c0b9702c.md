### Title
`native_per_pubdata` Truncates to Zero via Floor Division, Enabling Free Pubdata Exploitation — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` fee rate is computed with truncating integer division (`wrapping_div`). When `pubdata_price < native_price`, the result is **zero**, making all pubdata (state-diff bytes published to L1) completely free for any transaction sender. There is no minimum check on this value anywhere in the codebase. The operator bears the full L1 data-availability cost with zero compensation.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK transaction flow), the per-pubdata-byte native cost is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is plain integer (floor) division. When `pubdata_price < native_price`, the quotient is **0**. No lower-bound guard exists. [1](#0-0) 

Contrast this with `native_per_gas`, which uses **ceiling** division to protect the operator:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:135
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The same asymmetry is present in the public API helper `validate_l2_tx_intrinsic_native_resources`, where the comment even documents the inconsistency explicitly:

```rust
// api/src/helpers.rs
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price   ← floor, no ceil
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata == 0`, every downstream pubdata charge becomes zero:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // → 0 * N = 0
   ``` [4](#0-3) 

2. **Post-execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)?;
   // → M * 0 = 0
   ``` [5](#0-4) 

The result: a transaction that writes to hundreds of storage slots generates zero native resource charge for pubdata, regardless of how many bytes it publishes to L1.

---

### Impact Explanation

The operator must pay L1 data-availability costs for every pubdata byte included in a batch. When `native_per_pubdata == 0`, those costs are entirely unrecovered from transaction senders. An attacker who observes this condition can flood the chain with storage-write-heavy transactions (e.g., writing to many distinct storage slots), each generating hundreds of pubdata bytes at zero marginal cost. The operator absorbs the full L1 DA bill with no compensation, constituting a direct financial loss and a resource-accounting invariant violation.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic in normal operation. `native_price` reflects ZK proving costs (which can be high), while `pubdata_price` reflects L1 blob/calldata costs (which fluctuate and can be low). Any time the sequencer sets these parameters such that `pubdata_price < native_price`, every L2 transaction sender can exploit the zero pubdata charge. The attacker requires no privileged access — only the ability to submit standard L2 transactions and observe the block context parameters.

---

### Recommendation

Replace the floor division for `native_per_pubdata` with ceiling division, consistent with how `native_per_gas` is computed:

```rust
// Before (floor):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling, matching native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Optionally, add an explicit guard rejecting transactions when `native_per_pubdata == 0` and `pubdata_price > 0`, analogous to the `native_price.is_zero()` guard already present for `native_per_gas`. [6](#0-5) 

---

### Proof of Concept

**Setup:**
- `native_price = 1_000_000` (high proving cost)
- `pubdata_price = 999_999` (slightly cheaper L1 DA)
- `native_per_pubdata = 999_999 / 1_000_000 = 0` (floor division)

**Attack:**
1. Attacker observes `native_per_pubdata == 0` from block context.
2. Attacker deploys a contract that writes to N distinct storage slots (each write = 32 bytes of pubdata key + compressed value ≈ 33–65 bytes).
3. Attacker submits a transaction calling that contract. With N = 100 slots, ~3,300–6,500 bytes of pubdata are generated.
4. `get_resources_to_charge_for_pubdata` returns `3300 * 0 = 0` native resources charged.
5. Transaction succeeds; operator pays L1 DA for 3,300+ bytes with zero recovery.
6. Attacker repeats across many transactions and blocks, accumulating unbounded operator losses at minimal cost. [7](#0-6)
