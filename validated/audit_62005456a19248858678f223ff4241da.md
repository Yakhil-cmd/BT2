### Title
ETH Sent to `RewardSupplier.tick()` Is Permanently Frozen When No Mint Requests Are Pending — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is declared `payable` and accepts ETH as L1→L2 message fees. However, the ETH is only forwarded inside the `if (amountToMint > 0)` branch. When no L2 mint requests are pending, the branch is skipped entirely and the ETH sent by the caller is permanently trapped in the contract with no refund path and no withdrawal mechanism.

---

### Finding Description

`tick()` is a public, permissionless, `payable` function: [1](#0-0) 

The ETH is split into two fees and forwarded only inside the conditional block: [2](#0-1) 

Specifically:
- `msgFee = msg.value / 2` is forwarded to `bridge().depositWithMessage{value: msgFee}(...)` at line 126.
- The remainder is forwarded to `messagingContract().sendMessageToL2{value: msgFee}(...)` at line 137.

Both calls are gated by `if (amountToMint > 0)`. [3](#0-2) 

When `amountToMint == 0` — i.e., `messagingContract().l2ToL1Messages(msgHash)` returns 0 — the function returns immediately after the check, consuming all supplied ETH without using or refunding it. [4](#0-3) 

Neither `RewardSupplier` nor its parent `RewardSupplierStorage` defines a `receive`, `fallback`, or ETH-withdrawal function. [5](#0-4) 

The contract has no mechanism to recover stranded ETH.

---

### Impact Explanation

Any ETH sent to `tick()` when there are no pending L2 mint requests is permanently frozen in the `RewardSupplier` contract. There is no admin withdrawal, no `receive` function, and no refund logic. The funds are irrecoverable at the current code level.

**Impact: High — Permanent freezing of funds.**

---

### Likelihood Explanation

`tick()` carries no access control — any address may call it. [1](#0-0) 

A keeper or bot that routinely calls `tick()` with ETH to pre-pay fees will lose funds whenever the call lands in a window where no L2 mint requests have been queued. This is a normal operational scenario (e.g., between reward epochs), making the likelihood realistic and recurring.

---

### Recommendation

Add an early-return refund or revert when no minting is required:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        // Refund ETH to caller; nothing to do.
        if (msg.value > 0) {
            (bool ok, ) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
        return;
    }
    // ... existing logic
}
```

Alternatively, revert with a descriptive error when `amountToMint == 0` and `msg.value > 0`, forcing callers to check `requiredMinting()` off-chain before sending ETH.

---

### Proof of Concept

1. L2 reward epoch has just completed; no new mint requests have been queued yet (`l2ToL1Messages(msgHash) == 0`).
2. A keeper calls `RewardSupplier.tick{value: 0.1 ether}()`.
3. `requiredMinting()` returns `(0, 0)`. [4](#0-3) 
4. The `if (amountToMint > 0)` branch is skipped. [2](#0-1) 
5. `tick()` returns. The 0.1 ETH is now held by `RewardSupplier` with no path to recover it.
6. No `receive`, `fallback`, or withdrawal function exists in the contract or any of its parents. [5](#0-4)

        uint256[] memory messagePayload = new uint256[](1);
        messagePayload[0] = TOKENS_PER_MINT_REQUEST;

        bytes32 msgHash = l2ToL1MsgHash(mintRequestSource(), address(this), messagePayload);
        // Limit the number of msgs to consume to limit.
        uint256 numMsgsToConsume = Math.min(
            messagingContract().l2ToL1Messages(msgHash),
            MAX_MESSAGES_TO_PROCESS_PER_TICK
        );

        return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
    }
