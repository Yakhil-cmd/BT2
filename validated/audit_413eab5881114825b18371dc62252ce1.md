### Title
Presignature Reuse via Non-Consuming `rerandomize_presign` API Enables Secret Key Extraction — (File: `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both the robust and OT-based ECDSA schemes document that each presignature is "one-time use," but the `rerandomize_presign` function takes `&PresignOutput` by shared reference instead of consuming it, and `PresignOutput` derives `Clone`. This means the library provides no type-level or runtime enforcement of the single-use invariant. A malicious coordinator can rerandomize the same presignature twice with different `(msg_hash, tweak, entropy)` parameters, run two concurrent signing sessions, and use the resulting pair of signatures — which share underlying nonce material — to extract the aggregate secret key via standard ECDSA nonce-reuse techniques.

---

### Finding Description

**Root cause — robust ECDSA (`src/ecdsa/robust_ecdsa/mod.rs`):**

`PresignOutput` derives `Clone`: [1](#0-0) 

`rerandomize_presign` takes the presignature by shared reference, not by value: [2](#0-1) 

Because the function signature is `rerandomize_presign(presignature: &PresignOutput, args: &RerandomizationArguments)`, the same `PresignOutput` can be passed to `rerandomize_presign` an unlimited number of times with different `RerandomizationArguments`. The Rust type system does not prevent this; ownership is never transferred.

**Root cause — OT-based ECDSA (`src/ecdsa/ot_based_ecdsa/mod.rs`):**

The identical structural problem exists: `PresignOutput` derives `Clone` and `rerandomize_presign` takes `&PresignOutput`: [3](#0-2) 

**Exploit path (robust ECDSA):**

1. The malicious coordinator participates in the presigning protocol and obtains `PresignOutput { big_r, c, e, alpha, beta }` for each participant.
2. The coordinator calls `rerandomize_presign(&presig, &args1)` with `args1 = (Y1, epsilon1, h1, R, rho1)` → `RerandomizedPresignOutput1` with `delta1 = HKDF(...)`.
3. The coordinator calls `rerandomize_presign(&presig, &args2)` with `args2 = (Y2, epsilon2, h2, R, rho2)` (different parameters) → `RerandomizedPresignOutput2` with `delta2 ≠ delta1`.
4. The coordinator initiates two concurrent signing sessions, distributing `RerandomizedPresignOutput1` to participants for session 1 and `RerandomizedPresignOutput2` for session 2.
5. Each participant computes and sends their linearized share `s_i = h * alpha_i + (beta_i * Rx + e_i)` to the coordinator. Since `alpha_i` and `beta_i` in both sessions are deterministic linear functions of the same underlying `presig.alpha`, `presig.beta`, `presig.c`, the two resulting signatures `(R1, s1)` and `(R2, s2)` have multiplicatively related nonces.
6. The coordinator applies standard ECDSA nonce-reuse algebra to the two signatures to recover the aggregate secret key.

The library's own security documentation explicitly acknowledges this attack class: [4](#0-3) 

The orchestration documentation for OT-based ECDSA also states the one-time-use requirement but provides no enforcement: [5](#0-4) 

The `sign()` entry points for both schemes take `presignature: RerandomizedPresignOutput` by value, which only prevents reuse of the *rerandomized* output — not the underlying `PresignOutput` from which it was derived: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Critical.** A malicious coordinator who participates in presigning can rerandomize the same `PresignOutput` twice with different `RerandomizationArguments`, run two signing sessions against honest participants, and recover the aggregate secret key from the resulting pair of signatures. This constitutes full extraction of private signing material, enabling the attacker to forge arbitrary signatures under the shared key without further participation from honest parties.

---

### Likelihood Explanation

**Low.** The attacker must be the designated coordinator (or compromise it) and must have participated in the presigning round to hold the `PresignOutput`. The coordinator role is reachable without privileged assumptions — it is simply one of the `2t+1` presigning participants. No cryptographic primitive break is required; the attack is purely algebraic once two signatures with related nonces are obtained.

---

### Recommendation

1. **Change `rerandomize_presign` to consume `PresignOutput` by value** (`presignature: PresignOutput`) in both `src/ecdsa/robust_ecdsa/mod.rs` and `src/ecdsa/ot_based_ecdsa/mod.rs`. This transfers ownership and prevents the caller from passing the same value twice.
2. **Remove `Clone` from `PresignOutput`** in both modules. Deriving `Clone` allows callers to trivially circumvent the by-value consumption by cloning before the call.
3. These two changes together make presignature reuse a compile-time error rather than a runtime documentation warning.

```rust
let presign_output: robust_ecdsa::PresignOutput = /* presigning result */;

// Rerandomize the SAME presignature twice with different signing contexts
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presign_output, &args1).unwrap();
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presign_output, &args2).unwrap();
// ^^^ Both calls succeed because rerandomize_presign takes &PresignOutput, not PresignOutput

// Run two signing sessions — honest participants contribute shares for both
let sig1 = run_signing_session(rerand1, h1, &participants); // (R1, s1)
let sig2 = run_signing_session(rerand2, h2, &participants); // (R2, s2)

// R1 = R^delta1, R2 = R^delta2 — nonces are multiplicatively related
// Standard ECDSA nonce-reuse algebra recovers the secret key x:
// s1 = (h1 * alpha + beta1 * Rx1 + e) / delta1
// s2 = (h2 * alpha + beta2 * Rx2 + e) / delta2
// Solving the system yields x (the aggregate secret key)
```

```rust
impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
```

```markdown
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
```

```markdown
Each phase can be run many times in advance, recording the information
```

```rust
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
```
