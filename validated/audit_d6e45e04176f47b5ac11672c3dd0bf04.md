### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions with Nonce=1 When Any Account Transaction Exists in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce=1` from an account whose on-chain nonce is `0`, whenever `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` for **any** transaction from that account in the pool, not exclusively a `deploy_account` transaction. An attacker who observes a `deploy_account` (or any other) transaction from account A in the mempool can race-submit a malicious invoke with `nonce=1` and an **invalid signature** that the gateway admits without calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` is designed to improve UX for the simultaneous `deploy_account + invoke` pattern: because the account does not yet exist on-chain, the `__validate__` entry point cannot be called, so the gateway skips it when a `deploy_account` is known to be pending. [1](#0-0) 

The skip condition fires when:
1. The transaction is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain state).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, and `validate_tx` returns `Ok(None)` immediately — no `__validate__` call, no signature check. [3](#0-2) [4](#0-3) 

The invariant-breaking flaw is in `account_tx_in_pool_or_recent_block`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

This returns `true` for **any** transaction type from that account — not only `deploy_account`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch is the vulnerability: a future-nonce invoke that passed validation proves the account **is already deployed** (nonce=0 on-chain, contract exists), yet the skip still fires, suppressing signature verification for the incoming nonce=1 invoke.

The `validate_by_mempool` call that precedes the skip check only verifies duplicate hash and nonce ordering — it does not verify signatures. [6](#0-5) 

---

### Impact Explanation

The gateway admits an invoke transaction with an **invalid (attacker-controlled) signature** into the mempool without calling `__validate__`. This satisfies the High impact criterion:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Concrete consequences:
- The malicious nonce=1 tx occupies the nonce=1 slot for account A in the mempool.
- The legitimate owner cannot submit their own nonce=1 tx (rejected as `DuplicateNonce`) unless they pay a fee-escalation premium.
- When the batcher executes the malicious tx, `new_for_sequencing` sets `validate: true`, `__validate__` is called, the invalid signature causes a non-revertible failure, and the tx is rejected — creating a nonce gap that stalls all subsequent txs for account A.

---

### Likelihood Explanation

**Moderate.** The preconditions are:

1. Account A has `account_nonce = 0` on-chain **and** has any transaction in the mempool (the normal state immediately after a `deploy_account` is submitted but before it is committed, or for any account deployed via `deploy` syscall with nonce=0).
2. The attacker observes the mempool (public P2P gossip) and races to submit the malicious nonce=1 tx before the legitimate owner does.

The `deploy_account + invoke` UX pattern is explicitly supported and documented, making condition 1 a routine occurrence. The race window is the time between the `deploy_account` entering the mempool and the legitimate owner submitting their nonce=1 invoke — which can be seconds to minutes.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** for the sender address is pending in the mempool. Alternatively, add a check that the account contract does **not** exist on-chain before skipping validation — if the account is already deployed, `__validate__` must always be called.

---

### Proof of Concept

**Setup:** Account A is not yet deployed. `max_allowed_nonce_gap ≥ 1`.

**Step 1 — Legitimate owner submits `deploy_account`:**
```
POST /gateway/add_transaction
{ type: "DEPLOY_ACCOUNT", nonce: 0, sender: A, signature: <valid> }
→ accepted, deploy_account tx enters mempool
```

**Step 2 — Attacker observes mempool, submits malicious invoke:**
```
POST /gateway/add_transaction
{ type: "INVOKE", nonce: 1, sender: A, signature: [0xdead, 0xbeef] }
```

Gateway path:
- `StatelessTransactionValidator::validate` — passes (signature length ≤ max, resource bounds non-zero).
- `extract_state_nonce_and_run_validations`:
  - `get_nonce_from_state(A)` → `Nonce(0)`.
  - `validate_state_preconditions`: nonce check `0 ≤ 1 ≤ 0 + max_gap` → passes.
  - `validate_by_mempool`: no duplicate hash, nonce ordering valid → passes.
  - `skip_stateful_validations`: `nonce == 1 ∧ account_nonce == 0 ∧ account_tx_in_pool_or_recent_block(A) == true` → **returns `true`**.
  - `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate = false` → `validate_tx` returns `Ok(None)` — **`__validate__` never called, signature never verified**.
- Malicious tx **admitted to mempool**.

**Step 3 — Legitimate owner tries to submit their own nonce=1 invoke:**
```
→ MempoolError::DuplicateNonce  (or forced to pay fee-escalation premium)
```

**Step 4 — Batcher executes malicious tx:**
- `new_for_sequencing` sets `validate: true`.
- `__validate__` called → invalid signature → non-revertible failure → tx rejected.
- Nonce gap created; all subsequent txs for account A are stalled. [7](#0-6) [8](#0-7) [5](#0-4)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
