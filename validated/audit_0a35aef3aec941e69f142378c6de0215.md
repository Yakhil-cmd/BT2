### Title
ERC-20 Tokens Permanently Burned Without Refund When `ft_transfer` Promise Fails in `ExitToNear` Precompile (Without `error_refund` Feature) - (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not compiled in (the default production path), the `ExitToNear` precompile burns a user's ERC-20 tokens and schedules an `ft_transfer` promise to the NEP-141 contract with **no callback and no refund path**. If the `ft_transfer` promise fails for any reason (e.g., recipient not registered with the NEP-141 contract), the ERC-20 tokens are permanently destroyed while the NEP-141 tokens remain locked in Aurora's account. This is a permanent, irrecoverable fund loss for the user.

---

### Finding Description

The `ExitToNear` precompile flow for bridged ERC-20 tokens works as follows:

**Step 1 – Burn first, ask questions later.**
`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens unconditionally before calling the precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // <-- irreversible burn
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, ...)
    }
}
``` [1](#0-0) 

**Step 2 – Precompile constructs callback args with `refund: None` when feature is off.**
In `ExitToNear::run`, the `callback_args` struct is built with the refund field hardcoded to `None` when the `error_refund` feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // <-- no refund path in production
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

**Step 3 – For regular ERC-20 exits, no callback is attached at all.**
Because `transfer_near_args` is `None` for non-wNEAR exits, `callback_args` equals the default (both fields `None`), so the promise is created with no callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // <-- bare promise, no failure handler
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

**Step 4 – `exit_to_near_precompile_callback` only refunds if `args.refund` is `Some`.**
Even when a callback is attached (wNEAR unwrap case), the failure branch only executes if `args.refund` is populated, which requires the `error_refund` feature:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- silent failure, no refund
};
``` [4](#0-3) 

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

---

### Impact Explanation

When `ft_transfer` to the NEP-141 contract fails:
- The user's ERC-20 tokens are **permanently burned** (no re-mint occurs).
- The NEP-141 tokens remain in Aurora Engine's account with no mechanism to recover them.
- This constitutes **permanent freezing of user funds** — the ERC-20 side is destroyed and the NEP-141 side is locked.

This matches the Critical impact class: **Permanent freezing of funds**.

---

### Likelihood Explanation

The failure condition is realistic and user-triggerable:
- Sending to a NEAR account not registered with the NEP-141 contract (a common user error).
- The NEP-141 contract being paused at the time of the `ft_transfer` execution.
- Any transient failure in the NEP-141 contract.

The entry point (`withdrawToNear` on any bridged ERC-20) is a standard, publicly callable function available to any EVM user. No special privileges are required. [1](#0-0) 

---

### Recommendation

1. **Enable `error_refund` in the production build** so that `refund_call_args` is always populated and a failure callback is always attached.
2. Alternatively, **always attach a callback** regardless of the feature flag, and in the callback, re-mint burned ERC-20 tokens (or restore ETH balance) when the `ft_transfer` promise result is `Failed`.
3. Consider making `EvmErc20V2` (which encodes the sender address for refund purposes) the only deployed version, and ensure the engine is compiled with `error_refund` enabled.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (default production build).
2. Bridge a NEP-141 token to an ERC-20 on Aurora.
3. Call `erc20.withdrawToNear(unregistered_near_account, amount)` from an EVM account that holds ERC-20 tokens.
4. The ERC-20 tokens are burned immediately.
5. The `ft_transfer` promise to the NEP-141 contract fails because `unregistered_near_account` is not registered.
6. No callback fires; no re-mint occurs.
7. Observe: ERC-20 balance reduced by `amount`, NEP-141 balance of Aurora unchanged, user's funds permanently lost.

This is confirmed by the existing test `test_exit_to_near_refund` which asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` (i.e., tokens are gone) when `error_refund` is not enabled. [6](#0-5)
