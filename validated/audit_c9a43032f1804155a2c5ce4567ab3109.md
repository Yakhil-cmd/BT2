### Title
Precision Loss in `native_per_pubdata` Calculation via Floor Division Causes Systematic Pubdata Cost Undercharging — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` rate is computed using floor (truncating) integer division (`wrapping_div`), while the analogous `native_per_gas` rate uses ceiling division (`div_ceil`). This asymmetry causes pubdata costs to be systematically undercharged for every L2 transaction. In the extreme case where `pubdata_price < native_price`, `native_per_pubdata` truncates to zero, allowing users to publish pubdata without paying any native resources for it.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the `native_per_pubdata` rate is computed as:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same floor-division pattern appears in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

By contrast, `native_per_gas` is computed with ceiling division to ensure the user pays *at least* the actual cost:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [3](#0-2) 

The `native_per_pubdata` value is then used in two places that directly affect how much native resource is charged for pubdata:

1. **Intrinsic pubdata overhead** deducted from the native limit before execution:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [4](#0-3) 

2. **Per-byte pubdata charge** during and after execution:
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ``` [5](#0-4) 

The documentation confirms the intended formula is `nativePerPubdata := pubdataPrice / nativePrice`, but does not specify rounding direction: [6](#0-5) 

---

### Impact Explanation

**General case (always present):** For any `pubdata_price` not exactly divisible by `native_price`, the truncation error is up to `(native_price − 1)` native units per pubdata byte. For example, with `pubdata_price = 15` and `native_price = 10`:
- Exact rate: 1.5 native/byte
- Charged rate: `floor(15/10) = 1` native/byte
- Relative undercharge: **33%**

A transaction publishing 1000 pubdata bytes would pay 1000 native instead of 1500 native — a 500-native shortfall. The operator/protocol absorbs this difference.

**Extreme case (`pubdata_price < native_price`):** `native_per_pubdata` truncates to **zero**. The user pays zero native resources for all pubdata. Both the intrinsic pubdata overhead and the per-byte execution pubdata charge become zero, giving the user their full native limit for computation while publishing pubdata for free. The operator must still pay L1 calldata costs for this pubdata.

This is a resource accounting bug: users can underpay for pubdata, with the shortfall borne by the operator/protocol.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are set by the operator in block metadata and are not directly attacker-controlled. However:

- The truncation error (general case) is **always present** for any non-divisible price pair, which is the common case in practice.
- The `native_per_pubdata = 0` case requires `pubdata_price < native_price`. While unusual, this is a valid configuration (e.g., during low-L1-fee periods or misconfiguration).
- Any unprivileged transaction sender can exploit the undercharge simply by submitting a normal transaction — no special privileges are needed once the block metadata is set.

---

### Recommendation

Use ceiling division for `native_per_pubdata` to be consistent with `native_per_gas` and ensure users pay at least the actual cost:

```rust
// In validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// In api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

This mirrors the existing `div_ceil` pattern used for `native_per_gas` and eliminates the systematic undercharge.

---

### Proof of Concept

**Scenario 1 — 33% undercharge:**
- `pubdata_price = 15`, `native_price = 10`
- `native_per_pubdata = floor(15/10) = 1` (should be 2 with ceiling)
- Transaction publishes 1000 pubdata bytes
- Charged: `1 × 1000 = 1000` native
- Actual cost: `1.5 × 1000 = 1500` native
- Shortfall: **500 native (33%)**

**Scenario 2 — zero pubdata charge:**
- `pubdata_price = 9`, `native_price = 10`
- `native_per_pubdata = floor(9/10) = 0`
- `intrinsic_pubdata_overhead = 0 × intrinsic_pubdata = 0`
- `native charged for pubdata = 0 × pubdata_bytes = 0`
- User publishes pubdata at **zero native cost**; operator pays L1 calldata fees uncompensated
