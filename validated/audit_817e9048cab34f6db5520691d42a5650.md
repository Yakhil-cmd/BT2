### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The Confidential Key Derivation (CKD) coordinator aggregates participant contributions by blindly summing received group elements with no cryptographic verification. A single malicious participant can send arbitrary values for `(norm_big_y, norm_big_c)`, causing the coordinator to compute and accept a completely wrong derived-key ciphertext. The coordinator has no mechanism to specify or enforce what the expected output should be, and no way to detect the corruption without possessing the application secret key `app_sk`.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator:

1. Computes its own honest share `(norm_big_y, norm_big_c)` via `compute_signature_share`.
2. Calls `recv_from_others` to collect one `CKDOutput` from every other participant.
3. Unconditionally adds each received `big_y()` and `big_c()` to its running totals.
4. Returns the sum as the final `CKDOutput`.

```rust

There is no zero-knowledge proof, range check, or any other verification that the received `(norm_big_y, norm_big_c)` pair was honestly computed as `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · app_pk))`. The correct computation is defined in `compute_signature_share`: [2](#0-1) 

A malicious participant simply sends arbitrary BLS12-381 G1 elements instead of their honest contribution. The coordinator sums them in, producing a `CKDOutput` that does not correspond to `H(pk, app_id) · msk` encrypted under `app_pk`. The caller (coordinator) cannot specify what the expected output should be, and the protocol provides no mechanism to reject a corrupted result.

### Impact Explanation
The final `CKDOutput` is `(big_y, big_c)` where the intended semantic is an ElGamal encryption of the confidential derived key `msk · H(pk, app_id)`. When a malicious participant injects arbitrary `(δ_y, δ_c)`, the coordinator outputs `(big_y + δ_y, big_c + δ_c)`. The `unmask(app_sk)` operation then yields `(big_c + δ_c) − app_sk · (big_y + δ_y)`, which is not the correct derived key. The honest coordinator accepts this corrupted output as the protocol result, satisfying the **High** impact criterion: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs*.

### Likelihood Explanation
Any single participant in the CKD protocol can trigger this. The attacker only needs to send a malformed `CKDOutput` message to the coordinator — a one-round, zero-cost operation requiring no special privilege beyond being a listed participant. The coordinator has no defense.

### Recommendation
Each participant should accompany their `(norm_big_y, norm_big_c)` contribution with a non-interactive zero-knowledge proof of correct computation (e.g., a Chaum-Pedersen proof showing that `norm_big_c − norm_big_y · app_pk` lies on the correct coset of `H(pk, app_id) · G2`). The coordinator must verify all proofs before aggregating. Alternatively, the coordinator can verify the final output by checking it against a publicly reconstructable commitment derived from the participants' public key shares and the known `H(pk, app_id)` point.

### Proof of Concept
1. Honest participants: `P1` (coordinator), `P2`, `P3` with valid key shares summing to `msk`.
2. `P2` is malicious. Instead of sending `(λ_2 · y_2 · G, λ_2 · (x_2 · H(pk, app_id) + y_2 · app_pk))`, it sends `(G, G)` — the generator point for both components.
3. `P1` receives `P3`'s honest contribution and `P2`'s malicious `(G, G)`, sums them all.
4. The resulting `CKDOutput` is `(honest_big_y + G, honest_big_c + G)`.
5. `P1` returns this as `Some(ckd_output)` with no error.
6. Any downstream consumer calling `ckd_output.unmask(app_sk)` obtains a value that is not `msk · H(pk, app_id)`, silently corrupting the derived key. [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
