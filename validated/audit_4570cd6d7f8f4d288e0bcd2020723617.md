Audit Report

## Title
Unbounded `disburse_maturity_in_progress` Vec in SNS Governance Enables Periodic Task DoS — (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `disburse_maturity` function unconditionally pushes entries to `disburse_maturity_in_progress` with no cap on the vector length. The NNS governance enforces `MAX_NUM_DISBURSEMENTS = 10`; the SNS has no equivalent guard. A neuron owner with `DisburseMaturity` permission and sufficient maturity can enqueue ~3,000 pending disbursements, causing O(N) ledger calls during every `run_periodic_tasks` execution and potentially blocking SNS governance operations.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 1609–1706, the `disburse_maturity` function validates only that `percentage_to_disburse` is in `[1, 100]` and that `worst_case_maturity_modulation >= transaction_fee_e8s`. It then unconditionally pushes to the vec:

```rust
neuron
    .disburse_maturity_in_progress
    .push(disbursement_in_progress);  // lines 1696-1698
```

There is no check on `disburse_maturity_in_progress.len()` anywhere in the SNS governance code path.

By contrast, `rs/nns/governance/src/governance/disburse_maturity.rs` defines `MAX_NUM_DISBURSEMENTS: usize = 10` at line 40 and enforces it at lines 306–308:

```rust
if num_disbursements >= MAX_NUM_DISBURSEMENTS {
    return Err(InitiateMaturityDisbursementError::TooManyDisbursements);
}
```

The natural floor for the SNS is determined by the transaction fee check. With `percentage_to_disburse = 1`, each call deducts 1% of remaining maturity geometrically. Starting from `u64::MAX` maturity (~1.8×10¹⁹ e8s) and a 10,000 e8s fee, the worst-case modulation check (`0.95 × 0.01 × M × 0.99^N ≥ 10,000`) allows approximately **~3,031 entries** before calls begin failing. The `maybe_finalize_disburse_maturity` function, called from `run_periodic_tasks`, iterates all entries and issues one ledger call per ready entry, producing O(N) inter-canister calls in a single periodic task execution.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant SNS security impact with concrete user or protocol harm."*

With ~3,000 entries ready for finalization, `run_periodic_tasks` issues ~3,000 inter-canister ledger calls in a single invocation. This can exhaust the per-round instruction limit of the SNS governance canister, stall or skip other periodic tasks (proposal execution, reward distribution, upgrade scheduling), and render the SNS governance canister effectively non-functional until the disbursements are processed — which itself requires the same overloaded periodic task to complete.

## Likelihood Explanation
The attacker requires only: (1) a neuron in the target SNS with `DisburseMaturity` permission, achievable through normal participation; (2) sufficient accumulated maturity, achievable through normal reward accumulation; (3) ~3,000 sequential `manage_neuron(DisburseMaturity { percentage_to_disburse: 1, ... })` calls via the public ingress endpoint. No privileged access, key compromise, or governance majority is required. The attack is fully self-contained and repeatable.

## Recommendation
Add an explicit cap before the push in `disburse_maturity` in `rs/sns/governance/src/governance.rs`, mirroring the NNS pattern:

```rust
const MAX_DISBURSE_MATURITY_IN_PROGRESS: usize = 10;

let neuron = self.get_neuron_result(id)?;
if neuron.disburse_maturity_in_progress.len() >= MAX_DISBURSE_MATURITY_IN_PROGRESS {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Too many disbursements in progress for this neuron.",
    ));
}
```

This check should be inserted before line 1696. A corresponding test mirroring `test_initiate_maturity_disbursement_too_many_disbursements` from `rs/nns/governance/src/governance/disburse_maturity_tests.rs` should be added to the SNS governance test suite.

## Proof of Concept
1. Create an SNS neuron with large `maturity_e8s_equivalent` (e.g., set directly in a PocketIC or integration test).
2. Repeatedly call `manage_neuron(DisburseMaturity { percentage_to_disburse: 1, to_account: None })` as the neuron controller in a loop.
3. Assert that each call returns `Ok` and that `disburse_maturity_in_progress.len()` increments each time — confirming no cap is enforced.
4. After ~3,000 calls, advance the mock clock past `MATURITY_DISBURSEMENT_DELAY_SECONDS` and invoke `run_periodic_tasks`.
5. Observe that `maybe_finalize_disburse_maturity` issues O(N) ledger calls in a single execution, exhausting available instructions and blocking other periodic governance tasks.

The NNS equivalent guard and test at `rs/nns/governance/src/governance/disburse_maturity.rs` lines 306–308 and `rs/nns/governance/src/governance/disburse_maturity_tests.rs` lines 404–438 confirm the intended design; the SNS governance has no analogous protection.
