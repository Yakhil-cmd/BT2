### Title
Stale `isUpdater` Mapping Persists After `removeProvider` + `addProvider` Cycle, Enabling Unauthorized Confidence Manipulation — (File: `smart-contracts-poc/contracts/PriceProviderFactory.sol`, `smart-contracts-poc/contracts/PriceProviderFactoryL2.sol`)

### Summary

`PriceProviderFactory.removeProvider()` removes a provider from the tracked set but does not clear the nested `isUpdater[provider]` mapping. When `addProvider()` re-registers the same provider address, every previously-granted updater retains live `setConfidence()` access, bypassing the oracle role boundary and enabling unauthorized manipulation of the price provider's `confidenceParam`.

### Finding Description

`PriceProviderFactory` (and its L2 variant) maintains a nested mapping:

```solidity
mapping(address provider => mapping(address updater => bool)) public isUpdater;
```

`removeProvider` only removes the provider from the `_providers` `EnumerableSet` and from `_providersByCreator[owner]`. It does not touch `isUpdater[provider]`:

```solidity
function removeProvider(address provider) external override onlyRole(ADMIN_ROLE) {
    if (!_providers.remove(provider)) revert ProviderNotTracked();
    address owner = providerOwner[provider];
    if (owner != address(0)) {
        _providersByCreator[owner].remove(provider);
    }
    emit ProviderRemoved(provider);
}
```

`addProvider` re-registers the same contract address without clearing stale updater grants:

```solidity
function addProvider(address provider) external override onlyRole(ADMIN_ROLE) {
    require(PriceProvider(provider).factory() == address(this));
    if (!_providers.add(provider)) revert ProviderAlreadyTracked();
    address owner = providerOwner[provider];
    if (owner != address(0)) {
        _providersByCreator[owner].add(provider);
    }
    emit ProviderAdded(provider);
}
```

After re-registration, `setConfidence` passes both guards for any stale updater U:

```solidity
require(_providers.contains(providers[i]), ProviderNotTracked()); // passes — provider is back
_requireUpdater(providers[i]);                                     // passes — isUpdater[P][U] still true
PriceProvider(providers[i]).setConfidenceParam(values[i]);        // executes with attacker-chosen value
```

This is the direct analog of the external report: a logical "deletion" (removal from the tracked set) that leaves an inner mapping (`isUpdater`) intact, so re-adding the entity silently restores all previously-granted permissions.

### Impact Explanation

`confidenceParam` controls the spread/confidence shaping applied to the oracle mid-price before it is delivered to pools as a bid/ask quote. A stale updater who has been implicitly decommissioned (by the admin removing the provider) can set an extreme `confidenceParam` value on the re-added provider. This produces a stale or distorted bid/ask quote that reaches every pool whose `PriceProvider` is this contract, enabling bad-price execution: traders receive more than the oracle curve permits, or LPs suffer adverse fills at manipulated spreads. The corrupted value is `isUpdater[provider][staleUpdater] = true` persisting across the remove/add cycle.

### Likelihood Explanation

The admin removes a provider to decommission it or revoke all associated roles (a natural operational step, e.g., after an updater key is compromised). The admin then re-adds the same provider address (e.g., after the underlying issue is resolved, or by mistake). The stale updater — now an unprivileged actor whose grant was implicitly revoked — retains live `setConfidence()` access and can immediately manipulate the confidence parameter. No malicious setup is required; the trigger is a valid admin remove+add cycle combined with an unprivileged stale-updater call.

### Recommendation

Clear all updater grants for the provider inside `removeProvider`, or alternatively clear `providerOwner[provider]` so that `addProvider` treats the re-added provider as a fresh entry with no inherited permissions. The minimal fix is to require the caller to enumerate and revoke all updaters before removal, or to iterate and zero the known updater set (if tracked) inside `removeProvider`.

### Proof of Concept

```
1. Admin calls factory.addProvider(P)          → _providers contains P; providerOwner[P] = owner
2. Owner calls factory.grantUpdater(P, U)      → isUpdater[P][U] = true
3. Admin calls factory.removeProvider(P)       → _providers no longer contains P
                                                  isUpdater[P][U] still = true  ← stale
4. Admin calls factory.addProvider(P)          → _providers contains P again
5. Stale updater U calls:
       factory.setConfidence([P], [extreme])
   → _providers.contains(P)  ✓
   → isUpdater[P][U] == true ✓
   → PriceProvider(P).setConfidenceParam(extreme) executes
   → All pools using provider P now receive a distorted bid/ask quote
