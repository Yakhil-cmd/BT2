### Title
Signature Verification Bypass via Overly Broad `skip_stateful_validations` Condition — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (the account contract's signature verification) for any invoke transaction with `nonce=1` from an address that has **any** transaction in the mempool — not exclusively a `deploy_account` transaction. An unprivileged attacker can exploit this to inject an invoke transaction with an arbitrary/invalid signature into the mempool by first ensuring the target address appears in the mempool state through a different transaction type.

### Finding Description

The UX feature for simultaneous `deploy_account + invoke` submission is implemented in `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:
1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The comment claims this is safe because the address being in the mempool implies it has a `deploy_account` transaction. This is incorrect. `account_tx_in_pool_or_recent_block` checks: [2](#0-1) 

which resolves to: [3](#0-2) 

This returns `true` if the address appears in `staged`, `committed`, or `tx_pool` — regardless of whether the transaction type is `deploy_account`. An already-deployed account with on-chain nonce `0` (deployed but never sent a transaction) can have an `invoke(nonce=0)` in the mempool, satisfying condition 3 without any `deploy_account` being present.

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

The blockifier's `StatefulValidator` then short-circuits before calling `__validate__`: [5](#0-4) 

And `validate_tx` in `AccountTransaction` also returns early: [6](#0-5) 

No other path in the stateless validator checks the cryptographic signature: [7](#0-6) 

### Impact Explanation

An attacker can submit an `invoke(nonce=1)` transaction with an **arbitrary/forged signature** from any account `A` that satisfies:
- On-chain nonce of `A` is `0`
- `A` has any pending transaction in the mempool (e.g., a legitimate `invoke(nonce=0)` submitted by the real owner)

The gateway admits the forged transaction to the mempool without running `__validate__`. This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions before sequencing."*

Secondary effects:
- The forged `nonce=1` transaction occupies the nonce slot, blocking the legitimate owner's `nonce=1` transaction (griefing/DoS)
- Mempool can be flooded with signature-invalid transactions targeting any account with a pending `nonce=0` invoke

### Likelihood Explanation

The precondition — an account with on-chain nonce `0` having a pending `nonce=0` invoke in the mempool — is common. Any deployed account that has never committed a transaction and is currently submitting its first one satisfies this. The attack requires only observing the mempool and submitting a crafted transaction, which is unprivileged and requires no special access.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction (not just any transaction) is pending for the address. Alternatively, only skip `__validate__` when the account contract does not yet exist on-chain (i.e., `get_class_hash_at(sender_address)` returns the zero class hash), which is the actual condition under which `__validate__` cannot be run.

### Proof of Concept

1. Deploy account `A` on-chain (class hash set, nonce = 0 in state).
2. Submit a valid `invoke(nonce=0, sender=A, valid_sig)` → admitted to mempool; `A` now appears in `tx_pool`.
3. Submit `invoke(nonce=1, sender=A, sig=[0xdeadbeef, 0xdeadbeef])` (garbage signature).
4. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (on-chain state unchanged)
   - `account_tx_in_pool_or_recent_block(A)` → `true` (step 2 tx is in pool) ✓
   - Returns `true`
5. `run_validate_entry_point` sets `validate=false`; `__validate__` is never called.
6. Forged transaction is forwarded to the mempool and accepted.
7. The legitimate owner's `invoke(nonce=1)` is now blocked (duplicate nonce) or must pay a higher fee to replace the forged entry. [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        // TODO(Arni, 1/5/2024): Add a mechanism that validate the sender address is not blocked.

        Self::validate_contract_address(tx)?;

            self.validate_declare_tx(declare_tx)?;
        }
