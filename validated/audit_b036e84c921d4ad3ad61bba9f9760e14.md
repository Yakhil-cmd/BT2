### Title
Gas under-charge for cached `CompileModuleError` on disk-cache hit when `fix_contract_loading_cost = true` — (`runtime/near-vm-runner/src/wasmtime_runner/mod.rs`)

### Summary
When `fix_contract_loading_cost = true` and a contract's compilation error is served from the **disk cache** (not the in-memory cache), `before_loading_executable` is called with `err.size_bytes_approximate()` (~4–100 bytes) instead of the actual wasm code size stored in `CompiledContractInfo.wasm_bytes`. This causes a systematic under-charge of `contract_loading_bytes * (actual_wasm_size - err.size_bytes_approximate())` gas units.

### Finding Description
In `WasmtimeVM::with_compiled_and_loaded`, the `try_lookup` closure returns a two-element tuple. For a disk-cache hit with `CompileModuleError`, the first element (which becomes `wasm_bytes` at line 816) is set to `err.size_bytes_approximate()` rather than `CompiledContractInfo.wasm_bytes`. The actual wasm size is available in the destructured `wasm_bytes` variable on line 712 but is only forwarded into the `to_any(...)` payload (for the memory cache), not used as the gas-charging size.

The correct value (`CompiledContractInfo.wasm_bytes`) is used on the memory-cache hit path (second closure, lines 805–811) and on the fresh-compile path (line 727). Only the disk-cache-hit path for `CompileModuleError` is wrong.

### Impact Explanation
`VMOutcome::burnt_gas` is under-reported by up to `contract_loading_bytes * actual_wasm_size` gas. With `contract_loading_bytes = 216,750` gas/byte and a 4 MB contract, the maximum under-charge is ~867 billion gas (~867 TGas). The user receives a larger-than-correct gas refund. Additionally, nodes with a warm memory cache charge the correct amount while nodes that just restarted (disk cache only) charge the wrong amount — a potential consensus divergence for the same receipt.

### Likelihood Explanation
Triggering requires: (1) deploying a contract that fails compilation (valid unprivileged action), (2) the error being persisted to the disk cache (happens automatically), and (3) a subsequent call after the memory cache is cold (e.g., after a node restart or cache eviction). This is a realistic scenario in production.

### Recommendation
In the `CompileModuleError` disk-cache-hit branch (lines 714–718), replace `err.size_bytes_approximate() as u64` with the already-destructured `wasm_bytes` from `CompiledContractInfo`:

```rust
CompiledContract::CompileModuleError(err) => {
    return Ok((
        wasm_bytes,  // actual wasm size, not err.size_bytes_approximate()
        to_any((wasm_bytes, Err(err))),
    ));
}
``` [7](#0-6) 

### Proof of Concept
1. Deploy a contract with invalid wasm bytes (e.g., `[42; 10_000]`) — this triggers `WasmtimeCompileError` and caches the error with `wasm_bytes = 10_000`.
2. Call the contract once (populates disk cache with `CompileModuleError`).
3. Clear the memory cache (simulate node restart).
4. Call the contract again with `fix_contract_loading_cost = true`.
5. Assert: `burnt_gas == loading_base + loading_bytes * err.size_bytes_approximate()` (actual observed) vs `loading_base + loading_bytes * 10_000` (expected). The difference is `loading_bytes * (10_000 - ~4)` gas units of under-charge.

    pub fn size_bytes_approximate(&self) -> usize {
        const BASE_SIZE: usize = 4; // to accommodate for String/Box/Box<str> etc

        let dynamic_size = match self {
            CompilationError::CodeDoesNotExist { account_id } => account_id.len(),
            CompilationError::WasmerCompileError { msg }
            | CompilationError::WasmtimeCompileError { msg } => msg.len(),
            CompilationError::PrepareError(_) => 0,
        };

        BASE_SIZE + dynamic_size
    }
