### Title
First Pool Deployed by Factory Is Never Recognized as a Valid Pool, Breaking Extensions and Router Validation — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.isPool()` uses `poolToIdx[pool] != 0` as its validity sentinel. Because `nextPoolIdx` is an uninitialized `uint256` that starts at `0`, the very first pool created receives index `0`, which is indistinguishable from "not registered." Every downstream guard that calls `isPool` therefore treats the first pool as illegitimate for its entire lifetime.

---

### Finding Description

**Registration logic** in `createPool`:

```solidity
uint256 poolIdx = nextPoolIdx;   // == 0 on first call
nextPoolIdx++;
idxToPool[poolIdx] = pool;
poolToIdx[pool] = poolIdx;       // poolToIdx[firstPool] = 0
``` [1](#0-0) 

**Validity check**:

```solidity
function isPool(address pool) external view override returns (bool) {
    return poolToIdx[pool] != 0;   // false for firstPool forever
}
``` [2](#0-1) 

The first pool is stored with index `0`, which is the Solidity default for any unregistered address. The two cases are permanently aliased.

**Extension guard** (`BaseMetricExtension.onlyPool`):

```solidity
modifier onlyPool() {
    if (!IMetricOmmPoolFactory(FACTORY).isPool(msg.sender)) {
        revert OnlyPool(msg.sender, FACTORY);
    }
    _;
}
``` [3](#0-2) 

Every extension callback (`beforeSwap`, `afterSwap`, `beforeAddLiquidity`, `afterAddLiquidity`, `beforeRemoveLiquidity`, `afterRemoveLiquidity`) is gated by `onlyPool`. When the first pool calls any extension, `isPool` returns `false` and the call reverts, bubbling up through `_callExtensionsInOrder` → `CallExtension.callExtension` → the pool action itself. [4](#0-3) 

The router also consults `isPool` (4 call-sites in `MetricOmmSwapRouterBase`) to authenticate pools before routing, so even extension-free swaps through the periphery are rejected for the first pool.

---

### Impact Explanation

| Scenario | Effect |
|---|---|
| First pool has any extension with `beforeSwap` / `afterSwap` order set | Every `swap()` call reverts; pool is permanently frozen for trading |
| First pool has `beforeRemoveLiquidity` / `afterRemoveLiquidity` extension | LPs cannot withdraw; principal is trapped |
| First pool has no extensions | Direct pool calls work, but all router-mediated swaps revert |

The worst case — extensions covering remove-liquidity hooks — causes complete, irrecoverable loss of LP principal deposited into the first pool, with no admin escape hatch (the factory has no mechanism to override extension calls or re-index a pool).

---

### Likelihood Explanation

The trigger is deterministic: the very first `createPool` call on any freshly deployed factory produces the broken pool. No attacker action is required; normal protocol deployment is sufficient. The severity of the impact scales with whether extensions are wired to the first pool, which is a routine configuration choice made at pool creation time.

---

### Recommendation

Initialize `nextPoolIdx` to `1` in the factory constructor so that the first registered pool receives index `1` and `poolToIdx[firstPool] != 0` holds:

```solidity
constructor(...) {
    nextPoolIdx = 1;   // reserve 0 as the "not registered" sentinel
    ...
}
```

Alternatively, maintain a separate `mapping(address => bool) private _isPool` and set it to `true` on registration, removing the index-as-sentinel pattern entirely.

---

### Proof of Concept

```solidity
// 1. Deploy factory (nextPoolIdx == 0 by default)
MetricOmmPoolFactory factory = new MetricOmmPoolFactory(owner, 0, 0);
factory.setPoolDeployer(address(new MetricOmmPoolDeployer(address(factory))));

// 2. Create the first pool
address firstPool = factory.createPool(params);
// poolToIdx[firstPool] == 0  (assigned nextPoolIdx before increment)

// 3. isPool returns false for the first pool
assert(!factory.isPool(firstPool));   // PASSES — pool is not recognized

// 4. Any extension callback from firstPool reverts
// BaseMetricExtension.onlyPool: isPool(msg.sender) == false → revert OnlyPool
// → swap(), modifyLiquidity() revert if any extension order is non-zero
// → LP funds are trapped if beforeRemoveLiquidity/afterRemoveLiquidity is set
```
