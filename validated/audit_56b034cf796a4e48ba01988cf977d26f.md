Audit Report

## Title
ckBTC Minter Finalizes `AmountTooLow` Withdrawal Requests Without Reimbursement, Causing Permanent ckBTC Loss - (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

## Summary

When `retrieve_btc` or `retrieve_btc_with_approval` is called, the user's ckBTC is burned irreversibly before the request is queued. If Bitcoin network fees rise between request acceptance and batch processing, `build_tx` returns `BuildTxError::AmountTooLow`, and the minter finalizes the request via `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` — with no call to `reimburse_canceled_requests`. The user's burned ckBTC is permanently lost with no BTC sent and no reimbursement minted.

## Finding Description

**Step 1 — Burn precedes queuing.**

In `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` lines 209–232, `burn_ckbtcs` is called at line 210 before `accept_retrieve_btc_request` at line 232. The `RetrieveBtcRequest` is constructed with `reimbursement_account: Some(Account { owner: caller, subaccount: None })` (lines 218–221), confirming the infrastructure for reimbursement is present on the request object.

**Step 2 — `AmountTooLow` arm discards without reimbursement.**

In `rs/bitcoin/ckbtc/minter/src/lib.rs` lines 412–434, when `build_tx` returns `BuildTxError::AmountTooLow`, the handler iterates over the batch and calls `state::audit::remove_retrieve_btc_request(s, request, state::FinalizedStatus::AmountTooLow, runtime)` for each request. No call to `reimburse_canceled_requests` is made. The `remove_retrieve_btc_request` function in `rs/bitcoin/ckbtc/minter/src/state/audit.rs` lines 67–80 only records an event and pushes a `FinalizedBtcRequest`; it does not schedule any reimbursement mint.

**Step 3 — Asymmetry with `InvalidTransaction`.**

The immediately preceding arm (lines 400–411) for `BuildTxError::InvalidTransaction` calls `reimburse_canceled_requests(s, batch, reason, reimbursement_fee, runtime)`, crediting users back their burned ckBTC minus a penalty fee. The `AmountTooLow` arm has no equivalent call, despite the `reimbursement_account` being populated on every request.

**Step 4 — The min-amount guard is insufficient.**

`fee_based_retrieve_btc_min_amount` is checked at call time (retrieve_btc.rs lines 169–171) against the fee estimate at that moment. Bitcoin fees are volatile. A request accepted at fee level F can become unprocessable if fees rise to F' > amount before the minter's timer fires. The already-queued request is not re-validated or re-queued; it is permanently discarded.

**Step 5 — `DustOutput` arm has the same gap.**

Lines 436–467 show the `DustOutput` arm also finalizes individual requests as `FinalizedStatus::AmountTooLow` without reimbursement, compounding the exposure.

## Impact Explanation

A user who calls `retrieve_btc` with an amount that passes the minimum check at call time can permanently lose their ckBTC if Bitcoin fees spike before the minter processes the request. The ckBTC ledger burn is irreversible; the minter finalizes the request as `AmountTooLow` and records no reimbursement obligation. The user's `retrieve_btc_status_v2` will show `AmountTooLow` with no recovery path. This constitutes a direct, permanent loss of user funds in an in-scope ck-token/Chain Fusion component, matching the **High** impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."

## Likelihood Explanation

Bitcoin fees are known to spike sharply (e.g., during inscription/ordinal activity). The minter processes requests in batches on a timer, creating a window between burn and BTC transaction construction. `retrieve_btc` is an unprivileged public update call requiring no special access. Any user submitting a request just above the current minimum during a low-fee period is at risk if fees rise before the batch is processed. The scenario is realistic and has historical precedent in Bitcoin fee markets.

## Recommendation

Add a reimbursement path for `BuildTxError::AmountTooLow`, mirroring the `InvalidTransaction` handler in `rs/bitcoin/ckbtc/minter/src/lib.rs`. When a request cannot be fulfilled because the amount is too low to cover fees, call `reimburse_canceled_requests` (or an equivalent) with a new `WithdrawalReimbursementReason` variant (e.g., `FeeTooHigh`), deducting a small penalty fee to cover the cost of the failed attempt. Apply the same fix to the `DustOutput` arm. This ensures users are never left with burned ckBTC and no BTC, consistent with the existing `InvalidTransaction` reimbursement design.

## Proof of Concept

1. Bitcoin fees are currently low; `fee_based_retrieve_btc_min_amount` = 5,000 sat.
2. User calls `retrieve_btc { amount: 5,500, address: "bc1q..." }`. Amount passes the check at retrieve_btc.rs line 169. `burn_ckbtcs` executes at line 210; ckBTC is burned. `RetrieveBtcOk { block_index: N }` is returned.
3. Before the minter's timer fires, Bitcoin fees spike. The minter updates `fee_based_retrieve_btc_min_amount` to 10,000 sat.
4. The minter timer fires, calls `submit_pending_requests`, builds a batch including the 5,500-sat request, then calls `build_tx`. `build_tx` returns `BuildTxError::AmountTooLow` because 5,500 < current fee.
5. The `AmountTooLow` arm at lib.rs line 412 executes: `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)`. No reimbursement is scheduled.
6. User queries `retrieve_btc_status_v2(N)` → `AmountTooLow`. User's ckBTC is gone; no BTC was sent; no reimbursement mint will ever occur.

A deterministic integration test can reproduce this by: (a) initializing the minter with a low fee estimate, (b) calling `retrieve_btc` with an amount just above the minimum, (c) updating the fee estimate to exceed the request amount before the timer fires, (d) triggering the timer, and (e) asserting that `retrieve_btc_status_v2` returns `AmountTooLow` and that no reimbursement event was emitted in the event log.
