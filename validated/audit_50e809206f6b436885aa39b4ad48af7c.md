### Title
Unrestricted `update_rewards(disable_rewards: true)` Allows Anyone to Permanently Deny Block Rewards to All Stakers - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract has no access control and accepts a caller-supplied `disable_rewards` flag. Any unprivileged address can call `update_rewards(any_active_staker, disable_rewards: true)` to mark the current block as processed while suppressing reward distribution, permanently denying block rewards to every staker for that block. This is the direct analog of the `cancelOrder()` front-run: a pending `update_rewards(..., false)` transaction in the mempool reveals the staker's intent to collect rewards, and an attacker front-runs it with `disable_rewards: true` to nullify the payout.

### Finding Description
`update_rewards` is gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — no role check of any kind. [1](#0-0) 

The function enforces a single-call-per-block invariant by writing `last_reward_block` before the `disable_rewards` branch: [2](#0-1) 

When `disable_rewards: true` is passed, execution returns at line 1488 after `last_reward_block` has already been updated to the current block. Any subsequent call in the same block hits the `REWARDS_ALREADY_UPDATED` assertion and reverts. The reward distribution path (`_update_rewards`) is never reached. [3](#0-2) 

The front-running scenario mirrors the original report exactly:

1. Staker (or sequencer) submits `update_rewards(staker_address, false)` to collect block rewards.
2. Attacker observes the pending transaction in the public mempool.
3. Attacker submits `update_rewards(any_active_staker, true)` with higher priority (higher fee).
4. Attacker's transaction lands first: `last_reward_block` is stamped, no rewards flow.
5. Staker's transaction reverts with `REWARDS_ALREADY_UPDATED`.
6. Block rewards for that block are permanently lost.

Because `last_reward_block` is a single contract-wide slot, one attacker call per block is sufficient to deny rewards to **all** stakers for that block. [4](#0-3) 

### Impact Explanation
Repeated execution across consecutive blocks constitutes **permanent freezing of unclaimed yield** for every staker and every delegation pool in the protocol. Rewards are not redirected to the attacker; they are simply never minted/distributed. This matches the High-severity impact tier: *Permanent freezing of unclaimed yield or unclaimed royalties*.

### Likelihood Explanation
Medium. The attack requires one low-cost transaction per block. On Starknet, transaction fees are low, making sustained griefing economically viable for a motivated adversary. No privileged access, leaked key, or external dependency is required — only a non-zero caller address. The attack is also executable without front-running (proactive call at the start of any block), making it independent of mempool visibility.

### Recommendation
Restrict `update_rewards` to a trusted caller (e.g., the sequencer address or a dedicated `REWARDS_MANAGER` role). The `disable_rewards` flag should not be a caller-supplied parameter on a permissionless function. If the intent is to allow the sequencer to skip rewards for non-attesting stakers, enforce that via an on-chain attestation check rather than an unchecked boolean argument.

### Proof of Concept

```
// Block N
// Step 1: Attacker calls with disable_rewards = true
staking.update_rewards(any_active_staker_address, disable_rewards: true);
// → last_reward_block is written to N
// → function returns early, zero rewards distributed

// Step 2: Legitimate staker (or sequencer) attempts reward collection
staking.update_rewards(staker_address, disable_rewards: false);
// → PANICS: current_block_number (N) is NOT > last_reward_block (N)
// → Error::REWARDS_ALREADY_UPDATED

// Repeat for every block → all stakers permanently denied block rewards
``` [5](#0-4)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
