### Title
COA Ownership Proof Signature Replayable Across Flow Networks Due to Chain-Agnostic Domain Separation Tag - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over arbitrary caller-supplied `signedData` using the hardcoded domain separation tag `"FLOW-V0.0-user"`. This tag is identical across every Flow network (mainnet, testnet, previewnet). Because neither the tag nor the signed payload is required to encode a chain ID, a valid COA ownership proof signature produced on one Flow network is cryptographically valid on any other Flow network where the same Cadence key exists. The COA's ERC-1271 `isValidSignature` entry point is directly reachable by any unprivileged EVM transaction sender, making cross-network replay exploitable without any privileged access.

---

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` calls `Crypto.KeyList.verify` with a fixed domain separation tag:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"   // ← identical on every Flow network
)
``` [1](#0-0) 

The `signedData` parameter is arbitrary bytes supplied by the caller; the function explicitly does not enforce that it encodes the EVM address or any chain-specific context:

> "Note: this function does not enforce that `signedData` includes `evmAddress`." [2](#0-1) 

This function is invoked by the Cadence Arch precompile's `verifyCOAOwnershipProof` handler:

```go
func coaOwnershipProofValidator(...) func(proof *types.COAOwnershipProofInContext) (bool, error) {
    return func(proof *types.COAOwnershipProofInContext) (bool, error) {
        value, err := backend.Invoke(... "validateCOAOwnershipProof" ...)
``` [3](#0-2) 

The EVM-side entry point is the COA contract's ERC-1271 `isValidSignature`, callable by any unprivileged EVM transaction:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig));
``` [4](#0-3) 

The `_hash` passed to `isValidSignature` becomes the `signedData` verified by `validateCOAOwnershipProof`. There is no protocol-level enforcement that this hash encodes the Flow EVM chain ID (`747` for mainnet, `545` for testnet, `646` for previewnet). [5](#0-4) 

---

### Impact Explanation

A Cadence account key pair reused across Flow Mainnet and Flow Testnet (a common developer practice) produces identical signatures for the same `signedData` under the same domain tag. An attacker who observes a valid COA ownership proof signature on testnet — e.g., from a public transaction or off-chain authentication flow — can replay it verbatim on mainnet. The replayed proof passes `validateCOAOwnershipProof` and causes the COA's `isValidSignature` to return `ValidERC1271Signature` (ERC-1271 success) for a hash the account owner never authorized on mainnet. Any EVM contract on Flow Mainnet that relies on ERC-1271 to gate asset transfers, approvals, or privileged actions against a COA is therefore bypassable without the COA owner's knowledge or consent. This constitutes unauthorized transaction authorization — a direct cross-VM asset-loss or access-control bypass depending on the consuming contract.

---

### Likelihood Explanation

The preconditions are realistic:

1. **Same key on multiple networks**: Developers routinely generate one key pair and fund accounts on both testnet and mainnet. Flow's account model does not prevent this.
2. **Observable signature**: COA ownership proofs are submitted as EVM transaction calldata, which is public on-chain. Any proof submitted on testnet is visible to an attacker.
3. **COA on both networks**: A user who deploys a COA-backed dApp on testnet before mainnet will have COAs on both networks.
4. **No attacker privilege required**: The replay is submitted as an ordinary EVM transaction calling `isValidSignature` on the mainnet COA contract.

---

### Recommendation

Include the Flow EVM chain ID in the signed message. The simplest approach is to prepend the chain ID to `signedData` inside `validateCOAOwnershipProof` before verification, or to encode it into the domain separation tag (e.g., `"FLOW-V0.0-user-747"` for mainnet). The chain ID is already available via `EVMChainIDFromFlowChainID`: [6](#0-5) 

Alternatively, enforce at the protocol level that `signedData` must encode both the EVM address and the chain ID, and validate this inside `validateCOAOwnershipProof` rather than leaving it as a caller convention.

---

### Proof of Concept

**Setup**: Alice has the same ECDSA key at index 0 on both Flow Testnet and Flow Mainnet. She has a COA at `0xCOA_TESTNET` on testnet and `0xCOA_MAINNET` on mainnet.

**Step 1 — Testnet**: Alice signs `H = keccak256("authorize_action")` with her key under tag `"FLOW-V0.0-user"` and submits a COA ownership proof on testnet. The proof is recorded in public testnet transaction calldata.

**Step 2 — Attacker observes**: The attacker extracts `(H, sig, keyIndex=0, flowAddress, capabilityPath="coa")` from the testnet transaction.

**Step 3 — Mainnet replay**: The attacker calls `isValidSignature(H, encodedProof)` on `0xCOA_MAINNET` on Flow Mainnet, where `encodedProof` is the same RLP-encoded `COAOwnershipProof` from testnet.

**Step 4 — Verification passes**: `validateCOAOwnershipProof` fetches Alice's mainnet key (same public key), verifies `sig` over `H` under `"FLOW-V0.0-user"` — which succeeds because the domain tag and key are identical — and returns `isValid: true`.

**Step 5 — Impact**: `isValidSignature` returns `0x1626ba7e` (ERC-1271 valid). Any mainnet EVM contract that checks `COA.isValidSignature(H, ...)` to gate an action now treats Alice as having authorized that action on mainnet, which she never did. [1](#0-0) [4](#0-3)

func EVMChainIDFromFlowChainID(flowChainID flow.ChainID) *big.Int {
	// default evm chain ID is previewNet
	switch flowChainID {
	case flow.Mainnet:
		return FlowEVMMainNetChainID
	case flow.Testnet:
		return FlowEVMTestNetChainID
	default:
		return FlowEVMPreviewNetChainID
	}
}
