### Title
Malicious Coordinator Extracts Aggregate Secret Key via Presignature Reuse Across Parallel Signing Sessions — (`src/ecdsa/robust_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

The Robust ECDSA signing protocol is entirely coordinator-centric: each participant computes a signature share and sends it **only** to the coordinator, with no in-protocol step to verify that all participants are operating on the same `(msg_hash, tweak, presignature)`. A malicious coordinator can initiate two parallel signing sessions using the same `PresignOutput` with different `(h, ε)` pairs. Because participants never cross-verify the signing context, the coordinator collects two sets of shares derived from the same nonce material, yielding two signatures with multiplicatively related nonces, from which the aggregate secret key can be recovered via standard ECDSA nonce-reuse techniques.

---

### Finding Description

**Root cause — no in-protocol consensus on signing context.**

In `do_sign_participant()`, each non-coordinator participant computes its share and sends it exclusively to the coordinator:

```rust
// src/ecdsa/robust_ecdsa/sign.rs  lines 118-123
fn do_sign_participant(...) -> Result<SignatureOption, ProtocolError> {
    let s_me = compute_signature_share(presignature, msg_hash, participants, me)?;
    let wait_round = chan.next_waitpoint();
    chan.send_private(wait_round, coordinator, &s_me)?;   // ← only to coordinator
    Ok(None)
}
```

There is no broadcast round in which participants exchange `(msg_hash, big_r)` to confirm they are all operating on the same signing context. [1](#0-0) 

**Root cause — rerandomization is performed locally with caller-supplied parameters.**

`RerandomizedPresignOutput::rerandomize_presign()` is called independently by each participant with locally-provided `RerandomizationArguments`. The library imposes no constraint preventing the same `PresignOutput` from being rerandomized twice with different `(msg_hash, tweak)`:

```rust
// src/ecdsa/robust_ecdsa/mod.rs  lines 54-86
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,   // caller-supplied; no uniqueness check
) -> Result<Self, ProtocolError> { ... }
``` [2](#0-1) 

**The security documentation acknowledges the constraint but the code does not enforce it.**

`docs/ecdsa/robust_ecdsa/signing.md` states:

> "Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers."

and

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [3](#0-2) 

The `sign()` initializer enforces `N = 2t+1` and `msg_hash ≠ 0`, but it does **not** enforce that the same `PresignOutput` is used at most once, nor that all participants agreed on the same `(h, ε)` before contributing their shares:

```rust
// src/ecdsa/robust_ecdsa/sign.rs  lines 84-95
// The next two conditions prevent split-view attacks
if participants.len() != robust_ecdsa_threshold { ... }   // enforced ✓
if bool::from(msg_hash.is_zero()) { ... }                 // enforced ✓
// "Ensure all participants agree on (h, ε)" — NOT enforced ✗
``` [4](#0-3) 

The `RerandomizationArguments` struct is fully public and caller-constructed; nothing in the library prevents a coordinator from supplying different `(msg_hash, tweak)` to different participants or across two parallel sessions:

```rust
// src/ecdsa/mod.rs  lines 94-103
pub struct RerandomizationArguments {
    pub pk: AffinePoint,
    pub tweak: Tweak,
    pub msg_hash: [u8; 32],
    pub big_r: AffinePoint,
    pub participants: ParticipantList,
    pub entropy: [u8; 32],
}
``` [5](#0-4)

### Impact Explanation

**Critical — extraction of the aggregate secret key.**

A malicious coordinator collects two valid signatures `(R1, s1)` and `(R2, s2)` where `R1 = δ1·R` and `R2 = δ2·R` (both derived from the same presignature nonce `R`). The nonces are multiplicatively related: `R1 = (δ1/δ2)·R2`. Standard ECDSA nonce-reuse recovery then yields the aggregate secret key `x`, enabling the coordinator to forge arbitrary signatures for any message under the shared key.

This falls squarely within the allowed Critical impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**High.** The coordinator is a normal participant role — no privileged access is required. Any participant who is selected as coordinator (e.g., by random selection in the orchestration layer) can execute this attack. The attack requires only two signing sessions using the same presignature, which is a realistic scenario in any system that pre-generates presignatures in bulk. The library provides no API-level guard against it.

---

### Recommendation

1. **Add an in-protocol consensus round.** Before participants send their shares to the coordinator, have each participant broadcast a commitment to `(msg_hash, big_r, tweak)` via echo broadcast. Each participant verifies that all others committed to the same values before proceeding. This mirrors the `lastChangeBlock` mitigation in the AeraVault report — checking that the state has not been manipulated before executing the sensitive operation.

2. **Enforce single-use presignatures at the API level.** Wrap `PresignOutput` in a consuming type (e.g., move semantics enforced by the type system) so that `rerandomize_presign` takes ownership and the original cannot be reused. Alternatively, add a runtime token/nonce that is invalidated on first use.

3. **Document the caller obligation more prominently.** At minimum, add a `#[must_use]` annotation and a compile-time or runtime assertion that the same `PresignOutput` is not passed to `rerandomize_presign` more than once.

### Proof of Concept

```
Setup: t = 1, participants = {P1 (malicious coordinator), P2, P3}
       N = 2t+1 = 3 (satisfies the enforced constraint)

Step 1 — Presigning:
  All three parties run presign() → PresignOutput { big_r: R, alpha_i, beta_i, c_i, e_i }

Step 2 — Session A (coordinator presents h1, ε1):
  P1 tells P2 and P3: "sign message h1 with tweak ε1 using presignature R"
  Each Pi calls rerandomize_presign(presign_out, args{h1, ε1}) → RerandomizedPresignOutput_A
  P2, P3 compute s_i_A and send to P1 (coordinator)
  P1 assembles signature (R1, s_A) where R1 = δ1·R  [valid for h1 under derived key Y1]

Step 3 — Session B (coordinator presents h2, ε2, same presignature):
  P1 tells P2 and P3: "sign message h2 with tweak ε2 using presignature R"
  Each Pi calls rerandomize_presign(presign_out, args{h2, ε2}) → RerandomizedPresignOutput_B
  P2, P3 compute s_i_B and send to P1
  P1 assembles signature (R2, s_B) where R2 = δ2·R  [valid for h2 under derived key Y2]

Step 4 — Key recovery:
  P1 observes: R1 = δ1·R, R2 = δ2·R  →  R1 = (δ1/δ2)·R2  (multiplicatively related nonces)
  P1 applies ECDSA nonce-reuse recovery:
    s_A = (h1·k + Rx·x) / δ1   (simplified)
    s_B = (h2·k + Rx·x) / δ2
  Two equations, two unknowns (k, x) → P1 recovers aggregate secret key x.

P2 and P3 detect nothing: they each sent one share per session to the coordinator,
with no cross-participant verification of the signing context.
```

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
