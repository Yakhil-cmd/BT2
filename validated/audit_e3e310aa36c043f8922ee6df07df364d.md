### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Disable Mechanism Never Enforced - (`File: x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` message handler in the Cronos keeper is completely unimplemented. It accepts any `MsgTurnBridge` transaction, returns success, but performs no state change. As a result, the bridge can never actually be disabled: any unprivileged user can continue to bridge CRC20/CRC21 tokens to Ethereum via `send_to_evm_chain` even after an admin has "turned off" the bridge.

### Finding Description

The `TurnBridge` gRPC message is defined in the Cronos module to allow an authorized admin to enable or disable the Gravity bridge. The proto definition and CLI wiring are complete: [1](#0-0) [2](#0-1) 

However, the actual server-side handler in `msg_server.go` is a stub that does nothing: [3](#0-2) 

The function body is `return nil, nil` — no permission check, no parameter update, no state write. The `enable` field from `MsgTurnBridge` is silently discarded.

The integration test `test_gravity_turn_bridge` explicitly documents the intended behavior: after `turn_bridge("false")` succeeds, a subsequent `send_to_evm_chain` call on a CRC21 contract **must fail**: [4](#0-3) 

Because the handler is a no-op, the bridge-enabled state is never written, and the EVM hook that processes `__CronosSendToEvmChain` log events never sees a disabled state. Token bridge operations proceed unimpeded regardless of admin intent.

The `send_to_evm_chain` function on the canonical `ModuleCRC21` contract burns or transfers tokens and emits the bridge event: [5](#0-4) 

The EVM hook dispatcher in `evm_hooks.go` processes these logs post-transaction with no bridge-enabled guard: [6](#0-5) 

### Impact Explanation

**High — Bypass of Cronos admin bridge authorization check.**

An admin who calls `MsgTurnBridge{enable: false}` receives a success response but achieves nothing. Any unprivileged user can immediately call `send_to_evm_chain` on any CRC21 contract and successfully initiate a bridge transfer to Ethereum. The admin's bridge-control authority is permanently bypassed for as long as the no-op handler is deployed. This directly matches the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation

The `TurnBridge` message is a documented, user-facing transaction with a CLI command and integration test. Any operator who relies on it to halt bridge activity during an incident or upgrade will find the bridge remains fully operational. The bypass requires no special knowledge — any user simply continues calling `send_to_evm_chain` normally.

### Recommendation

Implement `TurnBridge` to:
1. Verify the caller holds the `CanTurnBridge` permission (consistent with how `UpdateTokenMapping` checks `CanChangeTokenMapping`).
2. Write the `enable` flag into the module's `Params` (e.g., a `BridgeEnabled` field).
3. Ensure the EVM hook handler for `__CronosSendToEvmChain` reads this parameter and returns an error when the bridge is disabled, causing the EVM transaction to revert.

### Proof of Concept

1. Admin submits `MsgTurnBridge{sender: admin, enable: false}` — transaction succeeds with code 0.
2. Unprivileged user calls `crc21.send_to_evm_chain(recipient, amount, chain_id, 0, "")` — transaction succeeds and tokens are burned/transferred, emitting `__CronosSendToEvmChain`.
3. The Gravity EVM hook processes the event and queues the outbound transfer to Ethereum.
4. The bridge was never actually disabled; the admin's action had no effect.

This is directly analogous to the external report: just as `PhiNFT1155` inherits `PausableUpgradeable` but the pause state is never wired into the ERC1155 transfer path, Cronos exposes `TurnBridge` but the bridge-enabled state is never written or checked — the control mechanism exists in the interface but is entirely absent from the execution path.

// CmdTurnBridge returns a CLI command handler for enable or disable the bridge
func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{
		Use:   "turn-bridge [true/false]",
		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
