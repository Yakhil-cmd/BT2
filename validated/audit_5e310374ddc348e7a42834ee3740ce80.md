### Title
Blocklist EVM Destination Check Bypassed via Contract Internal Calls — (File: app/proposal.go)

### Summary
The Cronos encrypted blocklist enforces a destination-address check on direct EVM transactions (`ethTx.To()`), but this check is never applied to EVM-internal (contract-to-contract) calls. Any unprivileged, non-blocked address can route a token transfer to a blocked address through a smart-contract intermediary, fully bypassing the block-list authorization check.

### Finding Description
`ProposalHandler.ValidateTransaction` in `app/proposal.go` iterates over every message in a transaction and, for `MsgEthereumTx`, checks whether `ethTx.To()` is in the blocklist:

```go
for _, msg := range tx.GetMsgs() {
    msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
    if ok {
        ethTx := msgEthTx.AsTransaction()
        if ethTx.To() != nil {
            if _, ok := h.blocklist[encoded]; ok {
                return fmt.Errorf("destination address is blocked: %s", encoded)
            }
        }
    }
}
``` [1](#0-0) 

The identical check is replicated in `BlockAddressesDecorator.AnteHandle` (which runs only during `ctx.IsCheckTx()`): [2](#0-1) 

Both checks inspect **only the top-level `To` field** of the Ethereum transaction. Neither check is applied to EVM internal calls generated during contract execution. When a non-blocked address sends an EVM transaction whose `To` is a contract (not the blocked address), the blocklist check passes unconditionally. If that contract then calls `transfer` on any ERC-20, invokes the bank precompile, or calls another contract that ultimately credits the blocked address, the transfer executes without any blocklist validation.

The `ProcessProposalHandler` calls `ValidateTransaction` for every transaction in the proposed block and would reject the block only if the direct `To` is blocked — it has no visibility into EVM execution traces: [3](#0-2) 

The bank precompile's `checkBlockedAddr` only checks the Cosmos SDK module-account blocklist (`bankKeeper.BlockedAddr`), not the e2ee blocklist: [4](#0-3) 

### Impact Explanation
This is a **bypass of block-list authorization checks** (High). The blocklist explicitly guards EVM destination addresses to prevent blocked addresses from receiving EVM-layer assets. The internal-call path circumvents that guard entirely, allowing a blocked address to accumulate CRO, CRC20/CRC21 tokens, or any EVM-managed asset through a contract intermediary. The blocked address retains the received balance and can act on it once the blocklist entry is removed or through pre-deployed contract logic.

### Likelihood Explanation
High. No special privilege is required. Any non-blocked address can deploy (or reuse) a trivial forwarding contract. The blocked address can arrange this before being added to the blocklist. The attack requires only a single EVM transaction from a non-blocked sender.

### Recommendation
Enforce the blocklist check at EVM execution time, not only at the transaction-routing layer. Options include:
1. Add a post-execution hook (e.g., in `LogProcessEvmHook` or a new `PostTxProcessing` hook) that inspects all EVM state-change recipients against the blocklist and reverts if a blocked address received funds.
2. Alternatively, extend `ValidateTransaction` to also reject transactions whose `To` is a contract known to forward to blocked addresses — though this is incomplete without execution-time enforcement.

### Proof of Concept
1. `Alice` is added to the encrypted e2ee blocklist.
2. Before or after being blocked, `Alice` (or an accomplice `Bob`) deploys contract `C`:
   ```solidity
   function forward(address token, address to, uint256 amt) external {
       IERC20(token).transferFrom(msg.sender, to, amt);
   }
   ```
3. `Bob` (non-blocked) approves `C` to spend his tokens, then calls `C.forward(tokenAddr, Alice, 1000)`.
4. `ProposalHandler.ValidateTransaction` checks: signer = `Bob` (not blocked), `To` = `C` (not blocked) → **passes**.
5. The block is accepted; during EVM execution `C` transfers 1000 tokens to `Alice`.
6. `Alice` receives 1000 tokens despite being on the blocklist — the destination check is bypassed.

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}

func (bc *BankContract) checkBlockedAddr(addr sdk.AccAddress) error {
	to, err := sdk.AccAddressFromBech32(addr.String())
	if err != nil {
		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {
		return errorsmod.Wrapf(errortypes.ErrUnauthorized, "%s is not allowed to receive funds", to.String())
	}
	return nil
}
