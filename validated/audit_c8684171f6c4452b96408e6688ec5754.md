### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` Allows Threshold=1 to Corrupt Signing Outputs - (File: src/frost/mod.rs)

---

### Summary
`assert_sign_inputs` in `src/frost/mod.rs` validates the threshold only against an upper bound (`threshold > participants.len()`), but never enforces a lower bound of `>= 2`. A malicious coordinator or caller can pass `threshold = 1` (or `0`), which passes all validation checks. When the FROST signing protocol then runs with this extreme low threshold, Lagrange interpolation uses wrong coefficients, producing a silently corrupt and unusable signature that honest parties accept as a completed protocol run.

---

### Finding Description

Every other threshold-bearing entry point in the codebase enforces `threshold >= 2`:

- `assert_key_invariants` (DKG): [1](#0-0) 
- `validate_triple_inputs` (OT-ECDSA triple generation): [2](#0-1) 

However, `assert_sign_inputs` in the FROST module only checks the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [3](#0-2) 

There is no corresponding lower-bound check (`threshold < 2`). The same omission exists in the FROST `presign` function:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` newtype with no internal invariant enforcing a minimum: [5](#0-4) 

So `threshold = 1` (or `0`) is a valid value that passes all checks in `assert_sign_inputs`.

---

### Impact Explanation

**threshold = 1**: The FROST signing protocol proceeds without error. Each participant computes their signature share using Lagrange coefficients derived from a 1-of-N interpolation. With only one evaluation point, the Lagrange coefficient is trivially `1`, so the "reconstructed" secret is just one participant's raw share — not the actual aggregate private key. The aggregated signature is cryptographically invalid and unusable. Honest parties complete the protocol and receive a corrupt output with no indication of failure.

**threshold = 0**: `compute_lagrange_coefficient` is called with an empty point set. The polynomial interpolation code rejects this with `ProtocolError::InvalidInterpolationArguments` (enforced at `n <= 1`): [6](#0-5)  This causes a hard protocol error, permanently denying signing for all honest participants in that session.

Both outcomes match allowed High impacts:
- *Corruption of sign outputs so honest parties accept unusable cryptographic outputs* (threshold=1 case).
- *Permanent denial of signing for honest parties under valid protocol inputs* (threshold=0 case).

---

### Likelihood Explanation

`assert_sign_inputs` is a **public** library function. Any caller — including a malicious coordinator who is also a participant — can invoke the FROST EdDSA or RedJubjub signing entry points with `threshold = 1`. No special privilege, leaked key, or external assumption is required. The `ReconstructionLowerBound` type accepts any `usize`, so the caller simply passes `1usize.into()`. The attack is trivially reachable from the public API surface.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and the FROST `presign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `participants.len() < 2` check in both functions, consistent with the pattern already established in `src/dkg.rs` and `src/ecdsa/ot_based_ecdsa/triples/generation.rs`.

---

### Proof of Concept

1. Run DKG with `participants = [P0, P1, P2]`, `threshold = 2` (valid). All parties receive correct key shares.
2. A malicious coordinator calls `assert_sign_inputs(&[P0, P1, P2], 1u32, P0, P0)`.
3. The function returns `Ok(participants)` — no error, because `1 <= 3` and there is no lower-bound check. [7](#0-6) 
4. The coordinator launches the FROST signing protocol with `threshold = 1`.
5. Each participant computes their signature share with Lagrange coefficient `= 1` (single-point interpolation).
6. The coordinator aggregates the shares. The result is a signature over the wrong "secret" (one participant's raw share, not the aggregate key).
7. The protocol completes successfully from all honest parties' perspective, but the output signature fails external verification — a silently corrupt, unusable signing output.
