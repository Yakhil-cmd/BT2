### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Global Spread Fee Cap System - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap validation. The factory enforces hard caps on global spread fees (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, 20%) and admin spread fees (`maxAdminSpreadFeeE6`), but the per-bin additional fee path has no analogous guard. A pool admin can set per-bin fees up to `uint16.max = 65535` (6.5535% in E6 units) on any bin, making the effective swap fee on that bin exceed the protocol-enforced ceiling.

### Finding Description
The factory's `setPoolAdminFees` correctly validates that `newAdminSpreadFeeE6 ≤ maxAdminSpreadFeeE6` before updating the global fee: [1](#0-0) 

However, `setPoolBinAdditionalFees` performs no such check — it forwards the caller-supplied values straight to the pool: [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes: [3](#0-2) 

These per-bin values are then added directly to the base fee in every swap iteration: [4](#0-3) [5](#0-4) 

The hard cap constants that bound every other fee path are: [6](#0-5) 

`uint16.max = 65535` in E6 units equals 6.5535%. Combined with a global spread fee already at the 20% hard cap, the effective fee on a targeted bin reaches ~26.5%, silently exceeding the protocol ceiling with no revert.

### Impact Explanation
Traders routing through a bin whose `addFeeBuyE6` or `addFeeSellE6` has been set to the `uint16` maximum pay a fee that exceeds the protocol-enforced hard cap. This is bad-price execution: the pool extracts more from the trader than the oracle/bin curve and the factory's fee cap system permit. The excess fee accrues as LP surplus and is later swept by the admin via `collectPoolFees`, constituting a direct loss of trader principal above the intended protocol ceiling.

### Likelihood Explanation
Any pool admin — a role that is semi-trusted only within the cap and timelock system — can call `setPoolBinAdditionalFees` at any time with no timelock and no cap guard. The trigger requires no special privilege beyond holding the `poolAdmin` role, which is assigned at pool creation and transferable.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` (and mirror it in `setBinAdditionalFees` as a defence-in-depth guard) that rejects `addFeeBuyE6` or `addFeeSellE6` values exceeding `maxAdminSpreadFeeE6` (or a dedicated per-bin hard cap constant). The same check should be applied when unpacking bin data in `_unpackAndValidateBinStates` at pool creation time.

```solidity
// In setPoolBinAdditionalFees (factory):
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
```

### Proof of Concept
1. Deploy a pool with `adminSpreadFeeE6 = 0` (within cap) and `maxAdminSpreadFeeE6 = 200_000`.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs — `65535 > 200_000` is never checked.
4. A trader swapping through bin 0 now pays `baseFee + 6.5535%` additional spread, exceeding the 20% hard cap.
5. The excess fee accumulates as pool surplus and is collected by the admin, extracting value from traders beyond the protocol-permitted ceiling.

  function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    onlyFactory
    nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
  {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
  }
