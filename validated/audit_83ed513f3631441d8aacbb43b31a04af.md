### Title
Missing Domain Separation in `validateCOAOwnershipProof` Allows Cross-COA Signature Replay - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` commits to the `evmAddress` being proven. Because both the signed payload and the target COA address are free parameters supplied by the caller, a valid signature produced for one COA can be replayed to prove ownership of a different COA belonging to the same Cadence account. This is the direct Flow analog of the Nomic commitment-format collision: two "destination" types share the same byte-level representation with no type prefix, so a relayer/caller can substitute one for the other.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six caller-controlled arguments:

```
address: Address          // Cadence account
path: PublicPath          // where the COA capability is stored
signedData: [UInt8]       // arbitrary bytes the account keys signed
keyIndices: [UInt64]
signatures: [[UInt8]]
evmAddress: [UInt8; 20]   // the EVM address being "proven"
```

The function verifies that the account's keys produced valid signatures over `signedData` under the `"FLOW-V0.0-user"` domain tag, then checks that the COA stored at `path` has address `evmAddress`. Crucially, it never checks that `signedData` encodes or commits to `evmAddress`.

The code itself acknowledges this gap:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

Because `path` and `evmAddress` are both attacker-supplied and are never bound to the signed payload, an attacker who observes a valid proof `(signedData=D, sig, path=/public/coa1, evmAddress=COA_1)` can construct a new call `(signedData=D, sig, path=/public/coa2, evmAddress=COA_2)` that passes all checks, provided the target account has a second COA at `/public/coa2`.

The precompile entry point `verifyCOAOwnershipProof(address, bytes32, bytes)` in the Cadence Arch contract is callable by any unprivileged EVM transaction. The ERC-1271 `isValidSignature` implementation in `coa.sol` delegates directly to this precompile, making it the primary consumer of forged proofs.

### Impact Explanation

An EVM contract that gates access using `verifyCOAOwnershipProof` (e.g., via ERC-1271 `isValidSignature`) can be deceived into accepting a proof for `COA_2` when the account holder only authorized `COA_1`. If the two COAs have different privilege levels or asset balances in the EVM contract, the attacker gains unauthorized access to the higher-privilege COA's resources — a cross-VM authentication bypass and potential asset loss.

### Likelihood Explanation

The attack requires the target Cadence account to have published capabilities for at least two COAs at different public paths, and the attacker must have observed a valid proof for one of them. While multi-COA accounts are not the current norm, the protocol does not prevent them, and the `validateCOAOwnershipProof` function is `access(all)` and callable from any Cadence script or EVM transaction. The precompile is reachable by any unprivileged EVM sender with no staking or special role required.

### Recommendation

- **Short term**: Enforce inside `validateCOAOwnershipProof` that `signedData` is a commitment that includes `evmAddress` (e.g., require `signedData == SHA3_256(evmAddress ++ userPayload)` or reject calls where `signedData` does not contain the 20-byte `evmAddress` as a prefix/suffix).
- **Long term**: Define and document a canonical signed-message format for COA ownership proofs that unambiguously binds the proof to a single EVM address, analogous to EIP-712 typed structured data. Provide a decoding routine that can be used in unit tests to verify injectivity.

### Proof of Concept

1. Cadence account `A` creates two COAs and publishes them:
   - `COA_1` at `/public/coa1` with EVM address `0xAAAA...`
   - `COA_2` at `/public/coa2` with EVM address `0xBBBB...`

2. Account `A` signs 32 bytes of data `D` with their Flow key to prove ownership of `COA_1` to some EVM contract, producing signature `sig`.

3. An attacker observes `(D, sig)` on-chain.

4. The attacker calls the Cadence Arch precompile from an EVM transaction:
   ```
   verifyCOAOwnershipProof(
     0xBBBB...,   // COA_2's EVM address
     D,           // same signed data
     rlp_encode({ KeyIndices: [0], Address: A, CapabilityPath: "coa2", Signatures: [sig] })
   )
   ```

5. Inside `validateCOAOwnershipProof`:
   - Signature verification over `D` with account `A`'s key → **passes** (same key, same data).
   - COA at `/public/coa2` has address `0xBBBB...` → **passes**.
   - Returns `ValidationResult(isValid: true)`.

6. The EVM contract believes account `A` authorized an action for `COA_2`, which `A` never signed for.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
