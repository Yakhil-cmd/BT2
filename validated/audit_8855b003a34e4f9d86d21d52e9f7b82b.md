### Title
Unauthenticated Invoke Admission via `skip_stateful_validations` Signature-Check Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point (the account's signature check) for any invoke transaction with `nonce = 1` sent to an account whose address appears in the mempool — regardless of who submitted that address. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a spoofed invoke with `sender_address = victim`, `nonce = 1`, and an arbitrary or empty signature. The gateway accepts it without running `__validate__`, admitting an unauthenticated transaction into the mempool.

---

### Finding Description

**Trigger condition in `skip_stateful_validations`**

```
crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
```

The function returns `true` (skip validation) when all three conditions hold:

1. The incoming transaction is `ExecutableTransaction::Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [1](#0-0) 

**What `account_tx_in_pool_or_recent_block` actually checks**

```rust

``` [2](#0-1) 

This returns `true` for **any** account that has ever had a transaction in the pool — it does not verify that the transaction is a `deploy_account` owned by the same signer as the incoming invoke. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but that reasoning only holds for the legitimate account owner. It does not prevent a third party from submitting an invoke for the same `sender_address`.

**How the skip propagates to disable `__validate__`**

When `skip_validate = true`, `run_validate_entry_point` constructs the `AccountTransaction` with `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside the blockifier's `ValidatableTransaction::validate_tx`, the very first check is:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [4](#0-3) 

The `__validate__` entry point — the account contract's signature verification — is never called. The mempool's own `validate_tx` only checks for duplicate hashes and nonce ordering; it performs no cryptographic check:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

---

### Impact Explanation

An attacker can inject an invoke transaction bearing an invalid or empty signature for any victim account whose `deploy_account` is pending in the mempool. The gateway admits it without signature verification. Concrete consequences:

- **Mempool pollution / DoS**: The attacker can flood the mempool with signature-less invokes targeting every newly-seen `deploy_account`, consuming mempool capacity and degrading throughput for legitimate users.
- **Transaction displacement**: If the victim also submits a legitimate invoke with `nonce = 1`, the attacker's transaction can displace it via fee escalation (`validate_fee_escalation`), causing the victim's transaction to be evicted.
- **Wasted block space**: The batcher uses `new_for_sequencing` (which sets `validate: true`) when executing transactions, so the attacker's transaction will revert on-chain — but it still occupies a slot in the block and consumes gas accounting resources.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."* [6](#0-5) 

---

### Likelihood Explanation

- **No privilege required**: Any network participant can submit RPC transactions.
- **Observable trigger**: `deploy_account` transactions are visible in the mempool before they are included in a block.
- **Trivial to craft**: The attacker only needs to copy the victim's `sender_address`, set `nonce = 1`, and submit an invoke with an empty signature array. All other fields (resource bounds, calldata) are attacker-controlled.
- **Condition window**: The window is open from the moment the victim's `deploy_account` enters the mempool until it is committed to a block.

---

### Recommendation

The skip should be restricted to the account owner. The simplest fix is to verify that the account has a **`deploy_account`** transaction specifically in the pool (not just any transaction), and that the incoming invoke's `sender_address` matches that deploy_account's derived contract address. Alternatively, require that the incoming invoke passes a lightweight off-chain signature pre-check (e.g., ECDSA verification against the class hash's known validation scheme) before the skip is granted. At minimum, `account_tx_in_pool_or_recent_block` should be replaced with a stricter `has_pending_deploy_account(address) -> bool` query that only returns `true` when a `deploy_account` transaction for that exact address is present in the pool.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
