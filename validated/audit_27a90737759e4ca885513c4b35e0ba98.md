### Title
Asymmetric Minimum-Amount Enforcement Between EVM-Side Deposits and COA Withdrawal Permanently Traps Sub-Minimum FLOW Balances - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

The `CadenceOwnedAccount.withdraw` path enforces a hard minimum of 1e10 attoFlow (the smallest UFix64-representable unit), but EVM-native transfers can credit arbitrary attoFlow amounts—including amounts below this minimum—directly to a COA's EVM address. Any sub-1e10 attoFlow balance accumulated in a COA through EVM-side transfers is permanently irrecoverable to Cadence. This is the structural analog of the Taiko quota-manager issue: a check enforced on the withdrawal path is absent on the deposit path, causing funds to be permanently stuck.

---

### Finding Description

**Deposit path — no minimum enforced:**

`EVMAddress.deposit` accepts any `@FlowToken.Vault`: [1](#0-0) 

Because `FlowToken.Vault` uses `UFix64`, the minimum Cadence-originated deposit is exactly 1e10 attoFlow (0.00000001 FLOW). However, EVM-native transactions (regular EVM EOA or contract calls) can send **any** attoFlow amount directly to a COA's EVM address, bypassing the Cadence `deposit` function entirely. The `InternalEVM.deposit` host function in Go has no minimum-amount guard: [2](#0-1) 

**Withdrawal path — minimum enforced:**

`CadenceOwnedAccount.withdraw` calls `InternalEVM.withdraw`, which in `newInternalEVMTypeWithdrawFunction` checks `AttoFlowBalanceIsValidForFlowVault`: [3](#0-2) 

`AttoFlowBalanceIsValidForFlowVault` requires the amount to be **≥ 1e10 attoFlow**: [4](#0-3) 

If the requested withdrawal amount is below 1e10 attoFlow, the call panics with `ErrWithdrawBalanceRounding`. This is confirmed by the existing test: [5](#0-4) 

The same check exists at the emulator level: [6](#0-5) 

**The asymmetry:**

| Path | Minimum enforced? |
|---|---|
| Cadence → COA (`EVMAddress.deposit`) | No (UFix64 floor is 1e10 attoFlow, but EVM-native transfers bypass this) |
| COA → Cadence (`CadenceOwnedAccount.withdraw`) | Yes — hard panic if amount < 1e10 attoFlow |

The `CadenceOwnedAccount.deposit` wrapper simply delegates to `EVMAddress.deposit` with no additional guard: [7](#0-6) 

---

### Impact Explanation

Any attoFlow balance below 1e10 in a COA account—whether from a deliberate EVM-side transfer or from EVM gas-fee remainders that leave a non-UFix64-aligned residue—**cannot be withdrawn to Cadence and is permanently stuck**. The COA owner can withdraw the largest multiple of 1e10 attoFlow that fits their balance, but the sub-minimum remainder is irrecoverable. This is cross-VM asset loss: FLOW tokens that exist in the EVM state are permanently inaccessible from the Cadence side.

---

### Likelihood Explanation

Low-to-medium. Any EVM EOA or smart contract can send an arbitrary attoFlow amount to a COA's EVM address. This can happen:
- Deliberately (attacker sends 1 attoFlow to a victim's COA to permanently trap a residue)
- Naturally (EVM gas fees or EVM contract interactions leave a non-UFix64-aligned balance)

The financial impact per incident is small (< 1e-8 FLOW), but the mechanism is permanent and requires no privileged access.

---

### Recommendation

1. In `InternalEVM.deposit` (or `EVMAddress.deposit`), reject or round deposits that would create a sub-1e10 attoFlow balance component, mirroring the withdrawal guard.
2. Alternatively, provide a COA-level `sweep` or `burn` function that allows the owner to explicitly discard a sub-minimum residue, unblocking future full withdrawals.
3. At minimum, document that EVM-native transfers of sub-1e10 attoFlow amounts to COA addresses result in permanently irrecoverable funds.

---

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Step 1: Create COA and deposit 1.0 FLOW from Cadence
        let coa <- EVM.createCadenceOwnedAccount()
        let admin = account.storage.borrow<&FlowToken.Administrator>(from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 1.0)
        let vault <- minter.mintTokens(amount: 1.0)
        destroy minter
        coa.deposit(from: <-vault)
        // COA balance: 1_000_000_000_000_000_000 attoFlow (1e18)

        // Step 2: (Off-chain) An EVM transaction sends 5_000_000_000 attoFlow (5e9 < 1e10)
        // to coa.address() — this bypasses Cadence deposit entirely.
        // COA balance is now: 1_000_000_005_000_000_000 attoFlow

        // Step 3: Attempt to withdraw full balance — PANICS with ErrWithdrawBalanceRounding
        // because 1_000_000_005_000_000_000 mod 1e10 != 0
        let bal = EVM.Balance(attoflow: 1_000_000_005_000_000_000)
        let stuck <- coa.withdraw(balance: bal)  // panics

        // Step 4: Withdraw only the UFix64-aligned portion — succeeds
        let alignedBal = EVM.Balance(attoflow: 1_000_000_000_000_000_000)
        let recovered <- coa.withdraw(balance: alignedBal)  // succeeds

        // Step 5: Remaining 5_000_000_000 attoFlow is permanently stuck in the COA.
        // No Cadence function can recover it.

        destroy coa
        destroy recovered
    }
}
```

func newInternalEVMTypeDepositFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeDepositFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from vault

			fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amountValue, ok := fromValue.GetField(
				context,
				fungibleTokenVaultTypeBalanceFieldName,
			).(interpreter.UFix64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))

			// Get to address

			toAddressValue, ok := invocation.Arguments[1].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			toAddress, err := AddressBytesArrayValueToEVMAddress(context, toAddressValue)
			if err != nil {
				panic(err)
			}

			// NOTE: We're intentionally not destroying the vault here,
			// because the value of it is supposed to be "kept alive".
			// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
			// and a withdrawal would then have to perform an actual mint of new tokens.

			// Deposit

			const isAuthorized = false
			account := handler.AccountByAddress(toAddress, isAuthorized)
			account.Deposit(types.NewFlowTokenVault(amount))

			return interpreter.Void
		},
	)

// AttoFlowBalanceIsValidForFlowVault returns true if the given balance,
// represented as atto-flow, can be stored in a Flow Vault, without loss
// in precision.
//
// Warning! The smallest unit of Flow token that a Flow Vault (Cadence)
// can store, is 1e-8 .
// This means the minimum balance, in atto-flow, that can be stored in a
// Flow Vault, is 1e10 .
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
	return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
