### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards by Calling `update_rewards` with `disable_rewards: true` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `staking.cairo` accepts a caller-controlled `disable_rewards` boolean with no access control. Any unprivileged address can call `update_rewards(any_valid_staker, disable_rewards: true)` every block, consuming the single per-block reward slot (`last_reward_block`) without distributing any rewards. This permanently freezes all consensus (V3) unclaimed yield for every staker in the protocol.

---

### Finding Description

`update_rewards` is a public function gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero. No role or identity check is performed. [1](#0-0) 

The function enforces a global, single-slot-per-block invariant via `last_reward_block`: [2](#0-1) 

After validating the staker, it unconditionally writes the current block number to `last_reward_block`: [3](#0-2) 

If `disable_rewards` is `true`, the function returns immediately after that write — no rewards are calculated or distributed. Because `last_reward_block` is now equal to the current block, every subsequent call to `update_rewards` in the same block will revert with `REWARDS_ALREADY_UPDATED`. [4](#0-3) 

`general_prerequisites` imposes no role restriction: [5](#0-4) 

The attacker only needs to supply any currently active staker address with non-zero balance, which is trivially discoverable from the public `stakers` vector: [6](#0-5) 

---

### Impact Explanation

In V3 consensus rewards, `update_rewards` is the sole mechanism by which a block proposer claims per-block STRK rewards. Because `last_reward_block` is a single global slot, only one call per block can succeed. An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block:

1. Consumes the per-block slot without distributing any rewards.
2. Causes every legitimate block-proposer call to revert with `REWARDS_ALREADY_UPDATED`.
3. Permanently prevents all stakers from accumulating consensus-era unclaimed yield.

This maps directly to **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- The entry point is fully public; no key, role, or privileged access is required.
- The attacker only needs a valid staker address (publicly enumerable from `stakers`) and enough gas to submit one transaction per block.
- The attack is economically rational for a competitor or adversary willing to spend gas to harm the protocol, matching the "griefing with damage to users" pattern and the MEV-drain scenario noted in the reference report.

---

### Recommendation

Restrict `update_rewards` so that only the operational address of the staker being rewarded (i.e., the block proposer) can call it, or derive `staker_address` from `get_caller_address()` via the `operational_address_to_staker_address` mapping rather than accepting it as a free parameter. Additionally, remove or gate the `disable_rewards` parameter behind a privileged role, or eliminate it entirely and handle the pre-consensus / ineligible-block case inside the function without exposing it to arbitrary callers.

---

### Proof of Concept

```
Block N:
  Attacker tx (ordered first by sequencer or submitted early):
    staking.update_rewards(staker = any_active_staker, disable_rewards = true)
    → last_reward_block := N
    → returns without distributing rewards

  Block proposer tx (ordered second):
    staking.update_rewards(staker = proposer, disable_rewards = false)
    → assert!(current_block_number > last_reward_block)  // N > N  → FALSE
    → reverts: REWARDS_ALREADY_UPDATED

Block N+1, N+2, … : attacker repeats.
Result: no staker ever receives consensus rewards; all unclaimed yield is permanently frozen.
```
