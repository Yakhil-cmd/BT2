### Title
Malicious CKD Participant Can Corrupt Protocol Output Without Detection — (`src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator blindly accumulates participant-supplied `(norm_big_y, norm_big_c)` values with no cryptographic proof of correct computation. Any single malicious participant can substitute arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that honest parties accept as valid.

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's contribution and sums them directly:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute and send `(λ_i · y_i · G₁, λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk))` where `x_i` is their secret signing share and `y_i` is a fresh random scalar:

```rust
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
Ok((norm_big_y, norm_big_c))
``` [2](#0-1) 

No proof of correct computation accompanies these values. The coordinator has no means to verify that a received `norm_big_c` encodes the participant's actual secret share `x_i` rather than an arbitrary point. The existing ZK-proof infrastructure (`dlogeq.rs`) is not invoked here. [3](#0-2) 

The analogy to the external report is direct: just as `ref_` and `artId_` were decoded from caller-supplied bytes and never bound into the Merkle hash, the participant-supplied `(norm_big_y, norm_big_c)` values are decoded from the wire and never bound into any cryptographic proof, leaving them fully attacker-controlled.

### Impact Explanation

The final `CKDOutput` is `(Σ λ_i y_i G₁, H(pk‖app_id)·msk + app_pk·Σ λ_i y_i)`. A malicious participant who sends `(P_arb, Q_arb)` instead of their correct share shifts both sums by `(P_arb − correct_Y_i, Q_arb − correct_C_i)`. The coordinator has no way to detect this because:

1. It does not know individual `x_i` shares.
2. It does not know individual `y_i` scalars.
3. The `unmask` step (subtracting `app_sk · total_big_y`) cannot recover the correct confidential key when `total_big_y` is corrupted.

The TEE that calls `unmask` will silently derive a wrong confidential key, matching the **High** impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs*. [4](#0-3) 

### Likelihood Explanation

Any single participant in the CKD session is a reachable, unprivileged attacker. The `ckd` public API accepts an arbitrary `me` identity and routes that participant's messages directly into the coordinator's accumulation loop with no authentication of the payload's mathematical content. No threshold of honest parties can prevent one malicious participant from corrupting the sum. [5](#0-4) 

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a `dlogeq` proof of knowledge. Concretely, the participant knows scalar `x_i` such that:

- `norm_big_s = norm_big_c − norm_big_y · (app_pk / G₁)` equals `λ_i · x_i · H(pk ‖ app_id)`
- The participant's public verification share `vk_i = x_i · G₂` is already known to the coordinator from the DKG output.

A `Statement { public0: vk_i, generator1: H(pk‖app_id), public1: norm_big_s / λ_i }` DLOGEQ proof would let the coordinator reject any malformed contribution before accumulating it, using the existing `dlogeq::verify` function. [6](#0-5) 

### Proof of Concept

```rust
// Malicious participant overrides compute_signature_share result
// and sends (identity, identity) instead of correct shares.
// The coordinator sums without verification:
//   norm_big_y += G1::identity()  -> total_big_y is missing participant i's term
//   norm_big_c += G1::identity()  -> total_big_c is missing x_i * H(pk||app_id) term
// unmask() then returns:
//   total_big_c - app_sk * total_big_y
//   = (correct_C - lambda_i*(x_i*H + y_i*app_pk)) - app_sk*(correct_Y - lambda_i*y_i*G1)
//   = H(pk||app_id)*msk - lambda_i*x_i*H(pk||app_id)
//   ≠ H(pk||app_id)*msk
// The TEE silently derives a wrong confidential key with no error returned.
```

The coordinator's accumulation loop at lines 50–55 of `protocol.rs` accepts and sums the forged zero elements, producing a `CKDOutput` that passes all downstream checks yet encodes an incorrect confidential key. [7](#0-6)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))

pub fn prove_with_nonce<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    witness: Witness<C>,
    k: Scalar<C>,
) -> Result<Proof<C>, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (big_k_0, big_k_1) = statement.phi(&k);

    // This will never raise error as k is not zero and generator1 is not the identity
    let enc = encode_two_points::<C>(&big_k_0, &big_k_1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    let s = k + e * witness.x.0;
    Ok(Proof {
        e: SerializableScalar::<C>(e),
        s: SerializableScalar::<C>(s),
    })
}

pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
