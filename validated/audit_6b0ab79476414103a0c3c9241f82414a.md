### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Fee Escalation on Non-Deploy-Account Mempool Entry — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call only when a `deploy_account` transaction for the sender is already in the mempool (UX pattern: send deploy + invoke simultaneously). However, the function checks `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction type from that address. When fee escalation is enabled, an attacker can first submit a valid invoke tx (nonce=1) to seed the mempool, then submit a second invoke tx at the same nonce with a higher fee and an invalid signature. The second tx passes `validate_by_mempool` (fee escalation), then `skip_stateful_validations` returns `true` because the account is already in the pool, and `__validate__` is never called. The invalid-signature tx is admitted to the mempool.

### Finding Description

**Root cause — wrong ownership check before skipping validation:** [1](#0-0) 

The function fires only when `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)`. When both conditions hold it calls:

```rust
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())
```

which resolves to: [2](#0-1) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`tx_pool.contains_account` checks only whether **any** transaction from that address is in the pool: [3](#0-2) 

There is no check that the existing pool entry is a `DeployAccount` transaction. The comment in the code acknowledges the second case explicitly:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second case is the exploitable path.

**Fee escalation enables the attack:**

`validate_by_mempool` (called before `skip_stateful_validations`) calls `mempool.validate_tx`, which accepts a replacement at the same nonce when fee escalation conditions are met: [4](#0-3) [5](#0-4) 

If `enable_fee_escalation = true` and the incoming tx raises both `tip` and `max_l2_gas_price` by the required percentage, `validate_tx` returns `Ok(())`. The gateway then calls `run_validate_entry_point` with `skip_validate = true`: [6](#0-5) 

`__validate__` is never invoked for the replacement tx.

**The gateway flow that exposes the gap:** [7](#0-6) 

Order: `get_nonce_from_state` → `run_pre_validation_checks` (which calls `validate_by_mempool` then `skip_stateful_validations`) → `run_validate_entry_point`. The mempool's `validate_tx` and the skip-check are evaluated against different invariants; neither verifies the tx type of the existing pool entry.

### Impact Explanation

An attacker can insert an invoke transaction carrying an **invalid signature** (or arbitrary malicious calldata) into the mempool without passing the account's `__validate__` entry point. The transaction will be sequenced by the batcher. During batcher execution `__validate__` is called with `validate=true`, so the tx will revert and the attacker forfeits their fee. However:

- The invalid tx is **admitted to the mempool** and occupies a slot, displacing the legitimate tx via fee escalation.
- The invalid tx **is sequenced** and consumes block resources.
- For account contracts whose `__validate__` does not enforce signature integrity (e.g., multisig contracts in intermediate states, or contracts with permissive validation logic), the tx could **execute successfully**.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

Requires three conditions to hold simultaneously:

1. **Fee escalation enabled** (`enable_fee_escalation = true` in `MempoolStaticConfig`) — this is a configurable production parameter.
2. **A deployed account with on-chain nonce = 0** — possible for genesis-funded accounts, accounts deployed via L1→L2 messages, or any account that has never sent a transaction.
3. **`max_allowed_nonce_gap >= 1`** — required for the deploy+invoke UX feature to function at all; the default must be ≥ 1.

All three are plausible in a live network. The attacker must first submit one valid tx (T1) to seed the mempool, then submit the malicious replacement (T2). No privileged access is required.

### Recommendation

`skip_stateful_validations` must verify that the existing mempool entry for the account is specifically a `DeployAccount` transaction, not merely that the account address appears in the pool. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type of the stored entry. The current `account_tx_in_pool_or_recent_block` API is insufficient for this authorization decision. [8](#0-7) 

### Proof of Concept

```

  - enable_fee_escalation = true, fee_escalation_percentage = 10
  - max_allowed_nonce_gap >= 1
  - Account A: class_hash set (deployed), on-chain nonce = 0

Step 1 — seed the mempool:
  Submit InvokeV3 T1 from A, nonce=1, tip=100, max_l2_gas_price=100, valid signature.
  Gateway flow:
    validate_nonce(nonce=1, account_nonce=0) → OK (within gap)
    validate_by_mempool → OK (no existing entry)
    skip_stateful_validations: account NOT in pool → skip=false
    __validate__ called → passes (account is deployed)
  T1 admitted; tx_pool.contains_account(A) = true.

Step 2 — bypass __validate__ via fee escalation:
  Submit InvokeV3 T2 from A, nonce=1, tip=111, max_l2_gas_price=111, INVALID signature.
  Gateway flow:
    validate_nonce(nonce=1, account_nonce=0) → OK
    validate_by_mempool → calls mempool.validate_tx:
      validate_incoming_tx: tx_hash(T2) ≠ tx_hash(T1), nonce=1 >= resolved_nonce=0 → OK
      validate_fee_escalation: tip 111 >= 100*1.1=110, price 111 >= 110 → OK (returns T1 ref)
    validate_by_mempool returns OK
    skip_stateful_validations: nonce=1, account_nonce=0,
      account_tx_in_pool_or_recent_block(A) = true (T1 in pool) → returns true
    run_validate_entry_point called with skip_validate=true → __validate__ SKIPPED
  mempool.add_tx(T2) replaces T1 via fee escalation.

Result: T2 (invalid signature) is in the mempool. __validate__ was never called.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
