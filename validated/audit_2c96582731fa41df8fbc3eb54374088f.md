### Title
Gateway Admits Nonce-1 Invoke with Invalid Signature via Overly Broad `skip_stateful_validations` Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` bypasses the `__validate__` entry-point call for any Invoke transaction with `nonce == 1` sent to an undeployed account (`account_nonce == 0`), provided `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** account transaction in the pool — not specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` in the mempool can immediately submit a nonce-1 Invoke with an arbitrary/invalid signature, have it admitted by the gateway without signature verification, and have it executed in a block where it reverts but consumes nonce 1, permanently disrupting the victim's post-deploy transaction sequence.

---

### Finding Description

The UX feature that allows a user to submit `deploy_account` (nonce 0) and `invoke` (nonce 1) simultaneously is implemented in `skip_stateful_validations`: [1](#0-0) 

The condition that triggers the skip is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

This returns `true` for **any** account transaction — including a `deploy_account` submitted by the victim. When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

This means `blockifier_validator.validate(account_tx)` is called with `validate = false`, so `__validate__` (the account's signature-verification entry point) is **never called** at the gateway level. [5](#0-4) 

The nonce check in `validate_nonce` passes because nonce 1 is within `[0, max_allowed_nonce_gap=200]`: [6](#0-5) 

The mempool's `validate_tx` only checks nonce ordering, not signatures: [7](#0-6) 

The broken invariant: **a nonce-1 Invoke for an undeployed account should only skip `__validate__` if a `deploy_account` for that account is specifically pending**. The actual check is satisfied by any account transaction.

---

### Impact Explanation

**Attack sequence:**

1. Victim submits `deploy_account` for address A (nonce 0). It enters the mempool.
2. Attacker observes A in the mempool and submits `Invoke` from A with `nonce = 1` and an arbitrary/invalid signature.
3. Gateway: nonce check passes (1 ∈ [0, 200]); `account_tx_in_pool_or_recent_block(A)` → `true` (victim's deploy_account); `skip_stateful_validations` → `true`; `__validate__` is **not called**; transaction is admitted to the mempool.
4. Block execution: `deploy_account` runs (nonce → 1); attacker's Invoke runs — `handle_nonce` increments nonce to 2 before `__validate__` is called; `__validate__` fails (invalid signature) → transaction reverts; nonce is now 2.
5. Victim's legitimate nonce-1 Invoke is permanently stuck (nonce 1 was consumed by the reverted attacker transaction).

The nonce increment happens unconditionally in `handle_nonce` before `__validate__`: [8](#0-7) 

This matches **High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing**, and also causes **wrong state** (nonce 2 instead of 1 after the victim's deploy).

---

### Likelihood Explanation

The attack requires only:
1. Observing a `deploy_account` transaction in the public mempool (trivially observable).
2. Submitting a nonce-1 Invoke with any signature to the same address.

No privileged access, no special knowledge, and no race condition beyond the window between the victim's `deploy_account` submission and the attacker's response. The default `max_nonce_for_validation_skip = Nonce(Felt::ONE)` limits the skip to nonce 1 only, but that is exactly the nonce used in the standard deploy+invoke UX flow, making every new account deployment a target. [9](#0-8) 

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the existing pool lookup should filter by transaction type. The invariant to enforce is:

> Skip `__validate__` for a nonce-1 Invoke from an undeployed account **only if** a `deploy_account` transaction for that exact address is currently pending in the mempool.

---

### Proof of Concept

```
1. Victim calls gateway: POST /add_transaction
   { type: "DEPLOY_ACCOUNT", sender_address: A, nonce: 0, signature: <valid> }
   → Accepted; A enters mempool.

2. Attacker calls gateway: POST /add_transaction
   { type: "INVOKE", sender_address: A, nonce: 1, calldata: [<drain_calldata>], signature: [0x0] }
   → skip_stateful_validations returns true (A is in mempool)
   → __validate__ is NOT called
   → Accepted; attacker's Invoke enters mempool.

3. Block is built:
   - deploy_account(A) executes → A deployed, nonce = 1
   - attacker's Invoke executes → handle_nonce increments nonce to 2
     → __validate__ called → fails (signature 0x0 invalid)
     → transaction reverts; nonce stays at 2

4. Victim's legitimate nonce-1 Invoke is rejected: account nonce is 2, tx nonce is 1.
``` [10](#0-9) [11](#0-10)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
