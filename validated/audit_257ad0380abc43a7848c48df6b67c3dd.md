### Title
WalletConnect Offer Confirmation Understates Total Spend Due to Incorrect Royalty Splitting in `formatAmountWithRoyalties` - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each royalty, rather than computing each royalty on the full amount. This causes the `amountWithRoyalties` field shown in the WalletConnect confirmation dialog to be systematically understated whenever a `take_offer` or `create_offer_for_ids` command involves multiple NFTs with royalties. A user approves a transaction believing they will spend less than they actually will.

### Finding Description

In `packages/gui/src/electron/commands/parseCommandDisplay.ts`, the function `formatAmountWithRoyalties` computes the total spend including royalties for display in the WalletConnect confirmation dialog:

```typescript
// lines 363-367
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

The function first divides `amount` by the number of NFTs (`splitAmount`), then computes each NFT's royalty on that split amount. This is wrong in two compounding ways:

1. **Conceptually incorrect**: In Chia's offer protocol, each NFT's royalty is a percentage of the full fungible purchase price, not a fraction of it. The `splitAmount` logic has no basis in the protocol.
2. **Integer truncation**: `amount / BigInt(royaltyPercentages.length)` truncates the remainder, and each `(splitAmount * royaltyPercentage) / 10_000n` truncates again — once per NFT.

This is the exact analog of the Velo Vault bug: one code path (the WalletConnect display) computes a value by splitting first then pricing each piece individually (truncating N times), while the correct path (the offer builder UI, which uses backend-calculated royalties via `useCalculateRoyaltiesForNFTsQuery`) computes each royalty on the full amount.

**Concrete example** (matching the existing test case at line 251):
- `amount` = 100,000,000 mojos (0.0001 XCH)
- NFT1 royalty = 500 (5%), NFT2 royalty = 10 (0.1%)

| Method | Calculation | Displayed total |
|---|---|---|
| Buggy `formatAmountWithRoyalties` | `splitAmount=50,000,000`; royalty1=2,500,000; royalty2=50,000 | **0.00010255 XCH** |
| Correct (per-NFT on full amount) | royalty1=5,000,000; royalty2=100,000 | **0.0001051 XCH** |

The test at line 320 asserts `amountWithRoyalties: '0.00010255'`, confirming the buggy value is what is actually displayed. [2](#0-1)

### Impact Explanation

The `amountWithRoyalties` field is the primary "total you will spend" figure shown to the user in the WalletConnect confirmation dialog for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` commands. When a user approves the command, the wallet backend executes the transaction using the correct royalty amounts — which are higher than what was displayed. The user is deceived into approving a transaction that costs more XCH or CAT than the confirmation dialog indicated.

This is a **High** impact: WalletConnect state causes a user to approve the wrong amount, fitting "Corruption, spoofing, or unsafe trust of... WalletConnect state that causes a user to... display the wrong asset, identity, amount, destination, or status."

The understatement scales with:
- Number of NFTs (more splits = more truncation)
- Royalty percentage magnitude
- Amounts not evenly divisible by the NFT count [4](#0-3) 

### Likelihood Explanation

Any dApp connected via WalletConnect can issue a `take_offer` command containing an offer with multiple NFTs that have royalties. This is a standard, supported offer type. No special privileges are required — only a WalletConnect pairing, which any dApp can request. The user has no way to detect the discrepancy from the confirmation dialog alone. [5](#0-4) 

### Recommendation

Replace the `splitAmount` logic with per-NFT royalty computation on the full amount:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Each NFT's royalty is a percentage of the full amount, not a split portion
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

This matches the correct pattern used in `OfferBuilderXCHSection.tsx` and `OfferBuilderTokensSection.tsx`, which sum per-NFT royalty amounts directly onto the full fungible amount. [6](#0-5) 

### Proof of Concept

1. A dApp pairs with the user's wallet via WalletConnect.
2. The dApp issues `chia_wallet.take_offer` with an offer blob containing 2 NFTs: NFT-A (royalty 500 = 5%) and NFT-B (royalty 500 = 5%), requesting 1,000,000,000 mojos (0.001 XCH).
3. The WalletConnect confirmation dialog calls `parseCommandDisplay`, which calls `formatAmountWithRoyalties` with `amount=1,000,000,000n` and `royaltyPercentages=[500, 500]`.
4. **Buggy display**: `splitAmount = 500,000,000`; royalty = `(500,000,000×500/10,000) + (500,000,000×500/10,000)` = 25,000,000 + 25,000,000 = 50,000,000; displayed total = **1,050,000,000 mojos (0.00105 XCH)**.
5. **Actual spend**: royalty = `(1,000,000,000×500/10,000) + (1,000,000,000×500/10,000)` = 50,000,000 + 50,000,000 = 100,000,000; actual total = **1,100,000,000 mojos (0.0011 XCH)**.
6. The user sees 0.00105 XCH and approves; the wallet deducts 0.0011 XCH — a 4.76% overcharge relative to what was displayed. [4](#0-3)
