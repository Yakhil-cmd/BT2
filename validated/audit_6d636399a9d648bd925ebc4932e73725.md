### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both `change_reward_address` in the staking contract and in the delegation pool contract accept a zero address as a valid `reward_address` without reverting. A staker or pool member can set their reward address to `0x0`, causing all future reward claims to transfer tokens to the zero address, permanently destroying the unclaimed yield.

---

### Finding Description

In `src/staking/staking.cairo`, the `change_reward_address` function performs only one validation on the incoming address — it checks that the address is not a registered token address — but does **not** check for zero address: [1](#0-0) 

The same pattern exists in `src/pool/pool.cairo` for pool members: [2](#0-1) 

The same omission is present in the initial `stake` call, where `reward_address` is also only checked against the token registry: [3](#0-2) 

When rewards are later claimed, `send_rewards_to_staker` unconditionally transfers to whatever address is stored as `reward_address`, including zero: [4](#0-3) 

---

### Impact Explanation

If a staker or pool member sets their `reward_address` to `ContractAddress::zero()`, all accumulated `unclaimed_rewards_own` (staker) or `unclaimed_rewards` (pool member) are transferred to the zero address upon the next `claim_rewards` call. The tokens are permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

---

### Likelihood Explanation

The entry path is reachable by any unprivileged staker or pool member with no special privileges required. The call `change_reward_address(0)` is a single transaction. While deliberate self-harm is unlikely, accidental zero-address submission is realistic in scripted deployments or integrations where the reward address variable is uninitialized or incorrectly serialized. The `stake` path (passing `reward_address: 0` at registration) is equally unguarded and affects the staker from the very first reward cycle.

---

### Recommendation

Add an explicit zero-address guard in both `change_reward_address` implementations and in `stake`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `ZERO_ADDRESS` guard already used elsewhere in the codebase (e.g., in `add_token`): [5](#0-4) 

---

### Proof of Concept

1. A staker calls `stake(reward_address: 0, operational_address: X, amount: MIN_STAKE)`. The call succeeds because the only reward-address check is `!does_token_exist(token_address: 0)`, which passes since zero is not a registered token. [6](#0-5) 

2. Alternatively, an existing staker calls `change_reward_address(reward_address: 0)`. The call succeeds for the same reason. [7](#0-6) 

3. After rewards accumulate, `claim_rewards` is called. `send_rewards_to_staker` reads `staker_info.reward_address` (which is `0`) and executes `checked_transfer(recipient: 0, amount: rewards)`. [8](#0-7) 

4. The STRK tokens are transferred to the zero address and are permanently lost. The staker's `unclaimed_rewards_own` is zeroed out with no recovery path.

The identical path exists for pool members via `pool.cairo`'s `change_reward_address`: [9](#0-8)
