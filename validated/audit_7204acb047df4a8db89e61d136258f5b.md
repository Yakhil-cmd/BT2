### Title
Unchecked Precompile Call Return Value After `_burn` in `withdrawToNear`/`withdrawToEthereum` Enables Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20V2` execute `_burn` before calling the exit precompile, then capture the assembly `call` result in a local variable `res` that is never inspected. When the precompile sub-call fails for any reason, the EVM returns `0` in `res`, the parent call frame continues normally, and the burned tokens are permanently destroyed with no corresponding NEP-141 transfer or Ethereum withdrawal scheduled.

---

### Finding Description

**Root cause — `EvmErc20V2.withdrawToNear`:**

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // ① tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // ② res is never checked — no revert, no restore
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum`: [2](#0-1) 

**Precompile failure path — `get_nep141_from_erc20`:**

When the ERC-20 address is absent from `Erc20Nep141Map` storage, the function returns `Err(ExitError::Other(ERR_TARGET_TOKEN_NOT_FOUND))`:

```rust
fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    ...
}
``` [3](#0-2) 

This error propagates through `process_precompile`, which converts it to `PrecompileFailure::Error`:

```rust
p.run(input, gas_limit.map(EthGas::new), context, is_static)
    .map_err(|exit_status| PrecompileFailure::Error { exit_status })
``` [4](#0-3) 

In standard EVM semantics, `PrecompileFailure::Error` causes the sub-call opcode to return `0` (failure) but does **not** revert the calling frame. The parent `withdrawToNear` / `withdrawToEthereum` call continues and returns normally. No promise log is emitted, so no `ft_transfer` or `withdraw` NEAR promise is ever scheduled.

**EVM-level confirmation:** `ExitError::Other` maps to `TransactionStatus::Other`, which the engine treats as a completed (non-fatal) transaction: [5](#0-4) 

---

### Impact Explanation

The ERC-20 `totalSupply` decreases and the caller's balance is zeroed, but no NEP-141 `ft_transfer` promise is scheduled and no Ethereum `withdraw` promise is scheduled. The underlying NEP-141 tokens remain locked in the Aurora engine contract with no recovery path. This constitutes **permanent freezing of funds** (Critical).

---

### Likelihood Explanation

The trigger condition — an `EvmErc20V2` whose address is absent from `Erc20Nep141Map` — is reachable via a standard EVM contract deployment transaction (a supported production path). Any EVM user can deploy `EvmErc20V2` directly without going through `deploy_erc20_token`, become the admin, mint tokens, and then call `withdrawToNear`. The same silent-failure path is also reachable for legitimately registered tokens if the eth-connector storage key is missing or if the precompile runs out of gas in the sub-call, making the surface broader than just the unregistered-ERC-20 case.

---

### Recommendation

1. **Check the precompile call result and revert on failure.** Replace the unchecked assembly block with one that reverts if `res == 0`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Alternatively, invert the order of operations** — call the precompile first and only burn if it succeeds. This is the safer pattern because `_burn` is irreversible within the same call frame.

3. Apply the same fix to `EvmErc20.sol`, which contains the identical unchecked pattern. [6](#0-5) 

---

### Proof of Concept

1. Deploy `EvmErc20V2` directly via an EVM transaction (do **not** call `deploy_erc20_token`; the ERC-20 address is therefore absent from `Erc20Nep141Map`).
2. As the deployer (admin), call `mint(attacker_address, 1000)`.
3. Call `withdrawToNear("victim.near", 1000)`.
4. Observe: `balanceOf(attacker_address) == 0`, `totalSupply` decreased by 1000, but no `ft_transfer` NEAR promise was scheduled (no promise log emitted by the precompile). The 1000 units are permanently destroyed.
5. Invariant broken: a failed precompile call did not prevent the burn and did not revert the transaction.
