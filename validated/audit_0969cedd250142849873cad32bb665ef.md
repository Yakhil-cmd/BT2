### Title
Gateway L2 Gas Price Threshold Uses Previous Block Price Instead of Next Block Price, Causing Incorrect Admission Decisions - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` compares a transaction's `max_price_per_unit` against the **previous** (latest committed) block's L2 gas price, but the transaction will be executed in the **next** block whose price is computed by EIP-1559 mechanics. A developer TODO comment in the code explicitly acknowledges this is wrong. The mismatch causes the gateway to admit transactions that will fail the blockifier's fee check during actual execution (rising-price scenario) and to reject valid transactions (falling-price scenario).

### Finding Description

In `validate_resource_bounds`, the gateway reads the L2 gas price from the latest committed block and uses it as the admission threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is then computed as `min_gas_price_percentage% × previous_block_l2_gas_price`. With the production default of `min_gas_price_percentage = 100`, the threshold equals the previous block's price exactly. [2](#0-1) 

The `gateway_fixed_block_state_reader` is always initialized from `latest_block_number` — the most recently committed block — not the pending next block: [3](#0-2) 

The actual next block's L2 gas price is computed by the consensus orchestrator using EIP-1559 mechanics (`calculate_next_base_gas_price`), which can produce a price meaningfully different from the previous block's price: [4](#0-3) 

During batcher execution, `check_fee_bounds` in `AccountTransaction::perform_pre_validation_stage` compares the transaction's `max_price_per_unit` against the **actual** next block's gas price embedded in the block context: [5](#0-4) 

This creates a split-brain between gateway admission and blockifier execution:

| Gas price trend | Gateway threshold | Blockifier threshold | Result |
|---|---|---|---|
| Rising (P_next > P_prev) | P_prev | P_next | Transactions with `P_prev ≤ price < P_next` admitted by gateway, rejected by blockifier |
| Falling (P_next < P_prev) | P_prev | P_next | Transactions with `P_next ≤ price < P_prev` rejected by gateway, would pass blockifier |

### Impact Explanation

**Rising prices (invalid transactions admitted):** The gateway admits transactions whose `max_price_per_unit` is between `P_prev` and `P_next`. These transactions enter the mempool and are handed to the batcher, where `check_fee_bounds` rejects them because the actual block price is `P_next > max_price_per_unit`. The sequencer wastes execution resources on transactions that are guaranteed to fail, and the mempool fills with unexecutable transactions.

**Falling prices (valid transactions rejected):** The gateway rejects transactions whose `max_price_per_unit` is between `P_next` and `P_prev`. These transactions would pass the blockifier's fee check (since `P_next < max_price_per_unit`) but are turned away at the gateway. Users receive a spurious `GAS_PRICE_TOO_LOW` error and must resubmit with a higher price than the network actually requires.

Both outcomes match the "High" impact scope: the gateway makes incorrect admission decisions — accepting transactions that will fail execution and rejecting transactions that would succeed.

### Likelihood Explanation

The EIP-1559 price adjustment is bounded per block but is continuous and directional. During any sustained period of falling L2 gas usage (which drives prices down), every block transition produces a `P_next < P_prev` condition. With `min_gas_price_percentage = 100` (the production default), the full gap between `P_next` and `P_prev` becomes a rejection zone for valid transactions. This is a systemic condition, not a rare edge case, and is triggered by normal network activity without any attacker involvement.

### Recommendation

Replace the previous block's L2 gas price with the computed next block's L2 gas price, as the TODO comment already acknowledges:

```rust
// Use the next block's L2 gas price, not the previous block's.
let next_block_l2_gas_price = compute_next_l2_gas_price(
    previous_block_l2_gas_price,
    previous_block_l2_gas_used,
    ...
);
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    next_block_l2_gas_price,
)?;
```

Alternatively, if the next block price is not yet available at gateway admission time, apply a configurable tolerance margin (e.g., `min_gas_price_percentage = 90`) so that the threshold is set below the previous block's price, ensuring transactions valid for the next block are not spuriously rejected.

### Proof of Concept

1. Observe the production default: `min_gas_price_percentage = 100`, so `threshold = previous_block_l2_gas_price`.
2. Suppose block N has `l2_gas_price = 1000 fri`. Due to low gas usage, EIP-1559 computes block N+1's price as `900 fri`.
3. A user submits an invoke transaction with `max_price_per_unit = 950 fri` (above the actual next block price of 900, so the transaction is economically valid).
4. `validate_resource_bounds` computes `threshold = 100% × 1000 = 1000`. Since `950 < 1000`, it returns `GAS_PRICE_TOO_LOW` and rejects the transaction.
5. The transaction would have passed `check_fee_bounds` in the blockifier (950 ≥ 900), so the rejection is incorrect.

The inverse (rising prices, invalid admission) follows symmetrically: a transaction with `max_price_per_unit = 1000` passes the gateway threshold of 1000 but fails the blockifier check when the next block price is 1100.

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }
