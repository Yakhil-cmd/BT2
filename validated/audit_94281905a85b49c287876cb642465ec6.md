### Title
Signature Validation Bypass via `skip_stateful_validations` Overly Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (i.e., account signature verification) for any invoke transaction with `nonce = 1` submitted against an account whose on-chain nonce is still `0`, provided `account_tx_in_pool_or_recent_block` returns `true` for that sender address. The mempool presence check is not restricted to `DeployAccount` transactions — it returns `true` for **any** transaction type associated with the address. An unprivileged attacker who observes a victim's `DeployAccount` transaction enter the mempool can immediately inject an invoke transaction bearing the victim's `sender_address`, `nonce = 1`, and an arbitrary/invalid signature, and have it admitted to the mempool without any signature check.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:
1. The transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

**The mempool check is type-agnostic:** [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in `tx_pool` (any pending transaction) or in `state` (staged/committed). It does **not** verify that the matching transaction is a `DeployAccount`.

**The comment's reasoning is incorrect:**

The inline comment at line 440–443 states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This is wrong: a `DeployAccount` for the victim's address placed by the victim themselves satisfies the check, and the attacker's subsequent nonce-1 invoke is then admitted without any signature verification.

**No other guard catches the invalid signature:**

- `StatelessTransactionValidator::validate` performs no signature check. [3](#0-2) 
- `validate_by_mempool` → `validate_tx` only checks for duplicate hashes and stale nonces. [4](#0-3) 
- `run_validate_entry_point` is called with `execution_flags.validate = false` when `skip_validate = true`, so `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`. [5](#0-4) [6](#0-5) 

**The transaction is then forwarded to the mempool:** [7](#0-6) 

### Impact Explanation

The gateway admits an invoke transaction carrying an arbitrary/invalid signature for an account it does not control. This is a direct violation of the admission invariant: *every transaction in the mempool must have passed account-level signature validation*. The corrupted admission decision propagates to:

1. **Mempool pollution** — the attacker's transaction occupies the victim's nonce-1 slot.
2. **Nonce consumption / DoS** — when the batcher later executes the block, it calls `__validate__` (with `execution_flags.validate = true`), which fails; the transaction is reverted, but in Starknet reverted transactions still consume the nonce. Alice's legitimate nonce-1 transaction is then rejected as `NonceTooOld`.
3. **Fee-escalation griefing** — the attacker can set a high tip on the injected transaction, forcing Alice to pay more to replace it.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The precondition is simply that the victim has a pending `DeployAccount` transaction in the mempool — a normal, observable on-chain event. No privileged access, no private key material, and no special knowledge is required. The attacker only needs to know the victim's contract address (publicly derivable from `class_hash`, `salt`, and `constructor_calldata`). The window of opportunity lasts from the moment the `DeployAccount` enters the mempool until it is committed.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction for the sender address is present in the mempool. For example, expose a `deploy_account_in_pool(address)` query on the mempool that inspects the transaction type, and use that in `skip_stateful_validations` instead of the current generic presence check.

### Proof of Concept

```
1. Alice broadcasts DeployAccount(class_hash=C, salt=S, constructor_calldata=D, nonce=0, sig=valid).
   → Gateway validates __validate_deploy__, transaction enters mempool pool.
   → account_tx_in_pool_or_recent_block(Alice) == true.

2. Eve constructs InvokeV3(sender_address=Alice, nonce=1, calldata=<arbitrary>, sig=0xdeadbeef).

3. Eve submits to gateway:
   - StatelessTransactionValidator::validate → OK (no sig check).
   - get_nonce_from_state(Alice) → 0.
   - validate_state_preconditions: nonce=1 >= account_nonce=0 → OK.
   - validate_by_mempool: no duplicate hash, nonce not stale → OK.
   - skip_stateful_validations:
       nonce==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(Alice)==true ✓
       → returns true (SKIP __validate__).
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false → StatefulValidator returns Ok immediately.
   → Gateway calls mempool_client.add_tx(Eve's tx). Transaction admitted.

4. Batcher executes block:
   - Executes Alice's DeployAccount → Alice's account created, nonce becomes 1.
   - Executes Eve's Invoke (nonce=1):
       perform_pre_validation_stage: nonce 1 >= 1 → increments Alice's nonce to 2.
       __validate__ called → signature check fails → transaction reverted.
   - Alice's nonce is now 2.

5. Alice submits her legitimate Invoke(nonce=1) → rejected: NonceTooOld (account nonce=2).
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
