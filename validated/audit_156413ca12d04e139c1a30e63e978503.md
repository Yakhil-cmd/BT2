### Title
Unchecked `RerandomizationArguments.participants` Enables Malicious-Coordinator Split-View Attack Leading to Secret Key Extraction — (`src/ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The `RerandomizationArguments` struct contains a `participants` field that is hashed into the HKDF-derived rerandomization scalar `delta`. Neither `robust_ecdsa::sign::sign()` nor `ot_based_ecdsa::sign::sign()` validates that the `participants` embedded in the already-rerandomized `RerandomizedPresignOutput` matches the `participants` list actually used in the signing session. A malicious coordinator can distribute different `RerandomizationArguments` (with different `participants` lists) to different honest signers, causing each to compute a distinct `delta`, effectively reusing the same presignature under two different rerandomizations — the exact split-view attack the codebase documents as leading to full secret key recovery.

---

### Finding Description

`RerandomizationArguments` is defined in `src/ecdsa/mod.rs` and contains six caller-supplied fields, including `participants: ParticipantList`. [1](#0-0) 

`derive_randomness()` feeds `participants` into the HKDF info string, so any difference in the participant list produces a completely different scalar `delta`. [2](#0-1) 

`RerandomizedPresignOutput::rerandomize_presign()` in `src/ecdsa/robust_ecdsa/mod.rs` consumes `args.participants` (via `derive_randomness()`) to produce the rerandomized shares, but stores **no record** of which participant list was used. [3](#0-2) 

`robust_ecdsa::sign::sign()` accepts the already-rerandomized `presignature: RerandomizedPresignOutput` and a separate `participants: &[Participant]`. It validates the signing-session participant list (size, membership, `msg_hash != 0`) but **never checks** that the `participants` embedded in the rerandomization matches the signing-session `participants`. [4](#0-3) 

The same gap exists in `ot_based_ecdsa::sign::sign()`. [5](#0-4) 

---

### Impact Explanation

The codebase's own security documentation states:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [6](#0-5) 

Because `participants` is part of the HKDF input that produces `delta`, supplying a different `participants` list to different signers is equivalent to giving them different `(h, ε)` tuples while reusing the same presignature. The coordinator collects the resulting inconsistent shares and applies the standard nonce-reuse key-recovery attack.

**Impact**: Critical — full extraction of the aggregate ECDSA secret key.

---

### Likelihood Explanation

The coordinator role is a documented, reachable participant in every signing session. The library exposes `RerandomizationArguments::new()` as a plain constructor with no cross-participant consistency enforcement. A malicious coordinator needs only to call `rerandomize_presign()` twice on the same `PresignOutput` with two `RerandomizationArguments` structs that differ only in their `participants` field, distribute the resulting `RerandomizedPresignOutput` values to the respective signers, and collect two sets of signing shares. No cryptographic primitive break or external compromise is required.

---

### Recommendation

1. **Bind rerandomization to the signing session**: Include the signing-session `participants` list in `RerandomizationArguments` and, inside `sign()`, verify that `presignature` was rerandomized with the same participant set (e.g., by storing a commitment/hash of the `participants` inside `RerandomizedPresignOutput` and checking it in `sign()`).

2. **Validate `msg_hash` consistency**: Similarly verify that the `msg_hash` embedded in `RerandomizationArguments` matches the `msg_hash` argument passed to `sign()`.

3. **Document the trust boundary explicitly**: Until enforcement is added, the API documentation must state that callers are responsible for ensuring all participants receive identical `RerandomizationArguments`.

---

### Proof of Concept

```
// Attacker is the coordinator; presign_output is a shared PresignOutput.

// Session 1: rerandomize with participants = {A, B, C}
let args1 = RerandomizationArguments::new(pk, tweak, msg_hash, big_r,
    ParticipantList::new(&[A, B, C]).unwrap(), entropy);
let rerand1_A = RerandomizedPresignOutput::rerandomize_presign(&presign_output_A, &args1);
let rerand1_B = RerandomizedPresignOutput::rerandomize_presign(&presign_output_B, &args1);
// ... collect shares s1_A, s1_B, s1_C → signature (R·δ1, s1)

// Session 2: rerandomize with participants = {A, B, D}  ← different list, same presignature
let args2 = RerandomizationArguments::new(pk, tweak, msg_hash, big_r,
    ParticipantList::new(&[A, B, D]).unwrap(), entropy);
let rerand2_A = RerandomizedPresignOutput::rerandomize_presign(&presign_output_A, &args2);
// ... collect shares → signature (R·δ2, s2)

// δ1 ≠ δ2 ⟹ two signatures with multiplicatively related nonces over the same key.
// Standard ECDSA nonce-reuse recovery extracts the secret key x.
```

`sign()` accepts both calls without error because it only validates the signing-session `participants` slice, not the `participants` baked into the `RerandomizedPresignOutput`. [7](#0-6) [8](#0-7)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
