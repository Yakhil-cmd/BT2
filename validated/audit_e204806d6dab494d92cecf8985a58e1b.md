### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Fee Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` imposes no cap on `addFeeBuyE6` / `addFeeSellE6`, while `setPoolAdminFees` correctly enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees up to `type(uint16).max = 65535` (6.5535% in E6 units) on any active bin, making the total effective swap fee exceed the protocol's intended cap system and causing traders to pay more than the bounded admin fee allows.

---

### Finding Description

The factory enforces a layered fee cap system. The factory owner sets `maxAdminSpreadFeeE6` (≤ `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%) via `setFeeCaps`. The pool admin's base spread component is then bounded by this cap inside `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, the same pool admin can call `setPoolBinAdditionalFees` with no cap check whatsoever:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap validation — only a bin-range check: [3](#0-2) 

During every swap, the per-bin additional fee is added directly on top of the base spread fee:

```solidity
// MetricOmmPool.sol L910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears for all four swap directions (token0/token1, exact-in/exact-out). [5](#0-4) 

The `uint16` type allows `addFeeBuyE6` / `addFeeSellE6` up to `65535`, which equals **6.5535%** in E6 units. This is additive on top of the already-capped base spread fee (`spreadFeeE6`, which can be up to `maxProtocolSpreadFeeE6 + maxAdminSpreadFeeE6 ≤ 40%`). The total effective fee for a swap in the targeted bin can therefore reach **~46.5%**, far exceeding the protocol's intended hard cap of 20% per component.

The hard cap constants confirm the intended ceiling: [6](#0-5) 

---

### Impact Explanation

Traders swapping through the affected bin pay a fee of up to `spreadFeeE6 + 65535/1e6` (≈ 6.5535% extra) instead of the fee bounded by `maxAdminSpreadFeeE6`. The excess fee accrues as spread surplus inside the pool and is collected by the admin via `collectPoolFees`. This is a direct loss of user principal (overpaid swap cost) caused by the pool admin exceeding the protocol's intended fee cap — an admin-boundary break with real fund loss.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly bounded by caps. The bypass requires only a single call to `setPoolBinAdditionalFees` with `addFeeBuyE6 = type(uint16).max` on the current active bin. No special conditions, timelocks, or multi-step sequences are required. Any pool admin can trigger this immediately after pool creation.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the check in `setPoolAdminFees`. The combined effective fee (`spreadFeeE6 + addFeeBuyE6`) should not exceed the hard cap:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    // addFee values are E6; cap them so total effective fee stays within hard limit
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that `spreadFeeE6 + addFeeBuyE6 ≤ HARD_MAX_SPREAD_FEE_E6` by reading the pool's current `spreadFeeE6` before accepting the new per-bin value.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 50_000` (5%) via `setFeeCaps`.
2. Pool admin calls `setPoolAdminFees(pool, 50_000, 0)` — accepted, base spread = 5%.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no revert**, sets per-bin additional fee to 6.5535% on bin 0.
4. A trader calls `swap(...)` routing through bin 0. The effective fee applied is `baseFeeX64 + 65535/1e6 * ONE_X64`, i.e. 5% + 6.5535% = **11.5535%**, more than double the intended 5% cap.
5. The excess fee accrues as spread surplus and is swept to the admin via `collectPoolFees`.
