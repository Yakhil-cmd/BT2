### Title
Stale `target_shard_index` from epoch-N layout used to index epoch-N+1 `outcome_roots`, producing an unverifiable `outcome_root_proof` after resharding — (`chain/client/src/view_client_actor.rs`)

### Summary

`Handler<GetExecutionOutcome>::handle` computes `target_shard_index` from the shard layout of the **outcome block's epoch (N)**, then calls `get_next_block_hash_with_new_chunk` which may return a block in **epoch N+1** with a different shard layout. The returned child `shard_id` (in epoch N+1's layout) shadows the outer variable, but `target_shard_index` is never recomputed. The Merkle path `merklize(&outcome_roots).1[target_shard_index]` is then indexed using the old epoch-N index into an `outcome_roots` array ordered by epoch N+1's layout, producing a structurally wrong `outcome_root_proof` that fails `verify_path`.

### Finding Description

The handler computes `target_shard_index` once, from epoch N's layout:

```rust
let epoch_id = *self.chain.get_block(&outcome_proof.block_hash)?.header().epoch_id();
let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id).into_chain_error()?;
let target_shard_id =
    account_id_to_shard_id(self.epoch_manager.as_ref(), &account_id, &epoch_id)
        .into_chain_error()?;
let target_shard_index = shard_layout
    .get_shard_index(target_shard_id)
    .map_err(Into::into)
    .into_chain_error()?;
``` [1](#0-0) 

Then `get_next_block_hash_with_new_chunk` is called. When it crosses an epoch boundary with a different shard layout, it replaces `shard_ids` with the children shards in the new layout and returns `(block_hash_in_N+1, child_shard_id_in_N+1)`:

```rust
shard_ids = shard_ids
    .into_iter()
    .flat_map(|id| {
        next_shard_layout.get_children_shards_ids(id).unwrap_or_else(|| { ... })
    })
    .collect();
shard_layout = next_shard_layout;
// ...
return Ok(Some((block_hash, shard_id)));  // shard_id is now a child in N+1's layout
``` [2](#0-1) 

Back in the handler, the returned `target_shard_id` shadows the outer binding (now a child shard in epoch N+1), but `target_shard_index` is **never updated**:

```rust
if let Some((h, target_shard_id)) = res {   // shadows outer target_shard_id
    outcome_proof.block_hash = h;
    let outcome_roots = self.chain.get_block(&h)?.chunks().iter()
        .map(|header| *header.prev_outcome_root())
        .collect::<Vec<_>>();
    // target_shard_index is still from epoch N's layout!
    Ok(GetExecutionOutcomeResponse {
        outcome_proof: outcome_proof.into(),
        outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone(),
    })
``` [3](#0-2) 

`outcome_roots` is ordered by epoch N+1's shard layout (one entry per chunk in block `h`), but `target_shard_index` is the position of the **parent** shard in epoch N's layout. In V2/V3 layouts, shard IDs and shard indices are decoupled, so the parent's index in epoch N is not the child's index in epoch N+1.

**Concrete V2 example** (from the test layout):

| Layout | Shards (ordered by index) | Shard 1 index |
|--------|--------------------------|---------------|
| V1 (epoch N) | [0, 1, 2, 3] | 1 |
| V2 (epoch N+1) | [3, 8, 4, 7] | — (shard 1 is gone) |

`get_children_shards_ids(1)` in V2 returns `[7, 8]`. In the new layout, shard 7 has index **3** and shard 8 has index **1**. [4](#0-3) 

`get_next_block_hash_with_new_chunk` returns `(h, shard_7)` (first child with a new chunk). `target_shard_index` = 1 (parent's old index). Shard 7's correct index in epoch N+1 = 3. The code returns `merklize(&outcome_roots).1[1]` instead of `merklize(&outcome_roots).1[3]` — a Merkle path for the wrong shard.

The bounds check at line 1174 does not catch this because the old index (1) is still within `outcome_roots.len()` (4): [5](#0-4) 

### Impact Explanation

The `outcome_root_proof` field of `GetExecutionOutcomeResponse` contains the Merkle path for the wrong shard index. A light client computing:

```python
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
```

gets a value that does not match `block_header_lite.inner_lite.outcome_root`, so `verify_path(...)` returns `false`. The proof is permanently unverifiable for any receipt whose receiver's shard was split at an epoch boundary. This breaks the client-proof invariant documented in the light client spec. [6](#0-5) 

### Likelihood Explanation

This triggers on any `EXPERIMENTAL_light_client_proof` / `GetExecutionOutcome` call for a receipt executed in epoch N when the shard layout changed at the N→N+1 boundary. This has already occurred on NEAR mainnet (Simple Nightshade V1 → V2 resharding). With dynamic resharding (V3) now enabled, it will recur at every future resharding event. No special attacker capability is required — any user who submitted a receipt before a resharding event and queries the proof afterward will observe the failure.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns, recompute `target_shard_index` using the shard layout of block `h` (epoch N+1), not the layout of the outcome block (epoch N):

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let next_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let next_shard_layout =
        self.epoch_manager.get_shard_layout(&next_epoch_id).into_chain_error()?;
    let target_shard_index = next_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

The `target_shard_id` returned by `get_next_block_hash_with_new_chunk` is already a valid shard ID in the new layout, so `get_shard_index` on the new layout will return the correct index.

### Proof of Concept

1. Configure a test-loop environment with resharding enabled (V2 or V3 layout).
2. Submit a receipt whose `receiver_id` maps to shard S in epoch N (e.g., shard 1 in a 4-shard V1 layout).
3. Let the receipt execute in epoch N. Record `receipt_id` and `receiver_id`.
4. Advance past the epoch boundary so resharding takes effect (epoch N+1 uses V2 layout with shards [3, 8, 4, 7]).
5. Call `view_client.handle(GetExecutionOutcome { id: TransactionOrReceiptId::Receipt { receipt_id, receiver_id } })`.
6. Retrieve the block at `execution_outcome.outcome_proof.block_hash` (in epoch N+1).
7. Assert:
   ```rust
   assert!(verify_path(
       block.header().outcome_root(),
       &execution_outcome.outcome_root_proof,
       &chunk_outcome_root,
   ));
   ```
   This assertion **fails** because `outcome_root_proof` uses index 1 (parent's old index) instead of index 3 (child shard 7's correct index in the new layout).

The existing test `check_outcome_proofs` in `test-loop-tests/src/tests/light_client.rs` does not cover a cross-epoch-resharding scenario and would catch this bug if extended with a resharding epoch boundary. [7](#0-6)

fn check_outcome_proofs(
    env: &mut TestLoopEnv,
    account_id: &AccountId,
    seed_hash: CryptoHash,
    tx: SignedTransaction,
) {
    let outcome = env.rpc_runner().execute_tx(tx, Duration::seconds(5)).unwrap();
    // Advance so the outcome's block is final and included in later blocks' merkle roots.
    env.rpc_runner().run_for_number_of_blocks(4);

    let mut ids = vec![TransactionOrReceiptId::Transaction {
        transaction_hash: outcome.transaction_outcome.id,
        sender_id: account_id.clone(),
    }];
    for receipt_outcome in &outcome.receipts_outcome {
        ids.push(TransactionOrReceiptId::Receipt {
            receipt_id: receipt_outcome.id,
            receiver_id: account_id.clone(),
        });
    }

    let mut rpc = env.rpc_node_mut();
    let view_client = rpc.view_client_actor();

    // The light client head and the block merkle root all proofs are checked against.
    let light_client_block = view_client
        .handle(GetNextLightClientBlock { last_block_hash: seed_hash })
        .unwrap()
        .unwrap();
    let light_client_head = light_client_block_hash(&light_client_block);
    let block_merkle_root = light_client_block.inner_lite.block_merkle_root;

    for id in ids {
        let execution_outcome = view_client.handle(GetExecutionOutcome { id }).unwrap();
        let outcome_proof = execution_outcome.outcome_proof;
        let block_proof = view_client
            .handle(GetBlockProof {
                block_hash: outcome_proof.block_hash,
                head_block_hash: light_client_head,
            })
            .unwrap();

        // outcome -> chunk outcome root -> block outcome root.
        let chunk_outcome_root =
            compute_root_from_path_and_item(&outcome_proof.proof, &outcome_proof.to_hashes());
        assert!(verify_path(
            block_proof.block_header_lite.inner_lite.outcome_root,
            &execution_outcome.outcome_root_proof,
            &chunk_outcome_root,
        ));

        // The light block header recomputes to the proof's block hash.
        assert_eq!(block_proof.block_header_lite.hash(), outcome_proof.block_hash);

        // block hash -> light client head's block merkle root.
        assert!(verify_hash(block_merkle_root, &block_proof.proof, outcome_proof.block_hash));
    }
