### Title
Attacker Can Bypass `__validate__` to Inject Invalid Invoke Transactions and Block Legitimate Ones via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway omits the `__validate__` entry-point check for any invoke transaction with `nonce == 1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that check is purely address-based (not type-filtered), an attacker who observes a victim's pending `DeployAccount` in the mempool can submit a forged invoke with `nonce = 1` and an arbitrary wrong signature for the same address. The gateway admits the forged invoke without running `__validate__`. The forged invoke then occupies `nonce = 1` in the mempool, causing the victim's legitimate invoke to be rejected with `DuplicateNonce`.

`account_tx_in_pool_or_recent_block` (mempool.rs line 697–700) returns `true` for **any** transaction type from the address:


The comment in `skip_stateful_validations` asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This assumption is exploitable: an attacker can be the one who places a transaction for the victim's address in the mempool.

**Attack sequence:**

1. Victim submits `DeployAccount` for address `A` (class_hash `X`, salt `Y`, constructor_calldata `Z`). The gateway runs `__validate_deploy__` (full execution for deploy_account), the signature is valid, and the transaction is admitted to the mempool.

2. Attacker observes the `DeployAccount` in the mempool and derives address `A` from the public parameters `(X, Y, Z)`.

3. Attacker submits `Invoke(sender=A, nonce=1, calldata=<anything>, signature=<garbage>)`.

4. Gateway stateful validation for the attacker's invoke:
   - `validate_nonce`: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap=200` → **passes**.
   - `validate_by_mempool` → `Mempool::validate_tx` → `validate_incoming_tx` (no duplicate tx_hash, `tx_nonce ≥ account_nonce`) + `validate_fee_escalation` (no existing tx at nonce=1 for address A yet) → **passes**.
   - `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` (victim's `DeployAccount` is in the pool) → returns **`true`**.
   - `run_validate_entry_point` with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` is **never called**.

5. Attacker's forged invoke (garbage signature) is admitted to the mempool at `(A, nonce=1)`.

6. Victim now submits their legitimate `Invoke(sender=A, nonce=1, valid_signature)`.

7. Gateway stateful validation for the victim's invoke:
   - `validate_by_mempool` → `validate_fee_escalation`: a transaction already exists at `(A, nonce=1)` (the attacker's) → returns `MempoolError::DuplicateNonce` (when fee escalation is disabled) or forces the victim to pay a strictly higher fee (when fee escalation is enabled).
   - Victim's legitimate invoke is **rejected** or must outbid the attacker's invalid transaction.

8. If the attacker's forged invoke is eventually executed by the batcher, `__validate__` runs inside the blockifier and fails (wrong signature), the transaction reverts, and the fee is charged to the victim's account.

---

### Impact Explanation

Two distinct harms:

**Admission of invalid transaction:** The gateway admits an invoke transaction with an arbitrary wrong signature into the mempool, violating the invariant that every mempool transaction has passed signature validation. This matches: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

**Rejection of valid transaction:** The victim's legitimate invoke with `nonce=1` is rejected with `DuplicateNonce` because the attacker's forged invoke already occupies that slot. The victim's transaction is blocked before sequencing.

**Secondary economic harm:** If the attacker's forged invoke is batched and executed, `__validate__` fails inside the blockifier, the transaction reverts, and the fee is charged to the victim's pre-funded account (the account is pre-funded before `DeployAccount` is submitted, since the address is deterministic).

---

### Likelihood Explanation

- The victim's `DeployAccount` parameters (`class_hash`, `contract_address_salt`, `constructor_calldata`) are fully public once the transaction is submitted to the gateway/mempool.
- The target address `A` is deterministically computable from those parameters via `calculate_contract_address`.
- The attacker only needs to submit a single forged invoke before the victim submits their invoke. No privileged access, no private key knowledge, and no special network position is required — only the ability to observe the mempool and submit transactions.
- The `max_nonce_for_validation_skip` default is `Nonce(Felt::ONE)`, so the window is exactly `nonce=1`, but that is precisely the nonce a user sends alongside their `DeployAccount` in the common UX pattern this feature is designed to support.

---

### Recommendation

1. **Filter by transaction type in `skip_stateful_validations`:** Instead of checking whether *any* transaction from the address is in the mempool, query specifically whether a `DeployAccount` transaction for address `A` is pending. The mempool should expose a typed query such as `deploy_account_in_pool(address)`.

2. **Alternatively, verify the deploy_account tx_hash:** The gateway already computes the `DeployAccount` tx_hash during conversion. Pass it through to `skip_stateful_validations` and verify that the mempool contains a transaction with that specific hash, not just any transaction from the address.

3. **Do not skip `__validate__` based solely on an address-level mempool presence check.** The current check is too coarse and can be satisfied by any transaction type, including ones submitted by adversaries.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
