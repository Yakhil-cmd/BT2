### Title
Cross-Chain Replay of Unprotected Legacy Transactions (v=27/28) — (`basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary

The ZKsync OS bootloader accepts pre-EIP-155 legacy transactions (v=27 or v=28) without any chain-ID binding. A raw signed transaction from any other EVM chain can be submitted verbatim to ZKsync OS and will pass signature verification, nonce checks, and execution — draining the victim's account.

### Finding Description

`LegacySignatureData::is_eip155()` classifies v=27 or v=28 as "unprotected": [1](#0-0) 

When `is_eip155()` returns `false`, `try_parse_and_hash_for_signature_verification` computes the signing hash as `keccak256(RLP([nonce, gasPrice, gasLimit, to, value, data]))` — the 6-field hash — and **never consults `expected_chain_id`**: [2](#0-1) 

The `expected_chain_id` parameter is only used in the EIP-155 branch (lines 97–114). For unprotected transactions it is silently ignored. No chain-ID check is added at the call site in `parse_and_compute_signed_hash` either: [3](#0-2) 

Contrast this with every typed transaction (EIP-2930, EIP-1559, EIP-4844, EIP-7702), each of which has an explicit `chain_id != expected_chain_id` guard immediately after parsing: [4](#0-3) 

In `validation_impl.rs`, ecrecover is called against `suggested_signed_hash`. Because the 6-field hash is chain-agnostic, ecrecover recovers the original signer's address identically on any chain: [5](#0-4) 

### Impact Explanation

An attacker who obtains a raw pre-EIP-155 signed transaction (v=27/28) from any EVM chain can submit it to ZKsync OS. If the victim's nonce on ZKsync OS matches the nonce in the replayed transaction and the victim holds funds there, the transaction executes and transfers/calls exactly as originally intended — direct loss of funds from the victim's account with no action required from the victim on ZKsync OS.

### Likelihood Explanation

Pre-EIP-155 transactions are rare in modern wallets but are not extinct: hardware wallets, old tooling, and some dApps still produce them. The nonce-match constraint is a real but not prohibitive barrier — an attacker can monitor both chains and wait for the nonce to align, or target fresh accounts (nonce=0 on both chains). The attacker needs no privileged access; submitting a raw RLP transaction to the bootloader is the normal unprivileged entry point.

### Recommendation

Reject unprotected legacy transactions (v=27 or v=28) entirely. If backward compatibility is required, at minimum add a deployment-time configuration flag that disables the unprotected path. The fix is a one-line guard in `try_parse_and_hash_for_signature_verification`:

```rust
if legacy_signature.is_eip155() == false {
    return Err(InvalidTransaction::InvalidEncoding.into()); // reject unprotected
}
```

### Proof of Concept

1. On Ethereum mainnet (chain_id=1), sign a legacy transfer with v=27 (pre-EIP-155): `RLP([nonce, gasPrice, gasLimit, to, value, data, 27, r, s])`.
2. Submit the identical raw bytes to a ZKsync OS instance configured with `expected_chain_id=324`.
3. `try_parse_and_hash_for_signature_verification` takes the `is_eip155() == false` branch, computes `keccak256(RLP([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the mainnet signing hash.
4. `validation_impl` calls ecrecover with this hash; it recovers the victim's address.
5. `recovered_from == from` passes; nonce and balance checks proceed normally.
6. Execution transfers funds from the victim's ZKsync OS account.

The existing test `test_legacy_wrong_chain_id_rejected` only covers EIP-155-protected transactions (where v encodes chain_id=1 and the test uses chain_id=2); it does not test the v=27/28 case against a different `expected_chain_id`, leaving this path untested and unguarded. [6](#0-5)
