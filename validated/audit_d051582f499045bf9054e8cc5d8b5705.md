### Title
Unauthorized ERC-20 Token Minting via Unguarded `deploy_erc20_token` + `ft_on_transfer` — (File: `engine/src/contract_methods/connector.rs`)

### Summary
`deploy_erc20_token` carries no caller-authorization check. Any NEAR account can register an arbitrary NEP-141 account ID as a bridged ERC-20 token. Once registered, the same attacker can call `ft_on_transfer` directly on the Aurora engine (with themselves as `predecessor_account_id`) to mint an unbounded amount of the corresponding ERC-20 token, because `receive_erc20_tokens` issues the EVM `mint` call from the engine's own admin address, bypassing the `onlyAdmin` guard on the Solidity contract.

---

### Finding Description

Compare with other privileged methods such as `factory_set_wnear_address` or `mirror_erc20_token`, which both call `require_owner_only`: [2](#0-1) [3](#0-2) 

**Step 2 — Attacker-controlled NEP-141 → ERC-20 registration**

`engine::deploy_erc20_token` deploys an `EvmErc20` contract whose `admin` is set to Aurora's own EVM address (`current_address(current_account_id)`), then writes the NEP-141 → ERC-20 bidirectional mapping into storage via `register_token`: [4](#0-3) 

**Step 3 — Minting via `ft_on_transfer`**

`ft_on_transfer` is a public NEAR entry point. When the caller (`predecessor_account_id`) is not the configured ETH-connector account, it unconditionally routes to `receive_erc20_tokens`, passing `predecessor_account_id` as the NEP-141 token identity: [5](#0-4) 

`receive_erc20_tokens` looks up the ERC-20 address for the supplied NEP-141 account, then issues an EVM `call` to the ERC-20 contract's `mint` selector **from the engine's own admin address** (`erc20_admin_address = current_address(current_account_id)`): [6](#0-5) 

Because the EVM call originates from the admin address, the `onlyAdmin` modifier in `EvmErc20.mint` passes: [7](#0-6) [8](#0-7) 

The `amount` field in `FtOnTransferArgs` is fully attacker-controlled JSON input; there is no on-chain verification that real NEP-141 tokens were actually transferred. [9](#0-8) 

---

### Impact Explanation

**Critical — Direct theft of funds / permanent bridge disruption.**

1. **Arbitrary ERC-20 minting**: An attacker registers their own NEAR account as a NEP-141 token, then calls `ft_on_transfer` with an arbitrary `amount`. The engine mints that many ERC-20 tokens to any EVM address the attacker specifies. If any DeFi protocol on Aurora accepts these tokens as collateral or in a liquidity pool, the attacker can drain it.

2. **Front-running legitimate bridge tokens**: An attacker can register a well-known NEP-141 account ID (e.g., `usdc.near`) before the legitimate bridge operator does. `register_token` rejects subsequent registrations with `TokenAlreadyRegistered`, permanently blocking the real bridge for that token. The attacker then mints unlimited fake USDC ERC-20 tokens and sells them to users.

---

### Likelihood Explanation

**High.** The entry point `deploy_erc20_token` is a standard public NEAR contract method callable by any NEAR account with no deposit or special permission required. The follow-up call to `ft_on_transfer` is equally open. No leaked keys, governance capture, or social engineering is needed. The attack is fully executable by any unprivileged NEAR account in a single block (two sequential transactions).

---

### Recommendation

Add a caller-authorization check to `deploy_erc20_token`, consistent with other privileged methods:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
``` [10](#0-9) 

Alternatively, introduce a dedicated role (e.g., a bridge-operator whitelist) if non-owner accounts must be permitted to register tokens.

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

/// Used to bridge NEP-141 tokens from NEAR to Aurora. On Aurora the NEP-141 becomes an ERC-20.
pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
