### Title
Gateway L2 Gas Price Threshold Uses Floor Division, Admitting Transactions Below Intended Minimum - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The stateful gateway's `validate_tx_l2_gas_price_within_threshold` computes the minimum acceptable L2 gas price using `Ratio::to_integer()`, which truncates (floors) the result. Because the threshold is a lower bound enforced in the protocol's favor, it should be computed with ceiling division. The floor causes the effective threshold to be 1 fri lower than intended whenever `min_gas_price_percentage × previous_block_l2_gas_price` is not exactly divisible by 100, allowing transactions with a gas price 1 fri below the intended minimum to pass gateway admission.

### Finding Description

In `validate_tx_l2_gas_price_within_threshold`:

```rust
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();          // ← floor, not ceiling
if tx_l2_gas_price.0 < threshold {
    return Err(...);
}
```

`Ratio::to_integer()` from `num_rational` returns the integer part of the ratio, which for positive values is the mathematical floor. For example, with `min_gas_price_percentage = 50` and `previous_block_l2_gas_price = 101`:

- True threshold = 50/100 × 101 = **50.5**
- `to_integer()` → **50** (floor)
- Correct ceiling → **51**

A transaction with `max_l2_gas_price = 50` satisfies `50 < 50 == false`, so it passes the check and is admitted to the mempool, even though the intended threshold is 51.

By contrast, the mempool's own fee-escalation check in `increased_enough` explicitly uses `div_ceil(100)` and is covered by the test `test_fee_escalation_rounds_up_required_bump`, confirming the codebase is aware that rounding direction matters here. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

The gateway's stateful admission check is the first line of defense against transactions whose L2 gas price is below the operator-configured minimum percentage of the previous block price. With floor rounding, the effective threshold is 1 fri lower than intended whenever the product is non-integer. A transaction with `max_l2_gas_price` equal to `floor(percentage × prev_price / 100)` — which is below the intended threshold — passes the gateway check and enters the mempool. This matches the **High** impact category: "Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."

The practical economic magnitude per transaction is at most 1 fri (the smallest unit), but the invariant is structurally broken: the protocol-favor rounding direction is wrong, and the gateway systematically admits a class of transactions it is configured to reject.

### Likelihood Explanation

The condition is triggered whenever `min_gas_price_percentage × previous_block_l2_gas_price` is not divisible by 100. With a typical `min_gas_price_percentage` of 50 and gas prices that are not multiples of 2, this occurs on every block. Any user who queries the previous block's gas price and submits a transaction with `max_l2_gas_price = floor(percentage × prev_price / 100)` will trigger the bypass without any special privileges.

### Recommendation

Replace `.to_integer()` with ceiling division so the threshold is computed in the protocol's favor:

```rust
// Before (floor):
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();

// After (ceiling):
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .ceil()
    .to_integer();
```

`Ratio::ceil()` returns the smallest integer greater than or equal to the ratio, matching the pattern already used in `convert_l1_to_l2_gas_price_round_up` and `l1_gas_to_sierra_gas_amount_round_up` elsewhere in the codebase. [4](#0-3) 

### Proof of Concept

1. Observe `min_gas_price_percentage = 50` (default-like config) and `previous_block_l2_gas_price = 101 fri`.
2. Intended threshold = ceil(50 × 101 / 100) = ceil(50.5) = **51**.
3. Actual threshold computed = floor(50 × 101 / 100) = floor(50.5) = **50**.
4. Submit a transaction with `max_l2_gas_price = 50`.
5. Gateway check: `50 < 50` → `false` → **no error, transaction admitted**.
6. The transaction is below the operator's intended minimum gas price but passes gateway admission. [5](#0-4) [6](#0-5)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
