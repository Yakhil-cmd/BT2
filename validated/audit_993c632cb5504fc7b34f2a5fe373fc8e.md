### Title
Unchecked Batch-Level Log Accumulation Causes `ArrayVec::push` Panic in Multiblock Batch Proving — (`basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`, `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

In the multiblock batch proving path, each block's L2-to-L1 logs are appended into a single batch-level `ArrayVec<Bytes32, 16384>`. The per-block log limit is also 16,384. Because `ArrayVec::push` panics on overflow and there is no batch-level log count guard, any multiblock batch whose total log count across all blocks exceeds 16,384 will abort proving with a panic, making a valid execution permanently unprovable.

---

### Finding Description

`ZKBatchDataKeeper` holds a batch-level log accumulator:

```rust
pub logs_storage: ArrayVec<Bytes32, 16384>,
``` [1](#0-0) 

After each block is processed in the multiblock proving path, its logs are appended to this accumulator:

```rust
io.logs_storage
    .apply_to_array_vec(&mut batch_data.logs_storage);
``` [2](#0-1) 

The implementation of `apply_to_array_vec` uses `ArrayVec::push`, which **panics** when the vector is full:

```rust
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())   // panics if capacity exceeded
    });
}
``` [3](#0-2) 

The per-block log limit is enforced in `check_for_block_limits`:

```rust
} else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
    Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
``` [4](#0-3) 

where `MAX_NUMBER_OF_LOGS = 16_384`: [5](#0-4) 

This limit is enforced **per block**, not per batch. In a multiblock batch with N blocks, the total log count can reach N × 16,384. The batch-level `ArrayVec` capacity is still only 16,384. There is no guard before `apply_to_array_vec` that checks `batch_data.logs_storage.remaining_capacity() >= block_log_count`. The `multiblock-batch` feature is a first-class production configuration: [6](#0-5) 

---

### Impact Explanation

When the total L2-to-L1 log count across all blocks in a multiblock batch exceeds 16,384, `apply_to_array_vec` panics inside the proving circuit. In a zkVM context, a panic aborts proof generation entirely. The batch execution is valid (each block individually satisfies all per-block limits), but it becomes permanently unprovable — a liveness/DoS impact on the proving system.

---

### Likelihood Explanation

The threshold is low: two blocks each with 8,193 logs (both individually valid) are sufficient to trigger the panic. L2-to-L1 logs are emitted via the L1 Messenger system hook, which is accessible to any user. L1 priority transactions that emit logs must be included by the operator. An operator running a multiblock batch configuration may not be aware that the batch-level capacity equals the per-block limit, making this easy to hit inadvertently or through deliberate log-heavy transactions.

---

### Recommendation

Replace `ArrayVec::push` with `try_push` in `apply_to_array_vec` and propagate the error, or enforce a batch-level log limit before appending. The batch-level capacity should be `num_blocks_per_batch × MAX_NUMBER_OF_LOGS`, or the per-block limit should be reduced to `MAX_NUMBER_OF_LOGS / max_blocks_per_batch`. A guard such as:

```rust
assert!(
    array_vec.remaining_capacity() >= self.list.len(),
    "batch log capacity exceeded"
);
```

would at minimum produce a clear diagnostic rather than an opaque panic, but the correct fix is to size the `ArrayVec` appropriately or enforce a batch-level cap.

---

### Proof of Concept

1. Enable the `multiblock-batch` feature.
2. Construct a batch of 2 blocks. In block 1, include transactions that emit 9,000 L2-to-L1 logs via the L1 Messenger hook (within the per-block limit of 16,384). In block 2, include transactions that emit 8,000 logs.
3. Run the multiblock proving path (`ZKHeaderStructurePostTxOpProvingMultiblockBatch::post_op`) for each block.
4. After block 2's `post_op`, `apply_to_array_vec` attempts to push the 16,385th element into the 16,384-capacity `ArrayVec` and panics.
5. Proof generation aborts; the valid 2-block batch is unprovable.
