### Title
Attacker Can Front-Run `skip_stateful_validations` to Inject Signature-Unverified Invoke Transactions and Block Legitimate User Invokes — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX feature that allows a user to submit a `deploy_account` + `invoke` pair simultaneously (`skip_stateful_validations`) can be exploited by an attacker who monitors the mempool. Once the attacker sees a victim's `deploy_account` (nonce 0) enter the mempool, they can immediately submit an `invoke` with nonce 1 from the victim's address carrying a garbage signature. Because `skip_stateful_validations` skips the `__validate__` entry-point call for exactly this nonce/state combination, the attacker's transaction is admitted to the mempool without any signature verification. The victim's legitimate invoke is then rejected as a duplicate nonce.

---

### Finding Description

**The skip-validate path**

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions are simultaneously true for an incoming `Invoke` transaction:

1. `tx.nonce() == Nonce(Felt::ONE)`
2. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When all three hold, the function returns `true`, which propagates to `run_validate_entry_point` as `skip_validate = true`, setting `execution_flags.validate = false`. [2](#0-1) 

Inside `StatefulValidator::perform_validations`, an `Invoke` transaction with `validate = false` returns immediately before the `__validate__` call — the account's signature is **never checked**. [3](#0-2) 

**The broken invariant**

The comment in the code states:

> "We verify that a deploy_account transaction exists for this account. It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

However, `account_tx_in_pool_or_recent_block` does **not** verify that the transaction in the pool is specifically a `deploy_account`. It returns `true` for **any** transaction from that address: [4](#0-3) [5](#0-4) 

**Attack sequence**

1. Victim Alice submits `deploy_account` (nonce 0) for address `A`. It passes gateway validation (nonce check, `__validate_deploy__` succeeds) and enters the mempool.
2. Attacker observes Alice's `deploy_account` in the mempool.
3. Attacker constructs an `Invoke` with `sender_address = A`, `nonce = 1`, and an **arbitrary garbage signature**. The attacker controls all other fields (resource bounds, calldata, tip).
4. Gateway processes the attacker's invoke:
   - Stateless checks pass (resource bounds, DA modes, etc.).
   - `validate_nonce`: `account_nonce = 0`, `tx_nonce = 1`, within `max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate tx_hash, nonce 1 ≥ account nonce 0 → passes.
   - `skip_stateful_validations`: `tx_nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` (Alice's deploy_account is in pool) → returns `true`.
   - `run_validate_entry_point`: `validate = false` → `__validate__` is **never called**; garbage signature is never checked.
5. Attacker's invalid invoke enters the mempool at `(A, nonce=1)`.
6. Alice submits her legitimate invoke (nonce 1). The mempool rejects it: `MempoolError::DuplicateNonce`. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Mempool/gateway admission accepts an invalid transaction**: The attacker's invoke with a garbage signature is admitted to the mempool without any signature verification. This matches the impact category: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

**Valid transaction rejected**: Alice's legitimate invoke with nonce 1 is rejected as `DuplicateNonce`. Alice is forced to either wait for the attacker's transaction to expire (TTL) or pay a fee-escalation premium to replace it — if fee escalation is enabled.

The attacker pays only the gas cost of submitting one transaction. The attack is repeatable: every time Alice deploys a new account, the attacker can block her first post-deployment invoke.

---

### Likelihood Explanation

- The mempool is observable by any network participant.
- The window between Alice's `deploy_account` entering the mempool and Alice's invoke being submitted is non-zero (two separate RPC calls, or even a single batch submission where the gateway processes them sequentially).
- The attacker needs no special privilege: any address can submit an invoke with `sender_address = A`.
- The attack requires no knowledge of Alice's private key; the garbage signature is never verified.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** for the address is present in the mempool. For example, expose a `deploy_account_in_pool(address)` query from the mempool that only returns `true` when the pending transaction for that address at nonce 0 is of type `DeployAccount`. This closes the gap between the intended invariant ("a deploy_account exists") and the actual check ("any transaction exists").

Alternatively, require that the `deploy_account` transaction hash be provided alongside the invoke (as the legacy `native_blockifier` path does via `deploy_account_tx_hash: Option<TransactionHash>`), and verify it is present in the mempool before skipping validation. [8](#0-7)

### Proof of Concept

```
# Setup
alice_address = compute_contract_address(salt, class_hash, ctor_calldata, deployer=0)

# Step 1: Alice submits deploy_account (nonce 0) — enters mempool
gateway.add_tx(alice_deploy_account_tx)

# Step 2: Attacker observes alice_address in mempool, constructs garbage invoke
attacker_invoke = InvokeV3 {
    sender_address: alice_address,
    nonce: 1,
    signature: [0xDEAD, 0xBEEF],   # garbage — never verified
    calldata: [...],
    resource_bounds: valid_bounds,
    ...
}

# Step 3: Attacker submits — gateway accepts without calling __validate__
# Conditions met: nonce==1, account_nonce==0, account_tx_in_pool_or_recent_block==true
gateway.add_tx(attacker_invoke)   # → ACCEPTED, no signature check

# Step 4: Alice submits her legitimate invoke (nonce 1)
gateway.add_tx(alice_invoke_nonce_1)
# → REJECTED: MempoolError::DuplicateNonce { address: alice_address, nonce: 1 }
```

The attacker's transaction occupies Alice's nonce-1 slot in the mempool. Alice's legitimate invoke is blocked until the attacker's transaction expires or Alice pays a fee-escalation premium.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
