Audit Report

## Title
Precision Loss and Denial-of-Service in ICRC-21 Consent Message Amount Display for Large Token Amounts - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
Two functions in the ICRC-21 consent message library fail for token amounts exceeding `u64::MAX`: `convert_tokens_to_string_representation` silently loses precision via `f64` conversion (Bug 1), and `nat_to_u64` returns a hard `GenericError` (Bug 2). For the ckETH ledger (u256 token, 18 decimals), any transfer exceeding ~18.4 ETH in wei triggers Bug 2, returning an error instead of a consent message to any caller of the public `icrc21_canister_call_consent_message` endpoint.

## Finding Description

**Bug 1 — Silent precision loss in `convert_tokens_to_string_representation`:** [1](#0-0) 

`Nat` wraps a `BigUint` (arbitrary precision). Calling `.to_f64()` silently loses precision for values beyond 2^53 (~9×10^15). For ckETH (18 decimals), `1_000_000_000_000_000_001` wei converts to `1.0_f64`, so the consent message displays `1 ckETH` instead of `1.000000000000000001 ckETH`. The error message in the `ok_or` branch even reads `"Failed to convert tokens to u64"` — a copy-paste error confirming this was not designed for large values.

**Bug 2 — Hard error in `nat_to_u64` for `FieldsDisplayMessage` mode:** [2](#0-1) 

`Value::TokenAmount` stores `amount: u64`: [3](#0-2) 

All four public methods — `add_amount`, `add_fee`, `add_allowance`, `add_existing_allowance` — call `nat_to_u64(amount)?` in `FieldsDisplayMessage` mode: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The call chain is: `icrc21_canister_call_consent_message` (public `#[update]`, no auth) → `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints` → `ConsentMessageBuilder::build()` → `add_amount(self.amount, ...)`: [8](#0-7) [9](#0-8) 

The ckETH ledger is compiled with the `u256-tokens` feature, meaning `Tokens = ic_icrc1_tokens_u256::U256`, and amounts can far exceed `u64::MAX`: [10](#0-9) 

The ckETH ledger proposal confirms 18 decimals and the target canister `ss2fx-dyaaa-aaaar-qacoq-cai`: [11](#0-10) 

`u64::MAX` = 18,446,744,073,709,551,615 wei ≈ 18.4 ETH. Any ckETH transfer above this threshold with `FieldsDisplay` requested returns `Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })` instead of a consent message.

## Impact Explanation

**Bug 2** is a denial-of-service on the `icrc21_canister_call_consent_message` endpoint for any ckETH (or other u256 ledger token) transfer exceeding ~18.4 ETH when `FieldsDisplay` is requested. ICRC-21 specifies `FieldsDisplay` as the recommended mode for hardware wallets and structured signing UIs. Wallets relying on this mode cannot obtain consent messages for large ckETH transfers, breaking the signing flow entirely. This constitutes a **High** impact: application/platform-level DoS with concrete user and protocol harm to a live mainnet ck-token ledger.

**Bug 1** causes the consent message to display a silently incorrect token amount, undermining the informed-consent guarantee that ICRC-21 is designed to provide. A user approving a transfer of `1_000_000_000_000_000_001` wei sees `1 ckETH` — the 1-wei difference is invisible. This is a **High** impact: significant ck-token security impact with concrete user harm (incorrect consent display for a financial transaction).

## Likelihood Explanation

The `icrc21_canister_call_consent_message` endpoint is a public `#[update]` callable by any principal with no authentication or rate limiting. ckETH is a live mainnet token. Transfers above 18.4 ETH are routine for institutional users and DeFi protocols. Any wallet or dApp using `FieldsDisplay` mode will encounter Bug 2 for such amounts. Bug 1 affects `GenericDisplay` mode for any amount where `f64` precision is insufficient (above ~9×10^15 wei, i.e., ~0.009 ETH precision loss at 1 ETH scale). Both bugs are deterministically reproducible with a single canister call.

## Recommendation

1. Replace `to_f64()` in `convert_tokens_to_string_representation` with exact arbitrary-precision decimal formatting using `BigUint` integer division and modulo by `10^decimals`, avoiding all floating-point precision loss.
2. Replace `Value::TokenAmount { amount: u64 }` with a `Nat`- or `String`-backed representation so that u256 token amounts are not truncated at the type level.
3. As an immediate mitigation, `nat_to_u64` should fall back to a string representation rather than returning a hard error, so consent messages remain available for large amounts even before the type is fixed.

## Proof of Concept

**Bug 2 (DoS) — deterministic local test:**
Call `icrc21_canister_call_consent_message` on the ckETH ledger (`ss2fx-dyaaa-aaaar-qacoq-cai`) or a local u256 ledger replica with:
```
method = "icrc1_transfer"
arg = TransferArg { amount = 18_446_744_073_709_551_616, to = <any account>, ... }
user_preferences.device_spec = FieldsDisplay
```
Expected (buggy) result: `Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })`.
Expected (correct) result: a `ConsentInfo` with a `FieldsDisplayMessage` containing the amount.

**Bug 1 (precision loss) — unit test:**
```rust
assert_eq!(
    convert_tokens_to_string_representation(
        Nat::from(1_000_000_000_000_000_001_u128),
        18
    ).unwrap(),
    "1.000000000000000001"  // fails: actual output is "1"
);
```
This is directly verifiable because `(1_000_000_000_000_000_001_u128 as f64) / 1e18 == 1.0` in IEEE 754 double precision.

fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,

        error_code: Nat::from(500_u64),

    })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
