### Title
Hardcoded NEAR Gas Constants in `ExitToNear` Precompile Can Cause Permanent Freezing of Bridged Funds - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile forwards fixed, hardcoded NEAR gas amounts to every `ft_transfer` and refund-callback cross-contract call it schedules. If NEAR protocol gas costs increase — exactly as EVM opcode repricing has historically broken Solidity `.transfer()` — those calls will silently fail with out-of-gas, while the user's tokens have already been burned on the Aurora side. The result is permanent loss of bridged funds with no recovery path.

---

### Finding Description

`engine-precompiles/src/native.rs` defines a private `costs` module with three hardcoded NEAR gas budgets:

```rust
// engine-precompiles/src/native.rs  lines 42-62
mod costs {
    // TODO(#483): Determine the correct amount of gas
    pub(super) const EXIT_TO_NEAR_GAS: EthGas = EthGas::new(0);          // EVM cost: free
    pub(super) const EXIT_TO_ETHEREUM_GAS: EthGas = EthGas::new(0);      // EVM cost: free

    pub(super) const FT_TRANSFER_GAS: NearGas = NearGas::new(10_000_000_000_000);   // 10 TGas
    pub(super) const FT_TRANSFER_CALL_GAS: NearGas = NearGas::new(70_000_000_000_000);
    pub(super) const EXIT_TO_NEAR_CALLBACK_GAS: NearGas = NearGas::new(10_000_000_000_000); // 10 TGas
    pub(super) const WITHDRAWAL_GAS: NearGas = NearGas::new(100_000_000_000_000);
}
``` [1](#0-0) 

These constants are consumed directly when the precompile builds its outgoing NEAR promise:

```rust
// lines 456-480
let attached_gas = if method == "ft_transfer_call" {
    costs::FT_TRANSFER_CALL_GAS
} else {
    costs::FT_TRANSFER_GAS          // ← always exactly 10 TGas
};

let transfer_promise = PromiseCreateArgs {
    ...
    attached_gas,
};

// callback (refund path)
PromiseCreateArgs {
    ...
    attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,  // ← always exactly 10 TGas
}
``` [2](#0-1) 

The execution sequence is:

1. User calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`).
2. ERC-20 tokens are **burned** (or ETH balance is **deducted**) inside the EVM — state is committed.
3. A NEAR promise is scheduled to call `ft_transfer` on the NEP-141 contract with exactly `FT_TRANSFER_GAS = 10 TGas`.
4. If that promise runs out of gas, it fails.
5. The `error_refund` callback (if the feature is compiled in) is then scheduled with `EXIT_TO_NEAR_CALLBACK_GAS = 10 TGas`. If that also runs out of gas, the refund never executes.

The `error_refund` feature is a **compile-time flag**, not a runtime guarantee:

```rust
#[cfg(feature = "error_refund")]
refund: refund_call_args(&exit_to_near_params, &exit_event),
#[cfg(not(feature = "error_refund"))]
refund: None,
``` [3](#0-2) 

When `error_refund` is disabled (the default in many build configurations), there is **no refund path at all** — a failed `ft_transfer` means the burned tokens are gone forever.

The same structural problem applies to `ExitToEthereum`, which uses `WITHDRAWAL_GAS = 100 TGas` for its `withdraw` call. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails due to insufficient attached gas:

- **Without `error_refund`**: tokens are burned on Aurora and never minted on NEAR. They are unrecoverable.
- **With `error_refund` but insufficient callback gas**: the refund callback itself fails, leaving the same outcome.

The EVM state change (burn) is committed before the NEAR promise is dispatched. There is no rollback mechanism at the NEAR layer once the promise gas budget is exhausted.

---

### Likelihood Explanation

**Medium.** NEAR protocol has repriced operations in past upgrades (storage reads, function-call overhead, etc.). The `FT_TRANSFER_GAS` value of 10 TGas was calibrated against a specific mainnet snapshot (noted in the comment at line 51-52). Any future NEAR protocol upgrade that increases the base cost of a cross-contract `ft_transfer` — or increases the cost of the storage operations it performs — can push the actual gas consumption above 10 TGas, triggering silent failure. The `EXIT_TO_NEAR_GAS` EVM cost is also hardcoded to `0`, meaning users pay no EVM gas for calling this precompile, making it trivially cheap to trigger at scale. [5](#0-4) 

---

### Recommendation

1. **Replace fixed gas constants with dynamic calculation**: query `env::prepaid_gas()` at runtime and forward a proportional fraction, similar to how the XCC precompile derives its NEAR gas budget from the EVM gas limit via `CROSS_CONTRACT_CALL_NEAR_GAS`.
2. **If static constants must be kept**, add a generous safety margin and document a governance process to update them when NEAR protocol gas schedules change.
3. **Ensure `error_refund` is always enabled** in production builds, and verify that `EXIT_TO_NEAR_CALLBACK_GAS` is sufficient even under repriced conditions.
4. **Charge non-zero EVM gas** for `EXIT_TO_NEAR_GAS` and `EXIT_TO_ETHEREUM_GAS` (currently `0`) to reflect the real NEAR gas cost consumed, preventing free-of-charge abuse. [6](#0-5) 

---

### Proof of Concept

**Scenario (no `error_refund` build):**

1. Alice holds 1000 `wBTC` ERC-20 tokens on Aurora.
2. Alice calls the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with flag `0x01` (ERC-20 exit), specifying her NEAR account as recipient.
3. The precompile burns 1000 `wBTC` from Alice's EVM balance — **committed**.
4. A NEAR promise is dispatched: `ft_transfer(receiver_id: alice.near, amount: 1000)` with `attached_gas = 10_000_000_000_000` (10 TGas).
5. After a NEAR protocol upgrade that increased `ft_transfer` base cost to 12 TGas, the promise fails with `GasExceeded`.
6. No callback exists (feature disabled). Alice's 1000 `wBTC` are permanently destroyed.

**Scenario (with `error_refund` but callback also starved):**

Steps 1–5 as above, but with `error_refund` enabled. The callback `exit_to_near_precompile_callback` is scheduled with `EXIT_TO_NEAR_CALLBACK_GAS = 10 TGas`. If the callback's own NEAR execution (which calls `Engine::refund_on_error` → EVM re-mint) also exceeds 10 TGas under the new gas schedule, the refund fails and Alice's funds are still permanently lost.
