The code confirms this is a real vulnerability. Let me trace it precisely.

## Root Cause

**`SetBlockList` silently no-ops when `Identity == nil`:** [1](#0-0) 

When `h.Identity == nil`, the function returns `nil` immediately — the comment even documents this as intentional ("don't fail if the identity is not set"). The `h.blocklist` map is never updated and remains the empty map set at construction time.

**`ProcessProposalHandler` fast-paths to ACCEPT when blocklist is empty:** [2](#0-1) 

An empty `h.blocklist` causes every proposal to be unconditionally accepted, regardless of which addresses signed the transactions.

## The Bypass Path

```
Admin stores encrypted blocklist (MsgStoreBlockList)
        ↓
EndBlocker → RefreshBlockList → SetBlockList(blob)
        ↓
Identity == nil → return nil (blocklist stays empty)
        ↓
ProcessProposalHandler: len(blocklist)==0 → ACCEPT all proposals
        ↓
Blocklisted attacker's tx included in block
```

The `BlockAddressesDecorator` AnteHandler in `block_address.go` only runs during `CheckTx` (guarded by `ctx.IsCheckTx()`), not during `DeliverTx`: [3](#0-2) 

So there is no fallback enforcement layer at execution time.

## Assessment

### Title
Validators Without e2ee Identity Silently Bypass Blocklist in ProcessProposal — (`app/proposal.go`)

### Summary
A validator node started without a valid age e2ee identity (`ProposalHandler.Identity == nil`) will never populate its in-memory blocklist. `ProcessProposalHandler` then fast-paths to `ACCEPT` every proposal, allowing transactions from blocklisted addresses to be included in any block proposed by that validator.

### Finding Description
`SetBlockList` is the sole mechanism that populates `h.blocklist`. Its first guard is:

```go
if h.Identity == nil {
    return nil   // silent no-op; h.blocklist stays empty
}
``` [1](#0-0) 

Even when the admin has stored a non-empty encrypted blocklist on-chain, `RefreshBlockList` calling `SetBlockList` with that blob will silently succeed without decrypting or updating the map. `ProcessProposalHandler` then hits:

```go
if len(h.blocklist) == 0 {
    return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
}
``` [4](#0-3) 

The `noneIdentity` type exists in the same file as a dummy that defers failure to decryption time, but it is never used when `Identity` is simply `nil`: [5](#0-4) 

### Impact Explanation
Blocklist enforcement is non-universal. Any validator that omits e2ee identity configuration becomes a blind spot: a blocklisted address can submit transactions and, when that validator is the CometBFT proposer, those transactions will be included in a committed block. The `BlockAddressesDecorator` only fires on `CheckTx`, not `DeliverTx`, so there is no execution-time safety net. [3](#0-2) 

This is a **High** impact: bypass of the on-chain block-list authorization check.

### Likelihood Explanation
Moderate. Validators that have not configured e2ee (a non-default, operator-level setup step) are silently non-enforcing. The attacker only needs to observe which proposer slot belongs to such a validator and submit their transaction timed to that slot — no key material or privileged access required.

### Recommendation
Replace the silent `return nil` with an explicit enforcement posture. Two options:

1. **Use `noneIdentity` as the default** instead of `nil`. This causes `age.Decrypt` to return `ErrIncorrectIdentity`, making `SetBlockList` return an error that the caller (`RefreshBlockList`) can log/panic on, alerting the operator.
2. **Reject proposals when blocklist cannot be loaded**: if `Identity == nil` and the on-chain blob is non-empty, `ProcessProposalHandler` should `REJECT` rather than `ACCEPT`.

### Proof of Concept
```go
handler := NewProposalHandler(txDecoder, nil /*Identity=nil*/, addressCodec)
blob := encryptBlocklist([]string{"blocked-address"}) // non-empty encrypted blob
err := handler.SetBlockList(blob)
// err == nil, but handler.blocklist is still empty
resp, _ := handler.ProcessProposalHandler()(ctx, reqContainingBlockedAddrTx)
// resp.Status == ACCEPT  ← blocklisted tx accepted
``` [6](#0-5) 

---

**Verdict: Valid High vulnerability** — concrete, locally testable, no privileged preconditions beyond a validator omitting e2ee identity configuration.
