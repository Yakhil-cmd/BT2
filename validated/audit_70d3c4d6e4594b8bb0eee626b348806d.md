### Title
Incorrect Multi-NFT Royalty Calculation in WalletConnect Offer Confirmation Understates User Cost - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

---

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` applies an incorrect formula when computing the displayed "total with royalties" for a `take_offer` WalletConnect signing confirmation that involves multiple NFTs. It divides the fungible amount by the number of NFTs before applying each NFT's royalty percentage, understating the displayed cost by a factor of N. A user shown this confirmation may approve a transaction where significantly more XCH is deducted than the confirmation indicated.

---

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect `take_offer` command display to compute `amountWithRoyalties` — the total the user will spend including creator royalties — shown in the signing confirmation dialog before the user approves. [1](#0-0) 

The current formula:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This computes:

```
royaltyAmount = (amount / N) × Σ(royaltyPercentage_i) / 10_000
```

The correct formula — consistent with how `calculateNFTRoyalties` handles the single-NFT case and how Chia's protocol applies each NFT's royalty to the full fungible trade value — is:

```
royaltyAmount = amount × Σ(royaltyPercentage_i) / 10_000
``` [2](#0-1) 

The single-NFT path in `calculateNFTRoyalties` applies `royaltyPercentage` to the full `amount` with no division. The multi-NFT path in `formatAmountWithRoyalties` silently divides by N, producing a result that is N times smaller than the actual royalty the backend will charge.

The `amountWithRoyalties` field produced by this function is surfaced directly in the WalletConnect signing confirmation via `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay`: [3](#0-2) [4](#0-3) 

The test suite encodes the incorrect behavior as the expected value, confirming the bug is present and untested against the correct formula: [5](#0-4) 

---

### Impact Explanation

**High** — The WalletConnect signing confirmation is the sole security gate between a dApp-initiated `take_offer` request and the user's wallet. When the offer contains N ≥ 2 NFTs with royalties, the displayed "total with royalties" is understated by a factor of N. The actual on-chain transaction deducts the correct (larger) royalty amount as computed by the Chia backend. The user approves based on a materially lower figure than what is actually spent, constituting approval of the wrong amount.

---

### Likelihood Explanation

Any unprivileged actor can craft a `take_offer` WalletConnect request containing multiple NFTs with non-zero royalty percentages. No special access, leaked keys, or social engineering is required beyond the normal WalletConnect dApp connection flow. The bug is triggered automatically whenever `royaltyPercentages.length > 1`.

---

### Recommendation

Remove the `splitAmount` division. Apply each NFT's royalty percentage to the full `amount`:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```

Update the corresponding test expectations to reflect the corrected values.

---

### Proof of Concept

**Scenario**: WalletConnect dApp sends `take_offer` for 2 NFTs (NFT₁ royalty = 500 bp = 5%, NFT₂ royalty = 10 bp = 0.1%) in exchange for 1,000,000,000,000 mojos (1 XCH).

**Current (buggy) display**:
- `splitAmount` = 1,000,000,000,000 / 2 = 500,000,000,000
- royalty₁ = 500,000,000,000 × 500 / 10,000 = 25,000,000,000
- royalty₂ = 500,000,000,000 × 10 / 10,000 = 500,000,000
- `amountWithRoyalties` shown = **1.0255 XCH**

**Correct value** (each royalty on full amount):
- royalty₁ = 1,000,000,000,000 × 500 / 10,000 = 50,000,000,000
- royalty₂ = 1,000,000,000,000 × 10 / 10,000 = 1,000,000,000
- Correct total = **1.051 XCH**

The user approves believing they spend **1.0255 XCH**; the blockchain deducts **1.051 XCH** — a silent overcharge of **0.0255 XCH** (≈2.5%) per NFT added to the bundle. With higher royalty percentages or more NFTs, the gap grows proportionally.

  amounts: Record<string, bigint>,

  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
