### Title
Unbounded Loop in `calculate_pubdata_used_by_tx()` Called Multiple Times Per Transaction Without Native Resource Charging — (File: `basic_system/src/system_implementation/flat_storage_model/storage_cache.rs`, `basic_system/src/system_implementation/flat_storage_model/account_cache.rs`)

---

### Summary

Both `NewStorageWithAccountPropertiesUnderHash::calculate_pubdata_used_by_tx()` and `NewModelAccountCache::calculate_pubdata_used_by_tx()` iterate over every pending storage/account change accumulated since the last transaction commit. Neither loop charges any native (RISC-V cycle) resources for the iteration work. These functions are invoked at least twice per transaction — once after execution and once in the refund phase — via the call chain `check_enough_resources_for_pubdata` → `net_pubdata_used` → `pubdata_used_by_tx`. An attacker who maximises the number of warm-slot `SSTORE` operations within a single transaction forces these loops to perform O(n log n) unaccounted work on every pubdata check, creating a resource-accounting divergence between charged and actual native cost.

---

### Finding Description

`calculate_pubdata_used_by_tx()` in `storage_cache.rs` iterates over `self.0.cache.iter_altered_since_commit()`:

```rust
pub fn calculate_pubdata_used_by_tx(&self) -> u32 {
    let mut visited_elements = BTreeSet::new_in(self.0.alloc.clone());
    let mut pubdata_used = 0u32;
    for element_history in self.0.cache.iter_altered_since_commit() {   // ← unbounded
        ...
        if visited_elements.contains(element_key) { continue; }
        visited_elements.insert(element_key);                           // O(log n) each
        ...
        pubdata_used += ValueDiffCompressionStrategy::optimal_compression_length(...) as u32;
    }
    pubdata_used
}
```

The parallel function in `account_cache.rs` has the same structure over `self.cache.iter_altered_since_commit()`.

Neither function charges `resources` for the loop body. The collection iterated — `pending_updated_elements` in `HistoryMap` — grows by one entry per storage write (and per `storage_touch` call, per the TODO comment in `EthereumStorageCache::touch`).

The call chain that reaches these loops on every transaction:

```
check_enough_resources_for_pubdata()          [gas_helpers.rs]
  └─ get_resources_to_charge_for_pubdata()
       └─ system.net_pubdata_used()           [io_subsystem.rs:397-400]
            └─ self.storage.pubdata_used_by_tx()  [flat_storage_model/mod.rs:118-121]
                 ├─ account_data_cache.calculate_pubdata_used_by_tx()
                 └─ storage_cache.calculate_pubdata_used_by_tx()
```

`check_enough_resources_for_pubdata` is called:
- After execution body (ZK flow, `mod.rs` line 880)
- In `before_refund` via `get_resources_to_charge_for_pubdata` (line 415)
- After L1 transaction execution (line 714)

So the loop runs **at least twice per transaction**, each time scanning the full set of pending changes.

---

### Impact Explanation

**Vulnerability class**: Resource accounting bug / valid-execution unprovability.

The native resource (RISC-V cycle) cost of iterating over `n` pending storage entries — including `BTreeSet::contains`, `BTreeSet::insert` (each O(log n)), and `optimal_compression_length` — is never charged to the transaction. The transaction pays for each `SSTORE` in ergs and native cost, but pays nothing for the repeated full-scan of the accumulated change set during pubdata accounting.

A transaction that writes to `n` warm storage slots (100 gas each) forces the system to perform O(n log n) uncharged RISC-V cycles per pubdata check call, and the check is called multiple times. If the block's native resource budget is exhausted by this uncharged work, the block becomes unprovable, or the sequencer's forward execution diverges from the prover's resource accounting — a forward/proving divergence.

---

### Likelihood Explanation

Any unprivileged EVM transaction sender can trigger this by:
1. Pre-warming many storage slots (via access lists or prior reads, costing 2100 gas cold → 100 gas warm).
2. Issuing a large number of warm `SSTORE` operations within a single transaction.
3. Each write adds one entry to `pending_updated_elements`; each subsequent call to `calculate_pubdata_used_by_tx()` scans all of them.

No privileged access, oracle manipulation, or governance control is required. The attack is bounded by the block gas limit but the uncharged native overhead scales super-linearly relative to the charged cost.

---

### Recommendation

Charge native resources proportional to the number of entries iterated inside `calculate_pubdata_used_by_tx()`. One approach is to maintain a running pubdata counter that is updated incrementally on each write (as is already done for logs in `logs_storage.rs` via `calculate_pubdata_used_by_tx` returning a pre-computed delta), eliminating the need for a full scan. Alternatively, add a per-entry native charge inside the loop body so that the iteration cost is reflected in the transaction's resource consumption.

    pub fn calculate_pubdata_used_by_tx(&self) -> u32 {
        let mut visited_elements = BTreeSet::new_in(self.0.alloc.clone());

        let mut pubdata_used = 0u32;
        for element_history in self.0.cache.iter_altered_since_commit() {
            // Elements are sorted chronologically

            let element_key = element_history.key();

            // we publish preimages for account details, so no need to publish hash
            if element_key.address == ACCOUNT_PROPERTIES_STORAGE_ADDRESS {
                continue;
            }

            // Skip if already calculated pubdata for this element
            if visited_elements.contains(element_key) {
                continue;
            }
            visited_elements.insert(element_key);

            let current_value = element_history.current().value();
            let initial_value = element_history.initial().value();
            let at_tx_start_value = element_history.committed().value();

            // If the current value is resetting to the initial one,
            // we don't consider this diff in the pubdata charging.
            // This change will be optimized away, so it's actually reducing
            // pubdata.
            if current_value == initial_value {
                continue;
            }

            if at_tx_start_value != current_value {
                // TODO(EVM-1074): use tree index instead of key for repeated writes
                pubdata_used += 32; // key
                pubdata_used += ValueDiffCompressionStrategy::optimal_compression_length(
                    at_tx_start_value,
                    current_value,
                ) as u32;
            }
        }

        pubdata_used
    }
