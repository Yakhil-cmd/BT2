### Title
Gateway Stateful Validator Uses Previous Block's L2 Gas Price for Fee Bound Validation, Causing Admission of Transactions That Fail Execution or Rejection of Valid Transactions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` and `run_validate_entry_point` both build their fee-check context from the **previous** committed block's L2 gas price. The batcher executes transactions against the **next** block's L2 gas price, which is computed independently by the consensus orchestrator via EIP-1559 mechanics. The mismatch means the gateway can admit V3 transactions whose `max_price_per_unit` is below the actual execution block's price (causing a reverted pre-validation failure in the batcher) or reject V3 transactions whose `max_price_per_unit` is above the next block's price but below the previous block's price (valid transactions incorrectly denied admission).

---

### Finding Description

**Root cause — `validate_resource_bounds`:**

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
```

The developer-authored TODO confirms the wrong price is used. The threshold computed is:

```
threshold = (min_gas_price_percentage / 100) * previous_block_l2_gas_price
```

With the production default `min_gas_price_percentage = 100`, the threshold equals the previous block's price exactly. [1](#0-0) [2](#0-1) 

**Root cause — `run_validate_entry_point`:**

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(
    block_info,   // gas_prices still from the previous block
    ...
);
```

Only `block_number` is incremented; all gas prices remain those of the previous block. The blockifier's `check_fee_bounds` inside `StatefulValidator::validate` therefore compares `tx.max_price_per_unit` against the **previous** block's L2 gas price, not the next block's. [3](#0-2) 

**Correct price used by the batcher:**

The consensus orchestrator computes the next block's L2 gas price via EIP-1559 (`calculate_next_base_gas_price`) and places it in `ProposalInit.l2_gas_price_fri`. `initiate_build` converts this into `ProposeBlockInput.block_info` via `convert_to_sn_api_block_info`, and the batcher's `BlockBuilderFactory::preprocess_and_create_transaction_executor` builds a `BlockContext` from that `block_info`. The blockifier's `check_fee_bounds` during actual execution therefore uses the **next** block's L2 gas price. [4](#0-3) [5](#0-4) 

**EIP-1559 price change magnitude:**

`gas_price_max_change_denominator = 48`, so the L2 gas price can shift by up to `price / 48` (~2.08%) per block in either direction. [6](#0-5) 

---

### Impact Explanation

**Scenario A — Gateway admits a transaction that the batcher rejects:**

1. Previous block L2 gas price: `P`.
2. Transaction sets `max_price_per_unit = P` (exactly meets the 100% threshold).
3. Gateway admits the transaction.
4. Next block L2 gas price: `P + P/48 ≈ 1.021 P` (full block, EIP-1559 increase).
5. Batcher's `check_fee_bounds`: `P < 1.021 P` → `InsufficientResourceBounds` pre-validation error.
6. Transaction is reverted; user is charged the minimum fee for a failed transaction.

This matches **"High. Mempool/gateway/RPC admission accepts invalid transactions."**

**Scenario B — Gateway rejects a transaction the batcher would accept:**

1. Previous block L2 gas price: `P`.
2. Next block L2 gas price: `P - P/48 ≈ 0.979 P` (underutilized block, EIP-1559 decrease).
3. Transaction sets `max_price_per_unit = 0.99 P` (above next block price, below previous block price).
4. Gateway rejects: `0.99 P < P` (threshold = 100% of previous block price).
5. The transaction would have executed successfully in the next block.

This matches **"High. Mempool/gateway/RPC admission rejects valid transactions before sequencing."**

---

### Likelihood Explanation

- The L2 gas price changes every block via EIP-1559. Any sustained period of full blocks (common during high activity) causes a monotonically increasing price, making Scenario A a routine occurrence.
- The default `min_gas_price_percentage = 100` leaves zero tolerance: a transaction priced at exactly the previous block's price will fail in the next block whenever the price rises even slightly.
- No existing guard in the gateway uses the actual next block's L2 gas price. The `StatelessTransactionValidator` only checks a static `min_gas_price` floor (8 Gwei), which is unrelated to the dynamic block price.
- The issue is unprivileged: any user submitting a V3 (`AllResources`) transaction with `max_price_per_unit` near the current block price can trigger either scenario without any special access. [7](#0-6) [8](#0-7) 

---

### Recommendation

1. **Use the next block's L2 gas price** in `validate_resource_bounds`. The consensus orchestrator already exposes `calculate_next_l2_gas_price`; the gateway should query or receive this value rather than reading the previous block's header. The existing TODO comment (`// TODO(Arni): getnext_l2_gas_price from the block header`) confirms this is the intended fix.

2. **Propagate the correct gas prices into `run_validate_entry_point`**. When constructing the `BlockContext` for blockifier validation, update `block_info.gas_prices` to reflect the anticipated next block's prices, not just the block number.

3. **Consider a tolerance band** in `validate_tx_l2_gas_price_within_threshold` (e.g., `min_gas_price_percentage = 95`) to absorb the ±2% EIP-1559 swing even before the exact next-block price is available.

---

### Proof of Concept

```
State:
  previous_block_l2_gas_price = 100 FRI
  gas_price_max_change_denominator = 48
  min_gas_price_percentage = 100 (default)

Step 1 — User submits V3 invoke with max_price_per_unit = 100 FRI.

Step 2 — Gateway stateful validation:
  threshold = (100/100) * 100 = 100 FRI
  100 >= 100  →  PASS  →  transaction admitted to mempool.

Step 3 — Consensus orchestrator computes next block price:
  gas_used > gas_target  →  price increases
  next_block_l2_gas_price = 100 + 100/48 ≈ 102 FRI

Step 4 — Batcher executes transaction with block_context.gas_prices.l2_gas_price = 102 FRI:
  check_fee_bounds: max_price_per_unit(100) < actual_gas_price(102)
  → InsufficientResourceBounds error
  → transaction reverts; user charged minimum fee.

Invariant broken:
  Gateway admitted a transaction that the blockifier rejects at execution time,
  because the gateway validated against the previous block's price (100 FRI)
  while execution used the next block's price (102 FRI).
``` [9](#0-8) [10](#0-9) [11](#0-10)

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
