### Title
Gateway Skips `__validate__` Signature Verification for Invoke Transactions Targeting Accounts With a Pending Deploy-Account, Enabling Unsigned Transaction Injection — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway unconditionally bypasses the account's `__validate__` entry point (the only on-chain signature check) for any invoke transaction whose nonce is `1` and whose sender address already appears in the mempool or a recent block. An unprivileged attacker who observes a victim's pending `deploy_account` transaction can submit a forged invoke with nonce `1` for the victim's address, carrying an arbitrary or empty signature. The gateway accepts it without calling `__validate__`, and the transaction enters the mempool. When the batcher later executes it, `__validate__` fails, the fee is charged, and—if the attacker used fee escalation—the victim's legitimate invoke is silently displaced.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bound checks but **before** `run_validate_entry_point`: [1](#0-0) 

When it returns `true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, … }` and calls `blockifier_validator.validate(account_tx)`, which immediately returns `Ok(())` for invoke transactions without ever calling `__validate__`: [2](#0-1) [3](#0-2) 

The skip condition is:

```
tx.nonce() == 1  AND  account_nonce == 0  AND  account_tx_in_pool_or_recent_block(sender) == true
``` [4](#0-3) 

The code comment claims the pool check is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, `account_tx_in_pool_or_recent_block` checks only whether **any** transaction for the address exists in the pool or state — it does **not** verify the type is `DeployAccount`: [5](#0-4) 

Because a legitimate `deploy_account` for address `V` passes full execution-based validation (including `__validate_deploy__`) and enters the mempool, the pool check immediately becomes `true` for `V`. Any attacker who knows `V` can now submit an invoke with nonce `1` and an arbitrary signature; the gateway will skip `__validate__` and admit it.

The only other pre-admission check, `validate_by_mempool`, performs nonce-gap and duplicate checks but **no signature verification**: [6](#0-5) 

The batcher constructs transactions with `new_for_sequencing`, which always sets `validate: true` and `strict_nonce_check: true`: [7](#0-6) 

So the batcher **will** call `__validate__` and reject the forged invoke — but only after it has already displaced the victim's legitimate invoke via fee escalation and charged a fee from the victim's pre-funded balance.

---

### Impact Explanation

**Impact: High** — Mempool/gateway admission accepts invalid transactions (wrong signature) before sequencing.

Concrete consequences:

1. **Unsigned invoke admitted to mempool.** The gateway accepts a forged invoke for any account whose `deploy_account` is pending, without any signature check.
2. **Legitimate invoke displaced.** The attacker submits the forged invoke with a tip slightly above the victim's legitimate invoke. The mempool's fee-escalation logic replaces the victim's transaction with the attacker's. The victim's invoke is silently dropped.
3. **Fee charged on failure.** When the batcher executes the forged invoke, `__validate__` fails and the fee is charged against the victim's pre-funded balance (required for the `deploy_account` to succeed). The victim loses funds and their first post-deploy invoke is never executed.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker needs only to observe a `deploy_account` transaction in the mempool (or know the victim's address and that a deploy is pending). Mempool contents are observable via the public RPC.
- No privileged access, special keys, or prior on-chain state is required.
- The attack window is the time between the `deploy_account` entering the mempool and the batcher committing the block containing it — typically seconds to minutes.
- The attacker must outbid the victim's invoke tip, which is a small economic cost.

---

### Recommendation

**Short term:** In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a check that the pool contains a transaction of type `DeployAccount` for the sender address. This ensures the skip is only triggered when a genuine deploy is pending, not for arbitrary pool membership.

**Long term:** Consider requiring the `deploy_account` and its companion invoke to be submitted as an atomic bundle, or cryptographically binding the invoke to the deploy (e.g., by including the deploy tx hash in the invoke's calldata and verifying it in `__validate__`). This eliminates the need for the skip entirely.

### Proof of Concept

```
1. Victim generates a new account at address V and submits a valid deploy_account
   transaction (correct signature, sufficient pre-funded balance).

2. The gateway executes the deploy_account through StatefulValidator::perform_validations
   (which calls self.execute(tx)), __validate_deploy__ passes, and the deploy_account
   enters the mempool.

3. Attacker observes V in the mempool (e.g., via get_mempool_snapshot RPC).

4. Attacker constructs an invoke transaction:
     sender_address = V
     nonce          = 1
     signature      = [0x0, 0x0]   // arbitrary / empty
     tip            = victim_tip + 1  // to win fee escalation

5. Attacker submits the invoke to the gateway's add_tx endpoint.

6. Gateway flow:
   a. stateless_tx_validator.validate() — passes (signature size ≤ max, nonce DA mode L1, etc.)
   b. convert_rpc_tx_to_internal_and_executable_txs() — succeeds
   c. extract_state_nonce_and_run_validations():
        get_nonce_from_state(V)  → Nonce(0)   // account not yet deployed
        validate_state_preconditions():
          validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap  → OK
        validate_by_mempool()  → OK (no signature check)
        skip_stateful_validations():
          tx.nonce() == 1  ✓
          account_nonce == 0  ✓
          account_tx_in_pool_or_recent_block(V) == true  ✓  (deploy_account is in pool)
          → returns true
        run_validate_entry_point(skip_validate=true):
          ExecutionFlags { validate: false, … }
          StatefulValidator::perform_validations → returns Ok(()) immediately
   d. Forged invoke is added to the mempool.

7. If the victim also submitted a legitimate invoke with nonce=1 and lower tip,
   the mempool's fee-escalation logic replaces it with the attacker's forged invoke.

8. Batcher builds a block:
   - Executes deploy_account (nonce 0): succeeds, V is now deployed.
   - Executes forged invoke (nonce 1): __validate__ is called with validate=true,
     signature [0x0, 0x0] fails verification, transaction is rejected, fee is charged.

9. Victim's legitimate invoke is gone; victim's balance is reduced by the failed fee.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
