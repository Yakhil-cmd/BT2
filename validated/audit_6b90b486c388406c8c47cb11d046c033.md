Audit Report

## Title
`add_expiration` Lossy `u64→i64` Cast Silently Corrupts ICRC-21 `FieldsDisplayMessage` Expiration to Epoch 0 — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
In `ConsentMessage::add_expiration`, the `FieldsDisplayMessage` arm casts a `u64` nanosecond timestamp to `i64` before dividing by 10⁹. For any `expires_at > i64::MAX` (e.g., `u64::MAX`), the cast wraps to `-1`, integer division toward zero yields `0`, and the final `as u64` cast stores `0` — Unix epoch — in `Value::TimestampSeconds`. A hardware wallet implementing the ICRC-21 `FieldsDisplay` spec renders this as "Jan 1 1970 00:00:00 UTC" while the on-chain approval stores the original far-future timestamp unchanged, enabling a malicious dApp to deceive users into granting permanent allowances.

## Finding Description
The defective code is at lines 264–271 of `packages/icrc-ledger-types/src/icrc21/responses.rs`:

```rust
let seconds = (expires_at as i64) / 10_i64.pow(9);
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds as u64 },
))
```

For `expires_at = u64::MAX`:
1. `u64::MAX as i64` → `-1` (defined wrapping behavior in Rust)
2. `-1_i64 / 1_000_000_000_i64` → `0` (truncation toward zero)
3. `0_i64 as u64` → `0`

The `GenericDisplayMessage` arm at lines 232–233 has the same cast but subsequently calls `replace_nanosecond(4294967295)` which fails and returns the string `"Invalid nanosecond: 4294967295"` — visibly wrong, not silently wrong. The `FieldsDisplayMessage` arm only computes `seconds` and stores it directly, so the silent corruption is exclusive to that path.

The `approve` guard at `rs/ledger_suite/common/ledger_core/src/approvals.rs:247` checks `expires_at.unwrap_or_else(remote_future) <= now`, where `remote_future()` returns `TimeStamp::from_nanos_since_unix_epoch(u64::MAX)`. Since `u64::MAX <= now` is always false for any realistic `now`, the ledger accepts and stores `expires_at = u64::MAX` unchanged. `ApproveArgs.expires_at` is `Option<u64>` with no upper bound.

**Exploit flow:**
1. Malicious dApp constructs `icrc2_approve` with `expires_at = u64::MAX` and calls `icrc21_canister_call_consent_message` (unprivileged query).
2. Ledger returns `FieldsDisplayMessage` containing `TimestampSeconds { amount: 0 }`.
3. Hardware wallet (Ledger, Keystone) implementing ICRC-21 `FieldsDisplay` renders "Approval expiration: Jan 1 1970 00:00:00 UTC".
4. User interprets the already-elapsed date as "this approval expires immediately / is harmless" and signs.
5. On-chain approval carries `expires_at = u64::MAX` (~year 2554), granting the spender a permanent allowance to drain the account.

## Impact Explanation
The ICRC-21 consent message system is a security mechanism specifically designed to protect hardware wallet users from malicious dApps by displaying accurate transaction details before signing. This bug corrupts the core output of that mechanism for the `FieldsDisplay` path — the path explicitly designed for constrained hardware devices. The result is that users can be deceived into granting permanent token allowances while believing the approval is already expired. This constitutes a significant ledger security impact with concrete user fund harm, fitting the **High ($2,000–$10,000)** impact class: "Significant ledger security impact with concrete user or protocol harm."

## Likelihood Explanation
The entry point is a public query call requiring no privilege, key, or governance majority. Any dApp can invoke `icrc21_canister_call_consent_message` with a crafted `expires_at`. The `FieldsDisplay` path is specifically targeted at hardware wallets, which are used precisely because users trust their display. A user seeing a 1970 date may plausibly interpret it as a UI glitch or as meaning the approval is already expired and harmless — both interpretations lead to signing. The attack is repeatable, low-cost, and requires no special access.

## Recommendation
Replace the lossy `as i64` cast with pure `u64` arithmetic in both arms of `add_expiration`:

```rust
// FieldsDisplayMessage arm — correct fix
let seconds = expires_at / 1_000_000_000_u64;
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds },
));
```

For the `GenericDisplayMessage` arm (lines 232–233), use `u64` division for `seconds` and handle the nanosecond remainder as `u32` via checked conversion to avoid the "Invalid nanosecond" error for large timestamps.

## Proof of Concept

```rust
#[test]
fn test_add_expiration_fields_display_no_truncation() {
    let expires_at: u64 = u64::MAX;

    // Buggy path (current code)
    let seconds_buggy = (expires_at as i64) / 10_i64.pow(9); // = 0
    let displayed = seconds_buggy as u64;                     // = 0
    assert_eq!(displayed, 0); // epoch 0 — WRONG

    // Correct path
    let seconds_correct = expires_at / 1_000_000_000_u64;    // = 18446744073
    assert_ne!(displayed, seconds_correct);                   // invariant violated

    // Verify via ConsentMessage directly
    let mut msg = ConsentMessage::FieldsDisplayMessage(FieldsDisplay::default());
    msg.add_expiration(Some(u64::MAX), None);
    if let ConsentMessage::FieldsDisplayMessage(fd) = &msg {
        let (_, val) = fd.fields.last().unwrap();
        // Currently stores 0; should store 18446744073
        assert_eq!(val, &Value::TimestampSeconds { amount: 18446744073 });
    }
}
```
