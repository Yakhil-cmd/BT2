### Title
Unprivileged Caller Can Corrupt Settlement Layer Chain ID via Unguarded Event Hook - (`system_hooks/src/event_hooks/system_context.rs`)

### Summary

The `system_context_event_hook` and `interop_root_reporter_event_hook` event hooks perform critical system state mutations (updating the settlement layer chain ID and adding interop roots) based solely on matching an event topic signature. Neither hook receives nor checks the emitting contract's address. Any unprivileged user who deploys a contract that emits the matching event signature can trigger these hooks and corrupt system-level state, directly analogous to the RollerPeriphery `approve()` DOS where a public function with no caller restriction allowed state corruption.

### Finding Description

**Root cause — no emitter address check in event hooks:**

`system_context_event_hook` fires whenever `topics[0] == SL_CHAIN_ID_UPDATED_EVENT_SIG` and immediately calls `system.io.update_settlement_layer_chain_id(...)` with the attacker-supplied `topics[1]` value: [1](#0-0) 

The inner handler `new_sl_chain_id_event_hook` receives `_caller_ee` (prefixed `_`, explicitly unused) and has no `caller` address parameter at all: [2](#0-1) 

Compare this to every **call hook**, which explicitly receives and validates the caller address before doing anything: [3](#0-2) [4](#0-3) [5](#0-4) 

The same pattern applies to `interop_root_reporter_event_hook`, which adds an interop root from attacker-controlled event data with no emitter check: [6](#0-5) [7](#0-6) 

**Exploit path:**

1. Attacker deploys a contract that emits `SettlementLayerChainIdUpdated(uint256)` (topic `0x208daf0b...`) with an arbitrary `uint256` value in `topics[1]`.
2. The event hook dispatch (which, by design, fires for all matching-topic events regardless of emitter) invokes `system_context_event_hook`.
3. `update_settlement_layer_chain_id` is called with the attacker-controlled value, overwriting the canonical settlement layer chain ID.
4. For `interop_root_reporter_event_hook`: attacker emits `InteropRootAdded(uint256,uint256,bytes32[])` (topic `0x6b451b84...`) with crafted `chain_id`, `block_or_batch_number`, and `root` values, injecting a fabricated interop root into the system.

**Note on certainty:** The event hook function signatures contain no `caller`/emitter address parameter (unlike all call hooks), and the `_caller_ee` argument is explicitly unused. This is strong structural evidence that the dispatch layer does not filter by emitter address. Full confirmation requires reading the `HooksStorage` event dispatch implementation in `zk_ee`.

### Impact Explanation

- **Settlement layer chain ID corruption**: Setting the chain ID to `0` or an arbitrary value breaks all cross-chain operations that depend on the canonical settlement layer chain ID — a protocol-level DOS.
- **Fake interop root injection**: Adding attacker-controlled interop roots could allow forging cross-chain messages, potentially enabling unauthorized asset movement across chains.
- Both impacts are permanent within the block and affect all users of the rollup, not just the attacker.

### Likelihood Explanation

Any unprivileged user can submit a transaction that calls a deployed contract emitting the matching event signature. No special keys, governance access, or oracle manipulation is required. The cost is a single transaction.

### Recommendation

Add an emitter address check to both event hooks. The `SystemEventHook` dispatch should pass the emitting contract's address to the hook, and each hook should validate it against the expected system contract address (e.g., `SystemContext` at `0x800b` for chain ID updates, and the designated interop reporter contract for interop roots). This mirrors the pattern already correctly applied in all call hooks.

### Proof of Concept

```
1. Deploy attacker contract:
   contract Attacker {
       event SettlementLayerChainIdUpdated(uint256 newChainId);
       function attack(uint256 fakeChainId) external {
           emit SettlementLayerChainIdUpdated(fakeChainId);
       }
   }

2. Submit tx calling attacker.attack(999999).

3. The emitted event has:
   topics[0] = keccak256("SettlementLayerChainIdUpdated(uint256)")
             = 0x208daf0b9291c1e9a1697737d736630c808045f81f5bc5ae7b8ed740eb5a4d7a
   topics[1] = 999999 (attacker-controlled)

4. system_context_event_hook fires, passes topics[1] to
   update_settlement_layer_chain_id → settlement layer chain ID = 999999.

5. All subsequent cross-chain operations using the settlement layer chain ID
   are broken for the remainder of the batch.
``` [8](#0-7) [9](#0-8)

fn new_sl_chain_id_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // Internal error if the data supplied isn't empty
    if !data.is_empty() {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad data").into(),
        );
    }
    // Same if there's a mismatch in expected topics
    if topics.len() != 2 {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad topics").into(),
        );
    }

    let new_sl_chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    system.io.update_settlement_layer_chain_id(
        ExecutionEnvironmentType::NoEE,
        resources,
        new_sl_chain_id,
    )?;
```
