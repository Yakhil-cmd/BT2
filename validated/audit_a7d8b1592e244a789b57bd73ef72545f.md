### Title
Zero-Fee Transactions Execute With Unbounded Gas Due to Hardcoded Zero Base Fee — (`engine/src/engine.rs`)

---

### Summary

`Engine::charge_gas` contains an early-return path that skips all ETH collection when `max_fee_per_gas == 0`, `fixed_gas == None`, and `block_base_fee_per_gas == 0`. Because `block_base_fee_per_gas` is **hardcoded to always return zero**, the condition permanently reduces to just `max_fee_per_gas == 0 && fixed_gas == None`. Any attacker can submit a zero-fee EIP-1559 transaction with an arbitrarily large `gas_limit`, execute the full EVM gas budget for free, and leave the relayer paying NEAR gas costs with zero ETH revenue.

---

### Finding Description

**`block_base_fee_per_gas` is permanently zero:** [1](#0-0) 

This is not a configurable parameter — it is hardcoded unconditionally.

**The early-return guard in `charge_gas`:** [2](#0-1) 

Because `block_base_fee_per_gas` is always zero, the three-way conjunction permanently collapses to:

```
transaction.max_fee_per_gas.is_zero() && fixed_gas.is_none()
```

When this fires, `GasPaymentResult::default()` is returned — `prepaid_amount = 0`, `effective_gas_price = 0`, `priority_fee_per_gas = 0`. No ETH is deducted from the sender.

**EVM executes with the full attacker-supplied `gas_limit`:** [3](#0-2) 

The `gas_limit` from the transaction is passed directly to `engine.call` / `engine.deploy_code` with no cap or minimum-fee enforcement.

**`refund_unused_gas` also skips all accounting:** [4](#0-3) 

When `effective_gas_price == 0`, the refund function immediately returns `Ok(())`. No relayer reward is computed, no balance adjustments are made. The entire gas accounting cycle is a no-op.

---

### Impact Explanation

The relayer submits a NEAR function call to execute the EVM transaction. NEAR charges gas for that call regardless of EVM-level fees. With `max_fee_per_gas = 0`:

- The attacker's EVM execution consumes up to `gas_limit` EVM gas (which maps to NEAR gas consumption).
- The relayer pays the NEAR gas cost in NEAR tokens.
- The relayer receives **zero ETH** in return.
- There is no on-chain mechanism inside the engine to reject or cap zero-fee transactions.

An attacker can repeat this with maximum `gas_limit` transactions, draining the relayer's NEAR balance while paying nothing. This is a structural, repeatable deficit — **insolvency** of the relayer subsidy pool.

---

### Likelihood Explanation

- No special privilege is required. Any Ethereum address can submit a transaction with `max_fee_per_gas = 0`.
- The condition is always satisfied because `block_base_fee_per_gas` is hardcoded to zero.
- `fixed_gas` is `None` for all normal transaction submissions (it is only set in specific relayer-controlled paths).
- The exploit is trivially repeatable with no cost to the attacker.

---

### Recommendation

1. **Enforce a minimum effective gas price.** If `effective_gas_price == 0` and `fixed_gas == None`, reject the transaction rather than silently executing it for free.
2. **Cap gas for zero-fee transactions.** If free execution is intentionally supported (e.g., for subsidized transactions), enforce a maximum `gas_limit` for the zero-fee path.
3. **Make `block_base_fee_per_gas` configurable** via `EngineState` so operators can set a non-zero floor, making the early-return condition unreachable in production.
4. **Add a relayer-side minimum fee check** before calling `charge_gas`, rejecting transactions where `max_fee_per_gas < relayer_minimum`.

---

### Proof of Concept

```rust
// Locally testable against unmodified engine code
let mut engine = Engine::new_with_state(EngineState::default(), sender, account_id, io, &env);

let tx = NormalizedEthTransaction {
    address: sender,
    chain_id: Some(aurora_chain_id),
    nonce: U256::zero(),
    gas_limit: U256::from(u64::MAX),          // maximum gas budget
    max_priority_fee_per_gas: U256::zero(),   // zero tip
    max_fee_per_gas: U256::zero(),            // zero fee — triggers early return
    to: Some(expensive_contract),
    value: Wei::zero(),
    data: calldata_for_heavy_computation,
    access_list: vec![],
    authorization_list: vec![],
};

// charge_gas returns GasPaymentResult::default() — zero prepaid, zero price
let gas_result = engine.charge_gas(&sender, &tx, None, None).unwrap();
assert_eq!(gas_result.prepaid_amount, Wei::zero());
assert_eq!(gas_result.effective_gas_price, U256::zero());

// EVM executes with gas_limit = u64::MAX — full budget consumed
// refund_unused_gas returns Ok(()) immediately — no ETH collected
// Relayer paid NEAR gas; attacker paid nothing
```

The invariant "gas consumed must always be paid for" is broken: the engine executes arbitrary EVM computation and collects zero ETH, while the relayer bears the full NEAR execution cost.
