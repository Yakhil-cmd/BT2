### Title
Malicious CKD Participant Corrupts Derived Key via Unverified Share Submission — (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator aggregates cryptographic shares from all participants without any verification of their correctness. A single malicious participant can submit arbitrary group elements in place of their legitimate `(big_y, big_c)` share, causing the coordinator to derive a wrong confidential key and rendering the CKD output cryptographically unusable.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs` lines 35–57), the coordinator receives a `CKDOutput` tuple `(big_y, big_c)` from every other participant via `recv_from_others` and unconditionally adds each value to a running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

No proof of knowledge, commitment binding, or consistency check is performed on the received values before they are incorporated into the final output. This is in stark contrast to the DKG protocol in `src/dkg.rs`, which verifies proofs of knowledge via `verify_proof_of_knowledge`, commitment hashes via `verify_commitment_hash`, and secret share consistency via `validate_received_share` before accepting any participant contribution. [2](#0-1) 

Each participant's correct contribution is computed in `compute_signature_share` as:
- `norm_big_y_i = lambda_i * y_i * G` (randomness commitment)
- `norm_big_c_i = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)` (masked BLS share) [3](#0-2) 

A malicious participant can instead send `(G1::identity(), G1::identity())` or any arbitrary group elements. The coordinator sums all contributions and outputs `CKDOutput::new(norm_big_y, norm_big_c)`. The final derived key is `C - a * Y`, which will be wrong whenever any participant's contribution is incorrect. [4](#0-3) 

The `recv_from_others` helper correctly deduplicates and filters messages to only those from known participants via `ParticipantCounter`, so the attacker must be a legitimately enrolled participant — but no further validation of message content occurs. [5](#0-4) 

The analog to the bridge service report is direct: just as the mobile app loaded a JavaScript bundle from a server it did not authenticate — allowing a malicious server to substitute arbitrary executable content — the CKD coordinator loads cryptographic share material from participants it does not authenticate, allowing a malicious participant to substitute arbitrary group elements that corrupt the derived key.

### Impact Explanation
A single malicious participant (one legitimately enrolled in the participant list) can corrupt the CKD output. The coordinator and any downstream consumer will accept a cryptographically invalid derived key — one that does not equal `msk * H(pk, app_id)`. The correct confidential key is never produced, and the application relying on it cannot function. This maps to:

**High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation
Any one of the `n` participants in the CKD protocol can mount this attack with zero cryptographic effort — the attacker simply sends the identity element or any arbitrary `G1Projective` point. The attack is deterministic, always succeeds, and requires no knowledge of other participants' shares or the app secret key. Because the CKD protocol runs over all `n` participants with Lagrange interpolation over the full set, a single corrupted contribution breaks the reconstruction. [6](#0-5) 

### Recommendation
Add a zero-knowledge proof of correct share formation to each participant's message, proving that `norm_big_c_i` is consistent with the participant's public key share and that `norm_big_y_i` is a valid scalar multiple of the generator. Alternatively, adopt a commitment-and-reveal protocol analogous to the one used in `do_keyshare` in `src/dkg.rs` (commitment hash broadcast in round 1, opening in round 2, verification before aggregation). [7](#0-6) 

### Proof of Concept
1. Honest participants compute correct `(norm_big_y_i, norm_big_c_i)` and send them to the coordinator via `chan.send_private`.
2. Malicious participant `P_m` sends `(G1Projective::identity(), G1Projective::identity())` to the coordinator instead of their correct share.
3. Coordinator sums all contributions: `norm_big_y = sum_honest_contributions + identity`, `norm_big_c = sum_honest_contributions + identity` — effectively dropping `P_m`'s Lagrange-weighted BLS share from the reconstruction.
4. Coordinator outputs `CKDOutput::new(norm_big_y, norm_big_c)` where `norm_big_c - a * norm_big_y ≠ msk * H(pk, app_id)`.
5. The derived confidential key is wrong and cryptographically unusable by the intended application. [8](#0-7)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,

    app_pk: PublicKey,

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
