Audit Report

## Title
Unbounded State Sync Restart Loop via Byzantine Peer Returning HTTP 204 — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A single Byzantine subnet peer can force a victim replica into an infinite state-sync start/stop cycle by advertising a valid state height and returning HTTP 204 for every chunk request. Each cycle creates and immediately destroys an `IncompleteState` object, wasting CPU and filesystem I/O. Because the Byzantine peer's 10 downloads complete (with 204) in milliseconds—before honest peers re-advertise (every 5 seconds)—the Byzantine peer can ensure it is always the sole entry in `active_downloads` when the loop exits, permanently preventing honest peers from delivering chunks and stalling the lagging node.

## Finding Description
**Root cause:** `OngoingStateSync::run` (`ongoing.rs:173–176`) breaks out of its event loop unconditionally whenever `self.active_downloads.is_empty()`. The only mechanism that removes a peer from `active_downloads` is `handle_downloaded_chunk_result` on `DownloadChunkError::NoContent` or `RequestError` (`ongoing.rs:194–209`), which calls `active_downloads.remove(&peer_id)` and decrements `allowed_downloads` by `PARALLEL_CHUNK_DOWNLOADS` (10). There is no retry, no backoff, and no minimum-lifetime guard before the break.

**Exploit flow:**

1. Byzantine peer sends an advert for height H with a valid hash. `handle_advert` (`lib.rs:168`) calls `maybe_start_state_sync`, which calls `IncompleteState::try_new` (`chunkable.rs:249–259`), acquiring the write lock and setting `state_sync_refs.active = Some((H, hash))`.
2. The peer is added to `active_downloads` with count 0; `allowed_downloads += 10`; `spawn_chunk_downloads` fires 10 concurrent RPC tasks (`ongoing.rs:147–158`).
3. All 10 tasks return HTTP 204 → `DownloadChunkError::NoContent` → `active_downloads.remove(&peer_id)` (`ongoing.rs:194–197`). `active_downloads` is now empty.
4. The loop checks `active_downloads.is_empty()` → `true` → `break` (`ongoing.rs:173–176`). The cleanup drain at line 179 completes, `new_peers_rx` is closed, and `run` returns.
5. `tracker` (the `Arc<Mutex<Box<dyn Chunkable>>>`) is dropped. `IncompleteState::drop` acquires the write lock and calls `active.take()` (`chunkable.rs:199–200`), resetting `state_sync_refs.active` to `None`.
6. `shutdown.completed()` returns `true`. Back in `handle_advert` (`lib.rs:156–158`), `self.ongoing_state_sync = None` is set. Execution falls through to `maybe_start_state_sync` (`lib.rs:168`). Because `state_sync_refs.active` is `None` and `should_download` still returns `true`, a new `IncompleteState` is created and the cycle repeats.

**Why existing guards are insufficient:** The `IncompleteState::try_new` guard at `chunkable.rs:249–257` only prevents two *concurrent* state syncs; it does not rate-limit sequential restarts. The advert channel (capacity 20, `lib.rs:65`) is drained as fast as the event loop runs. There is no per-peer failure counter, no per-height restart counter, and no minimum inter-start interval anywhere in the path.

## Impact Explanation
This matches the allowed High impact: **"Application/platform-level DoS … subnet availability impact not based on raw volumetric DDoS."** A node that cannot complete state sync cannot participate in consensus for the target height. At sub-second cycle times (Byzantine peer responds with 204 in microseconds), the victim node is locked out of consensus for the affected height indefinitely, degrading subnet liveness proportionally to the number of affected nodes. This is concrete, not hypothetical: the existing `SharableMockStateSync::maybe_start_state_sync_calls` counter (`tests/common.rs:358–360`) can directly measure the unbounded call rate.

## Likelihood Explanation
The attacker is any single Byzantine subnet peer—well below the one-third BFT fault threshold. The only capability required is returning HTTP 204 instead of 200 for `/chunk` requests, a trivial modification to a replica binary. No key material, governance majority, or network-level capability is needed. The attack is self-sustaining: the Byzantine peer only needs to keep re-advertising the same `StateSyncArtifactId` (which it can do at any rate, since the advert channel has capacity 20 and is drained immediately). The 5-second honest-peer re-advertisement interval (`ADVERT_BROADCAST_INTERVAL`, `lib.rs:46`) ensures the Byzantine peer's fast-completing cycles dominate before honest peers can be added.

## Recommendation
1. **Minimum state-sync lifetime:** Do not break from the `OngoingStateSync` loop immediately when `active_downloads` becomes empty. Instead, wait for a configurable minimum duration (e.g., 30 seconds) or until the cancellation token fires, giving honest peers' adverts time to arrive.
2. **Per-peer failure tracking:** Count consecutive `NoContent`/`RequestError` responses per peer. Suppress re-adding a peer that has exceeded a failure threshold within a sliding window.
3. **Rate-limit `maybe_start_state_sync` per height:** Record the last start time for each height in `StateSyncManager` and enforce a minimum inter-start interval (e.g., 60 seconds) before calling `maybe_start_state_sync` again for the same height.

## Proof of Concept
```
1. Instrument maybe_start_state_sync with the existing
   SharableMockStateSync::maybe_start_state_sync_calls counter
   (tests/common.rs:358-360).

2. Configure a MockTransport that always returns HTTP 204 for /chunk requests.

3. Configure MockStateSync::maybe_start_state_sync to always return
   Some(Box::new(chunkable)) where chunkable has at least one chunk to download.

4. Send a continuous stream of adverts for height H from a single peer
   into the advert channel (capacity 20).

5. Run the StateSyncManager event loop for 60 seconds.

6. Assert: maybe_start_state_sync_calls >> 1 (expect O(thousands));
            no chunks are ever successfully delivered;
            the node never advances past height H.

The test structure in rs/p2p/state_sync_manager/tests/ already provides
all necessary mock infrastructure to implement this as a deterministic
integration test without touching mainnet.
```
