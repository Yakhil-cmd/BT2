### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Invoke Nonce-1 Transactions When Account Has Any Mempool Transaction, Not Just a Deploy-Account - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call for invoke transactions with nonce 1 whenever `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` if the account has **any** transaction in the pool — not specifically a `deploy_account` transaction. An attacker can therefore seed the pool with a future-nonce invoke transaction (nonce ≥ 2) for an undeployed address, then submit a nonce-1 invoke with an invalid signature that the gateway admits without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is the UX shortcut that lets a user broadcast `deploy_account` (nonce 0) and `invoke` (nonce 1) simultaneously without waiting for the deploy to confirm:

```
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // "It is sufficient to check if the account exists in the mempool since it means
            //  that either it has a deploy_account transaction or transactions with future
            //  nonces that passed validations."
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool for that address: [3](#0-2) 

**Attack path:**

1. Attacker picks an undeployed address `A` (on-chain nonce = 0).
2. Attacker submits `invoke(nonce=2, sig=VALID)` for `A`. The gateway's `validate_nonce` allows it because `max_allowed_nonce_gap` defaults to 10 (nonce 2 ≤ 0 + 10). `__validate__` is called and passes. The transaction lands in `tx_pool`.
3. Attacker submits `invoke(nonce=1, sig=INVALID)` for `A`. `skip_stateful_validations` fires: `tx.nonce() == 1`, `account_nonce == 0`, and `account_tx_in_pool_or_recent_block(A) == true` (because the nonce-2 tx is in the pool). `skip_validate = true` is returned.
4. `run_validate_entry_point` is called with `execution_flags.validate = false`, so `__validate__` is **never invoked** at the gateway.
5. The nonce-1 invoke with an invalid signature is forwarded to the mempool and accepted. [4](#0-3) 

The comment's reasoning — "transactions with future nonces that passed validations" implies the nonce-1 tx would also pass — is incorrect. The nonce-2 tx and the nonce-1 tx can carry entirely different signatures; passing `__validate__` for one does not imply passing it for the other.

The batcher does call `__validate__` during execution via `AccountTransaction::new_for_sequencing` which hard-codes `validate: true`:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        validate: true,
        strict_nonce_check: true,
        ...
    };
    ...
}
``` [5](#0-4) 

So the invalid-signature transaction will revert during execution. However, the gateway's admission invariant — "every transaction admitted to the mempool has passed `__validate__`" — is broken.

### Impact Explanation

The gateway admits an invoke transaction carrying an invalid (or entirely absent) signature to the mempool without running `__validate__`. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Concretely: an attacker can flood the mempool with nonce-1 invoke transactions that carry arbitrary calldata and invalid signatures, bypassing the only gateway-level signature check. Each such transaction occupies mempool capacity and will be handed to the batcher, which must execute and revert it before discarding it — degrading sequencer throughput.

### Likelihood Explanation

The trigger requires no privileges. Any user can:
- Submit a future-nonce invoke (nonce ≥ 2) for a fresh address — allowed by the default `max_allowed_nonce_gap = 10`.
- Immediately submit a nonce-1 invoke with a garbage signature.

The default `StatefulTransactionValidatorConfig` has `max_allowed_nonce_gap` large enough to make step 1 trivially possible. [6](#0-5) 

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction (nonce 0) is present in the pool for the sender address. For example, query the pool for a transaction with `(address, nonce=0)` of type `DeployAccount`, rather than checking for the existence of any transaction. This preserves the intended UX shortcut while closing the bypass.

### Proof of Concept

```
# 1. Choose an undeployed address A (on-chain nonce = 0).

# 2. Submit invoke(sender=A, nonce=2, sig=VALID_SIG, calldata=...)
#    → gateway validate_nonce: 0 <= 2 <= 10 ✓
#    → __validate__ called, passes ✓
#    → tx lands in tx_pool for address A

# 3. Submit invoke(sender=A, nonce=1, sig=0xDEADBEEF_INVALID, calldata=...)
#    → gateway validate_nonce: 0 <= 1 <= 10 ✓
#    → skip_stateful_validations:
#         tx.nonce() == 1 ✓
#         account_nonce == 0 ✓
#         account_tx_in_pool_or_recent_block(A) == true  ← nonce-2 tx is in pool
#    → skip_validate = true → __validate__ NOT called
#    → tx admitted to mempool with invalid signature

# 4. Batcher picks up nonce-1 tx, calls __validate__, signature fails → revert.
#    Attacker has successfully bypassed gateway-level signature verification.
``` [1](#0-0) [2](#0-1) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
