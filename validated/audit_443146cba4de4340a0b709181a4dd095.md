### Title
`validateCOAOwnershipProof` Does Not Enforce `signedData` Encodes `evmAddress`, Enabling Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account signatures are valid over an arbitrary `signedData` blob, then separately checks that the `evmAddress` argument matches the COA resource stored at the given `path`. Because the function never enforces that `signedData` itself encodes `evmAddress`, a signature produced to prove ownership of one COA can be replayed verbatim to prove ownership of a different COA belonging to the same Cadence account. The code acknowledges this gap in an inline comment but does not mitigate it at the protocol level.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six caller-controlled parameters:

```
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
```

The function performs two independent checks:

1. **Signature check** — verifies that the supplied `signatures` are cryptographically valid over `signedData` using the account keys at `address`.
2. **Address check** — borrows the COA resource at `path` from the account and asserts its on-chain EVM address equals `evmAddress`.

Neither check requires `signedData` to contain `evmAddress`. The two checks are therefore decoupled: a valid `(signedData, signatures)` pair for COA_A can be submitted with `path` pointing to COA_B and `evmAddress` set to COA_B's address, and both checks will pass independently.

The function is declared `access(all)` and is callable from any unprivileged Cadence script or transaction. The inline comment at lines 1003–1009 explicitly acknowledges the gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The EVM-side path (`coa.sol` → `verifyCOAOwnershipProof` precompile → `coaOwnershipProofValidator` → `validateCOAOwnershipProof`) passes `address(this)` as the EVM address, which fixes the address for that call path. However, the Cadence-level function is independently reachable and all six parameters are fully attacker-controlled.

---

### Impact Explanation

**Cross-COA ERC-1271 signature forgery.**

A Cadence account that owns two or more COAs (stored at different `PublicPath`s) is vulnerable. Suppose Alice owns COA_A (at `/public/coaA`) and COA_B (at `/public/coaB`). Alice signs `signedData = H` to prove ownership of COA_A to an off-chain or on-chain EVM protocol that calls `isValidSignature(H, proof_A)` on COA_A.

An attacker who observes `(H, proof_A)` can construct `proof_B` — identical to `proof_A` except `CapabilityPath = "coaB"` — and call `validateCOAOwnershipProof(Alice, /public/coaB, H, keyIndices, signatures, COA_B_address)`. Both checks pass:

- Signature check: same keys, same `signedData = H` → valid.
- Address check: COA at `/public/coaB` has address `COA_B_address` → matches.

The function returns `isValid: true` for COA_B, even though Alice never signed anything for COA_B. Any EVM protocol relying on ERC-1271 (NFT marketplace approvals, ERC-4337 account abstraction, DeFi permit flows, DAO voting) that accepts a COA's `isValidSignature` response is now deceived into treating COA_B as having authorized the same message that was only authorized for COA_A.

---

### Likelihood Explanation

- The function is `access(all)` — reachable by any unprivileged Cadence script or transaction with no special role or staked node required.
- The attacker only needs to observe a single valid `(signedData, proof)` pair for any COA owned by the target Cadence account (e.g., from a public on-chain event or an off-chain authentication flow).
- Multi-COA accounts are a natural pattern for DeFi protocols, DAOs, and multi-purpose wallets; the assumption that "Flow wallets historically create at most one COA per account" is a social convention, not a protocol invariant.
- No brute force, front-running, or privileged access is required.

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` before accepting the proof as valid. One approach:

```cadence
// Require that signedData contains the evmAddress bytes as a prefix or suffix,
// or require callers to pass signedData = hash(evmAddress || application_data).
pre {
    signedData.length >= 20: "signedData must encode evmAddress"
    // verify first 20 bytes of signedData equal evmAddress
}
```

Alternatively, the function can internally construct the message to verify as `hash(evmAddress || signedData)` so that the binding is enforced regardless of what the caller supplies. This mirrors the fix pattern from M-12: include the specific contract/resource address inside the signed payload so that signatures are non-transferable across instances.

### Proof of Concept

**Setup:** Alice's Cadence account (`0xAlice`) owns two COAs:
- COA_A stored at `/storage/coaA`, published at `/public/coaA`, EVM address `0xAAAA...`
- COA_B stored at `/storage/coaB`, published at `/public/coaB`, EVM address `0xBBBB...`

**Step 1 — Legitimate signature for COA_A:**

Alice signs `signedData = [0x01, 0x02, 0x03]` (arbitrary, does not contain `0xAAAA...`) with her account key 0. She produces `proof_A = {KeyIndices: [0], Address: 0xAlice, CapabilityPath: "coaA", Signatures: [sig]}`.

**Step 2 — Attacker replays against COA_B:**

Attacker submits a Cadence script (no authorization required):

```cadence
import EVM from 0x...

access(all) fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: 0xAlice,
        path: /public/coaB,          // ← changed to COA_B's path
        signedData: [0x01, 0x02, 0x03], // ← same signedData Alice signed
        keyIndices: [0],
        signatures: [sig],            // ← same signature
        evmAddress: [0xBB, 0xBB, ...] // ← COA_B's actual EVM address
    )
}
```

**Step 3 — Result:**

- Signature check at line 1082: `keyList.verify(signatureSet, signedData=[0x01,0x02,0x03], ...)` → **passes** (Alice's key 0 did sign this data).
- Address check at line 1095: COA at `/public/coaB` has address `0xBBBB...` which equals the supplied `evmAddress` → **passes**.
- `ValidationResult(isValid: true)` is returned for COA_B.

The attacker has successfully forged a proof of ownership for COA_B using a signature Alice only produced for COA_A.

**EVM-side exploitation:** The attacker calls `COA_B.isValidSignature(bytes32([0x01,0x02,0x03 padded]), proof_B_encoded)` on the COA_B EVM contract. The `isValidSignature` implementation in `coa.sol` calls `verifyCOAOwnershipProof(address(this)=COA_B, hash, proof_B)`, which routes through the precompile to `validateCOAOwnershipProof` with the same parameters as above, returning `true`. 					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}

		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")

		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
