### Title
`OracleValueStopLossExtension` stop-loss timelock fully bypassed via `uint32` overflow in `_afterTimelock` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` casts `block.timestamp + timelock` to `uint32`. When `timelock` is set to `type(uint32).max` (a valid, unvalidated `uint32` input), the sum overflows the 32-bit range and wraps to a value **already in the past**, making every subsequent `_requireElapsed` check pass immediately. The pool admin can therefore execute drawdown, decay, and high-watermark changes with zero delay, defeating the entire LP-protection guarantee of the stop-loss extension.

---

### Finding Description

`_afterTimelock` computes the proposal deadline as:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space (no revert), then **silently truncated** to `uint32`. In Solidity 0.8+, explicit narrowing casts never revert — they just discard the high bits.

With `block.timestamp ≈ 1 753 000 000` (July 2026) and `timelock = type(uint32).max = 4 294 967 295`:

```
1 753 000 000 + 4 294 967 295 = 6 047 967 295   (uint256, no overflow)
uint32(6 047 967 295)          = 1 752 999 999   (wraps: 6 047 967 295 mod 2^32)
```

`1 752 999 999 < 1 753 000 000` — the deadline is **one second in the past**.

`_requireElapsed` then checks:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
``` [2](#0-1) 

`block.timestamp (uint256) < executeAfter (uint32 → uint256 = 1 752 999 999)` evaluates to `false` → **no revert**. Every propose+execute pair completes in a single block.

Neither `proposeOracleStopLossTimelock` nor `initialize` validates the timelock value:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // uses current timelock, not newTimelock
    sched.pendingTimelock = newTimelock;
    ...
}
``` [3](#0-2) 

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// no _validateTimelock
``` [4](#0-3) 

Once `timelock = type(uint32).max` is live, every call to `proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, and `proposeOracleStopLossHighWatermarks` produces a wrapped `executeAfter` that is already elapsed, so the matching `execute*` call succeeds in the same transaction.

---

### Impact Explanation

The `OracleValueStopLossExtension` is explicitly documented as an LP-protection mechanism: *"Drawdown and decay changes are timelocked so LPs can react."* [5](#0-4) 

Bypassing the timelock lets the admin:

1. Set `drawdownE6 = 1_000_000` (100 % — stop-loss never triggers) in one block.
2. Set `decayPerSecondE8 = 1e8` (watermarks decay to zero in one second).
3. Set per-bin high watermarks to zero (stop-loss permanently disabled per bin).

Any of these removes the swap-blocking guard that LPs relied on when depositing. A colluding or compromised admin can then allow a draining swap to proceed against a pool whose oracle is being manipulated, causing direct LP principal loss. This is an **admin-boundary break**: the timelock is the only cap constraining the admin's ability to weaken stop-loss parameters, and it is fully defeated.

---

### Likelihood Explanation

The attack requires the admin to first push `timelock = type(uint32).max` through the **current** timelock (e.g., 7 days). That one-time cost is the only barrier. After that, all future parameter changes bypass the timelock permanently. A pool created with `timelock = type(uint32).max` at initialization (no validation in `initialize`) requires zero waiting time from day one.

---

### Recommendation

Add an upper-bound check in both `initialize` and `proposeOracleStopLossTimelock`. A safe ceiling is any value that cannot cause `block.timestamp + timelock` to exceed `type(uint32).max`:

```solidity
uint32 private constant MAX_TIMELOCK = 365 days; // ~31 536 000 s, well within uint32

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock` in `initialize` alongside `_validateDrawdown`/`_validateDecay`, and at the top of `proposeOracleStopLossTimelock` before writing `newTimelock` to storage.

---

### Proof of Concept

```
// Assume pool initialized with timelock = 7 days, LPs have deposited.

// Step 1 – admin proposes timelock = type(uint32).max
//   executeAfter = block.timestamp + 7 days  (current timelock, no overflow yet)
admin.proposeOracleStopLossTimelock(pool, type(uint32).max);

// Step 2 – wait 7 days, execute
vm.warp(block.timestamp + 7 days);
admin.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == type(uint32).max

// Step 3 – propose drawdown = 100% (disables stop-loss)
//   _afterTimelock returns uint32(block.timestamp + type(uint32).max)
//                        = uint32(~6.0e9) = ~1.75e9  < block.timestamp
//   executeAfter is already in the past
admin.proposeOracleStopLossDrawdown(pool, 1_000_000);

// Step 4 – execute in the SAME block (no warp needed)
admin.executeOracleStopLossDrawdown(pool);
// drawdownE6 == 1_000_000 → stop-loss never triggers → pool unprotected
```
