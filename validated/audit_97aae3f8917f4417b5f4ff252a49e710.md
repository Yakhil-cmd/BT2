### Title
Third-party invoke with arbitrary signature bypasses gateway `__validate__` via `skip_stateful_validations` when deploy_account is in mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point for an invoke transaction with nonce=1 when the account's on-chain nonce is 0 and `account_tx_in_pool_or_recent_block` returns true. The check does not verify that the transaction originates from the account owner. A third party can submit an invoke with an arbitrary/garbage signature for a non-deployed account whose `deploy_account` is already in the mempool, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the deploy_account + invoke bundle flow. It returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`.
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 3 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`contains_account` checks whether the address appears in the mempool's `staged` or `committed` nonce maps, or in the transaction pool: [3](#0-2) 

The code comment in `skip_stateful_validations` states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This assumption is correct for the legitimate owner flow, but it is **not** a proof that the *current* invoke is from the account owner. Any party can submit an invoke for any `sender_address`. When the owner's `deploy_account(nonce=0)` is in the mempool, `account_tx_in_pool_or_recent_block` returns `true` for that address. A third party can then submit `invoke(sender=A, nonce=1, calldata=arbitrary, signature=garbage)` and satisfy all three conditions, causing the gateway to set `execution_flags.validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the function returns immediately after `perform_pre_validation_stage`, never calling `validate_tx` / `__validate__`: [5](#0-4) 

The gateway's `run_pre_validation_checks` calls `validate_by_mempool` before `skip_stateful_validations`, but the mempool's `validate_tx` only checks for duplicate tx hashes and `NonceTooOld`; it does not verify signatures: [6](#0-5) 

The full admission path therefore accepts the attacker's invoke into the mempool with no signature check.

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The gateway's invariant "every invoke transaction must have its `__validate__` entry point called before admission" is broken for the specific window when a `deploy_account` is pending in the mempool. The attacker's invoke carries an arbitrary signature and arbitrary calldata, yet it is admitted as a valid pending transaction. Concretely:

- The attacker occupies the nonce=1 slot for the victim account. The owner's legitimate invoke(nonce=1) is either rejected as a duplicate nonce or must win a fee-escalation race.
- The attacker's transaction will fail during batcher execution (because `__validate__` is called with `validate=true` at execution time and the signature is invalid), but the gateway has already accepted and propagated an unauthorized transaction.
- With zero or minimal resource bounds (legacy `L1Gas` bounds bypass the `validate_resource_bounds` threshold check), the attacker pays no fee for the griefing. [7](#0-6) 

### Likelihood Explanation

The attack requires:
1. Monitoring the public mempool for `deploy_account` transactions (trivially observable).
2. Submitting an invoke(nonce=1) for the same address before the owner does, with a fee high enough to win any fee-escalation race.
3. Setting resource bounds to `L1Gas` (legacy) with `max_fee=0` to avoid the balance check on the non-deployed account.

No privileged access is required. The window is the time between the owner's `deploy_account` entering the mempool and the owner's own invoke(nonce=1) being submitted.

### Recommendation

`skip_stateful_validations` should verify that the transaction present in the mempool for the account is specifically a `deploy_account` transaction, not merely any transaction. One approach: query the mempool for the specific `deploy_account` tx hash (as `PyValidator::should_run_stateful_validations` does via `deploy_account_tx_hash`) rather than using the coarse `account_tx_in_pool_or_recent_block` predicate. [8](#0-7) 

Alternatively, the mempool could expose a dedicated `has_deploy_account_in_pool(address)` query that only returns `true` when a `DeployAccount` transaction (not any transaction) is pending for that address.

### Proof of Concept

```
1. Account A is not yet deployed; on-chain nonce = 0.

2. Owner submits deploy_account(contract_address=A, nonce=0, valid_sig).
   → Gateway validates it (runs constructor via full execute path).
   → deploy_account enters the mempool.
   → account_tx_in_pool_or_recent_block(A) now returns true.

3. Attacker submits invoke(sender=A, nonce=1, calldata=<arbitrary>, signature=<garbage>)
   before the owner submits their own invoke.

4. Gateway stateful validation:
   a. get_nonce_from_state(A) → 0  (account not deployed on-chain)
   b. validate_nonce: 0 <= 1 <= max_allowed_nonce_gap → OK
   c. validate_by_mempool: no duplicate hash, nonce not too old → OK
   d. skip_stateful_validations:
        tx.nonce() == 1  ✓
        account_nonce == 0  ✓
        account_tx_in_pool_or_recent_block(A) == true  ✓  (deploy_account is in pool)
      → returns true (skip __validate__)
   e. run_validate_entry_point(skip_validate=true):
        execution_flags.validate = false
        perform_validations → returns after perform_pre_validation_stage,
        __validate__ is NEVER called.

5. Attacker's invoke(garbage_sig) is admitted to the mempool.

6. Owner's invoke(nonce=1, valid_sig) is now competing with the attacker's
   for the same nonce slot; it may be rejected or must escalate fees.

7. Batcher executes deploy_account(A) → A is deployed.
   Batcher attempts attacker's invoke(nonce=1):
     __validate__ is called with validate=true → signature check fails → tx reverts.
   Owner must resubmit their invoke.
``` [9](#0-8) [2](#0-1) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
