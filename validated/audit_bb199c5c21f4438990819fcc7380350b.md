### Title
Missing Proof of Correct Formation for ElGamal Shares Allows Any Participant to Corrupt CKD Output - (`src/confidential_key_derivation/protocol.rs`)

### Summary
`do_ckd_coordinator` blindly sums `(big_y, big_c)` contributions from all participants with no cryptographic proof that each pair was honestly formed. A single malicious participant can send an arbitrary `(big_y, big_c)` pair, causing the coordinator to produce a `CKDOutput` that, when unmasked by the client, yields an attacker-biased G1 point instead of the correct BLS signature `msk · H(pk ‖ app_id)`.

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and unconditionally adds the two group elements to its running sums: [1](#0-0) 

`recv_from_others` performs only deserialization — no cryptographic check of any kind: [2](#0-1) 

An honest participant's contribution satisfies the invariant `C_i = x_i · H(pk ‖ app_id) + y_i · A` and `Y_i = y_i · G`, so that `C_i − a · Y_i = x_i · H(pk ‖ app_id)` (where `a = app_sk`). Nothing in the protocol enforces this relationship for received messages.

The `unmask` function recovers the derived key as: [3](#0-2) 

### Impact Explanation

Let `(Ŷ, Ĉ)` be the honest aggregate. A malicious participant sends `(big_y = 0_G1, big_c = δ)` for any chosen G1 point `δ`. The coordinator computes:

```
final_Y = Ŷ + 0 = Ŷ
final_C = Ĉ + δ
```

The client then calls `unmask(app_sk)`:

```
result = (Ĉ + δ) − app_sk · Ŷ
       = msk · H(pk ‖ app_id) + δ
```

The derived key is offset by `δ`, which is fully attacker-controlled. The client receives a wrong, non-deterministic key. This corrupts the CKD output for all honest parties and violates the core protocol invariant that the derived key is determined solely by `msk` and `(pk, app_id)`.

This matches the **High** impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation

Any registered participant in the CKD session can mount this attack. It requires no knowledge of `app_sk`, `msk`, or any other secret — the attacker simply sends a crafted `CKDOutput` struct (a pair of valid G1 points) instead of their honest contribution. The attack is a single-message deviation in a single-round protocol, requiring no coordination.

### Recommendation

Each participant must accompany their `(big_y, big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that `big_c = big_s + y · A` and `big_y = y · G` for the same scalar `y`, without revealing `y` or `big_s`. A standard approach is a Chaum-Pedersen / sigma-protocol proof of discrete-log equality:

- Statement: `(big_y, big_c − big_s)` are both scalar multiples of `G` and `A` respectively by the same `y`
- Since `big_s = x_i · H(pk ‖ app_id)` is publicly computable from the participant's public share commitment, the coordinator can verify the proof before adding the contribution

Alternatively, use a commitment-then-reveal scheme similar to the DKG round (see `src/dkg.rs` lines 409–426 and 514–522 for the pattern already used elsewhere in this codebase).

### Proof of Concept

```rust
// Malicious participant: instead of calling compute_signature_share,
// send a crafted CKDOutput to the coordinator.

// Choose any arbitrary bias point delta (e.g., the generator)
let delta = ElementG1::generator();

// Send (identity, delta) — no y knowledge needed
let crafted = CKDOutput::new(ElementG1::identity(), delta);
chan.send_private(waitpoint, coordinator, &crafted)?;

// The coordinator sums without verification:
//   norm_big_y += identity  => unchanged
//   norm_big_c += delta     => biased

// After unmask(app_sk):
//   result = msk * H(pk || app_id) + delta   ≠ expected BLS signature
// verify_signature will fail, or the client silently uses a wrong key.
```

The coordinator has no mechanism to detect or reject this crafted message. [4](#0-3)

    waitpoint: u64,
    participants: &ParticipantList,

    T: serde::de::DeserializeOwned,

    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
