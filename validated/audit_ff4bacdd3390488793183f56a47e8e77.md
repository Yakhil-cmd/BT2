Audit Report

## Title
Missing Intra-Payload Duplicate `callback_id` Check in Bitcoin `SelfValidatingPayload` Validation â€” (`rs/bitcoin/consensus/src/payload_builder/validate_payload`)

## Summary
`bytes_to_payload` decodes Bitcoin adapter responses into a plain `Vec` with no uniqueness enforcement on `callback_id`. The `validate_payload` function computes `delivered_ids` from past payloads but never uses it to cross-check the current payload, and `validate_self_validating_payload_impl` performs no callback ID validation at all. A single Byzantine block proposer can craft a payload with two entries sharing the same `callback_id`, which passes all validation, gets finalized, and causes the execution layer to push two `ConsensusResponse` entries for the same callback into `consensus_queue`, invoking the Bitcoin management canister's callback twice.

## Finding Description

**Parse layer â€” no deduplication:**
`bytes_to_payload` collects decoded protobuf messages into a plain `Vec` with no uniqueness check: [1](#0-0) 

By contrast, `bytes_to_chain_key_payload` inserts into a `BTreeMap` and explicitly returns an error if `insert` returns `Some`, catching duplicates at decode time: [2](#0-1) 

HTTPS outcalls validation similarly tracks a `delivered_ids` set and returns `DuplicateResponse` on collision: [3](#0-2) 

**Validation layer â€” `delivered_ids` computed but never applied to current payload:**
`validate_payload` calls `parse_past_payload_ids` to build `delivered_ids` at line 369, then calls `bytes_to_payload` and `validate_self_validating_payload_impl`, but never uses `delivered_ids` to cross-check the decoded entries: [4](#0-3) 

`validate_self_validating_payload_impl` performs only an emptiness check and a byte-size return â€” no callback ID validation: [5](#0-4) 

**Execution layer â€” unconditional double-push:**
`DemuxImpl::process_payload` iterates over all `bitcoin_adapter_responses` and calls `push_response_bitcoin` for each with no deduplication guard: [6](#0-5) 

For `GetSuccessorsResponse`, `get_mut` is used (not `remove`), so the context remains in the map and both duplicate calls succeed, each pushing a `ConsensusResponse` to `consensus_queue`: [7](#0-6) 

For `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject`, there is no context lookup at all â€” each variant unconditionally constructs and pushes a `ConsensusResponse`: [8](#0-7) 

## Impact Explanation
Two `ConsensusResponse` entries with the same `callback_id` enter `consensus_queue`. The execution environment processes both, invoking the Bitcoin management canister's callback twice for a single request. For `GetSuccessors`, this means the canister processes the same Bitcoin block data twice, corrupting its chain-tip tracking state. For `SendTransaction`, the canister receives two confirmations for one broadcast. This violates the invariant that each `callback_id` maps to exactly one pending request and is delivered exactly once, causing concrete state corruption in the in-scope Bitcoin/Chain Fusion integration. This matches the **High ($2,000â€“$10,000)** impact class: significant Chain Fusion security impact with concrete protocol harm.

## Likelihood Explanation
A single Byzantine block proposer â€” one node within the standard `f < n/3` fault threshold â€” can craft and propose such a block. Because `validate_payload` has no intra-payload duplicate check, all honest nodes accept and notarize the block. No threshold corruption, no key compromise, and no external dependency is required. The attack is repeatable every time the Byzantine node is selected as block proposer.

## Recommendation
Add an intra-payload duplicate `callback_id` check in `validate_payload` (or in `bytes_to_payload` itself, mirroring `bytes_to_chain_key_payload`):

```rust
// In validate_payload, after bytes_to_payload:
let mut seen_ids = std::collections::BTreeSet::new();
for response in &payload {
    if !seen_ids.insert(response.callback_id) {
        return Err(ValidationError::InvalidArtifact(
            consensus::InvalidPayloadReason::InvalidSelfValidatingPayload(
                InvalidSelfValidatingPayloadReason::DuplicateCallbackId(response.callback_id),
            ),
        ));
    }
}
```

Additionally, cross-check each `callback_id` against `delivered_ids` (already computed from past payloads at line 369) to reject responses for already-delivered callbacks. For defense-in-depth, change `get_mut` to `remove` in the `GetSuccessorsResponse` branch of `push_response` so that a context can only be consumed once even if a duplicate somehow reaches execution.

## Proof of Concept
1. Encode two `PbBitcoinAdapterResponse` protobuf messages with identical `callback_id = X` and valid `response` fields (e.g., `SendTransactionReject`) as length-delimited bytes.
2. Submit this as the `SelfValidatingPayload` bytes in a block proposal from a controlled block-proposer node.
3. Observe that `validate_payload` returns `Ok(())` on all honest nodes (no duplicate check present).
4. After finalization, observe that `consensus_queue` contains two `ConsensusResponse` entries with `callback_id = X`.
5. Observe the Bitcoin management canister's callback is invoked twice for the same request.

A deterministic unit test can verify this directly: call `bytes_to_payload` with two length-delimited protobuf messages sharing the same `callback_id` and assert the result contains two entries â€” confirming the absence of any deduplication guard.
