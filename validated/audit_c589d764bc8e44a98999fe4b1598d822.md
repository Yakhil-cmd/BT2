### Title
`StatefulTransactionValidatorConfig::max_nonce_for_validation_skip` Is Ignored in `skip_stateful_validations` — Gateway Rejects Valid Invoke Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` declares and documents a `max_nonce_for_validation_skip` field that is meant to control the maximum transaction nonce for which the gateway skips the `__validate__` entry-point check in the deploy-account + invoke UX flow. However, the free function `skip_stateful_validations` never reads this config field; it hardcodes `Nonce(Felt::ONE)` unconditionally. Any operator-configured value above `1` is silently ignored, causing the gateway to reject valid invoke transactions that should have been admitted.

### Finding Description

`StatefulTransactionValidatorConfig` defines and serialises `max_nonce_for_validation_skip`:

```rust
pub max_nonce_for_validation_skip: Nonce,
// "Maximum nonce for which the validation is skipped."
``` [1](#0-0) 

The default value is `Nonce(Felt::ONE)`. [2](#0-1) 

`run_pre_validation_checks` (which has `&self` access to the config) calls the free function `skip_stateful_validations`:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [3](#0-2) 

`skip_stateful_validations` receives no config parameter and hardcodes the nonce threshold:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [4](#0-3) 

The config field `max_nonce_for_validation_skip` is therefore never consulted on the Rust gateway path. By contrast, the Python-facing `PyValidator::should_run_stateful_validations` correctly reads `self.max_nonce_for_validation_skip` and applies a `<=` comparison:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
``` [5](#0-4) 

The two code paths are therefore inconsistent: the Python validator honours the configured threshold; the Rust gateway ignores it.

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip` to any value `N > 1` (e.g. `2`), the intent is that invoke transactions with nonces `1..=N` from an as-yet-undeployed account should be admitted without running `__validate__` (because the deploy-account has not been processed yet). Because `skip_stateful_validations` hardcodes `Nonce(Felt::ONE)`, only nonce `1` is ever skipped. Invoke transactions with nonces `2..=N` reach `run_validate_entry_point` with `skip_validate = false`, causing `StatefulValidator::perform_validations` to attempt to call the `__validate__` entry point on a contract that does not yet exist. The gateway returns a `ValidateFailure` error and rejects the transaction. [6](#0-5) 

Conversely, if an operator sets `max_nonce_for_validation_skip` to `0` intending to disable the skip entirely, the hardcoded `== Nonce(Felt::ONE)` check still fires for nonce-1 invoke transactions, admitting them without `__validate__` contrary to the operator's configuration.

### Likelihood Explanation

The field is publicly documented in the config schema and serialised via `SerializeConfig`, making it a natural knob for operators to tune. Any deployment that raises the threshold above `1` (or lowers it to `0`) will silently observe the wrong behaviour. The root cause is a straightforward omission: the free function signature does not accept the config, so the field can never be used even if a developer intends to.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded `Nonce(Felt::ONE)` equality check with a range check mirroring `PyValidator`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,   // <-- add parameter
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() <= max_nonce_for_validation_skip   // was: == Nonce(Felt::ONE)
            && account_nonce == Nonce(Felt::ZERO)
        {
            ...
        }
    }
    Ok(false)
}
```

### Proof of Concept

1. Configure the gateway with `max_nonce_for_validation_skip = 2`.
2. Submit a `DeployAccount` transaction for address `A` (nonce 0).
3. Immediately submit an `Invoke` transaction from `A` with nonce `2` (second post-deploy tx).
4. The gateway reads `account_nonce = 0` from state (account not yet deployed).
5. `skip_stateful_validations` checks `tx.nonce() == Nonce(Felt::ONE)` → `false` (nonce is 2).
6. `skip_validate = false`; `run_validate_entry_point` runs `__validate__` on the non-existent contract.
7. Gateway returns `ValidateFailure` and rejects the transaction — even though the operator configured `max_nonce_for_validation_skip = 2` precisely to allow this. [4](#0-3) [1](#0-0)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
