Audit Report

## Title
Unprivileged Caller Can Permanently Corrupt SNS Upgrade State via `fail_stuck_upgrade_in_progress` — (`rs/sns/governance/canister/canister.rs`)

## Summary
`fail_stuck_upgrade_in_progress` is exposed as an unrestricted `#[update]` endpoint with no caller authentication. Any principal can invoke it after `mark_failed_at_seconds` elapses to mark a successfully completed SNS upgrade as FAILED, leaving `deployed_version` permanently stale. This also races with the in-flight `check_upgrade_status` callback, causing an unconditional `unwrap()` on a now-`None` `pending_version` to panic and trap the periodic task.

## Finding Description

**No access control on the endpoint.** `fail_stuck_upgrade_in_progress` in `rs/sns/governance/canister/canister.rs` (lines 526–535) carries no `caller()` check — any principal may invoke it: [1](#0-0) 

**Timestamp-only gate.** The implementation in `rs/sns/governance/src/governance.rs` (lines 6328–6361) proceeds solely on `now > pending_version.mark_failed_at_seconds`, without verifying whether the upgrade actually succeeded. The code itself contains a TODO comment acknowledging the missing `checking_upgrade_lock` guard (lines 6337–6338): [2](#0-1) 

**`complete_sns_upgrade_to_next_version` unconditionally clears state.** When called with `deployed_version = None`, it sets `self.proto.pending_version = None` and leaves `deployed_version` unchanged (lines 6309–6313): [3](#0-2) 

**Race with `check_upgrade_status` callback.** `check_upgrade_status` increments `checking_upgrade_lock` before the `await` at line 6174 (committed to state). After the inter-canister call returns, it unconditionally does `self.proto.pending_version.as_mut().unwrap()` (lines 6177–6181). If `fail_stuck_upgrade_in_progress` ran during the suspension and set `pending_version = None`, this `unwrap()` panics, trapping the callback: [4](#0-3) 

The comment at line 6108 explicitly states the `expect` is safe only because `should_check_upgrade_status` was checked — an assumption broken by the external endpoint: [5](#0-4) 

**Exploit flow:**
1. SNS `UpgradeSnsToNextVersion` proposal passes; `pending_version` is set with `mark_failed_at_seconds = now + 300` and `deployed_version = V1`.
2. Canister upgrade to V2 completes on-chain.
3. `check_upgrade_status` increments `checking_upgrade_lock` to 1 (committed) and suspends at `await get_running_version(...)`.
4. Clock advances past `mark_failed_at_seconds`.
5. Attacker calls `fail_stuck_upgrade_in_progress`: timestamp check passes → `complete_sns_upgrade_to_next_version(..., None)` → `pending_version = None`, proposal marked FAILED — all committed.
6. `get_running_version` response arrives; callback resumes and hits `pending_version.as_mut().unwrap()` → **panic**; callback state rolled back.

**Result:** `deployed_version = V1` (stale), `pending_version = None`, upgrade proposal = FAILED, canister actually runs V2.

## Impact Explanation

This is a **High** severity SNS governance impact. An unprivileged attacker can permanently corrupt the SNS governance canister's view of its own software version: `deployed_version` is left at the pre-upgrade value while the canister runs the new version. All subsequent `UpgradeSnsToNextVersion` proposals will compute upgrade steps from the wrong baseline, potentially targeting an already-applied version or skipping required intermediate versions. The incorrectly-failed proposal also blocks the SNS from recognizing the upgrade as successful, requiring manual recovery. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation

- The endpoint is callable by **any principal** with zero privileges.
- `pending_version.mark_failed_at_seconds` is readable from canister state; an attacker can time the call precisely.
- The exploitable window opens at `mark_failed_at_seconds` (5 minutes after upgrade initiation) and closes only when `check_upgrade_status` successfully confirms the new version. Because `check_upgrade_status` makes an inter-canister call spanning multiple rounds, the window can be seconds to minutes wide.
- The attack is repeatable: every SNS upgrade proposal creates a fresh `pending_version` with a new deadline.
- No special tooling is required beyond a standard IC agent.

## Recommendation

1. **Add access control**: restrict `fail_stuck_upgrade_in_progress` to the SNS root canister or to neurons with sufficient voting power, analogous to other privileged SNS operations.
2. **Respect `checking_upgrade_lock`**: if `checking_upgrade_lock > 0`, a status check is in-flight; `fail_stuck_upgrade_in_progress` should return early (the TODO comment at lines 6337–6338 already identifies this gap).
3. **Verify upgrade success before marking failure**: before calling `complete_sns_upgrade_to_next_version` with a failure status, attempt to confirm the running version matches the target.
4. **Defensive `unwrap` in callback**: replace the unconditional `unwrap()` at lines 6177–6181 with a graceful `if let Some(pv) = self.proto.pending_version.as_mut()` guard, since `pending_version` can be externally cleared between the `await` suspension and callback resumption.

## Proof of Concept

```
// PocketIC / integration test outline:

1. Install SNS governance canister with deployed_version = V1.
2. Submit and execute UpgradeSnsToNextVersion proposal → pending_version set,
   mark_failed_at_seconds = now + 300.
3. Simulate canister upgrade to V2 completing on-chain.
4. Advance PocketIC time by 301 seconds (past mark_failed_at_seconds).
5. Call fail_stuck_upgrade_in_progress({}) from an anonymous/unprivileged principal.
6. Assert: governance.proto.pending_version == None
           governance.proto.deployed_version == V1  // stale
           proposal.status == FAILED
           // canister actually runs V2
```

A deterministic unit test can be written using the existing `fail_stuck_upgrade_in_progress_tests.rs` harness, advancing the mock clock past `mark_failed_at_seconds` and asserting the stale `deployed_version` outcome without any privileged caller.
