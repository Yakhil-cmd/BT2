### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield via Trace Growth - (File: `src/pool/pool.cairo`)

### Summary

The `calculate_rewards` internal function in `Pool` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace. Because every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new checkpoint to `pool_member_epoch_balance`, a delegator who makes frequent balance changes across many epochs without claiming rewards will eventually accumulate a trace so large that the `claim_rewards` transaction runs out of gas and permanently reverts, freezing all unclaimed yield with no recovery path.

### Finding Description

`calculate_rewards` (pool.cairo lines 837–888) iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` to the first entry whose epoch is ≥ `until_epoch`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
```

The comment itself admits the loop is unbounded. Each iteration performs multiple storage reads (`pool_member_trace.at(...)` and `find_sigma` which reads `cumulative_rewards_trace`), making each iteration non-trivial in gas cost.

The trace grows via `set_member_balance` / `increase_member_balance`, called from:
- `add_to_delegation_pool` (line 242) — any balance increase by the pool member
- `exit_delegation_pool_intent` (line 278) — any partial or full exit intent
- `enter_delegation_pool_from_staking_contract` (line 456/464) — pool switching

The `PoolMemberBalanceTrace` insert logic only merges entries at the **same epoch**; entries at distinct epochs are always appended as new checkpoints. Therefore, one balance change per epoch adds one entry permanently.

`entry_to_claim_from` is only advanced when `claim_rewards` succeeds (line 358). If the transaction runs out of gas, the counter is never updated, and every subsequent attempt must re-iterate the same oversized trace — permanently failing.

There is no partial-claim mechanism. `claim_rewards` (line 335) always attempts to process the entire range from `reward_checkpoint` to the current epoch in a single transaction.

### Impact Explanation

A pool member who makes balance changes across N distinct epochs without claiming rewards will face a loop of N iterations on their next `claim_rewards` call. Once N is large enough to exhaust Starknet's per-transaction gas limit, the call reverts unconditionally. Because `entry_to_claim_from` is only persisted on success, every future attempt faces the same N-iteration loop and also reverts. The pool member's entire accumulated unclaimed yield is permanently frozen with no on-chain recovery path.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

The attack path requires only normal delegator behavior: stake, adjust balance periodically across epochs, and defer reward claims. A delegator who increases or partially exits their delegation once per epoch and claims rewards infrequently (e.g., annually) will accumulate hundreds of trace entries. The exact threshold depends on Starknet's gas limit and the per-iteration cost (multiple storage reads per iteration), but the developers themselves acknowledge the loop is unbounded. No privileged access, external dependency, or special tooling is required — only repeated calls to `add_to_delegation_pool` or `exit_delegation_pool_intent` across different epochs.

### Recommendation

1. **Allow partial reward claims**: Add a `max_iterations` parameter to `claim_rewards` so a pool member can process rewards in chunks, advancing `entry_to_claim_from` incrementally across multiple transactions.
2. **Enforce a claim cadence**: Require or incentivize reward claims at regular intervals (e.g., before each balance change) to keep the trace window small.
3. **Cap trace growth**: Automatically trigger a reward checkpoint (updating `entry_to_claim_from` and `reward_checkpoint`) whenever a balance change is recorded, so the loop never needs to span more than one epoch per call.

### Proof of Concept

1. Deploy a pool with a staker.
2. As a delegator, call `add_to_delegation_pool` with a small amount once per epoch for N epochs (N = several hundred), never calling `claim_rewards`.
3. After N epochs, call `claim_rewards`. The transaction will attempt to iterate N times through `pool_member_epoch_balance`, each iteration reading `pool_member_trace.at(i)` and calling `find_sigma` (which reads `cumulative_rewards_trace`). At sufficient N, the transaction runs out of gas and reverts.
4. Repeat step 3 — every attempt reverts identically because `entry_to_claim_from` was never persisted. All accumulated yield is permanently inaccessible.
