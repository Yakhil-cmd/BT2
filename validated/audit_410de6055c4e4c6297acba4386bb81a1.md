### Title
Caller-Supplied `app_pk` Has No Protocol-Level Consistency Enforcement in CKD, Allowing Malicious Orchestrator to Corrupt CKD Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol accepts `app_pk` (the application's ElGamal public key `A`) as an independent, locally-supplied parameter for each participant with no protocol-level broadcast or cross-participant agreement. A malicious orchestrator can supply a different `app_pk` to different participants within the same CKD session. Because each participant's encrypted contribution `C_i = S_i + y_i ÂṖ A_i` is computed against their locally-supplied key, the coordinator's aggregated output becomes a linear combination of heterogeneous encryption keys, producing a `CKDOutput` that the application cannot correctly unmask. Honest participants complete the protocol without error and the coordinator returns a `Some(CKDOutput)` â but the output is cryptographically unusable.

This value is forwarded directly into `compute_signature_share()`, where it is used to encrypt each participant's secret contribution:

```rust

``` [2](#0-1) 

The coordinator aggregates all received `(norm_big_y, norm_big_c)` pairs by simple addition, with no verification that participants used the same `app_pk`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();

``` [3](#0-2) 

The protocol specification requires that the aggregated `C` equals `msk ÂṖ H(pk, app_id) + a ÂṖ Y`, where `a` is the application's secret key corresponding to `A`. This identity holds only when every participant encrypts against the **same** `A`. There is no round in the protocol where participants broadcast or verify `app_pk` against each other. The protocol is single-round: participants send their shares privately to the coordinator and immediately return `Ok(None)`. [4](#0-3) 

The analogous vulnerability in the external report is `InsuranceFund#syncDeps()` changing `vusd` without a balance consistency check. Here, the critical parameter `app_pk` can vary across participants without any consistency check, producing the same class of silent corruption.

### Impact Explanation

If participant `P_i` is given `app_pk = A_i` (distinct per participant), the coordinator aggregates:

```
C = Î£ ÎṠ_i ÂṖ (S_i + y_i ÂṖ A_i)
  = msk ÂṖ H(pk, app_id) + Î£ ÎṠ_i ÂṖ y_i ÂṖ A_i
```

Because `A_i` differ, the blinding term `Î£ ÎṠ_i ÂṖ y_i ÂṖ A_i` is not equal to `a ÂṖ Y` for any single secret `a`. The application's `unmask(app_sk)` call computes `C â a ÂṖ Y`, which does not cancel the blinding term, yielding a random group element instead of `msk ÂṖ H(pk, app_id)`.

The `CKDOutput` is returned as `Some(...)` with no error â honest participants and the coordinator all complete successfully â but the derived key is permanently wrong. The application cannot detect this without an out-of-band oracle. [5](#0-4) 

**Impact category matched**: *High â Corruption of CKD outputs so honest parties accept an unusable cryptographic output.*

---

### Likelihood Explanation

In any real deployment the orchestrator (the entity that calls `ckd()` for each participant) controls which `app_pk` is passed to each node. A malicious or compromised orchestrator â explicitly listed as an attacker profile in `RESEARCHER.md` â can trivially supply distinct `app_pk` values per participant. No cryptographic capability is required; only the ability to call the library API with different arguments. The protocol provides zero detection: participants never communicate `app_pk` to each other, and the coordinator performs no cross-participant consistency check. [6](#0-5) 

---

### Recommendation

Add a protocol-level broadcast-and-agree step for `app_pk` (and `app_id`) before participants compute their shares. Each participant should broadcast a commitment to their locally-supplied `app_pk`, verify that all received commitments match, and abort if any mismatch is detected. Alternatively, derive `app_pk` deterministically from `app_id` and the shared public key so that no external parameter is needed.

A minimal guard analogous to the recommended fix in the external report:

```rust
// After receiving all participant commitments to app_pk,
// assert all match the locally-supplied value before proceeding.
for (from, their_app_pk_hash) in recv_from_others(...) {
    require!(their_app_pk_hash == my_app_pk_hash,
        "app_pk mismatch: participant {:?} used a different key", from);
}
```
