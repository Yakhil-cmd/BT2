### Title
`ExitToEthereum` Precompile Does Not Handle Failed `withdraw` Promise — Permanent Loss of Bridged Funds - (File: `engine-precompiles/src/native.rs`)

### Summary
The `ExitToEthereum` precompile burns ERC-20 tokens (or deducts ETH) from the Aurora EVM state and then schedules a NEAR `withdraw` promise to the ETH connector. Unlike `ExitToNear`, which attaches an `exit_to_near_precompile_callback` to handle promise failure and refund the user, `ExitToEthereum` always emits a bare `PromiseArgs::Create` with no callback. If the `withdraw` promise fails on the NEAR side, the tokens are permanently destroyed with no recovery path.

### Finding Description
In `ExitToEthereum::run()`, after burning the user's ERC-20 tokens or deducting ETH, the precompile constructs a `withdraw` promise and serialises it as `PromiseArgs::Create`:

```rust
// engine-precompiles/src/native.rs  lines 977-985
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

`PromiseArgs::Create` carries no callback. The EVM transaction has already committed (tokens burned / ETH deducted) before this asynchronous NEAR promise executes. If the promise fails — e.g., because the ETH connector is paused, the connector account is missing, or gas is exhausted — there is no code path that re-mints the burned tokens or returns the ETH to the caller.

By contrast, `ExitToNear::run()` conditionally wraps its transfer promise in a `PromiseArgs::Callback` that invokes `exit_to_near_precompile_callback`, which calls `engine::refund_on_error` to re-mint burned ERC-20 tokens or transfer ETH back from the precompile address:

```rust
// engine-precompiles/src/native.rs  lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
```

`ExitToEthereum` has no equivalent guard. The `error_refund` feature and `refund_on_error` logic exist in the engine but are never wired to the Ethereum-exit path.

### Impact Explanation
Any user who calls `ExitToEthereum` while the ETH connector's `withdraw` method is paused, out of gas, or otherwise failing will have their ERC-20 tokens permanently burned (or ETH permanently deducted) with no on-chain recovery. The funds are neither on Aurora nor on Ethereum — they are destroyed. This constitutes **permanent freezing / permanent loss of user funds**.

### Likelihood Explanation
The ETH connector exposes a `pa_pause_feature` admin function that can pause `withdraw`. During any such pause window, every `ExitToEthereum` call silently destroys the caller's tokens. Additionally, gas exhaustion on the NEAR side (e.g., if the connector performs expensive storage operations) can cause the promise to fail without any user error. Both scenarios are realistic in production.

### Recommendation
Mirror the `ExitToNear` pattern: attach a callback to the `withdraw` promise in `ExitToEthereum::run()`. The callback should inspect the promise result and, on failure, invoke `engine::refund_on_error` to re-mint burned ERC-20 tokens or transfer ETH back from the precompile address to the original caller. The existing `refund_on_error` function already handles both the ERC-20 and ETH cases and can be reused directly.

### Proof of Concept

1. Deploy Aurora with the ETH connector.
2. Bridge an ERC-20 token to Aurora; user holds `N` tokens on Aurora.
3. Owner calls `pa_pause_feature("engine_withdraw")` on the ETH connector (a legitimate operational action).
4. User calls `ExitToEthereum` for `N` tokens.
   - EVM executes: ERC-20 `burn` succeeds; tokens are gone from Aurora EVM state.
   - NEAR promise: `withdraw` on the connector reverts because the feature is paused.
   - No callback fires; no re-mint occurs.
5. User now has 0 tokens on Aurora and 0 tokens on Ethereum — funds are permanently lost.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
