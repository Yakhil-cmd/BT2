### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions When Any Sender Transaction Exists in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is intended to skip the `__validate__` entry-point call for the first invoke (nonce=1) from an account whose `deploy_account` is still pending in the mempool. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` whenever **any** transaction from that sender address exists in the pool, not specifically a `deploy_account`. An unprivileged attacker who observes a victim's `deploy_account` in the public mempool can immediately submit an invoke with nonce=1 carrying an arbitrary (invalid) signature for the victim's address. The gateway skips `__validate__`, admits the transaction, and the invalid invoke is sequenced into a block where it reverts and charges a fee to the victim.

### Finding Description

In `run_validate_entry_point` the gateway builds a `BlockContext` with `disable_casm_hash_migration()` and `block_number.unchecked_next()`, then calls `StatefulValidator::validate`. For invoke transactions the validator calls `perform_pre_validation_stage` and then the `__validate__` entry point. The skip path is decided earlier in `run_pre_validation_checks`:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
```

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

1. Transaction type is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

Condition 4 delegates to:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` is true for **any** transaction type in the pool — not exclusively `deploy_account`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is not enforced.

Neither `validate_by_mempool` nor the mempool's `validate_tx` inspects the cryptographic signature:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [3](#0-2) 

`ValidationArgs` carries only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — no signature field. [4](#0-3) 

The stateless validator checks only signature **length**, not validity: [5](#0-4) 

### Impact Explanation

An attacker who observes a victim's `deploy_account` (nonce=0) in the public mempool can submit an invoke with nonce=1 carrying a fabricated signature for the victim's address. All gateway checks pass:

| Check | Result |
|---|---|
| Stateless: signature size | Pass (any size ≤ max) |
| Stateful nonce: `0 ≤ 1 ≤ max_gap` | Pass |
| Mempool `validate_tx`: no duplicate nonce=1 | Pass |
| `skip_stateful_validations`: nonce=1, account_nonce=0, account in pool | Returns `true` → skip |
| `__validate__` entry point | **Skipped** |

The invalid invoke is admitted to the mempool. The batcher later executes `deploy_account` (success, account created), then the invoke (reverts at `__validate__` because the signature is invalid). The revert fee is charged to the victim's newly-deployed account. The attacker can front-run any new account deployment visible in the mempool.

**Matching impact**: *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

The mempool is observable by any network participant. A `deploy_account` transaction reveals the target address immediately. The attacker needs only to submit a single invoke with nonce=1 and any bytes as the signature before the victim submits their own nonce=1 invoke. No privileged access, no special knowledge, and no on-chain state is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender address exists in the pool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the pool check should filter by transaction type before returning `true`. Alternatively, the skip should only be granted when the gateway itself has already validated the `deploy_account` transaction for that address in the same admission session (e.g., by tracking a per-address flag set when a `deploy_account` passes full validation).

### Proof of Concept

```
1. Victim submits RpcDeployAccountTransaction for address A (class_hash C, salt S).
   → deploy_account lands in mempool; account_nonce(A) = 0 on-chain.

2. Attacker observes A in the mempool.

3. Attacker submits RpcInvokeTransaction V3:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef]   // arbitrary invalid bytes
     resource_bounds, tip, calldata  // any valid values

4. Gateway stateless validation:
     validate_contract_address(A)          → Ok (A is valid felt)
     validate_tx_signature_size([0xdeadbeef]) → Ok (length 1 ≤ max)
     validate_resource_bounds(...)          → Ok

5. Gateway stateful validation:
     account_nonce = get_nonce(A) = 0
     validate_nonce: 0 ≤ 1 ≤ max_gap      → Ok
     validate_by_mempool(nonce=1, account_nonce=0) → Ok (no dup)
     skip_stateful_validations:
       nonce==1 && account_nonce==0
       account_tx_in_pool_or_recent_block(A) → true (deploy_account is in pool)
       → returns true (SKIP __validate__)

6. run_validate_entry_point called with skip_validate=true
   → execution_flags.validate = false
   → __validate__ is NOT called
   → invalid invoke admitted to mempool

7. Batcher sequences block:
     execute deploy_account(A)  → Ok, account deployed
     execute invoke(A, nonce=1) → __validate__ called → REVERT (bad signature)
     fee charged to A

Result: victim's account pays fee for a transaction they never signed.
``` [6](#0-5) [7](#0-6)

        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,

        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.

        let cur_span = Span::current();

        tokio::task::spawn_blocking(move || {

/// Perform transaction validation by the mempool.
async fn validate_by_mempool(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<()> {
    let validation_args = ValidationArgs::new(tx, account_nonce);
    mempool_client
        .validate_tx(validation_args)
        .await
        .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
}

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
