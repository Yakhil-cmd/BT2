### Title
`native_per_pubdata` Silently Truncates to Zero via Floor Division, Enabling Free Pubdata Exploitation — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the native-resource cost charged per byte of pubdata — is computed with truncating integer division (`wrapping_div`). When `pubdata_price < native_price`, the result is **0**, making all pubdata completely free for any L2 transaction sender. This is a direct resource accounting bug analogous to the external report's wrong numeric value in a fee formula.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the two key per-unit ratios are computed as follows:

```rust
// native_per_gas uses CEILING division
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata uses FLOOR (truncating) division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`native_per_gas` correctly uses `div_ceil` to ensure the user always pays at least the true cost. `native_per_pubdata` uses `wrapping_div` (floor division), which truncates. When `pubdata_price < native_price`, the result is exactly **0**.

The same floor-division pattern is replicated in the public API helper:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

With `native_per_pubdata = 0`, two downstream charging sites both produce zero cost:

1. **Intrinsic pubdata pre-charge** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [3](#0-2) 

2. **Post-execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)...
   ``` [4](#0-3) 

Both return 0 native cost, so the entire pubdata bill is waived.

The default test block context in `BlockMetadataFromOracle::new_for_test()` already demonstrates this condition: `pubdata_price = 0`, `native_price = 10` → `native_per_pubdata = 0`. [5](#0-4) 

---

### Impact Explanation

When `pubdata_price < native_price` (including the trivially reachable case `pubdata_price = 0`), any unprivileged L2 transaction sender can generate arbitrarily large pubdata (via SSTORE operations, contract deployments, etc.) at zero native-resource cost. The operator must still publish this state-diff data to L1 and pay the associated DA costs, but receives no compensation. This is a direct, repeatable financial loss for the operator and a resource accounting bypass for the attacker.

Even when `pubdata_price >= native_price` but not an exact multiple, the systematic floor truncation undercharges every transaction by up to `(native_price − 1)` native units per pubdata byte, allowing attackers to generate slightly more pubdata than they paid for across many transactions.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are operator-set block-level parameters. The condition `pubdata_price < native_price` can arise from:

- Operator misconfiguration or a deliberate low pubdata price during low-DA-cost periods.
- The default test/bootstrap value (`pubdata_price = 0`) being carried into production.
- Market conditions where the L1 DA cost drops below the native computation price.

An attacker only needs to observe the current block context (publicly readable) and submit a storage-heavy transaction. No privileged access is required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division, mirrors native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, add a protocol-level invariant that rejects blocks where `pubdata_price = 0` unless pubdata is intentionally free by design.

---

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 999`.
2. Bootloader computes: `native_per_pubdata = 999 / 1000 = 0` (floor division).
3. Attacker submits an L2 transaction that writes to 100 new storage slots (each slot write produces ~65 bytes of pubdata → ~6500 bytes total).
4. `get_resources_to_charge_for_pubdata` computes: `native = 6500 * 0 = 0`.
5. Transaction succeeds; attacker pays zero native resources for 6500 bytes of pubdata.
6. Operator publishes 6500 bytes to L1 at their own cost with no compensation.

With `pubdata_price = 0` (the default test value), the attack works unconditionally regardless of `native_price`. [6](#0-5) [7](#0-6)
