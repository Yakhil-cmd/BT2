### Title
Pool admin bypasses `maxAdminSpreadFeeE6` cap via `setPoolBinAdditionalFees`, charging traders uncapped per-bin fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no validation against `maxAdminSpreadFeeE6`. The sibling function `setPoolAdminFees` explicitly enforces this cap, but the per-bin path does not, allowing the pool admin to set per-bin fees up to `uint16` max (65535 in E6, ~6.55%) regardless of the factory-configured cap.

### Finding Description
The factory maintains `maxAdminSpreadFeeE6` (a `uint24` set by the factory owner, hard-capped at 200,000 = 20%) to constrain how much fee a pool admin can impose on traders. `setPoolAdminFees` enforces this:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

But `setPoolBinAdditionalFees` has no equivalent check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The per-bin fees are applied additively to `baseFeeX64` in every swap direction:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

So a pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` and impose ~6.55% additional fee on swaps through that bin, even if `maxAdminSpreadFeeE6` has been lowered to, say, 1000 (0.1%) by the factory owner.

### Impact Explanation
Traders swapping through the affected bin pay fees far above the factory-configured cap. The per-bin fee is applied directly in the swap math, reducing the output token amount the trader receives. With `addFeeBuyE6 = 65535` and `maxAdminSpreadFeeE6 = 1000`, the effective fee is 65x the intended cap, causing direct, measurable loss of swap output above Sherlock thresholds for any non-trivial trade size.

### Likelihood Explanation
The pool admin is a semi-trusted role constrained by factory caps. The cap system exists precisely because pool admins are not fully trusted. A malicious or compromised pool admin can exploit this bypass immediately, with no timelock, no additional preconditions, and no protocol-level defense. The affected bins can be the active (current price) bins, maximizing impact on live traders.

### Recommendation
Add cap enforcement in `setPoolBinAdditionalFees`, mirroring `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept
1. Factory owner sets `maxAdminSpreadFeeE6 = 1000` (0.1%) via `setFeeCaps`.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. Trader calls `swap` routing through bin 0.
4. Effective fee applied = `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ 6.55% additional, far above the 0.1% cap.
5. Assert trader receives significantly less output than they would under the capped fee; assert the difference exceeds Sherlock's medium threshold for the trade size.
