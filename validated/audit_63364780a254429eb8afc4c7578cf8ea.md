### Title
Accept Offer Button Loading Guard Bypassed via Wrong Prop Name, Enabling Multiple Simultaneous Offer Acceptances - (File: packages/gui/src/components/offers2/OfferBuilderViewer.tsx)

### Summary
The "Accept Offer" button in `OfferBuilderViewer` passes `isLoading={isAccepting}` to `ButtonLoading`, but `ButtonLoading`'s internal click guard only checks the `loading` prop. Because `loading` is always `undefined` at this call site, the guard never activates. A user can click "Accept Offer" repeatedly while an acceptance is already in progress, spawning multiple concurrent `takeOffer` RPC calls before any in-flight state is reflected in the React closure.

### Finding Description

`ButtonLoading` destructures `loading` from its props and uses it both to gate clicks and to pass to MUI's `LoadingButton`:

```tsx
// packages/core/src/components/ButtonLoading/ButtonLoading.tsx
const { color = 'secondary', loading, onClick, ...rest } = props;

function handleClick(event) {
  if (!loading && onClick) {   // guard checks `loading`, not `isLoading`
    onClick(event);
  }
}
return <LoadingButton onClick={handleClick} loading={loading} {...rest} />;
``` [1](#0-0) 

The "Accept Offer" button in `OfferBuilderViewer` passes the in-flight flag under the wrong prop name `isLoading`:

```tsx
<ButtonLoading
  onClick={handleAcceptOffer}
  isLoading={isAccepting}   // ← wrong prop; `loading` is never set
  disabled={disableAccept}  // disableAccept = missingOfferedCATs || showInvalid || isExpired
>
  Accept Offer
</ButtonLoading>
``` [2](#0-1) 

Because `loading` is `undefined`, `ButtonLoading.handleClick` always fires `onClick`. `handleAcceptOffer` itself contains no `isAccepting` guard:

```tsx
async function handleAcceptOffer() {
  if (!isWalletSynced) { ... }
  else { offerBuilderRef.current?.submit(); }  // no isAccepting check
}
``` [3](#0-2) 

`handleSubmit` does check `isAccepting`, but `setIsAccepting(true)` is only called inside `acceptOffer` **after** the async confirmation dialog resolves:

```tsx
// handleSubmit
if (isAccepting || !canAccept) { return; }   // stale closure value
await acceptOffer(..., (accepting) => setIsAccepting(accepting), ...);
``` [4](#0-3) 

```tsx
// inside acceptOffer hook
const confirmedAccept = await openDialog(<OfferAcceptConfirmationDialog .../>);
if (!confirmedAccept) return;
try {
  onUpdate?.(true);                                    // setIsAccepting(true) fires HERE
  const response = await takeOffer({ offer: offerData, fee: feeInMojos }).unwrap();
``` [5](#0-4) 

The window between the first click and `onUpdate?.(true)` spans at least one async `offerBuilderDataToOffer` call and one modal dialog interaction. During this entire window `isAccepting` remains `false` in every concurrent closure, so every additional click passes the guard and enqueues another `takeOffer` call.

### Impact Explanation
Each concurrent `takeOffer` call is a signed spend bundle submitted to the Chia full node. If two calls race before the first is confirmed on-chain, both may be accepted, draining the taker's wallet of the offered XCH/CAT/NFT assets twice for a single offer. Even if the node rejects the second spend, the user receives no visual feedback that an acceptance is already in progress (the button shows no spinner and is not disabled), making accidental double-submission highly likely.

### Likelihood Explanation
Any user viewing an importable offer can trigger this by clicking "Accept Offer" more than once before the confirmation dialog appears or while it is open. No special privileges, network access, or attacker involvement is required — the wallet owner alone can trigger the condition through normal UI interaction. The missing spinner makes it especially likely on slow connections where the async operations take noticeable time.

### Recommendation
Pass `loading={isAccepting}` (not `isLoading`) to `ButtonLoading` so the built-in click guard activates:

```tsx
<ButtonLoading
  onClick={handleAcceptOffer}
  loading={isAccepting}        // ← correct prop name
  disabled={disableAccept || isAccepting}
>
```

Additionally, add an early-return guard in `handleAcceptOffer` itself:

```tsx
async function handleAcceptOffer() {
  if (isAccepting) return;
  ...
}
```

### Proof of Concept
1. Open the Chia GUI and import a valid offer file for an XCH/CAT trade.
2. Navigate to the offer viewer (`/dashboard/offers/view`).
3. Click "Accept Offer" once. Observe that the button shows **no loading spinner** and remains fully clickable.
4. Immediately click "Accept Offer" a second time before the confirmation dialog appears (or while it is open in a second tab/window scenario).
5. Confirm both dialogs.
6. Two `takeOffer` RPC calls are dispatched. On a fast node both may be accepted, debiting the taker's wallet twice for the offered assets.

The root cause is confirmed at:
- `ButtonLoading` click guard: [6](#0-5) 
- Wrong prop at call site: [7](#0-6) 
- `setIsAccepting(true)` called only after dialog confirmation: [8](#0-7)
