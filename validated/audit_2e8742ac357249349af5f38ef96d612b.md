### Title
Delegation Rewards Permanently Frozen in Pool Contract When Pool Balance Falls Below `min_delegation_for_rewards` — (File: src/pool/pool.cairo)

---

### Summary

When a pool's total delegated balance is non-zero but below `min_delegation_for_rewards` (1 STRK for STRK pools), the staking contract calculates and transfers STRK reward tokens to the pool contract, but the pool contract's `compute_rewards_per_unit` returns zero, so the cumulative rewards trace never increases. The transferred tokens are permanently frozen in the pool contract with no recovery path for pool members.

---

### Finding Description

The pool contract's `compute_rewards_per_unit` function contains a guard that returns zero when `total_stake < min_delegation_for_rewards` to prevent arithmetic overflow: [1](#0-0) 

For STRK pools, `min_delegation_for_rewards` is `10^18` (1 STRK in native units), set via `STRK_CONFIG.min_for_rewards`: [2](#0-1) 

The pool's `update_rewards_from_staking_contract` inserts `last + compute_rewards_per_unit(...)` into the cumulative rewards trace. When `compute_rewards_per_unit` returns zero, the trace entry is `last + 0 = last` — no rewards are credited to pool members: [3](#0-2) 

However, in the staking contract's `update_pool_rewards`, the STRK reward tokens are **transferred to the pool contract first** via `send_rewards_to_delegation_pool`, and only then is `update_rewards_from_staking_contract` called: [4](#0-3) 

The staking contract's `calculate_staker_pools_rewards` computes `pool_rewards_including_commission` as non-zero whenever `pool_balance_curr_epoch > 0` and `total_stake > 0`, regardless of whether `pool_balance` is below `min_delegation_for_rewards`: [5](#0-4) 

The code itself acknowledges this gap in a comment on `compute_rewards_per_unit`: [6](#0-5) 

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case.

The result: STRK tokens are physically deposited into the pool contract but are permanently inaccessible to pool members because the cumulative rewards trace never reflects them. There is no admin sweep, no recovery function, and no alternative claim path in the pool contract.

---

### Impact Explanation

This is a **permanent freezing of unclaimed yield**. The reward tokens exist in the pool contract's STRK balance but can never be claimed by pool members via `claim_rewards` (which reads the cumulative rewards trace) or recovered via `exit_delegation_pool_action` (which only returns the delegated principal). The amount frozen equals the sum of all `pool_rewards` forwarded during epochs where `pool_balance < 1 STRK`.

---

### Likelihood Explanation

The scenario is reachable through normal, unprivileged protocol operations:

1. **Partial exits**: If pool members call `exit_delegation_pool_intent` and reduce the total pool balance below 1 STRK, subsequent `update_rewards` calls will freeze rewards in the pool contract.
2. **Small initial delegation**: If the minimum delegation amount enforced by `enter_delegation_pool` is less than `min_delegation_for_rewards` (1 STRK), a single pool member delegating a sub-threshold amount is sufficient.

The staking contract's `update_rewards` is called every block by the sequencer, so rewards accumulate and freeze continuously for every block during which the pool balance is sub-threshold.

---

### Recommendation

In `calculate_staker_pools_rewards`, skip pools whose balance is below `min_delegation_for_rewards` before adding them to `pool_rewards_array`. This prevents the staking contract from forwarding rewards that the pool contract cannot distribute:

```cairo
// Before computing pool_rewards_including_commission:
let pool_balance_native = pool_balance_curr_epoch.to_native_amount(:decimals);
let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
if pool_balance_native < pool_dispatcher.get_min_delegation_for_rewards() {
    continue;
}
```

Alternatively, `update_rewards_from_staking_contract` in the pool contract should return any rewards it cannot distribute back to the reward supplier rather than silently accepting them.

---

### Proof of Concept

1. Staker stakes and creates a STRK pool.
2. Pool member delegates 0.5 STRK (below `min_delegation_for_rewards = 1 STRK`) to the pool.
3. After K epochs, the delegation becomes effective (`pool_balance_curr_epoch = 0.5 STRK`).
4. Sequencer calls `update_rewards(staker_address, disable_rewards: false)`.
5. `calculate_staker_pools_rewards` computes `pool_rewards > 0` because `pool_balance_curr_epoch > 0`.
6. `update_pool_rewards` calls `send_rewards_to_delegation_pool`, transferring `pool_rewards` STRK to the pool contract.
7. `pool_dispatcher.update_rewards_from_staking_contract(rewards: pool_rewards, pool_balance: 0.5e18)` is called.
8. Inside the pool, `compute_rewards_per_unit(staking_rewards: pool_rewards, total_stake: 0.5e18)` returns `0` because `0.5e18 < 1e18`.
9. `cumulative_rewards_trace` is updated with `last + 0 = last` — no change.
10. Pool member calls `claim_rewards` → receives 0 STRK.
11. The `pool_rewards` STRK tokens remain permanently locked in the pool contract.

        ///
        /// `staking_rewards` is in `STRK_DECIMALS` decimals.
        /// `total_stake` is in the contract's token decimals.
        /// **Note**: Delegation rewards lost when pool balance is less than
        /// `min_delegation_for_rewards`. The staking contract continues to forward
        /// `pool_rewards` to the pool contract even in this case.
        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
