### Title
Signed Integer Truncation in `delta_gas` Calculation Causes Incorrect Gas Refund Accounting — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment that compensates for native resource consumption is computed by casting two `u64` values to `i64`. When `native_used / native_per_gas` exceeds `i64::MAX`, the cast produces a large negative number, making `delta_gas` incorrectly negative. The `if delta_gas > 0` guard is never entered, so `gas_used` is never incremented to reflect actual native resource consumption. The user receives a larger-than-deserved gas refund, effectively underpaying for native resources.

---

### Finding Description

In `compute_gas_refund` at line 72:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
```

Both `native_used` and `gas_used` are `u64`. Rust's `as i64` cast is a **bit-reinterpretation truncation**: any value above `i64::MAX` (9,223,372,036,854,775,807) becomes negative. There is no checked cast, no saturation, and no bounds assertion.

The `delta_gas` value is the documented mechanism for bridging the gap between EVM gas consumption and native resource consumption (see `docs/double_resource_accounting.md` line 48: `deltaGas := (nativeUsed / nativePerGas) - gasUsed`). When the cast overflows, `delta_gas` is silently negative, the `if delta_gas > 0` branch is skipped, and `gas_used` is left at its EVM-only value — far below the native-resource-equivalent gas that should be charged.

The overflow condition is: `native_used / native_per_gas > i64::MAX`. Since `native_used ≤ gas_limit * native_per_gas` (line 62), this requires `gas_limit > i64::MAX ≈ 9.2 × 10¹⁸`. There is no enforced block-level gas limit cap visible in the validation path that would prevent a user from submitting such a transaction. [1](#0-0) 

The `compute_gas_refund` function is called for every L2 transaction (via `before_refund` in `zk/mod.rs` line 427) and every L1 transaction (via `process_l1_transaction.rs` line 266). [2](#0-1) [3](#0-2) 

---

### Impact Explanation

The `delta_gas` mechanism exists precisely to ensure that a transaction whose native resource consumption exceeds its EVM gas consumption pays the difference. When the cast overflows:

1. `delta_gas` is computed as a large negative number.
2. The `if delta_gas > 0` guard at line 75 is never entered.
3. `gas_used` remains at its EVM-only value (e.g., 21,000 for a minimal call).
4. `total_gas_refund = gas_limit - gas_used` is nearly equal to `gas_limit`.
5. The user is refunded `gas_price × (gas_limit − gas_used) ≈ gas_price × gas_limit`.
6. The user effectively pays only for EVM gas, not for the native resource consumption that the `delta_gas` adjustment was designed to charge.

The operator/protocol absorbs the unrecovered native resource cost. This is a **resource accounting bug** with a direct economic impact: users can underpay for native resource consumption. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**Trigger condition**: `gas_limit > i64::MAX ≈ 9.2 × 10¹⁸`. The attacker must hold enough balance to prepay `gas_price × gas_limit`. With `gas_price = 1` (minimum), this is 9.2 × 10¹⁸ tokens — a large but not impossible amount for a whale or a protocol-level actor on a chain with low-value tokens or a testnet. No block-level gas cap is enforced in the visible validation path. The test suite itself uses gas limits as high as 34 trillion (`gas_limit: 34_158_000_000_000`) without rejection, confirming the absence of a hard cap. [6](#0-5) 

The attacker controls `gas_limit` directly in the transaction fields, making this an **unprivileged, externally reachable** entry path.

---

### Recommendation

Replace the unchecked `as i64` casts with saturating or checked arithmetic. The semantically correct fix is to compute `delta_gas` entirely in `u64` (or `i128`) and only convert at the comparison boundary:

```rust
// Safe version using i128 to avoid truncation
let native_gas_equiv = (native_used / native_per_gas) as i128;
let gas_used_i128 = gas_used as i128;
let delta_gas = native_gas_equiv - gas_used_i128;
if delta_gas > 0 {
    gas_used = gas_used.saturating_add(delta_gas as u64);
}
```

Alternatively, enforce a maximum `gas_limit` of `i64::MAX` during transaction validation, consistent with the implicit assumption in this arithmetic.

---

### Proof of Concept

1. Attacker holds balance ≥ `gas_price × (i64::MAX + 1)`.
2. Attacker submits an L2 EIP-1559 transaction with:
   - `gas_limit = i64::MAX + 1 = 9_223_372_036_854_775_808u64`
   - `max_fee_per_gas = 1`, `max_priority_fee_per_gas = 1`
   - Calldata that triggers significant native resource consumption (e.g., many storage writes) but minimal EVM gas (e.g., 21,000 gas).
3. Transaction executes. `native_used ≈ gas_limit × native_per_gas`.
4. In `compute_gas_refund`:
   - `native_used / native_per_gas ≈ 9_223_372_036_854_775_808u64`
   - `(9_223_372_036_854_775_808u64) as i64 = i64::MIN = -9_223_372_036_854_775_808`
   - `gas_used ≈ 21_000`
   - `delta_gas = -9_223_372_036_854_775_808 - 21_000 = very negative`
   - `delta_gas > 0` → false → `gas_used` stays at 21,000.
5. `total_gas_refund = gas_limit - 21_000 ≈ gas_limit`.
6. Attacker is refunded nearly the entire prepaid fee, paying only for 21,000 EVM gas despite consuming native resources equivalent to `gas_limit` gas units. [7](#0-6)
