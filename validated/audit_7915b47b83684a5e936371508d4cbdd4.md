### Title
`protocolUnpausePool` Sets Wrong Pause Level (1 Instead of 0), Leaving Pool Permanently Swap-Blocked — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` transitions the pool to pause level `1` (admin-paused) instead of `0` (active). After the protocol "unpauses" a pool, swaps remain blocked and the protocol has no path to restore full functionality without admin cooperation.

### Finding Description

The pool defines three pause levels:

> `0 = active, 1 = paused by admin, 2 = paused by protocol` [1](#0-0) 

Any non-zero level blocks swaps: [2](#0-1) 

`protocolPausePool` correctly escalates from level `0` **or** `1` to level `2`: [3](#0-2) 

But `protocolUnpausePool` always writes level `1`, not `0`: [4](#0-3) 

When the protocol pauses a pool that was **active** (level `0`) and later calls `protocolUnpausePool`, the pool lands at level `1` (admin-paused). The `swap` modifier `whenNotPaused` still reverts, so swaps remain completely blocked. [5](#0-4) 

The only recovery path is for the pool admin to separately call `unpausePool` (level `1 → 0`): [6](#0-5) 

If the admin is unresponsive, malicious, or simply unaware, the pool is permanently swap-frozen despite the protocol having "unpaused" it.

### Impact Explanation

- Swaps are blocked at `whenNotPaused` for any pool that was active before a protocol pause, after `protocolUnpausePool` is called.
- The protocol owner has no direct path to restore level `0`; the factory enforces `2 → 1` as the only valid transition from `protocolUnpausePool`.
- Liquidity removal remains available (no `whenNotPaused` on `removeLiquidity`), so LP principal is not trapped, but the core swap flow is broken.
- This matches the allowed impact: **broken core pool functionality causing unusable swap flows**.

### Likelihood Explanation

- Trigger: the protocol owner calls `protocolPausePool` on an active pool, then later calls `protocolUnpausePool`. Both are normal operational actions.
- No attacker required; the bug fires on every legitimate protocol-pause/unpause cycle for a pool that was at level `0`.
- The admin must then independently call `unpausePool`; if they do not, the pool stays frozen indefinitely.

### Recommendation

Change `protocolUnpausePool` to set the pause level to `0` (active) instead of `1`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0); // target is 0, not 1
    IMetricOmmPoolFactoryActions(pool).setPause(0);       // restore to active
}
```

If the design intent is to hand control back to the admin (level `1`) when the pool was admin-paused before the protocol pause, the factory must store the pre-protocol-pause level and restore it. The current implementation does not do this and always writes `1` regardless of prior state.

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → pool transitions `0 → 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` → pool transitions `2 → 1` (admin-paused, **not** active).
4. Any call to `swap(...)` reverts with `PoolPaused()` because `pauseLevel == 1 != 0`.
5. The protocol owner has no further factory function to set the pool back to `0`; only the pool admin's `unpausePool` can do so.
6. If the admin does not act, all swap activity on the pool is permanently frozen.
