Let me search the nearcore codebase for patterns analogous to the buffer size miscalculation bug - specifically looking at storage accounting, capacity calculations, and size-related invariants.

### Title
Receipt Size Limit Bypassed via `promise_return` Post-Validation Mutation of `output_data_receivers` - (File: `runtime/runtime/src/lib.rs`)

### Summary

The `max_receipt_size` pre-inclusion invariant is broken by a two-phase receipt construction pattern: a new action receipt is validated for size before the runtime appends `output_data_receivers` to it during `promise_return` processing. The final, enlarged receipt is forwarded without re-validation, allowing an unprivileged user to inject receipts that exceed `max_receipt_size` into the network. The nearcore codebase explicitly acknowledges this as an open bug (issue #12606).

### Finding Description

The analog to the Buffer `init` bug is exact: a size bound is checked against an incomplete object, then the object is grown past that bound with no second check.

**Step 1 — Receipt C is created and validated at size ≤ `max_receipt_size`.**

`validate_receipt` with `ValidateReceiptMode::NewReceipt` measures the borsh-serialized size of each newly generated receipt and rejects it if it exceeds the limit: [1](#0-0) 

**Step 2 — After validation, the runtime appends the parent receipt's `output_data_receivers` to receipt C.**

When a contract calls `promise_return(C)` and the parent action receipt A has non-empty `output_data_receivers` (i.e., A is itself a callback dependency), the runtime mutates C's `output_data_receivers` in-place: [2](#0-1) 

Each appended `DataReceiver` adds `sizeof(CryptoHash) + sizeof(AccountId)` bytes to the receipt. No re-validation of the receipt size occurs after this mutation.

**Step 3 — The oversized receipt is forwarded without re-check.**

The receipt is then assigned an ID and forwarded or buffered: [3](#0-2) 

**Step 4 — Incoming processing uses the lenient `ExistingReceipt` mode.**

When the oversized receipt arrives at the destination shard, `validate_receipt` is called with `ValidateReceiptMode::ExistingReceipt`, which skips the size check entirely: [4](#0-3) 

The comment in `ExistingReceipt` explicitly states: *"There is a bug which allows to create receipts that are above the size limit."* [5](#0-4) 

The test suite confirms and demonstrates the bug: [6](#0-5) 

### Impact Explanation

The corrupted invariant is: `borsh::object_length(receipt) ≤ limit_config.max_receipt_size` for every receipt in the network. An oversized receipt:

- Bypasses the `max_receipt_size` = 4 MiB limit (protocol version 69+).
- Can push state witness sizes past `main_storage_proof_size_soft_limit` and `per_receipt_storage_proof_size_limit`, since the witness must include the full receipt.
- Corrupts congestion control accounting: `compute_receipt_size` correctly measures the actual larger size, so the congestion byte counters will reflect a receipt that was never supposed to exist.
- The `max_receipt_size` limit is a pre-inclusion transaction validation invariant; its violation means the network carries data that all validators agreed to exclude.

### Likelihood Explanation

Any unprivileged user who can deploy a contract can trigger this. The attack requires:
1. Deploy a contract that calls `promise_create` + `promise_return` with args sized to `max_receipt_size - base_receipt_overhead`.
2. Call that contract from a transaction that has a `.then()` continuation (so the parent receipt has non-empty `output_data_receivers`).

The test `test_max_receipt_size_promise_return` demonstrates this is reliably reproducible with a standard test contract. [7](#0-6) 

### Recommendation

Re-validate the receipt size **after** appending `output_data_receivers` in `runtime/runtime/src/lib.rs`. Specifically, after the `extend_from_slice` calls at lines 1029–1035, call `validate_receipt` with `ValidateReceiptMode::NewReceipt` on the mutated receipt and return a `NewReceiptValidationError::ReceiptSizeExceeded` action error if the limit is exceeded. This mirrors the existing guard applied to directly-oversized receipts.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept:

1. Deploy `rs_contract`.
2. Call `max_receipt_size_promise_return_method1` with `args_size = max_receipt_size - base_receipt_size`. This creates promise C at exactly `max_receipt_size` bytes, which passes `NewReceipt` validation.
3. The runtime then appends the parent's `output_data_receivers` to C, making it exceed `max_receipt_size`.
4. `assert_oversized_receipt_occurred` confirms the oversized receipt is present in the chain. [8](#0-7)
