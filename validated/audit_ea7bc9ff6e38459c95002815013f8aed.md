### Title
Unvalidated Participant Contributions in CKD Protocol Allow Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary

In `do_ckd_coordinator`, the coordinator accumulates `CKDOutput` values (`big_y`, `big_c`) from each participant and sums them directly without any proof of correct formation. A malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted `(Y, C)` pair. Honest parties accept this corrupted output, permanently invalidating the derived confidential key for that `app_id`.

### Finding Description

The CKD protocol is designed so that each participant `P_i` computes:

- `big_y_i = y_i * G` (ephemeral public key)
- `big_c_i = x_i * H(pk || app_id) + y_i * app_pk` (masked secret share)
- `norm_big_y_i = λ_i * big_y_i`, `norm_big_c_i = λ_i * big_c_i`

and sends `(norm_big_y_i, norm_big_c_i)` privately to the coordinator. The coordinator sums all contributions to produce the final `(Y, C)`.

The root cause is in `do_ckd_coordinator`: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no proof of knowledge that the participant knows `y_i` (the discrete log of `big_y_i`), and no proof that `big_c_i` is correctly formed as `x_i * H(pk || app_id) + y_i * app_pk`. The coordinator blindly accumulates whatever group elements are received.

Compare this to the DKG protocol, which requires a Schnorr proof of knowledge for every polynomial commitment before accepting it: [2](#0-1) 

and validates each received secret share against the commitment: [3](#0-2) 

No equivalent validation exists in the CKD coordinator path. [4](#0-3) 

### Impact Explanation

A malicious participant sends crafted `(norm_big_y, norm_big_c)` values — for example, the identity element `(0, 0)`, or arbitrary group elements `(Δ_Y, Δ_C)`. The coordinator computes:

```
Y_final  = Σ norm_big_y_i  (includes attacker's Δ_Y)
C_final  = Σ norm_big_c_i  (includes attacker's Δ_C)
```

The resulting `CKDOutput` is accepted by the coordinator as valid. The application then calls `ckd_output.unmask(app_sk)` to recover `C - app_sk * Y`, which equals `msk * H(pk || app_id) + Δ_C - app_sk * Δ_Y` — a value that is not the intended confidential key. The derived key is permanently corrupted for that `(pk, app_id)` pair.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any participant in the CKD protocol can trigger this. The attacker needs only a valid participant role (no privileged access, no leaked keys). The attack is a single-round, single-message substitution: replace the honest `(norm_big_y, norm_big_c)` with arbitrary group elements. No cryptographic primitive needs to be broken. The coordinator has no mechanism to detect the substitution.

### Recommendation

Add a Sigma/Schnorr proof of knowledge for `y_i` (proving knowledge of the discrete log of `big_y_i`) and a proof that `big_c_i` is correctly formed relative to `big_y_i` and the participant's committed public key share. This is the same pattern already used in the DKG and triple-generation protocols: [5](#0-4) 

Concretely:
1. Each participant generates a `dlog` proof for `y_i` (proving `big_y_i = y_i * G`).
2. Each participant generates a `dlogeq` proof proving that the same `y_i` used in `big_y_i` is used in `big_c_i` (i.e., `big_c_i - x_i * H(pk || app_id) = y_i * app_pk`).
3. The coordinator verifies both proofs before accumulating each participant's contribution.

### Proof of Concept

1. Run the CKD protocol with `N` participants, one of which is malicious.
2. The malicious participant, instead of computing the honest `(norm_big_y, norm_big_c)`, sends `(identity, identity)` (or any arbitrary pair) to the coordinator.
3. The coordinator's loop at lines 50–55 of `protocol.rs` adds these values without any check.
4. The final `CKDOutput` returned by the coordinator is `(Y - honest_Y_i, C - honest_C_i)` — missing the malicious participant's legitimate contribution.
5. `ckd_output.unmask(app_sk)` returns a value that differs from `msk * H(pk || app_id)`, confirming the derived confidential key is corrupted. [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
