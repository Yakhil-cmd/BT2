### Title
Pool Admin Can Set Uncapped Bin Additional Fees, Bypassing Protocol Fee Caps - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the pool admin's fee components, but the sibling function `setPoolBinAdditionalFees` passes per-bin fee values directly to the pool with **no cap check at all**. A semi-trusted pool admin can exploit this gap to charge traders fees that exceed the protocol-intended hard maximum.

### Finding Description

The factory enforces fee caps in two places:

**`setPoolAdminFees` — cap enforced:** [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

**`setPoolBinAdditionalFees` — no cap enforced:** [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` also applies no cap: [3](#0-2) 

During a swap, the bin additional fee is added directly on top of the oracle-derived base fee: [4](#0-3) 

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

The `uint16` type permits values up to `65,535`, which in E6 units equals **6.5535%**. The hard cap on the total spread fee is `HARD_MAX_SPREAD_FEE_E6 = 200,000` (20%): [5](#0-4) 

A pool admin who has already set `adminSpreadFeeE6` to the maximum allowed value can additionally set bin additional fees to `uint16.max`, making the total effective fee for that bin **26.5535%** — 6.5535 percentage points above the hard cap.

This is the direct analog to the external oracle bug: the minimum-stake check is enforced at `submit_price` but silently absent at `handle_payout`. Here, the fee cap is enforced at `setPoolAdminFees` but entirely absent at `setPoolBinAdditionalFees`.

### Impact Explanation

Traders swapping through a bin with `addFeeBuyE6 = 65535` or `addFeeSellE6 = 65535` receive a worse execution price than the protocol's hard cap permits. The excess fee accrues to LPs (including the pool admin if they hold LP shares). This is a direct loss of user principal — traders pay more than the protocol-intended maximum on every swap through the affected bin.

### Likelihood Explanation

The pool admin is explicitly described as semi-trusted "only inside caps and timelocks." Any pool admin — whether malicious, compromised, or simply misconfigured — can trigger this with a single transaction. No timelock applies to `setPoolBinAdditionalFees`. The call requires only the `onlyPoolAdmin` modifier, which is a normal operational role.

### Recommendation

Add cap validation to `setPoolBinAdditionalFees` in `MetricOmmPoolFactory.sol`, analogous to the checks in `setPoolAdminFees`. Define a `maxAdminBinAdditionalFeeE6` cap (or reuse `maxAdminSpreadFeeE6`) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept

1. Pool is deployed with `adminSpreadFeeE6 = maxAdminSpreadFeeE6 = 200_000` (20% — the hard cap).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — passes with no revert.
3. A trader calls `swap(... zeroForOne=true ...)` routing through bin 0.
4. Inside `_swapToken0ForToken1SpecifiedInput`, the effective buy fee is computed as:
   `buyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`
   — adding 6.5535% on top of the already-maximum 20% spread fee.
5. The trader receives 6.5535% less token1 than the hard cap permits, with the excess retained in the pool for LPs.
6. The pool admin, holding LP shares, collects the excess via `removeLiquidity`.
