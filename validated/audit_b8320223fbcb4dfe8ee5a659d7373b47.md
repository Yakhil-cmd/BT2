### Title
Front-runnable `SaveGrant` allows grantee to spend old spend limit before reduction takes effect - (`sei-cosmos/x/authz/keeper/keeper.go`)

### Summary

`SaveGrant` in the `x/authz` module unconditionally overwrites an existing authorization grant. When a granter reduces a grantee's `SendAuthorization` spend limit by submitting a new `MsgGrant`, the grantee can observe the pending transaction in the mempool and front-run it — spending the full old limit before the new lower limit is written — then spend the new limit as well, draining more funds than the granter intended.

### Finding Description

`SaveGrant` in `sei-cosmos/x/authz/keeper/keeper.go` directly sets the grant store key with the new authorization, overwriting any existing grant for the same `(granter, grantee, MsgTypeURL)` triple without any check on whether the new limit is lower than the current remaining limit:

### Impact Explanation

A granter who wants to reduce a grantee's `SendAuthorization` from X to Y submits a new `MsgGrant` with `spend_limit = Y`. The grantee sees this in the mempool, submits `MsgExec { MsgSend amount=X }` with a higher gas price (priority), gets included first, drains X from the granter's account, and then after the new grant lands, submits another `MsgExec { MsgSend amount=Y }`. Total drained: X + Y instead of Y. The granter loses X extra tokens — an amount they explicitly tried to revoke access to.

If X ≥ $5k this is Critical (direct fund loss ≥ $5k). If X < $5k it is Medium.

### Likelihood Explanation

- The grantee is an unprivileged user who already holds a valid grant.
- Cosmos mempool ordering is by transaction priority (gas price). A grantee can trivially submit a higher-priority `MsgExec` to race ahead of the granter's `MsgGrant`.
- No special access (validator key, governance, admin) is required.
- The attack is deterministic once the granter's `MsgGrant` is visible in the mempool.

### Recommendation

1. **Require revoke-then-grant** (as `x/feegrant` already does — it rejects `MsgGrantAllowance` if a grant already exists): force the granter to call `MsgRevoke` first, then `MsgGrant`. This eliminates the overwrite window, though it introduces a brief revocation window; that window is safer because the grantee has *no* authorization during it.
2. **Alternatively**, add an explicit decrease-only path: if a new `SendAuthorization` is submitted for an existing grant and the new limit is lower than the current remaining limit, subtract the difference rather than overwriting.

### Proof of Concept

1. Granter calls `MsgGrant` → `SendAuthorization{SpendLimit: 1000usei}` for grantee.
2. Granter decides to reduce to 200usei and broadcasts `MsgGrant` → `SendAuthorization{SpendLimit: 200usei}`.
3. Grantee observes the pending `MsgGrant` in the mempool.
4. Grantee immediately broadcasts `MsgExec { MsgSend from=granter to=grantee amount=1000usei }` with a higher gas price.
5. Grantee's `MsgExec` is ordered first in the block; `SendAuthorization.Accept` subtracts 1000 from 1000, limit hits zero, grant is deleted.
6. Granter's `MsgGrant` is processed next; `SaveGrant` writes a fresh `SendAuthorization{SpendLimit: 200usei}`.
7. Grantee broadcasts another `MsgExec { MsgSend amount=200usei }` — succeeds.
8. **Total drained: 1200usei. Granter intended to cap at 200usei.**

// SaveGrant method grants the provided authorization to the grantee on the granter's account
// with the provided expiration time. If there is an existing authorization grant for the
// same `sdk.Msg` type, this grant overwrites that.
func (k Keeper) SaveGrant(ctx sdk.Context, grantee, granter sdk.AccAddress, authorization authz.Authorization, expiration time.Time) error {
	store := ctx.KVStore(k.storeKey)

	grant, err := authz.NewGrant(authorization, expiration)
	if err != nil {
		return err
	}

	bz := k.cdc.MustMarshal(&grant)
	skey := grantStoreKey(grantee, granter, authorization.MsgTypeURL())
	store.Set(skey, bz)
	return ctx.EventManager().EmitTypedEvent(&authz.EventGrant{
		MsgTypeUrl: authorization.MsgTypeURL(),
		Granter:    granter.String(),
		Grantee:    grantee.String(),
	})
}

func (k Keeper) Grant(goCtx context.Context, msg *authz.MsgGrant) (*authz.MsgGrantResponse, error) {
	ctx := sdk.UnwrapSDKContext(goCtx)
	grantee, err := sdk.AccAddressFromBech32(msg.Grantee)
	if err != nil {
		return nil, err
	}

	granter, err := sdk.AccAddressFromBech32(msg.Granter)
	if err != nil {
		return nil, err
	}

	authorization := msg.GetAuthorization()
	if authorization == nil {
		return nil, sdkerrors.ErrUnpackAny.Wrap("Authorization is not present in the msg")
	}

	t := authorization.MsgTypeURL()
	if k.router.HandlerByTypeURL(t) == nil {
		return nil, sdkerrors.Wrapf(sdkerrors.ErrInvalidType, "%s doesn't exist.", t)
	}

	err = k.SaveGrant(ctx, grantee, granter, authorization, msg.Grant.Expiration)
	if err != nil {
		return nil, err
	}

	return &authz.MsgGrantResponse{}, nil

func (a SendAuthorization) Accept(ctx sdk.Context, msg sdk.Msg) (authz.AcceptResponse, error) {
	mSend, ok := msg.(*MsgSend)
	if !ok {
		return authz.AcceptResponse{}, sdkerrors.ErrInvalidType.Wrap("type mismatch")
	}
	limitLeft, isNegative := a.SpendLimit.SafeSub(mSend.Amount)
	if isNegative {
		return authz.AcceptResponse{}, sdkerrors.ErrInsufficientFunds.Wrapf("requested amount is more than spend limit")
	}
	if limitLeft.IsZero() {
		return authz.AcceptResponse{Accept: true, Delete: true}, nil
	}

	return authz.AcceptResponse{Accept: true, Delete: false, Updated: &SendAuthorization{SpendLimit: limitLeft}}, nil
