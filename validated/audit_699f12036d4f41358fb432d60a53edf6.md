### Title
`key_manager` Can Bypass Owner Emergency Pause via `start_hashchain` - (File: engine/src/contract_methods/admin.rs)

### Summary
The `start_hashchain` function, restricted to the `key_manager` role, unconditionally sets `state.is_paused = false` as a side effect, allowing the `key_manager` to silently re-enable a contract that the `owner` deliberately paused as an emergency measure. This is a direct privilege-hierarchy inversion: the lower-privileged `key_manager` can override the higher-privileged `owner`'s security action.

### Finding Description
Aurora Engine defines two distinct administrative roles relevant to the pause mechanism:

- **`owner`** — the highest-privilege account; the only role permitted to call `pause_contract` and `resume_contract`.
- **`key_manager`** — a lower-privilege account whose intended scope is managing relayer function-call keys (`add_relayer_key`, `remove_relayer_key`).

`pause_contract` sets `state.is_paused = true` and is gated by `require_owner_only`: [1](#0-0) 

`resume_contract` sets `state.is_paused = false` and is also gated by `require_owner_only`: [2](#0-1) 

However, `start_hashchain` — gated only by `require_key_manager_only` — also unconditionally sets `state.is_paused = false` at line 460 before writing state: [3](#0-2) 

The `key_manager` is set by the owner via `set_key_manager` and is a separate, lower-privileged account in production. The `start_hashchain` function's primary purpose is to initialize the hashchain integrity mechanism, but its side effect of clearing `is_paused` is unguarded by any owner check. This means the `key_manager` can call `start_hashchain` at any time the contract is paused — including during an emergency pause — and the contract will resume.

The `EngineState` struct confirms `is_paused` is the single flag controlling the running/paused state: [4](#0-3) 

The `require_running` guard used throughout all state-mutating methods (submit, deploy, bridge exits) reads this same flag: [5](#0-4) 

### Impact Explanation
When the owner pauses the contract as an emergency response (e.g., to halt an active bridge exploit), all EVM transaction submission and bridge precompile execution is blocked. The `exit_to_near` and `exit_to_ethereum` precompiles — which move ETH and ERC-20 tokens out of the EVM — are gated by `require_running`. If the `key_manager` calls `start_hashchain`, the pause is lifted and the bridge precompiles are immediately re-enabled, allowing an ongoing exploit to continue draining bridged funds. This constitutes a **temporary (or permanent, if repeated) bypass of the emergency freeze**, enabling continued theft of funds in motion. [6](#0-5) 

### Likelihood Explanation
In production, the `owner` and `key_manager` are distinct accounts (e.g., a multisig DAO vs. a hot-wallet relayer manager). The `key_manager` may call `start_hashchain` without understanding it resumes the contract, or a compromised `key_manager` key could intentionally exploit this. The design flaw is structural: the privilege hierarchy is not enforced at the code level, mirroring the keep-core pattern exactly.

### Recommendation
Remove the `state.is_paused = false` assignment from `start_hashchain`. The hashchain can be initialized independently of the pause state. If resuming the contract after hashchain initialization is desired, it should be a separate explicit call to `resume_contract` gated by `require_owner_only`, preserving the privilege hierarchy.

### Proof of Concept
1. Owner calls `pause_contract` — `state.is_paused = true`. All `submit`, `deploy_code`, and bridge precompile calls now fail with `ERR_PAUSED`.
2. `key_manager` calls `start_hashchain` with any valid `StartHashchainArgs` (block height ≤ current block height).
3. `start_hashchain` passes `require_paused` (contract is paused — check passes), passes `require_key_manager_only` (caller is key_manager — check passes), initializes the hashchain, then executes `state.is_paused = false` and writes state.
4. Contract is now running. The owner's emergency pause has been silently overridden.
5. Bridge precompiles (`exit_to_near`, `exit_to_ethereum`) are re-enabled; an attacker can resume draining bridged funds. [3](#0-2)

#[named]
pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }

    hashchain.add_block_tx(
        block_height,
        function_name!(),
        &input,
        &[],
        &Bloom::default(),
    )?;
    crate::hashchain::save_hashchain(&mut io, &hashchain)?;

    state.is_paused = false;
    state::set_state(&mut io, &state)?;

    Ok(())
}
