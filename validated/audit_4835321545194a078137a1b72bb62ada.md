### Title
Stale `DeprecatedPendingBlock` Fallback Silently Zeros `l2_gas_price` and `l1_data_gas_price` in `estimate_message_fee` Pending Context — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When `estimate_message_fee` is called with `block_id=Tag(Pending)` and the cached pending data is stale (its `parent_block_hash` does not match the latest committed block hash), `read_pending_data` synthesizes a `DeprecatedPendingBlock` fallback. That struct has no `l1_data_gas_price` or `l2_gas_price` fields, so both return `GasPricePerToken::default()` (all zeros). Those zeros flow through `client_pending_data_to_execution_pending_data` into `ExecutionPendingData`, and then into `create_block_context`, where each zero is silently replaced by `NonzeroGasPrice::MIN = 1`. The `BlockContext` used for the L1Handler fee estimation therefore carries `l2_gas_price = 1` and `l1_data_gas_price = 1` instead of the real committed-block values, producing a drastically wrong `overall_fee`.

### Finding Description

**Step 1 — `read_pending_data` stale-data fallback** [1](#0-0) 

When `pending_data.block.parent_block_hash() != latest_header.block_hash`, the function constructs a synthetic `DeprecatedPendingBlock`. It copies `eth_l1_gas_price` and `strk_l1_gas_price` from the committed header, but uses `..Default::default()` for all other fields. `DeprecatedPendingBlock` has no `l1_data_gas_price` or `l2_gas_price` fields at all. [2](#0-1) 

**Step 2 — `l2_gas_price()` and `l1_data_gas_price()` return zero for `Deprecated` variant** [3](#0-2) 

Both methods explicitly return `GasPricePerToken::default()` (all-zero) for the `Deprecated` arm, with comments acknowledging this is intentional for old format compatibility.

**Step 3 — `client_pending_data_to_execution_pending_data` passes zeros through** [4](#0-3) 

The converter reads `block.l2_gas_price()` and `block.l1_data_gas_price()` directly and stores them in `ExecutionPendingData` without any correction.

**Step 4 — `estimate_message_fee` uses this data for the pending `BlockContext`** [5](#0-4) 

**Step 5 — `create_block_context` silently replaces zero with `NonzeroGasPrice::MIN = 1`** [6](#0-5) 

`NonzeroGasPrice::MIN` is defined as `GasPrice(1)`: [7](#0-6) 

The `BlockContext` used for the L1Handler execution therefore has `l2_gas_price = 1 wei` and `l1_data_gas_price = 1 wei` instead of the real values (which are typically in the billions of wei range on mainnet).

### Impact Explanation

`estimate_message_fee` is the canonical RPC method users and bridges call to determine how much ETH to attach to an L1→L2 message. In modern Starknet (post-0.13.1), L1Handler execution consumes L2 gas and L1 data gas in addition to L1 gas. With `l2_gas_price = 1` and `l1_data_gas_price = 1`, the returned `overall_fee` is off by a factor equal to the real gas price (billions), making it appear the message is nearly free. Users who rely on this estimate will attach far too little fee, causing their L1→L2 messages to fail on L2 or be under-funded. The `l1_gas_price` is correctly set from the committed block, so only the L2 and data-gas components are corrupted.

### Likelihood Explanation

The stale-pending-data path is a **normal operational condition**, not an edge case. It fires whenever a new block has been committed but the pending-sync loop has not yet refreshed the `Arc<RwLock<PendingData>>`. Given that pending sync polls on a timer, there is always a window — potentially seconds wide — after every block where any call to `estimate_message_fee(block_id=Pending)` hits this path. No special attacker capability is required; any unprivileged RPC caller can trigger it by timing their call to this window, or simply by calling repeatedly.

### Recommendation

In the stale-data fallback inside `read_pending_data`, use a `PendingBlockOrDeprecated::Current(PendingBlock {...})` instead of `Deprecated`, and populate `l1_data_gas_price` and `l2_gas_price` from the committed block header (same as `l1_gas_price` is already copied). Alternatively, add a `PendingBlock`-style struct to the fallback path that carries all three price fields from the latest committed header, so that `l2_gas_price()` and `l1_data_gas_price()` never silently return zero.
