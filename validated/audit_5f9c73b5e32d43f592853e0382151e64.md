### Title
Missing Domain Binding in `validateCOAOwnershipProof` Enables Cross-Context Signature Replay — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof()` verifies Cadence account key signatures over a caller-supplied `signedData` blob without enforcing that `signedData` encodes the target COA's EVM address. The function's own inline comment acknowledges this gap. Because the domain separation tag is the generic `"FLOW-V0.0-user"` and `signedData` is fully attacker-controlled, any valid signature a user has ever produced under that tag — for any purpose — can be replayed as a COA ownership proof. The ERC-1271 `isValidSignature` entry point on the COA Solidity contract makes this directly exploitable from EVM.

---

### Finding Description

`EVM.validateCOAOwnershipProof()` in `fvm/evm/stdlib/contract.cdc` (lines 1011–1116) accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. It verifies that the provided signatures cover `signedData` using the account's keys and the fixed domain tag `"FLOW-V0.0-user"`, then checks that the COA stored at `path` has address `evmAddress`.

The critical gap: **`signedData` is never required to contain `evmAddress`** (or any other context-binding field). The function's own comment states this explicitly:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The verification call at line 1082–1086 uses the generic `"FLOW-V0.0-user"` domain tag: [2](#0-1) 

The EVM-side entry point is the Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)`, where the `bytes32` argument is passed verbatim as `signedData`: [3](#0-2) [4](#0-3) 

The COA Solidity contract's ERC-1271 `isValidSignature` implementation passes the caller-supplied `_hash` directly as the `bytes32` argument: [5](#0-4) 

This means any EVM contract can invoke `isValidSignature(_hash, _sig)` on a COA with an arbitrary `_hash` and a previously-observed signature, and the proof will validate if the signature was ever produced over that hash with the `"FLOW-V0.0-user"` tag.

---

### Impact Explanation

**Vulnerability class:** Cross-context signature replay / missing domain binding in a signature verification primitive.

An attacker who observes a valid `(signedData, signature)` pair that a Flow account holder produced under the `"FLOW-V0.0-user"` tag for *any* purpose (e.g., a dApp authentication flow, a prior COA proof for a different protocol) can replay it as a COA ownership proof. Because COAs are used as EVM smart-contract wallets and implement ERC-1271, a successful replay causes EVM protocols to accept the attacker's call as if it were authorized by the COA owner. Concrete consequences include:

- Unauthorized ERC-20/ERC-721 approvals or transfers gated behind ERC-1271 checks on the COA.
- Bypassing access controls in EVM DeFi protocols that use `isValidSignature` to authenticate the COA owner.
- Cross-address replay if a Cadence account holds multiple COAs at different paths (the attacker supplies a different `path`/`evmAddress` pair while reusing the same signature). [6](#0-5) 

---

### Likelihood Explanation

**Attacker-controlled entry path:** Any unprivileged EVM transaction sender can call the Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)` with arbitrary `bytes32` and `bytes` arguments. Any Cadence script can call `EVM.validateCOAOwnershipProof()` directly with arbitrary `signedData`. No privileged keys or staked nodes are required.

**Preconditions:**
1. The target Cadence account has a COA.
2. The attacker has observed a valid signature the account holder produced under the `"FLOW-V0.0-user"` tag over any 32-byte value (e.g., from a prior on-chain proof, a dApp authentication, or any other use of `Crypto.KeyList.verify()` with that tag).
3. An EVM protocol uses ERC-1271 `isValidSignature` on the COA to gate an asset operation.

All three conditions are realistic in a live ecosystem where COAs are used as smart-contract wallets.

---

### Recommendation

Enforce that `signedData` encodes the target `evmAddress` (and ideally the Flow chain ID and a nonce or timestamp) before verifying signatures. The simplest fix is to require inside `validateCOAOwnershipProof` that `signedData` is at least 20 bytes and that its first 20 bytes equal `evmAddress`. A stronger fix follows the EIP-712 pattern: define a structured message type that includes `evmAddress`, `chainId`, and a nonce, and require callers to sign that structure. The `"FLOW-V0.0-user"` domain tag should also be made context-specific (e.g., `"FLOW-V0.0-COA-ownership"`) to prevent cross-protocol replay.

---

### Proof of Concept

1. Alice holds a Flow account with key index 0 and a COA at `/public/coa`.
2. Alice signs `hash32` with the `"FLOW-V0.0-user"` tag for dApp A (e.g., to authenticate). The signature `sig` is published on-chain.
3. An attacker constructs an EVM transaction that calls a DeFi protocol's function gated by ERC-1271 on Alice's COA.
4. The DeFi protocol calls `isValidSignature(hash32, encodedProof)` on Alice's COA contract.
5. The COA contract calls the Cadence Arch precompile: `verifyCOAOwnershipProof(coaAddress, hash32, encodedProof)`.
6. The precompile calls `EVM.validateCOAOwnershipProof(aliceFlowAddr, /public/coa, hash32, [0], [sig], coaAddressBytes)`.
7. The function verifies `sig` over `hash32` with `"FLOW-V0.0-user"` — this passes because Alice signed exactly that data.
8. The function checks the COA address — this passes because the attacker supplied the correct `evmAddress`.
9. `isValidSignature` returns `0x1626ba7e` (valid), and the DeFi protocol executes the attacker's requested operation as if Alice authorized it. [2](#0-1) [5](#0-4) [4](#0-3)
