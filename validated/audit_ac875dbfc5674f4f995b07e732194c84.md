### Title
Replacing ETH Connector Account Without State Migration Permanently Freezes All User ETH Balances - (File: engine/src/contract_methods/connector.rs)

### Summary

`set_eth_connector_contract_account` unconditionally overwrites the stored ETH connector NEAR account ID with no check for a pre-existing value and no migration of the NEP-141 token balances held by the old connector. After the swap, every existing EVM ETH balance in Aurora becomes permanently unwithdrawable because the `ExitToNear` and `ExitToEthereum` precompiles will direct withdrawal promises to the new connector, which holds no NEP-141 tokens.

### Finding Description

The ETH connector is the NEAR account that custodies the NEP-141 nETH tokens that back every EVM ETH balance inside Aurora. The engine stores a single pointer to this account under `EthConnectorStorageId::EthConnectorAccount`.

`set_eth_connector_contract_account` (owner-only) simply overwrites that pointer: [1](#0-0) 

The write helper is unconditional: [2](#0-1) 

Every withdrawal path reads this single pointer at execution time:

- **`ExitToNear` precompile** (base-ETH branch) calls `get_eth_connector_contract_account()` and issues an `ft_transfer` promise to whatever account is currently stored. [3](#0-2) [4](#0-3) 

- **`ExitToEthereum` precompile** (base-ETH branch) does the same, calling `withdraw` on the stored connector. [5](#0-4) [6](#0-5) 

When the connector is replaced, the new account holds zero NEP-141 tokens. Every `ft_transfer` / `withdraw` promise it receives will fail for insufficient balance. Meanwhile the EVM ETH is already burned (consumed by the precompile call) before the NEAR promise executes. Without the `error_refund` feature the callback is omitted entirely: [7](#0-6) [8](#0-7) 

So the ETH is burned in the EVM and the NEP-141 transfer silently fails → funds are permanently frozen.

Additionally, `ft_on_transfer` identifies base-ETH deposits by comparing `predecessor_account_id` to the stored connector ID: [9](#0-8) 

After the swap, any in-flight deposit from the old connector is misrouted as an ERC-20 transfer and will fail or credit the wrong token.

The `BijectionMap` used for the NEP-141↔ERC-20 mapping correctly guards against re-registration (`TokenAlreadyRegistered`), but no analogous guard exists for the connector pointer: [10](#0-9) 

### Impact Explanation

**Permanent freezing of funds (Critical).** All EVM ETH balances in Aurora are backed by NEP-141 tokens held in the old connector. After the connector pointer is overwritten, no user can successfully withdraw their ETH. The EVM burn is irreversible; the NEAR promise to the new (empty) connector fails silently when `error_refund` is disabled. The total value at risk equals the entire ETH TVL of the Aurora instance.

### Likelihood Explanation

**Low.** The call requires the engine owner. However, the function is explicitly designed to be called for legitimate connector upgrades (e.g., migrating to a new connector contract). An owner performing a routine upgrade without first draining the old connector or migrating balances will trigger this freeze inadvertently. The CHANGES.md records that the `SetEthConnectorContractAccountArgs` struct was already extended once (v3.2.0), confirming this function is actively used in production upgrades. [11](#0-10) 

### Recommendation

Add a guard in `set_eth_connector_contract_account` that reverts if a connector account is already stored, unless an explicit migration flag is provided and the old connector's NEP-141 balance has been fully transferred to the new one. At minimum, emit a clear error when the existing pointer is non-empty, mirroring the `TokenAlreadyRegistered` guard used for the NEP-141↔ERC-20 bijection map.

### Proof of Concept

1. Users deposit 1000 ETH into Aurora via the old connector (`old-connector.near`). Aurora's EVM state credits their addresses; `old-connector.near` holds 1000 nETH (NEP-141).
2. Owner calls `set_eth_connector_contract_account` with `account = "new-connector.near"`. The pointer at `EthConnectorStorageId::EthConnectorAccount` is overwritten. `new-connector.near` holds 0 nETH.
3. A user calls `withdrawToNear` on their `EvmErc20` (or sends ETH to the `ExitToNear` precompile address). The EVM burns their balance. The precompile reads the connector pointer → `new-connector.near`. A NEAR promise `ft_transfer { receiver_id: user, amount: X }` is dispatched to `new-connector.near`.
4. `new-connector.near` has 0 nETH → the promise fails. Without `error_refund`, no callback is scheduled. The user's EVM ETH is gone and no NEP-141 was received → **permanent fund loss**.
5. The 1000 nETH remain locked in `old-connector.near` with no mechanism to retrieve them through the Aurora Engine interface. [12](#0-11) [13](#0-12)

#[named]
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_eth_connector_contract_account<I: IO>(io: &I) -> Result<AccountId, ExitError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(ExitError::Other(Cow::Borrowed("ERR_KEY_NOT_FOUND")))
    .and_then(|x| {
        x.to_value()
            .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_DESERIALIZE")))
    })
}
