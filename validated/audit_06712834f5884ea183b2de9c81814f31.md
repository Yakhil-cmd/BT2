### Title
Unrestricted `ft_on_transfer` Allows Any NEAR Account to Mint Unbacked ERC-20 Tokens — (File: `engine/src/contract_methods/connector.rs`)

### Summary
The `ft_on_transfer` entrypoint on the Aurora Engine contract performs no verification that the caller is a legitimate NEP-141 token contract that actually transferred tokens. Any NEAR account can call it directly, and if an ERC-20 has been deployed for that account's ID, the engine will mint arbitrary amounts of ERC-20 tokens without any NEP-141 backing, causing bridge insolvency.

### Finding Description

The `ft_on_transfer` function in `engine/src/contract_methods/connector.rs` is the NEP-141 receiver callback. In the legitimate flow, a user calls `ft_transfer_call` on a NEP-141 contract, which transfers tokens to Aurora and then calls `ft_on_transfer` on Aurora as a callback. The engine is supposed to mint ERC-20 tokens only in exchange for real NEP-141 tokens received.

The function's only guard is `require_running`: [1](#0-0) 

The routing logic is:

```rust
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        ...
    )
};
``` [2](#0-1) 

There is no check that `predecessor_account_id` is a legitimate NEP-141 contract, nor any check that the `amount` in `args` corresponds to tokens actually received by Aurora. The engine blindly trusts the caller-supplied `amount`.

`receive_erc20_tokens` then looks up the ERC-20 mapped to the caller's account ID and mints `args.amount` tokens to the specified recipient: [3](#0-2) 

The `deploy_erc20_token` entrypoint is also unrestricted (only `require_running` is checked), so any NEAR account can register an ERC-20 for itself: [4](#0-3) 

The public WASM entrypoints confirm both functions are callable by any NEAR account: [5](#0-4) 

### Impact Explanation

**Critical — Bridge insolvency / direct theft of user funds.**

An attacker can:
1. Create a NEAR account (e.g., `fake_usdt.near`).
2. Call `deploy_erc20_token` on Aurora Engine to register an ERC-20 for `fake_usdt.near`.
3. Directly call `ft_on_transfer` on Aurora Engine from `fake_usdt.near`, supplying an arbitrary `amount` and their EVM address as `msg`.
4. Aurora Engine mints that arbitrary amount of ERC-20 tokens to the attacker's EVM address — with zero NEP-141 tokens ever deposited.
5. The attacker sells the unbacked ERC-20 tokens on Aurora DEXes, draining funds from buyers.
6. Any user who later tries to exit (burn ERC-20 → receive NEP-141) finds Aurora holds no NEP-141 backing, making the bridge permanently insolvent for that token.

### Likelihood Explanation

**High.** The attack requires only a NEAR account (free to create) and two contract calls. No special privileges, no leaked keys, no governance capture. The `ft_on_transfer` method is a standard public WASM export callable by any NEAR account at any time. The only prerequisite — deploying an ERC-20 for the attacker's account — is itself unrestricted.

### Recommendation

Add a caller-verification guard inside `ft_on_transfer` for the ERC-20 path. The engine must confirm that the `predecessor_account_id` is a known, registered NEP-141 token (i.e., one that already has an ERC-20 mapping **and** that the call arrived as a genuine `ft_transfer_call` callback). One approach is to maintain a set of registered NEP-141 accounts and reject `ft_on_transfer` calls from accounts not in that set. Alternatively, require that `ft_on_transfer` can only be reached as a NEAR callback (checking `promise_results_count() > 0`), which would prevent direct top-level calls.

// Step 2: Directly call ft_on_transfer from attacker.near
// predecessor_account_id = "attacker.near"
// Aurora looks up ERC-20 for "attacker.near" → found

    amount: Balance::new(1_000_000_000),
    msg: hex::encode(attacker_evm_address.as_bytes()),

// Called directly from attacker.near — no ft_transfer_call involved
aurora_engine.ft_on_transfer(args);

// Result: attacker holds 1_000_000_000 ERC-20 tokens backed by 0 NEP-141 tokens
// Bridge is insolvent; attacker sells tokens on Aurora DEXes

        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
