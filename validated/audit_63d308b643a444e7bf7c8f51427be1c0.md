### Title
Gateway Resource-Bounds Check Uses Wrong L2 Gas Price Field (`l2_gas_price` Instead of `next_l2_gas_price`) — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` checks an incoming transaction's `max_price_per_unit` against `block_info.gas_prices.strk_gas_prices.l2_gas_price` — the L2 gas price **of** the latest committed block N. The transaction will actually be executed in block N+1, whose gas price is `next_l2_gas_price` (a distinct EIP-1559-derived field stored in the same block header). The code itself carries a `TODO` acknowledging the wrong field is used. This mismatch causes the gateway to admit transactions that will fail blockifier execution (when the price is rising) and to reject transactions that are actually valid (when the price is falling).

---

### Finding Description

`BlockHeaderWithoutHash` carries two separate L2 gas price fields:

- `l2_gas_price`: the price **used** for transactions inside block N
- `next_l2_gas_price`: the EIP-1559-adjusted price **for** block N+1 [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` maps only `block_header.l2_gas_price` into the returned `BlockInfo`; `next_l2_gas_price` is never surfaced through the `GatewayFixedBlockStateReader` interface. [2](#0-1) 

`validate_resource_bounds` then reads that stale field and uses it as the threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;          // ← block N's price, not block N+1's price
``` [3](#0-2) 

The threshold is then applied as:

```
threshold = (min_gas_price_percentage / 100) * l2_gas_price   // wrong base
``` [4](#0-3) 

The blockifier's own `check_fee_bounds` (called during `perform_pre_validation_stage`) checks the transaction's `max_price_per_unit` against the **actual** block context gas price, which is `next_l2_gas_price`. [5](#0-4) 

The `run_validate_entry_point` path in the gateway also builds its `BlockContext` from the same `get_block_info()` call (with only `block_number` bumped), so the gas price used for blockifier validation at the gateway is also `l2_gas_price`, not `next_l2_gas_price`. [6](#0-5) 

---

### Impact Explanation

**Case 1 — Gas price rising (`next_l2_gas_price > l2_gas_price`):**
A transaction with `max_price_per_unit = X` where `l2_gas_price ≤ X < next_l2_gas_price` passes the gateway threshold (computed from the lower `l2_gas_price`) and is admitted to the mempool. When the batcher executes it in block N+1 at `next_l2_gas_price`, blockifier rejects it with `MaxGasPriceTooLow`. The gateway has admitted a transaction that cannot be included in any block at the current price.

**Case 2 — Gas price falling (`next_l2_gas_price < l2_gas_price`):**
A transaction with `max_price_per_unit = Y` where `next_l2_gas_price ≤ Y < l2_gas_price` is rejected by the gateway (Y is below the threshold computed from the higher `l2_gas_price`), even though it would be perfectly valid for block N+1. Valid transactions are incorrectly denied admission.

Both outcomes match the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The EIP-1559 mechanism can move `next_l2_gas_price` by up to ~0.3% per block relative to `l2_gas_price`, and the default `min_gas_price_percentage` is 100 (full threshold). During sustained high-load periods the divergence accumulates across many blocks, widening the window of incorrectly admitted or rejected transactions. [7](#0-6) 

---

### Likelihood Explanation

The bug is always active whenever `validate_resource_bounds = true` (the production default). Any block where the EIP-1559 algorithm adjusts the gas price — i.e., any block with non-zero L2 gas usage — produces a `next_l2_gas_price` that differs from `l2_gas_price`. The divergence is small per block but the admission error is systematic and reproducible without any special attacker capability; a normal user submitting a transaction priced at exactly `next_l2_gas_price` during a rising-price period will be incorrectly admitted and then fail execution.

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a new method) to also return `next_l2_gas_price` from the block header.
2. In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, read `block_header.next_l2_gas_price` and expose it.
3. Replace the `l2_gas_price` read in `validate_resource_bounds` with `next_l2_gas_price`.
4. Pass the same corrected price into `run_validate_entry_point`'s `BlockContext` so the gateway blockifier validation is consistent with what the batcher will enforce.

---

### Proof of Concept

```rust
// In a test analogous to the oracle PoC:
// 1. Commit block N with l2_gas_price = 100 and next_l2_gas_price = 110
//    (high gas usage caused EIP-1559 to raise the price).
// 2. Submit an invoke tx with max_price_per_unit = 105.
// 3. Gateway validate_resource_bounds: threshold = 100% * 100 = 100; 105 >= 100 → PASS.
// 4. Tx is admitted to mempool.
// 5. Batcher builds block N+1 with l2_gas_price = 110.
// 6. Blockifier check_fee_bounds: 105 < 110 → MaxGasPriceTooLow → FAIL.
//
// The gateway admitted a transaction that blockifier will always reject.
//
// Conversely, a tx with max_price_per_unit = 108 (>= next_l2_gas_price=110? No, 108 < 110)
// would also fail. But a tx with max_price_per_unit = 99 (< l2_gas_price=100) is correctly
// rejected. The window [100, 109] is the incorrectly-admitted zone.
```

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
