### Title
`skip_stateful_validations` admits invoke transactions with invalid signatures by checking any mempool presence instead of a deploy-account transaction — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to improve UX by allowing a user to submit a `deploy_account` + `invoke` (nonce=1) pair simultaneously. It skips the `__validate__` entry point (account signature verification) for an invoke with nonce=1 when the on-chain account nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. However, that check returns `true` for **any** transaction from the sender address in the mempool — not specifically a `deploy_account` transaction. An external attacker who observes that an account has any pending transaction in the mempool can submit a nonce=1 invoke with an **invalid signature** that bypasses `__validate__` at the gateway and is admitted to the mempool.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` contains the following logic:

```rust
// check if the transaction nonce is 1, meaning it is post deploy_account, and the
// account nonce is zero, meaning the account was not deployed yet.
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    // We verify that a deploy_account transaction exists for this account. It is sufficient
    // to check if the account exists in the mempool since it means that either it has a
    // deploy_account transaction or transactions with future nonces that passed
    // validations.
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is skipped entirely — the blockifier's `__validate__` entry point is never called for that invoke:

```rust

Ok(skip_validate)
``` [2](#0-1) 

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The predicate `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

`tx_pool.contains_account` returns `true` if **any** transaction from that address is in the pool — a `deploy_account`, a future-nonce invoke, or any other type. The comment's reasoning ("it means that either it has a deploy_account transaction or transactions with future nonces that passed validations") is the broken invariant: a future-nonce invoke (e.g., nonce=2) that passed `__validate__` for an already-deployed account also satisfies the predicate, yet it provides no guarantee that a nonce=1 invoke from the same sender is authentic.

**Concrete broken invariant:** `account_tx_in_pool_or_recent_block` returning `true` does **not** imply that a `deploy_account` transaction for this address is pending. It only implies that *some* transaction from this address is known to the mempool.

**Attack path (external attacker, no private key required for the target account):**

1. Account A has a `deploy_account` tx (nonce=0) in the mempool. The account does not yet exist on-chain (state nonce = 0).
2. Attacker submits an invoke for account A with nonce=1 and a **crafted/invalid signature** (e.g., all-zero signature bytes, or a signature for a different payload).
3. Gateway stateless validator passes (signature length/format is valid).
4. `validate_nonce`: nonce=1 ≥ account_nonce=0 and ≤ 0+200 → passes.
5. `validate_by_mempool`: mempool accepts the nonce=1 invoke (no existing nonce=1 tx yet, or attacker uses a higher tip for fee escalation).
6. `skip_stateful_validations`: `account_tx_in_pool_or_recent_block` returns `true` (the `deploy_account` tx is in the pool) → returns `true`.
7. `run_validate_entry_point` is **skipped** — `__validate__` is never called.
8. The invalid invoke is forwarded to the mempool via `add_tx`. [5](#0-4) 

If the account owner's legitimate nonce=1 invoke is already in the mempool, the attacker can replace it by submitting with a higher tip (fee escalation is permitted by `validate_fee_escalation`). The attacker's invalid invoke then occupies the nonce=1 slot.

When the batcher later executes the block, it creates its own `AccountTransaction` with `validate: true` (default `ExecutionFlags`):

```rust
impl Default for ExecutionFlags {
    fn default() -> Self {
        Self { only_query: false, charge_fee: true, validate: true, strict_nonce_check: true }
    }
}
``` [6](#0-5) 

The batcher calls `__validate__`, which fails for the invalid signature → the transaction is **rejected** (not reverted; no fee charged). The legitimate nonce=1 invoke has been evicted from the mempool and must be resubmitted. The attacker can repeat this indefinitely, permanently griefing the account's first post-deployment invoke.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An unauthenticated invoke transaction (invalid account signature) is admitted to the mempool without the `__validate__` entry point being called. This enables:

1. **Griefing via fee escalation:** An attacker can continuously replace a victim's legitimate nonce=1 invoke with an invalid-signature invoke at a higher tip, preventing the victim's first post-deployment transaction from ever being sequenced.
2. **Mempool pollution:** Invalid transactions occupy mempool slots and consume batcher execution resources (the batcher must attempt execution before discovering the signature is invalid).
3. **No credentials required:** The attacker only needs to observe that a `deploy_account` tx is pending for the target address — information that is publicly visible in the mempool.

---

### Likelihood Explanation

- `deploy_account` transactions are publicly observable in the mempool.
- The attack requires only a single RPC call (`add_invoke_transaction`) with a crafted payload.
- The nonce=1 + account_nonce=0 condition is the normal state for every new account during the deploy_account + invoke UX window.
- No privileged access, no private key for the target account, and no special network position is required.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects transaction types in the pool. Alternatively, the gateway can inspect the `InternalRpcTransaction` type of the pending transaction before granting the skip.

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await

// Use a type-aware check:
mempool_client.has_pending_deploy_account(tx.sender_address()).await
```

This ensures the skip is only granted when the specific precondition (a pending `deploy_account`) that justifies bypassing `__validate__` is actually satisfied.

### Proof of Concept

```
Precondition:
  - Account 0xABCD has a deploy_account tx (nonce=0, valid) in the mempool.
  - On-chain nonce for 0xABCD = 0 (not yet deployed).

Step 1 — Attacker submits:
  starknet_addInvokeTransaction({
    sender_address: 0xABCD,
    nonce: 1,
    calldata: [...],
    signature: [0x0, 0x0],   // invalid / all-zero signature
    resource_bounds: { l2_gas: { max_amount: ..., max_price_per_unit: <high tip> } }
  })

Step 2 — Gateway stateless validation: passes (signature array length is valid).

Step 3 — validate_nonce: 0 <= 1 <= 200 → passes.

Step 4 — validate_by_mempool: no existing nonce=1 tx → passes.

Step 5 — skip_stateful_validations:
  account_tx_in_pool_or_recent_block(0xABCD)
    → tx_pool.contains_account(0xABCD) == true  (deploy_account is in pool)
    → returns true
  → __validate__ is SKIPPED.

Step 6 — add_tx: attacker's invalid invoke is added to the mempool.

Step 7 — Victim submits legitimate nonce=1 invoke with valid signature, lower tip
         → rejected as DuplicateNonce (or requires higher tip to replace attacker's tx).

Step 8 — Batcher executes: __validate__(invalid_sig) → FAIL → tx rejected.
         Attacker repeats from Step 1 with a higher tip.
         Victim's first post-deployment invoke is permanently suppressed.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
