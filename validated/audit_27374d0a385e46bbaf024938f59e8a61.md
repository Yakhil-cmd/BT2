### Title
Stale "recent-block" presence bypasses `__validate__` for invoke transactions after a deploy-account revert — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the blockifier `__validate__` entry-point (signature check) for an invoke with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true`. The "or recent block" branch consults historical sequencer state. If the account's `deploy_account` transaction was included in a block that was subsequently **reverted**, the account's on-chain nonce returns to 0 (not deployed), yet the mempool's recent-block record still exists. An attacker can therefore submit an invoke with nonce=1 for that address and have it admitted to the mempool with no signature verification — the exact structural analog of the validator-snapshot bug: authorization revoked in current state, but historical presence still grants power.

When the result is `true`, the caller sets:

```rust
let skip_validate = skip_stateful_validations(...).await?;
// ...
let execution_flags = ExecutionFlags {
    validate: !skip_validate,   // false → __validate__ is NOT called
    ...
};
``` [2](#0-1) 

The two conditions that must hold simultaneously are:

1. **`account_nonce == Nonce(Felt::ZERO)`** — the account is not yet deployed on-chain (current state).
2. **`account_tx_in_pool_or_recent_block` returns `true`** — the account appeared in a recent block (historical state).

These two conditions are normally mutually exclusive: if the deploy was included in a block, the nonce advances to ≥1. They become simultaneously satisfiable only after a **block revert**: the deploy_account is rolled back (nonce returns to 0), but the mempool's "recent block" record is not cleared. The code comment itself acknowledges the design intent is only the mempool branch:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [3](#0-2) 

The "or recent block" half of the check is the stale-snapshot surface. The gateway's blockifier validation runs with `validate: false`, so the account's `__validate__` entry point — which performs signature verification — is never called for this transaction at admission time.

The gateway then calls `mempool_client.add_tx(...)` with the unvalidated transaction: [4](#0-3) 

The batcher later re-executes with `validate: true` (via `new_for_sequencing`): [5](#0-4) 

So the transaction will ultimately revert during block building, but it was **admitted to the mempool without any signature check**, consuming a mempool slot and passing the gateway's admission gate.

---

### Impact Explanation

An attacker who knows any address that appeared in a recently-reverted block can craft an invoke transaction with nonce=1 and an arbitrary or empty signature. The gateway will skip `__validate__`, accept the transaction as valid, and forward it to the mempool. This matches the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The gateway is the trust boundary for admission. Bypassing signature verification at this layer means the mempool can be flooded with cryptographically invalid transactions for any address whose deploy was reverted, with no cost beyond the RPC call.

---

### Likelihood Explanation

Block reverts are a supported, tested production flow (see `integration_test_revert_flow.rs`). The window between a revert and the mempool's "recent block" record expiring is the attack surface. An attacker monitoring the chain for reverted deploy_account transactions can immediately exploit the window. No privileged access is required — only a standard `add_transaction` RPC call.

---

### Recommendation

Remove the `_or_recent_block` branch from the skip condition, or re-validate the recent-block record against current on-chain state before skipping `__validate__`. The comment already states the mempool-only check is sufficient for the intended UX case:

```rust
// Before:
return mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await ...;

// After (mempool-only, consistent with the stated design intent):
return mempool_client.account_tx_in_pool(tx.sender_address()).await ...;
```

If the "recent block" branch must be retained for performance, it should be gated on a check that the block in question has not been reverted (i.e., the account's current nonce is still ≥1 in the canonical chain).

---

### Proof of Concept

1. Alice submits a `deploy_account` transaction; it is included in block N.
2. Block N is reverted (e.g., due to a consensus revert). Alice's on-chain nonce returns to 0.
3. The mempool's `account_tx_in_pool_or_recent_block` record for Alice's address still reflects block N.
4. Attacker Bob submits an `Invoke V3` for Alice's address with `nonce=1` and a garbage signature.
5. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == Nonce(ONE)` ✓
   - `account_nonce == Nonce(ZERO)` ✓ (revert restored nonce to 0)
   - `account_tx_in_pool_or_recent_block(Alice)` → `true` ✓ (stale record from block N)
6. `skip_validate = true` → `ExecutionFlags { validate: false, ... }` → `__validate__` is **not called**.
7. The transaction passes gateway stateful validation and is forwarded to the mempool with no signature check.
8. The mempool slot is consumed; the transaction will only be rejected later during batcher execution, after the admission gate has been bypassed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
