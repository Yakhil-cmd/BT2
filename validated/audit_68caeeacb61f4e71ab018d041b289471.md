### Title
Unbounded `stakers` Vec Growth Causes Unbounded Gas Consumption in `get_stakers` - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage `Vec` in the `Staking` contract grows monotonically as new stakers join. Exited stakers are never removed from this Vec. The `get_stakers` function iterates over the **entire** Vec on every call using `into_iter_full_range()`. Any unprivileged user can register as a staker (with minimum stake), exit after the wait window to recover funds, and leave a permanent entry in the Vec. Repeating this across many addresses inflates the Vec without bound, eventually making `get_stakers` exceed the Starknet transaction gas limit.

### Finding Description
`get_stakers` in `src/staking/staking.cairo` iterates unconditionally over every element ever pushed into `self.stakers`:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;   // exited stakers are skipped but still iterated
    }
    ...
}
``` [1](#0-0) 

Starknet's `Vec` storage type supports only `push`; there is no `remove` or compaction primitive. When a staker calls `unstake_action` and is fully removed from the protocol, their address is **not** deleted from `self.stakers`. The Vec length is therefore strictly non-decreasing over the lifetime of the contract. [2](#0-1) 

The `pool_member_epoch_balance` trace in the Pool contract has an analogous acknowledged comment:

> "This loop is unbounded but unlikely to exceed gas limits." [3](#0-2) 

That comment reveals the developers are aware of unbounded-iteration risk in this codebase, but the `stakers` Vec is the more severe instance because it is inflatable by any external actor, not just the member themselves.

### Impact Explanation
`get_stakers` is the consensus-layer query used to determine per-epoch staking power for all validators. If the transaction gas limit is exceeded, the call reverts. This prevents the consensus layer from obtaining staking-power data, which can block reward distribution and attestation processing — constituting **unbounded gas consumption** and potential **temporary freezing of unclaimed yield**.

Allowed impact matched: *Unbounded gas consumption (Medium)* and *Temporary freezing of unclaimed yield (High)*.

### Likelihood Explanation
- Minimum stake is the only capital barrier; funds are fully recovered after the exit wait window, so the net cost to the attacker is only gas fees.
- The attack is permissionless: any address can call `stake()` followed by `unstake_intent()` / `unstake_action()`.
- The `stakers` Vec is already expected to grow organically; a targeted griefing campaign accelerates it to the point of DoS.
- Starknet's per-transaction gas cap is finite and well-documented; a sufficiently large Vec will reliably breach it.

### Recommendation
1. **Lazy deletion / tombstone flag**: When a staker fully exits (`unstake_action`), mark their slot as deleted (e.g., write `ContractAddress::zero()`) and skip zero entries in `get_stakers`. This avoids the need for Vec compaction.
2. **Active-staker index**: Maintain a separate, compacted `active_stakers` Vec that entries are removed from on exit, and iterate only over that Vec in `get_stakers`.
3. **Pagination**: Add a paginated variant of `get_stakers` (offset + limit) so callers can chunk the iteration across multiple transactions if the set grows large.

### Proof of Concept
1. Deploy the staking contract with `min_stake = M`.
2. Create N addresses, each funded with `M` STRK.
3. For each address: call `stake(amount: M, ...)`. Each call appends to `self.stakers`.
4. After K epochs, for each address: call `unstake_intent()`, wait `exit_wait_window`, call `unstake_action()`. Funds are returned; Vec entries remain.
5. Call `get_stakers(epoch_id: current_epoch)`. The function iterates all N dead entries plus any live stakers. For sufficiently large N, the transaction reverts with an out-of-gas error, denying the consensus layer staking-power data and blocking reward distribution. [4](#0-3) [5](#0-4)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
