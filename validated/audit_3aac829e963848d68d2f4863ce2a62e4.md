### Title
Missing STRK Token Address Validation for `reward_address` in BTC Pool — (`File: src/pool/pool.cairo`)

### Summary
The `Pool` contract's `enter_delegation_pool` and `change_reward_address` functions validate that `reward_address` is not equal to the pool's own staking token, but do not validate that `reward_address` is not the STRK token contract address. Since all pool rewards are always disbursed in STRK, a BTC pool member who sets `reward_address = STRK_TOKEN_ADDRESS` will have their accrued STRK rewards transferred into the STRK token contract itself, where they are permanently unrecoverable.

### Finding Description

In `pool.cairo`, `enter_delegation_pool` performs the following guard:

```cairo
let token_address = token_dispatcher.contract_address;   // BTC token for a BTC pool
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [1](#0-0) 

`token_address` here is the pool's staking token (e.g., a BTC ERC-20). The check therefore only rejects `reward_address == BTC_token`. It does **not** reject `reward_address == STRK_TOKEN_ADDRESS`.

The same incomplete guard appears in `change_reward_address`:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

When `claim_rewards` is later called, rewards are always paid in STRK regardless of the pool's staking token:

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [3](#0-2) 

If `reward_address` was set to `STRK_TOKEN_ADDRESS`, the transfer succeeds (the STRK ERC-20 contract accepts any incoming transfer), but the tokens are irrecoverably locked inside the token contract.

By contrast, the `Staking` contract's `stake` function uses the broader `does_token_exist` check, which covers all registered tokens including STRK:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [4](#0-3) 

The pool contract lacks this equivalent protection.

### Impact Explanation
A BTC pool member whose `reward_address` is `STRK_TOKEN_ADDRESS` will have every STRK reward payment sent into the STRK token contract. Those tokens cannot be retrieved by any protocol function. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

### Likelihood Explanation
The scenario requires a BTC pool member to supply `STRK_TOKEN_ADDRESS` as their `reward_address`. This can happen by mistake (a user confusing the reward token address with a personal wallet), or through a front-end bug. The staking contract already treats this as a dangerous condition and blocks it; the pool contract's weaker check creates an inconsistency that a careless user can fall into. Likelihood is **Low-to-Medium** given BTC pools are a newer feature and users may not be aware that rewards are always in STRK.

### Recommendation
Mirror the staking contract's validation in both `enter_delegation_pool` and `change_reward_address`. The simplest fix is to add an explicit check:

```cairo
assert!(
    reward_address != STRK_TOKEN_ADDRESS,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Alternatively, expose a helper (analogous to `does_token_exist`) that the pool can call to check all registered token addresses, keeping the validation consistent across contracts.

### Proof of Concept

1. Token admin adds and enables a BTC ERC-20 token in the staking contract.
2. A staker opens a BTC delegation pool via `set_open_for_delegation`.
3. An unprivileged delegator calls:
   ```
   pool.enter_delegation_pool(
       reward_address: STRK_TOKEN_ADDRESS,   // passes the BTC-only check
       amount: X
   )
   ```
   The call succeeds because `BTC_token_address != STRK_TOKEN_ADDRESS`.
4. After one or more epochs with attestations, the delegator (or anyone) calls `pool.claim_rewards(pool_member)`.
5. `claim_rewards` executes `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
6. The STRK rewards land inside the STRK token contract with no recovery path — permanently frozen.

The same outcome is reachable via `change_reward_address` for an existing BTC pool member.
