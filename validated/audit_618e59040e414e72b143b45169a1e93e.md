### Title
Equivocating Send+Echo by a Single Byzantine Participant Triggers Honest-Party Early-Abort in `reliable_broadcast_receive_all`, Permanently Aborting DKG/Reshare/Refresh — (`src/protocol/echo_broadcast.rs`)

---

### Summary

A single malicious participant M within the documented Byzantine threshold (n ≥ 3f+1) can permanently abort DKG, reshare, or refresh for at least one honest party by sending equivocating `Send` messages and then echoing the minority value. The early-abort guard in `reliable_broadcast_receive_all` terminates the honest party's execution rather than merely skipping the malicious session, violating the Byzantine fault-tolerance guarantee the library explicitly claims.

---

### Finding Description

**Threshold arithmetic for n=4:** [1](#0-0) 

```
broadcast_threshold f = (4-1)/3 = 1
echo_t = usize::midpoint(4, 1) = (4+1)/2 = 2   (integer division)
```

To advance to the Ready phase a value needs a count **strictly greater than** `echo_t` (i.e., ≥ 3).

**Attack (n=4: P0, P1, P2 honest; M malicious):**

M sends `Send(true)` to P0 and P1, and `Send(false)` to P2. M then calls `reliable_broadcast_receive_all` with `vote_false`, so it simulates `Echo(false)` for its own session.

Honest parties echo whatever `Send` they received:
- P0 → `Echo(true)` to all
- P1 → `Echo(true)` to all
- P2 → `Echo(false)` to all

From **P2's** perspective for M's session, messages arrive in order:

| Step | Source | Value | Counts |
|------|--------|-------|--------|
| 1 | P2 (self-simulated) | Echo(false) | false=1 |
| 2 | P0 | Echo(true) | false=1, true=1 |
| 3 | P1 | Echo(true) | false=1, true=2 |
| 4 | M | Echo(false) | **false=2, true=2** |

After step 4, the early-abort check fires: [2](#0-1) 

```
received_echo_cnt     = 4
non_received_echo_cnt = 4 - 4 = 0

For false: 2 + 0 = 2  → NOT > echo_t (2)
For true:  2 + 0 = 2  → NOT > echo_t (2)
is_enough = false  →  return Err(AssertionFailed("...malicious..."))
```

**P2, an honest party, permanently aborts.** Because `do_broadcast` propagates this error directly: [3](#0-2) 

and `do_keyshare` calls `do_broadcast` at the very first round: [4](#0-3) 

the entire DKG execution for P2 terminates with an unrecoverable error.

**The codebase's own test confirms this exact failure path:** [5](#0-4) 

The test comment at line 549 states "threshold is assumed to be n >= 3*threshold + 1", acknowledging the Byzantine fault-tolerance claim, yet `version_1` (M echoes the minority value) is asserted to produce the `AssertionFailed` error — meaning an honest party aborts under a single Byzantine fault at the minimum tolerated n.

---

### Impact Explanation

- **Scope**: DKG (`do_keygen`), reshare (`do_reshare`), and refresh all route through `do_keyshare` → `do_broadcast` → `reliable_broadcast_receive_all`.
- **Effect**: At least one honest party permanently aborts its protocol execution. No key shares are produced for that party. The abort is non-recoverable without restarting the entire ceremony.
- **Threshold violation**: The library explicitly documents n ≥ 3f+1 as the Byzantine fault-tolerance bound. With n=4 and f=1 (exactly the minimum), a single malicious participant can abort an honest party, violating that invariant.

---

### Likelihood Explanation

- Requires only one malicious participant who can send private messages to individual peers — a standard capability in any threshold protocol.
- The attack is deterministic and requires no cryptographic breaks.
- The exact attack is already encoded in the test suite (`do_broadcast_dishonest_consume_version_1`), making it trivially reproducible.
- Any deployment with n=4 (or more generally any n where `(n-1)/3 ≥ 1`) is affected.

---

### Recommendation

The early-abort check should **not** call `return Err(...)` on the honest party's execution. Instead, it should mark the sender's session as permanently failed (e.g., set `finish_send`, `finish_echo`, `finish_ready` to `true` for that `sid` and record the sender as malicious) and continue the outer loop. The honest party's own session and all other sessions remain unaffected. The malicious sender's slot in `vote_output` is simply left empty or flagged, and the caller (DKG) can decide whether to abort based on whether the faulty sender was within the tolerated Byzantine set.

---

### Proof of Concept

The existing test `test_three_honest_one_dihonest` is the proof of concept: [6](#0-5) 

`version_1` (M echoes the minority value, splitting votes 2-2) returns:
```
Err(AssertionFailed("The original sender in session 3 is malicious! Could not collect enough echo votes to meet the threshold"))
```
This error originates inside an **honest party's** execution of `reliable_broadcast_receive_all`, not M's. `version_2` (M echoes the majority value) succeeds, confirming the split-vote scenario is the sole trigger. Applying this to `do_keyshare` by having M send different `my_session_id` bytes to different peers produces the identical abort in the first `do_broadcast` call at `src/dkg.rs:362`.
