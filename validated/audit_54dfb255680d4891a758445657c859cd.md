Audit Report

## Title
Markdown Injection via Unsanitized UTF-8 Memo in ICRC-21 `GenericDisplayMessage` — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary

`add_memo` in `responses.rs` decodes the raw ICRC-1 memo bytes as UTF-8 and places the result verbatim inside a single-backtick inline code span in a `GenericDisplayMessage`. A backtick character in the memo terminates the code span, allowing everything that follows to be interpreted as raw Markdown by any wallet that renders `GenericDisplayMessage`. Because the memo is supplied by the transaction initiator (the dapp, not the ledger developer), this is reachable by an unprivileged caller and directly undermines the security guarantee of ICRC-21. The `token_name`/`token_symbol` injection vectors (Vector 1) are not independently meaningful because a canister developer who controls the ledger canister can already return arbitrary consent messages from `icrc21_canister_call_consent_message` without exploiting the shared library.

## Finding Description

**Root cause — `add_memo` (Vector 2):**

`add_memo` at [1](#0-0)  decodes the memo bytes as UTF-8 and interpolates the result directly into a single-backtick code span:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

A backtick anywhere in `memo_str` closes the code span at that point. All subsequent bytes are rendered as raw Markdown by any ICRC-21-compliant wallet that renders `GenericDisplayMessage`. There is no escaping, stripping, or fenced-block fallback.

**Exploit flow:**

1. Malicious dapp constructs a `TransferArg` whose `memo` field is the UTF-8 byte sequence `` ` `` + `\n\n# You are sending 1000 ICP to attacker\n\n**To:**\n` `` attacker_address ``.
2. Wallet calls `icrc21_canister_call_consent_message` with those args (standard ICRC-21 flow). [2](#0-1) 
3. `build_icrc21_consent_info` decodes the `TransferArg`, passes the memo to `add_memo`. [3](#0-2) 
4. `add_memo` emits: `\n\n**Memo:**\n`` ` ``\n\n# You are sending 1000 ICP to attacker\n\n**To:**\n`` ` ``attacker_address`
5. Wallet renders the `GenericDisplayMessage` as Markdown. The injected heading and fake "To" field appear prominently; the real fields appended by `add_amount`/`add_account` appear below and may be scrolled off-screen.
6. User approves based on the spoofed display; the actual transaction executes with the real (different) parameters.

**Why existing checks are insufficient:**

The only guard in `build_icrc21_consent_info` is a size check on the raw arg bytes ( [4](#0-3) ). There is no validation or sanitization of the decoded memo string before it is interpolated into Markdown. The ckBTC and ckDOGE minters explicitly call out this exact risk in comments and validate their user-supplied string (the destination address) before interpolation. [5](#0-4) [6](#0-5)  The shared ICRC-1/ICRC-2 library applies no equivalent guard.

## Impact Explanation

ICRC-21 is the mechanism by which wallets show users a human-readable, trustworthy summary of what they are about to sign. A malicious dapp can craft a memo that causes the consent dialog to display a completely fabricated set of fields (fake recipient, fake amount, fake heading) while the real transaction parameters appear below the fold. A user who approves based on the displayed dialog may authorize a transaction with materially different parameters than shown — resulting in unintended fund transfers. This constitutes a significant ledger security impact with concrete user harm, matching the High impact class: "Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm." However, because the real fields are also present in the message (below the injected content) and exploitation requires a malicious dapp plus user approval, the practical severity is constrained to **Medium ($200–$2,000)**: a difficult attack with meaningful security impact requiring strict target conditions (wallet renders GenericDisplay as Markdown, user does not read past the injected content).

## Likelihood Explanation

The memo is an unprivileged, user-supplied field on every `icrc1_transfer`, `icrc2_approve`, and `icrc2_transfer_from` call. Any dapp that constructs transactions on behalf of users (the standard DeFi pattern) can set the memo without the user's knowledge. No special privileges, canister deployment, or key material are required. The attack is repeatable and silent. Any wallet that implements ICRC-21 and renders `GenericDisplayMessage` as Markdown is affected.

## Recommendation

1. **Escape backticks in `memo_str`** before placing it inside a single-backtick code span: replace every `` ` `` with `` \` ``, or switch to a triple-backtick fenced code block (` ``` `…` ``` `), which is immune to single-backtick injection.
2. **Apply the same pattern used in ckBTC/ckDOGE**: validate or encode any user-supplied string before interpolating it into a `GenericDisplayMessage`, with an explicit comment noting the Markdown-injection risk.
3. **Prefer `FieldsDisplayMessage`** for structured data: the `FieldsDisplay` variant carries typed `Value` entries that wallets render without Markdown interpretation, eliminating the injection surface entirely. [7](#0-6) 

## Proof of Concept

```rust
// Craft a memo that breaks out of the backtick code span
let malicious_memo = b"`\n\n# You are sending 1000 ICP to attacker\n\n**To:**\n`attacker_address".to_vec();

let transfer_args = TransferArg {
    memo: Some(Memo::from(malicious_memo)),
    amount: Nat::from(1_u64),   // actual amount: 1 e8s
    to: victim_account,
    ..Default::default()
};

// Encode transfer_args as Candid and pass to icrc21_canister_call_consent_message.
// add_memo produces:
//   \n\n**Memo:**\n`\n\n# You are sending 1000 ICP to attacker\n\n**To:**\n`attacker_address`
// A Markdown-rendering wallet displays a fake heading and fake "To" field
// above the real fields.
```

A deterministic integration test can be written using PocketIC: deploy the NNS ICP ledger (or any ledger using `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints`), call `icrc21_canister_call_consent_message` with the crafted `TransferArg`, decode the returned `GenericDisplayMessage`, and assert that the string contains the injected heading `# You are sending 1000 ICP to attacker` — confirming the injection without any mainnet interaction.

```rust
    pub fn add_memo(&mut self, memo: GenericMemo) {
        match memo {
            GenericMemo::Icrc1Memo(memo) => {
                // Check if the memo is a valid UTF-8 string and display it as such if it is.
                let memo_str = match std::str::from_utf8(memo.as_slice()) {

                    Err(_) => hex::encode(memo.as_slice()),

                        .fields
```

```rust
/// Verifies that `address` parses as a valid Bitcoin address on the configured
/// network before it gets interpolated into a consent message. This both
/// guarantees the user is shown a meaningful (parseable) destination and rules
/// out Markdown-injection vectors in the GenericDisplay output (e.g. an
/// "address" that contains newlines or backticks crafted to fake additional
/// fields). Uses the same parser as `retrieve_btc_with_approval`, so any
/// address the consent endpoint accepts is also accepted by the actual call.
fn validate_address(address: &str, network: Network) -> Result<(), Icrc21Error> {
    BitcoinAddress::parse(address, network).map_err(|e| {
        Icrc21Error::UnsupportedCanisterCall(ErrorInfo {
            description: format!("Invalid Bitcoin destination address: {e}"),
        })
    })?;
    Ok(())
}
```
