### Title
`DIFFICULTY`/`PREVRANDAO` Opcode Returns Hardcoded Constant `1` in Production Builds - (`zk_ee/src/system/mod.rs`)

---

### Summary

In production builds of ZKsync OS, the EVM `DIFFICULTY` opcode (opcode `0x44`, which post-Merge serves as `PREVRANDAO`) always returns the hardcoded constant `1` instead of the actual consensus-layer randomness beacon. This is caused by the `prevrandao` Cargo feature being absent from all production feature sets. Any EVM contract that uses `block.prevrandao` or `block.difficulty` as a source of randomness is trivially exploitable because the value is a known constant.

---

### Finding Description

`get_mix_hash()` in `zk_ee/src/system/mod.rs` is gated on the `prevrandao` Cargo feature. When that feature is absent, it unconditionally returns `U256::ONE`:

```rust
pub fn get_mix_hash(&self) -> Result<Bytes32, InternalError> {
    #[cfg(feature = "prevrandao")]
    {
        self.metadata
            .block_randomness()
            .ok_or(internal_error!("block randomness should be provided"))
    }

    #[cfg(not(feature = "prevrandao"))]
    {
        Ok(Bytes32::from_array(
            ruint::aliases::U256::ONE.to_be_bytes::<32>(),
        ))
    }
}
``` [1](#0-0) 

The EVM `DIFFICULTY` instruction handler calls `get_mix_hash()` directly:

```rust
pub fn difficulty(&mut self, system: &mut System<S>) -> InstructionResult {
    // Mix hash is the source of randomness, currently holding
    // the value of prevRandao.
    let value = U256::from_be_bytes(system.get_mix_hash()?.as_u8_array());
    self.stack.push(&value)?;
    Ok(())
}
``` [2](#0-1) 

The `prevrandao` feature is **not** included in either production feature set:

- `forward_system`: `production = ["basic_bootloader/eip-7702", "system_hooks/p256_precompile"]`
- `proof_running_system`: `production = ["basic_bootloader/eip-7702", "system_hooks/p256_precompile"]` [3](#0-2) [4](#0-3) 

The feature is only enabled for non-production modes (`eth_runner`, `evm_tester`): [5](#0-4) 

The `prevrandao` feature definition in `zk_ee/Cargo.toml` confirms it is not part of `default`: [6](#0-5) 

---

### Impact Explanation

Any EVM contract deployed on ZKsync OS that reads `block.prevrandao` (or `block.difficulty`) as a randomness source will always receive the value `1`. This completely breaks the randomness assumption for:

- On-chain lotteries and prize draws
- Random NFT trait assignment
- Commit-reveal schemes seeded by `prevrandao`
- Any protocol that uses `block.prevrandao % N` for selection

An attacker who knows this constant (trivially discoverable from the public source code or by calling the opcode once) can deterministically predict all "random" outcomes and exploit any such contract to drain funds, claim all prizes, or manipulate selection results. The impact is direct theft of user funds from any protocol relying on this opcode.

---

### Likelihood Explanation

Likelihood is high. The `PREVRANDAO`/`DIFFICULTY` opcode is a standard and widely-used source of on-chain randomness. Many contracts deployed on EVM-compatible chains use it. The constant value `1` is publicly known from the open-source codebase. No special privileges, oracle manipulation, or brute force are required — the attacker simply reads the source code and exploits any target contract accordingly. The entry path is a standard unprivileged EVM transaction.

---

### Recommendation

Enable the `prevrandao` feature in the production feature sets of both `forward_system` and `proof_running_system`:

```toml
# forward_system/Cargo.toml
production = ["basic_bootloader/eip-7702", "system_hooks/p256_precompile", "zk_ee/prevrandao"]

# proof_running_system/Cargo.toml
production = ["basic_bootloader/eip-7702", "system_hooks/p256_precompile", "zk_ee/prevrandao"]
```

Additionally, ensure the oracle always supplies a valid `mix_hash` (prevRandao value) in the `BlockMetadataFromOracle` for every production block, and add an assertion or compile-time enforcement that `block_randomness()` is never `None` in production execution paths. [7](#0-6) 

---

### Proof of Concept

1. Deploy a simple lottery contract on ZKsync OS production:
   ```solidity
   contract Lottery {
       function pickWinner(address[] calldata players) external view returns (address) {
           uint256 rand = uint256(block.prevrandao); // always == 1 in production
           return players[rand % players.length];
       }
   }
   ```
2. Call `pickWinner` with any list of players. Because `block.prevrandao` always returns `1`, the winner is always `players[1 % players.length]`.
3. An attacker who knows this places themselves at index `1 % N` in any player list and wins every time, draining the lottery contract.

The root cause is confirmed at:
- `zk_ee/src/system/mod.rs` lines 119–124: constant `U256::ONE` returned when `prevrandao` feature is absent
- `evm_interpreter/src/instructions/environment.rs` lines 36–44: `DIFFICULTY` opcode directly uses `get_mix_hash()`
- `forward_system/Cargo.toml` line 49 and `proof_running_system/Cargo.toml` line 46: `prevrandao` absent from `production` feature [8](#0-7) [2](#0-1) [3](#0-2) [4](#0-3)
