### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Overly Broad Mempool Presence Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator is designed to allow a UX shortcut: when a user sends a `deploy_account` + `invoke` pair simultaneously, the gateway skips `__validate__` for the invoke because the account does not yet exist on-chain. The guard for this skip is `account_tx_in_pool_or_recent_block`, which returns `true` if **any** transaction from the sender's address is in the mempool — not specifically a `deploy_account`. An attacker who observes a victim's `deploy_account` in the mempool can immediately send an invoke with the victim's address, nonce=1, and an arbitrary/fake signature. The gateway will skip `__validate__` (the only place the signature is cryptographically verified at admission time) and admit the forged transaction into the mempool.

---

### Finding Description

The vulnerable function is `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← too broad
                .await
                ...
        }
    }
    Ok(false)
}
```

The three conditions that must all be true to skip `__validate__`:

| Condition | Attacker control |
|---|---|
| `tx.nonce() == 1` | Attacker sets nonce=1 freely |
| `account_nonce == 0` | True for any not-yet-deployed account |
| `account_tx_in_pool_or_recent_block(sender)` | True as soon as victim's `deploy_account` lands in mempool |

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

It checks for **any** transaction from the address — it does not distinguish `deploy_account` from `invoke`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is circular: the attacker's own forged invoke is the transaction being admitted, and it has not yet passed signature validation.

The result of `skip_stateful_validations` returning `true` is that `run_validate_entry_point` sets `validate: false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate: false`, `StatefulValidator::perform_validations` returns immediately without calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:79-81
if !tx.execution_flags.validate {
    return Ok(());
}
```

The mempool's `validate_tx` (called before `skip_stateful_validations`) only checks nonce ordering and duplicate hashes — it never inspects the signature:

```rust
// crates/apollo_mempool/src/mempool.rs:402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
```

The gateway's `validate_nonce` check also passes for nonce=1 with account_nonce=0 as long as `max_allowed_nonce_gap >= 1` (the default is non-zero):

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:287-296
_ => {
    let max_allowed_nonce =
        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
    if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
        return Err(...);
    }
}
```

The forged invoke is therefore admitted to the mempool with no signature check at any point in the gateway path.

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid (unsigned) transaction before sequencing.**

The gateway admits a forged invoke transaction for any account whose `deploy_account` is currently pending in the mempool. The forged transaction occupies the victim's nonce=1 slot. When the batcher later executes the block, it runs `__validate__` on the forged invoke (the batcher always validates), the transaction reverts, but the nonce increment is committed (Starknet reverted transactions still consume the nonce). The victim's legitimate nonce=1 transaction is displaced; the victim must resubmit with nonce=2.

Additionally, the forged transaction is broadcast to P2P peers as a valid mempool entry, polluting the network view of the victim's account state.

---

### Likelihood Explanation

**Medium.** The attack window is the time between a victim's `deploy_account` entering the mempool and being committed to a block. The mempool is observable (P2P gossip, RPC `get_mempool_snapshot`). The attacker needs only to craft an invoke with the victim's `sender_address`, `nonce=1`, and any signature bytes — no cryptographic material is required. The attack is cheap (only gas for the forged transaction's resource bounds) and can be automated to monitor the mempool continuously.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender's address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type in the pool, rather than merely checking address presence. Alternatively, the gateway can pass the `deploy_account` transaction hash alongside the invoke (as the `native_blockifier` `PyValidator` already does via `deploy_account_tx_hash`) and verify it is present in the mempool before skipping `__validate__`.

---

### Proof of Concept

```
1. Alice submits RpcTransaction::DeployAccount (nonce=0, class_hash=X, salt=S)
   → Gateway admits it; mempool now contains Alice's deploy_account.
   → Alice's committed nonce = 0 (account not yet on-chain).

2. Attacker (Bob) observes Alice's deploy_account in the mempool
   (via P2P gossip or RPC get_mempool_snapshot).

3. Bob constructs RpcTransaction::Invoke {
       sender_address: Alice's address,   // computed from class_hash + salt + constructor_calldata
       nonce: 1,
       signature: [0xdeadbeef],           // arbitrary bytes — never verified
       calldata: [...],                   // any calldata
       resource_bounds: <valid bounds>,
   }

4. Bob submits the forged invoke to the gateway.

5. Gateway stateless validation passes (valid resource bounds, nonce DA mode L1, etc.).

6. Gateway stateful validation:
   a. account_nonce = get_nonce(Alice) = 0  ✓
   b. validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap  ✓
   c. validate_by_mempool: no duplicate hash, nonce ≥ 0  ✓
   d. skip_stateful_validations:
        tx.nonce() == 1  ✓
        account_nonce == 0  ✓
        account_tx_in_pool_or_recent_block(Alice) == true  ✓  (Alice's deploy_account is in pool)
      → returns true (skip __validate__)
   e. run_validate_entry_point: validate=false → returns Ok immediately, NO signature check

7. Bob's forged invoke is added to the mempool at Alice's nonce=1 slot.

8. Batcher builds block:
   - Executes Alice's deploy_account → Alice's account deployed, nonce becomes 1.
   - Executes Bob's forged invoke → __validate__ called on Alice's account contract
     → signature check fails → transaction REVERTS.
   - Nonce increment (1→2) is committed despite revert.

9. Alice's nonce=1 slot is consumed. Alice must use nonce=2 for her intended first invoke.
   Alice's intended nonce=1 invoke (if already submitted) is now stale and will be rejected.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
