### Title
Missing Active Token Validation in `add_stake_from_pool` Allows Delegation to Disabled Token Pools - (File: `src/staking/staking.cairo`)

### Summary
`add_stake_from_pool` and `set_open_for_delegation` in the Staking contract do not verify that the token associated with a delegation pool is currently active. This allows delegators to lock funds into pools for disabled tokens, earning zero rewards while their capital is frozen for the full exit wait window.

### Finding Description
The Staking contract supports multiple tokens (STRK and BTC variants) with an enable/disable lifecycle managed via `enable_token` and `disable_token`. When a token is disabled, it stops contributing to reward calculations. However, two critical entry points omit the `is_active_token` check:

**`set_open_for_delegation`** (line 542–572, `src/staking/staking.cairo`) only asserts `does_token_exist`:

```cairo
assert!(self.does_token_exist(:token_address), "{}", Error::TOKEN_NOT_EXISTS);
// ← no is_active_token check
```

`does_token_exist` returns `true` for any registered token regardless of its active/inactive state:

```cairo
fn does_token_exist(self: @ContractState, token_address: ContractAddress) -> bool {
    token_address == STRK_TOKEN_ADDRESS || self.btc_tokens.read(token_address).is_some()
}
```

**`add_stake_from_pool`** (lines 1003–1046, `src/staking/staking.cairo`) performs no active-token check either:

```cairo
fn add_stake_from_pool(
    ref self: ContractState, staker_address: ContractAddress, amount: Amount,
) {
    self.general_prerequisites();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
    let pool_contract = get_caller_address();
    let token_address = self
        .staker_pool_info
        .entry(staker_address)
        .get_pool_token(:pool_contract)
        .expect_with_err(Error::CALLER_IS_NOT_POOL_CONTRACT);
    // ← no is_active_token(:token_address, epoch_id: curr_epoch) check
    ...
    self.add_to_total_stake(:token_address, amount: normalized_amount);
```

The contrast is clear: `get_total_stake_for_token` (line 820–832) *does* enforce the active check:

```cairo
assert!(
    self.is_active_token(:token_address, epoch_id: curr_epoch),
    "{}",
    Error::TOKEN_NOT_ACTIVE,
);
```

The full delegation entry path is:
1. Delegator → `Pool::enter_delegation_pool` (only checks `assert_staker_is_active`, i.e. `!staker_removed`)
2. Pool → `Staking::add_stake_from_pool` (no active-token check)

The `AddTokenWithoutEnableFlow` test (line 5641–5671, `src/flow_test/flows.cairo`) confirms the outcome:

```cairo
system.staking.add_token(token_address: token.contract_address());
// token NOT enabled
let pool = system.set_open_for_delegation(:staker, token_address: token.contract_address());
system.delegate_btc(:delegator, :pool, :amount, :token);
...
assert!(delegator_rewards == Zero::zero());
```

The same gap exists in `switch_staking_delegation_pool` (lines 1148–1245): no check that `to_pool`'s token is active before re-locking switched funds.

### Impact Explanation
A delegator who stakes into a pool for a disabled (or never-enabled) token:
- Earns **zero rewards** for the entire duration their funds remain locked.
- Must wait the full `exit_wait_window` (minimum 1 week, up to 12 weeks per `MAX_EXIT_WAIT_WINDOW`) to recover principal.
- Suffers a concrete, measurable yield loss with no recourse during the lock period.

This constitutes **griefing with damage to users** (Medium): a staker can open a pool for a disabled token and attract delegators who unknowingly lock capital for zero return.

### Likelihood Explanation
- `disable_token` can be called by the security agent at any time; existing pools for that token immediately become zero-reward traps for new entrants.
- `set_open_for_delegation` permits pool creation for a token that was never enabled (`add_token` without `enable_token`), creating a permanently zero-reward pool that is indistinguishable from a valid pool to an external observer.
- No on-chain guard or revert prevents the delegation; the delegator receives no warning.

### Recommendation
Add an `is_active_token` assertion in both `set_open_for_delegation` and `add_stake_from_pool`:

```cairo
// In set_open_for_delegation:
assert!(
    self.is_active_token(:token_address, epoch_id: self.get_current_epoch()),
    "{}",
    Error::TOKEN_NOT_ACTIVE,
);

// In add_stake_from_pool (after resolving token_address):
assert!(
    self.is_active_token(:token_address, epoch_id: self.get_current_epoch()),
    "{}",
    Error::TOKEN_NOT_ACTIVE,
);
```

Apply the same guard to `switch_staking_delegation_pool` for the `to_pool` token.

### Proof of Concept
1. Token `T` is registered via `add_token` but never enabled (or is later disabled via `disable_token`).
2. Staker calls `set_open_for_delegation(token_address: T)` — succeeds because `does_token_exist(T)` is `true`.
3. Delegator calls `Pool::enter_delegation_pool(reward_address, amount)` — `assert_staker_is_active()` passes.
4. Pool calls `Staking::add_stake_from_pool(staker_address, amount)` — no active-token check; funds transferred and locked.
5. Delegator calls `claim_rewards` after any number of epochs — receives **zero** (confirmed by `AddTokenWithoutEnableFlow` test at line 5669: `assert!(delegator_rewards == Zero::zero())`).
6. Delegator must call `exit_delegation_pool_intent` and wait the full `exit_wait_window` to recover principal, with no yield earned. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)

        /// Assumes that the token exists.
        ///
        /// Precondition: `get_current_epoch() <= epoch_id < get_current_epoch() + K`.
        fn is_active_token(
            self: @ContractState, token_address: ContractAddress, epoch_id: Epoch,
        ) -> bool {
            token_address == STRK_TOKEN_ADDRESS
                || is_btc_active(
