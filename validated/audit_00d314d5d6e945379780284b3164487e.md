### Title
Gateway Stateless Validator Unconditionally Rejects Valid L1-Gas-Only V3 Transactions via Unconditioned `l2_gas.max_price_per_unit` Check — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies the `min_gas_price` floor check unconditionally against `l2_gas.max_price_per_unit`, even when `l2_gas.max_amount == 0`. A valid V3 `AllResourceBounds` transaction that allocates only L1 gas (with `l2_gas` zeroed out) is rejected at the gateway with `MaxGasPriceTooLow` before it ever reaches the mempool or sequencer.

---

### Finding Description

The check at line 71 of `stateless_transaction_validator.rs`:

```rust
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

is applied to **every** `AllResourceBounds` transaction, regardless of whether `l2_gas.max_amount` is zero. [1](#0-0) 

The production default `min_gas_price` is `8_000_000_000`: [2](#0-1) 

A transaction with `AllResourceBounds { l1_gas: {max_amount:1000, max_price:1_000_000}, l2_gas: {max_amount:0, max_price:0}, l1_data_gas: {max_amount:0, max_price:0} }` passes the first check (non-zero total fee via `max_possible_fee`, which sums all three resources): [3](#0-2) 

But then immediately fails the unconditional `l2_gas.max_price_per_unit < min_gas_price` check, since `0 < 8_000_000_000`.

The existing test suite **explicitly acknowledges** that L1-gas-only transactions are valid (the `valid_l1_gas` positive-flow test case), but it sidesteps the bug by using `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` which sets `min_gas_price: 0`: [4](#0-3) [5](#0-4) 

With `min_gas_price: 0`, the condition `0 < 0` is false and the test passes. With the production default of `8_000_000_000`, the same transaction is rejected.

---

### Impact Explanation

Any user submitting a valid V3 `AllResourceBounds` transaction that allocates only L1 gas (or only L1 data gas) with `l2_gas.max_price_per_unit=0` is permanently rejected at the gateway with a misleading `MaxGasPriceTooLow` error. The transaction never reaches the mempool or sequencer. This is a concrete, deterministic admission corruption: a structurally valid transaction (non-zero total fee, correct format) is rejected before sequencing.

**Impact: High — Mempool/gateway admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

Any client that constructs a V3 transaction paying only via L1 gas (a supported and documented pattern) with `l2_gas` zeroed out will hit this on any deployment using the default or any non-zero `min_gas_price`. This is a straightforward, reproducible path requiring no special privileges.

---

### Recommendation

Guard the `min_gas_price` check on `l2_gas` with a condition that `l2_gas.max_amount > 0`:

```rust
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

Alternatively, apply the floor check to each resource dimension only when its `max_amount` is non-zero, consistent with how the blockifier's `check_fee_bounds` handles per-resource price checks. [6](#0-5) 

The test `valid_l1_gas` should also be updated to use a non-zero `min_gas_price` to actually exercise the corrected guard.

---
