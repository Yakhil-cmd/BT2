### Title
Missing Fee Flush Before Admin Fee Destination Change Redirects Accrued Fees to New Recipient — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first collecting accrued fees to the old destination. All spread-surplus and notional fees that accumulated before the change are subsequently paid to the new destination when any future `collectPoolFees` call executes.

### Finding Description

Every fee-rate mutation in the factory correctly flushes accrued fees before applying the new configuration. `setPoolAdminFees` calls `collectFees` with the current destination before writing new rates:

```solidity
// MetricOmmPoolFactory.sol lines 417-425
PoolFeeConfig memory c = poolFeeConfig[pool];
IMetricOmmPoolCollectFees(pool)
  .collectFees(
    c.protocolSpreadFeeE6,
    c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8,
    c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination used here
  );
```

`setPoolProtocolFee` follows the same pattern. However, `setPoolAdminFeeDestination` performs no such flush:

```solidity
// MetricOmmPoolFactory.sol lines 438-447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

When `collectFees` is later called (permissionlessly via `collectPoolFees`), the pool distributes the entire accumulated spread surplus and notional balance to `adminFeeDestination_`, which is now the new address:

```solidity
// MetricOmmPool.sol lines 416-420
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
```

The old destination receives nothing for the period it was entitled to.

### Impact Explanation

The old admin fee destination loses all fees that accrued before the destination change. Both fee types are affected:

- **Spread fees**: the entire token0/token1 surplus accumulated since the last collection is split by the new destination's ratio.
- **Notional fees**: `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` (on-pool accumulators) are zeroed and paid entirely to the new destination.

This is a direct, quantifiable loss of owed fees to the old recipient. The magnitude equals all fees accrued since the last `collectPoolFees` call, which can be arbitrarily large if collection has been infrequent.

### Likelihood Explanation

The trigger is a routine, within-scope admin action. Two realistic paths exist:

1. **Admin role transfer followed by destination change**: Old admin calls `proposePoolAdminTransfer`; new admin calls `acceptPoolAdmin` (no fee flush occurs there either), then immediately calls `setPoolAdminFeeDestination` to redirect to their own treasury. All fees earned under the old admin's tenure are captured by the new admin.

2. **Admin changes their own destination without collecting first**: A pool admin updating their treasury address without manually calling `collectPoolFees` first loses all previously accrued fees to the new address. This is an easy operational mistake given that `setPoolAdminFees` handles the flush automatically and admins may assume `setPoolAdminFeeDestination` does the same.

Both paths require only valid semi-trusted admin actions within documented caps.

### Recommendation

Apply the same flush-before-update pattern used in `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the current destination before switching
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool has been live for N blocks; `poolAdminFeeDestination[pool] = Alice`.
2. Significant swap volume has accumulated spread surplus and notional fees on the pool (e.g., 1000 USDC worth of admin fees).
3. Pool admin calls `setPoolAdminFeeDestination(pool, Bob)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is now `Bob`.
4. Any caller (keeper, Bob, or anyone) calls `collectPoolFees(pool)`.
5. The pool transfers the full 1000 USDC admin share to `Bob`. Alice receives 0.
6. Alice's accrued fees are permanently lost to her.

The same outcome occurs after an admin role transfer: old admin proposes transfer → new admin accepts → new admin calls `setPoolAdminFeeDestination` → all historical fees redirect to new admin's treasury. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
