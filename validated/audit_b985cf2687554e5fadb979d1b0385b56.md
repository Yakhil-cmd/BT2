### Title
`setPoolAdminFeeDestination` Does Not Flush Accrued Fees Before Redirecting the Admin Fee Recipient — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees` at the old destination. All admin fees (spread surplus and notional accumulator) that accrued under the old destination are silently redirected to the new one on the next collection, causing a direct loss of earned fees for the old recipient.

---

### Finding Description

The factory enforces a consistent pre-collection pattern before any fee-parameter change:

- `setPoolAdminFees` (lines 408–435): calls `collectFees` with the current config **before** writing new rates.
- `setPoolProtocolFee` (lines 318–360): calls `collectFees` with the current config **before** writing new rates.

`setPoolAdminFeeDestination` (lines 438–447) breaks this pattern — it writes the new destination directly with no prior flush:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

When `collectFees` is subsequently called (via `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee`), the pool computes:

1. **Spread surplus** — `balance * SCALE_MULTIPLIER − binTotals.scaledToken − notionalFeeScaled` — all tokens sitting in the pool above LP entitlements, earned since the last collection.
2. **Notional accumulator** — `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` — explicitly tracked per-swap notional fees.

Both are transferred to `adminFeeDestination_`, which is now the **new** address. The old destination receives nothing for the period it was active.

---

### Impact Explanation

The old `adminFeeDestination` loses 100 % of its accrued admin share of both spread fees and notional fees — real token balances already held in the pool. The magnitude equals the admin-fee portion of all swap volume since the last collection. For an active pool with a non-trivial `adminSpreadFeeE6` or `adminNotionalFeeE8`, this can be material. The loss is permanent: once the new destination is set and fees are collected, the old destination has no on-chain recourse.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that can legitimately call `setPoolAdminFeeDestination` at any time. The trigger requires no special conditions — just a routine destination update. The admin may act:

- **Maliciously**: deliberately redirect accumulated fees to a new address they control before the old recipient can call `collectPoolFees`.
- **Accidentally**: unaware that fees must be flushed first, mirroring the exact class of omission in the reference bug.

`collectPoolFees` is permissionless, so the old destination could theoretically front-run the admin's update, but the admin can also front-run them. There is no timelock on `setPoolAdminFeeDestination`.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the **current** destination before writing the new one.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the current destination before switching.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = Alice`, `adminSpreadFeeE6 = 5_000`, `adminNotionalFeeE8 = 1_000_000`.
2. Swaps occur; the pool accumulates spread surplus and `notionalFeeToken0/1Scaled` owed to Alice.
3. Pool admin calls `setPoolAdminFeeDestination(pool, Bob)`. No fees are collected; `poolAdminFeeDestination[pool]` is now Bob.
4. Anyone calls `collectPoolFees(pool)`. Inside `collectFees`, `adminFeeDestination_` is Bob. Both the spread surplus and the notional accumulator are transferred to Bob.
5. Alice receives zero tokens despite having earned fees during step 2.
