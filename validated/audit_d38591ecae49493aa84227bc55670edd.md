### Title
Integer Division Truncation in `native_per_pubdata` Silently Zeros Pubdata Resource Charges - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary
The `native_per_pubdata` ratio used to charge users for publishing data is computed via floor integer division (`wrapping_div`). When `pubdata_price < native_price`, the result truncates to zero, causing every user to pay zero native resources for all pubdata they generate, regardless of volume.

### Finding Description
In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 142:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is floor integer division. When `pubdata_price < native_price` (e.g., `pubdata_price = 99`, `native_price = 100`), the result is `0`. This zero propagates through the entire resource accounting pipeline:

1. **Intrinsic pubdata overhead** (`gas_helpers.rs` line 352):
   `intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` → `0`

2. **Execution pubdata charge** (`gas_helpers.rs` lines 430–431):
   `native = current_pubdata_spent.checked_mul(native_per_pubdata)` → `0`

3. **`check_enough_resources_for_pubdata`** always returns "sufficient" because 0 native resources are required.

The identical floor-division pattern appears in `api/src/helpers.rs` line 427:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

The asymmetry with `native_per_gas` is the key indicator this is unintentional. `native_per_gas` uses **ceiling** division to guarantee a non-zero result:
```rust
// validation_impl.rs line 135
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
```

Gas fees cannot round to zero; pubdata fees can. The two resource dimensions are treated inconsistently. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
When `native_per_pubdata = 0`, every call to `get_resources_to_charge_for_pubdata` returns zero native resources required. Concretely:

- **Operator financial loss**: The operator pays L1 calldata/blob costs for all published state diffs but collects zero native-resource fees from users for that pubdata. The fee shortfall scales with pubdata volume.
- **Pubdata spam without native-resource cost**: Any user can write to many storage slots (each generating 32–65 bytes of pubdata) up to the block gas limit without consuming any native resource budget. The post-execution pubdata check (`check_enough_resources_for_pubdata`) always passes, so transactions are never reverted for insufficient native resources due to pubdata. [4](#0-3) 

### Likelihood Explanation
The condition `pubdata_price < native_price` is a realistic and commonly tested configuration. The standard test `BlockContext::default()` sets `native_price = 10` and `pubdata_price = 0`. The test `test_check_pubdata_encoding_version` uses `pubdata_price = 2, native_price = 100`, giving `native_per_pubdata = 0`. Any block produced with such operator-configured prices exposes every transaction sender to this zero-cost pubdata path — no special privileges required beyond submitting a normal transaction. [5](#0-4) 

### Recommendation
Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with `native_per_gas`:

```rust
// Before (validation_impl.rs line 142)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that even when `pubdata_price < native_price`, users are charged at least 1 native unit per byte of pubdata, preventing zero-cost pubdata generation.

### Proof of Concept
1. Operator sets `pubdata_price = 99`, `native_price = 100` (L1 data slightly cheaper than proving — a plausible production configuration).
2. `native_per_pubdata = 99 / 100 = 0` (floor division in `wrapping_div`).
3. User submits a transaction calling a contract that writes to 1,000 storage slots, generating approximately 32,000 bytes of pubdata.
4. `get_resources_to_charge_for_pubdata` computes `32_000 * 0 = 0`

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
```
