### Title
Gateway Stateless Validator Unconditionally Checks `l2_gas.max_price_per_unit` Against `min_gas_price`, Incorrectly Rejecting Valid L1-Gas-Only Transactions - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` first accepts transactions whose total fee is non-zero across *any* resource (L1 gas, L2 gas, or L1 data gas), but then unconditionally compares `l2_gas.max_price_per_unit` against the production `min_gas_price` of `8_000_000_000`. A transaction that legitimately carries only L1 gas or L1 data gas bounds has `l2_gas.max_price_per_unit = 0`, which is always less than `8_000_000_000`, so the gateway rejects it with `MaxGasPriceTooLow` even though the transaction is structurally valid.

### Finding Description

In `validate_resource_bounds`:

```rust
// Line 66-68: passes for any tx where at least one resource is non-zero
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
    return Err(ZeroResourceBounds { resource_bounds });
}

// Line 71-76: unconditionally checks l2_gas price — no guard for "l2_gas not used"
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(MaxGasPriceTooLow {
        gas_price: resource_bounds.l2_gas.max_price_per_unit,
        min_gas_price: self.config.min_gas_price,
    });
}
```

The first check is intentionally permissive: the comment says "at least one resource bound (L1, L2, or L1 data) must be non-zero." The test suite even has explicit positive cases `valid_l1_gas` and `valid_l1_data_gas` that set only those fields and leave `l2_gas` at its zero default. Those tests pass only because `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` sets `min_gas_price: 0`. The production default is `min_gas_price: 8_000_000_000`, so the second check fires for every transaction that does not set an L2 gas price. [1](#0-0) [2](#0-1) 

The test config masks the defect by zeroing `min_gas_price`: [3](#0-2) 

The positive test cases that prove L1-only bounds are intended to be valid: [4](#0-3) 

### Impact Explanation

Every transaction that sets only `l1_gas` or `l1_data_gas` bounds (leaving `l2_gas.max_price_per_unit = 0`) is rejected at the stateless gateway layer with `MaxGasPriceTooLow` before it ever reaches the mempool or blockifier. The gateway is the public admission point; this is a permanent, deterministic rejection of a class of structurally valid transactions. Impact: **High — gateway admission rejects valid transactions before sequencing**.

### Likelihood Explanation

The `AllResourceBounds` type is the only resource-bounds representation accepted by the RPC/gateway layer. Any client that constructs a V3 transaction using only L1 gas (e.g., for legacy compatibility or explicit L1-DA cost control) will hit this rejection unconditionally in production. The protocol's own `ZeroResourceBounds` guard explicitly acknowledges L1-only transactions as valid. Likelihood: **Medium** — the scenario is uncommon today because most wallets set L2 gas, but it is a reachable, unprivileged path.

### Recommendation

Guard the `MaxGasPriceTooLow` check so it only fires when L2 gas is actually being used (i.e., when `l2_gas.max_amount > 0` or `l2_gas.max_price_per_unit > 0`):

```diff
-if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
+if resource_bounds.l2_gas.max_price_per_unit.0 > 0
+    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
+{
     return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
         gas_price: resource_bounds.l2_gas.max_price_per_unit,
         min_gas_price: self.config.min_gas_price,
     });
 }
```

Similarly audit the `MaxGasAmountTooHigh` check on line 80 for the same class of transactions.

### Proof of Concept

Submit the following RPC transaction to the production gateway (where `min_gas_price = 8_000_000_000`):

```json
{
  "type": "INVOKE",
  "resource_bounds": {
    "l1_gas":      { "max_amount": "0x1", "max_price_per_unit": "0x1" },
    "l2_gas":      { "max_amount": "0x0", "max_price_per_unit": "0x0" },
    "l1_data_gas": { "max_amount": "0x0", "max_price_per_unit": "0x0" }
  },
  ...
}
```

**Step 1** — `ZeroResourceBounds` check: `max_possible_fee(Tip::ZERO)` = `1 * 1 = 1 ≠ 0` → **passes**.

**Step 2** — `MaxGasPriceTooLow` check: `l2_gas.max_price_per_unit.0 = 0 < 8_000_000_000` → **fails**, transaction rejected.

The transaction is valid by the protocol's own zero-bounds invariant but is permanently blocked at the stateless gateway layer. The test suite does not catch this because `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING.min_gas_price = 0`, making `0 < 0` false and masking the production behaviour. [5](#0-4)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
