### Title
Missing Validation of `app_pk` Allows Malicious Coordinator to Extract Confidential Derived Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd` function accepts an `app_pk` (application ElGamal public key) parameter with no check that it is a valid non-identity group element. A malicious coordinator who controls the `app_pk` value distributed to participants can supply the G1 identity point, collapsing the ElGamal blinding term to zero and allowing the coordinator to read the confidential derived key directly from the aggregated protocol output.

### Finding Description
The CKD protocol is designed so that each participant computes an ElGamal-encrypted share of the confidential key and sends it to the coordinator. The coordinator aggregates the shares but is supposed to be unable to recover the plaintext without the application's secret key `app_sk`.

In `compute_signature_share` (called by both `do_ckd_participant` and `do_ckd_coordinator`):

```
big_s  = H(pk ‖ app_id) · private_share_i          // the secret share contribution
big_y  = y · G                                       // ephemeral public key
big_c  = big_s + app_pk · y                          // ElGamal ciphertext
```

The coordinator receives `(λᵢ · big_y, λᵢ · big_c)` from every participant and sums them:

```
Σ λᵢ · big_c  =  Σ λᵢ · big_s  +  app_pk · Σ λᵢ · y
               =  msk · H(pk ‖ app_id)  +  app_pk · (blinding)
```

The blinding term `app_pk · (blinding)` is what hides `msk · H(pk ‖ app_id)` from the coordinator. If `app_pk` is the G1 identity element `O`, then `app_pk · y = O` for any scalar `y`, so:

```
Σ λᵢ · big_c  =  msk · H(pk ‖ app_id)   (plaintext confidential key, no blinding)
```

The coordinator can read the confidential derived key directly from the aggregated `big_c` without ever knowing `app_sk`.

Neither `ckd` (the public entry point) nor `compute_signature_share` performs any check on `app_pk`: [1](#0-0) [2](#0-1) 

The only input-validation performed in `ckd` concerns participant-list membership and deduplication; `app_pk` is forwarded unchecked all the way into the scalar multiplication.

### Impact Explanation
If `app_pk = O` (identity), the coordinator aggregates `total_big_c = msk · H(pk ‖ app_id)`, which is exactly the confidential derived secret the protocol is designed to protect. This constitutes **direct extraction of a confidential derived key** — a Critical-severity impact under the allowed scope ("Extraction, reconstruction, or disclosure of … confidential derived secrets").

### Likelihood Explanation
In coordinator-based deployments the coordinator is the natural party that distributes session parameters — including `app_pk` — to the other participants before the protocol round begins. A malicious coordinator can trivially substitute `ElementG1::identity()` for the legitimate application public key. No cryptographic break, no leaked key, and no external dependency is required; the attack is a single-field substitution in the session setup message.

### Recommendation
Add an explicit identity-point check in `ckd` before the protocol is started:

```rust
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Optionally, also verify that `app_pk` is in the correct prime-order subgroup (cofactor check) if the underlying library does not enforce this automatically during deserialization.

### Proof of Concept
1. Coordinator calls `ckd(participants, coordinator, me, key_pair, app_id, ElementG1::identity(), rng)` and instructs all other participants to do the same (e.g., by distributing `app_pk = identity` in the session-setup message).
2. Every participant executes `compute_signature_share` with `app_pk = O`:
   - `big_c = big_s + O · y = big_s` — the blinding term vanishes.
   - Each participant sends `(λᵢ · big_y, λᵢ · big_s)` to the coordinator.
3. The coordinator sums the received shares:
   - `total_big_c = Σ λᵢ · big_s_i = msk · H(pk ‖ app_id)`.
4. The coordinator now holds the confidential derived key in plaintext, without ever possessing `app_sk`, defeating the entire confidentiality guarantee of the CKD protocol.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
