### Title
Unrestricted `update_rewards` Allows Any Caller to Grief Consensus Block Reward Distribution - (File: src/staking/staking.cairo)

---

### Summary

The `IStakingRewardsManager::update_rewards` function in `src/staking/staking.cairo` has no access control and can be called by any non-zero address. A global `last_reward_block` variable enforces a one-call-per-block limit. An attacker can call `update_rewards` with `disable_rewards: true` before the legitimate consensus mechanism, consuming the block's reward slot without distributing any rewards, permanently denying yield to all stakers.

---

### Finding Description

`update_rewards` is exposed as a public entrypoint with no role-based access control. Its only caller restriction is `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. [1](#0-0) 

The function enforces a global one-call-per-block invariant via `last_reward_block`: [2](#0-1) 

`last_reward_block` is a single global storage slot, not per-staker: [3](#0-2) 

When called with `disable_rewards: true`, the function writes `current_block_number` to `last_reward_block` and returns immediately without distributing any rewards: [4](#0-3) 

Because `last_reward_block` is global, a single attacker call with `disable_rewards: true` for any valid staker address exhausts the reward slot for the entire block. All subsequent calls to `update_rewards` in that block — for any staker — revert with `REWARDS_ALREADY_UPDATED`.

The `IStakingRewardsManager` interface documents no access restriction: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` once per block prevents every staker from receiving consensus block rewards for that block. Repeated every block, this permanently freezes all consensus-phase yield for all stakers at the cost of only transaction fees per block. The stakers' `unclaimed_rewards_own` fields are never incremented, and the rewards are never minted/transferred.

---

### Likelihood Explanation

- The function is fully public; no role, signature, or privileged key is required.
- The only prerequisite is a valid, active staker address, which is trivially observable on-chain from `NewStaker` events.
- The attack requires one cheap transaction per block and can be fully automated.
- There is no economic cost beyond gas; the attacker does not need to hold any stake.

---

### Recommendation

Restrict `update_rewards` to an authorized caller (e.g., a dedicated consensus role, the attestation contract, or a specific operational address). Alternatively, make `last_reward_block` per-staker so that one call cannot block reward distribution for all other stakers.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. In block N, the attacker submits: `update_rewards(valid_staker_address, disable_rewards: true)`.
3. The function passes all checks (non-zero caller, staker exists and is active, `current_block_number > last_reward_block`).
4. `last_reward_block` is written to N; the function returns early without distributing rewards. [4](#0-3) 
5. The legitimate consensus mechanism attempts `update_rewards(staker_address, disable_rewards: false)` in block N.
6. The assertion `current_block_number > self.last_reward_block.read()` fails (`N > N` is false) → reverts with `REWARDS_ALREADY_UPDATED`.
7. No staker receives block rewards for block N.
8. The attacker repeats step 2 every block, permanently freezing all consensus block rewards.
