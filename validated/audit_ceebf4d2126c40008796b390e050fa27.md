### Title
Unbounded `MessageBuffer` Growth via Malicious Message Injection - (File: `src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` inside `Comms` accumulates incoming messages in a `HashMap<MessageHeader, SubMessageQueue>` with no size cap. Each `SubMessageQueue` is backed by a `futures::channel::mpsc::unbounded()` channel. A malicious participant can call `Protocol::message()` with crafted payloads containing arbitrary `MessageHeader` values (novel channel tags or waitpoints the honest protocol never polls), causing unbounded HashMap entry creation and unbounded per-queue memory growth. The codebase itself contains a test that explicitly confirms this attack path is reachable and effective.

---

### Finding Description

`MessageBuffer` is defined as:

```rust
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
``` [1](#0-0) 

Each `SubMessageQueue` is created with an explicitly unbounded channel:

```rust
let (sender, receiver) = futures::channel::mpsc::unbounded();
``` [2](#0-1) 

The `push` method unconditionally inserts a new `HashMap` entry for any unseen `MessageHeader` and enqueues the message with no size check:

```rust
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
``` [3](#0-2) 

`push_message`, the function called from the public `Protocol::message()` interface, performs only a minimum-length check and header parse — no validation that the header corresponds to a waitpoint the honest protocol will ever consume:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;
    };
    self.incoming.push(header, from, message);
}
``` [4](#0-3) 

The public `Protocol` trait exposes `message()` as the delivery point for all incoming network bytes:

```rust
fn message(&mut self, from: Participant, data: MessageData);
``` [5](#0-4) 

`ProtocolExecutor::message()` forwards directly to `Comms::push_message()` with no additional filtering:

```rust
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
``` [6](#0-5) 

A `MessageHeader` is 40 bytes: a 32-byte `ChannelTag` plus an 8-byte `Waitpoint`. The space of valid-looking headers is astronomically large (2^320), so an attacker can generate an unbounded number of distinct headers, each creating a new `HashMap` entry that is never consumed.

The codebase itself contains a test that explicitly documents and confirms this behavior:

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    // Attacker injects messages for waitpoints the honest code never polls.
    comms.push_message(attacker, message);
    // ...
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [7](#0-6) 

---

### Impact Explanation

A malicious participant sends a stream of crafted `MessageData` payloads, each with a distinct `MessageHeader` (varying the 8-byte waitpoint field is sufficient). Every payload passes the `push_message` length check, parses as a valid header, and is inserted as a new `HashMap` entry backed by an unbounded channel. Memory grows without bound proportional to the number of injected messages. This can exhaust the heap of any honest party running a DKG, reshare, refresh, presign, or sign protocol, causing an OOM crash or severe degradation — permanently denying the honest party the ability to complete the protocol.

This matches the allowed Medium impact: **Griefing or resource-exhaustion by a malicious participant causing unbounded memory beyond documented behavior.**

---

### Likelihood Explanation

The `Protocol::message()` method is the standard delivery interface that every application using this library must call for each received network message. Any participant in the protocol set — or any network-level attacker who can inject bytes to an honest party's message handler — can trigger this. No cryptographic material, no key compromise, and no special privilege is required. The attacker only needs to be able to call `Protocol::message()` on the victim's protocol instance, which is the normal operating condition for all protocols in this library.

---

### Recommendation

Enforce a hard cap on the total number of distinct `MessageHeader` entries in `MessageBuffer` and/or on the per-queue depth. For example:

- Reject `push_message` calls once the `HashMap` exceeds a configurable maximum entry count (e.g., `max_participants * max_rounds * safety_factor`).
- Reject per-queue enqueue once a queue exceeds a configurable depth (e.g., `max_participants`).
- Optionally, validate that the incoming `ChannelTag` matches one of the known root tags (`root_shared` or a known `root_private`) before accepting the message, discarding messages with unrecognized channel tags entirely.

---

### Proof of Concept

The existing test in the repository already demonstrates the attack:

```rust
let comms = Comms::new();
let attacker = Participant::from(99_u32);
let attack_count = 512_u64;

for i in 0..attack_count {
    let header =
        MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
    let mut message = header.to_bytes().to_vec();
    message.extend_from_slice(&i.to_le_bytes());
    comms.push_message(attacker, message);
}

// 512 distinct HashMap entries created, none ever consumed
assert!(messages.len() == usize::try_from(attack_count).unwrap());
``` [7](#0-6) 

Scaling `attack_count` to millions and running this against an honest party's live `ProtocolExecutor` during any DKG, reshare, or signing session exhausts heap memory. The honest party's protocol instance never drains these entries because the honest code only ever polls the small, fixed set of waitpoints it actually uses; all injected entries accumulate permanently for the lifetime of the protocol.
