### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Delegator's Unclaimed Yield via Trace Inflation — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over a delegator's `pool_member_epoch_balance` trace. Because every balance change in a distinct epoch appends a new checkpoint to this trace, a delegator who repeatedly modifies their balance across many epochs without ever claiming rewards will accumulate an arbitrarily large trace. When `claim_rewards` is eventually called, the loop must traverse every accumulated entry, and if the trace is large enough the transaction will exceed the Starknet gas limit, permanently freezing the delegator's unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current trace length:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`: [2](#0-1) 

The underlying `insert` implementation only deduplicates entries with the **same epoch key**; a different epoch always appends a new checkpoint:

```cairo
if last.key == key {
    last.value = value;          // update in-place
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, ...);
    checkpoints.push(...);       // NEW entry
}
``` [3](#0-2) 

`set_member_balance` is called by every balance-mutating entrypoint:

- `enter_delegation_pool` → `set_member_balance` [4](#0-3) 
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance` [5](#0-4) 
- `exit_delegation_pool_intent` → `set_member_balance` [6](#0-5) 

The `entry_to_claim_from` cursor stored in `pool_member_info` is only advanced when `claim_rewards` succeeds: [7](#0-6) 

If the delegator never calls `claim_rewards`, the cursor stays at 0 and the loop must traverse every entry ever written.

---

### Impact Explanation

A delegator who makes one balance change per epoch and defers claiming rewards accumulates one new trace entry per epoch. After sufficiently many epochs the `claim_rewards` (or `pool_member_info_v1`) call will exceed the Starknet per-transaction gas limit, reverting every time. Because the cursor is only advanced inside a successful `claim_rewards` execution, there is no recovery path: the delegator's accrued yield is permanently inaccessible.

**Impact**: High — Permanent freezing of unclaimed yield.

---

### Likelihood Explanation

The attack requires only that a delegator:
1. Makes at least one balance-changing call per epoch (minimum-amount `add_to_delegation_pool` suffices).
2. Defers `claim_rewards` for a large number of epochs.

Both conditions are entirely within the control of an unprivileged delegator. Additionally, `add_to_delegation_pool` permits the **reward address** to be the caller and to supply the funds: [8](#0-7) 

This means a malicious reward address (e.g., a contract) can inflate another pool member's trace without the pool member's ongoing participation, as long as the reward address was set to a hostile contract at entry time.

The developers themselves flag the risk inline: *"This loop is unbounded but unlikely to exceed gas limits."* [9](#0-8) 

**Likelihood**: Medium — deliberate self-infliction or via a hostile reward address.

---

### Recommendation

1. **Enforce a maximum trace length** per pool member (e.g., cap at a constant such as 256 entries) and reject balance changes that would exceed it, or require the member to claim rewards first.
2. **Alternatively**, require `claim_rewards` to be called (or auto-claim) whenever a balance change is made, so `entry_to_claim_from` is always advanced and the loop never accumulates more than `K` entries between claims.
3. **At minimum**, document the maximum safe trace length and enforce it with an assertion so the failure mode is a clean revert rather than an OOG.

---

### Proof of Concept

```
Epoch 0:  delegator enters pool (trace length = 1, entry_to_claim_from = 0)
Epoch 1:  add_to_delegation_pool(1 token)  → trace length = 2
Epoch 2:  add_to_delegation_pool(1 token)  → trace length = 3
...
Epoch N:  add_to_delegation_pool(1 token)  → trace length = N+1

claim_rewards() called at epoch N:
  → calculate_rewards loops from entry 0 to N
  → each iteration: storage read (pool_member_trace.at(i)) + find_sigma call
  → for large N, transaction gas exceeds Starknet limit → revert
  → entry_to_claim_from never advances → all subsequent claim_rewards calls also revert
  → delegator's accumulated yield is permanently frozen
```
