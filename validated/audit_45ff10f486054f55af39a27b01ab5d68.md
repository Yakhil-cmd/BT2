Audit Report

## Title
Unbounded Body Buffering via `to_bytes(body, usize::MAX)` in `RosettaMetricsMiddleware` Enables OOM DoS — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`RosettaMetricsMiddleware::call` unconditionally buffers the entire HTTP request body into heap memory with no size limit via `to_bytes(body, usize::MAX)` before forwarding to any route handler. An unauthenticated attacker can POST an arbitrarily large body to any Rosetta endpoint, exhausting process memory and causing OOM termination of the Rosetta server, interrupting ledger sync and making all API endpoints unavailable.

## Finding Description
In `extract_canister_id` (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`, line 389), the body is consumed with `to_bytes(body, usize::MAX)`. The `usize::MAX` argument is the explicit "no limit" sentinel in axum's `to_bytes` API, causing the entire streaming body to be collected into a single heap allocation. This function is called unconditionally from `RosettaMetricsMiddleware::call` (line 336) for every request that is not `/metrics`. The middleware is applied as the innermost layer (line 383 of `rs/rosetta-api/icrc1/src/main.rs`) wrapping all 20 POST routes. The outer layers are only `add_request_span()` (a tracing layer) and `RequestIdLayer` — neither imposes a body size limit. A search across the entire `rs/rosetta-api/` tree for `RequestBodyLimitLayer`, `DefaultBodyLimit`, `body_limit`, or `max_body` returns zero matches. The error path at line 338 (`Err(_) => (None, Body::empty())`) only handles application-level errors returned by `to_bytes`; an OOM kill is delivered by the OS as `SIGKILL` and is never caught. The server binds to `0.0.0.0` (line 393 of `main.rs`), making every endpoint publicly reachable.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* A single HTTP POST with a multi-gigabyte streaming body to any Rosetta endpoint (e.g., `/account/balance`, `/construction/submit`) causes the Rosetta process to be OOM-killed. This terminates the ledger sync thread and renders all Rosetta API endpoints unavailable until the process is manually restarted, causing concrete harm to exchange integrations relying on the ICRC-1 Rosetta API.

## Likelihood Explanation
The attack requires only a single unauthenticated HTTP connection and a streaming POST body — no credentials, no special knowledge, no privileged access. The Rosetta API is exchange-facing and publicly reachable by design. The vulnerable code path runs on every non-`/metrics` request, so all 20 endpoints are equally affected. The attack is trivially repeatable: after a restart, a single new request re-triggers the condition.

## Recommendation
Replace `usize::MAX` with a reasonable maximum body size in `extract_canister_id` (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`, line 389):

```rust
const MAX_BODY_SIZE: usize = 1 * 1024 * 1024; // 1 MB
let bytes = to_bytes(body, MAX_BODY_SIZE).await.map_err(|_| {
    // propagate as error; caller returns Body::empty() or a 413 response
})?;
```

Alternatively, add `axum::extract::DefaultBodyLimit::max(1_048_576)` as an outer layer in `main.rs` before `metrics_layer` so oversized requests are rejected with HTTP 413 before any buffering occurs.

## Proof of Concept
```bash
# Send a 512 MB body to any Rosetta endpoint
dd if=/dev/zero bs=1M count=512 | \
  curl -X POST http://<rosetta-host>:<port>/account/balance \
       -H "Content-Type: application/json" \
       --data-binary @- &

# Repeat a few times concurrently; monitor RSS of the rosetta process
watch -n1 'ps aux | grep rosetta'
# Process RSS climbs until OOM kill
```

Call chain: `POST /account/balance` → `RosettaMetricsMiddleware::call` (line 304) → `extract_canister_id(body)` (line 336) → `to_bytes(body, usize::MAX)` (line 389) → heap exhaustion → `SIGKILL`.
