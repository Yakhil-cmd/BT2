### Title
Gateway Stateful Admission Omits L1 and L1-Data Gas Price Validation, Allowing Transactions That Deterministically Fail at Blockifier Pre-Validation — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful admission path checks only the L2 gas price against the previous block's price. It explicitly skips L1 gas and L1-data gas price validation for all transaction types. The blockifier's `check_fee_bounds`, executed later during sequencing, checks all three gas prices (L1, L2, L1-data) against the actual block prices. Any transaction whose `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` is below the actual block price passes gateway admission but deterministically fails at blockifier pre-validation — the exact asymmetric-validation pattern from the external report.

---

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` fetches the previous block's L2 gas price and delegates to `validate_tx_l2_gas_price_within_threshold`:

```rust

            // ... checks only l2_gas price ...
        }
        ValidResourceBounds::L1Gas(_) => {
            // No validation required for legacy transactions.
        }
    }
    Ok(())

``` [1](#0-0) 

The developer-acknowledged TODO at line 358 confirms the gap: L1 gas and L1-data gas prices are never compared against the actual block price at admission time.

The stateless validator also only checks `l2_gas.max_price_per_unit` against a configured floor (`min_gas_price`), and never touches L1 or L1-data gas prices: [2](#0-1) 

In contrast, `AccountTransaction::check_fee_bounds` — called inside `perform_pre_validation_stage` during blockifier sequencing — checks **all three** gas prices against the actual block gas prices for `AllResources` transactions, and checks L1 gas price for `L1Gas` transactions: [3](#0-2) 

The error type `ResourceBoundsError::MaxGasPriceTooLow` is raised for L1Gas, L2Gas, and L1DataGas independently: [4](#0-3) 

The gap is therefore:

| Check | L2 gas price | L1 gas price | L1-data gas price |
|---|---|---|---|
| Stateless gateway | ✓ (floor only) | ✗ | ✗ |
| Stateful gateway | ✓ (vs prev block) | ✗ | ✗ |
| Blockifier `check_fee_bounds` | ✓ | ✓ | ✓ |

---

### Impact Explanation

A transaction with `AllResources` bounds where `l1_gas.max_price_per_unit = 0` (or any value below the actual L1 gas price) and a valid `l2_gas.max_price_per_unit`:

1. Passes `StatelessTransactionValidator::validate` — only L2 gas price is checked against `min_gas_price`.
2. Passes `StatefulTransactionValidator::validate_resource_bounds` — only L2 gas price is compared to the previous block price; L1 and L1-data prices are never read.
3. Is admitted to the mempool.
4. Fails deterministically at `check_fee_bounds` inside `perform_pre_validation_stage` with `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }`.

The same applies to `L1Gas` (legacy) transactions: the stateful validator explicitly returns `Ok(())` for them with the comment "No validation required for legacy transactions," yet `check_fee_bounds` does compare their `max_price_per_unit` against the actual L1 gas price.

This matches the "High" impact scope: **Mempool/gateway/RPC admission accepts invalid transactions before sequencing.** The sequencer wastes CPU running `perform_pre_validation_stage` on transactions that are guaranteed to fail, and the mempool can be filled with such transactions by any unprivileged sender who sets a valid L2 gas price but a zero or stale L1/L1-data gas price.

---

### Likelihood Explanation

- **Unprivileged trigger**: any account can submit a V3 `AllResources` transaction with `l1_gas.max_price_per_unit = 0` and a valid `l2_gas.max_price_per_unit`. No special role or privilege is required.
- **Natural occurrence**: L1 gas prices fluctuate with Ethereum base fee. A transaction submitted during a low-L1-fee period with a tight `l1_gas.max_price_per_unit` will pass admission but fail execution if L1 fees rise before the block is built.
- **Deliberate exploitation**: an attacker can set `l1_gas.max_price_per_unit = 0` and `l2_gas.max_price_per_unit` just above the threshold, guaranteeing admission and guaranteed execution failure, at the cost of only the L2 gas component of the balance check (`verify_can_pay_committed_bounds` uses `max_possible_fee` which sums all three resources — but with `l1_gas.max_price_per_unit = 0` the committed fee is only the L2 component, making the attack cheap).

---

### Recommendation

Extend `validate_tx_l2_gas_price_within_threshold` (or replace it with a general `validate_tx_gas_prices_within_threshold`) to also compare `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against the previous block's corresponding prices (with the same configurable percentage threshold). The function's own TODO comment (`// TODO(Arni): Consider running this validation for all gas prices.`) already identifies this gap. For `L1Gas` legacy transactions, apply the same threshold check to `l1_gas.max_price_per_unit` instead of silently returning `Ok(())`.

---

### Proof of Concept

1. Deploy a funded account on the network.
2. Submit an `AllResources` invoke transaction with:
   - `l2_gas.max_price_per_unit` = current L2 gas price (passes stateful threshold check)
   - `l1_gas.max_price_per_unit` = 0
   - `l1_data_gas.max_price_per_unit` = 0
   - `l2_gas.max_amount` = sufficient for the call
   - Balance covers `l2_gas.max_amount * l2_gas.max_price_per_unit` (so `verify_can_pay_committed_bounds` passes)
3. Observe: `StatelessTransactionValidator::validate` passes (L2 price ≥ `min_gas_price`, total fee > 0 because L2 component is non-zero). [5](#0-4) 
4. Observe: `StatefulTransactionValidator::validate_resource_bounds` passes (L2 price ≥ threshold; L1 and L1-data prices are never read). [6](#0-5) 
5. Transaction is admitted to the mempool.
6. When the sequencer calls `perform_pre_validation_stage`, `check_fee_bounds` compares `l1_gas.max_price_per_unit = 0` against the actual L1 gas price (e.g., 100 fri) and raises `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }`. [7](#0-6) 
7. The transaction fails at pre-validation; the sequencer has wasted resources on a transaction that was guaranteed to fail at the moment of admission.

        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)

                gas_price: resource_bounds.l2_gas.max_price_per_unit,

        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {

                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
