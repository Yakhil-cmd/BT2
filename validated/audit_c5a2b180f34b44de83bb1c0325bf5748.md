### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield — (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract iterates over every balance-change checkpoint in a delegator's `pool_member_epoch_balance` trace since the last claim. The trace grows by one entry per epoch in which the delegator makes a balance change. Because there is no cap on trace length and no partial-claim mechanism, a delegator who accumulates enough entries without claiming will eventually be unable to call `claim_rewards` at all — permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}

If `claim_rewards` is never called (or is called infrequently), the loop must traverse every accumulated entry on the next invocation. Each iteration performs at least one storage read (`pool_member_trace.at`) and a `find_sigma` lookup against the cumulative rewards trace. Starknet transactions are bounded by a finite step budget (~10 M steps). Once the trace is large enough, every `claim_rewards` attempt reverts with out-of-resources, and there is no partial-claim escape hatch.

---

### Impact Explanation

A delegator whose trace grows beyond the per-transaction step budget can never successfully call `claim_rewards`. Because `entry_to_claim_from` is only advanced inside `claim_rewards`, and because there is no mechanism to split the computation across multiple transactions, the delegator's accumulated yield is permanently inaccessible. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The cost per epoch is one `add_to_delegation_pool` call with a minimal token amount (cheap on Starknet L2). With K = 2 and epochs measured in hours, an attacker can accumulate thousands of entries over months. The code itself acknowledges the loop is unbounded. A legitimate delegator who simply does not claim for an extended period faces the same outcome without any malicious intent. [5](#0-4) 

---

### Recommendation

1. **Cap trace length**: Enforce a maximum number of unclaimed balance-change entries per pool member (e.g., reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if `pool_member_trace_length - entry_to_claim_from` exceeds a safe bound).
2. **Partial-claim mechanism**: Allow `claim_rewards` to accept a `max_entries` parameter so the loop can be split across multiple transactions, advancing `entry_to_claim_from` incrementally.
3. **Auto-claim on balance change**: Force a reward settlement (advancing `entry_to_claim_from`) whenever a balance-change operation is performed, keeping the pending window bounded to at most one epoch.

---

### Proof of Concept

1. Delegator enters pool with `reward_address = victim`.
2. Each epoch, delegator calls `add_to_delegation_pool(amount: 1)` — one new checkpoint is appended to `pool_member_epoch_balance` at `current_epoch + K`.
3. Neither the delegator nor the victim calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs (N large enough that N loop iterations × per-iteration step cost > Starknet step budget), any call to `claim_rewards` for this pool member reverts with out-of-resources.
5. The victim's accumulated yield is permanently frozen; there is no recovery path because every attempt to claim re-enters the same loop from `entry_to_claim_from = 0`. [6](#0-5) [7](#0-6)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
