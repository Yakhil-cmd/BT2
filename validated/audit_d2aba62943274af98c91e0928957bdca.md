### Title
`from_shard_id` Label Not Cryptographically Bound to Merkle Proof Position in `set_state_header` Breaks Pigeonhole Invariant, Enabling Receipt Omission After State Sync — (`chain/chain/src/state_sync/adapter.rs`)

---

### Summary

`set_state_header` relies on the Pigeonhole principle to guarantee that all K included shards' outgoing receipts are represented. The uniqueness check on `from_shard_id` (step 4d) is the sole guard for this invariant, but `from_shard_id` is never cryptographically bound to the position of the `RootProof` in the block's outgoing-receipts Merkle tree. A malicious peer can serve a `ShardStateSyncResponseHeader` with K distinct `from_shard_id` labels but two entries sharing the same `RootProof`, passing every check while hiding one shard's receipts entirely.

---

### Finding Description

**Construction (honest server) — `adapter.rs` lines 198–216:**

For each `ReceiptProof`, the server derives the `RootProof` by:
1. `from_shard_index = prev_shard_layout.get_shard_index(*from_shard_id)` [1](#0-0) 
2. `root_proof = *block.chunks()[from_shard_index].prev_outgoing_receipts_root()` [2](#0-1) 

So on the honest path, `RootProof[j]` is the Merkle root of the chunk whose shard index matches `from_shard_id` of `receipt_proofs[j]`.

**Verification (`set_state_header`) — `adapter.rs` lines 475–502:**

```
for (j, receipt_proof) in receipt_proofs.iter().enumerate() {
    let ShardProof { from_shard_id, proof, .. } = shard_proof;
    // 4d — uniqueness only
    visited_shard_ids.insert(*from_shard_id);
    let RootProof(root, block_proof) = &root_proofs()[i][j];   // indexed by j, not from_shard_id
    let receipts_hash = hash_borsh(ReceiptList(shard_id, receipts)); // uses dest shard_id, NOT from_shard_id
    verify_path(*root, proof, &receipts_hash);                  // 4e
    verify_path(block_receipts_root, block_proof, root);        // 4f
}
``` [3](#0-2) 

Three structural gaps:

| Gap | Detail |
|-----|--------|
| `root_proofs()[i][j]` is indexed by **position** `j`, not by `from_shard_id` | An attacker controls which `RootProof` sits at position `j` |
| `receipts_hash` is `hash(dest_shard_id, receipts)` — `from_shard_id` is absent | The Merkle leaf does not encode which source shard produced the receipts |
| No check that `root` equals `block.chunks()[from_shard_index].prev_outgoing_receipts_root()` | The verifier cannot tell whether `root` belongs to shard 0 or shard 1 |

**Concrete attack (K = 2 included chunks, shard 0 → R0, shard 1 → R1):**

Attacker constructs:

```
receipt_proofs[0]: from_shard_id=0, receipts=R0, proof=P0   (valid)
receipt_proofs[1]: from_shard_id=1, receipts=R0, proof=P0   (copy of slot 0, different label)

root_proofs[0]: RootProof(root_0, BP0)   (valid path: block_root → root_0)
root_proofs[1]: RootProof(root_0, BP0)   (identical copy)
```

Check-by-check:
- **4c** `receipt_proofs.len() == root_proofs.len() == chunks_included() == 2` ✓ [4](#0-3) 
- **4d** `{0, 1}` are distinct → `visited_shard_ids` accepts both ✓ [5](#0-4) 
- **4e** `verify_path(root_0, P0, hash(shard_id, R0))` — same call for both slots ✓ [6](#0-5) 
- **4f** `verify_path(block_root, BP0, root_0)` — same call for both slots ✓ [7](#0-6) 

`set_state_header` stores the header. `set_state_finalize` then calls `collect_receipts_from_response`, which flat-maps all `ReceiptProof.receipts` fields without deduplication: [8](#0-7) 

Result: R0 appears twice in the receipt list; R1 is absent. `apply_chunk` is called with this corrupted input. [9](#0-8) 

---

### Impact Explanation

The syncing node applies the chunk with a receipt set that omits all receipts from shard 1 (R1). Every effect of R1 — token transfers, function-call side-effects, promise resolutions — is permanently absent from the node's state trie. The resulting state root diverges from the canonical chain. Any subsequent block the node processes will produce a wrong `ChunkExtra`, causing the node to be unable to participate correctly in the network until it re-syncs from scratch.

The exact corrupted value is the `prev_state_root` written by `apply_chunk_postprocessing` after `set_state_finalize`, which will differ from the canonical state root for that shard at that height.

---

### Likelihood Explanation

Exploiting this requires a malicious peer to serve a crafted `ShardStateSyncResponseHeader`. No validator, block-producer, or privileged role is needed — any node in the network can respond to state-sync requests. The attacker only needs public on-chain data (block headers, chunk roots, Merkle paths) to construct valid `root_0` / `BP0` / `P0` values. The crafted header is accepted without error, so there is no rate-limit or disconnect trigger. The attack is targeted: only nodes performing state sync against the malicious peer are affected.

---

### Recommendation

After step 4d, verify that `root` in `RootProof(root, block_proof)` equals the `prev_outgoing_receipts_root` of the chunk whose shard index corresponds to `from_shard_id`. Concretely, inside the inner loop, after resolving `from_shard_id` to `from_shard_index`, assert:

```rust
let expected_root = block.chunks()[from_shard_index].prev_outgoing_receipts_root();
if root != expected_root {
    return Err(Error::Other("set_shard_state failed: root_proof root mismatch".into()));
}
```

This binds the `RootProof` cryptographically to the shard identified by `from_shard_id`, restoring the Pigeonhole invariant. The block object is already available during `compute_state_response_header`; the verifier side needs to either fetch it or encode the expected root into the header in a verifiable way.

### Proof of Concept

```rust

// 1. Run a two-shard localnet for one epoch so block B has chunks_included=2.
// 2. Obtain a legitimate ShardStateSyncResponseHeader H for shard 1 from node 0.

//    - Copy receipt_proofs[0] into receipt_proofs[1], changing only from_shard_id to shard 1's id.
//    - Copy root_proofs[0] into root_proofs[1] verbatim.
// 4. Call node1.chain.state_sync_adapter.set_state_header(shard_id, sync_hash, H_mutated).
// 5. Assert Ok(()) — the header is accepted despite the duplicate RootProof.
// 6. Call set_state_finalize and observe that receipts from shard 0 appear twice
//    and receipts from shard 1 are absent from the applied receipt list.
```

The test does not require any cryptographic forgery: all Merkle paths are reused verbatim from the honest header. The only change is the `from_shard_id` label in slot 1 and the duplication of `root_proofs[0]` into `root_proofs[1]`.
