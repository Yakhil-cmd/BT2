### Title
STRK Rewards Transferred to Pool Contract But Not Recorded When Pool Balance Is Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a delegation pool's total balance is below `min_delegation_for_rewards`, the staking contract still calculates non-zero `pool_rewards` and transfers STRK tokens to the pool contract. However, the pool's `compute_rewards_per_unit` returns zero in this case, so the cumulative rewards trace is not updated. The transferred STRK tokens are permanently stuck in the pool contract with no recovery mechanism.

---

### Finding Description

The reward flow involves two steps in `update_pool_rewards` (`src/staking/staking.cairo`):

1. STRK tokens are unconditionally sent to the pool contract via `send_rewards_to_delegation_pool`.
2. The pool's `update_rewards_from_staking_contract` is called to record the rewards in the cumulative trace. [1](#0-0) 

Inside `update_rewards_from_staking_contract`, the pool calls `compute_rewards_per_unit`: [2](#0-1) 

`compute_rewards_per_unit` explicitly returns zero when `total_stake < min_delegation_for_rewards`: [3](#0-2) 

The code comment on this function acknowledges the consequence:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [4](#0-3) 

The `min_delegation_for_rewards` values are:
- **STRK**: `10^18` (1 STRK) — set in `STRK_CONFIG`
- **BTC (d decimals)**: `10^(d-5)` — e.g., 1000 satoshis for 8-decimal BTC [5](#0-4) [6](#0-5) 

The staking contract calculates `pool_rewards` using the normalized (18-decimal) pool balance, which can be non-zero even when the native-decimal balance is below `min_delegation_for_rewards`. For example, a STRK pool with `5 × 10^17` fractions (0.5 STRK) produces non-zero `pool_rewards_including_commission` in `calculate_staker_pools_rewards`, so STRK is sent to the pool. But `pool_balance.to_native_amount(18) = 5 × 10^17 < 10^18 = min_delegation_for_rewards`, so `compute_rewards_per_unit` returns 0 and the cumulative trace is not updated. [7](#0-6) 

The cumulative trace update becomes `last + 0 = last` — no change — while the STRK tokens now reside in the pool contract with no path to recovery. [8](#0-7) 

---

### Impact Explanation

STRK tokens transferred to the pool contract when `pool_balance < min_delegation_for_rewards` are permanently unclaimable. The pool contract has no admin recovery function; the only egress for STRK is `claim_rewards`, which is gated on the cumulative trace. Since the trace was not updated, the tokens are frozen forever.

This constitutes **permanent freezing of unclaimed yield** — a High-severity impact per the allowed scope.

---

### Likelihood Explanation

The existing test `delegate_min_strk_for_rewards_flow_test` confirms that delegating `min_for_rewards - 1` (i.e., just below the threshold) is a valid, non-reverting operation: [9](#0-8) 

Similarly, `PoolWithMinBtcFlow` delegates `TEST_MIN_BTC_FOR_REWARDS - 1` and confirms pool rewards are zero while staker rewards are non-zero — confirming STRK was sent to the pool but not credited: [10](#0-9) 

Any unprivileged delegator can trigger this by delegating any amount below `min_delegation_for_rewards`. The threshold is low enough (0.5 STRK, or 999 satoshis for 8-decimal BTC) that this can occur accidentally or deliberately. **Likelihood: Medium.**

---

### Recommendation

Align the staking contract's decision to send rewards with the pool's ability to record them. Before sending STRK to the pool, check whether `pool_balance.to_native_amount(decimals) >= min_delegation_for_rewards`. If not, either skip the transfer entirely or redirect the unrecordable rewards elsewhere (e.g., back to the staker or to a protocol treasury). Alternatively, add a recovery function in the pool contract that allows the staking contract to reclaim unrecorded STRK.

---

### Proof of Concept

1. Staker stakes and opens a STRK delegation pool.
2. Delegator calls `enter_delegation_pool` with `amount = STRK_CONFIG.min_for_rewards - 1` (e.g., `10^18 - 1` fractions). This succeeds — no minimum delegation check prevents it.
3. Staker attests. `_update_rewards` → `calculate_staker_pools_rewards` computes non-zero `pool_rewards` (since `pool_balance_curr_epoch.to_amount_18_decimals()` is non-zero).
4. `update_pool_rewards` sends `pool_rewards` STRK to the pool contract.
5. `update_rewards_from_staking_contract` is called; `compute_rewards_per_unit` returns 0 because `pool_balance = 10^18 - 1 < 10^18 = min_delegation_for_rewards`.
6. Cumulative trace is updated with `last + 0 = last`.
7. Delegator calls `claim_rewards` → receives 0.
8. The STRK tokens sent in step 4 remain in the pool contract permanently, with no mechanism to recover them. [11](#0-10) [1](#0-0)

        /// Compute the rewards for the pool trace.
        ///
        /// `staking_rewards` is in `STRK_DECIMALS` decimals.
        /// `total_stake` is in the contract's token decimals.
        /// **Note**: Delegation rewards lost when pool balance is less than
        /// `min_delegation_for_rewards`. The staking contract continues to forward
        /// `pool_rewards` to the pool contract even in this case.
        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
