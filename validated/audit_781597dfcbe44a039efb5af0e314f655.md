### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Unauthorized Invoke Transactions into Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the account's `__validate__` entry point — and therefore all signature verification — for any invoke transaction with `nonce=1` targeting an account that has **any** transaction in the mempool, regardless of who submitted the invoke. An attacker can submit an invoke transaction with an arbitrary (invalid) signature for any victim account that has a pending `deploy_account` in the mempool, and the gateway will admit it without verifying the signature.

### Finding Description

The `skip_stateful_validations` function returns `true` (skip validation) when all four conditions hold:

1. The transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags` with `validate: !skip_validate = false`, so the blockifier's `validate_tx` path returns `Ok(None)` immediately without calling the account's `__validate__` entry point: [2](#0-1) [3](#0-2) 

The authorization check at condition 4 only asks whether the account has **any** transaction in the mempool pool or recent-block state: [4](#0-3) 

It does **not** verify that the caller of the incoming invoke is the same party who submitted the `deploy_account`. The mempool's `validate_tx` (called earlier in `run_pre_validation_checks`) only checks for duplicate hashes and nonce ordering — it never inspects the signature: [5](#0-4) 

The full pre-validation chain is:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
```

None of these three steps verify the signature when `skip_validate` ends up `true`. [6](#0-5) 

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an invoke transaction carrying an arbitrary (invalid) signature into the mempool for any victim account that has a pending `deploy_account`. Concrete consequences:

1. **Nonce-slot squatting**: The attacker's invalid invoke occupies the `(victim_address, nonce=1)` slot. The victim's legitimate first invoke must then win a fee-escalation race to displace it.
2. **Batcher resource waste**: The batcher pulls the invalid transaction, executes it with `validate: true` (set by `new_for_sequencing`), and only then rejects it after running the `__validate__` entry point — consuming CPU and gas accounting resources.
3. **Admission invariant broken**: The gateway's stated invariant is that only transactions whose signatures pass the account's `__validate__` logic are admitted. That invariant is violated for every account in the deploy-account-pending state. [7](#0-6) 

### Likelihood Explanation

- **Unprivileged trigger**: Any external caller can submit an `RpcTransaction` to the gateway's `add_tx` endpoint.
- **Observable precondition**: The victim's `deploy_account` is visible in the mempool (P2P gossip or direct RPC query).
- **Deterministic**: The four conditions in `skip_stateful_validations` are fully attacker-controllable once the victim's `deploy_account` is in the mempool.
- **No cost to attacker**: A failed `__validate__` at the batcher does not charge the attacker; the attacker's only cost is the RPC call.

### Recommendation

The `skip_stateful_validations` path should not bypass signature verification entirely. Two complementary mitigations:

1. **Require a deploy-account hash witness**: Accept the skip only when the caller supplies the hash of the pending `deploy_account` transaction (as `native_blockifier`'s `PyValidator` already does via `deploy_account_tx_hash`). The gateway should require this field and verify it matches a `deploy_account` in the mempool **from the same sender**. [8](#0-7) 

2. **Restrict `account_tx_in_pool_or_recent_block` to deploy-account transactions**: The current check returns `true` for any transaction type. It should be narrowed to only match a pending `deploy_account` for the given address.

### Proof of Concept

```
1. Victim broadcasts:
     deploy_account { sender_address: A, nonce: 0, signature: <valid> }

2. Attacker observes A in the mempool via account_tx_in_pool_or_recent_block.

3. Attacker broadcasts:
     invoke { sender_address: A, nonce: 1, calldata: <arbitrary>, signature: [0xff, 0xff] }

4. Gateway stateless check: passes (valid address range, resource bounds, etc.)

5. convert_rpc_tx_to_internal: computes tx_hash with chain_id — no signature check here.

6. extract_state_nonce_and_run_validations:
     account_nonce = get_nonce_from_state(A) = 0   ← A not yet on-chain
     validate_state_preconditions: nonce 1 ∈ [0, 0+max_gap] → OK
     validate_by_mempool: no duplicate, nonce gap allowed → OK
     skip_stateful_validations:
       tx.nonce()==1 ✓  account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(A) == true ✓
       → returns true (SKIP)
     run_validate_entry_point(skip_validate=true):
       ExecutionFlags { validate: false, ... }
       blockifier validate_tx → returns Ok(None) immediately
       → NO __validate__ called, NO signature checked

7. mempool.add_tx(attacker_invoke) → accepted.

8. Victim's legitimate invoke { nonce: 1, signature: <valid> } now must
   win fee-escalation against the already-queued attacker transaction.
``` [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
