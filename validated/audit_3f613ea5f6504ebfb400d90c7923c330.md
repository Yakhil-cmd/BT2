### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent NEP-141 Fund Freeze — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** calling the Aurora exit precompile, and never check the return value of that low-level `call`. If the precompile call fails for any reason (e.g., an invalid NEAR account ID supplied as `recipient`), the burn is not reverted, the exit never executes, and the corresponding NEP-141 tokens are permanently locked inside the connector — a permanent fund freeze.

---

### Finding Description

Both bridge-exit functions follow the same pattern:

**`EvmErc20.sol` — `withdrawToNear`** [1](#0-0) 

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, ...);
        // res is NEVER checked — silent failure
    }
}
```

**`EvmErc20V2.sol` — `withdrawToNear`** [2](#0-1) 

The same pattern appears in `withdrawToEthereum` in both contracts. [3](#0-2) 

The hardcoded precompile address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is the `ExitToNear` precompile, confirmed correct in the engine: [4](#0-3) 

The `ExitToNear` precompile validates the `recipient` bytes as a NEAR account ID via `parse_recipient`. If the bytes are empty, non-UTF-8, or fail NEAR account ID rules, the precompile returns `ExitError`, causing the EVM sub-call to return `0`. [5](#0-4) 

Because `res` is never checked in the assembly block, the outer Solidity call does **not** revert. The `_burn` — which already executed in the outer call frame — is permanent. The ERC-20 tokens are gone, but the `ExitToNear` precompile never schedules the NEAR-side `ft_transfer` promise, so the NEP-141 tokens remain locked in the connector forever.

Both `EvmErc20.bin` and `EvmErc20V2.bin` are the contracts deployed by the engine for every bridged NEP-141 token: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The NEP-141 tokens that back the burned ERC-20 tokens are permanently locked inside the connector contract on NEAR. No recovery path exists: the ERC-20 supply has decreased (burn succeeded) but the connector's NEP-141 balance has not decreased (exit failed silently). The affected user loses their bridged assets with no recourse.

---

### Likelihood Explanation

**Medium.** Any unprivileged EVM user who holds bridged ERC-20 tokens and calls `withdrawToNear` with:
- an empty `recipient` (`bytes("")`),
- bytes that are not valid UTF-8, or
- a string that fails NEAR account ID validation (e.g., uppercase letters, leading/trailing dots, etc.)

will trigger the silent failure. This can happen accidentally (e.g., a frontend bug encoding the recipient incorrectly) or deliberately. The `withdrawToEthereum` variant is less likely to fail because the recipient is a 20-byte Ethereum address (always structurally valid), but it shares the same unchecked-return-value pattern and would freeze funds if the NEP-141 mapping lookup ever fails.

---

### Recommendation

Add a revert inside the assembly block if the precompile call fails, **or** restructure the function to call the precompile first and only burn on success:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. A NEP-141 token is bridged to Aurora; `deploy_erc20_token` deploys an `EvmErc20` (or `EvmErc20V2`) contract and registers the NEP-141↔ERC-20 mapping. [7](#0-6) 
2. Alice holds 1000 units of the ERC-20 token.
3. Alice calls `withdrawToNear(bytes(""), 1000)` — an empty NEAR recipient.
4. `_burn(alice, 1000)` executes; Alice's ERC-20 balance drops to 0.
5. The assembly `call` invokes the `ExitToNear` precompile. `parse_recipient` receives empty bytes, `"".parse::<AccountId>()` fails, and the precompile returns `ExitError`. The sub-call returns `res = 0`.
6. `res` is not checked; the function returns normally. The transaction is mined successfully.
7. Alice's 1000 ERC-20 tokens are permanently destroyed. The connector on NEAR still holds Alice's 1000 NEP-141 tokens with no mechanism to release them. The funds are permanently frozen.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}

pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
