### Title
Missing `app_pk` Identity-Point Validation Causes Coordinator to Learn Derived Secret Directly — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

`ckd()` accepts `app_pk = G1::identity()` without any validation. When this degenerate value propagates through `compute_signature_share`, the ElGamal blinding term `y_i · A` collapses to the identity, so every participant's `C_i` equals their unblinded share `S_i = x_i · H(pk‖app_id)`. After Lagrange aggregation the coordinator's `big_c` field equals `msk · H(pk‖app_id)` — the derived secret `s` — in the clear, violating the explicit security requirement that no single MPC node can compute `s`.

---

### Finding Description

`ckd()` performs four input checks before launching the protocol: [1](#0-0) 

None of them touch `app_pk`. The value is forwarded unchanged into `compute_signature_share`:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;   // line 174
``` [2](#0-1) 

When `app_pk = G1Projective::identity()`:

- `app_pk * y.0` = `identity * y` = `identity` (the additive identity of G1)
- `big_c` = `big_s` = `x_i · H(pk‖app_id)` — the raw, unblinded share

After Lagrange normalization and coordinator aggregation: [3](#0-2) 

```
C = Σ λ_i · C_i = Σ λ_i · x_i · H(pk‖app_id) = msk · H(pk‖app_id)
Y = Σ λ_i · y_i · G1   (nonzero — random blinding points still accumulate)
```

The coordinator's `CKDOutput.big_c` is exactly `msk · H(pk‖app_id)`, the BLS signature that is the basis of the derived secret `s`. `big_y` is nonzero but irrelevant — the coordinator can read `big_c` directly without needing `a`. [4](#0-3) 

---

### Impact Explanation

The protocol's stated security requirement is explicit:

> "No single node in the MPC network should be capable of computing `s`. … Example values that should not be leaked even if a node is malicious are `s`, `msk` and private shares of other nodes." [5](#0-4) 

With `app_pk = identity`, the coordinator — a single MPC node — obtains `msk · H(pk‖app_id)` directly from `big_c`, from which `s = HKDF(msk · H(pk‖app_id))` is trivially derived. This is a **Critical** disclosure of a confidential derived secret to a single party.

---

### Likelihood Explanation

`app_pk` (`A`) originates from the on-chain CKD request submitted by the TEE app. The developer contract is responsible for verifying `A` is bound to the attestation `report_data`, but there is no documented or enforced requirement that `A ≠ identity`. The library itself is the last line of defense and performs no such check. A malicious or buggy app, or a developer contract that omits the `A ≠ identity` check, can submit `A = identity` on-chain; all MPC nodes then call `ckd()` with `app_pk = identity`, and the coordinator learns `s`. The attack requires no cryptographic break and is locally testable.

---

### Recommendation

Add an explicit rejection of the identity point in `ckd()` before the protocol is launched:

```rust
use elliptic_curve::Group;

if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::InvalidPublicKey);
}
```

Place this check alongside the existing participant-list guards in `ckd()`. [6](#0-5) 

---

    let ckd_output = check_one_coordinator_output(result, coordinator).unwrap();

The coordinator observes `big_c = msk · H(pk‖app_id)` without needing the app's secret scalar `a`, directly violating the confidentiality invariant. [7](#0-6)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
