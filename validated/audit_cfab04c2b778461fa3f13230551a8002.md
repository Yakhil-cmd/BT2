### Title
Pool Member's Remaining Balance Temporarily Frozen When Staker Exits During Partial Exit Intent - (File: `src/pool/pool.cairo`)

### Summary

When a pool member has a **partial** `exit_delegation_pool_intent` in progress and the staker subsequently calls `unstake_action()`, the pool member's remaining (non-intent) balance is transferred to the pool contract but becomes temporarily inaccessible. The `UNDELEGATE_IN_PROGRESS` guard in `undelegate_from_staking_contract_intent` blocks any new intent while `unpool_time` is set, preventing the pool member from initiating withdrawal of the remaining balance until the first intent's exit window fully expires.

---

### Finding Description

**Root cause** — `src/pool/pool.cairo`, `undelegate_from_staking_contract_intent`:

```cairo

``` [1](#0-0) 

When `staker_removed == true`, any call to `exit_delegation_pool_intent` (regardless of amount, including `0`) unconditionally reverts with `UNDELEGATE_IN_PROGRESS` if `unpool_time` is already `Some(...)`. This check is invoked on every call path through `exit_delegation_pool_intent`: [2](#0-1) 

**Vulnerable sequence:**

1. Pool member calls `exit_delegation_pool_intent(amount: X)` where `X < total_balance`.
   - `unpool_time = T1 = now + exit_wait_window`, `unpool_amount = X`, `get_last_member_balance = total - X`.

2. Staker calls `unstake_intent()` then `unstake_action()`.
   - `transfer_to_pools_when_unstake` transfers the remaining delegated balance (`total - X`) to the pool contract.
   - `set_staker_removed()` sets `staker_removed = true`. [3](#0-2) 

3. Pool member attempts `exit_delegation_pool_intent(amount: total - X)` to withdraw the remaining balance.
   - `undelegate_from_staking_contract_intent` is called.
   - `!is_staker_active()` → `true`.
   - `unpool_time.is_none()` → `false` (still `Some(T1)`).
   - **Reverts with `UNDELEGATE_IN_PROGRESS`.** [4](#0-3) 

4. The remaining balance `total - X` sits in the pool contract, unreachable until:
   - Time advances past `T1`.
   - Pool member calls `exit_delegation_pool_action()` to recover `X`.
   - Pool member calls `exit_delegation_pool_intent(total - X)` (now succeeds since `unpool_time = None`).
   - Pool member calls `exit_delegation_pool_action()` to recover `total - X`. [5](#0-4) 

The test `test_exit_delegation_pool_intent_assertions` explicitly confirms this revert path: [6](#0-5) 

---

### Impact Explanation

The pool member's remaining balance (`total - X`) is **temporarily frozen** in the pool contract for up to `exit_wait_window` (default: 1 week) beyond the point at which the staker exits. The pool member cannot initiate withdrawal of the remaining balance until the first intent's exit window expires and `exit_delegation_pool_action` is called to clear `unpool_time`. This matches the allowed impact: **High — Temporary freezing of funds**.

---

### Likelihood Explanation

The scenario requires:
1. A pool member to have called `exit_delegation_pool_intent` with a **partial** amount (a common pattern, e.g., withdrawing half of a position).
2. The staker to call `unstake_intent()` + `unstake_action()` while the pool member's intent is still pending.

Both actions are unprivileged and routine. The pool member has no way to anticipate or prevent the staker's exit. The staker's exit window is at least `exit_wait_window` (1 week), during which the pool member's intent is likely still active. **Likelihood: Medium.**

---

### Recommendation

When `staker_removed == true` and the pool member already has an active intent, allow the pool member to **extend** their intent to cover the full remaining balance. Since the staker is already removed, no call to `remove_from_delegation_pool_intent` on the staking contract is needed — the remaining balance is already in the pool contract. The guard should be relaxed to permit updating `unpool_amount` upward (or to the full balance) when the staker is inactive, without requiring the existing intent to be cleared first.

Alternatively, allow `exit_delegation_pool_intent(0)` to cancel an existing intent even when `staker_removed == true`, so the pool member can immediately re-intent for the full balance.

---

### Proof of Concept

```
1. Pool member delegates 100 tokens.
2. Pool member calls exit_delegation_pool_intent(50).
   → unpool_amount = 50, unpool_time = now + 1 week, get_last_member_balance = 50.
3. Staker calls unstake_intent() + unstake_action().
   → Pool contract receives 50 tokens (remaining balance).
   → staker_removed = true.
4. Pool member calls exit_delegation_pool_intent(50) to exit remaining balance.
   → REVERTS: "Undelegate from pool in progress, pool member is in an exit window"
5. Pool member must wait 1 week, call exit_delegation_pool_action() to get first 50,
   then call exit_delegation_pool_intent(50) + exit_delegation_pool_action() to get the rest.
   → The second 50 tokens are frozen for up to 1 week beyond the first intent's expiry.
```

        fn exit_delegation_pool_action(
            ref self: ContractState, pool_member: ContractAddress,
        ) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let unpool_time = pool_member_info
                .unpool_time
                .expect_with_err(GenericError::MISSING_UNDELEGATE_INTENT);
            assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberExitAction {
                        pool_member, unpool_amount: pool_member_info.unpool_amount,
                    },
                );

            // Perform removal action in the staking contract, receiving funds if needed.
            // Note that if the intent was done after the staker was removed (unstake_action),
            // the funds will already be in the pool contract, and the following call will do
            // nothing.
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_action(identifier: pool_member.into());

            let unpool_amount = pool_member_info.unpool_amount;
            pool_member_info.unpool_amount = Zero::zero();
            pool_member_info.unpool_time = Option::None;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer delegated amount to the pool member.
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());

            unpool_amount
        }

        fn undelegate_from_staking_contract_intent(
            self: @ContractState, pool_member: ContractAddress, amount: Amount,
        ) -> Timestamp {
            if !self.is_staker_active() {
                // Don't allow intent if an intent is already in progress and the staker is erased.
                // Avoid the following flow:
                // 1. Member intent - moves member balance from the staker's `pool_amount` to
                // `UndelegateIntentKey`.
                // 2. Staker intent.
                // 3. Staker action - transfer the `pool_amount` from the staker to the pool
                // contract.
                // 4. Member intent - here it is no longer possible to move balance between the
                // `pool_amount` and the `UndelegateIntentKey`.
                assert!(
                    self.internal_pool_member_info(:pool_member).unpool_time.is_none(),
                    "{}",
                    Error::UNDELEGATE_IN_PROGRESS,
                );
                return Time::now();
            }
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            let staker_address = self.staker_address.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_intent(
                    :staker_address, identifier: pool_member.into(), :amount,
                )
        }
