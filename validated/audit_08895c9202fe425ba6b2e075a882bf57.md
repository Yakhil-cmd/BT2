### Title
Missing Zero Address Validation for `reward_address` Allows Permanent Loss of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary
Both `change_reward_address` in `staking.cairo` and `change_reward_address` in `pool.cairo` accept a zero address as a valid `reward_address` without any check. Once set, all subsequent reward claims transfer tokens to address `0x0`, permanently burning the staker's or pool member's accumulated yield.

---

### Finding Description

`change_reward_address` in `staking.cairo` performs only one validation — that the new address is not a registered token address — but does not check for the zero address: [1](#0-0) 

The same omission exists in `pool.cairo`: [2](#0-1) 

The initial `stake` entry point has the same gap — it checks `REWARD_ADDRESS_IS_TOKEN` and `OPERATIONAL_IS_TOKEN` but never asserts `reward_address.is_non_zero()`: [3](#0-2) 

Likewise, `enter_delegation_pool` in `pool.cairo` only guards against the token address, not zero: [4](#0-3) 

The codebase already defines a `ZERO_ADDRESS` error and uses it in other contexts (e.g., `add_token`), confirming the pattern is known but was not applied here: [5](#0-4) 

---

### Impact Explanation

When `claim_rewards` is called on a staker whose `reward_address` is `0x0`, the internal `send_rewards_to_staker` helper transfers the accumulated STRK balance to address zero. On Starknet, address `0x0` is not a burner sink with a recovery path — the tokens are permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield**, which maps to the **High** impact tier.

The same applies to pool members: `claim_rewards` in `pool.cairo` transfers to `pool_member_info.reward_address` without any guard: [6](#0-5) 

---

### Likelihood Explanation

The call is permissionless for any active staker or pool member. No privileged role is required. A user who mistakenly passes `0` (a common default/uninitialized value in Cairo) to `change_reward_address` will silently succeed, with no on-chain revert to warn them. The error is irreversible once rewards are claimed.

---

### Recommendation

Add a non-zero assertion at the top of both `change_reward_address` implementations and at the reward-address assignment sites in `stake` and `enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `ZERO_ADDRESS` guard already used in `add_token` and is consistent with the protocol's own error catalogue.

---

### Proof of Concept

1. A staker calls `staking.change_reward_address(reward_address: 0)`.
2. The only check — `does_token_exist(token_address: 0)` — returns `false` (zero is not a registered token), so the call succeeds and `staker_info.reward_address` is written as `0x0`.
3. Any subsequent call to `staking.claim_rewards(staker_address)` invokes `send_rewards_to_staker`, which executes `token_dispatcher.checked_transfer(recipient: 0x0, amount: unclaimed_rewards)`.
4. The transfer succeeds on-chain; the STRK tokens are sent to address `0x0` and are permanently unrecoverable.
5. The identical path exists for pool members via `pool.change_reward_address(0)` followed by `pool.claim_rewards(pool_member)`. [7](#0-6) [8](#0-7)

            // Calculate rewards and update entry_to_claim_from.
