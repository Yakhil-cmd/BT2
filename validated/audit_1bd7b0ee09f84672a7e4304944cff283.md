### Title
Missing Active-Token Validation in `set_open_for_delegation` and `add_stake_from_pool` Allows Delegation to Disabled-Token Pools — (File: src/staking/staking.cairo)

---

### Summary

`set_open_for_delegation` only validates that a BTC token *exists* (`does_token_exist`) but never checks whether it is currently *active* (`is_active_token`). This allows any staker to open a delegation pool for a disabled BTC token. Pool members who subsequently delegate into that pool have their funds transferred to the staking contract and locked for the full exit-wait window, while earning zero rewards, because disabled tokens are excluded from all staking-power and reward calculations.

---

### Finding Description

**Root cause — `set_open_for_delegation` (src/staking/staking.cairo line 551):**

```cairo
fn set_open_for_delegation(
    ref self: ContractState, token_address: ContractAddress,
) -> ContractAddress {
    self.general_prerequisites();
    let staker_address = get_caller_address();
    let staker_info = self.internal_staker_info(:staker_address);
    let staker_pool_info = self.staker_pool_info.entry(staker_address);
    assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
    assert!(self.does_token_exist(:token_address), "{}", Error::TOKEN_NOT_EXISTS);
    // ← NO is_active_token check here
    assert!(
        !staker_pool_info.has_pool_for_token(:token_address),
        "{}",
        Error::STAKER_ALREADY_HAS_POOL,
    );
    ...
}
``` [1](#0-0) 

The function checks `does_token_exist` (line 551) but never calls `is_active_token`, which is the function that enforces the epoch-gated active/disabled state for BTC tokens. [2](#0-1) 

**Root cause — `add_stake_from_pool` (src/staking/staking.cairo lines 1003–1046):**

After the pool is created, pool members call `enter_delegation_pool` or `add_to_delegation_pool` on the pool contract, which internally calls `add_stake_from_pool` on the staking contract. That function also performs no active-token check:

```cairo

``` [3](#0-2) 

**Why disabled-token funds earn no rewards:**

Reward calculations in `get_staker_total_strk_btc_balance_at_epoch` and `get_total_staking_power_at_epoch` explicitly gate on `is_active_token` / `is_btc_active`, so any stake held under a disabled token is silently excluded from staking power and reward distribution: [4](#0-3) [5](#0-4) 

**Protocol-confirmed behavior:**

The existing test `test_set_open_for_delegation_with_disabled_btc_token` (lines 2480–2519) explicitly demonstrates that a pool can be opened for a disabled token with no error, confirming the missing guard is not a test artifact: [6](#0-5) 

---

### Impact Explanation

An unprivileged staker can open a delegation pool for a disabled BTC token. Any pool member who calls `enter_delegation_pool` or `add_to_delegation_pool` on that pool:

1. Has their BTC transferred to the staking contract (real funds locked).
2. Earns **zero rewards** for the entire duration their funds remain staked, because the disabled token is excluded from staking-power calculations.
3. Must wait the full `exit_wait_window` before recovering principal via `exit_delegation_pool_intent` → `exit_delegation_pool_action`.

This constitutes **temporary freezing of funds** (funds are locked for the exit-wait window with no yield) and **theft of unclaimed yield** (yield that would have accrued on an active-token pool is permanently lost for that period). Both are in the allowed impact scope.

---

### Likelihood Explanation

Two realistic paths exist:

1. **Malicious staker (no privileged access required):** After a security agent disables a BTC token, any staker can immediately call `set_open_for_delegation` with the disabled token address. The staker then advertises the pool; unsuspecting delegators enter and lose yield.

2. **Token disabled after pool creation:** A pool is legitimately created while the token is active. The security agent later disables the token. Pool members who are unaware of the status change continue calling `add_to_delegation_pool`, locking additional funds with no reward accrual.

Both paths are reachable by an unprivileged actor (pool member / staker) without any privileged key or bridge compromise.

---

### Recommendation

Add an `is_active_token` guard in `set_open_for_delegation`:

```cairo
assert!(self.does_token_exist(:token_address), "{}", Error::TOKEN_NOT_EXISTS);
assert!(
    self.is_active_token(:token_address, epoch_id: self.get_current_epoch()),
    "{}",
    Error::TOKEN_NOT_ACTIVE,
);
```

Add the same guard in `add_stake_from_pool` after resolving `token_address`:

```cairo
assert!(
    self.is_active_token(:token_address, epoch_id: self.get_current_epoch()),
    "{}",
    Error::TOKEN_NOT_ACTIVE,
);
```

This mirrors the check already present in `get_total_stake_for_token` (lines 824–829). [7](#0-6) 

---

### Proof of Concept

```
1. Security agent calls disable_token(btc_token_address).
   → btc_tokens[btc_token_address] = (epoch_plus_k, false)

2. After K epochs the token is inactive (is_btc_active returns false).

3. Malicious staker calls:
     set_commission(commission)
     set_open_for_delegation(token_address: btc_token_address)
   → Succeeds; pool contract deployed for disabled token.
   (Confirmed by test_set_open_for_delegation_with_disabled_btc_token)

4. Victim pool member calls:
     pool.enter_delegation_pool(reward_address, amount)
   → Pool calls add_stake_from_pool on staking contract.
   → Funds transferred; no active-token check fires.

5. Reward update runs:
     get_staker_total_strk_btc_balance_at_epoch checks is_active_token → false
   → BTC balance excluded; victim earns 0 rewards.

6. Victim must wait exit_wait_window to recover principal,
   losing all yield for that period.
```
