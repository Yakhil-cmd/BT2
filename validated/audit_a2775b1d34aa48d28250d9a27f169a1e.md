### Title
Delegators Cannot Set Maximum Commission Threshold, Enabling Staker to Suddenly Increase Commission to 100% — (`File: src/staking/staking.cairo`)

---

### Summary

The `set_commission_commitment` + `set_commission` flow in `src/staking/staking.cairo` allows a staker to increase their commission rate up to 100% with no notice period and no mechanism for delegators to specify a maximum acceptable commission. Delegators who joined a pool at a low commission rate have no on-chain protection against a sudden commission spike that diverts all future yield to the staker.

---

### Finding Description

Without an active commitment, `set_commission` enforces that commission can only decrease: [1](#0-0) 

However, once a staker calls `set_commission_commitment` with a high `max_commission` (up to `COMMISSION_DENOMINATOR` = 10 000, i.e. 100%), the guard changes: the staker may set commission to **any value ≤ `max_commission`**, including values far above the current rate: [2](#0-1) 

The `set_commission_commitment` function itself imposes no minimum notice period — the commitment can expire as soon as `expiration_epoch = current_epoch + 1`, meaning the staker can commit and immediately raise commission within the same epoch: [3](#0-2) 

The developers explicitly acknowledged this residual risk in a code comment: [4](#0-3) 

On the delegator side, neither `enter_delegation_pool` nor `add_to_delegation_pool` accepts a `max_commission` parameter, so there is no on-chain slippage guard: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When commission is raised to 100% (`COMMISSION_DENOMINATOR`), the entire reward stream that would have been distributed to delegators is instead routed to the staker's reward address. Delegators who do not notice the `CommissionCommitmentSet` / `CommissionChanged` events in time continue to stake while earning zero yield. Even after noticing, they must call `exit_delegation_pool_intent` and wait out the full exit window before recovering their principal; during that window rewards are paused, so they earn nothing regardless. The net effect is permanent loss of yield for the period between the commission spike and the delegator's successful exit.

---

### Likelihood Explanation

**Medium.**

Any registered staker — an unprivileged role that any user can assume — can execute this two-step sequence (`set_commission_commitment` → `set_commission`) atomically within a single epoch. No leaked key, governance vote, or external dependency is required. The only friction is that the staker must already have an active pool with delegators, which is the normal operating state for any validator seeking delegation rewards. A rational but adversarial staker has a direct financial incentive to execute this once a large enough delegated balance has accumulated.

---

### Recommendation

1. **Delegator-side slippage guard**: Add a `max_commission: Commission` parameter to `enter_delegation_pool` and `add_to_delegation_pool`. Revert if the pool's current commission exceeds the caller's stated maximum at the time of the call.
2. **Notice period for commission increases**: Require that a commission increase (via commitment) only takes effect after a mandatory delay (e.g., one full epoch after `set_commission_commitment` is called), giving delegators time to observe the `CommissionCommitmentSet` event and exit before the higher rate applies.
3. **Cap on commitment range**: Restrict `max_commission` in `set_commission_commitment` to be no more than `current_commission + delta` for some protocol-defined `delta`, preventing a single commitment from enabling a jump to 100%.

---

### Proof of Concept

```
// Step 1 – Staker creates pool at 1% commission (commission = 100).
staking.stake(amount, pool_enabled: true, commission: 100);

// Step 2 – Delegators join, expecting 1% commission forever
//           (no max_commission guard in enter_delegation_pool).
pool.enter_delegation_pool(reward_address, large_amount);

// Step 3 – Staker sets a commitment allowing up to 100% commission,
//           expiring next epoch (minimum allowed).
staking.set_commission_commitment(
    max_commission: 10_000,          // 100%
    expiration_epoch: current_epoch + 1,
);

// Step 4 – Staker immediately raises commission to 100%.
//           update_commission allows this because commitment is active
//           and 10_000 <= max_commission (10_000).
staking.set_commission(commission: 10_000);

// Result: all future pool rewards flow to the staker.
//         Delegators earn 0% yield until they notice and exit.
//         Exit intent pauses rewards, so they earn nothing during
//         the exit wait window either.
```
