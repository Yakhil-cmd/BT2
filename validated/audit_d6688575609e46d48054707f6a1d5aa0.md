### Title
Unrestricted `disable_rewards` Parameter in `update_rewards` Allows Permanent Denial of Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is callable by any unprivileged user and accepts a user-controlled `disable_rewards` boolean. When called with `disable_rewards = true`, the function unconditionally updates the global `last_reward_block` lock but distributes no rewards. Because `last_reward_block` prevents any second call to `update_rewards` within the same block, an attacker can permanently deny all stakers their consensus block rewards by calling this function once per block.

### Finding Description
`update_rewards` is exposed through the `IStakingRewardsManager` public interface with no access control beyond `general_prerequisites`, which only checks the contract is not paused and the caller is non-zero. [1](#0-0) 

The critical flaw is that `last_reward_block` is written **unconditionally** before the `disable_rewards` branch: [2](#0-1) 

This means:
1. An attacker calls `update_rewards(any_active_staker, disable_rewards: true)` on every block.
2. `last_reward_block` is set to the current block number.
3. The function returns immediately — no rewards are distributed.
4. Any legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The rewards for that block are permanently lost and cannot be recovered. The attacker only needs to supply a valid active staker address (any staker with non-zero STRK balance satisfies the guard). [4](#0-3)
