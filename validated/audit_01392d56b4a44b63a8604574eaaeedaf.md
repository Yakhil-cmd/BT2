### Title
Insecure Reliance on `balance_of` in `request_funds` Suppresses L1 Mint Requests, Causing Temporary Freeze of Unclaimed Yield - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `request_funds` internal function in `RewardSupplier` uses `token_dispatcher.balance_of(account: get_contract_address())` to determine whether to send a mint request to the L1 reward supplier. Because this reads the live on-chain token balance rather than an internally tracked value, any unprivileged actor can donate STRK tokens directly to the contract to artificially inflate the perceived credit, suppressing L1 mint requests. As `unclaimed_rewards` continues to grow through normal attestation activity, the contract's actual token balance eventually falls below the amount owed, causing `claim_rewards` → `checked_transfer` to revert and temporarily freezing all unclaimed yield for stakers and delegators.

---

### Finding Description

`request_funds` is called every time `update_unclaimed_rewards_from_staking_contract` is invoked (which happens on every successful attestation). Its job is to decide whether the contract needs to request more STRK from L1:

```cairo
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // ← manipulable
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount; // ← inflated by donation
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);
    if credit < debit + threshold {                     // ← guard bypassed
        // ... send L1 mint request
    }
}
``` [1](#0-0) 

An attacker calls `IERC20.transfer(reward_supplier_address, D)` directly, donating `D` STRK tokens. This raises `balance` by `D`, raises `credit` by `D`, and makes the guard condition `credit < debit + threshold` harder to satisfy. The contract therefore skips the L1 mint request for one or more epochs.

Meanwhile, `unclaimed_rewards` is tracked correctly via internal accounting and keeps growing with each attestation. Once `unclaimed_rewards` exceeds the contract's actual token balance (which was never replenished from L1 because the guard was suppressed), the `checked_transfer` inside `claim_rewards` reverts:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    ...
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into()); // ← reverts
}
``` [2](#0-1) 

`claim_rewards` is called from the staking contract's `claim_from_reward_supplier` helper during `_update_rewards`, which is triggered on every attestation. A revert here propagates upward and blocks reward distribution for all stakers. [3](#0-2) 

---

### Impact Explanation

**High — Temporary freezing of unclaimed yield.**

Once the donated tokens are consumed paying out rewards and the contract has not requested a sufficient top-up from L1, `claim_rewards` reverts on every subsequent attestation. No staker or delegator can receive their accrued rewards until the L1 bridge message is eventually triggered (when `unclaimed_rewards` grows large enough to re-satisfy the guard) and the L1→L2 deposit is confirmed. During this window — which can span multiple epochs — all unclaimed yield is frozen.

---

### Likelihood Explanation

**Medium.** The attack requires the attacker to spend real STRK tokens (a direct ERC-20 transfer to the contract). The cost is bounded: the attacker only needs to donate enough to keep `credit >= debit + threshold` for the desired suppression window. Given that `threshold = base_mint_amount / 2` and `base_mint_amount` is a fixed protocol parameter, the required donation is predictable and finite. Any holder of STRK with a griefing motive (e.g., a competing validator, or someone who benefits from delayed reward distribution) can execute this with no special privileges. [4](#0-3) 

---

### Recommendation

Replace the `balance_of` call in `request_funds` with an internally tracked storage variable (e.g., `internal_balance`) that is incremented only when tokens arrive via `on_receive` (the authorised StarkGate path) and decremented when `claim_rewards` transfers tokens out. The guard should then read `internal_balance` instead of the live ERC-20 balance:

```cairo
// In on_receive: self.internal_balance.write(self.internal_balance.read() + amount_u128);
// In claim_rewards: self.internal_balance.write(self.internal_balance.read() - amount);
// In request_funds:
let balance = self.internal_balance.read(); // ← not manipulable by donation
let credit = balance + l1_pending_requested_amount;
```

This ensures that unsolicited token donations cannot inflate the credit calculation and suppress legitimate L1 mint requests.

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
