### Title
Permanent ERC-20 Token Loss on Failed `ft_transfer` in Legacy Exit Path When `error_refund` Feature Is Absent — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is **not** enabled, the legacy ERC-20 exit path (flag `0x1`, no `:unwrap`/omni suffix) schedules a bare `PromiseArgs::Create(ft_transfer)` with **no callback**. If the NEP-141 `ft_transfer` promise fails — for example because the recipient account has no storage deposit — the ERC-20 tokens that were already burned on Aurora are permanently destroyed with no on-chain recovery path.

---

### Finding Description

**Step 1 — Entry point.**
Any holder of an Aurora-mirrored ERC-20 token calls `withdrawToNear(recipient, amount)` on the ERC-20 contract. The ERC-20 contract burns the caller's tokens and calls the `ExitToNear` precompile at `0xe9217bc7…` with flag `0x1`.

**Step 2 — `exit_erc20_token_to_near` legacy arm.**
Inside `exit_erc20_token_to_near`, the catch-all `_` arm (no `:unwrap`, no omni message) returns `transfer_near_args = None`: [1](#0-0) 

**Step 3 — `callback_args` evaluates to `default()`.**
Back in `run()`, `callback_args` is constructed as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // always None without the feature
    transfer_near: None,   // None from the legacy arm
};
```

`ExitToNearPrecompileCallbackArgs` derives `Default` with both fields `None`: [2](#0-1) 

So `callback_args == ExitToNearPrecompileCallbackArgs::default()` is `true`.

**Step 4 — No callback is scheduled.**
The branch at line 470 selects `PromiseArgs::Create` (no callback) instead of `PromiseArgs::Callback`: [3](#0-2) 

**Step 5 — `ft_transfer` fails silently.**
The NEP-141 `ft_transfer` call fails because the recipient has no storage deposit. Since there is no `exit_to_near_precompile_callback` scheduled, the failure is never observed by Aurora. The `refund` branch in the callback handler is never reached: [4](#0-3) 

**Step 6 — Permanent loss.**
The ERC-20 tokens were burned in Step 1. The NEP-141 tokens remain in Aurora's account. The user has no ERC-20 balance and no NEP-141 balance. There is no recovery path.

---

### Impact Explanation

This is **Critical — Permanent freezing of funds**. Any user who calls `withdrawToNear` with a recipient that has no NEP-141 storage deposit permanently loses their ERC-20 tokens. The NEP-141 tokens remain locked in Aurora's account with no mechanism to retrieve them.

---

### Likelihood Explanation

- **Reachable by any unprivileged user** — no admin access required; any ERC-20 token holder can trigger this.
- **Condition**: the `error_refund` feature must be absent from the production build. The test suite explicitly acknowledges this behavior under `#[cfg(not(feature = "error_refund"))]`: [5](#0-4) 

- **Trigger**: supplying any NEAR account ID that has not registered a storage deposit with the NEP-141 contract — a common situation for newly created accounts or accounts that have never interacted with the specific NEP-141 token.

---

### Recommendation

1. **Enable `error_refund` in all production builds** so that `refund_call_args` populates `callback_args.refund` and the callback is always scheduled.
2. Alternatively, unconditionally schedule the `exit_to_near_precompile_callback` for the ERC-20 legacy `ft_transfer` path regardless of the feature flag, so that a failed promise always triggers a refund mint.
3. Add a compile-time or runtime assertion that prevents deployment without `error_refund` when ERC-20 exit is supported.

---

### Proof of Concept

The existing test `test_exit_to_near_refund` already proves the loss: [6](#0-5) 

The test explicitly asserts that without `error_refund`, `erc20_balance` drops by `FT_EXIT_AMOUNT` while `nep_141_balance_of(aurora)` stays at `FT_TRANSFER_AMOUNT` — confirming the burned ERC-20 tokens are not refunded and the NEP-141 tokens remain stranded in Aurora's account.
