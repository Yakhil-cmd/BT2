### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State (1), Leaving Pool Permanently Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` is designed to restore pool operation after a protocol-level pause, but it unconditionally calls `setPause(1)` — the admin-paused state — rather than the active state (0). When the protocol pauses an active pool (0 → 2) and then unpauses it (2 → 1), the pool remains paused. All swaps and liquidity operations continue to revert. The pool admin must independently discover and execute a second unpause call to restore the pool to state 0.

---

### Finding Description

The pool has three pause levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`_checkNotPaused()` reverts on **any non-zero** pause level: [1](#0-0) 

The factory's `protocolUnpausePool` enforces that the current state is 2, then calls `setPause(1)`: [2](#0-1) 

State 1 is still a paused state. The `whenNotPaused` modifier on `swap`, `addLiquidity`, and `removeLiquidity` will continue to revert after the protocol "unpause": [3](#0-2) 

The factory stores no record of the pool's pre-protocol-pause state. There is no mechanism to distinguish "was active before protocol pause" from "was admin-paused before protocol pause." The transition 2 → 1 is hardcoded regardless of prior state.

This is the direct analog to the external bug: `addSession` requires `startTime > block.timestamp` but calls `newStakePeriod` which requires `startTime <= block.timestamp`, making the combined operation self-defeating. Here, `protocolUnpausePool` is intended to restore pool operation but calls `setPause(1)` which keeps the pool paused — the intent (restore operation) contradicts the outcome (still paused).

---

### Impact Explanation

After a protocol unpause, the pool is stuck in state 1. Every call to `swap`, `addLiquidity`, and `removeLiquidity` reverts with `PoolPaused`. LPs cannot withdraw liquidity. Traders cannot execute swaps. The pool is functionally dead until the pool admin independently calls `unpausePool`. If the admin is unaware, unavailable, or the admin key is lost, the pool is permanently bricked. This is broken core pool functionality causing loss of access to LP assets and swap flows.

---

### Likelihood Explanation

This occurs every time the protocol pauses a pool that was in state 0 (active) — the normal operating state — and subsequently unpauses it. This is the most common protocol-pause scenario (emergency pause of a live pool). The broken transition is deterministic and unconditional.

---

### Recommendation

Track the pre-protocol-pause state in factory storage and restore it on `protocolUnpausePool`:

```solidity
mapping(address => uint8) public prePausedState;

function protocolPausePool(address pool) external onlyOwner nonReentrant {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    prePausedState[pool] = cur; // save original state
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external onlyOwner nonReentrant {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    uint8 restoreTo = prePausedState[pool]; // restore original state
    IMetricOmmPoolFactoryActions(pool).setPause(restoreTo);
    delete prePausedState[pool];
}
```

---

### Proof of Concept

1. Pool is deployed and active: `pauseLevel == 0`.
2. Protocol calls `protocolPausePool(pool)`:
   - Guard: `cur == 0`, passes (`cur != 0 && cur != 1` is false).
   - Pool transitions: `pauseLevel = 2`.
3. Emergency resolved. Protocol calls `protocolUnpausePool(pool)`:
   - Guard: `cur == 2`, passes.
   - Pool transitions: `setPause(1)` → `pauseLevel = 1`.
4. Any user calls `swap(...)`:
   - `whenNotPaused` fires: `pauseLevel != 0` → `revert PoolPaused()`.
5. Pool admin must independently discover the issue and call `unpausePool(pool)` (1 → 0) to restore operation. If admin is unavailable, the pool is permanently paused despite the protocol having "unpaused" it. [4](#0-3) [5](#0-4) [6](#0-5)
