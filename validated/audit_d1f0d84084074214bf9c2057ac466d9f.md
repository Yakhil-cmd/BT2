### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in `Pool` contains an explicitly acknowledged unbounded `while` loop that iterates over a pool member's entire balance-change trace (`pool_member_epoch_balance`). A pool member who accumulates enough trace entries between reward claims will cause `claim_rewards` to exceed the Starknet gas limit, permanently freezing their unclaimed yield with no recovery path.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function is called by the public `claim_rewards` entry point. It iterates over the `pool_member_epoch_balance` trace for the calling pool member:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`: [2](#0-1) 

The `insert` implementation appends a new checkpoint when the epoch key differs from the last entry, and overwrites the last entry when the key matches: [3](#0-2) 

Because `set_member_balance` always inserts at `current_epoch + K`, each balance change in a **distinct epoch** appends a new entry. The operations that trigger `set_member_balance` are all pool-member-controlled: `enter_delegation_pool`, `add_to_delegation_pool`, and `exit_delegation_pool_intent`.

The `entry_to_claim_from` field in `InternalPoolMemberInfoV1` tracks the starting index for the next claim and is only advanced upon a **successful** `claim_rewards` call: [4](#0-3) 

If a pool member makes balance changes across N distinct epochs without ever claiming, the loop must iterate N times on the next claim. Each iteration performs multiple storage reads (via `pool_member_trace.at(...)` and `find_sigma`), which are expensive on Starknet. Once N is large enough to exceed the block gas limit, `claim_rewards` reverts on every call.

There is **no partial-claim mechanism** and the trace is **append-only** (entries are never removed). The `entry_to_claim_from` pointer can only advance through a successful execution of the full loop. Therefore, once the trace is too long, the pool member's rewards are permanently inaccessible.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a pool member grows beyond the gas-limit threshold, every call to `claim_rewards` for that member reverts. Because:
1. The trace is append-only (no pruning mechanism exists).
2. `entry_to_claim_from` can only advance through a successful full-loop execution.
3. There is no batched or partial claim path.

The pool member's accumulated STRK rewards are permanently locked in the pool contract with no recovery path.

---

### Likelihood Explanation

**Low-Medium.**

The trace grows at most one entry per epoch (multiple changes within the same epoch overwrite the same slot). A pool member must make balance changes across many consecutive epochs **without claiming** to accumulate a dangerous trace length. This can occur:

- Naturally: a delegator who frequently adjusts their stake (e.g., via `add_to_delegation_pool` each epoch) but rarely or never calls `claim_rewards`.
- Deliberately: a pool member who intentionally inflicts this on themselves (self-griefing), or whose `reward_address` is a contract that reverts on token receipt, preventing successful claims and allowing the trace to grow unboundedly.

The second scenario (reward address that reverts) is particularly concerning because it creates a feedback loop: `claim_rewards` fails due to the transfer revert, `entry_to_claim_from` never advances, and the trace keeps growing with each balance change.

---

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` calls once the trace exceeds a safe bound (e.g., 1000 entries).
2. **Introduce a partial-claim mechanism.** Allow `claim_rewards` to accept a `max_entries` parameter, advancing `entry_to_claim_from` by at most that many steps per call, so rewards can be claimed in batches.
3. **Prune claimed entries.** After `entry_to_claim_from` advances past old entries, compact or zero them out to prevent unbounded storage growth.
4. **Validate reward address.** Reject reward addresses that are contracts without a safe token-receive path, or use a pull-pattern that does not revert on failed transfers.

---

### Proof of Concept

1. Staker deploys a pool with `pool_enabled: true`.
2. Attacker (pool member) calls `enter_delegation_pool` with a minimal amount.
3. Each epoch, the attacker calls `add_to_delegation_pool` with 1 unit. Because `set_member_balance` inserts at `current_epoch + K`, each call in a new epoch appends a new entry to `pool_member_epoch_balance`.
4. The attacker never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
5. After N epochs of balance changes, `pool_member_epoch_balance` has N entries.
6. The attacker (or anyone) calls `claim_rewards(pool_member: attacker)`. The loop at line 859 iterates N times, each iteration reading from storage via `pool_member_trace.at(entry_to_claim_from)` and calling `find_sigma` (additional storage reads).
7. Once N is large enough, the transaction exceeds the Starknet gas limit and reverts.
8. All subsequent `claim_rewards` calls also revert. The attacker's accumulated STRK rewards are permanently frozen.

The self-reinforcing variant: set `reward_address` to a contract that reverts on ERC-20 `transfer`. Every `claim_rewards` attempt fails at the transfer step (line 366), `entry_to_claim_from` is never written back, and the trace continues to grow with each `add_to_delegation_pool` call, accelerating the path to permanent freeze.     fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
