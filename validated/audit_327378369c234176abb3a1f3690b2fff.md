### Title
Unredacted `Debug` Derive on Secret-Bearing Types Exposes Private Signing Shares and Presign Nonce Material — (File: `src/lib.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`, `src/frost/mod.rs`)

---

### Summary
Multiple production types that hold secret cryptographic scalars unconditionally derive the `Debug` trait without redaction. Any code path that formats these types with `{:?}` — including logging frameworks, `dbg!()` macros, panic handlers, and error reporters — will emit the raw private signing share or presign nonce scalars in plaintext to whatever sink is configured. This is the structural analog to the Peggo CLI-argument exposure: instead of leaking a key through `/proc`, the key leaks through any observable debug output channel.

---

`KeygenOutput<C>` is the root type: [1](#0-0) 

It derives `Debug` unconditionally. `SigningShare<C>` from `frost_core` is itself a newtype over a field scalar and also derives `Debug`, so `{:?}` on a `KeygenOutput` emits the raw scalar bytes of the private share.

`PresignArguments` for OT-based ECDSA also derives `Debug` and embeds `keygen_out: KeygenOutput` directly: [2](#0-1) 

`PresignOutput` and `RerandomizedPresignOutput` for OT-based ECDSA derive `Debug` and contain the per-signature nonce shares `k` and `sigma`: [3](#0-2) 

`PresignOutput` for Robust ECDSA derives `Debug` and contains four secret nonce shares `c`, `e`, `alpha`, `beta`: [4](#0-3) 

`RerandomizedPresignOutput` for Robust ECDSA also derives `Debug` with `e`, `alpha`, `beta`: [5](#0-4) 

`PresignOutput<C>` for FROST derives `Debug` and contains `nonces: SigningNonces<C>`, which holds the secret binding and hiding nonces: [6](#0-5) 

None of these `Debug` derives are gated behind `#[cfg(test)]` or `#[cfg(debug_assertions)]`. They are unconditional on all production builds.

---

### Impact Explanation

**For `KeygenOutput` / `PresignArguments`:** Formatting either type with `{:?}` emits the raw `private_share` scalar. An observer who reads that output can directly use the scalar as a signing key share. Collecting `t` such shares (the reconstruction threshold) allows full private key reconstruction. This is disclosure of a private signing share — Critical impact under the allowed scope.

**For `PresignOutput` / `RerandomizedPresignOutput`:** Formatting with `{:?}` emits `k` and `sigma` (OT) or `c`, `e`, `alpha`, `beta` (Robust). These are the per-signature nonce shares. An attacker who obtains `k` and `sigma` for a presignature can compute the private key from any resulting signature using the standard ECDSA relation `x = (s·k − h) / r`. This is disclosure of presign secrets and nonce material — Critical impact under the allowed scope.

**For FROST `PresignOutput`:** `SigningNonces` contains the binding and hiding nonce scalars. Leaking these allows nonce reuse attacks that recover the signing key.

---

### Likelihood Explanation

The `Debug` derive is present on every public-facing output type in the library. Any integrating application that uses standard Rust logging (`tracing`, `log`, `env_logger`) and logs these structs at debug or trace level will expose the secrets. Common triggers include:

- `tracing::debug!("{:?}", keygen_output)` or `log::debug!("{:?}", presign_output)` in application code
- `dbg!(keygen_output)` left in during development
- Panic messages that include these structs in format strings
- Error-reporting frameworks (e.g., `anyhow`, `eyre`) that capture and log debug representations of context values

This is a well-documented class of accidental secret exposure in Rust cryptographic libraries. The library provides no protection against it — there is no `secrecy`-style wrapper, no `ZeroizeOnDrop`-equivalent for `Debug`, and no documentation warning callers not to log these types.

---

### Recommendation

1. Remove `Debug` from all types containing secret scalar material, or implement `Debug` manually to redact secret fields:

```rust
impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        f.debug_struct("KeygenOutput")
            .field("private_share", &"[REDACTED]")
            .field("public_key", &self.public_key)
            .finish()
    }
}
```

2. Apply the same manual `Debug` implementation to `PresignOutput`, `RerandomizedPresignOutput`, and `PresignArguments` across all scheme modules.

3. Consider wrapping secret scalar fields in a `Secret<T>` newtype (e.g., from the `secrecy` crate) that implements `Debug` as `[REDACTED]` by default, preventing accidental exposure regardless of how the outer type is formatted.

### Proof of Concept

```rust
use threshold_signatures::{KeygenOutput, Ciphersuite};
use frost_secp256k1::Secp256K1Sha256 as C;

// After running keygen, a developer adds a debug log:
let keygen_output: KeygenOutput<C> = /* result of keygen protocol */;

// Any of the following emit the raw private_share scalar:
println!("{:?}", keygen_output);
// Output: KeygenOutput { private_share: SigningShare(Scalar(0xADD8BEA1...)), public_key: ... }

tracing::debug!(?keygen_output);  // same exposure via tracing

// Similarly for presign output:
let presign: PresignOutput = /* result of presign protocol */;
println!("{:?}", presign);
// Output: PresignOutput { big_r: ..., k: Scalar(0x28B22444...), sigma: Scalar(0x4A258164...) }
// k and sigma are now known to any log reader; combined with the signature,
// the private key x can be recovered as x = (s*k - h) / r mod q.
```
