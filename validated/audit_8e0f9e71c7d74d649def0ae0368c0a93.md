### Title
Re-entrancy in `claim_rewards` Allows Theft of Unclaimed Staker Yield - (File: src/staking/staking.cairo)

### Summary
`staking.cairo`'s `claim_rewards` function calls `send_rewards_to_staker`, which performs an external token transfer to the staker's `reward_address` **before** the updated `staker_info` (with `unclaimed_rewards_own = 0`) is committed to storage. A malicious staker who sets their `reward_address` to a re-entrant contract can re-invoke `claim_rewards` during the transfer callback, reading the stale (non-zeroed) `unclaimed_rewards_own` from storage and draining rewards multiple times.

### Finding Description

In `claim_rewards` (staking.cairo lines 411–431), the execution order is:

1. Read `staker_info` from storage (`unclaimed_rewards_own = X`).
2. Call `send_rewards_to_staker` (lines 1614–1629):
   - `claim_from_reward_supplier(X)` — reward supplier decrements its `unclaimed_rewards` by X and transfers X tokens to the staking contract.
   - `token_dispatcher.checked_transfer(recipient: reward_address, amount: X)` — **external call to reward_address**.
   - `staker_info.unclaimed_rewards_own = Zero::zero()` — zeroed only in the **local variable**, not in storage.
3. `write_staker_info(:staker_address, :staker_info)` — storage write happens **after** the external call. [1](#0-0) [2](#0-1) 

Because storage is not updated until after the external transfer, a re-entrant call to `claim_rewards` during step 2 reads the original `unclaimed_rewards_own = X` from storage and repeats the full reward disbursement.

The `claim_from_reward_supplier` helper (utils.cairo lines 50–60) includes a balance-delta check (`balance_after - balance_before == amount`), but this check passes during re-entry: the staking contract's balance was reduced by X when the first transfer went out, then replenished by X when the reward supplier sends tokens again. [3](#0-2) 

The reward supplier's `claim_rewards` (reward_supplier.cairo lines 205–220) decrements `unclaimed_rewards` before transferring, so the re-entrant call succeeds as long as `unclaimed_rewards >= 2X` — a realistic condition in a live protocol with multiple stakers. [4](#0-3) 

Notably, the developers were aware of this class of bug: `unstake_action` (staking.cairo line 497) contains the comment *"This is done here to avoid re-entrancy"* and explicitly calls `write_staker_info` before subsequent transfers. The same protection was not applied to `claim_rewards`. [5](#0-4) 

### Impact Explanation
**High — Theft of unclaimed yield.** A malicious staker can repeatedly claim the same `unclaimed_rewards_own` balance, draining tokens from the reward supplier that belong to other stakers. Each re-entrant loop extracts an additional X tokens until the reward supplier's `unclaimed_rewards` is exhausted.

### Likelihood Explanation
Any staker can set their `reward_address` to an arbitrary contract address via `change_reward_address`. No privileged role is required. The attacker only needs accumulated unclaimed rewards and a reward supplier with sufficient `unclaimed_rewards` (guaranteed in a live multi-staker deployment). The entry point (`claim_rewards`) is public and callable by the staker or their reward address. [6](#0-5) 

### Recommendation
Apply the check-effects-interactions pattern: call `write_staker_info` (zeroing `unclaimed_rewards_own` in storage) **before** invoking `send_rewards_to_staker`. Alternatively, zero `unclaimed_rewards_own` in storage before the external transfer, mirroring the pattern already used in `unstake_action`.

```cairo
// Fixed claim_rewards
fn claim_rewards(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    self.general_prerequisites();
    let mut staker_info = self.internal_staker_info(:staker_address);
    // ... access control checks ...
    let amount = staker_info.unclaimed_rewards_own;
    let token_dispatcher = strk_token_dispatcher();
    staker_info.unclaimed_rewards_own = Zero::zero();
    self.write_staker_info(:staker_address, :staker_info); // <-- write BEFORE external call
    self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
    amount
}
```

### Proof of Concept

1. Attacker deploys `MaliciousRewardAddress` contract that, on receiving STRK, calls `staking.claim_rewards(attacker_staker_address)`.
2. Attacker stakes STRK and calls `change_reward_address(malicious_contract_address)`.
3. Rewards accumulate: `unclaimed_rewards_own = X`.
4. Attacker (or anyone) calls `staking.claim_rewards(attacker_staker_address)`.
5. Staking contract calls `claim_from_reward_supplier(X)` → reward supplier sends X to staking contract.
6. Staking contract calls `STRK.transfer(malicious_contract, X)`.
7. `MaliciousRewardAddress.on_receive` fires → calls `staking.claim_rewards(attacker_staker_address)`.
8. Re-entrant call reads storage: `unclaimed_rewards_own = X` (not yet zeroed).
9. Re-entrant call calls `claim_from_reward_supplier(X)` again → succeeds if reward supplier `unclaimed_rewards >= X`.
10. Re-entrant call transfers another X to `malicious_contract`.
11. Re-entrant call writes `unclaimed_rewards_own = 0` to storage.
12. Original call writes `unclaimed_rewards_own = 0` to storage (no-op).

**Result**: Attacker received `2X` tokens; reward supplier lost `2X` instead of `X`, stealing X tokens that belonged to other stakers' reward pool.

pub(crate) fn claim_from_reward_supplier(
    reward_supplier_dispatcher: IRewardSupplierDispatcher,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
) {
    let staking_contract = get_contract_address();
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}
