### Title
Gateway Resource-Bounds Validation Uses Previous Block's `l2_gas_price` Instead of `next_l2_gas_price`, Causing Admission of Transactions That Will Fail Execution and Rejection of Valid Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current block's** `l2_gas_price` from the previous block header to gate admission, but the actual price used for the next block's execution is `next_l2_gas_price` — a distinct field computed by the EIP-1559-style fee market and stored in the same header. The code even carries a self-documenting TODO acknowledging the bug. The mismatch causes the gateway to (a) admit transactions whose `max_price_per_unit` is below the actual next-block price, which will be rejected by the batcher at execution time, and (b) reject transactions whose `max_price_per_unit` is above the next-block price but below the stale threshold, which are perfectly valid.

### Finding Description

In `validate_resource_bounds`, the gateway fetches the previous block's info and extracts `strk_gas_prices.l2_gas_price`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is then `min_gas_price_percentage% × previous_block_l2_gas_price`. The default `min_gas_price_percentage` is 100, so the check is `tx.max_price >= previous_block_l2_gas_price`. [2](#0-1) 

However, the Starknet fee market computes a **separate** `next_l2_gas_price` field in the block header — the price that will actually govern the next block — via `calculate_next_l2_gas_price_for_fin`: [3](#0-2) 

This value is stored in `BlockHeaderWithoutHash.next_l2_gas_price` and propagated to state sync: [4](#0-3) 

The gateway's `GatewayFixedBlockSyncStateClient` reads the block header but only surfaces `l2_gas_price` (the price *used by* the previous block), not `next_l2_gas_price` (the price *to be used by* the next block): [5](#0-4) 

The same stale block info is also passed to `run_validate_entry_point`, so the blockifier pre-validation at the gateway level also uses the wrong price: [6](#0-5) 

Additionally, `validate_tx_l2_gas_price_within_threshold` only checks `ValidResourceBounds::AllResources` (v0.13.3+ transactions) and silently skips `ValidResourceBounds::L1Gas` (legacy v0.13.x transactions) entirely: [7](#0-6) 

### Impact Explanation

**Admission of transactions that will fail execution (gas price rising):**
When blocks are consistently full, `next_l2_gas_price > l2_gas_price`. A transaction with `max_price_per_unit = X` where `X ≥ l2_gas_price` (passes gateway) but `X < next_l2_gas_price` (fails batcher) is admitted to the mempool and forwarded to the batcher, where it will be rejected by `check_fee_bounds` in `perform_pre_validation_stage`. This pollutes the mempool with transactions that cannot be sequenced.

**Rejection of valid transactions (gas price falling):**
When blocks are consistently under-target, `next_l2_gas_price < l2_gas_price`. A transaction with `max_price_per_unit = X` where `X ≥ next_l2_gas_price` (valid for the next block) but `X < min_gas_price_percentage% × l2_gas_price` (fails gateway threshold) is incorrectly rejected. Users receive a `GAS_PRICE_TOO_LOW` error for a transaction that would have been sequenced successfully.

Both outcomes match: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The Starknet fee market adjusts `next_l2_gas_price` every block based on gas consumption relative to `gas_target`. Any sustained period of high or low block utilization — which is routine, not exceptional — causes `next_l2_gas_price` to diverge from `l2_gas_price`. The discrepancy compounds across consecutive blocks. No privileged access or adversarial action is required; normal network load triggers the bug. The TODO comment in the source confirms the developers are aware the wrong field is being read.

### Recommendation

Replace the read of `l2_gas_price` with `next_l2_gas_price` from the block header in `validate_resource_bounds`. The `GatewayFixedBlockStateReader` trait and its `GatewayFixedBlockSyncStateClient` implementation should expose `next_l2_gas_price` from `BlockHeaderWithoutHash`, and `validate_resource_bounds` should use that value as the reference price. The same correction should be applied to the `BlockInfo` passed to `run_validate_entry_point` so that blockifier pre-validation at the gateway also uses the correct upcoming price.

### Proof of Concept

1. Previous block N has `l2_gas_price = 1000 fri` and was full, so `next_l2_gas_price = 1003 fri` (≈0.3% increase per `MIN_GAS_PRICE_INCREASE_DENOMINATOR = 333`).
2. User submits an invoke transaction with `l2_gas.max_price_per_unit = 1001 fri`.
3. Gateway `validate_resource_bounds` computes threshold = `100% × 1000 = 1000`. Since `1001 ≥ 1000`, the transaction **passes** gateway validation and is admitted to the mempool.
4. Batcher builds block N+1 with `l2_gas_price = 1003 fri` (the `next_l2_gas_price` from block N's header).
5. Blockifier `check_fee_bounds` checks `1001 < 1003` → `MaxGasPriceTooLow` → transaction is **rejected** at execution time.
6. The transaction consumed gateway validation resources, a mempool slot, and a batcher execution attempt, but was never sequenced.

Conversely, if block N was empty and `next_l2_gas_price = 997 fri`, a transaction with `max_price = 999 fri` satisfies `999 ≥ 997` (valid for block N+1) but fails the gateway check `999 < 1000 = threshold`, producing a spurious `GAS_PRICE_TOO_LOW` rejection.

pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
