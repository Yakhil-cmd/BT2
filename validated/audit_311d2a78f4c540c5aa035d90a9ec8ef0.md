### Title
Gateway Admits Invoke Transactions with Invalid Signatures via Overly Broad `skip_stateful_validations` Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature verification) for invoke transactions with `nonce == 1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. This check is too broad: it returns `true` for **any** transaction from the account in the mempool, not specifically a `deploy_account` transaction. An unprivileged attacker can exploit this by submitting an invoke transaction with a garbage signature for any address that has a legitimate `deploy_account` pending in the mempool, causing the invalid transaction to be admitted and eventually executed, draining fees from the victim's account.

---

### Finding Description

The `skip_stateful_validations` function is designed to improve UX for the `deploy_account + invoke` pattern: when a user submits a `deploy_account` (nonce=0) and an `invoke` (nonce=1) simultaneously, the gateway cannot yet verify the invoke's signature because the account doesn't exist on-chain yet. The skip is intended to allow this legitimate pattern.

The condition for skipping is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The `account_tx_in_pool_or_recent_block` check is:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` if the account has **any** transaction in the pool or committed state — not specifically a `deploy_account`. The code comment acknowledges this: *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate = false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, skipping the `__validate__` call entirely:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

**Attack scenario:**

1. Legitimate user pre-funds address X and submits a `deploy_account` transaction (nonce=0). This transaction is now visible in the mempool.
2. Attacker observes the mempool and sees the `deploy_account` for address X.
3. Attacker crafts an `invoke` transaction with `sender_address=X`, `nonce=1`, arbitrary `calldata`, and a garbage `signature`.
4. Gateway stateless validation passes (signature length check only, not validity).
5. `validate_nonce` passes: `account_nonce=0`, `tx_nonce=1`, `0 ≤ 1 ≤ 200`. [5](#0-4) 

6. `validate_by_mempool` passes: unique tx hash, nonce=1 is a valid future nonce.
7. `skip_stateful_validations` returns `true`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` (because the legitimate `deploy_account` is in the pool).
8. `run_validate_entry_point` skips `__validate__` — the garbage signature is never checked.
9. The invalid invoke is admitted to the mempool.
10. Batcher executes `deploy_account` (nonce=0) → account deployed.
11. Batcher executes attacker's invoke (nonce=1) → `__validate__` is called with garbage signature → fails → transaction reverts but **fee is charged to address X**.

The `verify_can_pay_committed_bounds` check in `perform_pre_validation_stage` passes because address X was pre-funded (required for `deploy_account` to succeed): [6](#0-5) 

---

### Impact Explanation

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

An unprivileged attacker can cause any account that has a pending `deploy_account` in the mempool to have an invalid invoke transaction admitted. When executed, the transaction fails `__validate__` but the fee is charged to the victim's account (Starknet charges fees on validation failure to prevent DoS). The attacker can set high resource bounds to maximize the fee drain. The attacker pays nothing — the fee is charged to the victim's address.

---

### Likelihood Explanation

- The mempool is publicly observable; any attacker can monitor for `deploy_account` transactions.
- The attack window is the time between `deploy_account` submission and execution (which can be multiple blocks).
- No special privileges or on-chain state are required.
- The attack is cheap: the attacker only needs to submit a single RPC call.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the account in the mempool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction before granting the validation skip.

A minimal fix:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // Only skip if a deploy_account specifically is pending, not any transaction.
            return mempool_client
                .has_pending_deploy_account(tx.sender_address())  // new specific check
                .await
                ...
        }
    }
    Ok(false)
}
``` [7](#0-6)

### Proof of Concept

1. Monitor the mempool for `deploy_account` transactions targeting address `X`.
2. Construct an `RpcInvokeTransactionV3` with:
   - `sender_address = X`
   - `nonce = 1`
   - `calldata = [<any target contract>, <any selector>, ...]`
   - `signature = [0x1, 0x2]` (garbage)
   - `resource_bounds` set to maximum allowed values
3. Submit via `POST /add_transaction`.
4. The gateway's `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(X)` is `true` (the legitimate `deploy_account` is in the pool).
5. The transaction is admitted to the mempool without signature verification.
6. After the `deploy_account` is executed, the batcher executes the attacker's invoke, `__validate__` fails, and the fee is charged to address X. [8](#0-7) [9](#0-8)

```rust
/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
