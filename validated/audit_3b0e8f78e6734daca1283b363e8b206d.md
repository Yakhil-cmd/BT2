### Title
NFT Metadata Fetched Without Integrity Verification When `metadataHash` Is Empty — Enables Offer-Time Metadata Spoofing - (File: packages/gui/src/hooks/useFetchAndProcessMetadata.ts)

---

### Summary

When an NFT is minted with an empty or absent `metadataHash`, the Chia GUI fetches and fully trusts metadata from the attacker-controlled URI without any hash verification. The spoofed metadata (name, description, preview images) is rendered in the NFT offer viewer, potentially deceiving a victim into accepting an offer for a worthless NFT.

---

### Finding Description

In `useFetchAndProcessMetadata.ts`, the integrity check is gated on `hash` being truthy:

```js
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
``` [1](#0-0) 

When `hash` is `undefined`, `null`, or an empty string, the condition short-circuits and the metadata is accepted unconditionally. The caller in `useMetadataData` passes `metadataHash` directly from the on-chain NFT record:

```js
const { metadataUris = [], metadataHash } = nft;
const metadata = await fetchAndProcessMetadata(firstUri, metadataHash);
``` [2](#0-1) 

The `NFTInfo` type declares `metadataHash: string` but the Chia NFT1 standard permits minting with an empty hash field. `NFTDetails.tsx` itself guards display of the hash with `nft.metadataHash && nft.metadataHash !== '0x'`, confirming that empty/`0x` hashes are a real, observed case. [3](#0-2) 

The accepted (unverified) metadata object — including `name`, `description`, `preview_image_uris`, and `preview_video_uris` — is stored in state and rendered throughout the NFT UI, including the offer viewer. [4](#0-3) 

---

### Impact Explanation

An attacker mints an NFT with an empty `metadataHash` and a `metadataUris` entry pointing to their own server. They initially serve metadata that makes the NFT appear to belong to a valuable collection (fake name, description, preview image of a desirable asset). They create a sell offer for XCH. A victim opens the offer in the GUI: the NFT card renders the attacker-controlled name, description, and preview image without any integrity check. The victim, believing they are purchasing a valuable NFT, accepts the offer and transfers XCH for a worthless coin.

This satisfies the **High** impact criterion: *"Corruption, spoofing, or unsafe trust of... NFT metadata... that causes a user to approve... the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

- Minting an NFT with an empty `metadataHash` is a valid, permissible on-chain operation.
- The attacker controls the metadata server and can serve arbitrary JSON.
- The `NFTHashStatus` component does display an "Invalid hash" chip in error/red when `data` is `undefined` (which occurs because `findValidUri` returns `undefined` when `!hash`). However, this chip does not block rendering of the spoofed metadata, and most users do not understand its significance — it is indistinguishable from a transient network error. [5](#0-4) [6](#0-5) 

---

### Recommendation

In `useFetchAndProcessMetadata`, treat a missing or empty `hash` as a verification failure rather than a bypass. Either throw an error when `hash` is absent, or at minimum expose a distinct `isHashMissing` flag so callers can block offer acceptance and display an explicit, actionable warning (e.g., "This NFT has no integrity hash — metadata cannot be verified. Do not accept offers for this NFT.").

```ts
// Proposed guard
if (!hash) {
  throw new Error('No metadata hash provided; cannot verify integrity');
}
if (!compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
``` [7](#0-6) 

---

### Proof of Concept

1. Attacker mints an NFT on Chia mainnet with `metadata_hash = ""` (or `0x`) and `metadata_uris = ["https://attacker.example/meta.json"]`.
2. `attacker.example/meta.json` initially returns `{"name":"Rare Chia Punk #1","description":"1-of-1 genesis mint","preview_image_uris":["https://attacker.example/fake_punk.png"]}`.
3. Attacker creates a sell offer: 1 XCH for this NFT.
4. Victim opens the offer file in the Chia GUI. `useMetadataData` fetches `meta.json`; `fetchAndProcessMetadata` is called with `metadataHash = ""`. The `if (hash && ...)` guard is skipped. The spoofed metadata is stored and rendered.
5. The NFT card displays "Rare Chia Punk #1" with the fake preview image. The "Invalid hash" chip is present but ambiguous.
6. Victim clicks "Accept Offer" and pays 1 XCH for a worthless NFT.
7. Attacker can also mutate `meta.json` after the offer is shared (e.g., swap the preview image to a different asset) since there is no hash pinning. [8](#0-7) [9](#0-8)
