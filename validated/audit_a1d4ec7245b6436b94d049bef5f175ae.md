### Title
Staker Can Sandwich Delegator Deposits to Raise Commission via Active Commitment, Stealing Delegator Yield - (File: `src/staking/staking.cairo`)

---

### Summary

When a staker holds an active `CommissionCommitment`, the `set_commission` function permits raising commission to any value up to `max_commission` — not just decreasing it. A staker can monitor the mempool for incoming `enter_delegation_pool` or `add_to_delegation_pool` transactions, frontrun with a commission increase, and then lower it back after the deposit executes. This lets the staker extract excess commission from delegators without any lasting penalty, directly analogous to the reported sandwich-on-ratio pattern.

---

### Finding Description

Without a commitment, `update_commission` enforces a strict monotone-decrease invariant:

```cairo
} else {
    assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
}
```

But when a `CommissionCommitment` is active, the only constraints are that the new commission does not exceed `max_commission` and is not identical to the current value:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission,
        "{}",
        Error::INVALID_COMMISSION_WITH_COMMITMENT,
    );
    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
``` [1](#0-0) 

This means a staker who has set a commitment with `max_commission = X` can freely oscillate their commission between 0 and X at any time during the commitment window. The code itself acknowledges this gap:

> **Note**: Current commission increase safeguards still allow for sudden commission changes. [2](#0-1) 

Commission changes are written directly to storage and take effect immediately for reward calculations in `calculate_staker_pools_rewards`, which reads `staker_pool_info.commission()` at distribution time: [3](#0-2) 

The `set_commission_commitment` function requires only that `max_commission >= current_commission` and that no active commitment already exists, making it trivially obtainable by any staker with a pool: [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Delegators submit `enter_delegation_pool` or `add_to_delegation_pool` transactions expecting the publicly visible commission rate. If the staker raises commission immediately before those transactions land and lowers it immediately after, delegators earn less yield than they agreed to for the affected epoch(s). The excess commission flows to the staker's reward address. Because commission applies to all rewards earned during the epoch in which the change is active, a single sandwich on a large deposit can extract a meaningful fraction of that delegator's epoch rewards.

---

### Likelihood Explanation

**High.** Any staker who has called `set_commission_commitment` — a feature explicitly designed to attract delegators by signalling a commission ceiling — can execute this attack. The commitment window can be up to one year (`epochs_in_year`). Starknet's public mempool makes pending delegation transactions visible. The staker needs only two transactions (raise, then lower) around the target deposit. The code comment at line 745 confirms the developers are aware the safeguard is incomplete, indicating this is a known gap rather than an oversight in a single edge case.

---

### Recommendation

When a commitment is active, restrict `set_commission` to only allow **decreases** relative to the current commission (not just relative to `max_commission`). The commitment's purpose is to cap the maximum commission, not to grant a free-raise window. Alternatively, apply a time-lock (e.g., effective next epoch) to any commission increase so that in-flight delegator transactions cannot be sandwiched.

---

### Proof of Concept

1. Alice (staker) has `commission = 500` (5%) and calls `set_commission_commitment(max_commission: 1000, expiration_epoch: current + 10)`.
2. Alice observes Bob's pending `enter_delegation_pool` transaction for a large STRK amount in the mempool.
3. Alice frontruns Bob with `set_commission(1000)`. This passes because `1000 <= max_commission` and `1000 != 500`. Commission is now 10%.
4. Bob's transaction lands. Bob is now a pool member at 10% commission.
5. The epoch's attestation fires; `calculate_staker_pools_rewards` reads `commission = 1000` and splits rewards accordingly — Alice takes 10% of Bob's delegation rewards as commission.
6. Alice calls `set_commission(500)` to restore her advertised rate, attracting the next delegator.
7. Bob received ~half the yield he expected for that epoch. The difference went to Alice's reward address with no on-chain penalty. [5](#0-4) [6](#0-5)

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
        }
