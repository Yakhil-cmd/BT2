Audit Report

## Title
Hard deserialization error on `Unspecified` `on_low_wasm_memory_hook_status` breaks backward compatibility during rolling upgrades — (`rs/replicated_state/src/canister_state/system_state/task_queue/proto.rs`)

## Summary

`TryFrom<pb::TaskQueue> for TaskQueue` hard-errors when `on_low_wasm_memory_hook_status` equals `0` (`Unspecified`), which is the proto3 wire default for any checkpoint written before this field existed. During a rolling upgrade, new replicas loading old checkpoints (either their own or via state-sync) will fail deserialization for every canister, causing fatal checkpoint loading failures and crash-looping the upgraded replicas.

## Finding Description

The proto field `on_low_wasm_memory_hook_status` in `TaskQueue` is a plain (non-`optional`) proto3 enum:

```proto
// canister_state_bits.proto line 428
OnLowWasmMemoryHookStatus on_low_wasm_memory_hook_status = 2;
```

Any checkpoint written by a replica that predates this field will have wire value `0` for this field. The deserialization path in `proto.rs` lines 27–32 performs two conversions:

```rust
on_low_wasm_memory_hook_status: pb::OnLowWasmMemoryHookStatus::try_from(
    item.on_low_wasm_memory_hook_status,  // = 0 from old checkpoint
)
.map_err(|e| ProxyDecodeError::Other(...))?
.try_into()?   // <-- fails: Unspecified → Err
```

The first `try_from` succeeds (prost accepts `0` as `Unspecified`). The second `.try_into()` invokes the domain-level `TryFrom<pb::OnLowWasmMemoryHookStatus> for OnLowWasmMemoryHookStatus` in `management_canister_types/src/lib.rs` lines 4619–4626, which explicitly returns `Err(ProxyDecodeError::ValueOutOfRange)` for `Unspecified`.

The call site in `state_layout/proto.rs` lines 154–156 uses `unwrap_or_default()` for the outer `tasks` field (handling a missing `tasks` message), but then propagates the inner error unconditionally:

```rust
let tasks: pb_canister_state_bits::TaskQueue =
    try_from_option_field(value.tasks, "CanisterStateBits::tasks").unwrap_or_default();
let task_queue = TaskQueue::try_from(tasks)?;  // propagates Err
```

A missing `tasks` field (old checkpoint) produces a default `pb::TaskQueue` with `on_low_wasm_memory_hook_status = 0`, which then fails `TaskQueue::try_from`. The `?` propagates this error up through `CanisterStateBits::try_from`, causing checkpoint loading to fail for every canister in the old checkpoint.

## Impact Explanation

Checkpoint loading failure in IC state manager is fatal — a replica that cannot deserialize its checkpoint cannot participate in consensus. During a rolling upgrade that introduces `on_low_wasm_memory_hook_status` into `TaskQueue`, every new replica that loads an old checkpoint (its own pre-upgrade checkpoint or a state-synced checkpoint from an old replica) will crash-loop. If enough replicas upgrade before old checkpoints are replaced, the subnet loses liveness. This matches the allowed impact: **Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact** — **High severity ($2,000–$10,000)**.

## Likelihood Explanation

This is triggered automatically by the standard rolling upgrade procedure — no external attacker action is required. Any subnet upgrade that introduces `on_low_wasm_memory_hook_status` into `TaskQueue` while old replicas are still writing checkpoints is affected. The vulnerability window is the duration of the rolling upgrade. Once any replica upgrades and attempts to load a pre-upgrade checkpoint, the crash-loop begins.

## Recommendation

Treat `Unspecified = 0` as `ConditionNotSatisfied` (the safe default) instead of returning a hard error in `management_canister_types/src/lib.rs`:

```rust
pb_canister_state_bits::OnLowWasmMemoryHookStatus::Unspecified => {
    Ok(OnLowWasmMemoryHookStatus::ConditionNotSatisfied)
}
```

This is the standard IC pattern for backward-compatible proto enum migration. The serialization path (`From<&OnLowWasmMemoryHookStatus>`) already never writes `0`, so this change only affects the backward-compatibility window during upgrades.

## Proof of Concept

Construct a `pb::TaskQueue` with `on_low_wasm_memory_hook_status = 0` (simulating an old checkpoint or a default-constructed `TaskQueue` from `unwrap_or_default()`) and assert the current behavior:

```rust
let old_proto = pb::TaskQueue {
    paused_or_aborted_task: None,
    on_low_wasm_memory_hook_status: 0, // Unspecified — old replica default
    queue: vec![],
};
// Currently returns Err — should return Ok with ConditionNotSatisfied
assert!(TaskQueue::try_from(old_proto).is_err());
```

This can be run as a unit test in `rs/replicated_state/src/canister_state/system_state/task_queue/proto.rs`. A complementary integration test can verify that `CanisterStateBits::try_from` fails when `tasks` is `None` (triggering `unwrap_or_default()`) on a new replica binary, confirming the crash path in `state_layout/proto.rs` line 156.
