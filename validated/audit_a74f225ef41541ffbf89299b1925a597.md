### Title
WalletConnect `chia_spendCAT` Confirmation Dialog Omits CAT Token Identity, Allowing Wallet-ID Substitution — (`packages/gui/src/electron/commands/humanizeParamValue.ts`, `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

A WalletConnect dApp that has been granted `chia_spendCAT` permission can freely supply any `wallet_id` in its request. The confirmation dialog shows only the raw numeric `wallet_id` and a generic CAT amount — the CAT token name/symbol lookup is explicitly commented out as a TODO. A user with multiple CAT wallets cannot distinguish which specific token is being spent, allowing a malicious dApp to target a more valuable CAT wallet than the one the user expects to approve.

---

### Finding Description

**1. No wallet_id type or identity validation in the command schema**

The `chia_wallet.cat_spend` command definition accepts `wallet_id` as a plain `number` with no constraint that it must correspond to a CAT-type wallet or any wallet the user has previously selected: [1](#0-0) 

`parseDappParams` validates parameter types but performs no semantic check that `wallet_id` refers to a CAT wallet: [2](#0-1) 

**2. CAT token name lookup is commented out — confirmation dialog shows only a raw number**

`formatMojoCat` in `humanizeParamValue.ts` reads `wallet_id` from the params but then immediately returns the formatted amount without resolving the CAT name. The lookup is explicitly deferred with a `// TODO add lookupCat` comment: [3](#0-2) 

The result is that the confirmation dialog shows:
- **Wallet Id**: `7` (raw number)
- **Amount**: `1.0` (generic CAT units, no token symbol)
- **Address**: `xch1...`

A user with wallet_id=5 (SBX, low value) and wallet_id=7 (DBX, high value) sees identical-looking dialogs for both.

**3. `parseCommandDisplay` returns `undefined` for `cat_spend`**

The richer wallet-delta display (which resolves asset names for offers) is only implemented for `take_offer` and `create_offer_for_ids`. For `cat_spend`, `parseCommandDisplay` returns `undefined`, so no enriched identity information is shown: [4](#0-3) 

**4. `dispatchPairRequest` enforces fingerprint but not wallet_id**

The authorization layer validates pair existence, command grant, network, and fingerprint — but there is no check that `wallet_id` belongs to a CAT wallet or matches any user-selected wallet: [5](#0-4) 

---

### Impact Explanation

A dApp paired with fingerprint F and granted `chia_spendCAT` can call `chia_spendCAT` with `wallet_id` pointing to any CAT wallet under fingerprint F. Because the confirmation dialog shows only a raw numeric wallet_id and a generic CAT amount (no token name/symbol), the user cannot distinguish which CAT is being spent. The dApp can substitute a high-value CAT wallet_id for a low-value one, and the user will see an identical-looking approval prompt. If the user confirms, the spend executes against the substituted wallet.

**Impact class**: High — WalletConnect state causes a user to approve the wrong asset/wallet.

---

### Likelihood Explanation

- Requires a dApp that has already been paired and granted `chia_spendCAT` — not zero-click, but a realistic threat from a malicious or compromised dApp.
- Users with multiple CAT wallets (common for DeFi users) are directly affected.
- The missing CAT name in the dialog is a confirmed code gap (explicit TODO), not a theoretical issue.

---

### Recommendation

1. **Resolve CAT token identity in the confirmation dialog**: Implement the commented-out `lookupCat` call in `formatMojoCat` so the dialog shows the token name/symbol alongside the amount.
2. **Validate wallet_id type at the GUI layer**: Before executing `cat_spend`, verify that the supplied `wallet_id` corresponds to a wallet of type `CAT`, `RCAT`, or `CRCAT` under the current fingerprint.
3. **Show a wallet delta row for `cat_spend`**: Extend `parseCommandDisplay` to handle `chia_wallet.cat_spend`, resolving the CAT name via `cat_get_name` and displaying it in the rich confirmation UI alongside the amount.

---

### Proof of Concept

1. User has two CAT wallets: wallet_id=5 (low-value token, e.g., 1 SBX) and wallet_id=7 (high-value token, e.g., 1000 DBX).
2. Malicious dApp is paired and granted `chia_spendCAT`.
3. dApp sends `chia_spendCAT` with `{ wallet_id: 7, address: attacker_address, amount: <1000 DBX in mojos>, fee: 0 }`.
4. Confirmation dialog shows: **Wallet Id: 7 / Amount: 1,000 / Address: xch1attacker...** — no token name.
5. User, expecting to approve a small SBX spend, sees "1,000" and may not recognize wallet_id=7 as their DBX wallet.
6. User confirms → 1000 DBX drained to attacker address. [3](#0-2) [6](#0-5)

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
}

export async function dispatchPairRequest<T>(
  topic: string,
  command: string,
  params: Record<string, unknown>,
  process: (context: DispatchPairRequestContext) => Promise<T>,
  confirm: () => Promise<boolean>,
): Promise<T> {
  const [loggedInFingerprint, isMainnetValue] = await Promise.all([getLoggedInFingerprint(), isMainnet()]);

  const pair = findPair(topic);
  if (!pair) {
    throw new WcError(`Pair not found`, WcErrorCode.USER_REJECTED);
  }

  // verify if the command is allowed for this pair
  if (!pair.commands.includes(command)) {
    throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const { fingerprint } = params;

  // verify if the network is the same as the pair's network
  if (isMainnetValue !== pair.mainnet) {
    throw new WcError(`Network mismatch`, WcErrorCode.UNSUPPORTED_CHAINS);
  }

  // verify if the requested fingerprint is allowed for this pair
  const requestedFingerprint = fingerprint ?? loggedInFingerprint;
  if (typeof requestedFingerprint !== 'number' || !requestedFingerprint || requestedFingerprint !== pair.fingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const context = {
    pair,
    fingerprint: requestedFingerprint,
  };

  // Dapps may not switch the active key for an existing pair.
  if (fingerprint !== undefined && fingerprint !== loggedInFingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  // if command is bypassed return true
  if (pair.bypass.includes(command)) {
    return process(context);
  }

  const isAllowed = await confirm();
  if (isAllowed === true) {
    return process(context);
  }

  throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
}
