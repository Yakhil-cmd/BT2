### Title
Malicious CKD Participant Can Corrupt Confidential Key Derivation Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function accepts `CKDOutput` shares from participants and sums them with no cryptographic verification. A single malicious participant can send arbitrary group elements, causing the coordinator to assemble a corrupted `CKDOutput`. Every honest party that subsequently calls `unmask` on that output receives an incorrect confidential key, permanently breaking the CKD result for that invocation.

### Finding Description
The analog vulnerability class from the reference report is **"update accounting state without verifying the underlying transfer actually occurred."** In the CKD protocol the equivalent is: **"accumulate participant shares without verifying they encode the correct cryptographic relationship."**

In `do_ckd_coordinator` the coordinator computes its own share and then blindly adds every other participant's contribution:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each honest participant is supposed to send `(λ_i · y_i · G,  λ_i · (x_i · H(pk‖app_id) + y_i · app_pk))` where `x_i` is their private share and `y_i` is a fresh random scalar:

```rust
// src/confidential_key_derivation/protocol.rs  lines 165-180
let big_y = ElementG1::generator() * y.0;
let big_s = hash_point * private_share.to_scalar();
let big_c = big_s + app_pk * y.0;
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [2](#0-1) 

No proof is generated, and the coordinator performs no check that:
- `norm_big_y` is a Lagrange-weighted commitment to any scalar at all.
- `norm_big_c` uses the **same** scalar `y_i` as `norm_big_y` (a DLEQ relationship).
- `norm_big_c` encodes the participant's actual key share `x_i` (verifiable against the public key share from DKG).

The participant-side function simply sends the raw pair and returns `None`:

```rust
// src/confidential_key_derivation/protocol.rs  lines 29-32
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
Ok(None)
``` [3](#0-2) 

### Impact Explanation
The coordinator's final output is `(Y, C)` where the invariant is `C = msk · H(pk‖app_id) + Y · app_pk`. The application recovers the confidential key as `C − app_sk · Y`. If any participant sends `(norm_big_y', norm_big_c')` that violates the DLEQ relationship, the coordinator computes:

```
Y_final  = Y_honest + norm_big_y'
C_final  = C_honest + norm_big_c'
unmask   = C_final − app_sk · Y_final
         = msk · H(pk‖app_id) + (norm_big_c' − app_sk · norm_big_y')
```

The result is an arbitrary wrong group element. The honest coordinator and all downstream consumers silently accept it as the legitimate confidential key. This matches:

> **High: Corruption of CKD outputs so honest parties accept … unusable cryptographic outputs.**

The test in `protocol.rs` confirms the expected invariant `confidential_key == msk · H(pk‖app_id)` — any deviation from it is a protocol failure. [4](#0-3) 

### Likelihood Explanation
Any single registered participant can trigger this. The attacker needs only to be included in the `participants` list passed to `ckd()`. No key material, coordinator privilege, or external assumption is required. The malicious participant calls `ckd()` normally but substitutes arbitrary `ElementG1` values before sending to the coordinator. The coordinator has no detection path.

### Recommendation
Add a **DLEQ (Discrete Log Equality) proof** to each participant's contribution, proving that the same scalar `y_i` underlies both `norm_big_y = λ_i · y_i · G` and the `y_i · app_pk` component of `norm_big_c`. Additionally, verify the key-share component against the participant's public verification share (available from the DKG output) before accumulating. The coordinator should reject any contribution that fails either check, identifying the malicious participant.

### Proof of Concept
1. Honest participants P_1 … P_{n-1} compute and send correct `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant P_m sends `(norm_big_y' = R, norm_big_c' = S)` where `R, S` are arbitrary `ElementG1` points unrelated to P_m's key share or any random scalar.
3. Coordinator accumulates: `Y_final = Σ_{i≠m} norm_big_y_i + R`, `C_final = Σ_{i≠m} norm_big_c_i + S`.
4. No error is raised; `CKDOutput::new(Y_final, C_final)` is returned as `Some(ckd_output)`.
5. Application calls `ckd_output.unmask(app_sk)` and receives `msk · H(pk‖app_id) + (S − app_sk · R)`, an incorrect and attacker-influenced group element, silently accepted as the legitimate confidential key.
