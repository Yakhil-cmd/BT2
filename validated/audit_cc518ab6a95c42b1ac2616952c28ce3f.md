### Title
Missing Zero Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary
Both `change_reward_address` in the staking contract and in the pool contract accept the zero address as a valid new reward address. When `claim_rewards` is subsequently called, accumulated yield is transferred to `address(0)` and permanently frozen.

---

### Finding Description
In `staking.cairo`, `change_reward_address` performs only one validation: that the new address is not a registered token address. There is no check that the address is non-zero. [1](#0-0) 

The identical gap exists in `pool.cairo`: [2](#0-1) 

Neither function contains `assert!(reward_address.is_non_zero(), ...)` or any equivalent guard. The zero address passes both checks (`does_token_exist` returns false for address(0), and `token_dispatcher.contract_address != address(0)` is trivially true).

The downstream consequence is in `send_rewards_to_staker`, which blindly transfers to whatever `reward_address` is stored: [3](#0-2) 

And in `pool.cairo`'s `claim_rewards`: [4](#0-3) 

In both cases, if `reward_address` is `0`, the ERC-20 transfer sends tokens to the zero address. On Starknet, no private key controls address(0), so those tokens are permanently unrecoverable.

The same gap exists at initial `stake()` time — `reward_address` is accepted without a non-zero check: [5](#0-4) 

---

### Impact Explanation
Any accumulated STRK yield (staker's own rewards or pool member rewards) sent to `address(0)` is permanently frozen. This matches the allowed impact: **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation
The entry point is a standard, unprivileged user-facing function callable by any active staker or pool member. A user could accidentally pass `0` (e.g., a front-end bug, a scripting error, or a mistaken default value). The call requires no special role, no bridge interaction, and no external dependency. Likelihood is **Medium** — accidental misuse is realistic and the protocol provides no safety net.

---

### Recommendation
Add a non-zero guard at the top of both `change_reward_address` implementations, and in `stake` / `enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
```

Apply the same guard to the `reward_address` parameter in `stake()` (staking.cairo ~line 307) and `enter_delegation_pool()` (pool.cairo ~line 170).

---

### Proof of Concept

**Staker path:**
1. Staker calls `staking.stake(reward_address: 0, ...)` — accepted, no zero-address check.
   OR staker calls `staking.change_reward_address(reward_address: 0)` after staking — also accepted.
2. Rewards accumulate in `staker_info.unclaimed_rewards_own`.
3. Anyone calls `staking.claim_rewards(staker_address)`.
4. `send_rewards_to_staker` executes `token.checked_transfer(recipient: 0, amount: rewards)`.
5. All accumulated STRK yield is sent to `address(0)` and permanently frozen.

**Pool member path:**
1. Pool member calls `pool.change_reward_address(reward_address: 0)` — accepted, no zero-address check.
2. Pool member calls `pool.claim_rewards(pool_member)`.
3. Line 366 executes `reward_token.checked_transfer(recipient: 0, amount: rewards)`.
4. All accumulated STRK yield is permanently frozen. [6](#0-5) [7](#0-6)
