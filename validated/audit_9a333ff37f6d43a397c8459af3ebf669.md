### Title
Revoking a key does not revoke duplicate key indices sharing the same public key bytes, allowing continued transaction signing after revocation — (`fvm/environment/accounts.go`, `fvm/environment/account-key-metadata/metadata.go`)

### Summary
When an account has two key indices (key_0 and key_1) that share the same underlying public key bytes (a "duplicate key"), revoking key_0 only sets the revoked flag on key_0's own storage. Key_1 retains an independent `revoked=false` bit. An attacker who holds the private key for key_0 can continue signing transactions using key_1's index, bypassing the revocation invariant.

### Finding Description
The deduplication system introduced in `AppendDuplicateKeyMetadata` stores per-index `revoked` and `weight` fields independently for each logical key index, even when two indices map to the same stored public key bytes. `RevokeAccountPublicKey` operates only on the specified key index and has no mechanism to propagate revocation to other indices that share the same underlying cryptographic material.

Both the sequence number checker (`transactionSequenceNum.go:56`) and the transaction verifier (`transactionVerifier.go:305`) check `Revoked` on the specific key index used in the transaction. Neither checks whether any other index sharing the same public key bytes has been revoked. [5](#0-4) [6](#0-5) 

### Impact Explanation
An account owner who revokes key_0 (e.g., after a suspected compromise) believes the corresponding private key can no longer authorize transactions. If key_1 (a duplicate of key_0 with weight ≥ 1000) was added before revocation, the private key remains usable via key_1. This violates the security invariant that revoking a cryptographic key prevents all future use of that key material for transaction signing, enabling unauthorized account control.

### Likelihood Explanation
The attacker must control the account (to add key_1 before revocation) and retain the private key after revoking key_0. This is a realistic scenario: an account owner adds a duplicate key for redundancy, later revokes the "original" index after a suspected leak, but the duplicate index remains active. The exploit requires no privileged node access, no admin control, and is reachable via standard transaction submission.

### Recommendation
When revoking a key at index `i`, the runtime should scan all other key indices and revoke any that share the same `storedKeyIndex` (i.e., the same underlying public key bytes). Alternatively, revocation should be stored at the `storedKeyIndex` level rather than the logical key index level, so that revoking any one index automatically revokes all indices sharing the same stored key.

### Proof of Concept
1. Create account with key_0 (ECDSA P-256, weight 1000).
2. Add key_1 as a duplicate of key_0 (same public key bytes, weight 1000) via `AppendAccountPublicKey`.
3. Revoke key_0 via `RevokeAccountKey(address, 0)`.
4. Verify: `GetAccountPublicKeyRevokedStatus(addr, 0)` → `true`; `GetAccountPublicKeyRevokedStatus(addr, 1)` → `false`.
5. Submit a transaction with `ProposalKey{Address: addr, KeyIndex: 1, SequenceNumber: 0}`, signed with key_0's private key using key_1's index.
6. Assert the transaction is accepted (sequence number check passes, signature verification passes using key_0's public key bytes retrieved via key_1's `storedKeyIndex`). [1](#0-0) [3](#0-2)
