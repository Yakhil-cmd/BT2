### Title
Missing Setter for `avg_block_duration` Config Field Causes Uncorrectable Reward Miscalculation - (File: src/reward_supplier/reward_supplier.cairo)

### Summary
The `RewardSupplier` contract stores `avg_block_duration` as a mutable storage field that directly controls per-block reward amounts, but exposes no public setter function for it. The codebase itself acknowledges this with a `// TODO: Setter.` comment. The sibling field `block_duration_config` (min/max bounds) does have a setter (`set_block_duration_config`), making the asymmetry a direct analog to the reported missing-update-function pattern.

### Finding Description
In `src/reward_supplier/reward_supplier.cairo`, the storage field `avg_block_duration` is initialized to `DEFAULT_AVG_BLOCK_DURATION` (300, representing 3 seconds × 100 scale) in the constructor and is updated automatically by the internal `set_avg_block_duration()` function each time `update_current_epoch_block_rewards()` is called. [1](#0-0) 

The field is used directly in the reward formula: [2](#0-1) 

The `IRewardSupplierConfig` interface exposes `set_block_duration_config` (for min/max bounds) but has no counterpart for `avg_block_duration`: [3](#0-2) 

The internal `set_avg_block_duration()` function skips the update on the very first call (when `snapshot_block_number` is zero), leaving `avg_block_duration` at the constructor default for the entire first epoch: [4](#0-3) 

The only non-automatic way to set `avg_block_duration` is via the upgrade EIC, which requires a full contract upgrade cycle: [5](#0-4) 

### Impact Explanation
`avg_block_duration` is the sole multiplier that converts `yearly_mint` into per-block rewards. For the entire first epoch after deployment or after an EIC upgrade, rewards are computed using the hardcoded default (300) rather than the actual Starknet block time. If actual block time deviates from 3 s (e.g., 2 s → 66 % of correct rewards; 5 s → 167 % of correct rewards), all stakers and pool members receive systematically wrong rewards for that epoch with no admin recourse short of another contract upgrade. Under-payment constitutes temporary freezing of unclaimed yield; over-payment constitutes theft of unclaimed yield. [6](#0-5) 

### Likelihood Explanation
This condition is triggered deterministically on every fresh deployment and on every EIC-based upgrade (the EIC itself sets `avg_block_duration` to a caller-supplied value, but the first post-upgrade epoch still uses that value without correction until the second `update_current_epoch_block_rewards` call). The protocol has already undergone multiple upgrades (V0→V1→V2→V3), making recurrence likely. [7](#0-6) 

### Recommendation
Add a privileged setter to `IRewardSupplierConfig` analogous to `set_block_duration_config`:

```rust
fn set_avg_block_duration(
    ref self: TContractState, avg_block_duration: u64,
) {
    self.roles.only_app_governor();
    let config = self.block_duration_config.read();
    assert!(
        config.min_block_duration <= avg_block_duration
            && avg_block_duration <= config.max_block_duration,
        "{}",
        Error::INVALID_AVG_BLOCK_DURATION,
    );
    self.avg_block_duration.write(avg_block_duration);
}
```

This mirrors the existing `set_block_duration_config` pattern and resolves the `// TODO: Setter.` note. [8](#0-7) 

### Proof of Concept

1. Deploy `RewardSupplier` (or apply the V3 EIC). `avg_block_duration` is written to `DEFAULT_AVG_BLOCK_DURATION = 300`.
2. The staking contract calls `update_current_epoch_block_rewards()` for the first time. Inside `set_avg_block_duration()`, `snapshot_block_number` is 0, so the function returns early — `avg_block_duration` stays at 300.
3. Rewards for epoch 1 are computed as `yearly_mint * 300 / (100 * SECONDS_IN_YEAR)`.
4. If actual Starknet block time is 5 s (500 in scaled units), correct rewards would be `yearly_mint * 500 / (100 * SECONDS_IN_YEAR)` — 67 % higher.
5. The `app_governor` calls `set_block_duration_config` to adjust bounds but has no function to correct `avg_block_duration` itself.
6. Epoch 1 closes with all stakers under-rewarded by up to 40 % (or over-rewarded by up to 67 %) with no on-chain remedy. [9](#0-8)

        fn set_avg_block_duration(ref self: ContractState) {
            let current_block_number = starknet::get_block_number();
            let current_timestamp = starknet::get_block_timestamp();
            let (snapshot_block_number, snapshot_timestamp) = self.block_snapshot.read();
            // Sanity asserts.
            assert!(
                current_block_number > snapshot_block_number,
                "{}",
                InternalError::INVALID_BLOCK_NUMBER,
            );
            assert!(
                current_timestamp > snapshot_timestamp.into(),
                "{}",
                InternalError::INVALID_BLOCK_TIMESTAMP,
            );
            self
                .block_snapshot
                .write((current_block_number, Timestamp { seconds: current_timestamp }));
            // If this is the first time we're setting the block snapshot, can't calculate avg block
            // time yet.
            if snapshot_block_number.is_zero() || snapshot_timestamp.is_zero() {
                return;
            }
            let time_delta = current_timestamp - snapshot_timestamp.into();
            // *Note*: `num_blocks` should match the epoch length in blocks. This calculation is
            // expected to run on the first block of each epoch, assuming `update_rewards` is called
            // every block.
            // We calculate `num_blocks` instead of using the configured value to keep the average
            // accurate even if some calls are missed.
            let num_blocks = current_block_number - snapshot_block_number;
            let mut calculated_block_duration = mul_wide_and_div(
                lhs: time_delta, rhs: BLOCK_DURATION_SCALE, div: num_blocks,
            )
                .expect_with_err(err: Error::BLOCK_DURATION_OVERFLOW);
            // Adjust calculated_block_duration with min and max block duration.
            let block_duration_config = self.block_duration_config.read();
            calculated_block_duration = calculated_block_duration
                .clamp(
                    block_duration_config.min_block_duration,
                    block_duration_config.max_block_duration,
                );
            self.avg_block_duration.write(calculated_block_duration);
        }
