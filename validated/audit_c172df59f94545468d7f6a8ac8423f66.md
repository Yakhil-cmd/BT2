### Title
Unauthenticated `from: Participant` in `Protocol::message` Enables Sender Impersonation to Corrupt or Abort DKG - (File: `src/protocol/mod.rs`)

---

### Summary

The `Protocol::message(from: Participant, data: MessageData)` API accepts a caller-supplied sender identity with no cryptographic authentication. Any entity that can deliver messages to a participant's protocol instance — including a malicious coordinator or message router — can inject messages with an arbitrary spoofed `from` field. In DKG, this allows an attacker to pre-fill a victim's slot for a target participant, causing the honest party to process attacker-crafted data attributed to an innocent participant, fail share validation, and permanently abort key generation while falsely blaming the innocent party.

---

### Finding Description

The `Protocol` trait in `src/protocol/mod.rs` defines the message delivery interface:

```rust
fn message(&mut self, from: Participant, data: MessageData);
```

The `from: Participant` value is a plain `u32` wrapper with no associated cryptographic credential. The implementation in `src/protocol/internal.rs` unconditionally trusts it:

```rust
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
```

`push_message` routes the message into the internal `MessageBuffer` keyed by `(channel_tag, waitpoint)`, tagging it with the caller-supplied `from`. No signature, MAC, or any other proof of identity is checked.

In `src/protocol/helpers.rs`, `recv_from_others` uses this `from` field to deduplicate and collect exactly one message per participant:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

`ParticipantCounter::put` returns `false` for any participant already seen, so the first message tagged with a given `from` wins — subsequent messages from the real participant are silently dropped.

In `do_keyshare` in `src/dkg.rs`, this function is used to receive private signing shares (Round 5):

```rust
for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?
{
    let full_commitment_from = all_full_commitments.index(from)?;
    validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
    my_signing_share = my_signing_share + signing_share_from.to_scalar();
}
```

An attacker who can call `victim_protocol.message(target_participant, crafted_bytes)` before the real message from `target_participant` arrives will:

1. Pre-fill `target_participant`'s slot in `seen` with attacker-crafted data.
2. Cause `validate_received_share` to fail (the crafted share will not satisfy the commitment polynomial of `target_participant` established in Round 3).
3. The victim returns `Err(ProtocolError::InvalidSecretShare(target_participant))`, permanently aborting DKG and attributing the failure to the innocent `target_participant`.

The same path applies to the commitment hash round (`recv_from_others` at `wait_round_1`) and the `broadcast_success` final round, both of which also use unauthenticated `from` routing.

---

### Impact Explanation

A malicious coordinator or any entity controlling the message delivery layer can permanently abort DKG, reshare, or refresh for all honest parties by injecting a single spoofed message per victim per target participant. Because the first message wins and the real message is dropped, the honest party cannot recover without restarting the entire protocol. This maps to:

**High: Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs.**

Additionally, if the attacker can craft a share that passes `validate_received_share` (e.g., by knowing the target's commitment and solving for a consistent but wrong scalar — not trivially possible but structurally enabled by the missing authentication), the impact escalates to **corruption of the resulting key share**, which is Critical.

---

### Likelihood Explanation

The `Protocol` trait is the sole public interface for message delivery. Any integrator that routes messages through a coordinator (the common deployment pattern for threshold protocols) gives that coordinator the ability to supply arbitrary `from` values. A malicious or compromised coordinator, or a network-level attacker with message injection capability, can trivially exploit this. No cryptographic break is required — only the ability to call `protocol.message(spoofed_from, data)` before the legitimate message arrives.

---

### Recommendation

The library should either:

1. **Authenticate the `from` field at the transport layer and document this as a hard requirement** — add explicit documentation to `Protocol::message` stating that callers MUST guarantee the `from` field is cryptographically authenticated (e.g., via TLS mutual auth or a signed envelope), and provide a reference implementation.
2. **Embed sender authentication inside the protocol messages** — include a participant-specific signature or MAC over `(channel_tag, waitpoint, payload)` inside `MessageData`, verified inside `push_message` or `recv`, so the `from` field is corroborated by the message content itself.

Option 2 is preferable because it removes the implicit trust assumption on the transport layer and makes the security property self-contained within the library.

---

### Proof of Concept

```
Setup: 3-of-3 DKG with participants A (victim), B (honest), M (malicious coordinator).

1. M starts the DKG protocol for A and B normally through Rounds 1–3.
   A and B broadcast their commitments; all parties receive valid commitments.

2. In Round 4 (private share delivery), B sends its legitimate share to A via
   Action::SendPrivate(A, real_share_bytes).

3. M, acting as message router, intercepts real_share_bytes and instead calls:
       protocol_A.message(B, crafted_garbage_bytes)
   before delivering B's real message.

4. Inside recv_from_others on wait_round_3, A's ParticipantCounter marks B as seen.
   When M later delivers B's real message, seen.put(B) returns false → dropped.

5. A calls validate_received_share(me=A, from=B, crafted_garbage, commitment_B).
   The crafted scalar does not satisfy B's committed polynomial → returns
   Err(ProtocolError::InvalidSecretShare(B)).

6. DKG is permanently aborted for A. B is falsely blamed.
   M can repeat this for every honest participant, preventing any key from being generated.

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
```
