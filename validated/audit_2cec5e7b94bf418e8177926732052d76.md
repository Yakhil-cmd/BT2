### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Factory's Fee-Cap Invariant — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
The factory enforces explicit upper-bound caps on every admin-controlled fee parameter **except** the per-bin additional fees (`addFeeBuyE6` / `addFeeSellE6`). A pool admin can set these to the full `uint16` maximum (65 535 ≡ 6.5535 % in E6 units) with no factory-level guard, violating the design invariant that all semi-trusted admin fee parameters are bounded by factory-enforced caps.

### Finding Description
The factory defines hard caps for every other admin-controlled fee:

- `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) — enforced in `setPoolAdminFees` [1](#0-0) 
- `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1 %) — enforced in `setPoolAdminFees` [2](#0-1) 

`setPoolBinAdditionalFees`, however, passes the caller-supplied values straight through to the pool with **zero validation**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool-level `setBinAdditionalFees` likewise applies no cap beyond the bin-index range check:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [4](#0-3) 

These per-bin fees are then added directly to the oracle-derived base fee in every swap step:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [5](#0-4) 

and in `getSellAndBuyPrices`:

```solidity
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [6](#0-5) 

The `uint16` type provides an implicit ceiling of 65 535 (6.5535 %), but no factory constant, no `maxAdminBinFeeE6` state variable, and no revert guard exist to enforce a lower, policy-consistent bound.

### Impact Explanation
The pool admin is explicitly described as **semi-trusted only inside caps and timelocks**. The factory's fee-cap system is the mechanism that enforces this boundary. Because `setPoolBinAdditionalFees` has no cap check, the pool admin can unilaterally raise the effective swap fee for any individual bin to `oracle_spread + 6.5535 %`, exceeding the 20 % hard cap that governs the spread fee. This is an admin-boundary break: the semi-trusted role can push a fee parameter beyond the policy maximum without any factory-level guard, making the fee-cap invariant incomplete and bypassable for the bin-fee dimension.

### Likelihood Explanation
Any pool admin — a role reachable by the pool creator at deployment — can call `setPoolBinAdditionalFees` at any time with no timelock. The call path is short and requires no special preconditions beyond holding the pool-admin role.

### Recommendation
Introduce a `HARD_MAX_BIN_ADDITIONAL_FEE_E6` constant (e.g., `200_000`, consistent with the spread-fee hard cap) and enforce it in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
uint24 internal constant HARD_MAX_BIN_ADDITIONAL_FEE_E6 = 200_000;

function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > HARD_MAX_BIN_ADDITIONAL_FEE_E6) revert BinAdditionalFeeTooHigh();
    if (addFeeSellE6 > HARD_MAX_BIN_ADDITIONAL_FEE_E6) revert BinAdditionalFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept
1. Pool is deployed with `maxAdminSpreadFeeE6 = 200_000` (20 %) and any valid configuration.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Factory forwards the call; pool stores `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` for bin 0 with no revert.
4. Any subsequent swap routed through bin 0 incurs an effective fee of `oracle_half_spread + 6.5535 %` — a fee level that the factory's cap system was designed to prevent but fails to enforce for this parameter.
5. No other factory function, timelock, or on-chain guard can retroactively block or clamp the stored value.
