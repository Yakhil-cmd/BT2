### Title
`skip_stateful_validations` accepts any mempool transaction as proof of `deploy_account`, allowing invalid-signature invoke transactions to bypass gateway `__validate__` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to skip `__validate__` for an invoke transaction with nonce=1 when `account_nonce==0`, to support the UX pattern of sending `deploy_account` and the first invoke simultaneously. The guard condition `account_tx_in_pool_or_recent_block` is supposed to confirm a `deploy_account` exists for the account, but it returns `true` for **any** transaction for that address in the pool or recent block. An attacker who observes a victim's `deploy_account` in the mempool can submit an invoke(nonce=1) with an invalid/forged signature, bypass `__validate__` at the gateway, get the transaction admitted to the mempool, and have it executed by the batcher — consuming the victim's first nonce slot and charging a fee to the victim's account.

### Finding Description

`skip_stateful_validations` in `stateful_transaction_validator.rs` fires when:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        ...
``` [1](#0-0) 

When it returns `true`, `run_validate_entry_point` is called with `skip_validate=true`, meaning the account's `__validate__` entry point is never executed at gateway admission time. [2](#0-1) 

The `account_tx_in_pool_or_recent_block` implementation returns `true` if the account has **any** transaction in the pool or any committed/staged nonce in the mempool state — not specifically a `deploy_account`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

The code comment acknowledges this: *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second branch is the problem: if the legitimate account owner submitted invoke(nonce=2) before invoke(nonce=1) (a common pattern when submitting multiple transactions simultaneously), an attacker can submit invoke(nonce=1) with an invalid signature. The gateway sees nonce=2 in the pool, concludes the account is controlled by its owner, and skips `__validate__` for the attacker's nonce=1 transaction.

The attacker's transaction is then admitted to the mempool and forwarded to the batcher. The batcher runs the full execution path, which **does** call `__validate__`. `__validate__` fails (invalid signature), but by this point `handle_nonce` in `perform_pre_validation_stage` has already incremented the nonce:

```rust
pub fn perform_pre_validation_stage<S: State + StateReader>(
    &self,
    state: &mut S,
    tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    let tx_info = &tx_context.tx_info;
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
``` [4](#0-3) 

In Starknet's execution model, a transaction that fails `__validate__` is still included in the block as a failed transaction, the nonce is consumed, and a fee is charged to the sender's account. The victim's legitimate invoke(nonce=1) is now permanently invalid (nonce too old).

### Impact Explanation

**High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An attacker can inject an invalid-signature invoke transaction for any victim account that has a future-nonce transaction in the mempool (or a `deploy_account` in flight). The transaction bypasses the gateway's `__validate__` check, is admitted to the mempool, and when executed by the batcher:
- Consumes the victim's nonce slot
- Charges a fee to the victim's account
- Blocks the victim's legitimate transaction at that nonce

### Likelihood Explanation

**Medium.** The attack requires:
1. The victim's account has `account_nonce==0` in the latest committed block state.
2. Any transaction for the victim's address is in the mempool pool or recent committed state.

Condition 2 is trivially satisfied during the `deploy_account` + first-invoke UX flow, which is the exact scenario this code path was designed for. An attacker monitoring the public mempool can observe `deploy_account` transactions and immediately submit a fake invoke(nonce=1) before the victim does. No special privileges are required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a specific check that only returns `true` when a `deploy_account` transaction for the address is present in the pool. Introduce a dedicated `deploy_account_in_pool(address)` query on the mempool, or store the transaction type alongside the pool entry and filter accordingly. The current check conflates "account is known to the mempool" with "a deploy_account exists for this account," which are not equivalent.

### Proof of Concept

1. Victim submits `deploy_account` for address `X` (account_nonce=0 in committed state).
2. Attacker observes `deploy_account` in the public mempool.
3. Attacker constructs `invoke(sender=X, nonce=1, calldata=<drain_funds>, signature=<garbage>)`.
4. Gateway calls `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)` returns `true` (deploy_account is in pool).
5. `skip_stateful_validations` returns `true` → `run_validate_entry_point` skips `__validate__`. [5](#0-4) 

6. Attacker's fake invoke(nonce=1) is admitted to the mempool and forwarded to the batcher.
7. Batcher executes `deploy_account` → account `X` is deployed.
8. Batcher executes attacker's invoke(nonce=1): `perform_pre_validation_stage` increments nonce to 1→2; `__validate__` runs and fails (invalid signature); transaction is included in block as failed; fee charged to `X`; nonce=1 consumed.
9. Victim submits their legitimate invoke(nonce=1) → rejected: `InvalidNonce` (account nonce is now 2).

The victim's first transaction is permanently blocked at nonce=1, a fee has been drained from their account, and they must resubmit at nonce=2 — with the attacker able to repeat the attack indefinitely for each new nonce.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
