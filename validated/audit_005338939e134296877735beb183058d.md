### Title
Unbounded `stakers` Vec Growth Causes Unbounded Gas Consumption in `get_stakers` - (File: `src/staking/staking.cairo`)

### Summary
The `stakers` storage vector in the Staking contract grows monotonically as new stakers register and is **never pruned** when stakers unstake. The `get_stakers` function iterates over the entire vector on every call. Any unprivileged user can register as a staker (paying `min_stake`) to append an entry to this vector, causing `get_stakers` to consume ever-increasing gas until it becomes impractical or exceeds execution resource limits.

### Finding Description

The `stakers` field is declared as a `Vec<ContractAddress>` in storage: [1](#0-0) 

The comment explicitly acknowledges that stakers are **never removed** from this vector:

> `stakers: Vec<ContractAddress>` — **Note**: Stakers are not removed from this vector when they unstake.

Every call to `stake()` appends the new staker address unconditionally: [2](#0-1) 

The `get_stakers` function then iterates over the **full range** of this vector on every invocation: [3](#0-2) 

For each entry, it performs multiple storage reads and computations (`is_staker_active`, `get_staker_staking_power_at_epoch`, `get_public_key_at_epoch`, `get_peer_id_at_epoch`). The cost scales linearly with the total number of stakers ever registered, including those who have long since unstaked and exited.

### Impact Explanation

`get_stakers` is the consensus-critical view function used by Starknet consensus nodes to determine the active validator set for a given epoch. As the `stakers` vector grows without bound, the gas cost of `get_stakers` grows linearly. Eventually:

1. On-chain callers (any contract invoking `get_stakers` via `call_contract`) will hit execution resource limits and revert.
2. Off-chain consensus nodes will face increasing latency and resource consumption, degrading the ability to determine the validator set.

This matches the **Medium: Unbounded gas consumption** impact category.

### Likelihood Explanation

Any unprivileged address can call `stake()` with `min_stake` tokens to append a permanent entry to the `stakers` vector. The cost per entry is `min_stake` (a protocol-defined minimum), which is non-trivial but finite. A motivated attacker can register many staker addresses over time. Additionally, organic protocol growth naturally inflates the vector since legitimate stakers who exit are never removed. No privileged access is required.

### Recommendation

1. **Remove exited stakers from the `stakers` Vec**: When a staker completes `unstake_action`, swap-and-pop their address from the `stakers` vector, or maintain a separate active-staker count/set.
2. **Alternatively, use a paginated iterator**: Add an offset/limit parameter to `get_stakers` so callers can retrieve results in bounded chunks.
3. **Track active staker count separately**: Maintain a separate `active_stakers: Vec<ContractAddress>` that is pruned on exit, keeping `stakers` only for historical reference.

### Proof of Concept

An attacker registers `N` staker addresses (each with `min_stake` tokens) across separate accounts. Each call to `stake()` appends to `stakers`:

```
for i in 0..N:
    stake(staker_address=attacker_wallet[i], amount=min_stake, ...)
    // stakers.push(attacker_wallet[i]) is called each time
```

After `N` registrations, a call to `get_stakers(epoch_id)` must iterate over all `N` entries (plus all legitimate stakers), performing multiple storage reads per entry. At sufficient `N`, the call exceeds Starknet's per-transaction resource limits, causing it to revert or become impractical for consensus nodes.

The `stakers` vector is confirmed to never shrink: [1](#0-0) 

And `get_stakers` confirms the full-range unbounded iteration: [4](#0-3)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
