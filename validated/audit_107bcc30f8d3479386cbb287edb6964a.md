### Title
Permanent Loss of ERC-20 Tokens When `ExitToEthereum` NEAR Promise Fails Due to Missing Failure Callback - (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns ERC-20 tokens on the Aurora EVM side before the NEAR-side `withdraw` promise executes, and unlike `ExitToNear`, it **never attaches a failure callback**. If the NEAR `withdraw` call fails for any reason, the user's ERC-20 tokens are permanently destroyed with no recovery path.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToEthereum` burns the caller's tokens first, then calls the `ExitToEthereum` precompile:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← tokens destroyed here
    // ... then calls ExitToEthereum precompile
}
```

The `ExitToEthereum` precompile (`engine-precompiles/src/native.rs`) then schedules a NEAR promise to call `withdraw` on the ETH connector contract. Critically, the promise is always constructed as a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

Contrast this with `ExitToNear`, which (when the `error_refund` feature is enabled) attaches a `exit_to_near_precompile_callback` that re-mints burned tokens on failure via `engine::refund_on_error`. The `ExitToEthereum` path has no equivalent — no callback, no re-mint, no recovery.

The `ExitToNear` callback structure for comparison:

```rust
PromiseArgs::Callback(PromiseWithCallbackArgs {
    base: transfer_promise,
    callback: PromiseCreateArgs {
        method: "exit_to_near_precompile_callback".to_string(),
        ...
    },
})
```

`ExitToEthereum` has no such branch at all.

---

### Impact Explanation

If the NEAR `withdraw` promise fails (ETH connector paused, gas exhaustion, connector accounting mismatch, or any NEAR-side error), the user's ERC-20 tokens are permanently burned on Aurora with no corresponding release on the NEAR/Ethereum side. This is a **permanent, irrecoverable loss of user funds** — the exact analog of the `BurnUnlock` vulnerability in the report.

---

### Likelihood Explanation

The NEAR `withdraw` call can fail due to:
- The ETH connector contract being paused (an admin-controlled but externally observable state)
- Insufficient attached gas (`WITHDRAWAL_GAS = 100_000_000_000_000` TGas) for complex connector logic
- ETH connector balance accounting errors
- Any NEAR runtime-level failure

Any EVM user holding a bridged ERC-20 token can trigger this path by calling `withdrawToEthereum`. The failure scenario does not require attacker privilege — it can occur due to ordinary operational conditions.

---

### Recommendation

Attach a failure callback to the `ExitToEthereum` promise, mirroring the `error_refund` pattern already implemented for `ExitToNear`. On callback failure, call `engine::refund_on_error` to re-mint the burned ERC-20 tokens (or restore the ETH balance) to the original sender.

---

1. Alice holds 1000 units of a bridged ERC-20 on Aurora.
2. Alice calls `withdrawToEthereum(eth_address, 1000)` — 1000 tokens are burned on Aurora EVM.
3. The NEAR `withdraw` promise is scheduled but fails (e.g., ETH connector is paused).
4. No callback fires. No re-mint occurs.
5. Alice has 0 ERC-20 tokens on Aurora and 0 ETH on Ethereum — funds are permanently lost.

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
