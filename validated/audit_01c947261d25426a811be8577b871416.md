### Title
Re-initialization via Corrupted State Bypasses `is_ok()` Guard in `new` — (`engine/src/contract_methods/admin.rs`)

### Summary
The `new` function guards against re-initialization with `state::get_state(&io).is_ok()`. This check only blocks when state is successfully deserialized. When the STATE storage key contains bytes that fail Borsh deserialization (`Err(DeserializationFailed)`), `is_ok()` returns `false`, the guard is skipped, and any caller can overwrite the engine's `owner_id`, `chain_id`, and `upgrade_delay_blocks` with attacker-controlled values.

### Finding Description
`state::get_state` returns three distinct outcomes:
- `Err(NotFound)` — key absent (correct case for first-time init)
- `Ok(state)` — key present and valid (guard correctly blocks re-init)
- `Err(DeserializationFailed)` — key present but bytes fail both `BorshableEngineState` and `BorshableEngineStateV1` deserialization

The guard `if state::get_state(&io).is_ok()` treats `Err(NotFound)` and `Err(DeserializationFailed)` identically — both bypass the guard. The correct check should be `if !matches!(get_state(&io), Err(EngineStateError::NotFound))` (i.e., block re-init unless state is provably absent).

The `DeserializationFailed` scenario is explicitly anticipated: the error variant's doc comment reads *"possibly due to failed state migration"*, and the `state_migration` entrypoint has a `// TODO: currently we don't have migrations` comment, confirming future migrations are planned. A buggy `state_migration` writing malformed bytes to the STATE key is a realistic non-malicious operational failure.

### Impact Explanation
Once `new` succeeds with attacker-controlled args, the attacker becomes `owner_id`. They can immediately call:
- `attach_full_access_key` — adds a full-access key to the engine account, enabling arbitrary on-chain actions
- `upgrade` — deploys attacker-controlled WASM
- `set_owner` — locks out the legitimate owner

This constitutes complete engine takeover and direct theft of all user funds held in the engine.

### Likelihood Explanation
The precondition is a corrupted STATE key. In the current production code, `state_migration` is a no-op, so the precondition requires either (a) a future upgrade whose `state_migration` has a bug writing malformed bytes, or (b) any other code path that writes to the STATE key with invalid borsh. The developers explicitly anticipated scenario (a) by naming the error `DeserializationFailed` with the migration comment. The window between state corruption and attacker exploitation is a race, but `new` has no access control, so any NEAR account can win it.

### Recommendation
Replace the `is_ok()` check with an explicit `NotFound` check:

```rust
// Before (vulnerable):
if state::get_state(&io).is_ok() {
    return Err(b"ERR_ALREADY_INITIALIZED".into());
}

// After (correct):
match state::get_state(&io) {
    Ok(_) => return Err(b"ERR_ALREADY_INITIALIZED".into()),
    Err(EngineStateError::NotFound) => {} // proceed with initialization
    Err(_) => return Err(b"ERR_ALREADY_INITIALIZED".into()), // corrupted state also blocks re-init
}
```

This ensures that any bytes present at the STATE key — valid or corrupted — prevent re-initialization.
