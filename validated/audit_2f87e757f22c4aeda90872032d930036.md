### Title
Pool Admin Bypasses All Timelocks via uint32 Overflow in `_afterTimelock` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
`_afterTimelock` computes `uint32(block.timestamp + timelock)`. When `timelock = type(uint32).max`, the addition overflows uint32 and wraps to a timestamp in the past. `_requireElapsed` then passes immediately for every subsequent proposal, allowing the pool admin to execute any parameter change (drawdown, decay, watermarks) with zero delay, bypassing the LP protection window entirely.

### Finding Description
`_afterTimelock` returns the timestamp after which a pending change may be executed: [1](#0-0) 

The result is cast to `uint32`. When `oracleStopLossConfig[pool_].timelock == type(uint32).max`, the sum `block.timestamp + type(uint32).max` exceeds `type(uint32).max` and wraps to `block.timestamp - 1` (a past timestamp). `_requireElapsed` then trivially passes: [2](#0-1) 

`proposeOracleStopLossTimelock` accepts any `uint32` value with no upper-bound validation: [3](#0-2) 

Similarly, `initialize` validates `drawdownE6` and `decayPerSecondE8` but applies no cap to `timelock`: [4](#0-3) 

Once `type(uint32).max` is installed as the active timelock, every call to `_afterTimelock` — used by `proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, and `proposeOracleStopLossHighWatermarks` — produces a wrapped past timestamp, so the corresponding `execute*` call succeeds in the same block as the `propose*` call. [5](#0-4) 

### Impact Explanation
The timelock is the sole LP protection mechanism in this extension. LPs join pools expecting that any weakening of stop-loss parameters (raising drawdown tolerance, accelerating watermark decay, zeroing watermarks) will be visible on-chain for the full timelock window before taking effect. With the bypass active, the pool admin can:

- Immediately set `drawdownE6 = 0`, disabling the stop-loss entirely, exposing LPs to unbounded value drain through swaps.
- Immediately set `decayPerSecondE8 = E8` (maximum), causing watermarks to decay to zero within one second, making the stop-loss permanently ineffective.
- Immediately set per-bin watermarks to zero, removing all high-watermark floors.

All three actions remove the stop-loss guard that `afterSwap` relies on to revert value-draining swaps, directly exposing LP principal to loss.

### Likelihood Explanation
The pool admin role is semi-trusted and is explicitly constrained to act "only inside caps and timelocks" per the contest scope. The attack requires only two admin transactions (propose + execute timelock to `type(uint32).max`, then propose + immediately execute any parameter change). No external oracle manipulation, no special token behavior, and no factory owner involvement is needed. Any pool admin — including one who turns adversarial after deployment — can execute this.

### Recommendation
Add an upper-bound validation for the timelock value, consistent with how `drawdownE6` and `decayPerSecondE8` are validated:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days; // example cap

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Call `_validateTimelock` in both `initialize` and `proposeOracleStopLossTimelock`. Alternatively, use `uint256` arithmetic with an explicit overflow check before casting:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    require(result <= type(uint32).max, "timelock overflow");
    return uint32(result);
}
```
