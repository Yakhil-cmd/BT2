### Title
Notional Fee Accumulator Unconditionally Cleared Before Floor-Rounding Conversion Causes Permanent Protocol Fee Loss — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

`collectFees` resets `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` to zero unconditionally after computing external token amounts with `Math.Rounding.Floor`. When the accumulated scaled notional fees are smaller than `TOKEN_X_SCALE_MULTIPLIER`, the floor division produces zero external tokens transferred, yet the accumulator is still wiped. The fees are permanently destroyed — they cannot be recovered in a future collection because the accumulator has already been zeroed. `collectPoolFees` in the factory is permissionless, so any caller can trigger this at will.

### Finding Description

In `MetricOmmPool.collectFees`, after computing the split between admin and protocol, the code converts scaled amounts to external token units using floor rounding:

```solidity
(uint256 totalFee0ToAdmin, uint256 totalFee1ToAdmin) =
    deltasScaledToExternal(totalFee0ToAdminScaled, totalFee1ToAdminScaled, Math.Rounding.Floor);
(uint256 totalFee0ToProtocol, uint256 totalFee1ToProtocol) =
    deltasScaledToExternal(totalFee0ToProtocolScaled, totalFee1ToProtocolScaled, Math.Rounding.Floor);
```

`deltasScaledToExternal` with `Floor` performs integer division:

```solidity
deltaAmount0 = scaledDeltaAmount0 / TOKEN_0_SCALE_MULTIPLIER;
```

Immediately after, regardless of whether any tokens were actually transferred:

```solidity
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;
```

For a token with fewer than 18 decimals, `TOKEN_0_SCALE_MULTIPLIER = 10^(18 - decimals)`. For USDC (6 decimals) this is `1e12`. Any accumulated notional fee below `1e12` scaled units (i.e., below 1 USDC in external terms) produces `totalFee0ToProtocol = 0`, no transfer occurs, but the accumulator is zeroed. The scaled fees are permanently lost.

The factory's `collectPoolFees` carries no access control:

```solidity
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
}
```

The interface comment explicitly states: *"Callable by any address (keepers, admins, or bots)."* An attacker can call `collectPoolFees` after every swap, ensuring the accumulator never grows large enough to survive the floor conversion.

### Impact Explanation

Every call to `collectPoolFees` when `notionalFeeToken0Scaled < TOKEN_0_SCALE_MULTIPLIER` (or the token1 analog) permanently destroys up to `TOKEN_0_SCALE_MULTIPLIER - 1` scaled units of notional fees. For USDC/USDT pools (6 decimals, `TOKEN_0_SCALE_MULTIPLIER = 1e12`), this is up to just under 1 token per call. On low-gas L2 chains (Base, HyperEVM — the protocol's stated targets), an attacker can profitably call this after every small swap, draining all notional fees. Even without a deliberate attacker, keeper bots that call `collectPoolFees` frequently will cause the same loss naturally. The notional fee accumulator is the only fee type that is explicitly zeroed; spread fees are not explicitly cleared and therefore survive across calls.

### Likelihood Explanation

The condition is reachable on any pool that:
1. Uses a token with fewer than 18 decimals (USDC, USDT — both explicitly in scope per the contest rules)
2. Has a non-zero notional fee configured
3. Receives small swaps, or has `collectPoolFees` called frequently

`collectPoolFees` requires no privilege. On L2 chains with sub-cent gas, the attack is economically viable. Even without a deliberate attacker, the loss occurs naturally whenever a keeper or bot calls `collectPoolFees` before the accumulator crosses the `TOKEN_X_SCALE_MULTIPLIER` threshold.

### Recommendation

Only zero the notional fee accumulators when the corresponding external transfer amount is non-zero, or track the residual and carry it forward:

```solidity
// Only clear what was actually converted and transferred
uint256 residual0 = totalFee0ToProtocolScaled + totalFee0ToAdminScaled
    - (totalFee0ToProtocol + totalFee0ToAdmin) * TOKEN_0_SCALE_MULTIPLIER;
notionalFeeToken0Scaled = residual0.toUint128();
```

Alternatively, skip the accumulator reset entirely when `totalFee0ToProtocol == 0 && totalFee0ToAdmin == 0` for the notional component, so the dust accumulates until it is large enough to transfer.

### Proof of Concept

Setup: USDC/WETH pool, `TOKEN_0_SCALE_MULTIPLIER = 1e12`, `protocolNotionalFeeE8 = 1_000_000` (1%), `adminNotionalFeeE8 = 0`, `spreadFeeE6 = 0`.

1. LP adds liquidity.
2. Trader swaps 9 USDC out (9e6 external = 9e18 scaled). Notional fee on output: `9e18 * 1e6 / 1e8 = 9e16` scaled units → `notionalFeeToken0Scaled = 9e16`.
3. Attacker calls `factory.collectPoolFees(pool)`.
   - `notionalFee0ToProtocolScaled = 9e16`
   - `totalFee0ToProtocol = 9e16 / 1e12 = 90_000` external units (90,000 USDC micro-units = $0.09) — transferred correctly.
4. Trader swaps 0.000009 USDC out (9 external = 9e12 scaled). Notional fee: `9e12 * 1e6 / 1e8 = 9e10` scaled units → `notionalFeeToken0Scaled = 9e10`.
5. Attacker immediately calls `factory.collectPoolFees(pool)`.
   - `notionalFee0ToProtocolScaled = 9e10`
   - `totalFee0ToProtocol = 9e10 / 1e12 = 0` — **no transfer**.
   - `notionalFeeToken0Scaled = 0` — **9e10 scaled units permanently destroyed**.

Repeating step 4–5 after every small swap causes all notional fees from small trades to be silently discarded. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    returns (uint256 deltaAmount0, uint256 deltaAmount1)

    if (rounding == Math.Rounding.Ceil) {
      deltaAmount0 = Math.ceilDiv(scaledDeltaAmount0, TOKEN_0_SCALE_MULTIPLIER);
      deltaAmount1 = Math.ceilDiv(scaledDeltaAmount1, TOKEN_1_SCALE_MULTIPLIER);
