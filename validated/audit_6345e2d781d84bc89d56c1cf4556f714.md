### Title
Immutable extensions with no disable mechanism permanently DoS `removeLiquidity` when a registered extension reverts — (`metric-core/contracts/ExtensionCalling.sol`, `metric-core/contracts/libraries/CallExtension.sol`)

### Summary

Extensions are baked into pool immutables at deployment and can never be removed or bypassed. `CallExtension.callExtension` propagates every revert from an extension directly to the caller with no try/catch. `removeLiquidity` invokes `_beforeRemoveLiquidity` and `_afterRemoveLiquidity` through this path, but unlike `swap`, it carries no `whenNotPaused` guard. If any extension registered on those hooks permanently reverts — due to an external dependency failure, self-destruct, or any other cause — every LP in the pool is permanently unable to withdraw their tokens, with no admin recovery path.

### Finding Description

**Extensions are stored as immutables and cannot be changed post-deployment.** [1](#0-0) 

`EXTENSION_1` through `EXTENSION_7` are set in the constructor and are `immutable`. There is no factory function, admin function, or protocol-owner function that can replace, zero-out, or skip an extension after the pool is live.

**`CallExtension.callExtension` propagates reverts unconditionally.** [2](#0-1) 

If `extension.call(data)` returns `success == false`, the library re-reverts with the extension's own revert data (or `ExtensionCallFailed` if there is none). There is no try/catch, no skip-on-failure, and no fallback path.

**`removeLiquidity` calls extensions but has no pause guard.** [3](#0-2) 

`_beforeRemoveLiquidity` and `_afterRemoveLiquidity` are called unconditionally. The `whenNotPaused` modifier is present only on `swap`: [4](#0-3) 

The interface explicitly documents this asymmetry: *"Pause level is factory-controlled; liquidity mutators are intentionally not blocked by the same check."* [5](#0-4) 

**The pool admin has no mechanism to bypass or disable extensions.**

The full set of pool-admin actions is: [6](#0-5) 

`pausePool` only sets `pauseLevel = 1`, which only gates `swap`. There is no `disableExtension`, no `bypassExtension`, and no emergency withdrawal path that skips the extension chain.

**`BaseMetricExtension` defaults all unimplemented hooks to `revert ExtensionNotImplemented()`.** [7](#0-6) 

Any extension that inherits `BaseMetricExtension` and is registered on `beforeRemoveLiquidity` or `afterRemoveLiquidity` without overriding those functions will always revert. `ValidateExtensionsConfig` does not verify that the extension actually implements the hooks it is registered for — it only checks address uniqueness and order index bounds: [8](#0-7) 

### Impact Explanation

If any extension registered on `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` permanently reverts, every call to `removeLiquidity` on that pool reverts. LP shares remain recorded in `_positionBinShares` and `_binTotalShares`, but they can never be redeemed. The underlying token balances are locked in the pool contract with no recovery path. This is a direct, permanent loss of LP principal — matching the "unusable withdraw/liquidity flows" and "loss of user principal" impact gates.

### Likelihood Explanation

The trigger requires a pool to be deployed with an extension registered on `beforeRemoveLiquidity` or `afterRemoveLiquidity`. This is a valid, documented configuration path — `ValidateExtensionsConfig` accepts it, the factory wires it, and the `IMetricOmmExtensions` interface defines those hooks. Once such a pool exists and LPs have deposited, any of the following causes a permanent DoS:

- The extension reads from an external oracle or contract that is later paused, killed, or self-destructed.
- The extension's own admin tightens a parameter (e.g., a drawdown threshold) to a value that always triggers a revert.
- The extension contract is upgraded to a broken implementation (if it is behind a proxy).
- The extension inherits `BaseMetricExtension` and the `beforeRemoveLiquidity`/`afterRemoveLiquidity` override is accidentally omitted, causing immediate permanent revert from the first `removeLiquidity` call.

The pool creator is an unprivileged actor (anyone can call `createPool`). The pool admin is semi-trusted only within caps and timelocks, and has no capability to fix this.

### Recommendation

1. **Add an emergency extension bypass on `removeLiquidity`**: Allow the protocol owner (or a timelocked admin path) to set a per-pool flag that skips extension calls on `removeLiquidity`, preserving LP withdrawal access even when an extension is broken.
2. **Wrap extension calls on withdrawal hooks in try/catch**: For `beforeRemoveLiquidity` and `afterRemoveLiquidity`, catch extension reverts and emit an event rather than propagating the revert, so LP withdrawals are never permanently blocked by extension failures.
3. **Validate hook implementation at pool creation**: In `createPool`, call each extension's `beforeRemoveLiquidity` and `afterRemoveLiquidity` with a dry-run (or check via ERC-165 / selector introspection) to confirm the extension actually implements the hooks it is registered for, reverting if it would always revert.

### Proof of Concept

1. A pool is deployed via `createPool` with `extensionOrders.beforeRemoveLiquidity` set to reference `EXTENSION_1`, where `EXTENSION_1` is a legitimate `BaseMetricExtension`-derived contract that reads from an external allowlist oracle to gate LP withdrawals.
2. LPs add liquidity; their shares are recorded in `_positionBinShares`.
3. The external allowlist oracle is compromised or paused; the extension's `beforeRemoveLiquidity` now always reverts.
4. Every LP calls `removeLiquidity`. The call path is:

```
removeLiquidity
  → _beforeRemoveLiquidity
    → _callExtensionsInOrder(BEFORE_REMOVE_LIQUIDITY_ORDER, ...)
      → CallExtension.callExtension(EXTENSION_1, data)
        → extension.call(data) returns (false, revertData)
        → revert(revertData)   // propagated unconditionally
```

5. The pool admin calls `pausePool` — this sets `pauseLevel = 1`, which only gates `swap`. `removeLiquidity` is unaffected.
6. There is no factory function to disable `EXTENSION_1`. It is `immutable`. LP funds are permanently locked. [9](#0-8) [10](#0-9) [3](#0-2) [11](#0-10)

  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }

  function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
      if (result.length > 0) {
        assembly ("memory-safe") {
          revert(add(result, 32), mload(result))
        }
      }
      revert ExtensionCallFailed();
    }
    if (result.length < 32) {
      revert InvalidExtensionResponse();
    }
    bytes4 returnedSelector;
    assembly ("memory-safe") {
      returnedSelector := mload(add(result, 32))
    }
    bytes4 expectedSelector;
    assembly ("memory-safe") {
      expectedSelector := mload(add(data, 32))
    }
    if (returnedSelector != expectedSelector) {
      revert InvalidExtensionResponse();
    }
  }

  function validateExtensionsConfig(
    address[] calldata extensions,
    ExtensionOrders calldata orders,
    bytes[] calldata extensionInitData
  ) internal pure {
    if (extensions.length > MAX_EXTENSIONS) revert InvalidExtensionsConfig();
    if (extensionInitData.length != extensions.length) revert InvalidExtensionsConfig();

    if (extensions.length == 0) {
      if (!_ordersAreZero(orders)) revert InvalidExtensionsConfig();
      return;
    }

    for (uint256 i = 0; i < extensions.length; i++) {
      if (extensions[i] == address(0)) revert InvalidExtensionsConfig();
      for (uint256 j = 0; j < i; j++) {
        if (extensions[i] == extensions[j]) revert InvalidExtensionsConfig();
      }
    }

    if (_ordersAreZero(orders)) revert InvalidExtensionsConfig();

    _validateExtensionOrder(orders.beforeAddLiquidity, extensions.length);
    _validateExtensionOrder(orders.afterAddLiquidity, extensions.length);
    _validateExtensionOrder(orders.beforeRemoveLiquidity, extensions.length);
    _validateExtensionOrder(orders.afterRemoveLiquidity, extensions.length);
    _validateExtensionOrder(orders.beforeSwap, extensions.length);
    _validateExtensionOrder(orders.afterSwap, extensions.length);
  }
