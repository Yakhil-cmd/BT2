### Title
Gateway `skip_stateful_validations` Accepts Unsigned Invoke Transactions by Conflating Any Mempool Entry with a Pending Deploy-Account - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips blockifier `__validate__` (signature verification) for any Invoke transaction with `nonce=1` when `account_nonce=0` in state, provided `account_tx_in_pool_or_recent_block` returns `true`. However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction in the pool for that address — not specifically a `DeployAccount`. An unprivileged attacker who observes a victim's `DeployAccount` in the mempool can submit a forged Invoke with `nonce=1` and an invalid signature for the same address; the gateway accepts it without running `__validate__`, admitting an invalid transaction into the mempool.

### Finding Description

`skip_stateful_validations` is a UX feature: when Alice submits `deploy_account(A)` + `invoke(A, nonce=1)` simultaneously, the gateway cannot yet run `__validate__` on the invoke (the account doesn't exist yet), so it skips blockifier validation if a deploy-account is pending. The check used is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

The proxy check delegates to:

```rust
// crates/apollo_mempool/src/mempool.rs
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if the address has **any** transaction in the pool — including a `DeployAccount` submitted by an attacker for their own address, or any other transaction type. The code comment claims this is "sufficient" because the account "either has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is circular: future-nonce invokes for a non-existent account also pass via the same skip path. [3](#0-2) 

The gateway's `validate_nonce` for invoke transactions only checks that `account_nonce <= tx_nonce <= account_nonce + max_gap`, so `nonce=1` passes when `account_nonce=0`: [4](#0-3) 

`validate_by_mempool` (called before `skip_stateful_validations`) only checks for duplicate tx_hash and fee-escalation rules — it does not verify the signature: [5](#0-4) 

### Impact Explanation

**Attack scenario:**

1. Alice submits `deploy_account(A)` + `invoke(A, nonce=1, valid_sig, fee=X)` to the gateway. Both are accepted; the deploy-account passes full blockifier execution, and the invoke passes via `skip_stateful_validations`.
2. Bob observes Alice's `deploy_account(A)` in the mempool (public). Bob submits `invoke(A, nonce=1, invalid_sig, fee=X+1)`.
3. Gateway evaluation for Bob's invoke:
   - `validate_nonce`: `0 <= 1 <= max_gap` → passes.
   - `validate_by_mempool`: no duplicate hash, fee escalation allowed → passes.
   - `skip_stateful_validations`: `nonce=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block(A)=true` (Alice's deploy-account is in pool) → returns `true`.
   - `run_validate_entry_point`: **skipped**.
4. Bob's invalid invoke is admitted to the mempool. Because Bob paid higher fee, it replaces Alice's valid invoke via fee-escalation.
5. Batcher executes `deploy_account(A)` (succeeds), then Bob's `invoke(A, nonce=1, invalid_sig)` — which fails `__validate__` and is dropped. Alice's invoke has been evicted and her nonce=1 slot is wasted.

This satisfies: **"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."**

### Likelihood Explanation

The attack requires only:
- Observing the public mempool for `DeployAccount` transactions (trivially available).
- Submitting an invoke with `nonce=1` and a higher fee than the victim's invoke.

No privileged access, no special keys, no on-chain state required. Any user can execute this against any account in the deploy-account + invoke UX flow.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a specific check that verifies a **`DeployAccount` transaction** (not just any transaction) is pending for the address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type of the pooled entry at `nonce=0` for the given address. Only if a `DeployAccount` is confirmed pending should `skip_stateful_validations` return `true`.

### Proof of Concept

```
1. Alice: gateway.add_tx(deploy_account(A, class=C, salt=S))
   → accepted; mempool now has deploy_account(A) at nonce=0

2. Alice: gateway.add_tx(invoke(A, nonce=1, calldata=X, sig=valid, fee=100))
   → skip_stateful_validations=true (deploy_account in pool); accepted

3. Bob: gateway.add_tx(invoke(A, nonce=1, calldata=DRAIN_FUNDS, sig=0x0, fee=200))
   → validate_nonce: 0<=1<=max_gap ✓
   → validate_by_mempool: no dup hash, fee 200>100 → replaces Alice's invoke ✓
   → skip_stateful_validations: account_tx_in_pool_or_recent_block(A)=true ✓
   → run_validate_entry_point: SKIPPED
   → Bob's invalid invoke admitted, Alice's valid invoke evicted

4. Batcher executes block:
   deploy_account(A) → success, account deployed
   invoke(A, nonce=1, sig=0x0) → __validate__ fails, tx dropped
   Alice's invoke → no longer in mempool, lost
```
