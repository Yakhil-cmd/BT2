### Title
Missing Access Control on `Staking::update_rewards` Allows Anyone to Monopolize Per-Block Reward Distribution — (File: `src/staking/staking.cairo`)

---

### Summary

The `Staking::update_rewards` function is specified as callable only by the Starkware sequencer, but the implementation enforces no such check. Any address can call it. Because a single global `last_reward_block` variable gates the entire contract to one reward-distribution call per block, an attacker can call `update_rewards` for a chosen staker in every block, causing that staker and their delegators to accumulate block rewards continuously while all other stakers are permanently starved of rewards.

---

### Finding Description

The protocol specification explicitly states:

> **access control**: Only starkware sequencer.

for `update_rewards`. [1](#0-0) 

The implementation, however, only calls `general_prerequisites()` (which checks the paused flag, not the caller identity) and asserts that the current block number is strictly greater than `last_reward_block`: [2](#0-1) 

There is no check that `get_caller_address() == sequencer_address`. The `general_prerequisites()` helper is shared with fully public functions such as `stake`, `increase_stake`, and `claim_rewards`, confirming it performs no privileged-caller check.

The `last_reward_block` storage variable is a **single global counter**, not per-staker: [3](#0-2) 

When `update_rewards` succeeds it writes `current_block_number` into this global: [4](#0-3) 

This means only **one** call to `update_rewards` (for any staker) can succeed per block. Every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

Each successful call computes and immediately transfers block rewards to the named staker's own `unclaimed_rewards_own` and to every pool contract attached to that staker: [5](#0-4) 

Pool rewards are transferred unconditionally on every call via `update_pool_rewards` → `send_rewards_to_delegation_pool`: [6](#0-5) 

The existing flow test explicitly documents and asserts this accumulation behaviour — pool balances double and triple with successive same-epoch calls while the staker's own claimable balance resets to the same value after each `claim_rewards`: [7](#0-6) 

---

### Impact Explanation

**Theft of unclaimed yield / Permanent freezing of unclaimed yield.**

An attacker (e.g., a delegator in staker A's pool) calls `update_rewards(staker_A)` in every block:

1. Staker A and their pool receive block rewards every block.
2. Because `last_reward_block` is now set for that block, the sequencer's intended call for any other staker reverts with `REWARDS_ALREADY_UPDATED`.
3. All other stakers and their delegators receive zero rewards for every block the attacker front-runs.

Over an epoch of N blocks the attacker's chosen staker accumulates N × `block_rewards × staker_share` instead of the single-call-per-epoch amount the sequencer would have distributed. Delegators in the targeted pool receive proportionally inflated yield. Delegators in all other pools receive nothing — a permanent freeze of their unclaimed yield.

The reward supplier's `unclaimed_rewards` counter is incremented on every call: [8](#0-7) 

and funds are immediately claimed from the reward supplier for pool transfers: [9](#0-8) 

This drains the reward supplier faster than the minting curve intends, threatening protocol solvency.

---

### Likelihood Explanation

High. The function is publicly callable with no role or address restriction. The only barrier is the one-call-per-block limit, which the attacker satisfies trivially by submitting one transaction per block. No special privileges, leaked keys, or external dependencies are required — only a funded Starknet account.

---

### Recommendation

Add an explicit sequencer-address check at the top of `update_rewards`, analogous to the pattern already used in `update_current_epoch_block_rewards` in the reward supplier: [10](#0-9) 

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   let sequencer_address = starknet::get_execution_info().block_info.sequencer_address;
+   assert!(
+       get_caller_address() == sequencer_address,
+       "{}",
+       Error::CALLER_IS_NOT_SEQUENCER,
+   );
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
}
```

---

### Proof of Concept

The existing flow test already demonstrates the unbounded accumulation. Calling `update_rewards` for the same staker on successive blocks within the same epoch causes pool balances to grow linearly (2×, 3×, …) with each call, while the `last_reward_block` guard prevents any other staker from receiving rewards in those blocks: [11](#0-10) 

To demonstrate the griefing vector, deploy two stakers (A and B), start consensus rewards, then have an unprivileged address call `update_rewards(staker_A)` in every block. Assert that staker B's `unclaimed_rewards_own` remains zero across the entire epoch while staker A's pool balance grows with every block.

fn test_update_rewards_same_epoch_flow_test() {

    system.advance_epoch();

    let stake_amount = min_stake * 2;
    let staker = system.new_staker(amount: stake_amount);
    let commission = 200;
    system.stake(:staker, amount: stake_amount, pool_enabled: true, :commission);
    system.start_consensus_rewards();
    let pool = system.staking.get_pool(:staker);
    let delegator = system.new_delegator(amount: stake_amount);
    system.delegate(:delegator, :pool, amount: stake_amount);
    let btc_pool = system
        .set_open_for_delegation(:staker, token_address: system.btc_token.contract_address());
    let btc_amount = TEST_MIN_BTC_FOR_REWARDS;
    let delegator = system.new_btc_delegator(amount: btc_amount, token: system.btc_token);
    system.delegate_btc(:delegator, pool: btc_pool, amount: btc_amount, token: system.btc_token);
    let epoch_length = system.staking.get_epoch_info().epoch_len_in_blocks();
    let mut blocks_till_next_epoch = epoch_length.into();
    system.update_rewards(:staker, disable_rewards: false);
    let mut staker_rewards = system.staker_claim_rewards(:staker);
    // Same epoch - same rewards.
    advance_blocks(blocks: 1, block_duration: AVG_BLOCK_DURATION);
    blocks_till_next_epoch -= 1;
    system.update_rewards(:staker, disable_rewards: false);
    assert!(system.staker_claim_rewards(:staker) == staker_rewards);
    advance_blocks(blocks: 100, block_duration: AVG_BLOCK_DURATION);
    blocks_till_next_epoch -= 100;

    assert!(system.staker_claim_rewards(:staker) == staker_rewards);
    // Next epoch - different rewards due to block duration change.
    advance_blocks(blocks: blocks_till_next_epoch, block_duration: AVG_BLOCK_DURATION - 1);
    blocks_till_next_epoch = epoch_length.into();
    system.update_rewards(:staker, disable_rewards: false);
    let mut prev_staker_rewards = staker_rewards;
    staker_rewards = system.staker_claim_rewards(:staker);
    assert_ne!(staker_rewards, prev_staker_rewards);
    // Same epoch - same rewards.
    advance_blocks(blocks: 1, block_duration: AVG_BLOCK_DURATION);
    blocks_till_next_epoch -= 1;
    system.update_rewards(:staker, disable_rewards: false);
    assert!(system.staker_claim_rewards(:staker) == staker_rewards);
    advance_blocks(blocks: 100, block_duration: AVG_BLOCK_DURATION);

    // Next epoch - different rewards due to balance change.
    advance_blocks(blocks: blocks_till_next_epoch, block_duration: AVG_BLOCK_DURATION - 1);
    blocks_till_next_epoch = epoch_length.into();
    system.update_rewards(:staker, disable_rewards: false);
    prev_staker_rewards = staker_rewards;
    staker_rewards = system.staker_claim_rewards(:staker);
