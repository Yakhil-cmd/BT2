### Title
`setPoolAdminFeeDestination` Does Not Collect Accrued Fees Before Changing Destination, Sending Previously Earned Admin Fees to Wrong Recipient - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. All spread-surplus and notional fees that accumulated while the old destination was active are subsequently swept to the new destination on the next `collectPoolFees` call, permanently depriving the old destination of fees it was owed.

### Finding Description

The factory maintains two fee-change functions for pool admins. `setPoolAdminFees` correctly calls `collectFees` at the old rates before updating the fee config, ensuring fees accrued under the old rates go to the correct destination: [1](#0-0) 

However, `setPoolAdminFeeDestination` simply overwrites `poolAdminFeeDestination[pool]` with no prior fee flush: [2](#0-1) 

When `collectPoolFees` (or any subsequent fee-triggering call) is later invoked, `collectFees` on the pool computes the spread surplus as `balance - binTotals - notionalAccumulator` and the notional accumulator, then transfers the admin share to whatever `poolAdminFeeDestination[pool]` currently holds: [3](#0-2) 

All fees that accrued while the old destination was registered are therefore sent to the new destination.

### Impact Explanation

The old `adminFeeDestination` â€” which may be a separate treasury contract, staking contract, or DAO-controlled address with its own stakeholders â€” permanently loses all admin fees (both spread surplus and notional accumulator) that were earned during its tenure. The new destination receives a windfall of fees it did not earn. This is a direct loss of protocol/admin fee assets with no recovery path, since the pool holds no per-destination accounting.

### Likelihood Explanation

Pool admins legitimately rotate fee destinations (e.g., treasury migrations, multisig upgrades, DAO votes). The documentation for `setPoolAdminFeeDestination` says only "Coordinate with treasury operations" without enforcing a prior collection, making it easy to overlook. The inconsistency with `setPoolAdminFees` (which does enforce a prior collection) increases the chance of an admin calling `setPoolAdminFeeDestination` without manually calling `collectPoolFees` first. [4](#0-3) 

### Recommendation

Mirror the pattern used in `setPoolAdminFees`: call `collectFees` on the pool using the current `poolFeeConfig` and the **current** `poolAdminFeeDestination[pool]` before overwriting the destination. This ensures all fees accrued under the old destination are flushed to it before the switch takes effect.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the OLD destination before switching
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is created with `adminFeeDestination = oldTreasury`. Swaps occur; spread surplus and notional fees accumulate in the pool.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is immediately set to `newTreasury`.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` (now `newTreasury`) and passes it to `collectFees`: [5](#0-4) 

4. Inside `collectFees`, the entire spread surplus and notional accumulator â€” all earned while `oldTreasury` was the registered destination â€” are transferred to `newTreasury`: [6](#0-5) 

5. `oldTreasury` receives nothing. The loss is permanent and proportional to the volume of swaps that occurred before the destination change.
