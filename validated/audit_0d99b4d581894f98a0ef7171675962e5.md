### Title
Attacker-Controlled `puzzleHash` in Blockchain Offer Notifications Redirects Counter-Offer to Attacker Address - (File: packages/gui/src/hooks/useBlockchainNotifications.tsx)

### Summary
The Chia GUI blockchain notification system extracts the `ph` (puzzleHash) field directly from the on-chain notification message payload with no verification that it belongs to the actual notification sender. Any unprivileged actor can craft a blockchain notification with an arbitrary `puzzleHash` pointing to their own address. When the victim clicks "Counter" on such a notification, the GUI silently routes the counter-offer notification to the attacker's address instead of the legitimate offer creator's address, allowing the attacker to intercept and accept the counter-offer.

### Finding Description

**Root cause — unvalidated attacker-controlled field in notification payload:**

In `useBlockchainNotifications.tsx`, the `prepareNotifications` function decodes the on-chain message and blindly trusts the `ph` field:

```
const { u: url, ph: puzzleHash } = data;   // line 99
if (puzzleHash) {
  return {
    type: NotificationType.OFFER,
    ...
    offerURL: url,
    puzzleHash,          // attacker-supplied value stored verbatim
  };
}
``` [1](#0-0) 

There is no check that `puzzleHash` matches the coin sender, the notification's on-chain origin address, or any other verifiable identity. The blockchain notification protocol (`utils.ts`) documents `ph` as "puzzlehash of the notification sender, for sending a response (counter offer)" — but nothing enforces this invariant. [2](#0-1) 

**Exploit path — counter-offer destination hijack:**

1. Attacker pays the minimum notification fee and sends a blockchain notification to the victim's address. The message payload is crafted as:
   ```json
   {"v":1,"t":1,"d":{"u":"<legitimate-offer-url>","ph":"<attacker-puzzle-hash>"}}
   ```
   This is a standard `sendNotification` call — no privilege required.

2. The victim's GUI receives the notification. `useBlockchainNotifications` stores `puzzleHash = <attacker-puzzle-hash>` in the notification object. [1](#0-0) 

3. In `OfferIncomingTable`, the "Counter" button is enabled because `puzzleHash` is truthy. When the victim clicks it, `handleCounterOffer` converts the attacker-supplied `puzzleHash` to a bech32m address and passes it as the counter-offer destination:
   ```ts
   const address = currencyCode && puzzleHash
     ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';
   navigate('/dashboard/offers/builder', {
     state: { isCounterOffer: true, address, offer },
   });
   ``` [3](#0-2) 

4. The same attacker-controlled `puzzleHash` is also forwarded through `NotificationOffer.handleClick` → `navigate('/dashboard/offers/view', { state: { address: notification.puzzleHash } })`. [4](#0-3) 

5. The victim creates a counter-offer in the builder. When they share it, `NotificationSendDialog` calls `sendNotification` with `target = fromBech32m(address)` — the attacker's puzzle hash — as the on-chain destination. [5](#0-4) 

6. The attacker receives the counter-offer notification, obtains the counter-offer file URL, and can accept the counter-offer. The legitimate offer creator receives nothing.

### Impact Explanation

**High** — Spoofed blockchain notification state causes the victim to send their counter-offer to the wrong destination. The attacker intercepts the counter-offer and can accept it, completing a trade the victim intended for a different party. This directly matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of … notification … state that causes a user to … send … the wrong … destination."*

The victim has no indication that the counter-offer address is attacker-controlled; the UI presents the "Counter" button as a normal response to a legitimate offer notification, and the address field in `NotificationSendDialog` is rendered as `disabled` (read-only), so the victim cannot detect or correct the substitution. [6](#0-5) 

### Likelihood Explanation

**Medium-High.** The attack requires only the victim's XCH receive address (publicly derivable from any prior transaction) and a small notification fee (~0.0001 XCH). No keys, no privileged access, and no social engineering beyond the notification itself are needed. The victim must click "Counter" on the spoofed notification, which is the natural action for any user who wants to negotiate on an offer they received.

### Recommendation

1. **Verify sender identity:** When parsing a blockchain notification, compare the `ph` field against the puzzle hash of the coin that funded the notification (available from the on-chain coin record). Reject or warn if they differ.
2. **Display the counter-offer address to the user:** Show the resolved bech32m address in the offer view and counter-offer flow so the user can verify it before proceeding, rather than silently pre-filling a disabled field.
3. **Treat `ph` as untrusted input:** Add a validation step in `prepareNotifications` that either cryptographically verifies the `ph` field or marks the notification as "unverified sender" and disables the Counter button accordingly.

### Proof of Concept

```bash
# 1. Attacker obtains victim's XCH address (public info).
# 2. Attacker crafts notification payload with their own puzzle hash:
PAYLOAD='{"v":1,"t":1,"d":{"u":"https://dexie.space/offers/legitimate-offer-id","ph":"<attacker_puzzle_hash_hex>"}}'
HEX_MSG=$(echo -n "$PAYLOAD" | xxd -p | tr -d '\n')

# 3. Attacker calls send_notification via chia RPC (standard wallet API):
chia rpc wallet send_notification \
  '{"target":"<victim_puzzle_hash>","amount":10000000,"message":"'"$HEX_MSG"'","fee":0}'

# 4. Victim's GUI receives the notification, displays "You have a new offer".
#    puzzleHash stored = attacker's puzzle hash.
# 5. Victim clicks "Counter" → offer builder opens with address = attacker's address.
# 6. Victim creates and sends counter-offer notification → attacker receives it.
# 7. Attacker accepts the counter-offer.
```

The only cost to the attacker is the notification fee (~0.0001 XCH). The victim has no way to distinguish this from a legitimate offer notification.
