### Title
Missing `max_l2_gas_amount` Admission Check for Declare Transactions Allows Unbounded L2 Gas Bound at Gateway - (File: crates/apollo_gateway/src/stateless_transaction_validator.rs)

### Summary

The `StatelessTransactionValidator::validate_resource_bounds` function explicitly skips the `max_l2_gas_amount` upper-bound check for `Declare` transactions. For every other transaction type (Invoke, DeployAccount), a transaction whose `l2_gas.max_amount` exceeds `config.max_l2_gas_amount` (default `1_210_000_000`) is rejected at the gateway before any stateful work is done. For `Declare` transactions the branch is an empty no-op, so any `l2_gas.max_amount` up to `u64::MAX` passes stateless admission. Because `l2_gas.max_amount` is later used verbatim as the initial Sierra gas counter for execution (`initial_sierra_gas()`), a well-funded sender can force the sequencer to execute a Declare transaction with a `u64::MAX` gas budget, bypassing the per-transaction resource cap that protects the sequencer from oversized execution.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs` the `validate_resource_bounds` method contains:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The `if let RpcTransaction::Declare(_) = tx { }` arm is an intentional empty branch that causes the `else if` to be unreachable for Declare transactions. The companion test `valid_l2_gas_amount_on_declare` (lines 173–201 of the test file) explicitly asserts that a Declare transaction with `l2_gas.max_amount = 200` passes when `max_l2_gas_amount = 100`, confirming the bypass is load-bearing.

The stateful validator (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, `validate_resource_bounds`) only checks `l2_gas.max_price_per_unit` against the previous block price; it never checks `max_amount`.

The blockifier's `perform_pre_validation_stage` (`crates/blockifier/src/transaction/account_transaction.rs`, line 355) calls `check_fee_bounds` and `verify_can_pay_committed_bounds`. `check_fee_bounds` only rejects a transaction when `minimal_gas_amount > resource_bounds.max_amount`; with `max_amount = u64::MAX` this check always passes. `verify_can_pay_committed_bounds` computes `max_possible_fee` using saturating arithmetic (`crates/starknet_api/src/transaction/fields.rs`, lines 393–413): `u64::MAX * min_gas_price` saturates to `Fee(u128::MAX)`. The balance check (`crates/blockifier/src/fee/fee_utils.rs`, line 158) is:

```rust
let has_sufficient_balance = balance_high > Felt::ZERO || balance_low >= Felt::from(fee.0);
```

Any account whose ERC-20 balance has `balance_high > Felt::ZERO` (i.e., balance ≥ 2¹²⁸ FRI) passes this check regardless of the saturated fee, so the transaction is admitted to the mempool.

Once admitted, `initial_sierra_gas()` (`crates/blockifier/src/context.rs`, lines 63–71) returns `l2_gas.max_amount` verbatim as the starting gas counter for the `__validate_declare__` and constructor entry points. With `u64::MAX` initial gas the execution engine never hits an out-of-gas condition, removing the per-transaction gas cap that the `max_l2_gas_amount` limit is designed to enforce.

### Impact Explanation

An attacker controlling an account with `balance_high > 0` can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX`. The transaction:
1. Passes stateless validation (the check is skipped).
2. Passes stateful resource-bounds validation (only price is checked).
3. Passes `check_fee_bounds` (u64::MAX ≥ any minimal gas amount).
4. Passes `verify_can_pay_committed_bounds` (balance_high > 0 short-circuits the fee comparison).
5. Executes with `u64::MAX` initial Sierra gas, bypassing the `max_l2_gas_amount` cap enforced for all other transaction types.

This matches the **High** impact scope: "Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."

### Likelihood Explanation

The precondition (account with `balance_high > 0`) requires a balance ≥ 2¹²⁸ FRI, which is extremely large in practice. However, the stateless bypass itself is unconditional and requires no special privilege — any Declare transaction with an oversized `l2_gas.max_amount` passes the first admission gate. The stateful gate is the only backstop, and it is defeatable by a sufficiently funded account. The TODO comment and the dedicated positive test (`valid_l2_gas_amount_on_declare`) confirm the gap is known but unresolved.

### Recommendation

Apply the same `max_l2_gas_amount` upper-bound check to `Declare` transactions in `validate_resource_bounds`. Remove the empty `if let RpcTransaction::Declare(_) = tx { }` branch and let the existing `else if` apply uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas ceiling, introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.

### Proof of Concept

1. Create an account with `balance_high > 0` (balance ≥ 2¹²⁸ FRI).
2. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas = ResourceBounds { max_amount: GasAmount(u64::MAX), max_price_per_unit: GasPrice(min_gas_price) }`.
3. Submit via `add_tx` to the gateway.
4. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare transactions (confirmed by the existing test `valid_l2_gas_amount_on_declare` at lines 173–201 of `stateless_transaction_validator_test.rs`).
5. The stateful validator's `validate_resource_bounds` only checks `max_price_per_unit`; it passes.
6. `check_fee_bounds` passes because `u64::MAX ≥ minimal_gas_amount`.
7. `verify_can_pay_committed_bounds` passes because `balance_high > 0`.
