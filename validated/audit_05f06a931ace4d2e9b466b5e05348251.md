### Title
Unprivileged Attacker Can Overwrite Any Account's E2EE Encryption Key — (`x/e2ee/keeper/keeper.go`)

---

### Summary

The `RegisterEncryptionKey` message handler in the `x/e2ee` module does not verify that the transaction signer matches the `Address` field in `MsgRegisterEncryptionKey`. Any unprivileged user can overwrite any other account's registered encryption key with their own public key, corrupting the e2ee key state for that account.

---

### Finding Description

`MsgRegisterEncryptionKey` carries two fields: `Address` (the account whose key is being registered) and `Key` (the X25519 public key). [1](#0-0) 

The keeper's `RegisterEncryptionKey` handler directly uses `req.Address` as the storage key without any check that the transaction signer equals `req.Address`: [2](#0-1) 

The internal `registerEncryptionKey` function unconditionally writes the supplied key to the KV store under the given address: [3](#0-2) 

`ValidateBasic()` only checks bech32 validity and X25519 key format — it performs no signer-address equality check: [4](#0-3) 

The CLI client correctly sets `Address` to the sender's own address, but this is a client-side convention only — the on-chain handler enforces nothing: [5](#0-4) 

---

### Impact Explanation

An attacker who controls any funded account can submit a `MsgRegisterEncryptionKey` with `Address` set to any victim's address and `Key` set to the attacker's own X25519 public key. After this transaction is accepted, all senders who query the victim's registered key and encrypt messages to it will unknowingly encrypt to the attacker's key. The attacker can decrypt all such messages while the victim cannot. This is a direct corruption of e2ee key/message state with security impact, matching the **High** impact category: *Corruption of e2ee key/message state with direct security impact*.

---

### Likelihood Explanation

The entry path is fully unprivileged — any account with enough gas fees can submit this transaction. No special role, leaked key, or admin access is required. The `RegisterEncryptionKey` RPC is publicly reachable via the standard Cosmos SDK transaction pipeline. [6](#0-5) 

---

### Recommendation

In `RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address` before writing to the store:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signerAddr := sdk.UnwrapSDKContext(ctx).EventManager()... // use sdk signer extraction
    // Verify req.Address == signerAddr
    if req.Address != signerAddr.String() {
        return nil, sdkerrors.ErrUnauthorized.Wrap("signer must match address")
    }
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
```

Alternatively, remove the `Address` field from `MsgRegisterEncryptionKey` entirely and derive the address solely from the transaction signer inside the keeper, which eliminates the mismatch surface entirely.

---

### Proof of Concept

1. Victim (`victim_addr`) registers their legitimate X25519 key via `MsgRegisterEncryptionKey{Address: victim_addr, Key: victim_pubkey}`.
2. Attacker generates their own X25519 keypair: `attacker_identity`.
3. Attacker submits `MsgRegisterEncryptionKey{Address: victim_addr, Key: attacker_pubkey}` signed by their own account. The keeper writes `attacker_pubkey` under `victim_addr` in the KV store with no error.
4. Any third party queries `victim_addr`'s key (via `Key` or `Keys` RPC), receives `attacker_pubkey`, and encrypts a confidential message to it.
5. The attacker decrypts the message using `attacker_identity`; the victim cannot decrypt it. [7](#0-6)
