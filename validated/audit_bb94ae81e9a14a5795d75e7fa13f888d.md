### Title
Zero `native_per_pubdata` Due to Missing Floor on `gas_per_pubdata` Allows L1→L2 Transactions to Generate Pubdata Without Cost — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `prepare_and_check_resources`, the per-pubdata-byte native cost for L1→L2 transactions is computed as `native_per_pubdata = gas_per_pubdata * native_per_gas`. The `gas_per_pubdata_limit` field is user-supplied in the L1→L2 transaction. ZKsync OS enforces no minimum floor on this value. When it is zero, `native_per_pubdata` collapses to zero, and the transaction pays nothing for any pubdata it generates — the operator receives zero compensation for pubdata despite real pubdata being written.

---

### Finding Description

In `prepare_and_check_resources` the computation is:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| { ... u64::MAX });
``` [1](#0-0) 

When `gas_per_pubdata = 0`, `native_per_pubdata = 0`. This zero propagates through every pubdata-accounting site:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   // = 0 * intrinsic_pubdata = 0  → no deduction from native_limit
   ``` [2](#0-1) 

2. **Execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)
   // = pubdata_bytes * 0 = 0  → zero native charged
   ``` [3](#0-2) 

3. **Post-execution pubdata check** in `check_enough_resources_for_pubdata` always returns `enough = true` because the required resources are zero. [4](#0-3) 

The `gas_per_pubdata_limit` field is read directly from the user-submitted L1→L2 transaction without any floor check:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
``` [5](#0-4) 

The code acknowledges the dependency on L1 validation but does not enforce it:

> "L1 transactions cannot be invalidated. Therefore, the following function makes sure L1 transactions are processable even when some checks that should be performed by the L1 don't hold." [6](#0-5) 

A parallel truncation-to-zero exists for L2 transactions via integer division:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [7](#0-6) 

When `pubdata_price < native_price`, integer division truncates `native_per_pubdata` to zero for all L2 transactions in that block. No minimum floor is enforced here either.

---

### Impact Explanation

With `native_per_pubdata = 0`:

- The operator (coinbase) receives **zero** native-token compensation for pubdata generated by the transaction, despite real state diffs being written and published.
- The intrinsic pubdata budget is not deducted from the transaction's native limit, giving the transaction more native resources than it paid for.
- The post-execution pubdata sufficiency check always passes, so the transaction cannot be reverted for pubdata exhaustion regardless of how much pubdata it generates (up to the block pubdata limit).

This is a direct resource accounting bug: a party entitled to compensation (the operator, for pubdata costs) receives zero despite real pubdata being produced — the exact class of the reference vulnerability.

---

### Likelihood Explanation

**L1→L2 path**: Any L1 user can set `gas_per_pubdata_limit = 0` in their priority-queue transaction. ZKsync OS applies no floor. The only guard is the L1 mailbox contract. If the L1 contract does not enforce `gas_per_pubdata > 0` (or is updated without a corresponding ZKsync OS update), the path is directly exploitable.

**L2 path**: The operator controls `pubdata_price` and `native_price`. If `pubdata_price < native_price` (including the default test value of `pubdata_price = 0`), every L2 transaction in the block pays zero for pubdata. [8](#0-7) 

---

### Recommendation

Enforce a minimum floor for `gas_per_pubdata` in `prepare_and_check_resources`, analogous to the existing `minimal_gas_used` floor:

```rust
// Reject or saturate if gas_per_pubdata is below a protocol minimum
if gas_per_pubdata == 0 {
    system_log!(system, "L1 tx gas_per_pubdata is zero, using minimum instead");
    // or return an error / use a protocol-defined minimum
}
```

Similarly, enforce a minimum for `native_per_pubdata` in the L2 validation path to prevent integer-division truncation from silently zeroing out pubdata costs.

---

### Proof of Concept

1. Submit an L1→L2 priority transaction with `gas_per_pubdata_limit = 0`.
2. ZKsync OS reads `gas_per_pubdata = 0` and computes `native_per_pubdata = 0 * native_per_gas = 0`.
3. `create_resources_for_tx` charges zero for intrinsic pubdata; the transaction receives its full native limit.
4. The transaction executes and writes storage slots (generating pubdata).
5. `check_enough_resources_for_pubdata` returns `enough = true` (zero native required).
6. The transaction succeeds; the operator receives zero compensation for the pubdata written.
7. Repeat with maximum-pubdata-generating calldata to exhaust the block pubdata limit at zero cost. [1](#0-0) [9](#0-8)

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
```
