### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Undeployed Accounts - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point (the account's signature-verification step) for any Invoke transaction with nonce 1 whose sender address has *any* transaction present in the mempool or a recent block. An unprivileged attacker who observes a legitimate `deploy_account` transaction in the mempool can immediately submit a crafted Invoke with nonce 1 and an arbitrary/invalid signature for the same address. The gateway admits it without signature verification. The transaction enters the mempool, consumes block-building resources, and is only rejected—without fee—when the blockifier later calls `__validate__` during actual execution.

### Finding Description

**Admission path (gateway):**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

When this returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

With `validate: false`, the blockifier's `StatefulValidator::perform_validations` path for Invoke transactions skips the `__validate__` call entirely:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

No other check in the gateway path verifies the transaction signature. `validate_state_preconditions` checks only resource bounds and nonce; `validate_by_mempool` checks only mempool-internal nonce/gas-price state. [4](#0-3) 

**The overly broad `account_tx_in_pool_or_recent_block` predicate:**

The check returns `true` if the address has *any* transaction in the pool or any committed transaction in a recent block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

It does **not** verify that the matching pool entry is specifically a `deploy_account` transaction. The code comment claims this is "sufficient" because the only way an undeployed account (nonce 0 in state) can have a pool entry is via a `deploy_account`. That reasoning is correct for the *legitimate* UX path, but it means the predicate fires as soon as a legitimate user's `deploy_account` lands in the pool—giving any observer a window to inject an unsigned Invoke with nonce 1 for that address.

**Execution path (blockifier, block building):**

When the batcher later retrieves the transaction from the mempool and executes it, `new_for_sequencing` always sets `validate: true`:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        only_query: false,
        charge_fee: enforce_fee(&tx, false),
        validate: true,
        strict_nonce_check: true,
    };
    AccountTransaction { tx, execution_flags }
}
``` [6](#0-5) 

So `__validate__` is called during block execution, the invalid signature causes it to fail, and the transaction is rejected without charging a fee. The attacker bears zero cost.

**Attack steps:**

1. Alice broadcasts a `deploy_account` for address X. It enters the mempool; `account_tx_in_pool_or_recent_block(X)` becomes `true`.
2. Attacker submits `Invoke { sender: X, nonce: 1, signature: [0x0, 0x0], calldata: <arbitrary> }`.
3. Gateway: `account_nonce == 0`, `tx.nonce() == 1`, `account_tx_in_pool_or_recent_block(X) == true` → `skip_validate = true` → `__validate__` not called → transaction admitted.
4. Block execution: `__validate__` called with `validate: true` → signature check fails → transaction rejected, no fee charged.
5. Attacker repeats for every `deploy_account` visible in the mempool, at zero cost.

### Impact Explanation

The gateway admits transactions with invalid signatures into the mempool. These transactions consume mempool slots, block-builder execution threads, and state-reader resources before being discarded. Because no fee is charged on validation failure, the attacker bears no economic cost. This is a free, repeatable DoS against the admission and block-building pipeline, matching the **High** impact category: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

### Likelihood Explanation

The trigger is entirely unprivileged: any observer of the public mempool can execute it. `deploy_account` transactions are publicly visible. The attacker needs only to craft an Invoke with nonce 1 and submit it before the `deploy_account` is committed. No special keys, contracts, or permissions are required.

### Recommendation

1. **Verify the pool entry type.** Before returning `true` from `skip_stateful_validations`, confirm that the matching mempool entry for the address is specifically a `deploy_account` transaction, not an arbitrary transaction.

2. **Perform a lightweight signature pre-check at the gateway

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
