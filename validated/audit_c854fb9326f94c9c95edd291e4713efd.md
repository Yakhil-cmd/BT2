## Analysis

I need to trace the exact code path in `GetExecutionOutcome` and determine whether `target_shard_index` is correctly maintained across a resharding boundary.

**Exact code path in `view_client_actor.rs` lines 1145–1183:**

```
1. epoch_id  ← block containing the outcome's epoch_id
2. shard_layout ← get_shard_layout(epoch_id)          [OLD epoch layout]
3. target_shard_id ← account_id_to_shard_id(account_id, epoch_id)
4. target_shard_index ← shard_layout.get_shard_index(target_shard_id)  [OLD index, NEVER updated]
5. (h, target_shard_id) ← get_next_block_hash_with_new_chunk(outcome_block, target_shard_id)
   → if resharding occurred, returns a CHILD shard_id in the NEW epoch
6. outcome_roots ← block h's chunks' prev_outcome_root()  [NEW epoch, more shards]
7. outcome_root_proof ← merklize(&outcome_roots).1[target_shard_index]  [OLD index into NEW roots]
```

**The concrete mismatch:**

`get_next_block_hash_with_new_chunk` is resharding-aware — when it crosses an epoch boundary with a layout change, it replaces `shard_ids` with the children shards and returns the child `shard_id` in the new layout. [1](#0-0) 

However, `target_shard_index` is computed **before** this call, from the old epoch's `shard_layout`, and is **never recomputed** after the function returns a child shard. [2](#0-1) 

The `outcome_roots` vector is then built from block `h` (in the new epoch, with more shards), and `merklize(&outcome_roots).1[target_shard_index]` uses the stale old-epoch index to pick the Merkle path. [3](#0-2) 

**Why the indices diverge:**

In `ShardLayoutV2`/`V3`, shard IDs are arbitrary (not sequential). When shard `B` (index 1 in the old layout) splits into `B_left` and `B_right`, the new layout might be `[A, B_left, B_right, C]` with indices `[0, 1, 2, 3]`. An account that maps to `B_right` (index 2 in the new layout) would have `target_shard_index = 1` (from the old layout), producing a Merkle path for `B_left`'s outcome root instead of `B_right`'s. [4](#0-3) 

`ShardLayoutV0` and `V1` do not support resharding (`resharding_supported()` returns false for them), so this only affects V2/V3 layouts where shard IDs are non-sequential. [5](#0-4) 

**Attacker-controlled input:**

Any unprivileged user can trigger this by submitting a transaction before a resharding event and then calling `GetExecutionOutcome` after resharding. The `sender_id`/`receiver_id` in `TransactionOrReceiptId` is fully user-controlled. [6](#0-5) 

**Exact corrupted value:**

`outcome_root_proof` = `merklize(&outcome_roots).1[old_shard_index]` where `old_shard_index` is the parent shard's index in the pre-resharding layout, but `outcome_roots` is indexed by the post-resharding layout. The returned proof is for the wrong shard's outcome root. `verify_path(block_outcome_root, outcome_root_proof, shard_outcome_root)` returns false for the correct shard outcome root.

### Title
Stale `target_shard_index` after resharding produces wrong `outcome_root_proof` in `GetExecutionOutcome` — (`chain/client/src/view_client_actor.rs`)

### Summary
`GetExecutionOutcome` computes `target_shard_index` from the pre-resharding epoch's shard layout, then calls `get_next_block_hash_with_new_chunk` which may return a child shard in the post-resharding epoch. The `outcome_roots` vector is built from the post-resharding block (with more shards and different index ordering), but the stale `target_shard_index` is used to select the Merkle path, producing an `outcome_root_proof` for the wrong shard.

### Finding Description
In `ViewClientActor::handle(GetExecutionOutcome)`:

1. `epoch_id` is taken from the block that contains the outcome.
2. `target_shard_id` and `target_shard_index` are computed from that epoch's shard layout.
3. `get_next_block_hash_with_new_chunk` is called; if resharding occurred between the outcome block and the next new-chunk block, it returns a **child** `shard_id` in the new epoch.
4. `outcome_roots` is collected from the new-epoch block (which has a different number of shards and different index ordering).
5. `merklize(&outcome_roots).1[target_shard_index]` uses the **old** index, selecting the wrong entry.

The fix is to recompute `target_shard_index` from the new epoch's shard layout and the returned child `shard_id` after `get_next_block_hash_with_new_chunk` returns. [7](#0-6) [8](#0-7) 

### Impact Explanation
Light clients and RPC consumers that call `GetExecutionOutcome` for transactions submitted before a resharding event receive an `outcome_root_proof` that fails verification. The invariant `verify_path(block_outcome_root, outcome_root_proof, shard_outcome_root) == true` is broken. A light client cannot confirm transaction finality for affected transactions.

### Likelihood Explanation
Requires a resharding event between transaction submission and the `GetExecutionOutcome` query. Resharding is a planned, periodic protocol event. Any transaction submitted in the epoch before a resharding whose sender/receiver account maps to the "right" child shard (i.e., the child whose index in the new layout differs from the parent's index in the old layout) will produce a broken proof. No special privileges are required.

### Recommendation
After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the new block's epoch shard layout:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout =
        self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
}
```

### Proof of Concept
1. Configure a test-loop with two shards (V2/V3 layout) and a scheduled resharding that splits shard `B` into `B_left` and `B_right`.
2. Submit a transaction from an account that maps to shard `B` in the pre-resharding layout and to `B_right` (the child with a different index) in the post-resharding layout.
3. Wait for the transaction to execute (pre-resharding).
4. Advance past the resharding epoch boundary.
5. Call `GetExecutionOutcome` with the transaction hash and `sender_id`.
6. Assert `verify_path(block_header.outcome_root, outcome_root_proof, shard_outcome_root)` — it will return `false`, demonstrating the corrupted proof. [9](#0-8) [10](#0-9)

    pub fn get_next_block_hash_with_new_chunk(
        &self,
        block_hash: &CryptoHash,
        shard_id: ShardId,
    ) -> Result<Option<(CryptoHash, ShardId)>, Error> {
        let mut block_hash = *block_hash;
        let mut epoch_id = *self.get_block_header(&block_hash)?.epoch_id();
        let mut shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)?;
        // this corrects all the shard where the original shard will split to if sharding changes
        let mut shard_ids = vec![shard_id];

        while let Ok(next_block_hash) = self.chain_store.get_next_block_hash(&block_hash) {
            let next_epoch_id = *self.get_block_header(&next_block_hash)?.epoch_id();
            if next_epoch_id != epoch_id {
                let next_shard_layout = self.epoch_manager.get_shard_layout(&next_epoch_id)?;
                if next_shard_layout != shard_layout {
                    shard_ids = shard_ids
                        .into_iter()
                        .flat_map(|id| {
                            next_shard_layout.get_children_shards_ids(id).unwrap_or_else(|| {
                                panic!("invalid shard layout {:?} because it does not contain children shards for parent shard {}", next_shard_layout, id)
                            })
                        })
                        .collect();

                    shard_layout = next_shard_layout;
                }
                epoch_id = next_epoch_id;
            }
            block_hash = next_block_hash;

            let block = self.get_block(&block_hash)?;
            let chunks = block.chunks();
            for &shard_id in &shard_ids {
                let shard_index = shard_layout.get_shard_index(shard_id)?;
                let chunk_header =
                    &chunks.get(shard_index).ok_or(Error::InvalidShardId(shard_id))?;
                if chunk_header.height_included() == block.header().height() {
                    return Ok(Some((block_hash, shard_id)));
                }
            }
        }

        Ok(None)
    }
