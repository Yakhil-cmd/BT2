### Title
Cross-Address Replay in `EVM.validateCOAOwnershipProof` Allows Proof of Ownership for Unintended COA - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the `evmAddress` being proven. A signature produced to prove ownership of one Cadence-Owned Account (COA) can be replayed to prove ownership of a different COA belonging to the same Cadence account. This is the direct Flow analog of the reported vulnerability: both stem from an incorrect assumption that a credential (address/signature) produced in one context is safely bound to a specific target address in another context.

### Finding Description
`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function that verifies a user's ownership of a COA by:
1. Verifying that the provided `signatures` over `signedData` are valid for the keys of the Cadence `address`
2. Borrowing the COA resource at the provided `path` and checking that its EVM address matches `evmAddress`

The critical flaw is that **step 1 and step 2 are independent**: the signature is verified over `signedData`, but `signedData` is never required to commit to `evmAddress`. The code comment at line 1003 explicitly acknowledges this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The function signature accepts `evmAddress` as a caller-supplied parameter with no cryptographic binding to `signedData`: [2](#0-1) 

The verification logic checks the signature against `signedData` and separately checks the COA address, but never checks that `signedData` commits to `evmAddress`: [3](#0-2) 

### Impact Explanation
Any Cadence transaction, script, or EVM contract that calls `validateCOAOwnershipProof` (or the EVM precompile `verifyCOAOwnershipProof`) to gate access to assets or privileged operations is vulnerable to cross-address replay. Specifically:

- A Cadence account with two COAs published at different public paths (e.g., `/public/coaA` and `/public/coaB`) is vulnerable.
- A signature produced by the account owner for COA_B (e.g., for a low-value DApp login) can be replayed by an attacker to prove ownership of COA_A (e.g., a high-value DeFi vault).
- The attacker supplies `path = /public/coaA`, `evmAddress = COA_A_address`, and the victim's existing `signedData` + `signatures`. The function returns `isValid: true`.

The EVM precompile path (`coa.sol`'s `isValidSignature`) is safe because it hardcodes `address(this)` as the `evmAddress` argument: [4](#0-3) 

However, `validateCOAOwnershipProof` is `access(all)` and is a documented public authentication primitive intended for off-chain authentication flows and DApp integrations. Any such caller is exposed.

### Likelihood Explanation
- **Entry path is fully reachable**: Any unprivileged Cadence transaction or EVM contract can call `validateCOAOwnershipProof` or `verifyCOAOwnershipProof` with attacker-controlled arguments.
- **Precondition**: The victim must have two COAs published at public paths. While the comment notes wallets "historically create at most one COA per account," the protocol explicitly supports multiple COAs and the `createCadenceOwnedAccount` function is `access(all)`. [5](#0-4) 

- **Signature availability**: The attacker only needs to observe a single valid proof from the victim (e.g., from a public transaction or DApp interaction) to replay it against a different COA.
- The `AllocateCOAAddress` function is deterministic based on resource UUID, so COA addresses are predictable and enumerable by an attacker. [6](#0-5) 

### Recommendation
Require that `signedData` cryptographically commits to `evmAddress`. The simplest fix is to enforce at the protocol level that `signedData` is the ABI/RLP encoding of `evmAddress` (or a hash thereof), rather than leaving this as a caller responsibility. Alternatively, the function should internally hash `signedData || evmAddress` before verifying signatures, ensuring the proof is address-bound.

### Proof of Concept
1. Alice has a Cadence account with two COAs:
   - COA_A at `/public/coaA` (address `0xAAA...`)
   - COA_B at `/public/coaB` (address `0xBBB...`)
2. Alice signs arbitrary data `D` and submits a proof for COA_B to a low-stakes DApp. The proof `(address=Alice, path=/public/coaB, signedData=D, keyIndices=[0], signatures=[sig], evmAddress=0xBBB...)` is observed on-chain.
3. An attacker constructs a new call to `validateCOAOwnershipProof` with `(address=Alice, path=/public/coaA, signedData=D, keyIndices=[0], signatures=[sig], evmAddress=0xAAA...)`.
4. Step 1 (signature check): `sig` is valid over `D` for Alice's key — **passes**.
5. Step 2 (address check): the COA at `/public/coaA` has address `0xAAA...` which matches the supplied `evmAddress` — **passes**.
6. The function returns `ValidationResult(isValid: true)`.
7. Any protocol that uses this result to authorize operations on COA_A (e.g., asset withdrawals, bridge operations) now grants the attacker unauthorized access. [7](#0-6)
