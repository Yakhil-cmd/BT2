### Title
Multiblock Batch Prover Panic via `ArrayVec` Overflow in `apply_to_array_vec` - (`zk_ee/src/common_structs/logs_storage.rs` + `basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

---

### Summary

In the multiblock batch proving path, `ZKBatchDataKeeper.logs_storage` is declared as `ArrayVec<Bytes32, 16384>` — sized to hold exactly one block's worth of L2-to-L1 logs. However, `apply_to_array_vec` is called once per block during batch finalization, unconditionally pushing every block's logs into this single fixed-capacity accumulator. When the cumulative log count across all blocks in a batch exceeds 16,384, `ArrayVec::push` panics, crashing the prover and making the batch permanently unprovable.

---

### Finding Description

`MAX_NUMBER_OF_LOGS` is defined as 16,384 and is used both as the per-block log limit and as the capacity of the batch-level accumulator: [1](#0-0) 

The batch data keeper holds a single `ArrayVec<Bytes32, 16384>` for the entire multiblock batch: [2](#0-1) 

During multiblock batch proving, after each block is finalized, all of that block's logs are pushed into this shared accumulator: [3](#0-2) 

The `apply_to_array_vec` implementation uses `ArrayVec::push` with no bounds check: [4](#0-3) 

`ArrayVec::push` panics when the array is full. The per-block limit enforced by `check_for_block_limits` only prevents any single block from exceeding 16,384 logs: [5](#0-4) 

There is no per-batch log limit. A batch with N blocks can legally contain up to N × 16,384 logs in total, but the accumulator can only hold 16,384. For a 2-block batch where each block emits 8,193 logs (well within the per-block limit), the total of 16,386 overflows the `ArrayVec`, causing a panic in the RISC-V prover.

---

### Impact Explanation

**Valid-execution unprovability / service unavailability.** The forward (sequencing) execution of each block succeeds — every block individually respects the per-block log limit. However, when the prover attempts to finalize the multiblock batch, `apply_to_array_vec` panics, aborting the RISC-V proof run. The batch becomes permanently unprovable. This halts the chain's ability to settle on L1 for any batch that contains the attacker's transactions, constituting a critical integrity failure in the state transition function.

---

### Likelihood Explanation

L2-to-L1 logs are emitted via the L1Messenger system hook, which is accessible to any unprivileged user. The attacker needs to emit slightly more than `16384 / (number_of_blocks_in_batch)` logs per block. For a 2-block batch, this threshold is only 8,193 logs per block — half the per-block limit. The cost is bounded by EVM gas and native resources charged per log emission: [6](#0-5) 

Since the per-block log limit is 16,384 and the native cost per log is finite, an attacker with sufficient funds can reliably trigger this across any multiblock batch configuration.

---

### Recommendation

Replace the fixed-capacity `ArrayVec<Bytes32, 16384>` in `ZKBatchDataKeeper` with a dynamically-sized container (e.g., `Vec<Bytes32, A>`) that can grow to accommodate logs from all blocks in the batch. Alternatively, enforce a per-batch log limit equal to `16384` total (not per-block), and reject any block that would push the batch total over this limit before it is included in the batch. [7](#0-6) 

---

### Proof of Concept

1. Operator configures a multiblock batch with 2 blocks.
2. Attacker submits transactions in block 1 that call the L1Messenger hook 8,193 times, each emitting a minimal L2-to-L1 log. Block 1 is accepted (8,193 < 16,384).
3. Attacker submits transactions in block 2 that emit another 8,193 logs. Block 2 is accepted.
4. Prover runs `ZKHeaderStructurePostTxOpProvingMultiblockBatch::post_op` for block 1: `apply_to_array_vec` pushes 8,193 hashes into `batch_data.logs_storage` (len = 8,193, cap = 16,384). OK.
5. Prover runs `post_op` for block 2: `apply_to_array_vec` attempts to push 8,193 more hashes. At index 16,384, `ArrayVec::push` panics.
6. The RISC-V prover aborts. The batch is permanently unprovable. The chain cannot settle this batch on L1. [4](#0-3) [3](#0-2)
