### Title
Unverified NFT Metadata Spoofing in Offer Confirmation Dialogs Causes Users to Approve Wrong Asset Transfers - (File: `packages/gui/src/hooks/useFetchAndProcessMetadata.ts`)

### Summary
When an NFT is minted with an empty metadata hash (`metadataHash = '0x'`), the Chia GUI fetches and unconditionally trusts metadata from the attacker-controlled URI without any integrity check. This unverified metadata — including the NFT's `name`, `description`, and preview image — is rendered directly in offer confirmation dialogs and the WalletConnect signing confirmation screen. An unprivileged attacker can serve different metadata to different users, causing victims to believe they are accepting an offer for a valuable NFT when they are actually receiving a worthless one.

### Finding Description

The Chia NFT1 standard permits minting NFTs with no metadata hash (represented on-chain as `'0x'`). The GUI's `useFetchAndProcessMetadata` hook contains a conditional guard:

```ts
// packages/gui/src/hooks/useFetchAndProcessMetadata.ts, line 23
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
```

When `metadataHash` is falsy (empty string, `undefined`, or `'0x'`), the `if (hash && ...)` branch is never entered, so the downloaded metadata JSON is accepted without any integrity verification. [1](#0-0) 

The caller in `useMetadataData` passes `metadataHash` directly from the on-chain NFT record:

```ts
// packages/gui/src/components/nfts/provider/hooks/useMetadataData.ts, line 76-83
const { metadataUris = [], metadataHash } = nft;
const [firstUri] = metadataUris;
...
const metadata = await fetchAndProcessMetadata(firstUri, metadataHash);
``` [2](#0-1) 

The `NFTDetails` component itself confirms that `'0x'` is the canonical "no hash" sentinel:

```ts
// packages/gui/src/components/nfts/NFTDetails.tsx, line 262
if (nft.metadataHash && nft.metadataHash !== '0x') { ... }
``` [3](#0-2) 

The metadata `name` and `description` fields fetched from this unverified URI are then rendered in multiple offer-facing surfaces:

1. **`NFTSummary`** (used inside `OfferSummaryNFTRow` when `showNFTPreview=true`): renders `<NFTTitle>` (which reads `metadata.name`) and `metadata.description` directly: [4](#0-3) 

2. **`OfferDetails`** (used in incoming offer notifications): renders `<NFTTitle nftId={info.displayName} />`: [5](#0-4) 

3. **`OfferManager` offer list**: renders `<NFTTitle nftId={displayName} />` in the "Offered/Requested" columns: [6](#0-5) 

4. **WalletConnect `Confirm` dialog**: renders `line.name` (the NFT name from metadata) as the primary label in the signing confirmation screen: [7](#0-6) 

None of these surfaces display any warning that the metadata is unverified when `metadataHash` is absent. The `NFTHashStatus` chip (which would show "Invalid hash") is not rendered in any of the offer confirmation or WalletConnect approval flows.

`NFTTitle` itself is a thin wrapper that reads `metadata.name` via `NFTMetadata`: [8](#0-7) 

### Impact Explanation

A victim reviewing an offer sees the NFT's `name` and `description` as the primary identification signal. If those fields are attacker-controlled (because `metadataHash = '0x'`), the victim can be shown "Rare CryptoPunk #1234 — 1-of-1 genesis edition" while the actual on-chain asset is a worthless NFT. The victim approves the offer (spending real XCH or CATs), and the transaction is irreversible. This satisfies the **High** impact criterion: *"Corruption, spoofing, or unsafe trust of … NFT metadata … that causes a user to approve … the wrong asset."*

The WalletConnect path (`Confirm.tsx` line 171) is additionally impactful because the confirmation dialog is a sandboxed iframe with no secondary verification UI at all — the NFT name is the only human-readable identifier shown.

### Likelihood Explanation

The attack requires no special privilege. Any user can mint an NFT on Chia mainnet with `metadataHash = '0x'` and point `metadataUris` to a server they control. The server can serve different JSON responses based on the requesting IP or user-agent, showing legitimate-looking metadata to the victim and a different payload to anyone auditing the URI. The attacker then creates and publishes an offer. The only social-engineering step is convincing the victim to view and accept the offer, which is the normal offer-acceptance flow.

### Recommendation

1. **Treat absent `metadataHash` as unverified, not as "skip check":** In `useFetchAndProcessMetadata`, when `hash` is falsy, the returned `Metadata` object should be tagged as `unverified: true` rather than silently accepted.

2. **Surface a prominent warning in offer confirmation dialogs:** `NFTSummary`, `OfferSummaryNFTRow`, and the WalletConnect `Confirm` dialog should display a visible "Unverified metadata" badge whenever the NFT's `metadataHash` is `'0x'` or absent, so users know the displayed name/description may not reflect the actual asset.

3. **Align with `useNFTVerifyHash` behavior:** `useNFTVerifyHash` already returns `undefined` (no verification) when `hash` is absent (line 49: `if (!uris || !uris.length || !hash) { return undefined; }`). The offer confirmation path should consume this signal and warn accordingly. [9](#0-8) 

### Proof of Concept

1. Attacker mints NFT on mainnet with:
   - `metadataUris = ["https://attacker.example/meta.json"]`
   - `metadataHash = "0x"` (no hash)

2. `attacker.example/meta.json` returns `{"name":"Rare CryptoPunk #1234","description":"1-of-1 genesis edition"}` to victim IPs, and `{"name":"Test NFT","description":""}` to all other IPs.

3. Attacker creates an offer: "my NFT for 50 XCH" and shares the offer file.

4. Victim opens the offer in Chia GUI. `useMetadataData` calls `fetchAndProcessMetadata(firstUri, "0x")`. Because `hash` is `"0x"` (falsy after `if (hash && ...)`), the checksum check is skipped. The metadata `{"name":"Rare CryptoPunk #1234",...}` is stored and rendered.

5. `NFTSummary` renders "Rare CryptoPunk #1234" as the NFT title in the offer preview card. No hash-warning badge appears.

6. Victim clicks "Accept Offer", spending 50 XCH. The transaction is broadcast and irreversible. The victim receives a worthless NFT whose metadata server can now switch back to `{"name":"Test NFT"}`.
