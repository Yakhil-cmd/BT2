### Title
Missing Caller Restriction on `update_rewards` Allows Any Actor to Suppress Consensus Rewards via `disable_rewards: true` — (File: src/staking/staking.cairo)

---

### Summary

`update_rewards` in `StakingRewardsManagerImpl` has no caller restriction. Any unprivileged actor can invoke it with `disable_rewards: true` for any valid staker every block, advancing the global `last_reward_block` checkpoint without distributing consensus rewards, thereby continuously starving all stakers of block-based yield.

---

### Finding Description

The analog vulnerability class is **missing access-control guard on a function that should have one**. In the ENS case, `acceptRegistrarTransfer` was missing the `live` modifier that verified the contract was the proper registry owner. Here, `update_rewards` is missing any caller-identity check that would restrict who may invoke it and, critically, who may set `disable_rewards: true`.

`update_rewards` is the sole path through which V3 consensus (block-based) rewards are calculated and credited to stakers. Its logic is:

1. Call `general_prerequisites()` — only checks the contract is not paused and caller is non-zero. [1](#0-0) 

2. Assert `current_block_number > last_reward_block`. [2](#0-1) 

3. Validate the staker exists and is active. [3](#0-2) 

4. **Write `current_block_number` into `last_reward_block`** — this happens unconditionally, before the `disable_rewards` branch. [4](#0-3) 

5. If `disable_rewards || self.is_pre_consensus()` → **return early with no rewards distributed**. [5](#0-4) 

6. Otherwise, calculate and distribute block rewards. [6](#0-5) 

Because step 4 commits the new `last_reward_block` before step 5 exits, any caller who passes `disable_rewards: true` consumes the block's reward slot without paying out anything. No subsequent call can distribute rewards for that block because the guard at step 2 will reject it.

There is no check that the caller is the staker, the staker's operational address, or any privileged role. `general_prerequisites` is the only gate: [7](#0-6) 

---

### Impact Explanation

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` at every block:

- Advances `last_reward_block` to the current block on each call.
- Causes the legitimate block-proposer's subsequent call to revert with `REWARDS_ALREADY_UPDATED`.
- Prevents any consensus reward from ever being credited to `staker_info.unclaimed_rewards_own` or forwarded to delegation pools.

This constitutes **permanent freezing of unclaimed yield** for all stakers while the attack is sustained, matching the High-severity impact category.

---

### Likelihood Explanation

- The attacker needs only a non-zero address and a valid (active) staker address — both are publicly observable on-chain.
- The only cost is gas per block. On Starknet, transaction fees are low, making sustained griefing economically feasible for a motivated adversary.
- No privileged access, leaked key, or external dependency is required.

---

### Recommendation

Add a caller restriction to `update_rewards`. The function should only be callable by the staker themselves, their operational address, or a designated privileged role (e.g., `only_app_governor` or a block-proposer role). Additionally, the `disable_rewards` flag should either be removed or restricted to privileged callers, since it allows consuming the block's reward slot without distributing anything.

---

### Proof of Concept

1. Staker `S` is active with non-zero STRK balance. Consensus rewards are live (`is_pre_consensus()` returns `false`).
2. Attacker `A` (any address ≠ zero) monitors the mempool.
3. At each new block `N`, `A` calls `update_rewards(S, disable_rewards: true)`.
4. `general_prerequisites()` passes (contract not paused, `A` ≠ zero). [8](#0-7) 
5. `current_block_number (N) > last_reward_block` passes. `last_reward_block` is written to `N`. [4](#0-3) 
6. `disable_rewards == true` → function returns. No rewards distributed. [5](#0-4) 
7. When the legitimate block proposer calls `update_rewards(S, false)` in the same block `N`, step 2 reverts: `N > N` is false.
8. Repeated every block → staker `S` (and all stakers, since `last_reward_block` is global) accumulates zero consensus rewards indefinitely.
