### Title
Missing Access Control on `update_rewards` Allows Anyone to Advance `last_reward_block` and Permanently Freeze Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` unconditionally writes the global `last_reward_block` state variable before checking whether rewards should actually be distributed. Because the function has no caller restriction in the implementation (despite the spec mandating "Only starkware sequencer"), any unprivileged address can call it with `disable_rewards=true` to advance `last_reward_block` to the current block without distributing any rewards. The legitimate sequencer call for that same block then fails with `REWARDS_ALREADY_UPDATED`, permanently losing all stakers' block rewards for that block.

---

### Finding Description

**Vulnerability class**: State-transition bug — a global sequencing/gating state variable is advanced even when no real work is done, and there is no access control preventing an unprivileged caller from triggering this.

`last_reward_block` is a single global `BlockNumber` stored in the staking contract: [1](#0-0) 

`update_rewards` writes this variable unconditionally at line 1485, **before** the `disable_rewards` early-return guard at line 1487: [2](#0-1) 

The function signature and its only prerequisite check: [3](#0-2) 

`general_prerequisites()` only checks the pause flag — there is no `only_sequencer` or equivalent caller assertion anywhere in the function body. The spec explicitly states access should be "Only starkware sequencer": [4](#0-3) 

Yet the test suite calls `update_rewards` without any `cheat_caller_address_once`, confirming no on-chain caller restriction exists: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

`last_reward_block` is a single global value. Once it is set to block `N`, the guard `current_block_number > self.last_reward_block.read()` prevents any further call in block `N`. Block rewards are calculated per-block; if the sequencer's legitimate `update_rewards(..., disable_rewards=false)` call is front-run or replaced by an attacker's `update_rewards(..., disable_rewards=true)` call in the same block, the rewards for that block are permanently unclaimable — there is no mechanism to retroactively distribute missed-block rewards. [6](#0-5) 

---

### Likelihood Explanation

**High.** The entry path requires only:
1. A valid (active, non-zero-balance) `staker_address` — trivially discoverable from on-chain events.
2. Calling `update_rewards(staker_address, disable_rewards=true)` from any address in the target block.

No funds, no privileged role, and no special setup are required. The attacker pays only gas. The attack can be repeated every block to continuously suppress reward distribution for all stakers.

---

### Recommendation

Add an explicit caller restriction to `update_rewards` so only the designated sequencer address (or a role-controlled address) may invoke it, mirroring the spec's stated access control:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   self.assert_caller_is_sequencer(); // enforce "Only starkware sequencer"
    ...
    self.last_reward_block.write(current_block_number);
    ...
}
```

Alternatively, move the `last_reward_block.write` to after the `disable_rewards` guard so that a no-op call does not consume the block's reward slot.

---

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`). Staker `S` has a non-zero balance.
2. At block `N`, attacker calls `staking.update_rewards(S, disable_rewards=true)`.
   - `current_block_number (N) > last_reward_block` → passes.
   - `last_reward_block` is written to `N`.
   - Function returns early; **no rewards distributed**.
3. The Starkware sequencer attempts `staking.update_rewards(S, disable_rewards=false)` in the same block `N`.
   - `current_block_number (N) > last_reward_block (N)` → **false** → reverts with `REWARDS_ALREADY_UPDATED`.
4. Block `N`'s rewards for all stakers are permanently lost. [2](#0-1)
