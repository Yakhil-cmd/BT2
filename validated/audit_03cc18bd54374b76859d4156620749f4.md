### Title
Pool Admin Bypasses Price Provider Timelock via CEI Violation in `createPool` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool()` calls extension `initialize` functions (external calls) **before** writing factory state variables including `poolAdmin[pool]` and `priceProviderTimelock[pool]`. During this window `poolAdmin[pool] == address(0)`, causing the `onlyPoolAdmin` guard to pass for **any** caller. A malicious extension exploits this to call `proposePoolPriceProvider` while `priceProviderTimelock[pool]` is still `0`, recording `executeAfter = block.timestamp`. After `createPool` finishes and sets the real timelock (e.g. 7 days), the pending proposal retains the zero-delay timestamp, letting the pool admin execute the provider swap immediately and bypass the timelock entirely.

---

### Finding Description

`createPool` executes in this order:

1. **Deploy pool** (lines 180–204)
2. **External calls** — `extension.initialize(pool, data)` for each extension (lines 206–210)
3. **State writes** — `poolAdmin[pool]`, `priceProviderTimelock[pool]`, `poolFeeConfig[pool]`, `poolAdminFeeDestination[pool]`, `poolToIdx[pool]`, `idxToPool[poolIdx]` (lines 212–225) [1](#0-0) 

`createPool` carries **no `nonReentrant` modifier**, so the factory's transient reentrancy lock is not held during step 2. [2](#0-1) 

The `onlyPoolAdmin` guard is:

```solidity
if (msg.sender != poolAdmin[pool]) revert NotPoolAdmin();
``` [3](#0-2) 

While `poolAdmin[pool]` is still `address(0)` (step 2), this resolves to `msg.sender != address(0)` — **always true** for any real contract. Every `onlyPoolAdmin`-gated function is therefore callable by the extension itself during `initialize`.

`proposePoolPriceProvider` is one such function: [4](#0-3) 

It reads `priceProviderTimelock[pool]` — also `0` at this point — and records:

```solidity
uint256 executeAfter = block.timestamp + timelock;   // = block.timestamp + 0
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
``` [5](#0-4) 

After `createPool` completes and writes `priceProviderTimelock[pool] = params.priceProviderTimelock` (e.g. 7 days), the pending proposal still carries `executeAfter = block.timestamp` from the time of the `initialize` call. `executePoolPriceProviderUpdate` then succeeds immediately: [6](#0-5) 

---

### Impact Explanation

The price provider is the sole source of bid/ask prices for every swap. A malicious provider that returns inverted or extreme prices allows the pool admin to drain LP positions through bad-price execution. The timelock exists precisely to give LPs time to exit before a provider change takes effect; bypassing it removes that protection entirely, constituting a direct loss of LP principal.

---

### Likelihood Explanation

- Requires the pool admin to be adversarial and to supply a malicious extension at pool creation — a semi-trusted actor operating outside their permitted boundary.
- The pool must use a mutable price provider (`priceProviderTimelock != type(uint256).max`); immutable-provider pools are unaffected.
- The malicious provider must pass `_validatePriceProvider` (correct `token0`/`token1` return values), which is trivially satisfied by a purpose-built contract.
- No privileged factory-owner cooperation is needed; the pool admin acts alone.

---

### Recommendation

1. **Add `nonReentrant` to `createPool`** so the factory lock is held during extension `initialize` calls, preventing any re-entry into `proposePoolPriceProvider` or other guarded functions.
2. **Move all factory state writes before extension `initialize` calls** (strict CEI order): set `poolAdmin[pool]`, `priceProviderTimelock[pool]`, `poolFeeConfig[pool]`, `poolAdminFeeDestination[pool]`, and registry mappings before invoking any external extension code.

---

### Proof of Concept

```solidity
// MaliciousExtension.sol
contract MaliciousExtension {
    address immutable factory;
    address immutable maliciousProvider; // passes _validatePriceProvider checks

    function initialize(address pool, bytes calldata) external {
        // During this call: poolAdmin[pool] == address(0)  →  onlyPoolAdmin passes
        // priceProviderTimelock[pool] == 0  →  executeAfter = block.timestamp
        IMetricOmmPoolFactory(factory).proposePoolPriceProvider(pool, maliciousProvider);
    }
    // ... other IMetricOmmExtensions stubs ...
}

// Attack sequence
factory.createPool(PoolParameters({
    admin:                  attacker,
    priceProvider:          legitimateProvider,
    priceProviderTimelock:  7 days,          // intended timelock
    extensions:             [address(maliciousExtension)],
    // ... other params ...
}));
// createPool calls maliciousExtension.initialize(pool, "")
//   → proposePoolPriceProvider(pool, maliciousProvider) succeeds
//   → pendingPriceProviderExecuteAfter[pool] = block.timestamp
// createPool then sets priceProviderTimelock[pool] = 7 days  (too late)

// Immediately in the same block:
factory.executePoolPriceProviderUpdate(pool);
// block.timestamp >= pendingPriceProviderExecuteAfter[pool]  ✓
// Pool now uses maliciousProvider; attacker swaps at manipulated prices, draining LPs.
```

    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
