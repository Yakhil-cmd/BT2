### Title
`skip_stateful_validations` admits unsigned invoke transactions for any address whose deploy_account is in the mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to improve UX by allowing a user to submit a `deploy_account` + `invoke` pair simultaneously. It skips the `__validate__` entry-point check for an invoke with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true`. However, that helper returns `true` for **any** transaction from the sender's address in the pool — not exclusively a `deploy_account`. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can submit an invoke for the victim's address with an arbitrary (invalid) signature, bypass the `__validate__` entry point at the gateway, and have the transaction admitted to the mempool.

### Finding Description

The gateway stateful validation path is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce + resource bounds)
       ├─ validate_by_mempool            (duplicate hash / nonce gap)
       └─ skip_stateful_validations      ← decides whether __validate__ runs
  └─ run_validate_entry_point           (calls __validate__ unless skip=true)
```

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 4 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if **any** transaction from that address is in the pool — including a `deploy_account` submitted by the legitimate owner. The code comment acknowledges this looseness:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [3](#0-2) 

The reasoning is circular: the only way a fresh account (nonce=0) can have a transaction in the pool is via a `deploy_account` (which requires a valid `__validate_deploy__` signature). However, once that `deploy_account` is in the pool, **any third party** can submit an invoke for that address with nonce=1 and an arbitrary signature. The gateway will skip `__validate__` for that invoke because condition 4 is satisfied by the victim's `deploy_account`.

The `validate_nonce` check passes because `0 ≤ 1 ≤ max_allowed_nonce_gap`: [4](#0-3) 

The `validate_by_mempool` check passes because the attacker's invoke has a unique transaction hash and a valid nonce gap: [5](#0-4) 

The `run_validate_entry_point` then sets `validate: !skip_validate = false`, so `StatefulValidator::perform_validations` returns early without calling `__validate__`: [6](#0-5) [7](#0-6) 

The attacker's invoke is admitted to the mempool. During block production the batcher creates a fresh `AccountTransaction` with `validate=true`, so `__validate__` **is** called at execution time and the transaction reverts. However, the reverted transaction is still included in the block and the fee is charged to the victim's address.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

- An invoke transaction with an arbitrary (invalid) signature is admitted to the mempool, violating the invariant that every admitted invoke must carry a signature that will pass `__validate__`.
- If fee escalation is enabled (`enable_fee_escalation = true`), the attacker can pay a higher fee to replace the victim's legitimate invoke(nonce=1) with their own invalid one, causing the victim's transaction to be lost from the pool.
- The victim's account is charged fees for the attacker's reverted transaction after the `deploy_account` executes.
- The attack window is the period between the victim's `deploy_account` entering the mempool and being committed to a block.

### Likelihood Explanation

**Medium.** The attacker must monitor the public mempool for `deploy_account` transactions and race to submit the malicious invoke before the block is sealed. No privileged access is required. The attack is limited to one invoke per victim per deployment window (nonce=1 only), but it is repeatable across victims.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account`** transaction exists in the pool for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, rather than returning `true` for any transaction type. This closes the gap between the intended invariant ("skip validation only when the account's own deploy_account is pending") and the actual check ("skip validation when any transaction from that address is pending").

### Proof of Concept

```
1. Victim generates a fresh account at deterministic address A
   (class_hash, salt, constructor_calldata are public).

2. Victim submits deploy_account(sender=A, nonce=0, valid_sig).
   → Gateway: account_nonce=0, tx_nonce=0, passes all checks.
   → Mempool: deploy_account for A is now in tx_pool.

3. Attacker observes deploy_account for A in the mempool.

4. Attacker submits invoke(sender=A, nonce=1, calldata=drain_funds, sig=0x1337).
   → validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap  ✓
   → validate_by_mempool: unique hash, valid nonce gap  ✓
   → skip_stateful_validations:
       tx.nonce()==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(A)==true (victim's deploy_account) ✓
       → returns true (skip __validate__)
   → run_validate_entry_point: validate=false → returns Ok immediately
   → Attacker's invoke admitted to mempool.

5. Block is produced:
   a. deploy_account executes → A is deployed.
   b. Attacker's invoke executes → __validate__ called → invalid sig → REVERT.
   c. Fee charged to A.

6. Victim's legitimate invoke(nonce=1) was either:
   - Never submitted (victim only sent deploy_account), or
   - Replaced by attacker's invoke via fee escalation.
   Victim must resubmit and pays fees for the attacker's reverted tx.
``` [8](#0-7) [2](#0-1)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
