### Title
BLS12-381 Pairing Precompile Charges Zero Native Resources, Enabling Prover Resource Exhaustion — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/pairing.rs`)

---

### Summary

The `Bls12381PairingCheckPrecompile` charges EVM gas (ergs) proportionally to the number of input pairs but hardcodes `cost_native = 0`, meaning no native resources (RISC-V cycles) are charged. ZKsync OS uses Double Resource Accounting — both ergs and native resources must be charged to prevent forward/proving divergence. An unprivileged caller can submit a transaction calling this precompile with the maximum number of pairs the block gas limit allows, consuming significant RISC-V cycles without any native resource deduction, potentially making the block unprovable.

---

### Finding Description

In `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/pairing.rs`, the `execute` function for `Bls12381PairingCheckPrecompile` computes EVM gas proportionally to `num_pairs` but explicitly sets `cost_native = 0`:

```rust
let num_pairs = input.len() / BLS12_381_PAIR_LEN;
let cost_ergs = Ergs(
    ((num_pairs as u64) * BLS12_381_PAIRING_PER_PAIR_GAS + BLS12_381_PAIRING_FIXED_GAS)
        * ERGS_PER_GAS,
);
let cost_native = 0;                          // ← hardcoded zero
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
``` [1](#0-0) 

This is structurally inconsistent with every other expensive precompile in the codebase. The BN254 pairing precompile, for example, charges a large per-pair native cost:

```rust
pub const BN254_PAIRING_BASE_NATIVE_COST: u64 = native_with_delegations!(13_000_000, 500_000, 0);
pub const BN254_PAIRING_PER_PAIR_NATIVE_COST: u64 = BN254_PAIRING_BASE_NATIVE_COST;
``` [2](#0-1) 

And the BN254 pairing implementation charges both ergs and native:

```rust
let native_cost = (num_pairs as u64) * BN254_PAIRING_PER_PAIR_NATIVE_COST
    + BN254_PAIRING_BASE_NATIVE_COST;
resources.charge(&R::from_ergs_and_native(ergs_cost, ...native_cost...))?;
``` [3](#0-2) 

ZKsync OS's Double Resource Accounting model requires that native resources cover the RISC-V proving cost of every operation. The design documentation confirms this:

> "Double Resource Accounting," tracking both EVM Gas and "Native Resources" (RISC-V cycles) to ensure proving costs are covered. [4](#0-3) 

BLS12-381 pairing is among the most computationally expensive cryptographic operations. With `cost_native = 0`, the native resource budget of a transaction is never decremented by this precompile, regardless of how many pairs are processed.

---

### Impact Explanation

**Forward/proving divergence leading to an unprovable block.**

The sequencer (forward mode, x86/ARM) accepts a transaction because EVM gas is correctly charged and the transaction fits within the block gas limit. The prover (RISC-V mode) must then execute the actual BLS12-381 pairing computations, consuming RISC-V cycles that were never accounted for in the native resource budget. If the actual cycle cost of N pairs exceeds the native resource budget allocated to the transaction (which was never decremented by the pairing calls), the prover exhausts its resources and cannot produce a valid proof for the block.

This is a resource accounting bug that directly enables a valid-execution-unprovability attack: the sequencer's state transition is valid, but the prover cannot reproduce it within its resource envelope.

---

### Likelihood Explanation

The BLS12-381 pairing precompile is reachable by any unprivileged EOA via a standard `CALL` to the EIP-2537 precompile address. No privileged role, governance action, or special access is required. The attacker pays only EVM gas (which is charged correctly), and the cost is bounded by the block gas limit. With `BLS12_381_PAIRING_FIXED_GAS = 37700` and `BLS12_381_PAIRING_PER_PAIR_GAS = 32600`, a 30M-gas block allows roughly 919 pairs per transaction — each pair requiring a full BLS12-381 Miller loop and final exponentiation in RISC-V, with zero native resource deduction. [5](#0-4) 

---

### Recommendation

Set `cost_native` to a value proportional to `num_pairs`, mirroring the pattern used by `Bn254PairingCheckImpl`. The per-pair native cost should be benchmarked against the actual RISC-V cycle count for a BLS12-381 Miller loop + final exponentiation, and a constant analogous to `BN254_PAIRING_PER_PAIR_NATIVE_COST` should be defined in `basic_system/src/cost_constants.rs` and applied here.

---

### Proof of Concept

1. Compute the maximum number of BLS12-381 pairs that fit within the block gas limit:
   `N = floor((block_gas_limit - BLS12_381_PAIRING_FIXED_GAS) / BLS12_381_PAIRING_PER_PAIR_GAS)`
2. Construct a transaction calling the BLS12-381 pairing precompile with `N` valid (or invalid — gas is charged before the validity check at line 42) pairs.
3. The sequencer accepts the transaction: EVM gas is charged, native resources are not decremented (remain at their initial budget).
4. The prover attempts to execute the same transaction in RISC-V. Each of the N pairing operations consumes a large number of RISC-V cycles. The native resource counter, which was never pre-decremented, overflows or the prover exhausts its cycle budget.
5. The block is unprovable, causing a forward/proving divergence. [6](#0-5)

In ZKsyncOS we need a concept of "resources" to limit and charge for computation (mainly proving) and data.
This is trickier that it may seem: ZKsyncOS is built to be EVM gas-equivalent, meaning that EVM code execution should follow the same gas schedule as Ethereum. The main issue with this is that for several reasons the EVM gas schedule doesn't correspond to the costs of proving it. The solution to this problem is to perform double accounting of both Execution Environment gas (think of EVM gas) and a "native" computational resource, the latter modeling the cost of proving. More details about this solution can be found in [Double resource accounting](./double_resource_accounting.md)
