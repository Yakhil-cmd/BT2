### Title
Pusher delegation consent signature is replayable within its deadline, permanently defeating `revokePusher` — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` verifies a pusher's EIP-191 consent signature but stores no record that the signature was consumed. Because the only replay guard is a deadline check (`block.timestamp <= deadline`), the same signature is accepted an unlimited number of times before the deadline. A creator who holds a pusher's consent signature can call `allowPushers` again after the pusher calls `revokePusher`, silently re-establishing the delegation the pusher just cancelled.

---

### Finding Description

`allowPushers` constructs the signed digest as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only freshness gate is `_ensureDeadline`, which simply checks `block.timestamp <= deadline`: [2](#0-1) 

There is no nonce, no per-pusher sequence counter, and no `usedSignatures` mapping. The same `(deadline, pusher, creator)` tuple — and therefore the same signature bytes — is accepted on every call before the deadline.

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

But the creator still holds the original signature. One additional `allowPushers` call with the same bytes restores `namespaceRemapping[pusher] = creator`, overwriting the revocation: [4](#0-3) 

The code comment on `allowPushers` explicitly acknowledges the risk ("an undated signature could re-establish a delegation AFTER the pusher revoked it") and names the deadline as the mitigation, but the deadline only prevents replay *after* it expires — it does nothing to prevent repeated replay *before* it expires. [5](#0-4) 

---

### Impact Explanation

After the pusher calls `revokePusher`, the `fallback` push path resolves `creator = msg.sender` (own namespace): [6](#0-5) 

Once the creator replays the old signature, `namespaceRemapping[pusher]` is restored to `creator`, so every subsequent push from the pusher's wallet lands in the creator's namespace again — without the pusher's knowledge or consent. If the pusher believes they have revoked and stops pushing, the creator's namespace goes stale. If the pusher continues pushing (unaware of the re-delegation), the creator captures live oracle data the pusher intended to write to their own namespace. Either outcome corrupts the price feed consumed by any pool whose `priceProvider` reads from the creator's namespace, producing stale or misattributed bid/ask quotes that reach live swaps — a bad-price execution impact.

---

### Likelihood Explanation

The creator already holds the pusher's consent signature from the original `allowPushers` call; no additional off-chain capability is required. The creator only needs to submit one extra transaction after observing the pusher's `revokePusher` on-chain. The deadline is typically set days or weeks in the future (as shown in tests using `block.timestamp + 1 days`), giving the creator a wide replay window. [7](#0-6) 

---

### Recommendation

Track consumed signatures. The minimal fix is a `mapping(bytes32 => bool) public usedSignatures` keyed on the digest, set to `true` on first acceptance and checked before `ECDSA.recover`. Alternatively, introduce a per-pusher nonce (`mapping(address => uint256) public pusherNonce`) that is incremented on every successful `allowPushers` and `revokePusher`, and include it in the signed digest so each consent is single-use.

---

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with a far-future deadline.
uint256 deadline = block.timestamp + 365 days;
bytes memory sig = _signConsent(PUSHER_KEY, deadline, pusher, creator);

// 2. Creator establishes delegation — pusher's pushes land in creator namespace.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig));
assertEq(oracle.namespaceRemapping(pusher), creator);

// 3. Pusher revokes — intends to push into own namespace from now on.
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revocation confirmed

// 4. Creator replays the SAME signature — revocation is silently undone.
vm.prank(creator);
oracle.allowPushers(deadline, _arr(pusher), _arr(sig)); // no revert
assertEq(oracle.namespaceRemapping(pusher), creator);   // pusher re-delegated without consent

// 5. Pusher's next push lands in creator's namespace, not their own.
uint56 tsMs = uint56(block.timestamp * 1000);
vm.prank(pusher);
(bool ok,) = address(oracle).call(_wordAt(0, 0, _packRaw(999_000, 1, 1), tsMs));
assertTrue(ok);
// Creator namespace has the data; pusher's own namespace is empty.
assertGt(oracle.getOracleData(oracle.feedIdOf(creator, 0, 0)).price, 0);
assertEq(oracle.getOracleData(oracle.feedIdOf(pusher,  0, 0)).price, 0);
```
