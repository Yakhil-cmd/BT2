### Title
Pool Admin Can Permanently Lock All Accrued Protocol and Admin Fees by Setting a Reverting Fee Destination â€” (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` accepts any non-zero address with no timelock and no validation beyond the zero-address check. `collectFees` in `MetricOmmPool` transfers admin fees **before** protocol fees. If the admin fee destination reverts on ERC20 receipt, every code path that calls `collectFees` â€” including the permissionless `collectPoolFees`, the admin's own `setPoolAdminFees`, and the factory owner's `setPoolProtocolFee` â€” permanently reverts. The factory owner has no function to override `poolAdminFeeDestination`; only the pool admin can change it. All accrued fees are permanently locked in the pool with no recovery path.

---

### Finding Description

`setPoolAdminFeeDestination` stores the new destination immediately with only a zero-address guard: [1](#0-0) 

Inside `collectFees`, admin transfers execute first, before protocol transfers: [2](#0-1) 

`collectPoolFees` (permissionless) passes the stored destination directly: [3](#0-2) 

`setPoolAdminFees` and `setPoolProtocolFee` both call `collectFees` with the same stored destination before applying their config changes, so they also revert: [4](#0-3) [5](#0-4) 

The factory owner interface (`IMetricOmmPoolFactoryOwner`) exposes no function to override `poolAdminFeeDestination`: [6](#0-5) 

---

### Impact Explanation

All accrued protocol fees and admin fees are permanently locked in the pool. The factory owner cannot collect protocol fees, cannot update protocol fee rates for the pool (since `setPoolProtocolFee` also calls `collectFees` first), and cannot recover the locked tokens. The `collectTokens` rescue function on the factory only rescues tokens held **by the factory**, not by the pool.

---

### Likelihood Explanation

The pool admin is semi-trusted. The scope rules explicitly call out "fee collection destinations" as an admin-path attack surface. The destination change requires no timelock and takes effect immediately. A malicious or compromised pool admin can execute this in a single transaction after fees have accrued.

---

### Recommendation

1. In `setPoolAdminFeeDestination`, call `collectFees` with the **current** destination before updating it (mirroring the pattern used in `setPoolAdminFees`).
2. Add a factory-owner override function for `poolAdminFeeDestination` as a backstop.
3. In `collectFees`, transfer protocol fees independently of admin fees (separate try/catch or separate collection paths) so a reverting admin destination cannot block protocol fee extraction.

---

### Proof of Concept

```solidity
// 1. Pool admin sets a reverting contract as fee destination
factory.setPoolAdminFeeDestination(pool, address(new RevertOnReceive()));

// 2. Fees accrue via normal swaps (standard ERC20 tokens, no non-standard behavior)
router.swap(...);

// 3. Anyone calls collectPoolFees â€” reverts because admin transfer fires first
factory.collectPoolFees(pool); // REVERTS

// 4. Factory owner tries to update protocol fee â€” also reverts
factory.setPoolProtocolFee(pool, newFee, newNotional); // REVERTS

// All accrued fees are permanently locked in the pool.
```
