### Title
Unprivileged Signer Can Overwrite Any Account's E2EE Encryption Key — (`x/e2ee/keeper/keeper.go`, `x/e2ee/types/msg.go`)

---

### Summary

`MsgRegisterEncryptionKey` contains an `Address` field that is fully attacker-controlled. Neither `ValidateBasic` nor the keeper's `RegisterEncryptionKey` handler verifies that the transaction signer matches `req.Address`. Any funded account can overwrite any victim account's encryption key in the KV store.

---

### Finding Description

The message type is defined with two free fields: [1](#0-0) 

`ValidateBasic` only checks bech32 format and age key validity — no signer-vs-address equality check: [2](#0-1) 

The keeper handler writes directly to the KV store using `req.Address` with no signer check: [3](#0-2) 

The internal write path: [4](#0-3) 

A `GetSigners()` method returning `[Address]` is never defined anywhere in the module — a grep across all `x/e2ee/**/*.go` files returns zero matches for `GetSigners`, `signers`, or `signer`. Without this, the Cosmos SDK framework only requires that *someone* signs the transaction; it does not enforce that the signer equals the `Address` field in the message body.

---

### Impact Explanation

An attacker (account A) broadcasts `MsgRegisterEncryptionKey{Address: B, Key: attacker_age_pubkey}` signed by A. The keeper writes `attacker_age_pubkey` to `KeyPrefix(B)`: [5](#0-4) 

After this:
- Any sender who queries B's on-chain key and encrypts a message to it will encrypt to the attacker's key.
- The attacker can decrypt all future messages intended for B.
- B's legitimate key is silently overwritten; B loses the ability to receive encrypted messages.

This is **corruption of e2ee key/message state with direct security impact** — a High-severity finding per the scope rules.

---

### Likelihood Explanation

The attack requires only a funded account and a single broadcast transaction. No special privileges, leaked keys, or external assumptions are needed. It is trivially repeatable and permanent until B re-registers (which the attacker can immediately overwrite again in a race).

---

### Recommendation

In `keeper.RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    signers := sdkCtx.TxSigner() // or retrieve from msg server context
    // enforce signers[0] == req.Address
    ...
}
```

Alternatively, define `GetSigners()` on `MsgRegisterEncryptionKey` to return `[]sdk.AccAddress{sdk.MustAccAddressFromBech32(m.Address)}` so the SDK framework enforces the constraint at the ante-handler level, and add an explicit equality check in the keeper as defense-in-depth.

---

### Proof of Concept

```go
// Unit test sketch
addrA := createAccount(t, "attacker")
addrB := createAccount(t, "victim")
attackerAgeKey := generateAgeKey(t)

msg := &types.MsgRegisterEncryptionKey{
    Address: addrB.String(),   // victim's address
    Key:     attackerAgeKey,   // attacker's age pubkey
}
// Sign and broadcast with addrA's key
_, err := msgServer.RegisterEncryptionKey(ctx, msg)
require.NoError(t, err)

// Assert victim's slot now holds attacker's key
resp, _ := keeper.Key(ctx, &types.KeyRequest{Address: addrB.String()})
require.Equal(t, attackerAgeKey, resp.Key) // passes — victim's key is overwritten
```

```go
func (k Keeper) registerEncryptionKey(
	ctx context.Context,
	address string,
	key []byte,
) error {
	bz, err := k.addressCodec.StringToBytes(address)
	if err != nil {
		return err
	}
	sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
	return nil
}
```
