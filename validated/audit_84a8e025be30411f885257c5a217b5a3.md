### Title
`TurnBridge` Is a No-Op Stub — Bridge Cannot Be Disabled and No Bridge-Active Guard Exists in Any Bridge Path - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler in `msg_server.go` is an unimplemented stub that unconditionally returns `nil, nil` without setting any state. No bridge-active flag exists in the module params, and none of the bridge execution paths (`IbcTransferCoins`, `ConvertVouchersToEvmCoins`, `SendToIbcHandler`, `SendToIbcV2Handler`) check any such flag. As a result, the bridge can never be disabled: an admin calling `TurnBridge(false)` receives a success response but the bridge continues operating normally, and any unprivileged user can continue to bridge assets indefinitely.

### Finding Description
`msgServer.TurnBridge` at `x/cronos/keeper/msg_server.go:85-87` is:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

It performs no permission check, sets no state, and returns no error. The `CanTurnBridge` permission constant (`x/cronos/keeper/permissions.go:15`) exists and is defined, but is never consulted inside `TurnBridge`. Compare with `UpdateTokenMapping`, which correctly gates on `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`.

No `bridge_active` (or equivalent) field exists in `Params` (`x/cronos/types/params.go`). Consequently, every bridge execution path proceeds unconditionally:

- `IbcTransferCoins` (`x/cronos/keeper/ibc.go:80`) — no bridge-active check.
- `ConvertVouchersToEvmCoins` (`x/cronos/keeper/ibc.go:21`) — no bridge-active check.
- `SendToIbcHandler.Handle` (`x/cronos/keeper/evmhandlers/send_to_ibc.go:67`) — no bridge-active check.
- `SendToIbcV2Handler.Handle` (`x/cronos/keeper/evmhandlers/send_to_ibc_v2.go:70`) — no bridge-active check.
- `SendCroToIbcHandler.Handle` (`x/cronos/keeper/evmhandlers/send_cro_to_ibc.go:68`) — checks only `CroBridgeContractAddresses`, not a bridge-active flag.

The `MsgTransferTokens` server method (`msg_server.go:47`) calls `IbcTransferCoins` directly with no guard, and the EVM hook dispatcher (`evm_hooks.go:28`) calls every registered handler unconditionally.

### Impact Explanation
An authorized admin or governance actor calling `TurnBridge(false)` to halt the bridge in an emergency receives a success response but achieves nothing. Every unprivileged user can continue to:
- Emit `__CronosSendToIbc` / `__CronosSendCroToIbc` EVM logs that trigger IBC transfers of CRO and IBC vouchers.
- Submit `MsgTransferTokens` to move IBC vouchers and CRC20-backed tokens across chains.
- Submit `MsgConvertVouchers` to mint EVM-side tokens from incoming IBC packets.

This is a permanent bypass of the bridge-disable control: the bridge-active toggle is structurally inoperative. This maps to **High: Bypass of Cronos admin/governance bridge authorization checks**, and depending on the emergency scenario (e.g., a compromised CRC20 contract or IBC channel exploit), it can escalate to **Critical: unauthorized bridge/conversion/escrow-release of CRO, IBC vouchers, or CRC20 assets**.

### Likelihood Explanation
The `CanTurnBridge` permission, the `MsgTurnBridge` proto message, and the CLI command `CmdTurnBridge` all exist and are wired into the module, signaling clear design intent to support bridge disabling. The stub implementation is reachable by any node operator or governance actor who submits a `MsgTurnBridge` transaction. The absence of any bridge-active flag in params means the missing guard in bridge functions is not a latent issue — it is structurally impossible to add the guard without first fixing the stub.

### Recommendation
1. Add a `BridgeActive bool` field to `Params` in `proto/cronos/cronos.proto` and regenerate.
2. Implement `TurnBridge` in `msg_server.go` to: (a) check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`, (b) read current params, (c) set `params.BridgeActive = msg.Enable`, (d) call `k.SetParams(ctx, params)`.
3. Add a guard at the top of `IbcTransferCoins` and `ConvertVouchersToEvmCoins`:
   ```go
   if !k.GetParams(ctx).BridgeActive {
       return errors.New("bridge is disabled")
   }
   ```
4. Add the same guard in `SendToIbcHandler.handle`, `SendToIbcV2Handler.Handle`, and `SendCroToIbcHandler.Handle`.

### Proof of Concept
1. Governance or admin calls `MsgTurnBridge{Enable: false}` → tx succeeds, `TurnBridge` returns `nil, nil`, no state written.
2. Unprivileged user submits `MsgTransferTokens{From: alice, To: "cosmos1...", Coins: [1000basecro]}` → `TransferTokens` calls `IbcTransferCoins` with no bridge-active check → IBC transfer executes, CRO leaves Cronos.
3. Alternatively, unprivileged user deploys or uses an existing CRC20 contract mapped to an IBC denom, emits `__CronosSendToIbc` log → `SendToIbcHandler.Handle` fires, mints/transfers coins, initiates IBC transfer — all without any bridge-active check.

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
