### Title
Missing Participant-Set Consistency Check When Combining Beaver Triples in OT-Based ECDSA Presign - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA `presign` function explicitly omits validation that `triple0` and `triple1` were generated by the same participant set as the current presign session. A malicious participant can supply two Beaver triples from different triple-generation sessions (different participant sets) into a single presign invocation. Because the only cross-triple check is threshold equality, the resulting presignature is built from cryptographically inconsistent polynomial material, which honest parties accept without detecting the mismatch.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign` initializer performs the following validation on the two input triples:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The code comment explicitly acknowledges the omission: the participant set used during triple generation is **not** verified against the presign participant set, and is not verified to be consistent between `triple0` and `triple1`. The only enforced invariant is that both triples carry the same threshold value.

This is a direct structural analog to the ZeroLend NFT merge bug. In that bug, two individually valid NFTs (each within MAXTIME) are merged: the `start` is taken from NFT1 and the `end` from NFT2, producing a combined time span that exceeds MAXTIME because no validation is applied to the *combined* result. Here, two individually valid Beaver triples (each generated under a valid participant set and threshold) are combined into a single presign session. The threshold check passes for each triple in isolation, but no check enforces that both triples share the same participant set, or that either matches the presign participant set.

In the OT-based ECDSA scheme, Beaver triples are polynomial sharings: each participant holds an evaluation of the triple polynomial at their own identifier. If `triple0` was generated by participant set P₁ and `triple1` by participant set P₂ ≠ P₁, the polynomial evaluations embedded in the two triples are over different domains. When the presign protocol combines these evaluations to compute the presignature components, the resulting values are cryptographically inconsistent — the combined presignature does not correspond to a coherent secret sharing of any well-defined nonce. ---

### Impact Explanation

**Impact: High — Corruption of presign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs.**

Honest participants executing the presign protocol with mismatched triples will each compute their local presignature shares using polynomial evaluations that belong to different sessions. The protocol's internal consistency checks (e.g., OT-based verification of triple correctness) operate per-triple and do not cross-validate the two triples against each other or against the presign participant set. As a result, the protocol completes and outputs a `PresignOutput` that appears structurally valid but is cryptographically incoherent. This presignature cannot produce a valid ECDSA signature, permanently denying the ability to sign for that presignature slot, and honest parties have no mechanism to detect the corruption at presign time.

---

### Likelihood Explanation

**Likelihood: Medium.**

A malicious participant who has taken part in two or more triple-generation sessions with different participant sets can trivially mix triples across sessions. The attack requires no cryptographic break, no external compromise, and no privileged access — only the ability to call `presign` with attacker-chosen `PresignArguments`. The omission is explicitly documented in the source code comment, confirming it is a known gap rather than an oversight in the comment. Any library caller who constructs `PresignArguments` manually (the public API) can trigger this path.

---

### Recommendation

1. **Store the participant set inside the triple output** (`TripleOutput` or equivalent) at the time of triple generation.
2. **In `presign`, assert** that `triple0.participant_set == triple1.participant_set == current presign participant set` before proceeding.
3. Remove or update the comment that describes this check as intentionally omitted, since the assumption ("presumably they need to have been present") is not enforced and can be violated by a malicious caller.
