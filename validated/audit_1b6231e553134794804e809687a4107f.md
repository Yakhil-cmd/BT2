### Title
Missing `max_l2_gas_amount` Bound Check for Declare Transactions in Gateway Stateless Validator — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces a `max_l2_gas_amount` ceiling for Invoke and DeployAccount transactions but explicitly skips this check for Declare transactions. Any unprivileged user can submit a Declare transaction with `l2_gas.max_amount = u64::MAX`, bypassing the gateway's admission control and forcing the mempool to accept a transaction that violates the configured resource-bound invariant.

---

### Finding Description

In `validate_resource_bounds`, lines 78–85, the code explicitly branches on `RpcTransaction::Declare(_)` and skips the `max_l2_gas_amount` check:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The production config sets `max_l2_gas_amount = 1,210,000,000`: [2](#0-1) 

For Invoke and DeployAccount, any transaction with `l2_gas.max_amount > 1,210,000,000` is rejected at the gateway. For Declare, no such limit exists. The test `valid_l2_gas_amount_on_declare` explicitly documents and confirms this asymmetry — a Declare with `max_amount: GasAmount(200)` when `max_l2_gas_amount: 100` passes validation: [3](#0-2) 

The test for the enforced path (`test_invalid_max_l2_gas_amount`) only covers `TransactionType::DeployAccount` and `TransactionType::Invoke`, confirming Declare is intentionally excluded: [4](#0-3) 

The stateful validator's `validate_resource_bounds` only checks the L2 gas **price** against the previous block, not the gas **amount**, so it provides no downstream protection: [5](#0-4) 

The blockifier's `perform_pre_validation_stage` calls `verify_can_pay_committed_bounds`, which computes `max_possible_fee = max_amount × max_price_per_unit`. With `max_amount = u64::MAX` and any non-zero price, this produces an astronomically large fee commitment. The balance check then fails at execution time rather than at admission time: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

The gateway admits Declare transactions that violate the `max_l2_gas_amount` invariant enforced for all other transaction types. These transactions enter the mempool and are handed to the batcher, which wastes execution resources on Sierra compilation and blockifier pre-validation before the balance check ultimately fails. The admission decision is wrong: a transaction that should be rejected at the stateless gateway layer is instead accepted, matching the **High** impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

---

### Likelihood Explanation

Any unprivileged user can submit a valid Declare V3 transaction (with a valid Sierra class) and set `l2_gas.max_amount = u64::MAX`. No special privileges, keys, or on-chain state are required. The stateless validator is the first and only line of defense for this check, and it is bypassed unconditionally for Declare.

---

### Recommendation

Remove the `if let RpcTransaction::Declare(_) = tx { }` branch and apply the `max_l2_gas_amount` check uniformly across all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

Update `test_invalid_max_l2_gas_amount` to include `TransactionType::Declare` in its `#[values(...)]` list, and remove the separate `valid_l2_gas_amount_on_declare` test that currently documents the broken behavior.

---

### Proof of Concept

1. Construct a valid Declare V3 transaction with a well-formed Sierra class (passes `validate_declare_tx`).
2. Set `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`.
3. Set `resource_bounds.l2_gas.max_price_per_unit` to any value ≥ `min_gas_price` (e.g., `GasPrice(1)`).
4. Submit to the gateway's `add_declare_transaction` endpoint.
5. **Observed**: The stateless validator passes — the `max_l2_gas_amount` check is skipped for Declare. The transaction enters the mempool with `max_l2_gas_amount = 18,446,744,073,709,551,615`, violating the configured limit of `1,210,000,000`.
6. **Expected**: The gateway should reject the transaction with `StatelessTransactionValidatorError::MaxGasAmountTooHigh`, exactly as it does for Invoke and DeployAccount.

The existing test at line 191–201 already demonstrates this path passes without error.

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
