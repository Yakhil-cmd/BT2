### Title
EVM `StateDB.Commit()` Silently Drops Re-Created Account State, Causing Cross-VM Asset Loss — (`fvm/evm/emulator/state/stateDB.go`)

---

### Summary

When an EVM account is self-destructed and then re-created in the same transaction (e.g., via CREATE2 re-deployment), `StateDB.Commit()` deletes the account from the base view but never re-creates it. The new account's balance, code, and storage slots are silently dropped. Any FLOW tokens deposited to the re-created EVM account during that transaction are permanently lost.

---

### Finding Description

`DeltaView.CreateAccount` handles the "re-creation of an existing account" case by internally calling `SelfDestruct(addr)` (which adds the address to `toBeDestructed`) and simultaneously adding the address to both `created` and `recreated`: [1](#0-0) 

This means for a recreated account, both `HasSelfDestructed(addr)` and `IsCreated(addr)` return `true`.

In `StateDB.Commit()`, the commit loop processes dirty addresses as follows: [2](#0-1) 

The `HasSelfDestructed` check fires first. It returns `true` for the recreated account (because `toBeDestructed` contains the address), so `DeleteAccount` is called and `deleted = true`. The `if deleted { continue }` guard then **skips the `IsCreated` branch entirely**. The re-created account's new state — balance, code, and new storage slots — is never written to the `baseView`.

`StateDB.HasSelfDestructed` delegates to `latestView().HasSelfDestructed`, which checks `toBeDestructed` without any awareness of the `recreated` flag: [3](#0-2) 

The `recreated` map exists in `DeltaView` specifically to handle this scenario (it is used in `GetState` to return empty values instead of falling through to the parent), but it is **never consulted in `Commit()`**: [4](#0-3) 

The `BaseView.CreateAccount` function, which would write the new account to persistent storage, is never reached for recreated accounts: [5](#0-4) 

---

### Impact Explanation

After `Commit()`, the re-created account does not exist in the `baseView`. Any FLOW tokens deposited to the re-created EVM address during the transaction are permanently lost — the balance is recorded in the delta view but never flushed to persistent storage. The re-deployed contract code is not stored. The transaction completes without error, making the loss invisible to the caller. This constitutes **cross-VM asset loss**: FLOW tokens bridged into EVM and deposited to a re-created address are unrecoverable.

---

### Likelihood Explanation

CREATE2 re-deployment (self-destruct then re-deploy to the same address) is a standard EVM pattern used in upgradeable proxy systems and factory contracts. Any unprivileged EVM transaction sender can trigger this path. No special privileges, staked node access, or compromised keys are required. The attacker only needs to submit a valid EVM transaction that performs a CREATE2 re-deployment.

---

### Recommendation

In `StateDB.Commit()`, check for the recreated case before skipping. When an account is both self-destructed and re-created, the correct behavior is to delete the old account and then re-create it with the new state. The fix should be:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    if err != nil { ... }
    // If also re-created, fall through to the IsCreated branch below
    if !db.IsCreated(addr) {
        continue
    }
}
// IsCreated branch handles both fresh creates and recreated accounts
if db.IsCreated(addr) {
    err = db.baseView.CreateAccount(addr, bal, nonce, code, codeHash)
    ...
    continue
}
```

Alternatively, expose and check the `recreated` flag from `DeltaView` in the `Commit()` logic.

---

### Proof of Concept

1. An EVM transaction deploys contract A at address X via CREATE2. `DeltaView.CreateContract(X)` is called, setting `newContract[X]`.
2. Contract A executes `SELFDESTRUCT`. `SelfDestruct6780(X)` fires (since `IsNewContract(X)` is true), adding X to `toBeDestructed` with its balance.
3. The same transaction re-deploys to address X via CREATE2. The EVM calls `StateDB.CreateAccount(X)`.
4. Inside `DeltaView.CreateAccount(X)`: X already exists (in `toBeDestructed`), so `HasSelfDestructed` returns true; the pre-destruction balance is carried over; `recreated[X]` and `created[X]` are set. [6](#0-5) 
5. The new contract receives FLOW tokens; `AddBalance(X, amount)` updates `balances[X]` in the delta view.
6. `StateDB.Commit()` is called. For address X: `HasSelfDestructed(X)` = true → `baseView.DeleteAccount(X)` → `deleted = true` → `continue`. The `IsCreated(X)` branch is never reached. [2](#0-1) 
7. After the transaction, address X does not exist in the `baseView`. The FLOW tokens deposited to X are permanently lost. The transaction returned no error.

```go
func (d *DeltaView) CreateAccount(addr gethCommon.Address) error {
	// if is already created return
	if d.IsCreated(addr) {
		return nil
	}
	exist, err := d.Exist(addr)
	if err != nil {
		return err
	}
	if exist {
		// check if already destructed
		destructed, balance := d.HasSelfDestructed(addr)
		if !destructed {
			balance, err = d.GetBalance(addr)
			if err != nil {
				return err
			}
			err = d.SelfDestruct(addr)
			if err != nil {
				return err
			}
		}

		d.nonces[addr] = 0
		d.codes[addr] = nil
		d.codeHashes[addr] = gethTypes.EmptyCodeHash
		// carrying over the balance. (legacy behavior of the Geth stateDB)
		d.balances[addr] = balance

		// flag addr as recreated, this flag helps with postponing deletion of slabs
		// otherwise we have to iterate over all slabs of this account and set the to nil
		d.recreated[addr] = struct{}{}

		// remove slabs from cache related to this account
		for k := range d.slots {
			if k.Address == addr {
				delete(d.slots, k)
			}
		}
	}
	d.dirtyAddresses[addr] = struct{}{}
	d.created[addr] = struct{}{}
	return nil
```

```go
// GetState returns the value of the slot of the main state
func (d *DeltaView) GetState(sk types.SlotAddress) (gethCommon.Hash, error) {
	val, found := d.slots[sk]
	if found {
		return val, nil
	}
	// if address is deleted in the scope of this delta view,
	// don't go backward. this has been done to skip the step to iterate
	// over all the state slabs and delete them.
	_, recreated := d.recreated[sk.Address]
	if recreated {
		return gethCommon.Hash{}, nil
	}
	return d.parent.GetState(sk)
}
```

```go
// CreateAccount creates a new account
func (v *BaseView) CreateAccount(
	addr gethCommon.Address,
	balance *uint256.Int,
	nonce uint64,
	code []byte,
	codeHash gethCommon.Hash,
) error {
	var colID []byte
	// if is an smart contract account
	if len(code) > 0 {
		err := v.updateAccountCode(addr, code, codeHash)
		if err != nil {
			return err
		}
	}

	// create a new account and store it
	acc := NewAccount(addr, balance, nonce, codeHash, colID)

	// no need to update the cache , storeAccount would update the cache
	return v.storeAccount(acc)
}
```
