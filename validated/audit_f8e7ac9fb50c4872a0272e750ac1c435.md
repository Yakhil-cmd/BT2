### Title
Global `last_reward_block` Enforces a Single-Staker-Per-Block Reward Update Limit, Causing Permanent Loss of Unclaimed Yield at Scale - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function uses a single **global** `last_reward_block` storage variable to enforce a one-call-per-block limit. Because this variable is not keyed per staker, only **one staker** can have their block rewards updated per block. With a large number of stakers, the sequencer cannot update all stakers within a single epoch. Stakers who are not updated during an epoch receive **zero rewards** for that epoch — a permanent, irrecoverable loss of unclaimed yield.

---

### Finding Description

In `StakingRewardsManagerImpl::update_rewards` (V3 consensus rewards path), the guard at the top of the function reads a single global `last_reward_block`:

```cairo
// src/staking/staking.cairo:1453-1458
let current_block_number = starknet::get_block_number();
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

After any staker's rewards are processed, the global is updated:

```cairo
// src/staking/staking.cairo:1484-1485
// Update last block rewards.
self.last_reward_block.write(current_block_number);
```

The storage declaration confirms this is a single scalar, not a per-staker map:

```cairo
// src/staking/staking.cairo:186-187
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

This means: once `update_rewards` is called for **any** staker in block N, `last_reward_block = N`, and every subsequent call in block N reverts with `REWARDS_ALREADY_UPDATED`. Only one staker can be served per block.

The spec documents the access control as "Only starkware sequencer," but the implementation contains no caller-identity check beyond `general_prerequisites()` (which only enforces the pause flag). Tests confirm the function is callable by any address without `cheat_caller_address_once`:

```cairo
// src/staking/tests/test.cairo:3515
staking_rewards_dispatcher.update_rewards(:staker_address, disable_rewards: false);
```

The reward model for V3 (consensus rewards) is strictly block-by-block. If `update_rewards` is never called for a staker during an epoch, that staker earns **zero** rewards for the epoch — this is not recoverable in a later epoch. The flow test explicitly confirms this:

```cairo
// src/flow_test/test.cairo:4119-4121
// New epoch - Miss epoch.
system.advance_epoch();
assert!(system.staker_claim_rewards(:staker) == Zero::zero());
```

---

### Impact Explanation

**High — Permanent freezing / theft of unclaimed yield.**

With `E` blocks per epoch and `S` stakers, at most `E` stakers can receive rewards per epoch. Any staker beyond the `E`-th slot receives **zero** rewards for that epoch, permanently. As the protocol scales (more stakers join), an increasing fraction of stakers will be systematically excluded from reward distribution each epoch. The lost yield is never retroactively credited.

---

### Likelihood Explanation

**Medium.**

Starknet epochs are measured in blocks. The epoch length is configurable (`set_epoch_info`). If the number of active stakers exceeds the epoch length in blocks, the bottleneck is guaranteed to cause reward loss. Even below that threshold, any actor who calls `update_rewards` for their own staker in a block "consumes" that block's slot, preventing the sequencer from updating a different staker. Because `update_rewards` is publicly callable (no enforced access control), any staker can selfishly consume every block's slot for themselves, starving other stakers of rewards.

---

### Recommendation

Replace the global `last_reward_block: BlockNumber` with a **per-staker** mapping:

```cairo
last_reward_block: Map<ContractAddress, BlockNumber>,
```

Update the guard and write to key on `staker_address`:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(staker_address),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
// ...
self.last_reward_block.write(staker_address, current_block_number);
```

This allows all stakers to be updated within the same block, eliminating the artificial serialization bottleneck.

Additionally, enforce the "Only starkware sequencer" access control stated in the spec to prevent public griefing of the slot.

---

### Proof of Concept

1. Deploy the protocol with epoch length `E = 100` blocks.
2. Register `S = 200` stakers (S > E).
3. Consensus rewards are active.
4. The sequencer calls `update_rewards` for one staker per block across the 100-block epoch — updating stakers 1–100.
5. Stakers 101–200 are never updated during the epoch.
6. The epoch advances.
7. `staker_claim_rewards` for any staker in the 101–200 range returns `0` — they earned nothing for the epoch despite having valid, active stakes.
8. The lost rewards are permanent; no catch-up mechanism exists.
