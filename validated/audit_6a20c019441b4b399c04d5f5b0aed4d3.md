### Title
EVM Block Timestamps Advance During EVM Pause, Enabling Instant Liquidations After Unpause - (File: fvm/evm/stdlib/contract.cdc)

### Summary
The `EVM.Heartbeat.heartbeat()` function is not guarded by `EVM.isPaused()`, causing EVM blocks to continue being committed with advancing timestamps even when all EVM user operations are paused. EVM DeFi users with time-sensitive positions (loans, options, liquidation deadlines) cannot take protective action during the pause, and may be instantly liquidated when the pause is lifted.

### Finding Description
The `EVM` system contract exposes a governance-controlled pause mechanism. When paused, every state-mutating EVM operation checks `isPaused()` and reverts:

- `EVMAddress.deposit()` — `!EVM.isPaused()` [1](#0-0) 
- `CadenceOwnedAccount.withdraw()` — `!EVM.isPaused()` [2](#0-1) 
- `EVM.run()`, `EVM.batchRun()`, `CadenceOwnedAccount.call()`, `deploy()`, `withdrawNFT()`, `withdrawTokens()`, etc. — all guarded [3](#0-2) 

The contract documentation explicitly states the intent: *"The EVM enters a read-only mode, where all EVM state is available for reading, but no state updates are executed."* [4](#0-3) 

However, the `EVM.Heartbeat` resource's `heartbeat()` function — which is invoked unconditionally every Flow block via the system chunk transaction — is **not** guarded by `isPaused()`:

```cadence
access(all) resource Heartbeat {
    access(all)
    fun heartbeat() {
        InternalEVM.commitBlockProposal()  // ← no isPaused() check
    }
}
``` [5](#0-4) 

The system chunk transaction calls this every block:

```cadence
let evmHeartbeat = serviceAccount.storage
    .borrow<&EVM.Heartbeat>(from: /storage/EVMHeartbeat)
evmHeartbeat?.heartbeat()
``` [6](#0-5) 

`InternalEVM.commitBlockProposal()` commits a new EVM block whose timestamp is derived from the Flow block header timestamp. This is confirmed by the EVM block data test: [7](#0-6) 

```go
require.Equal(t, ctx.BlockHeader.Timestamp/1000, new(big.Int).SetBytes(res.ReturnedData).Uint64())
```

After each commit, a new block proposal is immediately constructed with the next timestamp: [8](#0-7) 

The result is that **EVM `block.timestamp` advances with every Flow block, even while EVM is paused**. EVM DeFi protocols (lending, options, perpetuals) that use `block.timestamp` for liquidation deadlines, loan expiry, or collateral ratio checks will see time pass during the pause. Users cannot submit any EVM transactions to repay loans, add collateral, or close positions because all such operations are blocked by `isPaused()`. When the pause is lifted, liquidators can immediately call liquidation functions on positions that became eligible during the pause.

### Impact Explanation
**High.** Users holding positions in EVM DeFi protocols deployed on Flow EVM can lose their collateral assets to instant liquidation immediately after the EVM is unpaused. The loss is permanent. The attacker-controlled entry path is: any unprivileged EVM transaction sender submits a liquidation call to a DeFi contract in the first block after unpause, targeting positions whose time-based conditions expired during the pause. The Flow protocol's `EVM` system contract is the necessary vulnerable step — it is the component that continues advancing `block.timestamp` while blocking all user-protective actions.

### Likelihood Explanation
**Low.** Exploitation requires the Governance Committee to pause the EVM for a duration long enough for time-based conditions in deployed DeFi contracts to expire (e.g., a loan term, an option expiry, or a grace period). This is a realistic operational scenario (emergency maintenance, upgrade), not a theoretical one, as the pause mechanism is explicitly designed for governance use.

### Recommendation
Guard `EVM.Heartbeat.heartbeat()` with `isPaused()` to halt EVM block production during a pause:

```cadence
access(all)
fun heartbeat() {
    if !EVM.isPaused() {
        InternalEVM.commitBlockProposal()
    }
}
```

Alternatively, if EVM block production must continue for protocol reasons, add a post-unpause grace period during which liquidation-type calls are disabled, analogous to the `GossipSubAppSpecificScoreRegistry`'s startup silence period pattern already present in the codebase. [9](#0-8) 

### Proof of Concept

1. A user deposits FLOW collateral into an EVM lending protocol deployed on Flow EVM and borrows against it, with a 30-day loan term enforced by `block.timestamp`.
2. The Governance Committee pauses the EVM (e.g., for a protocol upgrade) by storing `true` at `/storage/evmOperationsPaused`. [10](#0-9) 
3. During the pause, the user attempts to repay the loan via `coa.call(...)` — this reverts with `"EVM operations are temporarily paused"`. [11](#0-10) 
4. Meanwhile, `EVM.Heartbeat.heartbeat()` is called every Flow block via the system chunk transaction, committing new EVM blocks with advancing timestamps. After 30 days of Flow blocks, the EVM `block.timestamp` has advanced past the loan expiry. [12](#0-11) 
5. The Governance Committee unpauses the EVM.
6. In the first block after unpause, a liquidator calls the lending protocol's `liquidate()` function. The protocol checks `block.timestamp > loanExpiry` — this is now `true`. The user's collateral is seized with no opportunity to repay.
