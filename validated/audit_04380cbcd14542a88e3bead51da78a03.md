### Title
L2 Mint Requests Sent Without Checking L1 `PERIOD_MINT_CAP`, Causing Temporary Freeze of Unclaimed Rewards - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The L2 `RewardSupplier` sends mint request messages to L1 without validating against the L1 `PERIOD_MINT_CAP` (6,500,000 STRK/week). When accumulated pending requests exceed this cap, the L1 `RewardSupplier.tick()` reverts on the second call within the same week, tokens are never bridged to L2, and stakers/delegators cannot claim their accrued rewards until the next period resets the cap.

---

### Finding Description

**Asymmetric validation pattern (direct analog to the Taiko QuotaManager bug):**

**Send path (L2 → L1, no cap check):**

In `src/reward_supplier/reward_supplier.cairo`, `request_funds` calculates how many mint request messages to send based purely on L2-local state (`balance`, `l1_pending_requested_amount`, `unclaimed_rewards`). It sends as many messages as needed and immediately increments `l1_pending_requested_amount` — with no awareness of the L1 weekly minting cap: [1](#0-0) 

Each message is sent via `send_mint_request_to_l1_reward_supplier`, which fires a raw L2→L1 syscall with `base_mint_amount` as payload: [2](#0-1) 

**Fulfillment path (L1, cap IS enforced):**

On L1, `RewardSupplier.tick()` processes at most `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5` messages per call, minting `5 × 1,300,000 = 6,500,000 STRK` — exactly the weekly cap: [3](#0-2) 

`MintManager.mintRequest` then calls `checkAndUpdatePeriodicalQuota`, which enforces `PERIOD_MINT_CAP = 6,500,000 STRK` per week: [4](#0-3) 

If `tick()` is called a second time within the same 7-day window (because more than 5 messages are queued), `checkAndUpdatePeriodicalQuota` reverts with `"EXCEED_PERIOD_MINTING"`. The excess L2→L1 messages remain unconsumed in the Starknet messaging queue.

**Consequence on L2:**

`l1_pending_requested_amount` was already incremented when the messages were sent. The L2 contract believes these funds are "in transit" and counts them as credit: [5](#0-4) 

But `on_receive` (which decrements `l1_pending_requested_amount`) is never called because the tokens never arrive: [6](#0-5) 

The actual STRK token balance of the reward supplier is therefore insufficient to cover `unclaimed_rewards`. When the staking contract calls `claim_rewards` on the reward supplier, the `checked_transfer` reverts: [7](#0-6) 

---

### Impact Explanation

Stakers and delegators calling `claim_rewards` on the staking or pool contracts will receive a revert because the reward supplier cannot transfer tokens it does not hold. Unclaimed yield is temporarily frozen until the next weekly period resets the `PERIOD_MINT_CAP` and a caller invokes `tick()` again to process the backlogged messages. This matches the allowed impact: **High — Temporary freezing of unclaimed yield**.

---

### Likelihood Explanation

Under steady-state conditions (≈215,000 STRK/week in rewards at 700M STRK staked), the weekly cap is not approached. However, the scenario becomes realistic after:

- A protocol pause followed by resumption, causing a large backlog of `unclaimed_rewards` to be registered at once.
- A rapid growth in total staked STRK pushing weekly rewards above 6,500,000 STRK.
- Multiple independent callers triggering `update_unclaimed_rewards_from_staking_contract` in rapid succession within one week, each causing `request_funds` to emit additional messages.

No privileged access is required. Any staker or delegator triggering normal reward accrual paths can cause messages to accumulate.

---

### Recommendation

Before sending mint request messages in `request_funds`, validate that the total amount being requested in the current L1 period does not exceed `PERIOD_MINT_CAP`. Since L2 cannot read L1 state directly, the simplest mitigation is to cap the number of messages sent per epoch to `MAX_MESSAGES_TO_PROCESS_PER_TICK` (5), matching the L1 processing limit. Alternatively, implement a cooldown on `request_funds` so that no more than 5 messages are emitted per 7-day window, aligning L2 send-rate with the L1 cap.

---

### Proof of Concept

1. Protocol is paused for 3 weeks; `unclaimed_rewards` accumulates to 20,000,000 STRK.
2. Protocol is unpaused; `update_unclaimed_rewards_from_staking_contract` is called.
3. `request_funds` computes `diff` requiring 16 messages (16 × 1,300,000 = 20,800,000 STRK). All 16 messages are sent via `send_message_to_l1_syscall`; `l1_pending_requested_amount` is set to 20,800,000 STRK.
4. A public caller invokes `tick()` on L1. It processes 5 messages, mints 6,500,000 STRK, and bridges them to L2. `PERIOD_MINT_CAP` is now exhausted for this week.
5. A second caller invokes `tick()` in the same week. `requiredMinting()` returns 5 more messages. `mintManager().mintRequest(token, 6_500_000e18)` reverts: `"EXCEED_PERIOD_MINTING"`. The 11 remaining messages stay in the queue.
6. The reward supplier's L2 balance is only 6,500,000 STRK, but `unclaimed_rewards` is 20,000,000 STRK.
7. A staker calls `claim_rewards` for 8,000,000 STRK. The `assert!(amount <= unclaimed_rewards)` passes, but `checked_transfer` reverts — insufficient balance. Rewards are frozen until the next weekly period. [8](#0-7) [9](#0-8) [10](#0-9)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

    function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
        bytes32 periodSlot = periodAccountingSlot(token);
        uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
        uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
        require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
        periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
    }
