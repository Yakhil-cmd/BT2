### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator's Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary
The `calculate_rewards` function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over every unclaimed balance checkpoint in a delegator's `pool_member_epoch_balance` trace. Because each balance-modifying action in a new epoch appends a new checkpoint with no cap, a delegator who makes frequent balance changes across many epochs without claiming rewards will eventually be unable to claim — permanently freezing their unclaimed yield when the loop exceeds the Starknet transaction gas limit.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates from `entry_to_claim_from` to `pool_member_trace_length`, calling `find_sigma` (which performs up to 3 storage reads from `cumulative_rewards_trace`) on every iteration:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

The developers themselves flag this as unbounded. [1](#0-0) 

The trace grows because `set_member_balance` (called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`) inserts a new checkpoint keyed at `current_epoch + K`. The `insert` logic only deduplicates if the key is identical to the last entry; otherwise it appends a new checkpoint:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

So every balance change in a **different epoch** appends a new, permanent checkpoint. The `entry_to_claim_from` cursor is only advanced when `claim_rewards` is successfully called:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [3](#0-2) 

If a delegator accumulates N unclaimed checkpoints, the next `claim_rewards` call must iterate all N entries in a single transaction. Once N is large enough to exhaust the block gas limit, the call reverts permanently — the delegator can never advance `entry_to_claim_from` past the bloated range, and their unclaimed yield is frozen forever.

### Impact Explanation

A delegator who makes balance changes across many epochs without claiming rewards accumulates an unbounded number of checkpoints. When the checkpoint count is large enough, `claim_rewards` (and `pool_member_info_v1`) will always revert due to gas exhaustion. The delegator's accrued STRK rewards become permanently unclaimable — matching the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- Any unprivileged delegator triggers this path via `add_to_delegation_pool` or `exit_delegation_pool_intent`.
- The minimum amount per call is 1 wei (only `amount.is_non_zero()` is checked). [5](#0-4) 
- Each call in a new epoch appends one checkpoint. A delegator who is active across thousands of epochs without claiming will hit the gas ceiling.
- The developers explicitly acknowledge the loop is unbounded ("unlikely to exceed gas limits" is not a bound, it is a hope). [6](#0-5) 
- Likelihood is **medium**: requires sustained activity over many epochs without claiming, which is realistic for long-term delegators who automate deposits but not claims.

---

### Recommendation

1. **Enforce a claim before balance changes**: Require `claim_rewards` to be called (or auto-call it internally) before any balance-modifying operation, so `entry_to_claim_from` is always advanced to the current position before a new checkpoint is appended. This bounds the loop to at most K+1 entries per claim.
2. **Alternatively, cap the trace**: Reject balance changes if the number of unclaimed checkpoints (`pool_member_trace_length - entry_to_claim_from`) exceeds a safe constant (e.g., 100).
3. **Partial claim**: Allow `claim_rewards` to accept a `max_iterations` parameter so a delegator can drain the backlog over multiple transactions.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with 1 STRK in epoch E₀. Checkpoint appended at E₀+K.
2. Each subsequent epoch, delegator calls `add_to_delegation_pool(pool_member, 1)` — appending one new checkpoint per epoch. No `claim_rewards` is ever called.
3. After N epochs, `pool_member_epoch_balance` has N entries; `entry_to_claim_from = 0`.
4. Delegator calls `claim_rewards`. `calculate_rewards` enters the loop and iterates all N entries, each performing multiple storage reads via `find_sigma`. [7](#0-6) 
5. For sufficiently large N (bounded by Starknet's per-transaction gas limit), the transaction reverts.
6. Every subsequent `claim_rewards` attempt also reverts — the delegator's accumulated STRK rewards are permanently frozen.
