Looking at the Cronos codebase for an analog to the LiFi "arbitrary address approval" vulnerability class — specifically, user-controlled inputs used in token transfer/approval operations without proper authorization checks.

The most direct analog is in the bank precompile's `transfer` method.

---

### Title
Bank Precompile `transfer` Accepts Arbitrary `sender` Without Authorization — Unauthorized Token Drain from Any Holder - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile (`bank.go`) accepts a fully user-controlled `sender` address and transfers native bank tokens (`evm/{callerContract}`) from that address to a user-controlled `recipient`, with no check that the calling contract is authorized to move tokens on behalf of `sender`. Any contract can drain `evm/{self}` tokens from any holder without the holder's consent.

### Finding Description
In `bank.go`, the `TransferMethodName` case unpacks three arguments from the call input:

```go
sender    := args[0].(common.Address)   // fully user-controlled
recipient := args[1].(common.Address)   // fully user-controlled
amount    := args[2].(*big.Int)
from      := sdk.AccAddress(sender.Bytes())
to        := sdk.AccAddress(recipient.Bytes())
denom     := EVMDenom(contract.Caller()) // "evm/0x<callerContract>"
amt       := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is **no check** that `contract.Caller() == sender`, and no allowance/approval mechanism is consulted. The only constraint is that the denom is `evm/{contract.Caller()}`, meaning the calling contract can move tokens with its own denom from **any** address to **any** address.

Compare with the `mint`/`burn` cases, which also lack a caller-equals-recipient guard but at least operate on the module account. The `transfer` case uniquely allows moving tokens out of arbitrary third-party accounts. [2](#0-1) 

### Impact Explanation
A malicious contract `M` deployed at `0xM` can:

1. Call `bank.mint(victim, N)` → victim now holds `N` units of `evm/0xM`.
2. Call `bank.transfer(victim, attacker, N)` → the precompile executes `bankKeeper.SendCoins(victim, attacker, evm/0xM coins)` with no consent from `victim`.

The victim's balance is drained. Because `denom = evm/{callerContract}`, the attacker controls both the token supply (via `mint`) and the ability to move any holder's balance (via `transfer`). This is an **unauthorized transfer of precompile-controlled assets**, matching the Critical impact tier. [3](#0-2) 

### Likelihood Explanation
The entry path is fully unprivileged: any account can deploy an EVM contract on Cronos. No admin key, governance vote, or validator access is required. The attack is two transactions (mint + transfer) and is deterministic.

### Recommendation
Add an authorization check in the `TransferMethodName` case. The simplest fix is to require that `sender == contract.Caller()` (i.e., a contract may only transfer tokens from itself, not from arbitrary third parties):

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: caller is not the sender")
}
```

Alternatively, implement an ERC20-style allowance table keyed on `(owner, spender, denom)` so that a contract can only transfer from an address that has explicitly approved it.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract DrainAttack {
    IBank constant bank = IBank(address(0x64)); // bank precompile

    // Step 1: attacker mints evm/DrainAttack tokens to victim
    function seed(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: attacker drains victim's evm/DrainAttack balance to themselves
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // succeeds: denom = evm/DrainAttack, no consent from victim required
    }
}
```

After `seed(victim, 1000)`, calling `drain(victim, attacker, 1000)` transfers 1000 `evm/DrainAttack` tokens from `victim` to `attacker` with no approval from `victim`. The `bankKeeper.SendCoins` call succeeds because the only guard is that the denom matches `evm/{callerContract}`, which it does. [1](#0-0)
