### Title
WalletConnect `chia_createOfferForIds` Attacker-Controlled `driver_dict` Spoofs Royalty Amount in Confirmation Dialog - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

---

### Summary
A malicious dapp connected via WalletConnect can supply an arbitrary `royalty_percentage` inside the `driver_dict` parameter of `chia_createOfferForIds`. The GUI's confirmation dialog computes `amountWithRoyalties` — the total XCH the user will spend including creator royalties — from this attacker-controlled value whenever `nftGetInfo` fails (the common case when the NFT is not yet in the user's wallet). The user is shown a deflated total cost, approves the offer creation, and later pays the real on-chain royalty when the offer is accepted, spending more XCH than the dialog indicated.

---

### Finding Description

In `parseCommandDisplay.ts`, the `parseCommandDisplay` function handles the `chia_wallet.create_offer_for_ids` WalletConnect command by calling `createOfferRoyaltyPercentages(walletDelta, driverDict)`, which reads `royalty_percentage` directly from the dapp-supplied `driver_dict`: [1](#0-0) 

The extracted royalty is passed into `walletDeltaToDisplay`, which calls `parseWalletDeltaItem` for each NFT asset. Inside that function, the `driver_dict` royalty is set as the initial value, and then `nftGetInfo` is called as a "best-effort" override: [2](#0-1) 

The comment explicitly marks this as best-effort. When `nftGetInfo` returns `success: false` — which is the normal outcome when the NFT being requested is not in the user's own wallet — the `driver_dict` royalty is silently retained as the authoritative value.

That royalty is then fed into `formatAmountWithRoyalties`, which computes the `amountWithRoyalties` field displayed in the WalletConnect confirmation dialog: [3](#0-2) 

`parseRoyaltyPercentage` accepts any positive integer or digit-only string with no upper or lower bound: [4](#0-3) 

There is no cross-check of the `driver_dict` royalty against the actual on-chain NFT royalty before the `amountWithRoyalties` figure is shown to the user.

The `chia_createOfferForIds` command does not carry `allowConfirmationBypass: true`, so the confirmation dialog is always shown — but the amount displayed in it is derived from attacker-controlled input: [5](#0-4) 

---

### Impact Explanation

**High — WalletConnect state spoofing causes the user to approve an offer at the wrong displayed amount.**

When a user creates an offer to buy an NFT they do not yet own (the standard case for any NFT purchase offer), `nftGetInfo` returns `success: false` because the NFT is not in their wallet. The `amountWithRoyalties` shown in the confirmation dialog is therefore computed entirely from the attacker-supplied `driver_dict`. A malicious dapp can set `royalty_percentage: 1` (0.01 %) for an NFT whose real on-chain royalty is, say, 1 000 (10 %). The dialog shows the user a total cost of approximately 1.0001 XCH; the user approves; the offer is created with the real 10 % royalty enforced on-chain; when the offer is accepted the user actually spends 1.1 XCH — a 10 % overpayment relative to what the approval dialog stated.

This satisfies the allowed High impact: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."*

---

### Likelihood Explanation

The attack requires only that the user has granted a connected dapp the `chia_createOfferForIds` WalletConnect permission, which is the standard permission requested by any NFT-trading dapp. The `nftGetInfo` failure path (the trigger condition) is the **default** path for any offer to receive an NFT the user does not already own — no special network conditions are needed. The attacker controls the `driver_dict` entirely and can set any royalty value that passes `parseRoyaltyPercentage`'s minimal validation.

---

### Recommendation

Before computing `amountWithRoyalties` for the confirmation dialog, resolve the royalty percentage from the on-chain NFT record rather than from the dapp-supplied `driver_dict`. Concretely:

1. In `parseWalletDeltaItem`, if `nftGetInfo` fails or returns `success: false`, do **not** fall back to the `driver_dict` royalty for the `amountWithRoyalties` calculation. Either omit `amountWithRoyalties` entirely (show only the base amount) or display an explicit warning that the royalty could not be verified.
2. Alternatively, call `nftGetInfo` with a broader lookup (e.g., by launcher ID against the full node) so that NFTs not yet in the user's wallet can still be resolved to their real on-chain royalty.
3. Add an upper-bound sanity check in `parseRoyaltyPercentage` (e.g., reject values above 10 000, i.e., 100 %) to prevent absurdly inflated display values regardless of source.

---

### Proof of Concept

**Setup**: A malicious dapp is connected via WalletConnect and has been granted `chia_createOfferForIds`. The target NFT has a real on-chain `royalty_percentage` of `1000` (10 %).

**Malicious request**:
```json
{
  "command": "chia_createOfferForIds",
  "params": {
    "offer": {
      "1": "-1000000000000",
      "<nft_launcher_id_hex>": "1"
    },
    "driver_dict": {
      "<nft_launcher_id_hex>": {
        "type": "singleton",
        "launcher_id": "0x<nft_launcher_id_hex>",
        "also": {
          "type": "metadata",
          "also": {
            "type": "ownership",
            "transfer_program": {
              "type": "royalty transfer program",
              "royalty_percentage": "1"
            }
          }
        }
      }
    }
  }
}
```

**Execution trace**:

1. `createOfferRoyaltyPercentages` reads `royalty_percentage: 1` from `driver_dict`. [1](#0-0) 

2. `parseWalletDeltaItem` initialises `result.royaltyPercentage = 1`. [6](#0-5) 

3. `nftGetInfo` returns `{ success: false }` — NFT not in user's wallet. The catch/guard block leaves `result.royaltyPercentage = 1` unchanged. [7](#0-6) 

4. `formatAmountWithRoyalties` computes `royaltyAmount = (1_000_000_000_000n × 1n) / 10_000n = 100_000_000n` mojos → `amountWithRoyalties = "1.0001 XCH"`. [8](#0-7) 

5. The confirmation dialog presents **"Total with royalties: 1.0001 XCH"**. The user approves.

6. The offer is created on-chain with the real 10 % royalty. When accepted, the user's wallet is debited **1.1 XCH** — 0.0999 XCH more than the dialog stated.

```typescript
function parseRoyaltyPercentage(value: unknown): number | undefined {
  if (typeof value === 'number' && Number.isInteger(value) && value > 0) {
    return value;
  }

  if (typeof value === 'string' && /^\d+$/.test(value)) {
    return Number(value);
  }

  return undefined;
}
```

```typescript
function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  for (const assetId of Object.keys(walletDelta.spending)) {
    royaltyPercentages.spending[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  for (const assetId of Object.keys(walletDelta.receiving)) {
    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  return royaltyPercentages;
}
```

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
