### Title
Single Malicious Participant Permanently Aborts DKG/Reshare/Refresh via Echo Broadcast Early-Abort Equivocation - (File: src/protocol/echo_broadcast.rs)

---

### Summary
A single malicious participant (within the documented `n/3` Byzantine tolerance) can permanently abort DKG, reshare, and refresh for all honest parties by sending conflicting `Send` messages during the echo broadcast phase. The early-abort optimization in `reliable_broadcast_receive_all` fires when it determines no value can accumulate enough echo votes, causing every honest party to return a fatal `ProtocolError::AssertionFailed`. This is confirmed by the library's own test suite.

---

### Finding Description

The `do_keyshare` function in `src/dkg.rs` — which is the shared core of DKG, reshare, and refresh — invokes `do_broadcast` (which calls `reliable_broadcast_receive_all`) at three critical points:

- **Line 362**: broadcasting session IDs
- **Line 435**: broadcasting commitments and proofs of knowledge
- **Line 531**: the final `broadcast_success` round

Inside `reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs`, there is an early-abort optimization at lines 241–263. After each echo message is processed, if the function determines that no single value can ever accumulate enough echo votes to exceed `echo_t` (even counting all remaining undelivered echoes), it immediately returns:

```rust
return Err(ProtocolError::AssertionFailed(format!(
    "The original sender in session {sid:?} is malicious! ..."
)));
```

The threshold `echo_t` is computed by `echo_ready_thresholds` at lines 67–78:

```rust
let broadcast_threshold = (n - 1) / 3;
let echo_threshold = usize::midpoint(n, broadcast_threshold);
```

For `n = 4` (3 honest + 1 malicious, which is exactly the documented tolerance): `broadcast_threshold = 1`, `echo_threshold = 2`. The echo phase requires **strictly more than 2** votes for a single value.

**Attack**: The malicious participant sends `Send(true)` to 2 honest parties and `Send(false)` to 1 honest party (and echoes `false` itself). Honest parties echo what they received. After all 4 echo votes are delivered, the tally for every honest party is `{true: 2, false: 2}`. Since `2 + 0 = 2` is not `> 2` for either value, `is_enough = false`, and every honest party aborts with `AssertionFailed`. The protocol is permanently terminated — there is no recovery path.

---

### Impact Explanation

**High: Permanent denial of key generation, reshare, and refresh for honest parties under valid protocol inputs and documented trust assumptions.**

Once `reliable_broadcast_receive_all` returns `Err`, the error propagates through `do_broadcast` → `do_keyshare` → the public `keygen`/`reshare`/`refresh` API. There is no retry mechanism, no fallback, and no way for honest parties to complete the protocol. The key material is never produced (or the reshared key is never updated), permanently blocking all downstream signing, CKD, and key management operations.

---

### Likelihood Explanation

**High.** Any participant in a DKG/reshare/refresh session can execute this attack. The attacker only needs to:
1. Be a registered participant (no special privilege required).
2. Send different `Send` messages to different subsets of honest parties during the echo broadcast phase — a trivial equivocation.

The attack works whenever `n ≡ 0 (mod 4)` or more generally whenever the malicious participant can split echo votes so that no value exceeds `echo_t`. With `n = 4` (the minimum group size where `broadcast_threshold > 0`), a single malicious party achieves a perfect 2-2 split. The library's own test `test_three_honest_one_dihonest` (version 1) at lines 547–563 of `src/protocol/echo_broadcast.rs` explicitly asserts this failure as expected behavior, confirming the attack is reproducible.

---

### Recommendation

The early-abort optimization at lines 241–263 of `src/protocol/echo_broadcast.rs` is the root cause. It conflates "we cannot reach the threshold" with "the sender is malicious and we must abort." The correct behavior per the BRB protocol is to **wait** (not abort) when the threshold cannot be reached — the protocol's liveness guarantee under `f < n/3` faults means honest parties should eventually deliver a value or wait indefinitely, not hard-abort.

Remove or gate the early-abort: only abort if the number of already-received conflicting echoes **provably exceeds** the Byzantine bound (i.e., more than `f` distinct values each with at least one vote), not merely because the remaining undelivered votes are insufficient to tip any single value over the threshold. Alternatively, replace the abort with a `Wait` / `continue` so that honest parties remain in the loop until the protocol either succeeds or times out externally (consistent with the documented caller-managed timeout model in `README.md` lines 165–172).

### Proof of Concept

The library's own test suite demonstrates the attack directly. In `src/protocol/echo_broadcast.rs`, `test_three_honest_one_dihonest` (lines 547–563):

```
n = 4 (3 honest + 1 malicious, within documented n/3 = 1 tolerance)

Malicious sends:
  Send(true)  → honest party 0
  Send(true)  → honest party 1
  Send(false) → honest party 2
  (echoes false itself)

After all echoes delivered, every honest party sees:
  data_echo = {true: 2, false: 2}
  non_received_echo_cnt = 0

Early-abort check:
  true:  2 + 0 = 2, NOT > echo_t(2) → not enough
  false: 2 + 0 = 2, NOT > echo_t(2) → not enough
  → is_enough = false → Err(AssertionFailed(...))

Test assertion:
  assert_eq!(result, Err(ProtocolError::AssertionFailed(
      "The original sender in session 3 is malicious! ...")));
```

This abort propagates through `do_broadcast` → `do_keyshare` (called at `src/dkg.rs` line 362, 435, or 531) → public `keygen`/`reshare`/`refresh` API, permanently denying key generation and all dependent operations to all honest parties.
