### Title
Signature Bypass via `skip_stateful_validations` Allows Unauthorized Invoke Transactions for Undeployed Accounts — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function skips the `__validate__` entry point (the account's signature-verification step) for any Invoke transaction with `nonce == 1` submitted against an account whose on-chain nonce is still `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type in the pool — not exclusively a `deploy_account` — an unprivileged attacker who observes a victim's pending `deploy_account + invoke` pair can submit a second Invoke at `nonce = 1` carrying an **arbitrary (invalid) signature** and have it admitted to the mempool without signature verification. This is a direct analog to the external "Unprivileged Unstaking" report: a cost-free, repeated operation (here: signature-free transaction injection) that disrupts a protocol invariant (here: every admitted transaction must have a verified signer).

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have passed its account's `__validate__` entry point, which is the sole on-chain mechanism that verifies the sender's signature.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

**`account_tx_in_pool_or_recent_block` checks for *any* transaction, not a `deploy_account`:** [2](#0-1) 

It returns `true` if `self.state.contains_account(address) || self.tx_pool.contains_account(address)` — i.e., any transaction type for that address is sufficient.

**Effect when `skip_validate = true`:**

`run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the Invoke branch calls `perform_pre_validation_stage` (nonce + fee-bounds + balance checks) and then **returns early** before the `__validate__` call: [4](#0-3) 

`perform_pre_validation_stage` only checks nonce, fee bounds, and balance — it does **not** verify the signature: [5](#0-4) 

**Attack chain:**

1. Alice submits `deploy_account` (nonce 0) → enters mempool. `account_tx_in_pool_or_recent_block(Alice)` now returns `true`.
2. Alice submits `invoke` (nonce 1) → `skip_stateful_validations` fires, `__validate__` is skipped, tx enters mempool.
3. Bob (attacker) submits a second `invoke` (nonce 1, **invalid/arbitrary signature**, higher fee) for Alice's address. Conditions: nonce == 1 ✓, on-chain nonce == 0 ✓, `account_tx_in_pool_or_recent_block` == `true` ✓ → `skip_stateful_validations` returns `true` again.
4. The mempool's `validate_fee_escalation` accepts Bob's tx because it carries a higher fee, replacing Alice's invoke.
5. At execution time: `deploy_account` succeeds; Bob's invoke runs `__validate__` (now mandatory in the batcher), fails with an invalid signature, and is reverted. Alice's invoke is permanently gone from the mempool.

The comment in the code itself acknowledges the assumption that is violated: [6](#0-5) 

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is circular: Alice's own invoke at nonce 1 was *also* admitted via `skip_stateful_validations` (without `__validate__`), so its presence in the pool does **not** prove a valid signature was ever checked. Bob's tx satisfies the same condition and is admitted on the same basis.

---

### Impact Explanation

**Impact category:** High — Mempool/gateway admission accepts an invalid (signature-unverified) transaction and rejects a valid one before sequencing.

Concretely:
- An attacker can inject signature-free invoke transactions for any undeployed account that has a pending `deploy_account` in the mempool.
- Via fee escalation, the attacker displaces the legitimate user's invoke transaction.
- The attacker's transaction fails at execution (invalid signature), but the victim's transaction is permanently evicted from the mempool, breaking the `deploy_account + invoke` UX flow.
- The attack can be scripted to target every new `deploy_account` transaction observed in the mempool, analogous to the external report's "listen for stake → immediately unstake" loop.

---

### Likelihood Explanation

- The mempool is observable (P2P propagation, RPC `get_mempool_snapshot`).
- The attacker only needs to submit one transaction per victim with a fee slightly above the victim's invoke fee — a low, bounded cost.
- No privileged access, special contract, or pre-existing relationship with the victim is required.
- The `deploy_account + invoke` pattern is the standard new-account onboarding flow, making targets abundant.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects transaction type, not merely account presence. Alternatively, the gateway should verify the signature even when `skip_validate = true` by running `__validate__` against a speculative state that includes the pending `deploy_account` execution result.

---

### Proof of Concept

```
// Precondition: Alice's account at address X is undeployed (on-chain nonce = 0).

// Step 1 – Alice submits the standard deploy+invoke pair:
gateway.add_tx(deploy_account { sender: X, nonce: 0, sig: alice_sig_0 })
gateway.add_tx(invoke       { sender: X, nonce: 1, sig: alice_sig_1 })
// Both admitted; skip_stateful_validations fires for the invoke.

// Step 2 – Attacker observes Alice's deploy_account in the mempool.
// account_tx_in_pool_or_recent_block(X) == true.

// Step 3 – Attacker submits a replacement invoke with a higher fee and a bogus signature:
gateway.add_tx(invoke { sender: X, nonce: 1, fee: alice_fee + 1, sig: [0xDEAD, 0xBEEF] })
// skip_stateful_validations returns true (same three conditions hold).
// __validate__ is NOT called. Tx admitted; Alice's invoke is evicted via fee escalation.

// Step 4 – Batcher executes the block:
//   deploy_account(X) succeeds → X is now deployed.
//   attacker's invoke runs __validate__ → FAILS (invalid signature) → reverted.
//   Alice's invoke is gone. Alice must resubmit manually.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        state: &mut S,
