### Title
Greedy Regex in `parseOfferData` Always Extracts the Last `offer1` Blob, Enabling Wrong-Offer Acceptance — (`packages/gui/src/components/offers2/OfferBuilderImport.tsx`)

### Summary

The `parseOfferData` function uses a regex with a greedy `(?<leading>.*)` group that, by JavaScript regex backtracking semantics, always resolves to the **last** `offer1`-prefixed bech32 blob in the input. An attacker who controls the file content can prepend a decoy legitimate offer and append a malicious offer; the GUI will extract, summarize, and submit only the malicious one.

### Finding Description

In `OfferBuilderImport.tsx`, `parseOfferData` applies:

```js
data.match(/(?<leading>.*)(?<offer>offer1[qpzry9x8gf2tvdw0s3jn54khce6mua7l]+)(?<trailing>.*)/s)
``` [1](#0-0) 

With the `s` (dotAll) flag active, `(?<leading>.*)` greedily consumes the entire string, then backtracks only far enough to satisfy the `offer1[...]` group. Because the backtracking stops at the **rightmost** valid `offer1` match, any content before the last blob — including a complete, valid, decoy offer — is absorbed into `(?<leading>.*)` and silently discarded.

The extracted blob is then passed without any further validation directly to `getOfferSummary` and navigated to the viewer:

```ts
const [offerData] = parseOfferData(rawOfferData);
const { summary } = await getOfferSummary({ offerData }).unwrap();
navigate('/dashboard/offers/view', { state: { offerData, offerSummary: summary, ... } });
``` [2](#0-1) 

In `OfferBuilderViewer`, `handleSubmit` passes that same `offerData` directly to `acceptOffer`, which calls `takeOffer({ offer: offerData, ... })`: [3](#0-2) [4](#0-3) 

The same bug exists identically in the legacy `OfferImport.tsx`: [5](#0-4) 

### Impact Explanation

A user who opens a crafted file containing `offer1<legit_blob> offer1<malicious_blob>` will be shown the **malicious** offer's summary (assets, amounts, counterparty) and, upon clicking "Accept Offer," will execute `takeOffer` against the malicious blob. The legitimate blob is never seen or processed. This directly causes wrong-asset or wrong-amount execution — a concrete wallet security impact matching the High category: *"causes a user to approve, import, sign, send, revoke, burn, join, or display the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation

The attacker only needs to deliver a single crafted `.offer` file to the victim (e.g., via a marketplace, forum post, or direct message). The user performs a completely normal action — dropping or pasting the file into the import UI. No special permissions, local access, or key material are required. The malicious blob can be crafted to request a large XCH or CAT payment while the decoy blob shows a benign trade.

### Recommendation

Replace the greedy `(?<leading>.*)` with a non-greedy `(?<leading>.*?)` to match the **first** occurrence, or use `String.prototype.match` with the global flag `/g` and explicitly select the first capture group. Additionally, if multiple `offer1` blobs are found, the UI should warn the user or reject the file outright.

### Proof of Concept

1. Construct a file:
   ```
   offer1<valid_bech32_decoy_blob> offer1<valid_bech32_malicious_blob>
   ```
2. Open the Chia GUI, navigate to the Offers import page (`OfferBuilderImport`).
3. Drop the file onto the dropzone.
4. Observe that `getOfferSummary` is called with `offer1<malicious_blob>`, and the viewer displays the malicious offer's terms.
5. Click "Accept Offer" — `takeOffer` is invoked with `offer1<malicious_blob>`, not the decoy.
