### Title
Native Token Value Permanently Locked When Unauthorized Caller Sends Value to `l1_messenger_hook` - (`system_hooks/src/call_hooks/l1_messenger.rs`)

---

### Summary

The `l1_messenger_hook` at address `0x7001` silently returns **success** (pretending to be an empty account) when called by any address other than `L1_MESSENGER_ADDRESS`. Because the native token transfer from caller to `0x7001` is committed **before** the hook executes, and the hook returns a success result without checking `nominal_token_value`, any native tokens sent with such a call are permanently locked at `0x7001` with no recovery path.

---

### Finding Description

In `system_hooks/src/call_hooks/l1_messenger.rs`, the hook function first checks the caller and, if unauthorized, immediately returns a **success** result:

```rust
if caller != L1_MESSENGER_ADDRESS {
    // Pretend to be an empty account
    return Ok((
        make_return_state_from_returndata_region(available_resources, &[]),
        return_memory,
    ));
}

// This hook doesn't accept any native token value
error |= nominal_token_value != U256::ZERO;
``` [1](#0-0) 

The `nominal_token_value != U256::ZERO` guard is only reached for **authorized** callers. For unauthorized callers, the function returns `make_return_state_from_returndata_region(available_resources, &[])`, which produces a `CallResult::Successful` result: [2](#0-1) 

In the bootloader runner, the native token transfer from caller to callee is performed **before** the hook is invoked, inside the same rollback frame:

```rust
let rollback_handle = self.system.start_global_frame()?;

// Try to execute transfer if requested
if !self.perform_transfer_if_required(...) { ... }

// ...
if is_call_to_special_address {
    self.call_to_special_address_execute_callee_frame(
        external_call_launch_params,
        caller_ee_type,
        rollback_handle,
    )
}
``` [3](#0-2) 

Inside `call_to_special_address_execute_callee_frame`, the frame is committed (`finish_global_frame(None)`) when the hook result is **not** a revert: [4](#0-3) 

Because the hook returns `CallResult::Successful` for unauthorized callers, the rollback frame is committed, permanently transferring the value to `0x7001`. The hook address has no deployed bytecode and no withdrawal function; any subsequent call to `0x7001` by an unauthorized caller will again return success with empty data, making the locked balance irrecoverable.

---

### Impact Explanation

Any native tokens sent to `0x7001` (`L1_MESSENGER_ADDRESS_HOOK`) by a caller other than `L1_MESSENGER_ADDRESS` (`0x8008`) are permanently locked. The transaction succeeds with no revert, giving the user no indication of loss. Since `0x7001` is a pure hook address with no code and no withdrawal mechanism, the balance cannot be recovered by any on-chain action.

---

### Likelihood Explanation

Any unprivileged EVM transaction can call `0x7001` with a non-zero `value`. A user who mistakenly sends native tokens to `0x7001` (e.g., confusing the hook address with the L1 messenger system contract at `0x8008`) will silently lose funds. The transaction succeeds, providing no error signal. This is a realistic user mistake given the proximity of `0x7001` and `0x8008` in the system address space.

---

### Recommendation

Before the early-return "pretend to be empty account" path, check that `nominal_token_value` is zero and revert if it is not:

```rust
if caller != L1_MESSENGER_ADDRESS {
    if nominal_token_value != U256::ZERO {
        return Ok((make_error_return_state(available_resources), return_memory));
    }
    // Pretend to be an empty account
    return Ok((
        make_return_state_from_returndata_region(available_resources, &[]),
        return_memory,
    ));
}
```

This mirrors the existing guard already applied for authorized callers and ensures no value is silently accepted and locked.

---

### Proof of Concept

1. Deploy a contract (or use an EOA) that is **not** `L1_MESSENGER_ADDRESS` (`0x8008`).
2. Send a transaction calling `0x7001` (`L1_MESSENGER_ADDRESS_HOOK`) with `value = 1 ETH` and any calldata.
3. Observe: the transaction succeeds (`CallResult::Successful`), the caller's balance decreases by 1 ETH, and `0x7001`'s balance increases by 1 ETH.
4. Attempt to recover the ETH by calling `0x7001` again — the hook returns success with empty data; no withdrawal is possible.
5. The 1 ETH is permanently locked at `0x7001`. [1](#0-0) [5](#0-4)

fn make_return_state_from_returndata_region<S: SystemTypes>(
    remaining_resources: S::Resources,
    returndata: &[u8],

    let return_values = ReturnValues {
        returndata,
        return_scratch_space: None,
    };

        resources_returned: remaining_resources,
        result: CallResult::Successful { return_values },
