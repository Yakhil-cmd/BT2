### Title
Malicious CKD Participant Can Corrupt the Derived Key by Sending Unverified Share Contributions — (`src/confidential_key_derivation/protocol.rs`)

### Summary
In the Confidential Key Derivation (CKD) protocol, the coordinator (`do_ckd_coordinator`) blindly aggregates `(norm_big_y, norm_big_c)` tuples received from every participant without any proof of correctness. A single malicious participant can send arbitrary group elements in place of their honest share, causing the coordinator to output a corrupted CKD result. The application (e.g., a TEE) will then unmask a wrong derived key, silently and undetectably.

### Finding Description

The CKD protocol is designed so that the coordinator reconstructs `msk · H(pk, app_id)` by summing each participant's Lagrange-weighted contribution:

- Each honest participant `i` computes:
  - `Y_i = y_i · G` (fresh randomness)
  - `C_i = x_i · H(pk, app_id) + y_i · app_pk` (ElGamal encryption of their share)
  - Sends `(λ_i · Y_i, λ_i · C_i)` to the coordinator.

The coordinator then sums all received tuples:

```
norm_big_y = Σ λ_i · Y_i
norm_big_c = Σ λ_i · C_i
```

The application unmasks via `C - app_sk · Y = msk · H(pk, app_id)`.

**Root cause:** In `do_ckd_coordinator`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` and adds them directly to the running sum with no proof of correctness, no commitment check, and no consistency verification against the participant's known public key share:

```rust
// src/confidential_key_derivation/protocol.rs, lines 50–55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

A malicious participant's `do_ckd_participant` sends its share via:

```rust
// src/confidential_key_derivation/protocol.rs, lines 27–32
let (norm_big_y, norm_big_c) =
    compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
let waitpoint = chan.next_waitpoint();
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [2](#0-1) 

Nothing prevents a malicious participant from replacing `(norm_big_y, norm_big_c)` with arbitrary group elements before sending. The coordinator has no mechanism to detect this substitution.

The coordinator knows each participant's public key share (from `KeygenOutput`) and knows `app_pk`. A correct contribution satisfies `C_i - y_i · app_pk = x_i · H(pk, app_id)`, but since `y_i` is ephemeral and private, the coordinator cannot verify this relation without a zero-knowledge proof (e.g., a proof of correct ElGamal encryption). No such proof is sent or verified.

This is the direct analog of `amountOutMin = 0`: just as the Uniswap swap accepts any output amount including zero, the CKD coordinator accepts any participant contribution including a completely fabricated one.

### Impact Explanation

**High — Corruption of CKD output.**

A single malicious participant (out of any `n`) can cause the coordinator to output a `CKDOutput` whose `unmask(app_sk)` yields an attacker-chosen or random group element instead of `msk · H(pk, app_id)`. The TEE or application receiving this output will silently use a wrong derived key. There is no integrity check on the final `CKDOutput` that would allow honest parties to detect the corruption. This matches the allowed impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

### Likelihood Explanation

Any participant in the CKD protocol is a potential attacker — no privileged access is required. The participant role is reachable by any node that holds a key share. The attack requires only that the malicious participant substitute its honest `(norm_big_y, norm_big_c)` with crafted values before calling `chan.send_private`. This is a single-step deviation from the honest protocol and requires no coordination with other parties.

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct ElGamal encryption — specifically, a proof that `C_i` was formed as `x_i · H(pk, app_id) + y_i · app_pk` for the same `y_i` used in `Y_i = y_i · G`, and that `x_i` is consistent with the participant's committed public key share `X_i = x_i · G2`. The coordinator must verify this proof before including the contribution in the sum. A standard Chaum-Pedersen / sigma protocol suffices for this relation.

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` run `do_ckd_participant` honestly.
2. Malicious participant `P_n` overrides its `do_ckd_participant` to send `(G, G)` (the generator point) instead of its honest `(norm_big_y, norm_big_c)`.
3. The coordinator sums all contributions including `P_n`'s fabricated `(G, G)`.
4. The resulting `CKDOutput` satisfies:
   - `norm_big_y = (honest sum) + G`
   - `norm_big_c = (honest sum) + G`
5. `ckd_output.unmask(app_sk)` computes `norm_big_c - app_sk · norm_big_y`, which equals `msk · H(pk, app_id) + G - app_sk · G` — a value that differs from the correct key by `(1 - app_sk) · G`.
6. The TEE receives and uses this wrong key with no error or warning.
