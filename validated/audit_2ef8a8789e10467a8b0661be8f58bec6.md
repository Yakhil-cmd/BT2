### Title
Pool Admin Bypasses Fee Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` allows a pool admin to set per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) to any value up to `uint16.max` (65535 = 6.5535%) with no cap check, while the parallel `setPoolAdminFees()` path enforces `maxAdminSpreadFeeE6`. This is a direct admin-boundary break: the pool admin exceeds the cap system the protocol intends to enforce.

### Finding Description
The factory defines `maxAdminSpreadFeeE6` (hard-capped at `HARD_MAX_SPREAD_FEE_E6 = 200_000`) and enforces it in `setPoolAdminFees`:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

But `setPoolBinAdditionalFees` contains no analogous check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` only validates the bin index, then stores the raw values. During every swap, the per-bin fee is added directly to `baseFeeX64`:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

A pool admin can call `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` and immediately impose a 6.5535% additional fee on top of the global spread fee for all swaps touching bin 0, with no timelock, no cap, and no protocol override path.

### Impact Explanation
Traders executing swaps in the affected bin pay a higher effective fee than any documented cap permits. The excess fee is distributed to LPs (and partially to protocol via `spreadFeeE6`), but the trader's output is reduced by up to 6.5535% beyond what the cap system allows. This is a direct, quantifiable loss of swap output principal for every trader touching the bin.

### Likelihood Explanation
The pool admin is a semi-trusted role that is explicitly supposed to be bounded by caps. The path requires only that the pool admin call one factory function — no special conditions, no timelock, no oracle state. Any pool admin (including one who turns adversarial after deployment) can execute this immediately.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Introduce a `maxAdminBinAdditionalFeeE6` factory state variable (owner-settable, hard-capped) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
if (addFeeBuyE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
```
