### Title
Beaver Triple Reused in OT-Based ECDSA Presigning Without Participant-Set Validation — (`File: src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA presigning function explicitly omits checking that the participant set used to generate a Beaver triple matches the participant set used for presigning. The `TriplePub` struct records the generation participant set, but `presign()` only validates the threshold, not the participants. After a key reshare or participant-set rotation, stale triples generated under the old participant set can be fed into presigning under the new participant set, causing the protocol to fail for honest parties.

---

### Finding Description

`TriplePub` stores both the participant set and threshold used during triple generation: [1](#0-0) 

The `presign()` initializer validates only that the threshold matches, and contains an explicit comment acknowledging the participant-set check is omitted: [2](#0-1) 

The rationale given — *"presumably they need to have been present in order to have shares"* — is an assumption, not an enforced invariant. The `PresignArguments` struct accepts any `(TripleShare, TriplePub)` pair alongside an independent `participants` slice: [3](#0-2) 

The protocol specification explicitly requires that the triples be generated by the same participant set as the presigning parties: [4](#0-3) 

When the presign participant set `P_1` contains a participant `F` who was not in the triple generation set `P_0`, `F` holds no valid Shamir share of `(a, b, c)` or `(k, d, e)`. The Lagrange linearization step in `do_presign` applies coefficients derived from `P_1` to shares that were evaluated at participant IDs from `P_0`: [5](#0-4) 

This produces incorrect reconstructed values for `e`, `alpha`, and `beta`. The in-protocol consistency checks (`E = e·G`, `α·G = K+A`, `β·G = X+B`) will then fail: [6](#0-5) 

The protocol aborts with an `AssertionFailed` error, and no presignature is produced.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

After a key reshare or participant-set rotation (which the library explicitly supports via `src/dkg.rs`), the new participant set `P_new` may include participants who were not present during earlier triple generation sessions. If an application reuses pooled triples generated under the old participant set `P_old` for a presigning session under `P_new`, every participant in `P_new \ P_old` will lack a valid triple share. The consistency checks inside `do_presign` will abort the protocol. Because the library provides no mechanism to detect this mismatch at initialization time, honest parties cannot distinguish a legitimate protocol failure from a configuration error, and signing is permanently blocked until new triples are generated — which itself requires a full multi-round OT-based triple generation protocol.

---

### Likelihood Explanation

Any deployment that:
1. Pre-generates and pools Beaver triples (the documented and benchmarked usage pattern), and
2. Subsequently performs a key reshare or participant-set update,

will silently hold stale triples whose `TriplePub.participants` no longer matches the active signing set. The library provides no warning, no validation at presign initialization, and no mechanism to detect the staleness. A malicious coordinator can also deliberately supply triples generated under a different participant set to cause targeted denial of signing for specific honest parties.

---

### Recommendation

In `presign()` in `src/ecdsa/ot_based_ecdsa/presign.rs`, after constructing the `ParticipantList`, add an explicit check that the presign participant set is a subset of `triple0.1.participants` and `triple1.1.participants`:

```rust
// Enforce that presign participants are a subset of the triple generation participants
let triple0_participants: std::collections::HashSet<_> =
    args.triple0.1.participants.iter().collect();
let triple1_participants: std::collections::HashSet<_> =
    args.triple1.1.participants.iter().collect();
for p in participants.iter() {
    if !triple0_participants.contains(&p) || !triple1_participants.contains(&p) {
        return Err(InitializationError::BadParameters(
            "presign participant was not present during triple generation".to_string(),
        ));
    }
}
```

Remove the comment at lines 38–40 that documents the omission, replacing it with this enforced check.

### Proof of Concept

1. Run triple generation with `P_old = {A, B, C, D, E}`, threshold `t = 2`. Each party receives a `(TripleShare, TriplePub)` where `TriplePub.participants = [A, B, C, D, E]`.
2. Perform a key reshare, producing a new participant set `P_new = {A, B, C, D, F}` where `F ∉ P_old`.
3. Application reuses the pooled triples (generated under `P_old`) and calls `presign(&[A, B, C, D, F], F, PresignArguments { triple0: (stale_share, stale_pub), ... })`.
4. `presign()` passes initialization: threshold matches, `F` is in the presign list, no participant-set check is performed.
5. Inside `do_presign`, `F` applies Lagrange coefficients for `P_new` to a share that was evaluated at `F`'s ID under `P_old`'s polynomial — producing an incorrect `e_i`.
6. The aggregate `e` fails the check `E = e·G` at line 127, returning `AssertionFailed("received incorrect shares of kd")`.
7. All honest parties in `P_new` abort. Signing is denied. [2](#0-1) [1](#0-0) [7](#0-6)
