### Title
Domain Allowlist Bypass via Missing Dot-Prefix in `fetchOffer.ts` — (`packages/gui/src/util/fetchOffer.ts`)

### Summary
`fetchOffer.ts` uses `domain.endsWith(localDomain)` without a dot separator, allowing any domain that is a suffix of an allowlisted domain (e.g., `evilspacescan.io` matching `spacescan.io`) to pass the check. `OffersProvider.prepareOfferData` uses the correct `domain === localDomain || domain.endsWith('.'+localDomain)` guard.

### Finding Description
In `fetchOffer.ts` line 21, the domain check is:
```ts
domains.find((localDomain) => domain.endsWith(localDomain))
```
For `offerUrl = 'https://evilspacescan.io/offer'`, `domain = 'evilspacescan.io'`, and `'evilspacescan.io'.endsWith('spacescan.io')` evaluates to `true`, so the service is found and execution continues to fetch content from the attacker-controlled domain.

### Impact Explanation
If `fetchOffer.ts` is called in a production code path with an attacker-influenced URL, the attacker can serve crafted offer data (e.g., a swap with unfavorable amounts or a malicious asset) that gets parsed, summarized via `getOfferSummary`, and presented to the user as a legitimate offer from a trusted service. If the user accepts, they suffer direct asset loss.

### Likelihood Explanation
**Conditional on reachability.** The bug is unambiguous and locally testable. Reachability depends on whether `fetchOffer.ts` is invoked with attacker-controlled input in any production flow (WalletConnect, QR, deep link, notification). This could not be confirmed without tracing all callers.

### Proof of Concept
```ts
// In fetchOffer.ts, line 21:
const domain = new URL('https://evilspacescan.io/offer').hostname; // 'evilspacescan.io'
const match = 'spacescan.io';
console.log(domain.endsWith(match)); // true — bypass succeeds

// Correct check (OffersProvider.tsx):
console.log(domain === match || domain.endsWith('.' + match)); // false — correctly rejected
```

### Recommendation
Fix `fetchOffer.ts` line 21 to mirror `OffersProvider.prepareOfferData`:
```ts
domains.find((localDomain) => domain === localDomain || domain.endsWith(`.${localDomain}`))
```

---

**Verdict:** The bug is **real and demonstrable**, but full confirmation as a bounty-grade High finding requires verifying that `fetchOffer.ts` is called with attacker-influenced input in a production code path. If callers exist (which is likely for a utility in `src/util/`), this is a valid **High** finding under "Unsafe handling of external URLs... that produces direct asset loss or approval hijack."
