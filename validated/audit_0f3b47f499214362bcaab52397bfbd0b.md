### Title
WalletConnect `chia_pushTransactions`/`chia_pushTx` Confirmation Dialog Presents Opaque JSON Spend Bundle, Enabling Blind Signing of Arbitrary Asset Transfers — (File: `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

The WalletConnect commands `chia_pushTransactions` (dapp alias `chia_pushTransactions`) and `chia_pushTx` (dapp alias `chia_pushTx`) accept fully arbitrary, opaque transaction/spend-bundle JSON blobs from a connected dapp. The confirmation dialog shown to the user renders these blobs as raw JSON with no decoding of recipient addresses, coin amounts, or asset types. A malicious dapp granted either permission can craft a spend bundle that transfers the user's XCH, CAT, NFT, or DID to an attacker-controlled address, and the user will be presented with an unreadable JSON blob and asked to click "Push" — making meaningful review impossible.

---

### Finding Description

**Root cause — `Commands.ts` schema for `chia_wallet.push_transactions` and `chia_full_node.push_tx`:**

The `transactions` and `spend_bundle` parameters are declared with `type: 'json'` and carry no `humanize` field:

```ts
// packages/gui/src/electron/commands/Commands.ts L648-692
'chia_wallet.push_transactions': {
  params: [
    { name: 'transactions', label: ..., type: 'json' },   // ← opaque blob, no humanize
    { name: 'fee', ..., type: 'bigint', humanize: 'mojo-to-xch', isOptional: true },
    ...
  ],
  dapp: [{ command: 'chia_pushTransactions', ... }],
},

// L694-706
'chia_full_node.push_tx': {
  params: [{ name: 'spend_bundle', label: ..., type: 'json' }],  // ← opaque blob
  dapp: [{ command: 'chia_pushTx', ... }],
},
``` [1](#0-0) [2](#0-1) 

**Confirmation pipeline — `main.tsx`:**

When a dapp calls either command, `dispatchPairRequest` calls `humanizeDappCommand`, which calls `humanizeParams` over the schema. Because `transactions`/`spend_bundle` are `type: 'json'` with no `humanize` key, they are rendered verbatim as a raw JSON string in the `rows` array passed to the `Confirm` dialog. The user sees the raw wire-format blob — puzzle hashes, aggregated signatures, coin IDs — not decoded addresses or amounts. [3](#0-2) [4](#0-3) 

**`parseDappParams` does not inspect JSON blob contents:**

`parseDappParams` validates top-level parameter names and scalar types, but for `type: 'json'` parameters it passes the value through unchanged. No validation of the inner structure (recipient address, amount, asset type) is performed. [5](#0-4) 

**`chia_pushTransactions` is a spend command requiring confirmation, but confirmation is blind:**

`chia_wallet.push_transactions` is listed in `SPEND_COMMANDS`, so it always requires user confirmation and cannot be bypassed by default. However, the confirmation is meaningless because the user cannot decode the raw JSON. [6](#0-5) 

---

### Impact Explanation

A malicious dapp granted `chia_pushTransactions` or `chia_pushTx` can craft a spend bundle that:
- Sends the user's XCH to an attacker address
- Transfers the user's NFT or DID to the attacker
- Burns or redirects CAT tokens

The user is shown a confirmation dialog containing only the raw JSON blob (puzzle hashes, coin IDs, aggregated signatures). No human-readable recipient address, amount, or asset type is displayed. The user cannot make an informed decision and may click "Push," causing irreversible asset loss.

This matches the **High** allowed impact: *"Corruption, spoofing, or unsafe trust of... WalletConnect state that causes a user to approve... the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

- The attacker must be a dapp that the user has connected to via WalletConnect and explicitly granted `chia_pushTransactions` or `chia_pushTx`.
- This is a realistic scenario: legitimate dapps (DEXes, NFT marketplaces, DeFi protocols) routinely request `chia_pushTransactions` to submit pre-built transactions.
- A malicious dapp posing as a legitimate service can request this permission and then submit a crafted spend bundle.
- The user has no way to distinguish a legitimate transaction from a malicious one in the confirmation dialog.

---

### Recommendation

1. **Decode the `transactions`/`spend_bundle` JSON** in the confirmation dialog to extract and display: recipient puzzle hash (decoded to bech32 address), coin amounts (in XCH/CAT mojos humanized), asset type, and number of coins being spent.
2. Add a dedicated `humanize` handler for `type: 'json'` spend-bundle parameters that parses the Chia transaction format and renders a human-readable summary.
3. Consider adding a schema-level `parseDisplay` hook for `chia_pushTransactions` (analogous to the existing `parseCommandDisplay` used for offers and CAT display) that extracts and shows the net asset flow.

---

### Proof of Concept

**Setup:** Alice connects her Chia wallet to a malicious dapp via WalletConnect and grants `chia_pushTransactions`.

**Attack:**
```typescript
// Malicious dapp sends via WalletConnect session_request:
{
  method: 'chia_pushTransactions',
  params: {
    fingerprint: aliceFingerprint,
    transactions: [
      {
        // Pre-built spend bundle that sends Alice's 1 XCH to attacker
        spend_bundle: {
          coin_spends: [{
            coin: { parent_coin_info: "0x...", puzzle_hash: "0x...", amount: 1000000000000 },
            puzzle_reveal: "0xff...",  // Alice's puzzle
            solution: "0xff..."        // solution sending to attacker's address
          }],
          aggregated_signature: "0xc0..."
        }
      }
    ]
  }
}
```

**What Alice sees in the confirmation dialog:**
- Title: "Push Transactions"
- Transactions: `[{"spend_bundle":{"coin_spends":[{"coin":{"parent_coin_info":"0x...","puzzle_hash":"0x...","amount":1000000000000},...}],...}}]`
- Fee: (none)

Alice cannot decode the puzzle hashes or solution to understand that this sends 1 XCH to the attacker. She clicks "Push." The transaction is submitted and her 1 XCH is irreversibly transferred to the attacker. [1](#0-0) [7](#0-6)

import { getDappCommandSchema } from './getDappCommandSchema';
import { humanizeParams } from './humanizeParams';

export async function humanizeDappCommand(dappCommand: string, data: Record<string, unknown>, networkPrefix?: string) {
  const dappCommandSchema = getDappCommandSchema(dappCommand);

  const rows = await humanizeParams(dappCommandSchema.params, data, networkPrefix);

  return {
    destructive: dappCommandSchema.destructive === true,
    title: dappCommandSchema.title(),
    message: dappCommandSchema.message(),
    confirmLabel: dappCommandSchema.confirmLabel(),
    rows,
  };
}

  for (const param of dappParams) {
    const { name, type, isOptional } = param;

    const value = nextParams[name];

    // verify if all isOptional !== true params are provided
    if (isOptional !== true && value === undefined) {
      throw new Error(`param is required: ${name}`);
    }

    // parse value to the correct types
    if (value !== undefined) {
      if (type === 'string') {
        nextParams[name] = String(value);
      } else if (type === 'number') {
        nextParams[name] = Number(value);
        if (Number.isNaN(nextParams[name])) {
          throw new Error(`Invalid number value for argument ${name}. Value: ${value}`);
        }
      } else if (type === 'bool') {
        if (typeof value !== 'boolean') {
          throw new Error(`Invalid boolean value for argument ${name}. Value: ${value}`);
        }
        nextParams[name] = value;
      } else if (type === 'bigint') {
        if (typeof value !== 'string' && typeof value !== 'number' && typeof value !== 'bigint') {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        if (typeof value === 'number' && !Number.isSafeInteger(value)) {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        if (typeof value === 'string') {
          const trimmed = value.trim();
          if (trimmed === '') {
            throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
          }
        }

        const bigintValue = BigInt(value);
        if (bigintValue.toString() !== value.toString()) {
          throw new Error(`Invalid bigint value for argument ${name}. Value: ${value}`);
        }

        nextParams[name] = bigintValue;
      }
    }
  }
