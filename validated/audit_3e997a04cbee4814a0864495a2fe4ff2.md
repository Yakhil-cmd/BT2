### Title
FLOW Tokens Permanently Frozen When `CadenceOwnedAccount` Is Destroyed With Non-Zero EVM Balance — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVM.CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` guard that checks for a non-zero EVM balance before the resource is destroyed. Because COA addresses are derived from the resource UUID and carry **no private key**, once the Cadence resource is gone the FLOW tokens held at that EVM address are permanently frozen with no Cadence-side or EVM-side recovery path. Any unprivileged Cadence transaction can trigger this loss.

---

### Finding Description

`EVM.CadenceOwnedAccount` is the only Cadence-native way to own and control an EVM address. Its EVM address is deterministically derived from the resource UUID at creation time:

```cadence
let addr = InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)
``` [1](#0-0) 

FLOW tokens are deposited into the COA's EVM balance via `deposit`, which is `access(all)` — callable by any transaction:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)
}
``` [2](#0-1) 

Withdrawal back to Cadence requires the `Owner | Withdraw` entitlement and is only available on the live `CadenceOwnedAccount` resource:

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault {
``` [3](#0-2) 

The contract itself documents that COA addresses have **no private key**:

> "users are never able to access a key for their account, meaning that they cannot control their COA's address on other EVM blockchains." [4](#0-3) 

`CadenceOwnedAccount` defines **no `destroy` function** — there is no pre-condition that checks `self.balance().attoflow == 0` before the resource is destroyed. Cadence allows `destroy cadenceOwnedAccount` to succeed unconditionally. The EVM state retains the balance at the now-orphaned address, but:

- The `CadenceOwnedAccount` resource is gone → no Cadence-side `withdraw` is possible.
- The EVM address has no private key → no EVM-signed transaction can move the funds.

The `newInternalEVMTypeDepositFunction` in the Go layer confirms that the vault value is intentionally kept alive in the EVM state (not burned), so the tokens exist but are unreachable:

```go
// NOTE: We're intentionally not destroying the vault here,
// because the value of it is supposed to be "kept alive".
``` [5](#0-4) 

---

### Impact Explanation

FLOW tokens deposited to a `CadenceOwnedAccount` that is subsequently destroyed are **permanently frozen** in the EVM state. There is no Cadence-side recovery function (analogous to the missing stETH recovery in wstETH) and no EVM-side key to sign a withdrawal transaction. The frozen tokens remain visible in the EVM balance of the orphaned address but are irrecoverable. This constitutes a **cross-VM asset loss** with no remediation path.

---

### Likelihood Explanation

The likelihood is **medium**. The scenario is reachable by any unprivileged Cadence transaction author:

1. Any protocol or user-written Cadence contract that creates a COA, funds it, and later calls `destroy` on it (e.g., during cleanup, contract upgrade, or error handling) will silently freeze the balance.
2. The existing test suite already demonstrates the exact pattern — `destroy cadenceOwnedAccount` after `cadenceOwnedAccount.deposit(from: <-vault)` — and it succeeds without error, confirming no guard exists. [6](#0-5) 

As COA usage grows in DeFi protocols built on Flow EVM, the probability of accidental destruction with a live balance increases.

---

### Recommendation

Add a `destroy` function to `CadenceOwnedAccount` that panics if the EVM balance is non-zero:

```cadence
destroy() {
    pre {
        self.balance().isZero():
            "EVM.CadenceOwnedAccount: cannot destroy account with non-zero balance"
    }
}
```

This mirrors the standard Cadence pattern for resource safety and forces callers to explicitly withdraw all funds before destroying the COA, preventing silent asset loss.

---

### Proof of Concept

Any unprivileged Cadence transaction can execute the following to permanently freeze FLOW:

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>
import FungibleToken from <FungibleTokenAddress>

transaction(amount: UFix64) {
    prepare(signer: auth(BorrowValue) &Account) {
        // 1. Withdraw FLOW from signer's vault
        let vaultRef = signer.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
                from: /storage/flowTokenVault)
            ?? panic("no vault")
        let vault <- vaultRef.withdraw(amount: amount) as! @FlowToken.Vault

        // 2. Create a COA and fund it
        let coa <- EVM.createCadenceOwnedAccount()
        coa.deposit(from: <-vault)

        // 3. Destroy the COA — no guard fires, balance is permanently frozen
        //    at the COA's EVM address which has no private key
        destroy coa
        // FLOW is now irrecoverable: no Cadence withdraw path, no EVM private key
    }
}
```

The `createCadenceOwnedAccount` call registers the EVM address in the EVM state: [7](#0-6) 

The `deposit` call credits the EVM balance: [8](#0-7) 

The `withdraw` entitlement guard confirms no recovery is possible without the live resource: [9](#0-8) 

The balance precision constraint (`AttoFlowBalanceIsValidForFlowVault`, minimum 1e10 attoflow) further confirms that even partial recovery via a reconstructed path is impossible for sub-minimum dust balances: [10](#0-9)
