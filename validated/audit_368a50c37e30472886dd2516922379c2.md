### Title
`setPoolAdminFeeDestination` Skips Pre-Settlement, Redirecting Already-Accrued Admin Fees to New Destination — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

Every fee-parameter mutation in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` first to settle accrued fees at the current destination before the change takes effect. `setPoolAdminFeeDestination` is the one parallel path that changes a fee-routing parameter **without** first settling, so all fees that have accrued since the last collection are silently redirected to the new destination.

---

### Finding Description

Two fee-update functions correctly call `collectFees` before mutating fee state:

`setPoolAdminFees` (lines 417–425):
```solidity
PoolFeeConfig memory c = poolFeeConfig[pool];
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination used here
);
// ... then updates fee rates
```

`setPoolProtocolFee` (lines 327–335) does the same.

`setPoolAdminFeeDestination` (lines 438–447) does **not**:
```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

After this call, the next invocation of `collectFees` (via `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee`) distributes **all** previously accrued admin fees — both spread-fee surplus and the explicitly tracked `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` — to the **new** destination instead of the old one.

---

### Impact Explanation

The old `adminFeeDestination` is not necessarily the pool admin itself; it can be a separate revenue-sharing contract, a DAO treasury, or a third-party beneficiary set at pool creation. When the pool admin calls `setPoolAdminFeeDestination`, every wei of admin fees that has accrued since the last `collectFees` call is permanently redirected. Because swaps accrue fees continuously, there will almost always be a non-zero uncollected balance at the moment of the destination change. The old destination loses those funds with no recourse.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a legitimate reason to update the fee destination (e.g., rotating a multisig). The missing `collectFees` call is an oversight that fires automatically on every such rotation whenever uncollected fees exist — which is the normal operating state of any active pool.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Settle accrued fees to the OLD destination before changing it
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is active; 100 swaps have occurred, accruing `X` admin spread-fee surplus and `Y` notional fee tokens.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newDest)`. No `collectFees` is triggered.
3. Anyone calls `collectPoolFees(pool)`.
4. `collectFees` reads `poolAdminFeeDestination[pool]` — now `newDest` — and transfers all of `X` and `Y` to `newDest`.
5. The original fee destination receives nothing despite being the rightful recipient of fees accrued during its tenure.

The corrupted value is `poolAdminFeeDestination[pool]` being overwritten before the accrued `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` and spread surplus are settled, causing `collectFees` to route admin fees to the wrong address. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
