### Title
`protocolUnpausePool` Always Transitions to Admin-Paused (Level 1) Regardless of Pre-Pause State, Permanently Blocking Active Pools After a Protocol Pause/Unpause Cycle - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolPausePool` accepts pools at both level 0 (active) and level 1 (admin-paused), but `protocolUnpausePool` unconditionally sets the pool to level 1 (admin-paused) instead of restoring the pre-protocol-pause state. A pool that was fully active (level 0) before a protocol pause is left permanently stuck at level 1 after the protocol unpauses it, blocking all swaps and liquidity operations until the admin separately calls `unpausePool`.

---

### Finding Description

The pool pause state machine has three levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool` accepts the pool at **either** level 0 or level 1 before transitioning to level 2: [1](#0-0) 

`protocolUnpausePool` then **always** sets the pool to level 1, regardless of what the state was before the protocol pause: [2](#0-1) 

The pool's `whenNotPaused` modifier blocks swaps at any non-zero pause level: [3](#0-2) 

**Broken invariant**: When the protocol pauses a pool that was at level 0 (active) and then unpauses it, the pool should return to level 0. Instead it is left at level 1 (admin-paused), a state the admin never set. The admin must now call `unpausePool` to restore functionality — but `unpausePool` itself requires the pool to be at exactly level 1 and the caller to be the pool admin: [4](#0-3) 

The admin may not be aware that a protocol pause/unpause cycle silently placed their pool into admin-paused state, leaving it non-functional indefinitely.

---

### Impact Explanation

After a protocol pause/unpause cycle on a pool that was previously active (level 0), all calls to `swap`, `addLiquidity`, and `removeLiquidity` revert with `PoolPaused` because `pauseLevel == 1 != 0`. This is broken core pool functionality: traders cannot execute swaps, LPs cannot add or remove liquidity, and fees stop accruing. The pool is not permanently bricked (the admin can recover by calling `unpausePool`), but the unexpected state change can leave the pool non-functional for an extended period if the admin is unaware of the implicit side-effect.

---

### Likelihood Explanation

The protocol owner (`onlyOwner`) must call `protocolPausePool` on a pool that is currently at level 0 (active), and then call `protocolUnpausePool`. This is a normal operational sequence — protocol pauses are expected emergency responses. The pool admin is not notified on-chain that they must follow up with `unpausePool`. The likelihood is **medium**: the trigger is a valid, expected owner action, and the resulting stuck state is silent.

---

### Recommendation

`protocolUnpausePool` must restore the pool to its pre-protocol-pause state. Since `protocolPausePool` can be called from level 0 or level 1, the factory must record which level the pool was at before the protocol pause and restore to that level on unpause. One approach:

```solidity
// In factory storage:
mapping(address => uint8) public prePausePauseLevel;

function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    prePausePauseLevel[pool] = cur;   // record pre-pause state
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    uint8 restoreLevel = prePausePauseLevel[pool]; // 0 or 1
    delete prePausePauseLevel[pool];
    IMetricOmmPoolFactoryActions(pool).setPause(restoreLevel);
}
```

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)`. Pool transitions: 0 → 2. ✓
3. Protocol owner calls `protocolUnpausePool(pool)`. Pool transitions: 2 → **1** (admin-paused). ✗
4. Any call to `pool.swap(...)` reverts with `PoolPaused` because `pauseLevel == 1 != 0`.
5. The pool admin must discover the unexpected state and call `factory.unpausePool(pool)` to restore level 0.
6. If the admin is a multisig or automated system unaware of this side-effect, the pool remains non-functional indefinitely.
