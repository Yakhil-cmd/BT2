### Title
Invalid Assumption That First View After Spork Is 0 Causes UInt64 Underflow in Epoch Deployment Transaction - (File: cmd/util/cmd/epochs/cmd/templates.go)

---

### Summary

The `deployEpochTransactionTemplate` in `cmd/util/cmd/epochs/cmd/templates.go` explicitly assumes `// ASSUMPTION: first view after the spork is 0` when computing `adjustedNumViewsInEpoch` and `adjustedNumViewsInStakingAuction`. This assumption does not hold in all valid spork configurations. If the first view after a spork is non-zero (e.g., the network resumes from a non-zero view), both subtractions `numViewsInEpoch - currentBlock.view` and `numViewsInStakingAuction - currentBlock.view` will underflow on `UInt64` in Cadence, causing the transaction to abort with a runtime panic. This prevents the `FlowEpoch` contract from ever being deployed post-spork, halting the entire epoch lifecycle for the network.

---

### Finding Description

In `cmd/util/cmd/epochs/cmd/templates.go` lines 33–39, the Cadence transaction template contains two explicit `ASSUMPTION` comments:

```cadence
// ASSUMPTION: first view after the spork is 0
let adjustedNumViewsInEpoch = numViewsInEpoch - currentBlock.view

// ASSUMPTION: first view after the spork is 0
let adjustedNumViewsInStakingAuction = numViewsInStakingAuction - currentBlock.view
```

The `numViewsInEpoch` and `numViewsInStakingAuction` arguments are computed in `getDeployEpochTransactionArguments` in `cmd/util/cmd/epochs/cmd/deploy.go` (lines 145–147) as:

```go
numViewsInEpoch := (finalView + 1) - head.View
numViewsInStakingAuction := dkgPhase1FinalView - numViewsInDKGPhase - head.View + 1
```

These are computed relative to `head.View` (the snapshot head view), which is the view at the time the snapshot was taken — not the view at the time the transaction executes on-chain. The transaction then subtracts `currentBlock.view` again at execution time. If `currentBlock.view > numViewsInStakingAuction` (which can happen if the staking auction has already passed by the time the transaction is submitted, or if the spork starts at a non-zero view), the `UInt64` subtraction underflows and the Cadence runtime panics, aborting the transaction.

The Go-side argument computation in `deploy.go` already accounts for `head.View` in the subtraction. The Cadence template then subtracts `currentBlock.view` a second time, creating a double-adjustment that is only correct when `head.View == 0` (i.e., the first view after the spork is exactly 0).

---

### Impact Explanation

If the assumption is violated — for example, if the spork root block starts at a non-zero view, or if the `deploy-tx-args` CLI tool is run against a snapshot taken at a later view than expected — the generated Cadence transaction will underflow on `UInt64` arithmetic when executed on-chain. Cadence's `UInt64` subtraction panics on underflow. The `FlowEpoch` contract deployment transaction will abort, and the epoch smart contract cannot be deployed. Without the `FlowEpoch` contract, the network cannot transition epochs, effectively halting the epoch lifecycle. This is a correctness/liveness issue for the protocol's epoch management.

---

### Likelihood Explanation

This is a latent bug that manifests only during post-spork epoch contract deployment when the spork root view is non-zero. Flow sporks historically start at view 0, but this is not enforced anywhere in the protocol. The assumption is explicitly documented as an `ASSUMPTION` comment in the code, acknowledging it is not guaranteed. Any future spork that starts at a non-zero view, or any operator who runs the `deploy-tx-args` tool against a snapshot taken at a non-zero view, will trigger this underflow. The likelihood is low under current operational practice but non-zero and grows as the protocol evolves.

---

### Recommendation

Remove the double-adjustment. The Go-side argument computation in `getDeployEpochTransactionArguments` already computes `numViewsInEpoch` and `numViewsInStakingAuction` relative to `head.View`. The Cadence transaction template should not subtract `currentBlock.view` again. Instead, the template should pass the pre-adjusted values directly to `signer.contracts.add(...)` without further arithmetic, or the Go-side should pass absolute view numbers and let the Cadence template do a single, guarded subtraction with an explicit bounds check. Remove the `ASSUMPTION` comments and add an explicit guard:

```cadence
assert(numViewsInEpoch >= currentBlock.view, message: "numViewsInEpoch underflow")
assert(numViewsInStakingAuction >= currentBlock.view, message: "numViewsInStakingAuction underflow")
```

---

### Proof of Concept

1. Run a spork where the root block has `view = 500` (non-zero).
2. Run `cmd/util/cmd/epochs/cmd/deploy.go`'s `deployRun` against the resulting snapshot. `getDeployEpochTransactionArguments` computes `numViewsInEpoch = (finalView + 1) - 500` and `numViewsInStakingAuction = dkgPhase1FinalView - numViewsInDKGPhase - 500 + 1`.
3. Submit the generated transaction. At execution time, `currentBlock.view` is, say, `600`. The template computes `adjustedNumViewsInStakingAuction = numViewsInStakingAuction - 600`. Since `numViewsInStakingAuction` was already reduced by 500 on the Go side, this subtraction underflows `UInt64`, causing a Cadence runtime panic and transaction abort.
4. The `FlowEpoch` contract is never deployed; epoch transitions are impossible. [1](#0-0) [2](#0-1) [3](#0-2)

// getDeployEpochTransactionArguments pulls out required arguments for the `deploy_epoch_relative` transaction from the root
// protocol snapshot and takes into any required ajustments to align the state of the contract with the protocol state
// and returns an array of the cadence representations of the arguments.
func getDeployEpochTransactionArguments(snapshot *inmem.Snapshot) []cadence.Value {

	// current epoch
	currentEpoch, err := snapshot.Epochs().Current()
	if err != nil {
		log.Fatal().Err(err).Msgf("could not get current epoch")
	}

	head, err := snapshot.Head()
	if err != nil {
		log.Fatal().Err(err).Msgf("could not get head from snapshot")
	}

	epochContractName := systemcontracts.ContractNameEpoch

	// current epoch counter
	currentEpochCounter := currentEpoch.Counter()

	// get final view from snapshot
	finalView := currentEpoch.FinalView()

	dkgPhase1FinalView := currentEpoch.DKGPhase1FinalView()
	dkgPhase2FinalView := currentEpoch.DKGPhase2FinalView()

	numViewsInEpoch := (finalView + 1) - head.View
	numViewsInDKGPhase := dkgPhase2FinalView - dkgPhase1FinalView + 1
	numViewsInStakingAuction := dkgPhase1FinalView - numViewsInDKGPhase - head.View + 1

	// number of collectors clusters
	clustering, err := currentEpoch.Clustering()
	if err != nil {
		log.Fatal().Err(err).Msgf("could not get `clustering` for current epoch from snapshot")
	}
	numCollectorClusters := len(clustering)

	// random source
	randomSource := currentEpoch.RandomSource()

	return convertDeployEpochTransactionArguments(
		epochContractName,
		currentEpochCounter,
		numViewsInEpoch,
		numViewsInStakingAuction,
		numViewsInDKGPhase,
		numCollectorClusters,
		flagFlowSupplyIncreasePercentage,
		randomSource,
		clustering,
	)
}
