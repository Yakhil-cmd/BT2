### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded `while` loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because there is no cap on how many trace entries can accumulate, a pool member who makes repeated balance changes across many epochs without claiming rewards will eventually grow their trace to a size that causes every `claim_rewards` call to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the entire `pool_member_epoch_balance` trace between the member's last-claimed checkpoint and the current epoch: [1](#0-0) 

The developers themselves annotate this loop as unbounded:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.`

The trace is stored as a `PoolMemberBalanceTrace` (a `Vec` of checkpoints keyed by epoch). The `insert` function in `trace.cairo` appends a new checkpoint whenever the epoch key differs from the last entry: [2](#0-1) 

This means the trace grows by one entry per epoch in which a balance-changing operation occurs. The following public entrypoints each call `set_member_balance` or `increase_member_balance`, inserting a new trace entry:

- `enter_delegation_pool` â†’ `set_member_balance` [3](#0-2) 
- `add_to_delegation_pool` â†’ `increase_member_balance` [4](#0-3) 
- `exit_delegation_pool_intent` â†’ `set_member_balance` [5](#0-4) 

The `entry_to_claim_from` field in `pool_member_info` advances only upon a **successful** `claim_rewards` call: [6](#0-5) 

If `claim_rewards` never succeeds (because the loop already exceeds the gas limit), `entry_to_claim_from` never advances, and every subsequent call re-enters the same oversized loop and reverts again â€” permanently locking the member's accrued rewards.

The same unbounded `calculate_rewards` call is also made inside the `pool_member_info_v1` view function: [7](#0-6) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace grows beyond the gas-iteration threshold, every call to `claim_rewards` for that pool member reverts. The member's accumulated STRK rewards become permanently unclaimable. This matches the allowed High impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The only callers permitted to grow a specific member's trace are the pool member themselves or their designated `reward_address`: [8](#0-7) 

This means the scenario is self-inflicted: a pool member who calls `add_to_delegation_pool` (minimum 1 token) or `exit_delegation_pool_intent` once per epoch, across a large number of epochs, without ever calling `claim_rewards`, will accumulate enough trace entries to trigger the gas-limit failure. While this requires sustained activity over many epochs, there is no protocol-level guard preventing it. A legitimate user who adjusts their delegation frequently and defers claiming can reach this state inadvertently. Likelihood is **Low** for deliberate exploitation, but **Medium** for accidental self-infliction over a long protocol lifetime.

---

### Recommendation

1. **Enforce a claim before balance changes**: Require `claim_rewards` to be called (or auto-call it internally) before any operation that calls `set_member_balance` or `increase_member_balance`. This resets `entry_to_claim_from` and bounds the loop to at most a few entries per call.
2. **Cap trace growth**: Reject balance-change operations if the trace length since the last claim exceeds a safe constant (e.g., 1000 entries).
3. **Paginated claiming**: Expose a `claim_rewards_partial(max_entries)` entrypoint so members can drain a large trace incrementally across multiple transactions.

---

### Proof of Concept

1. Attacker (pool member `A`) calls `enter_delegation_pool(reward_address, 1_000_000)`.
2. Each epoch, `A` calls `add_to_delegation_pool(pool_member: A, amount: 1)`. Each call inserts a new checkpoint into `pool_member_epoch_balance[A]` because the epoch key advances.
3. `A` never calls `claim_rewards`. `entry_to_claim_from` stays at `0`.
4. After `N` epochs (where `N` exceeds the gas-iteration threshold), `A` calls `claim_rewards(A)`.
5. `calculate_rewards` enters the `while` loop, iterates `N` times (one `find_sigma` storage read per iteration), exhausts the transaction gas budget, and reverts.
6. Every future `claim_rewards(A)` call also reverts. `A`'s accumulated STRK yield is permanently frozen in the contract.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
