### Title
EVM Pause Blocks All User Withdrawals, Trapping Funds Until Governance Unpauses - (File: fvm/evm/stdlib/contract.cdc)

### Summary
When the Flow EVM is paused by governance, `CadenceOwnedAccount.withdraw()`, `withdrawNFT()`, and `withdrawTokens()` are all blocked by the same `!EVM.isPaused()` pre-condition. Users with FLOW tokens or bridged assets in their COA cannot exit the EVM at all until governance explicitly unpauses. This is the direct analog of the Reserve Protocol finding: a paused state that should permit user exits instead traps user funds behind a governance-only recovery action.

### Finding Description
`EVM.isPaused()` reads a `Bool` from `/storage/evmOperationsPaused` in the EVM contract account's storage. The contract comment states: *"Only the Governance Committee can pause the EVM transactions, with a multi-sig Cadence transaction."*

Every state-mutating user-facing function in the EVM contract carries the same guard:

```cadence

    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
```

No exemption exists for withdrawal/exit operations. The `isPaused()` guard is applied uniformly to every state-mutating path, including the paths that move user-owned FLOW tokens *out* of the EVM back to Cadence. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

### Impact Explanation
Any user who has deposited FLOW tokens into a `CadenceOwnedAccount` (COA) or bridged fungible/non-fungible tokens into Flow EVM cannot recover those assets while the EVM is paused. The only path to recovery is for the Governance Committee to unpause. If governance is slow, unresponsive, or the pause is extended for any reason, user funds are inaccessible for an indefinite period. This is a direct loss of asset availability for all COA holders — not merely a degraded experience. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
The EVM pause is a documented governance mechanism intended for maintenance or emergency use. The governance committee can invoke it via a multi-sig Cadence transaction at any time. Once paused, every COA holder is immediately locked out of their funds. The trigger is a routine governance action, not an exploit, making this a realistic operational scenario rather than a theoretical one. [4](#0-3) 

### Recommendation
Exempt withdrawal/exit operations from the `isPaused()` guard. Specifically, `CadenceOwnedAccount.withdraw()`, `withdrawNFT()`, and `withdrawTokens()` should be callable regardless of pause state, so users can always exit the EVM with their own assets. Only state-mutating operations that are not user exits (e.g., `run`, `call`, `deploy`, `depositNFT`, `depositTokens`) need to remain blocked during a pause.

```cadence
// Before (blocks user exits):
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    ...
}

// After (allows user exits even when paused):
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault {
    // No isPaused() guard — users must always be able to exit
    ...
}
``` [6](#0-5) 

### Proof of Concept
1. Governance sets `/storage/evmOperationsPaused` to `true` (e.g., for routine maintenance).
2. A user with FLOW tokens in their COA submits a Cadence transaction calling `coa.withdraw(balance: bal)`.
3. The pre-condition `!EVM.isPaused()` evaluates to `false`; the transaction aborts with `"EVM operations are temporarily paused"`.
4. The user has no alternative path to recover their FLOW tokens from the EVM.
5. The user remains locked out until governance submits a new multi-sig transaction to set `/storage/evmOperationsPaused` back to `false`.

This is confirmed by the existing test suite, which explicitly verifies that `CadenceOwnedAccount.withdraw` fails when `evmOperationsPaused` is `true`: [7](#0-6)

                amount: balance.inFLOW(),

            return <-vault
