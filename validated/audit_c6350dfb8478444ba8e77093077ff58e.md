### Title
`EXTCODEHASH` Opcode Returns `bytes32(0)` for Addresses with Nonzero Balance Received in the Same Transaction — (`File: fvm/evm/emulator/state/delta.go`)

### Summary
Flow EVM's `DeltaView.AddBalance()` does not create an account record when balance is added to a previously non-existent address. Because `StateDB.Empty()` short-circuits on `!Exist(addr)` without inspecting the balance map, the `EXTCODEHASH` opcode returns `bytes32(0)` for such addresses instead of `keccak256("")`, diverging from EIP-1052 / EIP-161 semantics.

### Finding Description
The go-ethereum EVM engine (used by Flow EVM) implements `EXTCODEHASH` by first calling `StateDB.Empty(address)`. If `Empty` returns `true`, the opcode returns `bytes32(0)`; otherwise it returns `StateDB.GetCodeHash(address)`.

Flow's `StateDB.Empty()` is:

```go
func (db *StateDB) Empty(addr gethCommon.Address) bool {
    if !db.Exist(addr) {
        return true          // ← short-circuits here; balance is never checked
    }
    return db.GetNonce(addr) == 0 &&
        db.GetBalance(addr).Sign() == 0 &&
        bytes.Equal(db.GetCodeHash(addr).Bytes(), gethTypes.EmptyCodeHash.Bytes())
}
```

`Exist()` delegates to `DeltaView.Exist()`, which checks `d.created[addr]` and the parent `BaseView`. It does **not** inspect `d.balances`.

`DeltaView.AddBalance()` only writes to `d.balances[addr]` and `d.dirtyAddresses[addr]`; it never inserts the address into `d.created`:

```go
func (d *DeltaView) AddBalance(addr gethCommon.Address, amount *uint256.Int) error {
    ...
    newBalance := new(uint256.Int).Add(orgBalance, amount)
    d.balances[addr] = newBalance          // ← existence NOT recorded
    d.dirtyAddresses[addr] = struct{}{}
    return nil
}
```

By contrast, go-ethereum's own `StateDB.AddBalance` creates the account object if it does not yet exist, so `Exist()` subsequently returns `true` and `Empty()` correctly evaluates the balance field.

The consequence: within a single EVM transaction, if contract A sends ETH to a previously non-existent address B (via a `CALL` with value, which triggers `Transfer → AddBalance`), and then calls `EXTCODEHASH(B)`:

- `Exist(B)` → `false` (B is in `d.balances` but not `d.created` or `BaseView`)
- `Empty(B)` → `true` (short-circuits on `!Exist`)
- `EXTCODEHASH` → `bytes32(0)`

Per EIP-161, B is **not** empty (it has nonzero balance), so EIP-1052 mandates `EXTCODEHASH` return `keccak256("")`. Flow EVM returns `bytes32(0)` instead.

### Impact Explanation
Any smart contract that (a) sends ETH to a fresh address and (b) inspects `EXTCODEHASH` of that address in the same transaction will observe `bytes32(0)` on Flow EVM instead of `keccak256("")`. Contracts that use `EXTCODEHASH` to distinguish "account with no code" from "non-existent account" (a common pattern for EOA detection and proxy guards) will malfunction. This is a cross-VM behavioral inconsistency reachable by any unprivileged EVM transaction sender.

### Likelihood Explanation
The sequence (value transfer to a new address followed by `EXTCODEHASH` in the same transaction) is uncommon but not exotic — it appears in factory patterns, airdrop contracts, and proxy-detection guards. The likelihood is low-to-medium; the bug is silent (no revert, just a wrong return value).

### Recommendation
`DeltaView.AddBalance()` should create the account in `d.created` (or an equivalent existence set) when balance is added to an address that does not yet exist in the parent view, mirroring go-ethereum's own `StateDB.AddBalance` behavior. Alternatively, `DeltaView.Exist()` should also return `true` when `d.balances[addr]` is present with a nonzero value.

### Proof of Concept

**Root cause chain:**

1. EVM executes `CALL{value: 1 wei, to: freshAddr}` → go-ethereum calls `Transfer(db, caller, freshAddr, 1)` → `db.AddBalance(freshAddr, 1)`.
2. `DeltaView.AddBalance` sets `d.balances[freshAddr] = 1`; `d.created` is untouched.
3. EVM executes `EXTCODEHASH(freshAddr)` → go-ethereum calls `db.Empty(freshAddr)`.
4. `StateDB.Empty` calls `db.Exist(freshAddr)` → `DeltaView.Exist` checks `d.created` (miss) → checks `BaseView` (miss) → returns `false`.
5. `Empty` returns `true` (short-circuit).
6. `EXTCODEHASH` pushes `bytes32(0)` — **wrong**; EIP-1052 requires `keccak256("")` because `freshAddr` has nonzero balance.

// AddBalance adds the amount to the current balance of the given address
func (d *DeltaView) AddBalance(addr gethCommon.Address, amount *uint256.Int) error {
	// if amount is 0 skip
	if amount.Sign() == 0 {
		return nil
	}
	// get the latest balance
	orgBalance, err := d.GetBalance(addr)
	if err != nil {
		return err
	}

	// flag the address as dirty

	return nil

// GetCodeHash returns the code hash of an address
//
// for non-existent accounts it returns gethCommon.Hash{}
// and for accounts without a code (e.g. EOAs) it returns default empty
// hash value (gethTypes.EmptyCodeHash)
func (v *BaseView) GetCodeHash(addr gethCommon.Address) (gethCommon.Hash, error) {
	acc, err := v.getAccount(addr)
	codeHash := gethCommon.Hash{}
	if acc != nil {
		codeHash = acc.CodeHash
	}
	return codeHash, err
}
