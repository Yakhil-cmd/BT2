### Title
`native_per_pubdata` Truncates to Zero via Integer Division, Allowing Pubdata Generation Without Native Resource Cost - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the `native_per_pubdata` ratio — which governs how many native (proving) resources are charged per byte of pubdata — is computed via a truncating integer division. When `pubdata_price < native_price`, this division rounds down to zero. A transaction sender can then generate arbitrarily large amounts of pubdata without being charged any native resources for it, bypassing the pubdata pricing mechanism entirely.

---

### Finding Description

The double resource accounting model in ZKsync OS derives a `native_per_pubdata` value from two operator-set prices:

```
native_per_pubdata := pubdata_price / native_price   (integer division, truncating)
```

This is computed in `validate_and_compute_fee_for_transaction` (L2 transactions):

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same truncating division appears in the API helper used for pre-validation:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

Note that `native_per_gas` uses `div_ceil` (rounding up, protecting the operator), but `native_per_pubdata` uses `wrapping_div` (truncating, rounding down). [3](#0-2) 

When `pubdata_price < native_price` (e.g., `native_price = 1000`, `pubdata_price = 500`), `native_per_pubdata = 0`.

This zero propagates through the entire pubdata charging pipeline:

**1. Intrinsic pubdata overhead becomes zero** in `create_resources_for_tx`:
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0
``` [4](#0-3) 

**2. Post-execution pubdata charge becomes zero** in `get_resources_to_charge_for_pubdata`:
```rust
let native = current_pubdata_spent.checked_mul(native_per_pubdata)?;
// = current_pubdata_spent * 0 = 0
``` [5](#0-4) 

**3. The pubdata sufficiency check always passes** in `check_enough_resources_for_pubdata`, since `resources_for_pubdata = 0` is always affordable. [6](#0-5) 

The documentation confirms the intended formula is `nativePerPubdata := pubdataPrice / nativePrice`, but does not specify rounding direction: [7](#0-6) 

---

### Impact Explanation

When `native_per_pubdata = 0`, any transaction sender can generate up to the block's `pubdata_limit` worth of pubdata (via SSTORE operations, contract deployments, etc.) while paying only EVM gas — not the native resource cost that represents the ZK proving overhead for that pubdata. The operator must prove all pubdata but receives no native resource compensation for it. This is a resource accounting bug: the operator's proving costs for pubdata are systematically underfunded whenever `pubdata_price < native_price`.

Concretely, an attacker can:
1. Submit transactions that write to many storage slots (each generating pubdata).
2. Pay only EVM gas (SSTORE costs), but zero native resources for the pubdata.
3. Force the operator to prove large pubdata payloads at a loss.

This is directly analogous to the Buyout.sol bug: just as `fractionPrice = buyoutPrice / totalSupply` rounding to zero allowed fractions to be purchased for free, `native_per_pubdata = pubdata_price / native_price` rounding to zero allows pubdata to be generated for free (in native resource terms).

---

### Likelihood Explanation

The operator independently sets `native_price` and `pubdata_price`. A configuration where `pubdata_price < native_price` is realistic — for example, when the operator lowers pubdata prices to attract users, or when `native_price` is set high to reflect expensive proving cycles while `pubdata_price` reflects a cheaper L1 data availability cost. No special privileges are required from the attacker; any transaction sender can exploit this once the operator's prices satisfy `pubdata_price < native_price`.

---

### Recommendation

Replace the truncating `wrapping_div` with a ceiling division (`div_ceil`) for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncating):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that even when `pubdata_price < native_price`, `native_per_pubdata` is at least 1, so pubdata always carries a non-zero native resource cost.

---

### Proof of Concept

**Setup**: Operator sets `native_price = 1000`, `pubdata_price = 500`.

**Computation**:
- `native_per_pubdata = 500 / 1000 = 0` (truncating integer division)

**Attack**:
1. Attacker submits a transaction with `gas_limit = 10_000_000`, `gas_price = 1000`.
2. Transaction executes a loop writing to 1000 storage slots (generating ~32,000 bytes of pubdata).
3. `get_resources_to_charge_for_pubdata` returns `0` native for 32,000 bytes of pubdata.
4. `check_enough_resources_for_pubdata` returns `enough = true`.
5. Transaction succeeds; attacker paid only EVM gas (SSTORE × 1000), zero native for pubdata.
6. Operator must prove 32,000 bytes of pubdata with no native resource compensation.

With `div_ceil`, `native_per_pubdata = ceil(500/1000) = 1`, and the 32,000 bytes would cost 32,000 native units — correctly charging the attacker.
