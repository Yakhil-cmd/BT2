### Title
Silent Asset Drop in Offer Display Enables Acceptance of Offers with Hidden Assets — (File: `packages/gui/src/util/offerToOfferBuilderData.ts`)

---

### Summary

`offerToOfferBuilderData` silently discards any asset whose `infos` entry is absent or whose `type` is not exactly `'CAT'` or `'singleton'`. Because the user's acceptance call sends the raw `offerData` blob — not the rendered summary — an offer containing an asset that falls through this filter is executed in full while the user sees only a partial picture of what they agreed to.

---

### Finding Description

**Vulnerability class (from external report):** Serialization/display code that does not account for the full variety of inputs it can receive, producing hard-to-reach logic bugs — here, silent omission of assets from the confirmation UI.

In `offerToOfferBuilderData`, every asset ID in `requested` and `offered` is classified by a three-branch chain:

```
if (info?.type === 'CAT')      → show as token
else if (info?.type === 'singleton') → show as NFT
else if (id === 'xch')         → show as XCH
// else: silently dropped — no error, no warning
``` [1](#0-0) 

Any asset whose `infos[id]` is `undefined` (no driver entry) or whose `type` string is anything other than the two hard-coded literals falls through all three branches and is never added to the returned `OfferBuilderData`. The function returns successfully with no indication that an asset was omitted.

The Chia wallet RPC `get_offer_summary` populates `infos` only for asset types it recognises. An offer containing a puzzle with an unusual or novel driver type can produce a summary where the asset appears in `offered`/`requested` with a non-zero amount but has no corresponding `infos` entry. The GUI's TypeScript type for `OfferSummaryInfos` is an open index signature, so the runtime value can legally have missing keys. [2](#0-1) 

The rendered `OfferBuilderData` is what the user sees in `OfferBuilderViewer`. When the user clicks Accept, however, the code calls `takeOffer` with the original raw `offerData` string, not the filtered display object: [3](#0-2) 

The same raw blob is used in the `OfferBuilderViewer` submit handler: [4](#0-3) 

So the wallet backend executes every asset in the blob regardless of what the display layer chose to render.

The WalletConnect path (`parseCommandDisplay` → `offerSummaryToWalletDelta`) does **not** have this gap — it iterates `summary.requested` and `summary.offered` unconditionally and includes every asset in the delta shown to the user: [5](#0-4) 

This inconsistency between the two display paths confirms the bug is specific to the `offerToOfferBuilderData` branch.

---

### Impact Explanation

A victim who imports and accepts a crafted offer is shown only the XCH/CAT/NFT assets that matched the three known type strings. Any additional asset with an unrecognised driver type is invisible in the confirmation UI but is still transferred on-chain when `takeOffer` executes the raw blob. This satisfies the **High** impact criterion: *unsafe trust of offer state that causes a user to approve the wrong asset or amount*.

---

### Likelihood Explanation

The attacker must be able to craft an offer blob that the Chia wallet backend parses into a summary where at least one asset appears in `offered`/`requested` without a corresponding `infos` entry. This requires knowledge of the Chia spend-bundle format and the ability to construct a puzzle with a driver type the backend does not classify. The victim must then import and accept the offer through the standard GUI flow (file drop, paste, URL, or WalletConnect notification). No privileged access, leaked keys, or cryptographic break is required. Likelihood is **medium** — the craft requires protocol-level knowledge, but the entry path (offer import) is fully unprivileged.

---

### Recommendation

1. **Fail closed in `offerToOfferBuilderData`:** After the three-branch classification, if an asset ID is not `'xch'` and `info` is absent or its `type` is unrecognised, throw an error (or at minimum surface a visible warning) rather than silently dropping the asset.

```typescript
// After the existing else-if chain:
} else {
  throw new Error(`Unknown asset type for id ${id}: ${info?.type}`);
}
```

2. **Add a completeness check:** After building `offeredTokens`, `offeredNfts`, `offeredXch`, etc., assert that `Object.keys(requested).length === offeredXch.length + offeredTokens.length + offeredNfts.length` (and similarly for `offered`). A mismatch means at least one asset was dropped.

3. **Align the two display paths:** The WalletConnect path already includes all assets. Apply the same completeness guarantee to the `offerToOfferBuilderData` path.

---

### Proof of Concept

1. Construct a Chia offer blob that includes a coin spend for an asset whose puzzle driver type is not `'cat'` or `'singleton'` (e.g., a custom puzzle or a future asset type). Ensure the spend bundle is otherwise valid so `get_offer_summary` returns `success: true`.

2. The backend returns a summary such as:
   ```json
   {
     "offered":    { "xch": "1000000000000" },
     "requested":  { "<custom_asset_id>": "500" },
     "infos":      {}
   }
   ```
   (`infos` has no entry for `<custom_asset_id>`.)

3. Victim imports the offer via the GUI (file drop, paste, or URL). `offerToOfferBuilderData` processes `requested`:
   - `info = infos["<custom_asset_id>"]` → `undefined`
   - None of the three branches match → asset silently dropped
   - Display shows only "You will receive 1 XCH"

4. Victim clicks Accept. `takeOffer({ offer: offerData })` sends the full blob. The custom asset is transferred on-chain. [1](#0-0) [6](#0-5) 

**Note on exploitability uncertainty:** The concrete reachability of step 2 — whether the Chia wallet backend can return an asset in `offered`/`requested` with no `infos` entry — depends on backend behaviour not visible in this repository. The GUI-side code gap (silent drop with no guard) is confirmed. If the backend ever omits `infos` entries for any asset class (novel puzzle types, future protocol extensions, or edge cases in the current parser), the impact path is fully open.
