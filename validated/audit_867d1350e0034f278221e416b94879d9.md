### Title
Malicious CKD Participant Can Corrupt the Coordinator's Derived Key Output by Sending Unverified Share Contributions — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator receives `CKDOutput` values `(big_y_i, big_c_i)` from every participant and blindly accumulates them with no cryptographic proof that each contribution was computed honestly. A single malicious participant can substitute arbitrary group elements, causing the coordinator to assemble an incorrect confidential derived key that honest parties will accept as valid.

---

### Finding Description

`do_ckd_coordinator` collects participant shares and sums them:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant is supposed to send:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is their private signing share and `y_i` is a fresh random scalar. The coordinator then forms the final output `(Σ norm_big_y_i, Σ norm_big_c_i)`.

**No proof is attached to the contribution.** There is no discrete-log equality proof, no ElGamal consistency proof, and no check that `big_c_i` is consistent with the participant's known public key share `X_i = x_i · G₂`. The only gate is successful deserialization.

A malicious participant can send any pair `(big_y_i', big_c_i')` of valid group elements. The coordinator adds them without complaint. The resulting output satisfies:

```
unmask(app_sk) = Σ norm_big_c_i' − app_sk · Σ norm_big_y_i'
               ≠ msk · H(pk ‖ app_id)
```

The coordinator returns a structurally well-formed `CKDOutput` that is cryptographically wrong.

The root cause is the direct analog of the oracle report: a multi-field response is received (`big_y`, `big_c`), only the raw values are consumed, and the implicit validity constraint (that `big_c_i` encodes the participant's actual secret share) is never verified.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator is the sole aggregator; it has no way to detect the bad contribution after the fact. Every downstream consumer of the `CKDOutput` (e.g., a TEE that calls `unmask`) will silently derive the wrong confidential key. Because the output is deterministically wrong rather than randomly wrong, the attacker can also bias the result toward a chosen value if they know `app_sk` (the application secret key), potentially enabling targeted key substitution.

---

### Likelihood Explanation

Any single participant in the CKD session can mount this attack. No special privilege is required beyond being a listed participant. The attacker needs only to deviate from the protocol by sending crafted group elements — a straightforward modification of the library's own `compute_signature_share` path. The attack is silent: the coordinator returns `Ok(Some(ckd_output))` with no error.

---

### Recommendation

Require each participant to attach a zero-knowledge proof of correct encryption alongside their `CKDOutput`. Concretely, a Chaum–Pedersen / discrete-log-equality proof showing that the same scalar `λ_i · y_i` was used in both `norm_big_y_i` and the randomness term of `norm_big_c_i`, **and** that `norm_big_c_i − λ_i · y_i · app_pk = λ_i · x_i · H(pk ‖ app_id)` is consistent with the public key share `X_i`, would close the gap. The coordinator must verify all proofs before accumulating any contribution.

---

### Proof of Concept

1. Honest participants P₁…P_{n-1} run `compute_signature_share` correctly.
2. Malicious participant Pₙ replaces its output with `(G₁, G₁)` (the generator point for both fields).
3. The coordinator's loop at lines 50–55 adds `G₁` to both running sums without error.
4. The final `CKDOutput` satisfies `big_c − app_sk · big_y ≠ msk · H(pk ‖ app_id)`.
5. Any caller of `ckd_output.unmask(app_sk)` receives a wrong key with no indication of failure. [1](#0-0) [2](#0-1)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
