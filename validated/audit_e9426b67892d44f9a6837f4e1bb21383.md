### Title
Gateway Signature Validation Skip Allows Invalid-Signature Invoke Transaction Admission via Deploy-Account UX Bypass - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for an invoke transaction with nonce 1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender. The check is intended to improve UX for the deploy-account + invoke pair, but the predicate is too broad: it returns `true` for **any** transaction in the pool for that account, not specifically a deploy-account. An unprivileged attacker who observes a legitimate user's pending deploy-account transaction can submit a replacement invoke transaction with an **invalid signature** and a higher fee, which the gateway accepts without signature verification. The mempool's fee-escalation mechanism then replaces the user's valid invoke with the attacker's invalid one. When the batcher executes the block, the deploy-account succeeds but the invalid invoke fails at `__validate__`, consuming the user's nonce and charging a fee.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip signature verification) when all three conditions hold:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

The comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction in the pool for that address: [2](#0-1) 

It checks `self.state.contains_account` (committed/staged nonces) **or** `self.tx_pool.contains_account` (any pending tx). There is no requirement that the pooled transaction is specifically a deploy-account.

**Validation order in `run_pre_validation_checks`:** [3](#0-2) 

`validate_by_mempool` is called first and only checks that the nonce is not too old — it does **not** verify the signature. Then `skip_stateful_validations` is evaluated. If it returns `true`, `run_validate_entry_point` is called with `validate: false`: [4](#0-3) 

The `__validate__` entry point (signature check) is therefore never executed for the attacker's transaction.

**Attack path:**

1. Alice submits `deploy_account` (nonce 0) + `invoke` (nonce 1) simultaneously. Both enter the mempool.
2. Attacker observes Alice's pending deploy-account in the mempool.
3. Attacker submits `invoke` for Alice's address with nonce 1, **invalid signature**, and a higher fee than Alice's invoke.
4. Gateway: nonce check passes (1 ≥ 0); `validate_by_mempool` passes (nonce not too old); `skip_stateful_validations` returns `true` (Alice's deploy-account is in the pool); `run_validate_entry_point` is skipped.
5. Gateway forwards the attacker's tx to the mempool via `add_tx`.
6. Mempool fee-escalation replaces Alice's valid invoke with the attacker's invalid-signature invoke.
7. Batcher executes: Alice's deploy-account succeeds (nonce 0→1); the attacker's invoke is executed, `__validate__` fails (invalid signature), the transaction is reverted, nonce 1→2, fee charged to Alice's account.
8. Alice's nonce 1 is permanently consumed; she must resubmit with nonce 2.

The `committed_nonce_retention_block_count` (default 100 blocks) also means that after a deploy-account is committed, `state.contains_account` returns `true` for up to 100 subsequent blocks: [5](#0-4) 

This extends the attack window beyond the initial pending period.

### Impact Explanation

The gateway/mempool admission path accepts an invalid transaction (one whose `__validate__` entry point will fail at execution time) without performing signature verification. This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The concrete consequence is that an attacker can grief any user who submits a deploy-account + invoke pair by replacing the invoke with an invalid-signature transaction, causing the user's nonce to be consumed and a fee to be charged for a failed transaction.

### Likelihood Explanation

The attack requires:
- Monitoring the public mempool for deploy-account transactions (trivial).
- Submitting a replacement invoke with a higher fee (low cost).
- Acting within the window before the deploy-account is committed (typically seconds to minutes, and extended by `committed_nonce_retention_block_count`).

No privileged access is required. The attack is repeatable and can target any new account deployment.

### Recommendation

The `skip_stateful_validations` predicate must be tightened. Instead of checking whether **any** transaction for the account exists in the pool, it should verify that a **deploy-account transaction specifically** is present. One approach:

- Add a dedicated mempool query `has_pending_deploy_account(address) -> bool` that checks only for a pending deploy-account transaction (nonce 0, type `DeployAccount`) for the given address.
- Replace the `account_tx_in_pool_or_recent_block` call in `skip_stateful_validations` with this stricter check.

Alternatively, remove the skip mechanism entirely and require users to wait for the deploy-account to be committed before submitting subsequent invoke transactions, accepting the UX trade-off.

### Proof of Concept

```
1. Alice: submit deploy_account(nonce=0, sig=valid) → accepted, enters mempool pool
2. Alice: submit invoke(nonce=1, sig=valid, tip=10) → accepted via skip_validate, enters mempool pool
3. Attacker: submit invoke(nonce=1, sig=0x0000...INVALID, tip=20) for Alice's address
   - validate_state_preconditions: nonce 1 >= account_nonce 0 → PASS
   - validate_by_mempool: nonce 1 >= resolved_nonce 0 → PASS
   - skip_stateful_validations: account_tx_in_pool_or_recent_block(Alice) == true → returns true (SKIP)
   - run_validate_entry_point: validate=false → __validate__ NOT called
   - Gateway sends to mempool
4. Mempool fee-escalation: tip 20 > tip 10 → replaces Alice's invoke with attacker's invoke
5. Batcher executes block:
   - deploy_account(nonce=0): SUCCESS, Alice's nonce → 1
   - invoke(nonce=1, sig=INVALID): __validate__ FAILS, reverted, nonce → 2, fee charged
6. Alice's nonce 1 is consumed; she must resubmit with nonce 2
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
