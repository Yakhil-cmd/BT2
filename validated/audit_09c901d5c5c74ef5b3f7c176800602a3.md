### Title
Missing Validation of Identity-Point `app_pk` in `ckd()` Allows Coordinator to Extract Confidential Derived Key — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The public `ckd()` entry-point accepts a caller-supplied `app_pk: PublicKey` (i.e., `blstrs::G1Projective`) without checking that it is not the group identity element. When `app_pk = G1::identity()`, the ElGamal masking term that is supposed to hide the BLS signature from the coordinator collapses to zero, causing `CKDOutput.big_c` to equal the raw confidential derived key `msk · H(pk ‖ app_id)` in the clear.

---

### Finding Description

The CKD protocol is designed so that each participant computes:

```
big_c_i = x_i · H(pk ‖ app_id)   +   y_i · app_pk
          ^^^^^^^^^^^^^^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^^
          BLS signature share           ElGamal mask
```

The coordinator aggregates these into:

```
big_C = msk · H(pk ‖ app_id)  +  (Σ λ_i · y_i) · app_pk
```

Because the coordinator knows `big_Y = Σ λ_i · y_i · G` but not the discrete log `Σ λ_i · y_i`, it cannot strip the mask — unless `app_pk = identity`, in which case the mask term is identically zero and `big_C = msk · H(pk ‖ app_id)` directly.

The `ckd()` function performs several input checks (participant count, duplicates, self-membership, coordinator membership) but never validates that `app_pk` is a non-identity curve point: [1](#0-0) 

The masking computation inside `compute_signature_share` is: [2](#0-1) 

Specifically, line 174 (`let big_c = big_s + app_pk * y.0;`) produces `big_c = big_s` when `app_pk` is the identity, because `identity * y = identity` and `big_s + identity = big_s`.

The coordinator then sums the Lagrange-weighted shares: [3](#0-2) 

When every participant uses `app_pk = identity`, the aggregated `norm_big_c` values sum to `msk · H(pk ‖ app_id)` — the exact confidential derived key — which is returned verbatim in `CKDOutput.big_c`.

The `CKDOutput` type and its `unmask` helper are defined here: [4](#0-3) 

---

### Impact Explanation

The CKD protocol's stated purpose is to prevent the key-derivation service (coordinator + participants) from learning the confidential derived key `msk · H(pk ‖ app_id)`. The ElGamal masking with `app_pk` is the sole mechanism providing this guarantee. When `app_pk = identity`, that mechanism is entirely bypassed: `CKDOutput.big_c` is the raw BLS signature, readable by the coordinator without any further computation. This constitutes **disclosure of a confidential derived secret**, matching the Critical impact tier.

---

### Likelihood Explanation

In a real deployment the coordinator is the party that receives `app_pk` from the client and distributes it to all participants out-of-band (the protocol itself has no round that authenticates or cross-checks `app_pk` among participants). A malicious coordinator can silently substitute `G1Projective::identity()` for the legitimate client public key. Because `ckd()` performs no identity-point check, every honest participant will accept the substituted value and the attack succeeds in a single protocol execution.

---

### Recommendation

Add an explicit identity-point guard at the top of `ckd()` in `src/confidential_key_derivation/protocol.rs`, immediately after the existing participant-list checks:

```rust
use elliptic_curve::Group; // already in scope via `use elliptic_curve::{Field, Group};`

if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the group identity element".to_string(),
    ));
}
```

This mirrors the pattern already used elsewhere in the codebase (e.g., `assert_key_invariants` rejecting a zero threshold) and closes the bypass with a single cheap comparison. [5](#0-4) 

---

### Proof of Concept

```
Setup
─────
participants = [P1, P2, P3]  (threshold-2 DKG already completed)
msk = λ1·x1 + λ2·x2 + λ3·x3   (master secret key)
app_id = b"target-app"
app_pk_honest = app_sk · G1::generator()   // legitimate client key

Attack
──────
Malicious coordinator distributes app_pk = G1Projective::identity()
to P1, P2, P3 instead of app_pk_honest.

Each Pi calls ckd(..., app_pk = identity, ...).
Inside compute_signature_share:
  big_c_i = x_i · H(pk ‖ app_id)  +  identity · y_i
           = x_i · H(pk ‖ app_id)          // mask term = 0

Coordinator aggregates:
  big_C = Σ λ_i · big_c_i
        = Σ λ_i · x_i · H(pk ‖ app_id)
        = msk · H(pk ‖ app_id)             // confidential derived key, in the clear

CKDOutput { big_y: ..., big_c: msk·H(pk‖app_id) }
                                ^^^^^^^^^^^^^^^^^^^
                                coordinator reads this directly —
                                no app_sk needed, masking fully bypassed.
```

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
