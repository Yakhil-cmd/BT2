### Title
TOCTOU Pool `targetPuzzleHash` Substitution Redirects All Farming Rewards to Attacker - (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
When a user joins or changes a pool, the GUI fetches `targetPuzzleHash` from the pool's HTTP endpoint **twice**: once to display it in the "Verify Pool Details" UI step, and again independently at submit time inside `prepareSubmitData`. A malicious pool operator can serve a legitimate-looking `targetPuzzleHash` for the display call and a different, attacker-controlled puzzle hash for the submit-time call. The transaction is signed and submitted with the attacker's hash, permanently redirecting all XCH farming rewards to the attacker.

### Finding Description

The pool-join flow involves two components:

**Display (check):** `PlotNFTSelectBase` calls `usePoolInfo(poolUrl)`, which internally calls `getPoolInfo(normalizedUrl)` → `window.appAPI.fetchPoolInfo(poolUrl)`. The returned `targetPuzzleHash` is rendered in the "Verify Pool Details" card for the user to review. [1](#0-0) [2](#0-1) 

**Submit (use):** When the user clicks "Create" or "Change", `handleSubmit` calls `prepareSubmitData`, which makes a **second, independent** HTTP call to `getPoolInfo(normalizedPoolUrl)`. The `targetPuzzleHash` from this second response is what is embedded in the on-chain transaction passed to `createNewPoolWallet` or `pwJoinPool`. [3](#0-2) [4](#0-3) 

There is no code that compares the `targetPuzzleHash` shown to the user against the one used in the transaction. The two fetches are completely decoupled. [5](#0-4) 

### Impact Explanation

`targetPuzzleHash` is the puzzle hash that receives all pooled farming rewards. By substituting it between display and submission, the attacker causes the user's Plot NFT singleton to be locked to the attacker's reward address on-chain. Every block reward earned by the user's plots is permanently paid to the attacker. The user has no recourse without paying another fee to change pools again — and the same attack can be repeated on that attempt.

This matches the **Critical** impact class: unauthorized accounting change affecting XCH pooled farming rewards.

### Likelihood Explanation

The attacker only needs to operate a pool HTTP server. No keys, no host compromise, no cryptographic break is required. The server simply returns different JSON for the two requests (e.g., by fingerprinting the request timing, User-Agent, or a session cookie set during the display fetch). The window between the display fetch and the submit fetch can be arbitrarily long (the user may spend minutes reviewing the UI), making the timing trivial to exploit.

### Recommendation

Cache the `poolInfo` result from the display fetch and pass it directly into `prepareSubmitData` instead of re-fetching. Concretely:

1. Lift the `usePoolInfo` result out of `PlotNFTSelectBase` and into `PlotNFTSelectPool`.
2. Pass the already-fetched `poolInfo` (including `targetPuzzleHash` and `relativeLockHeight`) into `prepareSubmitData` directly, eliminating the second HTTP call.
3. If a second fetch is ever needed for freshness, assert that `targetPuzzleHash` in the second response is byte-for-byte identical to the one shown to the user, and abort with an error if they differ.

### Proof of Concept

1. Attacker deploys a pool server at `https://evil-pool.example`.
2. The server tracks whether a given client IP has already fetched `/pool_info`:
   - **First request** (display): returns `{ "target_puzzle_hash": "0xLEGIT...", ... }`.
   - **Second request** (submit): returns `{ "target_puzzle_hash": "0xATTACKER_WALLET...", ... }`.
3. Victim opens Chia GUI → Pool → Add/Change Pool → enters `https://evil-pool.example`.
4. GUI renders "Verify Pool Details" showing `targetPuzzleHash = 0xLEGIT...`. Victim reviews and clicks "Create".
5. `prepareSubmitData` fires a second fetch; server returns `0xATTACKER_WALLET...`.
6. `createNewPoolWallet` / `pwJoinPool` is called with `targetPuzzleHash = 0xATTACKER_WALLET...`.
7. The Plot NFT singleton is committed on-chain with the attacker's puzzle hash. All future farming rewards flow to the attacker. [6](#0-5) [7](#0-6)

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
