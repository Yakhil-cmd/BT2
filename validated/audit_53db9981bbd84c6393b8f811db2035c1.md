### Title
Bank Precompile `transfer` and `burn` Allow Any EVM Contract to Move or Destroy Tokens from Arbitrary Holders Without Consent — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` and `burn` methods derive the token denom from `contract.Caller()` (correctly scoping the token namespace to the calling contract), but accept the **source address** as a fully attacker-controlled argument with no authorization check. Any deployed EVM contract can therefore transfer or burn `evm/<itself>` tokens held by any address without that address's knowledge or consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles `transfer` and `burn` without verifying that the source address equals the calling contract.

**`burn` path (lines 113–156):**

```go
recipient := args[0].(common.Address)   // address to burn FROM — caller-supplied
amount    := args[1].(*big.Int)
addr      := sdk.AccAddress(recipient.Bytes())
denom     := EVMDenom(contract.Caller()) // "evm/<callerAddress>"
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
``` [1](#0-0) 

`addr` is taken from `args[0]` — any address the caller supplies. There is no `require addr == contract.Caller()` guard.

**`transfer` path (lines 167–200):**

```go
sender    := args[0].(common.Address)   // source address — caller-supplied
recipient := args[1].(common.Address)
// ...
denom := EVMDenom(contract.Caller())    // "evm/<callerAddress>"
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [2](#0-1) 

`from` is derived from the caller-supplied `sender` argument. `SendCoins` executes unconditionally — no approval, no ownership check.

The only guard present is `checkBlockedAddr` on the **recipient**, which is irrelevant to the authorization of the **source** address. [3](#0-2) 

The bank precompile is registered at address `0x0000000000000000000000000000000000000064` and is callable by any EVM contract with no whitelist. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized transfer and burn of precompile-controlled assets.**

Any EVM contract can:

1. Call `transfer(victimAddress, attackerAddress, amount)` on the bank precompile to move `evm/<itself>` tokens from any holder to any destination without the holder's consent.
2. Call `burn(victimAddress, amount)` on the bank precompile to destroy `evm/<itself>` tokens held by any address without the holder's consent.

`evm/<contract>` tokens are Cosmos-native bank module coins. Once stolen via `transfer`, they are real, transferable, spendable assets. Once destroyed via `burn`, they are permanently lost. Both outcomes match the Critical impact class: *unauthorized transfer/burn of precompile-controlled assets*.

---

### Likelihood Explanation

- Any unprivileged attacker can deploy an EVM contract — no admin, governance, or validator key required.
- The bank precompile is unconditionally callable by any EVM contract.
- Victims hold `evm/<attackerContract>` tokens if they minted them through the attacker's contract, or if they received them from a third party (e.g., as payment, airdrop, or LP reward). Victims in the second category never interacted with the attacker's contract and have no reason to distrust the tokens.
- The attack is a single EVM call — no multi-step setup beyond deploying the contract and waiting for victims to accumulate balances.

---

### Recommendation

For `transfer`: add `require(sender == contract.Caller())` before executing `SendCoins`, so a contract can only move tokens it itself holds.

For `burn`: add `require(recipient == contract.Caller())` before executing `SendCoinsFromAccountToModule`, so a contract can only burn tokens it itself holds.

This mirrors the namespace-restriction fix recommended in the RocketStorage report: restrict each actor to operations within its own scope.

---

### Proof of Concept

```
1. Attacker deploys MaliciousContract at 0xDEAD.

2. Attacker (or any user) calls MaliciousContract, which calls:
       bankPrecompile.mint(victimAddress, 1_000_000)
   Victim now holds 1_000_000 evm/0xDEAD tokens in the Cosmos bank module.

3. Victim may transfer some of these tokens to other users via normal bank sends —
   those users also now hold evm/0xDEAD tokens.

4. Attacker calls MaliciousContract, which calls:
       bankPrecompile.transfer(victimAddress, attackerAddress, 1_000_000)
   The bank precompile executes:
       bankKeeper.SendCoins(ctx, victimAddress, attackerAddress, [1_000_000 evm/0xDEAD])
   No approval from victimAddress is checked. The transfer succeeds.

5. Victim's evm/0xDEAD balance is now 0. Attacker holds 1_000_000 evm/0xDEAD.

Alternatively, step 4 can use burn(victimAddress, 1_000_000) to permanently
destroy the victim's tokens instead of stealing them.
```
