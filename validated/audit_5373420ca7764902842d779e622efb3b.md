### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions After `deploy_account` Rejection Due to Stale `account_tx_in_pool_or_recent_block` Check - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry-point call for an invoke transaction with `nonce == 1` when the account's on-chain nonce is still `0`, provided a `deploy_account` transaction is believed to be pending. The guard for this belief is `account_tx_in_pool_or_recent_block`, which returns `true` if **any** transaction from the account address is in the mempool pool or in a recent committed block — not specifically a `deploy_account`. After a `deploy_account` transaction is rejected by the blockifier, the rewound invoke transaction (nonce=1) remains in the pool, causing `account_tx_in_pool_or_recent_block` to continue returning `true`. A subsequent invoke transaction with a forged signature and a higher fee (fee escalation) is then admitted to the mempool without `__validate__` being called, violating the admission invariant.

### Finding Description

**Root cause — `skip_stateful_validations` trusts a non-specific pool membership check:** [1](#0-0) 

The function checks `tx.nonce() == 1 && account_nonce == 0`, then delegates to `account_tx_in_pool_or_recent_block`. The inline comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This claim is false after a `deploy_account` rejection.

**`account_tx_in_pool_or_recent_block` is not deploy-account-specific:** [2](#0-1) 

It returns `true` if `state.contains_account` (staged or committed) **or** `tx_pool.contains_account`. After `deploy_account` rejection, the rewound invoke transaction keeps `tx_pool.contains_account` true.

**`MempoolState::contains_account` only checks staged/committed maps:** [3](#0-2) 

After `commit_block` with the `deploy_account` in `rejected_tx_hashes` and no committed nonce for the account, `staged` is cleared and `committed` is not updated, so `state.contains_account` returns `false`. But `tx_pool.contains_account` remains `true` because the invoke transaction is rewound.

**`commit_block` rewinds the invoke tx but removes the deploy_account:** [4](#0-3) 

The `state.commit` call clears `staged` and returns the account address as needing rewind. `remove_rejected_txs` removes the `deploy_account` from the pool. The invoke transaction (nonce=1) is rewound back into the pool, not removed.

**`run_validate_entry_point` skips `__validate__` when `skip_validate=true`:** [5](#0-4) 

When `skip_validate=true`, `execution_flags.validate = false` is set, and `StatefulValidator::perform_validations` returns early without calling `validate_tx` (the `__validate__` entry point). [6](#0-5) 

**Attack sequence:**

1. Attacker submits `deploy_account` (nonce=0) crafted to fail at execution (e.g., invalid constructor calldata or non-existent class hash).
2. Attacker submits invoke A (nonce=1, valid signature) → `skip_stateful_validations` returns `true` (deploy_account in pool) → admitted without `__validate__`.
3. Batcher stages both; `deploy_account` fails → rejected; invoke A is rewound to pool.
4. `commit_block` called: `deploy_account` removed from pool; invoke A rewound; `account_tx_in_pool_or_recent_block` still returns `true` (invoke A in pool).
5. Attacker submits invoke B (nonce=1, **forged signature**, higher fee for fee escalation) → `validate_by_mempool` passes (fee escalation replaces invoke A) → `skip_stateful_validations` returns `true` (invoke A still in pool at check time) → `__validate__` NOT called → invoke B admitted.
6. Invoke B is in the mempool with an unverified forged signature.

### Impact Explanation

An invoke transaction with a forged (or entirely absent) signature is admitted to the mempool without the account's `__validate__` entry point being called. This violates the gateway admission invariant that every invoke transaction from an undeployed account must either have its signature verified or have a confirmed pending `deploy_account`. The transaction will fail at execution time (account does not exist), but it occupies mempool capacity and consumes batcher execution resources. An attacker can repeat this cycle indefinitely to mount a resource-exhaustion attack against the sequencer's mempool and batcher, and can do so with zero valid cryptographic material.

### Likelihood Explanation

The trigger requires only two sequential RPC calls (submit `deploy_account` crafted to fail, then submit invoke with nonce=1) followed by waiting one block for the rejection. No privileged access is needed. The `deploy_account` can be made to fail deterministically by the attacker (e.g., using a class hash that is not declared). The fee escalation path is a standard mempool feature. The condition `tx.nonce() == 1 && account_nonce == 0` is always satisfied for a fresh account whose `deploy_account` was rejected.

### Recommendation

Replace the non-specific `account_tx_in_pool_or_recent_block` check in `skip_stateful_validations` with a check that specifically verifies a `deploy_account` transaction for the sender address is present in the mempool pool or was recently committed. The mempool should expose a dedicated `deploy_account_in_pool_or_recent_block(address)` query that inspects transaction types, or the gateway should track the deploy_account transaction hash explicitly (as the `native_blockifier` `PyValidator` already does via `deploy_account_tx_hash`). [7](#0-6) 

The `PyValidator::should_run_stateful_validations` correctly requires an explicit `deploy_account_tx_hash` parameter and checks `nonce == Nonce(Felt::ZERO)` on the account. The new gateway path should adopt the same explicit hash-based check rather than relying on the generic pool membership predicate.

### Proof of Concept

```
// Step 1: submit deploy_account crafted to fail (e.g., class_hash not declared)
POST /gateway/add_transaction  { type: DEPLOY_ACCOUNT, class_hash: 0xDEAD, nonce: 0, ... }
// → accepted, tx_hash = H_deploy

// Step 2: submit invoke A (nonce=1, valid sig) — skip_validate fires because H_deploy is in pool
POST /gateway/add_transaction  { type: INVOKE, nonce: 1, sender: A, signature: [valid], ... }
// → accepted (skip_validate=true, __validate__ not called)

// Step 3: wait for next block — deploy_account fails, invoke A rewound to pool
// account_nonce(A) is still 0; tx_pool.contains_account(A) = true (invoke A rewound)

// Step 4: submit invoke B (nonce=1, FORGED sig, higher fee — fee escalation replaces invoke A)
POST /gateway/add_transaction  { type: INVOKE, nonce: 1, sender: A, signature: [0x0, 0x0], tip: higher, ... }
// → validate_by_mempool passes (fee escalation)
// → skip_stateful_validations: account_nonce==0, tx.nonce==1, account_tx_in_pool_or_recent_block==true → skip=true
// → __validate__ NOT called
// → invoke B admitted to mempool with forged signature

// Repeat from Step 1 indefinitely for sustained mempool pollution / DoS
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
