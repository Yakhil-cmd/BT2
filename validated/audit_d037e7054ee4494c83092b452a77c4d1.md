### Title
Attached NEAR Deposit in `execute_scheduled` Is Permanently Trapped in XCC Router Contract — (`File: etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC Router contract is marked `#[payable]`, meaning it silently accepts an attached NEAR deposit from any caller. However, the function never reads or forwards `env::attached_deposit()`. Any NEAR attached to this call is absorbed into the router sub-account's balance with no recovery path available to the depositor, and no general withdrawal mechanism exists in the contract.

---

### Finding Description

The XCC Router contract exposes `execute_scheduled` as a public, permissionless function that anyone on NEAR can call to trigger execution of a previously scheduled cross-contract promise:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The `#[payable]` attribute causes NEAR to be accepted without panicking. However, `env::attached_deposit()` is never called inside the function body. The NEAR attached to the call is not forwarded to the created promise — the promise's NEAR budget comes exclusively from the stored `PromiseCreateArgs::attached_balance` field, which is funded separately via the wNEAR-unwrapping flow initiated by the Aurora Engine:

```rust
fn base_promise_create(promise: &PromiseCreateArgs) -> PromiseIndex {
    env::promise_create(
        promise.target_account_id.as_ref().parse().unwrap(),
        promise.method.as_str(),
        &promise.args,
        NearToken::from_yoctonear(promise.attached_balance.as_u128()),  // ← from stored args, not attached_deposit
        Gas::from_gas(promise.attached_gas.as_u64()),
    )
}
```

The router contract exposes no general withdrawal function. The only NEAR-moving method available to the parent is `send_refund`, which unconditionally transfers a hardcoded constant (`REFUND_AMOUNT = 2 NEAR`) and is only callable as a callback from the parent:

```rust
pub fn send_refund(&self) -> Promise {
    let parent = self.get_parent().unwrap_or_else(env_panic);
    require_caller(&parent)
        .and_then(|_| require_no_failed_promises())
        .unwrap_or_else(env_panic);
    Promise::new(parent).transfer(REFUND_AMOUNT)
}
```

This means any NEAR attached to `execute_scheduled` is absorbed into the router sub-account balance with no automatic or user-accessible refund path.

---

### Impact Explanation

Any NEAR tokens attached to a call to `execute_scheduled` are permanently trapped in the router sub-account. The router has no general withdrawal function. Recovery requires the Aurora Engine owner to:
1. Detect the stuck balance,
2. Schedule a recovery promise via the privileged `schedule` method,
3. Have someone call `execute_scheduled` again with the new nonce.

This is a multi-step, admin-gated recovery process. Until that intervention occurs, the funds are frozen. If the admin does not act, the freeze is permanent. This maps to **temporary freezing of funds (High)** at minimum, and **permanent freezing of funds (Critical)** if no recovery action is taken.

---

### Likelihood Explanation

`execute_scheduled` is intentionally open to any NEAR caller (the code comment confirms this). A NEAR user who wants to trigger execution of a scheduled XCC promise and mistakenly attaches a deposit — a natural mistake given that many NEAR contract calls require attached NEAR — will silently lose those tokens. Additionally, a malicious actor can grief any router sub-account by intentionally attaching NEAR to inflate its balance in a way that cannot be recovered without admin intervention. The function is reachable by any unprivileged NEAR account.

---

### Recommendation

Remove the `#[payable]` attribute from `execute_scheduled`. The function has no legitimate use for an attached deposit; the NEAR budget for the executed promise is sourced from the router's own balance via the wNEAR-unwrapping flow, not from the caller's deposit. Removing `#[payable]` will cause NEAR runtime to panic and reject any call that attaches a deposit, preventing accidental or malicious fund loss.

### Proof of Concept

1. Alice is a NEAR user. She observes that a scheduled promise exists at nonce `42` in her XCC router sub-account (`alice_evm_addr.aurora`).
2. She calls `execute_scheduled({ nonce: 42 })` and accidentally attaches `5 NEAR` (a common pattern when calling NEAR contracts that require storage deposits).
3. The call succeeds. The scheduled promise is executed using the router's own wNEAR-derived balance. Alice's `5 NEAR` is absorbed into the router sub-account.
4. Alice has no way to recover her `5 NEAR`. The router exposes no withdrawal function. `send_refund` only sends a fixed `2 NEAR` and is only callable by the Aurora Engine as a callback.
5. Recovery requires the Aurora Engine operator to notice the discrepancy, schedule a recovery promise via the privileged `schedule` method, and have someone call `execute_scheduled` again — a process entirely outside Alice's control. [1](#0-0) [2](#0-1) [3](#0-2)
