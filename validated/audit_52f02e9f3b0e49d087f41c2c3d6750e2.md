Looking at the `tick()` function in `L1/starkware/solidity/stake/RewardSupplier.sol`, I can identify a direct analog to M-08's token-freeze vulnerability class.

---

### Title
ETH Permanently Trapped in `RewardSupplier.tick()` When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
`RewardSupplier.tick()` is `payable` and accepts ETH for L1→L2 message fees. When `amountToMint == 0` (no pending mint requests), the function exits the `if` block without consuming or refunding `msg.value`. Any ETH sent by the caller is permanently locked in the contract with no recovery path.

### Finding Description
The `tick()` function sends two L1→L2 messages when minting is required, splitting `msg.value` between them:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // uses half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // uses other half
    }
    // ← if amountToMint == 0, msg.value is silently discarded
}
```

When `amountToMint == 0`, the entire `msg.value` is never forwarded, refunded, or tracked. The contract has no `receive()`, `fallback()`, or `withdraw()` function to recover stranded ETH.

The parallel to M-08 is exact: M-08's `_payNative()` returned only `_nativeFee` instead of `msg.value`, trapping the excess. Here, `tick()` uses `msg.value` only inside the `if (amountToMint > 0)` branch, trapping the full amount when the branch is not entered.

A realistic attack path exploiting this:
1. Caller A observes pending L2→L1 mint-request messages and prepares a `tick()` call with sufficient ETH.
2. An attacker front-runs with their own `tick()` call (minimal ETH), consuming all pending messages so `requiredMinting()` now returns `(0, 0)`.
3. Caller A's transaction executes; `amountToMint == 0`, the `if` block is skipped, and Caller A's ETH is permanently trapped.

No privileged role is required. Any public caller can trigger this by calling `tick{value: X}()` when no minting is pending, whether by mistake or via front-running.

### Impact Explanation
ETH sent to `tick()` when `amountToMint == 0` is permanently frozen in the `RewardSupplier` contract. There is no withdrawal function and no refund path. This constitutes griefing with direct financial damage to callers — matching the **Medium** impact tier: *griefing with no profit motive but damage to users or protocol*.

### Likelihood Explanation
`tick()` is a public, permissionless function intended to be called by bots or keepers. A race condition between `requiredMinting()` view call and the actual `tick()` execution is realistic on L1 (block-level front-running). Additionally, callers who do not pre-check `requiredMinting()` will lose ETH unconditionally. Likelihood is **Medium**.

### Recommendation
Refund unused ETH when no minting occurs:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund ETH if no minting is needed
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, remove `payable` from the no-op path by splitting into two functions, or add a `require(amountToMint > 0 || msg.value == 0)` guard.

### Proof of Concept

1. Deploy `RewardSupplier` with no pending L2→L1 mint-request messages (or consume them first).
2. Call `tick{value: 1 ether}()`.
3. `requiredMinting()` returns `(0, 0)` → `amountToMint == 0`.
4. The `if` block is skipped; 1 ETH is now held by the contract.
5. Confirm: `address(rewardSupplier).balance == 1 ether` with no way to recover it. [1](#0-0) [2](#0-1) [3](#0-2)

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.

                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.

                new uint256[](0)
