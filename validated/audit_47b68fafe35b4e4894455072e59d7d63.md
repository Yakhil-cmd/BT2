### Title
CKD Coordinator Blindly Aggregates Unverified Participant Shares, Allowing Any Malicious Participant to Corrupt the CKD Output — (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` receives `(norm_big_y, norm_big_c)` from every participant and adds them together with no ZK proof or consistency check. The ZK proof infrastructure (`dlog.rs`, `dlogeq.rs`) is explicitly present in the codebase and is enforced in every other multi-party protocol (DKG, OT-based ECDSA triple generation), but is never imported or applied in the CKD protocol. This is the direct analog of the Pausable bug: a security mechanism exists in the project but is never wired into the CKD path, leaving the coordinator unable to detect a malicious participant who submits arbitrary group elements.

### Finding Description

**Root cause — `do_ckd_coordinator`, lines 50–55:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is expected to compute and send:
- `norm_big_y = λ_i · (y_i · G)` — a blinding commitment
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` — the ElGamal-encrypted key share

The coordinator sums these values to reconstruct the encrypted confidential key. No proof is requested or verified. There is no check that `big_y` is a valid group element of the correct form, no dlog proof that the sender knows the discrete log of `big_y`, and no dlogeq proof that the same scalar `y_i` was used in both `big_y` and `big_c`, nor that `x_i` is the participant's actual key share.

**Contrast with other protocols:**

- DKG (`src/dkg.rs`, lines 452–460): calls `verify_proof_of_knowledge` for every participant before accepting any commitment.
- OT-based ECDSA triple generation (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`, lines 368–389, 483–491, 596–604): calls `dlog::verify` and `dlogeq::verify` for every received value before aggregating.
- CKD (`src/confidential_key_derivation/protocol.rs`, lines 50–55): no verification whatsoever.

The `dlog` and `dlogeq` modules are already compiled into the same crate and used elsewhere; they are simply never called from the CKD path.

**Attacker-controlled entry path:**

A malicious participant calls `ckd(...)` with a valid participant list and key pair, but inside `do_ckd_participant` (line 30) sends a crafted `(norm_big_y', norm_big_c')` — for example, the identity element, a random point, or a point designed to bias the output — instead of the honestly computed values. The coordinator receives this over the private channel and adds it unconditionally.

### Impact Explanation

**High.** The coordinator is the sole recipient of the final `CKDOutput`. A single malicious participant can replace their share with any pair of group elements, causing the aggregated `(norm_big_y, norm_big_c)` to be an arbitrary, attacker-chosen ciphertext. The coordinator then returns this corrupted value as the CKD result. Downstream, `unmask(app_sk)` will produce a wrong group element that does not equal `msk · H(pk, app_id)`, so the derived confidential key is invalid. This maps directly to: *"Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs."*

### Likelihood Explanation

**High.** Any single participant in the CKD session can trigger this. No special privilege, leaked key, or cryptographic break is required. The participant simply deviates from the protocol by sending arbitrary bytes that deserialize as a valid `CKDOutput`. The coordinator has no mechanism to detect or reject the deviation.

### Recommendation

Apply the same ZK-proof pattern already used in OT-based ECDSA triple generation:

1. Each participant generates a dlog proof `π_y` for `big_y = y_i · G` (proving knowledge of `y_i`).
2. Each participant generates a dlogeq proof `π_c` proving that the same `y_i` appears in `big_c = x_i · H(pk, app_id) + y_i · app_pk` (i.e., `y_i · G = big_y` and `y_i · app_pk` is consistent).
3. The coordinator verifies both proofs for every received share before aggregating, aborting on any failure.

Alternatively, if the threat model explicitly excludes malicious participants, this assumption must be documented clearly in the API and security policy.

### Proof of Concept

```
Setup: 3 participants P1, P2, P3 (honest), P_m (malicious), threshold = 2.
All run ckd(...) with the same (participants, coordinator, key_pair, app_id, app_pk).

P_m deviates in do_ckd_participant:
  Instead of computing (norm_big_y, norm_big_c) via compute_signature_share,
  P_m sends (ElementG1::identity(), ElementG1::identity()).

Coordinator (do_ckd_coordinator, lines 50-55):
  Receives honest shares from P1, P2, P3 and the zeroed share from P_m.
  Adds all four without verification.
  norm_big_y = Σ honest_big_y_i + identity  ← wrong
  norm_big_c = Σ honest_big_c_i + identity  ← wrong

Result:
  ckd_output.unmask(app_sk) ≠ msk · H(pk, app_id)
  The derived confidential key is invalid; the CKD protocol output is corrupted.
```
