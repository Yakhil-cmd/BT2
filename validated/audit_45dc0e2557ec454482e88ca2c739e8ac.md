### Title
Unbounded Loop in `calculate_rewards` Over `pool_member_epoch_balance` Trace Can Permanently Freeze Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded `while` loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. A pool member who makes repeated balance changes across many epochs without claiming rewards will accumulate an arbitrarily large trace. When `claim_rewards` is eventually called, the loop can exceed Starknet's per-transaction gas limit, causing the transaction to revert without updating `entry_to_claim_from`, permanently freezing the pool member's unclaimed yield.

### Finding Description
`calculate_rewards` iterates over the pool member's balance-change trace from `entry_to_claim_from` up to `pool_member_trace_length`: [1](#0-0) 

The developer comment at line 857–858 explicitly acknowledges the loop is unbounded. Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` inserts a new entry into `pool_member_epoch_balance` via `set_member_balance`: [2](#0-1) 

The key is `get_epoch_plus_k()` (current epoch + K), so one new trace entry is created per epoch in which the pool member changes their balance. The `entry_to_claim_from` field in `InternalPoolMemberInfoV1` tracks the starting index for the next claim: [3](#0-2) 

Critically, `entry_to_claim_from` is only written back to storage if `claim_rewards` succeeds: [4](#0-3) 

If the loop causes the transaction to revert (gas exhaustion), `entry_to_claim_from` is never advanced. Every subsequent `claim_rewards` attempt restarts from the same position and hits the same gas wall, permanently locking the pool member's accumulated rewards.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace grows large enough that a single `claim_rewards` call exceeds the Starknet transaction gas limit, the pool member's unclaimed STRK rewards become permanently inaccessible. The state cannot be unwound: `entry_to_claim_from` is never advanced because the transaction always reverts before the storage write at line 358.

### Likelihood Explanation
**Low-Medium.** The pool member must make at least one balance change per epoch (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) across many consecutive epochs without ever calling `claim_rewards`. Only the pool member or their designated `reward_address` can trigger balance changes or claims, so this is self-inflicted rather than externally forced. However, a pool member who automates frequent small top-ups (e.g., a bot compounding delegations) and whose reward-claiming logic fails silently could accumulate thousands of trace entries over months, eventually hitting the gas ceiling. The protocol provides no safeguard or warning.

### Recommendation
1. **Paginate `claim_rewards`**: Accept an optional `max_entries` parameter so the loop processes a bounded number of trace entries per call, advancing `entry_to_claim_from` incrementally across multiple transactions.
2. **Alternatively**, enforce a maximum number of unclaimed balance-change entries before allowing further `add_to_delegation_pool` calls, forcing the pool member to claim first.
3. **At minimum**, document the risk prominently and recommend that integrators call `claim_rewards` before each balance change.

### Proof of Concept
1. Pool member `A` enters the delegation pool via `enter_delegation_pool`.
2. Each epoch, `A` calls `add_to_delegation_pool` with a small amount. Each call invokes `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`, appending one entry to `pool_member_epoch_balance[A]`.
3. `A` never calls `claim_rewards`, so `entry_to_claim_from` remains at `0`.
4. After `N` epochs of balance changes, `pool_member_trace_length == N`.
5. `A` calls `claim_rewards`. The loop at line 859 iterates from `0` to `N`, calling `find_sigma` (multiple storage reads) on each iteration.
6. For sufficiently large `N`, the transaction exceeds the Starknet gas limit and reverts.
7. `entry_to_claim_from` is never updated (line 358 is never reached). Every future `claim_rewards` call repeats the same failing loop. `A`'s accumulated STRK rewards are permanently frozen. [5](#0-4) [6](#0-5)
