### Title
Unchecked Return Value of Low-Level Precompile Call in `EvmErc20` Allows Silent Token Burn Without Transfer - (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20::withdrawToNear` and `EvmErc20::withdrawToEthereum` burn the caller's tokens first, then invoke the Aurora exit precompile via a low-level assembly `call`. The return value of that `call` is captured in `res` but never checked. If the precompile call fails with an `ExitError` (sub-call failure, not a fatal error), the EVM returns `0` to the caller, the burn is not reverted, and the function returns successfully — permanently destroying the user's tokens with no corresponding NEAR or Ethereum transfer.

---

### Finding Description

In `etc/eth-contracts/contracts/EvmErc20.sol`, both withdrawal functions follow the same unsafe pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [1](#0-0) 

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is never checked
    }
}
``` [2](#0-1) 

The EVM `call` opcode returns `0` on sub-call failure and `1` on success. When the exit precompile returns an `ExitError` (not `ExitFatal`), the EVM treats it as a sub-call failure: it returns `0` to the caller and does **not** revert the outer call's state changes. Because `_burn` executes in the outer call frame before the sub-call, the burn is committed even when the precompile fails.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) has multiple code paths that return `ExitError` (not `ExitFatal`):

- `ExitError::OutOfGas` — if the gas forwarded to the sub-call is less than `EXIT_TO_NEAR_GAS`
- `ExitError::Other("ERR_GETTING_ERC20_FROM_NEP141")` — if the ERC-20 address is not registered in the NEP-141 mapping
- `ExitError::Other("ERR_INVALID_IN_DELEGATE")` — if called in delegate context
- `ExitError::Other("ERR_INVALID_MESSAGE")` — on malformed input [3](#0-2) 

Contrast this with the pausing path, which returns `PrecompileFailure::Fatal` (causing the entire transaction to fail and reverting the burn — that path is safe): [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

When the precompile sub-call fails with `ExitError`, the sequence is:

1. `_burn` destroys the user's ERC-20 tokens (state committed in outer frame).
2. Sub-call to exit precompile fails → EVM returns `0` to `EvmErc20`.
3. `res` is never checked; `withdrawToNear`/`withdrawToEthereum` returns normally.
4. No NEAR promise or Ethereum withdrawal is scheduled.
5. The user's tokens are permanently gone with no corresponding asset transfer.

The tokens are not locked in a contract — they are burned. There is no recovery path.

---

### Likelihood Explanation

**Medium.** The most realistic trigger is gas exhaustion: a user who provides a transaction gas limit that is sufficient for the `_burn` but insufficient for the precompile's `EXIT_TO_NEAR_GAS` requirement. The `call` forwards all remaining gas via `gas()`, but if the remaining gas at that point is below the precompile's required threshold, the precompile returns `ExitError::OutOfGas`. This is an ordinary user mistake (underestimating gas) that silently destroys funds instead of reverting the transaction. A second realistic path is any future state where the NEP-141 mapping for a given ERC-20 is absent or corrupted, causing `ERR_GETTING_ERC20_FROM_NEP141`.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is `0`, mirroring the pattern already used correctly in the XCC precompile: [5](#0-4) 

Apply the fix to both functions in `EvmErc20.sol`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }
    }
}
```

Apply the same `if iszero(res) { revert(0, 0) }` guard to `withdrawToEthereum`. Alternatively, restructure both functions to call the precompile **before** burning, so that a precompile failure reverts the entire call without any tokens being destroyed.

---

### Proof of Concept

1. Deploy `EvmErc20` with a NEP-141 mirror registered.
2. Mint tokens to `user`.
3. Call `withdrawToNear(recipient, amount)` with a gas limit carefully set so that after `_burn` executes, the remaining gas forwarded to the precompile sub-call is below `EXIT_TO_NEAR_GAS`.
4. The precompile returns `ExitError::OutOfGas`; the assembly `call` returns `0`; `res` is ignored.
5. The transaction succeeds (no revert).
6. Observe: `user`'s token balance is reduced by `amount`, but no NEAR transfer promise was scheduled and no tokens arrive at `recipient`. [6](#0-5) [7](#0-6)
