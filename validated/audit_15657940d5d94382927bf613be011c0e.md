### Title
Unbounded Growth of `stakers` Vec Causes Unbounded Gas Consumption in `get_stakers()` - (File: src/staking/staking.cairo)

---

### Summary

The `stakers` storage vector in `staking.cairo` is append-only: every call to `stake()` pushes a new address, but `unstake_action()` never removes entries. The `get_stakers()` function iterates over the **entire** vector on every call. An unprivileged attacker can register many staker addresses (recycling the same STRK capital through different addresses after each exit), permanently bloating the vector and making `get_stakers()` increasingly expensive until it exceeds Starknet's execution step limits.

---

### Finding Description

**Root cause — append-only `stakers` Vec:**

The storage declaration explicitly documents the invariant:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

Every successful `stake()` call appends to this vector:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [2](#0-1) 

`unstake_action()` calls `remove_staker()` which deletes the staker's `staker_info` map entry and transfers funds, but **never removes the address from `stakers`**: [3](#0-2) 

**Root cause — full linear scan in `get_stakers()`:**

`get_stakers()` iterates over `self.stakers.into_iter_full_range()` — the complete vector including all exited stakers — on every invocation:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [4](#0-3) 

Exited stakers are skipped via `is_staker_active`, but their addresses are still read from storage on every call, making the cost O(total stakers ever registered), not O(currently active stakers).

**No cap on staker count:**

Unlike the Ditto protocol's remediation (`max_n_validators`), there is no maximum staker limit in this codebase. The `stake()` function is permissionless — any address meeting `min_stake` may register: [5](#0-4) 

`assert_staker_address_not_reused` only prevents the *same* address from re-staking after exit; it does not prevent a new address from staking.

---

### Impact Explanation

`get_stakers()` is the `IStakingConsensus` interface function used by the consensus layer to determine the active validator set for each epoch: [6](#0-5) 

As the `stakers` vector grows without bound, each call to `get_stakers()` reads more storage slots and executes more loop iterations. On Starknet, both on-chain calls and off-chain `call` invocations are subject to execution step limits. Once the vector is large enough, `get_stakers()` will exceed those limits and revert, permanently preventing the consensus layer from reading the validator set.

**Impact: Medium — Unbounded gas consumption / griefing with damage to the protocol.**

---

### Likelihood Explanation

The attack is economically feasible because STRK capital is **fully recycled**:

1. Attacker stakes `min_stake` STRK from address A → address A is pushed to `stakers`.
2. Attacker calls `unstake_intent()` from address A.
3. After `exit_wait_window`, attacker calls `unstake_action()` → full `min_stake` STRK returned.
4. Attacker transfers STRK to address B and repeats.

The only cost per iteration is gas fees. The attacker needs only one `min_stake` worth of STRK at any time. Given that Starknet gas fees are low, a sustained campaign can bloat the vector to tens of thousands of entries over weeks, with each entry permanently increasing the cost of `get_stakers()`.

---

### Recommendation

1. **Impose a hard cap** on the total number of registered stakers (analogous to Ditto's `max_n_validators`).
2. **Remove exited stakers from the `stakers` vector** (swap-and-pop pattern), or maintain a separate active-staker set that is pruned on exit.
3. **Alternatively**, replace the append-only `Vec` with an `IterableMap` that supports deletion, so `get_stakers()` only iterates over currently active stakers.

---

### Proof of Concept

```
// Setup: attacker has min_stake STRK and N fresh addresses addr_1..addr_N

for i in 1..N:
    // Transfer min_stake STRK to addr_i
    strk.transfer(to: addr_i, amount: min_stake)

    // addr_i stakes → stakers.push(addr_i), vector length = i
    staking.stake(reward_address: addr_i, operational_address: op_i, amount: min_stake)
      // called as addr_i

    // addr_i signals exit
    staking.unstake_intent()  // called as addr_i

    // wait exit_wait_window blocks/time

    // addr_i exits, STRK returned to addr_i
    staking.unstake_action(staker_address: addr_i)  // called as addr_i

    // Transfer STRK back to attacker for next iteration
    strk.transfer(from: addr_i, to: attacker, amount: min_stake)

// After N iterations:
// stakers.len() == N (all N addresses remain, all inactive)
// get_stakers(epoch_id) now reads N storage slots and runs N loop iterations
// For large enough N, get_stakers() exceeds Starknet step limits and reverts
```

The `stakers` vector length after the attack equals the total number of unique addresses ever staked, regardless of how many are currently active. The consensus layer's call to `get_stakers()` will eventually fail.
