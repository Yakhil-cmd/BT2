### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Victim Accounts via Mempool Presence Check - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway bypasses the `__validate__` entry-point (signature check) for any invoke transaction with `nonce=1` when the sender address appears in the mempool — regardless of who submitted the mempool entry. An unprivileged attacker who observes a victim's pending `deploy_account` transaction can inject a malicious invoke transaction for the victim's account with an invalid signature, have it admitted to the mempool without signature verification, and thereby block the victim's legitimate first post-deploy invoke.

---

### Finding Description

The `skip_stateful_validations` function is intended to improve UX for the common pattern of submitting `deploy_account` + `invoke` simultaneously. It skips the blockifier's `__validate__` entry point for an invoke transaction when three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, and `StatefulValidator::perform_validations` returns early without ever calling `__validate__`: [2](#0-1) [3](#0-2) 

The authorization flaw is in condition 3. `account_tx_in_pool_or_recent_block` checks whether the address has **any** transaction in the pool or was recently committed — it does **not** verify that the mempool entry is a `deploy_account` transaction, nor that the invoke submitter is the same party who submitted the `deploy_account`: [4](#0-3) 

The comment in `skip_stateful_validations` asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect for the adversarial case: a third party can place the victim's address into the mempool by submitting the victim's `deploy_account` transaction (which is public once broadcast), and then immediately submit a malicious invoke for the same address with an arbitrary/empty signature.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee escalation — it does not verify the signature: [5](#0-4) [6](#0-5) 

---

### Impact Explanation

**Matching impact category: High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A transaction with an invalid (or empty) signature is admitted to the mempool for an account the attacker does not control. The concrete effects are:

1. The victim's legitimate `invoke` with `nonce=1` is rejected by the mempool with `DuplicateNonce` (or requires fee escalation to displace the attacker's entry).
2. The attacker's transaction will eventually fail during batcher execution (because `AccountTransaction::new_for_sequencing` sets `validate: true`, so `__validate__` is called at execution time), but the damage — blocking the victim's transaction — has already occurred. [7](#0-6) 

---

### Likelihood Explanation

- The victim's `deploy_account` transaction is publicly visible in the mempool the moment it is broadcast.
- The attacker only needs to submit an invoke for the victim's address with `nonce=1` before the victim does — a straightforward front-run.
- No privileged access, special keys, or knowledge of the victim's private key is required.
- The attack is repeatable: every new account deployment is a fresh opportunity.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`deploy_account` transaction** exists in the mempool for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, rather than returning true for any transaction type.

Alternatively, the gateway can require that the invoke transaction carry a proof-of-knowledge of the deploy parameters (e.g., the same `contract_address_salt` and `class_hash`), so that only the party who controls the deployment can submit the paired invoke.

---

### Proof of Concept

```
1. Victim broadcasts: deploy_account(class_hash=C, salt=S, ...) → deploys to address A, nonce=0
   → Mempool now contains address A (tx_pool.contains_account(A) = true)

2. Attacker submits: invoke(sender=A, nonce=1, calldata=<malicious>, signature=<empty>)
   Gateway checks:
     - tx_nonce=1, account_nonce=0 → within nonce gap, passes validate_nonce
     - validate_by_mempool: nonce 1 >= 0, no duplicate → passes
     - skip_stateful_validations: nonce==1, account_nonce==0,
       account_tx_in_pool_or_recent_block(A) == true → returns true (SKIP)
     - run_validate_entry_point called with skip_validate=true → __validate__ NOT called
   → Malicious invoke admitted to mempool with invalid signature

3. Victim submits: invoke(sender=A, nonce=1, calldata=<legitimate>, signature=<valid>)
   → Mempool rejects: DuplicateNonce { address: A, nonce: 1 }

4. Batcher executes deploy_account(A) then malicious invoke(A, nonce=1):
   → __validate__ called with validate=true → signature check fails → tx rejected
   → Mempool notified, malicious tx removed

5. Victim must resubmit their invoke — the deploy_account + invoke atomic UX flow is broken.
```

The root cause is identical in structure to the external report: a privileged action (submitting an invoke on behalf of an account) can be performed by an unauthorized third party because the authorization check (`account_tx_in_pool_or_recent_block`) does not verify the identity of the submitter relative to the account owner. [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
