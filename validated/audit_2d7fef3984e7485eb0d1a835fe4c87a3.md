### Title
Missing Mandatory Chain ID Validation for Pre-EIP-155 Legacy Transactions Enables Cross-Chain Replay - (File: engine/src/engine.rs)

### Summary
The `submit_with_alt_modexp` function in `engine/src/engine.rs` validates the chain ID only when it is present in the transaction signature. Legacy (pre-EIP-155) transactions with `v = 27` or `v = 28` produce `chain_id = None` and bypass the chain ID check entirely. This allows an attacker to replay a pre-EIP-155 Ethereum transaction on Aurora, or vice versa, against any victim whose Aurora nonce matches the replayed transaction's nonce.

### Finding Description

In `engine/src/engine.rs`, the chain ID check is guarded by an `if let Some(...)` pattern:

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

When a `LegacyEthSignedTransaction` is decoded, `chain_id()` returns `None` for any transaction where `v ∈ {27, 28}` (the pre-EIP-155 encoding):

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

This `None` propagates through `TryFrom<EthTransactionKind> for NormalizedEthTransaction`:

```rust
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v=27/28
    ...
}
``` [3](#0-2) 

The `CHANGES.md` confirms this is intentional — pre-EIP-155 transactions were re-enabled in v2.6.0 for EIP-1820 compatibility:

> "Original ETH transactions which do not contain a Chain ID are allowed again to allow for use of EIP-1820" [4](#0-3) 

The public entry point is the `submit` NEAR contract method, callable by any unprivileged user: [5](#0-4) 

### Impact Explanation

An attacker who finds a pre-EIP-155 signed transaction from a victim's Ethereum address (publicly available on-chain) can submit it to Aurora's `submit` endpoint. If the victim's Aurora account nonce matches the replayed transaction's nonce, the transaction executes on Aurora as if the victim signed it. If the replayed transaction transfers ETH or ERC-20 tokens to an attacker-controlled address, or calls a contract that benefits the attacker, the victim's Aurora funds are stolen. The impact is **direct theft of user funds** (Critical).

### Likelihood Explanation

Pre-EIP-155 transactions are permanently recorded on Ethereum mainnet and are publicly accessible. Many Aurora users share the same address as their Ethereum address. For a fresh Aurora account (nonce = 0), any pre-EIP-155 transaction from that address with nonce = 0 on Ethereum is a candidate. The constraint is that the replayed transaction must be beneficial to the attacker (e.g., a token transfer or approval to an attacker-controlled address). This limits likelihood to **Low-to-Medium**: the attacker must identify a victim with matching nonce and a beneficial pre-EIP-155 transaction in Ethereum history, but the attack requires no special privileges and the data is public.

### Recommendation

Reject legacy transactions that omit the chain ID (`v = 27` or `v = 28`) unless a specific allowlist of known pre-EIP-155 contract deployment transactions (e.g., the EIP-1820 deployer transaction) is maintained. Alternatively, enforce that `transaction.chain_id` must be `Some` and must match `state.chain_id` unconditionally in `submit_with_alt_modexp`, and handle EIP-1820 deployment through a separate privileged path.

### Proof of Concept

1. Identify a victim address `V` that has ETH on Aurora and has previously signed a pre-EIP-155 transaction on Ethereum mainnet with nonce `N` that transfers ETH to address `A` (attacker-controlled).
2. Confirm that `V`'s current Aurora nonce is also `N`.
3. RLP-encode the pre-EIP-155 transaction (with `v = 27` or `v = 28`).
4. Call Aurora's `submit` NEAR method with the encoded bytes as input.
5. The chain ID check at `engine/src/engine.rs:1055` is skipped because `transaction.chain_id` is `None`.
6. The transaction executes: ETH is transferred from `V` to `A` on Aurora. [1](#0-0) [6](#0-5) [3](#0-2)
