### Title
`skip_stateful_validations` Bypasses `__validate__` for Contracts Deployed via `deploy` Syscall — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function skips the blockifier `__validate__` entry-point call for any invoke transaction with `tx.nonce() == 1` and `account_nonce == 0` when `account_tx_in_pool_or_recent_block` returns `true`. The check is designed only for the deploy-account + invoke UX shortcut, but it fires equally for contracts deployed via the `deploy` syscall (not `deploy_account`). For those contracts the account already exists on-chain, `__validate__` is callable and checks the signature against the transaction hash, yet the gateway skips it entirely. An attacker who controls such a contract can submit a nonce-1 invoke with an arbitrary (invalid) signature and have it admitted to the mempool without any signature verification.

### Finding Description

`extract_state_nonce_and_run_validations` orchestrates gateway stateful validation in three steps:

1. Fetch `account_nonce` from the committed-block state reader.
2. Call `run_pre_validation_checks` (nonce range, resource bounds, mempool duplicate check, and the skip decision).
3. Call `run_validate_entry_point` with `validate = !skip_validate`. [1](#0-0) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

```
tx is Invoke  AND  tx.nonce() == 1  AND  account_nonce == 0
AND  mempool_client.account_tx_in_pool_or_recent_block(sender) == true
``` [2](#0-1) 

The comment justifies the skip: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

`account_tx_in_pool_or_recent_block` checks whether the sender address appears in the mempool pool **or** in the committed-block account state — it does not filter by transaction type: [3](#0-2) 

The flaw: a contract deployed via the `deploy` syscall (not `deploy_account`) has `account_nonce == 0` in the committed state and a class hash at its address. If the attacker first submits a valid nonce-0 invoke (which passes `__validate__` and enters the pool), the address now satisfies `account_tx_in_pool_or_recent_block`. A subsequent nonce-1 invoke with a completely invalid signature then satisfies all three skip conditions. `run_validate_entry_point` is called with `validate: false`: [4](#0-3) 

`StatefulValidator::perform_validations` returns after `perform_pre_validation_stage` without ever calling `validate_tx` (the `__validate__` entry point): [5](#0-4) 

The invalid-signature transaction is admitted to the mempool. This is the direct analog of the SnapshotERC20Guild bug: the gateway's outer layer (snapshot/committed nonce) decides to skip the inner layer's check (`__validate__`), but the inner layer would have used a different, transaction-specific value (the per-nonce transaction hash) that the outer layer never examined.

### Impact Explanation

An attacker controlling a contract deployed via `deploy` syscall can inject nonce-1 invoke transactions with forged signatures into the mempool without any signature verification. The mempool accepts and queues them. The batcher will later call `__validate__` and reject them, but by then resources have been consumed and the mempool has been polluted. This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

### Likelihood Explanation

Contracts deployed via `deploy` syscall are common in Starknet (wallet factories, proxy deployments). The attacker only needs to control such a contract, submit one valid nonce-0 invoke to seed the mempool entry, and then submit arbitrarily many nonce-1 invokes with invalid signatures. No privileged access is required.

### Recommendation

`skip_stateful_validations` must verify that the account does **not** already exist on-chain (i.e., `class_hash_at(sender) == ClassHash::default()`) before skipping `__validate__`. The skip is only safe when the account contract has not yet been deployed and `__validate__` is therefore uncallable. If the contract already exists, `__validate__` must always be executed regardless of the nonce-1 / account-nonce-0 pattern.

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // NEW: only skip if the account contract does not yet exist on-chain.
            let class_hash = state_reader.get_class_hash_at(tx.sender_address())?;
            if class_hash != ClassHash::default() {
                return Ok(false); // contract exists; must validate signature
            }
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
```

### Proof of Concept

1. Deploy contract `C` via `deploy` syscall from another contract. `C` has a standard ECDSA `__validate__`. Committed state: `nonce(C) = 0`, `class_hash(C) != 0`.
2. Submit `Invoke(sender=C, nonce=0, calldata=..., signature=valid_sig_for_nonce0)` to the gateway. All checks pass; `__validate__` is called and succeeds. `C` is now in `tx_pool`.
3. Submit `Invoke(sender=C, nonce=1, calldata=..., signature=[0,0])` (invalid signature) to the gateway.
   - `validate_nonce`: `0 <= 1 <= max_gap` → passes. [6](#0-5) 
   - `validate_by_mempool`: mempool expects nonce 1 next → passes.
   - `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true` → returns `true`. [7](#0-6) 
   - `run_validate_entry_point` with `validate=false`: `perform_pre_validation_stage` passes (nonce 0≤1, fee ok, balance ok); `__validate__` is **never called**. [4](#0-3) 
4. The invalid-signature transaction is admitted to the mempool. The batcher later calls `__validate__`, which fails, and the transaction is discarded — but the mempool slot was consumed and batcher CPU was wasted.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
