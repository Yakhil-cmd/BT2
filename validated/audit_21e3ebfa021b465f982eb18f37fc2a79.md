### Title
Unbounded Loop Over `pool_member_epoch_balance` Trace in `calculate_rewards` Enables Permanent Freezing of Delegator Yield — (`File: src/pool/pool.cairo`)

---

### Summary
The `calculate_rewards` function in `src/pool/pool.cairo` iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since the last claim. Because the trace grows by one entry per epoch in which a balance change occurs, and because `entry_to_claim_from` is only advanced on a successful `claim_rewards` call, a pool member who makes repeated balance changes across many epochs without claiming rewards accumulates an arbitrarily large trace. When `claim_rewards` is eventually called, the loop iterates over all accumulated entries and can exceed Starknet's gas/step limits, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` (lines 837–888) contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop variable `entry_to_claim_from` is read from `pool_member_info.entry_to_claim_from` and is only advanced when `claim_rewards` completes successfully. [2](#0-1) 

The trace is grown by `set_member_balance` / `increase_member_balance`, which are called on every `enter_delegation_pool`, `add_to_delegation_pool`, and `exit_delegation_pool_intent` invocation. The underlying `insert` function appends a new checkpoint only when the epoch key differs from the last entry; same-epoch changes overwrite the last entry. [3](#0-2) 

Therefore, one new trace entry is created per distinct epoch in which a balance change occurs. If a pool member makes balance changes across N distinct epochs without ever calling `claim_rewards`, the loop in `calculate_rewards` will iterate over all N entries on the next claim attempt.

`calculate_rewards` is invoked on-chain in two places:
- `claim_rewards` (line 349–355) — the only way to collect yield.
- `pool_member_info_v1` (line 532–538) — a view that also runs the loop on-chain when called from another contract. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

If the trace grows large enough that the loop exceeds Starknet's per-transaction step/gas budget, `claim_rewards` reverts on every call. Because `entry_to_claim_from` is only advanced inside a successful `claim_rewards` execution, there is no way to partially drain the trace across multiple transactions. The pool member's accumulated yield is permanently frozen in the pool contract with no recovery path.

This matches the allowed impact: **Permanent freezing of unclaimed yield (High)** and **Unbounded gas consumption (Medium)**.

---

### Likelihood Explanation

The pool member's `reward_address` is set independently of the `pool_member` address. A malicious pool member (caller of `enter_delegation_pool`) can set their own address as `pool_member` while pointing `reward_address` to a victim. The attacker then:

1. Makes one small `add_to_delegation_pool` or `exit_delegation_pool_intent` call per epoch (each costs only the minimum gas for a balance change).
2. Never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
3. After sufficiently many epochs, the trace is large enough that `claim_rewards` exceeds the step limit.
4. The victim's `reward_address` can never collect the accumulated yield.

The cost to the attacker is proportional to the number of epochs × gas per balance-change call — a low-cost griefing attack with no profit motive. Even without a deliberate attacker, a long-lived delegator who makes frequent balance adjustments and rarely claims rewards can accidentally trigger this condition.

---

### Recommendation

1. **Paginate `calculate_rewards`**: Accept a `max_iterations` parameter and return a partial result plus the updated `entry_to_claim_from`, allowing callers to drain the trace across multiple transactions.
2. **Enforce a claim cadence**: Require or incentivize `claim_rewards` to be called before each balance change, keeping `entry_to_claim_from` close to the trace head.
3. **Cap trace growth**: Reject balance changes if the unclaimed trace depth exceeds a protocol-defined maximum, forcing the member to claim first.

---

### Proof of Concept

```
1. Alice deploys a pool member with pool_member = Alice, reward_address = Bob.
2. Alice calls add_to_delegation_pool once per epoch for N epochs (N = ~500–1000).
   - Each call inserts a new entry into pool_member_epoch_balance (different epoch key each time).
   - entry_to_claim_from remains 0 because claim_rewards is never called.
3. After N epochs, pool_member_trace_length == N.
4. Bob (or Alice) calls claim_rewards(pool_member: Alice).
   - calculate_rewards loops from entry_to_claim_from=0 to pool_member_trace_length=N.
   - Each iteration reads pool_member_trace.at(i) (storage read) and calls find_sigma (another storage read).
   - At sufficient N, the transaction exceeds Starknet's step budget and reverts.
5. entry_to_claim_from is never updated (revert rolls back state).
6. All subsequent claim_rewards calls also revert — Bob's yield is permanently frozen.
``` [6](#0-5) [7](#0-6)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
