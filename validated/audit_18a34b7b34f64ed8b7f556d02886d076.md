### Title
Missing Signer-Address Authorization in `MsgRegisterEncryptionKey` Allows Overwriting Any Account's Encryption Key — (`x/e2ee/keeper/keeper.go`, `x/e2ee/types/msg.go`)

### Summary
`MsgRegisterEncryptionKey` accepts an arbitrary `Address` field that is never compared to the transaction signer. Any unprivileged account can overwrite any other account's registered e2ee public key with an attacker-controlled key.

### Finding Description
`MsgRegisterEncryptionKey` carries two independent fields: `Address` (the account whose key is being registered) and the signing account (the tx fee payer/signer). Neither `ValidateBasic()` nor `Keeper.RegisterEncryptionKey()` / `registerEncryptionKey()` ever retrieves the signer from the context and compares it to `req.Address`. The keeper unconditionally executes:

```go
sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
```

where `bz` is derived from the attacker-supplied `req.Address`. [4](#0-3) 

### Impact Explanation
An attacker controlling account A submits a valid, signed `MsgRegisterEncryptionKey` with `Address = <victim_bech32>` and `Key = <attacker_age_pubkey>`. After `DeliverTx`, the KVStore entry for the victim's address is permanently overwritten. All subsequent senders who query the victim's registered key (e.g., via `Keys` RPC) will encrypt messages to the attacker's key, allowing the attacker to decrypt all future e2ee messages intended for the victim. The victim cannot prevent this and may not detect it. This is a permanent, on-chain state corruption matching the defined High impact: "Corruption of... e2ee key/message state with direct security impact."

### Likelihood Explanation
Trivially exploitable by any account with enough funds to pay tx fees. No special privileges, leaked keys, or external assumptions required. The attack is a single standard Cosmos SDK transaction.

### Recommendation
In `Keeper.RegisterEncryptionKey`, extract the signer from the SDK context and enforce equality with `req.Address` before writing:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signerAddr := sdk.UnwrapSDKContext(ctx).EventManager()... // or use msg server signer extraction
    // enforce req.Address == signerAddr
    if req.Address != signerAddr.String() {
        return nil, sdkerrors.ErrUnauthorized.Wrapf("signer %s cannot register key for %s", signerAddr, req.Address)
    }
    ...
}
```

Alternatively, remove the `Address` field from the message entirely and derive it from the signer inside the keeper.

### Proof of Concept
Unit test plan (no external dependencies):

1. Create two test accounts: `accountA` (attacker) and `accountB` (victim).
2. Pre-register a legitimate key for `accountB` via `keeper.registerEncryptionKey(ctx, accountB.String(), legitimateKey)`.
3. Construct `MsgRegisterEncryptionKey{Address: accountB.String(), Key: attackerKey}` signed by `accountA`.
4. Call `keeper.RegisterEncryptionKey(ctx, msg)` — expect success (no error).
5. Query `KVStore.Get(types.KeyPrefix(accountB_bz))` — assert value equals `attackerKey`, not `legitimateKey`.

This test will pass against the current code, confirming the vulnerability.
