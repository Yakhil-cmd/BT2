### Title
Unprivileged Account Can Overwrite Any Other Account's Global Contract in AccountId Mode — (File: `runtime/runtime/src/global_contracts.rs`)

---

### Summary

`DeployGlobalContractAction` with `GlobalContractDeployMode::AccountId` stores the global contract indexed by the **receipt receiver's** account ID. No guard enforces that the predecessor (sender) must equal the receiver. Any unprivileged account can therefore overwrite the global contract namespace entry of an arbitrary victim account, silently replacing the code executed by every account that has opted into `UseGlobalContract(AccountId(victim))`.

---

### Finding Description

In `action_deploy_global_contract` the identifier for `AccountId` mode is derived from `account_id`, which is `receipt.receiver_id()`: [1](#0-0) 

The only pre-execution validation is a code-size check: [2](#0-1) 

The protocol explicitly lists the actions that require `predecessor_id == receiver_id` (`DeployContract`, `Stake`, `AddKey`, `DeleteKey`, `DeleteAccount`). `DeployGlobalContract` is absent from that list: [3](#0-2) 

Because no ownership check exists, an attacker can craft a transaction with `signer_id = attacker`, `receiver_id = victim`, `actions = [DeployGlobalContract(AccountId, malicious_wasm)]`. The runtime executes `action_deploy_global_contract` with `account_id = victim`, writes the attacker's code to `TrieKey::GlobalContractCode { identifier: AccountId(victim) }`, and deducts storage cost from the **victim's** balance: [4](#0-3) 

The same attack is reachable via a cross-contract promise: a malicious contract calls `promise_batch_action_deploy_global_contract_by_account_id` targeting the victim, producing an identical receipt with no additional authorization gate.

---

### Impact Explanation

**Exact corrupted value:** `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(victim_account_id) }` is overwritten with attacker-controlled WASM bytes.

Every account that previously called `UseGlobalContract(AccountId(victim))` stores `AccountContract::GlobalByAccount(victim_account_id)`: [5](#0-4) 

At the next function call on any such account, the runtime resolves the code from the now-corrupted trie key and executes the attacker's WASM. The attacker can steal funds, corrupt storage, or brick all downstream accounts. Additionally, the victim's NEAR balance is reduced by `global_contract_storage_amount_per_byte × |malicious_code|` without consent.

Scope match: **global contract/code selection** (exact).

---

### Likelihood Explanation

The attacker requires only a funded NEAR account and the ability to submit a signed transaction — no validator, operator, or admin privilege. The attack is a single transaction. The nonce-based freshness guard (`check_and_update_nonce`) only prevents *stale* re-delivery of the same distribution receipt; it does not prevent a fresh overwrite by a different account: [6](#0-5) 

---

### Recommendation

Add an ownership check in `action_deploy_global_contract` (or in `validate_deploy_global_contract_action`) that, when `deploy_mode == AccountId`, asserts `predecessor_id == receiver_id` (i.e., the account deploying the global contract under its own ID must be the receipt's own predecessor). This mirrors the existing guard applied to `DeployContract`, `AddKey`, and similar self-modifying actions.

---

### Proof of Concept

1. **Alice** deploys a legitimate global contract:
   `tx(signer=alice, receiver=alice, [DeployGlobalContract(AccountId, good_wasm)])`
   → `GlobalContractCode[AccountId(alice)] = good_wasm`

2. **Bob** opts in:
   `tx(signer=bob, receiver=bob, [UseGlobalContract(AccountId(alice))])`
   → `bob.contract = GlobalByAccount(alice)`

3. **Attacker** sends:
   `tx(signer=attacker, receiver=alice, [DeployGlobalContract(AccountId, evil_wasm)])`
   → Runtime executes on `account_id = alice`, writes `GlobalContractCode[AccountId(alice)] = evil_wasm`, deducts storage cost from Alice's balance.

4. Any subsequent function call on Bob's account resolves `GlobalByAccount(alice)` → `evil_wasm` and executes the attacker's code.

fn check_and_update_nonce(
    global_contract_data: &GlobalContractDistributionReceipt,
    identifier: &GlobalContractCodeIdentifier,
    state_update: &mut TrieUpdate,
) -> Result<bool, RuntimeError> {
    let nonce_key = TrieKey::GlobalContractNonce { identifier: identifier.clone() };
    let stored_nonce = get_nonce(state_update, &nonce_key)?;
    let incoming_nonce = global_contract_data.nonce();

    // Allow the same nonce since the nonce is updated immediately when
    // initiating distribution to prevent multiple distributions with the same
    // nonce from being initiated.
    if incoming_nonce < stored_nonce {
        return Ok(false);
    }

    set_nonce(state_update, nonce_key, incoming_nonce);
    Ok(true)
}

fn validate_deploy_global_contract_action(
    limit_config: &LimitConfig,
    action: &DeployGlobalContractAction,

            size: action.code.len() as u64,
            limit: limit_config.max_contract_size,
