### Title
Pool Contract `claim_rewards` Bypasses Staking Contract Pause — (`File: src/pool/pool.cairo`)

### Summary
The `Pool` contract's `claim_rewards` function contains no check against the staking contract's pause state. When the staking contract is paused by the security agent, delegators can still call `claim_rewards` on the pool contract and receive STRK transfers, bypassing the emergency freeze entirely.

### Finding Description
The staking contract (`src/staking/staking.cairo`) implements a pause mechanism via an `is_paused` storage flag. Every state-changing function on the staking contract asserts this flag before proceeding. The staking contract's own `claim_rewards` (for stakers) is blocked when paused, as confirmed by `test_claim_rewards_when_paused`. [1](#0-0) 

The `Pool` contract, however, has **zero pause-related logic**. A grep for `CONTRACT_IS_PAUSED`, `is_paused`, or `assert_paused` across all of `src/pool/` returns no matches. [2](#0-1) 

The `claim_rewards` function in the pool contract:
1. Reads pool member state.
2. Calculates accrued rewards.
3. Calls `get_current_epoch` on the staking contract — a **read-only** view function that is **not** paused-checked.
4. Directly transfers STRK from the pool contract's own balance via `IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS }.checked_transfer(...)`. [3](#0-2) 

Step 4 never touches the staking contract's guarded entry points. The pause on the staking contract therefore has no effect on this transfer path.

Compare with the pool's `enter_delegation_pool` and `exit_delegation_pool_intent`, which call `add_stake_from_pool` and `remove_from_delegation_pool_intent` respectively — both of which ARE paused-checked on the staking contract side, so those pool functions are indirectly blocked. `claim_rewards` has no such indirect protection. [4](#0-3) 

### Impact Explanation
When the security agent pauses the staking contract (e.g., due to a suspected reward-calculation exploit or accounting bug), the intent is to freeze all reward flows. Stakers cannot call `claim_rewards` on the staking contract. However, delegators can still drain all accrued pool rewards via the pool contract's `claim_rewards`. If the pause was triggered because rewards were inflated by a bug, delegators can claim those inflated rewards before the issue is remediated — constituting theft of unclaimed yield.

**Impact: High** — Theft of unclaimed yield; the emergency pause mechanism is rendered ineffective for the delegation pool reward path.

### Likelihood Explanation
Any pool member or their registered `reward_address` can call `claim_rewards` on the pool contract at any time. No privileged access is required. The call succeeds as long as the pool contract holds a STRK balance, which it does whenever `update_rewards_from_staking_contract` has been called in prior epochs. The attacker-controlled entry path is fully public and requires no special setup beyond being an existing pool member. [5](#0-4) 

### Recommendation
The pool contract's `claim_rewards` (and `change_reward_address`) should check the staking contract's pause state before executing. The pool contract already holds a dispatcher to the staking contract (`staking_pool_dispatcher`). A read call to `IStakingDispatcher::is_paused()` should be added at the top of `claim_rewards`, reverting with an appropriate error if the staking contract is paused. [6](#0-5) 

### Proof of Concept

Attack path:
1. Delegator enters a pool and accrues rewards over several epochs.
2. A reward-calculation bug is discovered; the security agent calls `staking_contract.pause()`.
3. The staking contract is now paused — stakers cannot call `claim_rewards` on the staking contract.
4. The delegator calls `pool_contract.claim_rewards(pool_member)` directly.
5. The pool contract executes without any pause check, calls the non-guarded `get_current_epoch` view, computes rewards, and calls `STRK_TOKEN.transfer(reward_address, rewards)`.
6. The delegator receives inflated STRK rewards despite the emergency pause.

            self

                );

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
