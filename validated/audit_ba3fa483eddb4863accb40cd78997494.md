### Title
Missing Lower-Bound Threshold Validation in FROST Presign and Sign Initialization — (`src/frost/mod.rs`)

### Summary

The `presign` and `assert_sign_inputs` functions in `src/frost/mod.rs` validate that the caller-supplied `threshold` does not exceed the participant count, but they omit the symmetric lower-bound check (`threshold >= 2`) that every other protocol entry-point in the codebase enforces. A caller can therefore pass `threshold = 0` or `threshold = 1` into the FROST EdDSA / RedJubjub signing pipeline, causing Lagrange interpolation to operate on a degree-0 (or degenerate) polynomial and producing an unusable or cryptographically incorrect signature output, or triggering a runtime panic that permanently denies signing for honest parties.

### Finding Description

`ReconstructionLowerBound` is a plain `usize` newtype with no enforced minimum: [1](#0-0) 

Every other protocol entry-point that accepts a threshold enforces `threshold >= 2`:

- `assert_key_invariants` in `src/dkg.rs`: [2](#0-1) 

- `validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs`: [3](#0-2) 

The FROST-specific functions are inconsistent. `presign` only checks the upper bound: [4](#0-3) 

`assert_sign_inputs` (called by the EdDSA and RedJubjub sign wrappers) likewise only checks the upper bound: [5](#0-4) 

Neither function contains the missing guard:
```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold.value(), min: 2 });
}
```

A caller can therefore construct a `PresignArguments` with `threshold = ReconstructionLowerBound::from(1usize)` (or `0`) and pass it directly to `presign`, then pass the same invalid threshold to `assert_sign_inputs` / the sign function. Both calls will return `Ok(...)` and proceed into the protocol.

### Impact Explanation

**High — Corruption of sign outputs so honest parties receive unusable cryptographic outputs.**

With `threshold = 1`, the Lagrange interpolation inside the signing round operates as if the secret polynomial has degree 0 (a constant), but the actual key shares were generated by DKG with a degree ≥ 1 polynomial (DKG enforces `threshold >= 2`). The resulting signature scalar `s` will be arithmetically incorrect; the final `sig.verify(...)` check inside the coordinator path will fail, and honest parties will receive a `ProtocolError::AssertionFailed` instead of a usable signature. With `threshold = 0`, the Lagrange coefficient computation receives an empty or zero-denominator input, which can produce a panic, permanently denying signing for all honest participants in that session.

### Likelihood Explanation

Any library caller that constructs `PresignArguments` directly controls the `threshold` field. There is no type-level enforcement preventing `threshold = 1`. The missing check is a straightforward omission visible by comparing the FROST validation path with the DKG and triple-generation paths. A misconfigured integrator or a malicious coordinator who controls the session parameters can trigger this with a single API call.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use, in both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately after the upper-bound check in each function, mirroring the pattern at `src/dkg.rs:580-582` and `src/ecdsa/ot_based_ecdsa/triples/generation.rs:699-703`.

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
