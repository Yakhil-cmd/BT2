### Title
`EVM.mustRun()` Silently Accepts Failed EVM Execution, Enabling Cross-VM Asset Loss - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in the Flow EVM system contract is designed as the "safe" variant of `EVM.run()`, but its internal `assert` only rejects `invalid` and `unknown` statuses. A `failed` EVM execution (e.g., a reverted token transfer, out-of-gas) passes the assertion without reverting the enclosing Cadence transaction. Any Cadence transaction that calls `mustRun()` to perform an EVM token transfer and then performs Cadence-side accounting based on the assumption the transfer succeeded will silently mis-account assets.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, the `EVM.mustRun()` function is defined as:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The `assert` condition is:
```
runResult.status == Status.failed || runResult.status == Status.successful
```

This means the assertion **passes** for both `Status.failed` and `Status.successful`. Only `Status.invalid` and `Status.unknown` cause a revert. The EVM `Status` enum has four values:

- `unknown` — something went very wrong
- `invalid` — rejected at validation (bad nonce, bad signature)
- `failed` — executed but the VM reported an error (out of gas, revert)
- `successful` — executed with no error [2](#0-1) 

A `failed` EVM transaction is one that was included in a block but whose execution reverted (e.g., an ERC-20 `transfer()` that reverted due to insufficient balance, or a call that ran out of gas). The EVM state changes from a `failed` transaction are rolled back at the EVM level, but the Cadence transaction is **not** reverted by `mustRun()`.

The root cause is identical to the reported vulnerability class: a function named to imply guaranteed success (`mustRun`) does not actually check whether the underlying operation succeeded, allowing the caller to continue executing after a silent failure.

---

### Impact Explanation

**Cross-VM asset loss / bridge escrow mis-accounting.**

Any Cadence transaction that:
1. Calls `EVM.mustRun()` to execute an EVM token transfer (e.g., ERC-20 `transfer`, native FLOW transfer via EVM)
2. Then performs Cadence-side state mutations (minting Cadence tokens, updating balances, recording a deposit, releasing escrowed assets)

…will perform those Cadence-side mutations even if the EVM transfer reverted. The EVM side holds no tokens transferred; the Cadence side records a successful transfer. This is a cross-VM accounting discrepancy that can be exploited to extract assets.

Concrete example: a bridge or relayer contract written in Cadence calls `EVM.mustRun()` to move ERC-20 tokens from a user's EVM address to an escrow, then mints wrapped Cadence tokens. If the ERC-20 transfer reverts (e.g., the user's EVM balance was drained by a front-run), `mustRun()` does not revert the Cadence transaction, so wrapped tokens are minted without any ERC-20 tokens being locked.

---

### Likelihood Explanation

**Low.** The vulnerability requires a Cadence transaction or contract that:
- Uses `EVM.mustRun()` for EVM token transfers, AND
- Performs Cadence-side accounting after the call without checking `runResult.status == Status.successful`

The misleading name `mustRun` increases the probability that developers will make this mistake, believing the function guarantees success. The behavior is documented in a comment, but the name contradicts the documentation.

---

### Recommendation

Change the `assert` in `EVM.mustRun()` to only pass for `Status.successful`, reverting the Cadence transaction on any non-successful EVM outcome:

```cadence
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.successful,
        message: "EVM.mustRun(): EVM transaction did not succeed"
    )
    return runResult
}
```

If the current behavior (allowing `failed` to pass) is intentional for gas-accounting reasons, rename the function to something that does not imply guaranteed success (e.g., `runIfValid`) and add a separate `mustSucceed()` variant that asserts `Status.successful`.

### Proof of Concept

A Cadence transaction demonstrating the silent failure:

```cadence
import EVM from <EVMContractAddress>

transaction(erc20TransferTx: [UInt8], coinbaseBytes: [UInt8; 20]) {
    execute {
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)

        // ERC-20 transfer that will revert (e.g., insufficient balance)
        // mustRun does NOT revert the Cadence transaction on Status.failed
        let result = EVM.mustRun(tx: erc20TransferTx, coinbase: coinbase)

        // result.status == EVM.Status.failed here, but execution continues
        // Any Cadence-side accounting below executes regardless:
        // e.g., mint wrapped tokens, release escrow, record deposit
    }
}
```

The `assert` at line 846–849 of `contract.cdc` passes when `result.status == Status.failed`, so the transaction does not revert. The EVM transfer is rolled back at the EVM level, but all subsequent Cadence operations execute normally, creating a cross-VM accounting discrepancy. [3](#0-2) [2](#0-1)
