### Title
Global-contract identifier overhead omitted from deletion storage check, permanently blocking `DeleteAccount` for affected accounts — (`File: runtime/runtime/src/actions.rs`)

### Summary

In `action_delete_account`, the legacy code path (pre-protocol-version 85) computes the account's non-contract storage usage by subtracting only the **local** contract code length. For accounts that called `UseGlobalContract`, the global-contract identifier bytes were added to `storage_usage` at use-time but are never subtracted at delete-time. The deletion guard therefore compares an inflated value against `MAX_ACCOUNT_DELETION_STORAGE_USAGE`, causing a false `DeleteAccountWithLargeState` error for any account whose non-contract storage is within the limit but whose total `storage_usage` (including the identifier) exceeds it.

### Finding Description

**Phase 1 — `UseGlobalContract` / `use_global_contract` in `runtime/runtime/src/global_contracts.rs`:**

When an account adopts a global contract, `identifier.len()` bytes are added to `account.storage_usage()`:

```rust
account.set_storage_usage(
    account.storage_usage()
        .checked_add(contract_identifier.len() as u64)…
);
``` [1](#0-0) 

For `AccountContract::Global(CryptoHash)` the identifier is 32 bytes; for `AccountContract::GlobalByAccount(AccountId)` it is the account-ID length (up to 64 bytes).

**Phase 2 — `action_delete_account` in `runtime/runtime/src/actions.rs` (legacy path):**

The deletion guard computes the non-contract storage usage by subtracting only the **local** contract code length:

```rust
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(), // returns 0 for global contracts
)?;
account_storage_usage.saturating_sub(code_len)  // subtracts 0 → full storage_usage retained
``` [2](#0-1) 

For a global-contract account, `local_contract_hash()` returns `None` → `code_len = 0` → `account_storage_usage = account.storage_usage()` (still includes the identifier overhead). The guard then fires:

```rust
if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
    result.result = Err(ActionErrorKind::DeleteAccountWithLargeState { … });
    return Ok(());
}
``` [3](#0-2) 

`MAX_ACCOUNT_DELETION_STORAGE_USAGE` is 10 000 bytes. [4](#0-3) 

**The broken invariant:** the deletion check is supposed to measure only non-contract storage, mirroring the logic used for local contracts. For global contracts the identifier overhead is included in `storage_usage` at use-time but is never removed at delete-time, so the measured value is `identifier.len()` bytes too large.

The protocol-transition test confirms the exact failure window:

```
storage = MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32   // 10 032 bytes
// Before fix: 10 032 - 0 = 10 032 > 10 000 → DeleteAccountWithLargeState (wrong)
// After fix:  10 032 - 32 = 10 000 ≤ 10 000 → success (correct)
``` [5](#0-4) 

### Impact Explanation

Any account that:
1. Called `UseGlobalContract` (adding `identifier.len()` to `storage_usage`), and
2. Has non-contract storage in the range `(MAX − identifier.len(), MAX]` bytes

is permanently blocked from `DeleteAccount`. The beneficiary never receives the account's liquid balance via the deletion path. The account cannot be closed. Because the identifier length is user-controlled (up to 64 bytes for `GlobalByAccount`), an attacker can craft a scenario where a victim account—or their own account—falls into this window. The denial is permanent until a protocol upgrade deploys the fix.

### Likelihood Explanation

- Any unprivileged user can call `UseGlobalContract` on their own account, then attempt `DeleteAccount`.
- The triggering window is narrow (at most 64 bytes wide), but it is deterministically reachable: a user who wants to delete their account and whose non-contract storage happens to land in `(MAX − identifier.len(), MAX]` will always hit the bug.
- No special privileges, validator access, or admin keys are required.
- The bug is active on any network running protocol version < 85.

### Recommendation

The fix is already present behind `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` (protocol version 85): use `get_contract_storage_usage` instead of `get_code_len_or_default` so that the identifier overhead is subtracted for global contracts, exactly as local code length is subtracted for local contracts. [6](#0-5) 

Ensure the feature is enabled on all production networks and that the legacy path is not reachable on any live chain still running protocol version < 85.

### Proof of Concept

1. Deploy a global contract (e.g., `GlobalContractDeployMode::CodeHash`).
2. Create an account whose non-contract `storage_usage` is exactly `MAX_ACCOUNT_DELETION_STORAGE_USAGE` (10 000 bytes).
3. Call `UseGlobalContract` on that account — this adds 32 bytes to `storage_usage`, making it 10 032.
4. Submit a `DeleteAccount` action on a network running protocol version < 85.
5. Observe `ActionErrorKind::DeleteAccountWithLargeState` even though the account's non-contract storage is exactly at the limit (10 000 bytes ≤ 10 000 bytes).
6. The account cannot be deleted; the beneficiary receives nothing via the deletion path.

The test `test_delete_account_global_contract_protocol_transition` in `runtime/runtime/src/actions.rs` reproduces this exactly. [5](#0-4)
