### Title
Unbounded Native Resource Consumption in EIP-7702 Authorization List Loop (Ethereum Flow) — (`basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

In the Ethereum transaction flow, `parse_authorization_list_and_apply_delegations` iterates over every EIP-7702 authorization entry without any upper-bound on native (RISC-V proving) resource consumption. Because the Ethereum flow's `create_resources_for_tx` initialises native resources to `u64::MAX`, a user can submit a transaction whose authorization list forces arbitrarily expensive proving work — including one `ecrecover` call per entry — while paying only EVM gas, not native resources.

---

### Finding Description

**Root cause — unlimited native in the Ethereum flow**

`create_resources_for_tx` in `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs` (lines 26–59) hard-codes the native limit to `u64::MAX`:

```rust
let native_limit =
    <<S as zk_ee::system::SystemTypes>::Resources as Resources>::Native
        ::from_computational(u64::MAX);
let main_resources = S::Resources::from_ergs_and_native(Ergs(ergs), native_limit);
``` [1](#0-0) 

This is in contrast to the ZK flow, which derives a finite native limit from `gas_price / native_price`.

**Unbounded loop**

`parse_authorization_list_and_apply_delegations` iterates over every entry in the authorization list with no maximum count enforced:

```rust
for entry in auth_list.iter() {
    let success = validate_and_apply_delegation(
        system, resources, &chain_id, nonce, address, (y_parity, r, s), &mut hasher,
    )?;
}
``` [2](#0-1) 

**Per-entry work**

Inside `validate_and_apply_delegation`, each iteration:
1. Charges `NEWACCOUNT` (25,000) EVM gas — the only real bound.
2. Charges `PER_AUTH_NATIVE_COMPUTATIONAL_OVERHEAD` native — but since native starts at `u64::MAX` this charge never fails.
3. Calls `secp256k1_ec_recover` (full ecrecover, expensive in RISC-V cycles).
4. Reads and writes account state via IO. [3](#0-2) 

**No explicit maximum on list length**

`AuthorizationList` is decoded as `HomList<'a, AuthorizationEntry<'a>, true>`. The only structural check is that the list is non-empty:

```rust
if authorization_list.count == Some(0) {
    return Err(InvalidTransaction::AuthListIsEmpty);
}
``` [4](#0-3) 

No upper bound (e.g. `MAX_AUTH_LIST_SIZE`) is enforced anywhere in the Ethereum flow.

**The ZK flow does charge native — the Ethereum flow does not**

In the ZK validation path, `calculate_l2_tx_intrinsic_computational_native_resources` pre-charges native per authorization entry before the loop runs:

```rust
intrinsic_computational_native_resources = intrinsic_computational_native_resources
    .saturating_add(
        authorization_list_num
            .saturating_mul(L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION),
    );
``` [5](#0-4) 

The Ethereum flow's `create_resources_for_tx` does not call this helper and does not pass `authorization_list_num` at all. [6](#0-5) 

---

### Impact Explanation

An unprivileged user submits a valid EIP-7702 type-4 transaction with the maximum number of authorization entries the gas limit permits (`gas_limit / 25_000`). With a 30 M gas block limit that is up to ~1,200 entries. Each entry triggers a full `secp256k1_ec_recover` in the RISC-V prover. The user pays only EVM gas; the native (proving-cycle) cost is borne by the protocol. This:

- Inflates proving cost for the block far beyond what the user paid for.
- Can cause the prover to exceed its cycle budget for the block, making the block unprovable.
- Is repeatable across every block, constituting a sustained, low-cost DoS against the proving pipeline.

---

### Likelihood Explanation

The attack requires only a standard EIP-7702 transaction with a large authorization list. No privileged access, no leaked keys, no oracle manipulation. Any externally-owned account can submit such a transaction. The `eip-7702` feature is present and imported in the Ethereum flow validation module, making this reachable in production.

---

### Recommendation

1. **Enforce a hard cap** on `authorization_list.count` in the Ethereum flow (e.g. `MAX_AUTH_LIST_ENTRIES = 255` as suggested by EIP-7702 implementors).
2. **Use the same `calculate_l2_tx_intrinsic_computational_native_resources` helper** in the Ethereum flow so that native resources are derived from a finite budget, not `u64::MAX`.
3. **Add a block-level native resource check** in the Ethereum loop op analogous to `check_for_block_limits` in the ZK loop op.

---

### Proof of Concept

1. Construct an EIP-7702 type-4 transaction with `gas_limit = 30_000_000`.
2. Populate the authorization list with `30_000_000 / 25_000 = 1_200` entries, each signed by a distinct EOA with a valid `ecrecover`-recoverable signature.
3. Submit the transaction through the Ethereum block flow.
4. Observe that `parse_authorization_list_and_apply_delegations` executes 1,200 `secp256k1_ec_recover` calls, each consuming significant RISC-V cycles, while `resources` (native = `u64::MAX`) never exhausts — the user paid only EVM gas, not native resources.
5. The resulting block requires far more RISC-V cycles to prove than the user's payment covers, inflating proving cost or causing the prover to exceed its cycle budget.
