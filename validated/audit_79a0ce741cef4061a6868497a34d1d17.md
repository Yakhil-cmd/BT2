### Title
`new_settlement_layer_chain_id_storage` Not Rolled Back on Frame Revert — (`File: basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIOStateSnapshot` omits a snapshot handle for `new_settlement_layer_chain_id_storage`. As a result, `start_io_frame` never snapshots it and `finish_io_frame` never rolls it back. If any EVM call frame that triggers the `SettlementLayerChainIdUpdated` event hook subsequently reverts, the in-memory `new_settlement_layer_chain_id_storage` retains the stale update while the underlying storage slot is correctly rolled back. This divergence causes a panic at block finalization and prevents the block from being proven.

---

### Finding Description

`FullIO` holds six sub-storages that must all be snapshotted and rolled back together on frame revert:

```
storage, transient_storage, logs_storage, events_storage,
interop_root_storage, new_settlement_layer_chain_id_storage
```

`FullIOStateSnapshot` captures only five of them:

---

### Impact Explanation

When a frame that triggered `update_settlement_layer_chain_id` is reverted:

1. The `SYSTEM_CONTEXT_ADDRESS` storage slot is correctly rolled back to the old chain ID.
2. `new_settlement_layer_chain_id_storage` is **not** rolled back — it retains `Some(new_id)`.
3. At block finalization, `read_batch_context_inputs` reads the old chain ID from storage but finds `Some(new_id)` in the in-memory store → `assert_eq!` panics → **block cannot be finalized or proven**.
4. Additionally, the single-update guard in `update()` now permanently blocks any legitimate SL chain ID update for the rest of the block, since `value()` returns `Some`.

The `settlement_layer_chain_id` field in `BatchOutput` is a ZK proof public input. Incorrect state here means the batch public input is wrong, which is a **state-transition correctness bug** with direct impact on provability.

---

### Likelihood Explanation

The `SettlementLayerChainIdUpdated` event hook fires whenever the event signature matches during any EVM call frame. A service transaction that calls `setSettlementLayerChainId` and then reverts (e.g., due to out-of-native resources, or a revert in a sub-call) is sufficient to trigger the bug. Because `begin_next_tx` also does not reset `new_settlement_layer_chain_id_storage`, the stale value persists for the entire block. [10](#0-9) 

The scenario is reachable whenever a service transaction that updates the SL chain ID reverts at any frame depth.

---

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and wire it into `start_io_frame` / `finish_io_frame`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    new_sl_chain_id: NewSettlementLayerChainIdSnapshotId, // ADD
}
```

In `start_io_frame`:
```rust
let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();
```

In `finish_io_frame`:
```rust
self.new_settlement_layer_chain_id_storage
    .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
```

### Proof of Concept

1. Block contains a service transaction that calls `SystemContext.setSettlementLayerChainId(42)`.
2. The call emits `SettlementLayerChainIdUpdated(42)` → hook fires → `new_settlement_layer_chain_id_storage` is set to `Some(42)`.
3. The transaction reverts (e.g., out-of-native resources).
4. `finish_io_frame(Some(&rollback_handle))` is called:
   - `storage.finish_frame(...)` rolls back the `SYSTEM_CONTEXT_ADDRESS` slot to the old chain ID ✓
   - `new_settlement_layer_chain_id_storage.finish_frame(...)` is **never called** ✗ → value stays `Some(42)`
5. Block finalization calls `read_batch_context_inputs`:
   - `settlement_layer_chain_id` = old value (from storage, correctly rolled back)
   - `new_settlement_layer_chain_id_storage.value()` = `Some(42)` (stale)
   - `assert_eq!(42, old_value)` → **panic** → block cannot be proven.
