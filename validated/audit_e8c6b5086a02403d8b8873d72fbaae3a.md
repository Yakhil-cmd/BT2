### Title
Missing Validation of Participant Contributions in CKD Coordinator Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The `do_ckd_coordinator` function aggregates `(big_y, big_c)` contributions from all participants by simple addition, with no proof-of-correctness check on any received value. A single malicious participant can send arbitrary group elements, silently corrupting the final `CKDOutput`. The client then calls `unmask(app_sk)` on a poisoned ciphertext and obtains an incorrect confidential derived key with no ability to detect the corruption.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, `do_ckd_coordinator` receives each participant's `CKDOutput` and unconditionally adds the two components:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each participant is supposed to send `(λ_i · y_i · G,  λ_i · (x_i · H(pk, app_id) + y_i · app_pk))`. No zero-knowledge proof, commitment, or consistency check is attached to these values. The participant sends them privately to the coordinator via `send_private`, so no other participant can observe or challenge them:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [2](#0-1) 

The `compute_signature_share` function that produces the honest values also attaches no proof: [3](#0-2) 

By contrast, both ECDSA signing coordinators verify the aggregated signature against the public key before returning, providing a final-output sanity check that the CKD coordinator entirely lacks: [4](#0-3) [5](#0-4) 

The DKG protocol similarly validates every received share against the sender's public polynomial commitment before accepting it: [6](#0-5) 

The CKD protocol has no equivalent guard.

---

### Impact Explanation

Let the honest aggregate be `(big_Y, big_C)` where `unmask(app_sk) = big_C − app_sk · big_Y = msk · H(pk, app_id)`.

A malicious participant P_m substitutes `(big_y_m + Δ_Y, big_c_m + Δ_C)` for arbitrary group elements `Δ_Y, Δ_C`. The coordinator computes:

```
big_Y' = big_Y + Δ_Y
big_C' = big_C + Δ_C
```

The client's `unmask` call then yields:

```
big_C' − app_sk · big_Y' = msk · H(pk, app_id) + Δ_C − app_sk · Δ_Y
``` [7](#0-6) 

This is an incorrect confidential derived key. The client has no way to detect the corruption because `unmask` performs no verification — it is a pure arithmetic operation. The coordinator also has no way to detect it because it does not hold `app_sk`.

**Matched allowed impact:** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

- The attacker is any single participant in the CKD session — the weakest possible trust assumption.
- The CKD protocol has no threshold structure: all `n` participants contribute, so even one corrupted party poisons the result.
- The attack requires only that the malicious participant send a modified pair of group elements instead of the correct ones — no cryptographic capability is needed.
- The private channel (`send_private`) ensures no other honest participant can observe or challenge the malicious contribution.
- The coordinator and client have no post-hoc verification path.

---

### Recommendation

1. **Add a Schnorr / DLEQ proof of correctness.** Each participant should accompany `(big_y_i, big_c_i)` with a non-interactive proof that `big_c_i − big_y_i · app_pk` lies on the correct coset of the participant's public key share. This requires exposing per-participant public key shares (e.g., `x_i · G2`) from the keygen output.

2. **Alternatively, use a verifiable ElGamal construction** where the coordinator can check each ciphertext against the sender's known public key share using a pairing: `e(big_c_i − big_y_i · app_pk, G2) =?= e(H(pk, app_id), x_i · G2)`.

3. **At minimum, add a final-output consistency check** analogous to the ECDSA coordinator's `sig.verify(...)` call — for example, by having the client return a verification token that the coordinator can check before distributing the result.

---

### Proof of Concept

1. Honest participants P_1 … P_{n-1} compute and send correct `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant P_m computes the correct values but instead sends `(norm_big_y_m + G1::generator(), norm_big_c_m)` — a one-generator shift in `big_y`.
3. `do_ckd_coordinator` adds all contributions without any check (lines 50–55 of `protocol.rs`).
4. The resulting `CKDOutput` has `big_Y' = big_Y + G1::generator()`.
5. The client calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk, app_id) − app_sk · G1::generator()` — a uniformly random-looking but incorrect group element.
6. No error is raised anywhere in the protocol; the corrupted output is silently accepted. [8](#0-7) [9](#0-8)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
