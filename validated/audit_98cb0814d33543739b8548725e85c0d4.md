### Title
Stale Per-Address Router Version Cache After `factory_set_wnear_address` Change — (File: `engine/src/contract_methods/xcc.rs`)

### Summary
When `factory_set_wnear_address` updates the wNEAR ERC-20 address in engine storage, the per-address router code version cache is not invalidated. Existing router sub-accounts are considered up-to-date and are never redeployed with the new wNEAR address. Any subsequent XCC call that requires NEAR payment will attempt to use the new wNEAR ERC-20 contract on the engine side while the already-deployed router sub-account still holds the old wNEAR NEAR account in its initialization state, causing the unwrap step to fail and breaking XCC for every affected address.

### Finding Description

The XCC system stores two independent pieces of state:

1. **Global wNEAR address** — written by `factory_set_wnear_address` to the `WNEAR_KEY` slot. [1](#0-0) 

2. **Per-address router code version** — written by `set_code_version_of_address` when a router is deployed or upgraded. [2](#0-1) 

`AddressVersionStatus::new` decides whether a router needs (re)deployment by comparing only code versions, never the wNEAR address: [3](#0-2) 

There is an additional hard-coded exception: routers deployed before the upgrade mechanism was introduced are permanently classified as `UpToDate` and can **never** be redeployed: [4](#0-3) 

When an XCC call is processed, the engine reads the **current** (new) wNEAR address from storage and uses it to call `withdrawToNear` on the new wNEAR ERC-20 contract: [5](#0-4) 

The new wNEAR NEAR contract then calls `ft_on_transfer` on the router sub-account. However, the router was initialized with the **old** wNEAR NEAR account and will reject the callback from the new wNEAR NEAR account, causing the unwrap and the entire XCC call to fail.

The initialization argument that embeds the wNEAR account into the router is constructed at deployment time from the then-current `get_wnear_address`: [6](#0-5) 

`factory_set_wnear_address` only writes the new address; it performs no version bump and no cache invalidation: [1](#0-0) 

`update_router_code` (called by `factory_update`) is the only function that increments the version counter, and it is entirely independent of `factory_set_wnear_address`: [7](#0-6) 

Because the two operations are decoupled, an owner who legitimately migrates the wNEAR contract and calls only `factory_set_wnear_address` leaves every existing router sub-account permanently out of sync.

### Impact Explanation

After `factory_set_wnear_address` is called without a simultaneous `factory_update`:

- Every EVM address whose router version is `>= first_upgradable_version` will not be redeployed (version still matches `latest_code_version`), so its router retains the old wNEAR NEAR account.
- Every EVM address whose router version is `< first_upgradable_version` can **never** be redeployed by design, making the breakage permanent for those addresses.
- Any XCC call from these addresses that attaches NEAR (i.e., `required_near != ZERO_YOCTO`) will fail at the unwrap step because the router rejects `ft_on_transfer` from the new wNEAR NEAR account.
- User wNEAR ERC-20 tokens that were transferred into the engine's implicit address as part of the failed call are at risk of being unrecoverable if the refund path also depends on the now-mismatched wNEAR address.

**Impact: High — Temporary (and for pre-upgrade routers, permanent) freezing of XCC functionality and associated funds.**

### Likelihood Explanation

`factory_set_wnear_address` exists precisely to support wNEAR contract migrations. Any routine upgrade of the wNEAR contract on Aurora (a realistic operational event) would trigger this bug if the operator does not also call `factory_update` in the same transaction or immediately after. There is no code-level enforcement or documentation requiring the two calls to be paired, making the omission easy to miss.

### Recommendation

`factory_set_wnear_address` should atomically increment the router code version (equivalent to calling `update_router_code` with the existing bytecode) so that every subsequent XCC call triggers a `deploy_upgrade` with the new wNEAR account embedded in `initialize_args`. Alternatively, the function should reset all per-address code versions to `CodeVersion::ZERO`, forcing full redeployment. A guard that prevents `factory_set_wnear_address` from being called independently of `factory_update` (or that bundles both into a single admin operation) would also mitigate the risk.

### Proof of Concept

1. Owner calls `factory_set_wnear_address(new_wnear_erc20_address)`.
   - Engine storage: `WNEAR_KEY → new_address`. Per-address version cache: **unchanged**.
2. User submits an EVM transaction that calls the XCC precompile with `attached_near > 0`.
3. `CrossContractCall::run_with_handle` calls `state::get_code_version_of_address` → returns the existing version.
4. `AddressVersionStatus::new` compares existing version with `latest_code_version` → `UpToDate`; no redeployment scheduled. [8](#0-7) 
5. Engine calls `withdraw_wnear_to_router` using `get_wnear_address(&io)` → **new** wNEAR ERC-20 address. [5](#0-4) 
6. New wNEAR ERC-20 burns tokens and triggers new wNEAR NEAR contract → `ft_transfer_call(router, amount, "unwrap")`.
7. New wNEAR NEAR contract calls `ft_on_transfer` on the router sub-account.
8. Router sub-account was initialized with **old** wNEAR NEAR account; it rejects the callback from the new wNEAR NEAR account → XCC call fails.
9. All existing router sub-accounts are broken until the owner also calls `factory_update`; pre-upgrade routers are broken permanently.

pub fn update_router_code<I: IO>(io: &mut I, code: &RouterCode) {
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, CODE_KEY);
    io.write_storage(&key, &code.0);

    let current_version = get_latest_code_version(io);
    let latest_version = current_version.increment();

    // Store the latest version, this will be the first one where the
    // router contract is upgradable.
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, FIRST_UPGRADABLE);
    if io.read_storage(&key).is_none() {
        let version_bytes = latest_version.0.to_le_bytes();
        io.write_storage(&key, &version_bytes);
    }

    set_latest_code_version(io, latest_version);
}

impl AddressVersionStatus {
    fn new<I: IO>(
        io: &I,
        latest_code_version: CodeVersion,
        target_code_version: Option<CodeVersion>,
    ) -> Self {
        let first_upgradable_version =
            get_first_upgradable_version(io).unwrap_or(CodeVersion::ZERO);
        match target_code_version {
            None => Self::DeployNeeded {
                create_needed: true,
            },
            Some(version) if version < first_upgradable_version => {
                // It is impossible to upgrade the initial XCC routers because
                // they lack the upgrade method.
                Self::UpToDate
            }
            Some(version) if version < latest_code_version => Self::DeployNeeded {
                create_needed: false,
            },
            Some(_version) => Self::UpToDate,
        }
    }
}
