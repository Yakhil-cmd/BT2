### Title
Caller-Supplied `participants` List Not Validated Against `KeygenOutput` in CKD Protocol Causes Lagrange Coefficient Mismatch and Corrupted Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts a caller-supplied `participants` list that is used to compute each party's Lagrange coefficient, but this list is never validated against the `key_pair: KeygenOutput` that was produced during DKG. A malicious participant can supply a different participant set than the one used by honest parties, causing their Lagrange coefficient to be computed over a mismatched domain. When the coordinator aggregates these inconsistent shares, the resulting CKD output is cryptographically invalid and the confidential derived key is permanently corrupted.

### Finding Description

The `ckd()` entry point accepts `participants: &[Participant]` and `key_pair: KeygenOutput` as independent, unrelated inputs:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,   // trusted DKG output
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError>
```

The only validation performed on `participants` is: [1](#0-0) 

- Minimum count ≥ 2
- No duplicates
- `me` is present
- `coordinator` is present

There is no check that `participants` is consistent with the participant set used during the DKG that produced `key_pair`. The `KeygenOutput` struct itself stores only `private_share` and `public_key`: [2](#0-1) 

The Lagrange coefficient used to normalize each party's share is computed directly from the caller-supplied `participants` list inside `compute_signature_share()`:

```rust
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [3](#0-2) 

A malicious participant P_evil calls `ckd()` with a `participants` list that differs from the one used by honest parties (e.g., omitting one honest participant, or using a different ordering that changes the Lagrange basis). P_evil's share `(norm_big_y, norm_big_c)` is computed over the wrong Lagrange domain and sent to the coordinator: [4](#0-3) 

The coordinator aggregates all received shares by simple addition, with no mechanism to detect that P_evil's share was computed over a different participant set: [5](#0-4) 

The final `CKDOutput` is therefore `Σ λ_i(honest_set) * share_i + λ_evil(fake_set) * share_evil`, which does not equal the intended `msk * H(pk, app_id)`. The derived confidential key is permanently wrong.

### Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Corruption of CKD outputs so honest parties accept inconsistent participant sets or unusable cryptographic outputs."*

The coordinator outputs a `CKDOutput` that is accepted as valid by all honest parties (no error is returned), but the embedded `big_y` and `big_c` values do not correspond to any valid confidential key derivation. Any downstream consumer that calls `ckd_output.unmask(app_sk)` will obtain a garbage value that is not `msk * H(pk, app_id)`. The corruption is silent and undetectable by honest parties.

### Likelihood Explanation

Any participant in the protocol can trigger this by simply passing a different `participants` slice when calling `ckd()`. No privileged access, leaked keys, or cryptographic breaks are required. The attacker only needs to be a legitimate participant in the CKD session. The library provides no session-binding or participant-set commitment mechanism that would prevent this.

### Recommendation

The `KeygenOutput` type should be extended to embed the participant list and threshold used during DKG, or the `ckd()` function should require a separate trusted participant-set commitment (e.g., a hash of the sorted participant list committed to during DKG) that is validated against the caller-supplied `participants` before computing Lagrange coefficients. At minimum, the protocol should include a broadcast round where all participants commit to their view of the participant set before computing shares, so any mismatch is detected and the protocol aborts rather than producing a silently corrupted output.

### Proof of Concept

1. Run DKG with participants `[A, B, C]` and threshold 2, producing `key_pair_A`, `key_pair_B`, `key_pair_C`.
2. Honest participants A and B call `ckd([A, B, C], coordinator=A, me=A/B, key_pair_A/B, app_id, app_pk, rng)`.
3. Malicious participant C calls `ckd([A, C], coordinator=A, me=C, key_pair_C, app_id, app_pk, rng)` — using a 2-party set instead of the correct 3-party set.
4. C's Lagrange coefficient `λ_C([A,C])` differs from `λ_C([A,B,C])`.
5. The coordinator A aggregates all three shares. The result satisfies no valid threshold reconstruction identity.
6. `ckd_output.unmask(app_sk)` returns a value ≠ `msk * H(pk, app_id)`, silently corrupting the confidential derived key with no error returned to honest parties.
