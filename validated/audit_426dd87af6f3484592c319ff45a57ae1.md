### Title
Single Malicious Participant Can Permanently Halt DKG, Reshare, Refresh, Presign, Sign, and CKD Protocols via Message Withholding — (`src/protocol/helpers.rs`)

---

### Summary

`recv_from_others` unconditionally waits until **every** participant has delivered a message. A single malicious participant who simply never sends their message causes the function to block forever with no timeout, retry, or fallback. Because this helper is used in every critical multi-party protocol (DKG, reshare, refresh, triple generation, presigning, signing, CKD), one bad actor can permanently deny all honest parties the ability to complete any of these operations.

---

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` loops on `while !seen.full()`, where `seen.full()` is only true once **all** `n` participants have contributed:

```rust
// src/protocol/helpers.rs  lines 19-24
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

`seen.full()` returns `true` only when every slot in the `ParticipantCounter` is set. There is no deadline, no partial-result path, and no error branch for a missing sender. The `await` on `chan.recv` suspends the async task indefinitely when no message arrives.

This helper is called at every round boundary across the entire codebase:

| OT Triple Generation | `src/ecdsa/ot_based_ecdsa/triples/generation.rs` | 7 |

The echo-broadcast layer (`do_broadcast`) does have BFT thresholds derived from `echo_ready_thresholds`, which tolerates up to `(n-1)/3` silent parties. However, `recv_from_others` is used for the **direct point-to-point rounds** (commitment hashes, secret shares, nonce commitments, etc.) and has **no such tolerance**. A participant who participates honestly in the broadcast rounds but withholds a single direct message at any `recv_from_others` waitpoint permanently stalls the protocol.

---

### Impact Explanation

**High — Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties.**

Any one of the following operations is permanently blocked by a single non-sending participant:

- **DKG / Reshare / Refresh** (`src/dkg.rs` lines 422–426, 514–528): commitment-hash collection and secret-share collection both use `recv_from_others`. Withholding either message prevents honest parties from ever obtaining a key share.
- **Presigning / Signing** (all ECDSA variants): nonce and partial-signature rounds use `recv_from_others`. Withholding blocks signature production indefinitely.
- **CKD** (`src/confidential_key_derivation/protocol.rs`): the derived-key protocol stalls, denying TEE applications their deterministic secrets.

There is no recovery path: the protocol future simply never resolves, and the library provides no timeout or abort signal to the caller.

---

### Likelihood Explanation

**Medium-High.** The attacker need only be a registered participant in the session — a role that is explicitly part of the documented trust model. No cryptographic break, no key compromise, and no external dependency is required. The attacker's action is purely passive (send nothing), making it undetectable until the timeout at the application layer (which the library itself does not implement). Any participant who wishes to grief a key generation or signing session can do so at zero cost.

---

### Recommendation

1. **Threshold-based collection**: Replace `recv_from_others` with a variant that returns once `n - t` messages have arrived (where `t` is the configured fault tolerance), consistent with the BFT guarantees already present in `do_broadcast`.
2. **Caller-supplied deadline**: Accept an optional `Duration` or `Instant` parameter; return a typed `Timeout` error if the deadline expires, allowing the caller to abort and retry with a different participant set.
3. **Identify and exclude non-senders**: On timeout, surface the set of participants who did not respond so the application layer can exclude them from future sessions.

---

### Proof of Concept

```
Setup: 4 participants P1, P2, P3, P4; threshold = 2.

1. All four participants call `keygen` / `do_keygen`.
2. Round 1 (session-id broadcast via `do_broadcast`): all four send — completes normally.
3. Round 2 (commitment-hash via `recv_from_others`, dkg.rs line 422):
   - P1, P2, P3 send their commitment hashes.
   - P4 (malicious) sends nothing.
4. `recv_from_others` on P1, P2, P3 loops forever on `chan.recv(wait_round_1).await`.
5. No timeout fires; no error is returned; the DKG never completes.
6. Honest parties P1–P3 are permanently denied their key shares.
```

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
