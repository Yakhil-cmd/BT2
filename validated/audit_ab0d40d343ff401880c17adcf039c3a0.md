### Title
Unprivileged caller can invoke `RewardSupplier.tick()` with zero ETH, sending zero-fee L1â†’L2 messages that desynchronize L2 state â€” (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is a public, permissionless, payable function with no access control and no minimum-fee guard. When called with `msg.value=0` while pending L2â†’L1 mint request messages exist, it irreversibly consumes those messages, mints tokens, and sends both the bridge deposit and the `update_total_supply` L1â†’L2 messages with a fee of zero. Zero-fee L1â†’L2 messages are accepted by the Starknet messaging contract but are never automatically processed by the sequencer, leaving the L2 `MintingCurve` totalSupply state permanently stale and the minted tokens stuck in the bridge until manual fee top-up.

---

### Finding Description

`tick()` is declared `external payable` with no role modifier: [1](#0-0) 

The fee split is computed as integer division of `msg.value`: [2](#0-1) 

With `msg.value = 0`:
- `msgFee = 0 / 2 = 0` â†’ `bridge().depositWithMessage{value: 0}(...)`
- `msgFee = 0 - 0 = 0` â†’ `messagingContract().sendMessageToL2{value: 0}(...)` [3](#0-2) 

The `IStarknetMessaging.sendMessageToL2` interface is `payable` with no minimum-fee requirement â€” the comment explicitly states "the payed amount is the message fee" with no floor: [4](#0-3) 

The production Starknet messaging contract stores L1â†’L2 messages keyed by `msg.value` as the fee. A zero-fee message is registered on-chain but the sequencer will not automatically process it. Neither the `IBridge` interface nor `IStarknetMessaging` enforce a non-zero fee: [5](#0-4) 

Before the zero-fee calls, the function has already irreversibly consumed the L2â†’L1 mint request messages and minted tokens: [6](#0-5) 

---

### Impact Explanation

- **L2â†’L1 mint request messages are consumed** â€” irreversible; the L2 reward supplier's `l1_pending_requested_amount` is never decremented (since `on_receive` is never called on L2), causing permanent accounting desynchronization.
- **Minted tokens are stuck in the bridge** â€” up to 5 Ă— 1,300,000 STRK per call, temporarily frozen until a manual fee top-up.
- **L2 `MintingCurve` totalSupply is stale** â€” the zero-fee `update_total_supply` message is never processed, causing the minting curve to compute incorrect yearly mint amounts until a legitimate `tick()` call with sufficient fee is made.

This matches **Medium: Griefing with no profit motive but damage to users or protocol**.

---

### Likelihood Explanation

The precondition (pending L2â†’L1 mint request messages) is a normal operational state â€” it occurs every time the L2 reward supplier requests funds. The attack requires only a single public call with `msg.value=0`, which any EOA can execute at any time. No privileged role, leaked key, or external compromise is needed.

---

### Recommendation

Add a minimum fee guard at the top of `tick()`:

```solidity
function tick() external payable {
    require(msg.value >= MIN_TICK_FEE, "INSUFFICIENT_FEE");
    ...
}
```

Or, more precisely, require that each of the two L1â†’L2 messages receives at least `messagingContract().getMaxL1MsgFee()` (or a protocol-defined minimum). The `getMaxL1MsgFee()` function is already available on the interface: [7](#0-6)
