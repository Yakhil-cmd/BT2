### Title
Silo `Address` Whitelist Not Enforced for Base Token (`receive_base_tokens`) Bridge Minting - (`engine/src/engine.rs`)

### Summary

In Silo mode, the `Address` whitelist is intended to restrict which EVM addresses can receive tokens bridged into Aurora. The `receive_erc20_tokens` path correctly enforces this restriction and redirects tokens to a configured `fallback_address` when the recipient is not whitelisted. However, the `receive_base_tokens` path — triggered when the ETH connector bridges native ETH — performs no whitelist check at all, allowing any EVM address to receive base tokens regardless of whitelist status.

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` dispatches to one of two paths depending on whether the caller is the ETH connector: [1](#0-0) 

When the caller is the ETH connector, `receive_base_tokens` is called. This function directly mints ETH to the recipient address parsed from `args.msg` with no whitelist validation: [2](#0-1) 

By contrast, `receive_erc20_tokens` explicitly checks `silo::is_allow_receive_erc20_tokens` and redirects to the `fallback_address` when the recipient is not whitelisted: [3](#0-2) 

The whitelist check itself is: [4](#0-3) 

The `Address` whitelist is disabled by default and only active when the silo operator explicitly enables it: [5](#0-4) 

When enabled, the silo operator's intent is that non-whitelisted addresses cannot receive tokens — those tokens should instead flow to the `fallback_address`. This intent is enforced for ERC-20 tokens but is completely absent for base tokens (ETH).

### Impact Explanation

**High — Theft of unclaimed yield / whitelist bypass.** When a silo operator enables the `Address` whitelist and configures a `fallback_address`, they expect that tokens bridged to non-whitelisted EVM addresses will be redirected to the fallback. For ERC-20 tokens this works correctly. For base tokens (ETH), the fallback mechanism is never invoked: the non-whitelisted recipient receives the ETH directly, and the fallback address receives nothing. The silo operator loses the tokens that should have been captured by the fallback mechanism, and a non-whitelisted address gains assets it should not have access to within the silo.

### Likelihood Explanation

**High.** Any silo deployment that enables the `Address` whitelist and configures a `fallback_address` is affected. The ETH bridge (`ft_on_transfer` from the ETH connector) is a standard, frequently used operation. Any user who initiates a bridge transfer of ETH to a non-whitelisted EVM address will trigger this bypass automatically — no special knowledge or exploit construction is required.

### Recommendation

Apply the same whitelist-and-fallback logic to `receive_base_tokens` that is already applied in `receive_erc20_tokens`. Before minting base tokens to the parsed recipient, check `silo::is_allow_receive_erc20_tokens` and, if the recipient is not whitelisted and a fallback address is configured, redirect the mint to the fallback address instead.

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with `set_silo_params` (sets `fixed_gas` and `erc20_fallback_address`).
2. Enable the `Address` whitelist via `set_whitelist_status(WhitelistKind::Address, active: true)`.
3. Do **not** add victim EVM address `A` to the whitelist.
4. From the NEAR side, initiate a bridge transfer of ETH to address `A` via the ETH connector, which calls `ft_on_transfer` on Aurora with `predecessor_account_id == connector_account_id`.
5. `receive_base_tokens` is called. It parses `A` from `args.msg` and calls `set_balance` directly — no whitelist check occurs.
6. Address `A` now holds ETH inside the silo despite not being whitelisted. The `fallback_address` receives nothing.
7. For comparison, repeat with an ERC-20 token: the tokens are correctly redirected to `fallback_address` because `receive_erc20_tokens` checks `is_allow_receive_erc20_tokens`. [2](#0-1) [3](#0-2) [1](#0-0)
