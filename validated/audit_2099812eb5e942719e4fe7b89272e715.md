### Title
Fee Parameter Not Shown in WalletConnect `create_offer_for_ids` Approval Dialog - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
When a dApp sends a `chia_wallet.create_offer_for_ids` WalletConnect request that includes a `fee` parameter, the `parseCommandDisplay` function ignores `params.fee` entirely and passes `undefined` to the display layer. The WalletConnect approval dialog therefore shows no fee line, even though the actual RPC call will deduct the fee from the user's XCH balance. A malicious dApp can silently inflate the fee to drain XCH beyond what the user believes they are approving.

### Finding Description
In `parseCommandDisplay`, the `chia_wallet.take_offer` branch correctly reads `summary.fees` and passes it to `walletDeltaToDisplay`:

```typescript
const fees = parseMojos(summary.fees);
return {
  walletDelta: await walletDeltaToDisplay(..., fees),
};
```

The `chia_wallet.create_offer_for_ids` branch, however, hardcodes `undefined` as the fee argument and never reads `params.fee`:

```typescript
return {
  walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
};
```

`walletDeltaToDisplay` only emits a fee line when `fee !== undefined`:

```typescript
fee: fee !== undefined ? mojoToChiaLocaleString(fee) : undefined,
```

So the "Offer Fees" row in the `WalletDeltaSection` confirmation dialog is never rendered for `create_offer_for_ids`, regardless of what fee the dApp requested.

The `fee` parameter is a fully documented, non-optional-in-practice field for this command:

```typescript
{ name: 'fee', label: () => i18n._({ id: 'Fee' }), type: 'bigint', humanize: 'mojo-to-xch', isOptional: true }
```

and is forwarded verbatim to the wallet RPC, which will deduct it from the user's spendable XCH balance.

### Impact Explanation
A connected dApp can craft a `chia_wallet.create_offer_for_ids` payload with a small, visible offer amount and a large, hidden `fee`. The user sees only the offer amounts in the approval dialog and approves, unaware that the fee will drain additional XCH. This is a direct, unprivileged asset-loss path: the user approves what appears to be a small spend but the actual on-chain transaction deducts significantly more XCH.

### Likelihood Explanation
Any dApp that has obtained a WalletConnect session can trigger this. No leaked keys, host compromise, or social engineering beyond the initial (legitimate) WalletConnect pairing is required. The dApp simply includes a large `fee` value in the RPC params.

### Recommendation
In the `chia_wallet.create_offer_for_ids` branch of `parseCommandDisplay`, read and parse `params.fee` and pass it to `walletDeltaToDisplay`:

```typescript
const fee = params.fee !== undefined ? parseMojos(params.fee) : undefined;
return {
  walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fee),
};
```

This mirrors the existing pattern used for `chia_wallet.take_offer`.

### Proof of Concept
1. A dApp establishes a WalletConnect session with the Chia GUI.
2. The dApp sends:
   ```json
   {
     "command": "chia_wallet.create_offer_for_ids",
     "params": {
       "offer": { "1": "-1000000000" },
       "fee": "10000000000000"
     }
   }
   ```
3. `parseCommandDisplay` builds the display with `undefined` fee → the approval dialog shows "You Spend: 0.001 XCH" and no fee row.
4. The user clicks "Create".
5. The wallet RPC executes with `fee: 10000000000000` mojos (10 XCH), deducting 10.001 XCH total from the user's balance.
