### Title
Unvalidated Participant Contributions Corrupt CKD Output Accumulator — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
In `do_ckd_coordinator`, the coordinator blindly accumulates `norm_big_y` and `norm_big_c` from every participant with no proof that each participant computed their share using the agreed-upon `app_id` and `app_pk`. A single malicious participant can substitute values derived from an attacker-chosen `app_id` or `app_pk`, silently corrupting the aggregated `CKDOutput` that honest parties accept as valid.

### Finding Description
`do_ckd_coordinator` initialises two running sums from the coordinator's own share and then adds every other participant's `CKDOutput` unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each honest participant is expected to compute their share as:

```
big_c = x_i · H(pk ∥ app_id) + y_i · app_pk
norm_big_c = λ_i · big_c
``` [2](#0-1) 

There is no zero-knowledge proof, commitment, or any other mechanism attached to the `CKDOutput` message that would let the coordinator verify the participant actually used the protocol-agreed `app_id` and `app_pk`. The coordinator simply deserialises two `G1` points and adds them to the accumulator.

Contrast this with `do_keyshare` in `src/dkg.rs`, where every received signing share is explicitly verified against the sender's published polynomial commitment before being accumulated:

```rust
validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
my_signing_share = my_signing_share + signing_share_from.to_scalar();
``` [3](#0-2) 

No equivalent guard exists in the CKD accumulation loop.

The structural parallel to the reported `CreditVault` bug is direct: just as `fundingFees` aggregates raw token amounts from heterogeneous assets into one scalar without per-asset accounting, `norm_big_y`/`norm_big_c` aggregate raw elliptic-curve points from all participants without per-participant correctness verification. In both cases the accumulator silently mixes contributions from different "contexts" (different tokens / different `app_id` inputs), and the downstream consumer of the aggregate has no way to detect the contamination.

### Impact Explanation
A malicious participant sends `(big_y', big_c')` computed with an attacker-chosen `app_id'` or `app_pk'` instead of the agreed values. The coordinator produces:

```
CKDOutput = (Σ λ_i · y_i · G + big_y',  Σ λ_i · (x_i · H(pk∥app_id) + y_i·app_pk) + big_c')
```

The application then calls `unmask(app_sk)` expecting `msk · H(pk∥app_id)`, but receives a corrupted group element. The derived confidential key is wrong and unusable, and honest parties have no way to detect the corruption because the coordinator returns a single opaque `CKDOutput`.

**Mapped impact:** High — Corruption of CKD outputs so honest parties accept an unusable cryptographic output.

### Likelihood Explanation
Any single participant in the protocol can trigger this. The participant already receives `app_id` and `app_pk` as protocol inputs, so they know the correct values and can trivially substitute alternatives. No special privilege, leaked key, or external assumption is required — only the ability to send a crafted message on the private channel to the coordinator, which is the normal communication path.

### Recommendation
Require each participant to attach a non-interactive zero-knowledge proof (e.g., a Schnorr-style proof of correct ElGamal encryption) demonstrating that `big_c` was formed as `x_i · H(pk∥app_id) + y_i · app_pk` for the agreed `app_id` and `app_pk`, and that `big_y = y_i · G`. The coordinator must verify every proof before adding the contribution to the accumulator, mirroring the `validate_received_share` pattern used in `do_keyshare`.

### Proof of Concept
1. Honest participants agree on `app_id = b"NearApp"` and `app_pk = A`.
2. Malicious participant `P_m` instead computes `big_c' = x_m · H(pk∥app_id') + y_m · A'` for attacker-chosen `app_id'` and `A'`, and `big_y' = y_m · G`.
3. `P_m` sends `CKDOutput { big_y: big_y', big_c: big_c' }` to the coordinator over the private channel.
4. `do_ckd_coordinator` adds `big_y'` and `big_c'` to the running sums with no check.
5. The final `CKDOutput` is returned to the application; `unmask(app_sk)` yields a garbage group element instead of `msk · H(pk∥app_id)`.
6. All honest parties accept this corrupted output as the protocol result.
