### Title
Staker Can Inflate `max_commission` in `set_commission_commitment` to Bypass Commission-Decrease Restriction and Steal Delegator Yield — (File: `src/staking/staking.cairo`)

---

### Summary

A staker can call `set_commission_commitment` with `max_commission = COMMISSION_DENOMINATOR` (10000 = 100%). Once the commitment is active, the `update_commission` internal function drops the normal "commission can only decrease" guard and replaces it with the weaker check `commission <= max_commission`. Because `max_commission` was set to 10000, the staker can raise commission from any low value (e.g. 2%) all the way to 100%, redirecting the entirety of delegator rewards to themselves.

---

### Finding Description

`set_commission_commitment` in `src/staking/staking.cairo` accepts any `max_commission` value between `current_commission` and `COMMISSION_DENOMINATOR` (10000):

```
assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
...
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
``` [1](#0-0) 

There is no upper bound tighter than 10000. A staker with commission 200 (2%) can legitimately call `set_commission_commitment(max_commission: 10000, expiration_epoch: current + 1)`.

Once the commitment is stored, `update_commission` (called from `set_commission`) branches on whether the commitment is active:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
} else {
    assert!(commission < old_commission, ...);
}
``` [2](#0-1) 

When the commitment is active the only constraints are:
1. `commission <= max_commission` — satisfied for any value ≤ 10000.
2. `commission != old_commission` — trivially satisfied by choosing a different value.

The invariant `commission < old_commission` (commission can only decrease) is **completely absent** from the active-commitment branch. A staker who set `max_commission = 10000` can therefore call `set_commission(commission: 10000)` and raise their rate to 100%.

The `set_commission` entry point itself only checks `commission <= COMMISSION_DENOMINATOR`: [3](#0-2) 

No additional floor or ceiling is applied before delegating to `update_commission`.

---

### Impact Explanation

Commission is deducted from pool rewards before they are forwarded to the pool contract:

```cairo
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [4](#0-3) 

At `commission = 10000`, `split_rewards_with_commission` returns `pool_rewards = 0` and `commission_rewards = total`. All delegator yield is redirected to the staker's reward address. Delegators receive nothing for every epoch the inflated commission is in effect.

**Impact class**: High — Theft of unclaimed yield. Delegators who staked based on an advertised low commission rate lose all accrued rewards for the duration of the inflated commission.

---

### Likelihood Explanation

- The attack requires no privileged role beyond being a registered staker with a pool.
- `set_commission_commitment` is a public staker-callable function.
- The commitment window is capped at one year (`expiration_epoch - current_epoch <= epochs_in_year`), but even a single-epoch commitment is sufficient to drain one full epoch of delegator rewards.
- The `CommissionCommitmentSet` event is emitted, but delegators have no on-chain mechanism to exit before the next epoch's rewards are calculated; the `exit_wait_window` (default 1 week) means they cannot withdraw in time if the epoch is short.
- The attack is cheap, repeatable after each commitment expires, and leaves no on-chain evidence distinguishing it from a legitimate commitment.

---

### Recommendation

Add a check in `set_commission_commitment` that `max_commission` does not exceed the current commission by more than a protocol-defined delta, **or** enforce `commission < old_commission` even when a commitment is active (allowing only decreases within the committed ceiling):

```cairo
// In update_commission, active-commitment branch:
assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
assert!(commission <= commission_commitment.max_commission, "{}", Error::INVALID_COMMISSION_WITH_COMMITMENT);
```

This mirrors the fusion-swap recommendation: treat the user-supplied ceiling as a protection bound, not as a free pass to raise the value to any level up to that ceiling.

---

### Proof of Concept

1. Staker stakes with `commission = 200` (2%) and opens a pool.
2. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
   - Passes: `10000 <= COMMISSION_DENOMINATOR` ✓, `200 <= 10000` ✓.
3. Within the same epoch, staker calls `set_commission(commission: 10000)`.
   - `update_commission` active-commitment branch: `10000 <= 10000` ✓, `10000 != 200` ✓. Commission is written as 10000.
4. Attestation fires; `calculate_staker_pools_rewards` computes `commission_rewards = pool_rewards_including_commission * 10000 / 10000 = pool_rewards_including_commission`, `pool_rewards = 0`.
5. Pool contract receives 0 STRK; staker's `unclaimed_rewards_own` absorbs the full pool reward.
6. Delegators call `claim_rewards` and receive 0. [5](#0-4) [6](#0-5) [7](#0-6)

            // Prerequisites and asserts.
            self.general_prerequisites();

                    );

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },

pub(crate) fn split_rewards_with_commission(

) -> (Amount, Amount) {

        :rewards_including_commission, :commission,
