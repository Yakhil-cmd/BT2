### Title
Hardcoded ECDSA Private Key Used in Production Consensus Signature Manager - (`File: crates/apollo_signature_manager/src/lib.rs`)

### Summary

The production `create_signature_manager()` function unconditionally instantiates `LocalKeyStore` with a hardcoded STARK-curve private key (`0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133`). This key is compiled into every deployed sequencer binary and is publicly readable from the repository. Any party who reads the source can compute valid ECDSA signatures over arbitrary messages using the sequencer's identity key, forging peer-identity challenges and consensus precommit-vote signatures.

### Finding Description

`LocalKeyStore::new_for_testing()` in `crates/apollo_signature_manager/src/signature_manager.rs` embeds a fixed private/public key pair as compile-time constants:

```rust
pub(crate) const fn new_for_testing() -> Self {
    const PRIVATE_KEY: PrivateKey = PrivateKey(Felt::from_hex_unchecked(
        "0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133",
    ));
    ...
}
``` [1](#0-0) 

Despite the `_for_testing` name, this constructor is called from the non-`#[cfg(test)]` production path in `crates/apollo_signature_manager/src/lib.rs`:

```rust
impl LocalKeyStoreSignatureManager {
    pub fn new() -> Self {
        Self(GenericSignatureManager::new(LocalKeyStore::new_for_testing()))
    }
}
// ...
pub fn create_signature_manager() -> SignatureManager {
    SignatureManager::new()
}
``` [2](#0-1) 

The accompanying TODO comment explicitly acknowledges the key store is not production-ready:

> `// TODO(Elin): understand how key store would look in production and better define the way the signature manager is created.` [3](#0-2) 

`create_signature_manager()` is called directly from the node's production component initialization in `crates/apollo_node/src/components.rs`:

```rust

The `SignatureManager` is wired into the `ConsensusManager` as `signature_manager_client` and exposes two signing operations:

- `sign_precommit_vote(block_hash)` — signs BFT consensus precommit votes
- `sign_identification(peer_id, challenge)` — signs peer-identity challenges for node authentication [5](#0-4) 

The `ConsensusManager` receives this client at construction time: [6](#0-5) 

### Impact Explanation

Because every deployed sequencer node uses the same publicly known private key, an attacker can:

1. Derive the key from the open-source repository.
2. Compute a valid ECDSA signature over any `(peer_id, challenge)` tuple, passing peer-identity verification as the legitimate sequencer node.
3. Compute a valid ECDSA signature over any `block_hash`, forging a precommit vote attributed to the sequencer's identity key.

If signature verification on received votes is enforced (or when it is enforced as the TODO is resolved), a single attacker can forge a full quorum of precommit votes for an arbitrary block hash, because all validators share the same key. This breaks the BFT invariant that a 2/3 quorum of distinct validators must agree.

**Impact category:** High — consensus signature/hash logic binds the wrong signer; the precommit-vote and peer-identity signatures are attributable to a publicly known key rather than a secret per-node identity.

### Likelihood Explanation

The private key is in plain text in the public repository. No special access, privilege, or network position is required. Any reader of the source code can immediately derive valid signatures. Likelihood is high.

### Recommendation

1. Remove `LocalKeyStore::new_for_testing()` from all non-`#[cfg(test)]` code paths.
2. Implement a production `KeyStore` that loads the private key from a secure external source (HSM, encrypted config file, environment secret) at startup.
3. Gate `LocalKeyStore::new_for_testing()` with `#[cfg(any(test, feature = "testing"))]` so it cannot be compiled into a production binary.
4. Rotate the exposed key immediately; treat `0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133` as permanently compromised.

### Proof of Concept

```
# 1. Read the private key from the public source:
PRIVATE_KEY=0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133

# 2. For any target block_hash, compute the precommit-vote message digest
#    using the same domain separator and blake2s hash as build_precommit_vote_message_digest():
#    message = b"PRECOMMIT_VOTE" || block_hash.to_bytes_be()
#    digest  = blake2s_to_felt(message)

# 3. Sign the digest with PRIVATE_KEY using starknet ECDSA:
#    signature = ecdsa_sign(PRIVATE_KEY, digest)

# 4. The resulting (r, s) is a valid precommit-vote signature accepted by
#    verify_precommit_vote_signature(block_hash, signature, PUBLIC_KEY)
#    where PUBLIC_KEY = 0x125d56b1fbba593f1dd215b7c55e384acd838cad549c4a2b9c6d32d264f4e2a
#    — the same key every production node presents.

fn build_precommit_vote_message_digest(block_hash: BlockHash) -> MessageDigest {
    let block_hash = block_hash.to_bytes_be();
    let mut message = Vec::with_capacity(PRECOMMIT_VOTE.len() + block_hash.len());
    message.extend_from_slice(PRECOMMIT_VOTE);
    message.extend_from_slice(&block_hash);

    MessageDigest(blake2s_to_felt(&message))
}
