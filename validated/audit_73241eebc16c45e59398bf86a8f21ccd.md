### Title
`receive_erc20_tokens` Mints ERC-20 Mirror Tokens Using Unvalidated Callback Amount, Enabling Over-Issuance With Fee-on-Transfer NEP-141 Tokens - (`engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` mints Aurora ERC-20 mirror tokens using `args.amount` taken directly from the NEP-141 `ft_on_transfer` callback argument, without cross-validating that this amount equals the actual NEP-141 balance Aurora received. When a NEP-141 token with a fee-on-transfer mechanism is bridged, Aurora mints more ERC-20 tokens than it holds in NEP-141 backing, permanently breaking the 1:1 accounting invariant and causing insolvency for that token's bridge pool.

---

### Finding Description

The inbound bridge flow for NEP-141 tokens is:

1. A user calls `ft_transfer_call(receiver_id=aurora, amount=X, msg=<evm_address>)` on a NEP-141 contract.
2. The NEP-141 contract transfers tokens to Aurora (potentially deducting a fee), then calls `ft_on_transfer(sender_id, amount=X, msg)` on Aurora — passing the **original requested amount**, not the post-fee amount actually credited to Aurora's balance.
3. Aurora's `ft_on_transfer` dispatches to `receive_erc20_tokens`.

Inside `receive_erc20_tokens`:

```rust

// ...

```

`args.amount` is the value supplied by the NEP-141 contract in the callback — the pre-fee amount. Aurora then mints exactly `amount` ERC-20 tokens:

`setup_receive_erc20_tokens_input` encodes a `mint(recipient, amount)` call to the `EvmErc20` contract:

There is no check that `args.amount` equals the actual NEP-141 balance increase Aurora received. The only link between the NEP-141 transfer and the ERC-20 mint is the callback argument, which is fully controlled by the NEP-141 contract and reflects the pre-fee amount.

**Accounting invariant broken:**

Let `B_nep141` = actual NEP-141 tokens credited to Aurora = `X - fee`.
Let `B_erc20` = ERC-20 tokens minted on Aurora = `X`.

The invariant `B_nep141 == B_erc20` is violated by `fee` on every deposit. The gap is cumulative and irreversible.

**Attacker-controlled entry path:**

1. Deploy a NEP-141 token with a fee-on-transfer mechanism (e.g., a 1% fee deducted from the transferred amount before crediting the receiver).
2. Call `deploy_erc20_token` on Aurora — this function has **no access control** beyond `require_running`, so any unprivileged account can register any NEP-141:

4. Repeat. Each deposit widens the gap. Eventually, when users exit via `ExitToNear` (burning ERC-20 tokens and triggering `ft_transfer` on the NEP-141 contract), Aurora's NEP-141 balance is insufficient to cover all outstanding ERC-20 tokens. The last users to exit lose funds permanently.

---

### Impact Explanation

**Impact: High — Insolvency / Permanent Fund Freeze**

Aurora's ERC-20 mirror tokens become over-issued relative to the NEP-141 backing. The bridge pool for the affected token becomes insolvent: the last users to exit cannot redeem their ERC-20 tokens for NEP-141 tokens because Aurora's NEP-141 balance is exhausted. Their ERC-20 tokens are permanently frozen with no backing.

---

### Likelihood Explanation

**Likelihood: Medium**

- `deploy_erc20_token` is permissionless — any account can register any NEP-141 token.
- A malicious actor can deploy a custom NEP-141 with a fee-on-transfer and register it on Aurora without any admin approval.
- Legitimate NEP-141 tokens with fees are less common on NEAR than on Ethereum, but the attack does not require a pre-existing token — the attacker can deploy one.
- No special privileges, leaked keys, or governance capture are required.

---

### Recommendation

In `receive_erc20_tokens`, after the ERC-20 mint succeeds, verify that the actual NEP-141 balance of the Aurora contract increased by at least `args.amount`. This requires a cross-contract view call to the NEP-141 contract before and after the transfer, or alternatively, Aurora should return `args.amount` (refund all) from `ft_on_transfer` for any NEP-141 token whose actual credited amount differs from the callback amount. A simpler mitigation is to document and enforce that only NEP-141 tokens without fee-on-transfer mechanisms may be registered via `deploy_erc20_token`, enforced by an allowlist.

---

### Proof of Concept

1. Deploy a NEAR contract implementing NEP-141 with a 1% fee deducted from the receiver's credited amount on every `ft_transfer_call`.
2. Call `aurora.deploy_erc20_token(nep141_account_id)` — succeeds with no access control.
3. Call `nep141.ft_transfer_call(receiver_id="aurora", amount="1000", msg="<evm_address>")`.
   - NEP-141 credits Aurora with 990 tokens (1% fee).
   - NEP-141 calls `aurora.ft_on_transfer(sender_id=attacker, amount="1000", msg="<evm_address>")`.
4. Aurora's `receive_erc20_tokens` reads `args.amount = 1000`, mints 1000 ERC-20 tokens to `<evm_address>`.
5. Aurora's NEP-141 balance: 990. Aurora's ERC-20 supply: 1000. Gap: 10.
6. Repeat N times. Gap grows to `N * 10`.
7. When legitimate users attempt to exit via `withdrawToNear` → `ExitToNear` precompile → `ft_transfer` on NEP-141, the last `N * 10` tokens worth of exits fail because Aurora's NEP-141 balance is exhausted.     pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
