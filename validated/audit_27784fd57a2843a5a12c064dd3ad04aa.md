### Title
Missing Validation That Signing Threshold Matches Presigning Threshold Enables Denial of Signing - (File: src/ecdsa/ot_based_ecdsa/sign.rs)

### Summary
The OT-based ECDSA `sign` function accepts a caller-supplied `threshold` parameter independently from the `presignature: RerandomizedPresignOutput`, without verifying that the two are consistent. Because `RerandomizedPresignOutput` does not carry the presigning threshold, the `sign` function cannot detect a mismatch. A malicious coordinator can pass a `threshold` lower than the one used during presigning, causing signing to proceed with fewer participants than the polynomial degree requires. The resulting incorrect Lagrange interpolation produces invalid signature shares that fail the coordinator's final verification, permanently denying signing for that presignature.

### Finding Description
In `src/ecdsa/ot_based_ecdsa/sign.rs`, the `sign` function performs only one threshold-related check:

```rust
// ensure number of participants during the signing phase is >= threshold
if participants.len() < threshold {
    return Err(InitializationError::NotEnoughParticipantsForThreshold { ... });
}
``` [1](#0-0) 

This check only validates that the caller-supplied `threshold` does not exceed the participant count. It does **not** validate that `threshold` matches the threshold used during presigning.

During presigning (`src/ecdsa/ot_based_ecdsa/presign.rs`), the threshold is validated against the triple thresholds:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [2](#0-1) 

However, this presigning threshold is **not** embedded in `RerandomizedPresignOutput`. The output only carries `big_r`, `k`, and `sigma`:

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<Scalar, ProtocolError> {
    let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
    let k_i = lambda * presignature.k;
    let sigma_i = lambda * presignature.sigma;
    ...
}
``` [3](#0-2) 

The `sign` function therefore has no way to enforce consistency between the caller-supplied `threshold` and the presigning threshold. A malicious coordinator calls `sign` with `threshold = t_low < t_presign` and only `t_low` participants. The check `participants.len() < threshold` passes. `compute_signature_share` computes Lagrange coefficients over only `t_low` participants for a degree-`(t_presign - 1)` polynomial — fewer points than the polynomial degree — producing incorrect signature shares. The coordinator's final check:

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed("signature failed to verify".to_string()));
}
``` [4](#0-3) 

fails, and the signing session is aborted. The presignature is consumed (passed by value, one-time use per library design) and cannot be reused.

This is the direct analog of the external report's missing `msg.value == amount` check: two values that must be equal (`threshold` in `sign` vs. `threshold` used in `presign`) are never compared, and the mismatch causes the protocol to produce an incorrect output.

### Impact Explanation
**High — Permanent denial of signing for honest parties.** A malicious coordinator can systematically pass a mismatched `threshold` (e.g., `threshold = 1`) to every `sign` invocation. Each call passes the participant-count check, proceeds to compute incorrect signature shares, and fails at final verification. Because each presignature is one-time use, every failed session permanently destroys a presignature. Honest participants cannot detect the mismatch because `RerandomizedPresignOutput` exposes no threshold field. The coordinator can repeat this indefinitely, exhausting the presignature pool and permanently denying signing.

### Likelihood Explanation
**Medium.** The coordinator role requires no privileged keys — any participant can act as coordinator. A malicious participant who becomes coordinator can trivially pass `threshold = 1` to every `sign` call. No cryptographic break, no leaked secrets, and no external assumptions are required. The attack is deterministic and repeatable.

### Recommendation
Embed the presigning threshold in `RerandomizedPresignOutput` and add a validation in `sign` that the caller-supplied `threshold` matches the embedded value:

```rust
if threshold != presignature.threshold {
    return Err(InitializationError::BadParameters(
        "signing threshold must match presigning threshold".to_string(),
    ));
}
```

Alternatively, remove the `threshold` parameter from `sign` entirely and derive it exclusively from the presignature, eliminating the possibility of a mismatch.

### Proof of Concept
1. Run presigning with `threshold = 3` and 3 participants `{P1, P2, P3}`, producing `RerandomizedPresignOutput` for each (shares lie on a degree-2 polynomial).
2. Malicious coordinator calls:
   ```rust
   sign(
       participants: &[P1, P2],   // only 2 participants
       coordinator: C,
       threshold: 2,              // mismatched: presigning used 3
       me: C,
       public_key,
       presignature,              // shares from degree-2 polynomial
       msg_hash,
   )
   ```
3. Check `participants.len() < threshold` → `2 < 2` → false → passes.
4. Each participant computes `lambda = participants.lagrange(me)` over `{P1, P2}` — a 2-point Lagrange basis applied to degree-2 polynomial shares.
5. Signature shares are incorrect (wrong interpolation).
6. Coordinator sums shares, constructs `sig`, calls `sig.verify(&public_key, &msg_hash)` → fails.
7. Returns `ProtocolError::AssertionFailed("signature failed to verify")`.
8. Presignature is consumed. Signing is permanently denied for this presignature. Coordinator repeats for every subsequent presignature.
