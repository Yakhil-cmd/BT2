### Title
`validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress`, Enabling Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account signatures are valid over an arbitrary `signedData` blob, then separately checks that the COA at the given `path` matches `evmAddress`. Because `signedData` is never required to commit to `evmAddress`, a valid signature produced for one COA's ownership proof can be replayed verbatim to forge a proof for any other COA owned by the same Cadence account. The code itself documents this gap at L1003‚Äì1009 but dismisses it as low-risk under a single-COA-per-account assumption that is not enforced by the protocol.

---

### Finding Description

`validateCOAOwnershipProof` (L1011‚Äì1116 of `fvm/evm/stdlib/contract.cdc`) performs two independent checks:

1. **Signature check** (L1082‚Äì1086): `keyList.verify(signatureSet: signatureSet, signedData: signedData, domainSeparationTag: "FLOW-V0.0-user")` ‚Äî verifies that the supplied signatures are valid over the caller-supplied `signedData` blob.

2. **Address check** (L1095‚Äì1109): borrows the COA capability at the caller-supplied `path` and compares its EVM address byte-by-byte against the caller-supplied `evmAddress`.

There is no step that ties these two checks together. `signedData` is never required to encode, hash, or otherwise commit to `evmAddress`. The two checks are therefore orthogonal: a signature that passes check 1 for COA_A will also pass check 1 for COA_B, because the same `signedData` is accepted for both. The attacker only needs to supply the correct `path` and `evmAddress` for the target COA in check 2.

The function is `access(all)` and is reachable via two independent paths:

- **EVM path**: any EVM contract (including one deployed by an unprivileged user) can call the Cadence Arch precompile at `verifyCOAOwnershipProof(address,bytes32,bytes)`, which invokes `validateCOAOwnershipProof` through `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go` (L115‚Äì153). The `bytes32` argument is the `signedData`; the caller controls it entirely.

- **Cadence path**: any Cadence script or transaction can call `EVM.validateCOAOwnershipProof(...)` directly, supplying all six parameters including `signedData` and `evmAddress`.

The code comment at L1003‚Äì1009 explicitly acknowledges the gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The mitigation offered ‚Äî "Flow wallets historically create at most one COA per account" ‚Äî is a social convention, not a protocol invariant. Nothing in the contract prevents an account from holding multiple COAs at different paths.

---

### Impact Explanation

An attacker who possesses a valid COA ownership proof signature from a victim (obtained from any prior legitimate ERC-1271 authentication flow, off-chain signing request, or any other use of the `"FLOW-V0.0-user"` domain tag) can replay it to forge a passing `ValidationResult` for any other COA owned by the same Cadence account. A forged `ValidationResult{isValid: true}` returned by `validateCOAOwnershipProof` causes the EVM-side `isValidSignature` in `coa.sol` (L114‚Äì125) to return `ValidERC1271Signature (0x1626ba7e)`, making the COA appear to have signed a message it never signed. Any on-chain system that relies on ERC-1271 for authentication or authorization (e.g., Safe multisig, token-gated contracts, permit-style approvals) would accept the forged signature as genuine, potentially granting unauthorized access to assets or operations controlled by the target COA.

---

### Likelihood Explanation

The attack requires the adversary to hold a valid Cadence account signature over some `signedData` blob produced by the victim under the `"FLOW-V0.0-user"` domain tag. This is obtainable from:

- A prior legitimate `verifyCOAOwnershipProof` call the victim made for a different COA or a different application context.
- Any off-chain or on-chain signing flow that uses the same domain tag (e.g., message-signing dApps, wallet authentication flows).

As multi-COA usage grows (DeFi protocols, DAOs, and multi-purpose accounts routinely deploy several COAs), the pool of replayable signatures widens. The likelihood is **medium**: the attacker does not need to compromise any key or node, only to observe or solicit a single prior signature from the victim.

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. The simplest protocol-level fix is to require callers to encode `evmAddress` into `signedData` and verify this binding before calling `keyList.verify`:

```cadence
// Require signedData to be at least 20 bytes and that its last 20 bytes equal evmAddress
pre {
    signedData.length >= 20:
        "signedData must encode evmAddress"
}
// Then verify the suffix matches evmAddress before proceeding
let dataLen = signedData.length
for i, b in evmAddress {
    if signedData[dataLen - 20 + i] != b {
        return ValidationResult(isValid: false, problem: "signedData does not commit to evmAddress")
    }
}
```

Alternatively, document a mandatory encoding convention (e.g., `signedData = keccak256(abi.encode(evmAddress, applicationData))`) and enforce it at the protocol level rather than leaving it as an off-chain recommendation.

### Proof of Concept

1. Alice deploys two COAs: `COA_A` stored at `/public/coa1` (EVM address `0xAAAA‚Ä¶`) and `COA_B` stored at `/public/coa2` (EVM address `0xBBBB‚Ä¶`).
2. A legitimate dApp asks Alice to prove ownership of `COA_A`. Alice signs `signedData = 0xdeadbeef‚Ä¶` and the dApp calls `EVM.validateCOAOwnershipProof(address: Alice, path: /public/coa1, signedData: 0xdeadbeef‚Ä¶, keyIndices: [0], signatures: [aliceSig], evmAddress: 0xAAAA‚Ä¶)` ‚Üí `isValid: true`.
3. The attacker, having observed `aliceSig` and `signedData`, calls:
   ```cadence
   EVM.validateCOAOwnershipProof(
       address: Alice,
       path: /public/coa2,          // ‚Üê target COA_B
       signedData: 0xdeadbeef‚Ä¶,     // ‚Üê same signed data
       keyIndices: [0],
       signatures: [aliceSig],      // ‚Üê replayed signature
       evmAddress: 0xBBBB‚Ä¶          // ‚Üê COA_B's address
   )
   ```
4. Step 1 (signature check) passes because `aliceSig` is valid over `0xdeadbeef‚Ä¶`. Step 2 (address check) passes because `COA_B`'s address is indeed `0xBBBB‚Ä¶`. The function returns `ValidationResult{isValid: true}`.
5. Via the EVM precompile path, the attacker deploys a contract that calls `verifyCOAOwnershipProof(Alice_COA_B_address, keccak256(0xdeadbeef‚Ä¶), encodedProof)` ‚Äî the precompile returns `true`, and `COA_B.isValidSignature(hash, sig)` returns `0x1626ba7e`, forging an ERC-1271 approval for `COA_B`.
