Audit Report

## Title
Unprivileged Caller Can DoS `update_balance` by Exhausting Guard Slots via Arbitrary `owner` Parameter - (File: `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs`)

## Summary
The `update_balance` endpoint keys its concurrency guard on the caller-supplied `args.owner` rather than on the actual `msg_caller()`. Any non-anonymous principal can supply 100 distinct arbitrary `owner` principals to fill all available guard slots (`MAX_CONCURRENT = 100`), causing every subsequent legitimate `update_balance` call to fail with `TooManyConcurrentRequests`. Since `update_balance` is the sole mechanism for minting ckBTC after a BTC deposit, this constitutes a complete, repeatable DoS of the ckBTC deposit flow.

## Finding Description
In `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` at lines 164–168, the guard account is constructed from the caller-supplied `args.owner`:

```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),   // attacker-controlled
    subaccount: args.subaccount,
};
let _guard = balance_update_guard(caller_account)?;
```

`balance_update_guard` delegates to `Guard::new(account)` in `rs/bitcoin/ckbtc/minter/src/guard.rs` (lines 45–60), which inserts the account into `state.update_balance_accounts`. If `accounts.len() >= MAX_CONCURRENT` (line 51), it returns `GuardError::TooManyConcurrentRequests`. `MAX_CONCURRENT` is hardcoded to `100` (line 6).

The guard is held across the async `get_utxos` call at line 175 of `update_balance.rs`, meaning each in-flight call occupies a slot for the duration of the Bitcoin canister round-trip. The entry point in `rs/bitcoin/ckbtc/minter/src/main.rs` (lines 197–199) only calls `check_anonymous_caller()`, imposing no further restrictions. An attacker with any valid IC identity can:

1. Submit 100 concurrent calls each with a distinct `owner = Some(P_i)` for freshly generated principals.
2. Each call acquires one guard slot and then awaits `get_utxos` from the Bitcoin canister.
3. While all 100 calls are in-flight, `update_balance_accounts.len() == 100`.
4. Every legitimate `update_balance` call returns `TemporarilyUnavailable("too many concurrent requests")`.
5. As slots free up, the attacker re-submits to maintain saturation.

The existing `check_anonymous_caller()` check does not mitigate this because any non-anonymous principal (including freshly generated self-authenticating keys) can call the endpoint. There is no per-caller rate limit, no fee, and no stake requirement.

## Impact Explanation
`update_balance` is the **only** mechanism by which users can mint ckBTC after depositing BTC on-chain. A sustained DoS of this endpoint prevents all users from completing BTC→ckBTC conversions for as long as the attacker maintains the attack. This matches the allowed High impact: **"Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS"** and **"Significant Chain Fusion, ck-token… security impact with concrete user or protocol harm."** The attack requires no special privileges, no funds, and is trivially repeatable.

## Likelihood Explanation
Any non-anonymous IC principal can call `update_balance` with an arbitrary `owner`. No tokens, governance rights, or prior state are required. The attacker needs only the ability to submit ingress messages to the minter canister, which is publicly accessible on mainnet. The attack is cheap: principals with no UTXOs cause calls to complete quickly (returning `NoNewUtxos`), so the attacker can cycle through slots at low cost. Likelihood is **High**.

## Recommendation
Key the concurrency guard on the actual `msg_caller()` rather than on `args.owner`. The `owner` parameter legitimately allows third parties to trigger minting on behalf of others, but the guard should track the actual caller to prevent abuse:

```rust
// Fix: guard on actual caller, not on args.owner
let _guard = balance_update_guard(Account { owner: caller, subaccount: None })?;
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),
    subaccount: args.subaccount,
};
```

This mirrors the pattern already used in `retrieve_btc` (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` lines 161–165), which correctly guards on `caller` directly.

## Proof of Concept
1. Generate 100 distinct IC self-authenticating principals `P_1 … P_100` (e.g., via `Principal::self_authenticating` with 100 different public keys).
2. Submit 100 concurrent ingress `update_balance` calls to the ckBTC minter, each with `{ owner: Some(P_i), subaccount: None }`. These principals have no UTXOs, so each call will eventually return `NoNewUtxos` after awaiting `get_utxos`.
3. While all 100 calls are in-flight, verify `state.update_balance_accounts.len() == 100` via canister logs or a local PocketIC integration test.
4. Submit a legitimate `update_balance` call from a real user account; observe it returns `UpdateBalanceError::TemporarilyUnavailable("too many concurrent requests")`.
5. As attacker calls complete, immediately re-submit new calls with fresh principals to maintain saturation.
6. A deterministic integration test using PocketIC can mock the Bitcoin canister to delay `get_utxos` responses, hold 100 guard slots open, and assert that the 101st call fails with `TooManyConcurrentRequests`.
