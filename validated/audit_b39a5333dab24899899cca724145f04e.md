### Title
Unprivileged Caller Can Grief Consensus Reward Distribution via `update_rewards` with `disable_rewards: true` — (File: src/staking/staking.cairo)

---

### Summary

`update_rewards` in `StakingRewardsManagerImpl` has no access control and unconditionally writes `last_reward_block` **before** checking the `disable_rewards` flag. Any unprivileged caller can invoke this function with `disable_rewards: true` every block to consume the global reward slot without distributing rewards, permanently blocking all stakers from receiving consensus-phase block rewards.

---

### Finding Description

`update_rewards` is an externally callable function (marked `#[abi(embed_v0)]`) with no role-based access control beyond `general_prerequisites()` (which only checks the paused flag). [1](#0-0) 

The function flow is:

1. Checks `current_block_number > last_reward_block` — enforcing at most one call per block globally.
2. **Unconditionally writes** `last_reward_block = current_block_number`.
3. Only then checks `if disable_rewards || self.is_pre_consensus() { return; }`. [2](#0-1) 

Because step 2 occurs before step 3, a caller who passes `disable_rewards: true` consumes the block's single reward slot while distributing nothing. Any subsequent legitimate call in the same block fails with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The only validation on the caller is that a valid, active staker address is supplied — any staker address with non-zero effective balance satisfies this. [4](#0-3) 

---

### Impact Explanation

During the consensus-rewards phase (`is_pre_consensus() == false`), block rewards are the primary yield mechanism for stakers. An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block prevents **all** stakers from ever accumulating consensus rewards. This constitutes **permanent freezing of unclaimed yield** for the entire staker set, matching the allowed High impact "Permanent freezing of unclaimed yield or unclaimed royalties," or at minimum the Medium impact "Griefing with no profit motive but damage to users or protocol."

---

### Likelihood Explanation

**Medium.** The attack requires the adversary to submit one transaction per block. On Starknet, transaction fees are low. A motivated attacker (e.g., a competing protocol, a short-seller of STRK, or a censorship actor) has clear economic incentive to suppress staking yields. The entry path is fully permissionless — no privileged key, no bridge compromise, no external dependency.

---

### Recommendation

Add access control to `update_rewards`. Only the designated consensus contract (or a whitelisted set of callers) should be permitted to invoke it. Alternatively, move the `last_reward_block.write` to after the `disable_rewards` guard so that a no-op call does not consume the block's reward slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number); // move here
// ... distribute rewards
```

---

### Proof of Concept

```
Epoch N, consensus rewards active (is_pre_consensus() == false):

Block B:
  Attacker calls: staking.update_rewards(valid_staker_address, disable_rewards: true)
    → last_reward_block is set to B
    → no rewards distributed
    → returns early

  Any legitimate caller tries: staking.update_rewards(staker_X, disable_rewards: false)
    → assert!(current_block_number > last_reward_block) FAILS (B == B)
    → reverts with REWARDS_ALREADY_UPDATED

Block B+1:
  Attacker repeats the call.

Result: No staker ever receives consensus block rewards.
```

The attacker only needs a valid staker address (publicly readable from `stakers` Vec in storage) and enough gas to submit one transaction per block. [5](#0-4)
