Audit Report

## Title
Cross-Operator `num_removed_same_ip_same_type` Quota Bypass in `do_add_node_` — (`rs/registry/canister/src/mutations/node_management/do_add_node.rs`)

## Summary
`do_add_node_` computes `num_removed_same_ip_same_type` by scanning all nodes at the target IP regardless of ownership, but computes `num_in_registry_same_type` by filtering only the caller's (OP2's) nodes. When a different operator (OP1) sharing the same `dc_id` and `node_provider_principal_id` has a node at the same IP with the same reward type, the subtrahend is inflated by OP1's node, causing the quota check `max_rewardable_nodes <= num_in_registry_same_type.saturating_sub(num_removed_same_ip_same_type)` to pass incorrectly. Combined with the same-DC/same-provider fallback in `make_remove_or_replace_node_mutations`, OP2 can register one node beyond its governance-approved `max_rewardable_nodes` quota per cycle.

## Finding Description

**Root cause — unscoped `num_removed_same_ip_same_type`:**

`scan_for_nodes_by_ip` at [1](#0-0)  returns every node at the target IP regardless of `node_operator_id`. The loop in `do_add_node_` at [2](#0-1)  increments `num_removed_same_ip_same_type` for any such node whose reward type matches — including nodes owned by OP1, not OP2.

**Asymmetric quota check:**

`get_node_operator_nodes(self, caller_id)` at [3](#0-2)  is filtered to OP2's nodes only, producing a per-operator numerator. But `num_removed_same_ip_same_type` was incremented for OP1's node. The check at [4](#0-3)  then evaluates `Q <= Q.saturating_sub(1)` → `Q <= Q-1` → `false`, so the quota check **passes** when OP2 is at quota Q and OP1 has one node of the same type at the target IP.

**Cross-operator removal permitted by design:**

`make_remove_or_replace_node_mutations` at [5](#0-4)  allows OP2 to remove OP1's node when `caller_id != node_operator_id` provided both operators share the same `dc_id` and `node_provider_principal_id`. The comment at line 81 ("This will only succeed if the same NO was in control of the original nodes") is factually incorrect — the same-DC/same-provider fallback is a second, broader path that permits cross-operator removal.

**Net result:** OP1's node is removed, OP2's new node is inserted, and OP2 ends up with Q+1 nodes of type T. The cycle can be repeated by having OP1 re-register a node at a new IP, then repeating the attack.

## Impact Explanation

A node provider controlling two operators in the same data center can cause OP2 to register nodes beyond its governance-approved `max_rewardable_nodes` quota for a given reward type. Each successful cycle yields one unauthorized extra node for OP2, which earns unauthorized node rewards from the NNS. This constitutes a significant NNS/governance security impact with concrete financial harm: the node provider receives ICP rewards for nodes that governance explicitly did not approve. This matches the **High ($2,000–$10,000)** impact class: "Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation

The preconditions are realistic and common in IC deployments: a single node provider operating multiple operator principals in the same data center is the standard multi-operator deployment pattern. The attack requires no admin keys, no governance majority, and no threshold corruption — only two legitimately registered operator principals under the same node provider. The bypass is repeatable (each cycle sacrifices one of OP1's nodes and gains one extra node for OP2), allowing gradual accumulation beyond quota.

## Recommendation

Scope `num_removed_same_ip_same_type` to only count nodes belonging to `caller_id` in the loop at [2](#0-1) :

```rust
for node_with_same_ip in &nodes_with_same_ip {
    // Only count nodes owned by the caller
    let node_operator = get_node_operator_id_for_node(self, *node_with_same_ip)
        .map_err(|e| format!("{LOG_PREFIX}do_add_node: {e}"))?;
    if node_operator != caller_id {
        continue;
    }
    let node_same_ip_reward_type =
        get_node_reward_type_for_node(self, *node_with_same_ip)
            .map_err(|e| format!("{LOG_PREFIX}do_add_node: {e}"))?;
    if Some(node_same_ip_reward_type) == node_reward_type {
        num_removed_same_ip_same_type += 1;
    }
}
```

This makes the subtrahend consistent with the per-operator numerator used in the quota check.

## Proof of Concept

```
Setup:
  NP = node provider principal
  OP1: dc_id="dc1", node_provider_principal_id=NP, max_rewardable_nodes={type1: 1}
  OP2: dc_id="dc1", node_provider_principal_id=NP, max_rewardable_nodes={type1: 1}

Step 1: OP1 registers node N1 at IP=192.0.2.1, type=type1
        → registry: {N1: operator=OP1, ip=192.0.2.1, type=type1}

Step 2: OP2 registers node N2 at IP=192.0.2.2, type=type1
        → registry: {N1: OP1, N2: OP2}
        → OP2 is now at quota (1 node of type1)

Step 3: OP2 calls do_add_node(http_endpoint=192.0.2.1, type=type1)
        → scan_for_nodes_by_ip → [N1]
        → get_node_reward_type_for_node(N1) = type1 == type1
          → num_removed_same_ip_same_type = 1
        → get_node_operator_nodes(OP2) filtered type1 → [N2]
          → num_in_registry_same_type = 1
        → quota check: 1 <= 1.saturating_sub(1) = 1 <= 0 → false → PASSES
        → make_remove_or_replace_node_mutations(N1, caller=OP2):
            OP2 != OP1, but dc_caller==dc_orig and provider_caller==provider_orig
            → assert passes; N1 removed, new node N3 inserted at 192.0.2.1

Result: registry contains N2 (OP2, type1) and N3 (OP2, type1)
        OP2 has 2 nodes of type1, quota is 1 → quota bypassed by 1
```

A deterministic integration test can be written using the existing test infrastructure in `do_add_node.rs` (see `should_succeed_for_adding_two_nodes_with_identical_ips` at [6](#0-5)  as a template), adding a second operator record under the same node provider and same DC, registering N1 under OP1 and N2 under OP2, then calling `do_add_node_` as OP2 targeting OP1's IP and asserting that OP2 ends up with 2 nodes of type1 despite a quota of 1.

    #[test]
    fn should_succeed_for_adding_two_nodes_with_identical_ips() {
        // Arrange
        let mut registry = invariant_compliant_registry(0);
        // Add node operator record first
        let node_operator_record = NodeOperatorRecord {
            // Set max rewardable nodes to 1 to make sure the first node is removed
            max_rewardable_nodes: btreemap! { "type1".to_string() => 1 },
            ..Default::default()
        };

        let node_operator_id = PrincipalId::new_user_test_id(0);
        registry.maybe_apply_mutation_internal(vec![insert(
            make_node_operator_record_key(node_operator_id),
            node_operator_record.encode_to_vec(),
        )]);

        // Use payloads with the same IPs
        let (payload_1, _) = prepare_add_node_payload(1, NodeRewardType::Type1);

        let (mut payload_2, _) = prepare_add_node_payload(2, NodeRewardType::Type1);
        payload_2.http_endpoint.clone_from(&payload_1.http_endpoint);
        assert_eq!(payload_1.http_endpoint, payload_2.http_endpoint);

        // Act: Add two nodes with the same IPs
        let node_id_1: NodeId = registry
            .do_add_node_(payload_1.clone(), node_operator_id, now_system_time())
            .expect("failed to add a node");
        let expected_node_record_1 = NodeRecord {
            xnet: Some(connection_endpoint_from_string(&payload_1.xnet_endpoint)),
            http: Some(connection_endpoint_from_string(&payload_1.http_endpoint)),
            node_operator_id: node_operator_id.into(),
            domain: Some("api-example.com".to_string()),
            node_reward_type: Some(NodeRewardType::from(payload_1.node_reward_type.unwrap()) as i32),
            ..Default::default()
        };

        let node_record_1 = registry.get_node_or_panic(node_id_1);
        assert_eq!(node_record_1, expected_node_record_1);
        // Add the second node, this should remove the first one from the registry
        let node_id_2: NodeId = registry
            .do_add_node_(payload_2.clone(), node_operator_id, now_system_time())
            .unwrap();
        // Assert second node record is in the registry and is correct
        let expected_node_record_2 = NodeRecord {
            xnet: Some(connection_endpoint_from_string(&payload_2.xnet_endpoint)),
            http: Some(connection_endpoint_from_string(&payload_2.http_endpoint)),
            node_operator_id: node_operator_id.into(),
            domain: Some("api-example.com".to_string()),
            node_reward_type: Some(NodeRewardType::from(payload_2.node_reward_type.unwrap()) as i32),
            ..Default::default()
        };
        let node_record_2 = registry.get_node_or_panic(node_id_2);
        assert_eq!(node_record_2, expected_node_record_2);
        // Assert first node record is removed from the registry because of the IP conflict
        assert!(
            registry
                .get(
                    make_node_record_key(node_id_1).as_bytes(),
                    registry.latest_version()
                )
                .is_none()
        );
        // Assert max_rewardable_nodes has not changed
        let observed_node_operator_record = get_node_operator_record(&registry, node_operator_id)
            .expect("failed to get node operator");
        assert_eq!(
            observed_node_operator_record.max_rewardable_nodes,
            node_operator_record.max_rewardable_nodes
        );
    }
