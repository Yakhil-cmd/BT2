### Title
Caller-Controlled `disable_rewards` Parameter Allows Permanent Griefing of Staker Block Rewards — (`File: src/staking/staking.cairo`)

### Summary

The `update_rewards` function in the Staking contract accepts a caller-controlled `disable_rewards: bool` parameter. When set to `true`, the function advances `last_reward_block` to the current block without distributing any rewards. Because `last_reward_block` is a global gate that enforces one reward update per block, any unprivileged caller can permanently deprive a staker of their block rewards for any given block by front-running the legitimate reward update.

### Finding Description

`update_rewards` is a public function with no role restriction beyond a non-zero caller check: [1](#0-0) 

The function unconditionally writes `last_reward_block` to the current block number before checking `disable_rewards`: [2](#0-1) 

If `disable_rewards` is `true`, execution returns immediately — no rewards are computed, no `unclaimed_rewards` are incremented in the reward supplier, and no pool rewards are distributed: [3](#0-2) 

Because `last_reward_block` is already set to the current block, any subsequent legitimate call to `update_rewards` in the same block will revert with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The `disable_rewards` flag is entirely attacker-controlled. There is no on-chain derivation of this value from protocol state (e.g., staker liveness, epoch phase, or governance decision). This is the direct analog to the Pico `quotient_degree` bug: a parameter that should be derived from protocol state is instead accepted as trusted input from an arbitrary caller.

### Impact Explanation

An attacker can permanently deprive any staker of their block rewards for any block in the consensus-rewards phase. The rewards are never added to `unclaimed_rewards` in the `RewardSupplier`: [5](#0-4) 

Since block rewards are inflationary and computed per-block, the lost yield is permanent and unrecoverable. This constitutes **permanent freezing / theft of unclaimed yield** (High impact).

### Likelihood Explanation

- `update_rewards` is callable by any non-zero address with no access control.
- Valid staker addresses are publicly enumerable from the `stakers` vector in storage.
- The attacker only needs to call `update_rewards(target_staker, true)` once per block to suppress that block's rewards.
- No capital, no special role, and no prior relationship with the staker is required.
- On Starknet, transaction ordering within a block is deterministic and front-running is feasible.

### Recommendation

Remove the `disable_rewards` parameter from the public interface entirely. Whether rewards should be distributed for a given block should be derived exclusively from on-chain protocol state (e.g., `is_pre_consensus()`, staker liveness flags, or epoch phase), not from caller-supplied input. If a "skip rewards" path is needed for internal migration purposes, it should be gated behind a privileged role or made an internal function.

### Proof of Concept

1. Staker `S` is active with non-zero STRK balance in the consensus-rewards phase.
2. Attacker `A` (any address) calls `update_rewards(S, disable_rewards: true)` at block `N`.
3. `last_reward_block` is set to `N`; no rewards are distributed.
4. The legitimate reward update for block `N` (called by `S` or anyone else) reverts with `REWARDS_ALREADY_UPDATED`.
5. `S` permanently loses block rewards for block `N`.
6. `A` repeats this every block to continuously suppress `S`'s rewards at negligible cost. [6](#0-5)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
