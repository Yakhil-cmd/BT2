### Title
Permanent Loss of ERC-20 Tokens When `ExitToNear` `ft_transfer` Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not compiled in, the `ExitToNear` precompile unconditionally sets `refund: None` in the callback arguments. For a standard ERC-20 exit, this causes the promise chain to be created **without any callback**. If the downstream `ft_transfer` NEAR promise fails (e.g., recipient account not registered with the NEP-141 token), the ERC-20 tokens that were already burned in the EVM are permanently unrecoverable.

---

### Finding Description

The `ExitToNear` precompile's `run()` function in `engine-precompiles/src/native.rs` constructs a `ExitToNearPrecompileCallbackArgs` struct:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← hardcoded None when feature absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a standard ERC-20 exit (non-wNEAR), `transfer_near_args` is also `None`. This means `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The subsequent branch:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

creates the promise **without** an `exit_to_near_precompile_callback`. The ERC-20 tokens are burned from the user's EVM balance **before** the promise is dispatched. If `ft_transfer` on the NEP-141 contract fails, there is no callback to invoke `refund_on_error`, which would otherwise re-mint the burned tokens:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [3](#0-2) 

The `refund_on_error` function re-mints the burned ERC-20 tokens to the original sender: [4](#0-3) 

Without `error_refund`, this path is never reachable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `error_refund` is not compiled in:
- The user's ERC-20 tokens are burned (destroyed) in the EVM.
- The corresponding NEP-141 `ft_transfer` promise is dispatched with no callback.
- If `ft_transfer` fails (e.g., recipient NEAR account not registered with the NEP-141 token), the NEP-141 tokens remain in Aurora's account and the burned ERC-20 tokens are gone.
- There is no recovery path: the user permanently loses their funds.

The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The failure condition — recipient NEAR account not registered with the NEP-141 token — is a realistic and common user error. NEAR's NEP-141 standard requires explicit `storage_deposit` registration before receiving tokens. Any user who exits to an unregistered account (e.g., a freshly created account, a contract that hasn't registered, or a typo in the account ID) will trigger this path. The `error_refund` feature is optional and not guaranteed to be enabled in all deployments; the feature flag itself is the sole guard against this loss. [6](#0-5) 

---

### Recommendation

1. **Enable `error_refund` as a default feature** in `engine-precompiles/Cargo.toml` and `engine/Cargo.toml` so that refund callbacks are always attached to `ft_transfer` promises in the `ExitToNear` precompile.
2. Alternatively, unconditionally attach the refund callback (removing the feature flag entirely) and always include a `refund_address` in the precompile input format.
3. Document clearly in the precompile interface that without `error_refund`, failed exits result in permanent token loss.

---

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora as an ERC-20.
2. Compile Aurora Engine **without** the `error_refund` feature.
3. From an EVM address, call the ERC-20's `burn` function targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
4. The `ExitToNear` precompile fires: ERC-20 tokens are burned, `ft_transfer` promise is dispatched without a callback.
5. `ft_transfer` fails (recipient not registered).
6. Observe: ERC-20 balance is permanently reduced; NEP-141 tokens remain in Aurora's account; no refund is issued.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) demonstrates exactly this scenario and confirms the token loss when `error_refund` is absent. [7](#0-6)
