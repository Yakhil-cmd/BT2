### Title
Gateway Admits Invoke Transactions Without Signature Verification for Undeployed Accounts - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the `__validate__` entry-point call (which performs signature verification) for any invoke transaction with `nonce == 1` sent from an account whose `account_nonce == 0` and that has any transaction in the mempool or a recent block. An unprivileged attacker who observes a `deploy_account` transaction in the mempool can inject an invoke transaction with arbitrary calldata and an invalid signature for the victim's address, and the gateway will admit it to the mempool without ever verifying the signature.

### Finding Description

In `skip_stateful_validations`, the gateway returns `skip_validate = true` when three conditions hold simultaneously:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs an `AccountTransaction` with `execution_flags.validate = false` and passes it to `StatefulValidator::validate`: [2](#0-1) 

`StatefulValidator::validate` calls `tx.validate_tx(...)`, which immediately returns `Ok(None)` when `execution_flags.validate == false`, performing no signature check whatsoever: [3](#0-2) 

The gateway therefore admits the transaction to the mempool with no cryptographic verification of the sender's authorization.

The condition `account_tx_in_pool_or_recent_block` is satisfied as soon as any transaction from the target address appears in the mempool pool or the mempool's committed-nonce history: [4](#0-3) 

This is publicly observable: the `deploy_account` transaction is visible in the mempool before it is sequenced.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool can:

1. Submit an `Invoke(nonce=1, sender=victim, calldata=<arbitrary>, signature=<garbage>)` with a fee higher than the victim's legitimate invoke.
2. The gateway admits it (skip_validate path) and the mempool's fee-escalation logic evicts the victim's legitimate invoke.
3. The batcher later executes the attacker's invoke with `validate: true` (via `new_for_sequencing`), `__validate__` fails, and the transaction is rejected.
4. The victim's legitimate first post-deploy invoke has been permanently evicted from the mempool. [5](#0-4) 

The victim must resubmit their invoke. The attacker can repeat this indefinitely at zero cost (the account does not exist, so no fee is charged for a failed `__validate__`), permanently blocking the victim's first post-deploy transaction from being sequenced. This matches the **High** impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool.
- The attacker needs only the victim's address (derivable from the `deploy_account` transaction itself).
- No privileged access, no special knowledge of the private key, and no on-chain state is required.
- The attack is repeatable and free (no fee charged for failed `__validate__` on a non-existent account).

### Recommendation

The skip-validate UX feature should not bypass signature verification entirely. Instead, the gateway should verify the transaction signature against the **declared class hash** in the `deploy_account` transaction before admitting the invoke. Alternatively, the skip should only suppress the blockifier's `__validate__` call during the gateway's stateful validation phase while still performing a lightweight off-chain signature check against the class hash committed in the pending `deploy_account`.

At minimum, the mempool should track that a skipped-validation invoke is paired with a specific `deploy_account` transaction hash, and evict the invoke if the `deploy_account` is evicted or replaced.

### Proof of Concept

```
1. Alice submits:
   - deploy_account(class_hash=C, salt=S, constructor_calldata=[], nonce=0, sig=valid)
   - invoke(sender=alice_addr, nonce=1, calldata=[transfer(bob, 100)], sig=valid, fee=10)

2. Attacker observes alice_addr in mempool (account_tx_in_pool_or_recent_block = true).

3. Attacker submits:
   - invoke(sender=alice_addr, nonce=1, calldata=[transfer(attacker, 1000000)], sig=0xdeadbeef, fee=20)

4. Gateway path for attacker's tx:
   - validate_nonce: account_nonce=0, tx_nonce=1, max_allowed_nonce_gap>=1 → PASS
   - validate_by_mempool: no duplicate hash → PASS
   - skip_stateful_validations: nonce==1, account_nonce==0, alice in mempool → returns true
   - run_validate_entry_point(skip_validate=true): validate_tx returns Ok(None) → PASS
   - Transaction admitted to mempool.

5. Mempool fee escalation: attacker's fee(20) > alice's fee(10) → alice's invoke evicted.

6. Batcher executes:
   - deploy_account → succeeds, alice_addr deployed.
   - attacker's invoke → __validate__ called → sig=0xdeadbeef → FAILS, rejected.

7. Alice's legitimate invoke is gone. Attacker repeats from step 3.

        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),

            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
