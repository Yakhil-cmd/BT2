### Title
Gateway Admits Invoke Transactions Without Signature Verification via `skip_stateful_validations` UX Path - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway stateful validator skips the `__validate__` entry-point call for any Invoke transaction with `nonce == 1` when the sender address has a pending `deploy_account` (or recent-block record) in the mempool. Because the only signature check in the entire gateway pipeline is that `__validate__` call, an unprivileged attacker can submit an Invoke with an arbitrary (invalid) signature for a victim's not-yet-deployed account and have it unconditionally admitted to the mempool — breaking the invariant that every admitted transaction carries a verified signature.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

When `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)` and `account_tx_in_pool_or_recent_block` returns `true`, the function returns `true` (skip).

**How the skip propagates to `run_validate_entry_point`** [2](#0-1) 

`execution_flags.validate = !skip_validate` → `false`. The blockifier `StatefulValidator::perform_validations` then short-circuits: [3](#0-2) 

`__validate__` is never called; the function returns `Ok(())` after only `perform_pre_validation_stage` (nonce increment + fee-bound check). No signature bytes are inspected.

**What `account_tx_in_pool_or_recent_block` actually checks** [4](#0-3) 

It returns `true` if the address appears in the pool (any transaction, including a `deploy_account` submitted by the legitimate owner) **or** in the mempool's recent-block state. An attacker who observes a victim's `deploy_account` in the pool immediately satisfies this condition for the victim's address.

**Attack path**

1. Victim broadcasts `deploy_account` for address X (nonce 0). It enters the mempool.
2. Attacker calls `add_transaction` with an Invoke for address X, `nonce = 1`, and a garbage signature.
3. Gateway stateless validator: checks only signature *length*, not validity.
4. Gateway stateful validator: `account_nonce = 0`, `tx.nonce() = 1`, `account_tx_in_pool_or_recent_block(X) = true` → `skip_validate = true` → `__validate__` skipped → **transaction admitted**.
5. Victim now tries to submit their own Invoke for address X with `nonce = 1` and a valid signature. `validate_by_mempool` rejects it: duplicate nonce / fee-escalation required.
6. Batcher executes `deploy_account` (account X deployed, nonce → 1), then executes attacker's Invoke: `__validate__` is called with `validate = true` (batcher uses `new_for_sequencing`), signature fails, transaction reverts, nonce advances to 2.
7. Victim's legitimate nonce-1 Invoke is permanently blocked; victim must resubmit at nonce 2.

**Why existing guards do not close the gap**

- `validate_tx_signature_size` (stateless) checks only that `signature.len() ≤ max_signature_length` — not that the signature is cryptographically valid.
- `validate_by_mempool` / `validate_incoming_tx` check duplicate hash and nonce-gap, not signature validity.
- `perform_pre_validation_stage` (called even on the skip path) only increments the nonce and checks fee bounds; it does not call `__validate__`.
- The batcher's `new_for_sequencing` does set `validate = true`, so the signature *is* checked at execution time — but by then the nonce slot is already occupied and the victim's transaction has been rejected.

### Impact Explanation

The gateway unconditionally admits Invoke transactions that carry an invalid (attacker-chosen) signature, violating the invariant that every mempool-admitted transaction has passed account-level signature verification. This matches the stated High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Concrete consequences:
- Any account whose `deploy_account` is visible in the mempool can have its nonce-1 slot poisoned by an attacker at the cost of one small fee payment.
- The victim's first post-deploy Invoke is rejected; they must resubmit at nonce 2, paying an extra transaction and experiencing unexpected UX breakage.
- At scale, an attacker can monitor the mempool and systematically grief every new account deployment.

### Likelihood Explanation

The trigger is fully unprivileged: any party can call `add_transaction` on the public HTTP endpoint. The only prerequisite is observing a `deploy_account` in the mempool (public information). The window is the time between the victim's `deploy_account` entering the pool and the victim submitting their first Invoke — a window that is deliberately widened by the UX feature itself (users are encouraged to send both transactions together).

### Recommendation

1. **Remove the unconditional skip**: The `skip_stateful_validations` path should not bypass `__validate__` entirely. Instead, the gateway should simulate the `deploy_account` execution first (in a read-only transactional state) and then run `__validate__` against the resulting deployed account state. This preserves the UX intent while enforcing signature verification.
2. **Alternatively**, restrict the skip to transactions whose sender address matches the deployer recorded in the pending `deploy_account` transaction already in the pool, so a third party cannot exploit the skip for an address they do not own.
3. **At minimum**, document the known griefing surface and add a rate-limit or fee-escalation requirement for nonce-1 Invokes on undeployed accounts.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
```
