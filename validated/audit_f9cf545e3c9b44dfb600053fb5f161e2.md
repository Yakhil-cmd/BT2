### Title
Missing Cryptographic Signature Verification in `validateGuarantors` Allows Forged Collection Guarantees - (File: `engine/consensus/ingestion/core.go`)

### Summary

The `validateGuarantors` function in the consensus ingestion engine checks that the claimed signers encoded in `guarantee.SignerIndices` have sufficient cumulative weight to meet the supermajority threshold, but **never verifies the cryptographic `Signature` field** of the `CollectionGuarantee`. The `Signature` field is explicitly documented as unused and set to `nil` at construction time. This is the direct Flow analog of the Cosmos SDK `ValidateVoteExtensions` issue: the weight/voting-power check is performed against a proposer-injectable set of claimed signers rather than against cryptographically proven signers, allowing a single malicious collection node to forge a guarantee that appears to carry a supermajority of cluster weight.

---

### Finding Description

`CollectionGuarantee` carries two fields relevant to signer authentication:

- `SignerIndices []byte` — a bit-vector encoding which cluster members are claimed to have signed
- `Signature crypto.Signature` — the aggregated BLS signature over those signers [1](#0-0) 

The constructor explicitly documents that `Signature` is **not validated** and is set to `nil` in production: [2](#0-1) 

The collection finalizer confirms this at the point of guarantee creation: [3](#0-2) 

`validateGuarantors` in the consensus ingestion engine decodes `SignerIndices` to obtain a list of identities, computes their total weight, and checks it against the supermajority threshold — but performs **no cryptographic verification** of `Signature`: [4](#0-3) 

The code itself acknowledges this is not BFT: [5](#0-4) 

The same gap exists at the block-extension layer. `guaranteeExtend` calls `protocol.FindGuarantors`, which only validates that `SignerIndices` decode to valid cluster member identifiers — again without touching `Signature`: [6](#0-5) [7](#0-6) 

The supermajority threshold itself is correctly computed from the **full** cluster weight: [8](#0-7) [9](#0-8) 

But because the weight is derived from the **claimed** `SignerIndices` rather than from cryptographically proven signers, the threshold check is meaningless without signature verification.

---

### Impact Explanation

A single malicious collection node can:

1. Craft a `CollectionGuarantee` for any `CollectionID` (including one containing invalid or attacker-chosen transactions).
2. Set `SignerIndices` to encode a supermajority of high-weight cluster members.
3. Set `Signature = nil` (matching the current protocol convention).
4. Broadcast the guarantee to consensus nodes.

`validateOrigin` only checks that the *sender* is a valid collection node — it does not check that the *claimed signers* actually signed anything: [10](#0-9) 

The guarantee passes all checks and is admitted to the mempool, then included in a block proposal. The block-extension check (`guaranteeExtend`) also passes for the same reason. The result is that an unauthorized collection — one that was never actually approved by a supermajority of honest cluster members — is finalized on-chain and subsequently executed by execution nodes.

This bypasses the BFT guarantee that is supposed to require ≥ 2/3 of cluster weight to approve a collection, reducing the attack requirement from a supermajority of cluster members to a single malicious collection node.

---

### Likelihood Explanation

The attacker must be a staked collection node (to pass `validateOrigin`). This is within the normal threat model (up to 1/3 of nodes may be Byzantine). The attack requires no special privileges beyond being a valid cluster participant, no key compromise, and no coordination with other nodes. The forged guarantee is structurally indistinguishable from a legitimate one because `Signature = nil` is the current production convention. Likelihood is **medium**: the attacker must be a staked collection node, but no supermajority or quorum is needed.

---

### Recommendation

1. **Verify the aggregated BLS signature** in `validateGuarantors` against the public staking keys of the decoded guarantors before admitting a guarantee to the mempool. The `WeightedSignatureAggregator` infrastructure already exists in the codebase for this purpose.
2. **Enforce the same check** in `guaranteeExtend` / `FindGuarantors` at the block-extension layer.
3. **Remove the `nil` signature convention** in `module/finalizer/collection/finalizer.go` once verification is in place, and enforce a non-nil signature in `NewCollectionGuarantee`.
4. Until full BFT verification is implemented, document the attack surface explicitly in security advisories.

---

### Proof of Concept

```
Attacker: single malicious collection node (nodeID = A, staked in cluster C)

1. Identify cluster C members with total weight W.
   Choose SignerIndices encoding members whose combined weight > 2W/3.

2. Craft guarantee:
   CollectionGuarantee{
       CollectionID:     <attacker-chosen collection hash>,
       ReferenceBlockID: <any valid recent block>,
       ClusterChainID:   <cluster C chain ID>,
       SignerIndices:    <bit-vector claiming supermajority>,
       Signature:        nil,   // matches production convention; never checked
   }

3. Send guarantee to consensus node via the ingestion engine channel.

4. validateOrigin: passes (A is a valid collection node).
   validateExpiry: passes (reference block is recent).
   validateGuarantors:
       - DecodeSignerIndicesToIdentities succeeds (indices are valid).
       - guarantors.TotalWeight() > threshold: passes (claimed signers have enough weight).
       - Signature: never read.
   → guarantee admitted to mempool.

5. Consensus leader includes guarantee in next block proposal.
   guaranteeExtend: FindGuarantors succeeds (same reason).
   → block extends chain; collection is finalized and executed.
```

The attacker-chosen collection is now on-chain with no honest cluster member having signed it.

func (e *Core) validateOrigin(originID flow.Identifier, guarantee *flow.CollectionGuarantee) error {

	}
	if !valid {
		return engine.NewInvalidInputErrorf("invalid collection origin (id=%x)", originID)
	}
	return nil

func FindGuarantors(state State, guarantee *flow.CollectionGuarantee) ([]flow.Identifier, error) {
	snapshot := state.AtBlockID(guarantee.ReferenceBlockID)
	epoch, err := snapshot.Epochs().Current()
	if err != nil {
		return nil, fmt.Errorf("could not get current epoch: %w", err)
	}
	cluster, err := epoch.ClusterByChainID(guarantee.ClusterChainID)

	if err != nil {
		return nil, fmt.Errorf(
			"fail to retrieve collector clusters for guarantee (ReferenceBlockID: %v, ClusterChainID: %v): %w",
			guarantee.ReferenceBlockID, guarantee.ClusterChainID, err)
	}

	guarantorIDs, err := signature.DecodeSignerIndicesToIdentifiers(cluster.Members().NodeIDs(), guarantee.SignerIndices)
	if err != nil {
		return nil, fmt.Errorf("could not decode signer indices for guarantee %v: %w", guarantee.ID(), err)
	}

	return guarantorIDs, nil

// WeightThresholdToBuildQC returns the weight (sum of unique, valid votes for this view)
// that is minimally required for a supermajority.
func WeightThresholdToBuildQC(totalWeight uint64) uint64 {
	// Given totalWeight, we need the smallest integer t such that 2 * totalWeight / 3 < t
	// Formally, the minimally required weight is: 2 * Floor(totalWeight/3) + max(1, totalWeight mod 3)
	floorOneThird := totalWeight / 3 // integer division, includes floor
	res := 2 * floorOneThird
	divRemainder := totalWeight % 3
	if divRemainder <= 1 {
		res = res + 1
	} else {
		res += divRemainder
	}
	return res
}
