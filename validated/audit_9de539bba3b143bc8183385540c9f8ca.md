### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Unverified Signatures for Undeployed Accounts - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` UX feature, designed to allow a deploy-account + invoke pair to be submitted atomically, unconditionally bypasses the `__validate__` entry-point (signature check) for any Invoke transaction with `nonce == 1` sent to an account whose on-chain nonce is still `0` and that has any transaction in the mempool. An unprivileged attacker who observes a victim's pending `DeployAccount` transaction can submit a spoofed Invoke with an arbitrary signature for the victim's not-yet-deployed address. The gateway admits it without verifying the signature. The transaction is later executed by the batcher, fails `__validate__`, reverts, and the fee is charged to the victim's pre-funded address — draining it before the account is even deployed.

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must carry a valid account signature (or, for the deploy-account UX path, must at minimum be verifiable as originating from the correct sender). The skip path removes the only gateway-level check that enforces this.

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

1. An attacker submits an Invoke with nonce=1 and a random/wrong signature for a victim's undeployed address. The gateway admits it (skip_validate=true, fee/balance checks pass because the address is pre-funded).
2. The batcher later executes `DeployAccount` (nonce=0), then the spoofed Invoke (nonce=1). The Invoke's `__validate__` fails; the transaction reverts.
3. The fee for the failed validation is charged to the victim's pre-funded address, draining it before the account is usable.
4. Because the mempool enforces uniqueness per (address, nonce), the attacker's spoofed tx occupies the nonce=1 slot, potentially blocking the victim's own legitimate nonce=1 transaction from being admitted until the spoofed one is committed and rejected.

### Likelihood Explanation

- The victim's `DeployAccount` transaction is public in the mempool.
- The victim's contract address is deterministic and computable from the `DeployAccount` parameters.
- The attacker only needs to submit one transaction (nonce=1) before the `DeployAccount` is sequenced — a standard front-run window.
- No special privileges, keys, or insider access are required.

### Recommendation

1. **Narrow the skip condition.** The skip should only suppress the `__validate__` call when the account contract does not yet exist on-chain (which is already the case), but it should still perform a lightweight, stateless signature-format check (e.g., verify the signature is non-empty and structurally valid for the declared account class).
2. **Alternatively, do not skip `__validate__` at the gateway.** Instead, accept that the gateway will reject the invoke tx if the account is not yet deployed, and rely on the client to retry after the `DeployAccount` is confirmed. The UX benefit does not justify admitting unverified signatures.
3. **At minimum, document and enforce that the skip path is only taken when the `DeployAccount` tx in the mempool was submitted by the same sender** (i.e., verify that the `DeployAccount` tx's sender address matches the Invoke's sender address and that the `DeployAccount` tx hash is provided and present in the pool).

### Proof of Concept

```
Precondition: Victim V wants to deploy account at address X.
V pre-funds address X with STRK.
V submits DeployAccount(nonce=0, class_hash=C, salt=S, ...) → mempool.

Step 1 (Attacker):
  Compute X = deploy_address(C, S, constructor_calldata)   // deterministic, public
  Craft Invoke(sender=X, nonce=1, calldata=[...], signature=[0xdeadbeef])
  Submit to gateway.

Step 2 (Gateway — skip_stateful_validations):
  account_nonce = get_nonce(X) = 0                         // X not deployed
  tx.nonce() == 1 && account_nonce == 0  → true
  account_tx_in_pool_or_recent_block(X)  → true (DeployAccount is in pool)
  skip_validate = true
  perform_pre_validation_stage: nonce 0<=1 ✓, fee bounds ✓, balance ✓
  __validate__ NOT called
  → Invoke admitted to mempool

Step 3 (Batcher):
  Execute DeployAccount(nonce=0) → X deployed successfully
  Execute Invoke(nonce=1):
    perform_pre_validation_stage: nonce 1==1 ✓, fee bounds ✓, balance ✓
    __validate__(signature=[0xdeadbeef]) → FAILS (wrong signature)
    Transaction reverts; fee charged to X

Result: X's pre-funded STRK balance is partially drained before V can use the account.
        V's own legitimate nonce=1 Invoke was blocked from the mempool by the attacker's tx.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
```

```rust
    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
```
