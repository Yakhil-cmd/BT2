### Title
Missing `to != treasury` Guard in `transfer_from_treasury` Allows Asset-Tracker Supply Inflation — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`transfer_from_treasury` performs a two-step balance update (subtract from treasury, add to `to`) without checking that `to` is not the treasury address itself (`BASE_TOKEN_HOLDER_ADDRESS`). Because `notify_l2_asset_tracker` is called **before** the balance update, an L1 transaction whose `to` field equals `BASE_TOKEN_HOLDER_ADDRESS` causes the asset tracker to record a bridging event for `value` tokens while the treasury balance remains unchanged — inflating the tracked L2 total supply without any real token movement.

---

### Finding Description

`mint_base_token` (lines 741–769) first calls `notify_l2_asset_tracker` with the full `amount`, then calls `transfer_from_treasury` with the same `amount` and the caller-supplied `to` address. [1](#0-0) 

`transfer_from_treasury` unconditionally performs:

1. Subtract `nominal_token_value` from `treasury_address` (`BASE_TOKEN_HOLDER_ADDRESS` = `0x10011`)
2. Add `nominal_token_value` to `to` [2](#0-1) 

When `to == BASE_TOKEN_HOLDER_ADDRESS`, steps 1 and 2 operate on the **same account**, so the net change to the treasury balance is zero. However, `notify_l2_asset_tracker` has already been called and has recorded that `amount` tokens were bridged from L1: [3](#0-2) 

The asset tracker's internal total-supply counter is incremented by `amount`, but no tokens actually left the treasury. The comment in `mint_base_token` explicitly notes that the asset tracker is notified first so that `totalSupply() == 0` can be used as a migration sentinel: [4](#0-3) 

There is no guard anywhere in `transfer_from_treasury` or `mint_base_token` that rejects or special-cases `to == BASE_TOKEN_HOLDER_ADDRESS`.

### Impact Explanation

Every L1→L2 transaction whose `to` field is `BASE_TOKEN_HOLDER_ADDRESS` with a non-zero `value` inflates the asset tracker's recorded total supply by `value` without any corresponding decrease in the treasury. Repeated calls accumulate unbounded inflation of the tracked supply. This corrupts the `totalSupply() == 0` sentinel used by `_needToForceSetAssetMigrationOnL2`, and any downstream protocol logic (cross-chain accounting, withdrawal limits, migration triggers) that relies on the asset tracker's supply figure will operate on incorrect data. The treasury balance itself is unaffected, creating a permanent divergence between on-chain token balances and the asset tracker's view.

---

### Likelihood Explanation

L1→L2 transactions are submitted by any unprivileged user. The `to` field is a free 20-byte parameter with no restrictions enforced at the bootloader level. The treasury (`BASE_TOKEN_HOLDER_ADDRESS`) is pre-funded with `2^128 − 1` tokens by default, so the subtract step never fails. An attacker needs only to submit a standard L1 transaction with `to = 0x0000000000000000000000000000000000010011` and any non-zero `value`. No special privileges, leaked keys, or governance access are required.

---

### Recommendation

Add an explicit guard in `transfer_from_treasury` (or at the call site in `mint_base_token`) that rejects or no-ops when `to == treasury_address`:

```rust
// In transfer_from_treasury, before the balance updates:
if to == treasury_address {
    return Err(interface_error!(BootloaderInterfaceError::InvalidRecipient));
}
```

Alternatively, add the check in `mint_base_token` before calling either `notify_l2_asset_tracker` or `transfer_from_treasury`, since notifying the asset tracker for a self-transfer is itself incorrect.

---

### Proof of Concept

1. Attacker submits an L1→L2 transaction:
   - `from`: any funded address
   - `to`: `0x0000000000000000000000000000000000010011` (`BASE_TOKEN_HOLDER_ADDRESS`)
   - `value`: `X` (any non-zero amount ≤ treasury balance)
   - `gas_limit` / `gas_price`: sufficient to cover fees

2. Bootloader calls `mint_base_token(system, …, amount=X, to=BASE_TOKEN_HOLDER_ADDRESS, …)`.

3. `notify_l2_asset_tracker` is called with `amount=X` → asset tracker increments its total supply by `X`.

4. `transfer_from_treasury` is called:
   - Treasury balance: `T − X` (step 1)
   - Treasury balance: `T − X + X = T` (step 2, same address)
   - Net treasury change: **zero**.

5. Asset tracker now reports `totalSupply` inflated by `X`; actual treasury balance is unchanged.

6. Repeating this attack with increasing `X` values allows unbounded inflation of the tracked supply at negligible cost (only gas fees).

/// Notify L2AssetTracker about base token bridging from L1.
///
/// Calls handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)
/// as L2_BASE_TOKEN_ADDRESS (0x800a) to pass the onlyBaseTokenHolderOrL2BaseToken modifier.
///
/// This is called separately for each token movement (value mint, operator
/// payment, refund) so that the asset tracker's accounting stays correct even
/// if the main transaction body reverts.
///
/// Resource usage depends on the caller — value-mint tracks native against user resources;
/// operator-fee and refund use FORMAL_INFINITE.
///
/// Failure halts block processing — if the asset tracker reverts, the
/// chain's token accounting would be inconsistent, so we treat it as
/// fatal rather than silently continuing with incorrect bookkeeping.
///
/// If no contract is deployed at L2AssetTracker, the call succeeds silently
/// (a call to an empty address returns success with no returndata in EVM).
/// However, we are certain that L2AssetTracker is available after the upgrade.
fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
