### Title
Hardcoded 2,300-Gas Stipend on `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` Causes Silent Revert When Depositing FLOW to EVM Smart Contracts - (File: `fvm/evm/types/call.go`)

---

### Summary

Flow EVM's `NewDepositCall` and `NewTransferCall` constructors hard-code a gas limit of `21,000 + 2,300 = 23,300` for all token transfers to EVM addresses. This mirrors the exact anti-pattern of Solidity's deprecated `transfer()` function. When the recipient EVM address is a smart contract whose `receive()` or `fallback()` function requires more than 2,300 gas, the deposit or transfer will revert at the EVM level, permanently blocking FLOW delivery to that address via the Cadence-side bridge path.

---

### Finding Description

In `fvm/evm/types/call.go`, the following constants are defined:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer
```

`NewDepositCall` uses `DepositCallGasLimit` (23,300) and `NewTransferCall` uses `DefaultGasLimitForTokenTransfer` (23,300). These are the only gas limits ever used for these call subtypes — the caller has no way to override them.

`Account.Deposit()` in `fvm/evm/handler/handler.go` calls `types.NewDepositCall(...)` with this fixed gas limit, and `Account.Transfer()` calls `types.NewTransferCall(...)` with the same fixed limit. Both then call `panicOnErrorOrInvalidOrFailedState`, meaning a revert at the EVM level causes the entire Cadence transaction to panic/revert.

The emulator test at `fvm/evm/emulator/emulator_test.go` line 83–110 explicitly demonstrates this: depositing to a smart contract that does not implement a payable `receive()` results in `gethVM.ErrExecutionReverted`. Any contract whose `receive()` or `fallback()` uses more than 2,300 gas (e.g., emits an event, writes storage, or is a proxy/multisig) will also revert.

---

### Impact Explanation

Any user who deploys or uses an EVM smart contract wallet (multisig, proxy, vault) as their EVM address and then attempts to receive FLOW via:
- `EVMAddress.deposit(from: @FlowToken.Vault)` (Cadence → EVM bridge path)
- `Account.Transfer(to, balance)` (COA-to-contract transfer)

will have their transaction permanently revert. The FLOW tokens are not lost (the Cadence transaction reverts entirely), but the recipient smart contract **cannot receive FLOW through these protocol-level paths at all**, regardless of how much gas the user is willing to pay. There is no user-facing parameter to increase the gas limit for these calls.

This is a cross-VM asset delivery failure: a class of EVM smart contract addresses is structurally unreachable via the Cadence-side deposit/transfer mechanism.

---

### Likelihood Explanation

This is reachable by any unprivileged user who:
1. Deploys or uses an EVM smart contract as their receiving address (e.g., a Safe multisig, any proxy wallet, any contract that emits an event on `receive()`), and
2. Calls `EVMAddress.deposit(from: vault)` or uses `Account.Transfer()` targeting that address.

COAs themselves are smart contracts and are explicitly tested to accept transfers (they implement a minimal `receive()`), but any third-party contract with a non-trivial `receive()` is affected. This is a realistic scenario given the prevalence of smart contract wallets on EVM chains.

---

### Recommendation

Replace the hardcoded `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` with a configurable or sufficiently large gas limit (e.g., matching what `COA.call()` allows, which is user-specified). At minimum, expose a gas limit parameter on `NewDepositCall` and `NewTransferCall`, or raise the default to a value that accommodates typical smart contract `receive()` implementations (e.g., 100,000 gas). The `WithdrawCallGasLimit` (bridge → user) is less affected since the bridge address is a known EOA-like address, but should also be reviewed.

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call

			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}
