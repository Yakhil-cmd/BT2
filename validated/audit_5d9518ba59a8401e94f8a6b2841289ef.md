### Title
Sequential Integer Division Precision Loss in `delta_gas` Native Resource Accounting — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` computes `delta_gas` — the extra gas charged to cover native (proving) resource consumption — using a two-step sequential integer division. The intermediate value `native_per_gas` is itself the result of a ceiling division (`gas_price / native_price`), and then `native_used / native_per_gas` performs a second floor division on that already-rounded value. Because both rounding errors compound in the same direction, `delta_gas` is always ≤ the mathematically correct value. Every transaction that triggers the `delta_gas > 0` branch pays less gas than it should, leaking fee revenue from the operator.

---

### Finding Description

The `delta_gas` adjustment is the mechanism by which ZKsync OS ensures that native (proving) resource consumption is reflected in the gas charged to the user. The docs define it as:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
nativePerGas := gasPrice / nativePrice
```

**Step 1 — first division** (`validation_impl.rs`, L2 path):

```rust
// zk/validation_impl.rs line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

`native_per_gas` is computed with **ceiling** division, so it is always ≥ the true ratio `gas_price / native_price`. [1](#0-0) 

The same ceiling division is applied for L1 transactions:

```rust
// process_l1_transaction.rs line 469
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

**Step 2 — second division** (`refund_calculation.rs`):

```rust
// refund_calculation.rs line 72
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [3](#0-2) 

Because `native_per_gas = ceil(gas_price / native_price)` is strictly larger than the true ratio whenever `gas_price` is not an exact multiple of `native_price`, the floor division `native_used / native_per_gas` yields a value strictly smaller than the mathematically correct `native_used * native_price / gas_price`. Both rounding errors push in the same direction:

```
computed delta_gas  =  native_used / ceil(gas_price / native_price)
true    delta_gas   =  native_used * native_price / gas_price

computed <= true   (always)
```

The `delta_gas > 0` branch then adds the underestimated value to `gas_used`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [4](#0-3) 

The final fee paid to the operator is `gas_used * gas_price`, so a smaller `gas_used` means a smaller operator payment and a larger user refund. [5](#0-4) 

---

### Impact Explanation

The operator receives less fee than the true cost of proving the transaction. The user is over-refunded. The magnitude of the error per transaction is:

```
error = (true_delta_gas - computed_delta_gas) * gas_price
      = (native_used * native_price / gas_price
         - native_used / ceil(gas_price / native_price)) * gas_price
```

**Worst-case example** (gas_price just above a multiple of native_price):

| Parameter | Value |
|---|---|
| `native_price` | 1,000 |
| `gas_price` | 1,001 |
| `native_per_gas` (computed, ceiling) | 2 |
| `native_per_gas` (true ratio) | 1.001 |
| `native_used` | 100,000 |
| True `delta_gas` | ≈ 99,900 |
| Computed `delta_gas` | 50,000 |
| Under-charged gas | 49,900 |
| Fee leak per tx | 49,900 × 1,001 ≈ **49.95M wei** |

At scale, across thousands of transactions per block, this constitutes a steady, systematic drain of operator fee revenue. The error is not random noise — it is always in the user's favour.

---

### Likelihood Explanation

This is triggered by any transaction where:
1. `gas_price` is not an exact multiple of `native_price` (the common case in practice), **and**
2. The transaction's native resource consumption exceeds what the ergs-based `gas_used` accounts for (`delta_gas > 0`).

Condition 2 is the normal operating mode for transactions that are native-resource-heavy (e.g., large pubdata, complex proving work). An unprivileged user can deliberately choose a `gas_price` value just above a multiple of `native_price` to maximise the rounding error. No special access, governance role, or oracle manipulation is required — only a standard transaction submission.

---

### Recommendation

Avoid the intermediate `native_per_gas` rounding in the `delta_gas` computation. Instead, pass the original `gas_price` and `native_price` values into `compute_gas_refund` and compute `delta_gas` in a single combined expression that multiplies before dividing:

```rust
// Correct: multiply first, divide once at the end
let delta_gas = if gas_price == 0 {
    0i64
} else {
    (native_used as u128 * native_price as u128 / gas_price as u128) as i64
        - gas_used as i64
};
```

This mirrors the recommendation in the reference report: consolidate all multiplications before the single final division to preserve maximum precision. The `native_per_gas` value can still be used for the `native_limit` budget (where ceiling rounding is intentionally conservative), but it must not be reused as the divisor in the `delta_gas` back-conversion.

---

### Proof of Concept

**Setup:**
- `native_price = 1000`
- `gas_price = 1001` (one unit above a multiple of `native_price`)
- `gas_limit = 200,000`
- Transaction executes and consumes `native_used = 100,000` native units
- EVM `gas_used` (from ergs) = 50,000

**Current code path:**

```
native_per_gas = ceil(1001 / 1000) = 2          // validation_impl.rs:135
delta_gas      = 100,000 / 2 - 50,000 = 0       // refund_calculation.rs:72
                                                 // delta_gas == 0, no adjustment
gas_used_final = 50,000
refund         = (200,000 - 50,000) * 1001 = 150,150,000 wei
```

**Correct computation:**

```
true_delta_gas = 100,000 * 1000 / 1001 - 50,000 ≈ 49,900
gas_used_final = 50,000 + 49,900 = 99,900
refund         = (200,000 - 99,900) * 1001 = 100,200,100 wei
```

**Difference:** the user receives `~49.95M wei` more than they should, and the operator receives `~49.95M wei` less. Repeated across every native-heavy transaction in every block, this constitutes a continuous, attacker-exploitable leak of operator fee revenue. The attacker's only required action is to submit transactions with `gas_price ≡ 1 (mod native_price)`.
