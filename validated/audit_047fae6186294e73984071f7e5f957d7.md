### Title
Missing Caller Validation in `update_rewards` Allows Any Address to Permanently Block Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, consuming the global `last_reward_block` slot and permanently preventing all stakers from ever receiving consensus block rewards.

### Finding Description
The specification at `docs/spec.md` lines 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1507) performs no such check. The function only validates:

1. Contract is not paused (`general_prerequisites`)
2. `current_block_number > self.last_reward_block.read()` (REWARDS_ALREADY_UPDATED)
3. The supplied `staker_address` exists and is active

There is no `assert_caller_is_sequencer()` or equivalent guard. The `last_reward_block` storage variable is a **single global value** shared across all stakers. When `update_rewards` is called with `disable_rewards: true`, it writes `current_block_number` into `last_reward_block` and returns immediately without distributing any rewards. Any subsequent call to `update_rewards` in the same block — including the legitimate sequencer call — will revert with `REWARDS_ALREADY_UPDATED`.

An attacker can front-run the sequencer every block by calling:
```
update_rewards(any_valid_staker_address, disable_rewards: true)
```
This costs only gas, requires no stake, and permanently prevents all consensus reward distribution for every block.

### Impact Explanation
This is a **High** impact finding: **Permanent freezing of unclaimed yield**.

All stakers and delegators are permanently denied their consensus block rewards. The `unclaimed_rewards_own` field of every staker is never incremented, and pool rewards are never transferred. Since the attack can be repeated every block at negligible cost, the yield freeze is permanent and protocol-wide.

### Likelihood Explanation
**High likelihood.** The entry point is a public, permissionless function. The attacker needs only:
- Any valid staker address (trivially obtained from on-chain events)
- Enough gas to call `update_rewards` once per block

No privileged access, no stake, no leaked keys, and no external dependency is required. The attack is fully self-contained and can be automated.

### Recommendation
Add a caller check at the top of `update_rewards` to enforce that only the authorized sequencer address (or a designated rewards-manager role) can invoke it. For example, add an assertion analogous to the existing `assert_caller_is_attestation_contract` pattern already used in the codebase:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
}
```

### Proof of Concept

1. Consensus rewards are enabled (post `set_consensus_rewards_first_epoch`).
2. Staker A has staked and K epochs have passed (balance is effective).
3. A new block N is produced.
4. **Attacker** (any address, zero stake) calls:
   ```
   IStakingRewardsManager(staking_contract).update_rewards(
       staker_address: staker_A,
       disable_rewards: true
   )
   ```
5. The call succeeds: `last_reward_block` is set to block N, no rewards are distributed.
6. The legitimate sequencer now calls `update_rewards(staker_A, disable_rewards: false)` — it reverts with `REWARDS_ALREADY_UPDATED`.
7. Staker A's `unclaimed_rewards_own` remains zero for block N.
8. Attacker repeats step 4 for every subsequent block → stakers receive zero consensus rewards indefinitely.
