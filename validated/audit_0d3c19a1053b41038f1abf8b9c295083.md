### Title
Missing Validation of Participant Contributions in CKD Coordinator Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator accumulates `(norm_big_y, norm_big_c)` shares from every participant with no validity checks whatsoever. A single malicious participant can inject arbitrary group elements, causing the coordinator to produce a `CKDOutput` that decrypts to a wrong derived key. Honest parties accept this corrupted output with no indication of failure.

### Finding Description
In `do_ckd_coordinator`, the coordinator iterates over all participant contributions and blindly adds them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

No check is performed on the received values. Contrast this with the honest computation in `compute_signature_share`, which produces:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)` [2](#0-1) 

The coordinator has no way to verify that the received pair is consistent with the sender's public key share, because doing so would require knowledge of `app_sk`. There are also no elementary sanity checks (e.g., non-identity element checks) applied to the received `big_y` or `big_c` values before they are accumulated.

The analog in the Chainlink report is exact: just as `latestRoundData()` outputs were consumed without checking `price > 0`, `updateTime != 0`, and `answeredInRound >= roundID`, the CKD coordinator consumes participant outputs without checking that they are non-identity and consistent with the protocol state.

### Impact Explanation
The final `CKDOutput` is an ElGamal ciphertext `(Y, C)` intended to encrypt `msk · H(pk, app_id)` under the application public key `A`. If participant `i` sends `(Y'_i, C'_i)` instead of its honest share, the coordinator computes:

```
Y = Y'_i + Σ_{j≠i} λ_j·y_j·G
C = C'_i + Σ_{j≠i} λ_j·(x_j·H(pk,app_id) + y_j·A)
```

Decryption yields `C - app_sk·Y = (C'_i - app_sk·Y'_i) + Σ_{j≠i} λ_j·x_j·H(pk,app_id)`, which differs from the correct derived key `msk·H(pk,app_id)` by an attacker-controlled additive offset. The coordinator outputs this corrupted ciphertext as a valid `CKDOutput` with no error, and all honest parties accept it.

This matches: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
- The CKD protocol requires **all** participants to contribute (no threshold reconstruction); a single malicious participant suffices.
- The attack requires only that the malicious participant deviate from the protocol when sending its private message to the coordinator — a trivial capability for any library caller who controls their own participant instance.
- There is no post-hoc verification of the coordinator's output against the public key or any commitment, so the corruption is undetectable by honest parties.

### Recommendation
1. **Non-identity checks**: At minimum, reject any received `big_y` or `big_c` that is the group identity element before accumulating it.
2. **Zero-knowledge consistency proof**: Each participant should accompany `(norm_big_y, norm_big_c)` with a proof of correct formation relative to their public key share (e.g., a Schnorr proof that `norm_big_c - norm_big_y·(app_pk/G)` lies on the correct subspace). This is the structural fix analogous to the `_checkAnswer()` function introduced in the Chainlink fix.
3. **Threshold reconstruction**: Consider restructuring CKD to tolerate up to `t-1` malicious participants by using a threshold-based aggregation with verifiable shares, rather than requiring all `N` participants to contribute honestly.

### Proof of Concept
A malicious participant overrides its protocol implementation to send `(G1::identity(), G1::identity())` as its `CKDOutput` to the coordinator.

The coordinator accumulates:
```
norm_big_y = identity + Σ_{j≠i} λ_j·y_j·G   // missing participant i's term
norm_big_c = identity + Σ_{j≠i} λ_j·(x_j·H(pk,app_id) + y_j·A)
```

Decryption by the application gives `Σ_{j≠i} λ_j·x_j·H(pk,app_id)`, which is not `msk·H(pk,app_id)`. The coordinator returns this as a successful `CKDOutput` (type `Some(ckd_output)`), and the TEE application derives a wrong confidential key with no error signal. [3](#0-2)
