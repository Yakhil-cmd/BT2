### Title
`induced_state_diff` Unconditionally Includes Class Hash for Reverted DeclareV0/V1 Transactions, Producing False State Diff in RPC Simulation — (`crates/apollo_rpc_execution/src/lib.rs`, `crates/apollo_rpc_execution/src/execution_utils.rs`)

### Summary

In `execute_transactions`, the `deprecated_declared_class_hash` is extracted from the transaction struct **before** execution and passed unconditionally to `induced_state_diff` regardless of whether the transaction reverted. `induced_state_diff` then unconditionally populates `deprecated_declared_classes` from this parameter, not from the actual blockifier state diff. A reverted DeclareV0 or DeclareV1 transaction therefore produces a simulation output whose `induced_state_diff.deprecated_declared_classes` contains the class hash even though the class was never actually declared.

### Finding Description

In `execute_transactions`: [1](#0-0) 

The `deprecated_declared_class_hash` is extracted from the tx struct at line 733–746 for `DeclareV0` and `DeclareV1`. Then at line 750–751, `blockifier_tx.execute(...)` is called and its result stored in `tx_execution_info_result`. At line 752–753, `induced_state_diff` is called with `deprecated_declared_class_hash` **before** the execution result is inspected. `transactional_state.commit()` is called at line 754, and only at line 755–761 is `tx_execution_info_result` checked for errors.

In the blockifier, a reverted transaction is **not** an `Err` — it returns `Ok(TransactionExecutionInfo { revert_error: Some("..."), ... })`. The `?` at line 761 therefore does not short-circuit for a revert. The `TransactionExecutionOutput` with the false `induced_state_diff` is pushed to `res` at line 762–766.

In `induced_state_diff`: [2](#0-1) 

`deprecated_declared_classes` is set purely from the `deprecated_declared_class_hash` parameter via `map_or_else`. There is no check of the execution result or revert status. If the parameter is `Some(class_hash)`, the class hash is unconditionally included.

The only place where `revert_error` is checked is in `estimate_fee`: [3](#0-2) 

`estimate_fee` returns early with `RevertedTransaction` on revert, so the wrong `induced_state_diff` is never surfaced there. But the simulation path (which calls `execute_transactions` and returns `TransactionExecutionOutput` including `induced_state_diff`) has no such guard.

### Impact Explanation

`simulateTransactions` RPC responses will include an `induced_state_diff` that falsely lists a deprecated class hash in `deprecated_declared_classes` for any DeclareV0 or DeclareV1 transaction whose `__validate__` or execution reverts. A caller relying on the simulation state diff to determine whether a class was declared will receive an authoritative-looking but incorrect answer. This fits the allowed impact: **High — RPC simulation returns an authoritative-looking wrong value**.

### Likelihood Explanation

Any unprivileged user can submit a `simulateTransactions` RPC call with a DeclareV0 or DeclareV1 transaction whose `__validate__` is crafted to revert (e.g., by using an account contract whose validate logic fails under certain calldata). No privileged access is required. The trigger is entirely attacker-controlled via public RPC inputs.

### Recommendation

In `execute_transactions`, after obtaining `tx_execution_info_result`, check whether the execution reverted before passing the class hash to `induced_state_diff`. Specifically, if `tx_execution_info_result` is `Ok` but `execution_info.revert_error` is `Some(...)`, pass `None` as `deprecated_declared_class_hash` to `induced_state_diff` so that the reverted class declaration is not included in the output state diff.

Alternatively, modify `induced_state_diff` to accept the full `TransactionExecutionInfo` (or a revert flag) and suppress the class hash when the transaction reverted.

### Proof of Concept

The existing ignored test at `crates/apollo_rpc_execution/src/execution_test.rs:658` already notes `// TODO(yair): Add a reverted transaction.` [4](#0-3) 

A concrete test would:
1. Build a `DeclareV1` (`ExecutableTransactionInput::DeclareV1`) with an account whose `__validate__` reverts.
2. Call `execute_simulate_transactions` (or `execute_transactions` directly).
3. Assert that `result[0].induced_state_diff.deprecated_declared_classes` is empty.

Under the current code, the assertion fails — the class hash from the tx struct appears in `deprecated_declared_classes` despite the revert.

```rust
pub fn induced_state_diff(
    transactional_state: &mut CachedState<MutRefState<'_, CachedState<ExecutionStateReader>>>,
    deprecated_declared_class_hash: Option<ClassHash>,
) -> ExecutionResult<ThinStateDiff> {
    let blockifier_state_diff =
        CommitmentStateDiff::from(transactional_state.to_state_diff()?.state_maps);

    Ok(ThinStateDiff {
        deployed_contracts: blockifier_state_diff.address_to_class_hash,
        storage_diffs: blockifier_state_diff.storage_updates,
        class_hash_to_compiled_class_hash: blockifier_state_diff.class_hash_to_compiled_class_hash,
        deprecated_declared_classes: deprecated_declared_class_hash
            .map_or_else(Vec::new, |class_hash| vec![class_hash]),
        nonces: blockifier_state_diff.address_to_nonce,
    })
}
```
