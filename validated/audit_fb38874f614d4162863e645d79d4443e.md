### Title
`EVMAddress.deposit()` Has No Corresponding Cadence-Level Withdrawal — FLOW Tokens Sent to Non-COA EVM Addresses Are Permanently Inaccessible - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVMAddress` struct exposes a public `deposit(from: @FlowToken.Vault)` function callable by any unprivileged Cadence transaction, but provides **no `withdraw()` function**. Only `CadenceOwnedAccount` (COA) resources have a Cadence-level `withdraw()`. FLOW tokens deposited to any non-COA EVM address — including EVM contracts with no withdrawal logic, addresses with no accessible private key, or the zero address — are permanently inaccessible from the Cadence layer with no protocol-level recovery path.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, the `EVMAddress` struct (lines 157–237) exposes:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) { ... }
```

This function is `access(all)` — callable by any Cadence transaction or script — and deposits FLOW tokens into the EVM state for the given address. There is **no `withdraw()` function** on `EVMAddress`.

The only Cadence-level withdrawal path exists exclusively on `CadenceOwnedAccount` (lines 586–606):

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault { ... }
```

This entitlement-gated function is only available to the owner of a COA resource. For any EVM address that is **not** a COA — including:
- EVM smart contracts deployed without a withdrawal function,
- EOA addresses whose private key is inaccessible or lost,
- The zero address or any burn address,

…there is no Cadence-level mechanism to retrieve deposited FLOW tokens. The only recovery path is an EVM-signed transaction from the private key of that address, which may not exist.

This is structurally identical to the FeeReceiver bug: a receiver contract (here, `EVMAddress`) accepts tokens via a deposit path but provides no withdrawal path.

---

### Impact Explanation

**Cross-VM asset loss.** Any FLOW tokens deposited via `EVMAddress.deposit()` to a non-COA EVM address are permanently locked at the Cadence layer if the EVM address has no EVM-level withdrawal mechanism. The tokens exist in the EVM state but cannot be recovered to Cadence without a private key for that EVM address. This is a one-way bridge with no guaranteed return path for non-COA targets.

---

### Likelihood Explanation

**Medium.** The `EVMAddress.deposit()` function is `access(all)` and callable by any unprivileged Cadence transaction. Users or contracts that deposit FLOW to an EVM contract address (e.g., a Solidity contract that only has a `receive()` fallback but no `withdraw()`) will permanently lose those tokens at the Cadence layer. The `EVM.run()` function also accepts a `coinbase: EVMAddress` parameter (line 828) — if the coinbase is set to a non-COA EVM contract address, all accumulated gas fees are permanently locked in that EVM address with no Cadence-level recovery.

---

### Recommendation

1. Add a clear warning in the `EVMAddress.deposit()` NatDoc that tokens deposited to non-COA EVM addresses can only be retrieved via EVM-signed transactions, and are permanently inaccessible from Cadence if the address has no private key or no EVM-level withdrawal function.
2. Consider restricting `EVMAddress.deposit()` to only accept COA addresses, or adding a protocol-level Cadence withdrawal path for non-COA EVM addresses (e.g., via a privileged admin function on the EVM contract).
3. For the `EVM.run()` coinbase parameter, validate or document that the coinbase address should be a COA address to ensure gas fees are recoverable from Cadence.

---

### Proof of Concept

Any unprivileged Cadence transaction can execute the following to permanently lock FLOW tokens:

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>

transaction(evmContractBytes: [UInt8; 20]) {
    prepare(account: auth(BorrowValue) &Account) {
        // Mint or obtain FLOW tokens
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin
        )!
        let minter <- admin.createNewMinter(allowedAmount: 10.0)
        let vault <- minter.mintTokens(amount: 10.0)
        destroy minter

        // Deposit to a non-COA EVM address (e.g., a Solidity contract with no withdraw())
        // EVMAddress has deposit() but NO withdraw()
        let target = EVM.EVMAddress(bytes: evmContractBytes)
        target.deposit(from: <-vault)

        // 10.0 FLOW is now permanently locked in the EVM state.
        // There is no Cadence-level function to retrieve it.
        // The only recovery path is an EVM transaction signed by the
        // private key of evmContractBytes — which may not exist.
    }
}
```

Root cause confirmed at:
- `EVMAddress.deposit()` — `fvm/evm/stdlib/contract.cdc` lines 201–223 (deposit exists, no withdraw)
- `CadenceOwnedAccount.withdraw()` — `fvm/evm/stdlib/contract.cdc` lines 586–606 (withdraw only on COA)
- `EVM.run(tx:coinbase:)` — `fvm/evm/stdlib/contract.cdc` lines 828–836 (coinbase is `EVMAddress`, not required to be a COA)
- Go-level deposit implementation — `fvm/evm/impl/impl.go` lines 631–683 (no withdrawal counterpart on `EVMAddress`)
