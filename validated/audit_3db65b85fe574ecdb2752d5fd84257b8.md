### Title
Cross-COA Proof Replay: Unsigned `CapabilityPath` and `evmAddress` Allow Proof for COA X to Authenticate as COA Y — (`fvm/evm/stdlib/contract.cdc`, `fvm/evm/precompiles/arch.go`)

---

### Summary

The `validateCOAOwnershipProof` Cadence function verifies signatures only over the caller-supplied `signedData` blob. Neither the `CapabilityPath` nor the `evmAddress` is included in what is signed. Because `COAOwnershipProof` is RLP-encoded and passed as an opaque `bytes` argument to the EVM precompile, an attacker can freely substitute a different `CapabilityPath` and a different `evmAddress` while reusing the original signatures verbatim. If the target account owns two COAs, a proof obtained for COA X can be replayed to authenticate as COA Y.

---

### Finding Description

**Call path:**

1. An EVM transaction calls the Cadence Arch precompile with selector `ProofVerifierFuncSig` (`verifyCOAOwnershipProof(address,bytes32,bytes)`).
2. `proofVerifier.Run` calls `DecodeABIEncodedProof`, which reads three independent fields from the ABI-encoded input:
   - `caller` — the attacker-supplied `evmAddress` (first ABI argument, type `address`)
   - `hash` — the `signedData` (second ABI argument, type `bytes32`)
   - `encodedProof` — an RLP blob containing `{KeyIndices, FlowAddress, CapabilityPath, Signatures}` (third ABI argument, type `bytes`) [1](#0-0) 

3. `NewCOAOwnershipProofInContext` assembles these three independently-decoded pieces into a `COAOwnershipProofInContext`. The `EVMAddress` field comes from the first ABI argument; the `CapabilityPath` comes from inside the RLP blob. [2](#0-1) 

4. `coaOwnershipProofValidator` invokes the Cadence function `validateCOAOwnershipProof` with all six fields. [3](#0-2) 

5. Inside `validateCOAOwnershipProof`, the signature check verifies `keyList.verify(signatureSet, signedData, ...)` — **only over `signedData`**. Neither `evmAddress` nor `CapabilityPath` is part of the signed message. [4](#0-3) 

6. After the signature check passes, the function borrows the COA capability from the attacker-supplied `path` and checks `coaRef.address().bytes == evmAddress`. If the attacker supplies `path=/public/coa2` and `evmAddress=Y`, and coa2 genuinely lives at that path with address Y, this check passes. [5](#0-4) 

**The `COAOwnershipProof` struct** (`KeyIndices`, `Address`, `CapabilityPath`, `Signatures`) is RLP-encoded and passed as an opaque blob. The signatures inside it are over `signedData`, not over the struct itself. An attacker can re-encode the struct with a different `CapabilityPath` without touching the signatures. [6](#0-5) 

The code itself acknowledges the gap: [7](#0-6) 

The comment's mitigation claim — "the EVM-side precompile always passes the calling COA's address as the evmAddress argument" — is false for the general case. `DecodeABIEncodedProof` reads `evmAddress` from the first ABI argument, which is fully attacker-controlled; there is no enforcement that it equals `msg.sender` or any other trusted value. [8](#0-7) 

---

### Impact Explanation

Any EVM smart contract that calls `verifyCOAOwnershipProof` to gate access based on COA identity can be bypassed. An attacker who holds a legitimately-obtained proof for COA X (e.g., obtained by running a service that asks users to prove COA ownership) can replay it against COA Y owned by the same Cadence account, causing the precompile to return `true` for Y. This constitutes unauthorized EVM-level authentication as COA Y — a sandboxed host capability bypass.

---

### Likelihood Explanation

Preconditions are realistic:
- A Cadence account owning two COAs is protocol-valid and increasingly common as multi-COA patterns emerge.
- A valid proof for coa1 can be obtained by any party that asked the account holder to sign arbitrary `signedData` (e.g., a DApp login flow, a bridge, or any off-chain authentication service). The `signedData` is explicitly described as "arbitrary" and set by the proof requester.
- The attacker only needs to re-RLP-encode the `COAOwnershipProof` with a different `CapabilityPath` — a trivial operation requiring no cryptographic capability.

---

### Recommendation

Include both `evmAddress` and `CapabilityPath` in the signed commitment. The simplest fix is to define `signedData` as a canonical encoding of `(evmAddress ‖ capabilityPath ‖ applicationData)` and enforce this inside `validateCOAOwnershipProof` before calling `keyList.verify`. Alternatively, bind the proof to a specific `evmAddress` by hashing it into the domain separation tag. The comment at line 1003 already identifies the correct fix ("Callers building off-chain authentication flows on top of this function should ensure `signedData` encodes `evmAddress`") but leaves it as a caller responsibility rather than a protocol invariant.

---

### Proof of Concept

```
Setup (localnet / emulator):
  1. Account A creates two COAs:
       coa1 → stored at /storage/coa1, published at /public/coa1, EVM address X
       coa2 → stored at /storage/coa2, published at /public/coa2, EVM address Y

  2. Account A signs signedData = 0xdeadbeef... with key 0.
     Produces proof1 = COAOwnershipProof{KeyIndices:[0], Address:A, CapabilityPath:"coa1", Signatures:[sig]}

Attack:
  3. Attacker re-encodes proof2 = COAOwnershipProof{KeyIndices:[0], Address:A, CapabilityPath:"coa2", Signatures:[sig]}
     (identical signatures, only CapabilityPath changed)

  4. Attacker ABI-encodes call:
       verifyCOAOwnershipProof(evmAddress=Y, signedData=0xdeadbeef..., encodedProof=RLP(proof2))

  5. EVM tx calls Cadence Arch precompile with the above input.

  6. validateCOAOwnershipProof:
       - keyList.verify(sig over 0xdeadbeef...) → true  (sig is valid)
       - borrow<&EVM.CadenceOwnedAccount>(/public/coa2) → succeeds
       - coaRef.address().bytes == Y → true
       → returns ValidationResult{isValid: true}

  7. Assert result.isValid == true  ← proof for coa1 authenticated coa2
```

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
