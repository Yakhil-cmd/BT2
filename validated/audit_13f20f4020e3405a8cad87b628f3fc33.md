### Title
Reward Address Can Be Set to the Pool Contract Itself, Permanently Freezing Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

Both `enter_delegation_pool` and `change_reward_address` in the Pool contract accept a caller-supplied `reward_address` without validating that it differs from the pool contract's own address (or the staking contract's address). When `claim_rewards` is subsequently called, the STRK reward tokens are transferred directly to the pool contract itself, where they are permanently frozen with no recovery path.

An identical gap exists in `src/staking/staking.cairo` for the `stake` and `change_reward_address` functions, where a staker can set `reward_address` to the staking contract address, causing the same freeze on the staking side.

---

### Finding Description

`enter_delegation_pool` in `src/pool/pool.cairo` validates only that `reward_address` is not the token contract address:

```cairo
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [1](#0-0) 

There is no check that `reward_address != get_contract_address()`. A pool member can therefore register (or later update via `change_reward_address`) with `reward_address` equal to the pool contract's own address.

`change_reward_address` in the pool contract performs the same single check:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

When `claim_rewards` is called, the reward tokens are unconditionally transferred to whatever address is stored as `reward_address`:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [3](#0-2) 

If `reward_address` is the pool contract itself, the STRK tokens land in the pool contract's balance. The pool contract exposes no sweep or recovery function; the only outbound token flows are `transfer_to_staking_contract` (deposits into staking) and `exit_delegation_pool_action` (returns principal to `pool_member`). Neither path can recover tokens that arrived as a misrouted reward transfer. The funds are permanently frozen.

The same root cause exists in `src/staking/staking.cairo`: `stake` and `change_reward_address` only guard against `reward_address` being a registered token address, not against it being the staking contract itself. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any accumulated STRK rewards for a pool member whose `reward_address` equals the pool contract address are transferred into the pool contract on every `claim_rewards` call and cannot be recovered. The same applies to a staker whose `reward_address` equals the staking contract address. The affected yield is permanently lost to the user and to the protocol.

---

### Likelihood Explanation

**Moderate.** The entry points (`enter_delegation_pool`, `change_reward_address`) are callable by any unprivileged pool member or staker with no special role required. The scenario is a realistic accidental mis-paste of the pool/staking contract address as the reward destination — exactly the class of user error the external report describes. A malicious actor could also deliberately trigger this to grief a specific staker/delegator (e.g., by front-running a `change_reward_address` call if the caller is a contract that can be tricked, though the primary risk is accidental).

---

### Recommendation

Add an explicit guard in both `enter_delegation_pool` and `change_reward_address` (pool contract) and in `stake` and `change_reward_address` (staking contract) rejecting the contract's own address as a reward destination:

```cairo
assert!(
    reward_address != get_contract_address(),
    "{}",
    GenericError::REWARD_ADDRESS_IS_SELF,
);
```

Apply the same guard against the staking contract address in the pool's `enter_delegation_pool` and `change_reward_address`.

---

### Proof of Concept

1. Alice deploys or uses an existing pool. She calls `enter_delegation_pool(reward_address: pool_contract_address, amount: X)`.
   - The only check is `token_address != reward_address` — the pool contract address is not the token, so this passes. [6](#0-5) 

2. Rewards accrue over epochs. Alice (or anyone) calls `claim_rewards(pool_member: alice)`.

3. Inside `claim_rewards`, the contract reads `reward_address = pool_member_info.reward_address` (which is `pool_contract_address`) and executes:
   ```cairo
   reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
   ``` [7](#0-6) 

4. STRK tokens are transferred to the pool contract itself. The pool contract has no recovery function. The rewards are permanently frozen.

5. The same scenario applies via `change_reward_address` for an existing pool member who updates their reward address to the pool contract address. [8](#0-7)

        fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
            assert!(
                self.token_dispatcher.contract_address.read() != reward_address,
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let old_address = pool_member_info.reward_address;

            // Update reward_address and commit to storage.
            pool_member_info.reward_address = reward_address;
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardAddressChanged {
                        pool_member, new_address: reward_address, old_address,
                    },
                );
        }
