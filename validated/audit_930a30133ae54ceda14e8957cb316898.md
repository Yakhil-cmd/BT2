### Title
Unchecked Precompile Call Return Value in Assembly Blocks Causes Silent Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

The `withdrawToNear` and `withdrawToEthereum` functions in `EvmErc20` and `EvmErc20V2` use undocumented inline assembly blocks to call the Aurora exit precompiles. In every case, the return value of the low-level `call` opcode is captured in a local variable (`res`) but **never checked**. Because the ERC-20 token burn (`_burn`) executes unconditionally before the assembly call, any failure of the precompile call results in tokens being permanently destroyed with no corresponding cross-chain transfer initiated and no revert to the caller.

---

### Finding Description

In `EvmErc20.withdrawToNear`:

```solidity
_burn(_msgSender(), amount);                          // tokens destroyed here

bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
uint input_size = 1 + 32 + recipient.length;

assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    // res is NEVER checked — a failed call is silently ignored
}
``` [1](#0-0) 

The same pattern appears in `EvmErc20.withdrawToEthereum`, `EvmErc20V2.withdrawToNear`, and `EvmErc20V2.withdrawToEthereum`: [2](#0-1) [3](#0-2) [4](#0-3) 

The assembly blocks are entirely undocumented: the magic flag byte `\x01`, the binary encoding layout, and the precompile addresses are unexplained inline. The Rust-side precompile validates inputs and returns `ExitError` on failure (e.g., `ERR_INVALID_INPUT`, `ERR_TARGET_TOKEN_NOT_FOUND`, `ERR_INVALID_RECEIVER_ACCOUNT_ID`): [5](#0-4) 

When the precompile returns an error, the EVM `call` opcode returns `0` (failure). Because `res` is never inspected, the Solidity function does not revert. The burn is already committed and irreversible.

The precompile enforces a `MAX_INPUT_SIZE` of 1,024 bytes and validates the recipient as a valid NEAR account ID: [6](#0-5) [7](#0-6) 

Any input that fails these checks causes the precompile to return an error, which the Solidity caller silently ignores.

Additionally, the `ExitToNear` and `ExitToEthereum` precompiles can be paused by the engine operator: [8](#0-7) 

If either precompile is paused, the call returns `ERR_PAUSED` and the Solidity function still does not revert, burning tokens with no transfer.

---

### Impact Explanation

**Critical — Permanent freezing (burning) of user funds.**

A token holder who calls `withdrawToNear` or `withdrawToEthereum` with a recipient that fails precompile validation (e.g., a NEAR account ID exceeding the length limit, containing invalid characters, or not registered in the NEP-141 map) will have their ERC-20 tokens burned on the Aurora side with no corresponding NEP-141 transfer on the NEAR side. The tokens are permanently destroyed. There is no revert, no error event, and no recovery path.

---

### Likelihood Explanation

**Medium.** The functions are `external` and callable by any token holder. Failure conditions are reachable without any privileged action:

1. A user supplies a `recipient` bytes value that is not a valid NEAR account ID (invalid UTF-8, invalid characters, or exceeds `MAX_INPUT_SIZE - 33` bytes). The precompile returns `ERR_INVALID_RECEIVER_ACCOUNT_ID` or `ERR_INVALID_INPUT`; the call returns 0; tokens are burned.
2. A user calls `withdrawToNear` on an ERC-20 token whose address has no NEP-141 mapping in engine storage. The precompile returns `ERR_TARGET_TOKEN_NOT_FOUND`; same outcome.
3. If the precompile is paused by the operator, every user withdrawal silently burns tokens.

Scenarios 1 and 2 are reachable by any unprivileged user without any admin involvement.

---

### Recommendation

1. **Check the return value of every assembly `call`** and revert on failure:
   ```solidity
   assembly {
       let res := call(gas(), EXIT_TO_NEAR_PRECOMPILE, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
2. **Document every assembly block**: explain the precompile address derivation (`keccak("exitToNear")[12..]`), the binary input format (flag byte, amount encoding, recipient encoding), and the expected failure modes.
3. **Define named constants** for the precompile addresses instead of embedding raw hex literals.
4. **Validate the recipient** in Solidity before calling the precompile, so failures are caught with a meaningful revert message before the burn.

---

### Proof of Concept

1. Deploy `EvmErc20` with a valid admin.
2. Mint 1000 tokens to `alice`.
3. `alice` calls `withdrawToNear(bytes("invalid account id!!!"), 1000)` — the `!` characters make this an invalid NEAR account ID.
4. `_burn(alice, 1000)` executes; alice's balance drops to 0.
5. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` returns 0 (precompile rejects with `ERR_INVALID_RECEIVER_ACCOUNT_ID`).
6. `res` is never checked; the transaction succeeds with status `1`.
7. Alice's 1000 tokens are permanently destroyed. No NEP-141 transfer occurs on NEAR. No event indicates failure. [1](#0-0) [9](#0-8)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
