### Title
`skip_stateful_validations` Admits Invalid-Signature Invoke Transactions via Overly Broad `account_tx_in_pool_or_recent_block` Proxy Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 when the account's deploy_account transaction is still pending in the mempool. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the pool for that address, not exclusively a deploy_account transaction. An unprivileged attacker who observes a victim's deploy_account transaction in the mempool can immediately submit an invoke transaction with nonce=1 carrying an **arbitrary/invalid signature** for the victim's address. The gateway skips signature validation and admits the transaction to the mempool.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when two conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).

When both are true it calls `account_tx_in_pool_or_recent_block(sender)` and, if that returns `true`, returns `true` (skip validation). [1](#0-0) 

The code comment reads:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`state.contains_account` returns `true` if the address appears in the mempool's `staged` or `committed` maps — i.e., for **any** transaction type that was ever staged or committed for that address. [3](#0-2) 

`tx_pool.contains_account` returns `true` if **any** transaction (invoke, declare, or deploy_account) for that address is currently in the pool. [4](#0-3) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the `__validate__` call is unconditionally skipped:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [6](#0-5) 

**Concrete attack path:**

1. Victim Alice submits a `deploy_account` transaction (nonce=0). It is admitted to the mempool. `tx_pool.contains_account(Alice)` is now `true`.
2. Attacker Bob immediately submits an `Invoke` transaction with `nonce=1` for Alice's address, carrying a completely invalid/arbitrary signature.
3. Gateway stateless validation passes (no signature check at that layer).
4. `validate_nonce` passes: `account_nonce(0) ≤ tx_nonce(1) ≤ max_allowed_nonce_gap`.
5. `validate_by_mempool` passes: no existing nonce=1 tx for Alice yet.
6. `skip_stateful_validations` fires: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(Alice)==true` → returns `true`.
7. `run_validate_entry_point` is called with `validate=false` → `__validate__` is never invoked.
8. Bob's invalid-signature invoke is admitted to the mempool.

If Bob submits before Alice's own nonce=1 invoke, he occupies the slot. If Alice later submits her legitimate nonce=1 invoke, it must pass fee-escalation against Bob's tx. If Bob's fee is higher, Alice's tx is rejected. When the batcher eventually executes Bob's tx (after Alice's deploy_account is processed), `__validate__` runs and fails, the tx is reverted — but Alice's legitimate nonce=1 tx has already been evicted from the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction.**

The gateway's core invariant is that every admitted transaction has passed signature validation. This invariant is broken: an invoke transaction with a completely invalid signature is admitted to the mempool. The attacker can:

- Occupy Alice's nonce=1 slot with an invalid-signature tx, forcing Alice's legitimate tx out via fee escalation.
- Repeat this for every account that has a pending deploy_account in the mempool, causing a systematic DoS against all new account deployments.
- The invalid tx will eventually fail at execution time (batcher runs `__validate__`), but by then the damage to the victim's mempool slot is done.

---

### Likelihood Explanation

**Medium-High.** The trigger is entirely unprivileged: the attacker only needs to observe that a deploy_account transaction exists in the mempool for a target address (observable via P2P propagation or RPC). No private key material, no privileged access, and no special timing beyond submitting before the victim's own nonce=1 invoke is required.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` proxy with a check that specifically verifies a **deploy_account** transaction exists in the pool for the sender address. The current check conflates "account has any pending transaction" with "account has a pending deploy_account," which is the only case that justifies skipping signature validation.

```rust
// Instead of:
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await ...;

// Use a dedicated query:
return mempool_client
    .deploy_account_tx_in_pool(tx.sender_address())
    .await ...;
```

Alternatively, add a type-discriminated check inside the mempool so that `account_tx_in_pool_or_recent_block` only returns `true` when the matching transaction is a `DeployAccount`.

---

### Proof of Concept

```
// State: Alice's account has on-chain nonce = 0 (not yet deployed).

// Step 1: Alice submits deploy_account (nonce=0) — admitted normally.
gateway.add_tx(alice_deploy_account_tx, None).await  // OK

// Step 2: Attacker submits invoke with nonce=1 and INVALID signature for Alice.
let attacker_invoke = RpcInvokeTransactionV3 {
    sender_address: alice_address,
    nonce: Nonce(Felt::ONE),
    signature: TransactionSignature(vec![Felt::from(0xdeadbeef_u64)]), // garbage
    // ... valid resource bounds, calldata, etc.
};
gateway.add_tx(RpcTransaction::Invoke(attacker_invoke), None).await
// Expected: Err(ValidateFailure)
// Actual:   Ok(tx_hash)  ← admitted without signature check
```

The gateway reaches `skip_stateful_validations`, finds `account_tx_in_pool_or_recent_block(alice) == true` (Alice's deploy_account is in the pool), sets `skip_validate = true`, and calls `run_validate_entry_point` with `validate: false`, bypassing `__validate__` entirely. async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
