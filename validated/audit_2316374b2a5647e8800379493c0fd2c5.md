### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Deployed Accounts with Pending Nonce-0 Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce 1 when the account is not yet deployed on-chain (nonce = 0), provided a deploy_account transaction exists in the mempool. However, the check used — `account_tx_in_pool_or_recent_block` — returns `true` for **any** account that has any transaction in the pool, not exclusively a deploy_account transaction. A deployed account with on-chain nonce 0 that has a pending nonce-0 invoke in the mempool satisfies the skip condition, allowing an attacker to submit a nonce-1 invoke with an **invalid signature** that bypasses `__validate__` and is admitted to the mempool.

---

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`, meaning the account's `__validate__` entry point (signature verification) is never executed:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

The skip condition fires when:
1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The account's on-chain nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender)` returns `true`

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`MempoolState::contains_account` returns `true` if the address appears in the staged or committed maps — it does **not** distinguish between deploy_account and invoke transactions:

```rust
// crates/apollo_mempool/src/mempool.rs:115-117
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [4](#0-3) 

The code comment asserts the check is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is incorrect for the following scenario:

**Exploit path:**
1. Account A is already deployed (class hash ≠ 0, on-chain nonce = 0).
2. Account A submits a valid invoke with nonce 0 → admitted to mempool with `__validate__` passing.
3. Attacker submits an invoke with nonce 1 from account A with an **invalid signature**.
4. Gateway stateful validator: `account_nonce = 0`, `tx.nonce() = 1` → `account_tx_in_pool_or_recent_block(A)` returns `true` (because the nonce-0 invoke is in the pool).
5. `skip_stateful_validations` returns `true` → `__validate__` is **not called**.
6. The nonce-1 invoke with invalid signature is admitted to the mempool.

The `validate_nonce` check permits nonce 1 when `account_nonce = 0` (within the `max_allowed_nonce_gap = 200` window):

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:287-296

        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));

        return Err(create_error(...));

}

---

### Impact Explanation

An unprivileged attacker who observes the public mempool can identify any deployed account with on-chain nonce 0 that has a pending nonce-0 transaction. The attacker can then submit a nonce-1 invoke with an arbitrary (invalid) signature for that account. The gateway admits the transaction without calling `__validate__`. When the batcher later executes the transaction, `__validate__` is called and fails, the transaction is reverted, and the victim account is charged a fee. This constitutes:

- **Mempool/gateway admission accepts invalid transactions before sequencing** (High impact per the allowed scope).
- The admitted transaction carries a wrong signer/signature, breaking the admission invariant that every accepted invoke transaction must have passed account-level signature verification.

---

### Likelihood Explanation

The mempool is publicly observable via P2P gossip. Any deployed account that has submitted its first transaction (nonce 0) but has not yet had it included in a block is a valid target. This is a common state for active accounts. The attacker requires no special privileges — only the ability to submit transactions to the gateway.

---

### Recommendation

The `skip_stateful_validations` function must verify that the account has a **deploy_account** transaction specifically, not just any transaction. The mempool should expose a dedicated query such as `has_deploy_account_in_pool(address)` that checks whether the pending transaction at nonce 0 for the given address is of type `DeployAccount`. Alternatively, the gateway can read the account's class hash from state: if `get_class_hash_at(address) != ClassHash::default()`, the account is already deployed and `__validate__` must never be skipped regardless of the nonce-0 on-chain state.

---

### Proof of Concept

```
1. Deploy account A on-chain (class hash set, nonce = 0).
2. Submit invoke_A_nonce0 (valid signature, nonce=0) from account A → admitted to mempool.
   mempool.account_tx_in_pool_or_recent_block(A) now returns true.
3. Construct invoke_A_nonce1_bad_sig (nonce=1, signature=[0xdead, 0xbeef]) from account A.
4. Submit invoke_A_nonce1_bad_sig to the gateway.
   Gateway stateful validator:
     account_nonce = get_nonce_from_state(A) = 0
     tx.nonce() = 1
     validate_nonce: 0 <= 1 <= 200 → OK
     validate_by_mempool: no duplicate → OK
     skip_stateful_validations:
       tx.nonce() == 1 && account_nonce == 0 → true
       account_tx_in_pool_or_recent_block(A) → true (invoke_A_nonce0 is in pool)
       returns skip=true
     run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false → __validate__ NOT called
5. invoke_A_nonce1_bad_sig is admitted to the mempool with an invalid signature.
6. Batcher executes invoke_A_nonce1_bad_sig:
     __validate__ is called → fails (bad signature) → transaction reverted, fee charged to A.

/// Perform transaction validation by the mempool.
async fn validate_by_mempool(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<()> {
    let validation_args = ValidationArgs::new(tx, account_nonce);
    mempool_client
        .validate_tx(validation_args)
        .await
        .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
}

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
