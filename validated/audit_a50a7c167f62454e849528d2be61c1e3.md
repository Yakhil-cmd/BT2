### Title
Missing Validation of `app_pk` Identity Point in `ckd()` Allows Coordinator to Learn Confidential Derived Secret - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd()` function does not validate that the caller-supplied `app_pk` (the TEE app's ElGamal public key `A`) is not the G1 identity point. When `app_pk = G1::identity()`, the ElGamal blinding term `y_i · A` collapses to the identity, stripping all confidentiality from the BLS signature shares. The coordinator then directly observes `msk · H(pk, app_id)` — the confidential derived secret — violating the core security requirement that no single MPC node can compute `s`.

### Finding Description

The CKD protocol is designed so that each node `i` computes:

```
C_i = S_i + y_i · A   where S_i = x_i · H(pk, app_id)
```

and sends `(λ_i · Y_i, λ_i · C_i)` to the coordinator. The coordinator aggregates:

```
C = msk · H(pk, app_id) + a · Y
```

The app recovers the BLS signature via `C − a · Y = msk · H(pk, app_id)`. The coordinator cannot compute this because it does not know `a`.

When `app_pk = G1::identity()`, the blinding term vanishes:

```
C_i = x_i · H(pk, app_id) + y_i · identity = x_i · H(pk, app_id)
```

The coordinator aggregates:

```
C = Σ λ_i · C_i = Σ λ_i · x_i · H(pk, app_id) = msk · H(pk, app_id)
```

The coordinator now holds the unblinded BLS signature directly in `CKDOutput.big_c`.

The `ckd()` function validates participant counts, duplicates, self-presence, and coordinator presence, but performs **no validation** on `app_pk`: [1](#0-0) 

The vulnerable computation in `compute_signature_share` where the missing guard would take effect: [2](#0-1) 

The `CKDOutput` type that the coordinator returns, now containing the unblinded secret: [3](#0-2) 

### Impact Explanation

The security requirement explicitly states:

> "No single node in the MPC network should be capable of computing `s`. This avoids key leakage in the case a single TEE is compromised."
> "Example values that should not be leaked even if a node is malicious are `s`, `msk` and private shares of other nodes." [4](#0-3) 

When `app_pk = identity`, the coordinator — a single MPC node — directly computes `msk · H(pk, app_id)` and therefore `s = HKDF(msk · H(pk, app_id))`. This is a **Critical** disclosure of a confidential derived secret to a single node, matching the allowed impact: *"Extraction, reconstruction, or disclosure of… confidential derived secrets."*

### Likelihood Explanation

The `app_pk` value originates from the TEE app and travels through the developer contract → MPC contract → MPC nodes before reaching `ckd()`. The library documentation explicitly delegates attestation and `A`-validation to the developer contract: [5](#0-4) 

The MPC library itself performs no such check. A malicious TEE app (or an operator whose developer contract omits the `A ≠ identity` check) can submit `A = G1::identity()` as a valid on-chain transaction. The `blstrs::G1Projective` type accepts the identity point without error, so no deserialization or type-level guard prevents this. The attack requires only a single malformed CKD request — no key material, no cryptographic break, no trusted-party compromise.

### Recommendation

Add an explicit identity-point guard in `ckd()` immediately after the coordinator-presence check:

```rust
use elliptic_curve::Group;

if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk cannot be the identity point: ElGamal blinding would be ineffective".to_string(),
    ));
}
``` [6](#0-5)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

    }

        self.big_y
    }

    /// Outputs `big_c`

        self.big_c
    }

    /// Takes a secret scalar and returns
    /// s <- C − a ⋅ Y = msk ⋅ H ( `app_id` )

        self.big_c - self.big_y * secret_scalar
    }
