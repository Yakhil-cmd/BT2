### Title
Malicious Participant Can Silently Corrupt CKD Output via Unverified Share Contributions — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates share contributions from all participants without verifying their correctness. Analogous to calling `transferFrom()` without checking its return value, the coordinator unconditionally incorporates received group elements into the derived key, silently accepting maliciously crafted values. A single malicious participant can corrupt the CKD output, causing honest parties to derive an incorrect and unusable confidential key with no detection mechanism.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`), the coordinator receives `CKDOutput` values from each participant and unconditionally adds them to the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

The protocol expects each participant to compute their share in `compute_signature_share` as:
- `big_y = λi · yi · G` (random blinding point)
- `big_c = λi · (xi · H(pk, app_id) + yi · app_pk)` (masked key share) [2](#0-1) 

No zero-knowledge proof, commitment, or consistency check is performed on the received `(big_y, big_c)` pair against the participant's known public key share. The coordinator blindly trusts the received values, exactly as the unchecked `transferFrom()` blindly trusts the token transfer succeeded.

By contrast, the DKG protocol — which handles a structurally similar aggregation — enforces proof-of-knowledge verification, commitment hash binding, and per-share validation before accepting any participant contribution: [3](#0-2) 

No equivalent safeguard exists in the CKD path.

---

### Impact Explanation

The final CKD output `(Y, C)` is unmasked via `C − app_sk · Y` to recover `msk · H(pk, app_id)`. If a malicious participant sends arbitrary `(big_y', big_c')`:

- `Y' = Y_honest + big_y'`
- `C' = C_honest + big_c'`
- Unmasked result: `C' − app_sk · Y' = msk · H(pk, app_id) + (big_c' − app_sk · big_y')`

The derived key is corrupted by an attacker-controlled offset. Honest parties have no mechanism to detect this — `CKDOutput::new` performs no validation, and `unmask` simply computes the linear combination. The output is silently wrong.

**Matched impact:** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

Any single participant in the CKD protocol can execute this attack. The attacker only needs to deviate from `compute_signature_share` and send arbitrary `ElementG1` values. No cryptographic capability, key material, or privileged position beyond being a protocol participant is required. The attack is trivially executable and undetectable by the coordinator or other honest participants.

---

### Recommendation

Add a zero-knowledge proof (e.g., a Schnorr-style proof of correct formation) to each participant's share contribution, proving that `(big_y, big_c)` is correctly computed relative to the participant's public key share and the public `app_pk`. This mirrors the proof-of-knowledge pattern already used in the DKG protocol (`proof_of_knowledge` / `internal_verify_proof_of_knowledge` in `src/dkg.rs`). [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. Run the CKD protocol with `n = 3` participants, one of which is malicious.
2. The malicious participant, instead of calling `compute_signature_share`, sends `(ElementG1::generator(), ElementG1::generator())` as their `CKDOutput` to the coordinator.
3. The coordinator in `do_ckd_coordinator` sums all contributions including the malicious `(G, G)`:
   - `Y' = Y_honest + G`
   - `C' = C_honest + G`
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned with no error.
5. When the caller invokes `ckd_output.unmask(app_sk)`, the result is:
   - `C' − app_sk · Y' = msk · H(pk, app_id) + G − app_sk · G = msk · H + (1 − app_sk) · G`
6. This is not equal to the expected `msk · H(pk, app_id)`. The derived confidential key is silently wrong, and no party detects the corruption. [6](#0-5) [7](#0-6)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}

fn internal_verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    participant: Participant,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: &Signature<C>,
) -> Result<(), ProtocolError> {
    // creates an identifier for the participant
    let id = participant.scalar::<C>();
    let vk_share = commitment
        .coefficients()
        .first()
        .ok_or_else(|| ProtocolError::AssertionFailed("Empty coefficient list".to_string()))?;

    let big_r = proof_of_knowledge.R();
    let z = proof_of_knowledge.z();
    let c = challenge::<C>(domain_separator, session_id, id, vk_share, big_r)?;
    if *big_r != <C::Group>::generator() * *z - vk_share.value() * c.to_scalar() {
        return Err(ProtocolError::InvalidProofOfKnowledge(participant));
    }
    Ok(())
}
