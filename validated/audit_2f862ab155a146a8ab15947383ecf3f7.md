### Title
ETH Permanently Frozen in `ExitToNear` Precompile When `ft_transfer` Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is absent (which is the default build configuration), the `ExitToNear` precompile sets `refund: None` in its callback arguments. If the downstream `ft_transfer` or `ft_transfer_call` NEAR promise fails, the ETH that was deducted from the caller's EVM balance and transferred to the `exit_to_near::ADDRESS` precompile address is permanently frozen with no recovery path.

---

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` handles bridging ETH and ERC-20 tokens from Aurora EVM to NEAR. When a user calls this precompile with ETH value, the EVM mechanics transfer the ETH from the caller's balance to `exit_to_near::ADDRESS` (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`), and the precompile schedules an async `ft_transfer` or `ft_transfer_call` promise on the eth-connector contract.

The critical defect is in how the callback arguments are constructed inside `ExitToNear::run()`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // ← no refund mechanism in default build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not compiled in, `refund` is unconditionally `None`. The callback handler in `engine/src/contract_methods/connector.rs` then reaches the `else` branch and does nothing:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← ETH is silently abandoned
}
``` [2](#0-1) 

The ETH remains permanently at `exit_to_near::ADDRESS`. The precompile has no "withdraw" or "rescue" entry point; it only handles the outbound exit flow. There is no way for any party — including the original sender — to recover the frozen ETH.

The `error_refund` feature is **not** part of the `default` feature set:

```toml
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [3](#0-2) 

A default build therefore ships without the refund path.

The refund mechanism itself (when the feature IS enabled) works by transferring ETH back from the precompile address to the original caller:

```rust
// ETH exit; transfer ETH back from precompile address
let exit_address = exit_to_near::ADDRESS;
engine.call(&exit_address, &refund_address, amount, ...)
``` [4](#0-3) 

Without the feature, this call is never scheduled.

---

### Impact Explanation

**Permanent freezing of user ETH funds (Critical).**

The ETH is transferred to `exit_to_near::ADDRESS` as part of the EVM call mechanics before the precompile logic runs. If the subsequent NEAR promise fails and `error_refund` is absent, the ETH is irrecoverably locked at that address. No user, relayer, or admin can extract it because the precompile exposes no withdrawal interface.

---

### Likelihood Explanation

The `ft_transfer` promise fails naturally and without any privileged action in at least one well-documented scenario: the recipient NEAR account is not registered with the NEP-141 eth-connector contract. This is a normal user error (sending to an unregistered account) that any unprivileged caller can trigger on themselves. The test suite explicitly models this exact failure mode:

```rust
exit_to_near(
    &ft_owner,
    // The ft_transfer will fail because this account is not registered with the NEP-141
    "unregistered.near",
    FT_EXIT_AMOUNT,
    ...
)
``` [5](#0-4) 

The test then confirms that without `error_refund`, the tokens are not returned:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

The ETH variant is confirmed by `test_exit_to_near_eth_refund`:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [7](#0-6) 

Any user who calls `ExitToNear` with ETH and a recipient that is not registered with the NEP-141 contract will permanently lose their ETH in a default build.

---

### Recommendation

Enable the `error_refund` feature in the production build, or promote it into the `default` feature set so it is always active. Alternatively, remove the compile-time gate entirely and always populate the `refund` field in `ExitToNearPrecompileCallbackArgs`, making the refund callback unconditional. The refund logic in `refund_on_error` is already correct and complete; the only missing piece is ensuring it is always reachable.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Alice holds 1 ETH on Aurora EVM.
2. Alice calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with `value = 1 ETH` and `recipient = "unregistered.near"` (any NEAR account not registered with the NEP-141 eth-connector).
3. The EVM deducts 1 ETH from Alice's balance and transfers it to `exit_to_near::ADDRESS`.
4. The precompile schedules `ft_transfer("unregistered.near", 1 ETH)` on the eth-connector, with a callback to `exit_to_near_precompile_callback` carrying `refund: None`.
5. The `ft_transfer` promise fails because `"unregistered.near"` is not registered.
6. The callback fires, finds `args.refund == None`, and returns `None` — no refund is issued.
7. Alice's 1 ETH is permanently frozen at `exit_to_near::ADDRESS`.

The precompile address has no mechanism to release the frozen ETH. The funds are permanently lost. [8](#0-7) [1](#0-0) [9](#0-8)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
```
