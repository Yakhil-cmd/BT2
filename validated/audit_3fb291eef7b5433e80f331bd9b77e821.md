### Title
Missing Cross-Participant Validation of `app_pk` in CKD Protocol Allows Malicious Participant to Corrupt CKD Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function accepts `app_pk` (the ElGamal public key used to encrypt the derived key) as a caller-supplied parameter with no cross-participant consistency check. Any participant can call `ckd()` with an attacker-chosen `app_pk` differing from the one agreed upon by the rest of the group. Because the coordinator aggregates all contributions blindly, the resulting `CKDOutput` is a mixed ciphertext that the legitimate app cannot decrypt, permanently denying the CKD result to honest parties.

---

### Finding Description

The CKD protocol is a single-round scheme. Each participant independently computes a blinded contribution and sends it privately to the coordinator:

```
C_i = S_i + y_i · A        (where A = app_pk, S_i = x_i · H(pk, app_id))
Y_i = y_i · G1
```

The coordinator aggregates all contributions:

```
C = Σ λ_i · C_i = msk · H(pk, app_id) + y_total · A
Y = Σ λ_i · Y_i = y_total · G1
```

The legitimate app decrypts with its secret `a` (where `A = a · G1`):

```
sig = C − a · Y = msk · H(pk, app_id)
```

This correctness relies on every participant using the **same** `app_pk = A`. However, the library never enforces this. In `do_ckd_participant`, the participant computes its share using whatever `app_pk` was passed to `ckd()` and immediately sends it to the coordinator: [1](#0-0) 

In `compute_signature_share`, `app_pk` is used directly without any broadcast or cross-check: [2](#0-1) 

The coordinator aggregates received `CKDOutput` values without any mechanism to verify that each participant used the correct `app_pk`: [3](#0-2) 

There is no protocol round where participants broadcast `app_pk` and verify mutual consistency before computing their shares. The public entry point `ckd()` performs no such check either: [4](#0-3) 

This is directly analogous to the WPunkGateway bug: just as `supplyPunk` accepted an arbitrary `onBehalfOf` without verifying the caller owned the punk, `ckd()` accepts an arbitrary `app_pk` without verifying it matches the value agreed upon by all other participants.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

If a malicious participant `m` calls `ckd()` with `app_pk = A′ ≠ A`, their contribution becomes:

```
λ_m · C_m = λ_m · (S_m + y_m · A′)
```

The coordinator aggregates this with honest contributions (encrypted under `A`):

```
C_total = msk · H(pk, app_id) + y_honest · A + λ_m · y_m · A′
```

The legitimate app attempts decryption:

```
C_total − a · Y_total = msk · H(pk, app_id) + λ_m · y_m · (A′ − a·G1)
```

Since `A′ ≠ a·G1`, the residual term is non-zero. The decryption fails. The CKD protocol permanently fails for this request. The malicious participant cannot extract the derived key from this corruption (they do not know `y_honest`), but they can reliably deny the CKD result to the legitimate app.

---

### Likelihood Explanation

**Medium.** Any participant in the protocol can call `ckd()` with an arbitrary `app_pk`. No privilege beyond being a member of the participant list is required. In the intended MPC network deployment, each node independently receives the CKD request parameters; a single compromised or malicious node can silently substitute `app_pk` without any other node detecting it, because the coordinator performs no consistency check on received shares. [5](#0-4) 

The security model states that secrecy does not depend on all nodes being honest, but correctness does. A single malicious participant is sufficient to break correctness.

---

### Recommendation

Add a protocol round before share computation where each participant broadcasts their `app_pk` and verifies that all received values are identical. Alternatively, include `app_pk` in the session-binding hash (analogous to how `session_id` is used in DKG) so that any inconsistency causes the protocol to abort:

```rust
// In do_ckd_participant / do_ckd_coordinator, before compute_signature_share:
let wait_pk = chan.next_waitpoint();
chan.send_many(wait_pk, &app_pk)?;
for (from, their_app_pk) in recv_from_others::<PublicKey>(&chan, wait_pk, &participants, me).await? {
    if their_app_pk != app_pk {
        return Err(ProtocolError::AssertionFailed(
            format!("app_pk mismatch from participant {:?}", from)
        ));
    }
}
```

### Proof of Concept

1. Three participants `P1`, `P2`, `P3` run CKD with `coordinator = P1` and legitimate `app_pk = A`.
2. Malicious `P3` calls `ckd(..., app_pk = A′)` where `A′ ≠ A`.
3. `P1` and `P2` compute shares encrypted under `A`; `P3` computes its share encrypted under `A′`.
4. `P1` (coordinator) aggregates all three contributions without detecting the mismatch:
   ```
   C_total = λ1·C1 + λ2·C2 + λ3·C3
           = msk·H(pk,app_id) + (λ1·y1 + λ2·y2)·A + λ3·y3·A′
   ```
5. The legitimate app receives `(Y_total, C_total)` and computes `C_total − a·Y_total`:
   ```
   = msk·H(pk,app_id) + λ3·y3·(A′ − a·G1)  ≠  msk·H(pk,app_id)
   ```
6. Decryption fails. The CKD output is permanently unusable. The attack requires no special privilege beyond being a listed participant. [2](#0-1) [6](#0-5)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
