### Title
`PresignOutput` Not Consumed on Rerandomization Enables Presignature Reuse and Secret Key Extraction вЂ” (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both `ot_based_ecdsa::RerandomizedPresignOutput::rerandomize_presign` and `robust_ecdsa::RerandomizedPresignOutput::rerandomize_presign` accept the `PresignOutput` as a shared reference (`&PresignOutput`) rather than consuming it by value. Because `PresignOutput` also derives `Clone`, the library provides no type-level enforcement of the one-time-use invariant that the protocol requires. A malicious coordinator can drive two signing sessions against the same presignature, producing signatures with multiplicatively related nonces, and recover the aggregate secret key via a standard ECDSA nonce-reuse attack.

---

### Finding Description

**Root cause вЂ” shared reference instead of consuming ownership**

In `src/ecdsa/ot_based_ecdsa/mod.rs`:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // в†ђ shared reference; PresignOutput is NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) 

The identical pattern appears in `src/ecdsa/robust_ecdsa/mod.rs`:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // в†ђ shared reference
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [2](#0-1) 

`PresignOutput` derives `Clone` in both variants: [3](#0-2) [4](#0-3) 

Because the function takes a reference and the type is `Clone`, nothing in the library prevents a caller from invoking `rerandomize_presign` twice on the same `PresignOutput` with two different `RerandomizationArguments` (different `msg_hash`, `tweak`, or `entropy`).

**What rerandomization produces**

For OT-based ECDSA, rerandomization computes:

```
Оґ  = HKDF(X, Оµ, h, R, ПЃ)
k' = k В· ОґвЃ»В№
``` [5](#0-4) 

If the same `PresignOutput` (same underlying `k`) is rerandomized twice with different arguments producing Оґв‚Ѓ and Оґв‚‚, the two signing sessions use nonces `kВ·Оґв‚ЃвЃ»В№` and `kВ·Оґв‚‚вЃ»В№`. The ratio `Оґв‚Ѓ/Оґв‚‚` is publicly computable from the two sets of `RerandomizationArguments`, so the nonces are multiplicatively related by a known constant вЂ” a textbook ECDSA nonce-reuse scenario.

**Exploit path (malicious coordinator)**

1. The coordinator runs the presigning protocol with `Nв‚Ѓ` participants, each obtaining their `PresignOutput` share.
2. The coordinator initiates **two** signing sessions for different messages `hв‚Ѓ` and `hв‚‚`, supplying each participant with two distinct `RerandomizationArguments` objects (different `msg_hash` or `entropy`).
3. Each honest participant calls `rerandomize_presign(&presign_out, &args1)` and `rerandomize_presign(&presign_out, &args2)` вЂ” both succeed because the API accepts a reference and the `PresignOutput` is still live.
4. Each participant sends their signature share `sбµў` for both sessions to the coordinator.
5. The coordinator aggregates both sets of shares into two valid signatures `(Rв‚Ѓ, sв‚Ѓ)` and `(Rв‚‚, sв‚‚)` where `Rв‚Ѓ = Оґв‚ЃВ·R` and `Rв‚‚ = Оґв‚‚В·R`.
6. Since `Оґв‚Ѓ` and `Оґв‚‚` are known (derived from public inputs), the coordinator solves the two-equation system to recover the aggregate secret key `x`.

The security documentation explicitly confirms this attack path:

> "If different subsets of size at least 2t+1 sign different (h, Оµ) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [6](#0-5) 

The documentation also states the invariant that the library fails to enforce at the API level:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [7](#0-6) 

The README for OT-based ECDSA similarly states "Each output is consumed **exactly once** (one-time use)" вЂ” a guarantee the type signature does not uphold. [8](#0-7) 

---

### Impact Explanation

**Critical вЂ” secret key extraction.**

A malicious coordinator who can initiate two signing sessions against the same presignature recovers the full aggregate secret key. This enables unauthorized creation of valid ECDSA signatures for any message and any derived key, and constitutes complete disclosure of the aggregate secret material. Both the OT-based and Robust ECDSA variants are affected.

---

### Likelihood Explanation

The coordinator role is a documented, reachable participant in every signing session and requires no privileged access beyond what the protocol grants. The attack requires only the ability to present two different `RerandomizationArguments` to participants holding the same `PresignOutput` вЂ” a straightforward capability for any coordinator. No cryptographic assumptions need to be broken. The library's own documentation identifies this as a known attack vector, confirming it is realistic.

---

### Recommendation

Change both `rerandomize_presign` functions to consume the `PresignOutput` by value, enforcing single-use at the type level:

```rust
// Before (both variants):
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// After:
pub fn rerandomize_presign(
    presignature: PresignOutput,   // consumed by value
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Additionally, remove the `Clone` derive from `PresignOutput` in both variants, or at minimum document it as `#[doc(hidden)]` with a strong warning, so that callers cannot trivially circumvent the ownership-based protection by cloning before passing.

This mirrors the fix described in the external report: just as the Solana program needed to deactivate the account *before* closing it to prevent re-entry, the library must consume (invalidate) the `PresignOutput` *at the point of rerandomization* to prevent reuse.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
