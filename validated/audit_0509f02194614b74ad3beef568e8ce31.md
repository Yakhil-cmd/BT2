### Title
Gateway `skip_stateful_validations` Accepts Unsigned Invoke Transactions at Nonce 1 When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the blockifier `__validate__` entry-point call (i.e., signature verification) for any Invoke V3 transaction whose nonce is exactly `1` and whose sender's on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. The guard is intended only for the deploy-account + invoke UX flow, but the mempool check is not restricted to deploy-account transactions: it returns `true` for **any** transaction for that address in the pool or a recently committed block. An unprivileged attacker who observes a victim's deploy-account transaction in the mempool can immediately submit a malicious invoke at nonce 1 with an arbitrary (invalid) signature and a higher fee, bypassing gateway signature validation entirely and having the transaction admitted to the mempool.

### Finding Description

`skip_stateful_validations` is called from `run_pre_validation_checks` after the nonce range check passes:

```
account_nonce (0) ≤ tx_nonce (1) ≤ account_nonce + max_allowed_nonce_gap (200)  → OK
``` [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator::perform_validations` path for Invoke exits immediately after `perform_pre_validation_stage` without ever calling `validate_tx` / `__validate__`: [3](#0-2) [4](#0-3) 

The mempool's `account_tx_in_pool_or_recent_block` is not restricted to deploy-account transactions; it returns `true` for **any** transaction for the address: [5](#0-4) 

The production `max_allowed_nonce_gap` is 200, so future-nonce invokes (nonce ≥ 2) from the legitimate owner are also accepted, and each one causes `account_tx_in_pool_or_recent_block` to return `true`: [6](#0-5) 

Note also that `max_nonce_for_validation_skip` is defined in `StatefulTransactionValidatorConfig` but is **not consulted** inside `skip_stateful_validations` in the gateway path (it is only used in the legacy `PyValidator`): [7](#0-6) 

### Impact Explanation

An attacker can submit an Invoke transaction at nonce 1 with a completely invalid (or empty) signature for any account that currently has a deploy-account or any future-nonce invoke in the mempool. The gateway accepts it without running `__validate__`. The mempool's fee-escalation logic allows the attacker's transaction to replace the legitimate owner's nonce-1 invoke if the attacker offers a higher fee. The malicious transaction is then handed to the batcher, occupies the nonce-1 slot, and when executed in a block `__validate__` fails — the nonce is incremented and a fee is charged from the victim's account balance — while the legitimate owner's original nonce-1 transaction has been evicted.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The deploy-account + invoke simultaneous submission pattern is the documented and actively tested UX flow. Any user performing this flow is immediately vulnerable. The production `max_allowed_nonce_gap = 200` also means any account with a future-nonce invoke in the mempool is vulnerable. The attack requires no privileged access: the attacker only needs to observe the mempool (or simply race any new account deployment) and submit a transaction with a higher fee.

### Recommendation

Restrict the skip to the case where the transaction in the pool is specifically a deploy-account for the same address, rather than any transaction. Alternatively, add a check that the account contract does not yet exist on-chain (class hash at the address is zero) before skipping validation, which is the actual precondition that makes `__validate__` impossible to run. The `max_nonce_for_validation_skip` config field already exists but is unused in this code path; it should be wired into `skip_stateful_validations`.

### Proof of Concept

1. Victim calls the gateway with a deploy-account transaction for address `X` (nonce 0). It passes all checks and enters the mempool.
2. Attacker observes the mempool and sees address `X` has a pending transaction.
3. Attacker submits `RpcInvokeTransactionV3 { sender_address: X, nonce: 1, signature: [0x0], ... }` with a fee higher than the victim's nonce-1 invoke.
4. Gateway stateless checks pass (signature length ≤ 4000, resource bounds non-zero, etc.).
5. `extract_state_nonce_and_run_validations` reads on-chain nonce = 0; nonce range check passes (0 ≤ 1 ≤ 200).
6. `skip_stateful_validations`: nonce == 1, account_nonce == 0, `account_tx_in_pool_or_recent_block(X)` == `true` → returns `true`.
7. `run_validate_entry_point` sets `validate = false`; blockifier exits without calling `__validate__`. Transaction is admitted to the mempool.
8. Fee escalation removes the victim's nonce-1 invoke; attacker's transaction takes its place.
9. Batcher executes: deploy-account succeeds (nonce 0→1); attacker's invoke runs `__validate__` with the bogus signature → fails; nonce advances to 2; fee deducted from `X`'s balance.
10. Victim's original nonce-1 invoke is gone; victim must resubmit at nonce 2.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
