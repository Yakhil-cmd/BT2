### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Yield Loss - (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both the staking contract and the pool contract accept `reward_address = 0` in every entrypoint that sets or updates a reward address. The only guard applied is a check that the address is not a registered token address. Because address zero passes that check, a staker or pool member can register or update to a zero reward address, after which all reward transfers silently succeed but deliver funds to the burn address, permanently destroying the yield.

### Finding Description
The vulnerability class is identical to the reported bridge bug: a validation helper enforces one constraint (not a token address) but omits the critical constraint (not the zero address), so a malformed address passes silently and funds are irrecoverably lost at the point of transfer.

**Affected entrypoints in `src/staking/staking.cairo`:**

`stake()` validates `reward_address` only against the token registry: [1](#0-0) 

`change_reward_address()` applies the same single check: [2](#0-1) 

**Affected entrypoints in `src/pool/pool.cairo`:**

`enter_delegation_pool()` checks only that `reward_address` is not the pool token: [3](#0-2) 

`change_reward_address()` in the pool applies the same single check: [4](#0-3) 

The `ZERO_ADDRESS` variant exists in `GenericError` and is used elsewhere in the codebase, but is never asserted for `reward_address` in any of these four entrypoints: [5](#0-4) 

**Transfer sites that deliver to the stored `reward_address` without re-checking:**

`send_rewards_to_staker` in staking.cairo transfers directly to whatever address is stored: [6](#0-5) 

`claim_rewards` in pool.cairo does the same: [7](#0-6) 

`unstake_action` calls `send_rewards_to_staker` before erasing the staker record, so rewards are irrecoverably sent to address zero and the staker record is then deleted: [8](#0-7) 

### Impact Explanation
Any staker or pool member who sets `reward_address = 0` — whether by mistake or due to a front-end bug — will have all accrued and future yield transferred to address zero. Because the transfer succeeds (Starknet ERC-20 does not revert on transfer to zero), the protocol emits a normal reward-claimed event, the internal accounting is zeroed, and there is no retry or recovery path. For a staker, this also occurs automatically during `unstake_action`, meaning the principal is returned but the yield is permanently destroyed. This maps directly to **Permanent freezing of unclaimed yield** (High severity in the allowed impact scope).

### Likelihood Explanation
`change_reward_address` is a routine maintenance call. A user migrating wallets, a front-end that passes an uninitialized variable, or a scripting error can trivially produce `reward_address = 0`. The call succeeds without any warning. The loss is not discovered until a reward claim or unstake is executed, at which point it is irreversible.

### Recommendation
Add an explicit non-zero assertion in all four entrypoints before storing `reward_address`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `ZERO_ADDRESS` guard already used for other address fields in the codebase and closes the gap consistently.

### Proof of Concept

1. Staker calls `change_reward_address(reward_address: 0)` on the staking contract. The only check is `!does_token_exist(0)`, which passes because address zero is not a registered token. The new `reward_address = 0` is written to storage.
2. Staker calls `unstake_intent()`, waits for the exit window, then calls `unstake_action()`.
3. Inside `unstake_action`, `send_rewards_to_staker` is invoked. It reads `reward_address = 0` and calls `checked_transfer(recipient: 0, amount: accrued_rewards)`. The transfer succeeds; tokens are sent to address zero and are permanently unrecoverable.
4. The staker record is then erased. There is no retry mechanism and no way to recover the yield.

The same sequence applies to a pool member via `pool.change_reward_address(0)` followed by `pool.claim_rewards()`.
