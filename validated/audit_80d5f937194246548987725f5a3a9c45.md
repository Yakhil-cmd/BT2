### Title
`protocolUnpausePool` Sets Pool to Admin-Paused State (1) Instead of Active (0), Leaving Pool Permanently Paused After Protocol Unpause — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). Because `whenNotPaused` reverts on any non-zero pause level, the pool remains fully paused after the protocol owner calls `protocolUnpausePool`. All swaps and liquidity operations continue to revert with `PoolPaused`.

---

### Finding Description

The pause-level semantics are:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`whenNotPaused` enforces `if (pauseLevel != 0) revert PoolPaused()`, so **both level 1 and level 2 are paused states**.

`protocolUnpausePool` is supposed to lift the protocol-level pause and restore the pool to an operational state:

```solidity
// MetricOmmPoolFactory.sol L399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← sets to admin-paused, NOT active
}
``` [1](#0-0) 

It calls `setPause(1)` — the admin-paused state — which is still a paused state. The pool never becomes active.

Compare with `protocolPausePool`, which can pause from **either** state 0 or state 1:

```solidity
// MetricOmmPoolFactory.sol L392-396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
``` [2](#0-1) 

Because the protocol can pause from state 0 (active), the corresponding unpause must restore to state 0. Instead it always lands on state 1, leaving the pool paused.

The `whenNotPaused` guard that blocks all swaps and liquidity operations:

```solidity
// MetricOmmPool.sol L643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [3](#0-2) 

The conditional inversion is the direct analog of the external report: just as `fillIPOrder` took the wrong payment path based on the `fundingVault` condition, `protocolUnpausePool` sets the wrong target state — 1 (still paused) instead of 0 (active).

---

### Impact Explanation

After `protocolUnpausePool` is called on a pool that was active before the protocol pause:

- The pool is left in admin-paused state (level 1).
- Every call to `swap`, `addLiquidity`, and `removeLiquidity` reverts with `PoolPaused`.
- LPs cannot withdraw their principal; traders cannot execute swaps.
- `protocolUnpausePool` cannot be called again (pool is not in state 2), so the protocol owner has no direct path to fix this.
- The only workaround is for the pool admin to call `unpausePool` — an action the admin never intended to take and that users/integrators would not anticipate.

This is broken core pool functionality causing unusable withdraw/swap/liquidity flows, matching the allowed impact gate.

---

### Likelihood Explanation

This triggers every time the protocol owner calls `protocolUnpausePool` on a pool that was in active state (0) before being protocol-paused. The protocol owner is a valid, non-malicious actor performing a routine operational action. No special setup or attacker is required.

---

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // ← restore to active
}
```

If the design intent is to preserve the admin-paused state when the pool was admin-paused before the protocol pause, the factory must record the pre-protocol-pause state and restore it on unpause.

---

### Proof of Concept

```
1. Pool is active: pauseLevel = 0
2. Protocol owner calls protocolPausePool(pool)
   → valid: cur == 0, setPause(2) → pauseLevel = 2
3. Protocol owner calls protocolUnpausePool(pool)
   → valid: cur == 2, setPause(1) → pauseLevel = 1  ← still paused!
4. User calls swap(...) → reverts: PoolPaused (pauseLevel != 0)
5. User calls addLiquidity(...) → reverts: PoolPaused
6. User calls removeLiquidity(...) → reverts: PoolPaused
7. Protocol owner calls protocolUnpausePool(pool) again
   → reverts: InvalidPauseTransition(1, 1) — cur is 1, not 2
8. Only fix: pool admin calls unpausePool(pool)
   → cur == 1, setPause(0) → pauseLevel = 0 — pool finally active
```

The pool is rendered permanently paused from the protocol's perspective after step 3, with no self-service recovery path for the protocol owner. The admin must perform an unexpected workaround — the exact pattern described in the external report.
