### Title
Unbounded Iteration Over All Block Events in `eip6110_events_parser` Without Native Resource Charge - (`basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over every EVM event accumulated across the entire block at block-finalization time. The iteration cost is proportional to the total number of events emitted in the block, yet no native resource (RISC-V cycle) charge is applied for this work. Because EVM events are bounded only by the block gas limit — not by any separate native-resource cap — an attacker can fill a block with cheap LOG-heavy transactions, causing the block-finalization phase to consume native cycles that were never deducted from any transaction's budget. In the proving mode this can push the RISC-V machine past its cycle limit, making the block unprovable.

---

### Finding Description

`eip6110_events_parser` is called unconditionally during block finalization in both the proving path (`post_tx_op_proving.rs`) and the sequencing path (`post_tx_op_sequencing.rs`):

```rust
// post_tx_op_proving.rs / post_tx_op_sequencing.rs
if eip6110_events_parser(&*system, &mut intermediate_hasher)
    .expect("must filter EIP-6110 deposit requests")
```

Inside the function, the full block-level event iterator is consumed:

```rust
for event in system.io.events_iterator() {   // iterates ALL events in the block
    if event.address != &DEPOSIT_CONTRACT_ADDRESS { continue; }
    if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
        let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
            panic!("invalid deposit event structure");   // ← hard panic
        };
    }
}
```

`events_iterator()` delegates to `EventsStorage::events_ref_iter()`, which walks the entire `HistoryList` of events accumulated across all transactions in the block. There is no cap on this list other than the block gas limit.

Each EVM LOG opcode charges the emitting transaction both EVM gas and a native cost:

```rust
let native = R::Native::from_computational(
    EVENT_STORAGE_BASE_NATIVE_COST
        + EVENT_TOPIC_NATIVE_COST * (topics.len() as u64)
        + EVENT_DATA_PER_BYTE_COST * (data.len() as u64),
);
resources.charge(&R::from_ergs_and_native(ergs, native))?;
```

However, the subsequent block-finalization scan over those same events in `eip6110_events_parser` carries **zero** native charge. The block-level native counter (`block_computational_native_used`) is only updated inside the transaction loop; `post_op_io_touching_impl` runs after that loop with no resource guard.

Additionally, for any event whose `address` matches `DEPOSIT_CONTRACT_ADDRESS` and whose first topic matches `DEPOSIT_EVENT_SIGNATURE_HASH` but whose data does not conform to the expected 576-byte SSZ layout, the function calls `panic!`, which aborts the entire proving run.

---

### Impact Explanation

**Proving-mode denial of service / unprovable block.** An attacker submits a block full of transactions that emit the maximum number of LOG events permitted by the block gas limit (≈ 40 000 LOG0 events at 375 gas each in a 15 M gas block). The per-event native cost is paid during execution, so the block passes all per-transaction resource checks. At finalization, `eip6110_events_parser` iterates over all 40 000 events, performing address and topic comparisons for each. This unaccounted native work can push the RISC-V prover past its fixed cycle budget, causing proof generation to fail and the block to become permanently unprovable.

A secondary impact exists if a contract can be deployed at `DEPOSIT_CONTRACT_ADDRESS` (possible if the deposit contract is not pre-deployed in the ZKsync OS genesis state): emitting a single event with the correct topic but malformed data triggers `panic!`, immediately aborting the proving run.

---

### Likelihood Explanation

The Ethereum mode of ZKsync OS is the primary execution target. Any unprivileged user can submit standard EIP-1559 transactions containing contracts that emit LOG events. No special role, leaked key, or governance action is required. The attacker only needs to fill a block with gas-heavy LOG transactions, which is a normal and permissionless operation. The mismatch between per-transaction native charging and block-finalization iteration cost is structural and reproducible.

---

### Recommendation

1. **Charge native resources for the finalization scan.** Before calling `eip6110_events_parser`, deduct a native cost proportional to `system.io.events_iterator().len()` from a block-level finalization resource budget, or include a per-event native overhead in the block's `block_computational_native_used` counter.

2. **Cap the number of events per block.** Introduce an explicit block-level limit on the total number of EVM events (separate from gas), analogous to the existing `block_logs_used` check for L2-to-L1 signals, and enforce it inside the transaction loop via `check_for_block_limits`.

3. **Replace the `panic!` with a graceful error.** `validate_and_write_event_data` already returns `Result<(), ()>`; propagate the error up through `eip6110_events_parser` and handle it as a block-validity failure rather than an abort.
