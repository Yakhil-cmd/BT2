### Title
WalletConnect `chia_farmer.set_payout_instructions` Confirmation Dialog Omits Launcher ID, Enabling Silent Pool Reward Redirection - (File: packages/gui/src/electron/commands/Commands.ts)

### Summary
A WalletConnect-paired dApp can call `chia_farmer.set_payout_instructions` to redirect pool farming rewards (XCH) to an attacker-controlled address. The confirmation dialog shown to the user displays only the new `payout_instructions` value and omits the `launcher_id` (which PlotNFT is being targeted), so the user cannot determine which NFT's payout is being changed before approving.

### Finding Description
In `packages/gui/src/electron/commands/Commands.ts`, the `chia_farmer.set_payout_instructions` command is registered in the WalletConnect command registry with only one displayed parameter:

```typescript
'chia_farmer.set_payout_instructions': {
    title: () => i18n._(/* i18n */ { id: 'Confirm Set Payout Instructions' }),
    message: () => i18n._(/* i18n */ { id: 'Are you sure you want to set payout instructions?' }),
    confirmLabel: () => i18n._(/* i18n */ { id: 'Set' }),
    params: [
      { name: 'payout_instructions', label: () => i18n._(/* i18n */ { id: 'Payout Instructions' }), type: 'string' },
    ],
  },
``` [1](#0-0) 

The underlying farmer RPC `set_payout_instructions` requires both `launcherId` and `payoutInstructions`:

```typescript
async setPayoutInstructions(args: { launcherId: string; payoutInstructions: string }) {
    return this.command<void>('set_payout_instructions', args);
}
``` [2](#0-1) 

The `launcher_id` parameter is not listed in the `params` array of the WalletConnect command definition. A dApp can pass it as an unlisted argument (which flows through to the daemon but is never rendered in the confirmation dialog). The user sees only the new payout address string — not which PlotNFT is being retargeted.

Additionally, `usePayoutAddress.ts` silently falls back to accepting any raw string when bech32m decoding fails, meaning an attacker-supplied arbitrary string (e.g., a different chain address) is accepted without validation:

```typescript
try {
  newPayoutInstructions = fromBech32m(newPayoutAddress);
} catch {
  newPayoutInstructions = newPayoutAddress;  // arbitrary string accepted
}
``` [3](#0-2) 

### Impact Explanation
Pool payout instructions are the mechanism by which a pool operator knows where to send XCH rewards for a given PlotNFT (identified by `launcher_id`). Changing them redirects all future pool payouts for that NFT to the attacker's address. The user approves the action without knowing which NFT is targeted, satisfying the "wrong asset/destination" spoofing criterion. This is a direct, permanent payout change affecting pooled farming rewards in XCH.

### Likelihood Explanation
Requires a WalletConnect-paired dApp (the attacker must have established a WalletConnect session with the victim). Once paired, the dApp can call `chia_farmer.set_payout_instructions` at any time. The victim sees a generic confirmation ("Are you sure you want to set payout instructions?") with only the new address shown — no NFT identity, no current value, no scope. A user who has multiple PlotNFTs or who trusts the dApp for other operations is likely to approve without scrutiny.

### Recommendation
1. Add `launcher_id` to the `params` array of the `chia_farmer.set_payout_instructions` command definition in `Commands.ts` so it is always rendered in the confirmation dialog.
2. In `usePayoutAddress.ts` / `PlotNFTPayoutInstructionsDialog.tsx`, enforce that the submitted value is a valid bech32m XCH address (or explicitly warn the user when a non-address string is accepted).
3. Display the current payout address alongside the proposed new value in the confirmation dialog so the user can detect unexpected changes.

### Proof of Concept
1. Attacker establishes a WalletConnect session with the victim's Chia GUI.
2. Attacker's dApp sends a `chia_farmer.set_payout_instructions` WalletConnect request with:
   - `payout_instructions`: attacker's XCH address (hex-decoded)
   - `launcher_id`: victim's PlotNFT launcher ID (obtainable from public blockchain data)
3. Victim's GUI renders the confirmation dialog showing only "Payout Instructions: `<attacker_address>`" — no NFT identity is shown.
4. Victim clicks "Set", believing this is a routine or expected operation.
5. The farmer daemon updates the payout instructions for that specific PlotNFT.
6. All subsequent pool payouts for that NFT are sent to the attacker's address. [1](#0-0) [4](#0-3) [5](#0-4)
