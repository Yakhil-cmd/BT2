Looking at the key audit pivots systematically across the gateway, mempool, and blockifier paths.

### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions When Any Transaction from the Sender Exists in the Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator skips the `__validate__` entry point (signature verification) for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. The check does not verify that the transaction in the mempool is specifically a `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` for address A can submit an invoke with nonce=1 for A carrying arbitrary calldata and no valid signature, and the gateway will admit it to the mempool without running `__validate__`.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after `validate_by_mempool` but before `run_validate_entry_point`:

```
run_pre_validation_checks
  → validate_state_preconditions   (nonce range, resource bounds)
  → validate_by_mempool            (duplicate hash / fee-escalation only)
  → skip_stateful_validations      ← returns true → __validate__ is skipped
``` [1](#0-0) 

The skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address has **any** transaction in the pool or appears in the mempool's staged/committed state:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." For the `tx_pool` branch this is true for an undeployed account (nonce=0 in chain state), since an invoke for an undeployed account would fail `__validate__` and never reach the pool. However, the check does not distinguish between transaction types — it only checks address presence. An attacker who observes a legitimate `deploy_account` for address A in the pool can immediately submit an invoke with nonce=1 for A with arbitrary calldata and a garbage signature. The gateway's `validate_nonce` accepts nonce=1 when `account_nonce=0` (within the allowed gap):

```rust
let max_allowed_nonce =
    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
    return Err(...);
}
``` [4](#0-3) 

`validate_by_mempool` only checks for duplicate hashes and fee escalation — it does not verify signatures:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

Because `skip_validate=true`, `run_validate_entry_point` is called with `validate: false`, so `validate_tx` returns `Ok(None)` immediately without executing `__validate__`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [6](#0-5) 

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [7](#0-6) 

The transaction is then forwarded to the mempool via `add_tx` without any signature check having been performed.

### Impact Explanation

An unprivileged attacker can admit invoke transactions with invalid or absent signatures into the mempool for any account that has a pending `deploy_account`. Concretely:

1. **Mempool pollution / DoS**: The attacker can flood the mempool with unsigned invokes targeting every observed `deploy_account`, consuming capacity and evicting legitimate transactions.
2. **Front-running the legitimate user's invoke**: If fee escalation is enabled, the attacker can submit a higher-fee unsigned invoke with nonce=1 before the legitimate user does, displacing the legitimate invoke. The attacker's transaction fails at execution time (the blockifier re-runs `__validate__` with `validate=true`), but the legitimate user's transaction is gone from the mempool.
3. **Broken admission invariant**: The invariant "every invoke transaction in the mempool has passed signature validation" is violated.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The mempool is observable (transactions are propagated via P2P). Any node can watch for `deploy_account` transactions and immediately race-submit unsigned invokes. No privileged access is required. The attack window is the time between the `deploy_account` entering the pool and the block being committed.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender address is present in the pool. For example, expose a `deploy_account_in_pool(address)` query on the mempool, or store the transaction type alongside the pool entry and filter on it inside `skip_stateful_validations`. Alternatively, document that the skip is intentionally permissive and add a rate-limit or fee floor on unsigned-skip invokes to bound the DoS surface.

### Proof of Concept

1. Attacker monitors the mempool and observes a `deploy_account` for address `A` (nonce=0) submitted by a legitimate user.
2. Attacker constructs an `RpcInvokeTransactionV3` with `sender_address=A`, `nonce=1`, arbitrary `calldata`, and a garbage `signature` (e.g., a single zero felt — passes the stateless size check).
3. Attacker submits the invoke to the gateway's `add_tx` endpoint.
4. Gateway stateless validator passes (signature size is within limit).
5. `convert_rpc_tx_to_internal` succeeds (invoke conversion does not verify signature).
6. `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(A)` → `Nonce(0)` (A not yet deployed).
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate hash, no fee-escalation conflict → passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` (the legitimate `deploy_account` is in the pool) → returns `true`.
   - `run_validate_entry_point` called with `validate=false` → `__validate__` is **not executed**.
7. `mempool_client.add_tx(...)` → attacker's unsigned invoke is now in the mempool.
8. If fee escalation is enabled and the attacker paid a higher tip, the legitimate user's invoke (if already submitted) is evicted.
9. At execution time the batcher runs `__validate__` with `validate=true`; the attacker's transaction fails. The legitimate user's transaction has been displaced. [8](#0-7) [2](#0-1)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
