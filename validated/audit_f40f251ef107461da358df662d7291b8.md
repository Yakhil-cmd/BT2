Audit Report

## Title
Single Node Operator Can Exhaust Shared Provider-Level Rate-Limit Bucket, Blocking All Sibling Operators — (`rs/registry/canister/src/rate_limits.rs`)

## Summary

`NODE_PROVIDER_MAX_SPIKE` and `NODE_OPERATOR_MAX_SPIKE` are both set to 140. Because every rate-limited registry call atomically consumes one token from both the per-operator bucket and the shared per-provider bucket, a single node operator can legitimately drain the entire provider-level budget in 140 calls, leaving every sibling operator under the same provider unable to perform any rate-limited node management operation for up to 7 days.

## Finding Description

In `rs/registry/canister/src/rate_limits.rs` lines 18–27, both tiers are configured identically:

```
NODE_PROVIDER_MAX_AVG_OPERATIONS_PER_DAY = 20
NODE_PROVIDER_MAX_SPIKE = 20 × 7 = 140
NODE_OPERATOR_MAX_AVG_OPERATIONS_PER_DAY = 20
NODE_OPERATOR_MAX_SPIKE = 20 × 7 = 140
```

`try_reserve_capacity_for_node_operator_operation` (lines 104–128) first reserves `requested_capacity` from the per-operator bucket keyed by `node_operator_key(node_operator_id)`, then reserves the same amount from the per-provider bucket keyed by `node_provider_key(node_provider_id)`. The provider key is shared across every operator registered under that provider. Because the per-operator spike cap equals the per-provider spike cap, operator A can issue 140 calls — each consuming 1 token from A's own bucket and 1 token from the shared provider bucket — until the provider bucket reaches zero. Operator B (same provider, separate operator bucket still at 140) then fails at the provider reservation step with `RateLimiterError::NotEnoughCapacity` on every subsequent attempt. The provider bucket refills at 1 token per ~72 minutes (`ONE_DAY_SECONDS / 20`), so full recovery from a complete drain takes 7 days. All four rate-limited entry points are affected: `do_update_node_domain_directly`, `do_update_node_ipv4_config_directly`, `do_add_node`, and `do_remove_node_directly`.

## Impact Explanation

This is an application/platform-level DoS against the NNS registry canister's node management surface. Sibling node operators under the same provider are completely locked out of adding, removing, and reconfiguring their nodes via direct calls for up to 7 days per attack cycle. The attack is repeatable: once the provider bucket partially refills, the attacker can drain it again. This matches the High ($2,000–$10,000) impact class: "Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

The attacker must be a registered node operator under the same provider as the victim — a realistic condition for large node providers with multiple independent operators. The attack requires only 140 cheap ingress calls to the registry canister, no special tooling, and no victim interaction. `do_update_node_domain_directly` is publicly callable by any registered node operator for their own nodes, making it the lowest-friction vector. The attack is fully repeatable.

## Recommendation

Size `NODE_PROVIDER_MAX_SPIKE` proportionally to the number of operators it covers rather than equal to a single operator's budget. Options include: (a) set `NODE_PROVIDER_MAX_SPIKE = N × NODE_OPERATOR_MAX_SPIKE` where N is the expected maximum operators per provider; (b) enforce a per-operator sub-quota within the provider bucket so no single operator can consume more than `NODE_OPERATOR_MAX_SPIKE` tokens from the shared pool; or (c) decouple the provider bucket's `max_capacity` and refill rate from the operator bucket's parameters entirely, sizing it to reflect aggregate provider throughput.

## Proof of Concept

Mirrors the existing `test_combined_rate_limiting` structure in `rs/registry/canister/src/rate_limits.rs`:

```rust
#[test]
fn test_provider_bucket_exhaustion_blocks_sibling_operator() {
    let now = SystemTime::now();
    let shared_provider = PrincipalId::new_user_test_id(1000);
    let op1 = PrincipalId::new_user_test_id(1);
    let op2 = PrincipalId::new_user_test_id(2);
    let mut registry = invariant_compliant_registry(0);

    // Register both operators under the same provider
    registry.do_add_node_operator(AddNodeOperatorPayload {
        node_operator_principal_id: Some(op1),
        node_provider_principal_id: Some(shared_provider),
        node_allowance: 10, dc_id: "dc1".into(),
        rewardable_nodes: btreemap!{}, ipv6: None, max_rewardable_nodes: None,
    });
    registry.do_add_node_operator(AddNodeOperatorPayload {
        node_operator_principal_id: Some(op2),
        node_provider_principal_id: Some(shared_provider),
        node_allowance: 10, dc_id: "dc2".into(),
        rewardable_nodes: btreemap!{}, ipv6: None, max_rewardable_nodes: None,
    });

    // Operator 1 exhausts the shared provider bucket (140 tokens)
    let available = registry.get_available_node_provider_op_capacity(shared_provider, now); // 140
    let reservation = registry
        .try_reserve_capacity_for_node_operator_operation(now, op1, available)
        .unwrap();
    registry
        .commit_used_capacity_for_node_operator_operation(now, reservation)
        .unwrap();

    // Operator 2's own bucket is untouched (still 140), but provider bucket is 0
    assert_eq!(registry.get_available_node_operator_op_capacity(op2, now), 140);
    assert_eq!(registry.get_available_node_provider_op_capacity(shared_provider, now), 0);

    // Operator 2 is now blocked
    let err = registry
        .try_reserve_capacity_for_node_operator_operation(now, op2, 1)
        .unwrap_err();
    assert_eq!(err, RateLimiterError::NotEnoughCapacity);
}
```
