### Title
Key-0 Revocation Does Not Propagate to Duplicate Key Indices — (`fvm/environment/accounts.go`, `fvm/environment/account_public_key_util.go`, `fvm/environment/account-key-metadata/metadata.go`)

### Summary
Revoking key index 0 only sets the `Revoked` flag inside the `apk_0` register. It does not touch `keyMetadataBytes` in the `AccountStatus` register. Any key index N > 0 that is a cryptographic duplicate of key 0 stores its own independent `revoked` bit in `keyMetadataBytes`. After key 0 is revoked, `GetRuntimeAccountPublicKey(addr, N)` returns `Revoked: false` for the duplicate index, and `TransactionVerifier.getAccountKeysAndAggregateWeights` accepts signatures produced with the same underlying private key via index N.

### Finding Description

**Storage split.** Key 0's revoked status lives inside the encoded bytes of the `apk_0` register. `revokeAccountPublicKey0` reads that register, flips `decodedPublicKey.Revoked = true`, and writes it back — touching nothing else. [1](#0-0) 

Key N's (N > 0) revoked status lives in the RLE-encoded `weightAndRevokedStatusBytes` section of `keyMetadataBytes` inside `AccountStatus`. `AccountStatus.RevokeAccountPublicKey(keyIndex)` calls `accountkeymetadata.SetRevokedStatus(a.keyMetadataBytes, keyIndex)`, which only modifies that section. [2](#0-1) [3](#0-2) 

**The branch in `RevokeAccountPublicKey`.** When `keyIndex == 0`, the function calls `revokeAccountPublicKey0` and returns immediately. When `keyIndex > 0`, it calls `status.RevokeAccountPublicKey(keyIndex)`. The two paths are completely disjoint; neither touches the other's storage. [4](#0-3) 

**Duplicate key metadata.** When key 1 is added as a duplicate of key 0, `AppendDuplicateKeyMetadata` stores key 1's own `revoked=false` and `weight` in `weightAndRevokedStatusBytes`, and records `storedKeyIndex=0` in the mapping section so that the actual public-key bytes are fetched from `apk_0`. [5](#0-4) 

**Read path for key N.** `GetRuntimeAccountPublicKey(addr, 1)` calls `status.AccountPublicKeyMetadata(1)`, which reads `revoked` exclusively from `keyMetadataBytes`. It then fetches the public-key bytes from `apk_0` (because `storedKeyIndex=0`), but the `Revoked` field it returns comes from `keyMetadataBytes`, not from `apk_0`. [6](#0-5) 

After key 0 is revoked: `apk_0.Revoked = true`, `keyMetadataBytes[key1].revoked = false`. `GetRuntimeAccountPublicKey(addr, 1).Revoked` returns `false`.

**Verifier accepts the signature.** `TransactionVerifier.getAccountKeysAndAggregateWeights` calls `GetRuntimeAccountPublicKey` for each signature's key index and rejects only if `accountKey.Revoked == true`. For key index 1 it sees `false`, so the signature passes. [7](#0-6) 

### Impact Explanation
An attacker who holds private key P and has added key index 1 (duplicate of key 0, same P) to the account retains full signing authority after the account owner revokes key 0. Every transaction signed with P and presented under key index 1 is accepted by the verifier. The account owner has no indication that revocation was incomplete.

### Likelihood Explanation
The precondition — attacker holds the private key and adds a duplicate index before the owner revokes — is exactly the sequence that occurs in a key-compromise response scenario. An attacker who has already compromised key P can submit one transaction to add key 1 (duplicate), then wait. When the owner revokes key 0, the attacker's access is silently preserved. No admin or staked-node control is required; the entire sequence goes through normal transaction submission.

### Recommendation
When `RevokeAccountPublicKey(addr, 0)` is called, after updating `apk_0`, iterate over all key indices N > 0 in `keyMetadataBytes` whose `storedKeyIndex == 0` (i.e., duplicates of key 0) and call `SetRevokedStatus` for each such N. Symmetrically, when revoking key N > 0, if its `storedKeyIndex` maps to a stored key that is also referenced by other indices, revoke those indices as well. Alternatively, store the revoked bit per stored-key rather than per logical-key-index, so a single revocation atomically covers all aliases.

### Proof of Concept
```
1. Create account A with key 0 = public key P (weight 1000).
2. Submit tx signed by key 0: account.keys.add(publicKey: P, hashAlgorithm: SHA3_256, weight: 1000)
   → key 1 is now a duplicate of key 0 (storedKeyIndex=0 in keyMetadataBytes, revoked=false).
3. Submit tx signed by key 0: account.keys.revoke(keyIndex: 0)
   → apk_0.Revoked = true; keyMetadataBytes[key1].revoked unchanged (false).
4. Submit tx signed with private key P, proposalKey.keyIndex=1, envelopeSignature.keyIndex=1.
5. TransactionVerifier calls GetRuntimeAccountPublicKey(A, 1) → Revoked=false → accepted.
6. Assert: transaction executes successfully despite key 0 being revoked.
```

func revokeAccountPublicKey0(
	a Accounts,
	address flow.Address,
) error {
	accountPublicKey0RegisterID := flow.AccountPublicKey0RegisterID(address)

	publicKey, err := a.GetValue(accountPublicKey0RegisterID)
	if err != nil {
		return err
	}

	const keyIndex = uint32(0)

	if len(publicKey) == 0 {
		return errors.NewAccountPublicKeyNotFoundError(
			address,
			keyIndex)
	}

	decodedPublicKey, err := flow.DecodeAccountPublicKey(publicKey, keyIndex)
	if err != nil {
		return fmt.Errorf(
			"failed to decode account public key 0: %w",
			err)
	}

	decodedPublicKey.Revoked = true

	encodedPublicKey, err := flow.EncodeAccountPublicKey(decodedPublicKey)
	if err != nil {
		encoded, _ := decodedPublicKey.MarshalJSON()
		return errors.NewValueErrorf(
			string(encoded),
			"failed to encode revoked account public key 0: %w",
			err)
	}

	return a.SetValue(accountPublicKey0RegisterID, encodedPublicKey)

func SetRevokedStatus(b []byte, keyIndex uint32) ([]byte, error) {
	// Key metadata only stores weight and revoked status for keys at index > 0.

	if keyIndex == 0 {
		return nil, errors.NewKeyMetadataUnexpectedKeyIndexError("failed to set revoked status", 0)
	}

	weightAndRevokedStatusBytes, rest, err := parseWeightAndRevokedStatusFromKeyMetadataBytes(b)
	if err != nil {
		return nil, err
	}

	newWeightAndRevokedStatusBytes, err := setRevokedStatus(slices.Clone(weightAndRevokedStatusBytes), keyIndex-1)
	if err != nil {
		return nil, err
	}

	newB := make([]byte, lengthPrefixSize+len(newWeightAndRevokedStatusBytes)+len(rest))
	off := 0

	binary.BigEndian.PutUint32(newB, uint32(len(newWeightAndRevokedStatusBytes)))
	off += 4

	n := copy(newB[off:], newWeightAndRevokedStatusBytes)
	off += n

	copy(newB[off:], rest)
	return newB, nil
}

func (m *KeyMetadataAppender) AppendDuplicateKeyMetadata(
	keyIndex uint32,

	// Append revoked status and weight
	m.weightAndRevokedStatusBytes, err = appendWeightAndRevokedStatus(m.weightAndRevokedStatusBytes, revoked, weight)
	if err != nil {
		return err
	}

	if !m.deduplicated {
		// Set deduplication flag

		m.startIndexForMapping = keyIndex
	}

	m.mappingBytes, err = appendStoredKeyIndexToMappings(m.mappingBytes, duplicateStoredKeyIndex)
	if err != nil {
		return err
	}

	return nil
}

func (a *StatefulAccounts) RevokeAccountPublicKey(
	address flow.Address,
	keyIndex uint32,
) error {
	err := a.accountPublicKeyIndexInRange(address, keyIndex)
	if err != nil {
		return err
	}

	if keyIndex == 0 {
		return revokeAccountPublicKey0(a, address)
	}

	status, err := a.getAccountStatus(address)
	if err != nil {
		return err
	}

	err = status.RevokeAccountPublicKey(keyIndex)
	if err != nil {
		return fmt.Errorf("failed to revoke public key at index %d for %s: %w", keyIndex, address, err)
	}

	return a.setAccountStatusAfterAccountStatusSizeChange(address, status)
}
