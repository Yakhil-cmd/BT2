### Title
`skip_stateful_validations` Bypasses `__validate__` for Nonce=1 Invoke When Any Nonce=0 Tx (Not Only `deploy_account`) Is in the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip `__validate__` only when a `deploy_account` is pending and the account does not yet exist on-chain. However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type in the pool for that address, including a nonce=0 invoke from an already-deployed account. An attacker who observes a legitimate nonce=0 invoke in the mempool for a freshly-deployed account (account_nonce=0 in state) can submit a nonce=1 invoke with a garbage signature and have it admitted to the mempool without `__validate__` running.

### Finding Description

**Step 1 — The skip predicate.**

`skip_stateful_validations` fires when:
- the incoming tx is `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)`
- `account_nonce == Nonce(Felt::ZERO)` (from on-chain state)
- `account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

**Step 2 — What `account_tx_in_pool_or_recent_block` actually checks.**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`tx_pool.contains_account` returns `true` if **any** transaction from that address is in the pool — it does not filter by transaction type. [2](#0-1) 

`MempoolState::contains_account` checks `staged` or `committed` maps, again with no type filter. [3](#0-2) 

**Step 3 — The false assumption in the comment.**

The code comment reads: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is flawed. A freshly-deployed account has `account_nonce=0` in state. A legitimate nonce=0 invoke from that account passes gateway validation (account exists, `__validate__` runs, signature is valid) and enters the pool. At that point `tx_pool.contains_account(A)` is `true` and `account_nonce` is still `0` — the exact same condition as the deploy_account-pending UX case. [4](#0-3) 

**Step 4 — How `skip_validate` propagates to suppress `__validate__`.**

`run_validate_entry_point` sets `execution_flags.validate = !skip_validate`. When `skip_validate=true`, `perform_validations` in the blockifier hits the early-return branch at line 79–81 without ever calling `validate_tx` (i.e., `__validate__`). [5](#0-4) [6](#0-5) 

**Step 5 — `perform_pre_validation_stage` does not check the signature.**

`perform_pre_validation_stage` only calls `handle_nonce`, `check_fee_bounds`, `verify_can_pay_committed_bounds`, and `validate_proof_facts`. No signature verification occurs here. [7](#0-6) 

### Impact Explanation

An attacker can submit a nonce=1 invoke with an arbitrary/garbage signature for any account that:
1. Is freshly deployed (account_nonce=0 in on-chain state), and
2. Has a legitimate nonce=0 invoke already in the mempool.

The garbage-signature transaction is admitted to the mempool without `__validate__` running. This occupies the nonce=1 slot for that account. The legitimate owner's nonce=1 transaction will be rejected as a duplicate nonce or forced into fee escalation. The admitted transaction will eventually fail at sequencer execution time, but the admission decision is already corrupted.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The mempool is publicly observable. Any attacker can monitor for accounts with `account_nonce=0` in state that have a pending nonce=0 invoke. Freshly-deployed accounts are common (every new wallet deployment followed immediately by a first action). No privileged access is required. The attack requires only submitting a single malformed invoke transaction.

### Recommendation

`skip_stateful_validations` must verify that the transaction in the pool for the sender address is specifically a `deploy_account` transaction, not merely that any transaction exists. One correct approach: query the mempool for the transaction type at `(sender_address, nonce=0)` and only skip validation if it is a `DeployAccount`. Alternatively, check whether the account contract code exists at the sender address in state — if it does, the account is already deployed and `__validate__` must always run regardless of mempool state.

### Proof of Concept

1. Deploy account A on-chain. `account_nonce(A) = 0` in state.
2. Submit a valid nonce=0 invoke from A (signed correctly). It passes `__validate__` and enters the mempool. `tx_pool.contains_account(A)` is now `true`.
3. Submit a nonce=1 invoke from A with `signature = [0xdeadbeef]` (garbage).
4. Gateway calls `extract_state_nonce_and_run_validations`:
   - `account_nonce = get_nonce_from_state(A) = 0`
   - `run_pre_validation_checks` → `skip_stateful_validations`:
     - `tx.nonce() == 1` ✓, `account_nonce == 0` ✓
     - `account_tx_in_pool_or_recent_block(A)` → `tx_pool.contains_account(A)` → `true` ✓
     - Returns `true` (skip)
   - `run_validate_entry_point(tx, skip_validate=true)`:
     - `execution_flags.validate = false`
     - `perform_validations` returns `Ok(())` at line 79–81 without calling `validate_tx`
5. The garbage-signature nonce=1 invoke is admitted to the mempool. `__validate__` never ran.

```rust
/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
```

```rust
        &self,
        state: &mut S,
```
