### Title
Incomplete Key Deletion Leaves WalletConnect Pair Records and Bypass Permissions Intact - (File: `packages/core/src/screens/SelectKey/WalletDeleteDialog.tsx`)

### Summary
When a user deletes a wallet key via `WalletDeleteDialog` or "Delete All Keys" in `SelectKey`, the deletion only removes the key from the keyring and clears two localStorage preference keys (`fingerprintSettings`, `sortedWallets`). It does **not** revoke the associated WalletConnect pair records persisted in `dapp-pairs.yaml`, including any "always allow" bypass permissions granted to dApps. If the user re-imports the same mnemonic (producing the same fingerprint), a previously authorized dApp can immediately execute bypassed commands — including `chia_sendTransaction` and `chia_createOfferForIds` — without any user confirmation dialog.

### Finding Description

`WalletDeleteDialog.handleSubmit` calls `deleteKey({ fingerprint })` and then `removeFingerprintPrefs()`: [1](#0-0) 

`removeFingerprintPrefs` only clears `fingerprintSettings` and `sortedWallets` prefs: [2](#0-1) 

Similarly, `handleDeleteAllKeys` in `SelectKey` only calls `deleteAllKeys().unwrap()` with no pair cleanup: [3](#0-2) 

WalletConnect pair records — including the `bypass` list of "always allow" commands — are stored persistently in `dapp-pairs.yaml` via `pairStore.ts`: [4](#0-3) [5](#0-4) 

Neither `deleteKey` nor `deleteAllKeys` calls `window.permissionsAPI.revokePair(topic)` or `removePair(topic)` for any pair associated with the deleted fingerprint. The `revokePair` path only exists in the explicit disconnect flow: [6](#0-5) 

In `dispatchPairRequest`, bypassed commands skip the confirmation dialog entirely and execute directly: [7](#0-6) 

The fingerprint check in `dispatchPairRequest` only blocks commands when the logged-in fingerprint differs from the pair's fingerprint: [8](#0-7) 

So if the user re-imports the same mnemonic (same fingerprint), all checks pass and bypassed commands execute silently.

### Impact Explanation

A dApp that was previously granted "always allow" for commands such as `chia_sendTransaction` or `chia_createOfferForIds` retains that bypass after the user deletes the key. When the user re-imports the same mnemonic and logs in, the dApp can immediately submit spend bundles or create/accept offers without any confirmation dialog. This constitutes unauthorized signing and spend authority over XCH, CAT, NFT, and other assets.

### Likelihood Explanation

The scenario is realistic: a user who suspects a dApp is malicious may delete their key intending to revoke all access, then re-import the same mnemonic to continue using their wallet. The user's mental model — that key deletion is a full revocation — is not met by the implementation. No warning is shown during key deletion that WalletConnect pairs will persist.

### Recommendation

In `WalletDeleteDialog.handleSubmit` and `SelectKey.handleDeleteAllKeys`, after the key is deleted, enumerate all pair records whose `fingerprint` matches the deleted fingerprint(s) and call `window.permissionsAPI.revokePair(topic)` (which calls `removePair` in `pairStore.ts`) for each. This mirrors the existing `disconnectPair` flow used in `WalletConnectProvider` and `SettingsIntegration`.

### Proof of Concept

1. Import a mnemonic (fingerprint `F`). Pair with a dApp via WalletConnect and grant "always allow" for `chia_sendTransaction`.
2. Navigate to the key selection screen. Delete the key for fingerprint `F` via `WalletDeleteDialog`.
3. Inspect `dapp-pairs.yaml` — the pair record for fingerprint `F` with `bypass: [chia_sendTransaction]` is still present.
4. Re-import the same mnemonic (fingerprint `F`). Log in.
5. The dApp sends a `session_request` for `chia_sendTransaction`. `dispatchPairRequest` finds the pair, confirms `loggedInFingerprint === pair.fingerprint === F`, sees the command in `bypass`, and calls `process(context)` — no confirmation dialog appears and the transaction is submitted. async function disconnectPair(client: Client, topic: string) {
  try {
    // revoke pair from main process
    await window.permissionsAPI.revokePair(topic);

    // get all sessions for the pairing
    const sessions = client.session.getAll();
    const relatedSessions = sessions.filter((s) => s.pairingTopic === topic);

    // disconnect all related sessions
    await Promise.all(
      relatedSessions.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect the pairing
    await client.core.pairing.disconnect({ topic });
  } catch (e) {
    processError(e as Error);
  }
}
