### Title
Signature-Bypass DoS via `skip_stateful_validations` Frontrun on Deploy-Account UX Path — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce=1` when the sender account has **any** transaction in the mempool or a recent block. An unprivileged attacker who observes a victim's `deploy_account` entering the mempool can immediately submit a malicious invoke for the victim's address with an arbitrary/invalid signature. Because the skip condition is satisfied, the gateway admits the unsigned transaction without calling `__validate__`. The attacker's transaction occupies the victim's `nonce=1` slot; the victim's legitimate invoke is then rejected as `DuplicateNonce`. When the batcher later executes the attacker's transaction, `__validate__` fails, the nonce is consumed, and the fee is charged to the victim's account.

---

### Finding Description

**Root cause — `skip_stateful_validations`:**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                // ...
        }
    }
    Ok(false)
}
```

The comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This assumption is false. `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction in the pool for that address — including one submitted by the attacker. [1](#0-0) 

**How the skip propagates to bypass signature verification:**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` on the `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate=false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, never calling `validate_tx` (the `__validate__` entry point):

```rust
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

**`account_tx_in_pool_or_recent_block` checks any account presence, not deploy_account specifically:**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

**Attack sequence:**

1. Victim pre-funds address X and submits `deploy_account` (nonce=0). The deploy_account enters the mempool; X is now visible.
2. Attacker submits `Invoke(sender=X, nonce=1, signature=<garbage>)` before the victim submits their own invoke.
3. Gateway evaluates: on-chain nonce of X = 0, tx nonce = 1, `account_tx_in_pool_or_recent_block(X)` = `true` (victim's deploy_account is there) → `skip_validate = true`.
4. `perform_pre_validation_stage` runs: nonce check passes (0 ≤ 1), fee bounds pass, `verify_can_pay_committed_bounds` passes (X has pre-loaded funds). `__validate__` is **never called**.
5. Attacker's invalid invoke is admitted to the mempool at `(X, nonce=1)`.
6. Victim submits their valid invoke (nonce=1). Mempool rejects it: `MempoolError::DuplicateNonce` (fee escalation disabled by default) or requires the victim to outbid the attacker's fee (which is charged to X, not the attacker). [5](#0-4) 

**Mempool duplicate-nonce enforcement (no fee escalation by default):**

```rust
if !self.config.static_config.enable_fee_escalation {
    if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
        return Err(MempoolError::DuplicateNonce { address, nonce });
    };
    return Ok(None);
}
``` [6](#0-5) 

When the batcher later executes the attacker's invoke, `__validate__` fails (invalid signature), the nonce is consumed, and the fee is charged to X's balance.

---

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool must either be a `deploy_account` (which runs `__validate_deploy__` during full execution) or have passed `__validate__` (signature verification). The skip path violates this for any invoke with nonce=1 targeting an account that has any mempool presence.

**Concrete effects:**
- **Admission of invalid transactions:** The gateway accepts an invoke with an arbitrary/invalid signature into the mempool. This directly matches the "High: Mempool/gateway/RPC admission accepts invalid transactions" impact category.
- **DoS on victim's first post-deploy invoke:** The victim's valid invoke is rejected as `DuplicateNonce`. The attacker can repeat this every time the victim retries.
- **Fee drain from victim's account:** When the batcher executes the attacker's transaction, `__validate__` fails, the nonce is consumed, and the fee is charged to the victim's account — not the attacker's. The attacker's cost is zero.

---

### Likelihood Explanation

- The victim's `deploy_account` is publicly visible in the mempool the moment it is submitted.
- The attacker only needs to submit a single transaction before the victim submits their invoke. In the common UX pattern (`deploy_account` + `invoke` submitted together), there is a small but real race window.
- The attack is free to the attacker: the fee is charged to the victim's account, not the attacker's. This makes repeated attacks economically rational.
- No special privileges, no privileged RPC access, and no knowledge of the victim's private key are required.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address in the mempool. The mempool should expose a dedicated `deploy_account_in_pool(address)` query, or the gateway should track the deploy_account tx hash and verify it is still present before granting the skip.

Alternatively, mirror the `PyValidator` approach in `crates/native_blockifier/src/py_validator.rs` which requires the caller to explicitly pass the `deploy_account_tx_hash` as proof, rather than inferring it from any mempool presence:

```rust
// PyValidator requires explicit deploy_account_tx_hash proof
let deploy_account_not_processed =
    deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
``` [7](#0-6)

### Proof of Concept

```
State: Account X has nonce=0 on-chain, X has been pre-funded with STRK.

Step 1: Victim submits deploy_account(sender=X, nonce=0, valid_sig).
        → Mempool: {X: deploy_account(nonce=0)}
        → account_tx_in_pool_or_recent_block(X) = true

Step 2: Attacker submits invoke(sender=X, nonce=1, signature=0xDEADBEEF).
        Gateway check:
          - account_nonce(X) = 0  ✓
          - tx.nonce() = 1        ✓
          - account_tx_in_pool_or_recent_block(X) = true  ✓
          → skip_validate = true
          → validate flag = false
          → __validate__ NOT called
          → perform_pre_validation_stage passes (nonce 0≤1, fee bounds OK, balance OK)
        → Mempool: {X: deploy_account(nonce=0), attacker_invoke(nonce=1)}

Step 3: Victim submits invoke(sender=X, nonce=1, valid_sig).
        Mempool: DuplicateNonce(X, 1) → REJECTED.

Step 4: Batcher executes:
          deploy_account(X, nonce=0) → success, X deployed, nonce→1
          attacker_invoke(X, nonce=1) → __validate__ fails (bad sig)
            → nonce consumed (→2), fee charged to X.

Result: Victim's first invoke is permanently blocked until they outbid the
        attacker's fee. Attacker pays nothing. Attack is repeatable.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
