### Title
Attacker-Controlled `puzzleHash` in Blockchain Notification Redirects Counter-Offer to Attacker's Address — (`packages/gui/src/components/notification/NotificationOffer.tsx`)

### Summary

An unprivileged blockchain attacker can send a crafted on-chain notification containing a valid `offerURL` and an arbitrary `puzzleHash` (their own). The GUI trusts the `puzzleHash` field verbatim as the counter-offer return address, with no cryptographic or identity binding to the notification sender. When the victim clicks "Counter", their counter-offer is sent to the attacker's address instead of the legitimate offer creator's.

---

### Finding Description

The Chia notification protocol embeds a `ph` (puzzle hash) field in the notification payload, documented as "puzzlehash of the notification sender, for sending a response (counter offer)": [1](#0-0) 

This field is parsed and stored as `notification.puzzleHash` on `NotificationCounterOffer` objects: [2](#0-1) 

In `NotificationOffer.handleClick()`, the `puzzleHash` is passed directly as the `address` state to the offer viewer route, with no validation that it matches the notification sender: [3](#0-2) 

The only guard applied by `useValidNotifications` is checking `offerState.offer?.valid` — it validates the offer file itself, not the `puzzleHash`: [4](#0-3) 

When the user clicks "Counter" in `OfferBuilderViewer`, the unvalidated `address` (attacker's puzzle hash) is forwarded to the offer builder: [5](#0-4) 

Similarly in `OfferIncomingTable.handleCounterOffer()`, the `puzzleHash` is converted to a bech32m address and used as the counter-offer destination without any sender-binding check: [6](#0-5) 

---

### Impact Explanation

The victim's counter-offer notification is sent to the attacker's address. The attacker receives the counter-offer file and can accept it, executing the trade. The victim believes they are negotiating with the legitimate offer creator but is actually trading with the attacker. This fits the High impact category: *"unsafe trust of notification state that causes a user to send to the wrong destination."*

---

### Likelihood Explanation

Sending a blockchain notification to any address is a standard, permissionless Chia network operation. The attacker only needs to:
1. Obtain or create any valid offer URL (e.g., from a public marketplace like Dexie).
2. Send a `send_notification` RPC to the victim's address with `ph` set to their own puzzle hash.

No privileged access, leaked keys, or social engineering beyond the notification itself is required.

---

### Recommendation

Bind the `puzzleHash` to the notification sender's coin. The on-chain notification coin's puzzle hash (the coin that funded the notification) should be compared against the `ph` field in the payload. If they do not match, the `ph` field must be discarded and `canCounterOffer` set to `false`. At minimum, display the raw sender address to the user alongside the counter-offer destination so they can verify the two match before proceeding.

---

### Proof of Concept

1. Attacker calls `send_notification` RPC targeting victim's address, with payload:
   ```json
   {"v":1,"t":1,"d":{"u":"https://dexie.space/offers/<valid_offer_id>","ph":"<attacker_puzzle_hash>"}}
   ```
2. Victim opens the GUI; the notification appears as a valid offer (offer passes `checkOfferValidity`).
3. Victim clicks the notification → `handleClick()` navigates to `/dashboard/offers/view` with `address = <attacker_puzzle_hash>`.
4. Victim clicks "Counter" → `handleCounterOffer()` navigates to `/dashboard/offers/builder` with `address = <attacker_puzzle_hash>`.
5. Victim completes and submits the counter-offer → a notification is sent to the attacker's address.
6. Attacker accepts the counter-offer, executing the trade on their terms.

**Test assertion**: Mount `NotificationOffer` with `notification.puzzleHash = '<attacker_hash>'` and a valid offer, click the wrapper, assert `navigate` is called with `state.address === '<attacker_hash>'` — confirming the unvalidated attacker hash flows through unchecked.

function filterNotifications(notifications: Notification[], getOffer: (offerId: string | undefined) => OfferState) {
  return notifications.filter((notification) => {
    if ([NotificationType.OFFER, NotificationType.COUNTER_OFFER].includes(notification.type)) {
      const offerId = getOfferId(notification);
      const offerState = getOffer(offerId);
      return !!offerState.offer?.valid;
    }

    return true;
  });
}

  async function handleCounterOffer(notification: Notification) {
    try {
      const puzzleHash = 'puzzleHash' in notification ? notification.puzzleHash : undefined;
      const offerId =
        'offerURL' in notification
          ? notification.offerURL
          : 'offerData' in notification
            ? notification.offerData
            : undefined;
      const offerState = getOffer(offerId);

      if (!offerState || !puzzleHash || !currencyCode) {
        return;
      }

      const address = currencyCode && puzzleHash ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';
      const offerSummary = offerState.offer?.summary;

      if (!offerSummary || isDataLayerOfferSummary(offerSummary)) {
        return;
      }

      const offer = offerToOfferBuilderData(offerSummary);

      navigate('/dashboard/offers/builder', {
        state: {
          referrerPath: location.pathname,
          isCounterOffer: true,
          address,
          offer,
        },
      });
