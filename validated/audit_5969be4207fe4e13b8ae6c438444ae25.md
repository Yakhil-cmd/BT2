### Title
Missing Zero-Address Validation in `change_reward_address()` Allows Permanent Freezing of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `Staking::change_reward_address()` and `Pool::change_reward_address()` accept a zero address as `reward_address` without any validation. Because reward transfers are sent directly to the stored `reward_address`, setting it to zero causes all future unclaimed yield to be transferred to the zero address (permanently burned/frozen). The protocol's own error catalogue defines a `ZERO_ADDRESS` error, confirming this check was intended but never implemented.

---

### Finding Description

`Staking::change_reward_address()` performs only one guard — it rejects token contract addresses — but does not reject the zero address: [1](#0-0) 

`Pool::change_reward_address()` mirrors the same pattern, checking only that `reward_address` is not the pool's token contract: [2](#0-1) 

`stake()` has the same omission at registration time — it checks `!does_token_exist(reward_address)` but never asserts `reward_address.is_non_zero()`: [3](#0-2) 

When rewards are later disbursed, `send_rewards_to_staker()` reads the stored `reward_address` and transfers directly to it with no further validation: [4](#0-3) 

`Pool::claim_rewards()` does the same for pool members: [5](#0-4) 

The protocol's own error catalogue explicitly defines `ZERO_ADDRESS = "Zero address"`, confirming the intent to guard against this case: [6](#0-5) 

---

### Impact Explanation

A staker who calls `change_reward_address(0)` (or stakes with `reward_address = 0`) will have every subsequent reward transfer sent to the zero address. Because `checked_transfer` to address zero succeeds on standard ERC20 implementations (it is a burn), the tokens are permanently destroyed. All accumulated unclaimed STRK yield for that staker — including rewards accrued across multiple epochs — is irreversibly lost. The same applies to pool members via `Pool::change_reward_address(0)`.

**Impact class: High — Permanent freezing (burning) of unclaimed yield.**

---

### Likelihood Explanation

Any registered staker or pool member can call `change_reward_address` at any time with no prerequisites beyond being registered. The call requires no privileged role. A user who accidentally passes a zero address (e.g., from an uninitialized variable in a front-end or script, directly analogous to the external report's uninitialized `address` state) will silently and irreversibly redirect all future rewards to the zero address. There is no recovery path once the address is set and rewards are claimed.

---

### Recommendation

Add a non-zero assertion to both `change_reward_address` functions and to `stake()`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing `ZERO_ADDRESS` error already defined in the protocol's error catalogue and is consistent with the pattern used elsewhere (e.g., `assert!(public_key.is_non_zero(), ...)` in `set_public_key`). [7](#0-6) 

---

### Proof of Concept

1. Staker registers via `stake(reward_address: 0, operational_address: <valid>, amount: min_stake)`. No assertion fires because `does_token_exist(0)` returns false and `0 >= min_stake` is false only if amount is too low — the zero-address check is simply absent.
2. Epochs pass; the attestation contract calls `update_rewards_from_attestation_contract`, accumulating STRK rewards in `staker_info.unclaimed_rewards_own`.
3. Anyone calls `claim_rewards(staker_address)`. `send_rewards_to_staker` reads `reward_address = 0` and executes `token_dispatcher.checked_transfer(recipient: 0, amount: rewards)`. The transfer succeeds; tokens are burned.
4. Alternatively, an already-registered staker calls `change_reward_address(reward_address: 0)`. From that point forward, every `claim_rewards` call burns the staker's yield.
5. The same sequence applies to pool members via `Pool::change_reward_address(0)` followed by `Pool::claim_rewards`.
