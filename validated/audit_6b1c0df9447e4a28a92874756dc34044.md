### Title
Gateway Stateless Validator Omits Blocked-Sender-Address Check, Allowing Any Address to Transact — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The `StatelessTransactionValidator::validate` function contains an explicit, unimplemented TODO to check whether the sender address is blocked. No such check is ever performed for `Invoke` or `DeployAccount` transactions, so any address — including one that the operator intends to block — is unconditionally admitted through the gateway's stateless path.

### Finding Description

`StatelessTransactionValidator::validate` is the first gate every incoming transaction must pass before conversion, stateful validation, and mempool admission. The function body opens with:

```rust
// TODO(Arni, 1/5/2024): Add a mechanism that validate the sender address is not blocked.
``` [1](#0-0) 

No blocked-address check is ever added. The function proceeds directly to structural checks (contract address validity, empty fields, resource bounds, sizes, DA modes, proof consistency) and returns `Ok(())` for any syntactically valid transaction regardless of the sender's blocked status.

By contrast, the gateway **does** enforce an address-level admission control for `Declare` transactions: `check_declare_permissions` calls `is_authorized_declarer` and rejects unauthorized senders before stateless validation even runs. [2](#0-1) 

This check is applied only inside the `if let RpcTransaction::Declare` branch: [3](#0-2) 

`Invoke` and `DeployAccount` transactions skip `check_declare_permissions` entirely and reach `stateless_tx_validator.validate` with no address-level gate. The stateless validator then passes them unconditionally because the TODO check was never implemented.

The prover subsystem has a separate `BlockingCheckClient` that calls an external `starknet_checkTransaction` service to decide whether a transaction should be blocked, but this runs only inside the prover — after the transaction has already been admitted through the gateway and mempool. [4](#0-3) 

The gateway therefore has no mechanism to enforce operator-level address blocking for the two most common transaction types.

### Impact Explanation

A sender address that the operator intends to block can submit `Invoke` and `DeployAccount` transactions through the HTTP gateway. Those transactions pass stateless validation, proceed to stateful validation (signature check via the blockifier's `validate` entry point), and — if they pass — are inserted into the mempool and eventually sequenced. The gateway's admission decision is wrong: it accepts transactions that should be rejected. This matches the **High** impact tier: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The trigger requires no privilege. Any user who knows (or guesses) that their address is on an operator blocklist can simply submit an `Invoke` transaction. The stateless validator performs no address lookup and returns `Ok(())`. The gap has existed since at least 2024-01-05 (the TODO date) and affects every production deployment of the gateway.

### Recommendation

Implement the missing check inside `StatelessTransactionValidator::validate`. The gateway config already carries an authorized-declarer list; a parallel blocked-sender list (or reuse of the same config structure) should be consulted for all transaction types:

```rust
pub fn validate(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    // Enforce blocked-address list for all transaction types.
    if let Some(sender) = tx.sender_address() {
        if self.config.is_blocked_sender(&sender) {
            return Err(StatelessTransactionValidatorError::BlockedSenderAddress { sender });
        }
    }
    // ... existing checks ...
}
```

The `check_declare_permissions` pattern in `gateway.rs` (lines 407–433) already demonstrates the correct place and style for this kind of address-level gate.

### Proof of Concept

1. Operator configures a blocklist containing address `0xDEAD`.
2. Attacker submits `RpcTransaction::Invoke { sender_address: 0xDEAD, nonce: 0, ... }` to the HTTP gateway.
3. `add_tx_inner` skips `check_declare_permissions` (only runs for `Declare`).
4. `stateless_tx_validator.validate` runs; the TODO check is absent; all structural checks pass; returns `Ok(())`.
5. Transaction proceeds to conversion, stateful validation, and mempool insertion.
6. The blocked address has successfully submitted a transaction — identical to M-1 where a blacklisted ERC-20 holder could still call `transfer`. [1](#0-0) [5](#0-4)

    fn check_declare_permissions(
        &self,
        declare_tx: &RpcDeclareTransaction,
    ) -> Result<(), StarknetError> {
        // TODO(noamsp): Return same error as in Python gateway.
        if self.config.static_config.block_declare {
            return Err(StarknetError {
                code: StarknetErrorCode::UnknownErrorCode(
                    "StarknetErrorCode.BLOCKED_TRANSACTION_TYPE".to_string(),
                ),
                message: "Transaction type is temporarily blocked.".to_string(),
            });
        }
        let RpcDeclareTransaction::V3(declare_v3_tx) = declare_tx;

                code: StarknetErrorCode::KnownErrorCode(
                    KnownStarknetErrorCode::UnauthorizedDeclare,
                ),
                message: format!(
                    "Account address {} is not allowed to declare contracts.",
                    &declare_v3_tx.sender_address
