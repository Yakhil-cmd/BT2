### Title
Missing Access Control on `update_rewards` Allows Anyone to Consume Per-Block Reward Slot with `disable_rewards: true`, Permanently Freezing Staker Yield - (File: src/staking/staking.cairo)

---

### Summary

`update_rewards` in `StakingRewardsManagerImpl` is intended to be callable only by the Starkware sequencer (per spec), but the implementation enforces no such restriction. Because `last_reward_block` is written unconditionally before the `disable_rewards` branch, any unprivileged caller can invoke `update_rewards(staker_address, disable_rewards: true)` to consume the single per-block reward slot without distributing any yield. A subsequent legitimate call with `disable_rewards: false` in the same block reverts with `REWARDS_ALREADY_UPDATED`. Repeated every block, this permanently freezes staker and delegator unclaimed yield.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` (lines 1449–1507) is the V3 consensus-rewards distribution entry point. Its guard is:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

Immediately after passing this guard, the function writes the current block number to storage **before** branching on `disable_rewards`:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // line 1485

if disable_rewards || self.is_pre_consensus() {
    return;                                            // line 1487-1489
}
```

When `disable_rewards: true`, the function returns here — no rewards are calculated or credited — but `last_reward_block` has already been advanced to the current block. Any subsequent call in the same block (including the sequencer's intended `disable_rewards: false` call) hits the `REWARDS_ALREADY_UPDATED` revert.

The protocol specification at `docs/spec.md` line 1644–1645 states:

> **access control**: Only starkware sequencer.

However, the actual implementation at line 1452 only calls `self.general_prerequisites()`, which checks the pause state — there is no caller identity check. The function is therefore callable by any address.

This is the direct analog of the TempleGold bug: a public function that updates the shared "last distribution" state variable without performing the actual distribution, allowing an attacker to front-run the legitimate distributor and block reward delivery.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once consensus rewards are active (`!is_pre_consensus()`), every block that the sequencer fails to call `update_rewards(disable_rewards: false)` is a block of yield permanently lost to all stakers and delegators. An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` at the start of every block ensures the sequencer's call always reverts with `REWARDS_ALREADY_UPDATED`. Stakers and delegators accumulate zero `unclaimed_rewards_own` indefinitely. The attack requires no capital and no privileged access.

---

### Likelihood Explanation

**High.** The function is publicly callable with no access control. Any address can submit a transaction each block targeting any active staker. The cost is only gas. The attacker has no requirement to hold stake, be a delegator, or control any privileged role. The only mitigation would be the sequencer ordering its own transaction first, but this is an operational assumption with no on-chain enforcement.

---

### Recommendation

Enforce the access control stated in the specification. Add a sequencer-identity check at the top of `update_rewards`, analogous to how `update_rewards_from_attestation_contract` asserts `assert_caller_is_attestation_contract()`. Alternatively, separate the `last_reward_block` write so it only occurs when `disable_rewards` is `false` and rewards are actually distributed — mirroring the TempleGold fix of introducing a separate variable updated only in the real distribution path.

---

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. At the start of block N, attacker calls:
   ```
   update_rewards(staker_address: any_active_staker, disable_rewards: true)
   ```
3. `last_reward_block` is written to N; function returns without distributing rewards.
4. Sequencer calls:
   ```
   update_rewards(staker_address: any_active_staker, disable_rewards: false)
   ```
5. Guard `current_block_number > self.last_reward_block.read()` → `N > N` → **false** → reverts `REWARDS_ALREADY_UPDATED`.
6. Staker's `unclaimed_rewards_own` is unchanged. Pool rewards are not transferred.
7. Repeat every block → permanent yield freeze for all stakers.
