Audit Report

## Title
Logo-Omission Guard Permanently Disabled in `convert_proposal` When `return_self_describing_action=true` — (`rs/nns/governance/src/pb/proposal_conversions.rs`)

## Summary
In `convert_proposal`, `is_create_service_nervous_system_proposal` is derived from the already-conditionally-suppressed `action` field, which is always `None` when `return_self_describing_action=true`. This causes the logo-omission guard passed to `convert_self_describing_action` to always evaluate to `false`, bypassing the `omit_large_fields` protection and returning full logo bytes in `self_describing_action` regardless of the caller's `omit_large_fields=true` request. Any unprivileged caller can trigger this via the public `list_proposals` query, potentially causing oversized responses that exceed the IC's 2 MB limit and trap the query.

## Finding Description
`ProposalDisplayOptions::for_list_proposals` sets `show_action: !return_self_describing_action` at line 24. When `return_self_describing_action=true`, `show_action=false`.

In `convert_proposal` (lines 495–499), `action` is set to `None` when `show_action()` is false:
```rust
let action = if display_options.show_action() {
    action.as_ref().map(|x| convert_action(x, display_options))
} else {
    None
};
```

Immediately after (lines 500–502), `is_create_service_nervous_system_proposal` is derived from this already-`None` value:
```rust
let is_create_service_nervous_system_proposal = action.as_ref().is_some_and(|action| {
    matches!(action, api::proposal::Action::CreateServiceNervousSystem(_))
});
```

Because `action` is always `None` in this branch, `is_create_service_nervous_system_proposal` is always `false`. The guard passed to `convert_self_describing_action` (lines 509–510) therefore evaluates to `false && true = false`:
```rust
convert_self_describing_action(
    self_describing_action,
    is_create_service_nervous_system_proposal  // always false
        && display_options.omit_create_service_nervous_system_large_fields(),
)
```

Inside `convert_self_describing_action` (lines 317–328), when `omit_create_service_nervous_system_logos=false`, `paths_to_omit` is an empty `vec![]`, so `logo` and `token_logo` fields pass through unfiltered. The method `omit_create_service_nervous_system_large_fields()` (lines 59–61) correctly returns `true` when `omit_large_fields_requested=true` and `multi_query=true`, but it is never reached due to the short-circuit. The existing unit test `test_self_describing_value_omit_logos` (lines 745–829) only tests `convert_self_describing_action` in isolation with a hardcoded boolean and does not exercise the `convert_proposal` integration path.

## Impact Explanation
This matches **High ($2,000–$10,000): Application/platform-level DoS**. The NNS governance canister's `list_proposals` query is a public, unauthenticated call. A `CreateServiceNervousSystem` proposal logo can be up to ~256 KiB (base64). With multiple such proposals in a single `list_proposals` response window, the combined payload can exceed the IC's 2 MB query response limit, causing response serialization to fail and the query to trap. This disrupts availability of the NNS governance query interface for all users.

## Likelihood Explanation
`list_proposals` is fully unauthenticated — any caller can set `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)` simultaneously. The NNS mainnet has had multiple real `CreateServiceNervousSystem` proposals, each potentially carrying logo blobs. No special privileges, victim interaction, or external compromise is required. The attack is repeatable and deterministic.

## Recommendation
Derive `is_create_service_nervous_system_proposal` from the original `pb::Proposal` action field, not from the already-conditionally-suppressed converted `action`. In `convert_proposal`, replace lines 500–502 with:

```rust
let is_create_service_nervous_system_proposal = item.action.as_ref().is_some_and(|action| {
    matches!(action, pb::proposal::Action::CreateServiceNervousSystem(_))
});
```

This decouples type-detection from display-suppression logic. Additionally, add an integration-level test for `convert_proposal` that exercises the `show_action=false, show_self_describing_action=true, omit_large_fields=true` path and asserts that logo fields are `Null` in the returned `self_describing_action`.

## Proof of Concept
1. Submit a `CreateServiceNervousSystem` proposal with a 256 KiB base64-encoded logo to NNS governance.
2. Call `list_proposals` with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`.
3. Observe that the returned `self_describing_action.value` map contains the full logo bytes under the `"logo"` key (not `Null`), confirming the guard is bypassed.
4. For the DoS: accumulate enough such proposals in the response window so the total response size exceeds 2 MB; the query traps.

A deterministic unit test can reproduce step 3 by calling `convert_proposal` directly with a `pb::Proposal` containing a `CreateServiceNervousSystem` action with a large logo, using `ProposalDisplayOptions::for_list_proposals(true, true)`, and asserting that `self_describing_action.value` has `logo = Null`.
