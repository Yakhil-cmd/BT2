### Title
Non-Coordinator Participants Consume One-Time Presignature Material Without Confirmation of Signing Completion - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

In both the OT-based and Robust ECDSA signing protocols, non-coordinator participants compute their signature share, send it to the coordinator, and immediately return `Ok(None)` — consuming their one-time-use presignature material — without ever receiving confirmation that the coordinator successfully assembled and verified the final signature. A malicious coordinator can exploit this asymmetry to exhaust all presignatures held by honest participants without producing any valid signature, permanently denying future signing capability until new presignatures are generated.

---

### Finding Description

The signing protocol is split into two asymmetric paths: the coordinator assembles the final signature and returns `Ok(Some(sig))`; all other participants return `Ok(None)`. The non-coordinator path in both implementations is:

**OT-based ECDSA** (`src/ecdsa/ot_based_ecdsa/sign.rs`, `do_sign_participant`):

```rust
fn do_sign_participant(
    mut chan: SharedChannel,
    participants: &ParticipantList,
    coordinator: Participant,
    me: Participant,
    presignature: &RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<SignatureOption, ProtocolError> {
    let s_i = compute_signature_share(participants, me, presignature, msg_hash)?;
    let wait0 = chan.next_waitpoint();
    chan.send_private(wait0, coordinator, &s_i)?;
    Ok(None)   // ← returns immediately, no confirmation
}

The presignature is passed **by value** into `sign()` and is therefore permanently consumed at the call site, regardless of whether the coordinator ever produces a valid signature: [3](#0-2) [4](#0-3) 

The codebase explicitly documents that presignatures are one-time-use:

> "it's crucial that a presignature is never reused" [5](#0-4) [6](#0-5) 

The coordinator, by contrast, performs full verification before returning: [7](#0-6) 

Non-coordinator participants have no equivalent verification step and no feedback path from the coordinator confirming the signature was assembled, verified, and distributed.

---

### Impact Explanation

This is a **High: Permanent denial of signing for honest parties**. A malicious coordinator can:

1. Initiate a signing session with honest participants.
2. Collect all signature shares — causing each participant to consume their presignature.
3. Abort without assembling or distributing the final signature.
4. Repeat across all available presignatures.

After this attack, honest participants have exhausted their presignature pool. They cannot sign until a new round of presignature generation (and, for OT-based ECDSA, a new round of triple generation) is completed. This is a direct analog to the `LimitedCallsEnforcer` counting a use even though the delegated action was never executed: the one-time-use state is permanently advanced with no corresponding output.

---

### Likelihood Explanation

The coordinator is a participant in the signing protocol — it is not a specially trusted external party. In the robust ECDSA scheme, the protocol is explicitly designed to tolerate up to `max_malicious` malicious participants, yet the coordinator role is assigned to a single participant with no accountability mechanism for whether it completes its obligation. Any participant who is assigned (or self-assigns) the coordinator role can execute this attack. The attack requires no cryptographic capability — only the ability to participate in the protocol and then drop the connection or return an error after collecting shares.

---

### Recommendation

Non-coordinator participants should receive a confirmation message from the coordinator containing the final assembled signature (or at minimum a commitment to it), and verify it against the public key before treating the presignature as consumed. This closes the gap: the presignature is only considered spent once the participant has cryptographic evidence that the signing action was completed. Alternatively, the application layer should be explicitly warned in documentation that presignatures are consumed even on coordinator failure, and presignature pools should be sized accordingly with a recovery path.

---

### Proof of Concept

1. Honest participants P1…Pn each hold a `RerandomizedPresignOutput` (one-time-use).
2. Malicious coordinator C calls `sign(participants, C, ...)` for each participant, initiating the protocol.
3. Each participant Pi executes `do_sign_participant`, computes `s_i`, sends it to C, and returns `Ok(None)` — their presignature is now consumed.
4. C receives all `s_i` values but never calls `do_sign_coordinator` to completion (or simply drops the assembled signature).
5. No valid signature is produced. All presignatures held by P1…Pn are permanently spent.
6. P1…Pn must re-run the full presignature protocol (and triple generation for OT-based ECDSA) before they can sign again — a significant resource cost that can be repeated indefinitely by the malicious coordinator. [8](#0-7) [2](#0-1)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    public_key: AffinePoint,

    msg_hash: Scalar,

    let mut s = compute_signature_share(&presignature, msg_hash, &participants, me)?.0;
    let wait_round = chan.next_waitpoint();

    for (_, s_i) in
        recv_from_others::<SerializableScalar<C>>(&chan, wait_round, &participants, me).await?
    {
        // Sum the linearized shares
        s += s_i.0;

    // raise error if s is zero
    if s.is_zero().into() {
        return Err(ProtocolError::AssertionFailed(
            "signature part s cannot be zero".to_string(),

    s.conditional_assign(&(-s), s.is_high());

        big_r: presignature.big_r,
        s,

        return Err(ProtocolError::AssertionFailed(
            "signature failed to verify".to_string(),
        ));
