### Title
WalletConnect `take_offer` Confirmation Dialog Understates Total Spend Due to Incorrect Multi-NFT Royalty Aggregation - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts`, used exclusively to render the WalletConnect `chia_wallet.take_offer` confirmation dialog, incorrectly calculates the total fungible amount a user will spend when accepting an offer containing multiple NFTs with royalties. It divides the fungible amount equally among NFTs before applying each royalty percentage, rather than applying each royalty to the full amount. The result is a systematically understated `amountWithRoyalties` displayed to the user, who then approves a WalletConnect transaction believing they will spend less than the blockchain will actually deduct.

### Finding Description

`parseCommandDisplay` is the function that generates the human-readable confirmation data shown in the WalletConnect approval dialog before a user signs a `chia_wallet.take_offer` command. [1](#0-0) 

For a `take_offer` command, it calls `walletDeltaToDisplay`, which in turn calls `withRoyaltyTotals` → `formatAmountWithRoyalties` to compute the `amountWithRoyalties` field shown to the user. [2](#0-1) 

The critical flaw is on line 363:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);  // integer division
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The function divides the fungible amount equally among all NFTs (`splitAmount`), then applies each NFT's royalty percentage to that split amount. This is mathematically wrong. The Chia protocol applies each NFT's royalty to the **full** fungible amount independently.

**Concrete example** (confirmed by the existing test at line 251–335):

- Offer: 100,000,000 mojos (0.0001 XCH) for 2 NFTs with royalties 500 (5%) and 10 (0.1%)
- **Display calculation** (buggy): `splitAmount = 50,000,000`; royalty = `(50M × 500/10000) + (50M × 10/10000)` = 2,550,000; total shown = **102,550,000 mojos (0.00010255 XCH)**
- **Actual on-chain**: royalty = `(100M × 500/10000) + (100M × 10/10000)` = 5,100,000; total deducted = **105,100,000 mojos (0.0001051 XCH)**

The user sees 0.00010255 XCH but the blockchain deducts 0.0001051 XCH — a ~2.4% understatement in this example. With higher royalties or more NFTs, the gap grows proportionally. [3](#0-2) 

The royalty percentages fed into this function come from `offerSummaryRoyaltyPercentages`, which reads them directly from the offer blob's `infos` driver dict — values fully controlled by the offer maker (the WalletConnect dApp). [4](#0-3) 

### Impact Explanation

A WalletConnect-connected dApp can craft a `take_offer` payload containing multiple NFTs with non-trivial royalty percentages. The confirmation dialog will display a materially lower `amountWithRoyalties` than what the blockchain will actually deduct. The user approves the transaction believing they are spending amount X, but the actual spend is X + (undisclosed royalty delta). This is a direct accounting spoofing of the WalletConnect approval state, causing the user to authorize the wrong spend amount.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of... WalletConnect state that causes a user to approve... the wrong... amount."*

### Likelihood Explanation

Any WalletConnect-paired dApp that has been granted `take_offer` permission can trigger this. The dApp constructs a valid multi-NFT offer (or sources one from a marketplace) and submits it via WalletConnect. No additional privileges, key access, or cryptographic breaks are required. The user's only defense is manually computing the correct royalty total before approving, which is not a reasonable expectation.

### Recommendation

Replace the split-amount approach with per-NFT full-amount royalty application:

```typescript
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

This matches the actual on-chain royalty semantics and ensures the confirmation dialog accurately reflects what will be deducted.

### Proof of Concept

The existing test at `packages/gui/src/electron/commands/parseCommandDisplay.test.ts` lines 251–335 already demonstrates the bug: it asserts `amountWithRoyalties: '0.00010255'` for a 2-NFT offer with royalties 500 and 10 on 100,000,000 mojos. The correct value is `'0.0001051'`. The test passes only because it was written to match the buggy output, not the correct on-chain value. [5](#0-4)

```typescript
function offerSummaryRoyaltyPercentages(offerSummary: OfferSummaryForDisplay): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},

  };

  const { infos } = offerSummary;
  if (!isPlainObject(infos)) {

  for (const assetId of Object.keys(offerSummary.requested)) {

    royaltyPercentages.spending[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  for (const assetId of Object.keys(offerSummary.offered)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.receiving[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  return royaltyPercentages;
}
```

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```

```typescript
export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
```
