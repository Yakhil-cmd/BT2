### Title
Unprivileged Caller Can Permanently Freeze Per-Block Consensus Rewards via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in `staking.cairo` updates the global `last_reward_block` storage variable **before** checking the `disable_rewards` flag. Because the function has no access control, any unprivileged caller can invoke it with `disable_rewards: true` to consume the per-block reward slot without distributing any rewards. The block's yield is then permanently unrecoverable.

### Finding Description

`update_rewards` is the consensus-phase reward distribution entry point. It enforces a global one-call-per-block invariant via `last_reward_block`: [1](#0-0) 

The function writes `last_reward_block` at line 1485, **before** the `disable_rewards` guard at line 1487: [2](#0-1) 

`last_reward_block` is a single global field shared across all stakers: [3](#0-2) 

The function carries no role-based access control — `general_prerequisites()` only asserts the contract is unpaused and the caller is non-zero: [4](#0-3) 

Therefore any address can call `update_rewards(valid_staker, disable_rewards: true)` in any block, advance `last_reward_block` to that block number, and cause every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`. The rewards that would have been distributed for that block are permanently lost — they are never added to `unclaimed_rewards` in the `RewardSupplier` and can never be reclaimed.

This is structurally identical to the reference vulnerability: a critical state variable (`last_reward_block` ↔ `startDate`) is mutated before the prerequisite condition (`disable_rewards == false` ↔ `drawableFunds > 0`) is validated, leaving the system in an inconsistent state where the slot is consumed but no value is delivered.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

For every block in which the attacker front-runs the legitimate `update_rewards` call:

- `last_reward_block` is set to the current block number.
- No STRK block rewards are calculated or forwarded to `RewardSupplier.unclaimed_rewards`.
- No pool rewards are transferred to delegation pool contracts.
- The staker's `unclaimed_rewards_own` is never incremented.

Because `last_reward_block` is a monotonically-consumed slot (one per block, never replayed), the yield for each attacked block is gone forever. An attacker running this every block eliminates all consensus-phase rewards for all stakers and delegators.

### Likelihood Explanation

**High.** The attack requires:
1. Any valid (active, non-zero-balance) staker address — trivially obtained from on-chain events.
2. One transaction per block — cheap on Starknet.
3. No privileged role, no leaked key, no bridge compromise.

The attacker has no profit motive but can cause severe, sustained damage to the protocol and all participants.

### Recommendation

Move the `last_reward_block` write to **after** the `disable_rewards` / `is_pre_consensus` guard, so the slot is only consumed when rewards are actually distributed:

```cairo
// Update last block rewards ONLY when rewards will be distributed.
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);
// ... rest of reward distribution logic
```

Alternatively, restrict `update_rewards` with `disable_rewards: true` to a privileged role (e.g., `only_app_governor`), or remove the `disable_rewards` parameter entirely if it has no legitimate public use.

### Proof of Concept

1. Consensus rewards are active: `current_epoch >= consensus_rewards_first_epoch`.
2. Attacker picks any active staker address `S` with non-zero STRK balance.
3. At the start of block `N`, attacker calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. Inside the call:
   - `current_block_number (N) > last_reward_block` → passes.
   - Staker `S` is active with non-zero balance → passes.
   - `last_reward_block` is written to `N`. [5](#0-4) 
   - `disable_rewards == true` → early return, zero rewards distributed. [6](#0-5) 
5. Any legitimate call to `update_rewards` in block `N` now fails:
   ```
   assert!(current_block_number > self.last_reward_block.read(), ...)
   // N > N  →  false  →  REWARDS_ALREADY_UPDATED
   ``` [7](#0-6) 
6. Block `N`'s rewards are permanently lost. Attacker repeats for every subsequent block.
