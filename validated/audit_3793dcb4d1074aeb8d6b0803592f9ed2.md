### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` to bound the pool admin's global spread fee, but `setPoolBinAdditionalFees` lets the pool admin set per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) with **no corresponding cap check**. A pool admin can set bin additional fees up to `uint16.max` (65 535 ≙ 6.5535 % in E6 units) on every bin, making the effective admin-controlled fee `adminSpreadFeeE6 + addFeeBuyE6`, which can far exceed `maxAdminSpreadFeeE6`.

---

### Finding Description

`setPoolAdminFees` correctly enforces the factory cap:

```solidity
// MetricOmmPoolFactory.sol  setPoolAdminFees
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, however, performs **no cap check at all** — it forwards the raw `uint16` values directly to the pool:

```solidity
// MetricOmmPoolFactory.sol  setPoolBinAdditionalFees
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, never the fee magnitudes:

```solidity
// MetricOmmPool.sol  setBinAdditionalFees
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap the per-bin additional fee is **added on top of** the global spread fee:

```solidity
// MetricOmmPool.sol  _swapToken1ForToken0SpecifiedOutput (representative call-site)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The effective admin-controlled fee for a swap in any bin is therefore `adminSpreadFeeE6 + addFeeBuyE6`, which is entirely uncapped by `maxAdminSpreadFeeE6`.

---

### Impact Explanation

The factory owner's ability to constrain pool admin fees via `maxAdminSpreadFeeE6` is rendered meaningless for per-bin fees. Even if the owner sets `maxAdminSpreadFeeE6 = 0`, the pool admin can impose up to 6.5535 % additional spread on every bin through `setPoolBinAdditionalFees`. Traders executing swaps in those bins pay fees far above the protocol-intended ceiling, constituting an admin-boundary break where the semi-trusted pool admin exceeds the caps the factory is designed to enforce.

---

### Likelihood Explanation

Any pool admin can call `setPoolBinAdditionalFees` at any time with no preconditions beyond holding the `poolAdmin` role. The bypass requires a single transaction per bin and is immediately effective for all subsequent swaps.

---

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` (or inside `MetricOmmPool.setBinAdditionalFees`) that enforces `addFeeBuyE6 ≤ maxAdminSpreadFeeE6` and `addFeeSellE6 ≤ maxAdminSpreadFeeE6` (or introduce a dedicated `maxAdminBinFeeE6` cap). This mirrors the guard already present in `setPoolAdminFees` and closes the bypass path.

---

### Proof of Concept

1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 1 000` (0.1 %).
2. Pool is created; pool admin calls `setPoolAdminFees(pool, 1000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no revert**, accepted.
4. A trader swaps through bin 0. The swap math applies:
   - `baseFeeX64` derived from `spreadFeeE6 = protocolSpreadFeeE6 + 1000`
   - plus `Math.mulDiv(65535, ONE_X64, 1e6)` ≈ 6.5535 % additional fee
5. Effective admin-controlled spread = `1 000 + 65 535 = 66 535` (6.6535 %), 66× the intended cap of 1 000.
6. The factory owner has no mechanism to prevent or retroactively clamp this without the pool admin's cooperation.
