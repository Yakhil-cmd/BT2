### Title
Incorrect Multi-NFT Royalty Amount Displayed in WalletConnect Confirmation Dialog Causes Users to Approve Transactions for More Than Shown - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the total fungible amount equally among all NFTs before computing each royalty, producing a systematically understated `amountWithRoyalties` in the WalletConnect `take_offer` / `create_offer_for_ids` confirmation dialog. A user approves what appears to be a lower cost; the actual on-chain deduction is higher, growing worse as the number of NFTs or royalty percentages increase.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect command display to show the user the total XCH/CAT they will spend including creator royalties:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The Chia protocol applies each NFT's royalty to the **full** fungible amount, not to a 1/N share of it. By dividing `amount` by `royaltyPercentages.length` before the reduction, the function computes royalties on a fraction of the actual price, understating the total by a factor of N (the number of NFTs).

**Concrete example** (confirmed by the existing test at line 251–335):

| Scenario | Displayed `amountWithRoyalties` | Correct value |
|---|---|---|
| 2 NFTs, royalties 500 bp + 10 bp, 0.0001 XCH | 0.00010255 XCH | 0.0001051 XCH |
| 5 NFTs, each 10% royalty, 1 XCH | 1.1 XCH | 1.5 XCH | [2](#0-1) 

The result is surfaced in the WalletConnect confirmation dialog via `withRoyaltyTotals` → `walletDelta.spending[*].amountWithRoyalties`, which is the primary cost figure shown to the user before they click "Approve". [3](#0-2) 

### Impact Explanation

A WalletConnect-connected dapp sends a `take_offer` or `create_offer_for_ids` request containing a multi-NFT offer. The GUI confirmation dialog shows an understated total cost. The user approves, and the full node deducts the correct (higher) royalty amount from their wallet. The discrepancy scales with both the number of NFTs and their royalty percentages — with 5 NFTs each carrying a 10% royalty, the user is shown 1.1 XCH but 1.5 XCH is actually spent. This constitutes displaying the wrong amount in a WalletConnect approval flow, causing the user to authorize a spend larger than what was presented.

### Likelihood Explanation

Any WalletConnect dapp that constructs a multi-NFT offer (two or more NFTs with non-zero royalties in a single `take_offer` or `create_offer_for_ids` call) triggers this path. No special privileges are required; the attacker only needs to craft such an offer and present it to the user via WalletConnect. The bug is unconditional — it fires whenever `royaltyPercentages.length > 1`.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty should be calculated on the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3) 

### Proof of Concept

The existing test at `parseCommandDisplay.test.ts:251–335` already encodes the wrong expected value (`0.00010255`) and passes, confirming the bug is live. With the correct formula the expected value would be `0.0001051`.

Attacker steps:
1. Connect a dapp to the user's Chia GUI via WalletConnect.
2. Issue `chia_wallet.take_offer` with an offer blob containing two NFTs, each with a 10% royalty, requesting 1 XCH.
3. The confirmation dialog shows `amountWithRoyalties ≈ 1.1 XCH`; the actual on-chain deduction is `1.2 XCH`.
4. User clicks Approve; 0.1 XCH more than displayed is deducted. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
