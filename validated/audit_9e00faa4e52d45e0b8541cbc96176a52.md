### Title
Double-Counting of Notional Fee Accumulator in `collectFees` Causes Pool Over-Allocation — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`collectFees` computes spread fees as a fraction of the full token surplus, then pays out notional fees from a separate accumulator. Because the notional fee accumulator is itself part of the surplus (notional fees are withheld from bin balances during swaps and held as surplus), the notional amount is counted twice: once inside the spread-fee base and once as a direct notional payout. The total attempted transfer exceeds the available surplus, draining LP principal to cover the shortfall.

---

### Finding Description

During a swap, the pool withholds the notional fee from the bin balance and accumulates it in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`. This means those amounts sit in the pool's token balance but are **not** reflected in any bin balance — they are part of the "surplus." [1](#0-0) 

When `collectFees` runs, it computes:

```
surplus0Scaled = token0.balanceOf(pool) * multiplier − totalScaledToken0InBins
```

This `surplus0Scaled` **includes** `notionalFeeToken0Scaled` because notional fees were never added to bin balances.

The function then computes spread fees as a fraction of `surplus0Scaled`:

```
spreadFee0ToAdmin    = (surplus0Scaled * adminSpreadFeeE6)    / spreadSumE6
spreadFee0ToProtocol = (surplus0Scaled * protocolSpreadFeeE6) / spreadSumE6
```

And separately pays out the full notional accumulator:

```
notionalFee0ToProtocol = notionalFeeToken0Scaled − notionalFee0ToAdmin
```

The total attempted payout is:

```
total = spreadFee(surplus)          // surplus already contains notional amount
      + notionalFee(notionalAccum)  // notional amount paid out again
```

The excess equals `spreadFeeRate × notionalAccumulator` — the spread fee rate applied to the notional amount that was already included in the spread-fee base. This excess must come from LP bin balances, causing insolvency.

The test suite explicitly documents and asserts this over-allocation: [2](#0-1) 

```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
```

---

### Impact Explanation

**Critical.** When both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` are configured and notional fees have accrued, every call to `collectFees` transfers more tokens than the legitimate fee surplus. The shortfall is drawn from LP bin balances. Repeated fee collections progressively drain LP principal, causing pool insolvency: LP withdrawal claims will exceed the pool's actual token balances.

---

### Likelihood Explanation

**High.** Both fee types are independently configurable and are expected to be used together in production pools. `collectFees` is called by the factory on any pool with non-zero fees. No special attacker action is required — the over-allocation occurs on every legitimate fee collection when both fee types are simultaneously active.

---

### Recommendation

Before computing spread fees, subtract the notional accumulator from the surplus so that the two fee bases are disjoint:

```solidity
uint256 spreadBase0Scaled = surplus0Scaled > notionalFeeToken0Scaled
    ? surplus0Scaled - notionalFeeToken0Scaled
    : 0;
uint256 spreadBase1Scaled = surplus1Scaled > notionalFeeToken1Scaled
    ? surplus1Scaled - notionalFeeToken1Scaled
    : 0;
```

Then compute spread fees from `spreadBase0Scaled` / `spreadBase1Scaled` and notional fees from the accumulators independently. This ensures the two fee streams are additive without overlap and the total payout never exceeds the actual surplus.

---

### Proof of Concept

The existing test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive` in `metric-core/test/MetricOmmPool.notionalFee.t.sol` already demonstrates the bug end-to-end: [3](#0-2) 

1. A pool is configured with both `spreadFeeE6 = PROTOCOL_FEE + ADMIN_FEE` and `notionalFeeE8 = FEE_1_PCT_E8`.
2. Eight rounds of swaps accumulate both spread surplus and notional fees.
3. The test mirrors the `collectFees` split math and computes `total0Attempted` and `total1Attempted`.
4. Both assertions confirm `total > surplus`, proving the pool would attempt to transfer more than it holds as fee surplus, drawing the difference from LP bin balances.
