### Title
Unverified Polynomial Evaluations in Robust ECDSA Presign Allow Malicious Participant to Silently Corrupt Nonce Shares - (File: src/ecdsa/robust_ecdsa/presign.rs)

### Summary
In `do_presign`, polynomial evaluations received from other participants in Round 1 are accumulated into local shares with no verification against any Feldman-style commitment. A malicious participant can send an arbitrary scalar as their `fk` evaluation to any honest party, silently biasing that party's nonce share `k_me`. The corrupted nonce propagates into the presign output without triggering any protocol abort, causing every subsequent signing attempt that uses the output to produce an invalid signature. The sender is never identified because the `from` field is discarded at the receive site.

### Finding Description

In `src/ecdsa/robust_ecdsa/presign.rs` lines 134–139, each participant receives a `package` of five polynomial evaluations (`fk`, `fa`, `fb`, `fd`, `fe`) from every other participant and immediately folds them into its running share accumulator:

```rust
for (_, package) in recv_from_others(&chan, wait_round_1, &participants, me).await? {
    // Step 2.2
    shares.add_shares(&package);
}
```

No commitment is broadcast before these private sends, and no Feldman-consistency check is performed on the received values. The `from` participant is silently discarded (`_`), so even a detected mismatch cannot be attributed.

Contrast this with the DKG protocol in `src/dkg.rs` lines 517–522, where every received share is verified against a broadcast Feldman commitment before being accumulated:

```rust
validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
```

The only post-hoc check in the presign protocol is the exponent-interpolation consistency check at lines 274–290, which verifies `W_i = R^{a_i}` values — covering only the `fa` polynomial. The `fk` polynomial (the nonce polynomial) has no corresponding public commitment and no interpolation check. A malicious participant `j` can therefore send `fk_j(me) + δ` (for any attacker-chosen `δ ≠ 0`) to honest participant `me`. Participant `me` computes:

```
k_me = Σ fk_j(me)   ← includes the poisoned evaluation
R_me = g^{k_me}     ← broadcast in Round 2, wrong value
```

Because `R_me` is wrong, the aggregate nonce `R = Σ R_i` is wrong, and the final presign output `(k_me, a_me, b_me, …)` is cryptographically inconsistent. The exponent-interpolation check on `W_i = R^{a_i}` still passes because `a_i` values are unaffected and the check only requires internal polynomial consistency of the `W` shares, not correctness of `R`.

### Impact Explanation

Every honest participant that receives a poisoned `fk` evaluation silently accepts a corrupted presign output. Any signature produced from that output is invalid and will be rejected by all verifiers. Because the `from` field is discarded, the malicious participant is never identified and can repeat the attack across every presign session, permanently denying signing to honest parties.

This matches **High: Corruption of presign outputs so honest parties accept unusable cryptographic outputs**, and **High: Permanent denial of signing for honest parties under valid protocol inputs**.

### Likelihood Explanation

The attacker only needs to be a legitimate participant in the presign protocol — no privileged keys, no external compromise. Sending a single wrong scalar in Round 1 is sufficient. The attack is trivially repeatable and unattributable because the sender identity is discarded at the accumulation site.

### Recommendation

Before sending private polynomial evaluations, each participant should broadcast a vector of Feldman commitments `{g^{fk_j(0)}, g^{fk_j(1)}, …}` (analogous to the DKG commitment broadcast). Each recipient must then verify the received evaluation against the commitment before calling `shares.add_shares`. This mirrors the existing `validate_received_share` pattern in `src/dkg.rs` and closes the gap for the `fk` polynomial. Alternatively, add an exponent-interpolation check on the `R_i = g^{k_i}` values (analogous to the existing `W_i` check) to detect inconsistent nonce contributions.

### Proof of Concept

1. Two-party presign: honest party `H`, malicious party `M`.
2. `M` generates its polynomials normally but, in Round 1, sends `fk_M(H) + δ` (with `δ ≠ 0`) to `H` instead of the correct evaluation.
3. `H` executes `shares.add_shares(&package)` at line 138 — no error, `k_H` is now `k_H_correct + δ`.
4. `H` broadcasts `R_H = g^{k_H_correct + δ}` in Round 2.
5. The exponent-interpolation check at lines 274–290 operates on `W_i = R^{a_i}` and passes because `a_i` values are unaffected.
6. `H` outputs a presign record with the poisoned `k_H`. Any signature produced with this record has an incorrect `s` component and fails ECDSA verification.

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
