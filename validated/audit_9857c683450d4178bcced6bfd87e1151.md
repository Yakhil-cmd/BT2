### Title
Unbounded `move_to_block` Loop Causes NEAR Gas Exhaustion, Freezing All Hashchain-Protected Engine Operations - (File: engine-hashchain/src/hashchain.rs)

### Summary

When the Aurora Engine's hashchain integrity feature is active, every protected contract call invokes `load_hashchain`, which calls `move_to_block` to catch up to the current NEAR block height. `move_to_block` contains an unbounded `while` loop that executes one keccak hash per skipped block with no iteration cap. If the engine is idle long enough for the accumulated block gap to exhaust NEAR's 300 TGas per-transaction gas limit, every subsequent hashchain-protected call (including `submit`) will fail with out-of-gas, effectively freezing the engine.

### Finding Description

`Hashchain::move_to_block` in `engine-hashchain/src/hashchain.rs` iterates unconditionally from the stored block height to the current NEAR block height:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
        &self.chain_id,
        self.contract_account_id.as_bytes(),
        self.current_block_height,
        self.previous_block_hashchain,
    );
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
``` [1](#0-0) 

Each iteration calls `compute_block_hashchain`, which invokes `aurora_engine_sdk::keccak` — a NEAR host function with a fixed gas cost per call. There is no maximum-gap guard, no early-exit, and no batching. [2](#0-1) 

This loop is triggered unconditionally inside `load_hashchain` whenever the stored block height lags behind the current block:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
``` [3](#0-2) 

`load_hashchain` is called at the start of both `with_hashchain` and `with_logs_hashchain`, which wrap every state-mutating engine method including `submit`: [4](#0-3) [5](#0-4) 

### Impact Explanation

NEAR's per-transaction gas limit is 300 TGas. Each iteration of the `move_to_block` loop performs at least one `keccak256` host-function call over a concatenated input of ~370 bytes (chain_id 32 B + account_id + block_height 8 B + previous hashchain 32 B + bloom 256 B). At NEAR's published gas schedule (~1.5 TGas base + ~1.5 GGas/byte for `keccak256`), each iteration costs roughly 2 TGas, meaning a gap of only ~150 NEAR blocks (~2–3 minutes of engine inactivity at ~1 block/second) is sufficient to exhaust the 300 TGas budget.

Once the gap exceeds this threshold, every call to `with_hashchain` or `with_logs_hashchain` fails before executing any user logic. Because `submit` (the primary EVM transaction entry point) is wrapped by `with_logs_hashchain`, no EVM transaction can be processed. Funds locked in EVM contracts become inaccessible until a contract upgrade is deployed. This constitutes **permanent freezing of funds** from the user's perspective until an operator intervention occurs.

### Likelihood Explanation

The hashchain must first be initialized via `start_hashchain` (an admin action). Once enabled, the vulnerability is triggered by ordinary inactivity — no adversarial transaction is required. Any period of low engine usage (maintenance windows, low-demand intervals, network congestion preventing relayer submissions) naturally grows the block gap. Because NEAR produces ~1 block/second, a gap of 150 blocks accumulates in under 3 minutes. An attacker who observes the engine is idle can simply wait and then be the first to attempt a transaction, confirming the freeze without spending any funds.

### Recommendation

1. **Cap the per-call catch-up**: In `load_hashchain`, limit the number of blocks processed per transaction (e.g., `MAX_BLOCKS_PER_CALL = 100`). If the gap exceeds the cap, process only the allowed number of blocks and save the intermediate state, allowing subsequent transactions to continue the catch-up incrementally.

2. **Add a gap guard in `move_to_block`**: Accept a `max_steps: u64` parameter and return a distinct error (e.g., `BlockchainHashchainError::GapTooLarge`) when the limit is exceeded, so callers can handle partial progress gracefully.

3. **Emit a warning / disable hashchain on large gap**: If the gap exceeds a safe threshold, disable hashchain recording for that call rather than aborting the entire transaction, preserving liveness.

### Proof of Concept

1. Deploy Aurora Engine and initialize the hashchain via `start_hashchain`.
2. Allow the engine to be idle for ~150 NEAR blocks (~2–3 minutes).
3. Submit any EVM transaction via `submit`. The call enters `with_logs_hashchain` → `load_hashchain` → `move_to_block(current_height)`.
4. The `while` loop executes ~150 iterations, each performing a `keccak256` host call costing ~2 TGas, totalling ~300 TGas.
5. The NEAR runtime aborts the transaction with `GasLimitExceeded` before any EVM execution occurs.
6. All subsequent `submit` calls fail identically because the stored hashchain height never advances past the gap. The engine is frozen.

    pub fn move_to_block(
        &mut self,
        next_block_height: u64,
    ) -> Result<(), BlockchainHashchainError> {
        if next_block_height <= self.current_block_height {
            return Err(BlockchainHashchainError::BlockHeightIncorrect);
        }

        while self.current_block_height < next_block_height {
            self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
                &self.chain_id,
                self.contract_account_id.as_bytes(),
                self.current_block_height,
                self.previous_block_hashchain,
            );

            self.block_hashchain_computer.clear_txs();
            self.current_block_height += 1;
        }

        Ok(())
    }

pub fn with_hashchain<I, E, T, F>(
    mut io: I,
    env: &E,
    function_name: &str,
    f: F,
) -> Result<T, ContractError>
where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {
        let cache_ref = cache.borrow();
        hashchain.add_block_tx(
            block_height,
            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}
