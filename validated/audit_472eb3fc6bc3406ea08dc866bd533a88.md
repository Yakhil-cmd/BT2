### Title
Gateway Admits Invoke Transactions with Unverified Signatures via `skip_stateful_validations` Overly-Broad Mempool Presence Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path skips the `__validate__` entry-point call (i.e., account-contract signature verification) for any Invoke transaction with `nonce == 1` when `account_nonce == 0` in state AND `account_tx_in_pool_or_recent_block` returns `true`. The presence check is satisfied by **any** transaction from that address in the pool — not specifically a `deploy_account`. An unprivileged attacker can therefore submit a legitimate `deploy_account` followed by an Invoke with `nonce=1` and an **arbitrary/invalid signature**; the Invoke bypasses `__validate__` at the gateway and is admitted to the mempool.

### Finding Description

`skip_stateful_validations` is the UX feature that allows a user to submit `deploy_account + invoke` atomically before the account is deployed on-chain. [1](#0-0) 

The guard condition is:

```
tx is Invoke
  AND tx.nonce() == Nonce(Felt::ONE)
  AND account_nonce == Nonce(Felt::ZERO)
  AND account_tx_in_pool_or_recent_block(sender_address) == true
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in the mempool's `tx_pool` **or** in the `staged`/`committed` nonce maps — for **any** transaction type: [3](#0-2) 

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false` and the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling `__validate__`: [4](#0-3) [5](#0-4) 

The `perform_pre_validation_stage` that does run still checks nonce, fee bounds, and balance — but **not** the account contract's `__validate__` logic (i.e., signature): [6](#0-5) 

**Attack path:**

1. Attacker pre-funds address `X` with enough STRK to cover the Invoke fee (transferring to an undeployed address is valid ERC-20).
2. Attacker submits a valid `deploy_account` for `X` → admitted to the mempool pool.
3. Attacker submits `Invoke(sender=X, nonce=1, signature=<garbage>)`:
   - Stateless check: passes (signature *length* is within limit, content not checked).
   - `validate_nonce`: `0 ≤ 1 ≤ 200` ✓.
   - `validate_by_mempool`: no duplicate, no nonce-too-old ✓.
   - `skip_stateful_validations`: all four conditions met → returns `true`.
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` is **never called**.
   - `verify_can_pay_committed_bounds`: balance check passes (account was pre-funded).
   - **Invoke with invalid signature is admitted to the mempool.**

The comment in the code acknowledges the intent but does not restrict the check to `deploy_account` transactions specifically: [7](#0-6) 

### Impact Explanation

The gateway's invariant — *every transaction admitted to the mempool has had its account-contract authorization (`__validate__`) verified* — is broken. An Invoke transaction with an arbitrary/invalid signature is accepted into the mempool. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The transaction will fail at execution time (the batcher runs `__validate__` with `validate=true`), so on-chain state is not corrupted. However, the mempool accepts an unauthorized transaction, wasting batcher execution resources and polluting the mempool. At scale, an attacker can repeatedly submit `(deploy_account, invalid_invoke)` pairs — each pair requires only a pre-funded address and a valid class hash — to continuously inject failing transactions into the sequencer pipeline.

### Likelihood Explanation

- **Unprivileged**: any external user can trigger this; no special role or key is required.
- **Low cost**: one `deploy_account` per attack iteration; the class hash can be any already-declared class.
- **Deterministic**: the four-condition check in `skip_stateful_validations` is always satisfied given the described setup; there is no race condition or timing dependency.
- **Config-independent**: `max_nonce_for_validation_skip` is hardcoded to `Nonce(Felt::ONE)` in the Rust gateway path (not configurable at runtime to a different value in the production config). [8](#0-7) 

### Recommendation

Replace the overly-broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the pool for the sender address. Add a dedicated mempool API (e.g., `deploy_account_in_pool(address)`) that inspects transaction types, and use that in `skip_stateful_validations` instead of the generic presence check. Alternatively, add a type assertion inside `skip_stateful_validations` that the matching pool transaction is a `DeployAccount`.

### Proof of Concept

```
// Pre-condition: class C is already declared on-chain.
// Pre-condition: address X = calculate_contract_address(salt, C, constructor_calldata, deployer=0)

// Step 1: fund X (no account needed, plain ERC-20 transfer)
erc20.transfer(X, fee_amount)

// Step 2: submit deploy_account — passes all gateway checks, enters mempool pool
gateway.add_tx(DeployAccount { class_hash: C, salt, nonce: 0, signature: <valid> })

// Step 3: submit invoke with garbage signature
gateway.add_tx(Invoke {
    sender_address: X,
    nonce: 1,
    calldata: [...],
    signature: [0xdeadbeef, 0xcafebabe],   // completely invalid
    resource_bounds: { l2_gas: { max_amount: N, max_price: P } },
})
// Expected result: ADMITTED to mempool (skip_stateful_validations returns true,
//                  __validate__ is never called at the gateway).
// Batcher result:  deploy_account executes OK; invoke fails with ValidateFailure.
```

The gateway returns a transaction hash for the invalid Invoke, confirming admission. The batcher later emits a `REVERTED` receipt for it.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        &self,
