### Title
Missing `isPaused()` Guard on `CadenceOwnedAccount.deposit` Allows Token Inflow When EVM Is Paused - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVM` Cadence contract implements a pause mechanism (`EVM.isPaused()`) to halt all state-mutating EVM operations during emergencies. Every state-mutating function on `CadenceOwnedAccount` checks this guard — except `deposit`, which allows any unprivileged caller to bridge FLOW tokens into the EVM environment even when the EVM is paused.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, the `CadenceOwnedAccount` resource exposes a set of state-mutating functions. Every one of them includes a `pre { !EVM.isPaused(): "EVM operations are temporarily paused" }` condition — except `deposit`:

```cadence
// Line 562-565 — NO isPaused() guard
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)
}
```

Compare this to `withdraw`, which is the symmetric operation:

```cadence
// Line 586-590 — HAS isPaused() guard
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    ...
}
```

The full set of guarded functions includes: `withdraw`, `deploy`, `call`, `callWithSigAndArgs`, `depositNFT`, `withdrawNFT`, `depositTokens`, `withdrawTokens`, `createCadenceOwnedAccount`, `run`, and `batchRun`. The `deposit` function is the sole state-mutating operation that is unguarded.

`deposit` is declared `access(all)`, meaning any unprivileged transaction sender can call it on any COA they hold a reference to, with no entitlement required.

---

### Impact Explanation

When governance pauses the EVM (e.g., in response to a bridge exploit or critical vulnerability), the intent is to freeze all EVM state changes. However, because `CadenceOwnedAccount.deposit` lacks the pause guard, any user can continue to move FLOW tokens from the Cadence side into EVM accounts. This:

1. Increases the EVM-side balance of attacker-controlled COA accounts during a pause window.
2. Undermines the invariant that a paused EVM is fully frozen — assets can still flow in, even if they cannot flow out or be acted upon until the pause is lifted.
3. Creates an asymmetric state: `withdraw` is blocked but `deposit` is not, meaning the pause does not achieve a symmetric halt of cross-VM token movement.

The impact class is **bridge escrow mis-accounting / cross-VM asset movement bypass** during an emergency pause.

---

### Likelihood Explanation

- The entry path is a standard Cadence transaction calling `coa.deposit(from: <-vault)` on any COA the caller controls.
- No special privileges, staked nodes, or leaked keys are required.
- The `deposit` function is `access(all)` and callable by any Flow account.
- The scenario is realistic whenever the EVM pause is activated (e.g., during a bridge incident), which is precisely when the guard matters most.

---

### Recommendation

Add the `!EVM.isPaused()` pre-condition to `CadenceOwnedAccount.deposit`, consistent with all other state-mutating functions on the same resource:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    self.address().deposit(from: <-from)
}
```

**Unguarded function (the vulnerability):**

`CadenceOwnedAccount.deposit` — `access(all)`, no pause check: [5](#0-4) 

**Attack transaction (pseudocode):**
```cadence
import EVM from <EVM_address>
import FlowToken from <FlowToken_address>

// EVM is paused. Governance expects all EVM state changes to be halted.
transaction {
    prepare(acct: auth(Storage) &Account) {
        // Attacker mints or withdraws FLOW from their Cadence vault
        let vault <- acct.storage.borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
            from: /storage/flowTokenVault
        )!.withdraw(amount: 1000.0)

        // Attacker's COA — deposit bypasses the pause guard
        let coa = acct.storage.borrow<&EVM.CadenceOwnedAccount>(from: /storage/coa)!
        coa.deposit(from: <-vault)  // succeeds even when EVM.isPaused() == true
    }
}
```

This transaction succeeds when `EVM.isPaused()` is `true`, while a symmetric `coa.withdraw(...)` call in the same state would revert with `"EVM operations are temporarily paused"`.
