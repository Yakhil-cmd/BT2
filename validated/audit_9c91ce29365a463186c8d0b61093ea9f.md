### Title
`protocolUnpausePool` Leaves Pool Permanently Swap-Paused After Protocol Unpause — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, so after the protocol "unpauses" a pool the pool remains in admin-paused state (level 1). Swaps are still blocked. The pool admin must issue a second, independent `unpausePool` call to restore swap functionality — a step that is not required by the function's name or documented intent, and that may never happen if the admin is unavailable.

---

### Finding Description

The pool uses a three-level pause system:

| Level | Meaning |
|-------|---------|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`swap` enforces `whenNotPaused`, which reverts whenever `pauseLevel != 0`. [1](#0-0) 

The intended protocol-pause lifecycle is:

1. `protocolPausePool` — owner pauses from level 0 or 1 → level 2.
2. `protocolUnpausePool` — owner unpauses from level 2 → **should reach level 0 (active)**.

But the implementation of `protocolUnpausePool` is:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← sets to 1, not 0
}
``` [2](#0-1) 

It calls `setPause(1)`, landing the pool in admin-paused state. Because `whenNotPaused` checks `pauseLevel != 0`, every subsequent `swap` call reverts with `PoolPaused`. [3](#0-2) 

The only recovery path is for the pool admin to call `unpausePool`, which requires `cur == 1` and then calls `setPause(0)`. [4](#0-3) 

This is a hidden, undocumented dependency: the protocol owner's "unpause" action does not restore swap functionality; it merely hands the problem to the pool admin.

---

### Impact Explanation

After `protocolUnpausePool` is called, `swap` remains permanently blocked until the pool admin independently calls `unpausePool`. If the admin is unavailable, malicious, or simply unaware that a second action is required, the pool is stuck in a non-functional state indefinitely. All swap volume and LP fee accrual is lost for that period. This is broken core swap functionality under the allowed impact gate.

---

### Likelihood Explanation

The trigger is a normal, expected operational sequence: the protocol owner pauses a pool for any reason (e.g., oracle anomaly, emergency) and then decides to lift the pause. This is a routine admin action. Every such cycle silently leaves the pool swap-disabled, making the likelihood medium.

---

### Recommendation

Change `protocolUnpausePool` to set the pause level to 0 (active) instead of 1 (admin-paused):

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the design intent is to require admin acknowledgment before full activation, that intent must be explicitly documented and the function renamed (e.g., `protocolUnpauseToAdminPaused`), and a separate `protocolFullUnpausePool` path should be provided.

---

### Proof of Concept

```
State: pauseLevel = 0 (active, swaps working)

1. owner calls protocolPausePool(pool)
   → setPause(2)  [pauseLevel = 2, swaps blocked]

2. owner calls protocolUnpausePool(pool)
   → setPause(1)  [pauseLevel = 1, swaps STILL blocked]

3. user calls swap(...)
   → _checkNotPaused() → pauseLevel != 0 → revert PoolPaused()

4. Pool admin must call unpausePool(pool)
   → setPause(0)  [pauseLevel = 0, swaps restored]

   If admin is absent → pool remains swap-disabled indefinitely.
``` [5](#0-4) [6](#0-5)
