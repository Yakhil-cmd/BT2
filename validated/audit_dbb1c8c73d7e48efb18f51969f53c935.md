### Title
Weight Aggregation Bypass via Duplicate Key Indices in `TransactionVerifier` — (`fvm/transactionVerifier.go`)

### Summary

`TransactionVerifier.getAccountKeysAndAggregateWeights` aggregates weights per `(address, keyIndex)` pair, while `newSignatureEntries` deduplicates only by `(address, keyIndex)`. Because the account storage layer explicitly supports adding the same cryptographic key at multiple indices with independent per-index weights, an attacker who controls an account can register the same private key at two indices each with weight=500, sign a transaction with both indices using the same private key, and have the verifier sum 500+500=1000 ≥ threshold — authorizing the transaction with a single private key that should only carry half the required weight.

---

### Finding Description

**Root cause 1 — Deduplication is by key-index, not by underlying key**

`newSignatureEntries` builds a `uniqueKey` struct keyed on `{address, keyIndex}`: [1](#0-0) 

Two signatures for the same address but different key indices (e.g., 0 and 1) are never deduplicated, even when both indices map to the same stored cryptographic key.

**Root cause 2 — Weight aggregation is additive over key indices**

`getAccountKeysAndAggregateWeights` fetches the per-index weight from `GetRuntimeAccountPublicKey` and adds it unconditionally: [2](#0-1) 

**Root cause 3 — `GetRuntimeAccountPublicKey` returns per-index weight from metadata, not the stored key's weight**

For key index > 0, the weight returned is the value stored in the metadata section for that specific index: [3](#0-2) 

This weight is set independently per index via `AppendDuplicateKeyMetadata`: [4](#0-3) 

**Root cause 4 — The account storage layer explicitly supports duplicate keys with independent weights**

`AppendAccountPublicKeyMetadata` detects a duplicate via `FindDuplicateKey` and calls `AppendDuplicateKeyMetadata`, which stores a separate weight for the new index while pointing `storedKeyIndex` back to the original stored key: [5](#0-4) 

The test suite confirms this is an intended, supported state ("account with 2 duplicate keys"): [6](#0-5) 

**Root cause 5 — Signature verification uses the stored key, which is the same for all duplicate indices**

`verify()` verifies against `entry.accountKey.PublicKey`: [7](#0-6) 

For a duplicate key index, `GetRuntimeAccountPublicKey` resolves `storedKeyIndex` via the mapping and fetches the same underlying public key bytes. The same private key therefore produces a valid signature for every duplicate index.

---

### Impact Explanation

An attacker who controls an account can:

1. Start with key K at index 0, weight=1000 (full control).
2. Add key K at index 1, weight=500 (duplicate; `saveKey=false`, no new stored key written).
3. Add key K at index 2, weight=500 (another duplicate).
4. Revoke index 0 (weight=1000). The account now appears to require two distinct 500-weight keys.
5. Sign any future transaction with both index 1 and index 2 using the single private key K.
6. The verifier aggregates 500+500=1000 ≥ 1000 and authorizes the transaction.

The account appears to enforce multi-sig (two independent 500-weight keys) but is actually controlled by a single private key. Any on-chain asset, capability, or contract owned by that account is fully accessible to the attacker. This directly violates the invariant stated in the comment at line 242: [8](#0-7) 

> "at this point, account keys are guaranteed to be unique across all signatures"

That comment is false: uniqueness is only guaranteed at the `(address, keyIndex)` level, not at the underlying cryptographic key level.

---

### Likelihood Explanation

The attacker must have had full control of the account at some prior point (to add the duplicate keys). This is not a remote, zero-prior-access exploit. However, it is realistic in:

- Accounts where one party previously held full control and wants to retain covert access after handing over apparent multi-sig governance.
- Accounts where a key-management script or tool accidentally adds the same key twice with partial weights.
- Any account migration or key-rotation workflow that does not verify key uniqueness before revoking the old full-weight key.

The attack is locally reproducible on an unmodified Flow emulator with no special node access.

---

### Recommendation

In `getAccountKeysAndAggregateWeights`, after fetching the `RuntimeAccountPublicKey` for each signature, track the set of `storedKeyIndex` values (or encoded public key bytes) already counted for each address. If the same underlying key has already contributed weight, skip the weight addition (or return an error). The fix should be applied at the verifier level rather than the storage level, since duplicate keys are a legitimate storage feature (e.g., for key rotation with overlapping validity windows).

---

### Proof of Concept

```go
// Emulator PoC (pseudocode, runnable on Flow emulator)
// 1. Create account with key K, weight=1000
// 2. Submit tx signed by K: account.keys.add(publicKey: K, weight: 500)  // index 1
// 3. Submit tx signed by K: account.keys.add(publicKey: K, weight: 500)  // index 2
// 4. Submit tx signed by K: account.keys.revoke(keyIndex: 0)             // revoke index 0
// 5. Build tx with authorizer=account, proposalKey=(account, index:1, seq:0)
//    PayloadSignatures: [{address, keyIndex:1, sig: Sign(payload, K)},
//                        {address, keyIndex:2, sig: Sign(payload, K)}]
//    EnvelopeSignatures: [{address, keyIndex:1, sig: Sign(envelope, K)}]
// 6. Submit — verifier aggregates weight[index1]=500 + weight[index2]=500 = 1000 >= 1000
//    Both crypto verifications pass (same underlying key K).
//    Transaction is AUTHORIZED. Expected: REJECTED.
```

The `newSignatureEntries` dedup check passes because `(addr, 1)` ≠ `(addr, 2)`. [9](#0-8) 

The weight aggregation sums both partial weights. [10](#0-9) 

Both `verify()` calls succeed because `GetRuntimeAccountPublicKey(addr, 1)` and `GetRuntimeAccountPublicKey(addr, 2)` both return the same stored public key bytes (via `storedKeyIndex=0`). [11](#0-10)

// AppendDuplicateKeyMetadata appends duplicate key metadata.
func (m *KeyMetadataAppender) AppendDuplicateKeyMetadata(
	keyIndex uint32,
	duplicateStoredKeyIndex uint32,
	revoked bool,
	weight uint16,
) (err error) {

	// Append revoked status and weight
	m.weightAndRevokedStatusBytes, err = appendWeightAndRevokedStatus(m.weightAndRevokedStatusBytes, revoked, weight)
	if err != nil {
		return err
	}

	if !m.deduplicated {
		// Set deduplication flag
		m.deduplicated = true

		// Save mapping start key index.
		m.startIndexForMapping = keyIndex
	}

	// Append duplicate stored key index to mapping
	m.mappingBytes, err = appendStoredKeyIndexToMappings(m.mappingBytes, duplicateStoredKeyIndex)
	if err != nil {
		return err
	}

	return nil
}
