### Title
L1 Transaction `refund_recipient` Set to Treasury (`BASE_TOKEN_HOLDER_ADDRESS`) Causes Phantom Deposit Recorded in L2AssetTracker Without Actual Token Movement - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`mint_base_token` unconditionally calls `notify_l2_asset_tracker` (which increments `totalSuccessfulDepositsFromL1` in the L2AssetTracker) and then calls `transfer_from_treasury`. Neither function checks whether the destination address `to` equals `BASE_TOKEN_HOLDER_ADDRESS` (the treasury). When an attacker sets `refund_recipient = BASE_TOKEN_HOLDER_ADDRESS` in an L1 priority transaction, `transfer_from_treasury` performs a self-transfer (subtract then add to the same address, net zero), but `notify_l2_asset_tracker` has already recorded the full refund amount as a successful L1 deposit. This inflates `totalSuccessfulDepositsFromL1` without any real token movement, breaking the L2AssetTracker's accounting invariant.

---

### Finding Description

During L1→L2 transaction processing in `process_l1_transaction`, the bootloader calls `mint_base_token` up to three times: once for the value mint (to `from`), once for the operator fee (to `coinbase`), and once for the refund (to `refund_recipient`).

`mint_base_token` always calls `notify_l2_asset_tracker` first, then `transfer_from_treasury`: [1](#0-0) 

`notify_l2_asset_tracker` calls `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2(fromChainId, amount)`, which increments `interopInfo[assetId].totalSuccessfulDepositsFromL1 += amount`: [2](#0-1) 

`transfer_from_treasury` then subtracts `amount` from `BASE_TOKEN_HOLDER_ADDRESS` and adds `amount` to `to`: [3](#0-2) 

**Neither `mint_base_token` nor `transfer_from_treasury` checks whether `to == BASE_TOKEN_HOLDER_ADDRESS`.**

The `refund_recipient` is read directly from the attacker-controlled `transaction.reserved[1]` field: [4](#0-3) 

The `ZKsyncL1Tx` struct confirms `refund_recipient` is a free user-supplied field: [5](#0-4) 

**Attack path:**

1. Attacker submits an L1 priority transaction with `refund_recipient = BASE_TOKEN_HOLDER_ADDRESS` and any nonzero `gas_limit * gas_price` (ensuring `to_refund_recipient > 0`).
2. The bootloader computes `to_refund_recipient = prepaid_fee - pay_to_operator > 0`.
3. `mint_base_token` is called with `to = BASE_TOKEN_HOLDER_ADDRESS` and `amount = to_refund_recipient`.
4. `notify_l2_asset_tracker` records `to_refund_recipient` as a successful deposit in `totalSuccessfulDepositsFromL1`.
5. `transfer_from_treasury` subtracts `to_refund_recipient` from treasury, then adds it back to treasury (same address) — **net zero balance change**.
6. The L2AssetTracker's `totalSuccessfulDepositsFromL1` is inflated by `to_refund_recipient` with no corresponding token movement.

The same self-transfer issue applies if `from == BASE_TOKEN_HOLDER_ADDRESS` (for the value-mint call at line 634–657), though `from` is also attacker-controlled. [6](#0-5) 

---

### Impact Explanation

`totalSuccessfulDepositsFromL1` in the L2AssetTracker is the canonical on-chain record of how much base token has been bridged from L1. Inflating it without actual token movement breaks the invariant between the real token supply and the recorded deposit history. The L2AssetTracker comment explicitly states that a revert of this call is treated as a fatal system error because "the chain's token accounting would be inconsistent": [7](#0-6) 

Downstream contracts or off-chain systems that rely on `totalSuccessfulDepositsFromL1` for cross-chain accounting, migration logic (`_needToForceSetAssetMigrationOnL2`), or bridge reconciliation will observe phantom deposits, leading to incorrect state and potential fund mismanagement.

---

### Likelihood Explanation

Any L1 transaction sender can freely set `refund_recipient` to any address, including `BASE_TOKEN_HOLDER_ADDRESS`. No privileged role is required. The only precondition is that `to_refund_recipient > 0`, which is trivially satisfied by any transaction with `gas_limit > gas_used`. This is a direct, low-cost, unprivileged attack.

---

### Recommendation

In `transfer_from_treasury` (or in `mint_base_token` before calling it), add a guard that rejects or redirects the transfer when `to == BASE_TOKEN_HOLDER_ADDRESS`:

```rust
pub fn transfer_from_treasury<...>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError> {
    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;
    // Guard: sending back to treasury is a no-op on balances but
    // notify_l2_asset_tracker has already been called — skip the transfer
    // entirely to avoid phantom accounting.
    if to == treasury_address {
        return Err(interface_error!(BootloaderInterfaceError::TreasuryTransferFailed));
        // or silently skip, depending on protocol intent
    }
    // ... existing subtract/add logic
}
```

Alternatively, validate `refund_recipient != BASE_TOKEN_HOLDER_ADDRESS` at the point where it is read from `transaction.reserved[1]`, and substitute a safe fallback address (e.g., `from`) if the check fails.

---

### Proof of Concept

1. Deploy a ZKsync OS chain with a funded treasury (`BASE_TOKEN_HOLDER_ADDRESS`).
2. Submit an L1 priority transaction:
   - `from`: any address
   - `gas_price`: 1000, `gas_limit`: 100_000
   - `to_mint` (reserved[0]): `gas_price * gas_limit` (e.g., 100_000_000)
   - `refund_recipient` (reserved[1]): `BASE_TOKEN_HOLDER_ADDRESS` (e.g., `0x0000000000000000000000000000000000010001`)
   - `value`: 0
3. After execution, read `L2AssetTracker.interopInfo[BASE_TOKEN_ASSET_ID].totalSuccessfulDepositsFromL1`.
4. Observe it equals `to_mint` (the full deposited amount), even though the refund portion was a self-transfer to the treasury with zero net balance change.
5. Repeat: each such transaction inflates `totalSuccessfulDepositsFromL1` by `to_refund_recipient` without any real token movement. [8](#0-7) [9](#0-8) [10](#0-9)
