Audit Report

## Title
`balance_update_guard` Global Slot Exhaustion via Subaccount Flooding Causes DoS for All ckBTC Depositors — (File: `rs/bitcoin/ckbtc/minter/src/guard.rs`)

## Summary
The ckBTC minter's `update_balance` endpoint enforces a single global cap of `MAX_CONCURRENT = 100` in-flight requests via `Guard::new`, keyed on `Account` (owner + subaccount). Because both `args.owner` and `args.subaccount` are fully caller-supplied with no per-principal sub-limit, a single unprivileged principal can exhaust all 100 slots by submitting 100 concurrent calls with distinct subaccounts. While those calls are suspended awaiting the async `get_utxos` response, every legitimate depositor receives `TemporarilyUnavailable("too many concurrent requests")` and cannot mint ckBTC.

## Finding Description
**Root cause — `rs/bitcoin/ckbtc/minter/src/guard.rs` lines 51–53:**
`Guard::new` checks only the global set size against `MAX_CONCURRENT = 100` with no per-owner sub-limit:
```rust
if accounts.len() >= MAX_CONCURRENT {
    return Err(GuardError::TooManyConcurrentRequests);
}
```
The guard key is a full `Account { owner, subaccount }`, so 100 distinct subaccounts of the same owner occupy 100 distinct slots.

**Vulnerable call site — `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` lines 164–183:**
```rust
let caller_account = Account {
    owner: args.owner.unwrap_or(caller),   // caller-supplied
    subaccount: args.subaccount,           // caller-supplied
};
let _guard = balance_update_guard(caller_account)?;
// ...
let utxos = get_utxos(...).await?   // guard held across async suspension
    .utxos;
```
The guard is acquired and then held across the entire `get_utxos` inter-canister call. When `satoshis_to_mint == 0` (which is always true for an attacker with no UTXOs), a second `get_utxos` call at line 229 is also awaited while the guard remains held, extending the slot occupation further.

**Caller-supplied fields — `rs/bitcoin/ckbtc/minter/src/updates/update_balance.rs` lines 35–41:**
`UpdateBalanceArgs` exposes `owner: Option<Principal>` and `subaccount: Option<Subaccount>` to any caller. The only restriction (line 149) is that `args.owner` must not equal the minter's own canister ID — no per-principal rate limit exists.

**Public endpoint — `rs/bitcoin/ckbtc/minter/src/main.rs` lines 196–200:**
`update_balance` is a `#[update]` endpoint gated only by `check_anonymous_caller()`, reachable by any non-anonymous IC principal.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* and *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

With all 100 guard slots occupied, every legitimate `update_balance` call returns `UpdateBalanceError::TemporarilyUnavailable("too many concurrent requests")`, completely blocking ckBTC minting for all users for the duration of the attack. The ckBTC minter is a production financial integration canister; denial of minting constitutes concrete, measurable harm to users and the protocol.

## Likelihood Explanation
- **No cost barrier**: The attacker needs no UTXOs, no ckBTC, and no privileged role — only the ability to send ingress messages.
- **Single principal sufficient**: 100 distinct subaccounts (e.g., `[0u8;32]` through `[99u8;32]`) of one principal fill all slots; subaccount creation is free.
- **Sustainable**: Each slot is held for the round-trip latency of one (or two) `get_utxos` calls to the Bitcoin canister. The attacker re-submits as calls complete (all return `NoNewUtxos`), maintaining the DoS indefinitely.
- **Directly reachable**: `update_balance` is a public endpoint callable by any non-anonymous principal on mainnet.

## Recommendation
1. **Add a per-principal concurrent-request cap** inside `Guard::new`: count how many entries in `accounts` share the same `owner` principal and reject if that count reaches a per-principal limit `K` (e.g., `K = 3`). This prevents a single principal from monopolising the global pool while preserving the per-account deduplication (`AlreadyProcessing`) check.
2. **Separate global cap from per-principal cap**: the existing `AlreadyProcessing` guard is correct and should be retained; only the `TooManyConcurrentRequests` path needs the per-principal sub-limit.
3. Alternatively, **move guard acquisition before `init_ecdsa_public_key().await`** to reduce the window, though this alone does not eliminate the attack.

## Proof of Concept

// Attacker principal P sends 100 concurrent update_balance ingress messages:
for i in 0u8..100 {
    update_balance(UpdateBalanceArgs {
        owner: Some(P),
        subaccount: Some([i; 32]),
    });
}
// Each call:
//   1. Passes mode check and anonymous check.
//   2. Awaits init_ecdsa_public_key() (no-op after first call).

    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
