### Title
Division-by-Zero in `calculate_staker_total_staking_power` Blocks Attestation and Reward Distribution When STRK Total Stake Is Zero - (File: src/staking/utils.cairo)

### Summary
`calculate_staker_total_staking_power` in `src/staking/utils.cairo` unconditionally divides by `strk_total_stake` without a zero-guard, while the analogous BTC branch is correctly guarded. If all STRK stakers complete their unstaking, `strk_total_stake` becomes zero, causing a panic on any subsequent call to this function and permanently blocking attestation and reward distribution.

### Finding Description
In `src/staking/utils.cairo`, the function `calculate_staker_total_staking_power` computes a staker's consensus power as a weighted sum of their STRK and BTC contributions: [1](#0-0) 

The BTC branch is correctly guarded against division by zero: [2](#0-1) 

However, the STRK branch has **no equivalent guard**: [3](#0-2) 

`mul_wide_and_div` returns `None` when the divisor is zero. The `.unwrap()` call (with no error message) will panic, reverting the transaction. This is the exact same class of unguarded arithmetic corner case as the normalization factor bug: a missing precondition check on a denominator that can legitimately reach zero.

`strk_total_stake` reaches zero when all STRK stakers complete their unstaking. The protocol's K-epoch delay means the balance trace eventually settles to zero after the exit wait window expires. Once that happens, any call path that invokes `calculate_staker_total_staking_power` — including attestation and reward distribution — will revert.

This function is called in `src/staking/staking.cairo` in at least four places: [4](#0-3) 

### Impact Explanation
When `strk_total_stake` is zero, every invocation of `calculate_staker_total_staking_power` panics. This blocks:
- **Attestation**: stakers cannot submit block hashes, losing liveness rewards.
- **Reward distribution / `update_rewards`**: the staking contract cannot push per-block rewards, permanently freezing all unclaimed yield for every staker and pool member.

This matches the allowed impact: **Permanent freezing of unclaimed yield** (High).

### Likelihood Explanation
Any unprivileged staker can trigger this by calling `unstake_intent` and waiting out the exit wait window. If the last STRK staker exits — a realistic scenario in a protocol that also supports BTC-only staking — `strk_total_stake` drops to zero. No privileged access, no external dependency, and no phishing is required. The attacker's only cost is unstaking their own tokens.

### Recommendation
Add a zero-check for `strk_total_stake` identical to the existing BTC guard:

```cairo
let strk_staking_power = if strk_total_stake.is_zero() {
    Zero::zero()
} else {
    mul_wide_and_div(
        lhs: staker_strk_total_amount.to_amount_18_decimals(),
        rhs: STRK_WEIGHT_FACTOR,
        div: strk_total_stake.to_amount_18_decimals(),
    )
    .unwrap()
};
```

Additionally, consider adding a protocol-level invariant that prevents the total STRK stake from reaching zero while any staker or pool member still has unclaimed rewards.

### Proof of Concept

1. A single STRK staker (`Alice`) is the only staker in the system; a BTC staker (`Bob`) also exists.
2. Alice calls `unstake_intent()` and waits for the exit wait window to expire.
3. Alice calls `unstake_action()`, completing her exit. `strk_total_stake` is now `0`.
4. Bob calls `attest(...)` for the current epoch. The staking contract calls `calculate_staker_total_staking_power` with `strk_total_stake = 0`.
5. `mul_wide_and_div(lhs: bob_strk_amount, rhs: STRK_WEIGHT_FACTOR, div: 0)` returns `None`.
6. `.unwrap()` panics. The transaction reverts.
7. Bob can never attest. `update_rewards` also reverts. All unclaimed yield is permanently frozen. [3](#0-2) [2](#0-1)
