### Title
EIP-7702 Authorization Authority Blocklist Check Silently Bypassed via `auth.Authority()` Error Swallow - (File: `app/proposal.go`, `app/block_address.go`)

---

### Summary

Both `ProposalHandler.ValidateTransaction` and `BlockAddressesDecorator.AnteHandle` silently skip the blocklist check for EIP-7702 authorization authorities when `auth.Authority()` returns an error. An unprivileged attacker can craft an EIP-7702 transaction with an invalid authorization signature for a blocked address, causing `auth.Authority()` to fail, which causes the `if err == nil` guard to skip the blocklist enforcement entirely. The transaction then passes both CheckTx admission and `ProcessProposal` validation and is included in a finalized block.

---

### Finding Description

Cronos implements a compliance blocklist enforced at two layers:

1. **`BlockAddressesDecorator.AnteHandle`** (`app/block_address.go`) — runs only during `CheckTx` (mempool admission).
2. **`ProposalHandler.ValidateTransaction`** (`app/proposal.go`) — runs during `PrepareProposal` / `ProcessProposal`.

Both locations check EIP-7702 authorization list entries for blocked authority addresses using the same pattern:

```go
addr, err := auth.Authority()
if err == nil {
    if _, ok := bad.blockedMap[...]; ok {
        return ctx, fmt.Errorf("signer is blocked: %s", ...)
    }
}
``` [1](#0-0) [2](#0-1) 

`auth.Authority()` performs ECDSA recovery from the authorization's `(v, r, s)` signature. It returns an error when the signature is invalid — for example, when the `s` value is in the upper half of the secp256k1 curve (malleable signature), when `r` or `s` are out of range, or when the recovered address is zero. When this error occurs, the `if err == nil` branch is not entered, and the blocked-address check for that authorization entry is **silently skipped** rather than treated as a rejection.

The `BlockAddressesDecorator` is additionally gated behind `ctx.IsCheckTx()`:

```go
func (bad BlockAddressesDecorator) AnteHandle(...) (...) {
    if ctx.IsCheckTx() {
        // all blocklist checks, including EIP-7702 authority check
    }
    return next(ctx, tx, simulate)
}
``` [3](#0-2) 

This means the `BlockAddressesDecorator` does not run at `FinalizeBlock` at all. The sole enforcement at block-inclusion time is `ProposalHandler.ValidateTransaction`, which has the same silent-skip flaw.

---

### Impact Explanation

The Cronos blocklist is a compliance/sanctions mechanism. Its intent is that blocked addresses cannot participate in any transaction capacity — including as EIP-7702 authorization signers (authorities). The `if err == nil` silent skip allows a transaction that names a blocked address as an EIP-7702 authority to pass both mempool admission and `ProcessProposal` validation and be committed to a finalized block. This is a direct bypass of the block-list authorization check, which falls under the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

---

### Likelihood Explanation

The attack requires no special privileges. Any user can construct an EIP-7702 type-4 transaction with a crafted authorization entry whose `(v, r, s)` values are chosen to make `auth.Authority()` return an error (e.g., `s` in the upper half of the curve). This is a standard, low-effort transaction manipulation. The attacker only needs to know which address is on the blocklist and submit a single transaction.

---

### Recommendation

Replace the silent `if err == nil` skip with an explicit rejection when `auth.Authority()` fails. An authorization entry whose signer cannot be recovered is either malformed or deliberately crafted; it should be treated as a hard error in both `ValidateTransaction` and `BlockAddressesDecorator.AnteHandle`:

```go
addr, err := auth.Authority()
if err != nil {
    return fmt.Errorf("failed to recover EIP-7702 authorization authority: %w", err)
}
if _, ok := h.blocklist[encoded]; ok {
    return fmt.Errorf("signer is blocked: %s", encoded)
}
```

Apply the same fix in both `app/proposal.go` (`ValidateTransaction`) and `app/block_address.go` (`BlockAddressesDecorator.AnteHandle`). [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. Identify a blocked address `B` (present in the on-chain blocklist via `MsgStoreBlockList`).
2. Construct an EIP-7702 type-4 transaction where the `authorizationList` contains one entry whose `(chainId, address, nonce)` fields target `B`'s account, but whose `(v, r, s)` signature has `s` in the upper half of the secp256k1 curve (i.e., `s > secp256k1n/2`).
3. Submit the transaction from any unblocked sender address.
4. At CheckTx: `BlockAddressesDecorator` calls `auth.Authority()` → error → `if err == nil` branch skipped → tx admitted to mempool.
5. At `ProcessProposal`: `ProposalHandler.ValidateTransaction` calls `auth.Authority()` → same error → check skipped → block accepted.
6. At `FinalizeBlock`: `BlockAddressesDecorator` does not run (`IsCheckTx()` is false) → no check.
7. EVM execution: the invalid authorization is skipped per EIP-7702 spec (no-op for `B`'s account), but the transaction is committed on-chain, bypassing the blocklist enforcement for the EIP-7702 authority address. [6](#0-5) [7](#0-6)

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}
