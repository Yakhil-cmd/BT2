### Title
No Cap on Bin Additional Fees Allows Pool Admin to Exceed Intended Fee Bounds — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound check, while every other admin fee setter (`setPoolAdminFees`, `setPoolProtocolFee`, `setFeeCaps`) is guarded by `maxAdminSpreadFeeE6` / `HARD_MAX_SPREAD_FEE_E6`. A pool admin can set per-bin additional fees to `type(uint16).max` (65 535 = 6.5535% in E6 units) on any or all bins, bypassing the factory's intended fee cap system and extracting excess fees from traders.

---

### Finding Description

The factory defines hard fee caps and enforces them on every admin fee path except one:

`HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) and `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1%) are the absolute ceilings. [1](#0-0) 

`setPoolAdminFees` enforces these caps before writing: [2](#0-1) 

But `setPoolBinAdditionalFees` passes the caller-supplied values straight through with zero validation: [3](#0-2) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index: [4](#0-3) 

These stored values are then added directly to `baseFeeX64` during every swap in the affected bin: [5](#0-4) 

The result is that the effective buy/sell price seen by a trader in that bin is `oracle_price × (1 + baseFee + addFee)`, where `addFee` can be up to 6.5535% with no factory-level guard.

---

### Impact Explanation

- Traders swapping through a bin with `addFeeBuyE6 = 65535` pay up to **6.5535% more** than the oracle price, on top of the already-permitted spread fee. This is a direct loss of trader principal on every swap routed through that bin.
- The excess fee accrues as pool surplus and is split between the protocol and the pool admin at collection time, meaning the pool admin directly profits from the uncapped extraction.
- The factory's cap system (`maxAdminSpreadFeeE6`, `HARD_MAX_SPREAD_FEE_E6`) is rendered meaningless for the bin-level fee path: a pool admin who is blocked from raising the global spread fee can instead raise per-bin fees to the `uint16` ceiling with no restriction.

---

### Likelihood Explanation

The trigger is a single call to `setPoolBinAdditionalFees` by the pool admin — a role that is explicitly described as "semi-trusted only inside caps and timelocks." No special preconditions, no timelock, no co-signer. The pool admin can act unilaterally and immediately. Any pool whose admin is compromised, mistaken, or acting adversarially can exploit this path.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
+   if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
+   if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, define a dedicated `maxAdminBinAdditionalFeeE6` constant and enforce it here, so that the sum of global spread fee and per-bin additional fee cannot exceed the hard cap.

---

### Proof of Concept

1. Factory is deployed; `HARD_MAX_SPREAD_FEE_E6 = 200_000`, `maxAdminSpreadFeeE6 = 200_000`.
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, global spread fee is at the hard cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no revert**, `addFeeBuyE6` and `addFeeSellE6` are stored as 65535 on bin 0.
4. A trader calls `swap(…, zeroForOne=true, …)` routed through bin 0. Inside `_swapToken0ForToken1SpecifiedInput`, the effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 6.5535% above the oracle mid-price, on top of the already-maxed 20% spread fee.
5. The trader receives ~6.5535% fewer tokens than the oracle price and the factory cap together should permit. The surplus accumulates in the pool and is collected by the admin via `collectPoolFees`.
