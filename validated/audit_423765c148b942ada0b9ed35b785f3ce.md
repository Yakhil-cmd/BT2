### Title
Sub-minimum attoFlow Balances Permanently Irrecoverable from COA EVM Accounts - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `CadenceOwnedAccount` (COA) resource in Flow EVM can accumulate sub-minimum attoFlow balances (< 1e10 attoFlow) through EVM-level operations such as gas refunds, small value transfers, and withdrawal truncation remainders. These sub-minimum balances cannot be withdrawn back to Cadence because `CadenceOwnedAccount.withdraw()` enforces a hard minimum of 1e10 attoFlow, and no alternative Cadence-level withdrawal path exists for amounts below this threshold. This is the direct Flow analog of the `HibernationDen` pattern: a component can receive assets (via EVM-level credits) but has no mechanism to recover them.

---

### Finding Description

The `CadenceOwnedAccount` resource exposes a `deposit` function that is `access(all)` â€” any Cadence transaction can deposit FLOW tokens into a COA's EVM address:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)
}
```

The only path to recover FLOW tokens from a COA back to Cadence is `CadenceOwnedAccount.withdraw()`, which requires the `Owner | Withdraw` entitlement and enforces a minimum balance check:

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault {
    ...
    let vault <- InternalEVM.withdraw(
        from: self.addressBytes,
        amount: balance.attoflow
    ) as! @FlowToken.Vault
```

The Go-layer implementation in `fvm/evm/impl/impl.go` enforces the minimum before any withdrawal:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
```

`AttoFlowBalanceIsValidForFlowVault` in `fvm/evm/types/balance.go` returns `false` for any amount below `1e10` attoFlow (= 0.00000001 FLOW):

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
```

Sub-minimum balances accumulate in COA EVM accounts through three realistic paths:

1. **Withdrawal truncation remainder**: When a user withdraws an amount whose attoFlow value has a sub-1e10 remainder (e.g., `1230000789912345678` attoFlow), the EVM account is only debited the truncated UFix64-aligned amount (`1230000780000000000`). The remainder `9912345678` attoFlow stays in the EVM account and is permanently irrecoverable to Cadence.

2. **EVM gas refunds**: EVM transactions that trigger gas refunds (e.g., storage slot clearing under EIP-3529) can credit sub-minimum amounts to the COA's EVM address.

3. **EVM-level value transfers**: Any EVM transaction sending a sub-minimum value (< 1e10 attoFlow) to a COA address leaves a balance that cannot be bridged back.

Once a COA's entire balance is below 1e10 attoFlow, every call to `withdraw()` panics with `ErrWithdrawBalanceRounding`, and there is no fallback path.

---

### Impact Explanation

**Medium.** FLOW tokens accumulate in COA EVM accounts with no Cadence-level recovery path. While individual stuck amounts are small (< 0.00000001 FLOW per occurrence), the effect is cumulative across all COAs on the network over time. The tokens are not destroyed â€” they remain in the EVM state â€” but they are permanently inaccessible to the Cadence owner of the COA. This is a cross-VM asset loss: assets bridged into EVM cannot be fully bridged back.

---

### Likelihood Explanation

**Medium.** The truncation remainder path is triggered on every withdrawal where the requested attoFlow amount is not a multiple of 1e10. This is a routine occurrence for any user performing EVM operations with non-UFix64-aligned balances. EVM gas refunds to COA addresses are also a normal operational event.

---

### Recommendation

Add a `sweepDust()` or `transferAll()` function on `CadenceOwnedAccount` that allows the COA owner to transfer the entire EVM balance (including sub-minimum amounts) to another EVM address via an EVM-level call, bypassing the UFix64 minimum constraint. Alternatively, allow the COA owner to transfer sub-minimum balances to the protocol's fee vault or burn address so they are not permanently locked.

---

### Proof of Concept

The truncation remainder path is directly confirmed by the existing test `"test coa withdraw with remainder truncation"` in `fvm/evm/evm_test.go`:

- Deposit: `2.34 FLOW` = `2_340_000_000_000_000_000` attoFlow
- Withdraw request: `1_230_000_789_912_345_678` attoFlow
- Actual EVM debit: `1_230_000_780_000_000_000` attoFlow (truncated to UFix64)
- Remainder stuck in EVM: `9_912_345_678` attoFlow (< 1e10, permanently irrecoverable)
- Balance after: `1_109_999_220_000_000_000` attoFlow (not `1_109_999_210_087_654_322`)

A subsequent attempt to withdraw the `9_912_345_678` attoFlow remainder panics with `ErrWithdrawBalanceRounding`, as confirmed by `"test coa withdraw with rounding error"`.
