### Title
No Maximum Commission Guard in `switch_delegation_pool` / `enter_delegation_pool` Allows Staker to Front-Run Delegators with Commission Increase — (File: `src/pool/pool.cairo`)

---

### Summary

`switch_delegation_pool` and `enter_delegation_pool` accept no `max_commission` parameter. A staker who holds an active `CommissionCommitment` can raise their commission up to `max_commission` between the moment a delegator submits a delegation/switch transaction and the moment it is executed, silently reducing the delegator's future yield with no recourse.

---

### Finding Description

The `update_commission` internal function in `src/staking/staking.cairo` enforces two distinct paths:

- **No active commitment**: commission can only *decrease*.
- **Active commitment**: commission can be set to any value `<= commitment.max_commission`, which may be *higher* than the current commission. [1](#0-0) 

`set_commission_commitment` requires only that `max_commission >= current_commission`, so a staker can legitimately set `max_commission` far above the current rate. [2](#0-1) 

The public entry-points that delegators use to commit funds to a pool are:

```
fn enter_delegation_pool(
    ref self: ContractState, reward_address: ContractAddress, amount: Amount,
)
``` [3](#0-2) 

```
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
) -> Amount
``` [4](#0-3) 

Neither function accepts a `max_commission` argument. The commission that will govern the delegator's future rewards is read from storage at reward-distribution time, not at delegation time. There is no on-chain check that the commission at execution matches what the delegator observed when they signed the transaction. [5](#0-4) 

---

### Impact Explanation

A delegator who switches or enters a pool expecting a 5 % commission may find themselves locked into a 50 % commission pool (or any value up to `max_commission`). Every subsequent reward distribution will transfer a larger fraction of the delegator's earned yield to the staker. This constitutes **theft of unclaimed yield** (High impact) because the delegator's principal is intact but their accrued rewards are permanently diverted to the staker at a rate the delegator never agreed to.

---

### Likelihood Explanation

The attack requires:
1. The staker to have set a `CommissionCommitment` with `max_commission` materially above the current commission — a legitimate, publicly visible on-chain action.
2. The staker to call `set_commission` to the higher value before the delegator's pending transaction is sequenced.

On Starknet the sequencer is centralised and the transaction pool is observable. A staker who monitors for incoming `switch_delegation_pool` or `enter_delegation_pool` calls targeting their pool can trivially submit `set_commission` first. Even without active monitoring, a staker can raise commission at any time; any delegator whose transaction is already in-flight will be affected. The precondition (an active commitment) is a normal protocol feature, not an exotic configuration.

---

### Recommendation

Add an optional `max_commission: Option<Commission>` parameter to both `enter_delegation_pool` and `switch_delegation_pool`. When provided, assert that the destination pool's current commission does not exceed it before completing the delegation:

```cairo
if let Option::Some(max_comm) = max_commission {
    let current_commission = self.get_commission_from_staking_contract();
    assert!(current_commission <= max_comm, Error::COMMISSION_EXCEEDS_MAX);
}
```

This mirrors the fix applied in the referenced `early-purchase` program (`max_lamports_to_spend`) and gives delegators a slippage guard with no impact on existing callers who pass `Option::None`.

---

### Proof of Concept

1. Staker calls `set_commission(500)` (5 %) then `set_commission_commitment(max_commission: 5000, expiration_epoch: current + 100)`. [6](#0-5) 

2. Delegator observes 5 % commission and calls `exit_delegation_pool_intent(amount)` on their current pool, then submits `switch_delegation_pool(to_staker, to_pool, amount)`. [7](#0-6) 

3. Staker sees the pending transaction and calls `set_commission(5000)` (50 %). Because the commitment is active and `5000 <= max_commission`, this succeeds. [8](#0-7) 

4. The delegator's `switch_delegation_pool` transaction executes. No commission check exists; the delegator is now a member of the 50 % commission pool. [9](#0-8) 

5. All future reward distributions apply the 50 % commission, silently diverting the majority of the delegator's yield to the staker for the remainder of the commitment period. [10](#0-9)

        fn switch_delegation_pool(
            ref self: ContractState,
            to_staker: ContractAddress,
            to_pool: ContractAddress,
            amount: Amount,
        ) -> Amount {
            // Asserts.
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            assert!(
                pool_member_info.unpool_time.is_some(),
                "{}",
                GenericError::MISSING_UNDELEGATE_INTENT,
            );
            assert!(amount <= pool_member_info.unpool_amount, "{}", GenericError::AMOUNT_TOO_HIGH);
            let reward_address = pool_member_info.reward_address;

            // Update pool_member_info and write to storage.
            pool_member_info.unpool_amount -= amount;
            if pool_member_info.unpool_amount.is_zero() {
                // unpool_amount is zero, clear unpool_time.
                pool_member_info.unpool_time = Option::None;
            }
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Serialize the switch pool data and invoke the staking contract to switch pool.
            let switch_pool_data = SwitchPoolData { pool_member, reward_address };
            let mut serialized_data = array![];
            switch_pool_data.serialize(ref output: serialized_data);
            self
                .staking_pool_dispatcher
                .read()
                .switch_staking_delegation_pool(
                    :to_staker,
                    :to_pool,
                    switched_amount: amount,
                    data: serialized_data.span(),
                    identifier: pool_member.into(),
                );

            // Emit event.
            self
                .emit(
                    Events::SwitchDelegationPool {
                        pool_member, new_delegation_pool: to_pool, amount,
                    },
                );

            pool_member_info.unpool_amount
        }
