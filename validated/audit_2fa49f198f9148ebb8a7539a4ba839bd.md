### Title
Missing `threshold < 2` Validation in `assert_sign_inputs` Allows Signing with Degenerate Threshold — (`File: src/frost/mod.rs`)

### Summary
`assert_sign_inputs`, the shared validation gate for all FROST EdDSA and RedJubjub signing entry points (`sign_v1`, `sign_v2`), omits the lower-bound threshold check (`threshold < 2`) that is explicitly enforced in the keygen validation function `assert_key_invariants`. A caller can pass `threshold = 1` (or `0`) to any FROST signing call, causing the protocol to proceed with a degenerate threshold that is inconsistent with the key material produced during DKG.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both an upper and a lower bound on the threshold: [1](#0-0) 

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

`assert_sign_inputs` in `src/frost/mod.rs`, which gates both `sign_v1` and `sign_v2`, only enforces the upper bound and omits the lower bound entirely: [2](#0-1) 

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 { ... }
    // ...
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge { ... });
    }
    // ← NO check: threshold.value() < 2
    // ...
}
```

Both `sign_v1` and `sign_v2` delegate entirely to `assert_sign_inputs` for their input validation: [3](#0-2) [4](#0-3) 

The same gap exists in the FROST `presign` entry point: [5](#0-4) 

`ReconstructionLowerBound` does not enforce a minimum of 2 at the type level — the runtime check in `assert_key_invariants` is the only guard, and it is absent from the signing path.

---

### Impact Explanation

When `threshold = 1` is passed to `sign_v1` or `sign_v2`:

1. `construct_key_package(threshold=1, ...)` is called inside `do_sign_coordinator_v1`, producing a `KeyPackage` with `min_signers = 1`.
2. FROST's `round2::sign` and `aggregate` operate under `min_signers = 1`, which is inconsistent with the key material produced by DKG (which enforced `threshold ≥ 2`).
3. The Lagrange interpolation used during signing is parameterized by the signing participant set, but the `min_signers` mismatch causes `aggregate()` to accept a subset of shares that is insufficient to reconstruct the correct group secret, producing a signature that fails external verification — or, in edge cases where the coordinator controls the signing package, allows the coordinator to aggregate with only their own share.

This constitutes **corruption of signing outputs**: honest parties complete the protocol and receive a result, but the output is cryptographically unusable or inconsistent with the public key established during DKG.

**Allowed impact matched:** High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

The entry point is a public library API. Any caller — including a malicious participant or a buggy orchestrator — can supply `threshold = 1`. There is no type-level barrier (`ReconstructionLowerBound` accepts any `usize`). The missing check is a single missing `if` branch, directly reachable on every FROST signing invocation.

---

### Recommendation

Add the lower-bound threshold check to `assert_sign_inputs` in `src/frost/mod.rs`, mirroring the check already present in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to the `presign` function in `src/frost/mod.rs` at the analogous location. [6](#0-5) 

---

### Proof of Concept

```rust
// Keygen enforces threshold >= 2 — this would be rejected:
// assert_key_invariants(&participants, me, 1) → ThresholdTooSmall

// But signing accepts threshold = 1 — this passes all validation:
let protocol = sign_v1(
    &participants,   // len = 2, valid
    1usize,          // threshold = 1 — NOT rejected by assert_sign_inputs
    me,
    coordinator,
    keygen_output,   // produced with threshold = 2
    message,
    rng,
).unwrap(); // succeeds, protocol proceeds with degenerate threshold
// → KeyPackage constructed with min_signers = 1
// → aggregate() accepts 1 share, produces signature inconsistent with DKG public key
``` [7](#0-6) [8](#0-7)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
