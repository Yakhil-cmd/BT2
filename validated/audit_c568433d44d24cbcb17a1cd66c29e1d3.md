### Title
Division Before Multiplication in `native_per_pubdata` Computation Causes Pubdata to Be Undercharged - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the native resource cost per pubdata byte (`native_per_pubdata`) is computed by first dividing `pubdata_price` by `native_price` using integer (floor) division, and then multiplying the result by the number of pubdata bytes consumed. This division-before-multiplication pattern causes precision loss: when `pubdata_price < native_price`, the intermediate result truncates to zero, making all pubdata consumption completely free in native resources regardless of how many bytes are written.

---

### Finding Description

In `validation_impl.rs`, the `native_per_pubdata` ratio is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

The same pattern appears in `api/src/helpers.rs`:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

This pre-divided scalar is then used in two downstream charging sites:

1. **Intrinsic pubdata overhead** (in `create_resources_for_tx`):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ```

2. **Post-execution pubdata charge** (in `get_resources_to_charge_for_pubdata`):
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ```

The mathematically correct computation is:

```
native_cost = current_pubdata_spent * pubdata_price / native_price
```

But the code computes:

```
native_cost = current_pubdata_spent * (pubdata_price / native_price)
```

When `pubdata_price < native_price`, integer division yields `native_per_pubdata = 0`, so `native_cost = 0` regardless of how much pubdata is consumed. The correct value for large `current_pubdata_spent` is non-zero. For example, with `pubdata_price = 5`, `native_price = 10`, `current_pubdata_spent = 1000`:

- **Correct**: `1000 * 5 / 10 = 500` native units charged
- **Actual**: `1000 * (5 / 10) = 1000 * 0 = 0` native units charged

---

### Impact Explanation

When `pubdata_price < native_price` (a valid operator configuration when L1 data is cheap relative to proving cost), `native_per_pubdata` truncates to zero. The consequence is:

- `check_enough_resources_for_pubdata` always returns `enough = true` because the computed cost is 0.
- Transactions can write an unbounded number of storage slots (up to the block pubdata limit) without consuming any native resource budget for pubdata.
- The operator bears the L1 data availability cost without receiving compensation through native resource charging.
- This is a resource accounting bug: the protocol's economic invariant that pubdata consumption is paid for in native resources is violated.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic operator configuration. The operator sets `pubdata_price` to reflect L1 calldata/blob costs and `native_price` to reflect proving costs. During periods of low L1 gas prices or high proving costs, `pubdata_price < native_price` is plausible. Any unprivileged transaction sender can exploit this by submitting transactions that write many storage slots, paying zero native cost for the pubdata they generate.

---

### Recommendation

Replace the pre-divided scalar with a full-precision computation at each charging site. Instead of:

```rust
let native_per_pubdata = pubdata_price / native_price;
// ...
let native = current_pubdata_spent * native_per_pubdata;
```

Compute:

```rust
let native = current_pubdata_spent * pubdata_price / native_price;
```

This ensures multiplication is performed before division, preserving precision for all values of `pubdata_price` and `native_price`.
