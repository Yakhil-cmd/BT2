### Title
Lack of Access Control in `SetContract` for Existing Contract Updates — (`fvm/environment/contract_updater.go`)

### Summary
The `SetContract` function in the FVM contract updater enforces the authorized-deployers list only for **new** contract deployments. When a contract already exists at the target address, the authorization check is skipped entirely and the update is unconditionally accepted. Any account owner holding the `UpdateContract` Cadence entitlement can overwrite their deployed contract code regardless of whether they remain on the authorized-deployers list, bypassing the intended deployment-restriction mechanism.

### Finding Description
`SetContract` at `fvm/environment/contract_updater.go` lines 374–403 reads:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}
// no authorization check when exists == true
updater.draftUpdates[location] = ContractUpdate{...}
``` [1](#0-0) 

The guard `!exists && !updater.isAuthorizedForDeployment(signingAccounts)` is only evaluated when the contract does **not** yet exist. When `exists == true`, the entire authorization branch is skipped. `isAuthorizedForDeployment` consults the on-chain authorized-deployers list stored in the service account: [2](#0-1) 

This list is the sole FVM-level gate controlling who may modify contracts when `RestrictedDeploymentEnabled()` is true. Because the gate is absent for updates, an account owner who was removed from the authorized-deployers list retains the ability to overwrite their contract code indefinitely.

The behavior is confirmed by a production test that explicitly asserts contract updates succeed without service-account authorization: [3](#0-2) 

The attacker-controlled entry path is a standard user transaction that calls `signer.contracts.update(name:, code:)` with the `UpdateContract` Cadence entitlement. The Cadence runtime grants this entitlement to any account key-holder; no privileged access is required beyond owning the account.

### Impact Explanation
When the network operator restricts contract deployment to a curated list of addresses (the default production posture), removing an address from that list is the expected mechanism to revoke their ability to modify contracts. Because `SetContract` skips the authorization check for updates, the revocation is ineffective: the removed party can still push arbitrary new bytecode to their existing contract. A malicious actor can silently alter contract logic — for example, redirecting token transfers, draining escrow balances, or inserting backdoors — affecting every user who interacts with that contract after the update. The impact is unauthorized on-chain state mutation and potential asset loss for users of the affected contract.

### Likelihood Explanation
The preconditions are realistic on a permissioned deployment:
1. An address is added to the authorized-deployers list and deploys a contract.
2. The operator later removes that address from the list (e.g., after a security incident or partnership end).
3. The address still controls its account keys and can submit transactions.

Step 3 is the normal state unless the operator also explicitly revokes the account's keys — an additional, non-obvious remediation step that the authorized-deployers mechanism does not require or document.

### Recommendation
Apply the `isAuthorizedForDeployment` check unconditionally in `SetContract`, regardless of whether the contract already exists:

```go
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("updating contract failed: ...")
}
```

If the intended design is to allow unrestricted updates by account owners, the authorized-deployers list documentation and operator guidance must explicitly state that it controls only initial deployments, so operators do not rely on it as a revocation mechanism.

### Proof of Concept
1. Bootstrap a Flow network with `WithContractDeploymentRestricted(true)`.
2. Add address `A` to the authorized-deployers list; deploy contract `Foo` from `A`.
3. Remove `A` from the authorized-deployers list via the service account.
4. Submit a transaction signed only by `A`:
   ```cadence
   transaction {
     prepare(signer: auth(UpdateContract) &Account) {
       signer.contracts.update(name: "Foo", code: "<malicious bytecode>".decodeHex())
     }
   }
   ```
5. The transaction succeeds and `Foo`'s bytecode is replaced, as confirmed by the existing test `"account update with update code succeeds if not signed by service account"` at `fvm/fvm_blockcontext_test.go:563`. [4](#0-3) [5](#0-4)

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}
