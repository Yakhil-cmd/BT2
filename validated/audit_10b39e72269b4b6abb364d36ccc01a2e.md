### Title
Cross-Network Replay of `DelegateAction` Signatures Due to Missing Chain Domain Separator - (`File: core/primitives/src/action/delegate.rs`)

---

### Summary

`DelegateAction` (NEP-366 meta transactions) and `DelegateActionV2` (NEP-611 gas-key variant) are signed over a payload that contains no chain-specific identifier — no genesis hash, no chain ID, no network name. The `SignableMessage` discriminant encodes only a NEP number. A signature produced on mainnet is cryptographically valid on testnet (or any other NEAR network) whenever the same account and key exist on both networks and the nonce and `max_block_height` constraints are satisfiable on the target network.

---

### Finding Description

`DelegateAction` is the struct a user signs to authorize a relayer to submit actions on their behalf. Its signed payload is:

```
hash( borsh( MessageDiscriminant(NEP_366) || DelegateAction { sender_id, receiver_id, actions, nonce, max_block_height, public_key } ) )
``` [1](#0-0) 

The `MessageDiscriminant` is derived solely from the NEP number (366 for V1, 611 for V2): [2](#0-1) 

Neither the discriminant nor the `DelegateAction` payload contains any chain-specific field. Compare this to a regular `SignedTransaction`, which includes a `block_hash` that is validated as an ancestor of the current chain's canonical head: [3](#0-2) 

The `block_hash` in `SignedTransaction` is checked by `check_transaction_validity_period`, which walks the chain store to confirm the hash belongs to the current chain: [4](#0-3) 

`DelegateAction` has no equivalent field. Its only temporal guard is `max_block_height`, a plain integer that is identical across all NEAR networks at the same block height: [5](#0-4) 

The runtime checks `max_block_height` and the nonce, but neither check is chain-specific: [6](#0-5) 

`DelegateActionV2` (carried by `Action::DelegateV2`) has the same structural gap — its `get_nep461_hash` uses `SignableMessageType::DelegateActionV2` (NEP 611), still with no chain binding: [7](#0-6) 

---

### Impact Explanation

An attacker (malicious relayer, or any party who obtains a signed `DelegateAction` from the mempool or off-chain) can submit the same signed payload to a different NEAR network. If the sender's account exists on both networks with the same key and a compatible nonce state, the inner actions execute on the unintended network. Inner actions can include `Transfer`, `AddKey`, `DeleteKey`, `FunctionCall`, `DeleteAccount`, and `DeployContract`. The exact corrupted values are: unauthorized balance deductions from `sender_id` on the target network, unauthorized key additions or deletions on `sender_id`'s account on the target network, and unauthorized contract calls or deployments on `receiver_id` on the target network.

---

### Likelihood Explanation

The preconditions are:
1. The same account ID exists on both networks (common for named accounts like `alice.near` / `alice.testnet` if the user controls both, or for implicit accounts derived from the same key).
2. The same access key is registered on both networks (common for users who set up accounts identically on mainnet and testnet).
3. The nonce in the `DelegateAction` is greater than the current access key nonce on the target network (likely if the target network has lower activity).
4. `max_block_height` has not been exceeded on the target network (trivially satisfiable if the user sets a generous expiry).

Relayers are explicitly untrusted in the NEP-366 model; the design assumes the user's signature is the only authorization. A malicious relayer is the primary threat actor and requires no special privilege.

---

### Recommendation

Include a chain-specific domain separator in the `DelegateAction` signed payload. The canonical approach is to add the genesis block hash (or a dedicated `chain_id` field) to `DelegateAction` and `DelegateActionV2`, and incorporate it into `get_nep461_hash`. This mirrors how `SignedTransaction` binds to a specific chain via `block_hash`. Alternatively, the `SignableMessage` scheme could be extended to include a network discriminant alongside the NEP number, so that the same NEP-366 action signed on mainnet produces a different hash than on testnet.

---

### Proof of Concept

**Setup:** Alice has account `alice.near` on mainnet and `alice.near` on testnet, both using the same ed25519 key pair. Alice's access key nonce on testnet is 0.

1. Alice signs a `DelegateAction` on mainnet authorizing a transfer of 10 NEAR to Bob, with `nonce = 1` and `max_block_height = 200_000_000` (far future). She hands the `SignedDelegateAction` to a relayer.

2. The relayer (or any observer who extracted the signed payload from the mainnet mempool) constructs a new `SignedTransaction` wrapping the same `SignedDelegateAction`, using a testnet `block_hash` and the relayer's own testnet key for the outer transaction signature.

3. The relayer submits this to a testnet node. The outer transaction passes `check_transaction_validity_period` because the outer `block_hash` is a valid testnet block. The inner `SignedDelegateAction.verify()` passes because the signature is over `hash(NEP_366_discriminant || DelegateAction{...})` — identical bytes on both networks.

4. `validate_delegate_action_key` checks: nonce `1 > 0` ✓, `max_block_height` not exceeded ✓, key exists ✓. The inner transfer executes on testnet, draining Alice's testnet balance without her authorization. [8](#0-7) [9](#0-8)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

pub fn check_transaction_validity_period(
    chain_store: &ChainStoreAdapter,
    prev_block_header: &BlockHeader,
    base_block_hash: &CryptoHash,
    transaction_validity_period: BlockHeightDelta,
) -> Result<(), InvalidTxError> {
    let base_header =
        chain_store.get_block_header(base_block_hash).map_err(|_| InvalidTxError::Expired)?;

    metrics::CHAIN_VALIDITY_PERIOD_CHECK_DELAY
        .observe(prev_block_header.height().saturating_sub(base_header.height()) as f64);

    // First check the distance between blocks
    if prev_block_header.height() > base_header.height() + transaction_validity_period {
        return Err(InvalidTxError::Expired);
    }

    // Then check if there is a path between the blocks (`base` is an ancestor of `prev`)
    validity_period_validate_is_ancestor(&base_header, prev_block_header, chain_store)
}

fn validate_delegate_action_key(

                    account_id: sender_id.clone(),
                    public_key: public_key.clone().into(),
                },
            )
            .into());
            return Ok(());
        }
    };
