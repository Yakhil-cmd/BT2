### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` Allows Threshold-1 Signing — (`src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate only the **upper** bound of the `threshold` parameter (`threshold > participants.len()`), but omit the **lower** bound check (`threshold < 2`). Because `ReconstructionLowerBound` is a plain `usize` newtype with no enforced minimum, a caller can pass `threshold = 1` (or `0`) and the library accepts it without error. With `threshold = 1`, a single participant acting as coordinator can reconstruct a valid FROST signature from just one signature share, bypassing the quorum requirement entirely.

---

### Finding Description

`ReconstructionLowerBound` in `src/thresholds.rs` is a transparent `usize` wrapper:

```rust
pub struct ReconstructionLowerBound(usize);
```

It accepts any value, including `0` and `1`. [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` performs only an upper-bound check:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

There is no corresponding `threshold < 2` guard. [2](#0-1) 

The same omission exists in the FROST `presign` entry point: [3](#0-2) 

By contrast, `assert_key_invariants` in `src/dkg.rs` — the DKG entry point — correctly enforces both bounds:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [4](#0-3) 

`validate_triple_inputs` for OT-based triple generation also enforces the lower bound: [5](#0-4) 

The signing path is therefore the only entry point that lacks this guard.

---

### Impact Explanation

**Critical — Unauthorized creation of a valid threshold signature.**

In FROST, the coordinator collects signature shares and combines them via Lagrange interpolation weighted by the threshold. When `threshold = 1`, the Lagrange coefficient for the single chosen participant collapses to `1`, and the coordinator needs only **one** signature share to produce a complete, verifiable signature over any message. This allows a malicious coordinator to:

1. Initiate a signing session with `threshold = 1`.
2. Collect a single signature share from any one participant.
3. Produce a cryptographically valid FROST signature without the quorum of honest parties that the key was generated under.

The resulting signature is valid under the group public key, so verifiers cannot distinguish it from a legitimately threshold-produced signature.

---

### Likelihood Explanation

The FROST signing API (`assert_sign_inputs`, `presign`) is a public library entry point. Any caller — including a malicious coordinator — supplies the `threshold` argument directly. No prior protocol step enforces a minimum of 2 on this value before it reaches the signing path. The `ReconstructionLowerBound` type provides no compile-time or runtime protection. The gap is reachable with a single well-formed API call.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already applies, inside both `assert_sign_inputs` and the FROST `presign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Alternatively, enforce the invariant at the type level by making `ReconstructionLowerBound::new` a fallible constructor that rejects values below 2, eliminating the need for repeated call-site checks across all protocol entry points. [6](#0-5) 

---

### Proof of Concept

```
1. Run DKG with participants = [P1, P2, P3], threshold = 2.
   → Each party holds a valid (2-of-3) key share.

2. Malicious coordinator calls assert_sign_inputs(
       participants = [P1, P2, P3],
       threshold    = 1,          // ← accepted, no lower-bound check
       me           = coordinator,
       coordinator  = coordinator,
   ) → Ok(participants)           // no error returned

3. Coordinator initiates FROST sign_v1 / sign_v2 with threshold = 1.

4. Coordinator receives the single signature share from P1.

5. Lagrange interpolation with threshold=1 uses coefficient λ_P1 = 1.
   Coordinator computes z = z_P1 and assembles signature (R, z).

6. Signature verifies successfully under the group public key,
   despite only 1 of the required 2 parties participating.
``` [7](#0-6) [8](#0-7)
