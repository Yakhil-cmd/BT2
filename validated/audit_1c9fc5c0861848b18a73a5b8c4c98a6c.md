### Title
Unbounded `pool_member_epoch_balance` Trace Causes Unbounded Gas Consumption in `claim_rewards` - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` trace for a pool member without any bound. A pool member who repeatedly changes their balance in different epochs (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) across many epochs will grow this trace without limit. When `claim_rewards` is eventually called, the loop over the trace can consume unbounded gas, permanently bricking the pool member's ability to claim rewards.

---

### Finding Description

The `pool_member_epoch_balance` storage structure is a `PoolMemberBalanceTrace` — an append-only Vec of checkpoints. Every time a pool member changes their balance in a **new epoch** (i.e., `current_epoch + K` differs from the last checkpoint's key), a new entry is pushed to the trace. [1](#0-0) 

The `set_member_balance` function, called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`, inserts into this trace at `current_epoch + K`: [2](#0-1) 

When `claim_rewards` is called, it invokes `calculate_rewards`, which loops over **all** entries in the trace from `entry_to_claim_from` up to the current epoch: [3](#0-2) 

The code itself acknowledges this: **"This loop is unbounded but unlikely to exceed gas limits."** [4](#0-3) 

The `entry_to_claim_from` pointer is advanced and saved after each `claim_rewards` call, so regular claimers are protected. However, a pool member who **never claims rewards** while repeatedly changing their balance each epoch will accumulate one new trace entry per epoch. After N epochs without claiming, `claim_rewards` must iterate over N entries in a single transaction.

The `claim_rewards` function in `pool.cairo` is the only entry point that advances `entry_to_claim_from`: [5](#0-4) 

There is no mechanism to partially claim or to prune the trace. Once the trace is large enough to exceed the Starknet transaction gas limit, `claim_rewards` will always revert, permanently freezing the pool member's accumulated yield.

---

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

A pool member who accumulates a sufficiently large `pool_member_epoch_balance` trace (by changing balance in many distinct epochs without ever claiming) will find that `claim_rewards` always reverts due to out-of-gas. Their unclaimed rewards are permanently locked in the pool contract with no recovery path, since there is no partial-claim or trace-pruning mechanism.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any pool member can trigger this by calling `add_to_delegation_pool` or `exit_delegation_pool_intent` in a different epoch each time, without ever calling `claim_rewards`.
- Starknet epochs are relatively short (the protocol uses `K`-epoch windows), so a long-lived delegator who simply never claims rewards will naturally accumulate trace entries over time.
- The attacker does not need any privileged role — `add_to_delegation_pool` is callable by the pool member or their reward address, and `exit_delegation_pool_intent` is callable by the pool member.
- The protocol itself acknowledges the risk in a comment but dismisses it as "unlikely," which is not a security guarantee. [4](#0-3) 

---

### Recommendation

1. **Enforce a maximum trace length** per pool member (e.g., reject balance changes if the trace exceeds a safe bound, or require the member to claim rewards first).
2. **Alternatively**, implement a paginated/partial `claim_rewards` that accepts a `max_entries` parameter, advancing `entry_to_claim_from` by at most that many steps per call, so the member can drain the trace over multiple transactions.
3. **Or**, automatically trigger a `claim_rewards` settlement whenever a balance change would push a new entry into the trace, keeping the trace length bounded at 1–2 entries at all times.

---

### Proof of Concept

1. Pool member Alice delegates to a pool via `enter_delegation_pool`.
2. Alice calls `add_to_delegation_pool` with a small amount in epoch E, epoch E+1, epoch E+2, …, epoch E+N, **never calling `claim_rewards`**.
3. Each call in a new epoch appends one entry to `pool_member_epoch_balance` for Alice.
4. After N epochs, Alice's trace has N entries.
5. Alice (or anyone) calls `claim_rewards(pool_member: alice)`.
6. `calculate_rewards` enters the `while entry_to_claim_from < pool_member_trace_length` loop and iterates N times, each iteration reading a storage slot and calling `find_sigma`.
7. For sufficiently large N (bounded by Starknet's per-transaction gas/step limit), the transaction reverts with an out-of-resources error.
8. All subsequent calls to `claim_rewards` for Alice also revert — her accumulated yield is permanently frozen. [6](#0-5) [1](#0-0)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards

            // **Note**: The loop iterates over the balance changes in the pool member's balance

            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
