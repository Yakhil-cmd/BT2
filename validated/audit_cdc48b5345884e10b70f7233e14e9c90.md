### Title
Unchecked `ArrayVec::push` in `apply_to_array_vec` Panics and Halts Batch Finalization When L2→L1 Log Count Exceeds 16384 - (File: `zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

`apply_to_array_vec` in `logs_storage.rs` unconditionally calls `ArrayVec::push` to accumulate L2→L1 log hashes into a fixed-capacity `ArrayVec<Bytes32, 16384>`. The log emission functions (`push_message`, `push_l1_l2_tx_log`) never enforce the `MAX_NUMBER_OF_LOGS = 16_384` limit. If a batch accumulates more than 16384 L2→L1 logs across its constituent blocks, the `push` call panics, halting batch finalization and permanently blocking the batch from being submitted to L1.

---

### Finding Description

`apply_to_array_vec` iterates over every log in the `LogsStorage` list and calls `array_vec.push(log.hash())` with no capacity guard:

```rust
pub fn apply_to_array_vec(&self, array_vec: &mut ArrayVec<Bytes32, 16384>) {
    self.list.iter().for_each(|el| {
        let log: L2ToL1Log = el.into();
        array_vec.push(log.hash())   // ← panics when array_vec is full
    });
}
```

`ArrayVec::push` is documented to panic when the array is at capacity. The caller-side field in `ZKBatchDataKeeper` is declared with the same fixed bound:

```rust
pub logs_storage: ArrayVec<Bytes32, 16384>,
```

`apply_to_array_vec` is called to populate `logs_storage` across all blocks in a batch before `l2_logs_root` computes the Merkle root. Neither of the two log-emission entry points enforces the `MAX_NUMBER_OF_LOGS` constant before appending to the internal list:

```rust
pub fn push_message(...) -> Result<(), SystemError> {
    // no check against MAX_NUMBER_OF_LOGS
    self.list.push(LogContent { ... }, total_pubdata);
    Ok(())
}

pub fn push_l1_l2_tx_log(...) -> Result<(), SystemError> {
    // no check against MAX_NUMBER_OF_LOGS
    self.list.push(LogContent { ... }, total_pubdata);
    Ok(())
}
```

Because a batch spans multiple blocks and each block can independently emit L2→L1 logs, the cumulative count across the batch is unbounded by any explicit guard.

---

### Impact Explanation

When `apply_to_array_vec` is called during batch finalization and the accumulated log count exceeds 16384, the Rust runtime panics. In the proving environment this terminates execution, making it impossible to produce a valid proof or submit the batch
