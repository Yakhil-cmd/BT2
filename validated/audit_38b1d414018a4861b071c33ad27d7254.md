### Title
Missing Verification of Participant Shares in CKD Coordinator Allows Malicious Participant to Corrupt Confidential Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function aggregates `(big_y, big_c)` shares from all participants without any cryptographic verification that those values were honestly computed. A single malicious participant can substitute arbitrary group elements, causing the coordinator to assemble and return a corrupted `CKDOutput` that honest parties accept as legitimate.

---

### Finding Description

The CKD protocol is designed so that each participant computes and sends:

- `norm_big_y = λᵢ · yᵢ · G` (Lagrange-weighted random nonce)
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)` (Lagrange-weighted share of the confidential key blinded by randomness)

The coordinator is then supposed to sum these to obtain `(Y, C)` such that `C − app_sk · Y = msk · H(pk ‖ app_id)` (the confidential derived key).

In `do_ckd_coordinator`, the coordinator receives each participant's output and adds it unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is no zero-knowledge proof, commitment binding, or any other mechanism that forces a participant to use their actual signing share `xᵢ` and a freshly sampled `yᵢ`. The `compute_signature_share` function that honest participants call is never enforced on the receiver side:

```rust
fn compute_signature_share(...) -> Result<(ElementG1, ElementG1), ProtocolError> {
    let private_share = Zeroizing::new(key_pair.private_share);
    let y = Scalar::random(rng);
    ...
    let big_s = hash_point * private_share.to_scalar();
    let big_c = big_s + app_pk * y.0;
    ...
}
``` [2](#0-1) 

A malicious participant simply replaces the honest `do_ckd_participant` send with arbitrary group elements. The coordinator has no way to distinguish a correct share from a forged one.

This is the direct analog of the external report's pattern: a validation check that should gate the aggregation step is entirely absent, so execution proceeds with unverified inputs and produces an incorrect output that honest parties accept.

---

### Impact Explanation

A malicious participant can inject any `(big_y′, big_c′)` pair. The coordinator computes:

```
Y  = Σ_{i≠j} λᵢ yᵢ G  +  big_y′
C  = Σ_{i≠j} λᵢ (xᵢ H + yᵢ A)  +  big_c′
```

When the application unmasks with `app_sk`:

```
C − app_sk · Y  ≠  msk · H(pk ‖ app_id)
```

The coordinator returns this corrupted `CKDOutput` as `Some(ckd_output)`, and the application receives a wrong confidential derived key with no indication of failure.

**Impact: HIGH** — Corruption of CKD output so honest parties accept an unusable or attacker-influenced cryptographic output, matching the allowed impact: *"Corruption of … CKD outputs so honest parties accept inconsistent … or unusable cryptographic outputs."* [3](#0-2) 

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. No leaked keys, no external compromise, and no cryptographic break are required. The attacker only needs to deviate from the honest protocol by sending arbitrary group elements instead of correctly computed shares. The coordinator has no detection mechanism.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof of knowledge of `(yᵢ, xᵢ)` such that:

- `norm_big_y = λᵢ · yᵢ · G`
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)`

where `xᵢ` is consistent with the participant's public verification share from the DKG output. A Schnorr-style sigma protocol over the two-component relation suffices. The coordinator must verify this proof before adding the participant's contribution to the running sum.

---

### Proof of Concept

1. Participant `j` is a malicious actor in a CKD session with `n ≥ 2` honest participants.
2. Instead of calling `compute_signature_share`, participant `j` sends `(big_y′, big_c′) = (G, G)` (or any arbitrary non-zero elements) to the coordinator via `chan.send_private`.
3. The coordinator's loop at lines 50–55 adds `G` to both `norm_big_y` and `norm_big_c` without any check.
4. The final `CKDOutput::new(norm_big_y, norm_big_c)` is returned as `Some(ckd_output)` — the coordinator signals success.
5. The application calls `ckd_output.unmask(app_sk)` and obtains a value that is not `msk · H(pk ‖ app_id)`.
6. All honest parties observe a successful protocol completion but hold an incorrect confidential derived key, with no error surfaced. [4](#0-3) [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

    let (mut norm_big_y, mut norm_big_c) =

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?

        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
