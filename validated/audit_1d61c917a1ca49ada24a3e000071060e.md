### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions with Nonce=1, Admitting Signature-Invalid Transactions into the Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX shortcut `skip_stateful_validations` unconditionally skips the `__validate__` entry-point call (the only place where the account contract verifies the transaction signature) for any Invoke V3 transaction whose nonce equals 1 and whose sender address already appears in the mempool or a recent block. Because the mempool also supports fee-escalation replacement, an attacker who observes a legitimate `deploy_account` transaction in the mempool can immediately submit a competing Invoke nonce=1 transaction carrying an **arbitrary/invalid signature** at a higher fee. The gateway admits it without signature verification, the fee-escalation logic evicts the legitimate user's Invoke, and the batcher later rejects the attacker's transaction — permanently destroying the victim's first post-deployment invoke.

**`account_tx_in_pool_or_recent_block` check is insufficient**


pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)

This returns `true` for any address that has *any* transaction in the pool or was seen in a recent block — including the deploy_account itself. The attacker only needs to observe the public mempool.

---

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool has passed `__validate__`, i.e., the account contract has verified the signature. This invariant is broken for Invoke nonce=1 transactions when a deploy_account for the same address is in the mempool.

**Concrete harm:**
1. An attacker can permanently destroy a victim's first post-deployment invoke transaction by replacing it with a signature-invalid transaction that will be rejected at execution time.
2. The attacker pays nothing (the invalid tx is rejected before fee transfer).
3. The victim must re-submit and pay again; in a congested mempool the window may be exploited repeatedly.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- The deploy_account + invoke UX flow is explicitly documented and tested as a supported pattern.
- The mempool is public; any observer can detect a deploy_account transaction and immediately race with a nonce=1 Invoke carrying an invalid signature.
- Fee escalation is a standard mempool feature; the attacker only needs to outbid the victim's tip/gas-price by the configured percentage.
- No privileged access is required; the attacker is an ordinary unprivileged user.

---

### Recommendation

1. **Do not skip `__validate__` for the UX shortcut.** Instead, run `__validate__` against a *simulated* post-deployment state (the account class is known from the deploy_account transaction already in the mempool) so the signature can be verified before admission.
2. Alternatively, restrict the skip to cases where the gateway can cryptographically confirm the signature is consistent with the account class declared in the pending deploy_account (e.g., by running a lightweight signature check using the known class hash).
3. At minimum, if the skip must remain, document it as a known admission of unverified transactions and prevent fee-escalation replacement of a validated nonce=1 transaction by an unvalidated one.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
        &self,

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;
```
