### Title
Missing Zero-Address Validation in `change_reward_address` Allows Staker/Pool Member to Freeze Their Own Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both `change_reward_address` in the staking contract and `change_reward_address` in the pool contract accept a `ContractAddress` parameter with no zero-address guard. A staker or pool member who passes `0` as the new reward address will have all subsequent reward transfers directed to the zero address, causing them to revert (OpenZeppelin Cairo ERC20 rejects transfers to address zero) and blocking the staker from completing `unstake_action` until the address is corrected.

### Finding Description
`change_reward_address` in `src/staking/staking.cairo` (lines 517–540) validates only that the supplied address is not a registered token address (`REWARD_ADDRESS_IS_TOKEN`). There is no `assert!(reward_address.is_non_zero(), …)` guard. [1](#0-0) 

The identical gap exists in `src/pool/pool.cairo` (lines 505–526), where the only check is that the address is not the pool's token address. [2](#0-1) 

Both functions are declared in the public interface and are callable by any registered staker or pool member respectively. [3](#0-2) [4](#0-3) 

### Impact Explanation
When `reward_address` is zero, every downstream call that transfers rewards to that address will revert. Critically, `unstake_action` calls `send_rewards_to_staker` before returning the principal; if that transfer reverts, the entire `unstake_action` transaction reverts, leaving the staker's funds locked in the exit-window state until the reward address is corrected. [5](#0-4) 

This matches the allowed impact: **Temporary freezing of unclaimed yield / funds** (High).

### Likelihood Explanation
`change_reward_address` is a routine self-service function called by ordinary stakers and pool members. An accidental zero value (e.g., an uninitialized variable in a script or front-end bug) is a realistic user-error scenario, directly analogous to the original report's "Alice calls `setOwner` with an uninitialized value." The function is reachable by any unprivileged staker or delegator with no preconditions beyond being registered.

### Recommendation
Add a non-zero guard at the top of both `change_reward_address` implementations:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the pattern already used in `add_token`: [6](#0-5) 

### Proof of Concept
1. A staker is registered and has accrued rewards.
2. The staker calls `change_reward_address(0)` — the call succeeds because no zero-address check exists.
3. The staker calls `unstake_intent()` and waits out the exit window.
4. The staker calls `unstake_action(staker_address)`.
5. Inside `unstake_action`, `send_rewards_to_staker` attempts `token.transfer(recipient: 0, …)`.
6. The OpenZeppelin ERC20 implementation reverts on a transfer to address zero.
7. The entire `unstake_action` transaction reverts; the staker's principal remains locked in the contract until they call `change_reward_address` with a valid address and retry.

        fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            assert!(
                !self.does_token_exist(token_address: reward_address),
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let staker_address = get_caller_address();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let old_address = staker_info.reward_address;

            // Update reward_address and commit to storage.
            staker_info.reward_address = reward_address;
            self.write_staker_info(:staker_address, :staker_info);

            // Emit event.
            self
                .emit(
                    Events::StakerRewardAddressChanged {
                        staker_address, new_address: reward_address, old_address,
                    },
                );
        }
