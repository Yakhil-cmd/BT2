### Title
Unbounded `pool_member_epoch_balance` Trace Causes Permanent Freezing of Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over a pool member's `pool_member_epoch_balance` trace in an explicitly unbounded loop. The trace grows by one checkpoint per epoch in which the pool member changes their balance. There is no cap on trace length. A pool member who accumulates enough balance-change epochs without claiming will eventually be unable to call `claim_rewards`, permanently freezing their unclaimed yield.

---

### Finding Description

Each pool member has a per-address `PoolMemberBalanceTrace` stored in `pool_member_epoch_balance`: [1](#0-0) 

The trace is a `Vec<PoolMemberBalanceCheckpoint>` that appends a new entry whenever a balance change occurs in a new epoch: [2](#0-1) [3](#0-2) 

Balance changes are triggered by `enter_delegation_pool`, `add_to_delegation_pool`, `exit_delegation_pool_intent`, and `enter_delegation_pool_from_staking_contract` — all of which call `set_member_balance` or `increase_member_balance`, which in turn call `insert` on the trace.

`claim_rewards` calls `calculate_rewards`, which iterates over every trace entry from `entry_to_claim_from` to the current trace length: [4](#0-3) 

The code itself acknowledges the risk with the comment:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. **This loop is unbounded but unlikely to exceed gas limits.** [5](#0-4) 

`entry_to_claim_from` is only advanced when `claim_rewards` succeeds: [6](#0-5) 

If a pool member accumulates N epochs of balance changes without claiming, the loop must iterate N times on the next `claim_rewards` call. Once N is large enough to exhaust the Starknet transaction gas limit, `claim_rewards` reverts on every attempt, permanently locking the accrued rewards.

The same unbounded loop is also triggered by the view function `pool_member_info_v1`: [7](#0-6) 

---

### Impact Explanation

A pool member whose `pool_member_epoch_balance` trace grows beyond the gas-iteration threshold can never successfully call `claim_rewards`. Their accrued STRK rewards are permanently frozen in the pool contract with no recovery path, because:

- `entry_to_claim_from` cannot advance past the gas-exhaustion point.
- There is no partial-claim or pagination mechanism.
- There is no administrative function to reset or truncate the trace.

This matches the allowed High impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

A pool member who is active (regularly adjusting delegation amounts) but infrequent in claiming rewards will naturally grow their trace. One new checkpoint is appended per epoch in which a balance change occurs. Starknet epochs are short (measured in blocks), so a delegator who adjusts their position every epoch for an extended period — a realistic pattern for yield-optimizing delegators — will accumulate a large trace. No adversarial intent is required; the freeze can occur organically. An adversary could also deliberately trigger this on their own account to make their rewards unclaimable (e.g., to avoid on-chain evidence of yield receipt).

---

### Recommendation

1. **Enforce a maximum trace length**: Add a constant `MAX_BALANCE_TRACE_LENGTH` and assert it is not exceeded in `set_member_balance` / `increase_member_balance`.
2. **Paginate `calculate_rewards`**: Allow partial reward calculation up to a configurable number of trace entries per call, advancing `entry_to_claim_from` incrementally across multiple transactions.
3. **Force-claim on balance change**: Automatically settle pending rewards before inserting a new trace checkpoint, keeping the unclaimed window to at most one entry.

---

### Proof of Concept

1. Pool member `A` calls `add_to_delegation_pool` once per epoch for `N` consecutive epochs without ever calling `claim_rewards`. Each call appends one entry to `pool_member_epoch_balance[A]` (since the epoch key differs each time).
2. After `N` epochs, `pool_member_epoch_balance[A].length() == N` and `pool_member_info[A].entry_to_claim_from == 0`.
3. `A` calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates `N` times, each iteration reading a storage slot and calling `find_sigma`. For sufficiently large `N`, the transaction exceeds the Starknet gas limit and reverts.
4. Every subsequent `claim_rewards` call also reverts because `entry_to_claim_from` was never updated. `A`'s accumulated rewards are permanently frozen.

The exact threshold for `N` depends on Starknet's per-transaction gas limit and the per-iteration cost (storage reads + arithmetic in `find_sigma`), but the code's own comment confirms the loop is unbounded and the risk is acknowledged by the developers. [8](#0-7) [9](#0-8)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    /// Inserts a (`key`, `value`) pair into a Trace so that it is stored as the checkpoint.
    /// This is done by either inserting a new checkpoint, or updating the last one.
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
