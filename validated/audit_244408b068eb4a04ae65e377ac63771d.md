### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last reward claim. Because any pool member can grow this trace by making one balance-changing call per epoch (e.g., `add_to_delegation_pool` or `exit_delegation_pool_intent`) without claiming rewards, a sufficiently active delegator will eventually cause every future `claim_rewards` call to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over all entries in `pool_member_epoch_balance` from `entry_to_claim_from` up to the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

The trace grows via `set_member_balance` → `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` logic in `src/pool/pool_member_balance_trace/trace.cairo` only updates the last entry when the key matches; otherwise it **appends** a new checkpoint. Since the key is always `current_epoch + K`, each call made in a distinct epoch appends a new entry.

The following public entrypoints each call `set_member_balance` or `increase_member_balance`, appending a new trace entry when called in a new epoch:

- `add_to_delegation_pool` (line 242 → `increase_member_balance`)
- `exit_delegation_pool_intent` (line 278 → `set_member_balance`)
- `enter_delegation_pool_from_staking_contract` (line 456/464 → `increase_member_balance` / `set_member_balance`)

The `entry_to_claim_from` field in `pool_member_info` is only advanced on a **successful** `claim_rewards` call (line 358). If a pool member accumulates N balance-change entries without successfully claiming, the next `claim_rewards` call must iterate all N entries. Once N is large enough to exceed the gas limit, every subsequent `claim_rewards` call reverts, and the member's accrued yield is permanently inaccessible.

`calculate_rewards` is called from two places:
- `claim_rewards` (line 350) — the only way to withdraw yield
- `pool_member_info_v1` (line 532) — a view function

There is no mechanism to partially advance `entry_to_claim_from` or to split the reward computation across multiple transactions.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the pool member's `pool_member_epoch_balance` trace accumulates enough entries between two consecutive successful claims, every future `claim_rewards` invocation will exceed the gas limit and revert. The member's accrued STRK rewards are permanently locked in the pool contract with no recovery path, because:

1. `entry_to_claim_from` only advances on a successful claim.
2. There is no partial-claim or checkpoint-reset mechanism.
3. The pool contract holds the reward tokens and can only release them via `claim_rewards`.

---

### Likelihood Explanation

**Medium.** The protocol is designed for long-term participation. A delegator who actively manages their position — calling `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch across many epochs without claiming — will naturally grow their trace. The developers themselves acknowledge the risk in the comment at line 857–858: *"This loop is unbounded but unlikely to exceed gas limits."* The word "unlikely" confirms the risk is known but unmitigated. On a live network running for months or years, active delegators will accumulate hundreds to thousands of trace entries, making gas exhaustion realistic.

---

### Recommendation

1. **Paginated claiming**: Allow `claim_rewards` to accept a `max_entries` parameter and advance `entry_to_claim_from` by at most that many steps per call, emitting partial rewards each time.
2. **Trace compaction**: After a successful `claim_rewards`, truncate or compact the already-processed prefix of `pool_member_epoch_balance` so the trace length is bounded by the number of balance changes since the last claim.
3. **Enforce a maximum trace length**: Reject balance-changing calls (e.g., `add_to_delegation_pool`) if the unclaimed trace segment exceeds a safe threshold, forcing the member to claim first.

---

### Proof of Concept

1. Delegator enters pool with `enter_delegation_pool` at epoch E.
2. For each epoch E+1, E+2, …, E+N, delegator calls `add_to_delegation_pool(amount: 1)` with a minimal amount. Each call in a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After N epochs, `pool_member_epoch_balance` has N entries.
5. Delegator calls `claim_rewards`. Inside `calculate_rewards`, the `while entry_to_claim_from < pool_member_trace_length` loop iterates N times, each iteration reading storage and calling `find_sigma`.
6. For sufficiently large N (dependent on Starknet's per-transaction gas cap), the transaction reverts with an out-of-gas error.
7. Every subsequent `claim_rewards` attempt also reverts. The delegator's accumulated STRK yield is permanently frozen.

```text
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
```
