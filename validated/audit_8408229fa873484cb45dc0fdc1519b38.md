### Title
Missing Participant-Set Validation Against `TriplePub.participants` in OT-Based ECDSA Presigning Enables Denial of Presignature Generation — (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA pipeline explicitly omits the check that the presigning participant set is a subset of the triple generation participant set recorded in `TriplePub.participants`. The `TriplePub` struct carries a `participants` field precisely to enable this binding, but it is never consulted during presigning initialization. A malicious caller or coordinator who controls `PresignArguments` construction can supply triples generated by a different participant set than the one used for presigning, causing the Lagrange linearization to be algebraically inconsistent and the presigning protocol to abort for all honest participants.

---

### Finding Description

The OT-based ECDSA presigning protocol requires, by specification, that both Beaver triples be generated by the same participant set $\mathcal{P}_0$ that is a superset of the presigning set $\mathcal{P}_1$:

> "Two multiplicative (Beaver) triples private shares $(a_i, b_i, c_i)$ and $(k_i, d_i, e_i)$ **generated by the same set of parties $\mathcal{P}_0$**" [1](#0-0) 

The `TriplePub` struct records the participant set that generated each triple:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    pub threshold: ReconstructionLowerBound,
}
``` [2](#0-1) 

However, the `presign()` initialization function contains an explicit acknowledgment that this check is omitted:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [3](#0-2) 

The only validation performed on the triples is a threshold equality check:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [4](#0-3) 

There is no check that `args.triple0.1.participants` or `args.triple1.1.participants` contains the presigning participant set. The `PresignArguments` struct is a plain public struct with no invariant enforcement: [5](#0-4) 

**Root cause:** The `TriplePub.participants` field is populated during triple generation and stored in the output, but is never read or validated in `presign()`. This is the direct analog of the MarinateV2 tokenId bug: the "type" of the resource (threshold) is checked, but the specific "identity" (which participants generated it) is not.

---

### Impact Explanation

When a presigning participant set $\mathcal{P}_1$ is used with triples generated by a different set $\mathcal{P}_0$ where $\mathcal{P}_1 \not\subseteq \mathcal{P}_0$, the Lagrange linearization step computes:

$$k'_i = \lambda_i(\mathcal{P}_1) \cdot k_i$$

where $k_i = f_k(P_i)$ is a polynomial evaluation from the triple generation over $\mathcal{P}_0$. Because $\mathcal{P}_1 \neq \mathcal{P}_0$, the Lagrange coefficients $\lambda_i(\mathcal{P}_1)$ do not correctly reconstruct the secret $k = f_k(0)$. The resulting $\alpha$ and $\beta$ values are algebraically inconsistent, causing the protocol to abort at the group-element consistency check:

```rust
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
    || (ProjectivePoint::GENERATOR * beta != big_x + big_b)
{
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of additive triple phase.".to_string(),
    ));
}
``` [6](#0-5) 

All honest participants in the presigning session receive this error and cannot produce a presignature. Since the triples are consumed (one-time use), the session cannot be retried with the same triples. A malicious coordinator who repeatedly supplies mismatched triples causes **permanent denial of presigning** for honest parties.

**Matched allowed impact:** High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs / Permanent denial of signing for honest parties under valid protocol inputs.

---

### Likelihood Explanation

The attack requires only the ability to construct a `PresignArguments` value — a public struct with no access control. Any party acting as a coordinator or orchestrator in the signing pipeline can supply `triple0` from one triple generation session (participant set $\mathcal{P}_A$) and `triple1` from another session (participant set $\mathcal{P}_B \neq \mathcal{P}_A$), or supply triples generated by a participant set that does not include all presigning participants. The `TriplePub.participants` field is serializable and deserializable, so a malicious party can also forge or swap the `participants` metadata to pass any future threshold check while providing shares from a mismatched session. The comment in the code explicitly acknowledges the omission, confirming this is a known gap rather than an intentional design decision.

---

### Recommendation

In `presign()`, after constructing the `ParticipantList`, validate that the presigning participant set is a subset of both `args.triple0.1.participants` and `args.triple1.1.participants`:

```rust
// Validate that the presigning set is a subset of each triple's generation set
for p in participants.participants() {
    if !args.triple0.1.participants.contains(p) {
        return Err(InitializationError::BadParameters(
            "presigning participant was not present for triple0 generation".to_string(),
        ));
    }
    if !args.triple1.1.participants.contains(p) {
        return Err(InitializationError::BadParameters(
            "presigning participant was not present for triple1 generation".to_string(),
        ));
    }
}
```

Additionally, validate that both triples were generated by the same participant set (as required by the specification), and remove the `NOTE` comment acknowledging the omission once the check is added.

---

### Proof of Concept

1. Run triple generation with participant set $\mathcal{P}_A = \{P_0, P_1, P_2\}$, threshold 2, producing `(triple0_pub_A, triple0_shares_A)`.
2. Run triple generation with participant set $\mathcal{P}_B = \{P_0, P_1, P_3\}$, threshold 2, producing `(triple1_pub_B, triple1_shares_B)`.
3. Run DKG with $\{P_0, P_1, P_2\}$.
4. Call `presign(&[P_0, P_1, P_2], P_0, PresignArguments { triple0: (triple0_shares_A[0], triple0_pub_A), triple1: (triple1_shares_B[0], triple1_pub_B), keygen_out, threshold: 2 })`.
5. The threshold check at line 43 passes (both thresholds are 2). No participant-set check is performed.
6. During `do_presign`, $P_0$'s share of `triple1` is `f_{a,B}(P_0)` — a polynomial evaluation from the $\mathcal{P}_B$ session. The Lagrange coefficient $\lambda_0(\{P_0,P_1,P_2\})$ is applied, but this does not reconstruct $a$ from the $\mathcal{P}_B$ polynomial.
7. The check `alpha * G == K + A` fails, returning `ProtocolError::AssertionFailed("received incorrect shares of additive triple phase.")` for all participants.
8. The presigning session is permanently aborted; the consumed triples cannot be reused. [7](#0-6) [8](#0-7) [5](#0-4)
