### Title
Overly Broad `skip_stateful_validations` Check Admits Invoke Transactions Without Signature Verification - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator uses `account_tx_in_pool_or_recent_block` as a proxy to confirm a `deploy_account` transaction exists before skipping `__validate__` for a nonce=1 invoke. However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from that account in the mempool — not just a `deploy_account`. An attacker can exploit this by submitting an invoke transaction with nonce=1 for any account that has a future-nonce transaction in the pool, causing the gateway to admit the invoke without running the account's `__validate__` entry point (signature verification).

### Finding Description

The `skip_stateful_validations` function is designed to improve UX for the `deploy_account + invoke` pattern: when a user submits both transactions simultaneously, the invoke (nonce=1) would normally fail gateway validation because the account doesn't exist yet. The function skips `__validate__` for the invoke if a `deploy_account` is believed to be pending.

The check used to confirm the `deploy_account` is pending is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

The implementation of `account_tx_in_pool_or_recent_block` is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`self.tx_pool.contains_account(account_address)` returns `true` if the account has **any** transaction in the pool — including future-nonce invoke transactions (nonce ≥ 2) that were admitted when `account_nonce=0` and `max_allowed_nonce_gap ≥ 2`. There is no check that the pooled transaction is specifically a `deploy_account`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };
...
blockifier_validator.validate(account_tx)
``` [3](#0-2) 

With `validate=false`, the blockifier dry-run does **not** call the account's `__validate__` entry point, meaning the transaction's signature is never verified during gateway admission.

The code comment acknowledges the assumption but states it incorrectly:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or **transactions with future nonces that passed validations**."

Future-nonce transactions passing validation does not imply a `deploy_account` exists. An account owner can submit a nonce=2 invoke (with `account_nonce=0` and `max_allowed_nonce_gap ≥ 2`) without ever submitting a `deploy_account`.

### Impact Explanation

An attacker can submit an invoke transaction with nonce=1 for any account that has a future-nonce transaction in the mempool (with `account_nonce=0`), bypassing the gateway's `__validate__` dry-run. The transaction is admitted to the mempool without signature verification. This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Additionally, if the attacker's nonce=1 invoke is admitted and the legitimate owner's nonce=2 tx is already in the pool, the attacker's tx will be sequenced first (lower nonce), executing arbitrary calldata under the account's identity before the account's `__validate__` can reject it at the blockifier level — depending on whether the batcher also propagates `skip_validate`.

### Likelihood Explanation

The precondition — an account with `account_nonce=0` having a future-nonce transaction in the pool — is a normal operational state. The `deploy_account + invoke` UX pattern itself creates this state (the invoke with nonce=1 is in the pool). Any account that submitted a future-nonce tx without a `deploy_account` (allowed by `max_allowed_nonce_gap > 0`) is also vulnerable. No privileged access is required; the attacker only needs to know the target account address (public information).

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the account. The mempool should expose a dedicated `deploy_account_in_pool(address)` query, or the pool should be queried for a transaction with `nonce=0` and type `DeployAccount` for the given address. Alternatively, store a separate set of addresses for which a `deploy_account` is pending.

### Proof of Concept

1. Account `A` has `account_nonce = 0` (not yet deployed on-chain).
2. Account `A`'s owner submits an invoke with `nonce=2` (allowed because `max_allowed_nonce_gap ≥ 2`). This tx passes full gateway validation including `__validate__` and is admitted to the mempool. `tx_pool.contains_account(A)` is now `true`.
3. Attacker submits an invoke with `nonce=1` for account `A` with an **invalid signature**.
4. Gateway `validate_nonce`: `account_nonce=0`, `tx_nonce=1`, within allowed range → passes.
5. Gateway `validate_by_mempool`: no existing nonce=1 tx for `A` → passes.
6. Gateway `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)` returns `true` (because nonce=2 tx is in pool) → returns `true`.
7. Gateway `run_validate_entry_point` with `validate=false` → `__validate__` is **not called**; the invalid signature is never checked.
8. Attacker's invoke is admitted to the mempool despite having an invalid signature.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
