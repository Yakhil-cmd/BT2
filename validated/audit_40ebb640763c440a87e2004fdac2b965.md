### Title
NFT Metadata Spoofing in Burn Confirmation Dialog Enables Misidentification of Asset Being Destroyed - (`packages/gui/src/components/nfts/NFTBurnDialog.tsx`)

### Summary

The burn confirmation dialog renders the NFT's display name exclusively from remotely-fetched metadata. When an NFT is minted without a `metadataHash`, the checksum guard in `useFetchAndProcessMetadata` is bypassed, allowing an attacker-controlled metadata server to serve an arbitrary `name` field. The dialog prominently displays this attacker-controlled name while the actual `nftCoinId` being burned is determined independently from on-chain state. A user who relies on the displayed name to identify the asset can be tricked into burning the wrong NFT.

---

### Finding Description

**Call chain:**

`NFTBurnDialog.renderNFTPreview()` → `NFTSummary(launcherId=nfts[0].launcherId)` → `useNFTMetadata(launcherId)` → `NFTProviderContext` → `useFetchAndProcessMetadata(uri, hash)` → `JSON.parse(metadataString)` → `NFTTitle` → `NFTMetadata(path="name")`

**The checksum guard is conditional on `hash` being truthy:** [1](#0-0) 

```ts
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
```

If an NFT is minted with no `metadataHash` (a valid on-chain state in the Chia NFT1 standard), `hash` is `undefined`/falsy, the guard is skipped entirely, and whatever JSON the metadata server returns is parsed and trusted.

**The burn dialog renders the attacker-controlled `name` prominently:** [2](#0-1) 

```tsx
<Typography variant="h6" fontWeight="bold" noWrap>
  <NFTTitle nftId={nftId} />
</Typography>
```

`NFTTitle` delegates to `NFTMetadata` with `path="name"`: [3](#0-2) 

`NFTMetadata` resolves the value via `get(metadata, path)` and renders it directly: [4](#0-3) 

**The actual burn target is determined separately from the display name:** [5](#0-4) 

```ts
await transferNFT({
  walletId: nfts[0].walletId,
  nftCoinIds: nfts.map((nft: NFTInfo) => nft.nftCoinId),
  targetAddress: destination,
  fee: feeInMojos,
}).unwrap();
```

The `nftCoinId` is taken from on-chain state; the displayed name has no bearing on which coin is spent. This creates a display/action mismatch exploitable via metadata.

**The only identity anchor visible to the user is a truncated NFT ID in small `body2` text:** [6](#0-5) 

The full ID is hidden behind a tooltip. The `h6`-bold name is the dominant identifier.

---

### Impact Explanation

**Bait-and-switch scenario (concrete):**

1. Attacker mints NFT-A with `metadataUri` pointing to an attacker-controlled server and **no `metadataHash`**.
2. Attacker initially serves legitimate-looking metadata (e.g., `name: "Rare Dragon #1"`) to make the NFT appear valuable.
3. Victim purchases or receives NFT-A.
4. Attacker later changes the served JSON to `name: "Worthless Airdrop"`.
5. Victim's gallery now shows NFT-A as "Worthless Airdrop."
6. Victim opens the burn dialog for NFT-A, intending to discard what they believe is a worthless NFT.
7. The dialog prominently displays "Worthless Airdrop" — consistent with the victim's expectation.
8. Victim confirms. NFT-A (which may have had real value at purchase) is irreversibly burned.

**Name-collision scenario (matching the PoC in the question):**

1. Attacker mints NFT-A with metadata `name` set to the exact display name of the victim's legitimate NFT-B.
2. Victim opens burn dialog for NFT-A.
3. Dialog shows NFT-B's name; victim believes they are burning NFT-B.
4. Victim confirms; NFT-A is burned. Victim is confused about which asset was destroyed.

Both scenarios result in irreversible on-chain asset destruction caused by attacker-controlled metadata.

---

### Likelihood Explanation

- NFTs minted without `metadataHash` are common; the Chia NFT1 standard does not require it.
- The attacker only needs to control a web server serving JSON — no cryptographic capability required.
- The attack is fully server-side and can be triggered at any time after the NFT is transferred to the victim.
- The only UI mitigation (truncated NFT ID in small text with full ID behind a tooltip) is easily overlooked by typical users who rely on names and images.

---

### Recommendation

1. **Warn on missing hash:** When `metadataHash` is absent, display a prominent warning in the burn dialog (and gallery) that the NFT's metadata is unverified and may have been altered.
2. **Surface the NFT ID prominently:** Show the full (or less-truncated) NFT ID in the burn confirmation dialog at a readable size, not only in a tooltip.
3. **Consider refusing to render unverified metadata names in destructive dialogs:** Fall back to the raw NFT ID as the display name when no hash is present, preventing attacker-controlled strings from appearing in burn/transfer confirmations.

---

### Proof of Concept

```
1. Start a local Chia simulator.
2. Mint NFT-A:
   - metadataUri: http://attacker.local/meta.json
   - metadataHash: (omitted / null)
3. Serve from attacker.local/meta.json:
   { "name": "NFT-B Display Name", "description": "Legitimate NFT" }
4. Mint NFT-B with name "NFT-B Display Name" and a real metadataHash.
5. Transfer NFT-A to the victim wallet.
6. In the GUI, open the burn dialog for NFT-A.
7. Observe: NFTSummary renders "NFT-B Display Name" in h6 bold.
8. Observe: the actual nftCoinId queued for burn belongs to NFT-A.
9. Confirm burn → NFT-A is irreversibly sent to the burn address.
10. Change attacker.local/meta.json to { "name": "Worthless Airdrop" }
    and repeat to demonstrate the bait-and-switch variant.
```
