### Title
Missing Cryptographic Proof of Correctness for CKD Participant Shares Allows Malicious Participant to Corrupt CKD Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator collects `(big_y, big_c)` shares from each participant and accumulates them without any cryptographic proof that the shares are correctly formed. A malicious participant can send arbitrary group elements, corrupting the final `CKDOutput` that the coordinator and honest parties accept as valid.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator receives a `CKDOutput` from every other participant and blindly adds the two group elements together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The only gate on these messages is `recv_from_others`, which filters by participant list — i.e., it verifies the *sender's identity* (analogous to checking `collection.key`). It does **not** verify that the received `(big_y, big_c)` pair is *cryptographically correct* (analogous to the missing `collection.verified` check).

Each honest participant is supposed to compute:

```rust
let big_y = ElementG1::generator() * y.0;          // Y_i = y_i · G
let big_s  = hash_point * private_share.to_scalar(); // S_i = x_i · H(pk, app_id)
let big_c  = big_s + app_pk * y.0;                  // C_i = S_i + y_i · A
``` [2](#0-1) 

There is no zero-knowledge proof that the submitted `big_c` was formed using the participant's actual private share `x_i` and the same randomness `y_i` used to produce `big_y`. The codebase already contains a discrete-log-equality proof implementation in `src/crypto/proofs/dlogeq.rs` that could enforce this relationship, but it is not used in the CKD protocol. [3](#0-2) 

---

### Impact Explanation

The correct aggregate output satisfies:

```
C = msk · H(pk, app_id) + Y · app_sk
```

so the application can recover the confidential key as `C − Y · app_sk`. If any participant substitutes an arbitrary `(big_y', big_c')`, the coordinator's final `CKDOutput` is shifted by `(big_y' − big_y_correct, big_c' − big_c_correct)`. The coordinator and all honest parties accept this corrupted ciphertext as the protocol output (there is no verification step after aggregation). The application then decrypts a wrong value, permanently producing an unusable or attacker-influenced confidential derived key.

This matches the allowed High impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

Any single participant in the CKD protocol can mount this attack. No special cryptographic capability is required — the attacker simply sends two arbitrary `ElementG1` values instead of the honestly computed ones. The attack is undetectable by the coordinator or any other participant because no proof of correctness is requested or verified.

---

### Recommendation

Require each participant to accompany their `(big_y, big_c)` with a zero-knowledge proof of correct formation. Concretely, the participant must prove:

- `big_y = λ_i · y_i · G` (discrete log of `big_y` in base `G` equals discrete log of `big_y / λ_i`)
- `big_c − big_s_pub = y_i · app_pk` (same `y_i` used in both `big_y` and `big_c`)

where `big_s_pub = λ_i · x_i_pub · H(pk, app_id)` is derivable from the participant's public key share. A discrete-log-equality proof (already present in `src/crypto/proofs/dlogeq.rs`) is the standard primitive for this. The coordinator must verify these proofs before accumulating any share. [4](#0-3) 

---

### Proof of Concept

1. Honest participants P₁ … Pₙ₋₁ and malicious participant Pₘ all join the CKD protocol.
2. Pₘ computes the honest `(norm_big_y, norm_big_c)` but instead sends `(ElementG1::identity(), ElementG1::identity())` (or any arbitrary pair) to the coordinator.
3. `recv_from_others` accepts the message because Pₘ is a valid participant.
4. The coordinator accumulates the corrupted values: `norm_big_y += identity` and `norm_big_c += identity`, shifting the aggregate ciphertext.
5. The coordinator returns a `CKDOutput` that does not satisfy `C = msk · H(pk, app_id) + Y · app_sk`.
6. The application calls `ckd_output.unmask(app_sk)` and obtains a wrong confidential key with no error or indication of failure.
7. No honest party can detect the corruption; the protocol terminates successfully from every participant's perspective. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,

    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {

        norm_big_c += participant_output.big_c();
    }
