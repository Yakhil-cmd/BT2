### Title
Silo Admin Role Cannot Manage Whitelists Despite Being Documented To Do So - (`engine/src/lib.rs`, `engine-types/src/parameters/silo.rs`)

---

### Summary

The `WhitelistKind::Admin` role in Aurora Engine's Silo mode is documented to allow adding new admins and managing whitelist entries, but the implementation restricts all whitelist management functions exclusively to the contract owner. Accounts in the `Admin` whitelist cannot exercise their documented privilege, making the role non-functional for its primary stated purpose.

---

### Finding Description

The `WhitelistKind` enum documents the `Admin` variant as follows:

> "The whitelist of this type is for storing NEAR accounts. Accounts stored in this whitelist have an admin role. **The admin role allows to add new admins and add new entities (`AccountId` and `Address`) to whitelists.** Also, this role allows to deploy of EVM code and submit transactions." [1](#0-0) 

However, every whitelist management entry point in `engine/src/lib.rs` enforces `require_owner_and_running`, which checks that the caller is strictly the `owner_id` stored in `EngineState`. Accounts in the `Admin` whitelist are never consulted:

```rust
pub extern "C" fn add_entry_to_whitelist() {
    let io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_owner_and_running(&state, &io.predecessor_account_id())  // owner only
        ...
    silo::add_entry_to_whitelist(&io, &args);
}
``` [2](#0-1) 

The same owner-only guard is applied to `add_entry_to_whitelist_batch`, `remove_entry_from_whitelist`, `set_whitelist_status`, and `set_whitelists_statuses`: [3](#0-2) [4](#0-3) 

`require_owner_and_running` is defined as a strict equality check against `state.owner_id` — it does not check the `Admin` whitelist at all: [5](#0-4) 

The `Admin` whitelist is only consulted in `is_account_allowed_deploy`, which gates EVM code deployment — not whitelist management: [6](#0-5) 

---

### Impact Explanation

In a Silo deployment with whitelists enabled, the `Account` and `Address` whitelists gate who can submit transactions. If the owner becomes operationally unavailable (e.g., owner is a multisig that loses quorum, or a DAO with a broken governance process), no new accounts can be added to the `Account`/`Address` whitelists, and no existing entries can be removed. Users whose addresses are not already whitelisted cannot submit transactions, including withdrawal transactions. This constitutes a **temporary (potentially permanent) freezing of user funds** in the Silo EVM environment, since users cannot move their assets out.

The `Admin` role was specifically designed to allow delegation of whitelist management below the owner level, precisely to avoid this single-point-of-failure. Because the implementation ignores the `Admin` whitelist for all management operations, this safety valve does not exist.

---

### Likelihood Explanation

Silo mode is an intentional production feature of Aurora Engine. Any Silo operator who follows the documented behavior and grants `Admin` whitelist entries expecting those accounts to manage whitelists will find the feature silently non-functional. The discrepancy between documentation and implementation is directly reachable by any NEAR account that calls `add_entry_to_whitelist` while holding `Admin` whitelist membership — the call will be rejected with `ERR_NOT_ALLOWED`. [1](#0-0) 

---

### Recommendation

The whitelist management entry points should check whether the caller is either the owner **or** a member of the `Admin` whitelist. For example:

```rust
pub extern "C" fn add_entry_to_whitelist() {
    let io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    let predecessor = io.predecessor_account_id();
    let is_owner = state.owner_id == predecessor;
    let is_admin = silo::is_admin(&io, &predecessor);
    if !is_owner && !is_admin {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED);
    }
    require_running(&state).map_err(ContractError::msg).sdk_unwrap();
    let args: WhitelistArgs = io.read_input_borsh().sdk_unwrap();
    silo::add_entry_to_whitelist(&io, &args);
}
```

The same pattern should be applied to `add_entry_to_whitelist_batch`, `remove_entry_from_whitelist`, `set_whitelist_status`, and `set_whitelists_statuses`.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode and enable all whitelists via `set_whitelists_statuses`.
2. As the owner, call `add_entry_to_whitelist` to add account `admin.near` to the `WhitelistKind::Admin` whitelist.
3. As `admin.near`, call `add_entry_to_whitelist` to add a new account `user.near` to the `WhitelistKind::Account` whitelist.
4. Observe: the call fails with `ERR_NOT_ALLOWED` because `require_owner_and_running` rejects any caller that is not the `owner_id`, regardless of `Admin` whitelist membership.
5. The `Admin` whitelist entry for `admin.near` is stored in state but never checked during whitelist management calls. [2](#0-1) [1](#0-0) [7](#0-6)

```rust
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}

pub fn require_owner_and_running(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    require_running(state)?;
    require_owner_only(state, predecessor_account_id)?;

    Ok(())
}
```
