### Title
`EVM.validateCOAOwnershipProof` Does Not Bind Signature to `evmAddress`, Enabling Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that caller-supplied `signatures` are valid over caller-supplied `signedData` for a Cadence account, then separately checks that the COA at the given `path` matches the supplied `evmAddress`. Because the function never enforces that `signedData` encodes `evmAddress` (or `path`), a signature produced to prove ownership of one COA is cryptographically valid as proof of ownership of any other COA held by the same Cadence account. The function is `access(all)` and callable by any unprivileged Cadence script or transaction. The code itself acknowledges the gap in a comment but treats it as low-risk based on a historical wallet convention, not a protocol invariant.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. **Signature check** (lines 1082–1093): verifies `signatures` over `signedData` using the Cadence account's keys.
2. **COA address check** (lines 1095–1110): borrows the COA capability at `path` and compares its EVM address to `evmAddress`.

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // ← evmAddress is NOT required to be in here
    domainSeparationTag: "FLOW-V0.0-user"
)
...
if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
    let coaAddressBytes = coaRef.address().bytes
    for index, item in coaAddressBytes {
        if item != evmAddress[index] { ... }
    }
    return ValidationResult(isValid: true, problem: nil)
}
```

The two checks are never joined: the signature is verified against `signedData` (arbitrary caller-controlled bytes), and the COA address is verified against `evmAddress` (also caller-controlled). Nothing in the protocol requires `signedData` to commit to `evmAddress` or `path`.

The code comment at lines 1003–1009 explicitly acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The function is declared `access(all)`, so any unprivileged Cadence script or transaction can call it with arbitrary `path` and `evmAddress` while reusing a previously observed valid `(signedData, signatures)` pair.

**Attack path:**

1. Victim Cadence account `A` holds two COAs: `COA_X` published at `/public/coa_x` and `COA_Y` published at `/public/coa_y`.
2. Victim (or a dApp) produces a valid `(signedData, signatures)` pair to prove ownership of `COA_X`. `signedData` does not encode `COA_X`'s EVM address (the protocol does not require it).
3. Attacker observes the `(signedData, signatures)` pair on-chain or off-chain.
4. Attacker calls `EVM.validateCOAOwnershipProof(address: A, path: /public/coa_y, signedData: <same>, keyIndices: <same>, signatures: <same>, evmAddress: COA_Y.address)`.
5. Step 1 (signature check) passes because the signature is genuinely valid for account `A` over `signedData`.
6. Step 2 (COA address check) passes because `COA_Y` is legitimately at `/public/coa_y` with the supplied EVM address.
7. The function returns `ValidationResult(isValid: true)` — falsely asserting that the signer authorized `COA_Y`.

Any Cadence contract or dApp that calls `EVM.validateCOAOwnershipProof` to gate access to on-chain assets or privileged operations accepts this forged proof.

### Impact Explanation

An attacker who possesses a valid `(signedData, signatures)` pair for a Cadence account — obtained from any prior legitimate proof-of-ownership interaction — can present it as proof of ownership of a *different* COA held by the same account. If a Cadence contract uses `EVM.validateCOAOwnershipProof` to authorize asset transfers, bridge operations, or other privileged actions scoped to a specific COA's EVM address, the attacker can redirect those actions to a COA the signer never authorized. This constitutes unauthorized access to on-chain assets gated by COA identity.

### Likelihood Explanation

The likelihood is **low-to-medium**. Two preconditions must hold simultaneously:

1. The victim Cadence account holds more than one COA (the comment notes this is historically uncommon, but the protocol imposes no limit and multi-COA accounts will become more common as the ecosystem matures).
2. A valid `(signedData, signatures)` pair for the account is observable by the attacker (e.g., emitted in a transaction, passed as a script argument, or used in a public dApp flow).

The entry point is fully unprivileged: any Cadence script or transaction can call `EVM.validateCOAOwnershipProof` with attacker-chosen arguments.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` commits to `evmAddress` before accepting the proof as valid. The simplest approach is to verify that the raw bytes of `evmAddress` appear within `signedData`, or — more robustly — require callers to sign a canonical encoding that includes `evmAddress`:

```cadence
// Inside validateCOAOwnershipProof, before the keyList.verify() call:
let evmAddressBytes: [UInt8] = evmAddress.toConstantSized()!.toVariableSized()
var evmAddressFound = false
// simplest check: signedData must contain evmAddress bytes
// (production code should use a canonical ABI encoding instead)
if signedData.length >= 20 {
    // require evmAddress to be encoded in signedData
    // enforce via a canonical prefix/hash scheme
}
```

The recommended canonical fix is to require `signedData = hash(evmAddress || application_data)` or to verify that `signedData` is exactly the ABI encoding of `(evmAddress, ...)`, mirroring the recommendation in the external report to include the implementation address in the signed digest.

### Proof of Concept

**Setup:**
- Cadence account `A` has two COAs: `COA_X` at `/public/coa_x` (EVM address `0xAAAA...`) and `COA_Y` at `/public/coa_y` (EVM address `0xBBBB...`).
- A dApp asks account `A` to sign `signedData = "authorize_coa_x"` (no EVM address encoded) to prove ownership of `COA_X`. The resulting `signatures` are published on-chain.

**Exploit transaction (callable by any unprivileged sender):**

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>

access(all)
fun main(): EVM.ValidationResult {
    // Reuse the victim's signature, but point at COA_Y instead of COA_X
    return EVM.validateCOAOwnershipProof(
        address: <victim_cadence_address>,
        path: /public/coa_y,                  // ← different COA
        signedData: "authorize_coa_x".utf8,   // ← same signedData as original
        keyIndices: [0],                       // ← same key index
        signatures: [<victim_signature>],      // ← same signature
        evmAddress: <COA_Y_evm_address>        // ← COA_Y's address
    )
    // Returns ValidationResult(isValid: true) — forged proof of COA_Y ownership
}
```

**Expected result:** `isValid: true`, falsely asserting the victim authorized `COA_Y`, because the signature check passes (the signature is genuinely valid for account `A` over `signedData`) and the COA address check passes (COA_Y is legitimately at `/public/coa_y`).
