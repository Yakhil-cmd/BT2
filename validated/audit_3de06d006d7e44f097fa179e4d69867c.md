### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With `msg.value > 0` and No Pending Mint Requests â€” (`L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
`tick()` is declared `external payable` but contains no refund path and no `receive()`/`fallback()`/`withdraw()` mechanism. When `amountToMint == 0`, the entire ETH-spending block is skipped and `msg.value` is silently absorbed by the contract with no recovery path.

### Finding Description
`tick()` at line 107 is `external payable`. [1](#0-0) 

When `requiredMinting()` returns `amountToMint == 0` (i.e., `messagingContract().l2ToL1Messages(msgHash) == 0`), the `if (amountToMint > 0)` block is skipped entirely. [2](#0-1) 

`msg.value` is only consumed inside that block via `bridge().depositWithMessage{value: msgFee}(...)` and `messagingContract().sendMessageToL2{value: msgFee}(...)`. [3](#0-2) 

Outside the `if` block, there is no `else` branch, no `require(msg.value == 0)` guard, and no refund statement. The function simply returns. [4](#0-3) 

A grep across all L1 Solidity files confirms there is no `receive()`, `fallback()`, or `withdraw` function anywhere in the contract or its inheritance chain (`RewardSupplierStorage`, `ProxySupportImpl`, `GovernanceStub`). [5](#0-4) 

### Impact Explanation
Any ETH sent to `tick()` when no L2â†’L1 mint-request messages are pending is permanently locked in the `RewardSupplier` contract. There is no owner-rescue function, no sweep, and no proxy-level ETH drain. The funds are irrecoverable. This maps to **High: Permanent freezing of funds** (the caller's ETH is frozen forever). The Critical framing in the question ("direct theft") is slightly overstated â€” no third party profits â€” but the permanent, unrecoverable loss of the caller's ETH is a concrete, bounty-grade impact.

### Likelihood Explanation
`tick()` is a public, permissionless function intended to be called by keepers or bots. Any caller that sends ETH (e.g., to cover anticipated bridge/messaging fees) during a period with no pending L2 mint requests will lose that ETH. This is a realistic operational scenario: the L2 side may not have queued a mint request yet, or all pending messages were already consumed in a prior call.

### Recommendation
Add a guard at the top of `tick()`:
```solidity
require(msg.value == 0 || amountToMint > 0, "NO_MINT_REQUIRED");
```
Or, unconditionally refund any unused ETH at the end of the function:
```solidity
if (address(this).balance > 0) {
    (bool ok,) = msg.sender.call{value: address(this).balance}("");
    require(ok, "ETH_REFUND_FAILED");
}
```
The cleanest fix is to remove `payable` from `tick()` entirely and only accept ETH when minting is actually required, reverting otherwise.

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)

            // Send a totalSupply update to L2MintCurve.
