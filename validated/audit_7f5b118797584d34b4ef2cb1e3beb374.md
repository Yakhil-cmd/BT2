### Title
`l1_messenger_hook` Accepts Unverified `address_sender` from Calldata While Ignoring `callers_caller` - (File: `system_hooks/src/call_hooks/l1_messenger.rs`)

### Summary

The `l1_messenger_hook` function verifies that the EVM-level `caller` is the L1Messenger system contract (`0x8008`), but it explicitly ignores the `callers_caller` field (the original user who called the L1Messenger contract) and blindly trusts the `address_sender` embedded in the first 20 bytes of calldata. This `address_sender` is then passed directly to `emit_l1_message` as the recorded sender of the L1 message, without any cross-check against the actual originating address.

### Finding Description

In `l1_messenger_hook`, the `ExternalCallRequest` is destructured and `callers_caller` is explicitly discarded:

```rust
let ExternalCallRequest {
    ...
    caller,
    callee,
    callers_caller: _,   // ← explicitly ignored
    ...
} = request;
```

The hook then performs only one identity check:

```rust
if caller != L1_MESSENGER_ADDRESS {
    // pretend to be empty account
    return Ok(...);
}
```

After passing this check, `send_to_l1_inner` reads the sender address directly from the first 20 bytes of the calldata:

```rust
let address_sender = B160::try_from_be_slice(&calldata[0..20])...;
let message = &calldata[20..];

system.io.emit_l1_message(
    ExecutionEnvironmentType::NoEE,
    resources,
    &address_sender,   // ← unverified, caller-supplied
    message,
)?;
```

There is no assertion that `address_sender == callers_caller`. The hook trusts the L1Messenger contract to supply the correct sender, but it never enforces this invariant itself.

The parallel to the external report is exact:

| External Report | ZKsync OS |
|---|---|
| `verify_origin()` checks `origin` | `l1_messenger_hook` checks `caller` (L1Messenger contract) |
| `previous_sender` is never verified | `callers_caller` is explicitly ignored (`callers_caller: _`) |
| `previous_sender` used in `contains_sender()` for authz | `address_sender` used in `emit_l1_message` as the recorded L1 message sender |

### Impact Explanation

`emit_l1_message` records the L1 message with `address_sender` as the originating address. L1-side bridge and governance contracts that rely on the sender field of L1 messages for authorization decisions (e.g., "only bridge contract X may send withdrawal messages") could be bypassed if an attacker can cause the hook to record an arbitrary `address_sender`.

If the L1Messenger system contract (`0x8008`) exposes any function that forwards caller-supplied data as the sender field (e.g., a `sendToL1WithSender(address, bytes)` variant, or any path where the first 20 bytes of the hook calldata are not strictly `msg.sender`), an unprivileged user can forge the recorded L1 message sender. The hook provides no defense-in-depth against this.

### Likelihood Explanation

The hook is reachable by any unprivileged EVM transaction that calls the L1Messenger system contract at `0x8008`. The `callers_caller` field is structurally available in `ExternalCallRequest` but is explicitly discarded. The L1Messenger contract's bytecode is not in this repository, so whether it currently exposes a forgeable path is not directly verifiable from this codebase alone. However, the hook's design provides zero enforcement of the sender invariant, making it entirely dependent on the L1Messenger contract never being upgraded or extended in a way that allows caller-controlled sender data.

### Recommendation

The hook should verify that the `address_sender` extracted from calldata matches `callers_caller` (the entity that called the L1Messenger contract):

```rust
let ExternalCallRequest {
    ...
    caller,
    callers_caller,   // ← capture instead of ignoring
    ...
} = request;

// ... existing caller check ...

// In send_to_l1_inner or before calling it:
let address_sender = B160::try_from_be_slice(&calldata[0..20])?;
if address_sender != callers_caller {
    return Ok((make_error_return_state(available_resources), return_memory));
}
```

This mirrors the fix suggested in the external report: both the "outer" identity field (`caller` / `origin`) and the "inner" identity field (`address_sender` / `previous_sender`) must be verified against the actual call context.

### Proof of Concept

The existing test `test_l1_messenger_hook_succeeds` already demonstrates the issue: it constructs an L1 transaction `from` the L1Messenger address and passes calldata containing `0x1111...1111` as the sender address. The hook accepts this without any check that `0x1111...1111` is the actual originating address.

```rust
// From tests/instances/system_hooks/src/lib.rs
let hook_calldata = hex::decode(
    // address field = 0x1111...1111, NOT the actual tx sender
    "000000000000000000000000111111111111111111111111111111111111111100..."
).unwrap();

let tx = L1TxBuilder::new()
    .from(l1_messenger_contract)   // caller = L1Messenger ✓
    .to(l1_messenger_hook)
    .input(hook_calldata)          // address_sender = 0x1111...1111, unverified ✗
    ...
    .build();
```

The hook passes, and `emit_l1_message` is called with `address_sender = 0x1111...1111` regardless of who actually initiated the transaction. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
pub(crate) fn send_to_l1_inner<S: EthereumLikeTypes>(
    calldata: &[u8],
    resources: &mut S::Resources,
    system: &mut System<S>,
) -> Result<Result<(), &'static str>, SystemError> {
    if calldata.len() < 20 {
        return Ok(Err(
            "L1 messenger failure: sendToL1 called with invalid calldata",
        ));
    }

    let address_sender = B160::try_from_be_slice(&calldata[0..20]).ok_or(
        SystemError::LeafDefect(internal_error!("Failed to create B160 from 20 byte array")),
    )?;

    let message = &calldata[20..];

    // emit L1 message (ignore returned hash)
    // TODO(EVM-1190): hash calculation is suboptimal, to be refactored in future
    system.io.emit_l1_message(
        // Gas should be charged by the L1Messenger system contract
        ExecutionEnvironmentType::NoEE,
        resources,
        &address_sender,
        message,
    )?;

    Ok(Ok(()))
```
