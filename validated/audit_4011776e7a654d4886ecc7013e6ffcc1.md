### Title
Missing Intra-Batch Block Timestamp Monotonicity Check Allows Prover to Corrupt Chain State Commitment - (File: `basic_bootloader/src/bootloader/block_flow/zk/batch_data.rs`)

---

### Summary
`ZKBatchDataKeeper::apply_block` updates `current_block_timestamp` for non-first blocks in a multi-block batch without verifying that the new timestamp is ≥ the previously stored timestamp. The only monotonicity check exists at the **batch boundary** (comparing against the oracle-provided `last_block_timestamp` from the prior batch's chain state commitment), but no such check exists **within** the batch. A malicious prover can supply oracle data with decreasing block timestamps across blocks in a batch, generating a valid proof where `BatchOutput.last_block_timestamp < BatchOutput.first_block_timestamp`, corrupting the `ChainStateCommitment.last_block_timestamp` committed to the settlement layer.

---

### Finding Description

`ZKBatchDataKeeper::apply_block` is called once per block during multi-block batch proving. For non-first blocks, it unconditionally overwrites `current_block_timestamp` with no monotonicity guard:

```rust
} else {
    assert_eq!(self.current_state_commitment.unwrap(), state_commitment_before);
    self.current_state_commitment = Some(state_commitment_after);
    self.current_block_timestamp = Some(block_timestamp); // ← no check vs. prior value
    assert_eq!(self.chain_id.unwrap(), chain_id);
    assert!(upgrade_tx_hash.is_zero());
    assert_eq!(self.settlement_layer_chain_id, Some(settlement_layer_chain_id));
}
``` [1](#0-0) 

The only timestamp monotonicity check in the proving path is in `post_tx_op_proving_multiblock_batch.rs`:

```rust
// validate that timestamp didn't decrease
assert!(metadata.block_timestamp() >= last_block_timestamp);
``` [2](#0-1) 

This checks the current block's timestamp against the oracle-provided `last_block_timestamp` from the **previous batch's** chain state commitment — not against the previous block's timestamp within the current batch.

The `ChainStateCommitment` struct explicitly documents that `last_block_timestamp` is included "to ensure that block timestamps are not decreasing": [3](#0-2) 

Yet this invariant is only enforced at the batch boundary, not within the batch. The `BatchOutput` produced by `into_public_input` directly uses the unchecked `current_block_timestamp` as `last_block_timestamp`: [4](#0-3) 

The `BatchOutput.hash()` commits this value into the batch public input submitted to the settlement layer: [5](#0-4) 

---

### Impact Explanation

A malicious prover constructs a multi-block batch where block timestamps decrease (e.g., Block 1: `t=1000`, Block 2: `t=500`). The resulting `BatchOutput` has `first_block_timestamp=1000` and `last_block_timestamp=500`. The `chain_state_commitment_after` commits to `last_block_timestamp=500`. The settlement layer accepts this batch and updates the stored state to `last_block_timestamp=500`. Future batches only need timestamps ≥ 500, not ≥ 1000, allowing the prover to effectively roll back the chain's timestamp anchor. This can affect time-dependent smart contract logic (e.g., contracts checking `block.timestamp >= deadline`) and undermines the settlement layer's ability to enforce forward-only time progression.
