### Title
Swap Allowlist Bypassed via Router: `SwapAllowlistExtension` Gates the Router Address, Not the End User — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension.beforeSwap` checks the `sender` argument, which is `msg.sender` of the `pool.swap()` call. When a user routes through `MetricOmmSimpleRouter`, the pool sees the router as `sender`, not the actual end user. If the pool admin allowlists the router address to enable router-mediated swaps for permitted users, every non-allowlisted user can bypass the curated-pool restriction by routing through the same public router contract.

### Finding Description

`SwapAllowlistExtension.beforeSwap` enforces:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
``` [1](#0-0) 

Here `msg.sender` is the pool (the extension is called by the pool) and `sender` is the first argument forwarded from `ExtensionCalling._beforeSwap`, which encodes `msg.sender` of the originating `pool.swap()` call:

```solidity
abi.encodeCall(IMetricOmmExtensions.beforeSwap, (sender, ...))
``` [2](#0-1) 

When a user calls `MetricOmmSimpleRouter.exactInput` or `exactOutput`, the router calls `pool.swap()` directly, so the pool receives `msg.sender = router`. The extension therefore checks `allowedSwapper[pool][router]`, not `allowedSwapper[pool][actualUser]`. [3](#0-2) 

The pool admin faces an impossible choice:

- **Do not allowlist the router** → no user (even permitted ones) can swap via the router on this pool; core router functionality is broken for the curated pool.
- **Allowlist the router** → every user, including those explicitly excluded from the allowlist, can bypass the restriction by calling `MetricOmmSimpleRouter` instead of `pool.swap()` directly.

The `DepositAllowlistExtension` correctly gates on `owner` (the position owner), not on `sender` (the payer), so the deposit path does not have the same flaw. [4](#0-3) 

### Impact Explanation

Any address excluded from a curated pool's swap allowlist can bypass the restriction by routing through the public `MetricOmmSimpleRouter`. The pool admin's curation policy is silently voided for all router-mediated swaps. Depending on the pool's purpose (e.g., KYC-gated, institutional-only, or compliance-restricted liquidity), this allows unauthorized parties to trade against LP funds, constituting a broken core pool functionality and a direct admin-boundary break. Severity: **High** — the allowlist guard fails open for the entire public router path.

### Likelihood Explanation

`MetricOmmSimpleRouter` is the primary public swap entry point documented and used by integrators. Any user aware of the router can trivially route around the allowlist without any privileged access. The bypass requires only a standard ERC-20 approval and a single router call.

### Recommendation

The `beforeSwap` hook must receive and check the **actual end user**, not the intermediary router. Two approaches:

1. **Pass the originating user through the router**: The router should forward the real caller's address in `extensionData`, and the extension should decode and check it. This requires a trusted router identity check in the extension.
2. **Check `tx.origin` as a fallback** (not recommended for general use, but acceptable if the pool is designed for EOA-only access).
3. **Preferred**: Redesign the hook signature so the pool passes both `msg.sender` (the immediate caller, e.g., router) and an authenticated `originatingUser` field that the router commits to via a signed payload or callback pattern, and the extension gates on `originatingUser`.

### Proof of Concept

1. Deploy a pool with `SwapAllowlistExtension` configured in `beforeSwap`.
2. Pool admin calls `setAllowedToSwap(pool, router, true)` to enable router-mediated swaps for allowed users.
3. Pool admin does **not** call `setAllowedToSwap(pool, attacker, true)`.
4. Attacker (not on allowlist) calls `MetricOmmSimpleRouter.exactInput(...)` targeting the pool.
5. The pool calls `_beforeSwap(router, ...)` → extension checks `allowedSwapper[pool][router]` → `true` → swap proceeds.
6. Attacker successfully swaps on the curated pool despite being excluded from the allowlist. [3](#0-2) [5](#0-4)
