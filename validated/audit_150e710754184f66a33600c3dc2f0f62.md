### Title
Unbounded Loop in `calculate_rewards` Over `pool_member_epoch_balance` Trace Enables Permanent Freezing of Unclaimed Yield - (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in the Pool contract iterates over every entry in a pool member's `pool_member_epoch_balance` trace without any bound. The trace grows by one entry per epoch in which the delegator modifies their balance. A delegator who makes balance changes across many epochs without claiming rewards will eventually be unable to claim them at all, because the loop will exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace is stored in `pool_member_epoch_balance` and grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`: [2](#0-1) 

The `insert` function in the trace implementation appends a **new checkpoint** whenever the key (epoch+K) differs from the last entry's key, and only **updates** the last entry when the key is the same: [3](#0-2) 

This means every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **new epoch** appends a new entry to the trace. Both functions call `set_member_balance` / `increase_member_balance`: [4](#0-3) [5](#0-4) 

The `entry_to_claim_from` cursor is only advanced on a **successful** `claim_rewards` call: [6](#0-5) 

If a delegator makes one balance change per epoch and defers claiming rewards, the trace grows linearly with the number of epochs elapsed. When `claim_rewards` is eventually called, the loop must iterate over every accumulated entry since the last claim, each iteration performing multiple storage reads (including `find_sigma` lookups). Beyond a sufficient number of entries, the transaction will exceed the Starknet gas limit and revert, making it permanently impossible to claim rewards.

The same loop is also executed inside the view function `pool_member_info_v1`: [7](#0-6) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace grows large enough, every call to `claim_rewards` (and `pool_member_info_v1`) will OOG-revert. Because `entry_to_claim_from` is only updated on a successful claim, the delegator is permanently locked out of their accrued yield. This constitutes **permanent freezing of unclaimed yield** (High severity per the allowed impact scope).

---

### Likelihood Explanation

Any delegator who:
1. Regularly adjusts their delegation balance (e.g., tops up or partially exits once per epoch), and
2. Does not claim rewards frequently enough to keep `entry_to_claim_from` advancing

will accumulate trace entries at a rate of one per epoch. On a protocol with short epochs and a long-lived delegator, hundreds to thousands of entries can accumulate. The minimum cost per entry is one `add_to_delegation_pool` call with a non-zero amount, which is a normal user action with no special privilege required. The delegator may not realize the risk until `claim_rewards` starts reverting.

---

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the trace already has more than a safe upper bound of entries since the last claimed checkpoint (`pool_member_trace_length - entry_to_claim_from > MAX_UNCLAIMED_ENTRIES`).
2. **Alternatively, require `claim_rewards` before allowing further balance changes** once the unclaimed trace window exceeds a threshold, forcing the delegator to periodically flush the loop.
3. Remove the comment "unlikely to exceed gas limits" and replace it with an enforced invariant.

---

### Proof of Concept

```
Epoch 1:  delegator calls add_to_delegation_pool(amount=1)  → trace length = 1
Epoch 2:  delegator calls add_to_delegation_pool(amount=1)  → trace length = 2
...
Epoch N:  delegator calls add_to_delegation_pool(amount=1)  → trace length = N

Epoch N+1: delegator calls claim_rewards()
  → calculate_rewards loops N times, each iteration reads storage + calls find_sigma
  → for large N (e.g., N > ~500 depending on gas costs), transaction OOGs
  → entry_to_claim_from is never updated (revert)
  → all subsequent claim_rewards calls also OOG
  → unclaimed yield is permanently frozen
```

```text
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
