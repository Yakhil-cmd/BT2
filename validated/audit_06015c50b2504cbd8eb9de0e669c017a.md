### Title
Unrestricted Overwrite of `eligible_operational_addresses` Enables Griefing of Staker Operational Address Changes - (File: src/staking/staking.cairo)

### Summary

`declare_operational_address` in `src/staking/staking.cairo` allows any caller to freely overwrite the `eligible_operational_addresses` mapping entry for their own address at any time, as long as the address is not yet actively assigned to a staker. There is no check that the slot is already occupied by a prior declaration. A malicious operational address can declare for staker A, then immediately re-declare for staker B, silently invalidating staker A's pending `change_operational_address` call.

### Finding Description

The two-step operational address change flow works as follows:

1. The new operational address calls `declare_operational_address(staker_address)`, writing `eligible_operational_addresses[op_addr] = staker_address`.
2. The staker calls `change_operational_address(op_addr)`, which asserts `eligible_operational_addresses[op_addr] == staker_address`.

The `declare_operational_address` function contains no guard preventing an overwrite of an existing declaration:

```cairo
fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
    self.general_prerequisites();
    let operational_address = get_caller_address();
    assert!(
        self.operational_address_to_staker_address.read(operational_address).is_zero(),
        "{}",
        Error::OPERATIONAL_IN_USE,
    );
    // ...
    if self.eligible_operational_addresses.read(operational_address) == staker_address {
        return;  // only skips if SAME staker; any other staker causes overwrite
    }
    self.eligible_operational_addresses.write(operational_address, staker_address);
    self.emit(Events::OperationalAddressDeclared { operational_address, staker_address });
}
``` [1](#0-0) 

The only guard (`OPERATIONAL_IN_USE`) checks `operational_address_to_staker_address`, which is only set after `change_operational_address` completes. Before that point, the `eligible_operational_addresses` slot is unprotected and can be overwritten an unlimited number of times.

The consuming assertion in `change_operational_address` is:

```cairo
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address,
    "{}",
    Error::OPERATIONAL_NOT_ELIGIBLE,
);
``` [2](#0-1) 

If the mapping has been overwritten between the declaration and the staker's call, this assertion fails and the staker's transaction reverts.

### Impact Explanation

A staker whose current operational address is compromised or lost must rotate to a new one. If the new operational address is controlled by an adversary (or if the adversary can observe the pending declaration and race to overwrite it), the staker's `change_operational_address` call will permanently fail for that address. The staker is forced to find yet another operational address, and the cycle can repeat. In the worst case — where the staker's current operational address is already compromised and the attacker can consistently race every new declaration — the staker is indefinitely blocked from rotating their operational address, preventing them from submitting valid attestations and causing permanent loss of attestation rewards.

This matches the allowed impact: **Griefing with no profit motive but damage to users or protocol (Medium)**, and potentially **Permanent freezing of unclaimed yield** if the staker cannot attest.

### Likelihood Explanation

The attack requires no privileged role. Any address that has not yet been assigned as an operational address can call `declare_operational_address` freely. The overwrite is a single public transaction. On Starknet, where the sequencer orders transactions, an adversary who controls the new operational address (or who monitors the mempool) can trivially execute this before the staker's `change_operational_address` is included. The scenario is realistic whenever a staker needs to rotate a compromised key.

### Recommendation

Add a guard in `declare_operational_address` that prevents overwriting an existing non-zero declaration:

```cairo
assert!(
    self.eligible_operational_addresses.read(operational_address).is_zero(),
    "{}",
    Error::OPERATIONAL_ALREADY_DECLARED,
);
```

This mirrors the existing `OPERATIONAL_IN_USE` check on `operational_address_to_staker_address` and closes the window between declaration and consumption. Alternatively, only allow the declaration to be overwritten if the previous target staker explicitly releases it, or introduce a timelock on the declaration.

### Proof of Concept

1. Staker A's current operational address is compromised. Staker A wants to rotate to `op_addr_new`.
2. `op_addr_new` calls `declare_operational_address(staker_A)`. Storage: `eligible_operational_addresses[op_addr_new] = staker_A`.
3. Before staker A's `change_operational_address(op_addr_new)` is sequenced, `op_addr_new` calls `declare_operational_address(staker_B)` (any other address). Storage: `eligible_operational_addresses[op_addr_new] = staker_B`.
4. Staker A's `change_operational_address(op_addr_new)` reverts with `OPERATIONAL_NOT_ELIGIBLE` because `eligible_operational_addresses[op_addr_new] == staker_B != staker_A`.
5. Staker A is still bound to the compromised operational address and cannot attest, losing all future attestation rewards until the situation is resolved.

The attacker repeats step 3 for every new address staker A attempts to use, as long as the attacker controls or can race those addresses.
