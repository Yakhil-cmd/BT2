### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield — (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract iterates over every balance-change checkpoint stored in a pool member's `pool_member_epoch_balance` trace without any upper bound. A pool member (or their designated reward address) can grow this trace to an arbitrary size by calling `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch. Once the trace is large enough, `claim_rewards` will exceed Starknet's per-transaction gas limit, permanently freezing the pool member's unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` up to `pool_member_trace_length`. The `entry_to_claim_from` cursor is only advanced when `claim_rewards` is successfully called and committed to storage. [2](#0-1) 

New entries are appended to the trace by `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: ...)`. The `insert` function only overwrites the last entry when the epoch key matches; otherwise it **appends** a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, ...);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

`set_member_balance` is called by both `enter_delegation_pool` and `add_to_delegation_pool` (via `increase_member_balance`), and also by `exit_delegation_pool_intent`: [4](#0-3) 

Because `get_epoch_plus_k()` returns `current_epoch + K` (K=2), each call made in a **different epoch** produces a distinct key and therefore a new appended checkpoint. A pool member who calls `add_to_delegation_pool` once per epoch, across N epochs without ever claiming rewards, accumulates N entries in their trace. When `claim_rewards` is eventually called, the loop must iterate over all N entries in a single transaction.

The same unbounded loop is also triggered by the view function `pool_member_info_v1`, which calls `calculate_rewards` without updating `entry_to_claim_from`: [5](#0-4) 

---

### Impact Explanation

Once the trace grows large enough that a single `claim_rewards` call exceeds Starknet's gas limit, the pool member's accumulated unclaimed yield becomes **permanently unclaimable**. There is no mechanism to partially claim rewards or to prune the trace. This matches the allowed impact: **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation

The attack is reachable by any unprivileged pool member or their reward address. `add_to_delegation_pool` is callable by either the pool member or the reward address: [6](#0-5) 

A malicious reward address can grief a pool member by repeatedly calling `add_to_delegation_pool` with the minimum non-zero amount across many epochs, spending only token dust per epoch. Over a long enough time horizon (e.g., years of protocol operation), even organic usage — a delegator who adds to their position every epoch — can naturally accumulate enough entries to hit the gas limit. The developers themselves acknowledged the risk in the code comment ("This loop is unbounded but unlikely to exceed gas limits"), confirming awareness of the issue without a mitigation.

---

### Recommendation

1. **Checkpoint compaction**: After a successful `claim_rewards`, truncate or compact all trace entries before `entry_to_claim_from`, since they are no longer needed for future reward calculations.
2. **Bounded iteration**: Introduce a maximum number of entries processed per `claim_rewards` call and allow partial claims, advancing `entry_to_claim_from` incrementally across multiple transactions.
3. **Rate-limit trace growth**: Enforce that `add_to_delegation_pool` in the same epoch (same `epoch + K` key) always overwrites rather than appends — the current `insert` already does this for the same key, but consider also merging entries within the same epoch window.

---

### Proof of Concept

1. Pool member Alice enters the delegation pool with minimum stake.
2. Alice (or her reward address) calls `add_to_delegation_pool(pool_member: alice, amount: 1)` once per epoch for N epochs, never calling `claim_rewards`.
3. Each call appends a new checkpoint to `pool_member_epoch_balance[alice]` because `get_epoch_plus_k()` returns a strictly increasing key each epoch.
4. After N epochs, `pool_member_epoch_balance[alice].length() == N`.
5. Alice calls `claim_rewards`. The `calculate_rewards` loop iterates N times, each iteration reading a storage slot and calling `find_sigma`.
6. For sufficiently large N (bounded by Starknet's per-transaction gas limit), the transaction reverts with an out-of-gas error.
7. Every subsequent `claim_rewards` call also reverts. Alice's accumulated yield is permanently frozen. [7](#0-6) [8](#0-7)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
