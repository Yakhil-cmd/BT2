### Title
Unprivileged Caller Can Permanently Freeze All Staker Consensus Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract has no caller access control and accepts an attacker-controlled `disable_rewards` boolean. Any non-zero address can call it once per block with `disable_rewards: true` to consume the global `last_reward_block` slot without distributing rewards, permanently destroying block-level yield for all stakers.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` is gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is not the zero address. [1](#0-0) 

The function enforces a global, single-use-per-block invariant via `last_reward_block`: [2](#0-1) 

After validating the staker, it unconditionally writes the current block number to `last_reward_block` **before** checking `disable_rewards`: [3](#0-2) 

If `disable_rewards` is `true`, the function returns immediately without distributing any rewards: [4](#0-3) 

Because `last_reward_block` is a **global** storage variable (not per-staker), any subsequent call to `update_rewards` in the same block — including the legitimate one — will revert with `REWARDS_ALREADY_UPDATED`. The block rewards that were skipped are not accumulated anywhere; they are permanently lost. [5](#0-4) 

The `last_reward_block` field is declared as a single global slot: [6](#0-5) 

---

### Impact Explanation

An attacker calling `update_rewards(staker_address: <any_active_staker>, disable_rewards: true)` once per block:

1. Consumes the block's single allowed `update_rewards` slot.
2. Writes `last_reward_block = current_block`.
3. Returns without distributing STRK or BTC block rewards.
4. Causes every legitimate `update_rewards` call in that block to revert with `REWARDS_ALREADY_UPDATED`.

Skipped block rewards are not deferred — they are permanently destroyed. Sustained over many blocks, this constitutes **permanent freezing of unclaimed yield** for all stakers in the protocol, matching the High-severity impact in scope.

---

### Likelihood Explanation

- No tokens, stake, or privileged role required.
- The attacker only needs any publicly observable active staker address (readable from the `stakers` vector or emitted `NewStaker` events).
- One cheap Starknet transaction per block suffices.
- The attack is fully permissionless and can be automated.

---

### Recommendation

Restrict who may supply `disable_rewards: true`. Options in increasing strictness:

1. **Require caller to be the staker or their operational address** when `disable_rewards: true` is passed.
2. **Remove `disable_rewards` from the public interface** entirely and handle the skip logic internally (e.g., keyed on a privileged role or on-chain condition).
3. **Gate the entire function** so only the staker's operational address (or a designated keeper role) may call it.

The simplest fix consistent with the existing design:

```cairo
if disable_rewards {
    let staker_info = self.internal_staker_info(:staker_address);
    let caller = get_caller_address();
    assert!(
        caller == staker_address || caller == staker_info.operational_address,
        "{}",
        Error::UNAUTHORIZED_DISABLE_REWARDS,
    );
}
```

---

### Proof of Concept

1. Attacker reads any active staker address `S` from on-chain events or the `stakers` storage vector.
2. At the first transaction of every block, attacker calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
3. `last_reward_block` is set to the current block; no rewards are distributed.
4. Any legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeated every block → all consensus block rewards are permanently destroyed.

This is the direct analog of the external report: just as the lender supplied attacker-controlled `zeroExTradeData` to bypass the balance check and steal funds, here an attacker supplies `disable_rewards: true` to bypass reward distribution and permanently freeze staker yield — with no access control preventing either action.
