### Title
Bank Precompile `transfer` Accepts Arbitrary `sender` Without Authorization Check — (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The bank precompile's `transfer` method takes the `sender` address directly from call arguments rather than from `contract.Caller()`, with no check that the caller is authorized to move tokens on behalf of the specified sender. Any EVM contract can call `bankPrecompile.transfer(victim, attacker, amount)` and drain the victim's native Cosmos token balance for the denom corresponding to the calling contract.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` from the ABI-encoded call data:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is correctly derived from `contract.Caller()` (the calling EVM contract), but the `from` address is taken directly from the caller-supplied argument with **no authorization check** — there is no assertion that `contract.Caller() == sender`, no allowance/approval mechanism, and no signature verification. Compare this to the `mint`/`burn` cases, which also take a target address from args but are one-directional (minting to or burning from a specified address), whereas `transfer` moves tokens *from* an address the caller does not own. [2](#0-1) 

The `mint` and `burn` paths use `contract.Caller()` only to derive the denom, and the recipient/target is an argument — this is acceptable because minting/burning is a privileged operation gated by the CRC20 contract itself. But `transfer` additionally allows specifying an arbitrary `from` address, which is the authorization gap.

### Impact Explanation

**Critical — Unauthorized transfer of CRC20/native token balances.**

Any EVM contract whose address maps to a Cosmos denom (i.e., any CRC20/CRC21 contract with a token mapping, or any contract that has previously minted native tokens to users via the bank precompile) can call:

```
bankPrecompile.transfer(victimAddress, attackerAddress, victimBalance)
```

This causes `bankKeeper.SendCoins` to move native Cosmos tokens from the victim's account to the attacker's account without the victim's consent. Because the denom is derived from `contract.Caller()`, the attacker needs to control a contract whose denom corresponds to tokens the victim holds — which is exactly the case for any legitimate CRC20 contract whose users have converted tokens to native form. [3](#0-2) 

### Likelihood Explanation

Any unprivileged EVM contract can call the bank precompile. A malicious or compromised CRC20 contract (or a contract deployed specifically to exploit this) can immediately drain all holders of the corresponding native denom. The entry path requires no special permissions, no admin keys, and no governance action.

### Recommendation

Enforce that the `sender` argument matches `contract.Caller()`, or remove the `sender` argument entirely and derive `from` from `contract.Caller()`:

```go
// Replace:
sender := args[0].(common.Address)
from := sdk.AccAddress(sender.Bytes())

// With:
from := sdk.AccAddress(contract.Caller().Bytes())
```

If a "transfer on behalf of" pattern is genuinely needed, it must be gated by an explicit on-chain allowance mechanism (similar to ERC-20 `approve`/`transferFrom`) checked inside the precompile before executing `SendCoins`.

### Proof of Concept

1. Alice holds 1,000,000 units of native denom `EVMDenom(crc20Contract.address)` (obtained by converting her CRC20 tokens to native form).
2. The CRC20 contract (or a malicious contract at the same denom) calls:
   ```solidity
   IBankPrecompile(BANK_PRECOMPILE).transfer(alice, attacker, 1_000_000);
   ```
3. The bank precompile unpacks `sender = alice`, `recipient = attacker`, derives `denom = EVMDenom(contract.Caller())`, and calls `bankKeeper.SendCoins(ctx, alice, attacker, coins)` with no authorization check.
4. Alice's entire native token balance is transferred to the attacker.

**Actual result:** `alice` native balance = 0, `attacker` native balance = 1,000,000.
**Expected result:** Transfer should revert — the calling contract is not `alice` and has no approval to move her funds. [4](#0-3)

		}

			if err := bc.bankKeeper.IsSendEnabledCoins(ctx, amt); err != nil {
				return err
			}
			if method.Name == "mint" {
				if err := bc.bankKeeper.MintCoins(ctx, types.ModuleName, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to mint coins in precompiled contract")
				}
				if err := bc.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.ModuleName, addr, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to send mint coins to account")
				}
			} else {
				if err := bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to send burn coins to module")

				if err := bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to burn coins in precompiled contract")

			}
			return nil
		})
		if err != nil {
			return nil, err
		}
		return method.Outputs.Pack(true)
