### Title
`skip_stateful_validations` admits unsigned invoke transactions to the mempool by checking for any account transaction rather than specifically a deploy-account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (signature verification) for invoke transactions with nonce=1 when the account has nonce=0 in state and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction in the pool for the account — not specifically a deploy-account — an attacker who observes a victim's deploy-account in the mempool can submit an invoke with nonce=1 carrying an arbitrary (invalid) signature that bypasses gateway-level signature verification and is admitted to the mempool.

---

### Finding Description

`skip_stateful_validations` is designed to improve UX for the deploy-account + invoke pattern: when a user submits both transactions simultaneously, the invoke (nonce=1) would otherwise fail gateway validation because the account does not yet exist on-chain. [1](#0-0) 

The function's guard condition is:

```
tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)
```

When both are true it calls `account_tx_in_pool_or_recent_block` and, if that returns `true`, returns `true` itself — causing `run_validate_entry_point` to set `execution_flags.validate = false`. [2](#0-1) 

With `validate = false`, `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling the account's `__validate__` entry point: [3](#0-2) 

The mempool check that gates this skip is: [4](#0-3) 

This returns `true` for **any** transaction in the pool for the account — including the victim's deploy-account. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch is circular: the only way a nonce=1 invoke for an account with on-chain nonce=0 could already be in the pool is if it also bypassed `__validate__` through this same path.

**Attack path:**

1. Victim submits a deploy-account transaction; it passes `__validate_deploy__` and enters the mempool.
2. Attacker observes the deploy-account in the mempool.
3. Attacker submits an invoke with nonce=1 for the victim's address, with arbitrary calldata and an invalid/empty signature.
4. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == Nonce(Felt::ONE)` → `true`
   - `account_nonce == Nonce(Felt::ZERO)` → `true`
   - `account_tx_in_pool_or_recent_block(victim_address)` → `true` (victim's deploy-account is in pool)
   - Returns `true`
5. `run_validate_entry_point` is called with `skip_validate = true`; `__validate__` is never called.
6. The attacker's unsigned invoke is admitted to the mempool.

The nonce and resource-bound checks that do run (`validate_state_preconditions`, `validate_by_mempool`) do not verify the signature: [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject unsigned invoke transactions (nonce=1) into the mempool for any account whose deploy-account is pending. The attacker pays no fees because the batcher's `__validate__` call will reject the transaction before block inclusion. Concrete consequences:

- **Mempool pollution / DoS on the deploy+invoke UX flow.** The attacker can repeatedly submit invalid invokes with escalating tips, forcing the victim to continuously outbid them to get their legitimate invoke included. Because the attacker's transactions are rejected at execution time (no fee charged), the attack is free.
- **Displacement of the victim's legitimate invoke.** If the attacker's invalid invoke occupies the nonce=1 slot with a high tip, the victim's legitimate invoke is either queued behind it or must pay a higher tip to replace it via fee escalation.

---

### Likelihood Explanation

The precondition — a deploy-account transaction visible in the mempool — is exactly the scenario this UX feature is designed for, so it is a routine, observable event. No privileged access is required. The attacker only needs to watch the public mempool and submit a single transaction per victim account.

---

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects transaction types, rather than returning `true` for any transaction type. This preserves the intended UX while closing the bypass.

Alternatively, restrict the skip to cases where the deploy-account transaction hash is explicitly provided by the submitter (as the `PyValidator` path does via `deploy_account_tx_hash: Option<TransactionHash>`): [6](#0-5) 

---

### Proof of Concept

```
// State: victim_address has on-chain nonce = 0, no contract deployed.

// Step 1: Victim submits deploy_account (passes __validate_deploy__).
gateway.add_tx(victim_deploy_account_tx, None).await  // → Ok, enters mempool

// Step 2: Attacker submits invoke with nonce=1, garbage signature.
let attacker_invoke = RpcInvokeTransactionV3 {
    sender_address: victim_address,
    nonce: Nonce(Felt::ONE),
    signature: TransactionSignature(vec![Felt::ZERO, Felt::ZERO].into()),
    calldata: calldata![felt!(0xdeadbeef_u64)],
    resource_bounds: valid_bounds,
    ..Default::default()
};
gateway.add_tx(RpcTransaction::Invoke(attacker_invoke), None).await
// → Ok(tx_hash)  ← admitted without __validate__ being called

// Step 3: Batcher later rejects attacker's invoke (no fee charged).
// Step 4: Attacker repeats step 2 indefinitely at zero cost.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
