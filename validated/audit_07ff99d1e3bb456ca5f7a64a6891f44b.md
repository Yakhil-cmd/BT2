### Title
Admin Fee Cap Reduction Not Enforced for Existing Pools — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

When the factory owner reduces `maxAdminSpreadFeeE6` or `maxAdminNotionalFeeE8` via `setFeeCaps`, the function auto-clamps only the global protocol-default storage variables but leaves every existing pool's per-pool admin fee components in `poolFeeConfig` untouched. Those pools continue charging the old, now-above-cap admin fees on every subsequent swap, bypassing the owner's intended fee reduction.

### Finding Description

The fee-cap system has three layers:

1. **Hard limits** — `HARD_MAX_SPREAD_FEE_E6 = 200_000` and `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` are compile-time constants that bound what `setFeeCaps` may accept.
2. **Soft caps** — `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` are mutable storage values that gate `setPoolAdminFees` and `createPool`.
3. **Per-pool config** — `poolFeeConfig[pool].adminSpreadFeeE6` / `adminNotionalFeeE8` are the live values used when `collectFees` splits accrued fees and when `setPoolFees` writes the total to the pool.

`setFeeCaps` correctly clamps the two global protocol-default variables when they exceed the new soft caps: [1](#0-0) 

However, it performs **no analogous clamping** of `poolFeeConfig[pool].adminSpreadFeeE6` or `poolFeeConfig[pool].adminNotionalFeeE8` for any already-deployed pool. The pool's on-chain `spreadFeeE6` (the total fee charged to swappers) is also not updated, because `setPoolFees` is never called: [2](#0-1) 

The clamping of per-pool admin fees only happens as a side-effect of `setPoolProtocolFee`, which the owner must call explicitly for each pool: [3](#0-2) 

`setPoolAdminFees` blocks the pool admin from *raising* fees above the new cap: [4](#0-3) 

But the existing above-cap fee persists passively — the pool admin need not take any action. Every swap on the affected pool continues to charge the old total fee (protocol + legacy-above-cap admin), and every `collectPoolFees` call distributes the admin share at the old above-cap rate: [5](#0-4) 

The pool's stored `spreadFeeE6` is the value actually applied to swappers: [6](#0-5) 

### Impact Explanation

**Medium/High.** Every swap on an affected pool charges users a spread fee whose admin component exceeds the owner's newly-reduced cap. The excess fee is a direct, continuous loss of user swap principal — users pay more than the protocol's own governance rules permit. The pool admin does not need to take any action; the legacy `poolFeeConfig` entry does the bypassing automatically. The owner's risk-management intent (reducing the cap) is silently nullified for all pre-existing pools until `setPoolProtocolFee` is called per-pool.

### Likelihood Explanation

**High.** Reducing admin fee caps is a routine governance action (e.g., responding to competitive pressure or user complaints). The factory already has the machinery to reduce caps; the missing step — retroactively clamping existing pools — is simply absent from `setFeeCaps`. Any pool whose admin previously set fees near the old cap will silently exceed the new cap after the reduction.

### Recommendation

Inside `setFeeCaps`, after updating the cap storage variables, iterate over registered pools (or accept a pool list parameter) and, for each pool whose `poolFeeConfig[pool].adminSpreadFeeE6 > newMaxAdminSpreadFeeE6` (or notional equivalent), clamp the stored value and call `setPoolFees` with the corrected total — mirroring the logic already present in `setPoolProtocolFee` at lines 337–357. Alternatively, add a dedicated `clampPoolAdminFees(address pool)` function that any caller can invoke after a cap reduction, so the enforcement can be applied pool-by-pool without requiring the owner to know the full pool list at cap-reduction time.

### Proof of Concept

```
State before:
  maxAdminSpreadFeeE6 = 200_000  (20 %)
  poolFeeConfig[pool].adminSpreadFeeE6 = 150_000  (15 %)
  pool.spreadFeeE6 = protocolFee + 150_000

Owner calls:
  factory.setFeeCaps(200_000, 50_000, 1_000_000, 1_000_000)
  // newMaxAdminSpreadFeeE6 = 50_000 (5 %)

After setFeeCaps:
  maxAdminSpreadFeeE6 = 50_000  ✓ (cap updated)
  poolFeeConfig[pool].adminSpreadFeeE6 = 150_000  ✗ (NOT clamped)
  pool.spreadFeeE6 = protocolFee + 150_000  ✗ (NOT updated)

Pool admin cannot raise fees further (setPoolAdminFees reverts with AdminFeeTooHigh),
but every swap continues charging 15 % admin spread — 3× the new 5 % cap —
until the owner separately calls setPoolProtocolFee(pool, ...) for this pool.
```
