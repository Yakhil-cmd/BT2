### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Deployed Accounts with Any Pending Transaction — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke transaction with nonce=1 when the account has not yet been deployed (nonce=0 in state) and a `deploy_account` transaction is pending in the mempool. The function uses `account_tx_in_pool_or_recent_block` as a proxy for "deploy_account exists in pool." However, that check returns `true` for **any** account that has **any** transaction in the pool — including a regular invoke with nonce=0. An attacker who already has a deployed account (nonce=0 in state) can plant a valid nonce=0 invoke to satisfy the check, then submit a nonce=1 invoke with an arbitrary/invalid signature that the gateway accepts without running `__validate__`.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` contains the following logic:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect.

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool for that address — not specifically `deploy_account`. An already-deployed account with nonce=0 in state that has a regular invoke (nonce=0) in the pool satisfies all three conditions:

1. `tx.nonce() == Nonce(Felt::ONE)` — attacker crafts the malicious tx with nonce=1
2. `account_nonce == Nonce(Felt::ZERO)` — account is deployed but nonce is still 0 in committed state
3. `account_tx_in_pool_or_recent_block` → `true` — because the attacker's valid nonce=0 invoke is in the pool

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false`, the function returns immediately after `perform_pre_validation_stage` without ever calling `validate_tx` (the `__validate__` entry point):

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

The gateway's `validate_nonce` check does not block this because the default `max_allowed_nonce_gap` is 200, so nonce=1 with account_nonce=0 passes:

```rust
let max_allowed_nonce =
    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
``` [5](#0-4) 

Default `max_allowed_nonce_gap = 200`: [6](#0-5) 

The mempool's `validate_tx` also passes because it only rejects nonces strictly less than the resolved account nonce:

```rust
if tx_nonce < account_nonce {
    return Err(MempoolError::NonceTooOld { ... });
}
``` [7](#0-6) 

The result: the gateway admits the nonce=1 invoke with an invalid/arbitrary signature into the mempool without any signature verification.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid (unauthorized) transactions before sequencing.**

The gateway's stateful validation is the only place where `__validate__` is run before a transaction enters the mempool. By bypassing it, the attacker injects a transaction that was never authorized by the account's signing logic. The invariant "every transaction in the mempool has passed the account's `__validate__`" is broken.

During block execution the blockifier uses `AccountTransaction::new_for_sequencing` which sets `validate: true` and `strict_nonce_check: true`:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        only_query: false,
        charge_fee: enforce_fee(&tx, false),
        validate: true,
        strict_nonce_check: true,
    };
``` [8](#0-7) 

So the invalid tx will fail `__validate__` at execution time and be rejected (no fee charged). The concrete impact is:

- **Mempool pollution / DoS**: The attacker can continuously inject invalid nonce=1 transactions at the cost of one valid nonce=0 transaction per cycle, filling the mempool with transactions that waste batcher resources.
- **Broken admission invariant**: The gateway's security boundary — that only account-authorized transactions enter the mempool — is violated.

---

### Likelihood Explanation

The preconditions are minimal and fully attacker-controlled:

1. The attacker needs any deployed account with nonce=0 in committed state (trivially achievable).
2. The attacker submits one valid nonce=0 invoke (costs one transaction fee).
3. The attacker then submits any number of nonce=1 invokes with invalid signatures; each is admitted without `__validate__`.

No privileged access, no special timing, no race condition required. The attack is deterministic and repeatable.

---

### Recommendation

`skip_stateful_validations` must verify that the account has a **`deploy_account` transaction specifically** in the mempool, not just any transaction. The current `account_tx_in_pool_or_recent_block` API does not distinguish transaction types.

Options:
1. Add a new mempool API `deploy_account_tx_in_pool(address)` that checks specifically for a pending `deploy_account` transaction for the given address.
2. Alternatively, tighten the skip condition: only skip `__validate__` when the account's class hash is not yet set in state (i.e., the contract does not exist at all), which is the true indicator that a `deploy_account` is pending.

The analogous fix in the SIZE report was: *"Do not trust the value of `lowestQuote` when determining the finalize state; use a dedicated state variable."* Here the fix is: do not trust `account_tx_in_pool_or_recent_block` as a proxy for "deploy_account is pending"; use a dedicated check.

---

### Proof of Concept

```
Setup:
  - Account A is deployed on-chain; committed nonce = 0.
  - Attacker controls Account A's private key.

Step 1 — Plant the trigger transaction:
  Submit Invoke(sender=A, nonce=0, calldata=<anything>, signature=<valid>)
  → Gateway: nonce check passes (0 == 0), __validate__ runs, tx accepted into mempool.
  → account_tx_in_pool_or_recent_block(A) now returns true.

Step 2 — Submit the unauthorized transaction:
  Submit Invoke(sender=A, nonce=1, calldata=<arbitrary>, signature=<invalid/empty>)
  → Gateway stateful validator:
      validate_nonce: 0 <= 1 <= 200  ✓
      validate_by_mempool: 1 >= 0    ✓
      skip_stateful_validations:
          tx.nonce() == 1            ✓
          account_nonce == 0         ✓
          account_tx_in_pool_or_recent_block(A) == true  ✓  (due to Step 1)
          → returns true (skip __validate__)
      run_validate_entry_point(skip_validate=true):
          execution_flags.validate = false
          StatefulValidator::perform_validations returns Ok(()) without calling __validate__
  → Nonce=1 tx admitted to mempool with no signature check.

Result:
  The mempool now contains a transaction with an invalid signature that was never
  authorized by the account's __validate__ entry point.
  The gateway's admission invariant is broken.
``` [9](#0-8) [10](#0-9)

        block_info.block_number = block_info.block_number.unchecked_next();

            versioned_constants,

**File:** crates/apollo_gateway/src/stateful_transaction_validator.rs (L429-461)
```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
