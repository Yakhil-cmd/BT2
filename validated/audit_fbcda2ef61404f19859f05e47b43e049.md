### Title
`protocolUnpausePool` Transitions to Admin-Paused State (Level 1) Instead of Active (Level 0), Leaving Swaps Permanently Blocked Until Admin Intervenes — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`protocolUnpausePool` sets the pool's pause level to **1** (admin-paused) rather than **0** (active). Because `whenNotPaused` blocks every `swap` call when `pauseLevel != 0`, a pool that was fully active before a protocol pause remains swap-disabled after the protocol "unpauses" it, until the pool admin separately calls `unpausePool`.

### Finding Description
The pool tracks three pause levels: 0 = active, 1 = paused by admin, 2 = paused by protocol. The `swap` function is gated by `whenNotPaused`, which reverts whenever `pauseLevel != 0`.

`protocolUnpausePool` is intended to remove a protocol-level pause:

```solidity
// MetricOmmPoolFactory.sol lines 399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);  // ← sets to 1, not 0
}
```

It transitions from level 2 → level 1. Level 1 is "paused by admin" and still satisfies `pauseLevel != 0`, so `_checkNotPaused()` continues to revert with `PoolPaused`. The pool is not actually unpaused.

The harmful scenario: a pool is at level 0 (active; admin never paused it). The protocol owner calls `protocolPausePool` (0 → 2). Later, the protocol owner calls `protocolUnpausePool` (2 → 1). The pool is now at level 1 — a state the admin never set — and swaps remain blocked. The admin must separately call `unpausePool` (1 → 0) to restore the pool to active, even though the admin never paused it.

The `protocolPausePool` guard confirms it can fire from either level 0 or level 1:

```solidity
// MetricOmmPoolFactory.sol lines 392-396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

So the protocol can pause from level 0 (active), but its corresponding unpause only reaches level 1, not level 0. The pre-pause state is not restored.

The `whenNotPaused` modifier and `swap` signature confirm the impact:

```solidity
// MetricOmmPool.sol lines 174-177, 217-224
modifier whenNotPaused() { _checkNotPaused(); _; }
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
```

`addLiquidity` and `removeLiquidity` do not carry `whenNotPaused`, so only the swap path is blocked — the pool's primary price-discovery and fee-earning function.

### Impact Explanation
All `swap` calls revert with `PoolPaused` after a protocol pause/unpause cycle on a previously-active pool. The pool's primary function is completely disabled: traders cannot execute, LPs cannot earn spread or notional fees, and the oracle-anchored price discovery mechanism is inoperative.
