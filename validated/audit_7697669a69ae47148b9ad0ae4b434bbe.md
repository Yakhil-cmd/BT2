### Title
Pre-EIP-155 Legacy Transaction Replay Across Chains — (`engine-transactions/src/legacy.rs`, `engine/src/engine.rs`)

### Summary

Aurora Engine accepts pre-EIP-155 legacy Ethereum transactions (those with `v = 27` or `v = 28`). For these transactions, the signed data contains no `chain_id`, so the same raw transaction and signature are cryptographically valid on every EVM chain simultaneously. Aurora's `submit_transaction` path skips the chain-ID guard when `chain_id` is `None`, meaning a transaction broadcast on Ethereum mainnet (or any other chain) can be replayed verbatim on Aurora to execute an identical state transition — including ETH/token transfers — against the sender's Aurora balance.

---

### Finding Description

**Root cause — missing chain binding in signed data**

`LegacyEthSignedTransaction::sender()` in `engine-transactions/src/legacy.rs` decodes the recovery ID and, when `v ∈ {27, 28}`, sets `chain_id = None` and calls `rlp_append_unsigned` with a 6-field RLP list that contains no chain identifier: [1](#0-0) 

`TransactionLegacy::rlp_append_unsigned` confirms: when `chain_id.is_none()` the list has 6 items (nonce, gas_price, gas_limit, to, value, data) — no chain context whatsoever: [2](#0-1) 

**Root cause — chain-ID guard is conditional on `Some`**

In `engine/src/engine.rs`, the chain-ID validation is wrapped in `if let Some(chain_id) = transaction.chain_id`. When the field is `None` (pre-EIP-155), the entire guard is bypassed and execution continues normally: [3](#0-2) 

The `NormalizedEthTransaction` produced from a legacy transaction carries `chain_id: tx.chain_id()`, which returns `None` for `v ∈ {27, 28}`: [4](#0-3) 

The nonce check that follows (`check_nonce`) only verifies that the transaction nonce matches the sender's current Aurora nonce — it does not compensate for the absent chain binding: [5](#0-4) 

---

### Impact Explanation

An attacker who observes a pre-EIP-155 signed transaction on Ethereum mainnet (or any other EVM chain) can submit the identical raw bytes to Aurora's `submit` entrypoint. If the originating account holds a balance on Aurora and its Aurora nonce happens to equal the transaction nonce, Aurora will execute the transaction: transferring ETH, calling contracts, or deploying code — exactly as the user intended on the original chain but without their consent on Aurora. This constitutes **direct theft of user funds** (Critical).

---

### Likelihood Explanation

Pre-EIP-155 transactions are still produced by hardware wallets in compatibility mode, certain DeFi scripts, and older tooling. An attacker passively monitoring public mempools or block explorers on any EVM chain can collect such transactions at zero cost. The nonce must coincide, but for accounts that are newly active on Aurora (nonce 0 or low) the window is realistic. No special privilege is required; any unprivileged actor can call `submit`.

---

### Recommendation

Reject legacy transactions that carry no chain-ID binding. Change the guard in `submit_transaction` from an `if let Some` (which silently passes `None`) to an explicit rejection:

```rust
let tx_chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(tx_chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This forces every submitted transaction to carry an EIP-155 chain-ID that matches Aurora's configured chain, eliminating the cross-chain replay surface entirely.

---

### Proof of Concept

1. On Ethereum mainnet, broadcast a pre-EIP-155 ETH transfer from address `A` (nonce 0, `v = 27`).
2. Ensure address `A` also holds ETH on Aurora with nonce 0 (e.g., freshly funded).
3. Submit the identical RLP-encoded bytes to Aurora's `submit` method.
4. `LegacyEthSignedTransaction::sender()` recovers `A` correctly (the signature is valid on any chain).
5. `chain_id = None` → the `if let Some` guard in `submit_transaction` is skipped.
6. `check_nonce` passes (nonce 0 == 0).
7. Aurora executes the transfer, debiting `A`'s Aurora balance to the mainnet recipient address — which the attacker controls on Aurora. [6](#0-5) [7](#0-6)

impl LegacyEthSignedTransaction {
    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);
        let message_hash = sdk::keccak(rlp_stream.as_raw());
        sdk::ecrecover(message_hash, &super::vrs_to_arr(rec_id, self.r, self.s))
            .map_err(|_| Error::EcRecover)
    }

    /// Returns chain id encoded in `v` parameter of the signature if that was done, otherwise None.
    #[must_use]
    pub const fn chain_id(&self) -> Option<u64> {
        match self.v {
            0..=34 => None,
            _ => Some((self.v - 35) / 2),
        }
    }
