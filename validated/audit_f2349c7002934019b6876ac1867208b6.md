I'll analyze the external bug pattern (inconsistent limit checks between deposit and withdrawal/transfer, where one check uses `amount` and another uses `amount + reward`) and search for a nearcore analog in the target scopes.

### Title
Receipt Size Limit Bypassed via `promise_return` `output_data_receivers` Injection After Validation - (File: `runtime/runtime/src/lib.rs`, `runtime/runtime/src/verifier.rs`)

### Summary

The `max_receipt_size` invariant is enforced only at the moment a new receipt is serialized and validated (`ValidateReceiptMode::NewReceipt`). However, after that check passes, the runtime unconditionally appends `output_data_receivers` entries from the parent receipt into the child receipt when `promise_return` is used. This post-validation mutation can push the receipt's serialized size above `max_receipt_size`. The receipt is then forwarded to the next shard/block and processed under `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check. The invariant is broken: a receipt larger than `max_receipt_size` is created, forwarded, and executed. This is the exact same two-check inconsistency as M-07 — one check uses the value before augmentation, the other uses the augmented value — except here the second check is omitted entirely.

### Finding Description

**Step 1 — Size check at creation (passes).**

In `validate_receipt`, the size check is gated on `ValidateReceiptMode::NewReceipt`:

```rust
// runtime/runtime/src/verifier.rs lines 533–541
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 =
        borsh::object_length(receipt).unwrap().try_into()...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

A contract calls `promise_create` with args sized so that the resulting receipt is exactly `max_receipt_size` bytes. The receipt passes validation.

**Step 2 — Post-validation mutation.**

After the function call executes and the new receipts are validated, the runtime checks whether the parent receipt had `output_data_receivers`. If the child used `promise_return`, the parent's `output_data_receivers` are appended directly into the child receipt's `output_data_receivers` field — **without re-running the size check**:

```rust
// runtime/runtime/src/lib.rs lines 1019–1037
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... =>
                new_action_receipt.output_data_receivers
                    .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
``` [2](#0-1) 

Each `DataReceiver` entry added to `output_data_receivers` is ~40 bytes (a `CryptoHash` + `AccountId`). Even a single appended entry pushes the receipt over `max_receipt_size`.

**Step 3 — Execution skips the size check.**

When the oversized receipt arrives at the destination shard as an incoming receipt, it is validated with `ValidateReceiptMode::ExistingReceipt`:

```rust
// runtime/runtime/src/lib.rs lines 2512–2518
validate_receipt(
    &processing_state.apply_state.config.wasm_config.limit_config,
    receipt,
    protocol_version,
    ValidateReceiptMode::ExistingReceipt,  // size check is skipped
)
``` [3](#0-2) 

The `ExistingReceipt` mode comment explicitly acknowledges this:

```
/// 2) There is a bug which allows to create receipts that are above the size limit. Runtime has
///    to handle them gracefully until the receipt size limit bug is fixed.
///    See https://github.com/near/nearcore/issues/12606 for details.
``` [4](#0-3) 

The receipt is executed without rejection.

**The two-check inconsistency (direct analog to M-07):**

| Stage | What is checked | What is used |
|---|---|---|
| Receipt creation | `size(receipt_without_output_data_receivers) ≤ max_receipt_size` | receipt before augmentation |
| Receipt execution | *(no size check)* | receipt after augmentation |

This is structurally identical to M-07: the first check uses `amount`, the second check uses `amount + reward` — except in nearcore the second check is absent entirely, making the bypass unconditional.

### Impact Explanation

Any unprivileged user can deploy a contract that crafts a `promise_return` scenario to produce receipts exceeding `max_receipt_size`. The `max_receipt_size` limit (4,194,304 bytes on mainnet) is a protocol-level invariant protecting:

- Cross-shard bandwidth (oversized receipts consume more than the budgeted bandwidth)
- State proof sizes (`per_receipt_storage_proof_size_limit` is related)
- Chunk/block production timing

The test `test_max_receipt_size_promise_return` confirms the oversized receipt is forwarded and executed successfully, proving the bypass is reachable in production. [5](#0-4) 

### Likelihood Explanation

Any user who can deploy a contract (no special privilege required) can trigger this. The attack requires:
1. Deploy a contract with a `promise_return` method.
2. Call it from a context where the parent receipt has at least one `output_data_receiver` (i.e., the call is part of a `promise_then` chain).
3. Size the child receipt's args so that `size(child) = max_receipt_size - size(output_data_receivers_to_be_appended)`.

This is a deterministic, reproducible exploit with no randomness or timing dependency.

### Recommendation

Re-validate the receipt size **after** `output_data_receivers` are appended in `apply_action_receipt` (in `runtime/runtime/src/lib.rs` around line 1031–1035). If the post-mutation size exceeds `max_receipt_size`, the function call should fail with `NewReceiptValidationError(ReceiptSizeExceeded)` and the deposit should be refunded to the predecessor, consistent with how other receipt validation errors are handled.

Alternatively, enforce the size check in `ValidateReceiptMode::ExistingReceipt` as well, so that oversized receipts are rejected at execution time rather than silently processed.
