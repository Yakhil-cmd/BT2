### Title
`L1GasPriceScraperConfig::finality` Defaults to Zero — Reorged L1 Gas Prices Corrupt Fee Validation Window - (File: `crates/apollo_l1_gas_price_config/src/config.rs`)

### Summary

The `finality` field in `L1GasPriceScraperConfig` defaults to `0` in both the Rust code and the production base `config_schema.json`. With `finality = 0`, the L1 gas price scraper reads from the very tip of the Ethereum chain — blocks that have not yet achieved any reorg safety depth. When an L1 reorg occurs, already-scraped gas prices from reorganized blocks remain in the provider's ring buffer for the entire polling interval before the reorg is detected. During that window, the incorrect L1 gas prices propagate into the L2 gas price used for gateway fee validation, causing valid transactions to be rejected or under-priced transactions to be admitted.

### Finding Description

`L1GasPriceScraperConfig::finality` is the number of L1 blocks the scraper subtracts from the chain tip before reading gas prices:

```rust
// crates/apollo_l1_gas_price/src/l1_gas_price_scraper.rs
let latest_l1_block_number = latest_l1_block_number
    .checked_sub(self.config.finality)  // finality = 0 → reads from tip
    ...
``` [1](#0-0) 

The Rust `Default` implementation hard-codes `finality: 0`: [2](#0-1) 

The production base schema `config_schema.json` also ships with `0`: [3](#0-2) 

The reorg-detection guard in `assert_no_l1_reorgs` only checks whether the *next* block's `parent_hash` matches the *last scraped* block's hash: [4](#0-3) 

This means: if the scraper has already advanced past blocks N through M in a previous polling cycle, and a reorg later reorganizes blocks N′ through M (N′ < M), those blocks' gas prices are already committed to the ring buffer. The guard fires only when the scraper next tries to read block M+1 — one full polling interval later. Until then, the corrupted prices remain live.

The deployment-specific overrides do raise `finality` to `10`: [5](#0-4) 

But `10` is not enforced by any `#[validate(range(min = ...))]` attribute on the field, and the base schema default of `0` is what any operator sees who does not apply the deployment overlay. The `L1GasPriceProviderConfig` has a `#[validate(range(min = 1))]` guard on `number_of_blocks_for_mean` but no analogous guard exists for `finality`: [6](#0-5) 

The corrupted L1 gas prices flow directly into the L2 gas price used by the stateful gateway validator: [7](#0-6) 

### Impact Explanation

Incorrect L1 gas prices in the ring buffer produce a wrong mean, which the consensus orchestrator uses to set the L2 gas price for the next block. The stateful validator then compares each incoming transaction's `max_price_per_unit` against that wrong L2 gas price. A reorg that inflates the sampled prices causes valid transactions to be rejected at the gateway; a reorg that deflates them causes under-priced transactions to be admitted. Both outcomes are incorrect admission decisions before sequencing.

### Likelihood Explanation

Ethereum mainnet reorgs deeper than 10 blocks are rare but documented. With the base-schema default of `finality = 0`, any deployment that omits the deployment overlay is fully exposed to even single-block reorgs. The polling interval for the deployment config is 120 seconds, meaning a reorg that occurs just after a scraping cycle leaves corrupted prices live for up to two minutes.

### Recommendation

1. Add a `#[validate(range(min = 1))]` (or a higher minimum, e.g., `min = 10`) to the `finality` field in `L1GasPriceScraperConfig` so that `0` is rejected at startup.
2. Raise the default in both `L1GasPriceScraperConfig::default()` and `config_schema.json` to match the deployment overlay value of `10` (or higher).
3. Consider adding a retroactive reorg check: after each polling cycle, re-verify the hashes of the most recently scraped `finality`-many blocks before committing their prices to the ring buffer.

### Proof of Concept

1. Deploy the sequencer using only `config_schema.json` defaults (no deployment overlay), so `finality = 0` and `polling_interval = 1s`.
2. Submit a transaction whose `max_price_per_unit` is just above the current L2 gas price threshold.
3. Trigger an L1 reorg (e.g., on a test network) that replaces the most recently scraped block with one whose `base_fee_per_gas` is significantly higher.
4. Observe that the scraper does not detect the reorg until the *next* block is scraped (one polling interval later).
5. During that interval, the gateway's `validate_resource_bounds` call reads the inflated L2 gas price derived from the reorged block and rejects the previously-valid transaction with `MaxGasPriceTooLow`. [2](#0-1) [3](#0-2) [8](#0-7) [4](#0-3) [7](#0-6)

    async fn assert_no_l1_reorgs(
        &self,
        new_header: &L1BlockHeader,
    ) -> L1GasPriceScraperResult<(), B> {
        // If no last block was processed, we don't need to check for reorgs.
        let Some(ref last_header) = self.last_l1_header else {
            return Ok(());
        };

        if new_header.parent_hash != last_header.hash {
            L1_GAS_PRICE_SCRAPER_REORG_DETECTED.increment(1);
            return Err(L1GasPriceScraperError::L1ReorgDetected {
                reason: format!(
                    "Last processed L1 block hash, {}, for block number {}, is different from the \
                     hash stored, {}",
                    new_header.parent_hash, last_header.number, last_header.hash,
                ),
            });
        }

        Ok(())
    }

    async fn latest_l1_block_number(&mut self) -> L1GasPriceScraperResult<L1BlockNumber, B> {
        let latest_l1_block_number = self
            .base_layer
            .latest_l1_block_number()
            .await
            .map_err(L1GasPriceScraperError::BaseLayerError)?;
        let latest_l1_block_number = latest_l1_block_number
            .checked_sub(self.config.finality)
            .ok_or(L1GasPriceScraperError::LatestBlockNumberTooLow {
                latest_l1_block_number,
                finality: self.config.finality,
            })?;
