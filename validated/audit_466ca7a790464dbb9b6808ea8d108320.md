### Title
`MsgTurnBridge` Handler Is a Permission-Less No-Op — Bridge Circuit Breaker Permanently Broken - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` gRPC message handler, which is the designated emergency circuit breaker for the Gravity Bridge, is implemented as a stub that performs no permission check and makes no state change. Any unprivileged address can submit `MsgTurnBridge` and receive a success response, while the bridge state is never modified. The `CanTurnBridge` permission bit is defined and tracked in the permission system but is never enforced in the actual handler.

### Finding Description

`MsgTurnBridge` is a registered Cosmos SDK message type with a defined proto surface, a CLI command, and a dedicated permission bit (`CanTurnBridge`). The permission system in `x/cronos/keeper/permissions.go` defines `CanTurnBridge` as a first-class permission, and the query endpoint `Permissions` correctly reports whether an address holds it. The ADR-009 design document explicitly states that `MsgTurnBridge` is one of the two messages that must be restricted to permissioned addresses.

However, the actual server-side handler in `x/cronos/keeper/msg_server.go` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly gates on `HasPermission`:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The `CanTurnBridge` constant is defined alongside `CanChangeTokenMapping` but is never consumed by the handler: [3](#0-2) 

The `HasPermission` function that would enforce it exists and is correct: [4](#0-3) 

The query layer even advertises `CanTurnBridge` as a meaningful capability: [5](#0-4) 

### Impact Explanation

Two distinct High-severity impacts are present simultaneously:

**1. Bypass of Cronos permission/admin authorization checks (High)**
The `CanTurnBridge` permission is never checked. Any unprivileged address can submit `MsgTurnBridge` and receive a success response (`nil` error). The entire permission system for this message is silently bypassed.

**2. Permanent inability to disable the Gravity Bridge (High)**
Because the handler is a no-op, the bridge state is never written regardless of who calls it. The Gravity Bridge emergency circuit breaker — the mechanism designed to halt bridge operations in case of an exploit or emergency — is permanently non-functional. Honest validators and the admin cannot disable the bridge through the intended protocol path. This maps directly to the allowed High impact: *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows."* [6](#0-5) 

### Likelihood Explanation

The entry path is fully unprivileged. `MsgTurnBridge` is a standard Cosmos SDK transaction reachable by any address on the network. `ValidateBasic` only checks that `Sender` is a valid bech32 address: [7](#0-6) 

No special role, key, or condition is required to trigger this. The broken circuit breaker impact is permanent and unconditional — it exists from genesis regardless of attacker activity.

### Recommendation

Implement the `TurnBridge` handler body following the same pattern as `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` on failure.
2. Write the `enable` flag to a persistent store key (e.g., a new `KeyBridgeEnabled` entry in the KV store).
3. Read that flag in the Gravity Bridge end-blocker and EVM hooks to gate bridge operations. [1](#0-0) 

### Proof of Concept

```
# Any unprivileged address can call TurnBridge and receive code=0 (success)
cronosd tx cronos turn-bridge false \
  --from <any_unprivileged_key> \
  --chain-id cronos_777-1 -y

# Expected: code != 0 (ErrUnauthorized)
# Actual:   code == 0, bridge state unchanged, circuit breaker silently fails
```

The integration test `test_gravity_turn_bridge` confirms the CLI path is live and that `from_="community"` (unprivileged) is expected to fail with `code != 0`: [8](#0-7) 

However, because the handler is a stub, even the `from_="validator"` (permissioned) call at line 664 succeeds only vacuously — no bridge state is written, and the subsequent assertions that the bridge is actually off rely on side-effects that the no-op handler never produces.

func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```
