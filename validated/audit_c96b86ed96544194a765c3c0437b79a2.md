## Analysis

The question requires tracing the exact execution path through `BankContract.Run` / `TransferMethodName` and verifying whether the IBC escrow account is protected.

**Key code facts established:**

**1. `TransferMethodName` handler — no sender authorization check** [1](#0-0) 

The handler unpacks `sender` (args[0]) as a free caller-supplied address, derives `denom = EVMDenom(contract.Caller())` (i.e. `evm/<callerContractAddress>`), then calls `bankKeeper.SendCoins(ctx, from, to, ...)`. The only guard is `checkBlockedAddr(to)` — checking the **recipient** is not a module account. There is **no check** that `from == contract.Caller()` or that the sender has authorized the transfer.

**2. `checkBlockedAddr` only covers module accounts** [2](#0-1) 

`BlockedAddr` is populated from `maccPerms` (module accounts only). [3](#0-2) 

IBC escrow accounts are **not** module accounts — they are derived from `ibctransfertypes.GetEscrowAddress(portID, channelID)` and are plain accounts. They are not in `BlockedAddr`.

**3. `evm/` tokens can reach the IBC escrow account**

A user holding `evm/A` tokens (minted via `bank.mint`) can call `MsgTransfer` directly on the IBC transfer module with those tokens. The IBC transfer module accepts any denom and escrows source tokens in the escrow account. The Cronos-specific `IbcTransferCoins` guard that rejects `evm/` denoms applies only to the EVM-hook path (`SendToIbcHandler`), not to direct `MsgTransfer` messages. [4](#0-3) 

**4. The exploit path is concrete**

```
1. Deploy Contract A (attacker-controlled)
2. Call bank.mint(attacker, N) from Contract A → attacker holds N evm/A tokens
3. Call MsgTransfer(evm/A, N, destChain, recipient) → IBC escrow account holds N evm/A tokens
4. Before ack/timeout, call bank.transfer(escrowAccount, attacker, N) from Contract A
   → denom = evm/A, from = escrowAccount (not blocked), to = attacker (not blocked)
   → bankKeeper.SendCoins succeeds; escrow drained
5. Destination chain receives packet, mints N tokens to recipient
6. Attacker holds N evm/A on Cronos AND N tokens on destination chain → double-spend
```

### Title
Unauthorized Drain of IBC Escrow Account via Bank Precompile `transfer` — (`x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract.Run` / `TransferMethodName` handler allows any EVM contract to move `evm/<contract>` tokens **from any arbitrary address** by supplying it as the `sender` argument. There is no check that the sender equals the calling contract. IBC escrow accounts are not in `bank.BlockedAddr`, so a contract can drain its own-denom tokens from the IBC escrow account before the packet is acknowledged, causing a double-spend.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case:

```go
sender := args[0].(common.Address)   // attacker-controlled
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil { ... }  // only recipient checked
denom := EVMDenom(contract.Caller())                   // evm/<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`from` is never validated against `contract.Caller()`. Any contract `A` can therefore call `bank.transfer(X, attacker, N)` to move `evm/A` tokens out of **any** account `X` that holds them — including the IBC transfer module's per-channel escrow accounts, which are plain (non-module) accounts and thus not in `BlockedAddr`.

### Impact Explanation
**Critical — unauthorized transfer / double-spend of IBC-escrowed `evm/` tokens.**

An attacker can:
1. Mint `evm/A` tokens to themselves via `bank.mint`.
2. Initiate an IBC transfer, causing the escrow account to hold `evm/A` tokens.
3. Before acknowledgement, drain the escrow via `bank.transfer(escrowAccount, attacker, N)`.
4. The destination chain still mints the tokens on receipt of the packet.

Result: the attacker recovers their tokens on Cronos while the destination chain also credits them — a complete double-spend. Additionally, any other holder of `evm/A` tokens (not just escrow accounts) can be drained by Contract A at any time, constituting unrestricted theft of `evm/` token balances.

### Likelihood Explanation
The exploit requires only: deploying a contract, minting tokens, sending an IBC packet, and calling the bank precompile — all unprivileged, on-chain operations. No leaked keys, governance, or external assumptions are needed.

### Recommendation
Add a sender authorization check in the `TransferMethodName` handler. The `from` address must equal `sdk.AccAddress(contract.Caller().Bytes())`:

```go
callerAddr := sdk.AccAddress(contract.Caller().Bytes())
if !from.Equals(callerAddr) {
    return nil, errors.New("transfer sender must be the calling contract")
}
```

This matches the design intent (each contract controls only its own `evm/<contract>` bank account) and is consistent with how `mint` and `burn` implicitly scope to `contract.Caller()`.

### Proof of Concept
```solidity
// AttackerContract.sol — deployed at address A

    // Step 3: drain escrow before ack
```
