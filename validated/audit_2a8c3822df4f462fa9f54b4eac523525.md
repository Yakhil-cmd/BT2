Looking at the external bug class: **an identifier is computed without including all relevant parameters, allowing frontrunning to create a duplicate entry with malicious values, permanently blocking the legitimate creation.**

I need to find the analog in Metric OMM's factory/deployer path.

**Key observation in `MetricOmmPoolDeployer.deploy()`:**

The CREATE2 pool address is determined by `params.salt` + the ABI-encoded constructor arguments of `MetricOmmPool`. The `priceProviderTimelock` is **not** a constructor argument — it is stored in the factory mapping after deployment. The only constructor-level signal derived from it is the boolean `immutablePriceProvider = (params.priceProviderTimelock == type(uint256).max)`.

This means two `createPool` calls with the same `salt` and all other parameters identical, but with different **non-max** `priceProviderTimelock` values (e.g., `30 days` vs `0`), produce the **same** CREATE2 address.

### Title
Frontrunning `createPool` with `priceProviderTimelock = 0` permanently bypasses the price-provider timelock — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.createPool()` stores `priceProviderTimelock` in a factory mapping after deployment, but this value is not encoded into the CREATE2 address. An attacker can frontrun any `createPool` transaction with an identical call except `priceProviderTimelock = 0`, deploying the pool at the same address with no timelock. The legitimate user's transaction then reverts permanently, and the pool is irrevocably misconfigured.

### Finding Description

In `createPool`, the pool address is computed via CREATE2 inside `MetricOmmPoolDeployer.deploy()`:

```solidity
pool = address(
  new MetricOmmPool{salt: params.salt}(
    params.factory,
    params.admin,
    params.adminFeeDestination,
    params.token0,
    params.token1,
    params.priceProvider,
    params.extensions,
    params.extensionOrders,
    immutablePriceProvider,   // ← bool: only true when timelock == type(uint256).max
    ...
  )
);
``` [1](#0-0) 

The `priceProviderTimelock` itself is **not** a constructor argument. It is stored in the factory only after the pool is deployed:

```solidity
priceProviderTimelock[pool] = params.priceProviderTimelock;
``` [2](#0-1) 

The only constructor-level distinction derived from `priceProviderTimelock` is:

```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
``` [3](#0-2) 

For any two calls where both use a non-`type(uint256).max` timelock (e.g., `30 days` vs `0`), `immutablePriceProvider = false` in both cases, so the constructor arguments are identical, and the CREATE2 address is identical.

An attacker who sees a `createPool` transaction in the mempool can frontrun it with `priceProviderTimelock = 0` and all other parameters copied verbatim. The attacker's transaction deploys the pool at the same address and stores `priceProviderTimelock[pool] = 0`. The legitimate user's transaction then reverts because the CREATE2 address already contains code.

There is no function in the factory to update `priceProviderTimelock[pool]` after creation, so the misconfiguration is permanent. [4](#0-3) 

With `priceProviderTimelock[pool] = 0`, `proposePoolPriceProvider` sets:

```solidity
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp + 0
``` [5](#0-4) 

And `executePoolPriceProviderUpdate` checks:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
``` [6](#0-5) 

With `timelock = 0`, `execAfter = block.timestamp`, so the check passes immediately. The price provider can be proposed and executed in the same block, completely bypassing the intended delay.

### Impact Explanation

The pool is permanently created with `priceProviderTimelock = 0`. The pool admin can change the price provider in the same block as the proposal, with no window for LPs to observe and exit. A compromised pool admin key can immediately redirect the price feed to a malicious oracle, causing bad-price execution on all subsequent swaps and draining LP principal. This is a direct loss of user funds via an admin-boundary break: the timelock cap is bypassed by an unprivileged frontrunner, not by the admin themselves.

### Likelihood Explanation

Any attacker with mempool visibility can execute this attack at the cost of gas only. The attack is most impactful against pools explicitly configured with a meaningful timelock (e.g., 7 days, 30 days) as a user-safety guarantee. The attacker copies all visible parameters from the pending transaction and substitutes `priceProviderTimelock = 0`.

### Recommendation

Include `priceProviderTimelock` in the CREATE2 address computation by passing it as a constructor argument to `MetricOmmPool` (even if stored only as an immutable). Alternatively, incorporate it into `params.salt` at the factory level before calling `deploy`:

```diff
- pool = MetricOmmPoolDeployer(poolDeployer).deploy(
-   MetricOmmPoolDeployer.DeployParams({ salt: params.salt, ... })
- );
+ bytes32 effectiveSalt = keccak256(abi.encode(params.salt, params.priceProviderTimelock));
+ pool = MetricOmmPoolDeployer(poolDeployer).deploy(
+   MetricOmmPoolDeployer.DeployParams({ salt: effectiveSalt, ... })
+ );
```

This ensures that two calls with different `priceProviderTimelock` values always produce different pool addresses, eliminating the frontrunning collision.

### Proof of Concept

1. Alice submits `createPool` with `salt = bytes32(0)`, `priceProviderTimelock = 30 days`, and all other parameters set legitimately.
2. Bob (attacker) sees Alice's transaction in the mempool. Bob submits the identical call with `priceProviderTimelock = 0` and a higher gas price.
3. Bob's transaction executes first. `immutablePriceProvider = false` (same as Alice's). The pool is deployed at the same CREATE2 address. The factory stores `priceProviderTimelock[pool] = 0`.
4. Alice's transaction executes next. `new MetricOmmPool{salt: params.salt}(...)` targets the same address, which already has code. The EVM reverts. Alice's `createPool` fails.
5. The pool is now live with `priceProviderTimelock[pool] = 0`. Bob (or anyone who later compromises the pool admin key) calls `proposePoolPriceProvider(pool, maliciousOracle)` followed immediately by `executePoolPriceProviderUpdate(pool)` in the same block. The malicious oracle returns an arbitrary price. All subsequent swaps execute at the attacker-controlled price, draining LP funds.

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
