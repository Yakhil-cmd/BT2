### Title
Unhandled `__CronosSendToEthereum` Event Causes Permanent Token Burn With No Gravity Bridge Credit — (`contracts/src/ModuleCRC20.sol`, `x/cronos/keeper/evm_hooks.go`)

### Summary

`send_to_ethereum` in `ModuleCRC20.sol` is callable by any unprivileged user, unconditionally burns the caller's tokens, and emits `__CronosSendToEthereum`. No Go-side handler for this event ID is registered in `LogProcessEvmHook`. The hook silently skips unhandled events. The result is permanent token destruction with no corresponding gravity bridge outbound transfer.

### Finding Description

**Step 1 — Entrypoint (no access control):**

`send_to_ethereum` has no `require(msg.sender == module_address)` guard or any other permission check. Any EVM caller can invoke it directly. [1](#0-0) 

**Step 2 — Unconditional burn:**

The function calls `unsafe_burn(msg.sender, add(amount, bridge_fee))` before emitting the event. The burn is not conditional on any downstream handler succeeding. [2](#0-1) 

**Step 3 — No handler registered:**

The `evmhandlers/` directory contains exactly four handlers: `send_to_account.go`, `send_to_ibc.go`, `send_cro_to_ibc.go`, `send_to_ibc_v2.go`. There is no `send_to_ethereum.go` or any handler whose `EventID()` matches the `__CronosSendToEthereum` topic.



**Step 4 — Silent skip in `PostTxProcessing`:**

When `log.Topics[0]` has no registered handler, the hook does `continue` — no error, no revert, no state rollback. [3](#0-2) 

**Step 5 — No gravity bridge integration visible:**

A search of `app/app.go` for `gravity`, `Gravity`, `SendToEthereum` returns zero matches. There is no evidence of a separate gravity bridge EVM hook being registered that would independently process `__CronosSendToEthereum` logs.

### Impact Explanation

Any user who calls `send_to_ethereum(recipient, amount, bridge_fee)` on a mapped `ModuleCRC20` contract loses `amount + bridge_fee` tokens permanently. The EVM state reflects the burn (balance reduced, `totalSupply` reduced), but no gravity bridge `SendToEthereum` message is ever created. The user receives nothing on the Ethereum side. This is a permanent, irreversible loss of user funds triggered by a normal, expected user action on a publicly callable function.

This fits **Critical** impact: unauthorized/unintended burn with no corresponding bridge credit — a direct accounting violation where tokens are destroyed without the protocol-intended counterpart action.

### Likelihood Explanation

- The function is publicly callable with no guard.
- `ModuleCRC20` contracts are the standard token contract for gravity-bridged assets (denom prefix `gravity0x`).
- Any user attempting to bridge tokens back to Ethereum via the documented `send_to_ethereum` path triggers the loss.
- No special conditions, admin keys, or race conditions are required.

### Recommendation

Either:
1. Register a Go-side `SendToEthereumHandler` in `LogProcessEvmHook` (and in `app.go`) that creates the corresponding gravity bridge outbound transfer when `__CronosSendToEthereum` is emitted, **or**
2. Remove `send_to_ethereum` from `ModuleCRC20.sol` if the gravity bridge integration is intentionally not supported in this version, to prevent users from calling a function that silently destroys their funds.

The burn must never precede a confirmed handler dispatch. Consider restructuring so the burn only occurs after the gravity bridge message is successfully enqueued (or use a check-effects pattern where the EVM call reverts if the hook fails).

### Proof of Concept

1. Deploy/use an existing `ModuleCRC20` contract mapped to a `gravity0x...` denom.
2. Call `send_to_ethereum(attacker_eth_addr, 1000, 0)` from any EOA holding ≥1000 tokens.
3. Observe: `balanceOf[msg.sender]` decreases by 1000, `totalSupply` decreases by 1000, `Burn` event emitted.
4. Check gravity bridge module state: no outbound transfer batch entry exists for `attacker_eth_addr`.
5. Check `PostTxProcessing` execution: `__CronosSendToEthereum` log topic has no handler in `h.handlers`, loop `continue`s, function returns `nil`.
6. Result: 1000 tokens permanently destroyed, zero Ethereum-side credit. [1](#0-0) [4](#0-3)

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
