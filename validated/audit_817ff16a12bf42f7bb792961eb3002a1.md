### Title
`StatefulTransactionValidator::validate_resource_bounds` Checks Transaction L2 Gas Price Against Previous Block Price Instead of Next Block Price, Causing Incorrect Gateway Admission Decisions - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway stateful validator checks whether an incoming transaction's `max_price_per_unit` for L2 gas meets a minimum threshold derived from the **previous (latest committed) block's** L2 gas price. However, the transaction will be executed in the **next block**, whose L2 gas price is computed via EIP-1559 mechanics from the previous block and may differ. This temporal mismatch causes the gateway to incorrectly reject valid transactions (when the next block price is lower) and incorrectly admit invalid transactions (when the next block price is higher). The developer acknowledged this in a TODO comment at the exact site of the bug.

---

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`, the threshold for the L2 gas price check is sourced from `gateway_fixed_block_state_reader.get_block_info()`, which returns the **latest committed block's** gas prices:

The `GatewayFixedBlockSyncStateClient` is constructed from `latest_block_number` and its `get_block_info` returns that block's committed gas prices: [3](#0-2) 

The factory always pins both the state reader and the fixed-block reader to the same `latest_block_number`: [4](#0-3) 

The **next block's** L2 gas price is computed by `calculate_next_base_gas_price` using EIP-1559 mechanics and can deviate from the previous block's price by up to `1/gas_price_max_change_denominator` per block: [5](#0-4) 

The blockifier's `check_fee_bounds` (called during actual execution) uses the **actual next block's** gas prices from the `BlockContext`, not the previous block's prices: [6](#0-5) 

The gateway's `run_validate_entry_point` correctly increments the block number by 1 (`block_info.block_number.unchecked_next()`) and uses the previous block's gas prices as the block context for blockifier validation — but the L2 gas price in that context is still the previous block's price, not the EIP-1559-adjusted next block price: [7](#0-6) 

---

### Impact Explanation

Two concrete failure modes arise:

**False rejection (valid transaction rejected):** When the next block's L2 gas price is lower than the previous block's (decreasing congestion), a transaction with `max_price_per_unit` satisfying `next_price <= max_price < previous_price` passes the blockifier's fee check but fails the gateway's threshold check. The transaction is permanently rejected at the gateway even though it would execute successfully. Users cannot submit valid transactions during periods of decreasing gas prices.

**False admission (invalid transaction admitted):** When the next block's L2 gas price is higher than the previous block's (increasing congestion), a transaction with `max_price_per_unit` satisfying `previous_price <= max_price < next_price` passes the gateway check but fails the blockifier's `check_fee_bounds`. The transaction is admitted to the mempool and wastes sequencer resources during block building before being rejected.

Both outcomes match the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The L2 gas price changes every block via EIP-1559. Any block with gas usage above or below the target produces a price change. The discrepancy between previous and next block price is bounded by `1/gas_price_max_change_denominator` but is non-zero in normal operation. Any user submitting a transaction with `max_price_per_unit` in the gap between the two prices triggers the incorrect decision. No special privileges are required.

---

### Recommendation

Replace the previous block's L2 gas price with the computed next block's L2 gas price in `validate_resource_bounds`. The next block's price can be computed by calling `calculate_next_l2_gas_price_for_fin` (already used in the consensus orchestrator) with the previous block's price and the gas used in the previous block. The TODO comment at line 229 already identifies this fix:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
```

The correct threshold is `calculate_next_base_gas_price(previous_block_l2_gas_price, previous_block_gas_used, gas_target, min_gas_price)`.

---

### Proof of Concept

1. Observe the current L2 gas price from the latest committed block header: `P_prev = 100 FRI`.
2. The previous block was at 90% of gas target, so EIP-1559 computes `P_next ≈ 94 FRI` (price decreases).
3. Submit an invoke transaction (v3, `AllResources`) with `l2_gas.max_price_per_unit = 96 FRI`.
4. Gateway check: `96 >= 100 * 100% = 100` → **FAIL** → transaction rejected with `GAS_PRICE_TOO_LOW`.
5. However, the blockifier check for the next block would compute: `96 >= 94` → **PASS**.
6. The transaction is valid for the next block but is incorrectly rejected at the gateway.

Conversely, with a rising price (`P_next = 106 FRI`), a transaction with `max_price_per_unit = 102 FRI` passes the gateway (`102 >= 100`) but fails the blockifier (`102 < 106`), entering the mempool and wasting block-building resources.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
```
