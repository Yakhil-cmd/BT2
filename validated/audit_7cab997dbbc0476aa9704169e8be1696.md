Audit Report

## Title
Silent ICP Burn Failure in CMC Allows Cycles Minting Without Corresponding ICP Destruction - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister deliberately swallows all ICP ledger burn errors, returning `()` regardless of outcome. After cycles are deposited or a canister is created, the three notify paths (`process_top_up`, `process_create_canister`, `process_mint_cycles`) call `burn_and_log` and unconditionally return `Ok(...)`, causing the block index to be permanently sealed as successfully processed even when the ICP burn never occurred. This breaks the ICP↔cycles conservation invariant: cycles are outstanding with no corresponding ICP destruction, and no recovery path exists.

## Finding Description
The root cause is in `burn_and_log` at lines 2014–2049 of `rs/nns/cmc/src/main.rs`. The function makes a protobuf call to the ICP ledger to transfer ICP from a CMC subaccount to the minting account (which destroys it). On any call-level error, it only prints and returns `()`:

```rust
Err((code, err)) => {
    let code = code as i32;
    print(format!("{msg} failed with code {code}: {err:?}"))
}
```

The design is explicitly documented at line 2015: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."*

All three callers treat `burn_and_log` as fire-and-forget:
- `process_top_up` (line 2001): calls `burn_and_log`, then returns `Ok(cycles)`
- `process_create_canister` (line 1945): calls `burn_and_log`, then returns `Ok(canister_id)`
- `process_mint_cycles` (line 1968): calls `burn_and_log`, then returns `Ok(NotifyMintCyclesSuccess{...})`

Because `burn_and_log` never propagates an error, the caller always returns `Ok(...)`. The `is_transient_error` guard at lines 1219–1221 only removes the block from `blocks_notified` when `process_top_up` itself returns a transient error — which it never does when the burn fails, since the burn failure is invisible to it. The block is therefore permanently sealed as `NotifiedTopUp(Ok(...))`, `NotifiedCreateCanister(Ok(...))`, or `NotifiedMint(Ok(...))`, preventing any retry of the notification.

The ICP remains in the CMC-controlled subaccount (keyed by `canister_id` or `controller`). The CMC has no heartbeat, sweep function, or retry queue for failed burns. The stuck ICP is inaccessible to the user and unrecoverable without a governance-controlled CMC upgrade.

## Impact Explanation
This constitutes illegal minting of cycles: cycles are created and delivered without the corresponding ICP being destroyed. The ICP↔cycles exchange rate invariant — every cycle minted must correspond to an equal-value ICP burn — is violated. The total ICP supply is higher than it should be relative to cycles outstanding. This matches the High impact category: *"Significant NNS/CMC security impact with concrete user or protocol harm"* — specifically, a permanent, irrecoverable conservation failure in the ICP ledger/cycles system with no on-chain audit trail beyond a canister `print` log.

## Likelihood Explanation
The trigger is a failed inter-canister call from the CMC to the ICP ledger during the `burn_and_log` step. This can occur when the ICP ledger is being upgraded (a routine, recurring event on mainnet), when the ledger returns a `TemporarilyUnavailable` response under load, or on any replica-level reject. The entry path is fully unprivileged: any principal can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` after sending ICP to the appropriate CMC subaccount. The attacker does not need to cause the ledger failure — it can occur naturally. The window is narrow (the burn call must fail after cycles are already deposited), but the condition is reachable without any privileged access and repeatable across every ledger upgrade cycle.

## Recommendation
The safest fix is to record failed burns in stable state and retry them. Specifically:

1. If `burn_and_log` fails, persist the pending burn (subaccount + amount) in a stable `Vec` or `BTreeMap` in CMC state.
2. A periodic heartbeat retries all pending burns against the ledger.
3. Only after a burn is confirmed should the entry be removed from the pending queue.

If the current seal-on-success semantics must be preserved for idempotency, at minimum the failed burn must be recorded in an on-chain, auditable stable structure (not just a `print`), and a governance-controlled method should exist to retry or account for unburned ICP. The `print`-only logging is insufficient because canister logs are ephemeral and not certified state.

## Proof of Concept
A deterministic integration test using PocketIC can reproduce this:

1. Deploy a mock ICP ledger that returns a `TemporarilyUnavailable` error on the `send_pb` call (simulating a ledger upgrade window).
2. Send 10 ICP to the CMC's top-up subaccount for canister `C`.
3. Call `notify_top_up { block_index: B, canister_id: C }`.
4. CMC fetches the transaction, verifies it, calls `deposit_cycles` — cycles are deposited to `C` (succeeds).
5. CMC calls `burn_and_log` — mock ledger returns error — `burn_and_log` prints and returns `()`.
6. CMC records `NotifiedTopUp(Ok(cycles))` in `blocks_notified`.
7. Assert: canister `C` has received the cycles; the 10 ICP remain in the CMC subaccount; calling `notify_top_up` with block index `B` again returns the cached `Ok` without re-attempting the burn.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
