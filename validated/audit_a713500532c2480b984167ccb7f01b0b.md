### Title
Overly Broad `skip_stateful_validations` Allows Invoke Transactions with Invalid Signatures to Bypass Gateway `__validate__` and Enter the Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is intended to skip the `__validate__` entry-point check only when a `deploy_account` transaction is pending for an account. However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type from the sender address, not exclusively `deploy_account` transactions. An unprivileged attacker who controls an existing deployed account can exploit this by first submitting a valid nonce-0 invoke transaction, then submitting a nonce-1 invoke transaction with an arbitrary/invalid signature. The gateway skips `__validate__` for the second transaction and admits it to the mempool, breaking the invariant that every admitted invoke transaction has passed account signature verification.

### Finding Description

The gateway stateful path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`: [1](#0-0) 

The skip condition fires when:
1. The transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain state nonce)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The comment in the code asserts this is sufficient to confirm a `deploy_account` is pending: [2](#0-1) 

But `account_tx_in_pool_or_recent_block` is implemented as: [3](#0-2) 

And `MempoolState::contains_account` checks only whether the address appears in the staged or committed maps — with no regard to transaction type: [4](#0-3) 

The mempool test confirms that `account_tx_in_pool_or_recent_block` returns `true` for any transaction type from the address, including plain invoke transactions: [5](#0-4) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false` and the blockifier's `StatefulValidator` skips the `__validate__` call entirely: [6](#0-5) [7](#0-6) 

### Impact Explanation

An invoke transaction carrying an invalid or forged signature is admitted to the mempool without passing the account's `__validate__` entry point. This breaks the gateway admission invariant. The transaction will revert during block execution (the batcher re-runs `__validate__` with `execution_flags.validate = true`), but it still occupies mempool and block space and forces the sequencer to include a reverted transaction. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The attacker only needs to control an existing deployed account (nonce = 0 on-chain). They submit a valid nonce-0 invoke transaction first (which passes all checks), then immediately submit a nonce-1 invoke transaction with an arbitrary signature. Both conditions (`tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block == true`) are trivially satisfied. No privileged access is required.

### Recommendation

Change `skip_stateful_validations` to verify that the transaction in the pool for the sender address is specifically a `deploy_account` transaction, not just any transaction. The mempool's `account_tx_in_pool_or_recent_block` API should either be replaced with a type-aware query (e.g., `deploy_account_tx_in_pool_or_recent_block`), or the gateway should query the mempool for the specific transaction type before deciding to skip `__validate__`.

Alternatively, align with the `PyValidator::should_run_stateful_validations` pattern in `native_blockifier`, which requires an explicit `deploy_account_tx_hash` parameter to be provided by the caller before skipping validation: [8](#0-7) 

### Proof of Concept

```
1. Attacker controls account A (deployed, on-chain nonce = 0).

2. Submit Invoke_0:
     sender_address = A
     nonce          = 0
     signature      = valid_sig   ← passes __validate__ at gateway

3. Invoke_0 enters the mempool.
   account_tx_in_pool_or_recent_block(A) → true

4. Submit Invoke_1:
     sender_address = A
     nonce          = 1
     signature      = 0xdeadbeef  ← invalid

5. Gateway evaluates skip_stateful_validations:
     tx.nonce()    == Nonce(1)  ✓
     account_nonce == Nonce(0)  ✓  (Invoke_0 not yet committed)
     account_tx_in_pool_or_recent_block(A) == true  ✓  (Invoke_0 is in pool)
   → skip_validate = true
   → execution_flags.validate = false
   → __validate__ is NOT called for Invoke_1

6. Invoke_1 (invalid signature) is admitted to the mempool.

7. During block production the batcher executes Invoke_1 with
   execution_flags.validate = true → __validate__ fails → transaction reverts.
   The reverted transaction is included in the block, consuming block space.
``` [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
