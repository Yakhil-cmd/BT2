### Title
Pool Admin Can Silently Overwrite a Pending Oracle Proposal, Resetting the Timelock Clock and Bypassing the Delay Guarantee — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` contains no guard against overwriting an already-pending proposal. A pool admin can call it a second time while a proposal is live, atomically replacing `pendingPriceProvider[pool]` and resetting `pendingPriceProviderExecuteAfter[pool]` to `block.timestamp + timelock`. This defeats the only protection the timelock provides: a guaranteed advance-notice window before an oracle change takes effect.

---

### Finding Description

`proposePoolPriceProvider` unconditionally writes to both pending-state slots:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  lines 487-490
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
``` [1](#0-0) 

There is no `if (pendingPriceProvider[pool] != address(0)) revert(...)` check. Any call by the pool admin while a proposal is already pending silently replaces it and restarts the clock.

The timelock mechanism was introduced specifically to fix M01 ("Admin can arbitrarily change the pool priceProvider"), as documented in `AuditFindings.md`:

> Fixed. Added constructor config `priceProviderTimelock`. Added admin flow: `proposePriceProvider` / `executePriceProviderUpdate`. [2](#0-1) 

The fix's intent is that users have a guaranteed window of `priceProviderTimelock[pool]` seconds to observe and react to an oracle change. The overwrite path eliminates that guarantee.

---

### Impact Explanation

**Attack: last-second oracle redirect**

1. Admin proposes provider A (a legitimate-looking oracle). Users observe the `PoolPriceProviderChangeProposed` event and begin monitoring.
2. At `executeAfter_A − 1` (one second before A can be executed), admin calls `proposePoolPriceProvider(pool, maliciousOracle)`. This atomically sets `pendingPriceProvider = maliciousOracle` and `pendingPriceProviderExecuteAfter = now + T`.
3. Users who were watching for A now have only 1 second of on-chain signal about the malicious oracle before the new clock starts. The full timelock `T` then runs against the malicious oracle, but the original warning window for A is gone.
4. After `T` seconds, admin executes the malicious oracle. Swaps now price against attacker-controlled bid/ask, enabling bad-price execution and direct loss of user principal.

**Attack: indefinite delay**

Admin repeatedly re-proposes any provider just before each `executeAfter`, preventing any oracle rotation from ever finalising. This is a functional DoS on the oracle-rotation flow, breaking core pool management.

Both paths are admin-boundary breaks: the pool admin exceeds the constraint imposed by the timelock, which is the only mechanism limiting their oracle-rotation power.

---

### Likelihood Explanation

The pool admin is the sole caller of `proposePoolPriceProvider` and `executePoolPriceProviderUpdate`. No external party can prevent the overwrite. The admin needs only to call the function twice — no special conditions, no race, no external dependency. Likelihood is high whenever a pool admin is malicious or compromised.

---

### Recommendation

Add a guard at the top of `proposePoolPriceProvider` that reverts if a proposal is already pending:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
```

Alternatively, add a separate `cancelPoolPriceProviderProposal` function (callable only by the pool admin) so that cancellation is an explicit, observable on-chain action, giving users a clear signal before a new proposal can be submitted.

---

### Proof of Concept

```
T=0:   admin calls proposePoolPriceProvider(pool, providerA)
         → pendingPriceProvider[pool] = providerA
         → pendingPriceProviderExecuteAfter[pool] = T + TIMELOCK

T=TIMELOCK-1:
       admin calls proposePoolPriceProvider(pool, maliciousOracle)
         → pendingPriceProvider[pool] = maliciousOracle   // providerA silently gone
         → pendingPriceProviderExecuteAfter[pool] = (TIMELOCK-1) + TIMELOCK

T=2*TIMELOCK-1:
       admin calls executePoolPriceProviderUpdate(pool)
         → block.timestamp >= execAfter  ✓
         → pool oracle set to maliciousOracle

Assert: pendingPriceProvider == maliciousOracle (not providerA)
Assert: executeAfter was reset to future (original timelock bypassed)
Assert: active price provider is now maliciousOracle
```

The original timelock window for `providerA` is bypassed entirely; users had at most 1 second of on-chain visibility of the switch to `maliciousOracle`.

## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
