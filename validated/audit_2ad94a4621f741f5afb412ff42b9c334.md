### Title
Gateway L2 Gas Price Admission Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Mempool/Execution Mismatch - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's stateful admission check validates a transaction's `max_price_per_unit` against the **previous block's `l2_gas_price`**, but the batcher executes that same transaction against the **EIP-1559-adjusted `next_l2_gas_price`** for the new block. When the two prices diverge (which happens every block that is not exactly at the gas target), the gateway admits transactions that the blockifier will hard-reject during block building, and rejects transactions that would have been valid.

### Impact Explanation

**Rising price (block above gas target):** `next_l2_gas_price > l2_gas_price`. A transaction with `l2_gas_price ≤ max_price_per_unit < next_l2_gas_price` passes both the gateway's custom threshold check and the gateway's blockifier `check_fee_bounds` (both use `l2_gas_price`). When the batcher later calls `perform_pre_validation_stage` with the actual block context (using `next_l2_gas_price`), `check_fee_bounds` returns `InsufficientResourceBounds { MaxGasPriceTooLow }` — a hard pre-validation error. The transaction is dropped from the block, the nonce is not advanced, and the transaction re-queues in the mempool indefinitely. An attacker can flood the mempool with such transactions to waste batcher CPU and delay honest transactions.

**Falling price (block below gas target):** `next_l2_gas_price < l2_gas_price`. A transaction with `next_l2_gas_price ≤ max_price_per_unit < l2_gas_price` is rejected by the gateway even though it would succeed in the next block. Valid user transactions are silently dropped.

### Likelihood Explanation

The EIP-1559 formula adjusts the price every block whenever gas consumption differs from the target. In practice, blocks are rarely exactly at the gas target, so `next_l2_gas_price ≠ l2_gas_price` on virtually every block. The divergence is bounded per block by `gas_price_max_change_denominator`, but it compounds across consecutive busy or idle blocks. No privilege is required; any user can craft a transaction with `max_price_per_unit` in the gap between the two prices.

### Recommendation

Replace the `l2_gas_price` read in `validate_resource_bounds` with `next_l2_gas_price` from the block header (the field already stored in `StorageBlockHeader.next_l2_gas_price`). Apply the same fix to `run_validate_entry_point`: when constructing the `BlockContext` for gateway blockifier validation, set `block_info.gas_prices.strk_gas_prices.l2_gas_price` to `header.next_l2_gas_price` rather than reusing the previous block's `l2_gas_price`. This aligns gateway admission with the price the batcher will actually enforce.

### Proof of Concept

1. Observe block N with `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (block N was above gas target).
2. Submit an `InvokeV3` transaction with `l2_gas.max_price_per_unit = P` (exactly equal to `l2_gas_price`).
3. Gateway `validate_tx_l2_gas_price_within_threshold` computes `threshold = 100% * P = P`; `P >= P` → **passes**.
4. Gateway blockifier `check_fee_bounds` uses block context with `l2_gas_price = P`; `P >= P` → **passes**.
5. Transaction is admitted to the mempool.
6. Batcher builds block N+1 with `l2_gas_price = P'`; blockifier `check_fee_bounds` checks `P < P'` → `MaxGasPriceTooLow` → **hard rejection**.
7. Transaction is dropped from the block, nonce unchanged, transaction re-queues. Repeat from step 2 with a fresh nonce to continuously occupy mempool slots and waste batcher execution cycles.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
```
