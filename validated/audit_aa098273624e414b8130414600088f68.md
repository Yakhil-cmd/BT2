### Title
Gateway Stateful Validator Skips L1 Gas and L1 Data Gas Price Validation, Admitting Transactions That Will Fail Execution — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_tx_l2_gas_price_within_threshold` only validates the L2 gas `max_price_per_unit` for `AllResources` transactions and explicitly skips all price validation for `L1Gas` legacy transactions. Transactions with `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` below the current block's actual gas prices pass gateway admission and enter the mempool, where they will deterministically fail at blockifier pre-validation (`check_fee_bounds`). This is a direct analog to the external report's "no upper-bound protection" pattern: here the missing guard is a *lower*-bound check on L1/L1-data gas prices at the gateway admission layer, causing the gateway to accept transactions it should reject.

---

### Finding Description

In `validate_tx_l2_gas_price_within_threshold`:

``` [1](#0-0) 

Two distinct gaps exist:

**Gap 1 — `AllResources` transactions**: Only `l2_gas.max_price_per_unit` is compared against the previous block's L2 gas price threshold. `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` are never checked. A transaction with `l1_gas.max_price_per_unit = 0` and a valid L2 gas price passes this check unconditionally.

**Gap 2 — `L1Gas` legacy transactions**: The arm explicitly returns `Ok(())` with the comment "No validation required for legacy transactions." No price check of any kind is performed, so a legacy transaction with `l1_gas.max_price_per_unit` far below the current block price is admitted without question.

The TODO comment at line 358 acknowledges the gap: `// TODO(Arni): Consider running this validation for all gas prices.` [2](#0-1) 

The stateless validator also only checks `l2_gas.max_price_per_unit` against a static floor; it never inspects `l1_gas` or `l1_data_gas` prices:

```rust
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
``` [3](#0-2) 

The blockifier's `check_fee_bounds` *does* validate all three resource prices against the actual block price:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [4](#0-3) 

This means the blockifier will reject the transaction at execution time — but only after the gateway has already admitted it to the mempool.

---

### Impact Explanation

Any unprivileged user can craft a transaction with `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` set below the current block's actual L1/L1-data gas price. The gateway stateful validator accepts it (only L2 gas price is checked), the transaction enters the mempool, and the sequencer wastes execution resources attempting it before the blockifier rejects it with `InsufficientResourceBounds { errors: [MaxGasPriceTooLow { resource: L1Gas, ... }] }`. The gateway is admitting transactions that are provably invalid at the time of submission.

**Matching impact**: *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

---

### Likelihood Explanation

L1 gas prices are volatile. During any L1 gas price spike, a large fraction of pending transactions signed with the pre-spike `l1_gas.max_price_per_unit` will have prices below the new block price. All of them pass gateway admission (only L2 gas price is gated) and flood the mempool with transactions that will fail execution. This is a normal, unprivileged, non-adversarial scenario that occurs organically during network congestion — exactly the scenario described in the external report.

---

### Recommendation

Extend `validate_tx_l2_gas_price_within_threshold` to also fetch and compare `l1_gas_price` and `l1_data_gas_price` from the previous block info, and apply the same `min_gas_price_percentage` threshold check to `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` for `AllResources` transactions. For `L1Gas` legacy transactions, apply the same threshold check to `l1_gas.max_price_per_unit`. The `validate_resource_bounds` caller already fetches `get_block_info()`, so the full `GasPriceVector` is available at that call site. [5](#0-4) 

---

### Proof of Concept

1. Read the current block's L1 gas price from the previous block header (e.g., `l1_gas_price = 100 STRK/gas`).
2. Submit an `AllResources` invoke transaction with:
   - `l2_gas.max_price_per_unit = 10_000` (above the L2 threshold — passes the only gateway check)
   - `l1_gas.max_price_per_unit = 1` (far below the actual L1 gas price of 100)
   - `l1_data_gas.max_price_per_unit = 1`
3. The stateless validator passes: L2 gas price ≥ `min_gas_price` config, and `max_possible_fee > 0`. [6](#0-5) 
4. The stateful validator passes: `validate_tx_l2_gas_price_within_threshold` only checks `l2_gas.max_price_per_unit`; L1 and L1-data gas prices are never compared. [7](#0-6) 
5. The transaction is admitted to the mempool.
6. When the sequencer calls `perform_pre_validation_stage` → `check_fee_bounds`, the blockifier compares `l1_gas.max_price_per_unit (1) < actual_l1_gas_price (100)` and returns `TransactionFeeError::InsufficientResourceBounds { errors: [MaxGasPriceTooLow { resource: L1Gas }] }`. [8](#0-7) 
7. The transaction fails execution. The gateway admitted an invalid transaction.
