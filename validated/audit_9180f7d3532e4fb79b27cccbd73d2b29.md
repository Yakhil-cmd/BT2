### Title
Missing Pointer-Alignment Check in `BitMap::from_raw` Enables Undefined Behavior During EVM Bytecode Execution - (`File: evm_interpreter/src/lib.rs`)

---

### Summary

`BitMap::from_raw` casts a `*const u8` byte pointer to `*const usize` and creates a `&[usize]` slice via `core::slice::from_raw_parts` without verifying that the pointer is aligned for `usize`. The length is checked to be a multiple of `BYTECODE_ALIGNMENT` (8 bytes), but pointer alignment is never verified. This is undefined behavior in Rust and can cause incorrect JUMPDEST validation or a prover crash when executing any EVM contract whose decommitted bytecode buffer is not `usize`-aligned at the bitmap offset.

---

### Finding Description

`BitMap::from_raw` is the only path used to load the pre-computed JUMPDEST bitmap from a decommitted bytecode slice:

```rust
pub fn from_raw(slice: &'a [u8]) -> Self {
    assert_eq!(slice.len() % BYTECODE_ALIGNMENT, 0);   // ← length check only
    let words = slice.len() / core::mem::size_of::<usize>();
    let ptr = slice.as_ptr() as *const usize;           // ← no alignment check
    let ws = unsafe { core::slice::from_raw_parts(ptr, words) };
    Self(ws)
}
``` [1](#0-0) 

The function's own safety comment lists only two invariants: length divisibility and lifetime. Pointer alignment is absent:

```
/// # Safety
/// * `slice` length is checked to be a multiple of `u64`.
/// * Caller guarantees the buffer lives at least `'a`.
``` [2](#0-1) 

`from_raw` is called from `parse_bytecode`, which computes `bitmap_slice` as a sub-slice of the decommitted bytecode:

```rust
let (code, rest) = bytecode.split_at(deployed_len);
let bitmap_slice = &rest[padding..];
// bitmap_slice starts at offset (deployed_len + padding) from bytecode base
let preprocessing = Self {
    jumpdest_bitmap: Either::Left(BitMap::from_raw(bitmap_slice)),
    ...
};
``` [3](#0-2) 

`parse_bytecode` is invoked during EVM contract execution for every contract stored with `ARTIFACTS_CACHING_CODE_VERSION_BYTE`:

```rust
ARTIFACTS_CACHING_CODE_VERSION_BYTE => {
    let (code, bytecode_preprocessing) = BytecodePreprocessingData::parse_bytecode(
        bytecode,
        unpadded_code_len as usize,
        artifacts_len as usize,
    )?;
``` [4](#0-3) 

The resulting misaligned `&[usize]` slice is then read on every `JUMP`/`JUMPI` instruction via `get_bit_unchecked`:

```rust
pub unsafe fn get_bit_unchecked(&self, pos: usize) -> bool {
    let (w, b) = (pos / usize::BITS as usize, pos % usize::BITS as usize);
    self.0.get_unchecked(w) & (1usize << b) != 0
}
``` [5](#0-4) 

**Why alignment can be violated:** `deployed_len + padding_len` is guaranteed to be a multiple of 8 by `bytecode_padding_len`. However, the *base address* of the decommitted bytecode buffer (`callee_account_properties.bytecode`, a `&'static [u8]`) is not guaranteed to be `usize`-aligned. On 64-bit hosts (where the prover runs), `align_of::<usize>() = 8`. If the preimage cache allocates the bytecode buffer with only 4-byte alignment, the bitmap pointer will be misaligned by 4 bytes, making every `usize` read undefined behavior. On 32-bit RISC-V (the zkVM target), `align_of::<usize>() = 4`; a 1-byte-aligned allocation would trigger the same issue.

The `debug_assert` pattern seen elsewhere in the codebase (e.g., `debug_assert_eq!(ptr.addr() % 32, 0)` in `u512.rs`) confirms the project is aware of alignment requirements — but `from_raw` has no such check, not even a `debug_assert`. [6](#0-5) 

---

### Impact Explanation

Dereferencing a misaligned `*const usize` is undefined behavior in Rust. Concretely:

- **On the prover (x86-64):** x86 tolerates misaligned loads in hardware, but Rust's optimizer is permitted to assume alignment and may emit instructions (e.g., SSE/AVX aligned loads) that fault or silently produce wrong values. A wrong bitmap read means a non-JUMPDEST byte is treated as a valid jump target, allowing EVM execution to jump into the middle of an instruction — producing a state transition that the prover cannot reproduce correctly, causing a **forward/proving divergence**.
- **On the zkVM (RISC-V 32):** RISC-V requires natural alignment; a misaligned `lw` traps, causing the zkVM to abort — a **denial-of-service** against any transaction that calls the affected contract.

---

### Likelihood Explanation

Every EVM contract deployed after the `ARTIFACTS_CACHING_CODE_VERSION_BYTE` upgrade stores its bitmap inline in the bytecode blob. Any such contract is affected if its bytecode buffer happens to be misaligned. The likelihood depends on the allocator's alignment guarantee for the preimage cache. If the allocator provides only 4-byte alignment (common in embedded/no-std environments targeting 32-bit), every contract on a 64-bit prover is affected. The attacker does not need to control the alignment — they only need to call any deployed EVM contract.

---

### Recommendation

Add an explicit pointer-alignment assertion inside `from_raw` before the unsafe cast:

```rust
pub fn from_raw(slice: &'a [u8]) -> Self {
    assert_eq!(slice.len() % BYTECODE_ALIGNMENT, 0);
    assert_eq!(
        slice.as_ptr().addr() % core::mem::align_of::<usize>(),
        0,
        "bitmap slice pointer is not usize-aligned"
    );
    let words = slice.len() / core::mem::size_of::<usize>();
    let ptr = slice.as_ptr() as *const usize;
    let ws = unsafe { core::slice::from_raw_parts(ptr, words) };
    Self(ws)
}
```

Additionally, ensure the preimage cache allocates bytecode buffers with at least `align_of::<usize>()` alignment (use `UsizeAlignedByteBox` or an equivalent aligned allocator). Update the `# Safety` doc-comment to list pointer alignment as a required invariant.

---

### Proof of Concept

1. Deploy an EVM contract. The system stores `deployed_code ++ padding ++ bitmap` in the preimage cache as a `&'static [u8]` buffer.
2. If the preimage cache allocator returns a buffer whose base address is `0x...4` (4-byte aligned but not 8-byte aligned), and `deployed_len + padding_len = 8k`, then `bitmap_slice.as_ptr() = base + 8k` has the same misalignment as the base.
3. `BitMap::from_raw` casts this to `*const usize` without checking alignment.
4. On the next `JUMP` opcode, `get_bit_unchecked` reads a misaligned `usize` — undefined behavior.
5. On x86-64 the optimizer may emit an aligned SSE load that reads wrong data, causing the JUMPDEST check to return an incorrect result; the prover and the forward executor diverge on the validity of the jump.
6. On RISC-V 32 the misaligned `lw` traps, aborting the zkVM for any caller of that contract.
