### Title
Forked `reddsa` Library with Disabled Cheater Detection Allows Malicious Participant to Cause Permanent Denial of FROST Signing — (File: src/frost/eddsa/sign.rs, src/frost/redjubjub/sign.rs)

### Summary
The project uses a git-pinned fork of `reddsa` that was created specifically to work around the cheater-detection feature present in the upstream Zcash library. As a direct consequence, both the EdDSA and RedJubjub FROST signing coordinators call `frost-core`'s `aggregate()` with an empty `verifying_shares` map, completely bypassing per-share signature verification. A single malicious participant can submit an invalid signature share every round, causing the coordinator to permanently fail to produce a valid threshold signature, with no mechanism to identify or exclude the offending party.

### Finding Description

`Cargo.toml` pins `reddsa` to a private git fork at a fixed commit:

```toml
# This project has been forked due to incompatibility problems with cheater detection feature activated on the original Zcash repo
reddsa = { git = "https://github.com/near/reddsa", rev = "c7cd92a55f7399d8d7f8c0ac386445b5f898f197", ... }
``` [1](#0-0) 

`deny.toml` additionally suppresses the only active security advisory that touches this dependency chain:

```toml
ignore = [
    "RUSTSEC-2023-0089" # deprecated atomic-polyfill necessary by the serialization feature of rerandomized-frost 2.2.0 called in reddsa
]
``` [2](#0-1) 

In both FROST signing coordinators the `PublicKeyPackage` is constructed with an empty `BTreeMap`, which is the mechanism `frost-core` uses to decide whether to perform per-share verification:

**EdDSA (v1 and v2):**
```rust
// We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
// Feature "cheater-detection" only points to a malicious participant, if there's such.
// It doesn't bring any additional guarantees.
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [3](#0-2) [4](#0-3) 

**RedJubjub:**
```rust
// We use empty BTreeMap because "cheater-detection" feature is disabled
let pk_package = PublicKeyPackage::new(BTreeMap::new(), keygen_output.public_key);
let signature = aggregate(&signing_package, &signature_shares, &pk_package, &randomized_params)
    .map_err(|_| ProtocolError::ErrorFrostAggregation)?;
``` [5](#0-4) 

When `verifying_shares` is empty, `frost-core`'s `aggregate()` skips the per-share check `s_i·G == R_i + c·λ_i·Y_i` for every participant `i`. It still verifies the final aggregated signature `(R, s)` against the group public key, so a bad share causes `aggregate()` to return an error — but the error carries no information about which participant submitted the invalid share.

The in-code comment that cheater detection "doesn't bring any additional guarantees" is incorrect: it is the only mechanism by which the coordinator can attribute a signing failure to a specific participant and exclude them from subsequent rounds.

### Impact Explanation

A single malicious participant enrolled in the signing set can, every round, submit a syntactically valid but arithmetically incorrect signature share. The coordinator's call to `aggregate()` will fail with a generic error. Because no per-share verification is performed, the coordinator receives no attribution data. It cannot determine which of the `n` participants is responsible.

Without the ability to identify and exclude the malicious participant, every retry of the signing protocol with the same participant set will fail in the same way. If the threshold `t` equals `n` (all participants required), or if the malicious participant is present in every feasible signing quorum, signing is permanently blocked for honest parties. Even when `t < n`, the coordinator must exhaustively try exponentially many subsets to find one that excludes the attacker — which is operationally equivalent to permanent denial.

**Impact class:** High — Permanent denial of signing for honest parties.

### Likelihood Explanation

Any participant who has been issued a valid key share and is included in a signing session can execute this attack. No cryptographic material beyond their own share is required. The attack requires only that the participant send a scalar value that does not satisfy the FROST share equation. This is trivially achievable by any participant who wishes to disrupt signing. The attack is repeatable across every signing round at negligible cost.

### Recommendation

1. **Re-enable cheater detection**: Populate `verifying_shares` in `PublicKeyPackage` with the actual per-participant verifying shares derived from DKG output. This allows `frost-core`'s `aggregate()` to identify and report the offending participant identifier on failure.

2. **Reconcile with upstream**: Investigate whether the "incompatibility problems" with the upstream Zcash `reddsa` cheater-detection feature can be resolved without forking, or whether the fork can be updated to re-enable the feature.

3. **Add application-level attribution**: If the library-level cheater detection cannot be re-enabled, implement an application-level per-share verification step before calling `aggregate()`, using the verifying shares from `KeygenOutput`.

4. **Track the pinned fork**: The git-pinned fork at `rev = c7cd92a...` will not receive upstream security fixes. Establish a process to monitor and merge upstream changes.

### Proof of Concept

1. Run a FROST EdDSA signing session with `n` participants and threshold `t`.
2. One participant (the attacker) replaces their honest `round2::sign()` output with a random scalar before sending it to the coordinator.
3. The coordinator collects all `n` shares and calls `aggregate()` with `PublicKeyPackage::new(BTreeMap::new(), vk_package)`.
4. `aggregate()` sums the shares, computes `(R, s)`, verifies against the group public key — verification fails because the attacker's share corrupted `s`.
5. `aggregate()` returns `Err(...)` with no participant identifier.
6. The coordinator has no basis to exclude any participant. Retrying with the same set repeats steps 2–5 indefinitely.
7. Signing never succeeds as long as the attacker participates.
