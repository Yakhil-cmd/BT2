### Title
Understated "Total Amount with Royalties" in Offer Confirmation Dialog Due to Incorrect BigInt Division in `formatAmountWithRoyalties` — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` uses an incorrect algorithm when computing the royalty-inclusive total for multi-NFT offers. It divides the trade amount by the number of NFTs before applying each royalty percentage, rather than applying each royalty to the full amount. This causes the "Total Amount with Royalties" shown in the offer-acceptance confirmation dialog to be systematically understated. A user who accepts a multi-NFT offer with royalties will approve a transaction that deducts significantly more XCH or CAT than the confirmation dialog indicated.

---

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` is called for every fungible spending line when the opposite side of the offer contains NFTs with royalties:

```ts
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  ...
  const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
``` [1](#0-0) 

The correct formula is: for each NFT, apply its royalty percentage to the **full** trade amount and sum the results. The code instead splits the amount equally across all NFTs first, then applies each percentage to the split portion. This is mathematically wrong for any multi-NFT offer.

**Concrete example** (matches the existing test case at line 251):

| | Displayed (GUI) | Correct (on-chain) |
|---|---|---|
| Base amount | 0.0001 XCH | 0.0001 XCH |
| NFT1 royalty (5%) | 0.0000025 XCH | 0.000005 XCH |
| NFT2 royalty (0.1%) | 0.0000005 XCH | 0.0000001 XCH |
| **Total shown** | **0.00010255 XCH** | **0.0001051 XCH** | [2](#0-1) 

The discrepancy scales with the number of NFTs and their royalty percentages. With 10 NFTs each carrying a 10% royalty on a 1 XCH trade, the GUI would display 1.1 XCH while the actual deduction is 2 XCH — a 90% understatement.

The `amountWithRoyalties` field is rendered directly in the confirmation dialog shown to the user before they approve:

```tsx
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties} ...
  </div>
)}
``` [3](#0-2) 

This dialog is shown for both the WalletConnect `chia_wallet.take_offer` command path and the regular GUI offer-acceptance path (via the WebSocket bridge interceptor in `main.tsx`): [4](#0-3) 

---

### Impact Explanation

A user accepting a multi-NFT offer with royalties is shown a "Total Amount with Royalties" that is lower than the actual on-chain cost. They approve the transaction believing they are spending amount X, but the blockchain deducts a materially larger amount. This is a spoofed accounting display in the offer confirmation flow that causes the user to approve the wrong amount — matching the **High** impact category: *"Corruption, spoofing, or unsafe trust of… offer… state that causes a user to approve… the wrong… amount."*

---

### Likelihood Explanation

Any unprivileged actor can craft a valid Chia offer containing two or more NFTs with royalties and share it (as an offer file, via WalletConnect, or via a dApp). No special access, leaked keys, or social engineering beyond presenting the offer is required. The victim only needs to open the offer in the GUI and click Accept. The discrepancy is invisible to the user because the confirmation dialog is the sole source of truth for the expected cost.

---

### Recommendation

Replace the `splitAmount` approach with the correct per-NFT calculation: apply each royalty percentage to the full `amount` and sum the results.

```ts
// Correct implementation
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [5](#0-4) 

---

### Proof of Concept

1. Seller creates an offer: sell 2 NFTs (NFT-A with 5% royalty, NFT-B with 0.1% royalty) in exchange for 0.0001 XCH.
2. Buyer opens the offer in the Chia GUI (or a WalletConnect dApp calls `chia_wallet.take_offer`).
3. The confirmation dialog displays:
   - Amount: 0.0001 XCH
   - **Total Amount with Royalties: 0.00010255 XCH** ← understated
4. Buyer clicks Accept.
5. On-chain, the actual deduction is **0.0001051 XCH** — 2.4% more than shown.

With 10 NFTs each at 10% royalty on a 1 XCH trade, the dialog shows 1.1 XCH while the actual deduction is 2 XCH.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],

  if (royaltyPercentages.length === 0 || line.kind === 'nft') {

  }

  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
