### Title
Unrestricted `declare_operational_address` Allows Anyone to Emit Spoofed `OperationalAddressDeclared` Events - (File: src/staking/staking.cairo)

### Summary
`declare_operational_address` in `src/staking/staking.cairo` accepts a fully attacker-controlled `staker_address` parameter with no validation that the supplied address is a real staker or that the caller has any relationship to it. Any unprivileged address can call the function with an arbitrary `staker_address`, causing the contract to write a false entry into `eligible_operational_addresses` storage and emit a spoofed `OperationalAddressDeclared` event. This is a direct structural analog to the `notifyFeeReceived` access-control gap: a public, permissionless function whose side-effect is an on-chain event whose authenticity the protocol cannot guarantee.

### Finding Description
The function at `src/staking/staking.cairo` lines 705–723 is:

```cairo
fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
    self.general_prerequisites();                          // only checks pause
    let operational_address = get_caller_address();
    assert!(
        self.operational_address_to_staker_address.read(operational_address).is_zero(),
        "{}", Error::OPERATIONAL_IN_USE,
    );
    assert!(
        !self.does_token_exist(token_address: operational_address),
        "{}", Error::OPERATIONAL_IS_TOKEN,
    );
    if self.eligible_operational_addresses.read(operational_address) == staker_address {
        return;
    }
    self.eligible_operational_addresses.write(operational_address, staker_address);
    self.emit(Events::OperationalAddressDeclared { operational_address, staker_address });
}
```

The three guards are:
1. Contract is not paused (`general_prerequisites`).
2. The caller's address is not already registered as an active operational address for any staker.
3. The caller's address is not a token address.

None of these guards validate `staker_address`. It is never checked against the `staker_info` map, never required to be non-zero, and never required to have any relationship to the caller. An attacker can therefore:

- Pass any arbitrary `ContractAddress` as `staker_address`, including addresses of real stakers, zero, or random values.
- Overwrite the `eligible_operational_addresses[caller]` slot with a fake staker address, emitting `OperationalAddressDeclared { operational_address: caller, staker_address: victim }`.
- Repeat from many different caller addresses to flood the event log with spurious declarations referencing any staker.

The `OperationalAddressDeclared` event is defined in `src/staking/interface.cairo` (lines 385–391) with both fields as indexed keys, making it a primary signal for off-chain indexers and monitoring infrastructure to track which addresses have declared intent to become operational addresses for which stakers.

### Impact Explanation
**Medium — Griefing with no profit motive but concrete damage to users and the protocol.**

1. **Event-log pollution / authenticity degradation**: Off-chain indexers, UIs, and monitoring tools that consume `OperationalAddressDeclared` events to display or act on pending operational-address declarations cannot distinguish legitimate declarations from attacker-injected ones. A staker's dashboard may show dozens of spurious addresses "declaring" for them, masking the real pending declaration.

2. **State pollution in `eligible_operational_addresses`**: Although the mapping key is `get_caller_address()` (so an attacker cannot overwrite a *different* address's slot), an attacker who controls many addresses can fill the mapping with garbage entries pointing to real stakers. This does not directly block `change_operational_address` for the victim staker, but it creates a permanently polluted on-chain state that cannot be cleaned up.

3. **Targeted griefing of a pending operational-address rotation**: If a legitimate operational address `op_addr` has declared itself for `staker_A` and is waiting for the staker to call `change_operational_address`, the owner of `op_addr` (or a social-engineering attack convincing `op_addr` to re-call the function) can overwrite the slot to point to a different staker, silently invalidating the pending rotation without any on-chain error.

### Likelihood Explanation
**High.** The function is public, requires no stake, no token balance, and no privileged role. The only cost is gas. A single transaction from any EOA or contract is sufficient to emit a spoofed event for any target staker address. There is no rate-limiting or cooldown.

### Recommendation
Add a validation that `staker_address` is a registered staker before writing state or emitting the event:

```cairo
fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
    self.general_prerequisites();
    // Require staker_address to be a real, existing staker.
    assert!(
        self.staker_info_map.entry(staker_address).is_some(),
        "{}", Error::STAKER_NOT_EXISTS,
    );
    let operational_address = get_caller_address();
    // ... existing checks ...
}
```

This mirrors the pattern used in `change_operational_address` (line 681), which calls `self.internal_staker_info(:staker_address)` and panics if the staker does not exist, ensuring the event is only emitted for real protocol participants.

### Proof of Concept

1. Deploy the staking contract and stake as `staker_A`.
2. From a fresh EOA `attacker` (no stake, no role), call:
   ```
   staking.declare_operational_address(staker_address: staker_A)
   ```
3. Observe that the transaction succeeds, `eligible_operational_addresses[attacker]` is written to `staker_A`, and `OperationalAddressDeclared { operational_address: attacker, staker_address: staker_A }` is emitted — despite `attacker` having no relationship to `staker_A`.
4. Repeat from 1 000 different attacker addresses in a single script. The event log now contains 1 000 spurious `OperationalAddressDeclared` entries all pointing at `staker_A`, indistinguishable from a real declaration by an off-chain consumer.
5. Confirm the existing test at `src/staking/tests/test.cairo` lines 1847–1858 already demonstrates that calling with `staker_address: Zero::zero()` succeeds and emits the event — a zero address is never a valid staker, yet the event is emitted unconditionally. [1](#0-0) [2](#0-1) [3](#0-2)
