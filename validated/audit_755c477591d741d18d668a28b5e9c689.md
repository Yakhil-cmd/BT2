### Title
GCS Proof Archive Write Failure Causes Valid Client-Side Proving Transactions to Be Rejected at Gateway Admission - (`crates/apollo_gateway/src/gateway.rs`)

### Summary

The `await_proof_archiving` function in the Gateway propagates GCS (Google Cloud Storage) write errors as hard failures, causing the entire `add_tx` flow to return an error and reject the transaction. Because the proof has already been verified and stored in the proof manager before the GCS write is awaited, the GCS archive is a supplementary/redundancy store — yet its failure is treated as a blocking condition. Any transient GCS outage, rate-limit, or timeout will cause all directly-submitted client-side proving transactions (Invoke V3 with non-empty `proof_facts`) to be rejected at the gateway admission layer, even though they are fully valid.

### Finding Description

In `add_tx_inner`, after stateless validation, stateful validation, and proof storage in the proof manager, the gateway spawns a GCS archive write and then awaits it:

`await_proof_archiving` converts every failure mode — write error, task panic, and timeout — into a hard `Err(StarknetError::internal_with_logging(...))`:

The `?` operator at line 284 propagates this error, causing `add_tx_inner` to return an error and the transaction to be rejected. This is confirmed by the existing test: [3](#0-2) 

By the time `await_proof_archiving` is called, the proof has already been:
1. Cryptographically verified (during `convert_rpc_tx_to_internal_rpc_tx`)
2. Stored in the proof manager (during `store_proof_and_spawn_archiving`, line 336) [4](#0-3) 

The GCS archive is a secondary/redundancy store. Its own comment describes it as running "in parallel" with the proof-manager store to avoid serializing latency — not as a gating condition for admission. [5](#0-4) 

The hard timeout is only 5 seconds: [6](#0-5) 

### Impact Explanation

**High. Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

Any client-side proving transaction (Invoke V3 with non-empty `proof_facts`) submitted directly to the gateway (not via P2P) will be rejected whenever GCS is unavailable, slow, or rate-limiting — even though the transaction is fully valid, the proof is cryptographically verified, and the proof is already durably stored in the proof manager. During a GCS outage or sustained latency spike exceeding 5 seconds, 100% of such transactions are rejected at the gateway. Users must retry, but retries will also fail for the duration of the outage.

### Likelihood Explanation

GCS is an external service subject to transient network errors, regional outages, and rate limits. The 5-second `PROOF_ARCHIVE_WRITE_TIMEOUT` means even brief network hiccups can trigger the failure path. The existing alert definition (`gateway_proof_archive_write_failure`) confirms the team anticipates this failure mode, but the current code treats it as a hard rejection rather than a degraded-mode continuation. [7](#0-6) 

### Recommendation

Degrade gracefully on GCS archive failure instead of rejecting the transaction. Log the error and increment the metric, but allow the transaction to proceed to the mempool since the proof is already in the proof manager:

```rust
async fn await_proof_archiving(proof_archive_handle: ProofArchiveHandle) -> GatewayResult<()> {
    let Some((handle, tx_hash)) = proof_archive_handle else {
        return Ok(());
    };
    let abort_handle = handle.abort_handle();
    match timeout(PROOF_ARCHIVE_WRITE_TIMEOUT, handle).await {
        Ok(Ok((_, Ok(())))) => Ok(()),
        // All failure arms: log + metric, but do NOT return Err
        Ok(Ok((proof_facts_hash, Err(e)))) => {
            GATEWAY_PROOF_ARCHIVE_WRITE_FAILURE.increment(1);
            warn!("Failed to archive proof to GCS (non-blocking): ...");
            Ok(())
        }
        Ok(Err(e)) => {
            GATEWAY_PROOF_ARCHIVE_WRITE_FAILURE.increment(1);
            warn!("Proof archive writer task panicked (non-blocking): ...");
            Ok(())
        }
        Err(_elapsed) => {
            abort_handle.abort();
            GATEWAY_PROOF_ARCHIVE_WRITE_FAILURE.increment(1);
            warn!("Proof archive writer timed out (non-blocking): ...");
            Ok(())
        }
    }
}
```

This mirrors the fix applied to the SponsorVault bug: wrap the supplementary external call so its failure does not block the core flow.

### Proof of Concept

1. Submit an Invoke V3 transaction with non-empty `proof_facts` and a valid proof directly to the gateway (not via P2P).
2. Configure the `ProofArchiveWriter` to return `Err(ProofArchiveError::WriteError(...))` (or let the 5-second timeout elapse by making GCS unreachable).
3. Observe that `add_tx` returns an internal error and the transaction is rejected, even though the proof was successfully verified and stored in the proof manager.

The existing test `test_add_tx_fails_when_proof_archive_write_fails` in `crates/apollo_gateway/src/gateway_test.rs` already demonstrates this exact behavior. [3](#0-2)
