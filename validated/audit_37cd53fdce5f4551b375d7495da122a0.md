### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Burn Without Exit — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear()` and `withdrawToEthereum()` by first burning the caller's ERC-20 tokens and then invoking the Aurora exit precompile via inline assembly. The return value of the assembly `call()` is captured in `res` but **never checked**. If the precompile call fails for any reason, execution continues silently: the tokens are permanently destroyed on the Aurora EVM side while the corresponding NEP-141 tokens on NEAR are never released to the recipient.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear()` is implemented as:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — no revert if call fails
    }
}
```

The identical pattern appears in `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol`. The `_burn()` is unconditional and executes before the precompile call. If the precompile call returns `0` (failure), the assembly block exits normally, the Solidity function returns normally, and the user's tokens are gone with no recourse.

The analog to the original report is direct: just as `token.transfer()` without a return-value check silently fails for non-standard tokens, `call(...)` in assembly without checking `res` silently fails when the exit precompile rejects the call — in both cases the caller loses funds with no error surfaced.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 balance is reduced to zero (burned via `_burn()`).
- The NEP-141 token balance on NEAR is never credited to the recipient.
- There is no mechanism to recover the burned tokens; `_burn()` is irreversible.

The funds are permanently destroyed from the user's perspective.

---

### Likelihood Explanation

**Medium.** Conditions under which the exit precompile call can return `0`:

1. **Out-of-gas inside the precompile**: Although `gas()` (all remaining gas) is forwarded, the precompile itself may exhaust it, causing the sub-call to fail. The outer call does not revert.
2. **Invalid or oversized `recipient` bytes**: A recipient byte string that is syntactically valid in Solidity but rejected by the NEAR account-ID validation inside the precompile will cause the precompile to return an error. The assembly block ignores this.
3. **Precompile address mismatch / engine upgrade**: If the hardcoded precompile address `0xe921...` is ever unreachable (e.g., during an engine migration or on a non-standard silo deployment), the call returns `0` and tokens are silently burned.

Any unprivileged EVM user calling `withdrawToNear` or `withdrawToEthereum` with a recipient that triggers a precompile-level rejection hits this path.

---

### Recommendation

Reverse the operation order and/or check the return value:

1. **Check `res` and revert on failure** (minimal fix):
```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Preferred — move `_burn()` after the precompile call succeeds**: Perform the burn only after confirming the exit precompile accepted the request, following a checks-effects-interactions pattern.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora with a valid NEP-141 backing token.
2. Mint tokens to `user`.
3. `user` calls `withdrawToNear(recipient_bytes, amount)` where `recipient_bytes` is a byte string that passes Solidity's `bytes` type but is rejected by the NEAR account-ID validator inside the precompile (e.g., an empty byte string or one containing invalid characters).
4. The precompile call returns `0`; the assembly block does not revert.
5. `withdrawToNear()` returns successfully.
6. `user`'s ERC-20 balance is now `0` (burned), but no NEP-141 tokens were transferred to any NEAR account.
7. Funds are permanently lost.
