### Title
Negative `delta_gas` Never Applied to `gas_used` — Users Systematically Overpay When EVM Gas Exceeds Native Resource Cost - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` in `refund_calculation.rs` implements a "double resource accounting" bridge that adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`). However, the symmetric downward adjustment — reducing `gas_used` when EVM gas consumption exceeds native resource consumption (`delta_gas < 0`) — is silently skipped. A developer-authored `TODO` comment at line 80 explicitly flags this as an unresolved open question. The result is that users are systematically overcharged whenever their EVM gas consumption exceeds the native resource equivalent, and the operator receives excess fees that were never earned.

---

### Finding Description

In `compute_gas_refund`, after computing `native_used` and `gas_used`, the code calculates `delta_gas`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // native resource consumption > EVM gas consumption → charge more gas
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The design specification in `docs/double_resource_accounting.md` defines `deltaGas := (nativeUsed / nativePerGas) - gasUsed` and states: *"If `deltaGas > 0`, we add it to `gasUsed`"*. The document is silent on the negative case. [2](#0-1) 

When `delta_gas < 0` — meaning `native_used / native_per_gas < gas_used` — the user consumed less native resource than their EVM gas implies. The correct behavior would be to reduce `gas_used` by `|delta_gas|`, giving the user a partial refund. Instead, `gas_used` is left at its full EVM-derived value, and the user is charged for the entire amount.

The final `gas_used` value flows directly into the operator payment calculation:

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
    ...
``` [3](#0-2) 

And into the user refund calculation:

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [4](#0-3) 

Because `gas_used` is inflated (not reduced by `|delta_gas|`), the user's refund is smaller than it should be, and the operator's payment is larger than it should be.

This same `compute_gas_refund` function is called for both L2 transactions (via `ZkTransactionFlowOnlyEOA::before_refund`) and L1→L2 transactions (via `process_l1_transaction`): [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Every transaction where EVM gas consumption exceeds the native resource equivalent results in the user paying more than the protocol's own resource model justifies. The overcharge per transaction is:

```
overcharge = (gas_used - native_used / native_per_gas) * gas_price
```

The operator receives this excess. Over a scaled system, this accumulates into a significant systematic transfer of value from users to the operator that is not backed by any actual resource consumption. This is a **resource accounting bug** causing direct, quantifiable token loss for transaction senders.

---

### Likelihood Explanation

`delta_gas < 0` occurs whenever `native_used / native_per_gas < gas_used`. This is a common condition for:

- Transactions with high EVM gas consumption but low proving complexity (e.g., simple storage reads/writes, arithmetic-heavy contracts that are cheap to prove in RISC-V)
- Any transaction where `native_per_gas` is low (i.e., `gas_price` is close to `base_fee` with no priority fee), making the native-to-gas conversion ratio small

The condition is reachable by any unprivileged transaction sender submitting a standard EVM transaction. No special privileges or oracle manipulation are required.

---

### Recommendation

Apply the negative `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` (subject to the `minimal_gas_used` floor):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native resource consumption < EVM gas consumption.
    // Reduce gas_used, but never below the minimal floor.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This resolves the `TODO` comment and makes the double-resource accounting symmetric in both directions.

---

### Proof of Concept

Consider a transaction with:
- `gas_limit = 100_000`
- `gas_price = 1_000` (= base fee, no priority fee)
- `native_price = 100`
- `native_per_gas = gas_price / native_price = 10`
- EVM execution consumes `gas_used = 50_000` gas (ergs exhausted)
- Native resource consumed: `native_used = 200_000` native units
- `native_used / native_per_gas = 200_000 / 10 = 20_000`

`delta_gas = 20_000 - 50_000 = -30_000` (negative)

**Current behavior:** `gas_used` stays at `50_000`. User is charged `50_000 * 1_000 = 50_000_000` tokens. Operator receives `50_000_000` tokens.

**Correct behavior:** `gas_used` should be reduced to `20_000`. User should be charged `20_000 * 1_000 = 20_000_000` tokens. Operator should receive `20_000_000` tokens.

**Overcharge:** `30_000_000` tokens per transaction, silently transferred to the operator with no corresponding resource justification. [7](#0-6)
