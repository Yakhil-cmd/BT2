### Title
First Deployed Pool Permanently Excluded from Canonical Registry Due to Zero-Index Collision — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`nextPoolIdx` is a `uint256` state variable that defaults to `0`. The first pool created receives index `0`, which is stored as `poolToIdx[pool] = 0`. Because `isPool()` checks `poolToIdx[pool] != 0`, the first deployed pool is permanently and incorrectly identified as non-canonical — indistinguishable from an address that was never registered.

---

### Finding Description

In `MetricOmmPoolFactory`, the registry uses two mappings to track canonical pools:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol
uint256 poolIdx = nextPoolIdx;   // nextPoolIdx == 0 on first call
nextPoolIdx++;
idxToPool[poolIdx] = pool;
poolToIdx[pool] = poolIdx;       // poolToIdx[firstPool] = 0
``` [1](#0-0) 

The canonical check is:

```solidity
function isPool(address pool) external view override returns (bool) {
    return poolToIdx[pool] != 0;
}
``` [2](#0-1) 

For the first pool ever deployed, `poolToIdx[firstPool] = 0`. The Solidity default value for an unregistered address in a `mapping(address => uint256)` is also `0`. The sentinel check `!= 0` cannot distinguish between "registered at index 0" and "never registered." `isPool(firstPool)` returns `false` permanently, even though the pool was legitimately deployed and fully registered in `idxToPool`.

The analog to the external report is exact: a check function (`isPool`) returns an incorrect boolean result because the underlying state (`poolToIdx`) was not initialized to distinguish the zero-index case from the absent-entry case — and downstream consumers act on that incorrect boolean without a secondary verification.

---

### Impact Explanation

Every consumer that calls `isPool()` to gate access to pool operations will treat the first canonical pool as unregistered:

- **Periphery routers and quoters** that guard `swap()` or `addLiquidity()` calls with an `isPool` check will reject all interactions with the first pool, making it permanently unusable through the standard periphery.
- **Fee collection paths** (`collectPoolFees`) that verify pool canonicality before disbursing accumulated protocol and admin fees will fail for the first pool, locking fees in the pool contract.
- **Off-chain indexers and integrators** that rely on `isPool` to enumerate or validate pools will silently omit the first pool from their view of the protocol.

The `idxToPool[0]` forward mapping correctly resolves to the first pool, creating an irreconcilable inconsistency: the pool exists in the forward registry but is invisible through the reverse lookup. This is a broken registry identity invariant with direct fund-loss potential (locked fees, inaccessible liquidity via periphery).

---

### Likelihood Explanation

This is deterministic and affects every factory deployment. The first `createPool` call on any factory instance always produces `nextPoolIdx = 0`, always stores `poolToIdx[pool] = 0`, and `isPool` always returns `false` for that pool. No special attacker action is required; the bug is triggered by the first legitimate pool creator.

---

### Recommendation

Initialize `nextPoolIdx` to `1` in the factory constructor so that index `0` is permanently reserved as the "not registered" sentinel:

```solidity
constructor(address owner) Ownable(owner) {
    nextPoolIdx = 1; // reserve 0 as the unregistered sentinel
}
```

This makes `poolToIdx[unregisteredAddress] == 0` and `poolToIdx[firstPool] == 1`, so `isPool` correctly returns `true` for all legitimately deployed pools.

### Proof of Concept

```solidity
// Foundry test
function test_isPool_returnsFalseForFirstPool() public {
    MetricOmmPoolFactory f = new MetricOmmPoolFactory(address(this));
    MetricOmmPoolDeployer d = new MetricOmmPoolDeployer(address(f));
    f.setPoolDeployer(address(d));

    address firstPool = f.createPool(_defaultParams());

    // BUG: isPool returns false for the first legitimately deployed pool
    assertFalse(f.isPool(firstPool));  // passes — demonstrates the bug

    // The forward mapping is correct, proving the pool IS registered
    assertEq(f.idxToPool(0), firstPool);

    // Second pool is correctly recognized
    address secondPool = f.createPool(_defaultParams2());
    assertTrue(f.isPool(secondPool));
}
