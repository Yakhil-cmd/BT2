### Title
Cross-Address COA Ownership Proof Replay via Missing `evmAddress` Binding in Signed Data - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over caller-supplied `signedData` but does not enforce that `signedData` encodes the `evmAddress` parameter. A signature produced by a Cadence account for one COA can be replayed to pass ownership verification for a different COA owned by the same Cadence account, because the signed message is not bound to the target EVM address.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six parameters: `address` (Cadence account), `path` (capability path), `signedData` (arbitrary bytes), `keyIndices`, `signatures`, and `evmAddress` (the COA being claimed). The function verifies that the signatures are valid over `signedData` using the Cadence account's keys, then separately checks that the COA at `path` has the address `evmAddress`. Critically, it never checks that `signedData` encodes `evmAddress`.

The code itself acknowledges this:

```cadence
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
/// In principle, a signature produced for one purpose could be replayed here against
/// a different COA owned by the same Cadence account.
```

The signature verification step:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // evmAddress is NOT required to be in here
    domainSeparationTag: "FLOW-V0.0-user"
)
```

passes with any `signedData` the attacker supplies, as long as the Cadence account's keys produced a valid signature over it. The `evmAddress` is only checked afterward against the COA resource at `path`, but the signature itself is never bound to `evmAddress`.

`validateCOAOwnershipProof` is `access(all)` — callable by any unprivileged Cadence script or transaction. The attacker controls all six input parameters.

### Impact Explanation

If a Cadence account owns multiple COAs (stored at different public paths), an attacker who has observed a valid `(signedData, signatures)` pair produced by that account for COA_A can replay it to pass `validateCOAOwnershipProof` for COA_B owned by the same account, by supplying COA_B's address as `evmAddress` and COA_B's capability path as `path`. Any Cadence application that calls `validateCOAOwnershipProof` directly and uses the `isValid` result to authorize privileged actions (e.g., asset transfers, governance votes, bridge operations) is vulnerable to this cross-address replay. The attacker does not need to control the victim's private key — only a previously observed valid signature over any data.

The EVM precompile path (`verifyCOAOwnershipProof` via `coa.sol`'s `isValidSignature`) is not directly vulnerable because it always passes `address(this)` as the EVM address argument, but the Cadence-native `validateCOAOwnershipProof` function is directly reachable by any unprivileged caller.

### Likelihood Explanation

The preconditions are: (1) the victim Cadence account owns more than one COA, and (2) a valid signature from that account over any `signedData` has been observed on-chain. Condition (2) is satisfied any time the account has previously submitted a COA ownership proof in a transaction. Condition (1) is currently rare (the comment notes wallets historically create at most one COA per account), but the protocol does not prevent multiple COAs, and as COA usage grows this assumption weakens. The attacker entry path is a single unprivileged Cadence script call with no special permissions required.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` — either by prepending/appending `evmAddress` to `signedData` before calling `keyList.verify`, or by rejecting calls where `signedData` does not contain the 20-byte `evmAddress`. This mirrors the EIP-712 recommendation in the original report: the target identity must be part of the signed digest.

### Proof of Concept

1. Account A owns two COAs: COA_1 at `/public/coa1` and COA_2 at `/public/coa2`.
2. Account A previously submitted a valid COA ownership proof for COA_1, producing `(signedData_X, sig_X)` observable on-chain.
3. Attacker calls (from any unprivileged Cadence script):
   ```cadence
   EVM.validateCOAOwnershipProof(
       address: accountA,
       path: /public/coa2,          // COA_2's path
       signedData: signedData_X,    // replayed from COA_1 proof
       keyIndices: [0],
       signatures: [sig_X],         // replayed from COA_1 proof
       evmAddress: coa2Address      // COA_2's EVM address
   )
   ```
4. The function verifies `sig_X` over `signedData_X` — valid, because account A signed it.
5. The function checks that the COA at `/public/coa2` has address `coa2Address` — true.
