### Title
`max_nonce_for_validation_skip` Config Field Is Never Read — Validation-Skip Window Permanently Hardcoded to Nonce 1 - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field that is serialized in the production config schema and intended to control how many post-deploy-account nonces may skip stateful validation. The free function `skip_stateful_validations` that implements this logic hardcodes the comparison to `Nonce(Felt::ONE)` and never receives or reads the config field. The configured value is permanently ignored, and the skip window is fixed at exactly nonce 1 regardless of operator configuration.

### Finding Description

`StatefulTransactionValidatorConfig` declares `max_nonce_for_validation_skip: Nonce` with a default of `Nonce(Felt::ONE)` and serializes it in `config_schema.json`: [1](#0-0) [2](#0-1) 

The free function `skip_stateful_validations` is the sole production code path that decides whether to skip the `__validate__` entry-point call for the deploy-account + invoke UX flow. It accepts no config argument and hardcodes the nonce comparison: [3](#0-2) 

`run_pre_validation_checks` calls this function without forwarding `self.config.max_nonce_for_validation_skip`: [4](#0-3) 

By contrast, the legacy `PyValidator` path correctly reads the equivalent field and applies it as an upper bound: [5](#0-4) 

The result is that `max_nonce_for_validation_skip` is a dead config knob: any operator value other than the default `0x1` silently has no effect.

### Impact Explanation

**High — Mempool/gateway admission rejects valid transactions before sequencing.**

When a user submits a `deploy_account` together with multiple invoke transactions (nonces 1, 2, 3, …, N) before the deploy is processed, only the invoke with nonce == 1 skips stateful validation. Invokes with nonces 2 through N are sent through the full `__validate__` entry-point call against a state where the account does not yet exist, causing them to fail with `ValidateFailure` and be dropped at the gateway. An operator who raises `max_nonce_for_validation_skip` to accommodate this pattern gets no benefit; the gateway permanently rejects those valid transactions.

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` (`0x1`) happens to match the hardcoded constant, so the bug is invisible in default deployments. It surfaces only when an operator explicitly raises the value — a documented, public configuration knob — to support users who batch more than one invoke with a fresh deploy. Any such deployment is silently broken.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method) and replace the hardcoded `Nonce(Felt::ONE)` upper-bound check with the configured value, mirroring the logic already present in `PyValidator::should_run_stateful_validations`:

```rust
// In run_pre_validation_checks, pass the config value:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,  // add this
).await?;

// In skip_stateful_validations, use it:
if tx.nonce() >= Nonce(Felt::ONE)
    && tx.nonce() <= max_nonce_for_validation_skip
    && account_nonce == Nonce(Felt::ZERO)
{ ... }
```

### Proof of Concept

1. Deploy a sequencer node with `max_nonce_for_validation_skip = 3` in `gateway_config.static_config.stateful_tx_validator_config`.
2. Submit `deploy_account` (nonce 0) + `invoke` (nonce 1) + `invoke` (nonce 2) + `invoke` (nonce 3) simultaneously for a fresh account.
3. Observe: invoke at nonce 1 is admitted (hardcoded skip fires). Invokes at nonces 2 and 3 are rejected with `ValidateFailure` because `skip_stateful_validations` only checks `tx.nonce() == Nonce(Felt::ONE)` and the account does not yet exist in state.
4. Grep the production source for any read of `config.max_nonce_for_validation_skip` inside `skip_stateful_validations` — none exists. [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
