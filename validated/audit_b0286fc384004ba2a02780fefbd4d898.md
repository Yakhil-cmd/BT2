### Title
Gateway `skip_stateful_validations` admits invoke with arbitrary signature when any pending deploy_account exists for the sender - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function skips the blockifier `__validate__` entry-point check for an Invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that check returns `true` for **any** transaction in the pool for that address — not specifically a deploy_account submitted by the same user — an unprivileged attacker can inject an Invoke with an arbitrary (fake) signature for any account whose deploy_account is currently pending in the mempool. The gateway admits the transaction without running signature verification, violating the invariant that every admitted Invoke must have passed `__validate__`.

### Finding Description

**Desync analogy.** In the external report, `BribeVotingReward` snapshots voting power at `EPOCH_END - 1`, but `Voter.distribute()` runs later; a voter who switches between those two moments receives bribes without delivering emissions. Here, the gateway snapshots the mempool state (via `account_tx_in_pool_or_recent_block`) at validation time, but the transaction it is validating belongs to a *different* sender than the one who submitted the deploy_account. The "snapshot" says "account is known", so the signature check is skipped, but the transaction that will actually execute carries a completely different (attacker-controlled) payload.

**Root cause — `skip_stateful_validations`.** [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:

1. The incoming transaction is an Invoke.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

**`account_tx_in_pool_or_recent_block` is not deploy_account-specific.** [3](#0-2) 

It returns `true` if the address appears in `tx_pool` (any transaction type) or in the mempool's `state` (staged/committed). It does **not** verify that the matching transaction is a deploy_account, nor that it was submitted by the same principal.

**`validate_by_mempool` does not close the gap.** [4](#0-3) 

`Mempool::validate_tx` only checks for duplicate tx_hash and that `tx_nonce >= account_nonce`. An attacker's Invoke with nonce=1 and a fresh tx_hash passes both checks. [5](#0-4) 

**`run_validate_entry_point` is then skipped entirely.** [6](#0-5) 

When `skip_validate=true`, `execution_flags.validate = false`, so `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`. [7](#0-6) 

**The transaction is then forwarded to the mempool.** [8](#0-7) 

### Impact Explanation

An Invoke transaction carrying an arbitrary (attacker-chosen) signature and calldata is admitted to the mempool without any signature verification. This matches the High impact category: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Concrete consequences:
- The attacker's Invoke occupies the nonce=1 slot for the victim's address. The victim's legitimate Invoke with nonce=1 is rejected with `DuplicateNonce` and must perform fee escalation to displace the attacker's transaction.
- The attacker's Invoke is eventually executed by the batcher (where `__validate__` does run and the transaction reverts), consuming block space and delaying the victim's account activation.
- In a full mempool, the attacker can repeatedly inject fake Invokes for every pending deploy_account, displacing legitimate transactions.

### Likelihood Explanation

- deploy_account transactions are publicly visible in the mempool (propagated via P2P).
- The attacker needs only to observe a pending deploy_account and submit a single Invoke with nonce=1 for the same sender address before the victim does.
- No privileged access, no special account state, and no cryptographic material is required.
- The attack window is the entire time the deploy_account sits in the mempool (potentially many blocks).

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the sender address in the mempool. Alternatively, expose a dedicated `deploy_account_in_pool(address)` query from the mempool that only returns `true` when the pending transaction for that address is of type `DeployAccount`. This closes the desync between "account is known to the mempool" and "a valid deploy_account for this exact account is pending".

### Proof of Concept

```
1. Victim calls POST /add_transaction with a valid DeployAccount(sender=A, nonce=0).
   → Gateway validates, adds to mempool. Mempool tx_pool now contains A.

2. Attacker calls POST /add_transaction with:
     Invoke(sender=A, nonce=1, calldata=<drain_funds>, signature=<garbage>)

3. Gateway stateful validation for attacker's Invoke:
   a. get_nonce_from_state(A) → 0  (A not deployed on-chain)
   b. validate_nonce: 0 <= 1 <= max_gap → OK
   c. validate_by_mempool (validate_tx): nonce=1 >= 0, no dup hash → OK
   d. skip_stateful_validations:
        tx.nonce()==1 ✓, account_nonce==0 ✓
        account_tx_in_pool_or_recent_block(A) → TRUE (victim's DeployAccount is in pool)
        → returns true (skip __validate__)
   e. run_validate_entry_point: execution_flags.validate=false → returns Ok immediately
   f. mempool_client.add_tx(attacker_invoke) → admitted

4. Mempool now contains: DeployAccount(A,nonce=0) + Invoke(A,nonce=1,garbage_sig)

5. Victim submits legitimate Invoke(A,nonce=1) → rejected: DuplicateNonce
   (must fee-escalate to displace attacker's transaction)

6. Batcher executes DeployAccount(A) → A deployed.
   Batcher executes attacker's Invoke(A,nonce=1) → __validate__ runs → FAIL (bad sig) → reverted.
   Block space consumed; victim's account activation delayed.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
