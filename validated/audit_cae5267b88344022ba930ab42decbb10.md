### Title
Hardcoded `L1_TX_NATIVE_PRICE` Assumption Misprices Native Resources for L1→L2 Transactions - (File: `basic_bootloader/src/bootloader/constants.rs`)

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` in the ZKsync OS bootloader and used as the denominator when computing the native-resource budget for every L1→L2 priority transaction. The operator-set `native_price` (which reflects actual proving costs for L2 transactions) is deliberately ignored for L1 transactions. When the real proving cost diverges from this hardcoded assumption, L1 transactions receive a native-resource budget that is disproportionate to what they actually pay, causing the protocol to subsidize the difference.

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant `L1_TX_NATIVE_PRICE` is set to `10` with an explicit open TODO:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

In `prepare_and_check_resources` inside `process_l1_transaction.rs`, this constant is unconditionally substituted for the operator-supplied `native_price` when computing `native_per_gas` for priority (L1→L2) transactions:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

For L2 transactions, the operator-set `native_price` is used instead:

```rust
let native_price = system.get_native_price();
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

The resulting `native_per_gas` drives two downstream quantities:

1. `native_prepaid_from_gas = native_per_gas * gas_limit` — the total native budget for the transaction.
2. `native_per_pubdata = gas_per_pubdata * native_per_gas` — the native cost per byte of pubdata. [4](#0-3) 

Inside `create_resources_for_tx`, any `native_prepaid_from_gas` exceeding `MAX_NATIVE_COMPUTATIONAL` is split: the cap goes to computation and the surplus is **withheld for pubdata**:

```rust
let (native_limit, withheld) = if native_limit <= MAX_NATIVE_COMPUTATIONAL {
    (native_limit, S::Resources::from_ergs(Ergs::empty()))
} else {
    let withheld = ...native_limit - MAX_NATIVE_COMPUTATIONAL...;
    (MAX_NATIVE_COMPUTATIONAL, S::Resources::from_native(withheld))
};
``` [5](#0-4) 

When `L1_TX_NATIVE_PRICE = 10` is much lower than the actual `native_price`, `native_per_gas` is inflated by the ratio `actual_native_price / 10`, giving L1 transactions a proportionally larger native budget — both for computation and for pubdata — than they have economically paid for.

### Impact Explanation

**Resource accounting bug / oracle IO mismatch.** The hardcoded price assumption (`10`) plays the same role as FraxPoolV3's hardcoded `$1` collateral price. When the actual proving cost (reflected in the operator-set `native_price`) diverges upward from `10`, every L1→L2 priority transaction receives a native-resource budget that is `actual_native_price / 10` times larger than warranted by its fee payment. The protocol (operator) must absorb the proving cost of the excess native resources consumed. Concretely:

- If `actual_native_price = 1000` and an attacker submits an L1 tx with `gas_price = 10,000` and `gas_limit = 72,000,000`, the L1 tx receives `native_per_gas = 10,000 / 10 = 1,000`, yielding a native budget of `72,000,000,000` — **100× larger** than an L2 tx with identical parameters would receive (`native_per_gas = 10,000 / 1,000 = 10`).
- The surplus above `MAX_NATIVE_COMPUTATIONAL` is withheld for pubdata, allowing the transaction to publish far more pubdata than its fee justifies.
- The protocol must pay the proving cost for all consumed native resources, regardless of what the L1 fee covers.

### Likelihood Explanation

The TODO comment `// TODO (EVM-1157): find a reasonable value for it.` confirms the developers acknowledge this value is provisional. As proving costs evolve (e.g., hardware improvements or increased demand), the operator will adjust `native_price` for L2 transactions, but `L1_TX_NATIVE_PRICE` remains frozen at `10`. Any deployment where `native_price > 10` creates the discrepancy. Since L1→L2 transactions are permissionless (any Ethereum user can submit them via the L1 bridge), the attacker-controlled entry path is fully open.

### Recommendation

- **Short term:** Add a runtime assertion or system log that fires when `L1_TX_NATIVE_PRICE` deviates significantly from the operator-set `native_price`, so operators are alerted to the mismatch.
- **Long term:** Replace the hardcoded `L1_TX_NATIVE_PRICE` with a value that is either (a) derived from the operator-set `native_price` with a bounded discount factor to preserve censorship-resistance, or (b) governed by an on-chain parameter that can be updated as proving costs change. Resolve `TODO (EVM-1157)`.

### Proof of Concept

Assume a deployment where the operator sets `native_price = 1000` (reflecting increased proving costs).

1. Attacker submits an L1→L2 priority transaction with:
   - `gas_price = 10,000`
   - `gas_limit = 72,000,000`
   - `gas_per_pubdata_limit = 1`

2. In `prepare_and_check_resources`:
   - `native_price = L1_TX_NATIVE_PRICE = 10` (hardcoded, ignores operator value of 1000)
   - `native_per_gas = ceil(10,000 / 10) = 1,000`
   - `native_prepaid_from_gas = 1,000 × 72,000,000 = 72,000,000,000`

3. An L2 transaction with identical parameters would compute:
   - `native_per_gas = ceil(10,000 / 1,000) = 10`
   - `native_prepaid_from_gas = 10 × 72,000,000 = 720,000,000`

4. The L1 tx receives a native budget **100× larger** than the L2 tx. The surplus above `MAX_NATIVE_COMPUTATIONAL` is withheld for pubdata, allowing the L1 tx to publish 100× more pubdata than an equivalently-priced L2 tx.

5. The protocol must prove all consumed native resources. The attacker pays the L1 fee (covering only `720,000,000` native units worth of proving cost) while the protocol absorbs the cost of the remaining `~71,280,000,000` native units. [1](#0-0) [6](#0-5) [7](#0-6)
