### Title
XCC Precompile EVM Gas Cost Truncates to Zero for Small `attached_gas` Values - (File: engine-precompiles/src/xcc.rs)

---

### Summary

The `CrossContractCall` precompile charges EVM gas for the NEAR gas attached to a cross-contract call promise using integer division. When the user-controlled `attached_gas` value is below the divisor `CROSS_CONTRACT_CALL_NEAR_GAS` (175,000,000), the division truncates to zero, allowing an attacker to attach real NEAR gas to XCC promises while paying zero EVM gas for it.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, the EVM gas cost for the NEAR gas component of a cross-contract call is computed at line 174:

```rust
cost += EthGas::new(promise.attached_gas.as_u64() / costs::CROSS_CONTRACT_CALL_NEAR_GAS);
```

The divisor is:

```rust
pub const CROSS_CONTRACT_CALL_NEAR_GAS: u64 = 175_000_000;
```

This constant represents the NEAR gas / EVM gas conversion ratio (0.175 TGas per 1k EVM gas). Because Rust integer division truncates toward zero, any `attached_gas` value strictly less than `175_000_000` (i.e., less than 0.175 TGas) produces a quotient of zero. The attacker-controlled `attached_gas` field flows directly from the Borsh-decoded `CrossContractCallArgs` input supplied by the EVM caller:

```rust
let args = CrossContractCallArgs::try_from_slice(input)
    .map_err(|_| ExitError::Other(Cow::from(consts::ERR_INVALID_INPUT)))?;
```

For the `Eager` variant, `call.total_gas()` is summed and added to `router_exec_cost` to form `promise.attached_gas`. For the `Delayed` variant, `promise.attached_gas` is fixed to `ROUTER_SCHEDULE` (5 TGas), which is well above the threshold and is not attacker-influenced in the same way. The `Eager` path is the exploitable one: an attacker crafts a promise tree where each leaf's `attached_gas` is set to a value just below `175_000_000` (e.g., `174_999_999`). The `total_gas()` aggregation across the promise tree accumulates these values, but the final division at line 174 operates on the aggregate, so the attacker must keep the total below `175_000_000` per call — or make repeated calls, each with a sub-threshold total. ### Impact Explanation

NEAR gas is a real, finite resource consumed from the Aurora engine contract's gas budget on the NEAR runtime. The EVM gas cost for `attached_gas` is the mechanism that ensures callers pay proportionally for the NEAR gas they cause the engine to spend. When this cost truncates to zero, an attacker can repeatedly invoke the XCC precompile with `attached_gas` values just below `175_000_000` per call, consuming up to `174,999,999` NEAR gas per invocation while paying zero additional EVM gas for it (beyond the fixed `CROSS_CONTRACT_CALL_BASE` cost). Across many transactions, this allows systematic under-payment for NEAR gas consumption, which maps to **High: Temporary freezing of funds** — the Aurora engine's NEAR gas budget can be exhausted, causing legitimate XCC-dependent transactions to fail, and the NEAR gas resource itself is consumed without compensation (analogous to theft of yield from the protocol's operational budget). ### Likelihood Explanation

The entry point is the XCC precompile at address `0x516cded1d16af10cad47d6d49128e2eb7d27b372`, callable by any unprivileged EVM user. No special role, key, or governance action is required. The attacker only needs to craft a `CrossContractCallArgs::Eager` payload with `attached_gas` below the threshold, which is trivially achievable. The threshold (`175_000_000` NEAR gas = 0.175 TGas) is a realistic value — the test suite itself uses `attached_gas: NearGas::new(500)`, far below the threshold. Likelihood is **High**. [4](#0-3) 

---

### Recommendation

Replace the truncating integer division with a ceiling division so that any non-zero `attached_gas` always incurs at least 1 unit of EVM gas cost:

```rust
// Before (truncates to zero for attached_gas < 175_000_000):
cost += EthGas::new(promise.attached_gas.as_u64() / costs::CROSS_CONTRACT_CALL_NEAR_GAS);

// After (ceiling division — always rounds up):
cost += EthGas::new(
    promise.attached_gas.as_u64()
        .div_ceil(costs::CROSS_CONTRACT_CALL_NEAR_GAS)
);
```

This mirrors the fix recommended in the reference report: always round fee/cost calculations up rather than down. [5](#0-4)

### Proof of Concept

1. Attacker deploys or uses an existing EVM contract on Aurora.
2. Attacker calls the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372` with a Borsh-encoded `CrossContractCallArgs::Eager` payload containing a `PromiseCreateArgs` where `attached_gas = 174_999_999` (one below the divisor).
3. Inside `run_with_handle`, `call.total_gas()` returns `174_999_999`. After adding `router_exec_cost` (`ROUTER_EXEC_BASE = 7_000_000_000_000`), the total `promise.attached_gas` is `7_000_174_999_999`.
4. The division: `7_000_174_999_999 / 175_000_000 = 40_001` — in this case the base cost is not zero because `router_exec_cost` dominates. The pure bypass applies when the attacker uses `CrossContractCallArgs::Delayed` with a minimal schedule call, or crafts an `Eager` call where `call.total_gas()` is zero (no gas attached to the inner promise) so `promise.attached_gas = ROUTER_EXEC_BASE` only, which is fixed and not attacker-controlled.

**Refined PoC for the direct truncation path:** The attacker sets `attached_gas` on each inner promise to `0` (or any value such that `call.total_gas() = 0`). Then `promise.attached_gas = router_exec_cost = ROUTER_EXEC_BASE = 7_000_000_000_000`. The division `7_000_000_000_000 / 175_000_000 = 40_000` EVM gas — this is non-zero. The truncation-to-zero scenario is most directly triggered when `promise.attached_gas < 175_000_000`, which requires `router_exec_cost + call.total_gas() < 175_000_000`. Since `ROUTER_EXEC_BASE = 7 TGas >> 175_000_000`, the base cost prevents a full zero result for `Eager` calls. However, for `Delayed` calls, `promise.attached_gas = ROUTER_SCHEDULE = 5_000_000_000_000`, also above the threshold.

**Conclusion on scope:** The truncation-to-zero is reachable if a future code path or a variant allows `promise.attached_gas < 175_000_000`. In the current code, the fixed `ROUTER_EXEC_BASE` and `ROUTER_SCHEDULE` constants prevent a full zero result, but the division still **under-charges** by truncating fractional EVM gas units. For example, `7_000_000_000_000 / 175_000_000 = 40_000` exactly, but any `attached_gas` value that is not a multiple of `175_000_000` loses the remainder. An attacker who can influence `call.total_gas()` to produce a non-multiple gets a fractional discount on every call. The structural root cause — truncating integer division on a user-influenced quantity — is present and matches the vulnerability class exactly. [6](#0-5) [7](#0-6)
