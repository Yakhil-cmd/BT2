### Title
Reentrancy in `claim_rewards` Allows Theft of Unclaimed Staker Yield via Malicious Reward Address - (File: src/staking/staking.cairo)

---

### Summary
The `claim_rewards` function in `staking.cairo` transfers tokens to the staker's `reward_address` before committing the zeroed `unclaimed_rewards_own` to storage. Because `reward_address` is fully user-controlled, a staker can set it to a malicious contract that re-enters `claim_rewards` during the token transfer callback, draining accumulated rewards multiple times.

---

### Finding Description

In `staking.cairo`, `claim_rewards` delegates to the internal helper `send_rewards_to_staker`:

```
// staking.cairo:411-431
fn claim_rewards(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    self.general_prerequisites();
    let mut staker_info = self.internal_staker_info(:staker_address);
    ...
    self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
    self.write_staker_info(:staker_address, :staker_info);   // ← storage write AFTER transfer
    amount
}
```

Inside `send_rewards_to_staker`:

```
// staking.cairo:1614-1629
fn send_rewards_to_staker(...) {
    let amount = staker_info.unclaimed_rewards_own;
    claim_from_reward_supplier(...);
    token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into()); // ← transfer first
    staker_info.unclaimed_rewards_own = Zero::zero();  // ← zeroed in memory only, not storage
    ...
}
```

The sequence is:
1. Read `unclaimed_rewards_own = X` from storage into local `staker_info`.
2. Transfer `X` tokens to `reward_address` (external call).
3. Zero `staker_info.unclaimed_rewards_own` in the local struct (memory only).
4. Return to `claim_rewards`, which calls `write_staker_info` to persist the zero.

On Starknet, storage writes take effect immediately within a transaction, but step 4 has not yet executed when the external call in step 2 fires. A reentrant call to `claim_rewards` at step 2 reads storage and still sees `unclaimed_rewards_own = X`, allowing a second (and further) transfer of the same amount.

The `reward_address` is fully attacker-controlled: any staker can call `change_reward_address` to point it at a malicious contract. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker who has accumulated `unclaimed_rewards_own` can drain those rewards repeatedly in a single transaction. Each reentrant call reads the un-updated storage value and triggers another transfer from the reward supplier. The attacker profits directly; the reward supplier's balance is depleted beyond what was legitimately owed.

---

### Likelihood Explanation

**High.** The attack requires only:
1. Deploying a malicious contract (trivial on Starknet).
2. Calling `change_reward_address` to point to that contract (permissionless, no privileged role needed).
3. Calling `claim_rewards` once rewards have accrued.

No leaked keys, no governance compromise, no third-party dependency. Any staker can execute this. [3](#0-2) 

---

### Recommendation

Apply the Checks-Effects-Interactions pattern: zero `staker_info.unclaimed_rewards_own` and commit it to storage **before** calling `token_dispatcher.checked_transfer`. Concretely, in `send_rewards_to_staker`, move the storage write ahead of the transfer, or restructure `claim_rewards` to call `write_staker_info` before `send_rewards_to_staker`.

The analogous fix is already applied in `pool.cairo`'s `claim_rewards`, which correctly calls `write_pool_member_info` (line 362) before `reward_token.checked_transfer` (line 366). [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `MaliciousReward` contract:
   ```
   fn on_receive_tokens(...) {
       // called when STRK is transferred to this address
       IStakingDispatcher { contract_address: STAKING }.claim_rewards(ATTACKER_STAKER);
   }
   ```
2. Attacker calls `staking.change_reward_address(MaliciousReward)`.
3. Attacker waits for rewards to accrue (e.g., `unclaimed_rewards_own = 1000 STRK`).
4. Attacker calls `staking.claim_rewards(ATTACKER_STAKER)`.
5. Execution enters `send_rewards_to_staker`:
   - `amount = 1000` (read from storage).
   - `checked_transfer(MaliciousReward, 1000)` fires.
   - `MaliciousReward.on_receive_tokens` re-enters `claim_rewards`.
   - Reentrant call reads storage: `unclaimed_rewards_own` still `1000` (not yet zeroed).
   - Another `checked_transfer(MaliciousReward, 1000)` fires.
   - Repeat until reward supplier is drained or gas runs out.
6. Outer call finally writes `unclaimed_rewards_own = 0` to storage — too late. [5](#0-4) [6](#0-5)
