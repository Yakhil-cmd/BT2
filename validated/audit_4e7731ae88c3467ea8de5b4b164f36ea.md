### Title
Pool Admin Can Bypass Oracle Timelock by Setting `priceProviderTimelock = 0`, Enabling Instant Oracle Swap to Drain LP Funds - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
No minimum `priceProviderTimelock` is enforced at pool creation. A pool admin who sets `priceProviderTimelock = 0` can propose and execute a malicious oracle change across two transactions in the same block, bypassing the timelock protection that is the sole guard against sudden oracle manipulation of LP funds.

### Finding Description

`createPool` stores the caller-supplied `params.priceProviderTimelock` directly into `priceProviderTimelock[pool]` with no lower-bound check: [1](#0-0) 

`_validatePoolParameters` validates tokens, admin, fee caps, and initial amounts, but never checks that `priceProviderTimelock` is above any minimum: [2](#0-1) 

When `priceProviderTimelock[pool] == 0`, `proposePoolPriceProvider` computes:

```
executeAfter = block.timestamp + 0  →  executeAfter == block.timestamp
``` [3](#0-2) 

`executePoolPriceProviderUpdate` then checks:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...)
``` [4](#0-3) 

Because `block.timestamp < block.timestamp` is always `false`, the guard never fires. The factory's `ReentrancyGuardTransient` prevents both calls in a single transaction, but transient storage is cleared between transactions, so the admin issues two transactions in the same block — both land at the same `block.timestamp` — and the oracle is swapped with zero effective delay.

The new oracle only needs to pass `_validatePriceProvider`, which checks token addresses but not price sanity: [5](#0-4) 

A malicious oracle returning the correct token pair but extreme bid/ask values passes this check. The pool's swap math then prices every bin relative to the manipulated mid-price, allowing the attacker to extract LP principal at the distorted rate.

The timelock mechanism was introduced specifically to fix audit finding M01 ("Admin can arbitrarily change the pool priceProvider"), but the fix did not enforce a minimum delay: [6](#0-5) 

### Impact Explanation

Direct loss of LP principal. After the oracle is swapped, any swap against the pool executes at the manipulated price. The pool's bin balances (`binTotals.scaledToken0/1`) are fully exposed to the distorted price curve, so a single swap can drain all token reserves held by LPs. `removeLiquidity` correctly enforces per-position share accounting, but that protection is irrelevant once the swap has already transferred LP value to the attacker. [7](#0-6) 

### Likelihood Explanation

Low. Requires a pool admin who acts maliciously after attracting LP deposits, and LPs who do not verify `priceProviderTimelock[pool]` on-chain before depositing. The value is emitted in `PoolCreated` and readable from factory storage, so diligent LPs can detect it — but most will not.

### Recommendation

Add a minimum timelock floor in `_validatePoolParameters` (or a dedicated constant):

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// in _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

This ensures that even a mutable-oracle pool gives LPs at least 24 hours to observe a pending oracle change and exit before it takes effect.

### Proof of Concept

```solidity
// Step 1 – pool admin creates pool with zero timelock
PoolParameters memory params = ...;
params.priceProviderTimelock = 0;          // no minimum enforced
address pool = factory.createPool(params); // priceProviderTimelock[pool] = 0

// Step 2 – LPs deposit; pool accumulates real value
pool.addLiquidity(lp, salt, deltas, callbackData, "");

// Step 3 – admin deploys malicious oracle (correct tokens, extreme prices)
MaliciousOracle mal = new MaliciousOracle(token0, token1);
mal.setBidAndAskPrice(1, 2);  // extreme: 1 wei bid, 2 wei ask

// Step 4 – Tx A (block N): propose
factory.proposePoolPriceProvider(pool, address(mal));
// pendingPriceProviderExecuteAfter[pool] = block.timestamp (N)

// Step 5 – Tx B (same block N): execute — block.timestamp == execAfter, guard passes
factory.executePoolPriceProviderUpdate(pool);
// pool now prices all swaps against the malicious oracle

// Step 6 – drain LP funds via bad-price swap
pool.swap(attacker, true, largeAmount, 0, callbackData, "");
``` [8](#0-7)

## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
