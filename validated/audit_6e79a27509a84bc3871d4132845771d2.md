### Title
`skip_stateful_validations` Accepts Nonce-1 Invoke with Invalid Signature When Any Transaction Exists in Mempool for Target Account - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function bypasses the `__validate__` entry-point (signature check) for an Invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. That helper returns `true` for **any** transaction type in the pool, not specifically a `deploy_account`. An attacker who observes a victim's `deploy_account` in the mempool can therefore inject a nonce=1 Invoke with an arbitrary/invalid signature for the victim's address, have it accepted by the gateway without signature verification, and — via fee escalation — replace the victim's legitimate nonce=1 Invoke. When the batcher later executes the block, `AccountTransaction::new_for_sequencing` re-enables `validate: true`, so the invalid Invoke fails `__validate__`, is reverted, and the victim's nonce=1 is permanently consumed.

### Finding Description

**Root cause — `skip_stateful_validations`** (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, lines 429–461):

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())   // ← any tx type
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The comment claims the check is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch is circular: nonce=1 invokes can themselves pass via this same skip, so their presence in the pool does not imply a `deploy_account` exists.

**`account_tx_in_pool_or_recent_block` is type-blind** (`crates/apollo_mempool/src/mempool.rs`, lines 697–700):

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for any transaction type (Invoke, Declare, DeployAccount) in the pool.

**`run_validate_entry_point` sets `validate: !skip_validate`** (`crates/apollo_gateway/src/stateful_transaction_validator.rs`, lines 308–312):

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

When `skip_validate = true`, `validate = false`, so `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`. [4](#0-3) 

**Batcher re-enables validation** (`crates/blockifier/src/transaction/account_transaction.rs`, lines 147–155):

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        only_query: false,
        charge_fee: enforce_fee(&tx, false),
        validate: true,          // ← always true
        strict_nonce_check: true,
    };
    AccountTransaction { tx, execution_flags }
}
``` [5](#0-4) 

So the gateway skips signature verification, but the blockifier enforces it during actual execution. The invalid Invoke fails `__validate__`, is reverted, yet the nonce is already incremented (in `handle_nonce`, called before `__validate__`). [6](#0-5) 

**Fee escalation enables replacement** (`crates/apollo_mempool/src/mempool.rs`, lines 760–792): when `enable_fee_escalation` is on, a higher-tip transaction at the same `(address, nonce)` replaces the existing one. The attacker can set an arbitrarily high tip to displace the victim's legitimate nonce=1 Invoke. [7](#0-6) 

### Impact Explanation

**Broken invariant**: The gateway must not admit a transaction whose `__validate__` entry point has not been verified, unless the account is already deployed on-chain. The skip is only safe when a `deploy_account` for the same address is pending; the current check accepts any transaction type.

**Corrupted value**: `execution_flags.validate = false` is set for an Invoke with an invalid/attacker-controlled signature, allowing it to enter the mempool without signature verification.

**Consequence**: An attacker can grief any user who submits a `deploy_account` by:
1. Replacing the victim's legitimate nonce=1 Invoke with an invalid-signature Invoke (via fee escalation).
2. Causing the victim's nonce=1 to be consumed by a failed transaction.
3. Forcing the victim to resubmit with nonce=2, while their original Invoke is permanently lost from the mempool.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

- The victim's `deploy_account` is publicly visible in the mempool.
- The victim's address is deterministically computable from the `deploy_account` parameters (class_hash, salt, constructor_calldata), all of which are public.
- The attacker only needs to submit a single transaction with a tip above the fee-escalation threshold.
- No privileged access is required; any unprivileged user can execute this attack.
- The `deploy_account + invoke` UX flow is explicitly documented and tested as a supported pattern, making it a high-value target.

### Recommendation

Replace the type-blind `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the target address. Add a dedicated mempool query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, or filter the pool contents to only count `DeployAccount` transactions when deciding whether to skip `__validate__`.

Alternatively, add a type check inside `skip_stateful_validations` itself: after `account_tx_in_pool_or_recent_block` returns `true`, query the mempool for the specific transaction at `(address, nonce=0)` and verify it is a `DeployAccount`.

### Proof of Concept

1. Victim submits `deploy_account` for address `A` (class_hash `C`, salt `S`). Address `A` is deterministic and publicly visible in the mempool.
2. Attacker computes `A` from the public `deploy_account` parameters.
3. Attacker submits `InvokeV3` for address `A`, nonce=1, with an empty or garbage signature, and a tip exceeding the victim's nonce=1 Invoke tip by `fee_escalation_percentage`.
4. Gateway path:
   - `validate_nonce`: account_nonce=0, tx_nonce=1, `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
   - `validate_by_mempool` → `mempool.validate_tx` → `validate_fee_escalation` → higher tip → returns `Some(victim_tx_reference)` (replacement allowed).
   - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)` = `true` (deploy_account is in pool) → returns `true`.
   - `run_validate_entry_point` with `validate=false` → `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`.
5. Mempool: attacker's Invoke replaces victim's nonce=1 Invoke via fee escalation.
6. Batcher executes block: `deploy_account` runs (nonce → 1), then attacker's Invoke runs with `validate=true` → `__validate__` fails (invalid signature) → transaction reverted, nonce consumed.
7. Victim's original nonce=1 Invoke is gone; victim must resubmit with nonce=2.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn handle_nonce(
        state: &mut dyn State,
        tx_info: &TransactionInfo,
        strict: bool,
    ) -> TransactionPreValidationResult<()> {
        if tx_info.is_v0() {
            return Ok(());
        }

        let address = tx_info.sender_address();
        let account_nonce = state.get_nonce_at(address)?;
        let incoming_tx_nonce = tx_info.nonce();
        let valid_nonce = if strict {
            account_nonce == incoming_tx_nonce
        } else {
            account_nonce <= incoming_tx_nonce
        };
        if valid_nonce {
            return Ok(state.increment_nonce(address)?);
        }
        Err(TransactionPreValidationError::InvalidNonce {
            address,
            account_nonce,
            incoming_tx_nonce,
        })
    }
