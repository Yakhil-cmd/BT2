### Title
Missing Access Control on Ledger Service gRPC `Set` RPC Allows Unauthorized Execution State Mutation - (`cmd/ledger/main.go`, `ledger/remote/service.go`)

---

### Summary

The standalone `LedgerService` gRPC server (`cmd/ledger`) exposes a `Set` RPC that directly mutates the execution state ledger (MTrie). The gRPC server is created with no authentication interceptor, no TLS client-certificate requirement, and no caller-identity check inside the handler. When the service is configured to listen on a TCP address (the documented and supported deployment mode), any network-reachable attacker can call `Set` with arbitrary keys and values, writing attacker-controlled data into the execution state trie that the Execution Node reads as ground truth.

---

### Finding Description

**Root cause 1 — gRPC server has no authentication or authorization:**

In `cmd/ledger/main.go`, the gRPC server is created with only message-size options:

```go
grpcServer := grpc.NewServer(
    grpc.MaxRecvMsgSize(int(*maxRequestSize)),
    grpc.MaxSendMsgSize(int(*maxResponseSize)),
)
```

No `grpc.Creds(...)`, no unary interceptor for token/certificate validation, and no mutual-TLS is configured. [1](#0-0) 

**Root cause 2 — TCP listener is a documented, supported deployment mode:**

The README and the flag definition both show `0.0.0.0:9000` as the TCP bind address, making the service reachable from any host that can reach that port: [2](#0-1) [3](#0-2) 

**Root cause 3 — `Set` handler performs no caller check:**

`ledger/remote/service.go` `Set` validates only argument shape (non-nil state, non-empty keys, matching lengths) and then unconditionally calls `s.ledger.Set(update)`: [4](#0-3) 

No identity, token, or peer-certificate check exists anywhere in the call path. The same is true for `InitialState`, `HasState`, `GetSingleValue`, `Get`, and `Prove`. [5](#0-4) 

**Root cause 4 — The client itself uses insecure credentials, confirming no mutual-TLS is expected:**

`ledger/remote/client.go` connects with `insecure.NewCredentials()`, confirming the server never requires a client certificate: [6](#0-5) 

---

### Impact Explanation

The execution state ledger (MTrie) is the canonical source of truth for all Flow account balances, contract bytecode, and storage registers. The `Set` RPC appends a new trie node with attacker-supplied key-value pairs and returns the resulting new state root. An attacker who can reach the TCP port can:

1. Overwrite arbitrary account storage registers (e.g., token balances, contract code) with attacker-chosen values.
2. Cause the Execution Node to produce execution results derived from the corrupted trie, which will diverge from honest nodes.
3. Permanently corrupt the WAL/checkpoint files on disk, making recovery require a full state re-sync.

This is an unauthorized state mutation of the execution layer — the highest-severity impact class in the Flow protocol.

---

### Likelihood Explanation

The README explicitly documents and encourages TCP deployment:

```
./flow-ledger-service \
  -triedir /path/to/trie \
  -ledger-service-tcp 0.0.0.0:9000
``` [7](#0-6) 

Any operator following the documented instructions exposes the service on a TCP port with no authentication. The attacker needs only a standard gRPC client (e.g., `grpcurl`) and network reachability to the port — no credentials, no staking, no privileged access.

---

### Recommendation

1. **Require mutual TLS** on the TCP listener: generate a server certificate and require client certificates signed by a trusted CA known only to the Execution Node. The existing `grpcutils.DefaultServerTLSConfig` / `DefaultClientTLSConfig` infrastructure used by other Flow gRPC servers should be reused.
2. **Add a gRPC unary interceptor** that verifies the caller's identity before dispatching any RPC, rejecting unauthenticated connections with `codes.Unauthenticated`.
3. **Restrict the `Set` (and `Prove`) RPCs** to authenticated callers only; read-only RPCs (`Get`, `HasState`, `InitialState`) may have a separate, less-privileged access tier.
4. **Document that TCP deployment without mTLS is insecure** and should only be used on a private, firewalled network segment.

---

### Proof of Concept

An attacker with TCP access to the ledger service port can corrupt the execution state with a standard gRPC client:

```bash
# 1. Obtain the current initial state root (no auth required)
grpcurl -plaintext <host>:9000 ledger.LedgerService/InitialState

# 2. Write an arbitrary value into account storage
grpcurl -plaintext -d '{
  "state": {"hash": "<base64_of_current_state_hash>"},
  "keys": [{"parts": [{"type": 0, "value": "<account_address>"}, {"type": 2, "value": "<storage_key>"}]}],
  "values": [{"data": "<attacker_controlled_bytes>", "is_nil": false}]
}' <host>:9000 ledger.LedgerService/Set
```

The call succeeds and returns a new state root. The Execution Node, which reads from this same ledger, will subsequently execute transactions against the corrupted state, producing incorrect results that diverge from the honest network.

The gRPC server creation at `cmd/ledger/main.go:121-128` and the `Set` handler at `ledger/remote/service.go:133-198` are the two necessary vulnerable steps — neither performs any caller authentication. [8](#0-7) [4](#0-3)

func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
	if req.State == nil || len(req.State.Hash) != len(ledger.State{}) {
		return nil, status.Error(codes.InvalidArgument, "invalid state")
	}

	if len(req.Keys) == 0 {
		return nil, status.Error(codes.InvalidArgument, "keys cannot be empty")
	}

	if len(req.Keys) != len(req.Values) {
		return nil, status.Error(codes.InvalidArgument, "keys and values length mismatch")
	}

	var state ledger.State
	copy(state[:], req.State.Hash)

	keys := make([]ledger.Key, len(req.Keys))
	for i, protoKey := range req.Keys {
		key, err := protoKeyToLedgerKey(protoKey)
		if err != nil {
			return nil, err // protoKeyToLedgerKey already returns status.Error
		}
		keys[i] = key
	}

	values := make([]ledger.Value, len(req.Values))
	for i, protoValue := range req.Values {
		var value ledger.Value
		// Reconstruct the original value type using is_nil flag
		// This preserves the distinction between nil and []byte{} that protobuf loses
		if len(protoValue.Data) == 0 {
			if protoValue.IsNil {
				// Original value was nil
				value = nil
			} else {
				// Original value was []byte{} (empty slice)
				value = ledger.Value([]byte{})
			}
		} else {
			// Non-empty value, use data as-is
			value = ledger.Value(protoValue.Data)
		}
		values[i] = value
	}

	update, err := ledger.NewUpdate(state, keys, values)
	if err != nil {
		return nil, status.Error(codes.InvalidArgument, err.Error())
	}

	newState, trieUpdate, err := s.ledger.Set(update)
	if err != nil {
		return nil, status.Error(codes.Internal, err.Error())
	}

	// Encode trie update using centralized encoding function to ensure
	// client and server use the same encoding method
	trieUpdateBytes := encodeTrieUpdateForTransport(trieUpdate)

	return &ledgerpb.SetResponse{
		NewState: &ledgerpb.State{
			Hash: newState[:],
		},
		TrieUpdate: trieUpdateBytes,
	}, nil
}
