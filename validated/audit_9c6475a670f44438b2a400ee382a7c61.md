### Title
`max_l2_gas_amount` Admission Guard Skipped for `Declare` Transactions, Allowing Oversized Gas Bounds to Bypass Gateway - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces a `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips this check for `Declare` transactions. An unprivileged user can submit a `Declare` transaction with `l2_gas.max_amount` set to any value up to `u64::MAX`, bypassing the gateway's admission guard and causing the transaction to be accepted into the mempool despite violating the protocol's own resource-bound invariant.

### Finding Description

In `validate_resource_bounds`, the check that rejects transactions whose `l2_gas.max_amount` exceeds `config.max_l2_gas_amount` (production value: 1,210,000,000) is gated behind a type match that silently passes for `Declare`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← no check; falls through
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The `TODO` comment confirms the team is aware the check is absent for `Declare`. The stateful validator's `validate_resource_bounds` only checks `max_price_per_unit` (not `max_amount`) for all transaction types, so it provides no compensating control: [2](#0-1) 

The full gateway admission path (`add_tx_inner`) runs stateless validation, then stateful validation, then sends the transaction to the mempool. Neither stage rejects a `Declare` with an oversized `l2_gas.max_amount`: [3](#0-2) 

The only downstream guard is `verify_can_pay_committed_bounds` inside `perform_pre_validation_stage`, which fires at **blockifier execution time** (not at admission time) and checks whether the account balance covers `max_amount × max_price_per_unit`. This is a post-admission check, not a pre-admission guard: [4](#0-3) 

The production `max_l2_gas_amount` is 1,210,000,000 gas units: [5](#0-4) 

The test `valid_l2_gas_amount_on_declare` explicitly documents and asserts this bypass as passing: [6](#0-5) 

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool must satisfy `l2_gas.max_amount ≤ max_l2_gas_amount`. This invariant holds for `Invoke` and `DeployAccount` but not for `Declare`.

**Admission of invalid transactions (High):** A `Declare` transaction with `l2_gas.max_amount = u64::MAX` passes all gateway checks and enters the mempool. The transaction is invalid by the sequencer's own admission rules yet is accepted. This matches the impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

**Downstream execution failure:** When the batcher picks up such a transaction, `verify_can_pay_committed_bounds` computes `committed_fee = u64::MAX × min_gas_price (8 × 10⁹) ≈ 1.47 × 10²⁹`, which no realistic account can cover, causing the transaction to fail pre-validation. The transaction occupies a mempool slot and wastes batcher CPU before being rejected.

**Asymmetry with other types:** An `Invoke` or `DeployAccount` with the same `l2_gas.max_amount` is rejected at the gateway with `MaxGasAmountTooHigh`. The `Declare` path has no equivalent rejection. [7](#0-6) 

### Likelihood Explanation

The trigger requires no privilege: any user with a valid sender address and a Sierra contract class can submit a `Declare` transaction. The bypass is unconditional — the `if let RpcTransaction::Declare(_) = tx {}` branch always skips the check. The production config has `validate_resource_bounds: true` for the stateless validator, so the guard is active for other transaction types but structurally absent for `Declare`.

### Recommendation

Remove the `Declare` exemption and apply the same `max_l2_gas_amount` ceiling uniformly:

```rust
// Remove the Declare exemption:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher gas ceiling (e.g., for large contract classes), introduce a separate `max_l2_gas_amount_declare` config parameter rather than removing the check entirely. Update the test `valid_l2_gas_amount_on_declare` to assert rejection once the check is applied.

### Proof of Concept

The existing test `valid_l2_gas_amount_on_declare` already demonstrates the bypass: it constructs a `Declare` transaction with `l2_gas.max_amount = 200` against a config that sets `max_l2_gas_amount = 100` and asserts `Ok(())` — i.e., the oversized amount is accepted: [6](#0-5) 

To confirm the asymmetry, the same `max_l2_gas_amount = 100` / `max_amount = 200` combination on `Invoke` or `DeployAccount` returns `MaxGasAmountTooHigh`: [7](#0-6) 

A concrete end-to-end exploit:

1. Craft an `RpcDeclareTransactionV3` with any valid Sierra class, `l2_gas.max_amount = u64::MAX`, and `l2_gas.max_price_per_unit = min_gas_price`.
2. Submit via `add_tx`. The stateless validator skips the `max_l2_gas_amount` check (line 79 match arm). The stateful validator checks only `max_price_per_unit` (passes). The transaction enters the mempool.
3. The batcher dequeues the transaction and calls `perform_pre_validation_stage`. `verify_can_pay_committed_bounds` computes `committed_fee ≈ 1.47 × 10²⁹` and fails because the account balance is insufficient.
4. The transaction is rejected post-admission, having consumed a mempool slot and batcher execution time.

Repeat with distinct class hashes to exhaust mempool capacity with transactions that are invalid at admission time but accepted by the gateway.
