### Title
Presign Accepts Triples Generated by a Different Participant Set Without Validation - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary

The OT-based ECDSA `presign` function explicitly omits the check that the participant set used for presigning matches the participant set that generated the input triples. This is directly analogous to the VaultsRegistry owner bypass: just as the registry owner can register vaults not created by an approved factory, a presign caller can supply triples generated by a different (unapproved) participant set, bypassing the intended participant-consistency invariant. A malicious coordinator or caller can exploit this to corrupt presign outputs, causing honest parties to accept cryptographically inconsistent presignatures.

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs` at lines 38–40, the presign initialization function contains an explicit comment acknowledging the omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The word "presumably" is the root cause: this is an unenforced assumption, not a validated invariant. The `presign` function accepts `PresignArguments` containing `triple0` and `triple1` (each a `(TripleShare, TriplePublic)` pair). It validates that the threshold matches between the presign call and the triples:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(...)
}
``` [2](#0-1) 

But it performs **no check** that the participants who generated the triples are the same as the participants listed in the presign call. The `TriplePublic` struct carries a `threshold` but not the originating participant set, so there is no data available to enforce this invariant even if the caller wanted to.

**Exploit path:**

1. A set of honest parties runs triple generation with participant set `{P1, P2, P3}`, producing `(TripleShare_i, TriplePublic)` for each `P_i`.
2. A malicious coordinator initializes presign with participant set `{P1, P2, P4}` (substituting `P4` for `P3`), passing in the triples from step 1.
3. `P4` was not present during triple generation and holds no valid triple shares. `P4` can supply arbitrary values as its triple shares.
4. The presign function proceeds without error, since the only check performed is threshold equality.
5. The presign protocol runs with mismatched Lagrange coefficients: the triple shares were evaluated at indices for `{P1, P2, P3}`, but the presign protocol interpolates using indices for `{P1, P2, P4}`.
6. The resulting presignature is cryptographically inconsistent — the combined nonce `k` and the combined `k^{-1} * x` shares do not correspond to any valid secret key reconstruction. [3](#0-2) 

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept inconsistent participant sets or unusable cryptographic outputs.**

Honest parties `P1` and `P2` complete the presign protocol and store a `PresignOutput` they believe is valid. When they later attempt to sign a message using this presignature, the signature will be invalid because the underlying nonce material was computed with mismatched Lagrange coefficients. This permanently denies signing for that presignature slot. If the attacker can repeatedly trigger this (e.g., by controlling the coordinator role), they can exhaust all pre-generated triples, causing a permanent denial of signing for honest parties.

Additionally, because `P4` supplies arbitrary values as triple shares, the presign protocol may leak information about honest parties' secret key shares through the combined presign outputs, depending on what `P4` injects — a potential path toward secret share extraction under repeated sessions.

This maps directly to the allowed High impact: *"Corruption of presign outputs so honest parties accept inconsistent participant sets or unusable cryptographic outputs."*

### Likelihood Explanation

The attacker needs only to control the coordinator or the presign initialization call — a role that is reachable without any privileged key material. In the library's architecture, the coordinator is a participant who drives the protocol; a malicious participant who is elected coordinator can trivially supply a mismatched participant list to `presign`. No cryptographic break is required. The omission is explicitly documented in the source code, confirming it is a known gap rather than an oversight.

### Recommendation

Enforce that the participant set used for presigning is a subset of (or identical to) the participant set that generated the triples. This requires either:

1. **Embedding the originating participant set into `TriplePublic`** at generation time, and then checking in `presign` that the presign participant set matches the stored set.
2. **Alternatively**, require each participant to prove possession of a valid triple share (e.g., via a commitment check against `TriplePublic`) before the presign protocol proceeds.

Remove the "presumably" comment and replace it with an enforced invariant, mirroring the pattern used elsewhere in the codebase (e.g., the threshold consistency check already present at lines 43–47).

### Proof of Concept

**Setup:**
- Participants: `P1 = Participant(1)`, `P2 = Participant(2)`, `P3 = Participant(3)`, `P4 = Participant(4)`
- Threshold: 2

**Steps:**
1. Run `generate_triple([P1, P2, P3], threshold=2)` to completion. Each of `P1`, `P2`, `P3` receives a `TripleShare` and the shared `TriplePublic`.
2. Construct `PresignArguments` using `P1`'s `TripleShare` for `triple0` and `triple1`, with `threshold=2`.
3. Call `presign([P1, P2, P4], me=P1, args)` — note `P4` replaces `P3` in the participant list.
4. Observe: `presign` returns `Ok(...)` without error, despite `P4` never having participated in triple generation.
5. `P4` supplies arbitrary scalars as its triple share contributions during the presign rounds.
6. The protocol completes; `P1` and `P2` store a `PresignOutput`.
7. Attempt to use this `PresignOutput` in `sign(...)`: the resulting signature fails verification against the public key, confirming the presignature is cryptographically corrupted.

The root cause is confirmed at `src/ecdsa/ot_based_ecdsa/presign.rs` lines 38–40: the participant-triple consistency check is explicitly omitted with no compensating control. [4](#0-3)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
