### Title
Bank Precompile `transfer` and `burn` Allow Any Contract to Steal or Destroy Any User's Native Tokens Without Authorization — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` and `burn` methods accept an arbitrary `sender`/`recipient` address as a caller-supplied argument and operate directly on that address's native token balance with no authorization check. Any EVM contract can call these methods to drain or destroy another user's `evm/0x{contractAddress}` native tokens without the victim's consent.

---

### Finding Description

`BankContract.Run` in `x/cronos/keeper/precompiles/bank.go` handles two mutating methods:

**`burn` (lines 113–156):**
```go
recipient := args[0].(common.Address)   // address to burn FROM — caller-supplied
addr := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())    // evm/0x{callerContract}
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
```

**`transfer` (lines 167–200):**
```go
sender := args[0].(common.Address)      // address to transfer FROM — caller-supplied
recipient := args[1].(common.Address)
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())    // evm/0x{callerContract}
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

In both cases the only guard present is `checkBlockedAddr`, which only rejects module-account destinations. There is **no check** that `contract.Caller() == sender/recipient`, and **no allowance/approval mechanism** is consulted. Any contract at address `C` can therefore call `burn(victimAddr, amount)` or `transfer(victimAddr, attackerAddr, amount)` to destroy or steal any user's `evm/0xC` native tokens. ---

### Impact Explanation

**Critical — unauthorized burn and transfer of native assets.**

`evm/0x{contractAddress}` tokens are the Cosmos-bank-side representation of CRC20/CRC21 source tokens. Users accumulate them when:
- A CRC21 source token's `send_to_ibc` flow mints native tokens before the IBC send.
- `BurnVouchersToEvmCoins` mints them on inbound bridge transfers.
- Any explicit EVM→native conversion path deposits them to a user's bank account.

A malicious contract at address `C` can call `transfer(victimAddress, attackerAddress, balance)` on the bank precompile (`0x64`) to move the victim's entire `evm/0xC` balance to the attacker, or call `burn(victimAddress, balance)` to destroy it. This is an unauthorized balance change for precompile-controlled assets, matching the Critical impact tier. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged actor can deploy an EVM contract and immediately call the bank precompile. No admin key, governance vote, or special permission is required. The attacker only needs victims to hold `evm/0x{attackerContract}` tokens, which occurs naturally whenever the attacker's contract is a registered CRC21 source token used in IBC or bridge flows. [6](#0-5) 

---

### Recommendation

Add a caller-authorization check before executing `burn` and `transfer`. For `burn`, require `contract.Caller() == recipient` (a contract may only burn from itself) or require an explicit on-chain allowance. For `transfer`, require `contract.Caller() == sender` or that the caller holds a sufficient allowance granted by `sender`:

```go
// burn: only allow a contract to burn its own balance
if contract.Caller() != recipient {
    return nil, errors.New("burn: caller is not the token holder")
}

// transfer: only allow a contract to transfer from itself
if contract.Caller() != sender {
    return nil, errors.New("transfer: caller is not the sender")
}
```

Alternatively, implement an ERC-20-style allowance table in the precompile so that users can explicitly approve a contract to spend on their behalf. [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. Attacker deploys contract `Exploit` at address `0xEEEE…`.
2. Legitimate users hold `evm/0xEEEE…` native tokens (e.g., after an IBC inbound transfer that minted them via `BurnVouchersToEvmCoins`).
3. `Exploit` calls the bank precompile at `0x0000000000000000000000000000000000000064`:
   ```solidity
   IBank(0x64).transfer(victimAddress, attackerAddress, victimBalance);
   ```
4. The precompile resolves `denom = "evm/" + address(this)` = `evm/0xEEEE…`, then calls `bankKeeper.SendCoins(victimAddr, attackerAddr, coins)` with no authorization check.
5. The victim's entire `evm/0xEEEE…` balance is transferred to the attacker in a single transaction. [9](#0-8) [10](#0-9)
