### Title
Missing Minimum Threshold Enforcement in FROST Presign/Sign Allows Malicious Coordinator to Corrupt Signing Outputs — (`File: src/frost/mod.rs`)

---

### Summary

The FROST presign and sign validation functions (`presign` and `assert_sign_inputs` in `src/frost/mod.rs`) do not enforce a minimum threshold of 2, unlike the DKG layer (`assert_key_invariants` in `src/dkg.rs`) which explicitly rejects `threshold < 2`. A malicious coordinator can supply `threshold = 1` to a FROST signing session whose key was generated with `threshold ≥ 2`, causing all honest participants to compute signature shares under incorrect Lagrange coefficients. The combined output is an invalid, unusable ECDSA/EdDSA signature — a corruption of the sign output that honest parties cannot detect until after they have already committed their shares.

---

### Finding Description

**DKG layer — minimum threshold enforced:**

`assert_key_invariants` in `src/dkg.rs` (lines 580–582) explicitly rejects any threshold below 2:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The same guard appears in `validate_triple_inputs` for OT-based triple generation: [2](#0-1) 

**FROST layer — minimum threshold NOT enforced:**

`presign` in `src/frost/mod.rs` (lines 55–77) only checks that `participants.len() ≥ 2` and that `threshold ≤ participants.len()`. There is no lower-bound check on `threshold`:

```rust
if participants.len() < 2 {
    return Err(InitializationError::NotEnoughParticipants { ... });
}
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: threshold >= 2 is MISSING
``` [3](#0-2) 

`assert_sign_inputs` in `src/frost/mod.rs` (lines 127–150) has the identical gap — it checks `threshold > participants.len()` but never `threshold < 2`:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: threshold >= 2 is MISSING
``` [4](#0-3)

### Impact Explanation

When a key is generated via `keygen` (threshold ≥ 2 enforced), each participant's signing share is a degree-(threshold−1) polynomial evaluation. During FROST signing, each participant computes their signature share as:

```
s_i = nonce_i + λ_i(S) · x_i · c
```

where `λ_i(S)` is the Lagrange coefficient over the signing set `S` at reconstruction threshold `t`. If a malicious coordinator supplies `threshold = 1`, the Lagrange coefficients collapse to `λ_i = 1` for every participant (trivial single-point reconstruction). The resulting sum `Σ s_i` does not equal the correct FROST aggregate because `Σ x_i ≠ x` (the master secret) under a degree-1 sharing. The final signature fails verification — an unusable cryptographic output — matching the allowed High impact:

> **High: Corruption of … sign … outputs so honest parties accept … unusable cryptographic outputs.**

---

### Likelihood Explanation

The coordinator role is explicitly modeled as potentially malicious throughout the codebase (see `docs/ecdsa/robust_ecdsa/signing.md` split-view attack mitigations). A malicious coordinator is a first-class attacker profile. The attack requires only calling the public `presign`/`sign` API with `threshold = 1`; no privileged access, leaked keys, or cryptographic breaks are needed. The missing check is a single missing `if threshold < 2 { return Err(...) }` guard, making the entry path trivially reachable. [6](#0-5) 

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use, to both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This makes the FROST signing layer consistent with the DKG layer's invariant that `threshold ≥ 2` is always required. [1](#0-0) [7](#0-6) 

---

### Proof of Concept

1. Run DKG for 3 participants with `threshold = 2` (accepted by `assert_key_invariants`).
2. Malicious coordinator initiates FROST presign by constructing `PresignArguments { threshold: 1.into(), ... }`. The `presign` function in `src/frost/mod.rs` accepts this because `1 ≤ 3` and there is no `threshold < 2` check.
3. Coordinator initiates FROST sign, calling `assert_sign_inputs` with `threshold = 1`. Again accepted.
4. Each honest participant computes `s_i = nonce_i + 1 · x_i · c` (Lagrange coefficient = 1 for threshold-1 reconstruction).
5. Coordinator aggregates: `S = Σ s_i = Σ nonce_i + (x_1 + x_2 + x_3) · c`. Since `x_1 + x_2 + x_3 ≠ x` (the master secret) under a degree-1 Shamir sharing, the resulting signature `(R, S)` fails standard ECDSA/EdDSA verification.
6. All honest parties have committed their shares and received an unusable, invalid signature — the sign output is corrupted. [8](#0-7) [9](#0-8) [10](#0-9)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
