### Title
`skip_stateful_validations` Bypasses Signature Verification for Any Invoke with Nonce=1 When Target Account Has Any Pending Mempool Transaction - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX feature for the `deploy_account + invoke` flow skips the `__validate__` entry-point call (the only place where the account's signature is cryptographically verified) for any invoke transaction with `nonce=1` submitted against an undeployed account, provided that account has **any** transaction in the mempool. The check does not verify that the existing mempool transaction is specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` can submit an invoke transaction for the victim's address with an arbitrary/invalid signature and have it admitted to the mempool without signature verification.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The third condition is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction from the account — not specifically a `deploy_account`. The comment in `skip_stateful_validations` claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but the second branch is circular: those future-nonce transactions would themselves have passed only because this same skip was applied.

When `skip_validate=true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

This means the `__validate__` entry point — the only place where the account contract cryptographically verifies the transaction signature — is never called during gateway admission. The stateless validator only checks signature **length** (≤ 4000 felts), not cryptographic validity. [4](#0-3) 

The default `max_allowed_nonce_gap` is 200, so `nonce=1` with `account_nonce=0` passes the nonce range check. [5](#0-4) 

**Attack path:**

1. Victim submits a `deploy_account` transaction (nonce=0); it enters the mempool.
2. Attacker derives the victim's contract address (deterministic from `class_hash`, `salt`, `constructor_calldata`).
3. Attacker crafts an invoke transaction: `sender_address=victim`, `nonce=1`, arbitrary calldata, arbitrary/invalid signature.
4. Gateway stateless check: signature length ≤ 4000 → passes.
5. Gateway stateful check: `account_nonce=0`, `tx.nonce()=1`, `account_tx_in_pool_or_recent_block(victim)=true` (victim's deploy_account is in pool) → `skip_validate=true` → `__validate__` is **not called** → transaction admitted.
6. Malicious invoke is now in the mempool occupying the victim's `nonce=1` slot.
7. Victim's legitimate invoke (also `nonce=1`) is rejected by the mempool as a duplicate nonce (unless fee escalation applies).
8. At block execution time, the batcher creates the transaction with `validate: true` (via `new_for_sequencing`), `__validate__` runs, the invalid signature is caught, and the transaction is rejected — but the victim's `deploy_account + invoke` UX flow has been disrupted. [6](#0-5) 

### Impact Explanation

The gateway admits an invoke transaction with an invalid (attacker-controlled) signature into the mempool, bypassing the account's `__validate__` entry point. This matches **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The concrete effect is that any account in the `deploy_account` pending state can have its `nonce=1` slot squatted by an attacker with zero knowledge of the account's private key. The victim's intended `deploy_account + invoke` atomic UX flow is broken: the legitimate invoke is either rejected from the mempool (duplicate nonce) or must win a fee-escalation race against the attacker.

### Likelihood Explanation

- **Trigger is unprivileged**: any network participant can submit an `RpcTransaction` to the gateway.
- **Precondition is observable**: the victim's `deploy_account` transaction is visible in the public mempool.
- **No cryptographic material required**: the attacker only needs the victim's contract address (derivable from the public `deploy_account` fields) and any non-empty signature bytes.
- **Window**: the attack is viable from the moment the victim's `deploy_account` enters the mempool until it is committed to a block.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** exists for the target address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when the pending transaction for that address is of type `DeployAccount`. Alternatively, require that the caller of the gateway is the account owner (i.e., enforce that `msg.sender` matches `sender_address` at the HTTP/RPC layer), analogous to the H-12 mitigation of checking `msg.sender == _loan.borrower`.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
