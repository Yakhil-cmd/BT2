### Title
Gateway Admits Invalid-Signature Invoke via Non-Deploy-Account `skip_stateful_validations` Trigger — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke with nonce 1 when the account's deploy-account transaction has not yet been processed. The guard it uses — `account_tx_in_pool_or_recent_block` — does **not** verify that the existing mempool entry is a `deploy_account` transaction. Any transaction from the sender address satisfies the check. An unprivileged attacker can therefore bootstrap the skip condition with a valid invoke, then replace it via fee escalation with an invoke carrying an invalid signature, causing the gateway to admit the invalid transaction to the mempool without ever calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0`.
3. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

Condition 3 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type from that address, not only `DeployAccount`: [3](#0-2) 

`MempoolState.contains_account` likewise checks `staged` or `committed` maps with no type filter: [4](#0-3) 

The code comment acknowledges the loose check: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."* [5](#0-4) 

The second branch ("future nonces that passed validations") is the exploitable path. Once an attacker's account appears in the pool via any transaction, the skip condition is permanently satisfiable for any subsequent nonce-1 invoke from that account while the account nonce remains 0.

The gateway's `validate_by_mempool` call (which runs before `skip_stateful_validations`) only checks for duplicate tx-hash and nonce ordering — it does not verify the signature: [6](#0-5) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so `StatefulValidator::validate` returns immediately without calling the account's `__validate__` entry point: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

The gateway/mempool admission path accepts an invoke transaction whose signature has never been verified by the account's `__validate__` entry point. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- An attacker can displace a valid nonce-1 invoke from the mempool with an invalid-signature invoke via fee escalation, causing the victim's transaction to be evicted.
- The invalid transaction occupies a mempool slot and will consume block space before failing during batcher execution.
- Repeated exploitation can be used for targeted DoS against specific accounts whose nonce is 0.

---

### Likelihood Explanation

The conditions are straightforward to satisfy for any attacker who controls an account with on-chain nonce 0 (a freshly deployed account or any account that has never sent a transaction). The attacker needs only two sequential gateway submissions:

1. A valid invoke (nonce 1) to seed the pool.
2. An invalid-signature invoke (nonce 1, higher tip) to trigger fee escalation while skipping `__validate__`.

No privileged access, no special contract, and no race condition is required.

---

### Recommendation

Replace the loose `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a `DeployAccount` transaction for the sender address is present in the pool or was recently committed. Alternatively, add a dedicated `deploy_account_in_pool_or_recent_block(address)` API to the mempool that filters by transaction type before returning `true`.

---

### Proof of Concept

```
Precondition: Account A is deployed on-chain, nonce = 0.

Step 1 — Seed the pool:
  Submit Invoke(sender=A, nonce=1, calldata=X, signature=VALID, tip=100)
  → Gateway: account_nonce=0, tx_nonce=1 (within gap), __validate__ called → OK
  → Mempool: tx admitted, tx_pool.contains_account(A) = true

Step 2 — Exploit skip_validate:
  Submit Invoke(sender=A, nonce=1, calldata=Y, signature=INVALID, tip=200)
  → Gateway stateful validator:
      account_nonce = 0, tx_nonce = 1
      validate_by_mempool: no duplicate hash, nonce 1 >= 0 → OK
      skip_stateful_validations:
          nonce==1 ✓, account_nonce==0 ✓
          account_tx_in_pool_or_recent_block(A) = true  ← Step 1 tx satisfies this
          → returns true (skip __validate__)
      run_validate_entry_point: validate=false → __validate__ NOT called
  → Fee escalation (tip 200 > 100): Step 1 tx evicted, Step 2 tx admitted

Result: Mempool contains Invoke(A, nonce=1, INVALID_SIGNATURE).
        The account's __validate__ was never called for this transaction.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
