### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions into Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path unconditionally bypasses the account's `__validate__` entry point — the only cryptographic signature check — for any invoke transaction with `nonce == 1` when the account has *any* transaction in the mempool or a recent block. An unprivileged attacker who observes a pending deploy-account (or any nonce-0 invoke) for a target address can immediately submit an invoke with nonce=1 carrying an arbitrary/fake signature, and the gateway will accept it into the mempool without ever verifying the signature.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks`, which is called by `extract_state_nonce_and_run_validations`: [1](#0-0) 

The function returns `true` (skip) when all three conditions hold:

1. The transaction is `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`. [2](#0-1) 

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, this causes an early return before the `__validate__` call: [4](#0-3) 

The stateless validator never checks the cryptographic signature — it only checks structural properties (size, format, DA modes, resource bounds): [5](#0-4) 

The `validate_by_mempool` call checks nonce ordering and mempool-level constraints, not the cryptographic signature: [6](#0-5) 

**Broken invariant:** Every transaction admitted to the mempool must have passed account signature verification (`__validate__`). This invariant is violated for invoke transactions with nonce=1 whenever the account has any pending transaction.

**Attacker-controlled field:** The `signature` field of `RpcInvokeTransactionV3` is accepted verbatim and never verified when the skip condition fires. [7](#0-6) 

**Condition 4 is trivially satisfiable:** The mempool is public. Any observer can see a pending deploy-account or nonce-0 invoke for a target address. The check is not restricted to deploy-account transactions — *any* transaction from the sender suffices: [8](#0-7) 

### Impact Explanation

An attacker can inject invoke transactions with arbitrary/forged signatures into the mempool for any account that has a pending transaction. These transactions pass all gateway checks (stateless + nonce + resource bounds + mempool validation) and are forwarded to the mempool. During block execution the blockifier will run `__validate__`, the signature will fail, and the transaction will revert — but it has already been admitted and occupies mempool and block space. Because the account is undeployed (nonce=0), no fee can be collected for the failed execution, making this a zero-cost DoS vector against the mempool and block-building pipeline.

**Matching impact:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

The attack requires no privileges. The mempool is publicly observable. The attacker only needs to:
- Watch for any transaction from a target address appearing in the mempool.
- Immediately submit an invoke with `nonce=1` and an all-zero (or random) signature for that address.

The exploit window is the time between the target's transaction entering the mempool and being included in a block. For deploy-account + invoke UX flows this window is intentionally kept open.

### Recommendation

1. **Restrict the skip condition to deploy-account specifically.** The `account_tx_in_pool_or_recent_block` check should be replaced with a check that a *deploy-account* transaction for the sender exists in the pool, not just any transaction.

2. **Add a lightweight off-chain signature pre-check.** Even when skipping the on-chain `__validate__` entry point, the gateway should perform a basic ECDSA/Stark-curve signature check against the sender address before admitting the transaction to the mempool.

3. **Scope the skip to the exact nonce.** The skip should only apply when `tx.nonce == account_nonce + 1` and a deploy-account for the same address is confirmed to be in the pool, not for any future nonce.

### Proof of Concept

```
1. Alice submits RpcDeployAccountTransaction for address A (valid signature).
   → deploy_account enters the mempool; account_nonce on-chain = 0.

2. Attacker observes the mempool and sees Alice's deploy_account for address A.

3. Attacker constructs RpcInvokeTransactionV3:
     sender_address = A
     nonce          = 1
     signature      = [0x0, 0x0]   ← arbitrary/fake
     resource_bounds = <any valid non-zero bounds>
     calldata        = <any>

4. Attacker submits the invoke to the gateway.

5. Gateway stateless validation passes (no signature check).

6. convert_rpc_tx_to_internal computes tx_hash from body (hash is correct for the body).

7. extract_state_nonce_and_run_validations:
     account_nonce = get_nonce_from_state(A) = 0
     validate_state_preconditions: nonce 1 ∈ [0, 0+max_gap] → OK
     validate_by_mempool: nonce ordering OK
     skip_stateful_validations:
       tx.nonce() == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(A) == true ✓  ← Alice's deploy_account
       → returns true (SKIP)

8. run_validate_entry_point(skip_validate=true):
     execution_flags.validate = false
     StatefulValidator::perform_validations returns Ok(()) without calling __validate__

9. Fake invoke is forwarded to the mempool and accepted.

10. During block execution, blockifier runs __validate__ → signature fails → tx reverts.
    No fee is collected (account undeployed, no balance).
``` [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
