### Title
No Slippage Protection in Delegation Functions Allows Staker to Sandwich Commission Increase Against Delegators - (File: src/pool/pool.cairo, src/staking/staking.cairo)

### Summary
The `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool` functions in `pool.cairo` accept no `max_commission` parameter. A staker who has set up a `CommissionCommitment` with a high `max_commission` can increase their commission to any value up to `max_commission` at any time via `set_commission`. Because the commission change takes effect immediately and there is no slippage guard in the delegation entry points, a staker can front-run a delegator's delegation transaction by raising commission to `max_commission` in the same block, causing the delegator to delegate under terms far worse than expected. The protocol's own code comment at line 745–746 of `staking.cairo` acknowledges this: *"Current commission increase safeguards still allow for sudden commission changes."*

### Finding Description

**Root cause — `update_commission` allows immediate commission increases under an active commitment:** [1](#0-0) 

When a `CommissionCommitment` is active (`current_epoch < expiration_epoch`), the only constraint on `set_commission` is `commission <= max_commission` and `commission != old_commission`. There is no lower-bound check, so the staker can raise commission from 1% to 100% in a single transaction.

**`set_commission_commitment` allows setting `max_commission` up to 10 000 (100%):** [2](#0-1) 

The only constraint is `current_commission <= max_commission`, so a staker can legitimately set `max_commission = COMMISSION_DENOMINATOR` (10 000 = 100%) while advertising a low current commission.

**`enter_delegation_pool` and `add_to_delegation_pool` accept no `max_commission` guard:** [3](#0-2) [4](#0-3) 

Neither function checks or accepts a caller-supplied maximum acceptable commission. The delegator has no on-chain protection against the commission changing between the moment they decide to delegate and the moment their transaction is included.

**`switch_delegation_pool` has the same gap:** [5](#0-4) 

**Commission is read at reward-calculation time, so all existing delegators are also affected:** [6](#0-5) 

The live commission value is used every time `_update_rewards` runs, meaning a post-delegation commission increase harms all current pool members, not only new entrants.

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

If a staker raises commission to 100% (`COMMISSION_DENOMINATOR = 10 000`), the `split_rewards_with_commission` call returns `pool_rewards = 0` for every delegator. All yield that would have accrued to delegators is instead credited to the staker as commission rewards. Delegators must wait for the full `exit_wait_window` (default one week, up to twelve weeks) before they can recover their principal, during which every reward epoch produces zero delegator yield.

### Likelihood Explanation

**Likelihood: Medium.**

Prerequisites that must hold:
1. The staker must have previously called `set_commission_commitment` with a high `max_commission` (publicly visible on-chain via the `CommissionCommitmentSet` event).
2. The staker must be willing to act maliciously.

Starknet's sequencer is currently centralised, so classic mempool front-running is not available to arbitrary actors. However, the staker does not need to observe a pending transaction: they can simply call `set_commission` to the maximum value at any time, and any delegator whose transaction lands in the same or a later block will delegate under the new rate. Because the commission change is immediate and affects all existing delegators simultaneously, the attack surface is broad. The acknowledged code comment confirms the developers are aware the safeguard is incomplete.

### Recommendation

1. **Add a `max_commission` parameter to `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool`.** Revert if the pool's current commission exceeds the caller-supplied bound at execution time.
2. **Enforce a time-delay on commission increases.** Require that any increase under a commitment takes effect only at `current_epoch + K` (analogous to how stake changes are deferred), giving delegators time to exit before the new rate applies.
3. **Emit a warning event and enforce a notice period** before a commission increase becomes effective, so off-chain tooling can alert delegators.

### Proof of Concept

```
// Step 1 – Staker sets up a commitment allowing 100% commission.
staking.set_commission_commitment(
    max_commission: 10_000,          // 100%
    expiration_epoch: current_epoch + 100,
);

// Step 2 – Staker advertises low commission to attract delegators.
staking.set_commission(commission: 100);  // 1%

// Step 3 – Delegator sees 1% commission and submits:
pool.enter_delegation_pool(reward_address: delegator_reward, amount: 1_000_000_STRK);

// Step 4 – In the same block, BEFORE the delegator's tx is sequenced,
//           the staker submits:
staking.set_commission(commission: 10_000);  // 100% — allowed by active commitment

// Step 5 – Delegator's tx executes. No max_commission check exists.
//           The delegator is now in a pool with 100% commission.

// Step 6 – Every subsequent call to _update_rewards reads commission = 10_000.
//           split_rewards_with_commission returns pool_rewards = 0.
//           The delegator earns zero yield until they exit after the wait window.
```

The staker captures 100% of the yield that would have been distributed to the delegator, constituting a direct theft of unclaimed yield with no recourse for the delegator during the exit window.

            // Asserts.
            self.assert_staker_is_active();
            let pool_member = get_caller_address();
            assert!(
                self.pool_member_info.read(pool_member).is_none(), "{}", Error::POOL_MEMBER_EXISTS,
            );
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);

            transfer_from_delegator(:pool_member, :amount, :token_dispatcher);

                .pool_member_info

            self

        fn add_to_delegation_pool(
            ref self: ContractState, pool_member: ContractAddress, amount: Amount,
        ) -> Amount {
            // Asserts.
            self.assert_staker_is_active();
            let pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            assert!(
                caller_address == pool_member || caller_address == pool_member_info.reward_address,
                "{}",
                Error::CALLER_CANNOT_ADD_TO_POOL,
            );
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);

            // Transfer funds from the delegator to the staking contract.
            let token_dispatcher = self.token_dispatcher.read();
            let staker_address = self.staker_address.read();
            transfer_from_delegator(pool_member: caller_address, :amount, :token_dispatcher);
            self.transfer_to_staking_contract(:amount, :token_dispatcher, :staker_address);

            // Update the pool member's balance checkpoint.
            let old_delegated_stake = self.increase_member_balance(:pool_member, :amount);
            let new_delegated_stake = old_delegated_stake + amount;

            // Emit events.
            self
                .emit(
                    Events::PoolMemberBalanceChanged {
                        pool_member, old_delegated_stake, new_delegated_stake,
                    },
                );

            new_delegated_stake
        }

        fn switch_delegation_pool(
            ref self: ContractState,
            to_staker: ContractAddress,
            to_pool: ContractAddress,
            amount: Amount,
        ) -> Amount {
            // Asserts.
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            assert!(
                pool_member_info.unpool_time.is_some(),
                "{}",
                GenericError::MISSING_UNDELEGATE_INTENT,
            );
            assert!(amount <= pool_member_info.unpool_amount, "{}", GenericError::AMOUNT_TOO_HIGH);
            let reward_address = pool_member_info.reward_address;

            // Update pool_member_info and write to storage.
            pool_member_info.unpool_amount -= amount;
            if pool_member_info.unpool_amount.is_zero() {
                // unpool_amount is zero, clear unpool_time.
                pool_member_info.unpool_time = Option::None;
            }
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Serialize the switch pool data and invoke the staking contract to switch pool.
            let switch_pool_data = SwitchPoolData { pool_member, reward_address };
            let mut serialized_data = array![];
            switch_pool_data.serialize(ref output: serialized_data);
            self
                .staking_pool_dispatcher
                .read()
                .switch_staking_delegation_pool(
                    :to_staker,
                    :to_pool,
                    switched_amount: amount,
                    data: serialized_data.span(),
                    identifier: pool_member.into(),
                );

            // Emit event.
            self
                .emit(
                    Events::SwitchDelegationPool {
                        pool_member, new_delegation_pool: to_pool, amount,
                    },
                );

            pool_member_info.unpool_amount
        }
