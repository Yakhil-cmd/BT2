### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required — (`File: L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary

`RewardSupplier.tick()` is declared `payable` and uses `msg.value` to pay bridge fees, but only when `amountToMint > 0`. When there are no pending L2 mint requests (`amountToMint == 0`), the function returns immediately without consuming or refunding `msg.value`. Any ETH sent by the caller in that branch is permanently locked in the contract with no recovery path.

### Finding Description

`tick()` in `L1/starkware/solidity/stake/RewardSupplier.sol` is a public, payable function. Its body is:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... consumes msg.value for bridge fees ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // ← if amountToMint == 0, execution falls through here.
    //   msg.value is never used and never refunded.
}
``` [1](#0-0) 

When `requiredMinting()` returns `(0, 0)` — which happens whenever there are no unconsumed L2→L1 mint-request messages — the `if` block is skipped entirely. Any ETH attached to the call is silently absorbed by the contract. The contract has no `receive()` fallback that tracks this ETH, no withdrawal function, and no mechanism to apply accumulated ETH to a future `tick()` call (each call only uses its own `msg.value`). [2](#0-1) 

The root cause is structurally identical to the reference bug: a variable (`amountToMint`) is effectively treated as zero, bypassing the branch that would consume the caller-supplied value, while the function signature still accepts that value without reverting.

### Impact Explanation

Any ETH sent to `tick()` when no minting is pending is permanently frozen inside the `RewardSupplier` L1 contract. There is no sweep, no refund, and no future reuse of that balance. This constitutes a permanent loss of the caller's native coin — matching the **Medium: Griefing with no profit motive but damage to users or protocol** impact tier (and potentially **High: Temporary/permanent freezing of funds** depending on the amount).

### Likelihood Explanation

`tick()` carries no access control; any public L1 caller can invoke it. Keepers or integrators who call `tick()` speculatively (e.g., in a cron job) while attaching ETH for bridge fees will lose that ETH whenever the L2 mint-request queue happens to be empty. This is a realistic operational scenario, not a contrived one.

### Recommendation

Add a guard at the top of `tick()` that refunds `msg.value` (or reverts) when `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        // Nothing to do; refund any accidentally sent ETH.
        if (msg.value > 0) {
            (bool ok, ) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
        return;
    }
    // ... rest of minting logic ...
}
```

Alternatively, remove `payable` from the signature and require callers to send exactly the required fee only when minting is needed.

### Proof of Concept

1. Deploy or interact with the live `RewardSupplier` L1 contract.
2. Confirm `requiredMinting()` returns `(0, 0)` (no pending L2 mint messages).
3. Call `tick{value: 1 ether}()` from any EOA.
4. Observe: the call succeeds (no revert), `msg.value` is not forwarded to the bridge or messaging contract, and the contract's ETH balance increases by 1 ETH with no way to recover it. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
