### Title
Missing `app_pk` Identity-Point Validation Strips ElGamal Blinding, Exposing Confidential Derived Key On-Chain — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

`ckd()` accepts `app_pk: PublicKey` (a `G1Projective`) without checking whether it is the group identity element. When `app_pk = G1Projective::identity()`, the per-node blinding term `y_i · A` collapses to the identity, so every participant's `C_i` equals their raw key-share contribution `S_i`. The coordinator aggregates these into `C = msk · H(pk ‖ app_id)` — the unblinded BLS signature — which is then published on-chain as part of `es = (Y, C)`, making the confidential derived key visible to any observer.

---

### Finding Description

In `compute_signature_share` the blinding step is:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;   // line 174
``` [1](#0-0) 

When `app_pk` is the identity element, `app_pk * y.0 = identity` for every scalar `y.0`, so:

```
C_i = S_i + identity = S_i = x_i · H(pk ‖ app_id)
```

After Lagrange-normalisation and coordinator aggregation:

```
C = Σ λ_i · C_i = Σ λ_i · x_i · H(pk ‖ app_id) = msk · H(pk ‖ app_id)
```

`Y = Σ λ_i · y_i · G` remains a random non-identity point, so the output `(Y, C)` looks structurally valid. The coordinator produces a `CKDOutput` where `big_c` is already the raw BLS signature. [2](#0-1) 

`ckd()` performs no validation on `app_pk` — only participant-list checks are present: [3](#0-2) 

The `CKDOutput` is sent on-chain as `es = (Y, C)` per the protocol design: [4](#0-3) 

---

### Impact Explanation

The confidentiality invariant requires that `C = msk · H(pk ‖ app_id) + a · Y`, so only the holder of `app_sk = a` can strip the blinding and recover the BLS signature. With `app_pk = identity`, `a = 0`, the blinding term vanishes, and `C` **is** the BLS signature. Because `es` is published on-chain, every observer — not just the coordinator — can read `C` and derive `s = HKDF(C)`. This is a **Critical** disclosure of the confidential derived key, directly violating the stated requirement:

> "No single node in the MPC network should be capable of computing s." [5](#0-4) 

---

### Likelihood Explanation

The developer contract is responsible for verifying TEE attestation and that `A` appears in `report_data`, but neither the library nor any documented caller contract enforces `A ≠ identity`. A malicious or buggy TEE app can submit `app_pk = G1Projective::identity()` on-chain; the MPC nodes will faithfully call `ckd()` with that value and produce the leaking output. No cryptographic assumption needs to be broken; it is a pure input-validation gap in the public API.

---

### Recommendation

Add an explicit identity-point check inside `ckd()` before the protocol runs:

```rust
use elliptic_curve::Group;

if app_pk.is_identity().into() {
    return Err(InitializationError::InvalidInput("app_pk must not be the identity element"));
}
```

Place it alongside the existing participant-list guards in `ckd()`. [3](#0-2)
