### Title
Missing Silo Whitelist Check in `receive_base_tokens` Allows ETH to Be Permanently Frozen - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode with the `Address` whitelist enabled, `receive_erc20_tokens` correctly redirects incoming ERC-20 tokens to a configured fallback address when the recipient is not whitelisted. However, the analogous function `receive_base_tokens` — which mints ETH (the base token) during `ft_on_transfer` from the ETH connector — performs no such whitelist check. Any ETH minted to a non-whitelisted address is permanently inaccessible because the Silo whitelist blocks all transaction submission from that address.

---

### Finding Description

The `ft_on_transfer` entry point in `engine/src/contract_methods/connector.rs` dispatches to one of two paths depending on the caller: [1](#0-0) 

When the ETH connector calls `ft_on_transfer`, execution goes to `receive_base_tokens`. This function parses the recipient address from the message and unconditionally mints ETH to it: [2](#0-1) 

There is no call to `silo::is_allow_receive_erc20_tokens` or any equivalent whitelist guard. Compare this to `receive_erc20_tokens`, which explicitly checks the silo whitelist and redirects to a fallback address when the recipient is not permitted: [3](#0-2) 

The fallback mechanism is named `erc20_fallback_address` and is stored under `ERC20_FALLBACK_KEY`. There is no equivalent `base_token_fallback_address` concept in the silo parameters: [4](#0-3) 

Once ETH is minted to a non-whitelisted address, the Silo `assert_access` check in `submit_with_alt_modexp` blocks every transaction from that address: [5](#0-4) 

`is_allow_submit` requires the sender's EVM address to be present in the `Address` whitelist: [6](#0-5) 

Because the address cannot submit any transaction, it cannot transfer, withdraw, or otherwise move the minted ETH. There is no admin-callable rescue function that moves balances on behalf of a frozen address.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH bridged into a Silo Aurora instance and minted to a non-whitelisted EVM address is irrecoverable by the recipient. The balance is recorded in engine state but the address is permanently barred from submitting the transaction needed to move it. Unlike ERC-20 tokens (which are redirected to the fallback address), there is no fallback mechanism for base-token deposits.

---

### Likelihood Explanation

**Medium.** The precondition is that the Aurora instance is running in Silo mode with the `Address` whitelist enabled — a deliberate operator configuration. Within that configuration, any NEAR user who calls `ft_transfer_call` on the ETH connector and specifies a non-whitelisted EVM address in the `msg` field triggers the freeze. No special privilege is required; the call is open to any NEAR account holder. The scenario is realistic whenever a user makes a bridging mistake (e.g., specifying a fresh address not yet added to the whitelist) or when an attacker deliberately targets another user's address.

---

### Recommendation

Apply the same silo whitelist guard to `receive_base_tokens` that already exists in `receive_erc20_tokens`. Introduce a `base_token_fallback_address` (or reuse the existing `erc20_fallback_address`) and, when the recipient is not whitelisted, redirect the ETH mint to that fallback address instead of proceeding unconditionally. The check should mirror the pattern at lines 818–822 of `engine/src/engine.rs`. [3](#0-2) 

---

### Proof of Concept

**Setup**: Deploy Aurora in Silo mode. Enable the `Address` whitelist. Do **not** add address `0xDEAD...` to the whitelist.

**Attack steps**:

1. A NEAR user calls `ft_transfer_call` on the ETH connector contract:
   ```
   receiver_id: aurora.near
   amount: 1_000_000_000_000_000_000   # 1 ETH in yoctoNEAR-equivalent
   msg: "DEAD000000000000000000000000000000000000"  # non-whitelisted EVM address
   ```
2. The ETH connector calls `ft_on_transfer` on the Aurora engine.
3. `predecessor_account_id == get_connector_account_id` → `receive_base_tokens` is invoked.
4. `receive_base_tokens` parses `0xDEAD...` as the recipient and calls `set_balance`, crediting 1 ETH to that address. [7](#0-6) 
5. Any subsequent attempt by `0xDEAD...` to submit a transaction is rejected by `assert_access` with `EngineErrorKind::NotAllowed` because the address is absent from the `Address` whitelist. [8](#0-7) 
6. The 1 ETH is permanently frozen. No admin function exists to rescue it.

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
