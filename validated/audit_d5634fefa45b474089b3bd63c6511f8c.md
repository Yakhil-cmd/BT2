### Title
Burned ERC-20 Mirror Tokens Are Permanently Lost on Failed NEP-141 Exit When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

When the `error_refund` Cargo feature is not compiled into the Aurora Engine binary (which is the case for the production `contract` feature set), a user who calls `withdrawToNear` or `withdrawToEthereum` on an `EvmErc20`/`EvmErc20V2` mirror token and whose subsequent NEP-141 `ft_transfer` fails will have their ERC-20 tokens permanently burned with no re-mint. The corresponding NEP-141 tokens remain locked inside the Aurora contract with no recovery path. This permanently destroys the user's bridged assets.

---

### Finding Description

The `EvmErc20` and `EvmErc20V2` contracts implement the bridge exit flow as follows:

1. `_burn` is called on the caller's ERC-20 balance, reducing `totalSupply`.
2. The exit precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is called via inline assembly to schedule a NEAR `ft_transfer` to the NEP-141 contract. [1](#0-0) [2](#0-1) 

The exit precompile (`ExitToNear::run`) constructs a `ExitToNearPrecompileCallbackArgs` struct. The `refund` field — which carries the information needed to re-mint burned tokens if the `ft_transfer` fails — is **conditionally compiled**:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // <-- always None without the feature
    transfer_near: transfer_near_args,
};
``` [3](#0-2) 

When the NEP-141 `ft_transfer` fails, `exit_to_near_precompile_callback` is invoked. It only calls `refund_on_error` (which re-mints the burned ERC-20 tokens) when `args.refund` is `Some`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- no refund, tokens permanently lost
};
``` [4](#0-3) 

The `error_refund` feature is **not** included in the `default` or `contract` feature sets:

```toml
# engine/Cargo.toml
[features]
default = ["std"]
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract"]
error_refund = ["aurora-engine-precompiles/error_refund"]   # optional, not in contract
``` [5](#0-4) 

The production WASM is built with the `contract` feature. Since `contract` does not pull in `error_refund`, the production binary always sets `refund: None`. The test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

The `refund_on_error` function, which would re-mint the burned tokens, is therefore never reachable in production for the ERC-20 exit path: [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls `withdrawToNear` or `withdrawToEthereum` and the downstream NEP-141 `ft_transfer` fails:

- The ERC-20 mirror tokens are permanently burned (subtracted from `totalSupply` with no re-mint).
- The NEP-141 tokens remain locked inside the Aurora contract with no mechanism to release them.
- The user loses both their ERC-20 representation and their ability to recover the underlying NEP-141 tokens.

This breaks the core bridge invariant: `ERC-20 totalSupply == NEP-141 tokens locked in Aurora`.

---

### Likelihood Explanation

**High.** The `ft_transfer` call fails whenever the recipient NEAR account is not registered with the NEP-141 token contract. This is a common, user-reachable condition — any user who mistypes a NEAR account ID, uses an unregistered account, or whose recipient account has been deleted will trigger this path. No special privileges are required; any ERC-20 token holder can call `withdrawToNear`. [1](#0-0) 

---

### Recommendation

Enable the `error_refund` feature in the production `contract` feature set in `engine/Cargo.toml`:

```toml
contract = ["log", "aurora-engine-sdk/contract", "aurora-engine-precompiles/contract", "error_refund"]
```

This ensures that `refund_call_args` is always populated in `ExitToNearPrecompileCallbackArgs`, and that `exit_to_near_precompile_callback` will always re-mint burned ERC-20 tokens when the NEP-141 transfer fails.

Note: `EvmErc20V2.sol` already encodes the `sender` address as the refund recipient in its calldata (compatible with `error_refund`), but `EvmErc20.sol` (V1) does not include a refund address, so V1 contracts would also need to be migrated to the V2 format. [2](#0-1) 

---

### Proof of Concept

1. User holds 100 units of a bridged NEP-141 token as ERC-20 on Aurora.
2. User calls `EvmErc20.withdrawToNear("unregistered.near", 100)`.
3. `_burn(msg.sender, 100)` executes — ERC-20 `totalSupply` decreases by 100, user balance goes to 0.
4. Exit precompile schedules `ft_transfer` to the NEP-141 contract for `unregistered.near`.
5. `ft_transfer` fails (account not registered).
6. `exit_to_near_precompile_callback` fires; `args.refund` is `None` (production binary, `error_refund` not compiled in).
7. No `refund_on_error` call is made; no re-mint occurs.
8. User has 0 ERC-20 tokens. Aurora contract still holds 100 NEP-141 tokens. Funds are permanently frozen.

The test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` (lines 623–666) directly demonstrates this outcome under the `#[cfg(not(feature = "error_refund"))]` branch. [8](#0-7)
