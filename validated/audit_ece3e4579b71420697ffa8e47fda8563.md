### Title
Incorrect Royalty Math in Multi-NFT WalletConnect Offer Confirmation Divides Total by NFT Count — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each NFT's royalty, causing the displayed "Total Amount with Royalties" in the WalletConnect `take_offer` confirmation dialog to be systematically understated when an offer contains more than one NFT with royalties.

---

### Finding Description

The function `formatAmountWithRoyalties` computes the royalty-inclusive total shown to the user in the WalletConnect approval dialog: [1](#0-0) 

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

The intent is to sum each NFT's royalty over the full fungible amount. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The actual formula computed is:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
             = (amount / N) × Σ royaltyPercentage_i / 10_000
```

This is exactly **N times smaller** than the correct value (where N = number of NFTs in the offer). The `splitAmount` division has no basis in the Chia royalty protocol; each NFT's royalty is independently applied to the full fungible amount.

**Concrete example** (from the existing test at line 251):

| | Displayed (buggy) | Correct |
|---|---|---|
| amount | 100,000,000 mojos | 100,000,000 mojos |
| NFT1 royalty (5%) | 2,500,000 mojos | 5,000,000 mojos |
| NFT2 royalty (0.1%) | 5,000 mojos | 100,000 mojos |
| Total with royalties | **102,505,000 mojos (0.00010255 XCH)** | **105,100,000 mojos (0.0001051 XCH)** |

The test at line 320 asserts `amountWithRoyalties: '0.00010255'`, which encodes the wrong value as the expected result. [2](#0-1) 

The computed `amountWithRoyalties` string is then rendered directly in the WalletConnect confirmation dialog: [3](#0-2) 

---

### Impact Explanation

When a WalletConnect dApp sends a `take_offer` command containing multiple NFTs with royalties, the confirmation dialog shows the user a "Total Amount with Royalties" that is **N times lower than what the protocol will actually deduct**. The user reads the understated total, approves the transaction, and the wallet signs and broadcasts it — paying the correct (higher) royalty amount enforced by the Chia smart contract, not the amount shown on screen.

This is a corruption of WalletConnect offer state that causes a user to approve a spend displaying the wrong amount, matching the High impact criterion: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."*

---

### Likelihood Explanation

Any WalletConnect-connected dApp can send a `take_offer` payload containing a multi-NFT offer. No special privilege is required. The attacker constructs or selects an offer with two or more NFTs carrying non-trivial royalty percentages. The larger N and the higher the royalty percentages, the greater the gap between the displayed and actual cost. The user has no other signal in the dialog to detect the discrepancy. [4](#0-3) 

---

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be applied to the full `amount`:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Each NFT's royalty is independently applied to the full amount.
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

Update the corresponding test assertion at line 320 from `'0.00010255'` to `'0.0001051'`. [5](#0-4) 

---

### Proof of Concept

1. A dApp paired via WalletConnect sends:
   ```json
   { "command": "chia_wallet.take_offer", "data": { "offer": "<offer-with-2-NFTs>" } }
   ```
   where the offer contains two NFTs with royalty percentages 500 (5 %) and 10 (0.1 %), and requests 100,000,000 mojos of XCH.

2. `parseCommandDisplay` calls `formatAmountWithRoyalties` with `amount = 100_000_000n` and `royaltyPercentages = [500, 10]`.

3. The dialog renders: **"Total Amount with Royalties: 0.00010255 XCH"**

4. The user approves. The wallet signs and broadcasts the transaction. The Chia protocol deducts the correct royalties: **0.0001051 XCH** — approximately 2.5× the displayed royalty cost. [6](#0-5) [7](#0-6)

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
