### Title
Unprivileged Attacker Can Overwrite Any Account's E2EE Encryption Key — (`x/e2ee/keeper/keeper.go`, `x/e2ee/types/msg.go`)

### Summary

`MsgRegisterEncryptionKey` contains an `Address` field that is fully attacker-controlled. Neither `ValidateBasic` nor `Keeper.RegisterEncryptionKey` ever checks that the transaction signer matches `req.Address`. Any unprivileged account can permanently overwrite any victim's registered encryption key.

### Finding Description

`MsgRegisterEncryptionKey` carries two fields — `Address` and `Key` — with no signer field: [1](#0-0) 

`ValidateBasic` only validates that `Address` is a valid bech32 string and `Key` is a valid age X25519 recipient. It performs **no signer-vs-address check**: [2](#0-1) 

`Keeper.RegisterEncryptionKey` directly passes `req.Address` to the store write with no authorization check: [3](#0-2) 

`registerEncryptionKey` then unconditionally writes the attacker-supplied key under the victim's KV prefix: [4](#0-3) 

The Cosmos SDK ante handler only verifies that the tx signature is valid for the **signer's own address** (the attacker's address A). It does not enforce that the signer matches any field inside the message body. Since the keeper never retrieves the signer from context and compares it to `req.Address`, the authorization gap is complete.

### Impact Explanation

An attacker submits:
```
MsgRegisterEncryptionKey{
    Address: victim_bech32,
    Key:     age1<attacker_controlled_pubkey>,
}
```

This overwrites the victim's KVStore entry. All future senders who look up the victim's encryption key (via `Keeper.Key` or `Keeper.Keys`) will encrypt messages to the attacker's public key. The victim can never decrypt those messages. The attacker, holding the corresponding private key, can decrypt them. The victim's original key is permanently gone unless they re-register — but an attacker can immediately overwrite again.

This is **e2ee key/message state corruption with direct security impact**, matching the High impact category.

### Likelihood Explanation

The attack requires only a funded account and a single transaction. No special privileges, no validator access, no leaked keys. It is trivially repeatable and can be automated to continuously overwrite any re-registration by the victim.

### Recommendation

In `Keeper.RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    signer := sdkCtx.EventManager()... // or retrieve from msg signer list
    // assert signer == req.Address
    ...
}
```

Alternatively, remove the `Address` field from the message entirely and derive the target address exclusively from the transaction signer, which is the standard Cosmos SDK pattern for self-service key registration.

```go
func (k Keeper) registerEncryptionKey(
	ctx context.Context,
	address string,
	key []byte,
) error {
	bz, err := k.addressCodec.StringToBytes(address)
	if err != nil {
		return err
	}
	sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
	return nil
}
```
