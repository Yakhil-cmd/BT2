### Title
Commission Commitment with Minimum 1-Epoch Duration Allows Staker to Immediately Raise Commission to Max and Expire the Commitment - (File: `src/staking/staking.cairo`)

---

### Summary

`set_commission_commitment` in `src/staking/staking.cairo` only enforces `expiration_epoch > current_epoch`, meaning a staker can set a commitment with `expiration_epoch = current_epoch + 1` (a single-epoch window). Within that same epoch the commitment is active, the staker can raise commission to `max_commission` (up to 100%). The commitment then expires at the next epoch boundary, leaving delegators permanently subject to the raised commission with no recourse.

---

### Finding Description

The commission commitment mechanism is designed to let stakers signal a cap on their commission to attract delegators. When a commitment is active, `update_commission` allows the staker to set commission to **any value ≤ `max_commission`**, including increases from the current commission. Without a commitment, commission can only decrease.

The validation in `set_commission_commitment` is:

```cairo
assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
assert!(
    expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
    "{}",
    Error::EXPIRATION_EPOCH_TOO_FAR,
);
``` [1](#0-0) 

The lower bound is `expiration_epoch > current_epoch`, so `expiration_epoch = current_epoch + 1` passes. There is no minimum commitment duration.

`is_commission_commitment_active` returns `true` while `current_epoch < expiration_epoch`:

```cairo
fn is_commission_commitment_active(...) -> bool {
    self.get_current_epoch() < commission_commitment.expiration_epoch
}
``` [2](#0-1) 

So a commitment set with `expiration_epoch = current_epoch + 1` is active for the remainder of the current epoch only. Within that window, `update_commission` permits raising commission up to `max_commission`:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
}
``` [3](#0-2) 

Once the epoch advances, the commitment is expired and the staker is locked into only decreasing commission — but the raise has already taken effect.

---

### Impact Explanation

**High — Theft of unclaimed yield from delegators.**

A staker with an existing pool and delegators can:

1. Call `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
2. In the same epoch, call `set_commission(commission: 10000)` — raising commission to 100%.
3. The commitment expires at the next epoch boundary.
4. All future delegator rewards are now entirely captured by the staker as commission.

Delegators who entered the pool based on a previously low commission have their unclaimed and future yield permanently redirected to the staker. The staker cannot be forced to lower commission; they must voluntarily call `set_commission` with a lower value.

---

### Likelihood Explanation

**High.** Any staker with a pool can execute this in two transactions within a single epoch. No special conditions, privileged access, or external dependencies are required. The attack is cheap, deterministic, and immediately profitable for a malicious staker. Delegators have no on-chain mechanism to exit before the commission change takes effect within the same epoch.

---

### Recommendation

Enforce a minimum commitment duration. Require `expiration_epoch` to be at least `current_epoch + MIN_COMMITMENT_EPOCHS` where `MIN_COMMITMENT_EPOCHS` is a meaningful value (e.g., `K` epochs, or a governance-defined minimum such as one month of epochs). Replace:

```cairo
assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
```

with:

```cairo
assert!(
    expiration_epoch >= current_epoch + MIN_COMMITMENT_EPOCHS,
    "{}",
    Error::EXPIRATION_EPOCH_TOO_EARLY
);
``` [4](#0-3) 

---

### Proof of Concept

```
Epoch N:
  staker.set_commission_commitment(max_commission=10000, expiration_epoch=N+1)
    → commitment is active (N < N+1)
  staker.set_commission(commission=10000)
    → passes: 10000 <= max_commission(10000), 10000 != old_commission
    → commission is now 100%

Epoch N+1:
  is_commission_commitment_active → false (N+1 < N+1 is false)
  commitment is expired
  staker can only decrease commission, but it is already at 100%
  all delegator rewards flow entirely to staker
```

The test suite itself demonstrates `expiration_epoch = current_epoch + 1` is accepted as valid input: [5](#0-4)
