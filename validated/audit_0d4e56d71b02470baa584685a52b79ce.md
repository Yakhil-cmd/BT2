### Title
Sub-Minimum attoFlow Balance Permanently Locked in Cadence-Owned Account After Gas Fee Deduction - (File: fvm/evm/stdlib/contract.cdc)

### Summary
FLOW tokens deposited into a `CadenceOwnedAccount` (COA) can become permanently unrecoverable when EVM gas fee deductions reduce the COA's EVM-side balance below the minimum withdrawable threshold of `1e10 attoFlow`. The `CadenceOwnedAccount.withdraw()` function enforces a hard floor due to the precision mismatch between EVM's 18-decimal attoFlow and Cadence's 8-decimal `UFix64`. Gas fees are deducted at full attoFlow precision, and no recovery mechanism exists for sub-minimum residual balances.

### Finding Description

`EVM.CadenceOwnedAccount` is the only mechanism for Cadence code to interact with Flow EVM. FLOW deposited into a COA is tracked at 18-decimal attoFlow precision in the EVM state. The sole path to move that FLOW back to Cadence is `CadenceOwnedAccount.withdraw()`, which internally calls `InternalEVM.withdraw()`.

In `fvm/evm/impl/impl.go`, `newInternalEVMTypeWithdrawFunction` enforces:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
``` [1](#0-0) 

`AttoFlowBalanceIsValidForFlowVault` in `fvm/evm/types/balance.go` defines the floor:

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0  // 1e10
}
``` [2](#0-1) 

Any withdrawal attempt with an amount below `1e10 attoFlow` panics with `ErrWithdrawBalanceRounding`: [3](#0-2) 

EVM gas fees are deducted at full attoFlow precision by the EVM emulator. If a COA executes EVM transactions and the cumulative gas cost leaves a residual balance anywhere in the range `[1, 9_999_999_999]` attoFlow, the following is true:

- Calling `withdraw(balance: EVM.Balance(attoflow: residual))` panics because `residual < 1e10`.
- Calling `withdraw(balance: EVM.Balance(attoflow: 1e10))` fails at the EVM level because the COA has insufficient balance.
- Calling `withdraw(balance: EVM.Balance(attoflow: 0))` returns an empty vault — no tokens recovered.

The `withdraw()` function in `contract.cdc` has no "withdraw all" or "sweep dust" variant: [4](#0-3) 

The only theoretical escape is to transfer the sub-minimum balance to another EVM address via `coa.call()`, but this itself requires paying gas from the COA's balance. If the residual is smaller than the minimum gas cost (21,000 gas × gas price), even this path is blocked. There is no protocol-level recovery mechanism.

### Impact Explanation

FLOW tokens that enter the EVM environment via a COA can become permanently unrecoverable. The per-COA loss is bounded by `9_999_999_999 attoFlow` (≈ `0.00000001 FLOW`), but this affects every COA that has executed EVM transactions and attempted a full withdrawal. At scale — with many COAs — the aggregate locked value grows monotonically with no recovery path. This is a cross-VM asset loss: valid Cadence FLOW tokens are deposited into the EVM environment and a subset of them can never be returned to Cadence.

### Likelihood Explanation

This is a near-certain outcome for any COA that:
1. Deposits FLOW tokens,
2. Executes one or more EVM transactions (paying gas from the COA balance), and
3. Attempts to withdraw the remaining balance.

This is the standard COA usage pattern. The precision mismatch is structural and affects every such interaction. The code comment in `contract.cdc` acknowledges the minimum but does not document the scenario where gas fees leave the total balance below the minimum, nor does it provide a recovery path. [5](#0-4) 

### Recommendation

- **Short term:** Add a `withdrawAll()` function that withdraws `floor(balance / 1e10) * 1e10` attoFlow and either burns or donates the sub-minimum remainder to a protocol treasury, preventing permanent lock-up.
- **Long term:** Document the precision gap and its implications for COA lifecycle management. Consider whether sub-minimum residuals should be automatically swept at COA destruction or at each withdrawal.

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction {
    prepare(account: auth(BorrowValue) &Account) {
        // Step 1: Deposit exactly 1e10 attoFlow (minimum withdrawable amount)
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 0.00000001)
        let vault <- minter.mintTokens(amount: 0.00000001)  // = 1e10 attoFlow
        destroy minter

        let coa <- EVM.createCadenceOwnedAccount()
        coa.deposit(from: <-vault)
        // COA balance: 10_000_000_000 attoFlow

        // Step 2: Execute an EVM transaction that costs 1 attoFlow in gas
        // (e.g., a minimal call; gas deducted at attoFlow precision)
        // COA balance after gas: 9_999_999_999 attoFlow  (<1e10)

        // Step 3: Attempt to withdraw the remaining balance — PANICS
        let bal = EVM.Balance(attoflow: 9_999_999_999)
        let recovered <- coa.withdraw(balance: bal)
        // ^ panics: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"

        destroy coa
        destroy recovered
    }
}
```

The `9_999_999_999 attoFlow` residual has no withdrawal path and is permanently locked in the EVM state. [6](#0-5) [7](#0-6)
