### Title
Accrued Admin Fees Are Misdirected When `setPoolAdminFeeDestination` Is Called Without Prior Collection - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolAdminFeeDestination` changes the admin fee destination immediately without first collecting accrued fees at the old destination. All spread-surplus and notional fees that accrued before the destination change are subsequently swept to the new address on the next `collectPoolFees` call, permanently depriving the old destination of fees it was owed.

### Finding Description

The factory exposes two fee-related admin setters. `setPoolAdminFees` correctly calls `collectFees` with the current `poolAdminFeeDestination` before updating rates, ensuring accrued fees are settled at the old rates and old destination before any change takes effect: [1](#0-0) 

`setPoolAdminFeeDestination`, however, performs no such pre-collection. It immediately overwrites `poolAdminFeeDestination[pool]` with the new address: [2](#0-1) 

The pool's `collectFees` function computes the admin share from two sources at collection time: (1) the spread surplus (`balance - binTotals - notionalAccumulator`) and (2) the stored `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` accumulators. Both represent fees that accrued since the last collection. The `adminFeeDestination_` parameter passed at collection time determines where the admin share goes: [3](#0-2) 

Because `setPoolAdminFeeDestination` does not flush these accumulators first, any fees that accrued before the destination change will be paid to the new destination on the next `collectPoolFees` call (which reads the now-updated `poolAdminFeeDestination[pool]`): [4](#0-3) 

### Impact Explanation

The old `adminFeeDestination` — which may be a separate treasury contract, DAO, or multisig distinct from the pool admin — permanently loses all admin fees that accrued between the last collection and the destination change. These fees are not recoverable: they are transferred to the new destination on the next collection, and `notionalFeeToken0Scaled`/`notionalFeeToken1Scaled` are zeroed out. The magnitude scales with trading volume and the time elapsed since the last collection.

### Likelihood Explanation

The pool admin is a semi-trusted actor who legitimately performs treasury migrations (e.g., rotating to a new multisig or treasury contract). This is a normal operational action. The inconsistency with `setPoolAdminFees` — which does collect first — means the admin is not warned by the code pattern that a pre-collection is required. Any treasury rotation without a manual prior `collectPoolFees` call triggers the loss.

### Recommendation

Mirror the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`: collect accrued fees at the old destination before updating `poolAdminFeeDestination[pool]`.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to the OLD destination before rotating
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool has been trading; `notionalFeeToken0Scaled = X > 0` and spread surplus `S > 0` have accumulated since the last collection. Admin fee destination is `oldTreasury`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`. No collection occurs; `poolAdminFeeDestination[pool]` is now `newTreasury`.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool] == newTreasury` and passes it to `collectFees`.
4. Inside `collectFees`, the admin share of `S` (spread) and `X` (notional) is transferred to `newTreasury`. `notionalFeeToken0Scaled` is zeroed.
5. `oldTreasury` receives nothing. All fees accrued before the destination change are permanently lost to it. [2](#0-1) [5](#0-4)
