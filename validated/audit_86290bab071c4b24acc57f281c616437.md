### Title
Missing Minimum Threshold Lower-Bound in FROST Sign Validation Allows Signing Below Protocol-Guaranteed Minimum — (File: src/frost/mod.rs)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that the caller-supplied `threshold` does not exceed the participant count, but omits the lower-bound check (`threshold >= 2`) that is explicitly enforced during key generation in `assert_key_invariants` (`src/dkg.rs`). A malicious coordinator can invoke FROST signing with `threshold = 1`, causing the protocol to proceed with a single participant's partial signature. Because the key was generated under a degree-≥1 polynomial (threshold ≥ 2), the Lagrange interpolation with a single point is incorrect and the aggregated signature is cryptographically invalid — an unusable output delivered to honest parties.

---

### Finding Description

During key generation, `assert_key_invariants` enforces a hard lower bound:

```rust
// src/dkg.rs line 580
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The signing-time validation function `assert_sign_inputs` performs only an upper-bound check:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. The `ReconstructionLowerBound` type is a plain `usize` wrapper with no built-in minimum: [3](#0-2) 

The same gap exists in the FROST `presign` entry point:

```rust
// src/frost/mod.rs lines 71-77
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
``` [4](#0-3) 

A caller who passes `threshold = 1` to `assert_sign_inputs` with two or more participants clears every guard: `participants.len() >= 2` ✓, no duplicates ✓, `me` present ✓, `threshold (1) <= participants.len() (2)` ✓. The protocol then collects only one partial signature. In FROST, the Lagrange coefficient for a single-point interpolation is 1, so the aggregated value becomes `z = nonce_A + share_A · challenge`. For this to verify against the public key `pk = g^{secret}`, we would need `share_A = secret`, which holds only for a degree-0 (threshold-1) polynomial — not for any key generated under the enforced `threshold >= 2` constraint. The output signature is therefore invalid.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties receive an unusable cryptographic output.**

Any honest party that relies on the FROST signing session receives a signature that fails standard verification. The corruption is silent at the protocol level: the signing session completes without error, but the produced `(R, z)` pair does not satisfy the verification equation. This matches the allowed impact: *"Corruption of … sign … outputs so honest parties accept … unusable cryptographic outputs."*

---

### Likelihood Explanation

The coordinator role is the natural attacker. `assert_sign_inputs` is a public library function; any caller constructing a FROST signing session controls the `threshold` argument directly. No special privilege, leaked key, or cryptographic break is required — passing the integer `1` is sufficient to trigger the condition. The `ReconstructionLowerBound` wrapper imposes no compile-time or runtime floor.

---

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` (and to the `presign` entry point) that already exists in `assert_key_invariants`:

```rust
// src/frost/mod.rs — inside assert_sign_inputs, after converting threshold
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the identical guard inside `presign` after line 72 in `src/frost/mod.rs`.

---

### Proof of Concept

1. Run DKG with `participants = [A, B]`, `threshold = 2`. Key shares are degree-1 polynomial evaluations; `share_A ≠ secret_key`.
2. Call `assert_sign_inputs(&[A, B], /*threshold=*/1u8, A, A)`.
   - `participants.len() = 2 >= 2` ✓
   - `threshold (1) <= participants.len() (2)` ✓
   - All other checks pass ✓ — function returns `Ok(participants)`.
3. Signing session proceeds; only participant A's partial signature is collected.
4. Aggregated `z = nonce_A + 1 · share_A · challenge`.
5. Verification: `g^z = g^{nonce_A} · (g^{share_A})^{challenge} ≠ R_A · pk^{challenge}` because `g^{share_A} ≠ pk`.
6. Honest verifier rejects the signature — the signing session produced a corrupted, unusable output with no protocol-level error raised. [5](#0-4) [6](#0-5)
