### Title
Gateway Admits Signature-Bypassed Invoke Transactions via Overly Broad `skip_stateful_validations` Check - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (account signature verification) for any invoke transaction with `nonce=1` when the sender's account has **any** transaction in the mempool or a recent committed block — not exclusively a `deploy_account` transaction. An unprivileged attacker can exploit this to inject invoke transactions with arbitrary/invalid signatures into the mempool for any account whose on-chain nonce is `0` and that has a pending transaction in the pool.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the simultaneous `deploy_account + invoke` submission pattern: when a user sends both transactions at once, the account does not yet exist on-chain, so the gateway cannot run `__validate__` on the invoke. The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The account's on-chain nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The code comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

However, `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type from that address — including a regular invoke with `nonce=0` submitted by the legitimate account owner. It does **not** verify that the transaction in the pool is specifically a `deploy_account`.

**Attack path:**

Consider an already-deployed account `A` with on-chain nonce `0` (e.g., freshly deployed, no committed transactions yet):

1. The legitimate owner submits `invoke_A_nonce0` (valid signature, nonce=0). It passes `__validate__` and enters the mempool.
2. An attacker observes `A` in the mempool and submits `invoke_A_nonce1` with an **invalid/arbitrary signature** and `nonce=1`.
3. In `run_pre_validation_checks`, `validate_state_preconditions` passes (nonce=1 is within the allowed gap from account_nonce=0). [3](#0-2) 

4. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(A)` is `true` (the legitimate nonce=0 invoke is in the pool).
5. `run_validate_entry_point` is called with `skip_validate=true`, setting `execution_flags.validate = false`. [4](#0-3) 

6. `StatefulValidator::perform_validations` skips the `__validate__` call entirely and returns `Ok(())`. [5](#0-4) 

7. The invalid invoke is forwarded to the mempool via `mempool_client.add_tx(...)`. [6](#0-5) 

The same attack applies to accounts that have a `deploy_account` in the mempool: any third party can submit an invalid invoke with `nonce=1` for that address, bypassing signature verification.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Invalid invoke transactions with arbitrary signatures are admitted to the mempool without account signature verification. While the batcher will reject them during execution (when `__validate__` is run again), the gateway's admission invariant — that all accepted transactions carry a valid account signature — is broken. This enables:

- **Mempool pollution**: An attacker can flood the mempool with invalid invokes for any account with `nonce=0` and a pending transaction, consuming mempool capacity.
- **Batcher resource waste**: The batcher must attempt execution of invalid transactions before rejecting them.
- **Griefing of deploy_account flows**: Any user submitting a `deploy_account` becomes a target; attackers can inject invalid invokes for their address before the account is even created.

### Likelihood Explanation

**Medium.** The attacker requires no special privileges. The only prerequisite is observing an account with `nonce=0` in state that has any pending transaction in the mempool — information that is publicly observable. The attack is cheap (one RPC call per invalid transaction) and can be automated.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address in the mempool. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that only returns `true` when a `deploy_account` transaction (not any transaction) is pending for that address. This preserves the intended UX while closing the signature-bypass path.

### Proof of Concept

```
1. Deploy account A (nonce=0 in state, account exists on-chain).
2. Legitimate owner submits invoke_A with nonce=0 and valid signature.
   → Passes __validate__, enters mempool.
   → account_tx_in_pool_or_recent_block(A) now returns true.

3. Attacker submits invoke_A with nonce=1 and signature = [0xDEAD, 0xBEEF] (invalid).
   → validate_nonce: account_nonce=0, tx_nonce=1, within allowed gap → PASS
   → validate_by_mempool: nonce check only → PASS
   → skip_stateful_validations: nonce==1, account_nonce==0,
     account_tx_in_pool_or_recent_block(A)==true → returns true (SKIP __validate__)
   → run_validate_entry_point: validate=false → __validate__ NOT called → PASS
   → mempool_client.add_tx(...) → invalid invoke ADMITTED to mempool

4. Batcher executes block:
   → invoke_A nonce=0 executes successfully.
   → invoke_A nonce=1 runs __validate__ → signature invalid → REJECTED.

Result: Invalid transaction with arbitrary signature was admitted to the mempool,
        bypassing the gateway's account signature verification invariant.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
