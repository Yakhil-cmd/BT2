Audit Report

## Title
Subaccount Scope Bypass in `get_allowances` Leaks Cross-Subaccount Allowance Data — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function in the ICRC-1 ledger terminates its `StableBTreeMap` range scan only when the iterated entry's `account.owner` differs from the requested `from.owner`, never comparing `account.subaccount`. When a caller supplies a specific subaccount in `from_account`, the function silently returns allowances belonging to every other subaccount of the same owner that sorts lexicographically after the requested one. Because `icrc103_get_allowances` is an unauthenticated public `#[query]`, any unprivileged principal can enumerate the full allowance table (spender identities, approved amounts, expiry times) for all subaccounts of any victim principal.

## Finding Description
`icrc103_get_allowances` is a public `#[query]` with no caller authentication or authorization check. [1](#0-0) 

It passes the caller-supplied `from_account` directly to `get_allowances`. Inside `get_allowances`, the range scan starts at `(from_account, min_spender)` and iterates forward through the `StableBTreeMap`. The only termination guard on the `from` dimension is: [2](#0-1) 

This checks `account_spender.account.owner != from.owner` — **owner only**. It never compares `account_spender.account.subaccount` against `from.subaccount`.

`AccountSpender` ordering is lexicographic on `(account.owner, account.effective_subaccount(), spender.owner, spender.effective_subaccount())`: [3](#0-2) [4](#0-3) 

So for `from = {owner: V, subaccount: Some([1;32])}`, the scan starts at the first entry whose account is `(V, [1;32])` and continues through `(V, [2;32])`, `(V, [3;32])`, …, `(V, [255;32])` — all subaccounts of `V` that sort after `[1;32]` — before finally hitting a different owner and breaking. Every one of those entries is appended to the result.

The `GetAllowancesArgs` struct takes `from_account: Option<Account>` (owner + subaccount), confirming the intended scope is a specific account, not all subaccounts of an owner: [5](#0-4) 

## Impact Explanation
Any unprivileged principal can enumerate the full set of allowances (spender identities, approved amounts, expiry times) for every subaccount of a victim principal whose subaccount sorts after the one supplied in `from_account`. By sweeping `from_account.subaccount` from `[0;32]` upward, an attacker can reconstruct the complete allowance table for any principal. This is a direct confidentiality violation against the ICRC-1 ledger — a production in-scope financial integration. This matches **High ($2,000–$10,000): Significant ledger security impact with concrete user harm**.

## Likelihood Explanation
The endpoint is a public `#[query]` with no caller authentication. Any IC principal can call it with an arbitrary `from_account`. The exploit requires a single canister call, is deterministically reproducible, and requires no special privileges, victim interaction, or economic cost beyond normal query fees.

## Recommendation
Replace the owner-only guard with a full `Account` equality check (owner **and** effective subaccount):

```rust
// rs/ledger_suite/icrc1/ledger/src/lib.rs — get_allowances(), inside the range loop
if account_spender.account != from {   // was: account_spender.account.owner != from.owner
    break;
}
```

`Account::eq` already compares both `owner` and `effective_subaccount()`: [6](#0-5) 

This confines the scan to the exact `(owner, subaccount)` pair the caller requested.

## Proof of Concept

```rust
// State-machine test
let victim = Principal::from_slice(&[1u8; 2]);
let sub1: Subaccount = [1u8; 32];
let sub2: Subaccount = [2u8; 32];  // sorts after sub1
let spender_a = Account { owner: Principal::from_slice(&[10u8; 2]), subaccount: None };
let spender_b = Account { owner: Principal::from_slice(&[11u8; 2]), subaccount: None };

// Victim approves from two different subaccounts
approve(&env, ledger, victim, sub1, spender_a, 1_000);
approve(&env, ledger, victim, sub2, spender_b, 2_000);

// Attacker (any principal) queries only sub1
let args = GetAllowancesArgs {
    from_account: Some(Account { owner: victim, subaccount: Some(sub1) }),
    prev_spender: None,
    take: None,
};
let result = icrc103_get_allowances(&env, ledger, attacker, args).unwrap();

// BUG: both sub1 and sub2 allowances are returned
assert_eq!(result.len(), 2);   // should be 1
assert!(result.iter().any(|a| a.from_account.subaccount == Some(sub2)));  // leaked
```

The scan starts at `(victim, [1;32], min_spender)`, passes through `(victim, [1;32], spender_a)` (correct), then continues to `(victim, [2;32], spender_b)` (leaked) because the break guard at line 1204 tests only `account.owner`, not `account.subaccount`. [7](#0-6)

fn icrc103_get_allowances(arg: GetAllowancesArgs) -> Result<Allowances, GetAllowancesError> {
    let from_account = arg.from_account.unwrap_or_else(|| Account {
        owner: ic_cdk::api::msg_caller(),
        subaccount: None,
    });
    let max_take_allowances = Access::with_ledger(|ledger| ledger.max_take_allowances());
    let max_results = arg
        .take
        .map(|take| take.0.to_u64().unwrap_or(max_take_allowances))
        .map(|take| std::cmp::min(take, max_take_allowances))
        .unwrap_or(max_take_allowances);
    Ok(get_allowances(
        from_account,
        arg.prev_spender,
        max_results,

    ))
