### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Incomplete User Refunds - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the dual-resource accounting reconciliation only adjusts `gas_used` upward when native resource consumption exceeds EVM gas consumption (`delta_gas > 0`). When native consumption is *less* than EVM gas consumption (`delta_gas < 0`), `gas_used` is never reduced. A `// TODO: return delta_gas to gas_used?` comment in the code explicitly acknowledges this asymmetry as unresolved. The result is that users are systematically overcharged: they receive a smaller gas refund than the actual resource consumption justifies, with the excess flowing to the operator.

---

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` implements ZKsync OS's double-resource accounting reconciliation. After execution, it computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

The code only handles the `delta_gas > 0` branch — adding extra gas when native resource consumption exceeds EVM gas consumption. The `delta_gas < 0` branch (native consumption less than EVM gas consumption) is silently ignored, with a `// TODO: return delta_gas to gas_used?` comment marking the gap:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [2](#0-1) 

The resulting `gas_used` is then used to compute the user's refund in `refund_and_commit_fee`:

```rust
let token_to_refund =
    context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);
``` [3](#0-2) 

And the operator payment:

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)
``` [4](#0-3) 

Because `gas_used` is inflated (not reduced when `delta_gas < 0`), the user's refund is smaller than it should be, and the operator receives more than it should.

The documentation for double resource accounting only describes the `delta_gas > 0` case and is silent on the negative case, confirming the asymmetry is unresolved:

> "If `deltaGas > 0`, we add it to `gasUsed`... Finally, any remaining gas left is refunded as usual." [5](#0-4) 

---

### Impact Explanation

**Resource accounting bug / incomplete refund.** When a transaction's EVM gas consumption exceeds its native resource consumption (in gas units), the user is overcharged by:

```
overcharge = |delta_gas| * gas_price
           = (gas_used - native_used / native_per_gas) * gas_price
```

In production (without `burn_base_fee`), the operator receives `gas_used * gas_price` and the user receives `(gas_limit - gas_used) * gas_price` as refund. The excess `|delta_gas| * gas_price` is transferred from the user to the operator rather than being refunded. This is a direct, quantifiable financial loss for every affected user.

---

### Likelihood Explanation

**High.** The `delta_gas < 0` condition occurs in any transaction where EVM gas consumption exceeds native resource consumption in gas units — a common scenario for computation-heavy transactions with low pubdata (e.g., pure arithmetic, in-memory operations, or transactions that touch few storage slots). Every such transaction submitted by any unprivileged sender is affected. No special conditions, privileges, or adversarial setup are required; the bug is triggered by normal transaction execution.

---

### Recommendation

Apply the negative `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so that the refund accurately reflects actual resource consumption:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Return unused gas to the user
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This resolves the TODO comment and ensures the refund equals the full unused gas, consistent with the design intent described in `double_resource_accounting.md`.

---

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`, `gas_price = 1_000`
- `native_price = 10` → `native_per_gas = gas_price / native_price = 100`
- Transaction uses `50_000` EVM gas but only `10_000` native units (computation-heavy, low pubdata)

**Current behavior:**
1. `gas_used (EVM) = 50_000`
2. `native_used = 10_000`
3. `delta_gas = (10_000 / 100) - 50_000 = 100 - 50_000 = -49_900`
4. Since `delta_gas < 0`, it is ignored; `gas_used` stays at `50_000`
5. Refund to user: `(100_000 - 50_000) * 1_000 = 50_000_000` wei
6. Operator receives: `50_000 * 1_000 = 50_000_000` wei

**Correct behavior (with fix):**
1. `delta_gas = -49_900` → `gas_used` reduced to `100`
2. Refund to user: `(100_000 - 100) * 1_000 = 99_900_000` wei
3. Operator receives: `100 * 1_000 = 100_000` wei

**User loss per transaction: `49_900_000` wei** — transferred to the operator instead of being refunded. [6](#0-5) [7](#0-6)
