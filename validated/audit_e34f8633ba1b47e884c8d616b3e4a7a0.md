### Title
WalletConnect Multi-NFT Royalty Amount Underestimation Causes User to Approve Wrong Spend — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in the WalletConnect confirmation pipeline incorrectly divides the fungible amount by the number of NFTs before computing each royalty. When a dapp sends a `take_offer` or `create_offer_for_ids` WalletConnect request containing two or more NFTs with royalties, the `amountWithRoyalties` shown in the approval dialog is systematically underestimated by a factor of N (the number of royalty-bearing NFTs). The user approves believing they will spend less XCH/CAT than the transaction actually deducts.

---

### Finding Description

`formatAmountWithRoyalties` is called from `withRoyaltyTotals` to compute the total-including-royalties figure displayed in the WalletConnect `Confirm` dialog for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` commands. [1](#0-0) 

The function collects all NFT royalty percentages from the opposite trade side via `royaltyPercentagesForSide`, then computes:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // BUG: divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [2](#0-1) 

**Correct formula** (each NFT royalty applies to the full amount):
```
royaltyTotal = Σ (amount × royaltyPercentage_i / 10_000)
```

**Actual formula** (amount is split before each royalty is applied):
```
royaltyTotal = Σ ((amount / N) × royaltyPercentage_i / 10_000)
             = correct_royaltyTotal / N
```

The displayed `amountWithRoyalties` is therefore `N` times smaller than the actual royalty obligation whenever N ≥ 2.

The existing test at line 251–335 of `parseCommandDisplay.test.ts` encodes and validates this buggy result (`0.00010255` XCH) rather than the correct value (`0.0001051` XCH) for a two-NFT offer with 5% and 0.1% royalties. [3](#0-2) 

The computed `amountWithRoyalties` is propagated through `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay` and rendered in the `Confirm` dialog that the user must approve before the WalletConnect command is dispatched. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

The `Confirm` dialog is the sole user-visible gate before a WalletConnect-initiated offer is executed. A user who sees `amountWithRoyalties: "1.025 XCH"` when the correct figure is `1.05 XCH` (two NFTs, 2.5% each) will approve a transaction that spends more XCH than they consented to. The underestimation scales linearly with the number of royalty-bearing NFTs: three NFTs → displayed royalty is 3× too small, etc. This constitutes WalletConnect state corruption that causes a user to approve the wrong amount, matching the **High** impact tier.

---

### Likelihood Explanation

Any dapp connected via WalletConnect can send a `take_offer` request containing a pre-constructed offer with two or more NFTs that carry royalties — a normal, supported offer type. No special privileges, leaked keys, or social engineering are required. The dapp does not need to craft a malicious offer; even a legitimate multi-NFT offer triggers the underestimation. The user has no way to detect the discrepancy from the dialog alone.

---

### Recommendation

Remove the erroneous division by `royaltyPercentages.length`. Each royalty percentage must be applied to the full `amount`:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;

- const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
-   (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
+   (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

Update the corresponding test expectation from `'0.00010255'` to `'0.0001051'`.

### Proof of Concept

**Scenario**: A dapp sends a WalletConnect `chia_wallet.take_offer` request for an offer where the user spends 0.0001 XCH (100,000,000 mojos) to receive two NFTs — NFT-A with 5% royalty (500 basis points) and NFT-B with 0.1% royalty (10 basis points).

**Displayed** (buggy):
```
splitAmount = 100_000_000 / 2 = 50_000_000
royaltyAmount = (50_000_000 × 500 / 10_000) + (50_000_000 × 10 / 10_000)
              = 2_500_000 + 50_000 = 2_550_000
amountWithRoyalties = 102_550_000 mojos → "0.00010255 XCH"
```

**Actual spend** (correct):
```
royaltyAmount = (100_000_000 × 500 / 10_000) + (100_000_000 × 10 / 10_000)
              = 5_000_000 + 100_000 = 5_100_000
actual total  = 105_100_000 mojos → 0.0001051 XCH
```

The user approves seeing `0.00010255 XCH` but the wallet deducts `0.0001051 XCH` — a 2× royalty overcharge relative to what was displayed. With N NFTs the factor grows to N×.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
