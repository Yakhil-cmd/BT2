### Title
`tick()` in `RewardSupplier.sol` Splits `msg.value` Arbitrarily Between Two L1→L2 Messages Without Fee Validation, Enabling Griefing That Freezes Minted STRK in the Bridge - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is publicly callable and payable. When pending L2→L1 mint requests exist, it mints STRK tokens and sends them to L2 via two separate L1→L2 messages. The ETH fee for both messages is taken entirely from `msg.value`, split as `msg.value / 2` each, with no minimum-fee check or fee estimation. Any caller can invoke `tick()` with a trivially small `msg.value` (e.g., 2 wei), causing both messages to be accepted by the Starknet messaging contract (which requires `msg.value >= 1` per message) but never processed by the sequencer due to insufficient fees. The result is that freshly minted STRK tokens are deposited into the StarkGate bridge and permanently locked there, because the `RewardSupplier` contract has no mechanism to cancel the bridge's L1→L2 message.

---

### Finding Description

`tick()` is `external payable` with no access control:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        ...
        mintManager().mintRequest(token(), amountToMint);          // tokens minted

        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(               // fee = msg.value/2
            token(), amountToMint, mintDestination(), new uint256[](0)
        );

        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(       // fee = msg.value - msg.value/2
            mintingCurve(), UPDATE_TOTAL_SUPPLY_SELECTOR, messagePayload
        );
    }
}
``` [1](#0-0) 

The `IStarknetMessaging` interface exposes `getMaxL1MsgFee()` but the contract never calls it, and there is no minimum-fee guard: [2](#0-1) 

The Starknet messaging contract enforces only `msg.value >= 1` per `sendMessageToL2` call. The StarkGate bridge's `depositWithMessage` internally calls `sendMessageToL2`, so it also requires only `msg.value >= 1`. Therefore, calling `tick()` with `msg.value = 2` satisfies both on-chain checks (each sub-call receives 1 wei), but 1 wei is far below the sequencer's minimum processing threshold. The messages are recorded on-chain but never consumed by the sequencer.

The `RewardSupplier` contract has no `startL1ToL2MessageCancellation` or `cancelL1ToL2Message` wrapper. The bridge is the actual sender of the deposit message, so only the bridge can cancel it. Without a bridge-level cancellation path, the deposited tokens are frozen. [3](#0-2) 

---

### Impact Explanation

Each `tick()` call can process up to `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5` pending mint requests, each for `TOKENS_PER_MINT_REQUEST = 1,300,000 × 10^18` STRK, totalling up to **6,500,000 STRK per tick**. [4](#0-3) 

If `tick()` is called with `msg.value = 2`:
1. Up to 6.5M STRK are minted from `mintManager`.
2. All tokens are deposited into the StarkGate bridge with a 1-wei fee — accepted on-chain but ignored by the sequencer.
3. The `update_total_supply` message to `MintingCurve` is also sent with 1-wei fee and never processed.
4. The L2 `RewardSupplier` never receives the tokens; `l1_pending_requested_amount` is never decremented; the minting curve receives a stale total supply.

This matches the allowed impact: **Temporary (potentially permanent) freezing of funds**.

---

### Likelihood Explanation

- `tick()` has no access control — any EOA or contract can call it.
- The only precondition is that pending L2→L1 mint requests exist, which is a normal protocol state that occurs every epoch when rewards accumulate.
- The cost to the attacker is 2 wei plus gas. There is no profit motive required; this is a pure griefing vector.
- The L2 `RewardSupplier` sends mint requests via `send_message_to_l1_syscall` whenever `credit < debit + threshold`, making the precondition reliably reachable. [5](#0-4) [6](#0-5) 

---

### Recommendation

Before executing the mint-and-bridge logic, validate that `msg.value` is sufficient to cover both L1→L2 message fees. Use `getMaxL1MsgFee()` as an upper bound or require a caller-supplied minimum:

```solidity
function tick(uint256 minFeePerMsg) external payable {
    uint256 msgFee = msg.value / 2;
    require(msgFee >= minFeePerMsg, "INSUFFICIENT_FEE");
    ...
}
```

Alternatively, enforce a protocol-defined minimum fee constant and revert if `msg.value / 2` falls below it. Also consider adding a `cancelBridgeDeposit` escape hatch that delegates to the bridge's cancellation mechanism so that stuck deposits can be recovered.

### Proof of Concept

1. L2 `RewardSupplier.request_funds()` sends N mint-request messages to L1 (normal protocol operation).
2. Attacker calls `RewardSupplier.tick{value: 2}()` on L1.
3. `requiredMinting()` returns `amountToMint = TOKENS_PER_MINT_REQUEST * N > 0`.
4. `mintManager().mintRequest(token(), amountToMint)` — STRK tokens are minted.
5. `bridge().depositWithMessage{value: 1}(token(), amountToMint, mintDestination(), [])` — accepted by bridge (1 wei ≥ minimum), but sequencer ignores the message.
6. `messagingContract().sendMessageToL2{value: 1}(mintingCurve(), UPDATE_TOTAL_SUPPLY_SELECTOR, [...])` — accepted by messaging contract (1 wei ≥ minimum), but sequencer ignores the message.
7. Minted STRK tokens are locked in the bridge. The L2 reward supplier never receives them. The minting curve never gets the updated total supply. The `l1_pending_requested_amount` on L2 is never decremented, causing future reward accounting to be incorrect. [7](#0-6)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
