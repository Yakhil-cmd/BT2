### Title
Unicast Stream Role Authorization Bypassed by Always-True Stub in Production Node Builders - (`network/message/authorization.go`, `cmd/scaffold.go`)

---

### Summary

`AlwaysAuthorizedUnicastSenderRole` — a function that unconditionally returns `true` — is wired into production node builders via `cmd/scaffold.go`, `follower/follower_builder.go`, `cmd/access/node_builder/access_node_builder.go`, and `cmd/observer/node_builder/observer_builder.go`. This directly replaces the production enforcement function `IsAuthorizedUnicastSenderRole`, which enforces a strict role-based allowlist governing which node roles may open unicast streams to which receiver roles. The result is that any peer — including an unstaked Access or Observer node — can open a unicast stream to any node type without role-based rejection at the stream-open boundary.

---

### Finding Description

`network/message/authorization.go` defines two functions for unicast stream authorization:

**Production enforcement function:**
```go
// IsAuthorizedUnicastSenderRole checks whether the given sender role is authorized
// to open a unicast stream to the given receiver role.
func IsAuthorizedUnicastSenderRole(sender flow.Role, receiver flow.Role) bool {
    senders, ok := unicastRoleAuthorization[receiver]
    return ok && senders.Contains(sender)
}
``` [1](#0-0) 

**Stub that always returns true:**
```go
// AlwaysAuthorizedUnicastSenderRole is unicast stream authorizer that always returns true.
//
// This is used for the public network where peers are not authorized based on role.
func AlwaysAuthorizedUnicastSenderRole(sender, receiver flow.Role) bool {
    return true
}
``` [2](#0-1) 

The `unicastRoleAuthorization` map enforces strict role-pair restrictions:
- Execution nodes may only receive unicasts from Consensus and Collection nodes.
- Verification nodes may only receive unicasts from Consensus and Execution nodes.
- Consensus nodes may only receive unicasts from Consensus, Execution, and Verification nodes. [3](#0-2) 

The `WithUnicastStreamAuthorizer` option in `network/underlay/network.go` documents that the network defaults to `message.IsAuthorizedUnicastSenderRole` when `nil` is passed, meaning the secure default exists but is being explicitly overridden: [4](#0-3) 

The override to `AlwaysAuthorizedUnicastSenderRole` is applied in the base node scaffold (`cmd/scaffold.go`) and in the Access, Observer, and Follower node builders: [5](#0-4) [6](#0-5) 

The `ErrUnauthorizedUnicastSender` error — which is supposed to be reported to the slashing violations consumer when a peer's role is not authorized — is never triggered because the authorizer always returns `true`: [7](#0-6) 

---

### Impact Explanation

The unicast stream authorization check is the **pre-read gate** — it fires before any message bytes are consumed from the stream. Bypassing it means:

1. Any peer (including an unstaked Access or Observer node) can open a unicast stream to an Execution node, a Verification node, or a Consensus node, roles that should only accept unicasts from a narrow set of authorized peer roles.
2. The slashing/violation reporting path for `ErrUnauthorizedUnicastSender` is never exercised, so unauthorized stream-open attempts are silently accepted rather than penalized.
3. Downstream message-level authorization (channel subscription checks) becomes the only gate, but it operates after the stream is already open and bytes are being read, increasing the attack surface for resource exhaustion and message injection.

---

### Likelihood Explanation

Any peer that can establish a libp2p connection to a staked node — including an unstaked Access or Observer node, which is an explicitly in-scope attacker profile — can trigger this path. No privileged keys or staked node control is required. The entry point is a standard libp2p stream open, which is reachable from the public network.

---

### Recommendation

Replace `AlwaysAuthorizedUnicastSenderRole` with `IsAuthorizedUnicastSenderRole` in all production node builders that are not explicitly public-facing (i.e., all node types except Access and Observer nodes that serve external clients). For Access and Observer nodes, document the intentional bypass explicitly and ensure downstream message-level authorization is sufficient. The base scaffold (`cmd/scaffold.go`) should not set the always-true stub as the default for all node types.

---

### Proof of Concept

An attacker operating as an unstaked Access node:

1. Establishes a libp2p connection to a staked Execution node.
2. Opens a unicast stream using the Flow unicast protocol ID.
3. The stream-open handler calls the configured `UnicastStreamAuthorizer` — which is `AlwaysAuthorizedUnicastSenderRole` — and receives `true` unconditionally.
4. The stream is accepted; the attacker can now send arbitrary encoded messages to the Execution node's unicast handler.
5. `ErrUnauthorizedUnicastSender` is never emitted; no slashing report is filed.

Under the production function `IsAuthorizedUnicastSenderRole`, step 3 would return `false` for an Access-role sender targeting an Execution-role receiver (Access is not in `flow.RoleExecution`'s allowed sender list), and the stream would be rejected before any message data is read.
