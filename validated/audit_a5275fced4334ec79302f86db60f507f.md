### Title
Missing Zero-Address Validation for `eth_recipient` in `ExitToEthereum` Precompile — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile (`ExitToEthereum::run`) accepts a user-supplied 20-byte Ethereum recipient address for both ETH base-token withdrawals (flag `0x0`) and ERC-20 withdrawals (flag `0x1`), but performs no check that the address is non-zero before scheduling the irreversible NEAR withdrawal promise. A user who passes the zero address as `eth_recipient` will have their funds permanently burned from the Aurora EVM with no recovery path.

---

### Finding Description

In `engine-precompiles/src/native.rs`, `ExitToEthereum::run` handles two withdrawal paths:

**Flag `0x0` — ETH base token:**

```rust
let recipient_address: Address = input
    .try_into()
    .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS")))?;
// ← No check: recipient_address != Address::zero()
``` [1](#0-0) 

**Flag `0x1` — ERC-20 token:**

```rust
let recipient_address = Address::try_from_slice(input)
    .map_err(|_| ExitError::Other(Cow::from("ERR_WRONG_ADDRESS")))?;
// ← No check: recipient_address != Address::zero()
``` [2](#0-1) 

In both branches, after parsing the recipient, the precompile unconditionally schedules a NEAR cross-contract promise to call `withdraw` on the ETH connector contract, passing the zero address as the Ethereum recipient:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,          // contains recipient_address = 0x000...000
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};
``` [3](#0-2) 

The ETH (or ERC-20 tokens) are deducted from the caller's EVM balance at the point the precompile executes. The NEAR promise is then scheduled with the zero address as the Ethereum-side recipient. There is no `error_refund` mechanism in `ExitToEthereum` (unlike `ExitToNear` which has a conditional `error_refund` feature), so if the downstream withdrawal fails or succeeds with the zero address, the EVM-side funds are already gone with no refund path.

The same gap exists in the NEAR-level `withdraw` entry point, which also forwards `recipient_address` from user-supplied `WithdrawCallArgs` without a zero-address check: [4](#0-3) 

---

### Impact Explanation

**Severity: High — Permanent freezing/loss of user funds.**

When a user calls `ExitToEthereum` with `eth_recipient = 0x0000000000000000000000000000000000000000`:

1. Their ETH (or ERC-20 tokens) are burned from the Aurora EVM state.
2. A NEAR promise is scheduled to withdraw those funds to the zero address on Ethereum.
3. Funds sent to the Ethereum zero address are permanently inaccessible — no private key controls it.
4. There is no refund mechanism in `ExitToEthereum` to return funds to the caller if the recipient is invalid.

The result is a permanent, irreversible loss of the user's funds.

---

### Likelihood Explanation

**Medium.** The zero address is a common mistake in EVM development — users or contracts that programmatically construct the `eth_recipient` field (e.g., from an uninitialized variable, a failed address lookup, or a misconfigured calldata encoder) can easily produce an all-zero 20-byte recipient. The precompile is directly callable by any unprivileged EVM account or contract. No special permissions are required. The analogous FETH finding was judged medium severity for the same reason.

---

### Recommendation

Add an explicit zero-address guard in `ExitToEthereum::run` immediately after parsing `recipient_address` in both the `0x0` and `0x1` branches:

```rust
if recipient_address == Address::zero() {
    return Err(ExitError::Other(Cow::from("ERR_RECIPIENT_IS_ZERO_ADDRESS")));
}
```

Apply the same guard in the NEAR-level `withdraw` function in `engine/src/contract_methods/connector.rs` before forwarding `recipient_address` to `engine_withdraw`. [5](#0-4) 

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Any EVM account calls the `ExitToEthereum` precompile at address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with:
   - `input[0] = 0x00` (flag: ETH base token withdrawal)
   - `input[1..21] = [0u8; 20]` (zero address as `eth_recipient`)
   - `msg.value = N` (any nonzero ETH amount to withdraw)

2. `ExitToEthereum::run` parses `recipient_address = Address([0u8; 20])`. [1](#0-0) 

3. No zero-address check is performed. The precompile serializes the zero address into the withdrawal args and schedules the NEAR promise. [3](#0-2) 

4. `N` ETH is burned from the caller's Aurora EVM balance. The NEAR ETH connector processes a withdrawal to `0x0000000000000000000000000000000000000000` on Ethereum.

5. Funds are permanently lost — the zero address on Ethereum has no controlling key, and there is no refund path back to the Aurora caller.

The same attack applies to the ERC-20 path (flag `0x1`) at lines 946–947, where `recipient_address` is equally unchecked. [6](#0-5)
