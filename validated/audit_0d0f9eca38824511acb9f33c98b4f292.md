### Title
Unrestricted `update_rewards` with `disable_rewards=true` Allows Any Caller to Permanently Freeze Consensus Block Rewards — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the `Staking` contract is callable by any non-zero address. When called with `disable_rewards=true`, it unconditionally advances the global `last_reward_block` counter without distributing any rewards. Because the contract enforces a strict one-call-per-block invariant on this global counter, an attacker can front-run every legitimate reward update, permanently starving all stakers of consensus block rewards at negligible cost.

---

### Finding Description

`update_rewards` is exposed as a public ABI function under `IStakingRewardsManager`. Its only access gate is `general_prerequisites`, which checks that the contract is not paused and the caller is non-zero — no role or identity check is performed. [1](#0-0) 

Inside the function, the global `last_reward_block` is written **before** the `disable_rewards` branch is evaluated: [2](#0-1) 

`last_reward_block` is a single, contract-wide storage slot — not per-staker: [3](#0-2) 

The one-call-per-block invariant is enforced by: [4](#0-3) 

Because the write to `last_reward_block` happens unconditionally (line 1485) and the early-return for `disable_rewards` happens after (line 1487), any caller who submits `update_rewards(any_active_staker, disable_rewards: true)` first in a block:

1. Advances `last_reward_block` to the current block number.
2. Returns without distributing STRK block rewards to the staker or its pools.
3. Causes every subsequent `update_rewards` call in the same block to revert with `REWARDS_ALREADY_UPDATED`.

Since `last_reward_block` is global, a single such call per block is sufficient to deny rewards to **every** staker for that block.

---

### Impact Explanation

An attacker who submits `update_rewards(any_active_staker, true)` at the start of each block permanently freezes all consensus block rewards across the entire protocol. No staker can receive STRK block rewards as long as the attacker continues the pattern. This constitutes **permanent freezing of unclaimed yield** for all stakers and delegators, matching the High-severity impact tier.

---

### Likelihood Explanation

The attack requires only:
- A non-zero caller address (no stake, no role).
- One transaction per block submitted before the legitimate reward update.

On Starknet, transaction ordering within a block is deterministic and publicly observable. An attacker can reliably front-run reward updates by submitting a low-cost transaction at the start of each block. The cost to the attacker is only gas; the cost to the protocol is the entire consensus reward stream.

---

### Recommendation

Restrict `update_rewards` to a trusted caller (e.g., the consensus layer contract or a designated operator role), or separate the `last_reward_block` advancement from the reward distribution so that a `disable_rewards=true` call does not consume the block's reward slot for all other stakers.

---

### Proof of Concept

1. Consensus rewards are active: `consensus_rewards_first_epoch` has been set and the current epoch has passed it.
2. Attacker (any EOA) calls `update_rewards(active_staker, disable_rewards: true)` at block N.
3. `general_prerequisites` passes (contract not paused, caller non-zero). [5](#0-4) 
4. `current_block_number > last_reward_block` passes (first call this block).
5. `last_reward_block` is written to N. [6](#0-5) 
6. `disable_rewards == true` → function returns early; zero rewards distributed. [7](#0-6) 
7. The legitimate staker (or consensus layer) calls `update_rewards(staker, false)` at block N → reverts with `REWARDS_ALREADY_UPDATED`.
8. Attacker repeats at block N+1, N+2, … — all stakers receive zero consensus block rewards indefinitely.
