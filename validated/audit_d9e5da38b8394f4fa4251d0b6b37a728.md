### Title
Missing Validation of `app_pk` in `ckd()` Allows Caller to Bypass ElGamal Encryption and Extract Confidential Derived Key in Plaintext - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts the application's ElGamal public key (`app_pk`) without validating that it is a non-identity point on G1. A malicious caller can supply `app_pk = G1::identity()`, which collapses the ElGamal encryption layer and causes the coordinator's `CKDOutput.big_c` to directly equal `msk · H(pk ‖ app_id)` — the confidential derived key — in plaintext.

---

### Finding Description

The CKD protocol is designed as an ElGamal-encrypted BLS signature scheme. Each participant computes:

```
big_s  = x_i · H(pk ‖ app_id)          // share of BLS signature
big_c  = big_s + app_pk · y             // ElGamal ciphertext
big_y  = y · G                          // blinding point
```

The coordinator aggregates Lagrange-weighted contributions:

```
C = Σ λ_i · big_c_i  =  msk · H(pk ‖ app_id)  +  app_pk · (Σ λ_i · y_i)
Y = Σ λ_i · big_y_i
```

The app recovers the secret as `C − app_sk · Y = msk · H(pk ‖ app_id)`.

The entire confidentiality guarantee rests on `app_pk` being a valid non-identity point. However, `ckd()` performs no such check:

```rust
// src/confidential_key_derivation/protocol.rs  lines 66-116

    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,

    app_pk: PublicKey,          // ← accepted without any validity check
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // checks: participant count, duplicates, self-presence, coordinator-presence
    // NO check: app_pk != G1::identity()
    ...
}
```

Inside `compute_signature_share` (lines 148–182):

```rust
let big_c = big_s + app_pk * y.0;   // line 174
```

When `app_pk = G1::identity()`:

```
app_pk * y  =  identity * y  =  identity
big_c       =  big_s + identity  =  big_s  =  x_i · H(pk ‖ app_id)
```

After aggregation by the coordinator:

```
C = Σ λ_i · x_i · H(pk ‖ app_id)  =  msk · H(pk ‖ app_id)
```

`big_c` in the returned `CKDOutput` is now the raw confidential key with no encryption applied. The `big_y` field is non-zero but irrelevant — calling `unmask(Scalar::ZERO)` or simply reading `big_c` directly yields the secret.

The `PublicKey` type alias is `blstrs::G1Projective`, which freely represents the identity element. No other layer in the call chain rejects it. The `BLS12381G1Group::serialize` method does reject the identity, but `app_pk` is never serialized through that path — it is used directly in scalar multiplication.

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

The confidential key `msk · H(pk ‖ app_id)` is the output that the CKD protocol is specifically designed to keep secret from all parties except the requesting app. With `app_pk = identity`, this value appears verbatim in `CKDOutput.big_c`, which is returned to the coordinator and propagated on-chain. Any observer of the CKD output (on-chain or off-chain) can read the confidential key directly, violating the core secrecy requirement stated in the protocol specification:

> `s` must be deterministic as a function of `app_id` and only known by *app*

---

### Likelihood Explanation

The `app_pk` is an externally supplied input — it is generated by the requesting TEE app and forwarded to the MPC network. A malicious app (or a malicious coordinator substituting a different value) can trivially supply `G1Projective::identity()`. The `blstrs` crate constructs the identity point with `G1Projective::identity()`, which is a valid Rust value that passes through all existing checks in `ckd()` without error. No cryptographic capability is required; this is a pure input-validation omission.

---

### Recommendation

Add an explicit check in `ckd()` that `app_pk` is not the identity element before proceeding:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used elsewhere in the codebase (e.g., `BLS12381G1Group::serialize` rejects the identity, and `robust_ecdsa::sign` rejects `msg_hash == 0` for analogous algebraic collapse reasons).

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
