### Title
Missing `msg_hash == 0` Validation in OT-Based ECDSA Signing Allows Presign Secret Disclosure - (File: src/ecdsa/ot_based_ecdsa/sign.rs)

### Summary
The OT-based ECDSA `sign()` function accepts `msg_hash = 0` without rejection. When `h = 0`, each participant's signature share collapses to `s_i = R_x · σ_i`, directly exposing the individual presign secret shares `σ_i = k⁻¹ · xᵢ` to the coordinator. The robust ECDSA `sign()` function enforces an identical guard, confirming the developers recognise this as a required invariant — but it was never ported to the OT-based path.

### Finding Description
In `src/ecdsa/ot_based_ecdsa/sign.rs`, the public entry-point `sign()` performs several input checks (participant count, threshold bounds, coordinator membership) but contains **no guard against `msg_hash == 0`**:

```rust
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,          // ← zero is accepted silently
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 { … }
    // … coordinator / threshold checks …
    // NO msg_hash.is_zero() guard
    let ctx = Comms::new();
    let fut = fut_wrapper(ctx.shared_channel(), participants, coordinator,
                          me, public_key, presignature, msg_hash);
    Ok(make_protocol(ctx, fut))
}
```

The signing formula for OT-based ECDSA is:

```
s_i = h · k_i  +  R_x · σ_i          (linearised share)
```

where `k_i` is the participant's share of the inverse nonce `k⁻¹` and `σ_i` is their share of `k⁻¹ · x` (the presign secret). When `h = 0` this reduces to:

```
s_i = R_x · σ_i
```

Each participant sends their individual `s_i` privately to the coordinator. Because `R_x` and the Lagrange coefficient `λ_i` are both public, the coordinator can immediately recover every participant's individual presign secret share:

```
σ_i = s_i / (λ_i · R_x)
```

The aggregate `σ = k⁻¹ · x` is also trivially recovered as `s / R_x`.

By contrast, the robust ECDSA `sign()` function in `src/ecdsa/robust_ecdsa/sign.rs` explicitly rejects this input:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
```

The function-level doc comment in the OT-based path even acknowledges the risk ("Allowing the signing of arbitrary scalars *is* a security risk") but provides no enforcement.

### Impact Explanation
**Critical — Extraction/disclosure of presign secrets and nonce material.**

When a signing session is initiated with `msg_hash = 0`, the coordinator receives individual shares `s_i = R_x · σ_i` from every other participant. Since `R_x` and `λ_i` are public, the coordinator recovers each participant's secret share `σ_i = k⁻¹ · xᵢ` and the aggregate `σ = k⁻¹ · x`. These are presign secrets that must never be revealed to any single party. Disclosure of `σ` also enables private-key recovery if the coordinator can correlate it with a second signing session over the same key (e.g., by observing `s₁ = h · k⁻¹ + R_x · σ` for any `h ≠ 0` using a different presignature, yielding `k⁻¹ = (s₁ − R_x · σ) / h` and then `x = σ · k`).

### Likelihood Explanation
**High.** The coordinator role is explicitly part of the threat model (the docs warn "The coordinator must not be able to present different message hashes … to different signers"). A malicious coordinator simply instructs all participants to sign the zero scalar. No cryptographic capability is required; the only barrier is the missing input check. The robust ECDSA path already demonstrates the correct fix, confirming this is an oversight rather than a deliberate design choice.

### Recommendation
Add the same zero-hash guard that exists in the robust ECDSA `sign()` function:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0: signing the zero scalar leaks presign secret shares".to_string(),
    ));
}
```

This should be inserted in `src/ecdsa/ot_based_ecdsa/sign.rs` immediately after the existing participant/threshold checks, before `fut_wrapper` is called.
