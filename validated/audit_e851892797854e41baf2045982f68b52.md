### Title
`DepositAllowlistExtension.beforeAddLiquidity` Checks `owner` Instead of `sender`, Allowing Unauthorized Callers to Bypass the Deposit Allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` silently discards the `sender` argument (the actual caller) and validates only `owner` (the position recipient). Because `MetricOmmPool.addLiquidity` explicitly supports an operator pattern where `msg.sender ≠ owner`, any address that is not on the allowlist can bypass the gate by calling `addLiquidity` with an authorized `owner`.

---

### Finding Description

`MetricOmmPool.addLiquidity` forwards two distinct addresses to the extension hook:

```solidity
// MetricOmmPool.sol L191
_beforeAddLiquidity(msg.sender, owner, salt, deltas, extensionData);
```

`msg.sender` is the actual caller (the depositor paying tokens); `owner` is the position recipient. The pool's own NatSpec confirms the operator pattern is intentional:

> `msg.sender` pays but need not equal `owner` (operator pattern).

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but names it `address` (unnamed, discarded) and checks only `owner`:

```solidity
// DepositAllowlistExtension.sol L32-42
function beforeAddLiquidity(address, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

`msg.sender` inside the extension is the pool (the extension is called by the pool via `CallExtension.callExtension`). The check `allowedDepositor[pool][owner]` therefore asks "is the position owner on the allowlist?" — not "is the depositing caller on the allowlist?". The admin-facing setter reinforces that the intent is to gate the caller:

```solidity
// DepositAllowlistExtension.sol L18
function setAllowedToDeposit(address pool_, address depositor, bool allowed) external onlyPoolAdmin(pool_) {
```

The parameter is named `depositor`, not `owner` or `recipient`.

---

### Impact Explanation

Any address that is not on the allowlist can call:

```
pool.addLiquidity(allowlistedOwner, salt, deltas, callbackData, extensionData)
```

The extension evaluates `allowedDepositor[pool][allowlistedOwner]` → `true`, the deposit proceeds, and the unauthorized caller pays the tokens. The allowlist — the sole access-control mechanism for `addLiquidity` on pools that deploy this extension — is completely bypassed. Pools relying on `DepositAllowlistExtension` for KYC, regulatory, or protocol-level deposit gating cannot enforce those restrictions.

The reverse is also broken: an authorized caller cannot use the operator pattern to deposit on behalf of an address that is not on the allowlist, even when that is the intended use (e.g., a router depositing on behalf of a user who has not been individually allowlisted).

---

### Likelihood Explanation

- Requires no special role or privilege — any EOA or contract can call `addLiquidity`.
- The only precondition is that at least one address is on the allowlist (which is the normal operating state of any pool using this extension).
- The exploit is a single direct call to the pool; no flash loan, sandwich, or multi-step setup is needed.

---

### Recommendation

Rename the first parameter and check `sender` (the actual caller) instead of `owner`:

```solidity
function beforeAddLiquidity(address sender, address owner, uint80, LiquidityDelta calldata, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the design intent is instead to gate by position owner (who may receive a position), the parameter name in `setAllowedToDeposit` should be changed from `depositor` to `owner`, and the NatSpec and error name (`NotAllowedToDeposit`) updated accordingly to avoid the misleading implication that the caller is being gated.

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` wired to `beforeAddLiquidity`.
2. Pool admin calls `setAllowedToDeposit(pool, alice, true)` — only Alice is authorized.
3. Bob (not on the allowlist) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")`.
4. The extension evaluates `allowedDepositor[pool][alice]` → `true` and does **not** revert.
5. Bob's callback pays the tokens; the position is credited to Alice.
6. Bob has successfully deposited into the pool despite not being on the allowlist.

The root cause is identical in structure to the external M-19 report: the authorization check tests the wrong address variable (`owner` / `receiver`) instead of the actual caller (`sender` / `msg.sender`), making the most natural operator-pattern flow bypass the intended gate.
