### Title
Multihop `exactInput` Reverts on Drift-Limited Partial Fill Instead of Proceeding with Actual Output — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`MetricOmmSimpleRouter::exactInput` enforces that each hop consumes **exactly** the full requested input amount. Because the pool can legitimately consume less when the drift limit is reached, this stricter per-hop guard causes the entire multihop swap to revert even though the swap could have succeeded — with the user's `amountOutMinimum` still providing slippage protection.

---

### Finding Description

In `exactInput`, after each pool call the router checks:

```solidity
int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);
``` [1](#0-0) 

`amount` is the full `amountSpecified` passed to the pool. The pool's own documented behavior confirms it can return `amountInActual < amount` when the drift limit is hit: [2](#0-1) [3](#0-2) [4](#0-3) 

The single-hop sibling `exactInputSingle` has **no such check** — it simply takes whatever output the pool returns and validates only against `amountOutMinimum`: [5](#0-4) 

The multihop path therefore imposes a stricter invariant than the user's actual protection (`amountOutMinimum`), creating an inconsistency: the same swap that succeeds as a single hop fails as a multihop when any intermediate pool is drift-limited.

---

### Impact Explanation

Any multihop exact-input swap where at least one pool hits its drift limit will revert with `InvalidInputAmountAtHop`, even though:

1. The pool did produce a valid (non-zero) output.
2. The remaining hops could have been executed with that reduced output as input.
3. The final `amountOutMinimum` guard would have caught any unacceptable slippage.

This breaks the core multihop swap flow for any large or volatile-oracle trade that crosses the drift boundary, making `exactInput` unreliable for the exact market conditions (high volume, fast oracle moves) where multihop routing is most needed.

No funds are lost because the transaction reverts atomically, but the swap is rendered unusable without the user manually splitting it into smaller single-hop calls.

---

### Likelihood Explanation

The drift limit (`MAX_DRIFT_E8 = 5%`) is a hard protocol cap applied per swap. Any single-hop input large enough to move the oracle-anchored price by more than 5% will trigger a partial fill. In a multihop path this is more likely because the user's total input is routed through pools that may each be near their drift boundary. The condition is reachable by any ordinary swapper with no privileged access.

---

### Recommendation

Remove the strict `amountInActual < amount` guard and instead propagate the actual output of each hop as the input to the next hop, mirroring the single-hop design. The existing `amountOutMinimum` check at the end of `exactInput` is sufficient slippage protection:

```solidity
// Remove:
// if (amountInActual < amount) revert InvalidInputAmountAtHop(...);

// Keep as-is:
amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
``` [6](#0-5) 

After the loop, the existing guard already enforces the user's minimum: [7](#0-6) 

---

### Proof of Concept

1. Deploy a two-pool path: `token0 → token1 → token2`.
2. Call `exactInput` with `amountIn = 10_000_000` (large enough to hit the drift limit on pool 0) and `amountOutMinimum = 0`.
3. Pool 0 is called with `amountSpecified = 10_000_000`; the drift limit caps consumption at, say, `8_000_000`, returning `amountInActual = 8_000_000 < 10_000_000`.
4. The router reverts with `InvalidInputAmountAtHop(0, 8_000_000, 10_000_000)`.
5. Calling `exactInputSingle` on pool 0 alone with the same `amountIn` succeeds and returns the partial output — confirming the revert is an artifact of the multihop guard, not a pool-level failure. [8](#0-7) [9](#0-8)

  function exactInput(ExactInputParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    _validatePath(params.tokens, params.pools, params.extensionDatas);

    uint256 last = params.pools.length - 1;
    int128 amount = MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn);

    for (uint256 i = 0; i <= last; i++) {
      address pool = params.pools[i];
      bool zeroForOne = MetricOmmSwapPath.resolveZeroForOneBitmap(params.zeroForOneBitMap, i);

      _setNextCallbackContext(pool, CALLBACK_MODE_JUST_PAY, i == 0 ? msg.sender : address(this), params.tokens[i]);
      (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(pool)
        .swap(
          i == last ? params.recipient : address(this),
          zeroForOne,
          amount,
          MetricOmmSwapPath.openLimit(zeroForOne),
          "",
          params.extensionDatas[i]
        );

      int128 amountInActual = MetricOmmSwapResults.extractAmountIn(zeroForOne, amount0Delta, amount1Delta);
      if (amountInActual < amount) revert InvalidInputAmountAtHop(uint8(i), amountInActual, amount);

      amount = MetricOmmSwapResults.extractAmountOut(zeroForOne, amount0Delta, amount1Delta);
    }

    if (amount <= 0) revert InvalidSwapDeltas();
    amountOut = MetricOmmSwapInputs.int128ToUint128(amount);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();
  }
```
