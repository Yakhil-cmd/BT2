### Title
Malicious CKD Coordinator Has Exclusive, Unverified Control Over Aggregated Output, Enabling Permanent Denial of CKD — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol grants the coordinator exclusive, unverified control over the aggregated `(Y, C)` output. Participants send their secret shares privately to the coordinator and immediately return `Ok(None)` with no ability to verify the coordinator's aggregation. A malicious coordinator can substitute any arbitrary `(Y', C')` for the correct aggregated result, causing the app's BLS signature verification to fail and permanently denying CKD for honest parties.

---

### Finding Description

In `do_ckd_participant`, each non-coordinator participant computes its Lagrange-weighted share `(norm_big_y, norm_big_c)` and sends it **privately** to the coordinator via `chan.send_private`, then immediately returns `Ok(None)`:

```rust
// src/confidential_key_derivation/protocol.rs lines 17-33
fn do_ckd_participant(...) -> Result<CKDOutputOption, ProtocolError> {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)   // ← participant terminates here; no verification of coordinator output
}
```

In `do_ckd_coordinator`, the coordinator aggregates all received shares and is the **sole party** that produces and observes the final `CKDOutput`:

```rust
// src/confidential_key_derivation/protocol.rs lines 35-58
async fn do_ckd_coordinator(...) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))   // ← only the coordinator sees and outputs the result
}
```

The protocol design is structurally identical to the TwapOracle pattern: a single privileged party (the coordinator) has exclusive, unverified control over the critical aggregation step. Just as the TwapOracle owner could selectively call or not call `update` to produce a manipulated price, the CKD coordinator can selectively output correct or incorrect `(Y, C)` values for different app requests, with no recourse for honest participants.

There is no round in which the coordinator broadcasts the aggregated result back to participants for cross-verification. Participants have no cryptographic means to detect that the coordinator substituted `(Y', C')` for the correct `(Y, C)`. [1](#0-0) [2](#0-1)

### Impact Explanation

The app receives `(Y', C')` from the coordinator and computes:

```
sig = C' + (−a) · Y'
```

For any `(Y', C')` not satisfying `C' = msk · H(pk, app_id) + a · Y'`, the BLS pairing check:

```
e(sig, G2) ≠ e(H(pk, app_id), pk)
```

fails. The app cannot derive its deterministic secret key `s = HKDF(sig)`. Because the coordinator is the only party that outputs `(Y, C)` and participants have no verification mechanism, there is no recovery path. The malicious coordinator can permanently deny CKD for any targeted app by outputting incorrect aggregated values.

This matches: **High — Permanent denial of CKD for honest parties under valid protocol inputs.** [3](#0-2) [4](#0-3)

### Likelihood Explanation

The coordinator is a regular participant designated by the caller at protocol invocation time:

```rust
// src/confidential_key_derivation/protocol.rs lines 66-101
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    ...
```

Any participant can be designated coordinator. A malicious participant who is assigned the coordinator role can exploit this without any additional privilege. The attack requires no leaked keys, no cryptographic breaks, and no external assumptions — only the ability to be designated coordinator and to output an incorrect group element pair. [5](#0-4) 

---

### Recommendation

Add a verification round in which the coordinator broadcasts the aggregated `(Y, C)` back to all participants before the protocol terminates. Each participant can verify their own contribution is correctly included using a pairing-based consistency check:

```
e(λ_i · C_i, G2) == e(λ_i · S_i + λ_i · y_i · A, G2)
```

Alternatively, require participants to commit to their shares (e.g., via a hash commitment broadcast) before sending them privately to the coordinator, and have the coordinator publish all individual contributions alongside the aggregate so participants can verify the sum.

### Proof of Concept

1. Participants P1 (coordinator), P2, P3 invoke `ckd(...)` for `app_id = "TargetApp"`.
2. P2 computes `(λ_2 · Y_2, λ_2 · C_2)` and sends it privately to P1 via `chan.send_private`.
3. P3 computes `(λ_3 · Y_3, λ_3 · C_3)` and sends it privately to P1 via `chan.send_private`.
4. P2 and P3 both return `Ok(None)` — they are done and cannot observe P1's output.
5. P1 (malicious) ignores the received shares and instead calls `CKDOutput::new(G1::generator(), G1::generator())` — outputting `(G, G)`.
6. The app receives `(Y', C') = (G, G)` and computes `sig = G + (−a) · G = (1 − a) · G`.
7. BLS verification: `e((1−a)·G, G2) ≠ e(H(pk, app_id), pk)` — fails.
8. The app cannot derive its key. P1 can repeat this for every CKD request, permanently denying CKD for all apps. [6](#0-5) [7](#0-6)
