### Title
`uint32` Truncation in `_afterTimelock` Allows Pool Admin to Bypass Stop-Loss Timelock Immediately — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`_afterTimelock` casts `block.timestamp + timelock` (computed in `uint256`) to `uint32`. When the stored `timelock` is large enough that the sum exceeds `type(uint32).max`, the result silently wraps to a value smaller than `block.timestamp`, making every subsequent `_requireElapsed` check pass instantly. A pool admin can exploit this to bypass the LP-protection timelock on all three timelocked parameters (drawdown, decay, high watermarks) in a single atomic sequence.

---

### Finding Description

`_afterTimelock` is the sole source of `executeAfter` timestamps for every timelocked mutation in `OracleValueStopLossExtension`:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `oracleStopLossConfig[pool_].timelock` is `uint32`. The addition is performed in `uint256` (no revert in Solidity 0.8+), but the result is then **silently truncated** to `uint32`. Whenever `block.timestamp + timelock > type(uint32).max` (= 4,294,967,295), the stored `executeAfter` wraps to a value strictly less than `block.timestamp`.

`_requireElapsed` then checks:

```solidity
// L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

Because the wrapped `executeAfter` is already in the past, the condition is false and the revert never fires — the timelock is completely bypassed.

There is **no validation** on the timelock value anywhere: `initialize` decodes it from `abi.encode` without bounds-checking, and `proposeOracleStopLossTimelock` accepts any `uint32` without a `_validateTimelock` guard (contrast with `_validateDrawdown` and `_validateDecay` which do exist).

---

### Impact Explanation

The `OracleValueStopLossExtension` is the only on-chain mechanism that prevents a pool admin from immediately weakening or disabling the per-bin value stop-loss that LPs rely on. Once the timelock is bypassed, the admin can in the same block:

1. Set `drawdownE6 = 1_000_000` (100% — stop-loss never triggers).
2. Set `decayPerSecondE8 = 1e8` (maximum — watermarks decay to zero in one second).
3. Set all bin high watermarks to zero (stop-loss floor becomes zero).

With the stop-loss disabled, a colluding or malicious admin can execute (or allow) swaps that drain LP principal from every bin, with no on-chain guard remaining. This is a direct loss of user principal above Sherlock thresholds.

---

### Likelihood Explanation

The pool admin is semi-trusted and controls the timelock value. The exploit requires only two transactions:

1. `proposeOracleStopLossTimelock(pool, type(uint32).max)` — if the current timelock is 0 (a valid and common initial value), `executeAfter` = `uint32(block.timestamp + 0)` = `block.timestamp`, which `_requireElapsed` passes immediately.
2. `executeOracleStopLossTimelock(pool)` — sets `timelock = type(uint32).max`.

After step 2, every future `_afterTimelock` call wraps and all subsequent proposals execute instantly. No waiting, no off-chain coordination, no special token balance required. Any pool initialized with `timelock = 0` is exploitable in a single block.

---

### Recommendation

Replace the unsafe `uint32` cast with a safe computation that reverts on overflow, and add an explicit cap on the timelock value:

```solidity
uint256 private constant MAX_TIMELOCK = 365 days; // example cap

function _validateTimelock(uint256 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}

function _afterTimelock(address pool_) private view returns (uint256) {
    return block.timestamp + oracleStopLossConfig[pool_].timelock;
}
```

Store and compare `executeAfter` as `uint256` throughout `PoolStopLossSchedule` and `PendingHighWatermarks`, or at minimum assert `block.timestamp + timelock <= type(uint32).max` before casting.
