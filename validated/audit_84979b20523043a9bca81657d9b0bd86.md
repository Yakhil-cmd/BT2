### Title
Unprivileged Caller Can Permanently Suppress Consensus Block Rewards via `disable_rewards=true` in `update_rewards` — (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as "Only starkware sequencer" but carries no on-chain caller restriction. Any address can call it with `disable_rewards=true`, which unconditionally advances the global `last_reward_block` sentinel without distributing rewards. Because the sentinel is global and the guard is `current_block_number > last_reward_block`, a single attacker call per block permanently forfeits that block's consensus rewards for every staker in the protocol.

---

### Finding Description

**Root cause — missing caller guard**

`update_rewards` is the sole entry point for distributing per-block consensus rewards. The spec explicitly states its access control is "Only starkware sequencer." The implementation, however, calls only `general_prerequisites()` (a pause/contract-state check) and then proceeds with no identity check on `get_caller_address()`. [1](#0-0) 

**Condition that gates reward distribution**

After validating the staker and updating the sentinel, the function branches on an attacker-controlled boolean:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // ← sentinel always advanced

if disable_rewards || self.is_pre_consensus() {
    return;                                            // ← rewards silently skipped
}
``` [2](#0-1) 

When `disable_rewards = true`, `last_reward_block` is written to `current_block_number` and the function returns before reaching `_update_rewards`. The sentinel is now consumed for this block.

**Global sentinel, not per-staker**

`last_reward_block` is a single contract-wide storage slot. One call with any valid `staker_address` and `disable_rewards=true` exhausts the reward slot for the entire block for every staker. [3](#0-2) 

**Exploit path**

1. Attacker monitors the mempool (or simply submits a transaction at the start of each block).
2. Attacker calls `update_rewards(any_active_staker, disable_rewards: true)`.
3. `last_reward_block` is set to the current block number.
4. The legitimate sequencer call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers lose their consensus block rewards for that block, permanently.
6. Attacker repeats every block.

**Spec vs. implementation mismatch confirmed by tests**

The test suite calls `update_rewards` without any `cheat_caller_address_once` setup, confirming no caller restriction exists in the deployed code: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Each block's consensus rewards are computed once and distributed once. If `last_reward_block` is advanced without distribution, those rewards are never minted or credited; there is no catch-up mechanism. A sustained attack (one cheap transaction per block) permanently zeroes all consensus-phase staker and delegator rewards across the entire protocol. This directly matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**High.** The function is publicly callable with no privilege requirement. The attack costs one transaction per block (gas only, no stake required). It is front-runnable and requires no special knowledge beyond knowing any active staker address (which is public on-chain). The attacker has no financial incentive but can inflict maximum damage at minimal cost (griefing / competitive sabotage).

---

### Recommendation

Add an explicit sequencer-only guard at the top of `update_rewards`, consistent with the spec. For example, store the authorized sequencer address in contract storage and assert `get_caller_address() == self.sequencer_address.read()` before any state mutation. Alternatively, if the function is intentionally public, remove the `disable_rewards` parameter and always distribute rewards, letting the sequencer control timing through transaction inclusion rather than a boolean flag.

---

### Proof of Concept

```
Block N:
  Attacker tx:   update_rewards(staker=<any_active_staker>, disable_rewards=true)
                 → last_reward_block := N
                 → returns early, zero rewards distributed

  Sequencer tx:  update_rewards(staker=<staker>, disable_rewards=false)
                 → assert!(N > N)  ← REWARDS_ALREADY_UPDATED, reverts

Result: All stakers earn 0 rewards for block N.

Repeat every block → all consensus rewards permanently suppressed.
```

The attacker needs only:
- Any valid active `staker_address` (readable from public events/storage).
- Enough gas to submit one transaction per block.
- No stake, no privileged role, no key compromise.
