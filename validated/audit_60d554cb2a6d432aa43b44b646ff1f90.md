### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace. Because any delegator can grow this trace by one entry per epoch (via `add_to_delegation_pool` or `exit_delegation_pool_intent`), a sufficiently active delegator who defers claiming will eventually cause `claim_rewards` to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description
`calculate_rewards` in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance` / `increase_member_balance`, both of which call `trace.insert(key: self.get_epoch_plus_k(), ...)`. The underlying `insert` implementation appends a new checkpoint whenever the key differs from the last stored key:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Because the key is always `current_epoch + K`, one new checkpoint is appended per epoch in which the delegator modifies their balance. There is no cap on the number of checkpoints.

`entry_to_claim_from` is only advanced on a successful `claim_rewards` call: [3](#0-2) 

If a delegator modifies their balance every epoch but defers claiming, `entry_to_claim_from` remains at 0 and the loop must traverse the entire trace on the next `claim_rewards` call. Each iteration performs at least one storage read (`pool_member_trace.at(...)`) and a `find_sigma` call, making the per-iteration cost non-trivial.

The same unbounded `calculate_rewards` call is also triggered by the public view `pool_member_info_v1`: [4](#0-3) 

### Impact Explanation
Once the trace is large enough that a single `claim_rewards` transaction exceeds the Starknet gas limit, the call will always revert. Because `entry_to_claim_from` is only updated inside `claim_rewards`, there is no partial-claim mechanism to drain the trace incrementally. The delegator's accumulated unclaimed yield becomes permanently inaccessible — matching the **High: Permanent freezing of unclaimed yield** impact category.

### Likelihood Explanation
The protocol is designed to run indefinitely. A delegator who adds or removes stake once per epoch (a normal, expected behavior) grows the trace by one entry per epoch. Starknet's per-transaction gas limit bounds the maximum safe trace length. At typical epoch durations (days to weeks) and realistic gas limits, a sufficiently active delegator who does not claim for an extended period (months to years) will cross the threshold. The developers themselves flag this risk in the code comment: *"This loop is unbounded but unlikely to exceed gas limits."* The likelihood is **Medium** — not immediate, but reachable over the protocol's intended lifetime without any privileged access.

### Recommendation
1. **Introduce a partial-claim mechanism**: Allow `claim_rewards` to accept a `max_entries` parameter and advance `entry_to_claim_from` by at most that many steps per call, so rewards can be drained incrementally.
2. **Cap trace growth**: Enforce a maximum number of unclaimed balance-change entries per pool member (e.g., reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if the unclaimed trace depth exceeds a protocol-defined limit), analogous to the `maxSignatures` / `maxBlocksSigned` fix in the referenced report.
3. **Encourage periodic claiming**: Document and enforce (via UI or contract-level incentives) that delegators should claim before their trace depth exceeds a safe threshold.

### Proof of Concept
1. Delegator calls `delegate` to enter a pool.
2. Each epoch, the delegator calls `add_to_delegation_pool` with the minimum allowed amount. Each call in a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`.
3. The delegator never calls `claim_rewards`, so `entry_to_claim_from` remains 0.
4. After `N` epochs, the trace has `N` entries.
5. The delegator (or their reward address) calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates all `N` entries, each requiring a storage read and arithmetic. For sufficiently large `N`, the transaction runs out of gas and reverts.
6. Every subsequent `claim_rewards` attempt also reverts. The delegator's yield is permanently frozen.
