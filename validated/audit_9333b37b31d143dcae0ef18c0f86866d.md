### Title
Missing Lower-Bound Threshold Validation in `assert_sign_inputs` and `presign` Allows Malicious Coordinator to Corrupt Signing Output — (File: src/frost/mod.rs)

---

### Summary

`assert_key_invariants` (used in keygen) enforces `threshold >= 2`, but the analogous validation functions for signing — `assert_sign_inputs` and `presign` — only check that `threshold <= participants.len()`, omitting the minimum-threshold lower-bound check. A malicious coordinator can invoke signing with `threshold = 1`, causing honest participants to compute signature shares that aggregate into a cryptographically invalid signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` explicitly enforces a minimum threshold of 2: [1](#0-0) 

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

However, `assert_sign_inputs` in `src/frost/mod.rs` only validates the upper bound: [2](#0-1) 

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

There is no corresponding `threshold < 2` guard. The same omission exists in `presign`: [3](#0-2) 

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

Both functions accept `threshold = 1` without error, while keygen would have rejected it.

---

### Impact Explanation

In FROST, the threshold controls how many participants' shares are Lagrange-interpolated to reconstruct the group secret during signing. If `threshold = 1` is supplied at signing time:

- Only one participant's nonce commitment is collected.
- The Lagrange coefficient for a singleton signing set evaluates to 1, so the aggregated signature share equals that participant's raw Shamir share (not the reconstructed secret).
- The coordinator aggregates a single share into a final signature that does not verify against the public key produced by keygen (which was generated under `threshold >= 2`).

Honest participants have expended computation and communication, and the protocol terminates with an **unusable cryptographic output** — a signature that will always fail verification.

**Matched impact:** *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

The coordinator role is explicitly modeled as a potentially adversarial participant in this library (the `coordinator: Participant` argument is caller-supplied). A malicious coordinator can trivially pass `threshold = 1` to the signing entry point. No key material, external compromise, or cryptographic break is required — only a single integer substitution in the call arguments.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` to both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold.value(), min: 2 });
}
```

This should be inserted immediately after the upper-bound check in each function. [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. Run keygen with `n = 3` participants and `threshold = 2`. This succeeds and produces valid key shares.
2. Call `presign` or the signing entry point with the same participants but `threshold = 1`. The validation in `assert_sign_inputs` / `presign` passes because `1 <= 3`.
3. The signing protocol proceeds: only one participant's nonce commitment is collected; the Lagrange coefficient for the singleton set is 1; the coordinator aggregates a single raw Shamir share.
4. The resulting signature fails verification against the public key, because the Shamir share of one participant is not the full secret key.
5. All honest participants have completed their protocol steps and received no usable output. [6](#0-5)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
