### Title
Attacker-Controlled Event Fields Poison Victim Account's Transaction Index — (`module/state_synchronization/indexer/extended/account_transactions.go`)

### Summary

`extractAddresses` blindly extracts every top-level `cadence.Address` field from every event emitted by a transaction and assigns `TransactionRoleInteracted` to those addresses in the account-transaction index. There is no guard that restricts extracted addresses to those that actually authorized, proposed, or paid for the transaction. Any Cadence contract can emit an event with an arbitrary `Address` field, so an attacker can inject a victim's address into the victim's `GetAccountTransactions` result set.

### Finding Description

`extractAddresses` iterates all top-level fields of a `flow.Event` and collects every `cadence.Address` (and `cadence.Optional<Address>`) value it finds: [1](#0-0) 

The caller, `ProcessBlockData`, then calls `addRole(addrRoles, addr, access.TransactionRoleInteracted)` for every address returned, with the only guard being `chain.IsValid(addr)`: [2](#0-1) 

There is no check that the extracted address is the payer, proposer, or one of the authorizers of the transaction. `groupEventsByTxIndex` performs no event-type filtering either — it groups all events indiscriminately: [3](#0-2) 

### Impact Explanation

An attacker deploys a Cadence contract such as:

```cadence
pub contract Poison {
    pub event Fake(victim: Address)
    pub fun attack(victim: Address) { emit Fake(victim: victim) }
}
```

The attacker submits a transaction authorized only by their own account that calls `Poison.attack(victim: <victimAddress>)`. The emitted event carries `victim` as a top-level `cadence.Address` field. When the access node indexes the sealed block, `extractAddresses` returns the victim's address, and `ProcessBlockData` writes an `AccountTransaction` entry for the victim with `Roles: [TransactionRoleInteracted]` and the attacker's `TransactionID`.

From that point on, `GetAccountTransactions` for the victim returns the attacker's transaction, violating the invariant that an account's transaction history only contains transactions it actually authorized or paid for. [4](#0-3) 

The existing test suite even documents this behavior as intentional for legitimate events (e.g., `AccountCreated`, FT deposit), confirming the code path is live and reachable: [5](#0-4) 

### Likelihood Explanation

The exploit requires only:
1. Deploying a Cadence contract (permissionless on Flow mainnet/testnet).
2. Submitting one transaction that emits an event with the victim's address as a field.

No privileged access, no staked node control, and no key compromise is needed. The path is fully reachable through the standard transaction-submission API.

### Recommendation

In `ProcessBlockData`, after calling `extractAddresses`, filter out any address that is already present in `addrRoles` (i.e., already a payer, proposer, or authorizer). Only add `TransactionRoleInteracted` for addresses that appear in **trusted, protocol-defined event types** (e.g., `flow.AccountCreated`, FungibleToken `Deposited`/`Withdrawn`, NFT `Deposited`/`Withdrawn`) whose emitters are known system or core contracts — not arbitrary user-deployed contracts.

Alternatively, restrict event-address extraction to events whose `event.Type` originates from a known, allowlisted contract address (e.g., the core FT/NFT contracts), rejecting events from user-deployed contracts entirely.

### Proof of Concept

1. Deploy `Poison` contract above from attacker account A.
2. Submit a transaction signed only by A that calls `Poison.attack(victim: B)`.
3. Wait for the block to be sealed and indexed by the access node.
4. Call `GetAccountTransactions(address: B)`.
5. Assert: the response contains the attacker's `TransactionID` with `role = Interacted` — even though B never signed, proposed, or paid for that transaction.
