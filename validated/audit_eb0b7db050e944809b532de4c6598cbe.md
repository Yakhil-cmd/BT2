### Title
ECDSA Signature Forgery via Zero-Digest Bypass in secp256r1 Verification — (`crypto/src/secp256r1/verify.rs`)

### Summary

When a caller submits `digest` equal to the secp256r1 group order `n`, `Scalar::reduce_be_bytes` returns `Scalar::ZERO`. This causes `u1 = 0`, which causes `ecmult` to skip the generator term entirely, reducing the verification equation from `u1*G + u2*pk` to just `u2*pk`. An attacker who knows the public key can trivially construct `(r, s)` satisfying this degenerate equation without knowledge of the private key.

### Finding Description

**Step 1 — `reduce_be_bytes(n_bytes)` returns ZERO**

In `scalar64.rs`, `reduce_be_bytes` calls `from_be_bytes_unchecked` then `subtract_modulus`: [1](#0-0) 

`subtract_modulus` computes `overflowing_sub(n, n)` = `(0, false)`. Since `borrow = false`, the add-back branch is not taken, and the result is `[0,0,0,0]` = `Scalar::ZERO`: [2](#0-1) 

The same holds for the Montgomery-form delegation path in `scalar_delegation.rs`: `n * R2 mod n = 0`, so `is_zero()` returns true there too. [3](#0-2) 

**Step 2 — `verify` does not guard against `z = 0`** [4](#0-3) 

`z = Scalar::reduce_be_bytes(digest)` is used directly with no zero-check. `u1 = z * s_inv = 0`.

**Step 3 — `ecmult` skips the generator term when `ng = 0`** [5](#0-4) 

When `ng.is_zero()`, `wnaf_ng` is set to `Wnaf::default()` (bits = -1), so no generator contribution is ever added in the loop. The result is purely `u2 * pk`.

**Step 4 — Forgery construction**

The verification equation degenerates to: does `(r · s⁻¹) · pk` have x-coordinate `≡ r (mod n)`?

Since the attacker knows `pk`, they can:
1. Choose any scalar `k`.
2. Compute `Q = k · pk` (public key scalar multiplication — no private key needed).
3. Set `r = Q.x mod n`.
4. Set `s = r · k⁻¹ mod n`.
5. Set `digest` = the 32-byte big-endian encoding of `n`.

Verification then computes `(r · s⁻¹) · pk = (r · k · r⁻¹) · pk = k · pk = Q`, and `Q.x mod n = r` ✓ — verification returns `true`.

### Impact Explanation

The `p256_verify` system function is a publicly callable precompile (per EIP-7951) with no privileged access requirement: [6](#0-5) 

Any contract that uses P256 signature verification for access control (e.g., passkey-based smart accounts, multisigs, or any contract gating fund transfers on `p256_verify` returning 1) can be bypassed. The attacker needs only the target's public key (which is public by definition) and can forge a valid `(r, s)` pair for `digest = n` in O(1) operations. This enables unauthorized contract execution and direct loss of funds.

### Likelihood Explanation

The attack requires no privileged access, no brute force, and no external oracle. The forged inputs are computable offline in milliseconds. The only precondition is knowing the victim's public key, which is always available on-chain. Likelihood is **high**.

### Recommendation

Add an explicit zero-check on `z` immediately after computing it in `verify`:

```rust
// In crypto/src/secp256r1/verify.rs
let z = Scalar::reduce_be_bytes(digest);
if z.is_zero() {
    return Err(Secp256r1Err::InvalidSignature);
}
```

This mirrors the existing guard on `r` and `s` in `Signature::from_scalars`: [7](#0-6) 

### Proof of Concept

```
1. Let pk = known secp256r1 public key (x_pk, y_pk)
2. Choose k = 2
3. Compute Q = 2 * pk  (standard point doubling, no private key needed)
4. r = Q.x mod n
5. s = r * modinv(2, n) mod n
6. digest = [0xFF,0xFF,0xFF,0xFF,0x00,0x00,0x00,0x00,
             0xFF,0xFF,0xFF,0xFF,0xFF,0xFF,0xFF,0xFF,
             0xBC,0xE6,0xFA,0xAD,0xA7,0x17,0x9E,0x84,
             0xF3,0xB9,0xCA,0xC2,0xFC,0x63,0x25,0x51]  // = n in big-endian
7. Call p256_verify(digest, r, s, x_pk, y_pk) → returns 1 (verified)
```

Unit test: call `crypto::secp256r1::verify(&n_bytes, &r_bytes, &s_bytes, &x_pk, &y_pk)` with the above values and assert it returns `Ok(true)` — this demonstrates the forgery without any private key.
