### Title
`DeleteAccountAction` with `beneficiary_id == account_id` Permanently Burns the Entire Account Balance — (File: `runtime/runtime/src/actions.rs`)

---

### Summary

`action_delete_account` unconditionally emits a `Receipt::new_balance_refund` addressed to `delete_account.beneficiary_id`, then removes the account from state. When an unprivileged user sets `beneficiary_id` equal to the account being deleted, the refund receipt is routed to a now-nonexistent account. Because system-predecessor refund receipts are explicitly ineligible for implicit-account creation, the transfer fails, and the runtime burns the entire account balance instead of delivering it. No validation in the production path rejects `beneficiary_id == account_id`.

---

### Finding Description

**Root cause — missing self-referential guard in `validate_delete_action`**

`validate_delete_action` only checks that `beneficiary_id` is a syntactically valid account ID:

```rust
fn validate_delete_action(action: &DeleteAccountAction) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    Ok(())
}
``` [1](#0-0) 

There is no check that `beneficiary_id != account_id`.

**Execution path — refund receipt sent to the deleted account**

`action_delete_account` first pushes a `new_balance_refund` receipt addressed to `beneficiary_id`, then calls `remove_account`:

```rust
let account_balance = account_ref.amount();
if account_balance > Balance::ZERO {
    result.new_receipts.push(
        Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance)
    );
}
let remove_result = remove_account(state_update, account_id)?;
``` [2](#0-1) 

`Receipt::new_balance_refund` sets `predecessor_id = "system"`: [3](#0-2) 

When `beneficiary_id == account_id`, this receipt is addressed to the account that was just deleted.

**Why the refund receipt cannot recreate the account**

In `apply_action`, the `implicit_account_creation_eligible` flag is set to `false` for any receipt whose predecessor is `"system"`:

```rust
let is_refund = receipt.predecessor_id().is_system();
let is_the_only_action = actions.len() == 1;
let implicit_account_creation_eligible = is_the_only_action && !is_refund;
``` [4](#0-3) 

`check_account_existence` for a `Transfer` action on a nonexistent account with `implicit_account_creation_eligible = false` returns `AccountDoesNotExist` regardless of whether the account ID is implicit or named: [5](#0-4) 

The comment in that function explicitly acknowledges this design: *"Refunds don't automatically create accounts … Account deletion with beneficiary creates a refund, so it'll not create a new account."*

**Tokens are burned when the refund receipt fails**

The runtime's receipt-processing path explicitly burns the deposit when a system-predecessor receipt fails:

```rust
let gas_refund_result = if receipt.predecessor_id().is_system() {
    // If the refund fails tokens are burned.
    if result.result.is_err() {
        stats.balance.other_burnt_amount = safe_add_balance(
            stats.balance.other_burnt_amount,
            total_deposit(&action_receipt.actions())?,
        )?
    }
    GasRefundResult::default()
``` [6](#0-5) 

The protocol specification confirms: *"If the execution of a refund fails, the refund amount is burnt."* [7](#0-6) 

---

### Impact Explanation

The accounting invariant **"DeleteAccount transfers `account.amount()` to `beneficiary_id`"** is broken. When `beneficiary_id == account_id`, the entire liquid balance of the deleted account is permanently burned (`stats.balance.other_burnt_amount`). The transaction returns success to the user; no error is surfaced. The tokens are irrecoverable.

**Scope**: accounting — balance conservation.
**Severity**: High — the full account balance (potentially large) is destroyed with no recovery path and no user-visible error.

---

### Likelihood Explanation

Any unprivileged user can craft a `SignedTransaction` containing `Action::DeleteAccount(DeleteAccountAction { beneficiary_id: own_account_id })`. The field is fully user-controlled, passes the only existing validation (`validate_action_account_id`), and reaches `action_delete_account` unchanged. The scenario arises naturally when a user intends to "send the balance to themselves before deleting" or when a client library defaults `beneficiary_id` to `signer_id` while `receiver_id == signer_id`.

The existing test `test_delete_account_ok` exercises exactly this path (`delete_account(eve_dot_alice_account(), eve_dot_alice_account())`) but asserts only that the account is deleted — it does not verify that the balance was preserved. [8](#0-7) 

---

### Recommendation

Add a self-referential guard in `validate_delete_action` (or at the start of `action_delete_account`):

```rust
fn validate_delete_action(
    action: &DeleteAccountAction,
    account_id: &AccountId,          // pass the receipt receiver
) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    if &action.beneficiary_id == account_id {
        return Err(ActionsValidationError::InvalidBeneficiaryId {
            account_id: action.beneficiary_id.clone(),
        });
    }
    Ok(())
}
```

This mirrors the fix applied to PSP22Wrapper (checking `account != self`) and closes the accounting invariant violation at the earliest possible validation point, before any state mutation occurs.

---

### Proof of Concept

1. Alice owns `alice.near` with a balance of 1 000 NEAR.
2. Alice submits:
   ```
   Transaction {
       signer_id:   "alice.near",
       receiver_id: "alice.near",
       actions: [DeleteAccount { beneficiary_id: "alice.near" }],
   }
   ```
3. `validate_delete_action` passes — `"alice.near"` is a valid account ID.
4. `action_delete_account` executes:
   - Pushes `Receipt::new_balance_refund("alice.near", 1_000 NEAR)` (predecessor = `"system"`).
   - Calls `remove_account(state_update, "alice.near")` — account deleted.
5. The refund receipt is processed in the next block:
   - `is_refund = true` → `implicit_account_creation_eligible = false`.
   - `"alice.near"` does not exist → `AccountDoesNotExist` error.
   - `predecessor_id().is_system() && result.is_err()` → `other_burnt_amount += 1_000 NEAR`.
6. Alice's 1 000 NEAR is permanently burned. The original transaction outcome shows `SuccessValue([])`. [9](#0-8) [6](#0-5)

pub(crate) fn action_delete_account(
    state_update: &mut TrieUpdate,
    account: &mut Option<Account>,
    actor_id: &mut AccountId,
    receipt: &Receipt,
    result: &mut ActionResult,
    account_id: &AccountId,
    delete_account: &DeleteAccountAction,
    config: &RuntimeConfig,
    current_protocol_version: ProtocolVersion,
) -> Result<(), StorageError> {
    let account_ref = account.as_ref().unwrap();
    let account_storage_usage = if ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage
        .enabled(current_protocol_version)
    {
        let contract_storage = get_contract_storage_usage(state_update, account_id, account_ref)?;
        account_ref.storage_usage().saturating_sub(contract_storage)
    } else {
        // Legacy behavior: only subtracts local contract code, misses the
        // global contract identifier overhead.
        let account_storage_usage = account_ref.storage_usage();
        let code_len = get_code_len_or_default(
            state_update,
            account_id.clone(),
            account_ref.local_contract_hash().unwrap_or_default(),
        )?;
        debug_assert!(
            code_len == 0 || account_storage_usage > code_len,
            "account storage usage should be larger than code size. storage usage: {}, code size: {}",
            account_storage_usage,
            code_len
        );
        account_storage_usage.saturating_sub(code_len)
    };
    if account_storage_usage > Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE {
        result.result =
            Err(ActionErrorKind::DeleteAccountWithLargeState { account_id: account_id.clone() }
                .into());
        return Ok(());
    }
    let gas_key_balance_to_burn = compute_gas_key_balance_sum(state_update, account_id)?;
    if gas_key_balance_to_burn > GasKeyInfo::MAX_BALANCE_TO_BURN {
        result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh {
            account_id: account_id.clone(),
            public_key: None,
            balance: gas_key_balance_to_burn,
        }
        .into());
        return Ok(());
    }
    // We use current amount as a pay out to beneficiary.
    let account_balance = account_ref.amount();
    if account_balance > Balance::ZERO {
        result
            .new_receipts
            .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
    }
    let remove_result = remove_account(state_update, account_id)?;
    result.tokens_burnt =
        result.tokens_burnt.checked_add(gas_key_balance_to_burn).ok_or_else(|| {
            StorageError::StorageInconsistentState("tokens_burnt overflow".to_string())
        })?;
    if remove_result.gas_key_nonce_count > 0 {
        let compute = storage_removes_compute(
            &config.wasm_config.ext_costs,
            remove_result.gas_key_nonce_count,
            remove_result.gas_key_nonce_total_key_bytes,
            AccessKey::NONCE_VALUE_LEN * remove_result.gas_key_nonce_count,
        );
        result.compute_usage = safe_add_compute(result.compute_usage, compute).map_err(|_| {
            StorageError::StorageInconsistentState("compute_usage overflow".to_string())
        })?;
    }
    *actor_id = receipt.predecessor_id().clone();
    *account = None;
    Ok(())
}
