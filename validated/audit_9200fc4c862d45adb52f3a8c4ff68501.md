### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees via `setPoolBinAdditionalFees`, Bypassing `maxAdminSpreadFeeE6` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound check, while the analogous `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees to `type(uint16).max = 65535` (6.5535%) on any bin, bypassing the factory's fee-cap system and extracting trader principal beyond the documented admin fee ceiling.

### Finding Description

`setPoolAdminFees` enforces the cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has **no analogous check**:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also has no cap — it stores whatever is passed: [3](#0-2) 

These per-bin fees are then **directly added** to the base spread fee in every swap:

```solidity
// MetricOmmPool.sol:1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same additive pattern applies on the buy side: [5](#0-4) 

### Impact Explanation

A pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`. Every swap through bin 0 now pays `spreadFeeE6 + 65535/1e6` (up to ~6.5535% extra) on both sides. This is a direct extraction of trader principal beyond any cap the protocol enforces. The `maxAdminSpreadFeeE6` hard ceiling (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%) is never consulted for bin additional fees, so the cap system is silently bypassed. [6](#0-5) 

### Likelihood Explanation

The pool admin is explicitly described as **semi-trusted only inside caps**. The scope rules and Smart Audit Pivots call out `setBinAdditionalFees` as a path to check for admin-boundary bypasses. Any pool admin — including one who turns adversarial after deployment — can exploit this immediately with a single transaction, with no timelock or protocol approval required.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, analogous to `setPoolAdminFees`:

```solidity
if (uint256(addFeeBuyE6) + uint256(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
// or per-side: if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxAdminBinFeeE6` cap (a `uint16`-sized sub-cap of `maxAdminSpreadFeeE6`) and enforce it in the factory before calling `setBinAdditionalFees`.

### Proof of Concept

1. Deploy a pool with `adminSpreadFeeE6 = 0` and `maxAdminSpreadFeeE6 = 200_000`.
2. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — succeeds with no revert.
3. Execute a swap through bin 0.
4. Assert effective fee = `spreadFeeE6 + 65535/1e6` ≈ 6.5535% extra, which exceeds `maxAdminSpreadFeeE6` (20% total cap) when combined with any non-trivial base spread, and is entirely unchecked by the factory. [2](#0-1)
