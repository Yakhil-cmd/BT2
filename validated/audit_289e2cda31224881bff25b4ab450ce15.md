### Title
Missing Proof of Correctness for Participant Shares in CKD Protocol Allows Malicious Participant to Corrupt Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function aggregates participant-supplied `CKDOutput` values (`norm_big_y`, `norm_big_c`) without any proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted confidential key derivation output that no honest party can use.

---

### Finding Description

In `do_ckd_coordinator` (lines 35–57), the coordinator receives a `CKDOutput` from every other participant and blindly adds the two group elements together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute:

```
norm_big_y = λ_i · (y_i · G)
norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)
```

where `x_i` is the participant's secret share and `y_i` is a fresh random scalar. The coordinator sums these to obtain `(big_y, big_c)` satisfying `big_c − app_sk · big_y = msk · H(pk, app_id)`. [2](#0-1) 

There is **no proof of correctness** that the received `(norm_big_y, norm_big_c)` was honestly derived from the participant's registered secret share. A malicious participant can send any two group elements — including the identity, a negation of another participant's contribution, or a completely unrelated point — and the coordinator will incorporate them without complaint.

Contrast this with the DKG protocol in `do_keyshare`, which validates every received share against a polynomial commitment and verifies a Schnorr proof of knowledge before accepting any contribution:

```rust
validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
``` [3](#0-2) 

```rust
verify_proof_of_knowledge(
    &session_id,
    &mut proof_domain_separator.clone(),
    threshold,
    p,
    old_participants.clone(),
    commitment_i,
    proof_i.as_ref(),
)?;
``` [4](#0-3) 

The CKD protocol has no analogous check. This is the direct structural parallel to the external report: `bigBang` and `singularity` addresses are whitelisted while `marketHelper` is not; here, DKG shares are cryptographically validated while CKD shares are not.

---

### Impact Explanation

The CKD output `(big_y, big_c)` is used by the application to unmask the confidential derived key via `big_c − app_sk · big_y`. If any participant sends malformed values, this relationship no longer holds and the coordinator outputs a garbage confidential key. Honest parties who rely on this output — e.g., to derive a client's secret key share in a TEE-based MPC network — receive an unusable or silently wrong cryptographic result with no indication of which participant misbehaved.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.** [5](#0-4) 

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. The attacker-controlled entry path is direct: the malicious participant simply deviates from the protocol in `do_ckd_participant` by sending arbitrary bytes that deserialize as valid group elements instead of the honestly computed `(norm_big_y, norm_big_c)`. No special privilege, leaked key, or external assumption is required — only participation in the protocol. [6](#0-5) 

---

### Recommendation

Add a zero-knowledge proof of correct computation for each participant's `(norm_big_y, norm_big_c)` contribution, analogous to the Schnorr proof of knowledge used in DKG. Concretely, each participant should prove in zero knowledge that:

- `norm_big_y = λ_i · y_i · G` for some `y_i` they know, and  
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` for the same `y_i` and their committed secret share `x_i`.

The coordinator must verify these proofs before aggregating contributions, and abort (identifying the culprit) if any proof fails.

---

### Proof of Concept

1. Honest participants P_1, …, P_{n−1} run `do_ckd_participant` correctly, sending valid `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant P_m sends `norm_big_y_m = G` (the generator) and `norm_big_c_m = G` (or any arbitrary point unrelated to their secret share `x_m`).
3. The coordinator executes:
   ```rust
   norm_big_y += participant_output.big_y();  // adds G instead of λ_m·y_m·G
   norm_big_c += participant_output.big_c();  // adds G instead of λ_m·(x_m·H+y_m·app_pk)
   ```
4. The final `ckd_output = CKDOutput::new(norm_big_y, norm_big_c)` is returned to the application.
5. When the application calls `ckd_output.unmask(app_sk)`, it computes `big_c − app_sk · big_y`, which no longer equals `msk · H(pk, app_id)`. The derived confidential key is silently wrong. [7](#0-6)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
