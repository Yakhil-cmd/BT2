### Title
Missing Lower-Bound Threshold Check in FROST `assert_sign_inputs` and `presign` Allows Signing with `threshold = 1`, Corrupting Signing Outputs — (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that `threshold <= participants.len()` but omit the symmetric lower-bound guard `threshold >= 2`. Every other initialization path in the library enforces both bounds. A caller (or malicious coordinator) can pass `threshold = 1`, which is silently accepted, causing the FROST signing protocol to run with a single participant's share and produce a cryptographically invalid, unusable signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces two threshold bounds:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`validate_triple_inputs` in the OT-based ECDSA triple generation enforces the same pair of bounds: [2](#0-1) 

By contrast, `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

There is no corresponding `threshold < 2` guard. The same omission exists in the FROST `presign` initializer: [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` newtype with no internal validation, so any value including `1` or `0` is accepted without error: [5](#0-4) 

---

### Impact Explanation

When `threshold = 1` is passed to `assert_sign_inputs`, the function returns `Ok(participants)`. The FROST signing protocol then proceeds collecting only one partial signature. In FROST, the aggregated signature `(R, z)` is valid only when:

```
z * G = R + challenge * public_key
```

With a single signer `i` and Lagrange coefficient `λ_i = 1` (degenerate interpolation over one point):

```
z = nonce_i + share_i * challenge
z * G = R_i + verifying_share_i * challenge
```

This equals `R + challenge * public_key` only if `verifying_share_i == public_key`, which requires the DKG threshold to have been `1`. Since `assert_key_invariants` enforces `threshold >= 2` at DKG time, this condition is never satisfied. The signing protocol completes but outputs a signature that fails verification — honest parties receive a corrupt, unusable signing output.

**Allowed impact matched:** *High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

`assert_sign_inputs` is a public library API. Any library caller — including an unprivileged participant or a malicious coordinator — can supply `threshold = 1` (e.g., an off-by-one error, a misconfiguration, or a deliberate attack). The type system provides no protection: `1usize` converts to `ReconstructionLowerBound(1)` via the derived `From` impl with no rejection. The DKG path would have caught this, but the signing validation path does not, creating an inconsistency that is easy to trigger accidentally or deliberately.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately before or after the existing upper-bound check in both functions, mirroring the pattern already established in `src/dkg.rs` and `src/ecdsa/ot_based_ecdsa/triples/generation.rs`.

---

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (accepted, since `2 >= 2`).
2. Call `assert_sign_inputs` with the same 3 participants but `threshold = 1`.
3. Observe: the function returns `Ok(participants)` — no error.
4. Proceed with FROST signing using `threshold = 1`; only one partial signature is collected.
5. The coordinator aggregates and outputs `(R, z)`.
6. Verify the signature against the public key produced in step 1 — verification fails, confirming the output is corrupt and unusable.

The root cause is the missing guard at: [3](#0-2) 
compared to the correct dual-bound enforcement at: [1](#0-0)
