### Title
Signature Validation Bypass via Overly Broad `skip_stateful_validations` Trigger — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, designed to allow a `deploy_account` + `invoke` pair to be submitted simultaneously, uses `account_tx_in_pool_or_recent_block` as its guard. That check returns `true` for **any** transaction in the pool for the target address — not only a `deploy_account`. An unprivileged attacker can therefore front-run the guard by inserting a cheap invoke(nonce=0) for any address, then submit a second invoke(nonce=1) with a **forged signature** for that same address. The gateway skips the `__validate__` entry-point entirely and admits the forged transaction to the mempool.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after nonce and resource-bound checks pass:

```
run_pre_validation_checks
  └─ validate_state_preconditions   (nonce, resource bounds)
  └─ validate_by_mempool            (duplicate / nonce-too-old)
  └─ skip_stateful_validations      ← decides whether __validate__ runs
``` [1](#0-0) 

The skip fires when all three conditions hold:

1. Transaction is `Invoke`
2. `tx.nonce() == 1`
3. `account_nonce == 0`
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [2](#0-1) 

The guard is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool for that address — not only `DeployAccount`. [4](#0-3) 

**Nonce validation for invoke(nonce=0) on an undeployed account:**

```rust
_ => {
    let max_allowed_nonce =
        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
    if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
        return Err(...)
    }
}
``` [5](#0-4) 

With `account_nonce = 0` and `tx_nonce = 0`, the condition `0 ≤ 0 ≤ max_allowed_nonce` is satisfied, so an invoke with nonce=0 for an undeployed account is admitted to the mempool.

Once that transaction is in the pool, `account_tx_in_pool_or_recent_block` returns `true`, and any subsequent invoke(nonce=1) for the same address — regardless of signature — skips `__validate__`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [6](#0-5) 

The `StatefulValidator::perform_validations` path for `Invoke` then short-circuits before calling `validate_tx`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [7](#0-6) 

### Impact Explanation

An unprivileged attacker can cause the gateway to admit an `Invoke` transaction carrying an **arbitrary/forged signature** for any undeployed account address. The transaction bypasses the account contract's `__validate__` entry point entirely at the gateway layer. The admitted transaction will eventually fail at execution time (the account does not exist), but it has been accepted into the mempool and may be included in a block, consuming block space and sequencer resources. More critically, the invariant "every admitted invoke transaction has passed its account's signature verification" is broken.

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The attack requires only two sequential RPC calls (`add_transaction` twice) with no privileged access, no special funds, and no timing dependency. The first call (invoke nonce=0) is always accepted for any undeployed address. The second call (invoke nonce=1, forged signature) is accepted whenever the first is still in the pool. The window is open for as long as the first transaction remains unprocessed.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction exists in the pool for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, rather than returning `true` for any transaction type.

Alternatively, the gateway can inspect the `InternalRpcTransactionWithoutTxHash` variant of the pooled transaction before granting the skip.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
