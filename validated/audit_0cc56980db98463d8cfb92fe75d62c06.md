### Title
Missing Zero-Address Validation in `change_reward_address` Permanently Burns Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both `Staking::change_reward_address` and `Pool::change_reward_address` accept the zero address as a valid new reward address. A staker or pool member who sets their reward address to `0` will have all future reward transfers sent to the zero address, permanently destroying their unclaimed yield with no recovery path.

### Finding Description
`Staking::change_reward_address` (staking.cairo:517–540) validates only that the new address is not a registered token address, but performs no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No assert!(reward_address.is_non_zero(), ...)
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;   // zero address accepted
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

The same pattern exists in `Pool::change_reward_address` (pool.cairo:505–526), which only checks the token address:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address check
    ...
    pool_member_info.reward_address = reward_address;   // zero address accepted
    ...
}
``` [2](#0-1) 

When rewards are later claimed, `send_rewards_to_staker` unconditionally transfers to `staker_info.reward_address`:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [3](#0-2) 

And `Pool::claim_rewards` does the same:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

There is no mechanism to recover funds sent to the zero address on Starknet.

### Impact Explanation
Any staker or pool member who calls `change_reward_address(0)` — whether by mistake, UI bug, or malicious self-harm — will have every subsequent reward transfer permanently sent to the zero address. The staker's `unclaimed_rewards_own` is zeroed out after the transfer, so the loss is irreversible. This constitutes **permanent freezing/destruction of unclaimed yield**, a High-severity impact under the allowed scope.

### Likelihood Explanation
The call is permissionless for any registered staker or pool member. A single erroneous transaction (e.g., a front-end passing an uninitialized address, a scripting error, or a user misunderstanding the field) is sufficient to trigger the loss. No privileged access is required.

### Recommendation
Add a non-zero assertion at the top of both `change_reward_address` implementations:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

Apply the same guard to the initial `stake` and `enter_delegation_pool` entry points for defense-in-depth.

### Proof of Concept

1. Staker calls `staking.change_reward_address(0)`.
   - The only check (`!does_token_exist(0)`) passes because `0` is not a registered token.
   - `staker_info.reward_address` is written as `0`. [5](#0-4) 

2. At the next attestation epoch, `update_rewards_from_attestation_contract` accumulates `unclaimed_rewards_own` for the staker.

3. Staker (or anyone) calls `claim_rewards` / `unstake_action`. Internally `send_rewards_to_staker` is invoked:
   ```cairo
   let reward_address = staker_info.reward_address; // == 0
   claim_from_reward_supplier(...);
   token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
   // Tokens sent to 0x0 — permanently lost
   staker_info.unclaimed_rewards_own = Zero::zero(); // state cleared, no recovery
   ``` [6](#0-5) 

4. The same sequence applies to a pool member via `Pool::change_reward_address(0)` followed by `Pool::claim_rewards`. [7](#0-6) [8](#0-7)
