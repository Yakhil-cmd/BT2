### Title
Malicious or Offline Coordinator Causes Permanent Denial of Signing for Honest Participants in FROST EdDSA `sign_v1` - (File: `src/frost/eddsa/sign.rs`)

---

### Summary

In the FROST EdDSA `sign_v1` protocol, non-coordinator participants send their Round 1 commitments to the coordinator and then enter an unbounded `loop` waiting for the coordinator to return the signing package. There is no timeout, abort signal, or fallback path. If the coordinator is offline, crashes, or deliberately withholds the signing package, every non-coordinator participant is permanently suspended at that wait point. The signing session can never complete or self-terminate, constituting permanent denial of signing for all honest parties.

---

### Finding Description

The FROST EdDSA two-round signing protocol (`sign_v1`) assigns the coordinator a mandatory relay role between Round 1 and Round 2. After each non-coordinator participant sends its commitment to the coordinator, it blocks unconditionally waiting for the coordinator to broadcast the `SigningPackage`.

In `do_sign_participant_v1` (`src/frost/eddsa/sign.rs`, lines 272–280):

```rust
let r2_wait_point = chan.next_waitpoint();
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;
    if from != coordinator {
        continue;
    }
    break signing_package;
};

`chan.recv(r2_wait_point)` resolves to `MessageBuffer::pop()`, which calls `receiver_lock.next().await` on an unbounded async channel:

```rust

    receiver_lock.next().await.expect("Reference to sender held")

``` [2](#0-1) 

There is no timeout, deadline, or cancellation token anywhere in the `Protocol` trait, `SharedChannel`, or `ProtocolExecutor`. The `Protocol` trait exposes only `poke()` and `message()` — no `abort()` or `cancel()`: [3](#0-2) 

The `ProtocolExecutor` will return `Action::Wait` indefinitely once the future is suspended at the `recv` await point, with no mechanism to break out: [4](#0-3) 

The coordinator is a single, statically-designated `Participant` passed at protocol initialization. There is no coordinator election, rotation, or replacement mechanism: [5](#0-4) 

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

If the coordinator participant is offline, crashes after receiving Round 1 commitments, or deliberately withholds the `SigningPackage`, every non-coordinator participant is permanently suspended at `r2_wait_point`. The signing session never produces a signature and never self-terminates. Because the `Protocol` trait provides no abort path, the caller has no in-protocol way to recover. A new signing session must be started from scratch with a different coordinator, but the library provides no mechanism to elect or substitute a coordinator mid-session. Any application relying on this signing session to produce a threshold signature is permanently blocked.

---

### Likelihood Explanation

The coordinator is a single participant. Any of the following realistic conditions triggers the denial:

- The coordinator node crashes or loses network connectivity after receiving Round 1 commitments but before broadcasting the `SigningPackage`.
- The coordinator is a Byzantine participant who intentionally withholds the `SigningPackage` to block a specific signing request.
- The coordinator is selectively censoring: it sends the `SigningPackage` to some participants but not others, causing a subset to hang indefinitely.

Because the coordinator is a required relay in `sign_v1` (participants send commitments only to the coordinator, not to each other), there is no peer-to-peer fallback. The condition is reachable by any participant who is assigned the coordinator role. [6](#0-5) 

---

### Recommendation

1. **Session timeout at the protocol layer:** Add a deadline or timeout to `MessageBuffer::pop` and expose a cancellation mechanism through the `Protocol` trait (e.g., an `abort()` method or a `tokio::time::timeout` wrapper at the executor level).
2. **Coordinator-free Round 2:** Allow participants to exchange commitments peer-to-peer (via `send_many`) rather than routing through the coordinator, eliminating the coordinator as a single point of failure for Round 2 progress.
3. **Coordinator rotation:** If a coordinator-relay design is retained, define a protocol-level timeout after which participants can elect a new coordinator and restart the round.

---

### Proof of Concept

**Setup:** 3 participants, threshold = 2. Participant 0 is the coordinator.

1. All participants call `sign_v1(...)`. Each non-coordinator participant executes `do_sign_participant_v1`.
2. Each non-coordinator sends its Round 1 commitment to the coordinator via `chan.send_private(commit_waitpoint, coordinator, &commitments)`. [7](#0-6) 

3. The coordinator receives all commitments but then goes offline (or a Byzantine coordinator simply never calls `chan.send_many(r2_wait_point, &signing_package)`). [8](#0-7) 

4. Each non-coordinator participant is now suspended at:

```rust
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;   // <-- blocks forever
    if from != coordinator { continue; }
    break signing_package;
};
``` [9](#0-8) 

5. The caller's `poke()` loop returns `Action::Wait` indefinitely. No error is ever returned. No signature is ever produced. The signing session is permanently stalled with no in-library recovery path. [10](#0-9)

    fn poke(&mut self) -> Result<Action<Self::Output>, ProtocolError> {
        let mut polled_once_already = false;
        loop {
            // If there's outgoing messages, request to send them.
            if let Some(outgoing) = self.comms.outgoing() {
                return Ok(match outgoing {
                    Message::Many(m) => Action::SendMany(m),
                    Message::Private(to, m) => Action::SendPrivate(to, m),
                });
            }
            // If we already have a return result, return it.
            if let Some(result) = self.result.take() {
                return Ok(Action::Return(result?));
            }
            // If this is the second iteration, we already polled the future and there's no
            // progress that can be made.
            if polled_once_already {
                return Ok(Action::Wait);
            }
            // If we don't have a future, this is an extraneous poke() call, so return Wait.
            let Some(fut) = self.fut.as_mut() else {
                return Ok(Action::Wait);
            };
            // Now poll the future. It may generate some more messages to send or a return value,
            // so go back and check all of those again.
            polled_once_already = true;
            let waker = noop_waker();
            let mut cx = Context::from_waker(&waker);
            if let std::task::Poll::Ready(result) = fut.poll_unpin(&mut cx) {
                self.result = Some(result);
                self.fut = None;
            }
        }

    type Output;

    /// The idea is that the protocol should be poked until it returns an error,
