### Title
Presignature Reuse Enabled by `Clone` Derivation and By-Reference API Allows Private Key Recovery — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both `PresignOutput` types (OT-based and robust ECDSA) publicly derive `Clone`, and both `RerandomizedPresignOutput::rerandomize_presign` functions accept the presignature by shared reference (`&PresignOutput`). The library provides no mechanism to mark a presignature as consumed. A malicious coordinator can therefore rerandomize and submit the same presignature for two distinct signing sessions, producing two signatures whose nonces are multiplicatively related. This is the classical ECDSA nonce-reuse condition, from which the full private key can be algebraically recovered.

---

### Finding Description

The OT-based ECDSA `PresignOutput` is declared as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub k: Scalar, pub sigma: Scalar }
``` [1](#0-0) 

The robust ECDSA `PresignOutput` is identically `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub c: Scalar, pub e: Scalar, pub alpha: Scalar, pub beta: Scalar }
``` [2](#0-1) 

Both `rerandomize_presign` implementations accept the presignature by shared reference, not by value:

```rust
// OT-based
pub fn rerandomize_presign(presignature: &PresignOutput, args: &RerandomizationArguments) -> Result<Self, ProtocolError>
// Robust
pub fn rerandomize_presign(presignature: &PresignOutput, args: &RerandomizationArguments) -> Result<Self, ProtocolError>
``` [3](#0-2) [4](#0-3) 

The library's own documentation acknowledges the catastrophic consequence of reuse:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [5](#0-4) 

The orchestration documentation similarly marks presignature outputs as one-time-use red boxes and states it is "**critical** that the output is then destroyed," but no enforcement exists in code: [6](#0-5) 

The analog to the token-gated-drop bug is exact: just as the same NFT token could be transferred between wallets and reused to drain a drop because `mintAllowedTokenHolder` never recorded the token as redeemed, the same `PresignOutput` can be `.clone()`-d and passed to `rerandomize_presign` a second time because the library never records the presignature as consumed. In both cases the "one-time credential" is structurally copyable and the consuming function accepts it by reference, leaving enforcement entirely to the caller with no in-library guard.

---

### Impact Explanation

Two calls to `rerandomize_presign` on the same `PresignOutput` with different `(h, ε)` pairs produce two `RerandomizedPresignOutput` values whose underlying nonce `k` is scaled by different `δ` values (`k·δ₁⁻¹` and `k·δ₂⁻¹`). The resulting on-chain signatures share a multiplicatively related nonce. Standard ECDSA lattice/nonce-reuse recovery then yields the full aggregate private key `x`. This satisfies the **Critical** impact tier: *Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, or nonce material.*

---

### Likelihood Explanation

The coordinator role is explicitly untrusted in the library's own security model ("The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers"). A malicious coordinator controls which `RerandomizedPresignOutput` each participant receives during the signing phase; participants have no independent way to detect that the underlying `big_r` was already used in a prior session. Because `PresignOutput` is `Clone + Serialize + Deserialize`, the coordinator can trivially persist and replay it. No out-of-band coordination among honest participants is required to detect the reuse.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput`** in both `src/ecdsa/ot_based_ecdsa/mod.rs` and `src/ecdsa/robust_ecdsa/mod.rs`. A one-time-use secret should not be freely copyable.
2. **Change `rerandomize_presign` to consume the presignature by value** (`presignature: PresignOutput`) so that the Rust ownership system statically prevents a second call on the same value.
3. **Optionally zeroize on move** by implementing a consuming wrapper that calls `zeroize` after the single permitted use, mirroring the "destroy after use" requirement stated in the orchestration docs.
4. Document in the `presign` function docstrings (both schemes) that the returned `PresignOutput` must be used exactly once and must be securely erased after the signing session completes or aborts.

---

### Proof of Concept

```
// Attacker is the coordinator.
// Honest presigning has already completed; coordinator holds presig_shares.

let presig: PresignOutput = presig_shares[coordinator_idx].clone(); // Clone is public

// Session 1 — legitimate
let args1 = RerandomizationArguments::new(pk, tweak, h1_bytes, presig.big_r, participants.clone(), entropy1);
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presig, &args1).unwrap();
// ... run sign protocol → (R1, s1)

// Session 2 — malicious reuse of the SAME presig with a different message hash
let args2 = RerandomizationArguments::new(pk, tweak, h2_bytes, presig.big_r, participants.clone(), entropy2);
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presig, &args2).unwrap();
// ... run sign protocol → (R2, s2)

// R1 = δ1·R,  R2 = δ2·R  (same underlying nonce k)
// s1 = h1·(k·δ1⁻¹) + r1·σ·δ1⁻¹
// s2 = h2·(k·δ2⁻¹) + r2·σ·δ2⁻¹
// Nonces are multiplicatively related → standard ECDSA nonce-reuse algebra recovers x.
```

The `presig` variable is never consumed; `rerandomize_presign` accepts `&PresignOutput`, so the second call compiles and executes without any library-level error.
