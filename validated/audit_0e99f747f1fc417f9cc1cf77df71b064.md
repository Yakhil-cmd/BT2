### Title
Unauthorized Invoke Transaction Bypasses Signature Verification via `skip_stateful_validations` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the account's `__validate__` entry point for invoke transactions with nonce=1 when the account has any transaction in the mempool. Because the check does not verify that the submitter is the account owner, any attacker can submit an invoke transaction with an invalid signature for any account that has a pending `deploy_account` transaction, and the gateway will admit it to the mempool without signature verification.

---

### Finding Description

`skip_stateful_validations` is designed to support the UX pattern of submitting `deploy_account` + `invoke` together. It returns `true` (skip validation) when three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (account not yet deployed on-chain). [1](#0-0) 

When all three conditions hold, the function calls `account_tx_in_pool_or_recent_block(tx.sender_address())`. If that returns `true`, it returns `true` — meaning the gateway will skip the `__validate__` entry point entirely. [2](#0-1) 

The `run_validate_entry_point` function then sets `validate: !skip_validate` in `ExecutionFlags`, so when `skip_validate = true`, the blockifier's `StatefulValidator::validate` is called with `validate: false` — the account's `__validate__` function is never invoked and the signature is never checked. [3](#0-2) 

The code comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is circular: any prior future-nonce invoke for this account would have also gone through this same `skip_stateful_validations` path (since `account_nonce == 0` at that time). The check `account_tx_in_pool_or_recent_block` verifies only that the account has **any** transaction in the pool — not that the submitter of the current invoke is the same party who submitted the `deploy_account`. [4](#0-3) 

The gateway's `run_pre_validation_checks` calls `validate_by_mempool` (nonce/fee checks) and then `skip_stateful_validations`, but `validate_by_mempool` does not verify the account signature — that is the account contract's responsibility, which is exactly what is being skipped. [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an invoke transaction with an arbitrary/empty signature for any account that has a pending `deploy_account` in the mempool. The transaction is admitted to the mempool without signature verification. This:

- Occupies the nonce=1 slot for the victim account in the mempool, potentially delaying or displacing the legitimate owner's nonce=1 transaction.
- Enables mempool pollution / DoS against any new account deployment: an attacker monitoring the mempool can immediately submit fake nonce=1 invokes for every observed `deploy_account`, systematically disrupting the deploy+invoke UX pattern for all new accounts.

When the batcher later executes the transaction, `__validate__` will fail and the transaction will be dropped (nonce not consumed), but the damage to mempool ordering and the legitimate user's UX has already occurred.

---

### Likelihood Explanation

**Medium.** The attacker must observe a `deploy_account` transaction in the mempool for the target account. This is a common, publicly visible event for every new Starknet account. No privileged access is required; any network participant can submit the malicious invoke.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the target address. Alternatively, require the mempool to confirm the pending transaction is of type `DeployAccount` before allowing signature-skip for the paired invoke. This ensures only the legitimate account owner (who submitted the `deploy_account`) can benefit from the skip-validation UX path.

---

### Proof of Concept

1. Alice submits a `deploy_account` transaction for her new account at address `X` (nonce=0). It enters the mempool.
2. Attacker observes Alice's `deploy_account` in the mempool.
3. Attacker constructs an `Invoke` transaction: `sender_address=X`, `nonce=1`, arbitrary `calldata`, **empty/invalid signature**.
4. Gateway stateless validation passes (no signature check there).
5. Gateway stateful validation: `account_nonce = 0`, `tx.nonce() = 1` → enters `skip_stateful_validations`.
6. `account_tx_in_pool_or_recent_block(X)` returns `true` (Alice's `deploy_account` is in the pool).
7. `skip_validate = true` → `ExecutionFlags { validate: false }` → `__validate__` is **never called**.
8. The attacker's unsigned invoke is admitted to the mempool, occupying Alice's nonce=1 slot.
9. Alice's legitimate nonce=1 invoke is rejected or delayed due to the occupied slot. [6](#0-5)

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
