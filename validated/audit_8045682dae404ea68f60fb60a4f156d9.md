Audit Report

## Title
Single-Principal Withdrawal Queue Saturation via Sequential Minimum-Amount Calls - (File: rs/ethereum/cketh/minter/src/guard/mod.rs)

## Summary
The `retrieve_withdraw_guard` in the ckETH minter enforces only an in-flight concurrency check per principal, not a per-principal cap on queued requests. A single unprivileged principal can sequentially fill all 100 slots of `pending_withdrawal_requests`, blocking every other user from withdrawing ckETH or ckERC20 until the attacker's requests are drained by the timer at 5 per tick.

## Finding Description
The guard in `rs/ethereum/cketh/minter/src/guard/mod.rs` tracks two things: the set of principals with in-flight calls (`pending_withdrawal_principals`) and the total count of pending withdrawal requests (`withdrawal_requests_len()`). [1](#0-0) 

`Guard::new` rejects a new call only if `pending_requests_count >= MAX_PENDING` (global) or if the principal is already in `pending_withdrawal_principals` (in-flight): [2](#0-1) 

The `Drop` impl removes the principal from `pending_withdrawal_principals` when the async function returns: [3](#0-2) 

However, by the time the guard is dropped, `process_event(AcceptedEthWithdrawalRequest(...))` has already appended the request to `pending_withdrawal_requests`: [4](#0-3) 

After the function returns, the principal is no longer in `pending_withdrawal_principals`, so the `AlreadyProcessing` check passes on the very next call. There is no check anywhere that counts how many entries in `pending_withdrawal_requests` share the same `from` principal. A grep across all ckETH minter source confirms no per-principal limit exists. The attacker repeats sequential calls until `withdrawal_requests_len() == 100`, at which point every other caller receives `TooManyPendingRequests`.

The timer drains only `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` requests per tick: [5](#0-4) [6](#0-5) 

Both `withdraw_eth` and `withdraw_erc20` share the same guard path: [7](#0-6) [8](#0-7) 

## Impact Explanation
During queue saturation, every call to `withdraw_eth` or `withdraw_erc20` by any other user is rejected with `TooManyPendingRequests`. This is a complete, temporary denial of the ckETH/ckERC20 withdrawal service — a financial integration component — for all users. The DoS window spans approximately 20 timer ticks (100 requests ÷ 5 per tick). This matches the allowed High impact: **"Application/platform-level DoS... not based on raw volumetric DDoS"** and **"Significant Chain Fusion, ck-token... security impact with concrete user or protocol harm."**

## Likelihood Explanation
The attack requires no special privileges — any non-anonymous principal can trigger it. The attacker needs 100 × `cketh_minimum_withdrawal_amount` in ckETH. Because the ckETH is burned and the ETH is returned to the attacker's Ethereum address (minus gas), the net cost is only Ethereum transaction fees — a bounded, recoverable expense. The attack can be repeated indefinitely after each drain cycle, making it a persistent griefing vector.

## Recommendation
Add a per-principal cap on queued requests. Before inserting a new request via `process_event(AcceptedEthWithdrawalRequest(...))`, count how many entries in `pending_withdrawal_requests` share the same `from` principal and reject if the count exceeds a small threshold (e.g., 3–5). This check should be placed inside `Guard::new` alongside the existing `pending_requests_count` check, using the principal being guarded. Alternatively, raise `cketh_minimum_withdrawal_amount` to make queue-filling economically prohibitive, though a code-level fix is more robust.

## Proof of Concept
1. Attacker holds 100 × `cketh_minimum_withdrawal_amount` ckETH.
2. Attacker calls `withdraw_eth(min_amount, attacker_eth_addr)` — call completes, request enters `pending_withdrawal_requests` (len = 1), guard dropped, principal removed from `pending_withdrawal_principals`.
3. Attacker immediately calls `withdraw_eth` again — `AlreadyProcessing` check passes (principal not in-flight), `TooManyPendingRequests` check passes (len = 1 < 100) — guard acquired. Call completes, len = 2.
4. Repeat until len = 100.
5. Any legitimate user calling `withdraw_eth` or `withdraw_erc20` now receives `ic_cdk::trap("Failed retrieving guard … TooManyPendingRequests")`.
6. The minter timer drains 5 requests per tick; during ~20 ticks the queue is occupied exclusively by attacker requests and no legitimate withdrawal can be accepted.

A deterministic integration test can reproduce this by: initializing state, calling `retrieve_withdraw_guard` + `record_withdrawal_request` in a loop 100 times for the same principal (dropping the guard each iteration), then asserting that a new `retrieve_withdraw_guard` for a different principal returns `Err(GuardError::TooManyPendingRequests)`. The existing test `should_allow_limited_number_of_pending_requests` in `rs/ethereum/cketh/minter/src/guard/tests.rs` already demonstrates the `TooManyPendingRequests` path but uses 100 distinct principals — the same test with a single principal filling all 100 slots proves the attack.     fn new(principal: Principal) -> Result<Self, GuardError> {
        mutate_state(|s| {
            if PR::pending_requests_count(s) >= MAX_PENDING {
                return Err(GuardError::TooManyPendingRequests);
            }
            let principals = PR::guarded_principals(s);
            if principals.contains(&principal) {
                return Err(GuardError::AlreadyProcessing);
            }
            if principals.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            principals.insert(principal);
            Ok(Self {
                principal,
                _marker: PhantomData,
            })
        })
    }
