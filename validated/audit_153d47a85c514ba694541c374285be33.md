### Title
`native_per_pubdata` Truncates to Zero When `pubdata_price < native_price`, Allowing Pubdata to Be Used Without Native Resource Cost — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's double resource accounting model, the `native_per_pubdata` ratio is computed using integer floor division (`wrapping_div`). When the operator-set `pubdata_price` is less than `native_price`, this ratio truncates to zero. As a result, all pubdata consumed by a transaction costs zero native resources, breaking the intended resource accounting invariant and allowing any unprivileged user to use unbounded pubdata without paying the corresponding native (proving) cost.

---

### Finding Description

ZKsync OS tracks two resources per transaction: EVM gas (ergs) and native resource (a proxy for ZK proving cost). Pubdata (storage writes published to L1) is charged from native resources at a rate of `native_per_pubdata` native units per byte.

The ratio is computed in `validation_impl.rs`:

```rust
// We checked native_price != 0 above
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs truncating (floor) integer division. When `pubdata_price < native_price`, the result is `0`. This is structurally identical to the reported bug: a small numerator divided by a large denominator rounds down to zero, silently eliminating the charge.

Contrast this with `native_per_gas`, which correctly uses ceiling division to avoid the same problem:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

The same truncating division appears in `api/src/helpers.rs`:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

When `native_per_pubdata == 0`, every downstream pubdata accounting function returns zero cost:

- In `create_resources_for_tx`: `intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata) = 0` — no native is deducted for intrinsic pubdata.
- In `get_resources_to_charge_for_pubdata`: `native = current_pubdata_spent.checked_mul(0) = 0` — no native is charged for any amount of execution pubdata.
- In `check_enough_resources_for_pubdata`: always returns `enough = true` regardless of pubdata used.
- In `compute_gas_refund`: `delta_gas = (native_used / native_per_gas) - gas_used` — since pubdata contributes 0 to `native_used`, the `deltaGas` adjustment never compensates for pubdata costs.

The condition `pubdata_price < native_price` is a valid and realistic operator configuration. The existing test suite demonstrates it:

```rust
let native_price = U256::from(100);
let pubdata_price = U256::from(2);
```

Any `pubdata_price` in the range `[1, native_price - 1]` is silently treated identically to `pubdata_price = 0`.

---

### Impact Explanation

When `pubdata_price < native_price`, any unprivileged user can submit transactions that perform many storage writes (SSTORE) and consume large amounts of pubdata without paying any native resource cost for it. The ZK prover must still prove all those storage writes, but the transaction sender is not charged. This:

1. Breaks the resource accounting invariant that pubdata costs native resources proportional to `pubdata_price`.
2. Allows pubdata-heavy transactions to be submitted at a lower cost than the operator intended, potentially enabling pubdata spam that degrades proving throughput.
3. Causes the operator to under-collect native fees for pubdata, misaligning economic incentives.

---

### Likelihood Explanation

This occurs in every block where the operator sets `pubdata_price < native_price`. This is a valid and expected configuration (the two prices are independently set). Any unprivileged user who observes this condition can exploit it by submitting storage-write-heavy transactions. No special privileges, leaked keys, or oracle manipulation are required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. This ensures that even when `pubdata_price` is small relative to `native_price`, at least 1 native unit is charged per pubdata byte, preserving the accounting invariant.

### Proof of Concept

**Setup**: Operator sets `native_price = 1000`, `pubdata_price = 999` (a realistic scenario where pubdata is slightly cheaper than native computation).

**Computation**:
```
native_per_pubdata = 999.wrapping_div(1000) = 0
```

**Effect**: A transaction that writes to 1000 storage slots (producing ~32,000 bytes of pubdata) is charged:
```
pubdata_native_cost = 32000 * 0 = 0 native units
```

The operator intended to charge `32000 * ceil(999/1000) = 32000 * 1 = 32000` native units. The attacker pays 0.

**Attacker entry path**: Submit any EIP-1559 L2 transaction with calldata that triggers many SSTORE operations. The transaction passes validation (EVM gas is still charged), executes successfully, and the pubdata cost in native resources is zero.
