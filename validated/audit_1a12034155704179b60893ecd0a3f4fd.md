### Title
Missing Caller Authorization in Bank Precompile `transfer` Allows Unauthorized Native Token Theft - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile (`BankContract`) accepts an arbitrary `sender` address from calldata and executes `bankKeeper.SendCoins` from that address without verifying that `contract.Caller()` is authorized to move the sender's tokens. Any EVM contract can exploit this to drain any user's `evm/<contract>` native bank tokens to an attacker-controlled address.

### Finding Description
The bank precompile at address `0x0000000000000000000000000000000000000064` exposes a `transfer(address sender, address recipient, uint256 amount)` function. Its implementation in `BankContract.Run` under the `TransferMethodName` case is:

```go
sender := args[0].(common.Address)   // taken from calldata — NOT contract.Caller()
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // evm/<calling_contract_address>
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The denom of the transferred coin is `evm/<contract.Caller()>`, meaning it is scoped to the calling contract. However, the `from` address is taken directly from calldata with **no check** that `contract.Caller() == sender` or that the caller holds any approval from the sender. The precompile unconditionally calls `bankKeeper.SendCoins(ctx, from, to, ...)`, transferring the victim's native bank tokens to any recipient the caller specifies.

The same pattern exists in the `BurnMethodName` case: `args[0]` (named `recipient` but used as the address to burn from) is passed directly to `bankKeeper.SendCoinsFromAccountToModule` without verifying the caller is authorized to burn from that address.

The intended usage pattern, shown in `TestBank.sol`, is for a contract to pass `msg.sender` as the sender:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);
    return bank.transfer(msg.sender, recipient, amount);
}
```

But nothing in the precompile enforces this. A malicious contract can pass any victim address as `sender`.

### Impact Explanation
**Critical.** Any EVM contract can call `bank.transfer(victim, attacker, victim_balance)` and the precompile will execute `bankKeeper.SendCoins(ctx, victim, attacker, coins)` for the `evm/<malicious_contract>` denom without any authorization. This constitutes an unauthorized transfer of native bank-module assets. Users who hold `evm/<contract>` tokens — obtained through legitimate use of a contract's `moveToNative`/`mint` flow — can have their entire balance stolen by the contract itself or any contract that can be made to call the precompile with the victim's address as `sender`.

### Likelihood Explanation
Any deployed EVM contract that calls the bank precompile's `transfer` function with a caller-controlled `sender` argument can trigger this. No special privileges, leaked keys, or governance access are required. The attacker only needs to deploy a contract and call it.

### Recommendation
In the `TransferMethodName` case, enforce that the caller is authorized to move the sender's tokens. The minimal fix is to require `contract.Caller() == sender`:

```go
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer from sender")
}
```

Alternatively, implement an allowance mechanism analogous to ERC-20 `approve`/`transferFrom` at the native bank precompile level. The same fix should be applied to the `BurnMethodName` case, where `args[0]` is the address being burned from.

### Proof of Concept
1. Deploy `MaliciousContract` which calls `bank.transfer(victim, attacker, amount)` via the bank precompile at `0x0000000000000000000000000000000000000064`.
2. Victim calls `MaliciousContract.moveToNative(100)`, which internally calls `bank.mint(victim, 100)`, crediting the victim with 100 `evm/<MaliciousContract>` native bank tokens.
3. Attacker calls `MaliciousContract.steal(victim, attacker, 100)`, which calls `bank.transfer(victim, attacker, 100)`.
4. The precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, [{denom: "evm/<MaliciousContract>", amount: 100}])` with no authorization check.
5. Victim's 100 `evm/<MaliciousContract>` tokens are transferred to the attacker.
