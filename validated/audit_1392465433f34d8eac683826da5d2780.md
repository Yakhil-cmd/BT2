### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Loss of Staker and Pool Member Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Neither the staking contract nor the pool contract validates that `reward_address` is non-zero when it is set at registration time or updated via `change_reward_address`. A staker or pool member who supplies `address(0)` as their reward address will have all accrued yield permanently transferred to the zero address, with no recovery path for already-transferred rewards.

---

### Finding Description
The only validation applied to `reward_address` in both contracts is a check that it does not equal a registered token address (`REWARD_ADDRESS_IS_TOKEN`). There is no guard against the zero address.

``` [6](#0-5) 

---

### Impact Explanation
Any rewards transferred to `address(0)` are permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield** for the affected staker or pool member. The already-transferred tokens cannot be retrieved even if the user subsequently calls `change_reward_address` with a valid address, because the transfer has already executed.

**Impact: High** — permanent freezing of unclaimed yield.

---

### Likelihood Explanation
The zero address is a natural default/sentinel value in many front-end libraries and transaction-building tools. A user who submits a malformed transaction, a front-end that fails to populate the field, or a script with an uninitialized variable can silently register `address(0)` as the reward destination. The missing validation means the contract accepts it without error, giving the user no on-chain signal that anything is wrong until rewards are irreversibly lost.

**Likelihood: Medium** — requires user/tooling error, but the absence of any guard makes it a silent failure mode with no on-chain warning.

---

### Recommendation
Add a non-zero assertion for `reward_address` in all four entry points:

In `src/staking/staking.cairo` (`stake` and `change_reward_address`):
```cairo
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

In `src/pool/pool.cairo` (`enter_delegation_pool` and `change_reward_address`):
```cairo
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

### Proof of Concept

1. A pool member calls `enter_delegation_pool(reward_address: 0, amount: X)` on `src/pool/pool.cairo`.
2. The only check performed is `token_address != reward_address` (line 195), which passes because `token_address != 0`.
3. The pool member accrues rewards over subsequent epochs.
4. The pool member (or anyone) calls `claim_rewards(pool_member)`.
5. `claim_rewards` reads `reward_address = pool_member_info.reward_address` (which is `0`) and executes `reward_token.checked_transfer(recipient: 0, amount: rewards)`.
6. All accrued yield is transferred to the zero address and is permanently lost.
7. The pool member can call `change_reward_address` to set a valid address going forward, but the already-transferred rewards are unrecoverable. [7](#0-6) [8](#0-7)
