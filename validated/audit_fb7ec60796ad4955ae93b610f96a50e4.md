### Title
Gateway proof-facts block-number window uses `unchecked_next()` block but reads block-hash state at current block, causing off-by-one false rejection of valid client-side-proving transactions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`run_validate_entry_point` advances the block number by one (`unchecked_next()`) before building the `BlockContext` used for blockifier validation. The resulting `block_context.block_info.block_number = N+1` is passed into `validate_proof_block_number`, which therefore permits proof facts referencing any block up to `(N+1) − STORED_BLOCK_HASH_BUFFER`. However, the state object passed to `validate_proof_block_hash` is the committed state at block N, in which the block hash for block `N − STORED_BLOCK_HASH_BUFFER + 1` has not yet been written (the OS writes it at the very start of block N+1 via `pre_process_block`). Any Invoke V3 transaction whose proof facts cite that boundary block therefore passes the number check but fails the hash check with "Block hash mismatch", producing a false rejection at the gateway for a transaction that the blockifier would accept when it actually executes block N+1.

---

### Finding Description

**Step 1 – block number is bumped in the gateway.**

In `run_validate_entry_point`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  ~L324
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // N → N+1
let block_context = BlockContext::new(block_info, ...);
``` [1](#0-0) 

**Step 2 – `validate_proof_block_number` uses the bumped number.**

```rust
// crates/blockifier/src/transaction/account_transaction.rs  ~L244
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// current_block_number == N+1  →  max_allowed == N − STORED_BLOCK_HASH_BUFFER + 1
if proof_block_number > max_allowed { return Err(...) }
``` [2](#0-1) 

A proof citing block `N − STORED_BLOCK_HASH_BUFFER + 1` passes this check.

**Step 3 – `validate_proof_block_hash` reads from state at block N.**

```rust
// crates/blockifier/src/transaction/account_transaction.rs  ~L278
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;
if stored_block_hash != proof_block_hash { return Err(...) }
``` [3](#0-2) 

The OS writes the hash of block `N − STORED_BLOCK_HASH_BUFFER + 1` only at the **start** of block N+1 (`pre_process_block` → `write_block_number_to_block_hash_mapping`):

```cairo
// crates/apollo_starknet_os_program/…/os_utils.cairo  ~L52
tempvar old_block_number = block_context.block_info_for_execute.block_number -
    STORED_BLOCK_HASH_BUFFER;
``` [4](#0-3) 

The gateway's state reader returns the committed state at block N, so `get_storage_at` returns `Felt::ZERO` for that slot. The non-zero proof hash therefore mismatches, and the transaction is rejected.

**Step 4 – the blockifier at execution time would accept it.**

When the sequencer actually builds block N+1, `pre_process_block` runs first, writing the missing hash. `perform_pre_validation_stage` then reads the now-populated slot and the proof passes. The gateway and the blockifier disagree on the same transaction.

---

### Impact Explanation

Any Invoke V3 transaction carrying SNOS proof facts that reference the most-recently-allowed block (`N − STORED_BLOCK_HASH_BUFFER + 1`) is permanently rejected by the gateway even though it is fully valid for inclusion in block N+1. The user receives a `ValidateFailure` / "Block hash mismatch" error and must either use a one-block-older reference or wait until block N+1 is committed and resubmit. This is a gateway false-rejection of a valid transaction, matching the **High** impact tier: *"Mempool/gateway/RPC admission … rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The boundary block number (`N − STORED_BLOCK_HASH_BUFFER + 1`) is the most natural choice for a client-side prover: it is the freshest block whose hash is guaranteed on-chain. Any wallet or SDK that automatically selects the newest permissible block will hit this rejection on every submission until the next block is committed. The condition is deterministic and reproducible without any privileged access.

---

### Recommendation

In `run_validate_entry_point`, do **not** apply `unchecked_next()` before constructing the block context used for proof-facts validation, or pass the un-bumped block number specifically to `validate_proof_block_number`. The block-number upper bound should be derived from the same block N whose committed state is used for the hash lookup:

```rust
// Option A: use current block number for proof-facts check
let max_allowed = N - STORED_BLOCK_HASH_BUFFER;   // consistent with state at N
```

Alternatively, pre-populate the cached state with the block-hash entry that `pre_process_block` would write at the start of N+1 before running `validate_proof_facts`.

---

### Proof of Concept

1. Let `N` = latest committed block number; `B = STORED_BLOCK_HASH_BUFFER` (e.g. 10).
2. Read the on-chain block hash of block `N − B + 1` (it is committed and publicly available).
3. Construct an Invoke V3 transaction with `proof_facts` citing block number `N − B + 1` and the correct hash.
4. Submit to the gateway.
5. **`validate_proof_block_number`** passes: `N − B + 1 ≤ (N+1) − B`. [5](#0-4) 
6. **`validate_proof_block_hash`** fails: `state.get_storage_at(block_hash_contract, N−B+1)` returns `Felt::ZERO` (not yet written); the non-zero proof hash mismatches → `InvalidProofFacts("Block hash mismatch …")`. [6](#0-5) 
7. Gateway rejects the transaction. The identical transaction submitted after block N+1 is committed succeeds, confirming the off-by-one.

    fn validate_proof_block_number(
        proof_block_number: u64,
        current_block_number: BlockNumber,
    ) -> TransactionPreValidationResult<()> {
        // Proof block must be old enough to have a stored block hash.
        // Stored block hashes are guaranteed only up to: current - STORED_BLOCK_HASH_BUFFER.
        let max_allowed =

                TransactionPreValidationError::InvalidProofFacts(format!(
                    "The current block number {current_block_number} is below the required \
                     block-hash retention buffer: {STORED_BLOCK_HASH_BUFFER}."
                ))

            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "The proof block number {proof_block_number} is too recent. The maximum allowed \
                 block number is {max_allowed}."
            )));
        }

        Ok(())
    }

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
