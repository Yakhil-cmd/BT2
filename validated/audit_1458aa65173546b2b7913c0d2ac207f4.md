### Title
Missing Identity Element Check for `app_pk` Allows Disclosure of Confidential Derived Secret - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts `app_pk: PublicKey` (an `ElementG1 = blstrs::G1Projective`) without validating that it is not the group identity element. If the identity element is passed as `app_pk`, the ElGamal blinding term collapses to zero, and the aggregated `C` value in `CKDOutput` directly equals `msk · H(pk ‖ app_id)` — the confidential derived secret — in plaintext, visible to the coordinator and any observer of the output.

### Finding Description

The `ckd()` entry point performs several input validations:

Inside `compute_signature_share`, each participant computes the ElGamal ciphertext:

```rust
// src/confidential_key_derivation/protocol.rs, lines 173–174
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
``` [2](#0-1) 

When `app_pk = G1::identity()`, the blinding term `y · A = y · identity = identity` vanishes, so `big_c = big_s = x_i · H(pk ‖ app_id)`. After the coordinator aggregates all shares:

```
C = Σ λᵢ · Cᵢ = Σ λᵢ · xᵢ · H(pk ‖ app_id) = msk · H(pk ‖ app_id)
```

The `CKDOutput { big_y, big_c }` returned to the coordinator contains the confidential derived secret directly in `big_c`, with no blinding. [3](#0-2) 

By contrast, the `verify_signature` function in the ciphersuite correctly rejects identity elements for signatures and verifying keys, but this check is never applied to the incoming `app_pk`: [4](#0-3) 

The `AppId::try_new` constructor only enforces a maximum length, not a minimum, so an empty `AppId` (zero bytes) is also accepted without error, though this is a lesser concern since `hash_to_curve` still produces a valid non-identity point: [5](#0-4) 

### Impact Explanation

The fundamental security requirement of CKD is that no single MPC node — including the coordinator — can learn the derived secret `s = msk · H(pk ‖ app_id)`. The ElGamal blinding (`y · A`) is the sole mechanism enforcing this. When `app_pk = identity`, this mechanism is entirely neutralized: the coordinator receives `CKDOutput` where `big_c = msk · H(pk ‖ app_id)` in the clear. This constitutes **disclosure of a confidential derived secret**, matching the Critical impact tier: *"Extraction, reconstruction, or disclosure of... confidential derived secrets."* [6](#0-5) 

### Likelihood Explanation

The `app_pk` value originates from the MPC contract on-chain and is forwarded to each MPC node. A misconfigured developer contract, a bug in the on-chain request handler, or a malicious MPC contract operator could supply `G1Projective::identity()` as `app_pk`. The library provides no defense-in-depth check at the protocol entry point. Because the identity element is a valid Rust value of type `blstrs::G1Projective` and requires no special construction, accidental or adversarial submission is realistic.

### Recommendation

Add an identity element check for `app_pk` inside `ckd()` before the protocol is launched, analogous to the checks already present in `verify_signature` and the `BLS12381G1Group::serialize` implementation:

```rust
// In ckd(), after the coordinator membership check:
use elliptic_curve::group::prime::PrimeCurveAffine;
if bool::from(G1Affine::from(app_pk).is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
``` [7](#0-6)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
