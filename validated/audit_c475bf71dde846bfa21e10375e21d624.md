### Title
`msg.value` Reuse via Inherited `Multicall` Allows Registration Fee Bypass — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase` inherits OpenZeppelin's `Multicall`, which executes each sub-call via `delegatecall`. Because `delegatecall` preserves `msg.value` from the outer frame, a caller can batch N `register(...)` calls inside a single `multicall(...)` invocation and pay the `registrationFee` only once, registering N pools for the price of one.

---

### Finding Description

`OracleBase` inherits `Multicall` from OpenZeppelin: [1](#0-0) [2](#0-1) 

The `register` function is `payable` and enforces a fee gate: [3](#0-2) 

OpenZeppelin's `Multicall.multicall` loops over sub-calls using `Address.functionDelegateCall(address(this), data[i])`. A `delegatecall` inherits the outer call's `msg.value` unchanged for every iteration. Therefore, if a caller sends `msg.value == registrationFee` and encodes two (or more) `register(...)` calls in the `data` array, every delegatecall sees the same `msg.value`, and every `require(msg.value >= registrationFee, ...)` check passes — even though only one fee payment was made.

The same `Multicall` pattern is present in the periphery contracts (`MetricOmmSimpleRouter`, `MetricOmmPoolLiquidityAdder`): [4](#0-3) 

However, those contracts' `pay()` helper reads `address(this).balance` (the real balance, which is consumed on the first hop), so they are not exploitable in the same way. The oracle `register` path is exploitable because it checks `msg.value` directly.

The comment in the code explicitly acknowledges that overpayment is intentional but does not address underpayment via multicall: [5](#0-4) 

---

### Impact Explanation

Every pool that wants to use the on-chain `price(feedId, pool)` path must call `register` and pay `registrationFee`. The fee is the protocol's sole revenue mechanism for oracle access control. An attacker can register an unbounded number of pools while paying the fee only once, causing a direct loss of protocol fee revenue proportional to `(N-1) * registrationFee`. The admin withdraws accumulated ETH via `withdrawEth()`: [6](#0-5) 

The shortfall is permanent and unrecoverable because the contract has no mechanism to retroactively charge for already-completed registrations.

---

### Likelihood Explanation

The trigger is fully permissionless — any EOA or contract can call `multicall`. No special role, no privileged setup, and no malicious token is required. The only prerequisite is knowing the `feedId`, a valid pool address, and an approved factory — all of which are public on-chain. The default `registrationFee` is `1 wei`, but the admin is expected to raise it to a meaningful value: [7](#0-6) 

Once the fee is set to a non-trivial amount, exploitation is trivially profitable.

---

### Recommendation

Either:

1. **Remove `payable` from `register` and enforce fee collection via a separate, non-delegatecall path**, or
2. **Override `multicall` in `OracleBase` to reject any call that bundles a `register` selector with `msg.value > 0`**, or
3. **Replace the `msg.value` check with an explicit ERC-20 fee token pull**, which is immune to delegatecall reuse, or
4. **Track cumulative ETH received per transaction** (e.g., via a transient-storage counter) and deduct from it on each `register` call within the same transaction.

The simplest fix matching the Spearbit recommendation in the external report is to make `register` non-payable and collect fees via `transferFrom` on a designated fee token.

---

### Proof of Concept

```solidity
// Attacker registers 3 pools for the price of 1
bytes[] memory calls = new bytes[](3);
calls[0] = abi.encodeCall(OracleBase.register, (feedId, pool1, factory));
calls[1] = abi.encodeCall(OracleBase.register, (feedId, pool2, factory));
calls[2] = abi.encodeCall(OracleBase.register, (feedId, pool3, factory));

// msg.value == registrationFee (paid once)
// Each delegatecall sees msg.value == registrationFee → all three pass
oracle.multicall{value: registrationFee}(calls);

// Result: 3 pools registered, only 1× fee received by the contract
assert(oracle.registeredPool(feedId, pool1));
assert(oracle.registeredPool(feedId, pool2));
assert(oracle.registeredPool(feedId, pool3));
assert(address(oracle).balance == registrationFee); // should be 3× registrationFee
```

    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
    }
