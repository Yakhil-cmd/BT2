### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Single-Party Signature Creation - (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that `threshold ≤ participants.len()` (upper bound) but never validate that `threshold ≥ 2` (lower bound). Because `ReconstructionLowerBound` is an unconstrained `usize` wrapper, a caller can pass `threshold = 1` (or `0`) to the FROST signing entry points. With `threshold = 1`, a single participant satisfies the reconstruction requirement and can unilaterally produce a valid threshold signature, bypassing the multi-party security property entirely.

---

### Finding Description

`ReconstructionLowerBound` is defined as a plain `usize` newtype with no minimum enforcement: [1](#0-0) 

`assert_sign_inputs` (the public validation gate for FROST signing) checks only the upper bound: [2](#0-1) 

It never checks `threshold.value() < 2`. The same omission exists in `presign`: [3](#0-2) 

By contrast, `assert_key_invariants` in `src/dkg.rs` — the DKG entry-point validator — explicitly enforces the lower bound: [4](#0-3) 

The DKG correctly rejects `threshold < 2`, but the signing path does not. Because the two validations are independent, a caller can invoke the FROST signing functions with an arbitrarily small threshold even when the key was generated with a proper threshold.

---

### Impact Explanation

In FROST, `threshold` is the reconstruction lower bound: the minimum number of participants whose signature shares must be combined to produce a valid signature. If `threshold = 1` is accepted by `assert_sign_inputs`, a single participant's share is sufficient to reconstruct and output a valid signature over the shared key. This directly violates the threshold security property: one party, acting alone, can create a valid signature that is indistinguishable from one produced by a legitimate quorum. This matches the **Critical** impact class: *Unauthorized creation of a valid threshold signature for attacker-chosen inputs*.

---

### Likelihood Explanation

The entry point is a public library function. Any caller — including a malicious coordinator or a compromised participant — can supply `threshold = 1` directly. No cryptographic break, key leakage, or external dependency compromise is required. The `ReconstructionLowerBound` type accepts any `usize`, so the call `assert_sign_inputs(participants, 1usize, me, coordinator)` compiles and passes all current validation checks.

---

### Recommendation

Add the same lower-bound check that `assert_key_invariants` already enforces, inside both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing guard in `assert_key_invariants`: [5](#0-4) 

---

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (accepted, correct).
2. Call `assert_sign_inputs(&participants, 1usize, me, coordinator)` — this currently returns `Ok(participant_list)` because `1 ≤ 3` passes the only threshold check.
3. Proceed to FROST signing with `threshold = 1`. Only participant `me` needs to contribute a signature share; the reconstruction succeeds with a single share.
4. The output is a valid signature over the shared public key, produced by one party without cooperation from any other participant.

The root cause is the missing `threshold.value() < 2` guard in `assert_sign_inputs` at: [6](#0-5) 

and in `presign` at: [7](#0-6)
