### Title
WalletConnect Offer Confirmation Understates Total Cost for Multi-NFT Royalty Offers - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible payment amount by the number of NFTs before applying each royalty percentage. When a WalletConnect dApp presents a `chia_wallet.take_offer` command containing multiple NFTs with royalties, the confirmation dialog shows a materially understated `amountWithRoyalties` total. The user approves based on the misleading figure, but the actual on-chain transaction deducts the correct (higher) amount.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect offer-acceptance confirmation to compute the total cost including creator royalties: [1](#0-0) 

The critical defect is on line 363:

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Each royalty percentage is applied to `amount / N` (where N is the number of NFTs with royalties) instead of to the full `amount`. The correct formula is to apply each royalty to the full payment amount and sum the results.

**Concrete example** (confirmed by the existing test at line 251–334):

| Scenario | Displayed `amountWithRoyalties` | Correct total |
|---|---|---|
| 2 NFTs, royalties 5 % + 0.1 %, payment 0.0001 XCH | 0.00010255 XCH | 0.0001051 XCH |
| 10 NFTs each 10 % royalty, payment 1 XCH | 1.1 XCH | 2 XCH |

The test suite itself encodes the wrong value as the expected result, confirming the bug is present and untested-against-correct-math: [2](#0-1) 

The `amountWithRoyalties` field is then surfaced in the WalletConnect confirmation dialog via `withRoyaltyTotals` → `walletDeltaToDisplay` → `WalletDeltaSection`: [3](#0-2) [4](#0-3) 

The royalty percentages themselves are read from the offer summary `infos` field (for `take_offer`) or from the `driver_dict` (for `create_offer_for_ids`), both of which are attacker-controlled inputs arriving over WalletConnect: [5](#0-4) [6](#0-5) 

### Impact Explanation

A user approves the WalletConnect prompt believing they will spend the displayed `amountWithRoyalties`. The actual `take_offer` RPC call uses the cryptographically-bound offer data, so the blockchain deducts the correct (higher) royalty-inclusive amount. The gap between the displayed figure and the real deduction grows with the number of NFTs and the royalty percentages involved, and is entirely attacker-controllable. This constitutes a WalletConnect state spoofing that causes the user to approve the wrong amount — a direct, concrete wallet balance impact.

### Likelihood Explanation

Any WalletConnect-connected dApp can issue a `chia_wallet.take_offer` command with an offer containing two or more NFTs that carry royalties. No special privileges, leaked keys, or social engineering beyond a normal WalletConnect session are required. The NFT royalty percentage is read directly from the attacker-supplied `infos` / `driver_dict` without an independent on-chain verification step before the confirmation dialog is rendered.

### Recommendation

Replace the split-then-multiply formula with the correct per-NFT full-amount calculation:

```ts
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Remove the `splitAmount` variable entirely. Update the corresponding test expectations in `parseCommandDisplay.test.ts` to reflect the corrected values.

### Proof of Concept

1. Attacker operates a WalletConnect dApp and establishes a session with the victim's Chia GUI wallet.
2. Attacker crafts an offer containing 10 NFTs each with a 10 % royalty (`royalty_percentage: 1000`) and a fungible payment of 1 XCH (1 000 000 000 000 mojos).
3. Attacker sends `chia_wallet.take_offer` over WalletConnect.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties` with `amount = 1_000_000_000_000n` and `royaltyPercentages = [1000, 1000, 1000, 1000, 1000, 1000, 1000, 1000, 1000, 1000]`.
5. `splitAmount = 1_000_000_000_000n / 10n = 100_000_000_000n`; each royalty contributes `100_000_000_000 * 1000 / 10_000 = 10_000_000_000`; total royalty = `100_000_000_000`; displayed total = **1.1 XCH**.
6. Correct total: each royalty = `1_000_000_000_000 * 1000 / 10_000 = 100_000_000_000`; total royalty = `1_000_000_000_000`; correct total = **2 XCH**.
7. Victim sees "You Spend: 1.1 XCH" in the WalletConnect confirmation dialog and approves.
8. The blockchain deducts 2 XCH from the victim's wallet — a 0.9 XCH loss relative to what was displayed.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
