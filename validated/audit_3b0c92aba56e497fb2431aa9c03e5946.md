### Title
Hardcoded Test Private Key Used in Production Consensus Signing — (`File: crates/apollo_signature_manager/src/lib.rs`, `crates/apollo_signature_manager/src/signature_manager.rs`)

---

### Summary

The production `create_signature_manager()` function, called unconditionally during node startup, instantiates `LocalKeyStore` via `new_for_testing()`, which contains a hardcoded, publicly-known ECDSA private key. Every sequencer node running this code signs consensus precommit votes and peer-identity challenges with the same known private key `0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133`.

---

### Finding Description

`LocalKeyStore::new_for_testing()` embeds a fixed private key directly in source code: [1](#0-0) 

The production factory function `create_signature_manager()` calls `SignatureManager::new()`, which calls `LocalKeyStore::new_for_testing()` — there is no other constructor path: [2](#0-1) 

A TODO comment in the same file explicitly acknowledges this is not production-ready: [3](#0-2) 

The node startup code in `crates/apollo_node/src/components.rs` calls `create_signature_manager()` for any `LocalExecutionWithRemoteDisabled` or `LocalExecutionWithRemoteEnabled` execution mode — which is the standard production deployment mode shown in the distributed deployment config: [4](#0-3) 

The `SignatureManager` is wired into the consensus pipeline and handles two operations:

1. `sign_precommit_vote(block_hash)` — signs BFT consensus precommit votes
2. `sign_identification(peer_id, challenge)` — signs peer identity challenges [5](#0-4) 

Both operations use the same hardcoded private key retrieved via `self.keystore.get_key()`: [6](#0-5) 

---

### Impact Explanation

The private key `0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133` is publicly visible in the git history. Any party who reads the source code can:

1. **Forge precommit votes** for arbitrary block hashes, impersonating any validator node running this code. Because all nodes share the same key, a forged signature is indistinguishable from a legitimate one.
2. **Pass peer identity challenges** as any node, enabling network-level impersonation.

This directly maps to the allowed High impact: *"Transaction conversion or signature/hash logic binds the wrong signer, hash, type, or executable payload."* The signature manager binds the wrong signer (a shared, publicly-known test key) to every consensus precommit vote, undermining the BFT security model that requires each validator to hold a unique, secret key.

---

### Likelihood Explanation

The private key is in plain text in a public repository. Any reader of the source code — including the git history — has immediate access. The production node startup path unconditionally uses this key with no configuration override. Likelihood is high.

---

### Recommendation

1. Remove the hardcoded key from `new_for_testing()` from any production code path. The `_new(private_key: PrivateKey)` constructor already exists but is unused in production.
2. Load the validator private key from a secure external source at startup (environment variable, secrets manager, HSM, or encrypted config file) and pass it to `LocalKeyStore::_new(...)`.
3. Rotate the exposed key immediately; treat `0x608bf2cdb1ad4138e72d2f82b8c5db9fa182d1883868ae582ed373429b7a133` as permanently compromised.
4. Add a compile-time or runtime guard that prevents `new_for_testing()` from being reachable outside `#[cfg(test)]`.

    → SignatureManager::new()                           // lib.rs:19
      → LocalKeyStore::new_for_testing()                // signature_manager.rs:101
        → PRIVATE_KEY = 0x608bf2cdb1ad4138...           // signature_manager.rs:103-104

consensus precommit vote path:
  → SignatureManager::sign_precommit_vote(block_hash)   // signature_manager.rs:68

Any attacker can reproduce a valid precommit signature for any block hash using the known private key, making it impossible to distinguish legitimate validator votes from forged ones.
