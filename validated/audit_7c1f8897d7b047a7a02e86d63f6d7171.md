Audit Report

## Title
Unconditional ballot overwrite in `vote_on_root_proposal_to_upgrade_governance_canister` allows a single NNS node operator to flip votes and block governance-canister upgrades — (`rs/nns/handlers/root/impl/src/root_proposals.rs`)

## Summary
The `vote_on_root_proposal_to_upgrade_governance_canister` function unconditionally overwrites a node operator's existing ballot at line 367 with no check on the prior ballot value. A single registered NNS node operator — explicitly within the Byzantine fault budget f = (N−1)/3 that the root proposal mechanism is designed to tolerate — can flip their vote arbitrarily many times within the 7-day proposal window, permanently blocking a legitimate governance-canister upgrade or forcing one through below the genuine-Yes threshold.

## Finding Description
`GovernanceUpgradeRootProposal` stores ballots as `Vec<(PrincipalId, RootProposalBallot)>` initialized to `Undecided` for every NNS node at proposal submission time (`rs/nns/handlers/root/impl/src/root_proposals.rs`, line 101). The `RootProposalBallot` enum has three variants: `Yes`, `No`, and `Undecided` (lines 34–38).

In `vote_on_root_proposal_to_upgrade_governance_canister` (lines 303–380), after checking proposal expiry (lines 331–341), registry version (lines 344–352), and wasm SHA (lines 354–361), the function iterates over `node_operator_ballots` and executes:

```rust
for (p, b) in &mut proposal.node_operator_ballots {
    if p == &caller {
        *b = ballot.clone();   // unconditional overwrite — lines 365–369
        voted_on += 1;
    }
}
```

There is no guard of the form `if !matches!(b, RootProposalBallot::Undecided)` before the overwrite. The only eligibility check (lines 372–378) verifies that the caller appears in the ballot list at all, not that they haven't already voted.

The threshold check `is_byzantine_majority_yes` (lines 110–122) requires `votes_yes >= num_nodes - max_faults`. Because any ballot can be overwritten at any time before the threshold is reached, the tally is not monotone.

**Attack path 1 — persistent blocking (Yes → No):**
1. Attacker (a registered NNS node operator) submits a proposal; their ballot is auto-set to `Yes`.
2. Other operators vote `Yes` until tally = threshold − 1.
3. Attacker calls `vote_on_root_proposal_to_upgrade_governance_canister(ballot=No)`; their `Yes` is overwritten, dropping tally to threshold − 2.
4. Each time another operator votes `Yes` to restore the tally, the attacker flips again. Repeatable indefinitely within the 7-day window.

**Attack path 2 — forced execution (No → Yes):**
1. A non-proposer node operator votes `No`.
2. Other operators vote `Yes` until tally = threshold − 1.
3. Attacker flips their `No` to `Yes`, pushing tally to threshold and triggering execution with only threshold − 1 genuine Yes votes.

## Impact Explanation
The governance canister controls all NNS canisters and all ICP/Cycles flows. A single malicious NNS node operator — a role the BFT model explicitly treats as potentially Byzantine — can permanently block any governance-canister upgrade for its full 7-day lifetime, or force an upgrade through with fewer genuine Yes votes than the Byzantine threshold requires. This constitutes a significant NNS security impact with concrete protocol harm, mapping to **High ($2,000–$10,000): Significant NNS security impact with concrete user or protocol harm** and **High ($2,000–$10,000): Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
Any registered NNS node operator can exploit this. No key compromise, social engineering, or majority collusion is required. The call is a standard ingress message to the NNS root canister. The only precondition is that a proposal is pending and has not yet reached majority — a normal operational state. The attack is repeatable for the full 7-day proposal lifetime and requires no special tooling beyond a standard canister call.

## Recommendation
Add an immutability check before overwriting the ballot. Only allow a transition from `Undecided`:

```rust
for (p, b) in &mut proposal.node_operator_ballots {
    if p == &caller {
        if !matches!(b, RootProposalBallot::Undecided) {
            return Err(format!(
                "{LOG_PREFIX}Caller: {caller} has already cast a ballot."
            ));
        }
        *b = ballot.clone();
        voted_on += 1;
    }
}
```

This enforces the invariant that each node operator's ballot is immutable once cast, consistent with the Byzantine fault tolerance model the proposal mechanism is built on.

## Proof of Concept
State-machine test with a 7-node NNS subnet (N=7, f=2, threshold=5):

1. Operator A submits proposal → tally: 1 Yes.
2. Operators B, C, D vote Yes → tally: 4 Yes (one short of threshold=5).
3. Operator A calls `vote_on_root_proposal_to_upgrade_governance_canister(ballot=No)` → tally: 3 Yes, 1 No.
4. Assert proposal is not executed and tally is 3 Yes.
5. Operator E votes Yes → tally: 4 Yes again.
6. Operator A flips back to No → tally: 3 Yes again.
7. Assert proposal is permanently blocked by a single operator within the 7-day window.

For the forced-execution variant: after step 2, have operator A (who voted No in a separate scenario) flip to Yes, pushing tally to 5 and triggering execution with only 4 genuine Yes votes.
