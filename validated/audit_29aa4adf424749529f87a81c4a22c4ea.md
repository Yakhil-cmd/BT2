### Title
`setPoolAdminFeeDestination` silently redirects all accrued admin fees to the new destination without prior collection — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` changes where admin fees are sent without first flushing accrued fees to the old destination. Every other fee-configuration mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` at the old rates/destination before applying the change. `setPoolAdminFeeDestination` omits this step entirely, so all fees that accrued under the old destination are silently redirected to the new one on the next `collectPoolFees` call.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the same invariant: collect all accrued fees at the current configuration before mutating it. [1](#0-0) 

`setPoolAdminFeeDestination` breaks this invariant. It updates `poolAdminFeeDestination[pool]` directly with no prior collection: [2](#0-1) 

`collectFees` on the pool uses the `adminFeeDestination_` argument passed by the factory at call time. The factory always reads `poolAdminFeeDestination[pool]` at that moment: [3](#0-2) 

So any spread-fee surplus and notional-fee accumulator (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) that built up while the old destination was active will be transferred to the new destination on the next `collectPoolFees` call: [4](#0-3) 

---

### Impact Explanation

The old admin fee destination (which may be a separate treasury contract, DAO vault, or third-party beneficiary distinct from the admin EOA/multisig) loses all fees that accrued before the destination change. Those tokens are transferred to the new destination instead. This is a direct, quantifiable loss of owed protocol/admin assets — the exact amount equals the admin share of all spread surplus and notional fees accumulated since the last `collectPoolFees` call.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a legitimate reason to rotate the fee destination (treasury migration, multisig key rotation, etc.). The trigger is a single admin call with no timelock. `collectPoolFees` is permissionless, so the window between the destination change and the next collection can be arbitrarily short or long, but the loss is locked in the moment `setPoolAdminFeeDestination` is called while fees are accrued.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the **old** destination before recording the new one.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Collect accrued fees to the OLD destination first
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool has been active; `notionalFeeToken0Scaled > 0` and spread surplus exists. Old admin fee destination is `treasuryA`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, treasuryB)`. No collection occurs; `poolAdminFeeDestination[pool]` is now `treasuryB`.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool] == treasuryB` and passes it to `collectFees`.
4. `collectFees` transfers the entire admin share of all previously accrued fees to `treasuryB`.
5. `treasuryA` receives nothing despite being the rightful destination for all fees accrued before step 2.

The corrupted value is `poolAdminFeeDestination[pool]` being updated before the accrued fee balances (`notionalFeeToken0Scaled`, `notionalFeeToken1Scaled`, and spread surplus) are flushed, causing the wrong address to receive tokens that were owed to the old destination. [5](#0-4) [6](#0-5)
