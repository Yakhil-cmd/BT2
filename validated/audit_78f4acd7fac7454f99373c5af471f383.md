### Title
Unauthorized Invoke Transaction Bypasses Signature Verification via `skip_stateful_validations` Deploy-Account UX Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature/authorization check) for invoke transactions with `nonce=1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. An attacker who observes a pending `deploy_account` transaction for address A in the mempool can submit an invoke with `nonce=1` for A carrying arbitrary calldata and a forged signature. The gateway skips `__validate__` because the deploy_account satisfies the pool-presence check, and the unauthorized invoke is admitted to the mempool without any signature verification.

---

### Finding Description

**The intended invariant.** The `skip_stateful_validations` UX feature is designed for the deploy-account + invoke flow: a user submits `deploy_account` (nonce 0) and `invoke` (nonce 1) together. Because the account does not yet exist on-chain, running `__validate__` on the invoke would fail. The gateway therefore skips `__validate__` when `tx_nonce == 1 && account_nonce == 0` and the account already has a transaction in the mempool, reasoning that the only way an account with on-chain nonce 0 can have a mempool entry is via a `deploy_account` submitted by the legitimate owner.

**The broken check.** The presence test is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())   // ← only checks presence
                .await ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the address in the pool or recent block — it does not verify the transaction type is `deploy_account`, nor that it was submitted by the legitimate account owner:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700

    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

**Effect of `skip_validate = true`.** When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust

    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`validate_tx` then returns immediately without executing `__validate__`:

```rust

}
``` [4](#0-3) 

**Attack path (step by step):**

1. Victim submits `deploy_account` for address A (nonce 0). It passes `__validate_deploy__` and enters the mempool's `tx_pool`.
2. Attacker observes the pending `deploy_account` (mempool is public to all nodes).
3. Attacker submits `invoke` with `nonce=1` for address A, arbitrary calldata, and a forged/empty signature.
4. **Gateway nonce check** (`validate_nonce`): `account_nonce=0`, `tx_nonce=1`, passes if `max_allowed_nonce_gap ≥ 1`. [5](#0-4) 

5. **Mempool `validate_tx`**: `resolve_nonce(A, 0) = 0` (not yet staged), `tx_nonce=1 ≥ 0`, passes. No duplicate nonce (attacker uses nonce 1, victim's deploy_account uses nonce 0). [6](#0-5) 

6. **`skip_stateful_validations`**: `tx_nonce=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block(A) = true` (victim's deploy_account is in `tx_pool`). Returns `true`.
7. **`run_validate_entry_point`**: `validate=false`, `__validate__` is never called.
8. Attacker's invoke is forwarded to the mempool and admitted.

---

### Impact Explanation

An unauthorized invoke transaction — carrying arbitrary calldata and a forged signature — is admitted to the mempool without any account-level authorization check. This violates the core invariant that every transaction in the mempool has been authorized by the account's `__validate__` entry point.

Concrete consequences:
- The attacker's `nonce=1` invoke occupies the nonce-1 slot for address A. The victim's legitimate `nonce=1` invoke is rejected as `DuplicateNonce` or must pay a fee-escalation premium to replace it.
- The attacker can repeat this for every new `deploy_account` observed in the mempool, systematically front-running the nonce-1 slot for all newly deploying accounts.
- The attacker's transaction will fail during block execution (blockifier runs `__validate__` unconditionally during execution), but the DoS effect on the victim's account persists until the attacker's tx is evicted or replaced.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

---

### Likelihood Explanation

- The mempool is observable by any connected node; pending `deploy_account` transactions are visible.
- The attack requires no special privileges, no funds beyond gas for the attacker's invoke, and no cryptographic capability.
- The condition (`tx_nonce=1`, `account_nonce=0`, pool presence) is precisely the condition that fires for every new account deployment.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a `deploy_account` transaction (not just any transaction) is pending for the address. For example, expose a `deploy_account_in_pool_or_recent_block(address)` method on the mempool that inspects the transaction type before returning `true`. This preserves the UX intent while closing the bypass.

---

### Proof of Concept

```
// State: address A not deployed (on-chain nonce = 0)

// Step 1 – Victim
gateway.add_tx(deploy_account { sender: A, nonce: 0, valid_sig })
// → deploy_account enters mempool tx_pool for address A

// Step 2 – Attacker (observes mempool)
gateway.add_tx(invoke { sender: A, nonce: 1, calldata: [arbitrary], sig: [0x0] })

// Gateway flow for attacker's invoke:
//   validate_nonce:                  1 >= 0  ✓  (max_allowed_nonce_gap >= 1)
//   mempool.validate_tx:             1 >= resolve_nonce(A,0)=0  ✓
//   skip_stateful_validations:       nonce==1 && account_nonce==0
//                                    account_tx_in_pool_or_recent_block(A) == true  ✓
//                                    → skip_validate = true
//   run_validate_entry_point:        validate=false → __validate__ NOT called
//   → attacker's invoke admitted to mempool

// Step 3 – Victim submits legitimate invoke
gateway.add_tx(invoke { sender: A, nonce: 1, calldata: [real], sig: [valid] })
// → MempoolError::DuplicateNonce  (attacker already holds nonce-1 slot)
//   Victim must pay fee-escalation premium to reclaim their own nonce slot.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
