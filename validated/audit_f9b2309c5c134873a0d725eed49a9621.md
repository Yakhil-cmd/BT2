### Title
CRC21 Contract Blocklist Causes Permanent IBC Receive Failure via Error Acknowledgement — (`x/cronos/middleware/conversion_middleware.go`)

### Summary
`IBCConversionModule.OnRecvPacket` auto-converts incoming IBC vouchers to CRC21 tokens by calling `ConvertVouchersToEvmCoins`. If the registered external CRC21 contract has a blocklist and the receiver is blocklisted, the EVM call to `mint_by_cronos_module` reverts. The middleware then returns `channeltypes.NewErrorAcknowledgement(err)` **without** calling `commit()`, discarding the underlying transfer state. The sender chain refunds the tokens; the receiver permanently cannot receive that IBC denom via the auto-conversion path.

### Finding Description

`OnRecvPacket` uses a `cacheCtx` pattern:

```
cacheCtx, commit := ctx.CacheContext()
ack := im.app.OnRecvPacket(cacheCtx, ...)   // underlying transfer succeeds → tokens credited in cacheCtx
...
if err := im.cronoskeeper.ConvertVouchersToEvmCoins(cacheCtx, data.Receiver, ...); err != nil {
    return channeltypes.NewErrorAcknowledgement(err)   // commit() is NEVER called
}
commit()
return ack
``` [1](#0-0) 

When conversion is attempted, `ConvertVouchersToEvmCoins` → `ConvertCoinFromNativeToCRC21` calls:

```go
_, err = k.CallModuleCRC21(ctx, contract, "mint_by_cronos_module", sender, coin.Amount.BigInt())
``` [2](#0-1) 

`CallModuleCRC21` executes the EVM call and checks `res.Failed()`:

```go
if res.Failed() {
    return nil, fmt.Errorf("call contract failed: %s, %s, %s", contract.Hex(), method, res.Ret)
}
``` [3](#0-2) 

If the registered external CRC21 contract implements a blocklist (as demonstrated by the integration-test fixture `TestBlackListERC20.sol`) and the receiver is blocklisted, `mint_by_cronos_module` reverts inside the EVM. The error propagates back to `OnRecvPacket`, which returns `NewErrorAcknowledgement` without committing `cacheCtx`. Consequences:

1. The underlying IBC transfer state (vouchers credited to receiver) is **discarded**.
2. An error acknowledgement is written on-chain.
3. The sender chain's ICS-20 module refunds the sender.
4. The receiver **permanently** cannot receive that IBC denom via the auto-conversion path as long as the blocklist is active. [4](#0-3) 

Contrast with `OnAcknowledgementPacket` and `OnTimeoutPacket`, which explicitly treat conversion failure as non-fatal (log and continue), preserving the refund:

```go
// Intentional: log and continue so the IBC refund is not blocked.
im.cronoskeeper.Logger(ctx).Error("failed to convert refund vouchers on acknowledgement", ...)
``` [5](#0-4) 

The inconsistency is the root cause: `OnRecvPacket` escalates a conversion error into a fatal IBC-level error, while the refund paths do not.

### Impact Explanation

**High** — Permanent, long-lived inability for honest users to receive valid IBC transfers. Any address blocklisted in a registered external CRC21 contract can never receive that IBC denom via the auto-conversion path. Because `commit()` is skipped, the receiver also loses the raw IBC vouchers (they are refunded to the sender, not held for the receiver). The receiver has no fallback path to claim the tokens on Cronos.

### Likelihood Explanation

An external CRC21 contract with a blocklist must be registered for an IBC denom by the Cronos admin. However, the contract owner (a separate, unprivileged entity) can deploy a contract that appears legitimate at registration time and later activate a blocklist, or can deploy a contract with a pre-existing hidden blocklist. Once registered, the contract owner can unilaterally blocklist any receiver address with no further admin action required. The `canBeConverted` check confirms the conversion path is taken for any denom with a registered contract:

```go
_, found := im.cronoskeeper.GetContractByDenom(ctx, denom)
return found
``` [6](#0-5) 

### Recommendation

Treat conversion failure in `OnRecvPacket` as non-fatal, consistent with the refund paths. Log the error, call `commit()` so the receiver retains the raw IBC vouchers, and return the success ack. The receiver can then manually convert via `MsgConvertVouchers`. Example:

```go
if err := im.cronoskeeper.ConvertVouchersToEvmCoins(cacheCtx, data.Receiver, sdk.NewCoins(token)); err != nil {
    im.cronoskeeper.Logger(ctx).Error("failed to convert vouchers on recv", "error", err)
    // fall through: commit the underlying transfer so receiver keeps the vouchers
}
commit()
return ack
```

### Proof of Concept

1. Deploy an external CRC21 contract with a blocklist (pattern shown in `integration_tests/contracts/contracts/TestBlackListERC20.sol`). [7](#0-6) 
2. Cronos admin registers this contract for an IBC denom via `MsgUpdateTokenMapping`.
3. Contract owner calls `setBlock(receiver, true)` to blocklist the target receiver — no admin action required from this point.
4. A third party sends IBC tokens of that denom to the receiver.
5. `OnRecvPacket` calls `ConvertVouchersToEvmCoins` → `mint_by_cronos_module` reverts → `NewErrorAcknowledgement` is returned without `commit()`.
6. Sender chain refunds the sender; receiver gets nothing.
7. Every subsequent IBC send to that receiver for that denom produces the same result indefinitely.

pragma solidity 0.8.21;
import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract TestBlackListERC20 is ERC20 {
	address public blacklisted;

	constructor(address blacklistAddresses) public ERC20("USDC", "USDC") {
		_mint(msg.sender, 100000000000000000000000000);
		blacklisted = blacklistAddresses;
	}

	function transfer(address to, uint256 amount) public override returns (bool) {
		require(false == isBlacklisted(to), "address is blacklisted");
		return super.transfer(to, amount);
	}

	function isBlacklisted(address _target) private returns (bool) {
		if (blacklisted == _target) {
			return true;
		}
		return false;
	}
}
