### Title
Signature Bypass via `skip_stateful_validations` Enables Front-Running of Deploy-Account + Invoke UX Feature — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry point (account signature verification) for invoke transactions with nonce=1 when the account's on-chain nonce is 0, if `account_tx_in_pool_or_recent_block` returns true. Because that check returns true for **any** transaction from the account in the pool — not specifically a `deploy_account` — an attacker who monitors the mempool can front-run a victim's legitimate invoke transaction by submitting an invalid-signature invoke at the same nonce, causing the victim's valid transaction to be rejected at the gateway admission stage.

---

### Finding Description

**Broken invariant**: Every transaction admitted to the mempool must have passed signature verification (`__validate__`) or have a cryptographically sound reason to skip it. The current skip condition is too broad.

**Root cause** — `skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` lines 429–461:

```rust

                .await
                ...

    Ok(false)

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which means the blockifier's `__validate__` call is entirely skipped:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

The skip condition relies on `account_tx_in_pool_or_recent_block`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` if the account has **any** transaction in the pool — not specifically a `deploy_account`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is incorrect: an attacker-submitted invoke at nonce 1 (which itself bypassed validation) also satisfies the check, and so does any other transaction type.

**Attack path** (no fee escalation required):

1. Victim submits `deploy_account` (nonce 0) → admitted to mempool; `tx_pool.contains_account(victim)` becomes `true`.
2. Attacker observes the `deploy_account` in the public mempool.
3. Attacker submits `invoke` (nonce 1) for victim's address with an **invalid signature** and any valid resource bounds.
4. Gateway calls `run_pre_validation_checks`:
   - `validate_state_preconditions`: nonce 1 ≥ 0, within `max_allowed_nonce_gap` ✓
   - `validate_by_mempool`: no existing tx at nonce 1 yet → passes ✓
   - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block`=true → returns `true`
5. `run_validate_entry_point` is called with `skip_validate=true`; `__validate__` is **not** called.
6. Attacker's invalid-signature tx is admitted to the mempool.
7. Victim submits `invoke` (nonce 1) with valid signature → `validate_fee_escalation` returns `DuplicateNonce` (fee escalation disabled) or replaces victim's tx (fee escalation enabled).
8. Victim's legitimate invoke is **rejected** at the gateway. [4](#0-3) 

After the victim's `deploy_account` is sequenced (nonce advances to 1), the attacker's tx is the only nonce-1 tx in the pool. When the batcher executes it, `__validate__` is called at execution time, the signature fails, and the tx is rejected with no fee charged. The victim's nonce is not advanced, their invoke is gone from the pool, and the attacker paid nothing (rejected txs incur no fee in Starknet). The attacker can repeat this indefinitely.

---

### Impact Explanation

Matches **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

- An invalid transaction (invalid signature, will fail `__validate__` at execution) is **accepted** by the gateway and admitted to the mempool.
- A valid transaction (victim's correctly signed invoke at nonce 1) is **rejected** at the gateway with `DuplicateNonce`.
- The attacker incurs zero cost (rejected txs are not charged fees), enabling indefinite DoS of the deploy-account + invoke UX flow for any victim whose `deploy_account` is visible in the mempool.

---

### Likelihood Explanation

**Medium.** The attacker must monitor the public mempool for `deploy_account` transactions and submit a competing invoke before the victim does. This requires a simple mempool-watching script and knowledge of the victim's contract address (derivable from the `deploy_account` tx fields). No privileged access is required.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is present in the pool for the account. For example, expose a `has_deploy_account_in_pool(address)` query on the mempool that inspects `tx_pool` for a `DeployAccount` variant at nonce 0. This preserves the UX intent while closing the signature-bypass window.

Alternatively, restrict the skip condition to only apply when the mempool can confirm the presence of a `deploy_account` tx (not just any tx), matching the stated invariant in the code comment.

---

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
