### Title
First-Swap Watermark Bootstrap Bypass in `OracleValueStopLossExtension` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` stores per-bin high watermarks in `highWatermarks[pool][binIdx]`, which default to `{token0: 0, token1: 0, lastDecayTs: 0}` at pool creation. The `_applyWatermark` helper treats `hwm = 0` as "no prior high" and unconditionally ratchets up to the current metric without ever setting `breached = true`. Because `uint256 metric >= 0` is always true, the first swap on any bin is guaranteed to pass the stop-loss check regardless of how much LP value it drains.

---

### Finding Description

The full call chain on every swap is:

```
pool.swap()
  → MetricOmmPool._afterSwap()          (line 281)
  → ExtensionCalling._afterSwap()       (line 194)
  → OracleValueStopLossExtension.afterSwap()   (line 185)
  → _afterSwapOracleStopLoss()          (line 207)
  → _checkAndUpdateWatermarks()         (line 258)
  → _applyWatermark()                   (line 328)
```

`_decayed` short-circuits on `hwm == 0`:

```solidity
// line 320
if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
```

So `_decayed(0, rate, dt)` always returns `0`. [1](#0-0) 

`_applyWatermark` then receives `hwm = 0`:

```solidity
// line 333
if (metric >= hwm) return (metric, false);   // metric >= 0 is always true for uint256
``` [2](#0-1) 

The function returns `(metric, false)` — `breached = false` — unconditionally. The breach check on lines 271–278 is never reached:

```solidity
(uint256 hwm0, bool breach0) = _applyWatermark(metricT0, _decayed(hwmS.token0, decayRate, dt), floorMultiplier);
if (breach0 && zeroForOne) {
    revert OracleStopLossTriggered(...);   // never fires on first swap
}
``` [3](#0-2) 

After the first swap completes, `hwmS.token0` and `hwmS.token1` are written to the **post-drain** metric values (lines 281–284), permanently anchoring the watermark at the already-drained level. All subsequent swaps are then protected only against further drawdown from that lower baseline. [4](#0-3) 

The `initialize` function stores only the config parameters (`drawdownE6`, `decayPerSecondE8`, `timelock`) and never seeds the watermarks: [5](#0-4) 

The `highWatermarks` mapping therefore remains at the Solidity zero-default for every bin until the first swap touches it: [6](#0-5) 

---

### Impact Explanation

The NatSpec explicitly states:

> "Guarantee: value per share at oracle marks cannot fall faster than drawdown (one-time) + decay * t (ongoing)."

This guarantee is broken on the first swap. An attacker who is the first trader on a pool protected by this extension can drain an arbitrary fraction of LP value in a single transaction — including 90 % or more — with no revert. The watermark is then set to the post-drain level, so the stop-loss only protects against incremental further loss from that point. LPs who deposited expecting the configured drawdown cap to apply from the moment of first trade suffer direct, unrecoverable principal loss.

---

### Likelihood Explanation

Every newly created pool with this extension starts with zero watermarks. The window is open from pool creation until the first swap on each bin. Any public trader can be the first swapper; no privileged role is required. The attacker only needs to submit a sufficiently large swap before any other trade establishes the watermark.

---

### Recommendation

Seed the per-bin watermarks at pool creation time. The `initialize` function should read the current bin states and compute initial metrics, or alternatively treat `hwm == 0` as a sentinel that blocks all swaps (fail-closed) until the admin explicitly sets a baseline via `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks`. The current admin-set path already exists and is timelocked; requiring it before the first swap would close the window without changing the overall architecture.

---

### Proof of Concept

```
1. Deploy pool with OracleValueStopLossExtension; drawdownE6 = 500_000 (50%).
2. LP adds liquidity: bin 0 holds 1_000 token0, 1_000 token1, BIN_SHARES shares.
   highWatermarks[pool][0] = {token0: 0, token1: 0, lastDecayTs: 0}  ← default
3. Attacker executes first swap (zeroForOne = true) draining ~90 % of bin value.
   Post-swap bin state: ~100 token0, ~100 token1.
4. afterSwap hook fires:
   _decayed(0, rate, dt) → 0
   _applyWatermark(metric_post_drain, 0, 500_000)
     → metric_post_drain >= 0 → true → return (metric_post_drain, false)
   breach0 = false → no revert
5. Swap settles. Attacker receives ~900 token1 out.
   highWatermarks[pool][0].token0 = metric_post_drain  ← anchored at drained level
6. Assert: no OracleStopLossTriggered revert was emitted.
   LP has lost ~90 % of principal; stop-loss never fired.
```
