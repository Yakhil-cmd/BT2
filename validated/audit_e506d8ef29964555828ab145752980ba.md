### Title
Unrestricted `update_rewards(disable_rewards: true)` Allows Any Caller to Block All Staker Reward Distribution — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the `Staking` contract is publicly callable by any address and accepts a `disable_rewards` boolean. When called with `disable_rewards: true`, it writes the current block number into the **global** `last_reward_block` storage variable but skips all reward calculation and distribution. Because `last_reward_block` is a single contract-wide value, any subsequent legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`. An unprivileged attacker can call `update_rewards(any_active_staker, disable_rewards: true)` once per block to permanently prevent every staker and pool member from accruing consensus rewards.

---

### Finding Description

`update_rewards` in `IStakingRewardsManager` has no caller access control — only a pause check via `general_prerequisites()`. [1](#0-0) 

The function first asserts the current block is strictly greater than `last_reward_block` (a single, non-keyed storage slot): [2](#0-1) 

It then unconditionally writes the current block number into `last_reward_block` **before** checking `disable_rewards`: [3](#0-2) 

Because the write happens at line 1485 and the early-return guard is at line 1487, any caller who passes `disable_rewards: true` "consumes" the block's reward slot without distributing anything. Every subsequent call in the same block hits the `REWARDS_ALREADY_UPDATED` assertion.

The `last_reward_block` field is a single global value (not a per-staker map), so one poisoned call blocks reward updates for **all** stakers in that block.

---

### Impact Explanation

An attacker who submits `update_rewards(any_active_staker, disable_rewards: true)` once per block causes every staker and every pool member to receive zero consensus rewards for as long as the attack continues. This is a sustained, externally-triggered freeze of unclaimed yield across the entire protocol.

Impact classification: **High — Temporary (or sustained) freezing of unclaimed yield for all stakers and delegators.**

---

### Likelihood Explanation

The attack requires one Starknet transaction per block. Transaction fees on Starknet are low, making this economically viable for a motivated adversary (e.g., a competing protocol, a griefing actor, or a staker who benefits from suppressing competitors' rewards). No privileged key, bridge access, or special role is needed — only the address of any currently active staker, which is publicly readable from chain state.

---

### Recommendation

1. **Add access control** to `update_rewards`: restrict callers to the attestation contract, the staker themselves, or a designated rewards-manager role.
2. **Alternatively**, remove `disable_rewards` from the public ABI entirely and handle the "skip rewards" path through an internal function callable only by trusted contracts.
3. **At minimum**, move the `last_reward_block.write` to occur only when rewards are actually distributed (after the `disable_rewards` guard), so a no-op call does not consume the block slot.

---

### Proof of Concept

```
1. Staker S stakes and becomes active with non-zero balance.
2. Attacker (any EOA) calls:
       update_rewards(staker_address = S, disable_rewards = true)
   at block N.
3. Inside the function:
   a. general_prerequisites() passes (contract not paused).
   b. assert!(N > last_reward_block) passes (first call this block).
   c. last_reward_block is written to N.          ← slot consumed
   d. disable_rewards == true → early return.     ← no rewards distributed
4. The consensus mechanism (or any legitimate caller) attempts:
       update_rewards(staker_address = S, disable_rewards = false)
   at block N → reverts with REWARDS_ALREADY_UPDATED.
5. All stakers miss rewards for block N.
6. Attacker repeats every block → all stakers earn zero rewards indefinitely.
``` [4](#0-3)
