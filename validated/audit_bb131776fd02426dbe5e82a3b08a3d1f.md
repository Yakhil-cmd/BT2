Audit Report

## Title
Missing Caller Authorization in `reset_timers` Allows Any Principal to Consume the Recovery Cooldown Window - (`rs/sns/root/canister/canister.rs`)

## Summary

The `reset_timers` update method on SNS Root performs no caller identity check, allowing any principal (including anonymous) to invoke it once the 7-day cooldown expires. An attacker monitoring the public `get_timers` query can race to call `reset_timers` the moment the cooldown window opens, consuming the recovery slot and blocking any legitimate operator or governance-initiated reset for the next 7 days. This can be repeated indefinitely every 7 days, permanently denying the lightweight timer-recovery path and forcing operators to resort to a full governance-gated canister upgrade to restart a stuck timer.

## Finding Description

`reset_timers` is declared at line 487 with the `#[update]` macro and contains exactly one guard — a cooldown check against `last_reset_timestamp_seconds`:

```rust
#[update]
fn reset_timers(_request: ResetTimersRequest) -> ResetTimersResponse {
    let reset_timers_cool_down_interval_seconds = RESET_TIMERS_COOL_DOWN_INTERVAL.as_secs();
    STATE.with(|state| {
        let state = state.borrow();
        if let Some(timers) = state.timers
            && let Some(last_reset_timestamp_seconds) = timers.last_reset_timestamp_seconds {
                assert!(
                    now_seconds().saturating_sub(last_reset_timestamp_seconds)
                        >= reset_timers_cool_down_interval_seconds, ...
                );
            }
    });
    init_timers();
    ResetTimersResponse {}
}
```

There is no `ic_cdk::api::caller()` check anywhere in the function. Every other state-mutating method on this canister gates on `assert_eq_governance_canister_id`: `change_canister` (line 218), `manage_dapp_canister_settings` (line 403), and the sensitive path of `get_sns_canisters_summary` (line 184) all enforce this. `reset_timers` is the sole exception.

The cooldown constant is one week:

```rust
const RESET_TIMERS_COOL_DOWN_INTERVAL: Duration = Duration::from_secs(60 * 60 * 24 * 7);
```

`init_timers()` (called unconditionally on success) sets `last_reset_timestamp_seconds` to `now_seconds()` and restarts the interval timer. After a successful attacker call, the cooldown is fully reset to the current time, locking out any subsequent caller — including the legitimate governance canister — for another 7 days.

The exploit path is:
1. Attacker reads `last_reset_timestamp_seconds` via the public `get_timers` query.
2. Attacker waits until `now - last_reset_timestamp_seconds >= 604800`.
3. Attacker submits an ingress `reset_timers` call as any principal (including anonymous).
4. `init_timers()` executes: `last_reset_timestamp_seconds` is updated to `now`, the periodic timer is restarted.
5. Any subsequent `reset_timers` call (including from governance) panics with the cooldown error for the next 7 days.
6. Attacker repeats every 7 days, permanently occupying the recovery slot.

If the periodic timer subsequently becomes stuck (e.g., after a canister upgrade clears timer state, or due to a trap in `run_periodic_tasks`), the only non-upgrade recovery path is blocked. The operator is forced to submit a governance upgrade proposal — a significantly heavier process — to call `canister_init` → `init_timers()` via `post_upgrade`.

## Impact Explanation

The periodic timer drives `run_periodic_tasks` → `SnsRootCanister::poll_for_new_archive_canisters`, which is the mechanism by which SNS Root discovers and registers new ledger archive canisters. A stuck timer with a consumed recovery window means archive canister discovery is halted for up to 7 days per attacker cycle. This constitutes an application-level DoS on a core SNS framework function — specifically the archive management subsystem — with concrete protocol harm: newly spawned archive canisters will not be registered with SNS Root, making them invisible to `get_sns_canisters_summary` and ineligible for governance-managed upgrades during the outage window.

This matches the allowed High impact: **"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS"** and **"Significant... SNS... security impact with concrete user or protocol harm."**

## Likelihood Explanation

The attack requires no privileges, no key material, and no social engineering. The SNS Root canister ID is public. The `get_timers` query is unauthenticated. The attacker needs only to monitor the timer state and submit a single ingress message at the right moment. This is fully automatable by a script. The attack is repeatable every 7 days with zero marginal cost.

## Recommendation

Add a caller authorization check at the top of `reset_timers`, consistent with every other state-mutating method on this canister:

```rust
#[update]
fn reset_timers(_request: ResetTimersRequest) -> ResetTimersResponse {
    assert_eq_governance_canister_id(PrincipalId(ic_cdk::api::caller()));
    // ... existing cooldown check ...
    init_timers();
    ResetTimersResponse {}
}
```

Optionally also allow the NNS root canister as an authorized caller, mirroring the pattern used elsewhere in the SNS framework.

## Proof of Concept

State-machine / PocketIC test outline:

1. Initialize SNS Root with a known governance canister ID.
2. Advance time by `ONE_WEEK_SECONDS` (≥ `RESET_TIMERS_COOL_DOWN_INTERVAL`).
3. Call `reset_timers` as the **anonymous principal** — assert it succeeds and `last_reset_timestamp_seconds` is updated to approximately `now`.
4. Immediately call `reset_timers` again as the anonymous principal — assert it panics with the cooldown error.
5. Call `reset_timers` as the legitimate governance canister — assert it also panics (cooldown consumed by attacker in step 3).
6. Confirm that the 7-day window is now locked out for all callers, including governance.

The absence of any `caller()` check at lines 487–506 makes step 3 succeed unconditionally for any principal, directly confirming the vulnerability.
