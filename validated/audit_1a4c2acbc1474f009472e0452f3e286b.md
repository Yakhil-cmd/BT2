### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Emergency Circuit-Breaker Is Permanently Non-Functional - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The Cronos module defines `MsgTurnBridge` as the privileged emergency circuit-breaker for the Gravity Bridge, protected by the `CanTurnBridge` permission. However, the actual handler implementation is an empty stub that returns `nil, nil` unconditionally. No permission check is performed, no bridge state is written, and the bridge cannot be disabled regardless of who calls the message. This is the direct analog to the external report's missing `pause()`/`unpause()` implementation.

### Finding Description

ADR-009 explicitly designates `MsgTurnBridge` as one of two restricted messages requiring the `CanTurnBridge` permission bit. The permission infrastructure is fully built: `CanTurnBridge` is defined as a permission constant, `HasPermission` checks it, `UpdatePermissions` grants it, and the CLI command `CmdTurnBridge` is wired up. The proto definition, message type, and gRPC handler registration are all present.

However, the actual server-side handler in `x/cronos/keeper/msg_server.go` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

The function body is completely empty. It performs no permission check, writes no state, and returns success (`nil` error) unconditionally. Compare this to `UpdateTokenMapping`, which correctly calls `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` before acting: [2](#0-1) 

The `CanTurnBridge` permission constant is defined and the `HasPermission` function is implemented, but `TurnBridge` never calls either: [3](#0-2) 

The integration test `test_gravity_turn_bridge` calls `turn_bridge("false", from_="validator")` and asserts `rsp["code"] == 0`, then expects bridge operations to fail — but since the handler does nothing, the bridge state is never actually changed: [4](#0-3) 

### Impact Explanation

**High — Bypass of bridge authorization checks + permanent inability to disable the bridge.**

Two distinct impacts:

1. **Authorization bypass**: Any unprivileged address can call `MsgTurnBridge` and receive a success response (`code == 0`), bypassing the `CanTurnBridge` permission gate entirely. The permission system for this message is dead code.

2. **Permanent inability to disable the bridge**: The Gravity Bridge emergency circuit-breaker is non-functional. In the event of a critical vulnerability in the bridge (e.g., malicious supply manipulation, unauthorized minting, or a relay exploit), the admin and all permissioned addresses are unable to halt bridge operations. The bridge remains permanently active regardless of any `MsgTurnBridge` call. This directly matches the "High: Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows" impact category.

### Likelihood Explanation

The missing implementation is reachable via any standard Cosmos SDK transaction. No special conditions are required. The CLI command `CmdTurnBridge` is publicly documented and accessible. The impact is realized the moment an emergency arises and the admin attempts to disable the bridge — the call succeeds silently but has no effect.

### Recommendation

Implement the `TurnBridge` handler body to:
1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if it fails.
2. Update the Gravity Bridge `bridge_active` parameter (or equivalent state) to `msg.Enable`.
3. Emit an appropriate event.

The pattern from `UpdateTokenMapping` should be followed directly.

### Proof of Concept

1. Any address submits `MsgTurnBridge{Sender: <any_address>, Enable: false}`.
2. The handler at `x/cronos/keeper/msg_server.go:85-87` executes `return nil, nil`.
3. The transaction succeeds with `code == 0` — no permission check was performed.
4. No bridge state is written. The Gravity Bridge remains fully active.
5. A permissioned admin subsequently calls `MsgTurnBridge{Sender: <admin>, Enable: false}` in response to an emergency — same result: success response, no state change, bridge still active.
6. Bridge operations (e.g., `send_to_evm_chain`, `submitBatch`) continue to execute normally, as the bridge was never actually disabled. [1](#0-0) [5](#0-4) [6](#0-5)

// HasPermission check if an account has a specific permission. by default cronos admin has all permissions
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
}
```
