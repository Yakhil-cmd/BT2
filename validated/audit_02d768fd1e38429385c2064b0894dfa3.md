### Title
Integer Division Truncation in `native_per_pubdata` Silently Zeroes Pubdata Cost When `pubdata_price < native_price` — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` conversion uses floor integer division (`wrapping_div`). When the operator-supplied `pubdata_price` is smaller than `native_price`, the result truncates to zero. Every downstream pubdata charge then multiplies by zero, making all pubdata — both intrinsic and execution — completely free for any transaction sender. This is a resource-accounting bug directly analogous to the external report's hardcoded-precision issue: just as a fixed `1e12` threshold ignores token decimals, a fixed floor-division ignores the sub-unit ratio between the two price axes.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` the two key per-unit rates are computed as:

```rust
// line 135 — uses ceiling division: always ≥ 1 when gas_price > 0
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))…;

// line 142 — uses FLOOR division: silently 0 when pubdata_price < native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same floor-division pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

`native_per_pubdata` flows into two charging sites:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [3](#0-2) 

2. **Execution pubdata charge** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)…;
   ``` [4](#0-3) 

When `native_per_pubdata = 0`, both products are zero regardless of how many pubdata bytes the transaction actually writes. The `native_per_gas` path correctly uses `div_ceil` to guarantee at least one native unit per gas unit; the `native_per_pubdata` path has no equivalent safeguard.

There is no system-level invariant enforcing `pubdata_price ≥ native_price`. The block metadata simply stores whatever the operator provides:

```rust
fn get_pubdata_price(&self) -> U256 { self.pubdata_price }
fn native_price(&self) -> U256 { self.native_price }
``` [5](#0-4) 

The default test configuration already demonstrates the condition: `pubdata_price = 0`, `native_price = 10`, giving `native_per_pubdata = 0`. [6](#0-5) 

---

### Impact Explanation

**Vulnerability class**: Resource accounting bug / state-transition correctness.

When `pubdata_price < native_price` the entire pubdata cost axis collapses to zero:

- Intrinsic pubdata (`L2_TX_INTRINSIC_PUBDATA` ≈ 134 bytes, `L1_TX_INTRINSIC_PUBDATA` ≈ 319 bytes) is uncharged.
- Every storage write a transaction makes is uncharged for L1 data-availability cost.
- An unprivileged transaction sender can spam storage writes — each costing only EVM gas, not native pubdata — bloating the L1 DA layer without paying for it.
- The operator bears the L1 DA cost but collects zero compensation from users, creating a direct financial loss path.

The `MAX_NATIVE_COMPUTATIONAL` cap (`2^35`) does not protect against this because pubdata native is withheld *separately* from computational native; with `native_per_pubdata = 0` the withheld amount is also zero. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The operator supplies `native_price` and `pubdata_price` as block-context oracle inputs. No on-chain or in-code constraint enforces `pubdata_price ≥ native_price`. A legitimate configuration — e.g. `native_price = 100` (wei per native unit), `pubdata_price = 50` (wei per pubdata byte) — silently zeroes all pubdata charges. The operator need not be malicious; the truncation is a silent precision loss that any sub-unit ratio triggers. The condition is also the default in the test harness (`pubdata_price = 0`), confirming it is a reachable, accepted input range.

---

### Recommendation

Replace the floor division with ceiling division for `native_per_pubdata`, consistent with `native_per_gas`:

```rust
// Before (floor — truncates to 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))…;

// After (ceiling — mirrors native_per_gas treatment)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))…;
```

Apply the same fix in `api/src/helpers.rs` line 427. Optionally add an explicit check: if `pubdata_price > 0` then `native_per_pubdata` must be `≥ 1`, returning a validation error otherwise.

---

### Proof of Concept

**Setup**:
- `native_price = 100`
- `pubdata_price = 50`
- `eip1559_basefee = 1`
- Transaction: EIP-1559, `max_fee_per_gas = 200`, `gas_limit = 500_000`, body writes to 20 distinct storage slots (≈ 640 bytes of pubdata).

**Trace**:

1. `native_per_gas = ceil(200 / 100) = 2` ✓ (correctly non-zero)
2. `native_per_pubdata = floor(50 / 100) = 0` ← truncation
3. `intrinsic_pubdata_overhead = 0 * 134 = 0` → full `native_prepaid_from_gas` available for computation
4. After execution, `get_resources_to_charge_for_pubdata(native_per_pubdata=0, …)` returns `(640, 0 native)` → no pubdata charge deducted
5. Sender pays only for EVM gas; 640 bytes of L1 DA are published at zero cost to the sender.

Repeating this across many transactions in a block allows an attacker to saturate the pubdata limit (`pubdata_limit` from block metadata) at negligible cost, forcing the operator to pay full L1 DA fees with no user-side recovery. [8](#0-7) [9](#0-8)
