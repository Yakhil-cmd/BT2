### Title
WalletConnect `chia_signMessageById` Confirmation Dialog Does Not Display the Signing Wallet Key Fingerprint - (File: `packages/gui/src/electron/dialogs/Confirm/Confirm.tsx`, `packages/gui/src/electron/commands/Commands.ts`)

### Summary
When a paired dapp issues a `chia_signMessageById` (or `chia_signMessageByAddress`) RPC call, the confirmation dialog shown to the user does not always display which wallet key fingerprint will be used to produce the signature. The dialog only surfaces a "Different wallet key" warning when the dapp-requested fingerprint differs from the currently logged-in one. When the dapp targets the currently active fingerprint, no fingerprint context is shown at all. A malicious dapp that has enumerated the user's NFT/DID IDs can request signing under any identity the user owns without the user being able to verify which wallet key is actually signing.

### Finding Description

The `chia_wallet.sign_message_by_id` command schema in `Commands.ts` exposes only `id` and `message` as visible confirmation rows: [1](#0-0) 

The `Confirm.tsx` dialog renders a "Different wallet key" banner only when `isDifferentFingerprint` is `true`, which requires both `fingerprint.requested` and `fingerprint.current` to be defined **and** unequal: [2](#0-1) 

When the dapp targets the currently logged-in fingerprint, `isDifferentFingerprint` is `false` and no wallet key information is rendered anywhere in the dialog: [3](#0-2) 

The `fingerprint` prop itself is optional: [4](#0-3) 

For `sign_message_by_id`, the NFT/DID `id` is shown, but the user has no way to see which wallet key fingerprint owns that NFT/DID from within the dialog. A user with multiple wallet keys may not immediately know which key is implicitly selected.

For `sign_message_by_address`, the `address` param is shown: [5](#0-4) 

This partially mitigates the issue for that command, but the wallet key fingerprint is still not shown.

### Impact Explanation

A malicious dapp paired with the user's Chia wallet can call `chia_signMessageById` with an NFT ID or DID ID belonging to a wallet key the user does not expect to be signing with. The confirmation dialog shows only the NFT/DID ID and the message content — not the wallet key fingerprint that will produce the signature. The user approves believing they are signing with one identity, but the backend resolves and signs under a different wallet key. This constitutes signing-context confusion: the user produces a valid cryptographic signature under a wallet key/DID/NFT identity they did not intend to authorize.

### Likelihood Explanation

Any dapp that has been granted the `chia_signMessageById` permission (classified as a "Sign messages" capability in the pairing flow) can trigger this. The pairing dialog does warn that the dapp "signs text with your key": [6](#0-5) 

But once paired, the per-command confirmation dialog does not reinforce which key is signing. A dapp that also holds `chia_getWallets` or read permissions can enumerate the user's NFT/DID IDs and target a specific one.

### Recommendation

The `Confirm.tsx` dialog for sign commands should unconditionally display the active wallet key fingerprint (and label if available), not only when it differs from the current session. For `sign_message_by_id`, the dialog should additionally resolve and display the owning wallet key fingerprint alongside the NFT/DID ID so the user can verify the full signing context before approving.

### Proof of Concept

1. Pair a dapp with the Chia GUI, granting `chia_signMessageById` permission.
2. From the dapp, enumerate the user's DID IDs (e.g., via `chia_getWallets`).
3. Call `chia_signMessageById` with a DID ID belonging to a wallet key the user does not expect.
4. Observe the confirmation dialog: it shows the DID ID and message, but no wallet key fingerprint.
5. The user approves; the daemon signs under the wallet key that owns the DID, which may differ from the user's expectation. [1](#0-0) [7](#0-6)
