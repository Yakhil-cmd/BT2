### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Unclaimed Yield — (`File: src/staking/staking.cairo`)

### Summary

`update_rewards` is documented as callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for that block and preventing the sequencer from distributing rewards in that block. Repeated across blocks, this permanently freezes unclaimed yield for all stakers and delegators.

### Finding Description

`update_rewards` is the V3 consensus-rewards entry point. Its spec explicitly states access control is "Only starkware sequencer." However, the implementation only calls `general_prerequisites()`, which checks for pause state and a non-zero caller — no sequencer identity check exists. [1](#0-0) 

`general_prerequisites` is: [2](#0-1) 

The function writes the current block number into the **global** `last_reward_block` storage variable immediately after the balance checks, before any reward distribution: [3](#0-2) 

When `disable_rewards: true` is passed, the function returns immediately after writing `last_reward_block`, distributing nothing: [4](#0-3) 

`last_reward_block` is a single global field — not per-staker: [5](#0-4) 

The spec confirms the intended restriction: [6](#0-5) 

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in a given block sets `last_reward_block` to that block number. Any subsequent call in the same block — including the sequencer's legitimate call — reverts with `REWARDS_ALREADY_UPDATED`. If the attacker does this in every block, no staker or delegator ever accumulates rewards. This constitutes **permanent freezing of unclaimed yield** for the entire protocol.

### Likelihood Explanation

Active staker addresses are publicly enumerable from the `stakers` vector and from emitted `NewStaker` events. The attacker needs only a valid active staker address and a non-zero caller address. The call itself is cheap (no token transfer, no approval). On Starknet, any user can submit transactions; if the sequencer does not atomically guarantee its own `update_rewards` call is the first transaction in every block, the attacker can race it. The attack requires no privileged access, no leaked keys, and no external dependency.

### Recommendation

Add a sequencer-only access control check to `update_rewards`. Store the authorized sequencer address in contract storage (or derive it from a roles component) and assert `get_caller_address() == sequencer_address` at the top of `update_rewards`, analogous to how `assert_caller_is_attestation_contract` is enforced for `update_rewards_from_attestation_contract`. [7](#0-6) 

### Proof of Concept

```
// Attacker script (pseudocode, one call per block):
loop {
    // Pick any known active staker (e.g., from NewStaker events)
    let victim_staker = known_active_staker_address;
    // Call update_rewards with disable_rewards = true
    staking.update_rewards(victim_staker, disable_rewards: true);
    // last_reward_block is now set to current block
    // Sequencer's update_rewards call in the same block will revert:
    //   assert!(current_block_number > self.last_reward_block.read(), REWARDS_ALREADY_UPDATED)
    wait_for_next_block();
}
// Result: no staker ever receives consensus rewards; all unclaimed yield is frozen.
```
