### Title
Permanent Fund Loss on Failed `ExitToNear` Precompile Call When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` Cargo feature is not compiled in (which is the default build configuration), a failed `ExitToNear` precompile call — triggered by any EVM user bridging ETH or ERC-20 tokens from Aurora to NEAR — results in permanent, unrecoverable loss of those funds. The refund callback is unconditionally set to `None` at compile time, so no recovery path exists if the downstream NEAR promise fails.

### Finding Description

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` bridges EVM assets to NEAR by:

1. Deducting ETH from the caller's EVM balance (for base-token exits) or burning ERC-20 tokens (for ERC-20 exits).
2. Scheduling a NEAR cross-contract promise (`ft_transfer` / `ft_transfer_call` / `near_withdraw`) to deliver the assets on the NEAR side.
3. Optionally attaching a callback (`exit_to_near_precompile_callback`) to handle promise failure and refund the user.

Step 3 is gated entirely behind the `error_refund` Cargo feature flag:

```rust
// engine-precompiles/src/native.rs  lines 449-453
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in the default build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `refund` is `None` and `transfer_near` is also `None` (the common case for plain ETH and ERC-20 exits), `callback_args` equals the `Default` value, so the promise is created **without any callback**:

```rust
// lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

Even in the wNEAR-unwrap path where a callback is attached, the callback's `refund` field is still `None`, so `exit_to_near_precompile_callback` takes the `else { None }` branch and performs no recovery:

```rust
// engine/src/contract_methods/connector.rs  lines 231-242
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // ← no refund, funds are gone
};
``` [3](#0-2) 

The `error_refund` feature is **not** in the `default` feature set of the engine crate:

```toml
# engine/Cargo.toml  lines 43-48
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

The test suite itself explicitly acknowledges the fund loss when the feature is absent:

```rust
// engine-tests/src/tests/erc20_connector.rs  lines 771-775
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing / loss of user funds.**

- **ETH base-token exit**: The ETH is transferred from the caller's EVM balance to the `ExitToNear` precompile address before the NEAR promise is dispatched. If the promise fails (e.g., recipient NEAR account not registered, insufficient storage deposit, account does not exist), the ETH remains locked in the precompile address with no recovery mechanism. There is no `refund_on_error` call and no way for the user to reclaim it.
- **ERC-20 exit**: The ERC-20 tokens are burned on the Aurora side before the `ft_transfer` promise is dispatched. If the promise fails, the tokens are permanently destroyed with no re-mint path. [6](#0-5) 

### Likelihood Explanation

The failure condition is easily triggered by any unprivileged EVM user:

- Sending an `ExitToNear` call targeting a NEAR account that is not registered with the NEP-141 token contract (storage deposit not paid) causes `ft_transfer` to fail — a common real-world scenario.
- The `error_refund` feature is not in the default build, so every production deployment compiled without explicitly enabling it is affected.
- The test `test_exit_to_near_refund` and `test_exit_to_near_eth_refund` demonstrate the exact failure scenario is reachable and results in fund loss. [7](#0-6) 

### Recommendation

Enable the `error_refund` feature unconditionally in the production build (add it to the `default` feature set in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`), or remove the compile-time gate entirely and always populate `refund` with the correct `RefundCallArgs`. The refund logic in `engine::refund_on_error` already handles both ETH and ERC-20 cases correctly when invoked. [8](#0-7) 

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature (the default).
2. Fund an EVM address with ETH on Aurora.
3. Call the `ExitToNear` precompile (flag `0x00`) targeting a NEAR account that has no storage deposit with the ETH connector (e.g., `"unregistered.near"`).
4. The EVM deducts ETH from the caller's balance and schedules `ft_transfer` on the ETH connector.
5. `ft_transfer` fails because `"unregistered.near"` is not registered.
6. No callback fires; no refund is issued.
7. Observe: caller's EVM ETH balance is permanently reduced by the exit amount; `"unregistered.near"` received nothing; the ETH is irrecoverable.

This is confirmed by the existing test at `engine-tests/src/tests/erc20_connector.rs:717-780`, which explicitly asserts the balance is `INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT` (i.e., funds are lost) when `error_refund` is not enabled. [9](#0-8)

```rust
pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
}
