### Title
`EVM.mustRun()` Silently Accepts Failed EVM Transactions, Enabling Cross-VM Asset Loss - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

`EVM.mustRun()` in the Flow EVM system contract asserts only that an EVM transaction was *executed* (status is not `invalid` or `unknown`), but explicitly does **not** revert when the EVM transaction fails (`status: failed`). Any Cadence transaction or contract that uses `mustRun` to deliver EVM-side tokens to a user before collecting Cadence-side payment can silently skip the delivery while still collecting payment — a direct structural analog to the Liquidator `ERC20Helper.transfer` silent-failure bug.

---

### Finding Description

`EVM.mustRun()` is defined in `fvm/evm/stdlib/contract.cdc` as:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The `assert` condition is `status == failed || status == successful`. This means a `failed` EVM execution (out-of-gas, Solidity `revert`, assertion failure, etc.) satisfies the assertion and the Cadence transaction **continues normally**. The EVM `Status` enum documents this explicitly:

```cadence
/// Returned when execution of an evm transaction/call
/// has been successful but the vm has reported an error in
/// the outcome of execution (e.g. running out of gas).
/// A failed tx/call is included in a block.
access(all) case failed
``` [2](#0-1) 

The emulator confirms that a `failed` result carries a `VMError` (e.g., `ErrOutOfGas`) but does **not** cause the surrounding Cadence execution to revert: [3](#0-2) 

The function is `access(all)` — callable by any unprivileged Cadence transaction or deployed contract.

---

### Impact Explanation

A Cadence contract implementing a cross-VM flash-loan or swap pattern follows this sequence:

1. Call `EVM.mustRun(tx)` where `tx` encodes an EVM `transfer` of tokens to `msg.sender`.
2. Invoke arbitrary Cadence logic on the caller (the "callback" step).
3. Withdraw Cadence-side payment from the caller via `FungibleToken.withdraw` / `transferFrom`.

If step 1 produces `status: failed` (e.g., the EVM token contract reverts because the escrow balance was already drained by a concurrent transaction), `mustRun` does **not** revert. Steps 2 and 3 execute normally. The caller's Cadence assets are debited while zero EVM tokens are delivered — a direct cross-VM asset loss matching the Liquidator scenario.

Because `EVM.run` state changes for a `failed` transaction are **not** committed to the EVM state (the emulator resets on failure), the EVM-side balance is unchanged while the Cadence-side debit is permanent.

---

### Likelihood Explanation

- `EVM.mustRun` is `access(all)` and is the natural choice for any Cadence developer building a cross-VM swap or flash-loan, because its name implies "this must execute" (analogous to Solidity's `require`).
- The misleading name increases the probability that developers omit a post-call `assert(result.status == EVM.Status.successful)` check.
- EVM-side failures are realistic: gas exhaustion, ERC-20 `revert` on insufficient balance, or a race condition where another transaction drains the EVM escrow first.
- No privileged access is required; any unprivileged account can deploy a Cadence contract using `mustRun` and expose users to this loss.

---

### Recommendation

**Short term:** Rename `mustRun` to a name that does not imply success (e.g., `runIfValid`), or add a companion `mustSucceed` function that asserts `status == Status.successful`:

```cadence
access(all)
fun mustSucceed(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.successful,
        message: "EVM.mustSucceed(): EVM transaction did not succeed"
    )
    return runResult
}
```

**Long term:** Audit all Cadence contracts (system and user-deployed) that call `EVM.mustRun`, `EVM.run`, or `CadenceOwnedAccount.call` and verify that every call site checks `result.status == Status.successful` before proceeding with any Cadence-side asset transfer. Ensure the pattern "deliver EVM asset → collect Cadence payment" always reverts atomically on EVM-side failure.

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FungibleToken from <FTAddress>
import FlowToken from <FlowTokenAddress>

// Malicious/buggy cross-VM flash-loan contract
transaction(
    evmTransferTx: [UInt8],   // EVM tx: transfer 100 USDC to msg.sender
    coinbase: EVM.EVMAddress,
    paymentAmount: UFix64     // Cadence FLOW to collect as fee
) {
    prepare(caller: auth(FungibleToken.Withdraw) &Account) {

        // Step 1: "deliver" EVM tokens — may silently fail
        let result = EVM.mustRun(tx: evmTransferTx, coinbase: coinbase)
        // result.status may be EVM.Status.failed here — NO revert occurs

        // Step 2: collect Cadence payment regardless
        let vault = caller.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
                from: /storage/flowTokenVault)!
        let payment <- vault.withdraw(amount: paymentAmount)
        // payment is now permanently debited from caller;
        // caller received zero EVM tokens
        destroy payment
    }
}
```

If the EVM escrow is drained between the caller's balance check and the `mustRun` call (e.g., by a concurrent transaction), `mustRun` returns `status: failed`, the Cadence transaction continues, and the caller's `paymentAmount` FLOW is destroyed with no EVM tokens delivered. [4](#0-3) [5](#0-4)
