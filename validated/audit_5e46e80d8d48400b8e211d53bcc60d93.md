### Title
Asymmetric Rounding in `native_per_pubdata` Causes Systematic Operator Underpayment for Pubdata Costs - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed with **floor (truncating) division** while `native_per_gas` uses **ceiling division**. This asymmetry causes every transaction that produces pubdata to systematically underpay the operator for pubdata processing costs. The underpayment can reach up to `(native_price − 1)` native units per pubdata byte — in extreme configurations, nearly the entire pubdata cost is unrecovered.

---

### Finding Description

In `validation_impl.rs` line 142, the ratio of pubdata cost to native cost is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` performs **integer floor division**, silently discarding the fractional remainder.

In contrast, `native_per_gas` at line 135 uses **ceiling division**:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

The same floor-division pattern appears in the public API helper at `api/src/helpers.rs` line 427:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

`native_per_pubdata` is then used in two critical places:

1. **`create_resources_for_tx`** (`gas_helpers.rs` line 352): deducts intrinsic pubdata overhead from the transaction's native budget.
2. **`get_resources_to_charge_for_pubdata`** (`gas_helpers.rs` line 430–432): charges native resources for actual pubdata bytes consumed.

Because `native_per_pubdata` is truncated downward, the native resource charged per pubdata byte is less than the true cost ratio. This reduces `native_used` in `compute_gas_refund` (`refund_calculation.rs` line 64), which in turn reduces `delta_gas` (line 72), which reduces `gas_used` (line 78), and therefore reduces the token fee transferred to the operator (line 514 of `zk/mod.rs`).

The full propagation chain:

```
native_per_pubdata (truncated)
  → less native charged per pubdata byte
  → lower native_used
  → lower delta_gas = (native_used / native_per_gas) - gas_used
  → lower gas_used
  → lower pay_to_operator = gas_used * gas_price
```

---

### Impact Explanation

**Operator/prover is systematically underpaid for pubdata costs.** The magnitude of underpayment per pubdata byte is:

```
error_per_byte = (pubdata_price mod native_price) / native_price
```

Concrete example with `pubdata_price = 19`, `native_price = 10`:
- `native_per_pubdata = floor(19/10) = 1` (true ratio: 1.9)
- Truncation error: 0.9 native units per pubdata byte (47% of actual cost)
- For a transaction producing 100 pubdata bytes: 90 native units unrecovered
- With `native_per_gas = 2`, `gas_price = 20`: operator underpaid by `(90/2) × 20 = 900` base-token units per 100 pubdata bytes

At scale (e.g., 100 KB pubdata): operator underpaid by ~900,000 base-token units per transaction.

Additionally, when `pubdata_price < native_price`, `native_per_pubdata = 0`, making pubdata entirely free in native resource terms — a degenerate case of the same root cause.

---

### Likelihood Explanation

**High.** Every L2 transaction that produces pubdata (i.e., virtually all transactions that write storage) is affected. The truncation is deterministic and occurs whenever `pubdata_price` is not an exact multiple of `native_price`. Both values are operator-set block-level parameters, and non-divisible combinations are the common case. No special attacker action is required — submitting any normal transaction with storage writes is sufficient to trigger the underpayment.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` to match the conservative rounding used for `native_per_gas`:

```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures the native resource charged per pubdata byte is never less than the true cost ratio, consistent with the ceiling-division approach already used for `native_per_gas`.

### Proof of Concept

**Setup:** `pubdata_price = 19`, `native_price = 10`, `gas_price = 20`, `gas_limit = 100_000`.

**Step 1 — `native_per_gas` (ceiling):**
`ceil(20 / 10) = 2` → `native_per_gas = 2`

**Step 2 — `native_per_pubdata` (floor, buggy):**
`floor(19 / 10) = 1` → `native_per_pubdata = 1`
Correct value would be `ceil(19 / 10) = 2`.

**Step 3 — Transaction produces 100 pubdata bytes:**
- Native charged for pubdata: `100 × 1 = 100` (should be `100 × 2 = 200`)
- `native_used` is 100 units lower than it should be.

**Step 4 — `compute_gas_refund` delta_gas:**
- Actual `delta_gas` contribution from pubdata: `100 / 2 = 50` gas units
- Correct contribution: `200 / 2 = 100` gas units
- `gas_used` is 50 units lower than it should be.

**Step 5 — Operator payment:**
- Actual: `gas_used × 20` (50 gas units short)
- Shortfall: `50 × 20 = 1,000` base-token units per 100 pubdata bytes
