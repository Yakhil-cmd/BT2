### Title
BTC Pool `change_reward_address` / `enter_delegation_pool` Incomplete Token Check Allows Permanent Freezing of Unclaimed STRK Rewards - (File: src/pool/pool.cairo)

### Summary

The `Pool` contract's `enter_delegation_pool` and `change_reward_address` functions validate the reward address only against the pool's own staking token (e.g., BTC). They do not validate against `STRK_TOKEN_ADDRESS`, which is the token used to pay all pool rewards. A BTC pool member can therefore set their reward address to the STRK token contract address. When `claim_rewards` is subsequently called, the STRK reward tokens are transferred to the STRK token contract itself, permanently freezing the unclaimed yield.

### Finding Description

In `pool.cairo`, both entry points that accept or update a reward address perform the following check:

**`enter_delegation_pool` (line 193–194):**
```cairo
let token_address = token_dispatcher.contract_address;
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

**`change_reward_address` (line 506–510):**
```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,

```

For a BTC pool, `self.token_dispatcher.contract_address` is the BTC token address. The guard therefore only rejects `reward_address == BTC_TOKEN_ADDRESS`. It does **not** reject `reward_address == STRK_TOKEN_ADDRESS`.
