### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Nonce-1 Invokes, Allowing Invalid Transactions into Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo gateway omits the `__validate__` entry-point call (account signature verification) for any `Invoke` transaction with `nonce=1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender. Because that check returns `true` for **any** transaction in the pool — not specifically a `DeployAccount` — an attacker who observes a pending `DeployAccount` in the mempool can submit a competing `Invoke(nonce=1)` with a garbage signature, bypass signature verification, and have the invalid transaction admitted to the mempool. Via fee escalation, the attacker can silently replace the legitimate user's first invoke with an invalid one at zero cost.

---

### Finding Description

`skip_stateful_validations` is the Sequencer-native analog of the EIP-777 `tokensReceived` hook omission: a mandatory validation step (`__validate__`) is intentionally skipped under a condition that is too broad.

**The skip condition** (lines 437–456 of `stateful_transaction_validator.rs`):

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, so the account's `__validate__` entry point — the only place the signature is verified — is never executed:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

**The overly broad check** — `account_tx_in_pool_or_recent_block` returns `true` if the account has *any* transaction in the pool or in the mempool's staged/committed state:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

The inline comment claims this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The first half is correct for the `tx_pool` case. The second half is the flaw: once a `DeployAccount` is in the pool, the pool check returns `true`, and any attacker-submitted `Invoke(nonce=1)` with a garbage signature also passes the check and skips `__validate__`.

**Fee escalation completes the attack.** The mempool's `validate_fee_escalation` path performs no signature check; it only requires the replacement transaction to offer a higher fee:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [4](#0-3) 

So the attacker's garbage-signature invoke, if it offers a higher fee, silently evicts the legitimate invoke from the mempool.

**Zero cost to attacker.** When the batcher later executes the attacker's invoke, `__validate__` is called and fails. In Starknet, a failed `__validate__` reverts the transaction including the nonce increment. If the account has no balance (freshly deployed), the fee transfer also fails, so the attacker pays nothing.

---

### Impact Explanation

This matches the **High** impact tier: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concretely:
- An `Invoke` transaction with an **invalid/garbage signature** is admitted to the mempool without signature verification.
- Via fee escalation, the attacker can **replace the legitimate user's first invoke** with the invalid one.
- The legitimate invoke is permanently lost from the mempool; the user must resubmit.
- The attack is **repeatable and free**: the attacker pays no fee because the account has no balance when the invalid invoke is executed.
- This enables a sustained, targeted DoS against any user who submits a `DeployAccount`.

---

### Likelihood Explanation

- **Trigger is unprivileged**: any party that can submit transactions to the gateway (or observe the P2P mempool) can execute this attack.
- **Observable precondition**: `DeployAccount` transactions are broadcast over P2P; an attacker monitoring the network can detect them.
- **Timing window**: the window is open from the moment the `DeployAccount` enters `tx_pool` until the block containing it is committed — typically several seconds to minutes.
- **No existing guard**: neither the gateway's stateless validator, the mempool's `validate_tx`, nor the fee-escalation path checks signatures.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction exists in the mempool for the sender address. For example, expose a `has_pending_deploy_account(address)` query on the mempool that inspects the transaction type, and use that instead of the generic presence check.

Alternatively, restrict the skip to only the `tx_pool` path (not the `staged`/`committed` state path), and add a type assertion that the pooled transaction is a `DeployAccount`.
