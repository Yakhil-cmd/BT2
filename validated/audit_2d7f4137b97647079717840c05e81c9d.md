### Title
Gateway Stateful Validator Only Checks L2 Gas Price, Admitting Transactions That Fail Blockifier Pre-Validation on L1/L1Data Gas Price - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful resource-bounds check (`validate_resource_bounds` → `validate_tx_l2_gas_price_within_threshold`) validates only the transaction's `l2_gas.max_price_per_unit` against the previous block's L2 gas price. It never checks `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit` against the corresponding on-chain prices. The blockifier's `check_fee_bounds`, called during block building, checks all three resource types. A transaction with a deliberately low L1 or L1-data gas price therefore passes gateway admission and enters the mempool, but is rejected by the blockifier during every block-building attempt.

---

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` fetches only `strk_gas_prices.l2_gas_price` from the previous block and passes it to `validate_tx_l2_gas_price_within_threshold`:


The blockifier's `check_fee_bounds`, executed during block building, checks all three resources for `AllResources` transactions:

```rust

// … then checks max_price_per_unit >= actual_gas_price for each
```

The gateway checks one resource; the blockifier checks three. The gap is the missing L1 and L1-data price checks at admission time.

---

### Impact Explanation

**High — Mempool/gateway admission accepts transactions that will be rejected by the blockifier on every block-building attempt.**

An attacker can flood the mempool with `AllResources` transactions that set `l2_gas.max_price_per_unit` just above the gateway threshold while setting `l1_gas.max_price_per_unit = 0` and `l1_data_gas.max_price_per_unit = 0`. Every such transaction:

1. Passes all gateway checks (stateless and stateful).
2. Enters the mempool.
3. Is pulled by the batcher and handed to the blockifier.
4. Fails `check_fee_bounds` with `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }`.
5. Is never included in a block but consumes batcher/blockifier CPU on every proposal.

This constitutes a sustained, low-cost denial-of-service against the block-building pipeline and wastes sequencer resources proportional to the number of admitted-but-unexecutable transactions.

---

### Likelihood Explanation

The attack requires only a valid Starknet account and the ability to submit `AllResources` transactions with crafted resource bounds. No privileged access, special contract, or chain-specific knowledge is needed beyond knowing the current L2 gas price (publicly observable). The TODO comment confirms the gap is known but unresolved.

---

### Recommendation

In `validate_resource_bounds`, fetch all three STRK gas prices from the previous block and extend `validate_tx_l2_gas_price_within_threshold` (or replace it with a new function) to apply the same percentage-threshold check to `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` for `AllResources` transactions. The blockifier already has the correct multi-resource loop in `check_fee_bounds` that can serve as a reference.

---

### Proof of Concept

1. Observe the current block's STRK gas prices: `l2_gas_price = P_l2`, `l1_gas_price = P_l1`, `l1_data_gas_price = P_l1d`.
2. Craft an `AllResources` invoke transaction with:
   - `l2_gas.max_price_per_unit = P_l2` (meets the gateway's `min_gas_price_percentage` threshold)
   - `l1_gas.max_price_per_unit = 0` (below `P_l1`, not checked by gateway)
   - `l1_data_gas.max_price_per_unit = 0` (below `P_l1d`, not checked by gateway)
   - `l2_gas.max_amount` set to any non-zero value
3. Submit via `POST /add_transaction`.
4. **Gateway result:** `Ok(tx_hash)` — transaction admitted.
   - `validate_tx_l2_gas_price_within_threshold` passes because `l2_gas.max_price_per_unit >= threshold`.
5. **Blockifier result (during block building):** `InsufficientResourceBounds { errors: [MaxGasPriceTooLow { resource: L1Gas, max_gas_price: 0, actual_gas_price: P_l1 }] }`.
   - `check_fee_bounds` at `account_transaction.rs:441` rejects the transaction.
6. Repeat at high frequency to saturate the batcher's execution pipeline. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
```
