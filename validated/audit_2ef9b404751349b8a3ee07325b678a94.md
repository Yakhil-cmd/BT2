### Title
Hardcoded 2,300-Gas Stipend in `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` Permanently Blocks FLOW Token Deposits to EVM Smart Contracts with Non-Trivial `receive()` Functions - (File: fvm/evm/types/call.go)

### Summary

`fvm/evm/types/call.go` defines `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` as `21,000 + 2,300 = 23,300` gas — the exact same hardcoded stipend as Solidity's deprecated `transfer()`. Any EVM smart contract whose `receive()` or `fallback()` function consumes more than 2,300 gas will permanently fail to receive FLOW tokens via the Cadence bridge. The failure propagates as a panic through `panicOnErrorOrInvalidOrFailedState`, aborting the entire enclosing Cadence transaction.

### Finding Description

In `fvm/evm/types/call.go`, the gas constants are defined as:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas  // 21,000

// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23,300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

The comment itself acknowledges the 2,300 figure is the "max gas allowed for receive/fallback methods" — the same stipend Solidity's `transfer()` forwards. `NewDepositCall` and `NewTransferCall` both embed this limit unconditionally: [2](#0-1) 

`Account.Deposit()` in `fvm/evm/handler/handler.go` calls `NewDepositCall` with `DepositCallGasLimit`: [3](#0-2) 

`Account.Transfer()` calls `NewTransferCall` with `DefaultGasLimitForTokenTransfer`: [4](#0-3) 

Both functions pass the result to `panicOnErrorOrInvalidOrFailedState`, which panics on any `Failed` or `Invalid` result: [5](#0-4) 

The panic propagates through the Cadence runtime, aborting the entire enclosing transaction. The existing test suite already documents the revert case for a contract with no `receive()` function: [6](#0-5) 

The same failure occurs for any contract whose `receive()` or `fallback()` function requires more than 2,300 gas — a common pattern in ERC-4626 vaults, contracts that emit events on receive, or contracts that update accounting state on receive.

The Cadence-facing entry point is `EVMAddress.deposit()` in `fvm/evm/stdlib/contract.cdc`, which calls `InternalEVM.deposit()` → `account.Deposit()`: [7](#0-6) 

`CadenceOwnedAccount.deposit()` delegates to the same path: [8](#0-7) 

### Impact Explanation

Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function consumes more than 2,300 gas is permanently unable to receive FLOW tokens via the Cadence bridge. Every Cadence transaction that calls `EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()` targeting such a contract will panic and be aborted. Tokens are not lost (the Cadence vault is not destroyed before the panic), but the deposit operation is permanently blocked for the affected contract address. This is a denial-of-service against a class of EVM smart contracts that is common in DeFi (vaults, yield aggregators, contracts with on-receive accounting).

### Likelihood Explanation

The attacker-controlled entry path is a standard unprivileged Cadence transaction calling `EVMAddress.deposit()`. No special privileges are required. Any user who deploys an EVM smart contract with a non-trivial `receive()` function and then attempts to fund it via the Cadence bridge will trigger this failure deterministically and permanently. The pattern is widespread in production EVM contracts.

### Recommendation

1. Expose a `gasLimit` parameter on `EVMAddress.deposit()` and `CadenceOwnedAccount.deposit()` (analogous to how `CadenceOwnedAccount.call()` already accepts a caller-specified `gasLimit`), so callers can provide sufficient gas for complex `receive()` functions.
2. Alternatively, raise `DepositCallGasLimit` to a higher default (e.g., the same cap used for `COA.call()`) and document the trade-off.
3. Do not panic on a failed deposit result; instead return the `Result` to the Cadence caller so it can handle the failure gracefully, consistent with how `EVM.run()` returns a `Result` without panicking.

### Proof of Concept

1. Deploy an EVM smart contract with a `receive()` function that writes to storage (costs > 2,300 gas):
   ```solidity
   contract Receiver {
       uint256 public count;
       receive() external payable { count += 1; }  // SSTORE costs 20,000 gas
   }
   ```
2. From a Cadence transaction, call:
   ```cadence
   import EVM from <EVM_address>
   import FlowToken from <FlowToken_address>
   transaction {
       prepare(signer: auth(BorrowValue) &Account) {
           let vault <- signer.storage.borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
               from: /storage/flowTokenVault)!.withdraw(amount: 1.0) as! @FlowToken.Vault
           let addr = EVM.EVMAddress(bytes: <Receiver_address_bytes>)
           addr.deposit(from: <-vault)  // panics: out of gas in receive()
       }
   }
   ```
3. The `NewDepositCall` is constructed with `GasLimit = 23,300`. After the 21,000 intrinsic cost, only 2,300 gas remains for the `receive()` function. The `SSTORE` requires 20,000 gas, so the EVM returns `ErrOutOfGas`. `panicOnErrorOrInvalidOrFailedState` panics with an EVM error, aborting the transaction. The deposit permanently fails for this contract address. [9](#0-8) [10](#0-9)

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
