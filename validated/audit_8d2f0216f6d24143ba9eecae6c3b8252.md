### Title
Pool Admin Bypasses Global Spread Fee Cap via Uncapped Per-Bin Additional Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the global spread fee, but `setPoolBinAdditionalFees` applies no analogous cap on `addFeeBuyE6` / `addFeeSellE6`. A pool admin can set per-bin additional fees up to `type(uint16).max` (65 535 E6 ≈ 6.55%) on top of an already-maxed global spread fee, making the effective per-bin fee exceed the protocol's hard cap and extracting more from traders than the cap permits.

### Finding Description

The factory enforces a hard cap hierarchy for the global spread fee:

- `HARD_MAX_SPREAD_FEE_E6` = 200 000 (20 %)
- `maxAdminSpreadFeeE6` ≤ 200 000
- `setPoolAdminFees` reverts if `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` [1](#0-0) 

However, `setPoolBinAdditionalFees` — also callable by the pool admin — passes `addFeeBuyE6` and `addFeeSellE6` straight through to the pool with **no cap check**: [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitude: [3](#0-2) 

During a swap, the effective spread fee applied to a bin is `spreadFeeE6 + addFeeBuyE6` (or `addFeeSellE6`). Because `addFeeBuyE6` / `addFeeSellE6` are `uint16`, their maximum is 65 535 E6 ≈ 6.55 %. A pool admin who has already set the global spread fee to the maximum (200 000 E6 = 20 %) can additionally set per-bin fees to 65 535 E6, yielding an effective bin fee of ≈ 26.55 % — 33 % above the protocol's hard cap.

This is the direct analog of the Notional M-20 bug: the minimum-borrow-size check is applied to the primary debt but not to secondary debts. Here, the fee cap is applied to the global spread fee but not to the per-bin additional fees, which are a second, additive fee of the same type.

### Impact Explanation

Traders executing swaps through a bin with maxed-out additional fees pay materially more than the protocol's hard cap permits. The excess fee accrues to the pool admin (via `collectFees`), constituting a direct loss of user principal above the protocol's intended ceiling. The pool's swap conservation invariant is broken: the trader's input exceeds what the oracle-derived price plus the capped fee should require.

### Likelihood Explanation

The pool admin is semi-trusted and has direct, permissionless access to `setPoolBinAdditionalFees` at any time with no timelock. No off-chain monitoring or governance step is required. The admin can target the most-traded bin (typically bin 0, the at-the-money bin) to maximise extraction. The call requires only `onlyPoolAdmin(pool)`, which is a single EOA or multisig.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (factory side) analogous to the check in `setPoolAdminFees`. For example:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce a combined cap: `spreadFeeE6 + addFeeBuyE6 ≤ HARD_MAX_SPREAD_FEE_E6`.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20 %, the hard cap).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, global spread fee = 20 %.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**, per-bin additional fee = 6.55 %.
4. A trader swaps through bin 0. Effective spread fee = 200 000 + 65 535 = 265 535 E6 ≈ 26.55 %, exceeding the 20 % hard cap.
5. The excess ≈ 6.55 % of notional is taken from the trader and flows to the admin fee destination on the next `collectPoolFees` call. [4](#0-3) [2](#0-1) [3](#0-2)
