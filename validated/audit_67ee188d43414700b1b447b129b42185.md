Audit Report

## Title
ckERC20 Withdrawal Permanently Stuck With No Reimbursement Path When Gas Fees Spike After Request Acceptance — (`rs/ethereum/cketh/minter/src/withdraw.rs`)

## Summary

When a user calls `withdraw_erc20`, both ckETH (gas fee) and ckERC20 (withdrawal amount) are burned immediately at request-acceptance time based on the gas fee estimate at that instant. If Ethereum gas fees spike before the minter's timer processes the queued request, `create_transaction` returns `CreateTransactionError::InsufficientTransactionFee` and the request is indefinitely rescheduled with no reimbursement triggered. Because `maybe_reimburse` is only populated inside `record_created_transaction` — which is only reached on success — a request that never passes `create_transaction` has no protocol-level escape hatch, leaving both the user's ckETH and ckERC20 tokens burned with no corresponding on-chain transfer and no recovery path.

## Finding Description

**Fee locked at request time.** In `main.rs` L430–451, `withdraw_erc20` calls `estimate_erc20_transaction_fee()` and immediately burns that amount of ckETH from the user's account. The burned amount is stored as `Erc20WithdrawalRequest.max_transaction_fee` and is immutable for the lifetime of the request. The ckERC20 withdrawal amount is also burned in the same function (L468–479). Both burns succeed before the request is enqueued.

**Asynchronous execution with fresh gas estimate.** `process_retrieve_eth_requests()` (`withdraw.rs` L150–183) fires on a periodic timer and calls `lazy_refresh_gas_fee_estimate()` to obtain the *current* gas fee (up to 60 seconds stale). This fresh estimate is passed to `create_transactions_batch`.

**Mismatch causes indefinite rescheduling.** Inside `create_transaction` (`transactions/mod.rs` L1155–1168), for `WithdrawalRequest::CkErc20`, the code computes `request_max_fee_per_gas` from the locked `max_transaction_fee` and checks `actual_min_max_fee_per_gas > request_max_fee_per_gas`. If gas fees have spiked, this returns `Err(CreateTransactionError::InsufficientTransactionFee)`. Back in `create_transactions_batch` (`withdraw.rs` L281–291), this error path calls only `reschedule_withdrawal_request` — moving the request to the back of the queue — with no reimbursement triggered.

**No reimbursement path for stuck pending requests.** The `maybe_reimburse` set is populated exclusively inside `record_created_transaction` (`transactions/mod.rs` L552), which is only reached when `create_transaction` *succeeds*. A request that never passes `create_transaction` never enters the reimbursement pipeline. There is no cancel endpoint (confirmed: no `cancel_withdrawal` or `cancel_erc20` function exists anywhere in the minter source), no retry counter, and no timeout-based reimbursement trigger.

## Impact Explanation

This is a concrete user-funds loss issue in the ckERC20 Chain Fusion component. Both ckETH (gas fee) and ckERC20 tokens (withdrawal amount) are burned on the IC ledgers at request time. If gas fees remain elevated indefinitely, the ckERC20 tokens are permanently burned with no corresponding ERC-20 transfer ever executed and no reimbursement mint. This is a direct ledger conservation violation: supply is reduced on the IC side without the corresponding asset being released on Ethereum. The impact matches **High ($2,000–$10,000): Significant Chain Fusion / ck-token security impact with concrete user or protocol harm**, as it requires specific timing conditions (gas fee spike in the window between request acceptance and timer execution) but is reachable by any unprivileged user via the public `withdraw_erc20` endpoint.

## Likelihood Explanation

Ethereum gas fees are historically volatile; 5–20× spikes within minutes are well-documented during network congestion events. The minter's 60-second gas fee cache (`MAX_AGE_NS = 60_000_000_000`) and the natural queue delay between request acceptance and timer execution create a window for fee divergence. Any user can trigger this condition by submitting a `withdraw_erc20` call during a low-fee window. No special privileges are required. The condition is repeatable and self-inflicted by normal protocol usage.

## Recommendation

1. **Add a reimbursement path for stuck pending requests.** When `InsufficientTransactionFee` is returned in `create_transactions_batch`, track a consecutive-failure count or a `created_at`-relative timeout per request. After N failures or after a configurable deadline, trigger the existing reimbursement flow to mint back both the burned ckETH and ckERC20 to the user.
2. **Add a user-callable cancellation endpoint.** Allow the original requester to cancel a withdrawal that has not yet had a transaction created, triggering reimbursement of both burned amounts (minus a small penalty fee to prevent abuse), mirroring the existing ckERC20 burn-failure reimbursement logic.
3. **Defer the ckETH burn until transaction creation time.** Alternatively, do not burn ckETH until the minter is about to create the transaction, so the burned amount reflects the actual execution-time fee rather than the request-time estimate.

## Proof of Concept

1. Ethereum gas fees are at 10 gwei. User calls `withdraw_erc20` for 100 ckUSDC. Minter calls `estimate_erc20_transaction_fee()`, computes `max_transaction_fee` ≈ 0.0014 ETH, burns 0.0014 ckETH and 100 ckUSDC from user. `Erc20WithdrawalRequest` is enqueued with `max_transaction_fee = 0.0014 ETH`.
2. Before the minter's next processing timer fires, Ethereum gas fees spike to 200 gwei.
3. `process_retrieve_eth_requests` fires. `lazy_refresh_gas_fee_estimate` returns 200 gwei. `create_transaction` computes `actual_min_max_fee_per_gas = 201.5 gwei`; `request_max_fee_per_gas = 0.0014 ETH / 65,000 ≈ 21.5 gwei`. Since `201.5 > 21.5`, `InsufficientTransactionFee` is returned.
4. `create_transactions_batch` calls `reschedule_withdrawal_request`. No reimbursement is scheduled. The request cycles indefinitely.
5. The user's 0.0014 ckETH and 100 ckUSDC are burned. `retrieve_eth_status` shows `Pending` indefinitely. No cancel endpoint exists. The user has no recourse within the protocol.

A deterministic integration test can reproduce this by: (a) submitting a `withdraw_erc20` call with a low gas estimate, (b) injecting a high gas fee estimate into the minter state before the timer fires, (c) asserting that after N timer iterations the request remains in `pending_withdrawal_requests` with `maybe_reimburse` empty and both ledger balances unreimbursed.
