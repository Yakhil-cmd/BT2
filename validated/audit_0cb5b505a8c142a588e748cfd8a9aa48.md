### Title
Stale Connector Account Reference Permanently Freezes All Bridged ETH After `set_eth_connector_contract_account` - (File: engine/src/contract_methods/connector.rs)

---

### Summary

When the ETH connector contract account is changed via `set_eth_connector_contract_account`, the Engine updates only its own stored pointer to the connector. It does not migrate NEP-141 token balances from the old connector to the new one, nor does it preserve any routing path to the old connector. All subsequent `withdraw` calls are routed exclusively to the new connector, which holds no NEP-141 ETH supply. Every user who deposited ETH through the old connector has their EVM balance permanently frozen: the Engine records the balance, but the withdrawal path is broken.

---

### Finding Description

`set_eth_connector_contract_account` (lines 418–438) writes a new connector `AccountId` into Engine storage and nothing else:

```rust
set_connector_account_id(io, &args.account);
set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
```

Every connector-facing operation in the Engine is dispatched through the single helper `return_promise` (lines 598–616), which always resolves the target at call time:

```rust
let promise_args = PromiseCreateArgs {
    target_account_id: get_connector_account_id(&io)?,   // always the current (new) connector
    ...
};
```

This means `withdraw` (line 58), `ft_transfer` (line 264), `storage_deposit` (line 302), `storage_unregister` (line 323), and `storage_withdraw` (line 338) all immediately begin targeting the new connector after the account is changed.

The old connector still holds the entire NEP-141 ETH supply that was minted for every prior deposit. The new connector starts with zero supply. When a user calls `withdraw`, the Engine debits their EVM balance and dispatches `engine_withdraw` to the new connector; the new connector has no tokens to burn and no record of the user, so the call fails or produces incorrect accounting. The user's EVM balance has already been decremented (or the transaction reverts), but either way the ETH locked on Ethereum cannot be released.

A secondary breakage occurs in `ft_on_transfer` (line 81):

```rust
if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(...)
}
```

Any in-flight deposit from the old connector that arrives after the account change is misclassified as an ERC-20 transfer rather than a base-token deposit, silently corrupting the accounting or returning the tokens.

The structural parallel to the reported Airdrop bug is exact: one contract (the Engine) stores a pointer to a "locker" (the connector); the pointer is updated in the Engine but the old locker's funds are never migrated and no fallback path is preserved, making those funds permanently unclaimable.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

All ETH that was bridged through the old connector becomes unwithdrawable the moment the connector account is changed. The Engine's EVM state still credits users with balances, but every withdrawal attempt targets a connector that cannot honour it. There is no recovery path in the current code: no fallback to the old connector, no migration function, no way for users to reclaim their ETged ETH.

---

### Likelihood Explanation

The function `set_eth_connector_contract_account` is an explicit, documented admin operation exposed as a named contract method (`CallSetEthConnectorContractAccount` in `engine-workspace/src/contract.rs`, line 174). It is the intended mechanism for upgrading or replacing the connector. Any legitimate connector upgrade — a routine operational event — triggers the freeze. The likelihood is therefore tied directly to the frequency of connector upgrades, which is non-negligible over the lifetime of the protocol.

---

### Recommendation

1. **Preserve the old connector reference.** Store a list (or at minimum the previous) connector account ID so that `withdraw` and `ft_on_transfer` can still route to the old connector for balances that were deposited through it.

2. **Require a migration step before cutover.** Before overwriting the connector account, verify (via a cross-contract view call or an explicit admin attestation) that the old connector's NEP-141 supply has been fully migrated to the new connector.

3. **Do not silently reroute in-flight deposits.** The `ft_on_transfer` guard at line 81 should accept calls from any connector that was ever registered, not only the current one, to avoid misclassifying in-flight deposits as ERC-20 transfers.

---

### Proof of Concept

**State before:**
- Old connector account: `eth-connector.aurora`; holds 1 000 ETH of NEP-141 supply.
- User Alice has 10 ETH credited in the Aurora Engine EVM state, deposited via `eth-connector.aurora`.

**Step 1 — Admin changes connector:**
```
aurora.set_eth_connector_contract_account({
    account: "eth-connector-v2.aurora",
    withdraw_serialize_type: Borsh
})
```
`set_connector_account_id` now stores `eth-connector-v2.aurora`. [1](#0-0) 

**Step 2 — Alice calls `withdraw`:**
```
aurora.withdraw({ recipient_address: 0xAlice, amount: 10_ETH })
```
`return_promise` resolves `get_connector_account_id` → `eth-connector-v2.aurora` and dispatches `engine_withdraw` there. [2](#0-1) 

**Step 3 — New connector fails:**
`eth-connector-v2.aurora` has zero NEP-141 ETH supply. It cannot burn 10 ETH. The call fails. Alice's EVM balance is either not debited (revert) or is debited with no corresponding unlock on Ethereum — in either case her ETH is permanently inaccessible.

**Step 4 — Old connector is unreachable:**
No code path in the Engine ever calls `eth-connector.aurora` again. The 1 000 ETH of NEP-141 supply sitting there is permanently stranded. [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}
