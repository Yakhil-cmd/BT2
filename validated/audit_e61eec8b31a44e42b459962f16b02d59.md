### Title
Hardcoded 2,300-Gas Stipend in `DepositCallGasLimit` / `DefaultGasLimitForTokenTransfer` Permanently Blocks FLOW Deposits to EVM Smart Contracts with Non-Trivial `receive()` - (File: `fvm/evm/types/call.go`)

---

### Summary

`fvm/evm/types/call.go` defines `DefaultGasLimitForTokenTransfer = 21_000 + 2_300 = 23_300`, which is used as the gas limit for every `NewDepositCall` and `NewTransferCall` directed at an EVM address. This leaves exactly 2,300 gas for the target contract's `receive()` or `fallback()` function — the same fixed stipend that Solidity's deprecated `transfer()` uses. After the Istanbul EVM hard-fork, a single `SLOAD` costs 2,100 gas, meaning any EVM smart contract whose `receive()` reads even one storage slot will exhaust the stipend and revert. The handler then calls `panicOnErrorOrInvalidOrFailedState`, which panics the entire Cadence transaction, permanently blocking FLOW deposits to such contracts.

---

### Finding Description

In `fvm/evm/types/call.go` the constants are:

```go
IntrinsicFeeForTokenTransfer        = gethParams.TxGas          // 21,000
DefaultGasLimitForTokenTransfer     = IntrinsicFeeForTokenTransfer + 2_300  // 23,300
DepositCallGasLimit                 = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit                = DefaultGasLimitForTokenTransfer
```

`NewDepositCall` and `NewTransferCall` both hard-wire this limit:

```go
func NewDepositCall(...) *DirectCall {
    return &DirectCall{...  GasLimit: DepositCallGasLimit, ...}
}
func NewTransferCall(...) *DirectCall {
    return &DirectCall{...  GasLimit: DefaultGasLimitForTokenTransfer, ...}
}
```

`Account.Deposit` in `fvm/evm/handler/handler.go` calls `NewDepositCall` and then:

```go
panicOnErrorOrInvalidOrFailedState(res, err)
```

which panics (reverts the Cadence transaction) on any `res.Failed()` result, including EVM out-of-gas.

The comment in the source even acknowledges the pattern: *"21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods"* — this is the exact 2,300-gas stipend that Solidity's `transfer()` imposes and that the Istanbul upgrade made unsafe.

The emulator test at `fvm/evm/emulator/emulator_test.go:83-111` already demonstrates the failure mode: depositing to a contract that does not accept native tokens (or whose `receive()` runs out of gas) yields `gethVM.ErrExecutionReverted`, which propagates to a panic.

---

### Impact Explanation

Any EVM smart contract whose `receive()` or `fallback()` function performs a storage read (`SLOAD`, 2,100 gas post-Istanbul) or any other operation costing more than 2,300 gas will always revert when FLOW is deposited to it via `Account.Deposit()` or `Account.Transfer()`. The Cadence transaction panics, and the deposit is permanently impossible for that address. Cadence protocols (bridges, escrows, DeFi integrations) that attempt to push FLOW into such EVM contracts are permanently broken for those targets. This constitutes a cross-VM asset-movement block: FLOW tokens held in Cadence cannot enter the EVM environment for a class of valid EVM contract addresses.

---

### Likelihood Explanation

Medium. The condition is triggered by any EVM smart contract with a non-trivial `receive()` or `fallback()` — a common pattern in DeFi (e.g., contracts that update accounting state on receipt). An unprivileged user can deploy such a contract and immediately reproduce the failure. No special privileges, leaked keys, or node compromise are required.

---

### Recommendation

Replace the hardcoded `2_300` stipend with a higher gas budget (e.g., `100_000`) for `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer`, or make the gas limit configurable per call. This mirrors the Solidity recommendation to replace `transfer()` with `call{value: ...}("")` which forwards all available gas. The existing `NewDeployCall` already accepts a caller-supplied `gasLimit`, demonstrating the pattern is available.

---

### Proof of Concept

1. Deploy an EVM contract on Flow EVM whose `receive()` reads one storage slot:
   ```solidity
   contract StorageReceiver {
       uint256 public counter;
       receive() external payable { counter += 1; }  // SLOAD + SSTORE > 2,300 gas
   }
   ```
2. From Cadence, create a COA and call `COA.deposit(from: <-vault)` targeting this contract address.
3. Internally, `Account.Deposit` constructs `NewDepositCall(..., GasLimit: 23_300)`.
4. The EVM executes the call; the `receive()` function exhausts the 2,300-gas stipend and reverts with `ErrExecutionReverted`.
5. `panicOnErrorOrInvalidOrFailedState` panics, reverting the entire Cadence transaction.
6. The deposit is permanently impossible for this contract address regardless of how much FLOW the caller holds.

```go
func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
```
