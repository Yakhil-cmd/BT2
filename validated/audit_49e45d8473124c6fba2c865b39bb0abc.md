### Title
`validateCOAOwnershipProof` Does Not Bind `evmAddress` to the Signed Message, Enabling Cross-COA Ownership Proof Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` accepts an `evmAddress` parameter that is used to verify which COA the proof is for, but the cryptographic signatures are verified only over the caller-supplied `signedData` — which is never required to include `evmAddress`. An attacker who obtains a valid signature from a victim (intended to prove ownership of COA_A) can replay it with a different `evmAddress` and `path` pointing to COA_B (another COA owned by the same Cadence account), and the proof will validate successfully for COA_B.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. **Signature check** — verifies that the provided `signatures` are valid over `signedData` using keys from the Cadence account at `address`.
2. **COA address check** — verifies that the COA resource at `path` has an EVM address matching the supplied `evmAddress`.

The critical flaw is that `evmAddress` is **not part of `signedData`** and is **not covered by the signatures**. The function explicitly acknowledges this in its own NatSpec comment:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The signature verification step:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // evmAddress is NOT in here
    domainSeparationTag: "FLOW-V0.0-user"
)
```

passes without any binding to `evmAddress`. After this check passes, the COA address check:

```cadence
if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
    let coaAddressBytes = coaRef.address().bytes
    for index, item in coaAddressBytes {
        if item != evmAddress[index] { ... }
    }
    return ValidationResult(isValid: true, problem: nil)
}
```

only verifies that the on-chain COA resource at `path` matches the attacker-supplied `evmAddress` — not that the signer intended to authorize that specific EVM address.

### Impact Explanation

An attacker who obtains a valid `(signedData, signatures, keyIndices)` tuple from a victim — intended to prove ownership of COA_A — can substitute `evmAddress = COA_B.bytes` and `path = /public/coaB` (another COA published by the same Cadence account). Both checks pass:

- Signature check passes because `signedData` is unchanged and the keys belong to the same Cadence account.
- COA address check passes because the attacker supplied the correct `evmAddress` for COA_B and the correct `path` to borrow it.

The result is a forged proof of ownership of COA_B. Any Cadence protocol or contract that gates asset access or privileged operations on `validateCOAOwnershipProof` returning `isValid: true` for a specific `evmAddress` is vulnerable to unauthorized access. This constitutes a cross-VM asset loss vector: an attacker can impersonate the owner of a COA they do not control, potentially draining assets or triggering privileged operations on that COA.

### Likelihood Explanation

The attack requires:
1. The victim to have **multiple COAs** under the same Cadence account (currently uncommon but protocol-permitted and growing).
2. The attacker to obtain a valid `(signedData, signatures)` from the victim — achievable by acting as a malicious proof requester who asks the victim to sign arbitrary data for a legitimate-looking purpose.
3. A Cadence protocol that calls `validateCOAOwnershipProof` directly (not through the EVM precompile) as an authentication gate.

The EVM precompile path (`verifyCOAOwnershipProof`) is **not** affected because it always passes `address(this)` as `evmAddress`, which the attacker cannot substitute. However, `validateCOAOwnershipProof` is `access(all)` and is part of the public EVM system contract API, making it callable from any Cadence script or transaction.

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof` itself, rather than leaving this as a caller responsibility. The function should hash or prepend `evmAddress` into the data that is verified:

```cadence
// Reconstruct the expected signed payload binding evmAddress
let expectedSignedData = signedData.concat(evmAddress.toVariableSized())
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: expectedSignedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Alternatively, add a pre-condition that panics if `signedData` does not contain `evmAddress` as a suffix/prefix, making the binding mandatory at the protocol level rather than advisory.

### Proof of Concept

**Setup**: Cadence account `0xAlice` owns two COAs:
- COA_A stored at `/storage/coaA`, published at `/public/coaA`
- COA_B stored at `/storage/coaB`, published at `/public/coaB`

**Step 1 — Attacker acts as proof requester for COA_A**: Attacker asks Alice to sign `signedData = [0xDE, 0xAD, 0xBE, 0xEF]` to prove she owns COA_A. Alice signs and returns `(signatures, keyIndices)`.

**Step 2 — Attacker replays against COA_B**: Attacker calls `EVM.validateCOAOwnershipProof` with:
- `address: 0xAlice`
- `path: /public/coaB`  ← substituted
- `signedData: [0xDE, 0xAD, 0xBE, 0xEF]`  ← unchanged
- `keyIndices: [0]`  ← unchanged
- `signatures: [aliceSig]`  ← unchanged
- `evmAddress: COA_B.address.bytes`  ← substituted

**Step 3 — Both checks pass**:
- `keyList.verify(signatureSet, signedData: [0xDE, 0xAD, 0xBE, 0xEF], ...)` → `true` (Alice's key signed this data)
- `coaRef.address().bytes == evmAddress` → `true` (COA_B's address matches the attacker-supplied `evmAddress`)

**Result**: `ValidationResult(isValid: true, problem: nil)` — the attacker has a valid proof of ownership of COA_B without Alice's consent for that specific COA. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
