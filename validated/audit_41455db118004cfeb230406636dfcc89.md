### Title
`KeygenOutput<C>` Derives `Debug` Exposing `private_share` Signing Share to Unintended Channels — (`File: src/lib.rs`)

---

### Summary

`KeygenOutput<C>`, the primary output type of every DKG, reshare, and refresh operation, derives `Debug` unconditionally. Its `private_share: SigningShare<C>` field — the participant's long-term secret signing share — is therefore emitted verbatim whenever any caller formats the struct with `{:?}`. The same pattern repeats on both `PresignOutput` and `RerandomizedPresignOutput` in the OT-based and Robust ECDSA modules, exposing ephemeral nonce secrets (`k`, `sigma`, `c`, `e`, `alpha`, `beta`) through the same channel.

---

### Finding Description

`src/lib.rs` line 48 declares:

```rust
#[derive(Debug, Clone, Deserialize, Serialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    #[zeroize[skip]]
    pub public_key: VerifyingKey<C>,
}
```

The `Debug` derive causes the Rust compiler to generate a `fmt::Debug` implementation that recursively formats every field, including `private_share`. Because `SigningShare<C>` (from `frost_core`) wraps a raw scalar and itself implements `Debug`, any `println!("{:?}", keygen_output)`, `log::debug!("{:?}", ...)`, `format!("{:?}", ...)`, or panic message that captures the struct will emit the raw secret scalar in plaintext.

The same `#[derive(Debug, ...)]` appears on:

- `PresignOutput` (OT-based) — fields `k: Scalar` (nonce share) and `sigma: Scalar` (sigma share) — `src/ecdsa/ot_based_ecdsa/mod.rs` line 40
- `RerandomizedPresignOutput` (OT-based) — fields `k`, `sigma` — `src/ecdsa/ot_based_ecdsa/mod.rs` line 54
- `PresignOutput` (Robust ECDSA) — fields `c`, `e`, `alpha`, `beta` — `src/ecdsa/robust_ecdsa/mod.rs` line 26
- `RerandomizedPresignOutput` (Robust ECDSA) — fields `e`, `alpha`, `beta` — `src/ecdsa/robust_ecdsa/mod.rs` line 42

The `Serialize` derive on `KeygenOutput` is independently confirmed to emit `private_share` as a hex string in JSON output, as demonstrated by the test at `src/ecdsa/mod.rs` lines 262–269 and the determinism snapshot files (e.g., `src/ecdsa/snapshots/threshold_signatures__ecdsa__test__keygen_determinism.snap`).

---

### Impact Explanation

**Critical — Extraction, reconstruction, or disclosure of private signing shares, presign secrets, and nonce material.**

- Disclosure of `private_share` from `KeygenOutput` directly yields a participant's long-term signing share. Collecting threshold-many such shares allows full private key reconstruction.
- Disclosure of `k` / `sigma` from `PresignOutput` (OT-based) or `c` / `e` / `alpha` / `beta` from `PresignOutput` (Robust) yields the per-session nonce material. An attacker who obtains a nonce share and the corresponding signature can solve for the private key algebraically.

---

### Likelihood Explanation

**High.** Rust's `Debug` trait is routinely invoked in:
- Application-level logging frameworks (`tracing`, `log`, `env_logger`) that format structs with `{:?}` at `DEBUG` or `TRACE` level.
- Panic messages and `unwrap()`/`expect()` call sites that capture the value.
- Test harnesses and integration test output.
- Error propagation chains that embed context values.

Any MPC node operator who enables debug-level logging — a standard practice during development, staging, or incident investigation — will emit every participant's signing share and every presign nonce to the log sink. Log aggregation systems (Elasticsearch, Splunk, CloudWatch) then persist and index this material, making it accessible to anyone with log-read access.

---

### Recommendation

1. **Remove `Debug` from all types containing secret scalars.** Replace the blanket `#[derive(Debug)]` on `KeygenOutput`, `PresignOutput`, and `RerandomizedPresignOutput` with a hand-written `Debug` implementation that redacts secret fields:
   ```rust
   impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
       fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
           f.debug_struct("KeygenOutput")
               .field("private_share", &"[REDACTED]")
               .field("public_key", &self.public_key)
               .finish()
       }
   }
   ```
2. **Audit `Serialize` usage.** The `Serialize` derive on `KeygenOutput` is necessary for persistence but should be clearly documented as producing output that contains secret material, and callers should be warned never to log the serialized form.
3. **Consider a dedicated secret-wrapping type** (e.g., using the `secrecy` crate's `Secret<T>`) for `private_share` and nonce scalars to enforce redaction at the type level across all formatting and serialization paths.

### Proof of Concept

```rust
use threshold_signatures::{KeygenOutput, Ciphersuite};
use frost_secp256k1::Secp256K1Sha256;
use frost_core::keys::SigningShare;

fn main() {
    // Simulate receiving a KeygenOutput after DKG
    let keygen_output: KeygenOutput<Secp256K1Sha256> = KeygenOutput {
        private_share: SigningShare::new(/* secret scalar */ ...),
        public_key: /* ... */,
    };

    // Any of the following emit the raw private_share scalar:
    println!("{:?}", keygen_output);          // direct debug print
    log::debug!("keygen result: {:?}", keygen_output); // logging framework
    let s = format!("{:?}", keygen_output);   // captured in error context

    // The output contains the secret scalar in plaintext, e.g.:
    // KeygenOutput { private_share: SigningShare(0xadd8bea1...), public_key: ... }
}
```
