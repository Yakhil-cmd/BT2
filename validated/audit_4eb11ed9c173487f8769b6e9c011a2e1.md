### Title
Global Contract Identifier Overhead Excluded from Deletion Storage-Usage Check Blocks Account Withdrawal — (`runtime/runtime/src/actions.rs`)

### Summary

`action_delete_account` contains a legacy code path (active for protocol versions < 85) that computes the deletable storage usage by subtracting only the **local** contract code length. For accounts whose contract field is `AccountContract::Global` or `AccountContract::GlobalByAccount`, the identifier overhead (32 bytes or `account_id.len()` bytes respectively) is never subtracted. The inflated `account_storage_usage` value is then compared against `Account::MAX_ACCOUNT_DELETION_STORAGE_USAGE`, causing a spurious `DeleteAccountWithLargeState` rejection and permanently blocking the account owner from recovering their balance.

### Finding Description

In `action_delete_account`, the storage-usage check that gates deletion is computed as follows in the legacy branch:

```rust
// Legacy behavior: only subtracts local contract code, misses the
// global contract identifier overhead.
let account_storage_usage = account_ref.storage_usage();
let code_len = get_code_len_or_default(
    state_update,
    account_id.clone(),
    account_ref.local_contract_hash().unwrap_or_default(),  // ← returns CryptoHash::default() for Global/GlobalByAccount
)?;
account_storage_usage.saturating_sub(code_len)  // ← subtracts 0; identifier overhead stays in
``` [1](#0-0) 

`local_contract_hash()` returns `None` for both `AccountContract::Global` and `AccountContract::GlobalByAccount`, so `unwrap_or_default()` produces `CryptoHash::default()`. `get_code_len_or_default` then returns `0` for the default hash, leaving the identifier overhead (32 bytes for `Global`; `id.len()` bytes for `GlobalByAccount`) fully included in `account_storage_usage`. [2](#0-1) 

The corrected path (enabled at protocol version 85) calls `get_contract_storage_usage`, which explicitly handles the global-contract cases:

```rust
AccountContract::Global(_) | AccountContract::GlobalByAccount(_) => {
    account.contract().identifier_storage_usage()
}
``` [3](#0-2) 

`identifier_storage_usage()` returns 32 for `Global` and `id.len()` for `GlobalByAccount`. [4](#0-3) 

The `UseGlobalContractAction` handler (`use_global_contract`) correctly **adds** the identifier length to `storage_usage` when the global contract is adopted:

```rust
account.set_storage_usage(
    account.storage_usage().checked_add(contract_identifier.len() as u64)…
);
``` [5](#0-4) 

So the account's `storage_usage` field legitimately includes the identifier overhead, but the legacy deletion check never removes it before the `> MAX_ACCOUNT_DELETION_STORAGE_USAGE` comparison. [6](#0-5)

### Impact Explanation

An account owner who has adopted a global contract and whose total `storage_usage` falls in the half-open interval `(MAX_ACCOUNT_DELETION_STORAGE_USAGE − identifier_len, MAX_ACCOUNT_DELETION_STORAGE_USAGE]` cannot delete their account. The `DeleteAccountAction` receipt returns `ActionErrorKind::DeleteAccountWithLargeState` even though the account's actual user-data storage is at or below the limit. Because `DeleteAccountAction` is the only mechanism to transfer the account balance to a beneficiary and remove the account, the balance is locked for as long as the protocol version remains below 85. If the deployed contract exposes no storage-removal method, the lock is permanent within that protocol era.

### Likelihood Explanation

The trigger conditions are:
1. The account uses `UseGlobalContractAction` (any unprivileged user may do this once a global contract exists on the shard).
2. The account's `storage_usage` (including the identifier overhead added by step 1) exceeds `MAX_ACCOUNT_DELETION_STORAGE_USAGE` by at most `identifier_len` bytes.

The window is narrow (32 bytes for `Global`; up to ~64 bytes for `GlobalByAccount`), but it is fully user-controllable: the user can store exactly enough contract data to land in the window. ETH-implicit accounts are created with `AccountContract::Global` by default, widening the affected population. [8](#0-7) 

### Recommendation

The fix already exists in the codebase behind `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` (protocol version 85). The corrected path calls `get_contract_storage_usage` which correctly handles all `AccountContract` variants. Ensure protocol version 85 is activated on all production networks; the legacy branch can be removed once the minimum supported protocol version exceeds 84.

### Proof of Concept

The existing unit test `test_delete_account_global_contract_protocol_transition` in `runtime/runtime/src/actions.rs` directly demonstrates the bug:

```rust
// Before the fix: the identifier is not subtracted, so `MAX + 32 > MAX`.
let before = test_delete_account_in_empty_trie(
    &account_id,
    AccountContract::Global(CryptoHash::default()),
    storage,   // = MAX_ACCOUNT_DELETION_STORAGE_USAGE + 32
    enabled - 1,  // protocol version < 85
);
expect_delete_account_too_large(&before);  // deletion blocked — balance locked
``` [9](#0-8) 

On-chain reproduction:
1. Deploy a global contract and call `UseGlobalContractAction` on a target account (adds 32 bytes to `storage_usage`).
2. Store contract data until `storage_usage == MAX_ACCOUNT_DELETION_STORAGE_USAGE` (i.e., actual user data = `MAX − 32`).
3. Submit a `DeleteAccountAction` receipt on a node running protocol version < 85.
4. Observe `ActionErrorKind::DeleteAccountWithLargeState`; the account balance is not transferred to the beneficiary and the account persists.

fn get_code_len_or_default(
    state_update: &TrieUpdate,
    account_id: AccountId,
    code_hash: CryptoHash,
) -> Result<StorageUsage, StorageError> {
    let code_len = state_update.get_code_len(account_id, code_hash)?;
    debug_assert!(
        code_len.is_some() || code_hash == CryptoHash::default(),
        "Non-default code hash for account with no contract deployed: {:?}",
        code_hash
    );
    Ok(code_len.unwrap_or_default().try_into().unwrap())
}
