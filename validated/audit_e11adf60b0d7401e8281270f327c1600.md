### Title
Presignature Reuse After Failed/Aborted Signing Sessions Enables Secret Key Extraction - (File: `src/ecdsa/robust_ecdsa/presign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The Robust ECDSA scheme returns `PresignOutput` as a plain data structure with no library-enforced single-use guarantee. When a signing session fails or is aborted before completion, each participant's `PresignOutput` material remains valid and reusable. A malicious coordinator can exploit this by deliberately aborting signing sessions and replaying the same presignature across multiple sessions with different message hashes or participant sets, enabling full secret key extraction via standard ECDSA nonce-reuse attacks.

---

### Finding Description

`PresignOutput` is produced by `do_presign` in `src/ecdsa/robust_ecdsa/presign.rs` and returned as a plain owned struct to the caller. [1](#0-0) 

The library's own security documentation explicitly acknowledges the attack surface:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [2](#0-1) 

And further documents the concrete cryptographic consequence:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [3](#0-2) 

The library provides **no API-level enforcement** of single-use semantics for `PresignOutput`. There is no "consumed" flag, no opaque wrapper that prevents re-submission, and no session-binding that ties a presignature to exactly one signing invocation. The security constraint is delegated entirely to the caller with no in-library guard.

This is the direct analog to the `NativeMetaTransaction` bug: just as a failed meta-transaction leaves the nonce unchanged and allows the same signed payload to be replayed, a failed or aborted signing session leaves each participant's `PresignOutput` intact and reusable, allowing the same nonce material to be used in a subsequent signing session.

---

### Impact Explanation

**Critical — Extraction of private signing shares / aggregate secret material.**

If a malicious coordinator initiates two signing sessions using the same presignature with distinct message hashes `h₁` and `h₂` (or distinct rerandomization tweaks `ε₁` and `ε₂`), the two resulting partial signatures expose the relationship between the nonce and the secret key. Standard ECDSA nonce-reuse algebra then allows full recovery of the aggregate secret key from the two signature equations. [3](#0-2) 

The documentation further notes a novel variant specific to the linearization optimization in this implementation:

> "a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions." [4](#0-3) 

---

### Likelihood Explanation

The coordinator role is explicitly part of the protocol design. [5](#0-4)  A malicious coordinator can trivially trigger the condition by:

1. Initiating a signing session and collecting partial signature shares from participants (who consume their `PresignOutput` locally).
2. Aborting or dropping the session before broadcasting the final signature.
3. Initiating a second signing session with the same presignature material (obtained by having participants re-run presign with the same inputs, or by the coordinator retaining the broadcast presignature values) and a different message hash.

Because the library places no enforcement barrier, any coordinator-level adversary following the documented API can reach this state without any out-of-band capability.

---

### Recommendation

1. **Consume `PresignOutput` on first use regardless of signing outcome.** The signing function should take `PresignOutput` by value and ensure it is cryptographically zeroed on both success and failure paths, so the caller cannot retain a copy.

2. **Bind presignatures to a session identifier at generation time.** Include a unique, unpredictable session ID in the presignature commitment so that the same presignature material cannot be submitted to a second signing invocation with a different session ID.

3. **Enforce single-use at the type level.** Wrap `PresignOutput` in a newtype that does not implement `Clone` or `Copy`, ensuring Rust's ownership system statically prevents duplication before the value is passed to `sign()`.

---

### Proof of Concept

**Setup:** 5 participants, `max_malicious = 2`, so `N = 2t+1 = 5`. A malicious coordinator controls the signing orchestration.

1. All 5 participants run `presign(...)` and each obtains their `PresignOutput` (containing `big_r`, `k_i`, `sigma_i`). [6](#0-5) 

2. The coordinator initiates **Signing Session 1** with message hash `h₁`. Participants send their signature shares `s_i` to the coordinator.

3. The coordinator **deliberately aborts** before finalizing, discarding the session. Participants' `PresignOutput` values have been consumed locally — but the coordinator has collected the broadcast presignature values `(R_i, w_i)` from Round 2 of presigning, which are public within the session. [7](#0-6) 

4. The coordinator instructs participants to re-run `presign()` with the same key material (or, in a split-view scenario, presents the same presignature transcript to a different subset). Participants who did not detect the abort still hold their `PresignOutput`.

5. The coordinator initiates **Signing Session 2** with message hash `h₂ ≠ h₁` using the same presignature. Two valid partial-signature sets are now available for the same nonce `R`.

6. From the two equations `s = k⁻¹(h + r·x)` evaluated at `h₁` and `h₂` with identical `k` and `r`, the coordinator solves for the secret key `x`:

   ```
   x = (s₁·h₂ - s₂·h₁) / (r·(s₂ - s₁))
   ```

Full secret key extracted. [3](#0-2)
