### Title
Non-Deterministic `ProcessProposalHandler` Across Validators with Divergent Blocklist State Causes Consensus Halt ‚Äî (`app/proposal.go`)

### Summary

`ProcessProposalHandler` produces different ACCEPT/REJECT decisions for the same block bytes depending on whether a validator's `Identity` field is `nil`. Validators with `Identity == nil` permanently skip `SetBlockList` updates and always fast-path to ACCEPT, while validators with a populated blocklist correctly REJECT proposals containing blocked-sender transactions. This divergence is a concrete consensus split.

### Finding Description

**Root cause ‚Äî `SetBlockList` is a silent no-op for nil-identity validators:** [1](#0-0) 

When `Identity == nil`, `SetBlockList` returns immediately without touching `h.blocklist`. The map stays as the empty map initialized in `NewProposalHandler`. [2](#0-1) 

**Root cause ‚Äî `ProcessProposalHandler` fast-paths to ACCEPT on empty blocklist:** [3](#0-2) 

Any validator whose `h.blocklist` is empty (because `Identity == nil` caused every `SetBlockList` call to be a no-op) will unconditionally return `ACCEPT` for every proposal, regardless of what transactions it contains.

**Root cause ‚Äî `EndBlocker` calls `RefreshBlockList` on every validator, but nil-identity validators silently ignore it:** [4](#0-3) 

After `MsgStoreBlockList` is committed on-chain, `EndBlocker` calls `RefreshBlockList` ‚Üí `SetBlockList`. Validators with a valid age identity decrypt the blob and populate `h.blocklist`. Validators with `Identity == nil` silently return without updating anything.

**The divergence path:**

| Validator state | `SetBlockList` effect | `ProcessProposalHandler` result for a block with a blocked-sender tx |
|---|---|---|
| `Identity == nil` | no-op; `h.blocklist` stays `{}` | fast-path ‚Üí **ACCEPT** |
| Valid identity, non-empty blocklist | blocklist populated | `ValidateTransaction` ‚Üí error ‚Üí **REJECT** | [5](#0-4) 

`ValidateTransaction` itself also fast-paths to `nil` when `h.blocklist` is empty, so nil-identity validators never inspect tx signers at all.

### Impact Explanation

When a nil-identity validator is the block proposer and includes a tx from a blocked sender:
- Validators with a populated blocklist call `ValidateTransaction` ‚Üí receive an error ‚Üí return `REJECT`
- Validators with `Identity == nil` hit the fast-path ‚Üí return `ACCEPT`

In CometBFT, `ProcessProposal` returning `REJECT` causes the validator to vote `nil` for that round. If the validators with non-empty blocklists hold ‚â•1/3 of voting power, no block from that proposer can ever reach 2/3 pre-commits, causing a **permanent consensus halt** for every round that proposer is selected. This matches the Critical impact category: *"Consensus divergence, deterministic state mismatch, invalid block acceptance, or chain halt triggered by an unprivileged transaction or proposal path."*

### Likelihood Explanation

Preconditions are realistic:
1. **Admin submits `MsgStoreBlockList`** ‚Äî normal, expected operation of the blocklist feature.
2. **At least one validator has `Identity == nil`** ‚Äî any validator that has not configured an age identity key (e.g., newly joined, misconfigured, or deliberately opting out) satisfies this. The code comment explicitly acknowledges this case: *"Identity is nil if it's not a validator node."*
3. **Nil-identity validator enters the proposer rotation** ‚Äî automatic, proportional to voting power.
4. **Blocked sender submits a tx** ‚Äî trivial; the blocked sender is unprivileged and only needs to broadcast a standard transaction.

The blocked sender does not need to control the blocklist or the proposer selection. They only need to submit transactions and wait for a nil-identity proposer to include one.

### Recommendation

`ProcessProposalHandler` must produce a deterministic result independent of local identity configuration. Two concrete fixes:

1. **Treat `Identity == nil` as "enforce no blocklist" consistently**: if `Identity == nil`, `ProcessProposalHandler` should still check the on-chain blocklist state (e.g., by reading it directly from the keeper rather than relying on the in-memory `h.blocklist`), or alternatively, validators without an identity must be prevented from proposing blocks when a non-empty blocklist is active.

2. **Remove the fast-path or gate it on identity**: the fast-path at line 340 (`if len(h.blocklist) == 0 { return ACCEPT }`) conflates two distinct cases ‚Äî "no blocklist is configured" and "identity is nil so we never loaded the blocklist." These must be distinguished. A separate boolean flag (e.g., `identityConfigured`) should gate whether the fast-path is safe to take.

### Proof of Concept

```
1. Start two validator nodes: node A with a valid age identity, node B with Identity == nil.
2. Admin submits MsgStoreBlockList containing address X (encrypted for node A's public key).
3. EndBlocker fires: node A decrypts and populates h.blocklist = {X}; node B silently skips.
4. Address X broadcasts a standard transfer tx.
5. Node B becomes proposer (nil identity ‚Üí empty blocklist ‚Üí PrepareProposal includes X's tx).
6. Node A receives the proposal ‚Üí ProcessProposalHandler ‚Üí h.blocklist non-empty ‚Üí ValidateTransaction(X's tx) ‚Üí "signer is blocked" ‚Üí REJECT.
7. Node B receives the same proposal ‚Üí ProcessProposalHandler ‚Üí h.blocklist empty ‚Üí fast-path ‚Üí ACCEPT.
8. Consensus halts: node A votes nil, node B votes for the block; neither reaches 2/3.
9. Every subsequent round where node B proposes repeats steps 5‚Äì8 ‚Üí permanent halt.
``` [6](#0-5) [1](#0-0)

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}
