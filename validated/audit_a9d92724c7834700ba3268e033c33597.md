### Title
Truncating Division in `native_per_pubdata` Calculation Allows Free or Underpriced Pubdata Consumption — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In the ZKsync L2 transaction validation path, the per-pubdata-byte native resource cost (`native_per_pubdata`) is computed using truncating integer division (`wrapping_div`), while the analogous per-gas native cost (`native_per_gas`) is computed using ceiling division (`div_ceil`). This asymmetry means pubdata costs are systematically rounded down in the user's favor. When `pubdata_price < native_price`, the result truncates to zero, and the user pays **no native resources** for any amount of pubdata written during execution.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (the ZKsync L2 transaction validation function), two resource rates are derived from block-level pricing parameters: [1](#0-0) 

`native_per_gas` uses `div_ceil` — rounding up, which is protocol-favorable: [2](#0-1) 

But `native_per_pubdata` uses `wrapping_div` — truncating division, which is user-favorable: [3](#0-2) 

This `native_per_pubdata` value is then passed into `create_resources_for_tx`, where it determines the intrinsic pubdata overhead deducted from the user's native budget: [4](#0-3) 

And it is also passed into `get_resources_to_charge_for_pubdata`, which computes the native cost of all pubdata written during execution: [5](#0-4) 

When `native_per_pubdata == 0`, both `intrinsic_pubdata_overhead` and the per-execution pubdata charge become zero. The user pays **zero native resources** for all pubdata consumed, regardless of how much state they write.

The same truncating division is replicated in the API helper used for off-chain validation: [6](#0-5) 

---

### Impact Explanation

When `pubdata_price < native_price` (a realistic condition during periods of low L1 gas prices or high ZK proving costs), `native_per_pubdata` truncates to `0`. Any user can then:

1. Submit a transaction that performs many `SSTORE` operations (each writing pubdata).
2. Pay only EVM gas for those operations — the native resource charge for pubdata is entirely waived.
3. Force the protocol/operator to absorb the L1 publication cost of that pubdata.

Even when `pubdata_price >= native_price`, the truncation causes every user to systematically underpay by up to `(native_price - 1)` native units per pubdata byte. Over many transactions this represents a consistent resource accounting shortfall that the protocol bears.

The direct impact is **loss of funds for the protocol**: pubdata publication costs are borne by the operator without corresponding user payment, and state bloat can be induced at below-cost rates.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is not exotic. `native_price` is the cost of one unit of native (computational) resource in the base token; `pubdata_price` is the cost of one pubdata byte. During periods of low L1 calldata/blob prices, `pubdata_price` can fall below `native_price`. No privileged access is required — any unprivileged transaction sender can exploit this whenever the block-level pricing satisfies the condition. The milder rounding-down case (where `pubdata_price >= native_price` but not a multiple) is always present.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates, user-favorable):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, protocol-favorable):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, add a guard that rejects (or at minimum logs) transactions when `native_per_pubdata == 0` and `pubdata_price` is nonzero, analogous to the `native_price.is_zero()` guard already present for `native_per_gas`.

---

### Proof of Concept

**Setup**: Configure a block with `pubdata_price = 999` and `native_price = 1000` (pubdata slightly cheaper than one native unit).

**Calculation**:
- `native_per_pubdata = 999.wrapping_div(1000) = 0`
- `native_per_gas = ceil(gas_price / 1000)` — nonzero, user pays for gas normally

**Exploit**: Submit a ZKsync transaction that performs 10,000 `SSTORE` writes (each generating pubdata). The user pays EVM gas for the SSTOREs but pays **zero native resources** for the pubdata. The operator must publish all 10,000 slots' worth of pubdata to L1 at their own expense.

**Contrast with correct behavior**: With `div_ceil`, `native_per_pubdata = ceil(999/1000) = 1`, and the user is charged 1 native unit per pubdata byte — correctly accounting for the cost.

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
```
