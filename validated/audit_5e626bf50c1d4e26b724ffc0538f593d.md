### Title
Hardcoded `L1_TX_NATIVE_PRICE` Constant Creates Systematic Native Resource Accounting Mismatch for L1→L2 Transactions - (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

For L1→L2 (priority) transactions, the bootloader uses a hardcoded constant `L1_TX_NATIVE_PRICE = 10` to convert gas price into native computational resources (`native_per_gas`). For L2 transactions, the same conversion uses the dynamic, oracle-provided `native_price` from the block metadata. When the actual market `native_price` diverges significantly from the hardcoded `10`, L1→L2 transactions are systematically allocated either far too many or far too few native resources per gas unit compared to L2 transactions. The code itself carries a `TODO (EVM-1157): find a reasonable value for it` comment, explicitly acknowledging the constant is a placeholder.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

This constant is consumed in `prepare_and_check_resources` inside `process_l1_transaction.rs`:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = ...
    u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [2](#0-1) 

For L2 transactions, the equivalent calculation in `validation_impl.rs` uses the oracle-provided block-level price:

```rust
let native_price = system.get_native_price();
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [3](#0-2) 

The `native_per_gas` ratio is the core resource allocation unit: it determines how many RISC-V proving cycles (`native` resources) are budgeted per EVM gas unit. The resulting `native_prepaid_from_gas = native_per_gas * gas_limit` is the total native budget for the transaction before the `MAX_NATIVE_COMPUTATIONAL` cap is applied. [4](#0-3) 

The `create_resources_for_tx` function then uses this budget to allocate computational resources: [5](#0-4) 

---

### Impact Explanation

**Case 1 — `L1_TX_NATIVE_PRICE` too low (e.g., actual `native_price` = 1000, constant = 10):**

- L2 `native_per_gas` = `ceil(gas_price / 1000)`
- L1 `native_per_gas` = `ceil(gas_price / 10)` → **100× more native resources per gas**

L1→L2 transactions receive up to 100× more RISC-V proving cycles per gas unit than L2 transactions paying the same gas price. An attacker submitting computationally intensive L1→L2 transactions (e.g., calling contracts with heavy storage writes) forces the prover to bear proving costs far exceeding what the user paid. This is a direct protocol-level value leak: the proving subsidy is borne by the protocol/operator, not the transaction sender.

**Case 2 — `L1_TX_NATIVE_PRICE` too high (e.g., actual `native_price` = 1, constant = 10):**

- L1 `native_per_gas` = `ceil(gas_price / 10)` → **10× fewer native resources per gas**

L1→L2 transactions are starved of native resources relative to L2 transactions. Legitimate L1→L2 transactions that would succeed as L2 transactions fail with out-of-native errors. Since L1 transactions cannot be invalidated without halting the priority queue, this creates a systematic DoS on the L1→L2 bridge for any workload that requires more than `native_per_gas * gas_limit` native resources.

The test suite confirms the hardcoded value is load-bearing and that the ratio directly determines success/failure: [6](#0-5) 

---

### Likelihood Explanation

The `native_price` is a dynamic oracle-provided block parameter that reflects real-time proving costs. As the ZKsync OS prover matures and proving costs change, the gap between the hardcoded `10` and the actual `native_price` will widen. The TODO comment in the source code confirms the developers are aware the value is not calibrated. Any deployment where `native_price` diverges from `10` by more than a small factor triggers the mismatch. An unprivileged L1 transaction sender can exploit Case 1 by submitting computationally intensive L1→L2 transactions at any time.

---

### Recommendation

Replace the hardcoded `L1_TX_NATIVE_PRICE` with a value derived from the block-level oracle `native_price`, subject to a minimum floor to preserve the censorship-resistance property. For example:

```rust
// Use oracle native_price but clamp to a minimum to prevent censorship
let native_price = system.get_native_price().max(MIN_L1_TX_NATIVE_PRICE);
```

Where `MIN_L1_TX_NATIVE_PRICE` is a conservatively low floor (e.g., `1`) that prevents operators from setting `native_price` to `u256::MAX` to censor L1 transactions, while still tracking actual proving costs. This mirrors the approach used for L2 transactions and eliminates the systematic resource accounting mismatch.

---

### Proof of Concept

Given a deployment where the block oracle reports `native_price = 1000`:

1. Submit an L1→L2 priority transaction with `gas_price = 10_000` and `gas_limit = 1_000_000`.
2. Bootloader computes: `native_per_gas = ceil(10_000 / 10) = 1_000` (using hardcoded `L1_TX_NATIVE_PRICE = 10`).
3. `native_prepaid_from_gas = 1_000 * 1_000_000 = 1_000_000_000`.
4. After `MAX_NATIVE_COMPUTATIONAL` cap, transaction receives the maximum computational native budget.
5. An equivalent L2 transaction with the same gas price would compute: `native_per_gas = ceil(10_000 / 1000) = 10`, giving `native_prepaid_from_gas = 10 * 1_000_000 = 10_000_000` — **100× less**.
6. The L1 transaction executes computationally intensive operations that would be rejected for an L2 transaction with the same gas, consuming proving resources the sender did not pay for. [7](#0-6) [8](#0-7)
