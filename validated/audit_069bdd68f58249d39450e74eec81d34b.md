### Title
Missing Caller Validation in `update_rewards` Allows Any Address to Permanently Freeze Staker Yield - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` is documented to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` on every block, consuming the global `last_reward_block` slot and permanently preventing the sequencer from distributing consensus rewards to stakers and pool members.

---

### Finding Description

The specification for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

However, the implementation in `StakingRewardsManagerImpl` performs no caller identity check whatsoever:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... NO assert_caller_is_sequencer() or equivalent ...
    let staker_info = self.internal_staker_info(:staker_address);
    ...
    self.last_reward_block.write(current_block_number);   // <-- global slot consumed

    if disable_rewards || self.is_pre_consensus() {
        return;   // <-- exits without distributing rewards
    }
    ...
``` [2](#0-1) 

The critical state variable `last_reward_block` is a single global value. Once it is written to the current block number, any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The `IStakingRewardsManager` interface exposes this function publicly with no access-control annotation: [4](#0-3) 

---

### Impact Explanation

An attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` on every new block. Each call:

1. Passes the `REWARDS_ALREADY_UPDATED` guard (new block number > stored value).
2. Writes `last_reward_block = current_block_number`.
3. Returns early at the `if disable_rewards` branch — no rewards are distributed.

The sequencer's subsequent call on the same block reverts. Because `last_reward_block` is global (not per-staker), a single attacker transaction per block is sufficient to deny rewards for **all** stakers and pool members indefinitely.

This constitutes **permanent freezing of unclaimed yield** for every staker and every pool member in the protocol. [5](#0-4) 

---

### Likelihood Explanation

- The function is publicly callable by any address with no preconditions beyond the contract being unpaused and a valid active staker address existing (trivially satisfiable).
- The attacker needs only to submit one transaction per block, which is cheap on Starknet.
- No privileged access, leaked key, or external dependency is required.
- The attack is fully griefing-viable: the attacker gains nothing but permanently denies yield to all protocol participants.

---

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the registered Starkware sequencer address (analogous to `assert_caller_is_attestation_contract` used in `update_rewards_from_attestation_contract`):

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
}
``` [6](#0-5) 

The sequencer address should be stored at initialization and exposed via a config getter, following the same pattern used for the attestation contract address.

---

### Proof of Concept

```cairo
// Attacker contract — called once per block via automation
fn grief_rewards(staking: IStakingRewardsManagerDispatcher, victim_staker: ContractAddress) {
    // No special role needed. Any caller works.
    // Consumes last_reward_block for this block; sequencer call will revert.
    staking.update_rewards(staker_address: victim_staker, disable_rewards: true);
}
```

After this call, the sequencer's `update_rewards(..., disable_rewards: false)` on the same block reverts with `REWARDS_ALREADY_UPDATED`. Repeated every block, no staker or pool member ever accumulates consensus-phase yield. [2](#0-1)
