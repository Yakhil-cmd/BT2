### Title
Silo Whitelist Bypass in `receive_base_tokens` Allows Non-Whitelisted EVM Addresses to Receive Base ETH — (`engine/src/engine.rs`)

---

### Summary

In Silo mode, `receive_erc20_tokens` correctly enforces the address whitelist by redirecting tokens to a configured fallback address when the recipient is not whitelisted. However, the sibling function `receive_base_tokens`, which handles bridged ETH (base token) deposits via the same `ft_on_transfer` entry point, performs no whitelist check at all. Any caller can bridge ETH to an arbitrary, non-whitelisted EVM address in Silo mode, bypassing the access control the operator intended to enforce.

---

### Finding Description

Aurora Engine's Silo mode exposes a whitelist system (`WhitelistKind::Address`) that is meant to restrict which EVM addresses may interact with the EVM. When a NEP-141 token is bridged into Aurora via `ft_on_transfer`, the engine branches on whether the predecessor is the ETH connector (base ETH) or an ERC-20 token contract:

```
ft_on_transfer  →  predecessor == eth_connector  →  receive_base_tokens
                →  predecessor == erc20 contract  →  receive_erc20_tokens
```

`receive_erc20_tokens` contains the following guard:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

This redirects ERC-20 tokens to the operator-configured fallback address when the recipient is not whitelisted.

`receive_base_tokens`, however, contains no such check:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    // ...
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
``` [2](#0-1) 

The recipient address is taken directly from the caller-controlled `msg` field and ETH is minted to it unconditionally, regardless of whether the address whitelist is enabled or whether the address is present in it.

The `ft_on_transfer` dispatcher that calls both functions performs no whitelist check of its own: [3](#0-2) 

The whitelist functions that should be consulted are already defined and used elsewhere: [4](#0-3) 

---

### Impact Explanation

In a Silo deployment with the `Address` whitelist enabled and a fallback address configured, the operator's intent is that only whitelisted EVM addresses may receive tokens. ERC-20 tokens respect this intent; base ETH does not. An attacker (or any user) can bridge ETH to any EVM address — including addresses the operator has deliberately excluded — by calling `ft_transfer_call` on the ETH connector with the target address encoded in the `msg` field.

Once ETH is minted to a non-whitelisted address:

- The address cannot submit EVM transactions (`assert_access` blocks it via `is_allow_submit`). [5](#0-4) 
- If the address is a contract address or otherwise has no corresponding NEAR account, there is no path to recover the ETH — **permanent freezing of funds**.
- If the address does correspond to a NEAR account, the NEAR-level `withdraw` function (which also lacks a whitelist check) may allow recovery, making this a **temporary freeze** in that narrower case.

The inconsistency with `receive_erc20_tokens` confirms this is unintended behavior, not a design choice.

---

### Likelihood Explanation

This requires Silo mode to be active (`set_silo_params` called with a fallback address), the `Address` whitelist to be enabled, and a user to bridge ETH to a non-whitelisted address. Silo is an explicitly supported production deployment mode. The entry point (`ft_on_transfer` via `ft_transfer_call` on the ETH connector) is a standard, publicly reachable bridge operation requiring no special privileges. Any token holder with ETH on NEAR can trigger this path.

---

### Recommendation

Add the same fallback-address guard to `receive_base_tokens` that exists in `receive_erc20_tokens`:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
{
    receipient = fallback_address;
}
```

This ensures that in Silo mode, base ETH deposits to non-whitelisted addresses are redirected to the operator-configured fallback address, consistent with ERC-20 token behavior.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a non-zero `erc20_fallback_address` and enable the `Address` whitelist via `set_whitelist_status`.
2. Do **not** add EVM address `0xAAAA...` to the `Address` whitelist.
3. Hold ETH-on-NEAR (NEP-141 from the ETH connector) in any NEAR account.
4. Call `ft_transfer_call` on the ETH connector with:
   - `receiver_id` = Aurora engine account
   - `msg` = hex-encoded `0xAAAA...`
5. Aurora's `ft_on_transfer` fires → `receive_base_tokens` is called → ETH is minted to `0xAAAA...` with no whitelist check.
6. Confirm: `0xAAAA...` has a non-zero ETH balance in Aurora but cannot submit any EVM transaction (`assert_access` returns `NotAllowed`). If `0xAAAA...` is a pure EVM address with no NEAR account mapping, the ETH is permanently unrecoverable. [6](#0-5) [7](#0-6)

    pub fn receive_base_tokens(

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}

fn is_account_allowed<I: IO + Copy>(io: &I, account: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Account);
    !list.is_enabled() || list.is_exist(account)
}
