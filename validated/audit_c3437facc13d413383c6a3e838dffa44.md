### Title
`TrieProof.Steps` Type Incompatible with `NodeMaxHeight` Causes Silent Proof Verification Failure at Maximum Trie Depth - (File: `ledger/trie.go`)

---

### Summary

`TrieProof.Steps` is typed as `uint8` (maximum value 255), but the MTrie's canonical height is `NodeMaxHeight = PathLen * 8 = 256`. When a proof traverses all 256 interim levels of a fully-expanded trie, the `Steps` counter silently wraps to `0` via unsigned integer overflow. `VerifyTrieProof()` then computes a wrong `leafHeight` of 256 (instead of 0), skips the entire hash-reconstruction loop, and returns `false` for a structurally valid inclusion proof. The code itself acknowledges the mismatch with an inline `TODO`.

---

### Finding Description

**Component 1 — Trie height constant:**

`NodeMaxHeight = PathLen * 8 = 32 * 8 = 256` [1](#0-0) 

The MTrie root is always at height 256. A fully-expanded leaf sits at height 0, meaning the path from root to leaf passes through exactly **256** interim nodes.

**Component 2 — Proof step counter typed as `uint8`:**

```go
Steps uint8 // number of steps for the proof (path len) // TODO: should this be a type allowing for larger values?
``` [2](#0-1) 

`uint8` holds values `[0, 255]`. The maximum representable step count is **255**, one less than the trie height.

**Component 3 — Proof generation increments `Steps` per interim node:**

```go
for _, p := range proofs {
    p.Steps++
}
``` [3](#0-2) 

For a leaf at height 0 (fully expanded), `prove()` visits 256 interim nodes and calls `p.Steps++` 256 times. After the 256th increment, `uint8` wraps: `255 + 1 = 0`.

**Component 4 — Verifier uses `Steps` to reconstruct `leafHeight`:**

```go
treeHeight := ledger.NodeMaxHeight          // 256
leafHeight := treeHeight - int(p.Steps)    // 256 - 0 = 256  (wrong; should be 0)
if !(0 <= leafHeight && leafHeight <= treeHeight) { // 256 ≤ 256 → passes!
    return false
}
computed := ledger.ComputeCompactValue(..., leafHeight) // computed at height 256
for h := leafHeight + 1; h <= treeHeight; h++ {        // 257 ≤ 256 → loop never runs
    ...
}
return computed == hash.Hash(expectedState)             // wrong hash → false
``` [4](#0-3) 

The sanity check `0 <= 256 <= 256` passes, so no error is returned. The hash-reconstruction loop is skipped entirely. The computed value is `ComputeCompactValue(path, payload, 256)` — the root hash of a trie containing **only** that one register — which will not match the actual multi-register trie root. `VerifyTrieProof` returns `false` for a structurally valid proof.

**The incompatibility in one line:**

| Component | Assumed depth |
|---|---|
| `NodeMaxHeight` (trie height) | **256** |
| `TrieProof.Steps` (`uint8` max) | **255** |

This is the direct Flow analog of the external report's "tree depth incompatible" class: two interacting components assume different maximum depths.

---

### Impact Explanation

Any inclusion proof generated for a register whose trie path requires traversal of all 256 interim levels will have `Steps = 0` after overflow. `VerifyTrieProof` will silently return `false` for that proof. In the Flow protocol, `VerifyTrieBatchProof` is used by Verification Nodes to validate chunk execution proofs. A false return causes a valid chunk to be treated as having an invalid proof, which can lead to a Verification Node incorrectly challenging a correct Execution Result — corrupting the slashing/challenge mechanism. [5](#0-4) 

---

### Likelihood Explanation

Triggering the overflow requires two storage registers whose SHA3-256-derived trie paths share all 255 leading bits (differing only in the last bit), forcing the trie to expand to full depth 256 for those registers. Because paths are derived via SHA3-256 from register keys, an unprivileged Cadence contract author cannot directly control path bits; finding such a collision is computationally infeasible. The likelihood is therefore **very low** in practice, though the structural incompatibility is real, present in production code, and acknowledged by the developers via the inline `TODO` comment.

---

### Recommendation

Change `Steps` from `uint8` to `uint16` (or larger) in `TrieProof`:

```go
Steps uint16 // number of steps for the proof (path len)
```

Update all encoding/decoding in `encodeTrieProof` / `decodeTrieProof` accordingly (currently encoded as a single byte via `AppendUint8` / `ReadUint8`). [6](#0-5) [7](#0-6) 

Add a compile-time or init-time assertion that `NodeMaxHeight` fits within the `Steps` type.

---

### Proof of Concept

1. Insert two registers into an MTrie whose paths share 255 leading bits (e.g., paths `0x00…00` and `0x00…01`). The trie expands to full depth 256 for those registers.
2. Call `mt.UnsafeProofs([]ledger.Path{path0})` to generate a proof for `path0`.
3. Observe that `proof.Steps == 0` (overflowed from 256).
4. Call `proof.VerifyTrieProof(proof, rootHash)` — it returns `false` despite the proof being structurally valid.

Relevant code path:

- `prove()` → `p.Steps++` × 256 → `Steps = 0` (overflow) [3](#0-2) 
- `VerifyTrieProof()` → `leafHeight = 256 - 0 = 256` → loop skipped → wrong hash → `false` [8](#0-7)

```go
func VerifyTrieProof(p *ledger.TrieProof, expectedState ledger.State) bool {
	treeHeight := ledger.NodeMaxHeight
	leafHeight := treeHeight - int(p.Steps)             // p.Steps is the number of edges we are traversing until we hit the compactified leaf.
	if !(0 <= leafHeight && leafHeight <= treeHeight) { // sanity check
		return false
	}
	// We start with the leaf and hash our way upwards towards the root
	proofIndex := len(p.Interims) - 1                                                        // the index of the last non-default value furthest down the tree (-1 if there is none)
	computed := ledger.ComputeCompactValue(hash.Hash(p.Path), p.Payload.Value(), leafHeight) // we first compute the hash of the compact leaf (at height leafHeight)
	for h := leafHeight + 1; h <= treeHeight; h++ {                                          // then, we hash our way upwards until we hit the root (at height `treeHeight`)
		// we are currently at a node n (initially the leaf). In this iteration, we want to compute the
		// parent's hash. Here, h is the height of the parent, whose hash want to compute.
		// The parent has two children: child n, whose hash we have already computed (aka `computed`);
		// and the sibling to node n, whose hash (aka `siblingHash`) must be defined by the Proof.

		var siblingHash hash.Hash
		flag := bitutils.ReadBit(p.Flags, treeHeight-h)

		if flag == 1 { // if flag is set, siblingHash is stored in the proof
			if proofIndex < 0 { // proof invalid: too few values
				return false
			}
			siblingHash = p.Interims[proofIndex]
			proofIndex--
		} else { // otherwise, siblingHash is a default hash
			siblingHash = ledger.GetDefaultHashForHeight(h - 1)
		}

		bit := bitutils.ReadBit(p.Path[:], treeHeight-h)
		// hashing is order dependent
		if bit == 1 { // we hash our way up to the parent along the parent's right branch
			computed = hash.HashInterNode(siblingHash, computed)
		} else { // we hash our way up to the parent along the parent's left branch
			computed = hash.HashInterNode(computed, siblingHash)
		}
	}
	return computed == hash.Hash(expectedState)
```
