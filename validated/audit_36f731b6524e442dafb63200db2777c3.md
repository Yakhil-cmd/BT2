### Title
`tick()` is `payable` but permanently locks ETH when `amountToMint == 0` - (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is declared `external payable` and is designed to accept ETH to pay L1→L2 message fees. However, when there are no pending L2 mint requests (`amountToMint == 0`), the entire fee-spending block is skipped and `msg.value` is never used or refunded. Any ETH sent in this case is permanently locked in the contract, which has no ETH withdrawal mechanism.

---

### Finding Description

`tick()` in `RewardSupplier.sol` is `payable` and splits `msg.value` between two downstream calls only inside the `if (amountToMint > 0)` branch:

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol
function tick() external payable {                          // line 107
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {                                 // line 111
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);    // line 126
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // line 137
    }
    // If amountToMint == 0: msg.value is silently accepted and never used
}
```

When `requiredMinting()` returns `(0, 0)` — which happens whenever there are no pending L2→L1 mint-request messages — the `if` block is entirely skipped. Any ETH sent by the caller is accepted by the contract but never forwarded or refunded.

The contract inherits from `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl`. None of these define a `receive()` function or any ETH withdrawal path, so the locked ETH is irrecoverable.

This is the direct analog of the reported vulnerability: in the original, a `payable` function's `msg.value` causes a revert via a non-`payable` delegatecall target; here, a `payable` function silently swallows `msg.value` when the active code path doesn't consume it — both stem from a `payable` function that does not correctly handle `msg.value` in all execution paths.

---

### Impact Explanation

Any ETH sent to `tick()` when `amountToMint == 0` is permanently frozen in the `RewardSupplier` contract. There is no `receive()` fallback, no `withdraw` function, and no admin rescue path visible in the contract or its storage/proxy parents. This constitutes **permanent freezing of funds** for the caller.

---

### Likelihood Explanation

`tick()` is a public, permissionless function callable by any account. The condition `amountToMint == 0` is the normal state of the contract between L2 mint-request cycles — it is not an edge case. A caller who sends ETH to `tick()` expecting to trigger minting (but arriving slightly early, or after another caller already processed the messages) will permanently lose their ETH. The likelihood is **medium-high** given the function is designed to be called by arbitrary accounts and the zero-mint path is the common steady state.

---

### Recommendation

Add a guard that reverts when `msg.value > 0` but `amountToMint == 0`, or refund unused ETH at the end of the function:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_NEEDED_ETH_REJECTED");
        return;
    }
    // ... existing logic
}
```

Alternatively, refund any unspent ETH after the branch executes.

---

### Proof of Concept

1. Deploy `RewardSupplier` with no pending L2→L1 mint-request messages (the normal idle state).
2. Call `tick{value: 1 ether}()` from any EOA.
3. `requiredMinting()` returns `(0, 0)` → `amountToMint == 0`.
4. The `if` block is skipped; the function returns successfully.
5. The contract's ETH balance increases by 1 ether.
6. There is no function in `RewardSupplier`, `RewardSupplierStorage`, `Identity`, or `ProxySupportImpl` to withdraw that ETH.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
