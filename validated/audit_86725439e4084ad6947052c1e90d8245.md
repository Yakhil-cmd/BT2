### Title
Missing Owner Authorization on `deploy_upgrade` Allows Anyone to Trigger Contract Code Deployment - (File: `engine/src/lib.rs`)

### Summary
The `deploy_upgrade` function in the Aurora Engine NEAR contract is callable by any NEAR account once the upgrade delay has elapsed. While `stage_upgrade` (which stores the code to be deployed) correctly enforces `require_owner_only`, `deploy_upgrade` only checks `require_running` and the block-height delay. This means an unprivileged NEAR account can force deployment of staged contract code at any time after the delay, including front-running an owner's attempt to cancel a flawed upgrade.

### Finding Description
`stage_upgrade` correctly gates code staging behind `require_owner_only`: [1](#0-0) 

However, `deploy_upgrade` — which actually executes `Runtime::self_deploy` to replace the live contract bytecode — performs no caller identity check: [2](#0-1) 

The only guards are `require_running` and a block-height comparison against the stored upgrade index. [3](#0-2) 

`require_owner_only` is defined and used consistently throughout the rest of the admin surface (e.g., `set_owner`, `pause_contract`, `upgrade`, `stage_upgrade`, `attach_full_access_key`): [4](#0-3) 

Its absence in `deploy_upgrade` is inconsistent with the rest of the access-control model.

### Impact Explanation
Once the owner calls `stage_upgrade` and the `upgrade_delay_blocks` window elapses, any NEAR account can call `deploy_upgrade` and atomically replace the live engine bytecode via `Runtime::self_deploy`. There is no `cancel_upgrade` function in the contract; the only way for the owner to abort a staged upgrade is to overwrite it with a new `stage_upgrade` call. An attacker can observe the mempool/block stream and call `deploy_upgrade` in the same block the delay expires, before the owner's cancellation transaction is processed. If the staged code contains a vulnerability (e.g., discovered after staging), the attacker forces its deployment, potentially enabling fund theft or permanent fund freeze across all ETH and NEP-141 assets held by the engine.

**Impact tier**: Critical — permanent freezing of funds / direct theft of user funds held in the Aurora Engine contract.

### Likelihood Explanation
The attack requires:
1. The owner to have called `stage_upgrade` (a routine operational action for upgrades).
2. The upgrade delay to have elapsed (publicly observable on-chain).
3. The owner to want to abort the upgrade (e.g., after a post-staging audit finding).

All three conditions are realistic in a production upgrade workflow. The attacker needs only to monitor the chain for a staged upgrade and call a single permissionless function — no special privileges, tokens, or capital are required.

### Recommendation
Add `require_owner_only` (or at minimum `require_owner_and_running`) to `deploy_upgrade`, consistent with every other privileged admin function in the contract:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    // ADD: enforce caller is the owner
    require_owner_only(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

### Proof of Concept

1. Owner calls `stage_upgrade` with new bytecode `B`. This stores `B` under `KeyPrefix::Config / CODE_KEY` and records `block_height + upgrade_delay_blocks` under `CODE_STAGE_KEY`. [5](#0-4) 
2. Owner discovers a critical bug in `B` and prepares a `stage_upgrade` transaction with safe bytecode `B'` to overwrite it.
3. Attacker monitors the chain. As soon as `block_height > stored_index`, the attacker submits a `deploy_upgrade` call (no special arguments needed).
4. `deploy_upgrade` passes both guards (`require_running` and block-height check) and executes `Runtime::self_deploy`, replacing the live contract with the buggy `B`. [6](#0-5) 
5. The owner's cancellation `stage_upgrade` transaction now executes against the already-replaced (buggy) contract, which may behave differently or be exploitable.
6. All ETH and NEP-141 funds held by the Aurora Engine are now at risk under the buggy bytecode.

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {

        let state = state::get_state(&io)?;

    })
}

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
