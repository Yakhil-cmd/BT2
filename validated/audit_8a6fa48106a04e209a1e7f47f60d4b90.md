### Title
Missing Whitelist Check for ETH Recipient in `receive_base_tokens` Bypasses Silo Address Whitelist — (`engine/src/engine.rs`)

---

### Summary

In Silo mode, `Engine::receive_base_tokens` mints ETH directly to a user-supplied EVM address without checking the `WhitelistKind::Address` whitelist. The sibling function `Engine::receive_erc20_tokens` correctly performs this check and redirects to a fallback address when the recipient is not whitelisted. The omission in `receive_base_tokens` means any NEAR user can deposit ETH to any EVM address — including addresses explicitly excluded from the Silo whitelist — bypassing the access control system. Because unwhitelisted addresses cannot submit EVM transactions, any ETH minted to them is permanently frozen.

---

### Finding Description

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to one of two paths depending on the predecessor: [1](#0-0) 

When the predecessor is the ETH connector account, `receive_base_tokens` is called. That function parses the recipient from the user-controlled `msg` field and immediately credits the balance — **no whitelist check is performed**: [2](#0-1) 

By contrast, `receive_erc20_tokens` explicitly calls `silo::is_allow_receive_erc20_tokens` and redirects to the configured fallback address when the recipient is not whitelisted: [3](#0-2) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which checks the `WhitelistKind::Address` list: [4](#0-3) [5](#0-4) 

The same `WhitelistKind::Address` list is enforced for transaction submission via `assert_access`: [6](#0-5) 

So an address that is not in the `Address` whitelist cannot submit any EVM transaction, but `receive_base_tokens` will still mint ETH to it.

---

### Impact Explanation

**Permanent freezing of funds.** ETH minted to an unwhitelisted EVM address in Silo mode is irrecoverable:

- The address cannot call `submit` or `call` (blocked by `assert_access` / `is_allow_submit`).
- The `exit_to_near` precompile is only reachable from within the EVM, which requires submitting a transaction — also blocked.
- There is no NEAR-level mechanism to drain an arbitrary EVM address's ETH balance on behalf of its owner.

The depositor's ETH is permanently lost inside the engine.

---

### Likelihood Explanation

Likelihood is **medium**. The preconditions are:

1. The Aurora deployment is running in Silo mode with `SiloParamsArgs` set (i.e., `erc20_fallback_address` configured).
2. The `WhitelistKind::Address` whitelist is enabled.
3. A user deposits ETH via the standard NEP-141 `ft_transfer_call` path targeting an EVM address that is not whitelisted.

Condition 3 can be triggered by any unprivileged NEAR account — the `msg` field of `ft_transfer_call` is fully user-controlled and is the sole source of the recipient address in `receive_base_tokens`. No special role or key is required.

---

### Recommendation

Apply the same whitelist guard used in `receive_erc20_tokens` to `receive_base_tokens`. When a fallback address is configured and the recipient is not whitelisted, redirect the ETH credit to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Mirror the guard already present in receive_erc20_tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    sdk::log!("Mint {amount} base tokens for: {}", recipient.encode());
    Ok(None)
}
```

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a non-zero `erc20_fallback_address`.
2. Enable the `Address` whitelist: call `set_whitelist_status` with `WhitelistKind::Address, active: true`.
3. Do **not** add victim EVM address `V` to the whitelist.
4. From any NEAR account, call `ft_transfer_call` on the ETH connector contract:
   - `receiver_id`: Aurora Engine account
   - `amount`: any ETH amount
   - `msg`: hex encoding of `V`
5. The ETH connector calls `ft_on_transfer` on Aurora Engine; `predecessor_account_id == connector_account_id`, so `receive_base_tokens` is invoked.
6. `receive_base_tokens` parses `V` from `msg` and calls `set_balance` — **no whitelist check occurs**.
7. ETH is now credited to `V`. Any attempt by `V` to submit a transaction is rejected by `assert_access` → `is_allow_submit` → `is_address_allowed` (whitelist enabled, `V` absent).
8. The ETH is permanently frozen. [2](#0-1) [3](#0-2) [1](#0-0) [4](#0-3) [7](#0-6)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
