### Title
Permanent Fund Loss on Failed `exit_to_near` Promise When `error_refund` Feature Is Absent - (File: `engine-precompiles/src/native.rs`)

### Summary
When the `error_refund` compile-time feature is not enabled, both ETH and ERC-20 tokens are permanently destroyed/frozen if the downstream NEP-141 `ft_transfer` promise fails during an `exit_to_near` call. No failure-handling callback is attached in this configuration, leaving burned ERC-20 tokens unrecoverable and ETH permanently locked at the precompile address.

### Finding Description

**Root cause — conditional callback attachment in `ExitToNear::run`:**

In `engine-precompiles/src/native.rs`, the precompile builds `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For a standard ERC-20 exit, `transfer_near_args` is also `None` (returned by `exit_erc20_token_to_near`). This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, triggering the branch that emits a bare promise with **no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

**ERC-20 path — tokens burned before promise:**

`EvmErc20.sol`'s `withdrawToNear` burns the caller's tokens unconditionally before invoking the precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← irreversible
    // ... calls exit_to_near precompile ...
}
``` [3](#0-2) 

If the subsequent `ft_transfer` to the NEP-141 contract fails (e.g., recipient not registered, storage deposit missing), there is no callback to re-mint the burned tokens. The `exit_to_near_precompile_callback` is never scheduled, so the silent loss is complete.

**ETH path — value locked at precompile address:**

For ETH exits, `context.apparent_value` is transferred to the `exit_to_near` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) during EVM execution. The `refund_on_error` function — which transfers ETH back from that address — is only reachable via the callback:

```rust
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op; ETH stays at precompile address forever
};
``` [4](#0-3) 

The precompile address has no private key; ETH stranded there is permanently inaccessible.

**The refund path itself (for reference):** [5](#0-4) 

**Test explicitly confirms the loss:**

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `withdrawToNear` (ERC-20) or the ETH exit precompile when the downstream NEP-141 `ft_transfer` fails will permanently lose their tokens. ERC-20 tokens are burned with no re-mint path; ETH is locked at a keyless precompile address. There is no administrative recovery mechanism.

### Likelihood Explanation

**Medium.** The `error_refund` feature is a non-default compile-time flag. Deployments compiled without it — including any Silo or custom Aurora deployment — are fully exposed. The trigger condition (failed `ft_transfer`) is easily reached by any unprivileged user: sending to an unregistered NEAR account, an account with insufficient storage deposit, or any NEP-141 that rejects the transfer. No special privilege is required.

### Recommendation

1. **Make `error_refund` a default feature** in `engine-precompiles/Cargo.toml` so it is always compiled into production binaries.
2. **Remove the conditional branch** that emits a bare `PromiseArgs::Create` for ERC-20 exits; always attach the failure-handling callback.
3. **Audit all exit paths** (ETH, ERC-20, wNEAR unwrap) to confirm a callback is unconditionally present whenever tokens are deducted before the promise is dispatched.

### Proof of Concept

1. Compile Aurora Engine **without** the `error_refund` feature flag.
2. Deploy a NEP-141 token and bridge it to an ERC-20 on Aurora.
3. As a normal EVM user, call `erc20.withdrawToNear("unregistered.near", amount)`.
4. Observe: ERC-20 balance decreases by `amount` (tokens burned).
5. The `ft_transfer` promise to the NEP-141 fails — `"unregistered.near"` is not registered.
6. No `exit_to_near_precompile_callback` is scheduled (bare `PromiseArgs::Create` was used).
7. Observe: ERC-20 balance remains at `original - amount`; NEP-141 balance of `"unregistered.near"` is 0. Tokens are permanently lost with no recovery path.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
