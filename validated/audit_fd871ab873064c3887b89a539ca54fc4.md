### Title
`MsgTurnBridge` Handler Is a No-Op Stub — Bridge Circuit Breaker Permanently Broken - (File: x/cronos/keeper/msg_server.go)

### Summary

The `TurnBridge` message handler, which is the only on-chain mechanism to disable the Gravity bridge in an emergency, is implemented as a no-op stub that performs no permission check and changes no state. Any unprivileged user can submit a `MsgTurnBridge` transaction that succeeds (returns `nil, nil`) while the bridge remains permanently enabled and uncontrollable.

### Finding Description

`MsgTurnBridge` is a fully registered, publicly reachable Cosmos SDK message defined in the proto service and codec. Its handler in `x/cronos/keeper/msg_server.go` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Two distinct defects are present simultaneously:

**1. Missing permission check.** Every other privileged handler in the same file enforces authorization. `UpdateTokenMapping` calls `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`, `UpdatePermissions` checks `admin != msg.From`, and `StoreBlockList` does the same. `TurnBridge` skips all of this. The `CanTurnBridge` permission bit is defined and queryable but is never enforced at the execution boundary. [2](#0-1) [3](#0-2) 

**2. No state mutation.** The `Keeper` struct contains no `gravityKeeper` field and no bridge-active flag. The handler returns immediately without writing anything to the KV store. The Gravity bridge's `bridge_active` parameter (queried as `cli.query_gravity_params()["bridge_active"]` in integration tests) is never touched. [4](#0-3) 

The integration test `test_gravity_turn_bridge` documents the intended behavior: an unpermissioned caller should be rejected, a permissioned caller should succeed, and subsequent `send_to_evm_chain` calls should fail. None of these invariants hold with the current implementation. [5](#0-4) 

### Impact Explanation

The Gravity bridge circuit breaker is permanently non-functional. In an emergency — a bridge exploit, an oracle manipulation, or a supply-inflation attack — the `CronosAdmin` and any address holding `CanTurnBridge` permission have no on-chain path to halt outbound bridge flows. Tokens can continue to be burned on Cronos and claimed on Ethereum (or vice versa) without any ability to intervene. This is a permanent inability for the admin to exercise a critical bridge authorization control, and it directly enables continued unauthorized asset movement through the bridge during an active exploit.

This maps to:
- **High: Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks** — `CanTurnBridge` is defined but never enforced.
- **High: Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows** — the circuit breaker is permanently broken.

### Likelihood Explanation

The defect is present in every deployment of this codebase. No special conditions are required to trigger it: any user can submit a `MsgTurnBridge` transaction that succeeds silently. The inability to disable the bridge is permanent until a chain upgrade replaces the handler. The Gravity bridge is an active, production-facing component with real asset flows.

### Recommendation

1. Add the missing permission check at the top of `TurnBridge`, mirroring the pattern used in `UpdateTokenMapping`:
   ```go
   if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
       return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
   }
   ```
2. Implement the actual bridge-disable logic. If the Gravity module exposes a `SetBridgeActive` or equivalent parameter setter, call it here. If not, store a `bridge_active` boolean in the Cronos KV store and check it in the relevant EVM hooks (`SendToEvmChainHandler`, `SendToEthereumHandler`).
3. Add a unit test that verifies (a) an unpermissioned caller is rejected, (b) a permissioned caller succeeds, and (c) the bridge-active state is actually mutated.

### Proof of Concept

```
# 1. Any unprivileged address submits MsgTurnBridge with enable=false
cronosd tx cronos turn-bridge false --from attacker

# Result: tx succeeds (code 0), no error
# Bridge state: unchanged, bridge_active remains true

# 2. Admin attempts the same
cronosd tx cronos turn-bridge false --from admin

# Result: tx succeeds (code 0), no error  
# Bridge state: still unchanged, bridge_active remains true

# 3. Gravity bridge exploit occurs; admin cannot halt outbound flows
# send_to_evm_chain continues to succeed and burn CRC21 tokens
```

The handler at `x/cronos/keeper/msg_server.go:85-87` returns `nil, nil` unconditionally, so both calls above produce a successful transaction receipt while the bridge remains permanently enabled and uncontrollable. [1](#0-0)
