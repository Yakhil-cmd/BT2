### Title
Missing Verification of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator receives `(big_y, big_c)` ElGamal ciphertext shares from every participant and accumulates them without any cryptographic verification that each share was computed honestly. A malicious participant can submit arbitrary group elements, causing the coordinator to output a corrupted derived key that honest parties will accept as valid.

---

### Finding Description

The external report's vulnerability class is **missing validation of received values in one code path while another code path performs the check**. The direct analog here is that the CKD coordinator receives participant outputs and blindly accumulates them with no proof-of-knowledge or commitment check, while the DKG protocol — which handles structurally similar secret-share contributions — enforces both `verify_proof_of_knowledge` and `verify_commitment_hash` before accepting any participant's material.

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55):

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant $P_i$ is supposed to compute:

- $y_i \xleftarrow{R} \mathbb{Z}_q^*$, $Y_i = y_i \cdot G$
- $S_i = x_i \cdot H(\mathsf{pk} \| \mathsf{app\_id})$ (secret share contribution)
- $C_i = S_i + y_i \cdot A$ (ElGamal encryption under app public key $A$)
- Normalize: $(\lambda_i Y_i,\ \lambda_i C_i)$

The coordinator sums all normalized pairs to obtain the aggregate ElGamal ciphertext. The app then decrypts with its secret key $a$:

$$\sum_i \lambda_i C_i - a \cdot \sum_i \lambda_i Y_i = \mathsf{msk} \cdot H(\mathsf{pk} \| \mathsf{app\_id})$$

**No proof is required or checked** that:
1. $Y_i$ is a well-formed commitment to $y_i$ (i.e., $Y_i = y_i \cdot G$).
2. $C_i$ was formed using the participant's actual secret share $x_i$ and the same $y_i$.

A malicious participant can send any pair $(\hat{Y}, \hat{C}) \in \mathbb{G}_1^2$ and the coordinator will incorporate it unconditionally. [1](#0-0) 

Contrast this with the DKG protocol, which verifies every received share against a published commitment before accumulating it: [2](#0-1) 

And verifies proofs of knowledge before accepting any commitment: [3](#0-2) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

A malicious participant $P_m$ sends $(\hat{Y}, \hat{C})$ instead of $(\lambda_m Y_m, \lambda_m C_m)$. The coordinator computes:

$$\text{out\_big\_y} = \sum_{i \neq m} \lambda_i Y_i + \hat{Y}, \quad \text{out\_big\_c} = \sum_{i \neq m} \lambda_i C_i + \hat{C}$$

When the app decrypts:

$$\text{out\_big\_c} - a \cdot \text{out\_big\_y} = \mathsf{msk} \cdot H(\mathsf{pk}\|\mathsf{app\_id}) + \underbrace{(\hat{C} - a \cdot \hat{Y})}_{\text{attacker-controlled bias}}$$

The derived confidential key is shifted by an attacker-controlled value. The coordinator outputs this corrupted `CKDOutput` as a valid result; honest parties have no mechanism to detect the corruption. The app will decrypt to a wrong key, silently breaking any downstream use of the derived key (e.g., decryption of data encrypted under the expected key). [4](#0-3) 

---

### Likelihood Explanation

**Medium-High.** Any single participant in the CKD protocol is an unprivileged attacker-controlled entry point. The attack requires only that the malicious participant send a malformed message in the single protocol round — no special access, no key material, and no coordination with other parties is needed. The protocol has exactly one round of participant-to-coordinator communication, making the attack trivially reachable. [5](#0-4) 

---

### Recommendation

Add a Sigma protocol (proof of knowledge / DLEQ proof) to each participant's contribution so the coordinator can verify that:

1. $Y_i = y_i \cdot G$ for the same $y_i$ used in $C_i$ (discrete-log proof on $Y_i$).
2. $C_i - S_i = y_i \cdot A$ for the publicly verifiable $S_i = x_i \cdot H(\mathsf{pk}\|\mathsf{app\_id})$ (DLEQ proof linking $Y_i$ and $C_i - S_i$ to the same $y_i$).

The coordinator must reject any participant whose proof fails before accumulating their contribution, mirroring the pattern already used in DKG (`verify_proof_of_knowledge`, `validate_received_share`). [6](#0-5) 

---

### Proof of Concept

**Setup:** 3 participants, threshold 2, coordinator = $P_1$. Honest participants $P_2, P_3$ compute correct shares. Malicious $P_2$ instead sends $(\hat{Y}, \hat{C}) = (0_{\mathbb{G}_1}, 0_{\mathbb{G}_1})$ (identity elements).

**Execution path:**

1. `ckd()` is called; `run_ckd_protocol` dispatches $P_1$ to `do_ckd_coordinator` and $P_2$ to `do_ckd_participant`.
2. $P_2$ calls `chan.send_private(waitpoint, coordinator, &(identity, identity))` — no validation prevents this.
3. Coordinator's loop at line 50–55 adds the identity elements: `norm_big_y += identity` (no-op), `norm_big_c += identity` (no-op).
4. The coordinator outputs `CKDOutput::new(norm_big_y, norm_big_c)` missing $P_2$'s secret share contribution entirely.
5. App decrypts: result is $(\mathsf{msk} - \lambda_2 x_2) \cdot H(\mathsf{pk}\|\mathsf{app\_id})$ — a wrong key — accepted without error. [7](#0-6)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}
