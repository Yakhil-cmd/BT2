### Title
Gateway L2 Gas Price Admission Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Admitting Transactions That Will Fail Execution - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_tx_l2_gas_price_within_threshold` computes the minimum acceptable L2 gas price threshold against the **current** block's `l2_gas_price` field. The transaction will be executed in the **next** block, which uses `next_l2_gas_price` — a distinct field stored in the block header. Under EIP-1559 dynamics the two values can diverge by up to 12.5 % per block. The gateway therefore admits transactions whose `max_price_per_unit` is below the actual execution-block price, and those transactions fail `check_fee_bounds` inside the blockifier, charging the user a fee for a reverted transaction. The same stale price is also forwarded into `run_validate_entry_point`'s `BlockContext`, so the blockifier pre-validation at the gateway also passes for these transactions.

### Finding Description

**Root cause — wrong price field in `validate_resource_bounds`:** [1](#0-0) 

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;          // ← current block's price, NOT next block's price
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
```

The threshold is then: [2](#0-1) 

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)   // ← wrong base
    .to_integer();
```

**Why `next_l2_gas_price` is the correct base:**

The block header stores two distinct prices: [3](#0-2) 

- `l2_gas_price` — the price used for the **current** block.
- `next_l2_gas_price` — the price that will be used for the **next** block, computed by `calculate_next_base_gas_price`.

A transaction admitted to the mempool will be executed in the next block at `next_l2_gas_price`. The gateway should therefore check `tx.max_price_per_unit >= threshold(next_l2_gas_price)`.

**`next_l2_gas_price` is not exposed through `BlockInfo`:**

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` only copies `l2_gas_price` into the returned `BlockInfo`; `next_l2_gas_price` is silently dropped: [4](#0-3) 

The `BlockInfo` struct has no `next_l2_gas_price` field, so the correct value is structurally inaccessible to the validator.

**Same stale price propagates into `run_validate_entry_point`:** [5](#0-4) 

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next(); // block number advanced…
let block_context = BlockContext::new(block_info, …);               // …but gas prices are NOT
```

The blockifier's `check_fee_bounds` inside `perform_pre_validation_stage` therefore also uses the current block's gas prices: [6](#0-5) 

So a transaction with `max_price_per_unit` between `l2_gas_price` and `next_l2_gas_price` passes **both** the gateway threshold check and the blockifier pre-validation at the gateway, and is admitted to the mempool.

**EIP-1559 price change magnitude:** [7](#0-6) 

The price change per block is `price * gas_delta / (gas_target * denominator)`. With a fully-congested block (`gas_used = max_block_size`) and the default `gas_price_max_change_denominator`, the price can rise by up to ~12.5 % in a single block.

### Impact Explanation

When the network is congested (`next_l2_gas_price > l2_gas_price`):

1. A user submits a transaction with `max_price_per_unit = P` where `l2_gas_price ≤ P < next_l2_gas_price`.
2. The gateway threshold check passes: `P ≥ min_gas_price_percentage% × l2_gas_price`.
3. The blockifier pre-validation at the gateway also passes (same stale price).
4. The transaction is admitted to the mempool.
5. During block execution the blockifier uses the actual `next_l2_gas_price`; `check_fee_bounds` fires `MaxGasPriceTooLow`.
6. The transaction is reverted and the user is charged a fee for a transaction the gateway should have rejected.

When the network is not congested (`next_l2_gas_price < l2_gas_price`):

1. A user submits a transaction with `max_price_per_unit = P` where `next_l2_gas_price ≤ P < l2_gas_price`.
2. The gateway threshold check fails: `P < min_gas_price_percentage% × l2_gas_price`.
3. The transaction is rejected even though it would succeed in the next block.

Both directions match **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The L2 gas price changes every block whenever gas usage deviates from the target, which is the normal operating condition. Any unprivileged user can trigger the incorrect admission path by submitting a transaction with a gas price in the gap between `l2_gas_price` and `next_l2_gas_price`. No special privileges, no malicious peer, no dependency behavior required.

### Recommendation

1. Extend `GatewayFixedBlockStateReader` to expose `next_l2_gas_price` (add it to the returned `BlockInfo` or as a separate accessor).
2. In `validate_resource_bounds`, read `next_l2_gas_price` from the block header and pass it to `validate_tx_l2_gas_price_within_threshold` instead of `l2_gas_price`.
3. In `run_validate_entry_point`, set `block_info.gas_prices` to use `next_l2_gas_price` for the L2 gas price so that the blockifier pre-validation at the gateway is consistent with actual execution conditions.

### Proof of Concept

```
Assume:
  current block l2_gas_price     = 100 FRI   (stored in block header as l2_gas_price)
  current block next_l2_gas_price = 112 FRI   (stored in block header as next_l2_gas_price)
  min_gas_price_percentage        = 100

1. User submits InvokeV3 with l2_gas.max_price_per_unit = 105 FRI.

2. validate_tx_l2_gas_price_within_threshold:
     threshold = 100% × 100 = 100
     105 >= 100  → PASS  (should have been: 105 < 112 → REJECT)

3. run_validate_entry_point builds BlockContext with l2_gas_price = 100 FRI.
   check_fee_bounds: 105 >= 100 → PASS

4. Transaction admitted to mempool.

5. Block N+1 is built with l2_gas_price = 112 FRI.
   perform_pre_validation_stage → check_fee_bounds:
     105 < 112 → MaxGasPriceTooLow → transaction reverted, fee charged.
```
