### Title
WalletConnect `chia_sendTransaction` Hides Source `wallet_id` from Confirmation Dialog, Enabling Silent Cross-Wallet Spend — (File: `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

The `wallet_id` parameter in the `chia_sendTransaction` WalletConnect command is marked `hide: true` in the confirmation-dialog definition. A dapp granted this command can target **any** wallet (XCH, CAT, CR-CAT) by supplying an arbitrary `wallet_id`, and the user's approval dialog will never reveal which wallet is being spent from — only amount, fee, and destination address are shown.

---

### Finding Description

In `Commands.ts`, the `chia_wallet.send_transaction` command definition marks `wallet_id` with `hide: true`:

```js
{ name: 'wallet_id', label: () => i18n._(/* i18n */ { id: 'Wallet Id' }), type: 'number', hide: true }
``` [1](#0-0) 

The corresponding dapp command `chia_sendTransaction` sets `defaults: { wallet_id: 1 }`, so if the dapp omits `wallet_id` it silently defaults to wallet 1 (the standard XCH wallet). However, the dapp can explicitly supply any `wallet_id` (e.g., a CAT wallet with ID 5). [2](#0-1) 

In `parseDappParams.ts`, the allowlist validation passes `wallet_id` through (it is a declared param), and the default is only applied when the value is `undefined`. The validated params — including the dapp-supplied `wallet_id` — are forwarded to `dispatchPairRequest`, which calls the confirmation-dialog builder (`humanizeDappCommand`). Because `wallet_id` is `hide: true`, it is excluded from the `rows` shown to the user in the `Confirm` dialog. [3](#0-2) 

The user sees: **Amount** (humanized as XCH via `mojo-to-xch`), **Fee**, and **Destination address** — but **not** which wallet is being spent from. [4](#0-3) 

The `dispatchPairRequest` main-process handler passes `parsedParams` (including the dapp-controlled `wallet_id`) directly to the wallet RPC after user confirmation: [5](#0-4) 

---

### Impact Explanation

A dapp granted `chia_sendTransaction` can silently target any wallet — XCH, CAT, CR-CAT — without the user knowing the source. The user approves what they believe is an XCH transaction, but the actual spend may come from a CAT wallet. This is a **signing-context confusion**: the user approves the wrong asset/wallet with no indication in the confirmation UI. This maps directly to the allowed High impact: *"WalletConnect state that causes a user to approve the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

The dapp must first be granted `chia_sendTransaction` permission by the user. Once granted, the dapp can freely target any wallet on every subsequent call without further disclosure. A malicious dapp presenting itself as an XCH-only service can exploit this after initial pairing.

---

### Recommendation

Remove `hide: true` from the `wallet_id` parameter in `chia_wallet.send_transaction`, or add a resolved wallet name/type row to the confirmation dialog so users can see which wallet is being spent from before approving. The wallet name can be resolved from the wallet list at confirmation time.

---

### Proof of Concept

1. User pairs a dapp and grants `chia_sendTransaction` permission (expecting XCH sends).
2. Dapp sends: `chia_sendTransaction({ amount: '1000', fee: '0', address: 'xch1...', walletId: 5 })` where wallet 5 is a CAT wallet.
3. `parseDappParams` converts `walletId` → `wallet_id: 5`, passes the allowlist check, skips the default (value is not `undefined`).
4. `dispatchPairRequest` calls `confirm()` → `humanizeDappCommand` builds rows excluding `wallet_id` (`hide: true`).
5. User sees confirmation: *"Amount: 0.000001 XCH, Fee: 0, Address: xch1..."* — no source wallet shown.
6. User approves. Transaction is sent from CAT wallet 5, not XCH wallet 1. [6](#0-5) [7](#0-6)
