### Title
Permanent NEP-141 ↔ ERC-20 Token Mapping Cannot Be Updated or Removed - (File: `engine/src/engine.rs`)

### Summary
Once a NEP-141 token is registered to an ERC-20 address via `register_token`, the mapping is permanently locked in storage. No privileged `remove_token` or `update_token` function exists anywhere in the codebase. If the ERC-20 contract at the registered address is compromised or must be replaced, the engine will continue routing all future NEP-141 bridge deposits to the compromised ERC-20 contract, enabling theft of bridged user funds.

### Finding Description
`Engine::register_token` in `engine/src/engine.rs` inserts a bidirectional NEP-141 ↔ ERC-20 mapping into `nep141_erc20_map` storage:

```rust
// engine/src/engine.rs L722-741
pub fn register_token(
    &mut self,
    erc20_token: Address,
    nep141_token: AccountId,
) -> Result<(), RegisterTokenError> {
    match get_erc20_from_nep141(&self.io, &nep141_token) {
        Err(GetErc20FromNep141Error::Nep141NotFound) => (),
        ...
        Ok(_) => return Err(RegisterTokenError::TokenAlreadyRegistered),
    }
    let erc20_token = ERC20Address(erc20_token);
    let nep141_token = NEP141Account(nep141_token);
    nep141_erc20_map(self.io).insert(&nep141_token, &erc20_token);
    Ok(())
}
```

The explicit `TokenAlreadyRegistered` guard on line 735 means re-registering the same NEP-141 token is rejected. A grep across the entire repository for `remove_erc20`, `unregister_token`, `deregister_token`, or any equivalent returns **no matches**. The mapping is write-once with no administrative escape hatch.

This mapping is consumed in two critical paths:

1. **Bridge deposit path** (`ft_on_transfer` → `receive_erc20_tokens`): uses `get_erc20_from_nep141` to determine which ERC-20 contract to mint tokens into when NEP-141 tokens are bridged from NEAR to Aurora.

2. **XCC router initialization path** (`engine/src/xcc.rs` L218-219): uses `nep141_erc20_map` to resolve the wNEAR account ID from the registered wNEAR ERC-20 address, which is required to initialize every XCC router sub-account.

### Impact Explanation
**Critical — Theft of user funds / Permanent freezing of funds.**

If the ERC-20 contract at the registered address is found to contain a vulnerability (e.g., an unrestricted `mint` or `transferFrom` bug), an attacker can:
- Drain ERC-20 token balances from all holders.
- Call the `exit_to_near` precompile with stolen ERC-20 tokens to redeem the underlying NEP-141 tokens held in custody by the Aurora engine contract, achieving direct theft of bridged user funds.

Because there is no `remove_token` function, the owner cannot redirect the NEP-141 → ERC-20 mapping to a patched replacement contract. Every subsequent `ft_on_transfer` deposit for that NEP-141 token will continue minting into the compromised ERC-20, compounding losses indefinitely.

If the wNEAR ERC-20 specifically is compromised, the XCC subsystem's router initialization path (`engine/src/xcc.rs`) will also be permanently broken, freezing all cross-contract call functionality for every user.

### Likelihood Explanation
**Medium.** The ERC-20 contracts deployed by `deploy_erc20_token` are EVM bytecode contracts. Any latent bug in the ERC-20 template (e.g., in `setup_deploy_erc20_input`) would affect every deployed token. The inability to update the mapping means a single discovered bug has no on-chain remediation path. The wNEAR token is a particularly high-value target given its role in XCC.

### Recommendation
Add a privileged `remove_erc20_token(nep141: AccountId)` function callable only by the contract owner that removes both directions of the `nep141_erc20_map` entry. This mirrors the recommendation in the original report: delete the stale mapping, then re-register with the replacement ERC-20 address via `deploy_erc20_token`. Access must be restricted to the owner (analogous to `require_owner_only`) to prevent griefing.

### Proof of Concept
1. Owner calls `deploy_erc20_token` for `usdt.near` → ERC-20 deployed at address `0xAAA...`, mapping `usdt.near → 0xAAA...` written to storage.
2. A critical bug is discovered in the ERC-20 contract at `0xAAA...` allowing arbitrary minting.
3. Owner attempts to call `deploy_erc20_token` again for `usdt.near` to deploy a patched ERC-20 at `0xBBB...`.
4. `register_token` returns `ERR_NEP141_TOKEN_ALREADY_REGISTERED` (line 735) — the call fails.
5. No `remove_erc20_token` function exists — confirmed by exhaustive grep.
6. Attacker exploits the bug in `0xAAA...`, mints ERC-20 tokens to themselves, calls `exit_to_near` precompile, and redeems the underlying NEP-141 tokens held by the Aurora engine — direct theft of bridged user funds.
7. All future `ft_on_transfer` deposits for `usdt.near` continue minting into the compromised `0xAAA...` contract.
