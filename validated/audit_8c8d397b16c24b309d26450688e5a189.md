### Title
FLOW Tokens Permanently Locked in EVM Environment When `CadenceOwnedAccount` Resource Is Destroyed Without Withdrawing Balance - (File: fvm/evm/stdlib/contract.cdc)

### Summary
The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` has no destructor. When a COA is destroyed while holding a FLOW token balance in the EVM environment, those tokens are permanently locked with no recovery mechanism — a direct analog to the SpiceAuction pattern where deposited tokens become irrecoverable under a specific terminal condition.

### Finding Description
The `CadenceOwnedAccount` resource (declared at line 518 of `contract.cdc`) is the Cadence-side key that controls a COA's EVM address. FLOW tokens deposited into a COA via `cadenceOwnedAccount.deposit(from: <-vault)` are transferred into the EVM state at the COA's EVM address. The only path to recover those tokens back to Cadence is `cadenceOwnedAccount.withdraw(balance:)`, which is guarded by `access(Owner | Withdraw)` — an entitlement that is only reachable through the live COA resource. [1](#0-0) 

The resource has no `destroy` function (destructor). Cadence allows any resource to be explicitly destroyed with `destroy`. When a caller executes `destroy cadenceOwnedAccount`, the Cadence resource is discarded without any cleanup. The EVM-side balance at the COA's address is unaffected — it remains in the EVM state — but it is now permanently inaccessible because:

1. `withdraw` requires the `Owner | Withdraw` entitlement, which is only reachable through the now-destroyed COA resource. [2](#0-1) 

2. The COA's EVM address is a smart contract wallet, not an EOA; it has no private key and cannot be controlled via `EVM.run`. [3](#0-2) 

3. There is no admin or protocol-level recovery function to drain an orphaned COA address. The `InternalEVM.withdraw` host function in `impl.go` is only reachable through the COA resource's `withdraw` method. [4](#0-3) 

The root cause is identical in structure to the SpiceAuction bug: tokens are deposited into a controlled context (COA / auction contract), a terminal condition occurs (COA destroyed / auction ends with no bids), and no recovery path exists because the only withdrawal function is gated on a condition that can no longer be satisfied.

### Impact Explanation
Any FLOW tokens held in the EVM environment at the time of COA destruction are permanently locked. They remain counted in the EVM total supply but are unreachable from both Cadence and EVM. This constitutes a cross-VM asset loss: tokens bridged from Cadence into the EVM environment via a COA are irrecoverably burned whenever the controlling resource is destroyed without a prior `withdraw`. The magnitude scales with the balance held at destruction time.

### Likelihood Explanation
The entry path is reachable by any unprivileged transaction sender who owns a COA. The pattern `destroy cadenceOwnedAccount` appears in test code throughout the repository (e.g., `fvm/evm/evm_test.go` lines 2373, 2445, 2517) and is the natural way to clean up a COA resource. A developer or user who deposits tokens and then destroys the COA — whether by mistake, in a cleanup path, or in a contract that creates temporary COAs — will silently lose the balance. No warning, no panic, and no recovery path exists. [5](#0-4) 

### Recommendation
Add a `destroy` function (destructor) to the `CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` that enforces one of the following:

1. **Panic on non-zero balance** — prevent destruction if `self.balance().attoflow > 0`, forcing the caller to withdraw first.
2. **Auto-withdraw** — automatically call `InternalEVM.withdraw` for the full balance and deposit it to a designated Cadence vault before the resource is destroyed.

Option 1 is the safer choice because it makes the invariant explicit and avoids the complexity of determining a destination vault at destruction time.

### Proof of Concept
The following Cadence transaction demonstrates the locked-token condition reachable by any unprivileged sender:

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Mint 1.0 FLOW for demonstration
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 1.0)
        let vault <- minter.mintTokens(amount: 1.0)
        destroy minter

        // Step 1: create a COA and deposit 1 FLOW into EVM
        let coa <- EVM.createCadenceOwnedAccount()
        let coaEVMAddress = coa.address()   // record address for verification
        coa.deposit(from: <-vault)

        // Step 2: confirm balance is 1 FLOW in EVM
        assert(coa.balance().inFLOW() == 1.0, message: "balance should be 1 FLOW")

        // Step 3: destroy the COA resource WITHOUT withdrawing
        // No destructor fires; no panic; no cleanup.
        destroy coa

        // Step 4: the 1 FLOW at coaEVMAddress is now permanently locked.
        // - withdraw() requires the COA resource (Owner | Withdraw entitlement) — gone.
        // - coaEVMAddress is a smart contract wallet with no private key — unreachable via EVM.run.
        // - No admin recovery function exists in the EVM contract.
    }
}
```

func newInternalEVMTypeWithdrawFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeWithdrawFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from address

			fromAddressValue, ok := invocation.Arguments[0].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			fromAddress, err := AddressBytesArrayValueToEVMAddress(context, fromAddressValue)
			if err != nil {
				panic(err)
			}

			// Get amount

			amountValue, ok := invocation.Arguments[1].(interpreter.UIntValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			_, overflow := uint256.FromBig(amountValue.BigInt)
			if overflow {
				panic(types.ErrInvalidBalance)
			}

			// check balance is not prone to rounding error
			if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
				panic(types.ErrWithdrawBalanceRounding)
			}

			// this is where rounding from Atto scale to UFix scale happens.
			value := new(big.Int).Div(amountValue.BigInt, types.UFixToAttoConversionMultiplier)
			amount := types.NewBalanceFromUFix64(cadence.UFix64(value.Uint64()))

			// Withdraw

			const isAuthorized = true
			account := handler.AccountByAddress(fromAddress, isAuthorized)
			vault := account.Withdraw(amount)

			ufix, roundedOff, err := types.ConvertBalanceToUFix64(vault.Balance())
			if err != nil {
				panic(err)
			}
			// We have already truncated the remainder above, but we still leave
			// the rounding check in as a redundancy.
			if roundedOff {
				panic(types.ErrWithdrawBalanceRounding)
			}
