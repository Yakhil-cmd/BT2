### Title
Unrestricted `update_rewards` Accepts User-Controlled `staker_address`, Allowing Any Staker to Steal Block Rewards from the Legitimate Block Proposer — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` is a public ABI function that accepts a fully user-controlled `staker_address` parameter and has no access control beyond basic prerequisites (unpaused + non-zero caller). Any registered staker can call it with their own address to claim the block rewards for the current block, while simultaneously locking out the legitimate block proposer via the `last_reward_block` guard.

---

### Finding Description

`update_rewards` is defined at `src/staking/staking.cairo` lines 1449–1507:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only: unpaused + caller != 0
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // locks out all other callers this block
    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    let (strk_block_rewards, btc_block_rewards) = self
        .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
    self._update_rewards(
        :staker_address,                                  // attacker-supplied address
        strk_total_rewards: strk_block_rewards,
        ...
        strk_total_stake: staker_total_strk_balance,      // attacker's own balance
        ...
    );
}
```

`general_prerequisites` (lines 1794–1797) only enforces:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

There is **no check** that the caller is the legitimate block proposer, nor any check that `staker_address` corresponds to the caller. The `last_reward_block` guard (lines 1454–1458) prevents a second call in the same block, but it does not prevent the *wrong* staker from being the first caller.

In V3 (consensus rewards), `_update_rewards` distributes `strk_block_rewards` proportionally among the supplied staker's own balance and their pool. The attacker therefore receives the full block reward allocation that should have gone to the legitimate proposer. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

The block rewards for every block in the consensus-rewards phase are a fixed mint allocated to the legitimate block proposer. An attacker who calls `update_rewards(attacker_staker_address, false)` before the legitimate proposer:

1. Receives the block rewards for that block (credited to `attacker_staker_info.unclaimed_rewards_own` and forwarded to the attacker's pool).
2. Writes `last_reward_block = current_block`, making the legitimate proposer's subsequent call revert with `REWARDS_ALREADY_UPDATED`.
3. The legitimate proposer earns **zero** rewards for that block.

This is a direct, repeatable theft of yield that belongs to another participant. [4](#0-3) 

---

### Likelihood Explanation

**High.** The attacker only needs to be a registered staker (no privileged role required). The attack requires a single transaction per block and can be automated. On Starknet, transaction ordering within a block is controlled by the sequencer, but the function is fully public and the attacker can submit the transaction at any time. There is no economic barrier beyond the gas cost of the call. [5](#0-4) 

---

### Recommendation

Remove the `staker_address` parameter from `update_rewards`. Instead, derive the staker address from the caller's operational address (mirroring how `attest` works in the Attestation contract), so only the legitimate block proposer can trigger their own reward update:

```cairo
fn update_rewards(ref self: ContractState, disable_rewards: bool) {
    self.general_prerequisites();
    let operational_address = get_caller_address();
    let staker_address = self.get_staker_address_by_operational(:operational_address);
    ...
}
```

This ensures the caller can only update rewards for themselves, eliminating the user-controlled input that enables the attack. [6](#0-5) 

---

### Proof of Concept

**Setup**: Staker A is the legitimate block proposer for block N. Attacker (Staker B) is also a registered staker.

1. Block N is produced. Staker A prepares to call `update_rewards(staker_A, false)`.
2. Attacker B calls `update_rewards(staker_B_address, false)` first (or in the same block before A).
3. Inside `update_rewards`:
   - `current_block_number > last_reward_block` → passes.
   - `staker_B` is active with non-zero balance → passes.
   - `last_reward_block` is written to block N.
   - `_update_rewards` is called with `staker_address = staker_B`, distributing the full block rewards to Staker B.
4. Staker A calls `update_rewards(staker_A, false)` in block N → **reverts** with `REWARDS_ALREADY_UPDATED`.
5. Staker A receives **zero** rewards for block N. Staker B receives rewards they were not entitled to.

This attack can be repeated every block, continuously siphoning block rewards away from legitimate proposers. [7](#0-6) [2](#0-1)
