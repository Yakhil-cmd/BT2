### Title
Gateway L2 Gas Price Admission Check Uses Previous Block Price Instead of Next Block Price - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` validates a transaction's L2 gas `max_price_per_unit` against the **current block's** L2 gas price instead of the **next block's** pre-computed L2 gas price (`next_l2_gas_price`). The developer explicitly acknowledged this with a TODO comment. The result is that the gateway admits transactions that will fail during execution when gas prices are rising, and incorrectly rejects valid transactions when gas prices are falling.

### Finding Description

In `validate_resource_bounds`, the gateway reads `gas_prices.strk_gas_prices.l2_gas_price` from the latest committed block and uses it as the reference for the admission threshold:

```rust
async fn validate_resource_bounds(
    &self,
    executable_tx: &ExecutableTransaction,
) -> StatefulTransactionValidatorResult<()> {
    if self.config.validate_resource_bounds {
        // TODO(Arni): getnext_l2_gas_price from the block header.
        let previous_block_l2_gas_price = self
            .gateway_fixed_block_state_reader
            .get_block_info()
            .await?
            .gas_prices
            .strk_gas_prices
            .l2_gas_price;
        self.validate_tx_l2_gas_price_within_threshold(
            executable_tx.resource_bounds(),
            previous_block_l2_gas_price,
        )?;
    }
    Ok(())
}
``` [1](#0-0) 

The threshold is then computed as:

```rust
let threshold = (gas_price_threshold_multiplier * previous_block_l2_gas_price.get().0).to_integer();
if tx_l2_gas_price.0 < threshold { return Err(...); }
``` [2](#0-1) 

The transaction will be included in the **next** block (N+1), whose L2 gas price is determined by the EIP-1559 fee market mechanism and is pre-computed and stored in the block header as `next_l2_gas_price`: [3](#0-2) 

The `calculate_next_base_gas_price` function adjusts the price up or down based on gas usage relative to the gas target, meaning the next block's price routinely differs from the current block's price: [4](#0-3) 

Compounding the issue, `run_validate_entry_point` also builds its block context from the current block's gas prices (only incrementing the block number), so the blockifier's `check_fee_bounds` inside gateway validation also uses the wrong gas prices: [5](#0-4) 

This means both the admission check and the gateway-side blockifier validation are anchored to the wrong price. The correct price (`next_l2_gas_price`) is already available in the block header but is not read.

The analog to the external report is exact: the external bug used the old state variable `amplification` instead of the new parameter `amplification_` in a calculation. Here, the code uses `l2_gas_price` (the current block's price, the "old" value) instead of `next_l2_gas_price` (the next block's price, the "new" value that will actually govern execution).

### Impact Explanation

**Impact: High** — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.

**Rising gas price (next > previous):** A transaction whose `max_price_per_unit` satisfies `threshold_wrong ≤ tx_price < threshold_correct` passes the gateway check but will fail `check_fee_bounds` during actual execution in the next block with `MaxGasPriceTooLow`. The gateway incorrectly admits transactions that are economically invalid for the block they will be sequenced into.

**Falling gas price (next < previous):** A transaction whose `max_price_per_unit` satisfies `threshold_correct ≤ tx_price < threshold_wrong` is valid for the next block but is incorrectly rejected at the gateway with `GAS_PRICE_TOO_LOW`. Legitimate user transactions are denied admission.

### Likelihood Explanation

**Likelihood: Medium.** The EIP-1559 fee market adjusts the L2 gas price every block based on gas consumption relative to the gas target. Any block that is not exactly at the gas target produces a `next_l2_gas_price` that differs from the current block's `l2_gas_price`. This is the normal operating condition, not an edge case. The developer's own TODO comment confirms the issue is known and unresolved.

### Recommendation

Replace the read of `l2_gas_price` with `next_l2_gas_price` from the block header in `validate_resource_bounds`:

```diff
-        // TODO(Arni): getnext_l2_gas_price from the block header.
-        let previous_block_l2_gas_price = self
+        let next_block_l2_gas_price = self
             .gateway_fixed_block_state_reader
             .get_block_info()
             .await?
-            .gas_prices
-            .strk_gas_prices
-            .l2_gas_price;
+            .next_l2_gas_price;  // use the pre-computed next-block price
         self.validate_tx_l2_gas_price_within_threshold(
             executable_tx.resource_bounds(),
-            previous_block_l2_gas_price,
+            next_block_l2_gas_price,
         )?;
```

`GatewayFixedBlockStateReader::get_block_info` should be extended to expose `next_l2_gas_price` from `StorageBlockHeader`. The same correction should be applied to the block context constructed in `run_validate_entry_point` so that the blockifier's `check_fee_bounds` also uses the correct next-block gas price.

### Proof of Concept

1. Observe the current L2 gas price in block N: `P_current`.
2. Observe that block N was heavily used, so `next_l2_gas_price = P_next > P_current`.
3. Submit a V3 `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P_current` (exactly at the threshold when `min_gas_price_percentage = 100`).
4. The gateway computes `threshold = 100% * P_current = P_current` and admits the transaction (`tx_price >= threshold`).
5. When the batcher executes the transaction in block N+1 with `l2_gas_price = P_next > P_current`, `check_fee_bounds` fires `MaxGasPriceTooLow` and the transaction fails.
6. Conversely, submit a transaction with `max_price_per_unit = P_next - 1` when gas price is falling (`P_next < P_current`). The gateway rejects it with `GAS_PRICE_TOO_LOW` even though the transaction would succeed in block N+1.

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
