### Title
Unvalidated `app_pk` Parameter in CKD Protocol Enables Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd()` function accepts an `app_pk` (TEE application public key) parameter that is completely caller-controlled and never validated against any broadcast or committed value. A single malicious participant can supply a different `app_pk` than the one agreed upon by honest parties, silently corrupting the aggregated `CKDOutput` so that the TEE cannot correctly decrypt it. All honest parties, including the coordinator, accept the corrupted output without error.

### Finding Description
In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` accepts `app_pk: PublicKey` as a caller-supplied argument. [1](#0-0) 

This value flows directly into `compute_signature_share()` where it is used as the ElGamal encryption key: [2](#0-1) 

The computation `big_c = big_s + app_pk * y` encrypts the participant's secret contribution `big_s` under `app_pk`. The protocol **never broadcasts `app_pk` among participants, never commits to it, and never validates that all participants are using the same value**. This is the direct analog to the auction bug: just as `params.endTime` was completely caller-controlled and not validated against the stored Seaport order hash, `app_pk` here is completely caller-controlled and not validated against any agreed-upon committed state.

The coordinator in `do_ckd_coordinator()` blindly aggregates all received `CKDOutput` values: [3](#0-2) 

There is no check that the received contributions were computed with the correct `app_pk`. A malicious participant `i` can call `ckd()` with an arbitrary `app_pk' ≠ app_pk`, causing their share to be encrypted under a different key. The coordinator accepts and aggregates this corrupted share without any error, outputting `Some(ckd_output)` as if the protocol succeeded.

### Impact Explanation
The coordinator outputs a `CKDOutput` where:

```
big_c = msk · H(pk, app_id)
      + Σ_{j≠i}(λ_j · y_j) · app_pk
      + λ_i · y_i · app_pk'
```

When the TEE decrypts using its secret key `app_sk`:

```
big_c − app_sk · big_y = msk · H(pk, app_id) + λ_i · y_i · (app_pk' − app_pk)
```

The result is not the correct confidential derived key `msk · H(pk, app_id)`. All honest parties accept this corrupted output as valid because there is no mechanism to detect the mismatch. This matches **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation
Any single malicious participant in the CKD protocol can trigger this attack by simply passing a different `app_pk` to the `ckd()` function. No special privileges, cryptographic capabilities, or colluding parties are required. The attack is trivially executable by any participant who controls their own software stack, making it a realistic threat in any deployment where participants are not fully trusted.

### Recommendation
The `app_pk` should be broadcast and validated as part of the CKD protocol, analogous to how session IDs are broadcast and validated in the DKG protocol: [4](#0-3) 

Specifically:
1. Each participant broadcasts their `app_pk` at the start of the CKD protocol.
2. All participants verify that all received `app_pk` values are identical to their own.
3. If any participant broadcasts a different `app_pk`, the protocol aborts with a `MaliciousParticipant` error, consistent with the existing error-handling pattern. [5](#0-4) 

### Proof of Concept
1. Honest participants agree on `app_pk = A` (the TEE's public key, `A = app_sk · G`).
2. Malicious participant `i` calls `ckd(..., app_pk = A', ...)` where `A' ≠ A`.
3. Participant `i` computes `big_c_i = big_s_i + A' · y_i` instead of `big_c_i = big_s_i + A · y_i`.
4. Participant `i` sends `(λ_i · big_y_i, λ_i · big_c_i)` to the coordinator via the private channel.
5. The coordinator aggregates without validation: `big_c = msk · H(pk, app_id) + Σ_{j≠i}(λ_j · y_j) · A + λ_i · y_i · A'`.
6. The TEE decrypts: `big_c − app_sk · big_y = msk · H(pk, app_id) + λ_i · y_i · (A' − A) ≠ msk · H(pk, app_id)`.
7. The coordinator returns `Ok(Some(ckd_output))` with no error; the confidential derived key is silently and permanently corrupted for this invocation. [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
