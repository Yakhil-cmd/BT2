### Title
Malicious Participant Can Corrupt CKD Output Without Cryptographic Verification — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly accumulates participant-supplied group elements with no proof of correct computation. Any single malicious participant can substitute arbitrary elliptic-curve points for their honest share, silently corrupting the confidential derived key that the coordinator returns as the protocol output.

---

### Finding Description

`do_ckd_coordinator` (lines 44–57) is the only party that receives the final `CKDOutput`. It first computes its own honest share via `compute_signature_share`, then loops over every other participant's private message and unconditionally adds the received `big_y` and `big_c` group elements to the running totals: [1](#0-0) 

There is no commitment, no zero-knowledge proof, and no broadcast-consistency check on the received values. The honest computation in `compute_signature_share` produces:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

A malicious participant is free to send any pair `(big_y', big_c')` of group elements instead. The coordinator has no mechanism to distinguish an honest contribution from a forged one.

The `ckd()` entry-point enforces only `participants.len() >= 2` and deduplication — it accepts no threshold parameter and performs no per-participant proof requirement: [3](#0-2) 

This is structurally analogous to the external report's finding: a role that is supposed to be constrained by a multi-party security guarantee (here, the honest-participant assumption) can unilaterally bypass that guarantee because no on-chain/in-protocol enforcement exists.

---

### Impact Explanation

**High — Corruption of CKD output.**

The final `CKDOutput` accepted by the coordinator will be:

```
big_y_final = Σ honest_i≠m (norm_big_y_i) + big_y'   (attacker-chosen)
big_c_final = Σ honest_i≠m (norm_big_c_i) + big_c'   (attacker-chosen)
```

When the application decrypts with `app_sk`:

```
confidential_key = big_c_final − app_sk · big_y_final
                 = msk · H(pk ‖ app_id) + (big_c' − app_sk · big_y')
```

The derived key is shifted by an attacker-controlled additive term. The coordinator and all downstream consumers accept this as the legitimate CKD result. No error is raised. This matches the allowed impact: *"Corruption of CKD outputs so honest parties accept inconsistent … unusable cryptographic outputs."*

---

### Likelihood Explanation

Any single participant in the protocol can trigger this. No special privilege, no key material beyond their own share, and no coordination with other parties is required. The attacker only needs to deviate from the honest message in the single private send to the coordinator: [4](#0-3) 

---

### Recommendation

1. **Commit-then-reveal**: Before sending `(norm_big_y, norm_big_c)`, each participant broadcasts a hash commitment. The coordinator verifies the opening before accumulating.
2. **Zero-knowledge proof of correct computation**: Each participant attaches a Schnorr-style proof that `norm_big_c_i` was formed using their actual signing share `x_i` and the committed randomness `y_i`, consistent with the public key and `app_id`.
3. **Exponent-interpolation consistency check** (analogous to the check already present in `do_presign` for the robust ECDSA path): broadcast

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
