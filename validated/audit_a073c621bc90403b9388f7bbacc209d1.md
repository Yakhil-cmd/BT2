### Title
Missing Threshold Validation Against Participant Count in CKD Protocol Allows Corrupted Key Derivation Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd()` function in the Confidential Key Derivation protocol only validates that `participants.len() >= 2`, but never validates that the participant count meets or exceeds the cryptographic threshold used during DKG. Because the threshold is not stored in `KeygenOutput`, a malicious coordinator or caller can invoke `ckd()` with a sub-threshold participant set, causing Lagrange interpolation to reconstruct an incorrect master secret contribution. The coordinator then aggregates these incorrect shares and outputs a corrupted confidential key that honest parties accept as valid.

### Finding Description

The `ckd()` entry point in `src/confidential_key_derivation/protocol.rs` performs the following input validation:

```rust
// not enough participants
if participants.len() < 2 {
    return Err(InitializationError::NotEnoughParticipants {
        participants: participants.len(),
    });
}
``` [1](#0-0) 

This is the only size check. There is no check that `participants.len() >= threshold`, where `threshold` is the reconstruction lower bound established during DKG.

The `KeygenOutput` struct passed to `ckd()` carries only `public_key` and `private_share`:

```rust
let key_pair = KeygenOutput {
    public_key: pk,
    private_share: private_shares[i],
};
``` [2](#0-1) 

No threshold field is present in `KeygenOutput`, so `ckd()` has no mechanism to recover or enforce the DKG threshold at all.

Inside `compute_signature_share()`, each participant computes a Lagrange coefficient over the supplied `participants` list and uses it to normalize their secret share contribution:

```rust
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [3](#0-2) 

The correctness of this Lagrange interpolation depends on the participant set being exactly the set for which the shares were generated (i.e., a set of size ≥ threshold). If the participant set is smaller than the DKG threshold, the Lagrange coefficients are computed over the wrong domain, and the sum `Σ λ_i · x_i` does not equal the master secret key `msk`. The coordinator then aggregates these incorrect contributions:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [4](#0-3) 

and returns a `CKDOutput` that does not correspond to the correct confidential key. There is no post-aggregation verification step that would detect this corruption.

Compare this to every other protocol entry point in the codebase, which does enforce the threshold:

- OT-based ECDSA `presign()`: `if args.threshold.value() > participants.len()` [5](#0-4) 
- OT-based ECDSA `sign()`: `if participants.len() < threshold` [6](#0-5) 
- FROST `assert_sign_inputs()`: `if threshold.value() > participants.len()` [7](#0-6) 
- Robust ECDSA `presign()` and `sign()`: enforce `participants.len() == 2*max_malicious+1` [8](#0-7) 

CKD is the only protocol that omits this check entirely.

### Impact Explanation

A sub-threshold CKD invocation produces a `CKDOutput` whose embedded `big_y` and `big_c` group elements do not encode the correct confidential key `msk · H(pk ‖ app_id)`. Any downstream consumer that calls `ckd_output.unmask(app_sk)` receives a wrong key silently. Honest participants have no way to detect the corruption because there is no verification step after aggregation. This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs**.

### Likelihood Explanation

The `ckd()` function is a public library API. A malicious coordinator controls which `participants` slice it passes to each honest party. By supplying a list shorter than the DKG threshold (but ≥ 2, to pass the only guard), the coordinator trivially triggers the corrupted output. No cryptographic break, key leakage, or external dependency failure is required. The attack is deterministic and requires only the ability to initiate a CKD session.

### Recommendation

Store the reconstruction threshold inside `KeygenOutput` for CKD (or accept it as an explicit parameter to `ckd()`), and add the missing guard before the protocol runs:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    threshold: ReconstructionLowerBound,   // <-- add this
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Add the missing threshold check (analog to the balance >= amount check)
    if participants.len() < threshold.value() {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold: threshold.value(),
            participants: participants.len(),
        });
    }
    // ... rest of validation
}
```

Alternatively, embed the threshold in `KeygenOutput` at DKG time so it is always available and cannot be omitted by callers.

### Proof of Concept

1. Run DKG with N=3 participants and threshold T=3 (all-or-nothing sharing). Each participant receives a share `x_i`.
2. A malicious coordinator initiates a CKD session passing only 2 participants (below T=3) to `ckd()`. The call passes the `participants.len() < 2` guard.
3. Each of the 2 honest participants computes `lambda_i` over the 2-participant set. These Lagrange coefficients are wrong for a degree-2 polynomial (T=3 requires 3 points).
4. The coordinator aggregates the 2 incorrect normalized shares and returns a `CKDOutput`.
5. The caller invokes `ckd_output.unmask(app_sk)` and receives a group element that is not `msk · H(pk ‖ app_id)`.
6. The honest parties have no way to detect the corruption; the protocol returns `Ok(Some(ckd_output))` with no error. [9](#0-8)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
```
