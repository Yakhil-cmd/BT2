### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Unclaimed Yield via Trace Inflation - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in the delegation pool contract contains an unbounded loop over a pool member's `pool_member_epoch_balance` trace. A pool member can inflate this trace by calling `add_to_delegation_pool` with a minimal amount each epoch. After sufficient inflation, any call to `claim_rewards` will exhaust the Starknet transaction gas limit, permanently freezing the pool member's accumulated unclaimed yield. The developers themselves acknowledge this in a code comment: *"This loop is unbounded but unlikely to exceed gas limits."*

---

### Finding Description

**Root cause — `calculate_rewards` in `src/pool/pool.cairo` lines 857–877:** [1](#0-0) 

The loop iterates from `entry_to_claim_from` to `pool_member_trace_length`, processing every balance-change checkpoint recorded since the last successful `claim_rewards`. There is no upper bound on how many iterations this loop can perform.

**How the trace grows — `set_member_balance` in `src/pool/pool.cairo` lines 718–729:** [2](#0-1) 

`set_member_balance` calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The underlying `insert` implementation appends a **new** checkpoint whenever the key (`current_epoch + K`) differs from the last stored key: [3](#0-2) 

Because the key advances by one each epoch, every call to `add_to_delegation_pool` (or `exit_delegation_pool_intent`) in a **different** epoch appends a distinct entry. Over N epochs this produces N trace entries.

**`add_to_delegation_pool` — the attacker-controlled entry path (`src/pool/pool.cairo` lines 221–254):** [4](#0-3) 

Any pool member (or their reward address) can call `add_to_delegation_pool` with `amount = 1` (the minimum non-zero value). Each call in a new epoch appends one entry to the trace. The only cost is 1 token unit per epoch plus gas.

**`claim_rewards` triggers the unbounded loop (`src/pool/pool.cairo` lines 335–377):** [5](#0-4) 

`claim_rewards` passes `entry_to_claim_from` (stored in `pool_member_info`) to `calculate_rewards`. This counter advances only after a **successful** claim. If the pool member never claims, the counter stays at 0 and the loop must traverse the entire trace on the next attempt.

---

### Impact Explanation

After accumulating a sufficiently large trace (achievable over many epochs), every call to `claim_rewards` for that pool member will exceed the Starknet transaction gas limit and revert. Because `entry_to_claim_from` is only updated on success, no partial progress is saved. The pool member's entire accumulated unclaimed yield becomes permanently unclaimable — matching the **"Permanent freezing of unclaimed yield"** (High) impact category.

---

### Likelihood Explanation

The attack is reachable by any unprivileged delegator. The cost is 1 token unit per epoch plus transaction gas. A realistic non-malicious scenario also exists: a long-term delegator who adds stake each epoch but rarely claims rewards will organically accumulate a large trace. The developers' own comment — *"This loop is unbounded but unlikely to exceed gas limits"* — confirms awareness of the risk without a concrete bound being enforced.

---

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` calls that would push the trace beyond a configurable cap (e.g., 1 000 entries).
2. **Alternatively, implement partial/paginated reward claiming.** Allow `claim_rewards` to process a bounded number of trace entries per call and save progress, so a large trace can be drained over multiple transactions.
3. **Remove or replace the developer comment** with an enforced invariant.

---

### Proof of Concept

```
Epoch 1:  pool_member calls enter_delegation_pool(amount=MIN_STAKE)
           → trace length = 1

Epoch 2:  pool_member calls add_to_delegation_pool(pool_member, 1)
           → trace.insert(key=2+K, ...) appends new entry → length = 2

Epoch 3:  pool_member calls add_to_delegation_pool(pool_member, 1)
           → trace.insert(key=3+K, ...) appends new entry → length = 3

... repeat for N epochs without ever calling claim_rewards ...

Epoch N+1: pool_member calls claim_rewards(pool_member)
            → calculate_rewards loops from entry_to_claim_from=0
              to pool_member_trace_length=N
            → each iteration: storage read (trace.at), find_sigma
              (up to 3 more storage reads), arithmetic
            → at sufficiently large N the transaction runs out of gas
            → revert; entry_to_claim_from unchanged
            → all subsequent claim_rewards calls also revert
            → unclaimed yield permanently frozen
```
