### Title
Gateway L2 Gas Price Admission Uses Stale Current-Block Price Instead of `next_l2_gas_price`, Admitting Transactions That Will Fail at Batcher Execution - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The gateway's stateful admission check compares a transaction's `l2_gas.max_price_per_unit` against the **current block's** `l2_gas_price`, but the transaction will actually be executed at the EIP-1559-adjusted `next_l2_gas_price` stored in the same block header. During periods of rising L2 gas prices, this discrepancy creates a window in which transactions pass all gateway checks but are then dropped by the batcher's `check_fee_bounds` pre-validation. The code itself acknowledges the bug with an explicit TODO: `// TODO(Arni): getnext_l2_gas_price from the block header.`

### Finding Description

`validate_resource_bounds` in `StatefulTransactionValidator` fetches the previous block's `BlockInfo` and extracts `gas_prices.strk_gas_prices.l2_gas_price` as the reference price for admission: [1](#0-0) 

This value is the L2 gas price **of the already-committed block**, not the EIP-1559-computed price for the **next** block. The block header stores both:

- `l2_gas_price` — the price used in the committed block
- `next_l2_gas_price` — the EIP-1559 adjustment that will govern the next block [2](#0-1) 

The threshold check then compares the transaction's `max_price_per_unit` against a fraction of this stale price: [3](#0-2) 

Critically, `run_validate_entry_point` also builds its `BlockContext` from the same stale `BlockInfo`, so the blockifier's `check_fee_bounds` inside the gateway also uses `prev_price`, not `next_price`: [4](#0-3) 

The batcher, however, builds its `BlockContext` with the actual next-block gas prices (derived from `next_l2_gas_price`). When it calls `perform_pre_validation_stage` → `check_fee_bounds`, it uses the real next-block price: [5](#0-4) 

The result is a two-tier mismatch:

| Stage | Price used | Outcome for tx with `prev_price ≤ max_price < next_price` |
|---|---|---|
| Gateway threshold check | `prev_price` | **PASS** |
| Gateway blockifier validation | `prev_price` | **PASS** |
| Mempool admission | — | **ADMITTED** |
| Batcher `check_fee_bounds` | `next_price` | **DROPPED** |

The EIP-1559 mechanism (`calculate_next_l2_gas_price_for_fin`) can increase the price by up to ~0.3% per block under the current `gas_target` constants, and the `fee_proposal_margin_ppt` allows the proposer to publish a price up to ~0.2% above `fee_actual`: [6](#0-5) 

During sustained high utilization, the cumulative drift between `l2_gas_price` and `next_l2_gas_price` grows monotonically, widening the admission window.

The symmetric case (falling prices) causes the gateway to **reject** transactions whose `max_price_per_unit` is above `next_price` but below `prev_price`, i.e., valid transactions are denied admission.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

- **Admits invalid**: An unprivileged sender can craft transactions with `max_price_per_unit` in the range `[prev_price, next_price)`. These pass every gateway check and enter the mempool, but are unconditionally dropped by the batcher's pre-validation. Repeated submission fills the mempool with permanently-unexecutable entries, degrading sequencer throughput.
- **Rejects valid**: When prices fall, transactions with `max_price_per_unit ∈ [next_price, prev_price)` are rejected at the gateway even though the batcher would accept them, causing unnecessary user-facing failures.

### Likelihood Explanation

The L2 gas price rises whenever block utilization exceeds `gas_target`. The sequencer's own EIP-1559 logic and the SNIP-35 oracle-driven `fee_proposal` mechanism are designed to push prices upward during congestion. Any period of sustained demand creates the rising-price condition. No special privilege is required; any account can submit a transaction.

### Recommendation

Replace the stale `l2_gas_price` lookup with `next_l2_gas_price` from the block header. The TODO comment already identifies the fix:

```rust
// In validate_resource_bounds:
// Instead of:
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;

// Use:
let next_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_header()   // expose next_l2_gas_price from BlockHeaderWithoutHash
    .await?
    .next_l2_gas_price;
```

The same correction must be applied to the `BlockContext` constructed inside `run_validate_entry_point` so that the blockifier's `check_fee_bounds` at the gateway also uses `next_l2_gas_price`.

### Proof of Concept

1. Observe the current block's `l2_gas_price` = P and `next_l2_gas_price` = P′ where P′ > P (network is congested).
2. Submit an `InvokeTransactionV3` with `l2_gas.max_price_per_unit = P` (exactly the previous block price, above the gateway threshold `min_gas_price_percentage * P / 100`).
3. The gateway's `validate_tx_l2_gas_price_within_threshold` passes because `P >= threshold`.
4. The gateway's blockifier validation (`run_validate_entry_point`) passes because it builds `BlockContext` with `l2_gas_price = P` and `check_fee_bounds` sees `max_price_per_unit (P) >= actual_gas_price (P)`.
5. The transaction is admitted to the mempool.
6. The batcher builds the next block with `l2_gas_price = P′`. Its `check_fee_bounds` sees `max_price_per_unit (P) < actual_gas_price (P′)` and raises `ResourceBoundsError::MaxGasPriceTooLow`, dropping the transaction.
7. Repeat at high frequency to exhaust mempool capacity with permanently-unexecutable transactions.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

/// Compute the fee_proposal an honest proposer should publish.
/// - If oracle failed (`fee_target` is `None`): freeze at `fee_actual`.
/// - Otherwise: clamp `fee_target` into the geometric bounds returned by `fee_proposal_bounds`.
pub fn compute_fee_proposal(
    fee_target: Option<GasPrice>,
    fee_actual: GasPrice,
    margin_ppt: u128,
) -> GasPrice {
    let Some(fee_target) = fee_target else {
        return fee_actual;
    };
    let (lower, upper) = fee_proposal_bounds(fee_actual, margin_ppt);
    GasPrice(fee_target.0.clamp(lower, upper))
}
