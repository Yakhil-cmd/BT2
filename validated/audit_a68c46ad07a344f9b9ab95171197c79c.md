### Title
FLOW tokens permanently locked when `CadenceOwnedAccount` resource is destroyed without withdrawing EVM balance - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
The `CadenceOwnedAccount` (COA) resource is the sole controller of its associated EVM address. The EVM address has no private key — it is derived from the COA resource's UUID. When a COA resource is destroyed without first withdrawing the EVM balance, the FLOW tokens held at that EVM address are permanently inaccessible. The COA resource defines no destructor (`destroy` function) that would automatically withdraw or redirect the balance before the resource is discarded, creating a direct analog to the burned-NFT reward-lock pattern in the reference report.

### Finding Description
In `fvm/evm/stdlib/contract.cdc`, the `CadenceOwnedAccount` resource stores `addressBytes` that map to a unique EVM address. The only way to move FLOW tokens out of that EVM address is through the COA's `withdraw` function, which is gated by `access(Owner | Withdraw)` — an entitlement that can only be held by the resource itself or a capability issued from it.

The `deposit` function is `access(all)`, meaning any party can send FLOW tokens into a COA's EVM address at any time. Once the COA resource is destroyed with `destroy coa`, the EVM address still exists in the EVM state trie with its full balance, but:

1. No private key exists for the address (it is a COA-type smart-contract wallet, not an EOA).
2. The `withdraw` entitlement is gone because the resource no longer exists.
3. No destructor runs to sweep the balance back to Cadence before destruction.

This is structurally identical to the reference vulnerability: a token/resource is destroyed, its associated on-chain value loses its only redemption path, and the assets are permanently locked.

### Impact Explanation
Any FLOW tokens held at a COA's EVM address at the moment the COA resource is destroyed are permanently burned from the perspective of the owner. Because `deposit` is public, a third party can also deposit tokens into a COA's EVM address at any time; if the COA is subsequently destroyed, those tokens are also permanently lost. The locked tokens remain visible in the EVM state but are unreachable by any transaction or script.

### Likelihood Explanation
Any unprivileged Cadence transaction can create a COA, fund it, and destroy it. The pattern `destroy cadenceOwnedAccount` without a preceding `withdraw` appears in the codebase's own test transactions (e.g., `fvm/evm/evm_test.go` lines 2373, 2445, 3264), demonstrating that the runtime permits and does not warn against this sequence. A user who migrates away from a COA, a contract that creates a temporary COA for a cross-VM operation, or any code path that destroys a COA after receiving an unexpected deposit will silently lock funds.

### Recommendation
Add a `destroy` function (destructor) to the `CadenceOwnedAccount` resource that sweeps the remaining EVM balance back to a Cadence vault before the resource is discarded, or alternatively add a precondition that panics if the balance is non-zero at destruction time, forcing callers to explicitly withdraw first.

```cadence
// Suggested destructor inside CadenceOwnedAccount
destroy() {
    let remaining = self.balance()
    if !remaining.isZero() {
        // withdraw remaining balance and deposit to a recovery vault,
        // or panic to force explicit withdrawal before destruction
        panic("EVM.CadenceOwnedAccount: cannot destroy a COA with a non-zero EVM balance")
    }
}
```

### Proof of Concept

The root cause is in `fvm/evm/stdlib/contract.cdc` at the `CadenceOwnedAccount` resource definition, which has no destructor: [1](#0-0) 

The `withdraw` function is the only egress path and requires an entitlement that ceases to exist once the resource is destroyed: [2](#0-1) 

The `deposit` function is unrestricted, allowing anyone to fund a COA before it is destroyed: [3](#0-2) 

The following transaction permanently locks 1.0 FLOW:

```cadence
import EVM from 0x<EVMContract>
import FlowToken from 0x<FlowToken>
import FungibleToken from 0x<FungibleToken>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        let vault <- account.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)!
            .withdraw(amount: 1.0) as! @FlowToken.Vault

        let coa <- EVM.createCadenceOwnedAccount()
        // EVM address is now live; deposit 1.0 FLOW into it
        coa.deposit(from: <-vault)

        // Destroy the COA without withdrawing.
        // The 1.0 FLOW at coa.address() is now permanently locked in EVM state.
        destroy coa
    }
}
```

The same pattern is observable in the existing test suite, confirming the runtime allows it without any warning or revert: [4](#0-3)
