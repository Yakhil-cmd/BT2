### Title
Blacklisted `adminFeeDestination` DOSes fee collection and blocks fee-rate updates for all three factory callers — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPool.collectFees()` performs unconditional `safeTransfer` calls to the stored `adminFeeDestination_`. Three factory-level functions — the permissionless `collectPoolFees`, the pool-admin-gated `setPoolAdminFees`, and the factory-owner-gated `setPoolProtocolFee` — all invoke `collectFees` as a mandatory first step before updating any state. If the pool's `adminFeeDestination` is blacklisted by the pool token (e.g., USDC), every one of those three entry points reverts, permanently blocking fee collection and freezing fee-rate configuration until the pool admin separately rotates the destination.

---

### Finding Description

`MetricOmmPool.collectFees` transfers the admin share of accrued fees directly to the caller-supplied `adminFeeDestination_` using `safeTransfer`: [1](#0-0) 

`safeTransfer` reverts on failure. There is no try-catch or fallback path.

The factory stores one `adminFeeDestination` per pool in `poolAdminFeeDestination[pool]` and passes it verbatim into every `collectFees` call. Three distinct factory entry points do this:

**1. `collectPoolFees` (permissionless)** [2](#0-1) 

**2. `setPoolAdminFees` (pool admin)** [3](#0-2) 

**3. `setPoolProtocolFee` (factory owner)** [4](#0-3) 

All three call `collectFees` **before** writing any new state. If the transfer to `adminFeeDestination_` reverts, the entire transaction reverts and no state is updated.

The only escape hatch is `setPoolAdminFeeDestination`, which updates the mapping without calling `collectFees`: [5](#0-4) 

However, this function is restricted to `onlyPoolAdmin(pool)`. The factory owner has no independent path to unblock `setPoolProtocolFee` — they must wait for the pool admin to rotate the destination first.

---

### Impact Explanation

When `adminFeeDestination` is blacklisted by the pool token:

- **`collectPoolFees` reverts** — all accrued spread and notional fees (both admin and protocol shares) are stuck inside the pool contract. The `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` accumulators are never cleared.
- **`setPoolAdminFees` reverts** — the pool admin cannot adjust fee rates. Fee configuration is frozen at its current values.
- **`setPoolProtocolFee` reverts** — the factory owner cannot update protocol fee rates for the affected pool. The factory owner has no independent path to bypass this; they are blocked until the pool admin acts.

Protocol fees accumulate in the pool but cannot be extracted. The factory owner's governance over per-pool protocol fees is held hostage by the pool admin's responsiveness.

---

### Likelihood Explanation

- USDC and USDT both implement token-level blacklisting and are the canonical examples of in-scope non-standard ERC-20 behavior per the contest rules.
- `adminFeeDestination` is set once by the pool admin and may point to a multisig, DAO treasury, or exchange deposit address — all realistic targets for USDC blacklisting.
- The blacklisting event can occur at any time after pool creation, independently of the pool admin's awareness.
- No attacker action is required; the trigger is an external USDC/USDT compliance action against the destination address.

---

### Recommendation

Wrap the admin-share transfer inside `collectFees` in a try-catch (or use a low-level call with a success check), and on failure, credit the admin's owed amount to a claimable balance mapping rather than reverting. Alternatively, separate the fee-collection step from the fee-rate-update step so that `setPoolAdminFees` and `setPoolProtocolFee` can proceed even when the current destination is unreachable.

```solidity
// Example: non-reverting admin transfer with fallback accumulation
if (totalFee0ToAdmin > 0) {
    try IERC20(TOKEN0).transfer(adminFeeDestination_, totalFee0ToAdmin) returns (bool ok) {
        if (!ok) pendingAdminFee0[adminFeeDestination_] += totalFee0ToAdmin;
    } catch {
        pendingAdminFee0[adminFeeDestination_] += totalFee0ToAdmin;
    }
}
```

---

### Proof of Concept

1. Deploy a pool with USDC as `token0`.
2. Pool admin sets `adminFeeDestination` to address `A`.
3. Swaps occur; spread and notional fees accumulate in the pool.
4. USDC blacklists address `A`.
5. Any caller invokes `collectPoolFees(pool)` → `collectFees` is called → `safeTransfer(A, fee0)` reverts → entire tx reverts. Fees remain stuck.
6. Factory owner calls `setPoolProtocolFee(pool, newFee, newFee)` → same `collectFees` call → same revert. Factory owner cannot update protocol fees.
7. Pool admin calls `setPoolAdminFees(pool, newFee, newFee)` → same revert. Admin cannot change fee rates.
8. Only recovery: pool admin calls `setPoolAdminFeeDestination(pool, newAddress)` to rotate the destination, after which steps 5–7 succeed.
