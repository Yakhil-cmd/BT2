### Title
EIP-2537 BLS12-381 Precompiles Reject Empty Input Instead of Returning Canonical Identity Values — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/pairing.rs`, `msm.rs`)

---

### Summary

The ZKsync OS EIP-2537 precompile implementations for `BLS12_PAIRING_CHECK`, `BLS12_G1MSM`, and `BLS12_G2MSM` unconditionally reject zero-length input with an `InvalidInputSize` error. The EIP-2537 specification mandates that each of these operations return a canonical identity value when given empty input. This is a direct EVM semantic mismatch: any smart contract that relies on the specified empty-input behavior will execute differently on ZKsync OS than on Ethereum mainnet, causing forward execution divergence and potential proving failures.

---

### Finding Description

EIP-2537 defines the following mandatory behaviors for empty input:

| Precompile | Empty-input required return value |
|---|---|
| `BLS12_G1MSM` | Point at infinity (identity) |
| `BLS12_G2MSM` | Point at infinity (identity) |
| `BLS12_PAIRING_CHECK` | `true` (1) |

In ZKsync OS, all three precompiles instead return an error:

**`pairing.rs`** — `Bls12381PairingCheckPrecompile::execute`:
```rust
if input.len() == 0 {
    return Err(interface_error!(
        Bls12PrecompileInterfaceError::InvalidInputSize
    ));
}
``` [1](#0-0) 

**`msm.rs`** — `Bls12381G1MSMPrecompile::execute`:
```rust
if input.len() == 0 {
    return Err(interface_error!(
        Bls12PrecompileInterfaceError::InvalidInputSize
    ));
}
``` [2](#0-1) 

**`msm.rs`** — `Bls12381G2MSMPrecompile::execute`:
```rust
if input.len() == 0 {
    return Err(interface_error!(
        Bls12PrecompileInterfaceError::InvalidInputSize
    ));
}
``` [3](#0-2) 

By contrast, the `parse_g1_with_subgroup_check` and `parse_g2_with_subgroup_check` helpers used by MSM and pairing are correctly implemented with subgroup validation: [4](#0-3) 

The on-curve parsing layer (`parse_g1_bytes`, `parse_g2_bytes`) is also correct: [5](#0-4) 

The defect is exclusively in the empty-input guard at the top of each precompile's `execute` function.

---

### Impact Explanation

**Vulnerability class:** EVM semantic mismatch / forward execution divergence.

Any smart contract deployed on ZKsync OS that calls `BLS12_PAIRING_CHECK`, `BLS12_G1MSM`, or `BLS12_G2MSM` with empty calldata will receive a revert, whereas the identical call on Ethereum mainnet succeeds and returns the identity value. Concrete consequences:

1. **BLS signature aggregation contracts** that use the empty-pairing-returns-true shortcut (a common optimization in BLS12-381 libraries) will have their verification logic silently broken.
2. **MSM-based protocols** that treat an empty point set as the identity will revert instead of continuing.
3. **Forward/proving divergence**: the forward execution engine records a revert; the prover, which must faithfully reproduce EVM semantics, would need to produce a proof for a different outcome than what Ethereum would produce. This breaks the EVM-equivalence guarantee of the rollup.

---

### Likelihood Explanation

Any unprivileged transaction sender can trigger this by calling address `0x0f` (BLS12_PAIRING_CHECK) with zero-length calldata. No special privileges, leaked keys, or governance access are required. The call is cheap (only the fixed gas cost of 37,700 gas is charged before the empty-input check fires). Smart contracts that implement BLS12-381 signature verification or aggregation are the primary realistic callers. [6](#0-5) 

---

### Recommendation

Replace the unconditional empty-input rejection in each affected precompile with the EIP-2537-mandated identity return:

**`pairing.rs`**: Remove the `input.len() == 0` early-return error. The subsequent `is_multiple_of` check already handles non-multiple-of-pair-length inputs. When `num_pairs == 0`, `multi_pairing` over empty iterators returns the identity of the target field (GT), which equals `ONE`, so the existing logic at line 74 will correctly write `1` to output.

**`msm.rs` (G1MSM and G2MSM)**: Remove the `input.len() == 0` early-return error. When `num_pairs == 0`, the MSM loop produces no points and the result is the group identity (point at infinity), which `write_g1`/`write_g2` will correctly serialize as all-zeros.

### Proof of Concept

```solidity

contract TestBLS12EmptyPairing {
    address constant BLS12_PAIRING = address(0x0f);

    function testEmptyPairing() external view returns (bool success, bytes memory result) {
        // EIP-2537: empty input MUST return abi.encode(true)
        (success, result) = BLS12_PAIRING.staticcall("");
        // On Ethereum mainnet: success == true, result == abi.encode(1)
        // On ZKsync OS:        success == false (revert) — DIVERGENCE
    }
}
```

On Ethereum mainnet the call succeeds and returns `0x0000...0001`. On ZKsync OS the call reverts due to the `InvalidInputSize` guard at `pairing.rs:26-30`, producing a different state transition and breaking EVM equivalence. [7](#0-6)

pub fn parse_g1_bytes(input: &[u8; G1_LEN]) -> Option<(G1Affine, bool)> {
    if input.iter().all(|el| *el == 0) {
        return Some((G1Affine::identity(), false));
    }
    let x = parse_fq_bytes(input[0..64].try_into().ok()?)?;
    let y = parse_fq_bytes(input[64..128].try_into().ok()?)?;
    let point = G1Affine::new_unchecked(x, y);

    if !point.is_on_curve() {
        return None;
    }

    Some((point, true))
}

#[inline(never)]
pub fn parse_g2_bytes(input: &[u8; G2_LEN]) -> Option<(G2Affine, bool)> {
    if input.iter().all(|el| *el == 0) {
        return Some((G2Affine::identity(), false));
    }
    let x = parse_fq2_bytes(input[0..128].try_into().ok()?)?;
    let y = parse_fq2_bytes(input[128..256].try_into().ok()?)?;
    let point = G2Affine::new_unchecked(x, y);

    if !point.is_on_curve() {
        return None;
    }

    Some((point, true))
}
