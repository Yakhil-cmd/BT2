### Title
Unauthenticated Sender in Echo Broadcast Aborts DKG/Reshare/Refresh via Unguarded `?` on `participants.index(from)` — (`File: src/protocol/echo_broadcast.rs`)

---

### Summary

In `reliable_broadcast_receive_all`, when a `MessageType::Send` message arrives from a `Participant` not present in the protocol's participant list, the expression `participants.index(from)?` at line 199 propagates `Err(ProtocolError::InvalidIndex)` via the `?` operator, immediately aborting the entire broadcast function. Because `do_broadcast` is called during every round of DKG, reshare, and refresh, a single injected message from any non-participant identity is sufficient to permanently deny key generation for all honest parties in that session.

---

### Finding Description

`reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` processes three message variants: `Send`, `Echo`, and `Ready`. The `Echo` and `Ready` arms handle unknown senders safely:

```rust
// Echo arm — safe: returns false for unknown participant, loop continues
if !state_sid.seen_echo.put(from) || state_sid.finish_echo {
    continue;
}
```

```rust
// Ready arm — safe: same pattern
if !state_sid.seen_ready.put(from) || state_sid.finish_ready {
    continue;
}
```

The `Send` arm is inconsistent:

```rust
// Send arm — UNSAFE: ? propagates Err(InvalidIndex) for unknown participant
if state_sid.finish_send || sid != participants.index(from)? {
    continue;
}
```

`participants.index(from)` returns `Err(ProtocolError::InvalidIndex)` when `from` is absent from the participant list. The `?` operator propagates this error out of the async function rather than treating it as a reason to `continue`.

The public entry point is `Protocol::message` in `src/protocol/internal.rs`:

```rust
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
```

`push_message` performs no participant-membership check — it routes any message with a parseable `MessageHeader` into the incoming buffer. The developers explicitly acknowledge this in the test at line 532 of `internal.rs`:

```rust
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    // Attacker injects messages for waitpoints the honest code never polls.
    comms.push_message(attacker, message);
}
```

The channel tag for the shared channel (`ChannelTag::root_shared()`) is a fixed, deterministic value derived from a public domain separator. The waitpoint sequence is also deterministic. An attacker can therefore craft a message with the correct header, a valid `sid` value (any integer in `0..n`), and a `MessageType::Send` payload, then deliver it via `Protocol::message` using any `Participant` ID not in the participant list.

`do_broadcast` in `echo_broadcast.rs` calls `reliable_broadcast_receive_all` directly:

```rust
pub async fn do_broadcast<'a, T>(
    chan: &mut SharedChannel,
    participants: &'a ParticipantList,
    me: Participant,
    data: T,
) -> Result<ParticipantMap<'a, T>, ProtocolError>
```

`do_keyshare` in `src/dkg.rs` calls `do_broadcast` three times — for session IDs, for commitments and proofs of knowledge, and for the final success vote. An abort at any of these points terminates the DKG session for the receiving party.

---

### Impact Explanation

**High — Permanent denial of key generation, reshare, and refresh for honest parties.**

A single crafted `MessageType::Send` message delivered to any honest participant's `Protocol::message` endpoint causes `reliable_broadcast_receive_all` to return `Err(ProtocolError::InvalidIndex)`. This propagates through `do_broadcast` → `do_keyshare` → `do_keygen` / `do_reshare`, aborting the session for that participant. Because the attacker can repeat the injection on every restart, the protocol can be denied indefinitely. All three broadcast rounds in `do_keyshare` are vulnerable.

---

### Likelihood Explanation

The `Protocol::message` function is the library's public message-delivery API and imposes no participant-membership guard. The channel tag and waitpoint sequence are fully deterministic and publicly derivable. Any party that can deliver a single network message to an honest participant — including a non-participant observer — can trigger the abort. No cryptographic material or insider access is required.

---

### Recommendation

Replace the `?` propagation with a safe fallback that treats an unknown sender as a reason to skip, consistent with the `Echo` and `Ready` arms:

```rust
// Before (unsafe):
if state_sid.finish_send || sid != participants.index(from)? {
    continue;
}

// After (safe):
let Ok(from_idx) = participants.index(from) else {
    continue;
};
if state_sid.finish_send || sid != from_idx {
    continue;
}
```

Additionally, consider adding a membership check at the top of the receive loop (before the `match`) to reject all messages from non-participants in a single place, rather than relying on per-arm handling.

---

### Proof of Concept

1. An honest set of participants `{P0, P1, P2}` begins DKG. `do_keyshare` calls `do_broadcast` for the session-ID round, allocating waitpoint `W`.
2. The attacker, holding identity `A` (not in `{P0, P1, P2}`), constructs a raw message:
   - Header bytes: `ChannelTag::root_shared()` tag (fixed, public) + waitpoint `W` (deterministic).
   - Payload: msgpack-encoded `(sid=0, MessageType::Send(some_data))`.
3. The attacker calls `P0.protocol.message(A, crafted_bytes)`.
4. `push_message` parses the header and enqueues the message for waitpoint `W`.
5. `reliable_broadcast_receive_all` dequeues the message: `from = A`, `sid = 0`.
6. `state.get_mut(0)` succeeds (valid index).
7. The `Send` arm executes: `participants.index(A)` → `Err(InvalidIndex)` → `?` propagates.
8. `reliable_broadcast_receive_all` returns `Err(InvalidIndex)` to `do_broadcast`, which returns it to `do_keyshare`, which returns it to `do_keygen`.
