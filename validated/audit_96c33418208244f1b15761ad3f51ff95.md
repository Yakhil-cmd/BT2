### Title
`max_l2_gas_amount` cap not enforced for `Declare` transactions in `StatelessTransactionValidator::validate_resource_bounds` — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The `StatelessTransactionValidator` has a configured `max_l2_gas_amount` cap that is enforced for `Invoke` and `DeployAccount` transactions but is explicitly skipped for `Declare` transactions. This allows a `Declare` transaction to enter the mempool with an `l2_gas.max_amount` value exceeding the configured cap — the exact analog of the `OmoVault` `supplyCap` not being enforced in `deposit()`/`mint()`.

---

### Finding Description

In `validate_resource_bounds`, the upper-bound check on `l2_gas.max_amount` is guarded by an empty `if let` branch that silently skips the check for `Declare` transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The `StatelessTransactionValidatorConfig` defines `max_l2_gas_amount: 1_210_000_000` as the production default: [2](#0-1) 

This cap is enforced for `Invoke` and `DeployAccount` but the `Declare` arm is an empty block — a deliberate bypass with a TODO acknowledging the gap. The test `valid_l2_gas_amount_on_declare` explicitly confirms that a `Declare` transaction with `l2_gas.max_amount = GasAmount(200)` passes when `max_l2_gas_amount = 100`: [3](#0-2) 

**No downstream check restores the invariant.** The blockifier's `check_fee_bounds` in `perform_pre_validation_stage` only enforces a *lower* bound (minimum gas required), not an upper bound: [4](#0-3) 

`verify_can_pay_committed_bounds` checks only that the sender's balance covers `max_amount × max_price_per_unit` — a balance check, not a gas-amount cap: [5](#0-4) 

The bouncer checks actual gas consumed, not the declared `max_amount`: [6](#0-5) 

**Attack path:** An attacker submits a `Declare` transaction with `l2_gas.max_amount` set to, e.g., `5_000_000_000` (equal to the full block `sierra_gas` capacity) and `l2_gas.max_price_per_unit = min_gas_price = 8_000_000_000`. The required balance is `5×10⁹ × 8×10⁹ = 4×10¹⁹` wei of STRK — achievable for a well-funded account. The transaction:

1. Passes `validate_resource_bounds` (Declare branch is empty).
2. Passes `verify_can_pay_committed_bounds` (balance covers the declared fee).
3. Enters the mempool with a declared L2 gas claim that equals the entire block's sierra gas budget.
4. Displaces or delays other transactions in mempool ordering (which is fee-based, derived from `max_amount × max_price_per_unit`).

The production config confirms `max_l2_gas_amount = 1_210_000_000` is the intended ceiling: [7](#0-6) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A `Declare` transaction with `l2_gas.max_amount > max_l2_gas_amount` is admitted to the mempool in violation of the gateway's own admission invariant. A well-funded attacker can submit `Declare` transactions whose declared L2 gas claim equals or exceeds the block's full `sierra_gas` capacity (`5_000_000_000`), inflating their apparent fee priority and crowding out legitimate transactions in mempool ordering.

---

### Likelihood Explanation

**High.** The bypass is unconditional for all `Declare` transactions — no special privilege is required. The only practical constraint is that the sender must hold enough STRK to satisfy `verify_can_pay_committed_bounds`. The TODO comment in the source confirms the developers are aware the check is absent. The test `valid_l2_gas_amount_on_declare` explicitly documents and exercises the bypass path.

---

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` guard and apply the same `max_l2_gas_amount` upper-bound check to `Declare` transactions, as the TODO comment already anticipates:

```rust
// Apply to all transaction types, including Declare.
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher L2 gas ceiling (e.g., for large Sierra programs), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the cap entirely.

---

### Proof of Concept

Using the existing test harness in `stateless_transaction_validator_test.rs`:

```rust
#[test]
fn declare_bypasses_max_l2_gas_amount_cap() {
    let config = StatelessTransactionValidatorConfig {
        validate_resource_bounds: true,
        max_l2_gas_amount: 100,          // cap = 100
        min_gas_price: 1,
        ..*DEFAULT_VALIDATOR_CONFIG_FOR_TESTING
    };
    let tx_validator = StatelessTransactionValidator { config };

    // l2_gas.max_amount = 200 >> cap of 100
    let rpc_tx_args = RpcTransactionArgs {
        resource_bounds: AllResourceBounds {
            l2_gas: ResourceBounds {
                max_amount: GasAmount(200),
                max_price_per_unit: GasPrice(1),
            },
            ..Default::default()
        },
        ..Default::default()
    };

    // Invoke/DeployAccount would be rejected; Declare is accepted — cap not enforced.
    let declare_tx = rpc_tx_for_testing(TransactionType::Declare, rpc_tx_args.clone());
    assert_matches!(tx_validator.validate(&declare_tx), Ok(()));   // BUG: should be Err

    let invoke_tx = rpc_tx_for_testing(TransactionType::Invoke, rpc_tx_args);
    assert_matches!(
        tx_validator.validate(&invoke_tx),
        Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { .. })
    );
}

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
