### Title
Malicious Participant Can Corrupt CKD Output by Injecting Arbitrary Group Elements Without Validation — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator accumulates `(big_y, big_c)` group-element contributions from every participant with no validity or consistency checks. A malicious participant can substitute arbitrary BLS12-381 G1 elements — including the identity — for their honest share, silently biasing the final ElGamal-encrypted derived key. The coordinator returns the corrupted `CKDOutput` as a successful result, and no honest party can detect the manipulation.

---

### Finding Description

`do_ckd_coordinator` (lines 35–57) collects each participant's `(norm_big_y, norm_big_c)` and adds them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no check that:
- `big_y` or `big_c` are non-identity elements,
- `big_y` is consistent with the participant's claimed randomness (`y_i · G`),
- `big_c` is consistent with the participant's key share (`x_i · H(pk, app_id) + y_i · app_pk`).

This contrasts sharply with the ECDSA presign protocols, which enforce consistency via exponent-interpolation checks: [2](#0-1) 

and scalar-commitment checks: [3](#0-2) 

The CKD protocol has no equivalent guard. The `ckd()` entry point also accepts no threshold parameter, so even a single malicious participant among N can corrupt the output. [4](#0-3) 

---

### Impact Explanation

The CKD output `(Y, C)` is an ElGamal encryption of the confidential derived key `msk · H(pk, app_id)`:

```
Y = Σ λᵢ · yᵢ · G
C = Σ λᵢ · (xᵢ · H(pk, app_id) + yᵢ · app_pk)
``` [5](#0-4) 

If a malicious participant substitutes `(Y', C')` for their honest contribution, the coordinator computes:

```
Y_final = Y_honest + Y'
C_final = C_honest + C'
```

Decryption yields `C_final − app_sk · Y_final = msk · H(pk, app_id) + (C' − app_sk · Y')`, which is an attacker-controlled offset from the correct derived key. The coordinator returns this as a valid `CKDOutput` with no error. This is a **High** impact: corruption of CKD outputs so honest parties accept an unusable or attacker-biased confidential derived key.

---

### Likelihood Explanation

Any participant in the CKD session can trigger this. The attacker needs only to be admitted to the participant list and deviate from the protocol by sending arbitrary G1 elements. No cryptographic break, key leakage, or external assumption is required. The CKD protocol is TEE-focused, but a compromised enclave or a rogue node admitted to the participant set is a realistic adversary in that deployment context.

---

### Recommendation

1. **Identity-element rejection**: Before accumulating, assert `big_y != G1::identity()` and `big_c != G1::identity()`.
2. **Consistency proof**: Require each participant to attach a Schnorr-style or sigma-protocol proof that `big_y = y_i · G` and `big_c = x_i · H(pk, app_id) + y_i · app_pk` for the same `y_i`, without revealing `y_i` or `x_i`. The existing `dlog` and `dlogeq` proof infrastructure in `src/crypto/proofs/` can be reused for this purpose.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
