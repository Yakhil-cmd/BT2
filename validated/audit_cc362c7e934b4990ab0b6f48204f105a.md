Audit Report

## Title
CMC `purge_old_notifications` Evicts `Processing` Entries, Permanently Locking User ICP - (File: `rs/nns/cmc/src/main.rs`)

## Summary

`purge_old_notifications` unconditionally removes the oldest entries from `blocks_notified` without checking whether their status is `Processing` (in-flight async call). When such an entry is evicted, `last_purged_notification` advances past that block index. If the in-flight call later returns a transient error, the entry is deleted and the user's retry is permanently rejected with `TransactionTooOld`, leaving their ICP locked in the CMC's subaccount with no on-chain recovery path.

## Finding Description

`purge_old_notifications` at [1](#0-0)  removes entries from `blocks_notified` purely by insertion order, with no guard on `NotificationStatus`. The `Processing` variant at [2](#0-1)  represents an in-flight inter-canister call; the canister has yielded execution and other messages can run.

The exploit path in `notify_top_up`:

1. User A calls `notify_top_up(X)`. `blocks_notified[X]` is set to `Processing` and the canister awaits `process_top_up`. [3](#0-2) 
2. While awaiting, User B (or an adversary) calls `notify_top_up(Y)`. `purge_old_notifications` fires because `len > MAX_NOTIFY_HISTORY`, removes block index `X` (oldest entry), and sets `last_purged_notification = X`. [4](#0-3) 
3. User A's async call returns a transient error. The post-await handler re-inserts then immediately removes `X`. [5](#0-4) 
4. User A retries `notify_top_up(X)`. The guard fires: `block_index X <= last_purged_notification` → `TransactionTooOld` returned permanently. [6](#0-5) 

The identical pattern exists in `notify_mint_cycles` and `notify_create_canister`. [7](#0-6) [8](#0-7) 

The ICP was transferred to the CMC's subaccount before `notify_top_up` was called. The CMC exposes no function to return ICP from a subaccount to its owner, and the automatic-refund path (`NotMeaningfulMemo`) only applies to transactions with non-meaningful memos.

## Impact Explanation

Permanent, irrecoverable loss of user ICP locked in the CMC's subaccount. The ICP is neither minted into cycles nor returned to the sender. This matches the allowed impact: **High** — unauthorized loss of canister-controlled funds where exploitation requires meaningful per-target work (filling the map and timing the purge). If the targeted transaction is large, the impact scales toward Critical (permanent loss of exorbitant ICP/Cycles assets).

## Likelihood Explanation

`MAX_NOTIFY_HISTORY = 1_000_000`. The CMC is a high-traffic system canister active since mainnet genesis; the map reaches capacity in steady-state operation. An adversary can observe a victim's in-flight `Processing` entry on-chain (block index is public), then submit enough ICP transfers to the CMC's subaccount (each costing only the ~0.0001 ICP transfer fee) to make the victim's entry the oldest and trigger its eviction. Once the map is at capacity, a single additional `notify_*` call suffices to evict the oldest entry. The attack is repeatable and low-cost relative to the ICP that can be locked.

## Recommendation

`purge_old_notifications` must skip `Processing` entries:

```rust
fn purge_old_notifications(&mut self, max_history: usize) {
    let mut last_purged = 0;
    let mut cnt = 0;
    while self.blocks_notified.len() > max_history && cnt < MAX_NOTIFY_PURGE {
        let block_height = *self.blocks_notified.iter().next().unwrap().0;
        if self.blocks_notified[&block_height] == NotificationStatus::Processing {
            break; // never evict in-flight entries
        }
        self.blocks_notified.remove(&block_height);
        last_purged = block_height;
        cnt += 1;
    }
    self.last_purged_notification = last_purged.max(self.last_purged_notification);
}
```

Additionally, `last_purged_notification` should only advance past block indices that were fully processed, not merely evicted, so that users whose entries were evicted mid-flight can still retry.

## Proof of Concept

```
Precondition: blocks_notified.len() == 1_000_000; smallest key = X.
blocks_notified[X] = Processing  (User A's notify_top_up, async call in-flight)

Step 1: Adversary calls notify_top_up(Y, Y > X) with a fresh ICP transfer.
        → purge_old_notifications: len > MAX_NOTIFY_HISTORY
        → blocks_notified.remove(X)
        → last_purged_notification = X

Step 2: User A's process_top_up returns Err(transient).
        → blocks_notified.insert(X, NotifiedTopUp(Err(transient)))
        → is_transient_error → blocks_notified.remove(X)

Step 3: User A retries notify_top_up(X).
        → block_index X <= last_purged_notification
        → returns Err(TransactionTooOld(X+1))

Result: User A's ICP is permanently locked in the CMC subaccount.
```

A deterministic integration test using PocketIC can reproduce this by: (1) pre-filling `blocks_notified` to `MAX_NOTIFY_HISTORY` entries, (2) injecting a `Processing` entry at the minimum key, (3) calling `purge_old_notifications` once, and (4) asserting that a subsequent `notify_top_up` for that block index returns `TransactionTooOld`.
