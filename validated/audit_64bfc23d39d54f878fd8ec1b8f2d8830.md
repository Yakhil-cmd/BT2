### Title
Secret Cryptographic Material Exposed via `#[derive(Debug)]` on Secret-Holding Types — (File: `src/ecdsa/ot_based_ecdsa/triples/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/lib.rs`)

---

### Summary

Multiple types containing secret cryptographic scalars unconditionally derive the `Debug` trait. Any library caller that formats these types with `{:?}` — through logging frameworks, error handling, or tracing — will emit the raw secret values in plaintext. The module documentation for `TripleShare` explicitly states that leaking the underlying triple scalars enables full private key recovery.

---

### Finding Description

The following production types derive `Debug` while holding secret scalar material:

**`TripleShare`** — `src/ecdsa/ot_based_ecdsa/triples/mod.rs:72–77`
```rust
#[derive(Clone, Debug, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct TripleShare {
    pub a: Scalar,
    pub b: Scalar,
    pub c: Scalar,
}
```
The module preamble at lines 12–13 explicitly warns:
> "It's important that the value of the underlying scalars in the triple is kept secret, otherwise the private key used to create a signature with that triple could be recovered."

Despite this, `Debug` is derived, so `format!("{:?}", triple_share)` prints `a`, `b`, and `c` in plaintext.

**`PresignOutput`** — `src/ecdsa/ot_based_ecdsa/mod.rs:40–49`
```rust

    pub big_r: AffinePoint,

**`PresignArguments`** — `src/ecdsa/ot_based_ecdsa/mod.rs:23–34`
```rust
#[derive(Debug, Clone)]
pub struct PresignArguments {
    pub triple0: (TripleShare, TriplePub),
    pub triple1: (TripleShare, TriplePub),
    pub keygen_out: KeygenOutput,          // contains private_share

```
This type aggregates both `TripleShare` instances and the `KeygenOutput` (with its `private_share`) under a single `Debug` derive, making a single log line sufficient to expose all secret inputs to a signing round.

The contradiction is sharpest for `TripleShare` and `PresignOutput`/`RerandomizedPresignOutput`: all three carry `ZeroizeOnDrop` to protect secrets at drop time, yet simultaneously carry `Debug` which can emit those same secrets to any log sink at any earlier point.

---

### Impact Explanation

- **`TripleShare`**: Leaking `a`, `b`, `c` allows an observer to recover the private signing key used in any signature produced with that triple, per the library's own documentation.
- **`PresignOutput` / `RerandomizedPresignOutput`**: Leaking `k` (nonce share) and `sigma` exposes presign secrets; combined with the public `big_r`, an attacker can reconstruct the full nonce and derive the private key.
- **`KeygenOutput`**: Directly exposes `private_share`, the participant's long-term signing share.
- **`PresignArguments`**: A single debug-format of this struct leaks all of the above simultaneously.

This maps to the allowed Critical impact: *Extraction, reconstruction, or disclosure of private signing shares, presign secrets, and nonce material.*

---

### Likelihood Explanation

The `Debug` trait is passively present but is routinely triggered in practice:
- Rust logging crates (`tracing`, `log`, `env_logger`) commonly format structs with `{:?}` in `DEBUG` or `TRACE` level spans.
- Error-handling patterns such as `unwrap_or_else(|e| panic!("{:?}", e))` or `anyhow`/`thiserror` context chains can transitively format these types.
- Any integrating application that stores these structs in a `tracing::instrument` span or passes them through a `Debug`-formatted error chain will silently emit the secret scalars.

The library already signals awareness of this risk by applying `ZeroizeOnDrop` to the same types, making the `Debug` derive an inconsistent configuration choice — directly analogous to the OneSignal SDK enabling `setWebContentsDebuggingEnabled` when the log level is set to verbose.

---

### Recommendation

- Remove `Debug` from the `#[derive(...)]` list for `TripleShare`, `PresignOutput`, `RerandomizedPresignOutput`, `KeygenOutput`, and `PresignArguments`.
- If a `Debug` implementation is needed for development ergonomics, implement it manually and redact all secret scalar fields, e.g.:
  ```rust
  impl fmt::Debug for TripleShare {
      fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
          f.debug_struct("TripleShare")
              .field("a", &"[REDACTED]")
              .field("b", &"[REDACTED]")
              .field("c", &"[REDACTED]")
              .finish()
      }
  }
  ```
- Consider adopting the `secrecy` crate's `Secret<T>` wrapper for scalar fields to enforce redaction at the type level.

### Proof of Concept

```rust
// Any library caller can trigger this with zero privilege:
let triple_share = TripleShare { a: secret_a, b: secret_b, c: secret_c };
// Emitted by any logging framework at DEBUG level:
tracing::debug!(?triple_share, "generated triple");
// Output: triple_share=TripleShare { a: 0x<secret>, b: 0x<secret>, c: 0x<secret> }
// → attacker observing logs recovers a, b, c → recovers private signing key

let presign = PresignOutput { big_r, k: secret_k, sigma: secret_sigma };
tracing::debug!(?presign, "presign complete");
// Output: presign=PresignOutput { big_r: ..., k: 0x<secret>, sigma: 0x<secret> }
// → attacker recovers nonce share k and sigma → derives private key

let args = PresignArguments { triple0, triple1, keygen_out, threshold };
tracing::debug!(?args, "starting sign");
// Single line leaks triple0.a/b/c, triple1.a/b/c, and keygen_out.private_share
