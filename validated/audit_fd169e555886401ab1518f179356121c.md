Audit Report

## Title
`RequestContext` Cache Key Omits `request_type`, Enabling Cross-Version Cache Poisoning — (File: `rs/boundary_node/ic_boundary/src/routes.rs`)

## Summary
The `RequestContext` struct used as the cache lookup key for the boundary node's query response cache omits the `request_type` field from both its `Hash` and `PartialEq` implementations. Because `QueryV2` and `QueryV3` are both treated as cacheable query types by `BypasserIC`, a response cached for one API version is served verbatim to callers of the other API version when all other key fields match. Since V2 and V3 query endpoints embed structurally different NNS delegation certificate formats in their responses, a V3 client that receives a cached V2 response will fail certificate delegation verification.

## Finding Description
`RequestContext` is defined in `rs/boundary_node/ic_boundary/src/routes.rs` (lines 67–82) with a `request_type: RequestType` field. The custom `Hash` implementation (lines 94–108) hashes only `canister_id`, `sender`, `method_name`, `ingress_expiry`, and `arg`/`http_request`. The custom `PartialEq` implementation (lines 111–125) compares only the same fields. `request_type` is absent from both.

`KeyExtractorContext` in `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs` (lines 33–46) uses `Arc<RequestContext>` directly as the cache key, relying entirely on these `Hash`/`PartialEq` implementations.

`BypasserIC::bypass` (lines 74–86 of the same file) bypasses caching only for non-query requests (`!ctx.request_type.is_query()`), nonce-bearing requests, and optionally non-anonymous requests. `RequestType::is_query()` in `rs/boundary_node/ic_boundary/src/http/mod.rs` (lines 62–64) returns `true` for `QueryV2`, `QueryV3`, and `QuerySubnetV3`. All three variants therefore pass the bypasser and are eligible for caching.

The two query API versions are served at distinct URL paths (`/api/v2/canister/{canister_id}/query` and `/api/v3/canister/{canister_id}/query`, lines 12–13 of `rs/boundary_node/ic_boundary/src/http/mod.rs`) and produce structurally different responses: the `Version` enum in `rs/http_endpoints/public/src/query.rs` (lines 50–58) shows V2 uses a flat canister-range format and V3 uses a tree format for the NNS delegation embedded in the signed query response.

For anonymous senders, `rs/validator/src/ingress_validation.rs` (lines 140–142) skips `validate_ingress_expiry`, so an attacker can freely set `ingress_expiry` to any value (including `0`) to match the cache key of a victim's anonymous request.

Exploit flow:
1. Attacker sends an anonymous `QueryV2` POST to `/api/v2/canister/C/query` with `{canister_id: C, method_name: M, arg: A, sender: anon, ingress_expiry: 0, nonce: null}`. The V2-formatted response is stored in the cache under key `(C, anon, M, 0, A)`.
2. Within the cache TTL window (default 1 s, `rs/boundary_node/ic_boundary/src/cli.rs` lines 336–338), a victim sends an anonymous `QueryV3` POST to `/api/v3/canister/C/query` with the same CBOR body.
3. The boundary node computes the same cache key (since `request_type` is absent) and returns the cached V2 response. The V3 client's certificate-delegation parser receives a flat-format delegation instead of the expected tree-format delegation, causing verification failure.

## Impact Explanation
This is a **High** boundary/API integrity and availability impact. Legitimate V3 callers are denied correct responses for the duration of the cache TTL. Because the NNS delegation format differs between V2 and V3, the poisoned response is not merely stale — it is structurally invalid for the recipient's parser, causing hard certificate verification failures. Any boundary node operator who enables the query cache (`--cache-size`) exposes all anonymous query callers to this attack. This matches the allowed impact: "Significant boundary/API security impact with concrete user or protocol harm."

## Likelihood Explanation
No privileged access is required. The attacker only needs to send an anonymous `QueryV2` request before the victim's `QueryV3` request arrives, within a 1-second window. Because `ingress_expiry` is not validated for anonymous senders, the attacker can fix it to `0` or any predictable value, eliminating the only remaining variable in the cache key. Popular canisters queried with fixed arguments (e.g., token balance queries) are the most exposed. The attack is repeatable and low-cost.

## Recommendation
Include `request_type` in both the `Hash` and `PartialEq` implementations of `RequestContext` in `rs/boundary_node/ic_boundary/src/routes.rs`:

```rust
impl Hash for RequestContext {
    fn hash<H: Hasher>(&self, state: &mut H) {
        self.request_type.hash(state); // add this line
        self.canister_id.hash(state);
        self.sender.hash(state);
        self.method_name.hash(state);
        self.ingress_expiry.hash(state);
        if self.http_request.is_some() {
            self.http_request.hash(state);
        } else {
            self.arg.hash(state);
        }
    }
}

impl PartialEq for RequestContext {
    fn eq(&self, other: &Self) -> bool {
        let r = self.request_type == other.request_type // add this field
            && self.canister_id == other.canister_id
            && self.sender == other.sender
            && self.method_name == other.method_name
            && self.ingress_expiry == other.ingress_expiry;
        if self.http_request.is_some() {
            r && self.http_request == other.http_request
        } else {
            r && self.arg == other.arg
        }
    }
}
```

`RequestType` already derives `Hash`, `PartialEq`, and `Eq` (confirmed at `rs/boundary_node/ic_boundary/src/http/mod.rs` lines 26–43), so no additional trait implementations are needed.

## Proof of Concept
1. Start a boundary node with `--cache-size=100MB`.
2. Send an anonymous `QueryV2` POST to `/api/v2/canister/<C>/query` with CBOR body `{canister_id: C, method_name: "balance", arg: [], sender: anon, ingress_expiry: 0}` (no nonce). Confirm the response is a V2-format signed query response (flat canister-range delegation). The response is now cached under key `(C, anon, "balance", 0, [])`.
3. Within 1 second, send an anonymous `QueryV3` POST to `/api/v3/canister/<C>/query` with the identical CBOR body.
4. Observe that the boundary node returns the cached V2 response. The V3 client's certificate-delegation parser receives a flat-format delegation instead of the expected tree-format delegation, producing a verification error.
5. A unit test can be added to `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs` mirroring the existing `test_cache` test: create two requests with `RequestType::QueryV2` and `RequestType::QueryV3` sharing the same `(canister_id, sender, method_name, ingress_expiry, arg)` tuple, assert that the second request results in a cache `Hit`, and assert that after the fix it results in a cache `Miss`.
