### Title
Missing Symmetric Validation in `assert_reshare_keys_invariants` Allows New Participant with Stale Key to Hang Reshare Protocol - (File: `src/dkg.rs`)

---

### Summary

`assert_reshare_keys_invariants` enforces only one direction of the participant/key relationship: it rejects an **old** participant who provides no key, but silently accepts a **new** participant who provides a key. A malicious new participant can exploit this to pass pre-flight validation, enter the reshare protocol with a garbage secret, and fail before sending any messages — leaving all honest participants hanging indefinitely.

---

### Finding Description

In `src/dkg.rs:662-667`, the validation check reads:

```rust
// if me is not in the old participant set then ensure that old_signing_key is None
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}
```

The comment explicitly states the intent: *"if me is not in the old participant set then ensure that old_signing_key is None"*. But the code only implements the **opposite** direction. The symmetric check is entirely absent:

```rust
// MISSING:
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(InitializationError::BadParameters(...));
}
```

When a new participant (not in `old_participants`) passes `old_signing_key = Some(...)`, the pre-flight check passes. Execution proceeds to `do_reshare` (`src/dkg.rs:600-635`), where the secret is computed:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?
    .unwrap_or_else(<C::Group as Group>::Field::zero);
```

Because `me` is **not** in `intersection`, `intersection.lagrange::<C>(me)` computes the Lagrange basis polynomial at `me`'s scalar value for a set that does not include `me`. This yields a non-zero scalar (all participant IDs are `u32 + 1`, so no division-by-zero occurs). The resulting `secret` is non-zero.

`do_keyshare` is then called with this non-zero secret. The secondary guard `assert_keyshare_inputs` (`src/dkg.rs:23-55`) catches it:

```rust
} else {
    // return error if me is part of the old participants set
    if !old_participants.contains(me) {
        return Err(ProtocolError::AssertionFailed(
            format!("{me:?} is running Resharing with a non-zero share but does not belong to the old participant set")));
    }
}
```

This causes the malicious participant's protocol future to fail **before any messages are sent**. The first outbound message (`chan.send_many` for the session ID broadcast) is at `src/dkg.rs:362`, after `assert_keyshare_inputs`. All honest participants are already waiting for the malicious participant's session ID broadcast and will hang indefinitely (or until an external timeout).

---

### Impact Explanation

All honest participants in the reshare session block waiting for the malicious new participant's first broadcast message, which is never sent. The reshare protocol cannot complete. Because the malicious participant can be re-admitted to any subsequent reshare attempt (they are a valid member of `new_participants`), they can repeatedly trigger this hang, constituting a **permanent denial of the reshare operation** for honest parties.

---

### Likelihood Explanation

Any participant included in `new_participants` but absent from `old_participants` can trigger this by passing `old_signing_key = Some(any_value)` to the public `reshare()` API. No privileged access, leaked keys, or cryptographic breaks are required. The attacker only needs to be a legitimate new joiner in the reshare session.

---

### Recommendation

Add the missing symmetric check immediately after the existing one in `assert_reshare_keys_invariants`:

```rust
// Existing check
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}
// ADD: symmetric check — new participant must not provide a key
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is not present in the old participant list but provided a share"
    )));
}
```

This mirrors the intent stated in the existing comment and matches the logic already enforced inside `assert_keyshare_inputs`, moving the rejection to the pre-flight stage before any protocol futures are spawned.

---

### Proof of Concept

1. Run DKG with `old_participants = [P1, P2, P3]`, threshold 2. All three obtain valid `KeygenOutput`.
2. Initiate reshare with `new_participants = [P1, P2, P3, P4]`, new threshold 2.
3. P4 (new participant, absent from `old_participants`) calls `reshare()` with `old_signing_key = Some(any_signing_share_value)`.
4. `assert_reshare_keys_invariants` at `src/dkg.rs:663` evaluates `old_participants.contains(P4)` → `false`, so the condition is `false && ...` → **no error returned**.
5. `do_reshare` computes `secret = intersection.lagrange(P4) * any_value` → non-zero scalar.
6. `do_keyshare` calls `assert_keyshare_inputs`; `is_zero_secret = false` and `!old_participants.contains(P4) = true` → `ProtocolError::AssertionFailed` returned.
7. P4's future exits before `chan.send_many` at `src/dkg.rs:362`.
8. P1, P2, P3 block indefinitely in `do_broadcast` waiting for P4's session ID. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

fn assert_keyshare_inputs<C: Ciphersuite>(
    me: Participant,
    secret: &Scalar<C>,
    old_reshare_package: Option<(VerifyingKey<C>, ParticipantList)>,
) -> Result<(Option<VerifyingKey<C>>, Option<ParticipantList>), ProtocolError> {
    let is_zero_secret = *secret == <C::Group as Group>::Field::zero();

    if let Some((old_key, old_participants)) = old_reshare_package {
        if is_zero_secret {
            //  return error if me is not a purely new joiner to the participants set
            //  prevents accidentally calling keyshare with extremely old keyshares
            //  that have nothing to do with the current resharing
            if old_participants.contains(me) {
                return Err(ProtocolError::AssertionFailed(
                    format!("{me:?} is running Resharing with a zero share but does belong to the old participant set")));
            }
        } else {
            //  return error if me is part of the old participants set
            if !old_participants.contains(me) {
                return Err(ProtocolError::AssertionFailed(
                    format!("{me:?} is running Resharing with a non-zero share but does not belong to the old participant set")));
            }
        }
        Ok((Some(old_key), Some(old_participants)))
    } else {
        if is_zero_secret {
            return Err(ProtocolError::AssertionFailed(format!(
                "{me:?} is running DKG with a zero share"
            )));
        }
        Ok((None, None))
    }
}
