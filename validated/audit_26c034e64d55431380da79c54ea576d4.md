### Title
Unprivileged Caller Can Suppress All Block Reward Distribution via Unvalidated `disable_rewards` Parameter — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract accepts a caller-controlled `disable_rewards` boolean without any role check. Any unprivileged address can call `update_rewards(any_valid_staker, disable_rewards: true)` once per block to advance the global `last_reward_block` counter while skipping reward distribution entirely, permanently starving all stakers of consensus block rewards.

---

### Finding Description

`update_rewards` is a public function in `StakingRewardsManagerImpl`. Its only access guard is `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero — no role or identity check is performed. [1](#0-0) 

The function unconditionally writes `current_block_number` to the global `last_reward_block` storage slot **before** inspecting `disable_rewards`: [2](#0-1) 

After that write, if `disable_rewards` is `true`, the function returns immediately without calling `_update_rewards`: [3](#0-2) 

Because `last_reward_block` is a single global slot (not per-staker), any subsequent call to `update_rewards` in the same block will fail with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The attacker therefore consumes the one allowed call per block for the entire protocol, with zero rewards distributed.

The `disable_rewards` parameter is analogous to the unvalidated `value` field in the Uniswap wallet report: a parameter that is accepted from an untrusted caller without validation, silently altering the outcome of a critical operation.

---

### Impact Explanation

`update_rewards` is the sole path through which consensus block rewards are computed and forwarded to stakers and their delegation pools in the post-`consensus_rewards_first_epoch` phase. Suppressing it every block means:

- Stakers accumulate zero `unclaimed_rewards_own`.
- Delegation pools receive zero `pool_rewards`, so pool members earn nothing.
- The yield that should have been minted and distributed is permanently lost (the minting curve does not retroactively compensate for skipped blocks).

This constitutes **permanent freezing of unclaimed yield** (High) for all stakers and delegators simultaneously, with a single cheap transaction per block.

---

### Likelihood Explanation

- The function is fully public; no role, no signature, no stake required.
- The attacker only needs any one valid, active staker address with non-zero balance — trivially obtained from on-chain events (`NewStaker`).
- On Starknet, transaction fees are low; calling once per block is economically viable for a motivated griever.
- The attack is entirely non-interactive and can be automated.

---

### Recommendation

**Short term**: Add an access-control check so that only a privileged role (e.g., `OPERATOR` or the staker's own operational address) may supply `disable_rewards: true`. Alternatively, remove the parameter from the public interface and derive the skip-rewards condition internally from protocol state.

**Long term**: Audit all public functions that accept boolean flags or mode selectors for similar missing caller-validation patterns.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker identifies any active staker `S` with non-zero STRK balance (readable from `NewStaker` events).
3. At the first transaction slot of every block, attacker calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. Inside the call:
   - `general_prerequisites()` passes (contract not paused, caller non-zero). [5](#0-4) 
   - `current_block_number > last_reward_block` passes (new block). [4](#0-3) 
   - `last_reward_block` is written to `current_block_number`. [6](#0-5) 
   - `disable_rewards == true` → early return, `_update_rewards` never called. [7](#0-6) 
5. Any legitimate call to `update_rewards` later in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeated every block: all stakers and pool members earn zero block rewards indefinitely, with no recourse.
