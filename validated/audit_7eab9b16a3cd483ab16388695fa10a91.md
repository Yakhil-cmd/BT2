### Title
Missing `app_pk` Identity-Point Validation Allows Masking Bypass and Confidential Key Disclosure - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` entry point in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk: PublicKey` (a BLS12-381 G1 element) without validating that it is not the group identity element. When `app_pk` is the identity, the masking term `y · app_pk` collapses to the identity, so the coordinator receives the unmasked confidential key `msk · H(pk ∥ app_id)` directly in the protocol output — without needing `app_sk`. This is a direct analog to the external report's pattern: a critical input is accepted without a validity check, causing a security-sensitive operation to silently succeed with insecure behavior.

### Finding Description

In `compute_signature_share()` (line 174), each participant computes:

```
C ← S + y · app_pk
```

where `S = x_i · H(pk ∥ app_id)` is the participant's secret share contribution and `y` is a fresh random scalar. The masking term `y · app_pk` is what prevents the coordinator from learning the confidential key without the application's secret key `app_sk`. [1](#0-0) 

If `app_pk = G1::identity()`, then `app_pk * y.0 = identity`, so `big_c = big_s = x_i · H(pk ∥ app_id)`. After Lagrange weighting and summation by the coordinator:

```
Σ λ_i · C_i = Σ λ_i · x_i · H(pk ∥ app_id) = msk · H(pk ∥ app_id)
```

The coordinator's `CKDOutput.norm_big_c` is the confidential key itself, readable without calling `unmask(app_sk)`.

The `ckd()` entry point performs four input checks — participant count, duplicates, self-membership, coordinator-membership — but performs **no validation on `app_pk`**: [2](#0-1) 

By contrast, `verify_signature()` in the same module explicitly guards against identity elements on both G1 and G2: [3](#0-2) 

The codebase is therefore aware of the need for identity-point checks but omits it at the CKD entry point.

### Impact Explanation

A caller who passes `app_pk = G1::identity()` causes every participant's `norm_big_c` to equal `λ_i · msk · H(pk ∥ app_id)`. The coordinator sums these to obtain `msk · H(pk ∥ app_id)` — the confidential derived key — directly from the protocol output, without possessing `app_sk`. This is **extraction/disclosure of a confidential derived secret**, matching the Critical impact tier.

### Likelihood Explanation

The `app_pk` is an externally supplied parameter. A malicious coordinator who controls the `app_pk` value distributed to participants (e.g., by substituting the identity point in the setup phase), or an application that accidentally supplies a zero/identity key, triggers the bypass with no cryptographic capability required. The library provides no guard, so the failure is silent and the output appears structurally valid.

### Recommendation

Add an explicit identity-point check for `app_pk` in `ckd()` before the protocol is launched, mirroring the pattern already used in `verify_signature()`:

```rust
// In ckd(), after the coordinator membership check:
if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
``` [4](#0-3) 

### Proof of Concept

1. Call `ckd()` with all valid parameters except `app_pk = G1Projective::identity()`.
2. Run the protocol to completion.
3. Observe that the coordinator's `CKDOutput.norm_big_c` equals `msk · H(pk ∥ app_id)`.
4. Verify: `ckd_output.norm_big_c == hash_app_id_with_pk(&public_key, &app_id) * msk` — the confidential key is directly readable without `app_sk`.

The existing test in `protocol.rs` already computes `expected_confidential_key = hash_app_id_with_pk(&pk, &app_id) * msk` and asserts equality with `ckd_output.unmask(app_sk)`. With `app_pk = identity`, `norm_big_c` equals `expected_confidential_key` directly, confirming the bypass. [5](#0-4)
