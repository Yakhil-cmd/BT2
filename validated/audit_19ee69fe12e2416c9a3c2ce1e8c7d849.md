### Title
Malicious CKD Participant Can Corrupt Coordinator Output Without Proof of Correct Computation — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator blindly aggregates `(norm_big_y, norm_big_c)` contributions from every participant with no proof that each contribution was honestly computed from the participant's actual key share. A malicious participant can inject an arbitrary additive delta into either component, corrupting the final confidential key that the coordinator outputs, while honest parties have no way to detect the manipulation.

### Finding Description

`compute_signature_share` correctly derives each participant's contribution:

```
big_s  = x_i · H(pk ∥ app_id)
big_c  = big_s + y · app_pk
big_y  = y · G
norm_* = λ_i · *
``` [1](#0-0) 

Each participant sends `(norm_big_y, norm_big_c)` privately to the coordinator. The coordinator then aggregates without any verification:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [2](#0-1) 

There is no zero-knowledge proof, no consistency check, and no binding between the sent `(big_y, big_c)` pair and the participant's public key share. The protocol verifies nothing about the received values before incorporating them into the final `CKDOutput`.

The analog to the external report is direct:

| External report (Illuminate) | This codebase (CKD) |
|---|---|
| Fee applied to `lent` (the declared order amount) | Protocol design expects correctly-formed `(big_y, big_c)` |
| `premium` (extra balance change after execution) flows through with no fee | Arbitrary additive delta `(Δ_y, Δ_c)` flows through the same summation with no check |
| Coordinator accepts the full balance without charging the premium | Coordinator accepts the full contribution without verifying the premium delta |

A malicious participant sends `(norm_big_y + Δ_y, norm_big_c + Δ_c)` for any group elements `Δ_y`, `Δ_c`. The coordinator accumulates these unchecked deltas:

```
big_Y_final = Σ λ_i · y_i · G  +  Δ_y
big_C_final = msk · H(pk∥app_id) + (Σ λ_i · y_i) · app_pk  +  Δ_c
```

The client's `unmask(app_sk)` then yields:

```
big_C_final − app_sk · big_Y_final
  = msk · H(pk∥app_id)  +  Δ_c − app_sk · Δ_y
```

Unless `Δ_c = app_sk · Δ_y` (which requires knowing the client's secret `app_sk`), the derived confidential key is permanently wrong. The client receives a silently corrupted key with no error signal. [3](#0-2) 

### Impact Explanation

A single malicious participant can cause the coordinator to output a corrupted `CKDOutput`. Every honest party that relies on this output — including the client calling `unmask` — will silently derive the wrong confidential key. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

Any participant in the CKD protocol can exploit this. No special privilege, leaked key, or external assumption is required. The attacker only needs to be one of the `n` participants and to deviate from the protocol by sending a modified `(big_y, big_c)` pair. The entry path is the private message sent at `waitpoint` in `do_ckd_participant`. [4](#0-3) 

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that:

1. `big_y = y · G` for some scalar `y` (proof of discrete log),
2. `big_c = x_i · H(pk∥app_id) + y · app_pk` where `x_i` is consistent with the participant's public key share (proof of discrete-log equality across two generators, analogous to the existing `dlogeq` proof already present in `src/crypto/proofs/dlogeq.rs`). [5](#0-4) 

The coordinator must verify these proofs before incorporating any contribution into the running sum.

### Proof of Concept

1. Run the CKD protocol with 3 participants, one of which is malicious.
2. The malicious participant, instead of calling `compute_signature_share`, sends `(norm_big_y + Δ_y, norm_big_c + Δ_c)` where `Δ_c` is an arbitrary non-identity group element and `Δ_y` is the identity.
3. The coordinator at lines 50–55 adds the malicious contribution without any check.
4. The final `CKDOutput` satisfies `big_C_final = correct_C + Δ_c`.
5. The client calls `unmask(app_sk)` and obtains `msk · H(pk∥app_id) + Δ_c` — a permanently wrong confidential key — with no error returned anywhere in the protocol. [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```

pub fn prove_with_nonce<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    witness: Witness<C>,
    k: Scalar<C>,
) -> Result<Proof<C>, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (big_k_0, big_k_1) = statement.phi(&k);

    // This will never raise error as k is not zero and generator1 is not the identity
    let enc = encode_two_points::<C>(&big_k_0, &big_k_1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    let s = k + e * witness.x.0;
    Ok(Proof {
        e: SerializableScalar::<C>(e),
        s: SerializableScalar::<C>(s),
    })
}
