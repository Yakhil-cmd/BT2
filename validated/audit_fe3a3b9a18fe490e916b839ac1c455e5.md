### Title
Pool Admin Bypasses Factory Fee Caps via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with **no cap validation**, while the parallel admin fee setter `setPoolAdminFees` explicitly enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin fees to the `uint16` maximum (65535 ≈ 6.5535% in E6) on every bin, bypassing the factory's fee-cap boundary that is supposed to constrain semi-trusted pool admins.

---

### Finding Description

`setPoolAdminFees` enforces the factory cap before writing:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has **no equivalent check** — it passes the caller-supplied values straight through:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it stores the raw values:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6) external onlyFactory ... {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

During every swap, the per-bin fee is added directly on top of the oracle-derived base fee:

```solidity
// MetricOmmPool.sol:999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

The hard cap `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) governs the global spread fee, and `maxAdminSpreadFeeE6` (≤ 20%) governs `setPoolAdminFees`. Neither cap is applied to per-bin fees. [5](#0-4) 

---

### Impact Explanation

A pool admin calls `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` for every bin in the pool. Each bin's effective swap fee becomes `baseFeeX64 + 6.5535%`, additive on top of the global spread fee (which can itself be up to 20%). The total effective fee per bin exceeds the hard cap. Traders receive materially less output than the oracle-permitted price allows, constituting a direct loss of principal on every swap. LPs cannot exit at fair prices because the pool's quoted bid/ask is distorted by the inflated per-bin fee applied in `_marginalBestBidAsk`. [6](#0-5) 

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted and bounded by factory caps. The attack requires only the pool admin to call `setPoolBinAdditionalFees` once per bin — no timelock, no co-signer, no oracle manipulation. The factory's own audit pivot (`setBinAdditionalFees` listed as an admin-boundary check point) confirms this path was intended to be guarded.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinSpreadFeeE6` cap if per-bin fees are intended to have a different ceiling than the global admin spread fee.

---

### Proof of Concept

1. Pool is deployed with `maxAdminSpreadFeeE6 = 50_000` (5%) and 3 bins (indices -1, 0, 1).
2. Pool admin calls `setPoolBinAdditionalFees(pool, -1, 65535, 65535)`, `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`, `setPoolBinAdditionalFees(pool, 1, 65535, 65535)` — all succeed with no revert.
3. A trader executes a swap crossing bin 0. The effective fee applied is `baseFeeX64 + (65535/1e6)*ONE_X64`, i.e., base + 6.5535%, far exceeding the 5% admin cap.
4. Assert: `trader output < oracle-permitted output at maxAdminSpreadFeeE6`; assert `addFeeBuyE6 stored = 65535 > maxAdminSpreadFeeE6 = 50_000`.
