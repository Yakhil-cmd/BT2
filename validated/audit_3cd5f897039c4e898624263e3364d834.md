### Title
Per-Block Entropy Precompile Exposes Predictable Zero-Cost Randomness, Enabling Fund Theft in Randomness-Dependent Contracts — (File: `engine-precompiles/src/random.rs`)

---

### Summary

The Aurora `RandomSeed` precompile (`0xc104f4840573bed437190daf5d2898c2bdf928ac`) exposes the NEAR block-level VRF seed as an on-chain entropy source. This seed is **identical for every transaction within the same block**, is publicly observable on NEAR before Aurora transactions finalize, and costs **zero gas** to query (an unresolved TODO). Any Aurora smart contract that uses this precompile for security-critical randomness — lotteries, key generation, commit-reveal schemes — is trivially exploitable by an attacker who reads the block seed from NEAR and predicts the outcome before submitting their own transaction.

---

### Finding Description

`engine-precompiles/src/random.rs` defines the `RandomSeed` precompile:

```rust
mod costs {
    // TODO(#483): Determine the correct amount of gas
    pub(super) const RANDOM_BYTES_GAS: EthGas = EthGas::new(0);
}
``` [1](#0-0) 

The precompile's own documentation states:

> "This is a per-block entropy source which could then be used to create a random sequence. **It will return the same seed if called multiple times in the same block.**" [2](#0-1) 

The `run` method unconditionally returns the block-level seed stored at construction time:

```rust
Ok(PrecompileOutput::without_logs(
    cost,
    self.random_seed.as_bytes().to_vec(),
))
``` [3](#0-2) 

The seed itself is sourced from the NEAR runtime's `random_seed` host function, which is a per-block VRF output:

```rust
fn random_seed(&self) -> H256 {
    unsafe {
        exports::random_seed(0);
        let mut bytes = H256::zero();
        exports::read_register(0, bytes.0.as_mut_ptr() as u64);
        bytes
    }
}
``` [4](#0-3) 

This seed is passed into every precompile set (Istanbul, Berlin, London, Prague, Osaka) at construction time:

```rust
Box::new(RandomSeed::new(ctx.random_seed)),
``` [5](#0-4) 

Two compounding defects exist:

1. **Entropy collapse**: The seed is fixed for the entire block. All transactions in the same block receive the identical 32-byte value. An attacker who reads the NEAR block's `random_seed` field (public, available before Aurora transactions are processed) knows the exact value any Aurora contract will receive.

2. **Zero gas cost**: `RANDOM_BYTES_GAS = EthGas::new(0)` means the precompile can be called an unlimited number of times within a transaction at no cost. The `TODO(#483)` comment confirms this was never resolved.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Any Aurora smart contract that uses the `RandomSeed` precompile for security-critical decisions (lottery winner selection, on-chain key derivation, commit-reveal resolution, random access-control gates) is fully predictable. An attacker who observes the NEAR block's random seed — which is public chain state — can compute the exact output the precompile will return for every transaction in that block. The attacker can then:

- Front-run a lottery contract by submitting a winning transaction after observing the seed.
- Derive the same "random" private key a contract generates and take ownership of associated funds.
- Resolve a commit-reveal scheme in their favor before honest participants.

The zero gas cost removes any economic friction from repeated probing or looping over the precompile within a single transaction.

---

### Likelihood Explanation

**High.** The precompile is part of Aurora's documented production interface, exposed across all EVM hard-fork configurations (Homestead through Osaka). Its own inline documentation explicitly describes it as an entropy source "which could then be used to create a random sequence," directly inviting contract authors to use it for randomness. The test contract `etc/eth-contracts/contracts/test/Random.sol` demonstrates calling it for a `uint256` random value, reinforcing the pattern. Any contract author following this example will produce exploitable code. [6](#0-5) 

---

### Recommendation

1. **Short term**: Assign a non-zero gas cost to the `RandomSeed` precompile (resolve `TODO(#483)`). Add prominent documentation warning that the seed is per-block, publicly observable, and **must not** be used as the sole entropy source for security-critical decisions.

2. **Long term**: Derive per-transaction entropy by mixing the block seed with the transaction hash or action hash (as NEAR's own runtime does via `create_random_seed(action_hash, random_seed)`), so each transaction receives a distinct value that cannot be predicted from the block seed alone. [7](#0-6) 

---

### Proof of Concept

1. Deploy a lottery contract on Aurora that calls the `RandomSeed` precompile and awards the pot to the caller whose submitted number matches `uint256(randomSeed()) % N`.
2. Before submitting any transaction, read the current NEAR block's `random_seed` field from the NEAR RPC (`block.header.random_value`).
3. Compute `uint256(seed) % N` off-chain.
4. Submit a transaction with the winning number. Because `RANDOM_BYTES_GAS = 0`, this costs no additional gas beyond the base transaction fee.
5. The contract pays out the pot to the attacker.

The attack requires no privileged access, no admin compromise, and no external oracle — only the ability to read public NEAR chain state and submit an Aurora transaction. [1](#0-0) [2](#0-1) [3](#0-2)
