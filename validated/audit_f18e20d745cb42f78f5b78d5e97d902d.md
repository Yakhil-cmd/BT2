### Title
Gateway Admits Invoke Transactions with Invalid Signatures via Overly Broad `skip_stateful_validations` Proxy Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway bypasses the `__validate__` entry-point (signature verification) for any invoke transaction whose nonce is `1` and whose sender's on-chain nonce is `0`, whenever `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction from the address — not exclusively a `deploy_account` transaction. An unprivileged attacker who observes a pending `deploy_account` in the mempool can therefore submit an invoke with an arbitrary (invalid) signature for the same address and have it admitted to the mempool without any signature check.

---

### Finding Description

`skip_stateful_validations` is the gateway's UX shortcut that lets a user broadcast `deploy_account` (nonce 0) and `invoke` (nonce 1) simultaneously, before the account exists on-chain. The function's guard is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    // We verify that a deploy_account transaction exists for this account.
    // It is sufficient to check if the account exists in the mempool since it means
    // that either it has a deploy_account transaction or transactions with future
    // nonces that passed validations.
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` and the blockifier's `StatefulValidator` skips the `__validate__` call entirely:

```rust
// lines 310-312
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

The proxy check `account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`state.contains_account` returns `true` if the address appears in either the `staged` or `committed` maps — populated by **any** transaction type, not only `deploy_account`:

```rust
// lines 115-117
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [4](#0-3) 

The code comment's second justification — *"transactions with future nonces that passed validations"* — is the broken invariant. It assumes that any account with a pending transaction must be a legitimate account whose nonce-1 invoke will also be legitimate. This is false: a third party who does not hold the account's private key can submit a nonce-1 invoke with a fabricated signature for any address that already has a pending `deploy_account` in the pool.

The mempool's own `validate_tx` path (called before `skip_stateful_validations`) only checks nonce ordering and fee escalation — it does not verify signatures:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

The only place a signature is ever verified is inside `__validate__`, which is the entry point that `skip_validate = true` suppresses. [6](#0-5) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an arbitrary number of syntactically valid but cryptographically invalid invoke transactions (nonce 1, wrong signature) for every address that has a pending `deploy_account` in the mempool. Each such transaction:

1. Passes all gateway stateless checks (signature length, resource bounds, DA mode).
2. Passes the gateway stateful nonce check (`0 ≤ 1 ≤ 0 + gap`).
3. Passes the mempool's `validate_tx` (nonce ordering only).
4. Bypasses `__validate__` entirely due to `skip_validate = true`.
5. Is inserted into the mempool pool and priority queue.

The batcher will later dequeue these transactions, attempt execution, call `__validate__` (which runs with `execution_flags.validate = true` by default during block building), observe the signature failure, and reject them. However, the admission invariant — *every transaction in the mempool has passed account-level validation* — is broken at the gateway boundary.

---

### Likelihood Explanation

**Medium.** The precondition (a pending `deploy_account` in the mempool for the target address) is publicly observable via the P2P gossip layer or the RPC. The attack requires no privileged access, no special account state, and no on-chain funds. An attacker monitoring the mempool can immediately submit a spoofed nonce-1 invoke for every newly seen `deploy_account`, saturating the mempool with invalid transactions and forcing the batcher to waste execution resources rejecting them.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` proxy with a check that specifically confirms a `deploy_account` transaction is present in the pool for the sender address. Concretely:

1. Add a `has_pending_deploy_account(address)` method to `Mempool` that inspects the transaction type stored in `tx_pool` for the given address at nonce 0.
2. In `skip_stateful_validations`, call this narrower predicate instead of `account_tx_in_pool_or_recent_block`.

Alternatively, pass the `deploy_account` transaction hash from the client (as `native_blockifier`'s `PyValidator::perform_validations` already does via `deploy_account_tx_hash`) and verify that hash is present in the pool before skipping `__validate__`. [7](#0-6) 

---

### Proof of Concept

```
1. Observe mempool: address A has a pending deploy_account (nonce=0).
   → tx_pool.contains_account(A) == true
   → account_tx_in_pool_or_recent_block(A) == true

2. Craft invoke_tx:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef]   // arbitrary invalid bytes
     resource_bounds = valid (passes stateless check)

3. Submit to gateway via add_tx RPC.

4. Gateway stateless validator: passes (signature length ≤ max, resource bounds OK).

5. extract_state_nonce_and_run_validations:
     account_nonce = get_nonce(A) = 0   (deploy_account not yet committed)
     validate_nonce: 0 ≤ 1 ≤ 0+gap → OK
     validate_by_mempool: nonce ordering OK, no duplicate
     skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → true
       account_tx_in_pool_or_recent_block(A) → true
       returns skip=true

6. run_validate_entry_point called with skip_validate=true:
     execution_flags.validate = false
     StatefulValidator::perform_validations → skips __validate__ call
     → returns Ok(())

7. Invalid invoke admitted to mempool.

8. Batcher dequeues it, executes with validate=true:
     __validate__ called → signature check fails → transaction rejected.

Result: invalid transaction admitted to mempool, breaking the admission invariant.
``` [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
