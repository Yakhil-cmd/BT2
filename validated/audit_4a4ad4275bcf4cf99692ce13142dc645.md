### Title
Declare Transactions Bypass `max_l2_gas_amount` Gateway Admission Check — (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` contains an intentionally empty branch for `RpcTransaction::Declare`, silently skipping the `max_l2_gas_amount` upper-bound check that is enforced for every other transaction type. This is the direct sequencer analog of the external bug: an empty implementation for one variant of a multi-variant dispatch, based on the current assumption that the check is unnecessary for Declare transactions.

### Finding Description

In `validate_resource_bounds` (lines 78–85 of `crates/apollo_gateway/src/stateless_transaction_validator.rs`):

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // empty — no check performed
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
```

The production config sets `max_l2_gas_amount = 1_210_000_000` (from `config_schema.json` line 3175). For `Invoke` and `DeployAccount` transactions, submitting `l2_gas.max_amount > 1_210_000_000` causes an immediate `MaxGasAmountTooHigh` rejection at the gateway. For `Declare` transactions, the same field is never checked — any value up to `u64::MAX` passes stateless validation.

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* (not the *amount*), so the gap is not closed there either:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:223-243
async fn validate_resource_bounds(&self, executable_tx: &ExecutableTransaction) -> … {
    if self.config.validate_resource_bounds {
        // only validates l2_gas_price, not l2_gas_amount
        self.validate_tx_l2_gas_price_within_threshold(
            executable_tx.resource_bounds(),
            previous_block_l2_gas_price,
        )?;
    }
    Ok(())
}
```

The test `valid_l2_gas_amount_on_declare` (lines 191–201 of `stateless_transaction_validator_test.rs`) explicitly confirms this behaviour: a Declare transaction with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount = 100`, which would be rejected for any other type.

The only downstream guard is `verify_can_pay_committed_bounds` inside `perform_pre_validation_stage` (blockifier, `account_transaction.rs` line 366), which checks `account_balance >= max_amount * max_price_per_unit`. An attacker with a funded account can set `max_amount` to any value that still satisfies the balance check, bypassing the gateway's intended admission limit entirely.

### Impact Explanation

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

A Declare transaction with `l2_gas.max_amount` above the configured `max_l2_gas_amount` threshold is invalid by the gateway's own rules (the same rule is enforced for Invoke and DeployAccount). The gateway admits it anyway. Consequences:

1. **Admission-control bypass**: An attacker can submit Declare transactions that would be rejected for any other type, circumventing the DoS protection the limit is designed to provide.
2. **Mempool pollution**: Declare transactions with inflated `max_amount` values are queued and consume sequencer resources (blockifier validation, state reads) before failing or succeeding at execution time.
3. **Fee-accounting inconsistency**: `max_possible_fee = max_amount × max_price_per_unit` is used in `verify_can_pay_committed_bounds`. With an unchecked `max_amount`, a funded account can commit to a fee envelope far larger than the protocol intends to allow at the gateway layer.

### Likelihood Explanation

Trivially reachable by any unprivileged user. Crafting a V3 Declare transaction with an oversized `l2_gas.max_amount` requires no special access — only a funded account and a valid Sierra class. The gateway's JSON-RPC endpoint is publicly exposed.

### Recommendation

Remove the empty branch and apply the same `max_l2_gas_amount` guard to Declare transactions:

```rust
// Remove the empty Declare branch; apply the check uniformly:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher gas-amount ceiling (e.g., due to large Sierra programs), introduce a separate `max_l2_gas_amount_declare` config field rather than leaving the check absent entirely.

### Proof of Concept

1. Fund an account with sufficient STRK balance.
2. Construct a valid `RpcDeclareTransaction::V3` with `resource_bounds.l2_gas.max_amount = 1_210_000_001` (one above the configured limit) and `max_price_per_unit` at or above `min_gas_price`.
3. Submit via the gateway's `add_declare_transaction` RPC endpoint.
4. **Observed**: The transaction passes `StatelessTransactionValidator::validate` and is forwarded to the stateful validator and mempool.
5. **Expected** (by analogy with Invoke/DeployAccount): The gateway returns `MaxGasAmountTooHigh`.
6. Repeat with an `RpcInvokeTransaction::V3` carrying the same `max_amount` to confirm it is rejected at step 3 — demonstrating the inconsistency.

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
