### Title
`EVM.validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress`, Enabling Cross-Purpose Signature Replay Through ERC-1271 - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account signed arbitrary `signedData` bytes with the `"FLOW-V0.0-user"` domain tag, but does not enforce that `signedData` encodes the `evmAddress` being validated. Any Flow user signature produced for any off-chain purpose can be replayed by an attacker to pass ERC-1271 (`isValidSignature`) validation for the victim's COA in EVM protocols, potentially authorizing unauthorized asset transfers on behalf of the victim's COA.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` (lines 1001–1116) is the Cadence-side implementation of COA ownership proof validation. It accepts caller-supplied `signedData: [UInt8]` and verifies that the Cadence account at `address` produced a valid signature over those bytes using the fixed domain separation tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The function then checks that the COA at `path` matches `evmAddress`, but **never checks that `signedData` encodes `evmAddress`**. The code itself acknowledges this explicitly:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The ERC-1271 entry point is `coa.sol`'s `isValidSignature(bytes32 _hash, bytes memory _sig)`, which calls the Cadence Arch precompile `verifyCOAOwnershipProof(address, bytes32, bytes)` with `_hash` passed directly as `signedData`:

```solidity
(bool ok, bytes memory data) = cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this), _hash, _sig
    )
);
```

The precompile (`fvm/evm/handler/precompiles.go`, `coaOwnershipProofValidator`, lines 115–153) invokes `validateCOAOwnershipProof` with the proof's `SignedData` field set to whatever `_hash` was passed by the EVM caller. There is no binding between `_hash` and the COA's EVM address.

This means: any valid Flow user signature over any 32-byte value `H` — produced for any purpose using the `"FLOW-V0.0-user"` domain tag — can be submitted as a proof to pass ERC-1271 validation for the victim's COA in any EVM protocol that calls `isValidSignature(H, sig)`.

---

### Impact Explanation

An attacker who observes a victim's Flow user signature (e.g., from an off-chain authentication flow, a signed challenge for a dApp, or any other use of the `"FLOW-V0.0-user"` domain tag) can replay that signature to pass ERC-1271 validation for the victim's COA in EVM protocols deployed on Flow EVM. EVM DeFi protocols that use ERC-1271 for authorization of asset operations (e.g., off-chain order books, permit-like approvals, NFT marketplace listings) would accept this replayed proof as a valid authorization from the victim's COA, enabling unauthorized movement of assets held by or approved to the victim's COA.

---

### Likelihood Explanation

Medium. The attack requires:
1. The victim to have produced a Flow user signature over some 32-byte value using the `"FLOW-V0.0-user"` domain tag.
2. The attacker to have observed that signature and the signed value (e.g., from a public off-chain authentication challenge-response).
3. An EVM protocol on Flow EVM to use ERC-1271 for authorization of asset operations.

As Flow EVM adoption grows and more EVM DeFi protocols deploy on it, condition 3 becomes increasingly likely. Condition 1 and 2 are realistic: off-chain authentication flows (wallet connect, sign-in-with-flow, etc.) routinely produce and publish user signatures over challenge hashes.

---

### Recommendation

Enforce that `signedData` encodes `evmAddress` before verifying the signature. The simplest fix is to compute the expected signed payload inside `validateCOAOwnershipProof` by hashing or concatenating `evmAddress` with the caller-supplied data, and verifying the signature against that bound payload — rather than verifying against the raw caller-supplied `signedData`. Alternatively, require that `signedData` begins with the 20-byte `evmAddress` and reject proofs where it does not. This mirrors the `permitWitnessTransferFrom` recommendation in the external report: the authorization must encode the specific target/action it is intended for.

---

### Proof of Concept

1. Victim signs a 32-byte challenge `H` with their Flow key using the `"FLOW-V0.0-user"` domain tag for a dApp login flow. The signature `sig` and `H` are publicly observable.
2. Victim has a COA stored at `/storage/coa` and published at `/public/coa` on their Cadence account `A`. The COA's EVM address is `coaAddr`.
3. Attacker constructs a `COAOwnershipProof` with `Address = A`, `CapabilityPath = "coa"`, `SignedData = H`, `Signatures = [sig]`, `KeyIndices = [0]`.
4. Attacker calls an EVM contract `C` on Flow EVM that uses ERC-1271 for authorization (e.g., an NFT marketplace that checks `isValidSignature(H, encodedProof)` on the COA before filling a listing).
5. Contract `C` calls `coaAddr.isValidSignature(H, encodedProof)`.
6. `coa.sol`'s `isValidSignature` calls `verifyCOAOwnershipProof(coaAddr, H, encodedProof)` on the Cadence Arch precompile.
7. The precompile invokes `EVM.validateCOAOwnershipProof(address: A, path: /public/coa, signedData: H, keyIndices: [0], signatures: [sig], evmAddress: coaAddr)`.
8. The function verifies `sig` over `H` with `"FLOW-V0.0-user"` tag — valid, because the victim signed exactly `H`.
9. The function checks that the COA at `/public/coa` matches `coaAddr` — valid.
10. Returns `isValid: true`. Contract `C` accepts the proof and executes the action (e.g., transfers an NFT or approves a token spend) on behalf of the victim's COA without the victim's consent for that specific action.

func coaOwnershipProofValidator(contractAddress flow.Address, backend backends.Backend) func(proof *types.COAOwnershipProofInContext) (bool, error) {
	return func(proof *types.COAOwnershipProofInContext) (bool, error) {
		value, err := backend.Invoke(
			environment.ContractFunctionSpec{
				AddressFromChain: func(_ flow.Chain) flow.Address {
					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		isValidValue := cadence.SearchFieldByName(data, ValidationResultTypeIsValidFieldName)
		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
}
