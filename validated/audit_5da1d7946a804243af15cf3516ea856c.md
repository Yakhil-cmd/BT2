### Title
NEP-141 Tokens Permanently Frozen When Sent via `ft_transfer` Instead of `ft_transfer_call` — (`engine/src/contract_methods/connector.rs`, `engine/src/engine.rs`)

---

### Summary

The Aurora Engine mints ERC-20 tokens for a user only when NEP-141 tokens arrive via `ft_transfer_call`, which triggers the `ft_on_transfer` callback. If a user instead calls `ft_transfer` directly on a NEP-141 contract with `receiver_id = aurora`, the tokens are credited to Aurora's NEP-141 balance but `ft_on_transfer` is never invoked, no ERC-20 tokens are minted, and no admin rescue function exists to recover the stuck tokens. The funds are permanently frozen.

---

### Finding Description

The Aurora Engine's token-bridging entry point is `ft_on_transfer`, which is only invoked by the NEP-141 standard when a sender uses `ft_transfer_call`. Inside `ft_on_transfer`, the engine dispatches to either `receive_base_tokens` (for bridged ETH) or `receive_erc20_tokens` (for any registered NEP-141 token), minting the corresponding ERC-20 balance in the EVM. [1](#0-0) 

If a user calls `ft_transfer` on a NEP-141 contract (e.g., USDC.near) with `receiver_id = aurora_engine_account`, the NEP-141 standard transfers the tokens to Aurora's account on that contract but does **not** call `ft_on_transfer`. The Aurora Engine receives no notification, mints no ERC-20 tokens, and the user's NEP-141 balance is silently consumed. [2](#0-1) 

The only way to move NEP-141 tokens back out of Aurora's account is through the `ExitToNear` precompile, which burns ERC-20 tokens and calls `ft_transfer` on the NEP-141 contract. Because no ERC-20 tokens were minted in the first place, this path is unavailable. [3](#0-2) 

The admin module contains no rescue or recovery function for arbitrary NEP-141 tokens held by the Aurora Engine account. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any NEP-141 token that has been registered with Aurora via `deploy_erc20_token` can be permanently frozen this way. The tokens exist in Aurora's NEP-141 balance on the external token contract, but the Aurora Engine has no built-in mechanism to recover them. A contract upgrade would be required to add a rescue path, and even then the upgrade process is gated behind the owner and an upgrade delay.

---

### Likelihood Explanation

**Medium.** The NEP-141 standard exposes two transfer functions — `ft_transfer` and `ft_transfer_call` — with nearly identical signatures. A user who is unfamiliar with the bridging protocol, or who copies a simple transfer snippet, will naturally reach for `ft_transfer`. This is the exact human-nature failure mode described in the reference report. The Aurora Engine is a high-value bridge target, making accidental misuse likely over time.

---

### Recommendation

Add an owner-only or admin-only function to the Aurora Engine that issues a cross-contract `ft_transfer` call on a specified NEP-141 contract to move tokens out of Aurora's account to a designated recipient. This is analogous to the `rescue` function fix applied in the reference report. The function should validate that the amount does not exceed the surplus (i.e., the difference between Aurora's actual NEP-141 balance and the amount accounted for by minted ERC-20 tokens) to prevent draining legitimately bridged funds.

---

### Proof of Concept

1. Deploy a NEP-141 token and register it with Aurora via `deploy_erc20_token`.
2. As an unprivileged user, call `ft_transfer` on the NEP-141 contract:
   ```json
   {
     "receiver_id": "aurora",
     "amount": "1000000",
     "memo": null
   }
   ```
3. Observe that Aurora's NEP-141 balance on the token contract increases by `1000000`.
4. Observe that no ERC-20 tokens are minted for the caller — `ft_on_transfer` was never called.
5. Attempt to recover via `withdrawToNear` on the ERC-20 contract — fails because the caller has zero ERC-20 balance.
6. Confirm no admin function exists in `engine/src/contract_methods/admin.rs` or `engine/src/contract_methods/connector.rs` to issue an outbound `ft_transfer` on behalf of the engine.

The `1000000` NEP-141 tokens are permanently frozen in Aurora's account with no protocol-level recovery path. [5](#0-4) [6](#0-5)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
