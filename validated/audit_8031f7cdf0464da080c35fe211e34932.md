### Title
`setPoolAdminFeeDestination` Does Not Flush Accumulated Fees to the Old Destination Before Rotating, Allowing Pool Admin to Redirect Already-Earned Admin Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory.setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees` at the old destination. Every other fee-parameter mutator (`setPoolAdminFees`, `setPoolProtocolFee`) flushes accumulated fees before the change. The missing flush lets the pool admin silently redirect all fees that accrued under the previous destination to a new address.

### Finding Description
The factory enforces a consistent "collect-before-change" pattern for every fee-rate mutation:

`setPoolAdminFees` (pool admin path): [1](#0-0) 

`setPoolProtocolFee` (owner path): [2](#0-1) 

Both functions call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before writing any new value, ensuring that all surplus (spread fees) and `notionalFeeToken{0,1}Scaled` that accrued under the old configuration are disbursed to the correct recipient first.

`setPoolAdminFeeDestination` breaks this invariant: [3](#0-2) 

It overwrites `poolAdminFeeDestination[pool]` with no prior flush. The two fee streams that are already sitting on the pool at the moment of the call are:

1. **Notional fees** — tracked explicitly in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`.
2. **Spread fees** — the implicit surplus `balance * SCALE - binTotals.scaled - notionalFee`.

Both are computed and disbursed only when `collectFees` is next called: [4](#0-3) 

After `setPoolAdminFeeDestination` returns, the next `collectPoolFees` call (permissionless) will send the entire accumulated admin share — including everything that accrued while the old destination was active — to the **new** destination.

### Impact Explanation
The old `adminFeeDestination` (which may be a separate treasury, DAO, or revenue-sharing contract distinct from the pool admin key) permanently loses all admin fees that accrued before the destination was changed. The pool admin can exploit this deliberately:

1. Let fees accumulate with destination = treasury A.
2. Call `setPoolAdminFeeDestination(pool, attackerAddress)` — no collection triggered.
3. Call `collectPoolFees(pool)` — all accumulated admin fees (both notional and spread) flow to `attackerAddress`.
4. Optionally restore destination to A.

This is a direct loss of owed protocol/admin fees for the original destination, matching the "direct loss of protocol fees" criterion in the allowed impact gate.

### Likelihood Explanation
The pool admin is semi-trusted and has a single-step, unrestricted call path to `setPoolAdminFeeDestination`. No timelock, no cap, no two-step confirmation is required. The asymmetry with `setPoolAdminFees` (which does flush) makes this easy to trigger either maliciously or by operational mistake (e.g., changing destination before a scheduled collection run).

### Recommendation
Mirror the collect-before-change pattern used by `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before rotating
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // <-- old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

```
State before attack:
  poolAdminFeeDestination[pool] = treasury          // legitimate recipient
  notionalFeeToken0Scaled       = X (accrued fees)
  surplus0Scaled                = Y (accrued spread)

Step 1 — pool admin calls:
  factory.setPoolAdminFeeDestination(pool, attacker)
  // No collectFees called; X and Y still sit on the pool

Step 2 — anyone calls:
  factory.collectPoolFees(pool)
  // collectFees(…, adminFeeDestination_ = attacker)
  // admin share of X + Y transferred to attacker

Result:
  treasury receives 0 of the fees it earned
  attacker receives the full admin share of X + Y
```

The root cause is structurally identical to the referenced external bug: a state variable (`poolAdminFeeDestination`) that governs how accumulated value is disbursed is mutated **before** the accumulated value is settled under the old state, causing the old beneficiary to lose their owed amount.
