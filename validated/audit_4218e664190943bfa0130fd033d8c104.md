### Title
Missing Operational Address Consent Check in `stake()` Bypasses the Eligibility Mechanism - (File: src/staking/staking.cairo)

### Summary
The `stake()` function in `src/staking/staking.cairo` allows a staker to register any unused address as their `operational_address` without that address having declared consent. The protocol enforces a two-step consent mechanism (`declare_operational_address` → `change_operational_address`) for all subsequent operational address changes, but the initial `stake()` call entirely skips the eligibility check, creating an inconsistency that can be exploited for griefing.

### Finding Description
The protocol implements a two-step consent flow for operational address assignment:

1. **`declare_operational_address(staker_address)`** — the intended operational address calls this to declare consent for a specific staker. It writes `eligible_operational_addresses[caller] = staker_address`.
2. **`change_operational_address(operational_address)`** — the staker calls this to finalize the change. It asserts `eligible_operational_addresses[operational_address] == staker_address`.

`change_operational_address` enforces the consent check at line 683–687:

```cairo
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address,
    "{}",
    Error::OPERATIONAL_NOT_ELIGIBLE,
);
```

However, the initial `stake()` function (lines 288–366) performs no equivalent check. It only verifies that the `operational_address` is not already in use:

```cairo
assert!(
    self.operational_address_to_staker_address.read(operational_address).is_zero(),
    "{}",
    Error::OPERATIONAL_EXISTS,
);
```

There is no check against `eligible_operational_addresses`. The `operational_address` is then written directly into the staker's record and the reverse mapping:

```cairo
self.operational_address_to_staker_address.write(operational_address, staker_address);
```

This means any staker can claim any currently-unused address as their operational address without that address's consent, bypassing the mechanism that `change_operational_address` enforces.

### Impact Explanation
**Medium — Griefing with no profit motive but damage to users or protocol.**

An attacker can front-run or pre-emptively call `stake()` with a victim's intended operational address. Once claimed, `operational_address_to_staker_address[victim_op_addr]` is non-zero, so:
- The victim cannot use that address as their operational address (blocked by `OPERATIONAL_EXISTS` in both `stake()` and `change_operational_address()`).
- The victim cannot reclaim it via `declare_operational_address` either, because `declare_operational_address` also checks `OPERATIONAL_IN_USE` (line 708–712).
- The attacker's staker will not receive attestation rewards (since the victim won't call `attest()` on the attacker's behalf), but the attacker's goal is denial-of-service, not profit.

The victim is forced to use a different operational address, disrupting their planned setup and potentially their node infrastructure.

### Likelihood Explanation
**Medium.** Any unprivileged account that can call `stake()` (i.e., any address with sufficient STRK and approval) can execute this attack. No privileged access is required. The attacker only needs to know the target's intended operational address (e.g., from a pending mempool transaction or a publicly announced node address) and front-run or pre-empt the victim's `stake()` call.

### Recommendation
Add an eligibility check in `stake()` analogous to the one in `change_operational_address()`. Before writing the operational address, verify that the operational address has declared consent for the calling staker:

```cairo
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address
        || operational_address == staker_address,
    "{}",
    Error::OPERATIONAL_NOT_ELIGIBLE,
);
```

Alternatively, if self-assignment (staker == operational address) is a valid use case, document it explicitly and only require the eligibility check when `operational_address != staker_address`.

### Proof of Concept

1. Victim (`Bob`) intends to call `stake(reward_addr, bob_op_addr, amount)`.
2. Attacker (`Alice`) observes Bob's intended `bob_op_addr` (e.g., from a pending transaction or public announcement).
3. Alice calls `stake(alice_reward_addr, bob_op_addr, min_stake)` first.
   - `operational_address_to_staker_address[bob_op_addr]` is zero → passes `OPERATIONAL_EXISTS` check.
   - No `eligible_operational_addresses` check is performed.
   - `operational_address_to_staker_address[bob_op_addr] = alice_staker_addr` is written.
4. Bob's subsequent `stake(reward_addr, bob_op_addr, amount)` reverts with `OPERATIONAL_EXISTS`.
5. Bob also cannot reclaim `bob_op_addr` via `declare_operational_address` because it is now `OPERATIONAL_IN_USE`.
6. Bob is forced to use a different operational address.
