### Title
Missing Predecessor Authorization in `action_deploy_global_contract` with `AccountId` Mode Allows Any Contract to Overwrite a Victim Account's Global Contract - (File: `runtime/runtime/src/global_contracts.rs`)

### Summary

`action_deploy_global_contract` in `runtime/runtime/src/global_contracts.rs` contains no check that the action is being executed by the account owner (i.e., that `predecessor_id == receiver_id`). When `deploy_mode == GlobalContractDeployMode::AccountId`, the global contract is stored under the **receiver's** account ID. Because any contract can create a cross-contract promise to any target account and attach a `DeployGlobalContract(AccountId)` action, an unprivileged attacker can overwrite the global contract stored under any victim account's ID. Every account that has previously called `UseGlobalContract(AccountId(victim))` — and therefore holds `AccountContract::GlobalByAccount(victim)` — will immediately execute the attacker's code on the next call.

### Finding Description

The host function `promise_batch_action_deploy_global_contract_by_account_id` (exposed in `runtime/near-vm-runner/src/logic/logic.rs`) appends a `DeployGlobalContract(AccountId)` action to any promise index, including promises targeting accounts other than the current contract. The only check performed is a code-size limit. [1](#0-0) 

The action is then appended to the receipt without any restriction on the target account: [2](#0-1) 

When the resulting action receipt is executed on the victim's account, `action_deploy_global_contract` is called. It deducts storage cost from the **receiver's** balance and calls `initiate_distribution` with `account_id = receiver_id`: [3](#0-2) 

Inside `initiate_distribution`, when `deploy_mode == AccountId`, the global contract identifier is set to `GlobalContractIdentifier::AccountId(account_id.clone())` — i.e., the victim's account ID — with no check that the caller is the owner: [4](#0-3) 

The action validation function `validate_deploy_global_contract_action` only checks code size: [5](#0-4) 

And `validate_actions_with_mode` applies no predecessor-vs-receiver restriction for `DeployGlobalContract`: [6](#0-5) 

The documentation for `GlobalContractDeployMode::AccountId` explicitly states: *"This allows the **owner** to update the contract for all its users."* — but no ownership invariant is enforced anywhere in the execution path. [7](#0-6) 

### Impact Explanation

**Corrupted value**: `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(victim_account_id) }` is overwritten with attacker-supplied Wasm bytecode.

Every account that previously called `UseGlobalContract(AccountId(victim))` stores `AccountContract::GlobalByAccount(victim_account_id)` in its account state: [8](#0-7) 

Because the reference is by account ID (not by code hash), those accounts resolve the contract at call time from the overwritten trie key. After the attack, all such accounts execute the attacker's Wasm on every subsequent function call — enabling arbitrary state manipulation, fund theft, or denial of service across all users of the victim's global contract. The victim's account also loses the storage deposit (paid involuntarily).

### Likelihood Explanation

The attack requires only:
1. An attacker-controlled account with a deployed contract.
2. The victim account has enough balance to cover `global_contract_storage_amount_per_byte * code_len`. A small malicious contract (e.g., a few hundred bytes) costs negligible storage, making this feasible against virtually any funded account.
3. The attacker calls their contract, which calls `promise_batch_create(victim)` then `promise_batch_action_deploy_global_contract_by_account_id(...)`.

No privileged role, validator access, or special permission is required. The attack is fully reachable by any unprivileged user submitting a standard signed transaction.

### Recommendation

In `action_deploy_global_contract`, add a predecessor check when `deploy_mode == GlobalContractDeployMode::AccountId`. The action should only be permitted when the predecessor is the receiver (i.e., the account is acting on itself), mirroring the ownership semantics stated in the documentation. Alternatively, restrict `DeployGlobalContract(AccountId)` to transactions only (not cross-contract receipts) by adding a `ValidateReceiptMode`-based guard in `validate_action_with_mode`.

### Proof of Concept

```
1. Attacker deploys `attacker.near` with a contract containing:
   fn exploit() {
       let p = promise_batch_create("victim.near");
       promise_batch_action_deploy_global_contract_by_account_id(
           p, MALICIOUS_WASM.len(), MALICIOUS_WASM.ptr()
       );
   }

2. Attacker calls `attacker.near::exploit()`.

3. Runtime creates an ActionReceipt:
     predecessor_id = "attacker.near"
     receiver_id    = "victim.near"
     actions        = [DeployGlobalContract { code: MALICIOUS_WASM, deploy_mode: AccountId }]

4. Receipt executes on "victim.near":
   - action_deploy_global_contract is called with account_id = "victim.near"
   - Storage cost deducted from victim.near's balance
   - initiate_distribution stores MALICIOUS_WASM under
     TrieKey::GlobalContractCode { identifier: AccountId("victim.near") }

5. All accounts holding AccountContract::GlobalByAccount("victim.near")
   now execute MALICIOUS_WASM on every subsequent call.
``` [3](#0-2) [4](#0-3)

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}

fn initiate_distribution(
    state_update: &mut TrieUpdate,
    account_id: AccountId,
    contract_code: Arc<[u8]>,
    deploy_mode: &GlobalContractDeployMode,
    current_shard_id: ShardId,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))
        }
        GlobalContractDeployMode::AccountId => {
            GlobalContractIdentifier::AccountId(account_id.clone())
        }
    };
    // Increment the nonce and write it to state immediately to prevent multiple
    // distributions with the same nonce from being initiated. This requires
    // allowing the same nonce in the freshness check when applying the
    // distribution receipt.
    let nonce = increment_nonce(state_update, &id)?;
    let distribution_receipt =
        GlobalContractDistributionReceipt::new(id, current_shard_id, vec![], contract_code, nonce);
    let distribution_receipts =
        Receipt::new_global_contract_distribution(account_id, distribution_receipt);
    // No need to set receipt_id here, it will be generated as part of apply_action_receipt
    result.new_receipts.push(distribution_receipts);
    Ok(())
}

/// Validates `DeployGlobalContractAction`. Checks that the given contract size doesn't exceed the limit.
fn validate_deploy_global_contract_action(
    limit_config: &LimitConfig,
    action: &DeployGlobalContractAction,
) -> Result<(), ActionsValidationError> {
    if action.code.len() as u64 > limit_config.max_contract_size {
        return Err(ActionsValidationError::ContractSizeExceeded {
            size: action.code.len() as u64,
            limit: limit_config.max_contract_size,

    }

    Ok(())
}
