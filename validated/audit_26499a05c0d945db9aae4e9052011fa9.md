### Title
Silent `continue` in `IbcTransferCoins` after `SendCoins` allows draining bridge contract bank balance without IBC transfer — (`x/cronos/keeper/ibc.go` + `x/cronos/keeper/evmhandlers/send_cro_to_ibc.go`)

### Summary

`SendCroToIbcHandler.Handle` unconditionally transfers EVM-denom CRO from the bridge contract's Cosmos bank account to the `sender` address **before** calling `IbcTransferCoins`. When the amount is below `10^10` (the 18→8 decimal truncation threshold), `IbcTransferCoins` silently `continue`s and returns `nil` — no burn, no IBC packet. `Handle` sees no error and returns `nil`. The net result is that EVM-denom CRO leaves the bridge contract's bank account and lands in the caller-controlled `sender` address with no corresponding IBC transfer.

### Finding Description

**`IbcTransferCoins` — `x/cronos/keeper/ibc.go` lines 97–102:**

```go
amount8decRem := c.Amount.Mod(sdkmath.NewIntFromBigInt(types.TenPowTen))
amountToBurn := c.Amount.Sub(amount8decRem)
if amountToBurn.IsZero() {
    // Amount too small

``` [2](#0-1) 

For any `amount` in `[1, 10^10 - 1]`:
- `amount8decRem = amount` (since `amount < 10^10`)
- `amountToBurn = amount - amount = 0`
- `amountToBurn.IsZero()` → `true` → `continue`
- `IbcTransferCoins` returns `nil`

`Handle` interprets `nil` as success and returns `nil`. The `bankKeeper.SendCoins` at line 96 has already executed and is **not rolled back**.

The existing test suite explicitly documents and accepts this silent-skip behavior:

```go

    ...

    func() {},

``` [3](#0-2) 

The test was written for `IbcTransferCoins` in isolation, where the silent skip is arguably acceptable. But in `Handle`, the `SendCoins` that precedes it is **not** guarded by the same threshold check, creating the accounting gap.

### Impact Explanation

An attacker who can call an authorized bridge contract (one listed in `CroBridgeContractAddresses`) with `amount = 1` (or any value `< 10^10`) causes:

1. `bankKeeper.SendCoins(contractAddr → attacker, 1 EVM-denom)` — executes and commits.
2. `IbcTransferCoins(attacker, recipient, 1 EVM-denom)` — silently no-ops, returns `nil`.
3. Transaction succeeds. Attacker holds 1 EVM-denom CRO. Bridge contract lost 1 EVM-denom CRO. No IBC packet was created.

Repeated calls drain the bridge contract's entire EVM-denom CRO bank balance. This is an **unauthorized balance change** for a bridge-controlled asset — Critical impact per the scope rules.

### Likelihood Explanation

Preconditions are the normal operating state of the bridge:
- The bridge contract is in `CroBridgeContractAddresses` (it must be, to be functional).
- The bridge contract holds EVM-denom CRO in its Cosmos bank account (it must, to service transfers).
- The bridge contract exposes a callable function that emits `__CronosSendCroToIbc` with caller-supplied `sender` and `amount` — standard design for a user-facing bridge.

No privileged access, no key compromise, no governance action is required by the attacker. Any unprivileged EVM caller can trigger this.

### Recommendation

Add a minimum-amount guard in `Handle` **before** `SendCoins`, mirroring the threshold used in `IbcTransferCoins`:

```go
// Reject amounts that would be silently dropped by IbcTransferCoins
if amount.LT(sdkmath.NewIntFromBigInt(types.TenPowTen)) {
    return fmt.Errorf("amount %s is below the minimum IBC transfer threshold", amount)
}
```

Alternatively, restructure `IbcTransferCoins` to return a sentinel error (rather than `continue`) when the amount is below threshold, so callers can detect and handle the no-op case explicitly.

    // Bridge lost 1 EVM-denom CRO with no IBC packet created
}
