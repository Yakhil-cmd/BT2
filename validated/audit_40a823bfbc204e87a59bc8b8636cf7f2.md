### Title
Missing Access Control on `update_rewards` Enables Transaction-Order-Dependent Permanent Freezing of Staker Yield — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract uses a **global** `last_reward_block` counter to enforce a single reward distribution per block. The spec designates this function as "Only starkware sequencer," but no on-chain caller restriction is implemented — only the generic `general_prerequisites()` pause check is present. Any unprivileged address can call `update_rewards` with an arbitrary `staker_address`, consuming the block's single reward slot. If a malicious actor's call lands before the sequencer's intended call in the same block, the sequencer's call reverts with `REWARDS_ALREADY_UPDATED` and the targeted staker's yield for that block is permanently lost.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` enforces a global, monotonically-advancing deduplication guard:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
// ...
self.last_reward_block.write(current_block_number);
``` [1](#0-0) [2](#0-1) 

`last_reward_block` is a **contract-wide** (not per-staker) state variable. Exactly one `update_rewards` call can succeed per block; every subsequent call in the same block reverts. The spec explicitly states the access control is "Only starkware sequencer": [3](#0-2) 

However, the implementation contains **no on-chain caller check**. The only guard is `self.general_prerequisites()`, which is also called by fully public functions such as `stake`, `increase_stake`, and `claim_rewards`, confirming it only checks the pause flag: [4](#0-3) 

By contrast, the analogous attestation-path entry point explicitly enforces its caller restriction: [5](#0-4) 

Because `update_rewards` accepts a caller-supplied `staker_address` and writes `last_reward_block` unconditionally before returning, any address can race the sequencer and consume the block's single reward slot for an arbitrary staker.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

When the attacker's `update_rewards(staker_A, ...)` call is included before the sequencer's intended `update_rewards(staker_B, ...)` call in the same block:

1. `last_reward_block` is set to the current block by the attacker's call.
2. The sequencer's call for `staker_B` reverts with `REWARDS_ALREADY_UPDATED`.
3. There is no retroactive mechanism to recover the missed block reward for `staker_B`; the block reward slot is permanently consumed.

Repeated across many blocks, this permanently freezes an unbounded amount of unclaimed yield for targeted stakers. If the attacker is themselves a staker, they additionally receive their own proportional block rewards on every block they front-run, compounding the harm.

---

### Likelihood Explanation

**Medium.** The function is publicly callable with no role check. An attacker needs only to submit a valid `update_rewards` transaction targeting any active staker before the sequencer's transaction lands in the same block. No special privilege, leaked key, or external dependency is required — only the ability to submit Starknet transactions, which is available to any address.

---

### Recommendation

Add an on-chain caller restriction to `update_rewards` matching the spec's intent. The simplest approach mirrors the attestation path: introduce a dedicated sequencer role and assert it at the top of `update_rewards`:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this
    // ...
}
```

Alternatively, store the authorized sequencer address in contract storage (similar to `attestation_contract`) and assert `get_caller_address() == self.sequencer_address.read()`.

---

### Proof of Concept

1. Staker B is a legitimate staker. The sequencer prepares `update_rewards(staker_B, false)` for block N.
2. Attacker (any address, including staker A) submits `update_rewards(staker_A, false)` targeting block N.
3. Attacker's transaction is included first in block N; `last_reward_block` is written to N; staker A receives their proportional block reward.
4. Sequencer's `update_rewards(staker_B, false)` is included second in block N; the assertion `current_block_number > last_reward_block` fails (`N > N` is false); the call reverts with `REWARDS_ALREADY_UPDATED`. [1](#0-0) 

5. Staker B's yield for block N is permanently lost. The attacker can repeat this every block.
