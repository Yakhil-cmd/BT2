### Title
Permanent Fund Freeze on Failed ERC-20 Exit-to-NEAR When `error_refund` Feature Is Disabled - (File: `engine-precompiles/src/native.rs`)

### Summary
When a user calls `withdrawToNear` on a bridged ERC-20 token, the ERC-20 tokens are burned immediately inside the EVM. A NEAR promise is then scheduled to transfer the backing NEP-141 tokens to the recipient. If that promise fails (e.g., recipient account not registered with the NEP-141 contract), and the `error_refund` compile-time feature is not enabled, the callback unconditionally skips the refund path. The burned ERC-20 tokens are never re-minted and the NEP-141 tokens remain permanently locked inside Aurora's account — a permanent fund freeze.

### Finding Description

The `ExitToNear` precompile's `run` method constructs `ExitToNearPrecompileCallbackArgs` with a compile-time conditional:

```rust
// engine-precompiles/src/native.rs lines 449-455
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None in production build
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled (the opt-in default), `refund` is always `None`. The callback `exit_to_near_precompile_callback` then handles a failed promise as:

```rust
// engine/src/contract_methods/connector.rs lines 231-242
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op: no refund, no error
};
``` [2](#0-1) 

The `refund_on_error` function, which would re-mint the burned ERC-20 tokens, is never reached: [3](#0-2) 

The ERC-20 burn happens inside `EvmErc20V2.withdrawToNear` before the precompile is called:

```solidity
// etc/eth-contracts/contracts/EvmErc20V2.sol lines 53-58
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);   // ← tokens destroyed here, before any promise
    ...
}
``` [4](#0-3) 

The test suite explicitly documents this loss as expected behavior when the feature is absent:

```rust
// engine-tests/src/tests/erc20_connector.rs lines 656-665
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

### Impact Explanation

**Permanent freezing of funds.** When the NEAR-side `ft_transfer` promise fails, the ERC-20 tokens are already burned and are never re-minted. The corresponding NEP-141 tokens remain locked in Aurora's account with no mechanism to recover them. The user suffers a total, irreversible loss of the exited amount.

### Likelihood Explanation

The trigger condition — a failed `ft_transfer` — is easily reached by any token holder who attempts to exit to a NEAR account that has not called `storage_deposit` on the NEP-141 contract. This is a common user mistake. The `error_refund` feature is opt-in and not a default feature, making the vulnerable code path the production default. [6](#0-5) 

### Recommendation

Enable the `error_refund` feature in the production build, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` regardless of the feature flag. The refund logic in `engine::refund_on_error` already exists and is correct; it simply needs to be reachable on every failed exit.

### Proof of Concept

1. Alice holds 100 units of a bridged ERC-20 token on Aurora (backed 1:1 by NEP-141 tokens held by Aurora's account).
2. Alice calls `erc20.withdrawToNear("alice-unregistered.near", 100)`.
3. `EvmErc20V2._burn(alice, 100)` executes — Alice's ERC-20 balance drops to 0.
4. The exit precompile schedules `ft_transfer(receiver_id: "alice-unregistered.near", amount: 100)` on the NEP-141 contract with `callback_args.refund = None`.
5. The NEP-141 `ft_transfer` fails because `alice-unregistered.near` has no storage deposit.
6. `exit_to_near_precompile_callback` fires; `args.refund` is `None`; the `else { None }` branch executes silently.
7. Alice's ERC-20 balance remains 0. The NEP-141 tokens remain in Aurora's account. Alice has permanently lost 100 tokens.
