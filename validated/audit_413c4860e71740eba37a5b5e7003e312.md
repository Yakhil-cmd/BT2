### Title
Stale `pendingPriceProvider` Survives Pool Admin Transfer, Allowing New Admin to Unknowingly Install a Bad Oracle - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`acceptPoolAdmin()` in `MetricOmmPoolFactory` transfers the `poolAdmin[pool]` role to a new address but does not clear `pendingPriceProvider[pool]` or `pendingPriceProviderExecuteAfter[pool]`. A price-provider proposal queued by the previous admin persists across the handover and can be finalized by the incoming admin — who may be unaware of it — installing an oracle the new admin never vetted.

### Finding Description

The two-step admin transfer flow is:

1. Current admin calls `proposePoolAdminTransfer(pool, newAdmin)` → sets `pendingPoolAdmin[pool] = newAdmin`.
2. New admin calls `acceptPoolAdmin(pool)` → sets `poolAdmin[pool] = newAdmin` and deletes `pendingPoolAdmin[pool]`.

`acceptPoolAdmin` only touches `poolAdmin` and `pendingPoolAdmin`:

```solidity
// MetricOmmPoolFactory.sol lines 518-526
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

It does **not** clear `pendingPriceProvider[pool]` or `pendingPriceProviderExecuteAfter[pool]`. [1](#0-0) 

Meanwhile, `executePoolPriceProviderUpdate` only checks that the caller is the **current** pool admin — it has no way to know whether the pending proposal was authored by the current admin or a previous one:

```solidity
// MetricOmmPoolFactory.sol lines 494-507
function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    ...
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
}
``` [2](#0-1) 

The only validation on the pending provider is a token-pair match (`_validatePriceProvider`), which does not prevent a provider that returns stale, manipulated, or otherwise bad prices from being installed. [3](#0-2) 

### Impact Explanation

Every swap in `MetricOmmPool` reads bid/ask prices exclusively from the active price provider. If a bad provider is installed, all subsequent swaps execute at wrong prices: traders receive more than the oracle permits (pool insolvency) or less (trader loss). This directly matches the "bad-price execution" and "swap conservation failure" impact gates.

The pool's oracle is the single source of truth for pricing; there is no on-chain fallback or circuit-breaker that detects a provider returning wrong prices.

### Likelihood Explanation

The scenario requires:

1. A pool with a **mutable** oracle (`priceProviderTimelock < type(uint256).max`).
2. The outgoing admin having an **in-flight** `proposePoolPriceProvider` call (pending but not yet executed).
3. The admin transfer completing while the proposal is still pending.
4. The incoming admin calling `executePoolPriceProviderUpdate` without first auditing `pendingPriceProvider(pool)`.

Condition 4 is realistic during operational handovers: the incoming admin may be told "there is a pending oracle rotation, please finalize it after the timelock" without being given the specific address to verify. The old admin cannot cancel the proposal after losing the admin role (cancellation also requires `onlyPoolAdmin`), so the stale proposal is permanently live until the new admin either overwrites it with a fresh `proposePoolPriceProvider` call or executes it. [4](#0-3) 

### Recommendation

In `acceptPoolAdmin`, clear any in-flight price-provider proposal alongside the admin handover:

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    // Clear stale price-provider proposal from previous admin
    if (pendingPriceProvider[pool] != address(0)) {
        delete pendingPriceProvider[pool];
        delete pendingPriceProviderExecuteAfter[pool];
    }
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

This mirrors the fix recommended in M-15: whenever ownership changes, all privileges and pending actions set by the previous owner must be reset.

### Proof of Concept

```
1. Factory deploys pool with mutable oracle (priceProviderTimelock = 7 days).
   poolAdmin[pool] = adminA.

2. adminA deploys `BadPriceProvider` — valid token0/token1 but returns
   manipulated bid/ask prices.

3. adminA calls proposePoolPriceProvider(pool, BadPriceProvider).
   → pendingPriceProvider[pool] = BadPriceProvider
   → pendingPriceProviderExecuteAfter[pool] = block.timestamp + 7 days

4. adminA calls proposePoolAdminTransfer(pool, adminB).
   adminB calls acceptPoolAdmin(pool).
   → poolAdmin[pool] = adminB
   → pendingPriceProvider[pool] still = BadPriceProvider  ← NOT CLEARED

5. 7 days pass. adminB (unaware of the stale proposal) calls
   executePoolPriceProviderUpdate(pool).
   → onlyPoolAdmin passes (adminB is current admin)
   → _validatePriceProvider passes (correct token pair)
   → pool.setPriceProvider(BadPriceProvider) executes

6. All subsequent swaps price against BadPriceProvider.
   Traders receive incorrect amounts; pool balances diverge from LP claims.
``` [5](#0-4)
