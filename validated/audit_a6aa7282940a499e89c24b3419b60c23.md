### Title
Missing Lower-Bound Threshold Validation in FROST Signing Initialization Allows Corrupt Sign Outputs - (File: src/frost/mod.rs)

### Summary

The `assert_sign_inputs` function and the `presign` entry point in `src/frost/mod.rs` accept a `threshold` parameter without enforcing the minimum value of 2 that the DKG layer mandates. A malicious coordinator or misconfigured caller can supply `threshold = 0` or `threshold = 1` to the signing phase. Because the shares were generated under a polynomial of degree `threshold_keygen − 1 ≥ 1`, using a mismatched lower threshold during Lagrange reconstruction produces a cryptographically wrong aggregate, causing every produced signature to fail verification and permanently denying signing to all honest participants in that session.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces a hard lower bound:

```rust
// src/dkg.rs  lines 580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The signing-phase validator `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs  lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. The same omission exists in the FROST `presign` entry point:

```rust
// src/frost/mod.rs  lines 71-77
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: args.threshold.into(),
        max: participants.len(),
    });
}
``` [3](#0-2) 

Neither function calls `assert_key_invariants`, so the minimum-threshold invariant established at key-generation time is never re-checked at the signing entry point. [4](#0-3) 

---

### Impact Explanation

FROST signing uses Lagrange interpolation over the participant set to reconstruct the aggregate nonce and signature scalar. The degree of the underlying secret polynomial is `threshold_keygen − 1`. If `threshold_sign = 1` is supplied, the library computes degree-0 Lagrange coefficients (each participant's coefficient becomes 1), which is mathematically inconsistent with the degree-1 shares produced during DKG. The resulting aggregate scalar does not equal the true secret, so the final signature fails `sig.verify(&public_key, &msg_hash)`. Every signing attempt in the session produces an unverifiable signature, permanently denying signing to all honest participants.

If `threshold_sign = 0` is supplied, `threshold.value() > participants.len()` evaluates to `0 > N`, which is false, so no error is returned; downstream Lagrange arithmetic on an empty or zero-degree basis will either panic or produce a degenerate result, again permanently breaking signing.

Matched allowed impact: **High — Corruption of sign outputs / Permanent denial of signing for honest parties under valid protocol inputs.**

---

### Likelihood Explanation

The `threshold` parameter is caller-supplied at every signing invocation. A malicious coordinator who controls the session setup can pass `threshold = 1` to all participants. Because `assert_sign_inputs` returns `Ok` for any value in `[0, participants.len()]`, every participant's local validation passes silently. No cryptographic secret or privileged access is required; only the ability to invoke the public signing API with an out-of-range threshold value.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces to both `assert_sign_inputs` and the `presign` entry point:

```rust
// In assert_sign_inputs and presign
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check at `src/dkg.rs` lines 580–582 and closes the gap between the DKG invariant and the signing entry points. [5](#0-4) 

---

### Proof of Concept

1. Run DKG with `N = 3`, `threshold = 2` — succeeds, each participant holds a degree-1 share.
2. Call `assert_sign_inputs(participants, /*threshold=*/1u8, me, coordinator)` — returns `Ok(participant_list)` with no error.
3. Call `presign(participants, me, &PresignArguments { keygen_out, threshold: 1.into() }, rng)` — returns `Ok(protocol)` with no error.
4. Execute the signing protocol; the coordinator aggregates shares using degree-0 Lagrange coefficients.
5. The produced signature fails `sig.verify(&public_key, &msg_hash)` because the reconstructed scalar is wrong.
6. All participants receive a `ProtocolError::AssertionFailed("signature failed to verify")`, permanently denying signing for this session. [6](#0-5) [4](#0-3)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
