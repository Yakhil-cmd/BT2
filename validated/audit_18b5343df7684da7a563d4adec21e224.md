### Title
COA `supportsInterface` Omits ERC-1271 Despite Implementing `isValidSignature`, Breaking Smart-Wallet Composability - (File: `fvm/evm/handler/coa/coa.sol`)

### Summary
The Flow EVM Cadence-Owned Account (COA) contract implements `isValidSignature` (ERC-1271) but does not declare ERC-1271 support in its `supportsInterface` function. The protocol documentation explicitly advertises ERC-1271 support, yet any EVM protocol that performs an ERC-165 compliance check before invoking `isValidSignature` will receive `false` and reject the COA as a valid smart-wallet signer.

### Finding Description
`fvm/evm/handler/coa/coa.sol` defines the COA contract, which is Flow's native EVM smart-contract wallet. The contract implements `isValidSignature(bytes32,bytes)` (lines 114–125), the sole function of the ERC-1271 interface, and even stores the correct magic return value as a named constant:

```solidity
// bytes4(keccak256("isValidSignature(bytes32,bytes)")
bytes4 constant internal ValidERC1271Signature = 0x1626ba7e;
```

Because ERC-1271 contains exactly one function, its ERC-165 interface ID equals that function selector: `0x1626ba7e`.

However, the `supportsInterface` function (lines 67–73) only covers four interfaces:

```solidity
function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId;
}
```

ERC-1271 (`0x1626ba7e`) is absent. This directly contradicts the Cadence-layer documentation in `fvm/evm/stdlib/contract.cdc` (line 474), which explicitly lists ERC-1271 as a supported standard:

> "COAs are smart contract wallets that accept native token transfers and support several ERCs including ERC-165, ERC-721, ERC-777, ERC-1155, **ERC-1271**."

The root cause is identical to the original report: sub-interface/extension functionality is implemented, but the base interface ID is not registered in `supportsInterface`, so ERC-165-based discovery fails.

### Impact Explanation
Any EVM protocol that guards its `isValidSignature` call with an ERC-165 check — a common pattern in DEXes, NFT marketplaces, governance systems, and multi-sig frameworks — will call `supportsInterface(0x1626ba7e)` on the COA, receive `false`, and refuse to treat the COA as a valid smart-wallet signer. This breaks the core use-case of COAs as EVM smart-contract wallets: users cannot authorize transactions or sign messages in ERC-1271-aware protocols using their COA, even though the underlying `isValidSignature` logic is fully functional and correct.

### Likelihood Explanation
ERC-1271 smart-wallet signature verification is a standard integration pattern across the EVM ecosystem. Protocols such as OpenSea (Seaport), Unipermit, Gnosis Safe integrations, and many governance frameworks perform the ERC-165 check before calling `isValidSignature`. Every COA deployed on Flow EVM is affected because the COA bytecode is fixed at bootstrap time. Any unprivileged user who creates a COA and attempts to use it as a signer in such a protocol will encounter the failure without any configuration error on their part.

### Recommendation
Add the ERC-1271 interface ID to `supportsInterface`. Since no `IERC1271` interface is currently declared in `coa.sol`, the check should use the hard-coded selector (matching the existing constant) or a newly declared interface:

```solidity
interface IERC1271 {
    function isValidSignature(bytes32 hash, bytes memory signature)
        external view returns (bytes4 magicValue);
}

function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId        ||
        id == type(IERC1271).interfaceId;       // 0x1626ba7e
}
```

### Proof of Concept
The following Solidity snippet demonstrates the discrepancy. Deploy the COA bytecode and call `doesSupportERC1271`; it returns `false` despite `isValidSignature` being present and functional.

```solidity
interface IERC165 {
    function supportsInterface(bytes4 id) external view returns (bool);
}

contract TestCOACompliance {
    // ERC-1271 interface ID = bytes4(keccak256("isValidSignature(bytes32,bytes)")) = 0x1626ba7e
    function doesSupportERC1271(IERC165 coa) external view returns (bool) {
        return coa.supportsInterface(0x1626ba7e);  // returns false for COA
    }
    // ERC-165 itself — returns true
    function doesSupportERC165(IERC165 coa) external view returns (bool) {
        return coa.supportsInterface(0x01ffc9a7);
    }
}
```
