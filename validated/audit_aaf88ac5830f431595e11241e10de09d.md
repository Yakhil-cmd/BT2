### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Bypass Oracle Rotation Delay and Drain LP Funds - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory._validatePoolParameters` does not enforce a minimum value for `priceProviderTimelock`. When a pool is created with `priceProviderTimelock = 0`, the pool admin can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same transaction, instantly substituting a malicious oracle with no delay. LPs have zero time to react and withdraw before the pool begins pricing swaps against an attacker-controlled feed, enabling complete drainage of LP principal.

---

### Finding Description

The `priceProviderTimelock` field in `PoolParameters` controls how long the pool admin must wait between proposing and executing an oracle rotation. The factory stores it verbatim:

```solidity
priceProviderTimelock[pool] = params.priceProviderTimelock;
``` [1](#0-0) 

`_validatePoolParameters` validates fees, tokens, admin, and initial amounts — but imposes **no minimum on `priceProviderTimelock`**: [2](#0-1) 

When `priceProviderTimelock[pool] = 0`, `proposePoolPriceProvider` computes:

```solidity
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp + 0 = block.timestamp
``` [3](#0-2) 

`executePoolPriceProviderUpdate` then checks:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp  →  false  →  does NOT revert
``` [4](#0-3) 

Both calls succeed in the same block (or even the same transaction via a multicall), making the timelock protection completely absent. The only validation on the new provider is that it exposes the correct `token0()`/`token1()` pair — trivially satisfied by a malicious contract the pool admin deploys.

---

### Impact Explanation

After atomically rotating to a malicious oracle, the pool admin controls the bid/ask prices returned to every subsequent `swap`. By setting extreme prices (e.g., bid ≈ 0, ask ≈ `type(uint128).max`), the admin can execute swaps that extract all token0 or token1 reserves from the pool at near-zero cost, directly stealing LP principal. The `OracleValueStopLossExtension` guard is optional and not present on pools that do not configure it at creation; even if present, it relies on the same oracle feed and can be bypassed by a sufficiently extreme price move within a single block. [5](#0-4) 

This is a **direct loss of LP principal** — the exact impact class listed as in-scope under "Critical/High direct loss of user principal."

---

### Likelihood Explanation

- `createPool` is permissionless; any actor can deploy a pool with `priceProviderTimelock = 0`.
- The pool admin is explicitly described as "semi-trusted only inside caps and timelocks." A zero timelock removes the only temporal constraint on oracle changes.
- The `PoolCreated` event emits `priceProviderTimelock` so the value is observable on-chain, but LPs have no on-chain protection and no guaranteed exit window.
- The attack requires only two factory calls by the pool admin — no flash loans, no external dependencies. [6](#0-5) 

---

### Recommendation

Enforce a minimum `priceProviderTimelock` in `_validatePoolParameters`. For example:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

This mirrors the remediation in the referenced HTLC report (doubling the minimum timelock) and ensures LPs always have a meaningful window to observe a pending oracle rotation and exit before it takes effect. [2](#0-1)
