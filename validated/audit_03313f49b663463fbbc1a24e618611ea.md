### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace. Because the trace grows by one entry per epoch in which the member makes a balance change, a pool member who makes frequent delegations across many epochs without claiming rewards will eventually cause `claim_rewards` (and the view `pool_member_info_v1`) to exceed the block gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` (called from both `claim_rewards` and `pool_member_info_v1`) iterates over the entire unprocessed portion of the pool member's balance trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace is backed by a `Vec` in storage (`pool_member_epoch_balance`). The underlying `insert` logic in `PoolMemberBalanceTrace` appends a **new** checkpoint whenever the epoch key differs from the last entry's key:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Every call to `set_member_balance` or `increase_member_balance` uses `get_epoch_plus_k()` as the key. Therefore, **one balance-change operation per epoch adds one new storage entry**. The following public entry points all trigger a trace insertion:

- `add_to_delegation_pool` → `increase_member_balance` [3](#0-2) 
- `exit_delegation_pool_intent` → `set_member_balance` [4](#0-3) 
- `enter_delegation_pool_from_staking_contract` → `increase_member_balance` / `set_member_balance` [5](#0-4) 

The `entry_to_claim_from` cursor stored in `pool_member_info` advances only when `claim_rewards` succeeds. If the member never calls `claim_rewards` (or calls it infrequently), the unprocessed window grows without bound. When the member finally calls `claim_rewards`, the loop must traverse every accumulated entry in a single transaction.

`pool_member_info_v1` (a read-only view) also calls `calculate_rewards` with `entry_to_claim_from` starting from the stored checkpoint, so it is equally affected:

```cairo
let (rewards, _) = self
    .calculate_rewards(
        :pool_member,
        from_checkpoint: pool_member_info.reward_checkpoint,
        until_checkpoint: self.get_current_checkpoint(:pool_member),
        entry_to_claim_from: pool_member_info.entry_to_claim_from,
    );
``` [6](#0-5) 

---

### Impact Explanation

Once the trace is large enough that a single `claim_rewards` call exceeds the Starknet block gas limit, the pool member's unclaimed yield is **permanently frozen**: every future attempt to claim will also exceed the gas limit, and there is no alternative code path to drain the rewards. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

An active delegator who calls `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch, without periodically calling `claim_rewards`, will grow the trace at a rate of one entry per epoch. The protocol imposes no cap on trace length and no warning to users. Over a sufficiently long protocol lifetime (or with a short epoch duration), the trace can grow to thousands of entries. Additionally, `add_to_delegation_pool` is callable by the pool member **or their `reward_address`**, so a pool member who sets a contract as their reward address could have the trace grown by that contract's logic without the member's direct awareness. [7](#0-6) 

---

### Recommendation

1. **Enforce a maximum trace length**: Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if the unprocessed trace window (`pool_member_trace_length - entry_to_claim_from`) exceeds a safe constant (e.g., 1 000 entries).
2. **Require periodic reward claims**: Before any balance-changing operation, automatically advance `entry_to_claim_from` (i.e., call an internal `claim_rewards` step) so the unprocessed window never grows large.
3. **Paginated claiming**: Expose a `claim_rewards_partial(max_entries)` entry point so a member with a large trace can drain it over multiple transactions before the window becomes unclaimable.

---

### Proof of Concept

```
1. Pool member Alice enters the pool with the minimum amount.
2. Each epoch, Alice calls `add_to_delegation_pool(1)` (minimum non-zero amount).
   - Each call in a new epoch appends one entry to `pool_member_epoch_balance`.
3. Alice never calls `claim_rewards`.
4. After N epochs, the trace has N unprocessed entries.
5. Alice calls `claim_rewards`.
   - `calculate_rewards` loops N times, each iteration reading from storage
     (`pool_member_trace.at(entry_to_claim_from)`) and calling `find_sigma`.
6. For sufficiently large N, the transaction exceeds the block gas limit and reverts.
7. All subsequent `claim_rewards` calls also revert — Alice's yield is permanently frozen.
```
