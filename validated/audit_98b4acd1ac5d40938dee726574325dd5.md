### Title
Unbounded Loop in `calculate_rewards` Over `pool_member_epoch_balance` Trace Can Permanently Freeze Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` balance-change trace for a pool member without any bound. The codebase itself acknowledges this at lines 857–858 with the comment: *"This loop is unbounded but unlikely to exceed gas limits."* A pool member who repeatedly changes their balance across many distinct epochs without claiming rewards will grow this trace unboundedly, eventually causing `claim_rewards` to exceed the Starknet gas limit and permanently freezing their unclaimed yield.

### Finding Description

Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` invokes `increase_member_balance` / `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`. [1](#0-0) 

The `insert` implementation in the balance trace appends a **new** checkpoint whenever the epoch key differs from the last stored key, and only updates in-place when the key matches. [2](#0-1) 

Therefore, each balance-changing operation executed in a **different epoch** appends one new entry to the trace. Over time, an active delegator who adds or partially exits across many epochs accumulates an ever-growing trace.

When `claim_rewards` is called, it invokes `calculate_rewards`, which loops over every entry in the trace from `entry_to_claim_from` up to the current epoch: [3](#0-2) 

The `entry_to_claim_from` cursor is only advanced upon a **successful** `claim_rewards` call: [4](#0-3) 

If the trace grows large enough between two successful claims, the loop will exceed the gas limit, the transaction will revert, `entry_to_claim_from` will not advance, and every subsequent `claim_rewards` attempt will also revert — permanently locking the pool member's accumulated yield.

The same unbounded loop is also triggered by the read-only `pool_member_info_v1` view: [5](#0-4) 

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the trace is large enough that a single `claim_rewards` call cannot complete within the gas limit, the pool member's rewards are permanently inaccessible. There is no mechanism to prune the trace, split the claim into batches, or otherwise recover. All STRK rewards accrued by that pool member are frozen in the pool contract forever.

### Likelihood Explanation

**Medium.** The scenario requires a pool member to:
1. Make balance changes (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) across many distinct epochs.
2. Defer claiming rewards for an extended period.

Both behaviors are realistic for active delegators who adjust their stake frequently (e.g., partial exit intents followed by re-delegation) and who do not claim rewards every epoch. The number of epochs required to hit the gas limit depends on Starknet's per-transaction gas cap, but given that each balance change in a new epoch costs one storage read per loop iteration (plus a `find_sigma` call), the limit is reachable within a few hundred to a few thousand balance-change operations — a plausible lifetime activity for a long-term delegator.

### Recommendation

1. **Batch claiming with a `max_iterations` parameter**: Allow `claim_rewards` to accept an optional upper bound on the number of trace entries to process per call, storing the updated `entry_to_claim_from` so the caller can resume in a subsequent transaction.
2. **Enforce a maximum trace length**: Cap the number of balance-change checkpoints per pool member (e.g., by requiring a claim before a new checkpoint can be appended once the trace exceeds a threshold).
3. **Periodic compaction**: After a successful `claim_rewards`, truncate all processed entries from the trace (entries before `entry_to_claim_from`) so the loop always starts from a small window.

### Proof of Concept

```
1. Alice delegates to a pool and never calls claim_rewards.
2. Alice calls add_to_delegation_pool once per epoch for N epochs,
   each call in a new epoch appends one entry to pool_member_epoch_balance.
3. After N epochs, Alice calls claim_rewards.
4. calculate_rewards loops over all N entries; for large N the
   transaction runs out of gas and reverts.
5. entry_to_claim_from is never updated; every future claim_rewards
   call also reverts. Alice's accumulated STRK rewards are permanently frozen.
```

The attacker-controlled entry path is entirely unprivileged: `add_to_delegation_pool` and `exit_delegation_pool_intent` are public functions callable by any pool member. [6](#0-5) [7](#0-6)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
