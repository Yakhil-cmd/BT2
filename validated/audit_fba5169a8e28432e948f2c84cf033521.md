### Title
`ExitToNear` Precompile Callback Does Nothing on Failed Transfer When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens on Aurora and schedules an `ft_transfer`/`ft_transfer_call` on the NEAR side. A callback (`exit_to_near_precompile_callback`) is meant to refund the burned tokens if the NEAR-side transfer fails. However, the refund argument is unconditionally set to `None` when the `error_refund` Cargo feature is not compiled in. The callback therefore silently returns `Ok(None)` on failure — executing no refund logic — and the burned tokens are permanently lost. This is structurally identical to the reported bug: the primary handler has the real logic, but the fallback/callback path calls an effectively empty function.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile constructs callback arguments before scheduling the NEAR-side transfer:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,                          // always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent and `transfer_near` is also `None` (the common ERC-20 exit case), `callback_args` equals the default value, so **no callback is scheduled at all** — only the bare `ft_transfer` promise is created:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

When a callback *is* scheduled (e.g., because `transfer_near` is `Some`), `exit_to_near_precompile_callback` handles the failure path:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- always reached when error_refund is disabled
};
``` [3](#0-2) 

Because `args.refund` is always `None` when the feature is absent, the `else { None }` branch is always taken on failure — the callback is a no-op, exactly like the empty `_nonblockingLzReceive` in the reported bug.

The engine test suite explicitly documents this behaviour:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the NEAR-side `ft_transfer` or `ft_transfer_call` fails (e.g., the recipient account is not registered with the NEP-141 contract, or the contract is paused), the ERC-20 tokens have already been burned inside the EVM. Without the `error_refund` feature, no refund is issued. The burned tokens are irrecoverably lost for the user.

---

### Likelihood Explanation

**Medium.** Any EVM user who calls the `ExitToNear` precompile (directly or via an ERC-20 burn function) is exposed if the NEAR-side transfer fails for any reason. Transfer failures are realistic: unregistered recipients, paused NEP-141 contracts, or insufficient storage deposits. The feature flag `error_refund` is a compile-time opt-in, meaning any deployment that omits it silently disables the entire refund path.

---

### Recommendation

1. Make `error_refund` a **default feature** in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml` so it is always compiled into production builds.
2. Alternatively, move the refund-argument construction out from behind the feature flag entirely, making the refund path unconditional.
3. Add an integration test that asserts a refund occurs on ft_transfer failure regardless of feature flags.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora (mint ERC-20 tokens to Alice).
3. Alice calls the `ExitToNear` precompile to send tokens to `unregistered.near` (an account not registered with the NEP-141 contract).
4. The precompile burns Alice's ERC-20 tokens and schedules `ft_transfer` to `unregistered.near`.
5. The `ft_transfer` fails because `unregistered.near` has no storage deposit.
6. No callback is scheduled (or the callback's `refund` field is `None`), so `exit_to_near_precompile_callback` returns `Ok(None)` without minting back the tokens.
7. Alice's ERC-20 balance is zero; the NEP-141 balance of `unregistered.near` is also zero. Tokens are permanently destroyed. [5](#0-4) [6](#0-5) [7](#0-6)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
