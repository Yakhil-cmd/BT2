### Title
Gateway Skips Account Signature Validation for Invoke(nonce=1) When Any Transaction (Not Just `deploy_account`) Exists in the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (account signature verification) for an Invoke transaction with nonce=1 when the on-chain account nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type in the pool, not exclusively `deploy_account` transactions. An attacker who places any legitimate transaction (e.g., Invoke(nonce=0)) from a target address into the mempool can then submit a forged Invoke(nonce=1) with an invalid signature and have it admitted to the mempool without signature verification.

### Finding Description

In `skip_stateful_validations` at `crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461`, the gateway skips the `__validate__` entry point when three conditions hold simultaneously:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The intent is to support the UX pattern of sending `deploy_account + invoke` simultaneously. The code comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

However, `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` if **any** transaction from the address is in the pool — including a legitimate `Invoke(nonce=0)` from an already-deployed account. It does **not** verify that the transaction in the pool is specifically a `deploy_account`.

The skip decision propagates directly into `run_validate_entry_point`, which sets `execution_flags.validate = !skip_validate`: [3](#0-2) 

When `skip_validate = true`, `StatefulValidator::perform_validations` returns `Ok(())` before calling `__validate__`: [4](#0-3) 

The full call chain is:

```
add_tx_inner
  → extract_state_nonce_and_run_validations
      → run_pre_validation_checks → skip_stateful_validations  [returns true]
      → run_validate_entry_point(skip_validate=true)
          → StatefulValidator::validate(account_tx{validate=false})
              → perform_validations → returns Ok(()) without __validate__
``` [5](#0-4) 

The broken invariant: *Every transaction admitted to the mempool must have passed `__validate__` (signature verification) OR the account is provably not yet deployed (deploy_account UX skip).* An already-deployed account with on-chain nonce=0 can have a forged Invoke(nonce=1) admitted to the mempool without signature verification, as long as any transaction from that account is already in the pool.

This is the direct analog of the external bug: `claimPromo()` updates state without checking `status='pending'` (insufficient precondition guard); here, `skip_stateful_validations` skips `__validate__` without verifying the transaction in the pool is specifically a `deploy_account` (insufficient precondition guard).

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

A forged Invoke(nonce=1) with an invalid/arbitrary signature is admitted to the mempool and a transaction hash is returned to the submitter as if the transaction were valid. The transaction will be reverted by the batcher when `__validate__` runs during execution, but the mempool is polluted with invalid transactions, batcher resources are wasted, and the gateway's admission invariant — that every accepted transaction has passed account signature verification — is violated.

### Likelihood Explanation

**Medium.** The preconditions are:
1. A deployed account with on-chain nonce=0 (account deployed but no committed transactions yet — common for newly deployed accounts).
2. Any legitimate transaction from that account in the mempool (trivially arranged by the attacker themselves, or by observing a pending transaction).
3. Submitting a forged Invoke(nonce=1) with an invalid signature.

All three conditions are achievable by an unprivileged external user with no special access.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender address is present in the mempool. Add a dedicated mempool API such as `deploy_account_tx_in_pool(address)` that inspects transaction types, and use that in `skip_stateful_validations` instead of the generic account-presence check.

Alternatively, restrict the skip to only apply when the account has no deployed code on-chain (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which is the true precondition for the UX skip to be safe.

### Proof of Concept

1. Deploy account `A` (e.g., via a factory or direct deployment in a prior block). Account `A` has on-chain nonce=0.
2. Submit a legitimate `Invoke(nonce=0)` from account `A` to the gateway. It passes all validation including `__validate__` and is admitted to the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Construct a forged `Invoke(nonce=1)` from account `A` with an invalid/arbitrary signature (e.g., `signature = [0x1337]`).
4. Submit the forged transaction to the gateway.
5. In `skip_stateful_validations`: `tx.nonce() == 1` ✓, `account_nonce == 0` ✓ → enters the skip branch.
6. `account_tx_in_pool_or_recent_block(A)` → `true` (the nonce-0 invoke is in the pool).
7. `skip_stateful_validations` returns `true` → `run_validate_entry_point` is called with `skip_validate=true`.
8. `execution_flags.validate = false` → `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`.
9. The forged `Invoke(nonce=1)` is admitted to the mempool and a transaction hash is returned. The invalid signature is never checked at the gateway. [6](#0-5) [7](#0-6)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
