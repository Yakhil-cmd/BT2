### Title
Hardcoded 2,300-Gas Stipend in `DepositCallGasLimit` Permanently Blocks FLOW Token Deposits to EVM Smart Contracts with Non-Trivial Receive/Fallback Functions - (File: `fvm/evm/types/call.go`)

---

### Summary

`DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` are hardcoded to `21_000 + 2_300 = 23_300` gas. After the 21,000-gas EVM intrinsic cost, only 2,300 gas remains for the recipient EVM contract's `receive()` or `fallback()` function — the exact same stipend as Solidity's deprecated `transfer()`. Any EVM smart contract on Flow EVM whose `receive()` or `fallback()` consumes more than 2,300 gas will permanently fail to receive FLOW tokens via the Cadence-EVM bridge deposit path.

---

### Finding Description

In `fvm/evm/types/call.go`, the gas constants are defined as:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas          // 21,000

// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23,300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer   // 23,300
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer   // 23,300
``` [1](#0-0) 

The code comment explicitly acknowledges that 2,300 is the "max gas allowed for receive/fallback methods" — the same fixed stipend that Solidity's `transfer()` forwards and that the Istanbul hardfork broke.

`NewDepositCall` is constructed with this fixed `DepositCallGasLimit`: [2](#0-1) 

This call is issued unconditionally inside `Account.Deposit()` in the EVM handler: [3](#0-2) 

`Account.Deposit()` is the Go-layer implementation invoked whenever a Cadence transaction calls `EVMAddress.deposit(from: <-vault)` (the public Cadence entry point): [4](#0-3) 

After the EVM charges 21,000 gas for the intrinsic transaction cost, only 2,300 gas is left for the recipient contract's `receive()` or `fallback()`. Post-Istanbul, a single cold `SLOAD` costs 2,100 gas. Any `receive()` that reads one storage slot, emits an indexed event, or calls another contract will exceed 2,300 gas and cause the deposit to revert with an out-of-gas error.

`panicOnErrorOrInvalidOrFailedState` is called on the result, which panics and reverts the entire Cadence transaction: [5](#0-4) 

The same 2,300-gas ceiling applies to `NewTransferCall` (used by `Account.Transfer()`): [6](#0-5) 

---

### Impact Explanation

Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function consumes more than 2,300 gas — a common pattern for contracts that track deposits, emit events, or update state — **cannot receive FLOW tokens via the Cadence bridge deposit path**. The deposit call will always revert with an out-of-gas error. Because the panic propagates to the Cadence layer, the entire transaction reverts and the tokens are returned to the sender, but the deposit is permanently impossible for those contracts. This breaks cross-VM asset movement for a broad class of EVM smart contracts.

---

### Likelihood Explanation

The pattern is common: ERC-4626 vaults, staking contracts, payment splitters, and any contract that records incoming deposits in storage all require more than 2,300 gas in their `receive()` or `fallback()`. Any such contract deployed on Flow EVM is silently incompatible with the Cadence bridge deposit mechanism. A user or protocol deploying such a contract and attempting to fund it via `EVMAddress.deposit()` will encounter a permanent, opaque revert with no clear error message explaining the gas ceiling.

---

### Recommendation

Increase `DepositCallGasLimit` (and `DefaultGasLimitForTokenTransfer`) to a value that accommodates non-trivial `receive()`/`fallback()` implementations — for example, a configurable limit aligned with the block gas limit, or at minimum a value large enough to cover one cold `SLOAD` plus event emission (≥ 10,000 gas). The `WithdrawCallGasLimit` direction (COA → bridge) is less affected since the bridge address is a known EOA-like account, but the deposit direction (bridge → arbitrary EVM address) must accommodate arbitrary contract logic.

---

### Proof of Concept

1. Deploy an EVM smart contract on Flow EVM with a `receive()` function that reads one storage variable (costs 2,100 gas cold SLOAD post-Istanbul):
   ```solidity
   contract Receiver {
       uint256 public totalReceived;
       receive() external payable {
           totalReceived += msg.value; // SLOAD (2100) + SSTORE (~5000) > 2300
       }
   }
   ```
2. From a Cadence transaction, call:
   ```cadence
   let addr = EVM.EVMAddress(bytes: receiverContractBytes)
   addr.deposit(from: <-vault)
   ```
3. The call reaches `Account.Deposit()` → `NewDepositCall(..., GasLimit: 23_300)`. After 21,000 intrinsic gas, 2,300 gas remains. The `receive()` function runs out of gas at the `SLOAD`, the EVM call reverts, `panicOnErrorOrInvalidOrFailedState` panics, and the Cadence transaction reverts. The deposit is permanently impossible for this contract.

func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}
