### Title
Donated STRK tokens inflate `RewardSupplier.request_funds` credit, suppressing L1 mint requests and temporarily freezing reward distribution - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`RewardSupplier.request_funds` uses the contract's live ERC-20 balance (`balance_of(get_contract_address())`) as the "credit" side of its L1 replenishment decision. Because any address can transfer STRK directly to the `RewardSupplier` contract without going through `on_receive`, an unprivileged attacker can inflate that balance, causing the contract to believe it is sufficiently funded and skip L1 mint requests. Once the donated tokens are consumed by legitimate `claim_rewards` calls, the contract has no real balance left and all reward transfers revert, temporarily freezing unclaimed yield for every staker and delegator.

---

### Finding Description

`request_funds` is the internal function that decides whether to send a cross-chain mint request to the L1 `RewardSupplier`:

```cairo
// src/reward_supplier/reward_supplier.cairo  L301-L331
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // <-- live ERC-20 balance
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount;   // inflatable by donation
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);  // base_mint_amount / 2
    if credit < debit + threshold {
        // send L1 mint messages
    }
    self.l1_pending_requested_amount.write(l1_pending_requested_amount);
}
```

`request_funds` is called every time the staking contract reports new rewards:

```cairo
// src/reward_supplier/reward_supplier.cairo  L189-L202
fn update_unclaimed_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount,
) {
    ...
    let unclaimed_rewards = self.unclaimed_rewards.read() + rewards;
    self.unclaimed_rewards.write(unclaimed_rewards);
    self.request_funds(:unclaimed_rewards);   // uses live balance
}
```

The `on_receive` callback (the only legitimate inbound path) is restricted to the StarkGate bridge:

```cairo
// src/reward_supplier/reward_supplier.cairo  L233-L236
assert!(
    get_caller_address() == self.starkgate_address.read(),
    "{}", Error::ON_RECEIVE_NOT_FROM_STARKGATE,
);
```

However, `on_receive` is **not** the only way tokens can arrive. A standard ERC-20 `transfer` call deposits tokens into the contract's balance without triggering `on_receive` and without updating `l1_pending_requested_amount`. The contract has no mechanism to distinguish legitimately bridged tokens from donated ones.

`claim_rewards` transfers tokens based on internal `unclaimed_rewards` accounting, not on the live balance:

```cairo
// src/reward_supplier/reward_supplier.cairo  L205-L219
fn claim_rewards(ref self: ContractState, amount: Amount) {
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
}
```

`checked_transfer` will revert if the actual token balance is insufficient, even though `unclaimed_rewards` says the amount is owed.

---

### Impact Explanation

**Temporary freezing of unclaimed yield.**

Once the donated tokens are consumed by `claim_rewards` calls, the contract's real balance is zero while `unclaimed_rewards` is still non-zero. Every subsequent `claim_rewards` call reverts. The staking contract cannot pay out staker or delegator rewards until an operator manually replenishes the contract or triggers a new L1 mint cycle. All accrued but unclaimed STRK rewards are frozen for the duration.

---

### Likelihood Explanation

The entry path requires only a standard ERC-20 `transfer` — no privileged role, no bridge access, no governance action. The attacker must hold enough STRK to cover `debit + threshold - l1_pending_requested_amount` at the moment of the attack, where `threshold = base_mint_amount / 2`. On mainnet `base_mint_amount` is a protocol parameter, but the attacker only needs to donate enough to push `credit >= debit + threshold` for the current epoch's reward cycle. The cost is bounded and the tokens are not burned — they are simply locked in the contract until consumed, so the attacker's only cost is opportunity cost on the donated amount.

---

### Recommendation

Replace the live `balance_of` call in `request_funds` with an internally tracked balance variable that is incremented only when tokens arrive via `on_receive` (the legitimate StarkGate path) and decremented when `claim_rewards` transfers tokens out. This mirrors the fix applied in the referenced report: use internal accounting rather than the raw ERC-20 balance, so that unsolicited token donations cannot influence the L1 replenishment logic.

---

### Proof of Concept

**Scenario:**

1. The `RewardSupplier` has `balance = 0`, `l1_pending_requested_amount = P`, `unclaimed_rewards = U`. Normally `credit = P < U + threshold`, so L1 mint requests are sent.
2. Attacker calls `strk_token.transfer(reward_supplier_address, D)` where `D >= U + threshold - P`. This is a plain ERC-20 transfer; no special permission is needed.
3. The staking contract calls `update_unclaimed_rewards_from_staking_contract(rewards)`, which calls `request_funds(U + rewards)`.
4. Inside `request_funds`: `balance = D`, `credit = D + P >= U + rewards + threshold`. The `if` branch is skipped — **no L1 mint request is sent**.
5. Over subsequent epochs, `claim_rewards` is called repeatedly, consuming `D` from the contract's balance. `l1_pending_requested_amount` is never increased because `request_funds` kept seeing sufficient credit.
6. Once `D` is exhausted, `claim_rewards(amount)` calls `checked_transfer(staking_contract, amount)` with a zero balance and **reverts**.
7. All staker and delegator reward claims revert until the contract is manually refunded. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
