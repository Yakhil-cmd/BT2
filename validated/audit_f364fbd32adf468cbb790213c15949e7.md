Audit Report

## Title
Silent negative-amount wrap in `ledgeramount_from_amount` produces `Tokens::from_e8s(1)` for Disburse operations — (`rs/rosetta-api/icp/src/models/amount.rs`)

## Summary
`ledgeramount_from_amount` calls `from_amount`, which validates only that `val.abs()` fits in `u64` but returns the original signed `i128`. The subsequent `inner as u64` cast in `ledgeramount_from_amount` silently wraps `-18446744073709551615_i128` to `1`, so the Rosetta construction pipeline produces a governance Disburse payload for 1 e8s with no error returned to the caller.

## Finding Description
`from_amount` at lines 34–36 of `rs/rosetta-api/icp/src/models/amount.rs` checks `u64::try_from(val.abs())` to ensure the magnitude fits in a `u64`, but returns `val` unchanged regardless of sign. For the input `"-18446744073709551615"`, `val.abs()` equals `u64::MAX` exactly, so the guard succeeds and the negative `i128` is returned.

`ledgeramount_from_amount` at lines 42–44 performs `inner as u64` with no sign check. Rust's `as` cast for `i128 → u64` truncates to the low 64 bits: `-18446744073709551615_i128` = `-(2⁶⁴ − 1)`, whose two's-complement low-64-bit representation is `1`. The function returns `Tokens::from_e8s(1)`.

In `rs/rosetta-api/icp/src/convert.rs` at lines 258–270, the result of `ledgeramount_from_amount` is passed directly to `state.disburse` with no further validation. The construction pipeline completes successfully and returns a signed Disburse payload for 1 e8s to the caller.

The existing guard is insufficient because it checks magnitude only, not sign. `from_amount` is intentionally shared with transfer debit legs (where negative values are valid), but `ledgeramount_from_amount` requires a strictly non-negative result and never enforces that invariant.

## Impact Explanation
The Rosetta construction server silently substitutes a caller-supplied large negative amount with `Tokens::from_e8s(1)` and constructs a governance Disburse payload for 1 e8s. No error is returned. Any client or automated service that relies on the Rosetta server to validate amounts before signing will sign and submit a transaction for the wrong amount. This constitutes a significant Rosetta API security impact with concrete user financial harm: a neuron owner's disburse operation is silently misdirected to 1 e8s rather than the intended amount, with no indication of failure. This fits the High impact class for significant Rosetta/ledger security impact with concrete user or protocol harm.

## Likelihood Explanation
The path is fully reachable via an unauthenticated HTTP POST to `/construction/payloads` with a crafted `DISBURSE` operation containing `value: "-18446744073709551615"`. No privileged role is required to reach the vulnerable code path. The caller must own or hold hotkey access to the target neuron and must sign the resulting payload, bounding cross-user impact. The specific input value is the unique i128 that passes the magnitude guard while being negative and wrapping to a non-zero u64, making it a targeted but mechanically trivial input.

## Recommendation
Add an explicit sign check in `ledgeramount_from_amount` before the cast:

```rust
pub fn ledgeramount_from_amount(amount: &Amount, token_name: &str) -> Result<Tokens, String> {
    let inner = from_amount(amount, token_name)?;
    if inner < 0 {
        return Err(format!("Amount must be non-negative, got {inner}"));
    }
    Ok(Tokens::from_e8s(inner as u64))
}
```

Alternatively, split `from_amount` into a signed variant for transfer debit legs and an unsigned variant for Disburse/Stake amounts, eliminating the shared-but-differently-constrained interface.
