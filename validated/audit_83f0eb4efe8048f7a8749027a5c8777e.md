### Title
Malicious or Unavailable Coordinator Permanently Denies Signing for All Honest Participants — (File: `src/frost/redjubjub/sign.rs`, `src/frost/eddsa/sign.rs`)

---

### Summary

In the FROST RedJubjub signing protocol and FROST EdDSA `sign_v1`, the coordinator is the sole entity responsible for broadcasting a critical protocol message to all non-coordinator participants (the `Randomizer` in RedJubjub; the `SigningPackage` in EdDSA v1). Non-coordinator participants block indefinitely waiting for this message with no timeout, no fallback, and no coordinator-rotation mechanism. A malicious or unavailable coordinator can permanently deny signing for all honest participants. In the RedJubjub case, participants have already consumed their one-time-use presignature nonces before the block occurs, making the denial irreversible without generating new presignatures.

---

### Finding Description

**RedJubjub — `src/frost/redjubjub/sign.rs`**

In `do_sign_coordinator`, the coordinator computes a `RandomizedParams` and broadcasts the `Randomizer` to all participants:

```rust
// src/frost/redjubjub/sign.rs:147-150
let randomizer = randomized_params.randomizer();
// Send the Randomizer to everyone
let wait_round_1 = chan.next_waitpoint();
chan.send_many(wait_round_1, &randomizer)?;
```

In `do_sign_participant`, every non-coordinator participant enters an unbounded `loop` waiting exclusively for this message from the coordinator:

```rust
// src/frost/redjubjub/sign.rs:216-223
let wait_round_1 = chan.next_waitpoint();
let randomizer = loop {
    let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;
    if from != coordinator {
        continue;
    }
    break randomizer;
};
```

There is no timeout on `chan.recv`. Messages from any party other than the coordinator are silently discarded and the loop continues. If the coordinator never sends the randomizer, this loop never terminates.

Critically, each participant has already loaded its `PresignOutput` (containing one-time-use nonces) into the function before reaching this wait point. Those nonces are consumed by the call regardless of whether the protocol completes.

**FROST EdDSA v1 — `src/frost/eddsa/sign.rs`**

The identical structural pattern exists in `do_sign_participant_v1`. After sending its commitment to the coordinator, each participant blocks waiting for the `SigningPackage`:

```rust
// src/frost/eddsa/sign.rs:272-280
let r2_wait_point = chan.next_waitpoint();
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;
    if from != coordinator {
        continue;
    }
    break signing_package;
};
```

The coordinator is the only entity that can produce and send this `SigningPackage` (assembled from all collected commitments at line 130 of `do_sign_coordinator_v1`). No other participant can substitute for it.

The `recv_from_others` helper used throughout the library (`src/protocol/helpers.rs:19`) also has no timeout — it loops until `seen.full()`, meaning any missing message from any participant (including the coordinator) causes indefinite blocking.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

A single malicious coordinator can:
1. Participate honestly through the commitment-collection round (receiving all participants' nonces/commitments).
2. Refuse to send the `Randomizer` (RedJubjub) or `SigningPackage` (EdDSA v1) back to participants.
3. Leave all honest participants permanently suspended at `chan.recv(...)`.

In the RedJubjub case, participants' `PresignOutput` nonces are already in scope and consumed by the protocol invocation. Those nonces cannot be reused (FROST nonce reuse breaks security), so the presignature is permanently wasted. A new presign round must be run before signing can be attempted again — and the same coordinator can repeat the attack.

There is no library-level mechanism to abort, time out, or elect a replacement coordinator. The honest parties have no recourse within the protocol.

---

### Likelihood Explanation

The coordinator is a regular participant selected by the application. Any participant who is designated coordinator can execute this attack trivially — they simply do not send the required broadcast message. No cryptographic capability, no key material beyond their own share, and no external resource is required. The attack is silent: honest participants cannot distinguish a malicious coordinator from a slow or crashed one, and there is no detection or slashing mechanism in the library.

---

### Recommendation

1. **Add a timeout to coordinator-dependent `recv` loops.** The unbounded `loop { chan.recv(...).await? }` in `do_sign_participant` (RedJubjub) and `do_sign_participant_v1` (EdDSA) should be bounded by a caller-configurable deadline, returning a `ProtocolError::Timeout` so the application can abort and retry with a different coordinator.
2. **Document the coordinator trust assumption explicitly.** The public API of `sign()` in both modules should state that the coordinator is a trusted role and that a malicious coordinator can deny signing.
3. **Consider a coordinator-free aggregation path.** For the EdDSA v2 and RedJubjub presign-based flows, the signing package can be constructed by any participant who holds the full commitment map, removing the coordinator as a single point of failure for that step.

---

### Proof of Concept

**RedJubjub scenario:**

1. Run a presign round to generate `PresignOutput` for all N participants.
2. Initiate a signing session: all participants call `sign(participants, threshold, me, coordinator, keygen_output, presignature, message, randomizer_opt)`.
3. The coordinator's `do_sign_coordinator` computes `RandomizedParams` but the coordinator process is killed (or the coordinator is malicious and simply drops the `chan.send_many` call).
4. All non-coordinator participants are now suspended at `chan.recv(wait_round_1).await?` in `do_sign_participant` (line 218), waiting for a `Randomizer` that will never arrive.
5. The signing session never completes. The `PresignOutput` nonces held by each participant are consumed and cannot be reused. A fresh presign round is required, and the same coordinator can repeat the attack indefinitely.
