### Title
Global `last_reward_block` Overwritten by Interleaved `update_rewards` Calls, Causing Permanent Loss of Staker Yield — (File: src/staking/staking.cairo)

---

### Summary

The `Staking` contract stores a single global `last_reward_block` variable to track the last block for which consensus block rewards were distributed. Because `update_rewards` is callable by any address for any staker, interleaved calls across different stakers overwrite this shared variable. Each call advances `last_reward_block` to the current block, so a subsequent call for a different staker "steals" the elapsed-block window from the first staker, permanently destroying that staker's unclaimed yield for the skipped blocks.

---

### Finding Description

In `src/staking/staking.cairo`, the `Storage` struct declares two single global (non-per-staker) variables:

```
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,          // line 187
/// Block rewards (STRK, BTC) for the current epoch.
block_rewards: (Amount, Amount),         // line 199
``` [1](#0-0) 

Neither field is a `Map<ContractAddress, …>`. They are shared across every staker.

The public entry point `update_rewards` (accessible by any caller, no access-control guard in the interface) distributes consensus block rewards to a specific staker:

```rust
fn update_rewards(
    ref self: TContractState, staker_address: ContractAddress, disable_rewards: bool,
);
``` [2](#0-1) 

The intended logic is:

```
num_blocks  = current_block_number − last_reward_block
staker_reward = block_rewards × staker_share × num_blocks
last_reward_block ← current_block_number          // ← global overwrite
```

Because `last_reward_block` is global, every call for **any** staker resets the counter for **all** stakers.

**Concrete overwrite sequence:**

| Step | Caller | Block | `last_reward_block` after | Blocks rewarded |
|------|--------|-------|--------------------------|-----------------|
| 1 | Staker A | 100 | 100 | 10 (90→100) |
| 2 | Staker B | 105 | **105** | 5 (100→105) |
| 3 | Staker A | 110 | 110 | **5** (105→110) ← should be 10 |

Staker A permanently loses 5 blocks of yield because Staker B's call at step 2 overwrote `last_reward_block`. The skipped rewards are never distributed to anyone.

This is the direct analog of the oracle report's pattern: a **single shared storage variable** (`lastHouseId` / `last_reward_block`) is overwritten by a second in-flight operation before the first operation's "callback" (here, the next `update_rewards` call for the same staker) can consume it, causing the wrong state to be used.

---

### Impact Explanation

The rewards for the "stolen" block window are never minted or transferred to any party — they are permanently frozen. This matches the allowed High impact: **Permanent freezing of unclaimed yield**. In a protocol with many active stakers calling `update_rewards` at different block heights (the normal operating condition), every staker continuously loses a fraction of their entitled yield. A malicious actor can amplify this by calling `update_rewards` for many stakers in rapid succession, maximally advancing `last_reward_block` and minimising every staker's next reward window.

---

### Likelihood Explanation

`update_rewards` carries no access-control restriction — any address may call it for any `staker_address`. In normal operation, stakers (or bots acting on their behalf) call it at different block heights, so the overwrite occurs organically. No special privilege, leaked key, or external dependency is required. The entry path is fully reachable by an unprivileged public caller.

---

### Recommendation

Replace the single global `last_reward_block` with a per-staker mapping:

```rust
// Before (global — vulnerable)
last_reward_block: BlockNumber,

// After (per-staker — safe)
last_reward_block: Map<ContractAddress, BlockNumber>,
```

Each staker's elapsed-block window is then computed and updated independently, eliminating the overwrite race.

---

### Proof of Concept

```
1. Deploy Staking contract; advance to consensus-rewards epoch.
2. At block 100: call update_rewards(staker_A, false).
   → staker_A receives rewards for 10 blocks (90→100).
   → last_reward_block = 100.
3. At block 105: call update_rewards(staker_B, false).
   → staker_B receives rewards for 5 blocks (100→105).
   → last_reward_block = 105.   ← global overwrite
4. At block 110: call update_rewards(staker_A, false).
   → staker_A receives rewards for only 5 blocks (105→110).
   → Expected: 10 blocks (100→110).
5. Assert staker_A.unclaimed_rewards < expected_10_block_reward.
   → Assertion passes; 5 blocks of yield are permanently lost.
```
