### Title
`OracleValueStopLossExtension` stop-loss timelock can be reduced to zero, allowing pool admin to atomically disable LP value protection — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`proposeOracleStopLossTimelock` accepts any `uint32` value for `newTimelock` with no lower-bound validation. After the current timelock elapses once, the pool admin can execute the reduction to zero. With a zero timelock, `_afterTimelock` returns `block.timestamp`, so `proposeOracleStopLossDrawdown(pool, 0)` and `executeOracleStopLossDrawdown(pool)` can be called atomically in the same transaction, setting `drawdownE6 = 0` and triggering the early-return bypass that completely disables the stop-loss check. LPs who deposited trusting the timelock protection have no time to react.

---

### Finding Description

The `OracleValueStopLossExtension` is designed to protect LPs by blocking swaps that drain per-bin value below a drawdown floor. The NatSpec explicitly states:

> "Drawdown and decay changes are timelocked so LPs can react; monitor at least as often as the timelock or trust the pool admin."

However, the timelock itself can be reduced to zero with no minimum bound check: [1](#0-0) 

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;   // ← newTimelock can be 0, no validation
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
}
```

`_afterTimelock` computes `block.timestamp + oracleStopLossConfig[pool_].timelock`. Once the stored timelock is 0, this returns exactly `block.timestamp`: [2](#0-1) 

`_requireElapsed` checks `block.timestamp < executeAfter`. When `executeAfter == block.timestamp`, the condition is `false` — no revert: [3](#0-2) 

`_validateDrawdown` only enforces an upper bound (`drawdownE6 > E6`), not a lower bound, so `drawdownE6 = 0` is accepted: [4](#0-3) 

With `drawdownE6 = 0` stored, `_afterSwapOracleStopLoss` returns immediately, skipping all watermark checks: [5](#0-4) 

```solidity
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;   // ← stop-loss completely disabled
```

There is no minimum timelock constant or validation anywhere in the codebase — confirmed by the absence of any `MIN_TIMELOCK` or lower-bound check in `proposeOracleStopLossTimelock` or `initialize`. [6](#0-5) 

---

### Impact Explanation

LPs who deposit into a pool with a non-zero stop-loss timelock trust that the admin cannot change stop-loss parameters without giving them time to react and withdraw. After the admin reduces the timelock to zero and atomically disables the drawdown check, swaps that drain per-bin value below the previous floor are no longer blocked. The core LP protection mechanism is removed, exposing deposited principal to value drain through oracle price movements. This is a direct loss of LP assets — the invariant "value per share at oracle marks cannot fall faster than drawdown + decay × t" is silently broken.

---

### Likelihood Explanation

The pool admin is semi-trusted only within caps and timelocks. The timelock is the cap on the admin's power to change stop-loss parameters. The bypass requires only one wait of the current timelock (a one-time cost), after which the admin has permanent unrestricted control over all stop-loss parameters. Any pool admin who turns adversarial after the initial timelock period can execute this attack. The attack is atomic (single transaction after the one-time wait) and leaves no on-chain warning for LPs.

---

### Recommendation

Add a minimum timelock validation in `proposeOracleStopLossTimelock` to prevent the timelock from being reduced below a meaningful floor. A one-way ratchet (new timelock ≥ current timelock) would also work:

```solidity
uint32 private constant MIN_TIMELOCK = 1 days;

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    if (newTimelock < MIN_TIMELOCK) revert OracleStopLossTimelockTooSmall(newTimelock);
    // ... rest unchanged
}
```

Alternatively, validate that `newTimelock >= oracleStopLossConfig[pool_].timelock` (monotone-increasing ratchet). Either approach ensures LPs always have a guaranteed reaction window.

---

### Proof of Concept

1. Pool is created with stop-loss extension: `drawdownE6 = 100_000` (10%), `timelock = 7 days`.
2. LPs deposit, trusting the 7-day reaction window.
3. Pool admin calls `proposeOracleStopLossTimelock(pool, 0)` → `pendingTimelockExecuteAfter = block.timestamp + 7 days`.
4. After 7 days, pool admin calls `executeOracleStopLossTimelock(pool)` → `oracleStopLossConfig[pool].timelock = 0`.
5. Pool admin deploys an attacker contract that in a single transaction:
   - Calls `proposeOracleStopLossDrawdown(pool, 0)` → `pendingDrawdownExecuteAfter = block.timestamp + 0 = block.timestamp`.
   - Calls `executeOracleStopLossDrawdown(pool)` → `_requireElapsed(block.timestamp)` passes (`block.timestamp < block.timestamp` is `false`); `drawdownE6` is set to `0`.
6. `_afterSwapOracleStopLoss` now returns immediately on every swap (`if (drawdown == 0) return`).
7. LPs had zero seconds to react after step 4; their stop-loss protection is permanently removed.

This is the direct analog of the NibblVault M-09 finding: just as the NibblVault admin could change the vault implementation (UPDATE_TIME = 2 days) before a buyout completed (BUYOUT_DURATION = 5 days), the Metric OMM pool admin can reduce the stop-loss timelock to zero after a single wait, then immediately change all stop-loss parameters with no delay — bypassing the timelock cap that defines the boundary of semi-trusted admin power.
