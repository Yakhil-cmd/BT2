### Title
`calculateNFTRoyalties` Hardcodes `nftSellerNetAmount` to Full Amount, Suppressing Negative-Proceeds Guard and Displaying Wrong Net Proceeds to Seller - (File: `packages/gui/src/components/offers/utils.ts`)

---

### Summary

In `packages/gui/src/components/offers/utils.ts`, the function `calculateNFTRoyalties` sets `nftSellerNetAmount` to the raw `amount` (the full sale price) instead of `amount - royaltyAmount`. The correct calculation is present but commented out. This causes the NFT offer editor and viewer to display the wrong "You will receive / Net Proceeds" figure to the seller, and permanently suppresses the guard that is supposed to block offers where royalties exceed the sale price.

---

### Finding Description

`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` computes royalty figures for the NFT offer UI:

```typescript
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;          // ← BUG: should be amount - royaltyAmount
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The correct formula — `amount - royaltyAmount - makerFee` — is commented out. `nftSellerNetAmount` is therefore always equal to the full `amount`, regardless of how large the royalty is.

This value flows into two security-relevant places in `NFTOfferEditor.tsx`:

1. **Display**: The "You will receive" label renders `nftSellerNetAmount` directly, so the seller always sees the full sale price, not the net proceeds after royalties.
2. **Guard**: `showNegativeAmountWarning` is derived as `(nftSellerNetAmount ?? 0) < 0`. Because `nftSellerNetAmount === amount` and `amount` is always a positive number entered by the user, this condition is permanently `false`. The guard that is supposed to prevent creation of offers where royalties exceed the sale price never fires. [2](#0-1) 

The same `nftSellerNetAmount` is also rendered as "Net Proceeds" in `NFTOfferViewer.tsx` when a buyer views an imported offer: [3](#0-2) 

---

### Impact Explanation

A seller using the NFT offer editor is shown that they will receive the full sale price (e.g., 1 XCH). On-chain, the royalty is enforced and deducted (e.g., 0.20 XCH for a 20% royalty NFT), so the seller actually receives only 0.80 XCH. The GUI never corrects this discrepancy and never warns the seller. Additionally, the guard meant to block economically nonsensical offers (royalty > sale price) is permanently disabled, so a seller can unknowingly publish an offer where they receive zero or negative net proceeds. The buyer's "Net Proceeds" view in the offer viewer is equally wrong.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of… offer… state that causes a user to approve… the wrong… amount."*

---

### Likelihood Explanation

Any user who creates or views an NFT offer for an NFT with a non-zero royalty percentage is affected. NFT royalties are common in the Chia ecosystem. No special attacker capability is required — the bug is triggered by normal offer creation flow. The attacker role is passive: a malicious NFT creator sets a high royalty percentage, and the GUI silently hides the true cost from the seller.

---

### Recommendation

Restore the correct net-amount calculation in `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [4](#0-3) 

This restores the `showNegativeAmountWarning` guard in `NFTOfferEditor.tsx` and ensures both the editor and viewer display the correct net proceeds to the seller. [5](#0-4) 

---

### Proof of Concept

1. Mint an NFT with a 20% royalty (`royaltyPercentage = 2000` in basis points, `convertRoyaltyToPercentage` → `20`).
2. Open the NFT offer editor, select "Sell an NFT", enter the NFT ID, and set a sale price of 1 XCH.
3. `calculateNFTRoyalties(1, 0, 20, NFTForToken)` is called:
   - `royaltyAmount = (20 / 100) * 1 = 0.2`
   - `nftSellerNetAmount = 1` ← hardcoded to full amount, not `1 - 0.2 = 0.8`
4. The UI displays "You will receive: **1 XCH**" — the seller believes they receive 1 XCH.
5. The offer is accepted on-chain; the royalty (0.2 XCH) is deducted; the seller receives **0.8 XCH**.
6. Repeat with a 110% royalty NFT: `nftSellerNetAmount` is still positive (= 1), so `showNegativeAmountWarning` stays `false` and the offer is not blocked, even though the seller would receive a negative net amount. [6](#0-5)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
