### Title
Gateway proof-facts block-hash lookup uses pre-`pre_process_block` state while the block-number bound uses the post-increment block number, causing valid proof-bearing transactions to be rejected — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`run_validate_entry_point` increments the latest committed block number by one (`unchecked_next()`) before building the `BlockContext` used for gateway validation. This makes `validate_proof_block_number` permit a `proof_block_number` up to `N + 1 − STORED_BLOCK_HASH_BUFFER`. However, the state passed to `validate_proof_block_hash` is the raw state at block N — `pre_process_block` has not been run — so the hash for block `N + 1 − STORED_BLOCK_HASH_BUFFER` is not yet stored (it reads as `Felt::ZERO`). The hash-equality check therefore always fails for that boundary block number, and the gateway rejects a class of transactions that the blockifier would accept during actual block execution.

---

### Finding Description

**Step 1 — block number is incremented for the validation context.**

In `run_validate_entry_point`:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // N → N+1
let block_context = BlockContext::new(block_info, ...);
``` [1](#0-0) 

The state reader, however, is rooted at block N — no `pre_process_block` is called.

**Step 2 — `validate_proof_block_number` uses the incremented number.**

```rust
Self::validate_proof_block_number(
    proof_block_number,
    block_context.block_info.block_number,   // N+1
)?;
``` [2](#0-1) 

Inside `validate_proof_block_number`:

```rust
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// max_allowed = (N+1) − STORED_BLOCK_HASH_BUFFER
``` [3](#0-2) 

So `proof_block_number = N + 1 − STORED_BLOCK_HASH_BUFFER` passes the number check.

**Step 3 — `validate_proof_block_hash` reads from the block-N state.**

```rust
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

if stored_block_hash != proof_block_hash {
    return Err(TransactionPreValidationError::InvalidProofFacts(
        format!("Block hash mismatch for block {proof_block_number}. ...")
    ));
}
``` [4](#0-3) 

The state at block N only contains hashes for blocks `0 … N − STORED_BLOCK_HASH_BUFFER`. The hash for block `N + 1 − STORED_BLOCK_HASH_BUFFER` is written by `pre_process_block` when block N+1 is actually executed:

```rust
// Writes hash of (current_block_number − STORED_BLOCK_HASH_BUFFER) into state.
func write_block_number_to_block_hash_mapping(block_context: BlockContext*) {
    tempvar old_block_number = block_context.block_info_for_execute.block_number
        - STORED_BLOCK_HASH_BUFFER;
``` [5](#0-4) 

Because the gateway never calls `pre_process_block`, `stored_block_hash` is `Felt::ZERO` for that slot, and the check `stored_block_hash != proof_block_hash` is always true (the proof carries a non-zero hash). The transaction is rejected with `"Block hash mismatch"`.

**Step 4 — the OS accepts the same transaction.**

The Cairo OS check is:

```cairo
assert_nn_le(
    os_output_header.base_block_number, current_block_number - STORED_BLOCK_HASH_BUFFER
);
``` [6](#0-5) 

`current_block_number` in the OS is also N+1, so the same boundary value is allowed. Crucially, `pre_process_block` runs before any transaction in the block, so the hash IS present when `read_block_hash_from_storage` is called. The transaction would succeed during actual execution.

---

### Impact Explanation

Any user who submits a valid Invoke V3 transaction whose `proof_facts` reference block `N + 1 − STORED_BLOCK_HASH_BUFFER` (the most recently allowed block hash according to the protocol) will have that transaction rejected at the gateway with a spurious `"Block hash mismatch"` error. The transaction is structurally valid and would be accepted and executed correctly by the blockifier. This is a gateway admission false-negative for a well-formed transaction class.

Impact category: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The trigger is unprivileged: any user can craft an Invoke V3 transaction with `proof_facts.block_number = N + 1 − STORED_BLOCK_HASH_BUFFER`. The block hash for that block is publicly known (it is the hash of a committed block). The condition is deterministic and reproducible on every block once the chain has advanced past `STORED_BLOCK_HASH_BUFFER` blocks. No special timing or race condition is required.

---

### Recommendation

In `run_validate_entry_point`, either:

1. **Do not increment the block number for the proof-facts check.** Use the committed block N's number when calling `validate_proof_block_number`, so the allowed range matches the hashes actually present in the state:

```rust
// Before incrementing, capture the committed block number for proof-facts validation.
let committed_block_number = block_info.block_number;
block_info.block_number = block_info.block_number.unchecked_next();
```

and pass `committed_block_number` to `validate_proof_block_number`; or

2. **Run `pre_process_block` on the gateway's cached state** before calling `perform_pre_validation_stage`, so the hash for block `N + 1 − STORED_BLOCK_HASH_BUFFER` is present when `validate_proof_block_hash` reads it.

Option 1 is simpler and avoids the need to supply the old block hash to the gateway.

---

### Proof of Concept

Let `N` be the latest committed block number and `B = STORED_BLOCK_HASH_BUFFER` (= 10).

1. Obtain the block hash of block `N + 1 − B` from any RPC node (`starknet_getBlockWithTxHashes`).
2. Construct an Invoke V3 transaction with `proof_facts` containing:
   - `proof_version = PROOF_VERSION_V1`
   - `base_block_number = N + 1 − B`
   - `base_block_hash = <hash obtained in step 1>` (non-zero, correct)
   - valid `program_hash`, `config_hash`
3. Submit to the gateway.

**Expected (correct) result:** transaction accepted and forwarded to the mempool.

**Actual result:** gateway returns `"Block hash mismatch for block N+1−B. Proof block hash: <correct>, stored block hash: 0x0"` and rejects the transaction.

The mismatch arises because `validate_proof_block_number` (using `block_number = N+1`) permits `proof_block_number = N+1−B`, but `validate_proof_block_hash` reads from the state at block N where that slot is still zero — `pre_process_block` for block N+1 has not yet been applied.     fn validate_proof_block_number(
        proof_block_number: u64,
        current_block_number: BlockNumber,
    ) -> TransactionPreValidationResult<()> {
        // Proof block must be old enough to have a stored block hash.
        // Stored block hashes are guaranteed only up to: current - STORED_BLOCK_HASH_BUFFER.
        let max_allowed =
            current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER).ok_or_else(|| {
                TransactionPreValidationError::InvalidProofFacts(format!(
                    "The current block number {current_block_number} is below the required \
                     block-hash retention buffer: {STORED_BLOCK_HASH_BUFFER}."
                ))
            })?;

        if proof_block_number > max_allowed {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "The proof block number {proof_block_number} is too recent. The maximum allowed \
                 block number is {max_allowed}."
            )));
        }

        Ok(())
    }

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }
