### Title
Missing Commission Slippage Protection in `enter_delegation_pool()`, `add_to_delegation_pool()`, and `switch_delegation_pool()` — (File: `src/pool/pool.cairo`)

### Summary
The `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool` functions in `pool.cairo` accept no `max_commission` parameter. A staker who holds an active `CommissionCommitment` can legally increase their commission up to `max_commission` at any time during the commitment window. Because delegators have no on-chain way to bound the commission they are willing to accept, a staker can front-run any of these three calls and redirect all future pool rewards to themselves, constituting theft of unclaimed yield.

---

### Finding Description

`enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool` in `pool.cairo` transfer tokens and register the caller as a pool member without any check on the pool's current commission rate. [1](#0-0) 

The commission that governs how pool rewards are split is stored in the staking contract and is read at reward-calculation time. It can be changed by the staker at any moment via `set_commission` in `staking.cairo`. [2](#0-1) 

Without a `CommissionCommitment`, `update_commission` enforces that the commission can only decrease: [3](#0-2) 

However, when an active `CommissionCommitment` exists, the staker may set the commission to **any value up to `max_commission`**, including values higher than the current commission: [4](#0-3) 

The code itself acknowledges this gap: [5](#0-4) 

A staker can therefore set a commitment with `max_commission = 10000` (100 %) and then raise the commission to 100 % in the same block as a delegator's entry transaction, capturing all pool rewards.

---

### Impact Explanation

**Impact: High — Theft of unclaimed yield.**

After the commission is raised to 100 %, `split_rewards_with_commission` routes the entire pool reward to the staker and zero to the pool contract. Every epoch the delegator remains in the pool, their accrued yield is silently redirected to the staker. The delegator's principal is safe (they can exit), but all yield earned from the moment of entry until exit is stolen.

---

### Likelihood Explanation

**Likelihood: Medium.**

The precondition is that the staker has set a `CommissionCommitment` with a high `max_commission`. This is a normal, permissionless staker action. Once the commitment is in place, the staker can raise the commission in a single transaction. On Starknet, sequencer ordering is controlled, making front-running straightforward for a staker who monitors the mempool or simply raises the commission before advertising the pool. The attack requires no privileged role beyond being the pool's own staker.

---

### Recommendation

Add a `max_commission` parameter to `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool` in `pool.cairo`. At the start of each function, assert that the pool's current commission does not exceed the caller-supplied bound:

```cairo
fn enter_delegation_pool(
    ref self: ContractState,
    reward_address: ContractAddress,
    amount: Amount,
    max_commission: Commission,   // <-- new
) {
    let current_commission = self.get_commission_from_staking_contract();
    assert!(current_commission <= max_commission, "commission exceeds max");
    // ... rest of function unchanged
}
```

Apply the same pattern to `add_to_delegation_pool` and `switch_delegation_pool`.

---

### Proof of Concept

1. Staker deploys a pool with commission = 500 (5 %).
2. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
   - Valid because `max_commission (10000) >= current_commission (500)`. [6](#0-5) 
3. Delegator observes 5 % commission and submits `enter_delegation_pool(reward_address, amount)`.
4. Before the delegator's transaction is included, staker calls `set_commission(10000)`.
   - Valid because an active commitment with `max_commission = 10000` exists. [7](#0-6) 
5. Delegator's `enter_delegation_pool` executes. Tokens are transferred; the pool member record is created. [8](#0-7) 
6. At the next reward distribution, `split_rewards_with_commission` applies commission = 10000/10000 = 100 %, sending the entire pool reward to the staker and zero to the pool contract.
7. Delegator calls `claim_rewards` and receives 0 STRK despite having staked for the full epoch.

            // Prerequisites and asserts.
            self.general_prerequisites();

            if let Option::Some(old_commission) = staker_pool_info.commission.read() {

                        :staker_address, :staker_pool_info, :old_commission, :commission,
                    );
            } else {
                staker_pool_info.commission.write(Option::Some(commission));
                self.emit(Events::CommissionInitialized { staker_address, commission });
