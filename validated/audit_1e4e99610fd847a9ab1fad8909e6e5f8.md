### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Enables Permanent Freezing of Unclaimed Yield - (`File: src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the Pool contract iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim checkpoint. There is no cap on how many entries can accumulate. A delegator who changes their balance across many epochs without claiming rewards will eventually be unable to call `claim_rewards` because the transaction will exceed Starknet's gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal function `calculate_rewards` contains an unbounded `while` loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop iterates over all entries in `pool_member_epoch_balance` from `entry_to_claim_from` (the index saved at the last `claim_rewards` call) up to the current epoch. Each iteration also calls `find_sigma`, which performs additional storage reads. [2](#0-1) 

Entries are appended to the trace by `set_member_balance`, which is called by `increase_member_balance`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(...);
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [3](#0-2) 

The underlying `insert` in the trace only deduplicates if the key (epoch) matches the last entry; otherwise it **appends a new checkpoint**:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [4](#0-3) 

This means every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **different epoch** appends a new entry. `add_to_delegation_pool` is callable by the pool member or their reward address: [5](#0-4) 

`exit_delegation_pool_intent` is callable by the pool member: [6](#0-5) 

`claim_rewards` calls `calculate_rewards` with the saved `entry_to_claim_from` index, which is only advanced when `claim_rewards` is successfully called: [7](#0-6) 

If `claim_rewards` never succeeds (OOG), `entry_to_claim_from` is never updated, so every subsequent attempt re-iterates the same growing trace — permanently bricking reward claims for that pool member.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.** Once the trace is large enough that `calculate_rewards` exceeds the Starknet transaction gas limit, the pool member's `claim_rewards` call will always revert. Because `entry_to_claim_from` is only updated inside a successful `claim_rewards` execution, there is no recovery path. The accumulated rewards are locked forever.

This matches the allowed impact: **"Permanent freezing of unclaimed yield or unclaimed royalties"** (High).

---

### Likelihood Explanation

A delegator who:
1. Calls `add_to_delegation_pool` (with any non-zero amount) once per epoch, and
2. Does not call `claim_rewards` between those calls

will accumulate one new trace entry per epoch. The cost per epoch is the minimum delegation amount plus gas. Over hundreds or thousands of epochs (which is realistic for a long-lived protocol), the trace grows large enough to cause OOG on `claim_rewards`. This can happen organically (an active delegator who forgets to claim) or deliberately (a delegator who intentionally stuffs their own trace to later claim a griefing scenario, or simply a user who is unaware of the risk). The codebase itself acknowledges the risk with the comment "This loop is unbounded but unlikely to exceed gas limits."

---

### Recommendation

1. **Enforce a maximum number of unclaimed balance-change entries** before allowing further balance changes (analogous to `maxPendingLocal` in the Perennial bug). Revert `add_to_delegation_pool` / `exit_delegation_pool_intent` if `pool_member_trace_length - entry_to_claim_from` exceeds a safe threshold.
2. **Alternatively**, allow partial reward claims by accepting a `max_entries` parameter in `claim_rewards`, so a delegator can drain their trace in multiple bounded transactions.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with minimum amount at epoch E.
2. For each subsequent epoch E+1, E+2, ..., E+N, delegator calls `add_to_delegation_pool` with minimum amount. Each call appends one entry to `pool_member_epoch_balance` trace (since the epoch key differs each time).
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs, delegator calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates N times, each iteration reading from storage and calling `find_sigma`. For sufficiently large N, the transaction exceeds the Starknet gas limit and reverts.
5. Every subsequent `claim_rewards` attempt also reverts. The delegator's accumulated yield is permanently frozen. [8](#0-7)

                pool_member_info.unpool_time = Option::Some(unpool_time);

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
