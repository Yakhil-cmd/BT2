### Title
`nftSellerNetAmount` Not Updated to Deduct Royalties in `TokenForNFT` Branch, Causing Inflated "Net Proceeds" Display That Induces Offer Acceptance - (File: packages/gui/src/components/offers/utils.ts)

---

### Summary

`calculateNFTRoyalties()` sets `nftSellerNetAmount = amount` unconditionally and never subtracts the royalty in the `TokenForNFT` branch. The commented-out code directly below the assignment shows the intended deduction was deliberately removed. This inflated value is then rendered as the authoritative **"Net Proceeds"** figure in `NFTOfferViewer` and is also used to override the per-row amount shown in the offer summary, causing an NFT seller to see a materially higher payout than they will actually receive, and potentially inducing acceptance of an offer they would otherwise reject.

---

### Finding Description

`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` has a dual-role variable pattern identical to the external report:

```typescript
// packages/gui/src/components/offers/utils.ts  lines 312-319
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;          // ← used first as raw amount
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );                                              // ← second role (net) is commented out
const totalAmount: number =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount + royaltyAmount
    : amount + makerFee + royaltyAmount;            // ← also wrong for TokenForNFT
```

For the `TokenForNFT` case the in-app tooltip explicitly states:

> *"those creator fees will be paid from the offered purchase price"*

meaning the seller's net is `amount − royaltyAmount`, not `amount`. The second update (subtracting the royalty) is the commented-out block and is never applied.

The stale value then flows into two display paths in `NFTOfferViewer.tsx`:

**Path 1 – "Net Proceeds" label (line 596):**
```typescript
// NFTOfferViewer.tsx  lines 580-598
{exchangeType === NFTOfferExchangeType.TokenForNFT && (
  ...
  <Trans>Net Proceeds</Trans>   // tooltip: "asking price minus creator fees"
  ...
  <FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} />
)}
```

**Path 2 – `overrideNFTSellerAmount` passed to the offer-summary row (lines 397-402):**
```typescript
// NFTOfferViewer.tsx  lines 397-402
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // inflated
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
```

This override replaces the raw on-chain amount shown in the `NFTOfferSummary` row (the "You will receive" line) with the same inflated figure.

---

### Impact Explanation

An NFT seller viewing an imported `TokenForNFT` offer sees two independent UI elements — the per-row "You will receive" amount and the dedicated "Net Proceeds" summary — both displaying `amount` (the full offered price). The seller's actual on-chain receipt is `amount − royaltyAmount`. For a 5 % royalty on a 2 XCH offer the seller sees **2 XCH** in both places but receives **1.9 XCH**. The discrepancy scales linearly with both the royalty percentage and the sale price. Because the seller's decision to accept is directly informed by these figures, the bug causes the user to approve a transaction where the displayed amount is wrong, satisfying the High-impact criterion: *"causes a user to approve … the wrong … amount."*

---

### Likelihood Explanation

The code path is exercised for every `TokenForNFT` offer involving an NFT that carries any non-zero `royaltyPercentage`. NFT royalties are a standard, widely-used feature of Chia NFTs. No special attacker capability is required; any user who creates a legitimate buy-side offer triggers the display bug for the seller. The seller needs only to open the offer in the GUI and click **Accept Offer**.

---

### Recommendation

In `calculateNFTRoyalties`, restore the branch-specific update for `nftSellerNetAmount` in the `TokenForNFT` case, mirroring the pattern used for `totalAmount`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - royaltyAmount).toFixed(12))
    : amount;
```

Also correct `totalAmount` for the same branch — royalties are paid *from* the offered amount, so the buyer's total outlay is `amount + makerFee`, not `amount + makerFee + royaltyAmount`:

```typescript
const totalAmount: number =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount + royaltyAmount
    : amount + makerFee;          // royalties come out of `amount`, not on top
```

---

### Proof of Concept

1. Mint an NFT with `royalty_percentage = 500` (5 %).
2. As the NFT buyer, create a `TokenForNFT` offer: offer **2 XCH** for the NFT.
3. As the NFT seller, open the offer file in `NFTOfferViewer` (imported = true).
4. Observe:
   - **"Net Proceeds"** displays **2 XCH** (tooltip says "asking price minus creator fees").
   - The per-row **"You will receive"** override also shows **2 XCH**.
   - **"Creator Fee (5%)"** correctly shows **0.1 XCH**.
5. Seller clicks **Accept Offer**.
6. On-chain settlement: seller receives **1.9 XCH**; royalty address receives **0.1 XCH**.
7. The seller received **0.1 XCH less** than every GUI element indicated.

The root cause is `nftSellerNetAmount = amount` at [1](#0-0) 
propagating through [2](#0-1) 
and rendered at [3](#0-2)
