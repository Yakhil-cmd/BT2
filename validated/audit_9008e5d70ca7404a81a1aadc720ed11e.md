### Title
`FeeEstimation` RPC Response Omits `l2_gas_consumed`, Making `overall_fee` Description Factually Wrong and Preventing Correct V3 Resource-Bound Setting - (File: crates/apollo_rpc_execution/src/objects.rs)

### Summary

The `FeeEstimation` struct returned by `starknet_estimateFee` and `starknet_simulateTransactions` exposes `l2_gas_price` but omits `l2_gas_consumed`. The `overall_fee` field is populated from `receipt.fee`, which correctly includes the L2 gas cost component (`l2_gas * l2_gas_price`), yet the struct's own doc comment states it equals only `gas_consumed * gas_price + data_gas_consumed * data_gas_price`. For any V3 transaction with non-zero L2 gas, the two sides of that equation diverge, and callers have no way to reconstruct the missing term or set a correct `l2_gas` resource bound.

### Finding Description

`tx_execution_output_to_fee_estimation()` builds the `FeeEstimation` response as follows:

```rust
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),       // L1 gas
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas
    l1_data_gas_price,
    l2_gas_price,                                   // price present …
    // l2_gas_consumed is ABSENT                    // … amount absent
    overall_fee: tx_execution_output.execution_info.receipt.fee,
    unit: tx_execution_output.price_unit,
})
```

`receipt.fee` is computed by `GasVector::cost()`, which sums all three gas components:

```rust
for (gas, price, resource) in [
    (self.l1_gas,      gas_prices.l1_gas_price,      Resource::L1Gas),
    (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
    (self.l2_gas,      tipped_l2_gas_price,           Resource::L2Gas),
] { … }
```

So `overall_fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * (l2_gas_price + tip)`.

The struct's doc comment, however, says:

```rust
/// The total amount of fee. This is equal to:
/// gas_consumed * gas_price + data_gas_consumed * data_gas_price.
```

This description omits the L2 gas term entirely. The same incorrect formula appears in the OpenRPC schema:

```json
"description": "The estimated fee for the transaction … equals to gas_consumed*gas_price + data_gas_consumed*data_gas_price"
```

A developer-acknowledged TODO confirms the missing field:

```rust
// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is
// close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.
```

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction with non-zero L2 gas receives a response where:

1. `overall_fee` is correct but cannot be verified from the returned components — `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price < overall_fee` by exactly `l2_gas_consumed * l2_gas_price`.
2. The caller has no way to determine the `l2_gas` amount to place in the `AllResourceBounds` of the follow-up transaction. Setting it too low causes the transaction to be rejected at pre-validation (`MaxGasAmountTooLow`); setting it too high causes overpayment.
3. The authoritative-looking description of `overall_fee` actively misleads callers into believing L2 gas is not a fee component, which is false for all post-0.13.3 V3 transactions.

This matches: **High — RPC fee estimation returns an authoritative-looking wrong value.**

### Likelihood Explanation

V3 transactions (`AllResourceBounds`) are the current standard for Starknet post-0.13.3. Every such transaction incurs non-zero L2 gas (computation, calldata, events). Any wallet, SDK, or dApp that calls `starknet_estimateFee` to populate resource bounds for a V3 transaction is affected.

### Recommendation

Add `l2_gas_consumed: Felt` to `FeeEstimation` and populate it in `tx_execution_output_to_fee_estimation()`:

```rust
l2_gas_consumed: gas_vector.l2_gas.0.into(),
```

Update the `overall_fee` doc comment and the OpenRPC schema description to reflect the correct formula:

```
overall_fee = gas_consumed * l1_gas_price
            + data_gas_consumed * l1_data_gas_price
            + l2_gas_consumed * l2_gas_price
```

### Proof of Concept

For a V3 transaction with `l2_gas = 740_000` (typical for a simple Cairo 1 invoke) and `l2_gas_price = 100_000_000_000` (0.1 Gwei in FRI):

1. Call `starknet_estimateFee` → response contains `overall_fee = X`, `gas_consumed = A`, `data_gas_consumed = B`, `l1_gas_price = P1`, `l1_data_gas_price = P2`, `l2_gas_price = P3`.
2. Compute `A * P1 + B * P2` → value is `X - 74_000_000_000_000` (the L2 gas cost is missing from the breakdown).
3. The caller cannot determine that `l2_gas_consumed = 740_000` is needed for the `l2_gas` resource bound.
4. If the caller sets `l2_gas.max_amount = 0` (following the misleading description), the transaction is rejected at `check_fee_bounds` with `MaxGasAmountTooLow { resource: L2Gas }`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

pub(crate) fn tx_execution_output_to_fee_estimation(
    tx_execution_output: &TransactionExecutionOutput,
    block_context: &BlockContext,
) -> ExecutionResult<FeeEstimation> {
    let gas_prices = &block_context.block_info().gas_prices;
    let (l1_gas_price, l1_data_gas_price, l2_gas_price) = (
        gas_prices.l1_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l1_data_gas_price(&tx_execution_output.price_unit.into()).get(),
        gas_prices.l2_gas_price(&tx_execution_output.price_unit.into()).get(),
    );

    let gas_vector = tx_execution_output.execution_info.receipt.gas;

    Ok(FeeEstimation {
        gas_consumed: gas_vector.l1_gas.0.into(),
        l1_gas_price,
        data_gas_consumed: gas_vector.l1_data_gas.0.into(),
        l1_data_gas_price,
        l2_gas_price,
        overall_fee: tx_execution_output.execution_info.receipt.fee,
        unit: tx_execution_output.price_unit,
    })
}

    /// Computes the cost (in fee token units) of the gas vector (panicking on overflow).
    pub fn cost(&self, gas_prices: &GasPriceVector, tip: Tip) -> Fee {
        let tipped_l2_gas_price =
            gas_prices.l2_gas_price.checked_add(tip.into()).unwrap_or_else(|| {
                panic!(
                    "Tip overflowed: addition of L2 gas price ({}) and tip ({}) resulted in \
                     overflow.",
                    gas_prices.l2_gas_price, tip
                )
            });

        let mut sum = Fee(0);
        for (gas, price, resource) in [
            (self.l1_gas, gas_prices.l1_gas_price, Resource::L1Gas),
            (self.l1_data_gas, gas_prices.l1_data_gas_price, Resource::L1DataGas),
            (self.l2_gas, tipped_l2_gas_price, Resource::L2Gas),
        ] {
            let cost = gas.checked_mul(price.get()).unwrap_or_else(|| {
                panic!(
                    "{resource} cost overflowed: multiplication of gas amount ({gas}) by price \
                     per unit ({price}) resulted in overflow."
                )
            });
            sum = sum.checked_add(cost).unwrap_or_else(|| {
                panic!(
                    "Total cost overflowed: addition of current sum ({sum}) and cost of \
                     {resource} ({cost}) resulted in overflow."
                )
            });
        }
        sum
    }
