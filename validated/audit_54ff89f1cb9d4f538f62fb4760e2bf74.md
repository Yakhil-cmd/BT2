Audit Report

## Title
Missing `break` After `Disconnected` Event Causes Infinite Task Loop and Channel Flood — (`rs/bitcoin/adapter/src/stream.rs`)

## Summary

The `handle_stream` task loop in `rs/bitcoin/adapter/src/stream.rs` never exits after emitting a `StreamEventKind::Disconnected` event. Any Bitcoin P2P peer that closes its TCP connection after the version handshake causes the spawned task to spin indefinitely, flooding the shared `stream_event_sender` channel with duplicate `Disconnected` events. This starves the router's `tokio::select!` loop of its other branches, stalling ckBTC/ckDOGE block ingestion.

## Finding Description

The `handle_stream` function spawns a Tokio task containing an unconditional `loop`. After a disconnect-class error is detected and a `Disconnected` event is sent, there is no `break` or `return`: [1](#0-0) 

The `tick()` method calls `read_message()`, which calls `try_read()` on the closed socket. A closed TCP socket returns 0 bytes on every subsequent read: [2](#0-1) 

This maps to `StreamError::Io(UnexpectedEof)`. Since `UnexpectedEof` is not `WouldBlock`, `disconnect = true` on every iteration: [3](#0-2) 

The `send(...).await.ok()` call blocks when the channel reaches its capacity of `DEFAULT_CHANNEL_BUFFER_SIZE = 64`: [4](#0-3) 

This creates a steady-state flood: the task blocks on `send`, resumes each time the router drains one slot, then immediately re-sends another `Disconnected` event.

The router's `tokio::select!` loop in `start_main_event_loop` selects over `connection_manager.receive_stream_event()`, `network_message_receiver.recv()`, `blockchain_manager_rx.recv()`, `transaction_manager_rx.recv()`, and `tick_interval.tick()`: [5](#0-4) 

When the `stream_event_sender` channel is continuously saturated with duplicate `Disconnected` events, the `select!` loop preferentially drains those events, starving the `network_message_receiver` branch (carrying block/header data) and the `tick_interval` branch (which drives `connection_manager.tick` and `blockchain_manager.tick`). Each additional peer that closes its connection leaks another spinning task, compounding the starvation.

## Impact Explanation

This is a **High** severity finding. The Bitcoin adapter is a core component of Chain Fusion (ckBTC, ckDOGE). Sustained starvation of the router's event loop halts block ingestion and tick processing for all connected peers, causing ckBTC/ckDOGE to stop processing new blocks. This constitutes a significant Chain Fusion / ck-token security impact with concrete user and protocol harm, matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation

The attacker only needs to be a Bitcoin P2P peer that the adapter connects to — any node on the public Bitcoin network qualifies. No privileged access, no key material, and no consensus-level corruption is required. The trigger is a single TCP `FIN`/`RST` after the version handshake, which is trivially achievable. The attack is repeatable: each new connection that is dropped leaks another spinning task. A small number of such peers (well below the adapter's `max_connections` limit) is sufficient to saturate the channel and stall the router.

## Recommendation

Add `break` immediately after the `Disconnected` event is sent:

```rust
if disconnect {
    stream_event_sender
        .send(StreamEvent {
            address,
            kind: StreamEventKind::Disconnected,
        })
        .await
        .ok();
    break; // <-- add this
}
``` [6](#0-5) 

## Proof of Concept

1. Spawn a `TcpListener` on a local port.
2. Call `handle_stream` with that address as the peer.
3. Accept the connection on the listener side, perform the Bitcoin version handshake, then immediately drop the socket (closing the TCP connection).
4. Collect all `StreamEvent`s from the receiver with a timeout (e.g., 2 seconds).
5. Assert that exactly one `StreamEventKind::Disconnected` is received and the `JoinHandle` completes within the timeout.

Without the fix, the `JoinHandle` never completes and the receiver accumulates an unbounded stream of `Disconnected` events until the channel fills (capacity 64) and the task blocks permanently. The existing test `read_huge_message_from_network` in `stream.rs` demonstrates the test harness pattern needed; a new test following the same structure but dropping the socket after connection would reproduce this bug deterministically. [7](#0-6)
