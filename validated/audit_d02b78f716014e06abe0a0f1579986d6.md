### Title
Unprivileged Caller Can Front-Run `update_rewards` Per-Block Slot to Permanently Deny Block Rewards to Other Stakers — (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in the `StakingRewardsManager` interface is callable by any address, despite the specification designating it as "Only starkware sequencer." The function enforces a single global `last_reward_block` guard that allows exactly one call per block. An attacker who is a staker can front-run the sequencer every block by calling `update_rewards` for their own staker address first, consuming the per-block slot and causing the sequencer's call for the intended staker to revert with `REWARDS_ALREADY_UPDATED`. Because rewards are computed against the current block at call time, the skipped staker's block-N rewards are permanently lost — they cannot be recovered in a later block.

---

### Finding Description

**Root cause — missing caller restriction on `update_rewards`:**

`StakingRewardsManagerImpl::update_rewards` at `src/staking/staking.cairo` lines 1449–1485 begins with:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks: contract not paused
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // global, single slot
    ...
}
```

`general_prerequisites()` enforces only the "contract is not paused" invariant. There is no `assert!(get_caller_address() == sequencer_address, ...)` guard. The spec at `docs/spec.md` line 1645 states access is "Only starkware sequencer," but this is not enforced in code.

**The global `last_reward_block` is a shared, exhaustible resource:**

`last_reward_block` is a single storage slot (not a per-staker mapping). Once any caller sets it to block N, every subsequent call in block N reverts with `REWARDS_ALREADY_UPDATED`. This is structurally identical to the `maxTransferAmountPerDay` in the Teleportation report: a shared quota that any caller can exhaust.

**Attack path:**

1. Attacker stakes the protocol minimum to become a valid staker with non-zero balance (required by the `INVALID_STAKER` check at line 1482).
2. At the start of every block, attacker submits `update_rewards(attacker_address, false)` before the sequencer's transaction lands.
3. The call succeeds: `last_reward_block` is written to the current block number; attacker receives their own block rewards.
4. The sequencer's call for the intended staker (say, staker B) reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker B's block-N rewards are permanently lost. When the sequencer retries in block N+1, `calculate_block_rewards` computes rewards for block N+1 only — block N's rewards are never distributed to staker B.

**Why block-N rewards are unrecoverable:**

`calculate_block_rewards` uses the current block's `avg_block_duration` and epoch data. There is no mechanism to retroactively credit a staker for a block that was skipped. The missed rewards are simply never minted or credited.

---

### Impact Explanation

**Permanent freezing / theft of unclaimed yield — High.**

Every block the attacker front-runs, the targeted staker permanently loses that block's share of inflation rewards. The attacker simultaneously collects their own rewards. Sustained over an epoch, this can drain a meaningful fraction of a competing staker's yield. Because the loss is per-block and irrecoverable, the cumulative impact scales linearly with the duration of the attack.

---

### Likelihood Explanation

**Medium-High.**

- Precondition: attacker must hold the minimum stake — a publicly known, finite threshold.
- Cost: one L2 transaction per block (cheap on Starknet L2).
- Motivation: a competing staker or delegator pool operator has direct financial incentive to suppress rivals' rewards.
- No privileged key, bridge compromise, or external dependency is required.
- The attack is deterministic and repeatable every block.

---

### Recommendation

Add an explicit caller check inside `update_rewards` to restrict it to the designated sequencer address (stored at construction time or via a governance-settable role), consistent with the specification:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, if the sequencer address is not reliably available on-chain, restrict the function to the `ONLY_APP_GOVERNOR` or a dedicated `REWARDS_MANAGER` role that is assigned only to the trusted sequencer account.

---

### Proof of Concept

```
Block N:
  Attacker tx:   update_rewards(attacker_staker_address, false)
                 → succeeds; last_reward_block = N; attacker earns block-N rewards
  Sequencer tx:  update_rewards(victim_staker_address, false)
                 → reverts: REWARDS_ALREADY_UPDATED

Block N+1:
  Sequencer tx:  update_rewards(victim_staker_address, false)
                 → succeeds; victim earns block-(N+1) rewards only
                 → block-N rewards for victim: permanently lost

Repeat every block → victim's cumulative yield is continuously drained.
```
