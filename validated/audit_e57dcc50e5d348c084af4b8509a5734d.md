Audit Report

## Title
Unbounded `ingress_expiries` Loop in `construction_payloads` Enables Single-Request OOM Crash of ICRC-1 Rosetta Node ÔÇö (`rs/rosetta-api/icrc1/src/construction_api/services.rs`)

## Summary
The `construction_payloads` function in the ICRC-1 Rosetta node accepts attacker-controlled `ingress_start` and `ingress_end` values from HTTP request metadata and uses them to drive an unbounded `while` loop that allocates heap memory proportional to `(ingress_end - ingress_start)`. With `ingress_end = u64::MAX`, both existing guards are bypassed, the loop runs approximately 134 million iterations, and the process exhausts heap memory and crashes. A single unauthenticated HTTP request is sufficient to take down the Rosetta node.

## Finding Description

In `construction_payloads` (`rs/rosetta-api/icrc1/src/construction_api/services.rs`), `ingress_start` and `ingress_end` are read directly from attacker-supplied request metadata with no range cap: [1](#0-0) 

The only two guards before the loop are:

1. `ingress_start >= ingress_end` Ôćĺ error (L148ÔÇô152): trivially bypassed by setting `ingress_end = u64::MAX`.
2. `ingress_end < now + ingress_interval` Ôćĺ error (L154ÔÇô158): requires `ingress_end` to be *at least* one interval in the future; `u64::MAX` is far larger and passes. [2](#0-1) 

The loop itself has no iteration cap: [3](#0-2) 

The step per iteration is `ingress_interval - INGRESS_INTERVAL_OVERLAP`. With `MAX_INGRESS_TTL = 300 s`, `PERMITTED_DRIFT = 60 s`, and `INGRESS_INTERVAL_OVERLAP = 120 s`: [4](#0-3) [5](#0-4) 

- `ingress_interval = (300 - 60) ├Ś 10Ôü╣ = 240,000,000,000 ns`
- `step = 240e9 - 120e9 = 120,000,000,000 ns`
- Iterations Ôëł `(u64::MAX Ôłĺ now) / 120e9 Ôëł 134,000,000`

The resulting `ingress_expiries` Vec is then passed to `handle_construction_payloads`, which allocates one `EnvelopeContent::Call` struct and one `SigningPayload` per entry: [6](#0-5) 

Both Vecs are then CBOR+hex-serialized into the `unsigned_transaction` string and returned in `ConstructionPayloadsResponse`, compounding the allocation: [7](#0-6) 

## Impact Explanation

The Rosetta node process exhausts heap memory and is OOM-killed or panics, making the entire ICRC-1 Rosetta API (transaction construction, signing, submission) unavailable. This matches the allowed High impact: *"Application/platform-level DoS, crash ÔÇŽ not based on raw volumetric DDoS"* and *"Significant ÔÇŽ Rosetta ÔÇŽ security impact with concrete user or protocol harm."* Severity: **High ($2,000ÔÇô$10,000)**.

## Likelihood Explanation

The ICRC-1 Rosetta construction API is a public HTTP endpoint requiring no authentication. The exploit requires one crafted JSON request with `ingress_end = 18446744073709551615` (`u64::MAX`). Both guards are bypassed by design. No keys, no prior state, no volumetric traffic, and no special privileges are needed. The attack is trivially repeatable.

## Recommendation

Add a hard cap on the number of ingress expiries before entering the loop, or validate the window size before the loop:

```rust
const MAX_INGRESS_WINDOW_NS: u64 = 24 * 3600 * 1_000_000_000; // 24 hours

if ingress_end.saturating_sub(ingress_start) > MAX_INGRESS_WINDOW_NS {
    return Err(Error::processing_construction_failed(
        &"ingress_end exceeds maximum allowed window of 24 hours",
    ));
}
```

Alternatively, break or error inside the loop once `ingress_expiries.len()` exceeds a safe constant (e.g., 360, covering ~24 h at the nominal interval).

## Proof of Concept

```bash
curl -X POST http://<rosetta-node>:8081/construction/payloads \
  -H 'Content-Type: application/json' \
  -d '{
    "network_identifier": {"blockchain":"Internet Computer","network":"00000000000000020101"},
    "operations": [<valid_transfer_operation>],
    "public_keys": [<valid_public_key>],
    "metadata": {
      "ingress_start": 1700000000000000000,
      "ingress_end": 18446744073709551615
    }
  }'
```

A unit-level reproduction requires no network: call `construction_payloads(ops, Some(meta_with_ingress_end_u64_max), ledger_id, keys, SystemTime::now())` in a test and observe the process OOM before the function returns. A fuzz target over `(ingress_start, ingress_end)` pairs will reliably find this within seconds.
