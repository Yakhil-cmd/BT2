### Title
Missing Validation of `old_signing_key` for New Participants in Reshare Enables Denial of Reshare Protocol — (File: `src/dkg.rs`)

---

### Summary

`assert_reshare_keys_invariants` in `src/dkg.rs` only enforces one direction of the signing-key invariant: it rejects an *old* participant that provides no key, but it never rejects a *new* participant (not in `old_participants`) that provides a non-`None` signing key. That invalid input passes pre-flight validation, is forwarded into `do_reshare`, and causes a runtime `ProtocolError` when the Lagrange coefficient for the new participant is looked up in the intersection set. Because the failing party stops sending protocol messages, all honest co-participants are left waiting indefinitely, permanently aborting the reshare.

---

### Finding Description

`assert_reshare_keys_invariants` contains the following guard:

```rust
// if me is not in the old participant set then ensure that old_signing_key is None
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}
```

The inline comment explicitly states the *intended* invariant — "if me is not in the old participant set then ensure that `old_signing_key` is None" — but the code only enforces the *opposite* direction. The symmetric check is absent:

```rust
// MISSING:
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(...)
}
```

Because the guard is incomplete, a new participant (not in `old_participants`) that supplies a non-`None` `old_signing_key` passes `assert_reshare_keys_invariants` without error. [1](#0-0) 

The validated (but invalid) `old_signing_key` is then forwarded unchanged into `do_reshare`: [2](#0-1) 

Inside `do_reshare`, the secret contribution is computed as:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)          // ← fails: me ∉ intersection
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?                       // ← propagates ProtocolError
    .unwrap_or_else(<C::Group as Group>::Field::zero);
```

Because `me` is not in `old_participants`, it is not in `intersection`. The call to `intersection.lagrange::<C>(me)` returns `Err(ProtocolError::InvalidIndex)` (consistent with `ParticipantList::index` returning that error for unknown participants). The `.transpose()?` propagates the error, causing `do_reshare` to return early before sending any protocol messages. [3](#0-2) [4](#0-3) 

All other honest participants are blocked in `recv_from_others` waiting for the failing party's messages, which never arrive, permanently aborting the reshare session.

---

### Impact Explanation

**High — Permanent denial of reshare for honest parties.**

A single malicious new participant can supply any non-`None` `SigningShare` value, pass the pre-flight check, and then silently abort during `do_reshare`. Every other honest participant in the session is left waiting for messages that will never come, making the reshare uncompletable without restarting the entire session (and excluding the attacker). This maps directly to the allowed impact: *"Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions."*

---

### Likelihood Explanation

**Medium.** Any participant that is newly joining a reshare session (i.e., not present in `old_participants`) can trigger this by passing any non-`None` value for `old_signing_key`. No cryptographic capability or leaked secret is required — only the ability to call the public `reshare` API with a crafted argument. The missing check is a single boolean condition that the library's own comment describes as required, making accidental omission plausible and deliberate exploitation trivial.

---

### Recommendation

Add the symmetric guard immediately after the existing check in `assert_reshare_keys_invariants`:

```rust
// Existing check (old participant must provide a key)
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}

// Missing check (new participant must NOT provide a key)
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is not present in the old participant list but provided a share"
    )));
}
```

This mirrors the pattern already used for the opposite direction and matches the intent expressed in the existing comment. [1](#0-0) 

---

### Proof of Concept

1. Run DKG with participants `[A, B, C]`, threshold `2`, producing `KeygenOutput` for each.
2. Initiate reshare: new participants `[A, B, C, D]`, new threshold `2`.
3. Participant `D` (not in old set) calls `reshare(...)` with `old_signing_key = Some(SigningShare::new(Scalar::ONE))` (any non-`None` value).
4. `assert_reshare_keys_invariants` returns `Ok(...)` — no error is raised.
5. `do_reshare` is entered; `intersection = {A, B, C} ∩ {A, B, C, D} = {A, B, C}`.
6. `intersection.lagrange::<C>(D)` → `Err(ProtocolError::InvalidIndex)`.
7. `.transpose()?` propagates the error; `D`'s protocol future terminates without sending any messages.
8. `A`, `B`, `C` are blocked in `recv_from_others` waiting for `D`'s round-1 message; the reshare never completes.
