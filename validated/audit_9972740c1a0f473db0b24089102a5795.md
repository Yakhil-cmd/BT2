### Title
WalletConnect `enabled` Flag Is Set But Never Enforced in Session Request Processing — Bypassed Commands Execute Without Confirmation After User Disables WalletConnect - (File: `packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`)

### Summary

The `enabled` flag from `useWalletConnectPreferences` is read inside `WalletConnectProvider` but is never checked in `processSessionRequest` or in the `useEffect` that registers the `session_request` handler. A previously-connected dApp whose commands are in the per-pair bypass list can therefore execute wallet transactions (send XCH/CAT, create/cancel offers, sign messages) without any user confirmation dialog, even after the user has toggled WalletConnect off believing they have cut off all dApp access.

### Finding Description

`WalletConnectProvider` is mounted **outside** `AppState` in the component tree:

```tsx
// packages/gui/src/components/app/AppProviders.tsx
<WalletConnectProvider projectId={WalletConnectChiaProjectId}>
  <AppState>{outlet ? <Outlet /> : children}</AppState>
  ...
</WalletConnectProvider>
```

Inside `WalletConnectProvider`, the `enabled` preference is fetched:

```ts
// WalletConnectProvider.tsx line 255
const { enabled } = useWalletConnectPreferences();
```

But `processSessionRequest` — the function that handles every incoming dApp command — never reads `enabled`:

```ts
// WalletConnectProvider.tsx lines 297-390
const processSessionRequest = useCallback(async (event) => {
  // ← no check for `enabled` here
  ...
  const result = await currentProcess(pairTopic, method, commandParams, { mainnet: isMainnet });
  ...
}, []);
```

Likewise, the `useEffect` that registers the `session_request` listener does not gate on `enabled`:

```ts
// WalletConnectProvider.tsx lines 392-425
useEffect(() => {
  if (!client) { return undefined; }
  // ← no check for `enabled` here
  client.on('session_request', handleSessionRequest);
  ...
}, [client, handleDisconnectPair, processSessionRequest, updateListOfPairs]);
```

The `enabled` flag is only consumed in the context value returned to UI consumers (line 533) and in `WalletConnectConnections.tsx` to show/hide the "Add Connection" button — it has zero effect on whether incoming session requests are processed.

When a command is in the pair's `bypass` list, `dispatchPairRequest` skips the confirmation dialog entirely:

```ts
// packages/gui/src/electron/utils/dispatchPairRequest.ts lines 57-58
if (pair.bypass.includes(command)) {
  return process(context);   // ← executes immediately, no confirmation
}
```

### Impact Explanation

A dApp that was previously connected and granted bypass permission for commands such as `chia_sendTransaction`, `chia_createOfferForIds`, `chia_cancelOffer`, or `chia_signMessageById` can continue to trigger those commands silently — without any confirmation dialog — after the user has disabled WalletConnect. The user believes they have revoked dApp access; in reality the session request handler is still active and bypassed commands execute immediately, resulting in unauthorized XCH/CAT transfers, offer creation/cancellation, or message signing against the user's wallet.

### Likelihood Explanation

Any dApp that was previously paired and had bypass permissions granted (a feature explicitly exposed in the UI via "Bypassed Commands" in Settings → Advanced) retains the ability to send commands. The WalletConnect relay connection remains open as long as the client is initialized, which happens unconditionally in `useWalletConnectClient` regardless of the `enabled` preference. The attacker path requires only a prior legitimate pairing with bypass grants — no additional privileges.

### Recommendation

Add an `enabled` guard at the top of `processSessionRequest` and inside the `useEffect` that registers the `session_request` listener. When `enabled` is `false`, respond with a rejection error and do not process the command. Additionally, consider tearing down the WalletConnect client (disconnecting from the relay) when `enabled` is toggled off, so no new session requests can arrive at all.

```ts
const processSessionRequest = useCallback(async (event) => {
  if (!enabled) {
    // respond with rejection and return
    ...
    return;
  }
  ...
}, [enabled]);
```

### Proof of Concept

1. User connects a dApp via WalletConnect and grants bypass permission for `chia_sendTransaction`.
2. User navigates to Settings → Advanced → Bypassed Commands and confirms the bypass is saved.
3. User navigates to Settings → Integrations and disables WalletConnect (`enabled = false`).
4. The dApp sends a `session_request` for `chia_sendTransaction` over the still-open relay connection.
5. `processSessionRequest` fires — it does not check `enabled` — and calls `currentProcess`.
6. `dispatchPairRequest` finds `chia_sendTransaction` in `pair.bypass` and calls `process(context)` directly, skipping the confirmation dialog.
7. The transaction is submitted to the blockchain without the user ever seeing a prompt, despite having "disabled" WalletConnect.
