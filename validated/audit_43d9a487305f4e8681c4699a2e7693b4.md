### Title
Wrong Condition Check on `uint64` in `ComputationRemaining` Causes Underflow-Based Incorrect Gas Limit Return - (`File: fvm/meter/computation_meter.go`)

### Summary
`ComputationRemaining` in `fvm/meter/computation_meter.go` applies the guard `if remainingComputationUsage <= 0` to a `uint64` variable. Because `uint64` is unsigned, this condition is equivalent to `== 0` and never catches the underflow case where `m.computationUsed > m.params.computationLimit`. When that underflow occurs, the function returns a huge wrapped-around value instead of `0`, directly mirroring the wrong-condition-check class of the reported Biconomy finding.

### Finding Description

In `ComputationRemaining` (lines 122–135 of `fvm/meter/computation_meter.go`):

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }

    remainingComputationUsage := m.params.computationLimit - m.computationUsed  // uint64 subtraction
    if remainingComputationUsage <= 0 {   // ← WRONG: uint64 is never < 0; only catches == 0
        return 0
    }

    return remainingComputationUsage / w
}
```

`MeterComputation` (lines 100–104) increments `m.computationUsed` **before** checking the limit:

```go
m.computationUsed += w * intensity
if m.computationUsed > m.params.computationLimit {
    return errors.NewComputationLimitExceededError(...)
}
```

After this error path, `m.computationUsed > m.params.computationLimit`. If `ComputationRemaining` is subsequently called, the subtraction `m.params.computationLimit - m.computationUsed` wraps around to a very large `uint64`. The guard `<= 0` is false (the wrapped value is enormous), so the function returns `huge_value / w` — a completely incorrect, attacker-exploitable gas budget — instead of `0`.

The correct guard must check the source variables, not the derived (potentially underflowed) result:

```go
if m.computationUsed >= m.params.computationLimit {
    return 0
}
```

### Impact Explanation

`ComputationRemaining` is the sole gate used by `fvm/evm/handler/handler.go` `batchRun` to determine how much EVM gas is available before executing a batch of EVM transactions submitted by an unprivileged Cadence transaction:

```go
remainingGasLimit := h.backend.ComputationRemaining(environment.ComputationKindEVMGasUsage)
for i, rlpEncodedTx := range rlpEncodedTxs {
    ...
    if remainingGasLimit < txGasLimit {
        return nil, types.ErrInsufficientComputation
    }
    remainingGasLimit -= txGasLimit
}
```

When the underflow occurs, `remainingGasLimit` is `~2^64`, so the check `remainingGasLimit < txGasLimit` never triggers, and the batch proceeds with effectively unlimited EVM gas — bypassing the computation budget that the FVM enforces for the enclosing Flow transaction. This allows an attacker to execute far more EVM computation than their transaction's gas limit permits, undermining the computation metering that protects Execution nodes from resource exhaustion.

### Likelihood Explanation

`MeterComputation` leaves `computationUsed > computationLimit` in the meter's internal state after returning `ComputationLimitExceededError`. In the normal Cadence execution path the runtime propagates this error and halts the transaction, so `batchRun` is not reached. However, the state is reachable in two concrete scenarios:

1. **Nested/deferred execution**: A Cadence transaction that catches a computation-limit error via a `pre`/`post` condition or a resource destructor callback and then invokes `EVM.batchRun` in that handler — the meter state is already over-limit but execution continues in the cleanup path.
2. **Future callers**: `ComputationRemaining` is a public interface method (`state.Meter`, `environment.Meter`) consumed by any present or future component. Any new caller that reads remaining capacity after a limit-exceeded event will receive a wrong value.

### Recommendation

Replace the dead `<= 0` guard with a correct pre-subtraction check:

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }
    if m.computationUsed >= m.params.computationLimit {   // correct: check before subtraction
        return 0
    }
    return (m.params.computationLimit - m.computationUsed) / w
}
```

### Proof of Concept

```
computationLimit  = 1000 (internal precision units)
computationUsed   = 1001  (after MeterComputation increments past limit and returns error)

remainingComputationUsage = uint64(1000) - uint64(1001)
                          = 18446744073709551615  (uint64 wrap-around)

remainingComputationUsage <= 0  →  false  (18446744073709551615 > 0)

return 18446744073709551615 / w   // ← returned to batchRun as "remaining gas"
```

`batchRun` then sees `remainingGasLimit ≈ 2^64`, passes every `remainingGasLimit < txGasLimit` check, and executes the entire EVM batch without any computation budget enforcement. [1](#0-0) [2](#0-1) [3](#0-2)
