### Title
Hardcoded 2,300-Gas Stipend in EVM Bridge Deposit/Withdraw/Transfer Direct Calls Mirrors Solidity `.transfer()` Vulnerability - (File: `fvm/evm/types/call.go`)

### Summary

`fvm/evm/types/call.go` defines `DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300` (i.e., 21,000 + 2,300 = 23,300 gas), which is used as the fixed gas limit for all bridge deposit, withdraw, and COA-to-COA transfer direct calls. This is the exact EVM analog of Solidity's `.transfer()` stipend: any EVM smart contract whose `receive()` or `fallback()` function consumes more than 2,300 gas will cause the direct call to revert, permanently blocking that address from receiving FLOW tokens via the bridge.

### Finding Description

In `fvm/evm/types/call.go`, three constants are defined:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer
```

These constants are consumed by three direct-call constructors:

- `NewDepositCall(...)` sets `GasLimit: DepositCallGasLimit` (23,300)
- `NewWithdrawCall(...)` sets `GasLimit: WithdrawCallGasLimit` (23,300)
- `NewTransferCall(...)` sets `GasLimit: DefaultGasLimitForTokenTransfer` (23,300)

All three are invoked from `fvm/evm/handler/handler.go` in `Account.Deposit()`, `Account.Withdraw()`, and `Account.Transfer()` respectively. After execution, `panicOnErrorOrInvalidOrFailedState` is called, which panics (reverting the Cadence transaction) if the EVM result is `Failed()` (i.e., `res.VMError != nil`).

The existing test at `fvm/evm/emulator/emulator_test.go:83-111` explicitly demonstrates the failure mode: depositing to a smart contract that does not implement a payable `receive()` function results in `res.VMError == gethVM.ErrExecutionReverted`. Any contract whose `receive()` or `fallback()` uses more than 2,300 gas (multisigs, proxy contracts, ERC-4337 smart wallets, etc.) will trigger the same revert.

The comment in the code itself acknowledges the intent to support smart contract addresses:
```go
// the value is set to the gas limit for transfer to facilitate transfers
// to smart contract addresses.
DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
```
Yet the 2,300 gas ceiling directly contradicts this intent.

### Impact Explanation

Any EVM smart contract address whose `receive()` or `fallback()` function requires more than 2,300 gas is permanently unable to receive FLOW token deposits or COA-to-COA transfers via the bridge. Because `panicOnErrorOrInvalidOrFailedState` reverts the entire Cadence transaction on EVM failure, the bridge deposit path is completely broken for those addresses. This includes:

- Multisig wallets (e.g., Gnosis Safe) that emit events or update state in `receive()`
- Proxy contracts that delegate to an implementation in `receive()`
- ERC-4337 smart wallets with non-trivial fallback logic
- Any contract using more than 2,300 gas in its receive hook

The impact is a permanent, protocol-level denial of the bridge deposit/transfer functionality for a broad class of EVM smart contract addresses — mirroring the Golom finding where the entire protocol breaks due to the same hardcoded gas ceiling.

### Likelihood Explanation

The likelihood is medium-to-high. Multisig wallets and proxy contracts are the dominant form of institutional and protocol-controlled EVM addresses. Any Flow EVM user who deploys a COA-backed smart wallet or uses a proxy pattern will encounter this failure when attempting to receive deposits. The failure is deterministic and reproducible — no special conditions are required beyond deploying a contract with a non-trivial `receive()` function.

### Recommendation

Replace the hardcoded `2_300` gas ceiling with a configurable or sufficiently large gas limit for deposit, withdraw, and transfer direct calls. The `NewDepositCall`, `NewWithdrawCall`, and `NewTransferCall` constructors should accept a caller-supplied or protocol-configured gas limit, or use a higher default (e.g., 100,000 gas) that accommodates complex `receive()` implementations. The `DefaultGasLimitForTokenTransfer` constant should be removed or renamed to reflect that it is not safe for smart contract recipients.

### Proof of Concept

1. Deploy an EVM smart contract with a `receive()` function that uses more than 2,300 gas (e.g., emits an event or writes to storage).
2. From a Cadence transaction, call `EVM.createCadenceOwnedAccount()` and `cadenceOwnedAccount.deposit(from: <-vault)` targeting that contract address.
3. Internally, `Account.Deposit()` in `handler.go` constructs `NewDepositCall(bridge, contractAddress, amount, nonce)` with `GasLimit = DepositCallGasLimit = 23_300`.
4. The EVM executes the deposit; the contract's `receive()` runs out of gas after 2,300 units and reverts.
5. `panicOnErrorOrInvalidOrFailedState` detects `res.Failed() == true` and panics, reverting the entire Cadence transaction.
6. The EVM smart contract address can never receive FLOW tokens via the bridge deposit path.

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
