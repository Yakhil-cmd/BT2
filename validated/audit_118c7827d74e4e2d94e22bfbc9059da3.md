### Title
Malformed Message Causes Unattributed Protocol Abort in `recv_from_others`, Enabling Persistent DoS on DKG / Reshare / Refresh — (File: `src/protocol/helpers.rs`)

---

### Summary

A malicious participant can send a single `send_many` message with a syntactically malformed payload at a fully predictable waitpoint during DKG, reshare, or refresh. The `recv_from_others` helper propagates the resulting deserialization error directly via `?`, aborting the protocol for every honest party simultaneously — without ever identifying the culprit. Because the attacker is never attributed, honest parties cannot exclude them from future runs, enabling a persistent denial of key generation.

---

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` collects exactly one message from every other participant before the caller may proceed:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // ← propagates on error
    if seen.put(from) {
        messages.push((from, msg));
    }
}
``` [1](#0-0) 

`chan.recv` resolves to `Comms::recv`, which pops a raw message from the `MessageBuffer`, strips the header, and attempts `rmp_serde::decode::from_slice`. When deserialization fails, the function returns `Err(DeserializationError(...))` **without exposing the `from` participant**:

```rust
let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
    rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
Ok((from, decoded?))
``` [2](#0-1) 

The `?` in `recv_from_others` immediately propagates this error, terminating the entire `do_keyshare` future for the honest party.

`recv_from_others` is called at two critical points inside `do_keyshare`:

1. **Round 1** — collecting commitment hashes from all participants: [3](#0-2) 

2. **Round 5** — collecting secret signing shares from all participants: [4](#0-3) 

Both call sites are reachable from `do_keygen`, `do_reshare`, and `do_refresh` (via `do_keyshare`). [5](#0-4) 

A malicious participant crafts a `send_many` message with the correct `MessageHeader` (root-shared channel tag + the deterministic waitpoint counter) but a garbage payload (e.g., `[0xFF, 0xFF, …]`). Because `send_many` broadcasts to all participants, **a single malformed message aborts the protocol for every honest party simultaneously**.

The `MessageHeader` values are fully deterministic: `ChannelTag::root_shared()` is a fixed constant, and the waitpoint counter starts at 0 and increments predictably with each `next_waitpoint()` call. [6](#0-5) [7](#0-6) 

By contrast, `reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` handles receive errors with `_ => continue`, silently skipping bad messages. `recv_from_others` has no such resilience:

```rust
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // ← non-blocking: bad messages are skipped
};
``` [8](#0-7) 

This inconsistency is the direct analog of the external report's **Issue B**: the non-blocking pattern is applied in the broadcast layer but not in the point-to-point collection layer, creating an exploitable gap.

---

### Impact Explanation

Every honest participant's DKG, reshare, or refresh instance terminates with `DeserializationError`. No `MaliciousParticipant` error is emitted, so the attacker is never identified. Honest parties cannot exclude the attacker from subsequent runs. The attacker can repeat the attack on every restart, permanently preventing key generation, resharing, and refresh as long as they remain active. This matches the **High** impact category: permanent denial of key generation for honest parties.

---

### Likelihood Explanation

The attack requires only that the malicious participant be included in the participant set — a normal precondition. The `MessageHeader` values (channel tag and waitpoint) are fully deterministic and predictable from the protocol structure alone. A single malformed `send_many` message is sufficient to abort the protocol for all honest parties simultaneously. No special privileges or cryptographic knowledge are required.

---

### Recommendation

In `recv_from_others`, separate the raw message pop from deserialization so that the `from` participant is always known. On deserialization failure, return `ProtocolError::MaliciousParticipant(from)` instead of propagating a generic `DeserializationError`. This mirrors the attribution logic already present in `validate_received_share` and `verify_proof_of_knowledge`. [9](#0-8) 

Alternatively, adopt the same resilience pattern used in `reliable_broadcast_receive_all` (skip bad messages) but augment it with sender attribution so that a participant who repeatedly sends malformed messages can be identified and excluded.

---

### Proof of Concept

1. Participants A (honest), B (honest), C (malicious) initiate `do_keygen`.
2. `do_keyshare` calls `recv_from_others` at `wait_round_1` (the waitpoint assigned after the two preceding `do_broadcast` calls).
3. C sends a `send_many` message with the correct `MessageHeader` (root-shared channel, the known waitpoint value) but a payload of `[0xFF, 0xFF, 0xFF, …]` (invalid msgpack).
4. A and B each pop C's message from their respective queues. `rmp_serde::decode::from_slice` fails. `Comms::recv` returns `Err(DeserializationError(...))`. The `?` in `recv_from_others` propagates the error.
5. Both A and B's `do_keyshare` futures terminate with `DeserializationError`. No `MaliciousParticipant(C)` is emitted.
6. A and B restart the protocol. C repeats step 3. The protocol never completes. [10](#0-9) [11](#0-10)

    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.

    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}
