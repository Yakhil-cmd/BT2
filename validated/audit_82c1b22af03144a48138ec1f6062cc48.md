### Title
CR-CAT Silently Dropped from WalletConnect Offer Confirmation Preview — (`File: packages/gui/src/util/createOfferForIdsToOfferBuilderData.ts`)

### Summary

`createOfferForIdsToOfferBuilderData` converts a `walletIdsAndAmounts` map (the raw payload of a `chia_createOfferForIds` / `chia_wallet.create_offer_for_ids` WalletConnect command) into the `OfferBuilderData` structure used to render the user-facing confirmation dialog. The switch statement that maps wallet types to offer sections handles `WalletType.CAT` and `WalletType.RCAT` but is missing `WalletType.CRCAT`. Any CR-CAT wallet ID in the dApp-supplied payload silently falls through to `default: break` and is omitted from the rendered preview. The underlying RPC call still receives the full, unmodified payload, so the offer that is actually submitted to the daemon includes the CR-CAT component that the user never saw.

### Finding Description

In `createOfferForIdsToOfferBuilderData`, the wallet-type dispatch is:

```ts
switch (asset.walletType) {
  case WalletType.STANDARD_WALLET:
    section.xch.push(...);
    break;
  case WalletType.CAT:
  case WalletType.RCAT:
    section.tokens.push(...);
    break;
  default:
    break;   // ← CRCAT (57) falls here, silently discarded
}
``` [1](#0-0) 

Every other part of the codebase that enumerates CAT-family wallet types includes `WalletType.CRCAT`:

- `Wallet.tsx` renders CR-CAT wallets: `[WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(wallet.type)` [2](#0-1) 
- `NFTOfferTokenSelector.tsx` lists CR-CAT in the token selector: `.filter((wallet) => [WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(wallet.type))` [3](#0-2) 
- `createDefaultValues.ts` pre-populates CR-CAT offers: `[WalletType.CAT, WalletType.RCAT, WalletType.CRCAT].includes(walletType)` [4](#0-3) 

The `chia_wallet.create_offer_for_ids` WalletConnect command is registered with a `dapp` entry (`chia_createOfferForIds`) and passes the raw `offer` map directly to the daemon without a custom handler. [5](#0-4) 

`createOfferForIdsToOfferBuilderData` is the function that converts that raw map into the human-readable confirmation view. Because `CRCAT` is absent from the switch, the confirmation dialog renders an offer that is missing the CR-CAT line item entirely, while the daemon receives and executes the full offer including the CR-CAT wallet ID.

The existing test suite confirms the gap: tests cover `STANDARD_WALLET`, `CAT`, `RCAT`, and an "unknown wallet type" (DID = 8) that is expected to be ignored — but there is no test for `CRCAT = 57`, and the "unknown wallet type" test explicitly validates that unrecognised types are silently dropped, which is exactly what happens to CR-CAT. [6](#0-5) 

### Impact Explanation

A connected WalletConnect dApp sends `chia_createOfferForIds` with a payload such as:

```json
{ "offer": { "1": -5000000000000, "7": -1000 }, "driver_dict": {}, "fee": 0 }
```

where wallet `7` is the user's CR-CAT wallet. The confirmation dialog renders only the XCH line (wallet `1`). The user sees "Offer 5 XCH" and approves. The daemon creates an offer that also locks 1000 mojos of CR-CAT. The user has unknowingly committed CR-CAT tokens to an offer they did not review.

This matches the **High** impact category: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation

Any WalletConnect-connected dApp can trigger this with a single crafted `chia_createOfferForIds` call. The user must have already approved the WalletConnect pair and granted the `chia_createOfferForIds` command permission, which is a normal part of dApp onboarding. No key leakage, host compromise, or cryptographic break is required. CR-CAT wallets are a live, supported wallet type (`WalletType.CRCAT = 57`). [7](#0-6) 

### Recommendation

Add `WalletType.CRCAT` to the switch statement in `createOfferForIdsToOfferBuilderData.ts`:

```ts
case WalletType.CAT:
case WalletType.RCAT:
case WalletType.CRCAT:
  section.tokens.push({ amount: mojoToCAT(numericValue.abs()).toFixed(), assetId: asset.assetId });
  break;
```

Add a corresponding unit test with `walletType: 57` (CRCAT) to the test file to prevent regression. [8](#0-7) 

### Proof of Concept

1. Establish a WalletConnect session with a wallet that holds both XCH and CR-CAT tokens; grant `chia_createOfferForIds` permission.
2. From the dApp, call:
   ```json
   {
     "method": "chia_createOfferForIds",
     "params": {
       "offer": { "1": -5000000000000, "<crcat_wallet_id>": -1000 },
       "driver_dict": {},
       "fee": 0
     }
   }
   ```
3. Observe the confirmation dialog: only the XCH amount is displayed; the CR-CAT line is absent.
4. Approve the offer.
5. Inspect the resulting trade record via `get_all_offers` — the offer locks both XCH and CR-CAT, confirming the CR-CAT was included in the actual RPC call despite being invisible in the confirmation UI.
