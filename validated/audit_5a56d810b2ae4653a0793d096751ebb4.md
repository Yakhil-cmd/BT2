### Title
ETH Sent to `tick()` Is Permanently Locked When `amountToMint == 0` — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is `payable` and accepts ETH to cover L1→L2 message fees. However, when `requiredMinting()` returns `amountToMint == 0`, the entire `msg.value` is silently trapped in the contract with no refund and no withdrawal path. This is a direct analog to the reported M-20 class: excess ETH paid for a cross-chain operation is not returned to the sender.

---

### Finding Description

`tick()` is declared `external payable` with no access control — any address may call it and send ETH: [1](#0-0) 

The function first calls `requiredMinting()` to determine how many L2→L1 mint-request messages are pending: [2](#0-1) 

All ETH-consuming logic is gated behind `if (amountToMint > 0)`: [3](#0-2) 

When `amountToMint == 0` the `if` block is skipped entirely. `msg.value` is never forwarded to any call and never refunded. The contract has no `receive()`, no `fallback()`, and no withdrawal function, so the ETH is permanently locked.

A secondary issue exists even when `amountToMint > 0`: the fee is split mechanically as `msg.value / 2` for each of the two outbound calls: [4](#0-3) 

The actual fees consumed by `depositWithMessage` and `sendMessageToL2` are determined by the StarkGate/messaging contracts at call time and may be less than `msg.value / 2`. Any surplus is not refunded to the caller.

---

### Impact Explanation

**Severity: High — Permanent freezing / direct loss of caller funds.**

Any ETH sent to `tick()` when no mint requests are pending is permanently locked in the contract. Because `tick()` is permissionless and the presence of pending mint requests is a runtime condition outside the caller's control, a caller cannot reliably predict whether their ETH will be consumed or trapped. The contract provides no recovery path.

---

### Likelihood Explanation

`tick()` is a public, permissionless function intended to be called by keepers or any external actor. The condition `amountToMint == 0` is common: it occurs whenever the L2 reward supplier has not yet emitted a new mint-request message (e.g., between reward epochs). Any caller who sends ETH during such a window loses it permanently.

---

### Recommendation

1. **Refund unused ETH**: After the `if (amountToMint > 0)` block, return any remaining balance to the caller:
   ```solidity
   if (address(this).balance > 0) {
       (bool ok, ) = msg.sender.call{value: address(this).balance}("");
       require(ok, "ETH_REFUND_FAILED");
   }
   ```
2. **Alternatively**, revert early if `amountToMint == 0` and `msg.value > 0`:
   ```solidity
   require(amountToMint > 0 || msg.value == 0, "NO_MINT_REQUIRED");
   ```
3. For the split-fee case, measure actual ETH consumed by each sub-call and refund the difference.

---

### Proof of Concept

1. No pending L2 mint-request messages exist (normal inter-epoch state).
2. A keeper (or any address) calls `tick{value: 1 ether}()`.
3. `requiredMinting()` returns `(0, 0)` → `amountToMint == 0`.
4. The `if` block is skipped; `msg.value = 1 ether` is never used.
5. The contract holds 1 ETH with no mechanism to recover it.
6. The caller has permanently lost 1 ETH. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
