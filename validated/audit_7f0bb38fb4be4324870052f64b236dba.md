### Title
Pending stop-loss parameter proposals execute under stale (shorter) timelock after timelock increase — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` stores an absolute `executeAfter` timestamp at proposal time by snapshotting `block.timestamp + currentTimelock`. When the pool admin subsequently increases the timelock, all already-pending proposals (drawdown, decay, high watermarks) retain their old, shorter `executeAfter` and become executable before the new timelock elapses. A semi-trusted pool admin can exploit this to disable LP stop-loss protection faster than LPs can observe and exit.

---

### Finding Description

Every propose function calls `_afterTimelock`:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

The returned value is stored as the absolute deadline:

```solidity
// L106-108  (proposeOracleStopLossDrawdown)
uint32 executeAfter = _afterTimelock(pool_);
sched.pendingDrawdownE6 = uint32(newMaxDrawdownE6);
sched.pendingDrawdownExecuteAfter = executeAfter;
```

The same pattern applies to `proposeOracleStopLossDecay` (L133-135) and `proposeOracleStopLossHighWatermarks` (L162-164). The timelock itself is also mutable via `proposeOracleStopLossTimelock` / `executeOracleStopLossTimelock`, and that proposal also uses `_afterTimelock` (L80), meaning both the timelock change and any co-pending parameter change become executable at the same absolute time.

**Attack sequence** (current timelock = `T_short`, e.g. 1 day):

1. Admin calls `proposeOracleStopLossDrawdown(pool, 0)` → `pendingDrawdownExecuteAfter = now + T_short`.
2. Admin calls `proposeOracleStopLossTimelock(pool, T_long)` (e.g. 30 days) → `pendingTimelockExecuteAfter = now + T_short` (uses the still-current short timelock).
3. LPs observe the timelock-increase proposal on-chain and believe any pending or future parameter changes will require 30 days to execute.
4. After `T_short` elapses, admin calls `executeOracleStopLossTimelock` → timelock is now 30 days.
5. Admin immediately calls `executeOracleStopLossDrawdown` → `drawdown = 0`, stop-loss disabled.

LPs had only `T_short` (1 day) to react, not the 30 days they inferred from the timelock-increase event. The stored `executeAfter` for the drawdown proposal was never updated when the timelock changed.

---

### Impact Explanation

`drawdownE6 = 0` causes `_afterSwapOracleStopLoss` to return immediately without any watermark check:

```solidity
// L216-217
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;
```

With the stop-loss disabled, swaps that drain LP value per share below any previously established watermark are no longer blocked. LPs suffer direct loss of principal through unchecked adverse swaps against a potentially manipulated or stale oracle price. The same bypass applies to `decayPerSecondE8` (setting it to `1e8` collapses all watermarks to zero within one second) and to high-watermark resets that permanently suppress the guard for a bin.

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted — trusted only within caps and timelocks. Bypassing the timelock is exactly the admin-boundary break the contest scope targets. The attack requires no external actors, no non-standard tokens, and no off-chain data. A single admin EOA or compromised multisig can execute all steps atomically across two blocks separated by `T_short`.

---

### Recommendation

Store the proposal timestamp instead of the absolute deadline, and compute `executeAfter` dynamically at execution time using the then-current timelock:

```solidity
// In PoolStopLossSchedule, replace pendingDrawdownExecuteAfter with pendingDrawdownProposedAt
// In proposeOracleStopLossDrawdown:
sched.pendingDrawdownProposedAt = uint32(block.timestamp);

// In executeOracleStopLossDrawdown:
uint32 executeAfter = sched.pendingDrawdownProposedAt + oracleStopLossConfig[pool_].timelock;
_requireElapsed(executeAfter);
```

This mirrors the fix recommended for StakedlvlUSD: store the start time, derive the end time at execution using the live duration. Any subsequent timelock increase then automatically extends the effective deadline for all pending proposals.

---

### Proof of Concept

```solidity
function testTimelockBypassViaTimelockIncrease() public {
    // Pool initialized with 1-day timelock
    uint32 SHORT = 1 days;
    uint32 LONG  = 30 days;

    // Step 1: admin proposes disabling stop-loss (drawdown = 0)
    vm.prank(admin);
    ext.proposeOracleStopLossDrawdown(pool, 0);
    // executeAfter stored = now + 1 day

    // Step 2: admin simultaneously proposes timelock increase to 30 days
    vm.prank(admin);
    ext.proposeOracleStopLossTimelock(pool, LONG);
    // pendingTimelockExecuteAfter = now + 1 day (uses current short timelock)

    // LPs see the timelock-increase event and believe they have 30 days to react.

    // Step 3: warp 1 day — both proposals are now executable
    vm.warp(block.timestamp + SHORT);

    // Step 4: execute timelock increase
    vm.prank(admin);
    ext.executeOracleStopLossTimelock(pool);
    assertEq(ext.oracleStopLossConfig(pool).timelock, LONG);

    // Step 5: execute drawdown disable — succeeds despite new 30-day timelock
    vm.prank(admin);
    ext.executeOracleStopLossDrawdown(pool);
    assertEq(ext.oracleStopLossConfig(pool).drawdownE6, 0);

    // Stop-loss is now disabled after only 1 day.
    // LPs had no 30-day window to exit.
}
```

  function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);

  function executeOracleStopLossTimelock(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingTimelockExecuteAfter == 0) revert OracleStopLossNoPendingTimelock(pool_);
    _requireElapsed(sched.pendingTimelockExecuteAfter);
    uint32 timelock = sched.pendingTimelock;
    oracleStopLossConfig[pool_].timelock = timelock;
    (sched.pendingTimelock, sched.pendingTimelockExecuteAfter) = (0, 0);
    emit OracleStopLossTimelockSet(pool_, timelock);
  }
