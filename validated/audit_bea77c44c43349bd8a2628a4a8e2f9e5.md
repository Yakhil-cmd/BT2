### Title
Sub-minimum EVM Balance Permanently Locked in COA When Total Balance Falls Below 1e10 attoFlow - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

The `CadenceOwnedAccount.withdraw()` function enforces a hard minimum of `1e10 attoFlow` (0.00000001 FLOW) per withdrawal, matching the precision floor of Cadence's `UFix64` type. There is no alternative withdrawal path. When a COA's entire EVM balance falls below this threshold — a reachable condition through normal EVM gas fee deductions — the remaining balance becomes permanently unwithdrawable to Cadence, with no recovery mechanism.

### Finding Description

`CadenceOwnedAccount` in `fvm/evm/stdlib/contract.cdc` exposes a single bridge-out path: `withdraw(balance: Balance)`. [1](#0-0) 

This calls `InternalEVM.withdraw()`, which is implemented in `fvm/evm/impl/impl.go` as `newInternalEVMTypeWithdrawFunction`. Before executing the withdrawal, it enforces: [2](#0-1) 

The guard calls `AttoFlowBalanceIsValidForFlowVault`, defined in `fvm/evm/types/balance.go`: [3](#0-2) 

`UFixToAttoConversionMultiplier` is `10^(18-8) = 10^10`, so any requested withdrawal amount strictly less than `10,000,000,000 attoFlow` is rejected with `ErrWithdrawBalanceRounding`: [4](#0-3) 

The same guard is applied at the emulator layer: [5](#0-4) 

**Root cause**: EVM balances use 18-decimal-place precision (attoFlow), while Cadence `UFix64` only supports 8 decimal places. The bridge enforces the UFix64 floor on every withdrawal. There is no "drain all" or "sweep" function that bypasses this floor when the COA's total balance is itself below the minimum.

**How a COA balance falls below the minimum**: EVM gas fees are deducted in attoFlow from the COA's EVM balance. A COA that holds exactly `1e10 attoFlow` and executes any EVM call (even a simple transfer) will have its balance reduced by the gas cost, leaving a remainder in the range `[1, 9_999_999_999]` attoFlow. At that point:

- Requesting withdrawal of the exact remaining balance fails `AttoFlowBalanceIsValidForFlowVault` (amount < 1e10).
- Requesting withdrawal of `1e10 attoFlow` fails at the EVM level (insufficient balance).
- There is no other Cadence-accessible exit path.

The existing test confirms this behavior is known but treats it as a design constraint rather than a recovery gap: [6](#0-5) 

### Impact Explanation

Any FLOW tokens stranded in a COA whose total EVM balance is in the range `[1, 9_999_999_999]` attoFlow are permanently unrecoverable to Cadence. The maximum per-COA loss is `9,999,999,999 attoFlow` (< `0.00000001 FLOW`). While individually small, this is a structural locked-funds condition with no protocol-level escape hatch, directly analogous to the stableswap finding where the judge held that "locked funds in edge cases" warrants Medium severity even when the developer considers it by-design.

### Likelihood Explanation

The condition is reachable by any unprivileged transaction sender:

1. Create a COA via `EVM.createCadenceOwnedAccount()`.
2. Deposit the minimum Cadence-representable amount (`0.00000001 FLOW` = `1e10 attoFlow`).
3. Execute any EVM call (e.g., a zero-value transfer) that consumes gas — even `21,000` gas at the minimum gas price leaves a sub-minimum remainder.
4. The COA's entire balance is now permanently locked.

Additionally, any EVM EOA or contract can send an arbitrary attoFlow amount directly to a COA's EVM address, forcing a sub-minimum balance onto a COA that previously had zero balance.

### Recommendation

Add a `drainToEVM(to: EVMAddress)` or equivalent function on `CadenceOwnedAccount` that transfers the entire EVM balance to another EVM address (bypassing the UFix64 floor, since no Cadence vault is involved). Alternatively, add a protocol-level "sweep" that converts sub-minimum remainders to zero by crediting them to a designated burn address, preventing permanent lock-up.

### Proof of Concept

```cadence
import EVM from 0xEVM
import FlowToken from 0xFlowToken

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin)!

        // Deposit exactly the minimum bridgeable amount: 1e10 attoFlow
        let minter <- admin.createNewMinter(allowedAmount: 0.00000001)
        let vault <- minter.mintTokens(amount: 0.00000001)
        destroy minter

        let coa <- EVM.createCadenceOwnedAccount()
        coa.deposit(from: <-vault)
        // COA balance: 10_000_000_000 attoFlow

        // Execute any EVM call that costs gas (e.g., 21_000 gas units).
        // After gas deduction, balance = 10_000_000_000 - gasCost < 10_000_000_000.
        // Assume balance is now 9_999_979_000 attoFlow (example).

        // Attempt to withdraw the entire remaining balance:
        let remaining = coa.balance()  // e.g. Balance(attoflow: 9_999_979_000)

        // PANICS: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"
        // because 9_999_979_000 < 10_000_000_000
        let recovered <- coa.withdraw(balance: remaining)
        destroy recovered
        destroy coa
    }
}
```

The panic is produced by `AttoFlowBalanceIsValidForFlowVault` returning `false` for any value below `UFixToAttoConversionMultiplier` (`1e10`), causing `newInternalEVMTypeWithdrawFunction` to `panic(types.ErrWithdrawBalanceRounding)`. [7](#0-6) [2](#0-1)
