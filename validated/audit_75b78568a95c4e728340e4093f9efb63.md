### Title
STRK Rewards Permanently Locked in Pool Contract When Total Delegation Falls Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When the total pool delegation falls below `min_delegation_for_rewards`, the staking contract still physically transfers STRK reward tokens to the pool contract, but the pool contract records zero rewards per unit. Those STRK tokens are permanently locked inside the pool contract because no pool member can claim them and no withdrawal function exists.

---

### Finding Description

In `pool.cairo`, `compute_rewards_per_unit` short-circuits to `Zero::zero()` when `total_stake < min_delegation_for_rewards`: [1](#0-0) 

The code comment on that function explicitly states:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case.

The execution path that causes the lock:

1. The staking contract's `_update_rewards` calculates non-zero `total_pools_rewards` and immediately claims them from the reward supplier and transfers them to the pool contract via `send_rewards_to_delegation_pool`: [2](#0-1) [3](#0-2) 

2. The staking contract then calls `update_rewards_from_staking_contract` on the pool with `rewards > 0` and `pool_balance < min_delegation_for_rewards`: [4](#0-3) 

3. `compute_rewards_per_unit` returns `0`, so the cumulative rewards trace receives a zero increment. The STRK tokens are physically inside the pool contract but the per-unit index never advances, making them unclaimable by any pool member.

4. The pool contract exposes no withdrawal or rescue function. The only token-exit paths are `exit_delegation_pool_action` (returns `unpool_amount` principal) and `claim_rewards` (returns rewards computed from the cumulative index). Neither can recover tokens that were never attributed to the index.

The flow test `DelegatorDidntUpdateAfterStakerUpdateCommissionFlow` independently confirms that a non-trivial STRK balance can remain in the pool after all parties have fully exited: [5](#0-4) 

---

### Impact Explanation

STRK tokens that were minted as protocol rewards and transferred to the pool contract are permanently frozen. No pool member can claim them (the cumulative rewards trace records `0` for those epochs), and no privileged or unprivileged function exists to recover them. This is **permanent freezing of unclaimed yield** — the yield was calculated, minted, and delivered to the pool, but is irrecoverable.

**Allowed impact matched**: High — Permanent freezing of unclaimed yield.

---

### Likelihood Explanation

The condition `total_pool_balance < min_delegation_for_rewards` (1 STRK = `10^18` wei for STRK pools) is reachable by any unprivileged actor:

- A delegator can enter a pool with exactly `min_delegation_for_rewards - 1` wei, causing every subsequent reward epoch to lock tokens.
- A delegator can partially exit, reducing the total pool balance below the threshold while leaving a residual delegation, trapping all future rewards.
- In a pool where all but one delegator have exited and the last delegator holds a sub-threshold amount, every attestation epoch permanently locks more STRK.

No privileged access is required; any delegator controlling the pool's total balance can trigger this state.

---

### Recommendation

Two complementary fixes mirror the `withdrawUnallocatedIntx` pattern from the external report:

1. **In the staking contract**: Before calling `send_rewards_to_delegation_pool`, check whether the pool balance is below `min_delegation_for_rewards` for the relevant token and skip the transfer (and the corresponding `claim_from_reward_supplier` call) when it is.

2. **In the pool contract**: Add a permissioned function (e.g., callable by the governance admin or after `staker_removed == true`) that transfers any STRK balance in excess of the sum of all members' `unpool_amount` values to a designated recovery address, analogous to `withdrawUnallocatedIntx`.

---

### Proof of Concept

1. Staker stakes and opens a STRK pool (`set_open_for_delegation`).
2. Delegator calls `enter_delegation_pool` with `amount = STRK_CONFIG.min_for_rewards - 1` (i.e., `10^18 - 1` wei).
3. Staker attests; `_update_rewards` calculates non-zero `total_pools_rewards`, claims them from the reward supplier, and transfers them to the pool contract.
4. `update_rewards_from_staking_contract` is called on the pool; `compute_rewards_per_unit` returns `0`; the cumulative trace does not advance.
5. Delegator calls `claim_rewards` → receives `0`.
6. Delegator calls `exit_delegation_pool_intent` + `exit_delegation_pool_action` → receives only principal back.
7. Staker calls `unstake_intent` + `unstake_action`.
8. All parties have exited; the pool contract still holds the STRK reward tokens with no function able to move them. [1](#0-0) [4](#0-3) [6](#0-5)

```text
        /// Compute the rewards for the pool trace.
        ///
        /// `staking_rewards` is in `STRK_DECIMALS` decimals.
        /// `total_stake` is in the contract's token decimals.
        /// **Note**: Delegation rewards lost when pool balance is less than
        /// `min_delegation_for_rewards`. The staking contract continues to forward
        /// `pool_rewards` to the pool contract even in this case.
        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
```
