### Title
`PresignOutput` Not Consumed After Use Enables Presignature Reuse and Secret Key Extraction — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both `ot_based_ecdsa::PresignOutput` and `robust_ecdsa::PresignOutput` derive `Clone` and are accepted by `rerandomize_presign` via a shared reference (`&PresignOutput`), not by value. The library therefore provides no API-level enforcement of the one-time-use invariant that the protocol requires. A malicious coordinator can drive two signing sessions against the same presignature material, producing two signatures whose nonces are algebraically related, and recover the aggregate secret key.

---

### Finding Description

Both ECDSA schemes expose the same structural defect:

**OT-based ECDSA** (`src/ecdsa/ot_based_ecdsa/mod.rs`):
- `PresignOutput` is declared `#[derive(Debug, Clone, …)]` at line 40 — it is freely copyable.
- `RerandomizedPresignOutput::rerandomize_presign` accepts `presignature: &PresignOutput` at line 66-67 — it borrows, never consumes.

**Robust ECDSA** (`src/ecdsa/robust_ecdsa/mod.rs`):
- `PresignOutput` is declared `#[derive(Debug, Clone, …)]` at line 26 — freely copyable.
- `RerandomizedPresignOutput::rerandomize_presign` accepts `presignature: &PresignOutput` at line 55-56 — borrows, never consumes.

Because `rerandomize_presign` takes a reference, the caller retains the original `PresignOutput` after the call and can invoke `rerandomize_presign` again with different `RerandomizationArguments` (different `msg_hash`, `tweak`, or `entropy`). Each call produces a distinct `RerandomizedPresignOutput` from the same underlying nonce shares `(k_i, sigma_i)` / `(alpha_i, beta_i, c_i, e_i)`.

The rerandomization for OT-based ECDSA computes:

```
delta  = HKDF(public_key, tweak, msg_hash, big_r, participants, entropy)
R'     = delta * R
k'_i   = k_i * delta^{-1}
sigma'_i = (sigma_i + tweak * k_i) * delta^{-1}
```

Two sessions with deltas `delta_1` and `delta_2` yield nonces satisfying `k'_1 = k'_2 * (delta_2 / delta_1)`. Both `delta` values are publicly derivable from the public `RerandomizationArguments`. This is a known-ratio nonce reuse, sufficient for full secret key recovery.

The security considerations in `docs/ecdsa/robust_ecdsa/signing.md` (lines 147–177) explicitly document the danger and recommend "Never reuse a presignature," but the code imposes no such constraint. The analog to the reported bug is exact: the presignature's "used" state is never recorded or enforced, so the operation can be repeated arbitrarily.

---

### Impact Explanation

Two signatures `(R_1, s_1)` and `(R_2, s_2)` produced from the same presignature satisfy:

```
k'_1 = k'_2 * (delta_2 / delta_1)   [publicly known ratio]
```

Standard ECDSA nonce-reuse algebra then yields the aggregate secret key `x`:

```
x = (s_2 * delta_2 * h_1 - s_1 * delta_1 * h_2)
    / (s_1 * delta_1 * r_2 - s_2 * delta_2 * r_1)
```

All quantities on the right-hand side are public. This is a **Critical** impact: full extraction of the aggregate secret signing key.

---

### Likelihood Explanation

The malicious coordinator role is explicitly in scope. A coordinator controls which `RerandomizationArguments` are distributed to participants. Participants call `rerandomize_presign` locally with coordinator-supplied arguments and have no way to detect that the same presignature is being reused across two sessions. The coordinator needs only to run two signing sessions against the same presignature pool entry — a single API call sequence with no special privileges beyond the coordinator role.

---

### Recommendation

1. **Consume `PresignOutput` by value** in `rerandomize_presign`:
   ```rust
   pub fn rerandomize_presign(
       presignature: PresignOutput,   // take ownership, not &PresignOutput
       args: &RerandomizationArguments,
   ) -> Result<Self, ProtocolError>
   ```
   This makes the Rust type system enforce single-use semantics: once rerandomized, the presignature is moved and cannot be used again.

2. **Remove `Clone` from `PresignOutput`** in both schemes, or at minimum document that cloning breaks the one-time-use guarantee and gate it behind an `unsafe` or explicitly named method.

3. Similarly, `RerandomizedPresignOutput` derives `Clone` in both modules (lines 54 and 42 respectively), allowing the rerandomized output itself to be fed into `sign` twice. Remove `Clone` or consume it by value in `sign`.

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
