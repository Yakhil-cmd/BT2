### Title
Missing Minimum Threshold Validation in FROST `assert_sign_inputs` Produces Unusable Signing Outputs — (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` function in `src/frost/mod.rs` validates signing parameters but omits the minimum-threshold check (`threshold >= 2`) that is consistently enforced in every other protocol entry point in the library. A malicious coordinator can invoke FROST signing with `threshold=1`, causing the protocol to produce a cryptographically invalid signature. Honest participants contribute their partial signatures and accept the combined output as the result of the protocol, receiving an unusable cryptographic output.

---

### Finding Description

**Root cause — missing lower-bound check in `assert_sign_inputs`**

`assert_sign_inputs` in `src/frost/mod.rs` performs the following threshold validation:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [1](#0-0) 

There is **no corresponding lower-bound check** (`threshold < 2`). Compare this with the check that is present in every other protocol entry point:

- `assert_key_invariants` (DKG, `src/dkg.rs`):

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [2](#0-1) 

- `validate_triple_inputs` (OT triple generation, `src/ecdsa/ot_based_ecdsa/triples/generation.rs`):

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
``` [3](#0-2) 

The `ThresholdTooSmall` error variant exists precisely for this purpose but is never raised by `assert_sign_inputs`. [4](#0-3) 

**Exploit path**

Because `ReconstructionLowerBound` is not a type that enforces a minimum of 2 at construction time (the application-level checks in DKG and triple generation would be redundant if it did), a caller can construct a `ReconstructionLowerBound` with value `1` and pass it to the FROST signing entry point. `assert_sign_inputs` accepts it without error.

With `threshold=1` in FROST signing:

1. The coordinator selects a single-participant subset `S = {i}`.
2. Participant `i` computes its Lagrange coefficient over `S`: `λ_i(S) = 1` (trivial interpolation over one point).
3. Participant `i` computes partial signature `s_i = r_i + c · 1 · x_i`, where `x_i` is its **share** of the secret key, not the full secret key.
4. The coordinator outputs `(R, s_i)` as the final signature.
5. Verification requires `s_i · G = R + c · PK`. This holds only if `x_i · G = PK`, i.e., `x_i` equals the full secret key — which is false for any key generated with `threshold ≥ 2` (the only keys the library can produce, since DKG enforces `threshold ≥ 2`).

The output signature is always cryptographically invalid when `threshold=1` is used in signing but `threshold ≥ 2` was used in key generation. [5](#0-4) 

---

### Impact Explanation

Honest participants contribute their nonces and partial signatures to the protocol and receive back an unusable cryptographic output — a signature that will fail verification against the group public key. This is a direct corruption of signing outputs. The impact maps to:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

The honest parties have no in-protocol mechanism to detect that the coordinator passed a wrong threshold; they only discover the failure when the signature is presented to a verifier.

---

### Likelihood Explanation

The coordinator role is a reachable, documented participant role in the FROST protocol. A malicious coordinator is an explicitly considered threat model for threshold protocols. The coordinator controls the `threshold` argument passed to the signing entry point. No privileged access, leaked keys, or cryptographic breaks are required — only the ability to call the public signing API with a crafted parameter value. [6](#0-5) 

---

### Recommendation

Add the missing lower-bound check to `assert_sign_inputs`, consistent with every other protocol entry point in the library:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `participants.len() < 2` check and before the upper-bound check, mirroring the structure in `assert_key_invariants`. [7](#0-6) 

---

### Proof of Concept

1. Run DKG with 5 participants and `threshold=3`. All participants receive valid key shares. Key generation succeeds (threshold ≥ 2 check passes).
2. Call the FROST signing entry point with the same 5 participants but `threshold=1`. `assert_sign_inputs` accepts this without error — no `ThresholdTooSmall` is raised.
3. The coordinator selects participant `P1` as the sole signer. `P1` computes `λ_{P1}({P1}) = 1` and produces partial signature `s = r + c · x_{P1}`.
4. The coordinator outputs `(R, s)` as the final signature.
5. Verification: `s · G = (r + c · x_{P1}) · G = R + c · X_{P1}`. Since `X_{P1} ≠ PK` (the group public key), verification fails.
6. All honest participants accepted the protocol run and received an unusable signature — the signing output is corrupted. [8](#0-7)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
