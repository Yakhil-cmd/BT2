### Title
Missing Zero-Address Validation in Base Token Minting via `ft_on_transfer` - (File: `engine/src/engine.rs`)

### Summary
The `receive_base_tokens` function in `engine/src/engine.rs` mints ETH (base token) balances to a recipient address parsed from the caller-controlled `msg` field of `FtOnTransferArgs`, without checking whether that address is the zero address (`0x0000...0000`). Any user who calls `ft_transfer_call` on the ETH connector NEP-141 contract with `msg = "0000000000000000000000000000000000000000"` will have their ETH permanently credited to the zero address in Aurora's storage, making it irrecoverable.

### Finding Description
The `receive_base_tokens` function is invoked from `ft_on_transfer` in `engine/src/contract_methods/connector.rs` when the predecessor is the ETH connector account. It parses the recipient address from `args.msg` via `FtTransferMessageData::try_from`, then directly calls `set_balance` to credit the amount to that address:

```rust
// engine/src/engine.rs lines 777-785
let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
let amount = Wei::new_u128(args.amount.as_u128());
let receipient = message_data.recipient;
let balance = get_balance(&self.io, &receipient);
let new_balance = balance
    .checked_add(amount)
    .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
set_balance(&mut self.io, &receipient, &new_balance);
``` [1](#0-0) 

The `FtTransferMessageData::try_from` parser in `engine-types/src/parameters/connector.rs` accepts any valid 40-character hex string as a recipient, including the all-zero string representing the zero address:

```rust
// engine-types/src/parameters/connector.rs lines 41-56
if message.len() == 40 {
    let mut address_bytes = [0; 20];
    hex::decode_to_slice(message, &mut address_bytes)
        .map_err(|_| errors::ParseOnTransferMessageError::InvalidHexData)?;
    Address::from_array(address_bytes)
    ...
}
``` [2](#0-1) 

There is no guard of the form `if recipient == Address::default() { return Err(...) }` anywhere in this path. By contrast, the ERC-20 minting path (`receive_erc20_tokens`) delegates to the `EvmErc20` Solidity contract, which inherits OpenZeppelin's `_mint` and therefore reverts on a zero-address recipient. The base-token path has no equivalent protection. [3](#0-2) 

### Impact Explanation
ETH (base token) balances credited to the zero address in Aurora's storage are permanently inaccessible. No private key controls `address(0)`, so the funds cannot be recovered. This constitutes **permanent freezing of funds** (Critical).

### Likelihood Explanation
The entry point is the standard NEP-141 `ft_transfer_call` interface, callable by any token holder without special permissions. A user who mistakenly (or programmatically) passes an all-zero hex string as the `msg` field will silently lose their ETH. The `ft_on_transfer` handler returns `"0"` (success, no refund) in this case, giving no indication of the loss. [4](#0-3) 

### Recommendation
Add a zero-address guard at the start of `receive_base_tokens` (and symmetrically in `receive_erc20_tokens`) before any balance mutation:

```rust
if receipient == Address::default() {
    return Err(errors::ERR_ZERO_ADDRESS_RECIPIENT.into());
}
```

This causes `ft_on_transfer` to return the full `args.amount` to the sender (the refund path at line 96), preventing permanent loss. [5](#0-4) 

### Proof of Concept
1. User holds ETH connector NEP-141 tokens (i.e., has ETH on Aurora).
2. User calls `ft_transfer_call` on the ETH connector contract with:
   - `receiver_id`: Aurora engine account ID
   - `amount`: e.g., `1000000000000000000` (1 ETH)
   - `msg`: `"0000000000000000000000000000000000000000"`
3. The ETH connector calls `ft_on_transfer` on Aurora with `predecessor = eth_connector_account_id`.
4. Aurora dispatches to `engine.receive_base_tokens(&args)`.
5. `FtTransferMessageData::try_from("0000000000000000000000000000000000000000")` succeeds, returning `recipient = Address([0u8; 20])`.
6. `set_balance` writes the ETH balance to the zero address in Aurora's storage.
7. `ft_on_transfer` returns `"0"` (no refund).
8. The user's ETH is permanently lost; the zero address balance is unspendable. [6](#0-5) [7](#0-6)
