### Title
Unverified `pubdata_price` / `native_price` in `BlockMetadataFromOracle` Enables Forward/Proving Divergence - (`zk_ee/src/system/metadata/zk_metadata.rs`)

---

### Summary

`BlockMetadataFromOracle` carries two ZKsync-specific pricing fields — `pubdata_price` and `native_price` — that are consumed by every L2 transaction's resource-accounting logic but are **never committed to the block header** and therefore **never constrained by the ZK public inputs**. A prover supplying the oracle data can silently substitute different values from those used during forward execution, producing a valid proof for a state transition that differs from what the sequencer actually executed.

---

### Finding Description

`BlockMetadataFromOracle` is the block-level metadata struct used in the ZK execution path:

```rust
pub struct BlockMetadataFromOracle {
    pub eip1559_basefee: U256,
    pub pubdata_price: U256,   // ← not in block header
    pub native_price: U256,    // ← not in block header
    pub block_number: u64,
    pub timestamp: u64,
    ...
}
``` [1](#0-0) 

The struct implements `ZkSpecificPricingMetadata`, exposing `get_pubdata_price()` and `native_price()` to the entire transaction-processing pipeline: [2](#0-1) 

These two values drive the critical `native_per_pubdata` and `native_per_gas` ratios used in every L2 transaction's resource budget: [3](#0-2) 

The documentation explicitly acknowledges that neither field is committed to the block header: [4](#0-3) 

The Ethereum block header (`PectraForkHeader`) — which is the only structure whose hash enters the public inputs — contains no `pubdata_price` or `native_price` field: [5](#0-4) 

In the proving path the entire `BlockMetadataFromOracle` is supplied by the prover as a CSR-based oracle response (`BLOCK_METADATA_QUERY_ID`). Because the block hash that anchors the public inputs does not cover `pubdata_price` or `native_price`, the prover is free to substitute any values it chooses without invalidating the proof.

---

### Impact Explanation

**Vulnerability class:** Forward/proving divergence — oracle IO mismatch on pricing parameters.

The `native_per_gas` ratio determines how many native (RISC-V cycle) resources a user's gas budget buys:

```
nativePerGas  = ceil(gasPrice / nativePrice)
nativeLimit   = gasLimit * nativePerGas
``` [6](#0-5) 

The `native_per_pubdata` ratio determines how many native resources each byte of pubdata costs:

```
nativePerPubdata = pubdataPrice / nativePrice
``` [7](#0-6) 

A prover that substitutes different values can:

1. **Prove transactions succeeded that should have reverted.** By supplying a very small `native_price` (inflating `nativePerGas`), the prover gives every transaction an artificially large native budget, making out-of-native reverts disappear from the proven state.

2. **Prove transactions reverted that should have succeeded.** By supplying a very large `pubdata_price`, the prover inflates `native_per_pubdata`, causing any transaction that writes storage to exhaust its native budget and revert — even though the forward execution succeeded.

3. **Diverge fee accounting.** Because `gas_used` is partially derived from native consumption (`deltaGas` adjustment), altering `native_price` changes the reported `gas_used` and therefore the ETH fees charged to users and paid to the operator.

The net result is that the on-chain proof attests to a state root that was never produced by honest forward execution, breaking the core ZK-rollup guarantee.

---

### Likelihood Explanation

The entry path is the prover supplying oracle data (`BLOCK_METADATA_QUERY_ID`) — explicitly listed as an in-scope attacker role ("prover/forward execution input"). No privileged key beyond the prover role is required. The gap is already acknowledged in the codebase documentation, confirming it is a real, reachable condition rather than a theoretical concern.

---

### Recommendation

Commit `pubdata_price` and `native_price` to the block header (e.g., in the `extra_data` field, as the documentation already contemplates) so that their values are covered by the block hash that enters the ZK public inputs. Until that is done, the proving-mode verifier should assert that the `pubdata_price` and `native_price` values supplied by the oracle match the values recorded in the committed block header, preventing any divergence between forward and proving execution.

---

### Proof of Concept

**Setup (forward execution):**
1. Operator sequences a block with `native_price = 100`, `pubdata_price = 1000`.
2. Transaction T writes 10 storage slots; `native_per_pubdata = 10`. T consumes exactly its native budget and succeeds.
3. Forward execution records T as successful; state root `S_forward` is produced.

**Attack (proving execution):**
4. Prover supplies `BlockMetadataFromOracle` with `native_price = 1`, `pubdata_price = 1_000_000` via the CSR oracle.
5. Now `native_per_pubdata = 1_000_000`. T's pubdata charge exhausts its native budget; T reverts.
6. Proving execution produces state root `S_proof ≠ S_forward`.
7. The ZK proof is valid because the block hash (public input) does not cover `pubdata_price` or `native_price` — the prover's substitution is unconstrained.
8. The settlement layer accepts the proof and finalises `S_proof`, which differs from the state the sequencer actually committed to users.

The divergence is reachable with no additional privileges beyond the prover role, and the root cause (`pubdata_price`/`native_price` absent from the block header) is confirmed by the codebase documentation at `docs/bootloader/bootloader.md:36`. [8](#0-7) [4](#0-3) [3](#0-2) [9](#0-8)
