### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Suppress Per-Block Rewards - (`File: src/staking/staking.cairo`)

### Summary

`update_rewards` in the staking contract is intended to be callable only by the Starknet sequencer (per spec), but the implementation contains **no caller check**. Because `last_reward_block` is a global variable and the function accepts a `disable_rewards: bool` parameter, any unprivileged caller can invoke `update_rewards(staker_address: any_valid_staker, disable_rewards: true)` in every block to consume the global "already updated" slot without distributing any rewards. The sequencer's legitimate call in the same block then reverts with `REWARDS_ALREADY_UPDATED`, and all stakers permanently lose their block rewards for that block.

---

### Finding Description

The spec for `update_rewards` explicitly states:

> **Access control:** Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1449–1507 contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // GLOBAL write

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits with no rewards
    }
    ...
``` [1](#0-0) 

Two properties combine to create the vulnerability:

1. **`last_reward_block` is a global (not per-staker) storage variable.** A single call to `update_rewards` for *any* valid staker address locks out all other stakers for that block.

2. **`disable_rewards: true` is a caller-controlled parameter.** When set, the function writes `last_reward_block` but immediately returns without computing or crediting any rewards.

An attacker who calls `update_rewards(staker_address: <any_active_staker>, disable_rewards: true)` before the sequencer in block N:
- Sets `last_reward_block = N` globally.
- Distributes zero rewards.
- Causes the sequencer's subsequent call for the intended staker to revert with `REWARDS_ALREADY_UPDATED`. [2](#0-1) 

The `calculate_block_rewards` path (called only when `disable_rewards == false`) invokes `reward_supplier_dispatcher.update_current_epoch_block_rewards()`, which updates `avg_block_duration` and returns the per-block reward amount. When the attacker bypasses this path, the rewards for that block are never computed or added to any staker's `unclaimed_rewards_own`. [3](#0-2) 

---

### Impact Explanation

Every block where the attack is active, **all stakers lose their entire block reward**. The rewards are not deferred or rolled over — they are simply never generated. A staker's `unclaimed_rewards_own` is never incremented for that block, so the loss is permanent. Sustained over many blocks, this constitutes **theft of unclaimed yield** for every active staker and their delegators.

This matches the allowed impact: **High — Theft of unclaimed yield**.

---

### Likelihood Explanation

- The attacker needs only a valid `staker_address` with non-zero balance at the current epoch. They can use any existing staker's address — they do not need to be a staker themselves.
- The call is cheap (no token transfer, no complex state change).
- The attack must be repeated each block, but since Starknet blocks are produced continuously, a motivated attacker can sustain it indefinitely.
- There is no economic cost to the attacker beyond gas fees.

The entry path is: **unprivileged public caller → `update_rewards(any_valid_staker, disable_rewards: true)`**.

---

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, analogous to how `update_rewards_from_attestation_contract` checks `assert_caller_is_attestation_contract()`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   assert!(
+       get_caller_address() == starknet::get_execution_info().block_info.sequencer_address,
+       "{}",
+       Error::CALLER_IS_NOT_SEQUENCER,
+   );
    ...
```

This aligns the implementation with the specification and closes the griefing vector. [4](#0-3) 

---

### Proof of Concept

```
1. Deploy the system in consensus-rewards mode (post `set_consensus_rewards_first_epoch`).
2. Advance to an epoch where staker S has non-zero STRK balance.
3. In block N, attacker A (any EOA) calls:
       staking.update_rewards(staker_address: S, disable_rewards: true)
   → last_reward_block is set to N; no rewards distributed.
4. Sequencer calls:
       staking.update_rewards(staker_address: S, disable_rewards: false)
   → Reverts with REWARDS_ALREADY_UPDATED.
5. Assert staker S's unclaimed_rewards_own is unchanged (zero new rewards for block N).
6. Repeat steps 3–5 for every block to permanently suppress all block rewards.
```

The `REWARDS_ALREADY_UPDATED` error is confirmed in the error definitions and tested in the existing test suite, confirming the global nature of `last_reward_block`. [5](#0-4)
