### Title
Missing Caller Validation in `update_rewards` Allows Theft or Permanent Freezing of Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract has no caller restriction and accepts an arbitrary `staker_address` and a caller-controlled `disable_rewards` flag. This is the direct analog to the PDA validation flaw: just as the Solana report's instruction accepted an unvalidated account allowing one authority to corrupt another's PDA, here any unprivileged actor can invoke `update_rewards` with any valid staker address and any `disable_rewards` value, either stealing block rewards or permanently freezing them.

---

### Finding Description

`update_rewards` is defined in `IStakingRewardsManager` and implemented in `src/staking/staking.cairo`: [1](#0-0) 

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
```

The function then validates the staker and unconditionally writes the current block as the last rewarded block: [2](#0-1) 

```cairo
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
```

`last_reward_block` is a **global, single-slot** variable: [3](#0-2) 

There is **no check** that `get_caller_address()` is the block producer, a sequencer, or any privileged role. The `staker_address` parameter is accepted verbatim and is never derived from or tied to the caller's identity. The `disable_rewards` flag is fully caller-controlled.

This mirrors the reported Solana flaw exactly: the account (here, `staker_address`) is passed in rather than derived/validated from the caller's context, and the instruction (here, `update_rewards`) modifies shared global state (`last_reward_block`) on behalf of that arbitrary address.

**Attack vector A — Theft of block rewards:**
An attacker who is a valid staker calls `update_rewards(staker_address: attacker_staker, disable_rewards: false)` at the start of every block. Because `last_reward_block` is now set to the current block, no other staker can call `update_rewards` for that block. The attacker's staker accumulates rewards for every block, far exceeding the share proportional to their stake. Legitimate block producers receive nothing for those blocks.

**Attack vector B — Permanent freezing of unclaimed yield:**
An attacker calls `update_rewards(staker_address: any_valid_staker, disable_rewards: true)` at the start of every block. `last_reward_block` is updated but the early-return path skips all reward distribution. Block rewards for that block are permanently lost — they are never added to any staker's `unclaimed_rewards_own`.

---

### Impact Explanation

- **Attack A**: Direct theft of unclaimed yield. The attacker's staker receives block rewards for every block in the consensus phase, stealing yield that belongs to the legitimate block producers.
- **Attack B**: Permanent freezing of unclaimed yield. Block rewards are irreversibly discarded because `last_reward_block` is consumed without distributing anything.

Both map to allowed impacts: *Theft of unclaimed yield* and *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation

**High.** The function is publicly callable with no role check. Any address that holds a valid, active staker position (a low barrier — just call `stake()` with the minimum amount) can execute either attack variant on every block. The attack is cheap, repeatable, and requires no special access.

---

### Recommendation

1. Restrict `update_rewards` to an authorized caller (e.g., the block producer identified on-chain, or a dedicated sequencer role). Add `self.roles.only_block_producer()` or equivalent at the top of the function.
2. Derive `staker_address` from the caller's registered identity rather than accepting it as a free parameter, analogous to the recommended fix in the external report (derive the PDA from seeds inside the context).
3. Remove or gate the `disable_rewards` parameter so it cannot be set to `true` by an unprivileged caller.

---

### Proof of Concept

```
1. Attacker calls `stake(reward_address, operational_address, min_stake)` to become a valid staker.
2. At the start of each new block (block_number > last_reward_block):
   a. Attacker calls `update_rewards(staker_address: attacker_staker, disable_rewards: false)`.
   b. `last_reward_block` is set to `current_block_number`.
   c. Block rewards are credited to `attacker_staker.unclaimed_rewards_own`.
3. Legitimate block producers call `update_rewards` → revert with REWARDS_ALREADY_UPDATED.
4. Attacker calls `claim_rewards(attacker_staker)` to withdraw the accumulated stolen yield.
``` [4](#0-3) [5](#0-4) [6](#0-5)
