### Title
Receipt Size Invariant Bypass via `promise_return` + Post-Validation `output_data_receivers` Injection — (`runtime/runtime/src/lib.rs`)

---

### Summary

The nearcore runtime validates new receipts for size compliance (`max_receipt_size = 4,194,304 bytes`) before committing them. However, the runtime then **mutates the already-validated receipt** by appending `output_data_receivers` from the parent receipt's action receipt — without re-validating the final size. This is structurally identical to the tBTC bug: one code path permits a state transition (receipt creation + validation), while a second code path (post-validation mutation) silently pushes the receipt past the invariant boundary, creating oversized receipts that are committed to state and forwarded across shards.

---

### Finding Description

**Phase 1 — Creation and validation (permitted):**

When a contract calls `promise_create` with args sized to exactly `max_receipt_size` minus the base receipt overhead, the resulting receipt passes `validate_receipt` with `ValidateReceiptMode::NewReceipt`: [1](#0-0) 

The check fires only in `NewReceipt` mode and measures the receipt at the moment of validation — before any post-execution mutation.

**Phase 2 — Post-validation mutation (the gap):**

After the function call executes and returns `ReturnData::ReceiptIndex` (i.e., `promise_return`), the runtime in `apply_action_receipt` appends the **parent receipt's `output_data_receivers`** directly onto the already-validated child receipt: [2](#0-1) 

No size re-check occurs after this mutation. Each `DataReceiver` entry (a `CryptoHash` + `AccountId`) adds tens of bytes. If the parent receipt had any `output_data_receivers` (i.e., it was itself part of a `.then()` chain), the child receipt now exceeds `max_receipt_size` and is committed to state and forwarded.

**The inconsistency:**

`validate_receipt(NewReceipt)` enforces `receipt_size ≤ max_receipt_size` at creation time, but the runtime's `output_data_receivers` injection path has no corresponding guard. The two code paths are inconsistent: one enforces the invariant, the other silently violates it.

**Acknowledged workaround (not a fix):**

The congestion control layer contains an explicit band-aid that clamps the reported size to `max_receipt_size` to prevent oversized receipts from getting permanently stuck in the outgoing buffer: [3](#0-2) 

The `ValidateReceiptMode::ExistingReceipt` mode was also added specifically to skip the size check for receipts already in state: [4](#0-3) 

Both workarounds are acknowledged as temporary mitigations for issue #12606. The root cause — the missing post-mutation size check — remains unfixed in production.

---

### Impact Explanation

**Broken invariant:** `receipt_size ≤ max_receipt_size` is violated for receipts produced via `promise_return` in a `.then()` promise chain. The exact corrupted value is the serialized receipt size, which exceeds `max_receipt_size` by `N × sizeof(DataReceiver)` where N is the number of `output_data_receivers` on the parent receipt.

**Congestion control accounting corruption:** The congestion control system uses the clamped (incorrect) size for bandwidth scheduling. An attacker can inject receipts whose true byte footprint exceeds what the scheduler accounts for, causing the outgoing bandwidth budget to be systematically underestimated. This breaks the receipt causality invariant that the congestion control system is designed to enforce.

**Scope:** Receipt causality — the receipt's final committed form is not the form that was validated, breaking the causal chain between validation and commitment.

---

### Likelihood Explanation

Any unprivileged user who can deploy a contract can trigger this. The attack requires:
1. Deploy a contract with a method that calls `promise_create` with args sized to `max_receipt_size - base_overhead`.
2. Call that method from a `.then()` chain so the parent receipt has non-empty `output_data_receivers`.
3. The contract calls `promise_return` on the large receipt.

This is confirmed reachable by the production test: [5](#0-4) 

The test explicitly documents the bug and confirms the oversized receipt is committed to state and forwarded.

---

### Recommendation

After the `output_data_receivers` injection at `runtime/runtime/src/lib.rs` lines 1029–1035, re-validate the mutated receipt's serialized size against `max_receipt_size`. If the size is exceeded, the action should fail with `NewReceiptValidationError::ReceiptSizeExceeded` rather than silently committing an oversized receipt. This mirrors the fix recommended for the tBTC bug: the finalization step's bound must be consistent with what the creation step permits.

---

### Proof of Concept

**Promise DAG:**
```
User tx → A[max_receipt_size_promise_return_method1(args_size)]
  A executes:
    promise_then(A_self, B[mark_test_completed])   // parent has output_data_receivers = [B]
    promise_return(C[noop(large_args)])             // C size == max_receipt_size
  Runtime injects B into C.output_data_receivers   // C size > max_receipt_size, no re-check
  C is committed to state and forwarded
```

**Exact corrupted value:**
- `base_receipt_size` = serialized size of a minimal action receipt (computed at test setup)
- `args_size = max_receipt_size - base_receipt_size` → receipt C passes validation
- After injection: `actual_size = max_receipt_size + sizeof(DataReceiver)` → invariant violated [6](#0-5) 

The `assert_oversized_receipt_occurred` call confirms the oversized receipt is present in the committed chain state, proving the invariant is broken in production.

// `max_receipt_size`, it'll pass validation, but then `output_data_receivers` will be modified and

// isn't because of a bug (See https://github.com/near/nearcore/issues/12606)
// Runtime shouldn't die when it encounters a receipt with size above `max_receipt_size`.
#[test]
fn test_max_receipt_size_promise_return() {
    init_test_logger();

    let account = create_account_id("account0");
    let account_signer = create_user_test_signer(&account);
    let mut env = TestLoopBuilder::new()
        .enable_rpc()
        .add_user_account(&account, Balance::from_near(10_000))
        .build();

    // Deploy the test contract
    let deploy_contract_tx = SignedTransaction::deploy_contract(
        101,
        &account,
        near_test_contracts::rs_contract().into(),
        &account_signer,
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(deploy_contract_tx, Duration::seconds(5));

    // User calls a contract method
    // Contract method creates a DAG with two promises: [A -then-> B]
    // When promise A is executed, it creates a third promise - `C` and does a `promise_return`.
    // The DAG changes to: [C ->then-> B]
    // The receipt for promise C is a maximum size receipt.
    // Adding the `output_data_receivers` to C's receipt makes it go over the size limit.
    let base_receipt_template = Receipt::V0(ReceiptV0 {
        predecessor_id: account.clone(),
        receiver_id: account.clone(),
        receipt_id: CryptoHash::default(),
        receipt: ReceiptEnum::Action(ActionReceipt {
            signer_id: account.clone(),
            signer_public_key: account_signer.public_key().into(),
            gas_price: Balance::ZERO,
            output_data_receivers: vec![],
            input_data_ids: vec![],
            actions: vec![Action::FunctionCall(Box::new(FunctionCallAction {
                method_name: "noop".into(),
                args: vec![],
                gas: Gas::ZERO,
                deposit: Balance::ZERO,
            }))],
        }),
    });
    let base_receipt_template = action_receipt_v1_to_latest(&base_receipt_template);
    let base_receipt_size = borsh::object_length(&base_receipt_template).unwrap();
    let max_receipt_size = 4_194_304;
    let args_size = max_receipt_size - base_receipt_size;

    // Call the contract
    let large_receipt_tx = SignedTransaction::call(
        102,
        account.clone(),
        account.clone(),
        &account_signer,
        Balance::ZERO,
        "max_receipt_size_promise_return_method1".into(),
        format!("{{\"args_size\": {}}}", args_size).into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(large_receipt_tx, Duration::seconds(5));

    // Make sure that the last promise in the DAG was called

        103,
        account.clone(),
        account,
        &account_signer,
        Balance::ZERO,
        "assert_test_completed".into(),
        "".into(),
        Gas::from_teragas(300),
        env.rpc_node().head().last_block_hash,
    );
    env.rpc_runner().run_tx(assert_test_completed, Duration::seconds(5));

    assert_oversized_receipt_occurred(&env.validator());
}
