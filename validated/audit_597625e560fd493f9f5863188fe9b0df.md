Audit Report

## Title
Silent Burn Failure in `burn_and_log` Allows Cycles Minting Without ICP Burn - (`rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function unconditionally returns `()` and only logs ledger transfer errors. Both `process_top_up` and `process_mint_cycles` call it after a successful cycle deposit and immediately return `Ok`, causing the block to be permanently marked as processed. If the ICP ledger is unavailable when the burn is attempted, cycles are minted and delivered while the corresponding ICP is never burned, violating the ICP/cycles conservation invariant.

## Finding Description
`burn_and_log` (lines 2017–2049) makes a `send_pb` call to the ICP ledger and on failure only prints a log message, returning `()` in all cases. The function comment explicitly states: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."*

In `process_top_up` (lines 1999–2003), after `deposit_cycles` succeeds:
```rust
Ok(()) => {
    burn_and_log(sub, amount).await;  // error silently discarded
    Ok(cycles)                         // always returned
}
```
In `process_mint_cycles` (lines 1967–1973), after `do_mint_cycles` succeeds:
```rust
Ok(deposit_result) => {
    burn_and_log(sub, amount).await;  // error silently discarded
    Ok(NotifyMintCyclesSuccess { ... }) // always returned
}
```
Both callers return `Ok(...)` regardless of burn outcome. Back in `notify_top_up` (lines 1214–1222) and `notify_mint_cycles` (lines 1305–1313), the result is written to `blocks_notified` as `NotifiedTopUp(Ok(cycles))` or `NotifiedMint(Ok(...))`. The `is_transient_error` check (lines 1125–1130) only removes the entry if the result is a retriable `Err`; since the result is `Ok`, the block is permanently finalized. Any subsequent call for the same block index hits the `Entry::Occupied` branch and returns the cached `Ok` result, making the burn permanently unretriable.

## Impact Explanation
This is illegal minting of cycles: cycles are created and deposited to the target canister while the backing ICP remains unburned in the CMC's subaccount. The ICP/cycles conservation invariant — every cycle minted corresponds to burned ICP — is violated. The unburned ICP accumulates in the CMC subaccount and cannot be recovered through the normal notification path. This matches the allowed impact class of **illegal minting of ICP/Cycles assets** (High, $2,000–$10,000), as each incident is bounded to a single transaction's worth of cycles and the ICP is not stolen but permanently stranded rather than burned.

## Likelihood Explanation
The ICP ledger undergoes routine upgrades via NNS governance proposals, during which it briefly stops accepting calls. Any `notify_top_up` or `notify_mint_cycles` call that reaches the `burn_and_log` step during this window triggers the condition. An unprivileged user can observe pending NNS upgrade proposals for the ledger canister and time their `notify_top_up` call to coincide with the upgrade execution. No special privileges are required. The window is short but the condition is deterministic and repeatable across every ledger upgrade cycle.

## Recommendation
`burn_and_log` should return a `Result` and propagate failures to callers. The block status should distinguish between "cycles deposited, burn pending" and "fully finalized." Options:
1. Introduce a `NotificationStatus::BurnPending { cycles, subaccount, amount }` state and retry the burn on a heartbeat/timer tick until it succeeds before transitioning to `NotifiedTopUp(Ok(...))`.
2. Do not mark the block as fully processed until the burn succeeds; return a retriable error to the caller so the notification can be re-submitted to complete the burn (while ensuring idempotency on the deposit side to prevent double-minting).

## Proof of Concept
1. User transfers ICP to `AccountIdentifier::new(CMC_ID, Some(Subaccount::from(&canister_id)))` with `MEMO_TOP_UP_CANISTER`.
2. User monitors NNS proposals; observes a pending ledger upgrade proposal about to execute.
3. User calls `notify_top_up { block_index, canister_id }` timed to be processed during the ledger upgrade window.
4. CMC: `fetch_transaction` succeeds; block set to `Processing`.
5. CMC: `deposit_cycles(canister_id, cycles, ...)` succeeds — cycles delivered.
6. CMC: `burn_and_log(sub, amount).await` — ledger rejects call (canister stopped/upgrading); error logged, `()` returned.
7. CMC: `process_top_up` returns `Ok(cycles)`.
8. CMC: `blocks_notified[block_index]` set to `NotifiedTopUp(Ok(cycles))` permanently (`is_transient_error` is false for `Ok`).
9. CMC returns `Ok(cycles)` to user.

**Result**: Target canister has cycles. ICP is unburned in CMC subaccount. Block is permanently finalized. Conservation invariant violated. Reproducible as a PocketIC integration test by pausing the mock ledger canister between `deposit_cycles` and `burn_and_log` and asserting the CMC subaccount balance remains non-zero after notification completes.
