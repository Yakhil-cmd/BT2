### Title
`ComputationRemaining` Unsigned Integer Underflow Returns Inflated Gas Limit, Bypassing EVM Batch Computation Cap — (`fvm/meter/computation_meter.go`)

---

### Summary

`ComputationRemaining` in the FVM computation meter performs an unsigned subtraction `computationLimit - computationUsed` without a valid underflow guard. The guard `remainingComputationUsage <= 0` is dead code for a `uint64` value. When `computationUsed` exceeds `computationLimit`, the subtraction silently wraps around to a near-`MaxUint64` value. This inflated result is consumed directly by `batchRun` in the EVM contract handler to gate EVM gas limits, allowing EVM batch transactions to bypass the FVM computation cap.

---

### Finding Description

In `fvm/meter/computation_meter.go`, `ComputationRemaining` computes how much EVM gas is still available within the FVM computation budget:

```go
remainingComputationUsage := m.params.computationLimit - m.computationUsed
if remainingComputationUsage <= 0 {   // ← dead code: uint64 is never < 0
    return 0
}
return remainingComputationUsage / w
``` [1](#0-0) 

Both `computationLimit` and `computationUsed` are `uint64`. The condition `remainingComputationUsage <= 0` is **always false** for an unsigned type; it can only equal zero, never go below it. When `computationUsed > computationLimit` (which occurs after `MeterComputation` increments `computationUsed` past the limit before returning an error, or after an unchecked `Merge`), the subtraction wraps around to approximately `MaxUint64`, and the guard never fires.

`MeterComputation` increments `computationUsed` **before** checking the limit:

```go
m.computationUsed += w * intensity
if m.computationUsed > m.params.computationLimit {
    return errors.NewComputationLimitExceededError(...)
}
``` [2](#0-1) 

After this error path, `computationUsed` is already above `computationLimit`. Similarly, `Merge` adds child usage with no limit check:

```go
m.computationUsed = m.computationUsed + child.computationUsed
``` [3](#0-2) 

The result of `ComputationRemaining` is used directly in `batchRun` to gate EVM transaction gas:

```go
remainingGasLimit := h.backend.ComputationRemaining(environment.ComputationKindEVMGasUsage)
...
if remainingGasLimit < txGasLimit {
    return nil, types.ErrInsufficientComputation
}
remainingGasLimit -= txGasLimit
``` [4](#0-3) 

When `ComputationRemaining` returns a near-`MaxUint64` value due to underflow, the check `remainingGasLimit < txGasLimit` passes for any realistic `txGasLimit`, and the subsequent `remainingGasLimit -= txGasLimit` still leaves a huge number, so all transactions in the batch pass the gate regardless of their gas limits.

---

### Impact Explanation

An EVM batch transaction that should be rejected for exceeding the FVM computation budget is instead allowed to execute. This enables:
- Running EVM computations beyond the FVM gas cap without paying the corresponding computation fees.
- Potential state changes in EVM that should have been blocked by the computation limit.

The analog to the external report is direct: an unsigned integer that cannot represent a negative intermediate value causes incorrect accounting — in the external report it causes unexpected reverts; here it causes an unexpected bypass of the computation cap.

---

### Likelihood Explanation

Exploitability requires `computationUsed > computationLimit` to be true at the point `batchRun` calls `ComputationRemaining`. This state is reached after `MeterComputation` returns a limit-exceeded error. In normal Cadence execution, such an error aborts the transaction immediately, preventing a subsequent `batchRun` call. However, edge cases exist:

- A Cadence transaction that meters computation in one code path and then calls `EVM.batchRun` in a separate path where the runtime does not immediately propagate the error.
- An unchecked `Merge` of a child meter whose `computationUsed` pushes the parent past the limit without triggering an error return.

The dead-code guard is unconditionally wrong regardless of exploitability and will silently produce incorrect results in any future code path that reaches this state.

---

### Recommendation

Replace the dead-code guard with a proper underflow check before the subtraction:

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }
    if m.computationUsed >= m.params.computationLimit {
        return 0
    }
    return (m.params.computationLimit - m.computationUsed) / w
}
```

Additionally, `Merge` should check whether the merged total exceeds the limit and return an error or clamp the value, consistent with how `MeterComputation` handles overages.

---

### Proof of Concept

1. Construct a Cadence transaction that calls `EVM.batchRun` with a set of EVM transactions whose combined gas exceeds the FVM computation limit for `ComputationKindEVMGasUsage`.
2. Arrange for `computationUsed` to exceed `computationLimit` before `batchRun` is reached (e.g., via a prior metered operation that pushes past the limit).
3. Observe that `ComputationRemaining` returns a value near `MaxUint64` instead of `0`.
4. The gas limit check `remainingGasLimit < txGasLimit` passes for all transactions in the batch.
5. All EVM transactions execute despite the FVM computation budget being exhausted.
