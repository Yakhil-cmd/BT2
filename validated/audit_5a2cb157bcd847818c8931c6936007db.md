### Title
`chia_getWalletAddresses` Bypasses Wallet-Selection Boundary, Exposing Non-Paired Wallet Key Addresses via WalletConnect - (File: `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

The `chia_getWalletAddresses` WalletConnect command (`daemon.get_wallet_addresses`) accepts a `fingerprints` (plural) array parameter that is forwarded directly to the daemon without being validated against the pair's authorized fingerprint. Both fingerprint-enforcement layers in the codebase only inspect `params.fingerprint` (singular), leaving `params.fingerprints` (plural) unchecked. A dapp paired with wallet key A can silently enumerate addresses for wallet keys B, C, D — or all keys — without any additional user confirmation, bypassing the explicit wallet-selection boundary the user established during WalletConnect pairing.

---

### Finding Description

The `daemon.get_wallet_addresses` command is registered in `Commands.ts` with `allowConfirmationBypass: true` and exposes the dapp-facing command `chia_getWalletAddresses`. Its parameter schema declares `fingerprints` (plural, optional JSON array): [1](#0-0) 

The underlying daemon API accepts this array and, when omitted, returns addresses for **all** stored keys: [2](#0-1) 

Both fingerprint-enforcement layers destructure only `params.fingerprint` (singular):

**Renderer layer** (`useWalletConnectCommand`): [3](#0-2) 

**Main-process layer** (`dispatchPairRequest`): [4](#0-3) 

Because `chia_getWalletAddresses` uses `fingerprints` (plural), `params.fingerprint` is always `undefined` for this command. Both guards fall back to `loggedInFingerprint`, which matches `pair.fingerprint` (the paired wallet), so the check passes. The full `params` object — including the attacker-controlled `fingerprints` array — is then forwarded verbatim to the daemon via `sendCommand`: [5](#0-4) 

The daemon returns addresses for every fingerprint in the array, or for all stored keys if the array is omitted.

---

### Impact Explanation

A dapp that has been granted `chia_getWalletAddresses` (with or without bypass) can:

1. Call `chia_getWalletAddresses` with `fingerprints: [fp2, fp3, fp4]` (fingerprints of other wallets the user holds).
2. Both fingerprint guards pass because they only check `params.fingerprint` (singular, which is `undefined`).
3. The daemon returns receive addresses for all requested wallet keys — keys the user never authorized this dapp to access.
4. If bypass was granted for this command (which the Pair dialog presents as an "innocuous" action), this happens **silently with zero user confirmation**.

This is a direct bypass of the wallet-selection security boundary: the user explicitly chose which wallet key to pair with the dapp, but the dapp can access addresses of all other keys in the keyring. Omitting `fingerprints` entirely causes the daemon to return addresses for every stored key. [6](#0-5) 

---

### Likelihood Explanation

- Any dapp that successfully completes WalletConnect pairing and is granted `chia_getWalletAddresses` (a command the Pair dialog classifies as "innocuous") can exploit this.
- The bypass path (no confirmation dialog at all) is reachable whenever the user checks the "Innocuous actions" capability group during pairing, which is the default-checked option.
- The attacker needs no leaked keys, no host compromise, and no cryptographic break — only a completed WalletConnect pairing session.

---

### Recommendation

In the `DISPATCH_AS_PAIR` handler (or in a command-specific handler for `chia_getWalletAddresses`), validate and restrict the `fingerprints` array to contain only the pair's authorized fingerprint before forwarding to the daemon. A dedicated `handler` function (like those used for `chia_requestPermissions`) should filter `params.fingerprints` to `[pair.fingerprint]`, rejecting any request that includes other fingerprints. If no `fingerprints` are specified, the handler should inject `[pair.fingerprint]` so the daemon never returns addresses for non-paired keys.

---

### Proof of Concept

1. Attacker operates a dapp and initiates a WalletConnect pairing with the victim's Chia GUI.
2. Victim approves the pairing for wallet key `111111`, granting `chia_getWalletAddresses` with bypass (presented as "innocuous").
3. Attacker's dapp sends a WalletConnect `session_request`:
   ```json
   {
     "method": "chia_getWalletAddresses",
     "params": {
       "fingerprints": [222222, 333333, 444444]
     }
   }
   ```
4. `useWalletConnectCommand` checks `params.fingerprint` (undefined) → falls back to `currentFingerprint` (111111) → matches `pair.fingerprint` (111111) → passes.
5. `dispatchPairRequest` performs the same check → passes. Command is in `pair.bypass` → no confirmation dialog shown.
6. `sendCommand('get_wallet_addresses', 'daemon', { fingerprints: [222222, 333333, 444444] })` executes.
7. Daemon returns receive addresses for wallet keys 222222, 333333, and 444444 — wallets the victim never authorized this dapp to access. [7](#0-6) [8](#0-7)
