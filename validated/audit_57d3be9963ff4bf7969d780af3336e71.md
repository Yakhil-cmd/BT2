### Title
`setFeeCaps` Does Not Clamp Existing Pools' Admin Fee Configs, Allowing Admin Fees to Exceed Newly Reduced Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

When the factory owner calls `setFeeCaps` to reduce `maxAdminSpreadFeeE6` or `maxAdminNotionalFeeE8`, the per-pool `poolFeeConfig[pool].adminSpreadFeeE6` / `adminNotionalFeeE8` values stored in the factory are **not clamped** to the new limits. All existing pools therefore continue to have their admin fee shares collected at the old (higher) rate, bypassing the cap reduction entirely. The protocol receives less than its intended share of fees while the pool admin silently retains a share above the new cap.

---

### Finding Description

`setFeeCaps` updates the four global cap variables and, as a partial guard, clamps the **protocol default** `spreadProtocolFeeE6` if it now exceeds the new protocol cap: [1](#0-0) 

However, the per-pool admin fee values stored in `poolFeeConfig` are never touched: [2](#0-1) 

Every subsequent call to `collectPoolFees` (or `setPoolProtocolFee` / `setPoolAdminFees`) reads those stale values directly: [3](#0-2) 

Inside `MetricOmmPool.collectFees`, the admin's share of the spread surplus is computed as: [4](#0-3) 

Because `adminSpreadFeeE6_` is sourced from the unclamped `poolFeeConfig[pool].adminSpreadFeeE6`, the admin continues to receive a proportion of the surplus that exceeds the new cap.

The only place that **does** clamp the admin fee is inside `setPoolProtocolFee`, but only when the factory owner explicitly calls it per-pool: [5](#0-4) 

This per-pool call is not automatic; `setFeeCaps` never triggers it.

---

### Impact Explanation

After `setFeeCaps` lowers `maxAdminSpreadFeeE6` (e.g., from 200 000 to 10 000), every existing pool whose `poolFeeConfig.adminSpreadFeeE6` was set at creation to, say, 100 000 continues to route 100 000 / (100 000 + protocolSpreadFeeE6) of the accumulated spread surplus to the pool admin. The protocol's share is correspondingly reduced. The pool admin — a semi-trusted actor who is supposed to be bounded by the caps — silently exceeds the new cap without any active exploit step. The factory owner's protective action is ineffective until they manually call `setPoolProtocolFee` for every existing pool.

---

### Likelihood Explanation

Any time the factory owner legitimately reduces admin fee caps (a normal governance action), all pre-existing pools are unaffected. With many pools deployed, it is operationally likely that at least some pools are never individually updated, making the bypass persistent.

---

### Recommendation

Inside `setFeeCaps`, after updating the cap variables, iterate over all registered pools and clamp each pool's `poolFeeConfig[pool].adminSpreadFeeE6` / `adminNotionalFeeE8` to the new caps, mirroring the clamping logic already present in `setPoolProtocolFee`:

```solidity
// pseudocode addition to setFeeCaps
for (uint256 i = 1; i < nextPoolIdx; i++) {
    address pool = idxToPool[i];
    PoolFeeConfig storage c = poolFeeConfig[pool];
    bool changed;
    if (c.adminSpreadFeeE6 > newMaxAdminSpreadFeeE6) {
        c.adminSpreadFeeE6 = newMaxAdminSpreadFeeE6;
        changed = true;
    }
    if (c.adminNotionalFeeE8 > newMaxAdminNotionalFeeE8) {
        c.adminNotionalFeeE8 = newMaxAdminNotionalFeeE8;
        changed = true;
    }
    if (changed) {
        IMetricOmmPoolFactoryActions(pool).setPoolFees(
            c.protocolSpreadFeeE6 + c.adminSpreadFeeE6,
            c.protocolNotionalFeeE8 + c.adminNotionalFeeE8
        );
    }
}
```

Alternatively, document that `setPoolProtocolFee` must be called for every pool immediately after `setFeeCaps` and enforce this via a multi-call helper.

---

### Proof of Concept

1. Factory owner deploys Pool P1 with `adminSpreadFeeE6 = 100_000`. `poolFeeConfig[P1].adminSpreadFeeE6 = 100_000`.
2. Factory owner calls `setFeeCaps(200_000, 10_000, 1_000_000, 1_000_000)` — `maxAdminSpreadFeeE6` drops to 10 000.
3. `poolFeeConfig[P1].adminSpreadFeeE6` is **still 100 000** — no clamping occurs.
4. Swaps accumulate surplus in P1.
5. Anyone calls `collectPoolFees(P1)`. The factory passes `adminSpreadFeeE6_ = 100_000` to `pool.collectFees`.
6. Inside `collectFees`, `spreadFee0ToAdminScaled = surplus0Scaled * 100_000 / (100_000 + protocolSpreadFeeE6)` — the admin receives a share 10× above the new cap, and the protocol receives a correspondingly reduced share. [6](#0-5) [7](#0-6) [8](#0-7)
