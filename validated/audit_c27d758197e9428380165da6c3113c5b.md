### Title
Participant-Set Mismatch Between Triple Generation and Presigning Corrupts Presignature Outputs — (`File: src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA `presign()` function accepts a `participants` slice and a pair of `PresignArguments` containing `TriplePub` structs that each carry their own `participants: Vec<Participant>` field recording who participated in triple generation. The function validates that the threshold matches between the presign call and the triples, but **never validates that the participant set matches**. The code even contains an explicit comment acknowledging this omission. Because Lagrange linearization of the triple shares is performed using the presign-time participant set rather than the triple-generation participant set, any mismatch between the two sets causes the linearized nonce `k'` and blinding value `sigma'` to be mathematically incorrect, producing a corrupted and unusable presignature.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the public entry point `presign()` performs the following validation:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The `TriplePub` struct records the participant set used during triple generation:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    /// The threshold which will be able to reconstruct it.
    pub threshold: ReconstructionLowerBound,
}
``` [2](#0-1) 

Inside `do_presign`, the Lagrange coefficient is computed from the presign-time `participants` (a `ParticipantList` built from the caller-supplied slice), not from `TriplePub.participants`:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;

let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
let x_prime_i = lambda_me * private_share;
``` [3](#0-2) 

The triple shares `k_i`, `a_i`, `b_i`, `c_i`, `e_i` are Shamir shares of secrets `k`, `a`, `b`, `c`, `e` relative to the participant set used **during triple generation**. Lagrange interpolation of these shares is only correct when the same participant set (and thus the same Lagrange basis) is used. If the presign participant set differs — even by a single participant being added or removed — the Lagrange coefficients change, and the linearized values `k'`, `sigma'` no longer satisfy the required algebraic relationships. The `TriplePub.participants` field is never read inside `do_presign`.

---

### Impact Explanation

When the presign participant set differs from the triple-generation participant set:

- `k' = Σ λᵢ_new · kᵢ ≠ k` (the actual secret nonce)
- `σ' = α · xᵢ − β · aᵢ + cᵢ` computed with wrong `α`, `β` (since `α = k' + a'`, `β = x' + b'`)
- The resulting `big_r = (D · e⁻¹)` is inconsistent with `k'`

The presignature output `(big_r, k, sigma)` is cryptographically invalid. Any subsequent call to `sign()` using this presignature will produce a signature that fails the internal verification check:

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed(
        "signature failed to verify".to_string(),
    ));
}
``` [4](#0-3) 

This constitutes **corruption of presign outputs** causing honest parties to accept unusable cryptographic material, matching the allowed High impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

The vulnerability is reachable by any caller of the public `presign()` API. Realistic triggering scenarios include:

1. **Participant dropout between phases**: Triples are generated with `{P1, P2, P3}` (threshold 2). Before presigning, P3 drops out. The application calls `presign()` with `{P1, P2}` and the same triples. The threshold check passes (both are 2), but the participant sets differ.
2. **Malicious coordinator**: A coordinator deliberately supplies a participant list that differs from the one used during triple generation, causing all honest participants to produce corrupted presignature shares.
3. **New participant added**: A new participant P4 joins between triple generation and presigning. The presign call includes P4, changing all Lagrange coefficients.

The code comment explicitly acknowledges the omitted check, indicating the developers were aware of the assumption but chose not to enforce it.

---

### Recommendation

Validate that the participant set passed to `presign()` matches `TriplePub.participants` for both triples. Since `TriplePub.participants` is a plain `Vec<Participant>` (unsorted), convert both to `ParticipantList` (which sorts) and compare:

```rust
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;

if triple0_participants.participants() != participants.participants()
    || triple1_participants.participants() != participants.participants()
{
    return Err(InitializationError::BadParameters(
        "Presign participant set must match the participant set used during triple generation".to_string(),
    ));
}
```

Remove the comment acknowledging the omitted check and replace it with this enforcement.

### Proof of Concept

**Setup**: Triple generation with `{P0, P1, P2}`, threshold 2.

```
triple0 and triple1 generated with participants = [P0, P1, P2], threshold = 2
Lagrange coefficient for P0 relative to {P0,P1,P2}: λ_P0 = (P1·P2)/((P1-P0)·(P2-P0))
```

**Attack**: Presign called with `{P0, P1}`, threshold 2 (threshold check passes).

```
Lagrange coefficient for P0 relative to {P0,P1}: λ'_P0 = P1/(P1-P0)  ≠  λ_P0
k'_P0 = λ'_P0 · k_P0  ≠  λ_P0 · k_P0
k' = λ'_P0·k_P0 + λ'_P1·k_P1  ≠  k  (the actual secret nonce)
```

**Result**: `big_r = D · e⁻¹` is inconsistent with `k'`. The presignature `(big_r, k_P0, sigma_P0)` is mathematically invalid. When `sign()` is called, the coordinator assembles `s = h·k' + r·σ'` which does not satisfy `s·G = h·R + r·X`, causing the internal `sig.verify()` check to fail with `ProtocolError::AssertionFailed("signature failed to verify")`. Signing is permanently denied for this presignature, and the triple is consumed and cannot be reused. [5](#0-4) [2](#0-1)
