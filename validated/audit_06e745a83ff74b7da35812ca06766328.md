### Title
`uint32` Truncation in `_afterTimelock` Allows Pool Admin to Bypass LP-Protection Timelocks — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` computes the unlock timestamp as `uint32(block.timestamp + timelock)`. When `timelock` is set to a sufficiently large `uint32` value at pool creation, the sum overflows `uint32` and wraps to a value already in the past, making every subsequent proposal immediately executable and completely bypassing the LP-protection timelock.

---

### Finding Description

`_afterTimelock` is the single function that stamps every pending-change deadline (drawdown, decay, high watermarks, and the timelock itself):

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256`; `oracleStopLossConfig[pool_].timelock` is `uint32`. The addition is performed in `uint256` space, then **silently truncated** to `uint32`. Solidity ≥ 0.8 does not protect against explicit narrowing casts.

`uint32.max` = 4,294,967,295 (≈ year 2106). Current `block.timestamp` (July 2026) ≈ 1,753,000,000. Any `timelock` value greater than `4,294,967,295 − 1,753,000,000 ≈ 2,541,967,295` (~80 years, well within `uint32` range) causes the sum to exceed `uint32.max`. The truncated result is a small number already in the past.

`_requireElapsed` then compares `block.timestamp` (`uint256`) against the wrapped `uint32` value (implicitly upcast to `uint256`):

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

Because `block.timestamp` (~1.75 billion) is far greater than the wrapped result (~1 or any small number), the check always passes immediately.

The `timelock` parameter is decoded from `extensionInitData` at pool creation with **no upper-bound validation**:

```solidity
// line 56-58
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// ← no _validateTimelock

### Impact Explanation

The `OracleValueStopLossExtension` is the sole LP-protection mechanism against pool-admin manipulation of stop-loss parameters. Its entire security model rests on the timelock giving LPs time to exit before the admin can:

- Lower `drawdownE6` to 0 → disable stop-loss entirely, removing LP protection against value leakage.
- Raise high watermarks to `type(uint104).max` → make the stop-loss trigger on any swap, permanently bricking the pool for all users.
- Set `decayPerSecondE8` to `E8` (maximum) → watermarks decay to zero in one second, making the stop-loss permanently inactive.

With the timelock bypassed, all three changes can be proposed and executed in the same block. LPs have zero time to react. This constitutes a direct admin-boundary break (pool admin exceeds the timelock cap) with potential for LP principal loss or complete pool DoS. [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The trigger requires only that the pool admin supply a `timelock` value ≥ ~2.54 billion (e.g., `type(uint32).max`) in `extensionInitData` at pool creation. This is a single-parameter choice made by the pool admin, who controls the creation call. No external conditions, oracle states, or other users are needed. The pool admin is semi-trusted only within caps and timelocks; this path lets them escape that constraint entirely. [7](#0-6) 

---

### Recommendation

Add an upper-bound check on `timelock` in `initialize`, and replace the unsafe narrowing cast in `_afterTimelock` with a checked addition:

```solidity
// In initialize():
uint32 MAX_TIMELOCK = 365 days; // or any protocol-chosen cap
if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);

// In _afterTimelock():
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + oracleStopLossConfig[pool_].timelock;
    if (result > type(uint32).max) revert OracleStopLossTimelockOverflow();
    return uint32(result);
}
```

Apply the same cap to `proposeOracleStopLossTimelock` so the admin cannot introduce an overflowing value post-deployment. [8](#0-7)
