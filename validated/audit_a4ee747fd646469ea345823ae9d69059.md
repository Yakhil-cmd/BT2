Audit Report

## Title
ckERC20 Withdrawal Funds Locked Indefinitely When Gas Prices Exceed Locked `max_transaction_fee` — (`rs/ethereum/cketh/minter/src/main.rs`, `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary
When `withdraw_erc20()` is called, the minter immediately and irrevocably burns both the user's ckETH fee and ckERC20 tokens, storing the fee estimate as the immutable `max_transaction_fee` field in `Erc20WithdrawalRequest`. If Ethereum gas prices rise above this locked cap before the transaction is created, the request is rescheduled indefinitely with no cancellation, fee top-up, or timeout-based reimbursement mechanism available to the user, leaving both the ckETH fee and the ckERC20 tokens (representing real ERC-20 value held by the minter) inaccessible for an unbounded duration.

## Finding Description
In `main.rs` at L430–432, `estimate_erc20_transaction_fee()` is called (using a gas estimate cached up to 60 seconds stale per `tx.rs` L611) and the result is immediately burned from the user's ckETH balance at L448–458. The ckERC20 tokens are then burned at L468–477. Both burns are committed before the `Erc20WithdrawalRequest` is enqueued at L480–492 with `max_transaction_fee: erc20_tx_fee` as a fixed field — the `Erc20WithdrawalRequest` struct (`transactions/mod.rs` L146–177) has no update path for this field.

At transaction creation time (`transactions/mod.rs` L1155–1168), `actual_min_max_fee_per_gas > request_max_fee_per_gas` causes `CreateTransactionError::InsufficientTransactionFee` to be returned. In `withdraw.rs` L281–291, this error unconditionally calls `reschedule_withdrawal_request`, moving the request to the back of the queue with no state change. This loop repeats on every timer tick indefinitely.

The reimbursement path at `transactions/mod.rs` L733–745 only triggers when a transaction has been finalized on-chain with `TransactionStatus::Failure` — a stuck-pending request never reaches this code path. The public interface (`cketh_minter.did` L696–750) exposes no `cancel_erc20_withdrawal`, `top_up_erc20_withdrawal_fee`, or equivalent endpoint.

## Impact Explanation
This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."** Any user who calls `withdraw_erc20()` just before a gas spike has their ckERC20 tokens (backed by real ERC-20 tokens held by the minter's Ethereum address) and ckETH fee burned with no recovery path. The ERC-20 tokens remain locked in the minter's Ethereum address for as long as gas prices stay elevated. There is no timeout, no cancellation, and no reimbursement for stuck-pending requests — only natural gas price recovery unblocks the user.

## Likelihood Explanation
Ethereum gas price spikes of 3–10× within minutes are well-documented during high-demand events. The 60-second cache (`MAX_AGE_NS = 60_000_000_000`) means the fee estimate used at burn time can already be stale. The fee formula (`2 * base_fee + priority_fee`) buffers for a few blocks but not sustained spikes. No special privileges are required; any user calling `withdraw_erc20()` during normal conditions just before a spike triggers this. The condition is repeatable and affects all pending ckERC20 withdrawal requests simultaneously during a spike.

## Recommendation
1. Add a `cancel_erc20_withdrawal` endpoint restricted to the original caller that cancels a pending (pre-signed) request and triggers reimbursement of both the ckETH fee and ckERC20 tokens via the existing `record_reimbursement_request` infrastructure.
2. Alternatively, extend the reimbursement logic to cover requests that have been stuck in the pending queue beyond a configurable timeout (e.g., 24 hours), automatically returning the ckERC20 tokens and ckETH fee to the user.
3. At minimum, add a `top_up_erc20_withdrawal_fee` endpoint allowing the original caller to burn additional ckETH to increase the `max_transaction_fee` of a pending request.

## Proof of Concept
1. User calls `eip_1559_transaction_price` and observes `max_transaction_fee = X`.
2. User approves minter for `X` ckETH and calls `withdraw_erc20(amount, ckerc20_ledger_id, recipient)`.
3. Minter burns `X` ckETH and `amount` ckERC20 tokens; `Erc20WithdrawalRequest { max_transaction_fee: X }` is enqueued.
4. Ethereum gas prices spike to `3X` before the minter's next timer tick.
5. `create_transactions_batch()` evaluates `actual_min_max_fee_per_gas > request_max_fee_per_gas` → `true` → returns `InsufficientTransactionFee`.
6. `reschedule_withdrawal_request` moves the request to the back of the queue.
7. Steps 5–6 repeat on every timer tick.
8. User queries `withdrawal_status` and observes the request cycling indefinitely. No endpoint exists to cancel or recover funds.
9. Verifiable via a deterministic integration test: set up a minter state with a pending `Erc20WithdrawalRequest`, inject a `GasFeeEstimate` exceeding `max_transaction_fee`, call `create_transactions_batch`, and assert the request remains in the pending queue with no reimbursement event emitted.
