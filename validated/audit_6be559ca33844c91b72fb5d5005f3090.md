### Title
Malicious CKD Participant Can Corrupt Confidential Key Derivation Output Without Detection - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The CKD coordinator aggregates `(big_y_i, big_c_i)` shares from all participants with no cryptographic verification that each share is correctly formed. A single malicious participant can send arbitrary group elements, causing every honest party to accept a corrupted derived key without any indication of failure.

### Finding Description

The CKD protocol is a threshold ElGamal-based scheme. Each participant is expected to compute:

- `big_y_i = y_i * G` (random blinding point)
- `big_c_i = x_i * H(pk, app_id) + y_i * app_pk` (masked secret contribution)
- Normalize both by `lambda_i` and send to the coordinator

The coordinator in `do_ckd_coordinator` receives these values and sums them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is **no zero-knowledge proof, no commitment, and no consistency check** verifying that:

1. `big_y_i` is actually `y_i * G` for some `y_i`
2. `big_c_i` uses the **same** `y_i` as `big_y_i`
3. `big_c_i` encodes the participant's **actual** secret share `x_i`

A malicious participant can send any `(big_y', big_c')` pair. The coordinator blindly adds it to the running sum. The final `CKDOutput` is then returned to the requester, who unmasks it with `app_sk` to recover what they believe is `msk * H(pk, app_id)`.

This is the direct analog of the external report's vulnerability: in the limit-order protocol, `thresholdAmount` only bounds one side of the swap while the other side (`getTakerAmount` return value) is freely controlled by the malicious maker. Here, the `big_y` component is not verified against `big_c`, so a malicious participant controls both sides of their contribution independently.

### Impact Explanation

A malicious participant sends `(big_y_i + Δ_y, big_c_i + Δ_c)` for attacker-chosen `Δ_y, Δ_c ∈ G1`. The coordinator computes:

```
sum_big_y = Σ lambda_j * big_y_j  +  lambda_malicious * Δ_y
sum_big_c = Σ lambda_j * big_c_j  +  lambda_malicious * Δ_c
```

The requester unmasks:

```
sum_big_c - app_sk * sum_big_y
  = msk * H(pk, app_id)  +  lambda_malicious * (Δ_c - app_sk * Δ_y)
```

Since the malicious participant does not know `app_sk`, the additive error term `lambda_malicious * (Δ_c - app_sk * Δ_y)` is non-zero with overwhelming probability for any non-trivial `(Δ_y, Δ_c)`. The requester receives a wrong key with no error signal. All honest parties accept this corrupted output as valid.

**Matched impact**: High — Corruption of CKD outputs so honest parties accept unusable or inconsistent cryptographic outputs.

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to be a legitimate participant (no special privilege required). The attack requires no cryptographic break, no leaked keys, and no external dependency. The malicious participant simply deviates from the protocol by sending arbitrary group elements instead of correctly computed shares. The coordinator has no mechanism to detect the deviation.

### Recommendation

Add a zero-knowledge proof of correct formation for each participant's `(big_y_i, big_c_i)` contribution. Specifically, each participant should prove in zero knowledge that:

- `big_y_i = y_i * G` for some `y_i` (discrete log proof)
- `big_c_i = x_i * H(pk, app_id) + y_i * app_pk` using the **same** `y_i` (a Chaum-Pedersen-style proof of equality of discrete logs across two bases: `G` and `app_pk`)

The coordinator must verify all proofs before aggregating. This is the threshold-signatures analog of the fix described in the external report: removing the unchecked truncation path and requiring the full value to be validated before use.

### Proof of Concept

1. Honest participants P1, P2 and malicious participant P3 run `ckd()`.
2. P1 and P2 compute and send correct `(norm_big_y_1, norm_big_c_1)` and `(norm_big_y_2, norm_big_c_2)` to the coordinator.
3. P3 computes its correct share but instead sends `(norm_big_y_3 + Δ, norm_big_c_3)` where `Δ` is an arbitrary non-identity point in G1.
4. The coordinator in `do_ckd_coordinator` sums all three contributions without any check.
5. The coordinator returns `CKDOutput::new(sum_big_y, sum_big_c)` where `sum_big_y` is polluted by `lambda_3 * Δ`.
6. The requester calls `ckd_output.unmask(app_sk)` and obtains `msk * H(pk, app_id) - app_sk * lambda_3 * Δ`, which is not the correct confidential key.
7. No error is raised anywhere in the protocol; all honest parties accept the corrupted output.

The entry path is `ckd()` → `run_ckd_protocol()` → `do_ckd_coordinator()` at the coordinator, receiving from `do_ckd_participant()` at the malicious party. [1](#0-0) [2](#0-1)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
