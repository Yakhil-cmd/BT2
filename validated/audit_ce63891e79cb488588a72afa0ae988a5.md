### Title
`MAX_NUMBER_OF_LOGS` Cap Exhaustion via L1 Messenger Allows Indefinite Delay of L1→L2 Priority Transactions — (`File: basic_bootloader/src/bootloader/block_flow/zk/mod.rs`)

---

### Summary

The ZK transaction loop enforces a hard per-block cap of `MAX_NUMBER_OF_LOGS = 16,384` on L2→L1 logs. This cap is checked uniformly for every transaction, including L1→L2 priority transactions. An unprivileged attacker can fill the block's log capacity by sending L2 transactions that emit L2→L1 messages via the L1 messenger system contract. Once the capacity is saturated, any subsequent L1→L2 priority transaction that would add its mandatory execution-result log is rejected with `BlockL2ToL1LogsLimitReached` and rolled back. If the attacker repeats this in every block, L1→L2 deposits are indefinitely deferred, locking user funds on L1.

---

### Finding Description

**Hard cap definition**

`MAX_NUMBER_OF_LOGS` is a fixed constant of `16_384`, derived from the depth of the L2→L1 Merkle tree: [1](#0-0) 

Two distinct log types share this single pool:
- **User messages** pushed by the L1 messenger hook (`push_message`).
- **L1→L2 execution-result logs** pushed unconditionally for every processed L1 priority transaction (`push_l1_l2_tx_log`). [2](#0-1) 

**Uniform cap enforcement in the ZK tx loop**

After every transaction (L1 or L2) is executed, `check_for_block_limits` is called. If the current log count exceeds `MAX_NUMBER_OF_LOGS`, the transaction is rolled back and reported as `BlockL2ToL1LogsLimitReached`: [3](#0-2) 

The rollback path in the tx loop applies identically to all transaction types: [4](#0-3) 

**L1→L2 transactions generate a mandatory log**

Every L1→L2 priority transaction unconditionally appends one `L1TxLog` entry to prove its execution result on L1. There is no bypass or reservation mechanism. If the pool is already at `MAX_NUMBER_OF_LOGS`, the transaction's log push would bring the count to `16,385 > 16,384`, triggering the rejection. [5](#0-4) 

**Design intent contradicted**

The codebase explicitly states that L1 transactions must not be invalidatable because doing so halts the chain: [6](#0-5) 

Yet `check_for_block_limits` provides no exemption for priority transactions, directly contradicting this invariant. [7](#0-6) 

**Attacker-controlled entry path**

The L1 messenger hook is callable by any L2 transaction through the L1 messenger system contract. The hook enforces only that the immediate caller is `L1_MESSENGER_ADDRESS` (the system contract), not that the originating user is privileged: [8](#0-7) 

An attacker deploys a contract that calls the L1 messenger system contract in a loop, emitting user-message logs. Each call consumes one slot from the shared `MAX_NUMBER_OF_LOGS` pool. The attacker fills the pool to exactly `16,384` across one or more transactions (each individual transaction is accepted because `16,384 > 16,384` is false). The very next L1→L2 priority transaction then attempts to push a 16,385th log and is rejected.

---

### Impact Explanation

L1→L2 priority transactions carry user funds locked on L1. If such a transaction is rejected due to `BlockL2ToL1LogsLimitReached`, the funds remain locked on L1 until the transaction is successfully included in a future block. If the attacker repeats the log-flooding attack in every block, the L1→L2 transaction is indefinitely deferred. This is a direct analog to the RageTrade utilization-cap bug: a protocol-level cap, unreachable by the affected party, prevents access to funds for an unbounded period. The impact is loss of liveness for L1→L2 deposits and potential chain instability if priority-queue processing is halted.

---

### Likelihood Explanation

The L1 messenger system contract is publicly callable. Emitting a single L2→L1 message costs a bounded amount of gas. With a block gas limit in the tens of millions, an attacker can emit thousands of messages per transaction and reach `16,384` total logs across a small number of transactions per block. The cost is proportional to gas fees, making sustained attacks expensive but feasible for a motivated adversary. No privileged access, governance majority, or oracle manipulation is required.

---

### Recommendation

1. **Reserve log slots for L1→L2 transactions.** Maintain a separate counter for user-message logs and cap it at `MAX_NUMBER_OF_LOGS - <max_l1_txs_per_block>`, ensuring L1 priority transactions always have available slots.
2. **Exempt L1→L2 transactions from the log-count block limit check**, consistent with the existing design intent that priority transactions cannot be invalidated.
3. **Charge per-message pubdata and gas costs** at a rate that makes filling `16,384` slots economically prohibitive within a single block.

---

### Proof of Concept

```
1. Deploy contract C on L2:
       fallback() { for i in 0..N { L1Messenger.sendToL1(""); } }

2. Send L2 transactions calling C with large gas limits until
   system.io.logs_len() == 16_384.
   Each transaction is accepted (16_384 > 16_384 is false).

3. Submit an L1→L2 priority transaction (e.g., a deposit).

4. The bootloader calls push_l1_l2_tx_log → logs_len becomes 16_385.
   check_for_block_limits: 16_385 > 16_384 → BlockL2ToL1LogsLimitReached.
   Transaction is rolled back; user funds remain locked on L1.

5. Repeat steps 2–4 in every block to maintain the denial indefinitely.
```

fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64,
    computational_native_used: u64,
    pubdata_used: u64,
    logs_used: u64,
    blob_gas_used: u64,
) -> Result<(), InvalidTransaction>
where
    S::IO: IOSubsystemExt,
    <S as SystemTypes>::Metadata: ZkSpecificPricingMetadata,
{
    if gas_used > system.get_gas_limit() {
        system_log!(
            system,
            "Block gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockGasLimitReached)
    } else if blob_gas_used > system.get_blob_gas_limit() {
        system_log!(
            system,
            "Block blob gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockBlobGasLimitReached)
    } else if !cfg!(feature = "resources_for_tester")
        && computational_native_used > MAX_NATIVE_COMPUTATIONAL
    {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block native limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockNativeLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block pubdata limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockPubdataLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block logs limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
    } else {
        Ok(())
    }
}
