### Title
Unauthorized Overwrite of `AccountId`-Mode Global Contract via Cross-Contract Call â€” (`runtime/runtime/src/global_contracts.rs`)

### Summary

`action_deploy_global_contract` uses the receipt's `receiver_id` as the `GlobalContractIdentifier::AccountId` key when `deploy_mode == AccountId`, but never verifies that the receipt's signer or predecessor is the same account. Any account that has a deployed contract can create a cross-contract call targeting a victim account and append `DeployGlobalContractAction { deploy_mode: AccountId }` to that call, overwriting the victim's AccountId-mode global contract with arbitrary code. Every account that has called `UseGlobalContract(AccountId("victim.near"))` will subsequently execute the attacker's code.

### Finding Description

`GlobalContractDeployMode::AccountId` is documented as owner-controlled: the contract is stored under the deployer's own account ID so that the owner can update the contract for all its users. [1](#0-0) 

The execution handler `action_deploy_global_contract` receives `account_id` (the receipt's `receiver_id`) and passes it directly to `initiate_distribution`, which stores the code under `GlobalContractIdentifier::AccountId(account_id)`: [2](#0-1) [3](#0-2) 

There is no check that `account_id == signer_id` or `account_id == predecessor_id`. For a direct transaction `signer_id == receiver_id` holds by convention, but for a cross-contract receipt generated by a contract via the host function `promise_batch_action_deploy_global_contract_by_account_id`, the receipt's `receiver_id` is the target account, not the calling contract: [4](#0-3) 

The host function is available in production (enabled at protocol version 78, `global_contract_host_fns: true`). The action validation path for `NewReceipt` mode only checks `ContractSizeExceeded` for `DeployGlobalContractAction`; there is no ownership guard: [5](#0-4) 

**Attack path:**

1. Attacker deploys a contract on `attacker.near`.
2. Attacker calls their contract.
3. Contract calls `promise_batch_create("victim.near")` â†’ creates a receipt with `receiver_id = "victim.near"`.
4. Contract calls `promise_batch_action_deploy_global_contract_by_account_id(promise_idx, malicious_code_len, malicious_code_ptr)`.
5. The receipt `{ receiver_id: "victim.near", actions: [DeployGlobalContractAction { code: malicious, deploy_mode: AccountId }] }` is submitted.
6. On execution, `action_deploy_global_contract` is called with `account_id = "victim.near"`.
7. `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId("victim.near") }` is overwritten with malicious code.
8. All accounts that previously called `UseGlobalContract(AccountId("victim.near"))` now execute the attacker's Wasm.

Additionally, the storage cost is deducted from the victim's account balance (line 39â€“46 of `global_contracts.rs`), draining NEAR from the victim without consent.

### Impact Explanation

- **Global contract/code selection**: The `AccountId`-mode global contract namespace is not owner-isolated. Any account's global contract can be silently replaced with malicious code.
- **Accounting**: Victim's NEAR balance is reduced by `global_contract_storage_amount_per_byte * code_len` without authorization.
- **Downstream execution integrity**: Every account that has opted into `UseGlobalContract(AccountId("victim.near"))` will execute attacker-controlled Wasm on all future calls, enabling theft of attached deposits, unauthorized state mutations, and key exfiltration.

### Likelihood Explanation

Any unprivileged user who has a contract deployed (a common, low-barrier precondition) can execute this attack against any account that has deployed an AccountId-mode global contract. No validator, admin, or governance privilege is required. The `global_contract_host_fns` feature is enabled in the current mainnet protocol version.

### Recommendation

In `action_deploy_global_contract`, when `deploy_mode == GlobalContractDeployMode::AccountId`, assert that the receipt's `signer_id` (or `predecessor_id`) equals `account_id`. Concretely, the receipt's `ActionReceipt::signer_id` is available via `action_receipt.signer_id()` and should be compared to `account_id` before calling `initiate_distribution`. Alternatively, restrict `DeployGlobalContractAction` with `AccountId` mode to receipts where `receiver_id == signer_id` (i.e., self-directed transactions only), rejecting cross-contract uses of this mode.

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}

fn initiate_distribution(
    state_update: &mut TrieUpdate,
    account_id: AccountId,
    contract_code: Arc<[u8]>,
    deploy_mode: &GlobalContractDeployMode,
    current_shard_id: ShardId,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))
        }
        GlobalContractDeployMode::AccountId => {
            GlobalContractIdentifier::AccountId(account_id.clone())
        }
    };
    // Increment the nonce and write it to state immediately to prevent multiple
    // distributions with the same nonce from being initiated. This requires
    // allowing the same nonce in the freshness check when applying the
    // distribution receipt.
    let nonce = increment_nonce(state_update, &id)?;
    let distribution_receipt =
        GlobalContractDistributionReceipt::new(id, current_shard_id, vec![], contract_code, nonce);
    let distribution_receipts =
        Receipt::new_global_contract_distribution(account_id, distribution_receipt);
    // No need to set receipt_id here, it will be generated as part of apply_action_receipt
    result.new_receipts.push(distribution_receipts);
    Ok(())
