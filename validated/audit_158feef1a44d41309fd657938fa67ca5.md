### Title
Incomplete Fee Breakdown in `FeeEstimation`: `overall_fee` Includes L2 Gas Cost But `l2_gas_consumed` Is Absent — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`tx_execution_output_to_fee_estimation` builds a `FeeEstimation` response where `overall_fee` is taken directly from `receipt.fee` (which includes L1 gas + L1 data gas + L2 gas costs for V3 transactions), but the response only exposes `gas_consumed` (L1 gas) and `data_gas_consumed` (L1 data gas). The L2 gas component is silently omitted. The struct's own documentation asserts the invariant `overall_fee = gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price`, which is false for any V3 transaction that consumes L2 gas.

### Finding Description

In `tx_execution_output_to_fee_estimation`:

```rust
let gas_vector = tx_execution_output.execution_info.receipt.gas;

Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),          // L1 gas only
    l1_gas_price,
    data_gas_consumed: gas_vector.l1_data_gas.0.into(), // L1 data gas only
    l1_data_gas_price,
    l2_gas_price,                                       // price present, but no l2_gas_consumed
    overall_fee: tx_execution_output.execution_info.receipt.fee, // includes L2 gas cost
    unit: tx_execution_output.price_unit,
})
``` [1](#0-0) 

`receipt.fee` is the total fee computed by blockifier for the full `GasVector { l1_gas, l1_data_gas, l2_gas }`. For a V3 transaction with non-zero `l2_gas`, this equals:

```
receipt.fee = l1_gas * l1_gas_price + l1_data_gas * l1_data_gas_price + l2_gas * l2_gas_price
```

But the response only exposes the first two terms. The struct documentation at line 109 explicitly states the invariant:

> "The total amount of fee. This is equal to: gas_consumed * gas_price + data_gas_consumed * data_gas_price." [2](#0-1) 

This invariant is violated for any V3 transaction with L2 gas usage. The developer TODO at line 104 acknowledges the missing field:

> `// TODO(Tzahi): Add l2_gas_consumed. Verify overall_fee estimation of l1_gas_price only is close enough (as there are roundings) to the fee of both l1_gas_price and l2_gas_price.` [3](#0-2) 

This function is called from both `estimate_fee` and `simulate_transactions`, meaning both `starknet_estimateFee` and `starknet_simulateTransactions` return this broken breakdown. [4](#0-3) [5](#0-4) 

### Impact Explanation

Any caller of `starknet_estimateFee` or `starknet_simulateTransactions` for a V3 transaction with L2 gas consumption receives a `FeeEstimation` where:

- `overall_fee` is correct (what blockifier actually computed and would charge)
- `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price < overall_fee` (the L2 gas portion is invisible)
- `l2_gas_price` is present in the response but `l2_gas_consumed` is absent, making the response internally inconsistent

Clients that trust the documented invariant and compute `gas_consumed * l1_gas_price + data_gas_consumed * l1_data_gas_price` to verify or display the fee will see a value lower than `overall_fee` with no explanation. This is an authoritative-looking wrong value from the RPC simulation/estimation endpoint, matching the defined High impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

Any V3 (STRK-fee) transaction that executes Cairo code consuming L2 gas triggers this. V3 transactions are the current standard for STRK fee payment. The condition is trivially reachable by any unprivileged user submitting a normal V3 invoke transaction.

### Recommendation

Add `l2_gas_consumed: gas_vector.l2_gas.0.into()` to the `FeeEstimation` struct and populate it in `tx_execution_output_to_fee_estimation`. Update the `overall_fee` documentation to reflect the three-component sum. This is already tracked by the developer TODO at line 104.

### Proof of Concept

The assertion that fails for a V3 transaction with L2 gas:

```rust
assert_eq!(
    fee_estimation.gas_consumed * fee_estimation.l1_gas_price
        + fee_estimation.data_gas_consumed * fee_estimation.l1_data_gas_price,
    fee_estimation.overall_fee.0
    // Fails: LHS is missing the l2_gas * l2_gas_price term that is included in overall_fee
);
```

The `gas_vector` at line 172 contains `l2_gas` from `receipt.gas`, but only `l1_gas` and `l1_data_gas` are placed into the response. The `l2_gas` contribution is absorbed silently into `overall_fee` with no corresponding `l2_gas_consumed` field to account for it. [6](#0-5)
