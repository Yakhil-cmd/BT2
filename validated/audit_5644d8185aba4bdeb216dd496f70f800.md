### Title
Malicious Participant Can Corrupt CKD Output by Submitting Shares for a Different Application Context — (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The CKD coordinator aggregates participant shares without any cryptographic verification that each share was computed for the correct `app_id` and `app_pk`. A malicious participant can submit a share computed for a different application identity (or arbitrary group elements), and the coordinator will silently incorporate it, producing a corrupted CKD output that honest parties accept as valid.

---

### Finding Description

The external report describes a root-bundle context mismatch: every spoke pool receives the same `slowRelayRoot` representing all slow relays across the entire system, and the Spoke Pool never filters on `destinationChainId`, so a relay intended for chain A can be executed on chain B. The analog here is structurally identical: the CKD coordinator receives shares from all participants and aggregates them without verifying that each share was computed for the session's `app_id` and `app_pk`.

In `do_ckd_coordinator` the aggregation is:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant computes their share as:

```
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + app_pk · y_i
``` [2](#0-1) 

The coordinator receives `(norm_big_y, norm_big_c)` from every participant over a private channel and adds them together. There is **no zero-knowledge proof, no commitment, and no binding** that ties each received pair to the session's `app_id` or `app_pk`. The protocol contains no analogue of the session-ID binding used in DKG (`do_keyshare` hashes all participants' random session IDs into a shared `session_id` and binds every proof of knowledge to it). In CKD there is no such session context at all. [3](#0-2) 

A malicious participant `P_m` can:

1. Compute `(big_y_m, big_c_m)` using a different `app_id'` (or any arbitrary BLS12-381 G1 elements).
2. Send those values to the coordinator in place of the honest share.
3. The coordinator adds them in unconditionally.

The resulting aggregated output is:

```
Y_out = Σ_{i≠m} λ_i · y_i · G  +  λ_m · y_m' · G
C_out = Σ_{i≠m} λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)
      + λ_m · (x_m · H(pk‖app_id') + y_m' · app_pk)
```

When the application unmasks with `app_sk`:

```
C_out − app_sk · Y_out  ≠  msk · H(pk ‖ app_id)
```

The output is silently wrong. No error is raised; honest parties accept it as the CKD result.

Compare with `do_keyshare`, which binds every proof of knowledge to the session ID and verifies it before accepting any participant's commitment: [4](#0-3) 

No equivalent verification exists in the CKD path.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator and all honest participants accept the aggregated `(Y_out, C_out)` as the legitimate CKD result. The application TEE decrypts it and obtains a secret that is not `msk · H(pk ‖ app_id)`. The derived key is silently wrong. Depending on how the application uses the secret, this can cause silent key derivation failure, cross-application key confusion (if `app_id'` was chosen deliberately), or permanent denial of the correct derived key for that `app_id`.

---

### Likelihood Explanation

**Medium.** Any single participant in a CKD session can trigger this unilaterally. No special privilege is required beyond being a listed participant. The attack requires no cryptographic break and is trivially executable: the malicious party simply calls `compute_signature_share` with a different `app_id` (or constructs arbitrary G1 elements) before sending to the coordinator. Because the coordinator performs no verification, the attack succeeds with probability 1 whenever a malicious participant is present.

---

### Recommendation

Add a non-interactive zero-knowledge proof (e.g., a Sigma/Maurer proof) that each participant's `(big_y, big_c)` was computed correctly with respect to their committed public verification share and the session's `app_id` and `app_pk`. Concretely, each participant should prove knowledge of `(y_i, x_i)` such that:

- `big_y = y_i · G`
- `big_c = x_i · H(pk ‖ app_id) + y_i · app_pk`

and that `x_i · G2` matches their public verification share from the DKG output. The coordinator must verify all proofs before aggregating. This mirrors the proof-of-knowledge verification already present in `do_keyshare` for DKG/reshare. [5](#0-4) 

---

### Proof of Concept

```
Setup: 3 participants P1, P2, P3 (threshold 2-of-3), app_id = "AppA", app_pk = app_sk·G.

Honest run:
  P1, P2 compute (big_y_i, big_c_i) using app_id="AppA"
  P3 (malicious) computes (big_y_3, big_c_3) using app_id="AppB"

Coordinator receives and sums all three pairs.

Result:
  C_out − app_sk·Y_out
    = λ1·x1·H(pk‖AppA) + λ2·x2·H(pk‖AppA) + λ3·x3·H(pk‖AppB)
    ≠ msk·H(pk‖AppA)

Application TEE decrypts and silently obtains a wrong key.
No error is returned by the protocol.
```

The entry path is: malicious participant → `do_ckd_participant` sends crafted `(norm_big_y, norm_big_c)` → coordinator's `do_ckd_coordinator` aggregates without verification → corrupted `CKDOutput` returned to caller. [6](#0-5) [7](#0-6)

/// Verifies the proof of knowledge of the secret coefficients used to generate the
/// public secret sharing commitment.
fn internal_verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    participant: Participant,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: &Signature<C>,
) -> Result<(), ProtocolError> {
    // creates an identifier for the participant
    let id = participant.scalar::<C>();
    let vk_share = commitment
        .coefficients()
        .first()
        .ok_or_else(|| ProtocolError::AssertionFailed("Empty coefficient list".to_string()))?;

    let big_r = proof_of_knowledge.R();
    let z = proof_of_knowledge.z();
    let c = challenge::<C>(domain_separator, session_id, id, vk_share, big_r)?;
    if *big_r != <C::Group>::generator() * *z - vk_share.value() * c.to_scalar() {
        return Err(ProtocolError::InvalidProofOfKnowledge(participant));
    }
    Ok(())
}
