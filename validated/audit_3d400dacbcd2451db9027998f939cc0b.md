### Title
Permanent Fund Freeze When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is disabled (the default), the `ExitToNear` precompile schedules a bare `ft_transfer` NEAR promise with **no failure callback**. If that promise fails — for example, because the recipient NEAR account is not registered with the NEP-141 token — the user's EVM-side ETH or ERC-20 tokens are permanently destroyed with no refund path. This is the direct analog of the BoxExchange push-over-pull vulnerability: a "push" transfer to an external account with no recovery mechanism on failure.

### Finding Description

The `ExitToNear` precompile's `run()` method in `engine-precompiles/src/native.rs` constructs a `callback_args` struct: [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is hardcoded to `None`. For a standard ETH base-token exit or a standard ERC-20 `ft_transfer` exit (neither of which sets `transfer_near`), `callback_args` equals the `Default` value. The branch at line 470 then selects a bare `PromiseArgs::Create` — **no callback is attached**: [2](#0-1) 

The `exit_to_near_precompile_callback` function, which is the only place `refund_on_error` is called, is never scheduled: [3](#0-2) 

The `refund_on_error` function itself would re-mint burned ERC-20 tokens or transfer ETH back from the precompile address — but it is unreachable without the callback: [4](#0-3) 

The `error_refund` feature is **not** in the default feature set: [5](#0-4) 

The test suite explicitly acknowledges the fund loss when the feature is absent: [6](#0-5) [7](#0-6) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

- **ETH exit path**: The user's EVM ETH balance is debited at EVM execution time. The `ft_transfer` NEAR promise is then dispatched. If it fails (e.g., recipient not registered with the eth-connector NEP-141), the ETH is gone from the EVM state and no NEAR-side tokens are delivered. There is no callback to restore the EVM balance. The ETH is permanently frozen inside the connector.
- **ERC-20 exit path**: The ERC-20 tokens are burned from the user's EVM balance before the `ft_transfer` promise is dispatched. On promise failure, the tokens are permanently destroyed with no re-mint.

### Likelihood Explanation

**High.** Any user who calls the `ExitToNear` precompile targeting a NEAR account that is not registered with the relevant NEP-141 token will trigger this path. Unregistered accounts are a normal operational condition (e.g., a new NEAR account that has never interacted with the token). No attacker action is required — an ordinary user mistake is sufficient to trigger permanent loss. The entry path is fully unprivileged: any EVM transaction can call the precompile.

### Recommendation

Enable the `error_refund` feature in all production builds, or restructure the precompile so that a refund callback is **always** attached regardless of compile-time features. The callback should unconditionally carry `RefundCallArgs` so that any `ft_transfer` failure triggers `refund_on_error`. The current design of making refund opt-in via a feature flag creates a silent fund-loss footgun in default builds.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default).
2. Fund an EVM address with ETH on Aurora.
3. Call the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with ETH value and a recipient NEAR account that has never registered storage with the eth-connector NEP-141.
4. The EVM ETH balance is debited immediately.
5. The `ft_transfer` NEAR promise fails because the recipient is unregistered.
6. No callback fires; no refund occurs.
7. The user's ETH is permanently lost — confirmed by the test `test_exit_to_near_eth_refund` which shows `expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT)` when `error_refund` is absent. [8](#0-7)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
