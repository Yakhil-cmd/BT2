### Title
WalletConnect Offer Approval Displays Understated Royalty Total Due to Integer Division Truncation in `formatAmountWithRoyalties` - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before applying each royalty percentage. This causes the `amountWithRoyalties` field shown in the WalletConnect approval dialog to be significantly lower than the amount that will actually be charged on-chain when a `take_offer` or `create_offer_for_ids` command involves multiple NFTs with royalties.

### Finding Description

`parseCommandDisplay` is called in the Electron main process to generate the `display` object shown to the user in the WalletConnect command-approval dialog before they confirm a wallet operation. [1](#0-0) 

Inside `parseCommandDisplay`, the function `formatAmountWithRoyalties` computes the total spending amount including royalties: [2](#0-1) 

The critical flaw is on line 363:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length); // integer division
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The function divides `amount` by the number of NFTs (`royaltyPercentages.length`) **before** applying each royalty percentage. This means each NFT's royalty is computed on `amount / N` instead of the full `amount`. The correct formula is:

```
royaltyAmount = amount * r1 / 10000 + amount * r2 / 10000 + ...
```

But the code computes:

```
royaltyAmount = (amount/N) * r1 / 10000 + (amount/N) * r2 / 10000 + ...
```

This understates the total royalty by a factor of N (the number of NFTs). The `amountWithRoyalties` string returned is then embedded in the approval dialog `display` object and shown to the user. [3](#0-2) 

The actual on-chain royalty calculation is performed by the Chia wallet backend (`nft_calculate_royalties`), which applies each NFT's royalty to the full fungible amount. The GUI display is a separate client-side approximation that is wrong for N > 1.

### Impact Explanation

A user approving a WalletConnect `take_offer` or `create_offer_for_ids` command sees an `amountWithRoyalties` that is N times lower than what will actually be deducted. For example:

- Buying 2 NFTs each with 10% royalty for 1 XCH:
  - **Displayed**: 1.1 XCH (split approach: `(1 XCH / 2) * 10% * 2 = 0.1 XCH royalty`)
  - **Actual charge**: 1.2 XCH (`1 XCH * 10% + 1 XCH * 10% = 0.2 XCH royalty`)
  - **User overpays**: 0.1 XCH without informed consent

This matches the **High** impact category: "Corruption, spoofing, or unsafe trust of WalletConnect state that causes a user to approve the wrong amount."

### Likelihood Explanation

Any dApp connected via WalletConnect can send a `take_offer` command referencing an offer with multiple royalty-bearing NFTs. No special privileges are required. The condition (multiple NFTs with royalties in a single offer) is a normal, supported use case. The user has no way to detect the discrepancy from the approval dialog alone.

### Recommendation

Replace the `splitAmount` division with per-NFT royalty computation applied to the full amount:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Apply each royalty to the full amount, not a split amount
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

### Proof of Concept

The existing test at line 251 of `parseCommandDisplay.test.ts` inadvertently documents the bug. With `amount = 100000000` mojos and two NFTs with royalty percentages `[500, 10]`:

- **Current (buggy) result**: `amountWithRoyalties = '0.00010255'` (102,550,000 mojos)
- **Correct result**: 105,100,000 mojos (`100000000 * 500/10000 + 100000000 * 10/10000 = 5,100,000` royalty) [4](#0-3) 

The test was written to match the implementation rather than the correct behavior, confirming the bug is present and undetected. The `parseCommandDisplay` function is called directly in the Electron main process approval flow, making this discrepancy visible to users in every multi-NFT WalletConnect offer approval. [5](#0-4)

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
