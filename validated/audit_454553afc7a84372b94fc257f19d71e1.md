### Title
Malicious Participant Can Inject Arbitrary Share Values in CKD Protocol Without Detection — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates per-participant ElGamal share contributions `(norm_big_y, norm_big_c)` with no cryptographic proof that each contribution was honestly computed from the participant's actual key share. Any single malicious participant can send arbitrary group elements to the coordinator, silently corrupting the final confidential key derivation output. This is the threshold-signatures analog of the clickjacking report's root cause: a missing security control (X-Frame-Options header / cryptographic proof-of-correct-computation) that allows an attacker to inject unauthorized values into a privileged operation performed on behalf of honest parties.

---

### Finding Description

In `do_ckd_coordinator` the coordinator collects one `CKDOutput` from every other participant via `recv_from_others` and unconditionally adds each received `(big_y, big_c)` pair into its running sum:

```rust

``` [1](#0-0) 

Each non-coordinator participant computes its share in `compute_signature_share` and sends it **privately** to the coordinator only:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [2](#0-1) 

The coordinator performs **no verification** that the received `(norm_big_y, norm_big_c)` pair is consistent with the sender's public key share or with the agreed `app_id` / `app_pk`. There is no zero-knowledge proof of correct encryption, no commitment-then-reveal, and no echo-broadcast consistency check — unlike the DKG protocol which uses proofs of knowledge and commitment hashes: [3](#0-2) 

The correct CKD output satisfies:

```
big_C_total - app_sk * big_Y_total  =  msk * H(pk || app_id)
```

where `msk` is the threshold-reconstructed master secret. A malicious participant who sends `(norm_big_y + Δ_Y, norm_big_c + Δ_C)` for arbitrary group elements `Δ_Y`, `Δ_C` shifts the coordinator's final sum by `(Δ_Y, Δ_C)`, producing:

```
(big_C_total + Δ_C) - app_sk * (big_Y_total + Δ_Y)
  = msk * H(pk || app_id)  +  Δ_C - app_sk * Δ_Y
```

which is an attacker-controlled point, not the correct confidential key.

Because the shares are sent privately to the coordinator, the other honest participants never see what the malicious participant sent and cannot raise an alarm. The `recv_from_others` helper only enforces that exactly one message per participant is accepted (deduplication), not that the content is valid: [4](#0-3) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The coordinator outputs a `CKDOutput` whose `unmask(app_sk)` value is not `msk * H(pk || app_id)` but an attacker-chosen point. Any TEE application that consumes this output derives a wrong deterministic secret. The master key is never reconstructed, so the attack cannot be detected by comparing against a known value. All honest participants return `None` (they never see the coordinator's output), so no cross-check is possible within the protocol. [5](#0-4) 

---

### Likelihood Explanation

**High.** The attacker only needs to be one of the `N` participants in a CKD session — a role that is reachable without any privileged key material beyond holding a valid key share. The attack requires sending a single crafted private message to the coordinator (one network message). No social engineering, timing, or external oracle is needed. The malicious deviation is completely undetectable by honest participants because the shares are private and no aggregate consistency check exists.

---

### Recommendation

Add a non-interactive proof of correct ElGamal encryption for each participant's contribution before the coordinator aggregates it. Concretely, each participant should prove in zero knowledge that `(norm_big_y, norm_big_c)` was computed as `(λ_i · y_i · G, λ_i · (x_i · H(pk||app_id) + y_i · app_pk))` for the committed `x_i` (their public key share) and a fresh `y_i`. A standard Chaum-Pedersen / sigma-protocol proof of discrete-log equality suffices and is consistent with the proof-of-knowledge pattern already used in the DKG: [6](#0-5) 

Alternatively, publish all shares via echo-broadcast so every honest participant can independently verify the aggregate before the coordinator finalises the output.

---

### Proof of Concept

1. Run a CKD session with participants `[P1, P2, P3]` and coordinator `P1`. `P2` is malicious.
2. `P2` computes its honest share `(norm_big_y_2, norm_big_c_2)` via `compute_signature_share`.
3. Instead of sending the honest share, `P2` sends `(norm_big_y_2 + Δ_Y, norm_big_c_2 + Δ_C)` to `P1` for arbitrary non-zero `Δ_Y, Δ_C ∈ G1`.
4. `P1` (coordinator) executes `do_ckd_coordinator`, receives the tampered pair from `P2`, and adds it to the running sum without any check.
5. The coordinator outputs `CKDOutput { big_y: Y_total + Δ_Y, big_c: C_total + Δ_C }`.
6. The TEE application calls `ckd_output.unmask(app_sk)` and obtains `msk * H(pk||app_id) + Δ_C - app_sk * Δ_Y` — an attacker-controlled wrong key — instead of the correct confidential key.
7. `P3` (honest non-coordinator) returns `None` and has no visibility into the corruption.

The entry path is entirely within `do_ckd_coordinator` / `do_ckd_participant` in `src/confidential_key_derivation/protocol.rs` with no external assumptions required. [7](#0-6)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
