### Title
Unprivileged Caller Can Permanently Suppress Consensus Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in `staking.cairo` is callable by any non-zero address. It unconditionally writes the global `last_reward_block` storage variable **before** checking the `disable_rewards` flag. An attacker can call `update_rewards(any_active_staker, true)` every block, consuming each block's reward slot without distributing any rewards, permanently suppressing consensus rewards for all stakers.

---

### Finding Description

`update_rewards` is gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero — no role-based access control exists. [1](#0-0) 

The function writes `last_reward_block` to the current block number **unconditionally**, before the `disable_rewards` branch: [2](#0-1) 

`last_reward_block` is a global singleton — not per-staker: [3](#0-2) 

Because the guard at the top of the function enforces `current_block_number > last_reward_block`: [4](#0-3) 

…any call to `update_rewards` in a given block — regardless of `disable_rewards` — permanently consumes that block's reward slot. An attacker who calls `update_rewards(any_active_staker, true)` in every block will:

1. Mark each block as "processed" (writing `last_reward_block`)
2. Return early without distributing rewards
3. Block any legitimate call to `update_rewards` in that block

The staker address passed must be active with non-zero STRK balance, but any such address is publicly visible on-chain.

---

### Impact Explanation

All stakers lose consensus rewards for every block the attacker calls this. If called every block, this is **permanent freezing of unclaimed yield** (High). Even sporadic calls cause **temporary freezing of unclaimed yield** (High). The attacker has no profit motive — this is pure griefing.

---

### Likelihood Explanation

Any unprivileged address can execute this attack. The only cost is gas per block. The attack requires no special knowledge beyond identifying any active staker address (public on-chain data). The attack is trivially automatable.

---

### Recommendation

Add role-based access control to `update_rewards` so only an authorized caller (e.g., the consensus mechanism or a designated `REWARDS_MANAGER` role) can invoke it. The `disable_rewards` flag in particular must not be settable by an arbitrary caller. Alternatively, separate the "mark block processed" write from the "distribute rewards" logic so that `last_reward_block` is only advanced when rewards are actually distributed.

---

### Proof of Concept

1. Attacker identifies any active staker address `S` with non-zero STRK balance at the current epoch.
2. Each block, attacker calls `update_rewards(S, disable_rewards: true)`.
3. `last_reward_block` is updated to the current block number.
4. The function hits `if disable_rewards || self.is_pre_consensus() { return; }` and exits without distributing rewards.
5. Any legitimate call to `update_rewards` in the same block fails with `REWARDS_ALREADY_UPDATED`.
6. All stakers receive zero consensus rewards indefinitely.

The state-transition bug is structurally identical to the reported analog: `last_reward_block` is advanced (state committed) without validating that the transition is legitimate (i.e., that rewards were actually distributed), and there is no guard preventing an unprivileged caller from triggering the transition with `disable_rewards: true`. [5](#0-4)
