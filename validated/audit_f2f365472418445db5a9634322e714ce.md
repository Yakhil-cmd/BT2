### Title
Missing Validation of Participant Signature Shares in CKD Coordinator Allows Output Corruption — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator aggregates participant contributions `(norm_big_y, norm_big_c)` without any proof of correctness. Just as `ssh.InsecureIgnoreHostKey()` accepts any server identity without validation, the coordinator blindly accepts any group elements from participants. A single malicious participant can send arbitrary values to corrupt the final `CKDOutput`, causing the application to derive an incorrect confidential key.

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and unconditionally adds them to the running sum: [1](#0-0) 

There is no ZK proof, commitment binding, or consistency check attached to these values. The protocol specification requires each participant to compute:

- `norm_big_y = λ_i · y_i · G₁`
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)`

but nothing in the code enforces this. A malicious participant `j` can instead send any arbitrary `(Y', C')` ∈ G₁ × G₁.

The coordinator then constructs the final output: [2](#0-1) 

The application later calls `ckd_output.unmask(app_sk)` to recover the confidential key `C - a·Y`. With a corrupted `(Y, C)`, the unmasked result is `(C_honest + C') - a·(Y_honest + Y')`, which is an incorrect value unrelated to `msk · H(pk, app_id)`.

The honest participants' `compute_signature_share` path is correct: [3](#0-2) 

but the coordinator never verifies that received values were produced by this computation.

### Impact Explanation

**High — Corruption of CKD output.** A single malicious participant (any non-coordinator party) can send arbitrary `(norm_big_y, norm_big_c)` values. The coordinator outputs a `CKDOutput` that, when unmasked by the application, yields an incorrect confidential derived key. All honest parties relying on the coordinator's output accept a corrupted result. The malicious participant cannot target a specific output value (they do not know the app's secret key `a`), so the impact is corruption/unusability rather than full secret extraction.

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. No special privilege is required — the attacker only needs to be a valid member of the participant list, which is the normal unprivileged attacker model for this library. The attack requires sending two arbitrary group elements in place of the honest contribution, which is trivially achievable by any participant controlling their own protocol instance.

### Recommendation

Attach a non-interactive ZK proof to each participant's contribution proving that `norm_big_c` was formed correctly relative to `norm_big_y` and the participant's public key share. Concretely, a DLOG-equality proof (already implemented in `src/crypto/proofs/dlogeq.rs`) can prove that the same scalar `y_i` was used in both `Y_i = y_i · G₁` and the blinding term `y_i · A` inside `C_i`, and a separate DLOG proof can bind `C_i - λ_i · x_i · H(pk, app_id)` to `A`. The coordinator must verify these proofs before aggregating. [4](#0-3) 

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` run `compute_signature_share` correctly.
2. Malicious participant `P_n` (non-coordinator) replaces its output with `(norm_big_y' = G₁, norm_big_c' = G₁)` — two arbitrary non-zero group elements.
3. `do_ckd_participant` sends `(G₁, G₁)` privately to the coordinator.
4. The coordinator's loop at lines 50–55 adds `G₁` to both accumulators.
5. The final `CKDOutput` is `(Y_honest + G₁, C_honest + G₁)`.
6. The application calls `unmask(app_sk)`: result is `(C_honest + G₁) - a·(Y_honest + G₁) = correct_key + G₁ - a·G₁`, which is not the correct confidential key.
7. The application silently accepts this incorrect value — there is no integrity check on the unmasked output.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))

pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
