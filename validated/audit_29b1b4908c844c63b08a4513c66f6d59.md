### Title
`broadcast_success` Never Broadcasts Failure, Enabling a Single Malicious Participant to Permanently Hang DKG/Reshare/Refresh — (File: `src/dkg.rs`)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` always broadcasts `(true, session_id)` and never broadcasts a failure signal. Its own doc-comment describes an `err`-parameterized design that was never implemented. When a participant fails share validation and aborts before reaching `broadcast_success`, it never sends a SEND message in the echo-broadcast round. The echo-broadcast implementation has no mechanism to detect a missing SEND and loops indefinitely, permanently hanging every other honest participant in the DKG, reshare, or refresh protocol.

---

### Finding Description

**Root cause — `broadcast_success` always broadcasts success:** [1](#0-0) 

The doc-comment explicitly states the function should accept an `err` input and broadcast failure when it is non-`None`. The actual signature takes no such parameter and unconditionally broadcasts `(true, session_id)`: [2](#0-1) 

**Consequence — a participant that fails validation never broadcasts:**

`broadcast_success` is the last step of `do_keyshare`. If a participant fails at share-validation step 5.2 (`validate_received_share`), the function returns an error and `broadcast_success` is never called: [3](#0-2) 

That participant therefore never sends a SEND message for its own echo-broadcast session.

**Echo-broadcast has no mechanism to detect a missing SEND:**

`reliable_broadcast_receive_all` only terminates when every participant's session reaches `finish_ready`: [4](#0-3) 

The only early-abort path fires when processing an ECHO message and the remaining potential echo count cannot reach the threshold: [5](#0-4) 

If the sender never sends a SEND message, no ECHO messages are ever generated for that session, so `received_echo_cnt = 0` and `non_received_echo_cnt = n`. The guard evaluates `0 + n > echo_t` = `n > (n+f)/2`, which is always true (since `n > f`). `is_enough` is set to `true`, the early-abort is skipped, and the loop waits forever for ECHO messages that will never arrive.

---

### Impact Explanation

Every honest participant that successfully validates its shares calls `broadcast_success` and enters the echo-broadcast loop. Because the failed participant's session never reaches `finish_ready`, the loop condition `state.iter().all(|x| x.finish_ready)` is permanently false. All honest participants hang indefinitely. DKG, reshare, and refresh — which all route through `do_keyshare` — are permanently denied for the entire participant set.

/// This function takes err as input.
/// If err is None then broadcast success
/// otherwise, broadcast failure
/// If during broadcast it receives an error then propagates it
/// This function is used in the final round of DKG
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}
