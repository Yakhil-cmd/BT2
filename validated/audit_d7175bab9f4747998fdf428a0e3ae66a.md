### Title
`RewardSupplier.claim_rewards` Transfers Against Accounting Balance That Includes In-Flight L1 Tokens, Causing Reward Claim Failures - (File: `src/reward_supplier/reward_supplier.cairo`)

### Summary

`RewardSupplier.claim_rewards` validates the requested amount against `unclaimed_rewards` (an accounting ledger), but the actual ERC-20 transfer requires tokens to be physically present in the contract. Because `request_funds` satisfies its invariant using `balance + l1_pending_requested_amount` (credit includes in-flight L1 tokens), the real token balance can be far below `unclaimed_rewards` during the L1→L2 bridge delay window. Any reward distribution call that triggers `claim_from_reward_supplier` during this window reverts, temporarily freezing all unclaimed yield for stakers and pool members.

### Finding Description

`RewardSupplier` tracks two separate quantities:

- `unclaimed_rewards` — the total amount owed to the staking contract (accounting).
- `l1_pending_requested_amount` — tokens already requested from L1 but not yet bridged.

`request_funds` is called every time rewards are registered. Its invariant is:

```
credit = balance + l1_pending_requested_amount >= unclaimed_rewards + threshold
``` [1](#0-0) 

`threshold = base_mint_amount / 2`, so the invariant is satisfied even when `balance` is zero, as long as enough L1 requests are in flight. [2](#0-1) 

`claim_rewards` then checks only the accounting value:

```cairo
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
// ...
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [3](#0-2) 

There is no check that `balance >= amount`. If `balance < amount`, `checked_transfer` reverts.

The staking contract calls `update_unclaimed_rewards_from_staking_contract` and then immediately calls `claim_from_reward_supplier` in the same transaction inside `_update_rewards`:

```cairo
reward_supplier_dispatcher
    .update_unclaimed_rewards_from_staking_contract(
        rewards: staker_rewards + total_pools_rewards,
    );
// Claim pools rewards.
claim_from_reward_supplier(
    :reward_supplier_dispatcher,
    amount: total_pools_rewards,
    ...
);
``` [4](#0-3) 

`claim_from_reward_supplier` asserts the balance increased by exactly `amount`:

```cairo
assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
``` [5](#0-4) 

If the reward supplier's token balance is below `total_pools_rewards` at the moment of the call (because L1 tokens are still in transit), the entire transaction reverts.

The same failure path exists in `send_rewards_to_staker`, which calls `claim_from_reward_supplier` for the staker's own accumulated rewards: [6](#0-5) 

### Impact Explanation

Any call to `update_rewards` (consensus phase) or `update_rewards_from_attestation_contract` (pre-consensus phase) that distributes pool rewards will revert if the reward supplier's actual token balance is below the amount being claimed. Similarly, any staker calling `claim_rewards` on the staking contract will have their transaction revert. This constitutes **temporary freezing of unclaimed yield** for all stakers and pool members during the L1→L2 bridge delay window.

### Likelihood Explanation

The L1→L2 bridge delay on Starknet is typically hours to days. During this entire window, every reward distribution transaction fails. This is not a rare edge case — it occurs systematically whenever rewards accumulate faster than bridged tokens arrive, which is the normal operating condition at protocol launch and after any period of low L1 throughput. The `request_funds` invariant explicitly permits `balance < unclaimed_rewards` (it only requires `balance + l1_pending_requested_amount >= unclaimed_rewards + threshold`), so this state is expected and routine.

### Recommendation

In `claim_rewards`, add a check that the contract's actual token balance is sufficient before attempting the transfer:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    // ... existing caller check ...
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

    // Add: verify actual balance is sufficient
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
    assert!(amount <= balance, "{}", Error::INSUFFICIENT_BALANCE);

    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
}
```

Alternatively, `_update_rewards` in the staking contract should only call `claim_from_reward_supplier` after confirming the reward supplier holds sufficient tokens, or the protocol should ensure the reward supplier is pre-funded before any reward distribution begins.

### Proof of Concept

1. Deploy the system with `balance(reward_supplier) = 0` (no pre-funding).
2. A staker stakes and earns rewards. The attestation contract calls `update_rewards_from_attestation_contract(staker_address)`.
3. Inside `_update_rewards`, `update_unclaimed_rewards_from_staking_contract(staker_rewards + pool_rewards)` is called. `unclaimed_rewards` increases; `request_funds` sends an L1 mint request and sets `l1_pending_requested_amount = base_mint_amount`. The invariant `0 + base_mint_amount >= (staker_rewards + pool_rewards) + threshold` is satisfied.
4. Immediately after, `claim_from_reward_supplier(pool_rewards)` is called. `assert!(pool_rewards <= unclaimed_rewards)` passes. `checked_transfer(pool_rewards)` reverts because `balance(reward_supplier) = 0 < pool_rewards`.
5. The entire transaction reverts. Pool members cannot receive their rewards. The same failure occurs for `claim_rewards` on the staking contract until L1 tokens arrive.
