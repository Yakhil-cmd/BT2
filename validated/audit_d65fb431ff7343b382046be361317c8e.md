### Title
`new_settlement_layer_chain_id_storage` Not Rolled Back on Frame Revert — Incomplete State Reset in `FullIO` - (`basic_system/src/system_implementation/system/io_subsystem.rs`)

### Summary

`FullIO::start_io_frame()` takes snapshots of all sub-storages except `new_settlement_layer_chain_id_storage`. Consequently, `finish_io_frame()` never rolls back that storage on frame revert. If any execution frame that triggers `update_settlement_layer_chain_id()` is later reverted, the "already-updated" guard inside `NewSettlementLayerChainIdStorage::update()` remains permanently armed for the rest of the block, making it impossible for any subsequent legitimate settlement-layer-chain-ID update to succeed in the same block.

### Finding Description

`FullIO` holds six sub-storages. `start_io_frame()` snapshots five of them and stores the handles in `FullIOStateSnapshot`:

```
io, transient, messages, events, interop_roots
```

`new_settlement_layer_chain_id_storage` is **absent** from the snapshot struct and from both `start_io_frame()` and `finish_io_frame()`. [1](#0-0) 

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    // new_settlement_layer_chain_id_storage — MISSING
}
```

`start_io_frame()` snapshots `interop_root_storage` but not `new_settlement_layer_chain_id_storage`: [2](#0-1) 

`finish_io_frame()` rolls back `interop_root_storage` but not `new_settlement_layer_chain_id_storage`: [3](#0-2) 

`NewSettlementLayerChainIdStorage` does have a `start_frame()` / `finish_frame()` / `rollback()` API — it is simply never called from the IO frame machinery: [4](#0-3) 

The guard inside `update()` checks `self.value().is_some()` and returns an `InternalError` if the value is already set: [5](#0-4) 

Because `begin_next_tx()` also does not reset `new_settlement_layer_chain_id_storage`, a value written in a reverted frame persists for the entire block: [6](#0-5) 

### Impact Explanation

If any execution frame that calls `update_settlement_layer_chain_id()` is subsequently rolled back (e.g., out-of-gas after the event hook fires, or an explicit revert), the `new_settlement_layer_chain_id_storage` retains the written value. For the remainder of the block:

1. The "only one update per block" guard fires as an `InternalError` for every subsequent attempt to update the settlement-layer chain ID.
2. A legitimate service transaction that must update the SL chain ID in the same block will fail with an internal error, which propagates as a fatal block-level error.
3. The batch output's `settlement_layer_chain_id` field may be computed from stale or inconsistent state, producing a divergence between forward execution and proving. [7](#0-6) 

### Likelihood Explanation

The `SettlementLayerChainIdUpdated` event hook is triggered whenever the matching event signature is emitted during EVM execution: [8](#0-7) 

The hook calls `update_settlement_layer_chain_id()` before the frame outcome is known. If the emitting frame later reverts (e.g., runs out of gas after the event is emitted, or the callee reverts), the event is rolled back via `events_storage` but `new_settlement_layer_chain_id_storage` is not. Any contract that can emit the matching event signature and then revert can trigger this condition. The test suite confirms the guard fires on a second update but does not test the revert-then-update scenario: [9](#0-8) 

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and include it in both `start_io_frame()` and `finish_io_frame()`, mirroring the existing treatment of `interop_root_storage`:

```rust
// In FullIOStateSnapshot
new_sl_chain_id: NewSettlementLayerChainIdSnapshotId,

// In start_io_frame()
let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();

// In finish_io_frame()
self.new_settlement_layer_chain_id_storage
    .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
``` [10](#0-9) 

### Proof of Concept

1. Block contains two transactions: Tx1 (attacker) and Tx2 (legitimate SL chain ID update service tx).
2. Tx1 calls a contract that emits `SettlementLayerChainIdUpdated(uint256)` with any value, then reverts (e.g., via `REVERT` opcode after the event).
3. The event hook fires during Tx1 execution, calling `update_settlement_layer_chain_id()`, which sets `new_settlement_layer_chain_id_storage`.
4. Tx1's frame reverts: `events_storage` is rolled back (event removed), but `new_settlement_layer_chain_id_storage` is **not** rolled back.
5. Tx2 attempts to call `update_settlement_layer_chain_id()` — the guard `self.value().is_some()` is `true`, returning `InternalError("Tried to update settlement layer chain id more than once in a block")`.
6. This internal error propagates as a fatal block-level failure, preventing the legitimate SL chain ID update and potentially halting block production. [5](#0-4) [3](#0-2)

    fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
        let io = self.storage.start_frame();
        let transient = self.transient_storage.start_frame();
        let messages = self.logs_storage.start_frame();
        let events = self.events_storage.start_frame();
        let interop_roots = self.interop_root_storage.start_frame();

        Ok(FullIOStateSnapshot {
            io,
            transient,
            messages,
            events,
            interop_roots,
        })
    }

    fn finish_io_frame(
        &mut self,
        rollback_handle: Option<&Self::StateSnapshot>,
    ) -> Result<(), InternalError> {
        self.storage.finish_frame(rollback_handle.map(|x| &x.io))?;
        self.transient_storage
            .finish_frame(rollback_handle.map(|x| &x.transient))?;
        self.logs_storage
            .finish_frame(rollback_handle.map(|x| x.messages));
        self.events_storage
            .finish_frame(rollback_handle.map(|x| x.events));
        self.interop_root_storage
            .finish_frame(rollback_handle.map(|x| x.interop_roots));

        Ok(())
    }
