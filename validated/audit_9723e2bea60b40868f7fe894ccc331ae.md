### Title
TOCTOU: `prepareSubmitData` re-fetches pool info independently at submit time, allowing a malicious pool server to substitute `targetPuzzleHash` after user confirmation — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

`PlotNFTSelectBase` calls `usePoolInfo` to display pool details for user confirmation, and `prepareSubmitData` makes a **second, independent** call to `getPoolInfo` at submit time. A malicious pool operator can return legitimate data on the first request (display) and a different `targetPuzzleHash` on the second request (signing), causing `createNewPoolWallet` to sign a transaction with an attacker-controlled payout address while the user believed they confirmed a different one.

### Finding Description

**Display path** — `PlotNFTSelectBase` calls `usePoolInfo(poolUrl)` reactively: [1](#0-0) 

`usePoolInfo` calls `getPoolInfo(normalizedUrl)` and renders the result in `<PoolInfo poolInfo={poolInfo.poolInfo} />`: [2](#0-1) 

**Submit path** — when the user clicks Create, `handleSubmit` calls `prepareSubmitData(data)`: [3](#0-2) 

`prepareSubmitData` makes a **fresh, independent** call to `getPoolInfo`: [4](#0-3) 

The `targetPuzzleHash` and `relativeLockHeight` extracted from this second fetch are placed directly into `initialTargetState` and passed to `createNewPoolWallet`: [5](#0-4) 

There is **no mechanism** that binds the pool info fetched for display to the pool info used in signing. The two fetches are entirely independent HTTP requests to the attacker-controlled server.

### Impact Explanation

A malicious pool operator (or MITM on the pool URL) implements a server that:
- Returns legitimate `targetPuzzleHash` (e.g., a well-known pool's puzzle hash) on the first `/pool_info` request → displayed to the user in the "Verify Pool Details" step.
- Returns an attacker-controlled `targetPuzzleHash` on the second `/pool_info` request → used in the signed `createNewPoolWallet` transaction.

The user approves the transaction based on the displayed (legitimate) pool info, but the on-chain transaction encodes the attacker's payout address. All future farming rewards from that Plot NFT flow to the attacker's puzzle hash. The user has no way to detect this from the GUI at confirmation time.

### Likelihood Explanation

- The attacker only needs to operate (or MITM) a pool server — a realistic threat for any user connecting to an untrusted pool URL.
- The timing window is the entire duration the user spends reviewing the "Verify Pool Details" card before clicking Create (seconds to minutes), making the race trivially winnable.
- No local host compromise, leaked keys, or broken cryptography is required.

### Recommendation

Capture the pool info from the display fetch and pass it directly into `prepareSubmitData` rather than re-fetching. Concretely: expose the `poolInfo` value already held by `usePoolInfo` in `PlotNFTSelectBase` up to `PlotNFTSelectPool`, and use those cached values in `prepareSubmitData` instead of calling `getPoolInfo` again. This ensures the `targetPuzzleHash` signed in the transaction is identical to the one the user reviewed.

### Proof of Concept

1. Stand up a pool server at `http://localhost:8080`.
2. On the **first** `GET /pool_info` request, return a response with a legitimate-looking `target_puzzle_hash` (e.g., copied from a real pool).
3. On the **second** `GET /pool_info` request, return a response with an attacker-controlled `target_puzzle_hash`.
4. In the GUI, enter `http://localhost:8080` as the pool URL.
5. Observe the "Verify Pool Details" card renders the legitimate puzzle hash.
6. Click Create.
7. Inspect the `createNewPoolWallet` RPC call (via network tab or node logs) — `initial_target_state.target_puzzle_hash` will contain the attacker's value, not the one displayed.
