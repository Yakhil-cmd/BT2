### Title
Silo Address Whitelist Bypassed via `ft_on_transfer` When No Fallback Address Is Set — (`engine/src/engine.rs`)

### Summary

In Silo mode, the EVM address whitelist is enforced when users submit EVM transactions via `submit`, but is **completely skipped** when ERC-20 tokens are deposited into Aurora via `ft_on_transfer` → `receive_erc20_tokens` if no ERC-20 fallback address has been configured. A non-whitelisted EVM address can receive ERC-20 tokens minted into the EVM, but cannot submit any transaction to move them, permanently freezing those funds.

### Finding Description

The Silo whitelist check for receiving ERC-20 tokens is gated behind a `Some(fallback_address)` guard in `receive_erc20_tokens`:

```rust
// engine/src/engine.rs:818-822
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

When `get_erc20_fallback_address` returns `None` (i.e., no fallback address is configured), the entire block is skipped. The `recipient` is used as-is regardless of whether it is in the Address whitelist. ERC-20 tokens are then minted to the non-whitelisted address via the subsequent `self.call(...)` to the ERC-20 contract. [2](#0-1) 

By contrast, every EVM transaction submitted via `submit` / `submit_with_args` goes through `assert_access`, which enforces both the Account and Address whitelists before any execution:

```rust
// engine/src/engine.rs:1051-1052
// Check if the sender has rights to submit transactions or deploy code.
assert_access(&io, env, &transaction)?;
``` [3](#0-2)

### Impact Explanation

**Permanent freezing of funds.**

When a Silo operator enables the Address whitelist but does not configure a fallback address, any user can call `ft_transfer_call` on a NEP-141 token contract targeting Aurora with a non-whitelisted EVM address as the message recipient. Aurora's `ft_on_transfer` will mint ERC-20 tokens to that address inside the EVM. Because the address is not whitelisted, every subsequent `submit` call from that address is rejected by `assert_access` with `EngineErrorKind::NotAllowed`. The `ExitToNear` precompile is only reachable via a submitted EVM transaction, so the tokens cannot be bridged back out. The funds are permanently frozen inside the EVM with no recovery path.

### Likelihood Explanation

The vulnerable configuration — Silo mode active, Address whitelist enabled, no fallback address set — is a valid and documented operational state. `SiloParamsArgs` stores `fixed_gas` and `erc20_fallback_address` as a pair; an operator can set a fixed gas cost (activating Silo mode) without setting a fallback address. The attacker-controlled entry path is the standard NEP-141 `ft_transfer_call` interface, callable by any unprivileged NEAR account. No special permissions are required. [7](#0-6) 

### Recommendation

Remove the `Some(fallback_address)` guard from the whitelist check in `receive_erc20_tokens`. The whitelist enforcement should be unconditional when the Address whitelist is enabled:

```rust
// Proposed fix
if !silo::is_allow_receive_erc20_tokens(&self.io, &recipient) {
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io) {
        recipient = fallback_address;
    } else {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
}
```

This mirrors the pattern used in `assert_access` for `submit`: the whitelist check is always enforced, and the absence of a fallback address results in an explicit rejection rather than a silent bypass.

### Proof of Concept

1. Deploy Aurora in Silo mode: call `set_silo_params` with a non-zero `fixed_gas` and `erc20_fallback_address = Address::zero()`, then call `set_silo_params` again with only `fixed_gas` set (or use `set_fixed_gas` directly) so that no fallback address is stored. Enable the Address whitelist via `set_whitelist_status`.
2. Do **not** add victim's EVM address to the Address whitelist.
3. As any NEAR account, call `ft_transfer_call` on a NEP-141 token that has an ERC-20 mirror on Aurora, with `receiver_id = aurora` and `msg = <victim_evm_address_hex>`.
4. Aurora's `ft_on_transfer` fires → `receive_erc20_tokens` is called → the `if let Some(fallback_address)` guard evaluates to `false` (no fallback) → whitelist check is skipped → ERC-20 tokens are minted to the victim address.
5. Attempt to call `submit` from the victim address: rejected with `NotAllowed`.
6. Attempt to call `ExitToNear` precompile: requires a submitted EVM transaction → also rejected.
7. Tokens are permanently frozen at the non-whitelisted EVM address. [8](#0-7) [9](#0-8)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
