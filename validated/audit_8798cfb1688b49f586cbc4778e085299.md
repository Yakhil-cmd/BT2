### Title
TOCTOU Race in Pool Wallet Creation Allows Malicious Pool Server to Substitute `targetPuzzleHash` After User Review — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

The pool-joining flow fetches pool info **twice** from the external pool server with no validation that the two responses are identical. The first fetch populates the "Verify Pool Details" UI; the second fetch, triggered at form submit inside `prepareSubmitData`, supplies the `targetPuzzleHash` that is actually sent to `createNewPoolWallet`. A malicious pool operator can serve legitimate data on the first request and substitute their own `targetPuzzleHash` on the second, causing the plot NFT to be created with an attacker-controlled reward destination while the user reviewed entirely different pool parameters.

---

### Finding Description

**First fetch — display only (`usePoolInfo` hook):**

`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)`, which calls `getPoolInfo(normalizedUrl)` → `window.appAPI.fetchPoolInfo(poolUrl)`. The result is stored in React state via `useAsync` and rendered in the "Verify Pool Details" card. [1](#0-0) [2](#0-1) 

**Second fetch — used for actual wallet creation (`prepareSubmitData`):**

When the user clicks "Create", `handleSubmit` calls `prepareSubmitData(data)`, which independently calls `getPoolInfo(normalizedPoolUrl)` again. The `targetPuzzleHash` and `relativeLockHeight` returned by **this second call** are what get passed to `createNewPoolWallet` — the cached/displayed values from the first call are never consulted. [3](#0-2) [4](#0-3) 

**No cross-check exists.** There is no code anywhere that compares the `targetPuzzleHash` shown to the user with the one returned at submit time. [5](#0-4) 

---

### Impact Explanation

`targetPuzzleHash` is the on-chain puzzle hash embedded in the plot NFT. It is the address that receives block rewards when the NFT farms a block. If an attacker substitutes their own puzzle hash, **all pooled farming rewards from that NFT are permanently redirected to the attacker**. The user's plot NFT is created, plots are farmed, but every block reward goes to the attacker's wallet. This is a direct, irreversible financial loss fitting the Critical category: *payout change affecting pooled farming rewards*. [6](#0-5) 

---

### Likelihood Explanation

The attacker must operate (or MITM) the pool server. This is a realistic threat: a malicious pool operator can trivially detect the two sequential HTTP requests (display fetch vs. submit fetch) and serve different `target_puzzle_hash` values. No special timing precision is required — the second request always arrives shortly after the user clicks "Create", and the operator controls the server response unconditionally. The user has no way to detect the substitution because the UI only shows the first response.

---

### Recommendation

**Cache and reuse the first fetch result.** `prepareSubmitData` should accept the already-fetched `poolInfo` object (from `usePoolInfo`) as a parameter instead of re-fetching. If a re-fetch is considered necessary, the returned `targetPuzzleHash` must be compared against the displayed value and the submit must be aborted with an error if they differ.

Concretely:
- Pass the `poolInfo` state from `PlotNFTSelectBase`/`usePoolInfo` into `prepareSubmitData` and use it directly.
- Remove the independent `getPoolInfo` call inside `prepareSubmitData`.
- If a freshness re-fetch is desired, add an explicit equality check: `if (freshInfo.targetPuzzleHash !== displayedInfo.targetPuzzleHash) throw new Error(...)`.

---

### Proof of Concept

```
1. Stand up a pool server at https://evil-pool.example.
2. On the first GET /pool_info request, return:
     { "target_puzzle_hash": "0xLEGIT...", "relative_lock_height": 32, ... }
3. On the second GET /pool_info request (arrives ~1–2 s later), return:
     { "target_puzzle_hash": "0xATTACKER_CONTROLLED...", "relative_lock_height": 32, ... }
4. User enters https://evil-pool.example in the pool URL field.
5. "Verify Pool Details" card renders with 0xLEGIT... — user approves.
6. User clicks "Create".
7. prepareSubmitData fires, calls getPoolInfo a second time, receives 0xATTACKER_CONTROLLED...
8. createNewPoolWallet is called with initialTargetState.targetPuzzleHash = "0xATTACKER_CONTROLLED..."
9. The plot NFT is created on-chain with the attacker's puzzle hash.
   All future block rewards from this NFT go to the attacker.

Assert: the targetPuzzleHash in the createNewPoolWallet RPC ≠ the targetPuzzleHash shown in the UI.
```

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
