### Title
Missing Zero Address Check in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Both `staking.cairo::change_reward_address` and `pool.cairo::change_reward_address` accept a `reward_address` parameter without validating it against the zero address. A staker or pool member can set their reward address to `0`, causing all subsequent reward transfers to revert, permanently freezing their unclaimed yield.

### Finding Description
`staking.cairo::change_reward_address` (line 517) performs only one validation — that the new address is not a registered token address — but does not check for the zero address:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;   // zero accepted here
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

The same omission exists in `pool.cairo::change_reward_address`:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
    pool_member_info.reward_address = reward_address;   // zero accepted here
    self.write_pool_member_info(:pool_member, :pool_member_info);
    ...
}
``` [2](#0-1) 

Additionally, the initial `stake()` call in `staking.cairo` also accepts `reward_address` without a zero check: [3](#0-2) 

### Impact Explanation
When `reward_address` is set to zero, every downstream reward transfer reverts:

1. **`claim_rewards` path (staking):** `send_rewards_to_staker` calls `token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into())`. If `reward_address` is zero and `amount > 0`, the ERC-20 transfer reverts, making unclaimed yield permanently unclaimable. [4](#0-3) 

2. **`unstake_action` path:** `unstake_action` unconditionally calls `send_rewards_to_staker` before returning the principal. If the reward transfer reverts, the entire `unstake_action` reverts, freezing the staker's principal stake as well. [5](#0-4) 

3. **Pool `claim_rewards` path:** `pool.cairo::claim_rewards` transfers directly to `reward_address`; a zero address causes the same revert. [6](#0-5) 

Impact: **High — Permanent freezing of unclaimed yield; Temporary freezing of staker principal** (until the user corrects the reward address via another `change_reward_address` call, but if the user is unaware, the freeze is effectively permanent).

### Likelihood Explanation
Any staker or pool member can trigger this by calling `change_reward_address(0)` — no privileged role is required. The call is permissionless and reachable from any externally-owned account. Accidental zero-address submission (e.g., a front-end bug, copy-paste error, or scripting mistake) is a realistic scenario.

### Recommendation
Add a zero address guard at the top of both `change_reward_address` functions and in the `stake()` constructor path:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply the same guard to the `reward_address` parameter in `stake()` in `staking.cairo` and `enter_delegation_pool` in `pool.cairo`.

### Proof of Concept

1. Staker calls `staking.change_reward_address(reward_address: 0)` — succeeds with no revert.
2. Protocol accrues rewards for the staker over subsequent epochs.
3. Staker calls `claim_rewards(staker_address)` — `send_rewards_to_staker` attempts `checked_transfer(recipient: 0, amount: rewards)` → ERC-20 reverts with "transfer to zero address".
4. Staker calls `unstake_intent()` then waits the exit window, then calls `unstake_action(staker_address)` — same revert path blocks principal withdrawal.
5. Unclaimed yield and principal are frozen until the staker calls `change_reward_address` with a valid address — if the staker is unaware of the root cause, the freeze is indefinite.
