### Title
Malicious Participant Can Corrupt CKD Output via Unverified Accumulated Share Contributions — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator accumulates `CKDOutput` contributions (`norm_big_y`, `norm_big_c`) from every participant by simple addition, with no cryptographic verification that each contribution was honestly computed. A single malicious participant can inject arbitrary group elements, permanently corrupting the aggregate CKD output that all honest parties accept.

---

### Finding Description

The CKD protocol coordinator aggregates participant shares in `do_ckd_coordinator`: [1](#0-0) 

```rust
let (mut norm_big_y, mut norm_big_c) =
    compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

let waitpoint = chan.next_waitpoint();

for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();   // no verification
    norm_big_c += participant_output.big_c();   // no verification
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant is supposed to send:

- `norm_big_y = λᵢ · yᵢ · G`
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)`

as computed in `compute_signature_share`: [2](#0-1) 

The coordinator performs no check that the received `(big_y, big_c)` pair satisfies any algebraic relation to the participant's committed public key share or to the `app_pk`. The participant side (`do_ckd_participant`) simply sends the values and returns `None`—there is no broadcast, no echo, and no proof-of-correctness step: [3](#0-2) 

**Analog to the dividend bug:** In the external report, `withdrawnDividendPerOwner` is accumulated (only ever increases) without the required compensating reduction when tokens change hands, so the mapping drifts from the true state. Here, `norm_big_y` and `norm_big_c` are accumulated by simple addition without the required compensating verification that each addend is honestly formed. In both cases, an actor injects a value into an accumulator that the protocol trusts unconditionally, causing the final aggregate to diverge from the cryptographically correct value.

---

### Impact Explanation

The final `CKDOutput` is the only output of the protocol. It encodes the ElGamal ciphertext `(Y, C)` from which the application recovers the confidential derived key via `unmask(app_sk)`: [4](#0-3) 

If any single participant substitutes an arbitrary group element for their honest contribution, the aggregate `(Y, C)` is irrecoverably wrong. `unmask` will produce a value unrelated to `msk · H(pk ‖ app_id)`. The honest coordinator and all downstream consumers accept this corrupted output with no indication of failure. This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The attack requires only that one participant in the CKD session be malicious. The malicious participant is a documented trust boundary (the protocol is a multi-party protocol, not a single-trusted-party protocol). No special privilege, leaked key, or external assumption is needed—the attacker simply deviates from the protocol when calling `chan.send_private`. The entry path is fully reachable by any participant who is admitted to the session.

---

### Recommendation

Add a zero-knowledge proof of correct share formation alongside each `(norm_big_y, norm_big_c)` message. Concretely, each participant should prove in zero knowledge that:

1. `norm_big_y = λᵢ · y · G` for some scalar `y` they know, and
2. `norm_big_c = λᵢ · (x_i · H(pk ‖ app_id) + y · app_pk)` where `x_i` is consistent with the participant's committed public key share from DKG.

A standard sigma-protocol (DLEQ proof) over the pair `(norm_big_y, norm_big_c - λᵢ · x_i · H(pk ‖ app_id))` with base points `(G, app_pk)` suffices to bind the ephemeral `y` across both components. The coordinator must verify all proofs before accumulating any contribution.

---

### Proof of Concept

```
// Malicious participant P_evil deviates from do_ckd_participant:
// Instead of computing compute_signature_share(...), it sends:
let evil_big_y = ElementG1::identity();   // or any arbitrary point
let evil_big_c = ElementG1::identity();   // or any arbitrary point
chan.send_private(waitpoint, coordinator, &(evil_big_y, evil_big_c))?;

// Coordinator in do_ckd_coordinator:
// norm_big_y += evil_big_y  →  aggregate Y is wrong
// norm_big_c += evil_big_c  →  aggregate C is wrong
// CKDOutput::new(norm_big_y, norm_big_c) is accepted as valid

// Application calls:
let confidential_key = ckd_output.unmask(app_sk);
// confidential_key ≠ msk · H(pk ‖ app_id)
// The derived key is garbage; all TEE operations using it fail silently.
```

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
```
