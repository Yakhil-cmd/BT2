### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over all balance-change checkpoints in a pool member's `pool_member_epoch_balance` trace since their last reward claim. A delegator who makes balance changes across many epochs without claiming rewards accumulates a trace large enough to cause `claim_rewards` to exceed Starknet's execution resource limits, permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` (pool.cairo:837–888) iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to `pool_member_trace_length`: [1](#0-0) 

The code itself acknowledges the risk:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits. [2](#0-1) 

A new checkpoint is appended to the trace (one per epoch) every time a pool member changes their balance via `add_to_delegation_pool`, `exit_delegation_pool_intent`, or `enter_delegation_pool_from_staking_contract`. The `insert` function only creates a new entry when the epoch key changes; same-epoch updates overwrite the last entry: [3](#0-2) 

The `entry_to_claim_from` pointer — which determines where the loop starts — is stored in `pool_member_info` and is **only advanced on a successful `claim_rewards` call**: [4](#0-3) 

If a pool member makes one balance change per epoch across N epochs without ever calling `claim_rewards`, the trace accumulates N entries. The next `claim_rewards` call must iterate all N entries. If N is large enough to exceed Starknet's execution resource cap, the call reverts — and because `entry_to_claim_from` is never updated on a failed call, every subsequent `claim_rewards` call also reverts.

`claim_rewards` is the primary state-changing caller of `calculate_rewards`: [5](#0-4) 

---

### Impact Explanation

`exit_delegation_pool_action` does **not** call `calculate_rewards`, so the pool member can still recover their principal. However, all accumulated unclaimed yield becomes permanently unclaimable once the loop exceeds execution resource limits. This maps directly to the allowed impact: **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation

This is triggered by ordinary delegator behavior: a long-term passive delegator who periodically adjusts their delegation amount (top-ups, partial exits) across many epochs without claiming rewards. No adversarial intent is required. As the protocol matures and epoch counts grow, the probability of hitting this condition increases. The minimum stake requirement does not limit how many balance changes a single pool member can make over time.

---

### Recommendation

1. **Auto-claim on balance change**: Force a reward claim (or checkpoint update) whenever `set_member_balance` / `increase_member_balance` is called, so `entry_to_claim_from` is always current and the loop length is bounded to at most K entries.
2. **Partial claim**: Allow `claim_rewards` to accept a `max_iterations` parameter and process only a bounded slice of the trace per call, advancing `entry_to_claim_from` incrementally.
3. **Enforce claim before balance change**: Revert any balance-changing operation if the unclaimed trace length exceeds a safe threshold, forcing the pool member to claim first.

---

### Proof of Concept

```
1. Pool member calls enter_delegation_pool (epoch E₀).
2. Each subsequent epoch Eᵢ, pool member calls add_to_delegation_pool
   with a minimal amount — one new checkpoint appended per epoch.
3. Pool member never calls claim_rewards.
4. After N epochs, pool_member_epoch_balance trace has N entries;
   entry_to_claim_from == 0.
5. Pool member calls claim_rewards.
6. calculate_rewards loop iterates N times (each iteration reads storage
   and calls find_sigma), consuming O(N) execution resources.
7. For sufficiently large N, the transaction exceeds Starknet's resource
   limit and reverts.
8. entry_to_claim_from remains 0; every future claim_rewards call
   also reverts. Unclaimed yield is permanently frozen.
``` [6](#0-5)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
