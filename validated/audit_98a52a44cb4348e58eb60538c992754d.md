### Title
Non-Consensus Nodes Silently Start with Unencrypted Secrets Database When Encryption Key Is Absent - (`File: cmd/scaffold.go`)

---

### Summary

In `cmd/scaffold.go`, the `initSecretsDB()` function enforces secrets database encryption only for consensus nodes. For all other node roles (collection, execution, verification, access), a missing encryption key file causes the node to silently fall back to starting with a completely unencrypted secrets database. This means the node's private staking key, networking key, and any other secrets stored in the database are written to disk in plaintext, accessible to anyone with filesystem access to the node's storage directory.

---

### Finding Description

The `initSecretsDB()` function in `cmd/scaffold.go` contains asymmetric enforcement of secrets database encryption:

```go
encryptionKey, err := loadSecretsEncryptionKey(fnb.BootstrapDir, fnb.NodeID)
if errors.Is(err, os.ErrNotExist) {
    if fnb.NodeRole == flow.RoleConsensus.String() {
        // missing key is a fatal error for SN nodes
        return fmt.Errorf("secrets db encryption key not found: %w", err)
    }
    fnb.Logger.Warn().Msg("starting with secrets database encryption disabled")
    // node continues to start with NO encryption
}
``` [1](#0-0) 

For consensus nodes, a missing `secretsdb-key` file is a fatal startup error. For every other node role, the node logs a single warning and proceeds to open the Badger secrets database **without any encryption key**, meaning all data written to it is stored in plaintext on disk.

The secrets database is explicitly designed to hold confidential material. The `RecoverablePrivateBeaconKeyStateMachine` constructor enforces that it only operates on a secrets-typed database: [2](#0-1) 

The low-level operations themselves carry explicit cautions: [3](#0-2) 

The secrets database stores the node's staking private key and networking private key, loaded from the bootstrap directory: [4](#0-3) 

The encryption key file path is `private-root-information/private-node-info_<NodeID>/secretsdb-key`: [5](#0-4) 

The `--insecure-secrets-db` flag is documented as a consensus-node-only escape hatch, yet the silent fallback for non-consensus nodes is unconditional and requires no explicit opt-in: [6](#0-5) 

---

### Impact Explanation

Any party with read access to the node's secrets database directory (e.g., via a filesystem backup, cloud storage snapshot, container image export, or compromised host) can read the node's private staking key and networking private key in plaintext. The staking private key is used to sign protocol messages and proves node identity on the Flow network. Compromise of this key allows an attacker to impersonate the node, sign fraudulent protocol messages, or extract the key for use in other attacks. This is a direct analog to the iOS keychain items being backed up without `ThisDeviceOnly` protection: the sensitive material is stored without the intended at-rest encryption layer.

---

### Likelihood Explanation

The encryption key file (`secretsdb-key`) is a separate bootstrap artifact that must be manually provisioned. Operators who follow incomplete deployment guides, restore from partial backups, or run non-consensus nodes in environments where the bootstrap directory is partially populated will silently operate with unencrypted secrets databases. The node emits only a `Warn`-level log message, which is easily missed in production log pipelines. There is no runtime indicator that encryption is disabled. Cloud-hosted nodes routinely have their data volumes snapshotted by infrastructure providers, making the unencrypted database accessible to the cloud provider and any attacker who obtains a snapshot.

---

### Recommendation

**Short term:** Treat a missing secrets database encryption key as a fatal startup error for **all** node roles that use the secrets database, not only consensus nodes. Remove the role-conditional fallback at lines 1156–1161 of `cmd/scaffold.go`.

**Long term:** Empirically validate that the secrets database is encrypted at rest for all node roles in integration tests. Add a startup self-check that reads back a known entry and verifies the database was opened with an encryption key. Consider making `--insecure-secrets-db` the universal explicit opt-in for all roles rather than an implicit default for non-consensus nodes.

---

### Proof of Concept

1. Bootstrap a collection, execution, verification, or access node normally (generating all key files).
2. Delete the `secretsdb-key` file from the bootstrap directory.
3. Start the node. Observe that it starts successfully with only a `WARN` log: `"starting with secrets database encryption disabled"`.
4. After the node has run and written data, inspect the Badger database files in the secrets directory using any Badger reader tool. The staking private key and networking private key are readable in plaintext without any decryption step.

The relevant code path is `initSecretsDB()` in `cmd/scaffold.go` lines 1154–1166, where the `os.ErrNotExist` branch for non-consensus nodes calls `bstorage.InitSecret(opts)` with `opts` that has no `WithEncryptionKey` applied. [7](#0-6)

func (fnb *FlowNodeBuilder) initSecretsDB() error {

	// if the secrets DB is disabled (only applicable for Consensus Follower,
	// which makes use of this same logic), skip this initialization
	if !fnb.BaseConfig.secretsDBEnabled {
		return nil
	}

	if fnb.BaseConfig.secretsdir == NotSet {
		return fmt.Errorf("missing required flag '--secretsdir'")
	}

	err := os.MkdirAll(fnb.BaseConfig.secretsdir, 0700)
	if err != nil {
		return fmt.Errorf("could not create secrets db dir (path: %s): %w", fnb.BaseConfig.secretsdir, err)
	}

	opts := badger.DefaultOptions(fnb.BaseConfig.secretsdir).
		WithLogger(sutil.NewLogger(
			fnb.Logger.With().Str("badgerdb", "secret").Logger()))

	// NOTE: SN nodes need to explicitly set --insecure-secrets-db to true in order to
	// disable secrets database encryption
	if fnb.NodeRole == flow.RoleConsensus.String() && fnb.InsecureSecretsDB {
		fnb.Logger.Warn().Msg("starting with secrets database encryption disabled")
	} else {
		encryptionKey, err := loadSecretsEncryptionKey(fnb.BootstrapDir, fnb.NodeID)
		if errors.Is(err, os.ErrNotExist) {
			if fnb.NodeRole == flow.RoleConsensus.String() {
				// missing key is a fatal error for SN nodes
				return fmt.Errorf("secrets db encryption key not found: %w", err)
			}
			fnb.Logger.Warn().Msg("starting with secrets database encryption disabled")
		} else if err != nil {
			return fmt.Errorf("failed to read secrets db encryption key: %w", err)
		} else {
			opts = opts.WithEncryptionKey(encryptionKey)
		}
	}

	secretsDB, err := bstorage.InitSecret(opts)
	if err != nil {
		return fmt.Errorf("could not open secrets db: %w", err)
	}
	fnb.SecretsDB = secretsDB

	fnb.ShutdownFunc(func() error {
		if err := fnb.SecretsDB.Close(); err != nil {
			return fmt.Errorf("error closing secrets database: %w", err)
		}
		return nil
	})

	return nil
}
```
