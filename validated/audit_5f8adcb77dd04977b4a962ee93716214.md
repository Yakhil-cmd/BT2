### Title
Griefing via `declare_operational_address` front-running permanently blocks a staker from changing their operational address - (File: src/staking/staking.cairo)

---

### Summary
The two-step operational address change flow (`declare_operational_address` → `change_operational_address`) is vulnerable to a TOCTOU front-running attack. Because `declare_operational_address` freely overwrites the `eligible_operational_addresses` mapping with no commitment window, an attacker who controls the target operational address can repeatedly invalidate a staker's pending `change_operational_address` transaction, permanently preventing the staker from adopting that address.

---

### Finding Description
Changing an operational address requires two transactions:

1. The new operational address calls `declare_operational_address(staker_address)`, which writes `eligible_operational_addresses[caller] = staker_address`.
2. The staker calls `change_operational_address(operational_address)`, which asserts `eligible_operational_addresses[operational_address] == staker_address` before committing the change.

The `declare_operational_address` function has **no commitment window, no time-lock, and no restriction on overwriting an existing declaration**. Any caller (i.e., the operational address itself) can call it at any time to silently replace the stored `staker_address` with any other value, including `zero`.

```cairo
// src/staking/staking.cairo:705-722
fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
    self.general_prerequisites();
    let operational_address = get_caller_address();
    assert!(
        self.operational_address_to_staker_address.read(operational_address).is_zero(),
        "{}", Error::OPERATIONAL_IN_USE,
    );
    // ...
    // No lock, no delay — overwrites unconditionally:
    self.eligible_operational_addresses.write(operational_address, staker_address);
    self.emit(Events::OperationalAddressDeclared { operational_address, staker_address });
}
```

And the consuming check in `change_operational_address`:

```cairo
// src/staking/staking.cairo:683-687
assert!(
    self.eligible_operational_addresses.read(operational_address) == staker_address,
    "{}", Error::OPERATIONAL_NOT_ELIGIBLE,
);
```

**Attack sequence:**

1. Attacker controls `op_addr_X` (e.g., a malicious operational-address-as-a-service provider).
2. Attacker calls `declare_operational_address(victim_staker)` from `op_addr_X` — the mapping is set.
3. `victim_staker` observes the declaration on-chain and submits `change_operational_address(op_addr_X)`.
4. Attacker watches the mempool and front-runs with `declare_operational_address(zero)` from `op_addr_X`, overwriting the mapping.
5. `victim_staker`'s transaction executes and reverts with `OPERATIONAL_NOT_ELIGIBLE`.
6. The attacker can repeat step 4 indefinitely, permanently blocking the staker from adopting `op_addr_X`.

This is structurally identical to the Fraxlend M-11 pattern: a shared mutable slot (here `eligible_operational_addresses[op_addr]`) can be pre-occupied or invalidated by a party who controls the key, causing a legitimate user's dependent transaction to revert.

---

### Impact Explanation
A staker whose current operational address is compromised or needs rotation is permanently blocked from adopting the target address. Because attestation eligibility and reward updates flow through the operational address, a staker stuck with a compromised operational address risks missing attestation windows and losing unclaimed epoch rewards. The attacker incurs zero cost (no tokens required, just gas for the `declare_operational_address` call) and can sustain the grief indefinitely.

This maps to **Medium: Griefing with no profit motive but damage to users or protocol**.

---

### Likelihood Explanation
The attacker must control the target operational address. This is realistic in scenarios where:
- A validator outsources their operational key to a third-party infrastructure provider who turns malicious.
- A staker is attempting to rotate to a new address they have not yet fully secured, and an adversary races to invalidate the declaration.

Starknet's public mempool makes the front-run straightforward to execute once the staker's `change_operational_address` transaction is visible.

---

### Recommendation
Introduce a commitment window: once `declare_operational_address(staker_address)` is called for a non-zero `staker_address`, prevent the operational address from overwriting that declaration for at least one block (or one epoch). Alternatively, require the staker to co-sign the declaration (e.g., by having the staker call a `confirm_operational_address` function), so the mapping cannot be invalidated without the staker's participation.

---

### Proof of Concept

```
1. op_addr_X (attacker-controlled) calls:
       staking.declare_operational_address(victim_staker)
   → eligible_operational_addresses[op_addr_X] = victim_staker

2. victim_staker submits (visible in mempool):
       staking.change_operational_address(op_addr_X)

3. Attacker front-runs with:
       staking.declare_operational_address(0)   // called from op_addr_X
   → eligible_operational_addresses[op_addr_X] = 0

4. victim_staker's tx executes:
       assert!(eligible_operational_addresses[op_addr_X] == victim_staker)
       // 0 != victim_staker → panics: "Operational address had not been declared by staker"

5. Repeat step 3 on every subsequent attempt by victim_staker.
```
