### Title
`calculateNFTRoyalties()` Returns Inflated `nftSellerNetAmount` Due to Commented-Out Royalty Deduction, Spoofing "Net Proceeds" in NFT Offer Acceptance UI - (File: packages/gui/src/components/offers/utils.ts)

---

### Summary

In `calculateNFTRoyalties()`, the `nftSellerNetAmount` field is unconditionally set to the full `amount` with the correct deduction logic commented out. For `TokenForNFT` offers (buyer-initiated), the NFT offer viewer uses this value to render the prominent "Net Proceeds" label and to override the "You will receive" row. Because royalties are never subtracted, the seller is shown an inflated figure and may accept an offer expecting to receive more XCH/CAT than the blockchain will actually deliver.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties()` computes the royalty amount correctly but then assigns `nftSellerNetAmount` to the raw `amount` with the subtraction commented out:

```typescript
// packages/gui/src/components/offers/utils.ts  lines 312-317
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;          // ← always the full amount
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

`nftSellerNetAmount` is consumed in two places in the offer acceptance flow:

**1. `NFTOfferViewer.tsx` — `overrideNFTSellerAmount` computation**

```typescript
// lines 397-402
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // inflated
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

This value is passed as `overrideNFTSellerAmount` to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the displayed amount in the "You will receive" row:

```typescript
// OfferSummaryTokenRow line 146
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [3](#0-2) 

**2. `NFTOfferViewer.tsx` — "Net Proceeds" display**

```typescript
// lines 595-597
<Typography variant="h5" fontWeight="bold">
  <FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} /> {displayName}
</Typography>
``` [4](#0-3) 

The tooltip for this field explicitly states: *"The net proceeds include the asking price, minus any associated creator fees (if the NFT has royalty payments enabled)."* — but the code never performs that subtraction. [5](#0-4) 

The same wrong value is also shown in `NFTOfferEditor.tsx` as "They will receive" when a buyer previews a `TokenForNFT` offer: [6](#0-5) 

---

### Impact Explanation

For any `TokenForNFT` offer (buyer offers XCH/CAT for an NFT that carries a royalty), the NFT seller who opens the offer file in the GUI sees:

| Label | Displayed | Actual |
|---|---|---|
| NFT Purchase Price | X XCH | X XCH |
| Creator Fee (R%) | X × R% XCH | X × R% XCH |
| **Net Proceeds** | **X XCH** ← wrong | **X × (1 − R%) XCH** |
| "You will receive" row | **X XCH** ← wrong | **X × (1 − R%) XCH** |

The seller accepts the offer believing they will receive `X` XCH; the blockchain delivers `X × (1 − R%)` XCH. The financial shortfall equals the royalty amount. For a 10 XCH offer with a 10% royalty, the seller loses 1 XCH relative to their displayed expectation.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of… offer… state that causes a user to approve… the wrong… amount."*

---

### Likelihood Explanation

- Any unprivileged buyer can create a `TokenForNFT` offer for any royalty-bearing NFT and share the offer file/link with the seller.
- The seller opens the offer in the standard GUI offer viewer — no special conditions required.
- The bug is triggered for every NFT with a non-zero `royaltyPercentage`; the Chia NFT standard supports royalties up to 100%.
- The seller has no in-GUI mechanism to detect the discrepancy: the "Net Proceeds" label is authoritative-looking and its tooltip explicitly promises the royalty-deducted figure.

---

### Recommendation

Restore the commented-out subtraction in `calculateNFTRoyalties()`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString)).toFixed(12),
);
```

For `NFTForToken` (seller-initiated), `nftSellerNetAmount` is not rendered as "Net Proceeds" in the viewer, so the fix is safe to apply unconditionally; the `exchangeType` guard in `overrideNFTSellerAmount` already limits the override to `TokenForNFT` paths.

---

### Proof of Concept

1. Mint an NFT with `royalty_percentage = 1000` (10%).
2. As an unprivileged buyer, create a `TokenForNFT` offer: offer 10 XCH for the NFT.
3. Share the offer file with the NFT owner (seller).
4. Seller opens the offer in the Chia GUI NFT offer viewer.
5. Observe: "Net Proceeds" displays **10 XCH**; "Creator Fee (10%)" displays **1 XCH**.
6. Seller accepts expecting 10 XCH; wallet receives **9 XCH**.
7. Root cause: `nftSellerNetAmount = amount` (line 314 of `utils.ts`) — the 1 XCH royalty is never subtracted.
