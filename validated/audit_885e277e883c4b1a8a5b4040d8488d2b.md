### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both `Staking.change_reward_address` and `Pool.change_reward_address` validate that the new reward address is not a token contract address, but neither validates that it is non-zero. A staker or pool member can set their reward address to `0x0`, causing all future reward transfers to be sent to the zero address and permanently lost. The `GenericError::ZERO_ADDRESS` error variant exists in the codebase but is never applied in these functions.

---

### Finding Description

`Staking.change_reward_address` (staking.cairo line 517) performs only one check before updating the reward address:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

No zero-address guard is present. The function then writes the zero address directly to storage and the next call to `send_rewards_to_staker` (staking.cairo line 1625) executes:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

with `recipient = 0x0`, permanently burning the staker's unclaimed yield.

The identical pattern exists in `Pool.change_reward_address` (pool.cairo line 505–526), which only checks:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

No zero-address check is present there either. The `GenericError::ZERO_ADDRESS` variant is defined in `src/errors.cairo` line 20 and is used elsewhere in the codebase (e.g., `add_token` assertions), confirming the pattern is known and intentionally applied in other places — but was omitted here.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

When a staker calls `change_reward_address(0x0)` and subsequently triggers reward distribution (via attestation or `claim_rewards`), `send_rewards_to_staker` transfers the accumulated `unclaimed_rewards_own` to address zero. Those tokens are irrecoverably lost. The same applies to pool member rewards via `Pool.claim_rewards` after `Pool.change_reward_address(0x0)`. There is no recovery path: the staker cannot re-stake from the same address (it is blacklisted after exit), and the reward address cannot be changed back to a valid address once rewards have already been sent to zero.

---

### Likelihood Explanation

**Low-Medium.** The most realistic trigger is a buggy front-end, SDK, or integration contract that passes an uninitialized or default-zero address when calling `change_reward_address`. On-chain integrations (e.g., smart-contract wallets, auto-compounders, or liquid staking wrappers) that manage reward routing programmatically are particularly susceptible if they fail to validate the address before forwarding it. The call is permissionless for the staker/pool-member role — no privileged access is required.

---

### Recommendation

Add a zero-address guard in both `change_reward_address` implementations, consistent with the existing `GenericError::ZERO_ADDRESS` pattern used elsewhere:

```cairo
assert!(!reward_address.is_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This should be inserted before the token-address check in both `staking.cairo` and `pool.cairo`.

---

### Proof of Concept

**Staking contract path:**

1. Staker calls `Staking.change_reward_address(reward_address: 0x0)`.
2. The only guard — `does_token_exist(0x0)` — returns `false` (zero is not a registered token), so the assert passes. [1](#0-0) 
3. `staker_info.reward_address` is written as `0x0`.
4. On the next attestation or explicit `claim_rewards`, `send_rewards_to_staker` executes `checked_transfer(recipient: 0x0, amount: unclaimed_rewards_own)`. [2](#0-1) 
5. Rewards are permanently transferred to the zero address.

**Pool contract path:**

1. Pool member calls `Pool.change_reward_address(reward_address: 0x0)`.
2. The only guard — `token_dispatcher.contract_address != 0x0` — passes (the token address is non-zero). [3](#0-2) 
3. `pool_member_info.reward_address` is written as `0x0`.
4. On `Pool.claim_rewards`, accumulated rewards are transferred to `0x0` and permanently lost.

**Existing `ZERO_ADDRESS` guard used elsewhere (confirming the omission is an oversight):** [4](#0-3)
