### Title
Unbounded Loop in `calculate_rewards` Over Unconstrained `pool_member_epoch_balance` Trace - (File: src/pool/pool.cairo)

---

### Summary

`calculate_rewards` in `src/pool/pool.cairo` contains a `while` loop that iterates over the `pool_member_epoch_balance` trace with no upper bound on its length. The trace grows by one checkpoint per epoch in which a pool member makes a balance change without claiming rewards. The developers themselves acknowledge this in a code comment. If a pool member accumulates enough unclaimed balance-change entries, a `claim_rewards` call will consume unbounded gas and eventually revert, permanently freezing that member's unclaimed yield.

---

### Finding Description

`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) iterates over the `pool_member_epoch_balance` trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace is stored as `pool_member_epoch_balance: Map<ContractAddress, PoolMemberBalanceTrace>` and grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation appends a **new** checkpoint whenever the key (epoch + K) differs from the last stored key — i.e., whenever a balance change occurs in a different epoch than the previous one. [2](#0-1) 

`set_member_balance` is called by two publicly reachable functions:

- `add_to_delegation_pool` (line 221) — callable by any pool member or their reward address.
- `exit_delegation_pool_intent` (line 256) — callable by any pool member. [3](#0-2) [4](#0-3) 

The `entry_to_claim_from` pointer stored in `pool_member_info` is advanced after each successful `claim_rewards` call, so the loop only re-processes entries since the last claim. However, if a pool member makes one balance change per epoch across many epochs **without ever claiming**, the trace accumulates one new entry per epoch. When `claim_rewards` is eventually called, the loop must iterate over all accumulated entries in a single transaction. [5](#0-4) 

There is no cap on the size of `pool_member_epoch_balance` and no size check in `set_member_balance` or `add_to_delegation_pool`. [6](#0-5) 

---

### Impact Explanation

Once the trace is large enough that a single `claim_rewards` call exceeds the Starknet transaction gas limit, the call will always revert. Because `entry_to_claim_from` is only updated on a **successful** completion of `claim_rewards`, there is no way to process the trace in partial batches. The pool member's accumulated unclaimed yield becomes permanently inaccessible — matching the **High** impact: *Permanent freezing of unclaimed yield*.

At minimum, even before the permanent-freeze threshold, each `claim_rewards` call consumes gas proportional to the number of unprocessed trace entries, matching the **Medium** impact: *Unbounded gas consumption*.

---

### Likelihood Explanation

Any pool member (delegator) who:
1. Makes at least one balance change per epoch (via `add_to_delegation_pool` or `exit_delegation_pool_intent`), and
2. Defers calling `claim_rewards` for an extended period,

will accumulate trace entries. This can happen inadvertently (a long-term delegator who rarely claims) or deliberately (a griefing attacker targeting their own account to demonstrate the issue, or a user who simply forgets to claim). No privileged access, key compromise, or external dependency is required. The entry path is fully permissionless.

---

### Recommendation

1. **Enforce a maximum trace length**: Add a size check in `set_member_balance` (or `add_to_delegation_pool` / `exit_delegation_pool_intent`) that reverts if the pool member's `pool_member_epoch_balance` trace would exceed a safe bound (e.g., 500 entries).
2. **Force periodic claiming**: Require `claim_rewards` to be called (or auto-claim) before a new balance change is recorded when the trace exceeds a threshold, resetting `entry_to_claim_from` and keeping the active window small.
3. **Paginated claiming**: Expose a partial-claim interface that processes a caller-specified number of trace entries per call, allowing the pool member to drain the trace over multiple transactions.

---

### Proof of Concept

1. Pool member `A` calls `add_to_delegation_pool` once per epoch for `N` consecutive epochs, never calling `claim_rewards`. Each call lands in a new epoch, so `set_member_balance` → `trace.insert` appends a new `PoolMemberBalanceCheckpoint`, growing `pool_member_epoch_balance` to length `N`.

2. After `N` epochs, `A` (or `A`'s reward address) calls `claim_rewards`. This invokes `calculate_rewards` with `entry_to_claim_from = 0` and `pool_member_trace_length = N`.

3. The `while` loop at line 859 iterates `N` times, performing a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` call on each iteration.

4. For sufficiently large `N`, the transaction exceeds the Starknet gas limit and reverts. Because `entry_to_claim_from` is never written back on a failed transaction, every subsequent `claim_rewards` attempt also reverts — permanently freezing `A`'s accumulated yield. [7](#0-6) [8](#0-7)

    /// Inserts a (`key`, `value`) pair into a Trace so that it is stored as the checkpoint.
    /// This is done by either inserting a new checkpoint, or updating the last one.
    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
