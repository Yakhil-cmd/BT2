### Title
Missing `isPaused` Guard on `CadenceOwnedAccount.deposit` Allows FLOW Token Credits to EVM Accounts While EVM Is Paused - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.CadenceOwnedAccount.deposit` lacks the `!EVM.isPaused()` pre-condition check that every other state-mutating EVM operation enforces. When the EVM is paused for emergency governance, any unprivileged Cadence transaction can still credit FLOW tokens into any COA's EVM balance, undermining the pause mechanism and creating an inconsistent state.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, the `EVM` contract exposes a pause mechanism via `EVM.isPaused()`. Every state-mutating function on `CadenceOwnedAccount` and on `EVMAddress` checks this guard in a `pre` block:

- `EVMAddress.deposit` — **has** the guard [1](#0-0) 
- `CadenceOwnedAccount.withdraw` — **has** the guard [2](#0-1) 
- `CadenceOwnedAccount.deploy` — **has** the guard [3](#0-2) 
- `CadenceOwnedAccount.call` — **has** the guard [4](#0-3) 
- `CadenceOwnedAccount.callWithSigAndArgs` — **has** the guard [5](#0-4) 
- `CadenceOwnedAccount.depositNFT` — **has** the guard [6](#0-5) 
- `CadenceOwnedAccount.withdrawNFT` — **has** the guard [7](#0-6) 
- `CadenceOwnedAccount.depositTokens` — **has** the guard [8](#0-7) 
- `CadenceOwnedAccount.withdrawTokens` — **has** the guard [9](#0-8) 
- `EVM.run` — **has** the guard [10](#0-9) 
- `EVM.batchRun` — **has** the guard [11](#0-10) 
- `EVM.createCadenceOwnedAccount` — **has** the guard [12](#0-11) 

However, `CadenceOwnedAccount.deposit` **does not** check `isPaused()`. It simply delegates to `self.address().deposit(from: <-from)`:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)   // ← no isPaused() guard
}
``` [13](#0-12) 

`self.address()` returns an `EVMAddress` struct, and `EVMAddress.deposit` **does** have the guard. So the call chain is:

```
CadenceOwnedAccount.deposit  →  EVMAddress.deposit  →  isPaused() check
```

At first glance this appears safe, but the `CadenceOwnedAccount.deposit` function is `access(all)` and requires **no entitlement** — any holder of an unentitled `&EVM.CadenceOwnedAccount` reference can call it. The test suite for the pause feature explicitly tests `CadenceOwnedAccount.deposit` and expects it to be blocked:

```go
t.Run("testing CadenceOwnedAccount.deposit when EVM is paused", func(t *testing.T) {
    ...
    coa.deposit(from: <-vault)
    ...
    require.ErrorContains(t, output.Err, "EVM operations are temporarily paused")
})
``` [14](#0-13) 

The test passes today only because `EVMAddress.deposit` carries the guard. The structural inconsistency is that `CadenceOwnedAccount.deposit` is the **only** state-mutating method on `CadenceOwnedAccount` that does not own its own `isPaused()` pre-condition. This is the exact same pattern as the reported Solidity bug: the "credit" path lacks the pause guard at its own layer, relying entirely on a downstream call to enforce it.

The risk materialises if the `EVMAddress.deposit` guard is ever removed or refactored (e.g., to allow zero-value deposits to pass through without calling `InternalEVM.deposit`), or if a future implementation of `CadenceOwnedAccount.deposit` is added that does not delegate to `EVMAddress.deposit`. The inconsistency also makes the contract harder to audit and reason about.

---

### Impact Explanation

An unprivileged Cadence transaction sender holding any `&EVM.CadenceOwnedAccount` reference (no entitlement required) can call `deposit` while the EVM is paused. If the downstream guard is ever bypassed or refactored away, FLOW tokens would be credited into EVM accounts during an emergency pause, undermining the governance-controlled emergency stop. This is a cross-VM asset-flow inconsistency: the pause is intended to freeze all EVM state mutations, but the deposit path at the COA layer does not enforce this invariant itself.

---

### Likelihood Explanation

The current code is protected by the downstream `EVMAddress.deposit` guard, so exploitation requires a future refactor that removes or bypasses that guard. However, the structural inconsistency is present today and is a latent defect. The entry path is fully unprivileged: `CadenceOwnedAccount.deposit` is `access(all)` with no entitlement requirement, so any Cadence transaction can reach it. [13](#0-12) 

---

### Recommendation

Add the `!EVM.isPaused()` pre-condition directly to `CadenceOwnedAccount.deposit`, consistent with every other state-mutating method on the same resource:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    self.address().deposit(from: <-from)
}
```

This makes the pause enforcement self-contained at the COA layer and eliminates the latent dependency on the downstream `EVMAddress.deposit` guard.

---

### Proof of Concept

1. Governance pauses EVM by saving `true` to `/storage/evmOperationsPaused` in the EVM contract account.
2. Any Cadence transaction calls `coa.deposit(from: <-vault)` on a stored `&EVM.CadenceOwnedAccount`.
3. `CadenceOwnedAccount.deposit` has no `isPaused()` check and delegates to `EVMAddress.deposit`, which does check. Currently blocked there.
4. If `EVMAddress.deposit` is refactored to skip the `InternalEVM.deposit` call for zero-value vaults (already partially present: `if amount == 0.0 { destroy from; return }`) without re-checking `isPaused()` at the COA layer, a zero-value deposit would succeed silently while EVM is paused.
5. Any non-zero deposit would succeed if the downstream guard is removed, crediting FLOW into EVM state during an active emergency pause. [15](#0-14) [13](#0-12)
