### Title
Commission Increased Mid-Epoch Without Settling Pending Rewards Steals Delegator Yield - (`src/staking/staking.cairo`)

### Summary

`set_commission` updates the commission rate immediately in storage without first settling the pending rewards accrued under the old rate. In the V2 (pre-consensus, epoch-based) reward model, `update_rewards_from_attestation_contract` uses the commission value present in storage at attestation time to calculate rewards for the **entire epoch**. A staker who holds a `commission_commitment` can raise their commission mid-epoch, causing the higher rate to be retroactively applied to the full epoch's pool rewards, stealing yield that delegators earned under the old rate.

---

### Finding Description

`set_commission` at [1](#0-0)  calls the internal `update_commission` helper, which validates the new value and writes it directly to storage: [2](#0-1) 

No reward-settlement step precedes the storage write. The commission is then consumed at attestation time inside `calculate_staker_pools_rewards`: [3](#0-2) 

`update_rewards_from_attestation_contract` (the V2 epoch reward path) reads the live commission and applies it to the **entire epoch's** pool rewards in a single call: [4](#0-3) 

The `commission_commitment` mechanism explicitly permits a staker to raise commission above the current value, up to `max_commission`: [5](#0-4) 

The combination means: a staker sets a commitment with a high `max_commission`, waits until late in an epoch, raises commission to `max_commission`, then attests. The attestation call applies the new (higher) commission to rewards that were earned throughout the entire epoch, not just from the moment of the change.

---

### Impact Explanation

**High — Theft of unclaimed yield from delegators.**

Delegators earned rewards under the old (lower) commission for the portion of the epoch before the commission change. Because no checkpoint is taken at the moment of the change, those accrued rewards are recalculated at attestation time using the new (higher) commission. The difference — commission on rewards earned before the change — is transferred to the staker instead of the pool, permanently reducing delegator yield for that epoch.

In the extreme case (commission raised to 100% just before attestation), the staker captures the entire epoch's pool rewards, leaving delegators with zero yield for that epoch.

---

### Likelihood Explanation

The attack requires only that the staker:
1. Call `set_commission_commitment` with a `max_commission` higher than the current commission (permissible per the spec, line 983).
2. Wait until near the end of an epoch.
3. Call `set_commission` to raise commission to `max_commission`.
4. Attest (or wait for attestation to be triggered).

All four steps are available to any active staker with a pool. No privileged access, leaked key, or external dependency is required. The staker is an explicitly listed attacker role in the scope.

---

### Recommendation

Before writing the new commission to storage in `update_commission`, call the internal reward-settlement logic to checkpoint the current epoch's accrued rewards at the old commission rate. This mirrors the fix applied in the Strata system: trigger an accounting update before any change to the rate variable, so the new rate only applies to future accruals.

---

### Proof of Concept

1. Staker stakes with commission = 200 (2%) and opens a delegation pool.
2. Delegator enters the pool with a large amount.
3. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
4. Staker calls `set_commission(commission: 10000)` — commission is now 100%, no rewards are settled.
5. Staker attests (calls `attest` on the attestation contract, which calls `update_rewards_from_attestation_contract`).
6. Inside `calculate_staker_pools_rewards`, `commission = staker_pool_info.commission()` returns 10000.
7. `split_rewards_with_commission` routes 100% of pool rewards to the staker as commission.
8. The pool receives zero STRK for the epoch; delegators receive zero yield despite having been staked the entire epoch at 2% commission.

The staker has stolen the delegators' unclaimed yield for the full epoch by retroactively applying the 100% commission to rewards earned under the 2% rate.

            // Prerequisites and asserts.

                    );

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
        }
