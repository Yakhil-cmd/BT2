### Title
`max_nonce_for_validation_skip` Config Field Stored But Never Read in Gateway — Validation-Skip Boundary Is Hardcoded to Nonce 1 Regardless of Configuration - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` declares and serializes a field `max_nonce_for_validation_skip` that is intended to control the upper nonce bound for which the gateway may skip the `__validate__` entry-point call (the account's signature check). The free function `skip_stateful_validations` — the only place this decision is made in the Rust gateway — never reads that field; it hardcodes `Nonce(Felt::ONE)` instead. The result is a permanent mismatch between the operator-configured policy and the actual admission behavior: setting the field to `0` (disable the skip) still admits unsigned invoke transactions, and setting it above `1` still rejects transactions the operator intended to admit without validation.

### Finding Description

`StatefulTransactionValidatorConfig` in `crates/apollo_gateway_config/src/config.rs` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,   // default Nonce(Felt::ONE)
``` [1](#0-0) 

The field is serialized, appears in `config_schema.json` and the deployment JSON files, and is documented as *"Maximum nonce for which the validation is skipped."*

The legacy Python gateway (`native_blockifier`) correctly consumes this field:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [2](#0-1) 

The new Rust gateway's `skip_stateful_validations` is a free function that receives no config argument and hardcodes the bound:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [3](#0-2) 

`run_pre_validation_checks` calls `skip_stateful_validations` without forwarding `self.config.max_nonce_for_validation_skip`:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [4](#0-3) 

A grep over the entire `crates/apollo_gateway/` tree returns **zero** matches for `max_nonce_for_validation_skip`, confirming the field is dead code in the production gateway path.

When `skip_validate` is `true`, the gateway sets `validate: false` in `ExecutionFlags`, suppressing the `__validate__` call entirely:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

### Impact Explanation

**Case 1 — operator sets `max_nonce_for_validation_skip: 0` to disable the skip.**  
The gateway ignores the setting and still skips `__validate__` for every invoke transaction whose on-chain nonce is 0 and whose tx nonce is 1, provided a `deploy_account` for that sender is in the mempool. An attacker who submits a valid `deploy_account` for any not-yet-deployed address can immediately follow with an invoke at nonce 1 carrying an arbitrary or empty signature; the gateway admits it to the mempool without signature verification. This matches the "High" impact: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

**Case 2 — operator sets `max_nonce_for_validation_skip: N > 1` to extend the skip window.**  
Invoke transactions at nonces 2 … N from accounts with a pending `deploy_account` are forced through `__validate__`, which fails because the account contract does not yet exist on-chain. Valid transactions are rejected. This also matches the "High" impact: *"… rejects valid transactions before sequencing."*

### Likelihood Explanation

Any operator who changes `max_nonce_for_validation_skip` from its default value of `1` — whether to harden security (set to `0`) or to widen the UX window (set to `2+`) — will silently experience the wrong behavior. The field is prominently documented and present in the deployment config files, making it a natural target for tuning. The attacker-controlled trigger for Case 1 requires only submitting a standard `deploy_account` transaction, which is unprivileged.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded `Nonce(Felt::ONE)` with the configured bound, mirroring the logic already present in `native_blockifier/src/py_validator.rs`:

```rust
// In run_pre_validation_checks:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,   // <-- add this
).await?;

// In skip_stateful_validations:
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
    max_nonce_for_validation_skip: Nonce,        // <-- add this
) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if Nonce(Felt::ONE) <= tx.nonce()
            && tx.nonce() <= max_nonce_for_validation_skip   // <-- use config
            && account_nonce == Nonce(Felt::ZERO)
        { ... }
    }
    Ok(false)
}
```

### Proof of Concept

1. Deploy the gateway with `max_nonce_for_validation_skip: 0` (intending to disable the skip).
2. Submit a valid `deploy_account` transaction for address `X` (nonce 0 on-chain, account not yet deployed).
3. Submit an invoke transaction from `X` with nonce `1` and an empty/invalid signature.
4. Observe: `skip_stateful_validations` returns `true` (because `tx.nonce() == Nonce(Felt::ONE)` is hardcoded), `validate` is set to `false`, and the gateway admits the transaction to the mempool without calling `__validate__`.
5. Expected (with correct fix): `skip_stateful_validations` returns `false` because `Nonce(1) > max_nonce_for_validation_skip (0)`, `validate` is `true`, `__validate__` is called, and the transaction is rejected with a signature error.

The discrepancy between the `native_blockifier` implementation (which correctly uses `max_nonce_for_validation_skip`) and the Rust gateway (which hardcodes `Nonce(Felt::ONE)`) confirms this is a porting omission, not an intentional design choice. [6](#0-5) [7](#0-6) [8](#0-7)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
