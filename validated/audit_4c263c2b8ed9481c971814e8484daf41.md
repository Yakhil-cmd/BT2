### Title
WalletConnect Multi-NFT Royalty Accounting Error Causes Underreported "Total Amount with Royalties" in Offer Confirmation Dialog - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` divides the fungible amount by the number of NFTs before applying each royalty percentage. When a WalletConnect dApp sends a `chia_wallet.take_offer` request containing multiple NFTs with royalties, the "Total Amount with Royalties" shown in the confirmation dialog is underestimated by a factor of N (the number of NFTs). The user approves based on a lower displayed cost; the wallet backend then deducts the correct, higher amount.

### Finding Description

The function `formatAmountWithRoyalties` computes the royalty-inclusive total shown to the user in the WalletConnect `Confirm` dialog:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts:363-368
const splitAmount = amount / BigInt(royaltyPercentages.length); // divides first — wrong
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The correct formula for total royalties when buying N NFTs with a single fungible asset is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The buggy formula computes instead:

```
royaltyAmount = (amount / N) × Σ royaltyPercentage_i / 10_000
```

This underestimates royalties by exactly 1/N. The existing test fixture encodes this wrong behavior as the expected value:

```
// 2 NFTs, royalties 500 (5%) and 10 (0.1%), amount = 100,000,000 mojos
// Displayed: 0.00010255 XCH  (2.55% total — half of correct)
// Correct:   0.00010510 XCH  (5.10% total)
amountWithRoyalties: '0.00010255'
``` [2](#0-1) 

The `amountWithRoyalties` field is rendered in the WalletConnect `Confirm` dialog under the label **"Total Amount with Royalties"** in the "You Spend" section: [3](#0-2) 

The locale files confirm this label is displayed at `Confirm.tsx:111`, `Confirm.tsx:132`, and `Confirm.tsx:150` — the primary cost summary the user reads before clicking Confirm.

`formatAmountWithRoyalties` is called from `withRoyaltyTotals`, which is called from `walletDeltaToDisplay`, which is called from `parseCommandDisplay` for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`: [4](#0-3) [5](#0-4) 

### Impact Explanation

The user sees a "Total Amount with Royalties" that is 1/N of the correct value (where N = number of NFTs in the offer). With 3 NFTs each carrying a 10% royalty, the dialog shows a 3.3% total royalty cost instead of 30%. The user approves; the Chia wallet backend applies the correct royalties and deducts the full amount. The discrepancy is unbounded as N grows. This constitutes spoofing of WalletConnect offer state that causes a user to approve spending the wrong (higher) amount of XCH/CAT.

### Likelihood Explanation

Any WalletConnect-connected dApp can send a `chia_wallet.take_offer` request. The attacker mints NFTs with high royalty percentages (up to the protocol maximum), bundles multiple into a single offer, and presents it to the user. No privileged access, leaked keys, or social engineering beyond the normal WalletConnect pairing flow is required. The trigger is a standard multi-NFT offer, which is a supported and common use case.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty is calculated on the full fungible amount, not a per-NFT share:

```typescript
// Correct
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the test at line 320 to expect `'0.0001051'` (5.1% of 0.0001 XCH) instead of `'0.00010255'`.

### Proof of Concept

Given a `chia_wallet.take_offer` WalletConnect request for an offer containing:
- NFT A: `royalty_percentage = 500` (5%)
- NFT B: `royalty_percentage = 500` (5%)
- Requested: 1,000,000,000,000 mojos (1 XCH)

**Buggy display:**
- `splitAmount = 1,000,000,000,000 / 2 = 500,000,000,000`
- royalty A = `500,000,000,000 × 500 / 10,000 = 25,000,000,000`
- royalty B = `25,000,000,000`
- Total shown: `1,050,000,000,000` mojos = **1.05 XCH**

**Correct total:**
- royalty A = `1,000,000,000,000 × 500 / 10,000 = 50,000,000,000`
- royalty B = `50,000,000,000`
- Total correct: `1,100,000,000,000` mojos = **1.10 XCH**

The user sees 1.05 XCH in the "You Spend / Total Amount with Royalties" field and approves. The wallet deducts 1.10 XCH. The 0.05 XCH difference goes to NFT royalty recipients — not to the attacker directly, but the user has been deceived into approving a transaction whose true cost was misrepresented in the security-critical approval dialog. function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
