### Title
Missing Access Control on `update_rewards` Allows Anyone to Permanently Block Per-Block Reward Distribution - (File: `src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function is documented as callable by "Only starkware sequencer" but has **no on-chain access control enforcement**. Any address can call it with `disable_rewards: true` to consume the global `last_reward_block` slot for the current block, permanently preventing the sequencer from distributing rewards for that block to any staker.

---

### Finding Description

In the V3 (consensus rewards) mode, `update_rewards` is the sole mechanism for distributing per-block staking rewards. The function uses a single global storage variable `last_reward_block` to enforce a one-call-per-block invariant:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
    self.last_reward_block.write(current_block_number);  // <-- global, not per-staker

    if disable_rewards || self.is_pre_consensus() {
        return;  // <-- exits without distributing rewards
    }
    // ... distribute rewards
}
```

The spec explicitly states access is "Only starkware sequencer," but the code contains **no caller check** — `general_prerequisites()` only checks the pause state, not the caller identity. This is confirmed by the test suite, which calls `update_rewards` without any `cheat_caller_address_once` override.

**Attack path:**

1. Attacker monitors the mempool/block production.
2. At the start of each block, attacker calls `update_rewards(any_valid_staker_address, disable_rewards: true)`.
3. `last_reward_block` is set to the current block number.
4. The sequencer's legitimate call to `update_rewards` for the intended staker reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers permanently lose the rewards for that block — they can never be recovered.

The attacker can sustain this indefinitely at low cost (one transaction per block), causing continuous, permanent loss of unclaimed yield across all stakers.

Additionally, an attacker can call `update_rewards(staker_A, disable_rewards: false)` to force rewards to be distributed to staker_A instead of the staker the sequencer intended to reward in that block, causing staker_B to miss their block rewards.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.** Each griefed block's rewards are permanently lost — the `last_reward_block` guard prevents any retroactive distribution. Over time, a sustained attacker can eliminate a significant fraction of all staker rewards. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**High.** No special privileges, tokens, or setup are required. Any EOA can call `update_rewards` with an arbitrary valid `staker_address` and `disable_rewards: true`. The cost is one transaction per block. The attack is trivially automatable.

---

### Recommendation

Add an explicit caller check to `update_rewards` to enforce the "Only starkware sequencer" access control stated in the spec. For example, introduce a stored `sequencer_address` and assert `get_caller_address() == self.sequencer_address.read()` at the top of the function, analogous to how `update_rewards_from_attestation_contract` enforces `assert_caller_is_attestation_contract()`.
