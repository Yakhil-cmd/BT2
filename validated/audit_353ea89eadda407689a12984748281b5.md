### Title
Truncating Division in `native_per_pubdata` Computation Systematically Undercharges Pubdata Costs - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the number of native resource units charged per byte of pubdata — is computed using truncating (floor) division (`wrapping_div`) instead of ceiling division (`div_ceil`). This is the exact rounding direction that is wrong for an incoming-cost calculation: whenever `pubdata_price % native_price != 0`, every pubdata byte costs 1 fewer native unit than it should, causing the user to systematically underpay for pubdata and the operator to be underpaid.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 142, `native_per_pubdata` is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The identical pattern appears in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

By contrast, `native_per_gas` — the analogous ratio for computational cost — is correctly computed with ceiling division:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;
``` [3](#0-2) 

The underestimated `native_per_pubdata` propagates into two downstream charging sites:

**1. Intrinsic pubdata overhead** (deducted from native budget at transaction start):
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
``` [4](#0-3) 

**2. Post-execution pubdata charge:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

Both sites multiply `native_per_pubdata` by the number of pubdata bytes. When `native_per_pubdata` is 1 too low, the total native charged for pubdata is `N` units too low (where `N` = pubdata bytes used).

This reduced native consumption then flows into `compute_gas_refund`, where `delta_gas` is computed as:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [6](#0-5) 

Because `native_used` is lower (less native was charged for pubdata), `delta_gas` is smaller than it should be, so `gas_used` is lower, and the user is refunded more gas than they should be.

---

### Impact Explanation

For every transaction where `pubdata_price % native_price != 0` (the common production case), the user pays 1 fewer native unit per pubdata byte than the actual cost. This translates directly into a lower `gas_used` and a larger gas refund. The operator receives less fee than the actual pubdata cost warrants.

**Concrete example:**
- `pubdata_price = 101`, `native_price = 10`
- Correct: `native_per_pubdata = ceil(101/10) = 11`
- Actual: `native_per_pubdata = floor(101/10) = 10` (9% undercharge per byte)
- For a transaction writing 1,000 pubdata bytes: 1,000 fewer native units charged
- With `native_per_gas = 11`: `delta_gas` is ~90 gas units lower than it should be
- User pays ~90 fewer gas units × `gas_price` fewer tokens to the operator

The discount is systematic and repeatable. A user who generates maximum pubdata per transaction (e.g., many storage writes) maximizes the underpayment per block.

---

### Likelihood Explanation

The condition `pubdata_price % native_price != 0` holds for virtually all realistic operator configurations, since `pubdata_price` and `native_price` are independently set oracle values with no alignment requirement. Any unprivileged L2 transaction sender can trigger this path simply by submitting a transaction that writes to storage (generating pubdata). No special access, leaked keys, or governance majority is required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

**In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`:**
```rust
// Before:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

**In `api/src/helpers.rs`:**
```rust
// Before:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;

// After:
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

This ensures that the native resource charged for pubdata is always at least the true cost, consistent with the principle that incoming costs must be rounded up.

---

### Proof of Concept

1. Operator sets `pubdata_price = 101`, `native_price = 10`, `base_fee = 1000`.
2. User submits an L2 EIP-1559 transaction with `max_fee_per_gas = 1000`, `gas_limit = 200_000`, and calldata that triggers 1,000 bytes of pubdata (e.g., 31 storage slot writes).
3. Bootloader computes `native_per_pubdata = floor(101/10) = 10` instead of `ceil(101/10) = 11`.
4. Post-execution pubdata charge: `1000 * 10 = 10_000` native units instead of `1000 * 11 = 11_000`.
5. `native_used` is 1,000 units lower than it should be.
6. `native_per_gas = ceil(1000/10) = 100`.
7. `delta_gas = (native_used / 100) - gas_used` is ~10 gas units lower than it should be.
8. `gas_used` is ~10 units lower; user is refunded ~10 extra gas × 1000 wei/gas = ~10,000 wei per transaction.
9. Repeating this across many transactions (or with larger pubdata) scales the underpayment linearly. [1](#0-0) [2](#0-1) [5](#0-4) [6](#0-5)
