### Title
Unauthorized Transfer of `evm/<contract>` Native Tokens from Any Holder — (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `TransferMethodName` case in `BankContract.Run()` accepts an arbitrary `sender` address from ABI arguments and passes it directly to `bankKeeper.SendCoins()` with no check that `contract.Caller()` is authorized to spend from that address. Any EVM contract can drain `evm/<contract>` native tokens from any account that holds them.

---

### Finding Description

In `Run()`, the `transfer` branch reads `sender` from caller-supplied ABI arguments: [1](#0-0) 

`checkBlockedAddr` is called only on the **recipient** (`to`), never on `from`: [2](#0-1) 

The denom is `EVMDenom(contract.Caller())` — scoped to the calling contract — but `from` is fully attacker-controlled. There is no assertion that `from == contract.Caller()` or any approval/allowance mechanism. The call then executes: [3](#0-2) 

Cosmos SDK's `bankkeeper.BaseKeeper.SendCoins` checks `BlockedAddr` only on the **recipient**, not the sender, so sending from any address — including module accounts — is not blocked at the keeper level.

---

### Impact Explanation

A malicious contract `C` can call `transfer(victim, attacker, amount)` via the bank precompile. The denom resolved is `evm/C`. Any address holding `evm/C` tokens (e.g., users who previously called `C.mint(...)`) has those tokens stolen without consent. This is an unauthorized transfer of native Cosmos-side assets, matching the Critical impact category.

The specific module-account-drain sub-scenario (draining the `evm` module account of `evm/C` tokens) requires the module account to hold such tokens. In normal operation the mint/burn flows are atomic within `ExecuteNativeAction` so the module account does not accumulate `evm/<contract>` tokens. That sub-scenario is therefore a harder precondition to satisfy. The primary, immediately exploitable path is theft from regular user accounts.

---

### Likelihood Explanation

- Attacker deploys contract `C` (no privilege required).
- Attacker or any user calls `C.mint(victim, amount)` — victim now holds `evm/C` native tokens.
- Attacker calls `C` which calls `bankPrecompile.transfer(victim, attacker, amount)`.
- No admin, governance, or approval gate exists. Fully reachable by any unprivileged EVM caller.

---

### Recommendation

In the `TransferMethodName` branch, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

This mirrors the design of `mint`/`burn`, where the denom and the acting address are both derived from `contract.Caller()`. [4](#0-3) 

Alternatively, implement an ERC20-style allowance mapping inside the precompile so that a contract can only transfer on behalf of an address that has explicitly approved it.

---

### Proof of Concept

```solidity
// MaliciousContract.sol
contract MaliciousContract {
    address constant BANK = 0x0000000000000000000000000000000000000064;

    // Step 1: victim calls this, giving them evm/<this> native tokens
    function giveVictimTokens(address victim, uint256 amount) external {
        IBankModule(BANK).mint(victim, amount);
    }

    // Step 2: attacker calls this — no victim consent required
    function steal(address victim, address attacker, uint256 amount) external {
        // sender=victim, recipient=attacker, denom=evm/<this>
        IBankModule(BANK).transfer(victim, attacker, amount);
    }
}
```

Unit test plan:
1. Deploy `MaliciousContract` as `C`.
2. Call `C.giveVictimTokens(victim, 100)` — verify victim holds 100 `evm/C` tokens.
3. Call `C.steal(victim, attacker, 100)` — verify attacker now holds 100 `evm/C` tokens and victim holds 0, with no signature or approval from victim.
