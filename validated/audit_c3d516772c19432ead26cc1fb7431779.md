In `MetricOmmPool.addLiquidity`, `msg.sender` (the real caller) is passed as `sender` and `owner` is a free caller-controlled parameter: [2](#0-1) 

So the bypass is direct: any unprivileged `callerAddr` calls `pool.addLiquidity(owner=allowlistedAddr, ...)`. ### Title
`DepositAllowlistExtension.beforeAddLiquidity` checks `owner` instead of `sender`, allowing any unprivileged caller to bypass the deposit allowlist — (`metric-periphery/contracts/extensions/DepositAllowlistExtension.sol`)

### Summary
`DepositAllowlistExtension` is intended to restrict which addresses may add liquidity to a pool. Its `beforeAddLiquidity` hook silently discards the `sender` argument (the actual `msg.sender` of the pool call) and instead validates the `owner` argument (the LP-position recipient). Because `owner` is a free parameter supplied by the caller, any address can pass an allowlisted address as `owner` and satisfy the check.

### Finding Description
`MetricOmmPool.addLiquidity` calls `_beforeAddLiquidity(msg.sender, owner, ...)`, forwarding the real caller as `sender` and the caller-supplied recipient as `owner`. [3](#0-2) 

Inside the extension, the first parameter is unnamed and unused: [4](#0-3) 

The guard `allowedDepositor[msg.sender][owner]` uses `msg.sender` = the pool address (correct) but `owner` = the LP recipient (wrong). It should use the `sender` argument, which is the actual depositing address.

### Impact Explanation
The deposit allowlist is completely bypassed. Any unprivileged address can add liquidity to a pool that is supposed to be restricted (e.g., KYC-gated, whitelist-only institutional pools). The attacker provides tokens via the callback and the LP position is minted to the allowlisted `owner`. This allows unauthorized manipulation of bin liquidity state, front-running of allowlisted depositors, and violation of any access-control invariant the pool admin intended to enforce. Severity: **High** — core access-control functionality is broken with no existing guard preserving the invariant.

### Likelihood Explanation
Exploitable by any external address with no preconditions beyond knowing an allowlisted address (which is readable from `allowedDepositor` public mapping). No privileged role or special setup is required.

### Recommendation
Replace `owner` with the `sender` parameter in the allowlist check:

```solidity
// Before (wrong):
function beforeAddLiquidity(address, address owner, ...) ...
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][owner]) { ... }

// After (correct):
function beforeAddLiquidity(address sender, address owner, ...) ...
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) { ... }
```

### Proof of Concept
1. Deploy pool with `DepositAllowlistExtension`.
2. Pool admin calls `setAllowedToDeposit(pool, allowlistedAddr, true)`.
3. `callerAddr` (not allowlisted) calls `pool.addLiquidity(owner=allowlistedAddr, ...)` with valid token approval.
4. Extension checks `allowedDepositor[pool][allowlistedAddr]` → `true` → passes.
5. LP position is minted to `allowlistedAddr`; `callerAddr` successfully deposited despite not being on the allowlist.
