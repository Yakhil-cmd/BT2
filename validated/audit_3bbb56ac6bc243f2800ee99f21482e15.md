### Title
Silo `Address` Whitelist Activation Permanently Freezes ERC-20 Token Balances of Pre-Existing Non-Whitelisted Holders - (File: `engine/src/engine.rs`, `engine/src/contract_methods/silo/mod.rs`)

### Summary
In Silo mode, ERC-20 tokens are minted to any recipient address when the `Address` whitelist is disabled (the default state). When the owner subsequently enables the whitelist via `set_whitelist_status`, or removes an address via `remove_entry_from_whitelist`, those addresses can no longer submit EVM transactions — including `withdrawToNear` — permanently freezing their bridged ERC-20 token balances. Neither operation updates any accounting state or provides a migration path for existing token holders.

### Finding Description

**Step 1 — Tokens minted without whitelist enforcement during open-access period.**

`receive_erc20_tokens` in `engine/src/engine.rs` is the bridge entry point called by `ft_on_transfer` when a NEP-141 token is deposited. It contains the following conditional redirect:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

The redirect to the fallback address only fires when **both** a fallback address is configured **and** the recipient is not whitelisted. When the `Address` whitelist is disabled (the default), `is_allow_receive_erc20_tokens` returns `true` for every address:

```rust
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)  // true when list is disabled
}
``` [2](#0-1) 

So during the open-access period, `ERC20.mint(recipient, amount)` is called for any arbitrary recipient address, writing a real ERC-20 balance into EVM state.

**Step 2 — Withdrawal requires passing `assert_access`, which enforces the whitelist.**

Every EVM transaction submitted via `submit` passes through `assert_access`:

```rust
fn assert_access<I: IO + Copy, E: Env>(io: &I, env: &E, transaction: &NormalizedEthTransaction)
    -> Result<(), EngineError>
{
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else { ... };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
``` [3](#0-2) 

`is_allow_submit` checks the `Address` whitelist:

```rust
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [4](#0-3) 

The only way to withdraw ERC-20 tokens is to call `withdrawToNear` (or `withdrawToEthereum`) on the `EvmErc20` contract, which calls `_burn(msg.sender, amount)` — an EVM call that must go through `submit`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
}
``` [5](#0-4) 

**Step 3 — Whitelist state change does not update existing token holder state.**

`set_whitelist_status` and `remove_entry_from_whitelist` only toggle storage flags or remove storage keys. They perform no scan of existing ERC-20 balances and provide no migration path:

```rust
pub fn remove_entry_from_whitelist<I: IO + Copy>(io: &I, args: &WhitelistArgs) {
    let (kind, entry) = get_kind_and_entry(args);
    Whitelist::init(io, kind).remove(entry);
}
``` [6](#0-5) 

```rust
pub fn set_whitelist_status<I: IO + Copy>(io: &I, args: &WhitelistStatusArgs) {
    whitelist::set_whitelist_status(io, args);
}
``` [7](#0-6) 

After either operation, any address that holds ERC-20 tokens but is no longer whitelisted is permanently unable to call `withdrawToNear`, because every attempt is rejected at `assert_access` with `EngineErrorKind::NotAllowed`. There is no admin-callable burn-and-refund path in the ERC-20 contract, and no alternative withdrawal route that bypasses `submit`.

### Impact Explanation

**Permanent freezing of funds (Critical).** Any ERC-20 token balance held by an address that is removed from — or never added to — the `Address` whitelist after the whitelist is enabled becomes irrecoverable. The NEP-141 tokens have already been transferred into Aurora (consumed by the bridge), and the corresponding ERC-20 balance exists in EVM state, but the holder cannot burn them to reclaim the underlying NEP-141 tokens. The loss is permanent unless the owner re-adds the address to the whitelist.

### Likelihood Explanation

**Medium.** The trigger requires the contract owner to either enable the `Address` whitelist after a period of open operation, or remove a specific address from the whitelist. Both are normal administrative operations explicitly supported by the contract (`set_whitelist_status`, `remove_entry_from_whitelist`). A Silo deployment that starts open and later restricts access — a common operational pattern — will silently freeze all pre-existing token holders who are not explicitly re-added. The owner has no on-chain prompt or guard warning them of this consequence.

### Recommendation

1. **Guard `remove_entry_from_whitelist`**: Before removing an address from the `Address` whitelist, check whether it holds any ERC-20 balances. If it does, either reject the removal or emit a warning event.
2. **Guard `set_whitelist_status` activation**: When enabling the `Address` whitelist, require the caller to supply a list of addresses to grandfather in, or emit an event that tooling can use to enumerate affected holders.
3. **Add an owner-callable emergency withdrawal**: Expose an admin function that can burn ERC-20 tokens on behalf of a frozen address and return the underlying NEP-141 tokens, analogous to the recommended mitigation in the reference report (preventing the state from becoming irrecoverable).

### Proof of Concept

```
1. Deploy Aurora Engine in Silo mode.
   - Address whitelist: DISABLED (default).
   - No fallback address set.

2. Alice (NEAR account `alice.near`, EVM address 0xAlice) calls ft_transfer_call
   on a NEP-141 token contract, routing tokens to Aurora.
   → ft_on_transfer fires → receive_erc20_tokens called
   → whitelist disabled, so is_allow_receive_erc20_tokens returns true
   → ERC20.mint(0xAlice, amount) executes
   → Alice now holds `amount` ERC-20 tokens in Aurora EVM state.

3. Owner calls set_whitelist_status({ kind: Address, active: true })
   without adding 0xAlice.
   → Whitelist is now enforced; Alice is not in it.

4. Alice submits an EVM transaction calling withdrawToNear(recipient, amount)
   on the ERC-20 contract.
   → submit → assert_access → is_allow_submit(io, "alice.near", 0xAlice)
   → is_address_allowed: list.is_enabled()=true, list.is_exist(0xAlice)=false → false
   → EngineError { kind: NotAllowed }

5. Alice's ERC-20 tokens are permanently frozen.
   The underlying NEP-141 tokens remain locked in Aurora with no recovery path.
```
