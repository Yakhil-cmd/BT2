### Title
Missing ERC1271 Interface ID in `COA.supportsInterface` Breaks ERC1271 Compatibility - (File: `fvm/evm/handler/coa/coa.sol`)

### Summary
The `COA` (Cadence-Owned Account) contract implements `isValidSignature` (ERC1271) but omits the ERC1271 interface ID (`0x1626ba7e`) from its `supportsInterface` function. Any EVM contract that guards `isValidSignature` calls with an ERC165 interface check will incorrectly conclude that COA does not support ERC1271, silently breaking COA-based signature verification for all such callers.

### Finding Description
The `COA` contract in `fvm/evm/handler/coa/coa.sol` is documented and designed to support ERC-1271 signature validation. The contract:

1. Defines the ERC1271 magic value constant `ValidERC1271Signature = 0x1626ba7e`, which is also the ERC1271 interface ID (since ERC1271 has exactly one function).
2. Implements `isValidSignature(bytes32,bytes)` with the comment `// ERC1271 requirement`.
3. Exposes `isValidSignature` in the ABI (`coa_abi.json`). [1](#0-0) [2](#0-1) 

However, the `supportsInterface` function only declares support for `ERC1155TokenReceiver`, `ERC721TokenReceiver`, `ERC777TokensRecipient`, and `IERC165`. It does **not** include the ERC1271 interface ID: [3](#0-2) 

The protocol documentation in `contract.cdc` explicitly states that COAs support ERC-1271: [4](#0-3) 

### Impact Explanation
Any EVM contract that follows the ERC165 standard by calling `supportsInterface(0x1626ba7e)` before invoking `isValidSignature` on a COA address will receive `false` and refuse to proceed with COA-based signature verification. This silently breaks COA accounts as ERC1271 signers in all such contracts (e.g., multisigs, DAOs, NFT marketplaces, DeFi protocols that use ERC1271 for off-chain order signing). The COA's core value proposition — acting as a smart contract wallet bridging Cadence and EVM — is undermined for any EVM-native protocol that correctly checks interface support.

**Impact: Medium**

### Likelihood Explanation
Checking `supportsInterface` before calling `isValidSignature` is standard ERC1271 integration practice and is recommended by EIP-1271 itself. Any EVM contract deployed on Flow EVM that follows this pattern will silently fail to use COA as a signer. This is reachable by any unprivileged EVM transaction sender interacting with such a contract.

**Likelihood: Medium**

### Recommendation
Add the ERC1271 interface ID to `supportsInterface`. Since `ValidERC1271Signature` (`0x1626ba7e`) is already defined as a constant and equals the ERC1271 interface ID, the fix is:

```diff
 function supportsInterface(bytes4 id) external view virtual override returns (bool) {
     return
         id == type(ERC1155TokenReceiver).interfaceId ||
         id == type(ERC721TokenReceiver).interfaceId ||
         id == type(ERC777TokensRecipient).interfaceId ||
+        id == ValidERC1271Signature ||
         id == type(IERC165).interfaceId;
 }
```

### Proof of Concept
1. Deploy any EVM contract on Flow EVM that checks ERC1271 support before calling `isValidSignature`, e.g.:
```solidity
function verifyWithCOA(address coa, bytes32 hash, bytes memory sig) external view returns (bool) {
    // Standard ERC1271 guard — checks interface support first
    require(IERC165(coa).supportsInterface(0x1626ba7e), "ERC1271 not supported");
    return IERC1271(coa).isValidSignature(hash, sig) == 0x1626ba7e;
}
```
2. Call `verifyWithCOA` with a valid COA address.
3. The call reverts at the `require` because `COA.supportsInterface(0x1626ba7e)` returns `false`, even though `isValidSignature` is fully implemented and functional.
4. The COA cannot be used as an ERC1271 signer in this contract despite the protocol documentation guaranteeing ERC1271 support. [5](#0-4) [6](#0-5)
