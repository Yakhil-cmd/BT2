### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Freeze Staker Rewards - (File: `src/staking/staking.cairo`)

---

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` before the sequencer, permanently burning that block's reward slot and denying stakers their consensus-era block rewards.

---

### Finding Description
The protocol specification explicitly restricts `update_rewards` to the Starkware sequencer:

> `docs/spec.md` line 1644–1645:
> ```
> #### access control
> Only starkware sequencer.
> ```

However, the implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1507) performs no caller identity check whatsoever. The only guards present are:

1. `self.general_prerequisites()` — checks the contract is not paused.
2. `current_block_number > self.last_reward_block.read()` — prevents two calls in the same block.
3. Staker existence and activity checks.

There is no `assert_caller_is_sequencer`, no role check, and no `get_caller_address()` comparison against any privileged address.

The critical state mutation is `self.last_reward_block.write(current_block_number)` at line 1485, which is written **unconditionally** before the `disable_rewards` branch. Once written, any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

An attacker calls `update_rewards(staker_address, disable_rewards: true)` at the first transaction of any block, before the sequencer. The function:
1. Passes all checks (contract unpaused, new block, staker active).
2. Writes `last_reward_block = current_block_number` (line 1485).
3. Returns early at line 1487–1489 because `disable_rewards == true`, distributing **zero** rewards.

The sequencer's legitimate call in the same block then reverts with `REWARDS_ALREADY_UPDATED`. That block's rewards are permanently lost for the targeted staker. Repeated across every block and every staker, the attacker can permanently suppress all consensus-era block rewards across the entire protocol.

---

### Likelihood Explanation
**High.** The entry point is a public, permissionless external function. No tokens, no stake, no role, and no special knowledge are required. The attacker only needs to front-run the sequencer's transaction in each block, which is straightforward on Starknet where transaction ordering is observable in the mempool.

---

### Recommendation
Add a sequencer-only caller check at the top of `update_rewards`, consistent with the spec. For example, store the authorized sequencer address in contract storage and assert:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, use the existing roles framework (`self.roles.only_operator()` or a dedicated sequencer role) to enforce the access control that the specification mandates.

**Attack steps:**

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).

3. `last_reward_block` is set to `N`; no rewards are distributed.
4. Sequencer submits `update_rewards(victim_staker, disable_rewards: false)` — reverts with `REWARDS_ALREADY_UPDATED`.
5. Victim staker's block rewards for block `N` are permanently lost.
6. Repeat every block to permanently freeze all consensus block rewards.
