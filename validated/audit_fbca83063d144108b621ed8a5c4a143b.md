### Title
Any Single Malicious Participant Can Permanently Abort DKG/Reshare/Refresh via `broadcast_success` — (File: `src/dkg.rs`)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` is called unconditionally at the end of every DKG, reshare, and refresh protocol. It requires **all n participants** to broadcast `true`. A single malicious participant can broadcast `false` and cause all honest parties to permanently abort the protocol, denying key generation even when the threshold of honest parties is met and all cryptographic checks have passed.

---

### Finding Description

`broadcast_success` (lines 307–338) is called at line 531 of `do_keyshare`, which is the shared core of `do_keygen`, `do_reshare`, and the refresh path:

```rust
// Step 5.4 and Step 5.5
broadcast_success(&mut chan, &participants, me, session_id).await?;

// Return the key pair
Ok(KeygenOutput { ... })
```

Inside `broadcast_success`, every participant unconditionally broadcasts `(true, session_id)` via `do_broadcast`, and then the result is checked with a strict all-or-nothing predicate:

```rust
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant seems to have failed its checks. Aborting Protocol!"
    ));
}
``` [1](#0-0) [2](#0-1) 

A malicious participant running a modified client can broadcast `(false, session_id)` instead of `(true, session_id)`. Because `do_broadcast` is the echo-broadcast protocol (`reliable_broadcast_receive_all`), it guarantees **consistency**: all honest parties will converge on the same delivered value. Once the malicious `false` vote is delivered, every honest party hits the `all(|&(boolean, _)| boolean)` check, gets `false`, and returns `Err`. The `KeygenOutput` is never returned. [3](#0-2) 

There is no fallback, no mechanism to exclude the malicious participant, and no way to recover the already-computed shares. The protocol must be restarted from scratch — and the same attack can be repeated indefinitely.

This is structurally identical to the external report's root cause: a critical operation (`broadcast_success`) is called **unconditionally** on every DKG/reshare/refresh, it depends on **every participant's cooperation** as a single point of failure, and there is no recovery path when that dependency fails.

---

### Impact Explanation

Permanent denial of key generation, reshare, and refresh for all honest parties. A single malicious participant — even one within the protocol's stated tolerance of `f` malicious parties — can abort DKG/reshare/refresh at zero cost by broadcasting `false` in the final round. Because the echo-broadcast protocol guarantees all honest parties see the same value, the abort is total and guaranteed. This maps directly to the allowed High impact: *Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions.*

---

### Likelihood Explanation

Any participant who wishes to disrupt the protocol can do so trivially by modifying the final broadcast value. No special cryptographic knowledge, no key material, and no prior protocol state is required. The attack is cheap, reliable, and undetectable until the final round — after all secret shares have already been distributed to all participants. The attack is especially dangerous in reshare and refresh scenarios, where it can permanently prevent key rotation or participant set changes.

---

### Recommendation

Replace the all-or-nothing `broadcast_success` check with a threshold-based success check: require only a quorum (e.g., at least `t` of `n`) of participants to broadcast `true` before proceeding. Participants who broadcast `false` or fail to participate should be identified and excluded, allowing the protocol to complete with the remaining honest participants. This mirrors the fix suggested in the external report: add a fallback path so that a single failing component cannot block the entire protocol.

---

### Proof of Concept

1. `n=3` participants run DKG with `threshold=2`.
2. Participant P3 (malicious) modifies the protocol to broadcast `(false, session_id)` in the final `broadcast_success` round instead of `(true, session_id)`.
3. The echo-broadcast protocol (`do_broadcast`) ensures P1 and P2 both receive and accept `(false, session_id)` from P3.
4. P1 and P2 both evaluate `vote_list.iter().all(|&(boolean, _)| boolean)` → `false`, and return `Err(ProtocolError::AssertionFailed("A participant seems to have failed its checks. Aborting Protocol!"))`.
5. Neither P1 nor P2 receives a `KeygenOutput`. DKG is permanently aborted despite both honest parties having valid shares.
6. P3 can repeat this attack on every subsequent DKG, reshare, or refresh attempt at no additional cost. [4](#0-3)

pub async fn do_broadcast<'a, T>(
    chan: &mut SharedChannel,
    participants: &'a ParticipantList,
    me: Participant,
    data: T,
) -> Result<ParticipantMap<'a, T>, ProtocolError>
where
    T: Serialize + Clone + DeserializeOwned + PartialEq,
{
    let wait_broadcast = chan.next_waitpoint();
    let send_vote = reliable_broadcast_send(chan, wait_broadcast, participants, me, data)?;
    let vote_list =
        reliable_broadcast_receive_all(chan, wait_broadcast, participants, me, send_vote).await?;
    Ok(vote_list)
}
