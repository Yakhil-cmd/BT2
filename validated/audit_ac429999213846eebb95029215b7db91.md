**`register()` analysis** (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`, lines 201–214):

```solidity
function register(bytes32 feedId, address pool, address factory) external payable {
    require(msg.value >= registrationFee, ...);   // 1 wei default
    require(pool != address(0));
    require(approvedFactories.contains(factory), ...);
    require(IPoolFactory(factory).isPool(pool), ...);

    if (blacklisted[pool]) {
        blacklisted[pool] = false;          // ← unconditional clear
        emit BlacklistUpdated(pool, false);
    }

    registeredPool[feedId][pool] = true;    // ← any feedId accepted
    emit PoolRegistered(feedId, pool, msg.sender, msg.value);
}
```

Key observations:
- No role check — fully permissionless
- No check that `feedId` is meaningful or that the pool was previously registered for it
- Blacklist is per-pool (`mapping(address => bool) public blacklisted`), not per-feedId
- `registrationFee` defaults to `1 wei`

The `price()` function enforces both `!blacklisted[pool]` and `registeredPool[feedId][pool]`, so the blacklist is the ADMIN's only mechanism to cut off a pool from oracle reads. [1](#0-0) 

`setBlacklist()` is `onlyRole(ADMIN_ROLE)`, making it an explicit oracle-admin security control. [2](#0-1) 

The bypass path is real and costs 1 wei. This is a genuine admin-boundary break where an unprivileged path overrides an `ADMIN_ROLE` security control.

### Title
Permissionless `register()` unconditionally clears ADMIN blacklist, allowing any blacklisted pool to restore oracle read access for 1 wei — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

### Summary
`OracleBase.register()` is intentionally permissionless but contains an unconditional blacklist-clearing side-effect. Because the blacklist is keyed per-pool (not per-feedId), registering a pool for any feedId — including a brand-new one — silently removes the ADMIN's blacklist entry. The registration fee defaults to 1 wei, making the bypass essentially free.

### Finding Description
`setBlacklist(pool, true)` is an `ADMIN_ROLE`-gated function intended to permanently revoke a pool's ability to call `price(feedId, pool)`. [2](#0-1) 

However, `register()` contains:

```solidity
if (blacklisted[pool]) {
    blacklisted[pool] = false;
    emit BlacklistUpdated(pool, false);
}
registeredPool[feedId][pool] = true;
``` [3](#0-2) 

There is no check that the caller holds any role, no check that the pool was previously registered for the supplied `feedId`, and no check that the `feedId` is a known/active feed. Any caller who can supply a valid `factory`+`pool` pair (both of which are public on-chain) can invoke `register()` with an arbitrary `feedId` and clear the blacklist.

### Impact Explanation
The blacklist is the oracle's only runtime mechanism to cut off a misbehaving or abusive pool from price reads. Once bypassed, the pool can call `price(feedId, pool)` again through its provider, receiving live oracle prices. This breaks the ADMIN's ability to enforce abuse protection and constitutes an admin-boundary break: an `ADMIN_ROLE` security control is overridden by an unprivileged path. Any downstream swap or liquidity action that depends on the oracle price is then exposed to a pool the ADMIN had explicitly revoked.

### Likelihood Explanation
- `register()` is public and payable with no role gate.
- The default `registrationFee` is `1 wei`. [4](#0-3) 
- The pool address and factory address are both discoverable on-chain from factory events.
- The `feedId` argument is unconstrained — any `bytes32` value works.
- The pool operator being blacklisted has direct economic incentive to execute the bypass.

### Recommendation
Remove the blacklist-clearing logic from `register()`, or gate it behind `ADMIN_ROLE`. If the intent is that paying the fee rehabilitates a pool, that decision should require explicit ADMIN approval, not be a silent side-effect of a permissionless call. A minimal fix:

```solidity
// Remove these lines from register():
// if (blacklisted[pool]) {
//     blacklisted[pool] = false;
//     emit BlacklistUpdated(pool, false);
// }
```

Additionally, add a check that `!blacklisted[pool]` before allowing registration to proceed, so blacklisted pools cannot even register for new feedIds without ADMIN intervention.

### Proof of Concept
```solidity
// 1. Pool operator registers pool for feedId1
oracle.register{value: 1 wei}(feedId1, pool, factory);
// registeredPool[feedId1][pool] == true, blacklisted[pool] == false

// 2. ADMIN blacklists the pool (e.g., for abuse)
oracle.setBlacklist(pool, true);
// blacklisted[pool] == true — pool cannot call price()

// 3. Pool operator (or anyone) registers pool for any feedId2
oracle.register{value: 1 wei}(feedId2, pool, factory);
// blacklisted[pool] == false  ← ADMIN blacklist silently cleared
// registeredPool[feedId2][pool] == true

// 4. Pool can now call price(feedId2, pool) again — blacklist fully bypassed
``` [5](#0-4)
