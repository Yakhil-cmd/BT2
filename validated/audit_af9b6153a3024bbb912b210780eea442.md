### Title
Rate Limit Token Consumed Before Signature Verification Enables Arbitrary Payer Quota Exhaustion DoS - (File: `access/validator/validator.go`)

---

### Summary

`TransactionValidator.initValidationSteps` places `checkRateLimitPayer` as the **first** validation step, before any signature verification. `AddressRateLimiter.IsRateLimited` calls `limiter.Allow()`, which is a **destructive token consumption**. Because Collection/Access Nodes explicitly do not perform cryptographic signature verification at ingestion time, any unprivileged sender can submit a transaction with an arbitrary victim address as `Payer` and a fake-but-format-valid ECDSA signature, causing the victim's rate-limit token to be consumed without the attacker controlling the victim's key.

---

### Finding Description

`initValidationSteps` in `access/validator/validator.go` defines the validation pipeline as:

```
checkRateLimitPayer      ← step 1: consumes rate-limit token
checkTxSizeLimit
checkMissingFields
checkGasLimit
checkExpiry
checkCanBeParsed
checkAddresses
checkSignatureFormat     ← step 8: only checks ECDSA byte format
checkAccounts            ← step 9: only checks presence of sig.Address == tx.Payer
``` [1](#0-0) 

`checkRateLimitPayer` calls `v.limiter.IsRateLimited(tx.Payer)`: [2](#0-1) 

`AddressRateLimiter.IsRateLimited` calls `limiter.Allow()`, which is a **destructive read** — it consumes one token from the token bucket regardless of whether the transaction is ultimately valid: [3](#0-2) 

The code itself documents that signature verification is not performed at this layer:

> `NOTE: This validation logic is a simplified interim approach: Collection/Access Nodes cannot reliably validate transaction signatures or payer balance.` [4](#0-3) 

`checkSignatureFormat` only validates that the raw bytes are a valid ECDSA encoding (P-256 or secp256k1), not that the signature is cryptographically bound to the transaction or the payer's public key: [5](#0-4) 

`checkAccounts` only checks that `tx.EnvelopeSignatures` contains **any** entry whose `Address` field equals `tx.Payer` — it does not verify the signature cryptographically: [6](#0-5) 

The Collection Node's ingest engine wires the `AddressRateLimiter` directly into `NewTransactionValidatorWithLimiter`: [7](#0-6) 

The `AddressRateLimiter` is instantiated at node startup with operator-configured rate and burst values: [8](#0-7) 

---

### Impact Explanation

The `AddressRateLimiter` is designed to allow a configured burst of transactions from a rate-limited payer address before blocking further ones. An attacker who knows (or discovers by probing) which addresses are in the rate-limiter can exhaust the entire burst allowance for a victim address by submitting crafted transactions with:

- `tx.Payer = victim_address`
- `tx.EnvelopeSignatures[0].Address = victim_address` with any valid-format ECDSA bytes (e.g., the attacker's own signature on a different message)

Each such submission causes `IsRateLimited` → `limiter.Allow()` to consume one token. Once the burst is exhausted, the victim's legitimate transactions are rejected with `InvalidTxRateLimitedError` for the remainder of the rate-limit window, preventing them from transacting on the network.

---

### Likelihood Explanation

The attack entry point is the `PushTransactions` network channel, reachable by any peer that can connect to a Collection Node. The attacker does not need the victim's private key — only a format-valid ECDSA byte string (their own signature suffices). Discovering which addresses are in the rate-limiter is possible by submitting probe transactions and observing whether the `InvalidTxRateLimitedError` is returned. The burst window resets on a token-bucket schedule, so the attack must be repeated periodically, but this is trivially automatable.

---

### Recommendation

Move `checkRateLimitPayer` to **after** `checkSignatureFormat` and `checkAccounts` in the `initValidationSteps` pipeline, so that the rate-limit token is only consumed for transactions that at minimum carry a structurally valid signature from the declared payer address. Alternatively, change `IsRateLimited` to use a **non-destructive peek** (e.g., `limiter.Tokens() < 1`) for the initial check, and only call `limiter.Allow()` (consuming the token) after signature verification passes.

---

### Proof of Concept

```
1. Operator adds victim_address to the AddressRateLimiter with burst=5.

2. Attacker constructs a transaction:
     tx.Payer = victim_address
     tx.EnvelopeSignatures = [{Address: victim_address, Signature: <attacker's own valid ECDSA bytes>}]
     tx.Script, GasLimit, ReferenceBlockID = any valid values

3. Attacker sends this transaction 5 times to the Collection Node via PushTransactions.

4. Each time, initValidationSteps runs:
     - checkRateLimitPayer: IsRateLimited(victim_address) → limiter.Allow() consumes 1 token → returns false (not yet limited)
     - checkSignatureFormat: ECDSA bytes are valid format → passes
     - checkAccounts: EnvelopeSignatures contains victim_address → passes
     → Transaction accepted into mempool (will fail at execution, but token is gone)

5. On the 6th call, limiter.Allow() returns false → IsRateLimited returns true → checkRateLimitPayer returns InvalidTxRateLimitedError.

6. Victim now submits a legitimate transaction signed with their actual key:
     → checkRateLimitPayer: IsRateLimited(victim_address) → bucket empty → returns InvalidTxRateLimitedError
     → Transaction rejected. Victim cannot transact until the token bucket refills.
```

func (v *TransactionValidator) checkSignatureFormat(tx *flow.TransactionBody) error {
	for _, signature := range tx.PayloadSignatures {
		valid, _ := signature.ValidateExtensionDataAndReconstructMessage(tx.PayloadMessage())
		if !valid {
			return InvalidAuthenticationSchemeFormatError{Signature: signature}
		}
	}

	for _, signature := range tx.EnvelopeSignatures {
		valid, _ := signature.ValidateExtensionDataAndReconstructMessage(tx.EnvelopeMessage())
		if !valid {
			return InvalidAuthenticationSchemeFormatError{Signature: signature}
		}
	}

	for _, signature := range append(tx.PayloadSignatures, tx.EnvelopeSignatures...) {
		// check the format of the signature is valid.
		// a valid signature is an ECDSA signature of either P-256 or secp256k1 curve.
		ecdsaSignature := signature.Signature

		// check if the signature could be a P-256 signature
		valid, err := crypto.SignatureFormatCheck(crypto.ECDSAP256, ecdsaSignature)
		if err != nil {
			return fmt.Errorf("could not check the signature format (%s): %w", signature, err)
		}
		if valid {
			continue
		}

		// check if the signature could be a secp256k1 signature
		valid, err = crypto.SignatureFormatCheck(crypto.ECDSASecp256k1, ecdsaSignature)
		if err != nil {
			return fmt.Errorf("could not check the signature format (%s): %w", signature, err)
		}
		if valid {
			continue
		}

		return InvalidRawSignatureError{Signature: signature}
	}

	return nil
}

// IsRateLimited returns whether the given address should be rate limited
func (r *AddressRateLimiter) IsRateLimited(address flow.Address) bool {
	r.mu.RLock()
	limiter, ok := r.limiters[address]
	r.mu.RUnlock()

	if !ok {
		return false
	}

	rateLimited := !limiter.Allow()
	return rateLimited
}
