### Title
Unverified Oracle-Sourced `native_price` / `pubdata_price` Not Committed to Public Input Enables Forward/Proving Divergence in Resource Accounting - (`zk_ee/src/system/metadata/zk_metadata.rs`, `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/public_input.rs`)

---

### Summary

`BlockMetadataFromOracle` carries `native_price` and `pubdata_price` that are fetched from the oracle at block start and used to derive every transaction's native resource limit and pubdata charge rate. Neither field is included in the ZK proof's public input (`BatchOutput` / `ChainStateCommitment`). A prover-controlled oracle can therefore supply values that differ from those used by the sequencer during forward execution. Because the settlement layer verifies only the committed public input, it cannot detect the divergence, allowing a malicious prover to include transactions that the sequencer rejected (or vice-versa) and to bypass the economic model that ensures proving costs are covered.

---

### Finding Description

**Step 1 â€“ Oracle provides pricing without commitment.**

`BlockMetadataFromOracle` is deserialized directly from the oracle response at block initialization: [1](#0-0) 

The struct carries `native_price` and `pubdata_price` alongside other block fields: [2](#0-1) 

**Step 2 â€“ These values drive all native-resource accounting.**

During L2 transaction validation, `native_per_gas` is computed as `ceil(gas_price / native_price)` and `native_per_pubdata` as `pubdata_price / native_price`. These two ratios set the transaction's entire native resource budget: [3](#0-2) 

**Step 3 â€“ Neither field appears in the public input.**

`BatchOutput.hash()` commits: `chain_id`, timestamps, DA commitment, L1/L2 tx counts, priority ops hash, L2 logs root, upgrade tx hash, interop roots hash, and settlement layer chain id. `native_price` and `pubdata_price` are absent: [4](#0-3) 

`ChainStateCommitment.hash()` commits: state root, next free slot, block number, last-256-block-hashes blake, and last block timestamp. Again, `native_price` and `pubdata_price` are absent: [5](#0-4) 

**Step 4 â€“ The project's own documentation acknowledges the gap.**

> "The block header should determine the block fully, i.e. include all the inputs needed to execute the block. Currently it misses `gas_per_pubdata` and `native_price`, but we already working on design and implementation to solve this issue." [6](#0-5) 

The `extra_data` field is listed as "TBD, possibly gas_per_pubdata and native price", confirming neither is currently committed: [7](#0-6)

### Impact Explanation

**Forward/proving divergence with resource accounting consequences.**

Because `native_price` and `pubdata_price` are oracle-injected inputs that influence which transactions pass validation and how much native resource they consume, but are not bound to the public input, a prover can substitute arbitrary values:

1. **Lowering `native_price`** (e.g., from 10 to 1) raises `native_per_gas` by 10Ã—. Transactions that the sequencer rejected with `NativeResourcesAreTooExpensive` or `OutOfNativeResourcesDuringValidation` now pass validation in the proving execution. The prover can include these transactions in a valid proof that the settlement layer accepts, even though the sequencer never included them.

2. **Setting `pubdata_price = 0`** makes `native_per_pubdata = 0`. Transactions that write large amounts of pubdata are no longer charged native resources for it. The prover can include storage-heavy transactions that would have been reverted post-execution due to insufficient native resources for pubdata.

3. **Raising `native_price`** to an extreme value makes `native_per_gas â‰ˆ 0`, granting every transaction an effectively unlimited native resource budget (`u64::MAX - 1`), bypassing the native resource cap entirely.

In all cases the settlement layer sees a valid proof over a valid state transition â€” it simply cannot verify that the pricing parameters used during proving matched those used during sequencing, because they are not part of the public input.

---

### Likelihood Explanation

The entry path is the **prover/forward execution input**, which the scope explicitly lists as an attacker-controlled vector. The prover supplies oracle responses via the CSR-based oracle (`CsrBasedIOOracle`): [8](#0-7) 

The oracle documentation itself states: *"Oracle responses are non-deterministic and MUST be treated as untrusted input. All responses should be validated against additional constraints."* [9](#0-8) 

`native_price` and `pubdata_price` are deserialized from the oracle stream without any cross-check against a committed value, violating this requirement. The only validation performed is a non-zero check on `native_price`: [10](#0-9) 

No upper-bound or commitment check exists.

---

### Recommendation

Include `native_price`, `pubdata_price`, and `pubdata_limit` in the committed public input. The most direct fix is to add them to `BatchOutput` (and its `hash()` function) so the settlement layer can verify they match the values agreed upon off-chain. Alternatively, include them in the block header's `extra_data` field (as the documentation already contemplates) so they flow into the block hash, which is part of the pubdata commitment. [11](#0-10) 

---

### Proof of Concept

1. Sequencer runs forward execution with `native_price = 10`, `pubdata_price = 1000`. A transaction with `max_fee_per_gas = 100` gets `native_per_gas = ceil(100/10) = 10`. With `gas_limit = 50_000`, `native_limit = 500_000`. The transaction is rejected because its intrinsic native cost exceeds this limit.

2. Prover runs proving execution with the same transaction set but supplies `native_price = 1` via the CSR oracle. Now `native_per_gas = 100`, `native_limit = 5_000_000`. The transaction passes validation and is included in the proven block.

3. The prover submits the proof. `BatchPublicInput.hash()` covers `state_before`, `state_after`, and `batch_output.hash()`. None of these fields encode `native_price`. The settlement layer verifies the proof successfully.

4. The state transition is accepted on L1 even though it includes a transaction the sequencer explicitly rejected, breaking the economic invariant that native resource consumption is bounded by the user's fee payment. [12](#0-11) [13](#0-12)

impl<NDS: NonDeterminismCSRSourceImplementation> IOOracle for CsrBasedIOOracle<NDS> {
    type RawIterator<'a> = CsrBasedIOOracleIterator<NDS>;

    fn raw_query<'a, I: UsizeSerializable + UsizeDeserializable>(
        &'a mut self,
        query_type: u32,
        input: &I,
    ) -> Result<Self::RawIterator<'a>, InternalError> {
        const {
            assert!(core::mem::size_of::<usize>() == core::mem::size_of::<u32>());
        }
        NDS::csr_write_impl(query_type as usize);
        let iter_to_write = UsizeSerializable::iter(input);
        // write length
        let iterator_len = iter_to_write.len();
        assert!(iterator_len == <I as UsizeSerializable>::USIZE_LEN);
        NDS::csr_write_impl(iterator_len);
        // write content
        let mut remaining_len = iterator_len;
        for value in iter_to_write {
            assert!(remaining_len != 0);
            NDS::csr_write_impl(value);
            remaining_len -= 1;
        }
        assert!(remaining_len == 0);
        // we can expect that length of the result is returned via read
        let remaining_len = NDS::csr_read_impl();
        let it = CsrBasedIOOracleIterator::<NDS> {
            remaining: remaining_len,
            _marker: core::marker::PhantomData,
        };

        Ok(it)
    }
