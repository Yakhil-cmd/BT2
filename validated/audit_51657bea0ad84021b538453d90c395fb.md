### Title
Hardcoded `L1_TX_NATIVE_PRICE` May Cause All L1→L2 Priority Transactions to Fail When Proving Costs Change — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` in the bootloader constants and is explicitly marked with a `TODO` acknowledging the value is a placeholder. This constant is the sole divisor used to derive the native resource budget for every L1→L2 priority transaction. For L2 transactions the equivalent value is operator-configurable at block-context time. If actual RISC-V proving costs diverge from what this constant implies — due to a protocol upgrade, prover change, or new opcode costs — all L1→L2 transactions will exhaust their native resource budget and revert, effectively breaking the L1→L2 bridge.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant is defined as:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

In `prepare_and_check_resources` inside `process_l1_transaction.rs`, this constant is unconditionally substituted for the operator-supplied `native_price` that L2 transactions use:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = ...
    u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [2](#0-1) 

The resulting `native_per_gas` drives `native_prepaid_from_gas = native_per_gas * gas_limit`, which is the entire native resource budget for the transaction.

For L2 transactions, the equivalent path reads the operator-configurable value from the block oracle:

```rust
let native_price = system.get_native_price();
...
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(
    TxError::Validation(InvalidTransaction::NativeResourcesAreTooExpensive),
)?
``` [3](#0-2) 

The operator-configurable `native_price` flows from `BlockMetadataFromOracle.native_price` through `ZkSpecificPricingMetadata::native_price()`: [4](#0-3) 

There is no equivalent runtime-adjustable path for L1→L2 transactions. The hardcoded `10` is the only divisor ever used.

---

### Impact Explanation

If the actual RISC-V proving cost per EVM gas unit rises above `1/10` (i.e., the true `native_price` the operator would set is greater than `10`), every L1→L2 priority transaction will exhaust its native resource budget mid-execution and revert. The deposit is refunded to the L2 refund recipient rather than permanently lost, but:

1. The L1→L2 bridge becomes non-functional for all users simultaneously.
2. Users who submitted L1→L2 transactions expecting to receive L2 tokens receive only a refund; their L1 funds remain locked in the L1 bridge contract with no path back to L1 without a separate L1-side claim mechanism.
3. The condition cannot be resolved without a code change and redeployment of ZKsync OS, since the constant is compiled in.

The TODO comment in the source code explicitly acknowledges the value is not finalized, confirming this is a known open risk.

---

### Likelihood Explanation

**Medium.** The TODO comment `// TODO (EVM-1157): find a reasonable value for it.` is a direct admission that the value is a placeholder. Any protocol upgrade that changes the RISC-V prover, adds new opcodes, or adjusts native cost constants (all of which are in active development) could shift the true `native_price` above or below `10`. The existing test `test_l1_tx_low_ratio` already demonstrates that L1 transactions with a gas_price/native_price ratio below a threshold fail, confirming the mechanism is live. [5](#0-4) 

---

### Recommendation

Replace the hardcoded `L1_TX_NATIVE_PRICE` with the operator-supplied `native_price` from the block context, the same source used for L2 transactions. The stated rationale for the constant — "to avoid censorship" — can be preserved by enforcing a minimum floor (e.g., `native_price.max(MIN_L1_TX_NATIVE_PRICE)`) rather than ignoring the operator value entirely. This mirrors the first mitigation option in the original report: allow the parameter to be set externally while the protocol front-end recommends a safe value.

```rust
// Use operator-supplied native_price with a minimum floor to prevent censorship
let native_price = system.get_native_price().max(MIN_L1_TX_NATIVE_PRICE);
```

---

### Proof of Concept

1. Deploy ZKsync OS with a RISC-V prover whose actual cost per EVM gas unit corresponds to a `native_price` of, say, `100` (10× the hardcoded value).
2. Submit an L1→L2 priority transaction with `gas_price = 1000`, `gas_limit = 100_000`.
3. `native_per_gas = ceil(1000 / 10) = 100`; `native_limit = 100 * 100_000 = 10_000_000`.
4. The actual proving cost at `native_price = 100` would require `native_limit = 100_000_000` — 10× more than allocated.
5. The transaction exhausts native resources, reverts, and the deposit is refunded to the L2 refund recipient.
6. Every subsequent L1→L2 transaction under the same conditions fails identically, breaking the bridge.
