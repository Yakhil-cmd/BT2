### Title
Integer Division Truncation in `native_per_pubdata` Silently Zeroes Pubdata Charges When `pubdata_price < native_price` — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the operator-set `pubdata_price` is less than `native_price`, the integer floor division used to compute `native_per_pubdata` truncates to zero. Because every downstream pubdata-charging call multiplies by this zero value, the transaction is charged **nothing** for pubdata regardless of how many state-diff bytes it produces. An unprivileged sender who observes this block-context condition can spam storage writes and force the operator to absorb the full L1 publication cost.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, the per-pubdata-byte native cost is derived as:

```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is floor integer division. Whenever `pubdata_price < native_price` (e.g. `pubdata_price = 5`, `native_price = 10`), the result is `0`. No error is raised; the zero propagates silently.

This zero then flows into every pubdata-charging site:

**1. Intrinsic pubdata overhead (resource creation)**

```rust
// gas_helpers.rs line 352
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
```

With `native_per_pubdata_byte = 0`, `intrinsic_pubdata_overhead = 0` — no native resources are reserved for the mandatory intrinsic pubdata bytes.

**2. Execution pubdata charge**

```rust
// gas_helpers.rs lines 430-431
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```

With `native_per_pubdata = 0`, `native = 0` — the entire execution pubdata cost is waived.

**3. Post-execution pubdata sufficiency check**

```rust
// gas_helpers.rs line 454
let enough = resources.has_enough(&resources_for_pubdata);
```

`resources_for_pubdata` is zero, so this check always passes, regardless of how many pubdata bytes were produced.

The same truncation is present in the API helper used by external tooling:

```rust
// api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

---

### Impact Explanation

When `pubdata_price < native_price`, any L2 transaction can produce an arbitrarily large number of state-diff bytes (e.g. by writing to many distinct storage slots) and pay **zero** native resources for them. The operator must still publish those bytes to the settlement layer and bears the full L1 data-availability cost. This is a direct, quantifiable financial loss to the operator proportional to the pubdata generated per block.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a plausible operator configuration. For example, if `native_price` is set to reflect RISC-V proving cost (e.g. 1000 units) and `pubdata_price` is set to a lower value reflecting current L1 calldata prices (e.g. 500 units), the operator intends to charge 0.5 native per pubdata byte but the floor division silently produces 0. There is no validation guard that rejects or warns when `pubdata_price > 0` yet `native_per_pubdata == 0`. An attacker monitoring the block context can detect this condition and immediately exploit it.

---

### Recommendation

After computing `native_per_pubdata`, add a guard that enforces a minimum of 1 when `pubdata_price` is non-zero:

```rust
let native_per_pubdata = {
    let raw = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
        .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
    if !pubdata_price.is_zero() && raw == 0 {
        1u64  // floor to minimum representable charge
    } else {
        raw
    }
};
```

Alternatively, use ceiling division (`div_ceil`) instead of floor division so that any non-zero `pubdata_price` always produces at least 1, consistent with how `native_per_gas` is computed:

```rust
// native_per_gas already uses div_ceil (line 135):
u256_try_to_u64(&gas_price.div_ceil(native_price))
// native_per_pubdata should match:
u256_try_to_u64(&pubdata_price.div_ceil(native_price))
```

---

### Proof of Concept

**Setup**: Deploy a contract that writes to 100 distinct storage slots in a single call. Set block context with `native_price = 1000`, `pubdata_price = 999`.

**Step 1** — `native_per_pubdata` truncates to zero:
```
native_per_pubdata = 999 / 1000 = 0  (floor division)
```

**Step 2** — Intrinsic pubdata overhead is zero:
```rust
intrinsic_pubdata_overhead = 0 * L2_TX_INTRINSIC_PUBDATA = 0
```

**Step 3** — Execution pubdata charge is zero:
```rust
native = 100_storage_slots * 32_bytes * 0 = 0
```

**Step 4** — Post-execution check passes trivially:
```rust
resources.has_enough(&Resources::from_native(0)) == true  // always
```

**Result**: The transaction succeeds, 100 storage slots are written (generating ~3200 bytes of pubdata), and the sender pays nothing for pubdata. The operator must publish those bytes to L1 at their own expense. Repeating this across many transactions in a block drains the operator's L1 budget.
