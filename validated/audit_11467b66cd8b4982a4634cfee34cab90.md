### Title
Missing Lower-Bound Validation on `threshold` in FROST Sign Inputs — (File: `src/frost/mod.rs`)

### Summary
The `assert_sign_inputs` function and the `presign` function in `src/frost/mod.rs` validate the `threshold` parameter against an upper bound (`threshold <= participants.len()`) but omit the lower-bound check (`threshold >= 2`). A malicious coordinator or misconfigured caller can supply `threshold = 1`, bypassing the intended threshold security property. The DKG initialization path enforces this lower bound correctly, creating an inconsistency that allows invalid threshold values to reach the FROST signing protocol, causing the signing round to produce an unusable output and denying honest participants a valid signature.

### Finding Description

In `src/dkg.rs`, `assert_key_invariants` correctly enforces both bounds on `threshold`:

```rust
// src/dkg.rs lines 580-582 — correct lower-bound guard
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

Likewise, `validate_triple_inputs` in the OT-based triple generation path enforces the same lower bound: [2](#0-1) 

However, `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs lines 145-150 — upper-bound only, no lower-bound check
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
// ← no check: threshold.value() < 2
``` [3](#0-2)
