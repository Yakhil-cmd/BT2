Audit Report

## Title
Untrusted Gzip Footer Inflates Compilation Cost, Exhausting Round Instruction Budget - (`rs/execution_environment/src/hypervisor.rs`)

## Summary
`Hypervisor::create_execution_state` reads the uncompressed size from the attacker-controlled gzip footer via `decoded_wasm_size()` and uses it — uncapped — to compute `compilation_cost`. When `wasm_executor.create_execution_state` subsequently rejects the module as `ModuleTooLarge`, the error branch charges the inflated cost (up to `6,000 × 4,294,967,295 ≈ 25.7 trillion instructions`) against the shared `round_limits.instructions` budget. A single malicious `install_code` call with a 4-byte-modified gzip footer can exhaust the entire round instruction budget, starving every other canister scheduled in that round.

## Finding Description

**Root cause — uncapped gzip footer read:**

`decoded_wasm_size` in `rs/embedders/src/wasm_utils/decoding.rs` reads the last 4 bytes of the gzip blob as a `u32` and returns it as `usize` with no upper-bound check:

```rust
isize_bytes.copy_from_slice(&module_bytes[module_bytes.len() - 4..module_bytes.len()]);
let uncompressed_size = u32::from_le_bytes(isize_bytes) as usize;
return Ok((WasmEncoding::Gzip, uncompressed_size));
``` [1](#0-0) 

The function's own documentation explicitly warns this value is untrusted: [2](#0-1) 

**Inflated cost computed from untrusted size:**

`Hypervisor::create_execution_state` uses the raw footer value to compute `compilation_cost` with `DEFAULT_COST_TO_COMPILE_WASM_INSTRUCTION = 6,000` instructions per byte:

```rust
let wasm_size_result = decoded_wasm_size(canister_module.as_slice());
let wasm_size = match wasm_size_result {
    Ok(size) => std::cmp::max(size, canister_module.len()),
    Err(_) => canister_module.len(),
};
let compilation_cost = self.cost_to_compile_wasm_instruction * wasm_size as u64;
``` [3](#0-2) [4](#0-3) 

**Error path charges inflated cost to shared round budget:**

`wasm_executor.create_execution_state` calls `decode_wasm` internally, which enforces `wasm_max_size`:

```rust
if uncompressed_size as u64 > max_size.get() {
    return Err(WasmValidationError::ModuleTooLarge { ... });
}
``` [5](#0-4) 

This causes the executor to return `Err`, and the error branch in `create_execution_state` charges the inflated estimate — not the actual cost — to the shared round budget:

```rust
Err(err) => {
    let total_cost = self.create_execution_state_base_cost + compilation_cost;
    round_limits.instructions -= as_round_instructions(total_cost);
    (total_cost, Err(err))
}
``` [6](#0-5) 

The success path uses the actual compilation cost returned by the executor and is unaffected. Only the error path uses the untrusted estimate.

**Existing guards are insufficient:** The `wasm_max_size` check in `decode_wasm` correctly rejects the oversized module, but this rejection is what *triggers* the vulnerable error branch. The guard that should exist — capping `wasm_size` before computing `compilation_cost` — is absent.

## Impact Explanation

`round_limits.instructions` is the shared instruction budget for an entire execution round on a subnet. With `cost_to_compile_wasm_instruction = 6,000` and a footer claiming `u32::MAX = 4,294,967,295` bytes, `compilation_cost ≈ 25.7 trillion instructions` — roughly 3,600× a typical round budget of ~7 billion instructions. A single malicious `install_code` call exhausts the round budget, causing all other canisters scheduled later in the same round to receive zero instructions and be skipped. The attack is repeatable every round, constituting a sustained subnet-wide denial of service. This matches the **High** impact category: application/platform-level DoS with subnet availability impact, not based on raw volumetric DDoS.

## Likelihood Explanation

Any canister developer with enough cycles to cover the normal `install_code` prepayment (bounded by the message instruction limit, not the inflated estimate) can execute this attack. The only technical requirement is overwriting 4 bytes of a valid gzip file with `\xFF\xFF\xFF\xFF`. No privileged access, social engineering, or threshold corruption is required. The existing test `test_decode_large_compressed_module_with_tweaked_size` already demonstrates that footer manipulation is trivial and causes `decode_wasm` to fail: [7](#0-6) 

## Recommendation

In `Hypervisor::create_execution_state`, cap `wasm_size` to the actual compressed module length (which is bounded by the ingress message size limit) before computing `compilation_cost`. The compressed size is attacker-controlled only within the ingress message size limit, making it a safe upper bound:

```rust
// Cap to compressed size to avoid trusting the untrusted gzip footer.
let safe_wasm_size = canister_module.len();
let compilation_cost = self.cost_to_compile_wasm_instruction * safe_wasm_size as u64;
```

Alternatively, apply the same `wasm_max_size` cap used in `decode_wasm` to the value returned by `decoded_wasm_size` before using it to compute `compilation_cost`, mirroring the existing guard.

## Proof of Concept

1. Take any minimal valid Wasm (`\x00asm\x01\x00\x00\x00`, 8 bytes).
2. Gzip-compress it with any standard tool to produce a valid `.gz` file.
3. Overwrite the last 4 bytes with `\xFF\xFF\xFF\xFF` (little-endian `u32::MAX`).
4. Submit the modified blob as the `wasm_module` field of an `install_code` ingress message to any canister on the target subnet.
5. `decoded_wasm_size` returns `Ok(4_294_967_295)`.
6. `compilation_cost = 6_000 × 4_294_967_295 ≈ 25.7 trillion instructions`.
7. `wasm_executor.create_execution_state` calls `decode_wasm`, which returns `ModuleTooLarge` (4 GiB > `wasm_max_size`).
8. The error branch reduces `round_limits.instructions` by `compilation_cost`, exhausting the entire round budget.
9. All other canisters scheduled in the same round are starved of instructions.

A deterministic integration test can be written using `ExecutionTestBuilder` with a manipulated gzip module, asserting that `round_limits.instructions` after the failed `create_execution_state` call is not less than the round budget minus a cost proportional to the *compressed* module size.
