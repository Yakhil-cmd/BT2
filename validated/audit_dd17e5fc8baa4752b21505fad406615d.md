### Title
WalletConnect Pair Bypass Permissions Not Revoked on Wallet Key Deletion - (File: `packages/core/src/screens/SelectKey/WalletDeleteDialog.tsx`)

### Summary
When a user deletes a wallet key (fingerprint), the WalletConnect pair records stored in `dapp-pairs.yaml` — including "always allow" bypass permissions for sensitive spending commands — are not revoked. If the user later re-imports the same mnemonic (same fingerprint), every previously connected dApp automatically regains its bypass permissions and can execute commands such as `chia_sendTransaction` and `chia_takeOffer` without any user confirmation, silently bypassing the WalletConnect approval gate.

### Finding Description
The `handleSubmit` function in `WalletDeleteDialog.tsx` performs two cleanup steps on key deletion:

1. `deleteKey({ fingerprint })` — removes the key from the daemon keyring.
2. `removeFingerprintPrefs()` — removes UI preferences (`fingerprintSettings`, `sortedWallets`). [1](#0-0) 

Neither step touches the WalletConnect pair store. Pairs are persisted to `dapp-pairs.yaml` by the main process via `pairStore.ts`, completely independently of the wallet key lifecycle. [2](#0-1) 

Each `PairRecord` carries a `fingerprint`, a list of granted `commands`, and a `bypass` array — the "always allow without confirmation" list. [3](#0-2) 

The runtime authorization gate in `dispatchPairRequest.ts` only blocks execution when the currently logged-in fingerprint does not match `pair.fingerprint`: [4](#0-3) 

This means:
- While the key is absent, the pair is dormant (no fingerprint is logged in that matches).
- The moment the user re-imports the same mnemonic (identical fingerprint), every pair for that fingerprint becomes live again — with all previously granted bypass permissions intact — without any re-approval dialog.

The bypass list can include high-impact commands such as `chia_sendTransaction`, `chia_takeOffer`, `chia_cancelOffer`, and `chia_transferNFT`, all of which are defined as bypassable in `Commands.ts`. [5](#0-4) 

The `removeFingerprintPrefs` function in `WalletDeleteDialog.tsx` is the only cleanup performed, and it is entirely UI-layer: [6](#0-5) 

There is no call to `window.permissionsAPI.revokePair()` or any equivalent for pairs matching the deleted fingerprint. [7](#0-6) 

### Impact Explanation
A dApp that was previously granted bypass permissions for `chia_sendTransaction` (or other spending commands) can, after the user re-imports the same key, silently drain funds or accept/cancel offers without triggering any confirmation dialog. The user's act of deleting the key — which a reasonable user interprets as revoking all third-party access — does not actually revoke WalletConnect authorization. This is a bypass of the WalletConnect approval mechanism with direct asset-loss impact (unauthorized spend, offer acceptance, NFT transfer).

### Likelihood Explanation
Key deletion followed by re-import of the same mnemonic is a common user action: backup restoration, "fresh start" after a sync issue, or migrating between machines. The scenario requires no attacker capability beyond having previously been a legitimately connected dApp. The dApp does not need to do anything special — it simply waits for the fingerprint to reappear.

### Recommendation
In `WalletDeleteDialog.tsx` `handleSubmit`, after `deleteKey` succeeds, enumerate all WalletConnect pairs whose `fingerprint` matches the deleted key and revoke them via `window.permissionsAPI.revokePair(topic)` for each. The same cleanup should be applied in the "Delete All Keys" flow in `SelectKey.tsx`. The `PermissionsService` already exposes `revokePair` for exactly this purpose. [8](#0-7) 

### Proof of Concept

1. **Pair and grant bypass**: User connects a dApp via WalletConnect with wallet fingerprint `X`. During pairing, the user (or the dApp via `chia_requestPermissions`) grants bypass for `chia_sendTransaction`.
2. **Bypass is persisted**: `dapp-pairs.yaml` now contains a record with `fingerprint: X`, `bypass: ["chia_sendTransaction"]`.
3. **User deletes the key**: User opens `WalletDeleteDialog`, types the fingerprint, and confirms. `deleteKey` is called; `removeFingerprintPrefs` cleans up UI prefs. `dapp-pairs.yaml` is **not modified**.
4. **User re-imports the same mnemonic**: The same fingerprint `X` is restored. The user logs in.
5. **dApp sends a transaction without confirmation**: The dApp issues a `chia_sendTransaction` WalletConnect request. `dispatchPairRequest` finds the pair, sees `bypass.includes("chia_sendTransaction")` is true, and calls `process(context)` directly — skipping `confirm()` entirely. [9](#0-8) 

The user receives no confirmation dialog and funds are transferred without consent.
