Looking at the CKD protocol implementation, I need to trace the exact aggregation path and check for any validation of participant contributions.

### Title
Missing Proof-of-Correct-Formation on CKD Participant Contributions Allows Malicious Participant to Permanently Corrupt CKD Output - (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates participant contributions `(norm_big_y, norm_big_c)` with no cryptographic validation. A single malicious participant can send an arbitrarily modified `CKDOutput` — including a negated `norm_big_c` — and the coordinator will silently incorporate it, producing a `CKDOutput` that cannot be unmasked to the correct confidential key.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives one `CKDOutput` per participant and adds the fields together unconditionally: [1](#0-0) 

The helper `recv_from_others` only deduplicates by sender identity; it performs no cryptographic check on the message content: [2](#0-1) 

Each participant's honest contribution is:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · A)
```

computed in `compute_signature_share`: [3](#0-2) 

No zero-knowledge proof, commitment, or consistency check is attached to the message sent to the coordinator: [4](#0-3) 

This is in stark contrast to the ECDSA triple generation code, which requires dlog and dlogeq proofs for every analogous contribution before aggregating. [5](#0-4) 

---

### Impact Explanation

A malicious participant sends `(-norm_big_c_i, norm_big_y_i)` — the negation of their honest `norm_big_c` with an honest `norm_big_y`. The coordinator accumulates:

```
big_y  = Σ_j λ_j·y_j·G                          (correct)
big_c  = Σ_j λ_j·C_j  −  2·λ_i·C_i              (corrupted)
       = msk·H(pk,app_id) + (Σ_j λ_j·y_j)·A  −  2·λ_i·C_i
```

When the app unmasks with `app_sk = a`:

```
big_c − a·big_y
  = msk·H(pk,app_id) + (Σ_j λ_j·y_j)·a·G  −  2·λ_i·C_i  −  a·(Σ_j λ_j·y_j·G)
  = msk·H(pk,app_id)  −  2·λ_i·C_i
  = (msk − 2·λ_i·x_i)·H(pk,app_id)  −  2·λ_i·y_i·A
```

The `−2·λ_i·y_i·A` term does **not** cancel (it depends on the malicious participant's fresh random `y_i` and the app's public key `A`), so the result is not a valid BLS signature and `verify_signature` will fail. The app cannot derive its confidential key. Because `y_i` is freshly randomized each run, the corruption is unpredictable and cannot be corrected by the app.

The attack is repeatable: the malicious participant can corrupt every CKD invocation for any `app_id`, permanently denying the service to all honest apps.

---

### Likelihood Explanation

Any single participant in the protocol can mount this attack. The participant is already a trusted node in the MPC network, but the protocol's security model must tolerate up to `t-1` malicious participants. The attack requires only sending a modified group element — no cryptographic hardness assumption needs to be broken. It is trivially executable by any participant who deviates from the protocol.

---

### Recommendation

Require each participant to attach a zero-knowledge proof of correct formation alongside their `(norm_big_y, norm_big_c)` contribution. Specifically, the participant must prove in zero knowledge that:

- `norm_big_y = λ_i · y_i · G` for some `y_i` they know
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)` using the **same** `y_i` and their committed signing share `x_i`

This can be expressed as a dlog-equality proof (the same `y_i` appears in both `norm_big_y` and the `y_i·A` term of `norm_big_c`), combined with a check that the `x_i` component is consistent with the public verification share `x_i·G2` from DKG. The coordinator must verify all proofs before aggregating, rejecting any participant whose proof fails. The existing `dlogeq` infrastructure in `src/crypto/proofs/dlogeq.rs` can be reused for this purpose.

    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,

where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }
