Looking at the actual production code to trace the full attack path.

**Step 1: Does `exit_delegation_pool_intent(0)` unconditionally call `set_member_balance`?** [1](#0-0) 

Yes — regardless of whether `amount` is zero or non-zero, line 278 always calls `self.set_member_balance(:pool_member, amount: new_delegated_stake)`.

**Step 2: Does `set_member_balance` always append a new checkpoint?** [2](#0-1) 

It calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation: [3](#0-2) 

`insert` only updates in-place if `last.key == key`. If the epoch has advanced, `get_epoch_plus_k()` returns a strictly larger key, so a **new checkpoint is appended**. Calling once per epoch → trace grows by 1 per epoch.

**Step 3: Is there a guard blocking `amount=0` calls?** [4](#0-3) 

The only assertion is `amount <= total_amount`. There is no `amount > 0` guard. The `UNDELEGATE_IN_PROGRESS` guard only fires when `staker_removed == true`: [5](#0-4) 

When the staker is active, there is **no rate-limiting guard** on repeated `exit_delegation_pool_intent(0)` calls across epochs.

**Step 4: Does `calculate_rewards` iterate over all accumulated entries?** [6](#0-5) 

The code itself acknowledges: *"This loop is unbounded but unlikely to exceed gas limits."* The loop iterates from `entry_to_claim_from` to `pool_member_trace_length`, processing every entry with `epoch < until_epoch`. After N epochs of `exit_delegation_pool_intent(0)`, all N entries have `epoch < current_epoch` and are processed.

**Step 5: Does `pool_member_info_v1` also iterate without updating the cursor?** [7](#0-6) 

`pool_member_info_v1` calls `calculate_rewards` but discards the returned `entry_to_claim_from` (the `_` discard). It always re-iterates from the stored `pool_member_info.entry_to_claim_from`, which is only advanced by `claim_rewards`.

**Step 6: Does `claim_rewards` advance the cursor, providing a mitigation?** [8](#0-7) 

Yes — `claim_rewards` updates `entry_to_claim_from`. But if the user never calls `claim_rewards` during the N-epoch accumulation window, the first call must iterate all N entries. If N is large enough to exceed the Starknet block gas limit, the transaction reverts, the cursor is never advanced, and **all subsequent `claim_rewards` calls also revert** — permanently locking the rewards.

### Verdict

### Title
Unbounded `pool_member_epoch_balance` trace growth via repeated `exit_delegation_pool_intent(0)` causes permanent OOG in `claim_rewards` and `pool_member_info_v1` — (`src/pool/pool.cairo`)

### Summary
A pool member can call `exit_delegation_pool_intent(amount=0)` once per epoch across N epochs to grow their `pool_member_epoch_balance` trace to N entries. A subsequent `claim_rewards` or `pool_member_info_v1` call iterates the unbounded `calculate_rewards` loop over all N entries, potentially exceeding the block gas limit and permanently locking the member's unclaimed rewards.

### Finding Description
`exit_delegation_pool_intent` unconditionally calls `set_member_balance` at line 278 regardless of whether `amount` is zero. `set_member_balance` calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` function appends a new checkpoint whenever the epoch key advances. There is no guard preventing `amount=0` calls, and the `UNDELEGATE_IN_PROGRESS` guard only applies when the staker is already removed. Calling `exit_delegation_pool_intent(0)` once per epoch for N epochs therefore grows the `pool_member_epoch_balance` trace to N entries.

`calculate_rewards` (called by both `claim_rewards` and `pool_member_info_v1`) iterates from `entry_to_claim_from` to `pool_member_trace_length`, processing every entry with `epoch < current_epoch`. The code itself notes: *"This loop is unbounded but unlikely to exceed gas limits."* If N is large enough, the loop exceeds the block gas limit. Since the transaction reverts on OOG, `entry_to_claim_from` is never advanced, making every future `claim_rewards` call also revert — permanently locking the rewards.

`pool_member_info_v1` additionally discards the updated cursor on every call, so it always re-iterates from the stored checkpoint, making it permanently unqueryable for the affected member.

### Impact Explanation
Permanent freezing of the pool member's unclaimed yield and permanent inability to query `pool_member_info_v1` for that member. Matches: *High: Permanent freezing of unclaimed yield* and *Medium: Unbounded gas consumption / Griefing*.

### Likelihood Explanation
Low-to-medium. The attacker must pay for N transactions (one per epoch) and the attack is self-inflicted. However, it requires no privileged access, no external dependencies, and is fully reachable via a public pool call. A sufficiently patient or malicious actor (e.g., to grief a pool's accounting) can execute it.

### Recommendation
1. **Reject `amount=0` in `exit_delegation_pool_intent`** — add `assert!(amount.is_non_zero(), ...)` at the top of the function, since the spec already defines `amount=0` as a special "cancel intent" path that should not re-checkpoint the balance.
2. **Skip `set_member_balance` when the balance is unchanged** — if `new_delegated_stake == old_delegated_stake` and the last trace entry already has the same value, do not insert.
3. **Add a trace length cap** or implement a "partial claim" mechanism that allows advancing `entry_to_claim_from` in batches.

### Proof of Concept
```
1. Deploy staking + pool contracts in test harness.
2. Pool member joins with any non-zero amount.
3. For i in 1..N:
     advance_epoch()
     pool.exit_delegation_pool_intent(amount=0)
4. Assert pool_member_epoch_balance trace length == N.
5. Call pool.claim_rewards(pool_member) — assert it reverts with OOG for large N.
6. Fuzz N from 100 to 10000, record gas consumed per iteration to find the OOG threshold.
```

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
