### Title
NEAR Block Producer Can Bias `RandomSeed` Precompile and `PREVRANDAO` Output to Game Aurora EVM Contracts - (`engine-precompiles/src/random.rs`)

---

### Summary

The Aurora Engine exposes NEAR's per-block VRF seed directly to EVM contracts through two surfaces: the `RandomSeed` precompile at `0xc104f4840573bed437190daf5d2898c2bdf928ac` and the EVM `PREVRANDAO` opcode (via `block_randomness()`). Both surfaces return the raw NEAR block-level `random_seed` with no confirmation delay, no additional entropy mixing, and no withholding-resistance. A NEAR validator scheduled to produce a block can observe the VRF output before committing the block and, if the outcome is unfavorable, simply skip producing that block and try again in the next slot. Any Aurora EVM contract using either surface as a source of randomness (lotteries, NFT mints, on-chain games) is vulnerable to this "last-reveal" attack.

---

### Finding Description

**Two exposed surfaces, same root cause:**

**Surface 1 — `RandomSeed` precompile** (`engine-precompiles/src/random.rs`):

The `RandomSeed` struct wraps the NEAR block's `random_seed` and returns it verbatim to any EVM caller. The precompile is registered in every hard-fork variant (Homestead through Osaka). [1](#0-0) [2](#0-1) 

The gas cost is explicitly set to zero, making it free to call and therefore attractive as a randomness source. [3](#0-2) 

**Surface 2 — EVM `PREVRANDAO` opcode** (`engine/src/engine.rs`):

`block_randomness()` in the `Backend` implementation returns `Some(self.env.random_seed())`, meaning any Solidity contract reading `block.prevrandao` (or `block.difficulty` on pre-merge semantics) receives the same NEAR block VRF seed. [4](#0-3) 

**Source of the seed — NEAR runtime host function** (`engine-sdk/src/near_runtime.rs`):

The `random_seed()` implementation calls the NEAR host function `exports::random_seed(0)` directly, reading the per-block VRF value produced by the current block producer. [5](#0-4) 

The codebase itself documents this as a VRF-derived value:

> "Each NEAR block has a 32-byte entropy source generated by a VRF." [6](#0-5) 

**The attack — block withholding ("last-reveal"):**

In NEAR's consensus model, a single validator is scheduled to produce each block. That validator runs the VRF locally to compute `random_seed` before broadcasting the block. If the validator is also a participant in an Aurora EVM protocol that uses this seed (e.g., a lottery), they can:

1. Compute the VRF output for the upcoming block.
2. Simulate the Aurora EVM contract outcome using that seed.
3. If the outcome is unfavorable, withhold the block (skip their slot) and let the next validator produce a block with a different seed.
4. Repeat until a favorable seed is produced.

The cost is forfeiting the block reward for the skipped slot. For any protocol with a prize pool exceeding one block reward, this attack is economically rational.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

Any Aurora EVM contract that uses `block.prevrandao` or calls the `RandomSeed` precompile at `0xc104f4840573bed437190daf5d2898c2bdf928ac` to determine winners, allocate prizes, or distribute NFTs is vulnerable. A NEAR validator can bias the outcome in their favor, stealing funds that would otherwise be distributed fairly to other participants. The attack is not theoretical: the seed is a single 32-byte value produced by one party (the block producer) with no commit-reveal delay and no multi-party contribution.

---

### Likelihood Explanation

**High.** NEAR's validator set is permissionless and rotates. Any entity that stakes enough NEAR to become a validator can be scheduled to produce blocks. The attack requires no special access beyond being a scheduled block producer. The economic incentive scales directly with the prize pool of any Aurora-deployed protocol using this randomness. Withholding a single block is a well-understood, low-complexity operation.

---

### Recommendation

1. **Document the limitation prominently.** The `RandomSeed` precompile and `block_randomness()` should carry explicit warnings that the seed is manipulable by the current block producer and must not be used as the sole source of randomness for high-value outcomes.

2. **Add a confirmation-delay pattern.** Protocols should commit to a future block height at request time and only reveal/use the seed from a block that is sufficiently deep (e.g., ≥5 blocks) relative to the commitment. This mirrors the Chainlink VRF recommendation in the original report.

3. **Consider mixing additional entropy.** Incorporate the transaction hash or a user-supplied commit into the seed derivation (similar to how `compute_random_seed` in `engine-standalone-storage/src/sync/mod.rs` mixes `action_hash` with `block_random_value`) so that the block producer cannot fully predict the per-transaction seed. [7](#0-6)

### Proof of Concept

1. Deploy a simple lottery contract on Aurora that picks a winner using `block.prevrandao` or by calling `0xc104f4840573bed437190daf5d2898c2bdf928ac`.
2. As a NEAR validator scheduled to produce the next block, compute the VRF output locally.
3. Simulate the lottery outcome: if you do not win, skip your block slot.
4. The next validator produces a block with a different `random_seed`; repeat until the simulated outcome is favorable.
5. Produce the block and claim the prize.
