### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Block Consensus Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is specified to be callable only by the Starkware sequencer, but the implementation contains no caller authentication check. Any unprivileged address can call this function with `disable_rewards: true` to consume the single per-block reward slot, permanently preventing legitimate reward distribution for that block. This is a direct analog to the external report's "unauthenticated endpoint" vulnerability class: a privileged action is exposed without any authentication, enabling griefing by any observer.

### Finding Description
The specification at `docs/spec.md` line 1645 explicitly states the access control for `update_rewards`:

> **access control**: Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1507) contains **no caller check whatsoever**:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no get_caller_address() check anywhere ...
    self.last_reward_block.write(current_block_number);   // consumes the block slot
    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits without distributing
    }
    // ... reward distribution logic ...
}
``` [1](#0-0) 

The critical invariant is that `last_reward_block` is written to the current block number unconditionally (line 1485), and any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED` (lines 1454–1458). There is exactly one reward slot per block.

**Exploit path:**
1. Attacker monitors the mempool or simply submits at the start of each block.
2. Attacker calls `update_rewards(any_valid_staker_address, disable_rewards: true)`.
3. `general_prerequisites()` passes (contract is unpaused).
4. `last_reward_block` is written to the current block number.
5. The function returns early at line 1487 without distributing any rewards.
6. The legitimate sequencer's call for the same block reverts with `REWARDS_ALREADY_UPDATED`.
7. All stakers and pool members lose their consensus block rewards for that block — permanently, since there is no catch-up mechanism.

The attacker needs only a valid `staker_address` (trivially obtained from on-chain events such as `NewStaker`) and the ability to submit transactions. No privileged keys, no leaked secrets. [2](#0-1) 

The specification's intent is confirmed by the contrast with `update_rewards_from_attestation_contract`, which does enforce its caller:

```rust
fn update_rewards_from_attestation_contract(...) {
    self.assert_caller_is_attestation_contract();  // enforced
    ...
}
``` [3](#0-2) 

The spec also confirms the intended restriction: [4](#0-3) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Once consensus rewards are active (`is_pre_consensus()` returns false), every block that the attacker front-runs results in zero rewards distributed to all stakers and pool members for that block. The `last_reward_block` slot is consumed and cannot be reclaimed. There is no retroactive distribution mechanism. A sustained attacker submitting one transaction per block can suppress 100% of consensus-phase block rewards across the entire protocol, permanently denying yield to all stakers and delegators.

### Likelihood Explanation
**High.** The attack requires:
- A valid staker address (publicly observable from `NewStaker` events).
- The ability to submit a transaction (any unprivileged address).
- No front-running in the strict sense is even required — the attacker simply needs to submit before the sequencer's reward transaction in the same block.

The cost is one transaction per block. On Starknet, transaction fees are low. The attacker has no profit motive but causes direct, measurable, irreversible harm to all protocol participants.

### Recommendation
Add a sequencer-only access control check at the top of `update_rewards`, consistent with the specification and analogous to the existing `assert_caller_is_attestation_contract` pattern:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer();   // add this
    ...
}
```

The sequencer address should be stored in contract storage and settable only by governance, following the same pattern used for `attestation_contract`.
