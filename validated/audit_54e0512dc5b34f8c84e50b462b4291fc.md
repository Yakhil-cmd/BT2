### Title
TOCTOU Pool Puzzle Hash Substitution: Displayed `targetPuzzleHash` Differs from Submitted Hash — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

The pool-joining flow makes two independent, uncached HTTP requests to the pool's `/pool_info` endpoint: one for display (via `usePoolInfo` in `PlotNFTSelectBase`) and a second at submit time (inside `prepareSubmitData`). A malicious pool operator can serve a different `targetPuzzleHash` on each request, causing the user to see a legitimate payout address in the UI while the `createNewPoolWallet` / `pwJoinPool` RPC is called with an attacker-controlled address.

---

### Finding Description

**Display path:**

`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)` at line 39, which internally calls `getPoolInfo(normalizedUrl)` → `window.appAPI.fetchPoolInfo(poolUrl)` → a live HTTP GET to `<poolUrl>/pool_info`. The returned `targetPuzzleHash` is rendered in the "Verify Pool Details" card. [1](#0-0) [2](#0-1) 

**Submit path:**

When the user clicks submit, `handleSubmit` calls `prepareSubmitData(data)`, which makes a **second, independent** call to `getPoolInfo(normalizedPoolUrl)` and uses the freshly returned `targetPuzzleHash` directly in `initialTargetState` — with no comparison against the previously displayed value. [3](#0-2) 

**No caching or binding between the two calls:**

`getPoolInfo` is a thin wrapper with zero caching: [4](#0-3) 

There is no code anywhere that compares `poolInfo.poolInfo.targetPuzzleHash` (the displayed value) with the `targetPuzzleHash` returned at submit time. The two HTTP round-trips are fully independent.

---

### Impact Explanation

`targetPuzzleHash` is the pool's payout puzzle hash — the on-chain address to which the user's farming rewards are sent. If the attacker substitutes `puzzle_hash_B` at submit time, all future block rewards from the user's plots are paid to the attacker's address, not the legitimate pool. This is a direct, irreversible financial loss of XCH farming rewards for the duration the NFT remains joined to the pool.

---

### Likelihood Explanation

Any pool operator can exploit this trivially: serve the legitimate `targetPuzzleHash` on the first request (so the UI looks correct) and an attacker-controlled hash on the second request (triggered by the submit button). No cryptographic break, no local compromise, and no social engineering beyond operating a pool that users choose to join is required. The timing window is the entire duration between page load and submit click — easily controlled server-side by counting requests per session/IP.

---

### Recommendation

At submit time, do not re-fetch pool info. Instead, pass the already-resolved `poolInfo` object (obtained from `usePoolInfo`) into `prepareSubmitData` or `handleSubmit` directly, and use its `targetPuzzleHash` and `relativeLockHeight` for the RPC call. If a re-fetch is considered necessary, assert that the returned `targetPuzzleHash` is byte-for-byte identical to the displayed one and abort with a visible error if they differ.

---

### Proof of Concept

1. Stand up an HTTP server at `http://attacker-pool.example/pool_info`.
2. On the **first** request, return `{ "target_puzzle_hash": "0xAAAA...AAAA", "relative_lock_height": 32, ... }`.
3. On the **second** request (and all subsequent), return `{ "target_puzzle_hash": "0xBBBB...BBBB", "relative_lock_height": 32, ... }`.
4. Open the GUI, navigate to "Join a Pool", enter `http://attacker-pool.example`.
5. Observe the "Verify Pool Details" card renders `0xAAAA...AAAA`.
6. Click the submit/Create button.
7. Intercept the `createNewPoolWallet` RPC call (or inspect the signed transaction) — `initialTargetState.targetPuzzleHash` will be `0xBBBB...BBBB`, not `0xAAAA...AAAA`.

The user approved a join to `0xAAAA...AAAA` but the blockchain transaction encodes `0xBBBB...BBBB` as the reward destination.
