### Title
Staker Can Permanently Freeze Unclaimed Yield by Setting Reward Address to Zero Address - (File: src/staking/staking.cairo)

---

### Summary
The `change_reward_address` function in `src/staking/staking.cairo` performs no zero-address validation on the incoming `reward_address` parameter. A staker who accidentally or mistakenly passes `0x0` as the new reward address will have all future reward transfers directed to the zero address, permanently destroying unclaimed yield with no protocol-level safeguard.

---

### Finding Description
`change_reward_address` (lines 517–540 of `src/staking/staking.cairo`) is callable by any registered staker and updates the address to which all staking rewards are sent. The only guard present is a check that the new address is not a registered token contract:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;

    staker_info.reward_address = reward_address;   // written unconditionally
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
```

There is no `assert!(!reward_address.is_zero(), ...)` guard. The zero address (`0x0`) is not a token contract, so it passes the only existing check and is written directly to storage.

Once the reward address is set to zero:

1. **Reward claims** – every call to `claim_rewards` (or the internal `send_rewards_to_staker` path invoked during `unstake_action`) will attempt `checked_transfer(recipient: 0x0, ...)`. OpenZeppelin Cairo ERC-20 reverts on transfers to the zero address (`ERC20: transfer to 0`), so the staker is unable to claim any accumulated rewards.
2. **Unstake action blocked** – `unstake_action` calls `send_rewards_to_staker` before erasing the staker record. If that transfer reverts, the entire `unstake_action` transaction reverts, permanently trapping the staker's self-stake inside the contract.
3. **Recovery path is narrow** – the staker can call `change_reward_address` again with a valid address to unblock themselves, but any rewards that were already transferred to the zero address in the window before correction are permanently destroyed.

The analogous pattern from the external report is exact: a critical address-update function accepts any value without confirming the new address is live/valid, and a single mistaken call can lock the caller out of all permissioned downstream flows.

---

### Impact Explanation
- **Permanent freezing of unclaimed yield** – rewards accumulated up to the point of the bad call, and all rewards earned afterward until the staker notices and corrects the address, are sent to `0x0` and irrecoverably burned.
- **Temporary freezing of staker's own stake** – while the zero reward address is in effect, `unstake_action` reverts (because `send_rewards_to_staker` reverts), preventing the staker from withdrawing their self-stake. This constitutes temporary freezing of funds.

Both impacts are within the allowed High-impact scope: *"Permanent freezing of unclaimed yield"* and *"Temporary freezing of funds."*

---

### Likelihood Explanation
**Low-to-Medium.** The call requires only the staker's own key; no privileged role is needed. A single fat-finger (e.g., passing `0` instead of a valid address in a script or UI) is sufficient. The staker population grows over time, increasing the probability of at least one such mistake. The absence of any zero-address guard means the protocol provides no safety net.

---

### Recommendation
Add an explicit zero-address assertion at the top of `change_reward_address`, mirroring the pattern already used elsewhere in the codebase:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(!reward_address.is_zero(), "Reward address cannot be zero address");
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

The same guard should be applied to the pool contract's equivalent `change_reward_address` for pool members.

---

### Proof of Concept

1. Staker `S` is registered with a valid `reward_address = R`.
2. `S` calls `staking.change_reward_address(reward_address: 0x0)`.
3. The only check — `!self.does_token_exist(token_address: 0x0)` — passes because `0x0` is not a registered token.
4. Storage is updated: `staker_info.reward_address = 0x0`.
5. `S` (or anyone) calls `staking.claim_rewards(staker_address: S)`.
   - Internally, `send_rewards_to_staker` calls `token_dispatcher.checked_transfer(recipient: 0x0, amount: accrued_rewards)`.
   - OpenZeppelin ERC-20 reverts: `"ERC20: transfer to 0"`.
   - All accumulated rewards are unclaimable.
6. `S` calls `staking.unstake_intent()` — succeeds (no reward transfer here).
7. After the exit window, `S` calls `staking.unstake_action(staker_address: S)`.
   - `send_rewards_to_staker` is called first → reverts on `checked_transfer(recipient: 0x0, ...)`.
   - The entire `unstake_action` reverts; `S`'s self-stake remains locked in the contract.
8. `S` must call `change_reward_address` again with a valid address to unblock themselves, but any rewards already accrued and "sent" to `0x0` before correction are permanently lost.
