### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `DelegationPool` contains an unbounded `while` loop that iterates over a pool member's entire balance-change trace. A pool member who makes balance changes across many epochs without claiming rewards will grow this trace indefinitely. Once the trace is large enough, any call to `claim_rewards` will exceed Starknet's execution step limit, permanently freezing the pool member's unclaimed yield with no recovery path.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current trace length: [1](#0-0) 

The code itself acknowledges the risk with the comment:

> **Note**: The loop iterates over the balance changes in the pool member's balance trace. **This loop is unbounded but unlikely to exceed gas limits.**

The trace (`pool_member_epoch_balance`) is a `Vec`-backed checkpoint structure stored per pool member: [2](#0-1) 

A new checkpoint is appended to the trace whenever `set_member_balance` or `increase_member_balance` is called with a key (epoch) that differs from the last stored key: [3](#0-2) 

The `insert` function only deduplicates within the same epoch; a balance change in each distinct epoch appends a new entry: [4](#0-3) 

Balance changes that grow the trace are triggered by two publicly callable functions:
- `add_to_delegation_pool` (line 221) calls `increase_member_balance` → `set_member_balance`
- `exit_delegation_pool_intent` (line 256) calls `set_member_balance` [5](#0-4) [6](#0-5) 

Inside the loop, each iteration calls `find_sigma`, which performs additional storage reads from `cumulative_rewards_trace`: [7](#0-6) 

The `entry_to_claim_from` cursor is only advanced when `claim_rewards` succeeds and writes back to storage: [8](#0-7) 

There is no partial-claim mechanism. If the loop cannot complete within the transaction's step budget, the entire `claim_rewards` call reverts and `entry_to_claim_from` is never updated, making the freeze permanent.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace grows large enough that iterating from `entry_to_claim_from` to the current length exceeds Starknet's per-transaction step limit, `claim_rewards` will always revert. Because there is no batched or partial-claim path, the pool member's accumulated STRK rewards become permanently unclaimable. The pool member also cannot exit cleanly, since `exit_delegation_pool_action` does not depend on `calculate_rewards`, but the rewards are lost forever.

### Likelihood Explanation

**Medium.** The attack requires one balance-change transaction per epoch without any intervening `claim_rewards` call. Epochs are time-bounded (e.g., weekly), so accumulating a dangerous number of trace entries takes months to years of deliberate or negligent behavior. However:

1. The protocol is designed to run indefinitely, making long-lived positions realistic.
2. A pool member who simply forgets to claim rewards while actively adjusting their delegation (e.g., dollar-cost averaging in/out) will naturally grow the trace.
3. There is no on-chain warning or cap to prevent the trace from reaching a dangerous size.
4. Each loop iteration is expensive (two `cumulative_rewards_trace` storage reads via `find_sigma`), so the threshold may be reached sooner than intuition suggests.

### Recommendation

1. **Cap the balance-change trace length**: Enforce a maximum number of unclaimed balance-change entries (e.g., 100–200). Reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the pool member's unclaimed trace length would exceed the cap, or require the member to call `claim_rewards` first.

2. **Support paginated / partial reward claims**: Allow `claim_rewards` to accept a `max_entries` parameter so a pool member with a large trace can drain it incrementally across multiple transactions.

3. **Benchmark the step cost**: Measure the actual Starknet step consumption of `calculate_rewards` as a function of trace length and set the cap conservatively below the network's per-transaction step limit, with headroom for future opcode cost changes.

### Proof of Concept

1. Deploy the staking system and create a staker with a STRK delegation pool.
2. As `pool_member`, call `enter_delegation_pool` with a small amount.
3. For each epoch `i` from 1 to N (e.g., N = 500):
   - Advance to epoch `i`.
   - Call `add_to_delegation_pool(pool_member, 1)` — this appends one new entry to `pool_member_epoch_balance` because `get_epoch_plus_k()` returns a new key each epoch.
   - Do **not** call `claim_rewards`.
4. After N epochs, call `claim_rewards(pool_member)`.
5. Observe that the transaction runs out of steps (Cairo VM step limit exceeded) and reverts.
6. Repeat step 4 — it reverts again. The rewards are permanently frozen.

The loop at `src/pool/pool.cairo:859` will attempt N iterations, each performing two storage reads into `cumulative_rewards_trace` via `find_sigma`, causing the transaction to exceed Starknet's execution budget.
