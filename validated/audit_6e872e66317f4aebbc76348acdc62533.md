### Title
Unchecked EVM Execution Result Status in `EVM.run()` Allows Silent Cross-VM Transfer Failures - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.run()`, `coa.call()`, and `coa.deploy()` in `fvm/evm/stdlib/contract.cdc` return an `EVM.Result` struct that carries a `status` field (`unknown`, `invalid`, `failed`, or `successful`). The protocol does not enforce that callers inspect this status. A Cadence transaction that uses `EVM.run()` to relay an EVM token transfer (e.g., an ERC20 `transfer()`) will succeed at the Cadence level even when the inner EVM execution reverts or runs out of gas, producing a silent cross-VM asset inconsistency — the exact analog of the unchecked ERC20 return-value class.

### Finding Description
`EVM.run()` is a public, unprivileged Cadence function exposed by the EVM system contract:

```cadence
access(all)
fun run(tx: [UInt8], coinbase: EVMAddress): Result {
    pre { !self.isPaused(): "EVM operations are temporarily paused" }
    return InternalEVM.run(tx: tx, coinbase: coinbase.bytes) as! Result
}
``` [1](#0-0) 

The returned `Result` struct carries three distinct failure states:

```cadence
access(all) enum Status: UInt8 {
    access(all) case unknown   // something went very wrong
    access(all) case invalid   // validation failed, tx not executed
    access(all) case failed    // executed but VM reported an error (e.g. out of gas, revert)
    access(all) case successful
}
``` [2](#0-1) 

The companion `mustRun()` helper only asserts that the status is **not** `unknown`; it explicitly does **not** revert on `failed`:

```cadence
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult   // status == failed is silently returned to caller
}
``` [3](#0-2) 

The same pattern applies to `coa.call()` and `coa.deploy()`, both of which return a `Result` with no post-condition on status: [4](#0-3) [5](#0-4) 

At the Go handler layer, `StatusFailed` is a defined, non-error outcome — the transaction is included in the EVM block and gas is consumed, but the state change (e.g., token transfer) is rolled back: [6](#0-5) 

### Impact Explanation
A Cadence transaction that:
1. Withdraws Cadence-side tokens (or records a Cadence-side accounting entry), **and**
2. Calls `EVM.run()` / `coa.call()` to relay the corresponding EVM-side transfer

…will commit the Cadence-side state change even when the EVM execution reverts. The EVM tokens are never moved, but the Cadence ledger records that they were. This is a direct cross-VM bridge escrow mis-accounting / asset loss scenario: one side of the ledger is updated while the other is not.

### Likelihood Explanation
`EVM.run()` is `access(all)` — callable by any unprivileged Cadence transaction. The `EVM.Result` type is a plain struct with no linear-type enforcement; Cadence does not require the caller to pattern-match on `status`. Any transaction author who omits the status check (a common omission, as `mustRun()` itself demonstrates by only guarding against `unknown`) triggers the silent failure path. The Flow EVM bridge and any cross-VM DeFi integration are realistic targets.

### Recommendation
1. Add a `post` condition to `EVM.run()` (or a new `safeRun()` variant) that panics when `status == failed` unless the caller explicitly opts into failure tolerance.
2. Rename `mustRun()` to reflect that it only guards against `invalid`/`unknown`, not `failed`, and add a `mustSucceed()` variant that also asserts `status == successful`.
3. Document clearly that `status == failed` means the EVM state change was **not** applied, so any Cadence-side accounting that depends on the EVM transfer must be gated on an explicit status check.

### Proof of Concept
```cadence
import EVM from <EVMAddress>
import SomeToken from <TokenAddress>

transaction(evmTransferTx: [UInt8], coinbaseBytes: [UInt8; 20]) {
    prepare(signer: auth(Storage) &Account) {
        // Step 1: debit Cadence-side balance (or escrow tokens)
        let vault <- signer.storage.load<@SomeToken.Vault>(from: /storage/vault)!
        // ... vault is now consumed / escrowed on Cadence side ...

        // Step 2: relay EVM-side ERC20 transfer — result is NEVER checked
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)
        let res = EVM.run(tx: evmTransferTx, coinbase: coinbase)
        // res.status may be EVM.Status.failed (e.g. ERC20 reverted, out of gas)
        // The Cadence transaction still succeeds; EVM tokens were NOT transferred.
        // Cadence escrow is now permanently locked with no corresponding EVM credit.
    }
}
```

The `res.status` field is available but nothing in the protocol forces the transaction to abort on `failed`. The Cadence transaction commits, the EVM transfer did not happen, and the cross-VM accounting is permanently inconsistent. [1](#0-0) [7](#0-6)

        /// has been successful but the vm has reported an error in

        /// Note that resubmission of a failed transaction would
        /// result in invalid status in the second attempt, given
        /// the nonce would become invalid.
        access(all) case failed

        /// Returned when execution of an evm transaction/call
        /// has been successful and no error is reported by the vm.
        access(all) case successful
    }

        /// is also returned here (see EIP-140 for more details).

        /// if the transaction caused such a deployment
        /// otherwise the value is nil.

        init(
