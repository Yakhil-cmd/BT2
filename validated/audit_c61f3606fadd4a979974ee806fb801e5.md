### Title
Missing Vec Length Validation After Deserialization Causes Panic in Triple Generation — (`File: src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
In `do_generation_many<const N: usize>`, multiple protocol rounds deserialize `Vec<T>` fields from network messages sent by other participants and immediately index them with `[i]` for `i in 0..N` without first validating that the received vector contains at least `N` elements. A malicious participant can send a message with an empty or short vector, triggering an out-of-bounds index panic in the honest party's protocol executor, permanently aborting triple generation and thus ECDSA signing for that party.

### Finding Description
The function `do_generation_many` is the core of the OT-based ECDSA triple generation protocol. It receives several messages whose payloads are `Vec<T>` (or structs containing `Vec<T>` fields), and then iterates `for i in 0..N` indexing into those vectors with `[i]`. No length check is performed before the indexed access.

**Round 1 — commitment collection** (`wait0`): [1](#0-0) 

A malicious participant sends a `Vec<Commitment>` with fewer than `N` elements. The loop `for i in 0..N` panics at `commitments[i]` when `i >= commitments.len()`.

**Round 2 — polynomial commitments** (`wait2`): [2](#0-1) 

`PolynomialCommitmentsMessageMany` contains six `Vec` fields (`big_e_v`, `big_f_v`, `big_l_v`, `randomizer_v`, `phi_proof0_v`, `phi_proof1_v`). Any of them can be sent with fewer than `N` elements, causing a panic at `their.big_e_v[i]` etc. [3](#0-2) 

**Round 3 — private share exchange** (`wait3`): [4](#0-3) 

**Round 4 — C-point and dlogeq proofs** (`wait4`): [5](#0-4) 

**Round 5 — c-share accumulation** (`wait6`): [6](#0-5) 

The deserialization layer (`rmp_serde::decode::from_slice`) successfully decodes a `Vec<T>` of any length — it does not enforce that the length equals `N`. [7](#0-6) 

### Impact Explanation
A Rust out-of-bounds index panics unconditionally. The panic propagates through the async future being polled inside `ProtocolExecutor::poke()`, crashing the protocol instance for the honest party. [8](#0-7) 

Without successfully completing triple generation, the honest party has no triples and cannot perform any OT-based ECDSA presignature or signing operation. A single malicious co-participant can repeat this attack on every new triple generation attempt, causing **permanent denial of signing** for honest parties under valid protocol inputs.

**Impact: High** — Permanent denial of triple generation (and thus ECDSA signing) for honest parties.

### Likelihood Explanation
Any participant in the triple generation protocol can trigger this. The attacker only needs to craft a MessagePack-encoded `Vec` with zero or fewer-than-`N` elements and deliver it at the correct waitpoint. No cryptographic material or privileged access is required beyond being a protocol participant.

**Likelihood: 3 out of 5** — Straightforward to exploit by any malicious co-participant; requires only crafting a short serialized vector.

### Recommendation
Before indexing into any deserialized `Vec` with `[i]` for `i in 0..N`, validate that the vector's length equals exactly `N` and return a `ProtocolError` (not a panic) if it does not. For example:

```rust
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "expected {N} commitments from {from:?}, got {}", commitments.len()
    )));
}
```

Apply this check at every receive site in `do_generation_many` before the `for i in 0..N` loop that indexes into the received vector.

### Proof of Concept
A malicious participant `P_evil` participates in `generate_triple_many::<N>` with honest parties. At `wait0`, instead of sending `N` commitments, `P_evil` sends a MessagePack-encoded `Vec<Commitment>` of length 0. When the honest party's loop executes:

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
// commitments.len() == 0, N >= 1
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]); // panics: index 0 out of bounds for slice of length 0
}
```

The honest party's `ProtocolExecutor` panics, aborting triple generation. The attacker repeats this on every subsequent attempt, permanently denying the honest party the ability to generate triples and thus sign. [9](#0-8)
