### Title
Pre-EIP-155 (Chain-ID-Free) Legacy Transactions Accepted, Enabling Cross-Chain Replay Attacks - (File: `engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

---

### Summary

Aurora Engine's `submit` endpoint accepts legacy Ethereum transactions signed without a chain ID (pre-EIP-155, `v = 27` or `v = 28`). Because the chain ID validation in `submit_with_alt_modexp` is guarded by `if let Some(chain_id) = transaction.chain_id`, transactions that carry no chain ID bypass the check entirely. An attacker who observes a pre-EIP-155 transaction broadcast on any other EVM chain (e.g., Ethereum mainnet) can replay it verbatim on Aurora and drain the victim's Aurora balance, and vice versa.

---

### Finding Description

**Deprecated protocol acceptance — the direct analog to TLS 1.0/1.1 support.**

`LegacyEthSignedTransaction::sender()` in `engine-transactions/src/legacy.rs` explicitly handles `v = 27` and `v = 28` as valid, returning `chain_id = None`:

```rust
// engine-transactions/src/legacy.rs  lines 68-78
let (chain_id, rec_id) = match self.v {
    0..=26 | 29..=34 => return Err(Error::InvalidV),
    27..=28 => (
        None,                                          // ← no chain ID
        u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
    ),
    _ => (
        Some((self.v - 35) / 2),
        u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
    ),
};
```

`LegacyEthSignedTransaction::chain_id()` propagates this `None`:

```rust
// engine-transactions/src/legacy.rs  lines 88-93
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,          // covers v=27 and v=28
        _ => Some((self.v - 35) / 2),
    }
}
```

When the transaction is normalized in `engine-transactions/src/lib.rs` (lines 106-118), `chain_id: tx.chain_id()` is `None` for these transactions.

The only chain-ID guard in the engine is:

```rust
// engine/src/engine.rs  lines 1054-1059
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

The `if let Some(...)` pattern means the entire block is **skipped** when `chain_id` is `None`. A pre-EIP-155 transaction passes this guard unconditionally and proceeds to full EVM execution.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

A pre-EIP-155 transaction is not bound to any chain. The identical signed bytes are valid on every EVM-compatible network simultaneously. An attacker who observes such a transaction on Ethereum mainnet (or any other chain) can submit the same raw bytes to Aurora's `submit` endpoint. If the signing address holds a balance on Aurora, the transaction executes and transfers/deploys exactly as intended by the original signer — but on Aurora, without the signer's knowledge or consent. Funds are stolen.

The reverse path is equally dangerous: a user who signs a pre-EIP-155 transaction on Aurora (e.g., using legacy tooling) exposes themselves to replay on Ethereum mainnet and every other EVM chain where the address has funds.

---

### Likelihood Explanation

**Medium.**

- Pre-EIP-155 transactions (`v = 27/28`) are still produced by hardware wallets in certain modes, by older Ethereum libraries, and by any tool that omits the chain ID parameter.
- Aurora addresses are derived from the same secp256k1 key pairs as Ethereum addresses, so the same private key controls the same address on both networks — a prerequisite for replay.
- The attacker's only requirement is to observe a pre-EIP-155 transaction on any public mempool or block explorer and submit it to Aurora's `submit` NEAR method. No privileged access is needed.
- The CHANGES.md entry for v2.4.0 states "Engine will no longer accept EVM transactions without a chain ID," indicating this was a known risk that was supposed to have been closed, but the current production code still contains the `v = 27/28 → chain_id = None` path and the conditional guard that skips validation when `chain_id` is `None`.

---

### Recommendation

- **Short term:** In `submit_with_alt_modexp` (`engine/src/engine.rs`), change the chain ID check from a conditional `if let Some(...)` to an unconditional rejection when `chain_id` is `None`:
  ```rust
  let chain_id = transaction.chain_id
      .ok_or(EngineErrorKind::InvalidChainId)?;
  if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
      return Err(EngineErrorKind::InvalidChainId.into());
  }
  ```
- **Long term:** Remove the `v = 27/28` branch from `LegacyEthSignedTransaction::sender()` entirely, making pre-EIP-155 transactions unparseable at the transaction layer, consistent with the stated intent in CHANGES.md v2.4.0.

---

### Proof of Concept

1. On Ethereum mainnet, observe any pre-EIP-155 transaction (v = 27 or v = 28) from address `A` that transfers ETH or calls a contract. The raw RLP bytes are publicly available on Etherscan.

2. Confirm address `A` holds a balance on Aurora mainnet (chain ID 1313161554). This is common because Aurora users often import the same private key.

3. Submit the **identical raw transaction bytes** to Aurora's `submit` NEAR method:
   ```
   near call aurora submit --base64 <raw_rlp_bytes_base64> --accountId attacker.near
   ```

4. `EthTransactionKind::try_from` parses the bytes as `EthTransactionKind::Legacy`.

5. `NormalizedEthTransaction::try_from` sets `chain_id: None` (because `v ∈ {27, 28}`).

6. In `submit_with_alt_modexp`, the guard at line 1055 evaluates `if let Some(chain_id) = None` → **false**, so no `InvalidChainId` error is returned.

7. `check_nonce` passes if the nonce matches Aurora's state for address `A`.

8. The transaction executes on Aurora, transferring `A`'s Aurora ETH balance to the attacker-controlled destination encoded in the original transaction.
