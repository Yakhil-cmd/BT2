### Title
Unbounded `subsidized_amount` accumulation via repeated `promise_batch_action_function_call_weight` on zero-balance contract — (`runtime/near-vm-runner/src/logic/logic.rs`)

### Summary
A zero-balance contract with `one_yocto_on_promise` enabled can call `promise_batch_action_function_call_weight` N times with `amount = 1 yoctoNEAR` in a single execution. Each call takes the `skip_deduct` path because `current_account_balance` is never modified by that path, so it remains zero throughout. `subsidized_amount` accumulates to N yoctoNEAR, violating the design invariant that at most one subsidized 1-yocto deposit is permitted per zero-balance contract execution.

### Finding Description
The guard condition at `logic.rs:2975-2977` checks three things:

1. `amount == 1 yocto` — satisfied by attacker choice
2. `self.config.one_yocto_on_promise` — a protocol-level config flag, enabled in production
3. `self.result_state.current_account_balance.is_zero()` — satisfied initially for a zero-balance account [1](#0-0) 

When `skip_deduct` is true, `deduct_balance` is **not** called, so `current_account_balance` remains zero. On the next call to `promise_batch_action_function_call_weight` within the same execution, condition (3) is still satisfied. There is no per-execution flag or counter to prevent re-entry into the `skip_deduct` branch.

The `subsidized_amount` field is initialized to `Balance::ZERO` at the start of each execution: [2](#0-1) 

And is propagated into `VMOutcome` via `compute_outcome`: [3](#0-2) 

The runtime then accumulates it into `ActionResult`: [4](#0-3) 

Each of the N calls also appends a real receipt via `ext.append_action_function_call_weight` with `amount = 1 yocto`: [5](#0-4) 

These N receipts each carry a 1-yocto deposit that was never deducted from the sender. The `subsidized_amount = N yocto` is the runtime's record of how much was "minted" to cover these receipts. The design comment at lines 2971–2974 states the intent is to allow "exactly 1 yoctoNEAR", but the implementation enforces no such limit. [6](#0-5) 

### Impact Explanation
The exact corrupted value is `VMOutcome::subsidized_amount`, which can be set to N yoctoNEAR (N ≥ 1) instead of the intended maximum of 1 yoctoNEAR per execution. Each of the N receipts carries a 1-yocto deposit that was not deducted from the sender's balance. The runtime uses `subsidized_amount` to account for tokens attached to receipts without a corresponding deduction — if it mints `subsidized_amount` tokens to cover these receipts, the total token supply increases by N yocto per execution instead of at most 1. The ledger invariant of token supply conservation is violated.

The economic magnitude per execution is bounded by gas (each call to `promise_batch_action_function_call_weight` costs gas), but the invariant violation is real and repeatable across many transactions.

### Likelihood Explanation
Any deployed contract on a zero-balance account can trigger this. The `one_yocto_on_promise` feature is a production protocol parameter. No special privileges are required — only the ability to deploy and call a contract. The attacker controls the number of calls within a single execution, bounded only by the gas limit.

### Recommendation
Add a per-execution guard that allows at most one subsidized 1-yocto deposit per execution. For example, track a boolean `subsidized_once: bool` in `ExecutionResultState` and return an error (or fall through to `deduct_balance`) if `skip_deduct` would be triggered a second time within the same execution. Alternatively, cap `subsidized_amount` at 1 yoctoNEAR and error on any subsequent attempt.

### Proof of Concept
Write a unit test that constructs a `VMLogic` with `account_balance = 0`, `one_yocto_on_promise = true`, calls `promise_batch_action_function_call_weight` 100 times with `amount = 1 yocto`, and asserts:

```rust
assert_eq!(outcome.subsidized_amount, Balance::from_yoctonear(1));
// Currently fails: subsidized_amount == 100 yoctoNEAR
```

The test will fail because `subsidized_amount` equals 100 yoctoNEAR, not 1, demonstrating the unbounded accumulation.
