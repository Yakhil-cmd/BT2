### Title
Opaque Unblockable Address Exemption Enables Permanent Blocklist Bypass - (File: `app/unblockable.go`)

### Summary
Cronos hardcodes 27 EVM addresses in `app/unblockable.go` that are permanently and silently exempt from the validator-operated blocklist. The selection criteria for these addresses is entirely undocumented, two entries are explicitly marked `// test`, and the list is publicly visible in source code. Any address in this list can never be blocked regardless of admin or governance action, and any user who controls or knows these addresses can use them as permanent proxies to bypass sanctions enforcement.

### Finding Description
`app/unblockable.go` defines `unblockableHexAddresses`, a hardcoded slice of 27 EVM addresses. [1](#0-0) 

`IsUnblockable()` is called in two enforcement paths. In `ProposalHandler.SetBlockList()`, any address matching the list is silently `continue`d — skipped — when building the active blocklist map: [2](#0-1) 

The same filter is applied in `setAnteHandler()` when constructing the static `blockedMap` from the node's config-level blacklist: [3](#0-2) 

The result is that `ValidateTransaction` (used in both `PrepareProposal` and `ProcessProposal`) and `BlockAddressesDecorator.AnteHandle` (used in `CheckTx`) will never reject a transaction whose signer or destination is one of these 27 addresses, regardless of what the admin submits via `MsgStoreBlockList`. [4](#0-3) 

The comment in the source states the list is "intentionally not configurable" but provides zero documentation of what these addresses represent, who controls them, or what criteria were used to include them. Two entries are explicitly annotated `// test`: [5](#0-4) 

### Impact Explanation
**High — Bypass of Cronos block-list authorization checks.**

The blocklist is Cronos's primary compliance and sanctions enforcement mechanism. Any address in `unblockableHexAddresses` is permanently immune to it. If any of these addresses are controlled by a malicious actor (which is plausible given the presence of `// test` entries and the complete absence of ownership documentation), that actor can:

1. Transact freely as a signer even when the admin has submitted a blocklist containing their address.
2. Receive CRO or CRC20/CRC21 tokens as a destination even when the admin has attempted to block that address.
3. Act as a permanent proxy for a sanctioned party: a blocked user pre-positions assets at a known unblockable address before the block takes effect, then directs that address to continue transacting on their behalf.

This directly matches the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation
The 27 addresses are publicly visible in the open-source repository. Any observer can enumerate them. The two `// test`-annotated addresses (`0xCC5d9bF5C3662D8A86A45ed23B300bc06ab36644`, `0xDaB2C01b1eBdf1D33eCF6Aff3a29b977a1EFba41`) are particularly concerning because test addresses are routinely controlled by developers or CI systems whose private keys may be widely known or poorly secured. No privileged access is required to exploit this — the attacker only needs to know the addresses (public) and control or interact with one of them.

### Recommendation
1. **Document every entry** in `unblockableHexAddresses` with its purpose, the entity that controls it, and the rationale for permanent exemption.
2. **Remove `// test` entries** from any production binary. Test-only addresses must not appear in production enforcement code.
3. **Publish the ownership mapping** of all unblockable addresses so that validators and governance can audit whether the exemptions remain appropriate.
4. Consider whether the unblockable list should be governed on-chain (e.g., via governance proposal) rather than hardcoded, so that compromised or misused entries can be removed without a binary upgrade.

### Proof of Concept
1. Identify any address from `unblockableHexAddresses`, e.g. `0xCC5d9bF5C3662D8A86A45ed23B300bc06ab36644` (marked `// test`).
2. Admin submits `MsgStoreBlockList` containing that address encrypted to validators.
3. `SetBlockList` is called; `IsUnblockable` returns `true` for the address; it is silently skipped and never inserted into `h.blocklist`. [6](#0-5) 
4. `ValidateTransaction` is called during `PrepareProposal`/`ProcessProposal`; the address is absent from `h.blocklist`; the check passes. [7](#0-6) 
5. The address transacts freely as signer or destination — the blocklist has no effect on it, permanently and without any on-chain recourse.

func (h *ProposalHandler) ValidateTransaction(tx sdk.Tx, txBz []byte) error {
	if len(h.blocklist) == 0 {
		// fast path, accept all txs
		return nil
	}

	var err error
	if tx == nil {
		tx, err = h.TxDecoder(txBz)
		if err != nil {
			return err
		}
	}

	sigTx, ok := tx.(signing.SigVerifiableTx)
	if !ok {
		return fmt.Errorf("tx of type %T does not implement SigVerifiableTx", tx)
	}

	signers, err := sigTx.GetSigners()
	if err != nil {
		return err
	}
	for _, signer := range signers {
		encoded, err := h.addressCodec.BytesToString(signer)
		if err != nil {
			return fmt.Errorf("invalid bech32 address: %s, err: %w", signer, err)
		}
		if _, ok := h.blocklist[encoded]; ok {
			return fmt.Errorf("signer is blocked: %s", encoded)
		}
	}
