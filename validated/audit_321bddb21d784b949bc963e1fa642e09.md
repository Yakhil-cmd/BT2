### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Switch Price Provider and Sandwich LP Funds - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.createPool` accepts `priceProviderTimelock = 0` with no minimum-delay enforcement. When the timelock is zero, `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` can be called in the same block (or the same transaction), making the timelock guard completely ineffective. A pool admin can atomically rotate to a price provider that quotes a manipulated bid/ask, sandwiching the switch with swaps to drain LP principal.

### Finding Description

`createPool` stores the caller-supplied `priceProviderTimelock` verbatim: [1](#0-0) [2](#0-1) 

Any value other than `type(uint256).max` is treated as "mutable with delayed execution." No minimum delay is enforced. When `priceProviderTimelock = 0`, `proposePoolPriceProvider` computes:

```
executeAfter = block.timestamp + 0  →  executeAfter = block.timestamp
``` [3](#0-2) 

`executePoolPriceProviderUpdate` then checks:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp  →  false  →  no revert
``` [4](#0-3) 

Both calls succeed in the same block. The only validation on the new provider is that its `token0()`/`token1()` match the pool's tokens: [5](#0-4) 

No price-range or oracle-source check is performed. An attacker-controlled provider that returns the correct token addresses but an arbitrary bid/ask passes validation.

Every swap reads the live oracle price from the active provider: [6](#0-5) [7](#0-6) 

Switching the provider therefore immediately changes the price used for all subsequent swaps in the same block.

### Impact Explanation

A pool admin who created the pool with `priceProviderTimelock = 0` can, in a single atomic bundle:

1. **Swap** token1 → token0 at the current (honest) oracle price.
2. **Propose + execute** a price provider switch to a contract that quotes token0 at a significantly higher price.
3. **Swap** token0 → token1 at the inflated price.

The pool pays out token1 at the inflated rate in step 3, drawing down LP reserves. The difference between the two prices is extracted from LP principal. Because the pool's bin accounting (`binTotals.scaledToken0/1`) is updated on every swap, the loss is permanent and socialised across all LPs.

This is a direct, quantifiable loss of LP principal — the exact analog of the external report's bad-debt extraction via oracle sandwich.

### Likelihood Explanation

- Any pool creator can set `priceProviderTimelock = 0`; the factory imposes no minimum.
- The pool admin role is "semi-trusted only inside caps and timelocks." With a zero timelock there is no constraint, so the admin operates outside the intended trust boundary.
- The attack requires no external oracle event; the admin fully controls both the price provider switch and the surrounding swaps.
- Pools with attractive TVL and a zero timelock are the target; this configuration is observable on-chain before depositing, but many LPs do not inspect `priceProviderTimelock`.

Likelihood: **Medium** (requires a pool admin who is malicious or compromised, but the factory provides no guard against this configuration).

### Recommendation

Enforce a minimum `priceProviderTimelock` in `_validatePoolParameters` for mutable pools:

```solidity
uint256 MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

if (params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

This ensures that even for mutable pools, LPs always have at least one day to observe a pending provider change and exit before it takes effect.

### Proof of Concept

```
Setup:
  Pool created with priceProviderTimelock = 0, token0 = ETH, token1 = USDC
  Honest provider quotes ETH at $2000 (bid/ask ≈ $1999/$2001)
  Pool has $1 000 000 USDC and 500 ETH in LP reserves

Attack (single block, single tx via attacker contract):

  Step 1 — swap(zeroForOne=false, amountSpecified=+500_000 USDC)
    Pool pays out ~250 ETH at $2000/ETH
    Attacker receives 250 ETH, pool receives 500 000 USDC

  Step 2 — proposePoolPriceProvider(pool, maliciousProvider)
    maliciousProvider.token0() == ETH, token1() == USDC  ✓
    maliciousProvider.getBidAndAskPrice() returns bid=$2400, ask=$2401
    executeAfter = block.timestamp + 0 = block.timestamp

  Step 3 — executePoolPriceProviderUpdate(pool)
    block.timestamp < block.timestamp → false → no revert
    pool.priceProvider = maliciousProvider

  Step 4 — swap(zeroForOne=true, amountSpecified=+250 ETH)
    Pool now prices ETH at $2400
    Pool pays out ~600 000 USDC for 250 ETH

  Net result:
    Attacker spent 500 000 USDC, received 250 ETH, then spent 250 ETH, received 600 000 USDC
    Attacker profit: 100 000 USDC
    LP loss:         100 000 USDC (pool reserves reduced by $100k relative to fair value)
```

The root cause is `MetricOmmPoolFactory.sol` lines 213 and 487–499: no minimum timelock is enforced at creation, and the `block.timestamp < block.timestamp` guard is trivially bypassed when `timelock = 0`. [2](#0-1) [8](#0-7)
