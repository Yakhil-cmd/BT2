Audit Report

## Title
Permanent Unvalidated-Pool Memory Leak via `start_block_height = u64::MAX` in DKG Message — (`rs/artifact_pool/src/dkg_pool.rs`)

## Summary

A Byzantine P2P peer below the consensus fault threshold can craft `dkg::Message` artifacts with `content.dkg_id.start_block_height = u64::MAX`. Because `DkgMessageId.height` is derived verbatim from that field, the purge predicate (`id.height < height`) is permanently false for any finite height, and the validator silently skips (rather than invalidates) height-mismatched messages. The result is an unbounded, permanent accumulation of entries in the unvalidated pool on every honest replica that receives such messages.

## Finding Description

**Root cause — ID derivation with no bounds check:**
`DkgMessageId::from` copies `start_block_height` directly into `DkgMessageId.height` with no validation or clamping. [1](#0-0) 

**Unvalidated pool insertion with no admission guard:**
`DkgPoolImpl::insert` places any received artifact directly into the unvalidated section without inspecting `start_block_height`. [2](#0-1) 

**Validator silently skips, does not invalidate:**
`validate_dealings_for_dealer` returns an empty `Mutations::new()` when `start_block_height != dkg_start_height`. A message with `start_block_height = u64::MAX` will never equal any real DKG start height, so it is skipped on every validation pass and never removed via the validation path. [3](#0-2) 

**Purge predicate never fires:**
`purge` evicts entries where `id.height < height`. For `id.height = u64::MAX`, this predicate is false for every finite purge height, including `u64::MAX - 1` and `u64::MAX` itself (strict less-than). [4](#0-3) 

**Invariant violated:**
The `DkgPool` interface states that all messages in both sections must correspond to the current DKG start height. Messages with `height = u64::MAX` permanently violate this invariant. [5](#0-4) 

No existing check in the insertion, validation, or purge path removes or rejects such a message. The `ChangeAction::HandleInvalid` path exists and is used for other invalid conditions (e.g., wrong replica version, missing config for local DKG), but is not triggered for height mismatches. [6](#0-5) 

## Impact Explanation

Every honest replica that receives a crafted message accumulates it in its unvalidated pool indefinitely. Because `DkgMessageId` is keyed by `(hash, height)`, a single Byzantine peer can inject many distinct messages (varying dealing content to produce different hashes) all sharing `height = u64::MAX`. None are ever purged. Over time this constitutes an unbounded memory leak on every honest replica in the subnet, potentially causing OOM conditions and replica crashes, which maps to: **High ($2,000–$10,000) — Application/platform-level DoS, subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation

The attack requires only a single Byzantine P2P peer — a replica that has been compromised but is below the consensus fault threshold. No privileged access, no key material, no governance majority, and no threshold corruption is needed. The peer sends crafted `DkgMessage` protobuf artifacts over the existing P2P gossip channel. The attack is repeatable and unbounded: each new message with a distinct dealing hash adds a permanent entry. The path is fully exercisable with a unit test against `DkgPoolImpl` directly.

## Recommendation

1. **Invalidate, do not skip, messages with unreachable `start_block_height`.** In `validate_dealings_for_dealer`, replace the silent `Mutations::new()` return for height mismatches with `ChangeAction::HandleInvalid` when the height is provably unreachable (e.g., greater than the current finalized height by more than the maximum DKG interval length, or simply equal to `u64::MAX`). [3](#0-2) 

2. **Add an upper-bound admission check in `DkgPoolImpl::insert`** that rejects messages whose `start_block_height` exceeds a reasonable future horizon relative to `current_start_height`. [2](#0-1) 

3. **Alternatively, change the purge predicate** from strict less-than (`<`) to less-than-or-equal (`<=`) so that `Purge(u64::MAX)` would evict such entries — though this is a weaker fix and does not address the root cause. [7](#0-6) 

## Proof of Concept

```rust
// Directly exercisable as a unit test in rs/artifact_pool/src/dkg_pool.rs
#[test]
fn test_umax_height_permanent_leak() {
    let mut pool = DkgPoolImpl::new(MetricsRegistry::new(), no_op_logger(), Height::from(0));

    // Byzantine peer crafts a message with start_block_height = u64::MAX
    let msg = make_message(Height::from(u64::MAX), node_test_id(1));
    pool.insert(UnvalidatedArtifact {
        message: msg,
        peer_id: node_test_id(99),
        timestamp: UNIX_EPOCH,
    });

    // Purge at u64::MAX - 1: u64::MAX < (u64::MAX - 1) is false → not removed
    pool.apply(vec![ChangeAction::Purge(Height::from(u64::MAX - 1))]);
    assert_eq!(pool.get_unvalidated().count(), 1); // still present

    // Purge at u64::MAX: u64::MAX < u64::MAX is false → still not removed
    pool.apply(vec![ChangeAction::Purge(Height::from(u64::MAX))]);
    assert_eq!(pool.get_unvalidated().count(), 1); // permanent leak confirmed
}
```

The `make_message` helper already exists in the test module. [8](#0-7)
