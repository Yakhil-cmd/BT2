### Title
Precision Loss in `native_per_pubdata` Calculation Allows Pubdata Undercharging — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio used to charge L2 transactions for pubdata consumption is computed via unscaled floor integer division (`pubdata_price / native_price`). When `pubdata_price < native_price`, the result truncates to zero, making pubdata completely free. Even when the ratio is non-zero, systematic rounding-down causes persistent undercharging. This is the direct analog of the reported Bottle module bug: an accumulator-style accounting value is derived by dividing one quantity by another without a scaling factor, causing precision loss that benefits the user at the protocol's expense.

---

### Finding Description

In `validation_impl.rs`, the two per-unit resource ratios are computed as follows:

```rust
// line 135 — uses ceiling division (rounds UP, protocol-favorable)
u256_try_to_u64(&gas_price.div_ceil(native_price))

// line 142 — uses floor division (rounds DOWN, user-favorable)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`native_per_gas` deliberately uses `div_ceil` to ensure the protocol is never undercompensated for computation. `native_per_pubdata` uses `wrapping_div` (floor), introducing a systematic downward bias.

`native_per_pubdata` is then passed into `get_resources_to_charge_for_pubdata`, which computes the native cost of all pubdata generated by the transaction:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

If `native_per_pubdata = 0`, the returned `to_charge_for_pubdata` resource is zero. In `compute_gas_refund`, this zero charge is applied unconditionally:

```rust
resources.charge_unchecked(&to_charge_for_pubdata);
``` [3](#0-2) 

The `deltaGas` correction that follows cannot compensate, because it is derived from `native_used`, which itself reflects the undercharged (zero) pubdata cost:

```rust
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
``` [4](#0-3) 

The same floor-division pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [5](#0-4) 

---

### Impact Explanation

**Scenario A — `pubdata_price < native_price` (truncation to zero):**
`native_per_pubdata = 0`. Every transaction pays zero native resources for pubdata regardless of how many storage slots it writes. An attacker can write the maximum number of storage slots per transaction at no pubdata cost, forcing the operator to publish large amounts of L1 data without compensation. This is a direct financial loss to the operator and a denial-of-service vector against the pubdata budget.

**Scenario B — `pubdata_price` slightly above `native_price` (non-zero but truncated):**
The truncation error per pubdata byte is up to `(native_price - 1) / native_price` of the true cost. Across many bytes of pubdata, this accumulates into a material undercharge. For example, with `native_price = 1000` and `pubdata_price = 1999`, `native_per_pubdata = 1` instead of the correct `~2`, halving the pubdata charge.

---

### Likelihood Explanation

The operator sets `native_price` and `pubdata_price` independently as block-level parameters. There is no on-chain constraint enforcing `pubdata_price >= native_price`. During periods of low L1 data costs relative to proving costs, `pubdata_price < native_price` is a realistic operating condition. Any unprivileged user submitting a standard L2 transaction during such a block can exploit this with no special access. The attacker only needs to submit a transaction that writes storage (e.g., a simple ERC-20 transfer or any contract that modifies state).

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — precision loss)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that when `pubdata_price < native_price`, `native_per_pubdata` rounds up to 1 rather than truncating to 0, and that the accumulated undercharge across many pubdata bytes is bounded to at most 1 native unit per byte rather than growing unboundedly.

---

### Proof of Concept

1. Operator sets block context: `native_price = 1000`, `pubdata_price = 999` (a realistic scenario where L1 data is cheap relative to proving cost).
2. `native_per_pubdata = 999 / 1000 = 0` (floor division).
3. Attacker submits a transaction writing 100 new storage slots (each slot write produces ~64 bytes of pubdata → ~6400 bytes total).
4. In `get_resources_to_charge_for_pubdata`: `native = 6400 * 0 = 0`.
5. `to_charge_for_pubdata` resource is zero; `compute_gas_refund` charges nothing for pubdata.
6. `native_used` does not include pubdata cost; `delta_gas = 0`.
7. Attacker pays only for EVM computation (gas for SSTORE opcodes), not for the 6400 bytes of pubdata the operator must publish to L1.
8. Repeat across many transactions to exhaust the block's pubdata budget at zero pubdata cost. [6](#0-5) [7](#0-6) [8](#0-7)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
