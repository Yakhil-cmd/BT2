### Title
Gateway `skip_stateful_validations` Allows Unauthorized Invoke Transactions to Bypass Signature Validation, Enabling Mempool DoS of Legitimate Post-Deploy Invokes — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the Apollo gateway's stateful validator unconditionally skips the `__validate__` entry point (signature check) for any invoke transaction with `nonce=1` targeting an undeployed account that has a pending `deploy_account` transaction in the mempool. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can submit a crafted invoke transaction with an arbitrary/invalid signature for the victim's address. The gateway accepts it without any signature verification. The attacker's transaction occupies the `(address, nonce=1)` slot in the mempool, causing the victim's legitimate invoke transaction to be rejected as `DuplicateNonce`.

---

### Finding Description

The gateway's stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block` returns `true` for the sender address [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in the execution flags: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate = false`, the `__validate__` call is entirely skipped and `Ok(())` is returned immediately: [4](#0-3) 

This means **no signature verification is performed at the gateway level** for any invoke transaction that satisfies the three conditions above. The transaction is then forwarded to the mempool via `add_tx`.

The mempool's `validate_incoming_tx` only rejects a transaction if its nonce is strictly less than the account's resolved nonce: [5](#0-4) 

For `tx_nonce=1` and `account_nonce=0`, this check passes. The attacker's transaction is inserted into the pool.

When the victim subsequently submits their legitimate invoke transaction with `nonce=1`, the mempool rejects it with `DuplicateNonce`: [6](#0-5) 

The `account_tx_in_pool_or_recent_block` check that gates the skip is trivially satisfiable by any observer of the mempool: [7](#0-6) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions and rejects valid transactions before sequencing.**

An attacker with no special privileges can:
1. Observe a victim's `deploy_account` transaction in the mempool (via P2P gossip or RPC)
2. Submit an invoke transaction for the victim's undeployed address with `nonce=1` and an arbitrary/invalid signature
3. The gateway accepts it without calling `__validate__` (no signature check)
4. The victim's legitimate invoke transaction with `nonce=1` is rejected as `DuplicateNonce`
5. The attacker's transaction eventually fails at execution time (invalid signature), but the victim's transaction slot has been consumed

The victim's `deploy_account` executes successfully, but their intended first invoke transaction never reaches execution. The victim must detect the failure, wait for the attacker's transaction to be evicted or executed, and resubmit — a persistent, repeatable DoS.

---

### Likelihood Explanation

**Medium.** The attack requires:
- Observing a `deploy_account` transaction in the mempool (straightforward via P2P or RPC)
- Submitting a crafted invoke transaction before the victim's invoke transaction arrives (a race, but the attacker has the full mempool propagation window)

No privileged access, special keys, or on-chain funds are required. The attacker only needs to pay the transaction fee for the crafted invoke.

---

### Recommendation

The `skip_stateful_validations` feature is designed for UX (allowing `deploy_account + invoke` to be submitted atomically). However, it must not allow arbitrary third parties to inject unsigned transactions into the victim's nonce space.

**Option 1 (preferred):** Wrap the `__validate__` call in a try-catch. If it fails solely because the account does not exist yet (contract not found / class hash zero), accept the transaction anyway. This preserves the UX intent while ensuring only the legitimate account owner (who holds the private key) can submit the invoke.

**Option 2:** Verify that the invoke transaction's signature is consistent with the public key embedded in the `deploy_account` transaction's constructor calldata before skipping `__validate__`.

**Option 3:** Remove the `skip_stateful_validations` shortcut entirely and require users to submit `deploy_account` and `invoke` in separate blocks.

---

### Proof of Concept

```
1. Alice submits deploy_account for address A (valid signature, nonce=0).
   → Accepted into mempool. account_tx_in_pool_or_recent_block(A) = true.

2. Attacker observes Alice's deploy_account in the mempool.

3. Attacker submits invoke(sender=A, nonce=1, calldata=arbitrary, signature=garbage).
   Gateway stateful validation:
     - account_nonce = 0 (A not deployed)
     - tx_nonce = 1
     - skip_stateful_validations: nonce==1 && account_nonce==0 && pool_has_A → true
     - run_validate_entry_point(skip_validate=true) → __validate__ skipped
   → Attacker's tx accepted into mempool at slot (A, nonce=1).

4. Alice submits invoke(sender=A, nonce=1, calldata=legitimate, signature=valid).
   Gateway → validate_by_mempool → Mempool::validate_tx:
     - (A, nonce=1) already occupied → DuplicateNonce error
   → Alice's tx REJECTED.

5. Alice's deploy_account executes. Her invoke tx is absent from the mempool.
   Attacker's invoke tx executes → fails (__validate__ called by batcher, invalid sig).

6. Alice must detect the failure and resubmit. Attacker can repeat indefinitely.
```

The broken invariant: the gateway's admission path must not accept a transaction for an account without verifying the transaction is authorized by that account's owner. The `skip_stateful_validations` shortcut violates this invariant for the specific case of `nonce=1` invoke transactions targeting undeployed accounts with a pending `deploy_account` in the mempool. ```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
