### Title
`div_ceil` Rounding in `native_per_gas` Causes Systematic Undercharging for Native Resource Consumption - (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`native_per_gas` is computed with ceiling division (`div_ceil`) during transaction validation, but the same inflated value is then used as the denominator in the `delta_gas` formula inside `compute_gas_refund`. This creates a rounding direction mismatch: the user receives more native resources than they paid for, and the `delta_gas` correction mechanism undercharges them for the native consumption. Any unprivileged L2 transaction sender can exploit this by choosing a `gas_price` that is not a multiple of `native_price`.

---

### Finding Description

The documentation specifies the intended formula as:

```
nativePerGas := gasPrice / nativePrice
deltaGas     := (nativeUsed / nativePerGas) - gasUsed
```

In `validation_impl.rs`, `native_per_gas` is computed with **ceiling** division:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

This inflated `native_per_gas` is then passed into `compute_gas_refund`, where it is used in two places:

1. To compute the native limit: `full_native_limit = gas_limit * native_per_gas`
2. To compute `delta_gas`: `(native_used / native_per_gas) - gas_used` [2](#0-1) 

Substituting `native_used = gas_limit * native_per_gas - remaining_native` into the `delta_gas` formula:

```
delta_gas = floor((gas_limit * native_per_gas - remaining_native) / native_per_gas) - gas_used
          = gas_limit - ceil(remaining_native / native_per_gas) - gas_used
```

When `native_per_gas` is inflated by `div_ceil`, `ceil(remaining_native / native_per_gas)` is **smaller**, so `delta_gas` is **smaller**, meaning the user is charged less gas for native consumption than they should be.

**Concrete example** (`gas_price = 3`, `native_price = 2`, `gas_limit = 100`, `gas_used_from_ergs = 60`, user consumes 150 native units):

| Scenario | `native_per_gas` | `native_limit` | `delta_gas` | Final `gas_used` | Fee paid |
|---|---|---|---|---|---|
| `div_ceil` (actual) | 2 | 200 | 75 − 60 = 15 | 75 | 75 × 3 = **225** |
| Exact ratio 1.5 | 1.5 | 150 | 100 − 60 = 40 | 100 | 100 × 3 = **300** |

The user pays **225** instead of **300** — a 25% discount — and also receives a larger native resource budget (200 vs. 150).

The same `div_ceil` path exists for L1→L2 priority transactions: [3](#0-2) 

---

### Impact Explanation

Every transaction where `gas_price mod native_price != 0` results in:

1. The sender receiving more native resources (RISC-V proving cycles) than their fee covers.
2. The `delta_gas` correction undercharging the sender for native consumption.
3. The operator/protocol receiving less fee than the actual proving cost warrants.

The maximum per-transaction fee shortfall is bounded by `gas_limit / native_per_gas_ceil * gas_price`. For a transaction with `gas_limit = 30,000,000`, `gas_price = native_price + 1`, and `native_per_gas_ceil = 2`, this is up to `15,000,000 * gas_price` in underpaid fee. Cumulatively across all transactions in a block, this represents a systematic drain on operator revenue and a subsidy of proving costs by the protocol.

---

### Likelihood Explanation

This is triggered by any L2 or L1→L2 transaction where `gas_price` is not an exact multiple of `native_price`. Since `native_price` is an operator-set parameter and `gas_price` is user-controlled, this condition holds for the vast majority of real transactions. No special privileges are required; any unprivileged transaction sender triggers this path. A sophisticated attacker can maximize the rounding error by setting `gas_price = native_price + 1`, which forces `native_per_gas = 2` regardless of the true ratio.

---

### Recommendation

Use two separate values for `native_per_gas`:

- **For the native limit** (giving users enough resources): keep `div_ceil` to avoid under-provisioning.
- **For the `delta_gas` denominator** (charging users for consumption): use `div_floor` (or the exact integer quotient) so that the fee correctly reflects actual native consumption.

Alternatively, store both `native_per_gas_ceil` and `native_per_gas_floor` in the transaction context and use each in its appropriate role.

---

### Proof of Concept

The rounding mismatch can be demonstrated with the following values:

- `native_price = 2`, `gas_price = 3` → `native_per_gas = ceil(3/2) = 2` (true ratio: 1.5)
- `gas_limit = 100`, EVM `gas_used = 60`, native consumed = 150 units

**Step 1 — Validation** (`validation_impl.rs:135`):
```
native_per_gas = ceil(3 / 2) = 2   // inflated by 1
native_prepaid = 100 * 2 = 200     // user gets 200 native units, true budget = 150
```

**Step 2 — Refund** (`refund_calculation.rs:62–72`):
```
full_native_limit = 100 * 2 = 200
native_used       = 200 - 50 = 150  (remaining = 50)
delta_gas         = floor(150 / 2) - 60 = 75 - 60 = 15
final gas_used    = 60 + 15 = 75
fee paid          = 75 * 3 = 225
```

**Expected** (with exact ratio 1.5):
```
native_limit = 100 * 1.5 = 150
native_used  = 150
delta_gas    = 150 / 1.5 - 60 = 100 - 60 = 40
gas_used     = 100
fee paid     = 100 * 3 = 300
```

The user pays **225** instead of **300**, consuming 150 native units while only paying for 75 gas worth of native work. The operator is shortchanged by 75 × 3 = 225 tokens per transaction. [4](#0-3) [5](#0-4) [6](#0-5)

```markdown
First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
