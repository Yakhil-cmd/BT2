### Title
Zero-Address `erc20_fallback_address` Causes Permanent Freeze of Bridged ERC-20 Tokens in Silo Mode — (`engine/src/contract_methods/silo/mod.rs`, `engine-types/src/parameters/silo.rs`)

---

### Summary

In Silo mode, when the `erc20_fallback_address` is set to `Address::zero()` — which is the Rust `Default` value of the `SiloParamsArgs` struct — all NEP-141 tokens bridged by non-whitelisted recipients are permanently frozen. The `set_silo_params` and `set_erc20_fallback_address` contract entry points perform no zero-address validation, so a deployer who forgets to supply a real fallback address (or who passes a default-constructed `SiloParamsArgs`) silently stores `0x0000…0000` as the fallback. Every subsequent `ft_on_transfer` call for a non-whitelisted recipient then mints ERC-20 tokens to the zero address, while the corresponding NEP-141 tokens remain locked inside the Aurora contract — a permanent, irrecoverable fund freeze.

---

### Finding Description

**Root cause — no zero-address guard on the fallback address setter**

`SiloParamsArgs` derives `Default`, so its `erc20_fallback_address` field silently initialises to `Address::zero()`:

```rust
// engine-types/src/parameters/silo.rs  L15-L24
#[derive(Debug, Default, Clone, PartialEq, Eq, BorshSerialize, BorshDeserialize)]
pub struct SiloParamsArgs {
    pub fixed_gas: EthGas,
    pub erc20_fallback_address: Address,   // defaults to Address::zero()
}
```

The two contract entry points that write this value perform no zero-address check:

```rust
// engine/src/lib.rs  L829-L839
pub extern "C" fn set_silo_params() {
    ...
    let args: Option<SiloParamsArgs> = io.read_input_borsh().sdk_unwrap();
    silo::set_silo_params(&mut io, args);   // no zero-address guard
}

// engine/src/lib.rs  L806-L815
pub extern "C" fn set_erc20_fallback_address() {
    ...
    let args: Erc20FallbackAddressArgs = io.read_input_borsh().sdk_unwrap();
    silo::set_erc20_fallback_address(&mut io, args.address);  // no zero-address guard
}
```

The underlying storage writer also has no guard:

```rust
// engine/src/contract_methods/silo/mod.rs  L65-L73
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes());  // stores 20 zero bytes if address is zero
    } else {
        io.remove_storage(&key);
    }
}
```

**Trigger path — `receive_erc20_tokens`**

When a user bridges NEP-141 tokens via `ft_on_transfer`, `receive_erc20_tokens` is called. If the recipient is not whitelisted and the stored fallback address is `Some(Address::zero())`, the recipient is silently replaced with the zero address:

```rust
// engine/src/engine.rs  L818-L822
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;   // becomes Address::zero()
}
```

The ERC-20 `mint` call then succeeds (minting to the zero address is valid EVM), the NEP-141 tokens are consumed from the sender and held by the Aurora contract, and the ERC-20 tokens are irrecoverably at `0x0000…0000`.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

- The NEP-141 tokens are transferred from the user to the Aurora contract as part of `ft_on_transfer`. They cannot be returned because the EVM call succeeds.
- The ERC-20 tokens are minted to `Address::zero()`, an address with no known private key. They are permanently inaccessible.
- Every non-whitelisted bridge transfer made while the zero fallback address is active results in a total, irrecoverable loss of the bridged amount.

---

### Likelihood Explanation

`SiloParamsArgs` derives `Default` and its `erc20_fallback_address` field is a plain `Address` (not `Option<Address>`), so the zero address is the natural Rust default. A deployer who calls `set_silo_params(Some(SiloParamsArgs { fixed_gas: X, ..Default::default() }))` — a common Rust idiom — silently stores the zero address. There is no compile-time or runtime warning. The existing test suite itself uses `Address::zero()` as the fallback address constant (`ERC20_FALLBACK_ADDRESS: Address = Address::zero()`), demonstrating that this value passes through the entire call chain without error. Any Silo-mode deployment that omits or forgets the fallback address field is immediately vulnerable.

---

### Recommendation

1. **Reject the zero address in both setters.** Add an explicit guard in `set_erc20_fallback_address` (and in `set_silo_params` before delegating):

```rust
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        if address == Address::zero() {
            // return an error / panic with a clear message
            sdk::panic_utf8(b"ERR_ZERO_FALLBACK_ADDRESS");
        }
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }
}
```

2. **Change `erc20_fallback_address` in `SiloParamsArgs` to `Option<Address>`** so that the absence of a fallback is explicit and cannot be confused with the zero address.

3. **Update the test constant** `ERC20_FALLBACK_ADDRESS` away from `Address::zero()` to a non-zero sentinel, so tests do not mask the bug.

---

### Proof of Concept

**Setup:**
1. Deploy Aurora Engine in Silo mode.
2. Owner calls `set_silo_params` with `SiloParamsArgs { fixed_gas: 1_000_000, erc20_fallback_address: Address::zero() }` (the Rust default).
3. Owner enables the `Address` whitelist (`set_whitelist_status { kind: Address, active: true }`).
4. Deploy a NEP-141 token and its ERC-20 mirror via `deploy_erc20_token`.

**Attack:**
5. A user (`alice`, whose EVM address is **not** whitelisted) calls `ft_transfer_call` on the NEP-141 contract, transferring `N` tokens to the Aurora contract with `msg = alice_evm_address_hex`.
6. Aurora's `ft_on_transfer` → `receive_erc20_tokens` is invoked.
7. `get_erc20_fallback_address` returns `Some(Address::zero())`.
8. `is_allow_receive_erc20_tokens` returns `false` (alice not whitelisted).
9. `recipient` is overwritten with `Address::zero()`.
10. The ERC-20 `mint(address(0), N)` call succeeds.

**Result:**
- Alice's `N` NEP-141 tokens are locked in the Aurora contract.
- `N` ERC-20 tokens are at `0x0000000000000000000000000000000000000000`.
