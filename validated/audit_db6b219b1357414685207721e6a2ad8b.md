### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Single-Party Signature Forgery - (File: `src/frost/mod.rs`, `src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

`assert_key_invariants` (used by `keygen`/`reshare`) enforces `threshold >= 2`, but `assert_sign_inputs` (used by every FROST `sign_v1`/`sign_v2` call) only enforces the upper bound `threshold <= participants.len()`. A caller can invoke FROST signing with `threshold = 1` (or `0`), which was explicitly rejected at key-generation time, allowing a single participant to produce a valid threshold signature unilaterally.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
if threshold > participants.len() { ... ThresholdTooLarge }
if threshold < 2 { ... ThresholdTooSmall }   // ← lower bound enforced
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs`, which is the sole validation gate for both `frost::eddsa::sign_v1` and `frost::redjubjub::sign`, only checks the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← NO lower-bound check; threshold = 1 or 0 is accepted
``` [2](#0-1) 

The same omission exists in the `presign` function in `src/frost/mod.rs`: [3](#0-2) 

Both `frost::eddsa::sign_v1` and `frost::redjubjub::sign` delegate entirely to `assert_sign_inputs` for threshold validation: [4](#0-3) [5](#0-4) 

The accepted `threshold` value is then forwarded directly into `construct_key_package`, which builds the FROST `KeyPackage` with `min_signers = threshold`: [6](#0-5) 

With `min_signers = 1`, the FROST `round2::sign` and `aggregate` calls succeed with a single participant's share, completely bypassing the multi-party threshold requirement.

---

### Impact Explanation

**Critical — Unauthorized creation of a valid threshold signature for attacker-chosen inputs.**

A single participant who controls the coordinator role can call `sign_v1` with `threshold = 1` over any message. The FROST aggregation step (`aggregate(...)`) will produce a valid signature verifiable against the group's public key, without any cooperation from other key-holders. This directly violates the core security property of the threshold scheme.

---

### Likelihood Explanation

The `threshold` parameter is caller-supplied at signing time and is not cross-checked against the threshold recorded in the `KeygenOutput`. Any participant who also acts as coordinator (a common deployment pattern) can trivially pass `threshold = 1`. No special privileges, leaked keys, or cryptographic breaks are required — only the ability to call the public API.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` to `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Additionally, apply the same fix to the `presign` function in `src/frost/mod.rs`. Long-term, store the threshold inside `KeygenOutput` and verify at signing time that the caller-supplied threshold matches the one agreed upon during DKG.

---

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (accepted, `>= 2`).
2. Call `frost::eddsa::sign_v1` with the same 3 participants but `threshold = 1`.
3. `assert_sign_inputs` passes (only upper bound `1 <= 3` is checked).
4. `construct_key_package` builds a `KeyPackage` with `min_signers = 1`.
5. The coordinator alone calls `round2::sign` and `aggregate`; both succeed.
6. The resulting `Signature` verifies correctly against the group public key — a fully valid threshold signature produced by one party. [7](#0-6) [8](#0-7)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
