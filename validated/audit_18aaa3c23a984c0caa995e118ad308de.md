### Title
Missing Lower-Bound Threshold Validation in FROST Signing Allows `threshold = 1`, Producing Corrupted Signing Outputs — (`src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` function used to validate FROST signing parameters does not enforce a minimum threshold of 2, unlike the DKG keygen validation (`assert_key_invariants`). A caller or malicious coordinator can supply `threshold = 1` to the FROST signing protocol. Because all keys are generated with `threshold >= 2`, using `threshold = 1` during signing causes Lagrange interpolation to be performed with a single share evaluated at the wrong polynomial degree, producing a cryptographically invalid and unusable signature. Honest participants complete the protocol and accept the output without detecting the inconsistency.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces a strict lower bound:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

This means every key generated by this library has a polynomial of degree `threshold - 1 >= 1`, so no participant's share equals the secret.

`assert_sign_inputs` in `src/frost/mod.rs` performs only an upper-bound check:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

There is **no lower-bound check** (`threshold >= 2`) in `assert_sign_inputs`. The same omission exists in the FROST `presign` function: [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no invariant enforced at construction time: [4](#0-3) 

So a caller can freely construct `ReconstructionLowerBound(1)` and pass it to `assert_sign_inputs` or `presign`. Both functions will accept it without error.

**What happens at `threshold = 1`:** FROST signing aggregates signature shares using Lagrange interpolation over the participant set. With `threshold = 1`, only one participant's share is used, and its Lagrange coefficient is 1 (trivially). The resulting aggregate is `f(x_i)` for a degree-`(t-1)` polynomial where `t >= 2`, not `f(0)` (the actual secret). The produced signature is cryptographically invalid and will fail verification. Honest participants complete the protocol and return a result they believe is correct, but the output is unusable.

This is the direct analog to the external report: just as `leverage = 1.0x` is accepted by `_leverageCheck` but produces the same position as `1.7x` (a non-monotonic, unexpected outcome at the boundary), `threshold = 1` is accepted by `assert_sign_inputs` but produces a corrupted signing output inconsistent with any valid keygen threshold.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

Every honest participant executes the signing protocol to completion and receives what appears to be a valid `PresignOutput` or final signature. No error is raised. The output is silently invalid because the Lagrange interpolation was performed with the wrong degree assumption. The mismatch between the keygen threshold (`>= 2`) and the signing threshold (`1`) is never detected by the protocol.

---

### Likelihood Explanation

**Medium.** The `assert_sign_inputs` function is a public API entry point. Any caller — including a malicious coordinator who controls the signing session parameters — can supply `threshold = 1`. The `ReconstructionLowerBound` type imposes no construction-time invariant, so no additional privilege or cryptographic capability is required. The only prerequisite is the ability to call the signing API, which is the normal unprivileged caller role.

---

### Recommendation

Add the same lower-bound check present in `assert_key_invariants` to both `assert_sign_inputs` and the FROST `presign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted in `assert_sign_inputs` after the `participants.len() < 2` check, and in `presign` after the corresponding threshold upper-bound check. Optionally, enforce the invariant at the `ReconstructionLowerBound` construction site so it is impossible to construct a value below 2.

---

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
