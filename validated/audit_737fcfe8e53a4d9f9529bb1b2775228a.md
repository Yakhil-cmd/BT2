### Title
Integer Division Truncation in `native_per_pubdata` Causes Pubdata to Be Undercharged When `pubdata_price < native_price` — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio is computed with floor division (`wrapping_div`), while the analogous `native_per_gas` ratio uses ceiling division (`div_ceil`). When `pubdata_price < native_price`, `native_per_pubdata` truncates to zero, making all pubdata effectively free. Any unprivileged transaction sender can then generate unbounded pubdata at no native-resource cost, causing the operator to absorb L1 data-publication costs without collecting fees.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` line 142:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is Rust's standard integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 500`, `native_price = 1000`), the result is `0`.

Compare this to `native_per_gas`, which explicitly uses ceiling division to prevent the same truncation:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(...)
``` [2](#0-1) 

The same floor-division bug is replicated in the off-chain helper `api/src/helpers.rs`:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata = 0`, two downstream effects occur:

**1. No native resources are reserved for pubdata upfront** — in `create_resources_for_tx`:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
``` [4](#0-3) 

`intrinsic_pubdata_overhead` becomes `0`, so no native budget is withheld for pubdata.

**2. No native resources are charged for pubdata after execution** — in `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

`native = 0` regardless of how many pubdata bytes the transaction wrote.

There is no special-case guard for `native_per_pubdata == 0` analogous to the `native_per_gas == 0` guard that sets `native_limit = u64::MAX - 1`: [6](#0-5) 

---

### Impact Explanation

When `pubdata_price < native_price`, every transaction in the block pays zero native resources for pubdata, regardless of how many storage slots it writes or how many L1 log entries it emits. An attacker who observes this price configuration can submit transactions that maximally dirty storage (maximizing pubdata bytes) at no additional cost. The operator must still publish all pubdata to L1 and pay the associated DA fees, but collects nothing for it. This is a direct resource-accounting loss for the operator and a potential griefing / economic-DoS vector.

**Impact: Medium** — operator suffers unbounded pubdata cost without fee recovery; no user funds are directly stolen, but the economic model of the rollup is broken for the affected block.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are both operator-set block-context parameters. In practice, `pubdata_price` reflects the L1 blob/calldata market price (volatile, denominated in wei/byte) while `native_price` reflects the proving-cycle cost (a different unit). It is realistic — especially during periods of low L1 gas prices or high proving costs — for `pubdata_price < native_price` to hold. The truncation is silent: no error is raised, no log is emitted, and the transaction proceeds normally.

**Likelihood: Medium** — requires a specific but plausible price relationship; no attacker action is needed beyond submitting a storage-heavy transaction.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [3](#0-2) 

---

### Proof of Concept

**Setup**: operator sets `native_price = 1000`, `pubdata_price = 999`.

**Step 1** — `native_per_pubdata` calculation:
```
native_per_pubdata = 999 / 1000 = 0   (floor division)
```

**Step 2** — intrinsic pubdata overhead in `create_resources_for_tx`:
```
intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0
```
No native budget is reserved for pubdata.

**Step 3** — attacker submits a transaction that writes to 100 new storage slots (each slot write generates pubdata). After execution, `get_resources_to_charge_for_pubdata` is called:
```
native = current_pubdata_spent * 0 = 0
```
The transaction is charged 0 native resources for pubdata, regardless of how many bytes were published.

**Step 4** — the operator must publish all pubdata to L1 at the real market price, but collected 0 fees for it. Repeating this across many transactions in a block causes unbounded operator loss.

With `div_ceil`, `native_per_pubdata = ceil(999/1000) = 1`, and each pubdata byte correctly costs 1 native unit.
