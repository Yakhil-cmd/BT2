### Title
Unbounded Execution Window on `execute_scheduled` Enables Timing Attacks on Scheduled XCC Promises — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `execute_scheduled` function is intentionally callable by any unprivileged NEAR account with no deadline or expiry enforced on the stored `PromiseArgs`. When a user schedules a price-sensitive cross-contract call (e.g., a DEX swap on NEAR) via `CrossContractCallArgs::Delayed`, an attacker can monitor the router's storage and trigger execution at an arbitrarily chosen moment — including after manipulating the target contract's state — extracting value from the user with no ability for the user to cancel or bound the execution window.

---

### Finding Description

The `CrossContractCallArgs::Delayed` variant of the XCC precompile stores a `PromiseArgs` payload in the router contract's `scheduled_promises` map for later execution. [1](#0-0) 

The `PromiseArgs` / `PromiseCreateArgs` structures carry only `target_account_id`, `method`, `args`, `attached_balance`, and `attached_gas`. There is no `deadline`, `not_before`, or any slippage-bound field. [2](#0-1) 

The router's `execute_scheduled` function is explicitly open to any caller:

```
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [3](#0-2) 

There is no mechanism for the user to cancel a scheduled promise, and no expiry is checked before execution. The comment's claim of "no security risk" is incorrect when the scheduled call targets a price-sensitive NEAR contract.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

When a user schedules a call to a price-sensitive NEAR contract (e.g., Ref Finance DEX swap, liquidity provision) via `CrossContractCallArgs::Delayed`, the stored `PromiseArgs` encodes the call arguments at scheduling time but carries no minimum-output or deadline constraint. An attacker who calls `execute_scheduled` at a manipulated price point causes the user's trade to settle at an unfavorable rate. The attacker captures the price difference via arbitrage. Because there is no cancellation path and no deadline, the user cannot protect themselves after scheduling.

---

### Likelihood Explanation

**Medium.** The `Delayed` XCC path is specifically designed for gas-intensive operations that cannot complete in a single transaction, making it the natural choice for complex DeFi interactions with NEAR-native DEXes. The router sub-account address is deterministic (`{evm_address}.{aurora}`), so any observer can enumerate and monitor `scheduled_promises` storage. `execute_scheduled` requires no special permission and accepts only a nonce argument, making it trivially callable by any NEAR account. [4](#0-3) 

---

### Recommendation

**Short term:** Add an optional `deadline_block_height: Option<u64>` field to `CrossContractCallArgs::Delayed` (or to the stored `PromiseArgs` wrapper). In `execute_scheduled`, reject execution if `env::block_height() > deadline`. This mirrors the standard DEX deadline pattern and prevents stale execution.

**Long term:** Add a `cancel_scheduled(nonce: U64)` method callable only by the parent (Aurora engine, on behalf of the originating EVM address) so users can revoke pending promises when market conditions change. Add integration tests that schedule a price-sensitive call, manipulate the target contract's state, and assert that execution reverts when a deadline has passed or a minimum-output bound is violated.

---

### Proof of Concept

1. Alice (EVM address `0xALICE`) calls the XCC precompile with `CrossContractCallArgs::Delayed` encoding a call to `ref-finance.near` → `swap` with no `min_amount_out` in the JSON args. The promise is stored at nonce `0` in `0xalice.aurora`'s router. [4](#0-3) 

2. Bob observes the `scheduled_promises` storage of `0xalice.aurora` (NEAR state is public).

3. Bob executes a large opposing swap on Ref Finance, moving the pool price against Alice's position.

4. Bob calls:
   ```
   near call 0xalice.aurora execute_scheduled '{"nonce": "0"}' --accountId bob.near
   ``` [5](#0-4) 

5. Alice's swap executes at the manipulated price. She receives far fewer output tokens than expected. No revert occurs because no minimum-output bound was enforced.

6. Bob reverses his swap on Ref Finance, capturing the spread as profit. Alice's loss equals Bob's gain minus fees.

The root cause is the combination of: (a) `execute_scheduled` being open to any caller with no deadline check, and (b) `PromiseArgs` carrying no slippage or expiry fields. [3](#0-2) [2](#0-1)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
