### Title
`convert_thin_state_diff` misclassifies pending-block replace_class as new deployment when contract was deployed in the same simulation batch — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
`is_deployed` checks only accepted storage. When `simulate_transactions` (or `trace_transaction`) is called against the pending block with a batch where Tx1 deploys address A and Tx2 calls `replace_class` on A, the per-transaction state diff for Tx2 incorrectly reports A in `deployed_contracts` instead of `replaced_classes`.

### Finding Description
`simulate_transactions` processes each transaction sequentially, committing each transaction's writes into the cumulative `cached_state`. For Tx2 (replace_class), `induced_state_diff` produces `deployed_contracts: {A: new_class_hash}` because the blockifier's `CommitmentStateDiff.address_to_class_hash` is a flat map covering both deployments and replacements.

`convert_thin_state_diff` is supposed to separate these by checking whether the contract existed before the transaction. For a pending block it sets `prev_block_number = Some(block_number)` (the latest accepted block) and calls `is_deployed(prev_block_number, A)`. This calls `maybe_get_class_hash_at` with a non-pending `BlockId::HashOrNumber`, which skips the pending state entirely and only queries accepted storage. Since A was deployed in Tx1 of the same simulation batch (not in any accepted block), `is_deployed` returns `false`, and A is left in `deployed_contracts` for Tx2's state diff.

The correct result for Tx2's state diff is `replaced_classes: [{address: A, class_hash: new_class_hash}]`.

### Impact Explanation
`starknet_simulateTransactions` and `starknet_traceTransaction` (for pending blocks) return an authoritative-looking wrong `state_diff` in the transaction trace. Any tool, wallet, or developer relying on the simulation response to understand what a transaction does (e.g., to display "this transaction replaces the class of contract X") will receive incorrect information. This fits the High impact category: **RPC simulation returns an authoritative-looking wrong value**.

### Likelihood Explanation
The scenario requires two transactions in the same simulation batch where the second transaction calls `replace_class` on a contract deployed by the first. This is a realistic pattern (e.g., deploy-and-upgrade flows, proxy patterns). Any unprivileged user can trigger this via the public `starknet_simulateTransactions` RPC endpoint.

### Recommendation
In `is_deployed`, pass the pending state when checking for a pending block. Specifically, `convert_thin_state_diff` should pass the pending deployed contracts and replaced classes to `maybe_get_class_hash_at` (using `BlockId::Tag(Tag::Pending)`) when `block_id` is pending, so that contracts deployed in earlier transactions of the same pending simulation are correctly recognized as already-deployed.

Alternatively, `convert_thin_state_diff` could check the pending state diff directly (the same `maybe_pending_data` used for execution) before falling back to accepted storage.

### Proof of Concept

        block_id: BlockId,

            thin_state_diff.deployed_contracts.swap_remove(address);
        });
        Ok(ThinStateDiff::from(thin_state_diff, replaced_classes))
