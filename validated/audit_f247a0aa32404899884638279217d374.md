### Title
`ExitToEthereum` Precompile Burns User Funds Before Unguarded `withdraw` Promise With No Failure Recovery Path - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile irrevocably burns a user's ETH or ERC-20 tokens from the EVM state, then schedules a NEAR cross-contract `withdraw` call to the ETH connector with **no failure callback**. If the `withdraw` promise fails for any reason, the burned tokens are permanently unrecoverable. The analogous `ExitToNear` precompile has an explicit `exit_to_near_precompile_callback` failure-handling path; `ExitToEthereum` has none.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run()` function handles two cases:

**ETH base-token exit (flag `0x0`):** The user's ETH is consumed from the EVM state via `context.apparent_value` at EVM execution time.

**ERC-20 exit (flag `0x1`):** The ERC-20 contract burns the user's tokens before calling the precompile.

In both cases, after the EVM-side burn is committed, the precompile constructs a NEAR promise to call `withdraw` on the ETH connector: [1](#0-0) 

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

This promise is emitted as `PromiseArgs::Create` — a fire-and-forget call with **no attached callback**: [2](#0-1) 

There is no `PromiseArgs::Callback` wrapping, no `exit_to_ethereum_precompile_callback`, and no refund logic anywhere in the codebase for this code path.

Contrast this with `ExitToNear`, which explicitly constructs a `PromiseArgs::Callback` wrapping `exit_to_near_precompile_callback` to handle the failure case: [3](#0-2) 

And the callback handler in `engine/src/contract_methods/connector.rs` that performs the refund on failure: [4](#0-3) 

Because NEAR promise execution is asynchronous and occurs **after** the EVM transaction is finalized, a failed `withdraw` promise cannot roll back the already-committed EVM state. The user's tokens are burned with no on-chain mechanism to restore them.

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

If the `withdraw` promise fails, the user's ETH or ERC-20 tokens are permanently destroyed on the Aurora side with no corresponding release on Ethereum and no refund path on Aurora. The funds are irrecoverably lost.

---

### Likelihood Explanation

**Likelihood: Medium.**

The `withdraw` call on the ETH connector can fail under realistic operational conditions:

- The ETH connector contract is paused (a documented admin capability).
- The ETH connector account is misconfigured or replaced (via `set_eth_connector_contract_account`, callable by owner).
- The connector contract encounters an internal error or is temporarily unavailable.
- Gas allocation (`WITHDRAWAL_GAS = 100 TGas`) is insufficient for a future connector implementation.

Any of these conditions, which are not exotic, causes permanent user fund loss with no recovery path.

---

### Recommendation

Mirror the `ExitToNear` pattern: wrap the `withdraw_promise` in a `PromiseArgs::Callback` that targets a new `exit_to_ethereum_precompile_callback` method on the engine. That callback should:

1. Check `promise_result(0)` for failure.
2. On failure, re-mint the burned ERC-20 tokens to the original sender (or credit the ETH back to the sender's EVM balance for the base-token case).
3. Emit a refund event for observability.

The `error_refund` feature flag pattern already used for `ExitToNear` provides a direct template.

---

### Proof of Concept

1. User calls the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with 1 ETH (flag `0x0`) targeting an Ethereum recipient address.
2. The EVM executor deducts 1 ETH from the user's Aurora balance. EVM state is committed.
3. The precompile emits a log containing `PromiseArgs::Create(withdraw_promise)` targeting the ETH connector's `withdraw` method.
4. The Aurora Engine submits this NEAR promise after the EVM transaction finalizes.
5. The ETH connector's `withdraw` call fails (e.g., connector is paused by admin, or connector account has been rotated).
6. No callback exists. The NEAR runtime discards the failed promise result.
7. The user's 1 ETH is gone from Aurora EVM state. No Ethereum withdrawal was initiated. No refund was issued. Funds are permanently lost.

The attacker-controlled entry path is the standard `exitToEthereum` precompile call, reachable by any unprivileged EVM user or ERC-20 contract. No admin compromise is required to trigger the loss — only a condition that causes the `withdraw` promise to fail, which can arise from normal operational events.
