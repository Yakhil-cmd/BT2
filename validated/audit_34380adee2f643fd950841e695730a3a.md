### Title
Staker Can Instantly Raise Commission to 100% via `set_commission_commitment` With No Timelock for Delegators - (`src/staking/staking.cairo`)

---

### Summary

A staker can call `set_commission_commitment` with `max_commission = 10000` (100%) and `expiration_epoch = current_epoch + 1`, making the commitment immediately active, then call `set_commission(10000)` in the same block to raise their pool commission from any value to 100%. Delegators have no timelock to react before their rewards are calculated at the new rate. The protocol's own developer comment acknowledges this gap.

---

### Finding Description

The `set_commission` function in `src/staking/staking.cairo` normally only allows commission to be **decreased**: [1](#0-0) 

However, when an active `CommissionCommitment` exists, the restriction changes: the staker may set commission to **any value up to `max_commission`**, as long as it differs from the current value: [2](#0-1) 

A commitment is considered "active" whenever `current_epoch < expiration_epoch`: [3](#0-2) 

`set_commission_commitment` only requires `expiration_epoch > current_epoch`, meaning `current_epoch + 1` is the minimum valid value — and a commitment set with that value is **immediately active**: [4](#0-3) 

There is no delay enforced between setting the commitment and using it to raise the commission. The protocol's own developer note explicitly acknowledges this: [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker who raises commission to 100% immediately redirects all pool rewards to themselves. Delegators who entered the pool at a low commission (e.g. 1%) will have their entire yield for the current epoch — and every subsequent epoch until they complete the exit window — stolen by the staker. The exit window introduces a mandatory waiting period during which delegators continue to accrue rewards at the new 100% commission rate, compounding the loss.

---

### Likelihood Explanation

**Medium-High.** Any staker who has opened a delegation pool can execute this attack at any time with two sequential transactions. No special access, leaked key, or external dependency is required. The attack is cheap, reversible in appearance (the staker can lower commission afterward), and leaves no on-chain evidence beyond the emitted `CommissionCommitmentSet` and `CommissionChanged` events — which most delegators are unlikely to monitor in real time.

---

### Recommendation

1. **Enforce a minimum delay between setting a commitment and using it to increase commission.** For example, require that `set_commission` with an upward change can only be called at least one full epoch after `set_commission_commitment` was recorded.
2. **Emit a prominent warning event** when a commitment is set with a `max_commission` significantly above the current commission, giving off-chain monitoring tools time to alert delegators.
3. **Consider a per-epoch cap on commission increases** (e.g., no more than X% per epoch) to limit the blast radius of any single sudden change.

---

### Proof of Concept

```
// Staker starts with commission = 100 (1%)
// Step 1: Set commitment with max_commission = 10000, expiration = current_epoch + 1
//         Commitment is immediately active (current_epoch < current_epoch + 1)
staking.set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1);

// Step 2: Immediately raise commission to 100% in the same block
//         update_commission branch: commitment is active, 10000 <= 10000, 10000 != 100 → passes
staking.set_commission(commission: 10000);

// Result: All delegator rewards from this epoch onward go to the staker
//         Delegators must wait through the full exit_wait_window before they can withdraw,
//         losing yield at 100% commission throughout that period.
```

Key code path:

- `set_commission_commitment` stores commitment with `expiration_epoch = current_epoch + 1` [6](#0-5) 
- `is_commission_commitment_active` returns `true` immediately [3](#0-2) 
- `update_commission` allows the increase because `commission <= max_commission` and `commission != old_commission` [2](#0-1) 
- Delegators must wait through `exit_wait_window` before funds are returned [7](#0-6)
