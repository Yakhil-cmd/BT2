### Title
`RegisterEncryptionKey()` Fails to Authorize Caller, Allowing Anyone to Overwrite Any Address's Encryption Key - (File: `x/e2ee/keeper/keeper.go`)

### Summary
The `RegisterEncryptionKey` message handler in `x/e2ee/keeper/keeper.go` stores an encryption key for an arbitrary `req.Address` without verifying that the transaction signer is the owner of that address. Any unprivileged actor can overwrite any other address's registered e2ee encryption key.

### Finding Description
The `RegisterEncryptionKey` function is the sole write path for e2ee public-key registration:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
``` [1](#0-0) 

The internal helper unconditionally writes the supplied key to the KV store keyed by `req.Address`:

```go
func (k Keeper) registerEncryptionKey(ctx context.Context, address string, key []byte) error {
    bz, err := k.addressCodec.StringToBytes(address)
    ...
    sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
    return nil
}
``` [2](#0-1) 

There is no check that the transaction signer equals `req.Address`. The message type carries both a signer field and a target `Address` field, but the handler never compares them.

### Impact Explanation
An attacker submits a `MsgRegisterEncryptionKey` with `Address` set to any victim address and `Key` set to an attacker-controlled public key. The victim's legitimate encryption key is silently overwritten. Any subsequent message encrypted to the victim's real key becomes undecryptable by the victim, and any party that queries the victim's key via `Key()`/`Keys()` will receive the attacker's key instead, enabling a key-substitution attack on the e2ee layer. This constitutes **corruption of e2ee key/message state with direct security impact**, matching the High impact tier.

### Likelihood Explanation
The entry point is a standard Cosmos SDK message (`MsgRegisterEncryptionKey`) reachable by any unprivileged account with no fee or stake requirement beyond normal transaction fees. No special role, leaked key, or cryptographic break is needed.

### Recommendation
Add a signer-equality check at the top of `RegisterEncryptionKey`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signers := req.GetSigners() // or equivalent signer extraction
    if len(signers) == 0 || signers[0].String() != req.Address {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "signer must match the target address")
    }
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
```

### Proof of Concept
1. Alice registers her legitimate encryption key: `MsgRegisterEncryptionKey{Address: alice, Key: aliceRealKey}` — succeeds normally.
2. Attacker (Bob) submits: `MsgRegisterEncryptionKey{Address: alice, Key: attackerKey}` — no authorization check, succeeds.
3. Query `Key(alice)` now returns `attackerKey`.
4. Any message encrypted to Alice's real key is now undecryptable; any sender who queries Alice's key before sending will encrypt to the attacker's key instead. [1](#0-0)
