### Title
Engine Pause Blocks `exit_to_near_precompile_callback`, Permanently Freezing Unwrapped NEAR in Engine Account — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The `exit_to_near_precompile_callback` function enforces `require_running` at the top of its body. This callback is the second leg of the wNEAR-unwrap promise chain: the first leg (`near_withdraw`) burns NEP-141 wNEAR and deposits real NEAR into the engine account, and the callback is supposed to forward that NEAR to the user's recipient. If the engine is paused between the two legs, the callback reverts, the NEAR is permanently stranded in the engine account, and the user's ERC-20 wNEAR tokens are already burned with no refund path.

---

### Finding Description

The wNEAR-unwrap exit flow in `ExitToNear::run` (when the `:unwrap` suffix is present) schedules a two-step NEAR promise chain:

**Step 1 – base promise** (`near_withdraw` on the wNEAR NEP-141 contract):
Burns the NEP-141 wNEAR tokens and returns real NEAR to the engine account.

**Step 2 – callback** (`exit_to_near_precompile_callback` on the engine itself):
Transfers the unwrapped NEAR from the engine account to the user's recipient. [1](#0-0) 

The callback is registered unconditionally whenever `transfer_near_args` is `Some` (the wNEAR-unwrap path): [2](#0-1) 

Inside `exit_to_near_precompile_callback`, the very first substantive check is `require_running`: [3](#0-2) 

`require_running` returns an error (causing the NEAR callback to revert) whenever `state.is_paused == true`: [4](#0-3) 

The engine can be paused by its owner at any time via `pause_contract`: [5](#0-4) 

**Sequence of events leading to fund freeze:**

1. User's EVM transaction calls the `ExitToNear` precompile with wNEAR ERC-20 tokens and the `:unwrap` suffix. The ERC-20 contract burns the user's wNEAR ERC-20 balance (irreversible at this point).
2. The precompile schedules the two-step promise chain (base + callback).
3. The engine owner pauses the engine (e.g., emergency response).
4. The base promise (`near_withdraw`) executes successfully — NEP-141 wNEAR is burned and real NEAR lands in the engine's NEAR account.
5. The callback (`exit_to_near_precompile_callback`) fires. `require_running` sees `is_paused = true` and returns `Err`, reverting the callback.
6. The NEAR is now permanently stranded in the engine's NEAR account. The user's ERC-20 wNEAR is already burned. There is no on-chain refund path because the refund branch inside the callback (`args.refund`) is also gated behind the same `require_running` check and never executes. [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

The user's wNEAR ERC-20 tokens are burned during EVM execution (before any promise fires). The real NEAR that results from `near_withdraw` is deposited into the engine account. If the callback reverts, that NEAR has no automated recovery path: the user cannot re-trigger the callback (NEAR promise callbacks cannot be retried), and the engine has no built-in sweep mechanism that credits the original user. The funds are permanently frozen from the user's perspective.

---

### Likelihood Explanation

**Medium.** The scenario requires the engine to be paused while a wNEAR-unwrap promise is in flight. NEAR's asynchronous execution model means there is always a non-zero window between the base promise and its callback. An emergency pause (e.g., in response to a bridge exploit or critical bug) is precisely the situation where many users may simultaneously be unwrapping wNEAR to exit the system — making the collision between "pause" and "in-flight callback" more likely, not less, during the most critical moments.

---

### Recommendation

Remove the `require_running` guard from `exit_to_near_precompile_callback`, or move it so that it only blocks new user-initiated exits but never blocks the completion of an already-committed promise chain. The callback must be allowed to complete (transferring NEAR to the recipient, or executing the refund branch) regardless of the engine's pause state, because the user's ERC-20 tokens are already burned by the time the callback fires.

### Proof of Concept

**Entry path (unprivileged EVM user):**

1. Any EVM user holding wNEAR ERC-20 tokens calls the `ExitToNear` precompile (address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x01`, the wNEAR amount, recipient account, and `:unwrap` suffix.
2. The ERC-20 contract burns the user's wNEAR ERC-20 balance.
3. The precompile schedules `near_withdraw` + `exit_to_near_precompile_callback`.
4. Engine owner calls `pause_contract` (legitimate emergency action).
5. `near_withdraw` executes: NEAR lands in engine account.
6. `exit_to_near_precompile_callback` executes: `require_running` returns `ERR_PAUSED`, callback reverts.
7. User's wNEAR is burned; NEAR is in engine account; no recovery mechanism exists.

**Root cause line:** `engine/src/contract_methods/connector.rs`, line 203 — `require_running(&state)?;` inside `exit_to_near_precompile_callback`. [8](#0-7)

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
