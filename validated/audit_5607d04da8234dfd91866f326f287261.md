### Title
`CadenceOwnedAccount.balance()` Returns Unwithdrawable Amount When EVM Balance Falls Below 1e10 attoFlow - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`CadenceOwnedAccount.balance()` returns the full raw attoflow balance of the EVM account, but `CadenceOwnedAccount.withdraw(balance:)` panics with `ErrWithdrawBalanceRounding` if the supplied balance is less than `1e10` attoflow (the minimum representable unit in a Cadence `UFix64` Flow vault). Any unprivileged EVM transaction sender can deposit a dust amount (< 1e10 attoflow) into a COA address, causing the COA's entire balance to become unwithdrawable via the natural `withdraw(balance: coa.balance())` pattern, permanently trapping those funds in the EVM environment.

### Finding Description

`CadenceOwnedAccount.balance()` is the only on-chain view function that reports how much FLOW a COA holds:

```cadence
access(all)
view fun balance(): Balance {
    return Balance(attoflow: InternalEVM.balance(address: self.addressBytes))
}
``` [1](#0-0) 

It returns the full 18-decimal-precision attoflow value with no floor or alignment applied.

`CadenceOwnedAccount.withdraw(balance:)` delegates to `InternalEVM.withdraw`, which is implemented in `newInternalEVMTypeWithdrawFunction`:

```go
// check balance is not prone to rounding error
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
``` [2](#0-1) 

`AttoFlowBalanceIsValidForFlowVault` enforces a hard minimum of `1e10` attoflow:

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
``` [3](#0-2) 

where `UFixToAttoConversionMultiplier = 10^(18-8) = 1e10`. [4](#0-3) 

The error triggered is:

```go
ErrWithdrawBalanceRounding = errors.New("withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow")
``` [5](#0-4) 

The mismatch: `balance()` reports the true EVM balance (any attoflow value), while `withdraw` rejects any amount below `1e10` attoflow. There is no `maxWithdraw()` or equivalent function that returns the largest amount safely passable to `withdraw`.

A COA's EVM balance can fall below `1e10` attoflow through a normal EVM-level transfer. Any EVM transaction can send an arbitrary wei amount to any address, including a COA address. After such a transfer the COA owner calling `coa.withdraw(balance: coa.balance())` — the natural "withdraw everything" pattern — will receive a panic.

### Impact Explanation

Funds below `1e10` attoflow (< `1e-8` FLOW) held in a COA are permanently unrecoverable via Cadence. The only exit path from the EVM environment for FLOW tokens is `CadenceOwnedAccount.withdraw`; there is no alternative bridge-out mechanism. Funds trapped this way cannot be returned to a Cadence vault. This constitutes a cross-VM asset loss: the COA owner loses access to on-chain assets they legitimately hold.

Additionally, any Cadence transaction or script that reads `coa.balance()` and passes the result directly to `coa.withdraw(balance:)` will revert, breaking the expected interface contract.

### Likelihood Explanation

The trigger requires only a standard EVM transaction sending any amount less than `1e10` attoflow (i.e., less than `10 gwei` in Ethereum terms) to the COA's EVM address. This is permissionless — any EVM account can do it at negligible cost. The pattern `coa.withdraw(balance: coa.balance())` is the obvious "withdraw full balance" idiom and is likely to appear in user scripts and integrations.

### Recommendation

Add a `maxWithdraw()` view function to `CadenceOwnedAccount` that returns the balance rounded down to the nearest `1e10` attoflow boundary:

```cadence
access(all)
view fun maxWithdraw(): Balance {
    let raw = InternalEVM.balance(address: self.addressBytes)
    // floor to nearest 1e10 attoflow (UFix64 precision)
    let aligned = raw - (raw % 10_000_000_000)
    return Balance(attoflow: aligned)
}
```

Alternatively, modify `balance()` to return the aligned (floor) value and add a separate `rawBalance()` for the full precision value, so that the value returned by `balance()` is always safe to pass directly to `withdraw`.

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // 1. Create a COA
        let coa <- EVM.createCadenceOwnedAccount()

        // 2. Simulate an EVM-level dust transfer of 5 attoflow to the COA address
        //    (in practice: any EVM tx sending < 1e10 attoflow to coa.address())
        //    After this, coa.balance().attoflow == 5

        // 3. coa.balance() returns Balance(attoflow: 5)
        let bal = coa.balance()
        assert(bal.attoflow == 5, message: "balance should be 5 attoflow")

        // 4. Attempt to withdraw the full balance — PANICS with ErrWithdrawBalanceRounding
        //    because 5 < 1e10 (UFixToAttoConversionMultiplier)
        let vault <- coa.withdraw(balance: bal)  // panics here

        destroy coa
        destroy vault
    }
}
```

The panic is confirmed by the existing test `"test coa withdraw with rounding error"` in `fvm/evm/evm_test.go` (line 2274), which demonstrates that `EVM.Balance(attoflow: 9999999999)` (just below `1e10`) causes `ErrWithdrawBalanceRounding` when passed to `withdraw`. [6](#0-5)
