### Title
Weekly `PERIOD_MINT_CAP` in `MintManager` Can Be Exhausted in a Single Public `tick()` Call, Temporarily Freezing Unclaimed Yield — (`L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

The L1 `MintManager` enforces a hard-coded weekly minting cap of `PERIOD_MINT_CAP = 6_500_000 × 10^18` STRK via `PeriodMintLimit.checkAndUpdatePeriodicalQuota()`. The L1 `RewardSupplier.tick()` function is permissionless and can consume up to `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5` pending L2→L1 mint-request messages per call, each worth `TOKENS_PER_MINT_REQUEST = 1_300_000 × 10^18` STRK. Because `5 × 1,300,000 = 6,500,000`, a single `tick()` call with five pending messages mints exactly the full weekly cap, causing every subsequent `tick()` call that week to revert with `"EXCEED_PERIOD_MINTING"`. No further STRK can be bridged to L2 for the remainder of the week, temporarily freezing unclaimed yield for all stakers and delegators.

---

### Finding Description

**Root cause — arithmetic identity between the three constants:** [1](#0-0) 

```
TOKENS_PER_MINT_REQUEST     = 1_300_000 × 10^18
MAX_MESSAGES_TO_PROCESS_PER_TICK = 5
``` [2](#0-1) 

```
PERIOD_MINT_CAP             = 6_500_000 × 10^18   (= 5 × 1,300,000)
MINTING_PERIOD_DURATION     = 1 weeks
```

`tick()` is a permissionless `external payable` function: [3](#0-2) 

It calls `requiredMinting()`, which caps the number of messages at `MAX_MESSAGES_TO_PROCESS_PER_TICK`: [4](#0-3) 

Then it calls `mintManager().mintRequest(token(), amountToMint)`: [5](#0-4) 

Inside `MintManager.mintRequest()`, `checkAndUpdatePeriodicalQuota` enforces the cap: [6](#0-5) [7](#0-6) 

When `amountToMint = 5 × 1,300,000 × 10^18 = PERIOD_MINT_CAP`, the `require(mintedThisPeriodAfter <= PERIOD_MINT_CAP)` passes exactly once. Every subsequent call that week will have `mintedThisPeriodBefore = PERIOD_MINT_CAP`, so even a single-message tick (1.3M STRK) will revert.

The L2 side sends mint requests whenever the reward supplier's credit falls below the debit + threshold: [8](#0-7) 

Under normal protocol operation (high staking participation, multiple epochs per week), five pending messages can accumulate naturally. An attacker simply waits for that condition and calls `tick()` once.

---

### Impact Explanation

Once the weekly cap is exhausted, `tick()` reverts for the rest of the week. The L2 `RewardSupplier` receives no new STRK from L1. When the L2 staking contract calls `RewardSupplier.claim_rewards()` to pay out staker and delegator rewards, it will fail if the contract's on-hand balance is insufficient: [9](#0-8) 

This constitutes a **temporary freeze of unclaimed yield** for all stakers and delegators, lasting up to one full week (until `block.timestamp / MINTING_PERIOD_DURATION` increments and the cap resets).

---

### Likelihood Explanation

- `tick()` has no access control — any EOA or contract can call it.
- The condition (≥5 pending L2→L1 messages) arises naturally during normal protocol operation whenever the reward supplier requests multiple mint batches in quick succession (e.g., after a large epoch reward update or after a period of `tick()` inactivity).
- The attacker bears only gas cost; there is no capital requirement.
- The cap is hard-coded and cannot be changed without a contract upgrade, so there is no admin escape hatch that can be applied in real time.

---

### Recommendation

Decouple `MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST` from `PERIOD_MINT_CAP` so that a single `tick()` call cannot exhaust the full weekly quota. For example:
- Reduce `MAX_MESSAGES_TO_PROCESS_PER_TICK` so the maximum single-tick mint is well below `PERIOD_MINT_CAP`, or
- Increase `PERIOD_MINT_CAP` to a multiple of the maximum single-tick amount, or
- Make `PERIOD_MINT_CAP` a governance-settable parameter rather than a compile-time constant, consistent with the recommendation in the reference report to remove or make the rate limit configurable.

---

### Proof of Concept

1. Observe that `5 × TOKENS_PER_MINT_REQUEST == PERIOD_MINT_CAP`:
   - `5 × 1_300_000e18 = 6_500_000e18 = PERIOD_MINT_CAP` ✓
2. Wait (or arrange) for ≥5 L2→L1 mint-request messages to be pending in the Starknet messaging contract.
3. Call `RewardSupplier.tick{value: fee}()` from any address.
4. `requiredMinting()` returns `(6_500_000e18, 5)`.
5. `MintManager.mintRequest(token, 6_500_000e18)` succeeds; `periodMintAccounting[slot]` is now `6_500_000e18 = PERIOD_MINT_CAP`.
6. Call `tick()` again (even with only 1 pending message → 1.3M STRK):
   - `checkAndUpdatePeriodicalQuota`: `6_500_000e18 + 1_300_000e18 > 6_500_000e18` → **reverts with `"EXCEED_PERIOD_MINTING"`**.
7. All `tick()` calls revert for the remainder of the week; no STRK is bridged to L2; staker/delegator reward claims that require new liquidity fail.

    function requiredMinting() public view returns (uint256, uint256) {
        uint256[] memory messagePayload = new uint256[](1);
        messagePayload[0] = TOKENS_PER_MINT_REQUEST;

        bytes32 msgHash = l2ToL1MsgHash(mintRequestSource(), address(this), messagePayload);
        // Limit the number of msgs to consume to limit.
        uint256 numMsgsToConsume = Math.min(
            messagingContract().l2ToL1Messages(msgHash),
            MAX_MESSAGES_TO_PROCESS_PER_TICK
        );

        return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
    }

    function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
        bytes32 periodSlot = periodAccountingSlot(token);
        uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
        uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
        require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
        periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
    }

    function mintRequest(address token, uint256 amount) external {
        address requester = AccessControl._msgSender();
        require(registeredMinters(token)[requester], "NOT_A_REGISTERED_MINTER");
        require(mintingAllowance(token)[requester] >= amount, "INSUFFICIENT_MINTING_ALLOWANCE");

        // Update allowance.
        checkAndUpdatePeriodicalQuota(token, amount);
        mintingAllowance(token)[requester] -= amount;

        // Mint.
        mintableToken(token).mint(requester, amount);
        emit MintProcessed(token, requester, amount);

            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
