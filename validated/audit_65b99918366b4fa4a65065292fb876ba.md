### Title
Unprotected `disable_rewards` Parameter in `update_rewards` Allows Any Caller to Suppress Block Reward Distribution — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` is specified as callable only by the Starkware sequencer, but the implementation contains **no access-control enforcement**. Any unprivileged caller can invoke it with `disable_rewards: true`, which writes the current block number into the global `last_reward_block` storage slot and returns early without distributing rewards. This permanently blocks the legitimate sequencer call for that block, causing stakers and delegators to lose all block rewards for every block the attacker targets.

---

### Finding Description

The spec at `docs/spec.md:1645` states:

> **Access control:** Only starkware sequencer.

The implementation at `src/staking/staking.cairo:1449–1507` does not enforce this:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits before distributing rewards
    }
    ...
```

`last_reward_block` is a **single global slot** — not per-staker. Any caller who reaches line 1485 first in a given block sets it for the entire contract. The `disable_rewards` parameter is fully attacker-controlled; there is no check that the caller is the sequencer.

The analog to the external report is direct:

| External report | This codebase |
|---|---|
| `ratio` accepted from user data without validation, used in token calculation | `disable_rewards` accepted from any caller without access control, used to gate reward distribution |
| Allows arbitrary inflation of token amounts | Allows permanent suppression of block rewards |

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

For every block in which an attacker front-runs the sequencer's `update_rewards` call with `disable_rewards: true`:

- `last_reward_block` is set to the current block number.
- The sequencer's subsequent call reverts with `REWARDS_ALREADY_UPDATED`.
- `_update_rewards` is never reached; `unclaimed_rewards_own` for the staker and pool balances are never incremented.
- The missed block rewards are **permanently lost** — there is no catch-up mechanism.

Delegators in the targeted staker's pool also lose their share of pool rewards for those blocks.

---

### Likelihood Explanation

**Medium.**

On Starknet, the sequencer controls transaction ordering within a block, which limits classical mempool front-running. However:

1. `update_rewards` is a public, permissionless entry point — any EOA or contract can call it.
2. An attacker can submit a transaction targeting any valid staker address with `disable_rewards: true` at the start of each block.
3. The attacker does not need to predict which staker the sequencer will target; calling it for *any* valid staker with a non-zero balance at the current epoch suffices to lock `last_reward_block`.
4. The attack is cheap (a single call per block) and requires no capital.

---

### Recommendation

Add a caller check at the top of `update_rewards`, consistent with the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
```

Alternatively, if the sequencer address is not stored, restrict via a role (e.g., `SEQUENCER_ROLE`) using the existing `RolesComponent`.

---

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Block N begins. The sequencer prepares `update_rewards(staker_A, disable_rewards: false)`.
3. Attacker submits `update_rewards(staker_A, disable_rewards: true)` — no special role required.
4. Attacker's transaction executes first:
   - `last_reward_block` ← N (line 1485).
   - Function returns at line 1487 (`disable_rewards == true`).
   - No rewards distributed.
5. Sequencer's transaction executes:
   - `current_block_number (N) > last_reward_block (N)` → **false** → reverts with `REWARDS_ALREADY_UPDATED` (line 1454–1458).
6. `staker_A.unclaimed_rewards_own` is unchanged; pool balances are unchanged.
7. Block N rewards are permanently lost for `staker_A` and all its delegators.
