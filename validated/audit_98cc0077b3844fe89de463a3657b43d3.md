### Title
`PresignOutput` Not Consumed After Rerandomization Enables Nonce Reuse and Private Key Recovery - (File: `src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both ECDSA implementations expose `rerandomize_presign` as accepting `&PresignOutput` (a shared reference) rather than consuming the presign output by value. Combined with `PresignOutput` deriving `Clone`, the presign output is never destroyed after use. A malicious coordinator can orchestrate two signing sessions against the same presignature, causing honest participants to rerandomize the same nonce material for two different messages. The coordinator collects both sets of signature shares and recovers the private key via standard ECDSA nonce-reuse analysis.

---

### Finding Description

**Root cause — Robust ECDSA:**

`PresignOutput` derives `Clone`: [1](#0-0) 

`rerandomize_presign` accepts `&PresignOutput` (borrow, not move): [2](#0-1) 

Because the function borrows rather than consumes the presign output, the caller retains full ownership of the `PresignOutput` after the call returns. Nothing in the type system prevents calling `rerandomize_presign` a second time with a different `RerandomizationArguments` (i.e., a different `(msg_hash, tweak)` pair), producing a second `RerandomizedPresignOutput` from the same underlying nonce material.

**Root cause — OT-based ECDSA (identical pattern):**

`PresignOutput` derives `Clone`: [3](#0-2) 

`rerandomize_presign` again accepts `&PresignOutput`: [4](#0-3) 

**Security requirement that is violated:**

The orchestration documentation explicitly states it is *critical* that presign output is destroyed after use and must never be reused: [5](#0-4) 

The Robust ECDSA signing documentation further warns that reusing a presignature across two signing sessions with different `(h, ε)` values allows the secret key to be recovered: [6](#0-5) 

Despite these warnings, the API provides no enforcement mechanism. The library documents the invariant but does not encode it in the type system.

---

### Impact Explanation

**Impact: Critical — Extraction of aggregate secret key material.**

In ECDSA, if two signatures `(R, s₁)` and `(R, s₂)` are produced using the same nonce `k` (i.e., the same `R`) but different message hashes `h₁` and `h₂`, the private key `x` is recoverable by solving the linear system:

```
s₁ = k·h₁ + r·x
s₂ = k·h₂ + r·x
```

Subtracting: `k = (s₁ - s₂) / (h₁ - h₂)`, then `x = (s₁ - k·h₁) / r`.

Because `rerandomize_presign` does not consume the `PresignOutput`, a malicious coordinator can cause honest participants to produce two complete sets of signature shares from the same nonce material for two different messages. The coordinator aggregates both sets and applies the above recovery formula to obtain the master private key.

---

### Likelihood Explanation

**Likelihood: High.**

The attack requires only a malicious coordinator, which is an explicitly documented trust boundary in the library. The coordinator controls which signing sessions are initiated and which presignatures are assigned to them. No cryptographic assumption needs to be broken. The only requirement is that honest participants hold a `PresignOutput` and are willing to participate in two signing sessions — a normal operational scenario. The `Clone` bound on `PresignOutput` further means even a non-malicious caller can accidentally trigger the condition by cloning and reusing a presign output.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput`** in both `src/ecdsa/robust_ecdsa/mod.rs` and `src/ecdsa/ot_based_ecdsa/mod.rs`. A one-time-use secret should not be cheaply duplicable.

2. **Change `rerandomize_presign` to consume `PresignOutput` by value** (i.e., `presignature: PresignOutput` instead of `presignature: &PresignOutput`). This encodes the single-use invariant in the type system: after rerandomization the presign output is moved into the function and dropped (zeroized via `ZeroizeOnDrop`), making a second call impossible without a fresh presign run.

3. Apply the same fix to `RerandomizedPresignOutput`, which also derives `Clone` and can be passed to `sign()` multiple times.

---

### Proof of Concept

```rust
// Attacker-controlled coordinator orchestrates two signing sessions
// against the same PresignOutput held by each honest participant.

// Step 1: Presigning produces PresignOutput for each participant.
let presign_out: PresignOutput = run_presign(...);

// Step 2: Coordinator requests signing session 1 with msg_hash h1.
// Participant calls rerandomize_presign — presign_out is NOT consumed.
let rerandomized_1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,   // <-- borrow; presign_out still alive
    &args_for_h1,
).unwrap();
let sig_share_1 = sign_participant(&rerandomized_1, h1);

// Step 3: Coordinator requests signing session 2 with msg_hash h2.
// Participant calls rerandomize_presign AGAIN on the same presign_out.
let rerandomized_2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,   // <-- same nonce material reused
    &args_for_h2,
).unwrap();
let sig_share_2 = sign_participant(&rerandomized_2, h2);

// Step 4: Coordinator aggregates both sets of shares, obtains (R, s1) and (R, s2).
// Applies nonce-reuse formula: k = (s1 - s2) / (h1 - h2), x = (s1 - k*h1) / r.
// Private key x is recovered.
```

The `PresignOutput` is never consumed or zeroized between steps 2 and 3 because `rerandomize_presign` takes `&PresignOutput`. [7](#0-6) [8](#0-7)

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
