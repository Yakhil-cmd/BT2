### Title
Service Transaction Replay — No Nonce and Ignored Salt Allow Identical `addInteropRootsInBatch` Calls to Execute Multiple Times - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/service_tx.rs`)

---

### Summary

ZKsync OS service transactions (type `0x7d`) carry no signature, no nonce, and a `salt` field that is **explicitly ignored** by the state transition function. Nothing in the protocol prevents the same service transaction bytes from being included and executed more than once — within a single block or across blocks. The most impactful target is `addInteropRootsInBatch`: replaying it inserts duplicate interop roots into the rolling-hash accumulator that forms part of the batch public input, corrupting the cross-chain commitment.

---

### Finding Description

`ServiceTx` is defined as:

```rust
pub(crate) struct ServiceTx<'a> {
    pub(crate) to: &'a [u8; 20],
    pub(crate) data: &'a [u8],
    salt: u64, // Some salt used by the server to identify service transactions. Ignored by ZKsync OS.
}
``` [1](#0-0) 

The comment is the smoking gun: the `salt` field was intended to give each service transaction a unique identity, but ZKsync OS parses it and then discards it — it is never used in any validation, deduplication, or replay-prevention logic.

The validation path for ZK transactions explicitly skips nonce tracking for service transactions:

```rust
let old_nonce = if transaction.nonce().is_some() {
    // ... increment nonce ...
} else {
    // For service transactions, nonce is not used
    0
};
``` [2](#0-1) 

And the nonce check is also skipped:

```rust
if !Config::SIMULATION {
    // Nonce validation - skipped for service transactions
    if let Some(originator_expected_nonce) =
        transaction.nonce().as_ref().map(u256_to_u64_saturated)
    { ... }
}
``` [3](#0-2) 

Service transactions also carry no signature:

> "They have no signature, as they are added directly by the operator." [4](#0-3) 

The three whitelisted operations are:

```rust
const SERVICE_DESTINATION_WHITELIST: &[(B160, [u8; 4])] = &[
    (L2_INTEROP_ROOT_STORAGE_ADDRESS, ADD_INTEROP_ROOTS_IN_BATCH_SELECTOR),
    (SYSTEM_CONTEXT_ADDRESS, SET_SL_CHAIN_ID_SELECTOR),
    (L2_INTEROP_CENTER_ADDRESS, SET_INTEROP_FEE_SELECTOR),
];
``` [5](#0-4) 

For `addInteropRootsInBatch`, each execution calls `push_root` unconditionally:

```rust
fn add_interop_root(..., interop_root: InteropRoot) -> Result<(), SystemError> {
    ...
    self.interop_root_storage.push_root(interop_root)
}
``` [6](#0-5) 

The pushed roots feed directly into the `interop_roots_rolling_hash` that becomes part of the batch public input:

```rust
self.interop_roots_rolling_hash = calculate_interop_roots_rolling_hash(
    self.interop_roots_rolling_hash,
    interop_roots,
    &mut crypto::sha3::Keccak256::new(),
);
``` [7](#0-6) 

Note: `setSettlementLayerChainId` has a per-block guard that prevents a second update within the same block, so it is not exploitable within a single block. `setInteropFee` is idempotent for the same value. The critical path is `addInteropRootsInBatch`.

---

### Impact Explanation

Replaying an `addInteropRootsInBatch` service transaction causes the same `InteropRoot` structs to be pushed into the interop root storage multiple times. Because `push_root` has no duplicate check, the `interop_roots_rolling_hash` accumulates the same root N times. This hash is committed to in the batch public input (`BatchPublicInput`). A corrupted rolling hash means:

- The settlement layer receives a batch commitment that does not correspond to the true set of cross-chain messages.
- Cross-chain message proofs that rely on the interop root commitment can be forged or invalidated, enabling double-spending of bridged assets or denial of legitimate cross-chain operations. [8](#0-7) 

---

### Likelihood Explanation

Service transactions are included in blocks by the operator/sequencer. The attack requires a malicious or compromised operator to include duplicate service transaction bytes in a service block. The ZKsync OS state transition function provides **zero protocol-level defense** against this: no nonce, no signature, and the only field that could serve as a uniqueness token (`salt`) is explicitly ignored. The Immunefi scope lists "prover/forward execution input" and "oracle-data-influencing caller" as valid attack vectors; the oracle data that feeds the state transition function is exactly where duplicate service transactions would be injected.

---

### Recommendation

1. **Use the `salt` field as a replay-prevention nonce.** Track consumed `(to, data_hash, salt)` tuples in a per-block or per-batch set and reject duplicates.
2. Alternatively, maintain a monotonically incrementing per-block service-transaction counter and require each service transaction to carry the expected counter value, analogous to a nonce.
3. At minimum, enforce within the state transition function that no two service transactions in the same block share identical `(to, data)` bytes, regardless of `salt`.

---

### Proof of Concept

The existing test `test_service_block_invariants` already demonstrates that multiple service transactions with the same `to` and `data` (only differing in `salt`) are accepted and executed:

```rust
let tx1 = ZKsyncTxEnvelope::from(ZKsyncServiceTx {
    to: ..., input: ADD_INTEROP_ROOTS_IN_BATCH_SELECTOR.into(), salt: 0,
});
let tx2 = ZKsyncTxEnvelope::from(ZKsyncServiceTx {
    to: ..., input: ADD_INTEROP_ROOTS_IN_BATCH_SELECTOR.into(), salt: 1,
});
let result = tester.execute_block(vec![tx1, tx2, tx3]);
assert!(result.tx_results.iter().all(|res| res.is_ok()), "All txs should succeed");
``` [9](#0-8) 

Since `salt` is ignored, `tx1` and `tx2` are functionally identical to ZKsync OS. A malicious operator can replace `tx2` with an exact byte-for-byte copy of `tx1` (same `salt`). Both will execute, `push_root` will be called twice with the same `InteropRoot`, and the `interop_roots_rolling_hash` will be computed over a duplicated root — producing a batch commitment that does not reflect the true cross-chain state.
