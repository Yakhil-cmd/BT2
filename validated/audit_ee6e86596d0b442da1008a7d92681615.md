### Title
Forward Execution Loop Missing Block-Limit Enforcement and Rollback — (`basic_bootloader/src/bootloader/block_flow/ethereum/loop_op.rs`)

### Summary
The Ethereum-mode transaction loop (`generic_loop_op` / `EthereumLoopOp`) processes every transaction and commits it unconditionally, while the ZK-mode loop (`zk/tx_loop.rs`) takes a per-transaction rollback snapshot and enforces block-level gas / native / pubdata / log limits before committing. A transaction that pushes the block over any of those limits is silently included by the forward path but would be rolled back by the proving path, creating a forward/proving divergence analogous to the LES bug: the transaction appears to succeed but is never finalisable on L1.

### Impact Explanation
A transaction that causes the running block totals to exceed any of the four block limits (gas, computational native, pubdata, logs) is committed by the forward path without rollback. The prover, which enforces those limits and rolls back the offending transaction, will produce a different post-state root. The resulting block is unprovable, causing a liveness failure: the batch can never be submitted to L1, and every transaction in it — including the attacker's — appears to succeed from the user's perspective but is never finalised.

### Likelihood Explanation
Any unprivileged user can submit a transaction whose execution pushes the block over a limit (e.g., a contract that writes many storage slots to exhaust pubdata, or a compute-heavy loop to exhaust native resources). The sequencer's forward loop will include it; the prover will reject it. No special access is required.

### Recommendation
Apply the same per-transaction `start_global_frame` / `check_for_block_limits` / conditional `finish_global_frame` pattern from `zk/tx_loop.rs` to `generic_loop_op` in `ethereum/loop_op.rs`. The existing `// TODO: unify with ZK version` comment already identifies this as pending work; it should be treated as a security-critical fix rather than a cleanup task.

### Proof of Concept

1. Deploy a contract that writes `N` distinct storage slots in a single call, where `N` is chosen so that the resulting pubdata exceeds `block_pubdata_limit`.
2. Submit the transaction to the sequencer running in forward mode (`generic_loop_op`).
3. Observe that `result_keeper.tx_processed(Ok(...))` is called and `system.finish_valid_tx()` commits the state — no limit check occurs.
4. Feed the same block to the prover (ZK tx_loop): `check_for_block_limits` fires, `finish_global_frame(Some(&pre_tx_rollback_handle))` reverts the transaction, and the prover's post-state root diverges from the sequencer's.
5. The batch cannot be submitted to L1; all transactions in the block are effectively dropped despite appearing successful to the user. [5](#0-4) [6](#0-5)
