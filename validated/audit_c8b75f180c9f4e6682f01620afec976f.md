### Title
Unchecked Wrapping Arithmetic in `fake_exponential` Silently Corrupts Blob Base Fee Computation - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/utils.rs`)

---

### Summary

The `fake_exponential` function used to compute the EIP-4844 blob base fee performs all intermediate arithmetic using Rust's default `*` and `+=` operators on `U256` values, which **panic in debug builds and wrap silently in release builds** on overflow. A prover/forward-execution input with a crafted `excess_blob_gas` value in the block header can cause the accumulator to overflow, producing a silently incorrect (wrapped/truncated) blob base fee. This corrupted value is then stored as `computed_blob_base_fee_per_gas` and used in all subsequent fee validation and payment logic for blob transactions, analogous to the MulDivGadget overflow described in the reference report.

---

### Finding Description

`fake_exponential` in `basic_bootloader/src/bootloader/block_flow/ethereum/utils.rs` implements the EIP-4844 blob base fee formula:

```rust
pub(crate) fn fake_exponential(prefactor: U256, numerator: &U256, denominator: &U256) -> U256 {
    let mut i = 1;
    let mut output = U256::ZERO;
    let mut numerator_accumulator = prefactor * denominator;   // ← unchecked mul
    while numerator_accumulator.is_zero() == false {
        output += &numerator_accumulator;                       // ← unchecked add
        numerator_accumulator =
            (numerator_accumulator * numerator) /              // ← unchecked mul
            (u256_mul_by_word(denominator, i).0);
        i += 1;
    }
    output / denominator
}
``` [1](#0-0) 

Every `*` and `+=` on `U256` (backed by `ruint`) **wraps silently in release mode** (Rust's default for `ruint` arithmetic). There is no `checked_mul`, `checked_add`, or overflow guard anywhere in this function. The code itself acknowledges the risk with a `TODO` comment at the call site:

```rust
// TODO: consider if it's numerically stable over u64

    U256::from(MIN_BASE_FEE_PER_BLOB_GAS),
    &U256::from(header.excess_blob_gas),
    &U256::from(BLOB_BASE_FEE_UPDATE_FRACTION),
);
``` [2](#0-1) 

The `excess_blob_gas` field is read directly from the oracle-provided block header bytes with no upper-bound validation before being passed to `fake_exponential`. [3](#0-2) 

The corrupted result is stored in `HeaderAndHistory::computed_blob_base_fee_per_gas` and returned by `blob_base_fee_per_gas()`, which is the authoritative source used in:

1. **ZK transaction validation** — `system.get_blob_base_fee_per_gas()` is compared against `tx_max_fee_per_blob_gas` and used to compute `fee_for_blob_gas` (the blob fee charged to the sender). [4](#0-3) 

2. **Ethereum transaction validation** — `system.metadata.blob_base_fee_per_gas()` is used to compute `fee_for_blob_gas` for the balance pre-charge. [5](#0-4) 

3. **EVM `BLOBBASEFEE` opcode** — the wrapped value is pushed directly onto the EVM stack. [6](#0-5) 

---

### Impact Explanation

A crafted `excess_blob_gas` value in the oracle-provided block header causes `fake_exponential` to silently wrap intermediate `U256` products, producing an arbitrarily small (or zero) blob base fee. Consequences:

- **Fee evasion**: A near-zero wrapped blob base fee means blob transactions pass the `block_base_fee_per_blob_gas > tx_max_fee_per_blob_gas` check with a trivially low `max_fee_per_blob_gas`, and the `fee_for_blob_gas` charged to the sender is negligible. This is the direct analog of the MulDivGadget overflow reducing fees to near-zero.
- **Forward/proving divergence**: The forward execution system and the prover may compute different blob base fees if one runs in debug mode (panic) and the other in release mode (wrap), causing a state-transition mismatch that breaks proof validity.
- **Incorrect `BLOBBASEFEE` opcode result**: Smart contracts reading `BLOBBASEFEE` receive a corrupted value, enabling contract-level exploits that depend on the blob fee being above a threshold.

---

### Likelihood Explanation

The `excess_blob_gas` field is a `u64` read from the oracle-provided block header. In the Ethereum Pectra context (ZKsync OS targets Pectra), `excess_blob_gas` is a legitimate block header field. A malicious prover or a sequencer providing a crafted oracle input can set `excess_blob_gas` to a value that causes overflow in the iterative accumulation loop. The `BLOB_BASE_FEE_UPDATE_FRACTION` is `5_007_716`, and `MIN_BASE_FEE_PER_BLOB_GAS` is `1`. With `excess_blob_gas` near `u64::MAX` (~1.8×10¹⁹), the exponential series terms grow rapidly and the `U256` accumulator overflows within a small number of iterations. The entry path is the oracle-provided block header, which is a standard prover/forward-execution input within scope.

---

### Recommendation

Replace all unchecked arithmetic in `fake_exponential` with checked or saturating variants:

```rust
pub(crate) fn fake_exponential(prefactor: U256, numerator: &U256, denominator: &U256) -> U256 {
    let mut i = 1u64;
    let mut output = U256::ZERO;
    let mut numerator_accumulator = prefactor.checked_mul(*denominator)
        .unwrap_or(U256::MAX); // saturate on overflow
    while !numerator_accumulator.is_zero() {
        output = output.saturating_add(numerator_accumulator);
        let denom_i = u256_mul_by_word(denominator, i).0;
        if denom_i.is_zero() { break; }
        numerator_accumulator = numerator_accumulator
            .checked_mul(*numerator)
            .map(|v| v / denom_i)
            .unwrap_or(U256::ZERO); // on overflow, series has converged past U256
        i += 1;
    }
    if denominator.is_zero() { return U256::ZERO; }
    output / denominator
}
```

Additionally, add an upper-bound sanity check on `excess_blob_gas` when parsing the block header, consistent with the Ethereum protocol maximum.

### Proof of Concept

1. Craft a block header oracle response with `excess_blob_gas = u64::MAX` (18_446_744_073_709_551_615).
2. `fake_exponential(1, &U256::from(u64::MAX), &U256::from(5_007_716))` is called.
3. `numerator_accumulator` starts at `1 * 5_007_716 = 5_007_716`.
4. After the first iteration: `numerator_accumulator = 5_007_716 * 18_446_744_073_709_551_615 / 5_007_716 = 18_446_744_073_709_551_615`.
5. After the second iteration: `numerator_accumulator = 18_446_744_073_709_551_615 * 18_446_744_073_709_551_615 / (5_007_716 * 2)`. The numerator `~3.4×10³⁸` exceeds `U256::MAX` (~1.16×10⁷⁷) only after many more iterations, but `output += numerator_accumulator` accumulates rapidly and wraps `U256` within tens of iterations.
6. The wrapped `computed_blob_base_fee_per_gas` (e.g., near zero) is stored and returned by `blob_base_fee_per_gas()`.
7. A blob transaction with `max_fee_per_blob_gas = 1` passes the fee check and is charged near-zero blob fees, bypassing the intended EIP-4844 fee market. [1](#0-0) [2](#0-1)
