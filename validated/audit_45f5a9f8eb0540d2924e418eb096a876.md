### Title
Zero-fee `tick()` call bypasses L1→L2 message fee enforcement, enabling gasless reward token bridging — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is publicly callable with `msg.value = 0`. It splits the provided ETH between two L1→L2 messages using integer division, with no minimum fee check. When called with zero ETH, both the StarkGate `depositWithMessage` and the `sendMessageToL2` calls for the total-supply update are submitted with zero fee. The Starknet messaging contract enforces only a **maximum** fee (`getMaxL1MsgFee()`), not a minimum, so zero-fee messages are accepted on-chain. This allows any unprivileged caller to trigger minting and bridging of up to `5 × 1,300,000 STRK` per call without paying for L2 message processing, and risks leaving minted reward tokens permanently stranded in the StarkGate bridge if the sequencer enforces a minimum fee.

---

### Finding Description

`tick()` is `external payable` with no access control and no `require(msg.value > 0)` guard.

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol  lines 107–143
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // Consume L2→L1 mint-request messages (irreversible within this tx)
        for (uint256 i = 0; i < numMsgsToConsume; i++) {
            messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
        }
        mintManager().mintRequest(token(), amountToMint);

        uint256 msgFee = msg.value / 2;                          // ← integer division
        bridge().depositWithMessage{value: msgFee}(              // ← fee = 0 when msg.value = 0
            token(), amountToMint, mintDestination(), new uint256[](0)
        );

        msgFee = msg.value - msgFee;                             // ← also 0
        messagingContract().sendMessageToL2{value: msgFee}(      // ← fee = 0
            mintingCurve(), UPDATE_TOTAL_SUPPLY_SELECTOR, messagePayload
        );
    }
}
```

Two compounding problems exist:

1. **Zero-fee path**: With `msg.value = 0`, both `depositWithMessage` and `sendMessageToL2` receive `0 wei`. The `IStarknetMessaging` interface exposes only `getMaxL1MsgFee()` — there is no on-chain minimum fee check. Zero-fee messages are stored in the messaging contract (as `fee + 1 = 1`) and accepted without revert.

2. **Off-by-one fee split**: Even with `msg.value = 1 wei`, integer division gives `msgFee = 0` for the bridge call and `1 wei` for the messaging call. The bridge deposit — which carries the actual reward tokens — always receives the smaller (potentially zero) share.

The Starknet sequencer currently processes all pending L1→L2 messages regardless of fee. However, this is an off-chain policy, not an on-chain invariant. If the sequencer begins enforcing a minimum fee (a documented future direction), any zero-fee `depositWithMessage` message will remain unprocessed indefinitely. The minted STRK tokens will be locked in the StarkGate bridge, and `on_receive` on the L2 `RewardSupplier` will never be called, so `l1_pending_requested_amount` is never decremented — causing the L2 contract to over-request future mints.

---

### Impact Explanation

**Primary (current):** Griefing — any caller can drain all pending L2→L1 mint-request messages and trigger minting of up to `5 × 1,300,000 STRK` per call without paying for L2 message processing. The sequencer bears the cost of including these messages. This matches **Medium: Griefing with no profit motive but damage to users or protocol**.

**Secondary (if sequencer enforces minimum fees):** Minted reward tokens are permanently stranded in the StarkGate bridge. Stakers and delegators cannot claim accrued rewards. This matches **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

`tick()` is permissionless — any Ethereum address can call it. No special role, key, or privilege is required. The attacker pays only the L1 gas cost of the call (~50–80k gas), which is negligible compared to the value of the minted tokens. The attack is trivially repeatable whenever new L2→L1 mint-request messages accumulate.

---

### Recommendation

Add a minimum `msg.value` guard at the top of `tick()` that ensures each of the two L1→L2 messages receives at least the sequencer's required minimum fee. A reasonable approach:

```solidity
function tick() external payable {
    require(msg.value >= 2 * MIN_L1_MSG_FEE, "INSUFFICIENT_FEE");
    ...
}
```

where `MIN_L1_MSG_FEE` is a configurable constant (or read from the messaging contract). Additionally, consider replacing the `msg.value / 2` split with an explicit parameter so callers can allocate fee proportionally to message priority.

### Proof of Concept

1. The L2 `RewardSupplier` calls `send_mint_request_to_l1_reward_supplier()`, emitting one or more L2→L1 messages. [1](#0-0) 

2. These messages become consumable on L1 via `messagingContract().l2ToL1Messages(msgHash)`. [2](#0-1) 

3. An attacker calls `RewardSupplier.tick()` with `msg.value = 0`. The function enters the `if (amountToMint > 0)` branch. [3](#0-2) 

4. L2→L1 messages are consumed (irreversible). Tokens are minted. [4](#0-3) 

5. `bridge().depositWithMessage{value: 0}(...)` is called. The `IBridge` interface is payable with no minimum fee check. [5](#0-4) 

6. `messagingContract().sendMessageToL2{value: 0}(...)` is called. `IStarknetMessaging` only enforces a maximum fee, not a minimum. [6](#0-5) 

7. Both L1→L2 messages are stored with fee = 0. The transaction succeeds. The attacker paid only L1 gas. If the sequencer stops processing zero-fee messages, the minted STRK is stranded in the bridge and `on_receive` on L2 is never triggered, freezing the reward tokens. [7](#0-6)
