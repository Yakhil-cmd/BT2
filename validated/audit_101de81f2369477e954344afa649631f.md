### Title
ERC-20 Tokens Permanently Frozen When `exitToNear` Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns a user's ERC-20 tokens during EVM execution and then schedules a `ft_transfer` (or `ft_transfer_call`) promise to deliver the corresponding NEP-141 tokens to the NEAR recipient. When the `error_refund` compile-time feature is disabled **and** the exit does not involve wNEAR unwrapping, no callback is attached to that promise. If `ft_transfer` fails on the NEP-141 contract side, the burned ERC-20 tokens are permanently unrecoverable — an exact structural analog to the multisig bug where state is committed before the cross-contract call is confirmed.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear` precompile constructs a `callback_args` struct:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is **not** enabled, `refund` is unconditionally `None`. `transfer_near` is only non-`None` when the exit involves unwrapping wNEAR. For a plain ERC-20 exit, `transfer_near` is also `None`, making `callback_args` equal to its default value.

The promise is then constructed conditionally:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

When `callback_args` is default, only a bare `PromiseArgs::Create` is emitted — no `exit_to_near_precompile_callback` is scheduled. The EVM state (token burn) has already been committed before this promise is dispatched. If `ft_transfer` on the NEP-141 contract subsequently fails (e.g., recipient account not registered for storage, NEP-141 contract paused, or insufficient Aurora engine balance), there is no callback to detect the failure and no refund path to restore the burned tokens.

The callback handler itself confirms the refund logic is gated on `args.refund` being `Some`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // ← silent no-op on failure when refund is None
};
``` [3](#0-2) 

---

### Impact Explanation

A user's ERC-20 tokens are burned in the EVM (state committed) before the NEAR-side `ft_transfer` resolves. If that promise fails and no callback is attached, the tokens are gone from the EVM ledger and never arrive on the NEAR side. This constitutes **permanent freezing of user funds** with no administrative recovery path, since the EVM burn is irreversible and no retry mechanism exists.

---

### Likelihood Explanation

The `error_refund` feature is a compile-time flag. If the production Aurora Engine binary is built without it (which the conditional compilation structure explicitly supports), every plain ERC-20 `exitToNear` call is exposed. Failure conditions for `ft_transfer` are realistic and user-reachable: a NEAR recipient that has not registered storage on the NEP-141 contract will cause `ft_transfer` to panic/fail, which is a common operational condition on NEAR. Any EVM user can trigger this by specifying such a recipient.

---

### Recommendation

- **Short term:** Unconditionally attach the `exit_to_near_precompile_callback` to every `exitToNear` promise, regardless of whether `error_refund` is enabled or `transfer_near` is set. The callback should always check the promise result and, on failure, re-mint the burned ERC-20 tokens to the original sender.
- **Long term:** Remove the `error_refund` compile-time gate for the refund path; make failure handling mandatory. Document the invariant that EVM state must never be committed before the corresponding NEAR-side promise is confirmed successful.

---

### Proof of Concept

1. User holds 100 units of an ERC-20 token on Aurora backed by a NEP-141 contract.
2. User calls `exitToNear` specifying a NEAR recipient account that has **not** registered storage on the NEP-141 contract.
3. The EVM executes: 100 tokens are burned from the user's ERC-20 balance (state committed).
4. Aurora schedules `PromiseArgs::Create(ft_transfer_promise)` — no callback, because `error_refund` is not compiled in and `transfer_near` is `None`.
5. `ft_transfer` panics on the NEP-141 contract because the recipient has no storage deposit.
6. No callback fires. The EVM burn is permanent. The user has lost 100 tokens with no recovery path. [4](#0-3) [5](#0-4)
