### Title
Missing Caller Authorization in `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as restricted to "Only starkware sequencer," but the implementation enforces no such check. Any unprivileged address can call it every block with `disable_rewards: true`, consuming the single global `last_reward_block` slot and permanently zeroing out all consensus block rewards for every staker.

---

### Finding Description

The spec explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation's only gate is `general_prerequisites()`:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [2](#0-1) 

There is no sequencer check anywhere in `src/staking/staking.cairo` — the word "sequencer" does not appear in the file at all.

The `update_rewards` entry point:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    ...
``` [3](#0-2) 

Two properties combine to make this exploitable:

1. **`last_reward_block` is a single global variable** — not per-staker. Only one call to `update_rewards` succeeds per block, for any staker. Once it fires, every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

2. **`disable_rewards: true` skips reward distribution but still writes `last_reward_block`** — the block's reward opportunity is consumed and permanently lost.

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` once per block, before the legitimate sequencer call, burns every block's reward slot. No staker ever accumulates `unclaimed_rewards_own`.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

All consensus block rewards are suppressed indefinitely. Stakers with large STRK pools accrue zero yield. The lost rewards are not deferred or recoverable; each block's reward window is a one-shot opportunity gated by `last_reward_block`.

---

### Likelihood Explanation

The call is fully permissionless (any non-zero address), costs only gas, and requires no capital. A single bot running one transaction per block is sufficient to sustain the attack indefinitely. The attacker has no profit motive requirement — pure griefing is sufficient.

---

### Recommendation

Add an explicit sequencer-only guard to `update_rewards`, consistent with the spec. For example, store the authorized sequencer address at construction time and assert `get_caller_address() == self.sequencer_address.read()` at the top of `update_rewards`, or reuse whatever role-based mechanism governs other privileged operations in the contract.

---

### Proof of Concept

1. Deploy the staking system and advance past the consensus rewards activation epoch.
2. From an arbitrary EOA (attacker), call:
   ```
   update_rewards(staker_address: <any_active_staker>, disable_rewards: true)
   ```
   once per block, every block.
3. The legitimate sequencer's call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
4. After N blocks, call `claim_rewards` for any staker — the returned amount is zero.
5. Compare against a baseline run where the attacker does not interfere — the baseline shows non-zero rewards, confirming permanent suppression.

The existing test `update_rewards_disable_rewards_consensus_rewards_flow_test` already demonstrates that calling with `disable_rewards: true` after consensus is active yields zero rewards and blocks any same-block retry: [4](#0-3) 

The only missing piece in the PoC is that the caller in the test is the privileged system actor — substituting an arbitrary address produces the same outcome because no caller check exists.
