### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `change_reward_address` in the Staking contract and in the Pool contract, as well as the initial `stake()` and `enter_delegation_pool()` entry points, only guard against the reward address being a registered token address. There is no check that `reward_address` is non-zero. A staker or pool member can set their `reward_address` to `0x0`, causing all accumulated STRK rewards to be permanently transferred to the zero address when rewards are claimed.

---

### Finding Description

In `staking.cairo`, `change_reward_address` performs only one validation:

```cairo
// src/staking/staking.cairo lines 520-524
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

The zero address (`0x0`) is not a registered token, so this check passes. The function then unconditionally writes the zero address as the staker's `reward_address`.

The same gap exists in `pool.cairo`'s `change_reward_address`:

```cairo
// src/pool/pool.cairo lines 506-510
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Again, zero address passes this check.

The same gap exists at registration time in `stake()` (staking.cairo lines 307-311) and `enter_delegation_pool()` (pool.cairo line 195).

When rewards are later claimed, `send_rewards_to_staker` in `staking.cairo` unconditionally transfers to whatever `reward_address` is stored:

```cairo
// src/staking/staking.cairo lines 1620-1625
let reward_address = staker_info.reward_address;
let amount = staker_info.unclaimed_rewards_own;
...
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

Likewise in `pool.cairo`'s `claim_rewards`:

```cairo
// src/pool/pool.cairo line 366
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

There is no zero-address guard at the transfer site either.

Critically, `unstake_action` is callable by **any address** (per the spec: "Any address can execute"). Its flow calls `send_rewards_to_staker` before returning the principal. So once a staker sets `reward_address = 0`, any third party can trigger `unstake_action` after the exit window and cause the rewards to be burned.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A staker or pool member who sets `reward_address` to `0x0` will have all accrued STRK rewards transferred to the zero address upon the next `claim_rewards` or `unstake_action` call. These tokens are irrecoverably lost. The principal is returned to the staker address, but the yield is permanently destroyed. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**Medium.** The entry path requires only the staker or pool member themselves to call `change_reward_address(0)` — no privileged role is needed. This could happen by mistake (user error, front-end bug, scripting error) or deliberately as a griefing/self-sabotage vector. Once set, the loss is irreversible because there is no recovery mechanism and `unstake_action` is permissionlessly callable by anyone after the exit window.

---

### Recommendation

Add a non-zero check in every function that accepts or stores a `reward_address`:

1. `stake()` in `src/staking/staking.cairo`
2. `change_reward_address()` in `src/staking/staking.cairo`
3. `enter_delegation_pool()` in `src/pool/pool.cairo`
4. `change_reward_address()` in `src/pool/pool.cairo`

Example fix (Cairo):
```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::REWARD_ADDRESS_IS_ZERO);
```

Optionally, also add a defensive zero-address guard at the transfer site in `send_rewards_to_staker` and `claim_rewards` as a defense-in-depth measure.

---

### Proof of Concept

**Staker path:**

1. Staker calls `staking.stake(reward_address: 0x0, ...)` — passes all current checks since `0x0` is not a token address. [1](#0-0) 
2. Staker earns rewards over epochs via attestation.
3. Anyone calls `staking.unstake_action(staker_address)` after the exit window. [2](#0-1) 
4. `send_rewards_to_staker` executes `checked_transfer(recipient: 0x0, amount: rewards)` — rewards are permanently lost. [3](#0-2) 

**Pool member path:**

1. Pool member calls `pool.enter_delegation_pool(reward_address: 0x0, amount)` — passes the `REWARD_ADDRESS_IS_TOKEN` check. [4](#0-3) 
2. Alternatively, an existing pool member calls `pool.change_reward_address(0x0)`. [5](#0-4) 
3. Pool member calls `pool.claim_rewards(pool_member)`.
4. `checked_transfer(recipient: 0x0, amount: rewards)` executes — rewards are permanently lost. [6](#0-5)
