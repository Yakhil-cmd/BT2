### Title
Gateway L2 Gas Price Validation Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Incorrect Admission Decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful resource-bounds check and blockifier pre-validation both use the **current block's** `l2_gas_price` as the reference price. The actual block that will include the transaction uses `next_l2_gas_price`, which is computed from the current block's gas consumption via an EIP-1559-style formula. Because these two values diverge by up to ~2% per block, the gateway systematically admits transactions that will fail in the real block (when the price is rising) and rejects transactions that would succeed (when the price is falling). A developer TODO in the source code explicitly acknowledges the wrong field is being read.

---

### Finding Description

**Root cause — `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`**

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from the synced block header. It reads `block_header.l2_gas_price` (the price *used in* the current block) and discards `block_header.next_l2_gas_price` (the price *for* the next block). The `BlockInfo` struct has no `next_l2_gas_price` field, so the information is silently dropped. [1](#0-0) 

**First use — `validate_resource_bounds`**

`validate_resource_bounds` reads `gas_prices.strk_gas_prices.l2_gas_price` from the `BlockInfo` returned above and checks that the transaction's `max_price_per_unit` is at or above a configured percentage of that value. The developer TODO in the same function acknowledges the wrong field is being used:

```
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [2](#0-1) 

**Second use — `run_validate_entry_point`**

`run_validate_entry_point` builds the blockifier `BlockContext` from the same `get_block_info()` call, only incrementing the block number. The gas prices in this context are therefore the current block's prices, not the next block's prices. The blockifier's `check_fee_bounds` inside `perform_pre_validation_stage` then checks `resource_bounds.max_price_per_unit >= block_info.gas_prices.l2_gas_price`, again using the wrong reference. [3](#0-2) [4](#0-3) 

**What the correct value is**

`StorageBlockHeader` and `BlockHeaderWithoutHash` both carry `next_l2_gas_price`, which is the EIP-1559-derived price for the *next* block. This is the value written into the block header by the consensus orchestrator and is what the actual block will use. [5](#0-4) [6](#0-5) 

**How `next_l2_gas_price` diverges from `l2_gas_price`**

`calculate_next_base_gas_price` implements EIP-1559 with `gas_price_max_change_denominator = 48` and `gas_target = 1,040,000,000`. The price change per block is:

```
price_change = price * |gas_used - gas_target| / (gas_target * 48)
```

At 75% block fullness the price rises ~2.1%; at 25% fullness it falls ~1.0%; at 0% (empty block) it falls ~2.1%. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Two distinct failure modes arise, both matching the "High" admission impact:

**Mode A — valid transactions incorrectly rejected (price falling)**

When the previous block was below the gas target, `next_l2_gas_price < l2_gas_price`. A transaction with `next_l2_gas_price ≤ tx_price < l2_gas_price` satisfies the actual block's requirement but is rejected by the gateway's `validate_resource_bounds` check (and would also fail the blockifier pre-validation inside `run_validate_entry_point`). The user receives a `GAS_PRICE_TOO_LOW` error even though their transaction would have been accepted by the sequencer. [9](#0-8) 

**Mode B — invalid transactions incorrectly admitted (price rising)**

When the previous block was above the gas target, `next_l2_gas_price > l2_gas_price`. A transaction with `l2_gas_price ≤ tx_price < next_l2_gas_price` passes both gateway checks but fails `check_fee_bounds` in the actual block with `InsufficientResourceBounds`. The mempool accepts and forwards a transaction that will be rejected during execution. [10](#0-9) 

---

### Likelihood Explanation

`next_l2_gas_price` equals `l2_gas_price` only when a block is exactly at the gas target. In practice every block deviates, so the mismatch is present in every block. The magnitude (~1–2% per block) is large enough to affect real transactions whose `max_price_per_unit` is set close to the current market price, which is the normal behavior of well-behaved wallets. No special privileges are required; any user submitting a V3 (`AllResources`) transaction is affected.

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a separate method) to also return `next_l2_gas_price` from `BlockHeaderWithoutHash`.
2. In `validate_resource_bounds`, replace `gas_prices.strk_gas_prices.l2_gas_price` with the `next_l2_gas_price` value from the block header (resolving the existing TODO).
3. In `run_validate_entry_point`, populate the `BlockContext`'s L2 gas price with `next_l2_gas_price` so that the blockifier pre-validation uses the same reference as the actual block. [11](#0-10) 

---

### Proof of Concept

**Scenario: valid transaction rejected (Mode A)**

1. Block N executes with `l2_gas_price = 30,000,000,000 fri` and is 25% full.
2. `calculate_next_base_gas_price(30 Gwei, gas_target/4, gas_target, min)` ≈ **29,687,500,000 fri** (−1.04%).
3. Block N is committed; `next_l2_gas_price = 29,687,500,000` is stored in the header.
4. User submits a V3 invoke with `l2_gas.max_price_per_unit = 29,800,000,000 fri`.
5. Gateway calls `validate_resource_bounds`:
   - reads `l2_gas_price = 30,000,000,000` (current block's price, **wrong**).
   - threshold = 100% × 30,000,000,000 = 30,000,000,000.
   - 29,800,000,000 < 30,000,000,000 → **rejected** with `GAS_PRICE_TOO_LOW`.
6. The actual block N+1 would use `next_l2_gas_price = 29,687,500,000`:
   - 29,800,000,000 ≥ 29,687,500,000 → **would have been accepted**.

**Scenario: invalid transaction admitted (Mode B)**

1. Block N executes with `l2_gas_price = 30,000,000,000 fri` and is 75% full.
2. `next_l2_gas_price` ≈ **30,312,500,000 fri** (+1.04%).
3. User submits a V3 invoke with `l2_gas.max_price_per_unit = 30,100,000,000 fri`.
4. Gateway `validate_resource_bounds`: 30,100,000,000 ≥ 30,000,000,000 → **passes**.
5. `run_validate_entry_point` blockifier check: 30,100,000,000 ≥ 30,000,000,000 → **passes**.
6. Transaction enters mempool.
7. Actual block N+1 uses `next_l2_gas_price = 30,312,500,000`:
   - `check_fee_bounds`: 30,100,000,000 < 30,312,500,000 → **fails** with `InsufficientResourceBounds`. [12](#0-11) [2](#0-1)

        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
