### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State (1) Instead of Restoring Pre-Pause State, Leaving Pool Permanently Swap-Blocked — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` hardcodes the destination pause level to `1` (admin-paused) regardless of what state the pool was in before the protocol pause. When the protocol pauses an **active** pool (state `0`) and then unpauses it, the pool is left in state `1` (admin-paused), where swaps remain blocked. The pre-pause state is never stored, so the protocol's own unpause mechanism cannot restore the pool to its original active state.

---

### Finding Description

The pool's pause state machine has three levels:

| Level | Meaning | Swaps |
|---|---|---|
| `0` | Active | Allowed |
| `1` | Admin-paused | Blocked |
| `2` | Protocol-paused | Blocked |

`protocolPausePool` accepts both state `0` and state `1` as valid source states: [1](#0-0) 

`protocolUnpausePool` always transitions to state `1`, unconditionally: [2](#0-1) 

The factory stores no record of the pre-protocol-pause state. When the sequence is:

```
state 0 (active) → protocolPausePool → state 2 → protocolUnpausePool → state 1 (admin-paused)
```

the pool is left in state `1`, which is still a paused state. The `whenNotPaused` modifier on `swap` checks `pauseLevel != 0`: [3](#0-2) 

State `1` satisfies `pauseLevel != 0`, so all swaps continue to revert with `PoolPaused` after the protocol "unpause."

The `swap` function is the only entry point gated by `whenNotPaused`: [4](#0-3) 

`addLiquidity` and `removeLiquidity` are not gated, so only swap functionality is broken — but swap is the core revenue-generating operation of the pool.

---

### Impact Explanation

After a `protocolPausePool` → `protocolUnpausePool` cycle on a previously active pool:

- All calls to `swap` revert with `PoolPaused`.
- The pool is functionally dead for traders and arbitrageurs.
- Protocol fees from spread and notional accumulate at zero (no swaps occur).
- LP positions earn no fees and cannot be rebalanced via swaps.

The pool admin must independently discover the broken state and call `unpausePool` (state `1` → `0`) to restore swap functionality. If the admin is unavailable, the pool remains permanently swap-blocked with no on-chain signal that an additional action is required.

This matches the **Broken core pool functionality** impact gate: the pool's primary operation (swap) is rendered unusable by a normal protocol operation.

---

### Likelihood Explanation

This triggers every time the protocol owner pauses and then unpauses a pool that was in state `0` (active) — the most common operational scenario (e.g., emergency pause during an incident, followed by resumption). No special conditions, malicious actors, or non-standard tokens are required. The factory owner is semi-trusted and performing a legitimate action; the bug is in the protocol's own state machine, not in the owner's intent.

---

### Recommendation

Store the pre-protocol-pause level in the factory and restore it on unpause:

```solidity
mapping(address => uint8) public preProtocolPauseLevel;

function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    preProtocolPauseLevel[pool] = cur;          // ← store original state
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    uint8 restore = preProtocolPauseLevel[pool]; // ← restore original state
    delete preProtocolPauseLevel[pool];
    IMetricOmmPoolFactoryActions(pool).setPause(restore);
}
```

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel = 0`). Swaps succeed.
2. Protocol owner calls `protocolPausePool(pool)`.
   - Check: `cur == 0` → passes. Pool set to state `2`. Swaps revert.
3. Incident resolved. Protocol owner calls `protocolUnpausePool(pool)`.
   - Check: `cur == 2` → passes. Pool set to state `1` (admin-paused). **Swaps still revert.**
4. Users call `swap(...)` → `_checkNotPaused()` → `pauseLevel == 1 != 0` → `revert PoolPaused()`.
5. Pool admin must independently call `unpausePool(pool)` (state `1` → `0`) to restore swaps. Until then, the pool is swap-dead despite the protocol having "unpaused" it.
