### Title
`PriceVelocityGuardExtension` velocity guard is completely bypassed on the first swap due to uninitialized `lastMidPriceX64` — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` is a `beforeSwap` hook that caps how fast the oracle mid-price can move between blocks to protect LPs from oracle price manipulation. Because `priceVelocityState[pool].lastMidPriceX64` is never initialized at pool creation, it defaults to `0`. The guard's check `if (prevMid != 0)` unconditionally skips the entire velocity enforcement on the first swap for every pool using this extension, regardless of how large the oracle price jump is or whether `maxChangePerBlockE18` is set.

---

### Finding Description

`PriceVelocityGuardExtension` stores per-pool state in a mapping:

```solidity
mapping(address pool => PriceVelocityState) public priceVelocityState;
``` [1](#0-0) 

`PriceVelocityState` contains `lastMidPriceX64`, `lastUpdateBlock`, and `maxChangePerBlockE18`, all of which default to `0` for a new pool entry. [2](#0-1) 

In `beforeSwap`, the guard reads the previous mid-price and then gates the entire velocity check behind `if (prevMid != 0)`:

```solidity
uint128 prevMid = s.lastMidPriceX64;   // == 0 on first swap
uint64 prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64 = midPrice;
s.lastUpdateBlock = uint64(block.number);

if (prevMid != 0) {                     // always false on first swap → guard skipped
    uint64 maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        // velocity check ...
    }
}
``` [3](#0-2) 

Unlike `OracleValueStopLossExtension`, which implements an `initialize` function called by the factory at pool creation to seed its config, `PriceVelocityGuardExtension` has no `initialize` override. The factory calls `initialize` on all registered extensions:

```solidity
for (uint256 i = 0; i < params.extensions.length; i++) {
    CallExtension.callExtension(
        params.extensions[i], abi.encodeCall(IMetricOmmExtensions.initialize, (pool, params.extensionInitData[i]))
    );
}
``` [4](#0-3) 

But since `PriceVelocityGuardExtension` does not implement `initialize`, `lastMidPriceX64` is never seeded. The only way to pre-seed it is via the admin-only `setLastMidPrice`, which is a separate, optional, post-deployment call:

```solidity
function setLastMidPrice(address pool_, uint128 newLastMidPriceX64) external onlyPoolAdmin(pool_) {
    PriceVelocityState storage s = priceVelocityState[pool_];
    s.lastMidPriceX64 = newLastMidPriceX64;
    s.lastUpdateBlock = uint64(block.number);
    ...
}
``` [5](#0-4) 

If the admin does not call `setLastMidPrice` before the first swap (which is the natural state after deployment), any user who executes the first swap bypasses the velocity guard entirely, regardless of how extreme the oracle price movement is.

---

### Impact Explanation

The `PriceVelocityGuardExtension` is the protocol's primary defense against oracle price manipulation at the pool level. Its stated purpose is to cap how fast the oracle mid-price can move between blocks. On the first swap, this defense is completely absent. An attacker who can influence the oracle price (e.g., via a flash-loan-assisted manipulation of a Chainlink or Pyth feed, or by timing a swap during a legitimate but extreme price spike) can execute a swap at a price that would be rejected on any subsequent swap. Because the pool prices all trades directly from the oracle bid/ask, a dramatically moved oracle price translates directly into LP assets being sold at a stale or manipulated rate, causing LP principal loss. After the first swap, `lastMidPriceX64` is set to the current mid-price and the guard operates correctly from that point forward — the damage is limited to the first swap window.

---

### Likelihood Explanation

Every pool that registers `PriceVelocityGuardExtension` as a `beforeSwap` hook is affected from the moment of deployment until the first swap completes. The window is open to any unprivileged user. An attacker who monitors pool creation events can front-run or race the first legitimate swap. The admin must proactively call `setLastMidPrice` before any swap occurs to close the window, but there is no protocol-level enforcement of this step and no documentation requiring it. The likelihood is medium: it requires oracle price movement or manipulation coinciding with the first-swap window, but the window is always present and the trigger is fully unprivileged.

---

### Recommendation

Implement an `initialize` function in `PriceVelocityGuardExtension` that seeds `lastMidPriceX64` and `lastUpdateBlock` at pool creation time, using the current oracle mid-price:

```solidity
function initialize(address pool, bytes calldata) external override onlyFactory returns (bytes4) {
    // Fetch current mid-price from the pool's price provider and seed the state
    // so the first swap is subject to the velocity guard immediately.
    // (Alternatively, accept an initial mid-price as ABI-encoded `data`.)
    PriceVelocityState storage s = priceVelocityState[pool];
    s.lastUpdateBlock = uint64(block.number);
    // s.lastMidPriceX64 must be set to a non-zero seed; accept it via `data` or read from the provider.
    return IMetricOmmExtensions.initialize.selector;
}
```

The simplest safe approach is to require the deployer to pass the initial mid-price as `extensionInitData` for this extension, decode it in `initialize`, and store it. This mirrors the pattern already used by `OracleValueStopLossExtension`. [6](#0-5) 

---

### Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` as a `beforeSwap` hook and set `maxChangePerBlockE18` to a tight value (e.g., 1% per block).
2. Do **not** call `setLastMidPrice` (the natural post-deployment state).
3. Manipulate or wait for the oracle price to move 50% in one block.
4. Execute the first swap on the pool. `beforeSwap` reads `prevMid = 0`, skips the `if (prevMid != 0)` branch entirely, and the swap succeeds at the 50%-moved price.
5. Execute a second identical swap in the same block. `beforeSwap` now reads `prevMid = <previous mid>`, enters the velocity check, computes `actualSq >> allowedSq`, and reverts with `PriceVelocityExceeded`.

The first swap executes at a price the guard was specifically configured to block; the second does not. LP assets are sold at the manipulated price on the first swap with no recourse. [7](#0-6)
