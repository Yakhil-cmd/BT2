### Title
Gateway `skip_stateful_validations` bypasses `__validate__` for any account with a mempool entry, not only a pending `deploy_account` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is intended to skip the `__validate__` entry point (signature verification) only when a `deploy_account` transaction is pending in the mempool for a not-yet-deployed account. However, the proxy check it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the mempool for that address, not specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` can submit an invoke with nonce=1 carrying an arbitrary payload and a forged/empty signature, and the gateway will admit it to the mempool without ever calling `__validate__`.

---

### Finding Description

**Root cause — wrong authorization proxy**

`skip_stateful_validations` is the Sequencer-native analog of the external report's `fetchVirtualAccount` flaw: both make an authorization decision based on an address alone, without verifying the *type* of the authorizing context.

```rust

The mempool function it calls is:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)

``` [2](#0-1) 

This returns `true` whenever **any** transaction for the address is in the pool — an invoke with nonce=0, a prior skip-validated invoke with nonce=1, or the intended `deploy_account`. The code comment acknowledges the gap ("either it has a deploy_account transaction or transactions with future nonces that passed validations") but does not recognise that the second branch is itself exploitable.

**How the gateway uses the result**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false` and calls `blockifier_validator.validate(account_tx)`. Inside `StatefulValidator::perform_validations`, the `validate` flag is checked and the `__validate__` call is skipped entirely:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 76-81
ApiTransaction::Invoke(_) => {
    ...
    if !tx.execution_flags.validate {
        return Ok(());   // ← signature never checked
    }
    let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

The gateway then calls `run_validate_entry_point` with `skip_validate = true`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

The transaction is then forwarded to the mempool with no signature check having occurred.

**Attack steps**

1. Victim broadcasts a `deploy_account` (nonce=0) for address `A`. It enters the mempool.
2. Attacker submits `Invoke{sender=A, nonce=1, calldata=<arbitrary>, signature=<empty>}`.
3. Gateway `validate_nonce`: account_nonce=0, tx_nonce=1, within `max_allowed_nonce_gap` → passes.
4. Gateway `validate_by_mempool`: no duplicate hash, nonce gap acceptable → passes.
5. Gateway `skip_stateful_validations`: nonce==1 ∧ account_nonce==0 ∧ `account_tx_in_pool_or_recent_block(A)` == true → returns `true`.
6. `run_validate_entry_point` called with `skip_validate=true` → `__validate__` never executed.
7. Attacker's invoke is admitted to the mempool with an invalid/forged signature.

**What happens at block-building time**

The batcher creates `AccountTransaction` via `new_for_sequencing` which always sets `validate: true`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 147-155
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        ...
        validate: true,
        ...
    };
``` [5](#0-4) 

So the batcher will call `__validate__` and the attacker's transaction will fail validation and be rejected. The nonce is not consumed. However, the attacker's transaction **occupied the nonce=1 slot in the mempool** for the duration of the block cycle. The victim's legitimate invoke with nonce=1 is rejected by the mempool as a duplicate nonce until the attacker's transaction is evicted. The attacker can repeat this with fee-escalated replacements, creating a persistent DoS.

---

### Impact Explanation

The gateway admits an invoke transaction with an invalid/forged signature to the mempool, violating the invariant that every transaction in the mempool has passed `__validate__`. This matches the allowed High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Concretely: any user who broadcasts a `deploy_account` is immediately vulnerable to having their nonce=1 slot poisoned by an attacker, preventing them from submitting a legitimate first invoke until the attacker's transaction is evicted (which requires fee escalation or waiting for block inclusion and rejection).

---

### Likelihood Explanation

- The mempool is observable (P2P propagation, public RPC).
- The attack requires no privileged access — any unprivileged sender can submit an invoke to the gateway.
- The condition (nonce=1, account_nonce=0, any tx in pool) is trivially satisfied whenever a user deploys a new account.
- The UX feature is explicitly tested and enabled in production configuration, so `max_allowed_nonce_gap >= 1` is guaranteed.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a `deploy_account` transaction is present for the address:

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,

    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.

                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
