### Title
`update_rewards` Lacks Access Control — Any Caller Can Permanently Freeze Staker Block Rewards - (File: src/staking/staking.cairo)

### Summary

`StakingRewardsManagerImpl::update_rewards` in `src/staking/staking.cairo` is callable by any address. The function enforces a per-block guard (`REWARDS_ALREADY_UPDATED`) that prevents it from being called more than once per block. An unprivileged attacker can front-run the legitimate sequencer call every block by invoking `update_rewards(staker_address, disable_rewards: true)`, consuming the one-call-per-block slot without distributing rewards. This permanently freezes all consensus-phase block rewards for any targeted staker.

---

### Finding Description

The spec for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation contains no such check. The function only calls `self.general_prerequisites()` (which checks the pause flag) and then asserts `current_block_number > self.last_reward_block.read()`.

```cairo
// src/staking/staking.cairo lines 1449–1507
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
    self.last_reward_block.write(current_block_number);   // ← consumes the slot

    if disable_rewards || self.is_pre_consensus() {
        return;   // ← exits without distributing rewards
    }
    // ... distribute block rewards
}
```

When `disable_rewards = true`, the function:
1. Passes all guards.
2. Writes `current_block_number` into `last_reward_block`, consuming the per-block slot.
3. Returns immediately without distributing any rewards.

Because `last_reward_block` is a **single global value** (not per-staker), any subsequent call in the same block — including the legitimate sequencer call with `disable_rewards = false` — reverts with `REWARDS_ALREADY_UPDATED`.

The analog to the original report is exact:
- **Original**: Anyone calls `distribute()` → resets `periodFinish` → `setRewardsDuration` can never execute.
- **Analog**: Anyone calls `update_rewards(staker, disable_rewards: true)` → sets `last_reward_block` → the sequencer's `update_rewards(staker, disable_rewards: false)` reverts for that block.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

In the consensus rewards phase (`!is_pre_consensus()`), stakers earn block rewards only when the sequencer calls `update_rewards(..., disable_rewards: false)`. If an attacker calls `update_rewards(..., disable_rewards: true)` at the start of every block, `last_reward_block` is always equal to the current block number, so the sequencer's call always reverts. The staker accrues zero rewards indefinitely. Because the attacker can repeat this for every block at negligible cost (a single Starknet transaction per block), the freeze is effectively permanent.

---

### Likelihood Explanation

**High.** The entry path requires no privileged access, no tokens, and no special state. Any address can call `update_rewards` with any `staker_address` and `disable_rewards = true`. The attacker only needs to submit one transaction per block ahead of the sequencer. On Starknet, where the sequencer ordering is observable, this is straightforward. The attack can target a single staker or all stakers simultaneously (by cycling through staker addresses in one transaction or across multiple).

---

### Recommendation

Add an access-control guard that restricts `update_rewards` to the Starknet sequencer (or a designated operator address), matching the specification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_sequencer(); // enforce spec: "Only starkware sequencer"
    // ...
}
```

Alternatively, if the intent is to allow any caller, the `disable_rewards` flag must be removed and the function must always distribute rewards, or the per-block guard must be made per-caller rather than global.

---

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Staker `S` is registered and has non-zero balance.
3. At block `N`, attacker calls:
   ```
   update_rewards(staker_address: S, disable_rewards: true)
   ```
   - `current_block_number (N) > last_reward_block` → passes.
   - `last_reward_block` is written to `N`.
   - Returns early; no rewards distributed.
4. Sequencer calls:
   ```
   update_rewards(staker_address: S, disable_rewards: false)
   ```
   - `current_block_number (N) > last_reward_block (N)` → **false** → reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` receives zero rewards for block `N`.
6. Attacker repeats step 3 at every subsequent block. Staker `S` permanently receives zero consensus-phase rewards.
