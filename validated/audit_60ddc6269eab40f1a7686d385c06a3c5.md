### Title
No Minimum Fee Validation in `tick()` Allows Griefing That Permanently Freezes Unclaimed Yield - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is a permissionless, payable function that sends two L1→L2 messages when minting is required. It splits `msg.value` between the two messages with no minimum fee check. An unprivileged caller can invoke `tick()` with `msg.value = 0`, causing both messages to be sent with zero fee. The Starknet sequencer will not process zero-fee L1→L2 messages, so the minted tokens become permanently stuck in the StarkGate bridge and the L2 reward supplier's `l1_pending_requested_amount` accounting is corrupted, blocking all future reward claims.

---

### Finding Description

`tick()` in `L1/starkware/solidity/stake/RewardSupplier.sol` has no access control and no minimum `msg.value` requirement:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // Irreversibly consume L2→L1 mint request messages
        for (uint256 i = 0; i < numMsgsToConsume; i++) {
            messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
        }
        mintManager().mintRequest(token(), amountToMint);

        // Both fees are 0 when msg.value == 0
        uint256 msgFee = msg.value / 2;                          // = 0
        bridge().depositWithMessage{value: msgFee}(...);         // 0-fee L1→L2

        msgFee = msg.value - msgFee;                             // = 0
        messagingContract().sendMessageToL2{value: msgFee}(...); // 0-fee L1→L2
    }
}
``` [1](#0-0) 

The two critical consequences:

**1. Minted tokens stuck in bridge.** `bridge().depositWithMessage{value: 0}(...)` registers an L1→L2 message with zero fee. The Starknet sequencer requires a non-zero fee to include L1→L2 messages. The message is never processed, so `on_receive` on the L2 `RewardSupplier` is never called, and the minted tokens remain locked in StarkGate.

**2. L2 accounting permanently corrupted.** On L2, `request_funds` tracks in-flight minting via `l1_pending_requested_amount`. This variable is only decremented inside `on_receive`:

```cairo
fn on_receive(...) -> bool {
    ...
    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    if amount_u128 > l1_pending_requested_amount {
        self.l1_pending_requested_amount.write(Zero::zero());
    } else {
        l1_pending_requested_amount -= amount_u128;
        self.l1_pending_requested_amount.write(l1_pending_requested_amount);
    }
    true
}
``` [2](#0-1) 

Since `on_receive` is never called, `l1_pending_requested_amount` remains inflated. The `request_funds` function computes `credit = balance + l1_pending_requested_amount` and only sends new mint requests when `credit < debit + threshold`:

```cairo
let credit = balance + l1_pending_requested_amount;
let debit = unclaimed_rewards;
if credit < debit + threshold {
    // send new mint requests
}
``` [3](#0-2) 

With `l1_pending_requested_amount` inflated by the ghost amount, `credit` appears sufficient and no new mint requests are ever sent. The L2 reward supplier's actual token balance never grows, so all subsequent `claim_rewards` calls fail with `AMOUNT_TOO_HIGH` once the existing balance is exhausted.

---

### Impact Explanation

**Temporary (potentially permanent) freezing of unclaimed yield.** Once the attack succeeds:
- Minted STRK tokens are locked in StarkGate with no recovery path (the bridge contract is not the message sender, so it cannot cancel the L1→L2 message on behalf of the protocol without a governance action).
- The L2 reward supplier's `l1_pending_requested_amount` is permanently inflated, suppressing all future mint requests.
- All stakers and delegators are unable to claim their accrued rewards.

This matches the allowed impact: **Temporary freezing of unclaimed yield / unclaimed royalties**.

---

### Likelihood Explanation

- `tick()` is `external` with no access control — any EOA or contract can call it.
- The only cost to the attacker is the L1 gas for the transaction itself (no ETH needs to be sent).
- Pending mint requests exist regularly as rewards accumulate each epoch, so the precondition (`amountToMint > 0`) is routinely satisfied.
- A single call is sufficient to trigger the full impact.

Likelihood: **High**.

---

### Recommendation

Add a minimum `msg.value` check before executing the minting path. The minimum should cover the fees for both L1→L2 messages. A simple guard:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        require(msg.value >= 2 * messagingContract().getMaxL1MsgFee() / someMinFactor,
                "INSUFFICIENT_FEE");
        // ... rest of logic
    }
}
```

At minimum, require `msg.value > 0` so that neither message is sent with a zero fee. Ideally, enforce a protocol-defined minimum fee per message (e.g., via a configurable `minMsgFee` storage variable) and revert if `msg.value < 2 * minMsgFee`.

---

### Proof of Concept

1. Wait until `requiredMinting()` returns `amountToMint > 0` (i.e., the L2 reward supplier has sent at least one mint request to L1).
2. Call `RewardSupplier.tick{value: 0}()` from any EOA.
3. Observe:
   - `ConsumedL2MintRequests` event is emitted — the L2→L1 messages are irreversibly consumed.
   - `mintManager().mintRequest(token(), amountToMint)` succeeds — tokens are minted.
   - `bridge().depositWithMessage{value: 0}(...)` succeeds on L1 — the deposit message is registered with 0 fee.
   - `messagingContract().sendMessageToL2{value: 0}(...)` succeeds on L1 — the total supply update is registered with 0 fee.
4. Neither L1→L2 message is ever processed by the Starknet sequencer (0 fee).
5. `on_receive` on the L2 `RewardSupplier` is never called.
6. `l1_pending_requested_amount` on L2 remains inflated; `request_funds` never sends new mint requests.
7. All staker/delegator `claim_rewards` calls eventually fail once the existing L2 balance is depleted. [1](#0-0) [4](#0-3)

            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
