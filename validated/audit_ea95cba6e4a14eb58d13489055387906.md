### Title
Unvalidated `app_pk` Parameter in CKD Protocol Allows Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts `app_pk` (the application's ElGamal public key `A`) as a caller-supplied local parameter with no cross-participant consistency validation. Because there is no protocol round where participants broadcast and verify that they all hold the same `app_pk`, a malicious participant can silently use a different value. The coordinator then sums contributions computed under mismatched keys, producing a `(Y, C)` output that the application cannot correctly unmask. Honest parties accept and return this corrupted CKD output without detecting the inconsistency.

---

### Finding Description

The CKD protocol computes, for each participant `i`:

```
C_i = S_i + y_i * A     where S_i = x_i * H(pk, app_id)
```

and sends `(λ_i * Y_i, λ_i * C_i)` to the coordinator, who sums all contributions to produce `(Y, C)`. The correctness of the final unmask step `C − a·Y = msk·H(pk, app_id)` depends entirely on every participant using the **same** `A`.

In `compute_signature_share()`:

```rust
let big_c = big_s + app_pk * y.0;   // app_pk used directly, no validation
``` [1](#0-0) 

`app_pk` is passed in as a plain argument to `ckd()` and flows straight into `compute_signature_share()` without any check: [2](#0-1) 

There is no protocol round in which participants exchange their `app_pk` values and abort on mismatch. The entire protocol is a single-round fan-in to the coordinator: [3](#0-2) 

The coordinator simply accumulates whatever `(Y_j, C_j)` it receives and returns the sum — no consistency check on the inputs: [4](#0-3) 

This is the direct analog of the external report's root cause: a critical initialization parameter (`priceOracle` / `packedPrices` there; `app_pk` here) is accepted without authentication or cross-party consistency enforcement, allowing a malicious caller to corrupt the protocol output.

---

### Impact Explanation

If malicious participant `m` calls `ckd()` with `app_pk = A′ ≠ A`, the coordinator accumulates:

```
C = msk·H(pk, app_id) + (Σ_{j≠m} λ_j·y_j)·A + λ_m·y_m·A′
```

The application then computes:

```
C − a·Y = msk·H(pk, app_id) + λ_m·y_m·(A′ − A)
```

which is not `msk·H(pk, app_id)`. The BLS signature verification fails and the application cannot derive its confidential key. All honest participants, including the coordinator, accept and return this corrupted `(Y, C)` without detecting the inconsistency.

**Impact class**: High — Corruption of CKD output so honest parties accept an unusable cryptographic output.

---

### Likelihood Explanation

Any participant in the CKD session can trigger this unilaterally by passing an arbitrary `app_pk` to their local `ckd()` call. No special privilege, leaked key, or cryptographic break is required. The malicious participant need only deviate from the agreed `app_pk` value. Because the protocol has no broadcast or consistency-check round for `app_pk`, the deviation is invisible to all other participants until the application fails to unmask the result.

---

### Recommendation

Add a reliable-broadcast round at the start of the CKD protocol in which every participant broadcasts their `app_pk` (or a hash of it). Each participant must verify that all received values match before proceeding to compute their share. This mirrors the session-ID broadcast already used in the DKG protocol: [5](#0-4) 

Alternatively, derive `app_pk` deterministically from a value already authenticated by the protocol (e.g., include it in the `AppId` or bind it to the `key_pair.public_key` via a hash), so that any deviation is cryptographically detectable.

---

### Proof of Concept

1. An honest session is initiated: all `n` participants agree out-of-band to use `app_pk = A` (the app's real ElGamal public key).
2. Malicious participant `m` calls `ckd(participants, coordinator, m, key_pair, app_id, A_prime, rng)` where `A_prime ≠ A`.
3. `m`'s contribution is `C_m = S_m + y_m * A_prime`; it sends `(λ_m·Y_m, λ_m·C_m)` to the coordinator.
4. All other participants send correct contributions `(λ_j·Y_j, λ_j·C_j)` with `C_j = S_j + y_j * A`.
5. The coordinator (honest) sums everything and returns `(Y, C)` where `C = msk·H(pk, app_id) + (Σ_{j≠m} λ_j·y_j)·A + λ_m·y_m·A_prime`.
6. The application computes `C − a·Y = msk·H(pk, app_id) + λ_m·y_m·(A_prime − A)`, which is not the expected BLS signature. The CKD output is permanently corrupted for this session. [6](#0-5)

    me: Participant,
    key_pair: KeygenOutput,

    app_pk: PublicKey,

        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates

        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list

        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });

    let chan = comms.shared_channel();

        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
