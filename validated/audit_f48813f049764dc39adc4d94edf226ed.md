### Title
`DelegateAction` Signed Payload Lacks Chain-Specific Domain Binding, Enabling Cross-Chain Replay - (File: `core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

### Summary

`DelegateAction` (meta-transaction, NEP-366) signatures are computed over a payload that contains no chain identifier. The `MessageDiscriminant` used in `SignableMessage` encodes only a NEP number, not a chain ID or genesis hash. Because `DelegateAction` uses an absolute `max_block_height` instead of a chain-specific `block_hash`, a signed delegate action created for one NEAR network (e.g., testnet) is cryptographically valid on any other network (e.g., mainnet) where the same account ID and key pair exist and the nonce/height conditions are satisfied.

### Finding Description

Regular `SignedTransaction` objects include a `block_hash` field — a hash of a recent block on the specific chain — which is validated against `transaction_validity_period`. Because block hashes are chain-specific, a transaction signed on testnet cannot be replayed on mainnet.

`DelegateAction` (NEP-366 meta-transactions) does not include a `block_hash`. Instead it uses `max_block_height`, an absolute block number:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,   // absolute number, not chain-specific
    pub public_key: PublicKey,
}
``` [1](#0-0) 

The hash signed by the user is:

```rust
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
``` [2](#0-1) 

`SignableMessage` prepends only a `MessageDiscriminant` — a 4-byte NEP number (`(1 << 30) + 366`):

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
``` [3](#0-2) 

The discriminant encodes only the NEP number, not a chain ID, genesis hash, or any network-specific value:

```rust
const NEP_366_META_TRANSACTIONS: u32 = 366;
// ...
SignableMessageType::DelegateAction => {
    MessageDiscriminant::new_on_chain(NEP_366_META_TRANSACTIONS).unwrap()
}
``` [4](#0-3) [5](#0-4) 

The runtime's `apply_delegate_action` verifies the signature, then checks `max_block_height`, `sender_id`, and nonce — none of which are chain-specific:

```rust
if !signed_delegate_action.verify() {
    result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
    return Ok(());
}
let delegate_action = signed_delegate_action.delegate_action();
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
``` [6](#0-5) 

### Impact Explanation

An attacker who observes a `SignedDelegateAction` broadcast on one network (e.g., testnet) can wrap it in their own `SignedTransaction` and submit it on a different network (e.g., mainnet) where:

1. The same `sender_id` account exists with the same key pair (common for developers/users who reuse keys across networks).
2. The `nonce` in the delegate action is greater than the current access key nonce on the target chain.
3. The target chain's block height is below `max_block_height`.

When accepted, the inner `actions` execute on the target chain with `sender_id` as the predecessor. This can include `Transfer`, `FunctionCall`, `AddKey`, `DeleteKey`, or `DeleteAccount` actions. The exact corrupted values are: the access key nonce on the target chain is incremented (consuming a nonce slot), and the inner actions execute against the user's account on the wrong chain without the user's consent.

### Likelihood Explanation

The conditions are realistic:
- Many NEAR users and developers maintain the same account ID and key pair on both mainnet and testnet.
- `DelegateAction` payloads are broadcast to relayer networks/APIs, making them observable by third parties.
- Mainnet and testnet block heights are in the same order of magnitude, so a `max_block_height` set for testnet is often also valid on mainnet.
- The nonce condition is satisfied whenever the user has been more active on the source chain than the target chain.

### Recommendation

Include a chain-specific domain separator in the `DelegateAction` signed payload. The canonical approach is to add the genesis block hash (or a dedicated chain ID field from genesis config) to the `SignableMessage` or directly to `DelegateAction`. This mirrors how regular `SignedTransaction` achieves chain binding via `block_hash`. Concretely:

- Add a `chain_id: CryptoHash` field (genesis hash) to `DelegateAction` and `DelegateActionV2`, or
- Extend `SignableMessage` to include a chain-specific domain tag alongside the NEP discriminant.

The same fix should be applied to `DelegateActionV2` / `VersionedDelegateActionPayload` which uses `SignableMessageType::DelegateActionV2` and has the same structural gap. [7](#0-6) [8](#0-7) 

### Proof of Concept

1. User `alice.near` exists on both mainnet and testnet with the same ed25519 key pair.
2. On testnet at block height 150,000,000, user creates and signs:
   ```
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "bob.near",
     actions: [Transfer { deposit: 10 NEAR }],
     nonce: 1000001,
     max_block_height: 160_000_000,
     public_key: <alice's key>,
   }
   ```
   Hash = `SHA256( LE32(0x4000016E) || borsh(DelegateAction) )` — no chain ID anywhere.
3. Attacker observes this `SignedDelegateAction` from the testnet relayer API.
4. Attacker wraps it in a `SignedTransaction` on **mainnet** (attacker pays gas) and submits it.
5. Mainnet `apply_delegate_action`:
   - `verify()` → passes (same key, same payload bytes, same hash).
   - `block_height (155_000_000) <= max_block_height (160_000_000)` → passes.
   - `sender_id` matches → passes.
   - `nonce (1000001) > ak_nonce on mainnet (e.g., 1000000)` → passes.
6. 10 NEAR is transferred from `alice.near` on **mainnet** to `bob.near` without Alice's mainnet consent.

pub struct DelegateAction {

    pub max_block_height: BlockHeight,
    /// Public key used to sign this delegated action.

}
