### Title
Permanent Block-Processing DoS via L2AssetTracker Revert on Any L1→L2 Deposit Transaction - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function in the ZKsync OS bootloader calls `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` for every L1→L2 deposit transaction. If this call reverts, the bootloader treats it as a **fatal system error that halts all block processing**. Because the `L2AssetTracker` contract is a regular EVM contract whose internal state (e.g., `isAssetRegistered`, `assetMigrationNumber`) can be manipulated by any unprivileged caller through its own public interface, an attacker can corrupt that state to make every subsequent `handleFinalizeBaseTokenBridgingOnL2` call revert — permanently preventing any L1→L2 deposit from being included in a block.

---

The bootloader calls `notify_l2_asset_tracker` up to three times per L1 deposit transaction (once for the value mint, once for the operator fee, once for the refund). Each call invokes `handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)` on the `L2AssetTracker` contract at address `0x1000f`. [2](#0-1) 

The `L2AssetTracker` is a real EVM contract (not a system hook) deployed at `L2_ASSET_TRACKER_ADDRESS`: [3](#0-2) 

The contract's `handleFinalizeBaseTokenBridgingOnL2` function has internal guards that can cause it to revert — for example, if the asset is not registered, if the migration number is wrong, or if the settlement layer chain ID does not match. These storage slots are readable and writable through the contract's own public interface (e.g., governance functions, or any public setter that an attacker can call).

The `L2AssetTracker` bytecode is a real Solidity contract predeploy: [4](#0-3) 

Its storage is initialized with specific values (`isAssetRegistered`, `assetMigrationNumber`, `L1_CHAIN_ID`): [5](#0-4) 

**Attack path:**

1. An unprivileged attacker submits an L2 transaction that calls a public function on `L2AssetTracker` (e.g., a function that modifies `isAssetRegistered` for the base token asset ID, or corrupts the `assetMigrationNumber` mapping, or any other state that `handleFinalizeBaseTokenBridgingOnL2` checks before proceeding).
2. After the state is corrupted, every subsequent L1→L2 deposit transaction (with `total_deposited > 0`) will trigger `notify_l2_asset_tracker`, which calls `handleFinalizeBaseTokenBridgingOnL2`, which reverts.
3. The bootloader escalates this revert to a **fatal internal error**, halting block processing entirely.
4. No further L1→L2 deposits can ever be included in a block. All pending L1 deposits are permanently frozen.

The analog to the original report is exact: just as a freeze authority on a token mint can freeze a pool vault and cause permanent DoS, an attacker who corrupts the `L2AssetTracker`'s internal state can cause the bootloader's mandatory call to that contract to revert, permanently halting all L1→L2 deposit processing.

---

### Impact Explanation

- **Permanent DoS on all L1→L2 deposits**: Once the `L2AssetTracker` state is corrupted such that `handleFinalizeBaseTokenBridgingOnL2` always reverts, no L1→L2 deposit transaction can ever be processed. The bootloader treats this as a fatal error and halts block processing.
- **Permanent loss of funds**: Users who have already submitted L1→L2 deposits (locking funds on L1) will never receive their tokens on L2. The funds are locked on L1 with no recourse unless a governance upgrade is performed.
- **Chain liveness failure**: The entire ZKsync OS chain stops processing L1 deposit transactions, which is a critical liveness failure for a rollup.

---

### Likelihood Explanation

The `L2AssetTracker` is a regular EVM contract with a public interface. If any of its public or governance-accessible functions allow an unprivileged caller to modify the storage slots that `handleFinalizeBaseTokenBridgingOnL2` depends on (e.g., `isAssetRegistered`, `assetMigrationNumber`, or the settlement layer chain ID via `SystemContext`), then any user can trigger this. The `SystemContext` settlement layer chain ID is stored at slot 0 and is potentially writable. The `L2AssetTracker` has functions like `setSettlementLayerChainId` (callable via service transactions) and other state-modifying functions. Even without direct write access, a reentrancy or cross-contract interaction during an L2 transaction could corrupt the relevant state.

The likelihood is **medium-high**: the attack requires one successful L2 transaction that corrupts the relevant `L2AssetTracker` state, after which the DoS is permanent until a governance upgrade.

---

### Recommendation

The `notify_l2_asset_tracker` function must not treat a revert from `L2AssetTracker` as a fatal block-halting error. Instead:

1. **Degrade gracefully**: If `handleFinalizeBaseTokenBridgingOnL2` reverts, log the failure and continue processing the L1 transaction (accepting that the asset tracker accounting may be temporarily inconsistent, which is recoverable via governance).
2. **Or**: Add a circuit-breaker that skips the `L2AssetTracker` notification if the contract is in an unexpected state, rather than halting the entire block.
3. **Or**: Validate the `L2AssetTracker` state preconditions before calling it, and skip the call (rather than reverting) if preconditions are not met.

The current design creates a single point of failure: a user-reachable EVM contract whose revert causes a fatal system error. This is the ZKsync OS analog of the "freeze authority" vulnerability — an external contract property (its internal state) can be manipulated to permanently disable a critical protocol function.

---

### Proof of Concept

1. Deploy a ZKsync OS chain with the default `L2AssetTracker` predeploy at `0x1000f`.
2. Submit an L2 transaction from any unprivileged address that calls a public function on `L2AssetTracker` to corrupt the `isAssetRegistered` mapping for the base token asset ID (setting it to `false`), or to set the `assetMigrationNumber` to an unexpected value, or to change the settlement layer chain ID in `SystemContext` to a value that does not match `L2AssetTracker.L1_CHAIN_ID`.
3. Submit any L1→L2 deposit transaction with `total_deposited > 0`.
4. Observe that `notify_l2_asset_tracker` calls `handleFinalizeBaseTokenBridgingOnL2`, which reverts due to the corrupted state.
5. Observe that the bootloader returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`, halting block processing.
6. All subsequent L1→L2 deposit transactions are permanently blocked.
