### Title
Echo-Broadcast Echo-Threshold Rounded to Zero for n ≤ 3 Allows Split-View in DKG — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

`echo_ready_thresholds` returns `(0, 0)` for all `n ≤ 3`, setting the echo threshold to `0` instead of the correct `floor(n/2) = 1` for `n = 3`. Because the echo-phase check is `> echo_t`, a threshold of `0` means a party advances to READY after receiving only its own simulated echo — before seeing any other party's echo. A malicious sender in a 3-party DKG/reshare/refresh can exploit this to deliver different messages to different honest parties, breaking broadcast agreement and corrupting the DKG output.

---

### Finding Description

`echo_ready_thresholds` in `src/protocol/echo_broadcast.rs` computes the echo and ready thresholds used by the Byzantine Reliable Broadcast protocol:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // <-- echo_threshold = 0 for n = 3
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
``` [1](#0-0) 

The protocol specification in `docs/network-layer.md` states:

> Step 6: Once a counter for ECHO(m) is **greater than** `(N + MaxFaulty) / 2`, send READY(m). [2](#0-1) 

For `n = 3`, `MaxFaulty = floor((3-1)/3) = 0`, so the correct echo threshold is `floor((3+0)/2) = 1`. The check `> 1` requires at least **2** echo votes. The code returns `0`, so the check `> 0` requires only **1** echo vote — the party's own simulated echo, which is injected before any network message is received.

The echo phase in `reliable_broadcast_receive_all` processes the sender's own simulated echo immediately:

```rust
if state_sid.data_echo.get(&data)? > echo_t  // > 0 for n=3
{
    vote = MessageType::Ready(data);
    chan.send_many(wait, &(&sid, &vote))?;
    state_sid.finish_echo = true;
    is_simulated_vote = true;
    from = me;
}
``` [3](#0-2) 

With `echo_t = 0`, the party transitions to READY and sets `finish_echo = true` after its own simulated echo alone, before any real network echo arrives. All subsequent echoes from other parties are then discarded (`state_sid.finish_echo` is already `true`).

The ready delivery threshold `> 2 * ready_t = > 0` is also satisfied by the party's own simulated READY, so the party delivers immediately.

This broadcast is used in DKG rounds 1, 3, and 5.5 via `do_broadcast`: [4](#0-3) 

---

### Impact Explanation

**High — Corruption of DKG outputs so honest parties accept inconsistent public keys, participant sets, or transcripts.**

A malicious participant acting as the sender in any broadcast round of a 3-party DKG, reshare, or refresh can send a different message to each honest party. Because each honest party delivers its received message immediately (after only its own simulated echo), the two honest parties deliver different values. In DKG Round 1 (session-ID broadcast), this causes the two honest parties to compute different aggregate session IDs `sid = H1(sid_1, sid_2, sid_3)`, leading to different polynomial commitments, different share evaluations, and ultimately different public keys. The honest parties complete the protocol believing they hold consistent shares of the same key, but they do not.

---

### Likelihood Explanation

**Low.** The attack requires exactly `n = 3` participants and a malicious participant who is the designated sender in at least one broadcast round. Since every participant is the sender in their own broadcast session (the multi-sender protocol runs `n` parallel instances), any one of the 3 participants can trigger this. The 3-party configuration is a common minimal deployment.

---

### Recommendation

Remove the `n ≤ 3` special case and compute the echo threshold uniformly using the same formula as for larger `n`:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n == 0 {
        return (0, 0);
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

For `n = 3`: `broadcast_threshold = 0`, `echo_threshold = midpoint(3, 0) = 1`. The check `> 1` now requires at least 2 echo votes, preventing a single-party simulated echo from advancing the protocol.

---

### Proof of Concept

**Setup:** 3 participants: A (malicious sender), B (honest), C (honest). `n = 3`, `echo_t = 0`, `ready_t = 0`.

**Attack (DKG Round 1 — session-ID broadcast, session `sid = A`):**

1. A sends `SEND(sid_A_fake1)` to B and `SEND(sid_A_fake2)` to C (two different values).
2. B receives `SEND(sid_A_fake1)`, echoes it. B's simulated echo count for `sid_A_fake1` = 1. `1 > 0` → B sends `READY(sid_A_fake1)`, sets `finish_echo = true`. B's simulated ready count = 1. `1 > 0` → B delivers `sid_A_fake1`.
3. C receives `SEND(sid_A_fake2)`, echoes it. C's simulated echo count for `sid_A_fake2` = 1. `1 > 0` → C sends `READY(sid_A_fake2)`, sets `finish_echo = true`. C delivers `sid_A_fake2`.
4. B receives `ECHO(sid_A_fake2)` from C — ignored (`finish_echo = true`).
5. C receives `ECHO(sid_A_fake1)` from B — ignored (`finish_echo = true`).

**Result:** B holds `sid_A = sid_A_fake1`, C holds `sid_A = sid_A_fake2`. They compute different aggregate session IDs, different polynomial commitments, and ultimately inconsistent key shares. The DKG output is corrupted. [1](#0-0) [5](#0-4) [6](#0-5)

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}

pub async fn do_broadcast<'a, T>(
    chan: &mut SharedChannel,
    participants: &'a ParticipantList,
    me: Participant,
    data: T,
) -> Result<ParticipantMap<'a, T>, ProtocolError>
where
    T: Serialize + Clone + DeserializeOwned + PartialEq,
{
    let wait_broadcast = chan.next_waitpoint();
    let send_vote = reliable_broadcast_send(chan, wait_broadcast, participants, me, data)?;
    let vote_list =
        reliable_broadcast_receive_all(chan, wait_broadcast, participants, me, send_vote).await?;
    Ok(vote_list)
}
