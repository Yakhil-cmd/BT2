### Title
Presignature Reuse Enabled by Non-Consuming API Allows Private Key Recovery - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based and Robust ECDSA presignature types derive `Clone` and expose `rerandomize_presign` accepting `&PresignOutput` (a shared reference). This means the library provides no API-level enforcement of the documented "one-time use" invariant. A malicious coordinator can orchestrate two signing sessions against the same presignature material, producing two ECDSA signatures that share the same underlying nonce `k`. Standard ECDSA nonce-reuse algebra then recovers the aggregate private key.

---

### Finding Description

Both ECDSA presignature types are defined with `#[derive(Clone)]`:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs:40
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub k: Scalar, pub sigma: Scalar }

// src/ecdsa/robust_ecdsa/mod.rs:26
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub c: Scalar, pub e: Scalar, pub alpha: Scalar, pub beta: Scalar }
```

The rerandomization step — the only gate between a raw presignature and a usable signing input — takes the presignature by shared reference, not by value:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs:66-67
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // <-- reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// src/ecdsa/robust_ecdsa/mod.rs:55-56
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // <-- reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Because `PresignOutput` is `Clone` and `rerandomize_presign` does not consume it, the same `PresignOutput` can be rerandomized an arbitrary number of times with different `(msg_hash, tweak, entropy)` tuples, producing distinct `RerandomizedPresignOutput` values that all embed the same underlying nonce scalar `k`.

The library's own security documentation acknowledges the consequence:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> — `docs/ecdsa/robust_ecdsa/signing.md:151-154`

The documentation mandates "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions" (`docs/ecdsa/robust_ecdsa/signing.md:176-177`), but the library enforces no such constraint at the API level. The contrast with the other two mitigations is stark: `N = 2t+1` is enforced in code (`src/ecdsa/robust_ecdsa/presign.rs:75-78`, `src/ecdsa/robust_ecdsa/sign.rs:86-90`) and `h ≠ 0` is enforced in code (`src/ecdsa/robust_ecdsa/sign.rs:91-95`), but single-use of the presignature is left entirely to the caller.

---

### Impact Explanation

Two signing sessions using the same presignature nonce `k` but different message hashes `h1`, `h2` yield signatures `(R, s1)` and `(R, s2)` where:

```
s1 = h1·k⁻¹ + Rx·x·k⁻¹   (mod q)
s2 = h2·k⁻¹ + Rx·x·k⁻¹   (mod q)
```

Subtracting: `s1 - s2 = (h1 - h2)·k⁻¹`, so `k = (h1 - h2)·(s1 - s2)⁻¹`. With `k` recovered, the private key `x = (s1·k - h1)·Rx⁻¹` follows immediately. This is a **Critical** impact: full extraction of the aggregate private signing key from two publicly observable signatures.

For the Robust ECDSA variant the security documentation further notes a novel split-view attack requiring only `2t+2` presigning participants and two signing sessions (`docs/ecdsa/robust_ecdsa/signing.md:156-158`), making the attack surface even broader.

---

### Likelihood Explanation

A malicious coordinator is an explicitly documented threat actor in this library. The coordinator controls which `(msg_hash, tweak, entropy)` tuple is presented to each participant during rerandomization. Because `PresignOutput` is `Clone` and `rerandomize_presign` takes `&PresignOutput`, the coordinator's orchestration layer can trivially call `rerandomize_presign` twice on the same stored presignature with two different `RerandomizationArguments`, then run two signing sessions. Participants have no per-presignature usage counter and no way to detect or refuse the second use. The attack requires no cryptographic break, no leaked keys, and no external dependency — only the ability to orchestrate two signing rounds, which is the coordinator's normal role.

---

### Recommendation

Consume the `PresignOutput` by value in `rerandomize_presign` so that Rust's ownership system enforces single-use at compile time:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // owned, not a reference
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Additionally, remove the `Clone` derive from both `PresignOutput` types. Together these changes make it a compile-time error to reuse a presignature, mirroring the pattern the external report recommends: mark the resource as consumed before any sensitive operation proceeds.

### Proof of Concept

```rust
// Attacker-controlled coordinator orchestration (pseudocode)
let presign_output: PresignOutput = run_presign_protocol(...);

// Session 1: sign message h1 with entropy ρ1
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,                          // reference — not consumed
    &RerandomizationArguments::new(..., h1, big_r, participants, ρ1),
).unwrap();
let sig1 = run_sign(rerand1, h1, ...);        // produces (R, s1)

// Session 2: sign message h2 with entropy ρ2 — SAME presign_output reused
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,                          // still valid — Clone + &ref
    &RerandomizationArguments::new(..., h2, big_r, participants, ρ2),
).unwrap();
let sig2 = run_sign(rerand2, h2, ...);        // produces (R, s2) — same nonce k

// Key recovery from (R, s1, h1) and (R, s2, h2):
// k   = (h1 - h2) * (s1 - s2)^{-1}  mod q
// x   = (s1 * k - h1) * Rx^{-1}     mod q
```

Both calls to `rerandomize_presign` succeed because `presign_output` is never consumed. The two resulting signatures share the same underlying nonce `k` (the rerandomization factor `δ` differs, but `k·δ⁻¹` and `k·δ'⁻¹` are both linear in `k`, and the relationship is recoverable). The private key is then extracted with standard nonce-reuse arithmetic.
