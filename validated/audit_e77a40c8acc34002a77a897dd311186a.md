### Title
Cross-Chain Replay of `DelegateAction` Signatures Due to Missing Chain Identifier in `get_nep461_hash` - (File: `core/primitives/src/action/delegate.rs`)

---

### Summary

`DelegateAction` (NEP-366 meta transactions) and `DelegateActionV2` (NEP-611 gas keys) compute their signing hash without any chain-specific identifier. A `SignedDelegateAction` produced on one NEAR network (mainnet, testnet, or a fork) is cryptographically valid on any other NEAR network where the same account, key, and nonce exist. An unprivileged attacker who observes a broadcast `SignedDelegateAction` can wrap it in a fresh outer `SignedTransaction` on a different chain and cause the user's actions to execute without the user's consent on that chain.

---

### Finding Description

**Root cause — `get_nep461_hash` omits chain identity**

`DelegateAction.get_nep461_hash()` computes:

```rust
let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
hash(&bytes)
``` [1](#0-0) 

`SignableMessage` serialises only a fixed 4-byte NEP discriminant followed by the Borsh-encoded `DelegateAction` body:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,
    pub msg: &'a T,
}
``` [2](#0-1) 

The `DelegateAction` struct contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key` — **no `chain_id`, no genesis hash, no block hash**: [3](#0-2) 

`DelegateActionV2` (via `VersionedDelegateActionPayload.get_nep461_hash`) has the identical omission: [4](#0-3) 

**Verification path confirms no chain check**

`apply_delegate_action` is the sole production execution entry point for both `Action::Delegate` and `Action::DelegateV2`: [5](#0-4) 

Inside `apply_delegate_action`, the checks are:

1. `signed_delegate_action.verify()` — calls `get_nep461_hash()`, no chain binding.
2. `apply_state.block_height > delegate_action.max_block_height()` — a raw block-height ceiling, not chain-specific.
3. `sender_id` match.
4. `validate_delegate_action_key` — nonce and access-key permission checks. [6](#0-5) 

None of these checks bind the inner `DelegateAction` signature to a specific chain.

**Contrast with regular `SignedTransaction`**

A regular `SignedTransaction` carries a `block_hash` field that is verified to be an ancestor on the current chain via `check_transaction_validity_period` / `validity_period_validate_is_ancestor`, returning `InvalidTxError::InvalidChain` on mismatch: [7](#0-6) 

`DelegateAction` has no equivalent field. The outer `SignedTransaction` that carries the `DelegateAction` is chain-bound, but the inner `SignedDelegateAction` bytes are not — an attacker can extract them and re-wrap them in a fresh outer transaction on any chain.

---

### Impact Explanation

An attacker who observes a `SignedDelegateAction` (which is public once broadcast) can replay it on any NEAR network where:

- The same `sender_id` account exists with the same `public_key` registered.
- The access-key nonce on that network is exactly `delegate_action.nonce - 1`.
- The current block height is ≤ `max_block_height`.

Successful replay executes the inner `actions` on the victim's account on the unintended chain. Depending on the actions signed, this can result in:

- **Unauthorized token transfers** — draining the user's balance on the replayed chain.
- **Key additions or deletions** — full account takeover.
- **Arbitrary contract calls** — any state mutation the user authorised for the original chain.

The corrupted state values are: the victim's account balance (decremented), access-key nonce (advanced), and any storage mutations caused by the inner actions — all on a chain the user never intended.

---

### Likelihood Explanation

The conditions are realistic in two concrete scenarios:

1. **Mainnet / testnet key reuse** — many users and dApps use the same account ID and key pair on both networks. A `DelegateAction` signed for mainnet is immediately replayable on testnet if the nonce matches, and vice versa.
2. **Chain fork** — if NEAR undergoes a contentious fork, both forks share the same account state at the fork point, including nonces. Any `SignedDelegateAction` created before the fork is valid on both branches simultaneously.

The attacker needs no special privilege: the `SignedDelegateAction` is transmitted in plain sight inside the outer transaction, and creating a new outer transaction on the target chain requires only a funded account on that chain.

---

### Recommendation

Include a chain-specific identifier in the `DelegateAction` signing payload. The cleanest approach is to add a `chain_id: String` field to `DelegateAction` (and `DelegateActionV2`) and incorporate it into `get_nep461_hash()`:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
    pub chain_id: String,   // <-- add this
}
```

The hash then covers `chain_id`, making the signature non-transferable across networks. Alternatively, the genesis block hash (already available in `GenesisId`) could be embedded in the `SignableMessage` discriminant or as a mandatory prefix in the serialised payload, without a protocol-breaking struct change. [8](#0-7) 

Validation in `apply_delegate_action` must then assert that `delegate_action.chain_id` matches the node's configured chain ID before accepting the signature.

---

### Proof of Concept

1. Alice signs a `DelegateAction` on **mainnet** to transfer 500 NEAR to Bob, with nonce `N`. She hands the `SignedDelegateAction` to a relayer.
2. The relayer wraps it in a mainnet `SignedTransaction` (with a valid mainnet `block_hash`) and broadcasts it. It executes; Alice's mainnet nonce advances to `N`.
3. Alice also has the same account and key on **testnet**, where her nonce is still `N-1` (she has not yet used that key on testnet).
4. An attacker extracts the `SignedDelegateAction` bytes from the mainnet transaction (public data).
5. The attacker creates a new testnet `SignedTransaction` wrapping the identical `SignedDelegateAction`, using a valid testnet `block_hash`.
6. The attacker submits it to testnet. `apply_delegate_action` calls `signed_delegate_action.verify()` — the NEP-461 hash matches because it contains no chain identifier — nonce `N` is valid on testnet, `max_block_height` is not exceeded, and the action executes: 500 testnet NEAR is transferred to Bob without Alice's consent on testnet.

The same attack applies symmetrically in the fork scenario, where both chains share identical state at the fork point.

fn validity_period_validate_is_ancestor(
    base_header: &BlockHeader,
    prev_block_header: &BlockHeader,
    chain_store: &ChainStoreAdapter,
) -> Result<(), InvalidTxError> {
    let base_height = base_header.height();
    let prev_height = prev_block_header.height();
    let base_block_hash = base_header.hash();

    // Base can't be an ancestor of prev if its height is bigger
    if base_height > prev_height {
        return Err(InvalidTxError::InvalidChain);
    }

    // if both are on the canonical chain, comparing height is sufficient
    // we special case this because it is expected that this scenario will happen in most cases.
    if let Ok(base_block_hash_by_height) = chain_store.get_block_hash_by_height(base_height) {
        if &base_block_hash_by_height == base_block_hash {
            if let Ok(prev_hash) = chain_store.get_block_hash_by_height(prev_height) {
                if &prev_hash == prev_block_header.hash() {
                    return Ok(());
                }
            }
        }
    }

    // if the base block height is smaller than `last_final_height` we only need to check
    // whether the base block is the same as the one with that height on the canonical fork.
    // Otherwise we walk back the chain to check whether base block is on the same chain.
    let last_final_height = chain_store
        .get_block_height(prev_block_header.last_final_block())
        .map_err(|_| InvalidTxError::InvalidChain)?;

    if last_final_height >= base_height {
        let base_block_hash_by_height = chain_store
            .get_block_hash_by_height(base_height)
            .map_err(|_| InvalidTxError::InvalidChain)?;
        if &base_block_hash_by_height == base_block_hash {
            Ok(())
        } else {
            Err(InvalidTxError::InvalidChain)
        }
    } else {
        let header =
            get_block_header_on_chain_by_height(chain_store, prev_block_header.hash(), base_height)
                .map_err(|_| InvalidTxError::InvalidChain)?;
        if header.hash() == base_block_hash { Ok(()) } else { Err(InvalidTxError::InvalidChain) }
    }
