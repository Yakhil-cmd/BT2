### Title
Cross-Transaction EVM Refund Counter Leak Allows Undeserved Gas Discount - (`basic_system/src/system_implementation/caches/generic_pubdata_aware_plain_storage.rs`)

---

### Summary

The `evm_refunds_counter` inside `GenericPubdataAwarePlainStorage` is never reset between transactions within a block. SSTORE-earned refunds that exceed the EIP-3529 cap (`gas_used / 5`) in Transaction A silently carry over into Transaction B. Any subsequent transaction can then consume those stale refunds as its own EVM gas refund, paying less gas than it legitimately earned — directly analogous to the Meta-protocol borrower who retains a positive `borrowed` balance after their collateral is wiped and continues to claim mint rewards.

---

### Finding Description

**Root cause — stale refund counter not cleared between transactions**

`GenericPubdataAwarePlainStorage` holds a per-block-persistent `evm_refunds_counter`:

```rust
pub struct GenericPubdataAwarePlainStorage<...> {
    pub(crate) cache: HistoryMap<...>,
    pub(crate) current_tx_id: TransactionId,
    pub(crate) evm_refunds_counter: NonEmptyHistoryCounter<R, SF, M, A>,
    ...
}
``` [1](#0-0) 

The counter is correctly rolled back when a call frame reverts:

```rust
pub fn finish_frame_impl(&mut self, rollback_handle: Option<&StorageSnapshotId>) -> Result<(), InternalError> {
    if let Some(x) = rollback_handle {
        self.evm_refunds_counter.rollback(x.evm_refunds_counter);
        self.cache.rollback(x.cache)
    } else {
        Ok(())
    }
}
``` [2](#0-1) 

But when a transaction **commits** (`rollback_handle = None`), the `else` branch is a no-op — the counter value is preserved in the persistent storage structure and is never zeroed before the next transaction begins. There is no `begin_new_tx` equivalent for `GenericPubdataAwarePlainStorage` that resets `evm_refunds_counter` (contrast with `GenericTransientStorage`, which explicitly calls `self.cache.clear()` on `begin_new_tx`).

**How refunds are consumed**

At the end of every transaction, `compute_gas_refund` reads the counter directly:

```rust
let evm_refund = {
    let full_refund_ergs = system.io.get_refund_counter().ergs();
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
gas_used -= evm_refund;
``` [3](#0-2) 

`get_refund_counter` delegates straight to `evm_refunds_counter.value()`: [4](#0-3) 

Because the counter is never zeroed between transactions, any refund balance left over from Transaction A (i.e., the portion that exceeded `gas_used_A / 5`) is visible to Transaction B and can be consumed up to `gas_used_B / 5`.

**Concrete scenario**

| Step | Detail |
|---|---|
| Tx A | Zeros 1 000 storage slots → earns 4 800 000 gas of refunds |
| Tx A cap | `gas_used_A ≈ 5 000 000` → max refund = 1 000 000 gas; 3 800 000 gas remain in counter |
| Tx B | Any transaction with `gas_used_B = 100 000` → claims `min(3 800 000, 20 000) = 20 000` gas refund it never earned |
| Operator loss | Operator receives `gas_used_B − 20 000` × gas_price instead of `gas_used_B` × gas_price |

This is a direct EVM-semantics deviation: the Yellow Paper resets the refund counter to zero at the start of every transaction.

---

### Impact Explanation

- **Operator/coinbase underpayment**: every transaction in the same block that follows a high-SSTORE transaction can claim up to `gas_used / 5` of unearned gas refunds, reducing the fee transferred to the coinbase.
- **Token conservation violation**: the total fees collected from users in a block are less than the sum of `gas_used × gas_price` for each transaction, breaking the invariant that the treasury/coinbase receives exactly what users paid.
- **Severity**: Medium — requires a prior high-SSTORE transaction in the same block; the per-transaction cap (`gas_used / 5`) limits the per-transaction gain, but the effect compounds across many transactions in a block.

---

### Likelihood Explanation

Any block that contains a transaction zeroing storage slots (common in DeFi interactions, ERC-20 approvals set to zero, contract self-clears, etc.) leaves residual refund balance. The very next transaction in the block — sent by any unprivileged user — automatically benefits. No special privilege, oracle manipulation, or governance access is required. The attacker can also deliberately craft Transaction A to maximise the leftover balance and then immediately submit Transaction B.

---

### Recommendation

Reset `evm_refunds_counter` to zero at the start of each new transaction, mirroring the EVM specification. The natural place is wherever `current_tx_id` is incremented (the transaction-boundary initialisation path in the bootloader or storage model). A minimal fix:

```rust
// In the transaction-start initialisation path:
self.storage_cache.0.evm_refunds_counter.update(R::empty());
```

Alternatively, expose a `begin_new_tx` method on `GenericPubdataAwarePlainStorage` (analogous to `GenericTransientStorage::begin_new_tx`) that resets the counter, and call it from the bootloader's per-transaction setup.

---

### Proof of Concept

1. Deploy a contract `Cleaner` that writes non-zero values to 1 000 storage slots in its constructor.
2. **Transaction A** (attacker): call `Cleaner.clearAll()` which zeros all 1 000 slots.
   - Each slot earns 4 800 gas refund → counter = 4 800 000 gas.
   - `gas_used_A ≈ 5 000 000`; EIP-3529 cap = 1 000 000 gas; counter after Tx A commits = **3 800 000 gas**.
3. **Transaction B** (attacker, same block): send a simple ETH transfer (`gas_used_B = 21 000`).
   - `compute_gas_refund` reads counter = 3 800 000 gas; cap = 21 000 / 5 = 4 200 gas.
   - `gas_used_B` is reduced by 4 200 gas; attacker pays 4 200 × gas_price less than the actual work performed.
4. Repeat Transaction B for every subsequent transaction in the block until the counter is exhausted.

The attacker-controlled entry path is two ordinary L2 transactions; no privileged role is needed. The root cause — `evm_refunds_counter` not being reset — is a necessary step: without it, Transaction B's counter would be zero and no unearned refund would be granted.
