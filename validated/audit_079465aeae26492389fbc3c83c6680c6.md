### Title
Signature Bypass via `skip_stateful_validations` Enables Front-Running of `deploy_account + invoke` UX Pattern — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (the account contract's signature verification) for any invoke transaction with `nonce == 1` when the sender address already appears in the mempool. An unprivileged attacker who observes a pending `deploy_account` transaction can immediately submit an invoke with `nonce = 1` for the same address carrying an arbitrary/invalid signature. Because signature verification is skipped, the gateway accepts the attacker's invalid invoke into the mempool. When the legitimate user subsequently submits their own valid invoke with `nonce = 1`, the mempool rejects it as `DuplicateNonce`. The legitimate user's `deploy_account` executes successfully, but their intended first invoke is permanently blocked for that block cycle.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` — the transaction is the first post-deploy invoke.
3. `account_nonce == Nonce(Felt::ZERO)` — the account is not yet deployed on-chain.
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

**Effect of `skip_validate = true`**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is then entirely skipped: [3](#0-2) 

`__validate__` is the only place where the account contract verifies the transaction signature. Skipping it means the gateway accepts the transaction without any cryptographic proof that the sender authorized it.

**Why `validate_by_mempool` does not save it**

`validate_by_mempool` is called before `skip_stateful_validations`: [4](#0-3) 

`validate_by_mempool` calls `mempool.validate_tx()`: [5](#0-4) 

`validate_tx` only checks for duplicate transaction hashes and duplicate nonces — it does not verify signatures. At the moment the attacker submits their invoke, no other invoke with `nonce = 1` exists for the target address, so both `validate_incoming_tx` and `validate_fee_escalation` pass. [6](#0-5) 

**`account_tx_in_pool_or_recent_block` is too broad**

The check used to decide whether to skip validation is: [7](#0-6) 

It returns `true` if *any* transaction from the address is in the pool or a recent block — not specifically a `deploy_account`. The comment in `skip_stateful_validations` acknowledges this:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

But the attacker's own invalid invoke (nonce=1) is the transaction that "passed validations" — it passed only because `skip_validate` was triggered by the legitimate `deploy_account` already in the pool. The reasoning is circular and the invariant is broken.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions and rejects valid transactions before sequencing.**

1. The gateway accepts an invoke transaction whose signature has never been verified.
2. The legitimate user's valid invoke with the same nonce is rejected as `DuplicateNonce` by `validate_by_mempool` / `validate_fee_escalation`.
3. The legitimate user's `deploy_account` executes successfully, but their intended first invoke is never sequenced.
4. The attacker's invalid invoke will revert during batcher execution (since `__validate__` runs with `validate = true` during actual execution), but by then the legitimate invoke has already been permanently rejected at the gateway.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker only needs to monitor the public mempool for `deploy_account` transactions (no privileged access required).
- The target address is deterministically computable from the `deploy_account` fields, which are visible in the mempool.
- The attacker must submit their invalid invoke before the legitimate user submits theirs — a race condition, but one that is straightforward to win by watching the mempool and immediately reacting.
- The attack is cheap: the attacker pays only the gateway admission cost; the invalid invoke will revert and the attacker loses the fee, but the damage to the victim is done.
- The attack can be repeated for every new `deploy_account` observed in the mempool, making it a systematic DoS against the `deploy_account + invoke` UX pattern.

---

### Recommendation

1. **Narrow the skip condition**: Instead of checking `account_tx_in_pool_or_recent_block` (which matches any transaction type), verify that the account has a *pending `deploy_account` transaction specifically*. Expose a dedicated mempool query such as `has_pending_deploy_account(address)` and use it in `skip_stateful_validations`.

2. **Alternative — run `__validate__` with a lenient state**: Rather than skipping `__validate__` entirely, execute it against a speculative state that includes the pending `deploy_account` (i.e., treat the account as deployed for the purpose of validation). This preserves signature verification while still supporting the UX pattern.

3. **Protect the nonce slot**: Apply the same `validate_no_delayed_declare_front_run`-style guard to the skip-validate invoke slot, so that once a legitimate invoke with `nonce = 1` is admitted for an undeployed account, no replacement is possible without fee escalation.

---

### Proof of Concept

```
1. Alice submits deploy_account for address A (class_hash C, salt S, calldata D).
   → Gateway: nonce=0, account nonce=0, deploy_account validation passes.
   → Mempool: deploy_account for A is stored.

2. Attacker observes A in the mempool.
   Attacker crafts invoke(sender=A, nonce=1, calldata=anything, signature=GARBAGE).

3. Gateway processes attacker's invoke:
   a. Stateless validation: passes (signature length within limit).
   b. validate_state_preconditions: account_nonce=0, tx_nonce=1 ≤ max_allowed_nonce_gap → passes.
   c. validate_by_mempool → mempool.validate_tx():
      - validate_incoming_tx: no duplicate hash, nonce not too old → Ok.
      - validate_fee_escalation: no existing invoke(nonce=1) for A → Ok(None).
      → passes.
   d. skip_stateful_validations:
      - tx.nonce()==1, account_nonce==0 → condition met.
      - account_tx_in_pool_or_recent_block(A) == true (Alice's deploy_account is in pool).
      → returns true.
   e. run_validate_entry_point(skip_validate=true):
      - execution_flags.validate = false → __validate__ NOT called.
      → passes.
   f. mempool_client.add_tx(attacker_invoke) → attacker's GARBAGE-signature invoke is in mempool.

4. Alice submits invoke(sender=A, nonce=1, calldata=transfer, signature=VALID).

5. Gateway processes Alice's invoke:
   a–b. Pass as before.
   c. validate_by_mempool → mempool.validate_tx():
      - validate_fee_escalation: existing invoke(nonce=1) for A found (attacker's).
        fee_escalation disabled → DuplicateNonce error.
      → REJECTED.

6. Batcher executes Alice's deploy_account → success, A is deployed.
7. Batcher executes attacker's invoke → __validate__ runs → GARBAGE signature → revert/reject.
8. Alice's valid invoke was never sequenced. Alice must resubmit and pay again.
```

The attacker can repeat step 2–5 for every new `deploy_account` observed, systematically denying the `deploy_account + invoke` UX pattern to all users.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
