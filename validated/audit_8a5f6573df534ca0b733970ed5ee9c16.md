### Title
Unvalidated Participant Contributions in CKD Coordinator Allow Malicious Participant to Corrupt Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator()`, the coordinator blindly sums `(big_y, big_c)` group elements received from each participant with no zero-knowledge proof or commitment check. A single malicious participant can inject arbitrary group elements, shifting the final `CKDOutput` to an attacker-chosen value and causing every honest party to accept a corrupted confidential key.

---

### Finding Description

The CKD coordinator collects each participant's share of the blinded secret and accumulates them: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each honest participant is supposed to send `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · app_pk))`, computed in `compute_signature_share`: [2](#0-1) 

No proof of correct formation accompanies these values. The coordinator performs no exponent-consistency check, no Pedersen commitment opening, and no Schnorr/DLEQ proof verification before adding each participant's contribution to the running sum.

Compare this with the robust ECDSA presign, which does perform exponent-interpolation consistency checks on every received share before accepting it: [3](#0-2) 

No equivalent guard exists in the CKD coordinator path.

---

### Impact Explanation

The final CKD output `(Y, C)` satisfies:

```
C - app_sk · Y  =  msk · H(pk, app_id)   (honest case)
```

If a malicious participant sends `big_c' = big_c_honest + Δ` for an arbitrary group element `Δ`, the coordinator computes `C_final = C_honest + Δ`, and the requester unmasks:

```
C_final - app_sk · Y_final  =  msk · H(pk, app_id) + Δ
```

The requester receives a wrong confidential key. The attacker can choose `Δ` freely, so they can:
- Make the output completely unusable (random `Δ`).
- Bias the derived key toward a value they control (structured `Δ`).

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session is a reachable, unprivileged attacker. The participant path (`do_ckd_participant`) sends values privately to the coordinator with no authentication of content beyond transport-layer delivery. No threshold of honest parties can detect or correct the manipulation because the coordinator sums contributions without verification. One malicious participant out of `n` is sufficient. [4](#0-3) 

---

### Recommendation

Add a DLEQ (Discrete Log Equality) or Pedersen commitment proof alongside each participant's `(big_y, big_c)` contribution, proving that the same scalar `y_i` was used in both `big_y = y_i · G` and the `y_i · app_pk` term inside `big_c`, and that `big_c` encodes a correctly Lagrange-weighted share of the participant's actual signing key. The coordinator must verify all proofs before accumulating any contribution. This is the standard technique used in verifiable secret sharing and threshold ElGamal encryption.

---

### Proof of Concept

1. Honest participants `P_1 … P_{n-1}` compute and send correct `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant `P_n` instead sends `(norm_big_y_n, norm_big_c_n + Δ)` where `Δ` is an arbitrary non-identity point.
3. The coordinator sums all contributions at lines 53–55 of `protocol.rs` without any proof check.
4. `CKDOutput::new(norm_big_y, norm_big_c + Δ)` is returned as the protocol result.
5. The requester calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk, app_id) + Δ` — a wrong key — with no error or warning. [5](#0-4)
