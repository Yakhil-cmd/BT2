### Title
Signature Validation Bypass via `skip_stateful_validations` Admits Unsigned Invoke Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator unconditionally skips the `__validate__` entry point (the account's signature check) for any Invoke transaction with `nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** account that has **any** transaction in the pool — not specifically a `deploy_account` — an unprivileged attacker who observes a legitimate `deploy_account` for victim address `X` in the mempool can immediately submit an Invoke with `nonce = 1` from `X` carrying an **arbitrary or empty signature**. The gateway admits this transaction to the mempool without ever calling `__validate__`.

---

The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." However, the function does **not** distinguish between a `deploy_account` and any other transaction type. An attacker who has no key material for address `X` can trigger this condition by observing that a legitimate `deploy_account` for `X` is already in the pool.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate tx hash, nonce ordering, and fee escalation — it carries no signature field and performs no cryptographic check:

### Impact Explanation

An attacker can submit an Invoke transaction from a victim address `X` with a completely invalid (or empty) signature and have it admitted to the mempool. This breaks the invariant that every transaction in the mempool has passed account-level signature validation. Concretely:

- The attacker can race Alice's legitimate `deploy_account + invoke(nonce=1)` pair: by submitting an unsigned Invoke with `nonce = 1` from `X` first, the attacker occupies that nonce slot. Alice's legitimate Invoke is then either rejected (`DuplicateNonce`) or must pay a higher fee to escalate past the attacker's transaction.
- The attacker can repeat this for every observed `deploy_account` in the mempool, performing a low-cost DoS on the deploy-account UX flow.
- The batcher wastes execution resources attempting to run the invalid transaction (which will fail at `__validate__` during actual execution).

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The mempool is observable by any network participant. A `deploy_account` transaction is visible as soon as it is submitted. The attacker needs only to craft an Invoke with the victim's address, `nonce = 1`, and any bytes as the signature — all fields that are freely chosen. No privileged access, no key material, and no on-chain state is required. The attack is repeatable at negligible cost.

---

### Recommendation

1. **Verify the transaction type, not just account presence.** The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that specifically checks whether the pooled transaction for that address is a `deploy_account`. `skip_stateful_validations` should call this instead of `account_tx_in_pool_or_recent_block`.

2. **Apply `max_nonce_for_validation_skip` consistently.** The config field `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip` exists and is used in `PyValidator::should_run_stateful_validations`, but `skip_stateful_validations` ignores it entirely. The gateway path should respect the same bound. [7](#0-6)

### Proof of Concept

```
1. Alice submits:
     RpcTransaction::DeployAccount { sender: X, nonce: 0, signature: <valid> }
   → gateway admits it; mempool now has deploy_account for X
   → account_tx_in_pool_or_recent_block(X) == true

2. Attacker submits (no knowledge of X's private key):
     RpcTransaction::Invoke { sender_address: X, nonce: 1,
                              signature: [],          // empty / garbage
                              resource_bounds: <valid non-zero> }

3. Gateway stateless validation:
   - validate_contract_address(X)          → Ok (valid felt address)
   - validate_resource_bounds              → Ok (non-zero bounds)
   - validate_tx_size                      → Ok
   - validate_nonce_data_availability_mode → Ok (L1)
   → passes

4. Gateway stateful validation:
   a. get_nonce_from_state(X)              → Nonce(0)   (X not deployed)
   b. validate_state_preconditions:
        validate_nonce: 0 ≤ 1 ≤ 200       → Ok
   c. validate_by_mempool:
        no duplicate hash, nonce 1 ≥ 0    → Ok
   d. skip_stateful_validations:
        tx.nonce()==1 && account_nonce==0
        account_tx_in_pool_or_recent_block(X) == true
        → returns true  (SKIP __validate__)
   e. run_validate_entry_point(skip=true):
        execution_flags.validate = false
        validate_tx() returns Ok(None) immediately
        → __validate__ never called

5. Transaction forwarded to mempool with invalid signature.
   Mempool::validate_tx checks only hash/nonce/fee — no signature check.
   → Invalid Invoke admitted.

6. Batcher later executes deploy_account(X, nonce=0) → succeeds.
   Batcher executes attacker's Invoke(X, nonce=1) → __validate__ called,
   signature invalid → transaction rejected at execution time.
   Alice's legitimate Invoke(X, nonce=1) was either displaced or must
   fee-escalate past the attacker's slot.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
