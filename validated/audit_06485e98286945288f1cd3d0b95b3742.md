### Title
`MsgTurnBridge` Handler Is a No-Op: Missing Permission Check and Bridge State Mutation Inconsistent with `UpdateTokenMapping` - (File: `x/cronos/keeper/msg_server.go`)

---

### Summary

`MsgTurnBridge` is one of two explicitly permission-gated messages in the Cronos module (per ADR-009), yet its handler unconditionally returns `nil, nil` — performing no permission check and writing no state. Any unprivileged address can submit `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` authorization guard. Simultaneously, the bridge circuit-breaker is permanently non-functional: the bridge can never be disabled regardless of who calls it.

---

### Finding Description

ADR-009 explicitly designates `MsgUpdateTokenMapping` and `MsgTurnBridge` as the two restricted messages requiring the `CanTurnBridge` / `CanChangeTokenMapping` permission bits. The `UpdateTokenMapping` handler correctly enforces this:

```go
// x/cronos/keeper/msg_server.go:68-82
func (k msgServer) UpdateTokenMapping(...) (...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil { ... }
    return &types.MsgUpdateTokenMappingResponse{}, nil
}
```

The `TurnBridge` handler, however, is a stub:

```go
// x/cronos/keeper/msg_server.go:84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Two invariants are simultaneously violated:

1. **Missing permission check** — `HasPermission(ctx, ..., CanTurnBridge)` is never called. Any address can submit `MsgTurnBridge` and receive a `nil` error (success).

2. **Missing state mutation** — No bridge-enabled flag is ever written. A search across all of `x/cronos/**/*.go` for `BridgeEnabled`, `IsBridgeEnabled`, `CanTurnBridge` (in runtime paths), and `bridge_enabled` returns zero hits outside the permission constant definition. The bridge state is never stored or read anywhere.

The `CanTurnBridge` constant is defined but only used in the permissions query response, never in the `TurnBridge` execution path: [2](#0-1) 

The integration test confirms the intended behavior — an unprivileged caller must be rejected and a privileged caller must actually disable bridge operations: [3](#0-2) 

With the current implementation, both the unprivileged and privileged calls silently succeed and change nothing.

---

### Impact Explanation

**High — Bypass of bridge authorization check + permanent inability to halt bridge flows.**

- **Auth bypass**: Any unprivileged user can broadcast `MsgTurnBridge` and receive a success response. The `CanTurnBridge` permission system — which exists specifically to restrict this operation to admin/delegated addresses — is entirely unenforced.

- **Permanent circuit-breaker failure**: The bridge can never be disabled. In an emergency (e.g., a bridge exploit draining funds), the admin/validator has no on-chain mechanism to halt `send_to_evm_chain` / IBC conversion flows. The circuit-breaker advertised in ADR-009 and exposed via CLI (`turn-bridge`) is permanently non-functional.

Both sub-impacts fall within the allowed High scope: bypass of bridge authorization checks, and permanent inability for honest validators to halt bridge/conversion flows.

---

### Likelihood Explanation

The auth bypass is trivially reachable by any unprivileged address submitting a standard Cosmos SDK transaction. The broken circuit-breaker is a latent risk that becomes critical the moment an emergency requires halting the bridge. No special privileges, leaked keys, or cryptographic breaks are required.

---

### Recommendation

Implement `TurnBridge` consistently with `UpdateTokenMapping`:

1. Add the `CanTurnBridge` permission check mirroring the pattern in `UpdateTokenMapping`.
2. Introduce a persistent bridge-enabled flag in the module's KV store (e.g., under a `KeyPrefixBridgeEnabled` key).
3. Read that flag in the EVM hook handlers (e.g., `send_to_evm_chain` processing) and reject bridge operations when the bridge is disabled.
4. Ensure the integration test `test_gravity_turn_bridge` passes end-to-end against the fixed implementation. [4](#0-3) 

---

### Proof of Concept

1. Any address (e.g., `signer2` with no permissions) broadcasts:
   ```
   cronosd tx cronos turn-bridge false --from signer2
   ```
2. The transaction is accepted on-chain and returns code `0` (success) — the `CanTurnBridge` check is never executed.
3. Bridge operations (`send_to_evm_chain`, IBC conversion) continue unaffected because no bridge-enabled state is ever written or read.
4. The legitimate admin subsequently calls `turn-bridge false --from validator`; it also returns code `0` but the bridge remains fully operational — the circuit-breaker is permanently broken. [1](#0-0) [5](#0-4) [6](#0-5)

func (k msgServer) UpdateTokenMapping(goCtx context.Context, msg *types.MsgUpdateTokenMapping) (*types.MsgUpdateTokenMappingResponse, error) {

// HasPermission check if an account has a specific permission. by default cronos admin has all permissions
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```
