### Title
Recreated Account Permanently Deleted in `StateDB.Commit` Due to Missing `IsCreated` Check After Self-Destruct — (`fvm/evm/emulator/state/stateDB.go`)

### Summary

`StateDB.Commit` unconditionally skips the `IsCreated` branch whenever `HasSelfDestructed` is true. Because `DeltaView.CreateAccount` on an existing address internally calls `SelfDestruct` **and** sets `created[addr]` in the same call, the commit loop deletes the account from the base view and then `continue`s past the recreation step. The account is permanently erased; its carried-over balance and new state are never written back.

### Finding Description

**Step 1 — `DeltaView.CreateAccount` produces the contradictory state**

When `CreateAccount(A)` is called on an address that already exists in the base view, the function:

- Calls `d.SelfDestruct(addr)` at delta.go:136, which inserts `A` into `toBeDestructed`.
- Sets `d.recreated[addr]` at delta.go:150.
- Sets `d.created[addr]` at delta.go:160.

After this call, for address `A`: `toBeDestructed[A]` is set **and** `created[A]` is set simultaneously. [1](#0-0) 

**Step 2 — `StateDB.Commit` handles only the pure-destruct case**

The commit loop processes each dirty address with this logic:

```
if HasSelfDestructed(addr):          // true for A
    baseView.DeleteAccount(addr)     // A erased from persistent store
    deleted = true
if deleted:
    continue                         // ← jumps here, never reaches IsCreated
...
if IsCreated(addr):                  // never reached for A
    baseView.CreateAccount(...)      // A never recreated
``` [2](#0-1) 

The `deleted = true` / `continue` path at lines 484–488 is unconditional. There is no check for `IsCreated` before the `continue`. The `recreated` map tracked in `DeltaView` is never consulted by `Commit` at all.

**Step 3 — `Extract` cannot see the account**

`Extract` reads accounts exclusively from the `BaseView` iterator. Because `baseView.DeleteAccount(A)` was called and `baseView.CreateAccount(A)` was never called, `A` is absent from the iterator and absent from the returned `EVMState`. [3](#0-2) 

### Impact Explanation

Any EVM transaction that deploys a contract to an address that already exists in committed state triggers this path. The concrete effects are:

1. The account's carried-over balance (captured at delta.go:146 into `d.balances[addr]`) is computed correctly inside the delta but is never written to the base view — the funds are silently destroyed.
2. The account's new code, nonce, and storage are also never written.
3. Subsequent reads from the base view (including `Extract`, state proofs, and cross-transaction reads) see the address as non-existent. [4](#0-3) 

### Likelihood Explanation

The trigger is a standard EVM CREATE2 deployment to a pre-funded address — a well-known and intentional pattern (e.g., counterfactual contract wallets). An attacker who controls a deployer contract can:

1. Pre-compute `A = keccak256(0xff || deployer || salt || initcode_hash)`.
2. Cause funds to be sent to `A` (or wait for a user to do so).
3. Deploy a contract to `A` via CREATE2 in a new transaction.

`CreateAccount(A)` fires, `Commit` deletes `A` and skips recreation, and the balance at `A` is permanently lost. No privileged access is required beyond the ability to submit an EVM transaction.

### Recommendation

In `StateDB.Commit`, after calling `baseView.DeleteAccount`, check whether the address is also marked as created (i.e., the recreated case) before issuing `continue`. If `IsCreated(addr)` is true, fall through to the `CreateAccount` branch instead of skipping it:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    ...
    deleted = true
}
// Only skip if truly deleted and NOT being recreated in the same tx
if deleted && !db.IsCreated(addr) {
    continue
}
``` [5](#0-4) 

### Proof of Concept

```
1. Commit account A with balance=100 to base view.
2. Reset StateDB (new transaction).
3. Call StateDB.CreateAccount(A).
   → DeltaView: toBeDestructed[A]=100, created[A], recreated[A], balances[A]=100
4. Call StateDB.Commit(true).
   → HasSelfDestructed(A)=true → baseView.DeleteAccount(A), deleted=true
   → if deleted { continue }   ← IsCreated(A) never checked
   → baseView.CreateAccount(A) never called
5. Call Extract(root, baseView).
   → AccountIterator returns no entry for A.
   → EVMState.Accounts does not contain A.
   → Balance of 100 is permanently lost.
```
