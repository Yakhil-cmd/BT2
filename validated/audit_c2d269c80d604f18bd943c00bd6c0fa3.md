Audit Report

## Title
SNS Governance Missing Voting Power Spike Detection Allows Proposal-Time Stake Inflation - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `compute_ballots_for_new_proposal` reads live neuron state at proposal submission time with no historical baseline or spike-detection guard. An attacker who controls the proposal submission can stake additional SNS tokens immediately before calling `make_proposal`, receiving inflated ballot voting power. The NNS governance canister explicitly mitigates this with a daily `SnapshotVotingPowerTask` and a spike-detection fallback in `compute_ballots_for_standard_proposal`; no equivalent mechanism exists anywhere in the SNS governance code.

## Finding Description
`compute_ballots_for_new_proposal` at `rs/sns/governance/src/governance.rs:5226` iterates over `self.proto.neurons`, reads `cached_neuron_stake_e8s` and `dissolve_delay_seconds` live at `now_seconds`, and assigns ballot voting power directly from those values (lines 5255–5279). There is no comparison against any pre-committed historical snapshot.

The NNS path (`rs/nns/governance/src/governance.rs:5497–5532`) first calls `compute_voting_power_snapshot_for_standard_proposal`, then queries `VOTING_POWER_SNAPSHOTS.previous_ballots_if_voting_power_spike_detected`. If the current total potential voting power exceeds 1.5× the historical minimum (defined in `rs/nns/governance/src/governance/voting_power_snapshots.rs:21`), the NNS falls back to the historical per-neuron snapshot for ballot creation. This snapshot is populated by `SnapshotVotingPowerTask` (`rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs:30–57`), which runs every 24 hours.

A grep for `SnapshotVotingPower`, `snapshot_voting_power`, and `VOTING_POWER_SNAPSHOT` across all of `rs/sns/**/*.rs` returns zero matches, confirming the SNS has no equivalent mechanism.

The exploit path for an attacker who is the proposer:
1. Attacker holds neuron `N` with a small stake and a dissolve delay ≥ `neuron_minimum_dissolve_delay_to_vote_seconds`.
2. Attacker acquires additional SNS tokens and calls `claim_or_refresh_neuron`, updating `cached_neuron_stake_e8s` immediately.
3. Attacker calls `make_proposal`. `compute_ballots_for_new_proposal` reads the freshly updated `cached_neuron_stake_e8s` and assigns inflated ballot voting power.
4. Attacker votes with disproportionate weight, causing the proposal to pass or fail contrary to the true token-holder distribution.
5. After the voting period, attacker starts dissolving and recovers tokens after the dissolve delay expires.

No front-running of another user's proposal is required; the attacker simply controls the ordering of their own `claim_or_refresh_neuron` and `make_proposal` calls.

## Impact Explanation
SNS DAOs control treasury funds, canister upgrades, and governance parameter changes. An attacker who can transiently inflate their ballot voting power on a specific proposal can cause it to pass or fail contrary to the true long-term token-holder distribution. This constitutes a concrete governance integrity impact on in-scope SNS framework canisters, matching: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm.**

## Likelihood Explanation
The attack requires no privileged access and no key compromise. The attacker must own (not borrow) a sufficient quantity of SNS tokens and commit them for the dissolve delay period, which is a real economic cost. For SNS DAOs with short minimum dissolve delays and liquid token markets, this cost is feasible for a well-funded attacker. The entry path is two standard unprivileged ingress calls (`claim_or_refresh_neuron` then `make_proposal`) with no technical barriers. Likelihood is **medium**.

## Recommendation
Introduce a periodic voting power snapshot task for SNS governance analogous to `SnapshotVotingPowerTask` in `rs/nns/governance/src/timer_tasks/snapshot_voting_power.rs`. In `compute_ballots_for_new_proposal`, compare the current total potential voting power against the historical snapshot minimum. If a spike is detected (current > 1.5× historical minimum), use the historical per-neuron voting power for ballot creation instead of live state. This is the exact mitigation already deployed for NNS governance.

## Proof of Concept
**Minimal integration test plan (PocketIC or local replica):**

1. Deploy an SNS governance canister with `neuron_minimum_dissolve_delay_to_vote_seconds = 1` (or the SNS minimum).
2. Create neuron `N` with 100 tokens staked and dissolve delay = 1 second.
3. Create neurons `A`, `B`, `C` each with 100 tokens staked (total honest voting power = 300).
4. Transfer 900 additional tokens to neuron `N`'s subaccount and call `claim_or_refresh_neuron`. Assert `cached_neuron_stake_e8s = 1000`.
5. Call `make_proposal` from neuron `N`.
6. Assert that the ballot for `N` has `voting_power` proportional to 1000 tokens (≈ 77% of total 1300).
7. Vote YES from `N`; vote NO from `A`, `B`, `C`.
8. Assert the proposal passes (attacker's 1000 > adoption threshold despite 3:1 honest opposition).
9. Repeat the test with the NNS governance canister to confirm the spike detection prevents the same outcome.
