### Title
Unchecked Index Access on Attacker-Controlled Empty `Vec` in `do_generation_many` Causes Panic, Permanently Aborting Triple Generation â€” (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

A malicious participant can send a `PolynomialCommitmentsMessageMany` message in round 2 of triple generation with empty outer `Vec` fields (`big_e_v: vec![]`, etc.). Because the struct's derived `Deserialize` imposes no length constraint on the outer `Vec`, deserialization succeeds. The code then immediately indexes into the empty vector with `their.big_e_v[i]` using Rust's panicking `[]` operator â€” before any degree check or commitment check is reached â€” causing a panic that aborts triple generation for every honest party.

### Finding Description

`PolynomialCommitmentsMessageMany` is defined with `#[derive(Serialize, Deserialize)]`: [1](#0-0) 

The custom `Deserialize` for `PolynomialCommitment` validates that each individual commitment is non-empty and non-all-identity: [2](#0-1) 

However, this guard only applies to each *element* of the outer `Vec`. An outer `Vec<PolynomialCommitment>` of length zero (`[]`) is perfectly valid MessagePack and deserializes without error. `recv_from_others` performs no further validation: [3](#0-2) 

After deserialization, the code immediately indexes into the received vectors using the panicking `[]` operator: [4](#0-3) 

The degree check and commitment check that would catch a misbehaving participant are located *after* these index accesses: [5](#0-4) 

They are never reached. The panic at line 337 fires first.

### Impact Explanation

Triple generation is the prerequisite for all OT-based ECDSA presigning and signing. A single malicious participant who sends an empty-vector `PolynomialCommitmentsMessageMany` in every session causes a panic in every honest party's async task, aborting the protocol unconditionally. Because the attacker can repeat this in every attempted session, triple generation â€” and therefore ECDSA signing â€” is permanently denied for all honest parties.

### Likelihood Explanation

The attacker only needs to be a registered participant in the triple generation session. No cryptographic material needs to be forged. Constructing and sending a MessagePack-encoded struct with empty arrays is trivial. The attack is deterministic and requires no special timing or environment.

### Recommendation

Before the inner loop at line 335, validate that all six vectors in the received message have exactly `N` elements, returning a `ProtocolError::AssertionFailed` (not panicking) if any length is wrong:

```rust
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(format!(
        "message from {from:?} has wrong number of elements"
    )));
}
```

Replace all bare `[i]` index accesses on attacker-supplied vectors with `.get(i).ok_or(...)` or add the length pre-check above.
