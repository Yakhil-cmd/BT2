### Title
Signature Validation Bypass via `skip_stateful_validations` Admits Unsigned Invoke Transactions to Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (the only place where the account contract verifies the transaction signature) for any invoke transaction whose nonce is `1` and whose sender has nonce `0` on-chain, provided the mempool reports *any* transaction for that address. Because the mempool check is not restricted to deploy-account transactions, an unprivileged attacker who observes a victim's `deploy_account` in the mempool can immediately submit a crafted invoke with an arbitrary (invalid) signature for the victim's address, and the gateway will admit it without ever verifying the signature.

---

### Finding Description

**Broken invariant:** Every transaction admitted to the mempool must carry a signature that passes the account contract's `__validate__` entry point.

**Root cause — `skip_stateful_validations`:**

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-460
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())   // ← only checks existence
                .await
                ...;
        }
    }
    Ok(false)
}
```

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false`:

```rust
// lines 302-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate = false`, `validate_tx` in the blockifier returns `Ok(None)` immediately without calling the account contract:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 999-1001
fn validate_tx(...) -> ... {
    if !self.execution_flags.validate {
        return Ok(None);
    }
```

**The mempool check is too broad:**

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` for *any* transaction type in the pool, not only `deploy_account`. The feature was designed for the UX case where a user submits `deploy_account` (nonce 0) and `invoke` (nonce 1) together, but the guard does not enforce that the pooled transaction is actually a `deploy_account`.

**The mempool's own `validate_tx` does not check signatures:**

```rust
// crates/apollo_mempool/src/mempool.rs  lines 402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;   // nonce/duplicate checks only
    self.validate_fee_escalation(tx_reference)?;                    // fee-escalation only
    Ok(())
}
```

`ValidationArgs` carries `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, `max_l2_gas_price` — no signature field.

**Complete admission path for an invalid transaction:**

1. `StatelessTransactionValidator::validate` — checks signature *length* only (not validity).
2. `validate_state_preconditions` — nonce range and resource-bounds checks; passes for nonce 1 when account nonce is 0.
3. `validate_by_mempool` → `mempool.validate_tx` — nonce/duplicate/fee-escalation only; no signature check.
4. `skip_stateful_validations` — returns `true` because `account_tx_in_pool_or_recent_block` is satisfied by the victim's `deploy_account`.
5. `run_validate_entry_point` — **skipped entirely** (`validate = false`).

The invalid invoke is then forwarded to `mempool.add_tx` and stored.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions.**

An attacker can inject invoke transactions carrying an arbitrary (or empty) signature for any account that has a `deploy_account` pending in the mempool. Concrete consequences:

- **Mempool pollution / DoS:** The attacker can flood the mempool with invalid transactions for every newly-deploying account observed in the public mempool.
- **Victim invoke displacement:** The mempool enforces at most one transaction per `(address, nonce)` pair via fee-escalation. If the attacker's invalid invoke carries a higher tip than the victim's legitimate invoke (nonce 1), the victim's transaction is evicted. The attacker's transaction then fails during batcher execution (the batcher always sets `validate: true` via `AccountTransaction::new_for_sequencing`), but the victim's legitimate invoke is permanently lost from the mempool.
- **Targeted censorship:** A time-sensitive invoke (e.g., a DeFi operation) submitted alongside a `deploy_account` can be silently replaced before it is ever sequenced.

---

### Likelihood Explanation

- `deploy_account` transactions are broadcast publicly over P2P and visible in the mempool.
- The victim's contract address is deterministic from the `deploy_account` fields and can be computed off-chain before the transaction is even mined.
- The attack requires no privileged access, no special contract, and no on-chain funds beyond the gas cost of submitting the crafted invoke.
- The window of vulnerability is the entire time the `deploy_account` sits in the mempool (potentially many blocks).

---

### Recommendation

Replace the broad existence check with a type-specific check. The mempool should expose a method such as `has_deploy_account_in_pool(address)` that returns `true` only when the pooled transaction for that address is a `deploy_account`. The `skip_stateful_validations` function should call this narrower predicate:

```rust
// Proposed fix
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .deploy_account_tx_in_pool(tx.sender_address())   // type-restricted check
        .await
        ...;
}
```

This preserves the intended UX (deploy + invoke submitted together) while closing the path for attackers who exploit the generic `account_tx_in_pool_or_recent_block` check.

### Proof of Concept

```
1. Victim broadcasts:
       deploy_account { class_hash, salt, constructor_calldata, nonce=0, signature=<valid> }
   → Victim's contract address A is deterministic and publicly computable.
   → Mempool now has a tx for address A; account nonce on-chain = 0.

2. Attacker observes the deploy_account in the mempool and computes address A.

3. Attacker crafts:
       invoke { sender_address=A, nonce=1, calldata=<anything>,
                tip=<victim_tip + 1>,   // outbid victim's legitimate invoke
                signature=[]            // empty / invalid }

4. Gateway stateless check: signature length 0 ≤ max_signature_length → passes.

5. Gateway stateful check:
   a. validate_state_preconditions: account_nonce=0, tx_nonce=1, 0 ≤ 1 ≤ max_gap → passes.
   b. validate_by_mempool: nonce/fee checks only → passes.
   c. skip_stateful_validations:
        tx.nonce()==1 ✓  account_nonce==0 ✓
        account_tx_in_pool_or_recent_block(A) == true (victim's deploy_account) ✓
        → returns true (skip)
   d. run_validate_entry_point: validate=false → __validate__ NOT called.

6. Attacker's invalid invoke is admitted to the mempool.

7. If victim also submitted a legitimate invoke (nonce=1, valid signature, lower tip):
   → Fee-escalation evicts victim's invoke; attacker's invalid tx takes its slot.

8. Batcher executes deploy_account (nonce=0) → account A is deployed.
   Batcher executes attacker's invoke (nonce=1, validate=true) → __validate__ fails
   → tx reverts / is rejected.
   Victim's legitimate invoke is gone from the mempool and never executed.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
