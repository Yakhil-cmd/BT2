### Title
Extension Registered and Initialized Without Errors But Never Called Due to Missing Coverage Check in `validateExtensionsConfig` — (`metric-core/contracts/libraries/ValidateExtensionsConfig.sol`)

---

### Summary

`ValidateExtensionsConfig.validateExtensionsConfig` verifies that registered extension addresses are non-zero, non-duplicate, and that at least one `ExtensionOrders` slot is non-zero. However, it never checks that **every** registered extension index appears in at least one order slot. An extension can therefore be registered, pass all factory validation, be initialized via `callExtension`, and yet be permanently unreachable during any pool operation — a direct analog to the NFT-receiver fallback handler that can be installed without error but is never executed.

---

### Finding Description

`createPool` calls `ValidateExtensionsConfig.validateExtensionsConfig` before deploying the pool: [1](#0-0) 

The validation function enforces:

1. `extensions.length ≤ 7`
2. `extensionInitData.length == extensions.length`
3. No zero or duplicate extension addresses
4. `!_ordersAreZero(orders)` — at least one order field is non-zero
5. Each non-zero order field references only valid indices (1 … extensionCount) with no intra-field duplicates [2](#0-1) 

What it **does not** enforce: that every index `1 … extensionCount` appears in at least one of the six order fields.

Concrete example with two extensions:

```
extensions        = [allowlistExt, loggingExt]   // length = 2
orders.beforeSwap = encode(index=2)               // only loggingExt
all other orders  = 0
```

`_validateExtensionOrder(encode(2), 2)` passes — index 2 is valid and non-duplicate. `_ordersAreZero` returns false. The call to `callExtension(allowlistExt, initialize(...))` succeeds and `allowlistExt.boundPool` is set to the new pool. [3](#0-2) 

But `BEFORE_SWAP_ORDER` (and every other order immutable) never encodes index 1, so `_callExtensionsInOrder` never reaches `allowlistExt`: [4](#0-3) 

The extension is permanently silent for the lifetime of the immutable pool.

---

### Impact Explanation

The protocol ships a `SwapAllowlistExtension` as a first-class periphery extension intended to gate who may call `swap()`. If that extension is registered at index 1 but index 1 is absent from `beforeSwap` (or `afterSwap`) order, every swap proceeds without the allowlist check. Any address — including those the pool creator intended to block — can trade freely. Because pool immutables are set at construction and extension orders are immutable, there is no post-deployment remedy short of migrating all liquidity to a new pool.

The same silent-bypass applies to any extension that enforces access control on `addLiquidity` or `removeLiquidity`.

---

### Likelihood Explanation

Pool creation is permissionless. The `ExtensionOrders` encoding (packed 3-bit indices in a `uint256`) is non-obvious; a creator who registers N extensions and sets orders for only M < N of them receives no error or warning. The factory's own documentation and validation give no indication that uncovered extensions are a problem. The likelihood of an honest misconfiguration is medium; the factory's silence makes it indistinguishable from a correct configuration.

---

### Recommendation

After the existing per-field `_validateExtensionOrder` calls, add a coverage check that accumulates the union of all extension indices referenced across all six order fields and asserts it equals `{1, 2, …, extensionCount}`:

```solidity
// After all _validateExtensionOrder calls:
uint256 covered;
covered |= _coveredIndices(orders.beforeAddLiquidity);
covered |= _coveredIndices(orders.afterAddLiquidity);
covered |= _coveredIndices(orders.beforeRemoveLiquidity);
covered |= _coveredIndices(orders.afterRemoveLiquidity);
covered |= _coveredIndices(orders.beforeSwap);
covered |= _coveredIndices(orders.afterSwap);

// Every index 1..extensionCount must appear in at least one order.
for (uint256 i = 1; i <= extensions.length; i++) {
    if (covered & (1 << i) == 0) revert InvalidExtensionsConfig();
}
```

This mirrors the external report's recommendation: check for the custom handler first (ensure coverage) before accepting the configuration as valid.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {ValidateExtensionsConfig} from
    "metric-core/contracts/libraries/ValidateExtensionsConfig.sol";
import {ExtensionOrders} from
    "metric-core/contracts/types/PoolExtensionsConfig.sol";

contract PoC {
    function demonstrate() external pure {
        address[] memory exts = new address[](2);
        exts[0] = address(0x1111); // intended allowlist guard — index 1
        exts[1] = address(0x2222); // logging extension   — index 2

        bytes[] memory initData = new bytes[](2);

        ExtensionOrders memory orders;
        // Only index 2 (logging) is wired into beforeSwap.
        // Index 1 (allowlist guard) is absent from every order.
        orders.beforeSwap = 2; // 3-bit encoding: single entry = 2

        // Passes without revert — allowlist guard is silently uncovered.
        ValidateExtensionsConfig.validateExtensionsConfig(exts, orders, initData);
    }
}
```

After `createPool` completes, `allowlistExt.initialize` has been called and `boundPool` is set, yet `_callExtensionsInOrder(BEFORE_SWAP_ORDER, ...)` never invokes it. Any caller can `swap()` freely, bypassing the intended guard. [5](#0-4) [6](#0-5)

```text
  function validateExtensionsConfig(
    address[] calldata extensions,
    ExtensionOrders calldata orders,
    bytes[] calldata extensionInitData
  ) internal pure {
    if (extensions.length > MAX_EXTENSIONS) revert InvalidExtensionsConfig();
    if (extensionInitData.length != extensions.length) revert InvalidExtensionsConfig();

    if (extensions.length == 0) {
      if (!_ordersAreZero(orders)) revert InvalidExtensionsConfig();
      return;
    }

    for (uint256 i = 0; i < extensions.length; i++) {
      if (extensions[i] == address(0)) revert InvalidExtensionsConfig();
      for (uint256 j = 0; j < i; j++) {
        if (extensions[i] == extensions[j]) revert InvalidExtensionsConfig();
      }
    }

    if (_ordersAreZero(orders)) revert InvalidExtensionsConfig();

    _validateExtensionOrder(orders.beforeAddLiquidity, extensions.length);
    _validateExtensionOrder(orders.afterAddLiquidity, extensions.length);
    _validateExtensionOrder(orders.beforeRemoveLiquidity, extensions.length);
    _validateExtensionOrder(orders.afterRemoveLiquidity, extensions.length);
    _validateExtensionOrder(orders.beforeSwap, extensions.length);
    _validateExtensionOrder(orders.afterSwap, extensions.length);
  }
```

```text
  function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;

    while (true) {
      uint256 extensionIndex = order & 0x7;
      if (extensionIndex == 0) break;
      address extension = _extensionAddress(extensionIndex);
      if (extension == address(0)) revert PanicEmptyExtension();
      CallExtension.callExtension(extension, data);
      order >>= 3;
    }
  }
```
