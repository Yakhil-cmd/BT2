### Title
Unprotected `new` Initialization Allows Any Caller to Seize Engine Ownership - (File: engine/src/contract_methods/admin.rs)

### Summary
The Aurora Engine's `new` initialization function performs no caller verification. Any NEAR account can call it before the legitimate deployer, supplying an arbitrary `owner_id` and `key_manager`, thereby gaining full administrative control over the engine contract — including the ability to upgrade it with arbitrary bytecode.

### Finding Description

The `new` function in `engine/src/contract_methods/admin.rs` is the sole initialization entry point for the Aurora Engine. It sets the entire `EngineState`, including the `owner_id` (the account authorized to upgrade the contract and perform all privileged operations) and the `key_manager` (the account that manages relayer function-call access keys). [1](#0-0) 

The only guard present is a re-initialization check:

```rust
if state::get_state(&io).is_ok() {
    return Err(b"ERR_ALREADY_INITIALIZED".into());
}
```

There is **no check on `env.predecessor_account_id()`**. The function accepts `NewCallArgs` from any caller and writes the supplied `owner_id` and `key_manager` directly into persistent state: [2](#0-1) 

The `owner_id` field is the gating identity for every privileged operation — `stage_upgrade`, `upgrade`, `set_owner`, `set_upgrade_delay_blocks`, etc.: [3](#0-2) [4](#0-3) 

The public WASM entrypoint is: [5](#0-4) 

On NEAR Protocol, contract deployment and initialization are **separate transactions** unless the deployer explicitly batches them. If the deployer sends the deploy action and the `new()` call in separate transactions, a window exists in which any NEAR account can call `new()` first with attacker-controlled arguments.

The `NewCallArgs` variants all accept a caller-supplied `owner_id`: [6](#0-5) 

### Impact Explanation

If an attacker initializes the contract before the deployer:

1. They set `owner_id` to their own NEAR account.
2. They immediately call `upgrade()` (which requires only `require_owner_only`) to deploy arbitrary WASM bytecode to the Aurora Engine account.
3. The malicious contract can drain all ETH and ERC-20 token balances held by the engine (bridged assets from the ETH connector), constituting **direct theft of user funds** or **permanent freezing of funds**.

**Impact: Critical** — direct theft of all bridged assets at rest.

### Likelihood Explanation

On NEAR, deployment and initialization are separate unless explicitly batched. If the deployer uses two separate transactions (deploy, then call `new`), the window is exploitable by any NEAR account that monitors the mempool or chain state. Even with batching, the absence of a caller check is a latent vulnerability that could be triggered by any future deployment or redeployment scenario (e.g., after a contract wipe). **Likelihood: Medium.**

### Recommendation

Add a check that `env.predecessor_account_id() == env.current_account_id()` (i.e., only a self-call or the contract account itself may initialize), or verify against the NEAR contract's upgrade authority. The NEAR SDK pattern is:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Add: only the contract account itself (i.e., called in a batch with deploy) may initialize
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    // ... rest of initialization
}
```

This ensures `new()` can only be called as part of a deploy batch (where `predecessor == current_account`), eliminating the front-running window.

### Proof of Concept

1. Deployer sends transaction A: deploy Aurora Engine WASM to account `aurora.near`.
2. Before deployer sends transaction B (`new()`), attacker sends their own call to `aurora.near::new()` with `owner_id = "attacker.near"`.
3. Attacker's transaction is included first (NEAR has no strict ordering guarantee between independent senders).
4. `state::get_state` returns `Err` (no state yet), so the guard passes.
5. `EngineState { owner_id: "attacker.near", ... }` is written to storage.
6. Deployer's `new()` call now fails with `ERR_ALREADY_INITIALIZED`.
7. Attacker calls `upgrade()` from `attacker.near`, deploying a malicious contract that transfers all bridged ETH balances to the attacker. [7](#0-6)

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
