### Title
Unregistered Pool Address Bypasses `onlyPoolAdmin` Guard, Enabling Attacker to Hijack Pool Admin Role — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

The `onlyPoolAdmin` modifier in `MetricOmmPoolFactory` silently passes for **any non-zero caller** when the target `pool` address has never been registered, because `poolAdmin[unregisteredAddress]` returns `address(0)` and the check `msg.sender != address(0)` is always true. An attacker can exploit this to pre-stage a pending admin transfer on a future pool's deterministic CREATE2 address before the pool is created, then call `acceptPoolAdmin` immediately after creation to seize the admin role.

---

### Finding Description

The `_checkPoolAdmin` function is: [1](#0-0) 

```solidity
function _checkPoolAdmin(address pool) private view {
    if (msg.sender != poolAdmin[pool]) revert NotPoolAdmin();
}
```

When `pool` is not yet registered, `poolAdmin[pool]` is `address(0)`. The condition becomes `msg.sender != address(0)`, which is **always true** for any real caller. Every function gated by `onlyPoolAdmin` — including `proposePoolAdminTransfer` — is therefore callable by anyone on an unregistered address. [2](#0-1) 

```solidity
function proposePoolAdminTransfer(address pool, address newAdmin) external override nonReentrant onlyPoolAdmin(pool) {
    if (newAdmin == address(0)) revert InvalidAdmin();
    if (newAdmin == poolAdmin[pool]) revert InvalidAdmin(); // poolAdmin[pool] == address(0), so newAdmin != address(0) passes
    pendingPoolAdmin[pool] = newAdmin;
    emit PoolAdminTransferProposed(pool, poolAdmin[pool], newAdmin);
}
```

`acceptPoolAdmin` has no check that the pool is registered or that `poolAdmin[pool]` is non-zero: [3](#0-2) 

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;   // overwrites the legitimate admin set at creation
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

Pool addresses are deterministic via CREATE2 in `MetricOmmPoolDeployer.deploy`: [4](#0-3) 

The salt and all constructor arguments are visible in the `createPool` calldata, so the CREATE2 address is computable from the mempool.

---

### Impact Explanation

Once the attacker seizes the admin role they can:

1. Call `setPoolAdminFeeDestination(pool, attacker)` — redirects all future admin fee sweeps to themselves.
2. Call `setPoolAdminFees(pool, maxAdminSpreadFeeE6, maxAdminNotionalFeeE8)` — this first collects accrued fees to the now-attacker-controlled destination, then raises fees to the factory cap (up to 20% spread / 1% notional), harming traders and LPs.
3. Call `pausePool(pool)` — halts all swaps and liquidity operations.
4. For pools with a mutable oracle, call `proposePoolPriceProvider` after the timelock to rotate to a malicious price feed, enabling bad-price execution. [5](#0-4) 

---

### Likelihood Explanation

- `createPool` is permissionless; any caller can create a pool.
- The CREATE2 address is fully computable from public mempool data (deployer address, salt, constructor args).
- Front-running a single `createPool` transaction is sufficient; no special privileges are required.
- The legitimate admin has no on-chain way to prevent the attack once the pending transfer is staged, because `cancelPoolAdminTransfer` can itself be front-run.

---

### Recommendation

Add a registered-pool guard to `_checkPoolAdmin` (or to `proposePoolAdminTransfer` / `acceptPoolAdmin` individually):

```solidity
function _checkPoolAdmin(address pool) private view {
    if (poolToIdx[pool] == 0) revert NotARegisteredPool();   // <-- add this
    if (msg.sender != poolAdmin[pool]) revert NotPoolAdmin();
}
```

Alternatively, add the same guard to `acceptPoolAdmin` so that a pending transfer staged before pool creation can never be finalised:

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    if (poolToIdx[pool] == 0) revert NotARegisteredPool();
    ...
}
``` [6](#0-5) 

---

### Proof of Concept

1. Attacker watches the mempool for a `createPool(params)` transaction.
2. Attacker computes the CREATE2 address `futurePool` from `params.salt` and all constructor arguments visible in the calldata.
3. Attacker front-runs with:
   ```solidity
   factory.proposePoolAdminTransfer(futurePool, attacker);
   // passes: poolAdmin[futurePool] == address(0), so onlyPoolAdmin allows any caller
   // sets: pendingPoolAdmin[futurePool] = attacker
   ```
4. The legitimate `createPool` transaction lands; `poolAdmin[futurePool]` is set to `params.admin`.
5. Attacker immediately calls:
   ```solidity
   factory.acceptPoolAdmin(futurePool);
   // pending == attacker (non-zero), msg.sender == attacker → passes
   // poolAdmin[futurePool] = attacker  (overwrites legitimate admin)
   ```
6. Attacker calls `setPoolAdminFeeDestination(futurePool, attacker)` then `setPoolAdminFees(futurePool, maxFee, maxFee)` — accrued admin fees are swept to the attacker and future fees are maximised.
