### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check for Nonce-1 Invokes from Already-Deployed Accounts with Any Pending Pool Transaction — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path is intended to skip the `__validate__` entry-point call (signature verification) only when a `deploy_account` transaction is pending in the mempool for a not-yet-deployed account. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from that address in the pool — not specifically a `deploy_account`. This allows an attacker who controls an already-deployed account (on-chain nonce = 0) to submit a nonce-1 invoke with an **arbitrary/invalid signature** that bypasses gateway signature verification and is admitted to the mempool.

---

### Finding Description

The relevant code path in `crates/apollo_gateway/src/stateful_transaction_validator.rs`:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce + resource bounds)
       ├─ validate_by_mempool            (mempool duplicate/fee-escalation check)
       └─ skip_stateful_validations      ← broken guard here
            └─ run_validate_entry_point(skip_validate=true)  ← __validate__ skipped
```

`skip_stateful_validations` fires when **all three** conditions hold:

1. The transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)`
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

Condition 3 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction from that address currently in the pool — including a nonce-0 invoke from an already-deployed account. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but a nonce-0 invoke (equal to the account nonce, not a future nonce) satisfies the pool check without implying a `deploy_account` exists.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`:

```rust
let execution_flags = ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

This causes `StatefulValidator::perform_validations` to return `Ok(())` without calling `tx.validate_tx(...)` (the `__validate__` entry point): [4](#0-3) 

The transaction is then forwarded to the mempool via `mempool_client.add_tx(...)` without any signature verification having occurred. [5](#0-4) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker who controls an already-deployed account (on-chain nonce = 0) can:

1. Submit a valid nonce-0 invoke (passes all checks including `__validate__`), placing it in the pool.
2. Submit a nonce-1 invoke with a **completely arbitrary/invalid signature**.
3. The gateway's `skip_stateful_validations` fires: nonce=1, account_nonce=0, pool contains the nonce-0 tx → `skip_validate = true`.
4. The nonce-1 invoke bypasses `__validate__` and is admitted to the mempool.

The broken invariant: *every invoke transaction admitted to the mempool must have passed `__validate__` (signature verification) at the gateway, unless the account does not yet exist on-chain (the legitimate deploy_account + invoke UX case).*

The corrupted admission decision is the mempool accepting a transaction whose signature was never verified. This enables mempool pollution with signature-invalid transactions from any deployed account, and can be used to fill the mempool with transactions that will revert at execution time, degrading sequencer throughput.

---

### Likelihood Explanation

**Likelihood: Medium.** The attacker must control a deployed account (nonce=0 on-chain) and must have a pending transaction from that account in the mempool. Both conditions are easily arranged by any user. No privileged access is required. The attack is repeatable as long as the attacker keeps a valid transaction in the pool.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that only returns `true` when a `DeployAccount` transaction (not any transaction) is present in the pool for that address.

Alternatively, restrict the skip condition to cases where the account contract does not yet exist on-chain (i.e., `get_class_hash_at(sender_address)` returns the zero class hash), which is the true precondition for why `__validate__` cannot run. [6](#0-5) 

---

### Proof of Concept

**Setup:**
- Account `X` is deployed on-chain; its nonce is `0`.
- Attacker controls `X` and knows its valid signing key.

**Steps:**

1. **Submit valid nonce-0 invoke from X** (any calldata, valid signature). This passes all gateway checks including `__validate__` and lands in the mempool pool.

2. **Submit nonce-1 invoke from X with invalid signature** (e.g., `signature = [Felt::from(0xdeadbeef)]`).

3. **Gateway stateless validation** (`StatelessTransactionValidator::validate`): passes — signature length is within bounds, resource bounds are non-zero, nonce DA mode is L1. [7](#0-6) 

4. **Gateway stateful validation** (`extract_state_nonce_and_run_validations`):
   - `account_nonce = get_nonce_from_state(X) = Nonce(0)` ✓
   - `validate_nonce`: `0 <= 1 <= 200` ✓
   - `validate_by_mempool`: no duplicate, fee escalation ok ✓
   - `skip_stateful_validations`: `tx.nonce() == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(X) == true` (nonce-0 invoke is in pool) ✓ → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: `validate = false` → `__validate__` **not called** → returns `Ok(())` [8](#0-7) 

5. **Nonce-1 invoke with invalid signature is admitted to the mempool** — signature was never verified.

6. **At execution time** (batcher): `__validate__` runs with `validate=true`, fails with `ValidateFailure`, transaction reverts. But the mempool admission invariant was already violated.

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
