### Title
Missing Validation of Participant Contributions in CKD Coordinator Allows Malicious Participant to Corrupt Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` receives `(norm_big_y, norm_big_c)` contributions from each participant and immediately accumulates them into the final output without any verification. This violates the Checks-Effects-Interactions pattern: the interaction (receiving messages) and effect (mutating accumulated state) occur with no preceding check on the validity of the received group elements. A malicious participant can send arbitrary curve points, causing the coordinator to output a corrupted `CKDOutput` that does not correspond to the actual master secret key.

---

### Finding Description

In `do_ckd_coordinator` at lines 50–55 of `src/confidential_key_derivation/protocol.rs`, the coordinator iterates over contributions received from all other participants and unconditionally adds each `big_y` and `big_c` into the running accumulators:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is no proof of knowledge, no consistency check, and no range check on the received `(norm_big_y, norm_big_c)` pair before it is folded into the output. The protocol specification for each participant (in `compute_signature_share`, lines 148–182) requires:

- `norm_big_y_i = lambda_i * y_i * G` — a Lagrange-weighted commitment to a fresh random scalar
- `norm_big_c_i = lambda_i * (x_i * H(pk || app_id) + y_i * app_pk)` — the masked share of the BLS signature

Neither of these properties is verified by the coordinator. Any participant can substitute arbitrary `G1` elements for their contribution.

This is the direct analog of the CEI violation in the referenced report: the **interaction** (receiving the message) and the **effect** (mutating `norm_big_y` / `norm_big_c`) happen without any preceding **check**, exactly mirroring the pattern where a token transfer occurs before state is updated.

The non-coordinator path (`do_ckd_participant`, lines 17–33) is equally unchecked: it computes and sends `(norm_big_y, norm_big_c)` with no commitment or binding to the public parameters that the coordinator could later verify.

---

### Impact Explanation

The CKD protocol is designed to produce a coordinator output `(Y, C)` such that the app can unmask the confidential key as:

```
confidential_key = C - app_sk * Y = msk * H(pk || app_id)
```

If a malicious participant M sends `(big_y_m', big_c_m')` instead of their correct Lagrange-weighted contribution, the coordinator computes:

```
Y'  = Σ_{i≠m}(λ_i · y_i · G) + big_y_m'
C'  = Σ_{i≠m}(λ_i · big_c_i) + big_c_m'
```

The app then unmasks:

```
C' - app_sk · Y'
  = Σ_{i≠m}(λ_i · x_i) · H(pk||app_id)
    + (big_c_m' - app_sk · big_y_m')
```

Because M can freely choose `big_y_m'` and `big_c_m'`, the term `(big_c_m' - app_sk · big_y_m')` is an arbitrary group element under M's control (M does not need to know `app_sk` to set this term; they simply pick any two points). Setting both to the identity causes the output to be missing M's share of the master secret, producing a key that is cryptographically incorrect and unusable. Setting them to non-trivial values injects an attacker-controlled additive offset into the derived key.

**Mapped impact**: High — Corruption of CKD outputs so honest parties accept an unusable cryptographic output.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker role is an unprivileged protocol participant — no special privilege, no key leakage, and no external assumption is required. The attack requires only that the participant deviate from the protocol when sending their private message to the coordinator. There is no detection mechanism: the coordinator has no way to distinguish a correct contribution from a malicious one.

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that:

1. `norm_big_y = lambda_i * y * G` for some scalar `y` (proof of discrete log)
2. `norm_big_c = lambda_i * (x_i * H(pk||app_id) + y * app_pk)` is consistent with the participant's public verification share and the same `y`

This can be implemented as a Chaum-Pedersen proof binding `norm_big_y` and `norm_big_c` to the participant's known public share commitment. The coordinator must verify all proofs before accumulating any contribution (Checks → Effects → Interactions order).

Alternatively, restructure the protocol so that each participant broadcasts their contribution publicly, allowing all parties to verify consistency before the coordinator finalises the output.

---

### Proof of Concept

1. Honest participants P1, P2, P3 run the CKD protocol. P3 is malicious.
2. P1 and P2 correctly compute and send `(norm_big_y_1, norm_big_c_1)` and `(norm_big_y_2, norm_big_c_2)` to the coordinator.
3. P3 sends `(G1::identity(), G1::identity())` — the identity element for both fields — to the coordinator instead of their correct contribution.
4. The coordinator at lines 50–55 accumulates without any check:
   ```rust
   norm_big_y += G1::identity();  // no-op, P3's share silently dropped
   norm_big_c += G1::identity();  // no-op, P3's share silently dropped
   ```
5. The coordinator outputs `CKDOutput::new(norm_big_y, norm_big_c)` at line 56, which encodes only the contributions of P1 and P2.
6. The app calls `ckd_output.unmask(app_sk)` and obtains `(λ_1·x_1 + λ_2·x_2)·H(pk||app_id)` instead of `msk·H(pk||app_id)`.
7. The derived confidential key is incorrect; any downstream use (e.g., TEE decryption, key agreement) silently fails or produces wrong results with no error surfaced to honest parties. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
