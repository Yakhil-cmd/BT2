### Title
Global `last_reward_block` Allows Any Caller to Permanently Block Consensus Reward Distribution â€” (File: `src/staking/staking.cairo`)

---

### Summary
The `update_rewards` function uses a single global `last_reward_block` storage variable shared across all stakers, with no on-chain access control. Any unprivileged caller can invoke `update_rewards` with `disable_rewards: true` to consume the block's reward slot without distributing any rewards, permanently preventing the sequencer from distributing consensus block rewards to any staker.

---

### Finding Description

The `update_rewards` function in `src/staking/staking.cairo` enforces a per-block uniqueness constraint using a single global `last_reward_block` variable:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();  // only checks pause state
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    // ... distribute rewards
``` [1](#0-0) 

`last_reward_block` is a single global slot in storage, not keyed per staker: [2](#0-1) 

`general_prerequisites()` is called identically in every user-facing function (`stake`, `claim_rewards`, `unstake_intent`, etc.), confirming it only checks the pause flag â€” it carries no role-based access control. The spec annotation "Only starkware sequencer" is not enforced on-chain.

The critical ordering is:
1. `last_reward_block` is written **before** the `disable_rewards` branch.
2. Calling with `disable_rewards: true` still writes `last_reward_block` to the current block, then returns immediately without distributing rewards.

This is the direct analog to the Squads ephemeral-key reuse bug: a **shared seed/state** (`last_reward_block`) that should be unique per operation (per staker, per block) is instead global, causing any subsequent legitimate call in the same block to collide and fail with `REWARDS_ALREADY_UPDATED`.

---

### Impact Explanation

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block:
- Writes `last_reward_block` to the current block without distributing any rewards.
- Causes the sequencer's legitimate `update_rewards(..., disable_rewards: false)` call in the same block to revert.
- Permanently freezes all consensus block rewards for all stakers as long as the attack is sustained.

This matches: **High â€” Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- No special privilege is required; any Starknet address can call `update_rewards`.
- The attacker only needs to supply any currently active staker address (publicly readable from the `stakers` vector or events).
- The cost is one transaction per block. On Starknet, transaction fees are low, making sustained griefing economically feasible.
- The attack is entirely permissionless and requires no coordination or leaked keys.

---

### Recommendation

**Short Term**: Add an on-chain access control check to `update_rewards` restricting callers to a designated sequencer/operator role (e.g., `self.roles.only_operator_contract()`), consistent with the spec's stated intent of "Only starkware sequencer."

**Long Term**: Consider making `last_reward_block` per-staker (keyed by `staker_address`) so that a call for one staker cannot block reward distribution for another staker in the same block.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker observes any valid active staker address `S` from on-chain state.
3. In block `N`, attacker calls `update_rewards(S, disable_rewards: true)`.
   - Passes `REWARDS_ALREADY_UPDATED` check (N > last stored block).
   - Passes staker-active check.
   - Writes `last_reward_block = N`.
   - Returns immediately â€” no rewards distributed.
4. Sequencer attempts `update_rewards(T, disable_rewards: false)` in block `N`.
   - Fails: `current_block_number (N) > last_reward_block (N)` is `false` â†’ `REWARDS_ALREADY_UPDATED`.
5. No staker receives block rewards in block `N`.
6. Attacker repeats step 3 in every subsequent block â†’ all stakers permanently frozen out of consensus rewards. [3](#0-2)
