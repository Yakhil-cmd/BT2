### Title
Malicious Participant Can Corrupt CKD Output via Unverified Group Elements вҖ” (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
In the Confidential Key Derivation (CKD) protocol, the coordinator receives `CKDOutput` values from each participant and blindly accumulates them without any cryptographic verification. A malicious participant can send arbitrary group elements in place of their correctly computed share, corrupting the final derived key accepted by honest parties. No proof of correct computation is required or checked.

### Finding Description

**Root Cause**

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35вҖ“57), the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and adds them directly to a running sum: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no zero-knowledge proof, commitment binding, or consistency check verifying that the received `(big_y, big_c)` pair was computed correctly from the participant's secret share. The deserialized values are used as-is.

**Protocol Specification**

Each honest participant is supposed to compute:
- `y вҶҗ в„Өq*`, `Y вҶҗ yВ·G`
- `S вҶҗ x_i В· H(pk вҲҘ app_id)` (using their private share `x_i`)
- `C вҶҗ S + yВ·A` (where `A` is the application public key)
- `norm_big_y = О»_i В· Y`, `norm_big_c = О»_i В· C` [2](#0-1) 

The coordinator sums all `norm_big_y` and `norm_big_c` values to produce the final `CKDOutput`. The `unmask(app_sk)` operation then recovers `msk В· H(pk вҲҘ app_id)` by computing `ОЈ norm_big_c вҲ’ app_sk В· ОЈ norm_big_y`.

**Attack Path**

A malicious participant substitutes arbitrary group elements `(big_y', big_c')` for their correct contribution. The coordinator adds these without verification:

```
final_big_c = (honest_sum_c) + big_c'
final_big_y = (honest_sum_y) + big_y'
```

The `unmask` result becomes:
```
(msk вҲ’ О»_iВ·x_i)В·H(pk вҲҘ app_id) + big_c' вҲ’ app_skВ·big_y'
```

This is not equal to `msk В· H(pk вҲҘ app_id)` for arbitrary `(big_y', big_c')`, producing a corrupted derived key. The malicious participant does not need to know `app_sk` to corrupt the output вҖ” sending identity elements or random group elements is sufficient to make the output unusable.

**Analog to External Report**

This is structurally identical to the `fd_store` vulnerability: data from an untrusted source (`fd_shred` / malicious participant) is received and used downstream (`fd_ext_blockstore_insert_shreds` / `CKDOutput::unmask`) after only partial or absent validation. In both cases, the receiving component trusts the structure of the incoming data without verifying its cryptographic correctness. [3](#0-2) 

### Impact Explanation

A single malicious participant can corrupt the CKD output accepted by the coordinator and all honest parties. The resulting `CKDOutput` does not encode `msk В· H(pk вҲҘ app_id)`, so any downstream use of the derived confidential key (e.g., TEE key derivation) produces an incorrect or attacker-influenced result. This matches the **High** impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

### Likelihood Explanation

Any participant in the CKD protocol can trigger this with no special privileges. The attacker only needs to be a valid protocol participant вҖ” the minimum trust assumption for the CKD protocol. The attack requires sending a single malformed message and is trivially reproducible.

### Recommendation

Require each participant to attach a zero-knowledge proof of correct computation alongside their `(norm_big_y, norm_big_c)` contribution вҖ” specifically, a proof that `norm_big_c вҲ’ norm_big_yВ·(app_sk)` encodes a correctly Lagrange-weighted secret share (a discrete-log equality proof suffices). The coordinator must verify all proofs before accumulating values. Alternatively, use a commit-then-reveal scheme so participants are bound to their inputs before the coordinator aggregates them.
