### Title
Missing Session-Unique Binding in Triple Generation Transcript Enables Full Cross-Session Proof Replay — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

`create_transcript` initializes the Fiat-Shamir transcript with only the participant list and threshold. The only session-specific data added later is the **confirmation** (a hash of all participants' commitments). Because there is no random session ID or nonce in the transcript, a malicious coordinator who can replay Round 1 commitment messages from a prior session can force the transcript state to be identical across two sessions, making every dlog and dlogeq proof from session 1 verify in session 2.

---

### Finding Description

`create_transcript` absorbs exactly three values into the transcript: [1](#0-0) 

```
NEAR_TRIPLE_GENERATION_LABEL | NAME | participants | threshold
```

No session ID, no per-run random nonce. The only session-specific material is the confirmation, added at: [2](#0-1) 

```rust
let my_confirmation = hash(c)?;          // c = ParticipantMap<Commitment>
transcript.message(b"confirmation", &enc_confirmations);
```

The confirmation is `H(Com_1, …, Com_N)` — a deterministic function of the Round 1 commitments. A malicious coordinator who replays the Round 1 commitment messages from session 1 into session 2 causes every honest participant to compute the **same confirmation**, producing an **identical transcript state** before any proof is generated.

All three proof families are then derived from forks of this transcript: [3](#0-2) 

```rust
dlog::prove_with_nonce(&mut transcript.fork(b"dlog0", &me.bytes()), …)
dlog::prove_with_nonce(&mut transcript.fork(b"dlog1", &me.bytes()), …)
``` [4](#0-3) 

```rust
dlogeq::prove_with_nonce(&mut transcript.fork(b"dlogeq0", &me.bytes()), …)
``` [5](#0-4) 

```rust
dlog::prove_with_nonce(&mut transcript.fork(b"dlog2", &me.bytes()), …)
```

Because the transcript state is identical, the Fiat-Shamir challenge `e` derived in `prove_with_nonce` / `verify` is identical: [6](#0-5) 

```rust
let mut rng = transcript.challenge_then_build_rng(NEAR_DLOG_CHALLENGE_LABEL);
let e = frost_core::random_nonzero::<C, _>(&mut rng);
```

A proof `(e, s)` from session 1 will therefore pass `verify` in session 2 for the same statement, because the reconstructed `big_k = s·G − e·X` is fed into the same transcript state and produces the same challenge `e`.

**Contrast with DKG**, which explicitly generates and broadcasts a 32-byte per-participant session ID before any proof material is produced: [7](#0-6) 

```rust
let mut my_session_id = [0u8; 32];
rng.fill_bytes(&mut my_session_id);
let session_ids = do_broadcast(&mut chan, &participants, me, my_session_id).await?;
```

Triple generation has no equivalent protection.

---

### Impact Explanation

A successful replay forces all honest participants to accept the **same triple material** (same `(A, B, C)` and same secret shares) in two distinct signing sessions. In ECDSA, reusing the same nonce triple with two different messages allows any observer to solve for the private signing key algebraically. This is a **Critical** impact: unauthorized reconstruction of the private signing key from two signatures produced with the same triple.

Additionally, the confirmation is passed as the session ID to `multiplication_many`: [8](#0-7) 

```rust
multiplication_many::<N>(
    comms.clone(),
    my_confirmations.clone(),   // ← used as OT session ID
    …
)
```

Replaying the same confirmation also replays the OT session ID into the multiplication sub-protocol, compounding the attack surface.

---

### Likelihood Explanation

The attacker must be a malicious coordinator (controls message routing). This is an explicitly documented trust assumption for threshold protocols. The attack requires only:
1. Recording all Round 1 and Round 2 messages from one completed session.
2. Routing those messages verbatim in a subsequent session with the same participants and threshold.

No cryptographic assumption needs to be broken. The attack is purely a message-relay operation.

---

### Recommendation

Add a per-session random nonce to `create_transcript`, mirroring the DKG approach. Each participant broadcasts a fresh 32-byte random value at the start of the protocol; all participants hash the collected values into a session ID and absorb it into the transcript before any commitment or proof is generated:

```rust
fn create_transcript(
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
    session_id: &[u8; 32],          // ← new: agreed-upon per-session random value
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);
    transcript.message(b"group", NAME);
    transcript.message(b"participants", &rmp_serde::encode::to_vec(participants)…);
    transcript.message(b"threshold", &…);
    transcript.message(b"session-id", session_id);   // ← binds transcript to this run
    Ok(transcript)
}
```

The session ID must be derived from contributions from all participants (e.g., XOR or hash of each participant's fresh random value) so no single party can control it.

---

### Proof of Concept

```rust
// Session 1: run normally, record all channel messages
let (share1, pub1) = run_triple_generation(participants, threshold, rng1);
// Attacker records: commitments_s1, poly_commitments_s1, proofs_s1

// Session 2: same participants, same threshold, different rng
// Malicious coordinator:
//   - intercepts each participant's Round 1 commitment, replaces with commitments_s1[i]
//   - intercepts each participant's Round 2 message, replaces with poly_commitments_s1[i] + proofs_s1[i]
// Result: all participants compute confirmation = H(commitments_s1) == confirmation from session 1
//         transcript state is identical to session 1
//         all dlog/dlogeq proofs from session 1 pass verify() in session 2
//         session 2 outputs the same (A, B, C) and same secret shares as session 1

// Two ECDSA signatures produced with the same triple → private key extraction
assert_eq!(pub1.big_a, pub2.big_a);  // same triple reused
```

fn create_transcript(
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);

    transcript.message(b"group", NAME);

    let enc = rmp_serde::encode::to_vec(participants).map_err(|_| ProtocolError::ErrorEncoding)?;
    transcript.message(b"participants", &enc);
    // To allow interop between platforms where usize is different
    transcript.message(
        b"threshold",
        &u64::try_from(threshold.value())
            .expect("threshold should always fit in u64")
            .to_be_bytes(),
    );
    Ok(transcript)
}
