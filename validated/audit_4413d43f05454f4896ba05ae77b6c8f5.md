### Title
Unchecked Precompile Call Return Value in `EvmErc20` Causes Permanent Token Burn Without NEAR Transfer - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens before calling the Aurora exit precompile via inline assembly. The return value of the `call()` opcode is assigned to `res` but **never checked**. If the precompile call fails at the EVM level for any reason, the burn is not reverted, permanently destroying the user's tokens with no corresponding NEAR-side transfer.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed first, irreversibly

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — a 0 return (failure) is silently ignored
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum`: [2](#0-1) 

And in `EvmErc20V2.sol`: [3](#0-2) 

The `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) and `ExitToEthereum` precompile (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) can return `ExitError` — causing the EVM `call()` to return `0` — in multiple reachable conditions:

- The `recipient` bytes in `withdrawToNear` are user-controlled and may encode an invalid NEAR account ID, causing `parse_recipient` or account ID validation inside the precompile to fail.
- `get_eth_connector_contract_account()` fails if the connector is not configured in storage.
- `get_nep141_from_erc20()` fails if the ERC-20 address has no registered NEP-141 mapping. [4](#0-3) 

When the precompile returns an error, the EVM `call()` returns `0`. Because `res` is never inspected and no `revert` is issued, the Solidity function returns successfully. The `_burn` that already executed is **not rolled back**, permanently destroying the user's tokens.

The `error_refund` callback mechanism (in `exit_to_near_precompile_callback`) only triggers when the NEAR-side `ft_transfer` promise fails after the precompile has already scheduled it. It does **not** trigger when the precompile itself fails at the EVM level and never schedules any promise. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user's ERC-20 tokens (representing bridged NEAR-side NEP-141 assets) are irreversibly burned inside the EVM state. No NEAR-side `ft_transfer` or `withdraw` promise is ever scheduled. The NEP-141 balance on the NEAR side remains with the Aurora engine contract and is unrecoverable by the user. The total EVM supply and NEAR supply become permanently mismatched (insolvency of the bridge accounting).

---

### Likelihood Explanation

**Medium-High.**

The `recipient` parameter of `withdrawToNear` is a raw `bytes` value supplied by the caller. NEAR account IDs have strict validity rules (lowercase alphanumeric, dots, underscores, length limits). Any user who passes a malformed recipient — an empty byte string, an uppercase string, a string exceeding 64 characters, or bytes that are not valid UTF-8 — will trigger a precompile error. The precompile validates the account ID and returns `ExitError`, the `call()` returns `0`, and the tokens are silently burned. This is a realistic user mistake, especially for users unfamiliar with NEAR account ID constraints.

---

### Recommendation

Check the return value of the assembly `call()` and revert if it is zero, so that the `_burn` is also rolled back:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora with a registered NEP-141 mapping.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear(bytes("INVALID ACCOUNT ID WITH SPACES"), amount)`.
4. Inside the EVM: `_burn(alice, amount)` executes — alice's balance drops to zero.
5. The precompile at `0xe921...` receives the input, calls `parse_recipient`, finds the account ID invalid, and returns `ExitError`.
6. The EVM `call()` returns `0` into `res`.
7. `res` is never checked; the function returns normally.
8. Alice's ERC-20 tokens are gone. No NEAR-side transfer was scheduled. The NEP-141 balance on the NEAR side is unchanged (still held by Aurora). Alice's funds are permanently frozen. [1](#0-0) [6](#0-5)
