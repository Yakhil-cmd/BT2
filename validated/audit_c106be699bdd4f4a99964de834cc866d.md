### Title
Gateway Admits Invoke Transactions With Unverified Signatures via `skip_stateful_validations` Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (which is the only place the account signature is checked) for any Invoke transaction whose nonce equals 1 when the on-chain account nonce is 0 and the account address appears in the mempool. An attacker who controls a fresh address can submit a valid `deploy_account` to seed the mempool, then immediately submit an Invoke with nonce=1 carrying an **arbitrary or invalid signature**. The gateway admits the Invoke to the mempool without ever verifying the signature, breaking the invariant that every admitted transaction has passed account-level authentication.

### Finding Description

The gateway stateful validation path is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, resource bounds)
       ├─ validate_by_mempool            (nonce/fee-escalation rules)
       └─ skip_stateful_validations      ← decides whether __validate__ runs
  └─ run_validate_entry_point           ← calls __validate__ (signature check)
```

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 434-458
if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
    if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
        return mempool_client
            .account_tx_in_pool_or_recent_block(tx.sender_address())
            .await
            ...
    }
}
```

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// lines 308-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations` the early-return guard fires before `__validate__` is ever called:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 79-81
if !tx.execution_flags.validate {
    return Ok(());
}
```

`perform_pre_validation_stage` (which does run) checks nonce, fee bounds, and balance — but **not** the signature. The signature is exclusively verified inside the account contract's `__validate__` entry point, which is skipped.

### Impact Explanation

An attacker can submit an Invoke transaction carrying a completely invalid (or forged) signature and have it admitted to the mempool without any cryptographic verification. The broken invariant is: *every transaction admitted to the mempool has been authenticated by the account's `__validate__` entry point*. This directly matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

When the batcher later executes the block, `__validate__` is called with `validate=true` and the transaction is reverted; the fee is charged. The attacker therefore loses the fee, but the mempool has accepted an unauthenticated transaction, which can be used to:

- Flood the mempool with signature-invalid transactions (paid DoS, but at low cost per slot since the attacker only needs one deploy_account to unlock unlimited nonce-1 invokes).
- Cause any downstream component that trusts "mempool-admitted = validated" to act on unauthenticated data.

### Likelihood Explanation

The trigger is fully unprivileged. Any user can:
1. Pre-fund a deterministic address (address is computable before deployment).
2. Submit a `deploy_account` with a valid signature → it enters the mempool.
3. Submit an Invoke with nonce=1 and an arbitrary/invalid signature → `skip_stateful_validations` returns `true` → admitted without signature check.

The default production config sets `max_nonce_for_validation_skip = 0x1`, so the window is exactly nonce=1, but that is sufficient for one unauthenticated Invoke per fresh account. The attacker can repeat this with new addresses.

### Recommendation

The skip should only suppress the `__validate__` call when the account contract does not yet exist on-chain (i.e., the deploy_account has not been committed). It must not skip signature verification entirely. Two options:

1. **Verify the signature statically** before skipping `__validate__`: extract the public key from the class constructor calldata and verify the Invoke signature off-chain before admitting it.
2. **Narrow the skip condition**: only skip `__validate__` when the account class is known and the signature can be verified against the expected public key embedded in the deploy_account calldata.

At minimum, add a comment and a test that explicitly documents that nonce-1 Invokes admitted via this path carry unverified signatures, so downstream consumers do not treat mempool admission as proof of authentication.

### Proof of Concept

```
1. Attacker derives address A = deploy_account_address(class_hash, salt, ctor_calldata).
2. Attacker funds address A with enough STRK to cover fees.
3. Attacker submits RpcTransaction::DeployAccount for address A with a valid signature.
   → Gateway: account_nonce=0, tx_nonce=0, deploy_account passes full validation.
   → Mempool: address A is now in tx_pool.
4. Attacker submits RpcTransaction::Invoke from address A with:
     nonce = 1
     signature = [0xdeadbeef, 0xdeadbeef]   ← completely invalid
   → Gateway stateful path:
       account_nonce = 0  (A not yet on-chain)
       tx_nonce = 1
       skip_stateful_validations:
           tx.nonce() == Nonce(Felt::ONE)  ✓
           account_nonce == Nonce(Felt::ZERO)  ✓
           account_tx_in_pool_or_recent_block(A) == true  ✓  (deploy_account is in pool)
       → returns skip=true
       run_validate_entry_point called with validate=false
       → __validate__ is NEVER called
       → invalid signature is NEVER checked
   → Invoke is admitted to the mempool.
5. Batcher processes block: deploy_account executes, then Invoke executes with validate=true.
   __validate__ fails (bad signature) → Invoke reverted, fee charged.
   Result: invalid-signature transaction was in the mempool; invariant broken.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }
