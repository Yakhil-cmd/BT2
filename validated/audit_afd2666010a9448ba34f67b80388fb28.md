### Title
Unbounded `stakers` Vector Causes Unbounded Gas Consumption in `get_stakers` — (File: `src/staking/staking.cairo`)

---

### Summary

The `stakers` storage vector in the Staking contract grows without bound as new stakers join. The `get_stakers` function in `IStakingConsensus` iterates over the **entire** vector on every call with no cap. Any unprivileged actor can repeatedly call `stake()` with the minimum stake amount to inflate the vector, eventually making `get_stakers` exceed the Starknet execution gas limit and permanently denying the consensus layer access to the active-staker set.

---

### Finding Description

Every call to `stake()` unconditionally appends the new staker's address to `self.stakers`: [1](#0-0) 

There is no maximum staker count enforced anywhere in the codebase — no `MAX_STAKERS` constant, no length check before the push.

`get_stakers`, the function the consensus layer calls to enumerate active validators for a given epoch, iterates over the **full range** of that vector: [2](#0-1) 

Crucially, stakers who have already exited (`unstake_action` completed) are **never removed** from `self.stakers`; they are merely skipped by the `is_staker_active` check inside the loop. The vector is append-only, so historical stakers permanently inflate the iteration cost of every future `get_stakers` call.

For each staker that is active, the loop also calls `get_staker_staking_power_at_epoch`, `get_public_key_at_epoch`, and `get_peer_id_at_epoch` — multiple storage reads per iteration — making the per-element cost non-trivial.

---

### Impact Explanation

**Medium — Unbounded gas consumption / griefing with damage to the protocol.**

Once the `stakers` vector is large enough, every invocation of `get_stakers` will exceed the Starknet execution gas ceiling. `get_stakers` is the sole on-chain interface through which the consensus layer retrieves the active validator set for an epoch. Rendering it permanently non-callable breaks the consensus query path and constitutes griefing with real protocol damage (consensus layer cannot obtain the staker set), matching the "Unbounded gas consumption" and "Griefing with no profit motive but damage to users or protocol" categories in the allowed Medium impact scope.

---

### Likelihood Explanation

The attack requires no special privilege. Any address can call `stake()` with the protocol's minimum stake amount. An attacker willing to lock up `N × min_stake` STRK can register `N` distinct staker addresses. Because stakers are never pruned from the vector even after exiting, the attacker can also cycle: stake → wait for exit window → unstake → repeat with a new address, permanently growing the vector while recovering most capital. The cost is proportional to the minimum stake and the exit wait window, both of which are finite and known.

---

### Recommendation

1. **Enforce a hard cap** on the total number of stakers (analogous to the M-04 recommendation). Add a `MAX_STAKERS` constant and assert `self.stakers.length() < MAX_STAKERS` before pushing in `stake()`.
2. **Prune exited stakers** from the vector during `unstake_action`, or maintain a separate active-staker count so the iteration in `get_stakers` can be bounded.
3. Alternatively, replace the full-range iteration with a paginated interface so callers can retrieve the staker set in bounded chunks.

---

### Proof of Concept

1. Deploy the Staking contract with `min_stake = M`.
2. Create `N` distinct EOA addresses, each funded with `M` STRK.
3. Each address calls `stake(reward_address, operational_address, M)`. Each call executes `self.stakers.push(staker_address)` with no guard. [1](#0-0) 
4. After `N` registrations, call `get_stakers(epoch_id: current_epoch + 1)`.
5. The function enters `for staker_address_ptr in self.stakers.into_iter_full_range()` and performs multiple storage reads for each of the `N` entries. [3](#0-2) 
6. At sufficiently large `N`, the call exhausts the Starknet execution gas limit and reverts, permanently blocking the consensus layer from querying the active staker set.
