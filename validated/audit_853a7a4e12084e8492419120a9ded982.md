### Title
Panic in `balance_at_epoch` When Trace Has Fewer Than Three Entries With Both Last Two in Future Epochs - (File: `src/staking/utils.cairo`)

### Summary

`balance_at_epoch` in `src/staking/utils.cairo` unconditionally calls `trace.third_last().unwrap_or_else(|err| panic!("{err}"))` without first checking that the trace contains at least three entries. When a staker has exactly two balance-trace checkpoints and both are scheduled for future epochs, the `third_last()` call returns an error and the function panics, reverting every transaction that touches that staker's balance — including `update_rewards`.

### Finding Description

`balance_at_epoch` is a shared helper used throughout the staking contract to read a staker's (or the global total-stake) balance at a given epoch:

```

```cairo
pub(crate) fn balance_at_epoch(trace: StoragePath<Trace>, epoch_id: Epoch) -> NormalizedAmount {
    let (epoch, balance) = trace.last().unwrap_or_else(|err| panic!("{err}"));
    let current_balance = if epoch <= epoch_id {
        balance
    } else {
        let (epoch, balance) = trace.second_last().unwrap_or_else(|err| panic!("{err}"));
        if epoch <= epoch_id {
            balance
        } else {
            let (epoch, balance) = trace.third_last().unwrap_or_else(|err| panic!("{err}"));
            assert!(epoch <= epoch_id, "{}", InternalError::INVALID_THIRD_LAST);
            balance
        }
    };
    NormalizedAmountTrait::from_amount_18_decimals(amount: current_balance)
}
``` [1](#0-0) 

The `starkware_utils::trace::Trace` methods `last()`, `second_last()`, and `third_last()` all return `Result` and error with `"Index out of bounds"` when the trace does not contain enough entries. The `unwrap_or_else(|err| panic!("{err}"))` pattern converts that error into a hard panic.

The `third_last()` branch is reached only when both the last and second-last checkpoints have an epoch strictly greater than `epoch_id`. This is a normal intermediate state: the protocol schedules balance changes K epochs ahead. A staker who makes two balance changes in two different epochs within the K-epoch window will have exactly two future-epoch entries in their trace. If the trace has never had a third entry, `third_last()` fails.

The same helper is called from three distinct internal functions:

- `get_staker_own_balance_at_epoch` → `staker_own_balance_trace` [2](#0-1) 
- `get_staker_delegated_balance_at_epoch` → `staker_delegated_balance_trace` [3](#0-2) 
- `get_total_staking_power_at_epoch` → `tokens_total_stake_trace` [4](#0-3) 

The public entry point that exercises the first path is `update_rewards`:

`update_rewards` calls `get_staker_total_strk_btc_balance_at_epoch`, which calls `get_staker_own_balance_at_epoch`, which calls `balance_at_epoch`. [5](#0-4) 

### Impact Explanation

When the panic fires, the entire transaction reverts. Because `update_rewards` is the mechanism by which a staker's unclaimed rewards are accumulated, a staker stuck in this trace state cannot have their rewards updated. Every call to `update_rewards` for that staker reverts until the trace acquires a third entry (i.e., the staker makes a third balance change or the epoch window advances past both future entries). During that window, unclaimed yield accrues but cannot be claimed, constituting **temporary freezing of unclaimed yield**.

If the panic is triggered on `tokens_total_stake_trace` (the global total-stake trace), `get_total_staking_power_at_epoch` panics for every staker, escalating the impact to a **protocol-wide reward freeze**.

### Likelihood Explanation

Any staker who calls `stake` (or `increase_stake`) in two different epochs within the K-epoch lookahead window will produce exactly this two-entry future-epoch trace state. This is a routine operation requiring no special privilege. The window during which the panic is reachable lasts until the current epoch advances past the earlier of the two future entries, which can span multiple epochs.

### Recommendation

Before calling `third_last()`, check the trace length. If the trace has fewer than three entries and both existing entries are in the future, the staker has no historical balance for the queried epoch and the function should return zero (or the appropriate default). A safe pattern:

```cairo
// After second_last() also returns a future epoch:
let len = trace.length(); // or equivalent
if len < 3 {
    // No historical entry exists; balance at epoch_id is zero.
    return NormalizedAmountTrait::from_amount_18_decimals(amount: 0);
}
let (epoch, balance) = trace.third_last().unwrap_or_else(|err| panic!("{err}"));
assert!(epoch <= epoch_id, "{}", InternalError::INVALID_THIRD_LAST);
balance
```

### Proof of Concept

1. Protocol is at epoch `E`. K = 2 (balance changes take effect 2 epochs later).
2. Staker calls `stake()` at epoch `E` → trace gets entry `(E+2, balance_1)`. Trace length = 1.
3. Staker calls `increase_stake()` at epoch `E+1` → trace gets entry `(E+3, balance_2)`. Trace length = 2.
4. At epoch `E+1`, the attestation contract calls `update_rewards(staker_address)`.
5. Inside `update_rewards`, `get_staker_own_balance_at_epoch(staker_address, epoch_id = E+1)` is called.
6. `balance_at_epoch` executes:
   - `last()` → `(E+3, balance_2)`. `E+3 > E+1` → fall through.
   - `second_last()` → `(E+2, balance_1)`. `E+2 > E+1` → fall through.
   - `third_last()` → **error** (only 2 entries). `unwrap_or_else` → **panic**.
7. Transaction reverts. Staker's rewards cannot be updated until epoch `E+2` passes. [1](#0-0) [6](#0-5)
