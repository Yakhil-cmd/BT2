### Title
Unbounded Pre-Charge Authorization List Parsing Enables CPU Exhaustion DoS - (`basic_bootloader/src/bootloader/transaction/rlp_encoded/rlp/minimal_rlp_parser.rs`)

### Summary

EIP-7702 transactions carry an `AuthorizationList` typed as `HomList<'a, AuthorizationEntry<'a>, true>` (VALIDATE = true). When `VALIDATE = true`, `HomList::decode_list_from` performs a full O(N) iteration over every entry to count and validate them during transaction parsing. This parsing pass occurs before any gas or native resource is charged. An unprivileged attacker can submit a crafted EIP-7702 transaction with a maximally large authorization list and a gas limit too low to cover intrinsic cost, forcing the sequencer to perform unbounded CPU work before the transaction is rejected without fee collection.

### Finding Description

**Root cause — double-parse before gas charging:**

`HomList::decode_list_from` with `VALIDATE = true` unconditionally iterates all entries at parse time:

```rust
let count = if VALIDATE {
    let mut chk = Rlp::new(all);
    let mut c = 0;
    while !chk.is_empty() {
        T::decode_from_item(&mut chk)?;   // full RLP decode of each entry
        c += 1;
    }
    Some(c)
} else {
    None
};

`AuthorizationList` is defined as this validated variant:

```rust

This validation pass is triggered inside `EIP7702Tx::decode_list_body`:

``` [3](#0-2) 

**Execution order — parsing precedes all gas checks:**

In the transaction loop, `try_begin_next_tx` reads the raw bytes (bounded only by `u32::MAX - 3`), then `process_transaction` is called. Transaction parsing (including the full authorization list validation pass) happens first; `validate_and_compute_fee_for_transaction` — which enforces the gas limit, EIP-7623 calldata floor, and intrinsic gas per authorization entry — is called only afterward. [4](#0-3) [5](#0-4) 

The intrinsic gas per authorization entry (`NEWACCOUNT = 25,000 gas`) is computed and charged only after the full parse: [6](#0-5) 

**Attack path:**

1. Attacker submits an EIP-7702 transaction (type `0x04`) with a maximally large authorization list — e.g., ~37,000 minimum-size entries (~27 bytes each ≈ 1 MB total).
2. Gas limit is set to any value below `authorization_list_num × 25,000` (intrinsic cost).
3. The sequencer calls `HomList::decode_list_from`, iterating all N entries (N × 6 RLP field decodes) with no resource charge.
4. `validate_and_compute_fee_for_transaction` rejects the transaction: intrinsic gas exceeds gas limit.
5. No fee is collected; the attacker repeats at will.

Each `AuthorizationEntry` decode parses 6 RLP fields (chain_id U256, 20-byte address, nonce u64, y_parity u8, r bytes, s bytes): [7](#0-6) 

### Impact Explanation

The sequencer performs O(N) RLP decode work — proportional to the number of authorization entries — before any gas is charged or any fee is collected. Because rejected transactions incur no fee, the attacker can flood the sequencer with large-authorization-list transactions at zero cost, degrading sequencer throughput. In the proving context (RISC-V / Talc allocator), the same unbounded parse work inflates proof generation cycles, increasing proving cost and latency without any corresponding resource payment from the attacker. This is a **resource accounting bug** causing **service unavailability / severe degradation** under realistic attacker input.

### Likelihood Explanation

EIP-7702 (type `0x04`) is a standard transaction type accepted from any unprivileged sender. No special role, key, or governance access is required. The attacker only needs to submit a valid-format (but gas-insufficient) EIP-7702 transaction to the sequencer's public RPC endpoint. The practical upper bound on N is set by the block gas limit via calldata cost (EIP-7623 floor), but this check occurs after parsing, so the CPU work is already done before rejection. Likelihood is **medium-high** given the zero-cost repeatability.

### Recommendation

1. **Enforce a maximum authorization list entry count before parsing.** Read the outer RLP list length header and reject the transaction immediately if the encoded list byte length implies more entries than `block_gas_limit / NEWACCOUNT_GAS`.
2. **Defer the VALIDATE pass.** Change `AuthorizationList` to `HomList<'a, AuthorizationEntry<'a>, false>` (lazy, non-validating) during initial parse. Perform per-entry validation only inside `parse_authorization_list_and_apply_delegations`, after intrinsic gas has been charged and deducted.
3. **Add a pre-parse size gate.** In `try_begin_next_tx` or at the start of `process_transaction`, reject transactions whose raw byte length exceeds a configurable maximum (e.g., `block_gas_limit / CALLDATA_NON_ZERO_BYTE_TOKEN_FACTOR`) before any RLP parsing begins.

    pub fn decode_list_from(r: &mut Rlp<'a>) -> Result<Self, InvalidTransaction> {
        let inner = r.list()?;
        let all = inner.remaining();

            }

pub fn calculate_tx_intrinsic_gas(
    calldata_len: u64,
    calldata_tokens: u64,
    is_deployment: bool,
    access_list_accounts: u64,
    access_list_storage_keys: u64,
    authorization_list_num: u64,
) -> u64 {
    let mut intrinsic_gas = TX_INTRINSIC_GAS;

    if is_deployment {
        intrinsic_gas = intrinsic_gas.saturating_add(DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS);
        let initcode_gas_cost =
            evm_interpreter::gas_constants::INITCODE_WORD_COST * calldata_len.div_ceil(32);
        intrinsic_gas = intrinsic_gas.saturating_add(initcode_gas_cost);
    }
    intrinsic_gas =
        intrinsic_gas.saturating_add(calldata_tokens.saturating_mul(CALLDATA_TOKEN_GAS_COST));

    // EIP-2930 access list: per-address + per-storage-key.
    intrinsic_gas = intrinsic_gas.saturating_add(
        access_list_accounts.saturating_mul(evm_interpreter::gas_constants::ACCESS_LIST_ADDRESS),
    );
    intrinsic_gas = intrinsic_gas.saturating_add(
        access_list_storage_keys
            .saturating_mul(evm_interpreter::gas_constants::ACCESS_LIST_STORAGE_KEY),
    );

    // EIP-7702 authorization list: per-authorization. We precharge the
    // empty-account cost; when the authority turns out to be non-empty the
    // delta (NEWACCOUNT - PER_AUTH_BASE_COST) is added back as a gas refund
    // inside `validate_and_apply_delegation`.
    intrinsic_gas = intrinsic_gas.saturating_add(
        authorization_list_num.saturating_mul(evm_interpreter::gas_constants::NEWACCOUNT),
    );

pub fn parse_authorization_list_and_apply_delegations<S: EthereumLikeTypes>(
    system: &mut System<S>,
    resources: &mut S::Resources,
    auth_list: AuthorizationList<'_>,
) -> Result<(), TxError>
where
    S::IO: IOSubsystemExt,
{
    use crate::bootloader::transaction::rlp_encoded::AuthorizationEntry;
    let mut hasher = crypto::sha3::Keccak256::new();

    for entry in auth_list.iter() {
        let AuthorizationEntry {
            chain_id,
            address,
            nonce,
            y_parity,
            r,
            s,
        } = entry;
        let success = validate_and_apply_delegation(
            system,
            resources,
            &chain_id,
            nonce,
            address,
            (y_parity, r, s),
            &mut hasher,
        )?;
        system_log!(system, "Delegation success: {success}\n");

        if !success {}
    }
    Ok(())
}
