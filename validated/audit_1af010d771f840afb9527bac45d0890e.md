### Title
Missing Maximum Commission Guard in Delegation Entry Functions Enables Front-Running Commission Increase — (File: src/pool/pool.cairo)

---

### Summary

`enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool` in `src/pool/pool.cairo` accept no `max_commission` parameter. A staker who holds an active `CommissionCommitment` can atomically raise commission to `max_commission` (up to 100 %) immediately before a delegator's transaction lands, capturing all future yield as commission. The delegator has no on-chain mechanism to bound the commission rate they will accept.

---

### Finding Description

**Commission increase path (confirmed in code)**

`update_commission` in `src/staking/staking.cairo` (lines 1580–1597) contains two branches:

```
if commitment is active:
    assert commission <= commitment.max_commission   // can INCREASE
    assert commission != old_commission
else:
    assert commission < old_commission               // can only decrease
```

When a `CommissionCommitment` is active the only constraint is `commission ≤ max_commission`. There is no lower-bound requirement, so a staker can move commission from 0 % to 100 % in a single call. The code itself acknowledges this at lines 745–746:

> `/// **Note**: Current commission increase safeguards still allow for sudden commission changes.`

**Entry functions carry no slippage guard**

`enter_delegation_pool` (pool.cairo ~line 182) signature:
```rust
fn enter_delegation_pool(ref self: ContractState, reward_address: ContractAddress, amount: Amount)
```

`add_to_delegation_pool` (pool.cairo ~line 221):
```rust
fn add_to_delegation_pool(ref self: ContractState, pool_member: ContractAddress, amount: Amount) -> Amount
```

`switch_delegation_pool` (pool.cairo ~line 379):
```rust
fn switch_delegation_pool(ref self: ContractState, to_staker: ContractAddress, to_pool: ContractAddress, amount: Amount) -> Amount
```

None of these accept a `max_commission` argument. The delegator cannot express "only enter if commission ≤ X".

**`set_commission_commitment` constraints** (staking.cairo ~line 748):
- `max_commission >= current_commission` — trivially satisfied when current commission is 0 %
- `expiration_epoch > current_epoch` — one epoch in the future is sufficient
- `expiration_epoch - current_epoch <= epochs_in_year`

A staker can therefore set `max_commission = 10000` (100 %) at any time, then later call `set_commission(10000)` in the same block as a delegator's entry transaction.

---

### Impact Explanation

Once the delegator is inside the pool at 100 % commission, every reward epoch transfers the entire pool reward to the staker as commission. The delegator's `unclaimed_rewards` accumulates zero. This constitutes **theft of unclaimed yield** for the lifetime of the delegation (until the delegator notices and exits, which itself requires waiting through the full `exit_wait_window`).

Impact: **High — Theft of unclaimed yield.**

---

### Likelihood Explanation

- Any staker can call `set_commission_commitment` permissionlessly at any time.
- Starknet currently uses a single sequencer, making transaction ordering observable and front-running feasible for a staker who monitors the mempool.
- The attack is cheap (one extra `set_commission` call) and leaves no on-chain evidence distinguishing it from a legitimate commission change.
- Delegators have no way to detect the commitment's `max_commission` from the pool interface alone; they see only the current commission.

Likelihood: **Medium** (requires a malicious staker and mempool visibility, but both are realistic).

---

### Recommendation

1. Add a `max_commission: Commission` parameter to `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool`. Assert `current_commission <= max_commission` at the start of each function, reverting if the staker has already raised commission above the delegator's tolerance.

2. Alternatively, expose the `CommissionCommitment` (specifically `max_commission`) in the pool's public view interface so delegators can make an informed decision off-chain, and document the risk clearly.

---

### Proof of Concept

```
// Setup
staker calls set_commission(commission: 0)          // advertise 0 % to attract delegators
staker calls set_commission_commitment(
    max_commission: 10000,                           // 100 %
    expiration_epoch: current_epoch + 1              // valid: > current, <= epochs_in_year
)

// Attack (same block or epoch as delegator's tx)
// Delegator submits:
//   pool.enter_delegation_pool(reward_address, amount)   <-- no max_commission guard

// Staker front-runs with:
staker calls set_commission(commission: 10000)       // raises to 100 %
// Allowed because commitment is active and 10000 <= max_commission (10000)

// Result
// Delegator is now in pool with 100 % commission.
// Every subsequent reward epoch:
//   commission_rewards = pool_rewards * 10000 / 10000 = pool_rewards
//   delegator_rewards  = 0
// All yield flows to staker indefinitely.
```
