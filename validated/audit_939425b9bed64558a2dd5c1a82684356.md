### Title
Malicious Coordinator Can Undetectably Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol's coordinator-centric design grants the coordinator unchecked, unilateral control over the final `CKDOutput`. Non-coordinator participants send their secret-share-derived contributions privately to the coordinator and receive no output, making it impossible for them to verify the coordinator's aggregation. A malicious coordinator can output any arbitrary `CKDOutput`, causing the application to derive a wrong confidential key with no detection mechanism.

---

### Finding Description

In `do_ckd_participant`, each non-coordinator participant computes their Lagrange-weighted ElGamal contribution `(norm_big_y, norm_big_c)` and sends it **privately** to the coordinator via `chan.send_private`, then immediately returns `None`: [1](#0-0) 

The coordinator in `do_ckd_coordinator` receives all participants' individual shares, sums them, and is the **sole** party that receives the `CKDOutput`: [2](#0-1) 

There is no mechanism for:
1. Participants to verify the coordinator correctly aggregated their shares.
2. The coordinator to prove the output is the honest sum.
3. Any cross-check between participants about the coordinator's behavior.

The `CKDOutput` type has no embedded proof of correct construction: [3](#0-2) 

A malicious coordinator can:
- **Drop** some participants' `(norm_big_y_i, norm_big_c_i)` contributions entirely.
- **Substitute** crafted values for any participant's contribution.
- **Output a completely fabricated** `CKDOutput::new(big_Y', big_C')` for any arbitrary group elements.

None of these manipulations are detectable by honest participants, who return `None` and observe no protocol output.

---

### Impact Explanation

Each participant's contribution encodes their Lagrange-weighted secret share as:

```
norm_big_c_i = λi · (x_i · H(pk ‖ app_id) + y_i · app_pk)
norm_big_y_i = λi · y_i · G
```

The correct aggregate satisfies `big_C - app_sk · big_Y = msk · H(pk ‖ app_id)`. When the coordinator outputs a corrupted `(big_Y', big_C')`, the application's call to `unmask(app_sk)`: [4](#0-3) 

yields a wrong group element instead of `msk · H(pk ‖ app_id)`. The application silently accepts an unusable or attacker-chosen confidential derived key.

**Matched impact**: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

Unlike the ECDSA signing coordinator (which verifies the final signature before returning it at `src/ecdsa/ot_based_ecdsa/sign.rs` lines 129–133 and `src/ecdsa/robust_ecdsa/sign.rs` lines 159–163), the CKD coordinator performs **no correctness check** on the output it returns.

---

### Likelihood Explanation

The coordinator is a required, named participant in every CKD invocation: [5](#0-4) 

Any participant in the set can be designated coordinator. The attack requires only that the coordinator deviate from the honest aggregation loop — a trivial, single-step deviation with no cryptographic barrier. The protocol provides no accountability: participants cannot distinguish a correct aggregation from a fabricated one.

---

### Recommendation

Add a verifiable aggregation mechanism so participants can audit the coordinator's output:

1. **Broadcast the output**: Have the coordinator broadcast `CKDOutput` to all participants. Each participant can verify that `big_C - big_Y · app_pk_contribution` is consistent with their own `norm_big_c_i - norm_big_y_i · app_pk`.
2. **Commitment-then-reveal**: Participants commit to their `(norm_big_y_i, norm_big_c_i)` before sending, and the coordinator must prove the output is the sum of the revealed values.
3. **Coordinator-proof**: Require the coordinator to provide a zero-knowledge proof of correct aggregation alongside the `CKDOutput`.

---

### Proof of Concept

1. Participants A, B, C run `ckd(...)` with C designated as coordinator.
2. A and B each compute honest `(norm_big_y, norm_big_c)` and send them privately to C via `chan.send_private`. [6](#0-5) 

3. C (malicious) ignores A's and B's contributions and calls `CKDOutput::new(ElementG1::identity(), ElementG1::identity())` — or any arbitrary pair — as the return value.
4. A and B return `Ok(None)` and observe no output; they cannot detect the substitution.
5. The application receives the corrupted `CKDOutput`, calls `unmask(app_sk)`, and derives a wrong confidential key, silently accepting it as valid.

This is the direct analog of the staking contract's `withDrawFunds()`: just as the owner could drain any token without participant consent, the CKD coordinator can unilaterally corrupt the protocol output without any participant being able to detect or prevent it.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

#[derive(Debug, Clone, Deserialize, Serialize)]
pub struct CKDOutput {
    big_y: ElementG1,
    big_c: ElementG1,
}

impl CKDOutput {
    pub fn new(big_y: ElementG1, big_c: ElementG1) -> Self {
        Self { big_y, big_c }
    }

    /// Outputs `big_y`
    pub fn big_y(&self) -> ElementG1 {
        self.big_y
    }

    /// Outputs `big_c`
    pub fn big_c(&self) -> ElementG1 {
        self.big_c
    }

    /// Takes a secret scalar and returns
    /// s <- C − a ⋅ Y = msk ⋅ H ( `app_id` )
    pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
        self.big_c - self.big_y * secret_scalar
    }
}
