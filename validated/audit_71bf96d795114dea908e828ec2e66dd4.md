### Title
Fixed Gas Limit (21,000 + 2,300) on Native Token Transfers Breaks Compatibility with Smart Contract Wallets - (`fvm/evm/types/call.go`)

### Summary

Flow EVM's `NewDepositCall`, `NewWithdrawCall`, and `NewTransferCall` all hard-code a gas limit of `21,000 + 2,300 = 23,300` for native FLOW token transfers. This is the exact EVM equivalent of Solidity's `transfer()` gas stipend. Any EVM smart contract whose `receive()` or `fallback()` function requires more than 2,300 gas (e.g., a Gnosis Safe multisig, a proxy wallet, or any contract that emits events or writes storage on receipt) will always revert with out-of-gas when receiving tokens via the Cadence-to-EVM bridge.

### Finding Description

In `fvm/evm/types/call.go`, three constants define the gas budget for all protocol-level native token transfers:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21,000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23,300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

Every constructor that builds a protocol-level token transfer `DirectCall` uses this fixed limit:

- `NewDepositCall` — `GasLimit: DepositCallGasLimit` (23,300)
- `NewWithdrawCall` — `GasLimit: WithdrawCallGasLimit` (23,300)
- `NewTransferCall` — `GasLimit: DefaultGasLimitForTokenTransfer` (23,300) [2](#0-1) 

These constructors are called unconditionally from the handler layer whenever a Cadence transaction deposits or transfers FLOW tokens to an EVM address: [3](#0-2) 

The existing test suite already documents the failure mode — depositing to a contract that consumes more than 2,300 gas in its `receive()` function produces `ErrExecutionReverted`: [4](#0-3) 

### Impact Explanation

Any EVM smart contract wallet (Gnosis Safe, proxy wallet, ERC-4337 account, or any contract that writes storage or emits events in `receive()`) deployed on Flow EVM will permanently be unable to receive FLOW tokens via:

1. `CadenceOwnedAccount.deposit()` — calls `NewDepositCall` with 23,300 gas
2. `Account.Transfer()` — calls `NewTransferCall` with 23,300 gas
3. `CadenceOwnedAccount.withdraw()` path back through the bridge — calls `NewWithdrawCall` with 23,300 gas

The Cadence transaction reverts (via `panicOnErrorOrInvalidOrFailedState`), so tokens are not permanently lost in a single call, but the recipient smart contract address is permanently incompatible with the bridge. Users who hold assets in multisig or smart-contract wallets on Flow EVM cannot receive bridged FLOW tokens through any Cadence-initiated path.

### Likelihood Explanation

Smart contract wallets (multisigs, proxy accounts, ERC-4337 accounts) are the dominant custody model for institutional and protocol-level EVM users. Gnosis Safe alone requires ~6,000–10,000 gas in its `receive()` hook. Any such wallet deployed on Flow EVM is affected. The entry path requires only an unprivileged Cadence transaction sender attempting a deposit — no special privileges needed.

### Recommendation

Replace the hard-coded `21,000 + 2,300` gas limit with a configurable or sufficiently large default (e.g., matching what `COA.call()` allows — a caller-specified `gasLimit`). At minimum, `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` should be raised to a value that accommodates common smart contract wallet patterns. The comment in the code itself acknowledges the risk:

```go
// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300
``` [5](#0-4) 

The fix should allow callers to specify a gas limit for deposit/transfer calls, or raise the default to a value sufficient for common smart contract wallet `receive()` implementations.

### Proof of Concept

1. Deploy a Gnosis Safe (or any contract with a `receive()` that writes to storage) on Flow EVM.
2. Submit a Cadence transaction that calls `cadenceOwnedAccount.deposit(from: <-vault)` targeting the Safe's EVM address.
3. The `NewDepositCall` is constructed with `GasLimit: 23,300`.
4. The Safe's `receive()` function requires >2,300 gas → EVM execution reverts with out-of-gas.
5. `panicOnErrorOrInvalidOrFailedState` panics → the Cadence transaction reverts.
6. The Safe address can never receive FLOW tokens via any Cadence-initiated bridge path. [1](#0-0) [6](#0-5)

// Deposit deposits the token from the given vault into the flow evm main vault
// and update the account balance with the new amount
func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}

// Withdraw deducts the balance from the account and
// withdraw and return flow token from the Flex main vault.
func (a *Account) Withdraw(b types.Balance) *types.FLOWTokenVault {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMWithdraw).End()

	res, err := a.executeAndHandleAuthorizedCall(
		types.NewWithdrawCall(
			a.fch.addressAllocator.NativeTokenBridgeAddress(),
			a.address,
			b,
			a.Nonce(),
		),
		b,
		true,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)

	return types.NewFlowTokenVault(b)
}

// Transfer transfers tokens between accounts
func (a *Account) Transfer(to types.Address, balance types.Balance) {
	res, err := a.executeAndHandleAuthorizedCall(
		types.NewTransferCall(
			a.address,
			to,
			balance,
			a.Nonce(),
		),
		nil,
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}
