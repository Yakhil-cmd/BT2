### Title
Gateway Signature Bypass via `skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Account with a Pending Deploy — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path unconditionally skips the `__validate__` entry-point call (the only place where the account's signature is verified) for any Invoke transaction whose nonce equals 1 and whose sender address appears in the mempool. Because the mempool check (`account_tx_in_pool_or_recent_block`) is keyed only on the sender address — not on the identity of the submitter — any unprivileged attacker who observes a victim's pending `deploy_account` transaction can immediately submit an Invoke transaction for that same address carrying an arbitrary/fake signature, and the gateway will admit it without any signature verification.

---

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have its signature verified by the account's `__validate__` entry point before admission. The `skip_stateful_validations` UX feature breaks this invariant for Invoke transactions with `nonce = 1` targeting any account whose address is already present in the mempool.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in state).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

Condition 4 is satisfied whenever **any** transaction for that address is in the mempool pool or in a recently committed block — it does not verify that the submitter of the new Invoke is the same party who submitted the `deploy_account`. [3](#0-2) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false`, meaning the `__validate__` entry point is never executed: [4](#0-3) 

The full admission path is: [5](#0-4) 

**Pre-validation nonce check does not block the attack.** The `validate_nonce` check for Invoke allows any nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]`. With `account_nonce = 0` and `max_allowed_nonce_gap = 200` (production default), `nonce = 1` passes trivially: [6](#0-5) [7](#0-6) 

**Mempool `validate_tx` does not check the signature.** It only checks for duplicate tx hash, nonce-too-old, and fee escalation: [8](#0-7) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can submit an Invoke transaction with a completely fabricated signature for any account that has a pending `deploy_account` in the mempool. The transaction passes all gateway checks and is inserted into the mempool. Concrete consequences:

1. **Admission of cryptographically invalid transactions.** The mempool now holds an Invoke whose signature was never verified. This violates the core admission invariant.

2. **Griefing / displacement of legitimate transactions.** If the victim also submitted a legitimate Invoke with `nonce = 1`, the attacker can replace it via fee escalation (by providing a higher tip/gas price). The victim's valid transaction is evicted; the attacker's fake transaction occupies the slot and will fail at execution time, wasting the victim's `deploy_account` gas and delaying their account setup.

3. **Repeated DoS.** The attacker can repeat this for every new `deploy_account` that appears in the mempool, systematically disrupting the deploy-account + invoke UX flow for all users.

The fake transaction will revert at blockifier execution time (because `validate: true` is set during block building), but the damage — admission, displacement, and execution-time failure — has already occurred at the gateway/mempool layer.

---

### Likelihood Explanation

**Likelihood: High.** The attack requires no privileged access, no special contract deployment, and no cryptographic capability. The attacker only needs to:

- Monitor the public mempool for `deploy_account` transactions (trivially observable via the RPC or P2P layer).
- Submit an Invoke transaction with the victim's sender address, `nonce = 1`, and any bytes as the signature.

The production configuration (`max_allowed_nonce_gap = 200`, `max_nonce_for_validation_skip = 0x1`) is confirmed in both the Rust default and the deployed config: [9](#0-8) 

The existing test `test_skip_validate` with `case::should_skip_validation` explicitly confirms that an Invoke with `nonce = 1`, `account_nonce = 0`, and `contains_tx = true` skips validation: [10](#0-9) 

---

### Recommendation

The `skip_stateful_validations` mechanism must be tightened so that it only skips signature verification when the **same sender** submitted the pending `deploy_account`. Concretely:

1. **Bind the skip to the deploy_account transaction hash.** The gateway should require the caller to supply the hash of the pending `deploy_account` transaction (as the `native_blockifier` path already does via `deploy_account_tx_hash`), and verify that this hash is present in the mempool for the same sender address before skipping validation. [11](#0-10) 

2. **Alternatively, remove the `account_tx_in_pool_or_recent_block` shortcut** and always run the `__validate__` entry point. The UX benefit of skipping validation is not worth the security cost of admitting unsigned transactions.

3. **At minimum**, the `skip_stateful_validations` function should verify that the mempool entry for the sender address is specifically a `deploy_account` transaction with `nonce = 0`, not just any transaction.

---

### Proof of Concept

```
1. Victim submits deploy_account tx for address A (class_hash C, salt S).
   → Mempool now contains A; account_tx_in_pool_or_recent_block(A) = true.
   → On-chain state: nonce(A) = 0 (account not yet deployed).

2. Attacker observes A in the mempool (via RPC or P2P).

3. Attacker constructs an Invoke V3 tx:
     sender_address = A
     nonce          = 1
     calldata       = [<any malicious calldata>]
     signature      = [0xdeadbeef]   ← completely fake
     resource_bounds = <valid, above min_gas_price>

4. Attacker submits to gateway.

5. Gateway stateful validation:
   a. get_nonce_from_state(A) → 0
   b. validate_state_preconditions:
        validate_resource_bounds → passes (valid bounds)
        validate_nonce: 0 <= 1 <= 0+200 → passes
   c. validate_by_mempool:
        no duplicate tx_hash → passes
        nonce 1 >= account_nonce 0 → passes
   d. skip_stateful_validations:
        tx.nonce == 1 ✓
        account_nonce == 0 ✓
        account_tx_in_pool_or_recent_block(A) == true ✓
        → returns true (SKIP VALIDATION)
   e. run_validate_entry_point(skip_validate=true):
        ExecutionFlags { validate: false, ... }
        → __validate__ entry point NOT called
        → signature NOT verified

6. Attacker's Invoke with fake signature is admitted to the mempool.

7. If victim also submitted a legitimate Invoke(nonce=1):
   - Attacker re-submits with higher tip → fee escalation replaces victim's tx.
   - Victim's valid tx is evicted.

8. At block execution time:
   - deploy_account(nonce=0) executes successfully.
   - Attacker's Invoke(nonce=1) runs __validate__ → fake signature → REVERT.
   - Victim's legitimate Invoke was already evicted; victim must resubmit.
```

```rust
    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
```
