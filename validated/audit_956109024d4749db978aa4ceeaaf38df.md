Audit Report

## Title
Unbounded Slot Table Growth via `SLOT_TABLE_NO_LIMIT` Allows Single Byzantine Peer to Exhaust Replica Heap — (`rs/p2p/consensus_manager/src/receiver.rs`, `rs/replica/setup_ic_network/src/lib.rs`)

## Summary

`SLOT_TABLE_NO_LIMIT = usize::MAX` is assigned to every artifact channel except ingress (consensus, certification, DKG, IDKG, HTTPS outcalls). A single authenticated Byzantine subnet peer can send an unbounded number of slot updates with distinct `slot_number` values and distinct artifact IDs, causing `slot_table`, `active_assembles`, and `artifact_processor_tasks` to grow without bound, exhausting heap memory and saturating the Tokio runtime, stalling consensus and IDKG progress.

## Finding Description

In `rs/replica/setup_ic_network/src/lib.rs`, two constants are defined:

```rust
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
``` [1](#0-0) 

Every channel except ingress is registered with `SLOT_TABLE_NO_LIMIT`: [2](#0-1) 

The slot limit guard in `handle_slot_update_receive` is:

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => {
    empty_slot.insert(new_slot_entry);
    ...
    (true, None)
}
Entry::Vacant(_) => {
    // dropped
    (false, None)
}
``` [3](#0-2) 

With `slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice. Every distinct `(slot_number, artifact_id)` pair from a peer is accepted unconditionally.

For each accepted new slot entry where the artifact ID is not already tracked, the receiver inserts into `active_assembles` and spawns a new task into `artifact_processor_tasks`: [4](#0-3) 

The `slot_number` is decoded directly from the peer-controlled protobuf field `slot_id` (a `u64`): [5](#0-4) 

Additionally, `handle_artifact_processor_joined` restarts tasks if the peer is still advertising, compounding CPU exhaustion: [6](#0-5) 

The existing test `slot_table_limit_exceeded` at line 847 confirms the guard works correctly when a finite limit is set — it is simply not applied to non-ingress channels: [7](#0-6) 

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

- **Memory exhaustion**: Each of N distinct slot updates allocates a `SlotEntry`, a `watch::Sender<PeerCounter>`, and a Tokio task. At millions of entries, the replica's heap is exhausted.
- **Consensus/IDKG stall**: The Tokio runtime is saturated with N tasks waiting on `peer_rx.wait_for(|p| p.is_empty())`, starving legitimate consensus and IDKG processing tasks.
- **Sustained attack**: Tasks are never cleaned up until the peer disconnects or is removed from topology, so the Byzantine peer can maintain the attack indefinitely.

## Likelihood Explanation

The attacker must be a registered subnet node authenticated via TLS/QUIC — a meaningful barrier requiring a compromised or malicious node operator. However, the IC's BFT model explicitly tolerates up to `f < n/3` Byzantine nodes, and the invariant that a **single** below-threshold peer cannot cause unbounded resource consumption is violated here. The `slot_id` field is a peer-controlled `u64`, so generating N distinct slot numbers requires no special knowledge. The attack is local-testable with the existing `ReceiverManagerBuilder` test infrastructure.

## Recommendation

Apply a per-peer slot table limit to all artifact channels, not just ingress. Derive appropriate limits from protocol bounds (e.g., number of active transcripts × dealers for IDKG/DKG):

```rust
const SLOT_TABLE_LIMIT_IDKG: usize = 10_000;
const SLOT_TABLE_LIMIT_DKG: usize = 1_000;
const SLOT_TABLE_LIMIT_CONSENSUS: usize = 1_000;
const SLOT_TABLE_LIMIT_CERTIFICATION: usize = 1_000;
const SLOT_TABLE_LIMIT_HTTPS_OUTCALLS: usize = 1_000;
```

Pass these constants to `abortable_broadcast_channel` in place of `SLOT_TABLE_NO_LIMIT`. Additionally, add a metric alert when any peer approaches its slot limit to detect misbehaving nodes early.

## Proof of Concept

Using the existing `ReceiverManagerBuilder` test infrastructure in `rs/p2p/consensus_manager/src/receiver.rs`:

```rust
#[tokio::test]
async fn byzantine_peer_exhausts_memory_via_idkg_slots() {
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        .with_slot_limit(usize::MAX) // production default for IDKG
        .build();
    let cancellation = CancellationToken::new();
    let n = 1_000_000usize;
    for i in 0..n {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i as u64),
                commit_id: CommitId::from(1),
                update: Update::Id(i as u64),
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    assert_eq!(mgr.slot_table.get(&NODE_1).unwrap().len(), n);
    assert_eq!(mgr.active_assembles.len(), n);
    assert_eq!(mgr.artifact_processor_tasks.len(), n);
}
```

This mirrors the existing `slot_table_limit_exceeded` test pattern and requires no network access. RSS growth proportional to N is observable, and OOM occurs on production replicas at sufficient N.
