### Title
Presignature Nonce Material Committed Irreversibly With No Reuse-Prevention After Failed Signing — (`src/ecdsa/robust_ecdsa/mod.rs`, `src/frost/mod.rs`)

---

### Summary

The presign → sign pipeline is a two-phase protocol where the first phase (presign) irreversibly commits nonce material to all participants, analogous to the token-burn in the Axelar report. If the signing phase subsequently fails or is aborted — a realistic outcome when the coordinator is malicious or the network partitions — the library provides no mechanism to invalidate or mark the presignature as consumed. Because `PresignOutput` is `Clone` and carries no "used" flag, an orchestration layer will naturally retry signing with the same presignature. The library's own documentation explicitly states this leads to private-key extraction.

---

### Finding Description

**Phase 1 — Irreversible commitment (presign):**

In `src/ecdsa/robust_ecdsa/mod.rs`, `PresignOutput` is declared:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,   // public nonce commitment g^k
    pub c: Scalar,
    pub e: Scalar,
    pub alpha: Scalar,
    pub beta: Scalar,
}
``` [1](#0-0) 

During the presign protocol (`do_presign` in `src/ecdsa/robust_ecdsa/presign.rs`), each participant broadcasts `big_r_me = g^{k_me}` and `w_me = a_me * k_me + b_me` to all other participants. Once these values are sent, the nonce `k` is permanently committed — it cannot be "un-broadcast." [2](#0-1) 

The same pattern holds for FROST in `src/frost/mod.rs`, where `PresignOutput<C>` is also `Clone` and nonces are committed via `commit(&signing_share, &mut rng)` and broadcast before signing begins: [3](#0-2) [4](#0-3) 

**Phase 2 — Signing can fail:**

In `src/ecdsa/robust_ecdsa/sign.rs`, non-coordinator participants compute and send their signature share `s_me` to the coordinator in a single round: [5](#0-4) 

A malicious coordinator can receive these shares and then simply not complete the protocol (crash, drop messages, or return a `ProtocolError`). The honest participants have already irrevocably sent their nonce-derived shares. The signing protocol returns an error, but the presignature's nonce `k` is now "spent" from a security standpoint.

**No invalidation mechanism:**

The library provides no mechanism to mark a `PresignOutput` as consumed. Because it is `Clone` and `Serialize`, an orchestration layer will naturally retry the failed signing session by cloning or re-deserializing the same `PresignOutput` and calling `rerandomize_presign` again with a new `entropy`/`tweak`. The library's own security documentation explicitly acknowledges the consequence: [6](#0-5) 

> "a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."
> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."

The `sign` function in `src/ecdsa/robust_ecdsa/sign.rs` enforces `N1 = N2 = 2t+1` and `msg_hash ≠ 0`, but it performs **no check** that the supplied `RerandomizedPresignOutput` was not derived from a previously-used `PresignOutput`: [7](#0-6) 

---

### Impact Explanation

A malicious coordinator who causes signing to fail after collecting participant shares, then triggers a retry with the same presignature (different `msg_hash` or `tweak`), obtains two sets of linearized signature shares `s_i` for the same underlying nonce `k`. The documentation confirms this is sufficient to reconstruct the aggregate secret key via standard ECDSA nonce-reuse algebra. This maps directly to:

> **Critical: Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, or nonce material.**

---

### Likelihood Explanation

The coordinator role is explicitly untrusted in the robust setting (the entire point of the robust scheme is to tolerate malicious participants). A malicious coordinator can deterministically trigger this condition by aborting after collecting shares. Any orchestration layer that retries a failed signing session — the natural and expected behavior — will reuse the presignature. The library provides no API-level guard against this.

---

### Recommendation

1. **Wrap `PresignOutput` in a single-use type** that consumes itself on the first call to `rerandomize_presign`, preventing cloning or re-use:
   ```rust
   pub struct PresignOutput { /* fields */ }
   // Remove Clone; provide only a consuming rerandomize method
   impl PresignOutput {
       pub fn rerandomize(self, args: &RerandomizationArguments) -> Result<RerandomizedPresignOutput, ProtocolError> { ... }
   }
   ```
2. **Remove `#[derive(Clone)]`** from both `PresignOutput` types (robust ECDSA and FROST) so that accidental reuse is a compile-time error rather than a runtime security failure.
3. **Document at the API level** (not only in a separate markdown file) that a `PresignOutput` must be discarded after any signing attempt, successful or not.

---

### Proof of Concept

**Attack scenario (robust ECDSA, malicious coordinator):**

1. Participants run `presign(...)` → each holds a `PresignOutput` with shared nonce commitment `big_r = g^k`.
2. Coordinator initiates signing session 1: participants call `rerandomize_presign(presign_out.clone(), args1)` and `sign(...)`, sending `s_me^{(1)}` to the coordinator.
3. Coordinator aborts (returns error / drops connection). Signing fails.
4. Orchestration layer retries: participants call `rerandomize_presign(presign_out.clone(), args2)` (different `entropy`/`tweak`) and `sign(...)`, sending `s_me^{(2)}` to the coordinator.
5. Coordinator now holds two sets of linearized shares `{s_i^{(1)}}` and `{s_i^{(2)}}` for the same underlying nonce `k` but different rerandomization scalars `delta_1`, `delta_2`. Using the algebraic relationship documented in `docs/ecdsa/robust_ecdsa/signing.md` lines 150–158, the coordinator recovers the aggregate secret key `x`.

The enabling code path is `PresignOutput` being `Clone` with no consumed-state tracking: [1](#0-0) [8](#0-7)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
