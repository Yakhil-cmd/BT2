### Summary

The factory enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on pool-admin fees through `setPoolAdminFees`, but the parallel admin path `setPoolBinAdditionalFees` carries no corresponding cap check. A pool admin can therefore set per-bin buy/sell fees to any value up to `uint16.max` (65 535 E6 â‰ˆ 6.55 %), silently exceeding whatever limit the factory owner intended.

### Finding Description

The factory owner's caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) are therefore only enforced on the base admin spread fee, not on the per-bin additive fees. The `uint16` type bounds `addFeeBuyE6`/`addFeeSellE6` to 65 535 (â‰ˆ 6.55 % in E6 units), but this bound is independent ofâ€”and can exceedâ€”whatever `maxAdminSpreadFeeE6` the factory owner has configured.

### Impact Explanation

The factory owner uses `maxAdminSpreadFeeE6` to limit how much a pool admin can charge traders. If the owner sets `maxAdminSpreadFeeE6 = 0` (zero admin spread fee) or any value below 65 535, the pool admin can still impose up to 6.55 % additional fee per bin via `setPoolBinAdditionalFees`. Traders pay more than the factory owner's intended ceiling; the excess accrues to LPs (including the pool admin if they hold LP shares). This is a direct loss of trader principal and an admin-boundary break: the pool admin exceeds the factory-owner-set cap through an alternative, unchecked code pathâ€”the exact bug class of the seeded report.

---

### Likelihood Explanation

Any pool admin can trigger this at will with a single transaction. No special preconditions are required beyond being the registered pool admin. The incentive is clear: a pool admin who is also an LP benefits from higher per-bin fees. Likelihood is high given the low barrier and direct financial incentive.

---

### Recommendation

Add cap enforcement inside `setPoolBinAdditionalFees` before forwarding to the pool, analogous to `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinFeeE6` cap that the factory owner can configure independently, and enforce it here.

---

### Proof of Concept

1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 5000` (0.5 %).
2. Pool is created; pool admin is `alice`.
3. `alice` calls `setPoolAdminFees(pool, 5001, 0)` â†’ reverts with `AdminFeeTooHigh`. Cap is enforced.
4. `alice` calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` â†’ **succeeds**. No cap check.
5. Every swap through bin 0 now pays an additional â‰ˆ 6.55 % fee on top of the oracle spread, far exceeding the 0.5 % limit the factory owner intended.
6. Factory owner's `maxAdminSpreadFeeE6` cap is completely bypassed via the alternative admin path.
