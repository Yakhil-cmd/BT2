### Title
Presignature Reuse Enables Secret Key Extraction by Malicious Coordinator — (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

### Summary

The `PresignOutput` type in both the robust and OT-based ECDSA schemes derives `Clone` and `Serialize/Deserialize`, and `rerandomize_presign` accepts it by shared reference (`&PresignOutput`). This means the library API provides no enforcement of the documented single-use requirement. A malicious coordinator — a role reachable by any participant without privileged access — can initiate two signing sessions against the same presignature with different message hashes, obtain two ECDSA signatures whose nonces are multiplicatively related by a known factor, and recover the aggregate secret key using a standard related-nonce attack.

### Finding Description

**Root cause — API does not consume or invalidate the presignature:**

In `src/ecdsa/robust_ecdsa/mod.rs`, `PresignOutput` is defined as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }
```

`rerandomize_presign` takes it by shared reference:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // shared ref — not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

The same pattern exists in `src/ecdsa/ot_based_ecdsa/mod.rs` for the OT-based scheme.

Because `PresignOutput` is `Clone` and `rerandomize_presign` borrows it, any caller can invoke `rerandomize_presign` on the same `PresignOutput` an arbitrary number of times with different `RerandomizationArguments` (different `msg_hash`, `entropy`, `tweak`). The library provides no session registry, no "used" flag, and no ownership transfer that would prevent this.

**Attack path — malicious coordinator:**

The coordinator role is one of the `2t+1` participants; it requires no privileged access. The signing protocol in `src/ecdsa/robust_ecdsa/sign.rs` routes all signature shares exclusively to the coordinator:

```rust
chan.send_private(wait_round, coordinator, &s_me)?;
```

A malicious coordinator M executes:

1. All `2t+1` participants complete presigning and hold their `PresignOutput` shares (same `big_r = R`).
2. M initiates **signing session 1**: broadcasts `RerandomizationArguments` with `(h₁, ε, ρ₁)`. Each honest participant calls `rerandomize_presign(&presign_out, &args1)` — producing rerandomized nonce `R' = δ₁·R` — and sends share `s_i^(1)` to M. M assembles final signature `(R', s₁)`.
3. M initiates **signing session 2**: broadcasts `RerandomizationArguments` with `(h₂, ε, ρ₂)` (same presignature, different message/entropy). Each honest participant calls `rerandomize_presign(&presign_out, &args2)` — producing rerandomized nonce `R'' = δ₂·R` — and sends share `s_i^(2)` to M. M assembles final signature `(R'', s₂)`.
4. M knows `δ₁` and `δ₂` (it chose the rerandomization arguments), so it knows `λ = δ₁/δ₂`. The two nonces satisfy `k'' = λ·k'` — a known multiplicative relationship.
5. M applies the standard related-nonce key recovery formula:

   `x = (λ·s₁·h₂ − s₂·h₁) / (s₂·r₁ − λ·s₁·r₂)`

   recovering the aggregate secret key `x`.

Honest participants have no mechanism to detect they are being asked to use the same presignature twice: the library tracks no presignature state, and the `RerandomizationArguments` differ between sessions so the `RerandomizedPresignOutput` values are distinct.

**Why the existing N₁ = N₂ = 2t+1 enforcement does not prevent this:**

The code in `src/ecdsa/robust_ecdsa/presign.rs` and `sign.rs` enforces `participants.len() == 2*max_malicious+1`, which prevents split-view attacks where *different subsets* sign different messages. However, it does not prevent the *same* `2t+1` participants from being asked to sign two different messages using the same presignature. The `e` field is also copied without rerandomization in `rerandomize_presign` (`e: presignature.e`), meaning `e_i` is identical across both sessions, further leaking nonce-related information to the coordinator.

### Impact Explanation

A malicious coordinator can extract the aggregate ECDSA secret key from two signing sessions that reuse the same presignature. This constitutes **Critical** impact: full extraction of the private signing key material, enabling the attacker to forge arbitrary signatures for any message under the compromised key.

### Likelihood Explanation

The coordinator role is assigned to one of the `2t+1` participants and requires no special privilege. Any participant who is selected as coordinator can execute this attack. The attack requires only two completed signing sessions — a realistic operational scenario — and no external resources, leaked credentials, or cryptographic breaks. The library API actively facilitates the attack by making `PresignOutput` cloneable and accepting it by shared reference.

### Recommendation

1. **Consume `PresignOutput` in `rerandomize_presign`**: Change the signature from `presignature: &PresignOutput` to `presignature: PresignOutput`. This forces the caller to move the value, making reuse require an explicit `.clone()` call that is visible in code review.

2. **Remove `Clone` from `PresignOutput`** (or gate it behind a `#[cfg(test)]` derive): Without `Clone`, the Rust type system enforces single-use at the call site.

3. **Apply the same fix to the OT-based scheme** in `src/ecdsa/ot_based_ecdsa/mod.rs`.

4. **Rerandomize the `e` field**: In `RerandomizedPresignOutput::rerandomize_presign` for the robust scheme, `e` is copied unchanged (`e: presignature.e`). It should be rerandomized consistently with the other fields to avoid leaking nonce-correlated information across sessions.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,

        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }

        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
