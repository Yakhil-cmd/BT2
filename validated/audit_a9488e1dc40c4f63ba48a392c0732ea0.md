### Title
Mandatory Reward Transfer in `unstake_action` Can Block Staker From Retrieving Principal Funds - (File: src/staking/staking.cairo)

### Summary
`unstake_action` mandatorily calls `send_rewards_to_staker`, which performs an ERC20 transfer to the staker's `reward_address`, before returning the staker's principal. If `reward_address` is a smart contract that reverts on token receipt, the entire `unstake_action` reverts. There is no emergency exit path to bypass this reward transfer, leaving the staker's principal permanently inaccessible.

### Finding Description
In `unstake_action` (`src/staking/staking.cairo:483`), the very first substantive step is a call to `send_rewards_to_staker` (line 495): [1](#0-0) 

`send_rewards_to_staker` unconditionally calls `checked_transfer` to the staker's `reward_address`: [2](#0-1) 

The `reward_address` is fully user-controlled. `change_reward_address` only validates that the address is not a registered token address — it accepts any arbitrary contract address: [3](#0-2) 

A staker may legitimately register a multisig wallet, DAO treasury, or other smart contract as their `reward_address`. If that contract later becomes non-functional (e.g., due to an upgrade, a bug, or self-destruction) and reverts on ERC20 receipt, every call to `unstake_action` will revert at the `checked_transfer` line. The staker's principal stake remains locked in the staking contract with no alternative withdrawal path.

The same failure also blocks `claim_rewards`, since it calls the same `send_rewards_to_staker` helper: [4](#0-3) 

The staker can recover by calling `change_reward_address` (which has no exit-window restriction) to point to a working address. However, if the staker has lost access to their staker key, no recovery is possible and the principal is permanently frozen. There is no protocol-level emergency exit that separates principal withdrawal from reward delivery.

### Impact Explanation
The staker's entire principal stake is frozen inside the staking contract. The staker cannot call `unstake_action` to retrieve it, and cannot call `claim_rewards` to drain the pending reward balance to zero first (which would otherwise allow a zero-amount transfer to succeed). The only self-help path — `change_reward_address` — requires the staker's private key. Loss of that key, or failure to discover the workaround, results in permanent loss of the principal. This matches **Temporary (potentially permanent) freezing of funds**.

### Likelihood Explanation
Stakers operating as validators are likely to use smart-contract wallets (multisigs, DAO treasuries, protocol-owned accounts) as their `reward_address`. Any such contract that is later upgraded, paused, or broken causes this freeze. The staker has no on-chain signal that their `reward_address` has become non-functional until `unstake_action` reverts. Likelihood is low-to-medium given the prevalence of smart-contract reward addresses in production validator setups.

### Recommendation
Decouple principal withdrawal from reward delivery in `unstake_action`. Two concrete options:

1. **Skip reward transfer on failure**: Wrap the `send_rewards_to_staker` call in a safe-dispatch pattern; if it fails, leave `unclaimed_rewards_own` intact and allow the principal withdrawal to proceed.
2. **Add an emergency exit entrypoint**: Provide a separate `unstake_action_no_rewards` (or equivalent) that skips the reward transfer entirely, allowing the staker to recover their principal even when the `reward_address` is broken. Unclaimed rewards can remain claimable once the staker fixes their `reward_address`.

### Proof of Concept

1. Staker registers with `reward_address = <multisig_contract>`.
2. Staker earns rewards via attestations; `unclaimed_rewards_own > 0`.
3. `<multisig_contract>` is upgraded and now reverts on ERC20 `transfer`.
4. Staker calls `unstake_intent` — succeeds (no token transfer involved).
5. Staker calls `claim_rewards` to drain rewards first — **reverts** at `send_rewards_to_staker` → `checked_transfer(recipient: <multisig_contract>, ...)`.
6. Staker calls `unstake_action` — **reverts** at the same point.
7. Principal funds remain locked in the staking contract.
8. Recovery requires calling `change_reward_address` with the staker key; if the key is unavailable, funds are permanently frozen.

        fn unstake_action(ref self: ContractState, staker_address: ContractAddress) -> Amount {
            // Prerequisites and asserts.
            self.general_prerequisites();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let unstake_time = staker_info
                .unstake_time
                .expect_with_err(Error::MISSING_UNSTAKE_INTENT);
            assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Send rewards to staker's reward address.
            // It must be part of this function's flow because staker_info is about to be erased.
            let token_dispatcher = strk_token_dispatcher();
            self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
            // Update staker info to storage (it will be erased later).
            // This is done here to avoid re-entrancy.
            self.write_staker_info(:staker_address, :staker_info);

            let staker_amount = self.get_own_balance(:staker_address).to_strk_native_amount();
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            self.remove_staker(:staker_address, :staker_info, :staker_pool_info);

            // Return stake to staker.
            token_dispatcher
                .checked_transfer(recipient: staker_address, amount: staker_amount.into());
            // Return delegated stake to pools and zero their balances.
            self
                .transfer_to_pools_when_unstake(
                    :staker_address, staker_pool_info: staker_pool_info.as_non_mut(),
                );
            // Clear staker pools.
            staker_pool_info.pools.clear();
            staker_amount
        }

pub(crate) fn claim_from_reward_supplier(
    reward_supplier_dispatcher: IRewardSupplierDispatcher,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
) {
    let staking_contract = get_contract_address();
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}
