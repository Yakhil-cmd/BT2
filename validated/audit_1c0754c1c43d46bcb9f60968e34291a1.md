### Title
Unchecked Precompile Call Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Silent Token Burn Without Exit — (`File: etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens **before** calling the exit precompile, and then silently ignore the precompile call's return value. If the precompile call fails for any reason, the burn is committed but no NEAR-side exit promise is ever scheduled. The user's ERC-20 tokens are permanently destroyed while the corresponding NEP-141 tokens remain locked inside the Aurora contract forever.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
// EvmErc20.sol – withdrawToNear (lines 53–63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn committed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert if call returns 0
    }
}
``` [1](#0-0) 

The identical pattern appears in `withdrawToEthereum`: [2](#0-1) 

And in `EvmErc20V2.sol` for both functions: [3](#0-2) 

The `ExitToNear` precompile can return failure (`ExitError`) in several reachable paths:

- **`ERR_INVALID_IN_DELEGATE`**: `context.address != exit_to_near::ADDRESS.raw()` — triggered if the call is made via `DELEGATECALL`.
- **`ERR_TARGET_TOKEN_NOT_FOUND`**: `get_nep141_from_erc20` fails if the ERC-20 address is not in the NEP-141 registry.
- **Input parsing failure**: `ExitToNearParams::try_from(input)?` fails on malformed input (e.g., an empty or oversized `recipient` bytes argument). [4](#0-3) [5](#0-4) 

In the EVM, a failed `call` to a precompile returns `0` in the assembly result variable. Because `res` is assigned but never inspected and no `require(res != 0)` follows, the Solidity function returns normally. The `_burn` state change is **not rolled back**.

---

### Impact Explanation

When the precompile call fails silently:

1. The user's ERC-20 token balance is permanently reduced (burned).
2. No NEAR-side `ft_transfer` or `withdraw` promise is ever scheduled.
3. The NEP-141 tokens that were backing those ERC-20 tokens remain locked inside the Aurora contract with no corresponding ERC-20 supply to redeem them.

This is a **permanent freezing of user funds**: the ERC-20 tokens are gone and the NEP-141 tokens are irrecoverably locked. The bridge accounting becomes permanently insolvent for the affected amount.

---

### Likelihood Explanation

The most directly reachable trigger is passing an empty or invalid `recipient` bytes argument to `withdrawToNear`. The `ExitToNear` precompile will fail during `ExitToNearParams::try_from(input)` (invalid NEAR account ID), returning `ExitError`. This is callable by any unprivileged EVM user who holds ERC-20 tokens. A secondary path is any contract that calls `withdrawToNear` via `DELEGATECALL`, which triggers `ERR_INVALID_IN_DELEGATE`. Both paths require no special privileges.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero, mirroring the fix applied in the referenced external report:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. This ensures the burn is atomically rolled back if the precompile call fails, preserving correct bridge accounting.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) on Aurora and register it against a NEP-141 token.
2. Mint ERC-20 tokens to address `A`.
3. From address `A`, call `withdrawToNear("", amount)` — passing an empty bytes recipient.
4. The `_burn(A, amount)` executes and commits.
5. The precompile at `0xe921...` receives input `\x01 || amount_b || ""`. `ExitToNearParams::try_from` fails on the empty/invalid account ID, returning `ExitError`. The assembly `call` returns `0`.
6. `res` is `0` but is never checked. The function returns without reverting.
7. Observe: `balanceOf(A)` has decreased by `amount`; no NEP-141 `ft_transfer` promise was scheduled; the NEP-141 tokens remain locked in Aurora permanently. [1](#0-0) [6](#0-5) [7](#0-6)
