### Title
Gateway Stateful Validator Uses `l2_gas_price` Instead of `next_l2_gas_price` for Resource Bounds Threshold — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`validate_resource_bounds` reads `gas_prices.strk_gas_prices.l2_gas_price` (the **current** block's L2 gas price) to compute the admission threshold, but the transaction will be executed in the **next** block whose price is `next_l2_gas_price`. The same wrong price propagates into `run_validate_entry_point`, where the block context is built with the current block's gas prices even though `block_number` is advanced to `unchecked_next()`. A developer TODO comment in the source explicitly acknowledges the error.

### Finding Description

In `validate_resource_bounds`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;          // ← current block's price, not next block's
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,   // ← wrong field passed here
)?;
``` [1](#0-0) 

The `BlockInfo` struct returned by `get_block_info()` carries only `gas_prices` (current block prices); it does not expose `next_l2_gas_price`. That field lives in `BlockHeaderWithoutHash` and `StorageBlockHeader` but is never surfaced through the gateway's state-reader API. [2](#0-1) [3](#0-2) 

The same wrong price is used a second time in `run_validate_entry_point`, which increments `block_number` to the next block but leaves the gas prices unchanged:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
// gas_prices are NOT updated to next_l2_gas_price
let block_context = BlockContext::new(block_info, ...);
``` [4](#0-3) 

Consequently, the blockifier's `check_fee_bounds` inside `perform_pre_validation_stage` also evaluates against the current block's `l2_gas_price`, not the next block's price that will govern actual execution. [5](#0-4) 

### Impact Explanation

**Case 1 — price rising (`next_l2_gas_price > l2_gas_price`):**  
A transaction with `max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` passes both the gateway's `validate_resource_bounds` check and the blockifier's `check_fee_bounds` inside `run_validate_entry_point` (both use the lower current price). The transaction is admitted to the mempool. When the batcher executes it against the actual next block context (which uses `next_l2_gas_price`), `check_fee_bounds` fires `MaxGasPriceTooLow` and the transaction is dropped — **invalid transaction admitted before sequencing**.

**Case 2 — price falling (`next_l2_gas_price < l2_gas_price`):**  
A transaction with `max_price_per_unit` in the range `[next_l2_gas_price, l2_gas_price)` is rejected by the gateway even though it would satisfy the actual next block's price — **valid transaction rejected before sequencing**.

Both cases match the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The fee market algorithm (`MIN_GAS_PRICE_INCREASE_DENOMINATOR = 333`) allows ~0.3 % price movement per block. [6](#0-5) 

During sustained high-throughput periods the gap between `l2_gas_price` and `next_l2_gas_price` accumulates. Both values are publicly readable from block headers, so an attacker can trivially craft transactions that land in the gap. The developer TODO comment confirms the discrepancy is known but unresolved.

### Recommendation

1. Extend `GatewayFixedBlockStateReader` (and its `get_block_info` implementation) to also return `next_l2_gas_price` from the stored `BlockHeaderWithoutHash`.
2. In `validate_resource_bounds`, replace `gas_prices.strk_gas_prices.l2_gas_price` with the retrieved `next_l2_gas_price`.
3. In `run_validate_entry_point`, set `block_info.gas_prices.strk_gas_prices.l2_gas_price` (and the corresponding wei price) to `next_l2_gas_price` before constructing the `BlockContext`, so the blockifier's `check_fee_bounds` evaluates against the price that will govern actual execution.

### Proof of Concept

1. Read the latest committed block header; note `l2_gas_price.price_in_fri = P` and `next_l2_gas_price = P'` where `P' > P` (network under load).
2. Submit an invoke V3 transaction with `l2_gas.max_price_per_unit = P + 1` (above current threshold, below next-block price).
3. Gateway `validate_resource_bounds` computes threshold as `min_gas_price_percentage% × P`; the transaction passes.
4. `run_validate_entry_point` builds a block context with `l2_gas_price = P`; blockifier `check_fee_bounds` passes.
5. Transaction is admitted to the mempool.
6. Batcher builds the next block with `l2_gas_price = P'`; blockifier `check_fee_bounds` evaluates `P + 1 >= P'` → **false** → `MaxGasPriceTooLow` → transaction dropped from the block.

The structural analog to the oracle report is exact: `l2_gas_price` (the current-block field) is passed to `validate_tx_l2_gas_price_within_threshold` where `next_l2_gas_price` (the next-block field) should be passed, mirroring the `oracle` vs `oracle_stats` substitution error. [7](#0-6)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
