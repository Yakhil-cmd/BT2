### Title
`MsgTurnBridge` Handler Is a No-Op Stub — Bridge Circuit Breaker Permanently Broken - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` gRPC message handler in `x/cronos/keeper/msg_server.go` is an empty stub that immediately returns `nil, nil` without performing any state change or permission check. The Gravity Bridge circuit breaker is therefore permanently non-functional: the bridge can never be disabled via `MsgTurnBridge`, and any unprivileged address can call the message and receive a success response.

### Finding Description

`MsgTurnBridge` is the designated emergency circuit breaker for the Gravity Bridge. ADR-001 explicitly states the bridge "can be controlled through the governance parameter `bridge_active`", and ADR-009 defines `CanTurnBridge` as one of the two core restricted permissions in the Cronos permission system, specifically for "disabling the bridge module in case of emergency."

The full implementation of the handler is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly checks permissions and writes state:

```go
func (k msgServer) UpdateTokenMapping(...) (...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, ...)
    }
    if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil { ... }
    return &types.MsgUpdateTokenMappingResponse{}, nil
}
``` [2](#0-1) 

The `TurnBridge` handler has two distinct failures:

1. **No permission check** — `CanTurnBridge` is defined and the permission system is wired, but the handler never calls `HasPermission(ctx, ..., CanTurnBridge)`. Any unprivileged address can submit `MsgTurnBridge` and receive a success response. [3](#0-2) 

2. **No state write** — There is no `bridge_active` key anywhere in the keeper store (confirmed by absence of any `BridgeActive`/`SetBridgeActive` implementation). The handler writes nothing, so the bridge state is permanently fixed as active. [1](#0-0) 

### Impact Explanation

The Gravity Bridge circuit breaker is permanently non-functional. In the event of a live bridge exploit — e.g., an attacker draining funds via unauthorized `send_to_cosmos` or batch manipulation — the admin and all `CanTurnBridge`-permissioned addresses are unable to halt bridge flows. The bridge remains active indefinitely regardless of how many `MsgTurnBridge{Enable: false}` transactions are submitted. This is a **permanent inability for honest operators to stop bridge/conversion flows**, matching the High impact criterion.

Additionally, the missing permission check means any unprivileged address can submit `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` authorization check entirely. [4](#0-3) 

### Likelihood Explanation

The `MsgTurnBridge` message is fully wired: it has a proto definition, a CLI command (`CmdTurnBridge`), a `ValidateBasic` implementation, and integration tests that call it and assert `code == 0`. The broken handler is invisible to callers — the transaction succeeds, fees are paid, but no state changes. Any operator attempting to use the circuit breaker during an active exploit would be silently misled into believing the bridge was disabled. [5](#0-4) [6](#0-5) 

### Recommendation

Implement `TurnBridge` to:
1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` on failure.
2. Write a `bridge_active` boolean to the KV store (add a new store key under `x/cronos/types/keys.go`).
3. Return `&types.MsgTurnBridgeResponse{}` (not `nil`) on success.
4. All bridge-processing paths (Gravity hooks, EVM handlers) must read this flag and reject operations when the bridge is inactive.

### Proof of Concept

```
# Any address — no CanTurnBridge permission required
cronosd tx cronos turn-bridge false --from=<any_unprivileged_key> -y

# Response: code: 0 (success) — but bridge_active state never written
# Bridge remains permanently active; subsequent bridge transfers proceed normally
```

The integration test at `integration_tests/test_gravity.py:661-665` already demonstrates the expected behavior (admin turns bridge off, subsequent transfers fail), but because the handler is a no-op, the bridge-disabled assertions in that test would fail against this implementation. [7](#0-6)

// CmdTurnBridge returns a CLI command handler for enable or disable the bridge
func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{
		Use:   "turn-bridge [true/false]",
		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}
