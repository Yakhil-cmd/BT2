### Title
Monotonically Growing `stakers` Vec Causes Unbounded Gas Consumption in `get_stakers` - (`File: src/staking/staking.cairo`)

---

### Summary

The `stakers` storage vector in `staking.cairo` is append-only: every new staker address is pushed into it on `stake()`, but addresses are **never removed** when a staker calls `unstake_intent` / `unstake_action`. The `get_stakers` consensus function iterates over the **full range** of this vector on every call, performing multiple storage reads per entry. As the vector grows over the protocol's lifetime through normal staker churn, `get_stakers` will eventually exceed the Starknet gas limit, permanently breaking the consensus validator-set query.

---

### Finding Description

**Root cause — storage declaration:**

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

**Growth path — `stake()` unconditionally pushes every new staker:**

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [2](#0-1) 

There is no corresponding pop or removal anywhere in the unstake flow. The `assert_staker_address_not_reused` guard at line 303 prevents the same address from re-staking, so there is no duplicate-counting issue, but it also means **every unique address that ever staked occupies a permanent slot in the vector**. [3](#0-2) 

**Consumption path — `get_stakers` iterates the full range:**

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    let staking_power = self
        .get_staker_staking_power_at_epoch(...);
    ...
    let public_key = self.get_public_key_at_epoch(...);
    let peer_id   = self.get_peer_id_at_epoch(...);
    stakers.append(...);
}
``` [4](#0-3) 

For every entry — including long-since-unstaked stakers — the loop performs at minimum one storage read (`staker_address_ptr.read()`) plus the `is_staker_active` check. For active stakers it performs several additional storage reads (`get_staker_staking_power_at_epoch`, `get_public_key_at_epoch`, `get_peer_id_at_epoch`). The gas cost scales linearly with the total number of addresses that have **ever** staked, not just those currently active.

The `IStakingConsensus` interface documents `get_stakers` as publicly callable by any address: [5](#0-4) 

The spec confirms access is unrestricted: [6](#0-5) 

---

### Impact Explanation

`get_stakers` is the consensus-layer entry point used to determine the validator set for each epoch. If the function reverts due to gas exhaustion, the consensus layer cannot retrieve the validator set, which disrupts the liveness of the Starknet network. This matches the allowed impact: **Unbounded gas consumption / Griefing with damage to the protocol**.

---

### Likelihood Explanation

Each unique staker address that stakes and then fully exits leaves a permanent dead entry in the vector. Over the protocol's lifetime, normal validator churn (validators joining and leaving) accumulates these entries organically. An adversary can accelerate this by deploying many addresses, each staking the minimum amount, waiting out the exit window, and withdrawing — each cycle permanently inflates the vector at the cost of only transaction fees and the temporary capital lock-up. The attack is irreversible without a contract upgrade.

---

### Recommendation

1. **Remove stakers from the vector on full exit.** In `unstake_action`, swap the exiting staker's slot with the last element and pop the vector (the same swap-and-pop pattern recommended in the original report for `allWhitelistedTokens`).

2. **Alternatively**, maintain a separate `active_staker_count` and a mapping from staker address to its index in the vector, enabling O(1) removal.

3. **Add a cap** on the maximum number of stakers that can ever be registered, consistent with the note already present for `add_token`: [7](#0-6) 

---

### Proof of Concept

1. Deploy the staking contract with `min_stake = M`.
2. Create `N` addresses, each funded with `M` STRK.
3. For each address: call `stake(...)` → `unstake_intent()` → advance time past `exit_wait_window` → `unstake_action(staker_address)`.
4. After step 3, the `stakers` vector has length `N` but zero active stakers.
5. Call `get_stakers(epoch_id: current_epoch)`.
6. Observe that the loop iterates `N` times, each time reading storage for the dead address. As `N` grows, the call approaches and eventually exceeds the Starknet block gas limit, causing a permanent revert.

The test at `src/flow_test/flows.cairo` lines 5898–5903 already demonstrates that unstaked stakers remain in the vector:

```cairo
let actual_stakers = system.staking.get_stakers();
assert!(actual_stakers.len() == 4);  // includes staker_in_action who already fully exited
``` [8](#0-7)
