### Title
`SwapAllowlistExtension` checks router address instead of actual user, allowing any caller to bypass per-pool swap allowlist via `MetricOmmSimpleRouter` — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary

`SwapAllowlistExtension` is designed to gate swaps by swapper identity. When swaps are routed through `MetricOmmSimpleRouter`, the pool's `swap()` receives `msg.sender = router`, and passes the router address as `sender` to `beforeSwap`. The extension then checks `allowedSwapper[pool][router]`, never inspecting the actual EOA that initiated the call. Any user blocked by the allowlist can bypass it by routing through the router.

### Finding Description

`MetricOmmPool.swap()` passes its own `msg.sender` as the `sender` argument to `_beforeSwap`: [1](#0-0) 

`_beforeSwap` forwards that value unchanged to the extension: [2](#0-1) 

`SwapAllowlistExtension.beforeSwap` then checks `allowedSwapper[msg.sender][sender]`, where `msg.sender` is the pool and `sender` is whatever address called `pool.swap()`: [3](#0-2) 

When `exactOutputSingle` is called on the router, the router calls `pool.swap()`, so `sender` = router address: [4](#0-3) 

The result: if the pool admin adds the router to the allowlist (a natural configuration for a pool that accepts router-based swaps), every EOA — including explicitly blocked ones — can swap freely through the router. The actual `msg.sender` of `exactOutputSingle` is never inspected.

This applies equally to `exactInputSingle`, `exactInput`, and `exactOutput`.

### Impact Explanation

A pool admin who configures a curated pool with `SwapAllowlistExtension` to restrict access to specific counterparties (e.g., KYC'd addresses, institutional traders) and adds the router to the allowlist inadvertently grants unrestricted swap access to all users. Any blocked EOA can drain token0 or token1 from the pool's liquidity by routing through `MetricOmmSimpleRouter`. The pool's accounting remains internally consistent (correct deltas), but the access-control invariant the admin intended is completely broken, constituting an admin-boundary bypass by an unprivileged path.

### Likelihood Explanation

The scenario requires a pool configured with `SwapAllowlistExtension` and the router address in the allowlist. Adding the router to the allowlist is the expected operational step for any pool that wants to support router-based trading while still restricting direct callers — making this a realistic and likely misconfiguration. No special privileges or unusual conditions are needed beyond calling `exactOutputSingle` as a blocked EOA.

### Recommendation

The extension must verify the true initiator of the swap, not the immediate caller of `pool.swap()`. Two approaches:

1. **Pass the original caller through the router**: Have `MetricOmmSimpleRouter` encode `msg.sender` into `extensionData` and have `SwapAllowlistExtension.beforeSwap` decode and verify it. This requires a trust relationship between the extension and the router.

2. **Check `recipient` instead of `sender`**: If the pool's design guarantees that `recipient` is the actual beneficiary, the extension could check `recipient`. However, this is semantically different and may not match the intended gating.

3. **Disallow router-mediated access for allowlisted pools**: Document that `SwapAllowlistExtension` is incompatible with router usage and enforce this at the extension or factory level.

### Proof of Concept

```
1. Deploy pool with SwapAllowlistExtension as beforeSwap hook.
2. Pool admin calls setAllowedToSwap(pool, router, true) — allows the router.
3. Pool admin does NOT add blockedEOA to the allowlist.
4. blockedEOA calls MetricOmmSimpleRouter.exactOutputSingle(..., pool=pool, ...).
5. Router calls pool.swap() with msg.sender=router.
6. Pool calls _beforeSwap(sender=router, ...).
7. Extension checks allowedSwapper[pool][router] == true → passes.
8. Swap executes. blockedEOA receives token output despite being blocked.
```
