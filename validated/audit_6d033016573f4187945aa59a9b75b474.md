### Title
Gateway L2 Gas Price Admission Check Uses Current Block's `l2_gas_price` Instead of `next_l2_gas_price`, Causing Wrong Admission Decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful admission check for V3 transaction resource bounds compares the transaction's `max_price_per_unit` against the **current** (previous) block's `l2_gas_price`. However, the transaction will be executed in the **next** block, which uses `next_l2_gas_price` — a distinct field computed via EIP-1559 adjustment. The code itself contains a TODO acknowledging this exact mistake. The result is that the gateway accepts transactions that will fail at blockifier execution (when the next-block price is higher) and rejects transactions that would succeed (when the next-block price is lower).

---

### Finding Description

In `validate_resource_bounds`, the gateway reads the previous block's `BlockInfo` and extracts `gas_prices.strk_gas_prices.l2_gas_price` as the reference price for the admission threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is then computed as `min_gas_price_percentage / 100 * previous_block_l2_gas_price` and the transaction is rejected if `tx.l2_gas.max_price_per_unit < threshold`. [2](#0-1) 

The block header stores **two distinct** L2 gas price fields:

- `l2_gas_price`: the price that was used for transactions **in the current block**
- `next_l2_gas_price`: the EIP-1559-adjusted price that will be used for transactions **in the next block** [3](#0-2) [4](#0-3) 

The `next_l2_gas_price` is computed by `calculate_next_base_gas_price` using EIP-1559 logic, which can adjust the price by up to `1 / gas_price_max_change_denominator` per block (e.g., 12.5% per block if the denominator is 8): [5](#0-4) 

The gateway validates incoming transactions against `l2_gas_price` (the old price), but the blockifier's `check_fee_bounds` — which runs during actual execution — validates against the **new block's** gas price. These two prices diverge whenever blocks are not exactly at the gas target. [6](#0-5) 

---

### Impact Explanation

**Case 1 — Invalid transactions admitted (price rising):**
When the previous block was above the gas target, `next_l2_gas_price > l2_gas_price`. A transaction with `max_price_per_unit` satisfying `l2_gas_price ≤ max_price < next_l2_gas_price` passes the gateway check but fails `check_fee_bounds` in the blockifier with `MaxGasPriceTooLow`. The gateway has admitted a transaction that will be rejected at execution, polluting the mempool.

**Case 2 — Valid transactions rejected (price falling):**
When the previous block was below the gas target, `next_l2_gas_price < l2_gas_price`. A transaction with `max_price_per_unit` satisfying `next_l2_gas_price ≤ max_price < l2_gas_price` is rejected by the gateway even though it would succeed at execution. A legitimate user transaction is incorrectly denied admission.

Both cases match: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The bug is limited to V3 (`AllResources`) transactions; legacy `L1Gas` transactions skip this check entirely. [7](#0-6) 

---

### Likelihood Explanation

The EIP-1559 adjustment fires on every block that deviates from the gas target. In any active network, blocks are routinely above or below target, so `next_l2_gas_price ≠ l2_gas_price` is the common case, not the exception. Any user submitting a V3 transaction near the price boundary triggers the wrong decision. No special privileges are required.

---

### Recommendation

Replace the read of `gas_prices.strk_gas_prices.l2_gas_price` with a read of `next_l2_gas_price` from the block header. The `GatewayFixedBlockStateReader` trait (or its `get_block_info` return type) must be extended to expose `next_l2_gas_price`, since `BlockInfo` currently does not carry that field. The threshold comparison should then be:

```rust
// Correct: use the price that will apply to the next block.
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_next_l2_gas_price()   // new method needed
    .await?;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    next_block_l2_gas_price,
)?;
```

---

### Proof of Concept

1. Suppose the latest committed block has `l2_gas_price = 100` and `next_l2_gas_price = 113` (block was above gas target, ~12.5% increase).
2. Attacker (or ordinary user) submits a V3 invoke transaction with `l2_gas.max_price_per_unit = 105`.
3. `validate_resource_bounds` computes threshold = `100 * 100% = 100`. Since `105 ≥ 100`, the transaction **passes** gateway admission and enters the mempool.
4. The batcher picks the transaction for the next block, which uses `l2_gas_price = 113`.
5. `check_fee_bounds` in the blockifier computes `actual_gas_price = 113 > max_price = 105` → `MaxGasPriceTooLow` error. The transaction is rejected at execution.
6. The mempool slot was consumed by a transaction that could never succeed.

Conversely, with `l2_gas_price = 100` and `next_l2_gas_price = 88`, a transaction with `max_price = 90` is **rejected** by the gateway (`90 < 100`) even though it would succeed in the next block (`90 ≥ 88`). [8](#0-7) [9](#0-8)
