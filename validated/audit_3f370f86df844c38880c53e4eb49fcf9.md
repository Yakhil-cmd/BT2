### Title
Missing `isPaused()` Guard on `CadenceOwnedAccount.deposit()` Allows FLOW Tokens to Be Trapped During EVM Pause - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

The `EVM.CadenceOwnedAccount` resource exposes a `deposit()` function that is `access(all)` and carries no `isPaused()` precondition. Every other state-mutating function on the same resource — `withdraw`, `deploy`, `call`, `callWithSigAndArgs`, `depositNFT`, `depositTokens`, `withdrawNFT`, `withdrawTokens` — enforces `!EVM.isPaused()`. When the EVM is paused, any unprivileged user can still call `deposit()` to move native FLOW tokens into their COA's EVM balance, but cannot call `withdraw()` to recover them, because `withdraw()` is blocked by the pause guard. This creates an asymmetric one-way valve: tokens flow in freely but cannot flow out, trapping user funds for the duration of the pause.

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, the `CadenceOwnedAccount` resource defines the following `deposit` function:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)
}
``` [1](#0-0) 

There is no `pre { !EVM.isPaused() }` block. By contrast, every other state-mutating function on the same resource carries the guard:

- `withdraw` (line 587–590): [2](#0-1) 
- `deploy` (line 617–625): [3](#0-2) 
- `call` (line 636–645): [4](#0-3) 
- `depositNFT` (line 734–741): [5](#0-4) 
- `depositTokens` (line 773–780): [6](#0-5) 
- `withdrawNFT` (line 755–763): [7](#0-6) 
- `withdrawTokens` (line 787–795): [8](#0-7) 

The fact that `depositNFT` and `depositTokens` both carry the `isPaused()` guard demonstrates that the protocol designers intended to block all inbound asset flows during a pause — the native FLOW `deposit` path was simply missed. [9](#0-8) 

**Exploit path:**

1. The EVM pause is activated (e.g., in response to an incident).
2. An unprivileged user holds a `CadenceOwnedAccount` resource and a `FlowToken.Vault`.
3. The user (or any third party with a reference to the COA's public `deposit` capability) calls `deposit(from: <-vault)`. The call succeeds because there is no `isPaused()` check.
4. The FLOW tokens are now credited to the COA's EVM-side balance via `InternalEVM`.
5. The user attempts `withdraw(balance: ...)`. The precondition `!EVM.isPaused()` fires and the transaction reverts.
6. The FLOW tokens remain locked inside the COA's EVM balance for the entire duration of the pause.

### Impact Explanation

FLOW tokens deposited into a COA during an EVM pause period are inaccessible to their owner. The pause duration is not bounded by the protocol; it is controlled by the service account administrator and could last indefinitely. During that window, the deposited tokens cannot be recovered via `withdraw`, `call`, or any other COA method, because all outbound paths enforce `isPaused()`. The result is a permanent (or indefinitely long) freeze of user-owned FLOW tokens — a direct cross-VM asset loss scenario. [2](#0-1) 

### Likelihood Explanation

The `deposit` function is `access(all)`, meaning any Cadence transaction can invoke it on any COA for which a public capability exists. No privileged key or special role is required. The only prerequisite is that the EVM is paused, which is an explicitly supported operational state of the protocol. Users who are unaware of the pause (e.g., automated bridging scripts, wallets) may continue depositing FLOW into COAs while the pause is active, inadvertently trapping their funds. [1](#0-0) 

### Recommendation

Add the same `isPaused()` precondition to `CadenceOwnedAccount.deposit()` that is already present on every other state-mutating function:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    self.address().deposit(from: <-from)
}
```

This is consistent with the treatment of `depositNFT` and `depositTokens`, which already carry the guard. [10](#0-9) 

### Proof of Concept

```cadence
import EVM from <EVM_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>

// Precondition: EVM is paused by the service account.
// The attacker/victim holds a COA and a FlowToken.Vault.

transaction(coaStoragePath: StoragePath, amount: UFix64) {
    prepare(signer: auth(BorrowValue) &Account) {
        // Borrow the COA resource
        let coa = signer.storage.borrow<&EVM.CadenceOwnedAccount>(from: coaStoragePath)
            ?? panic("No COA found")

        // Borrow the FLOW vault
        let vault = signer.storage.borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(
            from: /storage/flowTokenVault
        ) ?? panic("No vault found")

        // Step 1: deposit succeeds — no isPaused() guard
        coa.deposit(from: <-vault.withdraw(amount: amount))

        // Step 2: withdraw reverts — isPaused() guard fires
        // The following line will panic: "EVM operations are temporarily paused"
        let recovered <- coa.withdraw(balance: EVM.Balance(attoflow: UInt(amount * 1_000_000_000_000_000_000.0)))
        destroy recovered
    }
}
```

After step 1 the FLOW tokens are credited to the COA's EVM balance. Step 2 reverts, leaving the tokens permanently inaccessible for the duration of the pause. The root cause is the missing `isPaused()` precondition in `deposit()` at line 562–565 of `fvm/evm/stdlib/contract.cdc`. [1](#0-0)
