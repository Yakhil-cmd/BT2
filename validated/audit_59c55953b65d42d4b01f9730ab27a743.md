### Title
L2 `base_mint_amount` vs L1 `TOKENS_PER_MINT_REQUEST` Message Payload Mismatch Permanently Freezes Unclaimed Yield - (`src/reward_supplier/reward_supplier.cairo` / `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The L2 `RewardSupplier` sends L2→L1 mint-request messages whose single payload element is the **configurable** storage value `base_mint_amount`. The L1 `RewardSupplier` matches those messages against a **hardcoded** constant `TOKENS_PER_MINT_REQUEST = 1_300_000 * 10**18`. No on-chain enforcement guarantees the two values are equal. If they diverge—through a deployment misconfiguration or a future L1 upgrade—every mint-request message sent by L2 will hash to a value the L1 never finds, so L1 never mints, never bridges tokens back, and the reward supplier is permanently starved of funds, freezing all unclaimed yield.

---

### Finding Description

**L2 message construction** — `send_mint_request_to_l1_reward_supplier` in `src/reward_supplier/reward_supplier.cairo`:

```cairo

    let payload: Span<felt252> = array![self.base_mint_amount.read().into()].span();

    send_message_to_l1_syscall(:to_address, :payload).unwrap_syscall();

```

`base_mint_amount` is a constructor parameter with no setter and no validation against any L1 constant. [1](#0-0) 

**L1 message matching** — `requiredMinting` and `tick` in `L1/starkware/solidity/stake/RewardSupplier.sol`:

```solidity

        messagingContract().l2ToL1Messages(msgHash), ...

```

The L1 computes the message hash using its own hardcoded constant, not the value the L2 actually placed in the payload. [2](#0-1) [3](#0-2) 

The Starknet messaging layer hashes the payload verbatim; if `base_mint_amount ≠ TOKENS_PER_MINT_REQUEST`, the two hashes will never match and `l2ToL1Messages(msgHash)` will always return 0. [4](#0-3) 

`base_mint_amount` is written once at construction and there is no L2-side setter to correct it later. [5](#0-4) 

---

### Impact Explanation

When the payload values diverge:

1. `request_funds` emits L2→L1 messages with payload `[base_mint_amount]`.
2. L1 `tick()` looks for messages with payload `[TOKENS_PER_MINT_REQUEST]`; finds none; mints nothing; bridges nothing.
3. `l1_pending_requested_amount` on L2 grows without bound while the actual token balance stays at zero.
4. `claim_rewards` will eventually revert because `unclaimed_rewards > balance`, and there is no recovery path.

Result: **permanent freezing of all unclaimed staking yield** — a High-severity impact under the allowed scope. [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

The mismatch can arise in two realistic ways:

1. **Deployment misconfiguration**: the deployer passes any `base_mint_amount ≠ 1_300_000 * 10**18` to the L2 constructor. There is no on-chain validation that enforces equality with the L1 constant.
2. **L1 upgrade**: `TOKENS_PER_MINT_REQUEST` is a file-level Solidity constant. If the L1 contract is upgraded with a different value, the L2 has no setter to follow suit, making the break permanent and irrecoverable.

Because `base_mint_amount` is never validated against `TOKENS_PER_MINT_REQUEST` at any point in the protocol, the structural mismatch is latent in every deployment. [8](#0-7) [9](#0-8) 

---

### Recommendation

Eliminate the dual source of truth. Two concrete options:

1. **Hardcode on L2**: replace the configurable `base_mint_amount` storage slot with the same constant (`1_300_000 * 10**18`) and remove the constructor parameter, mirroring the L1 approach.
2. **Validate at construction**: add an assertion in the L2 constructor (or a dedicated initializer) that `base_mint_amount == TOKENS_PER_MINT_REQUEST` by reading the expected value from a shared constant or from the L1 contract address stored in configuration.

Additionally, if `TOKENS_PER_MINT_REQUEST` ever needs to change on L1, a corresponding L2 setter (gated to `APP_GOVERNOR`) must be added so both sides can be updated atomically.

---

### Proof of Concept

1. Deploy L2 `RewardSupplier` with `base_mint_amount = 1_000_000 * 10**18` (any value ≠ `1_300_000 * 10**18`).
2. Normal staking operations accumulate rewards; `update_unclaimed_rewards_from_staking_contract` is called, triggering `request_funds`.
3. `send_mint_request_to_l1_reward_supplier` emits messages with payload `[1_000_000 * 10**18]`.
4. On L1, call `tick()`. `requiredMinting` builds `messagePayload[0] = 1_300_000 * 10**18`, computes a different hash, and `l2ToL1Messages` returns 0. No minting occurs.
5. Repeat indefinitely: L2 accumulates `l1_pending_requested_amount` and `unclaimed_rewards`, but the token balance never increases.
6. Any call to `claim_rewards` with a non-zero amount eventually reverts with `AMOUNT_TOO_HIGH`, permanently freezing all staker and delegator yield. [1](#0-0) [10](#0-9)

            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

        uint256 fromAddress,
        address toAddress,

    function requiredMinting() public view returns (uint256, uint256) {
        uint256[] memory messagePayload = new uint256[](1);
        messagePayload[0] = TOKENS_PER_MINT_REQUEST;

        bytes32 msgHash = l2ToL1MsgHash(mintRequestSource(), address(this), messagePayload);
        // Limit the number of msgs to consume to limit.
        uint256 numMsgsToConsume = Math.min(
            messagingContract().l2ToL1Messages(msgHash),
            MAX_MESSAGES_TO_PROCESS_PER_TICK
        );

        return (TOKENS_PER_MINT_REQUEST * numMsgsToConsume, numMsgsToConsume);
