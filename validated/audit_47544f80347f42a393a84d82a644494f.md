### Title
Unprivileged Overwrite of Any Address's E2EE Encryption Key — (`File: x/e2ee/keeper/keeper.go`)

### Summary
`MsgRegisterEncryptionKey` accepts an arbitrary `Address` field with no check that the transaction signer owns that address. Any unprivileged account can overwrite the on-chain encryption key registered for any other address, including validators. This corrupts the e2ee key state used to distribute encrypted blocklists to validators.

### Finding Description
`RegisterEncryptionKey` in `x/e2ee/keeper/keeper.go` writes directly to the KV store using `req.Address` as the key, with no authorization check:

```go
// x/e2ee/keeper/keeper.go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
``` [1](#0-0) 

The internal helper unconditionally sets the key for whatever address is supplied:

```go
sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
``` [2](#0-1) 

`ValidateBasic()` only validates bech32 format and X25519 key format — it does not verify that `msg.Address` matches the transaction signer:

```go
func (m *MsgRegisterEncryptionKey) ValidateBasic() error {
    if _, err := sdk.AccAddressFromBech32(m.Address); err != nil {
        return fmt.Errorf("invalid address: %w", err)
    }
    return ValidateRecipientKey(m.Key)
}
``` [3](#0-2) 

There is no `signer == req.Address` guard anywhere in the message handler chain. The `MsgRegisterEncryptionKey` struct carries two independent fields (`Address`, `Key`); the signer of the outer transaction need not be `Address`. [4](#0-3) 

### Impact Explanation
The e2ee module is the on-chain public-key registry used to distribute encrypted blocklists to validators. An attacker who replaces a validator's registered public key with an attacker-controlled X25519 key causes the blocklist to be encrypted to the wrong key. The targeted validator can no longer decrypt the blocklist, effectively bypassing the blocklist enforcement mechanism for that validator. This matches the explicitly allowed High impact: **"Corruption of… e2ee key/message state with direct security impact"** and **"Bypass of Cronos admin… block-list… authorization checks."**

### Likelihood Explanation
The attack requires only a valid Cosmos SDK transaction with a fee — no special permissions, no leaked keys, no admin access. The attacker supplies any valid bech32 address and any valid X25519 public key. The transaction will be accepted and will overwrite the victim's key. The attack can be repeated continuously to prevent the victim from re-registering their legitimate key (exact analog to the credential front-running in the original report).

### Recommendation
Add an explicit authorization check in `RegisterEncryptionKey` (or in `ValidateBasic`) that asserts the transaction signer equals `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signers := sdk.UnwrapSDKContext(ctx).TxSigners() // or equivalent
    if len(signers) == 0 || signers[0].String() != req.Address {
        return nil, sdkerrors.ErrUnauthorized
    }
    ...
}
```

Alternatively, annotate the proto field with `(cosmos.msg.v1.signer) = "address"` and verify the SDK enforces it end-to-end.

### Proof of Concept

1. Validator `V` registers their legitimate X25519 public key `PK_V` for address `addr_V` via `MsgRegisterEncryptionKey{Address: addr_V, Key: PK_V}`.
2. Attacker `A` (any account) submits `MsgRegisterEncryptionKey{Address: addr_V, Key: PK_A}` where `PK_A` is an attacker-controlled X25519 key.
3. The keeper writes `PK_A` to `KVStore[KeyPrefix(addr_V)]` with no error — the transaction is accepted.
4. The blocklist admin queries `addr_V`'s key, receives `PK_A`, and encrypts the blocklist to `PK_A`.
5. Validator `V` cannot decrypt the blocklist (it is encrypted to `PK_A`, not `PK_V`).
6. Validator `V` cannot apply the blocklist; blocked addresses can transact through `V`.
7. If `V` re-registers `PK_V`, attacker `A` immediately front-runs with another `MsgRegisterEncryptionKey{Address: addr_V, Key: PK_A}`, maintaining the denial indefinitely. [5](#0-4)
