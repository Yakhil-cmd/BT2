### Title
`calculateBinPositionAtPrice` Reverts with Division-by-Zero When Oracle Price Causes `lowerPriceX64 == upperPriceX64` — (`metric-core/contracts/libraries/SwapMath.sol`)

---

### Summary

`SwapMath.calculateBinPositionAtPrice` divides by `(upperPriceX64 - lowerPriceX64)` with no guard. The bin price bounds are computed via integer-truncating `Math.mulDiv` in `distanceE6ToPriceX64`. When the oracle mid-price is small enough that `midPriceX64 * lengthE6 < 1e6`, both bounds round to the same value, making the denominator zero and reverting every swap that enters that bin.

---

### Finding Description

`calculateBinPositionAtPrice` performs inverse linear interpolation inside a bin:

```solidity
// metric-core/contracts/libraries/SwapMath.sol  lines 116-124
unchecked {
    // Numerator ≤ 2^128 × 2^104 = 2^232; denominator > 0 by assumption.
    if (rounding == Math.Rounding.Floor) {
        return ((priceX64 - lowerPriceX64) * MAX_POS_BIN) / (upperPriceX64 - lowerPriceX64);
    } else {
        return Math.ceilDiv((priceX64 - lowerPriceX64) * MAX_POS_BIN, upperPriceX64 - lowerPriceX64);
    }
}
```

The comment says "denominator > 0 by assumption," but the assumption is never enforced on-chain.

The bin bounds are computed in `MetricOmmPool._getInitialStateForSwap` (and in every bin-advance step during a swap):

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 850-851
lowerPriceX64 = distanceE6ToPriceX64(curBinDistE6Cache, params.midPriceX64);
upperPriceX64 = distanceE6ToPriceX64(_addDistE6(curBinDistE6Cache, binState.lengthE6), params.midPriceX64);
```

`distanceE6ToPriceX64` uses integer-truncating `Math.mulDiv`:

```solidity
// metric-core/contracts/MetricOmmPool.sol  lines 577-587
function distanceE6ToPriceX64(int256 distanceValueE6, uint256 midPriceX64) internal pure returns (uint256) {
    unchecked {
        if (distanceValueE6 >= 0) {
            return Math.mulDiv(midPriceX64, 1e6 + uint256(distanceValueE6), 1e6);
        } else {
            return Math.mulDiv(midPriceX64, 1e6 - uint256(-distanceValueE6), 1e6);
        }
    }
}
```

The difference between `upperPriceX64` and `lowerPriceX64` is approximately `midPriceX64 * lengthE6 / 1e6`. When this product is less than `1e6`, integer truncation makes both values identical.

The factory enforces `lengthE6 >= 1` per bin but places no lower bound on the oracle price:

The oracle is only validated for token-pair identity, not price magnitude:

`_getBidAndAskPriceX64` only checks `bid > 0` and `bid < ask`:


With `bid = 1, ask = 2` (both valid), `midPriceX64 = Math.sqrt(2) = 1`. For any bin with `lengthE6 = 1`:

```
lowerPriceX64 = Math.mulDiv(1, 1_000_000, 1_000_000) = 1
upperPriceX64 = Math.mulDiv(1, 1_000_001, 1_000_000) = 1   ← truncated
```

`calculateBinPositionAtPrice` is then called with denominator `= 0` inside `buyToken0InBinSpecifiedIn` and `buyToken1InBinSpecifiedOut`:

Division by zero always panics in Solidity even inside `unchecked`, so every swap call reverts.

---

### Impact Explanation

Every call to `swap` or `simulateSwapAndRevert` that reaches a bin whose computed `lowerPriceX64 == upperPriceX64` reverts unconditionally. The pool's core swap functionality is completely broken for the duration that the oracle returns a sufficiently small price. Liquidity providers cannot receive fees; traders cannot execute. This matches the contest gate: **broken core pool functionality causing unusable swap flows**.

---

### Likelihood Explanation

The trigger requires `midPriceX64 * lengthE6 < 1e6`. With the minimum factory-allowed `lengthE6 = 1`, this requires `midPriceX64 < 1e6`, i.e., a Q64.64 oracle price below `~5.4 × 10⁻¹⁴`. This is unrealistic for any real asset pair but is not blocked by any on-chain guard. A price provider that malfunctions, is misconfigured, or is deployed for a token pair with extreme denomination differences could produce such values. Likelihood is **low**.

---

### Recommendation

Add an explicit guard in `calculateBinPositionAtPrice` to handle the degenerate case, mirroring the external report's fix pattern:

```solidity
function calculateBinPositionAtPrice(
    uint256 lowerPriceX64,
    uint256 upperPriceX64,
    uint256 priceX64,
    Math.Rounding rounding
) internal pure returns (uint256) {
+   if (upperPriceX64 <= lowerPriceX64) return MAX_POS_BIN; // degenerate bin: treat as fully consumed
    unchecked {
        if (rounding == Math.Rounding.Floor) {
            return ((priceX64 - lowerPriceX64) * MAX_POS_BIN) / (upperPriceX64 - lowerPriceX64);
        } else {
            return Math.ceilDiv((priceX64 - lowerPriceX64) * MAX_POS_BIN, upperPriceX64 - lowerPriceX64);
        }
    }
}
```

Additionally, the factory should enforce a minimum effective bin width at creation time, e.g., requiring `lengthE6` to be large enough that `lowerPriceX64 < upperPriceX64` holds for any oracle price above a protocol-defined floor.

---

### Proof of Concept

1. Deploy a pool with `lengthE6 = 1` for bin 0 (minimum allowed by factory).
2. Configure the price provider to return `bid = 1, ask = 2` (both pass all on-chain checks).
3. `midPriceX64 = Math.sqrt(1 * 2) = 1`.
4. `lowerPriceX64 = Math.mulDiv(1, 1_000_000, 1_000_000) = 1`.
5. `upperPriceX64 = Math.mulDiv(1, 1_000_001, 1_000_000) = 1`.
6. Call `swap(recipient, true, 1e18, 0, "", "")`.
7. Inside `buyToken0InBinSpecifiedIn`, `upperPriceX64 (1) > priceLimitX64 (0)`, so `calculateBinPositionAtPrice(1, 1, 0, Floor)` is called.
