### Title
Excess ETH Sent to Payable Router/Adder Functions Is Silently Consumed by the Next WETH-Input Caller — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`MetricOmmSimpleRouter` and `MetricOmmPoolLiquidityAdder` expose multiple `payable` entry points. The shared `pay()` helper in `PeripheryPayments` unconditionally drains `address(this).balance` whenever the token being paid is WETH. There is no validation that `msg.value` equals the required swap/liquidity amount, and no validation that `msg.value == 0` when the input token is not WETH. Any ETH left in the contract from a prior call is silently consumed on behalf of the next WETH-input caller, constituting a direct theft of the prior caller's funds.

---

### Finding Description

`PeripheryPayments.pay()` contains the following branch for WETH payments:

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
``` [1](#0-0) 

Whenever `token == WETH`, the function first checks `address(this).balance` and uses any ETH sitting in the contract before pulling from the payer's wallet. This is the correct design for the intended use-case (user sends ETH → router wraps it → pays pool), but it creates a critical invariant: **the contract must never hold ETH that belongs to a prior caller at the time a new WETH-input call is processed.**

That invariant is broken by two independent paths:

**Path 1 — Non-WETH swap with `msg.value > 0`**

Every swap entry point (`exactInputSingle`, `exactInput`, `exactOutputSingle`, `exactOutput`) and every liquidity entry point (`addLiquidityExactShares`, `addLiquidityWeighted`) is declared `external payable`. [2](#0-1) [3](#0-2) 

The `receive()` fallback only rejects plain ETH transfers from non-WETH addresses:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [4](#0-3) 

`receive()` is **not** invoked when ETH is attached to a function call via `msg.value`. Therefore, any caller can send ETH alongside a call to `exactInputSingle(..., tokenIn=USDC, ...)` and the ETH is silently accepted and retained by the contract. The `pay()` branch for non-WETH tokens uses `safeTransferFrom` and ignores `address(this).balance` entirely, so the ETH is never consumed for that call.

**Path 2 — WETH swap with `msg.value > amount_owed`**

When `tokenIn == WETH` and the user sends more ETH than the pool actually requests (`msg.value > value`), `pay()` wraps and transfers exactly `value` ETH. The surplus (`msg.value - value`) remains in the contract with no automatic refund.

In both paths, the stranded ETH is consumed by the very next caller whose `tokenIn == WETH`, because `pay()` reads `address(this).balance` at call time and uses it unconditionally.

---

### Impact Explanation

**Direct loss of user principal.** The victim's ETH is transferred to the pool on behalf of a different user, reducing that user's WETH pull from their own wallet by the same amount. The attacker pays less (or nothing) for their swap/liquidity deposit; the victim receives nothing in return. The loss is bounded only by the amount of ETH the victim sent, which can be arbitrarily large.

---

### Likelihood Explanation

**Low.** The victim must make a mistake: either send ETH with a non-WETH swap, or send more ETH than the swap requires. However, the `multicall` pattern (used by both contracts) encourages batching operations, and a user who intends to do a WETH swap followed by `refundETH()` but omits the refund step, or who miscalculates `amountIn`, creates the condition. The theft requires no special privilege — any subsequent WETH-input caller benefits automatically.

---

### Recommendation

1. **Validate `msg.value` against the WETH input amount.** In the swap callback (`_justPayCallback`) or at the entry point, when `tokenIn == WETH`, assert `msg.value == amountOwed` (or `msg.value == 0` if the user intends to pay from their WETH balance).

2. **Reject non-zero `msg.value` when `tokenIn != WETH`.** Add a guard at each `payable` entry point:
   ```solidity
   if (params.tokenIn != WETH && msg.value != 0) revert UnexpectedETH();
   ```

3. **Alternatively**, make non-WETH entry points non-payable and provide separate ETH-input variants that enforce `msg.value == amountIn`.

---

### Proof of Concept

```
Setup:
  Router deployed with WETH = 0xWETH
  Pool A: token0=USDC, token1=WETH
  Pool B: token0=WETH, token1=DAI

Step 1 — Victim calls exactInputSingle for USDC→DAI (non-WETH path):
  router.exactInputSingle{value: 1 ether}(
      ExactInputSingleParams({
          pool: poolA,
          tokenIn: USDC,
          ...
          amountIn: 1000e6,
          ...
      })
  );
  // pay() branch: token != WETH → safeTransferFrom(USDC, victim, pool, 1000e6)
  // 1 ETH sits in router, unspent.

Step 2 — Attacker calls exactInputSingle for WETH→DAI:
  router.exactInputSingle{value: 0}(
      ExactInputSingleParams({
          pool: poolB,
          tokenIn: WETH,
          ...
          amountIn: 1 ether,
          ...
      })
  );
  // pay() branch: token == WETH, nativeBalance = 1 ether >= value = 1 ether
  // → WETH.deposit{value: 1 ether}()  ← uses victim's ETH
  // → WETH.transfer(poolB, 1 ether)
  // Attacker receives DAI without spending any ETH or WETH.

Result: Victim loses 1 ETH. Attacker gains 1 ETH worth of DAI.
```
