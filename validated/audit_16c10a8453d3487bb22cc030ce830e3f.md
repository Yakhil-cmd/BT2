### Title
Excess ETH Permanently Trapped in `RewardSupplier` When `tick()` Called With No Pending Mint Requests or Overpaid Fee - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is `external payable`, requiring callers to send ETH to cover L1→L2 messaging fees. When no pending mint requests exist (`amountToMint == 0`), the entire fee-spending block is skipped and all ETH sent with the call is permanently locked in the contract. Even when minting does occur, any ETH sent in excess of the actual fees required by `depositWithMessage` and `sendMessageToL2` is also permanently trapped. The contract has no `receive()`, `fallback()`, or ETH-withdrawal function, making recovery impossible.

---

### Finding Description

`RewardSupplier.tick()` is declared `external payable` and splits `msg.value` between two L1→L2 messaging calls:

```solidity
uint256 msgFee = msg.value / 2;
bridge().depositWithMessage{value: msgFee}(...);
// ...
msgFee = msg.value - msgFee;
messagingContract().sendMessageToL2{value: msgFee}(...);
```

This fee-spending logic is gated inside `if (amountToMint > 0)`. If `requiredMinting()` returns zero (no pending L2 mint requests), the block is entirely skipped and `msg.value` is never forwarded or refunded. [1](#0-0) 

Even when `amountToMint > 0`, the fee split is a fixed `msg.value / 2` per call. If the caller sends more ETH than the actual fees required by the bridge and messaging contract, the surplus remains in `RewardSupplier` with no recovery path.

Neither `RewardSupplierStorage`, `ProxySupportImpl`, nor any parent contract defines a `receive()`, `fallback()`, or ETH-withdrawal function. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any ETH sent to `tick()` when `amountToMint == 0` is permanently frozen in the `RewardSupplier` contract. There is no administrative or user-facing function to recover it. This constitutes **permanent freezing of the caller's ETH funds** with no recovery mechanism. Matches allowed impact: **High — Temporary/Permanent freezing of funds**.

---

### Likelihood Explanation

`tick()` is callable by any public account (`external`). A caller must estimate the required fee off-chain before calling. Two realistic scenarios trigger the loss:

1. **Race condition**: A caller calls `requiredMinting()` off-chain, sees pending messages, estimates fees, and submits `tick()` with ETH. Between the off-chain read and on-chain execution, another caller's `tick()` processes the pending messages first. The second caller's `tick()` now sees `amountToMint == 0` and all their ETH is trapped.
2. **Fee overestimation**: A caller sends more ETH than the actual fees consumed by `depositWithMessage` and `sendMessageToL2`. The surplus is trapped.

Scenario 1 is a realistic race condition on a public, permissionless function.

---

### Recommendation

Refund any unspent ETH to `msg.sender` at the end of `tick()`. If `amountToMint == 0`, refund the full `msg.value`. If minting occurs, track actual ETH consumed by the two sub-calls and refund the remainder:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ... existing logic ...
    }
    // Refund any unspent ETH to caller
    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok, ) = msg.sender.call{value: remaining}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

---

### Proof of Concept

1. Observe that `requiredMinting()` is a public view function — any caller can read it.
2. Caller A calls `requiredMinting()` off-chain and sees 1 pending message; estimates fee as `X` ETH.
3. Caller B's `tick()` transaction is mined first, consuming the pending message.
4. Caller A's `tick{value: X}()` is mined: `requiredMinting()` now returns `(0, 0)`, the `if` block is skipped, and `X` ETH is permanently locked in `RewardSupplier`.
5. No function exists to recover the ETH — confirmed by inspecting `RewardSupplier.sol`, `RewardSupplierStorage.sol`, and `ProxySupportImpl.sol`. [4](#0-3) [5](#0-4)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
