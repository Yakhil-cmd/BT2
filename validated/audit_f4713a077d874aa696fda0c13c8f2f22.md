### Title
`PresignOutput` Derives `Clone` Without Type-Level Single-Use Enforcement, Enabling Nonce Reuse and Private Key Extraction — (File: `src/frost/mod.rs`)

---

### Summary

The FROST `PresignOutput` struct contains secret nonce material that the protocol requires to be used **exactly once**. The `sign_v2` function accepts it by value (move semantics), which would normally enforce single-use via Rust's ownership system. However, `PresignOutput` derives `Clone`, as directly evidenced in production test code. This means any caller can clone the presignature before passing it to `sign_v2` and initiate a second signing session with the same nonces — a classic nonce-reuse scenario that allows full private key extraction in FROST.

---

### Finding Description

The FROST presigning protocol (`src/frost/mod.rs`) produces a `PresignOutput` containing two fields: secret `nonces` (the per-session randomness) and a shared `commitments_map` (the public nonce commitments from all participants). [1](#0-0) 

The library's own documentation and comments state unambiguously that presignatures must never be reused: [2](#0-1) [3](#0-2) 

The `sign_v2` function takes `presignature: PresignOutput` by value, which is the intended mechanism to enforce single-use through Rust's move semantics: [4](#0-3) 

However, `PresignOutput` implements `Clone`. This is directly confirmed by production test code that calls `.clone()` on a `frost::PresignOutput<C>` value before passing it to `sign_v2`: [5](#0-4) 

Because `PresignOutput` is `Clone`, any caller — including a malicious coordinator orchestrating retry logic, or an application layer implementing fault tolerance — can clone the presignature before the first signing call and retain a copy for a second signing session. The move-semantics guard is entirely bypassed.

Inside `do_sign_participant_v2`, the nonces from the presignature are used directly to produce the signature share: [6](#0-5) 

If the same `nonces` are used to sign two different messages `m1` and `m2`, the two resulting signature shares satisfy a linear system that directly reveals the participant's private signing share `x_i`. This is the standard FROST/Schnorr nonce-reuse attack.

The same structural problem exists in the robust ECDSA path. The security documentation for that scheme explicitly states the consequence: [7](#0-6) [8](#0-7) 

The analogy to the external report is direct: just as `KarmaAirdrop` inherits `Pausable` (the infrastructure exists) but `claim` lacks `whenNotPaused` (the guard is not applied to the critical function), this library documents "never reuse a presignature" (the awareness exists) but `PresignOutput` derives `Clone` (the type-level guard is not applied), leaving the critical invariant unenforced in code.

---

### Impact Explanation

**Critical — Extraction of private signing shares / nonce material.**

If the same `PresignOutput` is used in two signing sessions for different messages `m1` and `m2`, the two FROST signature shares `z1` and `z2` satisfy:

```
z1 = nonce_binding_1 + x_i * c1
z2 = nonce_binding_1 + x_i * c2   ← same nonce, different challenge
```

Subtracting: `z1 - z2 = x_i * (c1 - c2)`, so `x_i = (z1 - z2) / (c1 - c2)`.

The private signing share `x_i` is fully recovered. With `t` such pairs across `t` participants, the full aggregate private key is reconstructed. This enables unauthorized creation of valid threshold signatures for attacker-chosen inputs.

---

### Likelihood Explanation

**Medium.** The attack does not require breaking any cryptographic primitive. It requires only that the application layer clone `PresignOutput` before passing it to `sign_v2` — a natural pattern in fault-tolerant distributed systems where signing sessions can fail due to network issues and need to be retried. A malicious coordinator can deliberately abort a signing session after collecting signature shares (gaining partial information), then trigger a retry using the same presignature. Since `PresignOutput` is `Clone` and the library provides no mechanism to detect or prevent this, the invariant is entirely caller-enforced with no code-level backstop.

---

### Recommendation

Remove the `Clone` (and `Copy`, if present) derive from `PresignOutput` in `src/frost/mod.rs` and from `PresignOutput` / `RerandomizedPresignOutput` in the ECDSA modules. Rust's ownership system will then enforce single-use at compile time — any attempt to use a presignature twice will be a compile error. If retry logic is required, the application must generate a fresh presignature for each attempt. [9](#0-8)

### Proof of Concept

```rust
// 1. Run presigning — each participant gets a PresignOutput
let presig_outputs = frost_run_presignature(&keys, threshold, actual_signers, rng).unwrap();

// 2. Clone the PresignOutput BEFORE the first signing call
//    (PresignOutput derives Clone — this compiles and runs successfully)
let presig_clone: Vec<_> = presig_outputs
    .iter()
    .map(|(p, out)| (*p, out.clone()))   // ← confirmed by sign.rs:595
    .collect();

// 3. First signing session with message M1 — presig_outputs is moved/consumed
let _sig1 = run_sign_v2(&keys, coordinator, &presig_outputs, b"message_one");

// 4. Second signing session with message M2 — uses cloned nonces (NONCE REUSE)
let _sig2 = run_sign_v2(&keys, coordinator, &presig_clone,   b"message_two");

// 5. From sig1 and sig2, recover private key share x_i:
//    x_i = (z1 - z2) / (challenge1 - challenge2)
```

The two signature shares produced in steps 3 and 4 use identical nonce material (`presig_outputs[i].nonces == presig_clone[i].nonces`), satisfying the linear system above and yielding the private signing share.

async fn do_presign<C: Ciphersuite + Send>(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    signing_share: SigningShare<C>,
    mut rng: impl CryptoRngCore,
) -> Result<PresignOutput<C>, ProtocolError> {
    // Round 1
    let mut commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>> = BTreeMap::new();

    // Creating two commitments and corresponding nonces
    let (nonces, commitments) = commit(&signing_share, &mut rng);
    commitments_map.insert(me.to_identifier()?, commitments);

    let commit_waitpoint = chan.next_waitpoint();
    // Sending the commitments to all
    chan.send_many(commit_waitpoint, &commitments)?;

    // Collecting the commitments
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    Ok(PresignOutput {
        nonces,
        commitments_map,
    })
}
