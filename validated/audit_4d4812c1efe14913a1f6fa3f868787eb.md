### Title
Zero Oracle-Rotation Timelock Bypasses LP-Exit Window, Enabling Atomic Oracle Swap and Fund Drain — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`_validatePoolParameters` enforces minimum thresholds on fees, initial amounts, and liquidity, but imposes **no minimum on `priceProviderTimelock`**. A pool created with `priceProviderTimelock = 0` gives the pool admin a zero-delay oracle rotation path: `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` can be called in the same block, atomically replacing the live oracle before any LP can react.

### Finding Description

`createPool` stores `params.priceProviderTimelock` verbatim:

```solidity
// MetricOmmPoolFactory.sol line 213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` checks fees, admin address, provider token match, initial amounts, and minimal liquidity, but never checks `params.priceProviderTimelock`:

```solidity
// MetricOmmPoolFactory.sol lines 548-563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    ...
    if (params.minimalMintableLiquidity == 0) revert InvalidMinimalMintableLiquidity();
    // ← no check on params.priceProviderTimelock
}
```

In `proposePoolPriceProvider`, `executeAfter` is computed as:

```solidity
// MetricOmmPoolFactory.sol line 487
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp + 0 = block.timestamp
```

In `executePoolPriceProviderUpdate`, the guard is:

```solidity
// MetricOmmPoolFactory.sol line 499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp → false → no revert
```

Both calls succeed in the same block. The pool admin atomically rotates to a malicious `IPriceProvider` that returns an extreme bid/ask, then executes a swap that drains LP balances at the manipulated price.

This is the direct analog of the external report's "minimum password length of 4" weakness: the factory enforces minimum thresholds on every other parameter but enforces **no minimum on the timelock**, the sole constraint protecting LPs from instant oracle replacement.

### Impact Explanation

A pool admin controlling a pool with `priceProviderTimelock = 0` can:
1. Deploy a malicious `IPriceProvider` that returns `token0()` / `token1()` matching the pool (passes `_validatePriceProvider`) but returns an arbitrarily skewed bid/ask.
2. Call `proposePoolPriceProvider` → `executePoolPriceProviderUpdate` in the same block.
3. Immediately call `swap` against the pool using the manipulated oracle price.
4. Drain LP token0 and token1 balances.

LPs have no on-chain mechanism to exit between proposal and execution when the delay is zero. This is a direct loss of user principal.

### Likelihood Explanation

`createPool` is permissionless. Any caller can deploy a pool with `priceProviderTimelock = 0`. The pool admin role is semi-trusted only "inside caps and timelocks" (per protocol design); a zero timelock removes that constraint entirely. The attack requires the pool admin to be malicious or compromised, but the factory is supposed to prevent this configuration from being valid.

### Recommendation

Add a minimum timelock check in `_validatePoolParameters`:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days; // or protocol-chosen floor

if (params.priceProviderTimelock != type(uint256).max
    && params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

This mirrors the pattern already used for fee caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) and ensures the timelock always provides a meaningful LP-exit window.

### Proof of Concept

