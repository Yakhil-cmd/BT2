### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` Due to Missing `evmAddress` Binding in Signed Data - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over caller-supplied `signedData` but does not enforce that `signedData` encodes the `evmAddress` being proven. Because the signed payload is decoupled from the specific COA identity, a valid signature produced for one COA (COA-A) owned by a Cadence account can be replayed verbatim to obtain a `ValidationResult(isValid: true)` for a different COA (COA-B) owned by the same account. The code itself acknowledges this gap in a comment but leaves enforcement entirely to callers.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` (lines 1001–1110) accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function:

1. Fetches the Cadence account at `address` and builds a `Crypto.KeyList` from the supplied `keyIndices`.
2. Calls `keyList.verify(signatureSet: signatureSet, signedData: signedData, domainSeparationTag: "FLOW-V0.0-user")` — verifying that the signatures are valid over `signedData`.
3. Separately checks that the COA stored at `path` matches `evmAddress`. [1](#0-0) 

The critical gap: step 2 and step 3 are independent. The cryptographic verification in step 2 covers only `signedData`; it does not cover `evmAddress`. The code comment at lines 1001–1009 explicitly acknowledges this: [2](#0-1) 

Because `signedData` is arbitrary caller-supplied bytes with no protocol-enforced binding to `evmAddress`, any signature Alice produced for COA-A (e.g., visible in a prior on-chain transaction) can be submitted by an attacker with `path` pointing to COA-B and `evmAddress` set to COA-B's address. The signature check passes (same Cadence keys, same `signedData`), the COA address check passes (COA-B is at the supplied path), and the function returns `isValid: true` for COA-B — a COA Alice never authorized.

This is structurally identical to the reported AtlasVerification issue: the signed/hashed artifact covers an identity (Cadence address + key indices) but omits the mutable configuration context (which specific COA is being proven), so a signature remains valid across different contexts.

### Impact Explanation

Any Cadence contract that calls `validateCOAOwnershipProof` to gate a privileged action (e.g., authorizing an EVM asset transfer, granting a capability, or unlocking a resource) without independently binding `signedData` to the specific `evmAddress` is vulnerable. An attacker who observes a valid `(signedData, signatures, keyIndices)` tuple from a prior on-chain use — which is public — can replay it against a different COA owned by the same Cadence account to obtain a fraudulent `isValid: true` result. If the calling contract acts on that result (e.g., releases funds or grants access), the attacker achieves unauthorized access to on-chain assets or privileged state without possessing any private key.

The EVM-side precompile `verifyCOAOwnershipProof` is not directly vulnerable because it always passes the calling COA's own address as `evmAddress`. However, `validateCOAOwnershipProof` is `access(all)` and is callable from any Cadence transaction or contract, making the attack surface broader than the precompile alone.

### Likelihood Explanation

Likelihood is **low-medium**. Preconditions are:
1. The target Cadence account must own at least two COAs (at different paths).
2. A valid `(signedData, signatures, keyIndices)` tuple for one of those COAs must be observable on-chain (e.g., from a prior call to `validateCOAOwnershipProof` or the EVM precompile).
3. A Cadence contract must use `validateCOAOwnershipProof` to authorize a consequential action without independently binding `signedData` to `evmAddress`.

Condition 2 is realistic: any prior on-chain use of the function exposes the tuple. Condition 3 depends on ecosystem contracts. Condition 1 is currently uncommon but not impossible, and the protocol places no restriction on creating multiple COAs per account.

### Recommendation

Enforce at the protocol level that `signedData` encodes `evmAddress`. The simplest fix is to prepend or hash `evmAddress` into the data that is cryptographically verified before calling `keyList.verify`, rather than relying on callers to do so. Alternatively, the function can reject calls where `signedData` does not contain the expected `evmAddress` bytes, making the binding mandatory rather than advisory.

### Proof of Concept

1. Alice owns Cadence account `0xAlice` with key index 0 (weight 1000).
2. Alice creates COA-A at `/storage/coaA` (EVM address `0xAAAA`) and COA-B at `/storage/coaB` (EVM address `0xBBBB`).
3. Alice (or any contract) calls `EVM.validateCOAOwnershipProof(address: 0xAlice, path: /public/coaA, signedData: msg, keyIndices: [0], signatures: [sig], evmAddress: 0xAAAA)` — this call is recorded on-chain with `msg` and `sig` visible.
4. Attacker submits a Cadence transaction calling `EVM.validateCOAOwnershipProof(address: 0xAlice, path: /public/coaB, signedData: msg, keyIndices: [0], signatures: [sig], evmAddress: 0xBBBB)`.
5. `keyList.verify` passes — `sig` is valid over `msg` under Alice's key 0.
6. The COA address check passes — COA-B is at `/public/coaB` and its address is `0xBBBB`.
7. The function returns `ValidationResult(isValid: true, problem: nil)` for COA-B, even though Alice only signed `msg` in the context of COA-A. [3](#0-2)
