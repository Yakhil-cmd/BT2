### Title
`EvmErc20V2.withdrawToNear` permanently destroys user tokens due to V2/V1 input format mismatch with `ExitToNear` precompile when `error_refund` feature is disabled — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20V2.withdrawToNear` prepends a 20-byte sender address to the precompile input buffer, producing a V2 layout that the `ExitToNear` precompile cannot parse when compiled without the `error_refund` feature. The burn executes unconditionally before the precompile call, and the call's failure return value is never checked. The result is that ERC-20 tokens are permanently destroyed on the EVM side with no corresponding NEP-141 transfer on NEAR.

---

### Finding Description

**Input format divergence**

`EvmErc20.withdrawToNear` (V1) encodes:
```
[0x01][amount_b 32 bytes][recipient bytes]
``` [1](#0-0) 

`EvmErc20V2.withdrawToNear` (V2) encodes:
```
[0x01][sender 20 bytes][amount_b 32 bytes][recipient bytes]
``` [2](#0-1) 

**Precompile parsing without `error_refund`**

When the `error_refund` feature is disabled, `parse_input` strips only the single flag byte and returns `&input[1..]`: [3](#0-2) 

For flag `0x1`, the parser then reads `input[..32]` as the amount: [4](#0-3) 

When V2 input arrives, `input[..32]` is `[sender 20 bytes][first 12 bytes of amount_b]` — a 256-bit big-endian integer whose high 160 bits are the caller's Ethereum address. For any non-zero sender this value vastly exceeds `u128::MAX`, so `parse_amount` returns `ERR_INVALID_AMOUNT`: [5](#0-4) 

The EVM `call` to the precompile therefore returns `0` (failure). The Solidity assembly block captures `res` but never reverts on it: [6](#0-5) 

**Burn-before-call ordering**

`_burn` executes unconditionally before the precompile call: [7](#0-6) 

When the precompile call fails, no `PromiseArgs` log is emitted, so no NEP-141 `ft_transfer` is ever scheduled on NEAR. The tokens are gone.

**`error_refund` is not a default feature**

`error_refund` is an opt-in feature in both `engine-precompiles` and `engine`; neither crate lists it under `default`: [8](#0-7) [9](#0-8) 

The production `contract` feature does not pull in `error_refund`. A deployment compiled with the standard `contract` feature and `EvmErc20V2` deployed as the ERC-20 mirror is therefore vulnerable.

**With `error_refund` enabled the path is safe**

When `error_refund` is compiled in, `parse_input` consumes `input[1..21]` as the refund address and returns `&input[21..]`, so the subsequent `input[..32]` slice correctly lands on `amount_b`: [10](#0-9)

### Impact Explanation

Every call to `EvmErc20V2.withdrawToNear` on an engine compiled without `error_refund`:

1. Burns the exact requested token amount from the caller's EVM balance.
2. Silently fails the precompile call.
3. Creates no NEAR-side transfer promise.
4. Leaves the caller with neither EVM tokens nor NEAR-side NEP-141 tokens.

Impact: **Critical — permanent freezing/destruction of user funds.**

---

### Likelihood Explanation

- `EvmErc20V2` is a production contract in the repository alongside `EvmErc20`.
- `error_refund` is not enabled by default; a standard `contract`-feature build omits it.
- Any unprivileged EVM user holding a balance in a V2-mirrored token can trigger the loss by calling `withdrawToNear` with any non-zero amount.
- No admin compromise or special privilege is required.

---

### Recommendation

1. **Short-term:** Do not deploy `EvmErc20V2` on any engine instance compiled without `error_refund`. Add a compile-time or runtime guard that prevents `EvmErc20V2` from being used when the feature is absent.
2. **Medium-term:** Check the precompile call return value in `withdrawToNear` and revert if it fails, so the burn is rolled back atomically.
3. **Long-term:** Promote `error_refund` to a default feature, or merge V1/V2 into a single contract that always includes the sender field and update the precompile parser accordingly.

### Proof of Concept

Assume engine compiled with `contract` feature only (no `error_refund`). `EvmErc20V2` is deployed as the ERC-20 mirror for some NEP-141 token.

```
Attacker holds 1000 units of the mirrored ERC-20.
Attacker calls: EvmErc20V2.withdrawToNear("victim.near", 1000)

Execution trace:
  1. _burn(attacker, 1000)          → EVM balance: 0  (irreversible)
  2. input = [0x01][attacker_addr 20B][0x...03E8 32B]["victim.near"]
  3. call(precompile, input)
       parse_input → strips flag → remaining = [attacker_addr 20B][amount 32B][recipient]
       parse_amount([attacker_addr 20B][first 12B of amount])
         → U256 >> u128::MAX → ERR_INVALID_AMOUNT → precompile returns 0
  4. res = 0, not checked, function returns normally
  5. No promise log emitted → no ft_transfer on NEAR

Result: 1000 tokens burned, 0 tokens received on NEAR.
```

Differential assertion: calling the same precompile with V1 input `[0x01][0x...03E8 32B]["victim.near"]` succeeds and schedules the correct `ft_transfer`. Calling with V2 input fails. The divergence is deterministic and reproducible in a local sandbox without any privileged access.

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}
