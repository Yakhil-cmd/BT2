### Title
Missing Participant-Set Validation in `presign` Allows Non-Triple-Generation Participants to Trigger Presign Failure and Triple Consumption — (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA `presign` function explicitly omits validating that the presign participant set matches the participants who generated the Beaver triples. The `TriplePub` struct carries a `participants` field recording exactly who was present during triple generation, but `presign` never checks it. A presign session initiated with a participant set that diverges from the triple-generation set will fail mid-protocol after honest parties have already committed their triple material, permanently consuming the one-time-use triples and denying the signing round.

---

### Finding Description

`TriplePub` records the participant set from triple generation: [1](#0-0) 

The `presign` public entry point accepts a `participants` slice and `PresignArguments` containing `triple0: (TripleShare, TriplePub)` and `triple1: (TripleShare, TriplePub)`. It validates the threshold against the triples' threshold field, but contains an explicit comment acknowledging the omission of the participant-set check: [2](#0-1) 

The only triple-related validation performed is threshold equality: [3](#0-2) 

No check is ever made that `participants` is a subset of (or equal to) `args.triple0.1.participants` or `args.triple1.1.participants`. The `participants` field of `TriplePub` is populated during generation but is dead at the presign call site.

Inside `do_presign`, each participant linearizes their triple share using a Lagrange coefficient computed over the **presign** participant set, not the triple-generation participant set: [4](#0-3) 

If the presign participant set differs from the triple-generation set, the Lagrange basis changes, the reconstructed public values (`big_e`, `K+A`, `X+B`) will not match the committed public triple values, and the protocol aborts: [5](#0-4) [6](#0-5) 

Because `PresignArguments` takes ownership of both `TripleShare` values and the protocol is one-time-use by design, the triples are permanently consumed even when the protocol aborts.

---

### Impact Explanation

When the presign participant set diverges from the triple-generation participant set — whether due to a race condition (a participant is added or removed between triple generation and presign) or a malicious coordinator at the application layer — the presign protocol runs, fails at the consistency-check stage, and irrecoverably destroys the two Beaver triples that were passed in. Since triples are one-time-use and expensive to generate (requiring a full OT-based MPC sub-protocol), their loss permanently denies the signing round that depended on them. Honest parties must re-run the triple generation protocol before they can attempt signing again. This maps to **High: Permanent denial of signing for honest parties under valid protocol inputs**.

---

### Likelihood Explanation

The scenario is realistic in any deployment where participant sets are managed dynamically:

1. A group of participants `{A, B, C}` runs triple generation and stores `TriplePub` (which records `participants = [A, B, C]`).
2. Before presign is initiated, participant `D` joins the signing group (e.g., after a reshare or refresh).
3. The application layer — acting on the updated participant set — calls `presign` with `participants = [A, B, D]` while supplying the triples generated by `{A, B, C}`.
4. `presign` accepts the call without error (no participant-set check).
5. `do_presign` runs, the Lagrange coefficients are computed over `{A, B, D}`, the reconstructed `e` does not match `big_e` from the triple, and the protocol aborts.
6. Both triples are consumed; the signing round is lost.

The `TriplePub.participants` field exists precisely to enable this check, making the omission a clear design gap rather than an intentional trade-off.

---

### Recommendation

In `presign`, after constructing the `ParticipantList`, validate that the presign participants are a subset of both triples' recorded participant sets:

```rust
// After: let participants = ParticipantList::new(participants)...
for p in participants.iter() {
    if !args.triple0.1.participants.contains(p) {
        return Err(InitializationError::BadParameters(format!(
            "participant {p:?} was not present during triple0 generation"
        )));
    }
    if !args.triple1.1.participants.contains(p) {
        return Err(InitializationError::BadParameters(format!(
            "participant {p:?} was not present during triple1 generation"
        )));
    }
}
```

This mirrors the pattern used elsewhere in the codebase (e.g., `assert_reshare_keys_invariants` checking intersection size) and prevents triples from being consumed by an invalid presign invocation. [7](#0-6)

### Proof of Concept

```
1. Run triple generation with participants {P0, P1, P2}, threshold=2.
   → Produces TriplePub { participants: [P0, P1, P2], threshold: 2, big_a, big_b, big_c }
   → Produces TripleShare for each of P0, P1, P2.

2. Call presign(
       participants = &[P0, P1, P3],   // P3 was NOT in triple generation
       me = P0,
       PresignArguments {
           triple0: (share_for_P0, triple_pub_from_step_1),
           triple1: (share_for_P0, triple_pub_from_step_1),
           keygen_out: ...,
           threshold: 2,
       }
   )
   → presign() returns Ok(protocol) — no error, missing check.

3. Protocol runs do_presign():
   - Lagrange coefficients computed over {P0, P1, P3} (not {P0, P1, P2}).
   - P3 has no valid triple share; provides fabricated e_j.
   - Reconstructed e ≠ big_e from TriplePub.
   - Returns Err(ProtocolError::AssertionFailed("received incorrect shares of kd")).

4. Both TripleShare values (triple0.0, triple1.0) are permanently consumed.
   Signing round is denied; honest parties must regenerate triples.

            participants: participants.len(),

            threshold: args.threshold.value(),
