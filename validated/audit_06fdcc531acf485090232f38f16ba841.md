### Title
Zero `priceProviderTimelock` Bypasses Oracle Rotation Delay, Enabling Immediate Malicious Oracle Swap — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`createPool` accepts any `priceProviderTimelock` value without a minimum bound check. When set to `0` (not `type(uint256).max`), the pool admin can propose and execute an oracle rotation atomically in a single transaction, bypassing the intended delay that protects LPs from a rogue oracle substitution.

### Finding Description
The factory stores the caller-supplied timelock directly with no lower-bound validation:

```solidity
// MetricOmmPoolFactory.sol line 213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` checks tokens, admin, fees, and initial amounts, but never validates `priceProviderTimelock`:

```solidity
// MetricOmmPoolFactory.sol lines 548-563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    if (params.token0 == address(0) ...) revert InvalidTokenConfig();
    if (params.admin == address(0)) revert InvalidAdmin();
    ...
    // ← no check on params.priceProviderTimelock
}
```

The only branch on the field is the immutable-mode sentinel:

```solidity
// MetricOmmPoolFactory.sol line 164
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
```

Any value other than `type(uint256).max` is treated as mutable with a delay of exactly `priceProviderTimelock` seconds. When that value is `0`:

```solidity
// MetricOmmPoolFactory.sol lines 481, 487
uint256 timelock = priceProviderTimelock[pool];   // == 0
uint256 executeAfter = block.timestamp + timelock; // == block.timestamp
```

The execute guard then trivially passes in the same block:

```solidity
// MetricOmmPoolFactory.sol line 499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp → false → no revert
```

A pool admin who deployed with `priceProviderTimelock = 0` can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in a single atomic multicall, replacing the oracle with an arbitrary address that passes the token-pair check but returns manipulated bid/ask prices.

### Impact Explanation
Once a malicious oracle is active, every subsequent `swap` call reads prices from it. The pool admin (or a colluding party) can set the oracle to return an extreme bid/ask, execute swaps that extract the entire token0 or token1 balance from the bins, and leave LPs with worthless positions. This is a direct loss of LP principal with no recovery path because the oracle address is now live and the timelock offers no further protection.

### Likelihood Explanation
`createPool` is permissionless; any caller can supply `priceProviderTimelock = 0`. The pool admin role is set at creation and is semi-trusted by design — the timelock is the only on-chain constraint that limits how quickly the admin can rotate the oracle. Removing that constraint by setting the timelock to zero is a single-field choice that requires no special privilege beyond being the pool creator. A malicious or negligent deployer can trigger this silently; LPs have no on-chain signal that the timelock is absent until they inspect factory storage.

### Recommendation
Add a minimum timelock floor in `_validatePoolParameters` (or inline in `createPool`) for the mutable-provider path:

```solidity
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

`MIN_PRICE_PROVIDER_TIMELOCK` should be a protocol constant (e.g., `1 days`) that gives LPs enough time to observe a pending rotation and exit before it executes.

### Proof of Concept
1. Attacker deploys a pool with `priceProviderTimelock = 0` and themselves as `admin`.
2. Attacker deploys `MaliciousOracle` that returns `(bid=1, ask=2)` (extreme low price for token1 per token0) but correctly reports `token0`/`token1` to pass `_validatePriceProvider`.
3. Attacker calls `factory.proposePoolPriceProvider(pool, address(maliciousOracle))` — sets `pendingPriceProviderExecuteAfter[pool] = block.timestamp`.
4. In the same transaction, attacker calls `factory.executePoolPriceProviderUpdate(pool)` — `block.timestamp < block.timestamp` is `false`, so no revert; `setPriceProvider(maliciousOracle)` executes.
5. Attacker calls `pool.swap(...)` — `_getBidAndAskPriceX64` reads from `maliciousOracle`, swap math uses the manipulated price, attacker receives far more token1 than owed.
