### Title
Missing Integrity Proof on ElGamal Ciphertext Components Allows Malicious Participant to Corrupt CKD Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates ElGamal ciphertext components `(λ_i·Y_i, λ_i·C_i)` from participants with no zero-knowledge proof or integrity check. A single malicious participant can send arbitrary group elements in place of their well-formed contribution, corrupting the aggregated `CKDOutput` and permanently denying the requesting application its derived key.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's contribution via `recv_from_others` and unconditionally adds the received `big_y` and `big_c` fields together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant `i` is supposed to send:
- `Y_i = y_i · G₁` (a fresh blinding point)
- `C_i = x_i · H(pk, app_id) + y_i · A` (the encrypted BLS share)

scaled by their Lagrange coefficient `λ_i`. There is **no proof of correct formation** — no discrete-log equality proof (dlogeq) showing that `C_i − S_i` and `Y_i` share the same discrete log with respect to generators `G₁` and `A`. The coordinator has no mechanism to detect a malformed contribution.

This is the direct analog of the original report's AES-CBC-without-MAC issue: just as an attacker could modify an AES-CBC ciphertext and have it decrypted without detection, a malicious MPC participant can substitute arbitrary group elements for their ElGamal ciphertext components and have them aggregated without detection.

Contrast this with the OT-based triple generation protocol, which requires every participant to supply `dlog` and `dlogeq` proofs for their polynomial commitments before they are accepted:

```rust
if !dlogeq::verify(
    &mut transcript.fork(b"dlogeq0", &from.bytes()),
    statement,
    their_phi_proof,
)? {
    return Err(ProtocolError::AssertionFailed(...));
}
```

No equivalent check exists in the CKD protocol.

---

### Impact Explanation

The aggregated output `(Y, C)` is an ElGamal encryption of `msk · H(pk, app_id)` under the application's public key `A`. The application recovers the BLS signature as `C − a·Y` and verifies it against the MPC network's public key `pk`.

If a malicious participant sends `(Y_j', C_j')` where `C_j' ≠ x_j · H(pk, app_id) + y_j' · A`, the decrypted value becomes:

```
C − a·Y = msk · H(pk, app_id) + Δ
```

where `Δ` is the attacker-controlled bias. Since `Δ ≠ 0` with overwhelming probability, the BLS pairing check fails and the application cannot derive its key. The CKD output is permanently unusable for that session.

The attacker cannot forge a valid BLS signature for a different `app_id` (that would require knowing `msk`), so the impact is bounded to **corruption of CKD output → unusable cryptographic output → permanent denial of CKD for honest parties**, which falls squarely within the allowed High impact scope.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. No special cryptographic knowledge is required beyond participation — the attacker simply sends two arbitrary `G₁` points instead of their correct contribution. The attack is single-round, requires no coordination, and is undetectable by the coordinator or any other participant.

---

### Recommendation

Each participant should accompany their `(λ_i·Y_i, λ_i·C_i)` with a zero-knowledge proof of correct formation. Concretely, a `dlogeq` proof demonstrating that `Y_i` and `C_i − S_i` share the same discrete logarithm with respect to generators `G₁` and `A` (i.e., `Y_i = y_i·G₁` and `C_i − S_i = y_i·A`) is sufficient. The coordinator must verify all such proofs before aggregating, following the same pattern already used in the OT-based triple generation protocol.

Additionally, `S_i = x_i · H(pk, app_id)` can be verified against the participant's public key share (the commitment to `x_i` established during DKG), providing a complete integrity check on the contribution.

---

### Proof of Concept

1. Honest participants `1..n` except participant `j` compute and send correct `(λ_i·Y_i, λ_i·C_i)`.
2. Malicious participant `j` sends `(Y_j', C_j') = (G₁, 2·G₁)` — two arbitrary, unrelated group elements.
3. The coordinator's `do_ckd_coordinator` aggregates without complaint:
   - `Y_final = Σ_{i≠j} λ_i·Y_i + G₁`
   - `C_final = Σ_{i≠j} λ_i·C_i + 2·G₁`
4. The application decrypts: `C_final − a·Y_final = msk·H(pk, app_id) + (2·G₁ − a·G₁) = msk·H(pk, app_id) + (2−a)·G₁`.
5. Since `a` is a random scalar, `(2−a)·G₁ ≠ 0` with overwhelming probability.
6. The BLS pairing check `e(sig, G₂) = e(H(pk, app_id), pk)` fails.
7. The application cannot derive its key; the CKD session is permanently denied. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
```

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
```
