### Title
COA Ownership Proof Signature Replay Across Multiple COAs Due to Missing `evmAddress` Binding in `signedData` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over an arbitrary `signedData` blob but does not enforce that `signedData` encodes the target `evmAddress`. Because the RLP-encoded `COAOwnershipProof` struct (which carries the `CapabilityPath`) is not itself signed, an attacker who observes a valid proof for COA-A can strip and replace the `CapabilityPath` field to produce a structurally valid proof for COA-B — both owned by the same Cadence account — causing `isValidSignature` (ERC-1271) on COA-B to return `ValidERC1271Signature` for a hash the account holder never authorized for COA-B.

### Finding Description

**Root cause — `validateCOAOwnershipProof` does not bind `signedData` to `evmAddress`:**

The Cadence function `EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that the supplied signatures are valid over `signedData` using the Cadence account's keys and the fixed domain tag `"FLOW-V0.0-user"`.
2. Checks that the COA resource at the supplied `path` has an EVM address matching the supplied `evmAddress`.

The code itself documents the gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

The `COAOwnershipProof` struct that is RLP-encoded and passed as `_sig` to the EVM precompile contains `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`. The `Signatures` field covers only `signedData`, not the proof struct itself: [2](#0-1) 

The EVM-side entry point is `COA.isValidSignature` in `coa.sol`, which passes `address(this)` (the calling COA's EVM address) as `evmAddress` and the caller-supplied `_hash` as `signedData`: [3](#0-2) 

The precompile `verifyCOAOwnershipProof` decodes the ABI-encoded proof and delegates to `coaOwnershipProofValidator`, which calls `validateCOAOwnershipProof`: [4](#0-3) [5](#0-4) 

**Exploit flow:**

1. Alice owns a Cadence account with two COAs: COA-A published at `/public/coa` and COA-B published at `/public/coa2`.
2. Alice legitimately signs hash `H` with domain tag `"FLOW-V0.0-user"` to authorize an action on COA-A. The resulting proof is `{KeyIndices:[0], Address:alice, CapabilityPath:/public/coa, Signatures:[sig]}`.
3. An attacker observes this proof on-chain (it is submitted as calldata in an EVM transaction).
4. The attacker re-encodes the proof with `CapabilityPath` changed to `/public/coa2` while keeping `Signatures` unchanged: `{KeyIndices:[0], Address:alice, CapabilityPath:/public/coa2, Signatures:[sig]}`.
5. The attacker calls `COA-B.isValidSignature(H, modified_proof)` from any EVM transaction.
6. The precompile calls `validateCOAOwnershipProof(alice, /public/coa2, H, [0], [sig], COA-B_address)`.
7. Step 1 (signature check): `sig` is valid over `H` under Alice's key → **passes**.
8. Step 2 (address check): the COA at `/public/coa2` has address `COA-B_address` → **passes**.
9. `isValidSignature` returns `0x1626ba7e` (`ValidERC1271Signature`) for COA-B, for a hash Alice never authorized for COA-B. [6](#0-5) 

### Impact Explanation

Any EVM smart contract or DeFi protocol that relies on ERC-1271 (`isValidSignature`) to authenticate COA-B's owner — e.g., for permit-style token approvals, NFT marketplace order signing, or DAO voting — will accept the replayed proof as a valid authorization from Alice for COA-B. An attacker can drain or transfer assets held by COA-B, or perform any privileged action gated behind ERC-1271 verification, without Alice's consent. The impact is unauthorized movement of on-chain assets.

### Likelihood Explanation

The preconditions are:
- A Cadence account must own more than one COA. The protocol does not prevent this; it is a supported use case.
- The attacker must observe a valid COA ownership proof for COA-A. Proofs are submitted as EVM transaction calldata and are fully public.
- The attacker must be able to call `isValidSignature` on COA-B, which is a public EVM function callable by any unprivileged EVM transaction sender.

No privileged access, leaked keys, or compromised nodes are required. The attacker's only action is re-encoding a publicly visible proof struct and submitting an EVM transaction.

### Recommendation

**Short term:** Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress`. Reject the proof if `signedData` does not contain the 20-byte EVM address of the target COA. This mirrors the EIP-712 fix of including the contract address in the signed domain.

**Long term:** Define a canonical signed message format for COA ownership proofs that commits to `evmAddress`, `CapabilityPath`, and a nonce or timestamp to prevent both cross-COA and cross-context replay.
