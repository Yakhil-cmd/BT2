### Title
Unclaimed Rewards Redirected to New Address When `change_reward_address` Is Called Without Prior Claim - (`src/staking/staking.cairo` and `src/pool/pool.cairo`)

---

### Summary

Both `change_reward_address` in the staking contract and in the pool contract update the reward recipient address without first flushing accumulated unclaimed rewards to the old address. Any rewards earned up to the moment of the address change are subsequently paid out to the **new** reward address when `claim_rewards` is eventually called, permanently depriving the old reward address of yield it legitimately earned.

---

### Finding Description

**Staking contract** (`src/staking/staking.cairo`, `change_reward_address`, lines 517–540):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    // ...
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;

    // Update reward_address and commit to storage.
    staker_info.reward_address = reward_address;   // ← no claim before this
    self.write_staker_info(:staker_address, :staker_info);
    // ...
}
``` [1](#0-0) 

When `claim_rewards` is later called, `send_rewards_to_staker` reads `staker_info.reward_address` — which is now the **new** address — and transfers the entire `unclaimed_rewards_own` balance there, including all rewards accrued while the old address was registered. [2](#0-1) 

**Pool contract** (`src/pool/pool.cairo`, `change_reward_address`, lines 505–526):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    // ...
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    let old_address = pool_member_info.reward_address;

    // Update reward_address and commit to storage.
    pool_member_info.reward_address = reward_address;   // ← no claim before this
    self.write_pool_member_info(:pool_member, :pool_member_info);
    // ...
}
``` [3](#0-2) 

`claim_rewards` in the pool contract reads `reward_address` from the stored `pool_member_info` and transfers all accumulated rewards there. [4](#0-3) 

A compounding factor: both `claim_rewards` functions authorize the **current** reward address as a valid caller.

- Staking: `caller_address == staker_address || caller_address == reward_address` [5](#0-4) 
- Pool: `caller_address == pool_member || caller_address == reward_address` [6](#0-5) 

After `change_reward_address` executes, the **old** reward address is no longer authorized to call `claim_rewards`. It has no recourse to recover the rewards it earned.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

When the reward address is a distinct entity from the staker/pool member (a common setup: e.g., a business partner, a DAO treasury, or a separate cold/hot wallet split), the staker/pool member can:

1. Accumulate a large `unclaimed_rewards_own` / `unclaimed_rewards` balance under the old reward address.
2. Call `change_reward_address(attacker_wallet)` — no rewards are disbursed.
3. Call `claim_rewards` — the entire accumulated balance, including all yield earned while the old address was registered, is transferred to `attacker_wallet`.
4. The old reward address is permanently locked out of `claim_rewards` and receives nothing.

The stolen amount is bounded only by how long rewards have been accumulating without a claim, which can be arbitrarily large.

---

### Likelihood Explanation

**Medium.** The precondition is that the reward address is a different entity from the staker/pool member key. This is a standard operational pattern (operators separate signing keys from reward recipients). The staker/pool member key is the only party that can call `change_reward_address`, so no external attacker is needed — the key holder executes the theft unilaterally. The old reward address has no on-chain mechanism to prevent or detect the change before it is too late.

---

### Recommendation

In both `change_reward_address` implementations, flush accumulated rewards to the **old** reward address before updating the stored address. Concretely, insert a `claim_rewards` / `send_rewards_to_staker` call (or its internal equivalent) immediately before overwriting `reward_address` in storage.

For the staking contract:
```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    // ...
    let mut staker_info = self.internal_staker_info(:staker_address);
    // Flush rewards to the OLD address first.
    let token_dispatcher = strk_token_dispatcher();
    self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
    // Now safe to update.
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
    // ...
}
```

Apply the same pattern in the pool contract's `change_reward_address`.

---

### Proof of Concept

**Staking contract scenario:**

1. Alice stakes STRK. Her reward address is `alice_reward` (a separate wallet she shares with a business partner).
2. Over many epochs, `staker_info.unclaimed_rewards_own` accumulates to 10,000 STRK.
3. Alice calls `staking.change_reward_address(alice_personal_wallet)`. No rewards are transferred; `staker_info.reward_address` is now `alice_personal_wallet`.
4. `alice_reward` attempts to call `staking.claim_rewards(alice_staker_address)` — it reverts with `CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS` because it is no longer the registered reward address.
5. Alice calls `staking.claim_rewards(alice_staker_address)` — 10,000 STRK is sent to `alice_personal_wallet`.
6. `alice_reward` (the business partner) receives 0 STRK despite having been the registered reward address for the entire earning period.

**Pool contract scenario (identical structure):**

1. Bob delegates to a pool with `bob_reward` as his reward address.
2. Rewards accumulate in `pool_member_info.unclaimed_rewards`.
3. Bob calls `pool.change_reward_address(bob_personal_wallet)`.
4. `bob_reward` can no longer call `pool.claim_rewards(bob)`.
5. Bob calls `pool.claim_rewards(bob)` — all accumulated rewards go to `bob_personal_wallet`. [7](#0-6) [8](#0-7)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
