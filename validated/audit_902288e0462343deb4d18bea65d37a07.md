### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` function in the `Pool` contract contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim checkpoint. Because `add_to_delegation_pool` and `exit_delegation_pool_intent` each append a new trace entry whenever called in a distinct epoch, an unprivileged pool member (or their reward address) can grow this trace without bound by making small balance-change calls across many epochs while deferring `claim_rewards`. Once the trace is large enough, every future `claim_rewards` invocation exceeds Starknet's per-transaction gas limit, permanently freezing the pool member's accumulated unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance)`: [2](#0-1) 

The underlying `insert` implementation only deduplicates entries sharing the same epoch key; calls in distinct epochs each push a new `PoolMemberBalanceCheckpoint` onto the `Vec`: [3](#0-2) 

`set_member_balance` is invoked by both `add_to_delegation_pool` (line 242) and `exit_delegation_pool_intent` (line 278): [4](#0-3) [5](#0-4) 

`add_to_delegation_pool` is callable by the pool member **or their reward address**, with no minimum amount beyond `amount > 0`: [6](#0-5) 

`claim_rewards` passes `pool_member_info.entry_to_claim_from` into `calculate_rewards` and only advances it after a successful return. If the call reverts due to gas exhaustion, `entry_to_claim_from` is never updated, so every subsequent attempt re-enters the same oversized loop: [7](#0-6) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace is large enough to cause `calculate_rewards` to exceed the Starknet transaction gas limit, `claim_rewards` reverts on every call. Because `entry_to_claim_from` is only persisted on success, the pool member's accumulated unclaimed yield is permanently unclaimable — matching the allowed impact **"Permanent freezing of unclaimed yield"** (High).

---

### Likelihood Explanation

The attack requires calling `add_to_delegation_pool` with a minimal amount (1 token unit) once per epoch while never calling `claim_rewards`. The reward address — a separate account chosen by the pool member — is also authorized to call `add_to_delegation_pool`, meaning a malicious or compromised reward address can grief the pool member without the pool member's cooperation. Even without a malicious actor, a pool member who regularly tops up their delegation across many epochs without claiming can inadvertently reach the gas limit. The cost per epoch is one token unit plus transaction fees, making a deliberate attack economically feasible over time.

---

### Recommendation

1. **Bound the loop per call**: Accept a `max_iterations` parameter in `claim_rewards` (or internally in `calculate_rewards`) and process only up to that many trace entries per transaction, persisting the updated `entry_to_claim_from` after each partial execution. This allows incremental claiming.
2. **Alternatively, cap trace growth**: Consolidate balance-change entries within the same epoch+K window (the `insert` function already does this for identical keys) and consider compacting old entries after they are fully claimed.
3. **Emit a warning or enforce a maximum trace length** at the point of insertion so that the pool member is alerted before the trace becomes unclaimable.

---

### Proof of Concept

1. Pool member Alice calls `enter_delegation_pool` with the minimum stake.
2. For each of N epochs (without ever calling `claim_rewards`), Alice (or her reward address) calls `add_to_delegation_pool(pool_member: alice, amount: 1)`. Each call in a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`.
3. After N epochs, `pool_member_epoch_balance.length()` for Alice equals N.
4. Alice calls `claim_rewards`. `calculate_rewards` enters the `while entry_to_claim_from < pool_member_trace_length` loop and iterates N times, each iteration performing multiple storage reads (`pool_member_trace.at(...)`, `find_sigma(...)`).
5. For sufficiently large N, the transaction exceeds Starknet's gas limit and reverts. `entry_to_claim_from` remains 0.
6. Every future `claim_rewards` call for Alice also reverts. Alice's accumulated yield is permanently frozen.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
