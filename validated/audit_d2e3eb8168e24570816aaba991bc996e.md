Audit Report

## Title
Logo-omission guard bypassed in `convert_proposal` when `return_self_describing_action=true` — (`rs/nns/governance/src/pb/proposal_conversions.rs`)

## Summary
When `list_proposals` is called with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`, `ProposalDisplayOptions::for_list_proposals` sets `show_action=false`. This causes `action` to be `None` in `convert_proposal`, making `is_create_service_nervous_system_proposal` always `false`, so the logo-omission guard passed to `convert_self_describing_action` is always `false`. Full logo bytes are returned in `self_describing_action.value` regardless of `omit_large_fields`, defeating the documented purpose of that flag and enabling the response to exceed the IC 2 MB query limit.

## Finding Description
`ProposalDisplayOptions::for_list_proposals` sets `show_action = !return_self_describing_action`. [1](#0-0) 

When `return_self_describing_action=true`, `show_action=false`, so `convert_proposal` sets `action = None`: [2](#0-1) 

`is_create_service_nervous_system_proposal` is then derived from that `None` value and is always `false`: [3](#0-2) 

The guard passed to `convert_self_describing_action` is therefore `false && omit_create_service_nervous_system_large_fields()` = `false`: [4](#0-3) 

With `omit_create_service_nervous_system_logos=false`, `paths_to_omit` is always empty and no fields are stripped: [5](#0-4) 

The `self_describing_action` blob is stored at proposal-submission time with the full logo bytes embedded: [6](#0-5) 

The `omit_large_fields` flag is explicitly documented to prevent the response from exceeding the message size limit: [7](#0-6) 

## Impact Explanation
The IC query response size limit is 2 MB. A `CreateServiceNervousSystem` proposal can carry a logo up to ~256 KiB. `list_proposals` returns up to 100 proposals. With multiple SNS proposals carrying logos (a routine mainnet condition given every SNS launch produces one), the unstripped `self_describing_action` blobs push the serialized response over 2 MB, causing the query to trap and return an error. The `omit_large_fields` flag — whose sole documented purpose is to prevent exactly this — is silently ineffective in this code path. This constitutes an application-level DoS of the NNS governance `list_proposals` API for this parameter combination, matching the **High** impact class: *Application/platform-level DoS with concrete user or protocol harm*.

## Likelihood Explanation
The trigger is a standard, unauthenticated query call with two boolean flags set. No privileged access, governance majority, or key material is required. The precondition — multiple `CreateServiceNervousSystem` proposals with non-trivial logos — is routinely satisfied on mainnet (every SNS launch produces one). Any caller can reproduce this deterministically.

## Recommendation
Derive `is_create_service_nervous_system_proposal` from the raw `pb::Proposal.action` field directly, not from the already-converted (and potentially `None`) `action` output:

```rust
let is_create_service_nervous_system_proposal = matches!(
    item.action.as_ref(),
    Some(pb::proposal::Action::CreateServiceNervousSystem(_))
);
```

This makes type-detection independent of `show_action`.

## Proof of Concept
State-machine test:
1. Submit a `CreateServiceNervousSystem` proposal with a 256 KiB base64-encoded logo.
2. Call `list_proposals` with `omit_large_fields=Some(true)` and `return_self_describing_action=Some(true)`.
3. Assert that the returned `self_describing_action.value` map entry for `"logo"` is `SelfDescribingValue::Null` (currently it is the full blob, demonstrating the bypass).
4. Repeat with enough proposals to push the total serialized response over 2 MB and assert the query traps.
