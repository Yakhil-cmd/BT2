### Title
Missing Participant-Set Existence Check Before Presign Allows Mismatched Triple Usage - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The OT-based ECDSA `presign()` function explicitly omits the check that the presigning participant set matches the participant set that generated the Beaver triples. This is directly analogous to the reported vulnerability: a critical operation proceeds without verifying that the prerequisite material (triples) was generated by the same set of participants, allowing a malicious coordinator to mix participant sets and corrupt presignature outputs that honest parties will accept.

### Finding Description
In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign()` initialization function contains an explicit acknowledgment that a critical existence/membership check is omitted:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The function does validate that the threshold values match between the presign arguments and the triples:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [2](#0-1) 

However, it does **not** verify that the participants in the presign session are the same participants who generated `triple0` and `triple1`. The `TriplePublic` struct carries a `threshold` but the participant set that produced the triple is not cross-checked against the presigning participant set.

The analogy to the reported Solidity bug is exact:
- **Solidity**: `address(token).call(...)` returns `success = true` even when the token contract no longer exists, because the EVM does not check contract existence before low-level calls.
- **Here**: `presign()` proceeds and produces output even when the participant set does not match the triple generation set, because the code does not check participant-set existence/membership before using the triple material.

The comment's justification — *"presumably they need to have been present in order to have shares"* — is an assumption, not an enforced invariant. A malicious coordinator controls which participants are invited to each phase and can trivially violate this assumption.

### Impact Explanation
A malicious coordinator can:
1. Run triple generation with participant set `{A, B, C}`.
2. Initiate a presign session with participant set `{A, B, D}`, where `D` was never part of triple generation.
3. Supply `D` with fabricated or zero triple shares.
4. The `presign()` function accepts the call without detecting the mismatch.
5. The resulting presignature is cryptographically inconsistent — it was computed with triple material from one participant set and signing key shares from another.

Honest parties `A` and `B` will accept and store this corrupted `PresignOutput`, believing it is valid. When they later attempt to sign using it, the signature will either be invalid or the protocol will abort, permanently denying signing capability for that presignature slot.

**Mapped impact**: High — Corruption of presign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs.

### Likelihood Explanation
The coordinator role is explicitly part of the threat model (the prompt lists "malicious coordinator" as a valid attacker). The coordinator controls session membership for both triple generation and presigning. No cryptographic barrier prevents the coordinator from using different participant sets across the two phases. The omission is self-documented in the code, confirming it is a known gap rather than an intentional design choice with a compensating control.

### Recommendation
Before proceeding with the presign protocol, verify that every participant in the presigning session was also a participant in the triple generation sessions that produced `triple0` and `triple1`. The `TriplePublic` struct should carry the participant set (or a commitment to it) used during generation, and `presign()` should assert that the current participant set is a subset of (or equal to) that recorded set. This mirrors the fix recommended in the Solidity report: check existence before performing the operation.

### Proof of Concept
1. Coordinator runs `generate_triple` with participants `[A, B, C]`, producing `(triple_share_A, triple_pub)`, `(triple_share_B, triple_pub)`, `(triple_share_C, triple_pub)`.
2. Coordinator calls `presign` with participants `[A, B, D]`, passing `D` a fabricated `TripleShare` (e.g., a random scalar) and the same `triple_pub` (threshold matches, so the threshold check at line 43 passes).
3. `presign()` at line 49–57 only checks that `D` is in the presign participant list and that threshold values match — it does not check that `D` was in the triple generation participant list.
4. The protocol runs to completion, producing a `PresignOutput` for `A`, `B`, and `D` that is cryptographically inconsistent because `D`'s triple contribution is fabricated.
5. Any subsequent `sign` call using this `PresignOutput` will produce an invalid signature or abort, permanently denying signing for this presignature. [3](#0-2)
