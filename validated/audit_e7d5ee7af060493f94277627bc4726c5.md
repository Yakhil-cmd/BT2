The production build configuration (`CARGO_FEATURES_BUILD = "contract"`) does not include `error_refund`. The vulnerability is real and confirmed. Here is the full assessment:

---

### Title
ETH Permanently Lost When `ft_transfer` Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

### Summary
When the Aurora Engine is compiled without the `error_refund` feature (which is the production default), a user calling the `ExitToNear` precompile with ETH value targeting an unregistered NEAR account will have their ETH permanently burned from the EVM with no corresponding credit on NEAR and no refund mechanism.

### Finding Description

The production build is defined in `Makefile.toml` as:

```
CARGO_FEATURES_BUILD = "contract"
``` [1](#0-0) 

`error_refund` is not a default feature in either `engine/Cargo.toml` or `engine-precompiles/Cargo.toml`: [2](#0-1) [3](#0-2) 

In `ExitToNear::run`, when `error_refund` is not compiled in, `callback_args.refund` is hardcoded to `None`: [4](#0-3) 

For the base-token (ETH) path with flag `0x0` and no message (legacy `ft_transfer`), `exit_base_token_to_near` also returns `transfer_near_args = None`: [5](#0-4) 

This means `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()` (both fields `None`). The branch at line 470 then selects `PromiseArgs::Create` — a bare promise with **no callback**: [6](#0-5) 

`ExitToNearPrecompileCallbackArgs` derives `Default` with both fields as `Option::None`: [7](#0-6) 

The callback `exit_to_near_precompile_callback`, which is the only place a refund can be issued, is never scheduled: [8](#0-7) 

### Impact Explanation

ETH is deducted from the user's EVM balance via `context.apparent_value` at the EVM execution layer. The `ft_transfer` promise to the eth-connector NEP-141 is dispatched without a callback. If the promise fails (e.g., recipient not registered with the NEP-141 storage deposit), the ETH is neither credited on NEAR nor refunded to the EVM sender. The total ETH supply across EVM and NEAR is permanently reduced — this is **permanent freezing/destruction of funds** (Critical).

### Likelihood Explanation

The production WASM is built with `contract` feature only, not `error_refund`. Any user who calls `ExitToNear` with ETH value targeting a NEAR account that has not registered a storage deposit with the eth-connector NEP-141 will trigger this path. This is a realistic user mistake (e.g., sending to a freshly created NEAR account that has never interacted with the eth-connector). The existing test suite explicitly acknowledges this outcome: [9](#0-8) 

### Recommendation

Enable `error_refund` in the production build (`CARGO_FEATURES_BUILD = "contract,error_refund"`), or add a pre-flight check inside `ExitToNear::run` that verifies the recipient is registered with the eth-connector NEP-141 before deducting ETH from the EVM balance, reverting the EVM call if not.

### Proof of Concept

1. Deploy Aurora Engine WASM compiled with `contract` only (no `error_refund`).
2. Fund an EVM address with ETH.
3. Submit an EVM transaction calling the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with `value = N wei`, input = `[0x00] ++ "unregistered.near"` (flag=0x0, no refund address field, unregistered NEAR recipient).
4. Observe: EVM balance decreases by `N`. The `ft_transfer` promise to the eth-connector fails (recipient not registered). No callback fires. NEP-141 balance of `unregistered.near` remains 0. EVM balance is not restored.

The test `test_exit_to_near_eth_refund` in `engine-tests/src/tests/erc20_connector.rs` already encodes this exact scenario and asserts `INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT` as the expected post-failure balance when `error_refund` is absent. [10](#0-9)
