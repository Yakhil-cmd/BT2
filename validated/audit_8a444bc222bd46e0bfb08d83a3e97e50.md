### Title
`protocolUnpausePool` Sets Wrong Pause State (1 Instead of 0) When Pool Was Originally Unpaused — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` always calls `setPause(1)` (admin-paused) regardless of the pool's pre-protocol-pause state. When the pool was originally at state `0` (fully unpaused), a protocol pause/unpause cycle leaves the pool permanently stuck in admin-paused state until the admin takes an extra manual action. This is the direct Metric OMM analog of the external "wrong return value" bug class.

---

### Finding Description

The pause state machine uses three values:

| Value | Meaning |
|---|---|
| `0` | Fully unpaused — normal operation |
| `1` | Admin-paused — all pool actions blocked |
| `2` | Protocol-paused — all pool actions blocked |

`protocolPausePool` accepts pools in **either** state `0` or state `1` and moves them to state `2`: [1](#0-0) 

`protocolUnpausePool` then **always** calls `setPause(1)`, hardcoding the post-unpause state to admin-paused regardless of what the state was before the protocol pause: [2](#0-1) 

The pre-pause state is never saved anywhere in factory storage, so there is no way for `protocolUnpausePool` to restore it. The wrong literal `1` is the direct analog of the external "over1 instead of over" bug.

---

### Impact Explanation

After a protocol pause/unpause cycle on a pool that was originally at state `0` (unpaused):

- The pool lands in state `1` (admin-paused).
- The `whenNotPaused` modifier on `swap`, `addLiquidity`, and `removeLiquidity` blocks all user operations.
- Users cannot swap, deposit, or withdraw their principal until the pool admin separately calls `unpausePool`.

If the admin is unavailable (multisig quorum lost, admin contract broken, or admin simply unaware of the requirement), the pool remains permanently unusable and LP principal is inaccessible. This satisfies the "unusable withdraw/swap/liquidity flows" impact gate. [3](#0-2) 

---

### Likelihood Explanation

The trigger is the factory owner legitimately calling `protocolUnpausePool` on any pool that was in state `0` at the time of the protocol pause. This is the normal operational scenario (emergency pause → incident resolved → unpause). No malicious intent is required; the bug fires on every such cycle.

---

### Recommendation

`protocolUnpausePool` should restore the pool to state `0` (fully unpaused), not state `1`. The simplest fix:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // was: setPause(1)
}
```

Alternatively, store the pre-pause state in factory storage when `protocolPausePool` is called and restore it in `protocolUnpausePool`.

---

### Proof of Concept

1. Pool is deployed and fully operational — pause state = `0`.
2. Factory owner calls `protocolPausePool(pool)` → state transitions `0 → 2`. All swaps/liquidity blocked.
3. Incident resolved; factory owner calls `protocolUnpausePool(pool)` → state transitions `2 → 1` (wrong: should be `0`).
4. Users attempt `swap` / `addLiquidity` / `removeLiquidity` → all revert because `whenNotPaused` sees state `1`.
5. Pool is permanently stuck in admin-paused state. Admin must separately call `unpausePool` to reach state `0`, an action that was never needed before the protocol pause and that the admin may not know is required. [4](#0-3)
