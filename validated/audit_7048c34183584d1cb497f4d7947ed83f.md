### Title
Missing EIP-684 Storage Collision Check in CREATE/CREATE2 Enables Attacker to Pre-Corrupt Deployed Contract State — (`evm_interpreter/src/ee_trait_impl.rs`)

---

### Summary

ZKsync OS's EVM interpreter omits the EIP-684 storage-emptiness check during CREATE/CREATE2 collision detection. An attacker can pre-populate storage at a deterministic CREATE2 address by deploying and self-destructing a contract there first. When a victim subsequently deploys at the same address, ZKsync OS allows the deployment to proceed — unlike Ethereum, which would reject it — causing the victim's contract to silently inherit the attacker's pre-set storage values.

---

### Finding Description

In `before_executing_frame` inside `evm_interpreter/src/ee_trait_impl.rs`, the CREATE/CREATE2 collision guard checks only `deployee_code_len != 0 || deployee_nonce != 0`. The code itself acknowledges the missing storage check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // Burn all gas and return CreateCollision
``` [1](#0-0) 

EIP-684 mandates that a CREATE/CREATE2 deployment **must fail** if the target address has non-empty storage. ZKsync OS skips this check entirely. The omission is independently confirmed by multiple disabled test entries in the test index files:

```yaml
RevertInCreateInInitCreate2Paris.json:
  enabled: false
  comment: We do not check for storage collisions

create2collisionStorageParis.json:
  enabled: false
  comment: We do not check storage for collision
``` [2](#0-1) [3](#0-2) 

The `EvmError::CreateCollision` variant exists and is correctly emitted for nonce/code collisions, but the storage-collision path is never reached. [4](#0-3) 

---

### Impact Explanation

The attack mirrors the external report's griefing pattern — an attacker front-runs a victim's deterministic identifier claim — but operates at the EVM deployment layer:

1. A factory contract exposes CREATE2 with a user-supplied salt.
2. The attacker observes the victim's pending deployment (salt `S`, target address `A`).
3. The attacker calls the factory with salt `S`, deploys a contract at `A`, writes malicious values into storage slots (e.g., `initialized = true`, `owner = attacker`), then self-destructs the contract. After self-destruct, code and nonce are cleared but storage persists.
4. The victim calls the factory with salt `S`. The nonce/code check passes (both are zero). ZKsync OS proceeds with the constructor.
5. On Ethereum, EIP-684 would reject this deployment because storage is non-empty. On ZKsync OS, the deployment **succeeds**, and the victim's contract inherits the attacker's pre-set storage.
6. The victim's contract is live but operates with corrupted initial state — initialization guards are bypassed, access-control slots are pre-owned, or invariants are silently broken from block 0.

The victim has no indication anything went wrong; the deployment transaction succeeds and emits normal events.

---

### Likelihood Explanation

The attack is realistic wherever:
- A factory contract accepts arbitrary user-supplied salts (common pattern: `CREATE2` factories, minimal proxy deployers, counterfactual wallet deployers).
- The victim's target address is predictable from public parameters (salt, initcode hash, factory address — all observable on-chain or in the mempool).
- The attacker can front-run within the same block or across blocks (self-destruct in a prior block, deploy-corrupt in a later block).

No privileged access is required. Any unprivileged EOA can execute the attack.

---

### Recommendation

Implement the EIP-684 storage-emptiness check before executing a constructor frame. Before proceeding with deployment, verify that the target address has no non-zero storage slots. If a full storage scan is too expensive, at minimum track whether any storage was ever written to the address (a single boolean flag per address in the account properties) and treat a non-empty flag as a collision.

---

### Proof of Concept

```
// Step 1 – Attacker front-runs
factory.deploy(salt=S)          // deploys at address A
//   constructor: SSTORE(slot=0, value=0xdeadbeef)  // poison storage
//   then: SELFDESTRUCT(beneficiary)
// After tx: A.code = 0, A.nonce = 0, A.storage[0] = 0xdeadbeef

// Step 2 – Victim deploys
factory.deploy(salt=S)          // same factory, same salt → same address A
// ZKsync OS: code_len==0 && nonce==0 → collision check PASSES
// Constructor executes; A.storage[0] is already 0xdeadbeef
// If victim's constructor does: require(storage[0] == 0) → REVERT (DoS)
// If victim's constructor does not check → deployed with corrupted state
```

On Ethereum, step 2 would revert at the collision guard before the constructor ever runs. On ZKsync OS, the constructor runs against pre-poisoned storage, either silently corrupting the contract or causing an unexpected revert that consumes all gas — matching the griefing/DoS impact class of the reference report.
