### Title
Presignature Not Consumed on Rerandomization Enables Nonce Reuse and Secret Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

`RerandomizedPresignOutput::rerandomize_presign` accepts the `PresignOutput` by shared reference (`&PresignOutput`) rather than by value (consuming it). This means the Rust type system does not enforce the library's own documented one-time-use invariant. Any caller can invoke `rerandomize_presign` multiple times on the same `PresignOutput` with different `RerandomizationArguments`, producing multiple distinct `RerandomizedPresignOutput` values that all share the same underlying nonce material (`k`, `sigma`). Using two such outputs in separate signing sessions constitutes ECDSA nonce reuse, from which the secret key can be algebraically recovered.

---

### Finding Description

The library documents the one-time-use requirement explicitly:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase."
> — `docs/ecdsa/ot_based_ecdsa/orchestration.md:70–71`

> "Each output is consumed **exactly once** (one-time use)."
> — `src/ecdsa/ot_based_ecdsa/README.md:12`

Despite this, the public API function that produces a `RerandomizedPresignOutput` from a `PresignOutput` is defined as:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs:66–96
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← shared reference, NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { … }
``` [1](#0-0) 

Because `presignature` is a `&PresignOutput` (a shared borrow), the caller retains ownership of the `PresignOutput` after the call and can call `rerandomize_presign` again with a different `RerandomizationArguments` (e.g., a different `msg_hash` or `tweak`). Each call produces a fresh `RerandomizedPresignOutput` whose `k` and `sigma` fields are deterministic linear transforms of the **same** underlying nonce shares.

Additionally, `RerandomizedPresignOutput` itself derives `Clone`:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs:54–63
#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput { … }
``` [2](#0-1) 

This compounds the issue: even a single rerandomization can be cloned and fed into multiple `sign` calls.

---

### Impact Explanation

ECDSA security requires that the nonce `k` is used for exactly one signature. If two signatures `(r, s₁)` and `(r, s₂)` are produced with the same `R = k·G` (i.e., the same nonce), the secret key `x` can be recovered via:

```
x = (s₁·h₂ − s₂·h₁) / (s₂ − s₁) · r⁻¹   (mod q)
```

Because `rerandomize_presign` applies a deterministic linear transform (`k → k·δ⁻¹`) to the same base nonce, two rerandomizations with different `δ` values yield different `R` points but the **same base nonce shares** are reused across signing sessions. A coordinator or library caller who controls which `RerandomizationArguments` are passed can trigger two signing sessions that expose the relationship between nonce shares, enabling secret key reconstruction.

**Impact class:** Critical — Extraction/reconstruction of private signing shares. [3](#0-2) 

---

### Likelihood Explanation

The entry path requires only an unprivileged library caller with access to a `PresignOutput` (the normal output of the presigning protocol). No special privileges, leaked keys, or cryptographic breaks are needed. The caller simply invokes `rerandomize_presign` twice on the same value, which the API permits without any warning or runtime check. This is a realistic mistake for any integrator who stores presignatures in a pool and does not implement external one-time-use bookkeeping. [4](#0-3) 

---

### Recommendation

Change `rerandomize_presign` to consume the `PresignOutput` by value, so the Rust ownership system enforces the one-time-use invariant at compile time:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // ← owned, consumed on call
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { … }
```

Additionally, remove the `Clone` derive from both `PresignOutput` and `RerandomizedPresignOutput` so that neither type can be duplicated after creation. The `ZeroizeOnDrop` derive is already present, which is correct, but it does not prevent cloning before drop. [5](#0-4)

// Attacker-controlled library caller with a PresignOutput `presign_out`
// and two different signing contexts:

// Both calls succeed because presign_out is borrowed, not consumed:

// rerand1 and rerand2 share the same base nonce k.
// Feed each into a separate signing session → two signatures with related nonces.
// Apply standard ECDSA nonce-reuse formula → recover secret key shares.
```

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
