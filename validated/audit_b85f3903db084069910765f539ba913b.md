### Title
Missing `assert_staker_is_active()` in `enter_delegation_pool_from_staking_contract()` Allows Delegation to Removed Staker's Pool — (`src/pool/pool.cairo`)

---

### Summary

The `enter_delegation_pool_from_staking_contract()` function in `pool.cairo` is missing the `assert_staker_is_active()` check that is present in `enter_delegation_pool()`. This is the direct analog to the external report's pattern: a validation that guards a protocol invariant is applied in one entry path but omitted in a parallel entry path. A pool member can switch their delegation into a pool whose staker has already been removed, causing their funds to land in a dead pool with no ability to earn rewards.

---

### Finding Description

`enter_delegation_pool()` is the standard user-facing entry point for new delegators. It enforces the staker-active invariant at line 186:

```cairo
fn enter_delegation_pool(
    ref self: ContractState, reward_address: ContractAddress, amount: Amount,
) {
    self.assert_staker_is_active();   // ← invariant check present
    ...
}
``` [1](#0-0) 

`enter_delegation_pool_from_staking_contract()` is the parallel entry point invoked by the staking contract during a pool-switch (`switch_staking_delegation_pool`). It **omits** the same check:

```cairo
fn enter_delegation_pool_from_staking_contract(
    ref self: ContractState, amount: Amount, data: Span<felt252>,
) {
    assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
    self.assert_caller_is_staking_contract();
    // ← NO assert_staker_is_active() here
    ...
}
``` [2](#0-1) 

`assert_staker_is_active()` simply checks the `staker_removed` flag, which is set to `true` only when `set_staker_removed()` is called (during `unstake_action()`):

```cairo
fn assert_staker_is_active(self: @ContractState) {
    assert!(self.is_staker_active(), "{}", Error::STAKER_INACTIVE);
}
fn is_staker_active(self: @ContractState) -> bool {
    !self.staker_removed.read()
}
``` [3](#0-2) 

The `staker_removed` flag is set to `true` here:

```cairo
fn set_staker_removed(ref self: ContractState) {
    self.assert_caller_is_staking_contract();
    assert!(!self.staker_removed.read(), "{}", Error::STAKER_ALREADY_REMOVED);
    self.staker_removed.write(true);
    ...
}
``` [4](#0-3) 

The switch flow that reaches the vulnerable function is:

1. Pool member calls `switch_delegation_pool()` on the source pool.
2. Source pool calls `staking_pool_dispatcher.switch_staking_delegation_pool()` on the staking contract.
3. Staking contract calls `enter_delegation_pool_from_staking_contract()` on the **destination** pool — without the destination pool's staker-active status being validated at the pool level. [5](#0-4) 

---

### Impact Explanation

When a pool member switches into a pool whose staker has been removed (`staker_removed = true`):

- Their funds are recorded in the dead pool's balance trace.
- The removed staker can no longer attest, so no rewards are ever distributed to this pool via `update_rewards_from_staking_contract`.
- The pool member must go through the full exit process (`exit_delegation_pool_intent` + `exit_delegation_pool_action`) to recover principal, losing all yield they would have earned during the exit wait window.

This constitutes **theft of unclaimed yield** (rewards permanently lost for the exit window period) and **temporary freezing of funds** (principal locked for the exit wait window duration, defaulting to one week). [6](#0-5) 

---

### Likelihood Explanation

The attack path is reachable by any unprivileged pool member:

1. Staker A calls `unstake_intent()` then `unstake_action()` — pool A's `staker_removed` becomes `true`.
2. Pool member in pool B calls `exit_delegation_pool_intent()` to create an undelegate intent.
3. Pool member calls `switch_delegation_pool(to_staker: staker_A, to_pool: pool_A, amount: ...)`.
4. The staking contract calls `pool_A.enter_delegation_pool_from_staking_contract()` — no staker-active check fires.
5. Pool member's funds are now in pool A with no active staker.

No privileged access, leaked keys, or external dependencies are required. The only precondition is that a staker has completed `unstake_action()`, which is a normal protocol lifecycle event.

---

### Recommendation

Add `self.assert_staker_is_active()` at the top of `enter_delegation_pool_from_staking_contract()`, mirroring the guard in `enter_delegation_pool()`:

```cairo
fn enter_delegation_pool_from_staking_contract(
    ref self: ContractState, amount: Amount, data: Span<felt252>,
) {
    assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
    self.assert_caller_is_staking_contract();
    self.assert_staker_is_active();   // ← add this
    ...
}
``` [7](#0-6) 

---

### Proof of Concept

```
1. Deploy staking contract + pool A (staker A) + pool B (staker B).
2. Pool member delegates to pool B.
3. Staker A calls unstake_intent(), waits exit_wait_window, calls unstake_action().
   → pool_A.staker_removed == true
4. Pool member calls pool_B.exit_delegation_pool_intent(amount).
5. Pool member calls pool_B.switch_delegation_pool(to_staker=staker_A, to_pool=pool_A, amount).
   → staking contract calls pool_A.enter_delegation_pool_from_staking_contract(amount, data)
   → No revert — staker_active check is absent
6. Pool member's balance is now in pool A.
7. No attestations ever happen for staker A → no rewards ever distributed to pool A.
8. Pool member calls pool_A.exit_delegation_pool_intent(amount) + exit_delegation_pool_action()
   after the exit window to recover principal — all yield for the window is permanently lost.
```

            // Asserts.

            let old_delegated_stake = self.get_last_member_balance(:pool_member);
            let total_amount = old_delegated_stake + pool_member_info.unpool_amount;
            assert!(amount <= total_amount, "{}", GenericError::AMOUNT_TOO_HIGH);

            // Notify the staking contract of the removal intent.

            // Edit the pool member to reflect the removal intent, and write to storage.
            if amount.is_zero() {
