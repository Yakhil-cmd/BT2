### Title
Gateway `skip_stateful_validations` Incorrectly Skips `__validate__` for Fee-Escalation Invoke Transactions from Undeployed Accounts - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator silently skips the `__validate__` entry-point call for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. The check returns `true` for **any** account with any transaction in the pool, not specifically a `deploy_account`. An attacker can first submit a valid invoke (nonce=1) to seed the pool, then submit a fee-escalation replacement with an **invalid signature** — the replacement bypasses `__validate__` entirely and is admitted to the mempool.

### Finding Description

The `skip_stateful_validations` function is designed to improve UX for the simultaneous `deploy_account + invoke` flow. When an invoke transaction has `nonce == 1` and the on-chain account nonce is `0`, it queries `account_tx_in_pool_or_recent_block`. If that returns `true`, the function returns `true` (skip validation), and `run_validate_entry_point` is called with `skip_validate = true`, setting `execution_flags.validate = false` so `__validate__` is never executed. [1](#0-0) 

The comment in the code states the assumption:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This assumption is broken for **fee-escalation replacements**. `account_tx_in_pool_or_recent_block` checks for any account presence: [2](#0-1) 

The attack path:

1. **Seed step**: Attacker submits `invoke_1` from undeployed address A: `nonce=1`, `tip=100`, **valid signature**. At this point `account_tx_in_pool_or_recent_block(A)` is `false`, so `skip_stateful_validations` returns `false`, and `__validate__` runs and passes. `invoke_1` enters the mempool.

2. **Exploit step**: Attacker submits `invoke_2` from the same address A: `nonce=1`, `tip=200` (higher tip for fee escalation), **invalid signature** (one that would fail `__validate__`).
   - `validate_state_preconditions` passes: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap=200`.
   - `validate_by_mempool` passes: `validate_tx` checks for duplicate hash and fee escalation — the higher tip satisfies fee escalation, no signature check.
   - `skip_stateful_validations`: `tx.nonce() == Nonce(Felt::ONE)` ✓, `account_nonce == Nonce(Felt::ZERO)` ✓, `account_tx_in_pool_or_recent_block(A)` → **`true`** (because `invoke_1` is in the pool) → returns `true`.
   - `run_validate_entry_point` is called with `skip_validate = true` → `__validate__` is **never called**.
   - `invoke_2` replaces `invoke_1` in the mempool. [3](#0-2) [4](#0-3) 

The mempool now contains `invoke_2` with an invalid signature that bypassed account validation. The invariant — every transaction admitted to the mempool must have passed `__validate__` — is violated.

This is the direct analog to the external report's root cause: just as `payable(address(stEth)).call{value: _initialETH}("")` silently ignores a Lido revert and lets execution continue with a zero deposit, `skip_stateful_validations` silently skips `__validate__` based on an incorrect assumption (any pool presence = deploy_account present), letting an unauthorized transaction continue through admission.

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An attacker can admit an invoke transaction with an arbitrary (invalid) signature to the mempool without passing `__validate__`. The gateway's core security invariant — that every admitted transaction has been validated by the account contract — is broken for fee-escalation replacements targeting undeployed accounts. The admitted invalid transaction will fail during blockifier execution (the batcher always runs `__validate__`), but it occupies mempool capacity and can be used to displace legitimate transactions via fee escalation.

### Likelihood Explanation

**Medium.** The attack requires two sequential transactions from the same address with the second having a higher tip. No privileged access is needed. The only precondition is that the target account is undeployed (nonce=0 on-chain), which is the normal state for any new account. The `max_allowed_nonce_gap` of 200 ensures the nonce=1 seed transaction is always accepted. [5](#0-4) 

### Recommendation

`skip_stateful_validations` must verify that the account has a **`deploy_account` transaction specifically** in the pool, not just any transaction. The current `account_tx_in_pool_or_recent_block` API is insufficient for this purpose. Options:

1. Add a dedicated `has_deploy_account_in_pool(address)` query to the mempool that checks the transaction type.
2. Only skip `__validate__` when the account has **no prior transactions** in the pool (i.e., `invoke_1` is the first submission, not a replacement). This can be checked by verifying `!tx_pool.contains_account(address)` at the time of the skip decision.
3. Always run `__validate__` for fee-escalation replacements regardless of the skip condition.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
