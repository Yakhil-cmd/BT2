### Title
`EVM.validateCOAOwnershipProof` Does Not Bind Signatures to `evmAddress`, Enabling Cross-Address Replay of COA Ownership Proofs - (File: fvm/evm/stdlib/contract.cdc)

### Summary
`EVM.validateCOAOwnershipProof` is designed to prove that a Cadence account owns a specific EVM address (COA). The function verifies cryptographic signatures over `signedData`, then separately checks that the COA resource at the given path matches `evmAddress`. However, it never enforces that `signedData` commits to `evmAddress`. A signature produced for one COA can therefore be replayed verbatim to forge a valid ownership proof for any other COA owned by the same Cadence account. The code itself acknowledges this gap in a comment but does not fix it.

### Finding Description
`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. **Signature check** (lines 1082–1086): verifies that the provided signatures are valid over `signedData` using the Cadence account's keys.
2. **Address check** (lines 1095–1110): verifies that the COA resource stored at `path` has an EVM address equal to the caller-supplied `evmAddress`. [1](#0-0) [2](#0-1) 

These two checks are never joined: the signature is verified over `signedData` alone, and `signedData` is never required to include `evmAddress`. The code explicitly documents this gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [3](#0-2) 

This is the direct Flow analog of the Vader H-13 bug: in Vader, the veto protection reads from `calldata` (which holds only function arguments) instead of `signatures` (which holds the function selector), so the selector check always sees zero bytes and the protection is void. Here, the ownership-binding protection reads from `signedData` (which holds only the payload the user chose to sign) instead of a commitment that includes `evmAddress`, so the address-binding protection is void for any caller who supplies a different `path`/`evmAddress` pair.

**Concrete replay path:**

Suppose a Cadence account owns two COAs: `COA_A` published at `/public/coaA` and `COA_B` published at `/public/coaB`. A victim signs arbitrary bytes `D` (not encoding any address) to prove ownership of `COA_A`. An attacker who observes the resulting proof `(address, /public/coaA, D, keyIndices, sigs, COA_A_bytes)` can immediately construct a forged proof for `COA_B`:

```
EVM.validateCOAOwnershipProof(
    address:    victim_cadence_address,
    path:       /public/coaB,          // ← changed
    signedData: D,                     // ← same bytes, no address in them
    keyIndices: keyIndices,            // ← same
    signatures: sigs,                  // ← same
    evmAddress: COA_B_bytes            // ← changed
)
```

Step 1 passes because `sigs` are valid over `D` with the victim's keys.  
Step 2 passes because the COA at `/public/coaB` does have address `COA_B_bytes`.  
Result: `ValidationResult(isValid: true)` — a forged proof for `COA_B`.

The function is `access(all)`, so any unprivileged script or transaction can invoke it. [4](#0-3) 

The EVM precompile path (`verifyCOAOwnershipProof` called from `coa.sol`) is **not** affected because it always passes `address(this)` as `evmAddress`, binding the address at the EVM execution layer. [5](#0-4) [6](#0-5) 

### Impact Explanation
Any off-chain or on-chain Cadence authentication flow that calls `EVM.validateCOAOwnershipProof` directly (rather than through the EVM precompile) can be bypassed. An attacker who has observed a single valid proof for any COA of a target Cadence account can forge a valid ownership proof for every other COA that account holds. In the ERC-1271 context this means the attacker can make an EVM contract believe the victim has signed a message with `COA_B` when the victim only ever signed it in the context of `COA_A`, enabling unauthorized authentication or authorization in any EVM contract that relies on `isValidSignature` backed by this precompile path.

### Likelihood Explanation
The attack requires (a) the victim to hold more than one COA under the same Cadence account, and (b) the attacker to have observed a valid proof for one of those COAs (e.g., from a prior on-chain transaction). Both conditions are realistic as multi-COA accounts become more common and proofs are submitted on-chain. The entry point is a public, unprivileged Cadence script call with no special role required.

### Recommendation
Enforce that `signedData` commits to `evmAddress` before signature verification. The simplest fix is to prepend `evmAddress` to `signedData` inside `validateCOAOwnershipProof` before calling `keyList.verify`, so that a signature produced for one EVM address is cryptographically invalid for any other:

```cadence
let boundData = evmAddress.toConstantSized<[UInt8; 20]>()!.toVariableSized().concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Alternatively, document a mandatory convention that callers must include `evmAddress` in `signedData` and enforce it with a runtime check.

### Proof of Concept
1. Deploy two COAs under one Cadence account; publish them at `/public/coaA` and `/public/coaB`.
2. Sign arbitrary bytes `D` (no address content) with the account's key to produce `sigs`.
3. Verify the original proof succeeds: `EVM.validateCOAOwnershipProof(address, /public/coaA, D, [0], [sigs], COA_A_bytes)` → `isValid: true`.
4. Without any new signature, call: `EVM.validateCOAOwnershipProof(address, /public/coaB, D, [0], [sigs], COA_B_bytes)` → `isValid: true`.
5. Step 4 succeeds because the signature check (line 1082) only verifies `sigs` over `D`, and the address check (line 1095) only verifies that the resource at `/public/coaB` has address `COA_B_bytes`; neither check ties the signature to the address. [1](#0-0) [2](#0-1)
