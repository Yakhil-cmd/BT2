Audit Report

## Title
Off-by-One Timestamp Boundary in SNS Governance `register_vote` Allows Voting at Exact Deadline - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `register_vote` function uses a strict `>` comparison at line 3916 to gate votes, while the canonical `accepts_vote` predicate uses a strict `<` comparison at line 2102 of `rs/sns/governance/src/proposal.rs`. At exactly `now_seconds == deadline`, `register_vote` accepts the vote while `accepts_vote` considers voting closed. This allows an attacker to cast a vote in the final round that bypasses wait-for-quiet deadline extension and is immediately counted toward a binding governance decision.

## Finding Description
In `rs/sns/governance/src/governance.rs` at line 3916:
```rust
let deadline = proposal.get_deadline_timestamp_seconds();
if now_seconds > deadline {   // passes when now_seconds == deadline
    return Err(...);
}
``` [1](#0-0) 

In `rs/sns/governance/src/proposal.rs` at line 2102:
```rust
pub fn accepts_vote(&self, now_seconds: u64) -> bool {
    now_seconds < self.get_deadline_timestamp_seconds()  // false when now_seconds == deadline
}
``` [2](#0-1) 

At `now_seconds == deadline`, the `register_vote` guard does not fire, so the vote is cast and `process_proposal` is called. Inside `process_proposal`, the tally is recomputed because `status() == Open` is still true, and `can_make_decision` sees the proposal as expired (since `accepts_vote` returns false), causing an immediate binding decision: [3](#0-2) 

Critically, because `accepts_vote` returns false at this moment, the wait-for-quiet algorithm cannot extend the deadline — the extension logic depends on the proposal still being open. An attacker who deliberately delays their vote to the exact deadline round can therefore flip the tally without giving opponents the opportunity to respond via a deadline extension.

By contrast, NNS governance delegates the deadline check directly to `accepts_vote`, making it consistent: [4](#0-3) 

## Impact Explanation
This is a **High** severity governance impact. An attacker holding a neuron with sufficient voting power can cast a decisive vote in the single IC round where `env.now() == deadline`, bypassing the wait-for-quiet mechanism that would otherwise extend the deadline and allow counter-votes. This can flip an SNS governance proposal — enabling unauthorized parameter changes, treasury transfers, or canister upgrades — that would have been blocked or extended under correct behavior. This matches the allowed impact class: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
The deadline is publicly readable from canister state (`wait_for_quiet_state.current_deadline_timestamp_seconds`). IC rounds advance approximately every second. An attacker monitors the subnet block time and submits the ingress call timed to arrive in the target round. No privileged access, key material, or majority corruption is required — only a neuron with enough voting power to affect the tally and the ability to time an ingress message. The attack is repeatable for every SNS proposal.

## Recommendation
Change the deadline check in `register_vote` from strict `>` to `>=`, making it consistent with `accepts_vote`:

```rust
// Before:
if now_seconds > deadline {

// After:
if now_seconds >= deadline {
```

This mirrors the NNS governance approach of using `accepts_vote` as the single source of truth, and eliminates the one-second window where a vote is accepted by `register_vote` but rejected by `accepts_vote`.

## Proof of Concept
1. Deploy an SNS with a test neuron holding enough voting power to flip a proposal.
2. Submit a governance proposal and record `deadline = proposal.get_deadline_timestamp_seconds()`.
3. Do not vote until the round where `env.now() == deadline`. Submit `manage_neuron { RegisterVote { proposal_id, vote: Yes } }` in that round.
4. Observe: the call succeeds (the `now_seconds > deadline` guard is false at equality), the vote is counted in `recompute_tally`, `process_proposal` runs with `accepts_vote` returning false (expired), `can_make_decision` returns true, and the proposal is decided — without any wait-for-quiet extension having been triggered.
5. Contrast with voting at `deadline - 1`: if that vote would have flipped the majority, wait-for-quiet would have extended the deadline, giving opponents time to respond. Voting at exactly `deadline` skips this protection.
