Audit Report

## Title
Unbounded `Vec<u8>` Allocation in `TryFrom<pb::BasicSignature>` Before Crypto Check Enables Byzantine-Peer OOM — (`rs/types/types/src/signature.rs`, `rs/p2p/quic_transport/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

A Byzantine subnet peer can craft a `SlotUpdate` protobuf whose embedded `BasicSignature.signature` field is padded up to the 128 MB transport cap. The receiving replica allocates the full `Vec<u8>` inside `TryFrom<pb::BasicSignature>` with no size guard, before any cryptographic verification or slot-limit enforcement. With up to 1,000 concurrent QUIC streams per peer, each running an independent `handle_bi_stream` task, a single Byzantine node can drive the target replica into severe memory pressure or OOM, stalling or crashing it out of the active subnet set.

## Finding Description

**Root cause — no size check in `TryFrom<pb::BasicSignature>`.**
`rs/types/types/src/signature.rs` lines 86–94 unconditionally wrap the raw protobuf bytes:

```rust
impl<T> TryFrom<pb::BasicSignature> for BasicSignature<T> {
    fn try_from(value: pb::BasicSignature) -> Result<Self, Self::Error> {
        Ok(Self {
            signature: BasicSigOf::new(BasicSig(value.signature)), // no length check
            signer: node_id_try_from_option(value.signer)?,
        })
    }
}
```

The proto definition (`rs/protobuf/def/types/v1/signature.proto`) has no `max_length` constraint on the `bytes signature` field.

**Transport cap is 128 MB per stream.**
`rs/p2p/quic_transport/src/lib.rs:74` sets `MAX_MESSAGE_SIZE_BYTES = 128 * 1024 * 1024`. `rs/p2p/quic_transport/src/request_handler.rs:173–178` calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)`, reading up to 128 MB per stream into a heap `Vec` before any application processing. The QUIC `STREAM_RECEIVE_WINDOW` of 4 MB (`connection_manager.rs:73`) and connection-level `RECEIVE_WINDOW` of 200 MB (`connection_manager.rs:71`) are flow-control windows that are continuously extended as `read_to_end` drains them; they do not cap total per-stream allocation.

**Deserialization precedes slot-limit check.**
`rs/p2p/consensus_manager/src/receiver.rs:85–113` decodes the full artifact — including calling `TryFrom` on the embedded `BasicSignature` — inside `update_handler`, before the result is sent to the `ConsensusManagerReceiver` event loop where slot limits are enforced. The axum body limit is explicitly disabled for this route (`receiver.rs:51`: `DefaultBodyLimit::disable()`).

**Concurrent stream amplification.**
`rs/p2p/quic_transport/src/connection_manager.rs:74` sets `MAX_CONCURRENT_BIDI_STREAMS = 1_000`. `start_stream_acceptor` (`request_handler.rs:43–110`) spawns each accepted stream as an independent task in `inflight_requests` with no backpressure before the read completes. The channel from `update_handler` to the event loop has capacity 100 (`receiver.rs:43`), but this backpressure only applies after deserialization and allocation have already occurred.

**Existing checks are insufficient.**
TLS mutual authentication confirms the peer is a registered subnet node but does not prevent a Byzantine node from sending malformed payloads. The 128 MB cap prevents a single stream from exceeding that bound but does not prevent concurrent amplification. The slot-limit check is downstream of the allocation.

## Impact Explanation

A single Byzantine subnet peer (below the fault threshold) can force the target replica into OOM or severe memory pressure during consensus artifact ingestion. The affected replica stalls or crashes, dropping out of the subnet's active set until restarted. This constitutes an **application/platform-level DoS and subnet availability impact not based on raw volumetric DDoS**, matching the High ($2,000–$10,000) impact tier. Consensus safety is not broken (the subnet continues with remaining honest replicas), but liveness is degraded and the attack is repeatable.

## Likelihood Explanation

The attacker must be a valid subnet member (Byzantine peer below the fault threshold). TLS mutual authentication on the QUIC transport ensures only registered subnet nodes can connect, but this is an explicitly considered attacker model in the IC Byzantine fault model. No special privileges beyond subnet membership are required. The attack is trivially scriptable: open 1,000 QUIC streams and send a crafted `SlotUpdate` with a near-128 MB `BasicSignature.signature` on each. The attack can be repeated after the target restarts.

## Recommendation

1. Add a size guard in `TryFrom<pb::BasicSignature>` (and analogous `ThresholdSignature`, `ThresholdSignatureShare` conversions) that rejects any `signature` field exceeding the maximum expected cryptographic output size (e.g., 96 bytes for BLS12-381, 64 bytes for Ed25519):
   ```rust
   const MAX_BASIC_SIG_BYTES: usize = 256; // conservative upper bound
   if value.signature.len() > MAX_BASIC_SIG_BYTES {
       return Err(ProxyDecodeError::Other("signature field too large".into()));
   }
   ```
2. Apply the check before the `Vec<u8>` is moved into `BasicSig`, so no allocation occurs for oversized inputs.
3. Consider enforcing per-artifact-type size limits at the `update_handler` boundary (before protobuf decoding) rather than relying solely on the 128 MB transport cap.
4. Consider reducing `MAX_MESSAGE_SIZE_BYTES` or introducing per-route limits that reflect the actual maximum legitimate artifact size for each artifact type.

## Proof of Concept

```rust
// Unit test demonstrating allocation without rejection:
#[test]
fn oversized_basic_sig_is_accepted() {
    use ic_types::crypto::BasicSig;
    use ic_protobuf::types::v1 as pb;
    use std::convert::TryFrom;

    let oversized = vec![0u8; 100 * 1024 * 1024]; // 100 MB
    let pb_sig = pb::BasicSignature {
        signature: oversized,
        signer: Some(valid_node_id_proto()),
    };
    // Allocates 100 MB in BasicSig(value.signature) — no rejection
    let result = ic_types::crypto::BasicSignature::<()>::try_from(pb_sig);
    assert!(result.is_ok());
}
```
