### Title
Broken `block.number` and `block.timestamp` Time Assumptions Due to NEAR's ~1-Second Block Rate and Shared Block Number Across Same-Block EVM Transactions - (File: `engine/src/engine.rs`)

---

### Summary

Aurora Engine maps EVM `block.number` and `block.timestamp` directly to the raw NEAR block height and NEAR block timestamp with no adjustment. NEAR produces blocks approximately every **1 second** (vs Ethereum's ~12 seconds), and multiple EVM transactions included in the same NEAR block all observe the **identical** `block.number` and `block.timestamp`. Time-sensitive EVM contracts deployed on Aurora will have their block-based time assumptions broken in two compounding ways: (1) the block rate is ~12× faster than Ethereum, and (2) `block.number` does not increment between EVM transactions within the same NEAR block.

---

### Finding Description

**Root cause 1 — `block_number()` returns raw NEAR block height with no rate adjustment:**

`Engine::block_number()` is implemented as:

```rust
fn block_number(&self) -> U256 {
    U256::from(self.env.block_height())
}
``` [1](#0-0) 

`block_height()` is sourced directly from the NEAR host function `block_index()`:

```rust
fn block_height(&self) -> u64 {
    unsafe { exports::block_index() }
}
``` [2](#0-1) 

NEAR's block time is documented as **1 second** per block:

> "the [1000 Tgas gas limit] per block and the [1 second block time]" [3](#0-2) 

Ethereum's block time is ~12 seconds. A DeFi contract that uses `block.number` to measure a 20-day period (144,000 blocks × 12 s) would instead measure only ~1.67 days on Aurora (144,000 blocks × 1 s).

**Root cause 2 — Multiple EVM transactions in the same NEAR block share the same `block.number` and `block.timestamp`:**

The engine explicitly supports multiple EVM transactions per NEAR block via the `position` field:

> "If multiple Aurora transactions are included in the same block, this index gives the order in which they should be executed." [4](#0-3) 

Because `block_number()` and `block_timestamp()` both read from the NEAR block context (which is fixed for the entire NEAR block), every EVM transaction at positions 0, 1, 2, … within the same NEAR block sees the **same** `block.number` and `block.timestamp`. There is no per-EVM-transaction increment.

`block_timestamp()` has the same structure:

```rust
fn block_timestamp(&self) -> U256 {
    U256::from(self.env.block_timestamp().secs())
}
``` [5](#0-4) 

The engine's own comment confirms the 1:1 NEAR-to-Aurora block mapping:

> "There is one Aurora block per NEAR block height" [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield (block-rate mismatch):**
Any EVM staking, vesting, or reward contract that uses `block.number` to pace reward accrual (a common Ethereum pattern) will accrue rewards ~12× faster than intended on Aurora. An attacker can deploy or interact with such a contract and claim rewards that correspond to 12× more real-world time than has actually elapsed, draining the reward pool ahead of schedule.

**High — Temporary freezing / premature unlock (block-rate mismatch):**
Contracts that enforce a minimum lock period via `block.number` (e.g., `require(block.number >= unlockBlock)`) will unlock funds ~12× sooner than the protocol designer intended, breaking the security invariant of the time lock.

**High — Same-block manipulation bypass (shared block.number):**
Contracts that guard against same-block price manipulation or flash-loan attacks with a check such as `require(block.number > lastUpdateBlock)` can be bypassed: an attacker submits two transactions that land in the same NEAR block. Both see the same `block.number`, so the second transaction passes the guard even though it executes in the same logical "block." This enables price oracle manipulation and potential fund theft from lending/AMM protocols.

---

### Likelihood Explanation

- Any EVM DeFi contract ported from Ethereum to Aurora that uses `block.number` for time measurement is affected by the 12× rate mismatch — this is a structural, always-on condition.
- The same-block bypass requires two transactions to land in the same NEAR block. NEAR's 1-second block time and high throughput make this a realistic and repeatable condition; an attacker can submit a batch of transactions and statistically expect co-inclusion.
- No special privileges are required. Any EVM user, contract deployer, or calldata sender can trigger these paths through the standard `submit` entrypoint.

---

### Recommendation

1. **Document the incompatibility explicitly** in Aurora's developer-facing materials: `block.number` on Aurora advances at ~1-second intervals (not ~12 seconds), and multiple EVM transactions within the same NEAR block share the same `block.number` and `block.timestamp`. Warn that Ethereum contracts relying on `block.number` for time measurement must be audited before deployment.

2. **Consider a virtual block counter** that increments per EVM transaction rather than per NEAR block, so that `block.number` is unique across all EVM transactions. This would eliminate the same-block bypass issue and make `block.number` behave more like Ethereum's per-block increment.

3. **Consider a scaling factor** for `block.number` (e.g., multiply NEAR block height by 12) to approximate Ethereum's block cadence, or expose a separate "Aurora virtual block number" that accounts for the rate difference.

### Proof of Concept

**Scenario A — Reward drain via 12× faster block rate:**

```solidity
// Deployed on Ethereum: releases rewards after 144,000 blocks (~20 days at 12s/block)
// Deployed on Aurora:   releases rewards after 144,000 blocks (~1.67 days at 1s/block)
contract StakingReward {
    uint256 public startBlock;
    uint256 public constant LOCK_BLOCKS = 144_000;

    function claimReward() external {
        require(block.number >= startBlock + LOCK_BLOCKS, "Too early");
        // pay out reward
    }
}
```

On Aurora, `block.number` is `self.env.block_height()` advancing at ~1 s/block. [7](#0-6) 
The 144,000-block threshold is reached in ~1.67 days instead of ~20 days.

**Scenario B — Same-block oracle manipulation:**

```solidity
contract Oracle {
    uint256 public lastUpdateBlock;
    uint256 public price;

    function update(uint256 newPrice) external {
        require(block.number > lastUpdateBlock, "Same block");
        price = newPrice;
        lastUpdateBlock = block.number;
    }
}
```

An attacker submits two `update()` calls in the same NEAR block. Both calls observe the same `block.number` (the NEAR block height). [4](#0-3) 
The second call passes `require(block.number > lastUpdateBlock)` because `lastUpdateBlock` was set to the same value in the first call, and `block.number` has not changed. The attacker can sandwich a victim transaction between two oracle updates within a single NEAR block, manipulating the price and draining a dependent lending protocol.
