### Title
Missing Nonce in `allowPushers` Signed Data Enables Replay of Revoked Pusher Consent — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` verifies a pusher's EIP-191 signature over `(chainid, oracle, deadline, pusher, creator)` but includes **no nonce**. A creator who holds a valid, not-yet-expired signature can replay it an unlimited number of times before the deadline, re-establishing a delegation the pusher has already revoked and overwriting any subsequent delegation the pusher made to a different creator.

---

### Finding Description

`allowPushers` is the mechanism by which a pusher wallet consents to have its price pushes routed into a different creator's namespace: [1](#0-0) 

The signed digest is constructed as: [2](#0-1) 

The tuple `(block.chainid, address(this), deadline, pusher, msg.sender)` contains no per-pusher nonce or sequence counter. `_ensureDeadline` only checks `block.timestamp <= deadline`: [3](#0-2) 

Because there is no nonce, the same signature is valid for every call to `allowPushers` until `deadline` passes. The state written is simply: [4](#0-3) 

A pusher can self-revoke via `revokePusher()`: [5](#0-4) 

But because the old signature is still valid, creator A can immediately call `allowPushers` again with the same signature and deadline, overwriting `namespaceRemapping[pusher]` back to `A`. This cycle can repeat indefinitely until `deadline` expires.

---

### Impact Explanation

The `namespaceRemapping` table determines where every push from a given wallet lands. The `fallback` push path reads it unconditionally: [6](#0-5) 

If creator A keeps replaying the old signature, the pusher's data is permanently diverted into creator A's namespace instead of creator B's. Consequences:

1. **Creator B's feeds go stale.** Pools whose `IPriceProvider` reads creator B's `feedId` receive a timestamp that never advances. The pool's `_getBidAndAskPriceX64` call will eventually revert or return a stale price, breaking swap and liquidity flows. [7](#0-6) 

2. **Creator A's feeds receive unintended data.** The pusher's slot words (formatted for creator B's pair) are written into creator A's storage slots. If creator A's feeds are consumed by pools, those pools execute swaps against a corrupted bid/ask, satisfying the "bad-price execution" impact gate.

---

### Likelihood Explanation

- The pusher must have previously signed a delegation to creator A (a normal operational step).
- Creator A must be adversarial and retain the old signature.
- The deadline window is set by the pusher at signing time; typical values (hours to days) give creator A ample time to replay repeatedly.
- No special on-chain privilege is required — creator A calls a public function with a signature they legitimately received.

---

### Recommendation

Include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call:

```solidity
mapping(address => uint256) public pusherNonces;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid,
        address(this),
        deadline,
        pusher,
        msg.sender,
        pusherNonces[pusher]   // ← add nonce
    ))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
pusherNonces[pusher]++;        // ← invalidate old signatures
namespaceRemapping[pusher] = msg.sender;
```

This ensures each signed consent is single-use, so a revoked delegation cannot be replayed.

---

### Proof of Concept

```
t=0  Pusher P signs: (chainid, oracle, deadline=t+7days, P, creatorA)
t=1  creatorA calls allowPushers(deadline, [P], [sig])
       → namespaceRemapping[P] = creatorA  ✓

t=2  P calls revokePusher()
       → namespaceRemapping[P] = address(0)

t=3  P signs new consent for creatorB (deadline=t+7days)
t=4  creatorB calls allowPushers(...)
       → namespaceRemapping[P] = creatorB  ✓

t=5  creatorA replays OLD sig (still valid, deadline not expired):
     creatorA calls allowPushers(deadline_from_t0, [P], [old_sig])
       → namespaceRemapping[P] = creatorA  ← overwrites creatorB

t=6  P calls revokePusher() again → address(0)
t=7  creatorA replays again → creatorA
     ... loop until t+7days

Result: creatorB's feeds never receive P's pushes → stale prices →
        pools using creatorB's feedIds revert or execute at stale bid/ask.
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
