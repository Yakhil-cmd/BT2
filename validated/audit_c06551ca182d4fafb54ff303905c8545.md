### Title
Staker Rewards Temporarily Frozen When `RewardSupplier` Token Balance Is Insufficient — (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `RewardSupplier.claim_rewards` function validates only that `amount <= unclaimed_rewards` before executing a token transfer, without verifying that the contract's actual on-chain token balance is sufficient. Because the `request_funds` invariant permits `balance = 0` as long as `l1_pending_requested_amount` is large enough, any staker or delegator attempting to claim legitimately earned rewards during the L1-bridging latency window will have their transaction revert, temporarily freezing their funds.

---

### Finding Description

The `RewardSupplier` contract tracks two distinct quantities:

- `unclaimed_rewards` — accounting ledger of what is owed to stakers.
- `l1_pending_requested_amount` — tokens already requested from L1 but not yet received via StarkGate.

The `request_funds` internal function maintains the invariant:

```
balance + l1_pending_requested_amount >= unclaimed_rewards + threshold
``` [1](#0-0) 

This invariant explicitly allows `balance = 0` provided `l1_pending_requested_amount` is sufficiently large. However, `claim_rewards` — the function called by the staking contract whenever a staker or delegator collects rewards — only checks the accounting ledger, not the actual balance:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    let token_dispatcher = self.token_dispatcher.read();
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
    // ^^^ panics if actual balance < amount
}
``` [2](#0-1) 

`request_funds` is only triggered from `update_unclaimed_rewards_from_staking_contract`, which is called during reward updates — not during reward claims. [3](#0-2) 

Claiming rewards does not re-invoke `request_funds`, so the balance can reach zero between a reward update and a subsequent claim without any corrective action.

The staking contract's `send_rewards_to_staker` calls `claim_from_reward_supplier` (which calls `RewardSupplier.claim_rewards`) and then immediately transfers to the staker's reward address: [4](#0-3) 

If `RewardSupplier.claim_rewards` reverts due to insufficient balance, the entire staker claim transaction reverts.

---

### Impact Explanation

**Temporary freezing of funds.** Stakers and delegators cannot claim their legitimately earned STRK rewards for the duration of the L1→L2 bridge latency (typically hours to days). The rewards are not lost — they remain in `unclaimed_rewards` — but they are inaccessible until L1 delivers the bridged tokens via `on_receive`. [5](#0-4) 

---

### Likelihood Explanation

**Medium–High.** The scenario arises naturally under normal operation:

1. Accumulated rewards cause `request_funds` to send L1 mint requests (`l1_pending_requested_amount` grows).
2. Concurrent staker claims drain the existing `balance` to zero before L1 bridges the new tokens.
3. Any subsequent claim attempt reverts until `on_receive` is called by StarkGate.

No adversarial action is required; this is a structural timing gap between the accounting invariant and the actual transfer requirement.

---

### Recommendation

Add an explicit balance check inside `claim_rewards` before attempting the transfer. If the balance is insufficient, revert with a descriptive error (e.g., `INSUFFICIENT_REWARD_SUPPLIER_BALANCE`) so callers understand the condition is temporary and can retry. Alternatively, `request_funds` could be invoked at the start of `claim_rewards` to trigger an L1 request if the balance is about to be exhausted, mirroring the buffer-first pattern described in the reference report.

---

### Proof of Concept

1. Initial state: `balance = 100`, `l1_pending_requested_amount = 0`, `unclaimed_rewards = 0`, `threshold = 100`.
2. Rewards accumulate: `unclaimed_rewards = 200`.
3. `request_funds` is called: `credit = 100 < 200 + 100` → sends L1 request for 200; `l1_pending_requested_amount = 200`.
4. State: `balance = 100`, `pending = 200`, `unclaimed = 200`. Invariant holds: `100 + 200 >= 200 + 100` ✓.
5. Staker A claims 100: `RewardSupplier.claim_rewards(100)` → `checked_transfer(100)` succeeds; `balance = 0`, `unclaimed = 100`.
6. State: `balance = 0`, `pending = 200`, `unclaimed = 100`. Invariant still holds: `0 + 200 >= 100 + 100` ✓. No new L1 request triggered.
7. Staker B calls `staking.claim_rewards(staker_B)` → `send_rewards_to_staker` → `claim_from_reward_supplier(50)` → `RewardSupplier.claim_rewards(50)`: asserts `50 <= 100` ✓, then `checked_transfer(50)` from `balance = 0` → **REVERT**.
8. Staker B's rewards are frozen until StarkGate calls `on_receive` and replenishes the balance.
