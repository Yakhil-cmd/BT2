### Title
NFT Metadata Integrity Check Silently Skipped When `metadataHash` Is Absent, Enabling Metadata Spoofing - (File: packages/gui/src/hooks/useFetchAndProcessMetadata.ts)

### Summary

`useFetchAndProcessMetadata` skips the SHA-256 integrity check entirely when the NFT's on-chain `metadataHash` field is absent or empty. An attacker who mints an NFT with no metadata hash and controls the metadata URI server can serve arbitrary metadata content. The GUI renders this unverified metadata (name, description, attributes, preview URIs, sensitive-content flags) with no warning to the user, directly analogous to the external report's pattern of omitting a transcript component so the verifier accepts forged content.

### Finding Description

In `useFetchAndProcessMetadata`, the hash guard is:

```typescript
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
``` [1](#0-0) 

When `hash` is `undefined`, `null`, `''`, or `'0x'`, the entire `if`-block is skipped and execution falls through to parse and return the fetched content as trusted metadata. The caller in `useMetadataData` passes `metadataHash` directly from the on-chain NFT record:

```typescript
const { metadataUris = [], metadataHash } = nft;
const metadata = await fetchAndProcessMetadata(firstUri, metadataHash);
``` [2](#0-1) 

The Chia NFT standard permits minting with `metadataHash = '0x'` (empty hash). When this is the case, `NFTDetails` does not render the metadata hash row at all:

```typescript
if (nft.metadataHash && nft.metadataHash !== '0x') {
  rows.push({ key: 'metadataHash', ... });
}
``` [3](#0-2) 

The existing `NFTHashStatus` component only verifies the **data** hash (`dataHash` vs `dataUris`), not the metadata hash. There is no separate UI indicator for metadata hash verification status. [4](#0-3) 

The unverified metadata object is then consumed throughout the GUI: NFT name/description display, sensitive-content blurring (`hasSensitiveContent(metadata)`), preview image/video URI selection, and attribute rendering. [5](#0-4) 

### Impact Explanation

An attacker mints an NFT with `metadataHash = '0x'` and `metadataUri = 'https://attacker.com/metadata.json'`, then creates an offer or sends the NFT to a target. When the victim's GUI fetches metadata, no hash comparison is performed. The attacker's server returns metadata claiming the NFT is a high-value asset (e.g., a well-known collection, a specific DID-linked identity, or a VC credential). The victim sees the spoofed name, description, attributes, and preview images with no "unverified metadata" warning. This causes the user to display the wrong asset identity, which can lead to accepting an offer under false pretenses — a direct High-impact outcome under the allowed scope ("display the wrong asset, identity, amount, destination, or status").

Additionally, the attacker can set `sensitive_content: false` in the forged metadata to suppress the GUI's content-blurring filter for objectionable NFTs, or supply malicious `preview_image_uris`/`preview_video_uris` that are fetched and cached by the application.

### Likelihood Explanation

The attack requires only that the attacker mint one NFT (costs a small amount of XCH) with an empty metadata hash and a URI they control, then distribute it via an offer or transfer. No privileged access, leaked keys, or cryptographic break is needed. NFTs with absent or `'0x'` metadata hashes exist in the wild today, so the attacker does not even need to mint — they can find existing ones where they control or can redirect the metadata server (e.g., via expired domain, DNS hijack, or HTTP redirect on a non-pinned URI).

### Recommendation

1. **Treat absent `metadataHash` as unverifiable, not as "skip verification"**: When `hash` is falsy, either refuse to render metadata content or display a prominent "Metadata unverified — no hash on-chain" warning in the UI.
2. **Add a metadata hash status indicator** analogous to `NFTHashStatus` for data, so users can distinguish verified from unverified metadata.
3. **Harden the guard** in `useFetchAndProcessMetadata`:
   ```typescript
   if (!hash) {
     // metadata cannot be verified; surface warning, do not silently accept
   } else if (!compareChecksums(checksum, hash)) {
     throw new Error('Checksum mismatch');
   }
   ```
4. **Show `metadataHash` in `NFTDetails` even when it is `'0x'`**, so users can see that no hash was committed at mint time.

### Proof of Concept

1. Attacker mints NFT: `metadataHash = '0x'`, `metadataUri = 'https://attacker.com/nft-meta.json'`.
2. `nft-meta.json` returns `{ "name": "CryptoPunks #1", "description": "Rare original", "sensitive_content": false }`.
3. Victim opens the NFT in the GUI. `useMetadataData.limitedFetchMetadata` calls `fetchAndProcessMetadata('https://attacker.com/nft-meta.json', undefined)`.
4. Inside `useFetchAndProcessMetadata`, `hash` is `undefined` → the `if (hash && ...)` branch is not entered → no checksum comparison occurs.
5. The attacker's JSON is parsed and stored as trusted metadata.
6. The GUI renders "CryptoPunks #1" as the NFT name with no hash-mismatch warning. The data hash chip may show "Hash matches" (for the actual data file), giving the user false confidence.
7. Victim accepts an offer for this NFT believing it is a valuable asset. [6](#0-5) [7](#0-6)
