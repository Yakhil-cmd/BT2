### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator blindly aggregates `CKDOutput` contributions from all participants with no cryptographic proof of correctness. A single malicious participant can substitute arbitrary group elements for their `(norm_big_y, norm_big_c)` share, silently corrupting the final derived key delivered to the coordinator.

---

### Finding Description

The coordinator's aggregation loop in `do_ckd_coordinator` receives a `CKDOutput` from every other participant and unconditionally adds each field into the running sum: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

The honest computation each participant is supposed to perform is: [2](#0-1) 

```rust
let big_y = ElementG1::generator() * y.0;
let big_s  = hash_point * private_share.to_scalar();
let big_c  = big_s + app_pk * y.0;
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
```

No zero-knowledge proof, commitment, or consistency check is attached to the message sent to the coordinator: [3](#0-2) 

```rust
let (norm_big_y, norm_big_c) =
    compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
let waitpoint = chan.next_waitpoint();
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
```

Because the coordinator performs no verification before accumulating the received values, a malicious participant can replace their honest `(norm_big_y, norm_big_c)` with any pair of `G1` elements — including the identity, the generator, or a carefully chosen point — and the coordinator has no means to detect the substitution.

The final `unmask` step the application calls is: [4](#0-3) 

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
```

With a corrupted `big_y` and `big_c`, `unmask` silently returns a wrong group element. The application has no way to distinguish this from a legitimate derived key.

---

### Impact Explanation

A single malicious participant can make the coordinator output an incorrect derived key for any `app_id`. The application will accept and use this wrong key without any error signal. This directly matches the allowed impact:

> **High: Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs.**

Because the derived key is used as a secret (e.g., to encrypt or authenticate application data), a wrong key renders all downstream operations cryptographically invalid or, in the worst case, predictable to the attacker if they chose the substituted elements deliberately.

---

### Likelihood Explanation

Any participant enrolled in the CKD protocol can mount this attack. No special privilege, leaked key material, or external assumption is required. The attacker simply deviates from the protocol by sending arbitrary `G1` elements instead of their honest share. The attack is undetectable by the coordinator and by any other honest participant, because participants only send their share privately to the coordinator and never cross-check each other's contributions.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a non-interactive zero-knowledge proof of correct computation — specifically, a proof that `norm_big_y` and `norm_big_c` were formed using the same secret `y` and the participant's committed private share `x_i`, consistent with the public key and `app_id`. The coordinator must verify all proofs before aggregating. Alternatively, adopt a verifiable secret-sharing or commit-then-reveal pattern so that any deviation is publicly attributable and the protocol can abort before producing a corrupted output.

---

### Proof of Concept

1. Honest participants each call `compute_signature_share` and send the result privately to the coordinator.
2. The malicious participant instead sends `(norm_big_y, norm_big_c) = (G1::generator(), G1::generator())` — or any other chosen pair — to the coordinator via `chan.send_private`.
3. The coordinator's loop at lines 50–55 of `protocol.rs` adds these values unconditionally.
4. The resulting `CKDOutput` satisfies `big_y = Σ honest_big_y + G` and `big_c = Σ honest_big_c + G`.
5. When the application calls `ckd_output.unmask(app_sk)`, it computes `(Σ honest_big_c + G) − (Σ honest_big_y + G) * app_sk`, which differs from the correct derived key by `G * (1 − app_sk)`.
6. The application silently receives and uses the wrong key; no error is raised anywhere in the protocol.
