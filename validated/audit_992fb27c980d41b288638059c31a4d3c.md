### Title
WalletConnect `chia_wallet.create_offer_for_ids` Trusts Attacker-Supplied `driver_dict` Royalty Percentage for Confirmation Display - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

When a WalletConnect dapp sends `chia_wallet.create_offer_for_ids`, the GUI computes the "Total Amount with Royalties" shown in the confirmation dialog by reading `royalty_percentage` directly from the dapp-supplied `driver_dict`. The on-chain NFT royalty lookup (`nftGetInfo`) is best-effort and silently falls back to the dapp-supplied value on failure. A malicious dapp can supply a zero or absent `royalty_percentage` in the `driver_dict` for an NFT the user does not own (so `nftGetInfo` returns `success: false`), causing the confirmation dialog to display no royalty surcharge. The user approves the offer believing the cost is lower than it actually is; when the offer is taken on-chain, the real royalty (enforced by the NFT puzzle) is deducted from the user's wallet.

### Finding Description

`parseCommandDisplay` in `packages/gui/src/electron/commands/parseCommandDisplay.ts` handles the `chia_wallet.create_offer_for_ids` WalletConnect command. It builds the royalty percentages used for the confirmation display via `createOfferRoyaltyPercentages`, which reads `royalty_percentage` directly from the dapp-supplied `params.driver_dict` without any on-chain cross-check: [1](#0-0) 

The `royaltyPercentageForDriverInfo` helper traverses the dapp-controlled `driver_dict` structure and returns whatever `transfer_program.royalty_percentage` the dapp provided: [2](#0-1) 

In `parseWalletDeltaItem`, the code does attempt to override the driver_dict value with the real on-chain royalty via `nftGetInfo`: [3](#0-2) 

However, the override only occurs if `nftGetInfo` returns `success: true` **and** `nft_info.data_uris` is present. For NFTs the user is requesting (not yet in their wallet), `nftGetInfo` commonly returns `success: false`. The catch block explicitly treats this as best-effort: [4](#0-3) 

When `nftGetInfo` fails, `result.royaltyPercentage` retains the dapp-supplied value (or `undefined` if the dapp omitted royalty info). The `royaltyPercentagesForSide` function then filters out `undefined` and zero values: [5](#0-4) 

So `formatAmountWithRoyalties` receives an empty array and returns `undefined`, meaning the XCH spending line in the confirmation dialog shows no `amountWithRoyalties` field at all: [6](#0-5) 

The confirmation dialog renders only the base amount with no royalty warning: [7](#0-6) 

The existing test explicitly documents this behavior — when `nftGetInfo` returns `success: false`, the driver_dict royalty value is used for display: [8](#0-7) 

### Impact Explanation

A connected WalletConnect dapp sends `chia_wallet.create_offer_for_ids` with a valid NFT launcher ID but with `royalty_percentage: '0'` (or no royalty fields) in the `driver_dict`. The user is requesting an NFT they do not own, so `nftGetInfo` returns `success: false`. The confirmation dialog shows "Spending 1 XCH" with no royalty surcharge. The user approves. When the offer is taken on-chain, the Chia blockchain enforces the NFT's actual royalty (e.g., 25%), deducting 1.25 XCH from the user's wallet instead of the 1 XCH they saw in the approval dialog. This is a direct, concrete financial loss caused by the GUI displaying the wrong total amount in the WalletConnect signing approval flow.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount … or status."*

### Likelihood Explanation

Any WalletConnect dapp the user has connected to can trigger this. The condition for the fallback (nftGetInfo failing for a requested NFT not in the user's wallet) is the common case when buying an NFT via WalletConnect — the user does not yet own the NFT they are offering to purchase. No special privileges, leaked keys, or host compromise are required.

### Recommendation

In `parseWalletDeltaItem`, do not use the dapp-supplied `royaltyPercentage` as the initial value for the NFT result. Instead, initialize `royaltyPercentage` to `undefined` and only populate it from `nftGetInfo`. If `nftGetInfo` fails for a requested NFT, display a warning in the confirmation dialog that royalty information could not be verified rather than silently omitting it. Alternatively, block the confirmation display entirely when royalty data cannot be confirmed for an NFT being purchased via WalletConnect.

### Proof of Concept

1. User connects a WalletConnect dapp to their Chia wallet.
2. Dapp sends `chia_wallet.create_offer_for_ids` with:
   ```json
   {
     "offer": { "1": "-1000000000000", "<nftLauncherId>": "1" },
     "driver_dict": {
       "<nftLauncherId>": {
         "type": "singleton",
         "launcher_id": "0x<nftLauncherId>",
         "also": {
           "type": "metadata",
           "also": {
             "type": "ownership",
             "transfer_program": {
               "type": "royalty transfer program",
               "royalty_percentage": "0"
             }
           }
         }
       }
     }
   }
   ```
3. The NFT is not in the user's wallet; `nftGetInfo` returns `success: false`.
4. `createOfferRoyaltyPercentages` reads `royalty_percentage: '0'` from the driver_dict; `parseRoyaltyPercentage('0')` returns `undefined` (the regex `/^\d+$/` matches but `value > 0` is false for 0).
5. `royaltyPercentagesForSide` returns an empty array; `amountWithRoyalties` is not set.
6. Confirmation dialog shows: **"Spending 1 XCH"** — no royalty warning.
7. User approves. Offer is created locking 1 XCH.
8. When the offer is taken, the blockchain enforces the NFT's actual 25% royalty; the user's wallet is debited 1.25 XCH total — 0.25 XCH more than the approval dialog indicated.

function royaltyPercentageForDriverInfo(driverInfo: unknown): number | undefined {
  if (!isPlainObject(driverInfo)) {
    return undefined;
  }

  const { also } = driverInfo;
  if (!isPlainObject(also)) {
    return undefined;
  }

  const ownershipLayer = also.also;
  if (!isPlainObject(ownershipLayer)) {
    return undefined;
  }

  const transferProgram = ownershipLayer.transfer_program;
  if (!isPlainObject(transferProgram)) {
    return undefined;
  }

  return parseRoyaltyPercentage(transferProgram.royalty_percentage);
}

function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  for (const assetId of Object.keys(walletDelta.spending)) {
    royaltyPercentages.spending[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  for (const assetId of Object.keys(walletDelta.receiving)) {
    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  return royaltyPercentages;
}

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {

    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
