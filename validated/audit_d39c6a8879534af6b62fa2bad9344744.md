### Title
Unpermissioned `update_rewards` Allows Any Caller to Permanently Freeze Block Rewards - (File: `src/staking/staking.cairo`)

### Summary

`IStakingRewardsManager::update_rewards` is documented as callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, consuming the per-block `last_reward_block` slot and permanently preventing the legitimate sequencer from distributing rewards for that block.

### Finding Description

The specification at `docs/spec.md` line 1644–1645 states:

> **access control**: Only starkware sequencer.

The implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1448–1507) performs only a pause check via `general_prerequisites()` and a per-block deduplication guard:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks is_paused
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits without distributing
    }
    ...
}
```

There is no `assert!(get_caller_address() == sequencer_address, ...)` or equivalent role check. The `last_reward_block` field is a **global, single-slot** state variable: once written for a given block number, the `REWARDS_ALREADY_UPDATED` assertion prevents any second call in the same block, including the legitimate sequencer's call.

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` before the sequencer does will:
1. Write `last_reward_block = current_block_number`.
2. Return immediately without distributing any rewards.
3. Cause the sequencer's subsequent call to revert with `REWARDS_ALREADY_UPDATED`.

Rewards for that block are permanently lost — they are never credited to `unclaimed_rewards_own` of any staker, and pool rewards are never forwarded to delegation pool contracts.

### Impact Explanation

**Permanent freezing of unclaimed yield** for all stakers and delegators. By front-running every block's reward update, an attacker can suppress the entire consensus-era reward stream. Stakers' `unclaimed_rewards_own` and pool `cumulative_rewards_trace` are never updated, so neither stakers nor delegators can ever claim those block rewards. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

### Likelihood Explanation

**High.** The function is publicly callable with no role restriction. The attacker only needs to submit a transaction with a valid (active, non-zero-balance) `staker_address` and `disable_rewards: true` before the sequencer's transaction lands in each block. On Starknet, where the sequencer ordering is observable, this is straightforwardly achievable. The cost is one transaction per block.

### Recommendation

Add a caller check enforcing that only the authorized sequencer address (or a designated role) may call `update_rewards`. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

Alternatively, store the authorized caller address in contract storage and validate it at the top of the function, consistent with how `assert_caller_is_attestation_contract` is implemented for `update_rewards_from_attestation_contract`.

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Staker `S` is active with non-zero balance.
3. Attacker `A` (any EOA) calls `update_rewards(S, disable_rewards: true)` at block `N`.
4. `last_reward_block` is set to `N`; no rewards are distributed.
5. The sequencer calls `update_rewards(S, disable_rewards: false)` at block `N` → reverts with `REWARDS_ALREADY_UPDATED`.
6. Staker `S`'s `unclaimed_rewards_own` is unchanged; pool rewards are not forwarded.
7. Attacker repeats for every block → all consensus-era block rewards are permanently frozen.
