### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Can Permanently Freeze Unclaimed Yield - (File: `src/pool/pool.cairo`)

### Summary
The `calculate_rewards` internal function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace since their last claim. A delegator who makes balance changes across many epochs without claiming rewards will accumulate an arbitrarily large trace, eventually causing `claim_rewards` to exceed Starknet's per-transaction step limit and permanently freeze their unclaimed yield.

### Finding Description
In `src/pool/pool.cairo`, the `calculate_rewards` function iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to `pool_member_trace_length`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows by one entry per epoch in which the pool member modifies their balance. The `insert` function in `PoolMemberBalanceTrace` only deduplicates entries within the same epoch; changes in distinct epochs each append a new checkpoint:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

The `entry_to_claim_from` field is only advanced on a *successful* `claim_rewards` call:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
self.write_pool_member_info(:pool_member, :pool_member_info);
``` [3](#0-2) 

If the transaction reverts due to step exhaustion, `entry_to_claim_from` is never updated, so every subsequent attempt re-runs the same oversized loop. The cost per outer iteration is compounded because each iteration calls `find_sigma`, which performs a binary search on the `cumulative_rewards_trace`: [4](#0-3) 

The `claim_rewards` function is callable by the pool member or their reward address — both unprivileged roles:

```cairo
assert!(
    caller_address == pool_member || caller_address == reward_address,
    "{}",
    Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
);
``` [5](#0-4) 

### Impact Explanation
Once the balance trace grows large enough that a single `claim_rewards` transaction exceeds Starknet's maximum step budget, the delegator's accumulated rewards are permanently frozen. No partial-claim mechanism exists; the entire trace must be processed in one transaction. This maps to **High: Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The trace grows by at most one entry per epoch per delegator. A delegator who routinely adjusts their delegation (e.g., via `add_to_delegation_pool` / `exit_intent` / re-entry) across many epochs without claiming will accumulate entries proportional to the number of epochs elapsed. The epoch length is configurable and can be short. The codebase itself acknowledges the loop is unbounded (line 858 comment). A delegator who participates actively for a long period — a realistic scenario for a staking protocol — will eventually hit this limit. No privileged access is required; any delegator can reach this state through normal protocol usage.

### Recommendation
1. **Paginate `claim_rewards`**: Accept an optional `max_entries` parameter so the loop processes at most N entries per call, advancing `entry_to_claim_from` and writing partial state on each invocation.
2. **Alternatively, cap trace growth**: Collapse old entries into a running `unclaimed_rewards` accumulator during each balance-change operation, so the trace never grows beyond a small constant size (e.g., K+1 entries).
3. **Remove the "unlikely to exceed gas limits" comment** and replace it with an enforced bound.

### Proof of Concept
1. Delegator calls `enter_delegation_pool` to join a pool.
2. In each subsequent epoch, the delegator calls `add_to_delegation_pool` with a small amount, appending one new entry to `pool_member_epoch_balance` per epoch.
3. The delegator never calls `claim_rewards`.
4. After `N` epochs (where `N` is large enough to exhaust Starknet's step limit for one transaction), the delegator calls `claim_rewards`.
5. The `calculate_rewards` loop iterates `N` times, each time calling `find_sigma` (binary search on `cumulative_rewards_trace`). The transaction runs out of steps and reverts.
6. Because `entry_to_claim_from` was never written, every future `claim_rewards` attempt also reverts. The delegator's yield is permanently frozen.
