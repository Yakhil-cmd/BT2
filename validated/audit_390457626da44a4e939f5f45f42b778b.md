### Title
Gateway Stateful Validator Uses Previous Block's Stale L2 Gas Price Instead of Next Block's Price for Resource Bounds Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the L2 gas price from the **previous** committed block and uses it as the admission threshold. The batcher executes transactions against the **next** block's L2 gas price, which is computed by the EIP-1559 fee market. With the production default of `min_gas_price_percentage = 100`, transactions priced at exactly the previous block's L2 gas price pass gateway admission but fail during batcher execution whenever the fee market has moved the price upward. Conversely, when the fee market moves the price downward, transactions that would succeed in execution are incorrectly rejected at the gateway.

### Finding Description

In `validate_resource_bounds`, the gateway reads the L2 gas price from the previous block's stored `BlockInfo`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The embedded TODO comment explicitly acknowledges that the *next* block's price should be used instead. The threshold check computes `threshold = previous_block_l2_gas_price * min_gas_price_percentage / 100` and rejects if `tx.max_price_per_unit < threshold`. [2](#0-1) 

The production deployment config sets `min_gas_price_percentage = 100`, meaning the threshold equals the previous block's price exactly. [3](#0-2) 

`run_validate_entry_point` — the blockifier validation path inside the gateway — also uses the previous block's gas prices. It only increments the block number, leaving all gas prices unchanged:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [4](#0-3) 

The batcher, however, builds its `BlockContext` with the **next** block's L2 gas price, computed by the EIP-1559 fee market via `calculate_next_base_gas_price`. When the batcher calls `perform_pre_validation_stage` → `check_fee_bounds`, it compares `tx.max_price_per_unit` against the actual next block price:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [5](#0-4) 

The EIP-1559 formula is:

```
price_change = price * gas_delta / (gas_target * gas_price_max_change_denominator)
``` [6](#0-5) 

With the v0.14.4 constants (`gas_price_max_change_denominator = 48`, `gas_target = 1,040,000,000`, `max_block_size = 5,800,000,000`), a fully loaded block produces a next-block price increase of approximately **9.54%** per block:

```
Δ = (5,800,000,000 − 1,040,000,000) / (1,040,000,000 × 48) ≈ 9.54%
``` [7](#0-6) 

The `BlockHeaderWithoutHash` struct already carries a `next_l2_gas_price` field that stores the pre-computed next block price, but the gateway does not read it. [8](#0-7) 

### Impact Explanation

**Scenario A — Admission of transactions that fail execution (price rising):**
When a block is heavily loaded, the next block's L2 gas price is higher than the previous block's price. A transaction with `max_price = prev_price` passes the gateway threshold check (threshold = `prev_price × 100% = prev_price`) and also passes the gateway's blockifier validation (which uses the same stale price). The transaction is admitted to the mempool. When the batcher attempts to include it, `check_fee_bounds` compares against `next_price > prev_price` and rejects the transaction with `MaxGasPriceTooLow`. The transaction occupies a mempool slot and consumes batcher resources without ever being included.

**Scenario B — Rejection of valid transactions (price falling):**
When blocks are under-utilized, the next block's L2 gas price is lower than the previous block's price. A transaction with `max_price` in the range `[next_price, prev_price)` would succeed in execution but is rejected at the gateway because `max_price < prev_price = threshold`. This is a direct false rejection of a valid transaction before sequencing.

Both scenarios match the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The EIP-1559 fee market adjusts the L2 gas price every block. During any sustained period of high network activity — a normal condition on mainnet — the next block's price is consistently above the previous block's price by up to ~9.54% per block. A user who sets `max_price` at exactly the current network price (a rational strategy) will have their transaction admitted by the gateway but rejected by the batcher. No privileged access is required; any unprivileged user submitting a standard `AllResources` invoke transaction triggers this path.

### Recommendation

Replace the read of `previous_block_l2_gas_price` with the `next_l2_gas_price` field already stored in `BlockHeaderWithoutHash`. The `GatewayFixedBlockStateReader` should expose this field, and `validate_resource_bounds` should use it as the threshold base. The same correction should be applied to the `block_info` passed into `run_validate_entry_point` so that the blockifier's `check_fee_bounds` during gateway validation also uses the next block's price, making gateway admission and batcher execution consistent.

### Proof of Concept

```
Given:
  previous block L2 gas price P = 8,000,000,000 fri (min gas price, v0.14.4)
  previous block gas_used = 5,800,000,000 (fully loaded, max_block_size)
  gas_target = 1,040,000,000
  gas_price_max_change_denominator = 48
  min_gas_price_percentage = 100 (production default)

Compute next block price:
  gas_delta = 5,800,000,000 − 1,040,000,000 = 4,760,000,000
  price_change = 8,000,000,000 × 4,760,000,000 / (1,040,000,000 × 48)
               = 38,080,000,000,000,000,000 / 49,920,000,000
               ≈ 763,200,000
  next_price P' = 8,000,000,000 + 763,200,000 = 8,763,200,000 fri

User submits invoke tx with max_price = 8,000,000,000 (= P):

  Gateway validate_resource_bounds:
    threshold = P × 100/100 = 8,000,000,000
    max_price (8,000,000,000) >= threshold (8,000,000,000) → PASS

  Gateway run_validate_entry_point (blockifier, uses prev block prices):
    check_fee_bounds: max_price (8,000,000,000) >= prev_price (8,000,000,000) → PASS

  Transaction admitted to mempool.

  Batcher execution (blockifier, uses next block prices):
    check_fee_bounds: max_price (8,000,000,000) < next_price (8,763,200,000) → FAIL
    Error: MaxGasPriceTooLow { resource: L2Gas, max_gas_price: 8000000000,
                               actual_gas_price: 8763200000 }

  Transaction rejected during block building despite passing all gateway checks.
```
