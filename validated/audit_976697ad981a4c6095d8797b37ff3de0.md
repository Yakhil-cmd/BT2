### Title
Unpermissioned `execute_scheduled` Allows Anyone to Force Execution of a User's Scheduled XCC Promise at an Unfavorable Time - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC Router contract's `execute_scheduled` function has no caller restriction. Any NEAR account can call it to force execution of another user's scheduled cross-contract promise at any time, including at a time that is financially unfavorable to the user. This is a direct analog to the BalancedVault vulnerability where anyone could call `claim()` for another account.

---

### Finding Description

Each EVM address on Aurora has a dedicated XCC Router sub-account. The `schedule` function (restricted to the parent Aurora Engine) stores a `PromiseArgs` entry keyed by a monotonic nonce: [1](#0-0) 

The `execute_scheduled` function, however, has **no access control**. The developer comment explicitly acknowledges this is intentional, reasoning that the promise content was already authorized by the engine: [2](#0-1) 

The test suite confirms this design, with the comment `// anyone can call this function` before calling `execute_scheduled` from an unrelated account (`bob`): [3](#0-2) 

The reasoning that "there is no security risk" is flawed. While the *content* of the promise is fixed and authorized, the **timing** of execution is not. For many DeFi operations (token swaps, liquidity provision, yield claims), the time at which a cross-contract call executes is financially material. A user who schedules a promise may intend to execute it only under certain market conditions. An attacker who calls `execute_scheduled` at a different time can force the operation to execute when it is unfavorable to the user.

The `schedule` function stores the promise with a fixed `attached_balance` (NEAR from the router's balance) and fixed `args`. Once `execute_scheduled` is called, the promise is **irrevocably removed from storage** and dispatched: [4](#0-3) 

There is no cancellation mechanism and no way for the user to prevent execution once an attacker calls `execute_scheduled`.

---

### Impact Explanation

**High — Theft of unclaimed yield / direct loss of user funds.**

A user who schedules a cross-contract call to a DeFi protocol (e.g., a token swap, liquidity deposit, or yield claim) may receive significantly less value than expected if an attacker forces execution at an unfavorable time. The attacker permanently removes the promise from storage and triggers the call, leaving the user with no recourse. In a sandwich-style scenario on NEAR, an attacker could observe the stored promise on-chain, manipulate the target protocol's state, force execution, and profit at the user's expense.

---

### Likelihood Explanation

**Medium.** The XCC precompile is an actively used Aurora feature for EVM contracts interacting with NEAR DeFi protocols. Scheduled promises are stored on-chain and visible to any observer. Any NEAR account can call `execute_scheduled` with zero cost beyond gas. The attacker only needs to know the nonce (which starts at 0 and increments sequentially, making it trivially guessable) and the router sub-account address (derived deterministically from the EVM address). [5](#0-4) 

---

### Recommendation

Restrict `execute_scheduled` to the parent account (Aurora Engine), mirroring the access control already applied to `execute` and `schedule`:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    self.assert_preconditions(); // add this line
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

If the protocol requires third-party relayers to call `execute_scheduled` for UX reasons (analogous to the MultiInvoker justification in the BalancedVault report), consider a whitelist of authorized executors, or add a time-lock so the user has a window to cancel before a third party can force execution. [6](#0-5) 

---

### Proof of Concept

1. Alice's EVM contract calls the XCC precompile to schedule a cross-contract call to a NEAR DEX: swap 100 wNEAR for TOKEN with no minimum output specified. The Aurora Engine calls `schedule` on Alice's router sub-account (e.g., `<alice_evm_addr>.aurora`), storing the promise at nonce `0`.

2. The market moves: TOKEN price drops 30%.

3. Bob (attacker) observes the stored promise on-chain. Bob calls:
   ```
   <alice_evm_addr>.aurora::execute_scheduled(nonce: 0)
   ```
   This is permissionless — no signature or authorization from Alice is required.

4. The DEX swap executes immediately at the current (unfavorable) price. Alice receives 70 TOKEN instead of the 100 TOKEN she would have received had she waited for the price to recover.

5. The promise is permanently deleted from storage. Alice cannot cancel or retry. She has lost 30% of her expected yield/principal with no recourse. [7](#0-6)

        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
