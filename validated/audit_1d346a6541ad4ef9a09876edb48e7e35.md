### Title
Unbounded Iteration Over Pool Member Balance Trace Freezes Unclaimed Yield - (File: `src/pool/pool.cairo`)

### Summary
`Pool::claim_rewards` calls `calculate_rewards`, which contains an unbounded loop over the pool member's `pool_member_epoch_balance` trace. A pool member who repeatedly calls `add_to_delegation_pool` across many distinct epochs without claiming rewards will grow this trace without bound. When the trace is large enough, the loop exceeds Starknet's per-transaction gas limit, permanently bricking `claim_rewards` for that member.

### Finding Description
`calculate_rewards` in `src/pool/pool.cairo` iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` up to the current trace length:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

The code itself acknowledges the loop is unbounded. [1](#0-0) 

The trace grows because `set_member_balance` (called by `increase_member_balance`, called by `add_to_delegation_pool`) calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` implementation appends a **new** checkpoint whenever the epoch key differs from the last entry's key — i.e., one new entry per distinct epoch in which the member changes their balance. [2](#0-1) 

The `entry_to_claim_from` cursor advances only when `claim_rewards` succeeds. If the member never claims, the cursor stays at 0 and the loop must traverse the entire trace on the next claim attempt. [3](#0-2) 

`add_to_delegation_pool` is callable by the pool member or their reward address with no restriction on call frequency. [4](#0-3) 

There is no mechanism to prune or compact old trace entries.

### Impact Explanation
Once the trace is large enough that the loop exceeds the Starknet transaction gas limit, every call to `claim_rewards` for that pool member reverts. The member's accrued STRK rewards are permanently locked in the pool contract with no recovery path — **permanent freezing of unclaimed yield** (High).

### Likelihood Explanation
A pool member who calls `add_to_delegation_pool` once per epoch across hundreds or thousands of epochs without ever claiming rewards will reach the gas limit. Epochs are protocol-defined time windows; a long-term delegator who tops up their stake regularly and defers claiming is a realistic user pattern. The vulnerability is self-inflicted but irreversible once triggered.

### Recommendation
1. **Prune the trace on claim**: After `calculate_rewards` advances `entry_to_claim_from`, compact or delete the consumed prefix of `pool_member_epoch_balance` so the effective trace length stays bounded.
2. **Enforce a maximum trace depth**: Reject `add_to_delegation_pool` calls that would push the trace beyond a safe limit (e.g., 500 entries), or require the member to claim rewards first.
3. **Alternatively**, restructure reward accounting so that `claim_rewards` is O(1) regardless of balance-change history (e.g., store a running cumulative reward per unit and compute the delta at claim time without iterating the full trace).

### Proof of Concept
1. Pool member calls `add_to_delegation_pool` with a small amount once per epoch for N epochs (N ≈ several hundred to a few thousand, depending on Starknet's gas limit per transaction), never calling `claim_rewards`.
2. Each call in a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert`. [5](#0-4) 
3. After N epochs, the member calls `claim_rewards`. `calculate_rewards` is invoked with `entry_to_claim_from = 0` and `pool_member_trace_length = N`. [6](#0-5) 
4. The loop executes N iterations, each reading a storage slot and calling `find_sigma`. The transaction runs out of gas and reverts.
5. All subsequent `claim_rewards` calls also revert because `entry_to_claim_from` was never updated. The member's accumulated rewards are permanently inaccessible.
