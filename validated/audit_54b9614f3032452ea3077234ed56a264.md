### Title
Unbounded O(N) Iteration Over All Block Events During Post-Block Finalization Without Gas Metering - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

The `eip6110_events_parser` function iterates over **every event emitted in the entire block** during post-block finalization (after all transactions have been processed), with no resource charging for this iteration. An attacker can craft transactions that emit a large number of events from non-deposit-contract addresses, forcing the finalizer to scan all of them at zero cost. This is a direct analog of the Allora "SLOW ABCI METHODS" class: an unbounded O(N) loop in the block-closing path that is not metered against the attacker's gas/native budget.

---

### Finding Description

`eip6110_events_parser` is called unconditionally during `post_op` in both the sequencing and proving paths of the Ethereum block finalizer: [1](#0-0) 

The function body is:

```rust
for event in system.io.events_iterator() {
    if event.address != &DEPOSIT_CONTRACT_ADDRESS {
        continue;
    }
    ...
}
```

It calls `system.io.events_iterator()`, which returns an iterator over **all events emitted in the block** across all transactions: [2](#0-1) 

This loop is invoked from `post_op` in both the sequencing and proving post-tx operations: [3](#0-2) [4](#0-3) 

The `post_op` function is called **after** the transaction loop ends, outside of any per-transaction resource accounting. No native resource or gas is charged to any transaction for the work done in this loop. The loop cost is entirely borne by the node/prover, not by the transaction sender.

The block-level event limit (`MAX_NUMBER_OF_LOGS = 16_384`) bounds L2→L1 logs, but **not EVM events (LOG0–LOG4)**: [5](#0-4) [6](#0-5) 

EVM events are stored in `events_storage`, which is a separate structure from `logs_storage`. The `check_for_block_limits` function checks `logs_used` (L2→L1 logs count) but does **not** check the total number of EVM events: [7](#0-6) 

Per-event native cost is charged at emission time (`EVENT_STORAGE_BASE_NATIVE_COST + EVENT_TOPIC_NATIVE_COST * topics + EVENT_DATA_PER_BYTE_COST * data_len`), but this only covers the cost of **storing** the event, not the cost of **scanning all events at finalization time**. The `eip6110_events_parser` loop itself has no corresponding charge. [8](#0-7) 

---

### Impact Explanation

An attacker fills a block with transactions that emit many EVM events from addresses other than `DEPOSIT_CONTRACT_ADDRESS`. Each such event passes the address filter check and is skipped, but the loop still iterates over it. The finalization cost grows linearly with the total number of events in the block. Since this work is done outside the transaction execution loop and is not charged to any transaction, it represents unmetered computation in the block-closing path.

In the proving path, this unmetered work directly inflates the RISC-V cycle count for the prover, potentially causing the block to exceed the prover's `MAX_NATIVE_COMPUTATIONAL` budget in a way that was not accounted for during transaction admission. In the sequencing path, it slows down block production proportionally to the number of events.

The same pattern applies to `eip7002_system_part` and `eip7251_system_part` which are also called unconditionally in `post_op` and perform storage reads outside of transaction resource accounting.

---

### Likelihood Explanation

Any unprivileged transaction sender can emit EVM events cheaply (LOG0 costs 375 gas + 8 gas/byte of data). A block filled with transactions that each emit many events from arbitrary addresses would force the finalizer to scan all of them. The block gas limit bounds the total number of events indirectly, but the per-event gas cost does not account for the O(N) finalization scan. This is reachable by any user submitting standard EVM transactions.

---

### Recommendation

1. Charge native resources for the `eip6110_events_parser` scan proportional to the number of events iterated, either by pre-charging at event emission time (adding a per-event "finalization scan" overhead to the native cost) or by enforcing a hard cap on the total number of EVM events per block (analogous to `MAX_NUMBER_OF_LOGS` for L2→L1 logs).
2. Alternatively, maintain a separate counter of deposit-contract events during transaction execution so that `eip6110_events_parser` only needs to iterate over the pre-filtered subset.
3. Apply the same analysis to `eip7002_system_part` and `eip7251_system_part`.

---

### Proof of Concept

1. Deploy a contract at any address ≠ `DEPOSIT_CONTRACT_ADDRESS` that emits a LOG4 event with maximum data on every call.
2. Fill a block with transactions calling this contract, each emitting as many events as gas allows.
3. Observe that `eip6110_events_parser` iterates over all emitted events during `post_op`, performing address comparisons for each, with no native resource charge attributed to any transaction for this work.
4. The total finalization work is O(total_events_in_block), which is attacker-controlled and unbounded by any per-transaction resource limit specific to the finalization scan. [9](#0-8)

pub fn eip6110_events_parser<S: EthereumLikeTypes>(
    system: &System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {
                event_encountered = true;
                requests_hasher.update(&[DEPOSIT_REQUEST_EIP_7685_TYPE]);
            }
            let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
                panic!("invalid deposit event structure");
            };
        }
    }

    Ok(event_encountered)

fn check_for_block_limits<S: EthereumLikeTypes>(
    system: &mut System<S>,
    gas_used: u64,
    computational_native_used: u64,
    pubdata_used: u64,
    logs_used: u64,
    blob_gas_used: u64,
) -> Result<(), InvalidTransaction>
where
    S::IO: IOSubsystemExt,
    <S as SystemTypes>::Metadata: ZkSpecificPricingMetadata,
{
    if gas_used > system.get_gas_limit() {
        system_log!(
            system,
            "Block gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockGasLimitReached)
    } else if blob_gas_used > system.get_blob_gas_limit() {
        system_log!(
            system,
            "Block blob gas limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockBlobGasLimitReached)
    } else if !cfg!(feature = "resources_for_tester")
        && computational_native_used > MAX_NATIVE_COMPUTATIONAL
    {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block native limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockNativeLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && pubdata_used > system.get_pubdata_limit() {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block pubdata limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockPubdataLimitReached)
    } else if !cfg!(feature = "resources_for_tester") && logs_used > MAX_NUMBER_OF_LOGS {
        // ZKsync OS-specific resources are not checked for evm tester
        system_log!(
            system,
            "Block logs limit reached, invalidating transaction\n"
        );
        Err(InvalidTransaction::BlockL2ToL1LogsLimitReached)
    } else {
        Ok(())
    }
}
