### Title
Silent `uint64` Overflow in `MeterComputation` Bypasses Computation Limit and Underreports Execution Fees — (File: `fvm/meter/computation_meter.go`)

---

### Summary

The `MeterComputation` function accumulates execution effort using plain `uint64` multiplication (`w * intensity`) with no overflow guard. When the product wraps around, `m.computationUsed` silently drops to a small value, causing the limit check to pass and the fee-deduction path to charge far less than the actual work performed. The same silent-overflow pattern appears in `ComputationAvailable` and `ComputationFromIntensities`.

---

### Finding Description

In `fvm/meter/computation_meter.go`, three arithmetic sites share the same root cause:

**Site 1 — `MeterComputation` (line 100):**
```go
m.computationUsed += w * intensity          // uint64 × uint64, no overflow check
if m.computationUsed > m.params.computationLimit {
    return errors.NewComputationLimitExceededError(...)
}
``` [1](#0-0) 

**Site 2 — `ComputationAvailable` (line 117):**
```go
potentialComputationUsage := m.computationUsed + w*usage.Intensity
return potentialComputationUsage <= m.params.computationLimit
``` [2](#0-1) 

**Site 3 — `ComputationFromIntensities` (line 36):**
```go
result += weight * intensity    // accumulates across all kinds; no overflow guard
return result >> MeterExecutionInternalPrecisionBytes
``` [3](#0-2) 

`w` and `intensity` are both `uint64`. Go integer arithmetic wraps silently on overflow. If `w * intensity ≥ 2^64`, the product wraps to a small value. The subsequent `> computationLimit` check then passes, and the transaction continues executing as if it consumed almost no computation.

The internal precision scheme (`MeterExecutionInternalPrecisionBytes = 16`) already encodes weights as `weight_value << 16`, meaning the effective multiplier is already large. For example, the mainnet weight for `ComputationKindCreateAccount` is `2_143_437` (raw, not shifted), and for `ComputationKindEVMGasUsage` the raw weight is `3`. [4](#0-3) 

The computation limit is stored shifted: `computationLimit = userLimit << 16`. [5](#0-4) 

So the effective overflow threshold for a single `MeterComputation` call is:

```
w * intensity ≥ 2^64
intensity ≥ 2^64 / w
```

For `ComputationKindEVMGasUsage` (raw weight `3`, stored as `3` — not shifted, confirmed by `fvm/evm/handler/handler.go`), the threshold is `intensity ≥ ⌈2^64 / 3⌉ ≈ 6.1 × 10^18`. The EVM block-level gas limit is `math.MaxUint64 ≈ 1.84 × 10^19`, which is above this threshold. [6](#0-5) 

The `Merge` path has the same issue:
```go
m.computationUsed = m.computationUsed + child.computationUsed
``` [7](#0-6) 

---

### Impact Explanation

When overflow occurs:

1. **Computation limit bypass**: `m.computationUsed` wraps to a value below `computationLimit`, so `NewComputationLimitExceededError` is never returned. The transaction runs to completion regardless of actual resource consumption.

2. **Fee underreporting**: `TotalComputationUsed()` returns `m.computationUsed >> 16`, which is the wrapped (tiny) value. This is passed to `DeductTransactionFees` as `executionEffort`, causing the payer to be charged a fraction of the correct fee — or zero. [8](#0-7) [9](#0-8) 

3. **`ComputationAvailable` false positive**: Callers that gate expensive operations on `ComputationAvailable` will be told capacity exists when it does not, allowing further unbounded work.

The financial impact (paying less than owed) is a direct analog to the moeda token-amount miscalculation.

---

### Likelihood Explanation

**Moderate-low.** The overflow threshold for current mainnet weights is very high (e.g., ~6.1 × 10^18 EVM gas units for `ComputationKindEVMGasUsage = 3`). However:

- The EVM block gas limit is `math.MaxUint64`, which exceeds the overflow threshold for that weight.
- Execution effort weights are updatable by the service account via `SetExecutionEffortWeightsTransaction`; a weight increase lowers the overflow threshold proportionally.
- The `Merge` path accumulates across child meters, so repeated sub-transactions can reach the threshold without a single call needing to.
- No overflow guard exists anywhere in the metering stack, so any future weight change that crosses the threshold immediately becomes exploitable.

An attacker who can submit EVM transactions (any unprivileged EOA or COA) controls the gas consumed and therefore the `intensity` value reported to the meter.

---

### Recommendation

Replace all bare `w * intensity` and `result += weight * intensity` expressions with overflow-safe arithmetic. In Go this can be done with `math/bits.Mul64`:

```go
hi, lo := bits.Mul64(w, intensity)
if hi != 0 || lo > math.MaxUint64 - m.computationUsed {
    // treat as limit exceeded
    return errors.NewComputationLimitExceededError(m.params.TotalComputationLimit())
}
m.computationUsed += lo
```

Apply the same pattern in `ComputationAvailable`, `ComputationFromIntensities`, and `Merge`. This eliminates the precision-loss and overflow risk with no change to the observable semantics for non-overflowing inputs — exactly the simplification recommended in the moeda report.

---

### Proof of Concept

```
Given:
  weight w = 3  (ComputationKindEVMGasUsage, stored raw)
  computationLimit = DefaultComputationLimit << 16  (e.g., 9999 << 16 = 655,344,384)

Attacker submits an EVM transaction with gas limit = (2^64 / 3) + 1 = 6,148,914,691,236,517,206

MeterComputation called with intensity = 6,148,914,691,236,517,206:
  w * intensity = 3 * 6,148,914,691,236,517,206
               = 18,446,744,073,709,551,618
               = 2^64 + 2  →  wraps to 2  (mod 2^64)

m.computationUsed += 2   →  m.computationUsed = 2
2 > 655,344,384?  NO  →  no error returned

TotalComputationUsed() = 2 >> 16 = 0
DeductTransactionFees(payer, inclusionEffort, executionEffort=0)
→ attacker pays inclusion fee only; execution effort fee = 0
```

```go
func (executor *transactionExecutor) deductTransactionFees() (err error) {
	if !executor.env.TransactionFeesEnabled() {
		return nil
	}

	computationLimit := executor.txnState.TotalComputationLimit()

	computationUsed, err := executor.env.ComputationUsed()
	if err != nil {
		return errors.NewTransactionFeeDeductionFailedError(
			executor.proc.Transaction.Payer,
			computationLimit,
			err)
	}

	if computationUsed > computationLimit {
		computationUsed = computationLimit
	}

	_, err = executor.env.DeductTransactionFees(
		executor.proc.Transaction.Payer,
		executor.proc.Transaction.InclusionEffort(),
		computationUsed)

	if err != nil {
		return errors.NewTransactionFeeDeductionFailedError(
			executor.proc.Transaction.Payer,
			computationUsed,
			err)
	}
	return nil
