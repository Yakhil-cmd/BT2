### Title
Untrusted External Pool Server Response Fetched Twice (TOCTOU) Enables Farming Reward Redirection — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
The pool join and pool change flows fetch pool info from an external server **twice**: once for display (preview) and once at submit time. A malicious pool operator can serve a legitimate-looking `targetPuzzleHash` during the preview phase and substitute an attacker-controlled puzzle hash at submit time. Because no confirmation dialog re-displays the actual values used in the transaction, the user unknowingly commits their Plot NFT to redirect all farming rewards to the attacker's address.

### Finding Description
`PlotNFTSelectBase` uses the `usePoolInfo` hook to reactively fetch and display pool details (name, fee, `relativeLockHeight`, `targetPuzzleHash`) as the user types a pool URL. [1](#0-0) 

When the user clicks **Create** or **Change**, `handleSubmit` calls `prepareSubmitData`, which makes a **second, independent** HTTP request to the same pool server via `getPoolInfo(normalizedPoolUrl)`: [2](#0-1) 

The `targetPuzzleHash` and `relativeLockHeight` returned by this second fetch are passed directly to `pwJoinPool` without any re-display or confirmation step: [3](#0-2) 

The main-process handler for `FETCH_POOL_INFO` simply proxies the request to the pool server with no caching or integrity check: [4](#0-3) 

The only confirmation dialog in `handleSubmit` is for the "too many NFTs" edge case — it does not show the `targetPuzzleHash` or `relativeLockHeight` that will actually be committed on-chain: [5](#0-4) 

### Impact Explanation
`targetPuzzleHash` is the puzzle hash that receives all pooled farming rewards. If a malicious pool server substitutes its own puzzle hash at submit time, every XCH block reward earned by the user's plots is permanently redirected to the attacker. `relativeLockHeight` controls how many blocks the Plot NFT is locked to the pool; a malicious server can inflate this value at submit time (e.g., to `4,294,967,295` blocks), making it prohibitively expensive to leave. Both outcomes constitute a direct, irreversible loss of pooled farming rewards — matching the Critical/High impact scope for payout change and pooled farming reward accounting.

### Likelihood Explanation
Any operator of a publicly reachable pool server can execute this attack. The attacker does not need access to the user's machine, keys, or network. The user must enter the malicious pool URL, but the TOCTOU nature of the attack means even a careful user who reviews the displayed pool info is deceived: the preview and the transaction use separate HTTP responses. On testnet, HTTP pool URLs are also accepted (`usePoolInfo` only enforces HTTPS on mainnet), widening the attack surface. [6](#0-5) 

### Recommendation
1. **Cache the first fetch**: Store the pool info object returned by `usePoolInfo` in component state and pass it directly into `prepareSubmitData` instead of making a second network call.
2. **Final confirmation dialog**: Before calling `onSubmit`, display a confirmation dialog that shows the exact `targetPuzzleHash` and `relativeLockHeight` that will be written on-chain, so the user can compare them to what was previewed.
3. **Cap `relativeLockHeight`**: Enforce a reasonable maximum (e.g., the Chia reference pool uses 32 blocks) and warn the user if the pool returns a value above it.

### Proof of Concept
1. Attacker deploys a pool server at `https://evil-pool.example/`.
2. For the first request (preview), the server returns a legitimate-looking response: `target_puzzle_hash: <real_pool_hash>`, `relative_lock_height: 32`.
3. User enters the URL, reviews the displayed pool info, and clicks **Create** / **Change**.
4. `prepareSubmitData` issues a second `GET /pool_info` request. The server now returns `target_puzzle_hash: <attacker_controlled_hash>`, `relative_lock_height: 4294967295`.
5. `pwJoinPool` is called with the attacker's puzzle hash and the inflated lock height.
6. The Plot NFT is committed on-chain: all future farming rewards flow to the attacker's address, and the user cannot leave the pool without paying a prohibitive double fee for an astronomically long lock period.
