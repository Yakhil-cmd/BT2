Audit Report

## Title
Unbounded `disburse_maturity_in_progress` Vec Growth in SNS Governance Enables Storage Exhaustion - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance `disburse_maturity` function at line 1609 of `rs/sns/governance/src/governance.rs` unconditionally appends a `DisburseMaturityInProgress` entry to a neuron's Vec on every call, with no cap on the number of pending disbursements. The NNS governance equivalent enforces `MAX_NUM_DISBURSEMENTS = 10` before allowing a new entry. Any SNS neuron holder with `DisburseMaturity` permission can exploit this asymmetry to grow the Vec to hundreds of entries per neuron, inflating the SNS governance canister's heap and potentially causing a denial-of-service.

## Finding Description
In `rs/sns/governance/src/governance.rs`, the `disburse_maturity` function performs only two guards before pushing a new entry:

1. `percentage_to_disburse` must be in `[1, 100]` [1](#0-0) 
2. `worst_case_maturity_modulation >= transaction_fee_e8s` [2](#0-1) 

Neither guard limits the *count* of pending disbursements. The push is unconditional: [3](#0-2) 

By contrast, NNS governance defines `MAX_NUM_DISBURSEMENTS = 10` and enforces it before any push: [4](#0-3) [5](#0-4) 

The `DisburseMaturityInProgress` struct is stored inline in the neuron protobuf and serialized to the SNS governance canister's heap on every state write. [6](#0-5) 

The natural termination condition — maturity falling below the minimum disbursement threshold — is reached only after approximately 680+ calls for a neuron starting with 10^9 e8s (geometric decay at 1% per call, stopping when `0.99^n * 10^9 * 0.01 * 0.95 < 10,000 e8s`), yielding ~682 entries per neuron before the attack self-limits.

## Impact Explanation
Each `DisburseMaturityInProgress` entry serialized into the neuron protobuf consumes heap memory in the SNS governance canister. With ~682 entries per neuron and multiple neurons exploited simultaneously, the cumulative heap growth can exhaust the canister's memory allocation. This causes the SNS governance canister to trap on state writes or become unable to accept new neurons — a concrete application/platform-level DoS against the entire SNS instance. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS with concrete SNS governance harm**.

## Likelihood Explanation
The `DisburseMaturity` permission is a standard, user-grantable SNS neuron permission. Any neuron holder who has staked SNS tokens (a low but non-zero cost) and accumulated maturity can trigger this via the standard `manage_neuron` ingress endpoint. No privileged role, governance majority, or subnet compromise is required. The attack is repeatable and can be parallelized across multiple neurons.

## Recommendation
Add a per-neuron cap on pending maturity disbursements in SNS governance, mirroring the NNS guard:

```rust
const MAX_DISBURSE_MATURITY_IN_PROGRESS: usize = 10;

if neuron.disburse_maturity_in_progress.len() >= MAX_DISBURSE_MATURITY_IN_PROGRESS {
    return Err(GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Too many maturity disbursements in progress.",
    ));
}
```

This check should be inserted in `disburse_maturity` after the authorization check and before the maturity calculation, consistent with the NNS pattern at `rs/nns/governance/src/governance/disburse_maturity.rs` line 306. [5](#0-4) 

## Proof of Concept
1. Deploy an SNS with default parameters; create a neuron holding 10^9 e8s of maturity.
2. In a loop, call `manage_neuron` with `Command::DisburseMaturity { percentage_to_disburse: 1, to_account: None }` from the neuron controller.
3. Each call succeeds (maturity decreases geometrically) and appends one `DisburseMaturityInProgress` entry.
4. After ~682 iterations the remaining maturity falls below the minimum disbursement threshold and calls begin failing — but 682 entries are already stored.
5. Assert `neuron.disburse_maturity_in_progress.len() > 600` to confirm unbounded growth.
6. Repeat across multiple neurons to demonstrate cumulative heap exhaustion of the SNS governance canister.

A deterministic PocketIC integration test in `rs/sns/governance/tests/governance.rs` can reproduce this without mainnet interaction. [7](#0-6)
