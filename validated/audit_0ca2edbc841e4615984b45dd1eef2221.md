### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing `maxAdminSpreadFeeE6` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` in the factory passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap check, while the parallel `setPoolAdminFees` path enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees up to `uint16` max (65,535 ≈ 6.55% in E6 terms) on any bin, bypassing the factory's cap system and causing traders to pay more than the protocol-intended maximum fee.

### Finding Description

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` (hard ceiling 200,000 = 20%) before writing to the pool:

```solidity
// MetricOmmPoolFactory.sol line 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no equivalent check — it delegates unconditionally:

```solidity
// MetricOmmPoolFactory.sol line 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee values:

```solidity
// MetricOmmPool.sol line 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

In every swap direction, the bin additional fee is added directly to `baseFeeX64` with no downstream clamp:

```solidity
// MetricOmmPool.sol line 910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern applies to the sell direction: [5](#0-4) 

`BinState.addFeeBuyE6` / `addFeeSellE6` are `uint16`, so the maximum settable value is 65,535, which equals **6.5535%** in E6 units (1e6 = 100%). [6](#0-5) 

### Impact Explanation

Traders swapping through a bin with `addFeeBuyE6 = 65_535` pay up to **6.5535% additional fee** on top of the base spread fee. This is a direct, quantifiable loss of swap proceeds extracted from traders above any protocol-intended limit. The fee is applied in the swap math before the output amount is settled, so the trader receives less output (or pays more input) than the pool's stated fee cap would permit. This meets the "direct loss of user principal" and "bad-price execution" criteria.

### Likelihood Explanation

The pool admin is a semi-trusted role that the contest rules explicitly scope as "semi-trusted only inside caps." The factory's cap system (`maxAdminSpreadFeeE6`) is the mechanism that bounds pool admin power. The absence of a cap on `setPoolBinAdditionalFees` is an asymmetric gap — every other admin fee setter has a cap check, this one does not. Any pool admin (including a compromised multisig or a pool created with a malicious admin) can exploit this without any factory owner involvement.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Introduce a `maxBinAdditionalFeeE6` constant or factory-level cap (e.g., bounded by `HARD_MAX_SPREAD_FEE_E6`) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
if (addFeeBuyE6 > maxBinAdditionalFeeE6) revert BinAdditionalFeeTooHigh();
if (addFeeSellE6 > maxBinAdditionalFeeE6) revert BinAdditionalFeeTooHigh();
```

### Proof of Concept

1. Deploy a pool with a controlled `poolAdmin`.
2. Call `factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` as the pool admin — no revert.
3. Execute a swap through bin 0. Measure the fee extracted: it will include the base `spreadFeeE6` **plus** 6.5535% additional fee.
4. Compare to a swap on the same pool with `addFeeBuyE6 = 0` — the difference is the uncapped additional fee extracted from the trader.

The test `test_setPoolBinAdditionalFees_updatesStorage_emitsEvent` in `MetricOmmPoolFactory.t.sol` confirms the factory accepts arbitrary `uint16` values (500, 700, 10, 20) with no cap enforcement. [7](#0-6)
