### Title
Missing Identity-Point Validation for `app_pk` in CKD Protocol Allows Coordinator to Learn Confidential Derived Secret - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts `app_pk: PublicKey` (a `blstrs::G1Projective` point) without checking whether it is the group identity element. If a malicious caller passes the identity point as `app_pk`, the ElGamal blinding term `y_i * A` collapses to zero for every participant, causing each participant to transmit their unblinded BLS signature share `x_i * H(pk, app_id)` directly to the coordinator. The coordinator aggregates these into the full unblinded BLS signature `msk * H(pk, app_id)`, which is the confidential derived secret `s`. This breaks the core confidentiality guarantee of CKD.

---

### Finding Description

The CKD protocol is designed so that each MPC node `i` computes:

```
C_i = S_i + y_i * A    where S_i = x_i * H(pk, app_id), A = app_pk
Y_i = y_i * G_1
```

and sends `(λ_i * Y_i, λ_i * C_i)` to the coordinator. The coordinator aggregates:

```
C = msk * H(pk, app_id) + (Σ λ_i * y_i) * A
Y = (Σ λ_i * y_i) * G_1
```

The app recovers the secret as `sig = C - a * Y`, where `a` is the app's private key satisfying `A = a * G_1`. The blinding term `(Σ λ_i * y_i) * A` is what hides `msk * H(pk, app_id)` from the coordinator.

In `compute_signature_share`, the blinding is computed as:

```rust
let big_c = big_s + app_pk * y.0;
``` [1](#0-0) 

If `app_pk` is the identity element `O ∈ G1`, then `app_pk * y.0 = O`, so `big_c = big_s = x_i * H(pk, app_id)`. Every participant sends their unblinded BLS share directly to the coordinator.

The `ckd()` entry point performs no check on `app_pk`:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,   // ← no identity check
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
``` [2](#0-1) 

The only validations performed are participant-list checks (count, duplicates, membership): [3](#0-2) 

`PublicKey` is a type alias for `blstrs::G1Projective`, which can freely hold the identity element: [4](#0-3) 

Note that while `BLS12381G1Group::deserialize` does reject the identity element, `app_pk` is passed as a native Rust value, not through that deserialization path: [5](#0-4) 

---

### Impact Explanation

**Critical — Disclosure of confidential derived secrets.**

When `app_pk = O`:

- Each participant's `C_i = x_i * H(pk, app_id)` (unblinded BLS share).
- The coordinator aggregates `C = msk * H(pk, app_id)` — the full unblinded BLS signature.
- The coordinator directly learns `s = msk * H(pk, app_id)`, which is the confidential derived secret the protocol is designed to keep hidden from all MPC nodes.

The CKD security requirement states: *"No single node in the MPC network should be capable of computing `s`."* A malicious app bypasses this entirely by supplying the identity point. [6](#0-5) 

---

### Likelihood Explanation

The `app_pk` is an externally supplied parameter originating from the requesting application. Per the protocol specification, the app generates a fresh ElGamal key pair `(a, A)` and submits `A` as part of the CKD request. A malicious app can trivially submit `A = G1Projective::identity()`. The `ckd()` function is a public library API with no guard against this value. No privileged access or cryptographic break is required — only the ability to call `ckd()` with a chosen `app_pk`. [7](#0-6) 

---

### Recommendation

Add an identity-point check for `app_pk` at the start of `ckd()`, before the protocol is launched:

```rust
use elliptic_curve::Group;

if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the pattern already used elsewhere in the codebase for identity-element rejection: [8](#0-7)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
