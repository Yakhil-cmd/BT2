### Title
ETH Sent to `tick()` Is Permanently Stuck When No Minting Is Required — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in the L1 `RewardSupplier` contract is `payable` and callable by any account. The `msg.value` ETH is only forwarded inside the `if (amountToMint > 0)` branch. When `amountToMint == 0` (no pending L2 mint requests), the function returns silently without using or refunding the ETH, permanently locking it in the contract. No withdrawal or sweep mechanism exists in the contract.

---

### Finding Description

`tick()` is declared `external payable` and splits `msg.value` between two downstream calls only when there is work to do:

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol
function tick() external payable {                          // line 107
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {                                 // line 111
        // ...consume messages, mint tokens...
        uint256 msgFee = msg.value / 2;                     // line 125
        bridge().depositWithMessage{value: msgFee}(...);    // line 126
        // ...
        msgFee = msg.value - msgFee;                        // line 135
        messagingContract().sendMessageToL2{value: msgFee}(...); // line 137
    }
    // ← if amountToMint == 0, msg.value is never used and never refunded
}
``` [1](#0-0) 

When `amountToMint > 0`, the full `msg.value` is consumed (`msg.value/2` + `msg.value - msg.value/2`). When `amountToMint == 0`, the function exits the `if` block and returns, leaving `msg.value` trapped in the contract with no refund path and no governance withdrawal function visible in `RewardSupplier.sol` or `RewardSupplierStorage.sol`. [2](#0-1) 

The analog to the original report is direct: in the original, ETH pre-funded for a batch of deployments is stranded when one deployment silently fails. Here, ETH pre-funded for a minting tick is stranded when the tick is a no-op.

---

### Impact Explanation

Any ETH sent to `tick()` when `amountToMint == 0` is permanently frozen in the `RewardSupplier` contract. Because the contract has no `receive()` fallback, no `withdraw()`, and no governance sweep function, the ETH cannot be recovered. This constitutes **permanent freezing of funds**.

Matches allowed impact: **Temporary/Permanent freezing of funds**.

---

### Likelihood Explanation

`tick()` has no access control — any account can call it. The condition `amountToMint == 0` is the normal state between reward epochs when no L2 mint-request messages are pending. A caller who:

1. Does not call `requiredMinting()` as a pre-check, or
2. Is front-run between their pre-check and their `tick()` call (another caller drains the pending messages first),

will have their ETH permanently locked. This is a realistic, low-effort path for any unprivileged caller. [3](#0-2) 

---

### Recommendation

Add an ETH refund when the function is a no-op, and/or add a governance sweep for stranded ETH:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund ETH when there is nothing to mint.
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, add a privileged `sweepETH(address recipient)` function so governance can recover any stranded ETH.

---

### Proof of Concept

1. Observe that no L2 mint-request messages are pending: `requiredMinting()` returns `(0, 0)`.
2. Call `tick{value: 1 ether}()` from any EOA.
3. The `if (amountToMint > 0)` branch is skipped; the function returns.
4. `address(rewardSupplier).balance` increases by `1 ether`.
5. No function in `RewardSupplier.sol` or `RewardSupplierStorage.sol` allows recovery of this ETH. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
