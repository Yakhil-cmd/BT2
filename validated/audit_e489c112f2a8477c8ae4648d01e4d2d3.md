### Title
Missing Sender Authorization in Bank Precompile `transfer` Allows Any Contract to Drain Another User's Native Tokens - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile (`BankContract`) accepts the `sender` address as a caller-supplied argument but never verifies that `sender == contract.Caller()`. Any EVM contract can invoke `bank.transfer(victim, attacker, amount)` and drain `evm/<callerContract>` native tokens from any holder without their consent.

### Finding Description
In `BankContract.Run`, the `TransferMethodName` case unpacks `sender` directly from the ABI-encoded call arguments:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())   // denom = "evm/<callerContract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no guard of the form `if sender != contract.Caller() { return error }`. The only check performed is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses. [2](#0-1) 

Compare this with the `Executor.exec` helper used by the relayer precompile, which explicitly enforces `caller == signer`:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
``` [3](#0-2) 

No equivalent guard exists in the bank precompile's `transfer` path.

### Impact Explanation
The denom transferred is `evm/<callerContract>`. These are native Cosmos SDK tokens representing a CRC20 balance in native form, created when a CRC20 contract calls `bank.mint`. A user who calls `moveToNative` (burns CRC20 tokens and receives `evm/<contract>` native tokens) holds real, convertible assets. The attacker's contract can call `bank.transfer(victim, attacker, amount)` to move those native tokens to the attacker without the victim's approval. This is an unauthorized transfer of precompile-controlled assets — Critical impact per the allowed scope.

The same structural flaw exists in the `burn` case: `addr` (the account to burn from) is taken from args with no check that `addr == contract.Caller()`, allowing any contract to destroy another user's `evm/<callerContract>` tokens. [4](#0-3) 

### Likelihood Explanation
The bank precompile is a publicly callable EVM precompile at address `0x0000000000000000000000000000000000000064`. Any deployed EVM contract can call it. An attacker deploys a contract that presents itself as a legitimate CRC20 token, induces users to call `moveToNative` (minting `evm/<attacker_contract>` native tokens to them), and then calls `bank.transfer(victim, attacker, amount)` to drain those tokens. No privileged access is required. [5](#0-4) 

### Recommendation
Add an authorization check in the `TransferMethodName` case to enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender is not authorized: must equal caller")
}
```

Similarly, add a check in the `BurnMethodName` case that `recipient == contract.Caller()` (i.e., a contract may only burn tokens from its own native account, not from arbitrary addresses).

### Proof of Concept

1. Attacker deploys `MaliciousToken` at `0xMalicious`, which exposes a `depositAndConvert` function that calls `bank.mint(msg.sender, amount)` — minting `evm/0xMalicious` native tokens to depositors.
2. Victim calls `depositAndConvert(100)`. Victim now holds 100 `evm/0xMalicious` native tokens.
3. Attacker calls `MaliciousToken.steal(victim, attacker, 100)`, which internally calls:
   ```solidity
   IBankModule(0x64).transfer(victim, attacker, 100);
   ```
4. `BankContract.Run` executes `bankKeeper.SendCoins(victim, attacker, 100 evm/0xMalicious)` with no authorization check.
5. Victim's 100 `evm/0xMalicious` tokens are transferred to attacker. Victim can no longer convert them back to CRC20 tokens. [6](#0-5)
