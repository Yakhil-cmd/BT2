### Title
Unhandled Refund Failure in `exit_to_near_precompile_callback` Can Cause Permanent User Fund Loss - (File: engine/src/contract_methods/connector.rs)

---

### Summary

When a user's `ExitToNear` cross-chain operation fails (e.g., the NEP-141 transfer to the recipient is rejected), `exit_to_near_precompile_callback` attempts to restore the user's burned EVM tokens via `engine::refund_on_error`. If that refund EVM execution itself fails or reverts, the callback propagates the error and panics. Because the token burn in the EVM was committed in the prior transaction, the user's funds are permanently destroyed with no recovery path.

---

### Finding Description

The `ExitToNear` precompile flow is a two-step cross-chain operation:

1. **Main transaction (`submit`):** The EVM burns the user's ERC-20 tokens (or ETH) and schedules a NEAR promise to deliver NEP-141 tokens (or NEAR) to the recipient.
2. **Callback transaction (`exit_to_near_precompile_callback`):** Checks whether the promise succeeded. On failure, it calls `engine::refund_on_error` to re-mint the burned tokens back to the user.

The callback code is:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    if !refund_result.status.is_ok() {
        return Err(errors::ERR_REFUND_FAILURE.into());
    }
    Some(refund_result)
}
``` [1](#0-0) 

There are two distinct failure paths that both cause the callback to return an error (and therefore panic in the NEAR runtime):

- **Path A:** `refund_on_error` returns `Err(...)` — propagated by the `?` operator.
- **Path B:** `refund_on_error` returns `Ok(result)` but `result.status` is a failed EVM status (e.g., `ExitReason::Revert`) — caught by the explicit `if !refund_result.status.is_ok()` check.

`refund_on_error` executes an EVM transaction to re-mint tokens. This EVM execution can fail due to:
- The bridged ERC-20 contract being paused or having a blacklist that includes the user's address.
- Insufficient gas allocated to the callback for the re-mint EVM execution.
- Any other EVM-level revert condition in the token contract.

Because the token burn happened in the committed main transaction and the callback's state changes are reverted on panic, the user's tokens are permanently destroyed.

---

### Impact Explanation

**Permanent freezing (destruction) of user funds.** The user's ERC-20 tokens are burned in the EVM during the main `submit` transaction. If the exit promise fails and the refund callback also fails, there is no on-chain state recording the owed amount and no recovery mechanism. The tokens are gone. This matches the **Critical — Permanent freezing of funds** impact category.

---

### Likelihood Explanation

**Low but realistic.** Two independent failures must coincide:

1. The NEP-141 transfer to the recipient fails (e.g., recipient account not registered for the token, rate-limit exceeded on the connector, or connector contract paused).
2. The re-mint EVM execution in `refund_on_error` also fails (e.g., the ERC-20 contract is paused at the moment of the callback, or the callback runs out of gas because the gas budget for the callback was not sized for a worst-case re-mint).

Both conditions are reachable by normal protocol state without any privileged action. A connector pause or ERC-20 pause is an admin action, but it can happen concurrently with a user's in-flight exit. The gas budget issue is a configuration risk that requires no admin involvement.

---

### Recommendation

1. **Do not panic on refund failure.** Instead of returning `Err` when `refund_on_error` fails, store the owed refund amount and recipient address in persistent contract storage and emit a log.
2. **Add a `claim_failed_refund` entry point** that allows the original sender to pull their owed tokens once the blocking condition (e.g., pause) is lifted.
3. **Increase the gas budget** allocated to `exit_to_near_precompile_callback` to ensure the re-mint EVM execution has sufficient gas even in worst-case token contract paths.

---

### Proof of Concept

1. User calls the `ExitToNear` precompile from an EVM contract, specifying 1000 USDC to bridge out. The EVM burns 1000 USDC from the user's balance. A NEAR promise is scheduled to transfer 1000 NEP-141 USDC to the recipient.
2. The NEP-141 transfer fails because the recipient account has not registered storage for the USDC token contract.
3. `exit_to_near_precompile_callback` is invoked. `handler.promise_result(0)` returns `Failed`, so the `else if let Some(args) = args.refund` branch is taken.
4. Concurrently (or due to a prior admin action), the bridged USDC ERC-20 contract on Aurora has been paused.
5. `engine::refund_on_error` executes the re-mint EVM call. The ERC-20 `mint` function reverts because the contract is paused. `refund_result.status` is `ExitReason::Revert(...)`.
6. The check `if !refund_result.status.is_ok()` triggers, returning `Err(errors::ERR_REFUND_FAILURE)`.
7. The NEAR runtime reverts the callback's state changes. The user's 1000 USDC burn from step 1 is permanent. The user has lost 1000 USDC with no recovery path. [2](#0-1)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
