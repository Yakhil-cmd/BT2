### Title
`calculateNFTRoyalties` Uses Wrong `nftSellerNetAmount` for `TokenForNFT` Direction, Displaying Inflated "Net Proceeds" to NFT Sellers - (File: `packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` unconditionally sets `nftSellerNetAmount = amount` (the full offered amount) regardless of exchange direction. For `TokenForNFT` offers (buyer creates offer, NFT seller is taker), royalties are deducted from the offered tokens before the seller receives them, so the correct net proceeds are `amount - royaltyAmount`. The commented-out code is the remnant of a ternary that was partially removed, leaving only the `NFTForToken`-correct branch active for both directions.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes royalty-related display values:

```typescript

  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,

  const royaltyAmountString: string = formatAmount(royaltyAmount);

  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken
      ? amount + royaltyAmount
      : amount + makerFee + royaltyAmount;   // <-- also wrong for TokenForNFT
``` [1](#0-0) 

The commented-out fragment is the discarded `else` branch of a ternary. The original intent was:

```typescript
const nftSellerNetAmount =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount
    : parseFloat((amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12));
```

**Correct semantics per exchange direction:**

| Direction | Who pays royalties | Seller net | Buyer total |
|---|---|---|---|
| `NFTForToken` (seller is maker) | Taker (buyer) pays on top | `amount` ✓ | `amount + royaltyAmount` ✓ |
| `TokenForNFT` (buyer is maker) | Taker (NFT seller) deducted from offered tokens | `amount - royaltyAmount` ✗ shows `amount` | `amount + makerFee` ✗ shows `amount + makerFee + royaltyAmount` |

The viewer tooltip at line 588-591 explicitly states the contract: *"The net proceeds include the asking price, minus any associated creator fees."* The code violates this. [2](#0-1) 

**Impact chain in `NFTOfferViewer.tsx`:**

1. `nftSaleInfo` is computed via `calculateNFTRoyalties(amount, ..., exchangeType)` — `nftSaleInfo.nftSellerNetAmount = amount` (wrong for `TokenForNFT`). [3](#0-2) 

2. `overrideNFTSellerAmount` is set to `chiaToMojo(nftSaleInfo?.nftSellerNetAmount)` = full offered amount in mojos, and passed into `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, overriding the "You will receive" line with the gross amount instead of the net. [4](#0-3) 

3. `OfferSummaryTokenRow` uses `overrideNFTSellerAmount ?? originalAmount` as the displayed amount — so the override actively replaces the correct on-chain amount with the inflated value. [5](#0-4) 

4. The "Net Proceeds" field (only shown for `TokenForNFT`) renders `nftSaleInfo?.nftSellerNetAmount` = `amount` — the full offered amount, not `amount - royaltyAmount`. [6](#0-5) 

**Same bug in `NFTOfferEditor.tsx`:** the "They will receive" label during offer creation also renders `nftSellerNetAmount` = `amount` for `TokenForNFT`, misleading the buyer about what the seller will actually receive. [7](#0-6) 

### Impact Explanation

An NFT seller who opens an imported `TokenForNFT` offer (a buyer's offer to purchase their NFT) is shown:

- **"You will receive: 100 XCH"** — wrong; actual receipt is `100 - royalty` XCH
- **"Creator Fee (10%): 10 XCH"** — correct
- **"Net Proceeds: 100 XCH"** — wrong; the label promises `amount - royalties` but delivers `amount`

The seller sees a prominently displayed "Net Proceeds" figure that is inflated by the full royalty amount. If they accept the offer based on this figure, they receive less than the displayed amount. For a 10% royalty on a 100 XCH offer, the seller loses 10 XCH relative to what the UI promised. This satisfies the High impact criterion: offer state is displayed with the wrong amount, causing a user to approve an offer for the wrong expected receipt.

### Likelihood Explanation

Any `TokenForNFT` offer involving an NFT with a non-zero `royaltyPercentage` triggers this path. NFT royalties are a standard feature of the Chia NFT1 standard and are common in practice. The offer viewer is the primary UI surface for accepting imported offers, making this reachable by any unprivileged NFT seller who receives a buyer's offer file.

### Recommendation

Restore the direction-aware calculation in `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount
    : parseFloat((amount - royaltyAmount).toFixed(12));

const totalAmount: number =
  exchangeType === NFTOfferExchangeType.NFTForToken
    ? amount + royaltyAmount
    : amount + makerFee;   // royalties come out of the offered amount, not added on top
```

### Proof of Concept

1. NFT owner (Alice) holds an NFT with 10% royalty.
2. Buyer (Bob) creates a `TokenForNFT` offer: "I'll give 100 XCH for Alice's NFT."
3. Alice opens the offer file in the GUI. The viewer calls `calculateNFTRoyalties(100, makerFee, 10, TokenForNFT)`.
4. `nftSellerNetAmount = 100` (bug — should be `90`).
5. Alice sees: **"Net Proceeds: 100 XCH"** and **"You will receive: 100 XCH"**.
6. Alice accepts the offer. On-chain, 10 XCH is routed to the royalty address; Alice receives **90 XCH**.
7. Alice lost 10 XCH relative to the amount the GUI displayed as her net proceeds.
