### Title
TOCTOU Race in Pool Join: Displayed `targetPuzzleHash` Differs from Submitted Value - (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
When a user joins or creates a PlotNFT pool, the GUI fetches pool info **twice** from the pool server: once to display the "Verify Pool Details" panel (via `usePoolInfo`) and once at submit time inside `prepareSubmitData` (via `getPoolInfo`). A malicious pool operator can serve a legitimate `targetPuzzleHash` for the first request and a different, attacker-controlled `targetPuzzleHash` for the second request. The user sees and verifies one puzzle hash but the on-chain transaction commits to a different one, silently redirecting all future farming rewards to the attacker.

### Finding Description

**Step 1 – Display fetch (for user verification):**
`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)`, which calls `getPoolInfo` → `window.appAPI.fetchPoolInfo(poolUrl)` → `FETCH_POOL_INFO` IPC → `fetchJSON(poolUrl + '/pool_info')`. The result is rendered in the "Verify Pool Details" card, including the `targetPuzzleHash` field. [1](#0-0) [2](#0-1) 

**Step 2 – Submit fetch (for actual transaction):**
When the user clicks "Create", `handleSubmit` calls `prepareSubmitData`, which independently calls `getPoolInfo(normalizedPoolUrl)` again. The `targetPuzzleHash` returned by this **second** HTTP request is what gets embedded in `initialTargetState` and sent to `createNewPoolWallet` / `pwJoinPool`. [3](#0-2) 

**Step 3 – Transaction submission:**
`PlotNFTAdd.handleSubmit` passes `initialTargetState` (containing the second-fetch `targetPuzzleHash`) directly to `createNewPoolWallet`, which issues `create_new_wallet` to the daemon. `PlotNFTChangePool.handleSubmit` passes it to `pwJoinPool` (`pw_join_pool`). [4](#0-3) [5](#0-4) 

**The gap:** There is no confirmation dialog that re-displays the `targetPuzzleHash` that will actually be used in the transaction. The user's only opportunity to verify it is the "Verify Pool Details" panel, which is populated from the first fetch. The second fetch happens silently at submit time with no UI feedback. [6](#0-5) [7](#0-6) 

### Impact Explanation

The `targetPuzzleHash` is the pool's reward collection address embedded in the PlotNFT singleton on-chain. If the attacker substitutes their own puzzle hash, **all XCH block rewards farmed by the user's plots are paid to the attacker's address** for as long as the PlotNFT remains joined to that pool. Changing pools requires a second on-chain transaction and a `relativeLockHeight` waiting period (typically 32 blocks), during which rewards continue to flow to the attacker. This is a direct, irreversible payout change affecting pooled farming rewards.

### Likelihood Explanation

Any operator of a pool server the user connects to can execute this attack with zero cryptographic capability. The attack window is the time between the display fetch and the submit fetch — typically seconds to minutes while the user reads the "Verify Pool Details" panel. The pool server simply needs to return a different JSON body for the second `/pool_info` request. No network interception or key material is required. The attack is invisible to the user because the GUI shows no confirmation of the `targetPuzzleHash` actually used.

### Recommendation

Eliminate the second fetch. Cache the result of the first `getPoolInfo` call (from `usePoolInfo`) and pass it into `prepareSubmitData` instead of re-fetching. Concretely:

- Expose the resolved `poolInfo` object from `PlotNFTSelectBase` / `usePoolInfo` up to `PlotNFTSelectPool`.
- In `prepareSubmitData`, accept the already-fetched `poolInfo` as a parameter and read `targetPuzzleHash` / `relativeLockHeight` from it instead of calling `getPoolInfo` again.
- Optionally, add a final confirmation dialog that displays the `targetPuzzleHash` that will be committed on-chain before the user clicks "Create".

### Proof of Concept

1. Attacker operates a pool server at `https://evil-pool.example`.
2. The server's `/pool_info` endpoint tracks request count. For the **first** request it returns `targetPuzzleHash: <legitimate_hash>` and `relativeLockHeight: 32`. For the **second** request it returns `targetPuzzleHash: <attacker_hash>`.
3. Victim opens the GUI, navigates to "Add a Plot NFT", selects "Connect to pool", enters `https://evil-pool.example`.
4. The "Verify Pool Details" panel renders with `<legitimate_hash>` — victim verifies and clicks "Create".
5. `prepareSubmitData` fires a second `fetchPoolInfo` call; the server returns `<attacker_hash>`.
6. `createNewPoolWallet` is called with `initialTargetState.targetPuzzleHash = <attacker_hash>`.
7. The PlotNFT singleton is created on-chain with the attacker's puzzle hash as the reward destination.
8. All subsequent block rewards farmed by the victim's plots are paid to the attacker. [8](#0-7) [9](#0-8) [10](#0-9)

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
