### Title
Gateway Stateful Validator Checks Only L2 Gas Price, Admitting Transactions With Insufficient L1/L1DataGas Prices - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's stateful resource-bounds validation (`validate_tx_l2_gas_price_within_threshold`) checks only the transaction's `l2_gas.max_price_per_unit` against the previous block's L2 gas price threshold. It silently skips the equivalent check for `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit`. A transaction with a valid L2 gas price but zero (or below-threshold) L1/L1DataGas prices passes gateway admission and enters the mempool, only to be rejected by the blockifier's `check_fee_bounds` during block building.

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` calls `validate_tx_l2_gas_price_within_threshold`, which contains an explicit `TODO` acknowledging the gap:

```rust
// TODO(Arni): Consider running this validation for all gas prices.
fn validate_tx_l2_gas_price_within_threshold(
    &self,
    tx_resource_bounds: ValidResourceBounds,
    previous_block_l2_gas_price: NonzeroGasPrice,
) -> StatefulTransactionValidatorResult<()> {
    match tx_resource_bounds {
        ValidResourceBounds::AllResources(tx_resource_bounds) => {
            let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
            // ... threshold check on l2_gas only ...
        }
        ValidResourceBounds::L1Gas(_) => {
            // No validation required for legacy transactions.
        }
    }
    Ok(())
}
``` [1](#0-0) 

The stateless validator has the same gap — it checks only `l2_gas.max_price_per_unit` against `min_gas_price`: [2](#0-1) 

By contrast, the blockifier's `check_fee_bounds` (called during `perform_pre_validation_stage`) validates **all three** gas prices — L1, L1DataGas, and L2 — against the actual block prices: [3](#0-2) 

The invariant broken is: every transaction admitted to the mempool must have `max_price_per_unit ≥ current_block_price` for every resource type it commits to. The gateway enforces this only for L2 gas, leaving L1 gas and L1 data gas unchecked.

### Impact Explanation

An attacker submits `AllResources` V3 transactions with:
- `l2_gas.max_price_per_unit` = valid (≥ threshold, passes gateway)
- `l1_gas.max_price_per_unit` = 0
- `l1_data_gas.max_price_per_unit` = 0

These transactions pass both the stateless and stateful gateway validators and are inserted into the mempool. When the batcher dequeues them, `check_fee_bounds` in the blockifier rejects them with `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }`. The mempool is polluted with transactions that can never be sequenced, consuming capacity and batcher cycles. This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

### Likelihood Explanation

Any unprivileged user can submit a V3 `AllResources` transaction via the public RPC `addInvokeTransaction` / `addDeclareTransaction` / `addDeployAccountTransaction` endpoints. No special privilege is required. The only constraint is that `l2_gas.max_price_per_unit` must meet the threshold, which is publicly observable from the previous block header.

### Recommendation

Extend `validate_tx_l2_gas_price_within_threshold` (or rename it to `validate_tx_gas_prices_within_threshold`) to also compare `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit` against their respective previous-block prices, mirroring the three-resource check already present in the blockifier's `check_fee_bounds`:

```rust
// TODO(Arni): Consider running this validation for all gas prices.
fn validate_tx_l2_gas_price_within_threshold(...) {
    match tx_resource_bounds {
        ValidResourceBounds::AllResources(tx_resource_bounds) => {
            // existing l2_gas check ...
            // ADD: check l1_gas.max_price_per_unit >= threshold(previous_block_l1_gas_price)
            // ADD: check l1_data_gas.max_price_per_unit >= threshold(previous_block_l1_data_gas_price)
        }
        ...
    }
}
```

The same fix should be applied to the stateless validator's `validate_resource_bounds`, which also only enforces a floor on `l2_gas.max_price_per_unit`.

### Proof of Concept

1. Read the current block's L2 gas price (valid), L1 gas price (P1), and L1 data gas price (P2) from the chain.
2. Craft a V3 invoke transaction with `AllResources`:
   - `l2_gas = { max_amount: 1, max_price_per_unit: <valid_l2_price> }`
   - `l1_gas = { max_amount: 1_000_000, max_price_per_unit: 0 }` ← below actual price P1
   - `l1_data_gas = { max_amount: 1_000_000, max_price_per_unit: 0 }` ← below actual price P2
3. Submit via `starknet_addInvokeTransaction`.
4. **Expected (correct) behavior**: gateway rejects with `GAS_PRICE_TOO_LOW` for L1 gas.
5. **Actual behavior**: gateway accepts; transaction enters mempool. Batcher later rejects it with `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }`.

The root cause is the incomplete check at: [4](#0-3) 

which is the direct analog of the PoolTogether M-27 bug where `prizePool.claimCount()` was used without `prizePool.canaryClaimCount()` — in both cases, a total that should include all categories only includes one.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,

            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);

                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
