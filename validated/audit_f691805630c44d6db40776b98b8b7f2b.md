### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` and `presign` Allows Threshold Security Requirement to Be Bypassed - (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate only the upper bound of the caller-supplied `threshold` parameter (i.e., `threshold <= participants.len()`), but omit the lower-bound check (`threshold >= 2`) that is consistently enforced in every other protocol entry point in the library. A malicious coordinator or participant can supply `threshold = 0` or `threshold = 1`, bypassing the minimum-threshold security requirement and causing FROST signing to produce a cryptographically invalid, unusable signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
// ← no lower-bound check; threshold = 0 or 1 passes silently
``` [2](#0-1) 

The same omission exists in `presign`:

```rust
// src/frost/mod.rs lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no built-in minimum enforcement: [4](#0-3) 

---

### Impact Explanation

The `threshold` value flows directly into Lagrange interpolation during FROST signing. If `threshold = 1` is accepted, the interpolation uses coefficients computed for a degree-0 polynomial, which is inconsistent with the degree-(t−1) polynomial used during key generation (where `t >= 2` was enforced). The resulting partial-signature aggregation produces a scalar `s` that does not satisfy the verification equation against the master public key. Every honest verifier rejects the output. If `threshold = 0`, the interpolation is degenerate and may panic or produce a zero scalar, which is also rejected.

**Impact category matched**: *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

The entry points `assert_sign_inputs` and `presign` are public library API. Any caller — including a malicious coordinator who controls the signing session setup — can supply an arbitrary `threshold` value. No out-of-band enforcement prevents `threshold = 1` from reaching these functions. The coordinator role is explicitly within the library's threat model (the robust ECDSA path documents split-view attack mitigations against a malicious coordinator). Likelihood is **Medium**: it requires a malicious or buggy coordinator but no cryptographic capability.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already uses, in both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place it immediately after the upper-bound check in each function, mirroring the pattern at `src/dkg.rs` lines 580–582. [5](#0-4) 

---

### Proof of Concept

1. A malicious coordinator calls `assert_sign_inputs` with a valid participant list (≥ 2 members) and `threshold = 1`. The function returns `Ok(participants)` — no error. [6](#0-5) 

2. The coordinator then calls `presign` with `args.threshold = 1`. Again, only the upper-bound check fires; `1 <= participants.len()` is true, so the presign protocol proceeds. [3](#0-2) 

3. During signing, Lagrange coefficients are computed for a threshold-1 scheme. Because the key was generated with `threshold >= 2` (enforced by `assert_key_invariants`), the interpolated signing scalar is incorrect. [7](#0-6) 

4. The aggregated FROST signature fails verification against the master public key. All honest parties receive an unusable output; signing is effectively denied for the session.
