### Title
SwapAllowlistExtension checks router address instead of original caller, enabling allowlist bypass via MetricOmmSimpleRouter — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

### Summary
`SwapAllowlistExtension.beforeSwap` receives the pool's `msg.sender` (the router) as `sender` and checks `allowedSwapper[pool][router]`. Any non-allowlisted user who routes through an allowlisted `MetricOmmSimpleRouter` instance passes the gate unconditionally.

### Finding Description
`MetricOmmPool.swap` passes `msg.sender` as the `sender` argument to `_beforeSwap`, which forwards it verbatim to every configured extension. When the caller is `MetricOmmSimpleRouter`, `sender` is the router's address, not the originating EOA. `SwapAllowlistExtension.beforeSwap` performs:

```solidity
if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
    revert IMetricOmmPoolActions.NotAllowedToSwap();
}
```

`msg.sender` here is the pool; `sender` is the router. If `allowedSwapper[pool][router] == true`, the condition is false and the revert is skipped for every caller of the router, including addresses that were never individually allowlisted.

### Impact Explanation
A pool configured with `SwapAllowlistExtension` to restrict trading to a curated set of addresses (e.g., KYC'd counterparties, protocol-owned addresses, or whitelisted market makers) can be freely traded against by any public user via the router. LP balances are exposed to unrestricted swap flow, defeating the access control the pool admin intended to enforce.

### Likelihood Explanation
The `MetricOmmSimpleRouter` is a public, permissionless periphery contract. A pool admin who wants to permit router-mediated swaps (for slippage protection, multi-hop routing, or UX reasons) would naturally call `setAllowedToSwap(pool, router, true)`. The extension's NatSpec ("Gates `swap` by swapper address") gives no indication that allowlisting the router opens the gate to all callers. The protocol's own audit target document explicitly identifies this exact path as a validation concern.

### Recommendation
Pass the original economic actor through the call chain. One approach: add an `originalSender` field to the swap callback data that the router populates with `msg.sender` before calling the pool, and have the pool forward it to extensions as a separate parameter. Alternatively, `SwapAllowlistExtension` could maintain a separate set of "trusted forwarders" and, when `sender` is a forwarder, require the extension data to carry a signed or ABI-encoded original-caller address that the extension verifies.

### Proof of Concept
1. Deploy a pool with `SwapAllowlistExtension` in `beforeSwap` order.
2. Call `swapExtension.setAllowedToSwap(pool, address(router), true)` — allowlist only the router.
3. As an address that is **not** individually allowlisted, call `router.exactInputSingle(...)` targeting that pool.
4. Assert the swap succeeds (no `NotAllowedToSwap` revert) and tokens are transferred.
5. As the same non-allowlisted address, call `pool.swap(...)` directly (bypassing the router) and assert it **reverts** with `NotAllowedToSwap` — confirming the router is the bypass vector.
