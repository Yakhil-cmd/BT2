### Title
Malicious Participant Can Corrupt CKD Output by Sending Unverified Contribution — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the CKD coordinator path, the coordinator accumulates `(big_y, big_c)` contributions from every participant with no proof of correctness. Any single malicious participant can substitute arbitrary group elements, causing the coordinator to produce a structurally valid but cryptographically wrong `CKDOutput`. The client's `unmask` call then yields a garbage confidential key.

---

### Finding Description

`do_ckd_coordinator` collects one `CKDOutput` per participant and blindly adds the received elements together:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Two problems are present simultaneously:

1. **The sender identity is discarded** (`_`). The coordinator does not record which participant contributed which `(big_y, big_c)` pair, so it cannot later audit or attribute a bad value.

2. **No proof of correctness is required.** A correctly-behaving participant computes:

   ```
   big_y  = y · G
   big_c  = x_i · H(pk ‖ app_id) + y · app_pk
   ``` [2](#0-1) 

   Nothing in the protocol forces the participant to use their actual `x_i` or a consistent `y`. The coordinator has no way to distinguish a correct share from an arbitrary pair of group elements.

This is the direct analog of the esFLUO `deposit` bug: just as anyone could call `deposit(victim, 0)` to corrupt the victim's vesting state without depositing anything, any enrolled participant can send `(G, G)` (or any other arbitrary pair) to the coordinator to corrupt the accumulated CKD state without contributing a valid share.

The `compute_signature_share` function that honest participants call is never enforced on the receiving side: [3](#0-2) 

The public entry point `ckd` performs participant-list and coordinator-membership checks but adds no constraint on the content of what participants will later send: [4](#0-3) 

---

### Impact Explanation

The coordinator's final output is:

```
Y = Σ λ_i · y_i · G
C = msk · H(pk ‖ app_id) + app_pk · Y
```

If participant `j` sends `(big_y', big_c')` instead of its correct share, the sums become:

```
Y'  = Y_honest + big_y'   (wrong)
C'  = C_honest + big_c'   (wrong)
```

The client's `unmask(a)` computes `C' − a · Y'`, which no longer equals `msk · H(pk ‖ app_id)`. The derived confidential key is wrong and unusable. Because the coordinator returns a `CKDOutput` that is structurally valid (two `G1` points), honest parties have no way to detect the corruption.

**Allowed impact matched:** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.* [5](#0-4) 

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. No special privilege is required beyond being enrolled in the `participants` list. The attack requires only that the malicious participant deviate from `compute_signature_share` and send arbitrary `G1` elements. The coordinator's loop processes every received value unconditionally, so the attack succeeds on the first malicious message.

---

### Recommendation

Require each participant to attach a non-interactive zero-knowledge proof (e.g., a Schnorr-style sigma proof) demonstrating that:

- `big_y = y · G` for some known `y`, and
- `big_c = x_i · H(pk ‖ app_id) + y · app_pk` for the same `y` and the participant's committed `x_i`.

The coordinator must verify this proof before adding the contribution to the running sum. This mirrors the esFLUO remediation of adding `require(user == msg.sender && amount > 0)` — the coordinator must validate both *who* is contributing and *that the contribution is well-formed*.

---

### Proof of Concept

1. Participants A (honest), B (malicious), C (coordinator/honest) run `ckd` for the same `app_id` and `app_pk`.
2. A computes and sends its correct `(norm_big_y_A, norm_big_c_A)` to C.
3. B, instead of calling `compute_signature_share`, sends `(G1::generator(), G1::generator())` to C.
4. C's loop at lines 50–55 adds both contributions without any check.
5. The resulting `CKDOutput` satisfies `big_c ≠ msk · H(pk ‖ app_id) + app_pk · big_y`.
6. The client calls `ckd_output.unmask(app_sk)` and receives a random group element instead of the expected confidential key, permanently breaking the derived key for this `app_id`. [6](#0-5)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
