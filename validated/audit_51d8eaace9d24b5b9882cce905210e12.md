### Title
Duplicate Authorizer Addresses Bypass Multi-Account Authorization Requirements - (`fvm/transactionVerifier.go`)

### Summary
Neither the Access/Collection node pre-validation nor the FVM's on-chain transaction verifier checks for duplicate addresses in `tx.Authorizers`. A sender who controls a single account can list that account's address twice (or more) in the `Authorizers` field, satisfying the Cadence runtime's authorizer-count check for a `prepare` function that expects multiple distinct signers, while only providing one account's cryptographic signature.

### Finding Description
`flow.TransactionBody.Authorizers` is a plain `[]flow.Address` slice. No deduplication or uniqueness enforcement is applied to it at any validation layer.

**Layer 1 – Access/Collection node pre-validation (`access/validator/validator.go`)**

`checkAddresses` iterates over `tx.Authorizers` and validates each address against the chain's address generator, but performs no uniqueness check:

```go
func (v *TransactionValidator) checkAddresses(tx *flow.TransactionBody) error {
    for _, address := range append(tx.Authorizers, tx.Payer) {
        if !v.chain.IsValid(address) {
            return InvalidAddressError{Address: address}
        }
    }
    return nil
}
```

`checkAccounts` checks that each authorizer address has at least one signature, but again iterates the raw (potentially duplicate) slice without deduplication:

```go
for _, authorizer := range tx.Authorizers {
    if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
        continue
    }
    if !observedEnvelopeSig[authorizer] && !observedPayloadSig[authorizer] {
        return MissingSignatureError{...}
    }
}
```

Because the check is a map lookup (`observedPayloadSig[authorizer]`), a duplicate entry for an already-signed address passes silently on every subsequent iteration.

**Layer 2 – FVM on-chain verifier (`fvm/transactionVerifier.go`)**

`verifyTransaction` iterates `tx.Authorizers` to enforce the key-weight threshold, again without deduplication:

```go
for _, addr := range tx.Authorizers {
    if addr == tx.Payer {
        continue
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
```

A duplicate address passes both iterations because `payloadWeights[addr]` is the same value each time.

**Layer 3 – Cadence runtime execution**

The Cadence runtime receives `len(tx.Authorizers)` account references and passes them positionally to the `prepare` function. With `Authorizers = [addrA, addrA]`, the runtime delivers two `&Account` references both pointing to `addrA`. A script declaring `prepare(signerA: &Account, signerB: &Account)` executes successfully with only `addrA`'s single signature, even though the script's interface implies two distinct accounts must authorize it.

### Impact Explanation
Any Cadence transaction or contract interaction that relies on the presence of two (or more) distinct `&Account` references in `prepare` as a multi-party authorization gate can be bypassed. An attacker who controls a single account can:

- Satisfy a 2-of-2 (or N-of-N) multi-account authorization scheme using only their own account.
- Drain or mutate resources in contracts that gate sensitive operations on receiving distinct account capabilities from multiple parties.
- Forge the appearance of multi-party consent in on-chain governance or escrow transactions.

This is a direct unauthorized account mutation / authorization bypass with potential for asset loss.

### Likelihood Explanation
The attack requires only the ability to submit a transaction — the lowest possible privilege level on Flow. No special role, leaked key, or social engineering is needed. The malformed `Authorizers` slice is trivially constructed by any SDK user. The absence of a uniqueness check at both the pre-validation and FVM layers means the transaction reaches execution without any rejection.

### Recommendation
Add a uniqueness check on `tx.Authorizers` in both validation layers:

1. In `access/validator/validator.go` `checkAccounts` (or a dedicated `checkAuthorizers` step), reject any transaction whose `Authorizers` slice contains a repeated address.
2. In `fvm/transactionVerifier.go` `verifyTransaction`, enforce the same uniqueness invariant before the weight-check loop.

A simple set-based deduplication (analogous to the existing `duplicate` map used for signatures in `newSignatureEntries`) is sufficient.

### Proof of Concept
```
tx.Authorizers = [addrA, addrA]
tx.PayloadSignatures = [sig_from_addrA]   // single valid signature
tx.Script = `
  transaction {
    prepare(signerA: &Account, signerB: &Account) {
      // signerA == signerB == addrA
      // operations that appear to require two distinct accounts proceed
    }
  }
`
```

1. `checkAddresses`: both entries are valid addresses — passes. [1](#0-0) 
2. `checkAccounts`: `observedPayloadSig[addrA]` is `true` for both iterations — passes. [2](#0-1) 
3. `newSignatureEntries`: builds `transactionAddresses` as a map, so `addrA` is inserted once; no duplicate-authorizer error is raised. [3](#0-2) 
4. `verifyTransaction` weight loop: `payloadWeights[addrA] >= threshold` is true on both iterations — passes. [4](#0-3) 
5. Cadence runtime receives `len([addrA, addrA]) == 2` account references, matching the `prepare(signerA, signerB)` parameter count — transaction executes with only one account's authorization. [5](#0-4)
