### Title
Gateway Skips `__validate__` for Nonce-1 Invoke Transactions from Already-Deployed Accounts — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the Apollo gateway stateful validator unconditionally skips the `__validate__` entry-point call for any Invoke transaction with `tx_nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The guard is designed for the deploy-account + invoke UX shortcut (account not yet on-chain), but it fires equally for an **already-deployed** account whose nonce is still 0 and that has any prior transaction in the mempool. An attacker who controls such an account can inject a nonce-1 invoke with an **invalid or unauthorized signature** that the gateway accepts without ever calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` is called from `run_pre_validation_checks` after the nonce and resource-bound checks pass:

```
validate_state_preconditions → validate_by_mempool → skip_stateful_validations
``` [1](#0-0) 

The function returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [2](#0-1) 

The inline comment claims this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." That reasoning is **incorrect**: an already-deployed account with on-chain nonce 0 can have a nonce-0 invoke in the mempool (which passed `__validate__` normally). Once that nonce-0 invoke is pooled, `account_tx_in_pool_or_recent_block` returns `true` for the same address. [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate = false`: [4](#0-3) 

`StatefulValidator::perform_validations` then returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling `__validate__`: [5](#0-4) 

The transaction is forwarded to the mempool with an invalid or unauthorized signature.

---

### Impact Explanation

The gateway/mempool admission layer accepts an Invoke transaction whose `__validate__` entry point was never executed. This violates the invariant that every accepted transaction has been authorized by the account's own validation logic. The corrupted admission decision is: **an invalid (unauthorized) transaction is accepted into the mempool**.

When the batcher later sequences the transaction, `AccountTransaction::new_for_sequencing` sets `validate: true`: [6](#0-5) 

`__validate__` then runs and the transaction reverts. The reverted transaction is included in a block and the sender is charged fees. The primary impact is **gateway/mempool admission of invalid transactions**, matching the High-impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Secondary effects include mempool pollution, wasted block space, and potential DoS amplification if the attacker submits many such transactions.

---

### Likelihood Explanation

The preconditions are reachable by any unprivileged user:

1. Deploy an account (nonce becomes 0 on-chain).
2. Submit a valid nonce-0 invoke (passes `__validate__` normally, enters mempool).
3. Submit a nonce-1 invoke with an invalid or unauthorized signature.

Step 2 requires a valid signature for nonce-0, which the account owner can always produce. Step 3 is then trivially achievable. The attack is most impactful against accounts with custom `__validate__` logic (multisig, guardian-locked, policy-enforcing accounts) where the attacker controls one key but not the full authorization set.

---

### Recommendation

`skip_stateful_validations` must verify that the account has **no deployed code** at its address before skipping `__validate__`. The correct invariant is: skip only when the account does not yet exist on-chain (i.e., `get_class_hash_at(sender) == ClassHash::default()`). Alternatively, the mempool check should be restricted to confirming the presence of a **deploy-account** transaction specifically, not any transaction.

The `max_nonce_for_validation_skip` field in `StatefulTransactionValidatorConfig` is defined but unused in the production `skip_stateful_validations` path (it is only consumed by the legacy `native_blockifier` Python bindings): [7](#0-6) 

This field should either be wired into the production path or removed to avoid confusion.

---

### Proof of Concept

```
1. Deploy account A (class with custom __validate__ requiring 2-of-3 multisig).
   → On-chain nonce for A = 0.

2. Submit Invoke(sender=A, nonce=0, calldata=<benign>, signature=<valid 2-of-3>).
   → validate_nonce: 0 <= 0 <= 200 ✓
   → run_validate_entry_point(skip_validate=false): __validate__ called, passes ✓
   → Transaction enters mempool.
   → account_tx_in_pool_or_recent_block(A) now returns true.

3. Submit Invoke(sender=A, nonce=1, calldata=<high-value transfer>, signature=<single-sig, invalid for 2-of-3>).
   → validate_nonce: 0 <= 1 <= 200 ✓
   → skip_stateful_validations:
       tx.nonce()==1 ✓, account_nonce==0 ✓, account_tx_in_pool==true ✓
       → returns true (skip)
   → run_validate_entry_point(skip_validate=true): __validate__ NOT called ✓
   → Invalid transaction accepted into mempool.

4. Batcher sequences the nonce-1 invoke with validate=true.
   → __validate__ runs, fails (invalid signature).
   → Transaction reverts; included in block; fees charged.
   → Gateway has admitted an unauthorized transaction.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
