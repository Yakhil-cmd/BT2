### Title
No Setter for `starkgate_address` in `RewardSupplier` Permanently Freezes Unclaimed Yield on StarkGate Migration — (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `RewardSupplier` contract stores the StarkGate bridge address (`starkgate_address`) and the STRK token address (hardcoded constant `STRK_TOKEN_ADDRESS`) at construction time with **no admin setter functions**. If StarkGate migrates to a new contract address — a realistic scenario given StarkGate's documented upgrade history — the `on_receive` callback will permanently revert for all incoming L1→L2 reward transfers, freezing all unclaimed staking yield for every staker and pool member.

---

### Finding Description

The `RewardSupplier` constructor sets `starkgate_address` once and hardcodes the STRK token address from a constant:

```cairo
pub fn constructor(...) {
    let token_address = STRK_TOKEN_ADDRESS;   // hardcoded constant, not a parameter
    ...
    self.token_dispatcher.contract_address.write(token_address);
    self.starkgate_address.write(starkgate_address);
    self.l1_reward_supplier.write(l1_reward_supplier);
``` [1](#0-0) 

The only config setter exposed by the contract is `set_block_duration_config` (restricted to `app_governor`). There is no `set_starkgate_address`, `set_l1_reward_supplier`, or `set_token` function anywhere in the contract or its interfaces. [2](#0-1) [3](#0-2) 

The `on_receive` function — the sole entry point for L1→L2 reward token delivery — enforces both the stale bridge address and the stale token address:

```cairo
fn on_receive(...) -> bool {
    assert!(
        get_caller_address() == self.starkgate_address.read(),
        "{}", Error::ON_RECEIVE_NOT_FROM_STARKGATE,
    );
    assert!(
        l2_token == self.token_dispatcher.contract_address.read(),
        "{}", Error::UNEXPECTED_TOKEN,
    );
``` [4](#0-3) 

If StarkGate migrates to a new contract address, the new bridge calls `on_receive` from the new address. Both assertions fail permanently. No STRK ever arrives at the `RewardSupplier`. Meanwhile, `update_unclaimed_rewards_from_staking_contract` keeps incrementing `unclaimed_rewards` and `l1_pending_requested_amount` grows unboundedly as mint requests are sent to L1 that can never be fulfilled: [5](#0-4) 

Eventually `unclaimed_rewards` exceeds the contract's actual STRK balance, causing `claim_rewards` to revert on the `amount <= unclaimed_rewards` check — permanently blocking every staker and pool member from claiming their accrued rewards: [6](#0-5) 

The only recovery path is a full contract upgrade via `ReplaceabilityComponent` (requiring governance action and EIC initialization), analogous to the "painful shutdown" described in M-20.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once StarkGate migrates, every call to `claim_rewards` from the staking contract will revert (balance < `unclaimed_rewards`). All stakers and pool members lose access to their accrued STRK rewards until a governance-gated contract upgrade is executed. The funds are not stolen — they are frozen in the pipeline.

---

### Likelihood Explanation

**Medium.** StarkGate has been upgraded on Starknet before. The Starknet ecosystem explicitly supports bridge upgrades. The `RewardSupplier` has already been upgraded three times (V1→V2→V3), demonstrating that the team expects to upgrade contracts — but the `starkgate_address` field has no corresponding setter, meaning any StarkGate migration requires a full `RewardSupplier` upgrade rather than a simple admin call. The window between StarkGate migration and `RewardSupplier` upgrade is the freeze period.

---

### Recommendation

Add privileged setter functions (restricted to `token_admin` or `app_governor`) for `starkgate_address` and `l1_reward_supplier` in `RewardSupplier`, analogous to how `set_reward_supplier` exists in the staking contract: [7](#0-6) 

```cairo
fn set_starkgate_address(ref self: ContractState, starkgate_address: ContractAddress) {
    self.roles.only_token_admin();
    self.starkgate_address.write(starkgate_address);
    // emit event
}

fn set_l1_reward_supplier(ref self: ContractState, l1_reward_supplier: felt252) {
    self.roles.only_token_admin();
    self.l1_reward_supplier.write(l1_reward_supplier);
    // emit event
}
```

---

### Proof of Concept

1. System is deployed normally; `starkgate_address` is set to `OLD_STARKGATE`.
2. Starknet governance upgrades StarkGate; the new bridge contract is at `NEW_STARKGATE`.
3. The L1 `RewardSupplier` sends STRK via the new bridge.
4. `NEW_STARKGATE` calls `RewardSupplier.on_receive(l2_token: STRK_TOKEN_ADDRESS, amount: X, ...)`.
5. `on_receive` asserts `get_caller_address() == self.starkgate_address.read()` → `NEW_STARKGATE != OLD_STARKGATE` → **REVERT** with `ON_RECEIVE_NOT_FROM_STARKGATE`.
6. No STRK arrives at `RewardSupplier`; `l1_pending_requested_amount` grows with each epoch as new mint requests are sent.
7. `unclaimed_rewards` accumulates past the contract's actual STRK balance.
8. Any staker or pool member calling `claim_rewards` (via the staking contract) triggers `assert!(amount <= unclaimed_rewards)` — which passes — but the subsequent `checked_transfer` fails due to insufficient balance.
9. All staking rewards are frozen until a governance-gated `RewardSupplier` upgrade is executed.

        /// Requests funds from L1 to account for new rewards, if the contract's balance is too low.
        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
