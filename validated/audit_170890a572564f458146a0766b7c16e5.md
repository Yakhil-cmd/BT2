### Title
Unbounded NFT `royalty_percentage` from on-chain data inflates XCH spending estimate, falsely triggering `conflictsWithNewOffer` and blocking offer creation unless user cancels legitimate open offers — (`packages/gui/src/util/offerBuilderDataToOffer.ts`)

### Summary

In `prepareNftTasks` inside `offerBuilderDataToOffer`, the `royalty_percentage` field read from an attacker-controlled NFT's driver data is used directly in a multiplier calculation with no bounds check. An attacker who mints an NFT with `royalty_percentage = 10000` (100%, the on-chain maximum) causes the multiplier to reach 2, doubling the computed XCH spending amount. This inflated figure is compared against the user's spendable balance, incorrectly setting `pendingXchOffer.status = 'conflictsWithNewOffer'`. The resulting dialog disables the "Proceed" button until the user cancels their legitimate open offers, or aborts offer creation entirely.

### Finding Description

**Root cause — no bounds check on `royaltyPercentageStr`:** [1](#0-0) 

```
royaltyPercentageStr = driver.also.also?.transfer_program.royalty_percentage
royaltyMultiplier    = 1 + 10000 / 10_000  →  2
newSpendingXch       = spendingXch × 2 + fee
```

`pendingXchOffer.spendingAmount` is then compared against the real spendable balance. If the user's spendable balance is, say, 1 XCH and they are offering 0.6 XCH, the inflated spending amount becomes 1.2 XCH, which exceeds the spendable balance, so `status` is set to `'conflictsWithNewOffer'`.

**`considerNftRoyalty: true` is hardcoded at both call sites:** [2](#0-1) [3](#0-2) 

**The conflict dialog's "Proceed" button is disabled while conflicting offers remain:** [4](#0-3) 

The dialog message tells the user: *"You need to cancel open offers to unlock coins/asset for the new offer."* This statement is false — the real balance is sufficient. The user cannot click "Proceed" without first canceling their legitimate open offers. [5](#0-4) 

**Contrast with the old NFT offer editor**, which does have a royalty bounds check: [6](#0-5) 

The new `offerBuilderDataToOffer` path has no equivalent guard.

### Impact Explanation

This is **unsafe trust of NFT metadata** (on-chain `royalty_percentage`) that causes the user to be presented with a false conflict state. The "Proceed" button is programmatically disabled, so the user's only options are:

1. Cancel their legitimate open XCH offers (revoking valid offers they did not intend to cancel), or
2. Close the dialog, which aborts offer creation entirely.

This fits the High impact category: *"unsafe trust of … NFT metadata … that causes a user to … revoke … the wrong asset … or status."*

### Likelihood Explanation

- Attacker mints one NFT with `royalty_percentage = 10000` — a valid on-chain operation.
- Attacker shares the NFT ID (e.g., via a marketplace listing, a notification, or a WalletConnect payload).
- User adds the NFT to the "requested" side of an offer while also offering XCH.
- User has at least one open XCH offer in `PENDING_ACCEPT` state — a common condition for active traders.
- No special privileges, no leaked keys, no local compromise required.

### Recommendation

Add a bounds check before using `royaltyPercentageStr` in the multiplier calculation:

```typescript
const royaltyPercentageNum = +royaltyPercentageStr;
if (royaltyPercentageNum > 0 && royaltyPercentageNum <= 10_000) {
  const royaltyMultiplier = 1 + royaltyPercentageNum / 10_000;
  // ... rest of calculation
}
```

Alternatively, clamp the value: `Math.min(Math.max(+royaltyPercentageStr, 0), 10_000)`. This mirrors the guard already present in `NFTOfferEditor.tsx`.

### Proof of Concept

Unit-test `offerBuilderDataToOffer` with a mocked `prepareNFTOfferFromNFTId` that returns a driver with `royalty_percentage = '10000'`, a user offering 0.6 XCH, spendable balance of 1 XCH, and one open XCH offer locking 0.5 XCH. Assert that `assetsToUnlock[0].status` is **not** `'conflictsWithNewOffer'` — without the fix, it will be.
