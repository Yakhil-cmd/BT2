### Title
Missing Lower-Bound Validation on `threshold` in `assert_sign_inputs` Allows Corrupted FROST Signing Outputs - (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that `threshold <= participants.len()` but omits the symmetric lower-bound check `threshold >= 2`. This is directly inconsistent with `assert_key_invariants` in `src/dkg.rs`, which enforces both bounds. Any caller of the FROST signing API (`sign_v1`/`sign_v2` for EdDSA, `sign` for RedJubjub) can supply `threshold = 1` (or `threshold = 0`) and the library will silently proceed, producing a cryptographically invalid signature.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces two bounds on `threshold`:

```
if threshold > participants.len() { ... ThresholdTooLarge ... }
if threshold < 2                  { ... ThresholdTooSmall ... }
``` [1](#0-0) 

`validate_triple_inputs` in the OT-based triple generation enforces the same pair of bounds: [2](#0-1) 

`assert_sign_inputs`, the single validation gate for all FROST signing entry-points, only enforces the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check for threshold < 2
``` [3](#0-2) 

This function is the sole validation path for:
- `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs` [4](#0-3) 
- `sign` in `src/frost/redjubjub/sign.rs` [5](#0-4) 

When `threshold = 1` is accepted, the coordinator collects exactly one signature share and performs Lagrange interpolation over a single point. The Lagrange coefficient for a lone participant is 1, so the aggregated value is simply that participant's raw share evaluation — not the correct reconstruction of the secret-key contribution. The resulting `(R, z)` pair will fail standard EdDSA/RedJubjub verification against the public key that was produced by DKG with `threshold ≥ 2`.

When `threshold = 0` is accepted, the coordinator collects zero shares and attempts to aggregate an empty set, which either panics or produces a zero/identity output — both are unusable.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties receive unusable cryptographic outputs.**

The signing protocol completes without returning an `InitializationError` or `ProtocolError`, so callers have no indication that anything went wrong. The produced signature is structurally well-formed but cryptographically invalid: it will fail verification against the correct public key. Any downstream system that trusts the library's error-free completion to mean "the signature is valid" will silently accept an unusable output.

---

### Likelihood Explanation

Low-to-medium. The threshold value is a plain caller-supplied parameter with no type-level enforcement of the `≥ 2` invariant (`ReconstructionLowerBound` is a transparent `usize` wrapper with no constructor guard). [6](#0-5) 

A misconfigured orchestrator, a coordinator that derives the signing threshold from an external configuration without re-validating it, or a library integrator who reads the DKG threshold as `1` from a serialized state and passes it directly to signing will all silently trigger this path. The DKG spec itself documents `1 < threshold < N` as a hard requirement (Step 1.1), making the omission in signing a clear inconsistency that an integrator would not expect. [7](#0-6) 

---

### Recommendation

Add the symmetric lower-bound check inside `assert_sign_inputs`, mirroring `assert_key_invariants`:

```rust
// validate threshold lower bound (mirrors assert_key_invariants)
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Place this check immediately before or after the existing upper-bound check at `src/frost/mod.rs` lines 144–150. [8](#0-7) 

---

### Proof of Concept

1. Run DKG for 3 participants with `threshold = 2` (passes `assert_key_invariants`).
2. Call `sign_v1` with the same 3 participants but `threshold = 1`.
3. `assert_sign_inputs` checks `1 > 3` → false; no error is returned.
4. The coordinator collects 1 signature share; Lagrange coefficient = 1; aggregated `z = nonce_share_i + key_share_i * challenge`.
5. Verify the resulting signature against the public key produced in step 1 → **verification fails**.
6. The library returned `Ok(protocol)` and the protocol completed without error, giving the caller no signal that the output is invalid.

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
