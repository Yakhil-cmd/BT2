### Title
Unprivileged Ethereum L2 Transaction Triggers `TxError::Internal` via `max_fee_per_gas × tx_gas_limit` U256 Overflow, Halting Block Execution — (`basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

---

### Summary

In `validate_and_compute_fee_for_transaction`, two overflow checks on the product of a user-supplied `max_fee_per_gas` (U256) and `tx_gas_limit` (u64) return `internal_error!(...)`, which converts to `TxError::Internal`. The block-processing loop in `generic_loop_op` treats `TxError::Internal` as a fatal block-halting error, while `TxError::Validation` is per-transaction and non-fatal. Because no prior guard caps `max_fee_per_gas` to a range that prevents U256 overflow, any unprivileged sender can craft a transaction that halts the block.

---

### Finding Description

**Overflow site 1** — `validation_impl.rs` lines 355–358:

```rust
let worst_case_fee_amount = {
    let (value, of) = u256_mul_by_word(transaction.max_fee_per_gas(), tx_gas_limit);
    if of > 0 {
        return Err(internal_error!("max gas price by tx gas limit").into());
    }
    value
};
``` [1](#0-0) 

**Overflow site 2** — `validation_impl.rs` lines 402–409:

```rust
let fee_amount_execution_gas = {
    let (value, of) = u256_mul_by_word(&effective_gas_price, tx_gas_limit);
    if of > 0 {
        return Err(internal_error!("effective gas price by tx gas limit").into());
    }
    value
};
``` [2](#0-1) 

`u256_mul_by_word` multiplies a U256 by a u64 and returns the carry word as overflow indicator:

```rust
pub fn u256_mul_by_word(input: &U256, word: u64) -> (U256, u64) {
    let mut result = *input;
    let of = unsafe { ruint::algorithms::mul_nx1(result.as_limbs_mut(), word) };
    (result, of)
}
``` [3](#0-2) 

`internal_error!` is explicitly documented as "should not be triggered by user input": [4](#0-3) 

The `From<InternalError> for TxError` impl wraps it as `TxError::Internal`: [5](#0-4) 

**Block-halting consequence** — in `generic_loop_op`, `TxError::Internal` causes an immediate `return Err(err)` that exits the entire transaction loop, while `TxError::Validation` merely records the rejection and continues:

```rust
match tx_result {
    Err(TxError::Internal(err)) => {
        return Err(err);          // ← halts the block
    }
    Err(TxError::Validation(err)) => {
        result_keeper.tx_processed(Err(err)); // ← per-tx rejection, loop continues
    }
    ...
}
``` [6](#0-5) 

**No prior guard prevents the overflow.** The only pre-checks on `tx_gas_limit` before line 355 are:

1. `tx_gas_limit.saturating_mul(ERGS_PER_GAS) < u64::MAX` — constrains ergs, not the fee product.
2. `tx_gas_limit <= individual_tx_gas_limit()` — caps gas limit to a block-level value (e.g., ~30 M gas), but does not constrain `max_fee_per_gas`. [7](#0-6) 

`max_fee_per_gas` is accepted as a raw U256 with no upper-bound validation — only relational checks (`>= base_fee`, `>= max_priority_fee_per_gas`): [8](#0-7) 

The balance check that would naturally reject an absurdly large fee is at lines 392–399, **after** the overflow check, so it is never reached: [9](#0-8) 

---

### Impact Explanation

A single crafted transaction halts the block-processing loop via `return Err(err)` in `generic_loop_op`. All transactions queued after the malicious one in the same block become unprocessable. The sequencer/prover receives a fatal `BootloaderSubsystemError` instead of a per-transaction rejection, violating the invariant that user-supplied field overflows during validation must produce per-tx rejection errors.

---

### Likelihood Explanation

The attack requires only a valid Ethereum keypair (zero cost) and knowledge of the overflow threshold. With a gas limit of 21 000, any `max_fee_per_gas > 2^256 / 21000 ≈ 2^241.6` triggers the overflow. This is a trivially constructable U256 value. No privileged access, governance, or oracle manipulation is needed. The attacker does not even need a balance — the balance check is bypassed because it executes after the overflow check.

---

### Recommendation

Replace both `internal_error!` returns with the existing `InvalidTransaction::OverflowPaymentInTransaction` validation variant, which already exists in the error enum: [10](#0-9) 

```rust
// line 357-358 — change to:
return Err(TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction));

// line 404-405 — change to:
return Err(TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction));
```

Optionally, add an explicit upper-bound check on `max_fee_per_gas` early in validation (before any arithmetic) to reject transactions whose fee fields cannot possibly produce a valid product.

---

### Proof of Concept

```rust
// Differential test: both Ethereum and ZK paths must return TxError::Validation,
// never TxError::Internal, for overflow inputs.

let max_fee_per_gas = U256::from(1u64) << 242; // > 2^256 / 21000
let tx_gas_limit: u64 = 21_000;

// Verify overflow is triggered
let (_, of) = u256_mul_by_word(&max_fee_per_gas, tx_gas_limit);
assert!(of > 0, "overflow must occur");

// Submit transaction with this max_fee_per_gas through the Ethereum L2 path.
// Expected: TxError::Validation(OverflowPaymentInTransaction)
// Actual:   TxError::Internal(...) → block halt
let result = validate_and_compute_fee_for_transaction(system, &mut tx, &mut tracer);
match result {
    Err(TxError::Validation(_)) => { /* correct */ }
    Err(TxError::Internal(_))   => panic!("BUG: internal error from user input"),
    Ok(_)                       => panic!("should have been rejected"),
}
```

fn get_gas_prices<S: EthereumLikeTypes>(

    max_fee_per_gas: &U256,

) -> Result<(U256, U256), TxError> {
    let max_priority_fee_per_gas = if let Some(max_priority_fee_per_gas) = max_priority_fee_per_gas
    {
        max_priority_fee_per_gas
    } else {
        max_fee_per_gas
    };
    require!(
        max_priority_fee_per_gas <= max_fee_per_gas,
        TxError::Validation(InvalidTransaction::PriorityFeeGreaterThanMaxFee,),
        system
    )?;

    let base_fee = system.get_eip1559_basefee();
    let (max_fee_minus_base_fee, uf) = max_fee_per_gas.overflowing_sub(base_fee);
    require!(
        uf == false,
        TxError::Validation(InvalidTransaction::BaseFeeGreaterThanMaxFee,),
        system
    )?;

    let priority_fee_per_gas = core::cmp::min(*max_priority_fee_per_gas, max_fee_minus_base_fee);

    let effective_gas_price = base_fee + priority_fee_per_gas;

    Ok((effective_gas_price, priority_fee_per_gas))
}
