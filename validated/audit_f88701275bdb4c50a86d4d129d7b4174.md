### Title
COA Ownership Proof Signature Replay Across Multiple COAs — (`fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A cryptographic signature produced by a Cadence account holder for one COA (or for any arbitrary purpose) can be replayed by an attacker to pass ownership verification for a *different* COA belonging to the same Cadence account. This breaks the authentication guarantee of the ERC-1271 `isValidSignature` interface implemented by every COA contract.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. It verifies that the supplied `signatures` are cryptographically valid over `signedData` using the account's keys.
2. It verifies that the COA resource stored at `CapabilityPath` has an EVM address equal to the caller-supplied `evmAddress`. [1](#0-0) 

Critically, it never checks that `signedData` encodes `evmAddress`. The two checks are entirely independent. The code itself acknowledges this gap: [2](#0-1) 

Because `signedData` is free-form and not bound to any specific COA address, a signature Alice produced over hash `H` for COA-A is equally valid as a proof of ownership for COA-B — the attacker only needs to swap the `CapabilityPath` field in the RLP-encoded `COAOwnershipProof` to point to COA-B's public path.

The `COAOwnershipProof` struct is fully attacker-controlled: it is RLP-decoded from the `_sig` bytes passed to `isValidSignature` in `coa.sol`, and all four fields (`KeyIndices`, `Address`, `CapabilityPath`, `Signatures`) are supplied by the caller. [3](#0-2) 

The EVM entry point is `isValidSignature` in the COA contract, which calls the Cadence Arch precompile with `address(this)` as the `evmAddress`: [4](#0-3) 

The precompile then invokes `validateCOAOwnershipProof` via `coaOwnershipProofValidator`: [5](#0-4) 

### Impact Explanation

Any EVM DApp that uses ERC-1271 (`isValidSignature`) to authenticate a COA owner and gate privileged actions (token claims, withdrawals, governance votes, permit-style approvals) is vulnerable. An attacker who obtains a valid `COAOwnershipProof` that Alice produced for COA-A can construct a modified proof pointing to COA-B and pass it to `COA-B.isValidSignature(H, modified_proof)`. The call returns `ValidERC1271Signature (0x1626ba7e)`, granting the attacker Alice's authenticated identity for COA-B without Alice's knowledge or consent. This constitutes unauthorized account mutation and can lead to direct asset theft on any EVM protocol that trusts ERC-1271 responses.

### Likelihood Explanation

The preconditions are realistic:
- A Cadence account can hold multiple COAs; the protocol does not restrict this.
- `COAOwnershipProof` bytes are transmitted off-chain (e.g., in wallet authentication flows, API calls, permit messages) and are observable by a network-level attacker or a malicious DApp.
- The attacker only needs to modify the `CapabilityPath` field in the RLP-encoded proof — no cryptographic material needs to be forged.
- `validateCOAOwnershipProof` is also a public Cadence function callable directly from any script or transaction, widening the attack surface beyond the EVM precompile path. [6](#0-5) 

### Recommendation

Bind `signedData` to the target `evmAddress` inside `validateCOAOwnershipProof`. The function should verify that `signedData` contains (or commits to) the 20-byte `evmAddress` before accepting the proof as valid. Concretely:

- Define a canonical signed payload: `signedData = keccak256(abi.encode(evmAddress, applicationData))` or equivalent.
- Reject any proof where `signedData` does not encode `evmAddress`.
- Alternatively, include `evmAddress` as a domain-separation prefix when constructing the hasher, analogous to how Flow transactions use `ChainID` to prevent cross-chain replay. [1](#0-0) 

### Proof of Concept

```
Setup:
  Alice's Cadence account: 0xALICE
  COA-A stored at /public/coa-a  →  EVM address: 0xCOA_A
  COA-B stored at /public/coa-b  →  EVM address: 0xCOA_B

Step 1 – Alice legitimately authenticates with a DApp using COA-A:
  proof_A = COAOwnershipProof{
      KeyIndices:     [0],
      Address:        0xALICE,
      CapabilityPath: "coa-a",
      Signatures:     [sign(H, alice_key_0)],
  }
  COA_A.isValidSignature(H, encode(proof_A))  →  0x1626ba7e ✓

Step 2 – Attacker intercepts proof_A (e.g., from network traffic or a malicious DApp).

Step 3 – Attacker constructs modified proof pointing to COA-B:
  proof_B = COAOwnershipProof{
      KeyIndices:     [0],
      Address:        0xALICE,
      CapabilityPath: "coa-b",       // ← only field changed
      Signatures:     [sign(H, alice_key_0)],  // ← same signature
  }

Step 4 – Attacker calls:
  COA_B.isValidSignature(H, encode(proof_B))

Step 5 – validateCOAOwnershipProof(0xCOA_B, H, proof_B):
  • keyList.verify(signatureSet, signedData=H, ...)  →  true  (Alice signed H)
  • acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coa-b)
      .address().bytes == 0xCOA_B  →  true
  • Returns ValidationResult(isValid: true)

Result: COA_B.isValidSignature returns 0x1626ba7e — attacker is authenticated
        as Alice for COA-B without Alice's consent.
``` [7](#0-6)
