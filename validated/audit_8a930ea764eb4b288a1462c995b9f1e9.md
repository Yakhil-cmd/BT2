### Title
Missing Lower-Bound Sanity Check on `threshold` in FROST `assert_sign_inputs` Allows Degenerate Signing - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that `threshold <= participants.len()` but omits the symmetric lower-bound check `threshold >= 2`. The keygen path (`assert_key_invariants` in `src/dkg.rs`) enforces both bounds. A caller who passes `threshold = 1` (or `0`) to the FROST signing entry-points bypasses initialization-time rejection, and the protocol runs with incorrect Lagrange coefficients, producing a cryptographically invalid/unusable signature.

### Finding Description

`assert_key_invariants` (used by `keygen` and `reshare`) enforces both bounds on `threshold`: [1](#0-0) 

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

`validate_triple_inputs` (used by OT-ECDSA triple generation) also enforces the lower bound: [2](#0-1) 

By contrast, `assert_sign_inputs` — the dedicated validation function for FROST signing — only checks the upper bound: [3](#0-2) 

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: threshold.value() < 2
```

`ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement at the type level: [4](#0-3) 

So `ReconstructionLowerBound(1)` or `ReconstructionLowerBound(0)` passes `assert_sign_inputs` without error.

The same gap exists in the FROST `presign` function: [5](#0-4) 

### Impact Explanation

FROST signing uses the caller-supplied `threshold` to compute Lagrange interpolation coefficients over the set of signers. If `threshold = 1` is supplied while the key was generated with `threshold = 2`, the Lagrange coefficients are computed for a 1-of-N scheme instead of the correct 2-of-N scheme. The aggregated partial signatures do not reconstruct the correct secret nonce contribution, and the final signature fails verification against the public key. All honest participants complete the protocol and accept the output, but the signature is cryptographically unusable.

This matches: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

- `ReconstructionLowerBound` is a plain numeric wrapper; nothing prevents a caller from constructing `ReconstructionLowerBound(1)`.
- The asymmetry between `assert_key_invariants` (which rejects `threshold < 2`) and `assert_sign_inputs` (which does not) makes it easy for an integrator to assume the signing path performs the same validation as the keygen path.
- A misconfigured or malicious coordinator supplying `threshold = 1` to `assert_sign_inputs` causes all honest participants to waste their signing round and receive an invalid signature, permanently denying a successful signing attempt for that session.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces:

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants { ... });
    }
+   if threshold.value() < 2 {
+       return Err(InitializationError::ThresholdTooSmall { threshold: threshold.value(), min: 2 });
+   }
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge { ... });
    }
    // ...
}
```

Apply the same fix to the `presign` function in `src/frost/mod.rs`.

### Proof of Concept

1. Run keygen with `threshold = 2`, `participants = [P1, P2, P3]`. All parties receive valid shares.
2. Call `assert_sign_inputs([P1, P2, P3], threshold=1, me=P1, coordinator=P1)`. **No error is returned** — the missing lower-bound check allows this to pass.
3. Each participant runs the FROST signing protocol with `threshold = 1`. Lagrange coefficients are computed as if only one share is needed (coefficient = 1 for the single signer), but the actual secret is split across shares with degree-1 polynomial (threshold-2 scheme). The aggregated signature `z = Σ z_i` does not equal the correct value `k + x·c` and fails `public_key.verify(msg, signature)`.
4. All honest parties have completed the protocol and accepted the (invalid) output — the signing session is permanently wasted. [6](#0-5) [7](#0-6)
