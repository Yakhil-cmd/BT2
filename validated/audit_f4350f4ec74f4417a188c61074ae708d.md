### Title
ETH Permanently Locked in `RewardSupplier.tick()` When Called With No Pending Mint Requests — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary
`RewardSupplier.sol`'s `tick()` function is declared `payable` but contains no ETH-refund path when `amountToMint == 0`. Any ETH forwarded by a caller in that branch is permanently trapped in the contract with no withdrawal mechanism.

---

### Finding Description
`tick()` accepts ETH via `msg.value` to cover StarkGate bridge fees and L2 messaging fees. The entire `msg.value` is consumed only inside the `if (amountToMint > 0)` branch. When `amountToMint == 0` — i.e., no pending L2→L1 mint-request messages exist — the function exits immediately without spending or returning `msg.value`.

Root cause: [1](#0-0) 

The `if (amountToMint > 0)` block at line 111 is the only place `msg.value` is consumed. [2](#0-1) 

There is no `else` branch and no `payable(msg.sender).call{value: msg.value}("")` refund. The contract inherits `ProxySupportImpl` and has no ETH-withdrawal function, so locked ETH is irrecoverable without a governance-controlled upgrade.

The analog to the original report's vulnerability class is direct: the original issue was an unsafe ETH-transfer pattern (deprecated `transfer()`/`send()` with no result check). Here the unsafe pattern is the inverse — ETH is accepted via a `payable` function but silently discarded on the zero-mint code path, producing the same class of unintended fund loss.

---

### Impact Explanation
**Medium — Griefing / permanent loss of caller-supplied ETH.**

Any ETH sent to `tick()` when there are no pending mint requests is permanently locked in the contract. This damages the caller (protocol operator or any public caller) and degrades the protocol's ability to fund future bridge-fee calls without a governance upgrade to recover the ETH. The impact matches: *Griefing with no profit motive but damage to users or protocol.*

---

### Likelihood Explanation
**Low-Medium.**

`tick()` is `external` with no access control; any address may call it with ETH. [3](#0-2) 

A realistic trigger is a **race condition**: a caller reads `requiredMinting()` off-chain, observes pending messages, prepares a `tick()` call with ETH, but a competing transaction processes those messages first. The caller's transaction then executes with `amountToMint == 0` and the ETH is locked. Alternatively, an off-chain automation script with a bug could repeatedly call `tick()` with ETH even when there is nothing to mint.

---

### Recommendation
Add an ETH refund at the end of `tick()` (or in an `else` branch) to return any unused `msg.value` to `msg.sender`:

```solidity
if (amountToMint > 0) {
    // ... existing logic ...
} else if (msg.value > 0) {
    (bool ok, ) = msg.sender.call{value: msg.value}("");
    require(ok, "ETH_REFUND_FAILED");
}
```

Alternatively, remove `payable` from `tick()` entirely and require callers to send ETH only when `amountToMint > 0` (checked off-chain via `requiredMinting()`).

---

### Proof of Concept

**Direct path (no race):**
1. Deploy `RewardSupplier` on a fork.
2. Ensure no L2→L1 mint-request messages are pending (`requiredMinting()` returns `(0, 0)`).
3. Call `tick{value: 1 ether}()`.
4. Observe: the call succeeds, `msg.value` is not returned, and `address(rewardSupplier).balance` increases by 1 ETH with no withdrawal path.

**Race-condition variant:**
1. Two callers both observe `requiredMinting()` returning `(X, Y)` with `X > 0`.
2. Caller A's `tick{value: 1 ether}()` is mined first and processes all messages.
3. Caller B's `tick{value: 1 ether}()` is mined second; `amountToMint == 0`; 1 ETH is permanently locked.

```text
    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
```
