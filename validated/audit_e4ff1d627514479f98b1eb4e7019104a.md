Audit Report

## Title
Malicious NNS Node Operator Can Indefinitely Prevent Governance Upgrade Proposal Rejection via Unconditional Proposal Overwrite - (File: rs/nns/handlers/root/impl/src/root_proposals.rs)

## Summary
The `submit_root_proposal_to_upgrade_governance_canister` function unconditionally overwrites any existing pending proposal from the same proposer principal, resetting all accumulated ballots to `Undecided` and restarting the 7-day expiry clock. A single malicious NNS node operator can exploit this to prevent their governance upgrade proposal from ever being rejected by resubmitting whenever the "No" vote count approaches the rejection threshold, keeping the proposal alive indefinitely and potentially enabling a future malicious governance upgrade.

## Finding Description
Proposals are stored in a `BTreeMap<PrincipalId, GovernanceUpgradeRootProposal>` keyed by proposer principal (L142-144). When a new proposal is submitted, the code at L255-259 only logs the existence of a prior proposal from the same principal — it applies no guard — and then unconditionally calls `proposals.borrow_mut().insert(caller, ...)` at L266-278, replacing the prior entry with a fresh struct that has `node_operator_ballots` reset to all-`Undecided` (except the proposer's own automatic `Yes`) and `submission_timestamp_seconds: now`.

The rejection threshold requires `f + 1` "No" votes where `f = (num_nodes - 1) / 3` (L128-138). The attack is straightforward:

1. Operator A submits a proposal → `PROPOSALS[A]` initialized with all ballots `Undecided` except A's `Yes`.
2. Other operators vote `No`; the count climbs toward `f + 1`.
3. When the count reaches `f` (one vote short of rejection), A calls `submit_root_proposal_to_upgrade_governance_canister` again with the same (or any valid) wasm.
4. `PROPOSALS[A]` is overwritten: all `No` ballots erased, `submission_timestamp_seconds` reset to `now`.
5. Repeat indefinitely — the proposal is never rejected and the 7-day timer never expires.

A secondary TOCTOU window exists: `vote_on_root_proposal_to_upgrade_governance_canister` clones the proposal at L309, then yields at L311 (`get_nns_membership(...).await`). After the yield, it re-reads the live proposal from `PROPOSALS` at L318. If A overwrites the proposal during that yield with the same wasm sha, the `wasm_sha256` check at L354 passes and the voter's ballot is applied to the fresh (reset) proposal, not the accumulated one. This compounds the primary attack but is not required for it.

## Impact Explanation
A single malicious NNS node operator can permanently neutralize the rejection mechanism for their own governance upgrade proposal. The 7-day expiry is rendered ineffective because each resubmission resets `submission_timestamp_seconds`. Honest node operators are forced into an indefinite re-voting loop with no recourse. If the attacker eventually coordinates or coerces enough NNS node operators to reach the `N - f` "Yes" threshold, the malicious wasm executes against the governance canister — compromising the entire NNS. This maps to **High ($2,000–$10,000): Significant NNS security impact with concrete user or protocol harm**, with escalation potential to Critical if the attacker achieves execution of a malicious governance upgrade.

## Likelihood Explanation
The attacker must hold the role of NNS node operator, which is a below-fault-threshold protocol peer (a single operator is far below `f = (N-1)/3`). No cryptographic break, key leak, or majority coordination is required. The public `get_pending_root_proposals_to_upgrade_governance_canister` endpoint allows the attacker to monitor vote counts in real time. The only action required is submitting a new proposal before the rejection threshold is crossed — a trivially automatable on-chain call. The motivation is high for any operator attempting a governance takeover.

## Recommendation
1. **Reject overwrite if any non-`Undecided` ballot exists.** Before the `insert` at L266, check whether any ballot in the existing proposal has been cast; if so, return an error requiring explicit withdrawal first.
2. **Disallow resubmission while a proposal is pending.** Require the proposer to call an explicit withdrawal/cancellation endpoint before a new proposal from the same principal is accepted.
3. **If overwrite is permitted, preserve `submission_timestamp_seconds`.** Do not reset the expiry clock to `now`; carry forward the original submission timestamp so the 7-day window cannot be extended.

## Proof of Concept
```
// Setup: NNS subnet with N=7 nodes, f=2, rejection threshold = f+1 = 3 "No" votes.

1. Operator A calls submit_root_proposal_to_upgrade_governance_canister(wasm_v1)
   → PROPOSALS[A] = { ballots: [(A,Yes),(B,Undecided),...], timestamp: T1 }

2. B votes No → ballots: [(A,Yes),(B,No),(C,Undecided),...]
3. C votes No → ballots: [(A,Yes),(B,No),(C,No),...]
   // One more "No" vote triggers is_byzantine_majority_no() → rejection.

4. A calls submit_root_proposal_to_upgrade_governance_canister(wasm_v1) // same wasm
   → PROPOSALS[A] = { ballots: [(A,Yes),(B,Undecided),(C,Undecided),...], timestamp: T2 }
   // All "No" votes erased. Timer reset to T2.

5. Repeat from step 2 indefinitely.
   // is_byzantine_majority_no() never returns true. Proposal never expires.
```

A deterministic integration test using PocketIC can confirm this by: (1) registering 7 node operators on a mock NNS subnet, (2) submitting a proposal from operator A, (3) casting 2 "No" votes, (4) calling submit again from A, (5) asserting that the ballot vector in `PROPOSALS[A]` has been reset and `submission_timestamp_seconds` advanced, and (6) verifying that a third "No" vote still does not trigger rejection on the fresh proposal.
