### Title
Blob Base Fee Validation Skipped in Simulation Mode — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary
In `validation_impl.rs`, the EIP-4844 blob base fee check is guarded by `!Config::SIMULATION`, causing it to be bypassed during simulation. A blob transaction whose `max_fee_per_blob_gas` is below the current `block_base_fee_per_blob_gas` will pass simulation but be rejected on-chain, giving the submitter a false positive and wasting gas.

### Finding Description
At line 405 of `validation_impl.rs`, the blob base fee check reads:

```rust
if &block_base_fee_per_blob_gas > tx_max_fee_per_blob_gas && !Config::SIMULATION {
    return Err(TxError::Validation(
        InvalidTransaction::BlobBaseFeeGreaterThanMaxFeePerBlobGas,
    ));
}
``` [1](#0-0) 

The `&& !Config::SIMULATION` guard unconditionally suppresses this check during simulation. The analogous EIP-1559 base fee check in `gas_helpers.rs` carries an explicit comment — `// Skip this check on simulation` — documenting the intentional design choice:

```rust
if !Config::SIMULATION {
    // Skip this check on simulation
    require!(
        &base_fee <= max_fee_per_gas,
        TxError::Validation(InvalidTransaction::BaseFeeGreaterThanMaxFee,),
        system
    )?;
}
``` [2](#0-1) 

The blob base fee check has no such comment and no corresponding test asserting that simulation should succeed when `max_fee_per_blob_gas < block_base_fee_per_blob_gas`. The base fee skip is justified because `eth_call`/`eth_estimateGas` must work even when `max_fee_per_gas` is zero; that rationale does not extend to blob fee validation, where the user is explicitly setting a ceiling price for blob inclusion. Skipping it means simulation cannot detect the condition that will cause on-chain rejection.

The simulation entry point is `simulate_tx` in `forward_system/src/run/mod.rs`, which runs `CallSimulationBootloader` with `BasicBootloaderCallSimulationConfig` (where `SIMULATION = true`): [3](#0-2) 

### Impact Explanation
A user or bundler calls `eth_estimateGas` (or any simulation path) for a blob transaction with `max_fee_per_blob_gas` set below the current `block_base_fee_per_blob_gas`. The simulation returns success and a gas estimate. The user submits the transaction on-chain. The bootloader's non-simulation path evaluates the same check without the `!Config::SIMULATION` guard and returns `InvalidTransaction::BlobBaseFeeGreaterThanMaxFeePerBlobGas`, rejecting the transaction. The submitter has paid for a failed transaction based on a misleading simulation result.

### Likelihood Explanation
Medium. Blob base fees fluctuate with blob demand (EIP-4844 fee market). Any user or tooling that relies on `eth_estimateGas` to validate a blob transaction before submission — a standard workflow — will encounter this divergence whenever the blob base fee rises between simulation and submission, or whenever the user sets `max_fee_per_blob_gas` conservatively below the current fee. The attacker-controlled entry path is simply submitting a blob transaction with a low `max_fee_per_blob_gas` to the simulation RPC endpoint.

### Recommendation
Remove the `!Config::SIMULATION` guard from the blob base fee check so it applies in both modes:

```rust
// Before
if &block_base_fee_per_blob_gas > tx_max_fee_per_blob_gas && !Config::SIMULATION {
    return Err(TxError::Validation(
        InvalidTransaction::BlobBaseFeeGreaterThanMaxFeePerBlobGas,
    ));
}

// After
if &block_base_fee_per_blob_gas > tx_max_fee_per_blob_gas {
    return Err(TxError::Validation(
        InvalidTransaction::BlobBaseFeeGreaterThanMaxFeePerBlobGas,
    ));
}
``` [1](#0-0) 

### Proof of Concept
1. Set `block_base_fee_per_blob_gas` to `1000` in the block context.
2. Construct an EIP-4844 blob transaction with `max_fee_per_blob_gas = 1` (below the blob base fee).
3. Call `simulate_tx` (or `tester.simulate_block`) — the simulation succeeds because the check at line 405 is skipped under `Config::SIMULATION = true`.
4. Submit the identical transaction via `execute_block` — the bootloader evaluates the check without the simulation guard and rejects it with `BlobBaseFeeGreaterThanMaxFeePerBlobGas`.
5. The submitter has wasted gas on a transaction that a correct simulation would have flagged as invalid.
