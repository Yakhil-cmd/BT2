### Title
Unpermissioned `deploy_erc20_token` Allows Any Caller to Permanently Front-Run NEP-141 ERC-20 Registration - (File: `engine/src/contract_methods/connector.rs`)

### Summary

`deploy_erc20_token` has no caller access control. Any NEAR account can invoke it with an arbitrary NEP-141 account ID, permanently registering that NEP-141 at an ERC-20 address derived from the attacker's own EVM address and nonce. Because the NEP-141 ↔ ERC-20 bijection map has no removal path, the legitimate deployment is permanently blocked.

### Finding Description

`deploy_erc20_token` in `engine/src/contract_methods/connector.rs` enforces only `require_running`: [1](#0-0) 

No `require_owner_only` or equivalent guard is present. The function delegates to `engine::deploy_erc20_token` in `engine/src/engine.rs`, which:

1. Constructs an EVM engine whose `origin` is `near_account_to_evm_address(predecessor_account_id)` — i.e., the attacker's NEAR account mapped to an EVM address.
2. Deploys the standard `EvmErc20` bytecode via `deploy_code_with_input`, producing an ERC-20 address deterministically derived from `(attacker_evm_address, attacker_nonce)`.
3. Calls `register_token(address, nep141)`. [2](#0-1) 

`register_token` checks whether the NEP-141 is already mapped and returns `TokenAlreadyRegistered` if so; otherwise it writes the bijection permanently: [3](#0-2) 

The underlying `BijectionMap` exposes only `insert` and `lookup_*`; there is no removal method: [4](#0-3) 

Once an attacker calls `deploy_erc20_token("usdt.tether-token.near")` (or any target NEP-141), the mapping `usdt.tether-token.near → attacker_derived_erc20_address` is written to storage and can never be overwritten or deleted.

### Impact Explanation

Any subsequent legitimate call to `deploy_erc20_token` for the same NEP-141 hits the `TokenAlreadyRegistered` branch and reverts: [5](#0-4) 

The error constant confirms the permanent nature: [6](#0-5) 

All future `ft_on_transfer` calls from that NEP-141 will mint ERC-20 tokens at the attacker-chosen address. Because the ERC-20 admin is the engine's own EVM address (set in `setup_deploy_erc20_input`), the attacker cannot directly mint or burn tokens, but:

- The canonical ERC-20 address for the NEP-141 is permanently fixed to an address the attacker chose by controlling their nonce.
- The legitimate operator can never re-deploy the ERC-20 at the expected address.
- Any protocol, DEX, or bridge that hard-codes or caches the "correct" ERC-20 address for a popular NEP-141 (e.g., `wrap.near`, `usdt.tether-token.near`) will be permanently broken, causing user funds bridged through those integrations to be unrecoverable from the expected contract — a permanent partial freeze of bridged assets.

Impact: **High — Permanent freezing of funds** (bridged NEP-141 assets are minted at an unexpected, attacker-selected ERC-20 address that downstream integrations cannot reach).

### Likelihood Explanation

- The entry point `deploy_erc20_token` is a public NEAR contract method callable by any NEAR account with no deposit or special key required.
- The attacker only needs to submit one NEAR transaction before the legitimate deployment.
- The target NEP-141 account IDs for popular tokens (`wrap.near`, stablecoin accounts) are publicly known and predictable.
- The attack is cheap (one transaction) and irreversible.

### Recommendation

Add an owner-only guard to `deploy_erc20_token`, mirroring the pattern already used in `factory_update` and `factory_set_wnear_address`:

```rust
require_owner_only(&state::get_state(&io)?, &env.predecessor_account_id())?;
```

Alternatively, restrict callers to the engine's own account (for callback flows) or to a whitelist of trusted bridge operators.

### Proof of Concept

1. Attacker NEAR account `attacker.near` submits a NEAR transaction calling `aurora.deploy_erc20_token` with input `borsh::to_vec("wrap.near")`.
2. Inside `deploy_erc20_token`, `predecessor_account_id` = `attacker.near`; `near_account_to_evm_address(b"attacker.near")` = `ATTACKER_EVM`.
3. `deploy_code_with_input` deploys `EvmErc20` at `create_legacy_address(ATTACKER_EVM, 0)` = `ATTACKER_ERC20`.
4. `register_token(ATTACKER_ERC20, "wrap.near")` writes `wrap.near → ATTACKER_ERC20` into the bijection map.
5. Aurora operator later calls `deploy_erc20_token("wrap.near")` → `register_token` returns `Err(TokenAlreadyRegistered)` → transaction reverts.
6. All subsequent `ft_on_transfer` calls from `wrap.near` mint wNEAR ERC-20 tokens at `ATTACKER_ERC20` instead of the expected canonical address, permanently breaking any integration that references the expected address. [7](#0-6) [8](#0-7) [3](#0-2)

pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let bytes = io.read_input().to_vec();
        let args =
            DeployErc20TokenArgs::deserialize(&bytes).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

        match args {
            DeployErc20TokenArgs::Legacy(nep141) => {
                let address = engine::deploy_erc20_token(nep141, None, io, env, handler)?;

                io.return_output(
                    &borsh::to_vec(address.as_bytes()).map_err(|_| errors::ERR_SERIALIZE)?,
                );
                Ok(PromiseOrValue::Value(address))
            }
            DeployErc20TokenArgs::WithMetadata(nep141) => {
                let args = borsh::to_vec(&nep141).map_err(|_| errors::ERR_SERIALIZE)?;
                let base = PromiseCreateArgs {
                    target_account_id: nep141,
                    method: "ft_metadata".to_string(),
                    args: vec![],
                    attached_balance: ZERO_YOCTO,
                    attached_gas: READ_PROMISE_ATTACHED_GAS,
                };
                let callback = PromiseCreateArgs {
                    target_account_id: env.current_account_id(),
                    method: "deploy_erc20_token_callback".to_string(),
                    args,
                    attached_balance: ZERO_YOCTO,
                    attached_gas: DEPLOY_ERC20_TOKEN_CALLBACK_ATTACHED_GAS,
                };
                // Safe because these promises are read-only calls to the main engine contract
                // and this transaction could be executed by the owner of the contract only.
                let promise_args = PromiseWithCallbackArgs { base, callback };
                let promise_id = handler.promise_create_with_callback(&promise_args);

                handler.promise_return(promise_id);

                Ok(PromiseOrValue::Promise(promise_args))
            }
        }
    })

pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}

    pub fn insert(&mut self, left: &L, right: &R) {
        let key = self.left_key(left);
        self.io.write_storage(&key, right.as_ref());

        let key = self.right_key(right);
        self.io.write_storage(&key, left.as_ref());
    }

    pub fn lookup_left(&self, left: &L) -> Option<R> {
        let key = self.left_key(left);
        self.io
            .read_storage(&key)
            .and_then(|v| v.to_vec().try_into().ok())
    }

    pub fn lookup_right(&self, right: &R) -> Option<L> {
        let key = self.right_key(right);

            .read_storage(&key)
            .and_then(|v| v.to_vec().try_into().ok())
    }
