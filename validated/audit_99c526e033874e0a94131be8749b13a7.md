### Title
`parse_fq2_bytes` / `serialize_fq2_bytes` Swap c0/c1 Relative to EIP-2537 Encoding — (`crypto/src/bls12_381/eip2537.rs`)

### Summary

`parse_fq2_bytes` assigns the first 64 bytes of input to `c0` (real part) and the second 64 bytes to `c1` (imaginary part). EIP-2537 mandates the opposite layout: `Im(x) || Re(x)` — imaginary (`c1`) first, real (`c0`) second. The same inversion is present in `serialize_fq2_bytes`. This affects every precompile that touches a G2 coordinate: `MAP_FP2_TO_G2`, `G2ADD`, `G2MSM`, and `PAIRING_CHECK`.

### Finding Description

`parse_fq2_bytes` in `crypto/src/bls12_381/eip2537.rs`:

```rust
pub fn parse_fq2_bytes(input: &[u8; FIELD_ELEMENT_LEN * 2]) -> Option<Fq2> {
    let c0 = parse_fq_bytes(input[0..64].try_into().ok()?)?;   // bytes 0-63 → Re
    let c1 = parse_fq_bytes(input[64..128].try_into().ok()?)?; // bytes 64-127 → Im
    Some(Fq2 { c0, c1 })
}
``` [1](#0-0) 

EIP-2537 specifies: bytes `[0..64]` = `Im(x)` = `c1`, bytes `[64..128]` = `Re(x)` = `c0`. The code has these reversed.

`serialize_fq2_bytes` has the same inversion:

```rust

    serialize_fq_bytes(el.c1, right.try_into().unwrap());  // Im → bytes 64-127

### Impact Explanation

For `MAP_FP2_TO_G2`: any caller-supplied Fq2 element with `c0 ≠ c1` is silently interpreted as its "conjugate" (`c0` and `c1` exchanged). The SWU map is applied to the wrong field element, producing a G2 point that differs from the EIP-2537 reference output. The output is then serialized with the same swap, so the returned bytes also deviate from the spec.

For `G2ADD` / `G2MSM` / `PAIRING_CHECK`: G2 points encoded per EIP-2537 (c1 first) are parsed with c0/c1 swapped. The on-curve check may reject valid points (since the swapped coordinates may not satisfy the curve equation), or may pass and compute on the wrong point. The serialized output is again swapped. The two swaps do **not** cancel: the addition/pairing is performed on the conjugate points, not the intended ones.

The most security-relevant case is `PAIRING_CHECK`: a contract performing BLS signature verification on Ethereum mainnet will receive a different boolean result on ZKsync OS for any signature whose G2 component has asymmetric Fq2 coordinates (i.e., virtually all real-world BLS signatures). This can cause valid signatures to be rejected or, depending on the contract's logic, invalid signatures to pass.

### Likelihood Explanation

The bug is triggered by any non-symmetric Fq2 input (`c0 ≠ c1`), which is the overwhelming majority of real-world inputs. Any unprivileged caller can reach the precompile via a normal EVM `CALL` to address `0x11` (MAP_FP2_TO_G2) or `0x0d`–`0x0f` (G2ADD/G2MSM/PAIRING_CHECK). No special role or privilege is required. [5](#0-4) 

### Recommendation

Swap the assignment in `parse_fq2_bytes` so that the first 64 bytes map to `c1` and the second 64 bytes map to `c0`:

```rust
pub fn parse_fq2_bytes(input: &[u8; FIELD_ELEMENT_LEN * 2]) -> Option<Fq2> {
    let c1 = parse_fq_bytes(input[0..64].try_into().ok()?)?;   // Im first
    let c0 = parse_fq_bytes(input[64..128].try_into().ok()?)?; // Re second
    Some(Fq2 { c0, c1 })
}
```

Apply the mirror fix to `serialize_fq2_bytes` (write `c1` to the first 64 bytes, `c0` to the second).

### Proof of Concept

1. Pick any asymmetric Fq2 element, e.g. `u = 1 + 2·i` (`c0 = 1`, `c1 = 2`).
2. Encode it per EIP-2537: bytes `[0..64]` = big-endian `2` (Im), bytes `[64..128]` = big-endian `1` (Re).
3. Call `MAP_FP2_TO_G2` (address `0x11`) with this 128-byte input on ZKsync OS.
4. Call the same precompile on an EVM reference (or use the EIP-2537 test vectors / `blst_map_to_g2`).
5. Assert the outputs differ.
6. Swap the two 64-byte halves in the input (i.e., feed `Re || Im` instead of `Im || Re`) and call ZKsync OS again — the output will now match the EVM reference, confirming the c0/c1 inversion.

pub fn parse_g2_bytes(input: &[u8; G2_LEN]) -> Option<(G2Affine, bool)> {
    if input.iter().all(|el| *el == 0) {
        return Some((G2Affine::identity(), false));
    }
    let x = parse_fq2_bytes(input[0..128].try_into().ok()?)?;
    let y = parse_fq2_bytes(input[128..256].try_into().ok()?)?;
    let point = G2Affine::new_unchecked(x, y);

    if !point.is_on_curve() {
        return None;
    }

    Some((point, true))
}
