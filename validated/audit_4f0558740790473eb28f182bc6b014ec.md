### Title
`max_nonce_for_validation_skip` config parameter is defined but never consulted in `skip_stateful_validations` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field (default `0x1`) described as "Maximum nonce for which the validation is skipped." The gateway's `skip_stateful_validations` free-function, which decides whether to bypass the account's `__validate__` entry point, hardcodes `Nonce(Felt::ONE)` instead of reading that field. The config parameter is never passed to the function and has zero effect on runtime behavior — a direct analog to SuperPool's `togglePause()` setting a state that no function ever checks.

---

### Finding Description

`StatefulTransactionValidatorConfig` declares `max_nonce_for_validation_skip` and it is wired into the config schema: [1](#0-0) 

The gateway's `run_pre_validation_checks` method calls `skip_stateful_validations` as a free function, passing only the transaction, the on-chain nonce, and the mempool client — **not** `self.config`: [2](#0-1) 

Inside `skip_stateful_validations`, the threshold is hardcoded to `Nonce(Felt::ONE)`: [3](#0-2) 

When `skip_validate == true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, … }`, meaning the account's `__validate__` entry point (signature check) is entirely skipped for that transaction: [4](#0-3) 

Because `skip_stateful_validations` is a free function with no access to `self.config`, the operator-controlled `max_nonce_for_validation_skip` value is silently ignored at runtime.

---

### Impact Explanation

Two concrete mis-admission scenarios arise whenever an operator sets `max_nonce_for_validation_skip` to a non-default value:

**Scenario A — operator sets the field to `0` to disable the skip entirely:**  
The hardcoded `Nonce(Felt::ONE)` check still fires. Any attacker who knows a deploy-account transaction is pending in the mempool for an undeployed address can submit an invoke with nonce=1 and an **arbitrary/invalid signature**. The gateway skips `__validate__`, the transaction passes all admission checks, and is forwarded to the mempool — an invalid transaction accepted through account-validation logic.

**Scenario B — operator sets the field to `2` to extend the UX window:**  
A legitimate user sending `deploy_account (nonce=0)` + `invoke (nonce=1)` + `invoke (nonce=2)` in one batch will have the nonce=2 invoke rejected at the gateway because the skip only fires for nonce=1. A valid transaction is rejected before sequencing.

Both outcomes fall within the allowed impact scope:
> *High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

---

### Likelihood Explanation

The default config value (`0x1`) happens to match the hardcoded constant, so the bug is latent with a stock deployment. It becomes exploitable the moment an operator changes `max_nonce_for_validation_skip` — a documented, public config knob — to any other value. No privileged on-chain action is required; the attacker only needs to know a deploy-account is pending (observable from the mempool).

---

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded literal:

```rust
// In run_pre_validation_checks:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,   // ← add this
).await?;

// In skip_stateful_validations signature:
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
    max_nonce_for_skip: Nonce,                   // ← new parameter
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() <= max_nonce_for_skip && account_nonce == Nonce(Felt::ZERO) {
            // ... existing mempool check
        }
    }
    Ok(false)
}
```

---

### Proof of Concept

1. Deploy a fresh Starknet node with `max_nonce_for_validation_skip = 0` (operator intends to disable the skip).
2. Submit a `deploy_account` transaction for address `A` (nonce=0). It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Submit an `invoke` transaction for address `A` with nonce=1 and a **garbage signature** (e.g., all-zero felt values).
4. In `skip_stateful_validations`, `tx.nonce() == Nonce(Felt::ONE)` is `true` and `account_nonce == Nonce(Felt::ZERO)` is `true`, so the function returns `true` regardless of the config value.
5. `run_validate_entry_point` is called with `validate: false`; the blockifier never invokes `__validate__`; the transaction is accepted and forwarded to the mempool with an invalid signature — contradicting the operator's explicit configuration to disable this bypass.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
