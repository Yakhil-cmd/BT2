### Title
Third-Party Invoke Bypasses `__validate__` Signature Check via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature, intended to allow a user to submit a `deploy_account` + `invoke` pair simultaneously, uses an overly broad mempool presence check (`account_tx_in_pool_or_recent_block`) to decide whether to skip the `__validate__` entry-point call. Any unprivileged third party who observes a victim's `deploy_account` transaction in the mempool can immediately submit an invoke transaction bearing the victim's `sender_address`, nonce `1`, and an **arbitrary/invalid signature**. Because the victim's deploy_account is already in the pool, the skip condition fires, `__validate__` is never called at the gateway, and the forged invoke is admitted to the mempool. The victim's own legitimate nonce-1 invoke is then rejected as a duplicate nonce.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) fires when all three conditions hold:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When the function returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `execution_flags.validate = false`: [2](#0-1) 

Inside `AccountTransaction::validate_tx`, the very first thing done when `execution_flags.validate` is `false` is an unconditional early return — the `__validate__` entry point (which performs the ECDSA signature check) is **never invoked**: [3](#0-2) 

The mempool's `account_tx_in_pool_or_recent_block` returns `true` if **any** transaction for the address is in the pool or was seen in a recent committed block — it does **not** verify that the pooled transaction is specifically a `deploy_account`: [4](#0-3) 

The code comment at line 441–443 asserts this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is vacuously true once the attacker's own unvalidated invoke is in the pool, making the reasoning circular. The first branch is the only one that matters when `account_nonce == 0`, but the check does not distinguish between a `deploy_account` and any other transaction type. [5](#0-4) 

The mempool's `validate_fee_escalation` path will reject a second nonce-1 invoke as `DuplicateNonce` unless fee escalation is enabled and the attacker's fee is surpassed: [6](#0-5) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid (unsigned) transaction.**

- An invoke transaction with an **arbitrary signature** for a victim's not-yet-deployed account is admitted to the mempool without any signature verification.
- The victim's own legitimate nonce-1 invoke is subsequently rejected as `DuplicateNonce`, permanently blocking the victim's first post-deployment transaction unless fee escalation is enabled and the victim out-bids the attacker.
- At execution time the forged invoke will revert (because `__validate__` IS called by the batcher with `execution_flags.validate = true`), but the victim's nonce is consumed and gas fees are charged to the victim's account.

---

### Likelihood Explanation

**Medium.** The attacker needs only:

1. A publicly observable `deploy_account` transaction for the victim (visible in the mempool or via the gateway's response to the victim's submission).
2. The ability to submit a standard invoke transaction to the gateway with the victim's `sender_address` and `nonce = 1` before the victim's own invoke is processed.

No private key material, privileged access, or special infrastructure is required. The race window is the time between the victim's `deploy_account` entering the mempool and the victim's invoke being processed — a window that is non-trivial in a busy network.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** for the sender address is present in the mempool. The mempool should expose a dedicated `deploy_account_in_pool(address)` query, or the gateway should track the deploy_account tx hash returned when the victim submitted their `deploy_account` and require it to be passed alongside the invoke (as the `native_blockifier` path already does via `deploy_account_tx_hash`): [7](#0-6) 

Alternatively, the gateway's `add_tx` API could require the caller to supply the `deploy_account_tx_hash` for the skip-validate path, and the mempool could verify that hash is present as a `DeployAccount` transaction.

---

### Proof of Concept

```
1. Alice submits deploy_account(class_hash=C, salt=S, nonce=0).
   → Gateway accepts; deploy_account enters mempool for address A.
   → account_tx_in_pool_or_recent_block(A) now returns true.

2. Mallory observes A in the mempool (e.g., via get_txs or the gateway response).

3. Mallory crafts:
     invoke_tx {
       sender_address: A,
       nonce: 1,
       calldata: [transfer(A's_tokens → Mallory)],
       signature: [0xdeadbeef, 0xcafebabe],   // arbitrary, invalid
     }

4. Mallory submits invoke_tx to the gateway.

5. Gateway stateful validation:
     account_nonce = get_nonce(A) = 0          ✓ (account not deployed)
     tx.nonce() == 1                            ✓
     account_tx_in_pool_or_recent_block(A) = true  ✓ (Alice's deploy_account)
     → skip_validate = true
     → run_validate_entry_point called with execution_flags.validate = false
     → __validate__ NOT called → invalid signature NOT checked
     → invoke_tx accepted into mempool

6. Alice submits her own invoke(nonce=1, valid_signature).
   → mempool.validate_tx → DuplicateNonce error → REJECTED.

7. Block is built:
     - Alice's deploy_account(nonce=0) executes → account A deployed.
     - Mallory's invoke(nonce=1) executes:
         __validate__ called → signature invalid → REVERT.
         Fee charged to A.
     - Alice's legitimate invoke was never sequenced.
```

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
