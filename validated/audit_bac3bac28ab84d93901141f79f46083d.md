### Title
Panic on Malformed EIP-6110 Deposit Event Halts Ethereum STF Block Processing - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

### Summary

The `eip6110_events_parser` function, called during Ethereum STF block finalization, uses `panic!` instead of returning an error when a deposit event from `DEPOSIT_CONTRACT_ADDRESS` carries the correct topic signature but malformed ABI data. An unprivileged attacker who can cause any contract at that address to emit a structurally valid but data-malformed `DepositEvent` log will trigger an unconditional panic, aborting the RISC-V execution environment and halting block processing entirely.

### Finding Description

In `eip6110_events_parser`, after filtering events by address and topic, the code calls `validate_and_write_event_data` and handles its `Err` variant with an explicit `panic!`:

```rust
let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
    panic!("invalid deposit event structure");
};
```

`validate_and_write_event_data` returns `Err(())` whenever the event data length is not exactly 576 bytes, or when any of the ABI offset/length fields fail the `validate_u16_at_most` checks. The function signature already returns `Result<bool, SystemError>`, so propagating the error is straightforward. Instead, the code panics unconditionally.

The panic handler in `zksync_os/src/main.rs` calls `riscv_common::rust_abort()`, which terminates the RISC-V execution context with no recovery path.

The attacker-controlled entry path: any EVM transaction that causes a contract at `DEPOSIT_CONTRACT_ADDRESS` (`0x00000000219ab540356cbb839cbe05303d7705fa`) to emit a log whose first topic equals `DEPOSIT_EVENT_SIGNATURE_HASH` (`0x649bbc62...`) but whose data field is not exactly 576 bytes (or has wrong ABI offsets) will trigger the panic. If the deposit contract is not pre-deployed at genesis, an attacker can deploy a malicious contract at that address via CREATE2 and emit such a log. Even if the canonical deposit contract is deployed, any future upgrade or misconfiguration that causes it to emit a non-conforming event would trigger the same abort.

### Impact Explanation

A single crafted transaction that emits a malformed deposit event from the deposit contract address causes the Ethereum STF block-finalization step to call `rust_abort()`. This terminates the entire block-processing execution, making it impossible to finalize the block. The result is a complete denial of service for the Ethereum-compatible state transition: no further blocks can be processed until the panic path is removed. Because the panic occurs in the post-execution block-level hook (not inside a single transaction's revert boundary), there is no sandboxing that would limit the blast radius to one transaction.

### Likelihood Explanation

The deposit contract address is a well-known, fixed address. If it is not pre-deployed as part of the genesis state, any user can deploy an arbitrary contract there via CREATE2 (by finding an appropriate factory/salt combination) and emit a single malformed log. Even if the canonical deposit contract is present, a single non-conforming event—whether from a future upgrade, a re-deployment, or a bug in the deposit contract—is sufficient to trigger the abort. The `panic!` is unconditional and has no fallback.

### Recommendation

Replace the `panic!` with a proper error propagation. The enclosing function already returns `Result<bool, SystemError>`, so the fix is:

```rust
validate_and_write_event_data(event, requests_hasher, &mut logger)
    .map_err(|_| SystemError::from(internal_error!("invalid deposit event structure")))?;
```

This converts the invalid-event condition into a recoverable `SystemError` that the caller can handle (e.g., by rejecting the block) without aborting the entire execution environment.

### Proof of Concept

1. Deploy a contract `MaliciousDeposit` at `DEPOSIT_CONTRACT_ADDRESS` (via CREATE2 with an appropriate factory and salt, or by being the first deployer at that address on a fresh chain).
2. `MaliciousDeposit.emitBadEvent()` emits a log:
   - `address`: `DEPOSIT_CONTRACT_ADDRESS`
   - `topics[0]`: `0x649bbc62d0e31342afea4e5cd82d4049e7e1ee912fc0889aa790803be39038c5`
   - `data`: any byte string whose length is **not** 576 (e.g., 64 bytes of zeros)
3. Submit a transaction calling `emitBadEvent()` to the Ethereum STF.
4. When the block finalizer calls `eip6110_events_parser`, `validate_and_write_event_data` returns `Err(())` because `data.len() != 576`, the `else` branch executes `panic!("invalid deposit event structure")`, and `rust_abort()` terminates the RISC-V execution context.
