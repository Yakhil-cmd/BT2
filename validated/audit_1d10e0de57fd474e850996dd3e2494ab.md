Audit Report

## Title
Third-Party Caller Can Bypass `confirmation_text` Consent Gate and Force Victim's Participation in SNS Swap — (`rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` endpoint on the SNS Swap canister accepts an arbitrary `buyer` principal without verifying that the caller equals that principal. Because the swap's `confirmation_text` is stored in public `Init` state and `validate_confirmation_text` only performs a string equality check with no caller-identity binding, any unprivileged attacker can read the public consent text and submit it on behalf of any victim who has already transferred ICP to their swap subaccount. This records the victim as having explicitly consented to the swap's terms, commits their ICP, and destroys their open ticket — all without any action by the victim.

## Finding Description

In `rs/sns/swap/canister/canister.rs` L130–134, the effective buyer principal is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // no caller == buyer check
};
``` [1](#0-0) 

The resolved `p` is passed directly to `refresh_buyer_token_e8s` along with the caller-supplied `confirmation_text`. Inside that function, the only consent check is:

```rust
self.validate_confirmation_text(confirmation_text)?;
``` [2](#0-1) 

`validate_confirmation_text` performs a pure string equality check against the value stored in `Init`:

```rust
(Some(expected_text), Some(text)) => {
    if &text != expected_text { Err(...) } else { Ok(()) }
}
``` [3](#0-2) 

There is no check that the caller is the buyer. The `confirmation_text` is part of the public `Init` struct returned by the unauthenticated `get_init` query endpoint: [4](#0-3) [5](#0-4) 

After passing the text check, the function reads the ICP balance of the victim's subaccount, registers it as the victim's participation, and deletes the victim's open ticket: [6](#0-5) [7](#0-6) 

The proto definition explicitly documents the `buyer` field as "If not specified, the caller is used," confirming the field is intentionally caller-overridable, but no guard was added to protect the consent gate when a `confirmation_text` is present: [8](#0-7) 

## Impact Explanation

This is a **High** severity finding matching: *"Significant SNS security impact with concrete user or protocol harm."*

The `confirmation_text` feature exists specifically to enforce explicit participant consent (e.g., legal disclaimers, risk acknowledgements) before ICP is committed. Because the text is public and the caller-identity check is absent, the consent gate provides zero protection. Any victim who has transferred ICP to their swap subaccount (the normal first step of the payment flow) can be:

1. Recorded on-chain as having explicitly agreed to terms they never read.
2. Forced into the swap with their ICP locked until the swap concludes (committed or aborted).
3. Their open ticket silently destroyed, breaking the ticket-based payment flow state machine for that user.

## Likelihood Explanation

The attack requires only: knowledge of the victim's principal (public on-chain data), knowledge of the swap's `confirmation_text` (readable from the public `get_init` query), and the victim having already transferred ICP to their swap subaccount (the standard first step). No privileged access, key material, or majority corruption is needed. The attacker pays only the cycles cost of a single update call and can repeat the attack against any number of victims in any open SNS swap that uses `confirmation_text`.

## Recommendation

When a `confirmation_text` is configured, require that the effective buyer principal equals the caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    if specified != caller_principal_id()
        && swap().init_or_panic().confirmation_text.is_some()
    {
        panic!("Caller must equal buyer when a confirmation_text is required");
    }
    specified
};
```

Alternatively, always use `caller_principal_id()` unconditionally and remove the `buyer` override field entirely, since the ICP subaccount is already keyed to the buyer's principal.

## Proof of Concept

1. Deploy an SNS swap with `confirmation_text = "I confirm I have read the terms."`.
2. Victim transfers 10 ICP to `swap_canister[principal_to_subaccount(victim)]` on the ICP ledger (normal first step).
3. Victim has not yet called `refresh_buyer_tokens` — still deciding whether to participate.
4. Attacker calls `get_init` on the swap canister (unauthenticated query) and reads `confirmation_text`.
5. Attacker sends an ingress update call to `refresh_buyer_tokens` with `buyer = victim_principal_text` and `confirmation_text = "I confirm I have read the terms."`.
6. The swap canister resolves `p = victim_principal`, passes `validate_confirmation_text` (public text matches), reads the victim's 10 ICP balance, registers it as the victim's participation, and deletes the victim's open ticket.
7. The victim is now a registered swap participant recorded as having consented to the terms, with their ICP locked — without ever having called the endpoint themselves.

This is directly reproducible as a unit test by adapting the existing `test_swap_participation_confirmation` test in `rs/sns/swap/tests/swap.rs` to call `refresh_buyer_token_e8s` with a different `buyer` principal than the one whose ledger balance is mocked, supplying the correct `confirmation_text`. /// Test that the `refresh_buyer_token_e8s` function handles confirmations correctly.
#[test]
fn test_swap_participation_confirmation() {
    let confirmation_text = "Please confirm that 2+2=4".to_string();
    let another_text = "Please confirm that 2+2=5".to_string();
    let user = PrincipalId::new_user_test_id(1);
    let amount = 101 * E8;

    let buy_token = |swap: &mut Swap, confirmation_text: Option<String>| {
        swap.refresh_buyer_token_e8s(
            user,
            confirmation_text,
            SWAP_CANISTER_ID,
            &mock_stub(vec![LedgerExpect::AccountBalance(
                Account {
                    owner: SWAP_CANISTER_ID.get().into(),
                    subaccount: Some(principal_to_subaccount(&user)),
                },
                Ok(Tokens::from_e8s(amount)),
            )]),
        )
        .now_or_never()
        .unwrap()
    };

    // A. SNS specifies confirmation text & client sends confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .with_confirmation_text(confirmation_text.clone())
            .build();
        // A.1. The texts match
        assert_is_ok!(buy_token(&mut swap, Some(confirmation_text.clone())));
        // A.2. The texts do not match
        assert_is_err!(buy_token(&mut swap, Some(another_text)));
    }

    // B. SNS specifies confirmation text & client does not send a confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .with_confirmation_text(confirmation_text.clone())
            .build();
        assert_is_err!(buy_token(&mut swap, None));
    }

    // C. SNS does not specify confirmation text & client sends a confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .without_confirmation_text()
            .build();
        assert_is_err!(buy_token(&mut swap, Some(confirmation_text)));
    }

    // D. SNS does not specify confirmation text & client does not send a confirmation text
    {
        let mut swap = SwapBuilder::new()
            .with_lifecycle(Open)
            .without_confirmation_text()
            .build();
        assert_is_ok!(buy_token(&mut swap, None));
    }
}
