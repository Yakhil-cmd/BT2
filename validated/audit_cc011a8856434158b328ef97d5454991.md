### Title
`skip_stateful_validations` Overly Broad Mempool Presence Check Allows Signature-Bypass Admission of Forged Invoke Transactions - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call only when a `deploy_account` is pending for an account (UX feature for simultaneous deploy+invoke). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account, not specifically a `deploy_account`. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can submit a forged invoke with `nonce=1` from the victim's address, bypassing gateway signature verification. The gateway admits the forged transaction; the batcher later calls `__validate__`, which fails, reverting the invoke but still charging fees to the victim's account and permanently displacing the victim's legitimate `nonce=1` invoke.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` triggers when all three conditions hold:

```
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

When those conditions are met, `run_validate_entry_point` sets `validate: false`, so the `__validate__` entry point is never called during gateway validation: [2](#0-1) 

The code comment claims the mempool check is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is flawed. The mempool implementation of `account_tx_in_pool_or_recent_block` is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

It returns `true` for **any** transaction type from the account. It does not distinguish between a `deploy_account` (which proves the signer controls the key) and an arbitrary invoke that itself skipped validation. The check is therefore circular: a forged `nonce=1` invoke can be admitted because the victim's `deploy_account` is in the pool, and that forged invoke then satisfies the condition for future forged invokes.

The two earlier checks — `validate_state_preconditions` (nonce range + resource bounds) and `validate_by_mempool` (duplicate hash + nonce ordering) — do not verify the transaction signature: [4](#0-3) [5](#0-4) 

During batcher execution, `AccountTransaction` is created with `execution_flags.validate = true` (the default), so `__validate__` **is** called: [6](#0-5) 

`validate_tx` checks `self.execution_flags.validate` and calls the account's `__validate__` entry point: [7](#0-6) 

The forged invoke fails `__validate__` (wrong signature), is reverted, but the fee is still charged to the victim's account per the post-execution fee flow: [8](#0-7) 

### Impact Explanation

The gateway admits a transaction with an invalid (attacker-controlled) signature from the victim's address. The victim's legitimate `nonce=1` invoke is displaced from the mempool by the higher-fee forged invoke. During batcher execution the forged invoke fails `__validate__`, is reverted, and the fee is charged to the victim's account — economic loss the victim did not authorize. The victim's legitimate invoke is permanently lost. This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack requires only that the victim's `deploy_account` be visible in the mempool, which is the normal state for any new account deployment. The victim's address is deterministically derivable from the public `deploy_account` fields. No privileged access, no special network position, and no cryptographic secret is required. Any mempool observer can execute this attack.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address. The mempool should expose a dedicated `deploy_account_in_pool_or_recent_block(address)` query that only returns `true` when a `DeployAccount` transaction (not any transaction) is present. This mirrors the fix applied in the Inverse Finance bug: tighten the guard to match the actual invariant being enforced.

### Proof of Concept

1. Alice submits `deploy_account` (nonce=0) — admitted to the mempool.
2. Attacker observes Alice's `deploy_account` in the mempool and extracts Alice's address `0xALICE`.
3. Attacker submits a forged invoke (nonce=1) from `0xALICE` with arbitrary calldata and fees higher than Alice's legitimate invoke.
4. Gateway evaluates `skip_stateful_validations`:
   - `tx.nonce() == Nonce(Felt::ONE)` ✓
   - `account_nonce == Nonce(Felt::ZERO)` ✓ (account not yet on-chain)
   - `account_tx_in_pool_or_recent_block(0xALICE) == true` ✓ (Alice's `deploy_account` is in the pool)
5. Gateway sets `validate: false`, skips `__validate__`, and admits the forged invoke.
6. Alice submits her legitimate invoke (nonce=1) — rejected by the mempool (duplicate nonce, lower fee).
7. Batcher executes: `deploy_account` succeeds; forged invoke runs with `validate: true`, `__validate__` fails (wrong signature), invoke is reverted, fee is charged to Alice's account.
8. Alice's legitimate invoke is never executed; Alice's account is debited fees for an unauthorized transaction. [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
