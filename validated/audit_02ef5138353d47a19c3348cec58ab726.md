Audit Report

## Title
ckBTC Minter Permanently Burns User ckBTC Without Reimbursement on `AmountTooLow` at Execution Time ‚Äî (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

The ckBTC minter gates `retrieve_btc` submissions against a live, mutable `fee_based_retrieve_btc_min_amount` threshold, burns the user's ckBTC irreversibly upon acceptance, then re-evaluates the same effective threshold at execution time using freshly fetched Bitcoin fee percentiles. If fees spike between submission and processing, `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, and the minter finalizes the request with `FinalizedStatus::AmountTooLow` without minting ckBTC back to the user. The `InvalidTransaction` error path does trigger reimbursement; the `AmountTooLow` path does not. The user permanently loses their ckBTC.

## Finding Description

**Step 1 ‚Äî Submission-time check uses the dynamic threshold.**

`retrieve_btc` reads `fee_based_retrieve_btc_min_amount` from live state and rejects requests below it: [1](#0-0) 

If the check passes, `burn_ckbtcs` is called and the burn is committed to the ledger. The request is queued with a `reimbursement_account` pointing back to the caller: [2](#0-1) 

**Step 2 ‚Äî `estimate_fee_per_vbyte` overwrites the threshold before processing.**

`submit_pending_requests` calls `estimate_fee_per_vbyte`, which fetches live Bitcoin fee percentiles and mutates `fee_based_retrieve_btc_min_amount` in state: [3](#0-2) 

The formula rounds up in 50,000-satoshi steps, so a single fee-rate jump can raise the effective minimum by 50,000 sat or more: [4](#0-3) 

**Step 3 ‚Äî `AmountTooLow` at execution time drops the request with no reimbursement.**

When `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, the minter calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow` for every request in the batch: [5](#0-4) 

`remove_retrieve_btc_request` only records an event and pushes to the finalized queue ‚Äî it does not mint ckBTC back: [6](#0-5) 

**Step 4 ‚Äî Contrast: `InvalidTransaction` does reimburse.**

The `InvalidTransaction` branch calls `reimburse_canceled_requests`, which mints ckBTC back to the `reimbursement_account`. The `AmountTooLow` branch has no equivalent call: [7](#0-6) 

The two state fields confirm the threshold is live and mutable, distinct from the static floor: [8](#0-7) 

## Impact Explanation

This is a ledger conservation bug in an in-scope ck-token financial integration: ckBTC is burned on the ICRC-1 ledger, no corresponding BTC is ever sent on-chain, and no ckBTC is minted back. Every affected user suffers a permanent, unrecoverable loss of funds. The impact matches the allowed High bounty class: "Significant Chain Fusion, ck-token, ledger ‚Ä¶ security impact with concrete user or protocol harm." Aggregate losses across many users during a fee spike event could be material.

## Likelihood Explanation

Bitcoin fee markets are volatile and fee spikes are routine. The `max_time_in_queue_nanos` parameter (10 minutes on mainnet) means requests routinely sit in the queue long enough for fees to change materially. The entry path is fully unprivileged ‚Äî any caller of `retrieve_btc` or `retrieve_btc_with_approval` is exposed. No special privileges, social engineering, or external compromise is required. A user submitting near the current minimum during a low-fee window faces a realistic probability of hitting this path during the next fee spike.

## Recommendation

In the `BuildTxError::AmountTooLow` branch of `submit_pending_requests` (`lib.rs` ~L412), replace the bare `remove_retrieve_btc_request` loop with a call to `reimburse_canceled_requests` (as already done for `BuildTxError::InvalidTransaction`), using an appropriate `WithdrawalReimbursementReason`. The `reimbursement_account` is already stored on every `RetrieveBtcRequest`. The same fix should be applied to the `DustOutput` branch for the specific request being dropped. Alternatively, re-queue the request and retry when fees drop rather than permanently discarding it.

## Proof of Concept

1. Bitcoin fees are low; `fee_based_retrieve_btc_min_amount` = 100,000 sat.
2. Unprivileged user calls `retrieve_btc` with `amount` = 100,001 sat. The check at `retrieve_btc.rs` L169 passes. `burn_ckbtcs` executes at L210 ‚Äî ckBTC is irreversibly burned.
3. Before `submit_pending_requests` runs, Bitcoin fees spike. `estimate_fee_per_vbyte` updates `fee_based_retrieve_btc_min_amount` to 150,000 sat (one 50,000-sat step) at `lib.rs` L248.
4. `build_unsigned_transaction` is called with the new fee rate. The queued 100,001-sat request cannot cover fees ‚Üí returns `BuildTxError::AmountTooLow`.
5. The `AmountTooLow` branch at `lib.rs` L412 calls `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No `reimburse_canceled_requests` call is made.
6. `audit::remove_retrieve_btc_request` at `audit.rs` L67 records an event and finalizes ‚Äî no ckBTC is minted back.
7. User's ckBTC is permanently lost. A deterministic integration test using PocketIC can reproduce this by: (a) setting a low fee percentile, (b) calling `retrieve_btc`, (c) updating the fee percentile to a higher value before the next `submit_pending_requests` tick, (d) asserting the user's ckBTC balance is unchanged (it will be lower, proving the loss).

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
