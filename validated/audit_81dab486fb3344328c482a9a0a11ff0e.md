### Title
Permanent ERC-20 Token Loss When `ft_transfer` Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile permanently destroys a user's ERC-20 tokens if the downstream NEAR `ft_transfer` promise fails. The ERC-20 tokens are burned synchronously in the EVM before the asynchronous NEAR promise is dispatched. If that promise fails, the callback receives `refund: None` and silently does nothing, leaving the user with neither their ERC-20 tokens nor the corresponding NEP-141 tokens.

---

### Finding Description

The `ExitToNear` precompile flow for ERC-20 tokens operates in two asynchronous steps:

**Step 1 — Synchronous EVM burn (in `EvmErc20.sol`):**

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // tokens destroyed here, unconditionally
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    assembly {
        let res := call(gas(), 0xe9217bc7..., 0, add(input, 32), input_size, 0, 32)
    }
}
```

The burn is irreversible and happens before any NEAR-side confirmation. [1](#0-0) 

**Step 2 — Asynchronous NEAR promise (in `ExitToNear::run`):**

The precompile schedules a `ft_transfer` (or `ft_transfer_call`) promise. The callback args are constructed with `refund: None` when the `error_refund` feature is absent:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // <-- no recovery path compiled in
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

When the promise is a simple `ft_transfer` (no `transfer_near` args, no `refund` args), `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so **no callback is attached at all**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // fire-and-forget, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [3](#0-2) 

Even when a callback is attached (e.g., for the wNEAR unwrap path), `exit_to_near_precompile_callback` does nothing on failure when `refund` is `None`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(...)?;
    ...
} else {
    None   // silent no-op: tokens are gone
};
``` [4](#0-3) 

Additionally, `EvmErc20.sol` (deployed when `error_refund` is off) does **not** encode the sender address in the calldata, so even if a refund were attempted, the engine would not know who to credit: [1](#0-0) 

Compare with `EvmErc20V2.sol` (deployed only when `error_refund` is on), which encodes `sender`: [5](#0-4) 

The engine's own deploy logic selects the contract binary based on the feature flag: [6](#0-5) 

The test suite explicitly acknowledges the token loss:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [7](#0-6) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing (destruction) of user funds.**

Any user who calls `withdrawToNear()` on a bridged ERC-20 token and whose downstream `ft_transfer` promise fails will permanently lose the burned tokens. The EVM state records the burn as final; there is no compensating mint or credit. The NEP-141 tokens remain held by the Aurora engine account with no mechanism to release them to the user.

---

### Likelihood Explanation

**Likelihood: Medium.**

The `ft_transfer` promise can fail for several realistic, user-triggerable reasons:
- The recipient NEAR account is not registered with the NEP-141 contract (a common user mistake).
- The NEP-141 contract is paused at the time the promise executes.
- The NEP-141 contract has been deleted or migrated.
- Insufficient gas is forwarded to the promise (the allocated `FT_TRANSFER_GAS = 10 TGas` may be insufficient for some NEP-141 implementations).

No admin action or privileged access is required. Any unprivileged EVM user can trigger this by calling `withdrawToNear()` with a recipient that is not registered. [8](#0-7) 

---

### Recommendation

1. **Enable `error_refund` by default** in the production build, or make the refund path unconditional rather than feature-gated.
2. **Always attach a callback** to the `ft_transfer` promise so that failures are observable and recoverable.
3. **Always encode the sender address** in the precompile calldata (as `EvmErc20V2.sol` does) so the refund destination is always known.
4. Consider a two-phase approach: record the pending withdrawal in storage before dispatching the promise, and only clear it upon confirmed success.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (default build). The engine deploys `EvmErc20.bin` for all bridged tokens.
2. Alice bridges 100 units of `token.near` to Aurora, receiving 100 ERC-20 tokens at her EVM address.
3. Alice calls `erc20.withdrawToNear("unregistered.near", 100)`.
4. `_burn(alice, 100)` executes — Alice's ERC-20 balance drops to 0.
5. The engine schedules `ft_transfer(receiver_id: "unregistered.near", amount: 100)` on `token.near`.
6. `token.near` rejects the transfer because `unregistered.near` has no storage deposit.
7. No callback fires (or the callback fires with `refund: None` and does nothing).
8. Alice has 0 ERC-20 tokens and 0 NEP-141 tokens. The 100 units remain in `token.near`'s balance for the Aurora engine account, permanently inaccessible to Alice.
