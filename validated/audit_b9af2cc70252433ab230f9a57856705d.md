### Title
Unrestricted Contract Update Bypasses Deployment Authorization Guard - (File: `fvm/environment/contract_updater.go`)

### Summary

`SetContract` in the FVM enforces the network-wide deployment authorization list only for **new** contract deployments. When a contract already exists, the update path skips the `isAuthorizedForDeployment` check entirely, with the explicit comment "Contract updates are always allowed." This is the direct analog of the Solidity `addOperator`/`removeOperator` missing `onlyOwner`: the privileged-role guard is applied to one operation (deploy) but omitted from the equivalent mutating operation (update).

### Finding Description

In `fvm/environment/contract_updater.go`, `SetContract` checks authorization only on the `!exists` branch:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}
// No authorization check when exists == true
updater.draftUpdates[location] = ContractUpdate{...}
``` [1](#0-0) 

`isAuthorizedForDeployment` consults the network-wide `ContractDeploymentAuthorizedAddressesPath` list when `RestrictedDeploymentEnabled()` is true: [2](#0-1) 

`RemoveContract` does apply its own guard (`isAuthorizedForRemoval`) unconditionally: [3](#0-2) 

The asymmetry is therefore: deploy → guarded, update → unguarded, remove → guarded.

`UpdateAccountContractCode` (the Cadence runtime entry point) passes `updater.signingAccounts` into `SetContract`, so the signing accounts are available but the check is simply not performed for the update case: [4](#0-3) 

### Impact Explanation

When the network operates with `RestrictedDeploymentEnabled = true` (the default, `RestrictContractDeployment: true`), only accounts on the authorized-deployers list may deploy new contracts. [5](#0-4) 

However, any account that already has a contract deployed — including one that has since been **removed** from the authorized list — can freely replace that contract's bytecode with arbitrary new code. Because Cadence's `auth(UpdateContract) &Account` entitlement only requires the transaction to be signed by the account that owns the contract (not by a network-authorized deployer), the FVM-level governance restriction is fully bypassed for updates.

Concrete impact: a contract author whose deployment privileges have been revoked can silently introduce malicious logic (e.g., drain capabilities, alter resource accounting, change access rules) into a previously-deployed contract that other contracts or users depend on. This constitutes unauthorized account mutation of on-chain contract state.

### Likelihood Explanation

The entry path is a standard user-submitted transaction using the `UpdateContract` Cadence entitlement on the account that owns the contract. No special node access, leaked keys, or admin cooperation is required. The attacker only needs to control the account that holds the contract. The condition is realistic on any network where restricted deployment is enabled and account authorization lists are managed over time.

### Recommendation

Apply `isAuthorizedForDeployment` to both the new-deployment and the update branch inside `SetContract`, or introduce a separate `isAuthorizedForUpdate` check backed by the same (or a dedicated) authorized-addresses path. The guard should be:

```go
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("updating contract failed: ...")
}
```

applied regardless of whether the contract already exists, mirroring the unconditional guard already present in `RemoveContract`.

### Proof of Concept

1. Network is bootstrapped with `RestrictContractDeployment: true`.
2. Account `0xALICE` is on the authorized-deployers list; she deploys `MyContract`.
3. Admin removes `0xALICE` from the authorized-deployers list via `SetContractDeploymentAuthorizersTransaction`.
4. `0xALICE` submits:
   ```cadence
   transaction {
     prepare(signer: auth(UpdateContract) &Account) {
       signer.contracts.update(name: "MyContract", code: "<malicious bytecode>".decodeHex())
     }
   }
   ```
5. `SetContract` is called; `ContractExists` returns `true`; the `!exists` branch is skipped; `isAuthorizedForDeployment` is never called; the malicious update is staged and committed.
6. `0xALICE` is no longer on the authorized list, yet her contract now contains attacker-controlled code. [6](#0-5) [7](#0-6)

func (updater *ContractUpdaterImpl) UpdateAccountContractCode(
	location common.AddressLocation,
	code []byte,
) error {
	defer updater.tracer.StartChildSpan(
		trace.FVMEnvUpdateAccountContractCode).End()

	err := updater.meter.MeterComputation(
		common.ComputationUsage{
			Kind:      ComputationKindUpdateAccountContractCode,
			Intensity: 1,
		},
	)
	if err != nil {
		return fmt.Errorf("update account contract code failed: %w", err)
	}

	err = updater.SetContract(
		location,
		code,
		updater.signingAccounts)
	if err != nil {
		return fmt.Errorf("updating account contract code failed: %w", err)
	}

	return nil
}

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}
```

func (updater *ContractUpdaterImpl) RemoveContract(
	location common.AddressLocation,
	signingAccounts []flow.Address,
) (err error) {
	// check if authorized
	if !updater.isAuthorizedForRemoval(signingAccounts) {
		return fmt.Errorf("removing contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"RemoveContract",
				"removing contracts requires authorization from specific "+
					"accounts"))
	}

	u := ContractUpdate{Location: location}
	updater.draftUpdates[location] = u

	return nil
}

func (updater *ContractUpdaterImpl) isAuthorizedForDeployment(
	signingAccounts []flow.Address,
) bool {
	if updater.RestrictedDeploymentEnabled() {
		return updater.isAuthorized(
			signingAccounts,
			blueprints.ContractDeploymentAuthorizedAddressesPath)
	}
	return true
}
