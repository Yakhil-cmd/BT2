### Title
Unchecked Exit Precompile Return Value in `withdrawToNear` / `withdrawToEthereum` Causes Permanent Token Freeze — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` call `_burn` on the sender's tokens **before** invoking the exit precompile via inline assembly, and **never check the return value** of that call. If the precompile rejects the input (e.g., an invalid or empty NEAR account ID), the EVM sub-call silently returns `0`, the outer transaction does not revert, and the already-burned ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer on NEAR. Any token holder can trigger this by supplying a malformed recipient.

---

### Finding Description

In `EvmErc20.sol`, `withdrawToNear` executes:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // (1) tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // (2) res is captured but NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

The identical pattern exists in `EvmErc20V2.sol`: [2](#0-1) 

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) validates the recipient bytes as a NEAR account ID via `parse_recipient`. If the bytes are empty, contain invalid UTF-8, or do not form a valid NEAR account ID, the precompile returns `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`: [3](#0-2) 

In EVM semantics, a failed sub-`call` returns `res = 0` but does **not** revert the caller's state. Because the assembly block never checks `res` and never executes `revert`, the `_burn` at step (1) is final. The tokens are gone; no NEP-141 transfer is scheduled on NEAR.

The same unchecked-return pattern applies to `withdrawToEthereum` in both contracts: [4](#0-3) 

For `withdrawToEthereum`, passing `address(0)` as `recipient` causes the `ExitToEthereum` precompile to succeed and schedule a withdrawal to the zero address on Ethereum — permanently destroying the bridged value with no revert path. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who calls `withdrawToNear(bytes(""), amount)` (or any bytes that fail NEAR account ID validation) will have their ERC-20 tokens burned with no corresponding NEP-141 release. The tokens are irrecoverable: the ERC-20 total supply decreases, the NEP-141 balance on NEAR is unchanged, and no refund path exists because the EVM transaction succeeded. The same outcome occurs for `withdrawToEthereum(address(0), amount)` — tokens are burned and a withdrawal to the uncontrolled zero address on Ethereum is initiated.

---

### Likelihood Explanation

**Medium.** Every holder of a bridged ERC-20 token (e.g., USDC, USDT, wBTC) can call `withdrawToNear` or `withdrawToEthereum` directly. The `recipient` parameter is `bytes memory` with no on-chain validation before the burn. A user who makes a typo in the NEAR account ID, passes an empty byte array, or passes a recipient that is too long (> 64 chars, rejected by `parse_recipient`) will permanently lose their tokens. The protocol provides no safety net.

---

### Recommendation

1. **Check the precompile return value and revert on failure.** Replace the unchecked assembly block with one that reverts if `res == 0`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Alternatively, move `_burn` after the precompile call** (check-effects-interactions pattern), so that a precompile failure causes the entire transaction to revert before any tokens are destroyed.

3. **Reject `address(0)` as a recipient** in `withdrawToEthereum` to prevent silent loss to the zero address.

---

### Proof of Concept

**Scenario — permanent freeze via invalid NEAR account ID:**

1. Alice holds 1000 USDC (bridged ERC-20) on Aurora.
2. Alice calls `withdrawToNear(bytes(""), 1000e6)`.
3. `_burn(alice, 1000e6)` executes — Alice's balance drops to 0, total supply decreases.
4. The precompile is called with input `\x01 || amount_b || ""`. `parse_recipient(b"")` calls `"".parse::<AccountId>()` which fails. [6](#0-5) 

5. The precompile returns `ExitError`; the EVM sub-call returns `res = 0`.
6. The assembly block does not check `res`; no revert is issued.
7. The outer transaction succeeds. Alice's 1000 USDC are permanently destroyed. No NEP-141 transfer occurs on NEAR. Funds are irrecoverable.

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
}
