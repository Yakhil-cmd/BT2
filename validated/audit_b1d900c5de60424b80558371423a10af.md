### Title
Malicious Participant Can Corrupt CKD Output via Unverified Share Contributions - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator aggregates participant shares without any cryptographic proof of correctness. A malicious participant can send arbitrary `(Y_i, C_i)` values, corrupting the final CKD output and permanently denying the requesting app its confidential key.

### Finding Description
In `do_ckd_coordinator`, the coordinator receives `CKDOutput` pairs from each participant via `recv_from_others` and blindly sums them: [1](#0-0) 

There is no verification that each participant's `(Y_i, C_i)` pair is consistent with their actual key share `x_i`. The correct computation (enforced only locally in `compute_signature_share`) is:
- `Y_i = y_i * G`
- `C_i = x_i * H(pk, app_id) + y_i * A` [2](#0-1) 

No zero-knowledge proof is required that `x_i` is the participant's actual key share, nor that `Y_i` and `C_i` are mutually consistent. This contrasts directly with the DKG protocol, which requires each participant to provide a proof of knowledge of their secret coefficients, verified by all other participants before shares are accepted: [3](#0-2) 

The CKD protocol has no equivalent check. The coordinator simply accumulates whatever group elements each participant sends.

### Impact Explanation
A malicious participant sends arbitrary `(Y', C')` instead of the correct `(λ_i * Y_i, λ_i * C_i)`. The coordinator computes the corrupted aggregate `(Y_honest + Y', C_honest + C')`. The app then computes `sig = (C_honest + C') − a * (Y_honest + Y')`, which is not a valid BLS signature over `H(pk, app_id)`. BLS verification against the MPC network public key `pk` fails, and the app cannot derive its confidential key. This is **corruption of CKD output causing honest parties to accept an unusable cryptographic output** — a High-severity impact under the defined scope.

### Likelihood Explanation
Any single malicious participant in the CKD protocol can trigger this deterministically. No leaked keys, cryptographic breaks, or privileged access are required — only membership in the participant list, which is the baseline attacker capability for this protocol. The attack always succeeds because the coordinator performs no share validation.

### Recommendation
Add a zero-knowledge proof of correctness for each CKD share, analogous to the proof of knowledge used in `do_keyshare`. Each participant should prove knowledge of `(y_i, x_i)` such that `Y_i = y_i * G` and `C_i = x_i * H(pk, app_id) + y_i * A`, where `x_i` is bound to the participant's committed key share from DKG. The coordinator should verify all proofs before summing contributions, mirroring the `verify_proof_of_knowledge` pattern already present in `src/dkg.rs`. [4](#0-3) 

### Proof of Concept
1. Malicious participant `P_m` is a valid member of the CKD participant list.
2. Instead of computing the correct `(λ_m * Y_m, λ_m * C_m)`, `P_m` constructs arbitrary group elements `(Y', C')` (e.g., random points or the identity).
3. `P_m` sends `(Y', C')` to the coordinator via `chan.send_private(waitpoint, coordinator, &(Y', C'))`.
4. The coordinator's loop at lines 50–55 adds `Y'` and `C'` to the running totals without any check.
5. The final output is `CKDOutput::new(Y_honest + Y', C_honest + C')` — a corrupted ciphertext.
6. The app computes `sig = (C_honest + C') − a * (Y_honest + Y')`, which is not `msk * H(pk, app_id)`.
7. BLS verification fails; the app cannot recover its confidential key. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

fn verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    threshold: ReconstructionLowerBound,
    participant: Participant,
    old_participants: Option<ParticipantList>,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: Option<&Signature<C>>,
) -> Result<(), ProtocolError> {
    let threshold = threshold.value();
    match proof_of_knowledge {
        // if participant did not send anything but he is actually an old participant
        None => {
            // if basic dkg or participant is old
            if old_participants.is_none_or(|p| p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since previous line did not abort, then we know participant is new indeed
            // check the commitment length is threshold - 1
            if commitment.coefficients().len() != threshold - 1 {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }
            // nothing to verify
            Ok(())
        }
        // now we know the proof is not none
        Some(proof_of_knowledge) => {
            // if participant sent something but he is actually a new participant
            if old_participants.is_some_and(|p| !p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since the previous did not abort, we know the participant is old or we are dealing with a dkg
            if commitment.coefficients().len() != threshold {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }

            // creating an identifier as required by the syntax of verify_proof_of_knowledge of frost_core
            internal_verify_proof_of_knowledge(
                session_id,
                domain_separator,
                participant,
                commitment,
                proof_of_knowledge,
            )
        }
    }
}
