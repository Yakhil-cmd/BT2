### Title
Missing `is_contract_class_declared` Check for Cairo V0 Classes in `get_compiled_class` When `class_manager_handle` Is `Some` — (`crates/apollo_rpc_execution/src/state_reader.rs`)

---

### Summary

`ExecutionStateReader::get_compiled_class` enforces a `state_number`-aware declaration check for `ContractClass::V1` classes when the class manager path is active, but **unconditionally returns** `RunnableCompiledClass::V0` for `ContractClass::V0` without any such check. A developer TODO comment at the exact branch explicitly acknowledges this omission. An unprivileged user can trigger RPC simulation or `starknet_call` at a historical block number that predates a Cairo V0 class declaration, causing the sequencer to execute the undeclared class and return an authoritative-looking wrong result.

---

### Finding Description

In `get_compiled_class` (lines 115–141), when `self.class_manager_handle` is `Some`: [1](#0-0) 

- The `ContractClass::V1` arm (lines 122–134) calls `is_contract_class_declared(…, self.state_number)` and returns `StateError::UndeclaredClassHash` if the class is not yet declared at the queried `state_number`.
- The `ContractClass::V0` arm (lines 138–140) has **no such check**. It directly returns `Ok(RunnableCompiledClass::V0(…))` regardless of whether the class is declared at `state_number`.

The TODO comment at line 136 explicitly acknowledges the missing guard: [2](#0-1) 

The class manager stores classes by hash with no concept of "declared at block N." A V0 class declared at block 100 is present in the class manager and will be returned by `get_executable` even when `state_number` corresponds to block 50.

The non-class-manager fallback path (`get_contract_class` in `execution_utils.rs`) does **not** have this bug — it uses `get_deprecated_class_definition_at(state_number, class_hash)` which is inherently state-number-scoped: [3](#0-2) 

The asymmetry is only in the class-manager-enabled path.

---

### Impact Explanation

When `class_manager_handle` is `Some` (the production configuration with the class manager service running), any RPC caller can:

1. Issue `starknet_call` or `starknet_simulateTransactions` at a `block_id` that predates a known Cairo V0 class declaration.
2. Include a `library_call` syscall targeting that class hash.
3. Receive execution results computed using the undeclared class — a class that **does not exist in the canonical state at the queried block**.

This produces an authoritative-looking wrong simulation/call result. Fee estimation, tracing, and simulation outputs are all affected. The executed class is concretely wrong: a `RunnableCompiledClass::V0` for a class hash that is undeclared at the queried state.

Impact: **High** — RPC execution/simulation/fee estimation returns a wrong value. Also touches **Critical** — wrong compiled class selected for execution within the simulation context.

---

### Likelihood Explanation

- The class manager is a production component; `class_manager_handle` being `Some` is the normal operating mode.
- Cairo V0 (deprecated) classes are publicly declared on-chain; their class hashes are observable by anyone.
- Historical block queries via `block_id` are a standard, unprivileged RPC feature.
- No special privileges, keys, or sequencer access are required.

---

### Recommendation

Apply the same `is_contract_class_declared` guard to the `ContractClass::V0` arm, mirroring the V1 logic:

```rust
ContractClass::V0(deprecated_contract_class) => {
    let is_declared = is_contract_class_declared(
        &self.storage_reader.begin_ro_txn().map_err(storage_err_to_state_err)?,
        &class_hash,
        self.state_number,
    )
    .map_err(|e| StateError::StateReadError(e.to_string()))?;

    if is_declared {
        Ok(RunnableCompiledClass::V0(deprecated_contract_class.try_into()?))
    } else {
        Err(StateError::UndeclaredClassHash(class_hash))
    }
}
```

Remove the TODO comment once the fix is applied.
