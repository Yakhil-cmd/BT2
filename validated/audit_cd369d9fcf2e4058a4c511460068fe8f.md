### Title
Zero `AllResources` `max_amount` Bypasses All Gateway Stateful Fee Enforcement, Admitting Zero-Balance Invoke Transactions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

An attacker can submit a V3 invoke transaction with `AllResources` resource bounds where all three `max_amount` fields are `GasAmount(0)` (with non-zero `max_price_per_unit` to pass the gateway's L2 gas price threshold check). This causes `enforce_fee()` to return `false`, which propagates as `charge_fee = false` into `ExecutionFlags`. With `charge_fee = false`, `perform_pre_validation_stage` skips both `check_fee_bounds` and `verify_can_pay_committed_bounds`, and `PostValidationReport::verify` returns `Ok(())` immediately. A zero-balance account's invoke transaction passes all gateway stateful validation and is admitted to the mempool. At sequencing time, `new_for_sequencing` re-derives `charge_fee` via the same `enforce_fee()` call, so the transaction is also executed without fee payment.

### Impact Explanation

A zero-balance account can submit an invoke transaction with all three `AllResources` `max_amount` fields set to `GasAmount(0)`. The transaction bypasses `check_fee_bounds`, `verify_can_pay_committed_bounds`, and `PostValidationReport::verify` in the gateway stateful validation path, and is admitted to the mempool. At sequencing time, the same `enforce_fee()` logic causes the transaction to be executed without fee payment. The corrupted admission decision is concrete: a transaction that should be rejected (zero balance, zero fee commitment) is accepted and executed.

---

### Likelihood Explanation

The attack requires only the ability to submit a V3 invoke transaction — a fully unprivileged public operation. The attacker needs to know the `AllResources` transaction format and set `max_amount = 0` for all three resources while keeping `max_price_per_unit` above the gateway's L2 gas price threshold. No special account state, privileged access, or race condition is required.

---

### Recommendation

In `run_validate_entry_point`, do not derive `charge_fee` from `enforce_fee()`. Instead, always set `charge_fee = true` for gateway admission validation, or add an explicit check that rejects `AllResources` transactions where all three `max_amount` values are zero before the `enforce_fee()` call. The gateway should treat a zero-fee-commitment transaction as invalid at the stateless or stateful validation boundary, not silently disable fee enforcement for it.

Alternatively, add a check in `perform_pre_validation_stage` (or in the gateway's `validate_resource_bounds`) that rejects `AllResources` transactions where `max_possible_fee == Fee(0)` with an explicit error, rather than silently skipping fee enforcement.

// Fund with zero balance
let state = &mut test_state(&block_context.chain_info, Fee(0), &[(account, 1)]);

let zero_all_resources = ValidResourceBounds::AllResources(AllResourceBounds {
    l1_gas:      ResourceBounds { max_amount: GasAmount(0), max_price_per_unit: GasPrice(1) },
    l2_gas:      ResourceBounds { max_amount: GasAmount(0), max_price_per_unit: GasPrice(1) },
    l1_data_gas: ResourceBounds { max_amount: GasAmount(0), max_price_per_unit: GasPrice(1) },
});

let tx = executable_invoke_tx(invoke_tx_args! {
    sender_address: account_address,
    resource_bounds: zero_all_resources,
    // charge_fee will be set to false by enforce_fee() in run_validate_entry_point

// enforce_fee() returns false → charge_fee = false
assert!(!tx.create_tx_info().enforce_fee());

let account_tx = AccountTransaction {
    tx: tx.tx.clone(),
    execution_flags: ExecutionFlags {
        only_query: false,
        charge_fee: false, // as set by run_validate_entry_point
        validate: true,
        strict_nonce_check: false,
    },

PostValidationReport::verify(&tx_context, &dummy_receipt, false)

let mut validator = StatefulValidator::create(

assert!(validator.perform_validations(account_tx).is_ok());
