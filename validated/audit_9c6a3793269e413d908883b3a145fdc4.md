### Title
Zero Base Fee Causes Silent Discard of Priority Fees, Depriving Operators of All Fee Revenue - (`basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs`)

### Summary
When `eip1559_basefee == 0`, the `get_gas_price` function unconditionally returns `U256::ZERO`, silently discarding any `max_priority_fee_per_gas` set by the transaction sender. This propagates through the ZK transaction flow so that `token_to_pay_operator = gas_used * 0 = 0`, and the coinbase (operator) receives zero fee even when users explicitly set a non-zero priority fee. This is a direct resource-accounting analog to the Alluvial donate-before-deposit bug: a zero-state condition (`base_fee == 0`) causes a reward/fee calculation to silently return zero, depriving operators of revenue they should receive.

### Finding Description

In `get_gas_price` (`gas_helpers.rs`), when `base_fee` is zero the function short-circuits and returns `U256::ZERO` without inspecting `max_priority_fee_per_gas` at all:

```rust
// If base fee is zero, then we ignore priority fee
if base_fee.is_zero() {
    Ok(U256::ZERO)
}
``` [1](#0-0) 

This `gas_price = 0` value is stored in `TxContextForPreAndPostProcessing::gas_price` and used in two critical places in `ZkTransactionFlow::refund_and_commit_fee`:

1. **Refund calculation** — `token_to_refund = gas_price * (gas_limit - gas_used) = 0`
2. **Operator payment** — `token_to_pay_operator = gas_used * gas_price_for_operator = 0` [2](#0-1) 

The operator balance update is called with `token_to_pay_operator = 0`, so the coinbase receives nothing. [3](#0-2) 

The Ethereum transaction flow (`EthereumTransactionFlow`) handles the same scenario correctly: `get_gas_prices` in `ethereum/validation_impl.rs` computes `effective_gas_price = base_fee + priority_fee_per_gas`, so when `base_fee == 0` the operator still receives `priority_fee_per_gas * gas_used`. [4](#0-3) [5](#0-4) 

This inconsistency between the two transaction flows confirms the ZK path is buggy, not intentionally designed.

### Impact Explanation

**Resource accounting bug / operator revenue loss.** Any ZK (L2-native) transaction submitted when `eip1559_basefee == 0` with a non-zero `max_priority_fee_per_gas` will have its priority fee silently zeroed. The operator (coinbase/sequencer) receives zero fee for processing the transaction. In a chain configuration where `base_fee` is set to 0 (e.g., during initial deployment, subsidized chains, or operator-configured free-fee periods), operators are entirely deprived of priority-fee revenue across all ZK transactions, undermining sequencer economic incentives. The user's balance is not reduced (they also pay nothing), so no user funds are lost — but the operator loses all fee revenue that users intended to pay.

### Likelihood Explanation

`eip1559_basefee` is an operator-controlled block context parameter that can legitimately be set to 0. The test suite explicitly exercises this scenario (`test_gas_price_zero_fee_zero`, `test_gas_price_zero_fee_one`), confirming it is a reachable production state. Any unprivileged transaction sender can trigger the bug simply by submitting a ZK transaction with `max_priority_fee_per_gas > 0` when the block's base fee is 0 — no special access is required. [6](#0-5) 

### Recommendation

When `base_fee == 0`, the gas price for ZK transactions should still include the priority fee, consistent with EIP-1559 semantics and the Ethereum transaction flow:

```rust
if base_fee.is_zero() {
    // Still honour priority fee even when base fee is zero
    let max_priority_fee_per_gas = max_priority_fee_per_gas.unwrap_or(max_fee_per_gas);
    Ok((*max_priority_fee_per_gas).min(*max_fee_per_gas))
} else { ... }
```

This aligns the ZK transaction flow with the Ethereum transaction flow, which correctly computes `effective_gas_price = base_fee + priority_fee_per_gas` regardless of whether `base_fee` is zero.

### Proof of Concept

**Scenario A — ZK transaction (buggy path):**
1. Set `BlockContext { eip1559_basefee: U256::ZERO, .. }`.
2. Submit a ZK (L2-native) transaction with `max_fee_per_gas = 1000`, `max_priority_fee_per_gas = 1000`, `gas_limit = 100_000`.
3. `get_gas_price` hits the `base_fee.is_zero()` branch and returns `U256::ZERO`.
4. `fee_to_prepay = 0 * 100_000 = 0` — user pays nothing.
5. `token_to_pay_operator = gas_used * 0 = 0` — operator receives nothing.
6. Operator balance unchanged despite user setting a 1000-unit priority fee.

**Scenario B — Ethereum transaction (correct path, same inputs):**
1. Same block context (`base_fee = 0`).
2. Submit an Ethereum-format transaction with `max_fee_per_gas = 1000`, `max_priority_fee_per_gas = 1000`.
3. `get_gas_prices` computes `priority_fee_per_gas = min(1000, 1000 - 0) = 1000`, `effective_gas_price = 0 + 1000 = 1000`.
4. Operator receives `1000 * gas_used` — priority fee is correctly paid.

The divergence between the two flows on identical inputs confirms the ZK path silently discards priority fees when `base_fee == 0`. [7](#0-6)

```rust
pub(crate) fn get_gas_price<S: EthereumLikeTypes, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    max_fee_per_gas: &U256,
    max_priority_fee_per_gas: Option<&U256>,
) -> Result<U256, TxError> {
    let base_fee = system.get_eip1559_basefee();
    // If base fee is zero, then we ignore priority fee
    if base_fee.is_zero() {
        Ok(U256::ZERO)
    } else {
        let max_priority_fee_per_gas = max_priority_fee_per_gas.unwrap_or(max_fee_per_gas);
        require!(
            max_priority_fee_per_gas <= max_fee_per_gas,
            TxError::Validation(InvalidTransaction::PriorityFeeGreaterThanMaxFee,),
            system
        )?;
        if !Config::SIMULATION {
            // Skip this check on simulation
            require!(
                &base_fee <= max_fee_per_gas,
                TxError::Validation(InvalidTransaction::BaseFeeGreaterThanMaxFee,),
                system
            )?;
        }
        let priority_fee_per_gas =
            (*max_priority_fee_per_gas).min(max_fee_per_gas.saturating_sub(base_fee));
        // Normally, max_fee_per_gas >= base_fee + priority_fee_per_gas,
        // but we add this min to make it work in simulation too, where we do not
        // enforce max_fee_per_gas > base_fee.
        let gas_price = (base_fee.saturating_add(priority_fee_per_gas)).min(*max_fee_per_gas);
        Ok(gas_price)
    }
}
