### Title
ETH sent to `tick()` is permanently locked when no minting is required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary
`RewardSupplier.tick()` is a public `payable` function that accepts ETH as L1→L2 message fees. When there are no pending mint requests (`amountToMint == 0`), the entire `if` block is skipped and any ETH sent by the caller is permanently locked in the contract. There is no refund path and no withdrawal function anywhere in the contract.

---

### Finding Description

`tick()` is declared `external payable` and uses `msg.value` to pay for two L1→L2 messages inside the `if (amountToMint > 0)` branch: [1](#0-0) 

When `requiredMinting()` returns `amountToMint == 0` (i.e., `messagingContract().l2ToL1Messages(msgHash) == 0` — no pending L2 mint requests), execution falls through the `if` block entirely: [2](#0-1) 

Any ETH in `msg.value` is never forwarded and never refunded. The contract has no `receive()`, `fallback()`, or withdrawal function — confirmed by the full contract surface in `RewardSupplier.sol` and `RewardSupplierStorage.sol`: [3](#0-2) 

The ETH is permanently locked.

A secondary variant exists even when `amountToMint > 0`: the fee split is a hard `msg.value / 2` each way. If either `depositWithMessage` or `sendMessageToL2` requires less than that half, any ETH refunded back to `RewardSupplier` by those calls is also permanently locked for the same reason. [4](#0-3) 

---

### Impact Explanation

Any ETH sent by the caller of `tick()` when no minting is pending is permanently frozen in the `RewardSupplier` contract. There is no admin rescue, no `withdraw`, and no `receive` function. This is a **permanent freezing of funds** matching the allowed High impact: *"Permanent freezing of unclaimed yield or unclaimed royalties; Temporary freezing of funds."*

---

### Likelihood Explanation

`tick()` has no access control — any public caller can invoke it. The condition `amountToMint == 0` is the normal steady-state between reward epochs (most blocks have no pending L2 mint requests). A caller who sends ETH expecting minting to occur (e.g., a keeper bot that always sends a fee budget) will lose their ETH silently whenever the queue is empty. This is a realistic, recurring scenario.

---

### Recommendation

Add a refund of unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    }

    // Refund any unused ETH to the caller
    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok, ) = msg.sender.call{value: remaining}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, require `msg.value == 0` when `amountToMint == 0` by reverting early, or split the function into a view check + execution step so callers only send ETH when minting is actually needed.

---

### Proof of Concept

1. No L2 mint requests are pending (`l2ToL1Messages(msgHash) == 0`).
2. Any address calls `RewardSupplier.tick{value: 1 ether}()`.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. The function returns. The 1 ETH is now held by `RewardSupplier` with no mechanism to recover it.
6. Repeat calls accumulate locked ETH indefinitely. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
