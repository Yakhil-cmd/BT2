### Title
Missing Zero-Address Validation on `reward_address` Causes Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both `change_reward_address` functions in `staking.cairo` and `pool.cairo`, as well as the `stake` entry point, accept `reward_address = 0x0` without any zero-address guard. When a staker or pool member sets their reward address to the zero address, every subsequent call to `claim_rewards` (and `unstake_action`) will revert — because the underlying ERC20 `checked_transfer` to address `0x0` panics — permanently freezing all accrued unclaimed yield in the `RewardSupplier` contract with no protocol-level recovery path.

---

### Finding Description

**Root cause — `staking.cairo::change_reward_address`** (lines 517–531):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // ← NO is_non_zero() check
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

The only guard is `REWARD_ADDRESS_IS_TOKEN`, which passes for `0x0` because the zero address is not a registered token.

**Root cause — `pool.cairo::change_reward_address`** (lines 505–526):

```cairo

    self.write_pool_member_info(:pool_member, :pool_member_info);
    ...

``` [2](#0-1) 

**Root cause — `staking.cairo::stake`** (lines 288–316):

The `stake` entry point also only checks `REWARD_ADDRESS_IS_TOKEN` for `reward_address`, with no zero-address guard, so a staker can register with `reward_address = 0x0` from inception. [3](#0-2) 

**Downstream failure — `send_rewards_to_staker`** (lines 1614–1628):

```cairo
fn send_rewards_to_staker(...) {
    let reward_address = staker_info.reward_address;
    let amount = staker_info.unclaimed_rewards_own;
    claim_from_reward_supplier(:reward_supplier_dispatcher, :amount, :token_dispatcher);
    token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
    // ← panics when reward_address == 0x0
    staker_info.unclaimed_rewards_own = Zero::zero();
    ...
}
``` [4](#0-3) 

`checked_transfer` to `0x0` panics in the STRK ERC-20 (OpenZeppelin Cairo reverts on zero-address recipient). Because the entire transaction reverts, `unclaimed_rewards_own` is never zeroed and the rewards remain permanently inaccessible.

The same pattern applies in `pool.cairo::claim_rewards` (line 366):

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [5](#0-4) 

---

### Impact Explanation

- **`claim_rewards` permanently reverts** for any staker or pool member whose `reward_address` is `0x0`. All accrued STRK yield is frozen inside the `RewardSupplier` contract.
- **`unstake_action` also reverts** because it calls `send_rewards_to_staker` internally, blocking the staker from ever recovering their principal stake as well. [6](#0-5) 

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- Any staker or pool member can trigger this by calling `change_reward_address(0x0)` — a single unprivileged transaction with no preconditions beyond being a registered staker/pool member.
- It can also be triggered at registration time via `stake(reward_address: 0x0, ...)`.
- No privileged role is required; the entry path is fully reachable by an ordinary staker or delegator.
- Accidental misconfiguration (e.g., passing an uninitialized variable) is a realistic scenario.

---

### Recommendation

Add a non-zero guard in all three locations:

1. `staking.cairo::stake` — before writing `reward_address` to storage.
2. `staking.cairo::change_reward_address` — at the top of the function.
3. `pool.cairo::change_reward_address` — at the top of the function.

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

---

### Proof of Concept

**Staker path:**

1. Staker calls `stake(reward_address: 0x0, operational_address: X, amount: MIN_STAKE)`. Passes all existing checks.
2. Staker attests and earns rewards; `unclaimed_rewards_own` grows.
3. Staker (or anyone) calls `claim_rewards(staker_address)`.
4. `send_rewards_to_staker` calls `checked_transfer(recipient: 0x0, amount: rewards)` → ERC-20 panics.
5. Transaction reverts; `unclaimed_rewards_own` is never cleared; rewards are permanently frozen.
6. `unstake_action` is equally blocked, locking the staker's principal as well.

**Pool member path:**

1. Pool member calls `enter_delegation_pool(reward_address: 0x0, amount: X)` (or later `change_reward_address(0x0)`).
2. Pool member earns rewards.
3. `claim_rewards(pool_member)` calls `checked_transfer(recipient: 0x0, ...)` → panics.
4. All accumulated pool rewards are permanently frozen.
