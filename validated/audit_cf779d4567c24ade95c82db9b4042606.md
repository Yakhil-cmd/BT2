### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` as a hard ceiling on pool-level admin spread fees, but `setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with **no analogous cap check**. A pool admin can set per-bin additional fees to the `uint16` maximum (65 535 ≙ 6.5535 % in E6 units) regardless of the factory-configured cap, bypassing the intended admin-fee boundary and causing bad-price execution for every swap routed through that bin.

---

### Finding Description

The factory exposes two paths for a pool admin to influence swap fees:

**Path A — pool-level spread fee (capped):**

```solidity
// MetricOmmPoolFactory.sol L408-435
function setPoolAdminFees(address pool, uint24 newAdminSpreadFeeE6, ...) external onlyPoolAdmin(pool) {
    if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();   // ← cap enforced
    ...
}
```

**Path B — per-bin additional fees (uncapped):**

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
    // ← no cap check whatsoever
}
```

The pool's `setBinAdditionalFees` also performs no cap check:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6) external onlyFactory ... {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;   // ← written unconditionally
    s.addFeeSellE6 = addFeeSellE6;
}
```

Both fees are consumed directly in swap pricing:

```solidity
// MetricOmmPool.sol L540-541
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

`uint16` allows values up to 65 535 (6.5535 % in E6 units). There is no factory-level or pool-level guard that relates this value to `maxAdminSpreadFeeE6`.

---

### Impact Explanation

The factory owner may set `maxAdminSpreadFeeE6` to a low value (e.g., 1 000 = 0.1 %) to protect traders from excessive admin-controlled fees. Despite this, the pool admin can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535`, adding 6.5535 % on top of the pool-level spread fee for every swap in that bin. Traders receive materially worse prices than the protocol's cap permits — a direct loss of user principal paid as excess fees to LPs. Because the pool admin controls which bin is active and can change bin fees instantly (no timelock), the attack is repeatable and requires no external conditions.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is expected to operate only within factory-enforced caps. The bypass requires only a single call to `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535`, which is a valid `uint16` value and will not revert. No special setup, flash loan, or oracle manipulation is needed. Any pool admin who wishes to extract additional value from traders can exploit this immediately.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinFeeE6` cap that the factory owner can configure independently.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 1_000` (0.1 %) via `setFeeCaps`.
2. Pool admin calls `setPoolAdminFees(pool, 1_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries 6.5535 % additional buy and sell fees.
4. A trader swaps through bin 0. The effective fee is `spreadFeeE6 (≤ 0.1 %) + addFeeBuyE6 (6.5535 %) = ~6.65 %`, 66× the intended cap.
