Audit Report

## Title
Unchecked Integer Overflow in `disburse_maturity` Traps SNS Governance Canister for Neurons with Large Maturity — (File: rs/sns/governance/src/governance.rs)

## Summary
The `disburse_maturity` function in SNS governance multiplies `maturity_e8s_equivalent` (a `u64`) by `percentage_to_disburse` (up to 100) using `checked_mul`, then immediately calls `.expect()` on the result. When `maturity_e8s_equivalent > u64::MAX / 100 ≈ 1.84 × 10¹⁷ e8s`, the multiplication overflows, `checked_mul` returns `None`, and `.expect()` panics, trapping the canister. The neuron's maturity is never reduced, so every subsequent disbursement call by that neuron owner also traps, permanently blocking them from converting their earned maturity into SNS tokens.

## Finding Description
At lines 1643–1649 of `rs/sns/governance/src/governance.rs`, the function computes:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)
    .expect("Overflow while processing maturity to disburse.")
    .checked_div(100)
    .expect("Error when processing maturity to disburse.")
    as u128;
``` [1](#0-0) 

The only input validation performed before this computation is a bounds check on `percentage_to_disburse`: [2](#0-1) 

No constraint is placed on `maturity_e8s_equivalent`. When `maturity_e8s_equivalent * percentage_to_disburse > u64::MAX`, `checked_mul` returns `None` and `.expect()` panics, trapping the canister execution. Because the trap rolls back the message without modifying neuron state, the maturity remains at its large value and every future call to `disburse_maturity` for that neuron will also trap.

The reward distribution loop that accumulates maturity uses bare `+=` with no overflow protection: [3](#0-2) 

In Rust release mode (the production build mode for IC canisters), integer `+=` wraps silently on overflow. This means `maturity_e8s_equivalent` can grow past the overflow threshold (`u64::MAX / 100`) without any error being raised during reward distribution, silently setting up the later trap in `disburse_maturity`.

## Impact Explanation
A neuron owner whose `maturity_e8s_equivalent` exceeds `u64::MAX / 100 ≈ 1.84 × 10¹⁷ e8s` is permanently unable to disburse their maturity through the only available disbursement path. Their earned SNS token rewards are effectively frozen. This constitutes a significant SNS-level DoS with concrete, irreversible harm to specific users' governance assets — matching the **High** impact class: *"Significant SNS security impact with concrete user or protocol harm."*

## Likelihood Explanation
An SNS with a large token supply (e.g., 10¹⁰ tokens = 10¹⁸ e8s) and a high initial reward rate (10–100% annually) can cause a neuron holding a large fraction of the stake to accumulate `maturity_e8s_equivalent > 1.84 × 10¹⁷ e8s` within years of normal operation. No attacker action is required — the neuron owner simply participates in governance and accumulates rewards. The overflow threshold corresponds to approximately 1.84 billion tokens of maturity, which is reachable for large-supply SNS deployments. Once the threshold is crossed, the neuron owner is blocked on every retry with no recovery path available to them.

## Recommendation
Replace the `.expect()` calls with proper error propagation so the function returns a `GovernanceError` instead of trapping:

```rust
let maturity_to_deduct = neuron
    .maturity_e8s_equivalent
    .checked_mul(disburse_maturity.percentage_to_disburse as u64)
    .and_then(|v| v.checked_div(100))
    .ok_or_else(|| GovernanceError::new_with_message(
        ErrorType::PreconditionFailed,
        "Arithmetic overflow computing maturity to disburse.",
    ))?;
```

Additionally, the reward distribution loop at line 5994 should use `saturating_add` or `checked_add` with error logging to prevent silent maturity accumulation past safe bounds: [4](#0-3) 

## Proof of Concept
1. Deploy an SNS with a token supply of 10¹⁰ tokens (10¹⁸ e8s) and a high reward rate.
2. Create a neuron holding a large fraction of the stake and vote on all proposals until `maturity_e8s_equivalent` reaches `184_467_440_737_095_517` (= `u64::MAX / 100 + 1`).
3. Call `manage_neuron` with `DisburseMaturity { percentage_to_disburse: 100, to_account: ... }`.
4. The canister traps at line 1646 with `"Overflow while processing maturity to disburse."`, the call fails, and all subsequent calls also fail.

Minimal triggering state: `maturity_e8s_equivalent = 184_467_440_737_095_517`, `percentage_to_disburse = 100`. The multiplication `184_467_440_737_095_517 × 100 = 18_446_744_073_709_551_700 > u64::MAX (18_446_744_073_709_551_615)`, so `checked_mul` returns `None` and `.expect()` panics. [1](#0-0)
