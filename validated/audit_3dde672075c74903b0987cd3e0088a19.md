### Title
Unprivileged Caller Can Suppress All Consensus Rewards by Calling `update_rewards` with `disable_rewards=true` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the staking contract is callable by any address and accepts a caller-controlled `disable_rewards` boolean. When called with `disable_rewards=true`, it advances the global `last_reward_block` checkpoint without distributing any rewards. Because this checkpoint is global and enforced with a strict `>` guard, a single such call per block permanently blocks all legitimate reward distributions for that block. An attacker can repeat this every block to indefinitely suppress consensus rewards for all stakers.

### Finding Description
`update_rewards` in `src/staking/staking.cairo` carries no access-control guard — the spec explicitly states "Any address can execute." [1](#0-0) 

The function first checks a global, non-per-staker `last_reward_block`: [2](#0-1) 

It then unconditionally writes the current block number to that global slot before inspecting `disable_rewards`: [3](#0-2) 

When `disable_rewards=true` the function returns immediately after writing `last_reward_block`, distributing nothing. Because `last_reward_block` is a single contract-wide storage slot (not keyed by staker), any subsequent call to `update_rewards` in the same block — regardless of which staker is named — will revert with `REWARDS_ALREADY_UPDATED`.

The interface confirms no restriction on the caller or on the value of `disable_rewards`: [4](#0-3) 

The spec confirms the same open access: [5](#0-4) 

The analog to the external report is direct: just as BunniZone's `validate()` checked only the *fulfiller* identity while ignoring the *contents* of the order (allowing anyone to trigger arbitrary hooks), the staking contract checks only that a valid staker address is supplied while ignoring the *value* of `disable_rewards`, allowing anyone to consume the per-block reward slot without paying out rewards.

### Impact Explanation
Once consensus rewards are active, every block that an attacker front-runs with `update_rewards(any_active_staker, disable_rewards=true)` is a block in which zero rewards are distributed to any staker or pool. Sustained over many blocks this constitutes a **permanent freezing of unclaimed yield** for all stakers and delegators. The attacker gains nothing financially; the sole effect is denial of yield to the entire staking set.

This matches the allowed impact: **High — Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation
- The function is public with no role check.
- The attacker only needs to know one currently-active staker address (trivially obtained from on-chain events).
- The only cost is gas per block; on Starknet this is low.
- No special knowledge, leaked key, or privileged access is required.
- The attack is fully automatable with a simple bot.

### Recommendation
1. **Short term**: Restrict the `disable_rewards=true` path to a privileged role (e.g., `only_app_governor` or a dedicated keeper role), or split the function into two separate entry points — one public (always distributes) and one restricted (skips distribution).
2. **Long term**: Make `last_reward_block` per-staker rather than global, so one call cannot block all others. Audit every other global-checkpoint pattern for the same class of griefing.

### Proof of Concept
```
// Precondition: consensus rewards are active, staker_address is any active staker.
// Attacker runs this bot every block:

loop {
    staking.update_rewards(
        staker_address: known_active_staker,
        disable_rewards: true,   // attacker-controlled
    );
    // last_reward_block is now current_block.
    // Any honest call to update_rewards in this block reverts with REWARDS_ALREADY_UPDATED.
    wait_for_next_block();
}
```

Step-by-step:
1. Attacker calls `update_rewards(known_staker, disable_rewards=true)` at the start of block N.
2. `last_reward_block` is written to N; the function returns without distributing rewards.
3. Any honest caller (staker, keeper, or protocol) attempts `update_rewards(any_staker, disable_rewards=false)` in block N — it reverts because `current_block_number > last_reward_block` is false.
4. No staker or pool receives rewards for block N.
5. Attacker repeats in block N+1, N+2, … indefinitely. [3](#0-2) [2](#0-1)
