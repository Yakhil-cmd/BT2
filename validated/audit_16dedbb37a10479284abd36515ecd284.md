### Title
Missing Proof of Correct Share Computation in CKD Protocol Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The CKD (Confidential Key Derivation) participant function sends its cryptographic share `(norm_big_y, norm_big_c)` to the coordinator with no zero-knowledge proof that the values were computed correctly using the participant's actual private key share. The coordinator blindly sums all received values. A single malicious participant can send arbitrary group elements, causing the coordinator to accept and output a corrupted, unusable CKD result.

---

### Finding Description

In `do_ckd_participant`, each non-coordinator participant computes its share via `compute_signature_share` and immediately sends the raw output `(norm_big_y, norm_big_c)` to the coordinator over a private channel:

```rust
fn do_ckd_participant(
    mut chan: SharedChannel,
    participants: &ParticipantList,
    coordinator: Participant,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)
}
``` [1](#0-0) 

The coordinator receives these values and sums them without any verification:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [2](#0-1) 

The correct computation that each participant is supposed to perform is:

- `S_i = x_i · H(pk ‖ app_id)` (private share applied to the hash point)
- `C_i = S_i + y_i · app_pk` (ElGamal encryption with fresh randomness `y_i`)
- `Y_i = y_i · G`
- Normalized: `(λ_i · Y_i, λ_i · C_i)` [3](#0-2) 

No ZK proof (e.g., a proof of discrete log equality showing that `norm_big_c` encodes the correct private share contribution) is attached to the message. The protocol provides no mechanism for the coordinator to detect that a participant deviated.

This is the direct analog to the external report's finding: just as `transferFee`/`transferCollateral` silently skip event emission for ETH collateral, `do_ckd_participant` silently skips the proof-of-correct-computation step that should accompany every share submission in a threshold protocol.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A malicious participant sends `(identity, identity)` or any arbitrary `(norm_big_y', norm_big_c')` instead of the correctly computed share. The coordinator sums all contributions and outputs a `CKDOutput` that does not equal `msk · H(pk ‖ app_id)`. The application that calls `ckd_output.unmask(app_sk)` receives a wrong confidential key and has no way to detect the corruption. The derived secret is silently wrong, breaking the correctness guarantee of the CKD protocol for all honest parties.

---

### Likelihood Explanation

**Medium.** Any single participant in the CKD session can trigger this. The participant is already an authenticated member of the threshold group (they hold a valid key share from DKG). No external capability is required beyond participation in a legitimate CKD invocation. The attack is trivially executable: the malicious participant simply substitutes arbitrary `ElementG1` values before calling `chan.send_private`.

---

### Recommendation

Attach a zero-knowledge proof of correct share computation to each participant's message. Specifically, each participant should prove:

1. **Proof of discrete log equality (dlogeq)**: that `norm_big_c` and `norm_big_y` encode the same secret scalar `y_i` (i.e., `norm_big_c - λ_i · S_i = y_i · (λ_i · app_pk)` and `norm_big_y = y_i · (λ_i · G)`).
2. **Proof of correct `S_i`**: that `S_i = x_i · H(pk ‖ app_id)` using the committed public share `λ_i · vk_share_i` from the DKG output.

The coordinator must verify these proofs before accepting and summing any participant's contribution. The existing `dlogeq::prove_with_nonce` / `dlogeq::verify` infrastructure in `src/crypto/proofs/dlogeq.rs` is already available and suitable for this purpose. [4](#0-3) 

---

### Proof of Concept

```
Setup: 3 participants (P1, P2, P3), coordinator = P1, threshold = 2.
       All run DKG to obtain valid key shares x_1, x_2, x_3.

Attack:
  P2 (malicious) intercepts its own do_ckd_participant call.
  Instead of computing (norm_big_y, norm_big_c) correctly,
  P2 sends (ElementG1::identity(), ElementG1::identity()) to P1.

Result:
  P1 (coordinator) computes:
    big_y = λ_1·Y_1 + 0 + λ_3·Y_3   (missing P2's randomness)
    big_c = λ_1·C_1 + 0 + λ_3·C_3   (missing P2's share contribution)

  unmask(app_sk) = big_c - app_sk·big_y
                 = λ_1·x_1·H + λ_3·x_3·H   (≠ msk·H)

  The coordinator outputs a wrong confidential key with no error.
  No proof check exists to detect P2's deviation.
```

The coordinator accepts the corrupted output because `do_ckd_coordinator` performs no verification on received `CKDOutput` values before summing them. [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))

/// Produce a proof for the given statement and witness, using a caller-provided nonce.
/// The nonce `k` must be sampled from a cryptographically secure RNG by the caller.
/// The challenge is derived via the Fiat-Shamir transform over the transcript.
pub fn prove_with_nonce<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    witness: Witness<C>,
    k: Scalar<C>,
) -> Result<Proof<C>, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (big_k_0, big_k_1) = statement.phi(&k);

    // This will never raise error as k is not zero and generator1 is not the identity
    let enc = encode_two_points::<C>(&big_k_0, &big_k_1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    let s = k + e * witness.x.0;
    Ok(Proof {
        e: SerializableScalar::<C>(e),
        s: SerializableScalar::<C>(s),
    })
}
