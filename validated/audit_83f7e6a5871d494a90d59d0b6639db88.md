### Title
Presign Protocol Omits Participant-Set Binding for Beaver Triples, Enabling Presignature Corruption — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA presign protocol validates that the threshold of the supplied Beaver triples matches the signing threshold, but **explicitly omits** verifying that the triples were generated by the same participant set now performing the presign. A malicious participant can supply triples generated under a different participant set, causing the Lagrange-interpolation-based reconstruction inside `do_presign` to operate on mismatched identifiers. The resulting presignature is cryptographically corrupted and unusable, permanently denying signing for all honest parties in that session.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign()` entry-point performs the following validation on the supplied triples:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
```

The comment at lines 38–40 explicitly acknowledges the omission. Only the **threshold scalar** is compared; the **participant set** (the list of `Participant` identifiers whose Lagrange basis was used when the triple shares were distributed) is never checked against the current `participants` argument.

Beaver triples `(a, b, c)` with `c = a·b` are secret-shared using Shamir's scheme over a specific set of participant identifiers. The share held by participant `p` is `f(id_p)` where `id_p = p.scalar::<C>()` is derived from `p`'s identity. During presign, each participant contributes its share and the protocol reconstructs masked values via Lagrange interpolation using the **current** participant identifiers.

If the triples were generated with participant set `P₁ = {A, B, C}` but the presign runs with `P₂ = {A, B, D}`, the Lagrange coefficients computed during presign use `id_D` instead of `id_C`. The interpolation reconstructs a wrong value (not `a`, `b`, or `c`), silently producing a corrupted presignature `PresignOutput` that passes all internal consistency checks but yields an invalid ECDSA signature at the sign step.

---

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept an unusable cryptographic output.**

The corrupted `PresignOutput` is indistinguishable from a valid one at the presign layer (no signature is produced there to verify). All honest parties complete the presign protocol and store the output. When the subsequent `sign` step is executed, the final ECDSA signature is invalid. Because a presignature must never be reused (reuse leaks the private key), the session is permanently lost. Honest parties cannot recover: they must abort, discard the presignature, and restart from triple generation — a full denial of the signing operation for that session.

---

### Likelihood Explanation

**Medium-High.** Any participant who can supply their own `PresignArguments` (i.e., any library caller) controls which triple shares they inject. The attacker only needs to:
1. Participate in a triple-generation session with participant set `P₁`.
2. Initiate or join a presign session with a different participant set `P₂` (same threshold).
3. Supply the `P₁` triples as `args.triple0` / `args.triple1`.

No cryptographic break is required. The omission is self-documented in the source code comment, making it a known gap rather than an accidental oversight.

---

### Recommendation

Bind the triple's participant set to the presign participant set. The `TripleOutput` (or its metadata struct) should record the `ParticipantList` used during generation. Add a check in `presign()`:

```rust
// After constructing `participants`:
if args.triple0.1.participants != participants || args.triple1.1.participants != participants {
    return Err(InitializationError::BadParameters(
        "Triple participant set must match the presign participant set".to_string(),
    ));
}
```

If storing the full participant list in `TripleOutput` is undesirable, store a cryptographic commitment (e.g., a hash of the sorted participant identifiers) and verify it here.

---

### Proof of Concept

1. **Setup:** Three participants `A`, `B`, `C` run triple generation → each holds shares of `(a, b, c)` keyed to identifiers `{id_A, id_B, id_C}`.
2. **Swap:** A new participant `D` joins. A presign session is started with `participants = {A, B, D}`, same threshold `t`.
3. **Inject:** `A` (malicious) supplies its `(a, b, c)` shares from the `{A, B, C}` triple generation as `args.triple0` / `args.triple1`. The only check at lines 43–46 passes because `threshold` matches.
4. **Corrupt:** Inside `do_presign`, Lagrange interpolation uses `id_D` (not `id_C`). The reconstructed masked values are wrong. The protocol completes without error.
5. **Deny:** The `PresignOutput` is stored. When `sign()` is called, the final ECDSA signature fails verification. The presignature cannot be reused. Signing is permanently denied for this session. [1](#0-0)
