### Title
Missing Exact Participant-Count Enforcement in OT-Based ECDSA `presign()` and `sign()` Enables Split-View Secret-Key Extraction — (File: `src/ecdsa/ot_based_ecdsa/presign.rs`, `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `presign()` and `sign()` entry-points accept any participant count `≥ threshold`, but the protocol's own security documentation requires **exactly** `N = 2t + 1` participants for both phases and that `N₁ = N₂`. The robust ECDSA variant correctly enforces this with a strict equality check; the OT-based variant does not. A malicious coordinator can exploit the missing bound to run presign with `N₁ > 2t + 1` participants and then drive two overlapping signing sessions with different `N₂`-sized subsets, obtaining signature shares computed under different Lagrange coefficients and recovering the aggregate secret key.

---

### Finding Description

**Robust ECDSA — correct enforcement (reference):**

`src/ecdsa/robust_ecdsa/presign.rs` lines 75–79 and `src/ecdsa/robust_ecdsa/sign.rs` lines 86–90 both contain:

```rust
if participants.len() != robust_ecdsa_threshold {
    return Err(InitializationError::BadParameters(
        "the number of participants … must be exactly 2*max_malicious+1 …".to_string(),
    ));
}
``` [1](#0-0) [2](#0-1) 

**OT-based ECDSA — missing enforcement (vulnerable):**

`src/ecdsa/ot_based_ecdsa/presign.rs` only checks an upper bound:

```rust
// Spec 1.1
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
``` [3](#0-2) 

`src/ecdsa/ot_based_ecdsa/sign.rs` only checks a lower bound:

```rust
// ensure number of participants during the signing phase is >= threshold
if participants.len() < threshold {
    return Err(InitializationError::NotEnoughParticipantsForThreshold { … });
}
``` [4](#0-3) 

Neither function enforces `participants.len() == 2 * threshold - 1`, and neither enforces `N₁ == N₂`.

**Why the exact count matters — Lagrange linearization in signing:**

`compute_signature_share` in `sign.rs` linearizes the raw presign shares using the *signing* participant set:

```rust
let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
let k_i = lambda * presignature.k;
let sigma_i = lambda * presignature.sigma;
``` [5](#0-4) 

The `RerandomizedPresignOutput` passed to `sign()` stores only raw scalar shares (`big_r`, `k`, `sigma`) with no record of the presign participant set: [6](#0-5) 

Consequently, if a coordinator presents a different `participants` slice to different signers, each signer computes a Lagrange coefficient for a different polynomial evaluation domain. The coordinator collects signature shares `s_i` computed under inconsistent Lagrange bases and can use standard nonce-reuse / split-view algebra to recover the secret key.

The security documentation for the related robust variant explicitly warns (lines 156–158):

> "due to protocol modifications relative to [DJNPO20] (notably signature-share linearization), **a novel split-view attack exists that can extract the secret key using as few as 2t + 2 presigning participants**, with as few as two signing sessions." [7](#0-6) 

The OT-based ECDSA uses the identical linearization step and is therefore subject to the same class of attack.

---

### Impact Explanation

A malicious coordinator can recover the aggregate ECDSA secret key by:

1. Running `presign` with `N₁ > 2t + 1` participants (allowed by the missing upper-bound check).
2. Driving two signing sessions over the same presignature, each with a different `N₂`-sized subset (allowed by the missing equality check and the absence of any participant-set binding in `RerandomizedPresignOutput`).
3. Collecting signature shares computed under different Lagrange bases and solving for the secret.

This maps to the **Critical** allowed impact: *Extraction, reconstruction, or disclosure of private signing shares / aggregate secret material*.

---

### Likelihood Explanation

The coordinator role is a documented, reachable participant in every signing invocation. The `sign()` API accepts an arbitrary `participants` slice supplied by the caller; no cryptographic binding ties that slice to the presign participant set. A coordinator who controls the orchestration layer (a documented trust assumption in the library) can trivially supply inconsistent slices to different signers without any honest party detecting the discrepancy, because each signer only sends its share privately to the coordinator and never sees the other signers' inputs.

---

### Recommendation

1. **Enforce exact participant count in `presign()`** — add a check analogous to the robust variant:
   ```rust
   let required = 2 * args.threshold.value() - 1;
   if participants.len() != required {
       return Err(InitializationError::BadParameters(
           "presign participant count must be exactly 2*threshold-1".to_string(),
       ));
   }
   ```

2. **Enforce exact participant count in `sign()`** — mirror the robust ECDSA guard:
   ```rust
   if participants.len() != 2 * threshold - 1 {
       return Err(InitializationError::BadParameters(
           "sign participant count must be exactly 2*threshold-1".to_string(),
       ));
   }
   ```

3. **Bind the presign participant set to `PresignOutput` / `RerandomizedPresignOutput`** — store a hash or sorted list of the presign participants so that `sign()` can verify `N₁ == N₂` and that the same set is used.

---

### Proof of Concept

**Setup:** threshold `t = 2`, participants `{P1, P2, P3, P4}` (`N = 4 > 2t-1 = 3`).

1. Coordinator calls `presign(&[P1,P2,P3,P4], …, threshold=2)` — accepted because `4 >= 2`. Each `PresignOutput` stores raw `k_i`, `sigma_i`.

2. Coordinator calls `sign(&[P1,P2], coordinator=P1, threshold=2, …, presignature_i)` for `i ∈ {P1,P2}`. Each signer computes `lambda` for the two-party set `{P1,P2}` and returns `s_i^(A)`.

3. Coordinator calls `sign(&[P2,P3], coordinator=P2, threshold=2, …, presignature_i)` for `i ∈ {P2,P3}`. Each signer computes `lambda` for the two-party set `{P2,P3}` and returns `s_i^(B)`.

4. Both calls succeed because `participants.len() (= 2) >= threshold (= 2)` — the missing equality check does not fire.

5. The coordinator now holds `{s_{P1}^(A), s_{P2}^(A)}` and `{s_{P2}^(B), s_{P3}^(B)}`. Since `s_{P2}^(A)` and `s_{P2}^(B)` are computed from the same raw `k_{P2}`, `sigma_{P2}` but with different Lagrange coefficients (one for `{P1,P2}`, one for `{P2,P3}`), the coordinator can set up a linear system in the unknown secret shares and solve for the aggregate secret key — the standard split-view / nonce-reuse recovery.
