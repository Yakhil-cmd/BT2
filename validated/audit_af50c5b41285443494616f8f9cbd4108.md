### Title
Echo Broadcast Delivers With Zero-Vote Threshold at n≤3, Enabling Split-Brain DKG Outputs — (`src/protocol/echo_broadcast.rs`)

### Summary

`echo_ready_thresholds(n)` returns `(0, 0)` for any `n ≤ 3`. Because the delivery condition is `count > 2 * ready_t`, a threshold of `0` means a single self-simulated Ready vote (count = 1 > 0) is sufficient to deliver. A malicious participant can send conflicting `MessageType::Send` values to the two honest parties in a 3-party session; each honest party then independently delivers a different value based solely on its own simulated votes, before any cross-party echo/ready messages are processed. In DKG/reshare/refresh this causes H1 and H2 to derive different `VerifiableSecretSharingCommitment` maps and compute divergent public keys.

---

### Finding Description

**Root cause — `echo_ready_thresholds`:** [1](#0-0) 

For `n = 3`, both `echo_t` and `ready_t` are `0`. The comment explicitly states "no malicious parties are assumed: when n ≤ 3."

**Delivery fires on a single self-vote:**

The Ready-phase delivery check is `data_ready.get(&data) > 2 * ready_t`, i.e., `count > 0`. [2](#0-1) 

When a party processes its own simulated `Send → Echo → Ready` chain, `data_ready[value] = 1` after the first self-Ready. Since `1 > 0`, delivery fires immediately — before any message from another party is received.

**The split-Send attack (n=3, M malicious, H1 and H2 honest):**

M (the broadcaster for session `sid`) sends:
- `MessageType::Send(value_A)` privately to H1
- `MessageType::Send(value_B)` privately to H2

H1's execution in `reliable_broadcast_receive_all`:
1. Receives `Send(value_A)` → echoes `Echo(value_A)`, sets `finish_send = true`, simulates Echo.
2. Simulated `Echo(value_A)`: `data_echo[value_A] = 1 > 0` → sends `Ready(value_A)`, sets `finish_echo = true`, simulates Ready.
3. Simulated `Ready(value_A)`: `data_ready[value_A] = 1 > 0` → amplification + **delivery of `value_A`**, sets `finish_ready = true`.

H2 follows the identical path and **delivers `value_B`**.

By the time H1 receives `Echo(value_B)` from H2, `finish_echo = true` so it is skipped: [3](#0-2) 

By the time H1 receives `Ready(value_B)` from H2, `finish_ready = true` so it is skipped: [4](#0-3) 

The inconsistency is permanent.

**Propagation into DKG:**

`do_broadcast` is called twice inside `do_keyshare`:

1. Line 362 — to agree on session IDs.
2. Line 435 — to agree on `(commitment, proof_of_knowledge)`. [5](#0-4) 

If M sends different `(commitment_A, proof_A)` to H1 and different `(commitment_B, proof_B)` to H2, each honest party's `commitments_and_proofs_map` contains a different entry for M. The commitment-hash binding check in round 1 does **not** prevent this because M also sends the hash via `chan.send_many` (not via broadcast), so M can send matching but distinct hashes to H1 and H2: [6](#0-5) 

H1 verifies `H(commitment_A) == commitment_hash_A` ✓  
H2 verifies `H(commitment_B) == commitment_hash_B` ✓  

Both checks pass. `all_full_commitments` diverges across honest parties, and `public_key_from_commitments` produces different `verifying_key` values. [7](#0-6) 

---

### Impact Explanation

Honest parties complete DKG/reshare/refresh with **different public keys**. Any threshold signature produced by one honest party will be unverifiable by the other. This matches the allowed High impact: *"Corruption of DKG, reshare, refresh outputs so honest parties accept inconsistent public keys."*

---

### Likelihood Explanation

The library's `assert_key_invariants` permits `n = 3, threshold = 2` without any warning that Byzantine fault tolerance is absent at this size. [8](#0-7) 

A 2-of-3 setup is the most common real-world deployment. Any participant in such a session can trivially execute the attack by sending two different `Send` payloads via the private-message API — no cryptographic break required.

---

### Recommendation

1. **Fix `echo_ready_thresholds`**: The Cachin-Guerraoui-Rodrigues protocol requires `n > 3f`. For `n = 3, f = 1` this is violated. Either:
   - Reject `n ≤ 3` in `assert_key_invariants` when Byzantine tolerance is required, or
   - Use a different broadcast primitive (e.g., authenticated echo broadcast, which tolerates `f < n/2` with PKI) for small `n`.

2. **Document the trust assumption**: If `n ≤ 3` is intentionally crash-only (no Byzantine faults), document this prominently and enforce it at the API boundary.

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
