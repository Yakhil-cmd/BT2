### Title
Gateway L2 Gas Price Admission Uses Stale Previous-Block Price, Admitting Transactions That Fail Blockifier Execution — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful admission check compares a transaction's `max_l2_gas_price` against the **previous block's** L2 gas price, while the batcher executes transactions against the **next block's** L2 gas price (computed by EIP-1559). When blocks are consistently full, the next block's price is strictly higher than the previous block's price. Transactions whose `max_l2_gas_price` falls between the two prices pass gateway admission but are rejected by the blockifier's `check_fee_bounds` during actual execution. The inverse holds when blocks are under-utilized: valid transactions are rejected at the gateway even though the next block's price would accept them.

### Finding Description

In `validate_resource_bounds`, the gateway reads the L2 gas price from the **previous** committed block and uses it as the admission threshold. The code itself carries an explicit acknowledgment that this is wrong:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
```

The same stale price propagates into `run_validate_entry_point`, which builds the block context for blockifier pre-validation using the previous block's `BlockInfo` (only the block number is incremented):

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
```

So the gateway's `perform_pre_validation_stage` → `check_fee_bounds` enforces `max_l2_gas_price ≥ P_prev`, while the batcher's `check_fee_bounds` enforces `max_l2_gas_price ≥ P_next`.

The EIP-1559 mechanism in `calculate_next_base_gas_price` adjusts the price upward when `gas_used > gas_target` and downward when `gas_used < gas_target`. The two prices therefore diverge every block, creating a persistent window of mismatch.

Additionally, `validate_tx_l2_gas_price_within_threshold` silently skips the check for `ValidResourceBounds::L1Gas` legacy transactions entirely:

```rust
ValidResourceBounds::L1Gas(_) => {
    // No validation required for legacy transactions.
}
```

This means legacy transactions bypass the L2 gas price admission check altogether, while the blockifier still enforces `check_fee_bounds` on them during execution.

### Impact Explanation

**Admitted-but-failing transactions (full-block scenario):** When `P_next > P_prev`, any transaction with `P_prev ≤ max_l2_gas_price < P_next` passes gateway admission and enters the mempool, but is rejected by `check_fee_bounds` in `perform_pre_validation_stage` during batcher execution. The batcher discards the transaction without including it in the block. An unprivileged sender can deliberately craft such transactions to fill the mempool with entries that consume sequencer validation resources but produce no block output — a gateway-level DoS.

**Rejected-but-valid transactions (under-utilized-block scenario):** When `P_next < P_prev`, a transaction with `P_next ≤ max_l2_gas_price < P_prev` is rejected by the gateway even though the batcher would accept it. Users who correctly price their transactions for the next block are denied admission.

Both outcomes match the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The L2 gas price changes every block under EIP-1559 whenever actual gas usage differs from the target. In a live network with variable load this is the common case, not the exception. The mismatch window is bounded by the per-block price-change cap (`gas_price_max_change_denominator`), but it is non-zero on every block where utilization deviates from the target. No special privilege is required; any user can submit a transaction.

### Recommendation

Replace the stale previous-block price with the **next block's** L2 gas price in both `validate_resource_bounds` and the block context constructed in `run_validate_entry_point`. The next block's price is deterministically computable from the previous block's header fields (`l2_gas_price`, `l2_gas_used`) using the same `calculate_next_base_gas_price` function already present in `crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`. The TODO comment at line 229 already identifies this fix.

### Proof of Concept

1. Observe that the previous block was full: `gas_used > gas_target`. EIP-1559 guarantees `P_next > P_prev`.
2. Compute `P_next` using `calculate_next_base_gas_price(P_prev, gas_used, gas_target, min_gas_price)`.
3. Submit an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P_prev` (i.e., exactly at the previous block's price, which is below `P_next`).
4. `validate_resource_bounds` computes `threshold = P_prev * 100 / 100 = P_prev`; the transaction satisfies `P_prev ≥ P_prev` → **admitted**.
5. `run_validate_entry_point` builds a block context with `P_prev` as the L2 gas price; `check_fee_bounds` passes because `P_prev ≥ P_prev`.
6. Transaction enters the mempool.
7. Batcher builds the next block with `P_next > P_prev`; `check_fee_bounds` fails because `P_prev < P_next` → transaction is rejected and never included.

Repeating step 3 at high rate fills the mempool with transactions that consume gateway and batcher validation cycles but produce zero block throughput.

```rust
    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
```

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
```
