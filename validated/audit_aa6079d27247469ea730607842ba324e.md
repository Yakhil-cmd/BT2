### Title
`CapabilityPath` Not Covered by Signatures in `validateCOAOwnershipProof` Enables Cross-COA Replay Attack - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account signatures are valid over `signedData`, then separately checks that the COA resource at the caller-supplied `CapabilityPath` has the expected EVM address. Because `CapabilityPath` is embedded inside the user-supplied RLP-encoded proof bytes and is **never included in what is signed**, an attacker who obtains a valid proof for one COA can decode the proof, swap the `CapabilityPath` to point to a different COA on the same Cadence account, re-encode it, and pass verification for that second COA — without possessing any key for it.

### Finding Description

`COAOwnershipProof` is an RLP-encoded struct containing `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`.

```
type COAOwnershipProof struct {
    KeyIndices     KeyIndices
    Address        FlowAddress
    CapabilityPath PublicPath
    Signatures     Signatures
}
```

The signatures inside the proof are produced over `signedData` — an **external** byte slice that is passed separately to `validateCOAOwnershipProof`. The `CapabilityPath` field is **not** part of `signedData` and is **not** cryptographically bound to the signatures in any way.

`validateCOAOwnershipProof` performs two independent checks:

1. `keyList.verify(signatureSet: signatureSet, signedData: signedData, ...)` — verifies signatures over `signedData` only.
2. `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path)` — borrows the COA at the caller-supplied `path` and checks its EVM address.

Because these two checks are decoupled, an attacker can:

1. Obtain a legitimately produced proof from Alice for **COA-1** (stored at `/public/coa1`, EVM address `0xA`). Alice's signatures are over some `signedData` blob `D`.
2. RLP-decode the proof, replace `CapabilityPath` from `"coa1"` to `"coa2"`, and RLP-re-encode.
3. Call `verifyCOAOwnershipProof(0xB, D, tampered_proof)` where `0xB` is the EVM address of Alice's **COA-2** (stored at `/public/coa2`).
4. Step 1 of validation passes: Alice's signatures are still valid over `D`.
5. Step 2 of validation passes: the COA at `/public/coa2` has address `0xB`, which matches.
6. The function returns `isValid: true`.

The code itself acknowledges a related but narrower limitation (that `signedData` need not include `evmAddress`), but does **not** acknowledge or mitigate the `CapabilityPath` substitution vector:

```cadence
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
/// In principle, a signature produced for one purpose could be replayed here against
/// a different COA owned by the same Cadence account.
```

The comment attributes the risk only to `evmAddress` omission, while the actual root cause is that **neither** `evmAddress` **nor** `CapabilityPath` is bound to the signatures.

### Impact Explanation

The `isValidSignature` function on the COA EVM contract (ERC-1271) calls the `verifyCOAOwnershipProof` precompile:

```solidity
cadenceArch.staticcall(abi.encodeWithSignature(
    "verifyCOAOwnershipProof(address,bytes32,bytes)",
    address(this), _hash, _sig
));
```

An attacker who intercepts Alice's proof for COA-1 can replay it against COA-2 (a different COA on the same Cadence account), causing `COA-2.isValidSignature(D, tampered_proof)` to return `ValidERC1271Signature`. Any EVM DeFi protocol that relies on ERC-1271 to authorize actions on behalf of COA-2 (e.g., off-chain order signing, permit-style approvals, multisig participation) would accept this forged authorization. The attacker does not need any private key for COA-2.

### Likelihood Explanation

The preconditions are:
- The victim Cadence account must have **two or more COAs** published at different public paths.
- The attacker must observe a valid proof for one of those COAs (e.g., from a public on-chain transaction or an off-chain authentication flow).

Multiple COAs per account are a supported and documented use case (the `CapabilityPath` parameter exists precisely to accommodate this). Off-chain authentication flows built on `validateCOAOwnershipProof` are explicitly mentioned in the code comment as a target use case. Likelihood is **medium**: not every account has multiple COAs today, but the attack surface grows as EVM-on-Flow adoption increases and multi-COA patterns become common.

### Recommendation

Require that `signedData` commits to both `evmAddress` and `CapabilityPath`. The simplest fix is to enforce this inside `validateCOAOwnershipProof` by verifying that `signedData` encodes a canonical struct containing at minimum `evmAddress` and `path`, or by documenting a mandatory encoding convention and rejecting proofs where `signedData` does not include those fields. Alternatively, include `CapabilityPath` and `evmAddress` in the RLP-encoded proof and sign over the entire proof blob rather than a separate `signedData`.

### Proof of Concept

**Setup:**
- Alice's Cadence account `F` has two COAs:
  - COA-1 at `/public/coa1`, EVM address `0xA`
  - COA-2 at `/public/coa2`, EVM address `0xB`

**Step 1 — Alice produces a legitimate proof for COA-1:**
```
signedData = D  (arbitrary 32 bytes)
proof = RLP({
    Address:        F,
    CapabilityPath: "coa1",
    KeyIndices:     [0],
    Signatures:     [sign(D, alice_key_0)]
})
```

**Step 2 — Attacker tampers with the proof:**
```go
decoded := rlp.Decode(proof)   // COAOwnershipProof struct
decoded.CapabilityPath = "coa2"
tampered := rlp.Encode(decoded)
```

**Step 3 — Attacker calls the precompile directly or via COA-2's `isValidSignature`:**
```
verifyCOAOwnershipProof(0xB, D, tampered)
```

**Expected result (current behaviour):**
- `keyList.verify(signedData=D, ...)` → **passes** (Alice's signature over `D` is valid)
- `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coa2)` → **succeeds** (COA-2 exists)
- EVM address of COA-2 == `0xB` → **matches**
- Return value: `isValid: true`

**Violated invariant:** A proof authorised by Alice for COA-1 is accepted as proof of ownership of COA-2, without Alice's knowledge or consent. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
