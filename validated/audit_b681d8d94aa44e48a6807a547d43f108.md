### Title
Stop-loss completely disabled after watermark fully decays to zero, allowing value-draining swaps without triggering `OracleStopLossTriggered` — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary
`_decayed` returns `0` when `ratePerSecondE8 * dt >= E8`. `_applyWatermark` with `hwm = 0` always satisfies `metric >= hwm` (uint256 comparison), returning `(metric, false)` unconditionally. The stop-loss floor check is therefore completely skipped for the first swap after extended pool inactivity, and the watermark is silently reset to the post-drain metric.

### Finding Description
In `_checkAndUpdateWatermarks`, the decayed watermark is computed via `_decayed(hwmS.token0, decayRate, dt)`. When `decayRate * dt >= E8`, `_decayed` returns `0`. This zero is passed to `_applyWatermark` as `hwm`. Inside `_applyWatermark`, the first branch `if (metric >= hwm)` evaluates to `metric >= 0`, which is always true for `uint256`, so the function returns `(metric, false)` — no breach is reported regardless of how low `metric` is. After the swap, `hwmS.token0` and `hwmS.token1` are overwritten with the current (drained) metric values and `lastDecayTs` is reset, permanently erasing the original high watermark.

### Impact Explanation
LP principal loss. An attacker can execute a value-draining swap on a pool that has been inactive long enough for the watermark to fully decay. The stop-loss that LPs rely on to cap drawdown fires no revert, and the watermark is reset to the post-drain level. Subsequent swaps are then checked against the new, lower watermark, so the original protection is permanently lost. This is a direct loss of LP principal above Sherlock High thresholds.

### Likelihood Explanation
Any public trader can trigger this. No privileged role is required. The only precondition is pool inactivity long enough that `decayRate * elapsed >= E8`. At the maximum allowed rate of `E8` per second, this is 1 second. At a typical rate of 58 (≈5%/day), it is ~20 days. Pools with low activity (e.g., long-tail pairs, weekend gaps, or deliberate griefing by withholding swaps) are realistic targets.

### Recommendation
Cap the decay so the watermark never reaches zero, or treat a zero decayed watermark as "no decay has occurred" (i.e., return the original `hwm` when `factor >= E8` instead of `0`). A minimal fix in `_decayed`:

```solidity
// Instead of: if (factor >= E8) return 0;
// Use a floor of 1 to prevent the watermark from fully decaying:
if (factor >= E8) return 1;
```

Alternatively, clamp `factor` to `E8 - 1` so the watermark asymptotically approaches but never reaches zero, preserving the floor check at all times.

### Proof of Concept
1. Deploy a pool with `OracleValueStopLossExtension`, `drawdownE6 = 50_000` (5%), `decayPerSecondE8 = 58`.
2. Execute a normal swap to set `hwmS.token0 = X` and `hwmS.lastDecayTs = T`.
3. Warp time forward by `E8 / 58 + 1 ≈ 1,724,139 seconds` (~20 days).
4. Execute a swap that drains the bin value to near zero.
5. Assert `OracleStopLossTriggered` is **not** emitted/reverted despite the metric being far below `X * (1 - 0.05)`.
6. Read back `hwmS.token0` — it now equals the drained metric, not `X`.
