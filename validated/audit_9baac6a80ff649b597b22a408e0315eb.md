### Title
Malicious Participant Can Corrupt CKD Output by Using a Divergent `public_key` in `compute_signature_share` — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` protocol accepts each participant's `KeygenOutput` (including its `public_key` field) as a caller-supplied parameter with no cross-participant consistency check. Inside `compute_signature_share`, the hash point `H(pk || app_id)` is derived exclusively from the caller's own `key_pair.public_key`. A malicious participant who substitutes a different G2 point for `public_key` computes a different hash point, injecting an inconsistent `big_s` contribution. The coordinator aggregates all contributions blindly, producing a `CKDOutput` whose `unmask()` result is not `msk · H(pk, app_id)` — an invalid, unusable BLS signature.

---

### Finding Description

**Root cause — `compute_signature_share`, line 168:**

```rust
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
```

`key_pair.public_key` is taken verbatim from the `KeygenOutput` the caller passed to `ckd()`. There is no check that it equals the aggregate public key agreed upon during DKG, and no check that it matches what any other participant is using. [1](#0-0) 

**Coordinator aggregation — `do_ckd_coordinator`, lines 50–55:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The coordinator performs a pure additive aggregation with zero verification of the received `(Y_i, C_i)` pairs. There is no commitment, proof of correct computation, or consistency check on the embedded hash point. [2](#0-1) 

**Entry point — `ckd()`, lines 66–117:**

The only guards at the entry point are: minimum participant count, no duplicates, self-membership, and coordinator-membership. There is no validation that `key_pair.public_key` is the correct aggregate public key. [3](#0-2)

### Impact Explanation

Let honest participants use aggregate public key `pk` and the malicious participant use `pk'`. After aggregation:

```
C = Σ λᵢ·Sᵢ + a·Y
  = (Σ_{honest j} λⱼ·xⱼ·H(pk,app_id)) + λ_bad·x_bad·H(pk',app_id) + a·Y
```

After `unmask(app_sk)`:

```
C − a·Y  ≠  msk·H(pk, app_id)
```

The app receives a `CKDOutput` that fails `verify_signature()`. The honest coordinator and all honest participants accept this output as the protocol result — they have no way to detect the corruption. The derived confidential key is unusable. [4](#0-3) 

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The attack requires only that the adversary be a valid member of the participant list — the weakest possible attacker assumption for a threshold protocol. No cryptographic assumption needs to be broken. The attacker simply passes a `KeygenOutput` with a modified `public_key` field when calling `ckd()`. The library exposes `KeygenOutput` as a plain struct with public fields, making this trivially constructible. [5](#0-4) 

---

### Recommendation

The CKD protocol must bind every participant's contribution to the same `public_key`. Options:

1. **Broadcast and verify `public_key` consistency**: Before computing contributions, have all participants broadcast their `key_pair.public_key`; abort if any mismatch is detected.
2. **Pass `public_key` as a separate, externally-agreed parameter**: Remove `public_key` from the per-participant `KeygenOutput` path and instead pass a single, coordinator-supplied (or on-chain-anchored) `VerifyingKey` to `ckd()`, so all participants hash the same `pk` by construction.
3. **Zero-knowledge proof of correct `big_s`**: Require each participant to attach a proof that `big_s = x_i · H(pk || app_id)` for the agreed `pk`, which the coordinator verifies before aggregating.

### Proof of Concept

```rust
// Honest participants use `pk` (the real aggregate public key from DKG).
// Malicious participant substitutes `pk_evil`.

let pk_evil = VerifyingKey::new(G2Projective::generator() * Scalar::random(&mut rng));

let mut protocols: GenProtocol<CKDOutputOption> = Vec::new();
for (i, p) in participants.iter().enumerate() {
    let used_pk = if *p == malicious_participant { pk_evil } else { pk };
    let key_pair = KeygenOutput {
        public_key: used_pk,          // <-- divergent for malicious participant
        private_share: private_shares[i],
    };
    protocols.push((*p, Box::new(ckd(&participants, coordinator, *p,
                                     key_pair, app_id.clone(), app_pk, rng_p).unwrap())));
}

let result = run_protocol(protocols).unwrap();
let ckd_output = check_one_coordinator_output(result, coordinator).unwrap();

let confidential_key = ckd_output.unmask(app_sk);
let expected = hash_app_id_with_pk(&pk, &app_id) * msk;

// This assertion FAILS — the output is corrupted.
assert_eq!(confidential_key, expected, "Keys should be equal");
// verify_signature also fails:
assert!(verify_signature(&pk, &app_id, &confidential_key).is_err());
```
