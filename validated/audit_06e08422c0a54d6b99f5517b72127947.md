### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
`RewardSupplier.tick()` is `external payable` and accepts ETH to cover L1→L2 messaging fees. When `amountToMint == 0` (no pending mint requests), the function exits the `if` block without using or refunding `msg.value`, permanently locking any ETH sent by the caller in the contract. Even when `amountToMint > 0`, any ETH in excess of the actual fees consumed by the bridge and messaging contract is not returned to the caller.

### Finding Description
In `RewardSupplier.sol`, `tick()` is declared `external payable`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // If amountToMint == 0: msg.value is silently consumed by the contract
}
``` [1](#0-0) 

Two distinct loss paths exist:

1. **Zero-mint path**: If `amountToMint == 0`, the entire `msg.value` is never forwarded and never refunded. The ETH is permanently locked in the `RewardSupplier` contract. No `withdraw`, `receive`, or `fallback` function is present in the contract to recover it.

2. **Excess-fee path**: When `amountToMint > 0`, the fee is mechanically split as `msg.value / 2` to the bridge and `msg.value - msg.value/2` to the messaging contract. [2](#0-1)  If the actual fees required by those contracts are less than the amounts forwarded, the surplus is absorbed by those external contracts — not returned to the caller.

### Impact Explanation
Any ETH sent to `tick()` when `amountToMint == 0` is permanently locked in the `RewardSupplier` contract with no recovery path. This constitutes a direct, irreversible loss of the caller's funds. The contract has no `withdraw` function, no `receive` fallback, and no admin rescue mechanism visible in the codebase.

This maps to **Critical: Direct theft/permanent loss of user funds** (the ETH is irrecoverably locked) or at minimum **High: Permanent freezing of funds**.

### Likelihood Explanation
`tick()` is `external` — any unprivileged address can call it. A realistic scenario:

1. A keeper/bot monitors pending mint requests and calls `tick()` with ETH.
2. A front-runner (or a race condition between two keepers) calls `tick()` first, consuming all pending L2→L1 mint-request messages.
3. The original caller's transaction executes: `requiredMinting()` returns `(0, 0)`, `amountToMint == 0`, and their ETH is permanently locked.

This is a realistic griefing vector requiring no privileged access.

### Recommendation
Add a refund of unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ... existing logic ...
    }
    // Refund any unused ETH
    if (address(this).balance > 0) {
        (bool ok, ) = msg.sender.call{value: address(this).balance}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, require `msg.value == 0` when `amountToMint == 0` (strict equality check), or split the fee calculation only after confirming minting is needed.

### Proof of Concept
1. Deploy `RewardSupplier` with no pending L2→L1 mint-request messages (or consume them via a front-run).
2. Call `tick{value: 1 ether}()` from any EOA.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped.
5. The function returns. The caller's 1 ETH is now locked in `RewardSupplier` with no recovery path. [1](#0-0)
