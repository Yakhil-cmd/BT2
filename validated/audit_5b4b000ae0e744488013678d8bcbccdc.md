### Title
Non-Whitelisted ERC-20 Recipient Silently Redirected to Owner-Controlled Fallback Address, Enabling Custodial Capture of Inbound Bridge Transfers - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, `receive_erc20_tokens` silently replaces the intended ERC-20 recipient with an owner-configured fallback address whenever the recipient is absent from the Address whitelist. Because the owner controls both the fallback address and the whitelist membership, the owner can deterministically divert any inbound NEP-141 bridge transfer to their own EVM address without reverting the transaction or notifying the sender.

---

### Finding Description

`Engine::receive_erc20_tokens` in `engine/src/engine.rs` is the inbound bridge path for all NEP-141 → ERC-20 token transfers. After parsing the intended recipient from the caller-supplied `msg` field, the function performs the following check:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which returns `true` only when the Address whitelist is disabled **or** the address is explicitly present in it. If the whitelist is enabled and the recipient is absent, `recipient` is silently overwritten with `fallback_address`, and the full token amount is minted to that address instead.

Both control surfaces are exclusively owner-gated:

- `set_erc20_fallback_address` (`engine/src/lib.rs` line 806) requires `require_owner_and_running`.
- `set_whitelist_status` / `add_entry_to_whitelist` / `remove_entry_from_whitelist` (same file, lines 842–908) all require `require_owner_and_running`.

The owner can therefore:
1. Call `set_erc20_fallback_address` pointing to an EVM address they control.
2. Enable the Address whitelist via `set_whitelist_status`.
3. Omit or remove any target address from the whitelist.
4. All subsequent `ft_transfer_call` → `ft_on_transfer` → `receive_erc20_tokens` flows for that address silently mint tokens to the owner's fallback address.

The sender receives no revert and no on-chain indication that the recipient was changed; the NEP-141 transfer succeeds and the tokens are gone.

Note that `receive_base_tokens` (ETH bridging path, lines 773–790) does **not** apply this redirect — the vulnerability is isolated to the ERC-20 path.

---

### Impact Explanation

**Critical — Direct theft of user funds in motion.**

Any NEP-141 token holder who bridges tokens to a non-whitelisted EVM address on Aurora loses the full bridged amount to the fallback address. The diversion is unconditional once the predicate fires; there is no slippage guard, no partial delivery, and no revert path. The sender's NEP-141 balance is debited in full, and the ERC-20 tokens are minted to the owner's address.

---

### Likelihood Explanation

**Medium.** The mechanism requires the owner to act adversarially, but no additional exploit primitive is needed — the capability is built directly into the production contract. The owner can activate it at any time by calling two permissioned functions. Users have no on-chain way to detect or prevent the redirection before submitting their transfer.

---

### Recommendation

Replace the silent redirect with a hard revert when the recipient is not whitelisted:

```rust
if let Some(_fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    return Err(errors::ERR_RECIPIENT_NOT_WHITELISTED.into());
}
```

If redirection is intentional for compliance, emit an explicit on-chain event identifying the original recipient and the redirected address, and document the behavior prominently so users can verify the recipient before bridging.

---

### Proof of Concept

**Setup (owner actions):**
1. Owner calls `set_erc20_fallback_address({ address: Some(<owner_evm_addr>) })` — sets the diversion target.
2. Owner calls `set_whitelist_status({ kind: WhitelistKind::Address, active: true })` — enables the Address whitelist.
3. Owner does **not** add victim's EVM address to the whitelist (or calls `remove_entry_from_whitelist` to remove it).

**Exploit (unprivileged user action):**
4. Victim holds NEP-141 tokens and calls `ft_transfer_call` on the NEP-141 contract:
   - `receiver_id`: Aurora Engine account (`aurora`)
   - `amount`: e.g. `1_000_000`
   - `msg`: hex-encoded victim EVM address (40 chars)
5. NEP-141 contract calls `ft_on_transfer` on Aurora Engine.
6. `ft_on_transfer` → `receive_erc20_tokens` fires. `get_erc20_fallback_address` returns `owner_evm_addr`; `is_allow_receive_erc20_tokens` returns `false` for victim's address.
7. `recipient` is overwritten with `owner_evm_addr`; `setup_receive_erc20_tokens_input` encodes a `mint(owner_evm_addr, 1_000_000)` call.
8. The ERC-20 `mint` executes successfully; victim's NEP-141 balance is debited; owner's ERC-20 balance increases by `1_000_000`. Victim receives nothing.
