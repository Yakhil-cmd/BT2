### Title
Empty-payload message (exactly 40 bytes) from a malicious participant causes `DeserializationError` and permanently aborts honest party's protocol — (`src/protocol/internal.rs`)

### Summary
A malicious participant who knows the deterministic shared-channel tag and the sequential waitpoint (both computable from public constants) can send a 40-byte message (valid header, zero-byte payload) that passes `push_message`'s length guard, is buffered, and when popped by `Comms::recv` causes `rmp_serde::decode::from_slice(&[])` to fail, propagating a `ProtocolError` that aborts DKG, reshare, refresh, or signing for the honest party.

### Finding Description

`push_message` guards with `message.len() < MessageHeader::LEN` (strict less-than), so a message of exactly 40 bytes passes. `MessageHeader::from_bytes` succeeds on a 40-byte slice because `split_at_checked(40)` returns `Some((&bytes[..40], &[]))`. The message is stored in `MessageBuffer`.

When `Comms::recv` pops it, `data.get(MessageHeader::LEN..)` returns `Some(&[])` — an empty slice, not `None` — so the `ok_or_else` guard does not trigger. `rmp_serde::decode::from_slice(&[])` then fails for any non-unit type `T`. The error is converted to `ProtocolError::Other(...)` via the blanket `From<Box<dyn error::Error + Send + Sync>>` impl and propagates.

Callers such as `recv_from_others` use `?` unconditionally, so the error immediately aborts the protocol. The shared channel tag is a deterministic SHA-256 constant; waitpoints are sequential from 0. Both are computable by any participant before the protocol starts.

### Impact Explanation
Any legitimate participant can permanently abort DKG, reshare, refresh, presign, or signing for one or more honest parties by sending a single 40-byte message at the start of a protocol round, before legitimate messages are delivered. Because the `MessageBuffer` is FIFO per header, the malformed message is consumed first, the honest party errors out, and the protocol cannot complete.

### Likelihood Explanation
The attacker only needs to be a registered participant (no key material required). The channel tag and waitpoint are fully deterministic and public. The attack requires sending one crafted message at the right time — straightforward for any participant who controls their own network timing.

### Recommendation
1. **Tighten the length guard** in `push_message` to `message.len() <= MessageHeader::LEN` (strict greater-than for payload), rejecting zero-payload messages at ingress.
2. **Make `Comms::recv` retry** on deserialization failure (skip and re-await) rather than propagating the error, consistent with how `reliable_broadcast_receive_all` already handles it.
3. **Validate the sender** in `SharedChannel::recv` against the expected participant set before attempting deserialization, so messages from unexpected senders are discarded.
