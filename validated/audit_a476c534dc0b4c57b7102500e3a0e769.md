### Title
Signature Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions for Undeployed Accounts to Enter Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry point for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. This check passes for **any** transaction from the account in the mempool — not specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` can submit an invoke with nonce=1 from the victim's address with an invalid signature. Because the victim's `deploy_account` is in the pool, `__validate__` is skipped and the unsigned invoke is admitted to the mempool without signature verification.

### Finding Description

`skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`) implements a UX feature allowing users to submit `deploy_account` + `invoke` simultaneously. When an invoke has `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`, it calls:

```rust
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())
```

If that returns `true`, `skip_validate = true` is returned, and `run_validate_entry_point` is called with `execution_flags.validate = false`, causing `StatefulValidator::perform_validations` to return `Ok(())` immediately without calling `__validate__`. [1](#0-0) 

The underlying mempool check is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` if **any** transaction from the address is in the pool — it does not distinguish `deploy_account` from `invoke`. The comment in `skip_stateful_validations` claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect: an attacker can place a transaction in the pool for the victim's address to trigger the skip.

The gateway's `validate_by_mempool` (called before `skip_stateful_validations`) delegates to the mempool's `validate_tx`:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [3](#0-2) 

`validate_incoming_tx` only checks for duplicate tx_hash and that `tx_nonce >= account_nonce`. `ValidationArgs` carries no signature field: [4](#0-3) 

No signature check exists anywhere in the mempool admission path. The `run_validate_entry_point` path in `StatefulValidator::perform_validations` short-circuits before calling `__validate__` when `execution_flags.validate == false`: [5](#0-4) 

**Attack path:**

1. Victim submits `deploy_account` (nonce=0) → enters mempool; `tx_pool.contains_account(victim_addr)` becomes `true`.
2. Attacker observes the `deploy_account`, computes `victim_addr` via `calculate_contract_address(salt, class_hash, constructor_calldata, DEPLOYER=0)`.
3. Attacker submits `invoke` with `sender_address=victim_addr`, `nonce=1`, `signature=[]` (empty/invalid).
4. Gateway stateful validation:
   - `validate_nonce`: `account_nonce=0`, `tx_nonce=1` → `0 ≤ 1 ≤ 200` → passes.
   - `validate_by_mempool`: nonce not too old, no duplicate hash → passes.
   - `skip_stateful_validations`: `account_tx_in_pool_or_recent_block(victim_addr)` → `true` → returns `true`.
   - `run_validate_entry_point(skip_validate=true)` → `__validate__` **not called**.
5. Attacker's unsigned invoke is added to the mempool.
6. Victim submits their legitimate invoke with nonce=1 → rejected with `DuplicateNonce`. [6](#0-5) 

### Impact Explanation

Matches **High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

- An unsigned (or arbitrarily signed) invoke transaction for any undeployed account with a pending `deploy_account` is admitted to the mempool without signature verification.
- The attacker occupies the nonce=1 slot, blocking the victim's legitimate first invoke (`DuplicateNonce`).
- If fee escalation is enabled (`enable_fee_escalation: true`), the attacker can also **replace** the victim's already-queued legitimate invoke by submitting a higher-fee unsigned invoke — again bypassing `__validate__` because `account_tx_in_pool_or_recent_block` still returns `true`.
- The attacker's invoke fails at blockifier execution time (when `__validate__` runs with `validate=true`), but the damage — blocking or replacing the victim's invoke — is already done at admission time.

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool.
- The victim's contract address is deterministic and computable from the public `deploy_account` fields (`class_hash`, `contract_address_salt`, `constructor_calldata`).
- The attacker only needs to react faster than the victim's second transaction — a straightforward mempool-watching race.
- No privileged access, special keys, or on-chain state is required.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender address is pending in the mempool. Expose a dedicated mempool query such as `deploy_account_tx_in_pool(address) -> bool` that inspects transaction type, and use that in `skip_stateful_validations` instead of the generic presence check.
