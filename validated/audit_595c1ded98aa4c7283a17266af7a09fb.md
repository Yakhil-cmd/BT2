Audit Report

## Title
Type-3 Node Provider Reward Over-Minting via Attacker-Controlled Lexicographic Ordering of Node Operator Registry Keys - (File: rs/registry/node_provider_rewards/src/lib.rs)

## Summary
`calculate_rewards_v0` applies a shared, order-dependent decay coefficient (`np_coefficients`) to type-3 node rewards in the lexicographic order of node operator registry keys. Because the registry key for each node operator is derived directly from the node operator's principal ID — which the node provider freely chooses when submitting an `AddNodeOperator` governance proposal — a node provider can generate principal IDs whose lexicographic ordering places their highest-rate data center first, receiving the undecayed coefficient (1.0) on the most valuable nodes and causing the NNS to mint more ICP than the reward schedule intends.

## Finding Description
In `calculate_rewards_v0`, the outer loop iterates over `node_operators` in the order supplied by the caller. Both call sites — `rs/registry/canister/src/get_node_providers_monthly_xdr_rewards.rs` (line 35–40) and `rs/node_rewards/canister/src/canister/mod.rs` (line 295–299) — collect node operator records via `get_key_family_iter_at_version` / `get_key_family_with_values` using the prefix `NODE_OPERATOR_RECORD_KEY_PREFIX = "node_operator_record_"`. These functions return records in lexicographic key order. The registry key for each node operator is constructed as `make_node_operator_record_key(node_operator_principal_id)`, i.e., `"node_operator_record_" + principal_id.to_string()`. The `node_operator_principal_id` is a free field in `AddNodeOperatorPayload` that the proposer specifies.

Inside the loop, for every `type3`-prefixed node type, the code reads the current `np_coeff` from the shared `np_coefficients` map (keyed by `node_provider_id + continent + country`), applies it to compute the node reward, then multiplies it by `dc_reward_coefficient_percent` and writes it back. The first operator processed for a given NP+country key always starts at coefficient 1.0; each subsequent operator starts at the decayed value left by the previous one. The code comment at lines 85–99 explicitly acknowledges this as a known ordering issue.

A node provider controlling two node operators in the same country can generate key pairs (self-authenticating principals) whose text representations sort in any desired order. By choosing a principal ID for the high-rate DC that sorts lexicographically before the principal ID for the low-rate DC, the high-rate DC is always processed first at coefficient 1.0. Generating principal IDs with a desired relative sort order requires only iterating over key pairs until the desired ordering is found — a trivial offline computation. The resulting `AddNodeOperator` proposals are indistinguishable from legitimate ones; NNS voters have no signal to detect the manipulation.

## Impact Explanation
The NNS mints more ICP to the node provider each month than the reward table and decentralization-incentive mechanism intend. The decay coefficient exists specifically to discourage concentration of type-3 nodes with a single provider; gaming the ordering neutralizes this mechanism for the highest-value nodes. This is a concrete, repeatable illegal-minting impact against the NNS reward accounting system. It matches the allowed High impact: "Significant NNS security impact with concrete user or protocol harm." The per-provider monthly excess is bounded by the number of nodes and rate differentials (the PoC demonstrates ~19% excess), but it is permanent, compounding, and requires no ongoing cost after the initial key selection.

## Likelihood Explanation
Any registered node provider can submit `AddNodeOperator` proposals. Generating principal IDs with a desired lexicographic ordering is a trivial offline computation (generate key pairs, compare their text representations, keep the desired pair). The proposals look entirely legitimate to NNS voters. The attack is low-cost, requires no privileged access beyond being a registered node provider, is repeatable every reward cycle, and the code comment at lines 85–99 confirms the developers are aware of the ordering dependency but have not treated it as a security issue.

## Recommendation
Replace the order-dependent shared coefficient accumulation with an order-independent calculation. For each NP+country group, collect all `(rate, coefficient, node_count)` tuples first, then sort them by rate descending before applying the decay sequence — exactly as the V2 performance-based algorithm already does in `rs/node_rewards/rewards_calculation/src/performance_based_algorithm/mod.rs` at lines 482–483. Alternatively, compute the average rate across all DCs in the same country before applying decay (as V1 does), which is also order-independent.

## Proof of Concept
Write a unit test against `calculate_rewards_v0` directly (no governance required):

1. Create a single node provider NP with two node operators in the same country (e.g., `"Europe,CH"`):
   - Operator key `"aaa..."` → DC with type3 rate = 30,000 XDR, coefficient = 70%
   - Operator key `"zzz..."` → DC with type3 rate = 10,000 XDR, coefficient = 70%
2. Call `calculate_rewards_v0` and record total rewards → expect 37,000 XDR (high-rate first).
3. Swap the keys so `"aaa..."` maps to the low-rate DC and `"zzz..."` maps to the high-rate DC.
4. Call `calculate_rewards_v0` again → expect 31,000 XDR (low-rate first).
5. Assert the two totals differ, confirming the 19% reward difference is solely a function of key ordering.

To demonstrate the real-world path: generate two Ed25519 key pairs offline, derive their self-authenticating principal IDs, verify their text representations sort in the desired order, and submit two `AddNodeOperator` proposals to a local replica assigning each principal to the desired DC. The monthly reward calculation will then consistently favor the attacker's chosen ordering.
