### Title
`ft_on_transfer` Mints ERC-20 Tokens Without Validating Claimed Amount Against Actual NEP-141 Balance — (`engine/src/engine.rs`)

### Summary

The Aurora Engine's `ft_on_transfer` callback mints ERC-20 tokens (or credits ETH) based solely on the caller-supplied `amount` field in `FtOnTransferArgs`, without ever verifying that the engine's actual NEP-141 token balance increased by that amount. Any account registered as a NEP-141 token in the engine can call `ft_on_transfer` directly with an arbitrarily large fabricated `amount`, causing the engine to mint unbacked ERC-20 tokens.

### Finding Description

`ft_on_transfer` is a public WASM export on the Aurora Engine contract. In NEAR Protocol, any account can call any public function on any contract. The function dispatches to one of two minting paths based on `predecessor_account_id`: [1](#0-0) 

For the ERC-20 path, `receive_erc20_tokens` is called: [2](#0-1) 

Inside `receive_erc20_tokens`, the engine takes `args.amount.as_u128()` directly from the callback argument and uses it to call the ERC-20 contract's mint function via `setup_receive_erc20_tokens_input`: [3](#0-2) 

At no point does the engine read its own NEP-141 balance before and after the call to confirm the balance actually increased by `amount`. The `amount` field is fully attacker-controlled.

The only guard is `get_erc20_from_nep141(&self.io, token)?`, which requires `predecessor_account_id` to be a registered NEP-141 token. `deploy_erc20_token` is a public function callable by any account, allowing an attacker to register a malicious NEP-141 contract they control. Once registered, the attacker calls `ft_on_transfer` directly from that account with any `amount`. [4](#0-3) 

The same structural absence of balance verification exists in `receive_base_tokens`, though that path requires `predecessor_account_id == connector_account_id` (admin-controlled), so it is not directly reachable by an unprivileged attacker. [5](#0-4) 

### Impact Explanation

An attacker can mint an unbounded quantity of ERC-20 tokens (representing bridged NEP-141 assets) without depositing any real tokens. These minted tokens are indistinguishable from legitimately bridged tokens and can be transferred, sold, or used within the EVM. This directly inflates the ERC-20 supply beyond the actual NEP-141 collateral held by the engine, causing **insolvency** and enabling **direct theft** of other users' bridged assets.

**Impact: Critical** — insolvency and direct theft of user funds.

### Likelihood Explanation

`ft_on_transfer` is a public WASM export reachable by any NEAR account. `deploy_erc20_token` appears to be a permissionless public function (no access-control guard is visible in the dispatch path, and tests invoke it without special roles). An attacker needs only to: (1) deploy a minimal NEAR contract that satisfies the NEP-141 metadata interface, (2) call `deploy_erc20_token` to register it, and (3) call `ft_on_transfer` directly. No privileged keys, governance capture, or oracle manipulation is required.

**Likelihood: High.**

### Recommendation

1. **Snapshot the engine's NEP-141 balance before processing `ft_on_transfer`** and compare it to the balance after. Only mint tokens equal to the actual observed increase, not the caller-supplied `amount`.
2. Alternatively, **reject direct calls to `ft_on_transfer`** that do not originate from a known, trusted NEP-141 contract (e.g., by cross-checking `predecessor_account_id` against a strict allowlist of registered tokens and requiring the call to arrive as a genuine NEP-141 `ft_transfer_call` callback, which NEAR enforces at the receipt level).
3. Add an explicit assertion: `actual_balance_after - actual_balance_before >= args.amount` before minting.

### Proof of Concept

```
1. Attacker deploys `evil.near` implementing NEP-141 metadata (name/symbol/decimals).
2. Attacker calls aurora_engine.deploy_erc20_token("evil.near")
   → Engine registers evil.near → ERC-20 token at address X.
3. Attacker calls aurora_engine.ft_on_transfer(
       sender_id: "evil.near",
       amount:    "1000000000000000000000",   // fabricated, no tokens transferred
       msg:       "<attacker_evm_address_hex>"
   )  from the evil.near account.
4. Engine executes receive_erc20_tokens("evil.near", amount=1e21, recipient=attacker_evm_addr).
   → get_erc20_from_nep141 succeeds (evil.near is registered).
   → Engine calls ERC-20.mint(attacker_evm_addr, 1e21) as admin.
5. Attacker now holds 1e21 unbacked ERC-20 tokens.
   → Swap/withdraw drains real user deposits → insolvency.
``` [6](#0-5) [7](#0-6)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_base_tokens(
        &mut self,
        args: &FtOnTransferArgs,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
        let amount = Wei::new_u128(args.amount.as_u128());
        let receipient = message_data.recipient;
        let balance = get_balance(&self.io, &receipient);
        let new_balance = balance
            .checked_add(amount)
            .ok_or(errors::ERR_BALANCE_OVERFLOW)?;

        set_balance(&mut self.io, &receipient, &new_balance);

        sdk::log!("Mint {amount} base tokens for: {}", receipient.encode());

        Ok(None)
    }

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
