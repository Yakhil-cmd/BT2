The critical code to analyze is `submit_promise_resume_data` in `ext.rs` and the `has_promise_yield_status` helper in `core/store/src/utils/mod.rs`.

**Guard logic in `submit_promise_resume_data`:** [1](#0-0) 

The guard passes if **either** `has_promise_yield_receipt` **or** `has_promise_yield_status` returns `true`.

**`has_promise_yield_status` checks key existence only — not the value:** [2](#0-1) 

It calls `trie.contains_key(...)` — it returns `true` for **any** `PromiseYieldStatus` variant stored at that key, including `ResumeInitiated`.

**What happens on the first call:** [3](#0-2) 

`set_promise_yield_status` overwrites `Yielded` with `ResumeInitiated` — but the key **remains present** in the `TrieUpdate`.

**What happens on the second call with the same `data_id`:**

- `has_promise_yield_receipt` → `true`: the `TrieKey::PromiseYieldReceipt` entry is **not removed** by `submit_promise_resume_data`; it is only removed later when the runtime processes the DataReceipt. [4](#0-3) 
- `has_promise_yield_status` → `true`: the key still exists with value `ResumeInitiated`. [2](#0-1) 

Both conditions are `true`, so the `if` branch is entered again, `create_promise_resume_receipt` is called a second time, and a second `DataReceipt` is enqueued for the same `data_id`. The second call returns `Ok(true)` instead of `Ok(false)`.

**Attacker path:** A contract calls `promise_yield_create` to obtain a `data_id`, stores it, then in a subsequent function call invokes `promise_yield_resume` (the host function that calls `submit_promise_resume_data`) twice with the same `data_id` and different payloads. Both calls succeed within the same `RuntimeExt` execution context.

**Invariant broken:** "each yield is resumed exactly once." Two `DataReceipt`s are produced for the same yield. The callback receives whichever arrives first; the second payload is stored as orphaned `ReceivedData` in the trie (a permanent state leak). If both DataReceipts land in the same chunk, the runtime processes the first, removes the yield receipt, and the second DataReceipt's data is written to trie storage with no matching pending receipt — corrupting the promise result delivery and leaking trie storage.

**Root cause:** The guard should check the *value* of the status (specifically `PromiseYieldStatus::Yielded`), not merely the *existence* of the key. The correct check would use `get_promise_yield_status` and match on `Some(PromiseYieldStatus::Yielded)`. [5](#0-4)

### Title
Double-resume of the same yield `data_id` creates duplicate `DataReceipt`, violating the exactly-once delivery invariant — (`runtime/runtime/src/ext.rs`)

### Summary
`submit_promise_resume_data` guards against duplicate resumes by checking whether a `PromiseYieldStatus` key exists in the trie. After the first resume, the key is overwritten with `ResumeInitiated` but **not removed**, so the existence check passes again on a second call, causing a second `DataReceipt` to be enqueued for the same `data_id`.

### Finding Description
`has_promise_yield_status` is a pure key-existence check (`trie.contains_key`). It returns `true` for any stored variant, including `ResumeInitiated`. After the first call to `submit_promise_resume_data`, the status key is updated to `ResumeInitiated` (not deleted), and the `PromiseYieldReceipt` trie entry is also still present (it is only removed by the runtime when the DataReceipt is processed in a later step). Therefore, both guard conditions remain `true`, and a second call unconditionally creates a second `DataReceipt` and returns `Ok(true)`.

### Impact Explanation
Two `DataReceipt`s are produced for the same yield `data_id`. The callback receives the first payload. The second `DataReceipt` is processed after the yield receipt has been consumed: its data is written as `ReceivedData` in the trie with no matching pending receipt, permanently leaking trie storage. If the attacker supplies different payloads in the two calls, the callback receives an attacker-chosen subset of the data, and the remaining payload corrupts the `ReceivedData` namespace. The "each yield is resumed exactly once" protocol invariant is broken.

### Likelihood Explanation
Any deployed contract can trigger this by calling `promise_yield_resume` (the WASM host function) twice with the same `data_id` in a single function call. No special privileges are required — only the ability to deploy and call a contract.

### Recommendation
Replace the existence check with a value check. Use `get_promise_yield_status` and only proceed if the result is `Some(PromiseYieldStatus::Yielded)`. Alternatively, remove the `PromiseYieldStatus` key (or set it to a sentinel that `contains_key` cannot distinguish — but removal is cleaner) immediately after the first successful resume, so that subsequent calls find no key and return `Ok(false)`.

### Proof of Concept
In a runtime integration test on `RuntimeExt`:
1. Call `create_promise_yield_receipt(receiver_id)` → obtain `(receipt_index, data_id)`.
2. Call `submit_promise_resume_data(data_id, b"payload1".to_vec())` → assert returns `Ok(true)`.
3. Call `submit_promise_resume_data(data_id, b"payload2".to_vec())` → **currently returns `Ok(true)`** (bug); should return `Ok(false)`.
4. Inspect `ReceiptManager` outgoing receipts → **two** `DataReceipt`s are present for the same `data_id` (bug); only one should exist.
