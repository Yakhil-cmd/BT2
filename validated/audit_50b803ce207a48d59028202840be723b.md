### Title
Shared `*flate.Writer` in deflate pool `New` closure causes concurrent gRPC response cross-contamination — (`engine/common/grpc/compressor/deflate/deflate.go`)

### Summary

The deflate compressor's `init()` creates a single `*flate.Writer w` and captures it in the `sync.Pool.New` closure. Every call to `New` returns a distinct `*writer` struct, but all such structs embed the **same** underlying `*flate.Writer`. Under concurrent gRPC load, multiple goroutines simultaneously hold different `*writer` pool objects that all share one `*flate.Writer`, causing data races on `Reset`/`Write`/`Close` and allowing one client's compressed response bytes to be written into a different client's wire buffer.

### Finding Description

In `engine/common/grpc/compressor/deflate/deflate.go`:

```go
func init() {
    c := &compressor{}
    w, _ := flate.NewWriter(nil, flate.DefaultCompression)   // single shared writer
    c.poolCompressor.New = func() any {
        return &writer{Writer: w, pool: &c.poolCompressor}   // always wraps same w
    }
    encoding.RegisterCompressor(c)
}
``` [1](#0-0) 

`sync.Pool.New` is called whenever the pool is empty. Because the closure always returns a new `*writer` wrapping the **same** `*flate.Writer w`, the pool can accumulate multiple distinct `*writer` objects that all share one `*flate.Writer`. Once two goroutines each hold a different pool object, they race on the shared writer's internal state.

The `Compress` path is:

```go
func (c *compressor) Compress(w io.Writer) (io.WriteCloser, error) {
    dw := c.poolCompressor.Get().(*writer)
    dw.Reset(w)   // calls flate.Writer.Reset on the shared *flate.Writer
    return dw, nil
}
``` [2](#0-1) 

`flate.Writer.Reset(dst)` sets the internal destination field. If goroutine A calls `Reset(dst_A)` and goroutine B then calls `Reset(dst_B)` on the same `*flate.Writer`, goroutine A's subsequent `Write` calls flush bytes to `dst_B`. This is not a theoretical TOCTOU — it is a direct data race on the same memory location.

Compare with the correct snappy implementation, where `New` creates a **fresh** writer on every call:

```go
c.poolCompressor.New = func() any {
    return &writer{Writer: snappy.NewBufferedWriter(nil), pool: &c.poolCompressor}
}
``` [3](#0-2) 

The deflate compressor is imported (via blank import) into the production gRPC server and the Access node connection manager: [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation

Any gRPC client that negotiates `deflate` content-encoding with an Access or Execution node gRPC server will use this compressor for every response. Under concurrent load (two or more simultaneous deflate-compressed responses), the shared `*flate.Writer` causes one response's compressed bytes to be written into another response's wire buffer. The receiving client decompresses bytes that belong to a different request — potentially exposing execution results, Cadence script outputs (account register values, balances), or randomness outputs from a different account's script execution to an attacker-controlled client.

### Likelihood Explanation

The race is triggered by any two concurrent gRPC calls that both use deflate compression. This is not a rare edge case: the pool starts empty, so the very first burst of concurrent requests at node startup triggers it. After a GC cycle clears the pool, it recurs. Even in steady state, the pool accumulates multiple `*writer` objects all wrapping the same `*flate.Writer`, so any two concurrent in-flight requests race. No special privileges are required — any unauthenticated client can open concurrent gRPC connections to a public Access node.

### Recommendation

Fix the `New` closure to allocate a fresh `*flate.Writer` per pool entry, matching the snappy pattern:

```go
c.poolCompressor.New = func() any {
    w, _ := flate.NewWriter(nil, flate.DefaultCompression)
    return &writer{Writer: w, pool: &c.poolCompressor}
}
``` [7](#0-6) 

### Proof of Concept

1. Start a Flow Access node with deflate compression enabled.
2. Open 100 concurrent gRPC connections, each sending a distinct script execution request with a unique payload.
3. Each client negotiates `grpc-encoding: deflate`.
4. Collect all 100 responses and decompress each.
5. Assert that each decompressed response matches its own request's expected output.
6. Under the current code, some responses will decompress to bytes belonging to a different concurrent request, demonstrating cross-client data leakage. Running with Go's race detector (`-race`) will also immediately flag the data race on the shared `*flate.Writer`.
