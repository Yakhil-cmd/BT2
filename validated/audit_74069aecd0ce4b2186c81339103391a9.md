### Title
Unprivileged Caller Can Permanently Freeze Consensus Block Rewards via `disable_rewards` Flag - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable by any address and accepts a caller-controlled `disable_rewards` boolean. When `disable_rewards=true`, the function writes the current block number to the global `last_reward_block` storage slot but returns before distributing any rewards. Because `last_reward_block` is the sole guard preventing duplicate reward updates within a block, an attacker can call `update_rewards(any_active_staker, disable_rewards=true)` once per block to permanently consume every reward slot without distributing a single token.

---

### Finding Description

`IStakingRewardsManager.update_rewards` is defined as a public interface function with no role restriction: [1](#0-0) 

Inside the implementation the function:

1. Asserts `current_block_number > last_reward_block` — the only guard against re-entry within a block.
2. Validates that `staker_address` is an active staker with non-zero STRK balance.
3. **Unconditionally** writes `current_block_number` to `last_reward_block`.
4. Immediately returns if `disable_rewards == true`, skipping all reward computation and distribution. [2](#0-1) 

The critical ordering is:

```
last_reward_block.write(current_block_number);   // ← slot consumed
if disable_rewards || self.is_pre_consensus() {
    return;                                       // ← rewards never sent
}
```

`last_reward_block` is a **global** variable shared across all stakers. A single call with `disable_rewards=true` exhausts the reward slot for the entire block for every staker, not just the supplied `staker_address`. [3](#0-2) 

The analog to the Hinkal bug is exact: in Hinkal, `inputNullifiers` was left empty so `insertNullifiers()` was skipped while new commitments were still inserted — a critical state update was bypassed while a secondary state update (commitment insertion) still occurred. Here, reward distribution is bypassed while the secondary state update (`last_reward_block`) still occurs, consuming the slot.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(valid_staker, disable_rewards=true)` once per block prevents every staker and every delegation pool from ever accumulating consensus block rewards. The rewards are not deferred; the per-block reward calculation window is simply never entered. Sustained over time this is equivalent to setting the reward rate to zero for the entire protocol.

---

### Likelihood Explanation

**High.** The function is permissionlessly callable. The attacker only needs to know any active staker address (trivially readable from on-chain events such as `NewStaker`). The gas cost of a single call is low (one storage read, one storage write, one early return). The attack can be automated with a simple keeper bot at negligible cost relative to the rewards being suppressed.

---

### Recommendation

Remove `disable_rewards` from the public interface entirely, or gate it behind a privileged role (e.g., `only_security_agent`). Alternatively, move the `last_reward_block.write` to occur **after** the `disable_rewards` guard so that a skipped distribution does not consume the block's reward slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number); // write only when rewards are actually sent
```

---

### Proof of Concept

1. Attacker identifies any active staker address `S` (e.g., from a `NewStaker` event).
2. Each block, attacker calls:
   ```
   staking.update_rewards(staker_address=S, disable_rewards=true)
   ```
3. Inside the call:
   - `current_block_number > last_reward_block` passes (new block).
   - Staker `S` is validated as active with non-zero balance.
   - `last_reward_block` is written to `current_block_number`.
   - Function returns early — no rewards distributed.
4. Any legitimate call to `update_rewards` later in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeated every block: all stakers and pool members accumulate zero consensus rewards indefinitely. [4](#0-3)

    impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }
