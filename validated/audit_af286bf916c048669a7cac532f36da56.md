### Title
Unprivileged Caller Can Permanently Freeze Block Rewards by Calling `update_rewards` with `disable_rewards: true` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` has no access control beyond a non-zero caller check. It accepts a `disable_rewards: bool` parameter. When called with `disable_rewards: true`, it advances the global `last_reward_block` to the current block without distributing any rewards. Because only one successful `update_rewards` call is permitted per block (enforced by the `last_reward_block` guard), any unprivileged caller can permanently destroy the block reward for every staker by issuing a single cheap transaction per block.

---

### Finding Description

`update_rewards` is gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero: [1](#0-0) 

No role check is performed. The function then enforces a single-call-per-block invariant via a global storage variable: [2](#0-1) 

After validating the staker, it unconditionally writes the current block number to `last_reward_block`: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after that write — before calling `calculate_block_rewards`, before calling `_update_rewards`, and before notifying the reward supplier. The rewards for that block are never accounted for and are permanently inaccessible to all stakers.

`last_reward_block` is a single global field (not per-staker): [4](#0-3) 

Therefore, one call with `disable_rewards: true` for **any** valid active staker blocks every other staker from receiving rewards at that block number.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

For every block at which the attacker fires the transaction, the entire block's consensus rewards are permanently lost for all stakers. The reward supplier is never notified (`update_unclaimed_rewards_from_staking_contract` is never called), so those rewards remain stranded. Repeated execution across consecutive blocks can drain the entire reward stream indefinitely.

---

### Likelihood Explanation

**High.** The attack requires:
- Any non-zero Starknet account (no tokens, no KYC, no role).
- Knowledge of any currently active staker address (trivially obtained from on-chain `NewStaker` events).
- One transaction per block.

There is no economic barrier. The attacker gains nothing directly but can inflict continuous, irreversible yield loss on all stakers and delegators.

---

### Recommendation

Restrict `update_rewards` to a trusted caller — for example, the sequencer infrastructure address, a dedicated role, or the attestation contract. Alternatively, remove the `disable_rewards` parameter entirely and handle the "no rewards" case internally based on on-chain conditions (e.g., `is_pre_consensus()`), so that no external caller can suppress reward distribution.

---

### Proof of Concept

```
// Attacker (any non-zero address) executes at block N:
staking_contract.update_rewards(
    staker_address = <any active staker>,  // public from NewStaker events
    disable_rewards = true,
);
// Effect:
//   last_reward_block := N   (line 1485)
//   returns early            (line 1487-1489)
//   no rewards distributed, reward supplier never notified

// Legitimate staker or sequencer tries at the same block N:
staking_contract.update_rewards(staker_address, disable_rewards = false);
// Panics: REWARDS_ALREADY_UPDATED  (line 1454-1458)
// Block N rewards are permanently lost for all stakers.
```

The attacker repeats this every block. The cost is one transaction per block; the damage is the complete suppression of all consensus-phase staking rewards.
