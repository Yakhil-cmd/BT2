### Title
Gateway Signature Validation Bypass via `skip_stateful_validations` Overly-Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (i.e., signature verification) for any Invoke transaction with `nonce == 1` targeting an account whose address appears in the mempool. The check uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from that address — not specifically a `DeployAccount` submitted by the legitimate owner. Any unprivileged third party who observes a pending `DeployAccount` for address `X` can immediately submit a crafted Invoke with `nonce = 1` for `X` carrying an invalid or forged signature, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is intended as a UX feature: when a user sends `DeployAccount + Invoke` atomically, the gateway cannot yet run `__validate__` on the Invoke because the account does not exist on-chain yet. The function therefore skips the entry-point call when:

1. The transaction is an `Invoke`.
2. `tx.nonce == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

The comment on condition 4 reads: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [2](#0-1) 

The flaw is that the check is **not scoped to the submitter**. `account_tx_in_pool_or_recent_block` checks whether the address appears in the pool or the committed-block state: [3](#0-2) 

Any observer who learns address `X` (deterministic from the `DeployAccount` fields) can submit `Invoke(sender=X, nonce=1, signature=<garbage>)`. Because `X` is already in the pool (from the legitimate `DeployAccount`), condition 4 is satisfied, `skip_stateful_validations` returns `true`, and `run_validate_entry_point` is called with `execution_flags.validate = false`: [4](#0-3) 

Inside `validate_tx`, the early-return on `!self.execution_flags.validate` means the `__validate__` entry point is never invoked: [5](#0-4) 

The malformed Invoke therefore passes all gateway checks and is forwarded to the mempool via `add_tx`: [6](#0-5) 

### Impact Explanation

**Mempool/gateway admission accepts invalid transactions** (High per scope).

- An unsigned or adversarially-signed Invoke for a not-yet-deployed account is admitted to the mempool.
- If the attacker's Invoke carries a higher tip/fee than the legitimate owner's Invoke for the same `(address, nonce=1)` slot, the mempool's fee-escalation path can **replace** the legitimate transaction with the invalid one, effectively cancelling the owner's first post-deploy call.
- When the batcher eventually executes the invalid Invoke, `new_for_sequencing` sets `validate: true`, so `__validate__` runs and the transaction reverts — but the legitimate transaction is already gone from the mempool. [7](#0-6) 

### Likelihood Explanation

`DeployAccount` transactions are broadcast over P2P and visible to all mempool peers. The target address is deterministic and computable from the transaction fields. No privileged access is required; any network participant can craft the attack. The only constraint is timing (the attack window is between `DeployAccount` entering the mempool and being committed on-chain).

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a **`DeployAccount`** transaction (not just any transaction) is pending for the target address. Alternatively, require the gateway to verify that the `Invoke` was submitted in the same request batch as the `DeployAccount` (e.g., by checking a shared nonce or session token), so the skip cannot be triggered by a third party.

### Proof of Concept

1. Alice submits `DeployAccount(class_hash=C, salt=S, ...)` for address `X = compute_address(C, S, ...)`. The transaction passes full gateway validation and enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.

2. Bob observes the pending `DeployAccount` (via P2P gossip or the RPC). Bob computes `X` and constructs:
   ```
   Invoke(sender_address=X, nonce=1, calldata=<anything>, signature=<random bytes>)
   ```

3. Bob submits this Invoke to the gateway. The gateway runs `skip_stateful_validations`:
   - `tx.nonce() == Nonce(Felt::ONE)` ✓
   - `account_nonce == Nonce(Felt::ZERO)` ✓ (X not yet on-chain)
   - `account_tx_in_pool_or_recent_block(X)` → `true` ✓ (Alice's DeployAccount is in pool)
   - Returns `true` → `execution_flags.validate = false`

4. `run_validate_entry_point` is called; `validate_tx` returns `Ok(None)` immediately without calling `__validate__`. The Invoke is admitted to the mempool.

5. If Bob's Invoke has a higher fee than Alice's legitimate Invoke (also nonce=1), the mempool's fee-escalation logic replaces Alice's Invoke with Bob's invalid one.

6. The batcher executes Bob's Invoke with `validate: true`; `__validate__` fails on the invalid signature; the transaction reverts. Alice's legitimate Invoke has been evicted from the mempool. [1](#0-0) [3](#0-2) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
