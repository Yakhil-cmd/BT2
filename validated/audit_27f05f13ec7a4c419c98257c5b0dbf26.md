### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo::update_rewards`)

### Summary
`update_rewards` is callable by any non-zero address. It writes to a **global** `last_reward_block` storage variable. An attacker calling it with `disable_rewards=true` each block permanently discards that block's rewards for all stakers, with no recovery path.

### Finding Description

The spec mandates "Only starkware sequencer" access control for `update_rewards`: [2](#0-1) 

The implementation enforces no such restriction. `general_prerequisites` only checks pause state and non-zero caller: [3](#0-2) 

Inside `update_rewards`, the global `last_reward_block` is written unconditionally before the `disable_rewards` branch: [4](#0-3) 

```rust
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
```

`last_reward_block` is a **single global slot** — not per-staker. The guard at the top of the function: [5](#0-4) 

```rust
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

...means only **one** `update_rewards` call is permitted per block across the entire contract. Once an attacker calls `update_rewards(any_valid_staker, disable_rewards=true)` in block N, `last_reward_block` is set to N and every subsequent call in block N reverts with `REWARDS_ALREADY_UPDATED`. Block N's rewards are permanently lost for every staker.

### Impact Explanation
An attacker can call `update_rewards(any_valid_staker, disable_rewards=true)` every block, permanently suppressing all consensus block rewards for all stakers. The rewards for each skipped block are never recoverable — the `last_reward_block` gate ensures no retroactive distribution. This matches **High: Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The call requires no capital, no privilege, and no setup beyond knowing any valid active staker address (which is public on-chain). A single bot running continuously can execute this attack indefinitely at minimal cost.

### Recommendation
Add a caller check at the top of `update_rewards` that restricts execution to the authorized sequencer address (or a role-gated address), consistent with the spec's stated access control. For example:

```rust
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::ONLY_SEQUENCER,
);
```

### Proof of Concept
1. Deploy with two active stakers (past the K-epoch activation window) and consensus rewards enabled.
2. Each block, call `update_rewards(staker_A, disable_rewards=true)` from an arbitrary EOA.
3. Observe `last_reward_block` advances each block but `unclaimed_rewards_own` for both stakers never increases.
4. Confirm any legitimate `update_rewards(staker_B, disable_rewards=false)` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Compare cumulative rewards after N blocks against the expected model — the difference equals permanently lost yield.

The existing test at `src/staking/tests/test.cairo:3887–3894` already demonstrates that `disable_rewards=true` followed by `disable_rewards=false` in the same block reverts, confirming the one-call-per-block global gate is enforced. [6](#0-5)
