### Title
Missing Verification of Participant Contributions in CKD Coordinator Allows Any Malicious Participant to Corrupt the CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` aggregates `(big_y, big_c)` shares from all participants without any proof of correct computation. Any single malicious participant can send arbitrary group elements in place of their honest share, causing the coordinator to produce a corrupted `CKDOutput`. When the client unmasks the result, it obtains an incorrect confidential key instead of `msk · H(pk ‖ app_id)`.

### Finding Description
The CKD coordinator collects one `CKDOutput` from every other participant and blindly adds the received elements to its own running totals:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to send:

```
norm_big_y_i = λ_i · (y_i · G)
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

computed inside `compute_signature_share`: [2](#0-1) 

No zero-knowledge proof, commitment, or any other cryptographic check is attached to the message. The coordinator has no way to distinguish a correctly-formed share from an arbitrary pair of group elements. Because `recv_from_others` only enforces that the sender is a known participant (via `ParticipantCounter`), a participant who is legitimately enrolled in the protocol can substitute any `(big_y', big_c')` of their choosing. [3](#0-2) 

The `ParticipantMap::put` used elsewhere also silently accepts the first value received per participant and ignores subsequent ones, so there is no second-chance rejection path. [4](#0-3) 

### Impact Explanation
After the coordinator sums all contributions, it returns a `CKDOutput { big_y, big_c }`. The client unmasks it as:

```
confidential_key = big_c − app_sk · big_y
``` [5](#0-4) 

If participant `j` sends `(big_y_j', big_c_j')` instead of their honest values, the coordinator's aggregate becomes:

```
big_y_total' = big_y_total + λ_j·(big_y_j' − big_y_j)
big_c_total' = big_c_total + λ_j·(big_c_j' − big_c_j)
```

The client then unmasks to `msk·H(pk‖app_id) + λ_j·(big_c_j' − big_c_j − app_sk·(big_y_j'−big_y_j))`, which is not the intended confidential key. The malicious participant can freely choose the additive bias, making the derived key completely wrong and unusable. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation
Any single participant enrolled in the CKD session can mount this attack. No special privilege beyond being a listed participant is required. The attacker only needs to deviate from the honest protocol by sending a malformed `CKDOutput` to the coordinator. The coordinator has no mechanism to detect the deviation. Likelihood is **High**.

### Recommendation
Require each participant to attach a non-interactive zero-knowledge proof (e.g., a Schnorr-style DLEQ proof) demonstrating that their `big_c_i` was computed using the same discrete-log witness as their public verification share `vk_i = x_i · G2`. Concretely, for each participant `i` the coordinator should verify:

```
DLEQ( big_s_i / H(pk‖app_id) == vk_i / G2 )
```

where `big_s_i = big_c_i − y_i · app_pk`. Because `y_i` is private, the proof must be provided by the participant alongside their share. Alternatively, a commitment-then-reveal scheme (hash the share before sending, reveal later) can prevent equivocation, though it does not prove correct key usage. The strongest fix is a full DLEQ proof bound to the session transcript.

### Proof of Concept
1. Honest participants `{P1, P2, P3}` run `ckd(...)` with coordinator `P1`.
2. Malicious `P2` overrides `do_ckd_participant` to send `(G1::identity(), G1::identity())` instead of its honest `(norm_big_y, norm_big_c)`.
3. The coordinator at line 53–55 adds the identity elements, effectively subtracting `P2`'s honest contribution from the aggregate.
4. The coordinator outputs `CKDOutput { big_y: big_y_honest − λ_2·big_y_2, big_c: big_c_honest − λ_2·big_c_2 }`.
5. The client calls `ckd_output.unmask(app_sk)` and obtains a value that differs from `msk·H(pk‖app_id)` by `−λ_2·(big_c_2 − app_sk·big_y_2)`, which is non-zero with overwhelming probability.
6. Every subsequent operation that depends on the derived confidential key fails or produces incorrect results, with no error surfaced to the honest coordinator or client. [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
