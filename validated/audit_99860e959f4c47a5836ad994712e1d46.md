### Title
Post-Logout WalletConnect Command Execution via No-Op `handleLogOut` — (`packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary
`handleLogOut` is an intentional no-op stub. When a user clicks "Log Out" and is navigated to the key selection screen, the Chia backend remains fully logged in as fingerprint A, `WalletConnectProvider` remains mounted and listening for `session_request` events, and the fingerprint guard in `useWalletConnectCommand` continues to pass because `currentFingerprint === pair.fingerprint === A`. A connected WalletConnect dApp can therefore dispatch wallet commands (send, sign, create/take offer) silently after the user believes they have logged out.

### Finding Description
`handleLogOut` in `AuthProvider.tsx` (lines 52–55) performs no backend logout, no session teardown, and no WalletConnect disconnection. `LogoutButton` calls `logOut()` then `navigate('/')`, giving the user the impression of a logged-out state. However:

- `WalletConnectProvider` is mounted as a child of `AppProviders`, which is the element for the root route `path="/"`. It is never unmounted during navigation to `/`.
- The `session_request` listener registered in `WalletConnectProvider`'s `useEffect` (line 418) remains active.
- `useWalletConnectCommand` reads `currentFingerprint` from `useGetLoggedInFingerprintQuery()`. Because the backend is still logged in as A, this returns A.
- The guard `currentFingerprint !== pair.fingerprint` evaluates to `A !== A = false`, so the check passes.
- `window.permissionsAPI.dispatchAsPair` is called with no additional user confirmation prompt.

### Impact Explanation
An already-paired WalletConnect dApp can send `chia_sendTransaction`, `chia_takeOffer`, `chia_createOfferForIds`, `chia_signMessageById`, etc. after the user has clicked "Log Out." These execute silently against the user's wallet under fingerprint A. This constitutes unauthorized signing, sending, or offer execution under the user's key after they believe they have logged out — a direct Critical/High wallet impact.

### Likelihood Explanation
Requires a WalletConnect session to already be established (user previously approved the dApp). The dApp then simply sends a `session_request` at any point after the user clicks logout. No race condition is required — the window is open indefinitely until the user selects a new key or closes the app.

### Recommendation
`handleLogOut` must perform at minimum: (1) disconnect all active WalletConnect sessions/pairings, and (2) set a flag that causes `useWalletConnectCommand` to reject all incoming commands. Ideally, the backend should also be called to log out. Until a proper backend logout API exists, the WalletConnect listener should be torn down and all pairs disconnected on logout.

### Proof of Concept
1. Establish a WalletConnect pairing with fingerprint A and approve `chia_sendTransaction`.
2. Click "Log Out" in the GUI — user is navigated to `/` (key selection screen).
3. From the dApp, send a `session_request` for `chia_sendTransaction` with a valid destination and amount.
4. Observe: the command passes the fingerprint guard and `dispatchAsPair` is called, executing the transaction against fingerprint A's wallet without any user confirmation.
