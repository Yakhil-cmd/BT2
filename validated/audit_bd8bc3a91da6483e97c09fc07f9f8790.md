### Title
Unprivileged Caller Can Force Deployment of Staged Contract Upgrade After Delay — (`engine/src/lib.rs`)

---

### Summary

The `deploy_upgrade()` WASM entrypoint in `engine/src/lib.rs` performs no caller-identity check. Any unprivileged NEAR account can invoke it after the staged-upgrade delay expires, stripping the owner of the ability to cancel a staged upgrade and potentially forcing deployment of code the owner has already decided to retract.

---

### Finding Description

Every other critical administrative entrypoint in the engine enforces `require_owner_only` before acting. `deploy_upgrade()` is the sole exception:

```rust
// engine/src/lib.rs  (deploy_upgrade, lines 171-185)
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)          // ← only liveness check
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {  // ← only timing check
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
``` [1](#0-0) 

By contrast, `stage_upgrade` — which writes the code that `deploy_upgrade` will execute — correctly enforces `require_owner_only`:

```rust
// engine/src/contract_methods/admin.rs  (stage_upgrade, lines 154-167)
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;   // ← present here
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        ...
    })
}
``` [2](#0-1) 

The design intent is that the owner controls both *what* is deployed (via `stage_upgrade`) and *when* it is deployed (via `deploy_upgrade`). The missing check on `deploy_upgrade` breaks the second half of that invariant.

### Impact Explanation

**Impact: High — Temporary or Permanent Freezing of Funds**

The concrete harm path:

1. Owner calls `stage_upgrade` with new WASM code (normal operation). The code is written to `CODE_KEY`; the delay block height is written to `CODE_STAGE_KEY`.
2. Before the delay expires, the owner discovers a critical bug in the staged code (e.g., a storage-layout regression that would brick the engine or a logic error in the bridge accounting).
3. The owner's only recourse is to call `stage_upgrade` again with corrected code, which overwrites `CODE_KEY` and resets the delay. However, there is a race window between the moment the original delay expires and the moment the owner's replacement `stage_upgrade` transaction lands.
4. An attacker monitors the chain, sees the delay expire, and immediately calls `deploy_upgrade()`. Because there is no caller check, the call succeeds and the buggy WASM is deployed as the live contract.
5. If the deployed code contains a vulnerability that freezes withdrawals or corrupts accounting, all user funds held in the engine (ETH, bridged ERC-20 tokens) become inaccessible — a permanent freeze.

The attacker controls no input to the deployed code; the harm is entirely a function of the bug in the staged code. However, the missing auth check is the necessary condition that prevents the owner from safely canceling the upgrade.

---

### Likelihood Explanation

**Likelihood: Low**

- Requires the owner to have staged code that contains a critical bug.
- Requires the attacker to observe the delay expiry and submit `deploy_upgrade()` before the owner's corrective `stage_upgrade` transaction.
- NEAR block times are ~1 second, so the race window is narrow but non-zero, especially under network congestion.
- The upgrade path is exercised infrequently, reducing exposure.

---

### Recommendation

Add `require_owner_only` to `deploy_upgrade()`, consistent with every other privileged entrypoint:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state).map_err(ContractError::msg).sdk_unwrap();
    require_owner_only(&state, &io.predecessor_account_id())   // add this
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

Alternatively, expose a `cancel_upgrade` entrypoint (owner-only) that clears `CODE_KEY` and `CODE_STAGE_KEY`, giving the owner a safe cancellation path independent of the race.

### Proof of Concept

```
# Attacker script (pseudo-NEAR CLI)
# 1. Watch chain for CODE_STAGE_KEY to appear (owner staged an upgrade)
# 2. Poll block_height until block_height > staged_delay_block_height
# 3. Submit:
near call aurora deploy_upgrade '{}' \
  --accountId attacker.near \
  --gas 300000000000000
# No owner check → succeeds → buggy WASM deployed
```

The call requires no special permissions, no attached deposit, and no knowledge of the staged code's contents. The only precondition is that the delay has elapsed, which is publicly observable on-chain. [4](#0-3) [5](#0-4)

pub fn require_owner_only(
    state: &state::EngineState,
