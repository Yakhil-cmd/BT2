### Title
Missing `requiresSync` Guard on WalletConnect Spending Commands Allows Execution Against Stale Wallet State - (File: `packages/gui/src/electron/commands/Commands.ts`)

### Summary

Several WalletConnect spending commands — including `chia_takeOffer`, `chia_cancelOffer`, `chia_transferNFT`, `chia_setNFTDID`, `chia_createOfferForIds`, and `chia_pushTransactions` — are missing the `requiresSync: true` flag in their dapp command schema. This means the `waitForWalletSync()` guard in `useWalletConnectCommand` is skipped for these commands, allowing a connected dApp to trigger asset-spending operations while the wallet is operating on stale chain state. The user's confirmation dialog is populated with data fetched from an unsynced wallet, causing them to approve spends, offer acceptances, or cancellations based on incorrect coin/offer state.

### Finding Description

The `requiresSync` mechanism is the GUI's explicit guard against spending on stale state. When `requiresSync: true` is set on a dapp command schema, `useWalletConnectCommand.handleProcess` calls `waitForWalletSync()` before dispatching the command, and then re-validates the fingerprint post-sync. [1](#0-0) 

This guard is correctly applied to `chia_sendTransaction` and `chia_spendCAT`: [2](#0-1) [3](#0-2) 

However, the following spending commands — all listed in `isSpendCommand` — have **no** `requiresSync: true` in their dapp entry:

| dApp command | Parent command key | Effect |
|---|---|---|
| `chia_takeOffer` | `chia_wallet.take_offer` | Accepts offer, spends XCH/CAT/NFT |
| `chia_cancelOffer` | `chia_wallet.cancel_offer` | Cancels offer on-chain |
| `chia_transferNFT` | `chia_wallet.nft_transfer_nft` | Transfers NFT |
| `chia_setNFTDID` | `chia_wallet.nft_set_nft_did` | Assigns NFT to DID |
| `chia_createOfferForIds` | `chia_wallet.create_offer_for_ids` | Creates offer, locks assets |
| `chia_pushTransactions` | `chia_wallet.push_transactions` | Pushes raw transaction bundle | [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

`getDappCommandMetadata` returns `requiresSync: false` for all of these, so `waitForWalletSync()` is never called: [8](#0-7) 

### Impact Explanation

When a dApp invokes `chia_takeOffer` or `chia_cancelOffer` via WalletConnect while the wallet is mid-sync:

1. `handleProcess` skips `waitForWalletSync()` because `commandMetadata.requiresSync` is `false`.
2. `dispatchAsPair` is called immediately, reaching the main process.
3. The confirmation dialog is rendered using `parseCommandDisplay`, which calls `getOfferSummary` against the unsynced daemon — returning stale offer/coin data.
4. The user sees and approves a confirmation showing incorrect asset amounts, offer state, or coin availability.
5. The spend is submitted to the daemon with stale UTXO state.

For `chia_pushTransactions` specifically, a dApp can push a pre-constructed transaction bundle without any sync gate, meaning the bundle may reference coins that have already been spent on-chain but are not yet reflected in the local wallet state.

This matches the allowed High impact: *"WalletConnect state that causes a user to approve, send, or display the wrong asset, identity, amount, destination, or status."*

### Likelihood Explanation

Any dApp with a granted WalletConnect pair that includes `chia_takeOffer`, `chia_cancelOffer`, or `chia_pushTransactions` can trigger this condition. The wallet is frequently in a partially-synced state during startup, after a long idle period, or on slow connections. The dApp does not need any special privilege beyond the already-granted command permission.

### Recommendation

Add `requiresSync: true` to the dapp command schema entries for all spending commands that currently lack it:

```ts
// chia_wallet.take_offer
dapp: [{ command: 'chia_takeOffer', requiresSync: true, ... }]

// chia_wallet.cancel_offer
dapp: [{ command: 'chia_cancelOffer', requiresSync: true, ... }]

// chia_wallet.nft_transfer_nft
dapp: [{ command: 'chia_transferNFT', requiresSync: true, ... }]

// chia_wallet.nft_set_nft_did
dapp: [{ command: 'chia_setNFTDID', requiresSync: true, ... }]

// chia_wallet.create_offer_for_ids
dapp: [{ command: 'chia_createOfferForIds', requiresSync: true, ... }]

// chia_wallet.push_transactions
dapp: [{ command: 'chia_pushTransactions', requiresSync: true, ... }]
```

Alternatively, enforce `requiresSync` automatically for all commands in `isSpendCommand` rather than relying on per-command opt-in.

### Proof of Concept

1. Establish a WalletConnect pair granting `chia_takeOffer` to a dApp.
2. Start the Chia GUI while the wallet is syncing (e.g., immediately after launch or after a long idle).
3. From the dApp, send a `chia_takeOffer` session request with a valid offer string.
4. Observe that `useWalletConnectCommand.handleProcess` does **not** call `waitForWalletSync()` — `commandMetadata.requiresSync` is `false` for this command.
5. The confirmation dialog opens immediately, populated with offer summary data fetched from the unsynced wallet daemon.
6. The user approves, and the offer acceptance is submitted against stale coin state — potentially referencing coins already spent or an offer already taken/cancelled on-chain. [9](#0-8) [10](#0-9)
