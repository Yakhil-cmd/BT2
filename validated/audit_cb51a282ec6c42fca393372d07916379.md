### Title
Spread-Fee Surplus Includes Notional-Fee Accumulator, Causing Double-Payout That Drains LP Principal on Every `collectFees` Call — (File: `metric-core/contracts/MetricOmmPool.sol` / `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

When both spread fees and notional fees are active on a pool, `MetricOmmPool.collectFees` double-counts the notional-fee accumulator: the spread-fee split consumes the **entire** surplus (which already contains the notional accumulator), and then the notional-fee split pays out the same accumulator a second time. The total attempted payout exceeds the available surplus by exactly the notional accumulator amount, and the shortfall is covered by draining LP bin funds.

### Finding Description

`collectFees` computes two independent payouts:

**Spread-fee split** (factory-stored `poolFeeConfig` rates):

```
spread0ToAdmin    = surplus0Scaled × adminSpreadFeeE6   / spreadFeeE6
spread0ToProtocol = surplus0Scaled × protocolSpreadFeeE6 / spreadFeeE6
```

Because `adminSpreadFeeE6 + protocolSpreadFeeE6 = spreadFeeE6`, the two terms sum to exactly `surplus0Scaled` — the **full** surplus, which is `balance × scale − binTotals`.

**Notional-fee split** (from the on-pool accumulators `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`):

```
notional0ToAdmin    = notional0 × adminNotionalFeeE8 / notionalFeeE8
notional0ToProtocol = notional0 − notional0ToAdmin
```

These sum to `notional0` — the full notional accumulator.

The notional accumulator is funded by swap-volume fees that are **already sitting inside the pool balance** and therefore **already included in `surplus0Scaled`**. Paying out both means:

```
total0Attempted = surplus0Scaled + notional0
               = (spreadSurplus + notional0) + notional0
               = spreadSurplus + 2 × notional0
```

The pool only holds `surplus0Scaled` tokens above bin totals. The extra `notional0` must come from LP bin funds.

The project's own test suite confirms this invariant break: [1](#0-0) 

```solidity
assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
```

The notional accumulators are zeroed after the transfer: [2](#0-1) 

so the drain is permanent and unrecoverable per collection cycle.

### Impact Explanation

Every call to `collectPoolFees` (permissionless) or any factory function that internally calls `collectFees` first (`setPoolProtocolFee`, `setPoolAdminFees`) drains LP bin funds by `notionalFeeToken0Scaled / token0ScaleMultiplier` and `notionalFeeToken1Scaled / token1ScaleMultiplier` tokens. Because bin totals are typically orders of magnitude larger than the notional accumulator, the ERC-20 transfer succeeds — the loss is silent. Over time, repeated fee collections compound the drain, making the pool insolvent relative to LP share claims. [3](#0-2) 

### Likelihood Explanation

- `collectPoolFees` is **permissionless** — any address can call it at any time.
- The condition requires only that both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` on the pool, which is the intended production configuration (protocol spread fee + admin notional fee).
- No special role, timing, or token behaviour is required. [4](#0-3) 

### Recommendation

Before computing the spread-fee split, subtract the notional accumulator from the surplus so the two fee pools are disjoint:

```solidity
uint256 spreadSurplus0 = surplus0Scaled - notionalFeeToken0Scaled;
uint256 spreadSurplus1 = surplus1Scaled - notionalFeeToken1Scaled;
// use spreadSurplus* for the spread split, notionalFee* for the notional split
```

Alternatively, accumulate spread fees and notional fees into separate storage slots so the surplus calculation never conflates them.

### Proof of Concept

1. Deploy a pool with `spreadFeeE6 = 15_000` (1.5 %) and `notionalFeeE8 = 1_000_000` (1 %).
2. Add liquidity and execute several round-trip swaps so both `notionalFeeToken0Scaled > 0` and a spread surplus accumulates.
3. Record `binTotals.scaledToken0` (LP claim) and `notionalFeeToken0Scaled`.
4. Call `factory.collectPoolFees(pool)` from any EOA.
5. Observe that `binTotals.scaledToken0` after the call is lower by `notionalFeeToken0Scaled / token0ScaleMultiplier` tokens — LP principal has been silently transferred to the fee recipients.

The project's own test at `metric-core/test/MetricOmmPool.notionalFee.t.sol:211-265` reproduces the overallocation arithmetic and asserts `total0Attempted > surplus0Scaled`, confirming the double-payout. [5](#0-4)
