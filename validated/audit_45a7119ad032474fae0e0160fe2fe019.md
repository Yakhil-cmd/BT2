### Title
ETH Permanently Frozen in `exit_to_near` Precompile When `error_refund` Feature Is Disabled — (File: engine-precompiles/src/native.rs)

---

### Summary

When the `error_refund` compile-time feature is not enabled, ETH sent to the `exit_to_near` precompile that fails its outbound `ft_transfer` NEAR promise is permanently frozen in the precompile's EVM address. No rescue mechanism — admin or user — exists to recover it.

---

### Finding Description

The `exit_to_near` precompile (address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) handles ETH base-token exits from Aurora EVM to NEAR. When a user sends ETH with flag `0x00`, the EVM deducts ETH from the caller's balance and credits it to the precompile's own EVM address. A NEAR promise is then scheduled to call `ft_transfer` on the eth-connector contract.

The critical issue is in how the error-refund callback is conditionally constructed: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

For the base ETH exit path (flag `0x00`), `transfer_near` is also `None`, so `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The branch that decides whether to attach a callback then fires the no-callback path:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no error callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

Without the callback, if `ft_transfer` fails (e.g., recipient account not registered with the NEP-141 eth-connector), the ETH credited to the precompile address is never returned. The `exit_to_near_precompile_callback` method — the only code path that calls `refund_on_error` for ETH — is gated behind `env.assert_private_call()`, meaning it can only be invoked as a NEAR promise callback, not directly by any admin or user: [3](#0-2) 

The ETH refund path inside `refund_on_error` transfers ETH from the precompile address back to the user, but this code is unreachable without the callback: [4](#0-3) 

---

### Impact Explanation

ETH sent to the `exit_to_near` precompile that fails its outbound NEAR promise is permanently frozen in the precompile's EVM balance. There is no admin function, no user-callable rescue path, and no automatic refund. The codebase itself confirms this in the integration test: [5](#0-4) 

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

**Impact: Critical — Permanent freezing of user funds.**

---

### Likelihood Explanation

Any EVM user who calls the `exit_to_near` precompile with ETH and specifies a recipient NEAR account that is not registered with the NEP-141 eth-connector will have their ETH permanently frozen. This can occur accidentally (wrong recipient) or due to external state changes (recipient account deregistered). The likelihood is conditioned on the production binary being compiled without the `error_refund` feature; the `#[cfg(not(feature = "error_refund"))]` branches are present and exercised in the test suite, confirming this is a supported build configuration.

---

### Recommendation

1. Make `error_refund` a **default feature** in `engine-precompiles/Cargo.toml` so it is always compiled into production builds.
2. Alternatively, implement an unconditional fallback rescue path for ETH stuck in the `exit_to_near` precompile address that does not depend on the `error_refund` feature flag — analogous to adding a `rescueETH()` function alongside `rescueERC20()` in the referenced Solidity report.

---

### Proof of Concept

1. Compile Aurora Engine **without** the `error_refund` feature.
2. An EVM user calls the `exit_to_near` precompile (address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with `msg.value > 0` and flag `0x00`, specifying a NEAR recipient account that is not registered with the NEP-141 eth-connector.
3. The EVM deducts ETH from the user's balance and credits it to the precompile address.
4. The `ft_transfer` NEAR promise fails (unregistered recipient).
5. Because `error_refund` is disabled, `callback_args == default()`, so no callback was attached — `exit_to_near_precompile_callback` is never invoked.
6. `refund_on_error` is never called; the ETH remains in the precompile's EVM balance indefinitely.
7. No admin method or user-callable function can recover it.

This exact scenario is confirmed by the existing test `test_exit_to_near_eth_refund`: [6](#0-5)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
