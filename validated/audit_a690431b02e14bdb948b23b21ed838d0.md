### Title
TOCTOU: Pool Server Can Substitute `targetPuzzleHash` Between Display and Signing Fetches — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

`PlotNFTSelectBase` and `prepareSubmitData` each call `getPoolInfo` independently. A malicious pool operator can serve legitimate data on the display fetch and a substituted `targetPuzzleHash` on the submit fetch, causing the user to sign a `createNewPoolWallet` transaction that routes farming rewards to the attacker's address.

### Finding Description

**Display path** — `PlotNFTSelectBase.tsx` calls `usePoolInfo(poolUrl)`, which internally calls `getPoolInfo(normalizedUrl)` reactively whenever `poolUrl` changes: [1](#0-0) [2](#0-1) 

**Submit path** — `prepareSubmitData` makes a completely independent second HTTP call to `getPoolInfo` at the moment the user clicks "Create": [3](#0-2) 

Specifically, line 29 re-fetches `targetPuzzleHash` and `relativeLockHeight` from the pool server at submit time: [4](#0-3) 

The `initialTargetState` built from this second fetch is passed directly to `createNewPoolWallet`: [5](#0-4) 

There is **no comparison** between the `targetPuzzleHash` shown to the user (from the display fetch) and the one used in the signed transaction (from the submit fetch). No confirmation dialog presents `targetPuzzleHash` to the user for verification.

### Impact Explanation

`targetPuzzleHash` is the puzzle hash to which all pooled farming rewards are paid. If an attacker substitutes their own puzzle hash on the second fetch, every block reward earned by the user's plots is redirected to the attacker's wallet. This is a direct, irreversible financial loss affecting pooled farming rewards — Critical impact under the scope rules ("payout change... affecting... pooled farming rewards").

### Likelihood Explanation

The attacker only needs to operate a pool server (a normal, expected external actor in this flow). No local host compromise, leaked keys, or broken cryptography is required. The window between the display fetch and the submit fetch is entirely under the attacker's control — they can hold the first response until the user has read and accepted the displayed info, then change the response for the second fetch. The attack is deterministic and locally testable.

### Recommendation

Capture the `poolInfo` result from the display fetch (already available in the `usePoolInfo` hook state) and pass it directly into `prepareSubmitData` / `handleSubmit` instead of re-fetching. The `targetPuzzleHash` and `relativeLockHeight` used to build `initialTargetState` must come from the same fetch whose result was displayed to the user for confirmation.

### Proof of Concept

1. Implement a pool server at `https://attacker.pool/`:
   - First `GET /pool_info` request: return a legitimate-looking response with `target_puzzle_hash = <legitimate_hash>`.
   - Second `GET /pool_info` request: return `target_puzzle_hash = <attacker_hash>`.
2. Run the Chia GUI, navigate to "Add a Plot NFT", enter `https://attacker.pool/` as the pool URL.
3. Observe the display fetch fires (`usePoolInfo`), the UI shows the legitimate pool info.
4. Click "Create". `prepareSubmitData` fires the second fetch, receives `<attacker_hash>`.
5. Inspect the `createNewPoolWallet` RPC call — `initialTargetState.targetPuzzleHash` will be `<attacker_hash>`, not `<legitimate_hash>`.
6. The plot NFT is created with the attacker's puzzle hash as the payout destination.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
