### Title
Hardcoded 2,300-Gas Stipend in `DepositCallGasLimit` Blocks FLOW Token Deposits to EVM Contract Recipients - (File: `fvm/evm/types/call.go`)

### Summary

`DepositCallGasLimit` and `WithdrawCallGasLimit` are hardcoded to `21_000 + 2_300 = 23_300` gas — the exact same pattern as Solidity's deprecated `.transfer()`. When `EVMAddress.deposit()` is called from Cadence with an EVM contract address as recipient, the underlying `DepositCall` is issued with only 2,300 gas available for the contract's `receive()`/`fallback()`. Any EVM contract requiring more than 2,300 gas in those hooks (e.g., multi-sig wallets, proxy contracts, ERC-777 receivers) will revert. The Cadence `deposit()` function does not inspect the EVM result and emits `FLOWTokensDeposited` regardless, while the FLOW vault has already been consumed — resulting in cross-VM asset loss.

### Finding Description

In `fvm/evm/types/call.go`, the gas limits for deposit and withdraw direct calls are defined as:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas          // 21_000
// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

`NewDepositCall` hardwires this limit:

```go
func NewDepositCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DepositCallGasLimit,  // always 23_300
    }
}
``` [2](#0-1) 

The Cadence-facing `EVMAddress.deposit()` in `contract.cdc` calls `InternalEVM.deposit` and unconditionally emits `FLOWTokensDeposited` without inspecting the EVM execution result:

```cadence
InternalEVM.deposit(from: <-from, to: self.bytes)
emit FLOWTokensDeposited(address: ..., amount: amount, ...)
``` [3](#0-2) 

The Go host function `newInternalEVMTypeDepositFunction` returns `interpreter.Void` unconditionally, giving the Cadence layer no signal about EVM-level failure:

```go
account := handler.AccountByAddress(toAddress, isAuthorized)
account.Deposit(types.NewFlowTokenVault(amount))
return interpreter.Void
``` [4](#0-3) 

Inside the emulator, `mintTo` resets EVM state on failure but the FLOW vault has already been consumed:

```go
if res.Invalid() || res.Failed() {
    proc.state.Reset()   // EVM balance NOT credited
    return res, nil      // no Go error propagated
}
``` [5](#0-4) 

The existing test explicitly confirms the revert behavior when depositing to a contract that does not accept native tokens:

```go
call := types.NewDepositCall(bridgeAccount, testContract, types.MakeBigIntInFlow(1), 0)
res, err := blk.DirectCall(call)
require.NoError(t, err)
require.NoError(t, res.ValidationError)
require.Equal(t, res.VMError, gethVM.ErrExecutionReverted)
``` [6](#0-5) 

### Impact Explanation

Any user who calls `EVMAddress.deposit(from: <-vault)` targeting an EVM contract address whose `receive()` or `fallback()` consumes more than 2,300 gas will:

1. Have their FLOW vault consumed (moved into `InternalEVM.deposit`).
2. Receive no EVM balance credit (EVM state is reset on revert).
3. See a spurious `FLOWTokensDeposited` event emitted despite the failed transfer.

The net result is **cross-VM asset loss**: FLOW tokens are permanently removed from Cadence without being credited in EVM. This affects any EVM contract wallet that performs non-trivial logic on receipt — multi-sig wallets, proxy contracts, ERC-777 hooks, and similar patterns all commonly exceed 2,300 gas.

### Likelihood Explanation

The entry path is fully unprivileged: any Cadence transaction author can call `EVMAddress.deposit()` with an arbitrary EVM address. Multi-sig wallets and proxy contracts are standard EVM infrastructure. The 2,300-gas stipend is a well-known limitation that has caused widespread issues on Ethereum (the exact motivation for the original report). The likelihood of users attempting to deposit FLOW to such contracts is high, and the failure is silent from the Cadence side.

### Recommendation

Replace the hardcoded `DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300` with a configurable or sufficiently large gas limit for `DepositCallGasLimit` (analogous to OpenZeppelin's `Address.sendValue` which forwards all available gas). Additionally, the Cadence `deposit()` function should inspect the EVM result and panic/revert if the EVM-level transfer failed, preventing silent token loss.

### Proof of Concept

1. Deploy an EVM contract whose `receive()` function performs any operation consuming more than 2,300 gas (e.g., a simple storage write or a multi-sig threshold check).
2. From a Cadence transaction, call:
   ```cadence
   let vault <- flowTokenVault.withdraw(amount: 1.0)
   let contractAddr = EVM.EVMAddress(bytes: <contract_address_bytes>)
   contractAddr.deposit(from: <-vault)
   ```
3. Observe: the Cadence transaction succeeds, `FLOWTokensDeposited` is emitted, but the EVM contract's balance remains zero. The 1.0 FLOW is permanently lost.

```go
func (proc *procedure) mintTo(
	call *types.DirectCall,
) (*types.Result, error) {
	// check and convert value
	value, isValid := checkAndConvertValue(call.Value)
	if !isValid {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrInvalidBalance,
		), nil
	}

	// create bridge account if not exist
	bridge := call.From.ToCommon()
	if !proc.state.Exist(bridge) {
		proc.state.CreateAccount(bridge)
	}

	// add balance to the bridge account before transfer
	proc.state.AddBalance(bridge, value, gethTracing.BalanceIncreaseWithdrawal)
	// check state errors
	if err := proc.state.Error(); err != nil {
		return nil, err
	}

	// withdraw the amount and move it to the bridge account
	res, err := proc.run(call.Message(), call.Hash(), types.DirectCallTxType)
	if err != nil {
		return res, err
	}

	// if any error (invalid or vm) on the internal call, revert and don't commit any change
	// this prevents having cases that we add balance to the bridge but the transfer
	// fails due to gas, etc.
	if res.Invalid() || res.Failed() {
		// reset the state to revert the add balances
		proc.state.Reset()
		return res, nil
	}

	// commit and finalize the state and return any stateDB error
	res.StateChangeCommitment, err = proc.commit(true)
	return res, err
}
