### Title
Exponent Interpolation Consistency Check Bypassed for First `threshold+1` Participants in Robust ECDSA Presign - (`File: src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

In the Robust ECDSA presign protocol, the exponent interpolation consistency check for `big_r_me` (and `big_w_me`) values is only applied to participants at sorted positions `threshold+1` through `2*threshold`. Participants occupying the first `threshold+1` sorted positions have their values accepted as the interpolation basis without any verification. A malicious participant who selects a sufficiently small participant ID can guarantee placement in this unchecked range and submit an arbitrary `big_r_me`, corrupting the shared nonce point `big_r` for all honest parties and producing an unusable presignature.

---

### Finding Description

In `do_presign` (`src/ecdsa/robust_ecdsa/presign.rs`), after all participants exchange their `big_r_me = g^{k_me}` values, the protocol performs an exponent interpolation check to detect inconsistent submissions:

```rust
// Lines 186–213
let (threshold_plus1_identifiers, _) = identifiers
    .split_at_checked(threshold + 1) ...;
let (threshold_plus1_verifying_shares, _) = verifying_shares
    .split_at_checked(threshold + 1) ...;

for (identifier, verifying_share) in identifiers
    .iter()
    .skip(threshold + 1)          // ← only checks positions threshold+1 .. 2*threshold
    .zip(verifying_shares.iter().skip(threshold + 1))
{
    let big_r_i = PolynomialCommitment::eval_exponent_interpolation(
        threshold_plus1_identifiers,
        threshold_plus1_verifying_shares,
        Some(identifier),
    )?;
    if big_r_i != *verifying_share {
        return Err(ProtocolError::AssertionFailed(
            "Exponent interpolation check failed.".to_string(),
        ));
    }
}
```

The `identifiers` and `verifying_shares` vectors are populated from a `ParticipantMap`, which stores entries at each participant's **sorted index** (see `ParticipantMap::put` in `src/participants.rs`). The first `threshold+1` entries in sorted order are taken as the interpolation basis and are **never verified against any external reference**. Only the remaining `threshold` entries (positions `threshold+1` through `2*threshold`) are checked for consistency with that basis.

The identical structural flaw appears in the `big_w_me` check:

```rust
// Lines 274–291
for (identifier, wshare) in identifiers
    .iter()
    .skip(threshold + 1)          // ← same bypass
    .zip(wshares.iter().skip(threshold + 1))
```

Because `Participant` IDs are caller-supplied (`Participant::from(u32)`) and `ParticipantList::new` simply sorts them, a malicious participant can choose a small ID (e.g., `Participant::from(0u32)`) to guarantee placement in the first `threshold+1` sorted positions. Their submitted `big_r_me` value is then used as part of the interpolation basis with no verification whatsoever.

---

### Impact Explanation

A malicious participant in the first `threshold+1` sorted positions submits an arbitrary `big_r_me'` (not equal to `g^{k_me}`). Because this value is part of the interpolation basis, the interpolated nonce point `big_r` computed by **all** honest parties is wrong. The presign output `PresignOutput { big_r, alpha, beta, c, e }` is therefore corrupted. During the subsequent signing phase, `do_sign_coordinator` computes a signature using the wrong `big_r` and calls `sig.verify(&public_key, &msg_hash)`, which fails, aborting the signing protocol. Honest parties accept and store a cryptographically unusable presignature, and signing is permanently denied for that presignature slot.

This matches the allowed High impact: **Corruption of presign outputs so honest parties accept unusable cryptographic outputs**, and **Permanent denial of signing for honest parties under valid protocol inputs**.

---

### Likelihood Explanation

The attack is straightforward and reliable:
1. The attacker registers with a small participant ID (e.g., `0`) to be in the first `threshold+1` sorted positions — this is always possible since IDs are caller-supplied.
2. During presign Round 2, the attacker sends a random or identity group element as `big_r_me` instead of `g^{k_me}`.
3. No check in the protocol detects this; the corrupted value is silently accepted as part of the interpolation basis.
4. Every subsequent signing attempt using this presignature fails.

The attacker needs no special privileges beyond being a registered participant. The attack is deterministic and repeatable.

---

### Recommendation

The exponent interpolation check must be applied to **all** participants, not just those beyond position `threshold+1`. One correct approach is to verify that all `2*threshold+1` received `big_r_me` values are mutually consistent by checking each one against the polynomial defined by the others. Concretely, for each participant `i`, interpolate the polynomial from the remaining `threshold+1` points and verify that the interpolated value at participant `i`'s identifier matches `big_r_me[i]`. Alternatively, use a proper Reed-Solomon error-detection scheme over all `2*threshold+1` points that does not privilege any fixed subset as the "trusted basis."

The same fix must be applied to the `big_w_me` consistency check at lines 274–291.

---

### Proof of Concept

**Setup:** 5 participants, `max_malicious = 2`, so `threshold = 2` and `N = 5`. Sorted participant IDs: `[0, 1, 2, 3, 4]`. First `threshold+1 = 3` positions: participants `0, 1, 2`.

**Attack:**
- Participant `0` (malicious, in position 0 of sorted list) sends `big_r_me = g` (the generator, not `g^{k_0}`) during Round 2.
- The `signingshares_map` and `verifyingshares_map` fill in sorted order; participant `0`'s corrupted `big_r_me` occupies index 0.
- The check loop `identifiers.iter().skip(threshold + 1)` skips indices 0, 1, 2 entirely — participant `0`'s value is never verified.
- The interpolation basis `threshold_plus1_verifying_shares` contains the corrupted `big_r_me` from participant `0`.
- The interpolated `big_r` is wrong.
- Participants `3` and `4` (honest, at positions 3 and 4) have their values checked against the corrupted basis. If their honest values happen to be consistent with the corrupted polynomial (unlikely but possible), the presign completes with a wrong `big_r`. If not, the protocol aborts with "Exponent interpolation check failed" — falsely blaming honest participants.
- In either case, the signing phase fails: either the signature does not verify, or the presign never completes. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
