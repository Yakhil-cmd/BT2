Audit Report

## Title
Unbounded Synchronous BFS in SNS `cast_vote_and_cascade_follow` Enables Governance DoS - (File: `rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance function `cast_vote_and_cascade_follow` executes an unbounded BFS over the neuron follow graph within a single synchronous message, with no instruction-limit guard and no mechanism to resume across multiple messages. An attacker who creates enough SNS neurons and configures a sufficiently large follow chain can cause every `register_vote` and `make_proposal` call to trap, permanently halting governance for the affected SNS. The NNS governance equivalent explicitly checks `is_over_instructions_limit()` at each BFS step and resumes across timer ticks; SNS governance has no analogous protection.

## Finding Description
`cast_vote_and_cascade_follow` (lines 3687–3837 of `rs/sns/governance/src/governance.rs`) performs BFS via an outer `while !induction_votes.is_empty()` loop. Each iteration processes one BFS tier: it resolves ballot entries via `BTreeMap::get_mut`, performs string conversions, and calls `vote_from_ballots_following` per follower neuron. There is no call to any instruction-limit check anywhere in this function or its callers within SNS governance.

This function is invoked synchronously in two user-reachable paths:
- `make_proposal` (line 3658): the proposer's initial `Yes` vote is immediately cascaded inline before the proposal is even inserted.
- `register_vote` (line 3931): any direct vote triggers the full cascade before returning.

Additionally, `compute_ballots_for_new_proposal` (lines 5255–5280) iterates over `self.proto.neurons` unconditionally to build the electoral roll, adding further per-neuron cost to `make_proposal`.

By contrast, NNS governance's `continue_processing` in `rs/nns/governance/src/voting.rs` (lines 506–573) calls `is_over_instructions_limit()` after processing each neuron and returns early, persisting intermediate state for resumption in subsequent timer ticks. The `is_over_instructions_limit` symbol does not appear anywhere in SNS governance source files.

SNS governance does have a configurable `max_number_of_neurons` parameter, but this is set per-SNS and may be configured to large values. Even at moderate neuron counts, the combined cost of the ballot-computation loop (all neurons) plus the BFS cascade (all neurons in the follow chain) can approach or exceed the IC's per-message instruction limit (~40 billion instructions for update calls), particularly given the non-trivial per-neuron work (BTreeMap lookups, string allocations, `vote_from_ballots_following` logic).

## Impact Explanation
If either the ballot-computation loop or the BFS cascade exceeds the per-message instruction limit, the message traps. Because both `make_proposal` and `register_vote` are synchronous update calls that invoke `cast_vote_and_cascade_follow` inline, a sufficiently large follow graph causes every vote and every proposal submission to trap. No state is committed on a trap, so the DoS is persistent. This constitutes an application/platform-level DoS of SNS governance — a concrete, non-volumetric availability impact matching the **High ($2,000–$10,000)** bounty tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS" and "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
SNS neuron creation is permissionless for any holder of sufficient SNS tokens. Follow relationships are freely configurable by neuron owners. An attacker needs only to create a linear follow chain (neuron[1] follows neuron[0], neuron[2] follows neuron[1], …) on a single `function_id`. The required chain length depends on the SNS's `max_number_of_neurons` setting and the per-neuron instruction cost; for SNS instances with large neuron caps, the threshold is reachable. The attack is repeatable and requires no special privileges beyond token ownership.

## Recommendation
1. Add an `is_over_instructions_limit()` guard inside the BFS loop in `cast_vote_and_cascade_follow`, returning early when the limit is approached, mirroring the NNS pattern in `rs/nns/governance/src/voting.rs`.
2. Persist in-progress cascade state and resume it in subsequent timer ticks, as NNS governance does via its `VotingStateMachine` and `continue_processing`.
3. Consider enforcing a tighter, well-reasoned cap on `max_number_of_neurons` and `max_followees_per_function` defaults to bound worst-case cascade size.

## Proof of Concept
1. Deploy an SNS with a large `max_number_of_neurons` (e.g., 50,000) and permissionless neuron creation.
2. Create N neurons each staking the minimum amount.
3. Configure a linear follow chain: neuron[i] follows neuron[i-1] on the same `function_id` for all i from 1 to N.
4. Have neuron[0] submit a proposal via `manage_neuron { MakeProposal(...) }`. The `compute_ballots_for_new_proposal` loop iterates all N neurons, then `cast_vote_and_cascade_follow` traverses the full chain of N neurons in one message. With N large enough, the message traps.
5. Repeat with `register_vote` from neuron[0]: the BFS cascade alone traverses all N followers and traps.
6. Observe that no neuron can successfully vote or submit proposals; SNS governance is permanently halted.

        let voting_finished = self.is_voting_finished();

        if !voting_finished {
            while let Some(neuron_id) = self.neurons_to_check_followers.pop_first() {

                // Before we check the next one, see if we're over the limit.
