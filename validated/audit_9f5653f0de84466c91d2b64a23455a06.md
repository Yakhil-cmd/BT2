### Title
Signature Validation Bypass via `skip_stateful_validations` Accepting Any Mempool Transaction as Deploy-Account Proof — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` skips the `__validate__` entry-point call for an Invoke transaction with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true` for the sender. That helper returns `true` for **any** transaction from the account in the pool — not specifically a deploy-account transaction. An unprivileged attacker who observes a legitimate deploy-account transaction in the mempool can immediately submit an Invoke transaction with nonce=1 for the same address, carrying arbitrary calldata and no valid signature, and the gateway will admit it without ever calling `__validate__`.

---

### Finding Description

The UX feature is documented in the code:

> "Check if validation of an invoke transaction should be skipped due to deploy_account not being processed yet."

The guard that is supposed to confirm a deploy-account exists is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  line 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type from the account — including the legitimate deploy-account transaction submitted by the real owner. The comment in `skip_stateful_validations` claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." That reasoning is circular: the very transaction being admitted right now is the one that would need to have passed validation.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate` is `false`, execution returns immediately after `perform_pre_validation_stage` — the `__validate__` call is never reached:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
``` [4](#0-3) 

`perform_pre_validation_stage` only checks nonce ordering, fee bounds, and balance — it does **not** verify the cryptographic signature:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 355-372
pub fn perform_pre_validation_stage<S: State + StateReader>(
    &self, state: &mut S, tx_context: &TransactionContext,
) -> TransactionPreValidationResult<()> {
    Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
    if self.execution_flags.charge_fee {
        self.check_fee_bounds(tx_context)?;
        verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
    }
    self.validate_proof_facts(&tx_context.block_context, state)?;
    Ok(())
}
``` [5](#0-4) 

**Attack steps:**

1. Alice pre-funds address A with STRK and submits a deploy-account transaction (nonce=0, valid signature). It enters the mempool.
2. Attacker observes Alice's deploy-account tx in the public mempool.
3. Attacker submits an Invoke transaction for address A with nonce=1, arbitrary calldata, and an **invalid or empty signature**.
4. Gateway stateless validation passes (no signature check there).
5. `validate_nonce` passes: `account_nonce=0 ≤ tx_nonce=1 ≤ max_allowed_nonce_gap`.
6. `validate_by_mempool` passes: no duplicate nonce yet for address A at nonce=1.
7. `skip_stateful_validations` returns `true` because Alice's deploy-account tx is in the pool.
8. `__validate__` is **never called**. The transaction is admitted to the mempool.

---

### Impact Explanation

**Mempool admission of an invalid (unsigned) transaction — High.**

Once the attacker's Invoke tx (nonce=1) is in the mempool, two concrete harms follow:

**a) Blocking Alice's first post-deployment invoke.** The mempool enforces duplicate-nonce rejection when fee escalation is disabled:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 768-773
if !self.config.static_config.enable_fee_escalation {
    if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
        return Err(MempoolError::DuplicateNonce { address, nonce });
    };
    return Ok(None);
}
``` [6](#0-5) 

Alice's legitimate nonce=1 invoke tx is rejected with `DuplicateNonce`. Even with fee escalation enabled, the attacker can set an arbitrarily high tip to prevent replacement.

**b) Batcher resource waste / DoS.** The batcher dequeues the attacker's tx, runs it, and `__validate__` fails (invalid signature). The tx is rejected and not included in the block, but batcher CPU and block-building time are consumed. The attack is repeatable for every new deploy-account tx observed in the mempool.

---

### Likelihood Explanation

The mempool is publicly observable. Any deploy-account transaction submitted by any user is visible to an attacker before it is sequenced. The attacker needs only to:
- Read the sender address from the pending deploy-account tx.
- Craft an Invoke tx for that address with nonce=1 (any calldata, any or no signature).
- Submit it to the gateway.

No privileged access, no special knowledge, and no on-chain funds are required. The attack is fully automated and can target every new account deployment on the network.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a **deploy-account** transaction (not just any transaction) is pending for the sender address. For example, expose a dedicated mempool method:

```rust
fn has_pending_deploy_account(address: ContractAddress) -> bool;
```

This method would inspect the pool and return `true` only when the pending transaction for `address` at nonce=0 is of type `DeployAccount`. This closes the gap between the intended invariant ("a deploy-account exists") and the actual check ("any transaction exists").

Alternatively, the gateway could pass the deploy-account transaction hash alongside the invoke transaction (as the `native_blockifier` `PyValidator` already does via `deploy_account_tx_hash`) and verify it is present in the mempool before skipping validation.

### Proof of Concept

```
// State: address A has account_nonce = 0 (not deployed), pre-funded with STRK.

// Step 1: Alice submits a valid deploy-account tx for address A.
//   -> mempool now contains: DeployAccount(A, nonce=0, valid_sig)
//   -> account_tx_in_pool_or_recent_block(A) == true

// Step 2: Attacker submits:
RpcInvokeTransactionV3 {
    sender_address: A,
    nonce: 1,
    calldata: [/* arbitrary: e.g. transfer all funds */],
    signature: [],   // empty / invalid
    resource_bounds: /* valid, above threshold */,
    ...
}

// Step 3: Gateway stateful validation path:
//   validate_nonce(nonce=1, account_nonce=0) -> Ok  (0 <= 1 <= max_gap)
//   validate_by_mempool(nonce=1) -> Ok  (no existing nonce=1 for A)
//   skip_stateful_validations:
//     tx.nonce() == 1 && account_nonce == 0  -> true
//     account_tx_in_pool_or_recent_block(A)  -> true  (Alice's deploy-account is there)
//     returns true  (SKIP __validate__)
//   run_validate_entry_point(skip_validate=true):
//     execution_flags.validate = false
//     perform_pre_validation_stage -> Ok (nonce/fee/balance pass)
//     __validate__ NOT called
//     returns Ok

// Step 4: Attacker's unsigned Invoke(A, nonce=1) is now in the mempool.
//   Alice's subsequent Invoke(A, nonce=1) is rejected: MempoolError::DuplicateNonce
//   (when fee escalation is disabled)
```

The exact code path is: [7](#0-6) 
→ calls `account_tx_in_pool_or_recent_block` which is implemented at: [2](#0-1) 
→ returns `true` for Alice's deploy-account tx, causing `skip_validate=true` to propagate to: [4](#0-3) 
→ where `__validate__` is bypassed entirely.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
