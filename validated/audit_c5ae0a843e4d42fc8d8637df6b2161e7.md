### Title
Cross-COA Signature Replay: `validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's keys signed some `signedData`, then separately checks that a COA at a given `path` owns the given `evmAddress`. Because the function never enforces that `signedData` encodes `evmAddress`, a signature produced for one COA can be replayed to prove ownership of a **different** COA owned by the same Cadence account. This is the direct Flow analog of the Maia DAO H-02 finding: both share the root cause that an identity proof does not bind to the specific context (source chain in Maia DAO; specific COA EVM address in Flow).

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that the Cadence account's keys produced valid signatures over `signedData`.
2. Verifies that the COA resource stored at `path` has the given `evmAddress`.

The function never checks that `signedData` encodes `evmAddress`. The code itself acknowledges this in a comment:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0)

### Impact Explanation

**Concrete attack scenario (Cadence path):**

1. Alice owns a Cadence account with two COAs: COA-A at `/public/coaA` (EVM address `0xAAA`) and COA-B at `/public/coaB` (EVM address `0xBBB`).
2. Alice signs some data `D` for COA-A (e.g., to authenticate with a DeFi protocol).
3. An attacker observes the signature `sig` and the proof `{Address: Alice, CapabilityPath: /public/coaA, KeyIndices: [0], Signatures: [sig]}`.
4. The attacker constructs a modified proof: `{Address: Alice, CapabilityPath: /public/coaB, KeyIndices: [0], Signatures: [sig]}`.
5. The attacker calls `EVM.validateCOAOwnershipProof(Alice, /public/coaB, D, [0], [sig], 0xBBB)`.
6. Step 1 passes: Alice's key signed `D` — valid.
7. Step 2 passes: COA at `/public/coaB` has address `0xBBB` — valid.
8. The function returns `isValid: true`, falsely asserting Alice authorized COA-B for data `D`.

**ERC-1271 impact:** Any EVM protocol relying on `isValidSignature` for COA-B will accept a signature that Alice produced in a completely different context (for COA-A). This enables unauthorized approvals, token transfers, NFT sales, or governance votes on behalf of COA-B. [7](#0-6) 

---

### Likelihood Explanation

- `EVM.validateCOAOwnershipProof` is `access(all)` — any unprivileged Cadence script can call it with arbitrary parameters. No special role or staked node is required.
- The EVM precompile `verifyCOAOwnershipProof` is callable from any EVM transaction or contract.
- The attacker only needs: (a) a target Cadence account that owns multiple COAs, and (b) any previously observed valid signature from that account over any data.
- The code comment explicitly acknowledges the vulnerability exists and is not mitigated at the protocol level.
- As the Flow EVM ecosystem grows, multi-COA accounts will become more common (e.g., contracts holding COAs for different purposes, DAO treasuries, protocol-owned liquidity). [8](#0-7) 

---

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof`. Specifically:

1. **Protocol-level fix:** Before verifying signatures, check that `signedData` contains the bytes of `evmAddress` (e.g., as a prefix or structured encoding). Reject the proof if `evmAddress` is not bound into the signed payload.

2. **Alternative:** Add a domain separation tag that includes `evmAddress` to the `keyList.verify` call, so signatures are scoped to a specific COA address.

3. **Documentation:** Until fixed, clearly document that callers of `validateCOAOwnershipProof` (especially off-chain authentication flows) **must** include `evmAddress` in `signedData`. [3](#0-2)

### Proof of Concept

**Setup:** Alice's Cadence account (`0xALICE`) owns two COAs:
- COA-A stored at `/storage/coaA`, published at `/public/coaA`, EVM address `0xAAAA...`
- COA-B stored at `/storage/coaB`, published at `/public/coaB`, EVM address `0xBBBB...`

**Step 1 — Legitimate signature for COA-A:**
```cadence
// Alice signs hash H for COA-A authentication
// sig = sign(H, aliceKey0, "FLOW-V0.0-user")
// proof_A = {Address: 0xALICE, CapabilityPath: "coaA", KeyIndices: [0], Signatures: [sig]}
```

**Step 2 — Attacker replays against COA-B:**
```cadence
import EVM from 0xEVM

// Attacker constructs modified proof pointing to coaB
// proof_B = {Address: 0xALICE, CapabilityPath: "coaB", KeyIndices: [0], Signatures: [sig]}

access(all) fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: 0xALICE,
        path: /public/coaB,       // <-- changed to COA-B's path
        signedData: H,             // <-- same signed data
        keyIndices: [0],           // <-- same key indices
        signatures: [sig],         // <-- same signature (originally for COA-A)
        evmAddress: 0xBBBB...      // <-- COA-B's EVM address
    )
    // Returns: ValidationResult(isValid: true, problem: nil)
    // Alice never authorized COA-B for data H
}
```

**Step 3 — ERC-1271 exploitation:**
```solidity
// Attacker calls isValidSignature on COA-B's EVM contract
// with H and proof_B (modified to point to coaB)
bytes4 result = ICOA(coaBAddress).isValidSignature(H, proof_B_encoded);
// Returns: 0x1626ba7e (ValidERC1271Signature)
// Any EVM protocol checking this will accept it as COA-B's authorization
