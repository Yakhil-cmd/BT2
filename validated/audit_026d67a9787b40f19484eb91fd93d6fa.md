### Title
Malicious Participant Can Corrupt CKD Output by Sending Unverified Cryptographic Contributions — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The Confidential Key Derivation (CKD) coordinator unconditionally sums participant-supplied `(norm_big_y, norm_big_c)` contributions without any proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a silently corrupted derived key that honest parties accept as valid. This is the direct analog of the external report's pattern: user-supplied objects are accepted without enforcing they were produced from a predefined, validated computation.

---

### Finding Description

``` [1](#0-0) 

Each honest participant is supposed to compute, per `compute_signature_share` (lines 148–181):

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

The coordinator then sums all contributions to reconstruct:

```
Y = Σ norm_big_y_i = (Σ λ_i · y_i) · G
C = Σ norm_big_c_i = msk · H(pk ‖ app_id) + Y · app_sk
```

The application unmasks with `C − app_sk · Y = msk · H(pk ‖ app_id)`.

**The protocol sends no zero-knowledge proof, no commitment, and no consistency check** that ties a participant's `norm_big_c` to their actual key share `x_i` or to the `norm_big_y` they sent. The coordinator has no mechanism to distinguish a correctly computed contribution from an arbitrary group element.

**Exploit path:**

A malicious participant `P_m` deviates from `compute_signature_share` and instead sends:

```
norm_big_y_m = 0_G1          (identity element)
norm_big_c_m = Δ             (arbitrary attacker-chosen G1 point)
```

The coordinator accumulates:

```
Y'  = Σ_{i≠m} λ_i · y_i · G  +  0  =  Y − λ_m · y_m · G
C'  = Σ_{i≠m} λ_i · (x_i · H + y_i · app_pk)  +  Δ
```

The unmasked output becomes:

```
C' − app_sk · Y' = (msk − λ_m · x_m) · H(pk ‖ app_id) + Δ + λ_m · y_m · app_pk
```

This is not `msk · H(pk ‖ app_id)`. The derived key is silently wrong. The coordinator and application have no way to detect the corruption; they accept the output as valid.

There is no `max_malicious` or threshold parameter in the `ckd()` API — the protocol implicitly requires all participants to be honest, yet provides no enforcement: [3](#0-2) 

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable cryptographic output.**

The derived key `msk · H(pk ‖ app_id)` is the sole output of the CKD protocol. If it is corrupted, any downstream use (e.g., deriving application-specific signing keys inside a TEE) silently fails or produces keys that are inconsistent with the master public key. Honest parties — the coordinator and the application — have no signal that the output is wrong. This matches the allowed impact: *"Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs."*

---

### Likelihood Explanation

**High.** Any single participant in the CKD session can trigger this. No special privilege is required beyond being a listed participant. The attack requires only sending two arbitrary group elements instead of the correct computation — a trivial deviation for a participant who controls their own node. There is no detection mechanism, no retry, and no abort signal visible to honest parties.

---

### Recommendation

Add a non-interactive zero-knowledge proof of correct computation to each participant's message. Concretely, each participant should prove in zero knowledge that:

1. `norm_big_y_i = λ_i · y_i · G` for some scalar `y_i`, and
2. `norm_big_c_i = λ_i · x_i · H(pk ‖ app_id) + λ_i · y_i · app_pk`, where `x_i` is consistent with the participant's public key share (derivable from the DKG output's `VerifyingKey`).

A standard Chaum–Pedersen proof of discrete-log equality suffices for (1) and (2) jointly, binding `y_i` across both components and binding `x_i` to the known public share. The coordinator must verify all proofs before summing contributions and abort if any proof fails.

### Proof of Concept

**Setup:** 3 participants `{P1, P2, P3}`, coordinator = `P1`, threshold = 3-of-3 (all honest required). `P3` is malicious.

**Honest execution (expected):**
- Each `P_i` calls `compute_signature_share`, sends `(λ_i · y_i · G, λ_i · (x_i · H + y_i · app_pk))`.
- Coordinator sums → `C − app_sk · Y = msk · H(pk ‖ app_id)`. ✓

**Malicious execution:**
- `P3` skips `compute_signature_share` entirely.
- `P3` sends `norm_big_y = G1::identity()`, `norm_big_c = 42 · G1::generator()` (arbitrary).
- Coordinator sums without verification (lines 50–55 of `protocol.rs`).
- Final output: `C' − app_sk · Y' ≠ msk · H(pk ‖ app_id)`.
- Application receives a wrong derived key with no error, no abort, and no indication of corruption. [4](#0-3)

        me,

        key_pair,

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
