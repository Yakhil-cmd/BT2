### Title
Unauthenticated invoke transaction bypasses `__validate__` entry point via `skip_stateful_validations`, admitted to mempool without signature check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator grants a UX bypass — skipping the blockifier's `__validate__` entry point — to any invoke transaction with `nonce = 1` targeting an account that has a pending transaction in the mempool. Because the `sender_address` field is attacker-controlled and no authorization check is performed, an unprivileged attacker can inject a fake-signature invoke transaction on behalf of any victim account that has a pending `DeployAccount` in the mempool, and the gateway will admit it without ever verifying the signature.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the mempool's fee/nonce checks pass:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:399-410
async fn run_pre_validation_checks(...) -> StatefulTransactionValidatorResult<bool> {
    self.validate_state_preconditions(executable_tx, account_nonce).await?;
    validate_by_mempool(executable_tx, account_nonce, mempool_client.clone()).await?;
    let skip_validate =
        skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
    Ok(skip_validate)
}
``` [1](#0-0) 

`skip_stateful_validations` returns `true` (skip) when:
- The transaction is an `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)` (nonce = 1)
- `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
- `account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true` [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, meaning the blockifier's `__validate__` entry point — which performs signature verification — is never called:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The `sender_address` field in an `RpcInvokeTransaction` is fully attacker-controlled. No prior step in the gateway pipeline authenticates the submitter as the owner of `sender_address`:

- `stateless_tx_validator.validate()` checks size, resource bounds, DA modes — not signature validity.
- `validate_by_mempool` checks for duplicate hash and nonce range — not signature validity.
- `account_tx_in_pool_or_recent_block` checks whether the address has any transaction in the pool or a recent committed block — it does not verify the submitter's identity. [4](#0-3) [5](#0-4) 

**Attack path:**

1. Victim submits a `DeployAccount` transaction for their address `V`. It enters the mempool, so `account_tx_in_pool_or_recent_block(V)` returns `true`.
2. Attacker submits an `RpcInvokeTransaction` with `sender_address = V`, `nonce = 1`, and an arbitrary/invalid signature.
3. Gateway stateless validation passes (size/bounds checks only).
4. `validate_nonce` passes: `account_nonce (0) <= tx_nonce (1) <= max_allowed_nonce_gap`.
5. `validate_by_mempool` passes: no duplicate hash, nonce not too old.
6. `skip_stateful_validations` returns `true` because `nonce == 1`, `account_nonce == 0`, and `account_tx_in_pool_or_recent_block(V) == true`.
7. `run_validate_entry_point` is called with `validate = false` — the `__validate__` entry point is skipped entirely.
8. The attacker's fake-signature invoke is admitted to the mempool at nonce slot 1 for victim address `V`. [6](#0-5) 

### Impact Explanation

The gateway admits an invoke transaction with an invalid/fake signature into the mempool without any signature verification. This breaks the invariant that every transaction in the mempool has passed authorization. Concretely:

- The attacker's fake invoke occupies nonce slot 1 for the victim in the mempool.
- The victim's legitimate nonce-1 invoke is rejected with `DuplicateNonce` (or requires fee escalation to replace the attacker's tx).
- The attacker can continuously re-submit with escalating fees to permanently block the victim's nonce-1 slot.
- The victim's `DeployAccount + Invoke` UX flow — the exact scenario this bypass was designed to support — is disrupted or denied.

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

- **Precondition**: The victim must have a pending `DeployAccount` in the mempool (or a recently committed account). This is the exact scenario the skip-validate feature targets, so it is a normal operational state.
- **Attacker capability**: Any unprivileged network participant can submit an `RpcInvokeTransaction` with an arbitrary `sender_address`. No special access is required.
- **Detectability**: The attack is silent from the gateway's perspective — the fake transaction is accepted with a success response.

Likelihood: **Medium** (requires victim to be in the deploy-account flow, but that is a common and expected state).

### Recommendation

Before returning `true` from `skip_stateful_validations`, verify that the submitter is authorized to act on behalf of `sender_address`. The simplest fix is to require that the `DeployAccount` transaction for the account is present in the mempool **and** was submitted with the same transaction hash that the gateway can correlate to the current invoke's sender. Alternatively, run a lightweight signature pre-check (e.g., verify the ECDSA signature against the account's expected public key derived from the `DeployAccount` constructor calldata) before skipping the full blockifier validation.

A minimal guard: only skip validation if the mempool contains a `DeployAccount` transaction (not just any transaction) for `sender_address`, and that `DeployAccount` was submitted in the same gateway session or carries a verifiable link to the current invoke's signer.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        }
        Ok(())
