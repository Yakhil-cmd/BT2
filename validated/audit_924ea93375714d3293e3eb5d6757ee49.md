### Title
Greedy Regex in `parseOfferData` Silently Extracts Last Offer Blob, Enabling Wrong-Offer Acceptance — (`packages/gui/src/components/offers/OfferImport.tsx`, `packages/gui/src/components/offers2/OfferBuilderImport.tsx`)

### Summary
Both `OfferImport.tsx` and `OfferBuilderImport.tsx` define a `parseOfferData` function that uses a greedy regex to extract an `offer1` bech32 blob from raw file content. Due to the greedy `(?<leading>.*)` group combined with the `s` (dotAll) flag, the regex always extracts the **last** `offer1` token in the file. A crafted `.offer` file containing a legitimate blob followed by a malicious blob will cause the malicious blob to be silently parsed, summarized, and presented to the user for acceptance — with no warning that multiple blobs were present.

### Finding Description

Both files define `parseOfferData` with the same regex: [1](#0-0) [2](#0-1) 

The `s` flag makes `.` match `\n`, so `(?<leading>.*)` greedily consumes the entire string including newlines, then backtracks only far enough to allow the last `offer1[...]` sequence to match. Given input:

```
offer1<LEGITIMATE_BLOB>
offer1<MALICIOUS_BLOB>
```

`leading` captures `offer1<LEGITIMATE_BLOB>\n`, and `offer` captures `offer1<MALICIOUS_BLOB>`. The legitimate blob is silently discarded.

The extracted `offerData` is then passed directly to `getOfferSummary` with no check for multiple blobs and no user warning: [3](#0-2) [4](#0-3) 

The resulting summary is then navigated to for user review and acceptance — the user sees the malicious offer's details, not the legitimate one's. [5](#0-4) 

### Impact Explanation
A user who receives a crafted `.offer` file (e.g., from a trading platform, Discord, or email) and imports it will be shown the summary of the **malicious** offer. If they accept, they execute a trade with wrong assets, wrong amounts, or a wrong counterparty — causing direct asset loss. The deception is compounded because the file appears to be a normal `.offer` file and the first (legitimate) blob is never surfaced to the user.

### Likelihood Explanation
Offer files are routinely shared between untrusted parties. An attacker only needs to distribute a crafted `.offer` file — a standard, expected user action. No special permissions, local access, or key material are required. The regex behavior is 100% deterministic.

### Recommendation
1. **Reject multi-blob files**: After matching, check whether `leading` or `trailing` contains another `offer1[...]` sequence. If so, reject the file with an error.
2. **Use a non-greedy or anchored match**: Change `(?<leading>.*)` to `(?<leading>.*?)` (non-greedy) to extract the **first** blob, and then verify no second blob exists in the remainder.
3. **Validate blob uniqueness**: Count all `offer1[...]` matches in the raw input; if count > 1, reject.
