Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Violates ICP/Cycles Conservation Invariant — (File: `rs/nns/cmc/src/main.rs`)

## Summary

The `burn_and_log` function in the Cycles Minting Canister (CMC) is explicitly designed to swallow all errors from the ICP burn operation. After cycles are successfully minted or deposited, if the ICP ledger is temporarily unavailable, the burn silently fails, cycles are permanently created, the block index is recorded as fully processed (preventing any retry), and the backing ICP remains permanently locked in the CMC subaccount — never burned. This violates the protocol invariant that every minted cycle corresponds to destroyed ICP.

## Finding Description

`burn_and_log` at `rs/nns/cmc/src/main.rs` lines 2014–2048 is declared `async fn burn_and_log(...) -> ()` and explicitly matches on the `call_protobuf` result, logging errors but returning `()` in all cases:

```rust
match res {
    Ok(block) => print(format!("{msg} done in block {block}.")),
    Err((code, err)) => {
        let code = code as i32;
        print(format!("{msg} failed with code {code}: {err:?}"))
    }
}
```

The design comment at line 2015 reads: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."*

Both callers proceed to return `Ok(...)` regardless of burn outcome:

- `process_mint_cycles` (lines 1966–1973): calls `do_mint_cycles`, then `burn_and_log`, then returns `Ok(NotifyMintCyclesSuccess {...})`.
- `process_top_up` (lines 1999–2002): calls `deposit_cycles`, then `burn_and_log`, then returns `Ok(cycles)`.

After these functions return `Ok(...)`, the callers `notify_mint_cycles` (lines 1305–1312) and `notify_top_up` (lines 1214–1222) insert the block index into `blocks_notified` as `NotifiedMint(Ok(...))` or `NotifiedTopUp(Ok(...))`. The `is_transient_error` check at lines 1219 and 1310 only removes the entry for retriable errors — since the result is `Ok(...)`, the block stays permanently recorded as processed. Any subsequent call with the same block index returns the cached success result, making the notification non-replayable.

The exploit path:
1. User sends ICP to CMC subaccount keyed to their principal.
2. User calls `notify_mint_cycles` or `notify_top_up`.
3. CMC successfully mints/deposits cycles.
4. CMC calls `burn_and_log`; the ICP ledger is mid-upgrade and returns `Err(...)`.
5. Error is logged only; `burn_and_log` returns `()`.
6. CMC returns `Ok(...)` to the caller.
7. Block index is stored as `NotifiedMint(Ok(...))` — permanently consumed.
8. Result: cycles exist in circulation; ICP remains unburned in CMC subaccount; ICP total supply is inflated relative to cycles minted.

No existing guard prevents this. The `is_transient_error` mechanism only allows retry on error results, not on success results where the burn silently failed.

## Impact Explanation

This is a concrete NNS/ledger protocol conservation violation. Every cycle minted without a corresponding ICP burn inflates the effective ICP supply relative to cycles in circulation. The ICP is permanently locked in the CMC subaccount — the user cannot recover it (notification consumed), and it is never burned. This matches the **High** impact class: *"Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm."* The per-transaction impact is bounded by the amount sent, but the violation is permanent and accumulates over the protocol's lifetime.

## Likelihood Explanation

The ICP ledger undergoes routine canister upgrades on the NNS subnet. During upgrades, the ledger is transiently unavailable for inter-canister calls. Any `notify_mint_cycles` or `notify_top_up` call that completes the cycles minting step but hits the ledger during its upgrade window will trigger this condition. No special privileges are required — any unprivileged user can trigger this by timing a notification call during a ledger upgrade. The probability per individual call is low, but the condition is deterministically reproducible and non-negligible at scale.

## Recommendation

Track failed burns in persistent CMC state (e.g., a `Vec<(Subaccount, Tokens)>` of pending burns). A timer task should periodically retry pending burns against the ledger. This achieves both goals: replay-attack prevention (the notification block is still consumed on first success) and conservation correctness (the burn is retried until it succeeds). Alternatively, expose a permissionless `retry_burn` endpoint that accepts a block index and re-attempts the burn if the subaccount still holds the expected balance.

## Proof of Concept

1. Deploy a local replica with the NNS (CMC + ICP ledger).
2. Send 1 ICP to the CMC subaccount for a test principal.
3. Intercept or mock the ICP ledger's `send_pb` endpoint to return a transient error (simulating an upgrade).
4. Call `notify_mint_cycles` with the block index of the transfer.
5. Observe: `do_mint_cycles` succeeds (cycles deposited to cycles ledger), `burn_and_log` logs an error and returns `()`, `notify_mint_cycles` returns `Ok(NotifyMintCyclesSuccess {...})`.
6. Verify: the block index is in `blocks_notified` as `NotifiedMint(Ok(...))` — retry is blocked.
7. Verify: the CMC subaccount still holds 1 ICP on the ICP ledger (not burned).
8. Verify: the cycles ledger shows the minted cycles credited to the test principal.

This can be implemented as a PocketIC integration test by injecting a mock ledger canister that returns `Err` on `send_pb` calls during the burn phase.
