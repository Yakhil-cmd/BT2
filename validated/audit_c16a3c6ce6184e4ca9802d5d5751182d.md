### Title
Incorrect BLS12-381 Precompile Addresses (EIP-2537 Off-by-One Shift) — (`engine-precompiles/src/bls12_381/`)

---

### Summary

Six of the seven BLS12-381 precompiles in Aurora Engine are registered at addresses that are shifted one position below the EIP-2537 specification. The `G1MUL` and `G2MUL` precompiles are absent entirely, causing every subsequent precompile to occupy the wrong address slot. Any EVM contract that calls BLS12-381 precompiles at their canonical EIP-2537 addresses will invoke the wrong operation, most critically causing BLS pairing-check calls (used for signature verification) to silently dispatch to `MapFp2ToG2` instead.

---

### Finding Description

EIP-2537 (finalized for Prague/Pectra) defines nine BLS12-381 precompiles at addresses `0x0b`–`0x13`:

| Address | EIP-2537 Precompile |
|---------|---------------------|
| 0x0b | BLS12_G1ADD |
| 0x0c | BLS12_G1MUL |
| 0x0d | BLS12_G1MSM |
| 0x0e | BLS12_G2ADD |
| 0x0f | BLS12_G2MUL |
| 0x10 | BLS12_G2MSM |
| 0x11 | BLS12_PAIRING_CHECK |
| 0x12 | BLS12_MAP_FP_TO_G1 |
| 0x13 | BLS12_MAP_FP2_TO_G2 |

Aurora Engine registers only seven precompiles, skipping `G1MUL` and `G2MUL`, and packs them consecutively starting at `0x0b`:

| Address in Code | Aurora Precompile | Correct EIP-2537 Address |
|-----------------|-------------------|--------------------------|
| 0x0b | `BlsG1Add` | 0x0b ✓ |
| **0x0c** | **`BlsG1Msm`** | **0x0d** ✗ |
| **0x0d** | **`BlsG2Add`** | **0x0e** ✗ |
| **0x0e** | **`BlsG2Msm`** | **0x10** ✗ |
| **0x0f** | **`BlsPairingCheck`** | **0x11** ✗ |
| **0x10** | **`BlsMapFpToG1`** | **0x12** ✗ |
| **0x11** | **`BlsMapFp2ToG2`** | **0x13** ✗ |

No unit test verifies that any BLS12-381 address matches its EIP-2537 canonical value (contrast with the Aurora-specific precompiles such as `cross_contract_call`, `prepaid_gas`, and `predecessor_account`, which all have `assert_eq!(ADDRESS, near_account_to_evm_address(...))` tests). [7](#0-6) 

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

The most critical mismatch is `BlsPairingCheck` at `0x0f` instead of `0x11`. BLS aggregate-signature verification contracts call `0x11` expecting a 32-byte pairing result (`0x00…01` for valid, `0x00…00` for invalid). On Aurora they instead invoke `BlsMapFp2ToG2`, which:

- Expects exactly `PADDED_FP2_LENGTH` (128) bytes of input.
- A pairing-check input is `384 × k` bytes (minimum 384 bytes).
- `BlsMapFp2ToG2` returns `ExitError::Other("ERR_BLS_MAP_FP2_TO_G2_LEN")` for any input that is not exactly 128 bytes. [8](#0-7) 

The precompile call reverts, causing the calling contract to revert. Any contract that gates fund withdrawals or state transitions behind a BLS signature check will be permanently unable to execute those paths, freezing the funds it controls.

Additionally, `G1MUL` (0x0c) and `G2MUL` (0x0f) are entirely absent; calls to those addresses fall through to the EVM as ordinary (empty) contract calls, returning empty data and consuming no gas beyond the call overhead — silently producing wrong results for any contract that interprets the empty return as success.

---

### Likelihood Explanation

**High.** Aurora is marketed as Ethereum-compatible. EIP-2537 is a finalized Prague EIP. Any Solidity library or contract that uses BLS12-381 (e.g., for BLS aggregate signatures, zkSNARK verifiers, or threshold cryptography) will use the canonical EIP-2537 addresses. Such contracts deployed on Aurora will silently call the wrong precompile on every BLS operation.

---

### Recommendation

1. Add the missing `BlsG1Mul` precompile at `make_address(0, 0x0C)` and `BlsG2Mul` at `make_address(0, 0x0F)`.
2. Update the remaining precompile addresses to match EIP-2537:
   - `BlsG1Msm` → `make_address(0, 0x0D)`
   - `BlsG2Add` → `make_address(0, 0x0E)`
   - `BlsG2Msm` → `make_address(0, 0x10)`
   - `BlsPairingCheck` → `make_address(0, 0x11)`
   - `BlsMapFpToG1` → `make_address(0, 0x12)`
   - `BlsMapFp2ToG2` → `make_address(0, 0x13)`
3. Add address-verification unit tests (analogous to those for `cross_contract_call::ADDRESS`) asserting each BLS precompile address equals its EIP-2537 value.

---

### Proof of Concept

An unprivileged EVM user deploys or calls any contract that performs BLS aggregate-signature verification:

```solidity
// Calls EIP-2537 PAIRING_CHECK at 0x11
(bool ok, bytes memory result) = address(0x11).staticcall(pairingInput);
// On Aurora: dispatches to BlsMapFp2ToG2, which reverts for input != 128 bytes.
// ok == false, result == abi-encoded "ERR_BLS_MAP_FP2_TO_G2_LEN"
// Signature verification always fails → funds gated by this check are frozen.
```

The attacker-controlled entry path is any `submit` call (standard Aurora EVM transaction entry) that executes EVM bytecode invoking address `0x11`. The incorrect `ADDRESS` constant in `pairing_check.rs` is the necessary vulnerable step: the precompile set lookup uses this constant to register the handler, so the wrong handler is always dispatched.
