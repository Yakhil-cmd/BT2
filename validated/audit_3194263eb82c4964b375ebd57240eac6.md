### Title
Domain Tag Null-Byte Padding Collision Enables Cross-Contract ECDSA Signature Reuse - (File: fvm/crypto/hash.go)

### Summary
The `paddedDomainTag` function in `fvm/crypto/hash.go` right-pads tag strings with zero bytes to a fixed 32-byte length. Because Go strings can contain embedded null bytes, two distinct tag strings that differ only in trailing `\x00` bytes produce identical padded tags. This violates the independence guarantee stated in the code comments and allows a malicious Cadence contract author to accept ECDSA signatures that were intended for a different contract's domain tag.

### Finding Description
The `paddedDomainTag` function converts a string tag into a fixed 32-byte array:

```go
func paddedDomainTag(s string) ([tagLength]byte, error) {
    var tag [tagLength]byte
    if len(s) > tagLength {
        return tag, fmt.Errorf("domain tag cannot be longer than %d characters, got %s", tagLength, s)
    }
    copy(tag[:], s)
    return tag, nil
}
``` [1](#0-0) 

Because `copy` fills only the bytes occupied by the string and leaves the rest as zero, the tag `"my-app"` (6 bytes) and `"my-app\x00"` (7 bytes) both produce the identical 32-byte padded array. The code comment at line 18 explicitly claims "Prefixes are padded tags till 32 bytes to **guarantee** prefixedHashers are independent hashers," but this guarantee is violated for any pair of tags sharing the same prefix and differing only in trailing null bytes. [2](#0-1) 

This collision is confirmed by the existing test in `fvm/crypto/crypto_test.go` (lines 233–238), which asserts that a signature made with tag `"padding test"` successfully verifies with tag `"padding test\x00\x00\x00\x00\x00"` for ECDSA — the test expects `require.True(t, sigOk)`: [3](#0-2) 

The `VerifySignatureFromRuntime` function accepts an arbitrary user-supplied `tag` string from Cadence scripts and passes it directly to `NewPrefixedHashing` → `paddedDomainTag`. There is no validation preventing null bytes in the tag: [4](#0-3) 

This function is reachable from any Cadence script or contract via `CryptoLibrary.VerifySignature`: [5](#0-4) 

Note: the collision does **not** affect BLS/KMAC128 — the BLS test at lines 158–163 correctly expects `False` for the same null-byte-padded tag pair, because KMAC128 does not use `paddedDomainTag`. [6](#0-5) 

### Impact Explanation
A malicious Cadence contract author (Contract B) deploys a contract using a domain tag equal to a legitimate contract's tag (Contract A) with one or more trailing null bytes appended. Any ECDSA (P-256 or secp256k1) signature a user creates for Contract A's domain tag will also verify successfully in Contract B. If the signed data is not context-specific (e.g., a user-provided message, a generic hash, or any data meaningful in both contexts), Contract B can accept and act on signatures the user never intended for it. This enables unauthorized cross-contract signature reuse, potentially leading to unauthorized on-chain asset transfers or account mutations.

### Likelihood Explanation
The attacker only needs to be a Cadence contract author — a low barrier on Flow. The attack requires knowing the target contract's domain tag (typically visible in contract source) and deploying a contract with a null-byte-padded version of that tag. The signed data must be reusable across both contexts, which is plausible for generic message-signing patterns. The collision is deterministic and requires no brute force.

### Recommendation
Normalize tag strings before padding: reject any tag string that contains embedded null bytes (`\x00`), or use a length-prefixed encoding instead of null-byte padding. This ensures two distinct tag strings always produce distinct padded tags, restoring the independence guarantee. Alternatively, document explicitly that tags must not contain null bytes and enforce this at the Cadence runtime level when `domainSeparationTag` is supplied by a contract.

### Proof of Concept
1. Contract A uses domain tag `"my-app"` (6 bytes) to verify user ECDSA signatures.
2. Attacker deploys Contract B using domain tag `"my-app\x00"` (7 bytes, trailing null byte).
3. Both tags produce the identical 32-byte padded array after `paddedDomainTag` in `fvm/crypto/hash.go`.
4. User signs data `D` with tag `"my-app"` for Contract A.
5. Attacker calls Contract B's verify path with the same signature and data `D`.
6. `VerifySignatureFromRuntime` in Contract B uses tag `"my-app\x00"`, which pads to the same 32 bytes as `"my-app"`, and the signature verifies successfully.
7. Contract B accepts the signature as if the user authorized it for Contract B's context — cross-contract signature reuse achieved.

This is directly confirmed by the existing test at `fvm/crypto/crypto_test.go:233–238`, which shows ECDSA signatures are interchangeable between `"padding test"` and `"padding test\x00\x00\x00\x00\x00"`. [1](#0-0) [7](#0-6) [3](#0-2)

	signature []byte,
	tag string,
	signedData []byte,
	publicKey []byte,
	signatureAlgorithm runtime.SignatureAlgorithm,
	hashAlgorithm runtime.HashAlgorithm,
) (
	bool,
	error,
) {
	defer lib.tracer.StartChildSpan(trace.FVMEnvVerifySignature).End()

	err := lib.meter.MeterComputation(
		common.ComputationUsage{
			Kind:      ComputationKindVerifySignature,
			Intensity: 1,
		},
	)
	if err != nil {
		return false, fmt.Errorf("verify signature failed: %w", err)
	}

	valid, err := crypto.VerifySignatureFromRuntime(
		signature,
		tag,
		signedData,
		publicKey,
		signatureAlgorithm,
		hashAlgorithm,
	)

	if err != nil {
		return false, fmt.Errorf("verify signature failed: %w", err)
	}

	return valid, nil
