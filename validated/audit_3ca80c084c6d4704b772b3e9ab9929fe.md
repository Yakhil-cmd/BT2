### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the base spread fee, but `setPoolBinAdditionalFees` applies per-bin additional fees with **no cap check**. Because the swap engine adds `addFeeBuyE6`/`addFeeSellE6` directly on top of the base fee, a pool admin who has already reached the admin spread cap can still push the effective per-bin fee above that cap, extracting excess cost from every trader who crosses the affected bin.

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` correctly enforces the admin spread cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`MetricOmmPoolFactory.setPoolBinAdditionalFees`, callable by the same pool admin, performs **no analogous check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool-level `setBinAdditionalFees` only validates the bin index, not the fee magnitude: [3](#0-2) 

During every swap, the bin additional fee is added directly on top of the oracle-derived base fee before being passed to `SwapMath`:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern applies to the sell direction (`addFeeSellE6`): [5](#0-4) 

`addFeeBuyE6` and `addFeeSellE6` are stored as `uint16`, allowing values up to 65 535 (≈ 6.55 % in E6 units): [6](#0-5) 

The hard cap on the admin spread component is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %): [7](#0-6) 

A pool admin who sets `adminSpreadFeeE6 = 200_000` (the maximum) and then calls `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65_535` raises the effective buy-side fee for that bin to `200_000 + 65_535 = 265_535` E6 units (≈ 26.55 %), exceeding the 20 % admin cap by more than 6 percentage points. The factory never re-checks the combined total.

### Impact Explanation

Every trader whose swap crosses the affected bin pays a fee above the protocol-enforced maximum. The excess fee is retained in the bin as LP income, so the admin (who may also be an LP) benefits indirectly while traders suffer a direct, quantifiable loss on each swap. The invariant that the admin spread fee is bounded by `maxAdminSpreadFeeE6` is broken.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained to operate within factory-set caps. No timelock, governance vote, or additional approval is required to call `setPoolBinAdditionalFees`; the admin can invoke it at any time after pool creation. The bypass is therefore available to any pool admin immediately and repeatably.

### Recommendation

Add a combined-fee cap check inside `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) that ensures `addFeeBuyE6` and `addFeeSellE6` do not cause the effective per-bin fee to exceed `maxAdminSpreadFeeE6`. At minimum, enforce:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

A stricter approach would check `poolFeeConfig[pool].adminSpreadFeeE6 + addFeeBuyE6 <= maxAdminSpreadFeeE6` so the combined admin-controlled spread never exceeds the cap regardless of the base fee setting.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, base admin spread is now at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries an additional 6.55 % buy and sell fee.
4. A trader swaps through bin 0. The swap engine computes the effective buy fee as `baseFeeX64 + mulDiv(65_535, ONE_X64, 1e6)`, charging ≈ 26.55 % total spread instead of the protocol-intended maximum of 20 %.
5. The excess ≈ 6.55 % is retained in the bin as LP income, directly benefiting the admin if they hold LP shares in that bin.
