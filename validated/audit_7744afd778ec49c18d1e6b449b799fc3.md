### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Unclaimed Yield via Trace Inflation - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace. Because each call to `add_to_delegation_pool` in a distinct epoch appends a new checkpoint to this trace with no upper bound, a pool member who repeatedly adjusts their delegation across many epochs without claiming rewards can grow the trace until a subsequent `claim_rewards` call exceeds the Starknet transaction gas limit, permanently bricking their ability to claim accumulated yield.

### Finding Description

**Root cause — unbounded loop with no iteration cap:**

In `calculate_rewards`, the loop iterates from `entry_to_claim_from` to `pool_member_trace_length` with no maximum bound:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves flag this as unbounded. Each loop iteration performs at least two storage reads (`pool_member_trace.at(...)` and `find_sigma` which reads `cumulative_rewards_trace`), making each iteration gas-heavy.

**Trace growth mechanism:**

`set_member_balance` inserts a new checkpoint keyed at `current_epoch + K`:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    let pool_member_balance = PoolMemberBalanceTrait::new(
        balance: amount,
        cumulative_rewards_trace_idx: self.cumulative_rewards_trace_length() + 1,
    );
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `insert` function only overwrites the last entry if the key matches; otherwise it appends a new checkpoint:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

Therefore, one call to `add_to_delegation_pool` per distinct epoch appends one new entry. There is no mechanism to prune or compact the trace.

**`entry_to_claim_from` only advances on successful `claim_rewards`:**

```cairo
let (mut rewards, updated_entry_to_claim_from) = self
    .calculate_rewards(
        :pool_member,
        from_checkpoint: pool_member_info.reward_checkpoint,
        :until_checkpoint,
        entry_to_claim_from: pool_member_info.entry_to_claim_from,
    );
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [4](#0-3) 

If `claim_rewards` reverts due to gas exhaustion, `entry_to_claim_from` is never updated, so every subsequent attempt must re-iterate the same oversized range. The freeze is permanent.

**Exploit path:**

1. Pool member calls `add_to_delegation_pool` with a minimal amount (1 token) once per epoch across N epochs, never calling `claim_rewards`.
2. Each call in a new epoch appends one entry to `pool_member_epoch_balance`.
3. After N epochs, calling `claim_rewards` triggers `calculate_rewards`, which loops N times with multiple storage reads per iteration.
4. Once N is large enough to exhaust the transaction gas limit, `claim_rewards` reverts on every attempt.
5. The pool member's accumulated yield is permanently frozen with no recovery path.

The same unbounded loop is also triggered by the `pool_member_info_v1` view function: [5](#0-4) 

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

Once the trace grows beyond the gas-limit threshold, the pool member can never successfully call `claim_rewards`. There is no partial-claim mechanism, no trace-pruning function, and no administrative escape hatch. All accumulated STRK rewards are permanently locked in the pool contract with no path to recovery.

### Likelihood Explanation

**Low-to-medium.** The attacker must be the pool member themselves (only the pool member or their reward address can call `add_to_delegation_pool`): [6](#0-5) 

Growing the trace requires one transaction per epoch. The number of epochs needed to exceed gas limits depends on Starknet's per-transaction gas ceiling and the cost of each loop iteration (two storage reads minimum). For a delegator who actively manages their position over a long protocol lifetime without ever claiming rewards, this threshold is reachable. The scenario is most realistic for a delegator who deliberately or inadvertently defers reward claims across hundreds of epochs while frequently topping up their delegation.

### Recommendation

1. **Cap the loop** — introduce a `MAX_ITERATIONS` constant and break after processing that many entries per call, storing the updated `entry_to_claim_from` so subsequent calls continue from where the previous one stopped (partial-claim pattern).
2. **Enforce a minimum claim frequency** — require `claim_rewards` to be called before `add_to_delegation_pool` if the unclaimed trace window exceeds a threshold.
3. **Trace compaction** — after a successful `claim_rewards`, compact all processed entries below `entry_to_claim_from` into a single baseline entry to prevent unbounded growth.

### Proof of Concept

```
Epoch 1:  delegator calls add_to_delegation_pool(amount=1)  → trace length = 1
Epoch 2:  delegator calls add_to_delegation_pool(amount=1)  → trace length = 2
...
Epoch N:  delegator calls add_to_delegation_pool(amount=1)  → trace length = N
Epoch N+1: delegator calls claim_rewards()
  → calculate_rewards loops N times, each iteration reads 2+ storage slots
  → transaction exceeds gas limit → reverts
Epoch N+2: delegator calls claim_rewards() again
  → entry_to_claim_from was never updated → same N-iteration loop → reverts again
  → yield permanently frozen
```

The minimum token cost per epoch is 1 unit of the pool's token. The attack is self-directed but results in permanent loss of all accumulated yield with no protocol-level recovery mechanism.
