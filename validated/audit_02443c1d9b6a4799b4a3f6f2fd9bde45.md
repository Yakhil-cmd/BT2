### Title
`MsgUpdateTokenMapping.ValidateBasic` unconditionally rejects empty `Contract`, permanently blocking token-mapping deletion via the direct message path - (File: x/cronos/types/messages.go)

### Summary

`RegisterOrUpdateTokenMapping` in the keeper explicitly treats an empty `Contract` field as a signal to delete an existing non-source token mapping. However, `MsgUpdateTokenMapping.ValidateBasic` unconditionally calls `common.IsHexAddress(msg.Contract)` — which returns `false` for an empty string — and rejects the message before it ever reaches the keeper. The deletion path is therefore permanently unreachable via the `MsgUpdateTokenMapping` gRPC surface.

### Finding Description

`RegisterOrUpdateTokenMapping` contains an intentional delete branch for non-source denoms:

```go
// x/cronos/keeper/keeper.go
} else {
    if len(msg.Contract) == 0 {
        // delete existing mapping
        k.DeleteExternalContractForDenom(ctx, msg.Denom)
    } else {
``` [1](#0-0) 

However, `MsgUpdateTokenMapping.ValidateBasic` unconditionally rejects any message where `Contract` is not a valid EVM hex address:

```go
// x/cronos/types/messages.go
if !common.IsHexAddress(msg.Contract) {
    return errors.Wrapf(sdkerrors.ErrInvalidRequest, "invalid contract address (%s)", msg.Contract)
}
``` [2](#0-1) 

`common.IsHexAddress("")` returns `false`, so an empty `Contract` always fails `ValidateBasic`. The Cosmos SDK `ValidateBasicDecorator` ante handler calls `ValidateBasic` on every message before routing, so the msg server's `UpdateTokenMapping` handler — which calls `RegisterOrUpdateTokenMapping` — is never reached with an empty contract.

The msg server comment at line 77 ("// msg is already validated") confirms the keeper relies on `ValidateBasic` having passed: [3](#0-2) 

By contrast, the legacy governance path (`TokenMappingChangeProposal.ValidateBasic`) correctly allows an empty contract for non-source denoms:

```go
// non-source mappings allow empty contract (delete) or a valid hex address
if len(tcp.Contract) > 0 && !common.IsHexAddress(tcp.Contract) {
    return fmt.Errorf("invalid contract address: %s", tcp.Contract)
}
``` [4](#0-3) 

This asymmetry means the direct `MsgUpdateTokenMapping` path — including accounts granted `CanChangeTokenMapping` permission — can never delete a non-source token mapping, while the governance proposal path can.

### Impact Explanation

Any account holding `CanChangeTokenMapping` permission (including the Cronos admin) is permanently unable to delete a non-source (IBC/gravity) token mapping via `MsgUpdateTokenMapping`. If a mapped CRC20/CRC21 contract becomes compromised, is upgraded to a malicious implementation, or must be rotated, the mapping cannot be removed through the direct administrative path. IBC vouchers received by users will continue to be converted through the stale or compromised contract binding. This constitutes **High: Corruption of token mappings, denom/contract binding, IBC channel/accounting state with direct security impact**, and **High: Permanent inability for honest users to process valid bridge/conversion flows under normal network assumptions**.

### Likelihood Explanation

The `CanChangeTokenMapping` permission exists precisely to allow fast administrative response without governance latency. Any operator who attempts to delete a non-source mapping via `MsgUpdateTokenMapping` with an empty `Contract` will receive a consistent `ErrInvalidRequest` error. The only workaround is the governance proposal path, which introduces significant delay and is not always available (e.g., if governance is paused or quorum cannot be reached).

### Recommendation

In `MsgUpdateTokenMapping.ValidateBasic`, add a bypass for the empty-contract case when the denom is not a source coin, mirroring the logic already present in `TokenMappingChangeProposal.ValidateBasic`:

```go
func (msg *MsgUpdateTokenMapping) ValidateBasic() error {
    // ... sender and denom checks ...

    if !IsSourceCoin(msg.Denom) {
        // empty contract is valid: signals deletion of existing mapping
        if len(msg.Contract) > 0 && !common.IsHexAddress(msg.Contract) {
            return errors.Wrapf(sdkerrors.ErrInvalidRequest, "invalid contract address (%s)", msg.Contract)
        }
    } else {
        if !common.IsHexAddress(msg.Contract) {
            return errors.Wrapf(sdkerrors.ErrInvalidRequest, "invalid contract address (%s)", msg.Contract)
        }
    }
    return nil
}
```

### Proof of Concept

1. Admin grants `CanChangeTokenMapping` to account `A`.
2. Account `A` submits `MsgUpdateTokenMapping{Sender: A, Denom: "gravity0x...", Contract: ""}` to delete an existing mapping.
3. The Cosmos SDK ante handler calls `msg.ValidateBasic()`.
4. `common.IsHexAddress("")` returns `false`; `ValidateBasic` returns `ErrInvalidRequest: invalid contract address ()`.
5. The transaction is rejected. `RegisterOrUpdateTokenMapping` is never called. The mapping persists indefinitely.
6. The same operation submitted as a `TokenMappingChangeProposal` with empty `Contract` passes `ValidateBasic` (line 61 of `proposal.go`) and successfully calls `DeleteExternalContractForDenom`. [5](#0-4) [6](#0-5) [7](#0-6)

// ValidateBasic validates the parameter change proposal
func (tcp *TokenMappingChangeProposal) ValidateBasic() error {
	if err := govtypes.ValidateAbstract(tcp); err != nil {
		return err
	}

	if !IsValidCoinDenom(tcp.Denom) {
		return fmt.Errorf("invalid coin denom: %s", tcp.Denom)
	}

	if IsSourceCoin(tcp.Denom) {
		// source-denom mappings always require a valid contract address
		if !common.IsHexAddress(tcp.Contract) {
			return fmt.Errorf("invalid contract address for source denom: %s", tcp.Contract)
		}
	} else {
		// non-source mappings allow empty contract (delete) or a valid hex address
		if len(tcp.Contract) > 0 && !common.IsHexAddress(tcp.Contract) {
			return fmt.Errorf("invalid contract address: %s", tcp.Contract)
		}
	}

	return nil
}
