### Title
Permanent ERC-20 Token Destruction When NEP-141 `ft_transfer` Fails Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is disabled (the default production build), ERC-20 tokens burned during a `withdrawToNear` exit are permanently destroyed if the subsequent NEP-141 `ft_transfer` NEAR promise fails. No refund callback is attached to restore the burned tokens. This is a direct structural analog to the Redis state-tracking failure: the EVM-side burn is committed, the cross-chain state update fails silently, and the user loses funds on both sides.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles ERC-20 bridge exits to NEAR. The flow is:

**Step 1 — EVM burn is committed first.**
`EvmErc20.sol::withdrawToNear` calls `_burn(sender, amount)` before calling the precompile. The EVM state change is committed atomically with the EVM transaction. [1](#0-0) 

**Step 2 — Precompile constructs the callback args.**
When `error_refund` is disabled, `refund` is hardcoded to `None`. For a standard ERC-20 exit (non-wNEAR), `transfer_near` is also `None`. [2](#0-1) 

**Step 3 — No callback is attached when both fields are `None`.**
The promise is only a bare `PromiseArgs::Create` (no callback) when `callback_args == ExitToNearPrecompileCallbackArgs::default()`. This is exactly the case when `error_refund` is off and the exit is not a wNEAR unwrap. [3](#0-2) 

**Step 4 — If `ft_transfer` fails, there is no handler.**
`exit_to_near_precompile_callback` is never scheduled. The NEP-141 contract rejects the transfer (e.g., recipient not registered, contract paused, out of gas), but the EVM burn has already been finalized. There is no code path to re-mint the burned tokens. [4](#0-3) 

**Step 5 — The refund path only exists behind the feature flag.**
`refund_on_error` (which re-mints burned ERC-20 tokens) is only reachable when `error_refund` is compiled in. [5](#0-4) 

**The test suite explicitly confirms this behavior:**

```
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

The `error_refund` feature is optional and is not part of the default `contract` build target: [7](#0-6) 

---

### Impact Explanation

Any ERC-20 token holder who calls `withdrawToNear` with a recipient that is not registered with the NEP-141 contract (or when the NEP-141 contract is paused, or when the promise runs out of gas) will have their ERC-20 tokens permanently burned with zero NEP-141 tokens received. The tokens are destroyed on the EVM side and never appear on the NEAR side — a permanent, irrecoverable loss of user funds. This maps to **Critical — Permanent freezing/destruction of funds**.

---

### Likelihood Explanation

The trigger condition is reachable by any unprivileged ERC-20 token holder:

- Calling `withdrawToNear` with a NEAR account that has never called `storage_deposit` on the NEP-141 contract is a common user mistake.
- NEP-141 contracts enforce storage registration; unregistered recipients cause `ft_transfer` to fail.
- The `error_refund` feature is not enabled in the default `contract` build, meaning this is the live production behavior.

Likelihood: **Medium** — requires a failed `ft_transfer`, which is a realistic and common condition.

---

### Recommendation

Enable the `error_refund` feature in all production contract builds, or unconditionally attach the refund callback in `ExitToNear::run` regardless of the feature flag. The refund path in `exit_to_near_precompile_callback` already exists and is correct; it simply must always be reachable when the `ft_transfer` fails. [8](#0-7) 

---

### Proof of Concept

1. Deploy Aurora with the default `contract` feature (no `error_refund`).
2. Bridge a NEP-141 token to Aurora EVM, receiving ERC-20 tokens.
3. Call `withdrawToNear(recipient_bytes, amount)` on the `EvmErc20` contract, where `recipient` is a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
4. Observe: ERC-20 tokens are burned (`_burn` executes in EVM), the `ft_transfer` NEAR promise fails (unregistered recipient), no `exit_to_near_precompile_callback` is invoked, and no re-mint occurs.
5. Result: ERC-20 balance reduced by `amount`, NEP-141 balance unchanged — tokens permanently destroyed.

This is confirmed by `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs`: [9](#0-8)
