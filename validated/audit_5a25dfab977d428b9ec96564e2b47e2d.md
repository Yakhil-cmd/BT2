### Title
COA Ownership Proof Lacks `evmAddress` Binding in `signedData`, Enabling Cross-Address Signature Replay - (File: fvm/evm/stdlib/contract.cdc)

---

### Summary

`EVM.validateCOAOwnershipProof` does not require `signedData` to commit to the `evmAddress` being claimed. A valid signature over arbitrary `signedData` can be replayed to prove ownership of any COA owned by the same Cadence account, allowing an unprivileged attacker who obtains a victim's signature to impersonate the victim's COA in EVM contracts that use `verifyCOAOwnershipProof` for authorization, leading to cross-VM asset loss.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that the provided `signatures` are valid over `signedData` using the Cadence account's keys (with domain separation tag `"FLOW-V0.0-user"`).
2. Verifies that the COA resource at the provided `path` has an EVM address matching `evmAddress`. [1](#0-0) 

The critical flaw: **`signedData` is never required to include or commit to `evmAddress`**. The two checks are structurally independent. A signature over `signedData` proves the signer controls the Cadence account keys, but it does not prove the signer intended to authorize the specific `evmAddress` being claimed. [2](#0-1) [3](#0-2) 

The code itself acknowledges this design flaw in a comment:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

This is the direct analog to the Taiko `receivedAt` dual-use flaw: in Taiko, a single field served both "proof status" and "suspension" purposes, so manipulating it for suspension inadvertently set proof status. Here, `signedData` serves as the cryptographic commitment for the proof, but it does not bind to the `evmAddress` context, so a signature produced for one COA context is valid in another.

The `validateCOAOwnershipProof` function is `access(all)` and is invoked by the `verifyCOAOwnershipProof` EVM precompile (the Cadence Arch contract), which is callable by any EVM transaction sender. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An attacker (Bob) who obtains a valid signature from a victim (Alice) over any `signedData` — including signatures Alice produced for a completely different purpose using the same `"FLOW-V0.0-user"` domain separation tag (e.g., a Flow transaction signature, which is public on-chain) — can construct a `COAOwnershipProof` pointing to Alice's COA at a chosen `CapabilityPath`. The proof will pass `validateCOAOwnershipProof` because:

- Alice's signature over `signedData` is cryptographically valid.
- The COA at the chosen path does have the claimed `evmAddress`.

Any EVM contract that uses `verifyCOAOwnershipProof` to gate asset withdrawals or privileged operations on a COA will then authorize Bob to act as Alice's COA owner, enabling full drainage of the COA's FLOW balance or bridged assets. This is a cross-VM asset loss impact. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The attack requires:

1. The victim to own **two or more COAs** under the same Cadence account (for cross-address replay), OR the attacker to obtain a signature Alice produced for a different purpose that shares the `"FLOW-V0.0-user"` domain tag (e.g., any public Flow transaction signature).
2. An EVM contract deployed on Flow EVM that uses `verifyCOAOwnershipProof` to authorize asset operations.
3. The attacker to observe or obtain a valid signature from the victim.

Flow transaction signatures are public on-chain, satisfying condition 3 without any social engineering. As the Flow EVM ecosystem grows and more contracts use `verifyCOAOwnershipProof` for authentication, condition 2 becomes increasingly likely. The comment's claim that "Flow wallets historically create at most one COA per account" is a historical observation, not a protocol constraint, and multi-COA accounts are architecturally supported.

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. Concretely:

- Add a pre-condition that verifies `signedData` encodes `evmAddress` (e.g., the first 20 bytes of `signedData` must equal `evmAddress`), or
- Require callers to pass `signedData = keccak256(evmAddress || nonce || timestamp)` and validate this structure inside the function.

This mirrors the fix recommended in the Taiko report: introduce a dedicated field or a reversible/binding computation so that the proof is unambiguously tied to the specific context it authorizes. [7](#0-6) 

---

### Proof of Concept

**Setup:**
- Alice's Cadence account owns two COAs: `COA_A` stored at `/storage/coaA` (published at `/public/coaA`) and `COA_B` stored at `/storage/coaB` (published at `/public/coaB`).
- Alice signs `signedData = 0xdeadbeef...` (32 bytes) for a dApp login challenge associated with `COA_A`.

**Attack:**

1. Bob observes Alice's signature `sig` over `signedData` (e.g., from a public API response or on-chain event).

2. Bob constructs a `COAOwnershipProof`:
   ```
   {
     KeyIndices:     [0],
     Address:        Alice's Cadence address,
     CapabilityPath: /public/coaB,   // points to COA_B, not COA_A
     Signatures:     [sig]           // Alice's signature over signedData
   }
   ```

3. Bob submits an EVM transaction calling a contract `AssetVault` that contains:
   ```solidity
   function withdraw(address coaAddr, bytes32 signedData, bytes calldata proof) external {
       require(arch.verifyCOAOwnershipProof(coaAddr, signedData, proof), "not owner");
       // transfer assets from coaAddr to msg.sender
   }
   ```
   with `coaAddr = COA_B.evmAddress`, `signedData = 0xdeadbeef...`, `proof = RLP(Bob's constructed proof)`.

4. The precompile calls `EVM.validateCOAOwnershipProof`:
   - Signature verification: Alice's `sig` over `signedData` is valid ✓
   - COA address check: the COA at `/public/coaB` has `evmAddress = COA_B.evmAddress` ✓
   - Returns `isValid: true`.

5. `AssetVault` transfers all assets from `COA_B` to Bob. [8](#0-7) [4](#0-3)

					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)

			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err

			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
