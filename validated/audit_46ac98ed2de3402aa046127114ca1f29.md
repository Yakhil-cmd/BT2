### Title
WalletConnect `take_offer` Confirmation Displays Understated Royalty Total When Multiple NFTs Are Involved — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

The `formatAmountWithRoyalties` function in the WalletConnect confirmation pipeline incorrectly divides the fungible amount by the number of NFTs before applying each NFT's royalty percentage. This causes the `amountWithRoyalties` field shown to the user in the approval dialog to be understated by a factor of N (number of NFTs), so the user consents to spending less than the transaction will actually deduct.

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the royalty-inclusive total shown to the user when they are asked to approve a `chia_wallet.take_offer` WalletConnect command:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The function splits the fungible amount equally across all NFTs before multiplying by each royalty rate. The Chia protocol, however, applies each NFT's royalty to the **full** fungible amount independently. The correct formula is:

```
royaltyAmount = Σ (amount × Rᵢ / 10_000)
```

but the code computes:

```
royaltyAmount = Σ ((amount / N) × Rᵢ / 10_000)
```

which is exactly `1/N` of the correct value.

The resulting `amountWithRoyalties` string is attached to the `DisplayWalletDeltaItem` and rendered in the WalletConnect confirmation dialog under "You Spend": [2](#0-1) [3](#0-2) 

The existing test encodes the wrong value as the expected output, confirming the bug is baked into the test suite: [4](#0-3) 

**Concrete numbers from that test:**
- 2 NFTs: royalty 500 (5 %) and royalty 10 (0.1 %)
- Requested XCH: 100,000,000 mojos (0.0001 XCH)
- GUI displays: `splitAmount = 50,000,000`; royalties = 2,500,000 + 50,000 = 2,550,000; total shown = **0.00010255 XCH**
- Protocol actually charges: royalties = 5,000,000 + 100,000 = 5,100,000; true total = **0.0001051 XCH**

The user is shown ~50 % less royalty cost than they will actually pay.

### Impact Explanation

The `amountWithRoyalties` field is the only royalty-inclusive figure presented to the user before they click "Approve" in the WalletConnect confirmation dialog. When a malicious or misconfigured dApp sends a `take_offer` command containing multiple NFTs with royalties, the dialog understates the true XCH cost. The user approves a transaction believing they will spend X XCH but the blockchain deducts X + (N−1)/N × royalties more. This constitutes displaying the wrong amount in a WalletConnect approval flow, matching the High-impact criterion: *"WalletConnect state that causes a user to approve … the wrong … amount."* [5](#0-4) 

### Likelihood Explanation

Any dApp that has an established WalletConnect session can send a `take_offer` request referencing an offer with two or more NFTs that carry royalties. No special privilege beyond an approved WalletConnect session is required. Multi-NFT offers are a supported and documented Chia feature, so the trigger condition is reachable in normal usage.

### Recommendation

Remove the division by `royaltyPercentages.length`. Each NFT's royalty is applied to the full fungible amount:

```typescript
// Before (wrong):
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);

// After (correct):
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectation: with 2 NFTs (royalties 500 and 10) and 100,000,000 mojos, the correct `amountWithRoyalties` is `'0.0001051'`, not `'0.00010255'`.

### Proof of Concept

1. Attacker dApp has an approved WalletConnect session with the victim's Chia GUI wallet.
2. Attacker constructs an offer containing two NFTs: NFT-A (royalty 500 = 5 %) and NFT-B (royalty 500 = 5 %), requesting 1 XCH (1,000,000,000,000 mojos).
3. Attacker sends `chia_wallet.take_offer` via WalletConnect.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties` with `amount = 1_000_000_000_000n` and `royaltyPercentages = [500, 500]`.
5. `splitAmount = 500_000_000_000n`; displayed royalties = 2 × (500,000,000,000 × 500 / 10,000) = 50,000,000,000; displayed total = **1.05 XCH**.
6. Actual protocol royalties = 2 × (1,000,000,000,000 × 500 / 10,000) = 100,000,000,000; true total = **1.1 XCH**.
7. User sees "You Spend: 1.05 XCH" and approves; the blockchain deducts 1.1 XCH — 0.05 XCH more than the user consented to. [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
