### Title
`setPoolAdminFeeDestination` Skips Fee Settlement Before Redirecting Destination, Allowing Pool Admin to Steal Accrued Admin Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` changes `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-mutating function in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before writing new config. The missing flush lets the pool admin atomically redirect all previously earned admin fees — both spread-surplus and stored notional fees — to an address they control.

---

### Finding Description

`MetricOmmPoolFactory` maintains two independent fee-related mappings per pool:

- `poolFeeConfig[pool]` — the split ratios (spread / notional, protocol / admin)
- `poolAdminFeeDestination[pool]` — the address that receives the admin share

`collectFees` in `MetricOmmPool` receives `adminFeeDestination_` as a parameter and sends the admin share directly to whatever address is passed at call time:

```solidity
// MetricOmmPool.sol L416-L421
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
```

`collectPoolFees` (the public entry point) reads `poolAdminFeeDestination[pool]` at the moment of the call and forwards it:

```solidity
// MetricOmmPoolFactory.sol L379-L389
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← live read
    );
}
```

`setPoolAdminFees` and `setPoolProtocolFee` both flush fees to the **current** destination before updating config:

```solidity
// MetricOmmPoolFactory.sol L417-L425  (setPoolAdminFees)
PoolFeeConfig memory c = poolFeeConfig[pool];
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // flush to OLD destination first
);
```

`setPoolAdminFeeDestination` performs no such flush:

```solidity
// MetricOmmPoolFactory.sol L438-L447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

All fees accrued since the last collection — spread surplus sitting in the pool's token balance and notional fees stored in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` — will be sent to the **new** destination on the next `collectPoolFees` call, not to the old one.

---

### Impact Explanation

`adminFeeDestination` is a distinct parameter from `admin` in `createPool`; it is explicitly designed to be a separate beneficiary (e.g., a protocol treasury, a revenue-sharing contract, or a DAO vault). When the pool admin changes the destination without flushing, all fees that accrued while the old destination was registered are silently redirected. The old destination receives nothing for the period it was entitled to. The magnitude equals the full admin share of every swap fee earned since the last collection, which can be arbitrarily large in a high-volume pool.

---

### Likelihood Explanation

The pool admin is semi-trusted and has unilateral, timelock-free authority to call `setPoolAdminFeeDestination` at any block. There is no on-chain mechanism that forces a flush before the destination changes. The admin can front-run any `collectPoolFees` transaction to ensure the redirect captures the maximum accrued balance. The only mitigation available to the old destination is off-chain monitoring, which is not a protocol-level guard.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the current destination before writing the new one.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush all accrued fees to the OLD destination before switching.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed with `admin = ADMIN`, `adminFeeDestination = TREASURY`.
2. The pool accumulates swap fees over time; `notionalFeeToken0Scaled > 0` and a spread surplus exists in the pool balance.
3. `ADMIN` calls `setPoolAdminFeeDestination(pool, ATTACKER_WALLET)` — no flush occurs, `poolAdminFeeDestination[pool]` is now `ATTACKER_WALLET`.
4. Anyone (or `ADMIN`) calls `collectPoolFees(pool)`.
5. `collectFees` inside the pool reads `adminFeeDestination_ = ATTACKER_WALLET` and transfers all accrued admin fees there.
6. `TREASURY` receives zero despite having been the registered destination for the entire accrual period.

The root cause is the missing `collectFees` call in `setPoolAdminFeeDestination`, directly analogous to the external report's finding that `claim` must call `refund` first so that the recipient's true settled amount is used before any state-changing payout is made.
