### Title
Block Processing Halted by Revert in `notify_l2_asset_tracker` External Call — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

`notify_l2_asset_tracker` makes an external EVM call to `L2_ASSET_TRACKER_ADDRESS` (0x1000f) during every L1→L2 transaction's post-execution accounting phase. If that call reverts for any reason, the code explicitly converts the revert into a **fatal internal error that halts block processing entirely**. The `amount` argument passed to the contract is derived directly from attacker-controlled L1 transaction fields (`gas_price`, `gas_limit`, `reserved[0]`). Any reachable revert path inside `L2_ASSET_TRACKER_ADDRESS.handleFinalizeBaseTokenBridgingOnL2` — whether from a bug, an edge-case input, or a future upgrade — can be triggered by a crafted L1 priority transaction, permanently stalling the chain.

---

### Finding Description

`notify_l2_asset_tracker` is invoked up to three times per L1→L2 transaction:

1. **Value mint** — inside `execute_l1_transaction_and_notify_result`, with `amount = to_transfer` (total deposited minus max fee).
2. **Operator payment** — in `process_l1_transaction`, with `amount = pay_to_operator` (gas used × gas price).
3. **Refund** — in `process_l1_transaction`, with `amount = to_refund_recipient` (unused gas × gas price). [1](#0-0) 

Inside `notify_l2_asset_tracker`, `run_single_interaction` is called with `should_make_frame = true` (state changes are isolated), but the **return value** is checked and a revert is escalated to a fatal error: [2](#0-1) 

The comment explicitly states: *"Treated as a fatal system error — block processing cannot continue."*

The callee address is the fixed constant `L2_ASSET_TRACKER_ADDRESS = 0x1000f`: [3](#0-2) 

This address is **not** a system hook (0x7xxx) and **not** a traditional system contract (0x8xxx). It is a regular EVM contract deployed via a protocol upgrade. Its `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` function receives:

- `_fromChainId` — the L1 chain ID (system-level, fixed per deployment).
- `_amount` — derived from attacker-controlled L1 transaction fields.

The three `amount` values are computed as: [4](#0-3) [5](#0-4) 

All three are arithmetic combinations of `gas_price`, `gas_limit`, `gas_used`, and `reserved[0]` (total deposited), all of which are set by the L1 transaction submitter.

The fatal-error path is reached regardless of whether the revert occurs during the value-mint call (inside `execute_l1_transaction_and_notify_result`) or the operator-payment / refund calls (directly in `process_l1_transaction`). In the latter two cases there is no enclosing rollback frame to absorb the error; it propagates straight to the block runner and halts processing.

The design flaw is compounded by the fact that `should_make_frame = true` already rolls back any state changes made by the asset-tracker call on revert, so the chain's token accounting would **not** actually be inconsistent if the revert were handled gracefully. The fatal escalation is therefore unnecessary and creates a single point of failure.

---

### Impact Explanation

If an attacker can craft an L1 priority transaction whose parameters cause `L2_ASSET_TRACKER_ADDRESS.handleFinalizeBaseTokenBridgingOnL2` to revert, the entire block processing pipeline halts with an internal error. No further transactions in the block (or subsequent blocks) can be processed until the operator manually intervenes or the contract is upgraded. This is a chain-halting denial-of-service, analogous to the external report's vault-settlement DoS but with a far wider blast radius.

---

### Likelihood Explanation

The likelihood is **conditional**: it requires a reachable revert path inside `L2_ASSET_TRACKER_ADDRESS`. Given that:

- The `amount` argument spans the full U256 range (attacker controls `gas_price` × `gas_limit` and `reserved[0]`).
- The contract is a non-trivial EVM contract with access-control modifiers (`onlyBaseTokenHolderOrL2BaseToken`) and state-dependent logic (`_needToForceSetAssetMigrationOnL2`).
- The contract is called with `msg.sender = L2_BASE_TOKEN_ADDRESS` (0x800a), which passes the modifier, so the full function body executes.
- Any arithmetic overflow, unexpected state, or future upgrade that introduces a revert path becomes immediately exploitable.

The attack surface is every L1→L2 priority transaction submitted to the chain, which is an unprivileged, externally reachable operation.

---

### Recommendation

1. **Do not treat a revert in `notify_l2_asset_tracker` as a fatal block-halting error.** Because `should_make_frame = true` already rolls back the asset-tracker's state changes, a revert leaves the chain in a consistent state. The correct response is to revert the entire L1 transaction (refund the deposit) and continue block processing.
2. Alternatively, wrap the `notify_l2_asset_tracker` call in its own rollback frame and propagate the revert as a transaction-level failure rather than a block-level failure.
3. Add fuzz/property tests that exercise `notify_l2_asset_tracker` with extreme `amount` values (0, U256::MAX, values near overflow boundaries) to surface revert paths in the asset-tracker contract before deployment.

---

### Proof of Concept

1. Deploy (or upgrade) `L2_ASSET_TRACKER_ADDRESS` (0x1000f) with a `handleFinalizeBaseTokenBridgingOnL2` implementation that reverts when `_amount == X` for some value `X` reachable via normal transaction parameters.
2. Submit an L1 priority transaction with `gas_price` and `gas_limit` chosen so that `pay_to_operator = gas_used × gas_price == X`.
3. `process_l1_transaction` calls `mint_base_token` → `notify_l2_asset_tracker` with `amount = X`.
4. `run_single_interaction` returns `failed = true`.
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. The error propagates through `mint_base_token` → `process_l1_transaction` → the block runner.
7. Block processing halts; no further transactions are executed. [6](#0-5) [7](#0-6)

fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
