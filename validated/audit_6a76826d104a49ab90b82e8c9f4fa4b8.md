Audit Report

## Title
Unchecked Ledger Burn Return Value in CMC `burn_and_log` Causes ICP Supply Invariant Violation - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister silently discards ICP ledger burn failures. Because the notification block is unconditionally committed to a terminal state (`NotifiedTopUp`, `NotifiedCreateCanister`, or `NotifiedMint`) after the primary operation succeeds — regardless of whether the subsequent burn call succeeds — any burn failure permanently locks ICP in CMC's subaccount with no recovery path. Cycles are minted (or a canister is created) without the corresponding ICP being burned, violating the protocol invariant that every minted cycle is backed by burned ICP.

## Finding Description
In `rs/nns/cmc/src/main.rs` at lines 2017–2049, `burn_and_log` calls the ICP ledger's `send_pb` endpoint and on failure only logs the error and returns `()`:

```rust
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    ...
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;
    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
```

This function is called after the primary operation succeeds in all three notify paths:
- `process_create_canister` (L1943–1946): `burn_and_log(sub, amount).await; Ok(canister_id)`
- `process_mint_cycles` (L1966–1973): `burn_and_log(sub, amount).await; Ok(...)`
- `process_top_up` (L1999–2002): `burn_and_log(sub, amount).await; Ok(cycles)`

In each caller, the result of `burn_and_log` is discarded and `Ok(...)` is returned unconditionally. The callers (`notify_top_up`, `notify_create_canister`, `notify_mint_cycles`) then commit the block to a terminal `NotifiedTopUp`/`NotifiedCreateCanister`/`NotifiedMint` status (L1214–1222, L1305–1313, L1411–1419). The `is_transient_error` check that would allow retry only fires on `Err` results; since `process_*` returns `Ok(...)` regardless of burn outcome, the block is always finalized.

The design comment at L2014–2016 explicitly acknowledges this: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* This prevents double-spend retries but creates a permanent supply invariant violation on burn failure.

Once the block is in a terminal state, no code path in the CMC can re-trigger the burn for that subaccount/amount pair. The ICP remains locked in CMC's subaccount indefinitely.

## Impact Explanation
This is a concrete violation of the ICP-to-cycles protocol invariant: cycles are minted and delivered to the recipient, but the corresponding ICP is not burned. The unburned ICP remains permanently locked in CMC's subaccount (inaccessible to any party, including the original sender), inflating the effective ICP supply relative to the outstanding cycles. This constitutes illegal cycles minting — cycles enter circulation without the required ICP destruction — and represents a significant NNS/ledger security impact with concrete protocol harm. Repeated occurrences (e.g., across multiple ledger upgrade windows) compound the supply discrepancy without bound.

## Likelihood Explanation
The ICP ledger undergoes canister upgrades via NNS governance proposals, which are publicly visible and scheduled in advance. During the brief stop/upgrade/start window, inter-canister calls to the ledger return a system-level reject. Any user who has already transferred ICP to CMC's subaccount and submits a `notify_*` call timed to land during this window will trigger the failure path. No special privileges are required: the attacker only needs to monitor the NNS proposal queue and submit a pre-funded notification at the right moment. The window is short but the attack is repeatable across every ledger upgrade cycle.

## Recommendation
Replace the fire-and-forget pattern with a persistent retry mechanism:

1. **Persistent retry queue**: On burn failure, record the pending burn `(subaccount, amount)` in stable state and retry on the next heartbeat/timer tick until the ledger confirms success. This is the most robust fix and aligns with how other critical CMC operations handle transient ledger unavailability.
2. **Pre-burn before operation**: Burn the ICP before creating the canister or minting cycles, and refund on failure — mirroring the existing refund path. This eliminates the window entirely but requires restructuring the operation order.
3. **Propagate and store on failure**: Return an error from `burn_and_log`, and on failure store the failed burn in stable state for administrative recovery or automatic retry, rather than silently discarding it.

## Proof of Concept
1. User transfers 10 ICP to CMC's subaccount derived from their principal on the ICP ledger.
2. User monitors the NNS proposal queue and identifies an upcoming ICP ledger upgrade proposal.
3. User submits `notify_create_canister` (or `notify_top_up` / `notify_mint_cycles`) timed to execute while the ledger canister is stopped mid-upgrade.
4. CMC sets block status to `Processing`, calls `do_create_canister` — this succeeds (canister management is independent of the ledger).
5. CMC calls `burn_and_log(sub, 10 ICP)`. The `call_protobuf(..., "send_pb", ...)` call returns a system reject because the ledger is stopped.
6. `burn_and_log` logs the error and returns `()`. CMC returns `Ok(canister_id)` to the caller.
7. Block status is set to `NotifiedCreateCanister(Ok(canister_id))` — terminal, non-retryable.
8. **Result**: User received a new canister. 10 ICP remains in CMC's subaccount. The ICP is never burned. Total ICP supply is 10 ICP higher than the cycles-backed invariant requires.
9. A deterministic integration test using PocketIC can reproduce this by: (a) setting up CMC + ledger, (b) stopping the ledger canister, (c) calling `notify_create_canister`, (d) restarting the ledger, and (e) asserting that the CMC subaccount balance is non-zero while the canister was successfully created.
