### Title
Zero Native Resource Charge in All EIP-2537 BLS12-381 Precompiles and EIP-152 Blake2F Enables Unmetered Proving-Cost Consumption - (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

Every EIP-2537 BLS12-381 precompile and the EIP-152 Blake2F precompile hardcodes `cost_native = 0`, charging **zero native (proving) resource** regardless of the actual RISC-V cycle cost of the operation. All other computationally expensive precompiles (BN254 pairing, ecrecover, P256, point evaluation) correctly charge a non-zero native cost. An unprivileged caller can exploit this to consume far more proving cycles than the native resource limit was designed to bound, causing prover resource exhaustion and block-proving delays.

---

### Finding Description

ZKsync OS implements a **double resource accounting** model: EVM gas (ergs) and a **native resource** that models RISC-V proving cost. The native resource limit is derived from `gasLimit * nativePerGas` and is intended to bound the actual proving work per transaction.

Every EIP-2537 BLS12-381 precompile sets `cost_native = 0`:

**`pairing.rs`** — `Bls12381PairingCheckPrecompile`:
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
``` [1](#0-0) 

**`msm.rs`** — `Bls12381G1MSMPrecompile` and `Bls12381G2MSMPrecompile`:
```rust
let cost_native = 0;
``` [2](#0-1) [3](#0-2) 

**`addition.rs`** — `Bls12381G1AdditionPrecompile` and `Bls12381G2AdditionPrecompile`: [4](#0-3) [5](#0-4) 

**`mappings.rs`** — `Bls12381G1MappingPrecompile` and `Bls12381G2MappingPrecompile`: [6](#0-5) [7](#0-6)

Contrast this with the BN254 precompiles, which correctly charge substantial native costs:

```rust
pub const BN254_ECADD_NATIVE_COST: u64 = native_with_delegations!(46_000, 1650, 0);
pub const BN254_ECMUL_NATIVE_COST: u64 = native_with_delegations!(600_000, 41_000, 0);
pub const BN254_PAIRING_BASE_NATIVE_COST: u64 = native_with_delegations!(13_000_000, 500_000, 0);
pub const ECRECOVER_NATIVE_COST: u64 = native_with_delegations!(350_000, 43_000, 0);
pub const P256_NATIVE_COST: u64 = native_with_delegations!(500_000, 71_000, 0);
pub const POINT_EVALUATION_NATIVE_COST: u64 = native_with_delegations!(49_900_000, 3_301_000, 0);
``` [9](#0-8) 

BLS12-381 pairing is **more** computationally expensive than BN254 pairing (larger field, more Miller loop iterations), yet charges 0 native while BN254 pairing charges `13,000,000+` native units per pair. The native resource is passed fully from frame to frame with no per-call cap:

> "The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend." [10](#0-9) 

---

### Impact Explanation

The native resource limit is calibrated assuming all operations charge native proportionally to their actual RISC-V proving cost. When BLS12-381 precompiles charge 0 native, the native resource counter is not depleted by these calls. An attacker can exhaust the entire EVM gas budget calling BLS12-381 pairing or MSM operations, consuming orders of magnitude more RISC-V proving cycles than the native limit was designed to allow. This causes:

1. **Prover resource exhaustion**: The prover must execute all these expensive elliptic curve operations, consuming far more cycles than the native resource limit bounds.
2. **Block-proving delays or halts**: If the prover cannot complete within resource bounds, block finalization is delayed.
3. **Sequencer/prover DoS**: Repeated transactions exploiting this can continuously overload the proving pipeline.

The `deltaGas` adjustment mechanism only compensates when native resources are actually consumed — it cannot compensate for native resources that were never charged. [11](#0-10) 

---

### Likelihood Explanation

- The EIP-2537 precompiles are enabled via the `eip-2537` feature flag and are part of the EVM equivalence target (Prague/Pectra upgrade).
- Any unprivileged EOA or contract can call these precompiles directly via `CALL` to addresses `0x0b`–`0x11`.
- No special permissions, governance access, or privileged roles are required.
- The attack requires only a standard transaction with sufficient EVM gas to call the precompile multiple times.
- The BLS12-381 pairing check (`0x0f`) is the most severe target: each pair costs 32,600 EVM gas but 0 native. With a 30M gas block, an attacker can submit ~900 pairs per transaction, each requiring a full BLS12-381 multi-pairing computation.

---

### Recommendation

Assign non-zero native resource costs to all EIP-2537 BLS12-381 precompiles and the EIP-152 Blake2F precompile, following the same pattern used for BN254 and other precompiles. The native cost should be benchmarked against actual RISC-V cycle counts. For example:

```rust
// In pairing.rs
pub const BLS12_381_PAIRING_BASE_NATIVE_COST: u64 = /* benchmark */;
pub const BLS12_381_PAIRING_PER_PAIR_NATIVE_COST: u64 = /* benchmark */;

let cost_native = BLS12_381_PAIRING_BASE_NATIVE_COST
    + (num_pairs as u64) * BLS12_381_PAIRING_PER_PAIR_NATIVE_COST;
resources.charge(&R::from_ergs_and_native(
    cost_ergs,
    <R::Native as zk_ee::system::Computational>::from_computational(cost_native),
))?;
```

Similarly for MSM (input-size-proportional), addition, mapping, and Blake2F (rounds-proportional) operations.

---

### Proof of Concept

1. Deploy a contract with the following loop (pseudocode):
```solidity
// Call BLS12-381 pairing check (0x0f) with maximum pairs per gas budget
// Each pair: 32,600 EVM gas, 0 native resource
// With 30M gas: ~900 pairs, each requiring a full BLS12-381 multi-pairing
for (uint i = 0; i < 900; i++) {
    (bool ok,) = address(0x0f).call{gas: 35000}(validPairInput);
}
```
2. Submit the transaction with a standard gas limit (e.g., 30M gas).
3. The EVM gas is fully consumed, but the native resource counter is barely decremented (only by the loop overhead, not by the pairing operations themselves).
4. The prover must execute ~900 BLS12-381 multi-pairing computations — each more expensive than a BN254 pairing (which would cost `13,000,000` native units) — while the native resource limit was only sized for the EVM gas consumed.
5. Repeated submissions of such transactions continuously overload the prover, causing proving delays.
