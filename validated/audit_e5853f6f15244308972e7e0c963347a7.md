### Title
Integer Division Truncation in `native_per_pubdata` Calculation Silently Zeroes Pubdata Native Charge — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When `pubdata_price < native_price`, the floor division `pubdata_price.wrapping_div(native_price)` truncates to `0`, setting `native_per_pubdata = 0`. This silently makes all pubdata (state diffs) free in terms of native resources for every transaction in that block, even though the operator configured a non-zero pubdata price. Any unprivileged user can then generate maximum-pubdata transactions at no native resource cost, forcing the operator to absorb the L1 data-publishing cost without compensation.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two resource-rate derivations are computed asymmetrically:

```rust
// native_per_gas uses div_ceil — never truncates to 0
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata uses wrapping_div (floor) — truncates to 0 when pubdata_price < native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

The same pattern is replicated in the public API helper:

```rust
// native_per_pubdata = pubdata_price / native_price  (floor)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

When `native_per_pubdata = 0`, two downstream charging sites both produce zero cost:

**1. Intrinsic pubdata overhead (upfront reservation):**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0, no native reserved for intrinsic pubdata
``` [3](#0-2) 

**2. Post-execution pubdata charge:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // → 0 regardless of pubdata_spent
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

The `native_per_gas` path deliberately uses `div_ceil` to prevent a zero result (which would grant unlimited native resources). No equivalent protection exists for `native_per_pubdata`.

---

### Impact Explanation

When `pubdata_price < native_price`:

- `native_per_pubdata` truncates to `0`
- Every transaction in the block pays **zero native resources** for any amount of pubdata it generates
- The operator still must publish all state diffs to L1 and bears that cost entirely
- A user can craft transactions that write to many unique storage slots (maximising pubdata) and pay only EVM gas, not the native resource cost that was supposed to compensate the operator for L1 data fees
- This directly contradicts the double-resource-accounting model described in `docs/double_resource_accounting.md`: *"If execution doesn't run out of native resources, we first charge for pubdata from native resource."* [5](#0-4) 

---

### Likelihood Explanation

`native_price` reflects the cost of a single RISC-V proving cycle; `pubdata_price` reflects the cost of publishing one byte to L1. These are independently set by the operator and denominated in different economic units. It is entirely realistic for `pubdata_price < native_price` — for example, if L1 calldata is cheap relative to proving cycles, or during periods of low L1 gas prices. Any block where this condition holds exposes every transaction to free pubdata. An attacker who monitors the block context oracle can time pubdata-heavy transactions to blocks where the condition is met.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — consistent with native_per_gas, never silently zeroes)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that even when `pubdata_price` is fractionally smaller than `native_price`, at least 1 native unit is charged per pubdata byte, preventing the silent zeroing.

---

### Proof of Concept

**Trigger condition:** operator sets `native_price = 100`, `pubdata_price = 50`.

**Calculation:**
```
native_per_pubdata = 50.wrapping_div(100) = 0   (floor division)
```

**Effect in `get_resources_to_charge_for_pubdata`:**
```
native = current_pubdata_spent * 0 = 0
```

A transaction that writes to 20 unique storage slots (generating ~640 bytes of pubdata) pays:
- With correct `div_ceil`: `ceil(50/100) = 1` native unit × 640 bytes = 640 native units charged
- With current `wrapping_div`: `0` native units charged — pubdata is completely free

The attacker submits a storage-spam transaction (e.g., the pattern in `tests/instances/transactions/src/lib.rs` `test_pubdata_native_calculation_overflow`) under a block context where `pubdata_price < native_price`. The transaction succeeds, generates maximum pubdata, and the `native_used` counter for pubdata remains zero, so no `delta_gas` adjustment compensates the operator either.
