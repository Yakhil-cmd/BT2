Audit Report

## Title
Byzantine node submits signature-valid, curve-invalid `VetKdEncryptedKeyShare` causing persistent `combine_encrypted_key_shares` failure — (`rs/crypto/src/vetkd/mod.rs`)

## Summary
`verify_encrypted_key_share_internal` validates only the Ed25519 node signature over the raw share bytes; it never deserializes or validates the embedded BLS12-381 curve points. `combine_encrypted_key_shares_internal` deserializes those points and aborts with `InvalidArgumentEncryptedKeyShare` if any share contains an invalid encoding. A single Byzantine subnet node can exploit this asymmetry to permanently prevent VetKD key derivation for every canister request whose combination round includes its share.

## Finding Description
`verify_encrypted_key_share_internal` (lines 272–302 of `rs/crypto/src/vetkd/mod.rs`) constructs a `BasicSigOf` from `key_share.node_signature` and calls `BasicSigVerifierInternal::verify_basic_sig` over `key_share.encrypted_key_share`. The content of `encrypted_key_share` is treated as an opaque byte blob; `EncryptedKeyShare::deserialize` is never called.

`combine_encrypted_key_shares_internal` (lines 347–366) iterates over all shares and calls `EncryptedKeyShare::deserialize` on each one. `EncryptedKeyShare::deserialize` (lines 446–465 of `rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs`) checks that the bytes decode to three valid BLS12-381 points (`c1: G1Affine`, `c2: G2Affine`, `c3: G1Affine`). Any failure returns `Err(EncryptedKeyShareDeserializationError::InvalidEncryptedKeyShare)`, which is mapped to `VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare` and propagated via `?` at line 366, before `EncryptedKey::combine_all` is ever invoked.

The `combine_valid_shares` fallback (lines 388–430) is only entered when `combine_all` returns `EncryptedKeyCombinationError::InvalidShares`. Because the deserialization failure exits the function at line 366 with a different error variant, the fallback is structurally unreachable in this path.

`EncryptedKeyShare::BYTES` is `2 * G1Affine::BYTES + G2Affine::BYTES` = 192 bytes (line 401 of `rs/crypto/internal/crypto_lib/bls12_381/vetkd/src/lib.rs`). A Byzantine node constructs a 192-byte payload whose first 48 bytes do not encode a valid compressed G1 point (e.g., all-zero bytes), signs it with their legitimate node key, and submits it. The share passes `verify_encrypted_key_share` (signature is valid) and is admitted to the validated pool. Every subsequent call to `combine_encrypted_key_shares` that includes this share fails immediately at deserialization.

In `rs/consensus/chain_key/src/lib.rs` (lines 327–344), a `combine_shares` failure with `InvalidArgumentEncryptedKeyShare` falls into the generic `Err(err)` arm, logs a warning, and returns `None`. The request is skipped for that round. Because all validated shares for a callback ID are passed to `combine_encrypted_key_shares` without any per-share exclusion mechanism, the Byzantine share is included in every subsequent attempt until the request expires.

## Impact Explanation
Any VetKD key derivation request assigned to a combination round that includes the Byzantine node's share will never produce a successful response. The request is silently skipped each round and eventually expires with `TimedOut`. The attacker can repeat the attack for every new VetKD request, rendering VetKD key derivation permanently unavailable on the affected subnet for as long as the Byzantine node participates. This matches the allowed impact: **High — Application/platform-level DoS, subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The attacker is a legitimate subnet node operating below the Byzantine fault threshold. No key leakage, majority corruption, or privileged access is required. The crafted payload is trivial: any 192-byte string with an invalid BLS12-381 point encoding suffices. The node signs it with their own valid key. The attack is repeatable for every new VetKD request and requires no victim interaction.

## Recommendation
Add `EncryptedKeyShare::deserialize` validation inside `verify_encrypted_key_share_internal`, immediately after the signature check succeeds. If deserialization fails, return a new `VetKdKeyShareVerificationError` variant (e.g., `InvalidEncryptedKeyShareContent`). This restores the invariant that any share admitted to the validated pool can always be deserialized during combination, closing the gap without changing the combination logic.

## Proof of Concept
```rust
// 1. Craft a 192-byte payload with an invalid G1 point (all zeros, no valid tag)
let bad_bytes = vec![0u8; EncryptedKeyShare::BYTES]; // 192 bytes

// 2. Byzantine node signs it with their legitimate node key
let signature = node_key.sign(&bad_bytes);
let share = VetKdEncryptedKeyShare {
    encrypted_key_share: VetKdEncryptedKeyShareContent(bad_bytes),
    node_signature: signature,
};

// 3. Passes verify_encrypted_key_share — only Ed25519 signature is checked
assert_eq!(crypto.verify_encrypted_key_share(node_id, &share, &args), Ok(()));

// 4. Fails combine_encrypted_key_shares — EncryptedKeyShare::deserialize rejects invalid points
let mut shares = BTreeMap::new();
shares.insert(node_id, share);
assert!(matches!(
    crypto.combine_encrypted_key_shares(&shares, &args),
    Err(VetKdKeyShareCombinationError::InvalidArgumentEncryptedKeyShare)
));
// All subsequent combination attempts for this callback_id also fail.
// The request expires after one DKG interval with TimedOut.
```
