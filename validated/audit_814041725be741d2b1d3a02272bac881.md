### Title
Global Contract Identifier Bytes Inflate `account_storage_usage` in `action_delete_account`, Incorrectly Blocking Account Deletion - (File: `runtime/runtime/src/actions.rs`)

### Summary

When an account adopts a global contract via `UseGlobalContractAction`, `use_global_contract` adds `contract_identifier.len()` bytes to `account.storage_usage()`. In `action_delete_account` (protocol versions < `FixDeleteAccountGlobalContractStorageUsage`, i.e., < 85), the legacy code path computes the non-contract storage by subtracting only `code_len` — which is **zero** for global contracts because `local_contract_hash()` returns `None`. The identifier bytes are never subtracted, so `account_storage_usage` is overstated by the identifier size, and the `> MAX_ACCOUNT_DELETION_STORAGE_USAGE` guard incorrectly blocks deletion for accounts whose actual non-contract storage is within the limit.

### Finding Description

**Step 1 — Identifier bytes are added to `storage_usage` in `use_global_contract`:** [1](#0-0) 

`contract_identifier.len()` is 32 bytes for `CodeHash` or `account_id.len()` bytes for `AccountId`. [2](#0-1) 

**Step 2 — `action_delete_account` legacy path only subtracts local contract code:** [3](#0-2) 

For an account with `AccountContract::Global` or `AccountContract::GlobalByAccount`, `account_ref.local_contract_hash()` returns `None`, so `code_len = 0`. The identifier bytes added in Step 1 are **not subtracted**. The resulting `account_storage_usage` is overstated by `identifier_len` bytes.

**Step 3 — Overstated value drives the deletion guard:** [4](#0-3) 

An account whose actual non-contract storage is exactly `MAX_ACCOUNT_DELETION_STORAGE_USAGE` but whose `storage_usage` field is `MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_len` (because the identifier was added in Step 1) is incorrectly rejected with `DeleteAccountWithLargeState`.

**The fix (protocol version 85) confirms the bug:** [5](#0-4) [6](#0-5) 

The test `test_delete_account_global_contract_protocol_transition` explicitly demonstrates the before/after: [7](#0-6) 

### Impact Explanation

An unprivileged user who has called `UseGlobalContractAction` on their account and whose `storage_usage` falls in the range `(MAX_ACCOUNT_DELETION_STORAGE_USAGE, MAX_ACCOUNT_DELETION_STORAGE_USAGE + identifier_len]` cannot delete their account. The `DeleteAccountAction` is permanently blocked with `DeleteAccountWithLargeState`, preventing the user from recovering their balance via the beneficiary transfer. The account persists indefinitely in a state where deletion is impossible despite the actual non-contract storage being within the allowed limit. This is a broken accounting invariant with High impact on account lifecycle.

### Likelihood Explanation

Any user can trigger this condition by:
1. Deploying a global contract with `GlobalContractDeployMode::AccountId` (choosing a long deployer account ID to maximize identifier size).
2. Calling `UseGlobalContractAction` on their own account.
3. Accumulating enough contract storage data to bring `storage_usage` into the affected range.

The condition is fully deterministic and user-controllable. No privileged role is required. The affected range is up to 64 bytes wide (for a maximum-length account ID identifier), which is reachable through normal contract storage writes.

### Recommendation

In `action_delete_account`, always use `get_contract_storage_usage` (which correctly handles all `AccountContract` variants including `Global` and `GlobalByAccount`) to compute the amount to subtract from `storage_usage` before comparing against `MAX_ACCOUNT_DELETION_STORAGE_USAGE`. This is exactly what `ProtocolFeature::FixDeleteAccountGlobalContractStorageUsage` implements at protocol version 85. [8](#0-7) 

### Proof of Concept

1. Deploy a global contract under `GlobalContractDeployMode::AccountId` from an account with a long name (e.g., 32-character account ID).
2. Call `UseGlobalContractAction` on a target account — this adds `account_id.len()` bytes to `storage_usage`.
3. Write contract storage data to the target account until `storage_usage == MAX_ACCOUNT_DELETION_STORAGE_USAGE + account_id.len()` (i.e., `10_000 + identifier_len`).
4. Submit `DeleteAccountAction` targeting the account. On protocol versions < 85, the runtime computes `account_storage_usage = storage_usage - 0 = 10_000 + identifier_len > 10_000 = MAX`, and returns `DeleteAccountWithLargeState`, permanently blocking deletion despite the actual non-contract storage being exactly at the limit.

fn get_contract_storage_usage(
    state_update: &TrieUpdate,
    account_id: &AccountId,
    account: &Account,
) -> Result<StorageUsage, StorageError> {
    Ok(match account.contract().as_ref() {
        AccountContract::None => 0,
        AccountContract::Local(code_hash) => {
            get_code_len_or_default(state_update, account_id.clone(), *code_hash)?
        }
        AccountContract::Global(_) | AccountContract::GlobalByAccount(_) => {
            account.contract().identifier_storage_usage()
        }
    })
}
