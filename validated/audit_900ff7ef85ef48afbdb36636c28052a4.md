Looking at the external bug's class: a state flag is reset without verifying whether the entity being processed was actually the one that set the flag. I need to find the same pattern in the Metric OMM pause-transition logic.

The key functions are `protocolPausePool` and `protocolUnpausePool` in `MetricOmmPoolFactory.sol`.

`protocolPausePool` accepts two entry states — level 0 (active) **or** level 1 (admin-paused) — and always writes level 2: [1](#0-0) 

`protocolUnpausePool` always writes level 1 (admin-paused), regardless of what the pool's level was before the protocol pause: [2](#0-1) 

The pool's `swap` function is gated by `whenNotPaused`, which reverts on any non-zero `pauseLevel`: [3](#0-2) [4](#0-3) 

The admin-only `unpausePool` is the only path back to level 0: [5](#0-4)

### Title
`protocolUnpausePool` Unconditionally Forces Pool to Admin-Paused State (Level 1) Regardless of Pre-Pause Level, Permanently Blocking Swaps Until Admin Acts — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`protocolUnpausePool` always writes `pauseLevel = 1` (admin-paused) when clearing a protocol pause, even when the pool was at level 0 (fully active) before the protocol pause was applied. Because `protocolPausePool` accepts both level-0 and level-1 pools as valid entry states, the pre-pause level is never recorded. After the protocol unpause the pool is left in admin-paused state that the admin never set, blocking all swaps until the admin separately calls `unpausePool`.

### Finding Description
`protocolPausePool` transitions level 0 → 2 **or** level 1 → 2:

```solidity
// MetricOmmPoolFactory.sol L392-396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

`protocolUnpausePool` always writes level 1, never level 0:

```solidity
// MetricOmmPoolFactory.sol L399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← always 1, never 0
}
```

The factory stores no record of the pool's level before the protocol pause. When the pool was at level 0 before the pause, the correct post-unpause level is 0, but the function writes 1. The pool is now in an admin-paused state that the admin never requested.

`swap` checks `pauseLevel != 0` and reverts:

```solidity
// MetricOmmPool.sol L643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

The only exit from level 1 is `unpausePool`, callable only by the pool admin:

```solidity
// MetricOmmPoolFactory.sol L467-471
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the admin is unresponsive, rotated out, or simply unaware that an extra step is required, the pool remains swap-frozen indefinitely.

### Impact Explanation
Every pool that was at level 0 when the protocol pause was applied will be left at level 1 after `protocolUnpausePool`. Swaps are blocked for all users of that pool. `removeLiquidity` and `addLiquidity` are not gated by `whenNotPaused`, so LP principal is not directly locked, but the core trading function is completely unusable until the admin acts. If the admin is unavailable or unaware, the pool is permanently frozen for swaps — a broken core pool functionality matching the allowed impact gate.

### Likelihood Explanation
The most common protocol-pause scenario is pausing an active (level-0) pool. `protocolPausePool` explicitly supports 0 → 2. Every such pause followed by a `protocolUnpausePool` call triggers the bug. No special attacker action is required; the owner's own normal operational flow produces the incorrect state.

### Recommendation
Record the pool's pause level before the protocol pause and restore it on unpause. One minimal fix: add a `mapping(address => uint8) prePausePauseLevel` in the factory, write it in `protocolPausePool`, and read it in `protocolUnpausePool`:

```solidity
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    prePausePauseLevel[pool] = cur;          // store pre-pause level
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    uint8 restore = prePausePauseLevel[pool]; // 0 or 1
    delete prePausePauseLevel[pool];
    IMetricOmmPoolFactoryActions(pool).setPause(restore);
}
```

### Proof of Concept
1. Pool is deployed and active — `pauseLevel = 0`.
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` → `pauseLevel = 1` (admin-paused, even though admin never paused it).
4. Any user calls `swap(...)` → reverts with `PoolPaused()`.
5. Pool admin must call `unpausePool(pool)` to reach level 0. If admin is unavailable, swaps are frozen indefinitely.
