### Title
Echo-Broadcast Threshold Collapse Enables DKG Commitment Equivocation with n ≤ 3 Participants - (File: src/protocol/echo_broadcast.rs)

### Summary

The `echo_ready_thresholds` function in `src/protocol/echo_broadcast.rs` returns `(0, 0)` for any participant count `n ≤ 3`. Because the DKG protocol (`do_keyshare`) uses `do_broadcast` to distribute polynomial commitments and proofs of knowledge, a single malicious participant in a 3-party DKG can equivocate — sending different commitments to different honest participants — and each honest participant will independently deliver a different value. The result is that honest parties complete DKG holding inconsistent public keys, permanently corrupting the key generation output.

### Finding Description

**Root cause — `echo_ready_thresholds` collapses to zero for n ≤ 3:** [1](#0-0) 

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // ← both echo_t and ready_t become 0
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

With `echo_t = 0` and `ready_t = 0`, the comparisons in `reliable_broadcast_receive_all` become:

- `data_echo.get(&data) > 0` — true after **one** echo vote (the participant's own simulated echo)
- `data_ready.get(&data) > 0` — true after **one** ready vote (the participant's own simulated ready)
- `data_ready.get(&data) > 2 * 0` — also true after **one** ready vote → immediate delivery [2](#0-1) [3](#0-2) 

This means each honest participant delivers whatever value it first received, with no cross-checking against what other honest participants received. The consistency guarantee of reliable broadcast is entirely absent.

**How DKG uses this broadcast:**

`do_keyshare` calls `do_broadcast` to distribute each participant's polynomial commitment and proof of knowledge: [4](#0-3) 

```rust
let commitments_and_proofs_map = do_broadcast(
    &mut chan,
    &participants,
    me,
    (commitment, proof_of_knowledge),
).await?;
```

**Exploit path with n = 3 (A, B honest; M malicious):**

1. M sends `Send(commitment_X, proof_X)` to A and `Send(commitment_Y, proof_Y)` to B in the broadcast round.
2. A's simulated echo of `commitment_X` immediately satisfies `count > 0 > echo_t`, so A sends `Ready(commitment_X)` and sets `finish_echo = true`. Any later `Echo(commitment_Y)` from B is skipped (`finish_echo` guard).
3. A's simulated ready of `commitment_X` immediately satisfies `count > 0 > 2*ready_t`, so A **delivers** `commitment_X` for M's session.
4. B independently delivers `commitment_Y` for M's session by the same path.
5. M also sends different commitment hashes to A and B in the earlier `send_many` round (Round 2 of DKG), so `verify_commitment_hash` passes independently for each honest party. [5](#0-4) 

6. A computes `public_key_A = f(commitment_A, commitment_X, commitment_B)`.
7. B computes `public_key_B = f(commitment_A, commitment_Y, commitment_B)`.
8. `public_key_A ≠ public_key_B`.

**The final `broadcast_success` check does not catch this:** [6](#0-5) 

`broadcast_success` only verifies that all participants broadcast `(true, session_id)`. It does not compare public keys across participants. Both A and B pass this check and return inconsistent `KeygenOutput` values.

**No library-level guard prevents n = 3:** [7](#0-6) 

`assert_key_invariants` permits any `n ≥ 2` without warning that n ≤ 3 removes the broadcast consistency guarantee.

### Impact Explanation

Honest parties complete DKG and accept **different public keys** for the same key generation session. Any subsequent signing attempt will fail or produce signatures that only verify under one party's view of the public key. The shared key is permanently corrupted and unusable. This matches:

> **High: Corruption of DKG outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs.**

### Likelihood Explanation

A 3-party threshold setup (e.g., 2-of-3) is one of the most common real-world configurations. The attacker needs only to be one of the three participants — no privileged access, no leaked keys. The attack requires sending two different `Send` messages to two different recipients in the broadcast round, which is a standard network-layer capability for any participant.

### Recommendation

1. **Remove the n ≤ 3 special case** or replace it with a correct threshold for small n. For n = 2 (no Byzantine tolerance possible), the protocol can use a simple authenticated channel. For n = 3, the standard echo-broadcast requires `f = 0` (no malicious parties), which should be enforced at the API level, not silently assumed.
2. **Add a guard in `assert_key_invariants`** that rejects n ≤ 3 or documents the security limitation explicitly.
3. **Alternatively**, use a different broadcast primitive for small n that provides consistency without relying on the `n > 3f` assumption.

### Proof of Concept

```
Participants: A (honest), B (honest), M (malicious)
n = 3, threshold = 2

Round 2 (commitment hash, plain send_many):
  M → A: hash(commitment_X)
  M → B: hash(commitment_Y)

Round 3 (commitment broadcast via do_broadcast):
  M → A: Send(commitment_X, proof_X)
  M → B: Send(commitment_Y, proof_Y)

A processes Send(commitment_X):
  echo_t = 0 → simulated Echo(commitment_X) satisfies count(X)=1 > 0
  → A sends Ready(commitment_X), finish_echo = true
  → simulated Ready(commitment_X) satisfies count(X)=1 > 0
  → A delivers commitment_X for M's session ✓

B processes Send(commitment_Y):
  → B delivers commitment_Y for M's session ✓

Round 4 verification:
  A: verify_commitment_hash(commitment_X, hash(commitment_X)) → OK
  B: verify_commitment_hash(commitment_Y, hash(commitment_Y)) → OK

DKG output:
  A: KeygenOutput { public_key: PK_A = g^(a_A + x + a_B) }
  B: KeygenOutput { public_key: PK_B = g^(a_A + y + a_B) }
  PK_A ≠ PK_B  →  DKG output is permanently inconsistent
```

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}

    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
