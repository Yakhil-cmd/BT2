Audit Report

## Title
Off-by-One Boundary Check in `disburse_neuron` Allows Recovery of Tokens That Should Be Burned as Ledger Fee - (File: rs/nns/governance/src/governance.rs)

## Summary
In `disburse_neuron`, the condition `if disburse_amount_e8s > transaction_fee_e8s` at line 2025 uses strict `>` instead of `>=`. When `disburse_amount_e8s` equals exactly `transaction_fee_e8s`, the fee is not subtracted and `transfer_funds` is called with `disburse_amount_e8s = transaction_fee_e8s`, causing the user to receive `transaction_fee_e8s` tokens that the governance accounting model says should net to zero. The identical bug exists in the SNS governance implementation at line 1170.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `disburse_neuron` computes `disburse_amount_e8s` from `neuron_minted_stake_e8s` (line 2016–2021), then attempts to subtract the ledger fee:

```rust
// line 2025
if disburse_amount_e8s > transaction_fee_e8s {
    disburse_amount_e8s -= transaction_fee_e8s
}
```

When `disburse_amount_e8s == transaction_fee_e8s`, the condition is false, so `disburse_amount_e8s` is unchanged. The code then unconditionally calls:

```rust
// line 2091–2100
self.ledger.transfer_funds(
    disburse_amount_e8s,      // = transaction_fee_e8s
    transaction_fee_e8s,
    Some(neuron_subaccount),
    to_account,
    now,
).await?;
```

This debits `2 * transaction_fee_e8s` from the neuron subaccount: `transaction_fee_e8s` goes to the user and `transaction_fee_e8s` is burned as the ledger fee. The governance model requires the user to receive zero when their minted stake equals the fee, but instead they receive `transaction_fee_e8s`.

A parallel fee-burn guard at line 2046 (`if fees_amount_e8s > transaction_fee_e8s`) also uses strict `>`, so when `fees_amount_e8s == transaction_fee_e8s`, the neuron management fees are not burned either, leaving the neuron subaccount with sufficient balance (`2 * transaction_fee_e8s`) for the transfer to succeed.

The identical pattern exists in `rs/sns/governance/src/governance.rs` at lines 1170–1172 and 1181.

## Impact Explanation
A neuron controller can recover `transaction_fee_e8s` tokens (10,000 e8s = 0.0001 ICP for NNS) per exploit that the governance accounting model requires to be consumed entirely by the ledger fee. The total token supply is conserved (the neuron account is debited `2 * transaction_fee_e8s`, the user receives `transaction_fee_e8s`, and the ledger burns `transaction_fee_e8s`), so this is not illegal minting. The impact is a governance accounting violation: the user recovers tokens that should have been burned, bypassing the intended fee-consumption mechanism in both NNS and SNS governance. This matches a Medium-severity impact — a governance/ledger accounting error with concrete but constrained per-exploit financial harm (0.0001 ICP per execution), requiring specific neuron conditions to trigger.

## Likelihood Explanation
Any neuron controller can reach the triggering condition without special privileges:
1. Stake exactly `2 * transaction_fee_e8s` (20,000 e8s for NNS) into a neuron.
2. Submit a proposal that is rejected, accumulating `neuron_fees_e8s = transaction_fee_e8s` (10,000 e8s).
3. Dissolve and disburse the neuron.

The conditions are fully attacker-controlled through normal governance participation. The exploit is repeatable but each iteration requires a new neuron and a rejected proposal, limiting throughput. The per-exploit gain (0.0001 ICP) is small, but the path is deterministic and requires no privileged access.

## Recommendation
Change the strict `>` to `>=` in both files so that when `disburse_amount_e8s` equals `transaction_fee_e8s`, the result is zero and no net transfer is issued to the user:

**`rs/nns/governance/src/governance.rs` line 2025:**
```rust
// Before
if disburse_amount_e8s > transaction_fee_e8s {
// After
if disburse_amount_e8s >= transaction_fee_e8s {
```

**`rs/sns/governance/src/governance.rs` line 1170:**
```rust
// Before
if disburse_amount_e8s > transaction_fee_e8s {
// After
if disburse_amount_e8s >= transaction_fee_e8s {
```

Additionally, consider adding a guard to skip `transfer_funds` entirely when `disburse_amount_e8s == 0` after the subtraction, to avoid a zero-value transfer call to the ledger.

## Proof of Concept
**NNS Governance — deterministic unit test path:**

1. Initialize governance with `transaction_fee_e8s = 10_000`.
2. Create a dissolved, KYC-verified neuron with `cached_neuron_stake_e8s = 20_000` and `neuron_fees_e8s = 10_000`.
3. Call `disburse_neuron` with no explicit amount.
4. `neuron_minted_stake_e8s = 20_000 − 10_000 = 10_000`; `disburse_amount_e8s = 10_000`.
5. Fee-burn guard: `10_000 > 10_000` → false → fees not burned; neuron subaccount retains 20,000 e8s.
6. Disburse guard: `10_000 > 10_000` → false → `disburse_amount_e8s` stays 10,000.
7. `transfer_funds(10_000, 10_000, neuron_subaccount, to_account, now)` succeeds (20,000 e8s available).
8. Assert: user account receives 10,000 e8s. Expected correct result: 0 e8s.

This can be implemented as a unit test in `rs/nns/governance/tests/` using the existing mock ledger infrastructure, asserting that the recipient balance equals zero after disbursement when `neuron_minted_stake_e8s == transaction_fee_e8s`.
