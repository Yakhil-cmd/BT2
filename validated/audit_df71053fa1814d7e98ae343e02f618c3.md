### Title
Unverified `new_proof_hash` in WalletConnect `chia_spendVC` Allows Dapp to Substitute Arbitrary Credential Root - (File: packages/gui/src/electron/commands/Commands.ts)

### Summary
A WalletConnect-connected dapp with `chia_addVCProofs` and `chia_spendVC` permissions can call `chia_spendVC` supplying a `new_proof_hash` that does not correspond to the proofs the user just approved via `chia_addVCProofs`. The wallet displays the raw hash to the user for confirmation but never verifies that it is the hash of any proof set the user actually approved. The user approves the VC spend believing the hash corresponds to the proofs they just added; the dapp substitutes the hash of a different, previously-stored proof set. The on-chain VC is updated to reference credentials the user did not intend to authorize.

### Finding Description

The `chia_wallet.vc_spend` WalletConnect command accepts a required `new_proof_hash` parameter alongside `vc_id` and `provider_inner_puzhash`: [1](#0-0) 

The `chia_wallet.vc_add_proofs` command stores proof key-value pairs in the wallet's local proof store: [2](#0-1) 

Both commands are exposed as dapp-callable WalletConnect methods (`chia_spendVC`, `chia_addVCProofs`). The entire dispatch path — from `DISPATCH_AS_PAIR` in `main.tsx` through `dispatchPairRequest` — only validates pair membership, command grants, network, and fingerprint. It performs no cross-parameter consistency check: [3](#0-2) [4](#0-3) 

The wallet daemon API layer (`VC.ts`) simply forwards the `new_proof_hash` to the daemon without any local validation: [5](#0-4) 

There is no code anywhere in the GUI that verifies `new_proof_hash` is the hash of any proof set stored via `vc_add_proofs`.

### Impact Explanation

**Attack sequence:**

1. Dapp (granted `chia_addVCProofs` + `chia_spendVC`) calls `chia_addVCProofs` with `{kyc: true}` — user approves, proofs stored locally with hash `H_kyc`.
2. In a prior session, the dapp also stored `{kyc: true, admin: true}` proofs with hash `H_admin` via `chia_addVCProofs` — user approved that too.
3. Dapp calls `chia_spendVC` with `new_proof_hash = H_admin`. The confirmation dialog shows the user the raw hash `H_admin` and the VC launcher ID. The user, expecting the hash to correspond to the `{kyc: true}` proofs they just added, approves.
4. The on-chain VC is updated to reference `H_admin` — the `{kyc: true, admin: true}` credential set — not the `{kyc: true}` set the user intended.

The user's VC now asserts credentials they did not intend to activate. Because Chia's CR-CAT access control is gated on VC proof hashes, this corrupts the user's credential identity state. Alternatively, the dapp can supply a `new_proof_hash` with no corresponding local proofs, permanently corrupting the VC (its proof set becomes unresolvable).

This matches the allowed High impact: *"Corruption… of WalletConnect state that causes a user to approve… the wrong… identity… or status."*

### Likelihood Explanation

The attacker must be a WalletConnect dapp that the user has already paired and granted both `chia_addVCProofs` and `chia_spendVC` permissions. This is a realistic scenario for any VC-issuing dapp. No leaked keys, host compromise, or cryptographic break is required. The dapp simply substitutes a different hash value in the `chia_spendVC` call. The user has no UI mechanism to verify which stored proof set a given hash corresponds to.

### Recommendation

Before executing `vc_spend`, the GUI should resolve `new_proof_hash` against the wallet's local proof store (via `vc_get_proofs_for_root`) and display the corresponding human-readable proof key-value pairs in the confirmation dialog alongside the hash. If the hash resolves to no stored proofs, the confirmation dialog should warn the user explicitly. This mirrors the fix pattern from the original report: verify the derived value against the raw data before signing/spending.

### Proof of Concept

```
// Attacker dapp (has chia_addVCProofs + chia_spendVC grants)

// Step 1: Store elevated proofs in wallet proof store (user approves)
await walletConnect.request({
  method: 'chia_addVCProofs',
  params: { fingerprint, proofs: { kyc: 'true', admin: 'true' } }
});
// H_admin = sha256(canonical({ kyc: 'true', admin: 'true' }))

// Step 2: Store legitimate proofs (user approves, thinking this is the active set)
await walletConnect.request({
  method: 'chia_addVCProofs',
  params: { fingerprint, proofs: { kyc: 'true' } }
});

// Step 3: Spend VC with H_admin instead of H_kyc
// User sees raw hash in confirmation dialog, approves
await walletConnect.request({
  method: 'chia_spendVC',
  params: {
    fingerprint,
    vc_id: '<user_vc_launcher_id>',
    new_proof_hash: '<H_admin>',   // ← substituted hash
    provider_inner_puzhash: '<known_provider_puzhash>',
  }
});
// On-chain VC now references {kyc: true, admin: true} — not {kyc: true}

export async function dispatchPairRequest<T>(
  topic: string,
  command: string,
  params: Record<string, unknown>,
  process: (context: DispatchPairRequestContext) => Promise<T>,
  confirm: () => Promise<boolean>,
): Promise<T> {
  const [loggedInFingerprint, isMainnetValue] = await Promise.all([getLoggedInFingerprint(), isMainnet()]);

  const pair = findPair(topic);
  if (!pair) {
    throw new WcError(`Pair not found`, WcErrorCode.USER_REJECTED);
  }

  // verify if the command is allowed for this pair
  if (!pair.commands.includes(command)) {
    throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const { fingerprint } = params;

  // verify if the network is the same as the pair's network
  if (isMainnetValue !== pair.mainnet) {
    throw new WcError(`Network mismatch`, WcErrorCode.UNSUPPORTED_CHAINS);
  }

  // verify if the requested fingerprint is allowed for this pair
  const requestedFingerprint = fingerprint ?? loggedInFingerprint;
  if (typeof requestedFingerprint !== 'number' || !requestedFingerprint || requestedFingerprint !== pair.fingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const context = {
    pair,
    fingerprint: requestedFingerprint,
  };

  // Dapps may not switch the active key for an existing pair.
  if (fingerprint !== undefined && fingerprint !== loggedInFingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  // if command is bypassed return true
  if (pair.bypass.includes(command)) {
    return process(context);
  }

  const isAllowed = await confirm();
  if (isAllowed === true) {
    return process(context);
  }

  throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
