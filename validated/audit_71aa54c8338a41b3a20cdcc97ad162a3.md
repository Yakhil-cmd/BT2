### Title
COA `supportsInterface` Omits ERC-1271, Breaking Smart-Contract-Wallet Authentication in Flow EVM - (File: fvm/evm/handler/coa/coa.sol)

---

### Summary

The Cadence-Owned Account (COA) contract deployed for every COA in Flow EVM implements `isValidSignature` (ERC-1271) but does **not** declare support for it in `supportsInterface` (ERC-165). Any EVM contract that follows the standard ERC-165 guard before calling `isValidSignature` will conclude that a COA does not support ERC-1271 and will either reject the COA as a valid signer or fall back to an insecure path, silently bypassing the COA-backed authentication that the Flow protocol intends to provide.

---

### Finding Description

The COA Solidity contract defines the ERC-1271 magic value as a named constant and implements `isValidSignature`: [1](#0-0) [2](#0-1) 

`isValidSignature` delegates to the Cadence Arch precompile `verifyCOAOwnershipProof`, which performs full Cadence-side signature verification. This is the only authentication mechanism available to a COA, because COAs have no private key. [3](#0-2) 

However, the `supportsInterface` implementation covers only ERC-1155, ERC-721, ERC-777, and ERC-165 itself. The ERC-1271 interface ID (`0x1626ba7e`) is **absent**: [4](#0-3) 

The contract's own documentation (in `contract.cdc`) explicitly lists ERC-1271 as a supported standard: [5](#0-4) 

The constant `ValidERC1271Signature = 0x1626ba7e` is defined but used only as a return value inside `isValidSignature`, never as a comparison target inside `supportsInterface`. This is the exact same structural omission described in the external report for OpenBrush: the contract implements a capability but provides no standard way for callers to discover it.

---

### Impact Explanation

ERC-1271 is the universal smart-contract-wallet signing standard. Virtually every EVM protocol that accepts smart-contract signers (Safe multisig, OpenSea Seaport, Uniswap permit2, ERC-4337 bundlers, etc.) performs the ERC-165 guard `supportsInterface(0x1626ba7e)` before calling `isValidSignature`. When that guard returns `false` for a COA:

1. **Authentication bypass via fallback**: Contracts that treat a `false` ERC-165 response as "not a smart-contract wallet, treat as EOA" will attempt ECDSA recovery against the COA address. Because COAs have no private key, ECDSA recovery always fails. Depending on the contract's logic, this can mean the COA is silently treated as an unauthorized signer, or the contract's access-control check is skipped entirely when no valid signer path exists.

2. **Cross-VM asset loss**: A COA owner who deposits ERC-721/ERC-1155 tokens into a protocol that uses ERC-1271 for withdrawal authorization cannot withdraw them, because the protocol's `supportsInterface` check causes it to reject the COA's signature. Assets become permanently locked in the EVM contract.

3. **Broken upgradeability guarantee**: The COA contract is deployed by the Flow protocol itself and cannot be upgraded by users. Any EVM contract that caches the `supportsInterface` result will permanently record that COAs do not support ERC-1271, even if the COA bytecode is later corrected.

---

### Likelihood Explanation

The entry path requires only an unprivileged EVM transaction sender interacting with any standard EVM protocol that follows ERC-165 before ERC-1271. This is the dominant pattern in the EVM ecosystem. The COA contract is deployed automatically for every user who calls `EVM.createCadenceOwnedAccount()`, so every COA on Flow EVM is affected. No special privileges, leaked keys, or node compromise are required.

---

### Recommendation

Add the ERC-1271 interface ID to `supportsInterface` in `fvm/evm/handler/coa/coa.sol`:

```solidity
function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId  ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId              ||
        id == ValidERC1271Signature;                  // ERC-1271: 0x1626ba7e
}
```

Also add a corresponding interface declaration so the compiler can enforce the conformance:

```solidity
interface IERC1271 {
    function isValidSignature(bytes32 hash, bytes memory signature) external view returns (bytes4);
}
contract COA is ERC1155TokenReceiver, ERC777TokensRecipient, ERC721TokenReceiver, IERC165, IERC1271 { ... }
```

### Proof of Concept

1. Deploy any EVM contract on Flow EVM that guards ERC-1271 calls with ERC-165, for example:

```solidity
contract Verifier {
    function verify(address signer, bytes32 hash, bytes calldata sig) external view returns (bool) {
        // Standard ERC-165 guard used by Safe, Seaport, permit2, etc.
        (bool ok, bytes memory data) = signer.staticcall(
            abi.encodeWithSelector(IERC165.supportsInterface.selector, bytes4(0x1626ba7e))
        );
        if (!ok || !abi.decode(data, (bool))) {
            // COA falls here: supportsInterface(0x1626ba7e) returns false
            // Fallback: treat as EOA, attempt ecrecover — always fails for COA
            return false; // or revert, or skip auth entirely depending on implementation
        }
        (bool ok2, bytes memory data2) = signer.staticcall(
            abi.encodeWithSelector(IERC1271.isValidSignature.selector, hash, sig)
        );
        return ok2 && abi.decode(data2, (bytes4)) == 0x1626ba7e;
    }
}
```

2. Call `Verifier.verify(coaAddress, hash, validProof)` from any EOA. The call returns `false` even with a fully valid COA ownership proof, because `supportsInterface(0x1626ba7e)` on the COA returns `false`, causing the guard to short-circuit before `isValidSignature` is ever called.

3. The root cause is confirmed at: [4](#0-3) 

The ERC-1271 interface ID `0x1626ba7e` (stored as `ValidERC1271Signature` at line 62) is never compared inside `supportsInterface`, making the COA invisible to any ERC-165-compliant caller that queries for ERC-1271 support.
