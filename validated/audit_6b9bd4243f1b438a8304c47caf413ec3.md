### Title
Wrong Multi-NFT Royalty Calculation in WalletConnect Command Display Causes User to Approve Understated Spend Amount - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the total payment amount by the number of NFTs before computing each NFT's royalty contribution. This causes the `amountWithRoyalties` field shown in the WalletConnect command-approval dialog to understate the true total spend by a factor of N (number of NFTs), leading a user to approve a `take_offer` or `create_offer_for_ids` WalletConnect command believing they will spend less than the wallet daemon will actually deduct.

### Finding Description

`formatAmountWithRoyalties` is called during WalletConnect command display to compute the total spend including creator royalties:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The Chia protocol computes each NFT's royalty on the **full** fungible payment amount, not a per-NFT split. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The code instead computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

which is `1/N` of the correct value. The test fixture for the multi-NFT case confirms the bug is baked in — it asserts `amountWithRoyalties: '0.00010255'` for two NFTs with 5% and 0.1% royalties on 0.0001 XCH, which matches the buggy formula (correct value would be `0.0001051`). [2](#0-1) 

`formatAmountWithRoyalties` is called by `withRoyaltyTotals`, which is called by `walletDeltaToDisplay`, which is called by `parseCommandDisplay` — the function that generates the approval payload shown to the user for every WalletConnect wallet command. [3](#0-2) 

### Impact Explanation

When a WalletConnect-connected dApp sends a `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command containing multiple NFTs with royalties, the approval dialog shows `amountWithRoyalties` that is `1/N` of the correct total. The user reads the understated figure, approves, and the Chia wallet daemon — which uses the correct `nft_calculate_royalties` RPC — deducts the full correct amount from the user's spendable balance. The user has approved a spend they would not have approved had the correct figure been shown.

This matches the **High** impact class: WalletConnect state causes a user to approve the wrong amount.

### Likelihood Explanation

Any WalletConnect dApp (including a legitimate one) that presents a multi-NFT offer triggers the bug. An attacker who can craft or relay an offer containing two or more NFTs with non-trivial royalties can reliably cause the displayed total to be materially lower than the actual spend. No key material, host access, or cryptographic break is required — only a valid multi-NFT offer file and an active WalletConnect session, both of which are normal user operations.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be applied to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test assertion to reflect the corrected value (`0.0001051` for the two-NFT case).

### Proof of Concept

Given a WalletConnect `take_offer` command for an offer where:
- Payment: 100,000,000 mojos (0.0001 XCH)
- NFT A: `royalty_percentage = 500` (5%)
- NFT B: `royalty_percentage = 10` (0.1%)

**Buggy code path:**
- `splitAmount = 100,000,000 / 2 = 50,000,000`
- `royaltyAmount = (50,000,000 × 500)/10,000 + (50,000,000 × 10)/10,000 = 2,500,000 + 50,000 = 2,550,000`
- Displayed `amountWithRoyalties` = 102,550,000 mojos = **0.00010255 XCH**

**Correct calculation:**
- `royaltyAmount = (100,000,000 × 500)/10,000 + (100,000,000 × 10)/10,000 = 5,000,000 + 100,000 = 5,100,000`
- Correct total = 105,100,000 mojos = **0.0001051 XCH**

The user sees 0.00010255 XCH and approves; the wallet daemon spends 0.0001051 XCH — a ~2.4% overcharge that scales with royalty rates and grows with each additional NFT in the offer. [4](#0-3)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
