### Title
Overly Broad `account_tx_in_pool_or_recent_block` Check Allows Signature Bypass for Nonce-1 Invoke Transactions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function skips the `__validate__` entry point (signature verification) for invoke transactions with nonce=1 when the account nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` for **any** account with **any** transaction in the pool, not only accounts with a pending `deploy_account`. An unprivileged attacker can exploit this to inject an invalid-signature invoke with nonce=1 for any factory-deployed account (on-chain nonce=0) that already has a pending transaction in the mempool, bypassing gateway signature validation entirely.

### Finding Description

`skip_stateful_validations` is a UX feature that lets users broadcast `deploy_account` + `invoke` (nonce=1) simultaneously before the account is deployed. The gateway skips `__validate__` for the invoke when it detects the deploy_account is already queued.

The guard is implemented as:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
```

`account_tx_in_pool_or_recent_block` is defined as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`tx_pool.contains_account` returns `true` if the account has **any** transaction in the pool — including a plain invoke at nonce=0 submitted by the legitimate owner. There is no check that the pooled transaction is specifically a `deploy_account`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

This means `validate_tx` returns `Ok(None)` immediately without executing `__validate__`:

```rust
// crates/blockifier/src/transaction/account_transaction.rs:999-1001
if !self.execution_flags.validate {
    return Ok(None);
}
```

The attacker's invalid-signature invoke passes all remaining checks:
- `validate_nonce`: nonce=1 is within `max_allowed_nonce_gap=200` of account_nonce=0 ✓
- `validate_by_mempool` (`validate_tx`): checks only duplicate hash and nonce-too-old, not signature ✓
- `run_validate_entry_point`: skipped entirely ✓

The transaction is then forwarded to the mempool via `mempool_client.add_tx`.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invalid-signature invoke transaction is admitted to the mempool and propagated to all sequencer nodes. When the batcher executes it, `__validate__` runs and fails, the transaction is reverted, and the fee is charged to the victim account's balance. The attacker can grief any factory-deployed account (nonce=0) that has a pending transaction, forcing it to pay fees for a reverted transaction it never authorized.

### Likelihood Explanation

**Medium.** The preconditions are:
1. The target account must have been deployed via a factory contract (on-chain nonce=0, account contract exists). This is a common pattern on Starknet.
2. The target account must have at least one pending transaction in the mempool (nonce=0 invoke submitted by the legitimate owner). This is the normal operating state for any active account.
3. The attacker must know the target account address (public information).

No privileged access is required. The default `max_allowed_nonce_gap=200` ensures nonce=1 always passes the nonce range check.

### Recommendation

Replace the overly broad pool-membership check with a type-specific check. `skip_stateful_validations` should only return `true` when a `deploy_account` transaction for the sender address is present in the pool or was recently committed — not when any arbitrary transaction is present. One approach:

```rust
// Only skip if a deploy_account for this address is pending
let has_pending_deploy_account = mempool_client
    .has_deploy_account_in_pool_or_recent_block(tx.sender_address())
    .await?;
return Ok(has_pending_deploy_account);
```

Alternatively, add a dedicated mempool query that checks specifically for a `deploy_account` transaction at nonce=0 for the given address.

Also note: `StatefulTransactionValidatorConfig` defines `max_nonce_for_validation_skip: Nonce` (default `0x1`) but `skip_stateful_validations` hardcodes `Nonce(Felt::ONE)` and never reads this field — the configurable guard is dead code.

### Proof of Concept

**Setup:** Account `A` deployed via factory contract. On-chain nonce = 0. Account contract class exists at address `A`.

**Step 1 — Seed the pool (legitimate owner):**
```
POST /add_transaction
{
  "type": "INVOKE",
  "sender_address": A,
  "nonce": "0x0",
  "signature": [<valid_sig>],
  ...
}
→ 200 OK, tx_hash: H0
```
`tx_pool.contains_account(A)` now returns `true`.

**Step 2 — Inject invalid-signature invoke (attacker):**
```
POST /add_transaction
{
  "type": "INVOKE",
  "sender_address": A,
  "nonce": "0x1",
  "signature": ["0xdeadbeef", "0xdeadbeef"],   // invalid
  ...
}
```

Gateway execution path:
1. `stateless_tx_validator.validate` — passes (signature length ≤ 4000) ✓
2. `convert_rpc_tx_to_internal_and_executable_txs` — passes (no signature check here) ✓
3. `extract_state_nonce_and_run_validations`:
   - `get_nonce_from_state(A)` → `Nonce(0)` ✓
   - `validate_nonce`: `0 ≤ 1 ≤ 200` ✓
   - `validate_by_mempool`: no duplicate, no nonce-too-old ✓
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true` ✓
   - `run_validate_entry_point(skip_validate=true)`: `validate=false`, `__validate__` never called ✓
4. `mempool_client.add_tx(...)` → invalid-signature tx admitted ✓

**Step 3 — Batcher execution:**
Batcher dequeues the nonce=1 invoke, runs `__validate__` → signature check fails → transaction reverted → fee charged to account `A`.

**File:** crates/apollo_gateway/src/stateful_transaction_validator.rs (L429-461)
```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
