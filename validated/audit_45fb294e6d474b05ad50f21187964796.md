### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is intended to skip the `__validate__` entry point only when a `deploy_account` transaction is pending for an account (UX feature for the deploy+invoke flow). However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** account that has **any** transaction in the mempool or committed state, not exclusively accounts with a pending `deploy_account`. An attacker who controls an account with on-chain nonce 0 can first submit a valid invoke at nonce 0 (which passes `__validate__` normally), then submit a second invoke at nonce 1 with an **invalid or arbitrary signature**. The gateway skips `__validate__` for the second transaction and admits it to the mempool without signature verification.

---

### Finding Description

**The skip predicate:** [1](#0-0) 

The function triggers when `tx.nonce() == 1` and `account_nonce == 0`, then calls: [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the account appears in either `self.state` (committed/staged nonce map) or `self.tx_pool` (any pending transaction): [3](#0-2) 

The `committed` map is populated by `commit_block` for every address whose transactions were included in a block, and it persists across blocks up to `committed_nonce_retention_block_count`: [4](#0-3) 

The code comment in `skip_stateful_validations` claims:
> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is incorrect. An account can appear in the mempool because it has a **regular invoke** at nonce 0 — not a `deploy_account`. The fact that a nonce-0 invoke passed `__validate__` says nothing about whether a nonce-1 invoke with a different (or absent) signature should also pass.

**The skip flag propagates to execution flags:** [5](#0-4) 

When `skip_validate = true`, `execution_flags.validate = false`, so `validate_tx` returns `None` immediately without calling the account's `__validate__` entry point: [6](#0-5) 

**The gateway flow that admits the transaction:** [7](#0-6) 

After `extract_state_nonce_and_run_validations` returns without error (because `__validate__` was skipped), the transaction is forwarded to the mempool via `add_tx`.

**The batcher re-enables validation:**

When the batcher later executes the transaction, it uses `new_for_sequencing` which always sets `validate: true`: [8](#0-7) 

So the invalid transaction will fail at execution time. However, the gateway has already admitted it to the mempool without verifying the signature, violating the admission invariant.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can submit invoke transactions with nonce 1 carrying invalid or forged signatures for any account with on-chain nonce 0 that has a concurrent pending transaction in the mempool. The gateway skips the account's `__validate__` entry point and admits these transactions. The mempool is polluted with signature-invalid transactions that consume pool capacity and batcher execution resources before being rejected at execution time.

The `validate_by_mempool` call that precedes the skip check only validates nonce ordering, not signatures: [9](#0-8) [10](#0-9) 

---

### Likelihood Explanation

**Medium.** The preconditions are:
1. An account with on-chain nonce 0 that has deployed contract code (so a nonce-0 invoke can pass `__validate__` normally). This applies to pre-deployed genesis accounts or accounts deployed via mechanisms that leave nonce at 0.
2. A concurrent pending nonce-0 transaction in the mempool for that account (submitted by the attacker or legitimately).

Once condition 2 is met, the attacker can submit arbitrarily many nonce-1 invokes with invalid signatures, each bypassing gateway validation. The mempool's TTL is the only natural bound.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the account. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type of the pending nonce-0 entry, rather than returning true for any transaction type.

Alternatively, the gateway can inspect the `executable_tx` type of the nonce-0 transaction already in the pool before granting the skip.

---

### Proof of Concept

**Setup:** Account `A` is pre-deployed at genesis with nonce 0 and a standard ECDSA `__validate__` function. Attacker controls the private key for `A`.

**Step 1 — Seed the mempool with a valid nonce-0 invoke:**
```
POST /gateway/add_transaction
{
  type: Invoke, sender: A, nonce: 0, signature: <valid_sig_for_nonce_0>, ...
}
```
This passes `__validate__` (account exists, signature valid). Account `A` is now in `tx_pool`.

**Step 2 — Submit a nonce-1 invoke with an invalid signature:**
```
POST /gateway/add_transaction
{
  type: Invoke, sender: A, nonce: 1, signature: [0x1337, 0xdead], ...
}
```

**Gateway evaluation in `skip_stateful_validations`:**
- `tx.nonce() == Nonce(1)` ✓
- `account_nonce == Nonce(0)` ✓ (on-chain nonce is still 0)
- `account_tx_in_pool_or_recent_block(A)` → `tx_pool.contains_account(A)` → **`true`** (nonce-0 invoke is in pool)
- Returns `true` → `execution_flags.validate = false`

**Result:** The nonce-1 invoke with `signature: [0x1337, 0xdead]` is admitted to the mempool without calling `A.__validate__`. The gateway returns a success response with a transaction hash.

**Step 3 — Batcher execution:**
When the batcher executes the block, `new_for_sequencing` sets `validate: true`. The nonce-0 invoke executes first (nonce incremented to 1). The nonce-1 invoke then runs `__validate__`, which rejects the invalid signature. The transaction is marked rejected and removed from the mempool — but it was already admitted and consumed resources. async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

                .committed

                .expect("Account in commit history must appear in the committed nonces.");
            if last_committed_nonce == removed_nonce {
                self.committed.remove(&address);
            }
        }

        addresses_to_rewind
    }
