After thorough investigation of the Metric OMM codebase, I found a genuine native analog.

---

### Title
First pool deployed by `MetricOmmPoolFactory` is permanently excluded from `isPool()` registry due to zero-index collision — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory` uses `poolToIdx[pool] != 0` as the canonical "is this a real pool?" predicate. Because `nextPoolIdx` starts at `0`, the very first pool created receives index `0`, making it indistinguishable from an unregistered address. Every extension's `onlyPool` guard rejects callbacks from that pool, permanently breaking swaps and liquidity operations for any first-deployed pool that uses extensions.

### Finding Description

`nextPoolIdx` is an uninitialized `uint256` state variable that defaults to `0`. During `createPool`, the factory assigns the current index to the new pool and then increments:

```solidity
uint256 poolIdx = nextPoolIdx;   // 0 for the first pool
nextPoolIdx++;                    // becomes 1
idxToPool[poolIdx] = pool;       // idxToPool[0] = firstPool
poolToIdx[pool] = poolIdx;       // poolToIdx[firstPool] = 0
``` [1](#0-0) 

The `isPool` view then checks:

```solidity
function isPool(address pool) external view override returns (bool) {
    return poolToIdx[pool] != 0;
}
``` [2](#0-1) 

For the first pool, `poolToIdx[firstPool] == 0`, so `isPool(firstPool)` returns `false` — identical to any address that was never registered.

Every production extension inherits `BaseMetricExtension`, whose `onlyPool` modifier gates all six hook entry-points:

```solidity
modifier onlyPool() {
    if (!IMetricOmmPoolFactory(FACTORY).isPool(msg.sender)) {
        revert OnlyPool(msg.sender, FACTORY);
    }
    _;
}
``` [3](#0-2) 

This modifier is applied to `beforeAddLiquidity`, `afterAddLiquidity`, `beforeRemoveLiquidity`, `afterRemoveLiquidity`, `beforeSwap`, and `afterSwap`. [4](#0-3) 

### Impact Explanation

Any pool created as the first pool from a factory deployment that configures one or more extensions will have every extension callback revert with `OnlyPool`. This means:

- **Swaps are permanently broken** if `beforeSwap` or `afterSwap` is wired.
- **Liquidity additions are permanently broken** if `beforeAddLiquidity` or `afterAddLiquidity` is wired.
- **Liquidity withdrawals are permanently broken** if `beforeRemoveLiquidity` or `afterRemoveLiquidity` is wired — locking deposited user principal with no recovery path.

The pool itself is legitimate and fully registered in `idxToPool`, but the `isPool` predicate permanently misidentifies it as unregistered. There is no admin function to re-index or repair the mapping.

### Likelihood Explanation

This affects every fresh factory deployment. The first `createPool` call on any newly deployed `MetricOmmPoolFactory` instance triggers the bug. Pools without extensions are unaffected, but the protocol's own periphery (allowlist extension, price guard extension, swap reporter extension) all extend `BaseMetricExtension` and are therefore affected whenever used with the first pool.

### Recommendation

Initialize `nextPoolIdx` to `1` so that the first pool receives index `1` and `poolToIdx[firstPool] != 0` correctly returns `true`:

```solidity
uint256 public override nextPoolIdx = 1;
```

Alternatively, use a separate boolean mapping `isRegisteredPool[pool]` that is set to `true` on registration, removing the zero-index ambiguity entirely.

### Proof of Concept

1. Deploy `MetricOmmPoolFactory` and `MetricOmmPoolDeployer`.
2. Deploy any `BaseMetricExtension`-derived extension (e.g. the periphery allowlist extension).
3. Call `factory.createPool(params)` with that extension in `params.extensions` and a non-zero `extensionOrders.beforeSwap`. This is the first pool; it receives `poolToIdx[pool] = 0`.
4. Assert `factory.isPool(pool) == false` — confirmed despite the pool being legitimately deployed.
5. Attempt a swap on the pool. The pool calls `extension.beforeSwap(...)`. The extension's `onlyPool` modifier calls `factory.isPool(msg.sender)` → `false` → reverts with `OnlyPool`.
6. Attempt `pool.removeLiquidity(...)` after adding liquidity. Same revert path locks deposited funds. [2](#0-1) [1](#0-0) [3](#0-2)
