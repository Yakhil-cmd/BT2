### Title
Malicious CKD Participant Can Corrupt Derived Key Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates participant contributions to the confidential derived key without any cryptographic verification. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a silently corrupted CKD output that honest parties will accept as valid.

### Finding Description
In `do_ckd_coordinator` (lines 35–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives `(norm_big_y, norm_big_c)` from every other participant and unconditionally adds them into the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is supposed to compute, in `compute_signature_share` (lines 148–182):

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

where `x_i` is the participant's private BLS share and `y_i` is a fresh random scalar. The coordinator has no mechanism — no zero-knowledge proof, no commitment, no consistency check against the participant's public key share — to verify that the received `(norm_big_y, norm_big_c)` pair was honestly computed. The protocol documentation for this function is absent; there is no comment explaining what invariants the received values must satisfy or what trust is assumed of participants.

The correct final output requires the sum of all `norm_big_c` values to equal `msk · H(pk ‖ app_id) + Y · app_sk`, where `Y = Σ λ_i · y_i · G`. Any deviation in a single participant's contribution breaks this equation silently.

### Impact Explanation
A single malicious participant can substitute arbitrary BLS12-381 G1 elements for their `(norm_big_y, norm_big_c)` contribution. The coordinator will accept and aggregate these without complaint, producing a `CKDOutput` that does not correspond to any valid derivation of the master secret key. Honest parties who receive this output will treat it as legitimate. The derived confidential key recovered by the application (`ckd_output.unmask(app_sk)`) will be an arbitrary group element unrelated to the true `msk · H(pk ‖ app_id)`, permanently corrupting the CKD result for that `(pk, app_id)` pair.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation
The attack requires only that the adversary control one participant slot in the CKD protocol — a realistic assumption for any unprivileged library caller who is admitted to the participant set. No cryptographic break, no key leakage, and no external dependency compromise is needed. The malicious participant simply deviates from the protocol by sending chosen group elements instead of honestly computed shares. Because there is no verification step, the attack is undetectable and deterministic.

### Recommendation
Each participant should accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct computation relative to their public BLS verification share. Concretely, a proof of knowledge of `(y_i, x_i)` such that:
- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · x_i · H(pk ‖ app_id) + λ_i · y_i · app_pk`
- `λ_i · x_i · G2 = λ_i · vk_share_i` (consistency with the public key share)

The coordinator must verify these proofs before aggregating. Additionally, the function should be documented to state explicitly what trust is assumed of participants and what invariants the received values must satisfy.

### Proof of Concept
1. Honest participants `P_1, …, P_{n-1}` compute and send correct `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant `P_n` sends `(norm_big_y_n, norm_big_c_n) = (G, G)` — arbitrary non-zero G1 points — instead of their honest contribution.
3. The coordinator executes the loop at lines 50–55 of `protocol.rs`, adding `G` to both accumulators without any check.
4. The final `CKDOutput` satisfies `big_c ≠ msk · H(pk ‖ app_id) + big_y · app_sk`.
5. When the application calls `ckd_output.unmask(app_sk)`, it recovers `big_c - big_y · app_sk`, which is not `msk · H(pk ‖ app_id)`.
6. The derived confidential key is permanently corrupted; no honest party can detect the deviation.
