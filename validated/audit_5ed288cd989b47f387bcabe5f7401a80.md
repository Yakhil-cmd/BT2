### Title
`DelegateAction` (meta-transaction) signatures contain no chain-specific binding, enabling cross-chain replay - (`File: core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

### Summary

`DelegateAction` (NEP-366 meta-transactions) signatures are computed over a payload that contains no chain-specific identifier. Regular NEAR `SignedTransaction`s bind to a specific chain via the `block_hash` field; `DelegateAction` has no equivalent. After a hard fork (or across mainnet/testnet when a user holds the same key on both networks), any unprivileged relayer can take a `SignedDelegateAction` produced for one chain and submit it verbatim on the other chain, where it will pass signature verification, nonce checks, and `max_block_height` checks and execute the inner actions without the user's consent.

### Finding Description

`DelegateAction::get_nep461_hash()` computes the signed hash as:

```
sha256( borsh( SignableMessage { discriminant: MessageDiscriminant(NEP_366_constant), msg: &delegate_action } ) )
``` [1](#0-0) 

The `DelegateAction` payload covers only `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`. [2](#0-1) 

The `MessageDiscriminant` is a compile-time constant (`1<<30 + 366`); it does not vary by chain. [3](#0-2) [4](#0-3) 

By contrast, a regular `SignedTransaction` includes a `block_hash` field that is chain-specific (it must be an ancestor block on the chain where the transaction is submitted), providing cross-chain isolation.

The runtime validation in `apply_delegate_action` / `validate_delegate_action_key` checks:
1. Signature validity — passes on both chains (same key, same payload hash).
2. Nonce > current nonce — passes on the forked chain because both chains share the same starting state.
3. `block_height <= max_block_height` — passes if the fork occurs before `max_block_height`.
4. Sender account existence — passes because both chains share the same genesis state. [5](#0-4) 

No guard checks that the action was intended for the current chain.

### Impact Explanation

After a NEAR hard fork (or when a user holds the same key on mainnet and testnet), a malicious relayer can replay a `SignedDelegateAction` on the unintended chain. The inner actions execute with `sender_id` as the predecessor, so the replay can:

- Transfer tokens from the user's account on the forked chain.
- Add or delete access keys on the user's account.
- Call arbitrary contract methods on behalf of the user.
- Consume the user's nonce, blocking legitimate use of that nonce on the forked chain.

The corrupted values are the user's token balance, access-key set, and nonce on the forked chain.

### Likelihood Explanation

The attack requires an unprivileged relayer (any network participant) and one of:
- A NEAR hard fork (both chains share identical account state at the fork point, so nonces match exactly).
- A user who holds the same key on mainnet and testnet with a testnet nonce lower than the `DelegateAction`'s nonce.

The relayer role is explicitly unprivileged and open to any party. [6](#0-5) 

### Recommendation

Include the chain identifier in the signed payload of `DelegateAction`. The simplest approach is to add `chain_id` (the genesis `chain_id` string, e.g. `"mainnet"`) to `DelegateAction` and cover it in `get_nep461_hash()`. Alternatively, include the genesis block hash. This mirrors the fix applied to the GMX oracle bug (adding `block.chainid` to the signed data at call time rather than caching it at construction).

The `chain_id` is already available to contracts via the `chain_id` host function: [7](#0-6) 

and is part of the genesis config exposed via RPC. It should be added as a field to `DelegateAction` and `DelegateActionV2`, and included in the borsh-serialized payload before hashing.

### Proof of Concept

1. Alice signs a `DelegateAction` on mainnet: `sender_id="alice.near"`, `receiver_id="token.near"`, `actions=[Transfer{deposit:1e24}]`, `nonce=5`, `max_block_height=10_000_000`, `public_key=<alice_key>`.
2. NEAR hard-forks at block 1_000_000 (before `max_block_height`). Both chains share Alice's state: nonce=4.
3. A malicious relayer wraps the same `SignedDelegateAction` in a transaction on the forked chain and submits it.
4. `SignedDelegateAction::verify()` calls `get_nep461_hash()` — the hash is identical on both chains (no chain-specific data). Signature check passes.
5. `validate_delegate_action_key` checks nonce: `5 > 4` ✓. Checks `max_block_height`: `1_000_001 <= 10_000_000` ✓.
6. The transfer executes on the forked chain, draining Alice's balance there without her consent. [8](#0-7) [9](#0-8)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

    pub fn get_nep461_hash(&self) -> CryptoHash {
        let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
        let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
        hash(&bytes)
    }

/// Validate access key which was used for signing DelegateAction:
///
/// - Checks whether the access key is present fo given public_key and sender_id.
/// - Validates nonce and updates it if it's ok.
/// - Validates access key permissions.
fn validate_delegate_action_key(
    state_update: &mut TrieUpdate,
    apply_state: &ApplyState,
    delegate_action: VersionedDelegateActionRef<'_>,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let sender_id = delegate_action.sender_id();
    let public_key = delegate_action.public_key();
    // 'sender_id' account existence must be checked by a caller
    let mut access_key = match get_access_key(state_update, sender_id, public_key)? {
        Some(access_key) => access_key,
        None => {
            result.result = Err(ActionErrorKind::DelegateActionAccessKeyError(
                InvalidAccessKeyError::AccessKeyNotFound {
                    account_id: sender_id.clone(),
                    public_key: public_key.clone().into(),
                },
            )
            .into());
            return Ok(());
        }
    };
