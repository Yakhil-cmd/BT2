- **ERC-1271 report**: `isValidSignature` fallback omits `address(this)` binding → same `(hash, sig)` pair accepted by multiple accounts on the same chain.
- **Aurora Engine analog**: `submit` omits chain_id validation when `transaction.chain_id` is `None` (pre-EIP-155, v=27/28) → same signed transaction accepted by multiple Aurora Engine instances with different chain IDs.

The root cause is in `engine/src/engine.rs:1054-1059` (conditional chain_id check) and `engine-transactions/src/legacy.rs:68-78` (v=27/28 produces `chain_id = None`).

### Title
Pre-EIP-155 Legacy Transaction Cross-Chain Replay Enables Unauthorized Fund Transfers - (File: `engine/src/engine.rs`)

### Summary

The `submit_with_alt_modexp` function in `engine/src/engine.rs` validates the chain ID only when it is present in the transaction signature. Legacy transactions with `v=27` or `v=28` (pre-EIP-155) carry no chain ID, so the validation is silently skipped. Because Aurora Engine explicitly re-enabled acceptance of pre-EIP-155 transactions (v2.6.0), the same signed transaction bytes are accepted by any Aurora Engine instance regardless of its configured chain ID, enabling cross-chain replay attacks that drain user funds.

### Finding Description

**Root cause — missing chain-ID binding for pre-EIP-155 transactions:**

`LegacyEthSignedTransaction::sender()` in `engine-transactions/src/legacy.rs` decodes the `v` field and, for `v=27` or `v=28`, sets `chain_id = None` and signs the message hash over only the six-field RLP (nonce, gas_price, gas_limit, to, value, data) with no chain ID appended:

```rust
// engine-transactions/src/legacy.rs:68-78
let (chain_id, rec_id) = match self.v {
    0..=26 | 29..=34 => return Err(Error::InvalidV),
    27..=28 => (
        None,                                          // ← no chain-ID binding
        u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
    ),
    _ => (
        Some((self.v - 35) / 2),
        u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
    ),
};
self.transaction.rlp_append_unsigned(&mut rlp_stream, chain_id);
``` [1](#0-0) 

`TransactionLegacy::rlp_append_unsigned` only appends the chain ID fields when `chain_id` is `Some`:

```rust

    s.begin_list(if chain_id.is_none() { 6 } else { 9 });

``` [2](#0-1) 

The `NormalizedEthTransaction` produced from a legacy transaction therefore carries `chain_id: None`:

```rust
// engine-transactions/src/lib.rs:106-118
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // ← None for v=27/28

``` [3](#0-2) 

In `submit_with_alt_modexp`, the chain-ID guard is wrapped in `if let Some(chain_id) = transaction.chain_id`, so it is entirely bypassed when `chain_id` is `None`:

```rust
// engine/src/engine.rs:1054-1059
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [4](#0-3) 

Aurora Engine deliberately re-enabled pre-EIP-155 transactions in v2.6.0 (after blocking them in v2.4.0 as a security improvement) to support EIP-1820 contract deployment. This means every production Aurora Engine instance — mainnet, testnet, and any Silo deployment — accepts transactions with `v=27/28`. [5](#0-4) 

**Exploit path:**

1. Alice holds ETH on Aurora mainnet (chain_id = 1313161554) and on an Aurora Silo (chain_id = X).
2. Alice's nonce is N on both instances.
3. Alice submits a pre-EIP-155 transaction (v=27) on Aurora mainnet: transfer Y ETH to Bob.
4. An attacker observes the raw transaction bytes (public mempool / on-chain).
5. The attacker submits the identical bytes to the Aurora Silo via `submit`.
6. `submit_with_alt_modexp` recovers Alice's address correctly (the signature is chain-agnostic), the nonce check passes (nonce = N on both chains), and the chain-ID guard is skipped because `chain_id = None`.
7. Alice loses Y ETH on the Silo without her consent.

The same path applies cross-ecosystem: a pre-EIP-155 transaction signed for Ethereum mainnet (or any EVM chain) can be replayed on Aurora if the sender's nonce and balance align.

### Impact Explanation

**Critical — direct theft of user funds.**

An unprivileged attacker who observes a single pre-EIP-155 transaction can replay it on any other Aurora Engine instance where the sender's nonce matches, transferring the sender's ETH to the original recipient without the sender's knowledge or intent. The attacker requires no keys, no governance access, and no special role — only the ability to submit a transaction to the target Aurora Engine contract.

### Likelihood Explanation

**Medium.** Pre-conditions required:

- The victim must submit a pre-EIP-155 transaction (v=27/28). This occurs when using legacy tooling, EIP-1820 deployment scripts, or any wallet/library that omits chain ID.
- The victim must hold funds on a second Aurora Engine instance (mainnet ↔ testnet, mainnet ↔ Silo, or cross-ecosystem from Ethereum mainnet).
- The victim's nonce on the target chain must equal the nonce in the replayed transaction.

Nonce alignment is not guaranteed but is realistic: users who deploy fresh Aurora Silo environments or who use both mainnet and testnet with similar transaction histories will frequently have matching nonces. The attacker's cost is zero beyond gas on the target chain.

### Recommendation

Reject pre-EIP-155 transactions at the engine level by returning `EngineErrorKind::InvalidChainId` whenever `transaction.chain_id` is `None`, restoring the v2.4.0 security posture:

```rust
// engine/src/engine.rs — replace the current conditional
match transaction.chain_id {
    None => return Err(EngineErrorKind::InvalidChainId.into()),
    Some(chain_id) if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) => {
        return Err(EngineErrorKind::InvalidChainId.into());
    }
    _ => {}
}
```

If EIP-1820 support is required, deploy the EIP-1820 contract through a privileged admin path (e.g., `call`) rather than through the public `submit` endpoint, so that the chain-ID enforcement can remain unconditional for all user-submitted transactions.

### Proof of Concept

```
Setup:
  Aurora mainnet  chain_id = 1313161554
  Aurora Silo     chain_id = 9999
  Alice's address = 0xAlice, nonce = 5 on both instances, balance = 1 ETH on both

Step 1 — Alice signs a pre-EIP-155 transfer on mainnet:
  tx = { nonce:5, gasPrice:1gwei, gasLimit:21000, to:0xBob, value:0.5 ETH, data:[] }
  sign(tx, chain_id=None)  →  v=27, r=R, s=S
  raw_bytes = rlp_encode(tx, v=27, r=R, s=S)

Step 2 — Alice submits raw_bytes to Aurora mainnet → succeeds, nonce becomes 6.

Step 3 — Attacker submits the same raw_bytes to Aurora Silo:
  submit(raw_bytes)
  → EthTransactionKind::Legacy decoded, sender() = 0xAlice (chain-agnostic ecrecover)
  → chain_id = None  →  chain-ID guard skipped
  → check_nonce(0xAlice, 5) passes (nonce is still 5 on Silo)
  → 0.5 ETH transferred from Alice to Bob on the Silo

Result: Alice loses 0.5 ETH on the Silo without signing any Silo transaction.
```
