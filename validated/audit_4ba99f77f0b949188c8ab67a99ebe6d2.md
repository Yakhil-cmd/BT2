### Title
`EVM.mustRun()` Silently Passes Failed EVM Transactions, Enabling Cross-VM Asset Accounting Discrepancies — (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in the Flow EVM system contract is the protocol-level analog to unchecked ERC-20 `transfer()` return values. Its internal assertion explicitly permits `status == failed` (EVM execution reverted) to pass without reverting the enclosing Cadence transaction. Any Cadence transaction or contract that calls `mustRun` to execute an EVM token transfer and then performs Cadence-side state mutations — without separately asserting `res.status == EVM.Status.successful` — will silently proceed even when the EVM-side transfer has failed, creating cross-VM asset accounting discrepancies.

---

### Finding Description

`EVM.mustRun()` is defined in the production system contract `fvm/evm/stdlib/contract.cdc` as `access(all)`, making it callable by any unprivileged Cadence transaction:

```cadence
// fvm/evm/stdlib/contract.cdc lines 844–851
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
```

The assert condition is:

```
runResult.status == Status.failed || runResult.status == Status.successful
```

The `Status` enum has four values:
- `unknown` — something went very wrong
- `invalid` — transaction rejected at validation (e.g. nonce mismatch)
- `failed` — **transaction executed but VM reported an error (e.g. EVM revert, out of gas)**
- `successful` — transaction executed without error

The assertion only rejects `unknown` and `invalid`. It explicitly allows `status == failed` — meaning a fully reverted EVM transaction (e.g. a failed ERC-20 `transfer()`, an ERC-20 `transferFrom()` that reverts due to insufficient allowance, or any EVM call that reverts) passes the assertion and returns to the caller with `status == failed`. The Cadence transaction is **not** rolled back.

The contract's own comment acknowledges this:

> "Note that this method does not rollback if transaction is executed but an vm error is reported as the outcome of the execution (status: failed)."

The name `mustRun` implies a "safe" or "enforced" execution — directly analogous to how `SafeERC20.safeTransfer()` was intended to be the safe wrapper over `transfer()`. But unlike `safeTransfer()`, `mustRun` does not enforce that the underlying operation succeeded.

---

### Impact Explanation

Any Cadence transaction or contract that:
1. Calls `EVM.mustRun()` to execute an EVM token transfer (ERC-20 `transfer`, `transferFrom`, ETH value transfer, etc.)
2. Does **not** explicitly assert `res.status == EVM.Status.successful` after the call
3. Proceeds with Cadence-side state mutations (e.g. releasing escrowed Cadence assets, updating ledger balances, minting Cadence tokens)

…will silently proceed with those Cadence-side mutations even when the EVM-side transfer has failed. This produces cross-VM asset accounting discrepancies: Cadence state reflects a completed transfer that never occurred on the EVM side, or vice versa. In a bridge context this is a direct bridge escrow mis-accounting / cross-VM asset loss.

---

### Likelihood Explanation

`EVM.mustRun()` is `access(all)` and is the documented "safe" entry point for running EVM transactions from Cadence. Contract authors building Cadence↔EVM bridges or relayer contracts are the primary consumers. The misleading name creates a strong expectation that the function enforces success — exactly the same footgun as unchecked ERC-20 `transfer()` return values. Any such contract that omits a post-call `assert(res.status == EVM.Status.successful, ...)` is silently vulnerable. The entry path requires only an unprivileged Cadence transaction submission.

---

### Recommendation

Change the assertion in `mustRun` to enforce that the EVM transaction actually succeeded:

```cadence
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.successful,
        message: "EVM.mustRun(): EVM transaction did not succeed (status: \(runResult.status.rawValue), error: \(runResult.errorMessage))"
    )
    return runResult
}
```

This makes `mustRun` the true analog of OpenZeppelin's `SafeERC20.safeTransfer()` — it reverts the Cadence transaction whenever the EVM-side execution fails or is invalid, eliminating the silent-fail class entirely for callers who use `mustRun`.

### Proof of Concept

An unprivileged user submits the following Cadence transaction. The inner EVM transaction is crafted to call an ERC-20 contract's `transfer()` with an amount exceeding the sender's balance (causing an EVM revert). `mustRun` returns `status == failed` but the Cadence transaction succeeds, and any Cadence-side state mutations after the call proceed on a false premise:

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>

transaction(revertingEvmTx: [UInt8], coinbaseBytes: [UInt8; 20]) {
    prepare(account: auth(Storage) &Account) {
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)

        // mustRun does NOT revert when the EVM tx reverts (status == failed).
        // The assert only rejects status == unknown or status == invalid.
        let res = EVM.mustRun(tx: revertingEvmTx, coinbase: coinbase)

        // res.status == EVM.Status.failed here — EVM transfer silently failed.
        // Any Cadence-side asset release or ledger update below executes
        // unconditionally, producing cross-VM accounting discrepancy.
        // e.g.: release escrowed Cadence vault, mint bridged tokens, etc.
    }
}
```
