### Title
`skip_stateful_validations` Bypasses Signature Check When Any Future-Nonce Transaction Exists in Pool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip blockifier `__validate__` for an invoke at nonce=1 when a `deploy_account` is still pending (UX improvement). Its guard uses `account_tx_in_pool_or_recent_block`, which returns `true` if **any** transaction from the account is in the pool — not only a `deploy_account`. An attacker who observes that a deployed account (on-chain nonce=0) has a future-nonce invoke in the mempool can submit a forged invoke at nonce=1 with an invalid signature, and the gateway will admit it without ever calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` at line 429 of `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when:

```
tx.nonce() == Nonce(Felt::ONE)  &&  account_nonce == Nonce(Felt::ZERO)
```

and `account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It does **not** distinguish between `DeployAccount` and `Invoke` transaction types. The code comment claims this is safe because "transactions with future nonces that passed validations" imply the account is valid, but that reasoning only proves the account's contract code exists — it does not prove the signature on the *current* nonce=1 transaction is valid.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The blockifier's `StatefulValidator::perform_validations` then skips the `__validate__` entry-point call entirely:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

The mempool's own `validate_tx` only checks for duplicate hashes and nonce ordering — it performs no signature verification:

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

The gateway's nonce check allows nonce=1 for an account at nonce=0 (within `max_allowed_nonce_gap`):

```rust
let max_allowed_nonce =
    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
    return Err(...)
}
``` [6](#0-5) 

So the forged invoke(nonce=1) passes every gateway check and is inserted into the mempool.

---

### Impact Explanation

**Critical — Invalid or unauthorized Starknet transaction accepted through account validation / signature logic.**

A forged invoke transaction carrying an arbitrary (invalid) signature for an account the attacker does not control is accepted by the gateway and inserted into the mempool. The gateway's sole mechanism for verifying account authorization — the `__validate__` entry-point call — is skipped. The transaction occupies mempool capacity, is handed to the batcher, and will only be rejected at execution time (when the batcher runs `__validate__` with `validate=true`). Until that point the sequencer treats it as a legitimate pending transaction.

---

### Likelihood Explanation

**Medium.** The precondition — a deployed account at on-chain nonce=0 with at least one future-nonce invoke already in the mempool — arises naturally whenever a user submits transactions out of order (e.g., nonce=2 arrives before nonce=1 due to a retry or race condition). An attacker can passively monitor the public mempool for such accounts and immediately submit a forged nonce=1 invoke. No privileged access is required.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific check that only returns `true` when a `DeployAccount` transaction for the sender is present in the pool or recent block. Alternatively, add a secondary guard that verifies the account has no deployed contract code on-chain before skipping validation, so that already-deployed accounts (nonce=0 but code present) are never eligible for the skip path.

---

### Proof of Concept

1. Account `A` is deployed on-chain; its nonce is `0` (no committed transactions yet).
2. The legitimate owner submits `Invoke(nonce=2)` with a valid signature. The gateway accepts it (nonce gap ≤ `max_allowed_nonce_gap`, blockifier `__validate__` passes). It enters `tx_pool` for address `A`.
3. Attacker submits `Invoke(nonce=1)` for address `A` with a **forged/invalid signature**.
4. Gateway stateful validation:
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate hash, nonce not too old → passes.
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `tx_pool.contains_account(A)==true` (from step 2) → returns `true`.
5. `run_validate_entry_point` is called with `validate=false`; `__validate__` is **never called**.
6. `add_tx` inserts the forged transaction into the mempool. The gateway returns success to the attacker.

The forged `Invoke(nonce=1)` with an invalid signature is now a live mempool entry for account `A`, accepted without any signature verification.
