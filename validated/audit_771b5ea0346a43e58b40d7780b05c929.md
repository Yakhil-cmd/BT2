### Title
Truncating Division in `native_per_pubdata` Computation Undercharges Users for Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio used to charge users for pubdata consumption is computed with truncating (floor) division, while the analogous `native_per_gas` ratio uses ceiling division. This inconsistency causes the protocol to systematically undercharge every transaction that generates pubdata whenever `pubdata_price` is not exactly divisible by `native_price`, favoring the user over the protocol.

---

### Finding Description

In `validation_impl.rs`, the two key resource-pricing ratios are computed as follows:

**`native_per_gas` — uses ceiling division (correct, favors protocol):** [1](#0-0) 

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

**`native_per_pubdata` — uses truncating division (incorrect, favors user):** [2](#0-1) 

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is standard truncating integer division. When `pubdata_price % native_price != 0`, the result is floored, so the user is charged `floor(pubdata_price / native_price)` native units per pubdata byte instead of `ceil(pubdata_price / native_price)`.

This `native_per_pubdata` value is then used directly to charge for all pubdata consumed during execution: [3](#0-2) 

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
```

The same truncating path is replicated in the public API helper: [4](#0-3) 

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

A secondary, smaller instance of the same class exists in `compute_gas_refund`, where `delta_gas` is computed with truncating division, giving the user a slightly larger gas refund than intended: [5](#0-4) 

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
```

---

### Impact Explanation

**Type:** Resource accounting bug — rounding direction favors user over protocol.

For every transaction that generates pubdata when `pubdata_price % native_price != 0`, the user pays up to `(native_price - 1)` fewer native units per pubdata byte than the operator intends. In the worst case (e.g., `pubdata_price = native_price + 1`), the user pays only `1` native unit per pubdata byte instead of `2` — a 50% discount on pubdata costs. An attacker who generates large amounts of pubdata (e.g., many storage writes) across many transactions can systematically drain the operator's expected revenue from pubdata fees. This is a direct financial loss to the protocol/operator, accumulating over time.

---

### Likelihood Explanation

**High.** The operator sets `pubdata_price` and `native_price` independently as block-level parameters. In practice, `pubdata_price % native_price != 0` is the common case. Every L2 transaction that writes storage or emits events generates pubdata and is affected. No special attacker capability is required — any unprivileged user submitting normal transactions triggers this path.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (truncates, favors user):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling, favors protocol):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs`. Additionally, replace the truncating `native_used / native_per_gas` in `compute_gas_refund` with `native_used.div_ceil(native_per_gas)` to ensure `delta_gas` does not undercharge.

---

### Proof of Concept

**Setup:**
- `native_price = 10`
- `pubdata_price = 19`
- A transaction writes 1000 storage slots → ~32,000 pubdata bytes

**Current behavior (truncating):**
- `native_per_pubdata = floor(19 / 10) = 1`
- Native charged for pubdata = `32,000 * 1 = 32,000`

**Correct behavior (ceiling):**
- `native_per_pubdata = ceil(19 / 10) = 2`
- Native charged for pubdata = `32,000 * 2 = 64,000`

The user pays 32,000 fewer native units than intended — a 50% discount on pubdata costs. Repeated across many transactions, this accumulates into a significant underpayment to the operator. The attacker-controlled entry path is any standard L2 transaction with storage writes, requiring no privileged access.
