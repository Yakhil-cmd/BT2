### Title
Service Account Single Point of Failure Permanently Locks EFM Recovery - (File: `cmd/util/cmd/epochs/cmd/recover.go`)

### Summary
The Flow protocol's Epoch Fallback Mode (EFM) can only be exited via the `EpochRecover` service event, which must originate exclusively from a manual service account transaction. No fallback mechanism exists to allow any other account to trigger recovery. If the service account's private key becomes inaccessible, the network is permanently stuck in EFM with no on-chain path to recovery — a direct structural analog to the Fantom `tokensTradeable` single-owner lock.

### Finding Description
When the Flow network enters Epoch Fallback Mode (EFM) — triggered by a failed DKG or an invalid epoch service event — the only protocol-defined exit path is the `EpochRecover` service event. This event must be emitted by the `FlowEpoch` smart contract, which requires a transaction authorized by the service account (the account holding the `FlowEpoch.Admin` resource).

The codebase explicitly documents this constraint in the `generateRecoverEpochTxArgsCmd` description:

> "EFM can be exited only by a special service event, EpochRecover, which initially originates from a **manual service account transaction**." [1](#0-0) 

The `MakeRecoverEpochTx` helper constructs the recovery transaction with `adminAccount.Address` as the sole authorizer — no alternative signer path exists: [2](#0-1) 

The `FallbackStateMachine.ProcessEpochRecover` in the protocol state machine processes the `EpochRecover` service event, but this event can only be produced upstream by the service account calling the `FlowEpoch` Cadence contract: [3](#0-2) 

There is no time-based fallback, no multi-sig escape hatch, and no mechanism for any other account to trigger recovery after a waiting period. The `ensureValidEpochRecover` check enforces a finalization safety threshold, but this only gates *when* within an epoch extension recovery can be submitted — it does not open an alternative authorization path: [4](#0-3) 

### Impact Explanation
If the service account's private key is lost or permanently inaccessible:
- The network is permanently stuck in EFM (`EpochFallbackTriggered = true`).
- No new epochs can ever be committed.
- Staking rewards cannot be distributed.
- New nodes cannot join the network.
- The epoch lifecycle is permanently paralyzed.

The network continues producing blocks via epoch extensions added by `FallbackStateMachine`, but it can never transition to a new epoch. This is a permanent, irreversible protocol paralysis caused by a single point of failure — structurally identical to the Fantom `tokensTradeable` lock. [5](#0-4) 

### Likelihood Explanation
The service account is controlled by the Flow Foundation with presumably robust key management. However, the vulnerability is a **design flaw**, not an operational one: there is no fallback mechanism whatsoever. Key loss due to hardware failure, organizational disruption, or key management error would permanently paralyze the epoch lifecycle with no on-chain recovery path. The EFM recovery path was explicitly designed as a governance-only action, making this a known architectural single point of failure.

### Recommendation
Add a time-based fallback mechanism analogous to the Fantom report's recommendation. The `recoverEpoch` Cadence transaction (referenced at `cmd/bootstrap/run/epochs.go` line 264 pointing to `flow-core-contracts`) should be modified to allow any account to call it after a sufficient number of epoch extensions have elapsed without a successful recovery — removing the exclusive service account requirement after a grace period. In Go terms, the `MakeRecoverEpochTx` helper and the underlying Cadence transaction should support an alternative authorization path:

```
require(msg.sender == serviceAccount || epochExtensionCount > N)
``` [6](#0-5) 

### Proof of Concept
1. The network enters EFM — e.g., a collection node is paused during DKG, causing DKG failure and setting `EpochFallbackTriggered = true` in the protocol state. [7](#0-6) 

2. The service account's private key becomes inaccessible.

3. No `EpochRecover` service event can be emitted — the `FlowEpoch` smart contract's `recoverEpoch` function requires service account authorization, and `MakeRecoverEpochTx` has no alternative authorizer path.

// MakeRecoverEpochTx makes an admin transaction to recover the network when it is in EFM mode.
func MakeRecoverEpochTx(
	env templates.Environment,
	adminAccount *sdk.Account,
	adminAccountKeyID uint32,
	latestBlockID sdk.Identifier,
	args []cadence.Value,
) (*sdk.Transaction, error) {
	accountKey := adminAccount.Keys[adminAccountKeyID]
	tx := sdk.NewTransaction().
		SetScript(templates.GenerateRecoverEpochScript(env)).
		SetComputeLimit(9999).
		SetReferenceBlockID(latestBlockID).
		SetProposalKey(adminAccount.Address, adminAccountKeyID, accountKey.SequenceNumber).
		SetPayer(adminAccount.Address).
		AddAuthorizer(adminAccount.Address)

	for _, arg := range args {
		err := tx.AddArgument(arg)
		if err != nil {
			return nil, err
		}
	}

	return tx, nil
}

func (m *FallbackStateMachine) ensureValidEpochRecover(epochRecover *flow.EpochRecover) error {
	if m.view+m.parentState.GetFinalizationSafetyThreshold() >= m.state.CurrentEpochFinalView() {
		return protocol.NewInvalidServiceEventErrorf("could not process epoch recover, safety threshold reached")
	}
	err := protocol.IsValidExtendingEpochSetup(&epochRecover.EpochSetup, m.state)
	if err != nil {
		return fmt.Errorf("invalid setup portion in EpochRecover event: %w", err)
	}
	err = protocol.IsValidEpochCommit(&epochRecover.EpochCommit, &epochRecover.EpochSetup)
	if err != nil {
		return fmt.Errorf("invalid commit portion in EpochRecover event: %w", err)
	}
	return nil
}

func (s *RecoverEpochSuite) TestRecoverEpoch() {
	// 1. Manually trigger EFM
	// pause the collection node to trigger EFM by failing DKG
	ln := s.GetContainersByRole(flow.RoleCollection)[0]
	require.NoError(s.T(), ln.Pause())
	s.AwaitFinalizedView(s.Ctx, s.GetDKGEndView(), 2*time.Minute, 500*time.Millisecond)
	// start the paused collection node now that we are in EFM
	require.NoError(s.T(), ln.Start())

	// get final view from the latest snapshot
	epoch1, err := s.Net.BootstrapSnapshot.Epochs().Current()
	require.NoError(s.T(), err)
	epoch1FinalView := epoch1.FinalView()

	// Wait for at least the first view past the current epoch's original FinalView to be finalized.
	s.TimedLogf("waiting for epoch transition (finalized view %d)", epoch1FinalView+1)
	s.AwaitFinalizedView(s.Ctx, epoch1FinalView+1, 2*time.Minute, 500*time.Millisecond)
	s.TimedLogf("observed finalized view %d", epoch1FinalView+1)

	// assert that we are in EFM
	snapshot, err := s.Client.GetLatestProtocolSnapshot(s.Ctx)
	require.NoError(s.T(), err)
	epochPhase, err := snapshot.EpochPhase()
	require.NoError(s.T(), err)
	require.Equal(s.T(), flow.EpochPhaseFallback, epochPhase, "network must enter EFM by this point")
