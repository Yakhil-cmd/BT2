### Title
Zero-Reward `claim_rewards` Executes Full State Write and Emits Spurious Event - (File: `src/pool/pool.cairo`)

### Summary
`claim_rewards` in the delegation pool contract does not guard against a zero-reward outcome. When called with no accrued rewards, it unconditionally writes three fields of `pool_member_info` to storage, dispatches a zero-amount ERC-20 transfer, and emits a `PoolMemberRewardClaimed` event with `amount: 0`. Any pool member or their reward address can invoke this path repeatedly at no protocol cost, producing unbounded spurious state writes and misleading on-chain events.

### Finding Description
`claim_rewards` in `src/pool/pool.cairo` follows this sequence unconditionally:

1. Calculates `rewards` (may be zero).
2. Clears `pool_member_info._unclaimed_rewards_from_v0`, advances `entry_to_claim_from`, and advances `reward_checkpoint`.
3. Writes the mutated `pool_member_info` back to storage via `write_pool_member_info`.
4. Calls `reward_token.checked_transfer(recipient: reward_address, amount: rewards.into())` — a zero-amount ERC-20 transfer.
5. Emits `Events::PoolMemberRewardClaimed { pool_member, reward_address, amount: rewards }` with `amount = 0`. [1](#0-0) 

There is no early-exit guard of the form `if rewards.is_zero() { return 0; }` anywhere before steps 3–5. The existing test `test_claim_rewards_no_rewards` explicitly confirms the function succeeds and returns zero without reverting. [2](#0-1) 

The access-control check permits both the `pool_member` address and the `reward_address` to be the caller. [3](#0-2) 

Because `reward_address` is a separate, caller-controlled address set at `enter_delegation_pool` time, it can be a contract that autonomously and repeatedly invokes `claim_rewards` on behalf of any pool member it is the reward address for.

The same pattern exists in the staking contract's `claim_rewards`, which calls `send_rewards_to_staker` and `write_staker_info` unconditionally even when `unclaimed_rewards_own == 0`. [4](#0-3) 

### Impact Explanation
Every zero-reward invocation performs a full storage write (three fields of `pool_member_info`) and a zero-value ERC-20 `transfer` call, both of which consume non-trivial gas. It also emits a `PoolMemberRewardClaimed` event with `amount: 0`, which pollutes the event log and can mislead off-chain indexers, dashboards, and reward-tracking tools into believing a reward distribution occurred. Because there is no rate limit or cost imposed on the caller beyond gas, the reward address (an unprivileged, attacker-controlled account) can loop this call to produce unbounded cumulative gas consumption and an unbounded stream of zero-amount reward events.

**Matched allowed impact:** Medium — Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption.

### Likelihood Explanation
The entry path requires only that the caller be the `pool_member` or its `reward_address`. The `reward_address` is set freely by the pool member at delegation time and is never validated beyond a token-address exclusion check. Any pool member who sets a contract as their reward address, or any pool member who wishes to self-grief, can trigger this path at will. No privileged role, bridge compromise, or external dependency is required.

### Recommendation
Add an early return in `claim_rewards` (pool) and `claim_rewards` (staking) when the computed reward amount is zero, mirroring the fix suggested in the reference report:

```cairo
// In pool.cairo claim_rewards, after computing `rewards`:
if rewards.is_zero() {
    return 0;
}
```

This eliminates the unnecessary storage write, the zero-value token transfer, and the spurious event emission in the no-reward case.

### Proof of Concept

1. Deploy the system and stake with a pool enabled.
2. Have a delegator call `enter_delegation_pool` with a reward address set to an attacker-controlled contract `RewardSpammer`.
3. Without advancing any epoch or accruing any rewards, `RewardSpammer` calls `pool.claim_rewards(pool_member)` in a loop.
4. Each call succeeds (confirmed by `test_claim_rewards_no_rewards`), writes three storage slots, executes a zero-value STRK transfer, and emits `PoolMemberRewardClaimed { amount: 0 }`.
5. The on-chain event log accumulates an unbounded number of zero-amount reward-claimed events; storage write gas is consumed on every iteration with no state change of consequence. [5](#0-4) [6](#0-5)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

fn test_claim_rewards_no_rewards() {
    let mut cfg: StakingInitConfig = Default::default();
    general_contract_system_deployment(ref :cfg);
    let token = cfg.test_info.strk_token;
    let token_address = token.contract_address();
    let token_dispatcher = IERC20Dispatcher { contract_address: token_address };

    let pool_contract = stake_with_strk_pool_enabled(:cfg);
    enter_delegation_pool_for_testing_using_dispatcher(:pool_contract, :cfg, :token);

    let pool_member = cfg.test_info.pool_member_address;
    // Balance is zero.
    let rewards = claim_rewards_for_pool_member(:pool_contract, :pool_member);
    assert!(rewards == Zero::zero());
    assert!(token_dispatcher.balance_of(cfg.pool_member_info.reward_address) == Zero::zero());

    advance_epoch_global();

    // Balance is not zero but no rewards.
    let rewards = claim_rewards_for_pool_member(:pool_contract, :pool_member);
    assert!(rewards == Zero::zero());
    assert!(token_dispatcher.balance_of(cfg.pool_member_info.reward_address) == Zero::zero());
